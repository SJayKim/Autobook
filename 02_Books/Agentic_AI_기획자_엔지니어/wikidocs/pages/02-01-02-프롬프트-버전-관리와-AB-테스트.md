# 2.1.2 프롬프트 버전 관리와 A/B 테스트

2.1.1에서 Role, Context, Few-shot Examples 등 8가지 기법으로 프롬프트를 설계하는 방법을 다뤘습니다. 이 기법들을 조합하면 모델의 응답 품질을 크게 높일 수 있었습니다. 그런데 실무에서는 한 가지 문제가 더 있습니다. 잘 만든 프롬프트를 운영 환경에 적용한 뒤, 시간이 지나면서 프롬프트를 수정해야 하는 상황이 반드시 찾아옵니다. 비즈니스 요구사항이 바뀌거나, 사용자 피드백을 반영하거나, 모델을 업그레이드할 때마다 프롬프트도 함께 바뀌어야 합니다.

문제는 이 변경 과정에서 혼란이 발생한다는 점입니다. 프롬프트가 코드 저장소의 설정 파일, 문서, 대시보드 등 여러 곳에 분산되어 있으면, 현재 운영 중인 버전이 어느 것인지 파악하기 어렵습니다. 특정 시점에 어떤 프롬프트로 생성된 응답인지 추적할 수 없습니다. 문제가 발생했을 때 이전 버전으로 돌아가려면 전체 애플리케이션을 다시 배포해야 합니다. 팀원 여러 명이 같은 프롬프트를 동시에 수정하면 충돌이 일어납니다.

이런 문제들을 해결하려면 프롬프트에 대한 **버전 관리**가 필요합니다. 소프트웨어 개발에서 코드를 Git 같은 도구로 관리하듯이, 프롬프트도 누가, 언제, 무엇을, 왜 변경했는지 기록하고 추적해야 합니다.

프롬프트 버전 관리가 소프트웨어 코드의 버전 관리와 다른 점이 하나 있습니다. 코드는 결정론적입니다. 같은 코드에 같은 입력을 넣으면 같은 출력이 나옵니다. 그래서 단위 테스트만으로 변경의 영향을 확인할 수 있습니다. 프롬프트는 다릅니다. 같은 프롬프트에 같은 입력을 넣어도 실행마다 출력이 달라질 수 있습니다. LLM의 출력은 토큰 확률 분포에서 샘플링되기 때문입니다. 이 **비결정론적** 특성 때문에, 프롬프트를 변경한 뒤에는 단위 테스트가 아닌 통계적 실험으로 효과를 검증해야 합니다. 이것이 바로 A/B 테스트가 필요한 이유입니다.

버전 관리가 가져다주는 이점을 구체적으로 살펴봅니다. 첫째, **투명성**입니다. 프롬프트가 언제 어떻게 바뀌었는지 이력이 남으므로, AI가 생성한 응답에 대한 감사 추적이 가능합니다. 둘째, **재현성**입니다. 과거 특정 시점의 프롬프트를 불러와 당시의 응답을 재현할 수 있습니다. 셋째, **안전한 실험**입니다. 새 버전을 일부 사용자에게만 먼저 적용하고, 문제가 없으면 전체로 확대할 수 있습니다. 넷째, **즉각 롤백**입니다. 변경 후 예상치 못한 문제가 생기면, 코드 재배포 없이 이전 버전으로 되돌릴 수 있습니다.

예를 들어 금융 서비스 챗봇을 운영한다고 가정합니다. 프롬프트를 수정하여 제품 추천 문구를 조금 바꿨는데, 수정된 프롬프트가 예상보다 공격적으로 제품을 권유하게 되었습니다. 컴플라이언스 규정 위반이 될 수 있는 상황입니다. 버전 관리가 없다면 수정 전 프롬프트를 찾아 코드를 다시 배포해야 합니다. 버전 관리가 있다면 이전 버전 번호를 지정하여 즉시 롤백할 수 있습니다. 문제가 발생한 버전 번호와 변경 이력이 남아 있으므로, 원인 분석과 감사 대응도 수월합니다.

이제 프롬프트 버전 관리를 지원하는 도구를 살펴봅니다. 먼저 **Langfuse**입니다. Langfuse는 LLM 애플리케이션의 관측 가능성과 프롬프트 관리를 제공하는 플랫폼입니다. 프롬프트를 Langfuse에 등록하면, 변경할 때마다 자동으로 새 버전 번호가 부여됩니다. 즉, 프롬프트를 수정하고 저장하면 버전 1, 버전 2, 버전 3 식으로 이력이 쌓입니다.

Langfuse의 핵심 기능은 **레이블 시스템**입니다. 레이블은 특정 버전에 붙이는 태그입니다. 환경을 구분하는 용도로 씁니다. 예를 들어 'staging' 레이블은 테스트 환경에서 쓸 버전에 붙입니다. 'production' 레이블은 실제 운영 환경에서 쓸 버전에 붙입니다. 애플리케이션 코드에서는 버전 번호 대신 레이블 이름으로 프롬프트를 불러옵니다. 레이블을 지정하지 않으면 'production' 레이블이 붙은 버전이 반환됩니다.

Langfuse에서 프롬프트를 불러오는 방법은 세 가지입니다. 첫째, 특정 버전 번호를 직접 지정하는 방법입니다. "버전 3을 가져와라"처럼 정확한 번호를 명시합니다. 디버깅이나 과거 출력 재현에 유용합니다. 둘째, 레이블 이름으로 불러오는 방법입니다. "production 레이블이 붙은 버전을 가져와라"처럼 요청합니다. 운영 환경에서 가장 일반적으로 쓰는 방식입니다. 셋째, 'latest' 레이블로 최신 버전을 자동으로 가져오는 방법입니다. 개발 중 빠른 테스트에 적합합니다.

롤백도 간단합니다. 'production' 레이블을 현재 버전에서 이전 버전으로 옮기면 됩니다. 애플리케이션 코드는 'production' 레이블로 프롬프트를 불러오므로, 레이블만 옮기면 코드 변경 없이 즉시 이전 버전이 적용됩니다.

```
Langfuse 프롬프트 버전 관리 흐름

버전 1  ---- [staging]
버전 2
버전 3  ---- [production]   <-- 현재 운영 중
버전 4  ---- [latest]

롤백 시: [production] 레이블을 버전 3에서 버전 2로 이동

버전 1  ---- [staging]
버전 2  ---- [production]   <-- 롤백 완료
버전 3
버전 4  ---- [latest]
```

위 다이어그램에서 'production' 레이블이 버전 3에 붙어 있으면, 운영 환경은 버전 3의 프롬프트를 사용합니다. 문제가 발생하여 'production' 레이블을 버전 2로 옮기면, 운영 환경은 즉시 버전 2로 전환됩니다. 코드 배포가 필요 없습니다.

Langfuse는 **프롬프트 비교(Prompt Diffs)** 기능도 제공합니다. 버전 간 변경 사항을 시간순으로 나란히 표시합니다. 어떤 단어가 추가되었고, 어떤 지시가 삭제되었는지 한눈에 확인할 수 있습니다. 팀이 프롬프트의 진화 과정을 추적하고, 문제 발생 시 어떤 변경이 원인인지 디버깅하는 데 도움을 줍니다.

엔터프라이즈 환경에서는 **보호된 레이블(Protected Labels)** 기능이 중요합니다. 이 기능을 켜면 Viewer나 Member 권한을 가진 팀원은 'production' 레이블을 변경할 수 없습니다. Admin이나 Owner만 운영 환경의 프롬프트 버전을 바꿀 수 있습니다. 실수로 운영 프롬프트가 변경되는 사고를 방지합니다.

다음으로 **LaunchDarkly**를 살펴봅니다. LaunchDarkly는 원래 소프트웨어의 **기능 플래그(feature flag)** 관리 플랫폼입니다. 기능 플래그란, 코드 배포 없이 특정 기능을 켜거나 끌 수 있는 스위치입니다. 새 기능을 코드에 포함시켜 배포하되, 플래그가 꺼져 있으면 사용자에게 노출되지 않습니다. 플래그를 켜면 그때부터 사용자에게 보입니다. 이 개념을 LLM 프롬프트에 적용한 것이 LaunchDarkly의 AI 플래그입니다.

LaunchDarkly의 AI 플래그는 두 종류가 있습니다. 첫 번째는 **AI 모델 플래그**입니다. 이 플래그는 어떤 LLM 모델을 사용할지, temperature나 max_token 같은 파라미터를 어떻게 설정할지를 제어합니다. JSON 형태로 모델 설정을 담고 있으며, 런타임에 값을 바꿀 수 있습니다. 코드를 다시 배포할 필요가 없습니다.

두 번째는 **AI 프롬프트 플래그**입니다. 이 플래그는 시스템 프롬프트와 사용자 프롬프트의 내용을 제어합니다. 2.1.1에서 다룬 시스템 프롬프트와 사용자 프롬프트의 구분을 그대로 활용합니다. 플래그 값을 바꾸면 모델에 전달되는 프롬프트 내용이 달라집니다.

LaunchDarkly의 가장 큰 강점은 **타게팅 규칙**입니다. 사용자를 세그먼트별로 나누어 서로 다른 프롬프트 버전을 할당할 수 있습니다. 예를 들어 교육 앱에서 초등학생에게는 쉬운 어휘를 쓰는 프롬프트를, 대학생에게는 전문 용어를 포함하는 프롬프트를 각각 적용할 수 있습니다. 이 과정에서 코드는 동일합니다. LaunchDarkly SDK가 사용자 속성을 보고 적절한 프롬프트 버전을 반환합니다.

카나리 배포도 지원합니다. 새 프롬프트 버전을 전체 사용자에게 한꺼번에 적용하는 대신, 소수의 사용자에게만 먼저 노출합니다. 문제가 없으면 비율을 점차 높이고, 문제가 발견되면 즉시 이전 버전으로 되돌립니다.

```
LaunchDarkly AI 플래그 작동 방식

+------------------+         +-----------------------+
| 애플리케이션     |         | LaunchDarkly 플래그   |
|                  |         |                       |
| SDK가 플래그 평가| ------> | 사용자 세그먼트 확인  |
|                  |         |                       |
|                  | <------ | 해당 프롬프트 반환    |
|                  |         +-----------------------+
| 반환된 프롬프트  |
| + 모델 설정으로  |         +-----------------------+
| LLM API 호출    | ------> | LLM API               |
+------------------+         +-----------------------+
```

위 다이어그램은 LaunchDarkly의 작동 방식을 요약합니다. 애플리케이션이 LaunchDarkly SDK를 통해 플래그를 평가합니다. SDK는 사용자의 세그먼트를 확인하고, 해당 세그먼트에 맞는 프롬프트와 모델 설정을 반환합니다. 애플리케이션은 반환된 값으로 LLM API를 호출합니다. 프롬프트나 모델을 바꾸고 싶으면 LaunchDarkly 대시보드에서 플래그 값만 변경하면 됩니다.

Langfuse와 LaunchDarkly의 접근 방식은 다릅니다. Langfuse는 프롬프트 자체의 버전을 관리합니다. 실행 이력과 성능 지표를 추적하는 데 강점이 있습니다. LaunchDarkly는 기능 플래그 인프라를 활용합니다. 사용자 세그먼트별 프롬프트 배포와 런타임 전환에 강점이 있습니다. 두 도구는 경쟁 관계라기보다, 서로 다른 계층의 문제를 해결합니다. 실무에서는 Langfuse로 프롬프트 버전과 성능을 관리하면서, LaunchDarkly로 사용자별 배포를 제어하는 방식으로 함께 쓸 수도 있습니다.

이제 프롬프트 **A/B 테스트 설계** 방법을 다룹니다. A/B 테스트란 두 가지 이상의 변형을 동시에 운영하고, 어느 쪽이 더 나은 성과를 내는지 데이터로 비교하는 실험 방법입니다. 웹 서비스에서 버튼 색상이나 문구를 바꿔 클릭률을 비교하는 것과 같은 원리입니다. 프롬프트 A/B 테스트에서는 프롬프트의 두 가지 버전을 동시에 운영하면서, 어느 버전이 더 나은 응답을 생성하는지 비교합니다.

프롬프트 A/B 테스트는 전통적인 소프트웨어 A/B 테스트와 한 가지 중요한 차이가 있습니다. LLM 출력의 **높은 분산**입니다. 같은 프롬프트로 같은 질문을 해도 매번 답이 달라질 수 있으므로, 적은 횟수의 시행으로는 결론을 내리기 어렵습니다. 따라서 전통 A/B 테스트보다 더 큰 표본 크기가 필요합니다.

프롬프트 A/B 테스트를 설계하는 과정을 5단계로 정리합니다.

```
프롬프트 A/B 테스트 5단계

+--------------------------------------------------+
| 1단계: 가설 수립                                  |
|   "프롬프트를 더 간결하게 하면                    |
|    관련성 점수가 10% 개선될 것이다"               |
+--------------------------------------------------+
                    |
                    v
+--------------------------------------------------+
| 2단계: 지표 정의                                  |
|   품질 지표 + 운영 지표 + 비즈니스 지표           |
+--------------------------------------------------+
                    |
                    v
+--------------------------------------------------+
| 3단계: 트래픽 분할 (카나리 배포)                  |
|   Control A: 기존 프롬프트 (90%)                  |
|   Variant B: 신규 프롬프트 (10%)                  |
+--------------------------------------------------+
                    |
                    v
+--------------------------------------------------+
| 4단계: 결과 모니터링                              |
|   통계적 유의성 확보 시까지 실시간 추적           |
+--------------------------------------------------+
                    |
                    v
+--------------------------------------------------+
| 5단계: 배포 또는 반복                             |
|   승자 확정 --> 점진적 확대                       |
|   불확실   --> 새 실험 설계                       |
+--------------------------------------------------+
```

1단계는 **가설 수립**입니다. 막연히 "더 좋은 프롬프트를 찾겠다"가 아니라, 구체적이고 측정 가능한 목표를 세웁니다. "프롬프트에서 불필요한 지시를 제거하면 관련성 점수가 10% 개선되고 응답 지연시간이 줄어들 것이다"처럼, 변경 내용과 기대 효과를 명시합니다.

2단계는 **지표 정의**입니다. 이 부분은 뒤에서 별도로 상세히 다룹니다. 단순히 "응답이 좋아졌는가"가 아니라, 품질, 운영, 비즈니스 세 축에서 다각도로 측정합니다.

3단계는 **트래픽 분할**입니다. 카나리 배포 방식을 적용합니다. 기존 프롬프트(Control A)에 전체 트래픽의 90%를 유지하고, 신규 프롬프트(Variant B)에 10%만 라우팅합니다. 신규 버전에 심각한 문제가 있어도 전체 사용자의 10%만 영향을 받으므로 리스크를 제한합니다.

이때 한 가지 반드시 지켜야 할 원칙이 있습니다. **동일 사용자 일관성**입니다. 한 사용자가 실험 기간 동안 같은 프롬프트 버전을 일관되게 경험해야 합니다. 요청할 때마다 A 버전과 B 버전이 번갈아 적용되면, 사용자 경험이 일관되지 않을 뿐 아니라 실험 결과의 공정성도 깨집니다. 사용자 ID를 기준으로 버전을 고정하는 것이 일반적입니다.

4단계는 **결과 모니터링**입니다. 실험이 시작되면 정의한 지표를 실시간으로 추적합니다. 충분한 표본이 쌓여 통계적 유의성이 확보될 때까지 실험을 유지합니다. 표본 크기가 부족한 상태에서 결론을 내리면 잘못된 판단을 할 수 있습니다.

통계적 유의성을 확보하려면 실험 전에 필요한 표본 크기를 미리 계산해야 합니다. 이를 **사전 파워 분석**이라 합니다. 먼저 감지하고 싶은 최소 효과 크기를 정합니다. 예를 들어 "관련성 점수가 최소 5% 이상 차이 나야 의미 있다"고 정하면, 이 5%가 **최소 감지 효과(MDE)**가 됩니다. MDE가 작을수록 미세한 차이까지 감지해야 하므로 더 많은 표본이 필요합니다. 표본 크기를 잘못 추정하면 성능 지표가 실제와 크게 달라질 수 있으므로, 실험 설계 시 신중하게 산정해야 합니다.

5단계는 **배포 또는 반복**입니다. 명확한 승자가 확정되면, Variant B의 트래픽 비율을 점진적으로 높여 전체 배포합니다. 결과가 불확실하면, 실험에서 얻은 통찰을 바탕으로 프롬프트를 다시 수정하고 새 실험을 설계합니다.

Langfuse에서 A/B 테스트를 구현하는 방법을 정리합니다. 앞서 다룬 레이블 시스템을 활용합니다. 프롬프트의 두 버전에 각각 'prod-a'와 'prod-b' 레이블을 붙입니다. 애플리케이션 코드에서 사용자 요청이 들어오면, 사용자 ID에 따라 'prod-a' 또는 'prod-b' 레이블의 프롬프트를 불러옵니다. Langfuse는 각 버전별 실행 이력과 성능 지표를 자동으로 추적합니다. 실험이 끝나면 Langfuse UI에서 두 버전의 지표를 나란히 비교 분석합니다.

이제 프롬프트 **성능 추적 지표**를 다룹니다. A/B 테스트의 2단계에서 언급한 지표 정의가 여기에 해당합니다. LLM 출력은 비결정론적이므로 전통 머신러닝의 정확도 한 가지만으로는 품질을 판단하기 어렵습니다. 프롬프트 성능은 품질, 운영, 비즈니스 세 축으로 나누어 추적합니다.

첫 번째 축은 **품질 지표**입니다. 모델이 생성한 응답의 내용적 품질을 측정합니다. 품질 지표를 측정하는 방법은 세 가지입니다.

자동화 평가는 사람이 개입하지 않고 프로그래밍으로 측정하는 방식입니다. **관련성(Relevance)**은 응답이 질문과 얼마나 관련 있는지를 봅니다. **충실성(Faithfulness)**은 응답이 주어진 근거 자료에 충실한지를 봅니다. 근거 없는 내용을 지어내는 환각(hallucination)을 감지하는 데 쓰입니다. **일관성(Coherence)**은 응답 내부의 논리적 흐름이 자연스러운지를 봅니다. **지시 준수(Instruction Following)**는 프롬프트에 명시한 제약 조건을 모델이 잘 따랐는지를 봅니다.

LLM-as-a-Judge는 AI 모델이 다른 AI 모델의 출력을 평가하는 방식입니다. 두 가지 방법이 있습니다. Pointwise 방식은 한 버전의 응답에 0점에서 5점 사이의 절대 점수를 부여합니다. Pairwise 방식은 두 버전의 응답을 나란히 놓고 어느 쪽이 더 나은지 비교합니다. Pairwise 방식의 결과를 승률(win rate)로 집계하면, A/B 테스트에서 어느 프롬프트 버전이 우세한지 직관적으로 파악할 수 있습니다.

인간 피드백은 실제 사용자나 평가자가 직접 판단하는 방식입니다. 명시적 신호로는 엄지 위/아래 평가나 만족도 설문이 있습니다. 암묵적 신호로는 사용자의 재시도 횟수나 조기 대화 종료 여부가 있습니다. 재시도가 잦다면 응답 품질에 불만이 있다는 간접 증거입니다.

두 번째 축은 **운영 지표**입니다. 응답의 내용이 아닌, 시스템의 기술적 성능을 측정합니다. **응답 지연시간**은 첫 토큰이 생성되기까지의 시간과 전체 응답 완료까지의 시간을 나누어 봅니다. **토큰 사용량 및 비용**은 요청당 얼마나 많은 토큰을 소비했는지, 그에 따른 금전적 비용은 얼마인지를 추적합니다. **오류 비율**은 타임아웃이나 API 오류가 얼마나 자주 발생하는지를 봅니다. 프롬프트를 더 간결하게 수정하면 토큰 수가 줄어 비용과 지연시간이 개선될 수 있습니다. 반대로 프롬프트를 더 상세하게 만들면 품질은 올라가지만 비용이 증가할 수 있습니다. A/B 테스트에서 품질 지표와 운영 지표를 함께 보는 이유가 여기에 있습니다.

세 번째 축은 **비즈니스 지표**입니다. 최종적으로 프롬프트 변경이 비즈니스 성과에 미치는 영향을 측정합니다. **채택률(Adoption Rate)**은 AI 기능을 실제로 사용하는 사용자 비율입니다. **태스크 완료율**은 사용자가 AI 응답을 바탕으로 원하는 작업을 완수하는 비율입니다. 챗봇의 경우 **대화 길이**나 **재사용률**도 중요한 지표입니다. 대화가 지나치게 길어지면 사용자가 원하는 답을 빨리 얻지 못하고 있다는 신호일 수 있습니다.

```
프롬프트 성능 추적 지표 3축

+-------------------+-------------------+-------------------+
|    품질 지표      |    운영 지표      |  비즈니스 지표    |
+-------------------+-------------------+-------------------+
| 관련성            | 응답 지연시간     | 채택률            |
| 충실성            | 토큰 사용량/비용  | 태스크 완료율     |
| 일관성            | 오류/타임아웃     | 대화 길이         |
| 지시 준수         |   비율            | 재사용률          |
| LLM-as-a-Judge    |                   | 세션당 쿼리 수    |
| 인간 피드백       |                   |                   |
+-------------------+-------------------+-------------------+
```

Langfuse에서는 이 세 축의 지표를 버전별로 자동 집계합니다. 응답 지연시간, 토큰 사용량, 요청당 비용을 기본으로 추적하며, 품질 평가 점수나 커스텀 지표도 함께 기록할 수 있습니다. Dataset Runs 기능을 활용하면, 동일한 테스트 입력에 대해 서로 다른 프롬프트 버전의 결과를 테이블 형태로 나란히 비교할 수 있습니다.

정리하면, 프롬프트는 코드와 달리 비결정론적이므로 버전 관리와 통계적 실험이 함께 필요합니다. Langfuse는 레이블 기반의 버전 관리와 성능 추적을 제공합니다. LaunchDarkly는 기능 플래그를 활용한 사용자 세그먼트별 배포와 런타임 전환을 제공합니다. A/B 테스트는 가설 수립, 지표 정의, 트래픽 분할, 결과 모니터링, 배포 판단의 5단계로 설계합니다. 품질, 운영, 비즈니스 세 축의 지표를 종합적으로 비교하여 판단합니다.

다음 단원인 2.1.3에서는 에이전트 시스템 프롬프트 설계 원칙을 다룹니다. 단일 프롬프트를 넘어, 에이전트가 자율적으로 동작할 때 시스템 프롬프트를 어떻게 구성해야 하는지를 살펴봅니다.

이 단원을 마치면 프롬프트 버전 관리의 필요성을 이해할 수 있습니다. Langfuse와 LaunchDarkly를 활용한 A/B 테스트 방법도 설명할 수 있습니다.
