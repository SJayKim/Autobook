# 1.1.2 AI 에이전트의 3대 구성요소 — LLM, 도구, 메모리

1.1.1에서 AI 에이전트가 생성형 AI와 다른 이유를 살펴보았습니다. 생성형 AI는 프롬프트를 받아 텍스트를 생성하는 데 그치지만, AI 에이전트는 스스로 계획을 세우고, 외부 도구를 사용하며, 맥락을 기억합니다. 그런데 이런 능력은 어디서 오는 것일까요. AI 에이전트 내부를 열어 보면 세 가지 구성요소가 맞물려 돌아갑니다. 추론을 담당하는 **LLM**, 실세계와 접점을 만드는 **도구(Tool)**, 맥락을 유지하는 **메모리(Memory)**입니다. 이 단원에서는 각 요소가 무엇이고, 왜 필요하며, 서로 어떻게 연결되는지를 단계별로 풀어 봅니다.

LLM부터 시작합니다. LLM은 Large Language Model의 줄임말로, 대규모 텍스트 데이터를 학습한 언어 모델입니다. 1.1.1에서 생성형 AI의 핵심 기술로 소개한 바 있습니다. 그런데 AI 에이전트에서 LLM이 하는 일은 단순히 글을 생성하는 것과 다릅니다.

AI 에이전트 안에서 LLM은 **추론 엔진(reasoning engine)** 역할을 합니다. 추론 엔진이란, 주어진 정보를 해석하고 다음에 무엇을 해야 할지 판단하는 핵심 장치를 뜻합니다. 구체적으로 LLM은 네 가지 기능을 수행합니다. 첫째, 사용자가 입력한 자연어를 이해합니다. "다음 주 서울 날씨에 맞는 출장 일정을 짜 줘"라는 요청에서 '날씨 확인'과 '일정 수립'이라는 두 과제를 분리해 내는 것이 자연어 이해입니다. 둘째, 과제를 해결하기 위한 단계를 추론합니다. 날씨부터 확인한 뒤 그 결과를 일정에 반영해야 한다는 순서를 스스로 도출합니다. 셋째, 추론 결과를 바탕으로 실행 계획을 세웁니다. 어떤 도구를 어떤 순서로 호출할지 결정합니다. 넷째, 도구가 돌려준 결과를 해석하여 사용자에게 전달할 최종 응답을 생성합니다.

이 네 기능이 왜 중요한지 한 가지 장면으로 정리하면 이렇습니다. 사용자가 "서울 날씨 알려줘"라고 입력합니다. LLM이 없다면 시스템은 이 문장이 날씨 API를 호출하라는 뜻인지, 일반 상식을 물어보는 것인지 구분하지 못합니다. LLM이 있기 때문에 '서울'이라는 지역과 '날씨'라는 의도를 파악하고, 날씨 조회 도구를 호출하기로 결정하며, 도구가 돌려준 JSON 데이터를 사람이 읽을 수 있는 문장으로 바꿔 줍니다. LLM은 에이전트의 두뇌에 해당합니다. 도구와 메모리가 아무리 잘 갖춰져 있어도 LLM이 없으면 에이전트는 사용자 의도를 해석하거나 다음 행동을 결정할 수 없습니다.

다음은 **도구(Tool)**입니다. LLM은 본질적으로 텍스트를 처리하는 모델입니다. 아무리 추론 능력이 뛰어나도 인터넷에 접속하거나, 파일을 저장하거나, 데이터베이스를 조회하는 행동은 LLM 혼자 할 수 없습니다. 도구는 이 한계를 메우는 확장 장치입니다. 검색 엔진, 코드 실행기, 이메일 발송 API, 일정 관리 시스템 등 에이전트가 실세계에서 수행하는 모든 행동은 도구를 통해 이루어집니다.

도구를 유형별로 나누면 다섯 가지로 분류할 수 있습니다. 정보 검색 도구(검색 엔진, 지식베이스 조회), 계산 도구(코드 인터프리터, 통계 분석기), 생성 도구(이미지 생성, 문서 생성), 통신 도구(이메일 전송, 일정 관리), 전문 API(날씨 조회, 금융 데이터, CRM 시스템)입니다.

그렇다면 LLM은 이 도구들을 어떻게 사용할까요. 여기서 **함수 호출(Function Calling)**이라는 메커니즘이 등장합니다. 함수 호출은 LLM이 "이 함수를 이 인수로 실행해 달라"고 지시를 내리는 방식입니다. 중요한 점은, LLM이 함수를 직접 실행하는 것이 아니라는 것입니다. LLM은 실행 지시만 출력하고, 실제 실행은 외부 시스템이 담당합니다.

함수 호출은 세 단계로 동작합니다.

```
단계 1: 도구 정의
  개발자가 함수 이름, 설명, 파라미터 스키마를 JSON으로 정의하여
  LLM에 전달합니다.

  예) get_weather(city: string, date: string)
      -> "지정한 도시의 해당 날짜 날씨를 조회합니다"

단계 2: 호출 결정
  LLM이 사용자 요청을 분석하다가 외부 정보가 필요하다고
  판단하면, 어떤 함수를 어떤 인수로 호출할지를
  JSON 형태로 출력합니다.

  예) {"function": "get_weather", "arguments": {"city": "서울", "date": "2026-04-06"}}

단계 3: 실행 및 결과 주입
  외부 시스템이 해당 함수를 실제로 실행하고,
  결과를 LLM의 컨텍스트에 다시 넣어 줍니다.

  예) {"temperature": 14, "condition": "맑음"} -> LLM이 이 결과를 해석
```

첫 번째 단계에서 개발자는 에이전트가 사용할 수 있는 도구 목록을 미리 정의합니다. 함수 이름, 함수가 하는 일에 대한 설명, 그리고 어떤 파라미터를 받는지를 JSON 스키마로 작성합니다. 두 번째 단계에서 LLM은 사용자 요청을 처리하면서 "이 작업을 완료하려면 외부 정보가 필요하다"고 판단합니다. 그러면 함수 이름과 인수를 JSON으로 출력합니다. 이 출력은 실행 결과가 아니라 "이것을 실행해 달라"는 지시입니다. 세 번째 단계에서 외부 시스템이 해당 함수를 실제로 호출하고, 그 결과를 다시 LLM에게 전달합니다. LLM은 이 결과를 해석하여 사용자에게 자연어로 응답합니다.

2023년 6월 OpenAI가 이 함수 호출 기능을 API에 네이티브로 탑재하면서, 개발자가 어떤 API든 에이전트의 도구로 연결할 수 있는 환경이 열렸습니다. 하지만 연결해야 할 도구가 많아지면 문제가 생깁니다. AI 애플리케이션 N개와 데이터 소스 M개를 연결하려면 N x M개의 개별 커넥터를 만들어야 합니다. 2024년 11월 Anthropic이 공개한 **MCP(Model Context Protocol)**는 이 문제를 해결하기 위한 표준 인터페이스입니다. USB-C 포트가 어떤 기기든 하나의 규격으로 연결하듯, MCP는 어떤 AI 애플리케이션이든 어떤 데이터 소스든 하나의 프로토콜로 연결할 수 있게 합니다. 도구 통합의 비용이 N x M에서 N + M으로 줄어드는 것입니다.

도구를 설계할 때 고려해야 할 과제도 있습니다. 어떤 도구를 선택할지 판단하는 로직, 각 도구에 맞는 파라미터 형식 준비, 도구가 실패했을 때 오류를 처리하는 방법, 도구 결과를 추론 과정에 통합하는 방식, 이 네 가지가 도구 설계의 핵심 과제입니다. 기획자 입장에서는 에이전트에 어떤 도구를 부여하느냐가 곧 에이전트의 능력 범위를 결정한다는 점을 기억하면 됩니다.

이제 세 번째 구성요소인 **메모리(Memory)**를 살펴봅니다. LLM 자체는 기억하지 못합니다. 하나의 대화가 끝나면 이전에 주고받은 내용이 사라집니다. 1.1.1에서 생성형 AI의 한계로 '무상태(Stateless)'를 꼽았던 이유가 바로 이것입니다. 메모리는 이 한계를 보완하여 에이전트가 맥락을 유지하고 과거 경험을 활용할 수 있게 하는 별도의 설계 컴포넌트입니다.

메모리는 크게 **단기 메모리(Short-Term Memory)**와 **장기 메모리(Long-Term Memory)**로 나뉩니다.

단기 메모리부터 설명합니다. 단기 메모리는 현재 진행 중인 세션 안에서 최근 입력과 출력을 담아 두는 장치입니다. 가장 단순한 형태는 LLM의 컨텍스트 창(context window)에 최근 대화 이력을 그대로 넣는 것입니다. 컨텍스트 창이란 LLM이 한 번에 처리할 수 있는 텍스트의 최대 길이를 말합니다. 예를 들어, 사용자가 "서울 날씨 알려줘"라고 물은 뒤 "그러면 부산은?"이라고 이어서 물었을 때, 단기 메모리에 이전 대화가 남아 있으므로 에이전트는 '부산 날씨'를 묻는다는 맥락을 파악합니다. 단기 메모리는 하나의 작업을 완료하는 데 충분하지만, 세션이 끝나면 데이터가 사라진다는 한계가 있습니다.

장기 메모리는 세션을 넘어서 정보를 영속적으로 저장하는 외부 저장소입니다. 데이터베이스, 지식 그래프, 벡터 임베딩 저장소 같은 시스템이 장기 메모리 역할을 합니다. 1.1.1에서 에이전틱 AI의 특성으로 '지속적 메모리(Persistent Memory)'를 언급했는데, 바로 이 장기 메모리가 그 실체입니다.

장기 메모리는 저장하는 내용의 성격에 따라 다시 세 유형으로 나뉩니다. 첫째, **에피소딕 메모리(Episodic Memory)**는 과거의 특정 사건이나 상호작용을 기록합니다. "지난주 사용자가 A 클라이언트 미팅 일정을 조정한 기록"처럼, 구체적인 경험을 저장하여 비슷한 상황에서 참고합니다. 둘째, **시맨틱 메모리(Semantic Memory)**는 사실, 정의, 규칙 같은 구조화된 지식을 저장합니다. "A 클라이언트의 선호 미팅 시간은 오후 2시"처럼, 반복적으로 참조할 일반 지식입니다. 셋째, **절차적 메모리(Procedural Memory)**는 자주 수행하는 행동 순서를 저장합니다. "미팅 일정 조정 시 캘린더 확인 -> 참석자 가용 시간 조회 -> 초대장 발송" 같은 작업 패턴을 기억해 두면, 매번 처음부터 추론하지 않고 효율적으로 실행할 수 있습니다.

메모리 설계에서 핵심적인 트레이드오프가 있습니다. 저장하는 정보가 많을수록 검색에 시간이 걸려 응답 속도가 느려집니다. 따라서 메모리 최적화는 "많이 저장하는 것"이 아니라 "필요한 정보를 빠르게 꺼내는 것"이 핵심입니다. 기획자가 에이전트의 메모리를 설계할 때는 "이 에이전트가 무엇을 기억해야 하고, 얼마나 오래 유지해야 하며, 얼마나 빨리 꺼내야 하는가"를 기준으로 판단합니다.

지금까지 LLM, 도구, 메모리를 각각 살펴보았습니다. 이 세 요소는 독립적으로 동작하지 않습니다. 하나의 반복 루프 안에서 서로 맞물려 작동합니다. 이것을 **에이전트 루프(Agent Loop)**라고 합니다.

에이전트 루프의 기본 사이클은 세 단계입니다. **인식(Perceive)**, **추론(Reason)**, **행동(Act)**입니다.

```
에이전트 루프: Perceive -> Reason -> Act

  +---> 인식(Perceive) --->  추론(Reason) ---> 행동(Act) ---+
  |     사용자 입력,          LLM이 메모리와      도구 호출     |
  |     도구 결과,            도구 목록을 참고해    또는          |
  |     환경 신호를 수집      다음 행동을 결정     최종 응답 생성 |
  |                                                            |
  +------------------<--- 결과 피드백 ---<---------------------+
```

인식 단계에서 에이전트는 입력을 수집합니다. 사용자의 새 요청일 수도 있고, 이전 단계에서 호출한 도구의 실행 결과일 수도 있습니다. 추론 단계에서 LLM이 수집된 입력을 메모리에 저장된 맥락, 사용 가능한 도구 목록과 함께 분석합니다. 목표를 달성했는지 평가하고, 아직 완료되지 않았다면 다음에 어떤 행동을 취할지 결정합니다. 행동 단계에서 에이전트는 결정된 행동을 실행합니다. 도구를 호출하거나, 사용자에게 최종 응답을 전달하거나, 추가 정보를 요청합니다. 도구를 호출한 경우 그 결과가 다시 인식 단계로 돌아가며, 루프가 반복됩니다.

이 루프를 좀 더 상세하게 풀면 여섯 단계로 나눌 수 있습니다. (1) LLM이 메모리의 맥락을 포함하여 사용자 입력을 처리합니다. (2) 추론을 통해 계획을 수립하고 적절한 도구를 결정합니다. (3) 선택한 도구를 LLM이 생성한 파라미터로 실행합니다. (4) 도구 결과를 추론 과정에 통합합니다. (5) 사용자에게 전달할 응답을 생성합니다. (6) 이번 상호작용을 메모리에 업데이트합니다.

에이전트 루프의 핵심 특성은 '반복'에 있습니다. 단일 패스, 즉 한 번 처리하고 끝나는 구조가 아닙니다. 첫 번째 시도가 실패하면 에이전트가 스스로 다른 전략을 시도할 수 있습니다. 날씨 API가 응답하지 않으면 다른 날씨 서비스를 호출하거나, 사용자에게 "날씨 정보를 가져오지 못했습니다. 다른 방법을 시도할까요?"라고 피드백을 요청할 수 있습니다. 이 반복 구조가 에이전트를 단순 챗봇과 구별하는 자율성의 근거입니다.

에이전트 루프의 가장 대표적인 구현이 **ReAct 패턴**입니다. ReAct는 Reason과 Act를 합친 이름으로, 2022년 10월 구글 연구팀이 발표했습니다. 1.1.1에서 AI 에이전트의 기술적 이정표로 언급한 바 있습니다.

ReAct 패턴은 **Thought(생각)**, **Action(행동)**, **Observation(관찰)** 세 단계를 반복합니다. 구체적인 예로 살펴봅니다. 사용자가 "서울과 부산의 내일 날씨를 비교해 줘"라고 요청했다고 가정합니다.

```
[Thought 1] 서울과 부산 두 도시의 날씨를 비교해야 한다.
            먼저 서울 날씨부터 조회한다.

[Action 1]  get_weather(city="서울", date="2026-03-31")

[Observation 1] {"temperature": 12, "condition": "흐림"}

[Thought 2] 서울 날씨를 확인했다. 이제 부산 날씨를 조회한다.

[Action 2]  get_weather(city="부산", date="2026-03-31")

[Observation 2] {"temperature": 16, "condition": "맑음"}

[Thought 3] 두 도시 날씨를 모두 확인했다.
            서울은 흐리고 12도, 부산은 맑고 16도다.
            비교 결과를 정리해서 응답한다.

[Action 3]  최종 응답 생성
```

Thought 단계에서 LLM은 현재 상황을 분석하고 다음에 무엇을 해야 할지 판단합니다. Action 단계에서 판단에 따라 도구를 호출합니다. Observation 단계에서 도구가 돌려준 결과를 확인합니다. 이 결과가 다음 Thought의 입력이 됩니다. 목표가 달성될 때까지 이 사이클이 반복됩니다.

ReAct 패턴이 현대 AI 에이전트의 사실상 표준 기반이 된 데는 두 가지 이유가 있습니다. 첫째, 추론과 행동을 분리하여 추론 과정이 명시적으로 기록됩니다. LLM이 왜 그런 판단을 내렸는지 Thought 단계에 남기 때문에, 잘못된 추론을 추적하기 쉽고 할루시네이션(사실과 다른 내용을 생성하는 현상)이 줄어듭니다. 둘째, 자기 평가와 도구 사용을 하나의 흐름 안에서 결합합니다. 에이전트가 스스로 "지금까지 무엇을 했고, 다음에 무엇이 필요한가"를 판단하면서 동시에 외부 도구를 활용합니다.

ReAct에도 한계는 있습니다. 중간에 추론이 막히거나 도구가 예상치 못한 오류를 반환하면 에이전트가 스스로 빠져나오지 못하고 인간 피드백이 필요할 수 있습니다. 이 한계를 보완하기 위해 여러 파생 패턴이 등장했습니다. RAISE는 ReAct에 단기/장기 메모리를 추가하여 과거 경험을 참고하게 하고, Reflexion은 별도의 LLM 평가자가 실행 결과를 점검하여 성공률을 높입니다. 이런 패턴들은 이후 단원에서 더 자세히 다룹니다.

이제 세 구성요소가 에이전트 루프 안에서 어떻게 맞물리는지 전체 흐름을 정리합니다.

```
사용자 요청
     |
     v
+-----------------------------+
|  인식(Perceive)              |
|  사용자 입력 + 메모리 맥락   | <--- 메모리에서 관련 정보 검색
+-----------------------------+
     |
     v
+-----------------------------+
|  추론(Reason)                |
|  LLM이 상황 분석,            |
|  실행 계획 수립              | <--- LLM (추론 엔진)
|  어떤 도구를 쓸지 결정       |
+-----------------------------+
     |
     v
+-----------------------------+
|  행동(Act)                   |
|  도구 호출, 결과 수신        | <--- 도구 (외부 시스템)
+-----------------------------+
     |
     v
  목표 달성? --아니오--> 결과를 메모리에 저장, 인식 단계로 복귀
     |
    예
     |
     v
  최종 응답 + 메모리 업데이트
```

사용자 요청이 들어오면 에이전트는 메모리에서 관련 맥락을 꺼내 입력과 합칩니다. LLM이 이 정보를 바탕으로 추론하고 계획을 세웁니다. 계획에 따라 도구를 호출하고, 결과를 확인합니다. 목표가 달성되지 않았으면 결과를 메모리에 저장한 뒤 다시 인식 단계로 돌아갑니다. 목표가 달성되면 최종 응답을 생성하고 이번 상호작용을 메모리에 기록합니다. LLM은 매 단계에서 판단을 내리고, 도구는 실행을 담당하며, 메모리는 맥락을 공급합니다. 셋 중 하나라도 빠지면 에이전트는 제대로 동작하지 않습니다.

정리하면, AI 에이전트는 LLM, 도구, 메모리라는 세 구성요소가 에이전트 루프 안에서 맞물려 동작하는 시스템입니다. LLM은 사용자 의도를 해석하고 다음 행동을 결정하는 추론 엔진이며, 도구는 함수 호출 메커니즘을 통해 에이전트가 실세계와 상호작용하게 하고, 메모리는 단기와 장기로 나뉘어 맥락 유지와 경험 축적을 가능하게 합니다. 이 세 요소가 인식-추론-행동의 반복 루프를 형성하며, ReAct 패턴은 이 루프를 Thought-Action-Observation 사이클로 구현한 대표적인 방식입니다.

다음 단원인 1.1.3에서는 확률적 출력과 비결정론성을 다룹니다. LLM이 매번 같은 입력에 다른 결과를 내놓는 이유와, 기획자가 이 특성을 어떻게 고려해야 하는지를 살펴봅니다.

이 단원을 마치면 AI 에이전트를 구성하는 LLM, 도구, 메모리 세 요소의 역할과 상호작용을 설명할 수 있습니다.
