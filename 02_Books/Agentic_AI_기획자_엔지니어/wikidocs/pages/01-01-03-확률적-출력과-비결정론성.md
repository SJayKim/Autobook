# 1.1.3 확률적 출력과 비결정론성 — 기획자가 알아야 할 동작 원리

1.1.2에서 AI 에이전트의 핵심 두뇌인 LLM이 사용자 의도를 해석하고 다음 행동을 결정한다고 설명했습니다. 그런데 같은 질문을 같은 에이전트에게 두 번 던지면 어떤 일이 벌어질까요. 전통 소프트웨어라면 결과가 같아야 정상입니다. 계산기에 3 + 5를 입력하면 언제나 8이 나옵니다. 하지만 LLM 기반 에이전트는 다릅니다. 같은 프롬프트를 보내도 문장 구성이 달라지고, 단어 선택이 바뀌며, 때로는 사실과 다른 내용을 생성하기도 합니다. 이 단원에서는 이런 현상이 왜 발생하는지 동작 원리를 풀어보고, 기획자가 제품을 설계할 때 무엇을 달리 해야 하는지를 다룹니다.

먼저 두 종류의 시스템을 구분합니다. **결정론적 시스템(deterministic system)**은 동일한 입력이 주어지면 항상 동일한 출력을 반환하는 시스템입니다. 정렬 알고리즘에 같은 숫자 목록을 넣으면 매번 같은 순서가 나옵니다. 데이터베이스에 같은 조건으로 조회하면 같은 행이 돌아옵니다. 전통 소프트웨어의 대부분이 여기에 해당합니다.

반면 **확률적 시스템(stochastic system)**은 동일한 입력에도 출력이 매번 달라질 수 있는 시스템입니다. 내부에 무작위성이 포함되어 있기 때문입니다. LLM 기반 AI가 바로 이 범주에 속합니다. 1.1.2에서 살펴본 에이전트의 두뇌인 LLM은 확률적 시스템이므로, 에이전트 전체의 행동 역시 확률적이 됩니다. 에이전트가 도구를 호출하는 순서, 사용자에게 전달하는 문장, 심지어 어떤 도구를 선택할지까지 매 실행마다 달라질 수 있습니다.

이 차이가 실무에서 왜 중요한지 한 가지 장면으로 설명합니다. 결정론적 시스템은 테스트 케이스 하나로 동작을 검증할 수 있습니다. 입력 A를 넣었을 때 출력 B가 나오면 통과, 아니면 실패입니다. 하지만 확률적 시스템은 같은 입력에 출력이 달라지므로 단일 테스트로 품질을 보장할 수 없습니다. 대신 여러 번 실행한 뒤 성공률이나 평균 품질 분포를 통계적으로 평가해야 합니다.

그렇다면 LLM은 왜 확률적으로 동작할까요. 이를 이해하려면 LLM이 텍스트를 생성하는 메커니즘을 들여다봐야 합니다. LLM은 글자나 문장을 한꺼번에 만들지 않습니다. **토큰(token)**이라는 작은 단위를 하나씩 이어 붙여서 텍스트를 생성합니다.

토큰은 단어 또는 단어 조각입니다. 영문 기준으로 한 토큰은 대략 네 글자에 해당합니다. 한글은 한두 글자가 한 토큰이 되는 경우가 많습니다. 예를 들어 "안녕하세요"는 여러 토큰으로 쪼개질 수 있습니다. **토큰화(tokenization)**란 입력 텍스트를 이런 토큰 단위로 분해하는 과정을 말합니다.

LLM의 텍스트 생성은 "다음 토큰 예측"의 반복입니다. 모델은 지금까지 생성된 토큰열을 보고, 다음에 올 수 있는 모든 토큰 각각에 확률을 부여합니다. 이 확률을 모아 놓은 것이 **확률 분포(probability distribution)**입니다. 모델이 알고 있는 전체 어휘(vocabulary)에 대해 "이 토큰이 다음에 올 가능성이 몇 퍼센트인가"를 계산하는 것입니다.

구체적인 과정을 단계별로 보겠습니다.

```
1단계: 입력 텍스트를 토큰으로 분해한다.
        "서울의 내일 날씨는" -> [서울, 의, 내일, 날씨, 는]

2단계: 모델이 토큰열을 처리하고,
        다음 토큰 후보마다 확률을 계산한다.
        "맑겠" -> 35%
        "흐리" -> 25%
        "좋"   -> 15%
        "비"   -> 10%
        (나머지 후보들) -> 15%

3단계: 확률 분포에서 토큰 하나를 뽑는다(샘플링).
        -> 이번에는 "맑겠"이 선택됨

4단계: 선택된 토큰을 이어 붙이고,
        2~3단계를 반복한다.
        "서울의 내일 날씨는 맑겠"
        -> 다시 다음 토큰의 확률 분포를 계산...
```

3단계에서 확률이 가장 높은 "맑겠"이 항상 선택되는 것은 아닙니다. 확률 분포에서 무작위로 뽑기 때문에 "흐리"나 "좋"이 선택될 수도 있습니다. 바로 이 샘플링 과정이 LLM 출력을 확률적으로 만드는 핵심입니다. 같은 입력을 줘도 매번 다른 토큰이 뽑히고, 그 결과 다른 문장이 만들어집니다.

토큰은 비용과도 직결됩니다. LLM API는 입력 토큰과 출력 토큰 수를 기준으로 과금합니다. 기획자가 에이전트 기능을 설계할 때, 프롬프트 길이와 예상 응답 길이를 토큰 단위로 추정하면 운영 비용을 가늠할 수 있습니다.

여기서 한 가지 의문이 생깁니다. 확률 분포에서 토큰을 뽑는 방식을 조절할 수는 없을까요. 있습니다. 그 역할을 하는 것이 **Temperature** 파라미터입니다.

Temperature는 확률 분포의 "날카로움"을 조절하는 설정값입니다. 온도라는 이름은 물리학의 열역학에서 빌려온 비유입니다. 온도가 낮으면 분자의 움직임이 줄어 안정적이고, 온도가 높으면 분자가 활발하게 움직여 예측이 어려워집니다. LLM에서도 마찬가지입니다.

Temperature를 0에 가깝게 설정하면 확률 분포가 뾰족해집니다. 가장 높은 확률을 가진 토큰이 거의 항상 선택됩니다. 결과적으로 출력이 매번 비슷해지고 예측 가능해집니다. 거의 결정론적으로 동작하는 셈입니다.

Temperature를 높이면 확률 분포가 평평해집니다. 원래 확률이 낮았던 토큰도 선택될 가능성이 올라갑니다. 결과적으로 다양하고 예상치 못한 표현이 등장합니다. 다만 사실과 동떨어진 내용이 나올 위험도 함께 올라갑니다.

이 차이를 표로 정리하면 다음과 같습니다.

```
Temperature 설정과 출력 특성

+------------------+-------------------+-------------------+
| 설정 범위        | 출력 특성          | 적합한 태스크      |
+------------------+-------------------+-------------------+
| 0 ~ 0.3 (낮음)  | 일관적, 예측 가능  | 사실 정보 추출     |
|                  | 거의 같은 답 반복  | 고객센터 응답      |
|                  |                   | 데이터 분류        |
+------------------+-------------------+-------------------+
| 0.4 ~ 0.6 (중간)| 적당한 변화        | 일반 대화          |
|                  | 자연스러운 다양성  | 요약               |
+------------------+-------------------+-------------------+
| 0.7 ~ 1.0 (높음)| 창의적, 예측 어려움| 브레인스토밍       |
|                  | 새로운 표현 등장   | 마케팅 카피 생성   |
|                  |                   | 창작 글쓰기        |
+------------------+-------------------+-------------------+
```

기획자가 알아야 할 점은, Temperature 값이 곧 제품의 성격을 결정한다는 것입니다. 고객센터 자동 응답처럼 정확성이 중요한 기능이라면 Temperature를 낮게 잡아야 합니다. 반대로 아이디어 생성이나 카피라이팅처럼 다양성이 가치인 기능이라면 높게 설정합니다. PRD에 "창의성 수준"이나 "일관성 우선/다양성 우선" 방향을 명시하면 엔지니어링 팀과의 커뮤니케이션이 훨씬 명확해집니다. Temperature 외에도 Top-p, Top-k 같은 샘플링 파라미터가 있지만, 핵심 원리는 동일합니다. 확률 분포에서 토큰을 뽑는 범위를 조절하여 출력의 다양성을 제어하는 것입니다.

이제 확률적 출력이 만들어내는 가장 심각한 부작용을 다룹니다. **환각(Hallucination)**입니다. 환각이란 LLM이 문법적으로 유창하고 그럴듯하게 들리지만 사실적으로 틀린 내용을 자신감 있게 생성하는 현상입니다.

환각이 발생하는 이유는 LLM의 동작 원리 자체에 있습니다. 앞에서 설명한 것처럼 LLM은 "무엇이 사실인가"를 판단하지 않습니다. "다음에 올 확률이 높은 토큰이 무엇인가"만 계산합니다. 확률 분포상 자연스러운 토큰 조합이 현실의 사실과 일치하지 않을 때 환각이 발생합니다. 모델은 "모른다"는 개념 자체가 없으므로, 확신에 찬 어조로 틀린 정보를 내놓습니다.

환각이 왜 위험한지 구체적으로 봅니다. 금융 에이전트가 존재하지 않는 규정을 인용하여 대출 심사를 거부할 수 있습니다. 의료 에이전트가 근거 없는 복용량을 안내할 수 있습니다. 이런 오류는 단순한 불편이 아니라 실질적 피해로 이어집니다.

환각을 완전히 없앨 수는 없지만 줄이는 방법은 여러 가지가 있습니다.

첫째, **RAG(Retrieval-Augmented Generation)**입니다. 외부 문서에서 관련 정보를 먼저 검색한 뒤, 그 정보를 바탕으로 답변을 생성하도록 하는 방식입니다. 모델이 자체 기억에만 의존하지 않고 실제 문서를 근거로 삼으므로 환각이 줄어듭니다.

둘째, Temperature를 낮추는 것입니다. 확률 분포를 뾰족하게 만들면 엉뚱한 토큰이 선택될 가능성이 줄어듭니다.

셋째, 프롬프트에 "근거가 없으면 모른다고 답하라"는 지시를 추가하는 것입니다.

넷째, 멀티에이전트 검증입니다. 여러 에이전트가 서로의 출력을 교차 검증하여 사실 오류를 걸러냅니다.

다섯째, 파인튜닝입니다. 특정 도메인 데이터로 모델을 추가 학습시켜 해당 영역의 환각 패턴을 교정합니다.

기획자는 기능을 설계할 때 "이 기능에서 환각이 발생하면 어떤 결과가 오는가"를 먼저 판단해야 합니다. "10번 중 1번 틀려도 괜찮은 기능"과 "절대 틀리면 안 되는 기능"은 설계 방식이 완전히 다릅니다. 의료, 금융, 법률처럼 오류 허용 범위가 극히 좁은 도메인에서는 RAG, 파인튜닝, 인간 검토 중 어떤 보완 전략을 조합할지 PRD에 명시해야 합니다.

지금까지 살펴본 확률적 출력과 환각은 PRD 작성 방식과 KPI 설계에 근본적인 변화를 요구합니다.

전통 소프트웨어의 PRD는 "버튼을 클릭하면 A 화면으로 이동한다"처럼 결정론적 동작을 명세합니다. 입력과 출력이 1:1로 대응하므로, 명세대로 동작하는지만 확인하면 됩니다. 하지만 AI 기능의 PRD는 같은 입력에도 출력이 달라질 수 있다는 전제에서 출발해야 합니다. 따라서 정확한 출력값 대신 출력의 "허용 범위"와 품질 평가 방법을 함께 정의해야 합니다.

AI PRD에 추가해야 하는 항목은 크게 네 가지입니다. 첫째, LLM 출력의 기대 사양입니다. 응답 길이, 톤, 형식, 다뤄야 할 영역과 제외할 영역을 구체적으로 기술합니다. 둘째, 피드백 메커니즘입니다. 사용자가 출력의 품질을 평가할 수 있는 장치(좋아요/싫어요, 상세 피드백 폼)를 설계합니다. 셋째, 품질 평가 방식입니다. 자동 평가와 수동 평가를 병행하는 계획을 포함합니다. 넷째, 책임 AI 항목입니다. AI가 생성한 결과임을 사용자에게 공개할지, 긴급 상황에서 AI 기능을 즉시 중단할 수 있는 킬 스위치(kill switch)가 있는지를 명시합니다.

KPI 역시 달라집니다. 전통 소프트웨어에서는 클릭률, 전환율 같은 결정론적 지표가 핵심이었습니다. 하지만 에이전트 AI에서는 이런 지표만으로 전체 상황을 파악할 수 없습니다. 대신 다음과 같은 지표를 측정해야 합니다.

```
에이전트 AI의 KPI 구성

+----------------------+--------------------------------------+
| 지표 범주            | 구체 예시                             |
+----------------------+--------------------------------------+
| 작업 성공률          | 에이전트가 사용자 요청을               |
| (Task Success Rate)  | 정상 완료한 비율                       |
+----------------------+--------------------------------------+
| 실패 지점 분석       | 어떤 단계에서 오류가 발생했는지         |
| (Failure Points)     | 추적                                  |
+----------------------+--------------------------------------+
| 인간 개입률          | 에이전트가 스스로 해결하지 못해         |
| (Human Intervention) | 사람이 개입한 비율                     |
+----------------------+--------------------------------------+
| 품질 점수            | 일관성(Coherence), 근거성             |
| (Quality Scores)     | (Groundedness), 안전성(Safety)        |
+----------------------+--------------------------------------+
| 사용자 신뢰도        | "이 답변이 도움이 됐나요?"             |
| (User Trust)         | 피드백 긍정/부정 비율                  |
+----------------------+--------------------------------------+
```

품질 점수를 측정할 때는 두 가지 방법이 있습니다. Pointwise 평가는 하나의 출력에 0점에서 5점까지 절대 점수를 매기는 방식입니다. Pairwise 평가는 두 출력을 나란히 놓고 어느 쪽이 더 나은지 비교하는 방식입니다. 기능의 성격에 따라 둘을 조합하여 사용합니다.

UX 측면에서도 비결정론성을 고려한 설계가 필요합니다. "AI가 생성한 내용입니다"라는 라벨을 표시하여 사용자가 출력의 성격을 인지하게 합니다. 재생성(Regenerate) 버튼을 두어 다른 결과를 원할 때 다시 시도할 수 있게 합니다. 에이전트가 확신이 낮을 때 그 불확실성을 시각적으로 표현하는 방법도 설계합니다. "이 답변이 도움이 됐나요?"와 같은 피드백 루프를 넣어 품질 데이터를 지속적으로 수집합니다.

정리하면, LLM은 다음 토큰의 확률 분포에서 샘플링하는 방식으로 텍스트를 생성하기 때문에 본질적으로 확률적 시스템입니다. Temperature 파라미터로 출력의 일관성과 다양성 사이의 균형을 조절할 수 있고, 환각은 이 확률적 특성에서 비롯되는 피할 수 없는 부작용입니다. 기획자는 결정론적 명세 대신 허용 범위와 품질 평가 방법을 PRD에 포함하고, 작업 성공률, 인간 개입률, 사용자 신뢰도 같은 새로운 KPI를 설계해야 합니다.

다음 단원인 1.1.4에서는 에이전트 워크플로우 패턴 개관을 다룹니다. 에이전트가 작업을 수행하는 다양한 구조적 패턴을 살펴봅니다.

이 단원을 마치면 AI 에이전트의 확률적 출력이 전통 소프트웨어와 어떻게 다른지 설명하고, 기획 시 고려해야 할 점을 나열할 수 있습니다.
