# 1.3.2 8가지 AI 모델 유형과 유스케이스 매핑

1.3.1에서 LLM의 토큰화, 컨텍스트 창, 추론 방식을 살펴봤습니다. LLM은 텍스트를 입력받아 텍스트를 생성하는 모델입니다. 그런데 AI 제품을 기획하다 보면 텍스트만으로 해결되지 않는 문제를 자주 만납니다. 고객이 보낸 스크린샷에서 오류를 찾아야 할 수도 있고, 콜센터 통화를 문자로 바꿔야 할 수도 있습니다. 마케팅 배너 이미지를 자동으로 만들고 싶을 수도 있고, 정형 데이터에서 이탈 가능성이 높은 고객을 골라내야 할 수도 있습니다.

이런 문제마다 적합한 AI 모델 유형이 다릅니다. "AI = LLM"이라고 생각하면 텍스트 생성 모델 하나로 모든 문제를 풀려는 함정에 빠집니다. 이번 단원에서는 AI 모델을 여덟 가지 유형으로 나누고, 각 유형이 어떤 문제에 적합한지를 살펴봅니다. 이 지도를 갖추면 "이 기능에는 어떤 모델이 필요한가"라는 질문에 근거 있는 답을 내릴 수 있습니다.

여덟 가지 유형을 한눈에 먼저 정리합니다. 각 유형의 입력과 출력이 무엇인지, 대표 모델은 무엇인지, 어디에 쓰이는지를 아래 표에 담았습니다. 이후 문단에서 유형별로 하나씩 풀어 설명합니다.

```
8가지 AI 모델 유형 요약

+--------------------+-----------------+------------------------+----------------------------+
| 모델 유형          | 입력 -> 출력    | 대표 모델               | 핵심 유스케이스            |
+--------------------+-----------------+------------------------+----------------------------+
| 텍스트 생성        | 텍스트 -> 텍스트 | GPT-4.5, Claude 4      | 챗봇, 요약, 문서 작성      |
| 이미지 생성        | 텍스트 -> 이미지 | DALL-E 3, Midjourney   | 마케팅 소재, 컨셉 아트     |
| 코드 생성          | 텍스트 -> 코드   | Codex, GitHub Copilot  | 코드 자동완성, 버그 수정   |
| 음성               | 오디오 <-> 텍스트| Whisper, OpenAI TTS    | 콜센터, 회의록, 음성 에이전트|
| 멀티모달           | 복합 -> 복합     | GPT-4V, Gemini 2.5    | 이미지 분석, 문서 이해     |
| 임베딩             | 텍스트 -> 벡터   | text-embedding-3       | 시맨틱 검색, RAG, 추천     |
| 분류/회귀          | 정형 데이터 -> 예측값| XGBoost, 신경망    | 사기 탐지, 이탈 예측       |
| 강화학습 에이전트  | 상태 -> 행동     | AlphaGo, RLHF 기반 LLM| 자율주행, 추천 최적화      |
+--------------------+-----------------+------------------------+----------------------------+
```

첫 번째 유형은 **텍스트 생성 모델**입니다. 1.3.1에서 다룬 LLM이 바로 이 유형에 해당합니다. 텍스트를 입력받아 텍스트를 출력합니다. 트랜스포머 아키텍처를 기반으로 방대한 텍스트 데이터로 사전학습한 뒤, RLHF(인간 피드백 강화학습)로 사용자 의도에 맞게 정렬한 모델입니다. 토큰 단위로 다음 단어를 예측하며, 수십억에서 수조 개의 파라미터가 문법, 의미, 추론, 문화적 맥락을 인코딩합니다.

대표 모델로는 GPT-4.5(OpenAI), Claude 4 Sonnet(Anthropic), Gemini 2.5(Google)가 있습니다. GPT-4.5는 ChatGPT와 Microsoft Copilot을 구동합니다. Claude 4 Sonnet은 200K 토큰 컨텍스트 창과 확장된 사고 모드를 제공합니다. Gemini 2.5는 100만 토큰 컨텍스트를 지원하며 멀티모달 기능을 갖추고 있습니다.

기획자가 이 모델을 제품에 적용할 때 주의할 점이 두 가지 있습니다. 하나는 확률적 출력입니다. 같은 입력을 넣어도 매번 다른 결과가 나올 수 있습니다. 성공 지표를 설계할 때 이 변동성을 반영해야 합니다. 다른 하나는 환각(hallucination)입니다. 모델이 사실과 다른 내용을 그럴듯하게 생성하는 현상입니다. 정확성이 중요한 업무에서는 사람의 검수 단계를 포함하거나, RAG(검색 보강 생성)를 결합하여 사실 기반 응답을 유도해야 합니다.

텍스트 생성 모델의 대표적인 유스케이스로는 고객 지원 챗봇, 콘텐츠 초안 작성, 문서 요약, PRD 초안, 이메일 자동화, 법률이나 의료 문서 검토 보조가 있습니다. 1.3.1에서 다룬 비용 구조(토큰 기반 과금)가 그대로 적용되므로, 컨텍스트 창 크기와 응답 길이가 비용에 직결됩니다.

두 번째 유형은 **이미지 생성 모델**입니다. 텍스트 프롬프트를 입력하면 이미지를 출력합니다. 텍스트 생성 모델이 다음 토큰을 예측하는 방식으로 작동하듯, 이미지 생성 모델에도 고유한 생성 메커니즘이 있습니다.

현재 주류인 방식은 **확산(Diffusion) 모델**입니다. 확산 모델의 작동 원리를 이해하려면 먼저 "노이즈"라는 개념을 알아야 합니다. 디지털 이미지에서 노이즈란 화면에 무작위로 흩뿌린 점들입니다. 오래된 TV에서 채널이 안 잡힐 때 나오는 "지직거림"을 떠올리면 됩니다.

확산 모델은 세 단계로 작동합니다. 먼저 학습 단계에서, 깨끗한 이미지에 노이즈를 조금씩 더해 결국 완전한 노이즈로 만드는 과정을 수천 번 반복하며 "노이즈를 제거하는 방법"을 학습합니다. 그 다음 생성 단계에서, 완전한 노이즈 이미지에서 출발하여 노이즈를 점진적으로 제거하며 깨끗한 이미지를 만듭니다. 이때 텍스트 프롬프트가 "어떤 방향으로 노이즈를 제거할지"를 안내합니다. 텍스트와 이미지 사이의 연결은 CLIP 같은 텍스트-이미지 임베딩 모델이 담당합니다.

```
확산 모델 생성 과정

[완전한 노이즈] --> 1단계 노이즈 제거 --> 2단계 노이즈 제거 --> ... --> [깨끗한 이미지]
                          ^                      ^
                          |                      |
                    텍스트 프롬프트          텍스트 프롬프트
                    "산 위의 일출"          "산 위의 일출"

* 매 단계마다 텍스트 프롬프트가 노이즈 제거 방향을 안내합니다.
* 단계가 많을수록 이미지 품질이 올라가지만 생성 시간도 늘어납니다.
```

대표 모델로는 DALL-E 3(OpenAI), Midjourney, Stable Diffusion이 있습니다. DALL-E 3는 ChatGPT에 통합되어 대화하듯 이미지를 생성할 수 있습니다. 프롬프트 충실도와 사실적 묘사에 강점이 있고, 상업 라이선스가 명확합니다. Midjourney는 예술적이고 영화적인 품질에 강점이 있어 컨셉 아트나 브랜드 비주얼 작업에 많이 쓰입니다. Stable Diffusion은 오픈소스여서 자체 서버에 설치하고 커스터마이징할 수 있습니다.

기획 시 고려할 점은 콘텐츠 정책입니다. 저작권 이슈와 딥페이크 방지를 위한 컴플라이언스 설계가 필요합니다. 유스케이스로는 마케팅 소재 제작, 제품 시각화, 게임이나 엔터테인먼트 컨셉 아트, 블로그 일러스트, 광고 캠페인 비주얼, SNS 콘텐츠, 브랜드 아이덴티티 탐색이 있습니다.

세 번째 유형은 **코드 생성 모델**입니다. 자연어로 된 명세를 코드로 변환하거나, 기존 코드를 분석하고 완성하고 설명합니다. 텍스트 생성 모델과 기본 아키텍처(트랜스포머)는 같지만, 학습 데이터가 다릅니다. 수십억 줄의 공개 소스코드(GitHub 등)를 추가로 학습하여 프로그래밍 언어의 문법과 패턴을 깊이 이해합니다.

대표 모델로는 OpenAI Codex와 GitHub Copilot이 있습니다. Codex는 GPT-3 변형으로 GitHub Copilot의 초기 구동 엔진이었습니다. 현재 Copilot은 GPT-4.1을 기본으로 사용하며, Claude, Gemini, Grok 등 다중 모델을 지원합니다. 30개 이상의 프로그래밍 언어를 처리할 수 있습니다.

기획자 관점에서 주목할 개념은 "코딩 에이전트" 모드입니다. Copilot Agent Mode는 이슈를 할당받아 클라우드 환경에서 자율적으로 개발한 뒤 Pull Request를 생성합니다. 사람이 명세를 작성하면 AI가 구현까지 수행하는 구조입니다. 이 모드를 도입할 때는 두 가지 KPI를 동시에 측정해야 합니다. 하나는 개발 속도(지연시간 단축)이고, 다른 하나는 코드 품질(버그율)입니다. 속도만 추구하면 품질이 떨어질 수 있습니다.

유스케이스로는 코드 자동완성, 버그 수정, 단위 테스트 생성, 코드 리뷰, 레거시 코드 설명, 프로그래밍 언어 간 번역, 보안 취약점 탐지가 있습니다.

네 번째 유형은 **음성 모델**입니다. 음성 모델은 크게 두 방향으로 나뉩니다. 하나는 사람의 말을 문자로 바꾸는 STT(Speech-to-Text)입니다. 다른 하나는 문자를 사람의 목소리로 바꾸는 TTS(Text-to-Speech)입니다.

**STT**의 대표 모델은 OpenAI Whisper입니다. Whisper는 웹에서 수집한 68만 시간 분량의 다국어 오디오로 학습한 인코더-디코더 트랜스포머입니다. 작동 방식은 다음과 같습니다. 먼저 오디오를 30초 단위로 잘라냅니다. 잘린 오디오를 로그-멜 스펙트로그램이라는 시각적 표현으로 변환합니다. 로그-멜 스펙트로그램이란 소리의 주파수와 세기를 시간 축에 따라 그린 그래프입니다. 이 그래프를 트랜스포머 인코더에 통과시키면 텍스트로 전사됩니다. Whisper는 다국어 음성인식, 번역, 언어 식별을 모두 지원하며 오픈소스로 자체 호스팅이 가능합니다.

**TTS**는 반대 방향입니다. 텍스트를 자연스러운 음성 파형으로 합성합니다. 최근에는 "공감하는 고객 서비스 에이전트처럼 말하라"는 식으로 감정 톤과 말투를 프롬프트로 제어할 수 있는 수준으로 발전했습니다.

기획자가 음성 모델을 제품에 적용할 때 핵심 구조는 STT + LLM + TTS 파이프라인입니다. 사용자의 말을 STT로 텍스트로 바꾸고, LLM이 응답을 생성하고, TTS가 그 응답을 음성으로 변환합니다. 이 세 단계를 이어 붙이면 음성 에이전트가 됩니다. 제품 품질의 핵심 지표는 실시간 처리(지연시간)와 배경 소음이나 억양에 대한 강건성입니다.

```
음성 에이전트 파이프라인

[사용자 음성] --> STT(Whisper) --> [텍스트] --> LLM(GPT/Claude) --> [응답 텍스트] --> TTS --> [음성 응답]
                    |                                                                    |
                    +-- 지연시간 측정 구간 1 --+---- 지연시간 측정 구간 2 ----+-- 구간 3 --+
```

유스케이스로는 고객 콜센터 통화 기록과 분석, 회의록 자동 생성, 언어 학습 앱, 팟캐스트 자막, 접근성 보조(청각 장애), 음성 에이전트, 다국어 실시간 번역이 있습니다.

다섯 번째 유형은 **멀티모달 모델**입니다. 앞서 살펴본 네 가지 유형은 각각 특정 입출력에 특화되어 있습니다. 텍스트 생성은 텍스트만, 이미지 생성은 텍스트에서 이미지만, 음성 모델은 오디오와 텍스트 사이만 다룹니다. 멀티모달 모델은 이 경계를 허뭅니다. 텍스트, 이미지, 오디오, 비디오 등 여러 종류의 입력을 하나의 모델에서 동시에 처리합니다.

작동 원리는 각 입력 유형(모달리티)을 공통된 수치 공간에 투영하는 것입니다. 텍스트든 이미지든 같은 형식의 숫자 표현으로 바꾸고, 그 숫자들 사이의 관계를 파악하여 교차 모달 추론을 수행합니다. 예를 들어 스크린샷 이미지를 넣으면서 "이 화면에서 오류가 있는 부분을 찾아줘"라고 텍스트로 요청할 수 있습니다. 모델은 이미지의 시각 정보와 텍스트의 의미를 함께 이해하여 답을 생성합니다.

대표 모델로는 GPT-4.5/5(텍스트+이미지), Claude 4(텍스트+이미지), Gemini 2.5(텍스트+이미지+음성+비디오), LLaMA 4(멀티모달, 오픈웨이트)가 있습니다.

기획자 관점에서 멀티모달 모델의 장점은 시스템 복잡도 감소입니다. 이전에는 이미지를 분석하려면 별도의 이미지 분석 모델을 호출하고, 그 결과를 텍스트로 변환해 LLM에 넘기는 복잡한 파이프라인이 필요했습니다. 멀티모달 모델은 이 과정을 단일 호출로 처리합니다. 다만 비용 설계에 주의가 필요합니다. 이미지 토큰은 텍스트 토큰보다 비용이 높습니다. 이미지 입력이 많은 서비스에서는 모달리티별 비용 차이를 예산에 반영해야 합니다.

유스케이스로는 고객 지원 스크린샷 분석, 의료 영상 분석, 연구 논문 다이어그램과 표 해석, 문서 레이아웃 이해(컴플라이언스 처리), 제품 사진 기반 설명 자동 생성, 시각 장애인을 위한 화면 해설이 있습니다.

여섯 번째 유형은 **임베딩 모델**입니다. 앞의 다섯 가지 유형은 모두 사람이 읽거나 보거나 듣는 결과물을 생성합니다. 임베딩 모델은 다릅니다. 사용자에게 직접 보이지 않지만, 검색과 추천 시스템의 핵심 부품 역할을 합니다.

임베딩 모델이 하는 일을 이해하려면 먼저 "의미적 유사도"라는 문제를 알아야 합니다. 키워드 검색은 단어가 정확히 일치해야 결과를 찾습니다. "자동차 보험료"를 검색하면 "차량 보험 비용"이라는 문서는 놓칠 수 있습니다. 단어는 다르지만 의미는 같기 때문입니다. 이 문제를 해결하는 것이 임베딩 모델입니다.

**임베딩(embedding)**이란 텍스트, 이미지, 코드 같은 비정형 데이터를 고밀도 수치 벡터로 변환하는 것입니다. 벡터란 숫자들의 나열입니다. 예를 들어 [0.12, -0.34, 0.78, ...] 같은 형태로, 수백에서 수천 개의 숫자로 이루어집니다. 핵심은 의미적으로 비슷한 텍스트일수록 벡터 공간에서 가까운 위치에 배치된다는 것입니다. "자동차 보험료"와 "차량 보험 비용"은 단어는 다르지만 벡터 공간에서 가까이 위치합니다.

대표 모델로는 BERT, Sentence-BERT, OpenAI의 text-embedding-3-large가 있습니다. 이 모델들이 생성한 벡터를 저장하고 검색하는 데 쓰이는 것이 벡터 데이터베이스(Pinecone, Weaviate, pgvector)입니다.

기획자가 임베딩 모델을 신경 써야 하는 이유는 RAG(검색 보강 생성) 시스템의 품질이 임베딩 모델에 달려 있기 때문입니다. 1.3.1에서 다룬 텍스트 생성 모델이 정확한 답을 내려면, 관련 문서를 먼저 찾아 컨텍스트에 넣어야 합니다. 이 "관련 문서를 찾는" 단계가 임베딩 모델의 역할입니다. 임베딩 모델의 선택(다국어 지원 여부, 도메인 특화 여부)이 검색 정확도에 직접 영향을 미칩니다. 비용은 텍스트 생성 모델의 추론 비용보다 낮지만, 대량의 문서를 벡터로 변환하는 인덱스 구축 비용을 고려해야 합니다.

유스케이스로는 시맨틱 검색, 문서 클러스터링, 추천 시스템(사용자-콘텐츠 유사도), 이상 탐지, 중복 문서 제거, 고객 리뷰 감성 분류 보조, RAG 검색 레이어가 있습니다.

일곱 번째 유형은 **분류/회귀 모델**입니다. 여기서 잠시 시야를 넓혀야 합니다. 앞의 여섯 가지 유형은 모두 비정형 데이터(텍스트, 이미지, 음성)를 다룹니다. 하지만 기업이 보유한 데이터 중 상당 부분은 정형 데이터입니다. 엑셀 표처럼 행과 열로 정리된 숫자, 범주, 날짜 데이터입니다. 고객의 구매 이력, 거래 기록, 센서 수치 같은 데이터가 여기에 해당합니다.

정형 데이터에서 예측을 수행하는 모델이 분류/회귀 모델입니다. 이 모델은 LLM보다 훨씬 오래된 전통적 **지도학습(Supervised Learning)** 방식을 사용합니다. 지도학습이란 "정답이 딸린 데이터"로 모델을 학습시키는 방식입니다. 예를 들어 과거에 사기로 판명된 거래 데이터와 정상 거래 데이터를 모델에 보여주면, 모델이 패턴을 학습하여 새 거래가 사기인지 정상인지 예측합니다.

**분류(Classification)**는 입력을 이산 범주로 나눕니다. "이 이메일은 스팸인가 정상인가", "이 거래는 사기인가 아닌가", "이 환자는 양성인가 음성인가"처럼 범주를 예측합니다. **회귀(Regression)**는 연속적인 숫자를 예측합니다. "이 집의 가격은 얼마인가", "다음 달 수요량은 얼마인가", "이 광고의 최적 입찰가는 얼마인가"처럼 값을 예측합니다.

대표 알고리즘으로는 Random Forest, XGBoost, 로지스틱 회귀, 신경망이 있습니다. LLM이 등장한 이후에도 정형 데이터 예측에서는 이 전통 모델들이 여전히 우위를 점합니다. LLM은 비정형 텍스트 감성 분류에는 활용되지만, 숫자 기반 예측은 전통 모델이 더 빠르고 해석 가능합니다.

기획자가 분류/회귀 모델을 알아야 하는 이유는 핵심 비즈니스 문제 상당수가 여기에 해당하기 때문입니다. 고객 이탈 예측, 가격 최적화, 사기 탐지, 대출 심사, 재고 관리, 리드 스코어링은 모두 정형 데이터 기반 문제입니다. 또한 금융이나 의료 같은 규제 산업에서는 **해석 가능성(Explainability)**이 중요한 선택 기준입니다. "왜 이 고객의 대출을 거절했는가"를 설명할 수 있어야 하는데, 전통 모델은 LLM보다 의사결정 근거를 투명하게 제시할 수 있습니다.

여덟 번째 유형은 **강화학습 에이전트**입니다. 앞의 일곱 가지 유형은 데이터를 입력받아 결과를 한 번 출력하는 구조입니다. 강화학습은 다릅니다. 에이전트가 환경과 반복적으로 상호작용하며 행동의 결과를 관찰하고, 더 나은 행동을 스스로 학습합니다.

작동 원리를 이해하기 위해 간단한 예를 들어보겠습니다. 게임 캐릭터가 미로를 탈출해야 한다고 가정합니다. 캐릭터(에이전트)는 현재 위치(상태)를 보고 상하좌우 중 하나를 선택(행동)합니다. 벽에 부딪히면 벌점을 받고, 출구에 가까워지면 보상을 받습니다. 수천 번 시행착오를 반복하면서 에이전트는 **보상(Reward)**을 최대화하는 **정책(Policy)**을 학습합니다. 정책이란 "이 상태에서는 이 행동을 하라"는 규칙 집합입니다.

```
강화학습 작동 구조

+----------+      행동       +-----------+
|          | -------------> |           |
|  에이전트 |                |   환경     |
|          | <------------- |           |
+----------+  상태 + 보상    +-----------+

1. 에이전트가 현재 상태를 관찰합니다.
2. 정책에 따라 행동을 선택합니다.
3. 환경이 새 상태와 보상을 돌려줍니다.
4. 보상을 최대화하도록 정책을 갱신합니다.
5. 1~4를 수천~수백만 번 반복합니다.
```

대표 적용 사례로는 DeepMind의 AlphaGo(바둑), AlphaStar(스타크래프트), Tesla Autopilot(자율주행), Google DeepMind의 데이터센터 냉각 효율화가 있습니다. 추천 시스템에서도 강화학습이 활용됩니다. Netflix나 YouTube가 피드 순서를 결정할 때 사용자의 클릭과 시청 시간을 보상 신호로 삼아 추천 정책을 최적화합니다.

기획자가 특히 알아야 할 점은 강화학습과 LLM의 연결 고리입니다. 1.3.1에서 언급한 RLHF(인간 피드백 강화학습)가 바로 강화학습의 변형입니다. GPT, Claude 같은 현대 LLM은 사전학습 후 RLHF로 사용자 선호도에 맞게 정렬됩니다. 인간의 피드백을 보상 모델로 변환하여, 모델이 더 안전하고 유익한 응답을 생성하도록 유도합니다. 강화학습은 직접 사용자와 대면하는 제품보다는 시스템 최적화와 자율 제어 레이어에서 주로 활용됩니다. 보상 함수를 설계할 때 제품 목표와 정확히 일치시켜야 하며, 어긋나면(오정렬) 의도치 않은 행동이 발생할 수 있습니다.

여덟 가지 유형을 모두 살펴봤으니, 기획자가 모델을 선택하는 판단 기준을 정리합니다. 핵심은 "풀려는 문제의 성격"에서 출발하는 것입니다. 비정형 텍스트를 다루는가, 이미지를 만들어야 하는가, 정형 데이터에서 예측이 필요한가, 시스템이 스스로 최적화해야 하는가. 이 질문에 따라 적합한 모델 유형이 달라집니다. 또한 여러 유형을 조합하는 경우도 흔합니다. 임베딩 모델로 관련 문서를 검색하고, 텍스트 생성 모델이 답변을 생성하는 RAG 아키텍처가 대표적인 조합 사례입니다.

정리하면, AI 모델은 텍스트 생성, 이미지 생성, 코드 생성, 음성, 멀티모달, 임베딩, 분류/회귀, 강화학습 에이전트의 여덟 가지 유형으로 나뉩니다. 각 유형은 입출력 형태, 학습 방식, 적합한 문제 영역이 다릅니다. 텍스트 생성 모델(LLM)이 가장 널리 알려져 있지만, 정형 데이터 예측에는 분류/회귀 모델이, 시스템 최적화에는 강화학습이, 검색 품질에는 임베딩 모델이 각각 더 적합합니다. 기획자는 "이 문제에 어떤 유형의 모델이 필요한가"를 먼저 판단하고, 필요하면 여러 유형을 조합하여 제품을 설계해야 합니다.
