# 1.3.4 AI 팀 구성과 역할별 협업 방법

1.2.2에서 AI PM의 세 가지 아키타입을 살펴보았습니다. Core Product PM, Platform Builder, AI-Powered PM은 각각 다른 깊이를 가지지만, 결국 혼자서 AI 제품을 만들지는 않습니다. AI 제품은 모델을 만드는 사람, 데이터를 정비하는 사람, 사용자 경험을 설계하는 사람, 그리고 이 모든 것을 하나의 제품으로 엮는 기획자가 함께 움직여야 완성됩니다. 이 단원에서는 AI 팀을 구성하는 핵심 역할을 하나씩 살펴보고, 기획자가 이들과 어떻게 협업하는지, 그리고 팀 규모와 조직 구조에 따라 역할 분배가 어떻게 달라지는지를 다룹니다.

AI 팀의 역할을 이해하려면 먼저 GenAI 제품이 전통 소프트웨어와 어떻게 다른지를 떠올려야 합니다. 전통 소프트웨어 제품에서 팀은 기획자가 요구사항을 정의하고, 개발자가 코드를 작성하며, 디자이너가 화면을 설계하는 비교적 명확한 분업 구조로 운영됩니다. GenAI 제품에서는 여기에 모델 훈련, 데이터 파이프라인, 프롬프트 설계, 출력 품질 평가라는 새로운 업무 영역이 추가됩니다. 이 영역들은 기존 직군이 자연스럽게 흡수하기 어렵기 때문에, 새로운 역할이 필요해집니다.

GenAI 팀의 역할은 크게 세 그룹으로 나뉩니다. 첫 번째는 **기술 클러스터**입니다. 모델과 데이터를 직접 다루는 엔지니어들이 여기에 속합니다. 두 번째는 **제품/비즈니스 클러스터**입니다. 사용자 가치를 정의하고 시장에 전달하는 역할들이 포함됩니다. 세 번째는 **거버넌스/안전 클러스터**입니다. AI 윤리, 규제 준수, 품질 보증을 담당합니다. 이 단원에서는 기획자와 가장 밀접하게 협업하는 기술 클러스터의 역할을 중심으로 살펴보겠습니다.

기술 클러스터에서 가장 먼저 알아야 할 역할은 **ML 엔지니어(ML Engineer)**입니다. ML 엔지니어는 모델을 제품에 통합하고 배포하는 사람입니다. 구체적으로는 모델 파인튜닝, 배포 인프라 구축, MLOps 파이프라인 운영을 담당합니다. 파인튜닝이란 범용 모델을 특정 업무에 맞게 추가 학습시키는 과정입니다. MLOps 파이프라인이란 모델의 학습, 평가, 배포, 모니터링을 자동화하는 일련의 시스템을 뜻합니다. ML 엔지니어가 없으면 모델이 연구 단계에 머물러 실제 제품으로 나오지 못합니다. 채용 시장에서도 ML 엔지니어는 GenAI 관련 직군 중 가장 수요가 높은 역할 가운데 하나입니다.

다음은 **데이터 사이언티스트(Data Scientist)**입니다. 전통적으로 데이터 사이언티스트는 데이터를 분석하고 모델을 직접 훈련하는 역할이었습니다. 그런데 GenAI 환경에서는 이 역할의 무게 중심이 크게 이동했습니다. 대규모 언어 모델을 처음부터 훈련하는 일은 소수의 연구 조직만 수행합니다. 대부분의 제품 팀은 이미 만들어진 모델을 활용합니다. 이 환경에서 데이터 사이언티스트의 핵심 업무는 모델 성능 평가, 실험 설계, 출력 품질 패턴 분석으로 전환되었습니다. 쉽게 말해, "모델을 만드는 사람"에서 "모델이 잘 작동하는지 검증하는 사람"으로 바뀐 것입니다.

**데이터 엔지니어(Data Engineer)**도 빼놓을 수 없습니다. 데이터 엔지니어는 데이터 파이프라인을 구축하고 유지합니다. 데이터 파이프라인이란 원시 데이터를 수집, 정제, 변환하여 모델이 사용할 수 있는 형태로 전달하는 일련의 처리 과정입니다. GenAI 제품에서 데이터 엔지니어의 중요성은 더욱 커졌습니다. RAG(검색 증강 생성) 기반 아키텍처에서는 모델이 참조하는 외부 데이터의 품질이 출력 품질을 직접 결정하기 때문입니다. 데이터가 부정확하거나 오래되었으면, 아무리 좋은 모델을 써도 결과가 나빠집니다.

**프롬프트 엔지니어(Prompt Engineer)**는 LLM의 동작을 형성하는 지시문을 설계하고, 테스트하며, 개선하는 역할입니다. 1.2.2에서 AI-Powered PM의 핵심 역량으로 프롬프트 엔지니어링을 언급했는데, 프롬프트 엔지니어는 이 역량을 전문적으로 수행하는 사람입니다. ML에 대한 깊은 이론적 지식보다 언어 모델의 행동 패턴을 실험적으로 파악하는 감각이 더 중요합니다.

마지막으로 **AI 디자이너(AI Product Designer)**가 있습니다. AI 디자이너는 AI 출력이 사용자 경험에 자연스럽게 녹아들도록 인터페이스를 설계합니다. 전통적인 UX 디자이너와의 차이는 설계 대상에 있습니다. 전통 UX 디자이너는 버튼을 누르면 항상 같은 결과가 나오는 확정적 인터페이스를 설계합니다. AI 디자이너는 매번 다른 결과가 나올 수 있는 비결정론적 출력을 사용자에게 어떻게 보여줄지를 고민합니다. 예를 들어, AI가 생성한 답변의 신뢰도를 어떻게 표시할지, 모델이 틀렸을 때 사용자가 어떻게 수정할 수 있게 할지 같은 문제가 AI 디자이너의 고유 영역입니다.

이제 기획자가 이 역할들과 어떻게 협업하는지를 살펴봅니다. AI 제품에서 기획자와 엔지니어의 분업은 세 가지 영역으로 나뉩니다. 기획자는 "무엇을(what)" 소유합니다. 사용자 관점에서 성공 기준을 정의하는 것입니다. 엔지니어는 "어떻게(how)"를 소유합니다. 그 기준을 측정하는 방법을 설계하는 것입니다. "왜 이 지표가 중요한가(why)"는 양쪽이 함께 소유합니다. 1.2.2에서 이미 이 원칙을 다루었는데, 이 단원에서는 이 원칙이 실제 협업 과정에서 어떻게 작동하는지를 구체적으로 살펴봅니다.

기획자와 엔지니어의 협업은 세 단계로 진행됩니다. 첫 번째 단계는 **초기 프로토타이핑**입니다. 기획자가 "왜 이 기능이 필요한가"를 주도하고, 엔지니어가 빠르게 프로토타입을 만듭니다. 이 단계에서 팀은 세 가지 활동을 수행합니다. 공유 언어를 개발하고, 엣지 케이스를 발견하며, 성공 기준을 스케치합니다. 공유 언어 개발이란 기획자와 엔지니어가 같은 용어를 같은 의미로 사용하도록 맞추는 과정입니다. 예를 들어, "정확도"라는 말이 기획자에게는 "사용자가 만족하는 답변의 비율"을 뜻하고, 엔지니어에게는 통계적 정밀도를 뜻할 수 있습니다. 이 차이를 초기에 맞추지 않으면 이후 단계에서 혼란이 생깁니다.

두 번째 단계는 **평가 체계 구축**입니다. 기획자가 골든셋(모범 답안 모음), 품질 차원(정확성, 유용성, 안전성 등), 실패 모드(모델이 틀리는 대표 유형)를 정의합니다. 엔지니어는 이 정의를 바탕으로 통계적 엄밀성을 확보하고, 대규모로 평가를 자동화할 수 있는 시스템을 구축합니다. 이 단계에서 기획자에게 세 가지 역량이 요구됩니다. 첫째, 통계적 리터러시입니다. 신뢰구간, 표본 크기, 통계적 유의성 같은 개념을 이해해야 엔지니어와 평가 결과를 놓고 대화할 수 있습니다. 둘째, AI 직관입니다. 모델이나 프롬프트가 왜 실패하는지를 감각적으로 이해하는 능력입니다. 셋째, 기술적 공감입니다. 엔지니어링 제약을 이해하고 무리한 요구를 하지 않는 태도입니다. 트랜스포머의 구현 방법을 알 필요는 없지만, 모델이 왜 특정 유형의 질문에 약한지 정도는 파악할 수 있어야 합니다.

세 번째 단계는 **프로덕션 운영**입니다. 제품이 출시된 후 실제 사용자 데이터를 기반으로 품질을 모니터링합니다. 기획자와 엔지니어가 공유 대시보드를 보며 공동으로 사후 분석을 수행합니다.

```
기획자-엔지니어 협업 3단계

단계 1: 초기 프로토타이핑
+------------------+     +------------------+
| 기획자           |     | 엔지니어         |
| why 주도         |<--->| 프로토타입 제작   |
| 공유 언어 개발   |     | 엣지 케이스 발견  |
+------------------+     +------------------+
         |
         v
단계 2: 평가 체계 구축
+------------------+     +------------------+
| 기획자           |     | 엔지니어         |
| 골든셋 정의      |<--->| 통계적 엄밀성    |
| 품질 차원 설계   |     | 자동화 시스템    |
| 실패 모드 분류   |     | 확장성 확보      |
+------------------+     +------------------+
         |
         v
단계 3: 프로덕션 운영
+--------------------------------------+
| 공유 대시보드 + 공동 사후 분석       |
| 기획자: 비즈니스 맥락                |
| 엔지니어: 기술 원인 분석             |
+--------------------------------------+
```

이 세 단계에서 특히 중요한 협업 활동이 세 가지 있습니다. 첫째, 평가 프롬프트 공동 작성입니다. 기획자가 프롬프트를 작성하고 엔지니어에게 넘기는 핸드오프 방식이 아니라, 둘이 나란히 앉아 함께 작성하는 페어 프로그래밍 방식으로 진행합니다. 둘째, 공동 레이블링 세션입니다. 기획자와 엔지니어가 함께 모델 출력을 보며 "이 답변은 좋은가, 나쁜가"를 판정합니다. 이 활동은 AI 제품 개발에서 가장 과소평가되는 활동이지만, 품질 기준에 대한 양측의 이해를 일치시키는 데 가장 효과적입니다. 셋째, 통계적 유의성 대화입니다. 기획자가 비즈니스 맥락을 제공하고, 엔지니어가 통계적 전문성을 제공하여, "이 정도 차이가 의미 있는 차이인가"를 함께 판단합니다.

여기서 기획자의 역할에 대해 한 가지 더 짚어야 할 점이 있습니다. AI 제품에서 기획자의 역할은 "만드는 사람(maker)"에서 "방향을 정하고 조율하는 사람(director)"으로 전환되고 있습니다. AI가 초안을 생성하고, 데이터를 분석하고, 사용자 피드백을 정리하는 업무를 점점 더 많이 대신합니다. 그러나 책임(accountability)은 AI에게 위임할 수 없습니다. 최종 의사결정, 윤리적 판단, 이해관계자 간 조율은 여전히 기획자의 몫입니다. 기획자에게 새롭게 요구되는 핵심 역량은 AI에 위임할 작업과 직접 보유해야 할 책임의 경계를 설정하는 것입니다.

이제 팀 전체의 조직 구조를 살펴봅니다. 앞서 개별 역할을 다루었다면, 여기서는 이 역할들을 어떻게 배치하느냐의 문제입니다. 조직 구조는 크게 세 가지 모델로 나뉩니다.

첫 번째는 **중앙화 AI CoE(Center of Excellence)** 모델입니다. 하나의 전담 팀이 조직 전체의 GenAI 역량, 도구, 거버넌스를 모두 소유합니다. 장점은 일관성입니다. 모든 AI 프로젝트가 동일한 표준과 인프라를 사용하므로 품질 관리가 수월합니다. 단점은 병목입니다. 모든 요청이 하나의 팀을 거쳐야 하므로, 사업부가 늘어날수록 대기 시간이 길어집니다. AI 도입 초기 단계에서 표준을 수립할 때 적합합니다.

두 번째는 **완전 분산형(Fully Embedded)** 모델입니다. AI 전문가를 각 사업부에 직접 배치합니다. 장점은 속도입니다. 각 사업부의 도메인 맥락을 깊이 이해한 AI 전문가가 바로 옆에서 일하므로, 가치 창출까지의 시간이 짧아집니다. 단점은 중복과 거버넌스 공백입니다. 사업부마다 비슷한 인프라를 따로 구축하거나, 서로 다른 품질 기준을 적용할 위험이 있습니다.

세 번째는 **허브-앤-스포크(Hub-and-Spoke) 하이브리드** 모델입니다. 중앙의 허브가 공유 인프라, 표준, 전문 역량을 제공하고, 각 사업부에 배치된 스포크(AI 스쿼드)가 도메인별 응용을 담당합니다. 앞의 두 모델의 장점을 결합한 구조로, 가장 일반적인 엔터프라이즈 패턴입니다. 속도와 거버넌스 사이의 균형을 잡을 수 있습니다.

```
GenAI 팀 3가지 조직 모델

(1) 중앙화 CoE          (2) 완전 분산형         (3) 허브-앤-스포크
+---------------+       +---------+---------+   +-------+-------+
|   AI CoE      |       | 사업부A | 사업부B |   |   허브(Hub)   |
| (전사 AI 팀)  |       |  +AI팀  |  +AI팀  |   | 공유 인프라   |
|               |       |         |         |   | 표준/거버넌스 |
| 모든 요청 처리|       | 독립 운영|독립 운영|   +---+-------+---+
+-------+-------+       +---------+---------+       |       |
        |                                      +----v--+ +--v----+
   모든 사업부에                               |스포크A| |스포크B|
   서비스 제공                                 |사업부 | |사업부 |
                                               |AI스쿼드| |AI스쿼드|
일관성 높음           속도 빠름                +-------+ +-------+
병목 위험             중복/공백 위험           균형 모델
초기 도입 적합        성숙 단계 적합           가장 일반적
```

조직 모델을 선택할 때 흔히 발생하는 실패 패턴도 알아두어야 합니다. 첫째, 데이터 사이언티스트를 과잉 채용하고 ML 엔지니어나 데이터 엔지니어를 경시하는 경우입니다. 모델을 평가하는 사람은 많지만 실제로 배포하고 운영하는 사람이 부족해집니다. 둘째, AI PM 없이 팀을 구성하는 경우입니다. 기술 역량은 갖추었지만 유스케이스를 정의하는 사람이 없어서, 팀이 "쓸 곳을 찾아 떠도는 기술"이 됩니다. 셋째, AI 랩을 사일로로 운영하는 경우입니다. 크로스펑셔널 통합 없이 연구실에만 머물면, 기술이 제품으로 이어지지 않습니다.

마지막으로 팀 규모에 따른 역할 분배를 살펴봅니다. 1.2.2에서 조직 규모에 따른 PM 아키타입의 분업을 다루었는데, 여기서는 PM을 포함한 팀 전체의 구성으로 시야를 넓힙니다.

소규모 팀(약 5명)에서는 최소한의 인원으로 시작합니다. 리드 데이터 엔지니어 1명, AI 기획자(AI PM) 1명, ML 엔지니어 1명, 프롬프트 엔지니어 1명, 그리고 AI 전환을 총괄하는 매니저 1명이 첫 5명의 전형적인 구성입니다. 이 규모에서는 한 사람이 여러 역할을 겸합니다. 데이터 사이언티스트 없이 ML 엔지니어가 평가까지 담당하고, AI 디자이너 없이 기획자가 UX 방향을 잡는 식입니다.

중규모 팀(20명 이상)에서는 전문화가 시작됩니다. 데이터 엔지니어가 증원되고, 평가 전담 데이터 사이언티스트가 합류합니다. AI 윤리 분석가, 규제 준수 담당자, 도메인별 AI 개발자가 추가됩니다. 소규모 팀에서 한 사람이 겸하던 역할이 분리되면서, 각 영역의 깊이가 늘어납니다.

엔터프라이즈 환경에서 이 구성이 안정적으로 운영되기까지는 상당한 시간이 걸립니다. 데이터 기반을 구축하는 데 3개월에서 6개월, 채용과 온보딩에 3개월, 프로덕션 유스케이스를 안정적으로 제공하기까지 6개월에서 12개월이 소요됩니다. 전체적으로 12개월에서 18개월 정도를 예상해야 합니다.


크로스펑셔널 팀 운영에 대해서도 짚어 두겠습니다. 크로스펑셔널 팀이란 기술, 비즈니스, 디자인 등 서로 다른 전문성을 가진 사람들이 하나의 팀으로 결합된 구조를 뜻합니다. GenAI 제품에서 크로스펑셔널 팀 구성이 중요한 이유는 명확합니다. 기술만 있고 비즈니스 맥락이 없으면 유스케이스를 찾지 못합니다. 비즈니스 요구만 있고 기술 이해가 없으면 실현 불가능한 계획을 세웁니다. 크로스펑셔널 통합이 없는 팀은 도입 정체의 주요 원인이 됩니다. 성공하는 팀은 GenAI를 연구실 실험이 아니라 제품 개발로 취급합니다.

이 크로스펑셔널 팀에서 기획자의 위치는 교차점에 있습니다. 비즈니스, 기술, 데이터가 만나는 지점에서 "사용자 니즈를 기술적 요구사항으로 전환"하는 중개자 역할을 합니다. AI PM이 없는 크로스펑셔널 팀은 유스케이스를 찾아 표류하는 기술 역량에 머물 위험이 큽니다.

정리하면, AI 팀은 ML 엔지니어, 데이터 사이언티스트, 데이터 엔지니어, 프롬프트 엔지니어, AI 디자이너 등 기술 클러스터의 전문가들과 기획자가 함께 구성됩니다. 기획자는 "무엇을" 소유하고 엔지니어는 "어떻게"를 소유하며, 초기 프로토타이핑, 평가 체계 구축, 프로덕션 운영의 세 단계를 거쳐 협업합니다. 조직 구조는 중앙화 CoE, 완전 분산형, 허브-앤-스포크 세 모델 중 상황에 맞게 선택하되, 크로스펑셔널 통합 없이는 어떤 모델도 성공하기 어렵습니다. 팀 규모가 작을수록 한 사람이 여러 역할을 겸하고, 팀이 성장하면서 역할별 전문화가 진행됩니다.

다음 단원인 2.1.1에서는 PM을 위한 8가지 프롬프트 기법을 다룹니다. 기획자가 AI 팀과 협업할 때 직접 활용할 수 있는 프롬프트 엔지니어링의 구체적 방법을 살펴봅니다.

이 단원을 마치면 AI 프로덕트 팀의 구성원인 ML 엔지니어, 데이터 사이언티스트, AI 디자이너 등의 역할을 이해하고, 기획자로서 효과적으로 협업하는 방법을 설명할 수 있습니다.
