# 2.1.1 PM을 위한 8가지 프롬프트 기법

1.3.1에서 LLM이 토큰 확률 분포를 바탕으로 다음 토큰을 예측한다는 원리를 살펴봤습니다. 컨텍스트 창 안에 어떤 토큰이 들어있느냐에 따라 모델의 응답이 달라진다는 것도 확인했습니다. 그렇다면 질문이 하나 생깁니다. 같은 모델에 같은 질문을 하더라도 프롬프트를 어떻게 구성하느냐에 따라 응답 품질이 크게 달라질 수 있지 않을까요. 실제로 그렇습니다. 프롬프트를 체계적으로 설계하면 모델이 더 정확하고, 더 일관되며, 더 유용한 답을 내놓습니다.

이 단원에서는 PM이 실무에서 바로 활용할 수 있는 8가지 프롬프트 기법을 다룹니다. 각 기법이 왜 효과가 있는지, 어떤 상황에서 쓰는지, 그리고 실제로 어떻게 적용하는지를 하나씩 살펴봅니다.

첫 번째 기법은 **Role 프롬프트**입니다. LLM에게 특정 역할을 부여하는 방식입니다. 모델은 학습 데이터에서 수많은 직업, 전문 분야, 사고 방식에 관한 텍스트를 학습했습니다. "시니어 프로덕트 매니저처럼 행동하라"고 지시하면, 모델은 학습 데이터에서 프로덕트 매니저와 관련된 어휘, 사고 패턴, 도메인 지식을 우선적으로 참조합니다. 역할을 구체적으로 명시할수록 응답 품질이 높아집니다.

예를 들어 "프로덕트 매니저처럼 답하라"보다 "헬스케어 도메인에서 5년 이상 경력을 가진 시니어 프로덕트 매니저처럼 답하라"가 더 효과적입니다. 후자는 도메인과 경력 수준을 함께 지정했기 때문에, 모델이 참조하는 토큰 분포가 더 좁고 정확해집니다.

```
Role 프롬프트 예시

역할 지정 없음:
  "사용자 이탈률을 줄이는 방법을 알려줘."

역할 지정 있음:
  "Act as a senior product manager with 5 years of experience
   in B2B SaaS. 사용자 이탈률을 줄이는 방법을 알려줘."
```

위 예시에서 첫 번째 프롬프트는 범용적인 답을 생성합니다. 두 번째 프롬프트는 B2B SaaS 맥락에 맞는 구체적인 전략을 생성할 가능성이 높습니다. Role 프롬프트는 단독으로 쓰기보다, 뒤에서 다룰 Context 설정이나 Output Format 지정과 함께 쓸 때 효과가 더 큽니다.

두 번째 기법은 **Context 설정**입니다. 1.3.1에서 컨텍스트 창이 모델이 한 번에 읽을 수 있는 토큰의 한계라고 설명했습니다. Context 설정은 이 컨텍스트 창 안에 적절한 배경 정보를 넣어주는 기법입니다. 제품 전략, 사용자가 겪는 문제, 비즈니스 제약 조건 등을 프롬프트에 포함하면, 모델은 그 범위 안에서 정합성 있는 답변을 생성합니다.

Context 설정에서 중요한 것은 시스템 프롬프트와 사용자 프롬프트의 구분입니다. 시스템 프롬프트는 매 대화에 고정으로 들어가는 배경 정보입니다. 제품의 비전, 타겟 고객, 브랜드 톤 같은 내용을 담습니다. 사용자 프롬프트는 매 요청마다 달라지는 동적 정보입니다. 특정 기능에 대한 질문, 특정 고객 세그먼트에 대한 분석 요청 같은 내용이 여기에 해당합니다.

```
Context 설정 구조

시스템 프롬프트 (고정 컨텍스트):
  "우리 제품은 B2B SaaS 프로젝트 관리 도구입니다.
   타겟 고객은 50~200명 규모의 IT 기업입니다.
   핵심 가치는 '팀 간 투명한 업무 가시성'입니다."

사용자 프롬프트 (동적 컨텍스트):
  "최근 3개월간 온보딩 완료율이 40%에서 25%로 하락했습니다.
   주요 이탈 지점은 첫 프로젝트 생성 단계입니다.
   이 문제를 해결할 기능 개선안을 제안해 주세요."
```

고정 컨텍스트와 동적 컨텍스트를 분리하면 두 가지 이점이 있습니다. 첫째, 매번 제품 배경을 반복 입력하지 않아도 됩니다. 둘째, 동적 부분만 바꿔가며 다양한 질문을 할 수 있어 재사용성이 높아집니다. 다만 1.3.1에서 다뤘듯이 컨텍스트 창에는 토큰 수 한계가 있습니다. 불필요한 정보까지 모두 넣으면 비용이 올라가고, 정작 중요한 정보가 묻힐 수 있습니다.

세 번째 기법은 **Few-shot Examples**입니다. 원하는 입출력 패턴의 예시를 프롬프트에 직접 포함시키는 방식입니다. 모델은 제공된 예시의 형식과 스타일을 모방하여 응답을 생성합니다.

예시의 수에 따라 세 가지로 구분합니다. 먼저 예시를 하나도 제공하지 않는 경우를 Zero-shot이라 합니다. 모델이 사전 학습된 지식만으로 응답해야 하므로, 형식이나 스타일을 세밀하게 제어하기 어렵습니다.

예시를 한 개 제공하면 One-shot입니다. 하나의 입출력 쌍만으로도 모델은 원하는 형식을 상당 부분 파악합니다. 비용 대비 효과가 높아 빠른 테스트에 적합합니다.

예시가 두 개 이상이면 **Few-shot**이라 합니다. 일반적으로 예시가 많을수록 출력의 정확도와 형식 일관성이 높아집니다.

```
Few-shot Examples 적용 예시

프롬프트:
  "다음 사용자 피드백을 [긍정/부정/중립]으로 분류해 주세요.

   예시 1:
   피드백: '로딩 속도가 빨라서 좋아요'
   분류: 긍정

   예시 2:
   피드백: '결제 과정에서 오류가 자주 발생합니다'
   분류: 부정

   예시 3:
   피드백: '새 기능이 추가됐군요'
   분류: 중립

   이제 아래 피드백을 분류해 주세요.
   피드백: '앱이 자꾸 꺼져서 업무에 지장이 큽니다'"
```

위 예시에서 세 개의 입출력 쌍을 보여준 뒤 새로운 입력을 제시했습니다. 모델은 앞선 패턴을 참조하여 "부정"이라고 분류할 가능성이 높습니다. Few-shot은 파인튜닝 없이도 새로운 태스크에 즉시 적용할 수 있다는 점에서, PM이 빠르게 프로토타이핑할 때 특히 유용합니다.

네 번째 기법은 **Chain-of-Thought(CoT)**입니다. 1.3.1에서 CoT의 기본 원리를 간략히 소개했습니다. 여기서는 PM이 실무에서 CoT를 적용하는 방법에 초점을 맞춥니다.

CoT는 복잡한 문제를 풀 때 "단계별로 생각하라"는 지시를 추가하여, 모델이 중간 추론 과정을 명시적으로 출력하게 만드는 기법입니다. 모델은 자기회귀 방식으로 토큰을 생성하므로, 중간 추론 과정이 토큰으로 출력되면 그 내용이 이후 생성의 맥락에 포함됩니다. 사람이 복잡한 계산을 종이에 적어가며 푸는 것과 같은 원리입니다. 머릿속으로만 하면 실수가 잦지만, 적어가면 정확도가 올라갑니다.

CoT는 특히 산술, 논리, 상식 추론 과제에서 정확도를 크게 높입니다. PM 실무에서는 우선순위 판단, 리스크 분석, 경쟁사 비교처럼 여러 요소를 종합해야 하는 의사결정에 적합합니다.

```
CoT 적용 예시

CoT 없이:
  "이 세 기능 중 다음 스프린트에 어떤 것을 우선 개발해야 할까요?
   A: 다크 모드, B: CSV 내보내기, C: 실시간 알림"

CoT 적용:
  "이 세 기능 중 다음 스프린트에 어떤 것을 우선 개발해야 할까요?
   A: 다크 모드, B: CSV 내보내기, C: 실시간 알림

   각 기능의 사용자 영향, 개발 난이도, 비즈니스 가치를
   단계별로 분석한 뒤 결론을 내려 주세요."
```

CoT 없이 질문하면 모델은 바로 하나를 고릅니다. CoT를 적용하면 각 기능을 기준별로 비교하는 과정을 먼저 출력합니다. 그 과정에서 논리적 근거가 드러나므로, PM이 모델의 판단을 검증하기도 쉬워집니다. 다만 1.3.1에서 다뤘듯이, CoT는 중간 과정을 토큰으로 출력하므로 비용이 증가합니다. 단순한 태스크에는 불필요한 기법입니다.

다섯 번째 기법은 **Zero-shot CoT**입니다. 앞서 다룬 CoT는 예시 안에 단계별 풀이를 포함시켜야 하는 경우가 많습니다. Zero-shot CoT는 그런 예시 없이 단 한 문구만 추가하여 같은 효과를 내는 기법입니다. 그 문구는 "Let's think step by step"입니다.

이 기법이 작동하는 과정은 두 단계로 나뉩니다. 첫 번째 단계에서 모델은 "단계별로 생각해 봅시다"라는 지시에 따라 추론 경로를 출력합니다. 두 번째 단계에서 그 추론 경로를 바탕으로 최종 답을 생성합니다.

```
Zero-shot CoT 작동 과정

+------------------------------------------+
| 1단계: 추론 경로 추출                     |
|                                          |
| 프롬프트:                                 |
|   "이 기능의 ROI를 평가해 주세요.         |
|    Let's think step by step."            |
|                                          |
| 모델 출력:                                |
|   "먼저 예상 사용자 수를 추정합니다...    |
|    다음으로 개발 비용을 산출합니다...      |
|    그 다음 매출 기여분을 계산합니다..."    |
+------------------------------------------+
              |
              v
+------------------------------------------+
| 2단계: 최종 답 생성                       |
|                                          |
| 모델 출력:                                |
|   "위 분석을 종합하면, 이 기능의          |
|    예상 ROI는 약 2.3배입니다."            |
+------------------------------------------+
```

Zero-shot CoT는 태스크별로 예시를 준비할 필요가 없다는 것이 장점입니다. 산술, 기호 추론, 논리 추론 등 다양한 유형의 문제에 동일한 문구를 재사용할 수 있습니다. 다만 모델 규모에 따라 효과가 다릅니다. 소형 모델에서는 효과가 거의 나타나지 않고, 대형 모델에서만 뚜렷한 성능 향상이 나타납니다. 이는 단계적 추론 능력이 충분히 큰 모델에서만 나타나는 특성이기 때문입니다.

여섯 번째 기법은 **Self-Consistency**입니다. 동일한 프롬프트로 여러 번 응답을 생성한 뒤, 가장 많이 등장하는 답을 최종 답으로 채택하는 방식입니다. 복잡한 문제는 여러 추론 경로를 통해 같은 정답에 도달할 수 있다는 직관에 기반합니다.

이 기법이 필요한 이유를 설명하겠습니다. CoT는 한 번의 호출에서 하나의 추론 경로만 생성합니다. 그 경로가 잘못되면 최종 답도 틀립니다. Self-Consistency는 여러 경로를 샘플링하여 이 위험을 줄입니다. 다수결로 답을 정하므로, 개별 경로의 오류가 전체 결과에 미치는 영향이 줄어듭니다.

```
Self-Consistency 작동 과정

동일한 프롬프트를 5회 호출

  경로 1 --> 답: A
  경로 2 --> 답: B
  경로 3 --> 답: A
  경로 4 --> 답: A
  경로 5 --> 답: C

다수결: A가 3회로 최다 --> 최종 답: A
```

실무에서는 5개에서 10개 정도의 응답을 샘플링하는 것이 비용 대비 효과가 좋습니다. 10개 응답을 생성하면 토큰 소비와 지연시간이 10배 늘어나므로, 모든 요청에 적용하기보다는 중요도 높은 판단에만 선택적으로 사용합니다. 예를 들어 기능 우선순위 결정이나 리스크 등급 분류처럼, 한 번의 잘못된 판단이 큰 영향을 미치는 상황에 적합합니다.

일곱 번째 기법은 **Output Format 지정**입니다. 모델에게 응답의 형식을 명시적으로 요청하는 방식입니다. 표, JSON, 마크다운 리스트, 특정 문서 템플릿 등 원하는 형식을 프롬프트에 직접 지정합니다.

이 기법이 중요한 이유는 후처리 비용에 있습니다. 모델이 자유 형식으로 답하면, PM이 그 답을 다시 정리하거나 다른 도구에 입력할 수 있는 형태로 변환해야 합니다. 처음부터 원하는 형식을 지정하면 이 과정이 생략됩니다.

```
Output Format 지정 예시

형식 미지정:
  "경쟁사 A, B, C의 가격 정책을 비교해 주세요."

형식 지정:
  "경쟁사 A, B, C의 가격 정책을 비교해 주세요.
   출력 형식: 아래 열을 가진 표로 작성해 주세요.
   | 경쟁사 | 무료 플랜 | 유료 플랜 시작가 | 엔터프라이즈 |"
```

형식 미지정 프롬프트는 문단형 서술을 생성할 수도 있고, 목록을 생성할 수도 있습니다. 매번 형식이 달라질 수 있습니다. 형식 지정 프롬프트는 일관된 표 형태의 출력을 생성합니다. PRD 초안, 사용자 스토리, 회의록 템플릿 등 PM이 반복적으로 사용하는 문서 형식을 미리 지정해 두면 업무 효율이 높아집니다.

Output Format은 단순히 시각적 형식만을 의미하지 않습니다. 독자 대상을 지정하는 것도 출력 스타일을 제어하는 방법입니다. "초등학생에게 설명하듯이 작성하라"는 지시는 어휘 수준과 문장 복잡도를 낮추는 효과를 냅니다. "경영진 보고용으로 작성하라"는 지시는 핵심 수치와 의사결정 포인트를 앞에 배치하는 효과를 냅니다.

여덟 번째 기법은 **Negative Prompting**입니다. 원하지 않는 출력이나 행동을 명시적으로 제외하도록 지시하는 방식입니다. "개인 의견은 제외하고 작성하라", "기술 용어 없이 설명하라" 같은 형태로 사용합니다.

Negative Prompting을 사용할 때 반드시 알아야 할 점이 있습니다. LLM은 부정 지시를 처리하는 데 구조적 약점이 있습니다. "욕설을 사용하지 마라"처럼 부정형으로 지시하면, 모델이 그 지시를 무시하거나 오히려 역효과를 내는 경우가 관찰됩니다. 모델 규모가 커져도 이 문제가 반드시 개선되지는 않습니다.

이 문제를 해결하는 실무적 방법은 핵심 제약을 긍정형으로 바꿔 표현하는 것입니다.

```
Negative Prompting 긍정형 전환 예시

부정형 (효과 불안정):
  "욕설을 사용하지 마라."
  "기술 용어를 쓰지 마라."

긍정형 (효과 안정):
  "항상 정중하고 전문적인 어조로 작성하라."
  "비전문가도 이해할 수 있는 일상 언어로 설명하라."
```

부정형 지시를 완전히 배제할 필요는 없습니다. 보조적으로 사용하되, 핵심 제약은 반드시 긍정형 지시로 병행하는 것이 실무 원칙입니다. 예를 들어 "기술 용어를 쓰지 마라"와 "비전문가도 이해할 수 있는 일상 언어로 설명하라"를 함께 넣으면, 어느 한쪽이 무시되더라도 다른 쪽이 제약을 유지합니다.

지금까지 다룬 8가지 기법을 실무에서는 단독으로 쓰기보다 조합하여 사용합니다. 하나의 프롬프트 안에 Role, Context, Output Format, CoT를 함께 배치하는 것이 일반적입니다. 아래는 조합 예시입니다.

```
8가지 기법 조합 프롬프트 구조

[Role]
  Act as a senior product manager with B2B SaaS experience.

[Context]
  우리 제품은 프로젝트 관리 도구입니다.
  최근 3개월간 월간 활성 사용자가 15% 감소했습니다.
  주요 이탈 구간은 온보딩 7일 이내입니다.

[Task + CoT]
  이탈 원인을 분석하고 개선안을 제안해 주세요.
  각 원인과 개선안을 단계별로 분석해 주세요.

[Output Format]
  출력 형식: 아래 열을 가진 표로 작성해 주세요.
  | 이탈 원인 | 근거 | 개선안 | 예상 효과 |

[Negative Prompting]
  경쟁사 이름은 언급하지 마세요.
  항상 데이터 기반 근거를 포함하세요.
```

위 구조에서 Role은 모델의 관점을 설정하고, Context는 배경 정보를 제공합니다. CoT는 단계별 분석을 유도하고, Output Format은 결과의 형식을 고정합니다. Negative Prompting은 원치 않는 요소를 제외합니다. 이처럼 기법을 조합하면 각 기법의 효과가 상호 보완됩니다.

정리하면, 8가지 프롬프트 기법은 각각 모델의 응답을 다른 방식으로 제어합니다. Role 프롬프트는 모델의 관점을 설정하고, Context 설정은 배경 정보를 제한합니다. Few-shot Examples는 입출력 패턴을 시범으로 보여주고, CoT와 Zero-shot CoT는 단계적 추론을 유도합니다. Self-Consistency는 다수결로 답의 신뢰성을 높이고, Output Format 지정은 결과의 형식을 고정합니다. Negative Prompting은 원치 않는 요소를 배제하되, 긍정형 지시와 병행해야 안정적입니다. PM은 태스크의 성격에 따라 이 기법들을 선택하고 조합하여 프롬프트를 설계합니다.

다음 단원인 2.1.2에서는 프롬프트 버전 관리와 A/B 테스트를 다룹니다. 프롬프트를 체계적으로 관리하고, 어떤 프롬프트가 더 나은 결과를 내는지를 실험으로 검증하는 방법을 살펴봅니다.

이 단원을 마치면 Role, Context, Few-shot Examples, Chain-of-Thought, Zero-shot CoT, Self-Consistency, Output Format 지정, Negative Prompting 등 8가지 프롬프트 기법을 설명하고, 상황에 맞게 적용할 수 있습니다.
