# 1.3.1 PM을 위한 LLM 기초 — 토큰, 컨텍스트 창, 추론 방식

1.1.3에서 확률적 출력의 원리를 다루면서, LLM이 토큰 단위로 텍스트를 생성하고 확률 분포에서 샘플링한다는 점을 살펴봤습니다. 그때는 "왜 출력이 매번 달라지는가"에 초점을 맞췄습니다. 이번 단원에서는 한 걸음 더 들어갑니다. 토큰이 어떻게 만들어지는지, 모델이 한 번에 처리할 수 있는 토큰의 한계가 무엇인지를 살펴봅니다. 그리고 모델이 답을 생성하는 두 가지 방식이 비용과 품질에 어떤 영향을 미치는지를 다룹니다. 이 세 가지를 이해하면 모델 선택, 프롬프트 설계, 비용 추정에서 엔지니어링 팀과 구체적인 숫자로 대화할 수 있습니다.

LLM을 활용하는 제품을 기획할 때, PM이 가장 먼저 부딪히는 질문은 "이 기능을 운영하면 비용이 얼마나 드는가"입니다. 그 질문에 답하려면 비용이 어떤 단위로 청구되는지부터 알아야 합니다. LLM API의 과금 단위는 토큰입니다. 토큰이 무엇인지, 같은 문장이 왜 모델마다 다른 수의 토큰으로 쪼개지는지를 이해하는 것이 출발점입니다.

**토큰(token)**은 LLM이 텍스트를 처리하는 최소 단위입니다. 사람은 문장을 단어나 글자 단위로 읽지만, LLM은 토큰 단위로 읽고 씁니다. 토큰은 단어 전체일 수도 있고, 단어의 일부일 수도 있습니다. 예를 들어 영어 단어 "unhappiness"는 세 개의 토큰 "un", "happy", "ness"로 분해될 수 있습니다. 자주 쓰이는 짧은 단어 "the"나 "is"는 그 자체가 하나의 토큰입니다. 반면 흔하지 않은 긴 단어는 여러 조각으로 나뉩니다.

그렇다면 LLM은 어떤 기준으로 텍스트를 토큰으로 나눌까요. 이 과정을 **토큰화(tokenization)**라고 합니다. 가장 널리 쓰이는 알고리즘이 **BPE(Byte Pair Encoding)**입니다.

BPE의 핵심 아이디어는 단순합니다. 훈련 데이터에서 가장 자주 나란히 등장하는 두 글자(또는 두 토큰)를 찾아 하나로 합치는 과정을 반복하는 것입니다. 처음에는 알파벳 한 글자 한 글자가 각각 토큰입니다. 여기서 가장 자주 붙어 나타나는 쌍을 찾아 합칩니다. 예를 들어 "t"와 "h"가 가장 자주 연달아 나타나면 "th"라는 새 토큰을 만듭니다. 이 과정을 수천에서 수만 번 반복하면, 자주 쓰이는 단어는 통째로 하나의 토큰이 되고, 드문 단어는 여러 조각으로 남습니다. 최종적으로 만들어진 토큰 목록을 **어휘 사전(vocabulary)**이라 합니다. GPT 계열 모델의 어휘 사전은 대략 5만에서 10만 개의 토큰으로 구성됩니다.

이 방식이 PM에게 중요한 이유는 언어마다 토큰 효율이 다르기 때문입니다. BPE는 훈련 데이터에서 빈도를 기준으로 어휘 사전을 구성합니다. 영어는 훈련 데이터에 풍부하므로 효율적으로 토큰화됩니다. 영어 기준으로 약 1,000단어가 약 750개 토큰에 해당합니다. 반면 한국어, 중국어, 일본어 같은 비라틴 문자 언어는 훈련 데이터 비중이 적습니다. 같은 의미의 문장을 표현하는 데 더 많은 토큰이 필요합니다. 토큰 수가 늘어나면 API 호출 비용이 그만큼 올라갑니다. 다국어 서비스를 기획한다면 언어별 토큰 효율 차이를 비용 추정에 반영해야 합니다.

토큰화를 이해했으니, 다음 질문으로 넘어갑니다. 모델이 한 번에 처리할 수 있는 토큰의 양에는 한계가 있을까요. 있습니다. 그 한계를 **컨텍스트 창(context window)**이라 합니다.

컨텍스트 창은 모델이 한 번의 추론에서 읽고 쓸 수 있는 최대 토큰 수입니다. 이 창 안에는 시스템 프롬프트, 대화 이력, 사용자가 입력한 텍스트, RAG로 검색한 문서, 그리고 모델이 생성하는 응답까지 모두 포함됩니다. 컨텍스트 창을 하나의 책상이라고 생각하면 됩니다. 책상 위에 올려놓을 수 있는 서류의 양이 정해져 있고, 그 양을 넘으면 오래된 서류부터 밀려나거나 아예 작업이 중단됩니다.

2025년 기준으로 주요 모델의 컨텍스트 창은 다음과 같습니다.

```
모델별 컨텍스트 창 비교

+---------------------+------------------+
| 모델                | 컨텍스트 창       |
+---------------------+------------------+
| GPT-4o              | 약 128k 토큰     |
| Claude Sonnet       | 약 200k 토큰     |
| Gemini 2.5          | 최대 1,000k 토큰 |
+---------------------+------------------+
```

숫자만 보면 "창이 클수록 좋겠다"고 생각할 수 있습니다. 긴 문서를 통째로 넣을 수 있고, 대화 이력을 잘리지 않게 유지할 수 있으니까요. 하지만 컨텍스트 창의 크기는 비용, 지연시간과 직결됩니다.

LLM API의 비용은 입력 토큰과 출력 토큰 수에 비례합니다. 여기서 중요한 점은 출력 토큰의 단가가 입력 토큰보다 3배에서 5배 비싸다는 것입니다. 컨텍스트 창을 가득 채워 긴 문서를 넣으면 입력 비용이 올라가고, 거기에 긴 응답을 요청하면 출력 비용까지 올라갑니다. 또한 토큰 수가 많아질수록 모델의 처리 시간도 길어져 사용자 체감 지연시간이 늘어납니다.

모델 등급에 따른 비용 차이도 큽니다. 소형 모델의 입력 비용은 백만 토큰당 0.15~0.60달러 수준입니다. 최고 사양 모델은 백만 토큰당 2.50~15달러 수준입니다. 모델 선택 하나만으로 비용이 20배에서 100배까지 차이 날 수 있습니다.

PM이 이 구조를 알아야 하는 이유는 "모든 정보를 컨텍스트 창에 다 넣으면 된다"는 발상이 비용 폭발로 이어지기 때문입니다. 제품 설계 초기부터 "사용자 한 번의 요청에 허용 가능한 비용(cost-per-action)"을 정의해야 합니다. 그 예산 안에서 어떤 정보를 컨텍스트에 넣고 어떤 정보를 생략할지를 결정해야 합니다.

비용을 줄이는 대표적인 전략 네 가지를 정리합니다.

```
비용 최적화 전략과 절감 효과

+-----------------------+----------------------------+
| 전략                  | 기대 절감 효과              |
+-----------------------+----------------------------+
| 프롬프트 압축         | 불필요한 토큰 제거,         |
|                       | 20~40% 비용 절감            |
+-----------------------+----------------------------+
| 시맨틱 캐싱           | 유사한 질문의 응답을 재활용, |
|                       | 30~60% 캐시 히트율          |
+-----------------------+----------------------------+
| 배치 처리             | 비실시간 작업을 모아 처리,  |
|                       | 최대 50% 할인               |
+-----------------------+----------------------------+
| 모델 라우팅           | 단순 태스크에 소형 모델,    |
|                       | 복잡한 태스크에만 대형 모델  |
|                       | 배정하여 20~100배 차이 활용 |
+-----------------------+----------------------------+
```

프롬프트 압축은 같은 의미를 더 적은 토큰으로 전달하는 기법입니다. 앞서 살펴본 BPE 토큰화에서, 토큰 단위와 의미 단위가 정확히 일치하지 않기 때문에 의미 손실 없이 불필요한 토큰을 줄일 여지가 있습니다. 실제로 GPT-4o 기준 평균 19.4%의 토큰을 줄이면서도 응답 품질을 유지한 사례가 보고되어 있습니다.

시맨틱 캐싱은 이전에 처리한 질문과 유사한 질문이 들어왔을 때, 모델을 다시 호출하지 않고 저장된 응답을 반환하는 방식입니다. 고객 문의처럼 반복되는 질문이 많은 서비스에서 효과가 큽니다.

모델 라우팅은 태스크의 복잡도에 따라 다른 모델을 배정하는 방식입니다. "주문 상태를 알려줘"처럼 단순한 요청은 소형 모델이 처리합니다. "이 계약서를 분석해서 위험 조항을 찾아줘"처럼 복잡한 요청만 대형 모델에 보냅니다. 이 전략을 적용하면 전체 시스템의 품질을 유지하면서도 평균 비용을 크게 낮출 수 있습니다.

이제 세 번째 주제로 넘어갑니다. LLM이 답을 생성하는 방식, 즉 추론 방식입니다.

1.1.3에서 LLM이 한 번에 한 토큰씩 생성한다고 설명했습니다. 이 방식을 **자기회귀(autoregressive) 생성**이라 합니다. "자기회귀"라는 이름은 "자기 자신이 만든 출력을 다시 입력으로 사용한다"는 뜻입니다. 첫 번째 토큰을 생성하면, 그 토큰이 두 번째 토큰을 생성할 때의 입력에 포함됩니다. 두 번째 토큰이 생성되면, 첫 번째와 두 번째 토큰이 모두 세 번째 토큰의 입력에 포함됩니다. 이 과정이 종료 조건에 도달할 때까지 반복됩니다.

종료 조건은 두 가지입니다. 하나는 모델이 **EOS(End of Sequence) 토큰**을 생성하는 것입니다. EOS 토큰은 "이 문장은 여기서 끝"이라는 신호 역할을 하는 특수 토큰입니다. 다른 하나는 API 호출 시 설정한 **max_tokens** 값에 도달하는 것입니다. max_tokens는 모델이 생성할 수 있는 최대 토큰 수를 제한하는 파라미터입니다.

이 방식의 실무적 함의는 응답 길이와 생성 시간의 관계입니다. 토큰을 하나씩 순서대로 생성하므로, 응답이 길어질수록 생성 시간이 거의 비례해서 늘어납니다. 100 토큰짜리 응답보다 1,000 토큰짜리 응답이 약 10배 오래 걸립니다. PM이 "응답 시간 2초 이내"라는 요구사항을 정한다면, 그 시간 안에 생성 가능한 토큰 수를 역산하여 응답 길이를 설계해야 합니다.

```
자기회귀 생성 과정

입력: "오늘 회의 안건은"

  [오늘 회의 안건은] ---> 모델 ---> "세" (1번째 토큰)
                                      |
  [오늘 회의 안건은 세] ---> 모델 ---> "가지" (2번째 토큰)
                                        |
  [오늘 회의 안건은 세 가지] ---> 모델 ---> "입니다" (3번째 토큰)
                                             |
  [오늘 회의 안건은 세 가지 입니다] ---> 모델 ---> [EOS] (종료)

* 매 단계마다 이전에 생성한 모든 토큰이 입력에 포함됩니다.
* 토큰 수가 늘수록 생성 시간이 선형으로 증가합니다.
```

자기회귀 생성은 모든 LLM의 기본 동작입니다. 하지만 복잡한 문제를 풀 때는 이 기본 방식만으로 부족한 경우가 있습니다. "대한민국의 수도는?"처럼 단순한 질문은 바로 "서울"이라는 토큰을 생성하면 됩니다. 하지만 "이 계약서에서 갑에게 불리한 조항을 세 가지 찾아 근거와 함께 설명하라"처럼 여러 단계의 사고가 필요한 문제는 다릅니다. 바로 답을 생성하면 중요한 단계를 건너뛰거나 논리적 오류를 범할 수 있습니다.

이 문제를 해결하기 위해 등장한 기법이 **CoT(Chain-of-Thought)**입니다. CoT는 "사고의 연쇄"라는 뜻으로, 최종 답을 바로 내놓는 대신 중간 추론 과정을 토큰으로 하나하나 출력하게 만드는 방식입니다. "단계별로 생각하라(think step-by-step)"라는 지시를 프롬프트에 추가합니다. 그러면 모델은 문제를 작은 단계로 나누어 각 단계의 결론을 먼저 생성하고, 그 위에서 다음 단계를 진행합니다.

CoT가 효과적인 이유는 자기회귀 생성의 특성에 있습니다. 모델은 이미 생성한 토큰을 다음 토큰 예측의 입력으로 사용합니다. 중간 추론 과정을 토큰으로 명시적으로 출력하면, 그 추론 결과가 이후 생성의 맥락에 포함됩니다. 마치 사람이 복잡한 계산을 할 때 중간 과정을 종이에 적어가며 푸는 것과 비슷합니다. 머릿속으로만 하면 실수가 잦지만, 적어가면 정확도가 올라갑니다.

하지만 CoT에는 비용이 따릅니다. 중간 추론 과정을 토큰으로 출력하므로, 같은 질문에 대해 단순 생성보다 훨씬 많은 토큰을 소비합니다. 추론 모드를 사용하면 쿼리당 비용이 몇 배에서 수십 배까지 높아질 수 있습니다. 토큰 단가가 매년 하락하고 있지만, 추론 모델이 생성하는 토큰 양이 크게 늘어나기 때문에 총비용은 상쇄되거나 오히려 증가할 수 있습니다.

PM이 이 차이를 알아야 하는 이유는 태스크마다 적합한 추론 방식이 다르기 때문입니다. 고객 인사말 생성, 간단한 분류, 짧은 요약처럼 단순한 태스크에 CoT를 적용하면 불필요한 비용만 늘어납니다. 반면 법률 문서 분석, 재무 데이터 해석, 다단계 의사결정처럼 정확도가 핵심인 태스크는 다릅니다. CoT가 오류를 줄여 오히려 전체 비용을 낮출 수 있습니다. 잘못된 답을 내놓아 재처리하는 비용이 CoT 추가 토큰 비용보다 클 수 있기 때문입니다.

이제 모델 크기와 성능의 관계를 살펴봅니다. 모델의 크기는 **파라미터(parameter)** 수로 표현합니다. 파라미터는 모델이 학습 과정에서 조정한 가중치(weight)로, 모델이 축적한 지식과 능력이 저장된 형태입니다. GPT-4 급 모델은 수천억 개 이상의 파라미터를 가지고 있고, 소형 모델은 수십억 개 수준입니다.

일반적으로 파라미터 수가 많을수록 복잡한 추론, 긴 맥락 이해, 다양한 도메인 지식에서 성능이 높습니다. 그러나 추론에 필요한 메모리, 처리 시간, API 단가도 함께 올라갑니다. 이것이 모델 크기와 성능의 트레이드오프입니다.

그런데 항상 큰 모델이 작은 모델보다 나은 것은 아닙니다. 특정 도메인에 맞춰 **파인튜닝(fine-tuning)**한 소형 모델이 범용 대형 모델을 능가하는 사례가 보고되어 있습니다. 파인튜닝이란 이미 학습된 모델에 특정 분야의 데이터를 추가로 학습시켜 그 분야의 성능을 높이는 기법입니다. 예를 들어 금융 데이터 감성 분석에서, 소형 모델을 파인튜닝하자 정확도가 34%에서 85%로 올라간 사례가 있습니다. 100개 정도의 예시만으로도 정확도가 25%p 향상된 경우도 있습니다.

이것이 PM에게 시사하는 바는 "가장 비싼 모델을 쓰면 된다"는 접근이 최선이 아닐 수 있다는 것입니다. 작업의 성격에 따라 소형 모델 + 파인튜닝 조합이 대형 범용 모델보다 더 빠르고, 더 저렴하며, 더 정확할 수 있습니다.

모델 크기를 줄이지 않고도 추론 효율을 높이는 방법도 있습니다. **양자화(quantization)**는 모델의 가중치를 표현하는 숫자의 정밀도를 낮추는 기법입니다. 원래 16비트(FP16)로 저장된 가중치를 8비트(INT8)나 4비트(INT4)로 변환합니다. 이렇게 하면 메모리 사용량과 연산량을 50~75%까지 줄일 수 있습니다. 정밀도를 낮추면 약간의 성능 저하가 있을 수 있지만, 많은 경우 실용적 수준의 품질을 유지합니다.

마지막으로, 지금까지 다룬 토큰, 컨텍스트 창, 추론 방식, 모델 크기에 대해 PM이 갖추어야 할 이해의 깊이를 정리합니다.

PM에게 요구되는 기술 이해는 구현(implementation) 수준이 아니라 개념적(conceptual) 수준입니다. BPE 알고리즘을 직접 코딩할 필요는 없지만, 토큰 수가 비용에 어떻게 영향을 미치는지는 설명할 수 있어야 합니다. 트랜스포머 아키텍처의 수식을 외울 필요는 없지만, 컨텍스트 창의 한계가 제품 설계에 어떤 제약을 주는지는 판단할 수 있어야 합니다.

PM의 본질적 역할은 번역자입니다. AI의 가능성과 한계를 비즈니스 언어로 옮기고, 비즈니스의 필요를 기술 가능성의 언어로 옮기는 사람입니다. 기술을 이해하는 목적은 직접 구현하기 위해서가 아닙니다. 엔지니어링 팀과 제품 전략을 정렬하고, 비용, 품질, 일정 간의 트레이드오프를 판단하기 위해서입니다.

기획 의사결정에 필요한 수준에 도달했는지 스스로 점검할 수 있는 질문 세 가지가 있습니다. 첫째, 이 기능에 CoT가 필요한가입니다. 둘째, 컨텍스트 창에 정보를 얼마나 넣을 것인가입니다. 셋째, 소형 모델로 라우팅할 태스크는 무엇인가입니다. 이 질문에 근거를 들어 답할 수 있다면 준비가 된 것입니다. 엔지니어링 팀과 신뢰 있는 논의를 시작할 수 있습니다.

정리하면, LLM은 BPE 알고리즘으로 텍스트를 토큰 단위로 분해합니다. 컨텍스트 창이라는 한정된 공간 안에서 자기회귀 방식으로 토큰을 하나씩 생성합니다. 컨텍스트 창의 크기는 처리 가능한 정보량을 결정하지만 비용과 지연시간도 함께 올립니다. CoT 추론은 복잡한 태스크의 정확도를 높이지만 토큰 소비가 몇 배로 늘어납니다. 모델 크기가 클수록 범용 성능은 높아지나, 도메인 특화 파인튜닝으로 소형 모델이 대형 모델을 능가할 수도 있습니다. PM은 이 네 가지 축의 트레이드오프를 이해하고, 태스크별로 적합한 조합을 판단할 수 있어야 합니다.

이 단원을 마치면 LLM의 토큰화, 컨텍스트 창, 추론 방식을 이해하고, 기획 의사결정에 필요한 수준으로 설명할 수 있습니다.
