# 1.2.2 AI PM 3가지 아키타입과 역할 분업

1.2.1에서 전통 PM과 AI PM의 차이를 살펴보았습니다. AI PM은 확률적 시스템, 데이터 의존성, 기획 대상의 자율성이라는 세 가지 조건 때문에 전통 PM보다 넓은 책임을 지게 됩니다. 그런데 현실에서 한 사람이 이 모든 책임을 동일한 깊이로 수행하기는 어렵습니다. 사용자 경험을 깎아 내리는 일과 인프라 비용을 최적화하는 일은 요구하는 전문성의 방향이 다르기 때문입니다. 이 단원에서는 AI PM이라는 역할 안에 존재하는 세 가지 실천 패턴, 즉 아키타입을 구분하고, 조직 규모에 따라 이 아키타입이 어떻게 분업되는지를 살펴봅니다.

**아키타입(archetype)**이란 같은 직함 아래에서도 핵심 관심사와 역량의 초점이 뚜렷하게 갈리는 유형을 뜻합니다. AI PM의 아키타입은 세 가지로 나뉩니다. 첫째는 최종 사용자 경험에 집중하는 **Core Product PM**, 둘째는 AI 인프라를 설계하고 운영하는 **Platform Builder**, 셋째는 AI를 자신의 업무 도구로 직접 활용하는 **AI-Powered PM**입니다. 이 세 아키타입은 서로 배타적인 직무가 아니라, 한 사람 안에서 상황에 따라 비중이 달라지는 역할 스펙트럼에 가깝습니다. 다만 각각이 요구하는 역량의 방향이 다르므로 하나씩 분리해서 살펴볼 필요가 있습니다.

Core Product PM부터 알아보겠습니다. Core Product PM은 "고객의 목소리"를 대변하는 사람입니다. AI 기능이 사용자 문제를 실제로 해결하는지, 기존 제품 경험에 자연스럽게 녹아드는지를 책임집니다. 핵심 관심사는 AI 기능의 **표현 형태(form factor)**와 **포장(packaging)**입니다. 표현 형태란 AI 기능이 사용자에게 어떤 모습으로 보이는지를 뜻합니다. 예를 들어, 같은 텍스트 요약 기능이라도 버튼을 눌러 결과를 받는 형태, 문서를 열면 자동으로 요약이 붙는 형태, 채팅 인터페이스로 대화하며 요약을 받는 형태는 사용자 경험이 전혀 다릅니다. 포장이란 이 기능을 어떤 맥락에서 어떤 이름으로 제공할지를 결정하는 것입니다. Core Product PM은 이 선택을 사용자 리서치에 기반하여 내립니다.

Core Product PM에게 요구되는 역량은 기술적 구현 깊이보다 사용자 공감과 UX 설계에 치우쳐 있습니다. 1.2.1에서 다룬 것처럼 AI 시스템의 출력은 비결정론적이므로, 전통 소프트웨어와 같은 확정적 화면 설계가 통하지 않습니다. Core Product PM은 이 비결정론적 출력에 맞는 UX를 설계해야 합니다. 또한 AI 출력의 품질을 정성적으로 판단하는 능력이 필요합니다. "이 답변이 너무 로봇 같다", "사용자가 원하는 수준의 자연스러움이 아니다"와 같은 피드백을 구체적인 평가 기준으로 전환하는 것이 이 아키타입의 고유한 역할입니다. 1.2.1에서 언급한 다차원 평가 지표, 즉 정확성, 일관성, 안전성, 유용성 같은 기준을 설계하는 일이 여기에 해당합니다.

다음은 Platform Builder입니다. Platform Builder는 AI 인프라를 설계하고 운영하는 사람입니다. 확장 가능한 기반을 구축하는 것이 이 아키타입의 핵심 관심사입니다. 구체적으로는 세 가지 의사결정을 주로 다룹니다. 첫째, 어떤 모델을 쓸 것인가. 둘째, 언제 더 저렴한 모델로 전환할 것인가. 셋째, 캐싱 전략을 어떻게 적용할 것인가.

이 세 가지 의사결정이 왜 중요한지를 이해하려면 LLM 비용 구조를 알아야 합니다. LLM 기반 제품은 API 호출 한 건마다 비용이 발생합니다. 사용자가 늘어나면 비용이 선형적으로, 때로는 그 이상으로 증가합니다. 전통 소프트웨어에서 서버 비용은 제품이 성장해도 비교적 예측 가능한 범위에서 관리됩니다. 하지만 LLM 비용은 제품의 경제성에 직접 영향을 줍니다. 응답 한 건의 비용이 1원이라도 하루 100만 건이면 일 100만 원이 됩니다. 이런 상황에서 정확도를 조금 양보하더라도 비용이 절반인 모델로 전환하는 것이 사업적으로 합리적인 선택일 수 있습니다. Platform Builder는 이런 비용-성능 트레이드오프를 판단합니다.

Platform Builder에게 요구되는 역량은 인프라와 운영 쪽에 집중됩니다. MLOps와 LLMOps 파이프라인에 대한 감각, 모델 평가 메트릭(정확도, F1 스코어, 지연시간, 환각률)에 대한 이해, 데이터 파이프라인 구조에 대한 지식이 필요합니다. Core Product PM이 "사용자가 만족하는가"를 묻는다면, Platform Builder는 "이 품질을 유지하면서 비용과 속도를 감당할 수 있는가"를 묻습니다.

세 번째 아키타입은 AI-Powered PM입니다. 이 아키타입은 앞의 두 아키타입과 성격이 다릅니다. Core Product PM과 Platform Builder가 "AI 제품을 기획하는 PM"이라면, AI-Powered PM은 "AI를 도구로 활용하는 PM"입니다. 프롬프팅, 빠른 프로토타이핑, 경우에 따라 코드 작성까지 직접 수행하며 제품, 엔지니어링, 디자인의 경계를 허뭅니다.

AI-Powered PM이 등장한 배경을 이해하려면 노코드/로우코드 도구의 확산을 살펴봐야 합니다. Dify, n8n, Lovable, Cursor, Replit 같은 도구가 보급되면서, 코드를 본격적으로 작성하지 않는 PM도 에이전트 워크플로우를 직접 설계하고 프로토타입을 만들 수 있게 되었습니다. 예를 들어, PM이 고객 문의 자동 분류 에이전트의 아이디어를 떠올렸다고 합시다. 과거에는 이 아이디어를 PRD로 작성하고 엔지니어에게 전달해야 했습니다. AI-Powered PM은 n8n 같은 도구로 직접 프로토타입을 만들어 당일 내부 테스트를 돌려 볼 수 있습니다. 아이디어에서 검증까지의 거리가 크게 줄어든 것입니다.

AI-Powered PM에게 요구되는 역량은 도구 활용과 프롬프팅에 집중됩니다. **프롬프트 엔지니어링** 역량이 핵심입니다. 시스템 프롬프트를 설계하고, 퓨샷 예제를 구성하며, 온도(temperature) 파라미터가 출력의 다양성에 어떤 영향을 미치는지 이해해야 합니다. 여기에 더해 **컨텍스트 엔지니어링** 역량도 필요합니다. 컨텍스트 엔지니어링이란 모델이 좋은 결정을 내리도록 올바른 정보를 올바른 형태로 전달하는 기술입니다. 같은 질문이라도 어떤 배경 정보를 함께 제공하느냐에 따라 모델의 응답 품질이 크게 달라집니다. AI-Powered PM은 이 차이를 실무적으로 체감하며 제품에 적용합니다.

세 아키타입을 나란히 놓고 보면, 기술 깊이의 방향이 서로 다르다는 점이 드러납니다. Core Product PM은 사용자 공감과 UX 설계에 깊이를 둡니다. Platform Builder는 인프라, 운영, 비용에 깊이를 둡니다. AI-Powered PM은 도구 활용, 프로토타이핑, 프롬프팅에 깊이를 둡니다. 그러나 세 아키타입 모두가 공유하는 공통 기술 기준선이 있습니다.

```
세 아키타입의 역량 구조

                    공통 기술 기준선
+-----------------------------------------------------------+
| ML 기본 개념 이해 (지도/비지도 학습, 정확도-편향 트레이드오프) |
| 데이터 유창성 (파이프라인 이해, 데이터 분석가와 협력)         |
| AI 라이프사이클 인식 (모델 학습의 복잡성, 가능/불가능 구분)   |
+-----------------------------------------------------------+
        |                   |                   |
        v                   v                   v
+---------------+  +----------------+  +------------------+
| Core Product  |  | Platform       |  | AI-Powered       |
| PM            |  | Builder        |  | PM               |
+---------------+  +----------------+  +------------------+
| 사용자 공감   |  | 비용-성능      |  | 프롬프트         |
| UX 설계       |  | 트레이드오프   |  | 엔지니어링       |
| 정성적 품질   |  | MLOps/LLMOps   |  | 노코드           |
| 피드백        |  | 모델 평가      |  | 프로토타이핑     |
| 평가 기준     |  | 메트릭         |  | 컨텍스트         |
| 설계          |  | 데이터         |  | 엔지니어링       |
|               |  | 파이프라인     |  |                  |
+---------------+  +----------------+  +------------------+
```

공통 기술 기준선은 1.2.1에서 다룬 AI 리터러시의 구체적 구성 요소에 해당합니다. ML의 기본 개념을 이해하고, 데이터 파이프라인의 구조를 파악하며, 모델 학습이 얼마나 복잡한 과정인지, 현재 기술로 가능한 것과 불가능한 것을 구분할 수 있어야 합니다. 이 기준선을 갖추지 못하면 어떤 아키타입을 맡더라도 엔지니어링 팀과 신뢰 있는 대화가 어렵습니다.

이제 조직 규모에 따라 이 세 아키타입이 실제로 어떻게 배치되는지를 살펴봅니다. 핵심 원리는 간단합니다. 조직이 작을수록 한 사람이 여러 아키타입을 겸하고, 조직이 클수록 아키타입별로 역할이 분리됩니다.

초기 스타트업(대략 200명 미만)에서는 한 명의 PM이 세 아키타입을 모두 담당하는 경우가 많습니다. 제품-시장 적합성(PMF)을 찾기 전에는 CTO나 CPO가 PM 역할까지 겸임하기도 합니다. 이 단계에서 가장 중요한 역량은 AI-Powered PM의 것입니다. 빠르게 프로토타입을 만들고, 사용자 반응을 확인하고, 방향을 수정하는 속도가 생존을 결정하기 때문입니다. 인프라를 정교하게 설계할 여유보다 아이디어를 빠르게 검증하는 능력이 우선입니다.

중간 규모 기업(200명에서 5,000명 사이)에서는 역할이 분화되기 시작합니다. Core Product PM과 Platform Builder가 분리되는 경향이 나타납니다. 제품 전략과 로드맵을 소유하면서 여러 팀과의 협업을 주도하는 미드레벨 PM이 이 단계에서 주로 채용됩니다. 한 사람이 사용자 경험과 인프라 비용을 동시에 깊이 다루기 어려워지는 지점이기 때문입니다.

대기업(10,000명 이상)에서는 세 아키타입이 명확히 분리됩니다. 시니어 PM은 조직 전체의 AI 비전과 다년 플랫폼 전략을 정의하는 역할을 맡습니다. 이 규모에서는 AI 조직을 어떻게 구성할 것인가라는 문제도 함께 등장합니다. 엔터프라이즈에서 흔히 볼 수 있는 조직 모델은 세 가지입니다. 첫째, 중앙집중형 CoE(Center of Excellence)로, AI 도입 초기 단계에서 전문성을 한곳에 모아 표준을 수립하는 방식입니다. 둘째, 완전 분산형으로, 각 사업부가 자체 AI 팀을 운영하여 속도를 최대화하는 방식입니다. 셋째, 허브-앤-스포크 하이브리드로, 중앙에서 공통 인프라와 가이드라인을 제공하고 각 사업부에 AI 스쿼드를 배치하는 방식입니다. 이 세 번째 모델이 가장 일반적입니다.

```
조직 규모에 따른 아키타입 분업

+--------------------+-------------------+-------------------+
| 초기 스타트업      | 중간 규모 기업    | 대기업            |
| (200명 미만)       | (200~5,000명)     | (10,000명 이상)   |
+--------------------+-------------------+-------------------+
|                    |                   |                   |
| PM 1명이           | Core Product PM과 | 세 아키타입이     |
| 세 아키타입을      | Platform Builder  | 명확히 분리       |
| 모두 수행          | 분리 시작         |                   |
|                    |                   |                   |
| 핵심 역량:         | 핵심 역량:        | 핵심 역량:        |
| AI-Powered PM      | 제품 전략 +       | 조직 AI 비전 +    |
| (빠른 검증)        | 크로스펑셔널 실행 | 다년 플랫폼 전략  |
|                    |                   |                   |
+--------------------+-------------------+-------------------+
```

세 아키타입의 분업에서 한 가지 더 짚어야 할 축이 있습니다. PM과 엔지니어 사이의 책임 경계입니다. AI 제품에서는 PM이 "무엇을(what)" 소유하고, 엔지니어가 "어떻게(how)"를 소유합니다. PM은 사용자 관점의 성공 기준을 정의하고, 평가용 골든셋을 큐레이션하며, 실패 모드의 우선순위를 매깁니다. 엔지니어는 측정 방법을 설계하고, 통계적 엄밀성을 확보하며, 확장 가능한 평가 시스템을 구축합니다. "왜 이 메트릭이 중요한가(why)"는 양쪽이 함께 소유합니다. 이 분업은 아키타입과 무관하게 모든 AI PM에게 적용되는 원칙입니다.

정리하면, AI PM이라는 역할은 Core Product PM, Platform Builder, AI-Powered PM이라는 세 가지 아키타입으로 세분화됩니다. Core Product PM은 사용자 경험과 AI 기능의 표현 형태에, Platform Builder는 인프라 비용과 모델 운영에, AI-Powered PM은 도구 활용과 빠른 프로토타이핑에 깊이를 둡니다. 세 아키타입은 ML 기본 개념, 데이터 유창성, AI 라이프사이클 인식이라는 공통 기준선을 공유하며, 조직이 작을수록 한 사람이 여러 역할을 겸하고 조직이 커질수록 역할이 분리됩니다.

다음 단원인 1.2.3에서는 AI가 자동화하는 PM 업무와 인간 고유 영역을 다룹니다. 세 아키타입의 업무 가운데 어떤 부분을 AI가 대체할 수 있고, 어떤 부분은 사람만이 수행할 수 있는지를 살펴봅니다.

이 단원을 마치면 Core Product PM, Platform Builder, AI-Powered PM 세 아키타입을 구별하고, 각각의 핵심 책임과 요구 역량을 설명할 수 있습니다.
