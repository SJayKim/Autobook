# 1.3.3 데이터 리터러시 — Goldilocks PM이 되는 법

1.3.1에서 AI PM의 역할을 정의하면서, AI 제품 개발이 비즈니스와 기술과 데이터의 교차점에서 이루어진다고 설명했습니다. 전통적인 PM이 사용자 경험을 중심에 놓았다면, AI PM은 그 자리에 데이터를 함께 놓아야 합니다. 그런데 "데이터를 이해한다"는 말은 막연합니다. SQL을 작성할 수 있어야 하는 걸까요. 통계학 학위가 필요한 걸까요. 아니면 데이터 과학자가 보여주는 차트를 읽을 수 있으면 충분한 걸까요. 이 단원에서는 AI 기획자에게 필요한 데이터 이해의 수준이 정확히 어디인지를 다룹니다. 그리고 데이터의 품질, 편향, 거버넌스가 기획 단계에서 왜 중요한지를 살펴봅니다.

AI 제품을 기획하는 PM이 데이터를 몰라서 생기는 문제부터 짚어 봅니다. 데이터 과학자들이 PM에게 가장 자주 느끼는 불만은 세 가지입니다. 첫째, PM이 올바른 질문을 던지지 못합니다. 둘째, PM이 문제를 머신러닝으로 풀 수 있는 형태로 정의하지 못합니다. 셋째, PM이 자신의 기술 이해에 대해 과잉 자신감을 가집니다. 이 세 가지 불만의 공통 원인은 하나입니다. PM에게 **데이터 리터러시**가 부족하다는 것입니다.

데이터 리터러시는 숫자를 읽는 능력이 아닙니다. AI 맥락에서 데이터 리터러시는 데이터가 수집되고 정제되고 레이블링되는 과정, 즉 **데이터 파이프라인**을 이해하는 역량입니다. 여기에 머신러닝의 기본 개념을 파악하고, 분석 결과의 의미를 판단하는 능력이 더해집니다. 핵심은 "이 문제가 머신러닝으로 풀 수 있는 문제인가"를 판단할 수 있느냐입니다. 그리고 데이터 과학팀과 대화할 때 올바른 질문을 던질 수 있느냐입니다.

데이터 파이프라인이라는 말이 처음 나왔으니 풀어서 설명합니다. 데이터 파이프라인은 원시 데이터가 모델이 사용할 수 있는 형태가 되기까지 거치는 일련의 단계입니다.

```
데이터 파이프라인 3단계

  [수집]           [정제]           [레이블링]
  여러 출처에서 --> 오류/중복을  --> 정제된 데이터에
  데이터를 모음     제거함           정답 표시를 붙임
```

데이터를 여러 출처에서 모으는 수집 단계가 있습니다. 모은 데이터에서 오류나 중복을 제거하는 정제 단계가 있습니다. 정제된 데이터에 정답 표시를 붙이는 레이블링 단계가 있습니다. 예를 들어 고객 리뷰 텍스트를 감성 분석 모델에 학습시키려면, 각 리뷰에 "긍정" 또는 "부정"이라는 레이블을 붙여야 합니다. PM이 이 흐름을 이해하면 "어떤 데이터를 모을 것인가", "레이블링 비용은 얼마인가", "데이터가 모델 학습에 충분한 양인가"를 기획 초기부터 질문할 수 있습니다.

그렇다면 PM은 이 데이터 지식을 얼마나 깊이 갖춰야 할까요. 여기서 **Goldilocks PM**이라는 개념이 등장합니다. Goldilocks는 영국 동화 '골디락스와 곰 세 마리'에서 온 표현입니다. 너무 뜨겁지도, 너무 차갑지도 않은 "딱 적당한" 상태를 가리킵니다. Goldilocks PM은 기술을 너무 깊이 파고들어 제품 감각을 잃지도 않고, 기술을 너무 모른 채로 과대광고에 휩쓸리지도 않는 PM을 뜻합니다.

Goldilocks PM의 기준선은 구현 역량이 아닙니다. 이해 역량입니다. 1.3.1에서 PM의 역할을 "비즈니스와 기술 사이의 번역자"로 정의했습니다. 번역자가 원어민 수준의 작문을 할 필요는 없지만, 상대방의 말을 정확히 알아듣고 핵심을 옮길 수 있어야 합니다. 마찬가지로 PM이 파이썬 코드를 작성할 필요는 없습니다. 하지만 모델이 왜 그런 결과를 냈는지 팀에 물을 수 있어야 합니다. 개인화 기능을 기획할 때 어떤 데이터가 필요한지 판단할 수 있어야 합니다. ML 결과물에 내재된 불확실성을 이해해야 합니다.

"비기술 PM은 위험하다"는 말이 업계에서 자주 나옵니다. 이 말의 뜻은 "코드를 못 쓰는 PM"이 위험하다는 것이 아닙니다. "제품이 어떻게 만들어지는지 신뢰 있게 말할 수 없는 PM"이 위험하다는 뜻입니다. 기술 유창성의 기준선이 영구적으로 높아졌습니다. Goldilocks PM은 그 높아진 기준선 위에 서되, 엔지니어로 변신하지는 않는 사람입니다.

구체적으로 Goldilocks PM이 이해해야 하는 영역은 네 가지입니다. 첫째, 데이터 모델링 기초입니다. 데이터가 어떤 구조로 저장되고 관계를 맺는지 이해합니다. 둘째, 데이터 인프라입니다. 데이터가 어디에 저장되고 어떻게 흐르는지 파악합니다. 셋째, 머신러닝 모델의 작동 원리입니다. 1.3.1에서 다룬 LLM 기초가 여기에 해당합니다. 넷째, 모델 평가 지표입니다. 모델의 성능을 어떤 숫자로 측정하는지, 그 숫자가 비즈니스에 무엇을 의미하는지 판단합니다.

이 네 영역 중에서 PM이 가장 큰 영향을 발휘하는 지점은 두 곳입니다. 하나는 **모델 출력(Model Output)** 영역입니다. 모델이 내놓는 결과를 비즈니스 가치로 연결하고, 출력의 제약 조건을 정의하는 일입니다. 다른 하나는 **모델 입력(Model Input)** 영역입니다. PM이 가진 도메인 지식을 활용해 어떤 데이터가 모델의 특성(feature)으로 유용할지 제안하는 일입니다. 예를 들어 대출 심사 모델을 기획하는 PM이 "신청자의 직업 안정성이 중요한 특성일 것"이라고 제안하면, 데이터 과학자는 그 특성을 수치화하는 방법을 설계합니다.

여기서 주의할 점이 있습니다. PM이 도메인 지식만으로 특성을 제안하면, 실제로 수집 가능한 데이터인지 확인하지 못하는 경우가 있습니다. 한 사례에서 PM이 "주당 근무 시간"을 모델 특성으로 포함하려 했지만, 대상 사용자 프로필에 해당 데이터 필드가 존재하지 않았습니다. 개발 후반부에 이 사실이 밝혀져 일정이 지연됐습니다. Goldilocks PM은 특성을 제안할 때 "이 데이터를 실제로 수집할 수 있는가"를 함께 확인합니다.

Goldilocks PM의 기준선을 실무에서 판별하는 방법도 있습니다. 데이터 과학자가 "저 PM은 질문을 제대로 한다"고 인정하면, 그 PM은 Goldilocks 수준에 도달한 것입니다. 이 수준은 고정된 체크리스트로 정의되지 않습니다. AI 기술은 빠르게 변합니다. 핵심은 새로운 패턴, 도구, 개념을 얼마나 빠르게 습득할 수 있느냐입니다. 정적인 지식보다 학습 속도가 Goldilocks PM의 진짜 역량입니다.

이제 Goldilocks PM이 데이터를 바라보는 첫 번째 렌즈를 다룹니다. **데이터 품질**입니다. AI 프로젝트가 실패하는 가장 흔한 원인은 데이터 부족과 품질 문제입니다. 모델의 성능은 학습 데이터의 품질에 직접적으로 의존합니다. 결함이 있는 데이터로 학습한 모델은 결함이 있는 출력을 냅니다. 이 원리를 "쓰레기가 들어가면 쓰레기가 나온다(Garbage in, garbage out)"고 표현합니다.

PM이 기획 단계에서 데이터 품질을 점검하지 않으면, 모델 성능 저하, 일정 지연, 편향된 출력이라는 세 가지 문제가 뒤늦게 터집니다. 그래서 기획 초기부터 데이터 가용성과 품질을 선제적으로 평가해야 합니다.

데이터 품질을 평가하는 기준은 다섯 가지 차원으로 나뉩니다.

```
데이터 품질 평가의 5가지 차원

+------------+--------------------------------------------+
| 차원       | PM이 확인할 질문                            |
+------------+--------------------------------------------+
| 정확성     | 데이터 값이 실제 상태를 올바르게             |
|            | 반영하는가?                                 |
+------------+--------------------------------------------+
| 완전성     | 필요한 필드에 빈 값이나 누락이               |
|            | 없는가?                                     |
+------------+--------------------------------------------+
| 시의성     | 데이터가 현재 상황을 반영할 만큼             |
|            | 최신인가?                                   |
+------------+--------------------------------------------+
| 일관성     | 서로 다른 출처의 데이터가 같은               |
|            | 기준으로 측정되었는가?                       |
+------------+--------------------------------------------+
| 관련성     | 수집한 데이터가 풀려는 문제에                |
|            | 실질적으로 도움이 되는가?                    |
+------------+--------------------------------------------+
```

이 다섯 가지를 PM 관점에서 좀 더 구체적으로 풀어 봅니다.

정확성은 레이블의 품질과 직결됩니다. 감성 분석 모델을 학습시키는데 레이블러가 "긍정"과 "부정"을 일관되지 않게 붙였다면, 모델은 혼란스러운 신호를 학습합니다. PM은 레이블링 작업의 일관성을 어떻게 보장할지 기획에 포함해야 합니다.

완전성은 데이터 볼륨과도 연결됩니다. 모델이 패턴을 학습하려면 최소한의 데이터 양이 필요합니다. 데이터가 부족하면 모델이 일반화하지 못합니다. PM은 "이 기능을 학습시키려면 데이터가 최소 얼마나 필요한가"를 엔지니어링 팀에 초기에 확인해야 합니다.

시의성은 시간이 지남에 따라 데이터의 유효성이 떨어지는 문제입니다. 3년 전 고객 행동 데이터로 학습한 추천 모델은 현재 고객의 취향을 반영하지 못할 수 있습니다. 이를 **시간 감쇠(time decay)** 문제라 합니다. PM은 데이터 갱신 주기를 제품 운영 계획에 포함해야 합니다.

일관성은 여러 출처에서 데이터를 합칠 때 특히 중요합니다. 한쪽에서는 나이를 "만 나이"로, 다른 쪽에서는 "세는 나이"로 기록했다면 같은 숫자가 다른 의미를 가집니다. PM은 데이터 통합 시 측정 기준의 차이를 확인해야 합니다.

관련성은 수집 가능한 데이터와 문제 해결에 필요한 데이터가 일치하는지를 따집니다. 앞서 "주당 근무 시간" 사례에서 봤듯이, 이론적으로 유용해 보이는 데이터라도 실제로 수집할 수 없으면 의미가 없습니다.

PM은 기획 문서 작성 단계에서 이 다섯 가지 차원을 점검 항목으로 포함하는 것이 좋습니다. 데이터 출처가 어디인지, 레이블링 품질을 어떻게 보장할지, 학습 데이터의 양이 충분한지, 데이터가 최신인지, 측정 기준이 통일되어 있는지를 문서에 명시합니다. 이 점검을 기획 단계에서 하면 개발 후반부에 "데이터가 부족합니다"라는 보고를 받는 상황을 줄일 수 있습니다.

두 번째 렌즈는 **데이터 편향**입니다. 편향은 학습 데이터에 내재된 역사적, 구조적 불균형이 모델 출력에 증폭되어 나타나는 현상입니다. 모델은 데이터에서 패턴을 학습합니다. 데이터 자체에 편향이 있으면, 모델은 그 편향을 패턴으로 인식하고 출력에 반영합니다. 단순히 반영하는 데 그치지 않고 증폭하는 경우도 있습니다.

PM이 편향을 기획 단계에서 인식해야 하는 이유는 출시 이후에 발견하면 대응 비용이 크기 때문입니다. 편향된 출력은 특정 사용자 그룹에 대한 차별로 이어질 수 있습니다. 비즈니스 신뢰도 하락, 법적 리스크, 사용자 이탈을 일으킵니다.

편향의 유형을 세 가지로 나누어 살펴봅니다.

첫째, **표현 편향(Representation bias)**입니다. 학습 데이터에서 특정 그룹이 과소 또는 과대 대표되는 경우입니다. 예를 들어 얼굴 인식 모델의 학습 데이터에 밝은 피부색 이미지가 80%를 차지하면, 어두운 피부색에 대한 인식 정확도가 떨어집니다. PM은 학습 데이터가 실제 사용자 분포를 반영하는지 확인해야 합니다.

둘째, **측정 편향(Measurement bias)**입니다. 데이터를 측정하는 방식 자체가 편향적인 경우입니다. 채용 모델에 "이력서 제출 횟수"를 특성으로 넣었다고 합시다. 이력서를 많이 제출하는 것은 구직 능력이 아니라 경제적 여유나 정보 접근성을 반영할 수 있습니다. 겉으로는 공정해 보이는 특성이 실제로는 차별적 결과를 만들 수 있습니다.

셋째, **집계 편향(Aggregation bias)**입니다. 서로 다른 특성을 가진 그룹을 하나의 모델로 처리하는 경우입니다. 전체 사용자를 대상으로 평균적인 성능이 좋더라도, 특정 하위 그룹에서는 성능이 크게 떨어질 수 있습니다. PM은 사용자 세그먼트별로 성능을 평가하는 계획을 기획에 포함해야 합니다.

편향 관리는 단순한 규정 준수 문제가 아닙니다. 기획 초기부터 윤리를 내재화하는 접근이 필요합니다. "출시 전에 편향 검사를 한 번 하면 된다"가 아니라, 어떤 사용자 그룹이 있는지, 각 그룹에 미치는 영향이 무엇인지를 기획 단계에서 설계합니다. 모델이 공정한지 판단하려면 모델이 왜 그런 결과를 냈는지 설명할 수 있어야 합니다. 이를 **설명 가능한 설계(Explainable Design)**라 합니다. 설명이 가능해야 편향을 감지하고 수정할 수 있습니다.

세 번째 렌즈는 **데이터 거버넌스**입니다. 거버넌스라는 단어는 "통치"나 "관리 체계"를 뜻합니다. 데이터 거버넌스는 AI 제품의 전체 생애주기에 걸쳐 데이터의 신뢰성, 추적 가능성, 책임 소재를 확보하는 정책과 프로세스와 역할 체계입니다.

거버넌스가 왜 필요한지부터 설명합니다. AI 모델은 데이터로 학습합니다. 그 데이터가 어디서 왔는지, 수집할 때 동의를 받았는지, 누가 관리하는지 추적할 수 없다면 문제가 생겼을 때 원인을 찾기 어렵습니다. 모델이 편향된 출력을 내도 "학습 데이터의 어느 부분 때문인지" 파악할 수 없습니다. 거버넌스는 이런 상황을 방지하는 운영 원칙입니다. 장식이 아니라 운영입니다.

PM 관점에서 거버넌스의 핵심은 네 가지 원칙으로 요약됩니다.

```
데이터 거버넌스 4대 원칙과 PM의 역할

+----------------------------+-------------------------------+
| 원칙                       | PM이 해야 할 일               |
+----------------------------+-------------------------------+
| 1. 전략 정렬               | AI 기능을 비즈니스 목표에     |
|                            | 연결하는 유스케이스 포트폴리오 |
|                            | 관점에서 관리합니다.          |
+----------------------------+-------------------------------+
| 2. 설계 시점의 데이터      | 데이터의 출처(리니지),        |
|    신뢰성 확보             | 소유자, 품질 모니터링을       |
|                            | 제품 설계 초기부터 정합니다.  |
+----------------------------+-------------------------------+
| 3. 실행 가능한 정책 수립   | 데이터 사용 허용 범위,        |
|                            | 모델 검증 기준,               |
|                            | 사람 검토(HITL) 요건을        |
|                            | 구체적으로 정합니다.          |
+----------------------------+-------------------------------+
| 4. 단대단 리스크 통제      | 배포 전 평가, 지속적 드리프트 |
|                            | 모니터링, 이상 발생 시        |
|                            | 대응 프로세스를 미리          |
|                            | 설계합니다.                   |
+----------------------------+-------------------------------+
```

이 표에서 몇 가지 용어를 풀어 봅니다. **리니지(lineage)**는 데이터의 출처와 변환 이력을 추적하는 것입니다. 원시 데이터가 어디서 왔고, 어떤 정제 과정을 거쳤고, 최종적으로 모델에 어떤 형태로 들어갔는지를 기록합니다. **HITL(Human-In-The-Loop)**은 모델의 판단에 사람이 개입하는 구조입니다. 고위험 결정에서 모델이 자동으로 최종 판단을 내리지 않고, 사람이 검토하는 단계를 둡니다. **드리프트(drift)**는 모델 배포 후 시간이 지나면서 입력 데이터의 분포가 변해 모델 성능이 떨어지는 현상입니다.

거버넌스는 PM 혼자 하는 일이 아닙니다. AI 제품 하나에도 여러 역할이 관여합니다. 비즈니스 오너는 결과를 정의합니다. PM은 제품 전달과 생애주기를 관리합니다. 데이터 스튜어드는 품질과 리니지를 담당합니다. 리스크 담당자는 통제와 승인을 맡습니다. PM의 역할은 이 역할들 사이에서 거버넌스가 실제로 작동하도록 조율하는 것입니다.

글로벌 규제 환경도 PM이 알아두어야 합니다. AI 제품에 법적 구속력이 있는 규제가 적용되기 시작했습니다. 위반 시 글로벌 매출의 상당 비율이 벌금으로 부과될 수 있습니다. 특히 의료, 금융, 채용처럼 사람의 삶에 직접 영향을 미치는 고위험 사용 사례에 대해서는 규제가 엄격합니다. PM은 자신이 기획하는 기능이 고위험 카테고리에 해당하는지를 초기에 식별해야 합니다.

AI 윤리 원칙은 안전성, 투명성, 비차별, 추적성, 환경 지속가능성의 다섯 가지로 정리됩니다. 이 원칙들은 서로 긴장 관계에 있습니다. 투명성을 높이면 프라이버시가 위협받을 수 있습니다. 안전성을 강화하면 혁신 속도가 느려질 수 있습니다. 추적성을 완벽히 구현하면 시스템 효율이 떨어질 수 있습니다. PM은 이 트레이드오프를 "규정 준수 체크박스"로 처리하지 않고, 사안별로 의식적으로 결정해야 합니다.

PM이 실무에서 거버넌스를 적용하는 구체적인 방법을 다섯 가지로 정리합니다. 첫째, 기획 문서에 데이터 출처, 라이선스, 동의 기록 항목을 포함합니다. 둘째, 모델 카드 작성을 엔지니어링 핸드오프 기준에 넣습니다. 모델 카드는 모델의 용도, 성능, 한계, 편향 정보를 정리한 문서입니다. 셋째, 배포 전 편향 감사 단계를 설정합니다. 넷째, 모델 드리프트 알림 기준을 출시 조건에 포함합니다. 다섯째, 기획 중인 기능이 고위험 AI 사용 사례에 해당하는지 식별합니다.

정리하면, 데이터 리터러시는 코드를 작성하는 능력이 아니라 데이터와 모델의 관계를 이해하는 역량입니다. Goldilocks PM은 너무 얕지도 깊지도 않은 수준에서 데이터를 이해하며, 핵심은 올바른 질문을 던지는 것입니다. 데이터 품질은 정확성, 완전성, 시의성, 일관성, 관련성의 다섯 차원으로 평가합니다. 데이터 편향은 표현, 측정, 집계 세 유형으로 나뉘며 기획 단계에서 사전에 식별해야 합니다. 데이터 거버넌스는 데이터의 신뢰성과 추적 가능성을 확보하는 운영 원칙으로, PM은 기획 문서부터 거버넌스 항목을 내재화해야 합니다.

다음 단원인 1.3.4에서는 AI 팀 구성과 역할별 협업 방법을 다룹니다. Goldilocks PM이 협업해야 할 데이터 과학자, ML 엔지니어, 디자이너 등 각 역할이 무엇을 하고, PM이 어떻게 이들과 효과적으로 일하는지를 살펴봅니다.

이 단원을 마치면 AI 기획자에게 필요한 데이터 리터러시 수준을 이해하고, 데이터 품질, 편향, 거버넌스를 기획에 반영하는 방법을 설명할 수 있습니다.
