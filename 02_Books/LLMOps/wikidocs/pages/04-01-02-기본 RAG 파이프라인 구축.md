# 4.1.2 기본 RAG 파이프라인 구축

4.1.1에서 RAG의 개념과 필요성을 살펴보았습니다. RAG는 외부 문서에서 관련 정보를 검색하여 LLM의 프롬프트에 덧붙이고, 이를 바탕으로 응답을 생성하는 방식입니다. 개념을 이해하는 것과 실제로 동작하는 파이프라인을 만드는 것은 다른 문제입니다. 이 단원에서는 문서를 불러오는 단계부터 최종 응답을 생성하는 단계까지, 기본 RAG 파이프라인을 구성하는 여섯 단계를 하나씩 짚어 봅니다.

기본 RAG 파이프라인은 크게 두 구간으로 나뉩니다. 첫 번째 구간은 문서를 벡터 데이터베이스에 저장하는 **인덱싱**(indexing) 구간이고, 두 번째 구간은 사용자 질문이 들어왔을 때 관련 문서를 찾아 응답을 만드는 **질의**(query) 구간입니다. 인덱싱 구간은 문서 로딩, 파싱과 전처리, 임베딩 생성과 벡터 저장의 세 단계로 구성됩니다. 질의 구간은 검색 쿼리 구성, 컨텍스트 조립과 프롬프트 구성, 응답 생성과 출처 추적의 세 단계로 구성됩니다. 전체 흐름을 먼저 그림으로 확인하겠습니다.

```
기본 RAG 파이프라인 전체 흐름

[인덱싱 구간]                          [질의 구간]

+-----------+    +-----------+    +----------+
| 1. 문서   | -> | 2. 파싱/  | -> | 3. 임베딩|
|    로딩   |    |    전처리 |    |    + 저장 |
+-----------+    +-----------+    +----------+
  PDF, HTML,       텍스트 추출,      벡터 변환,
  Markdown 등      청킹, 정리        벡터 DB 저장

                                       |
                                       v
+------------+    +-----------+    +----------+
| 6. 응답    | <- | 5. 컨텍스트| <- | 4. 검색  |
|    생성    |    |    + 프롬프트|   |    쿼리  |
+------------+    +-----------+    +----------+
  LLM 호출,        검색 결과 +       사용자 질문
  출처 추적         질문 조합         -> 벡터 검색
```

이 여섯 단계 중 어느 한 단계라도 부실하면 최종 응답의 품질이 떨어집니다. 문서 로딩에서 텍스트를 제대로 추출하지 못하면 검색할 내용 자체가 없고, 임베딩 품질이 낮으면 관련 문서를 찾지 못하며, 프롬프트 구성이 엉성하면 LLM이 검색 결과를 제대로 활용하지 못합니다. 각 단계를 순서대로 살펴보겠습니다.

**문서 로딩**(Document Loading)은 RAG 파이프라인의 출발점입니다. 조직이 보유한 지식은 다양한 형태로 존재합니다. 사내 매뉴얼은 PDF로 저장되어 있고, 기술 문서는 Markdown 파일로 관리되며, 웹 페이지는 HTML 형식입니다. 데이터베이스 테이블이나 CSV 파일에 담긴 정형 데이터도 있습니다. 문서 로딩이란 이러한 다양한 원본 파일을 프로그램이 처리할 수 있는 텍스트 형태로 읽어 들이는 작업을 뜻합니다.

문서 로딩이 별도의 단계로 존재하는 이유는 파일 형식마다 텍스트를 추출하는 방법이 다르기 때문입니다. PDF 파일은 시각적 배치 정보와 폰트 데이터를 담고 있어서, 단순히 파일을 열어 읽는 것만으로는 연속된 텍스트를 얻을 수 없습니다. PDF 파서 라이브러리가 페이지 레이아웃을 분석하여 텍스트 블록의 순서를 추정해야 합니다. HTML 파일은 태그 구조 안에 텍스트와 스타일, 스크립트, 광고 등이 뒤섞여 있으므로, 본문에 해당하는 텍스트만 골라내는 과정이 필요합니다. Markdown 파일은 비교적 단순하지만, 헤더 기호(#)나 목록 기호(-, *)를 어떻게 처리할지 결정해야 합니다.

실무에서는 각 파일 형식에 맞는 로더(loader)를 따로 준비합니다. 예를 들어 Python 환경에서는 PDF에 PyPDF, pdfplumber, Unstructured 같은 라이브러리를 사용하고, HTML에는 BeautifulSoup이나 Unstructured를 사용합니다. LangChain 같은 오케스트레이션 프레임워크는 이러한 로더들을 통합 인터페이스로 감싸서, 파일 형식에 관계없이 동일한 방식으로 문서를 불러올 수 있게 해 줍니다. 다음은 PDF 파일을 로딩하는 간단한 예입니다.

```python
from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader("company_manual.pdf")
documents = loader.load()
```

이 코드에서 `PyPDFLoader`는 PDF 파일 경로를 받아 각 페이지의 텍스트를 추출합니다. `loader.load()`를 호출하면 페이지별로 분리된 문서 객체 목록이 반환됩니다. 각 문서 객체는 추출된 텍스트(`page_content`)와 메타데이터(`metadata`)를 담고 있습니다. 메타데이터에는 원본 파일 이름, 페이지 번호 등이 포함됩니다. 이 메타데이터는 나중에 응답 생성 단계에서 출처를 추적할 때 사용합니다.

문서 로딩에서 자주 발생하는 문제는 스캔 PDF 처리입니다. 텍스트가 이미지로 들어간 스캔 PDF는 일반 PDF 파서로는 텍스트를 추출할 수 없습니다. 이런 경우에는 OCR(Optical Character Recognition, 광학 문자 인식) 도구를 사용하여 이미지에서 텍스트를 인식해야 합니다. 표가 많은 문서나 다단 레이아웃이 있는 문서도 텍스트 추출 품질이 떨어지기 쉬운 대상입니다. 문서 로딩 단계에서의 품질은 이후 모든 단계에 영향을 미치므로, 추출 결과를 직접 눈으로 확인하는 검증 과정이 중요합니다.

다음 단계는 **문서 파싱과 전처리**입니다. 문서 로딩으로 얻은 텍스트는 아직 RAG에 바로 사용할 수 있는 상태가 아닙니다. 두 가지 문제가 있습니다. 첫째, 원본 텍스트에는 불필요한 내용이 섞여 있을 수 있습니다. PDF에서 추출한 텍스트에 머리글, 바닥글, 페이지 번호가 반복적으로 나타나거나, HTML에서 추출한 텍스트에 네비게이션 메뉴나 광고 문구가 포함될 수 있습니다. 둘째, 문서 전체가 하나의 긴 텍스트 덩어리인 상태로는 검색에 적합하지 않습니다. 수십 페이지짜리 문서 전체를 하나의 벡터로 만들면, 문서 안의 특정 부분과 관련된 질문에 대해 정확한 검색이 어렵습니다.

전처리의 핵심은 두 가지입니다. 하나는 노이즈를 제거하는 **클리닝**이고, 다른 하나는 긴 문서를 검색에 적합한 크기의 조각으로 나누는 **청킹**(chunking)입니다.

클리닝에서는 불필요한 공백과 특수문자를 정리하고, 머리글/바닥글 같은 반복 요소를 제거하며, 인코딩 깨짐을 수정합니다. 이 작업은 단순해 보이지만, 원본 문서의 형태가 다양하기 때문에 실무에서는 예상보다 많은 시간이 들어가는 단계입니다.

청킹은 긴 텍스트를 일정 크기의 **청크**(chunk)로 분할하는 작업입니다. 청크란 검색의 최소 단위가 되는 텍스트 조각을 뜻합니다. 청크의 크기는 보통 수백에서 수천 토큰 사이로 설정합니다. 청크가 너무 크면 검색 결과에 무관한 내용이 많이 포함되고, 너무 작으면 문맥이 끊겨서 의미를 파악하기 어렵습니다. 가장 단순한 방식은 고정된 문자 수로 자르는 것이고, 좀 더 정교한 방식은 문단이나 문장 경계를 기준으로 자르는 것입니다. 청킹 전략의 종류와 선택 기준은 다음 단원인 4.1.3에서 자세히 다룹니다. 이 단원에서는 기본적인 고정 크기 청킹의 예를 보겠습니다.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200,
    separators=["\n\n", "\n", ". ", " ", ""]
)
chunks = text_splitter.split_documents(documents)
```

이 코드에서 `chunk_size=1000`은 각 청크의 최대 길이를 1,000자로 설정한다는 뜻입니다. `chunk_overlap=200`은 인접한 청크 사이에 200자를 겹치게 한다는 뜻입니다. 겹침을 두는 이유는 문장이 청크 경계에서 잘려 문맥이 끊기는 것을 완화하기 위해서입니다. `separators`는 텍스트를 나눌 때 우선적으로 시도할 구분자의 목록입니다. 빈 줄(`\n\n`)을 먼저 기준으로 시도하고, 그래도 청크가 너무 크면 줄바꿈(`\n`), 마침표와 공백(`. `), 공백(` `) 순서로 더 작은 단위로 쪼갭니다. 이렇게 하면 가능한 한 문단이나 문장 단위로 텍스트가 나뉘어 문맥이 보존됩니다.

청킹 과정에서 원본 문서의 메타데이터도 함께 유지해야 합니다. 청크 하나하나에 "이 조각이 어떤 문서의 몇 번째 페이지에서 왔는지"를 기록해 두어야, 나중에 응답을 생성할 때 출처를 추적할 수 있습니다.

세 번째 단계는 **임베딩 생성과 벡터 저장**입니다. 청킹으로 만들어진 텍스트 조각들을 벡터 데이터베이스에 저장하려면, 먼저 각 청크를 숫자 벡터로 변환해야 합니다. 이 변환을 **임베딩**(embedding)이라 합니다. 임베딩의 원리는 3.1.1에서 이미 다루었습니다. 핵심을 다시 짚으면, 임베딩은 텍스트의 의미를 고차원 공간의 좌표로 표현하는 것입니다. 의미가 비슷한 텍스트는 벡터 공간에서 가까운 위치에, 의미가 다른 텍스트는 먼 위치에 놓입니다.

임베딩을 생성하려면 임베딩 모델이 필요합니다. 3.1.2에서 다룬 것처럼 OpenAI의 text-embedding 모델, 오픈소스인 BGE, E5 등 다양한 선택지가 있습니다. 모델을 선택했으면 모든 청크를 임베딩 모델에 통과시켜 벡터를 얻습니다.

```python
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
```

이 코드는 OpenAI의 text-embedding-3-small 모델을 임베딩 생성기로 설정합니다. `model` 인자에 사용할 모델의 이름을 지정합니다. 이 객체를 이후 벡터 데이터베이스에 전달하면, 벡터 데이터베이스가 내부적으로 이 모델을 호출하여 각 청크의 임베딩을 생성합니다.

생성된 벡터는 **벡터 데이터베이스**에 저장합니다. 벡터 데이터베이스의 개념은 3.2.1에서 다루었습니다. 벡터 데이터베이스는 고차원 벡터를 효율적으로 저장하고, 주어진 쿼리 벡터와 가장 유사한 벡터를 빠르게 찾아주는 특화된 저장소입니다. Pinecone, Weaviate, Qdrant, Milvus, Chroma 등 여러 제품이 있습니다. 다음은 Chroma를 사용하여 청크를 벡터로 변환하고 저장하는 예입니다.

```python
from langchain_community.vectorstores import Chroma

vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)
```

`Chroma.from_documents`는 세 가지 일을 한 번에 수행합니다. 첫째, `chunks`에 담긴 각 청크의 텍스트를 `embeddings` 객체(앞서 설정한 OpenAI 임베딩 모델)에 전달하여 벡터를 생성합니다. 둘째, 생성된 벡터와 원본 텍스트, 메타데이터를 Chroma 데이터베이스에 저장합니다. 셋째, `persist_directory`에 지정된 경로에 데이터를 영구 저장하여, 프로그램을 종료했다가 다시 시작해도 데이터가 유지되도록 합니다.

여기까지가 인덱싱 구간입니다. 인덱싱은 문서가 변경되거나 추가될 때만 수행하면 되므로, 사용자 질문이 들어올 때마다 반복하지 않아도 됩니다. 인덱싱이 완료되면, 이제 사용자의 질문에 응답할 준비가 된 것입니다.

네 번째 단계는 **검색 쿼리 구성**입니다. 사용자가 질문을 입력하면, 그 질문을 벡터 데이터베이스에서 검색할 수 있는 형태로 바꿔야 합니다. 가장 단순한 방법은 사용자의 질문 텍스트를 그대로 임베딩하여 쿼리 벡터로 사용하는 것입니다. 이 쿼리 벡터와 벡터 데이터베이스에 저장된 청크 벡터들 사이의 유사도를 계산하여, 유사도가 높은 순서대로 청크를 가져옵니다.

```python
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4}
)
results = retriever.invoke("회사의 연차 휴가 정책은 어떻게 되나요?")
```

이 코드에서 `as_retriever`는 벡터 데이터베이스를 검색기(retriever) 객체로 변환합니다. `search_type="similarity"`는 코사인 유사도 기반 검색을 사용한다는 뜻이고, `search_kwargs={"k": 4}`는 가장 유사한 청크 4개를 가져온다는 뜻입니다. `retriever.invoke`에 사용자의 질문을 전달하면, 내부적으로 질문이 임베딩되어 쿼리 벡터가 만들어지고, 이 쿼리 벡터와 가장 유사한 4개의 청크가 반환됩니다.

사용자의 질문을 그대로 쿼리로 사용하는 것이 항상 최선은 아닙니다. 예를 들어 "연차 며칠?"처럼 짧고 불완전한 질문은 의미를 충분히 담지 못해 정확한 검색 결과를 얻기 어렵습니다. 이런 경우 LLM을 활용하여 질문을 검색에 적합한 형태로 재구성하는 방법이 있습니다. 사용자의 원래 질문을 LLM에 전달하여 "이 질문과 관련된 정보를 찾기 위해 검색할 문장을 작성해 주세요"라고 요청하는 것입니다. 이를 **쿼리 변환**(query transformation)이라 합니다. 쿼리 변환은 기본 RAG에서는 선택사항이지만, 검색 품질을 높이는 데 효과적입니다.

다섯 번째 단계는 **컨텍스트 조립과 프롬프트 구성**입니다. 검색으로 가져온 청크들을 LLM에 전달할 프롬프트 안에 배치하는 단계입니다. RAG의 핵심 원리가 바로 이 단계에 있습니다. 4.1.1에서 설명한 것처럼, RAG는 LLM이 학습하지 못한 외부 정보를 프롬프트에 "덧붙여"(augment) 제공하여, 모델이 이를 참고하여 답변하게 합니다.

프롬프트는 보통 세 부분으로 구성됩니다. 첫째, 시스템 지시(system instruction)로 모델의 역할과 응답 규칙을 정합니다. 둘째, 검색된 컨텍스트(context)로 관련 문서 조각들을 나열합니다. 셋째, 사용자의 질문(question)을 넣습니다. 다음은 이 구조를 보여주는 프롬프트 템플릿의 예입니다.

```python
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_template("""
당신은 사내 문서를 기반으로 질문에 답변하는 어시스턴트입니다.
아래 제공된 컨텍스트만을 사용하여 질문에 답변하세요.
컨텍스트에 답변할 정보가 없으면 "해당 정보를 찾을 수 없습니다"라고 답하세요.

컨텍스트:
{context}

질문: {question}

답변:
""")
```

이 템플릿에서 `{context}` 자리에는 검색으로 가져온 청크들의 텍스트가 들어가고, `{question}` 자리에는 사용자의 원래 질문이 들어갑니다. 시스템 지시에서 "컨텍스트만을 사용하여"라고 명시하는 것이 중요합니다. 이 지시가 없으면 LLM이 자신의 사전학습 지식을 섞어 답변할 수 있고, 그 경우 RAG를 도입한 목적인 "신뢰할 수 있는 출처에 근거한 답변"이 무력화됩니다. 또한 "정보가 없으면 없다고 답하라"는 지시도 환각 방지에 효과적입니다.

컨텍스트를 조립할 때 고려할 점이 있습니다. 검색된 청크가 4개라면 이 4개의 텍스트를 단순히 이어 붙이는 것이 기본 방식입니다. 그런데 LLM에는 한 번에 처리할 수 있는 입력 길이의 상한인 **컨텍스트 윈도우**(context window)가 있습니다. 컨텍스트 윈도우는 모델마다 다르며, 수천 토큰에서 수십만 토큰까지 다양합니다. 검색 결과가 많거나 청크가 클 경우, 전체를 프롬프트에 넣으면 컨텍스트 윈도우를 초과할 수 있습니다. 이때는 가져올 청크 수를 줄이거나, 각 청크에서 핵심 부분만 발췌하는 등의 조치가 필요합니다.

여섯 번째이자 마지막 단계는 **응답 생성과 출처 추적**입니다. 구성된 프롬프트를 LLM에 전달하면 모델이 응답을 생성합니다. 기본적인 호출 방식은 다음과 같습니다.

```python
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

answer = chain.invoke("회사의 연차 휴가 정책은 어떻게 되나요?")
```

이 코드를 한 줄씩 살펴보겠습니다. `ChatOpenAI(model="gpt-4o-mini", temperature=0)`는 응답 생성에 사용할 LLM을 설정합니다. `temperature=0`은 가장 확률이 높은 토큰만 선택하게 하여 응답의 일관성을 높이는 설정입니다. `format_docs` 함수는 검색된 문서 객체들의 텍스트를 빈 줄로 구분하여 하나의 문자열로 합칩니다. `chain`은 파이프라인의 실행 순서를 정의합니다. 사용자 질문이 들어오면 `retriever`가 관련 청크를 검색하고, `format_docs`가 이를 텍스트로 정리하며, `prompt`가 컨텍스트와 질문을 템플릿에 채우고, `llm`이 응답을 생성하며, `StrOutputParser`가 응답에서 텍스트 부분만 추출합니다.

출처 추적은 RAG의 중요한 장점 중 하나입니다. LLM이 단독으로 생성한 응답은 어떤 근거에서 나온 것인지 알 수 없지만, RAG에서는 응답의 근거가 된 청크를 특정할 수 있습니다. 각 청크에는 문서 로딩 단계에서 부여한 메타데이터(파일명, 페이지 번호 등)가 붙어 있으므로, 검색 결과와 함께 이 메타데이터를 사용자에게 보여주면 "이 답변은 어떤 문서의 어느 부분에 근거한 것인지" 확인할 수 있습니다. 실무에서는 응답 텍스트 아래에 "참고 문서: company_manual.pdf 3페이지"와 같은 형태로 출처를 표시합니다.

출처 추적을 구현하려면 응답 생성 체인을 약간 수정하여, LLM의 답변과 검색된 문서를 함께 반환하도록 만들면 됩니다. 검색된 청크 목록에서 각 청크의 `metadata` 필드를 꺼내 출처 정보를 추출합니다. 프롬프트에 "답변에 사용한 문서의 출처를 명시하세요"라는 지시를 추가하여 LLM이 답변 안에 직접 출처를 언급하게 할 수도 있습니다.

여기까지 여섯 단계를 모두 다루었습니다. 전체 흐름을 단계별로 정리하면 다음과 같습니다.

```
단계별 입력과 출력 요약

단계               입력                      출력
---------------------------------------------------------------
1. 문서 로딩       원본 파일(PDF, HTML 등)    문서 객체(텍스트 + 메타데이터)
2. 파싱/전처리     문서 객체                  청크 목록(정제된 텍스트 조각)
3. 임베딩 + 저장   청크 목록                  벡터 DB에 저장된 벡터
4. 검색 쿼리       사용자 질문                유사도 높은 청크 목록
5. 컨텍스트 조립   청크 + 질문                완성된 프롬프트
6. 응답 생성       프롬프트                   답변 텍스트 + 출처 정보
```

정리하면, 기본 RAG 파이프라인은 문서 로딩, 파싱과 전처리, 임베딩 생성과 벡터 저장, 검색 쿼리 구성, 컨텍스트 조립과 프롬프트 구성, 응답 생성과 출처 추적의 여섯 단계로 구성됩니다. 앞의 세 단계는 문서를 벡터 데이터베이스에 준비하는 인덱싱 구간이고, 뒤의 세 단계는 사용자 질문에 응답하는 질의 구간입니다. 각 단계에서 문서의 메타데이터를 유지하면 최종 응답에서 출처를 추적할 수 있고, 프롬프트에 "컨텍스트만 사용하라"는 지시를 넣으면 환각을 줄일 수 있습니다.

다음 단원인 4.1.3에서는 이 파이프라인의 두 번째 단계인 청킹을 더 깊이 다룹니다. 고정 크기 청킹, 재귀적 청킹, 의미 기반 청킹 등 다양한 전략의 특성과 선택 기준을 살펴봅니다.

이 단원을 마치면 문서 로딩부터 응답 생성까지 기본 RAG 파이프라인을 구축할 수 있습니다.
