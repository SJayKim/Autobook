# 1.2.2 토큰화와 어휘

1.2.1에서 Transformer 아키텍처의 내부 구조를 살펴보았습니다. Self-Attention이 단어 벡터를 입력으로 받아 문맥을 반영한 새로운 벡터를 만들어 낸다는 것까지 다루었는데, 여기서 한 가지 빠진 이야기가 있습니다. 사람이 입력하는 텍스트는 문자의 나열이지, 숫자 벡터가 아닙니다. 모델이 텍스트를 처리하려면 문자열을 숫자로 바꾸는 과정이 먼저 필요합니다. 이 과정의 첫 단계가 바로 **토큰화(tokenization)**입니다.

토큰화란 텍스트를 일정한 단위로 쪼개는 작업입니다. 쪼개진 각 조각을 **토큰(token)**이라고 부릅니다. 모델은 이 토큰 하나하나에 고유한 번호를 붙이고, 그 번호를 벡터로 변환하여 내부 연산을 수행합니다. 따라서 토큰화 방식에 따라 같은 문장이라도 모델이 받아들이는 입력의 형태가 달라집니다.

왜 토큰화가 필요한지를 좀 더 구체적으로 살펴보겠습니다. 가장 직관적인 방식은 공백을 기준으로 단어를 나누는 것입니다. 영어 문장 "I like cats"는 공백으로 나누면 "I", "like", "cats" 세 단어가 됩니다. 그런데 이 방식에는 두 가지 심각한 문제가 있습니다.

첫 번째 문제는 어휘 폭발입니다. 영어에서 "play"라는 단어는 "plays", "played", "playing", "player", "players" 등 수많은 변형을 만들어 냅니다. 단어 단위로 쪼개면 이 변형 각각을 별개의 어휘로 등록해야 합니다. 영어만 해도 수십만 개의 변형이 존재하고, 한국어처럼 조사와 어미가 붙는 언어에서는 그 수가 훨씬 더 늘어납니다. 어휘 목록이 커지면 모델의 입출력 층 크기도 비례하여 커지므로 메모리와 계산 비용이 급증합니다.

두 번째 문제는 미등록 단어(OOV, Out-of-Vocabulary)입니다. 학습 데이터에 한 번도 나타나지 않은 단어는 어휘 목록에 없으므로, 모델이 이 단어를 만났을 때 아무런 처리를 할 수 없습니다. 예를 들어 "ChatGPT"라는 단어가 학습 당시 어휘에 없었다면, 모델은 이 단어를 이해하지 못합니다.

반대로, 텍스트를 한 글자씩 쪼개는 방식도 생각할 수 있습니다. 영어라면 알파벳 26개와 몇 가지 기호만으로 모든 텍스트를 표현할 수 있으므로 어휘 크기가 매우 작아집니다. 그러나 "playing"이라는 단어를 "p", "l", "a", "y", "i", "n", "g" 일곱 개 토큰으로 쪼개면, 모델이 처리해야 할 토큰 수가 크게 늘어납니다. 1.2.1에서 살펴본 Self-Attention은 모든 토큰 쌍을 비교하므로, 토큰 수가 많아지면 계산량이 토큰 수의 제곱에 비례하여 증가합니다. 또한 개별 글자만으로는 의미를 파악하기 어렵기 때문에 모델 성능도 떨어집니다.

정리하면, 단어 단위로 쪼개면 어휘가 너무 커지고, 글자 단위로 쪼개면 토큰 수가 너무 많아집니다. 현대의 대규모 언어 모델은 이 두 극단 사이에서 적절한 균형을 찾는 **서브워드(subword)** 토큰화를 사용합니다. 서브워드 토큰화는 자주 등장하는 문자 조합은 하나의 토큰으로 묶고, 드물게 등장하는 문자 조합은 더 작은 단위로 분리합니다. 예를 들어 "playing"은 "play"와 "ing"로 나뉠 수 있습니다. "play"는 자주 등장하므로 하나의 토큰이 되고, "ing"도 여러 단어에서 반복되므로 하나의 토큰이 됩니다. 이렇게 하면 어휘 크기를 적정 수준으로 유지하면서도, 학습 데이터에 없던 단어가 나타나더라도 서브워드 조합으로 표현할 수 있어 미등록 단어 문제를 크게 줄일 수 있습니다.

서브워드 토큰화를 구현하는 대표적인 알고리즘으로 **BPE(Byte Pair Encoding)**, **WordPiece**, **SentencePiece**가 있습니다. 각각의 동작 방식이 다르므로 하나씩 살펴보겠습니다.

**BPE(Byte Pair Encoding)**는 가장 널리 쓰이는 서브워드 토큰화 알고리즘입니다. GPT-2, GPT-3, GPT-4, LLaMA 계열 모델이 BPE를 사용합니다. BPE라는 이름은 원래 데이터 압축 기법에서 유래했습니다. 텍스트에서 가장 자주 나란히 등장하는 바이트 쌍(pair)을 하나로 합치는 과정을 반복하여 압축하는 방식이었는데, 이를 토큰화에 응용한 것입니다.

BPE의 학습 과정을 단계별로 살펴보겠습니다. 먼저, 학습 데이터의 모든 텍스트를 글자(또는 바이트) 단위로 쪼갭니다. 초기 어휘는 개별 글자들의 집합입니다. 그다음, 전체 데이터에서 나란히 등장하는 두 토큰 쌍의 빈도를 셉니다. 가장 빈도가 높은 쌍을 하나의 새로운 토큰으로 합치고, 이 새 토큰을 어휘에 추가합니다. 데이터 전체에서 해당 쌍이 나타나는 곳을 모두 새 토큰으로 교체합니다. 이 과정을 원하는 어휘 크기에 도달할 때까지 반복합니다.

구체적인 예를 들어 보겠습니다. 학습 데이터에 "low", "lower", "lowest", "new", "newer"라는 단어가 있다고 가정합니다. 초기 상태에서 각 단어는 글자 단위로 분리되어 있습니다.

```
l o w        (빈도 5)
l o w e r    (빈도 2)
l o w e s t  (빈도 1)
n e w        (빈도 3)
n e w e r    (빈도 1)
```

전체 데이터에서 나란히 등장하는 쌍의 빈도를 세면, "l"과 "o"의 쌍이 8회(low 5 + lower 2 + lowest 1)로 가장 높습니다. "l"과 "o"를 합쳐 "lo"라는 새 토큰을 만듭니다.

```
lo w         (빈도 5)
lo w e r     (빈도 2)
lo w e s t   (빈도 1)
n e w        (빈도 3)
n e w e r    (빈도 1)
```

다음으로 "lo"와 "w"의 쌍이 8회로 가장 빈도가 높으므로, "low"라는 토큰을 만듭니다.

```
low          (빈도 5)
low e r      (빈도 2)
low e s t    (빈도 1)
n e w        (빈도 3)
n e w e r    (빈도 1)
```

이런 식으로 계속 합쳐 나가면, 자주 등장하는 문자열은 하나의 토큰이 되고, 드물게 등장하는 문자열은 여러 토큰으로 남게 됩니다. 이 과정 전체를 도식화하면 아래와 같습니다.

```
[BPE 학습 과정]

1단계: 글자 단위 분리
   "lower" --> l, o, w, e, r

2단계: 가장 빈번한 쌍 찾기
   (l, o) = 8회  <-- 최다 빈도

3단계: 쌍 병합, 어휘 추가
   "lo" 토큰 생성, 어휘에 추가

4단계: 데이터에서 해당 쌍 교체
   l, o, w, e, r --> lo, w, e, r

5단계: 2~4단계 반복 (어휘 크기 도달까지)
   (lo, w) = 8회 --> "low" 토큰 생성
   (e, r) = 3회  --> "er" 토큰 생성
   ...
```

BPE로 학습된 토크나이저를 실제 텍스트에 적용할 때는, 학습 과정에서 정해진 병합 규칙을 동일한 순서로 적용합니다. 입력 텍스트를 글자 단위로 쪼갠 뒤, 학습 시 가장 먼저 만들어진 병합 규칙부터 차례로 적용하여 토큰을 합쳐 나갑니다. 학습 데이터에서 자주 나타났던 문자열일수록 먼저 병합되므로, 빈도가 높은 문자열은 긴 토큰 하나가 되고, 빈도가 낮은 문자열은 짧은 토큰 여러 개로 남습니다.

GPT 계열 모델에서 사용하는 BPE는 한 가지 변형이 있습니다. 일반적인 BPE가 유니코드 글자 단위로 시작하는 것과 달리, **바이트 수준 BPE(Byte-level BPE)**는 텍스트를 UTF-8로 인코딩한 바이트 단위로 시작합니다. 영어 알파벳은 1바이트, 한글 한 글자는 3바이트로 표현됩니다. 바이트는 0부터 255까지 256가지뿐이므로, 초기 어휘 크기가 256으로 고정됩니다. 어떤 언어의 어떤 문자든 바이트의 조합으로 표현할 수 있으므로, 미등록 문자가 원천적으로 발생하지 않습니다. 이 방식 덕분에 GPT 계열 모델은 영어, 한국어, 일본어, 아랍어 등 다양한 언어를 하나의 토크나이저로 처리할 수 있습니다.

**WordPiece**는 Google이 BERT 모델에 사용한 서브워드 토큰화 알고리즘입니다. BPE와 기본 아이디어는 비슷합니다. 글자 단위에서 시작하여 토큰을 점진적으로 합쳐 나가는 것은 동일합니다. 차이는 합칠 쌍을 선택하는 기준에 있습니다.

BPE는 단순히 빈도가 가장 높은 쌍을 합칩니다. 반면 WordPiece는 두 토큰을 합쳤을 때 학습 데이터의 **우도(likelihood)**가 가장 크게 증가하는 쌍을 합칩니다. 우도란 데이터가 해당 모델로부터 생성되었을 확률을 뜻합니다. 쉽게 말해, 단순 빈도만 보는 것이 아니라 "이 두 토큰을 합치면 전체 데이터를 더 잘 설명할 수 있는가"를 기준으로 판단하는 것입니다. 예를 들어, "un"과 "happy"가 나란히 자주 등장하더라도, "un"이 "un" 단독으로도 매우 자주 나타난다면, 단순 빈도 기준으로는 높은 순위일 수 있지만 우도 기준으로는 합치는 이득이 크지 않을 수 있습니다.

WordPiece의 또 다른 특징은 단어의 중간이나 끝에 오는 서브워드에 "##" 접두사를 붙여 표시하는 것입니다. 예를 들어 "unhappy"가 "un"과 "##happy"로 분리됩니다. "##" 표시가 있으면 이 토큰이 단어의 시작이 아니라 앞 토큰에 이어지는 조각이라는 뜻입니다. 이 표시 덕분에 토큰을 다시 원래 텍스트로 복원할 때 어디서 단어가 나뉘는지 구분할 수 있습니다.

**SentencePiece**는 Google이 만든 토큰화 라이브러리로, BPE와 WordPiece가 갖는 한 가지 공통 제약을 해결합니다. BPE와 WordPiece는 토큰화를 시작하기 전에 텍스트를 공백 기준으로 먼저 단어로 분리하는 **전처리 단계**를 필요로 합니다. 그런데 공백으로 단어를 나누는 규칙은 언어마다 다릅니다. 영어는 공백이 단어 구분자 역할을 하지만, 중국어나 일본어는 공백 없이 문자를 이어 씁니다. 한국어도 조사가 띄어쓰기 없이 명사에 붙는 경우가 많습니다. 따라서 공백 기반 전처리는 특정 언어에 종속적인 방식입니다.

SentencePiece는 텍스트를 공백 구분 없이 하나의 문자열 그대로 입력받습니다. 공백 자체도 일반 문자와 동등하게 취급합니다. 공백 문자는 특수 기호(보통 밑줄 "_"로 표기)로 치환하여, 토큰화 과정에서 다른 문자와 동일하게 병합 대상이 됩니다. 이렇게 하면 어떤 언어든 동일한 방식으로 처리할 수 있어, 다국어를 지원하는 모델에서 특히 유용합니다. LLaMA, T5 등 여러 다국어 모델이 SentencePiece를 사용합니다.

SentencePiece는 내부적으로 BPE 알고리즘 또는 **유니그램(Unigram)** 알고리즘을 선택하여 사용할 수 있습니다. 유니그램 방식은 BPE와 반대 방향으로 동작합니다. BPE가 작은 토큰을 합쳐 가며 어휘를 키우는 상향식(bottom-up) 접근이라면, 유니그램은 큰 어휘에서 시작하여 불필요한 토큰을 제거해 가는 하향식(top-down) 접근입니다. 먼저 충분히 큰 후보 어휘를 만들고, 각 토큰이 제거되었을 때 전체 데이터의 우도가 얼마나 감소하는지를 계산합니다. 우도 감소가 가장 작은 토큰, 즉 없어도 전체 표현에 영향이 적은 토큰을 제거합니다. 이를 반복하여 원하는 어휘 크기에 도달합니다.

세 가지 알고리즘의 차이를 정리하면 아래와 같습니다.

```
알고리즘       병합 기준          방향       공백 전처리   대표 모델
----------------------------------------------------------------------
BPE            빈도 최다 쌍       상향식     필요         GPT-2/3/4, LLaMA
WordPiece      우도 최대 증가 쌍  상향식     필요         BERT
SentencePiece  BPE 또는 유니그램  양방향     불필요       LLaMA, T5
```

**어휘 크기(vocabulary size)**는 토크나이저가 보유한 전체 토큰의 수를 뜻합니다. 어휘 크기는 모델 설계 시 정하는 중요한 하이퍼파라미터입니다. 어휘 크기가 크면 자주 등장하는 단어나 구절이 하나의 토큰으로 표현될 가능성이 높아져, 같은 텍스트를 더 적은 수의 토큰으로 표현할 수 있습니다. 반면, 어휘 크기가 커질수록 모델의 임베딩 층과 출력 층의 파라미터 수가 늘어납니다. 1.2.1에서 설명했듯이, 각 토큰에 대응하는 벡터를 저장해야 하므로, 어휘 크기에 벡터 차원 수를 곱한 만큼의 파라미터가 필요합니다.

실제 모델들의 어휘 크기를 비교하면 다음과 같습니다.

```
모델           어휘 크기      토크나이저
-------------------------------------------
GPT-2          50,257        BPE (바이트 수준)
GPT-4          100,256       BPE (바이트 수준)
BERT           30,522        WordPiece
LLaMA 2        32,000        SentencePiece (BPE)
LLaMA 3       128,256        BPE (바이트 수준)
```

GPT-2에서 GPT-4로 가면서 어휘 크기가 약 두 배로 늘어난 것을 볼 수 있습니다. 어휘가 커지면 동일한 텍스트를 더 적은 토큰으로 표현하므로, Attention 계산의 부담이 줄어들고 처리 속도가 빨라질 수 있습니다. LLaMA 3가 어휘 크기를 32,000에서 128,256으로 대폭 늘린 것도 같은 이유입니다. 특히 영어가 아닌 다른 언어의 텍스트를 처리할 때 효과가 큽니다. 어휘가 작으면 한글 한 글자가 여러 토큰으로 쪼개지지만, 어휘가 크면 자주 쓰이는 한글 음절이나 단어가 하나의 토큰으로 등록되어 토큰 수가 줄어듭니다.

그러나 어휘 크기를 무한정 늘릴 수는 없습니다. 어휘가 커지면 임베딩 층 파라미터가 늘어나 메모리를 더 많이 차지합니다. 또한 드물게 등장하는 토큰은 학습 데이터에서 충분한 예시를 보지 못하므로, 해당 토큰의 벡터 표현이 부정확해질 수 있습니다. 적절한 어휘 크기를 정하는 것은 토큰 효율성(적은 토큰으로 텍스트를 표현하는 능력)과 학습 품질 사이의 균형을 찾는 문제입니다.

토큰화 과정에서는 일반 텍스트 조각 외에도 **특수 토큰(special token)**이 사용됩니다. 특수 토큰은 모델에 텍스트의 구조적 정보를 전달하기 위해 인위적으로 삽입하는 토큰입니다. 실제 텍스트에는 존재하지 않지만, 모델이 입력의 시작과 끝, 문장 사이의 경계, 빈 공간 등을 인식할 수 있도록 돕습니다.

대표적인 특수 토큰 세 가지를 살펴보겠습니다.

**BOS(Beginning of Sequence)** 토큰은 입력 시퀀스의 시작을 나타냅니다. 모델이 이 토큰을 만나면 "여기서부터 새로운 텍스트가 시작된다"는 신호로 인식합니다. 텍스트 생성 시 BOS 토큰을 첫 입력으로 넣으면, 모델은 이 토큰 다음에 올 단어부터 생성을 시작합니다.

**EOS(End of Sequence)** 토큰은 시퀀스의 끝을 나타냅니다. 모델이 텍스트를 생성하다가 EOS 토큰을 출력하면, "생성을 여기서 멈춰야 한다"는 신호가 됩니다. EOS 토큰이 없으면 모델은 언제 출력을 멈춰야 하는지 알 수 없으므로, 의미 없는 텍스트를 끝없이 생성할 수 있습니다. 또한 여러 개의 텍스트를 이어 붙여 학습할 때, EOS 토큰이 문서 사이의 경계 역할을 합니다.

**PAD(Padding)** 토큰은 서로 다른 길이의 입력을 동일한 길이로 맞출 때 사용합니다. GPU는 같은 크기의 데이터를 한꺼번에 처리할 때 효율이 높습니다. 그래서 여러 입력을 하나의 묶음(배치)으로 만들 때, 짧은 입력 뒤에 PAD 토큰을 채워 가장 긴 입력과 길이를 맞춥니다. 모델은 PAD 토큰을 실제 내용으로 취급하지 않도록 학습됩니다.

실제로 특수 토큰이 포함된 입력이 어떻게 구성되는지 예를 들어 보겠습니다. 세 개의 문장을 하나의 배치로 처리한다고 가정합니다.

```
문장 1: "오늘 날씨가 좋다"
문장 2: "내일은 비가 온다고 한다"
문장 3: "우산을 챙기자"

토큰화 후 배치 구성:
[BOS] 오늘 날씨 가 좋다 [EOS] [PAD] [PAD]
[BOS] 내일 은 비가 온다 고 한다 [EOS]
[BOS] 우산 을 챙기 자 [EOS] [PAD] [PAD]
```

세 문장의 토큰 수가 서로 다르지만, PAD 토큰을 넣어 같은 길이로 맞춘 것을 볼 수 있습니다. BOS 토큰은 각 문장의 시작을, EOS 토큰은 끝을 표시합니다.

이러한 특수 토큰 외에도, 모델에 따라 다양한 특수 토큰이 사용됩니다. 대화형 모델에서는 사용자 발화와 모델 응답을 구분하는 역할 토큰이 있고, BERT에서는 문장 쌍의 경계를 나타내는 [SEP] 토큰이나 빈칸 예측을 위한 [MASK] 토큰이 있습니다. 어떤 특수 토큰을 사용하는지는 모델의 학습 방식과 용도에 따라 달라집니다.

토큰 수는 LLMOps 관점에서 비용과 직결되는 요소입니다. 대부분의 LLM API 서비스는 처리한 토큰 수에 비례하여 요금을 부과합니다. 사용자가 보낸 입력 텍스트를 토큰화한 결과가 입력 토큰이 되고, 모델이 생성한 응답 텍스트의 토큰 수가 출력 토큰이 됩니다. 총 비용은 입력 토큰 수와 출력 토큰 수에 각각의 단가를 곱하여 합산합니다.

같은 내용이라도 토큰화 방식에 따라 토큰 수가 달라집니다. 영어 텍스트에서는 대략 1개 토큰이 약 4글자, 또는 약 0.75개 단어에 해당하는 것이 일반적입니다. 한국어는 영어보다 토큰 효율이 낮은 경우가 많습니다. 한글 한 글자가 UTF-8에서 3바이트를 차지하므로, 바이트 수준 BPE에서 어휘에 해당 음절이 없으면 여러 토큰으로 쪼개질 수 있습니다. 예를 들어, 영어 "Hello"가 1개 토큰으로 처리되는 동안 한국어 "안녕하세요"가 3~5개 토큰으로 처리될 수 있습니다. 이 차이는 API 사용 비용에 직접적으로 영향을 줍니다.

토큰 수가 늘어나면 비용만 증가하는 것이 아닙니다. 1.2.1에서 다루었듯이, Self-Attention의 계산량은 토큰 수의 제곱에 비례합니다. 토큰 수가 많아지면 추론 시간이 길어지고, GPU 메모리 사용량도 늘어납니다. 또한 모든 모델에는 한 번에 처리할 수 있는 토큰 수의 상한인 **컨텍스트 윈도우(context window)**가 있습니다. 컨텍스트 윈도우가 4,096 토큰인 모델에 5,000 토큰의 입력을 보내면 처리할 수 없습니다. 따라서 LLMOps에서는 불필요한 텍스트를 줄여 토큰 수를 최적화하거나, 긴 입력을 여러 조각으로 나누어 처리하는 전략을 사용합니다.

정리하면, 토큰화는 텍스트를 모델이 처리할 수 있는 단위로 쪼개는 과정이며, 단어 단위와 글자 단위의 한계를 절충하는 서브워드 방식이 표준으로 자리 잡았습니다. BPE는 빈도 기반으로 쌍을 병합하고, WordPiece는 우도 기반으로 병합하며, SentencePiece는 공백 전처리 없이 다국어를 처리합니다. 어휘 크기는 토큰 효율성과 모델 파라미터 사이의 균형 문제이고, BOS, EOS, PAD 같은 특수 토큰은 모델에 텍스트 구조 정보를 전달합니다. 토큰 수는 API 비용, 추론 속도, 메모리 사용량에 직접 영향을 주므로, LLMOps를 수행할 때 반드시 고려해야 하는 요소입니다.

다음 단원인 1.2.3에서는 추론 과정과 생성 전략을 다룹니다. 토큰화된 입력이 모델 내부에서 어떻게 처리되어 출력 토큰을 만들어 내는지, 그리고 생성 결과의 품질을 조절하는 온도, top-k, top-p 같은 전략이 어떻게 동작하는지를 살펴봅니다.

이 단원을 마치면 토큰화 과정을 단계별로 설명하고, BPE, WordPiece, SentencePiece의 차이를 구분할 수 있습니다.
