# 4.2.2 Advanced RAG 패턴

4.2.1에서 하이브리드 검색, 리랭킹, 쿼리 변환을 통해 RAG의 검색 품질을 높이는 방법을 다루었습니다. 이 기법들은 "더 좋은 문서를 찾아오는 것"에 초점을 맞추었습니다. 그런데 검색 결과가 충분히 좋더라도 여전히 해결되지 않는 문제가 있습니다. 검색된 문서가 정말 질문에 답하기에 충분한지를 시스템이 스스로 판단하지 못한다는 것입니다. 기본 RAG에서는 검색 결과를 무조건 LLM에 전달하고, LLM이 무조건 답변을 생성합니다. 검색 결과가 부적절해도, 답변이 검색 결과에 근거하지 않아도, 이를 감지하고 수정하는 메커니즘이 없습니다. 이 단원에서는 RAG의 검색과 생성 과정에 자기 점검, 수정, 다단계 추론을 도입하는 고급 패턴들을 다룹니다.

첫 번째 패턴은 **Self-RAG**(자기 반성 RAG)입니다. Self-RAG의 핵심 아이디어는 LLM이 RAG 파이프라인의 각 단계에서 스스로 판단하게 하는 것입니다. 기본 RAG에서 LLM의 역할은 "주어진 문서를 바탕으로 답변을 생성하는 것"에 한정됩니다. Self-RAG에서는 LLM이 추가적인 판단을 수행합니다. "검색이 필요한가?", "검색된 문서가 질문과 관련이 있는가?", "생성한 답변이 문서에 근거하고 있는가?"를 LLM 스스로 평가합니다.

Self-RAG의 동작 흐름은 다음과 같습니다. 사용자의 질문이 들어오면, 먼저 LLM이 이 질문에 외부 검색이 필요한지 판단합니다. "서울의 인구는?"처럼 사실 확인이 필요한 질문이면 검색을 수행하고, "이 시를 해석해 줘"처럼 모델의 기본 능력으로 충분한 질문이면 검색 없이 바로 답합니다. 검색을 수행한 경우, 검색된 각 문서가 질문에 관련이 있는지를 LLM이 평가합니다. 관련성이 낮은 문서는 걸러냅니다. 그 다음 LLM이 답변을 생성하고, 생성된 답변이 검색된 문서에 실제로 근거하고 있는지를 다시 한번 평가합니다.

```
Self-RAG 동작 흐름

사용자 질문
    |
    v
[검색 필요 판단]
  LLM: "이 질문에 검색이 필요한가?"
    |
    +-- 불필요 --> LLM이 직접 답변 생성
    |
    +-- 필요 --> 검색 수행
                    |
                    v
               [관련성 평가]
                LLM: "검색된 문서가 관련 있는가?"
                    |
                    v
               관련 문서만 선별
                    |
                    v
               [답변 생성]
                    |
                    v
               [근거 확인]
                LLM: "답변이 문서에 근거하는가?"
                    |
                    +-- 아니오 --> 답변 재생성 또는 거부
                    |
                    +-- 예 --> 최종 답변 출력
```

Self-RAG에서 LLM이 이 판단을 수행할 수 있는 이유는, 특별한 토큰(reflection token)을 도입하여 모델을 학습시켰기 때문입니다. 모델이 "Retrieve", "IsRel", "IsSup", "IsUse" 같은 특별 토큰을 생성하면서 각 단계의 판단 결과를 표현합니다. 이 접근의 장점은 불필요한 검색을 줄여 효율성을 높이고, 검색 결과의 관련성을 사전에 필터링하며, 답변의 근거를 자체적으로 검증하여 할루시네이션을 줄인다는 것입니다.

두 번째 패턴은 **Corrective RAG**(CRAG)입니다. CRAG의 초점은 "검색 결과가 불충분할 때 어떻게 보정할 것인가"에 있습니다. 기본 RAG에서는 검색 결과가 부실해도 그대로 LLM에 전달합니다. CRAG는 검색 결과의 품질을 평가하고, 품질이 낮으면 보정 조치를 취합니다.

CRAG의 핵심 구성요소는 **검색 평가기**(retrieval evaluator)입니다. 검색된 문서 각각에 대해 "이 문서가 질문에 답하기에 충분한 정보를 담고 있는가"를 평가합니다. 평가 결과에 따라 세 가지 경로로 분기합니다.

```
CRAG 동작 흐름

사용자 질문 --> 검색 수행 --> 검색 결과
                                |
                                v
                      [검색 결과 품질 평가]
                                |
          +---------------------+---------------------+
          |                     |                     |
          v                     v                     v
     [정확 (Correct)]     [모호 (Ambiguous)]    [오류 (Incorrect)]
     검색 결과를           검색 결과 일부 사용   검색 결과 폐기
     그대로 사용           + 추가 검색 수행      웹 검색 등 대안 탐색
          |                     |                     |
          v                     v                     v
                    LLM이 최종 답변 생성
```

첫 번째 경로(Correct)에서는 검색 결과의 품질이 높으므로 그대로 사용합니다. 두 번째 경로(Ambiguous)에서는 일부 문서는 관련이 있지만 불충분하므로, 관련 문서는 유지하면서 추가 검색을 수행합니다. 세 번째 경로(Incorrect)에서는 검색 결과가 질문과 무관하므로, 기존 결과를 폐기하고 웹 검색 같은 대안적 정보원을 탐색합니다.

CRAG의 장점은 검색 실패에 대한 자기 복구 능력입니다. 벡터 검색만으로는 적절한 문서를 찾지 못하는 경우에도, 대안적 경로를 통해 정보를 확보할 수 있습니다.

세 번째 패턴은 **Agentic RAG**입니다. Agentic RAG는 RAG 파이프라인을 하나의 고정된 흐름이 아니라, LLM 에이전트가 상황에 따라 동적으로 결정하는 유연한 과정으로 만드는 패턴입니다.

기본 RAG에서는 "검색 -> 증강 -> 생성"이라는 고정된 단계를 순서대로 실행합니다. Agentic RAG에서는 LLM 에이전트가 질문을 분석하고, 어떤 도구를 사용할지, 어떤 순서로 실행할지를 스스로 결정합니다. 에이전트가 사용할 수 있는 도구에는 벡터 검색, 키워드 검색, 데이터베이스 쿼리, 웹 검색, 계산기 등이 포함될 수 있습니다. 에이전트는 질문의 성격에 따라 적합한 도구를 선택하고, 한 도구의 결과가 불충분하면 다른 도구를 시도하며, 충분한 정보가 모이면 최종 답변을 생성합니다.

예를 들어, "지난달 매출 상위 5개 제품의 고객 리뷰 요약을 알려줘"라는 질문을 생각해 보겠습니다. 이 질문에 답하려면 두 가지 정보가 필요합니다. 매출 데이터(데이터베이스에서 조회)와 고객 리뷰(문서 저장소에서 검색). 기본 RAG는 하나의 벡터 검색만 수행하므로 이 질문을 처리하기 어렵습니다. Agentic RAG에서는 에이전트가 먼저 매출 데이터베이스에서 상위 5개 제품을 조회하고, 각 제품의 이름으로 리뷰 문서를 검색한 뒤, 이 정보를 종합하여 답변을 생성합니다.

```
Agentic RAG 예시 흐름

사용자 질문: "지난달 매출 상위 5개 제품의 리뷰 요약"
    |
    v
[에이전트 추론]
  "매출 데이터가 필요 -> DB 쿼리 도구 사용"
    |
    v
[DB 쿼리] --> 결과: 제품 A, B, C, D, E
    |
    v
[에이전트 추론]
  "각 제품의 리뷰가 필요 -> 벡터 검색 도구 사용"
    |
    v
[벡터 검색] x 5 --> 각 제품의 리뷰 문서
    |
    v
[에이전트 추론]
  "충분한 정보 확보 -> 답변 생성"
    |
    v
최종 답변 생성
```

Agentic RAG의 장점은 복잡한 질문을 분해하여 단계적으로 처리할 수 있다는 것입니다. 고정된 파이프라인으로는 대응하기 어려운 다양한 유형의 질문에 유연하게 대처합니다. 단점은 에이전트의 추론 과정에서 LLM 호출이 여러 번 발생하므로, 지연 시간과 비용이 증가한다는 것입니다.

네 번째 패턴은 **Multi-hop RAG**입니다. Multi-hop RAG는 한 번의 검색으로 답을 찾을 수 없는 질문에 대해, 여러 차례의 검색을 연쇄적으로 수행하는 패턴입니다. "hop"은 검색 한 번을 뜻하며, multi-hop은 여러 번의 검색을 거친다는 의미입니다.

예를 들어, "A 회사의 CEO가 졸업한 대학의 설립 연도는?"이라는 질문을 생각해 보겠습니다. 이 질문에 바로 답하는 문서는 존재하지 않을 가능성이 높습니다. 먼저 "A 회사의 CEO는 누구인가"를 검색하고, 그 결과에서 CEO의 이름을 얻은 뒤, "그 CEO가 졸업한 대학은 어디인가"를 검색하고, 마지막으로 "그 대학의 설립 연도는 언제인가"를 검색해야 합니다.

```
Multi-hop RAG 흐름

질문: "A 회사의 CEO가 졸업한 대학의 설립 연도는?"

  [Hop 1] 검색: "A 회사 CEO"
  결과: "A 회사의 CEO는 김철수입니다."
       |
       v
  [Hop 2] 검색: "김철수 졸업 대학"
  결과: "김철수는 한국대학교를 졸업했습니다."
       |
       v
  [Hop 3] 검색: "한국대학교 설립 연도"
  결과: "한국대학교는 1946년에 설립되었습니다."
       |
       v
  최종 답변: "A 회사의 CEO가 졸업한 한국대학교는 1946년에 설립되었습니다."
```

Multi-hop RAG에서 중요한 것은 각 hop의 검색 쿼리를 어떻게 생성하느냐입니다. 첫 번째 검색의 결과를 바탕으로 다음 검색 쿼리를 만들어야 하므로, LLM이 중간 결과를 해석하고 다음에 무엇을 검색할지를 결정하는 역할을 합니다. 이 과정은 Agentic RAG와 겹치는 부분이 있으며, 실제 구현에서도 에이전트 프레임워크 위에서 Multi-hop RAG를 구현하는 경우가 많습니다.

다섯 번째 패턴은 **Graph RAG**입니다. 지금까지 다룬 RAG 패턴은 모두 문서를 청크 단위로 분할하고, 벡터 유사도로 관련 청크를 검색하는 방식이었습니다. Graph RAG는 문서에서 지식 그래프(knowledge graph)를 추출하고, 그래프 구조를 활용하여 검색하는 패턴입니다.

**지식 그래프**란 개체(entity)와 관계(relation)로 이루어진 그래프입니다. 예를 들어, "김철수는 A 회사의 CEO이다"라는 문장에서 "김철수"와 "A 회사"가 개체이고, "CEO이다"가 관계입니다. 이 정보를 (김철수, CEO이다, A 회사)라는 삼중항(triple)으로 표현합니다. 문서 전체에서 이런 삼중항을 추출하면, 개체들이 어떤 관계로 연결되어 있는지를 나타내는 그래프가 만들어집니다.

Graph RAG는 이 그래프 위에서 검색을 수행합니다. 사용자의 질문에 포함된 개체를 식별하고, 그래프에서 해당 개체와 연결된 관계와 이웃 개체를 탐색합니다. 벡터 유사도 검색으로는 포착하기 어려운, 개체 간의 명시적 관계를 활용할 수 있습니다. 특히 "A와 B의 관계는?", "A에 연결된 모든 X를 나열하라" 같은 관계 중심 질문에서 강점을 보입니다.

```
Graph RAG의 지식 그래프 예시

  [김철수] --CEO--> [A 회사]
  [김철수] --졸업--> [한국대학교]
  [A 회사] --소재--> [서울]
  [한국대학교] --설립--> [1946년]

  질문: "김철수와 관련된 조직은?"
  그래프 탐색: 김철수 -> CEO -> A 회사
              김철수 -> 졸업 -> 한국대학교
  답변 근거: A 회사, 한국대학교
```

Graph RAG의 장점은 개체 간의 구조화된 관계를 활용하여 정확한 정보를 제공할 수 있다는 것입니다. 단점은 지식 그래프 구축 자체에 상당한 비용이 든다는 것입니다. 문서에서 개체와 관계를 추출하려면 LLM을 사용해야 하고, 추출 결과의 정확성을 검증해야 합니다.

마지막으로 **RAG 패턴 선택 가이드**를 정리하겠습니다.

```
패턴            핵심 아이디어             적합한 상황
----------------------------------------------------------------------
Self-RAG       검색/생성의 자기 점검     답변의 신뢰도가 중요한 환경
CRAG           검색 실패 시 보정         검색 결과가 불안정한 환경
Agentic RAG    도구 선택의 동적 결정     복합 질문, 다중 데이터 소스
Multi-hop RAG  연쇄적 다단계 검색        추론이 필요한 질문
Graph RAG      지식 그래프 기반 검색     관계 중심 질문, 구조화된 지식
```

모든 패턴을 한꺼번에 적용할 필요는 없습니다. 기본 RAG(4.1.2)로 시작하고, 검색 품질이 부족하면 4.2.1의 하이브리드 검색과 리랭킹을 적용합니다. 그래도 부족한 부분이 있으면 질문의 유형에 따라 위 패턴 중 적합한 것을 선택적으로 도입합니다. 패턴의 복잡도가 높아질수록 구현과 운영의 비용도 증가하므로, 실제로 해결해야 하는 문제가 무엇인지를 먼저 파악한 뒤 필요한 패턴만 도입하는 것이 효율적입니다.

정리하면, Advanced RAG 패턴은 기본 RAG의 한계를 보완하기 위해 검색과 생성 과정에 자기 점검, 보정, 다단계 추론을 도입합니다. Self-RAG는 LLM이 검색 필요성과 답변의 근거를 스스로 판단합니다. CRAG는 검색 결과의 품질을 평가하고 부족하면 대안적 정보원을 탐색합니다. Agentic RAG는 에이전트가 질문에 따라 도구를 동적으로 선택합니다. Multi-hop RAG는 여러 차례의 연쇄 검색으로 복합 질문에 답합니다. Graph RAG는 지식 그래프를 활용하여 개체 간 관계를 기반으로 검색합니다.

다음 단원인 5.1.1에서는 LLM 오케스트레이션의 개념을 다룹니다. 여러 LLM 호출과 도구를 연결하여 복잡한 워크플로를 구성하는 프레임워크의 기본 개념을 살펴봅니다.

이 단원을 마치면 Self-RAG, CRAG 등 고급 RAG 패턴의 원리를 설명하고 적합한 상황을 판단할 수 있습니다.
