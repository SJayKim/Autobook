# 9.1.1 LLM 옵저버빌리티 개요

LLM 애플리케이션을 프로덕션에 배포하고 나면, 개발 단계에서는 보이지 않던 문제가 하나둘 드러납니다. 어떤 사용자의 요청에 모델이 환각을 일으켰는지, 특정 시간대에 응답 지연이 왜 급증했는지, 한 달 동안 토큰 비용이 얼마나 발생했는지를 추적해야 합니다. 그런데 LLM 호출은 입력과 출력이 모두 자연어 텍스트이고, 같은 입력에도 매번 다른 응답이 나올 수 있어서 전통적인 서버 로그만으로는 "무엇이 잘못되었는지"를 파악하기 어렵습니다. 이 단원에서는 이런 상황을 해결하기 위한 접근법인 옵저버빌리티(observability)의 개념을 살펴보고, LLM 시스템에 특화된 옵저버빌리티가 왜 별도로 필요한지, 어떤 도구와 아키텍처로 구성되는지를 다룹니다.

**옵저버빌리티**(observability)란 시스템의 외부 출력만 관찰하여 내부 상태를 추론할 수 있는 능력을 뜻합니다. 원래 제어 공학에서 나온 개념이지만, 소프트웨어 분야에서는 "서비스가 지금 어떤 상태인지, 왜 그런 상태인지를 확인할 수 있는 체계"라는 의미로 쓰입니다. 옵저버빌리티와 모니터링이라는 말을 혼동하기 쉬운데, 두 개념은 범위가 다릅니다. **모니터링**(monitoring)은 미리 정해 놓은 지표(CPU 사용률, 오류율 등)를 반복적으로 관찰하며 임계값을 넘으면 알림을 보내는 활동입니다. 반면 옵저버빌리티는 사전에 예상하지 못한 문제까지 사후에 추적할 수 있는 역량을 포괄합니다. 모니터링이 "미리 정한 질문에 답하는 것"이라면, 옵저버빌리티는 "아직 던지지 않은 질문에도 답할 수 있도록 데이터를 확보하는 것"입니다.

소프트웨어 옵저버빌리티는 세 가지 핵심 데이터 유형으로 구성됩니다. 이 세 가지를 흔히 옵저버빌리티의 세 기둥(three pillars)이라 부릅니다.

첫 번째 기둥은 **메트릭**(metric)입니다. 메트릭은 시간 축에 따라 수치로 기록되는 측정값입니다. 응답 지연 시간, 초당 요청 수, 오류 발생 건수, 메모리 사용량 같은 것이 메트릭에 해당합니다. 메트릭의 장점은 집계가 쉽다는 것입니다. 수천 건의 요청에서 평균 지연 시간이 얼마인지, 오류율이 몇 퍼센트인지를 빠르게 계산할 수 있습니다. 다만 메트릭은 전체적인 추세를 보여줄 뿐, 개별 요청에서 구체적으로 무슨 일이 일어났는지까지 보여주지는 않습니다.

두 번째 기둥은 **로그**(log)입니다. 로그는 시스템에서 발생한 개별 이벤트를 시간순으로 기록한 것입니다. "2026-04-03 14:32:01에 모델 호출이 시작되었다", "타임아웃 오류가 발생했다" 같은 개별 사건을 텍스트나 구조화된 형태로 남깁니다. 로그는 메트릭과 달리 개별 이벤트의 상세 맥락을 담을 수 있지만, 양이 많아지면 필요한 정보를 찾기 어렵고, 여러 서비스에 흩어진 로그를 하나의 흐름으로 엮기가 까다롭습니다.

세 번째 기둥은 **트레이스**(trace)입니다. 트레이스는 하나의 요청이 시스템을 통과하는 전체 경로를 기록한 것입니다. 예를 들어 사용자의 질문이 들어와서 벡터 검색을 거치고, 프롬프트가 조합되고, LLM이 호출되어 응답이 반환되기까지의 전 과정을 하나의 연결된 흐름으로 보여줍니다. 트레이스는 여러 개의 **스팬**(span)으로 구성됩니다. 스팬은 흐름 안에서 하나의 작업 단위를 나타냅니다. "벡터 검색"이 하나의 스팬이고, "LLM 호출"이 또 하나의 스팬입니다. 각 스팬에는 시작 시간, 종료 시간, 입력값, 출력값, 오류 여부 등이 기록됩니다. 스팬은 부모-자식 관계로 중첩될 수 있어서, 전체 요청 안에서 어떤 단계가 어떤 순서로 실행되었고 각각 얼마나 걸렸는지를 계층적으로 파악할 수 있습니다.

이 세 기둥의 관계를 정리하면 다음과 같습니다.

```
옵저버빌리티의 세 기둥

+-------------------+-------------------+-------------------+
|     메트릭         |      로그          |     트레이스       |
|   (Metrics)       |     (Logs)        |    (Traces)       |
+-------------------+-------------------+-------------------+
| 수치 집계          | 개별 이벤트 기록   | 요청 흐름 추적     |
| 예: 평균 지연 2.1s | 예: 타임아웃 발생  | 예: 검색->조합->   |
|     오류율 0.3%    |     모델 교체 감지 |     LLM호출->응답  |
+-------------------+-------------------+-------------------+
| "전체 추세가       | "이 순간에         | "이 요청이         |
|  어떤가?"          |  무슨 일이?"       |  어떻게 처리?"     |
+-------------------+-------------------+-------------------+
```

메트릭으로 전체 추세를 감시하다가 이상을 발견하면, 로그에서 해당 시점의 이벤트를 확인하고, 트레이스에서 문제가 된 요청의 처리 경로를 따라가 근본 원인을 찾는 흐름이 일반적입니다. 세 기둥이 개별적으로도 유용하지만, 세 가지를 함께 활용할 때 "지금 무슨 문제가 있고, 왜 발생했으며, 어디서 생겼는지"에 대한 답을 종합적으로 얻을 수 있습니다.

여기까지가 소프트웨어 옵저버빌리티의 일반적인 개념입니다. 그렇다면 LLM 시스템에서는 왜 별도의 옵저버빌리티 체계가 필요할까요. 기존 웹 서비스의 옵저버빌리티, 즉 APM(Application Performance Management)은 HTTP 요청의 상태 코드, 응답 시간, 데이터베이스 쿼리 성능 같은 지표를 추적합니다. 이 지표들은 "요청이 성공했는가", "얼마나 빨리 처리되었는가"라는 질문에 답합니다. 그런데 LLM 시스템에서는 요청이 HTTP 200으로 성공하고 응답 시간도 정상이지만, 응답 내용이 사실과 다르거나 맥락에 맞지 않는 상황이 발생합니다. 1.1.1에서 설명한 '조용한 실패(silent failure)'가 바로 이 경우입니다. 기존 APM의 지표만으로는 이런 품질 문제를 감지할 수 없습니다.

LLM 옵저버빌리티가 기존 APM과 구별되는 고유한 요구사항을 하나씩 살펴보겠습니다.

첫째, **입출력 콘텐츠 추적**이 필요합니다. 전통적인 API에서는 요청 본문과 응답 본문을 일일이 저장하지 않는 경우가 많습니다. 구조가 정해져 있어 상태 코드만으로도 성공 여부를 판단할 수 있기 때문입니다. 그러나 LLM 호출에서는 프롬프트 전문과 모델 응답 전문을 저장해야 합니다. 어떤 프롬프트에 어떤 응답이 나왔는지를 볼 수 있어야 환각 여부를 확인하고, 프롬프트 개선 방향을 잡을 수 있습니다.

둘째, **토큰 단위 비용 추적**이 필요합니다. 기존 웹 서비스는 요청 건수나 서버 가동 시간으로 비용을 예측할 수 있습니다. LLM 서비스는 요청마다 소비하는 토큰 수가 다르고, 입력 토큰과 출력 토큰의 단가도 다릅니다. 모델마다 가격이 다르므로, 어떤 모델이 어떤 요청에 얼마의 토큰을 소비했는지를 개별 트레이스 수준에서 추적해야 비용을 통제할 수 있습니다.

셋째, **다단계 파이프라인 추적**이 필요합니다. 5.1.1에서 다룬 것처럼, LLM 애플리케이션은 단일 API 호출이 아니라 여러 단계의 체인으로 구성됩니다. 문서 검색, 프롬프트 조합, 모델 호출, 후처리가 순서대로 이어지고, 에이전트의 경우 도구 호출과 모델 호출이 여러 번 반복됩니다. 기존 APM이 HTTP 요청 단위로 추적하는 것과 달리, LLM 옵저버빌리티는 하나의 사용자 요청 안에서 일어나는 모든 내부 단계를 계층적으로 추적해야 합니다. 이를 위해 각 단계를 스팬으로 표현하고, 스팬을 부모-자식 관계로 연결하여 하나의 트레이스로 묶는 구조가 필요합니다.

넷째, **응답 품질 평가와 연동**이 필요합니다. 기존 APM에서 "품질"이란 주로 가용성(서비스가 작동하는가)과 성능(얼마나 빠른가)을 뜻합니다. LLM 시스템에서는 여기에 "응답이 정확하고 유용한가"라는 차원이 추가됩니다. 8.1.3에서 다룬 자동 평가 파이프라인처럼, 프로덕션에서 수집한 트레이스에 자동 평가자(evaluator)를 붙여 품질 점수를 매기고, 이 점수를 모니터링 지표로 활용하는 체계가 필요합니다. 기존 APM에는 이런 개념이 존재하지 않습니다.

이 네 가지 차이를 기존 APM과 나란히 비교하면 다음과 같습니다.

```
기존 APM vs LLM 옵저버빌리티 비교

관점             기존 APM                    LLM 옵저버빌리티
--------------------------------------------------------------
추적 단위        HTTP 요청/응답               트레이스 (다단계 스팬)
성공 판정        상태 코드 (200/500)          상태 코드 + 응답 품질
비용 추적        서버 자원 (CPU/메모리)       토큰 사용량 (모델별)
저장 대상        헤더, 지연시간, 오류         프롬프트 전문, 응답 전문
품질 측정        가용성, 성능                 가용성 + 정확성 + 관련성
디버깅 초점      코드 오류, 네트워크 장애     환각, 맥락 누락, 프롬프트 결함
```

이 차이 때문에, LLM 시스템의 옵저버빌리티를 기존 APM 도구만으로 구현하기에는 한계가 있습니다. APM은 인프라와 네트워크 계층의 상태를 잘 보여주지만, LLM 호출의 의미적(semantic) 측면인 프롬프트 내용, 응답 품질, 토큰 비용을 다루는 기능이 없습니다. 그래서 LLM 옵저버빌리티에 특화된 도구들이 등장했습니다.

LLM 옵저버빌리티 도구 생태계는 크게 세 가지 유형으로 나눌 수 있습니다.

첫 번째 유형은 **LLM 전용 옵저버빌리티 플랫폼**입니다. LLM 호출의 트레이싱, 프롬프트 추적, 토큰 비용 계산, 평가 연동을 핵심 기능으로 갖춘 도구들입니다. LangSmith, Langfuse, Arize Phoenix가 이 유형에 해당합니다. LangSmith는 LangChain 생태계와 깊이 통합되면서도 OpenAI SDK, Anthropic SDK, LlamaIndex 등 다양한 프레임워크와 연동됩니다. Langfuse는 오픈소스로 셀프 호스팅이 가능하며 MIT 라이선스로 제공됩니다. Arize Phoenix는 임베딩 드리프트 탐지와 LLM 평가를 결합한 접근을 취합니다. 이 도구들은 다음 단원인 9.1.2부터 9.1.4에서 각각 상세히 다룹니다.

두 번째 유형은 **기존 옵저버빌리티 플랫폼의 LLM 확장**입니다. Datadog, New Relic 같은 기존 APM 도구들이 LLM 추적 기능을 추가하고 있습니다. 이 접근의 장점은 이미 인프라 모니터링에 사용 중인 도구와 같은 화면에서 LLM 지표를 볼 수 있다는 것입니다. 다만 LLM 전용 플랫폼에 비해 트레이싱 깊이나 평가 연동 기능이 제한적일 수 있습니다.

세 번째 유형은 **범용 관측 도구와의 조합**입니다. Prometheus로 메트릭을 수집하고 Grafana로 시각화하는 방식이 대표적입니다. LLM 전용 플랫폼에서 정성적 트레이스 데이터(프롬프트 입출력, 단계별 실행 내역)를 수집하고, Prometheus에서 정량적 메트릭(지연 시간, 토큰 사용량, 오류율)을 시계열로 저장하며, Grafana에서 대시보드로 시각화하는 구성입니다. LLM 전용 플랫폼이 "무엇이 문제인가"를 보여주고, Prometheus-Grafana 조합이 "언제 얼마나 문제인가"를 보여주는 역할 분담이 됩니다.

LLM 옵저버빌리티 아키텍처를 설계할 때는 데이터가 수집되고 활용되는 흐름을 단계별로 구성합니다. 이 흐름은 네 단계로 나�� 수 있습니다.

```
LLM 옵저버빌리티 아키텍처 흐름

1. 계측             2. 수집              3. 저장/분석         4. 활용
(Instrumentation)   (Collection)         (Storage/Analysis)   (Action)

+---------------+   +---------------+   +----------------+   +-------------+
| 애플리케이션  |   | 트레이스 수집 |   | 시계열 DB      |   | 대시보드    |
| 코드에 계측   |-->| 서버로 전송   |-->| (메트릭)       |-->| 알림        |
| 포인트 삽입   |   |               |   | 트레이스 저장소|   | 평가 연동   |
| (@traceable,  |   | (SDK 배치     |   | (프롬프트/응답)|   | 디버깅      |
|  OTel 등)     |   |  전송)        |   | 로그 저장소    |   | 비용 분석   |
+---------------+   +---------------+   +----------------+   +-------------+
```

첫 번째 단계는 **계측**(instrumentation)입니다. 애플리케이션 코드의 주요 지점에 트레이싱 코드를 삽입하여, 각 단계의 입력, 출력, 소요 시간, 메타데이터를 캡처합니다. 계측 방법은 도구에 따라 다릅니다. LangSmith의 경우 Python 함수에 `@traceable` 데코레이터를 붙이거나, OpenAI 클라이언트를 `wrap_openai()`로 래핑하는 방식을 씁니다. LangChain이나 LangGraph를 사용하면 환경 변수 설정만으로 자동 계측이 됩니다. 계측 코드를 삽입하지 않아도 작동하는 이런 방식을 제로 코드 계측(zero-code instrumentation)이라 부릅니다.

두 번째 단계는 **수집**(collection)입니다. 계측 포인트에서 생성된 트레이스와 메트릭 데이터를 옵저버빌리티 플랫폼의 서버로 전송합니다. 대부분의 SDK는 데이터를 즉시 보내지 않고 일정량을 모아서 배치로 전송하여 애플리케이션 성능에 미치는 영향을 줄입니다.

세 번째 단계는 **저장과 분석**(storage and analysis)입니다. 수집된 데이터는 유형에 따라 다른 저장소에 보관됩니다. 수치 메트릭은 시계열 데이터베이스에, 트레이스와 프롬프트/응답 전문은 트레이스 전용 저장소에, 이벤트 로그는 로그 저장소에 저장됩니다. LLM 전용 플랫폼은 이 저장소들을 내부적으로 통합하여 제공하는 경우가 많습니다.

네 번째 단계는 **활용**(action)입니다. 저장된 데이터를 기반으로 대시보드를 구성하고, 임계값 초과 시 알림을 보내고, 수집된 트레이스에 자동 평가자를 적용하여 품질 점수를 매기고, 문제가 발생한 트레이스를 추적하여 디버깅하며, 비용 추이를 분석합니다. 이 단계가 옵저버빌리티의 궁극적 목적입니다. 단순히 데이터를 모으는 것이 아니라, 모은 데이터로 행동 가능한 정보(actionable insight)를 얻는 것입니다.

이 네 단계에서 핵심적인 역할을 하는 기술 표준이 **OpenTelemetry**(줄여서 OTel)입니다. OpenTelemetry는 메트릭, 로그, 트레이스를 수집하고 전송하기 위한 오픈소스 표준입니다. 특정 벤더에 종속되지 않는 중립적인 계측 방식을 제공하여, 옵저버빌리티 도구를 교체하더라도 계측 코드를 다시 작성할 필요가 없습니다.

LLM 옵저버빌리티에서 OpenTelemetry가 중요한 이유는 두 가지입니다. 하나는 기존 인프라 옵저버빌리티와의 통합입니다. 이미 OpenTelemetry로 웹 서버, 데이터베이스, 메시지 큐의 트레이스를 수집하고 있는 조직이라면, LLM 호출의 트레이스도 같은 파이프라인으로 통합할 수 있습니다. 예를 들어, 웹 서버에서 시작된 HTTP 요청 트레이스 안에 LLM 호출 스팬이 자연스럽게 포함되어, 전체 요청 경로를 하나의 트레이스로 볼 수 있게 됩니다. 다른 하나는 도구 유연성입니다. OpenTelemetry 형식으로 트레이스를 내보내면, 수신 측을 LangSmith, Langfuse, Jaeger, Datadog 중 어디로든 바꿀 수 있습니다. LangSmith는 `POST /otel/v1/traces` 엔드포인트를 통해 OpenTelemetry 트레이스를 수신하며, SDK에서는 `LANGSMITH_OTEL_ENABLED` 환경 변수를 설정하여 OTel 기반 트레이스 내보내기를 활성화합니다.

LLM 트레이싱에서 OpenTelemetry의 스팬 개념이 어떻게 적용되는지를 좀 더 구체적으로 살펴보겠습니다. OpenTelemetry에서 하나의 트레이스는 루트 스팬을 꼭대기로 하는 트리 구조입니다. 각 스팬에는 고유 ID가 있고, 부모 스팬의 ID를 참조하여 계층을 형성합니다. LLM 옵저버빌리티 도구들은 이 구조를 LLM 파이프라인에 맞게 활용합니다. 예를 들어, LangSmith에서는 하나의 트레이스가 여러 개의 **런**(run)으로 구성됩니다. 런은 OpenTelemetry의 스팬에 대응하는 개념으로, 단일 작업 단위를 나타냅니다. 루트 런이 전체 요청을 대표하고, 그 아래에 LLM 호출, 도구 호출, 검색 등의 자식 런이 중첩됩니다. 각 런에는 유형(llm, chain, tool, retriever 등)이 지정되어 있어, 어떤 종류의 작업인지 즉시 구분할 수 있습니다.

RAG 파이프라인을 예로 들면, 트레이스 구조는 다음과 같은 형태가 됩니다.

```
RAG 파이프라인의 트레이스 구조 예시

[루트 런] rag_pipeline (chain) -------- 전체 소요: 3.2s
  |
  +-- [자식 런] retriever (retriever) -- 소요: 0.4s
  |     입력: "LLM 옵저버빌리티란?"
  |     출력: [문서1, 문서2, 문서3]
  |
  +-- [자식 런] prompt_format (chain) -- 소요: 0.01s
  |     입력: 질문 + 검색 문서
  |     출력: 조합된 프롬프트
  |
  +-- [자식 런] llm_call (llm) -------- 소요: 2.7s
        입력: 조합된 프롬프트
        출력: 모델 응답 텍스트
        토큰: 입력 1,200 / 출력 350
        비용: $0.0045
```

이 트레이스를 보면, 전체 3.2초 중 LLM 호출이 2.7초를 차지하고 있다는 것을 바로 알 수 있습니다. 검색 단계에서 어떤 문서가 반환되었는지, 프롬프트에 어떤 내용이 포함되었는지, 모델이 어떤 응답을 생성했는지도 확인할 수 있습니다. 응답에 문제가 있다면, 검색된 문서가 부적절했는지(검색 단계의 문제), 프롬프트 구성이 잘못되었는지(조합 단계의 문제), 모델 자체가 잘못 추론했는지(LLM 호출 단계의 문제)를 단계별로 추적하여 근본 원인을 좁혀 갈 수 있습니다.

정리하면, LLM 옵저버빌리티는 LLM 애플리케이션의 내부 동작을 투명하게 관찰하기 위한 체계입니다. 메트릭, 로그, 트레이스라는 세 기둥으로 구성된 소프트웨어 옵저버빌리티의 기본 개념 위에, LLM 고유의 요구사항인 프롬프트/응답 콘텐츠 추적, 토큰 단위 비용 계산, 다단계 파이프라인 추적, 응답 품질 평가 연동이 더해집니다. 기존 APM이 "서비스가 살아 있는가, 빠른가"에 초점을 맞추는 데 비해, LLM 옵저버빌리티는 "응답이 정확한가, 비용은 적정한가, 어디서 문제가 생겼는가"까지 다룹니다. LangSmith, Langfuse, Arize Phoenix 같은 LLM 전용 플랫폼과 OpenTelemetry 같은 오픈 표준이 이 체계를 뒷받침하며, 계측, 수집, 저장, 활용의 네 단계로 아키텍처가 구성됩니다.

다음 단원인 9.1.2에서는 LLM 옵저버빌리티 도구 중 하나인 LangSmith의 트레이싱, 평가, 비용 추적 기능을 구체적으로 다룹니다.

이 단원을 마치면 LLM 옵저버빌리티의 핵심 구성요소를 설명하고 기존 APM과의 차이를 구분할 수 있습니다.
