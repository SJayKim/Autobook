# 3.2.5 벡터 인덱스 관리와 최적화

3.2.1에서 벡터 데이터베이스가 ANN(Approximate Nearest Neighbor) 검색으로 유사한 벡터를 빠르게 찾는다는 점을 다루었습니다. 그런데 "빠르게"라는 말 뒤에는 구체적인 메커니즘이 숨어 있습니다. 벡터 수백만 개가 저장된 데이터베이스에서 쿼리 벡터와 가장 가까운 벡터를 찾으려면, 모든 벡터를 하나하나 비교하는 것은 현실적이지 않습니다. 100만 개의 768차원 벡터가 있다면, 쿼리 한 번에 100만 번의 거리 계산이 필요합니다. 응답 시간이 수 초에서 수십 초까지 늘어나고, 동시에 여러 사용자가 검색하면 시스템이 감당할 수 없습니다.

이 문제를 해결하는 것이 **벡터 인덱스**입니다. 벡터 인덱스란 벡터들을 특정 자료 구조로 미리 정리해 두어, 검색 시 전체를 훑지 않고도 가까운 벡터를 빠르게 찾아낼 수 있게 하는 장치입니다. 관계형 데이터베이스에서 B-tree 인덱스가 행 전체를 스캔하지 않고 원하는 행을 찾게 해 주는 것과 비슷한 역할입니다. 다만 벡터 인덱스는 "정확히 같은 값"이 아니라 "가장 가까운 값"을 찾는다는 점에서 근본적으로 다른 설계가 필요합니다.

벡터 인덱스 알고리즘은 여러 종류가 있고, 각각 검색 속도, 정확도, 메모리 사용량 사이에서 다른 균형점을 잡습니다. 3.2.2에서 다룬 Pinecone, 3.2.3에서 다룬 Weaviate, 3.2.4에서 다룬 Qdrant와 Milvus 모두 내부적으로 이러한 인덱스 알고리즘을 사용합니다. 이 단원에서는 실무에서 가장 널리 쓰이는 세 가지 알고리즘인 HNSW, IVF, PQ를 다루고, 인덱스 파라미터를 조정하여 성능을 튜닝하는 방법을 살펴봅니다.

먼저 가장 기본이 되는 검색 방식부터 짚겠습니다. 쿼리 벡터가 들어오면 저장된 모든 벡터와 하나씩 거리를 계산하여 가장 가까운 k개를 돌려주는 방식을 **Flat 검색**(또는 brute-force 검색)이라고 합니다. 이 방식은 정확도가 완벽합니다. 가장 가까운 벡터를 절대 놓치지 않습니다. 그러나 벡터 수가 늘어나면 검색 시간이 선형으로 증가합니다. 벡터가 1만 개 이하일 때는 충분히 빠르지만, 수십만 개를 넘어가면 실용적이지 않습니다.

이 한계를 넘기 위해 등장한 것이 ANN 알고리즘입니다. ANN은 "가장 가까운 이웃을 반드시 찾겠다"는 보장을 약간 완화하는 대신, 검색 속도를 극적으로 높입니다. 실제로 대부분의 경우 진짜 최근접 이웃을 찾지만, 드물게 놓칠 수 있습니다. 이 "찾아낸 비율"을 **재현율**(recall)이라고 합니다. 재현율 0.95는 진짜 최근접 이웃 100개 중 95개를 찾았다는 뜻입니다. ANN 알고리즘의 목표는 재현율을 충분히 높게 유지하면서 검색 시간을 획기적으로 줄이는 것입니다.

**HNSW**(Hierarchical Navigable Small World)는 현재 벡터 데이터베이스에서 가장 널리 사용되는 ANN 알고리즘입니다. 3.2.2의 Pinecone, 3.2.3의 Weaviate, 3.2.4의 Qdrant와 Milvus 모두 HNSW를 기본 인덱스로 지원합니다.

HNSW의 원리를 이해하려면 "Small World 네트워크"라는 개념부터 시작해야 합니다. Small World 네트워크란 노드 수가 아무리 많아도, 소수의 연결만 거치면 임의의 두 노드에 도달할 수 있는 네트워크를 말합니다. 사회학에서 말하는 "여섯 다리만 건너면 아는 사이"와 같은 원리입니다. HNSW는 벡터 하나하나를 노드로 놓고, 가까운 벡터끼리 연결선(edge)을 만들어 이런 Small World 네트워크를 구성합니다.

HNSW에서 "Hierarchical"은 이 네트워크를 여러 계층(layer)으로 쌓는다는 뜻입니다. 맨 위 계층에는 소수의 노드만 존재하고, 아래 계층으로 갈수록 노드 수가 많아집니다. 가장 아래 계층(Layer 0)에는 모든 벡터가 포함됩니다.

검색은 꼭대기 계층에서 시작합니다. 꼭대기 계층에 노드가 몇 개 없으므로 그중에서 쿼리 벡터와 가장 가까운 노드를 빠르게 찾습니다. 이 노드를 진입점으로 삼아 한 계층 아래로 내려갑니다. 아래 계층에는 노드가 더 많으므로 진입점 주변에서 더 가까운 이웃을 찾습니다. 이 과정을 최하위 계층까지 반복하면, 전체 벡터를 순회하지 않고도 가까운 벡터에 도달합니다.

```
Layer 2 (최상위):    [A] ------------ [B]
                      |
Layer 1 (중간):      [A] --- [C] --- [B] --- [D]
                      |       |       |       |
Layer 0 (최하위):    [A]-[E]-[C]-[F]-[B]-[G]-[D]-[H]
```

위 다이어그램에서 쿼리 벡터가 들어오면 Layer 2에서 A와 B 중 더 가까운 쪽을 고릅니다. A가 더 가깝다면 Layer 1로 내려가 A 주변의 C를 확인합니다. 다시 Layer 0으로 내려가 C 주변의 E와 F를 비교하여 최종 결과를 결정합니다. 벡터가 수백만 개여도 계층 구조 덕분에 탐색 횟수가 로그 단위로 유지됩니다. Flat 검색이 벡터 수에 비례하여 느려지는 것과 대조적입니다.

HNSW 그래프를 만들 때 조정할 수 있는 핵심 파라미터가 두 가지 있습니다. 첫째는 **m** 파라미터입니다. m은 각 노드가 가질 수 있는 연결 수의 상한입니다. m이 크면 노드마다 더 많은 이웃과 연결되어 검색 정확도가 올라갑니다. 반면 그래프가 차지하는 메모리가 늘어나고, 인덱스를 구축하는 시간도 길어집니다. 일반적으로 기본값 16에서 시작합니다.

둘째는 **ef_construction** 파라미터입니다. 그래프를 만드는 과정에서 후보로 고려하는 이웃의 수를 뜻합니다. 이 값이 클수록 더 많은 후보를 살펴보며 연결선을 만들기 때문에 그래프 품질이 높아집니다. 대신 인덱스 빌드 시간이 늘어납니다. 일반적으로 100에서 200 사이의 값을 사용합니다.

검색 시점에는 **ef**(또는 ef_search)라는 별도의 파라미터가 사용됩니다. ef는 검색 과정에서 후보로 유지하는 노드 수입니다. ef가 클수록 더 넓은 범위를 탐색하므로 재현율이 높아지지만, 검색 시간이 늘어납니다. ef가 작으면 빠르지만 좋은 결과를 놓칠 수 있습니다. ef는 반환할 결과 수 k 이상이어야 합니다. 실무에서는 k의 5배에서 10배로 시작하여 재현율과 응답 시간을 측정하면서 조정합니다.

HNSW의 장점은 높은 재현율과 빠른 검색 속도입니다. 파라미터를 적절히 설정하면 재현율 0.95 이상을 밀리초 단위 응답 시간으로 달성할 수 있습니다. 단점은 메모리 사용량입니다. 그래프 구조를 메모리에 올려야 하므로, 벡터 원본 데이터에 더해 그래프 포인터만큼의 추가 메모리가 필요합니다. 벡터 수가 늘어날수록 이 비용이 커집니다. 메모리 계산 공식은 다음과 같습니다.

```
메모리(바이트) = 벡터 수 x 4 x (차원 수 + 12)
```

768차원 벡터 2천만 개를 저장한다면 약 62.4GB가 필요합니다. 이 데이터가 메모리에 들어가야 검색 성능이 보장됩니다.

**IVF**(Inverted File Index)는 HNSW와 다른 방식으로 검색 범위를 줄이는 알고리즘입니다. IVF의 핵심 아이디어는 벡터 공간을 여러 구역으로 미리 나누어 두고, 검색 시 쿼리 벡터와 가까운 구역만 살펴보는 것입니다.

IVF는 인덱스를 만드는 단계에서 k-means 클러스터링을 수행합니다. 전체 벡터를 정해진 수의 클러스터로 나누고, 각 클러스터의 중심점(centroid)을 기록합니다. 벡터를 저장할 때 가장 가까운 클러스터에 배정합니다.

검색 시에는 다음 과정을 거칩니다.

```
[인덱스 구축]
1. 전체 벡터를 k-means로 N개 클러스터로 분할
2. 각 클러스터의 중심점(centroid) 저장
3. 각 벡터를 가장 가까운 클러스터에 배정

[검색]
1. 쿼리 벡터와 모든 centroid의 거리 계산
2. 가장 가까운 nprobe개 클러스터 선택
3. 선택된 클러스터 안의 벡터들만 거리 계산
4. 상위 k개 반환
```

여기서 **nprobe**가 IVF의 핵심 튜닝 파라미터입니다. nprobe는 검색 시 방문할 클러스터 수를 지정합니다. 전체 클러스터가 1,000개인데 nprobe를 10으로 설정하면, 전체 벡터의 약 1%만 실제로 거리 계산을 합니다. 검색 속도가 크게 빨라지는 이유입니다.

nprobe를 늘리면 더 많은 클러스터를 살펴보므로 재현율이 올라갑니다. nprobe를 전체 클러스터 수와 같게 설정하면 Flat 검색과 동일해집니다. 반대로 nprobe가 너무 작으면 가까운 벡터가 이웃 클러스터에 있는 경우 놓칠 수 있습니다. 벡터 공간의 클러스터 경계 근처에 있는 벡터가 다른 클러스터에 배정되는 경우가 이에 해당합니다.

IVF의 장점은 메모리 효율입니다. HNSW처럼 그래프 구조를 통째로 메모리에 올리지 않아도 됩니다. 중심점 목록과 클러스터 배정 정보만 있으면 되므로 메모리 부담이 적습니다. 대규모 데이터셋에서 메모리 제약이 있을 때 유리합니다. 단점은 클러스터 경계 효과로 인해 HNSW보다 재현율이 낮을 수 있다는 점입니다. 또한 인덱스를 만들기 전에 k-means 학습이 필요하므로, 데이터가 충분히 모인 뒤에야 인덱스를 생성할 수 있습니다.

Milvus에서는 IVF_FLAT이라는 이름으로 이 알고리즘을 제공합니다. FLAT은 클러스터 안에서 벡터를 원본 그대로 비교한다는 뜻입니다. 여기에 뒤에서 설명할 PQ를 결합한 IVF_PQ도 있습니다.

**PQ**(Product Quantization)는 벡터 자체를 압축하여 메모리 사용량을 줄이는 기법입니다. HNSW나 IVF가 "어디를 검색할지"를 효율화하는 것이라면, PQ는 "벡터를 얼마나 작게 저장할지"를 효율화합니다. 두 접근은 독립적이므로 함께 사용할 수 있습니다.

PQ의 원리를 이해하기 위해 768차원 벡터를 예로 들겠습니다. 이 벡터를 float32로 저장하면 768 곱하기 4바이트, 즉 3,072바이트가 필요합니다. PQ는 이 벡터를 여러 조각(sub-vector)으로 나눕니다. 768차원을 8개 조각으로 나누면 각 조각은 96차원입니다.

각 조각 영역에서 전체 벡터들의 해당 조각을 모아 k-means 클러스터링을 수행합니다. 보통 256개의 클러스터를 만듭니다. 256개라면 클러스터 번호를 1바이트(0~255)로 표현할 수 있습니다. 이제 원본 96차원 조각(384바이트) 대신 클러스터 번호 1바이트만 저장합니다. 8개 조각이므로 벡터 하나가 8바이트로 압축됩니다. 원본 3,072바이트 대비 약 384분의 1로 줄어드는 것입니다.

```
원본 벡터 (768차원, 3,072바이트):
[v1, v2, ..., v96 | v97, ..., v192 | ... | v673, ..., v768]
 --- 조각 1 ---    --- 조각 2 ---         --- 조각 8 ---

PQ 압축 후 (8바이트):
[code1, code2, code3, code4, code5, code6, code7, code8]
 각 code는 0~255 사이의 클러스터 번호 (1바이트)
```

검색 시에는 쿼리 벡터의 각 조각과 256개 클러스터 중심점 사이의 거리를 미리 계산해 둡니다. 이 테이블을 만들어 두면, 저장된 벡터의 코드값만 보고 바로 거리를 조회할 수 있습니다. 원본 벡터끼리 거리를 계산하는 것보다 훨씬 빠릅니다.

PQ의 단점은 압축 과정에서 정보 손실이 발생한다는 점입니다. 클러스터 중심점으로 대체하므로 거리 계산에 오차가 생깁니다. 이 오차 때문에 재현율이 떨어질 수 있습니다. 그러나 메모리를 수십 배 절약할 수 있으므로, 벡터가 수억 개에 달하는 대규모 시스템에서는 PQ 없이 서비스하기 어려운 경우가 많습니다.

IVF와 PQ를 함께 사용하는 IVF_PQ는 두 기법의 이점을 결합합니다. IVF로 검색 범위를 좁히고, PQ로 벡터를 압축하여 메모리를 절약합니다. Milvus에서는 IVF_PQ 인덱스를 기본으로 제공하며, 대규모 데이터셋에서 널리 사용됩니다.

지금까지 설명한 세 알고리즘의 특성을 정리하면 다음과 같습니다.

```
알고리즘      검색 방식            메모리        재현율       빌드 속도
----------  -------------------  -----------  ----------  ----------
HNSW        그래프 계층 탐색      높음          매우 높음    느림
IVF         클러스터 선택 후 탐색  중간          높음        중간
PQ          벡터 압축 + 조회      매우 낮음     중간        느림
IVF_PQ      클러스터 + 압축       낮음          중간~높음   중간
```

HNSW는 메모리 여유가 있고 높은 재현율이 중요한 경우에 적합합니다. IVF는 메모리가 제한적이고 데이터가 대규모일 때 유용합니다. PQ는 단독보다 IVF나 HNSW와 결합하여 메모리 절감 목적으로 사용하는 것이 일반적입니다.

인덱스 알고리즘을 선택한 뒤에는 파라미터 튜닝이 필요합니다. 같은 알고리즘이라도 파라미터에 따라 성능이 크게 달라집니다. 앞에서 언급한 파라미터를 다시 한 번 정리합니다.

HNSW에서 m은 노드당 연결 수 상한입니다. 기본값 16에서 시작하되, 재현율이 부족하면 32로 올릴 수 있습니다. m을 올리면 메모리 사용량과 인덱스 빌드 시간이 함께 늘어납니다. ef_construction은 인덱스 구축 시 탐색 후보 수입니다. 100에서 200 사이가 일반적입니다. ef(ef_search)는 검색 시 탐색 후보 수이며, k의 5~10배에서 시작합니다.

IVF에서 nprobe는 검색 시 방문할 클러스터 수입니다. 전체 클러스터 수의 1~10%에서 시작하여 재현율을 측정하며 조정합니다. 클러스터 수(nlist)는 보통 벡터 수의 제곱근 근처로 설정합니다. 벡터가 100만 개라면 nlist를 1,000 정도로 잡는 것이 출발점입니다.

PQ에서 조각 수(m_pq, 위의 HNSW m과 다른 파라미터)는 벡터를 몇 개로 나눌지를 결정합니다. 조각이 많으면 압축률이 높아지지만 정밀도가 떨어집니다. 각 조각의 코드북 크기는 보통 256(8비트)을 사용합니다.

모든 인덱스 파라미터 튜닝의 핵심은 **재현율과 응답 시간의 트레이드오프**입니다. 재현율을 높이면 응답 시간이 늘어나고, 응답 시간을 줄이면 재현율이 떨어집니다. 이 관계는 직관적입니다. 더 넓은 범위를 탐색할수록 정확한 결과를 찾을 가능성이 높아지지만, 그만큼 계산에 시간이 걸립니다.

실무에서는 이 트레이드오프를 정량적으로 측정합니다. 테스트 데이터셋에서 Flat 검색(brute-force)으로 정답을 구해 두고, ANN 검색 결과와 비교하여 재현율을 계산합니다. 파라미터를 바꿔 가며 재현율과 응답 시간을 기록하면, 시스템 요구 사항에 맞는 최적점을 찾을 수 있습니다.

예를 들어 실시간 채팅 서비스라면 응답 시간 50ms 이내가 필수일 수 있습니다. 이 범위 안에서 재현율을 최대화하는 파라미터를 찾습니다. 반대로 문서 분석처럼 배치 처리 위주라면 응답 시간에 여유가 있으므로 재현율 0.99 이상을 목표로 파라미터를 높게 잡을 수 있습니다.

인덱스는 한 번 만들면 영구적이지 않습니다. 시스템 운영 과정에서 인덱스를 다시 만들거나 유지보수해야 하는 상황이 발생합니다.

첫째, 데이터가 크게 변할 때입니다. 벡터가 대량으로 추가되거나 삭제되면 인덱스 품질이 저하됩니다. HNSW의 경우 삭제된 노드가 그래프에 빈 공간을 남기고, IVF의 경우 클러스터 분포가 실제 데이터 분포와 달라집니다. 이런 경우 인덱스를 처음부터 다시 구축하면 검색 품질이 회복됩니다.

둘째, 파라미터를 변경할 때입니다. m이나 ef_construction 같은 파라미터는 인덱스 구축 시점에 결정되므로, 변경하려면 인덱스를 새로 만들어야 합니다. nprobe나 ef_search처럼 검색 시점에 적용되는 파라미터는 인덱스 재구축 없이 변경할 수 있습니다.

셋째, 임베딩 모델을 교체할 때입니다. 새 모델은 기존 모델과 다른 벡터 공간을 생성합니다. 기존 인덱스의 벡터와 새 모델의 벡터를 같은 인덱스에서 검색하면 의미 있는 결과가 나오지 않습니다. 모든 문서를 새 모델로 다시 임베딩하고 인덱스를 새로 구축해야 합니다.

인덱스 재구축은 시간과 자원이 많이 드는 작업입니다. 운영 중인 서비스에서 재구축하려면 기존 인덱스로 검색을 계속 서비스하면서 백그라운드에서 새 인덱스를 만들고, 완성되면 전환하는 방식이 일반적입니다. 3.2.4에서 다룬 Milvus는 컬렉션 단위로 인덱스를 관리하여 이런 전환을 지원하고, Qdrant는 컬렉션의 복제본을 활용하여 무중단 재구축을 수행합니다.

정리하면, 벡터 인덱스는 대량의 벡터에서 유사한 벡터를 빠르게 찾기 위한 자료 구조입니다. HNSW는 계층적 그래프를 사용하여 높은 재현율과 빠른 검색을 제공하지만 메모리를 많이 사용합니다. IVF는 벡터 공간을 클러스터로 나누어 검색 범위를 줄이며 메모리 효율이 좋습니다. PQ는 벡터를 압축하여 메모리를 크게 절약합니다. 각 알고리즘의 파라미터를 조정하여 재현율과 응답 시간의 균형을 잡는 것이 벡터 인덱스 최적화의 핵심입니다.

다음 단원인 4.1.1에서는 RAG의 개념과 필요성을 다룹니다. 벡터 데이터베이스에 저장된 문서를 검색하여 LLM의 답변 품질을 높이는 기법으로 넘어갑니다.

이 단원을 마치면 주요 벡터 인덱스 알고리즘의 원리를 설명하고 성능을 튜닝할 수 있습니다.
