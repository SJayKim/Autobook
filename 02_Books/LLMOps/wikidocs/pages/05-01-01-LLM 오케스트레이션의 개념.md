# 5.1.1 LLM 오케스트레이션의 개념

LLM을 활용하는 애플리케이션은 대부분 모델 호출 한 번으로 끝나지 않습니다. 사용자의 질문을 받아 외부 문서를 검색하고, 검색 결과를 프롬프트에 조합한 뒤 모델에 전달하며, 모델의 응답을 후처리하여 최종 결과를 반환하는 식으로 여러 단계가 이어집니다. 단계가 늘어날수록 각 단계 사이의 데이터 흐름을 관리하고, 오류가 발생했을 때 어디서 문제가 생겼는지 추적하며, 단계의 순서와 조건을 유연하게 바꿀 수 있어야 합니다. 이런 복잡한 흐름을 체계적으로 조율하는 것이 바로 오케스트레이션입니다.

**오케스트레이션**(orchestration)이라는 단어는 원래 관현악단의 지휘자가 여러 악기의 연주를 하나의 곡으로 엮어내는 행위를 뜻합니다. 소프트웨어 분야에서는 여러 개의 독립된 서비스, 함수, 또는 처리 단계를 정해진 순서와 조건에 따라 연결하고 조율하는 것을 가리킵니다. LLM 오케스트레이션에서는 이 대상이 프롬프트 구성, 모델 호출, 외부 데이터 검색, 도구 실행, 응답 파싱 등 LLM 애플리케이션을 구성하는 개별 처리 단계가 됩니다. 1.1.1에서 LLMOps 생태계의 네 계층을 살펴볼 때, 모델 계층 위에 '오케스트레이션 계층'이 자리하여 프롬프트 구성, RAG, 에이전트, 도구 호출 등 애플리케이션의 흐름을 조율한다고 설명한 바 있습니다. 이 단원에서는 그 오케스트레이션 계층이 구체적으로 무엇을 하고, 왜 필요하며, 어떤 프레임워크들이 이 역할을 맡고 있는지를 다룹니다.

오케스트레이션이 왜 필요한지 이해하려면, 오케스트레이션 없이 LLM 애플리케이션을 만드는 경우를 먼저 살펴보는 것이 도움이 됩니다. 가장 단순한 형태의 LLM 활용은 API를 한 번 호출하는 것입니다. 예를 들어, 사용자가 질문을 입력하면 그 질문을 그대로 LLM API에 보내고, 돌아온 응답을 화면에 표시하는 방식입니다. 이 경우 코드는 매우 단순합니다.

```python
import openai

response = openai.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "Python의 GIL이 무엇인가요?"}]
)
print(response.choices[0].message.content)
```

이 코드에서 일어나는 일은 명확합니다. 사용자의 질문을 messages 리스트에 담아 모델에 전달하고, 모델이 생성한 응답의 텍스트를 꺼내 출력합니다. model 인자는 사용할 모델을 지정하고, messages는 대화 이력을 담는 리스트입니다. 단일 질문-응답이므로 사용자 메시지 하나만 들어 있습니다.

이렇게 단순한 호출은 프로토타입이나 일회성 실험에는 충분합니다. 그러나 실제 서비스에서는 이 단순 호출만으로 해결할 수 없는 요구사항이 빠르게 쌓입니다. 대표적인 상황을 세 가지 들어 보겠습니다.

첫째, 모델이 알지 못하는 정보를 답해야 하는 상황입니다. LLM은 학습 시점까지의 데이터만 알고 있으므로, 회사 내부 문서나 최신 정보에 대해서는 정확한 답을 할 수 없습니다. 이를 해결하려면 사용자의 질문을 받아 관련 문서를 벡터 데이터베이스에서 검색하고, 검색된 문서 내용을 프롬프트에 포함시킨 뒤 모델에 전달해야 합니다. 이것이 4단원에서 다룬 RAG(Retrieval-Augmented Generation)입니다. RAG를 구현하려면 질문 임베딩, 벡터 검색, 프롬프트 조합, 모델 호출이라는 네 단계를 순서대로 실행해야 합니다.

둘째, 모델의 응답을 특정 형식으로 가공해야 하는 상황입니다. 예를 들어, 모델에게 JSON 형식으로 답하라고 요청했는데 모델이 마크다운 코드 블록으로 감싸서 응답하거나, 일부 필드를 누락하는 경우가 있습니다. 이때 응답을 파싱하고, 형식이 맞지 않으면 재시도하는 로직이 필요합니다.

셋째, 하나의 작업을 여러 번의 모델 호출로 나누어야 하는 상황입니다. 긴 문서를 요약할 때, 문서 전체를 한 번에 모델에 넣으면 컨텍스트 길이 제한에 걸릴 수 있습니다. 이때는 문서를 여러 조각으로 나누어 각각 요약한 뒤, 부분 요약들을 다시 합쳐 최종 요약을 만드는 방식을 씁니다. 이 과정에서 문서 분할, 반복 호출, 결과 병합이라는 여러 단계가 발생합니다.

이 세 가지 상황의 공통점은, 단일 API 호출로는 부족하고 여러 처리 단계를 정해진 순서와 조건에 따라 연결해야 한다는 것입니다. 단순 API 호출과 오케스트레이션의 차이를 정리하면 다음과 같습니다.

```
단순 API 호출 vs 오케스트레이션 비교

단순 API 호출                        오케스트레이션
----------------------------------   ----------------------------------
사용자 입력 --> LLM API --> 응답      사용자 입력
                                       |
                                       v
                                     전처리 (입력 검증, 분류)
                                       |
                                       v
                                     외부 데이터 검색 (벡터 DB 등)
                                       |
                                       v
                                     프롬프트 조합
                                       |
                                       v
                                     LLM API 호출
                                       |
                                       v
                                     응답 파싱 및 검증
                                       |
                                       v
                                     후처리 (형식 변환, 필터링)
                                       |
                                       v
                                     최종 응답
```

왼쪽의 단순 API 호출은 입력을 그대로 모델에 보내고 응답을 돌려받는 한 단계짜리 흐름입니다. 오른쪽의 오케스트레이션은 입력 전처리부터 최종 응답까지 여러 단계가 순서대로 이어지며, 각 단계의 출력이 다음 단계의 입력이 됩니다. 이 흐름을 코드로 직접 구현할 수도 있지만, 단계가 많아지고 조건 분기가 복잡해질수록 코드가 뒤엉키기 쉽습니다. 오케스트레이션 프레임워크는 이 문제를 해결하기 위해 만들어진 도구입니다.

**오케스트레이션 프레임워크**란 LLM 애플리케이션의 다단계 처리 흐름을 구성하고, 실행하고, 관리하기 위한 소프트웨어 라이브러리 또는 플랫폼을 뜻합니다. 프레임워크마다 설계 철학과 API 형태는 다르지만, 대부분의 오케스트레이션 프레임워크가 공통으로 제공하는 구성요소가 있습니다. 이를 다섯 가지로 나누어 살펴보겠습니다.

첫 번째 구성요소는 **모델 추상화**입니다. 다양한 LLM 제공자(OpenAI, Anthropic, 오픈소스 모델 등)의 API 형식은 제각각입니다. 모델 추상화는 이 차이를 감추고, 동일한 인터페이스로 어떤 모델이든 호출할 수 있게 합니다. 모델을 교체할 때 애플리케이션 코드 전체를 수정할 필요 없이, 모델 이름이나 설정만 바꾸면 됩니다.

두 번째 구성요소는 **프롬프트 템플릿**입니다. 프롬프트에는 고정된 지시문과 매번 달라지는 동적 데이터가 함께 들어갑니다. 프롬프트 템플릿은 고정 부분과 변수를 분리하여 관리할 수 있게 합니다. 예를 들어 "다음 문서를 참고하여 질문에 답하세요. 문서: {context}, 질문: {question}"처럼 중괄호로 변수 자리를 표시하고, 실행 시점에 실제 값을 채워 넣는 방식입니다. 2.2.2에서 다루는 프롬프트 버전 관리와 연계하면, 템플릿 단위로 버전을 추적하고 교체할 수 있습니다.

세 번째 구성요소는 **체인 또는 파이프라인**입니다. 체인은 여러 처리 단계를 순서대로 연결한 실행 흐름입니다. 앞 단계의 출력이 뒷 단계의 입력으로 전달되는 구조입니다. 단순한 직선 흐름뿐 아니라, 조건에 따라 분기하거나 반복하는 흐름도 체인으로 표현할 수 있습니다. 프레임워크에 따라 '체인', '파이프라인', '워크플로', '그래프' 등 다른 이름을 쓰지만, 핵심은 여러 단계를 하나의 단위로 묶어 실행한다는 점에서 같습니다.

네 번째 구성요소는 **데이터 커넥터**입니다. LLM 애플리케이션은 외부 데이터를 자주 활용합니다. 벡터 데이터베이스에서 유사 문서를 검색하거나, 관계형 데이터베이스에서 구조화된 데이터를 조회하거나, 웹 API를 호출하여 실시간 정보를 가져오는 식입니다. 데이터 커넥터는 이런 외부 데이터 소스와의 연결을 표준화된 방식으로 제공합니다.

다섯 번째 구성요소는 **메모리**입니다. LLM은 기본적으로 이전 대화를 기억하지 않습니다. 매 호출은 독립적이어서, 이전에 어떤 질문을 했는지 모델은 알지 못합니다. 메모리 구성요소는 대화 이력이나 중간 상태를 저장하고, 다음 호출 시 이를 프롬프트에 포함시켜 모델이 맥락을 유지할 수 있게 합니다.

이 다섯 가지 구성요소의 관계를 정리하면 다음과 같습니다.

```
오케스트레이션 프레임워크의 공통 구성요소

+-----------------------------------------------------------+
|                체인 / 파이프라인                            |
|  (단계들을 순서대로 연결, 조건 분기, 반복)                 |
|                                                           |
|   +---------------+   +-----------+   +---------------+   |
|   | 프롬프트      |   |   모델    |   |  응답 파서    |   |
|   | 템플릿        |-->|  추상화   |-->|  (출력 가공)  |   |
|   +---------------+   +-----------+   +---------------+   |
|          ^                                    |           |
|          |                                    v           |
|   +---------------+                   +---------------+   |
|   | 데이터        |                   |    메모리     |   |
|   | 커넥터        |                   | (대화 이력,   |   |
|   | (벡터 DB,     |                   |  중간 상태)   |   |
|   |  외부 API)    |                   +---------------+   |
|   +---------------+                                       |
+-----------------------------------------------------------+
```

프롬프트 템플릿이 변수 자리에 데이터 커넥터가 가져온 정보와 메모리에서 꺼낸 대화 이력을 채워 넣고, 완성된 프롬프트를 모델 추상화 계층을 통해 LLM에 전달합니다. 모델의 응답은 응답 파서가 가공하며, 필요하면 메모리에 저장합니다. 체인이 이 모든 단계를 하나의 실행 흐름으로 엮습니다.

현재 LLM 생태계에서 널리 쓰이는 오케스트레이션 프레임워크는 여러 가지가 있으며, 그중 대표적인 세 가지를 간략히 소개합니다.

**LangChain**은 가장 먼저 등장하여 가장 넓은 사용자 기반을 확보한 Python 기반 오케스트레이션 프레임워크입니다. '체인'이라는 이름에서 알 수 있듯이, 여러 처리 단계를 체인으로 연결하는 것이 핵심 설계입니다. 프롬프트 템플릿, 다양한 LLM 제공자 통합, 벡터 데이터베이스 연동, 에이전트, 메모리 등 앞서 설명한 공통 구성요소를 폭넓게 지원합니다. LangChain Expression Language(LCEL)라는 선언적 문법으로 체인을 구성할 수 있고, LangGraph라는 확장 라이브러리를 통해 상태 기반의 복잡한 워크플로도 구현할 수 있습니다. 다음 단원인 5.1.2에서 LangChain의 핵심 구조를 상세히 다룹니다.

**LlamaIndex**는 데이터 연결에 특화된 오케스트레이션 프레임워크입니다. LLM이 외부 데이터를 효과적으로 활용할 수 있도록, 다양한 형태의 데이터 소스(PDF, 데이터베이스, API, 웹 페이지 등)를 읽어 들이고 인덱싱하는 기능이 강점입니다. RAG 파이프라인을 구축할 때 데이터 수집부터 인덱스 생성, 쿼리 엔진 구성까지의 과정을 간결하게 표현할 수 있습니다. LangChain이 범용 오케스트레이션에 초점을 맞춘다면, LlamaIndex는 데이터 중심의 LLM 애플리케이션에 초점을 맞춥니다.

**Semantic Kernel**은 Microsoft가 개발한 오케스트레이션 프레임워크로, C#과 Python을 주요 언어로 지원합니다. 기존 엔터프라이즈 애플리케이션에 LLM 기능을 점진적으로 통합하는 데 초점을 맞추고 있습니다. '플러그인'이라는 단위로 기존 코드와 LLM 기능을 연결하며, Azure OpenAI Service와의 통합이 긴밀합니다. Microsoft 기술 스택을 사용하는 조직에서 특히 활용도가 높습니다.

이 세 프레임워크는 각각 강조하는 영역이 다릅니다. 어떤 프레임워크를 선택할지는 프로젝트의 성격에 따라 달라집니다. 선택 시 고려할 기준을 정리하면 다음과 같습니다.

첫째, 주요 사용 사례입니다. RAG 파이프라인이 핵심이라면 데이터 연결이 강한 LlamaIndex가 유리하고, 에이전트나 복잡한 워크플로가 핵심이라면 LangChain의 LangGraph가 적합하며, 기존 엔터프라이즈 시스템과의 통합이 우선이라면 Semantic Kernel이 유리합니다.

둘째, 기술 스택입니다. Python 환경이라면 LangChain과 LlamaIndex 모두 자연스럽게 사용할 수 있습니다. C# 기반 서비스라면 Semantic Kernel이 사실상 유일한 선택지입니다.

셋째, 커뮤니티와 생태계 규모입니다. 프레임워크의 사용자 수가 많을수록 문서, 예제, 서드파티 통합이 풍부합니다. 문제가 생겼을 때 해결 방법을 찾기도 쉽습니다.

넷째, 추상화 수준입니다. 높은 수준의 추상화를 제공하는 프레임워크는 빠르게 시작할 수 있지만, 내부 동작을 세밀하게 제어하기 어려울 수 있습니다. 반대로 낮은 수준의 추상화를 제공하는 프레임워크는 유연하지만 구현에 더 많은 코드가 필요합니다. 프로젝트의 복잡도와 팀의 숙련도에 따라 적절한 수준을 선택해야 합니다.

정리하면, LLM 오케스트레이션은 LLM 애플리케이션의 다단계 처리 흐름을 체계적으로 구성하고 관리하는 것을 뜻합니다. 단순 API 호출로는 외부 데이터 검색, 응답 후처리, 대화 맥락 유지 같은 실제 서비스 요구사항을 감당하기 어렵기 때문에, 오케스트레이션 프레임워크가 필요합니다. 오케스트레이션 프레임워크들은 모델 추상화, 프롬프트 템플릿, 체인, 데이터 커넥터, 메모리라는 공통 구성요소를 제공하며, LangChain, LlamaIndex, Semantic Kernel이 각각 범용 오케스트레이션, 데이터 중심 오케스트레이션, 엔터프라이즈 통합이라는 서로 다른 강점을 가지고 있습니다.

다음 단원인 5.1.2에서는 가장 널리 쓰이는 오케스트레이션 프레임워크인 LangChain의 핵심 구조를 다룹니다.

이 단원을 마치면 LLM 오케스트레이션의 필요성과 주요 프레임워크를 설명할 수 있습니다.
