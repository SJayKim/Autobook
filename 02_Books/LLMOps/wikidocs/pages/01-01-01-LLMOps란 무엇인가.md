# 1.1.1 LLMOps란 무엇인가

소프트웨어 서비스에 대규모 언어 모델(LLM)을 도입하는 팀이 늘고 있습니다. 챗봇, 문서 요약, 코드 생성 등 LLM이 처리할 수 있는 작업은 다양합니다. 그런데 실제 서비스에 LLM을 적용하면, 모델 자체의 성능과는 다른 차원의 문제가 나타납니다. 프롬프트를 조금만 바꿔도 응답 품질이 크게 흔들리고, 같은 질문에 대해 매번 다른 답이 나오기도 합니다. 모델 호출 한 건에 드는 비용이 기존 API 호출보다 수십 배 높아 예산 통제가 어렵고, 응답 지연이 수 초에 달해 사용자 경험을 해치기도 합니다. 모델이 사실과 다른 내용을 자신 있게 생성하는 환각(hallucination) 문제도 있습니다. 이런 문제들은 모델을 개발하는 단계가 아니라 운영하는 단계에서 드러나며, 기존의 소프트웨어 운영 방식만으로는 대응하기 어렵습니다. LLMOps는 바로 이 운영 단계의 문제를 체계적으로 다루기 위해 등장한 분야입니다.

**LLMOps**는 Large Language Model Operations의 줄임말로, 대규모 언어 모델을 실제 서비스 환경에서 안정적이고 효율적으로 운영하기 위한 기술, 절차, 도구의 총체를 뜻합니다. 여기서 "운영"이란 모델을 한 번 배포하고 끝나는 것이 아닙니다. 프롬프트를 설계하고 버전 관리하는 것, 모델 응답의 품질을 지속적으로 평가하는 것, 비용과 지연 시간을 모니터링하는 것, 유해하거나 부적절한 출력을 차단하는 것, 그리고 이 모든 과정을 반복 가능한 파이프라인으로 엮는 것을 포함합니다. 한마디로, LLM을 "만드는 것" 이후의 모든 것을 다루는 분야입니다.

LLMOps라는 용어가 독립적으로 쓰이기 시작한 배경에는 LLM 자체의 특수성이 있습니다. 전통적인 머신러닝 모델은 팀이 직접 훈련 데이터를 준비하고, 모델을 학습시키고, 성능을 측정한 뒤 배포합니다. 이 과정을 체계화한 것이 MLOps(Machine Learning Operations)입니다. MLOps는 데이터 파이프라인, 모델 학습, 배포, 모니터링을 자동화하는 데 초점을 맞춥니다. 그런데 LLM은 이 흐름과 근본적으로 다른 지점이 있습니다. 첫째, 대부분의 조직은 LLM을 밑바닥부터 학습시키지 않습니다. 이미 사전훈련된 모델을 API로 호출하거나, 오픈소스 모델을 가져와 미세조정(fine-tuning)하는 방식을 택합니다. 둘째, 모델의 동작을 바꾸는 주된 수단이 학습 데이터가 아니라 **프롬프트**(prompt)입니다. 프롬프트란 모델에 전달하는 입력 텍스트를 뜻하며, 프롬프트를 어떻게 구성하느냐에 따라 동일한 모델이 전혀 다른 품질의 응답을 내놓습니다. 셋째, 출력이 자연어 텍스트이므로 정답 여부를 자동으로 판별하기 어렵습니다. 숫자 하나를 예측하는 회귀 모델은 오차를 바로 계산할 수 있지만, "이 요약이 원문을 정확히 반영하는가"를 기계적으로 판단하는 것은 훨씬 복잡합니다. 이러한 차이 때문에 기존 MLOps의 도구와 절차를 그대로 적용하기 어렵고, LLM에 특화된 운영 체계가 필요해졌습니다. 이것이 LLMOps가 MLOps와 구분되어 독립적인 분야로 자리 잡은 이유입니다.

LLMOps가 필요해진 또 다른 이유는 LLM 도입의 속도와 규모에 있습니다. 2022년 말 ChatGPT가 공개된 이후, 기업들은 빠른 속도로 LLM을 서비스에 통합하기 시작했습니다. 초기에는 개발자가 프롬프트를 수동으로 작성하고, 응답을 눈으로 확인하며, 문제가 생기면 그때그때 수정하는 방식으로도 충분했습니다. 그러나 LLM 기반 기능이 하나의 프로토타입에서 수십 개의 프로덕션 서비스로 확산되면, 이 수동 방식은 금방 한계에 부딪힙니다. 프롬프트 변경 이력을 추적할 수 없고, 어떤 버전의 프롬프트가 어떤 성능을 냈는지 비교할 수 없으며, 비용이 어디서 얼마나 발생하는지 파악이 되지 않습니다. LLMOps는 이런 상황에서 운영의 투명성과 반복 가능성을 확보하기 위해 등장했습니다.

LLMOps가 다루는 핵심 영역은 크게 다섯 가지로 나눌 수 있습니다. **서빙**, **프롬프트 관리**, **평가**, **모니터링**, **거버넌스**입니다. 각 영역이 어떤 문제를 해결하는지 하나씩 살펴보겠습니다.

첫 번째 영역은 **서빙**(serving)입니다. 서빙이란 학습이 완료된 모델을 실제 요청을 받아 응답을 생성할 수 있는 상태로 배포하고 운영하는 것을 뜻합니다. LLM은 수십억에서 수천억 개의 파라미터를 가지고 있어, 모델을 메모리에 올리고 추론을 실행하는 것 자체가 상당한 컴퓨팅 자원을 요구합니다. 동시 요청이 늘어나면 GPU 메모리가 부족해지거나 응답 지연이 급격히 증가할 수 있습니다. 서빙 영역에서는 모델 양자화(quantization)를 통해 메모리 사용량을 줄이거나, 배치 처리를 최적화하거나, 요청량에 따라 인스턴스를 자동으로 늘리고 줄이는 오토스케일링을 설정하는 등의 작업을 수행합니다. 외부 API를 사용하는 경우에도 요청 속도 제한(rate limit) 관리, 장애 시 대체 모델로의 전환(failover), 비용 최적화 등이 서빙의 범위에 포함됩니다.

두 번째 영역은 **프롬프트 관리**(prompt management)입니다. 앞서 설명했듯이 LLM의 동작을 좌우하는 핵심 요소가 프롬프트입니다. 하나의 서비스에서도 용도별로 여러 개의 프롬프트를 사용하며, 각 프롬프트는 시간이 지남에 따라 계속 수정됩니다. 프롬프트 관리란 이 프롬프트들의 버전을 추적하고, 변경 전후의 성능을 비교하며, 특정 버전을 프로덕션에 배포하거나 롤백하는 체계를 뜻합니다. 소프트웨어 코드를 Git으로 관리하는 것과 비슷한 개념이지만, 프롬프트는 코드보다 자연어에 가까워서 변경의 영향을 예측하기 어렵다는 차이가 있습니다. 따라서 프롬프트 변경 시 자동으로 평가를 실행하여 품질 저하를 감지하는 파이프라인도 프롬프트 관리의 일부입니다.

세 번째 영역은 **평가**(evaluation)입니다. 평가란 모델의 응답이 기대한 수준을 충족하는지 측정하는 활동을 뜻합니다. 전통적인 소프트웨어는 단위 테스트와 통합 테스트로 정확성을 검증할 수 있지만, LLM의 출력은 자연어이므로 "맞다/틀리다"를 이분법으로 판단하기 어렵습니다. 평가 영역에서는 자동화된 지표(예: 사실 정확도, 관련성, 유해성 점수)와 사람에 의한 정성 평가를 조합하여 응답 품질을 측정합니다. 새 프롬프트나 새 모델을 배포하기 전에 기존 버전과 나란히 비교하는 A/B 테스트도 평가의 범위에 들어갑니다. 평가 체계가 없으면, 프롬프트를 수정할 때마다 "더 나아졌는지 나빠졌는지"를 감으로 판단해야 하므로 운영이 불안정해집니다.

네 번째 영역은 **모니터링**(monitoring)입니다. 모니터링이란 운영 중인 LLM 시스템의 상태를 실시간으로 관찰하고, 이상을 감지하는 활동을 뜻합니다. 전통적인 서비스 모니터링이 CPU 사용률, 메모리, 응답 시간, 오류율 등을 추적하는 것처럼, LLM 모니터링도 이러한 인프라 지표를 포함합니다. 그런데 LLM에는 추가로 관찰해야 할 지표가 있습니다. 토큰 사용량과 그에 따른 비용, 응답의 품질 변화 추이, 환각 발생 빈도, 사용자 피드백 등이 이에 해당합니다. 예를 들어, 모델 제공자가 모델을 업데이트하면 동일한 프롬프트에 대한 응답 품질이 달라질 수 있는데, 모니터링이 없으면 이러한 변화를 알아차리지 못한 채 서비스 품질이 저하됩니다.

다섯 번째 영역은 **거버넌스**(governance)입니다. 거버넌스란 LLM 시스템의 사용을 조직 차원에서 통제하고, 규정을 준수하도록 관리하는 체계를 뜻합니다. LLM은 학습 데이터에 포함된 편향을 그대로 출력에 반영할 수 있고, 개인정보가 포함된 응답을 생성할 수도 있습니다. 또한 산업별로 AI 사용에 관한 규제가 도입되고 있어서, 모델의 입력과 출력을 기록하고 감사할 수 있는 체계가 필요합니다. 거버넌스 영역에서는 접근 권한 관리, 입출력 로깅, 유해 콘텐츠 필터링, 개인정보 마스킹, 규제 대응 문서화 등을 다룹니다.

이 다섯 가지 영역의 관계를 정리하면 다음과 같습니다.

```
LLMOps 핵심 영역 전체 조감도

+-------------------------------------------------------------------+
|                        거버넌스 (Governance)                       |
|   접근 권한 / 입출력 로깅 / 유해 콘텐츠 필터 / 규제 준수          |
+-------------------------------------------------------------------+
|                                                                   |
|  +-----------+    +-----------+    +---------+    +-----------+   |
|  | 프롬프트  | -> |   서빙    | -> |  평가   | -> | 모니터링  |   |
|  |   관리    |    |           |    |         |    |           |   |
|  +-----------+    +-----------+    +---------+    +-----------+   |
|   버전 관리       모델 배포        응답 품질      비용/지연/품질   |
|   A/B 배포        추론 최적화      자동 지표      이상 감지        |
|   롤백            오토스케일링     사람 평가      알림              |
|                                                                   |
+-------------------------------------------------------------------+
```

프롬프트 관리에서 설계한 프롬프트가 서빙 단계에서 모델에 전달되고, 모델의 응답을 평가 단계에서 검증하며, 운영 중 상태를 모니터링 단계에서 관찰합니다. 거버넌스는 이 전체 흐름을 감싸면서 보안, 규제, 윤리적 기준을 적용합니다. 다섯 영역은 독립적으로 존재하는 것이 아니라 서로 연결된 순환 구조를 이루며, 평가와 모니터링에서 발견한 문제가 프롬프트 수정이나 서빙 설정 변경으로 이어지는 피드백 루프를 형성합니다.

LLMOps와 전통 소프트웨어 운영의 차이를 좀 더 구체적으로 짚어 보겠습니다. 전통적인 소프트웨어는 동일한 입력에 대해 항상 동일한 출력을 냅니다. 같은 HTTP 요청을 보내면 같은 응답이 돌아옵니다. 따라서 테스트도 "이 입력에 이 출력이 나와야 한다"는 결정론적 단언(assertion)으로 작성할 수 있습니다. 반면 LLM은 확률적으로 토큰을 생성하므로, 동일한 프롬프트에 대해서도 매번 다른 텍스트를 출력할 수 있습니다. 이 비결정성(non-determinism) 때문에 전통적인 단위 테스트 방식이 그대로 적용되지 않으며, 통계적 평가나 범위 기반 판정 같은 새로운 검증 방식이 필요합니다.

비용 구조도 다릅니다. 전통적인 소프트웨어에서 API 호출 한 건의 비용은 서버 자원 사용량에 비례하며, 대체로 예측 가능합니다. LLM 호출은 입력과 출력의 토큰 수에 따라 비용이 달라집니다. 토큰이란 모델이 텍스트를 처리하는 최소 단위로, 대략 한국어 한 글자 또는 영어 단어 일부에 해당합니다. 긴 문서를 요약하는 요청은 짧은 질문에 답하는 요청보다 토큰을 훨씬 많이 소비하므로, 사용 패턴에 따라 비용 편차가 큽니다. 이 때문에 LLMOps에서는 토큰 단위의 비용 추적과 예산 관리가 별도의 운영 과제가 됩니다.

장애의 성격도 다릅니다. 전통적인 소프트웨어의 장애는 서버 다운, 네트워크 단절, 데이터베이스 오류 등 명확하게 감지할 수 있는 형태가 대부분입니다. LLM 시스템에서는 서비스가 정상적으로 응답을 반환하더라도 그 내용이 틀리거나, 편향되거나, 유해할 수 있습니다. 이를 "조용한 실패(silent failure)"라 부릅니다. 서버 지표만 보면 모든 것이 정상인데, 실제로는 사용자에게 잘못된 정보를 전달하고 있는 상황입니다. 이런 종류의 장애를 감지하려면 인프라 모니터링 외에 응답 내용 자체를 분석하는 체계가 필요하며, 이것이 LLMOps 모니터링이 전통 모니터링보다 복잡해지는 이유입니다.

마지막으로 LLMOps 생태계의 전체 조감도를 살펴보겠습니다. LLMOps를 구성하는 요소는 크게 네 층으로 나눌 수 있습니다. 가장 아래에는 **모델 계층**이 있습니다. OpenAI의 GPT 시리즈, Anthropic의 Claude, Meta의 LLaMA, Google의 Gemini 등 사전훈련된 모델이 여기에 해당합니다. 조직은 이 모델을 API로 호출하거나, 오픈소스 모델을 자체 인프라에 배포하여 사용합니다. 그 위에 **오케스트레이션 계층**이 있습니다. LangChain, LlamaIndex 같은 프레임워크가 프롬프트 구성, 외부 데이터 연결(RAG), 도구 호출, 에이전트 실행 등 LLM을 활용하는 애플리케이션의 흐름을 조율합니다. 그 위에 **운영 계층**이 자리합니다. 서빙, 평가, 모니터링, 프롬프트 관리 도구들이 이 계층에 속하며, LangSmith, Weights & Biases, Arize AI 같은 플랫폼이 대표적입니다. 가장 위에는 **거버넌스 계층**이 있어 보안, 접근 제어, 감사 로그, 규제 대응을 담당합니다.

```
LLMOps 생태계 계층 구조

+-----------------------------------------------+
|            거버넌스 계층                        |
|  보안 / 접근 제어 / 감사 로그 / 규제 대응      |
+-----------------------------------------------+
|            운영 계층                            |
|  서빙 / 평가 / 모니터링 / 프롬프트 관리        |
|  (LangSmith, W&B, Arize AI 등)                |
+-----------------------------------------------+
|            오케스트레이션 계층                   |
|  프롬프트 구성 / RAG / 에이전트 / 도구 호출    |
|  (LangChain, LlamaIndex 등)                   |
+-----------------------------------------------+
|            모델 계층                            |
|  GPT / Claude / LLaMA / Gemini 등             |
|  (API 호출 또는 자체 배포)                     |
+-----------------------------------------------+
```

이 네 계층은 서로 밀접하게 연결되어 있습니다. 모델 계층에서 모델을 선택하면 오케스트레이션 계층에서 애플리케이션 로직을 구성하고, 운영 계층에서 배포와 모니터링을 수행하며, 거버넌스 계층에서 전체를 통제합니다. LLMOps는 이 중 주로 운영 계층과 거버넌스 계층의 활동을 지칭하지만, 실무에서는 오케스트레이션 계층과의 경계가 겹치는 부분도 있습니다.

정리하면, LLMOps는 대규모 언어 모델을 프로덕션 환경에서 안정적으로 운영하기 위한 기술과 절차의 체계입니다. LLM의 비결정적 출력, 프롬프트 의존성, 토큰 기반 비용 구조, 조용한 실패 가능성이라는 고유한 특성 때문에 기존 소프트웨어 운영이나 MLOps와는 다른 접근이 필요하며, 서빙, 프롬프트 관리, 평가, 모니터링, 거버넌스의 다섯 가지 핵심 영역으로 구성됩니다.

다음 단원인 1.1.2에서는 전통적인 머신러닝 운영 체계인 MLOps에서 LLMOps로의 전환이 왜, 어떻게 이루어졌는지를 구체적으로 다룹니다.

이 단원을 마치면 LLMOps의 정의와 등장 배경을 설명할 수 있고, LLMOps가 다루는 다섯 가지 핵심 영역을 나열할 수 있습니다.
