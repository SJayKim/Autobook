# 7.1.1 파인튜닝의 기본 개념과 전략

지금까지 6장에서 모델 서빙, 추론 최적화, 양자화, 모델 관리를 다루었습니다. 이 주제들은 이미 학습이 완료된 모델을 효율적으로 운영하는 방법에 해당합니다. 그런데 기존 모델이 우리의 요구 사항을 충분히 만족하지 못하는 상황이 있습니다. 예를 들어, 범용 언어 모델에게 회사 내부 규정에 따른 고객 응대를 요청하면 일반적인 답변만 돌아옵니다. 의료 분야의 전문 용어를 사용한 질문에 정확한 답변을 기대해도, 범용 모델은 해당 도메인의 지식이 부족하여 틀린 정보를 생성할 수 있습니다. 이런 상황에서 모델의 동작을 원하는 방향으로 조정하는 방법이 **파인튜닝(Fine-tuning)**입니다.

**파인튜닝**이란, 대규모 데이터로 사전 학습(pre-training)이 완료된 모델을 가져와, 특정 목적에 맞는 소규모 데이터로 추가 학습시키는 과정을 말합니다. 사전 학습은 인터넷에서 수집한 방대한 텍스트를 사용하여 언어의 일반적인 패턴과 지식을 모델에 심는 단계입니다. 이 과정에는 수천 개의 GPU와 수 주에서 수 개월의 시간이 소요됩니다. 파인튜닝은 이 사전 학습된 모델을 출발점으로 삼아, 우리가 원하는 특정 작업이나 도메인에 맞게 모델의 가중치를 조정합니다. 사전 학습에 비해 훨씬 적은 데이터와 컴퓨팅 자원으로 수행할 수 있습니다.

파인튜닝이 가능한 이유는 사전 학습 단계에서 모델이 이미 언어의 구조, 문법, 상식, 추론 능력 등 범용적인 능력을 갖추었기 때문입니다. 파인튜닝은 이 범용 능력을 유지하면서 특정 영역의 지식이나 특정 형식의 응답 패턴을 추가로 학습시킵니다. 비유하자면, 의과대학에 입학하는 학생이 이미 기초 과학과 언어 능력을 갖추고 있어서, 의학 지식만 추가로 배우면 되는 것과 비슷합니다. 처음부터 모든 것을 가르칠 필요가 없습니다.

파인튜닝의 구체적인 동작을 단계별로 설명하겠습니다. 먼저 사전 학습된 모델의 가중치를 로드합니다. 그다음 목적에 맞게 준비된 학습 데이터를 모델에 입력합니다. 모델이 출력한 결과와 기대하는 정답 사이의 차이(손실, loss)를 계산합니다. 이 손실을 줄이는 방향으로 가중치를 갱신합니다. 이 과정을 학습 데이터 전체에 대해 여러 번 반복합니다. 학습이 끝나면 조정된 가중치를 저장합니다.

```
파인튜닝 과정

사전 학습된 모델 (범용 능력 보유)
         |
         v
  [목적별 학습 데이터 입력]
         |
         v
  [모델 출력과 정답 비교 -> 손실 계산]
         |
         v
  [가중치 갱신 (손실을 줄이는 방향)]
         |
         v
  [반복 (수 에포크)]
         |
         v
  파인튜닝된 모델 (범용 능력 + 특정 목적 능력)
```

이제 파인튜닝의 두 가지 큰 범주를 살펴보겠습니다. **Full Fine-tuning**과 **Parameter-Efficient Fine-tuning(PEFT)**입니다.

**Full Fine-tuning**은 모델의 모든 파라미터를 학습 대상으로 두는 방식입니다. 사전 학습된 모델의 모든 가중치가 파인튜닝 데이터에 맞게 갱신됩니다. 모델이 학습 데이터에 가장 충실하게 적응할 수 있다는 장점이 있습니다. 그러나 모든 파라미터를 갱신해야 하므로, 학습에 필요한 GPU 메모리와 연산량이 매우 큽니다. 70억 파라미터 모델을 Full Fine-tuning하려면, 가중치뿐 아니라 옵티마이저 상태와 그래디언트까지 GPU 메모리에 올려야 합니다. FP32 기준으로 가중치 28GB, 그래디언트 28GB, 옵티마이저 상태(Adam 기준) 56GB가 필요하여, 총 112GB 이상의 GPU 메모리가 필요합니다. 80GB짜리 GPU 한 장으로는 부족하고, 여러 장의 GPU를 사용해야 합니다.

**Parameter-Efficient Fine-tuning(PEFT)**는 모델의 파라미터 중 극히 일부만 학습하거나, 소수의 새 파라미터를 추가하여 학습하는 방식의 총칭입니다. 원래 모델의 대부분의 가중치는 고정(freeze)해 두고, 작은 부분만 조정합니다. 학습해야 할 파라미터 수가 전체의 1% 미만인 경우도 흔합니다. 그만큼 필요한 GPU 메모리와 연산량이 Full Fine-tuning보다 크게 줄어듭니다.

PEFT의 대표적인 기법으로는 LoRA(Low-Rank Adaptation), QLoRA(Quantized LoRA), 프롬프트 튜닝(Prompt Tuning), 프리픽스 튜닝(Prefix Tuning) 등이 있습니다. 이 중 LoRA와 QLoRA는 7.1.2와 7.1.3에서 상세히 다룹니다. PEFT 기법들의 공통점은 원본 모델의 가중치를 변경하지 않고, 작은 추가 모듈이나 파라미터를 통해 모델의 동작을 조정한다는 것입니다.

```
Full Fine-tuning vs PEFT 비교

항목                Full Fine-tuning         PEFT (예: LoRA)
-----------         ------------------       ----------------
학습 대상           모든 파라미터             추가된 소수 파라미터
7B 모델 기준
학습 파라미터 수    약 70억 개               약 수백만 개 (전체의 1% 미만)
GPU 메모리 (FP32)   112 GB 이상              약 16~24 GB
학습 속도           느림                     빠름
품질                높음 (충분한 데이터 시)   대부분의 경우 유사
원본 모델 보존      변경됨                   보존됨
```

Full Fine-tuning과 PEFT 중 어떤 것을 선택할지는 상황에 따라 다릅니다. 대규모 GPU 자원이 있고, 충분한 양의 고품질 학습 데이터가 확보되어 있으며, 모델을 도메인에 깊이 적응시켜야 하는 경우에는 Full Fine-tuning이 더 나은 결과를 줄 수 있습니다. 그 외 대부분의 실무 상황에서는 PEFT가 비용 대비 효과가 뛰어납니다.

다음으로 파인튜닝의 주요 유형인 **Instruction Tuning**과 **Supervised Fine-Tuning(SFT)**을 살펴보겠습니다.

**Instruction Tuning**은 모델이 사람의 지시(instruction)를 이해하고 그에 맞는 응답을 생성하도록 학습시키는 파인튜닝 방식입니다. 사전 학습만 완료된 모델, 즉 기반 모델(base model)은 다음 토큰을 예측하는 능력만 학습했기 때문에, "한국의 수도는 어디인가요?"라는 질문을 입력하면 "한국의 수도는 서울입니다"라고 답하는 대신, "한국의 인구는 얼마인가요?"처럼 비슷한 형태의 질문을 이어서 생성할 수 있습니다. 기반 모델은 텍스트를 이어 쓰는 방법을 알지만, 지시에 따라 응답하는 방법은 알지 못합니다.

Instruction Tuning은 이 간극을 메웁니다. "지시-응답" 쌍으로 구성된 데이터를 사용하여 모델을 학습시킵니다. 예를 들어, "다음 문장을 요약하세요: [긴 문장]"이라는 지시와 그에 대한 적절한 요약 응답, "이 코드의 버그를 찾아 수정하세요: [코드]"라는 지시와 수정된 코드 응답 등의 데이터를 사용합니다. 이 과정을 거치면 모델이 다양한 형태의 지시를 이해하고, 지시에 맞는 형식과 내용으로 응답을 생성하게 됩니다.

**Supervised Fine-Tuning(SFT)**은 레이블이 있는 데이터, 즉 입력과 기대 출력이 짝지어진 데이터를 사용하여 모델을 학습시키는 방식입니다. Instruction Tuning도 SFT의 한 형태로 볼 수 있습니다. 다만 SFT는 더 넓은 개념으로, 지시-응답 쌍뿐 아니라 특정 도메인의 질문-답변 쌍, 문서 분류 데이터, 감정 분석 데이터 등 다양한 형태의 레이블 데이터를 사용할 수 있습니다.

SFT의 학습 데이터 형식은 보통 다음과 같습니다.

```
SFT 학습 데이터 예시

예시 1 (지시-응답):
  입력: "다음 텍스트의 감정을 분석하세요: '오늘 날씨가 정말 좋아서 기분이 좋습니다'"
  출력: "긍정적인 감정입니다. 좋은 날씨로 인한 기쁨을 표현하고 있습니다."

예시 2 (도메인 특화 질의응답):
  입력: "고객이 환불을 요청했을 때 절차를 알려주세요."
  출력: "환불 요청 시 다음 절차를 안내합니다. 1. 주문 번호 확인..."

예시 3 (코드 생성):
  입력: "Python으로 리스트에서 중복을 제거하는 함수를 작성하세요."
  출력: "def remove_duplicates(lst):\n    return list(set(lst))"
```

이제 파인튜닝과 다른 접근 방식의 비교를 살펴보겠습니다. 모델의 동작을 원하는 방향으로 조정하는 방법은 파인튜닝만 있는 것이 아닙니다. **프롬프트 엔지니어링**과 **RAG(Retrieval-Augmented Generation)**도 같은 목적을 달성할 수 있습니다. 세 가지 접근 방식은 각각 다른 상황에 적합합니다.

**프롬프트 엔지니어링**은 모델의 가중치를 전혀 변경하지 않고, 입력 프롬프트를 정교하게 설계하여 원하는 출력을 유도하는 방식입니다. 가장 빠르고 비용이 적게 드는 접근법입니다. 모델을 추가로 학습시킬 필요가 없으므로, 몇 분 안에 시도하고 결과를 확인할 수 있습니다. 하지만 프롬프트의 길이에는 제한이 있으며, 복잡한 도메인 지식이나 특정 응답 패턴을 프롬프트만으로 전달하기 어려운 경우가 있습니다.

**RAG**는 모델의 가중치를 변경하지 않으면서, 외부 데이터베이스에서 관련 정보를 검색하여 프롬프트에 포함시키는 방식입니다. 모델이 알지 못하는 최신 정보나 사내 문서를 참조하여 응답을 생성할 수 있습니다. 데이터베이스의 내용을 갱신하면 모델의 응답도 바로 변합니다. 하지만 모델의 응답 형식이나 추론 방식 자체를 바꿀 수는 없으며, 검색 품질에 의존한다는 한계가 있습니다.

**파인튜닝**은 모델의 가중치를 변경하여 동작을 근본적으로 조정합니다. 특정 도메인의 전문 용어와 지식을 모델 내부에 내재화하거나, 응답 형식과 톤을 일관되게 변경할 수 있습니다. 그러나 학습 데이터 준비, GPU 자원, 학습 시간이 필요하며, 잘못된 데이터로 학습하면 모델 성능이 오히려 떨어질 수 있습니다.

```
파인튜닝 vs RAG vs 프롬프트 엔지니어링 비교

항목                 프롬프트 엔지니어링   RAG                   파인튜닝
-----------          ------------------   ------------------    --------
모델 가중치 변경     없음                 없음                  있음
추가 학습 필요       아니오               아니오                예
구현 시간            수 분~수 시간        수 일                 수 일~수 주
비용                 매우 낮음            중간 (검색 인프라)    높음 (GPU 비용)
최신 정보 반영       어려움               쉬움 (DB 갱신)        어려움 (재학습 필요)
응답 형식 제어       제한적               제한적                높음
도메인 지식 내재화   불가                 부분적                가능
```

마지막으로 **파인튜닝 결정 프레임워크**를 정리하겠습니다. 실무에서 "파인튜닝을 해야 하는가"라는 질문에 체계적으로 답하기 위한 판단 기준입니다.

먼저 프롬프트 엔지니어링으로 충분한지 시도합니다. 프롬프트를 다양하게 변경하고 few-shot 예시를 추가해 봅니다. 원하는 결과가 나오면 파인튜닝이 불필요합니다.

프롬프트 엔지니어링으로 부족하다면, 부족한 부분이 지식의 문제인지 행동의 문제인지 구분합니다. 모델이 특정 정보를 모르는 것이 문제라면 RAG를 먼저 시도합니다. 모델이 응답 형식을 지키지 않거나, 도메인 특유의 추론 패턴을 따르지 않는 것이 문제라면 파인튜닝이 적합합니다.

파인튜닝을 결정했다면, 충분한 학습 데이터를 확보할 수 있는지 확인합니다. SFT에는 최소 수백에서 수천 개의 고품질 입력-출력 쌍이 필요합니다. 데이터가 부족하면 파인튜닝 효과가 미미하거나 과적합이 발생할 수 있습니다.

데이터와 자원이 준비되었다면, Full Fine-tuning과 PEFT 중 하나를 선택합니다. 대부분의 경우 LoRA나 QLoRA 같은 PEFT 기법으로 시작하여 결과를 확인한 뒤, 품질이 부족하면 Full Fine-tuning을 고려하는 단계적 접근이 효율적입니다.

```
파인튜닝 결정 흐름

프롬프트 엔지니어링으로 원하는 결과가 나오는가?
  |
  +-- 예 --> 파인튜닝 불필요
  |
  +-- 아니오 --> 부족한 부분이 무엇인가?
                   |
                   +-- 지식 부족 --> RAG 시도
                   |                  |
                   |                  +-- RAG로 해결됨 --> 파인튜닝 불필요
                   |                  |
                   |                  +-- RAG로도 부족 --> 파인튜닝 고려
                   |
                   +-- 행동/형식 문제 --> 파인튜닝 필요
                                          |
                                          v
                                     학습 데이터 확보 가능한가?
                                          |
                                          +-- 아니오 --> 데이터 수집 먼저
                                          |
                                          +-- 예 --> PEFT로 시작
                                                      |
                                                      +-- 품질 충분 --> 완료
                                                      |
                                                      +-- 품질 부족 --> Full FT 고려
```

정리하면, 파인튜닝은 사전 학습된 모델을 특정 목적에 맞게 추가 학습시키는 과정입니다. Full Fine-tuning은 모든 파라미터를 학습하여 가장 강력한 적응을 이루지만 자원 소모가 크고, PEFT는 소수의 파라미터만 학습하여 비용 대비 효과가 뛰어납니다. Instruction Tuning은 모델이 지시를 따르도록 학습시키는 방식이고, SFT는 레이블 데이터를 사용한 지도 학습의 총칭입니다. 프롬프트 엔지니어링, RAG, 파인튜닝은 각각 다른 상황에 적합하며, 프롬프트 엔지니어링부터 시작하여 단계적으로 복잡한 방법으로 전환하는 것이 실무에서 효율적인 접근법입니다.

다음 단원인 7.1.2에서는 PEFT의 대표 기법인 LoRA를 다룹니다. LoRA가 어떤 수학적 원리로 적은 파라미터만으로 파인튜닝 효과를 달성하는지, 그리고 실제 적용 방법을 살펴봅니다.

이 단원을 마치면 LLM 파인튜닝의 유형을 구분하고, 프롬프트 엔지니어링, RAG, 파인튜닝 중 적합한 전략을 선택할 수 있습니다.
