# 3.2.1 벡터 데이터베이스의 개념과 필요성

3.1.3에서 텍스트 데이터를 임베딩 벡터로 변환하는 파이프라인을 구축하는 방법을 살펴보았습니다. 전처리, 배치 생성, 캐싱, 갱신 정책까지 갖추면, 문서 수천 건에서 수백만 건에 이르는 임베딩 벡터가 만들어집니다. 그런데 이렇게 만들어진 벡터를 어디에, 어떻게 저장하고 검색해야 할까요. 파일 시스템에 NumPy 배열로 저장해 두었다가 매번 전체를 불러와 비교하는 방식은 데이터가 늘어날수록 현실적이지 않습니다. 이 단원에서는 임베딩 벡터를 대규모로 저장하고 빠르게 검색하기 위해 설계된 전용 저장소, 즉 벡터 데이터베이스가 무엇이고 왜 필요한지를 다룹니다.

벡터 데이터베이스가 해결하려는 문제부터 짚어 보겠습니다. 3.1.1에서 다룬 것처럼, 임베딩은 텍스트의 의미를 숫자 배열, 즉 벡터로 표현한 것입니다. 두 벡터 사이의 거리나 각도를 계산하면 원문의 의미적 유사도를 수치로 알 수 있습니다. 이 성질을 활용하면, 사용자가 질문을 입력했을 때 그 질문과 가장 의미가 가까운 문서 조각을 찾아내는 검색이 가능합니다. RAG(Retrieval-Augmented Generation) 파이프라인에서 검색 단계가 바로 이 과정입니다.

문제는 비교 대상이 많아질 때 생깁니다. 벡터 하나의 차원이 1,536이고 문서 조각이 100만 건이라고 가정해 보겠습니다. 질문 벡터 하나와 100만 건의 벡터를 모두 비교하려면, 1,536차원짜리 벡터 쌍을 100만 번 내적하거나 코사인 유사도를 계산해야 합니다. 이 방식을 **전수 검색(brute-force search)**이라고 부릅니다. 데이터가 1만 건 이하일 때는 전수 검색도 충분히 빠르지만, 수십만 건을 넘어가면 지연 시간이 눈에 띄게 늘어나고, 수백만 건에서는 초 단위 이상의 응답 지연이 발생할 수 있습니다. 사용자에게 실시간으로 답변을 제공해야 하는 서비스에서는 이 지연이 치명적입니다.

**벡터 데이터베이스(vector database)**는 이 문제를 해결하기 위해 만들어진 전용 데이터베이스입니다. 고차원 벡터를 저장하고, 주어진 질의 벡터와 가장 유사한 벡터를 빠르게 찾아 돌려주는 것을 핵심 목적으로 설계되었습니다. 일반적인 데이터베이스가 행과 열로 이루어진 테이블이나 키-값 쌍을 다루는 것과 달리, 벡터 데이터베이스는 수백에서 수천 차원에 이르는 실수 배열을 일급 데이터 타입으로 다룹니다. 저장뿐 아니라 유사도 기반 검색에 특화된 인덱스 구조를 내장하고 있어, 수백만 건 이상의 벡터에서도 밀리초 단위로 유사한 벡터를 찾아낼 수 있습니다.

벡터 데이터베이스가 빠른 검색을 가능하게 하는 핵심 기법은 **ANN(Approximate Nearest Neighbor)** 검색입니다. 이름에서 알 수 있듯이, "정확한" 최근접 이웃이 아니라 "근사적인" 최근접 이웃을 찾는 방식입니다. 왜 정확한 결과를 포기하는 걸까요. 앞서 설명한 전수 검색은 정확한 결과를 보장하지만, 데이터가 많아지면 너무 느립니다. ANN은 정확도를 약간 양보하는 대신 검색 속도를 극적으로 높입니다. 실제로 100만 건의 벡터에서 전수 검색이 수 초 걸리는 경우, ANN 인덱스를 사용하면 수 밀리초 안에 결과를 돌려받을 수 있습니다. 그리고 "근사적"이라고 해도 대부분의 경우 상위 결과의 95% 이상이 정확한 최근접 이웃과 일치합니다.

ANN 검색이 동작하는 원리를 이해하려면, 대표적인 알고리즘 두 가지를 살펴보는 것이 도움이 됩니다. 하나는 **IVF(Inverted File Index)**이고, 다른 하나는 **HNSW(Hierarchical Navigable Small World)**입니다.

IVF는 벡터 공간을 여러 구역으로 미리 나누어 두는 방식입니다. 동작 과정은 다음과 같습니다. 먼저 인덱스를 구축하는 단계에서, 전체 벡터를 비슷한 것끼리 묶어 여러 클러스터로 분류합니다. 각 클러스터의 중심점을 기록해 둡니다. 검색할 때는 질의 벡터와 중심점들만 먼저 비교하여, 가장 가까운 클러스터 몇 개를 선택합니다. 그리고 선택된 클러스터에 속한 벡터들만 대상으로 유사도를 계산합니다. 전체 100만 건이 아니라, 후보 클러스터에 속한 수천에서 수만 건만 비교하면 되므로 속도가 크게 빨라집니다. 다만 질의 벡터와 가장 유사한 벡터가 선택되지 않은 클러스터에 있을 가능성이 있기 때문에, 정확도가 100%가 아닌 "근사"가 됩니다. 탐색할 클러스터 수를 늘리면 정확도가 올라가지만, 그만큼 속도는 느려집니다.

HNSW는 그래프 구조를 활용합니다. 벡터들을 노드로 하고, 가까운 벡터끼리 간선으로 연결한 그래프를 여러 층으로 쌓아 올립니다. 가장 위 층에는 소수의 노드만 있고, 아래 층으로 내려갈수록 노드가 촘촘해집니다. 검색은 맨 위 층에서 시작합니다. 질의 벡터와 가까운 방향으로 간선을 따라 이동하면서 후보 노드를 좁혀 갑니다. 한 층에서 더 이상 가까워지지 않으면 아래 층으로 내려가서 같은 과정을 반복합니다. 마지막 층에 도달하면 최종 결과를 돌려줍니다. 이 과정을 시각화하면 다음과 같습니다.

```
[층 2]   (A) --------- (B)                   <- 소수 노드, 넓은 이동
           |
[층 1]   (A) --- (C) --- (B) --- (D)         <- 중간 밀도
           |       |       |       |
[층 0]   (A)-(E)-(C)-(F)-(B)-(G)-(D)-(H)    <- 전체 노드, 정밀 탐색

검색 흐름:
  1. 층 2에서 시작 -> A에서 B로 이동 (질의와 더 가까운 쪽)
  2. 층 1로 내려감 -> B 주변의 C, D 비교
  3. 층 0으로 내려감 -> F, G 등 인접 노드를 정밀 비교
  4. 최종 결과 반환
```

HNSW는 검색 속도와 정확도의 균형이 뛰어나서, Pinecone, Weaviate, Qdrant, Milvus 등 대부분의 벡터 데이터베이스가 기본 인덱스 알고리즘으로 채택하고 있습니다. IVF는 메모리 사용량이 상대적으로 적어, 데이터 규모가 매우 크거나 비용을 절감해야 할 때 선택됩니다. 어떤 알고리즘을 쓰든 핵심 원리는 같습니다. 전체를 다 비교하지 않고, 구조를 미리 만들어서 비교 대상을 줄이는 것입니다.

이제 벡터 데이터베이스가 기존의 **RDBMS**나 **NoSQL** 데이터베이스와 어떻게 다른지를 살펴보겠습니다. RDBMS는 관계형 데이터베이스로, MySQL이나 PostgreSQL처럼 행과 열로 이루어진 테이블에 데이터를 저장합니다. 데이터를 찾을 때는 SQL이라는 질의 언어를 사용하여, 특정 열의 값이 조건에 맞는 행을 정확히 찾습니다. "이름이 '홍길동'인 사용자를 찾아라"처럼 값이 일치하는지, 범위에 들어가는지를 기준으로 검색합니다. NoSQL은 MongoDB 같은 문서형 데이터베이스나 Redis 같은 키-값 저장소를 포함하는 넓은 범주로, 스키마가 유연하고 대규모 분산 처리에 유리하지만, 검색 방식은 여전히 "값의 일치나 범위"가 기본입니다.

벡터 데이터베이스의 검색은 이와 근본적으로 다릅니다. 값이 정확히 일치하는 행을 찾는 것이 아니라, 주어진 벡터와 "가장 가까운" 벡터를 찾습니다. "가깝다"는 기준은 코사인 유사도, 유클리드 거리, 내적 같은 수학적 거리 함수입니다. 이 차이를 표로 정리하면 다음과 같습니다.

```
구분              RDBMS              NoSQL              벡터 데이터베이스
------------------------------------------------------------------------
데이터 모델       테이블 (행, 열)    문서/키-값/그래프   벡터 + 메타데이터
검색 기준         값 일치/범위       키 조회/문서 조건   유사도 (거리 함수)
인덱스 구조       B-Tree, Hash       다양                ANN 인덱스 (HNSW, IVF 등)
대표 질의         SELECT WHERE       find({조건})        query(벡터, top_k)
적합한 용도       트랜잭션, 정형     반정형, 확장성      의미 검색, 추천
```

RDBMS에서 "이 문장과 의미가 비슷한 문서를 찾아라"라는 질의를 수행하려면, 벡터를 BLOB이나 배열 컬럼에 저장한 뒤 애플리케이션 레벨에서 전수 비교를 해야 합니다. ANN 인덱스가 없으므로 데이터가 늘어날수록 성능이 급격히 저하됩니다. 일부 RDBMS는 벡터 검색 확장 기능을 제공하기도 합니다. 예를 들어 PostgreSQL의 pgvector 확장은 벡터 타입과 ANN 인덱스를 추가합니다. 이런 확장은 기존 데이터베이스의 트랜잭션 기능과 벡터 검색을 하나의 시스템에서 사용할 수 있다는 장점이 있지만, 전용 벡터 데이터베이스에 비해 대규모 벡터 처리의 성능이나 최적화 수준에서 차이가 날 수 있습니다.

벡터 데이터베이스의 핵심 기능은 크게 네 가지로 나눌 수 있습니다. **저장**, **인덱싱**, **검색**, **필터링**입니다.

저장은 벡터와 그에 딸린 부가 정보를 함께 보관하는 기능입니다. 벡터 데이터베이스에 데이터를 넣을 때는 보통 세 가지를 함께 전달합니다. 고유 식별자(ID), 임베딩 벡터, 그리고 메타데이터입니다. 예를 들어 문서 조각 하나를 저장할 때, ID는 "doc_0042", 벡터는 1,536차원 실수 배열, 메타데이터는 {"source": "manual.pdf", "page": 15, "category": "installation"}처럼 원본 정보를 담는 키-값 쌍입니다. 메타데이터가 벡터와 함께 저장되기 때문에, 나중에 검색할 때 벡터 유사도뿐 아니라 메타데이터 조건도 함께 걸 수 있습니다.

인덱싱은 저장된 벡터에 대해 ANN 검색을 빠르게 수행할 수 있도록 내부 자료 구조를 구축하는 과정입니다. 앞서 설명한 IVF의 클러스터 분류나 HNSW의 그래프 구축이 이 단계에서 일어납니다. 벡터를 저장하기만 하면 전수 검색밖에 할 수 없지만, 인덱스를 구축하면 검색 속도가 수십 배에서 수백 배 빨라집니다. 대부분의 벡터 데이터베이스는 데이터를 삽입할 때 인덱스를 자동으로 갱신합니다.

검색은 질의 벡터를 받아 인덱스를 탐색하고, 유사도가 높은 순서대로 상위 k개의 결과를 돌려주는 기능입니다. 여기서 k는 사용자가 지정하는 값으로, 예를 들어 k=10이면 가장 유사한 벡터 10개와 그에 딸린 ID, 메타데이터, 유사도 점수를 함께 반환합니다. 유사도 측정에 사용하는 거리 함수는 데이터베이스나 인덱스 설정에 따라 코사인 유사도, 유클리드 거리, 내적 중 하나를 선택할 수 있습니다. 3.1.1에서 다룬 것처럼, 코사인 유사도는 벡터의 방향만 비교하고 크기를 무시하며, 유클리드 거리는 두 점 사이의 직선 거리를, 내적은 방향과 크기를 모두 반영합니다.

필터링은 검색 결과를 메타데이터 조건으로 좁히는 기능입니다. 이것이 **메타데이터 필터링**입니다. 예를 들어, "설치 관련 문서 중에서 질문과 가장 유사한 것 5개를 찾아라"라는 질의를 수행하려면, 벡터 유사도 검색과 동시에 category가 "installation"인 것만 필터링해야 합니다. 벡터 데이터베이스는 이 두 조건을 하나의 질의에서 처리합니다.

메타데이터 필터링의 처리 방식에는 두 가지 접근이 있습니다. 하나는 **사전 필터링(pre-filtering)**으로, 메타데이터 조건에 맞는 벡터만 먼저 걸러낸 뒤 그 안에서 ANN 검색을 수행합니다. 다른 하나는 **사후 필터링(post-filtering)**으로, ANN 검색을 먼저 수행한 뒤 결과에서 메타데이터 조건에 맞지 않는 것을 제거합니다.

사전 필터링은 정확한 결과를 보장하지만, 필터 조건이 까다로워 후보가 적어지면 ANN 인덱스의 효율이 떨어질 수 있습니다. 사후 필터링은 ANN 인덱스를 최대한 활용하지만, 조건에 맞지 않는 결과가 제거된 뒤 요청한 k개보다 적은 결과가 반환될 수 있습니다. 대부분의 벡터 데이터베이스는 두 방식을 조합하거나, 상황에 따라 자동으로 최적의 전략을 선택합니다.

```
사전 필터링:
  전체 벡터 -> [메타데이터 조건 적용] -> 후보 벡터 -> [ANN 검색] -> 결과

사후 필터링:
  전체 벡터 -> [ANN 검색] -> 상위 후보 -> [메타데이터 조건 적용] -> 결과
```

마지막으로, 벡터 데이터베이스를 선택할 때 고려해야 하는 기준을 살펴보겠습니다. 선택 기준은 프로젝트의 규모, 운영 환경, 요구사항에 따라 달라집니다.

첫 번째 기준은 **관리형 서비스인지 자체 호스팅인지**입니다. Pinecone처럼 클라우드에서 완전 관리형으로 제공되는 서비스는 인프라 관리 부담이 없지만, 데이터가 외부 서버에 저장됩니다. Milvus나 Qdrant처럼 자체 서버에 설치할 수 있는 오픈소스 데이터베이스는 데이터 통제권을 유지할 수 있지만, 클러스터 운영과 확장을 직접 관리해야 합니다.

두 번째 기준은 **확장성**입니다. 저장할 벡터가 수만 건인지, 수억 건인지에 따라 적합한 데이터베이스가 달라집니다. 수만에서 수십만 건 수준이라면 단일 서버로 충분하지만, 수천만 건 이상이면 데이터를 여러 서버에 분산 저장하는 샤딩(sharding) 기능이 필요합니다.

세 번째 기준은 **성능 특성**입니다. 검색 지연 시간, 초당 처리 가능한 질의 수(QPS), 인덱스 구축 시간 등이 해당합니다. 실시간 서비스에서는 검색 지연이 50밀리초 이내여야 하는 경우가 많습니다.

네 번째 기준은 **기능 범위**입니다. 메타데이터 필터링의 유연성, 하이브리드 검색(벡터 검색과 키워드 검색의 결합) 지원 여부, 다중 인덱스 관리, 접근 제어 같은 부가 기능이 프로젝트 요구사항에 맞는지 확인해야 합니다.

다섯 번째 기준은 **생태계 통합**입니다. LangChain, LlamaIndex 같은 오케스트레이션 프레임워크와의 연동이 잘 되는지, SDK가 사용하는 프로그래밍 언어를 지원하는지도 실무에서 중요한 고려 사항입니다.

정리하면, 벡터 데이터베이스는 고차원 임베딩 벡터를 대규모로 저장하고, ANN 알고리즘을 통해 밀리초 단위의 유사도 검색을 수행하는 전용 데이터베이스입니다. 기존 RDBMS나 NoSQL이 값의 일치나 범위를 기준으로 검색하는 것과 달리, 벡터 데이터베이스는 벡터 간 거리를 기준으로 "의미가 가까운" 데이터를 찾습니다. 핵심 기능은 벡터와 메타데이터의 저장, ANN 인덱스 구축, 유사도 기반 검색, 메타데이터 필터링이며, 선택 시에는 관리형 여부, 확장성, 성능, 기능 범위, 생태계 통합을 함께 고려해야 합니다.

다음 단원인 3.2.2에서는 Pinecone을 다룹니다. 대표적인 관리형 벡터 데이터베이스인 Pinecone의 아키텍처와 기본적인 데이터 조작 방법을 살펴봅니다.

이 단원을 마치면 벡터 데이터베이스의 동작 원리와 기존 데이터베이스와의 차이를 설명할 수 있습니다.
