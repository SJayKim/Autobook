# 7.2.3 데이터 플라이휠 구축

7.2.1에서 파인튜닝용 데이터를 큐레이션하고 품질을 관리하는 방법을, 7.2.2에서 LLM을 활용하여 합성 데이터를 생성하는 기법을 살펴보았습니다. 두 단원에서 다룬 내용은 파인튜닝에 필요한 초기 데이터셋을 마련하는 데 초점을 맞추었습니다. 그런데 모델을 서비스에 배포한 뒤에도 데이터의 문제는 끝나지 않습니다. 서비스를 사용하는 사람들은 학습 데이터에 없던 새로운 유형의 질문을 하고, 모델이 틀린 답변을 내놓기도 하며, 시간이 지나면서 세상의 정보도 변합니다. 초기 데이터셋만으로는 이런 변화에 대응할 수 없습니다.

이 문제의 해결 방향이 **데이터 플라이휠(data flywheel)**입니다. 플라이휠은 원래 기계 공학에서 회전 관성을 이용하여 에너지를 저장하는 바퀴를 뜻합니다. 한 번 돌기 시작하면 관성 덕분에 점점 빨라지는 특성이 있습니다. 데이터 플라이휠은 이 비유를 차용한 개념입니다. 서비스를 운영하면서 사용자의 피드백을 수집하고, 그 피드백을 학습 데이터로 변환하여 모델을 개선하고, 개선된 모델이 더 좋은 서비스를 제공하여 더 많은 사용자를 끌어들이고, 더 많은 사용자가 더 많은 피드백을 생성하는 선순환 구조입니다. 이 순환이 한 바퀴 돌 때마다 모델과 데이터의 품질이 함께 향상됩니다.

데이터 플라이휠의 순환 구조를 단계별로 표현하면 다음과 같습니다.

```
데이터 플라이휠 순환 구조

  +---> 모델 배포 (서비스 운영)
  |              |
  |              v
  |     사용자 상호작용
  |     (질문, 응답 열람)
  |              |
  |              v
  |     피드백 수집
  |     (평점, 수정, 선호)
  |              |
  |              v
  |     피드백 --> 학습 데이터 변환
  |     (정제, 포맷팅, 품질 검증)
  |              |
  |              v
  |     모델 재학습/파인튜닝
  |              |
  +------<-------+
          (반복)
```

이 순환이 효과적으로 동작하려면, 각 단계가 체계적으로 설계되어야 합니다. 첫 번째 단계인 사용자 피드백 수집부터 살펴보겠습니다.

**사용자 피드백 수집 메커니즘**은 서비스 환경에서 사용자의 반응을 체계적으로 기록하는 체계를 뜻합니다. 피드백에는 여러 유형이 있습니다.

가장 단순한 형태는 **명시적 피드백(explicit feedback)**입니다. 사용자가 모델의 응답에 직접 평가를 남기는 것입니다. 대표적인 예가 "좋아요/싫어요" 버튼입니다. ChatGPT를 비롯한 많은 LLM 서비스에서 응답 아래에 엄지 위/아래 아이콘을 두어, 사용자가 한 번의 클릭으로 응답의 만족도를 표시하도록 합니다. 좀 더 세밀한 형태로는 1점에서 5점까지의 점수 평가가 있고, 가장 풍부한 형태는 사용자가 직접 수정한 응답을 제출하는 것입니다. 수정된 응답은 "이 질문에 대한 더 나은 답변"이라는 고품질 학습 데이터를 직접 얻는 셈입니다.

다음으로 **암시적 피드백(implicit feedback)**이 있습니다. 사용자가 의도적으로 평가를 남기지 않아도, 행동 자체에서 피드백을 추출할 수 있습니다. 사용자가 응답을 읽은 뒤 추가 질문 없이 대화를 종료했다면, 응답이 만족스러웠을 가능성이 높습니다. 반대로 같은 질문을 다른 표현으로 반복하거나, "아니, 내가 원한 것은..."이라며 재질문을 한다면 이전 응답이 불만족스러웠다는 신호입니다. 응답의 특정 부분을 복사했다면 해당 부분이 유용했다는 뜻입니다. 이런 행동 패턴을 분석하여 피드백 신호로 변환합니다.

세 번째로 **비교 피드백(comparative feedback)**이 있습니다. 같은 질문에 대해 두 개의 응답을 나란히 보여주고, 사용자에게 어느 쪽이 더 나은지 선택하도록 합니다. 이 형태의 피드백은 8.3.1에서 다룰 RLHF의 선호 데이터로 직접 활용할 수 있습니다.

피드백 수집 시 주의할 점이 있습니다. 피드백 수집이 사용자 경험을 해치지 않아야 합니다. 모든 응답마다 팝업으로 평가를 요청하면 사용자가 피로를 느끼고 서비스를 떠날 수 있습니다. 명시적 피드백은 선택적으로 두고, 암시적 피드백을 주로 활용하는 것이 균형을 잡는 방법입니다. 또한 피드백 데이터에는 개인정보가 포함될 수 있으므로, 수집 시 개인정보 보호 정책을 준수해야 합니다.

수집된 피드백은 곧바로 학습 데이터로 쓸 수 있는 형태가 아닙니다. **피드백을 학습 데이터로 변환하는 파이프라인**이 필요합니다.

변환 파이프라인은 다음 단계를 거칩니다. 먼저 수집된 피드백을 정제합니다. 스팸, 악의적 피드백, 의미 없는 데이터를 걸러냅니다. 다음으로 피드백을 학습 데이터 포맷으로 변환합니다. 7.2.1에서 다룬 Instruction-Input-Output 포맷으로 구조화하는 것입니다. "좋아요"를 받은 응답은 해당 질문-응답 쌍을 긍정 예시로 변환합니다. 사용자가 수정한 응답이 있으면, 원래 질문과 수정된 응답을 짝지어 학습 데이터로 만듭니다. 그 다음에 7.2.1에서 다룬 품질 지표(다양성, 정확성, 일관성)에 따라 변환된 데이터를 검증합니다. 마지막으로 검증을 통과한 데이터를 기존 학습 데이터셋에 추가합니다.

```
피드백 --> 학습 데이터 변환 파이프라인

수집된 피드백
     |
     v
[1. 정제]
  스팸/악의적 피드백 제거
  개인정보 마스킹
     |
     v
[2. 포맷 변환]
  좋아요 응답 --> (질문, 응답) 긍정 쌍
  수정된 응답 --> (질문, 수정 응답) 쌍
  싫어요 응답 --> (질문, 응답) 부정 쌍 (RLHF용)
     |
     v
[3. 품질 검증]
  다양성 확인 (기존 데이터와 중복도)
  정확성 확인 (자동 검증 + 샘플 수동 검토)
  일관성 확인 (형식/어조 통일)
     |
     v
[4. 데이터셋 추가]
  기존 학습 데이터에 병합
  버전 태깅
```

이 파이프라인에서 핵심은 자동화와 품질 사이의 균형입니다. 완전 자동화하면 품질이 낮은 데이터가 섞일 수 있고, 모든 것을 수동으로 검토하면 플라이휠의 속도가 느려집니다. 실무에서는 자동화된 필터로 대부분의 데이터를 처리하되, 통계적 샘플링으로 품질을 모니터링하는 방식을 사용합니다.

피드백 수집과 변환만으로는 데이터 플라이휠이 충분히 빠르게 돌지 않을 수 있습니다. 사용자가 피드백을 남기는 빈도가 낮거나, 특정 유형의 질문에 대한 피드백이 부족할 수 있습니다. 이런 경우에 **능동 학습(Active Learning)** 전략이 도움이 됩니다.

능동 학습은 모델이 스스로 "어떤 데이터를 학습하면 가장 많이 개선될지"를 판단하여, 해당 데이터에 대한 레이블(또는 피드백)을 우선적으로 요청하는 전략입니다. 일반적인 학습은 주어진 데이터를 무작위 순서로 학습하지만, 능동 학습은 학습 효과가 높을 것으로 예상되는 데이터를 선별하여 학습합니다.

LLM 파인튜닝에서 능동 학습을 적용하는 방식을 구체적으로 살펴보겠습니다. 모델이 운영 중에 받는 질문들을 모니터링합니다. 이 중에서 모델이 확신 없이 답변한 질문을 찾습니다. 확신도를 측정하는 방법으로는, 같은 질문에 여러 번 응답을 생성했을 때 응답 간 일관성이 낮은 경우, 출력 토큰의 확률 분포가 균등에 가까운 경우 등이 있습니다. 확신도가 낮은 질문은 모델이 잘 모르는 영역에 해당하므로, 이 질문에 대한 고품질 학습 데이터를 확보하면 모델이 크게 개선됩니다. 이런 질문을 선별하여 사람 검토자에게 보내거나, 더 강력한 모델에게 응답을 생성하도록 요청하여 학습 데이터를 만듭니다.

능동 학습의 효과는 학습 데이터의 효율성에서 나타납니다. 무작위로 선택한 1,000건의 데이터보다, 능동 학습으로 선별한 300건의 데이터가 더 큰 성능 향상을 가져올 수 있습니다. 같은 비용으로 더 효과적인 학습 데이터를 확보하는 것이므로, 플라이휠의 회전 효율이 높아집니다.

데이터 플라이휠이 돌면서 학습 데이터가 지속적으로 쌓이면, **데이터 버전 관리**가 필수가 됩니다. 코드에 Git이 있듯이, 데이터에도 버전 관리 도구가 있습니다. 대표적인 것이 **DVC(Data Version Control)**입니다.

DVC는 Git과 함께 동작하도록 설계된 데이터 버전 관리 도구입니다. Git은 코드와 설정 파일의 변경 이력을 추적하는 데 뛰어나지만, 대용량 데이터 파일을 직접 관리하기에는 적합하지 않습니다. 수 GB에 달하는 학습 데이터셋을 Git 저장소에 직접 넣으면, 저장소 크기가 폭발적으로 늘어나고 복제(clone) 시간이 길어집니다. DVC는 이 문제를 해결합니다. 실제 데이터 파일은 별도의 스토리지(로컬 디스크, AWS S3, Google Cloud Storage 등)에 저장하고, Git 저장소에는 데이터 파일의 해시값과 위치 정보만 담은 메타파일(.dvc 파일)을 넣습니다. Git이 메타파일의 변경 이력을 추적하므로, 결과적으로 데이터의 버전도 추적됩니다.

DVC를 통한 데이터 버전 관리의 흐름은 다음과 같습니다.

```
DVC 데이터 버전 관리 흐름

Git 저장소                          원격 스토리지 (S3 등)
+----------------------------+      +---------------------+
| 코드, 설정 파일            |      |                     |
| train.py                   |      | v1/ dataset.json    |
| config.yaml                |      | v2/ dataset.json    |
| dataset.json.dvc  <--------|----->| v3/ dataset.json    |
|  (해시값, 위치 정보)       |      |                     |
+----------------------------+      +---------------------+

1. dvc add dataset.json
   --> dataset.json.dvc 메타파일 생성

2. git add dataset.json.dvc && git commit
   --> 메타파일을 Git으로 버전 관리

3. dvc push
   --> 실제 데이터를 원격 스토리지에 업로드

4. git checkout v1 && dvc checkout
   --> v1 시점의 데이터로 복원
```

데이터 버전 관리가 중요한 이유는 재현성과 추적성 때문입니다. 모델 A는 v1 데이터로, 모델 B는 v2 데이터로 학습했다고 기록하면, 나중에 성능 차이의 원인을 추적할 수 있습니다. 문제가 생겼을 때 이전 버전의 데이터로 돌아가서 학습을 재현할 수도 있습니다. 데이터 플라이휠이 돌면서 데이터가 계속 변하므로, 각 시점의 데이터 상태를 기록하는 것은 필수입니다.

마지막으로 **플라이휠 효과 측정**을 다루겠습니다. 데이터 플라이휠이 실제로 모델 개선에 기여하는지를 확인하려면, 순환이 한 바퀴 돌 때마다 성과를 측정해야 합니다. 측정 없이 플라이휠을 돌리면, 데이터는 쌓이지만 모델이 나아지지 않는 상황을 감지하지 못합니다.

플라이휠 효과를 측정하는 지표는 크게 세 가지 축으로 나뉩니다. 첫째, 모델 성능 지표입니다. 8.1.1에서 다룬 평가 지표를 사용하여, 플라이휠 순환 전후의 모델 성능을 비교합니다. 순환 이전 모델의 평가 점수와 순환 이후 모델의 평가 점수 차이가 플라이휠의 효과입니다. 둘째, 데이터 지표입니다. 수집된 피드백의 양, 학습 데이터로 변환된 비율, 데이터의 다양성 변화를 추적합니다. 셋째, 사용자 지표입니다. 사용자 만족도, 응답에 대한 긍정 피드백 비율, 재방문율 등을 측정합니다. 모델이 개선되면 사용자 만족도가 높아지고, 만족한 사용자가 더 많은 피드백을 남기면서 플라이휠이 가속되는지를 확인합니다.

이 세 축의 지표를 순환 주기별로 기록하면, 플라이휠이 실제로 가속되고 있는지, 정체되고 있는지, 또는 오히려 악화되고 있는지를 판단할 수 있습니다. 성능이 정체되면 데이터 품질, 변환 파이프라인, 학습 전략 중 어디에 병목이 있는지 진단하고 개선합니다.

정리하면, 데이터 플라이휠은 사용자 피드백 수집, 학습 데이터 변환, 모델 재학습, 서비스 개선이 순환하는 구조로, 모델과 데이터의 품질을 지속적으로 향상시킵니다. 명시적, 암시적, 비교 피드백을 통해 사용자 반응을 수집하고, 정제와 포맷 변환을 거쳐 학습 데이터로 전환합니다. 능동 학습 전략으로 가장 효과적인 데이터를 선별하여 학습 효율을 높이고, DVC 같은 도구로 데이터의 버전을 관리합니다. 플라이휠의 효과는 모델 성능, 데이터, 사용자 지표의 세 축으로 측정하여 순환이 실제로 가속되는지를 확인합니다.

다음 단원인 7.3.1에서는 모델 갱신 전략과 Continual Learning을 다룹니다. 데이터 플라이휠로 새 데이터가 계속 쌓일 때, 모델을 어떻게 갱신해야 기존에 학습한 지식을 잃지 않으면서 새로운 지식을 반영할 수 있는지를 살펴봅니다.

이 단원을 마치면 사용자 피드백을 학습 데이터로 순환시키는 데이터 플라이휠을 설계할 수 있습니다.
