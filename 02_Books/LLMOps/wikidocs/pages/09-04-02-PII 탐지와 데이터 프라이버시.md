# 9.4.2 PII 탐지와 데이터 프라이버시

LLM 기반 서비스를 운영하다 보면, 사용자가 입력하는 텍스트 안에 이름, 전화번호, 주민등록번호 같은 개인정보가 섞여 들어오는 상황을 자주 만나게 됩니다. 고객 상담 챗봇에 "저는 홍길동이고 010-1234-5678로 연락 주세요"라고 입력하는 경우를 생각해 보겠습니다. 이 텍스트가 그대로 LLM에 전달되면, 모델의 로그에 개인정보가 기록됩니다. 외부 API를 사용하는 경우에는 제3자 서버에까지 전송됩니다. 9.4.1에서 다룬 모델 거버넌스 체계가 조직 차원의 통제 틀이라면, 이 단원에서는 그 틀 안에서 개인정보를 기술적으로 탐지하고 보호하는 구체적인 방법을 다룹니다.

이 단원에서 말하는 **PII**는 Personally Identifiable Information의 줄임말로, 한국어로는 '개인식별정보'라고 옮깁니다. PII란 특정 개인을 직접 또는 간접적으로 식별할 수 있는 모든 정보를 뜻합니다. '직접 식별'이란 그 정보 하나만으로 누구인지 알 수 있는 경우입니다. 주민등록번호, 여권 번호, 운전면허 번호가 대표적입니다. '간접 식별'이란 단독으로는 개인을 특정하기 어렵지만, 다른 정보와 결합하면 특정할 수 있는 경우입니다. 생년월일, 우편번호, 직장명 등이 여기에 해당합니다. 예를 들어, "1995년 3월생, 서울 강남구 거주, A 회사 근무"라는 세 가지 정보를 조합하면 대상을 상당히 좁힐 수 있습니다.

PII의 범위는 생각보다 넓습니다. 이름, 전화번호, 이메일 주소처럼 명확한 것만이 아닙니다. IP 주소, 디바이스 고유 식별자, 위치 데이터, 생체 정보(지문, 홍채), 금융 계좌 번호, 의료 기록 번호도 모두 PII에 포함됩니다. 어떤 정보가 PII인지 아닌지는 맥락에 따라 달라지기도 합니다. "서울특별시"라는 주소만으로는 개인을 식별하기 어렵지만, "서울특별시 강남구 테헤란로 123 A동 456호"라면 거주자를 특정할 가능성이 높아집니다.

LLM 파이프라인에서 PII가 문제가 되는 지점은 여러 곳에 걸쳐 있습니다. 크게 세 가지 단계로 나누어 살펴보겠습니다.

첫째, **입력 단계**입니다. 사용자가 프롬프트에 개인정보를 직접 입력합니다. 상담 챗봇에 자신의 이름과 연락처를 적거나, 문서 요약 서비스에 계약서 원문을 붙여넣는 경우가 이에 해당합니다. RAG(Retrieval-Augmented Generation) 파이프라인에서는 검색 대상 문서에 포함된 개인정보가 컨텍스트로 모델에 전달되기도 합니다.

둘째, **처리 단계**입니다. 모델이 입력을 처리하는 과정에서 개인정보가 서버 메모리에 올라갑니다. 외부 API를 호출하는 구조라면, 사용자의 개인정보가 네트워크를 통해 제3자 서버로 전송됩니다. 파인튜닝 데이터에 개인정보가 포함되어 있었다면, 모델이 학습 과정에서 해당 정보를 기억하고 추론 시 재생성할 가능성도 있습니다.

셋째, **출력 및 저장 단계**입니다. 모델의 응답 텍스트에 개인정보가 포함될 수 있습니다. 입력에 있던 정보를 그대로 반복하거나, 학습 데이터에서 암기한 정보를 출력하는 경우입니다. 이 응답이 로그에 기록되면, 로그 저장소에도 개인정보가 남습니다. 아래 다이어그램은 PII가 흐를 수 있는 경로를 정리한 것입니다.

```
PII 유출 경로

사용자 입력          RAG 문서 저장소        파인튜닝 데이터
    |                     |                      |
    v                     v                      v
+----------------------------------------------------------+
|                   LLM 파이프라인                           |
|  [프롬프트 조립] --> [모델 추론] --> [응답 생성]           |
+----------------------------------------------------------+
    |          |                           |
    v          v                           v
  API 로그   외부 모델 서버              응답 로그
  (입력 기록)  (제3자 전송)              (출력 기록)
```

이처럼 PII는 파이프라인의 시작부터 끝까지 여러 지점에서 노출될 수 있으므로, 각 지점에 맞는 탐지와 보호 수단이 필요합니다.

PII를 탐지하는 기법은 크게 세 가지로 나뉩니다. **정규표현식 기반 탐지**, **NER 기반 탐지**, **전용 모델 기반 탐지**입니다. 각각의 원리와 적합한 상황이 다르므로 순서대로 살펴보겠습니다.

가장 단순한 방법은 **정규표현식**(Regular Expression) 기반 탐지입니다. 정규표현식이란 텍스트에서 특정 패턴을 찾기 위한 문자열 규칙을 뜻합니다. 예를 들어, 한국 휴대전화 번호는 "010"으로 시작하고 하이픈을 사이에 두고 숫자 네 자리가 두 번 이어지는 패턴을 갖습니다. 이를 정규표현식으로 표현하면 다음과 같습니다.

```
010-\d{4}-\d{4}
```

이 패턴에서 `\d`는 숫자 한 자리를 뜻하고, `{4}`는 앞의 패턴이 정확히 네 번 반복됨을 뜻합니다. 이메일 주소, 주민등록번호(예: `\d{6}-\d{7}`), 신용카드 번호(예: `\d{4}-\d{4}-\d{4}-\d{4}`)처럼 형식이 고정된 정보에 정규표현식은 매우 효과적입니다. 패턴이 일치하면 거의 확실하게 해당 유형의 PII라고 판단할 수 있기 때문입니다.

그러나 정규표현식에는 한계가 있습니다. 사람의 이름처럼 형식이 정해져 있지 않은 정보는 패턴으로 포착하기 어렵습니다. "홍길동"이 사람 이름인지, 소설 속 인물인지, 상호명인지를 정규표현식만으로는 구분할 수 없습니다. 또한 전화번호라도 "010.1234.5678"이나 "01012345678"처럼 구분자가 다르면 패턴을 별도로 추가해야 합니다. 형식이 다양한 PII에 대응하려면 패턴의 수가 급격히 늘어나고, 유지보수가 어려워집니다.

두 번째 기법은 **NER 기반 탐지**입니다. NER은 Named Entity Recognition의 줄임말로, '개체명 인식'이라고 옮깁니다. NER이란 텍스트에서 사람 이름, 조직명, 장소, 날짜 등 의미 있는 단위(개체)를 자동으로 찾아내는 자연어 처리 기술입니다. 예를 들어, "홍길동은 2024년 1월에 삼성전자에 입사했다"라는 문장에서 NER 모델은 "홍길동"을 사람, "2024년 1월"을 날짜, "삼성전자"를 조직으로 분류합니다.

NER을 PII 탐지에 활용하면, 정규표현식으로는 잡기 어려운 사람 이름, 주소, 조직명 같은 비정형 개인정보도 탐지할 수 있습니다. spaCy, Hugging Face Transformers 등의 라이브러리에서 사전훈련된 NER 모델을 바로 사용할 수 있으며, 한국어의 경우에도 KoNLPy 등을 활용하거나 한국어 데이터로 훈련된 모델을 사용할 수 있습니다. 다만 NER 모델의 정확도는 훈련 데이터의 품질과 도메인에 따라 달라집니다. 의료 분야의 환자명과 일상 대화의 사람 이름은 등장하는 맥락이 다르므로, 범용 NER 모델이 모든 도메인에서 동일한 성능을 내지는 않습니다.

세 번째 기법은 **전용 모델 기반 탐지**입니다. PII 탐지만을 목적으로 훈련된 모델을 사용하는 방식입니다. Microsoft의 Presidio가 대표적인 오픈소스 도구입니다. Presidio는 정규표현식과 NER을 결합한 방식으로 동작합니다. 내부적으로 여러 '인식기(recognizer)'를 두고, 각 인식기가 이메일, 전화번호, 신용카드 번호 등 특정 유형의 PII를 탐지합니다. 정규표현식 기반 인식기와 NER 기반 인식기를 함께 사용하므로, 형식이 고정된 PII와 비정형 PII를 모두 다룰 수 있습니다. 탐지 결과에는 PII의 유형, 텍스트 내 위치, 신뢰도 점수가 포함됩니다.

아래는 세 가지 탐지 기법의 특성을 비교한 표입니다.

```
기법              적합한 PII 유형          장점                 한계
---------------------------------------------------------------------------
정규표현식        전화번호, 주민번호,      빠름, 정확도 높음    비정형 PII 탐지
                  카드번호 등 고정 형식    (형식 일치 시)       불가
---------------------------------------------------------------------------
NER               사람 이름, 조직명,       비정형 정보 탐지     도메인별 정확도
                  주소, 날짜               가능                 편차
---------------------------------------------------------------------------
전용 모델         복합 유형                정규식+NER 결합,     설정 및 커스터
(Presidio 등)     (정형+비정형)            신뢰도 점수 제공     마이징 필요
```

실무에서는 이 기법들을 단독으로 쓰기보다 조합하여 사용합니다. 정규표현식으로 형식이 명확한 PII를 먼저 잡고, NER이나 전용 모델로 비정형 PII를 보완하는 구조가 일반적입니다.

PII를 탐지한 다음에는 해당 정보를 보호하는 조치가 필요합니다. 이를 **PII 마스킹**과 **비식별화**라고 합니다. 마스킹이란 원본 텍스트에서 PII에 해당하는 부분을 다른 값으로 대체하여, 원래 정보를 알 수 없게 만드는 것입니다. 비식별화는 마스킹을 포함하여 데이터에서 개인을 식별할 수 없도록 처리하는 모든 기법을 아우르는 넓은 개념입니다.

마스킹의 구체적인 방식은 여러 가지가 있습니다. 가장 단순한 방법은 **치환(replacement)**입니다. PII를 유형 태그로 바꾸는 것입니다. "홍길동에게 010-1234-5678로 연락하세요"라는 문장이 있다면, 마스킹 후에는 "[PERSON]에게 [PHONE_NUMBER]로 연락하세요"가 됩니다. 이렇게 하면 LLM에 전달되는 텍스트에는 실제 개인정보가 포함되지 않으면서, 문장의 구조와 맥락은 유지됩니다. 모델이 응답을 생성한 뒤에는 태그를 다시 원래 값으로 복원할 수도 있고, 태그 상태 그대로 사용자에게 전달할 수도 있습니다.

다른 방법으로는 **해싱(hashing)**이 있습니다. 해싱이란 원본 값을 되돌릴 수 없는 고정 길이 문자열로 변환하는 것입니다. "홍길동"을 해시 함수에 넣으면 "a3f2b8c1..."처럼 의미를 알 수 없는 문자열이 됩니다. 같은 입력은 항상 같은 해시값을 내므로, 원본을 복원하지 않고도 동일 인물 여부를 비교할 수 있습니다. 다만 해시값 자체가 문맥 안에서 의미를 갖지 않으므로, LLM이 해시값을 포함한 텍스트를 처리할 때 자연스러운 응답을 생성하기 어려울 수 있습니다.

**가명처리(pseudonymization)**도 널리 쓰이는 기법입니다. 가명처리란 실제 PII를 가짜이지만 형식이 동일한 값으로 바꾸는 것입니다. "홍길동"을 "김민수"로, "010-1234-5678"을 "010-9876-5432"로 바꾸는 식입니다. 가명처리된 데이터는 원본과 형식이 같으므로 LLM이 자연스럽게 처리할 수 있다는 장점이 있습니다. 다만 가명과 원본의 대응 관계를 관리해야 하며, 대응표가 유출되면 원본을 복원할 수 있으므로 대응표 자체의 보안이 중요합니다.

이러한 마스킹 기법을 LLM 파이프라인에 적용하는 흐름은 다음과 같습니다.

```
PII 마스킹 파이프라인

사용자 입력
    |
    v
[1. PII 탐지] --- 정규표현식 + NER + 전용 모델
    |
    v
[2. PII 마스킹] --- 치환 / 해싱 / 가명처리
    |
    v
[3. 마스킹된 텍스트를 LLM에 전달]
    |
    v
[4. LLM 응답 생성]
    |
    v
[5. 필요 시 마스킹 역복원] --- 태그를 원본으로 복원
    |
    v
최종 응답
```

1단계에서 입력 텍스트의 PII를 탐지합니다. 2단계에서 탐지된 PII를 선택한 방식으로 마스킹합니다. 3단계에서 마스킹된 텍스트를 모델에 전달합니다. 이 시점에서 모델이 처리하는 텍스트에는 실제 개인정보가 없습니다. 4단계에서 모델이 응답을 생성합니다. 5단계에서 응답에 포함된 태그를 원래 값으로 복원합니다. 복원이 필요 없는 경우(예: 내부 분석 용도)에는 5단계를 생략합니다.

PII 보호는 탐지와 마스킹만으로 끝나지 않습니다. 데이터가 얼마나 오래, 어디에 저장되는지를 관리하는 **데이터 보존 정책**도 필수입니다. 데이터 보존 정책이란 수집된 데이터의 저장 기간, 저장 위치, 접근 권한, 삭제 절차를 명시한 규정입니다.

LLM 파이프라인에서 데이터가 저장되는 지점은 여러 곳입니다. 사용자 입력 로그, 모델 응답 로그, RAG용 문서 저장소, 파인튜닝 데이터셋, 평가 데이터셋 등이 모두 해당합니다. 각 저장 지점에 대해 다음 항목을 정해야 합니다. 얼마나 오래 보관할 것인지(예: 입력 로그는 90일 후 삭제), 어디에 저장할 것인지(예: 암호화된 내부 스토리지), 누가 접근할 수 있는지(예: 운영팀만 접근 가능), 삭제 요청이 들어오면 어떻게 처리할 것인지(예: 30일 이내 완전 삭제)입니다.

데이터 보존 정책이 없으면 문제가 점진적으로 커집니다. 로그가 무한히 쌓이면서 저장 비용이 증가하고, 오래된 로그에 포함된 개인정보가 유출 사고의 범위를 넓히며, 규제 기관의 감사에서 "왜 이 데이터를 아직 보관하고 있는가"라는 질문에 답하지 못하는 상황이 발생합니다.

마지막으로, PII 보호와 직결되는 주요 **프라이버시 규제**를 살펴보겠습니다. LLM을 운영하는 조직이라면 서비스 대상 지역의 규제를 반드시 파악해야 합니다.

**GDPR**(General Data Protection Regulation)은 유럽연합(EU)의 개인정보 보호 규정입니다. 2018년 시행되었으며, EU 시민의 개인정보를 처리하는 모든 조직에 적용됩니다. 조직이 EU 밖에 있더라도 EU 시민의 데이터를 다루면 적용 대상입니다. GDPR에서 LLM 운영과 관련하여 특히 중요한 원칙은 다음과 같습니다. '목적 제한'은 수집 시 명시한 목적 외에 데이터를 사용하지 않아야 한다는 원칙입니다. '데이터 최소화'는 목적 달성에 필요한 최소한의 데이터만 수집해야 한다는 원칙입니다. '삭제권(잊힐 권리)'은 정보 주체가 자신의 데이터 삭제를 요청할 수 있다는 권리입니다. LLM 파이프라인에서 사용자의 프롬프트를 파인튜닝 데이터로 활용하거나 로그에 무기한 보관한다면, 이 원칙들에 위배될 수 있습니다. GDPR 위반 시 전 세계 연매출의 4% 또는 2천만 유로 중 높은 금액이 과징금으로 부과될 수 있습니다.

**CCPA**(California Consumer Privacy Act)는 미국 캘리포니아주의 소비자 프라이버시법입니다. 2020년 시행되었으며, 캘리포니아 주민의 개인정보를 다루는 일정 규모 이상의 기업에 적용됩니다. CCPA는 소비자에게 '알 권리'(기업이 어떤 개인정보를 수집하는지 알 권리), '삭제 요청권', '판매 거부권'(개인정보 판매를 거부할 권리)을 부여합니다. LLM 서비스에서 사용자의 대화 내용을 수집하고 이를 모델 개선에 활용한다면, 이 활용 사실을 사용자에게 고지하고 거부 수단을 제공해야 합니다.

한국에서는 **개인정보 보호법**이 PII 관련 주요 규제입니다. 개인정보의 수집, 이용, 제공, 파기 전반에 걸쳐 정보 주체의 동의를 기본 요건으로 하며, 가명정보 활용에 관한 조항도 포함하고 있습니다. LLM 서비스가 한국 사용자를 대상으로 한다면, 입력 데이터의 수집 목적과 보유 기간을 명시하고, 동의를 받는 절차가 필요합니다.

이러한 규제에 대응하기 위해 LLM 운영 조직이 갖추어야 할 사항을 정리하면 다음과 같습니다. 첫째, 파이프라인의 어느 지점에서 PII를 수집하는지 파악하는 데이터 흐름 매핑이 필요합니다. 둘째, 각 수집 지점에 PII 탐지 및 마스킹 모듈을 배치합니다. 셋째, 데이터 보존 정책을 수립하여 저장 기간과 삭제 절차를 명확히 합니다. 넷째, 삭제 요청에 대응할 수 있는 기술적 수단을 갖춥니다. 다섯째, 이 모든 절차를 문서화하여 감사 추적이 가능하도록 합니다.

```
프라이버시 규제 대응 체크리스트

[1단계] 데이터 흐름 매핑
  - 입력, 처리, 출력, 로그 각 지점에서 PII 식별

[2단계] PII 탐지 및 마스킹 배치
  - 입력 전처리 단계에 탐지 모듈 삽입
  - 로그 저장 전 마스킹 적용

[3단계] 데이터 보존 정책 수립
  - 저장 기간, 위치, 접근 권한, 삭제 절차 명시

[4단계] 삭제 요청 처리 체계 구축
  - 정보 주체의 삭제 요청을 정해진 기한 내에 이행

[5단계] 문서화 및 감사 추적
  - 처리 이력 기록, 정기 점검 수행
```

정리하면, PII는 LLM 파이프라인의 입력, 처리, 출력, 저장 전 단계에 걸쳐 노출될 수 있으며, 이를 체계적으로 보호하려면 탐지, 마스킹, 보존 정책, 규제 대응이라는 네 가지 축을 모두 갖추어야 합니다. 정규표현식은 형식이 고정된 PII를, NER과 전용 모델은 비정형 PII를 담당하며, 마스킹은 치환, 해싱, 가명처리 등 상황에 맞는 방식을 선택합니다. 데이터 보존 정책으로 저장 기간과 삭제 절차를 통제하고, GDPR, CCPA, 개인정보 보호법 등 서비스 대상 지역의 규제 요건을 충족하는 것이 LLM 운영 조직의 기본 책무입니다.

이 단원을 마치면 LLM 파이프라인에서 PII를 탐지하고 마스킹하는 기법을 선택하여 적용하고, 데이터 보존 정책과 프라이버시 규제 대응 체계를 수립할 수 있습니다.
