# 9.3.2 프롬프트와 모델의 A/B 테스트

9.3.1에서 LLM 애플리케이션의 CI/CD 파이프라인을 다루면서, 프롬프트나 모델이 변경될 때마다 자동화된 평가 게이트를 통과해야 한다는 점을 살펴보았습니다. 평가 게이트는 "이 변경이 기존보다 나쁘지 않은가"를 판단하는 장치입니다. 그런데 "나쁘지 않다"는 것과 "더 낫다"는 것은 다른 문제입니다. 프롬프트를 새로 작성했을 때, 또는 기존 모델을 더 새로운 모델로 교체했을 때, 실제 사용자 환경에서 어느 쪽이 더 좋은 결과를 내는지를 정량적으로 비교해야 하는 상황이 생깁니다. 이때 사용하는 방법이 A/B 테스트입니다.

A/B 테스트는 웹 서비스 분야에서 오래전부터 쓰여 온 실험 방법입니다. 원래는 웹 페이지의 버튼 색상, 레이아웃, 문구 등을 두 가지 버전으로 만들어 사용자 집단을 무작위로 나눈 뒤, 어느 버전이 클릭률이나 전환율에서 더 높은 성과를 내는지 비교하는 데 쓰였습니다. 핵심 원리는 간단합니다. 비교하고 싶은 두 가지 변형(A와 B)을 준비하고, 사용자를 무작위로 두 그룹에 배정하고, 각 그룹에 서로 다른 변형을 보여준 뒤, 결과 지표를 측정하여 차이가 우연인지 실제 효과인지를 통계적으로 판단합니다.

이 원리를 LLM 환경에 적용하면 두 가지 축이 생깁니다. 하나는 **프롬프트 A/B 테스트**이고, 다른 하나는 **모델 A/B 테스트**입니다. 프롬프트 A/B 테스트는 동일한 모델에 서로 다른 프롬프트를 적용하여 응답 품질을 비교합니다. 모델 A/B 테스트는 동일한 프롬프트를 서로 다른 모델에 전달하여 성능, 비용, 지연 시간 등을 비교합니다. 두 경우 모두 "어느 쪽이 더 나은가"를 감이 아닌 데이터로 판단한다는 공통점이 있습니다.

프롬프트 A/B 테스트를 설계하려면 먼저 무엇을 비교할지 명확히 해야 합니다. 예를 들어 고객 문의에 답변하는 챗봇을 운영하고 있다고 가정합니다. 현재 사용 중인 프롬프트(A)는 "고객의 질문에 정확하고 간결하게 답변하세요"라는 지시를 포함하고, 새로 테스트할 프롬프트(B)는 "고객의 질문에 답변하되, 관련 FAQ 링크를 함께 제공하세요"라는 지시를 포함합니다. 이때 비교 지표로는 사용자 만족도 점수, 추가 문의 비율, 평균 대화 턴 수 등을 선택할 수 있습니다.

프롬프트 A/B 테스트의 설계 과정을 단계별로 정리하면 다음과 같습니다.

```
프롬프트 A/B 테스트 설계 흐름

1. 가설 수립
   "프롬프트 B는 프롬프트 A보다 사용자 만족도를 높일 것이다"

2. 지표 선정
   주 지표: 사용자 만족도 점수 (1~5)
   보조 지표: 추가 문의 비율, 응답 지연 시간

3. 트래픽 분배 결정
   전체 요청의 10%를 B에, 90%를 A에 배정 (안전 우선)

4. 실험 기간 산정
   필요 표본 크기를 계산하여 최소 실험 기간을 정함

5. 실험 실행
   라우터가 요청을 무작위로 A 또는 B 프롬프트로 분기

6. 결과 수집 및 분석
   지표를 집계하고 통계적 유의성을 판단

7. 의사결정
   B가 유의미하게 우수하면 전체 트래픽을 B로 전환
   차이가 없거나 B가 열등하면 A를 유지
```

모델 A/B 테스트의 설계도 구조는 비슷하지만, 비교 대상과 관심 지표가 다릅니다. 예를 들어 현재 GPT-4를 사용하고 있는 요약 서비스에서 GPT-4o-mini로 교체하는 것을 검토한다고 가정합니다. 프롬프트는 동일하게 유지하되, 요청의 일부를 새 모델로 보내서 요약 품질, 호출당 비용, 응답 지연 시간을 비교합니다. 모델 A/B 테스트에서 비용은 특히 중요한 지표입니다. 품질이 거의 동등하다면, 토큰당 비용이 낮은 모델로 전환하는 것만으로도 운영 비용을 크게 줄일 수 있기 때문입니다.

모델 A/B 테스트에서 추가로 고려해야 할 점이 있습니다. 서로 다른 모델은 동일한 프롬프트에 대해서도 응답 형식이나 길이가 다를 수 있습니다. 따라서 평가 지표를 설계할 때 응답 형식의 차이가 지표에 영향을 미치지 않도록 주의해야 합니다. 예를 들어 "응답 길이"를 품질 지표로 쓰면, 단순히 더 긴 응답을 생성하는 모델이 유리해질 수 있습니다. 응답의 정확성, 관련성, 완결성 같은 내용 중심 지표를 사용하는 것이 바람직합니다.

트래픽을 두 변형에 어떻게 나누느냐도 중요한 설계 요소입니다. 이를 **트래픽 분배 전략**이라 합니다. 가장 단순한 방법은 50 대 50으로 균등 분배하는 것입니다. 이렇게 하면 가장 빠르게 충분한 데이터를 모을 수 있습니다. 그러나 새 프롬프트나 새 모델이 예상보다 나쁜 결과를 낼 위험이 있으므로, 프로덕션 환경에서는 보수적인 비율로 시작하는 것이 일반적입니다. 예를 들어 기존 버전(A)에 95%, 새 버전(B)에 5%를 배정합니다. 초기 결과가 안정적이면 B의 비율을 10%, 20%로 점진적으로 올립니다. 이 방식을 **점진적 롤아웃**이라 하며, 9.3.1에서 다룬 카나리 배포와 유사한 개념입니다.

트래픽 분배에서 한 가지 더 고려할 점은 사용자 고정(sticky assignment)입니다. 한 사용자가 같은 세션 안에서 요청을 여러 번 보낼 때, 매번 다른 변형으로 배정되면 일관성 없는 경험을 하게 됩니다. 따라서 사용자 ID나 세션 ID를 기준으로 한 번 배정된 변형을 유지하는 것이 바람직합니다. 구현 방법은 간단합니다. 사용자 ID를 해시 함수에 넣어 0에서 99 사이의 숫자를 구하고, 그 숫자가 5 미만이면 B, 5 이상이면 A로 배정하는 식입니다.

```
트래픽 분배 예시: 해시 기반 사용자 고정

사용자 ID: "user_1234"
해시값: hash("user_1234") % 100 = 37
분배 기준: 0~4 -> B, 5~99 -> A
결과: 사용자 "user_1234"는 A 그룹

사용자 ID: "user_5678"
해시값: hash("user_5678") % 100 = 2
분배 기준: 0~4 -> B, 5~99 -> A
결과: 사용자 "user_5678"는 B 그룹
```

이 방식을 사용하면 동일한 사용자는 항상 같은 그룹에 배정되면서도, 전체적으로는 설정한 비율에 가깝게 분배됩니다.

실험 결과가 모이면 "두 변형 사이의 차이가 실제로 존재하는가, 아니면 우연에 의한 것인가"를 판단해야 합니다. 이것이 **통계적 유의성** 판단입니다. 통계적 유의성이란, 관찰된 차이가 순전히 우연으로 발생했을 확률이 충분히 낮다는 것을 뜻합니다.

통계적 유의성을 판단하는 과정을 구체적인 시나리오로 설명합니다. 프롬프트 A를 받은 1,000명의 사용자 중 만족도 평균이 3.8이었고, 프롬프트 B를 받은 100명의 사용자 중 만족도 평균이 4.1이었다고 가정합니다. B가 0.3점 높습니다. 그런데 이 차이가 진짜 프롬프트의 효과인지, 아니면 B 그룹에 우연히 만족하기 쉬운 사용자가 몰린 것인지를 구분해야 합니다.

이를 판단하기 위해 **p값**(p-value)을 계산합니다. p값이란, 실제로는 두 변형 사이에 차이가 없는데도 관찰된 것만큼 또는 그 이상의 차이가 나타날 확률입니다. 일반적으로 p값이 0.05 미만이면 "통계적으로 유의하다"고 판단합니다. 이 기준은 "차이가 없는데 이 정도 결과가 나올 확률이 5% 미만이니, 실제 차이가 있다고 보는 것이 합리적이다"라는 의미입니다.

p값만으로 의사결정을 내리기에 부족한 경우가 있습니다. p값이 유의하더라도 실제 차이의 크기가 운영에 의미 있을 만큼 큰지를 함께 확인해야 합니다. 만족도 5점 척도에서 0.01점 차이는 통계적으로 유의할 수 있지만, 실무적으로는 무의미할 수 있습니다. 반대로, 비용이 20% 감소하면서 품질 차이가 유의하지 않다면, 비용 절감을 이유로 전환을 결정할 수도 있습니다. 통계적 유의성과 실무적 유의성을 함께 고려하는 것이 중요합니다.

실험에 필요한 **표본 크기**도 사전에 계산해야 합니다. 표본이 너무 적으면 실제 차이가 있더라도 유의성을 확인하지 못합니다. 표본 크기는 세 가지 요소에 의해 결정됩니다. 첫째, 검출하려는 최소 효과 크기(예: 만족도 0.2점 차이)입니다. 둘째, 데이터의 분산(값들이 얼마나 흩어져 있는가)입니다. 셋째, 원하는 통계적 검정력(실제 차이가 있을 때 이를 감지할 확률, 보통 80% 이상)입니다. 이 세 값을 정하면 필요한 최소 표본 수를 계산할 수 있고, 하루 평균 요청 수로 나누면 최소 실험 기간이 나옵니다.

```
표본 크기 결정 요소

+---------------------+     +------------------+     +------------------+
| 최소 효과 크기      |     | 데이터 분산      |     | 통계적 검정력    |
| (검출할 최소 차이)  |     | (값의 흩어짐)    |     | (감지 확률)      |
| 예: 만족도 0.2점    |     | 예: 표준편차 0.8 |     | 예: 80%          |
+---------------------+     +------------------+     +------------------+
          |                          |                         |
          +----------+---------------+-----------+-------------+
                     |                           |
                     v                           v
          +---------------------+     +---------------------+
          | 필요 표본 크기      |     | 최소 실험 기간      |
          | 예: 각 그룹 400명   | --> | 예: 5일             |
          +---------------------+     | (하루 요청 80건,    |
                                      |  B 그룹 5% 배정)    |
                                      +---------------------+
```

A/B 테스트를 매번 수동으로 설계하고, 트래픽을 수동으로 분배하고, 결과를 수동으로 분석하는 것은 반복적이고 실수가 나기 쉬운 작업입니다. LLM 서비스에서 프롬프트와 모델 변경은 빈번하게 일어나므로, 이 과정을 **자동화**하는 것이 중요합니다.

A/B 테스트 자동화의 기본 구조는 다음과 같습니다. 먼저 프롬프트나 모델의 새 버전을 등록합니다. 시스템이 미리 정해진 트래픽 비율에 따라 자동으로 요청을 분기합니다. 각 변형의 응답에 대해 자동화된 평가 지표(예: LLM 기반 판정, 사용자 피드백 수집)가 실행됩니다. 결과가 충분히 쌓이면 통계 분석이 자동으로 수행되고, 유의미한 차이가 확인되면 우수한 변형으로 전체 트래픽을 전환합니다. 차이가 없거나 새 변형이 열등하면 기존 버전을 유지합니다.

```
A/B 테스트 자동화 파이프라인

새 프롬프트/모델 등록
        |
        v
+------------------+
| 트래픽 라우터    |----> A (기존): 95% 트래픽
| (자동 분기)      |----> B (신규):  5% 트래픽
+------------------+
        |
        v
+------------------+
| 응답 수집 및     |
| 자동 평가        |
| (품질, 비용,     |
|  지연 시간)      |
+------------------+
        |
        v
+------------------+
| 통계 분석 엔진   |
| (p값, 효과 크기  |
|  자동 계산)      |
+------------------+
        |
   +----+----+
   |         |
   v         v
B 우수    B 열등 또는 차이 없음
   |         |
   v         v
B로 전환  A 유지 (B 폐기)
```

이 자동화 파이프라인이 9.3.1에서 다룬 CI/CD와 연결되면, 프롬프트나 모델 변경이 코드 저장소에 커밋되는 순간부터 A/B 테스트 실행, 결과 판정, 전체 배포까지 사람의 개입 없이 진행될 수 있습니다. 물론 완전 자동 전환이 위험한 경우(예: 의료, 금융 등 고위험 도메인)에는 자동 분석까지만 수행하고, 최종 전환 결정은 사람이 승인하는 반자동 방식을 택할 수도 있습니다.

LLM 환경의 A/B 테스트에서 특별히 주의할 점이 하나 더 있습니다. LLM의 출력은 비결정적이므로, 같은 프롬프트와 같은 모델이라도 매번 다른 응답이 나올 수 있습니다. 이 자체적인 변동성이 실험 결과의 분산을 키우고, 필요한 표본 크기를 늘립니다. 따라서 LLM A/B 테스트에서는 전통적인 웹 A/B 테스트보다 더 많은 표본이 필요할 수 있으며, temperature 같은 생성 파라미터를 고정하여 불필요한 변동성을 줄이는 것이 실험의 정밀도를 높이는 데 도움이 됩니다.

정리하면, A/B 테스트는 프롬프트나 모델의 변경이 실제 성능 향상으로 이어지는지를 데이터에 기반하여 판단하는 방법입니다. 프롬프트 A/B 테스트는 동일 모델에 다른 프롬프트를, 모델 A/B 테스트는 동일 프롬프트에 다른 모델을 적용하여 비교합니다. 트래픽 분배는 해시 기반 사용자 고정과 점진적 롤아웃으로 안전하게 수행하고, 결과는 p값과 효과 크기를 함께 고려하여 통계적으로 판단합니다. 이 과정을 CI/CD 파이프라인과 연결하여 자동화하면, 빈번한 프롬프트 및 모델 변경에도 체계적인 품질 관리가 가능합니다.

다음 단원인 9.3.3에서는 Rate Limiting과 API Gateway를 다룹니다.

이 단원을 마치면 LLM 환경에서 A/B 테스트를 설계하고 통계적으로 유의미한 결과를 도출할 수 있습니다.
