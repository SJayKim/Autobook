# 2.1.2 고급 프롬프트 기법

2.1.1에서 프롬프트 엔지니어링의 정의와 기본 기법을 살펴보았습니다. Zero-shot, Few-shot, Chain-of-Thought(CoT) 같은 기법은 프롬프트 하나로 모델의 응답 품질을 크게 개선할 수 있다는 점을 보여 주었습니다. 그런데 실무에서 LLM을 활용하다 보면, 기본 기법만으로는 해결하기 어려운 상황이 나타납니다. 모델이 논리적으로 그럴듯하지만 틀린 답을 내놓거나, 복잡한 문제에서 중간 추론 한 단계를 잘못 밟아 전체 결론이 어긋나는 경우가 대표적입니다. 답변이 자연어 문장이라 후속 시스템에서 파싱하기 어려운 상황, 하나의 프롬프트로 해결하기엔 작업이 너무 복잡한 상황, 외부 도구나 데이터를 실시간으로 조회해야 하는 상황도 있습니다. 이 단원에서는 이러한 한계를 넘기 위해 개발된 고급 프롬프트 기법 여섯 가지를 다룹니다.

2.1.1에서 다룬 CoT는 모델에게 "단계별로 생각하라"고 지시하여 추론 과정을 명시적으로 풀어내게 하는 기법이었습니다. CoT를 적용하면 복잡한 산술 문제나 논리 문제에서 정답률이 올라갑니다. 그런데 CoT에도 약점이 있습니다. 모델이 한 번의 추론 경로만 생성하기 때문에, 그 경로가 잘못되면 최종 답도 틀려집니다. 사람이 문제를 풀 때를 생각해 보겠습니다. 어려운 수학 문제를 만나면 한 가지 풀이만 시도하지 않습니다. 여러 방법으로 풀어 보고, 여러 풀이에서 같은 답이 나오면 그 답을 신뢰합니다. **Self-Consistency** 기법은 이 직관을 LLM에 그대로 적용한 것입니다.

Self-Consistency의 동작 방식은 다음과 같습니다. 먼저, 동일한 문제에 대해 CoT 프롬프트를 여러 번 실행합니다. 이때 temperature 파라미터를 0보다 높게 설정하여 매번 다른 추론 경로를 생성하게 합니다. temperature는 1.2.3에서 다룬 것처럼, 모델이 다음 토큰을 선택할 때 확률 분포를 얼마나 고르게 만들지를 조절하는 값입니다. temperature가 0이면 매번 같은 답이 나오고, 높을수록 다양한 답이 나옵니다. Self-Consistency에서는 이 다양성을 의도적으로 활용합니다.

여러 번 실행한 결과로 서로 다른 추론 경로와 최종 답이 모입니다. 예를 들어, 5번 실행해서 3번은 "42", 1번은 "38", 1번은 "45"가 나왔다면 가장 많이 나온 "42"를 최종 답으로 선택합니다. 이것을 **다수결 투표(majority voting)**라고 합니다. 핵심은 중간 추론 경로는 제각각이어도 최종 답이 일치하는지를 보는 것입니다. 서로 다른 경로로 풀었는데 같은 답이 나왔다면, 그 답이 맞을 가능성이 높습니다.

```
Self-Consistency 동작 흐름

[동일한 문제 + CoT 프롬프트]
         |
    temperature > 0으로 N번 실행
         |
   +-----+-----+-----+-----+
   |     |     |     |     |
경로1  경로2  경로3  경로4  경로5
답:42  답:42  답:38  답:42  답:45
   |     |     |     |     |
   +-----+-----+-----+-----+
         |
    다수결 투표 (majority voting)
         |
    최종 답: 42
```

Self-Consistency는 단순한 만큼 구현이 쉽습니다. 프롬프트를 바꿀 필요 없이, 같은 프롬프트를 여러 번 호출하고 결과를 집계하면 됩니다. 다만 모델을 N번 호출해야 하므로 비용과 지연 시간이 N배로 늘어납니다. 정답이 명확한 숫자나 선택지 형태일 때 가장 효과적이며, 긴 서술형 답변에서는 "같은 답"의 기준을 정하기가 어려워 적용이 제한적입니다.

Self-Consistency가 "같은 문제를 여러 경로로 풀어서 비교"하는 접근이라면, **Tree-of-Thought(ToT)**는 문제를 푸는 과정 자체를 체계적으로 탐색하는 접근입니다. CoT는 하나의 직선적인 추론 경로를 따릅니다. 1단계에서 2단계로, 2단계에서 3단계로 쭉 이어갑니다. 중간에 잘못된 방향으로 가더라도 되돌아오지 않습니다. ToT는 이 한계를 극복하기 위해, 각 추론 단계에서 여러 갈래의 가능성을 동시에 펼치고, 유망한 갈래만 골라 더 깊이 탐색합니다.

구체적인 시나리오로 설명하겠습니다. "24 게임"이라는 퍼즐이 있습니다. 4개의 숫자를 사칙연산으로 조합해서 24를 만드는 게임입니다. 예를 들어, 1, 5, 5, 5가 주어졌다면 5 * (5 - 1 / 5) = 24라는 해가 있습니다. CoT로 이 문제를 풀면, 모델은 하나의 조합을 시도하고 끝까지 밀고 나갑니다. 중간에 "이 조합으로는 24가 안 되는 것 같다"는 판단 없이 결론까지 가 버립니다.

ToT는 다르게 접근합니다. 첫 번째 단계에서 "어떤 두 숫자를 먼저 연산할까"를 여러 후보로 펼칩니다. 1+5=6, 5-1=4, 5*1=5 같은 분기가 생깁니다. 두 번째 단계에서는 각 분기에서 다시 남은 숫자로 가능한 연산을 펼칩니다. 이렇게 하면 트리(나무) 모양의 탐색 구조가 만들어집니다. 여기서 핵심은 **평가 단계**입니다. 각 분기가 24에 도달할 가능성이 있는지를 모델이 스스로 평가하고, 가능성이 낮은 분기는 가지치기(pruning)합니다. 유망한 분기만 남겨서 다음 단계를 탐색하므로, 전체 경우의 수를 무작정 늘리지 않으면서도 정답을 찾을 확률이 높아집니다.

```
Tree-of-Thought 탐색 구조

                    [문제: 1,5,5,5로 24 만들기]
                     /          |          \
              1+5=6          5-1=4         5*1=5
              /   \          /   \          /   \
         6*5=30  6+5=11  4*5=20  4+5=9  5*5=25  5+5=10
         (가지   (가지    (가지   (가지   (가지    (가지
         치기)   치기)    치기)   치기)   치기)    치기)
                          ...
                  [유망 분기를 깊이 탐색]
                          ...
              5*(5-1/5) = 5*4.8 = 24  (정답 발견)
```

ToT의 탐색 방식은 두 가지가 있습니다. 하나는 **너비 우선 탐색(BFS)**으로, 같은 깊이의 모든 분기를 평가한 뒤 상위 K개만 다음 단계로 보내는 방식입니다. 다른 하나는 **깊이 우선 탐색(DFS)**으로, 한 분기를 끝까지 따라가 보고, 막히면 되돌아가서 다른 분기를 시도하는 방식입니다. 문제의 성격에 따라 적합한 탐색 방식이 다릅니다. 경우의 수가 넓게 펼쳐지지만 깊지 않은 문제에는 BFS가, 분기는 적지만 깊이 있는 추론이 필요한 문제에는 DFS가 유리합니다.

ToT는 창의적 글쓰기, 퍼즐 풀이, 전략 수립처럼 탐색과 되돌아가기(backtracking)가 필요한 과제에서 CoT보다 높은 성능을 보입니다. 다만 여러 분기를 생성하고 각 분기를 평가하기 위해 모델을 여러 번 호출해야 하므로, Self-Consistency와 마찬가지로 비용이 증가합니다. 또한 구현의 복잡도가 높아 단순한 질의응답보다는, 기존 방법으로 풀리지 않는 복잡한 추론 과제에 선별적으로 적용하는 것이 적절합니다.

지금까지 다룬 Self-Consistency와 ToT는 모델이 "생각을 더 잘 하도록" 돕는 기법입니다. 그런데 아무리 추론을 잘해도, 모델의 지식이 학습 시점에 고정되어 있다는 근본적인 한계는 남습니다. 오늘의 날씨를 묻거나, 특정 데이터베이스를 조회하거나, 계산기를 사용해야 하는 상황에서 모델은 스스로 답을 만들어 내려다 사실과 다른 내용을 생성할 수 있습니다. **ReAct** 패턴은 이 문제를 해결합니다. ReAct는 **Reasoning**(추론)과 **Acting**(행동)을 결합한 이름 그대로, 모델이 생각하고 행동하는 것을 번갈아 수행하는 패턴입니다.

ReAct의 동작을 단계별로 살펴보겠습니다. 모델은 질문을 받으면 먼저 Thought(생각) 단계에서 "이 질문에 답하려면 무엇을 알아야 하는가"를 추론합니다. 그 다음 Action(행동) 단계에서 외부 도구를 호출합니다. 검색 엔진에 쿼리를 보내거나, 계산기를 실행하거나, API를 호출하는 것이 Action에 해당합니다. 도구가 결과를 돌려주면 Observation(관찰) 단계에서 그 결과를 읽습니다. 모델은 Observation을 바탕으로 다시 Thought를 수행하여, 추가 Action이 필요한지 아니면 최종 답을 내릴 수 있는지를 판단합니다. 이 Thought-Action-Observation 순환이 답을 얻을 때까지 반복됩니다.

```
ReAct 패턴의 Thought-Action-Observation 순환

질문: "2024년 노벨 물리학상 수상자의 출생 도시 인구는?"

Thought 1: 먼저 2024년 노벨 물리학상 수상자가 누구인지 검색해야 합니다.
Action 1:  Search("2024 Nobel Physics Prize winner")
Observation 1: John Hopfield와 Geoffrey Hinton이 수상했습니다.

Thought 2: Geoffrey Hinton의 출생 도시를 알아야 합니다.
Action 2:  Search("Geoffrey Hinton birthplace")
Observation 2: 영국 런던 윔블던에서 태어났습니다.

Thought 3: 런던의 인구를 검색하겠습니다.
Action 3:  Search("London population")
Observation 3: 약 900만 명입니다.

Thought 4: 충분한 정보를 모았으므로 답변할 수 있습니다.
Final Answer: Geoffrey Hinton의 출생 도시인 런던의 인구는 약 900만 명입니다.
```

ReAct의 장점은 모델의 추론 과정이 투명하게 드러나고, 외부 정보를 실시간으로 가져올 수 있다는 것입니다. 단순히 모델의 기억에 의존하는 것이 아니라, 필요한 정보를 직접 조회하여 사실에 기반한 답변을 만듭니다. 또한 Thought가 기록되므로 모델이 왜 그런 행동을 했는지를 사후에 추적할 수 있습니다. 이 투명성은 LLMOps의 모니터링과 평가 측면에서도 유용합니다. ReAct 패턴은 이후 5단원에서 다루는 에이전트 아키텍처의 핵심 토대가 됩니다.

다음으로 다루는 기법은 모델의 출력 형식에 관한 것입니다. LLM의 기본 출력은 자연어 텍스트입니다. 사람이 읽기에는 편하지만, 프로그램이 처리하기에는 까다롭습니다. 예를 들어, 모델에게 "이 리뷰의 감성을 분석하라"고 요청했을 때 "이 리뷰는 긍정적인 편입니다. 다만 배송에 대한 불만이 약간 있습니다."라고 답하면, 후속 코드에서 "긍정"인지 "부정"인지를 뽑아내기 위해 별도의 파싱 로직을 작성해야 합니다. 응답 형식이 매번 조금씩 달라질 수 있으므로 파싱이 깨지기도 합니다.

**Structured Output**은 모델이 JSON, XML 같은 구조화된 형식으로 출력하도록 강제하는 기법입니다. 많은 LLM API 제공자가 이 기능을 **JSON Mode**라는 이름으로 제공합니다. JSON Mode를 활성화하면 모델의 출력이 반드시 유효한 JSON 형태를 갖추도록 보장됩니다. 더 나아가 JSON Schema를 지정하면, 출력이 정해진 필드 이름과 데이터 타입을 따르도록 제약할 수 있습니다.

구체적으로 감성 분석 작업을 Structured Output으로 처리하는 예를 보겠습니다.

```
프롬프트에 포함할 JSON Schema 지정

{
  "type": "object",
  "properties": {
    "sentiment": {
      "type": "string",
      "enum": ["positive", "negative", "neutral"]
    },
    "confidence": {
      "type": "number",
      "minimum": 0,
      "maximum": 1
    },
    "key_phrases": {
      "type": "array",
      "items": { "type": "string" }
    }
  },
  "required": ["sentiment", "confidence", "key_phrases"]
}
```

```json
// 모델의 출력 예시
{
  "sentiment": "positive",
  "confidence": 0.85,
  "key_phrases": ["빠른 배송", "좋은 품질", "재구매 의향"]
}
```

위 예시에서 프롬프트와 함께 JSON Schema를 전달하면, 모델은 sentiment 필드에 "positive", "negative", "neutral" 중 하나만 넣고, confidence에는 0에서 1 사이의 숫자를 넣으며, key_phrases에는 문자열 배열을 넣습니다. 후속 코드에서는 별도의 파싱 없이 JSON 객체의 필드에 바로 접근할 수 있습니다. Structured Output은 LLM을 소프트웨어 파이프라인의 한 구성 요소로 통합할 때 필수적인 기법입니다. 모델의 출력이 예측 가능한 형식을 갖추어야 후속 단계에서 안정적으로 처리할 수 있기 때문입니다.

지금까지 살펴본 기법들은 각각 하나의 프롬프트 호출 안에서 모델의 성능을 높이는 데 초점을 맞추었습니다. 그런데 실무의 복잡한 작업은 한 번의 프롬프트 호출로 해결되지 않는 경우가 많습니다. 긴 문서를 요약하고, 요약 결과에서 핵심 주제를 추출하고, 각 주제에 대한 실행 계획을 만드는 작업을 생각해 보겠습니다. 이것을 하나의 프롬프트에 모두 넣으면 지시가 복잡해지고, 모델이 일부 단계를 건너뛰거나 혼동할 위험이 커집니다.

**프롬프트 체이닝(Prompt Chaining)**은 하나의 큰 작업을 여러 개의 작은 단계로 나누고, 각 단계를 별도의 프롬프트로 처리한 뒤, 앞 단계의 출력을 다음 단계의 입력으로 연결하는 기법입니다. 위의 문서 분석 작업을 체이닝으로 설계하면 다음과 같습니다.

```
프롬프트 체이닝 예시: 문서 분석 파이프라인

[원본 문서]
     |
     v
+--------------------+
| 프롬프트 1: 요약   |  "이 문서를 500자 이내로 요약하라"
+--------------------+
     |
     v
[요약 결과]
     |
     v
+--------------------+
| 프롬프트 2: 주제   |  "이 요약에서 핵심 주제 3개를 추출하라"
| 추출               |
+--------------------+
     |
     v
[주제 목록]
     |
     v
+--------------------+
| 프롬프트 3: 계획   |  "각 주제에 대한 실행 계획을 작성하라"
| 수립               |
+--------------------+
     |
     v
[최종 실행 계획]
```

체이닝의 장점은 여러 가지입니다. 첫째, 각 단계의 프롬프트가 단순해지므로 모델이 지시를 정확히 따를 확률이 높아집니다. 둘째, 중간 단계의 결과를 검증할 수 있습니다. 요약이 적절한지 확인한 뒤에 다음 단계로 넘어가면, 잘못된 요약이 후속 단계에 전파되는 것을 막을 수 있습니다. 셋째, 각 단계에서 서로 다른 모델이나 설정을 사용할 수 있습니다. 요약에는 비용이 낮은 모델을 쓰고, 계획 수립에는 성능이 높은 모델을 쓰는 식입니다. 넷째, Structured Output과 결합하면 단계 간 데이터 전달이 안정적이 됩니다. 앞 단계의 출력을 JSON으로 받고, 그 JSON을 다음 단계의 프롬프트에 삽입하면 파싱 오류 없이 연결됩니다.

프롬프트 체이닝에서 주의할 점도 있습니다. 체인이 길어지면 전체 지연 시간이 각 단계의 지연 시간을 합한 것이 됩니다. 또한 앞 단계의 오류가 뒤 단계로 전파될 수 있으므로, 각 단계 사이에 검증 로직을 넣는 것이 권장됩니다. 예를 들어, 요약 단계의 출력 길이가 지나치게 짧거나 입력과 무관한 내용이면 재실행하거나 오류를 반환하는 조건을 두는 것입니다.

마지막으로 다루는 기법은 **메타 프롬프트**와 **자기 반성(Self-Reflection)**입니다. 메타 프롬프트란 "프롬프트를 만드는 프롬프트"를 뜻합니다. 일반적인 프롬프트는 모델에게 특정 작업을 지시합니다. "이 텍스트를 번역하라", "이 코드의 버그를 찾아라" 같은 직접적인 지시입니다. 메타 프롬프트는 한 단계 위에서, 모델에게 "이 작업을 수행하기 위한 최적의 프롬프트를 생성하라"고 지시합니다.

메타 프롬프트가 필요한 상황을 생각해 보겠습니다. 고객 문의를 분류하는 프롬프트를 만들어야 하는데, 문의 유형이 20가지이고 각 유형마다 미묘한 차이가 있습니다. 사람이 직접 20가지 유형의 구분 기준을 하나하나 프롬프트에 작성하는 대신, 모델에게 "다음 문의 예시들을 보고, 각 유형을 정확히 분류할 수 있는 프롬프트를 작성하라"고 지시할 수 있습니다. 모델이 예시를 분석하여 분류 기준과 프롬프트 구조를 스스로 설계하는 것입니다.

자기 반성(Self-Reflection)은 모델이 자신의 출력을 되돌아보고 개선하는 기법입니다. 모델이 첫 번째 응답을 생성한 뒤, 두 번째 프롬프트에서 "방금 네가 생성한 답변을 검토하라. 논리적 오류, 누락된 정보, 개선할 점이 있으면 수정하라"고 지시합니다. 모델은 자신의 출력을 비판적으로 분석하고, 문제를 발견하면 수정된 답변을 내놓습니다.

자기 반성의 동작을 단계별로 보면 다음과 같습니다.

```
자기 반성(Self-Reflection) 흐름

[원래 질문]
     |
     v
+--------------------------+
| 1단계: 초안 생성         |  모델이 첫 번째 답변을 생성
+--------------------------+
     |
     v
[초안 답변]
     |
     v
+--------------------------+
| 2단계: 반성 프롬프트     |  "위 답변을 검토하라.
|                          |   사실 오류, 논리 비약,
|                          |   누락 정보가 있는가?"
+--------------------------+
     |
     v
[비판/개선점 목록]
     |
     v
+--------------------------+
| 3단계: 수정된 답변 생성  |  비판을 반영하여 최종 답변 생성
+--------------------------+
     |
     v
[최종 답변]
```

자기 반성은 프롬프트 체이닝의 특수한 형태로 볼 수 있습니다. 초안 생성, 비판, 수정이라는 세 단계의 체인입니다. 이 기법이 효과적인 이유는, 모델이 답변을 생성할 때와 검토할 때 서로 다른 "관점"을 취하기 때문입니다. 생성 단계에서는 정보를 종합하여 흐름을 만드는 데 집중하지만, 검토 단계에서는 논리적 일관성과 사실 정확성을 점검하는 데 집중합니다. 하나의 프롬프트에서 두 가지를 동시에 수행하는 것보다, 역할을 분리했을 때 각각의 품질이 높아지는 것입니다.

메타 프롬프트와 자기 반성은 결합하여 사용할 수도 있습니다. 메타 프롬프트로 프롬프트 초안을 생성하고, 자기 반성으로 그 프롬프트의 문제점을 점검한 뒤 개선하는 방식입니다. 이 반복 과정을 여러 차례 수행하면 프롬프트의 품질이 점진적으로 향상됩니다.

지금까지 다룬 여섯 가지 기법을 상황별로 어떻게 선택할지 정리하겠습니다. 정답이 명확하고 신뢰도를 높이고 싶다면 Self-Consistency가 적합합니다. 탐색과 되돌아가기가 필요한 복잡한 추론 문제에는 ToT가 효과적입니다. 외부 데이터 조회나 도구 사용이 필요하면 ReAct 패턴을 적용합니다. 모델의 출력을 프로그램이 안정적으로 처리해야 하면 Structured Output을 사용합니다. 작업이 여러 단계로 나뉘면 프롬프트 체이닝으로 분할합니다. 답변의 품질을 한 단계 더 끌어올리고 싶으면 자기 반성을 추가합니다. 이 기법들은 상호 배타적이지 않습니다. 프롬프트 체이닝의 각 단계에 Structured Output을 적용하고, 특정 단계에서 ReAct를 사용하며, 최종 단계에서 자기 반성을 추가하는 식으로 조합할 수 있습니다.

정리하면, 고급 프롬프트 기법은 기본 기법의 한계를 보완하기 위해 다양한 전략을 도입합니다. Self-Consistency는 다수결로 답의 신뢰도를 높이고, ToT는 체계적 탐색으로 복잡한 추론을 지원하며, ReAct는 외부 도구와의 상호작용을 가능하게 합니다. Structured Output은 모델 출력을 프로그래밍적으로 처리할 수 있게 만들고, 프롬프트 체이닝은 복잡한 작업을 단계별로 분할합니다. 메타 프롬프트와 자기 반성은 프롬프트 자체의 품질과 답변의 정확성을 반복적으로 개선합니다.
