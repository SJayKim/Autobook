# 7.1.3 QLoRA

7.1.2에서 LoRA가 사전 학습된 가중치를 고정한 채 두 개의 작은 행렬(B, A)만 학습하여 파인튜닝 비용을 줄이는 방법을 살펴보았습니다. LoRA 덕분에 학습 파라미터 수는 전체의 1% 미만으로 줄어들었지만, 원래 모델 가중치 자체는 여전히 GPU 메모리에 올려야 합니다. 70억 파라미터 모델은 16비트(FP16) 정밀도 기준으로 약 14GB의 메모리를 차지합니다. LoRA 어댑터가 아무리 작아도, 기본 모델을 올릴 메모리가 부족하면 파인튜닝 자체를 시작할 수 없습니다. 소비자용 GPU가 대부분 16GB에서 24GB 수준인 현실을 고려하면, 70억 파라미터 모델도 단일 GPU에서 LoRA 파인튜닝을 수행하기가 빠듯합니다. 130억 파라미터 이상의 모델은 사실상 불가능합니다.

**QLoRA**(Quantized LoRA)는 이 메모리 문제를 정면으로 해결하기 위해 등장한 기법입니다. 핵심 아이디어는 기본 모델의 가중치를 4비트로 양자화하여 메모리 사용량을 대폭 줄인 뒤, 그 위에 LoRA 어댑터를 얹어 학습하는 것입니다. 양자화란 높은 정밀도의 숫자를 낮은 정밀도로 변환하여 저장 공간을 줄이는 기법을 뜻합니다. 32비트 부동소수점(FP32)을 16비트(FP16)로 바꾸면 메모리가 절반으로 줄고, 16비트를 4비트로 바꾸면 다시 4분의 1로 줄어듭니다. QLoRA는 이 4비트 양자화를 도입하여, LoRA 단독 대비 메모리 사용량을 약 4분의 1 수준으로 추가 절감합니다.

QLoRA를 가능하게 만드는 핵심 기법은 세 가지입니다. 4-bit NormalFloat 데이터 타입, 이중 양자화, 그리고 페이지드 옵티마이저입니다. 이 세 기법이 어떻게 조합되는지를 하나씩 살펴보겠습니다.

첫 번째 핵심 기법은 **4-bit NormalFloat(NF4)** 데이터 타입입니다. 양자화에서 가장 중요한 문제는 "4비트로 줄였을 때 정보 손실을 얼마나 줄일 수 있느냐"입니다. 4비트로 표현할 수 있는 값은 2의 4승, 즉 16개뿐입니다. 원래 수십억 가지 값을 표현할 수 있던 32비트 숫자를 16개의 대표값 중 하나로 매핑해야 합니다. 이때 대표값을 어떻게 선정하느냐에 따라 정보 손실이 크게 달라집니다.

기존 4비트 양자화 방식은 대표값을 균등 간격으로 배치했습니다. 예를 들어 가중치의 최솟값이 -1이고 최댓값이 1이면, -1부터 1까지를 16등분하여 대표값을 놓는 식입니다. 그런데 이 방식에는 문제가 있습니다. 신경망의 가중치 분포는 균등하지 않습니다. 대부분의 가중치는 0 근처에 밀집되어 있고, 극단적으로 크거나 작은 값은 드뭅니다. 이 분포는 정규분포(가우시안 분포)에 가깝습니다. 균등 간격 양자화는 0 근처의 밀집 구간에 대표값이 부족하고, 드문 극단값 구간에 대표값을 낭비하는 셈입니다.

NF4는 이 문제를 해결합니다. 16개의 대표값을 균등 간격이 아니라, 정규분포의 분위수(quantile) 위치에 배치합니다. 분위수란 데이터를 크기 순으로 나열했을 때 전체를 동일한 비율로 나누는 경계값을 뜻합니다. 16개의 대표값을 정규분포의 분위수에 맞추면, 가중치가 밀집된 0 근처에는 대표값이 촘촘하게 배치되고, 가중치가 드문 극단 영역에는 대표값이 듬성듬성 배치됩니다. 이렇게 하면 실제 가중치와 가장 가까운 대표값 사이의 거리가 전체적으로 줄어들어, 같은 4비트 제약 안에서 정보 손실을 최소화합니다.

이 과정을 시각적으로 표현하면 다음과 같습니다.

```
가중치 분포 (정규분포 형태)

                  ***
                **   **
              **       **
            **           **
          **               **
  --------+--+--+--+--+--+--+--------
  -3     -2  -1  0  +1  +2  +3

균등 간격 양자화 (대표값 16개):
  |   |   |   |   |   |   |   |   |   |   |   |   |   |   |   |
  <-- 극단에도 동일 간격 --> 0 근처에 대표값 부족

NF4 양자화 (대표값 16개):
  |     |   | | ||| ||| | |   |     |
  <-- 극단은 듬성듬성 --> 0 근처에 대표값 촘촘
  정규분포 분위수 기반 배치 --> 정보 손실 최소화
```

NF4가 정보 이론적으로 최적인 이유를 좀 더 구체적으로 설명하겠습니다. 양자화의 목표는 원래 값과 양자화된 값 사이의 오차(양자화 오차)를 최소화하는 것입니다. 데이터가 정규분포를 따를 때, 분위수 기반으로 대표값을 배치하면 각 대표값이 담당하는 구간에 같은 수의 데이터 포인트가 들어갑니다. 데이터가 많은 곳에 대표값을 촘촘하게 배치한 셈이므로, 전체 양자화 오차의 기대값이 최소가 됩니다. 신경망 가중치가 정규분포에 가까운 분포를 따른다는 실험적 관찰과 결합하면, NF4는 신경망 가중치를 4비트로 양자화할 때 이론적으로 최적에 가까운 데이터 타입입니다.

두 번째 핵심 기법은 **이중 양자화(Double Quantization)**입니다. 양자화를 수행할 때는 대표값 외에 추가 정보가 필요합니다. 구체적으로, 양자화된 값을 원래 스케일로 복원하려면 양자화 상수(quantization constant)가 있어야 합니다. 양자화 상수란 양자화 전 값의 범위를 나타내는 스케일링 값으로, 보통 FP32(32비트)로 저장합니다. 가중치를 일정 크기의 블록 단위로 묶어서 양자화하는데, 각 블록마다 하나의 양자화 상수가 필요합니다. 블록 크기가 64일 때, 64개의 4비트 가중치마다 32비트 양자화 상수 하나가 추가됩니다. 이는 가중치당 0.5비트의 추가 메모리에 해당합니다.

이중 양자화는 이 양자화 상수 자체를 한 번 더 양자화하는 기법입니다. 첫 번째 양자화에서 생긴 FP32 양자화 상수들을 모아서, 이것들을 다시 8비트 정수(INT8)로 양자화합니다. 32비트였던 양자화 상수가 8비트로 줄어들므로, 가중치당 추가 메모리가 0.5비트에서 약 0.127비트로 감소합니다. 이 절감분은 개별적으로는 작아 보이지만, 수십억 개의 파라미터에 적용하면 수백 MB의 메모리를 아낄 수 있습니다.

이중 양자화의 과정을 단계별로 정리하면 다음과 같습니다.

```
이중 양자화 절차

1단계: 가중치 양자화
  원래 가중치 (FP16) --> 4비트 NF4 양자화
  블록 크기 64 --> 64개 가중치마다 양자화 상수 1개 (FP32) 생성

2단계: 양자화 상수 양자화
  1단계에서 생성된 FP32 양자화 상수들을 모은다
  이 양자화 상수들을 다시 8비트 (INT8)로 양자화한다
  2단계 양자화 상수도 생성되지만, 수가 훨씬 적어 오버헤드 미미

메모리 절감 효과 (가중치당):
  양자화 상수 없이:  4비트 = 0.5바이트
  1차 양자화 상수:  +0.5비트 (FP32 / 블록크기64)
  이중 양자화 후:   +0.127비트 (INT8 / 블록크기64)
  절감:             가중치당 약 0.373비트
```

세 번째 핵심 기법은 **페이지드 옵티마이저(Paged Optimizer)**입니다. 이 기법은 GPU 메모리 부족 상황을 처리하는 안전장치입니다. 파인튜닝 중에는 모델 가중치 외에 옵티마이저 상태도 메모리에 올라갑니다. 7.1.2에서 LoRA가 B와 A 행렬만 학습한다고 했지만, 이 작은 행렬에 대해서도 Adam 옵티마이저는 모멘텀(momentum)과 분산(variance) 두 가지 상태를 유지합니다. 학습 가능한 파라미터의 3배에 해당하는 메모리가 옵티마이저 상태에 필요한 셈입니다.

문제는 학습 중에 메모리 사용량이 일정하지 않다는 것입니다. 배치 크기가 크거나 시퀀스 길이가 긴 입력이 들어오면 일시적으로 메모리 사용량이 급증합니다. 이때 GPU 메모리가 부족하면 학습이 중단됩니다. 페이지드 옵티마이저는 NVIDIA의 통합 메모리(Unified Memory) 기능을 활용하여, GPU 메모리가 부족할 때 옵티마이저 상태를 자동으로 CPU RAM으로 이동시킵니다. CPU RAM은 GPU 메모리보다 훨씬 크므로, 일시적인 메모리 급증을 흡수할 수 있습니다. 메모리 여유가 생기면 다시 GPU 메모리로 가져옵니다.

운영체제의 가상 메모리가 디스크를 사용하여 물리 메모리 부족을 해결하는 것과 같은 원리입니다. "페이지드"라는 이름도 운영체제의 페이징 기법에서 유래했습니다. CPU RAM으로의 이동은 GPU 메모리 내 접근보다 느리지만, 학습이 중단되는 것보다는 낫습니다. 실제로 메모리 급증은 간헐적으로 발생하므로, 전체 학습 시간에 미치는 영향은 크지 않습니다.

이제 세 가지 기법이 QLoRA 안에서 어떻게 결합되는지 전체 흐름을 보겠습니다.

```
QLoRA 전체 동작 흐름

1. 사전 학습된 모델 가중치를 NF4 4비트로 양자화한다
   - 70억 파라미터: 약 14GB (FP16) --> 약 3.5GB (NF4 4비트)

2. 양자화 상수를 이중 양자화로 추가 압축한다
   - 양자화 상수 오버헤드: 약 0.373비트 추가 절감

3. 양자화된 모델 위에 LoRA 어댑터(B, A 행렬)를 삽입한다
   - LoRA 어댑터는 BFloat16 정밀도로 학습

4. 순전파:
   NF4 가중치 --> BFloat16으로 역양자화 --> 연산 수행
   + LoRA 경로: BFloat16 B * A * x
   두 결과를 합산하여 출력

5. 역전파:
   그래디언트는 LoRA의 B, A에 대해서만 계산
   기본 모델 가중치는 고정 (그래디언트 계산 안 함)

6. 옵티마이저 갱신:
   B, A 파라미터를 갱신
   메모리 부족 시 페이지드 옵티마이저가 상태를 CPU로 이동
```

4단계에서 NF4 가중치를 BFloat16으로 "역양자화"한다는 부분이 중요합니다. NF4는 저장 형식이지, 연산 형식은 아닙니다. 실제 행렬 곱셈은 BFloat16 정밀도에서 수행됩니다. 역양자화란 4비트 대표값과 양자화 상수를 사용하여 원래에 가까운 16비트 값을 복원하는 과정입니다. 이 복원은 완벽하지 않아 약간의 오차가 발생하지만, NF4의 분위수 기반 배치 덕분에 오차가 최소화됩니다. 역양자화는 연산이 필요할 때마다 수행되고, 연산이 끝나면 다시 4비트 상태로 돌아갑니다. 메모리에는 4비트만 상주하고, 연산 시에만 잠깐 16비트로 확장되는 것입니다.

이제 **QLoRA와 LoRA의 성능 비교**를 살펴보겠습니다. 메모리를 크게 줄였으니 성능도 크게 떨어질 것이라 생각할 수 있지만, 실험 결과는 다릅니다. QLoRA 논문에서 다양한 모델과 데이터셋에 걸쳐 실험한 결과, QLoRA로 파인튜닝한 모델의 성능은 16비트 LoRA로 파인튜닝한 모델의 성능과 거의 동등했습니다. 일부 벤치마크에서는 QLoRA가 LoRA와 통계적으로 유의미한 차이가 없는 성능을 보였습니다.

이것이 가능한 이유는 LoRA의 특성에 있습니다. 학습되는 것은 B와 A 행렬이고, 이 행렬들은 BFloat16 정밀도를 그대로 유지합니다. 기본 모델 가중치의 4비트 양자화로 인한 오차는 존재하지만, 이 오차는 LoRA 어댑터가 학습 과정에서 보상할 수 있는 범위 안에 있습니다. 결과적으로 QLoRA는 메모리 효율과 학습 품질 사이의 균형을 잘 잡은 기법입니다.

메모리 사용량과 성능을 비교하면 다음과 같습니다.

```
QLoRA vs LoRA 비교 (70억 파라미터 모델 기준)

+--------------------+------------+------------+
| 항목               | LoRA       | QLoRA      |
+--------------------+------------+------------+
| 모델 정밀도        | FP16       | NF4 (4bit) |
| 모델 메모리        | ~14 GB     | ~3.5 GB    |
| 어댑터 정밀도      | FP16       | BF16       |
| 어댑터 메모리      | ~수십 MB   | ~수십 MB   |
| 옵티마이저 상태    | GPU 고정   | 페이지드   |
| 총 GPU 메모리      | ~18 GB     | ~6 GB      |
| 학습 성능          | 기준선     | 기준선과   |
|                    |            | 거의 동등  |
| 학습 속도          | 기준선     | 약간 느림  |
+--------------------+------------+------------+
```

QLoRA의 학습 속도가 약간 느린 이유는 역양자화 연산이 추가되기 때문입니다. 매 순전파마다 NF4를 BFloat16으로 변환하는 과정이 들어가므로, 순수 연산량은 LoRA보다 많습니다. 그러나 이 추가 연산 비용은 전체 학습 시간의 수 퍼센트 수준이어서 실용적으로 큰 문제가 되지 않습니다.

마지막으로 **QLoRA 실전 설정**을 살펴보겠습니다. QLoRA는 bitsandbytes라는 파이썬 라이브러리를 통해 구현됩니다. bitsandbytes는 NVIDIA GPU에서 4비트 및 8비트 양자화를 지원하는 라이브러리로, Hugging Face Transformers와 통합되어 있습니다.

QLoRA를 적용하는 코드의 핵심 부분을 보겠습니다.

```python
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training
import torch

# 1. 4비트 양자화 설정
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_use_double_quant=True,
    bnb_4bit_compute_dtype=torch.bfloat16,
)

# 2. 양자화된 모델 로드
model = AutoModelForCausalLM.from_pretrained(
    "모델 이름",
    quantization_config=bnb_config,
    device_map="auto",
)

# 3. k-bit 학습을 위한 모델 준비
model = prepare_model_for_kbit_training(model)

# 4. LoRA 설정
lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)

# 5. PEFT 모델 생성
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
```

이 코드를 단계별로 풀어 보겠습니다. 1단계에서 BitsAndBytesConfig를 생성합니다. load_in_4bit=True는 모델을 4비트로 로드하겠다는 설정입니다. bnb_4bit_quant_type="nf4"는 앞서 설명한 NF4 데이터 타입을 사용하겠다는 뜻입니다. bnb_4bit_use_double_quant=True는 이중 양자화를 활성화합니다. bnb_4bit_compute_dtype=torch.bfloat16은 연산 시 역양자화할 때 사용할 데이터 타입을 BFloat16으로 지정합니다.

2단계에서 from_pretrained에 quantization_config를 전달하면, 모델이 로드되면서 자동으로 4비트 양자화가 적용됩니다. device_map="auto"는 모델을 사용 가능한 GPU에 자동으로 배치합니다.

3단계의 prepare_model_for_kbit_training은 양자화된 모델을 학습 가능한 상태로 준비하는 함수입니다. 양자화된 가중치의 그래디언트 계산을 비활성화하고, 입력 임베딩의 정규화 층을 FP32로 유지하는 등의 설정을 수행합니다. 이 단계를 건너뛰면 학습이 불안정해질 수 있습니다.

4단계에서 LoRA 설정을 생성합니다. 이 부분은 7.1.2에서 다룬 LoRA 설정과 동일합니다. QLoRA에서 달라지는 것은 기본 모델의 양자화 방식이지, LoRA 어댑터의 설정 자체는 같습니다.

5단계에서 get_peft_model이 양자화된 모델 위에 LoRA 어댑터를 삽입합니다. print_trainable_parameters를 호출하면 전체 파라미터 중 학습 가능한 파라미터의 비율을 확인할 수 있습니다.

페이지드 옵티마이저를 활성화하려면 학습 설정에서 옵티마이저 종류를 지정합니다.

```python
from transformers import TrainingArguments

training_args = TrainingArguments(
    output_dir="./output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    learning_rate=2e-4,
    optim="paged_adamw_8bit",
    save_strategy="epoch",
    logging_steps=10,
    bf16=True,
)
```

optim="paged_adamw_8bit"이 페이지드 옵티마이저를 활성화하는 설정입니다. AdamW 옵티마이저의 상태를 8비트로 양자화하면서, 메모리 부족 시 CPU로 페이징하는 기능을 함께 켭니다. bf16=True는 학습 과정에서 BFloat16 혼합 정밀도를 사용하겠다는 설정입니다.

정리하면, QLoRA는 사전 학습된 모델 가중치를 NF4 4비트로 양자화하고, 이중 양자화로 양자화 상수의 오버헤드를 추가 절감하며, 페이지드 옵티마이저로 메모리 급증에 대응하면서, 그 위에 LoRA 어댑터를 BFloat16 정밀도로 학습하는 기법입니다. 70억 파라미터 모델 기준으로 GPU 메모리 사용량을 약 18GB에서 약 6GB로 줄이면서도, 16비트 LoRA와 거의 동등한 학습 성능을 유지합니다. bitsandbytes 라이브러리와 Hugging Face PEFT의 통합 덕분에, 설정 몇 줄을 추가하는 것만으로 QLoRA를 적용할 수 있습니다.

다음 단원인 7.1.4에서는 LoRA, QLoRA를 포함한 다양한 파라미터 효율적 파인튜닝 기법을 통합 관리하는 PEFT 프레임워크를 다룹니다.

이 단원을 마치면 QLoRA의 동작 원리를 설명하고 제한된 GPU 환경에서 파인튜닝을 수행할 수 있습니다.
