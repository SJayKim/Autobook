# 6.2.3 KV Cache 관리와 최적화

대규모 언어 모델을 서빙할 때, 응답 속도를 좌우하는 요소 중 하나는 GPU 메모리를 얼마나 효율적으로 사용하느냐입니다. 1.2.3에서 살펴본 것처럼 모델은 자기회귀 방식으로 토큰을 하나씩 생성하며, 매 단계마다 이전 토큰들의 Key와 Value 벡터를 다시 참조해야 합니다. 이 벡터들을 매번 처음부터 재계산하면 같은 연산을 수십, 수백 번 반복하게 되므로, 이미 계산한 결과를 메모리에 보관해 두는 것이 필수적입니다. 이 보관소가 바로 **KV Cache**입니다. 6.2.1에서 GPU 메모리 구조와 연산 병목을, 6.2.2에서 배칭 전략을 다루었는데, KV Cache는 이 두 주제 모두와 깊이 맞닿아 있습니다. 배칭으로 여러 요청을 동시에 처리하려 할수록 KV Cache가 차지하는 메모리가 늘어나고, GPU 메모리가 부족해지면 배치 크기를 줄여야 하므로 처리량이 떨어집니다. 이 단원에서는 KV Cache가 정확히 무엇이고, 메모리를 얼마나 소비하며, 이를 효율적으로 관리하고 압축하는 기법에는 어떤 것이 있는지를 단계별로 살펴봅니다.

KV Cache의 역할을 이해하려면 먼저 Transformer의 Attention 연산을 되짚어야 합니다. 1.2.1에서 다룬 것처럼, Self-Attention은 각 토큰의 Query 벡터와 문장 안 모든 토큰의 Key 벡터를 비교하여 어텐션 가중치를 구하고, 그 가중치를 Value 벡터에 곱해 문맥이 반영된 새 표현을 만듭니다. Decoder-only 구조에서는 인과적 마스킹이 적용되어, 각 토큰은 자기 자신과 그 앞에 위치한 토큰들만 참조할 수 있습니다.

1.2.3에서 설명한 Prefill 단계에서는 사용자가 입력한 프롬프트의 모든 토큰을 한꺼번에 Transformer 층에 통과시킵니다. 이때 각 토큰의 Key 벡터와 Value 벡터가 계산되고, 이 결과가 GPU 메모리에 저장됩니다. 이것이 KV Cache의 초기 상태입니다. 이후 Decode 단계에서 토큰을 하나씩 생성할 때, 모델은 새로 생성한 토큰의 Query 벡터만 계산하면 됩니다. 이전 토큰들의 Key와 Value는 이미 KV Cache에 들어 있으므로 다시 계산할 필요가 없습니다. 새 토큰의 Query가 캐시에 저장된 모든 Key와 내적을 수행하여 어텐션 가중치를 구하고, 그 가중치를 캐시의 Value에 적용합니다. 그런 다음 새 토큰 자신의 Key와 Value를 캐시에 추가합니다. 이 과정이 매 토큰 생성마다 반복됩니다.

```
Decode 단계에서 n번째 토큰 생성 시:

1. 새 토큰의 Query(Q_n) 계산
2. KV Cache에서 Key_1, Key_2, ..., Key_{n-1} 읽기
3. Q_n과 모든 Key의 내적 --> 어텐션 가중치
4. 어텐션 가중치를 Value_1, Value_2, ..., Value_{n-1}에 적용
5. 새 토큰의 Key_n, Value_n을 KV Cache에 추가
6. 다음 토큰의 로짓 출력
```

만약 KV Cache가 없다면 어떻게 될까요. n번째 토큰을 생성할 때 1번부터 n-1번까지 모든 토큰의 Key와 Value를 매번 처음부터 다시 계산해야 합니다. 100번째 토큰을 만들 때는 99개 토큰을, 200번째 토큰을 만들 때는 199개 토큰을 매번 Transformer 전체 층에 통과시켜야 하므로, 생성이 진행될수록 연산량이 급격히 늘어납니다. KV Cache는 이 중복 연산을 제거하여, Decode 단계에서 매번 새 토큰 한 개만 처리하면 되도록 만들어 줍니다. 대규모 언어 모델 서빙에서 KV Cache는 선택이 아니라 사실상 필수입니다.

KV Cache가 차지하는 메모리를 구체적으로 계산해 보겠습니다. KV Cache의 크기를 결정하는 요소는 네 가지입니다.

첫째, Transformer **층의 수**(L)입니다. 각 층마다 독립적인 Key와 Value가 존재하므로, 층이 많을수록 캐시도 커집니다. 둘째, **어텐션 헤드의 수**(H)입니다. 1.2.1에서 다룬 Multi-Head Attention에서 각 헤드가 자체 Key와 Value를 갖습니다.

셋째, **헤드 차원**(d_head)입니다. 각 헤드가 처리하는 벡터의 크기입니다. 넷째, **시퀀스 길이**(S)입니다. 지금까지 처리한 토큰의 총 개수로, 프롬프트 길이와 이미 생성한 토큰 수를 합한 값입니다.

한 토큰이 한 층에서 차지하는 캐시 크기는 Key 벡터 하나와 Value 벡터 하나, 즉 2개의 벡터입니다. 각 벡터는 H개 헤드에 걸쳐 d_head 차원을 갖습니다. 이를 수식으로 정리하면 다음과 같습니다.

```
KV Cache 메모리 = 2 x L x H x d_head x S x (데이터 타입 바이트 수)
```

여기서 2는 Key와 Value 두 종류를 의미합니다. H x d_head는 모델의 히든 차원(d_model)과 같습니다. 따라서 더 간단하게 쓸 수 있습니다.

```
KV Cache 메모리 = 2 x L x d_model x S x (바이트 수)
```

실제 모델에 이 공식을 적용해 보겠습니다. LLaMA 2 7B 모델을 예로 들면, 층 수(L)는 32, 히든 차원(d_model)은 4096, FP16(2바이트) 정밀도를 사용한다고 가정합니다.

```
모델: LLaMA 2 7B
L = 32, d_model = 4096, 데이터 타입 = FP16 (2바이트)

시퀀스 길이 1,024 토큰일 때:
  2 x 32 x 4096 x 1024 x 2 바이트
  = 536,870,912 바이트
  = 약 512 MB (요청 1건당)

시퀀스 길이 4,096 토큰일 때:
  2 x 32 x 4096 x 4096 x 2 바이트
  = 2,147,483,648 바이트
  = 약 2 GB (요청 1건당)
```

7B 모델에서 시퀀스 길이가 4,096 토큰이면 요청 한 건의 KV Cache만으로 약 2 GB를 소비합니다. 모델 가중치 자체가 FP16 기준 약 14 GB를 차지하므로, 80 GB 메모리를 가진 GPU에서 모델을 올린 뒤 남는 공간은 약 66 GB입니다. 요청 한 건당 2 GB씩이라면 동시에 처리할 수 있는 요청 수는 최대 약 33건에 그칩니다. 모델 크기가 더 크거나 시퀀스가 더 길면 이 수치는 급격히 줄어듭니다.

```
모델: LLaMA 2 70B
L = 80, d_model = 8192, 데이터 타입 = FP16 (2바이트)

시퀀스 길이 4,096 토큰일 때:
  2 x 80 x 8192 x 4096 x 2 바이트
  = 약 10.7 GB (요청 1건당)
```

70B 모델에서는 요청 한 건의 KV Cache가 약 10.7 GB에 달합니다. 이 정도 크기가 되면 동시 요청 수가 극도로 제한되어, 아무리 좋은 배칭 전략을 적용하더라도 GPU 메모리가 병목이 됩니다. 이것이 KV Cache 관리가 서빙 최적화에서 핵심적인 이유입니다.

KV Cache의 메모리 문제를 더 악화시키는 요인이 있습니다. 전통적인 KV Cache 구현에서는 각 요청에 대해 컨텍스트 윈도우의 최대 길이만큼 메모리를 미리 할당합니다. 컨텍스트 윈도우가 4,096 토큰인 모델에서, 실제로 500 토큰만 사용하는 요청이 들어와도 4,096 토큰 전체에 해당하는 메모리가 예약됩니다. 나머지 3,596 토큰분의 메모리는 비어 있지만 다른 요청이 사용할 수 없습니다. 이런 낭비를 **내부 단편화(internal fragmentation)**라고 부릅니다.

또한 요청마다 연속된 메모리 공간을 확보해야 하므로, 전체 GPU 메모리에 빈 공간이 충분히 남아 있어도 연속된 구간이 없으면 새 요청을 받아들이지 못하는 상황이 생깁니다. 이것을 **외부 단편화(external fragmentation)**라고 합니다. 내부 단편화와 외부 단편화가 함께 발생하면, 실제로 사용 가능한 GPU 메모리의 상당 부분이 낭비됩니다.

**PagedAttention**은 이 단편화 문제를 해결하기 위해 고안된 KV Cache 관리 기법입니다. 이 기법의 핵심 아이디어는 운영체제의 가상 메모리 관리에서 가져왔습니다. 운영체제는 프로그램에 연속된 메모리 공간을 제공하는 것처럼 보이지만, 실제 물리 메모리에서는 고정 크기의 작은 조각인 '페이지'로 나누어 흩어진 위치에 저장합니다. PagedAttention은 KV Cache에 같은 원리를 적용합니다.

PagedAttention의 동작을 단계별로 살펴보겠습니다. 먼저, GPU 메모리를 고정 크기의 **블록(block)**으로 나눕니다. 각 블록은 정해진 수의 토큰에 해당하는 Key와 Value를 저장할 수 있습니다. 예를 들어 블록 크기가 16이면, 한 블록에 16개 토큰의 KV 데이터가 들어갑니다.

요청이 들어오면, 필요한 만큼의 블록만 할당합니다. 500 토큰이 필요하면 블록 크기 16 기준으로 32개 블록(512 토큰분)만 할당하고, 나머지는 다른 요청이 사용할 수 있도록 남겨 둡니다. 최대 컨텍스트 길이만큼 미리 예약하지 않으므로 내부 단편화가 크게 줄어듭니다.

블록들은 물리 메모리상에서 연속될 필요가 없습니다. 각 요청은 자신에게 할당된 블록들의 위치를 기록하는 **블록 테이블(block table)**을 갖고 있어, 흩어진 블록을 논리적으로 연결합니다. 이렇게 하면 연속된 메모리 공간을 확보하지 못해 요청을 거부하는 외부 단편화 문제도 해소됩니다.

시퀀스가 길어져 추가 토큰을 저장할 공간이 필요하면, 새 블록을 동적으로 할당하여 블록 테이블에 추가합니다. 요청이 완료되면 해당 블록을 해제하여 다른 요청이 재사용할 수 있게 합니다.

```
전통적 KV Cache 할당 (연속 메모리, 최대 길이 예약):

GPU 메모리
|[요청A: 4096토큰분 예약 (실제 500토큰 사용)]|[요청B: 4096토큰분 예약]|[빈 공간]|
  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  대부분 낭비

PagedAttention (블록 단위, 필요한 만큼만):

GPU 메모리
|[A-블록1][B-블록1][A-블록2][B-블록2][B-블록3][A-블록3][빈][빈][빈]...|
  블록 테이블(A): 블록1 -> 블록2 -> 블록3
  블록 테이블(B): 블록1 -> 블록2 -> 블록3

  --> 빈 블록을 새 요청에 즉시 할당 가능
```

PagedAttention은 메모리 낭비를 마지막 블록의 빈 슬롯 정도로 줄입니다. 블록 크기가 16이고 실제 시퀀스가 500 토큰이면, 마지막 블록에서 12 토큰분(512 - 500)만 낭비됩니다. 전통 방식에서 3,596 토큰분이 낭비되던 것과 비교하면 극적인 개선입니다. 이 덕분에 같은 GPU 메모리에서 더 많은 요청을 동시에 처리할 수 있어, 서빙 처리량이 크게 향상됩니다.

**Prefix Caching**은 여러 요청이 동일한 프롬프트 접두사를 공유할 때, 해당 접두사의 KV Cache를 한 번만 계산하고 여러 요청이 함께 사용하는 기법입니다. 실제 서비스에서 이런 상황은 자주 발생합니다.

예를 들어, 시스템 프롬프트를 생각해 보겠습니다. 2.1.3에서 다룬 것처럼 시스템 프롬프트는 모델의 역할과 제약을 설정하는 지시문으로, 대부분의 요청에 동일하게 포함됩니다. 시스템 프롬프트가 500 토큰이라면, 100건의 요청이 들어올 때 동일한 500 토큰에 대한 KV Cache를 100번 따로 계산하고 저장해야 합니다. Prefix Caching을 사용하면 이 500 토큰의 KV Cache를 한 번만 계산한 뒤, 100건의 요청이 읽기 전용으로 공유합니다.

PagedAttention 구조에서 Prefix Caching은 자연스럽게 구현됩니다. 공유되는 접두사 부분의 블록을 여러 요청의 블록 테이블이 동시에 가리키도록 하면 됩니다. 각 요청은 접두사 이후의 토큰에 대해서만 자체 블록을 할당받습니다.

```
시스템 프롬프트 (500 토큰) + 사용자 메시지 (각각 다름)

Prefix Caching 없이:
  요청 A: [시스템프롬프트 KV 500토큰] + [사용자A KV 200토큰]  = 700토큰분
  요청 B: [시스템프롬프트 KV 500토큰] + [사용자B KV 300토큰]  = 800토큰분
  요청 C: [시스템프롬프트 KV 500토큰] + [사용자C KV 150토큰]  = 650토큰분
  합계: 2,150 토큰분의 KV Cache

Prefix Caching 적용:
  공유 블록: [시스템프롬프트 KV 500토큰]  (1회만 저장)
  요청 A: 공유 블록 참조 + [사용자A KV 200토큰]
  요청 B: 공유 블록 참조 + [사용자B KV 300토큰]
  요청 C: 공유 블록 참조 + [사용자C KV 150토큰]
  합계: 500 + 200 + 300 + 150 = 1,150 토큰분의 KV Cache
```

위 예시에서 Prefix Caching을 적용하면 KV Cache 메모리 사용량이 약 46% 줄어듭니다. 동일한 시스템 프롬프트를 사용하는 요청이 많을수록, 그리고 시스템 프롬프트가 길수록 절감 효과가 커집니다. 메모리를 절약하는 것 외에도, 접두사에 대한 Prefill 연산을 반복하지 않으므로 첫 토큰 생성까지의 지연 시간(Time To First Token)도 줄어듭니다.

KV Cache의 크기를 줄이는 또 다른 접근법이 **KV Cache 압축 기법**입니다. 앞서 살펴본 PagedAttention이 메모리 할당 방식을 개선하는 것이라면, 압축 기법은 저장해야 하는 데이터의 양 자체를 줄이는 것입니다.

대표적인 압축 방법 중 하나는 **GQA(Grouped-Query Attention)**입니다. 1.2.1에서 다룬 Multi-Head Attention에서, 원래는 모든 어텐션 헤드가 각자 독립적인 Key와 Value를 갖습니다. 헤드가 32개이면 Key 세트 32개, Value 세트 32개가 필요합니다. GQA는 여러 헤드가 Key와 Value를 공유하도록 그룹을 만듭니다. 예를 들어 32개 헤드를 8개 그룹으로 묶으면, 4개 헤드마다 하나의 Key-Value 쌍을 공유합니다. Key와 Value는 8세트만 있으면 되므로, KV Cache 크기가 원래의 1/4로 줄어듭니다. Query는 여전히 헤드마다 독립적으로 유지되므로, 다양한 관점에서 정보를 조회하는 능력은 크게 손상되지 않습니다.

```
Multi-Head Attention (헤드 32개):
  Q: 32세트, K: 32세트, V: 32세트
  KV Cache: 32세트의 Key + 32세트의 Value

Grouped-Query Attention (8그룹):
  Q: 32세트 (독립), K: 8세트 (공유), V: 8세트 (공유)
  KV Cache: 8세트의 Key + 8세트의 Value --> 원래의 1/4
```

GQA의 극단적인 형태로 **MQA(Multi-Query Attention)**가 있습니다. MQA에서는 모든 헤드가 단 하나의 Key와 Value를 공유합니다. KV Cache가 원래의 1/H (헤드 수 분의 1)로 줄어들지만, 모델 품질에 미치는 영향이 GQA보다 클 수 있습니다. LLaMA 2 70B, Gemma 같은 모델은 GQA를 채택하여 KV Cache 크기와 모델 품질 사이의 균형을 맞추고 있습니다.

또 다른 압축 접근법은 KV Cache에 저장하는 데이터의 **정밀도를 낮추는** 것입니다. 모델 가중치는 FP16으로 유지하면서, KV Cache만 INT8(1바이트)이나 INT4(0.5바이트)로 양자화하는 방식입니다. FP16 대비 INT8로 양자화하면 KV Cache 크기가 절반으로 줄어듭니다. KV Cache는 가중치와 달리 추론 중에 동적으로 생성되는 값이므로, 약간의 정밀도 손실이 최종 출력 품질에 미치는 영향이 상대적으로 작다는 점을 활용하는 것입니다.

마지막으로, **컨텍스트 길이와 KV Cache의 관계**를 정리하겠습니다. 앞에서 계산한 공식에서 알 수 있듯이, KV Cache 크기는 시퀀스 길이에 정비례합니다. 시퀀스 길이가 2배가 되면 KV Cache도 2배가 됩니다. 최근 모델들은 컨텍스트 윈도우가 점점 길어지는 추세입니다. 1.2.3에서 언급한 것처럼 GPT-4는 128,000 토큰, Claude 3는 200,000 토큰을 지원합니다.

```
LLaMA 2 7B (FP16) 기준 요청 1건의 KV Cache 크기:

시퀀스 길이      KV Cache 크기
-------------------------------
  1,024 토큰       약 512 MB
  4,096 토큰       약 2 GB
 16,384 토큰       약 8 GB
 65,536 토큰       약 32 GB
131,072 토큰       약 64 GB
```

시퀀스 길이가 131,072 토큰에 이르면, 7B 모델임에도 요청 한 건의 KV Cache가 64 GB에 달합니다. 80 GB GPU 하나에 모델 가중치(14 GB)와 KV Cache(64 GB)를 합하면 78 GB로, 다른 요청을 처리할 여유가 거의 남지 않습니다. 이 때문에 긴 컨텍스트를 지원하는 서비스에서는 PagedAttention, Prefix Caching, GQA, KV Cache 양자화 같은 기법을 조합하여 메모리 효율을 극대화해야 합니다.

정리하면, KV Cache는 자기회귀 생성에서 이전 토큰의 Key와 Value를 재사용하여 중복 연산을 제거하는 핵심 장치이며, 그 크기는 층 수, 히든 차원, 시퀀스 길이에 비례합니다. 전통적인 연속 메모리 할당 방식은 내부 단편화와 외부 단편화로 메모리를 낭비하는데, PagedAttention은 블록 단위 할당으로 이 문제를 해결합니다. Prefix Caching은 공통 접두사의 KV Cache를 공유하여 메모리와 연산을 절약하고, GQA나 KV Cache 양자화 같은 압축 기법은 저장해야 하는 데이터 자체를 줄입니다. 컨텍스트 길이가 길어질수록 KV Cache 관리의 중요성은 더 커지며, 이들 기법의 적절한 조합이 서빙 처리량과 비용을 결정합니다.

다음 단원인 6.3.1에서는 양자화의 원리를 다룹니다. 모델 가중치의 정밀도를 낮추어 메모리 사용량과 연산 비용을 줄이는 방법과, 정밀도 변화가 모델 품질에 미치는 영향을 살펴봅니다.

이 단원을 마치면 KV Cache의 동작 원리를 설명하고, 메모리 사용량을 계산하며, PagedAttention, Prefix Caching, GQA 같은 기법을 활용하여 메모리 효율적인 KV Cache 관리 방법을 적용할 수 있습니다.
