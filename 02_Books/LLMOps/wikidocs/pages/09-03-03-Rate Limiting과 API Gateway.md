# 9.3.3 Rate Limiting과 API Gateway

LLM 서비스를 운영하다 보면, 특정 시간대에 요청이 한꺼번에 몰리는 상황을 자주 겪습니다. 마케팅 캠페인이 시작되거나, 내부 배치 작업이 동시에 실행되면 수백 건의 API 호출이 몇 초 사이에 쏟아집니다. 모델 추론은 GPU 자원을 집중적으로 사용하므로, 요청이 제한 없이 밀려들면 GPU 메모리가 고갈되거나 응답 지연이 급격히 치솟습니다. 한 사용자의 대량 호출이 다른 사용자의 정상적인 요청까지 느리게 만드는 현상이 발생하기도 합니다. 이 단원에서는 이러한 문제를 예방하기 위한 **Rate Limiting** 전략과, 요청의 흐름을 중앙에서 제어하는 **API Gateway**의 역할을 다룹니다.

Rate Limiting이란 일정 시간 동안 허용하는 요청의 수 또는 양을 제한하는 기법입니다. 웹 서비스에서 흔히 사용하는 개념이지만, LLM 환경에서는 일반적인 웹 API와 다른 점이 있습니다. 일반적인 REST API에서는 요청 한 건의 처리 비용이 대체로 균일합니다. 사용자 정보를 조회하는 GET 요청이든 주문을 생성하는 POST 요청이든, 서버가 소비하는 CPU와 메모리의 차이가 크지 않습니다. 반면 LLM API는 요청마다 소비하는 자원의 편차가 매우 큽니다. 짧은 질문에 한 문장으로 답하는 요청은 입출력 토큰을 합쳐 수백 개만 사용하지만, 긴 문서를 요약하는 요청은 수만 개의 토큰을 소비합니다. 1.1.1에서 다룬 것처럼 LLM 호출 비용은 토큰 수에 비례하므로, 요청 건수만으로 제한을 걸면 자원 소비를 정확히 통제할 수 없습니다.

이 차이 때문에 LLM 환경의 Rate Limiting은 두 가지 방식으로 나뉩니다. 첫 번째는 **요청 기반 Rate Limiting**입니다. 이 방식은 단위 시간당 허용하는 요청 건수를 제한합니다. 예를 들어 "1분에 60건"이라는 규칙을 설정하면, 한 사용자가 1분 동안 60번을 초과하여 API를 호출할 수 없습니다. 구현이 단순하고 이해하기 쉬운 것이 장점입니다. 그러나 앞서 설명한 것처럼 LLM 요청은 건별 비용 편차가 크므로, 60건 모두가 긴 문서 요약이면 GPU 자원이 고갈될 수 있고, 60건 모두가 짧은 질문이면 실제로는 여유가 남는 상황이 생깁니다. 요청 건수만으로는 실제 자원 소비를 반영하지 못하는 것입니다.

두 번째는 **토큰 기반 Rate Limiting**입니다. 이 방식은 단위 시간당 허용하는 토큰 수를 제한합니다. 예를 들어 "1분에 100,000 토큰"이라는 규칙을 설정하면, 해당 사용자가 1분 동안 소비할 수 있는 입출력 토큰의 합계가 100,000개로 제한됩니다. 긴 요청은 토큰을 많이 소비하므로 적은 횟수만 허용되고, 짧은 요청은 토큰을 적게 소비하므로 더 많은 횟수를 보낼 수 있습니다. 실제 자원 소비에 비례하여 제한이 걸리므로 GPU 사용량을 더 정밀하게 통제할 수 있습니다. 다만, 요청을 보내기 전에 출력 토큰 수를 정확히 알 수 없다는 어려움이 있습니다. 입력 토큰은 요청 시점에 계산할 수 있지만, 모델이 몇 개의 토큰을 생성할지는 추론이 끝나야 알 수 있습니다. 실무에서는 입력 토큰만 사전에 차감하고 출력 토큰은 응답 완료 후 사후에 차감하는 방식을 사용하거나, max_tokens 파라미터 값을 기준으로 최대치를 사전에 예약하는 방식을 사용합니다.

실무에서는 두 방식을 함께 적용하는 경우가 많습니다. "1분에 60건, 그리고 1분에 100,000 토큰" 같이 요청 건수와 토큰 수를 동시에 제한하면, 소량의 거대한 요청이 자원을 독점하는 것도, 대량의 소형 요청이 시스템을 과부하시키는 것도 함께 막을 수 있습니다.

```
요청 기반 Rate Limiting           토큰 기반 Rate Limiting
-----------------------------     -----------------------------
단위: 요청 건수/시간              단위: 토큰 수/시간
예: 60 req/min                    예: 100,000 tokens/min
장점: 구현 단순                   장점: 자원 소비에 비례
단점: 요청별 비용 편차 반영 못함  단점: 출력 토큰 사전 예측 어려움

실무 권장: 두 방식을 동시에 적용
```

Rate Limiting 규칙을 설정했다면, 이 규칙을 실제로 적용하는 알고리즘이 필요합니다. 대표적인 알고리즘으로 **Token Bucket**과 **Sliding Window**가 있습니다.

Token Bucket 알고리즘은 이름 그대로 양동이(bucket)에 토큰이 담겨 있는 것에 비유한 방식입니다. 여기서 말하는 "토큰"은 LLM의 텍스트 토큰이 아니라, 요청 허용 권한을 나타내는 추상적인 단위입니다. 혼동을 피하기 위해 이 알고리즘의 토큰을 "허용 토큰"이라 부르겠습니다. 동작 원리를 단계별로 살펴보겠습니다.

첫째, 양동이에는 최대 용량이 정해져 있습니다. 예를 들어 최대 용량이 10이라면, 양동이에는 최대 10개의 허용 토큰이 들어갈 수 있습니다. 둘째, 일정한 속도로 허용 토큰이 양동이에 추가됩니다. 예를 들어 초당 2개씩 추가된다면, 5초가 지나면 10개가 채워집니다. 양동이가 이미 가득 차 있으면 추가되는 허용 토큰은 버려집니다. 셋째, 요청이 들어오면 양동이에서 허용 토큰을 하나(또는 요청 크기에 비례하는 수만큼) 꺼냅니다. 양동이에 허용 토큰이 남아 있으면 요청을 승인하고, 비어 있으면 요청을 거부하거나 대기시킵니다.

```
Token Bucket 동작 흐름

[허용 토큰 보충]     [양동이 (최대 10개)]     [요청 도착]
   초당 2개    --->  | * * * * * * |     <---  요청 A: 토큰 1개 소비 --> 승인
                     | * * * * *   |     <---  요청 B: 토큰 1개 소비 --> 승인
                     | * * * *     |     <---  ...
                     |             |     <---  요청 K: 양동이 비어 있음 --> 거부
                     +-------------+

시간이 지나면 다시 토큰이 채워져 요청을 받을 수 있게 됩니다.
```

Token Bucket의 특징은 순간적인 트래픽 폭증을 어느 정도 허용한다는 점입니다. 양동이에 허용 토큰이 가득 차 있을 때 요청이 몰리면, 최대 용량만큼은 연속으로 처리할 수 있습니다. 그 이후에는 보충 속도에 맞춰 요청이 제한됩니다. 이 특성 덕분에 평소에는 여유롭게 운영하다가 갑자기 요청이 늘어나는 패턴에 유연하게 대응할 수 있습니다. LLM 서비스에서 Token Bucket을 사용할 때는 허용 토큰 하나가 LLM 토큰 일정 수에 대응하도록 설정하여, 토큰 기반 Rate Limiting과 결합하는 것이 일반적입니다.

Sliding Window 알고리즘은 시간 창(window)을 기준으로 해당 구간 내의 요청 수를 추적하는 방식입니다. 고정 창(fixed window) 방식과 비교하면 차이가 명확해집니다. 고정 창 방식은 "매 1분의 시작"을 기준으로 카운터를 초기화합니다. 예를 들어 12:00:00에 카운터가 0으로 리셋되고, 12:00:59까지의 요청을 세어 제한을 적용합니다. 문제는 12:00:50에 60건이 몰리고 12:01:00에 다시 60건이 몰리면, 10초 사이에 120건이 처리되는 상황이 발생한다는 점입니다. 카운터가 정각에 리셋되기 때문에 경계 시점에서 제한이 무력화됩니다.

Sliding Window는 이 문제를 해결합니다. 요청이 들어온 시점을 기준으로 직전 1분(또는 설정한 시간 창)을 거슬러 올라가, 그 구간 내의 요청 수를 셉니다. 12:00:50에 요청이 들어오면 11:59:50부터 12:00:50까지의 요청을 세는 것입니다. 창이 요청마다 함께 "미끄러지므로" 경계 시점의 폭증 문제가 발생하지 않습니다. 다만, 모든 요청의 타임스탬프를 기록해야 하므로 고정 창보다 메모리를 더 사용합니다. 실무에서는 정밀한 Sliding Window 대신, 이전 구간과 현재 구간의 가중 평균을 구하는 Sliding Window Counter 방식을 사용하여 메모리 사용을 줄이기도 합니다.

```
고정 창 (Fixed Window)

  12:00         12:01         12:02
  |----60건----|----60건----|
       ↑ 카운터 리셋         ↑ 카운터 리셋
  경계 전후 10초에 120건 처리 가능 (제한 무력화)


슬라이딩 윈도우 (Sliding Window)

  요청 시점 12:00:50 기준, 직전 60초를 봄
  |---------- 60초 ----------|
  11:59:50                12:00:50
  이 구간 안의 요청 수로 제한 판단 --> 경계 문제 없음
```

지금까지 다룬 Rate Limiting은 요청을 허용하거나 거부하는 이분법적 판단입니다. 그런데 요청을 거부하면 사용자 경험이 나빠지므로, 거부 대신 속도를 늦추는 방식을 사용하기도 합니다. 이것이 **Throttling**입니다. Throttling은 제한을 초과한 요청을 즉시 거부하지 않고, 대기열(큐)에 넣어 순서대로 처리합니다. 사용자는 응답이 느려지기는 하지만, 요청 자체가 실패하지는 않습니다.

큐 관리는 Throttling의 핵심입니다. 큐에 요청이 계속 쌓이기만 하면 메모리가 고갈되거나 대기 시간이 무한히 길어지므로, 큐의 최대 길이와 요청의 최대 대기 시간을 설정해야 합니다. 큐가 가득 차면 그때는 요청을 거부합니다. 또한 큐에서 대기 중인 요청의 우선순위를 두어, 유료 사용자의 요청을 무료 사용자보다 먼저 처리하는 방식도 있습니다.

```
Rate Limiting + Throttling + 큐 관리 흐름

[요청 도착]
    |
    v
[Rate Limit 확인]
    |
    +--> 허용 범위 내 --> [즉시 처리] --> [모델 추론] --> [응답]
    |
    +--> 허용 초과 --> [큐에 삽입]
                          |
                          +--> 큐 여유 있음 --> [대기] --> 순서 도래 시 처리
                          |
                          +--> 큐 가득 참 --> [429 Too Many Requests 반환]
                          |
                          +--> 대기 시간 초과 --> [408 Request Timeout 반환]
```

여기까지 Rate Limiting의 개념과 알고리즘을 살펴보았습니다. 실제 운영에서는 이 Rate Limiting을 개별 서비스마다 직접 구현하지 않고, 중앙의 관문 역할을 하는 시스템에서 일괄 처리하는 것이 효율적입니다. 이 관문이 바로 **API Gateway**입니다.

API Gateway는 클라이언트와 백엔드 서비스 사이에 위치하여 모든 요청을 중계하는 서버입니다. 클라이언트가 직접 백엔드 서비스에 접근하지 않고, 반드시 API Gateway를 거치도록 구성합니다. API Gateway가 수행하는 주요 역할은 다음과 같습니다.

첫째, **인증과 인가**입니다. 요청에 포함된 API 키나 토큰을 검증하여, 정당한 사용자의 요청만 백엔드로 전달합니다. 둘째, **Rate Limiting**입니다. 앞서 다룬 Token Bucket이나 Sliding Window 알고리즘을 API Gateway 수준에서 적용하여, 개별 서비스가 Rate Limiting을 직접 구현하지 않아도 되게 합니다. 셋째, **요청 라우팅**입니다. 요청의 경로, 헤더, 파라미터에 따라 적절한 백엔드 서비스로 전달합니다. 넷째, **로깅과 모니터링**입니다. 모든 요청과 응답을 기록하여 사용량 추적, 비용 분석, 이상 감지의 기반을 마련합니다. 다섯째, **로드 밸런싱**입니다. 동일한 서비스의 여러 인스턴스에 요청을 분산합니다.

```
API Gateway의 위치와 역할

[클라이언트 A]--+
                |     +------------------+     +------------------+
[클라이언트 B]--+---->| API Gateway      |---->| LLM 서빙 서버    |
                |     |                  |     +------------------+
[클라이언트 C]--+     | - 인증/인가      |     +------------------+
                      | - Rate Limiting  |---->| LLM 서빙 서버    |
                      | - 요청 라우팅    |     +------------------+
                      | - 로깅/모니터링  |
                      | - 로드 밸런싱    |
                      +------------------+
```

일반적인 API Gateway(Kong, NGINX, AWS API Gateway 등)도 LLM 서비스 앞에 둘 수 있지만, LLM 고유의 요구사항을 충족하기에는 한계가 있습니다. 일반 API Gateway는 토큰 기반 Rate Limiting을 기본 제공하지 않고, 모델별 사용량 추적이나 프롬프트 수준의 로깅을 지원하지 않습니다. 이러한 gap을 메우기 위해 LLM에 특화된 Gateway 솔루션이 등장했습니다. 이를 **LLM Gateway** 또는 **AI Gateway**라 부릅니다.

대표적인 LLM Gateway 솔루션으로 **LiteLLM**이 있습니다. LiteLLM은 100개 이상의 LLM 제공자(OpenAI, Anthropic, Azure, Bedrock, Vertex AI 등)를 하나의 통일된 인터페이스로 묶어 주는 오픈소스 프록시입니다. 6.1.1에서 다룬 OpenAI 호환 API 형식을 사용하므로, 클라이언트는 어떤 모델 제공자를 사용하든 동일한 형식으로 요청을 보낼 수 있습니다. LiteLLM이 제공하는 주요 기능을 살펴보겠습니다.

LiteLLM은 사용자별, 팀별, API 키별로 Rate Limiting과 예산 한도를 설정할 수 있습니다. 요청 건수와 토큰 수를 동시에 제한할 수 있어, 앞서 설명한 이중 제한 전략을 적용하기 쉽습니다. 또한 모든 요청과 응답의 메타데이터(모델명, 토큰 수, 지연 시간, 비용 등)를 자동으로 기록하여 비용 추적과 사용량 분석의 기반을 제공합니다.

아래는 LiteLLM 프록시를 설정하는 예입니다.

```yaml
model_list:
  - model_name: gpt-4o
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY
  - model_name: claude-sonnet
    litellm_params:
      model: anthropic/claude-sonnet-4-20250514
      api_key: os.environ/ANTHROPIC_API_KEY

litellm_settings:
  max_budget: 100          # 전체 월간 예산 (USD)
  budget_duration: monthly
```

'model_list'는 프록시가 중계할 모델들을 정의합니다. 'model_name'은 클라이언트가 사용할 이름이고, 'litellm_params' 안의 'model'은 실제 제공자의 모델 식별자입니다. 'api_key'에 'os.environ/'을 붙이면 환경 변수에서 키를 읽어옵니다. 'litellm_settings'에서 전체 예산 한도를 설정할 수 있습니다. 이 설정으로 프록시를 시작하면, 클라이언트는 프록시의 주소로 요청을 보내고 'model' 필드에 'gpt-4o' 또는 'claude-sonnet'을 지정하면 됩니다. 프록시가 해당 제공자로 요청을 전달하고 응답을 돌려줍니다.

**Portkey**는 또 다른 LLM Gateway 솔루션으로, 클라우드 서비스와 오픈소스 버전을 모두 제공합니다. Portkey는 LiteLLM과 마찬가지로 여러 LLM 제공자를 통합하면서, 자동 재시도(retry), 폴백(fallback), 캐싱, 요청 로깅 같은 운영 기능을 추가로 제공합니다. 예를 들어 OpenAI API가 일시적으로 오류를 반환하면 자동으로 Anthropic API로 전환하는 폴백 규칙을 설정할 수 있습니다. 이러한 폴백 기능은 서비스 안정성에 직접적으로 기여합니다.

LLM Gateway가 단일 제공자만 지원하면 되는 경우는 드뭅니다. 실무에서는 용도에 따라 여러 모델을 사용합니다. 간단한 분류 작업에는 비용이 낮은 소형 모델을, 복잡한 추론에는 대형 모델을, 코드 생성에는 코드 특화 모델을 사용하는 식입니다. 이때 요청의 특성에 따라 적절한 모델로 자동 전달하는 것이 **다중 모델 라우팅**입니다.

다중 모델 라우팅의 기본 구조는 라우팅 규칙을 API Gateway 수준에서 정의하는 것입니다. 규칙의 기준은 다양합니다. 요청 경로(예: '/v1/chat/simple'은 소형 모델, '/v1/chat/advanced'는 대형 모델)로 나눌 수도 있고, 요청 헤더에 모델 등급을 명시하는 방식도 있습니다. 9.3.2에서 다룬 A/B 테스트와 결합하여 트래픽의 일정 비율을 새 모델로 보내는 방식도 라우팅의 일종입니다. LiteLLM에서는 같은 model_name에 여러 제공자를 등록하고 라우팅 전략(예: 가장 저렴한 것, 가장 빠른 것, 라운드 로빈)을 설정하여 다중 모델 라우팅을 구현합니다.

```
다중 모델 라우팅 예시

                    +--- 분류 요청 -----> [소형 모델: gpt-4o-mini]
                    |
[API Gateway] ------+--- 추론 요청 -----> [대형 모델: gpt-4o]
                    |
                    +--- 코드 요청 -----> [코드 모델: claude-sonnet]
                    |
                    +--- 장애 감지 시 ---> [폴백 모델: 대체 제공자]
```

폴백 라우팅은 안정성 측면에서 특히 중요합니다. 특정 모델 제공자에 장애가 발생하면, API Gateway가 이를 감지하여 자동으로 대체 모델로 요청을 전환합니다. 이 과정에서 클라이언트 코드를 수정할 필요가 없습니다. API Gateway가 중간에서 라우팅을 변경하기 때문입니다. 이것이 API Gateway를 두는 핵심 이점 중 하나입니다.

정리하면, LLM 환경의 Rate Limiting은 요청 건수와 토큰 수를 동시에 제한하여 GPU 자원을 보호하고, Token Bucket과 Sliding Window 알고리즘으로 제한 규칙을 적용합니다. Throttling과 큐 관리를 통해 제한 초과 요청을 유연하게 처리할 수 있습니다. API Gateway는 인증, Rate Limiting, 라우팅, 로깅을 중앙에서 수행하며, LiteLLM이나 Portkey 같은 LLM 전용 Gateway는 토큰 기반 비용 관리와 다중 모델 라우팅까지 지원합니다.

다음 단원인 9.4.1에서는 모델 거버넌스 체계를 다룹니다. LLM 시스템을 조직 차원에서 통제하고 규정을 준수하기 위한 프레임워크와 프로세스를 살펴봅니다.

이 단원을 마치면 LLM API의 Rate Limiting 전략을 수립하고 API Gateway를 구성할 수 있습니다.
