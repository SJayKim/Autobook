# 5.1.4 체인과 파이프라인 구성

LLM을 활용하는 애플리케이션은 대부분 단 한 번의 모델 호출로 끝나지 않습니다. 사용자의 질문을 받아 문맥을 보강하고, 모델에 전달하며, 응답을 가공해 반환하는 과정에는 여러 단계가 얽힙니다. 5.1.2에서 다룬 LangChain의 LCEL과 5.1.3에서 다룬 LlamaIndex의 QueryEngine은 각각 이런 다단계 흐름을 구성하는 기초 도구를 제공합니다. 이 단원에서는 그 기초 위에 실제 워크플로를 설계할 때 등장하는 여섯 가지 구성 패턴을 살펴봅니다. 순차 체인, 병렬 체인과 분기, 라우팅 체인, 대화형 체인에서의 메모리 활용, 스트리밍 처리, 그리고 에러 핸들링과 폴백입니다.

하나의 LLM 호출만으로 원하는 결과를 얻기 어려운 상황은 흔합니다. 예를 들어, 사용자가 영어로 작성된 기술 문서에 대해 한국어로 질문하면, 먼저 질문을 영어로 번역하고, 번역된 질문으로 문서를 검색한 뒤, 검색 결과를 참고해 영어 답변을 생성하고, 마지막으로 그 답변을 한국어로 되돌려야 합니다. 이처럼 앞 단계의 출력이 뒤 단계의 입력이 되는 구조를 **순차 체인**(Sequential Chain)이라 합니다.

순차 체인의 핵심 원리는 단순합니다. 첫 번째 단계가 실행을 마치면 그 출력을 두 번째 단계의 입력으로 넘기고, 두 번째가 끝나면 세 번째로 넘기는 식으로 마지막 단계까지 한 줄로 이어집니다. LangChain의 LCEL에서는 파이프 연산자(|)를 사용해 이를 표현합니다. 각 단계를 **Runnable**이라는 공통 인터페이스로 감싸기 때문에, 프롬프트 템플릿, 모델, 출력 파서를 동일한 방식으로 연결할 수 있습니다.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

translate_prompt = ChatPromptTemplate.from_template(
    "Translate the following Korean question to English:\n{question}"
)
answer_prompt = ChatPromptTemplate.from_template(
    "Answer the following question:\n{english_question}"
)

model = ChatOpenAI(model="gpt-4o")

chain = translate_prompt | model | StrOutputParser()
```

위 코드에서 translate_prompt는 한국어 질문을 영어로 바꾸라는 지시를 담은 프롬프트 템플릿입니다. 파이프 연산자로 model과 StrOutputParser를 잇고 있습니다. translate_prompt가 프롬프트 문자열을 만들면 model이 LLM을 호출하고, StrOutputParser가 응답 객체에서 텍스트만 추출합니다. 이 세 단계가 순차적으로 실행되므로 하나의 순차 체인을 구성합니다.

실제 애플리케이션에서는 이런 단일 순차 체인을 여러 개 이어 붙여, 번역 체인의 출력을 검색 체인에 넘기고, 검색 체인의 출력을 답변 생성 체인에 넘기는 더 긴 파이프라인을 만듭니다. 순차 체인의 장점은 흐름이 직관적이어서 디버깅이 쉽다는 점입니다. 어느 단계에서 문제가 생겼는지 출력을 하나씩 확인하면 됩니다. 반면, 모든 단계가 직렬로 실행되므로 총 소요 시간이 각 단계 소요 시간의 합이 된다는 단점이 있습니다.

```
순차 체인 실행 흐름

[사용자 입력]
     |
     v
+-----------+     +-----------+     +-------------+
| 프롬프트  | --> |   모델    | --> | 출력 파서   |
| 템플릿    |     |  (LLM)   |     |             |
+-----------+     +-----------+     +-------------+
     |                                     |
     +-- 단계 1의 출력이 단계 2의 입력 ---+
                                           |
                                           v
                                    [최종 결과]
```

모든 단계가 직렬로 연결될 필요는 없습니다. 서로 독립적인 작업은 동시에 실행해 총 처리 시간을 줄일 수 있습니다. 이것이 **병렬 체인**입니다. 예를 들어, 하나의 문서에 대해 요약과 키워드 추출을 동시에 수행하는 경우를 생각해 보겠습니다. 요약과 키워드 추출은 서로의 결과를 필요로 하지 않으므로 병렬로 실행해도 됩니다.

LangChain의 LCEL에서는 **RunnableParallel**을 사용해 병렬 실행을 표현합니다. RunnableParallel에 여러 체인을 이름과 함께 등록하면, 각 체인이 동시에 실행되고 결과가 딕셔너리로 합쳐집니다.

```python
from langchain_core.runnables import RunnableParallel

summary_chain = summary_prompt | model | StrOutputParser()
keyword_chain = keyword_prompt | model | StrOutputParser()

parallel = RunnableParallel(
    summary=summary_chain,
    keywords=keyword_chain,
)
result = parallel.invoke({"document": doc_text})
# result == {"summary": "...", "keywords": "..."}
```

위 코드에서 RunnableParallel은 summary_chain과 keyword_chain을 동시에 실행합니다. invoke를 호출하면 동일한 입력 {"document": doc_text}가 두 체인에 전달되고, 두 체인의 결과가 "summary"와 "keywords"라는 키로 묶여 반환됩니다. 두 체인 중 오래 걸리는 쪽의 시간이 전체 소요 시간이 되므로, 순차 실행보다 빠릅니다.

병렬 체인은 종종 **분기 로직**과 결합됩니다. 분기란 입력의 특성에 따라 서로 다른 처리 경로를 선택하는 것입니다. 예를 들어, 입력 텍스트가 영어인지 한국어인지에 따라 서로 다른 프롬프트 체인을 실행하는 경우가 이에 해당합니다. 분기 로직은 조건 함수를 사용해 입력을 검사한 뒤, 그 결과에 따라 적절한 체인으로 보냅니다.

분기보다 더 유연한 패턴이 **라우팅 체인**입니다. 라우팅 체인은 입력의 내용을 분석해 여러 전문 체인 중 적합한 하나로 전달하는 구조입니다. 분기는 보통 두세 가지 경로 중 하나를 고르지만, 라우팅은 수십 개의 경로를 다룰 수도 있습니다.

구체적인 예를 들어 보겠습니다. 고객 문의 시스템에서 "결제", "배송", "반품"과 같은 주제별로 각각 다른 전문 체인을 두었다고 합시다. 사용자가 "환불 절차가 궁금합니다"라고 입력하면, 라우팅 체인은 먼저 이 입력이 어떤 주제에 해당하는지 판별합니다. 판별 방법에는 키워드 매칭처럼 간단한 방법부터 LLM을 이용한 분류까지 다양합니다. 판별 결과가 "반품"이면 반품 전문 체인으로, "결제"면 결제 전문 체인으로 입력을 전달합니다.

LangChain에서는 **RunnableBranch**를 사용해 라우팅을 구현합니다. RunnableBranch는 조건-체인 쌍의 목록과, 어떤 조건에도 해당하지 않을 때 실행할 기본 체인을 받습니다.

```python
from langchain_core.runnables import RunnableBranch

branch = RunnableBranch(
    (lambda x: "payment" in x["topic"], payment_chain),
    (lambda x: "shipping" in x["topic"], shipping_chain),
    (lambda x: "return" in x["topic"], return_chain),
    default_chain,
)
```

위 코드에서 첫 번째 인자부터 세 번째 인자까지는 각각 조건 함수와 해당 체인의 쌍입니다. 마지막 인자 default_chain은 어떤 조건에도 맞지 않을 때 실행할 기본 체인입니다. 입력이 들어오면 위에서부터 조건을 차례로 평가하고, 처음으로 True가 되는 조건의 체인을 실행합니다.

```
라우팅 체인 동작 흐름

[사용자 입력]
     |
     v
+------------------+
|   주제 분류기    |
| (조건 판별)      |
+------------------+
     |         |         |         |
     v         v         v         v
 결제 체인  배송 체인  반품 체인  기본 체인
     |         |         |         |
     +----+----+----+----+----+----+
          |
          v
     [최종 응답]
```

지금까지 살펴본 순차, 병렬, 라우팅 체인은 모두 단발 요청을 처리합니다. 그런데 챗봇처럼 사용자와 여러 차례 대화를 주고받는 애플리케이션에서는 이전 대화 내용을 기억해야 합니다. LLM 자체는 상태를 보관하지 않으므로, 이전 대화를 매번 프롬프트에 포함시켜야 합니다. 이를 위해 사용하는 것이 **Memory**입니다.

Memory는 대화 이력을 저장하고, 매 호출 시 이력을 프롬프트에 주입하는 구성 요소입니다. 가장 단순한 형태는 전체 대화 기록을 그대로 저장하는 **ConversationBufferMemory**입니다. 대화가 길어지면 토큰 수가 늘어나 비용이 증가하고 컨텍스트 윈도우 제한에 걸릴 수 있으므로, 최근 N개 메시지만 유지하는 **ConversationBufferWindowMemory**나 이전 대화를 요약해 압축하는 **ConversationSummaryMemory** 같은 변형도 있습니다.

대화형 체인은 Memory를 순차 체인과 결합한 구조입니다. 사용자 입력이 들어오면, Memory에서 이전 대화 이력을 꺼내 프롬프트에 추가하고, LLM을 호출한 뒤, 이번 대화(사용자 입력과 모델 응답)를 다시 Memory에 저장합니다. 이 과정이 매 턴마다 반복됩니다.

```
대화형 체인의 한 턴 처리 흐름

+---------+     +----------+     +----------+     +---------+
| Memory  | --> | 프롬프트 | --> |  모델    | --> | Memory  |
| (읽기)  |     | 구성     |     | (LLM)   |     | (쓰기)  |
+---------+     +----------+     +----------+     +---------+
     ^               ^                                  |
     |               |                                  |
 이전 대화 이력  사용자 입력                    새 대화 쌍 저장
```

이 흐름에서 Memory(읽기)는 저장된 이전 대화를 꺼내는 단계이고, 프롬프트 구성은 시스템 메시지, 이전 대화, 현재 사용자 입력을 하나의 프롬프트로 합치는 단계입니다. 모델이 응답을 생성하면 Memory(쓰기)가 이번 턴의 입력과 응답을 저장합니다.

LLM은 응답 전체를 한꺼번에 만들지 않고, 토큰 단위로 순차 생성합니다. **Streaming**(스트리밍) 처리란 이 토큰이 생성되는 대로 클라이언트에 전달하는 방식입니다. 스트리밍을 사용하지 않으면 모든 토큰이 생성될 때까지 기다렸다가 한 번에 응답을 보내므로, 사용자는 수 초간 빈 화면을 보게 됩니다. 스트리밍을 적용하면 첫 번째 토큰이 생성되는 즉시 화면에 표시되기 시작하므로 체감 응답 시간이 크게 줄어듭니다.

LangChain에서 스트리밍을 적용하려면 invoke 대신 stream 메서드를 호출합니다. stream은 응답을 한 번에 반환하는 대신, 생성되는 토큰 묶음(chunk)을 하나씩 내보내는 반복 가능 객체(iterator)를 반환합니다.

```python
for chunk in chain.stream({"question": "LLMOps란 무엇인가요?"}):
    print(chunk, end="", flush=True)
```

위 코드에서 chain.stream은 체인 전체를 스트리밍 모드로 실행합니다. for 반복문이 chunk를 하나씩 받아 출력하므로, 토큰이 생성되는 대로 화면에 표시됩니다. end=""는 줄바꿈 없이 이어 출력하기 위한 설정이고, flush=True는 출력 버퍼를 즉시 비워 지연 없이 표시하기 위한 설정입니다.

순차 체인에서 스트리밍을 사용할 때 주의할 점이 있습니다. 체인의 중간 단계가 입력 전체를 필요로 하면 해당 단계에서 스트리밍이 차단됩니다. 예를 들어, 출력 파서가 JSON 전체를 파싱해야 한다면 모든 토큰이 모일 때까지 기다려야 합니다. 따라서 스트리밍의 효과를 최대화하려면 체인의 마지막 단계에서 스트리밍이 가능하도록 설계해야 합니다.

비동기 환경에서는 astream 메서드를 사용합니다. astream은 비동기 반복 가능 객체(async iterator)를 반환하며, async for 문으로 소비합니다. 웹 서버처럼 동시에 여러 요청을 처리해야 하는 환경에서는 비동기 스트리밍이 자원을 효율적으로 사용하는 데 유리합니다.

LLM 호출은 외부 API에 의존하므로 네트워크 오류, 서버 과부하, 요청 한도 초과 등 다양한 이유로 실패할 수 있습니다. **에러 핸들링**은 이런 실패를 감지하고 적절히 대응하는 것이고, **폴백**(fallback)은 주 경로가 실패했을 때 대체 경로로 전환하는 것입니다.

가장 기본적인 에러 핸들링은 재시도(retry)입니다. 일시적인 네트워크 오류나 서버 과부하는 잠시 뒤에 다시 요청하면 성공하는 경우가 많습니다. 재시도 간격을 점진적으로 늘리는 지수 백오프(exponential backoff) 전략을 사용하면, 서버에 과도한 부하를 주지 않으면서 성공 확률을 높일 수 있습니다.

재시도로도 해결되지 않으면 폴백이 필요합니다. LangChain에서는 with_fallbacks 메서드로 폴백 체인을 설정합니다. 주 모델이 실패하면 대체 모델로 자동 전환되는 구조입니다.

```python
main_model = ChatOpenAI(model="gpt-4o")
fallback_model = ChatOpenAI(model="gpt-4o-mini")

model_with_fallback = main_model.with_fallbacks([fallback_model])

chain = prompt | model_with_fallback | StrOutputParser()
```

위 코드에서 main_model은 주로 사용할 모델이고, fallback_model은 main_model이 실패할 때 대체할 모델입니다. with_fallbacks 메서드에 대체 모델의 리스트를 전달하면, 주 모델 호출이 예외를 던질 때 리스트의 다음 모델로 자동 전환됩니다. 폴백 모델도 실패하면 최종적으로 예외가 발생합니다. 폴백 대상은 모델뿐 아니라 체인 전체가 될 수도 있습니다. 예를 들어, RAG 체인이 실패하면 단순 LLM 체인으로 전환하는 식입니다.

에러 핸들링에서 또 하나 중요한 것은 타임아웃 설정입니다. LLM 호출이 비정상적으로 오래 걸릴 때 무한정 기다리지 않도록, 최대 대기 시간을 정해 두어야 합니다. 타임아웃이 초과되면 해당 호출을 취소하고 폴백 경로를 실행하거나 사용자에게 재시도를 안내할 수 있습니다.

```
에러 핸들링과 폴백 흐름

[사용자 입력]
     |
     v
+-------------+    성공    +----------+
| 주 모델     | --------> | 응답     |
| (gpt-4o)    |           +----------+
+-------------+
     | 실패
     v
+-------------+    성공    +----------+
| 폴백 모델   | --------> | 응답     |
| (gpt-4o-mini)|          +----------+
+-------------+
     | 실패
     v
+-------------+
| 예외 발생   |
| (에러 반환) |
+-------------+
```

정리하면, LLM 워크플로를 구성하는 패턴은 크게 여섯 가지입니다. 순차 체인은 단계를 일렬로 연결해 앞 단계의 출력을 뒤 단계의 입력으로 전달합니다. 병렬 체인은 독립적인 작업을 동시에 실행해 처리 시간을 줄입니다. 라우팅 체인은 입력의 특성에 따라 적합한 전문 체인을 선택합니다. Memory를 활용한 대화형 체인은 이전 대화 이력을 프롬프트에 포함시켜 맥락 있는 응답을 생성합니다. 스트리밍은 토큰 단위로 응답을 전달해 체감 응답 시간을 줄입니다. 에러 핸들링과 폴백은 외부 API 실패에 대비해 재시도와 대체 경로를 마련합니다. 이 패턴들은 개별로 쓰이기보다 조합되어 하나의 파이프라인을 이룹니다.

다음 단원인 5.2.1에서는 LLM 에이전트의 개념과 아키텍처를 다룹니다.

이 단원을 마치면 순차 체인, 병렬 체인, 라우팅 체인, 대화형 체인, 스트리밍, 에러 핸들링과 폴백을 조합해 복잡한 LLM 워크플로를 체인과 파이프라인으로 구성할 수 있습니다.
