# 7.1.2 LoRA (Low-Rank Adaptation)

7.1.1에서 파인튜닝의 기본 개념과 전략을 살펴보았습니다. Full Fine-tuning은 모델의 모든 파라미터를 업데이트하므로 강력하지만, 대규모 언어 모델에서는 현실적인 제약이 큽니다. 70억 개의 파라미터를 가진 모델을 Full Fine-tuning하려면, 모델 가중치 자체뿐 아니라 그래디언트와 옵티마이저 상태까지 GPU 메모리에 올려야 합니다. 결과적으로 모델 크기의 수 배에 달하는 메모리가 필요합니다. 이 비용 문제를 해결하면서도, 특정 태스크에 맞게 모델을 조정하는 방법이 바로 **LoRA(Low-Rank Adaptation)**입니다.

LoRA를 이해하려면, 먼저 이 기법이 출발한 관찰부터 알아야 합니다. 대규모 언어 모델을 파인튜닝할 때 실제로 변하는 가중치의 변화량을 분석해 보면, 그 변화가 매우 낮은 차원의 공간에 집중되어 있다는 사실이 발견되었습니다. 수학적으로 표현하면, 가중치 변화 행렬의 **랭크(rank)**가 매우 낮다는 뜻입니다. 랭크란 행렬이 실질적으로 담고 있는 독립적인 정보의 차원 수를 말합니다. 예를 들어, 4096 x 4096 크기의 가중치 변화 행렬이 있더라도, 그 행렬의 랭크가 8이라면, 실질적으로 8차원 분량의 정보만 담고 있는 것입니다. 나머지 차원의 변화는 이 8개 방향의 조합으로 표현할 수 있습니다.

이 관찰이 LoRA의 핵심 아이디어인 **Low-Rank Decomposition(저랭크 분해)**으로 이어집니다. 가중치 변화 행렬이 낮은 랭크를 가진다면, 그 큰 행렬을 굳이 통째로 저장하고 학습할 필요가 없습니다. 대신, 두 개의 작은 행렬의 곱으로 근사할 수 있습니다. 원래 4096 x 4096 크기의 변화 행렬을 직접 다루는 대신, 4096 x 8 크기의 행렬과 8 x 4096 크기의 행렬 두 개로 분해하면, 저장해야 할 파라미터 수가 약 16,777,216개에서 65,536개로 줄어듭니다. 학습해야 할 파라미터가 약 256분의 1로 감소하는 것입니다.

이제 LoRA의 수학적 원리를 구체적으로 살펴보겠습니다. 원래 모델의 특정 가중치 행렬을 W0이라고 하겠습니다. Full Fine-tuning에서는 학습이 끝난 뒤의 가중치를 W라고 하면, 가중치 변화량은 delta_W = W - W0입니다. LoRA는 이 delta_W를 두 개의 작은 행렬 B와 A의 곱으로 표현합니다.

```
W = W0 + delta_W
delta_W = B * A

여기서:
  W0 : 원래 사전 학습된 가중치 (d x d 행렬, 학습 중 고정)
  B  : d x r 행렬 (학습 대상)
  A  : r x d 행렬 (학습 대상)
  r  : 랭크, d보다 훨씬 작은 값 (예: r = 8, 16, 64)
  d  : 원래 가중치의 차원 (예: 4096)
```

핵심은 W0을 고정한 채로, B와 A만 학습한다는 점입니다. 원래 모델의 가중치는 전혀 건드리지 않습니다. 추론 시에 입력 x가 들어오면, 원래 경로인 W0 * x와 LoRA 경로인 B * A * x를 더하여 최종 출력을 만듭니다.

```
h = W0 * x + B * A * x
```

학습을 시작할 때 B는 모든 값이 0인 영행렬로, A는 작은 랜덤 값으로 초기화합니다. B가 0이므로 B * A도 0이 되어, 학습 시작 시점에서는 LoRA가 원래 모델의 동작을 전혀 변경하지 않습니다. 학습이 진행되면서 B와 A가 점차 조정되어, 태스크에 필요한 변화만 반영합니다.

이 과정을 단계별로 정리하면 다음과 같습니다.

```
[LoRA 학습 과정]

1. 사전 학습된 가중치 W0을 고정한다 (requires_grad = False)
2. 작은 행렬 B (d x r, 영행렬)와 A (r x d, 랜덤 초기화)를 추가한다
3. 순전파: h = W0 * x + (alpha / r) * B * A * x
4. 역전파: B와 A에 대해서만 그래디언트를 계산하고 업데이트한다
5. W0은 변하지 않으므로 옵티마이저 상태도 B, A 분량만 유지한다
```

위 3단계에서 alpha / r이라는 항이 등장합니다. 이것이 **Alpha 파라미터**와 관련된 스케일링입니다. Alpha는 LoRA의 변화량이 원래 모델 출력에 얼마나 큰 영향을 줄지를 조절하는 하이퍼파라미터입니다. LoRA의 출력에 alpha / r을 곱하여 스케일링합니다. 이 비율을 **스케일링 팩터(scaling factor)**라고 부릅니다.

왜 이런 스케일링이 필요한지 생각해 보겠습니다. 랭크 r을 바꾸면 B * A의 출력 크기도 달라집니다. r이 커지면 더 많은 정보를 담을 수 있지만, 동시에 출력의 크기도 변할 수 있습니다. alpha를 r로 나누면, r을 변경하더라도 LoRA 출력의 전체 스케일이 대체로 일정하게 유지됩니다. 덕분에 r을 실험적으로 바꿔 볼 때 학습률 같은 다른 하이퍼파라미터를 매번 다시 조정하지 않아도 됩니다.

실무에서 흔히 쓰이는 설정은 alpha를 r의 2배로 두는 것입니다. 예를 들어 r = 16이면 alpha = 32으로 설정합니다. 이 경우 스케일링 팩터는 32 / 16 = 2가 됩니다. alpha를 r과 같은 값으로 두면 스케일링 팩터가 1이 되어, LoRA 출력이 그대로 더해집니다. alpha가 r보다 크면 LoRA의 영향력이 커지고, 작으면 영향력이 줄어듭니다.

다음으로 **Rank(r) 파라미터**의 의미와 선택 기준을 살펴보겠습니다. r은 LoRA에서 가장 중요한 하이퍼파라미터입니다. r이 작을수록 학습 파라미터 수가 줄어들어 메모리와 계산 비용이 절감되지만, 표현할 수 있는 변화의 복잡도가 제한됩니다. 반대로 r이 클수록 더 복잡한 패스크 적응이 가능하지만, LoRA의 효율성 이점이 줄어듭니다.

r 값의 선택은 태스크의 복잡도에 따라 달라집니다. 단순한 분류 태스크나 감정 분석처럼 모델이 학습해야 할 새로운 패턴이 적은 경우에는 r = 4나 r = 8 정도로도 충분한 성능을 낼 수 있습니다. 반면, 새로운 도메인의 전문 지식을 학습하거나 코드 생성처럼 복잡한 패턴을 익혀야 하는 경우에는 r = 32나 r = 64가 필요할 수 있습니다. 실무에서는 r = 8 또는 r = 16을 시작점으로 두고, 검증 데이터셋의 성능을 보며 조정하는 방식이 일반적입니다.

아래 표는 r 값에 따른 학습 파라미터 수 변화를 보여줍니다. 원래 가중치가 4096 x 4096인 경우를 가정합니다.

```
+-------+------------------+--------------------+-------------------+
| r     | A 파라미터 수    | B 파라미터 수      | 합계              |
+-------+------------------+--------------------+-------------------+
|   4   | 4 x 4096 = 16K  | 4096 x 4 = 16K    |  32K  (0.19%)     |
|   8   | 8 x 4096 = 32K  | 4096 x 8 = 32K    |  64K  (0.38%)     |
|  16   | 16 x 4096 = 64K | 4096 x 16 = 64K   | 128K  (0.76%)     |
|  64   | 64 x 4096 = 256K| 4096 x 64 = 256K  | 512K  (3.05%)     |
+-------+------------------+--------------------+-------------------+
  원래 가중치 파라미터 수: 4096 x 4096 = 16,777,216 (16M)
  괄호 안 비율은 원래 파라미터 대비 LoRA 파라미터 비율
```

r = 8일 때 원래 파라미터의 0.38%만 학습하면 되는 것입니다.

이제 LoRA를 모델의 어디에 적용할지, 즉 **타겟 모듈 선택**을 다루겠습니다. 1.2.1에서 Transformer 아키텍처의 핵심 구성 요소를 살펴보았습니다. Self-Attention 메커니즘에는 Query, Key, Value를 만드는 가중치 행렬(W_Q, W_K, W_V)과 출력을 합치는 가중치 행렬(W_O)이 있고, Feed-Forward Network(FFN)에는 확장과 축소를 담당하는 가중치 행렬이 있습니다. LoRA는 이 가중치 행렬들 중 어디에든 적용할 수 있지만, 어떤 모듈에 적용하느냐에 따라 효과가 달라집니다.

LoRA 원래 논문에서는 주로 **Query(W_Q)**와 **Value(W_V)** 행렬에 LoRA를 적용했을 때 좋은 성과를 보고했습니다. Query는 "어떤 정보를 찾을지"를 결정하고, Value는 "어떤 정보를 전달할지"를 결정하는 역할이므로, 이 두 행렬을 조정하면 모델이 태스크에 맞게 주의를 기울이는 방식을 효과적으로 바꿀 수 있습니다.

실무에서는 적용 범위를 넓히는 경향이 있습니다. Query, Key, Value, 출력 행렬 모두에 LoRA를 적용하거나, 여기에 더해 FFN의 가중치 행렬까지 포함하는 경우도 많습니다. 적용 모듈을 늘리면 학습 파라미터 수는 증가하지만, 모델이 조정할 수 있는 범위가 넓어져 성능이 향상되는 경우가 많습니다. 다만 모든 모듈에 높은 r 값으로 LoRA를 적용하면 메모리 이점이 줄어들므로, 적용 범위와 r 값 사이에서 균형을 잡아야 합니다.

Hugging Face의 PEFT 라이브러리에서는 target_modules 파라미터로 LoRA를 적용할 모듈을 지정합니다. 아래는 대표적인 설정 예시입니다.

```python
from peft import LoraConfig

# Query와 Value에만 적용 (기본적인 설정)
config_basic = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM"
)

# Attention 전체 + FFN까지 적용 (더 넓은 범위)
config_wide = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                     "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM"
)
```

위 코드에서 r은 랭크, lora_alpha는 스케일링에 쓰이는 alpha 값입니다. target_modules는 LoRA를 적용할 선형 계층의 이름 목록입니다. q_proj, k_proj, v_proj, o_proj는 각각 Query, Key, Value, Output 프로젝션 행렬에 해당합니다. gate_proj, up_proj, down_proj는 FFN 내부의 가중치 행렬입니다. lora_dropout은 LoRA 계층에 적용할 드롭아웃 비율로, 과적합을 방지하는 정규화 기법입니다. bias="none"은 편향 파라미터는 학습하지 않겠다는 설정입니다. task_type="CAUSAL_LM"은 텍스트 생성 모델임을 지정합니다. 모듈 이름은 모델 아키텍처에 따라 다를 수 있으므로, 사용하는 모델의 구조를 확인한 뒤 설정해야 합니다.

마지막으로 **LoRA 어댑터 병합과 배포**를 다루겠습니다. LoRA 학습이 끝나면, 결과물은 원래 모델 가중치 W0과 별도로 저장된 B, A 행렬 쌍입니다. 이 B, A 행렬 쌍을 **어댑터(adapter)**라고 부릅니다. 어댑터의 크기는 원래 모델에 비해 매우 작습니다. 70억 파라미터 모델의 경우 원래 가중치가 약 14GB라면, LoRA 어댑터는 수십 MB 수준에 불과합니다.

배포 시에는 두 가지 방식을 선택할 수 있습니다. 첫 번째는 어댑터를 분리 상태로 유지하는 것입니다. 원래 모델을 메모리에 올린 뒤, 요청에 따라 서로 다른 어댑터를 로드하여 적용합니다. 하나의 기본 모델 위에 고객 상담용 어댑터, 코드 생성용 어댑터, 요약용 어댑터를 상황에 맞게 교체할 수 있습니다. 기본 모델은 한 번만 메모리에 올리면 되므로, 여러 태스크를 지원하면서도 메모리를 효율적으로 사용할 수 있습니다.

두 번째는 어댑터를 원래 가중치에 **병합(merge)**하는 것입니다. W = W0 + B * A 연산을 수행하여 하나의 가중치 행렬로 합치는 것입니다. 병합하면 추론 시에 별도의 LoRA 연산이 필요 없으므로, 추론 속도가 LoRA를 적용하지 않은 원래 모델과 동일해집니다. 다만 병합된 모델은 크기가 원래 모델과 같아지고, 어댑터를 유연하게 교체할 수 없다는 단점이 있습니다.

```
[배포 방식 비교]

분리 배포:
  기본 모델 (W0) ──┬── 어댑터 A (상담용)  ──> 상담 응답
                    ├── 어댑터 B (코드용)  ──> 코드 생성
                    └── 어댑터 C (요약용)  ──> 문서 요약
  장점: 하나의 모델로 여러 태스크, 어댑터 교체 용이
  단점: 추론 시 LoRA 연산 추가 (미미한 지연)

병합 배포:
  병합 모델 (W0 + BA) ──> 단일 태스크 추론
  장점: 추론 지연 없음, 배포 구조 단순
  단점: 태스크별 별도 모델 필요, 유연성 감소
```

정리하면, LoRA는 사전 학습된 가중치를 고정한 채, 가중치 변화량을 두 개의 작은 행렬(B, A)의 곱으로 근사하는 파라미터 효율적 파인튜닝 기법입니다. 랭크 r로 학습 파라미터 수를 조절하고, alpha와 r의 비율로 LoRA 출력의 스케일을 관리합니다. Query, Value 등 Transformer 내부의 특정 모듈에 선택적으로 적용하며, 학습된 어댑터는 분리 배포하거나 원래 가중치에 병합하여 배포할 수 있습니다.

다음 단원인 7.1.3에서는 LoRA를 기반으로 4비트 양자화를 결합한 **QLoRA**를 다룹니다. QLoRA는 LoRA보다 더 적은 GPU 메모리로 파인튜닝을 가능하게 하는 기법입니다.

이 단원을 마치면 LoRA의 수학적 원리를 설명하고 파인튜닝에 적용할 수 있습니다.
