# 6.1.4 TensorRT-LLM

대규모 언어 모델을 서비스에 배포할 때 가장 먼저 부딪히는 문제는 추론 속도입니다. 1.2.3에서 살펴본 것처럼, 모델은 자기회귀 방식으로 토큰을 하나씩 생성하며 Decode 단계가 전체 추론 시간의 대부분을 차지합니다. 사용자가 체감하는 응답 지연을 줄이려면 이 과정을 가능한 한 빠르게 처리해야 합니다. NVIDIA GPU를 사용하는 환경이라면, NVIDIA가 제공하는 추론 최적화 도구를 활용하여 같은 모델에서 더 높은 처리량과 더 낮은 지연 시간을 달성할 수 있습니다. 이 단원에서는 그 도구인 TensorRT-LLM의 구조와 사용 방법을 다룹니다.

TensorRT-LLM이 어떤 도구인지 이해하려면, NVIDIA의 추론 최적화 생태계가 어떻게 구성되어 있는지를 먼저 알아야 합니다. NVIDIA는 GPU 하드웨어를 만드는 것에 그치지 않고, GPU 위에서 연산을 수행하기 위한 소프트웨어 스택을 함께 제공합니다. 이 스택의 핵심 요소 중 하나가 **TensorRT**입니다. TensorRT는 학습이 끝난 딥러닝 모델을 NVIDIA GPU에서 최적화된 형태로 실행하기 위한 추론 엔진입니다. 일반적인 딥러닝 프레임워크(PyTorch, TensorFlow 등)에서 모델을 실행하면 범용적인 연산 경로를 거칩니다. TensorRT는 이 연산 경로를 GPU 아키텍처에 맞게 재구성하여 불필요한 연산을 줄이고, 메모리 접근 패턴을 개선하며, 여러 연산을 하나로 합치는 등의 최적화를 수행합니다.

TensorRT가 일반적인 딥러닝 모델에 대한 추론 최적화 엔진이라면, **TensorRT-LLM**은 대규모 언어 모델에 특화된 추론 라이브러리입니다. 대규모 언어 모델은 일반적인 딥러닝 모델과 다른 특성을 가집니다. 자기회귀 생성, KV 캐시 관리, 여러 사용자 요청의 동시 처리 등 LLM 고유의 요구사항이 있습니다. TensorRT-LLM은 TensorRT의 최적화 기술 위에 이런 LLM 고유의 요구사항을 처리하는 기능을 추가한 것입니다. NVIDIA가 오픈소스로 공개하고 있으며, Python API를 통해 모델을 정의하고 최적화된 추론 엔진을 빌드할 수 있습니다.

NVIDIA의 추론 관련 소프트웨어 스택을 정리하면 다음과 같습니다.

```
NVIDIA GPU 하드웨어 (A100, H100 등)
    |
CUDA (GPU 범용 연산 플랫폼)
    |
cuBLAS, cuDNN (행렬 연산, 딥러닝 연산 라이브러리)
    |
TensorRT (범용 딥러닝 추론 최적화 엔진)
    |
TensorRT-LLM (LLM 특화 추론 라이브러리)
    |
Triton Inference Server (모델 서빙 서버)
```

이 다이어그램에서 아래로 갈수록 사용자에게 가까운 계층입니다. TensorRT-LLM은 하위 계층의 최적화 기술을 활용하면서 LLM에 필요한 기능을 제공하고, Triton Inference Server는 TensorRT-LLM으로 빌드한 엔진을 HTTP/gRPC 엔드포인트로 노출하여 실제 서비스에서 요청을 받을 수 있게 합니다.

TensorRT-LLM을 사용하여 모델을 서빙하려면, PyTorch 등에서 학습한 모델 가중치를 TensorRT-LLM이 실행할 수 있는 형태로 변환해야 합니다. 이 과정은 크게 두 단계로 나뉩니다.

첫 번째 단계는 **모델 가중치 변환**입니다. PyTorch 형식(Hugging Face의 safetensors 등)으로 저장된 모델 가중치를 TensorRT-LLM의 내부 표현으로 변환합니다. 이 과정에서 모델의 구조(레이어 수, 히든 사이즈, 어텐션 헤드 수 등)에 맞는 설정을 지정합니다. 양자화를 적용할 경우에도 이 단계에서 양자화 방식을 선택합니다.

두 번째 단계는 **엔진 빌드**입니다. 변환된 가중치를 바탕으로 TensorRT 최적화 엔진을 생성합니다. 이 단계에서 TensorRT는 모델의 연산 그래프를 분석하여 GPU 아키텍처에 맞는 최적의 실행 계획을 수립합니다. 여러 연산을 하나의 커널로 합치는 **레이어 퓨전(layer fusion)**, 메모리 사용 패턴 최적화, 최적의 커널 선택 등이 이 단계에서 이루어집니다. 빌드 결과물은 특정 GPU 아키텍처에 최적화된 바이너리 파일이며, 이것을 **엔진 파일**이라고 부릅니다.

전체 과정을 정리하면 다음과 같습니다.

```
[PyTorch 모델 가중치]
    |
    |  (1) 가중치 변환 (convert_checkpoint)
    v
[TensorRT-LLM 체크포인트]
    |
    |  (2) 엔진 빌드 (trtllm-build)
    v
[TensorRT 엔진 파일] --> GPU에서 최적화된 추론 실행
```

엔진 빌드에는 시간이 걸립니다. 모델 크기에 따라 수 분에서 수십 분이 소요될 수 있습니다. 엔진 파일은 빌드 시 지정한 GPU 아키텍처, 최대 배치 크기, 최대 시퀀스 길이 등의 설정에 종속됩니다. 다른 GPU로 옮기거나 설정을 변경하려면 엔진을 다시 빌드해야 합니다.

TensorRT-LLM이 추론 성능을 높이는 핵심 방법 중 하나가 **양자화(quantization)**입니다. 양자화는 모델 가중치와 연산에 사용하는 숫자의 정밀도를 낮추는 기법입니다. 딥러닝 모델은 기본적으로 32비트 부동소수점(FP32) 형식으로 가중치를 저장합니다. 하나의 가중치가 32비트, 즉 4바이트를 차지합니다. 70억 개의 파라미터를 가진 모델이라면 가중치만 약 28GB의 메모리가 필요합니다. 양자화를 적용하면 이 숫자의 정밀도를 낮추어 메모리 사용량을 줄이고 연산 속도를 높일 수 있습니다.

TensorRT-LLM은 여러 수준의 양자화를 지원합니다. **FP16**은 가중치를 16비트 부동소수점으로 표현합니다. FP32 대비 메모리 사용량이 절반으로 줄어듭니다. 정밀도 손실이 적어 대부분의 경우 모델 품질에 큰 영향을 주지 않습니다. NVIDIA GPU의 Tensor Core가 FP16 연산을 하드웨어 수준에서 가속하므로 연산 속도도 빨라집니다.

**INT8** 양자화는 가중치를 8비트 정수로 표현합니다. FP32 대비 메모리 사용량이 4분의 1로 줄어듭니다. FP16보다 더 공격적인 압축이므로 정밀도 손실이 발생할 수 있습니다. 이 손실을 줄이기 위해 TensorRT-LLM은 보정(calibration) 과정을 거칩니다. 대표적인 입력 데이터를 모델에 통과시켜 각 레이어의 활성값 범위를 측정하고, 이를 기반으로 양자화 파라미터를 결정합니다.

**INT4** 양자화는 가중치를 4비트 정수로 표현합니다. FP32 대비 메모리 사용량이 8분의 1로 줄어듭니다. 70억 파라미터 모델이 약 3.5GB만으로도 표현될 수 있어, 단일 GPU에 훨씬 큰 모델을 올릴 수 있습니다. 정밀도 손실이 더 크므로, 품질 저하를 최소화하기 위한 기법이 함께 사용됩니다. TensorRT-LLM은 GPTQ, AWQ 같은 INT4 양자화 기법을 지원합니다.

```
양자화 수준    비트 수    FP32 대비 메모리    정밀도 손실    속도 향상
-----------------------------------------------------------------------
FP32           32비트     1배 (기준)          없음           기준
FP16           16비트     1/2배               거의 없음      높음
INT8            8비트     1/4배               소량           더 높음
INT4            4비트     1/8배               중간           가장 높음
```

양자화 수준을 선택할 때는 모델 품질과 추론 성능 사이의 균형을 고려해야 합니다. FP16은 품질 손실 없이 성능을 높이는 안전한 선택입니다. INT8은 메모리가 제한된 환경에서 더 큰 모델을 서빙하거나 처리량을 높이고 싶을 때 사용합니다. INT4는 가장 공격적인 압축이 필요할 때 선택하되, 반드시 결과 품질을 평가한 뒤 적용해야 합니다.

TensorRT-LLM이 제공하는 또 다른 핵심 기능은 **In-Flight Batching**입니다. 이 기능을 이해하려면 먼저 일반적인 배치 처리의 한계를 알아야 합니다.

모델 서빙에서 배치(batch)란 여러 요청을 묶어서 한꺼번에 처리하는 것을 뜻합니다. GPU는 병렬 처리에 강하므로, 요청을 하나씩 처리하는 것보다 여러 요청을 묶어서 처리하는 것이 GPU 활용 효율이 높습니다. 가장 단순한 배치 방식은 **정적 배칭(static batching)**입니다. 여러 요청을 모아 한 배치로 만들고, 배치 안의 모든 요청이 생성을 완료할 때까지 기다린 뒤 결과를 한꺼번에 반환합니다.

정적 배칭의 문제는 배치 안에서 요청마다 생성하는 토큰 수가 다르다는 점에서 발생합니다. 예를 들어, 배치 안에 3개의 요청이 있고 각각 10개, 50개, 200개의 토큰을 생성한다고 가정합니다. 정적 배칭에서는 200개 토큰을 모두 생성할 때까지 배치가 유지됩니다. 10개 토큰 생성이 끝난 첫 번째 요청은 나머지 190 스텝 동안 GPU 자원을 낭비하며 대기합니다. 그 시간에 새 요청을 처리할 수 있었지만, 배치가 끝날 때까지 기다려야 합니다.

```
정적 배칭:

시간 -->
요청 A: [===].....................  (10 토큰 후 대기)
요청 B: [=============]..........  (50 토큰 후 대기)
요청 C: [============================]  (200 토큰)
         ^                              ^
         배치 시작                       배치 종료 (모두 완료될 때까지)
```

**In-Flight Batching**은 이 비효율을 해결합니다. "비행 중 배칭"이라는 이름처럼, 생성이 진행되는 도중에 배치 구성을 동적으로 바꿉니다. 요청이 생성을 완료하면 즉시 배치에서 빠지고, 대기 중인 새 요청이 빈 자리에 들어옵니다. 이를 **연속 배칭(continuous batching)**이라고도 부릅니다.

```
In-Flight Batching:

시간 -->
요청 A: [===]
요청 D:      [========]          (A 완료 후 즉시 투입)
요청 B: [=============]
요청 E:                [======]  (B 완료 후 즉시 투입)
요청 C: [============================]
```

이 방식에서는 요청 A가 10 토큰 생성을 마치는 즉시 GPU 자원이 해제되고, 대기 큐에 있던 요청 D가 바로 배치에 합류합니다. GPU가 쉬는 시간이 줄어들어 전체 처리량이 크게 향상됩니다. TensorRT-LLM은 이 In-Flight Batching을 내장하여, 별도의 배치 관리 로직을 구현하지 않아도 높은 처리량을 달성할 수 있습니다.

TensorRT-LLM으로 빌드한 엔진을 실제 서비스에 배포하려면, 외부에서 HTTP나 gRPC로 추론 요청을 보낼 수 있는 서빙 인프라가 필요합니다. NVIDIA는 이를 위해 **Triton Inference Server**를 제공합니다. Triton Inference Server는 NVIDIA가 개발한 오픈소스 모델 서빙 서버로, 다양한 딥러닝 프레임워크(TensorRT, PyTorch, TensorFlow, ONNX 등)의 모델을 하나의 서버에서 관리하고 서빙할 수 있습니다.

TensorRT-LLM과 Triton Inference Server의 통합은 **Triton TensorRT-LLM Backend**를 통해 이루어집니다. 이 백엔드는 Triton이 TensorRT-LLM 엔진을 로드하고, In-Flight Batching을 포함한 TensorRT-LLM의 기능을 Triton의 서빙 인터페이스와 연결합니다. 사용자는 Triton의 모델 저장소에 TensorRT-LLM 엔진 파일과 설정 파일을 배치하면, Triton이 해당 엔진을 로드하여 HTTP/gRPC 엔드포인트로 서비스합니다.

통합 배포의 전체 흐름은 다음과 같습니다.

```
(1) PyTorch 모델 --> (2) TensorRT-LLM 엔진 빌드
                              |
                              v
                    (3) Triton 모델 저장소에 엔진 배치
                              |
                              v
                    (4) Triton Inference Server 실행
                              |
                              v
                    (5) 클라이언트 --> HTTP/gRPC 요청 --> 추론 결과
```

Triton은 모델 버전 관리, 동적 배칭, 모델 앙상블, 헬스체크, 메트릭 수집 등 프로덕션 서빙에 필요한 기능을 제공합니다. TensorRT-LLM이 추론 엔진의 성능을 담당하고, Triton이 서빙 인프라의 운영 기능을 담당하는 구조입니다.

TensorRT-LLM의 성능을 다른 서빙 프레임워크와 비교하면, GPU 활용 효율에서 차이가 나타납니다. TensorRT-LLM은 NVIDIA GPU에 특화된 최적화를 수행하므로, 동일한 NVIDIA GPU에서 vLLM이나 TGI와 비교했을 때 일반적으로 더 높은 처리량(초당 처리 토큰 수)과 더 낮은 지연 시간을 달성합니다. 특히 레이어 퓨전, 커널 최적화, 양자화 지원 등 GPU 하드웨어에 밀착된 최적화가 성능 차이의 주요 원인입니다.

다만 TensorRT-LLM에는 절충점도 있습니다. 엔진 빌드 과정이 필요하므로 모델 배포까지의 시간이 더 깁니다. 빌드된 엔진은 특정 GPU 아키텍처에 종속되어 이식성이 낮습니다. 새 모델 아키텍처가 등장했을 때 지원까지 시간이 걸릴 수 있습니다. vLLM이나 TGI는 PyTorch 모델을 직접 로드하여 바로 서빙할 수 있어 배포 과정이 단순합니다. 따라서 최대 성능이 필요한 프로덕션 환경에서는 TensorRT-LLM이 유리하고, 빠른 프로토타이핑이나 다양한 모델을 자주 교체하는 환경에서는 vLLM이나 TGI가 더 적합할 수 있습니다.

```
항목             TensorRT-LLM        vLLM / TGI
------------------------------------------------------
추론 성능        높음 (GPU 최적화)    보통
배포 과정        엔진 빌드 필요       모델 직접 로드
GPU 종속성       특정 아키텍처        범용적
새 모델 지원     빌드 필요            빠른 지원
양자화 지원      FP16/INT8/INT4       FP16/INT8 (제한적)
배칭 방식        In-Flight Batching   Continuous Batching
```

정리하면, TensorRT-LLM은 NVIDIA GPU 환경에서 대규모 언어 모델의 추론 성능을 극대화하기 위한 라이브러리입니다. PyTorch 모델을 가중치 변환과 엔진 빌드 두 단계를 거쳐 GPU에 최적화된 형태로 변환하고, FP16, INT8, INT4 양자화로 메모리 사용량과 연산 속도를 조절하며, In-Flight Batching으로 GPU 활용 효율을 높입니다. Triton Inference Server와 결합하면 프로덕션 수준의 모델 서빙 인프라를 구축할 수 있습니다.

다음 단원인 6.2.1에서는 GPU 추론 최적화 기법을 다룹니다. TensorRT-LLM 같은 프레임워크가 내부적으로 사용하는 최적화 기술들의 원리를 더 깊이 살펴봅니다.

이 단원을 마치면 TensorRT-LLM의 최적화 과정을 설명하고 모델을 빌드할 수 있습니다.
