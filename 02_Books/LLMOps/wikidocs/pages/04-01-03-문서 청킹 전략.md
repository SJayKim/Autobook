# 4.1.3 문서 청킹 전략

4.1.2에서 기본 RAG 파이프라인을 구축하면서, 문서 로딩 후 청크로 분할하는 과정을 간략히 다루었습니다. 그때 RecursiveCharacterTextSplitter를 사용하여 1,000자 단위로 텍스트를 나누는 예를 보았습니다. 이것은 가장 단순한 형태의 청킹이었습니다. 실제 서비스에서는 청킹 전략에 따라 RAG의 검색 품질과 최종 응답 품질이 크게 달라집니다. 같은 문서, 같은 임베딩 모델, 같은 LLM을 사용하더라도, 청킹을 어떻게 하느냐에 따라 사용자의 질문에 대한 정답이 검색 결과에 포함되기도 하고 누락되기도 합니다. 이 단원에서는 다양한 청킹 전략의 원리, 장단점, 그리고 전략별 성능 차이를 다룹니다.

먼저 **청킹이 왜 필요한지**, 그리고 **어떤 원칙을 따라야 하는지** 짚어 보겠습니다.

문서를 청크로 나누는 이유는 두 가지입니다. 첫째, 임베딩 모델에는 한 번에 처리할 수 있는 입력 길이의 상한이 있습니다. 3.1.2에서 살펴본 것처럼, 임베딩 모델마다 최대 토큰 수가 다릅니다. all-MiniLM-L6-v2는 256토큰, text-embedding-3-small은 8,191토큰까지 처리할 수 있습니다. 이 상한을 넘는 텍스트는 잘려 나가거나 오류가 발생합니다. 둘째, 긴 텍스트를 통째로 하나의 벡터로 만들면 의미가 희석됩니다. 20페이지짜리 매뉴얼을 하나의 벡터로 압축하면, 매뉴얼의 어느 부분이 질문과 관련 있는지 구분하기 어렵습니다. 작은 단위로 나누면 각 청크가 특정 주제에 집중되므로, 질문과 관련된 부분만 정확히 검색할 수 있습니다.

청킹의 기본 원칙은 세 가지입니다. 첫째, 하나의 청크는 하나의 주제나 맥락을 담아야 합니다. 두 가지 다른 주제가 하나의 청크에 섞이면, 임베딩이 어느 쪽 의미도 정확히 포착하지 못합니다. 둘째, 청크의 경계에서 문맥이 끊기면 안 됩니다. 문장 중간에서 잘리면, 그 청크만으로는 의미를 파악할 수 없습니다. 셋째, 청크의 크기는 임베딩 모델의 최대 입력 길이와 LLM의 컨텍스트 윈도우를 함께 고려해야 합니다. 청크가 너무 크면 임베딩 모델의 입력 상한을 초과하고, 여러 청크를 LLM 프롬프트에 넣을 때 컨텍스트 윈도우를 소모합니다.

이 원칙을 바탕으로 구체적인 청킹 전략을 하나씩 살펴보겠습니다.

첫 번째 전략은 **고정 크기 청킹**(Fixed-size Chunking)입니다. 가장 단순한 방식으로, 텍스트를 고정된 문자 수(또는 토큰 수)마다 기계적으로 자릅니다. 1,000자마다 잘라서 하나의 청크를 만드는 것입니다.

고정 크기 청킹의 장점은 구현이 간단하고 예측 가능하다는 것입니다. 모든 청크의 크기가 동일하므로, 임베딩 모델의 입력 상한을 초과하지 않도록 관리하기 쉽습니다. 그러나 치명적인 단점이 있습니다. 텍스트의 내용을 전혀 고려하지 않으므로, 문장 중간이나 문단 중간에서 잘릴 수 있습니다. "환불은 구매 후 7일 이내에 신청해야 하며, 전자제"에서 잘리면 이 청크만으로는 전자제품의 환불 조건을 알 수 없습니다.

이 문제를 완화하기 위해 **오버랩**(overlap)을 사용합니다. 오버랩이란 인접한 청크 사이에 일정량의 텍스트를 겹치게 하는 것입니다. 1,000자 청크에 200자 오버랩을 설정하면, 첫 번째 청크는 0~999자, 두 번째 청크는 800~1,799자를 담습니다. 이렇게 하면 청크 경계에서 잘린 문장이 다음 청크에서 완전한 형태로 포함될 가능성이 높아집니다.

```
고정 크기 청킹 (오버랩 포함)

원본 텍스트:  |--------- 1000자 ---------|--------- 1000자 ---------|
청크 1:       |=====================|
청크 2:                      |=====================|
청크 3:                                     |=====================|
                              ^오버랩^
```

두 번째 전략은 **재귀적 청킹**(Recursive Character Splitting)입니다. 4.1.2에서 RecursiveCharacterTextSplitter를 사용한 예가 바로 이 전략입니다. 재귀적 청킹은 텍스트를 나눌 때 여러 구분자를 우선순위에 따라 순서대로 시도합니다.

동작 방식을 단계별로 설명하겠습니다. 구분자 목록이 ["\n\n", "\n", ". ", " "]라고 가정합니다. 먼저 전체 텍스트를 빈 줄("\n\n")로 나눕니다. 나눈 결과 중 청크 크기 상한(예: 1,000자)을 초과하는 조각이 있으면, 그 조각을 다음 구분자인 줄바꿈("\n")으로 다시 나눕니다. 여전히 초과하는 조각이 있으면 마침표와 공백(". ")으로 나누고, 그래도 초과하면 공백(" ")으로 나눕니다. 이 과정을 재귀적으로 반복하여, 모든 청크가 상한 이내에 들어올 때까지 분할합니다.

```
재귀적 청킹 과정

전체 텍스트
    |
    v
빈 줄("\n\n")로 분할 --> 문단 단위 조각들
    |
    v
크기 초과 조각이 있는가?
    |
    +-- 아니오 --> 완료
    |
    +-- 예 --> 해당 조각을 줄바꿈("\n")으로 분할
                    |
                    v
               크기 초과 조각이 있는가?
                    |
                    +-- 아니오 --> 완료
                    |
                    +-- 예 --> 마침표(". ")로 분할
                                    |
                                    v
                               (계속 반복)
```

재귀적 청킹의 장점은 가능한 한 문단이나 문장 경계를 유지한다는 것입니다. 고정 크기 청킹처럼 문장 중간에서 잘리는 일이 줄어듭니다. LangChain의 RecursiveCharacterTextSplitter가 이 전략을 구현한 대표적인 도구입니다. 실무에서 가장 널리 사용되는 청킹 전략이기도 합니다. 단점은 구분자에 의존하므로, 구분자가 명확하지 않은 텍스트(예: 줄바꿈 없이 이어진 긴 텍스트)에서는 고정 크기 청킹과 큰 차이가 없을 수 있다는 것입니다.

세 번째 전략은 **의미 기반 청킹**(Semantic Chunking)입니다. 고정 크기 청킹과 재귀적 청킹은 텍스트의 "형식"(문자 수, 줄바꿈, 마침표)을 기준으로 나눕니다. 의미 기반 청킹은 텍스트의 "의미"를 기준으로 나눕니다. 주제가 바뀌는 지점을 감지하여 그곳에서 청크를 분리하는 것입니다.

의미 기반 청킹의 동작 원리는 다음과 같습니다. 먼저 텍스트를 문장 단위로 분리합니다. 각 문장을 임베딩 모델로 벡터화합니다. 인접한 문장 쌍의 코사인 유사도를 계산합니다. 유사도가 급격히 떨어지는 지점, 즉 주제가 바뀌는 지점을 청크의 경계로 삼습니다.

```
의미 기반 청킹 원리

문장 1: "환불 정책은 구매 후 7일 이내입니다."     유사도
문장 2: "환불 시 영수증이 필요합니다."             0.85  (높음)
문장 3: "반품 배송비는 고객 부담입니다."           0.78  (보통)
---- 유사도 급락 ---- 경계선 ----
문장 4: "신제품 A가 출시되었습니다."               0.25  (낮음)
문장 5: "제품 A의 주요 특징은 ..."                0.82  (높음)
```

위 예시에서 문장 3과 문장 4 사이에 코사인 유사도가 급격히 떨어집니다. 환불 관련 내용에서 신제품 소개로 주제가 바뀌었기 때문입니다. 의미 기반 청킹은 이 지점에서 청크를 나눕니다. 문장 1~3이 하나의 청크가 되고, 문장 4~5가 다른 청크가 됩니다.

의미 기반 청킹의 장점은 하나의 청크가 하나의 주제를 담을 가능성이 높다는 것입니다. 임베딩 벡터가 해당 주제를 더 정확하게 표현하므로, 검색 품질이 높아집니다. 단점은 모든 문장을 임베딩해야 하므로 전처리 비용이 높다는 것입니다. 문서가 수만 건일 때는 청킹 단계에서만 상당한 API 호출 비용이나 GPU 연산이 필요합니다.

네 번째 전략은 **문서 구조 기반 청킹**(Markdown, HTML 헤더)입니다. 문서 자체에 구조적 표시가 있는 경우, 그 구조를 청킹의 기준으로 활용하는 전략입니다.

Markdown 문서에는 # 제목, ## 소제목 같은 헤더가 있고, HTML 문서에는 h1, h2, h3 태그가 있습니다. 이 헤더는 문서 작성자가 주제를 구분한 경계입니다. 따라서 헤더 단위로 청크를 나누면, 각 청크가 자연스럽게 하나의 주제를 담게 됩니다.

```
Markdown 기반 청킹 예시

원본 문서:
# 환불 정책
환불은 구매 후 7일 이내에 ...
영수증이 필요합니다 ...

# 배송 정책
배송은 결제 후 1~3영업일 ...
해외 배송은 별도 문의 ...

# 교환 정책
동일 상품 교환은 ...

청크 분할 결과:
  청크 1: [제목: 환불 정책] "환불은 구매 후 7일 이내에 ... 영수증이 ..."
  청크 2: [제목: 배송 정책] "배송은 결제 후 1~3영업일 ... 해외 배송은 ..."
  청크 3: [제목: 교환 정책] "동일 상품 교환은 ..."
```

문서 구조 기반 청킹의 장점은 문서 작성자의 의도를 그대로 반영한다는 것입니다. 추가적인 임베딩 계산이 필요 없으므로 비용도 들지 않습니다. 또한 청크에 헤더 정보를 메타데이터로 포함시키면, 검색 결과에 "이 청크는 '환불 정책' 섹션에서 온 것이다"라는 맥락을 함께 제공할 수 있습니다. 단점은 문서에 헤더가 없거나 구조가 불규칙한 경우에는 적용할 수 없다는 것입니다. 또한 하나의 섹션이 매우 길면 해당 청크가 임베딩 모델의 입력 상한을 초과할 수 있습니다. 이 경우 구조 기반 청킹을 먼저 적용한 뒤, 초과하는 청크에 대해 재귀적 청킹을 추가로 적용하는 혼합 전략이 효과적입니다.

다음으로 **청크 크기와 오버랩 설정**을 살펴보겠습니다. 어떤 청킹 전략을 사용하든 결국 청크의 크기를 결정해야 합니다. 청크 크기의 선택은 두 가지 상충 관계에 놓여 있습니다.

청크가 작으면 각 청크가 좁은 주제에 집중하므로, 정밀한 검색이 가능합니다. 질문과 관련된 핵심 문장만 포함된 청크를 정확히 찾아낼 확률이 높아집니다. 그러나 너무 작으면 문맥이 부족하여 LLM이 답변을 생성할 때 필요한 정보가 부족할 수 있습니다. 또한 같은 양의 문서에서 더 많은 청크가 생성되므로 저장 공간과 인덱싱 비용이 증가합니다.

청크가 크면 하나의 청크에 풍부한 맥락이 담기므로, LLM이 더 완전한 답변을 생성할 수 있습니다. 그러나 너무 크면 청크 안에 질문과 무관한 내용이 많이 포함되어 임베딩의 의미가 희석되고, 검색 정밀도가 떨어집니다.

일반적인 가이드라인은 다음과 같습니다. FAQ나 짧은 문답 형태의 문서에서는 200~500자가 적합합니다. 일반적인 문서(매뉴얼, 가이드)에서는 500~1,500자가 많이 사용됩니다. 법률 문서나 학술 논문처럼 긴 논리 흐름이 중요한 문서에서는 1,500~3,000자가 필요할 수 있습니다. 오버랩은 청크 크기의 10~20%를 설정하는 것이 일반적입니다. 1,000자 청크에 100~200자 오버랩이 흔한 설정입니다.

```
청크 크기별 특성

청크 크기     검색 정밀도     맥락 완전성     저장 비용
-------------------------------------------------------
작음 (200~500)    높음          낮음          높음
중간 (500~1500)   보통          보통          보통
큼 (1500~3000)    낮음          높음          낮음
```

마지막으로 **청킹 전략별 성능 비교**를 정리하겠습니다. 어떤 전략이 최선인지는 문서의 특성과 질문의 유형에 따라 달라집니다. 절대적으로 우월한 전략은 없습니다.

```
청킹 전략별 비교

전략             구현 난이도   전처리 비용   검색 품질     적합한 문서 유형
---------------------------------------------------------------------------
고정 크기        매우 낮음     없음          낮음~보통     구조가 없는 텍스트
재귀적           낮음          없음          보통          일반 문서 (가장 범용)
의미 기반        높음          높음 (임베딩)  높음          주제 전환이 잦은 문서
문서 구조 기반   낮음          없음          높음          헤더가 있는 문서
```

실무에서 권장하는 접근 방식은 다음과 같습니다. 먼저 재귀적 청킹으로 시작합니다. 재귀적 청킹은 구현이 간단하면서도 문장 경계를 유지하므로, 대부분의 경우 합리적인 기준선이 됩니다. 검색 품질이 부족하다고 판단되면, 문서에 헤더가 있으면 구조 기반 청킹을, 주제 전환이 빈번하면 의미 기반 청킹을 시도합니다. 여러 전략을 비교할 때는 동일한 질문 셋에 대해 각 전략의 검색 결과를 평가하여 정량적으로 판단합니다.

정리하면, 청킹은 긴 문서를 검색에 적합한 크기의 조각으로 나누는 과정으로, RAG의 검색 품질을 좌우하는 핵심 단계입니다. 고정 크기 청킹은 단순하지만 문맥을 훼손할 수 있고, 재귀적 청킹은 구분자 우선순위에 따라 문장 경계를 유지합니다. 의미 기반 청킹은 주제 변화를 감지하여 가장 자연스러운 분할을 만들지만 전처리 비용이 높습니다. 문서 구조 기반 청킹은 헤더를 활용하여 작성자의 의도를 반영합니다. 청크 크기는 검색 정밀도와 맥락 완전성의 균형점을 찾아야 하며, 오버랩으로 경계에서의 문맥 단절을 완화합니다.

다음 단원인 4.2.1에서는 검색 품질 최적화를 다룹니다. 하이브리드 검색, 리랭킹, 쿼리 변환 등 검색 단계에서의 품질을 높이는 기법을 살펴봅니다.

이 단원을 마치면 다양한 청킹 전략의 특성을 비교하고 데이터에 적합한 전략을 선택할 수 있습니다.
