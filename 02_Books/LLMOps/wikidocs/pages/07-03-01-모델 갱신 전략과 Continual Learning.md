# 7.3.1 모델 갱신 전략과 Continual Learning

7.2.3에서 사용자 피드백을 학습 데이터로 순환시키는 데이터 플라이휠을 다루었습니다. 플라이휠이 돌면서 새로운 학습 데이터가 계속 쌓입니다. 그런데 데이터가 쌓인다고 해서 모델이 자동으로 나아지지는 않습니다. 모델에 새로운 데이터를 반영하려면, 모델을 갱신하는 과정이 필요합니다. 이 단원에서는 모델 갱신이 왜 필요한지, 갱신 방식에는 어떤 선택지가 있는지, 갱신 과정에서 발생하는 핵심 문제인 카타스트로픽 포겟팅을 어떻게 완화하는지를 살펴봅니다.

모델 갱신이 필요한 첫 번째 이유는 **데이터 드리프트(data drift)**입니다. 데이터 드리프트란 서비스에 들어오는 입력의 패턴이 시간이 지남에 따라 변하는 현상을 뜻합니다. 모델을 학습할 때의 데이터 분포와 실제 운영 중에 들어오는 데이터의 분포가 달라지는 것입니다. 예를 들어, 고객 서비스 챗봇을 2024년 1월에 학습시켰는데, 6월에 새 제품이 출시되면 사용자들이 새 제품에 관한 질문을 쏟아냅니다. 이 질문들은 학습 데이터에 존재하지 않으므로, 모델의 응답 품질이 떨어집니다. 계절에 따라 질문 유형이 바뀌거나, 사회적 이슈에 따라 특정 주제의 질문이 급증하는 것도 데이터 드리프트의 사례입니다.

두 번째 이유는 **지식 갱신**입니다. 세상의 정보는 계속 변합니다. 법률이 개정되고, 기술 사양이 업데이트되고, 조직의 정책이 바뀝니다. LLM은 학습 시점까지의 정보만 가지고 있으므로, 학습 이후에 변한 정보에 대해서는 오래된 답변을 내놓을 수 있습니다. 모델이 최신 정보를 반영하려면 갱신이 필요합니다.

세 번째 이유는 **성능 개선**입니다. 7.2.3에서 다룬 데이터 플라이휠을 통해 새로운 고품질 학습 데이터가 쌓이면, 이 데이터를 모델에 반영하여 성능을 높일 수 있습니다. 사용자 피드백으로 발견된 약점을 보강하는 데이터, 능동 학습으로 선별된 데이터를 학습시키면 모델이 한 단계 나아집니다.

모델을 갱신하는 방식은 크게 두 가지로 나뉩니다. **주기적 재학습(periodic retraining)**과 **증분 학습(incremental learning)**입니다.

주기적 재학습은 일정 주기(매주, 매월 등)마다 누적된 전체 학습 데이터를 모아서 모델을 처음부터 다시 파인튜닝하는 방식입니다. 이 방식의 장점은 단순합니다. 데이터셋 전체를 한꺼번에 학습하므로, 과거 데이터와 새 데이터를 균형 있게 반영할 수 있고, 이전 학습의 오류가 누적되지 않습니다. 단점은 비용이 높다는 것입니다. 매번 전체 데이터로 학습을 처음부터 수행하므로, 데이터가 커질수록 학습 시간과 GPU 비용이 증가합니다. 또한 갱신 주기 사이에는 새 데이터가 반영되지 않습니다. 월 단위 재학습이면, 이번 달 초에 발견된 문제가 다음 달 초에나 반영됩니다.

증분 학습은 새로 수집된 데이터만으로 기존 모델을 추가 학습하는 방식입니다. 전체 데이터를 다시 사용하지 않고, 새 데이터에 대해서만 짧게 파인튜닝합니다. 장점은 비용이 낮고 갱신이 빠르다는 것입니다. 새 데이터의 양이 적으므로 학습 시간이 짧고, 필요할 때마다 수시로 갱신할 수 있습니다. 단점은 기존에 학습한 내용을 잊어버리는 위험이 있다는 것입니다. 이 문제가 바로 카타스트로픽 포겟팅입니다.

두 방식을 비교하면 다음과 같습니다.

```
모델 갱신 방식 비교

+------------------+------------------+------------------+
| 항목             | 주기적 재학습    | 증분 학습        |
+------------------+------------------+------------------+
| 학습 데이터      | 전체 누적 데이터 | 새 데이터만      |
| 학습 비용        | 높음             | 낮음             |
| 갱신 빈도        | 주기적 (주/월)   | 수시 가능        |
| 반영 속도        | 느림             | 빠름             |
| 과거 지식 유지   | 안정적           | 포겟팅 위험      |
| 오류 누적        | 없음 (매번 초기화)| 가능             |
+------------------+------------------+------------------+
```

이제 **Continual Learning(지속적 학습)**의 개념을 살펴보겠습니다. Continual Learning은 모델이 새로운 데이터를 학습하면서도 이전에 학습한 지식을 유지하는 것을 목표로 하는 학습 패러다임입니다. 증분 학습의 장점(빠른 갱신, 낮은 비용)은 유지하면서, 단점(지식 망각)을 극복하려는 연구 분야입니다.

Continual Learning이 해결해야 하는 핵심 과제는 **안정성-유연성 딜레마(stability-plasticity dilemma)**입니다. 모델이 새로운 정보를 잘 받아들이려면(유연성) 기존 파라미터를 크게 변경해야 하지만, 기존 정보를 유지하려면(안정성) 파라미터 변경을 최소화해야 합니다. 두 목표가 상충하는 것입니다. Continual Learning의 다양한 기법은 이 딜레마에서 적절한 균형점을 찾는 방법론입니다.

Continual Learning에서 가장 잘 알려진 문제가 **카타스트로픽 포겟팅(Catastrophic Forgetting)**입니다. 이 현상은 신경망이 새로운 데이터를 학습할 때, 이전에 학습한 내용을 급격히 잊어버리는 것을 뜻합니다. "카타스트로픽(catastrophic, 재앙적)"이라는 표현이 붙은 이유는, 잊어버리는 정도가 점진적이지 않고 급격하기 때문입니다.

카타스트로픽 포겟팅이 발생하는 원리를 이해하려면 신경망의 학습 방식을 돌아봐야 합니다. 신경망은 파라미터(가중치)의 조합으로 지식을 저장합니다. 특정 지식이 특정 파라미터 하나에 저장되는 것이 아니라, 여러 파라미터에 분산되어 인코딩됩니다. 문제는 서로 다른 지식이 같은 파라미터를 공유한다는 점입니다. A라는 지식을 인코딩하는 데 사용된 파라미터 일부가, B라는 지식을 인코딩하는 데도 사용됩니다. 새 데이터 B를 학습하면서 이 공유 파라미터들이 B에 맞게 변경되면, A에 대한 인코딩이 훼손됩니다. 결과적으로 B는 잘 수행하지만 A에 대한 성능은 급격히 떨어집니다.

LLM 파인튜닝에서 카타스트로픽 포겟팅이 나타나는 구체적인 상황을 예로 들겠습니다. 모델을 고객 상담 데이터로 파인튜닝했다고 합시다. 이 모델은 고객 질문에 잘 답합니다. 이후에 기술 문서 작성 데이터로 추가 파인튜닝하면, 기술 문서 작성 능력은 높아지지만 고객 상담 능력이 크게 저하될 수 있습니다. 심한 경우, 사전 학습에서 습득한 기본적인 언어 능력까지 훼손되어 문법 오류가 늘어나거나 의미 없는 텍스트를 생성하기도 합니다.

이제 카타스트로픽 포겟팅을 **완화하는 기법**들을 살펴보겠습니다.

첫 번째 기법은 **EWC(Elastic Weight Consolidation)**입니다. EWC의 핵심 아이디어는 "모든 파라미터가 동일하게 중요하지 않다"는 관찰에서 출발합니다. 이전 과제를 수행하는 데 중요한 파라미터는 크게 변하지 않도록 제약을 걸고, 덜 중요한 파라미터는 자유롭게 변하도록 허용하는 것입니다.

중요도를 어떻게 측정하느냐가 핵심입니다. EWC는 **피셔 정보 행렬(Fisher Information Matrix)**을 사용합니다. 피셔 정보 행렬은 각 파라미터가 이전 과제의 손실(loss)에 얼마나 민감하게 영향을 미치는지를 나타냅니다. 특정 파라미터를 조금만 바꿔도 이전 과제의 손실이 크게 변한다면, 그 파라미터는 이전 과제에 중요한 것입니다. 반대로 파라미터를 바꿔도 손실이 거의 변하지 않으면, 그 파라미터는 이전 과제에 덜 중요합니다.

EWC는 이 중요도를 학습의 손실 함수에 정규화 항으로 추가합니다. 새 과제의 손실을 최소화하면서, 동시에 중요 파라미터가 이전 값에서 크게 벗어나지 않도록 페널티를 줍니다. 이렇게 하면 이전 과제에 중요한 파라미터는 보존되면서, 덜 중요한 파라미터를 활용하여 새 과제를 학습합니다.

```
EWC 동작 원리

일반 학습 손실:
  L_new = 새 데이터에 대한 손실

EWC 손실:
  L_ewc = L_new + (lambda / 2) * sum(F_i * (theta_i - theta_old_i)^2)

  F_i        : 파라미터 i의 피셔 정보 (이전 과제 중요도)
  theta_i    : 현재 파라미터 값
  theta_old_i: 이전 학습 후 파라미터 값
  lambda     : 정규화 강도 (클수록 이전 지식 보존 강조)

효과:
  F_i가 큰 파라미터 --> 변화에 큰 페널티 --> 거의 고정
  F_i가 작은 파라미터 --> 변화에 작은 페널티 --> 자유롭게 갱신
```

두 번째 기법은 **리플레이(Replay)**입니다. 리플레이의 아이디어는 직관적입니다. 새 데이터를 학습할 때, 이전 데이터의 일부를 함께 섞어서 학습하는 것입니다. 모델이 새 데이터만 보면 이전 지식을 잊지만, 이전 데이터를 함께 보면 기억을 유지할 수 있습니다. 사람이 새로운 내용을 공부하면서 이전에 배운 내용을 간간이 복습하는 것과 같은 원리입니다.

리플레이에는 두 가지 변형이 있습니다. **경험 리플레이(Experience Replay)**는 이전 학습 데이터의 일부를 버퍼에 저장해 두었다가, 새 데이터 학습 시 버퍼에서 꺼내어 함께 학습합니다. 버퍼 크기가 제한되므로, 어떤 데이터를 버퍼에 유지할지 선택하는 전략이 필요합니다. 무작위로 선택하거나, 다양성을 극대화하는 방향으로 선택하거나, 모델이 어려워하는 데이터를 우선 유지하는 방식이 있습니다. **생성적 리플레이(Generative Replay)**는 이전 데이터를 직접 저장하는 대신, 이전 모델이 생성한 데이터를 사용합니다. 이전 모델에게 질문을 던져 응답을 생성하게 한 뒤, 이 질문-응답 쌍을 새 학습에 포함시킵니다. 저장 공간이 절약되지만, 생성된 데이터의 품질이 이전 모델의 능력에 의존한다는 한계가 있습니다.

```
리플레이 기법 비교

경험 리플레이:
  이전 데이터 [D1, D2, D3, ...] --> 버퍼에 일부 저장
  새 학습 시: 새 데이터 + 버퍼 데이터를 섞어 학습

생성적 리플레이:
  이전 모델에 질문 --> 이전 모델의 응답 수집
  새 학습 시: 새 데이터 + 생성된 (질문, 응답) 쌍을 섞어 학습
```

세 번째 기법은 **LoRA 기반 접근**입니다. 7.1.2에서 다룬 LoRA는 기본 모델 가중치를 고정하고 어댑터만 학습합니다. 이 특성이 포겟팅 완화에 자연스럽게 도움이 됩니다. 기본 모델의 가중치가 변하지 않으므로, 사전 학습에서 습득한 일반적인 언어 능력은 보존됩니다. 새로운 과제가 추가될 때마다 별도의 LoRA 어댑터를 학습하면, 과제 간 간섭 없이 각 과제에 특화된 어댑터를 유지할 수 있습니다. 다만 이 방식은 어댑터 수가 과제 수만큼 늘어나므로, 관리 복잡성이 증가합니다.

마지막으로 **모델 갱신 파이프라인 자동화**를 다루겠습니다. 수동으로 데이터를 모으고, 학습을 실행하고, 평가하고, 배포하는 과정을 매번 반복하면 실수와 지연이 발생합니다. 이 과정을 파이프라인으로 자동화하면 갱신 주기를 단축하고 일관성을 유지할 수 있습니다.

모델 갱신 자동화 파이프라인의 전형적인 구조는 다음과 같습니다.

```
모델 갱신 자동화 파이프라인

[1. 트리거]
  시간 기반: 매주 월요일
  데이터 기반: 새 데이터 1,000건 이상 축적 시
  성능 기반: 모니터링 지표가 임계값 이하로 하락 시
       |
       v
[2. 데이터 준비]
  새 데이터 수집 및 정제
  이전 데이터 버퍼 준비 (리플레이용)
  학습/검증/평가 데이터 분할
       |
       v
[3. 학습 실행]
  기존 모델 체크포인트 로드
  포겟팅 완화 기법 적용 (EWC, 리플레이 등)
  학습 실행 및 체크포인트 저장
       |
       v
[4. 평가]
  새 데이터 평가: 새 과제 성능 확인
  기존 데이터 평가: 이전 과제 성능 유지 확인
  안전성 평가: 유해 출력 발생 여부 확인
       |
       v
[5. 판정]
  평가 통과 --> 새 모델 배포
  평가 실패 --> 이전 모델 유지 + 원인 분석
```

이 파이프라인에서 핵심은 4단계의 평가입니다. 새 과제의 성능만 확인하면 안 되고, 이전 과제의 성능도 함께 확인해야 합니다. 카타스트로픽 포겟팅이 발생하면 새 과제의 성능은 올라가지만 이전 과제의 성능이 떨어지므로, 양쪽 모두를 평가해야 갱신의 전체 효과를 판단할 수 있습니다. 기존 과제 성능이 허용 임계값 이하로 떨어지면, 갱신을 롤백하고 완화 기법의 강도를 조정한 뒤 다시 시도합니다.

트리거 조건도 중요합니다. 시간 기반 트리거(매주, 매월)는 구현이 간단하지만, 새 데이터가 충분히 쌓이지 않았는데도 학습이 실행되어 자원이 낭비될 수 있습니다. 데이터 기반 트리거(새 데이터 N건 축적 시)는 데이터 양에 비례하여 갱신하므로 효율적이지만, 급한 지식 갱신이 필요할 때 대응이 느릴 수 있습니다. 성능 기반 트리거(모니터링 지표 하락 시)는 실제 문제가 발생했을 때 갱신하므로 가장 반응적이지만, 문제를 감지하는 모니터링 체계가 선행되어야 합니다. 실무에서는 이 세 가지를 조합하여 사용합니다.

정리하면, 모델 갱신은 데이터 드리프트, 지식 갱신, 성능 개선을 위해 필수적인 과정입니다. 주기적 재학습은 안정적이지만 비용이 높고, 증분 학습은 효율적이지만 카타스트로픽 포겟팅의 위험이 있습니다. Continual Learning은 새 지식을 학습하면서 기존 지식을 유지하는 것을 목표로 하며, EWC는 중요 파라미터를 보호하고, 리플레이는 이전 데이터를 함께 학습하며, LoRA 기반 접근은 기본 모델 가중치를 고정하여 포겟팅을 완화합니다. 이 과정을 자동화된 파이프라인으로 구성하고, 새 과제와 기존 과제 모두에 대해 평가를 수행하여 갱신의 품질을 보증합니다.

이 단원을 마치면 모델 갱신 전략을 수립하고 카타스트로픽 포겟팅을 완화할 수 있습니다.
