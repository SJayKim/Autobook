# 9.2.3 토큰 비용 최적화

LLM 서비스를 운영하면 비용이 빠르게 증가합니다. 개발 단계에서 하루 수십 건의 테스트 호출을 하는 것과, 프로덕션에서 하루 수만 건의 실제 요청을 처리하는 것은 비용 규모가 전혀 다릅니다. 9.2.2에서 구조화된 로그에 토큰 수와 비용을 기록하는 방법을 다루었습니다. 이 단원에서는 그 데이터를 바탕으로 비용 구조를 분석하고, 비용을 줄이는 구체적인 전략을 다룹니다.

LLM API의 과금 방식을 이해하려면 **토큰 기반 과금 구조**를 먼저 알아야 합니다. 대부분의 LLM API 제공자(OpenAI, Anthropic, Google 등)는 처리한 토큰 수에 비례하여 비용을 청구합니다. 토큰이란 모델이 텍스트를 처리하는 최소 단위이며, 1.2.2에서 다룬 토큰화 과정을 통해 텍스트가 토큰으로 변환됩니다. 과금 단위는 보통 100만 토큰(1M tokens)당 가격으로 표시됩니다.

여기서 중요한 점은 **입력 토큰과 출력 토큰의 단가가 다르다**는 것입니다. 대부분의 모델에서 출력 토큰이 입력 토큰보다 비쌉니다. 모델이 토큰을 생성하는 과정(디코딩)이 입력을 처리하는 과정(인코딩)보다 연산 비용이 높기 때문입니다. 예를 들어, 어떤 모델의 가격이 입력 1M 토큰당 3달러, 출력 1M 토큰당 15달러라면, 출력 토큰의 단가가 입력 토큰의 5배입니다.

이 구조에서 비용을 계산하는 공식은 다음과 같습니다.

```
요청 1건의 비용 = (입력 토큰 수 x 입력 단가) + (출력 토큰 수 x 출력 단가)

예시:
  모델: 입력 $3/1M tokens, 출력 $15/1M tokens
  요청: 입력 2,000 토큰, 출력 500 토큰

  비용 = (2,000 / 1,000,000 x 3) + (500 / 1,000,000 x 15)
       = 0.006 + 0.0075
       = $0.0135
```

한 건당 0.0135달러는 적어 보이지만, 하루 10만 건이면 1,350달러, 월 4만 달러가 넘습니다. 모델이 더 비싸거나, 입출력 토큰이 많거나, 호출 빈도가 높으면 비용은 급격히 올라갑니다.

비용을 최적화하려면 먼저 "비용이 어디서 발생하는지"를 파악해야 합니다. 이것이 **입력/출력 토큰 비용 분석**입니다. 9.1.2에서 LangSmith의 비용 추적 기능을, 9.1.3에서 Langfuse의 비용 분석 대시보드를 소개했습니다. 이 도구들은 각 트레이스의 토큰 사용량과 비용을 자동으로 기록합니다. 이 데이터를 분석하면 다음과 같은 질문에 답할 수 있습니다.

"전체 비용 중 입력 토큰과 출력 토큰의 비율은 얼마인가?" 입력 비중이 높다면 프롬프트를 줄이는 전략이 효과적이고, 출력 비중이 높다면 max_tokens를 제한하거나 응답 형식을 간결하게 유도하는 전략이 효과적입니다. "어떤 기능(엔드포인트)이 비용을 가장 많이 차지하는가?" 요약 기능이 전체의 60%를 차지한다면, 그 기능에 최적화를 집중하는 것이 전체 비용 절감에 가장 큰 효과를 냅니다. "비용이 급증한 시점이 있는가? 원인은 무엇인가?" 특정 날짜에 비용이 평소의 3배가 되었다면, 트래픽 급증인지, 프롬프트 변경으로 토큰이 늘어난 것인지를 추적합니다.

이런 분석을 일상적으로 수행하려면 **비용 모니터링 대시보드**가 필요합니다. 대시보드에 포함해야 할 핵심 지표는 다음과 같습니다. 일별/주별/월별 총비용 추이, 모델별 비용 분포, 기능별(엔드포인트별) 비용 분포, 평균 요청당 비용, 사용자별 비용 상위 목록입니다. 9.1.2와 9.1.3에서 다룬 LangSmith, Langfuse가 이러한 대시보드를 제공하며, Grafana와 Prometheus를 조합하여 직접 구축할 수도 있습니다.

비용 구조를 파악했으면, 이제 비용을 줄이는 전략을 적용합니다.

첫 번째 전략은 **프롬프트 압축**(prompt compression)입니다. 입력 토큰 수를 줄여 입력 비용을 직접 절감하는 방법입니다. 프롬프트에는 시스템 프롬프트, few-shot 예시, RAG로 검색된 컨텍스트 문서, 대화 이력 등이 포함됩니다. 이 중 불필요하게 긴 부분을 줄이면 토큰 수가 감소합니다.

프롬프트 압축의 구체적인 기법을 살펴보겠습니다. 첫째, 시스템 프롬프트를 간결하게 다듬습니다. 같은 지시를 반복하거나, 지나치게 상세한 예외 조건을 나열하는 경우가 있습니다. 핵심 지시만 남기고 불필요한 문구를 제거합니다. 둘째, RAG 컨텍스트의 양을 조절합니다. 검색된 문서 전문을 넣는 대신, 관련성이 높은 부분만 발췌하여 삽입합니다. 셋째, 대화 이력을 요약합니다. 멀티턴 대화에서 이전 대화를 전부 포함하면 토큰이 급증합니다. 최근 N턴만 포함하거나, 이전 대화를 짧게 요약하여 포함하는 방식으로 토큰을 절약합니다. 넷째, LLMLingua 같은 자동 프롬프트 압축 도구를 활용합니다. 이 도구들은 프롬프트에서 모델의 이해에 크게 영향을 미치지 않는 토큰을 자동으로 제거하여, 응답 품질을 유지하면서 입력 길이를 줄입니다.

```
프롬프트 압축 전후 비교 예시

[압축 전: 3,200 토큰]
시스템 프롬프트: 800 토큰 (상세 지시, 반복 포함)
대화 이력: 1,200 토큰 (전체 10턴)
RAG 컨텍스트: 1,000 토큰 (문서 3개 전문)
사용자 질문: 200 토큰

[압축 후: 1,600 토큰]
시스템 프롬프트: 400 토큰 (핵심만 유지)
대화 이력: 300 토큰 (이전 대화 요약 + 최근 2턴)
RAG 컨텍스트: 700 토큰 (관련 구절만 발췌)
사용자 질문: 200 토큰

절감 효과: 입력 토큰 50% 감소 --> 입력 비용 50% 절감
```

두 번째 전략은 **캐싱으로 비용 절감**입니다. 동일하거나 유사한 질문이 반복적으로 들어올 때, 모델을 다시 호출하지 않고 이전 응답을 재사용하면 비용을 아낄 수 있습니다. 캐싱에는 두 가지 수준이 있습니다.

**정확 캐싱**(exact caching)은 프롬프트가 정확히 일치하는 경우에만 캐시를 적중시킵니다. 프롬프트의 해시값을 키로 사용하여, 동일한 프롬프트에 대한 이전 응답을 저장해 두고 반환합니다. 구현이 단순하고 응답 정확도에 영향이 없다는 장점이 있습니다. 다만 프롬프트가 한 글자라도 다르면 캐시가 적중하지 않으므로, 적중률이 낮을 수 있습니다.

**시맨틱 캐싱**(Semantic Cache)은 프롬프트의 의미적 유사성을 기준으로 캐시를 적중시킵니다. "서울의 날씨 알려줘"와 "서울 지금 날씨 어때"는 문자열은 다르지만 의미는 같습니다. 시맨틱 캐시는 프롬프트를 임베딩 벡터로 변환하고, 벡터 유사도가 설정한 임계값(예: 0.95) 이상이면 캐시를 적중시킵니다. 정확 캐싱보다 적중률이 높지만, 임베딩 생성과 유사도 검색에 추가 비용이 들고, 유사하지만 다른 의도의 질문에 잘못된 캐시를 반환할 위험이 있습니다. 임계값을 높게 설정하면 안전하지만 적중률이 떨어지고, 낮게 설정하면 적중률은 높지만 부정확한 응답을 반환할 가능성이 커집니다.

```
정확 캐싱 vs 시맨틱 캐싱

정확 캐싱:
  키: hash("서울의 날씨 알려줘") = "a3f2..."
  "서울의 날씨 알려줘" --> 캐시 적중 (O)
  "서울 지금 날씨 어때" --> 캐시 미적중 (X, 해시 다름)

시맨틱 캐싱:
  키: embedding("서울의 날씨 알려줘")
  "서울의 날씨 알려줘" --> 유사도 1.0 --> 캐시 적중 (O)
  "서울 지금 날씨 어때" --> 유사도 0.97 --> 캐시 적중 (O, 임계값 0.95 이상)
  "부산의 맛집 추천해줘" --> 유사도 0.42 --> 캐시 미적중 (X)
```

세 번째 전략은 **모델 티어링**(model tiering)입니다. 모든 요청에 가장 크고 비싼 모델을 사용할 필요는 없습니다. 간단한 질문에는 저렴한 소형 모델을 사용하고, 복잡한 질문에만 고가의 대형 모델을 사용하면 전체 비용을 크게 줄일 수 있습니다. 이 전략을 "라우팅으로 비용 분배"라고도 합니다.

모델 티어링을 구현하는 방법은 다음과 같습니다. 먼저 요청의 복잡도를 판단하는 **라우터**(router)를 둡니다. 라우터가 들어온 요청을 분석하여, 간단한 요청은 소형 모델(예: GPT-4o-mini)로, 복잡한 요청은 대형 모델(예: GPT-4o)로 보냅니다. 라우터의 판단 기준은 규칙 기반(키워드, 입력 길이, 요청 유형)일 수도 있고, 별도의 분류 모델을 사용할 수도 있습니다.

예를 들어, 고객 지원 챗봇에서 단순 FAQ("영업 시간이 몇 시인가요?")는 소형 모델로 충분하고, 복잡한 기술 문의("API 인증 오류 401이 발생하는데 원인과 해결 방법을 알려주세요")는 대형 모델이 필요할 수 있습니다. 전체 요청의 70%가 단순 FAQ라면, 이 70%를 소형 모델로 처리하는 것만으로도 비용을 절반 이하로 줄일 수 있습니다.

```
모델 티어링 흐름

[요청 도착]
    |
    v
[라우터 (복잡도 판단)]
    |
    +---> 단순 (70%) ---> [소형 모델: $0.15/1M tokens] ---> 응답
    |
    +---> 복잡 (30%) ---> [대형 모델: $3.00/1M tokens] ---> 응답

비용 비교:
  모든 요청에 대형 모델 사용: 100% x $3.00 = $3.00 (기준)
  티어링 적용: (70% x $0.15) + (30% x $3.00) = $0.105 + $0.90 = $1.005
  절감률: 약 67%
```

9.3.3에서 다룰 API Gateway의 다중 모델 라우팅 기능과 결합하면, 이러한 티어링을 중앙에서 일괄 관리할 수 있습니다.

마지막으로 **예산 관리와 할당**을 다루겠습니다. 아무리 최적화를 하더라도, 예상치 못한 트래픽 급증이나 설정 오류로 비용이 폭증할 수 있습니다. 이를 방지하기 위해 예산 한도(budget cap)를 설정합니다.

예산 관리의 첫 번째 수준은 **전체 예산 한도**입니다. 월간 총비용이 설정한 한도를 초과하면 추가 요청을 차단하거나 저렴한 모델로 자동 전환합니다. 두 번째 수준은 **팀별/프로젝트별 할당**입니다. 조직 내 여러 팀이나 프로젝트가 LLM API를 공유할 때, 각각에 예산을 배분합니다. 한 팀이 예산을 소진해도 다른 팀의 서비스에는 영향이 없도록 합니다. 세 번째 수준은 **사용자별 한도**입니다. 개별 사용자가 일정 기간 동안 사용할 수 있는 토큰 수나 비용을 제한합니다. 한 사용자의 과도한 사용이 전체 예산을 소모하는 것을 방지합니다.

9.1.3에서 다룬 Langfuse나, 9.3.3에서 다룰 LiteLLM 같은 도구는 사용자별, 팀별 예산 한도 설정 기능을 제공합니다. 한도에 도달하면 요청을 차단하거나 알림을 보내는 것을 자동화할 수 있습니다.

```
예산 관리 계층

[전체 예산: 월 $10,000]
    |
    +--- [팀 A: 월 $5,000]
    |       +--- [프로젝트 1: 월 $3,000]
    |       +--- [프로젝트 2: 월 $2,000]
    |
    +--- [팀 B: 월 $3,000]
    |
    +--- [팀 C: 월 $2,000]
            +--- [사용자별: 일 $10]
```

정리하면, LLM 비용 최적화는 과금 구조의 이해에서 시작합니다. 입력/출력 토큰의 단가 차이를 파악하고, 모니터링 대시보드로 비용 발생 지점을 분석합니다. 프롬프트 압축으로 입력 토큰을 줄이고, 시맨틱 캐싱으로 반복 호출을 제거하며, 모델 티어링으로 요청 복잡도에 맞는 모델을 사용하는 세 가지 전략이 핵심입니다. 여기에 전체, 팀별, 사용자별 예산 한도를 설정하여 비용 폭증을 방지합니다.

다음 단원인 9.3.1에서는 LLM 애플리케이션의 CI/CD를 다룹니다. 프롬프트와 모델의 변경을 안전하게 배포하기 위한 파이프라인 설계를 살펴봅니다.

이 단원을 마치면 LLM API 비용 구조를 분석하고 비용 최적화 전략을 수립할 수 있습니다.
