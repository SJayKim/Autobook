# 4.1.1 RAG의 개념과 필요성

2.1.1에서 프롬프트 엔지니어링을 다루면서, LLM에 전달하는 입력 텍스트가 출력 품질을 좌우한다는 것을 확인했습니다. 3.2.1에서는 벡터 데이터베이스가 임베딩 벡터를 저장하고 유사한 벡터를 빠르게 검색하는 시스템이라는 것을 살펴보았습니다. 이 두 가지, 즉 프롬프트와 벡터 검색을 결합하면 LLM의 근본적인 한계를 극복하는 강력한 패턴이 만들어집니다. 이 단원에서는 그 패턴인 RAG(Retrieval-Augmented Generation)의 개념, 등장 배경, 동작 흐름을 다룹니다.

LLM은 대량의 텍스트 데이터를 사전학습하여 언어를 이해하고 생성하는 능력을 갖추고 있습니다. 그런데 이 사전학습에는 시간적 한계가 있습니다. 모델의 학습 데이터에는 특정 시점까지의 정보만 포함됩니다. 학습이 완료된 이후에 발생한 사건, 갱신된 문서, 새로 제정된 규정은 모델이 알 수 없습니다. 사용자가 "올해 개정된 연차 규정은 무엇인가?"라고 질문하면, 모델은 학습 시점의 오래된 정보로 답하거나, 존재하지 않는 내용을 그럴듯하게 지어내어 답할 수 있습니다.

이렇게 모델이 사실이 아닌 내용을 사실인 것처럼 생성하는 현상을 **할루시네이션**(hallucination)이라 합니다. 할루시네이션은 LLM의 구조적 특성에서 비롯됩니다. 1.2.3에서 다룬 것처럼, LLM은 입력 토큰 시퀀스에 이어 올 확률이 높은 토큰을 생성합니다. 모델은 "사실인지 여부"를 검증하는 메커니즘이 없고, 통계적으로 자연스러운 텍스트를 만들어낼 뿐입니다. 학습 데이터에 해당 정보가 없으면, 비슷한 패턴의 텍스트를 조합하여 그럴듯한 답을 만들어 냅니다. 이것이 할루시네이션의 원인입니다.

할루시네이션 외에도 **지식 한계**라는 문제가 있습니다. LLM의 학습 데이터는 공개된 웹 문서, 위키피디아, 서적 등을 주로 포함합니다. 특정 조직의 사내 문서, 비공개 데이터베이스, 도메인 전문 자료는 포함되지 않는 경우가 대부분입니다. 따라서 "우리 회사의 환불 정책은 무엇인가?"와 같은 조직 내부 지식에 기반한 질문에는 정확히 답할 수 없습니다.

이 두 가지 문제, 시간적 한계로 인한 할루시네이션과 조직 고유 지식의 부재를 해결하기 위한 접근 방식이 **RAG(Retrieval-Augmented Generation)**입니다. RAG라는 이름은 세 단어의 머리글자를 딴 것입니다. **Retrieval**(검색)은 외부 지식 저장소에서 관련 정보를 찾아오는 것이고, **Augmented**(증강)는 찾아온 정보를 프롬프트에 덧붙이는 것이며, **Generation**(생성)은 이렇게 보강된 프롬프트를 바탕으로 LLM이 응답을 생성하는 것입니다.

RAG의 정의를 한 문장으로 표현하면 다음과 같습니다. RAG는 사용자의 질문에 관련된 외부 문서를 검색하여 LLM의 프롬프트에 포함시키고, LLM이 이 문서를 근거로 응답을 생성하도록 하는 패턴입니다.

RAG의 핵심 흐름은 세 단계로 이루어집니다. **Retrieve**(검색), **Augment**(증강), **Generate**(생성)입니다. 각 단계를 구체적으로 살펴보겠습니다.

첫 번째 단계인 Retrieve에서는 사용자의 질문을 받아 외부 지식 저장소에서 관련 문서 조각을 검색합니다. 여기서 외부 지식 저장소는 주로 벡터 데이터베이스입니다. 3.2.1에서 다룬 것처럼, 문서를 임베딩 벡터로 변환하여 벡터 데이터베이스에 저장해 두면, 사용자의 질문도 같은 임베딩 모델로 벡터화한 뒤 유사도 검색으로 관련 문서 조각을 찾을 수 있습니다. 3.1.1에서 설명한 코사인 유사도를 기준으로 가장 유사한 문서 조각 몇 개를 가져옵니다.

두 번째 단계인 Augment에서는 검색된 문서 조각을 LLM에 전달할 프롬프트에 삽입합니다. 2.1.3에서 시스템 프롬프트의 구조를 다루면서, 프롬프트 안에 맥락(context)을 배치하는 방법을 살펴보았습니다. RAG에서는 이 맥락 자리에 검색된 문서 조각이 들어갑니다. "아래 문서를 참고하여 질문에 답하시오"와 같은 지시와 함께, 검색된 문서 텍스트를 프롬프트에 포함시킵니다.

세 번째 단계인 Generate에서는 프롬프트를 LLM에 전달하여 응답을 생성합니다. LLM은 프롬프트에 포함된 문서 조각을 근거로 답변을 만듭니다. 모델의 사전학습 지식이 아니라, 방금 검색한 최신 문서에 기반하여 답하므로 할루시네이션이 줄어들고, 조직 고유의 지식도 활용할 수 있습니다.

이 세 단계의 흐름을 시각화하면 다음과 같습니다.

```
RAG의 핵심 흐름 (Retrieve -> Augment -> Generate)

  사용자 질문: "우리 회사의 환불 정책은?"
       |
       v
  [1. Retrieve - 검색]
  질문을 벡터로 변환
       |
       v
  벡터 데이터베이스에서 유사 문서 검색
  결과: "환불은 구매 후 7일 이내..."
        "전자제품은 개봉 시 환불 불가..."
       |
       v
  [2. Augment - 증강]
  프롬프트 구성:
    "다음 문서를 참고하여 질문에 답하시오.
     문서1: 환불은 구매 후 7일 이내...
     문서2: 전자제품은 개봉 시 환불 불가...
     질문: 우리 회사의 환불 정책은?"
       |
       v
  [3. Generate - 생성]
  LLM이 문서를 근거로 응답 생성:
    "회사의 환불 정책은 구매 후 7일 이내에
     환불을 신청할 수 있으며, 전자제품의 경우
     개봉 시 환불이 불가합니다."
```

이 흐름에서 LLM은 자신의 사전학습 지식이 아니라, 검색된 문서에 기반하여 답합니다. 문서가 최신이면 답변도 최신이고, 문서가 조직의 사내 자료이면 답변도 조직 고유의 정보를 반영합니다.

RAG가 왜 LLM의 할루시네이션과 지식 한계를 해결할 수 있는지 이해했으니, RAG 외에 이 문제를 해결하는 또 다른 접근 방식인 **파인튜닝**(fine-tuning)과 비교해 보겠습니다. 파인튜닝은 사전학습된 LLM을 특정 데이터로 추가 학습시켜 모델 자체의 지식을 갱신하는 방법입니다.

```
RAG와 파인튜닝의 비교

비교 항목            RAG                         파인튜닝
----------------------------------------------------------------------
지식 주입 방식       검색 시점에 외부 문서 제공   모델 가중치에 학습
지식 갱신            문서 갱신만으로 즉시 반영    재학습 필요 (시간, 비용)
비용                 검색 인프라 + API 비용       GPU 학습 비용
환각 제어            출처가 있으므로 검증 가능    학습 데이터 품질에 의존
적합한 상황          최신 정보, 사실 기반 답변    특정 스타일, 형식, 도메인 적응
```

RAG는 외부 문서를 검색하여 프롬프트에 넣는 방식이므로, 모델 자체를 변경하지 않습니다. 문서를 업데이트하면 즉시 답변에 반영되고, 답변의 근거가 되는 문서를 특정할 수 있어 출처 추적이 가능합니다. 반면 파인튜닝은 모델의 가중치를 직접 수정하므로, 특정 도메인의 어휘나 문체, 또는 특수한 출력 형식에 적응시키는 데 효과적입니다. 그러나 새로운 정보를 반영하려면 모델을 다시 학습시켜야 하고, 학습에는 GPU 자원과 시간이 필요합니다.

실무에서는 RAG와 파인튜닝을 상호 배타적으로 보지 않고, 목적에 따라 병행하기도 합니다. 예를 들어, 파인튜닝으로 모델이 의학 용어를 정확하게 사용하도록 적응시킨 뒤, RAG로 최신 의학 논문을 검색하여 답변의 근거로 사용하는 조합이 가능합니다.

RAG가 활용되는 대표적인 적용 사례를 살펴보겠습니다. 첫 번째는 사내 지식 기반 질의응답 시스템입니다. 사내 매뉴얼, 정책 문서, FAQ를 벡터 데이터베이스에 저장해 두고, 직원이 질문하면 관련 문서를 검색하여 답변을 생성합니다. 두 번째는 고객 지원 챗봇입니다. 제품 설명서, 트러블슈팅 가이드를 검색하여 고객의 문의에 정확한 답변을 제공합니다. 세 번째는 법률이나 의료 분야의 전문 지식 검색입니다. 법률 조항이나 의학 논문을 검색하여 전문적인 질의에 근거 있는 답변을 생성합니다.

이 사례들의 공통점은 "정확한 정보에 기반한 답변"이 중요하다는 것입니다. 사내 정책에 대한 잘못된 답변은 업무 혼란을 초래하고, 법률 자문에서의 할루시네이션은 심각한 결과를 낳을 수 있습니다. RAG는 답변의 근거를 명시할 수 있으므로, 이런 환경에서 LLM을 안전하게 활용하는 핵심 패턴이 됩니다.

마지막으로 **Naive RAG의 한계**를 짚어 보겠습니다. 앞서 설명한 기본적인 RAG 흐름, 즉 질문을 벡터로 변환하고, 유사도 검색으로 문서를 가져오고, 프롬프트에 넣어 답변을 생성하는 것을 **Naive RAG**(기본 RAG)라 부릅니다. Naive RAG는 RAG의 가장 단순한 형태로, 구현이 간단하고 많은 경우에 효과적이지만 몇 가지 한계가 있습니다.

첫 번째 한계는 검색 품질의 문제입니다. 벡터 유사도 검색은 의미적으로 비슷한 문서를 찾지만, "비슷한 문서"가 항상 "정답을 담고 있는 문서"인 것은 아닙니다. 3.1.1에서 코사인 유사도가 벡터의 방향만 비교한다고 했는데, 질문과 방향이 비슷하지만 실제로는 다른 주제의 문서가 상위에 올라올 수 있습니다. 또한 고유명사나 기술 용어처럼 정확한 단어 일치가 중요한 경우에는 시맨틱 검색만으로는 부족합니다.

두 번째 한계는 청킹 품질의 문제입니다. 문서를 적절한 크기의 조각으로 나누는 청킹(chunking) 과정에서, 문맥이 끊기거나 관련 정보가 여러 청크에 분산될 수 있습니다. 사용자의 질문에 대한 답이 하나의 청크에 완전히 담기지 않으면, 검색된 청크만으로는 불완전한 답변이 생성됩니다.

세 번째 한계는 모델이 검색 결과를 제대로 활용하지 못하는 경우입니다. 검색된 문서가 관련성이 높더라도, 프롬프트에 포함된 여러 청크 중 어느 부분이 질문에 대한 핵심 답변인지를 모델이 정확히 식별하지 못할 수 있습니다. 특히 검색된 청크 수가 많거나 내용이 서로 모순될 때 이 문제가 두드러집니다.

이러한 한계를 극복하기 위해, 하이브리드 검색, 리랭킹, 쿼리 변환, 청킹 전략 최적화 등 다양한 개선 기법이 발전해 왔습니다. 이 기법들은 이후 단원에서 순서대로 다룹니다.

정리하면, RAG는 외부 지식 저장소에서 관련 문서를 검색하여 LLM의 프롬프트에 포함시키고, 모델이 이를 근거로 응답을 생성하는 패턴입니다. LLM의 할루시네이션과 지식 한계를 해결하기 위해 등장했으며, 모델을 재학습시키지 않고 문서 갱신만으로 최신 정보를 반영할 수 있습니다. 핵심 흐름은 Retrieve(검색), Augment(증강), Generate(생성)의 세 단계이며, 파인튜닝과는 상호 보완적인 관계에 있습니다. 기본적인 Naive RAG는 구현이 간단하지만, 검색 품질, 청킹, 모델의 정보 활용 측면에서 한계가 있어 추가적인 최적화가 필요합니다.

다음 단원인 4.1.2에서는 기본 RAG 파이프라인 구축을 다룹니다. 문서 로딩부터 응답 생성까지 실제 동작하는 RAG 파이프라인을 단계별로 구현하는 방법을 살펴봅니다.

이 단원을 마치면 RAG의 동작 원리와 LLM 단독 사용 대비 이점을 설명할 수 있습니다.
