# 6.1.1 모델 서빙 아키텍처의 이해

대규모 언어 모델을 학습시키는 것과 실제 사용자에게 응답을 돌려주는 것은 전혀 다른 문제입니다. 학습은 대량의 데이터를 수 시간에서 수 주에 걸쳐 처리하면 되지만, 사용자 요청에 응답하려면 몇 초 이내에 결과를 내놓아야 합니다. 챗봇에 질문을 보냈는데 30초 뒤에야 첫 글자가 나타난다면 서비스로서 쓸 수 없습니다. 모델을 학습하는 단계와 실제 요청에 답하는 단계 사이에는 이 간극을 메우는 별도의 시스템이 필요합니다. 이 시스템을 설계하고 운영하는 영역이 바로 **모델 서빙(Model Serving)**입니다.

모델 서빙이란, 학습이 완료된 모델을 메모리에 올려 두고, 외부에서 들어오는 요청을 받아 모델에 전달한 뒤, 모델이 생성한 결과를 요청자에게 돌려주는 전체 과정을 말합니다. 웹 애플리케이션에서 데이터베이스 서버가 쿼리를 받아 결과를 반환하는 것과 구조가 비슷합니다. 차이는, 데이터베이스 서버 대신 수십억 개의 파라미터를 가진 모델이 그 자리를 차지한다는 점입니다. 모델 서빙의 목적은 크게 세 가지입니다. 첫째, 사용자가 모델의 내부 구조나 실행 환경을 알 필요 없이 표준화된 인터페이스(주로 HTTP API)로 모델에 접근할 수 있게 합니다. 둘째, 동시에 여러 요청이 들어와도 안정적으로 처리할 수 있도록 자원을 관리합니다. 셋째, 응답 시간과 처리량 사이의 균형을 맞추어 운영 비용을 통제합니다.

이 단원에서는 모델 서빙 시스템이 어떤 구성요소로 이루어지는지, 요청이 들어왔을 때 내부에서 어떤 흐름을 거치는지, 그리고 서빙 성능을 측정하는 핵심 지표들을 다룹니다. 1.2.3에서 다룬 추론 과정(Prefill 단계와 Decode 단계, KV 캐시, 자기회귀 생성)을 기반으로, 이 추론 과정을 실제 서비스 환경에서 어떻게 감싸고 운영하는지를 살펴봅니다.

모델 서빙 시스템의 내부 구조를 이해하기 위해, 사용자 요청이 들어와서 응답이 나가기까지의 전체 흐름을 단계별로 따라가 보겠습니다. 이 흐름은 크게 세 단계로 나뉩니다. **모델 로딩**, **요청 처리**, **응답 반환**입니다.

**모델 로딩**은 서빙 시스템이 시작될 때 가장 먼저 일어나는 단계입니다. 학습이 끝난 모델의 가중치 파일은 보통 디스크에 저장되어 있습니다. 70억 개의 파라미터를 가진 모델이 16비트 부동소수점(FP16)으로 저장되어 있다면, 파일 크기는 약 14GB입니다. 모델 로딩이란 이 가중치 파일을 디스크에서 읽어 GPU 메모리(VRAM)에 올리는 과정입니다. GPU 메모리에 모델이 올라가야 비로소 추론 연산을 실행할 수 있습니다. 모델 크기가 GPU 한 장의 메모리 용량을 초과하면 여러 GPU에 나누어 올려야 하는데, 이를 모델 병렬화라고 합니다. 모델 로딩은 서버 시작 시 한 번 수행되며, 모델 크기에 따라 수십 초에서 수 분이 걸릴 수 있습니다. 로딩이 완료되면 서빙 시스템은 요청을 받을 준비가 된 상태가 됩니다.

**요청 처리**는 실제 추론이 이루어지는 핵심 단계입니다. 사용자가 HTTP API를 통해 프롬프트를 보내면, 서빙 시스템은 이 텍스트를 토큰으로 변환합니다. 1.2.2에서 다룬 토크나이저가 이 역할을 합니다. 토큰화된 입력은 GPU에 올라가 있는 모델에 전달됩니다. 모델은 1.2.3에서 다룬 것처럼 Prefill 단계에서 입력 토큰 전체를 한꺼번에 처리하여 KV 캐시를 생성하고, 이어서 Decode 단계에서 토큰을 하나씩 자기회귀적으로 생성합니다. 이때 서빙 시스템이 단순히 요청 하나를 모델에 넣고 끝날 때까지 기다리기만 한다면 효율이 매우 낮습니다. 실제 서빙 시스템은 여러 요청을 한데 모아 처리하는 **배칭(batching)** 기법을 사용합니다. 동시에 들어온 여러 사용자의 입력을 하나의 배치로 묶어 GPU에 한꺼번에 넣으면, GPU의 병렬 연산 능력을 더 효율적으로 활용할 수 있습니다.

**응답 반환**은 모델이 생성한 토큰을 다시 텍스트로 변환하여 사용자에게 돌려보내는 단계입니다. 토크나이저가 토큰 ID 시퀀스를 원래의 텍스트 문자열로 디코딩합니다. 응답 방식에는 두 가지가 있습니다. 하나는 모든 토큰 생성이 끝난 뒤 전체 텍스트를 한꺼번에 반환하는 방식입니다. 다른 하나는 **스트리밍(streaming)** 방식으로, 토큰이 생성될 때마다 즉시 사용자에게 보내는 것입니다. ChatGPT에서 글자가 한 글자씩 나타나는 것이 스트리밍 방식의 예입니다. 스트리밍은 사용자가 첫 글자를 보기까지 기다리는 시간을 줄여 주므로, 대화형 서비스에서 널리 사용됩니다.

아래 다이어그램은 이 세 단계의 전체 흐름을 정리한 것입니다.

```
[서버 시작]
  |
  v
[모델 로딩] 디스크 --> GPU 메모리에 가중치 적재
  |
  v
[요청 대기] <-- HTTP API 엔드포인트 개방
  |
  v (사용자 요청 도착)
[요청 처리]
  (1) 텍스트 --> 토크나이저 --> 토큰 ID
  (2) 토큰 ID --> 모델 추론 (Prefill --> Decode)
  (3) 배칭: 여러 요청을 묶어 GPU에 전달
  |
  v
[응답 반환]
  (1) 토큰 ID --> 토크나이저 --> 텍스트
  (2) 전체 반환 또는 스트리밍 반환
  |
  v
[사용자에게 응답 전달]
```

모델 서빙에는 사용 목적에 따라 두 가지 방식이 있습니다. **온라인 서빙(online serving)**과 **배치 서빙(batch serving)**입니다. 이 둘은 요청을 처리하는 시점과 응답 속도에 대한 요구 수준이 다릅니다.

온라인 서빙은 사용자 요청이 도착하는 즉시 처리하여 결과를 반환하는 방식입니다. 챗봇, 검색 보조, 실시간 번역 같은 서비스가 이에 해당합니다. 사용자가 메시지를 보내면 수 초 이내에 응답이 돌아와야 하므로, 낮은 지연 시간(레이턴시)이 핵심 요구사항입니다. 온라인 서빙에서는 GPU가 항상 켜져 있어야 하고, 요청 폭주(트래픽 급증)에 대비한 오토스케일링(자동 확장)도 필요합니다. 운영 비용이 높지만, 사용자 경험이 직결되는 서비스에서는 필수적인 방식입니다.

배치 서빙은 여러 건의 요청을 모아 두었다가 한꺼번에 처리하는 방식입니다. 대량의 문서를 요약하거나, 수천 건의 이메일에 분류 레이블을 붙이거나, 데이터셋 전체에 대해 모델 평가를 수행하는 경우가 이에 해당합니다. 개별 요청의 응답 시간은 중요하지 않습니다. 대신 전체 작업을 가능한 한 적은 비용으로, 높은 처리량(throughput)으로 끝내는 것이 목표입니다. 배치 서빙에서는 GPU를 필요한 시간만큼만 사용하고 끌 수 있으므로 비용 효율이 높습니다. 또한 요청들을 큰 배치로 묶을 수 있어 GPU 활용률도 극대화됩니다.

```
온라인 서빙                          배치 서빙
------------------------------      ------------------------------
요청 도착 즉시 처리                  요청을 모아 두었다가 일괄 처리
낮은 레이턴시 필수 (수 초 이내)      레이턴시 제약 완화 (수 분~수 시간)
GPU 상시 가동                        필요 시에만 GPU 사용
트래픽 변동 대응 필요                처리량(throughput) 극대화 목표
예: 챗봇, 실시간 번역                예: 문서 요약, 데이터셋 평가
```

실무에서는 하나의 서비스가 두 가지 방식을 모두 사용하기도 합니다. 사용자가 직접 대화하는 부분은 온라인 서빙으로, 야간에 쌓인 로그를 분석하는 부분은 배치 서빙으로 처리하는 식입니다.

모델 서빙 시스템의 성능을 평가할 때는 여러 지표를 사용합니다. 그중에서도 레이턴시, 즉 사용자가 요청을 보내고 응답을 받기까지 걸리는 시간을 구성하는 세부 지표가 특히 중요합니다. LLM 서빙의 레이턴시는 일반적인 웹 서비스의 레이턴시와 성격이 다릅니다. 일반적인 API는 요청을 보내면 응답 전체가 한꺼번에 돌아오므로, 전체 응답 시간 하나로 성능을 판단할 수 있습니다. 하지만 LLM은 1.2.3에서 다룬 것처럼 자기회귀 방식으로 토큰을 하나씩 생성합니다. 스트리밍 환경에서는 첫 토큰이 나오는 시점, 토큰이 생성되는 속도, 전체 응답이 완료되는 시점이 모두 따로 측정됩니다. 이 세 가지를 하나씩 살펴보겠습니다.

**TTFT(Time To First Token)**는 사용자가 요청을 보낸 시점부터 첫 번째 토큰이 반환되기까지 걸리는 시간입니다. 사용자 입장에서는 "내 질문을 이해하고 답변을 시작했구나"라고 느끼는 시점이 바로 TTFT입니다. TTFT에는 네트워크 전송 시간, 토큰화 시간, 그리고 Prefill 단계의 연산 시간이 포함됩니다. 이 중에서 Prefill 단계가 가장 큰 비중을 차지합니다. 입력 프롬프트가 길수록 Prefill에서 처리해야 할 토큰이 많아지므로 TTFT가 길어집니다. 대화형 서비스에서 TTFT가 길면 사용자는 시스템이 멈춘 것으로 오해할 수 있으므로, 온라인 서빙에서 TTFT는 핵심 관리 대상입니다.

**TPS(Tokens Per Second)**는 Decode 단계에서 초당 생성되는 토큰 수입니다. TPOT(Time Per Output Token)라는 이름으로 토큰 하나를 생성하는 데 걸리는 시간으로 표현하기도 합니다. TPS가 40이라면 초당 40개의 토큰을 생성한다는 뜻이고, TPOT으로 환산하면 토큰 하나당 25밀리초(1000ms / 40)입니다. 사람이 텍스트를 읽는 속도는 대략 초당 5~8단어 정도입니다. 한국어 기준으로 하나의 단어가 대략 2~3개의 토큰에 해당한다고 보면, TPS가 20 이상이면 사용자가 읽는 속도보다 빠르게 생성되어 체감상 대기 없이 텍스트가 흘러가는 느낌을 줍니다. TPS가 너무 낮으면 스트리밍 중에도 글자가 끊겨 보이는 현상이 나타납니다. TPS는 모델 크기, GPU 성능, 배치 크기, 그리고 KV 캐시의 메모리 대역폭에 의해 좌우됩니다.

**E2E 레이턴시(End-to-End Latency)**는 요청을 보낸 시점부터 마지막 토큰이 반환되기까지의 전체 소요 시간입니다. 수식으로 표현하면 다음과 같습니다.

```
E2E 레이턴시 = TTFT + (생성 토큰 수 - 1) x TPOT
```

첫 번째 토큰까지의 시간(TTFT)에, 나머지 토큰들을 생성하는 시간을 더한 것입니다. 예를 들어 TTFT가 200ms이고, TPOT가 25ms이며, 생성할 토큰이 100개라면, E2E 레이턴시는 200 + 99 x 25 = 2,675ms, 약 2.7초입니다. E2E 레이턴시는 스트리밍이 아닌 전체 응답을 한꺼번에 받는 경우에 특히 중요합니다. 배치 서빙에서는 개별 요청의 E2E 레이턴시보다 전체 처리량(단위 시간당 처리한 요청 수)을 더 중시하는 경우가 많습니다.

```
요청 전송
  |
  |<------ TTFT ------>|
  |                     |  첫 번째 토큰 도착
  |                     |
  |                     |<-- TPOT -->|<-- TPOT -->| ... |<-- TPOT -->|
  |                     |  2번째     |  3번째          |  마지막 토큰
  |                                                                  |
  |<----------------------- E2E 레이턴시 --------------------------->|
```

이 세 지표를 함께 보아야 하는 이유는, 하나만으로는 서빙 성능을 정확히 파악할 수 없기 때문입니다. TTFT가 빨라도 TPS가 낮으면 긴 응답을 생성할 때 사용자가 오래 기다립니다. 반대로 TPS가 높아도 TTFT가 느리면 첫 글자가 나오기까지 답답한 대기가 생깁니다. 온라인 서빙에서는 TTFT와 TPS를 모두 관리해야 하고, 배치 서빙에서는 E2E 레이턴시와 전체 처리량의 균형을 맞추어야 합니다.

이제 서빙 프레임워크를 선택할 때 고려해야 하는 기준을 살펴보겠습니다. 서빙 프레임워크란 위에서 설명한 모델 로딩, 요청 처리, 배칭, 응답 반환 같은 기능을 미리 구현해 둔 소프트웨어입니다. 직접 이 모든 것을 구현하는 대신 서빙 프레임워크를 사용하면 모델을 빠르게 서비스에 올릴 수 있습니다. 대표적인 프레임워크로는 vLLM, TGI(Text Generation Inference), TensorRT-LLM 등이 있으며, 이들은 6.1.2, 6.1.3, 6.1.4에서 각각 상세히 다룹니다. 여기서는 프레임워크를 비교할 때 어떤 기준으로 판단해야 하는지를 정리합니다.

첫 번째 기준은 **모델 호환성**입니다. 서빙하려는 모델의 아키텍처를 해당 프레임워크가 지원하는지 확인해야 합니다. 예를 들어 Llama 계열 모델을 서빙하려면 그 아키텍처를 지원하는 프레임워크를 선택해야 합니다. 새로운 모델이 공개되었을 때 프레임워크가 얼마나 빠르게 지원을 추가하는지도 중요한 고려 사항입니다.

두 번째 기준은 **추론 성능**입니다. 같은 모델과 같은 하드웨어에서도 프레임워크에 따라 TTFT, TPS, 최대 처리량이 달라집니다. 이는 프레임워크마다 배칭 전략, 메모리 관리 방식, GPU 커널 최적화 수준이 다르기 때문입니다. 실제 서비스 조건과 비슷한 환경에서 벤치마크를 수행하여 비교하는 것이 가장 확실합니다.

세 번째 기준은 **배칭 전략**입니다. 단순히 여러 요청을 모아서 한 번에 넣는 정적 배칭(static batching)만 지원하는지, 요청이 완료될 때마다 새 요청을 즉시 끼워 넣는 연속 배칭(continuous batching)을 지원하는지가 처리량에 큰 영향을 줍니다. 연속 배칭은 GPU 유휴 시간을 줄여 동시 처리 가능한 요청 수를 크게 늘립니다.

네 번째 기준은 **양자화 지원**입니다. 모델의 가중치를 더 적은 비트로 표현하면 메모리 사용량이 줄고 추론 속도가 빨라질 수 있습니다. 프레임워크가 어떤 양자화 방식(FP16, INT8, INT4, GPTQ, AWQ 등)을 지원하는지에 따라 사용할 수 있는 최적화 범위가 달라집니다.

다섯 번째 기준은 **배포 편의성**입니다. Docker 이미지 제공 여부, 설정 파라미터의 간결함, 문서의 충실도, 커뮤니티 활성도 등이 실제 운영에서의 생산성에 영향을 줍니다.

마지막으로 **OpenAI 호환 API 인터페이스**에 대해 다루겠습니다. OpenAI가 자사 모델의 API 형식으로 정의한 인터페이스가 사실상 업계 표준이 되었습니다. 이 인터페이스의 핵심은 '/v1/chat/completions'와 '/v1/completions' 두 가지 엔드포인트입니다. 클라이언트가 JSON 형식으로 모델 이름, 메시지 목록, 생성 파라미터(temperature, max_tokens 등)를 담아 HTTP POST 요청을 보내면, 서버가 같은 형식으로 생성된 텍스트를 돌려주는 구조입니다.

아래는 '/v1/chat/completions' 엔드포인트에 보내는 요청의 예입니다.

```json
{
  "model": "my-llama-model",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "서울의 날씨를 알려주세요."}
  ],
  "temperature": 0.7,
  "max_tokens": 256
}
```

'model' 필드는 서빙 중인 모델의 이름을 지정합니다. 'messages' 필드는 대화 이력을 역할(role)과 내용(content)의 쌍으로 담습니다. 'system'은 모델의 동작 지침을, 'user'는 사용자의 입력을, 'assistant'는 모델의 이전 응답을 나타냅니다. 'temperature'와 'max_tokens'는 1.2.3에서 다룬 생성 전략 파라미터입니다.

이 형식이 중요한 이유는 호환성에 있습니다. vLLM, TGI, Ollama 등 주요 서빙 프레임워크들은 모두 이 형식을 지원합니다. 따라서 OpenAI API를 호출하도록 작성된 애플리케이션 코드를 수정하지 않고, 엔드포인트 URL만 바꾸어 자체 서빙 서버로 전환할 수 있습니다. 예를 들어, 기존에 'https://api.openai.com/v1/chat/completions'로 요청을 보내던 코드를 'http://localhost:8000/v1/chat/completions'로 변경하면, 자체 서버에 올린 오픈소스 모델을 같은 방식으로 호출할 수 있습니다. 이 호환성 덕분에 모델을 교체하거나 서빙 인프라를 바꿀 때 애플리케이션 코드의 변경을 최소화할 수 있고, 개발 초기에는 OpenAI API로 프로토타이핑한 뒤 나중에 자체 서빙으로 전환하는 경로도 자연스럽게 열립니다.

정리하면, 모델 서빙은 학습된 모델을 GPU 메모리에 올리고, 외부 요청을 받아 추론을 수행한 뒤 결과를 반환하는 시스템입니다. 서빙 아키텍처는 모델 로딩, 요청 처리(토큰화, 배칭, 추론), 응답 반환(디코딩, 스트리밍)의 세 단계로 구성됩니다. 사용 목적에 따라 즉시 응답이 필요한 온라인 서빙과 대량 처리에 적합한 배치 서빙으로 나뉘며, 서빙 성능은 TTFT, TPS, E2E 레이턴시로 측정합니다. 서빙 프레임워크를 선택할 때는 모델 호환성, 추론 성능, 배칭 전략, 양자화 지원, 배포 편의성을 기준으로 비교하며, OpenAI 호환 API를 지원하는 프레임워크를 선택하면 애플리케이션 코드의 이식성을 확보할 수 있습니다.

다음 단원인 6.1.2에서는 vLLM으로 모델 서빙하기를 다룹니다. vLLM의 핵심 메커니즘인 PagedAttention과 연속 배칭이 어떻게 동작하는지, 그리고 실제로 오픈소스 모델을 서빙하는 과정을 살펴봅니다.

이 단원을 마치면 LLM 서빙 아키텍처의 핵심 구성요소를 설명할 수 있습니다.
