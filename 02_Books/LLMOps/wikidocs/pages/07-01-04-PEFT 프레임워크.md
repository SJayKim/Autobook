# 7.1.4 PEFT 프레임워크

7.1.2에서 LoRA가 거대한 가중치 행렬을 직접 수정하는 대신 작은 저랭크 행렬 두 개를 끼워 넣어 학습 파라미터 수를 줄이는 방법을 살펴보았고, 7.1.3에서는 QLoRA가 4비트 양자화와 이중 양자화를 결합하여 단일 GPU에서도 수십억 파라미터 모델을 파인튜닝할 수 있게 해 주는 과정을 다루었습니다. LoRA와 QLoRA는 파라미터 효율적 파인튜닝(Parameter-Efficient Fine-Tuning, PEFT)이라는 큰 범주에 속하는 기법 중 두 가지일 뿐입니다. 같은 범주에는 Prefix Tuning, Prompt Tuning, IA3처럼 접근 방식이 서로 다른 기법이 여러 가지 더 있습니다. 문제는 이 기법들을 직접 구현하려면 모델 내부 구조를 뜯어 고쳐야 하고, 기법마다 코드 패턴이 달라서 실험과 비교가 번거롭다는 점입니다. Hugging Face가 제공하는 **PEFT 라이브러리**는 이 문제를 해결하기 위해 만들어진 파이썬 패키지입니다. 하나의 통합된 인터페이스로 여러 PEFT 기법을 적용하고, 학습하고, 저장하고, 불러올 수 있게 해 줍니다.

PEFT 라이브러리가 해결하는 핵심 문제를 구체적으로 짚어 보겠습니다. 파인튜닝에서 가장 단순한 방법은 모델의 모든 파라미터를 갱신하는 전체 파인튜닝(Full Fine-Tuning)입니다. 7.1.1에서 다루었듯이, 70억 파라미터 모델을 전체 파인튜닝하려면 모델 가중치, 옵티마이저 상태, 그래디언트를 모두 메모리에 올려야 하므로 수십 GB의 GPU 메모리가 필요합니다. LoRA나 QLoRA를 쓰면 학습 파라미터 수가 전체의 0.1~1% 수준으로 줄어들어 메모리 부담이 크게 줄어듭니다. 그런데 LoRA만이 유일한 선택지가 아닙니다. 과제의 성격, 모델의 구조, 가용 자원에 따라 더 적합한 기법이 있을 수 있습니다. PEFT 라이브러리는 이런 기법들을 통일된 API로 묶어, 설정 객체 하나만 바꾸면 기법 전환이 가능하도록 설계되어 있습니다.

PEFT 라이브러리의 구조는 세 가지 핵심 요소로 이루어져 있습니다. 첫째, 어떤 기법을 어떤 설정으로 적용할지를 담는 **설정 클래스**(Config)입니다. LoraConfig, PrefixTuningConfig, PromptTuningConfig, IA3Config 등 기법마다 전용 설정 클래스가 있고, 각 클래스에는 해당 기법의 하이퍼파라미터가 속성으로 들어 있습니다. 둘째, 원본 모델을 감싸서 PEFT 기법을 적용한 **모델 래퍼**인 get_peft_model 함수입니다. 원본 모델과 설정 객체를 넣으면, 원본 파라미터는 동결(freeze)하고 기법에 해당하는 학습 가능한 파라미터만 추가한 새 모델을 돌려줍니다. 셋째, 학습이 끝난 뒤 추가된 파라미터만 따로 **저장하고 불러오는** 메서드입니다. 전체 모델이 아니라 추가된 소량의 파라미터만 저장하므로 파일 크기가 수 MB에서 수십 MB 수준에 머뭅니다. 이 세 요소의 흐름을 정리하면 다음과 같습니다.

```
원본 모델 (수십억 파라미터, 동결)
     |
     +--- 설정 객체 (Config) 생성
     |      예: LoraConfig(r=16, lora_alpha=32, ...)
     |
     +--- get_peft_model(원본 모델, 설정 객체)
     |      원본 파라미터 동결 + PEFT 파라미터 삽입
     |      학습 가능 파라미터: 전체의 0.1~1%
     |
     +--- 학습 (Trainer 또는 직접 루프)
     |
     +--- save_pretrained("경로")
     |      PEFT 파라미터만 저장 (수 MB)
     |
     +--- from_pretrained("경로")
            원본 모델에 PEFT 파라미터 다시 결합
```

이제 PEFT 라이브러리가 지원하는 주요 기법을 LoRA, QLoRA 외의 것들 위주로 하나씩 살펴보겠습니다.

**Prefix Tuning**은 Transformer 각 층의 어텐션 메커니즘 앞에 학습 가능한 벡터 시퀀스를 삽입하는 방법입니다. 이 기법이 등장한 배경을 먼저 이해할 필요가 있습니다. 1.2.1에서 다룬 것처럼, Transformer의 각 층은 Self-Attention을 통해 입력 토큰들 사이의 관계를 계산합니다. 이때 각 토큰은 Key와 Value 벡터를 만들어 내고, 다른 토큰의 Query가 이 Key-Value를 참조하여 문맥 정보를 모읍니다. Prefix Tuning의 핵심 아이디어는 "실제 입력 토큰 앞에 가상의 토큰을 추가하면, 모델이 이 가상 토큰의 Key-Value도 함께 참조하게 되어 출력 방향이 바뀔 수 있다"는 것입니다.

구체적으로, Prefix Tuning은 Transformer의 모든 층에 길이가 p인 접두사(prefix) 벡터 쌍을 추가합니다. 각 층의 어텐션 계산에서 Key 행렬 앞에 학습 가능한 접두사 Key 벡터가 붙고, Value 행렬 앞에 학습 가능한 접두사 Value 벡터가 붙습니다. 실제 입력 토큰의 Query가 어텐션을 계산할 때, 원래 입력 토큰의 Key-Value뿐 아니라 접두사의 Key-Value까지 함께 참조하게 됩니다. 접두사 벡터는 학습 과정에서 과제에 맞게 갱신되므로, 모델이 특정 과제에 적합한 방향으로 문맥을 해석하도록 유도합니다.

```
[Prefix Key 1][Prefix Key 2]...[Prefix Key p][입력 토큰 Key 1][입력 토큰 Key 2]...
[Prefix Val 1][Prefix Val 2]...[Prefix Val p][입력 토큰 Val 1][입력 토큰 Val 2]...

                    ^                                      ^
              학습 가능 (갱신됨)                     동결 (고정됨)

Query는 모든 Key를 참조 --> 접두사 벡터가 어텐션 분포에 영향
```

접두사 길이 p는 Prefix Tuning의 주요 하이퍼파라미터입니다. p가 클수록 모델이 참조할 가상 문맥이 많아져 표현력이 높아지지만, 실제 입력에 할당할 수 있는 시퀀스 길이가 줄어듭니다. 모델의 최대 시퀀스 길이가 2048이고 접두사 길이가 20이면, 실제 입력은 2028 토큰까지만 넣을 수 있습니다. 실무에서는 10에서 100 사이의 값이 흔히 쓰입니다. 학습 파라미터 수는 "층 수 x 접두사 길이 x 히든 차원 x 2(Key와 Value)"로 결정되며, 전체 모델 파라미터의 0.1% 미만에 해당하는 경우가 많습니다.

PEFT 라이브러리에서 Prefix Tuning을 적용하는 코드는 다음과 같습니다.

```python
from peft import PrefixTuningConfig, get_peft_model

config = PrefixTuningConfig(
    task_type="CAUSAL_LM",
    num_virtual_tokens=20,
)

model = get_peft_model(base_model, config)
model.print_trainable_parameters()
```

task_type은 모델의 과제 유형을 지정합니다. "CAUSAL_LM"은 GPT 계열처럼 다음 토큰을 예측하는 인과 언어 모델을 뜻합니다. num_virtual_tokens는 접두사 길이 p에 해당합니다. 이 두 줄의 설정만으로 Prefix Tuning이 적용되며, print_trainable_parameters를 호출하면 전체 파라미터 대비 학습 가능한 파라미터의 비율을 확인할 수 있습니다.

**Prompt Tuning**은 Prefix Tuning과 비슷해 보이지만 작동 방식이 다릅니다. 두 기법의 차이를 명확히 짚겠습니다. Prefix Tuning은 Transformer의 모든 층에 접두사 벡터를 삽입합니다. 반면 Prompt Tuning은 모델의 입력 임베딩 층에만 학습 가능한 벡터를 추가합니다. 즉, 실제 입력 텍스트의 토큰 임베딩 앞에 학습 가능한 "소프트 프롬프트" 벡터가 붙는 것이 전부입니다.

Prompt Tuning의 배경을 이해하려면, 프롬프트 엔지니어링과의 관계를 먼저 살펴봐야 합니다. 2.1.1에서 다룬 프롬프트 엔지니어링은 사람이 자연어 텍스트를 직접 작성하여 모델의 동작을 유도합니다. "다음 문장을 한국어로 번역하세요:"처럼 사람이 읽을 수 있는 문장을 모델 입력 앞에 붙이는 방식입니다. 이런 프롬프트를 **하드 프롬프트**(hard prompt)라고 합니다. 하드 프롬프트의 한계는, 사람이 읽을 수 있는 단어만 사용해야 하므로 모델이 표현할 수 있는 공간의 극히 일부만 활용한다는 점입니다. Prompt Tuning은 이 제약을 풀어 줍니다. 사람이 읽을 수 있는 단어 대신, 모델의 임베딩 공간에서 자유롭게 움직일 수 있는 연속 벡터를 프롬프트로 사용합니다. 이것을 **소프트 프롬프트**(soft prompt)라고 부릅니다.

소프트 프롬프트 벡터는 처음에 임의의 값이나 기존 단어의 임베딩으로 초기화된 뒤, 학습 과정에서 과제의 손실(loss)을 줄이는 방향으로 갱신됩니다. 학습이 끝나면 이 벡터들은 어떤 자연어 단어와도 대응하지 않는 고유한 값을 갖게 됩니다. 사람이 읽을 수 없지만, 모델 입장에서는 해당 과제에 최적화된 문맥 정보를 담고 있는 것입니다.

```
하드 프롬프트: ["번역", "하세요", ":"]  -->  토큰 임베딩  -->  모델 입력
소프트 프롬프트: [v1, v2, v3, ..., vn]   -->  학습 가능한 벡터  -->  모델 입력

v1~vn은 자연어 단어에 대응하지 않음 (연속 벡터)
나머지 모델 파라미터는 모두 동결
```

Prompt Tuning의 학습 파라미터 수는 "소프트 프롬프트 길이 x 임베딩 차원"입니다. 접두사 길이가 20이고 임베딩 차원이 4096이면 학습 파라미터는 약 81,920개로, 70억 파라미터 모델의 0.001% 수준입니다. Prefix Tuning보다도 학습 파라미터가 적습니다. 그 이유는 Prefix Tuning이 모든 층에 접두사를 넣는 반면, Prompt Tuning은 입력 층에만 벡터를 추가하기 때문입니다. 다만 이 극단적인 효율성에는 대가가 있습니다. 표현력이 상대적으로 제한되어, 모델 크기가 작을 때는 성능이 Prefix Tuning이나 LoRA에 비해 떨어질 수 있습니다. 반면 모델이 충분히 클 때(100억 파라미터 이상)는 Prompt Tuning만으로도 전체 파인튜닝에 근접하는 성능을 보이는 것으로 알려져 있습니다.

PEFT 라이브러리에서 Prompt Tuning을 적용하는 코드는 다음과 같습니다.

```python
from peft import PromptTuningConfig, PromptTuningInit, get_peft_model

config = PromptTuningConfig(
    task_type="CAUSAL_LM",
    num_virtual_tokens=20,
    prompt_tuning_init=PromptTuningInit.TEXT,
    prompt_tuning_init_text="Classify if the sentiment of this review is positive or negative.",
    tokenizer_name_or_path="모델 이름 또는 경로",
)

model = get_peft_model(base_model, config)
```

prompt_tuning_init 파라미터는 소프트 프롬프트 벡터의 초기화 방법을 지정합니다. PromptTuningInit.TEXT로 설정하면, prompt_tuning_init_text에 지정한 자연어 문장의 토큰 임베딩으로 초기화합니다. 이렇게 하면 임의 초기화보다 학습이 빠르게 수렴합니다. PromptTuningInit.RANDOM으로 설정하면 임의의 값으로 초기화합니다.

**IA3**(Infused Adapter by Inhibiting and Amplifying Inner Activations)는 LoRA나 Prefix Tuning과는 다른 각도에서 효율성을 추구하는 기법입니다. IA3의 이름에 담긴 핵심 아이디어는 "내부 활성값을 억제하거나 증폭한다"는 것입니다. 활성값(activation)이란 모델 내부에서 한 층의 연산 결과로 나오는 중간 벡터를 뜻합니다. IA3는 이 활성값에 학습 가능한 스케일링 벡터를 원소별로 곱합니다.

좀 더 구체적으로, IA3는 Transformer의 세 곳에 스케일링 벡터를 삽입합니다. 첫째, 어텐션 메커니즘에서 Key 벡터에 곱하는 벡터 l_k입니다. 둘째, 어텐션 메커니즘에서 Value 벡터에 곱하는 벡터 l_v입니다. 셋째, Feed-Forward Network(FFN)의 중간 활성값에 곱하는 벡터 l_ff입니다. 각 벡터의 차원은 해당 위치의 히든 차원과 같습니다. 스케일링 벡터의 각 원소가 1에 가까우면 원래 활성값을 거의 그대로 두고(억제도 증폭도 하지 않고), 0에 가까우면 해당 차원의 활성값을 억제하며, 1보다 크면 증폭합니다. 학습 과정에서 이 벡터들이 과제에 맞게 조정되면, 모델의 기존 지식 중 과제에 유용한 부분은 강화하고 불필요한 부분은 약화하는 효과를 얻습니다.

```
원래 어텐션:
  Key = X * W_k
  Value = X * W_v

IA3 적용 후:
  Key = l_k * (X * W_k)       <-- l_k: 학습 가능한 스케일링 벡터
  Value = l_v * (X * W_v)     <-- l_v: 학습 가능한 스케일링 벡터

원래 FFN:
  출력 = activation(X * W_1) * W_2

IA3 적용 후:
  출력 = (l_ff * activation(X * W_1)) * W_2   <-- l_ff: 학습 가능한 스케일링 벡터

*는 원소별(element-wise) 곱
```

IA3의 학습 파라미터 수는 LoRA보다도 적습니다. LoRA는 각 타겟 모듈에 "입력 차원 x 랭크 + 랭크 x 출력 차원" 개의 파라미터를 추가하는 반면, IA3는 각 위치에 히든 차원 크기의 벡터 하나만 추가합니다. 예를 들어, 히든 차원이 4096인 모델에서 LoRA(랭크 16)는 어텐션 모듈 하나당 약 131,072개의 파라미터를 추가하지만, IA3는 같은 위치에 4,096개만 추가합니다. 이 극도의 경량성 덕분에 IA3는 메모리가 극히 제한된 환경에서 유용합니다. 다만 표현력은 LoRA보다 제한적이어서, 복잡한 도메인 적응에는 LoRA가 더 나은 성능을 보이는 경우가 많습니다.

PEFT 라이브러리에서 IA3를 적용하는 코드는 다음과 같습니다.

```python
from peft import IA3Config, get_peft_model

config = IA3Config(
    task_type="CAUSAL_LM",
    target_modules=["k_proj", "v_proj", "down_proj"],
    feedforward_modules=["down_proj"],
)

model = get_peft_model(base_model, config)
```

target_modules는 IA3 스케일링 벡터를 삽입할 모듈을 지정합니다. feedforward_modules는 target_modules 중에서 FFN에 해당하는 모듈을 별도로 지정합니다. 이 구분이 필요한 이유는, 어텐션의 Key/Value와 FFN에서 스케일링 벡터를 곱하는 위치가 다르기 때문입니다.

지금까지 살펴본 네 가지 PEFT 기법(LoRA, Prefix Tuning, Prompt Tuning, IA3)은 각각 장단점이 다릅니다. 어떤 기법을 선택할지 결정할 때 고려할 주요 기준을 정리하겠습니다.

| 기준 | LoRA / QLoRA | Prefix Tuning | Prompt Tuning | IA3 |
|------|-------------|---------------|---------------|-----|
| 학습 파라미터 수 | 중간 (0.1~1%) | 적음 (0.1% 미만) | 매우 적음 (0.01% 미만) | 매우 적음 (0.01% 미만) |
| 삽입 위치 | 어텐션/FFN 가중치 옆 | 모든 층의 Key-Value 앞 | 입력 임베딩 층 | 어텐션 Key-Value + FFN 활성값 |
| 성능 (소형 모델) | 높음 | 중간 | 낮음~중간 | 중간 |
| 성능 (대형 모델) | 높음 | 높음 | 높음 | 중간~높음 |
| 추론 오버헤드 | 없음 (병합 가능) | 약간 있음 | 약간 있음 | 없음 (병합 가능) |
| 다중 과제 전환 | 어댑터 교체 | 접두사 교체 | 소프트 프롬프트 교체 | 스케일링 벡터 교체 |

LoRA와 QLoRA는 범용성이 가장 높습니다. 모델 크기와 관계없이 안정적인 성능을 보이고, 학습 후 어댑터를 원본 가중치에 병합하면 추론 시 추가 비용이 없습니다. 복잡한 도메인 적응이나 지시 따르기(instruction following) 개선에 적합합니다. Prefix Tuning은 학습 파라미터가 적으면서도 모든 층에 영향을 미치므로, 생성 과제에서 출력 스타일이나 형식을 바꾸는 데 효과적입니다. Prompt Tuning은 파라미터가 가장 적어 저장과 교체 비용이 최소이므로, 모델이 충분히 크고 과제가 여러 개라서 과제별 어댑터를 빈번하게 전환해야 하는 상황에 유리합니다. IA3는 LoRA보다 파라미터가 적으면서도 추론 시 병합이 가능하다는 장점이 있어, 메모리 제약이 심한 환경에서 가벼운 적응이 필요할 때 고려할 수 있습니다.

실무에서 기법을 선택하는 의사결정 흐름을 단순화하면 다음과 같습니다.

```

```

위 흐름은 출발점일 뿐, 실제로는 데이터의 양과 품질, 과제의 복잡도, 원하는 성능 수준에 따라 여러 기법을 실험하고 비교해야 합니다.

마지막으로, PEFT 라이브러리를 Hugging Face Trainer와 통합하는 방법을 살펴보겠습니다. Hugging Face의 **Trainer**는 학습 루프, 평가, 체크포인트 저장, 로깅 등을 자동으로 처리해 주는 클래스입니다. PEFT로 감싼 모델을 Trainer에 그대로 넣으면, Trainer가 학습 가능한 파라미터만 자동으로 갱신합니다. 별도의 학습 루프를 직접 작성할 필요가 없습니다.

```python
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
from peft import LoraConfig, get_peft_model

# 1. 원본 모델과 토크나이저 불러오기
base_model = AutoModelForCausalLM.from_pretrained("모델 이름")
tokenizer = AutoTokenizer.from_pretrained("모델 이름")

# 2. PEFT 설정 생성 (여기서는 LoRA 예시)
peft_config = LoraConfig(
    task_type="CAUSAL_LM",
    r=16,
    lora_alpha=32,
    lora_dropout=0.1,
    target_modules=["q_proj", "v_proj"],
)

# 3. PEFT 모델 생성
model = get_peft_model(base_model, peft_config)
model.print_trainable_parameters()

# 4. 학습 설정
training_args = TrainingArguments(
    output_dir="./output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    learning_rate=2e-4,
    save_strategy="epoch",
    logging_steps=10,
)

# 5. Trainer 생성 및 학습
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    tokenizer=tokenizer,
)

trainer.train()

# 6. PEFT 파라미터만 저장
model.save_pretrained("./peft-adapter")
```

이 코드의 흐름을 단계별로 풀어 보겠습니다. 1단계에서 사전훈련된 원본 모델과 토크나이저를 불러옵니다. 2단계에서 PEFT 설정 객체를 생성합니다. 이 예시에서는 LoraConfig를 사용했지만, PrefixTuningConfig, PromptTuningConfig, IA3Config 중 어떤 것으로 바꾸더라도 나머지 코드는 동일합니다. 이것이 PEFT 라이브러리의 통합 인터페이스가 주는 이점입니다. 3단계에서 get_peft_model이 원본 모델의 파라미터를 동결하고 설정에 따른 학습 가능한 파라미터를 삽입합니다. 4단계에서 TrainingArguments로 학습 에포크 수, 배치 크기, 학습률 등을 지정합니다. learning_rate는 PEFT에서 보통 1e-4에서 3e-4 사이의 값을 사용합니다. 전체 파인튜닝보다 높은 학습률을 쓰는 이유는, 갱신 대상 파라미터 수가 적어 한 번의 갱신에서 더 큰 보폭으로 움직여야 같은 횟수의 학습 내에 수렴할 수 있기 때문입니다. 5단계에서 Trainer가 학습을 실행합니다. Trainer 내부에서는 데이터 로딩, 포워드 패스, 로스 계산, 백워드 패스, 옵티마이저 스텝이 자동으로 반복됩니다. 이때 동결된 파라미터는 그래디언트가 계산되지 않으므로 메모리를 절약합니다. 6단계에서 save_pretrained를 호출하면 PEFT 파라미터만 저장됩니다. 나중에 같은 원본 모델에 이 파라미터를 불러와 결합하면 파인튜닝된 모델을 복원할 수 있습니다.

기법을 바꾸고 싶을 때는 2단계의 설정 객체만 교체합니다. 예를 들어, Prefix Tuning으로 전환하려면 LoraConfig 대신 PrefixTuningConfig를 생성하고 같은 흐름을 따르면 됩니다. 이처럼 PEFT 라이브러리는 기법 간 전환 비용을 최소화하여, 여러 기법을 실험하고 비교하는 과정을 효율적으로 만들어 줍니다.

정리하면, PEFT 라이브러리는 LoRA, Prefix Tuning, Prompt Tuning, IA3 등 다양한 파라미터 효율적 파인튜닝 기법을 통일된 API로 묶어 제공하는 Hugging Face의 파이썬 패키지입니다. 설정 클래스, get_peft_model 래퍼, 저장/불러오기 메서드라는 세 가지 핵심 요소를 통해, 원본 모델을 건드리지 않고 소량의 학습 가능한 파라미터만 추가하여 과제에 맞게 모델을 적응시킬 수 있습니다. Prefix Tuning은 모든 층의 어텐션 Key-Value 앞에 학습 가능한 접두사를 삽입하고, Prompt Tuning은 입력 임베딩 층에만 소프트 프롬프트 벡터를 추가하며, IA3는 어텐션과 FFN의 활성값에 스케일링 벡터를 곱합니다. 각 기법은 학습 파라미터 수, 성능, 추론 오버헤드에서 서로 다른 특성을 보이므로, 과제와 자원 조건에 맞는 기법을 선택해야 합니다. Hugging Face Trainer와의 통합을 통해 설정 객체만 바꾸면 기법 전환이 가능하여, 효율적인 실험과 비교가 이루어집니다.
