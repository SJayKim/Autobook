# 9.2.2 로깅, 트레이싱, 디버깅

9.2.1에서 지연 시간이라는 하나의 지표를 깊이 있게 다루었습니다. 그런데 프로덕션에서 발생하는 문제는 지연 시간 증가만이 아닙니다. 모델이 엉뚱한 응답을 생성하거나, 특정 사용자의 요청만 실패하거나, 어제까지 잘 작동하던 파이프라인이 갑자기 오류를 내는 상황도 있습니다. 이런 문제를 체계적으로 기록하고 추적하여 원인을 찾아내는 것이 로깅, 트레이싱, 디버깅의 영역입니다. 이 단원에서는 LLM 애플리케이션에 특화된 로깅과 트레이싱 체계를 설계하고, 수집된 데이터를 바탕으로 문제를 진단하는 디버깅 워크플로를 다룹니다.

전통적인 웹 애플리케이션에서 로깅이란 서버에서 발생하는 이벤트를 시간순으로 기록하는 것입니다. "14:32:01 GET /api/users 200 OK 23ms"처럼, 언제 어떤 요청이 어떤 결과로 처리되었는지를 남깁니다. LLM 애플리케이션의 로깅도 기본 원리는 같지만, 기록해야 하는 정보의 종류와 양이 크게 다릅니다. 이것이 **LLM 로깅의 특수성**입니다.

첫째, **프롬프트와 응답 전문**을 기록해야 합니다. 전통적인 API에서는 요청 본문을 일일이 저장하지 않는 경우가 많습니다. 요청의 구조가 정해져 있어 상태 코드만으로도 성공 여부를 판단할 수 있기 때문입니다. LLM 호출에서는 프롬프트의 내용이 응답 품질을 결정하므로, 어떤 프롬프트에 어떤 응답이 나왔는지를 확인할 수 있어야 합니다. 9.1.1에서 설명한 입출력 콘텐츠 추적이 바로 이것입니다.

둘째, **토큰 수**를 기록해야 합니다. 입력 토큰과 출력 토큰의 수는 비용 계산과 성능 분석에 필수적인 데이터입니다. 같은 엔드포인트에 대한 요청이라도 토큰 수에 따라 비용과 지연 시간이 크게 달라집니다.

셋째, **모델 파라미터**를 기록해야 합니다. 어떤 모델을 사용했는지, temperature, max_tokens, top_p 같은 생성 파라미터가 무엇이었는지를 함께 남겨야, 동일 조건에서 문제를 재현할 수 있습니다.

넷째, **파이프라인 단계 정보**를 기록해야 합니다. LLM 애플리케이션은 단일 호출이 아니라 여러 단계(검색, 프롬프트 조합, 모델 호출, 후처리 등)의 체인입니다. 어느 단계에서 오류가 발생했는지, 어느 단계가 병목인지를 파악하려면 단계별 로그가 필요합니다.

이처럼 기록할 정보가 많고 다양하므로, 자유 형식의 텍스트 로그로는 분석이 어렵습니다. 이를 해결하는 접근이 **구조화된 로깅**(structured logging)입니다. 구조화된 로깅이란 로그를 사전에 정의된 필드 구조로 기록하는 것입니다. 일반 텍스트 로그가 "2026-04-03 14:32:01 INFO model=gpt-4o tokens=1500 latency=2.3s"처럼 문자열 한 줄이라면, 구조화된 로그는 각 항목을 별도 필드로 분리합니다.

```json
{
  "timestamp": "2026-04-03T14:32:01Z",
  "level": "INFO",
  "trace_id": "abc123",
  "span_id": "def456",
  "event": "llm_call_complete",
  "model": "gpt-4o",
  "input_tokens": 1200,
  "output_tokens": 300,
  "total_tokens": 1500,
  "latency_ms": 2300,
  "cost_usd": 0.0045,
  "temperature": 0.7,
  "status": "success",
  "user_id": "user_1234",
  "session_id": "session_abc"
}
```

위 JSON 구조에서 각 필드의 의미를 살펴보겠습니다. 'timestamp'는 이벤트 발생 시간입니다. 'level'은 로그의 심각도(DEBUG, INFO, WARNING, ERROR 등)입니다. 'trace_id'와 'span_id'는 이 로그가 어떤 트레이스의 어떤 스팬에 속하는지를 나타내며, 뒤에서 다룰 분산 트레이싱과 연결됩니다. 'event'는 어떤 종류의 이벤트인지를 구분합니다. 나머지 필드는 LLM 호출의 구체적인 정보입니다.

구조화된 로깅의 장점은 기계가 로그를 파싱하고 분석하기 쉽다는 것입니다. "오류가 발생한 요청 중 입력 토큰이 5,000개를 넘는 것만 추려라", "지난 1시간 동안 model=gpt-4o인 요청의 평균 지연 시간을 구하라" 같은 질의를 로그 분석 도구에서 바로 수행할 수 있습니다. 텍스트 로그에서는 정규표현식으로 파싱해야 하는 작업이, 구조화된 로그에서는 필드 기반 검색으로 단순해집니다.

구조화된 로깅을 설계할 때 고려해야 할 점이 있습니다. 어떤 필드를 필수로 포함할지, 필드의 이름과 형식을 어떻게 표준화할지, 팀 전체가 동일한 규칙을 따르도록 어떻게 강제할지입니다. 로깅 필드 표준을 문서화하고, 로그 생성 함수를 공유 라이브러리로 제공하면 일관성을 유지하기 쉽습니다.

로깅이 개별 이벤트를 기록하는 것이라면, **분산 트레이싱**(distributed tracing)은 여러 이벤트를 하나의 흐름으로 엮는 것입니다. 9.1.1에서 트레이스와 스팬의 개념을 다루었습니다. 여기서는 분산 환경에서 트레이싱이 어떻게 작동하는지, 그리고 **Trace ID**와 **Span**이라는 두 핵심 개념을 더 깊이 살펴봅니다.

**Trace ID**는 하나의 사용자 요청 전체를 고유하게 식별하는 ID입니다. 사용자 요청이 들어오면 시스템 입구에서 Trace ID가 생성됩니다. 이후 이 요청이 거치는 모든 서비스, 모든 단계가 같은 Trace ID를 공유합니다. API Gateway에서 Trace ID가 생성되고, 벡터 검색 서비스에 전달되고, LLM 호출 서비스에 전달되고, 후처리 서비스에 전달됩니다. 각 서비스가 남기는 로그에 같은 Trace ID가 찍히므로, 나중에 이 ID로 검색하면 하나의 요청이 거친 모든 경로를 복원할 수 있습니다.

**Span**은 트레이스 안에서 하나의 작업 단위를 나타냅니다. 각 Span에는 고유한 Span ID가 있고, 부모 Span ID를 참조하여 계층 구조를 형성합니다. 하나의 트레이스가 하나의 루트 Span을 가지고, 그 아래에 자식 Span들이 달리는 트리 구조입니다.

```
분산 트레이싱 구조 예시

Trace ID: abc-123

[루트 Span] api_request (Span ID: s1, 부모: 없음) --- 전체 3.5s
  |
  +-- [Span] vector_search (Span ID: s2, 부모: s1) --- 0.3s
  |     서비스: retrieval-service
  |
  +-- [Span] prompt_assembly (Span ID: s3, 부모: s1) --- 0.02s
  |     서비스: orchestrator-service
  |
  +-- [Span] llm_call (Span ID: s4, 부모: s1) --- 3.1s
  |     서비스: inference-service
  |     |
  |     +-- [Span] token_generation (Span ID: s5, 부모: s4) --- 2.8s
  |
  +-- [Span] response_format (Span ID: s6, 부모: s1) --- 0.08s
        서비스: orchestrator-service
```

위 예시에서 Trace ID 'abc-123'으로 검색하면, 네 개의 서로 다른 작업과 그 계층 관계가 보입니다. LLM 호출(s4)이 전체 3.5초 중 3.1초를 차지하고 있으므로, 병목이 모델 추론에 있다는 것을 바로 알 수 있습니다.

트레이싱에서 중요한 것은 **컨텍스트 전파**(context propagation)입니다. 요청이 서비스 A에서 서비스 B로 넘어갈 때, Trace ID와 현재 Span ID를 함께 전달해야 합니다. HTTP 기반 통신에서는 요청 헤더에 이 정보를 실어 보냅니다. OpenTelemetry에서는 'traceparent'라는 표준 헤더를 사용합니다. 이 헤더에 Trace ID, 부모 Span ID, 플래그가 인코딩되어, 수신 서비스가 같은 트레이스의 자식 Span을 생성할 수 있습니다.

수집된 로그와 트레이스를 활용하여 문제를 찾아내는 과정이 **디버깅 워크플로**입니다. LLM 애플리케이션의 디버깅은 전통적인 소프트웨어 디버깅과 접근이 다릅니다. 전통 소프트웨어에서는 코드의 논리적 오류를 찾는 것이 핵심이지만, LLM 애플리케이션에서는 모델의 비결정적 출력, 프롬프트의 미묘한 차이, 검색 품질의 변동 등 원인이 다양합니다.

디버깅 워크플로를 단계별로 정리하면 다음과 같습니다.

```
LLM 디버깅 워크플로

1. 문제 감지
   - 알림 발동, 사용자 리포트, 품질 점수 하락
   |
   v
2. 문제 트레이스 식별
   - 시간대, 오류 유형, 태그 등으로 관련 트레이스 필터링
   |
   v
3. 트레이스 분석
   - 스팬별 입출력 확인
   - 어느 단계에서 이상이 발생했는지 특정
   |
   v
4. 원인 범주 분류
   - 프롬프트 문제? 검색 문제? 모델 문제? 인프라 문제?
   |
   v
5. 재현 및 검증
   - 문제 트레이스의 입력으로 동일 조건 재실행
   - Playground에서 프롬프트 변형 테스트
   |
   v
6. 수정 및 확인
   - 수정 적용 후 동일 입력으로 재실행하여 문제 해결 확인
```

1단계에서 문제를 인지합니다. 9.2.1에서 설정한 알림이 발동하거나, 사용자가 불만을 접수하거나, 자동 평가 점수가 떨어지는 것으로 문제가 드러납니다. 2단계에서 문제와 관련된 트레이스를 찾습니다. 특정 시간대, 특정 오류 코드, 특정 사용자에 대한 트레이스를 필터링합니다. 3단계에서 개별 트레이스의 스팬을 하나씩 열어 봅니다. 검색 스팬의 출력을 확인하여 관련 문서가 제대로 반환되었는지, 프롬프트 조합 스팬에서 의도한 프롬프트가 만들어졌는지, LLM 호출 스팬에서 모델의 응답이 적절한지를 순서대로 점검합니다. 4단계에서 원인의 범주를 좁힙니다. 검색 결과가 부적절하면 검색 문제이고, 프롬프트가 의도와 다르게 구성되었으면 프롬프트 문제이며, 프롬프트와 검색 결과 모두 정상인데 모델이 잘못된 응답을 내면 모델 문제입니다. 5단계에서 문제를 재현합니다. 문제 트레이스의 입력을 그대로 사용하여 동일 조건에서 다시 실행해 봅니다. 6단계에서 수정을 적용하고 동일 입력으로 다시 실행하여 문제가 해결되었는지 확인합니다.

수집된 로그와 트레이스를 어디에, 얼마나 오래 저장할지도 설계해야 합니다. 이것이 **로그 저장과 보존 정책**입니다. LLM 로그는 프롬프트와 응답 전문을 포함하므로, 전통적인 로그보다 용량이 훨씬 큽니다. 하나의 요청 로그에 프롬프트 수천 토큰과 응답 수백 토큰이 포함되면, 하루에 수만 건의 요청만으로도 수 기가바이트의 로그가 쌓입니다.

보존 정책에서 고려해야 할 항목은 네 가지입니다. 첫째, 저장 기간입니다. 디버깅 목적의 상세 로그(프롬프트/응답 전문)는 30일에서 90일 정도가 일반적입니다. 집계된 메트릭은 더 오래(1년 이상) 보존하여 장기 추세를 분석합니다. 둘째, 저장 계층입니다. 최근 데이터는 빠르게 검색할 수 있는 핫 스토리지에, 오래된 데이터는 비용이 낮은 콜드 스토리지에 보관합니다. 셋째, 접근 권한입니다. 프롬프트 전문에 사용자 개인정보가 포함될 수 있으므로, 로그 접근 권한을 제한해야 합니다. 넷째, 삭제 절차입니다. 보존 기간이 지난 로그를 자동으로 삭제하는 프로세스를 갖추어야 합니다.

마지막으로 **민감 데이터 마스킹**을 다루겠습니다. LLM 로그에는 사용자가 입력한 텍스트가 그대로 기록됩니다. 사용자가 이름, 전화번호, 주소 등의 개인정보를 입력했다면, 로그에도 해당 정보가 남습니다. 이 문제는 9.4.2에서 다루는 PII 보호와 직결됩니다. 로그를 저장하기 전에 PII를 탐지하여 마스킹하는 전처리 단계를 삽입하는 것이 일반적입니다.

마스킹의 적용 지점은 두 곳입니다. 하나는 로그를 저장소에 쓰기 직전입니다. 로그 파이프라인에 마스킹 필터를 추가하여, 저장되는 로그에서는 PII가 제거된 상태가 되도록 합니다. 다른 하나는 옵저버빌리티 플랫폼으로 트레이스를 전송하기 직전입니다. 외부 플랫폼을 사용하는 경우, 프롬프트 전문이 외부 서버로 전송되므로 전송 전에 마스킹을 적용해야 합니다.

```
민감 데이터 마스킹 적용 지점

[사용자 입력]
  "저는 홍길동이고 010-1234-5678로 연락 주세요"
      |
      v
[마스킹 필터]
  "저는 [PERSON]이고 [PHONE]로 연락 주세요"
      |
      +----> [로그 저장소] (마스킹된 텍스트만 저장)
      |
      +----> [옵저버빌리티 플랫폼] (마스킹된 텍스트만 전송)
```

마스킹을 적용하면 디버깅 시 원본 텍스트를 볼 수 없다는 단점이 있습니다. 이를 완화하기 위해, 마스킹과 원본의 대응표를 별도의 보안 저장소에 보관하고, 필요 시 권한이 있는 담당자만 원본을 복원하는 방식을 사용하기도 합니다.

정리하면, LLM 로깅은 프롬프트/응답 전문, 토큰 수, 모델 파라미터, 파이프라인 단계 정보를 구조화된 형식으로 기록해야 합니다. 분산 트레이싱은 Trace ID로 하나의 요청 경로를 엮고, Span으로 각 단계를 계층적으로 표현하여, 문제 발생 시 원인을 단계별로 추적할 수 있게 합니다. 디버깅 워크플로는 문제 감지, 트레이스 식별, 스팬 분석, 원인 분류, 재현, 수정의 여섯 단계로 진행합니다. 로그 보존 정책으로 저장 기간과 접근 권한을 통제하고, 민감 데이터 마스킹으로 PII가 로그에 노출되는 것을 방지합니다.

다음 단원인 9.2.3에서는 토큰 비용 최적화를 다룹니다. 토큰 기반 과금 구조를 분석하고, 프롬프트 압축, 캐싱, 모델 티어링 등의 전략으로 비용을 줄이는 방법을 살펴봅니다.

이 단원을 마치면 LLM 애플리케이션의 로깅과 트레이싱 체계를 설계하고 문제를 진단할 수 있습니다.
