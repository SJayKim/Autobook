# 2.2.2 프롬프트 버전 관리와 테스트

2.2.1에서 프롬프트를 코드에서 분리하고, 레지스트리와 카탈로그를 통해 중앙에서 관리하는 체계를 살펴보았습니다. 프롬프트를 중앙 저장소에서 관리하면, 다음으로 풀어야 할 문제가 분명해집니다. 프롬프트를 수정했을 때, 그 변경이 기존 기능을 망가뜨리지 않는다는 것을 어떻게 확인할 수 있을까요. 전통적인 소프트웨어 개발에서는 코드를 수정할 때마다 테스트를 실행하여 기존 기능이 정상인지 확인합니다. 프롬프트도 동일한 원칙을 적용해야 합니다. 이 단원에서는 프롬프트의 변경 이력을 체계적으로 추적하고, 변경이 품질에 미치는 영향을 검증하는 방법을 다룹니다.

프롬프트를 수정하는 상황은 다양합니다. 모델을 GPT-4에서 GPT-4o-mini로 교체했더니 기존 프롬프트로는 출력 형식이 깨지는 경우가 있습니다. 고객으로부터 "답변이 너무 딱딱하다"는 피드백을 받아 톤을 조정해야 할 수도 있습니다. 새로운 분류 범주를 추가해야 할 수도 있습니다. 이런 수정을 할 때마다 이전 버전의 프롬프트가 처리하던 입력들에 대해서도 여전히 올바르게 동작하는지 확인하지 않으면, 한쪽을 고치면서 다른 쪽을 망가뜨리는 일이 반복됩니다.

**프롬프트 버전 관리 전략**은 이 문제를 해결하기 위한 첫 번째 장치입니다. 버전 관리란 프롬프트의 모든 변경을 기록하고, 특정 시점의 프롬프트로 되돌아갈 수 있게 하며, 여러 버전을 동시에 비교할 수 있게 하는 체계를 뜻합니다.

프롬프트 버전 관리에서 가장 기본적인 원칙은, 프롬프트를 수정할 때 기존 버전을 덮어쓰지 않고 새 버전을 생성하는 것입니다. 2.2.1에서 프롬프트 레지스트리에 이름과 버전을 함께 기록한다고 했습니다. 고객 문의 분류 프롬프트의 버전이 1.0이었다면, 톤을 수정한 새 프롬프트는 1.1로 등록합니다. 분류 범주를 추가하는 더 큰 변경은 2.0으로 올릴 수 있습니다. 이렇게 하면 문제가 발생했을 때 이전 버전으로 즉시 되돌릴 수 있고, 두 버전의 텍스트를 나란히 비교하여 무엇이 달라졌는지 파악할 수 있습니다.

버전 번호를 어떻게 매길지는 조직의 규모와 관행에 따라 다릅니다. 소프트웨어에서 흔히 사용하는 시맨틱 버전 방식(major.minor.patch)을 프롬프트에도 적용할 수 있습니다. 출력 형식이 바뀌는 것처럼 하위 호환이 깨지는 변경은 major 번호를 올리고, 톤이나 표현을 미세하게 조정하는 변경은 minor 번호를 올리며, 오타 수정 같은 사소한 변경은 patch 번호를 올리는 식입니다. 중요한 것은 번호 체계 자체가 아니라, 모든 변경에 고유한 식별자가 부여되어 추적 가능하다는 점입니다.

프롬프트 버전 관리를 구현하는 가장 접근성 높은 방법은 **Git 기반 프롬프트 버전 관리**입니다. Git은 소프트웨어 소스 코드의 변경 이력을 관리하는 도구로 이미 널리 사용되고 있습니다. 프롬프트 파일을 Git 저장소에 넣으면, Git이 제공하는 변경 이력 추적, 브랜치, 병합, 되돌리기 기능을 프롬프트에도 그대로 활용할 수 있습니다.

Git 기반 관리의 전형적인 구조는 다음과 같습니다. 프로젝트의 Git 저장소 안에 프롬프트 전용 디렉토리를 만들고, 각 프롬프트를 별도의 파일로 저장합니다.

```
prompts/
  customer_inquiry_classifier/
    prompt.txt
    metadata.json
  document_summarizer/
    prompt.txt
    metadata.json
  code_review_assistant/
    prompt.txt
    metadata.json
```

이 구조에서 각 프롬프트는 자신의 디렉토리를 가지며, 프롬프트 텍스트(prompt.txt)와 메타데이터(metadata.json)를 분리하여 관리합니다. metadata.json에는 대상 모델, 용도, 변수 목록, 마지막 테스트 날짜 같은 정보를 넣습니다. Git의 커밋 기록이 곧 프롬프트의 변경 이력이 됩니다. 누가, 언제, 왜 프롬프트를 수정했는지 커밋 메시지에 남기면, 나중에 특정 변경의 맥락을 파악할 수 있습니다.

Git 기반 관리의 장점은 기존 개발 워크플로와 자연스럽게 통합된다는 것입니다. 코드 리뷰와 마찬가지로 프롬프트 변경도 풀 리퀘스트(Pull Request)를 통해 동료의 검토를 거칠 수 있습니다. 프롬프트 변경이 승인되면 메인 브랜치에 병합하고, CI/CD 파이프라인이 자동으로 테스트를 실행합니다. 단점은 프롬프트 전용 기능(변수 치환 미리보기, 실시간 성능 비교 등)을 Git 자체가 제공하지는 않는다는 것입니다. 이런 기능이 필요하면 2.2.1에서 다룬 전용 관리 도구와 Git을 병행합니다.

프롬프트의 변경 이력을 관리하는 것은 "무엇이 바뀌었는지"를 기록하는 일입니다. 그런데 "바뀐 것이 괜찮은지"를 확인하려면 테스트가 필요합니다. 여기서 **프롬프트 회귀 테스트**가 등장합니다. 회귀 테스트(regression test)란, 변경 이후에도 기존에 정상이던 기능이 여전히 정상인지 확인하는 테스트를 뜻합니다. 소프트웨어 개발에서 코드를 수정한 뒤 기존 테스트 케이스를 전부 다시 실행하는 것과 같은 원리입니다.

프롬프트 회귀 테스트에서는 코드 테스트와 다른 점이 하나 있습니다. 코드의 출력은 결정적(deterministic)입니다. 같은 입력에 항상 같은 출력이 나옵니다. 그런데 LLM의 출력은 확률적입니다. 같은 프롬프트와 같은 입력을 넣어도 실행할 때마다 표현이 달라질 수 있습니다. 따라서 프롬프트 회귀 테스트는 "출력이 정확히 이 문자열과 일치하는가"를 검증하는 방식이 아니라, "출력이 기대하는 기준을 만족하는가"를 검증하는 방식으로 설계해야 합니다.

예를 들어, 고객 문의 분류 프롬프트의 회귀 테스트에서는 다음과 같은 기준을 사용할 수 있습니다. "배송 관련 문의에 대해 category가 '배송'으로 분류되는가", "JSON 형식이 올바른가", "confidence 값이 high, medium, low 중 하나인가". 이런 기준은 LLM의 표현이 매번 달라지더라도 일관되게 검증할 수 있습니다.

프롬프트 회귀 테스트를 수행하려면 테스트에 사용할 입력-기대 출력 쌍이 필요합니다. 이것이 **골든 데이터셋**(Golden Dataset)입니다. 골든 데이터셋이란, 프롬프트의 품질을 검증하기 위해 사전에 준비한 입력과 기대 결과의 모음을 뜻합니다. "골든"이라는 표현은 이 데이터셋이 "정답"의 기준이 된다는 뜻입니다.

골든 데이터셋을 구축하는 과정은 다음과 같습니다. 첫째, 프롬프트가 처리해야 하는 대표적인 입력을 수집합니다. 실제 서비스에서 수집한 사용자 질문이 가장 좋고, 없다면 예상되는 입력을 수작업으로 작성합니다. 둘째, 각 입력에 대해 기대되는 결과를 사람이 직접 작성합니다. 분류 작업이라면 정답 범주를, 요약 작업이라면 핵심이 포함되어야 하는 키워드나 문장을 기록합니다. 셋째, 엣지 케이스를 의도적으로 포함합니다. 모호한 문의, 여러 범주에 걸치는 문의, 프롬프트 인젝션을 시도하는 입력 등 프롬프트가 실패하기 쉬운 케이스를 넣어야 테스트의 가치가 높아집니다.

골든 데이터셋의 규모는 프롬프트의 복잡도와 서비스의 중요도에 따라 다르지만, 최소 20~50개의 다양한 입력을 포함하는 것이 권장됩니다. 데이터셋이 너무 작으면 프롬프트 변경의 영향을 제대로 포착하지 못하고, 너무 크면 테스트 실행 비용(LLM API 호출 비용과 시간)이 부담이 됩니다.

골든 데이터셋이 준비되면, 프롬프트의 두 버전을 동일한 데이터셋으로 비교할 수 있습니다. 이것이 **프롬프트 성능 비교**, 즉 Evaluation Set을 활용한 평가입니다. 비교의 기본 흐름은 다음과 같습니다.

```
프롬프트 성능 비교 흐름

골든 데이터셋 (입력 N개 + 기대 결과)
       |
       +---> 프롬프트 v1.0 실행 ---> 출력 N개 ---> 평가 점수 A
       |
       +---> 프롬프트 v1.1 실행 ---> 출력 N개 ---> 평가 점수 B
       |
       v
  A vs B 비교 --> v1.1 채택 여부 결정
```

골든 데이터셋의 모든 입력에 대해 두 버전의 프롬프트를 각각 실행하고, 각 출력을 기대 결과와 비교하여 점수를 산출합니다. 분류 작업이라면 정확도(올바르게 분류한 비율)를, 생성 작업이라면 핵심 키워드 포함 여부, 형식 준수 여부, 또는 별도의 LLM 기반 평가(LLM-as-a-Judge)를 사용할 수 있습니다. LLM-as-a-Judge란, 또 다른 LLM에게 "이 출력이 기대 결과에 부합하는지 0~10점으로 평가하시오"라고 요청하는 방식입니다. 이 방식은 사람이 직접 평가하기 어려운 대량의 출력을 자동으로 평가할 수 있다는 장점이 있습니다.

비교 결과 v1.1이 v1.0보다 전반적으로 높은 점수를 받았다면 v1.1을 채택합니다. 특정 유형의 입력에서 v1.1이 더 낮은 점수를 받았다면, 해당 유형을 개선한 뒤 다시 비교합니다. 이렇게 변경 전후를 정량적으로 비교하면, "느낌상 좋아진 것 같다"가 아니라 "정확도가 85%에서 92%로 올랐다"처럼 근거 있는 판단이 가능해집니다.

마지막으로 **프롬프트 배포 파이프라인**을 살펴보겠습니다. 프롬프트의 변경, 테스트, 배포 과정을 자동화하는 흐름을 뜻합니다. 소프트웨어 개발에서 코드를 커밋하면 CI/CD 파이프라인이 자동으로 빌드, 테스트, 배포를 수행하는 것과 같은 원리입니다.

프롬프트 배포 파이프라인의 전형적인 흐름은 다음과 같습니다.

```
프롬프트 배포 파이프라인

1. 프롬프트 수정
   개발자가 프롬프트 파일을 수정하고 Git에 커밋
       |
       v
2. 자동 테스트 실행 (CI)
   골든 데이터셋으로 회귀 테스트 실행
   이전 버전 대비 성능 비교
       |
       v
3. 결과 확인
   테스트 통과 --> 코드 리뷰 요청
   테스트 실패 --> 개발자에게 알림, 수정 필요
       |
       v
4. 리뷰 및 승인
   동료가 프롬프트 변경 내용과 테스트 결과를 검토
       |
       v
5. 배포
   승인된 프롬프트를 프로덕션 레지스트리에 반영
   애플리케이션이 새 버전의 프롬프트를 자동으로 사용
```

이 파이프라인에서 2단계가 핵심입니다. 프롬프트가 변경될 때마다 자동으로 골든 데이터셋에 대한 테스트가 실행되고, 이전 버전과의 성능 차이가 보고됩니다. 테스트가 통과해야만 다음 단계로 넘어갈 수 있도록 설정하면, 품질이 떨어지는 프롬프트가 실수로 배포되는 것을 방지할 수 있습니다. 5단계에서 배포는 프롬프트 레지스트리의 프로덕션 버전을 갱신하는 것으로 이루어집니다. 2.2.1에서 설명한 것처럼, 애플리케이션 코드는 레지스트리에서 프롬프트를 가져오므로, 레지스트리의 버전만 갱신하면 애플리케이션 코드를 다시 배포하지 않아도 새 프롬프트가 적용됩니다.

프롬프트 배포에서 한 가지 더 고려할 점은 점진적 배포입니다. 새 프롬프트를 전체 트래픽에 즉시 적용하는 대신, 일부 트래픽에만 먼저 적용하여 실제 환경에서의 성능을 확인한 뒤 전체로 확대하는 방식입니다. 트래픽의 10%에 v1.1을 적용하고 나머지 90%는 v1.0을 유지한 뒤, v1.1의 성능 지표가 만족스러우면 비율을 점진적으로 높이는 것입니다. 이 방식은 소프트웨어 배포에서 사용하는 카나리(canary) 배포와 같은 원리입니다.

정리하면, 프롬프트 버전 관리는 모든 변경에 고유한 버전을 부여하고, 변경 이력을 추적하며, 필요시 이전 버전으로 되돌릴 수 있게 하는 체계입니다. Git을 활용하면 기존 개발 워크플로와 자연스럽게 통합할 수 있습니다. 프롬프트 회귀 테스트는 골든 데이터셋을 사용하여 변경 전후의 품질을 정량적으로 비교합니다. 골든 데이터셋에는 대표적인 입력과 엣지 케이스를 포함하고, LLM의 확률적 특성을 고려하여 기준 기반(정확도, 형식 준수 등)으로 평가합니다. 프롬프트 배포 파이프라인은 이 과정을 자동화하여, 수정된 프롬프트가 테스트를 통과한 뒤에만 프로덕션에 반영되도록 보장합니다.

다음 단원인 3.1.1에서는 텍스트 임베딩의 원리를 다룹니다. 텍스트를 숫자 벡터로 변환하는 임베딩의 개념과 유사도 계산 방식을 살펴봅니다.

이 단원을 마치면 프롬프트의 버전 관리 전략을 수립하고 회귀 테스트를 구성할 수 있습니다.
