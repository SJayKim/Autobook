# 1.1.3 Foundation Model Operations

1.1.2에서 기존 MLOps가 LLMOps로 전환되는 흐름을 살펴보았습니다. 전통적 머신러닝에서는 팀이 직접 모델을 설계하고, 데이터를 수집하여 처음부터 학습시키는 과정이 일반적이었습니다. 그런데 최근 몇 년 사이, 이 방식과는 근본적으로 다른 접근이 주류가 되었습니다. 누군가가 대규모 데이터로 미리 학습해 둔 범용 모델을 가져와서, 그 위에 자기 업무에 맞는 기능을 얹는 방식입니다. 이렇게 미리 학습된 범용 모델을 **Foundation Model**(파운데이션 모델)이라고 부릅니다. 이 단원에서는 Foundation Model이 무엇인지, 어떤 종류가 있는지, 그리고 이 모델을 운영할 때 어떤 과제가 생기는지를 다룹니다.

**Foundation Model**이라는 용어가 등장한 배경부터 짚겠습니다. 전통적 머신러닝에서는 '스팸 메일 분류'처럼 하나의 과제를 정하고, 그 과제에 맞는 데이터를 모아 모델을 처음부터 학습시켰습니다. 이렇게 만든 모델은 스팸 분류에는 뛰어나지만, 번역이나 요약 같은 다른 과제에는 쓸 수 없었습니다. 과제가 바뀔 때마다 데이터 수집과 학습을 처음부터 다시 해야 했습니다. 이 문제를 해결하기 위해 연구자들은 발상을 바꾸었습니다. 특정 과제에 맞추는 대신, 인터넷에 있는 방대한 텍스트를 활용하여 언어 자체의 패턴을 먼저 학습시키자는 것입니다. 이렇게 하면 하나의 모델이 번역, 요약, 질의응답, 코드 작성 등 여러 과제를 수행할 수 있는 '범용' 능력을 갖추게 됩니다. 이 범용 모델이 이후 다양한 응용의 기반, 즉 '토대(foundation)'가 된다는 의미에서 Foundation Model이라는 이름이 붙었습니다.

Foundation Model의 정의를 정리하면 다음과 같습니다. Foundation Model은 대규모 데이터로 **사전훈련**(pre-training)을 거쳐 범용적인 언어 이해 및 생성 능력을 갖춘 모델입니다. 여기서 '사전훈련'이란, 특정 업무를 지정하지 않고 텍스트의 다음 단어를 예측하는 방식 등으로 언어의 문법, 의미, 세상 지식을 대량으로 학습하는 과정을 뜻합니다. 사전훈련에는 수백 기가바이트에서 수 테라바이트에 이르는 텍스트 데이터가 사용되며, 수천 대의 GPU로 수주에서 수개월간 학습을 진행합니다. 이 과정의 결과물이 Foundation Model입니다.

**범용성**이라는 특성을 좀 더 구체적으로 살펴보겠습니다. 전통적 머신러닝 모델이 '스팸 분류'라는 하나의 과제만 수행할 수 있었다면, Foundation Model은 하나의 모델로 여러 과제를 처리합니다. 예를 들어, 같은 Foundation Model에 "이 이메일이 스팸인지 판단해 주세요"라고 요청하면 분류를 수행하고, "이 기사를 세 문장으로 요약해 주세요"라고 요청하면 요약을 수행합니다. 모델 자체를 바꾸지 않고, 요청하는 방식(프롬프트)만 바꾸면 다른 과제를 수행할 수 있다는 점이 범용성의 핵심입니다. 이 범용성 덕분에, 조직은 과제마다 별도의 모델을 학습시키는 대신 하나의 Foundation Model을 다양한 업무에 활용할 수 있습니다.

사전훈련만으로 끝나는 것은 아닙니다. Foundation Model은 사전훈련 이후 추가 단계를 거쳐 실제 서비스에 적합한 형태가 됩니다. 대표적인 추가 단계가 **파인튜닝**(fine-tuning)과 **정렬**(alignment)입니다. 파인튜닝은 사전훈련된 모델에 특정 도메인이나 과제의 데이터를 추가로 학습시켜, 해당 분야의 성능을 높이는 과정입니다. 정렬은 모델이 사용자의 의도에 맞게 답변하고, 유해하거나 부정확한 내용을 생성하지 않도록 조정하는 과정입니다. 사전훈련이 '언어를 이해하는 능력'을 만든다면, 파인튜닝과 정렬은 그 능력을 '실제 업무에 쓸 수 있는 형태'로 다듬는 단계입니다.

현재 널리 사용되는 Foundation Model의 종류를 살펴보겠습니다. 각 모델은 개발 주체, 공개 범위, 성능 특성이 서로 다르며, 이 차이가 운영 방식에 직접적인 영향을 미칩니다.

**GPT 계열**은 OpenAI가 개발한 모델입니다. GPT-3.5, GPT-4, GPT-4o 등의 버전이 있으며, API를 통해 사용하는 상용 모델입니다. 모델의 가중치(학습된 내부 파라미터)는 공개되지 않으며, 사용자는 OpenAI의 서버에 요청을 보내고 응답을 받는 방식으로 이용합니다.

**Claude 계열**은 Anthropic이 개발한 모델입니다. Claude 3 Haiku, Sonnet, Opus 등 성능 수준이 다른 여러 버전을 제공합니다. GPT와 마찬가지로 API 기반의 상용 모델이며, 모델 가중치는 비공개입니다.

**Gemini 계열**은 Google이 개발한 모델입니다. Gemini Flash, Pro, Ultra 등의 변형이 있으며, 텍스트뿐 아니라 이미지, 오디오 등 여러 형태의 입력을 처리하는 다중 모달(multimodal) 능력이 특징입니다. API를 통해 제공됩니다.

**LLaMA 계열**은 Meta가 개발하여 가중치를 공개한 모델입니다. LLaMA 2, LLaMA 3 등의 버전이 있으며, 연구 및 상업적 사용이 허용되는 라이선스 조건 아래 배포됩니다. 가중치가 공개되어 있으므로, 사용자가 자체 서버에 모델을 설치하고 직접 운영할 수 있습니다.

**Mistral 계열**은 프랑스의 Mistral AI가 개발한 오픈소스 모델입니다. 비교적 적은 파라미터로 높은 성능을 달성하여 주목받았으며, 자체 인프라에서 운영하기에 자원 부담이 상대적으로 적다는 장점이 있습니다.

이처럼 Foundation Model은 크게 두 가지 형태로 나뉩니다. 모델 가중치를 공개하지 않고 API로만 접근할 수 있는 **상용 API 모델**과, 가중치를 공개하여 누구나 내려받아 직접 운영할 수 있는 **오픈소스 모델**입니다. 이 구분은 단순한 비용 차이를 넘어서, 운영 방식 전체에 근본적인 차이를 만듭니다.

상용 API 모델을 사용하면, 모델을 직접 설치하거나 GPU 서버를 관리할 필요가 없습니다. API 제공 업체가 인프라를 운영하고, 사용자는 요청을 보내면 됩니다. 운영 부담이 적은 대신, 몇 가지 제약이 따릅니다. 첫째, API 호출마다 비용이 발생하므로 요청량이 늘어나면 비용이 빠르게 증가합니다. 둘째, 데이터가 외부 서버로 전송되기 때문에 민감한 데이터를 다루는 조직에서는 보안과 규정 준수에 대한 검토가 필요합니다. 셋째, 업체가 모델을 업데이트하거나 서비스 정책을 변경하면 사용자 측에서 통제할 수 없습니다. 넷째, 모델의 내부 동작을 수정하거나 파인튜닝할 수 있는 범위가 업체가 제공하는 기능에 한정됩니다.

오픈소스 모델을 사용하면, 모델 가중치를 내려받아 자체 서버에서 실행합니다. 데이터가 외부로 나가지 않으므로 보안 측면에서 유리하고, 모델을 자유롭게 파인튜닝하여 특정 업무에 최적화할 수 있습니다. 업체의 정책 변경에 영향을 받지 않는다는 점도 장점입니다. 그 대신, GPU 서버를 직접 확보하고 관리해야 합니다. 대규모 모델은 수십 기가바이트의 GPU 메모리를 요구하므로 인프라 비용이 상당합니다. 모델 서빙, 성능 최적화, 장애 대응까지 모두 운영 팀의 몫이 됩니다.

두 방식의 운영 차이를 정리하면 다음과 같습니다.

```
+-----------------+------------------------------+------------------------------+
|    구분         |      상용 API 모델           |       오픈소스 모델          |
+-----------------+------------------------------+------------------------------+
| 인프라 관리     | 불필요 (업체가 담당)         | 직접 GPU 서버 구축/관리      |
| 비용 구조       | 요청 단위 과금 (종량제)      | 인프라 고정 비용 + 운영 비용 |
| 데이터 보안     | 외부 서버로 데이터 전송      | 내부 서버에서 처리           |
| 모델 커스터마이즈| 업체 제공 범위 내 제한적     | 자유로운 파인튜닝 가능       |
| 모델 업데이트   | 업체 주도, 사용자 통제 불가  | 사용자가 업데이트 시점 결정  |
| 운영 복잡도     | 낮음                         | 높음                         |
+-----------------+------------------------------+------------------------------+
```

실제 현장에서는 두 방식 중 하나만 선택하지 않는 경우도 많습니다. 예를 들어, 빠른 프로토타이핑 단계에서는 상용 API 모델로 기능을 검증한 뒤, 트래픽이 안정되고 보안 요구가 구체화되면 오픈소스 모델로 전환하는 전략을 취하기도 합니다. 또는 일반적인 요청은 비용이 낮은 오픈소스 모델로 처리하고, 복잡한 요청만 상용 API 모델로 보내는 하이브리드 구성도 사용됩니다.

Foundation Model을 선택할 때는 여러 기준을 함께 고려해야 합니다. 가장 먼저 확인하는 것은 **성능**입니다. 모델이 수행해야 할 과제에서 얼마나 정확하고 유용한 결과를 내는지가 핵심입니다. 같은 모델이라도 과제에 따라 성능 차이가 크므로, 실제 업무 데이터로 평가해 보는 과정이 필요합니다. 벤치마크 점수는 참고 지표이지만, 특정 업무에서의 실제 성능과 반드시 일치하지는 않습니다.

다음으로 **비용**을 고려합니다. 상용 API 모델의 경우 입력 토큰과 출력 토큰 수에 따라 요금이 책정됩니다. 하루에 수천 건의 요청을 처리하는 서비스라면, 건당 단가가 조금만 달라도 월간 비용이 크게 변합니다. 오픈소스 모델의 경우 API 사용료는 없지만, GPU 서버 임대비와 운영 인건비를 합산하여 총소유비용(TCO)을 산정해야 합니다.

**라이선스** 조건도 빠뜨릴 수 없습니다. 오픈소스 모델이라고 해서 모두 같은 조건으로 사용할 수 있는 것은 아닙니다. 모델마다 상업적 사용 가능 여부, 파생 모델 배포 조건, 사용 제한 분야 등이 다릅니다. 예를 들어, 어떤 모델은 연구 목적으로만 사용할 수 있고, 어떤 모델은 상업적 사용을 허용하되 일정 규모 이상의 기업에는 별도 계약을 요구하기도 합니다. 라이선스를 확인하지 않고 서비스에 적용하면, 나중에 법적 문제가 발생할 수 있습니다.

이 세 가지 외에도, 모델이 지원하는 언어(한국어 성능이 중요한 서비스라면 이 점이 결정적일 수 있습니다), 컨텍스트 윈도우 크기(한 번에 처리할 수 있는 텍스트 길이), 응답 지연 시간 등을 함께 검토합니다.

마지막으로, Foundation Model을 운영할 때 마주하는 고유한 과제를 살펴보겠습니다. 1.1.1에서 LLMOps의 핵심 영역을 개관했고, 1.1.2에서 전통적 ML 운영과의 차이를 다루었습니다. Foundation Model 운영에서는 이러한 차이가 구체적인 과제로 드러납니다.

첫 번째 과제는 **추론 비용 관리**입니다. Foundation Model은 파라미터 수가 수십억에서 수천억에 이르며, 한 번의 요청을 처리하는 데에도 상당한 연산 자원이 필요합니다. 전통적 ML 모델은 한 번 추론하는 데 밀리초 단위의 시간과 미미한 비용이 들었지만, Foundation Model은 요청 하나에 수십 밀리초에서 수 초가 걸리고 비용도 눈에 띄게 발생합니다. 서비스 규모가 커질수록 이 비용이 빠르게 누적되므로, 비용을 통제하면서도 품질을 유지하는 전략이 필수적입니다.

두 번째 과제는 **환각**(hallucination) 대응입니다. Foundation Model은 학습 데이터에 없는 내용을 마치 사실인 것처럼 생성하는 경우가 있습니다. 이를 환각이라고 합니다. 전통적 ML 모델이 '정확도 90%'처럼 정량적으로 신뢰도를 측정할 수 있었던 것과 달리, Foundation Model의 출력은 자연어 문장이기 때문에 정답과 오답의 경계가 모호합니다. 운영 환경에서는 모델의 출력이 사실에 부합하는지 검증하는 체계를 갖추어야 합니다.

세 번째 과제는 **버전 관리와 재현성**입니다. 상용 API 모델을 사용하는 경우, 업체가 모델을 업데이트하면 같은 프롬프트에 대한 응답이 달라질 수 있습니다. 지난주까지 잘 동작하던 기능이 모델 업데이트 이후 갑자기 품질이 떨어지는 상황이 발생할 수 있습니다. 오픈소스 모델을 직접 운영하더라도, 모델 가중치 버전, 프롬프트 버전, 설정 파라미터를 함께 관리하지 않으면 결과를 재현하기 어렵습니다.

네 번째 과제는 **지연 시간**(latency) 관리입니다. Foundation Model은 토큰을 하나씩 순차적으로 생성하므로, 출력 길이가 길어질수록 응답 시간이 늘어납니다. 사용자가 실시간으로 상호작용하는 서비스에서는 이 지연이 사용자 경험에 직접적인 영향을 미칩니다. 모델 경량화, 캐싱, 스트리밍 응답 등의 기법으로 지연 시간을 줄이는 것이 운영의 주요 관심사입니다.

다섯 번째 과제는 **보안과 거버넌스**입니다. Foundation Model에 입력되는 데이터에 개인정보나 기업 기밀이 포함될 수 있습니다. 또한 모델이 부적절하거나 편향된 내용을 생성할 위험도 있습니다. 어떤 데이터가 모델에 입력되고 어떤 출력이 사용자에게 전달되는지를 추적하고 통제하는 체계가 필요합니다.

```
Foundation Model 운영의 고유 과제

+------------------+     +------------------+     +------------------+
| 추론 비용 관리   |     | 환각 대응        |     | 버전 관리/재현성 |
| - 대규모 연산    |     | - 사실성 검증    |     | - 모델 버전 추적 |
| - 비용 최적화    |     | - 출력 품질 보증 |     | - 프롬프트 버전  |
+------------------+     +------------------+     +------------------+

+------------------+     +------------------+
| 지연 시간 관리   |     | 보안/거버넌스    |
| - 실시간 응답    |     | - 데이터 보호    |
| - 경량화/캐싱    |     | - 편향 통제      |
+------------------+     +------------------+
```

이 과제들은 서로 독립적이지 않습니다. 예를 들어, 비용을 줄이기 위해 더 작은 모델을 선택하면 환각이 늘어날 수 있고, 보안을 위해 오픈소스 모델을 자체 운영하면 인프라 비용과 운영 복잡도가 올라갑니다. 이처럼 과제 간의 균형을 잡는 것이 Foundation Model 운영의 핵심이며, 이것이 곧 LLMOps가 체계적으로 다루어야 할 영역입니다.

정리하면, Foundation Model은 대규모 데이터로 사전훈련하여 범용적인 언어 능력을 갖춘 모델이며, GPT, Claude, Gemini 같은 상용 API 모델과 LLaMA, Mistral 같은 오픈소스 모델로 나뉩니다. 모델을 선택할 때는 성능, 비용, 라이선스를 종합적으로 평가해야 하며, 운영 단계에서는 추론 비용, 환각, 버전 관리, 지연 시간, 보안과 거버넌스라는 고유한 과제에 대응해야 합니다. 이 과제들이 바로 이후 단원에서 하나씩 깊이 다루게 될 LLMOps의 실질적인 대상입니다.

다음 단원인 1.2.1에서는 대규모 언어 모델의 구조를 다룹니다. Foundation Model의 내부가 어떤 구조로 이루어져 있는지, Transformer 아키텍처의 핵심 구성요소를 살펴봅니다.

이 단원을 마치면 Foundation Model의 개념과 운영 특성을 설명할 수 있습니다.
