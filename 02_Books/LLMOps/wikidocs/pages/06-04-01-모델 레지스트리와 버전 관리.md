# 6.4.1 모델 레지스트리와 버전 관리

LLM 프로젝트에서는 모델이 한 종류만 존재하는 일이 드뭅니다. 기본 모델을 다운로드한 뒤 양자화를 적용한 버전, 특정 도메인 데이터로 파인튜닝한 버전, 프롬프트를 변경하여 테스트 중인 버전 등이 동시에 존재합니다. 6.1.2에서 vLLM으로 서빙할 때 모델 이름을 지정하여 Hugging Face Hub에서 가중치를 다운로드하는 과정을 살펴보았는데, 모델이 여러 개가 되면 어떤 모델이 현재 프로덕션에 배포되어 있고, 어떤 모델이 테스트 중이며, 각 모델이 어떤 데이터와 설정으로 만들어졌는지를 추적하는 일이 중요해집니다. 이 추적 체계가 없으면 "지난주에 배포한 모델이 어떤 하이퍼파라미터로 학습된 건지 모르겠다"거나 "두 달 전 좋았던 모델로 롤백하고 싶은데 어디에 저장했는지 찾을 수 없다"는 상황이 발생합니다. 이런 문제를 해결하는 시스템이 **모델 레지스트리(Model Registry)**입니다.

모델 레지스트리란, 학습이 완료된 모델을 중앙에 등록하고, 각 모델의 메타데이터와 버전 이력을 관리하며, 필요할 때 특정 버전의 모델을 꺼내 쓸 수 있게 하는 저장소입니다. 소프트웨어 개발에서 코드 저장소(Git 리포지토리)가 소스 코드의 버전을 관리하는 것과 비슷합니다. 차이는, 모델 레지스트리가 관리하는 대상이 코드가 아니라 수 기가바이트에 달하는 모델 가중치 파일과 그에 딸린 메타데이터라는 점입니다.

모델 레지스트리가 관리하는 정보를 구체적으로 나열하면 다음과 같습니다. 모델 가중치 파일 자체, 모델을 만들 때 사용한 하이퍼파라미터(학습률, 배치 크기, 에포크 수 등), 학습에 사용한 데이터셋의 식별 정보, 평가 지표(정확도, 펄플렉서티, 벤치마크 점수 등), 모델 아키텍처와 프레임워크 버전, 라이선스 정보, 그리고 모델의 상태(실험 중, 스테이징, 프로덕션 등)입니다.

이 정보들이 왜 필요한지 구체적인 시나리오를 들어 설명하겠습니다. 팀에서 Llama 8B 모델을 고객 서비스 데이터로 파인튜닝하여 v1을 만들고 배포했다고 가정합니다. 한 달 뒤 새 데이터를 추가하여 v2를 만들었는데, v2의 응답 품질이 기대보다 낮습니다. 이때 v1으로 즉시 롤백해야 합니다. 모델 레지스트리가 있으면 v1의 가중치를 바로 꺼내 서빙 프레임워크에 로드할 수 있습니다. 모델 레지스트리가 없으면 v1의 가중치가 어느 서버의 어느 디렉토리에 있는지, 또는 이미 삭제되었는지 확인하는 것부터 시작해야 합니다.

이제 실무에서 가장 많이 사용되는 두 가지 모델 관리 시스템을 살펴보겠습니다. **Hugging Face Hub**과 **MLflow Model Registry**입니다.

**Hugging Face Hub**는 Hugging Face가 운영하는 모델 호스팅 플랫폼입니다. 6.1.2에서 vLLM으로 서빙할 때 `meta-llama/Llama-3.1-8B-Instruct`라는 모델 식별자를 사용했는데, 이 식별자가 가리키는 곳이 바로 Hugging Face Hub의 모델 저장소입니다. Hugging Face Hub은 오픈소스 모델의 공유 플랫폼으로 시작했지만, 조직 내부에서 비공개 모델을 관리하는 용도로도 사용할 수 있습니다.

Hugging Face Hub에서 모델은 Git 기반 저장소로 관리됩니다. 모델 가중치, 설정 파일, 토크나이저 파일 등이 하나의 저장소에 담기며, Git의 버전 관리 기능을 활용하여 파일의 변경 이력을 추적합니다. 대용량 가중치 파일은 Git LFS(Large File Storage)를 통해 저장합니다. 모델 저장소에는 모델 카드(Model Card)라는 문서가 포함되어 있어, 모델의 아키텍처, 학습 데이터, 성능 지표, 사용 제한 사항 등을 기술합니다.

Hugging Face Hub에 모델을 업로드하고 관리하는 방법을 살펴보겠습니다. `huggingface_hub` 라이브러리를 사용합니다.

```python
from huggingface_hub import HfApi

api = HfApi()

# 모델 저장소 생성
api.create_repo(
    repo_id="my-org/llama-8b-customer-service-v1",
    private=True
)

# 모델 파일 업로드
api.upload_folder(
    folder_path="./model_output",
    repo_id="my-org/llama-8b-customer-service-v1",
    commit_message="Initial fine-tuned model v1"
)
```

`create_repo`는 새 모델 저장소를 생성합니다. `repo_id`는 "조직명/모델명" 형식입니다. `private=True`로 설정하면 조직 외부에서 접근할 수 없는 비공개 저장소가 됩니다. `upload_folder`는 로컬 디렉토리의 파일을 저장소에 업로드합니다. `commit_message`는 Git 커밋 메시지와 같은 역할을 하여, 이 업로드가 어떤 변경인지를 기록합니다.

새 버전의 모델을 같은 저장소에 덮어쓰면 이전 버전의 가중치가 사라집니다. 버전을 분리하려면 두 가지 방법이 있습니다. 첫째, 저장소 이름에 버전을 포함하는 방법입니다. `my-org/llama-8b-cs-v1`, `my-org/llama-8b-cs-v2`처럼 별도 저장소를 만듭니다. 둘째, Git의 브랜치나 태그를 활용하는 방법입니다. 같은 저장소에서 `v1`, `v2` 태그를 붙여 특정 시점의 스냅샷을 보존합니다.

```python
# 태그를 사용한 버전 관리
api.create_tag(
    repo_id="my-org/llama-8b-customer-service",
    tag="v1.0",
    tag_message="Production release v1.0"
)

# 특정 태그의 모델 다운로드
from huggingface_hub import snapshot_download

snapshot_download(
    repo_id="my-org/llama-8b-customer-service",
    revision="v1.0",
    local_dir="./models/v1.0"
)
```

`create_tag`는 현재 저장소 상태에 태그를 붙입니다. `snapshot_download`에서 `revision="v1.0"`을 지정하면 해당 태그 시점의 파일을 다운로드합니다. 이 방식으로 이전 버전의 모델을 언제든 복원할 수 있습니다.

다음으로 **MLflow Model Registry**를 살펴보겠습니다. MLflow는 머신러닝 실험의 전체 생명주기를 관리하는 오픈소스 플랫폼입니다. MLflow의 여러 구성요소 중 Model Registry가 모델 버전 관리를 담당합니다. Hugging Face Hub이 모델 파일의 호스팅과 공유에 초점을 맞추었다면, MLflow Model Registry는 모델의 생명주기 관리에 초점을 맞춥니다.

MLflow Model Registry에서 모델은 세 가지 상태를 가질 수 있습니다. "Staging"은 테스트 중인 모델, "Production"은 실제 서비스에 배포된 모델, "Archived"는 더 이상 사용하지 않지만 이력 보존을 위해 보관하는 모델입니다. 이 상태 관리 덕분에 팀 전체가 어떤 모델이 현재 프로덕션에 있는지, 어떤 모델이 다음 배포 후보인지를 한눈에 파악할 수 있습니다.

MLflow에서 모델을 등록하고 관리하는 흐름을 살펴보겠습니다.

```python
import mlflow

# 실험 실행 중 모델 로깅
with mlflow.start_run():
    # 하이퍼파라미터 기록
    mlflow.log_param("learning_rate", 2e-5)
    mlflow.log_param("epochs", 3)
    mlflow.log_param("batch_size", 8)

    # 성능 지표 기록
    mlflow.log_metric("eval_loss", 1.23)
    mlflow.log_metric("perplexity", 3.42)

    # 모델 아티팩트 저장 및 레지스트리 등록
    mlflow.pyfunc.log_model(
        artifact_path="model",
        python_model=model_wrapper,
        registered_model_name="llama-8b-customer-service"
    )
```

`start_run`은 하나의 실험 실행을 시작합니다. `log_param`은 하이퍼파라미터를 기록하고, `log_metric`은 평가 지표를 기록합니다. `log_model`은 모델 아티팩트를 저장하면서 동시에 Model Registry에 등록합니다. `registered_model_name`에 같은 이름을 사용하면, MLflow가 자동으로 버전 번호를 증가시켜 새 버전으로 등록합니다. 첫 번째 등록이면 Version 1, 두 번째면 Version 2가 됩니다.

등록된 모델의 상태를 변경하는 방법은 다음과 같습니다.

```python
from mlflow import MlflowClient

client = MlflowClient()

# Version 2를 Staging으로 전환
client.transition_model_version_stage(
    name="llama-8b-customer-service",
    version=2,
    stage="Staging"
)

# 테스트 완료 후 Production으로 전환
client.transition_model_version_stage(
    name="llama-8b-customer-service",
    version=2,
    stage="Production"
)
```

이 코드에서 `transition_model_version_stage`는 특정 버전의 모델을 지정한 상태로 전환합니다. Version 2를 먼저 Staging에 올려 테스트한 뒤, 문제가 없으면 Production으로 승격시키는 흐름입니다. 이전에 Production이었던 버전은 자동으로 Archived 상태가 되거나, 수동으로 Archived로 전환합니다.

```
MLflow Model Registry의 모델 생명주기

  실험 실행
     |
     v
  [모델 로깅 + 레지스트리 등록]
     |
     v
  Version N 생성 (상태: None)
     |
     v
  Staging (테스트/검증)
     |
     +-- 검증 실패 --> Archived
     |
     +-- 검증 성공 --> Production (서비스 배포)
                          |
                          +-- 새 버전 배포 시 --> Archived
```

이제 **모델 메타데이터 관리**를 더 자세히 살펴보겠습니다. 모델 레지스트리에 가중치만 저장하고 메타데이터를 기록하지 않으면, 나중에 그 모델이 어떻게 만들어졌는지 추적할 수 없습니다. 관리해야 할 메타데이터는 크게 네 가지 범주로 나뉩니다.

첫째, **학습 하이퍼파라미터**입니다. 학습률, 배치 크기, 에포크 수, 옵티마이저 종류, 웜업 스텝 수 등이 여기에 해당합니다. 파인튜닝의 경우 LoRA의 rank 값이나 alpha 값도 포함됩니다.

둘째, **성능 지표**입니다. 학습 손실, 검증 손실, 펄플렉서티, 벤치마크 점수(MMLU, HumanEval 등), 그리고 도메인 특화 평가 결과가 이에 해당합니다. 성능 지표는 버전 간 비교의 기준이 되므로, 측정 시점과 평가 데이터셋도 함께 기록해야 합니다.

셋째, **라이선스 정보**입니다. 기반 모델의 라이선스(Llama Community License, Apache 2.0, MIT 등)에 따라 상업적 사용이 제한되거나 모델 재배포 시 조건이 붙을 수 있습니다. 파인튜닝 모델은 기반 모델의 라이선스를 상속하는 경우가 많으므로, 이를 명시적으로 기록해 두어야 합니다.

넷째, **환경 정보**입니다. 모델 학습에 사용한 프레임워크 버전(PyTorch, Transformers 등), GPU 종류, CUDA 버전 등을 기록합니다. 모델을 재현하거나 서빙 환경을 구성할 때 이 정보가 필요합니다.

다음으로 **모델 버전 관리 전략**을 살펴보겠습니다. 모델 버전을 어떻게 체계화할지는 프로젝트 규모와 팀 구조에 따라 달라지지만, 일반적으로 적용할 수 있는 원칙들이 있습니다.

버전 번호 체계는 소프트웨어의 시맨틱 버저닝(Semantic Versioning)과 유사하게 구성할 수 있습니다. 주 버전(Major)은 기반 모델이 바뀌거나 학습 데이터가 대폭 변경된 경우에 올립니다. 부 버전(Minor)은 학습 데이터가 소폭 추가되거나 하이퍼파라미터가 조정된 경우에 올립니다. 패치 버전(Patch)은 양자화 방식 변경이나 서빙 설정 조정처럼 모델 가중치 자체는 동일하지만 배포 형태가 달라진 경우에 올립니다.

마지막으로 **모델 아티팩트 저장과 배포 파이프라인**을 다루겠습니다. 모델 아티팩트란 서빙에 필요한 모든 파일의 묶음을 말합니다. 가중치 파일, 토크나이저 파일, 설정 파일, 그리고 필요한 경우 양자화 설정 파일까지 포함됩니다. 이 아티팩트를 저장소에 올려 두고, 배포 시 서빙 서버가 저장소에서 아티팩트를 내려받아 모델을 로드하는 방식이 일반적입니다.

배포 파이프라인은 모델 레지스트리에서 특정 버전의 모델을 가져와 서빙 환경에 배포하는 자동화된 흐름입니다. 일반적인 파이프라인은 다음과 같은 단계로 구성됩니다. 모델 레지스트리에서 Production 상태의 모델 버전을 확인합니다. 해당 버전의 아티팩트를 서빙 서버로 전송합니다. 서빙 프레임워크(vLLM, TGI 등)가 모델을 로드합니다. 상태 점검(health check)을 수행하여 모델이 정상적으로 응답하는지 확인합니다. 이상이 없으면 트래픽을 새 모델로 전환합니다.

```
모델 배포 파이프라인

 [모델 레지스트리]
       |
       | Production 버전 확인
       v
 [아티팩트 다운로드]
       |
       | 가중치, 토크나이저, 설정 파일
       v
 [서빙 서버에 모델 로드]
       |
       | vLLM / TGI 등
       v
 [상태 점검 (Health Check)]
       |
       +-- 실패 --> 이전 버전 유지, 알림 발송
       |
       +-- 성공 --> 트래픽 전환
                       |
                       v
                 [서비스 운영]
```

이 파이프라인을 CI/CD 도구(GitHub Actions, GitLab CI, Jenkins 등)와 연동하면, 모델 레지스트리에서 새 버전이 Production으로 전환될 때 자동으로 배포가 시작되도록 구성할 수 있습니다.

정리하면, 모델 레지스트리는 학습된 모델의 가중치와 메타데이터를 중앙에서 관리하고, 버전 이력을 추적하며, 필요할 때 특정 버전을 복원할 수 있게 하는 시스템입니다. Hugging Face Hub는 Git 기반 저장소로 모델을 호스팅하고 공유하는 데 적합하며, MLflow Model Registry는 모델의 생명주기 상태 관리(Staging, Production, Archived)에 강점이 있습니다. 모델 메타데이터(하이퍼파라미터, 성능 지표, 라이선스, 환경 정보)를 체계적으로 기록해야 버전 간 비교와 롤백이 가능합니다. 배포 파이프라인을 자동화하면 레지스트리의 상태 변경이 곧바로 서빙 환경에 반영됩니다.

다음 단원인 7.1.1에서는 파인튜닝의 기본 개념과 전략을 다룹니다. 모델 레지스트리에 등록할 새 버전의 모델을 만드는 핵심 방법인 파인튜닝이 무엇이고, 어떤 전략으로 접근하는지를 살펴봅니다.

이 단원을 마치면 모델 레지스트리의 구성요소를 설명하고, Hugging Face Hub과 MLflow를 활용한 버전 관리 전략을 수립할 수 있습니다.
