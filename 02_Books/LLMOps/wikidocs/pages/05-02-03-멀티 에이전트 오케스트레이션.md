# 5.2.3 멀티 에이전트 오케스트레이션

5.2.1에서 LLM 에이전트의 구성요소와 에이전트 루프를 살펴보았고, 5.2.2에서 에이전트가 외부 도구를 호출하는 매커니즘을 다루었습니다. 지금까지 다룬 에이전트는 하나의 LLM이 모든 추론과 행동을 담당하는 단일 에이전트였습니다. 단순한 작업이라면 하나의 에이전트로 충분하지만, 작업이 복잡해질수록 한계가 드러납니다. 이 단원에서는 여러 에이전트가 협업하여 작업을 수행하는 멀티 에이전트 시스템의 필요성과 대표적인 구성 패턴, 그리고 디버깅 방법을 다룹니다.

하나의 에이전트가 모든 작업을 처리할 때 발생하는 문제를 구체적으로 살펴보겠습니다. 첫째, 프롬프트가 비대해집니다. 에이전트가 코드 작성, 웹 검색, 데이터 분석, 보고서 작성을 모두 해야 한다면, 시스템 프롬프트에 모든 역할에 대한 지시와 수십 개의 도구 스키마를 담아야 합니다. 1.2.3에서 다룬 컨텍스트 윈도우의 토큰 제한에 가까워지면, 중요한 맥락이 밀려나거나 모델의 추론 정확도가 떨어집니다. 둘째, 역할이 모호해집니다. 하나의 에이전트에게 "연구원이면서 작가이면서 편집자"가 되라고 하면, 어느 역할의 관점에서 판단해야 하는지 혼란이 생깁니다. 셋째, 하나의 단계에서 실패하면 전체가 중단됩니다. 에이전트가 데이터 수집 중 오류를 만나면, 이미 완료한 분석 결과까지 처음부터 다시 진행해야 할 수 있습니다.

**멀티 에이전트 시스템**은 이 문제들을 역할 분리로 해결합니다. 각 에이전트가 특정 역할만 맡고, 해당 역할에 필요한 도구와 프롬프트만 가지므로, 개별 에이전트의 복잡도가 낮아집니다. 코드를 작성하는 에이전트에게는 코드 실행 도구만 주고, 웹 검색 에이전트에게는 검색 도구만 줍니다. 각자의 프롬프트가 짧고 명확해지므로 추론 정확도도 올라갑니다.

그러면 여러 에이전트를 어떻게 조율하고 작업을 분배하느냐가 다음 문제입니다. 이 조율 방식에 따라 멀티 에이전트 시스템을 여러 패턴으로 분류할 수 있습니다. 대표적인 네 가지 패턴을 살펴보겠습니다.

첫 번째는 **감독자(Supervisor) 패턴**입니다. 하나의 에이전트가 감독자 역할을 맡아, 사용자의 요청을 받고 하위 에이전트에게 작업을 배분합니다. 감독자 에이전트는 직접 작업을 수행하지 않습니다. 대신 "이 작업은 연구 에이전트에게", "이 작업은 작성 에이전트에게"처럼 적절한 하위 에이전트를 선택하여 작업을 위임합니다. 하위 에이전트가 결과를 반환하면, 감독자가 이를 검토하고 다음 하위 에이전트에게 넘기거나, 추가 작업이 필요하면 다시 위임하거나, 최종 결과를 종합하여 사용자에게 전달합니다.

```
감독자(Supervisor) 패턴

[사용자 요청]
      |
      v
+------------+
|   감독자   |  <-- 작업 배분, 결과 검토, 최종 종합
+------------+
   |    |    |
   v    v    v
 +--+ +--+ +--+
 |연| |작| |검|
 |구| |성| |토|   <-- 각자 전문 영역 수행
 +--+ +--+ +--+
   |    |    |
   +----+----+
        |
        v
+------------+
|   감독자   |  <-- 결과 수집, 최종 답변 구성
+------------+
      |
      v
[최종 답변]
```

이 패턴의 장점은 구조가 단순하고 제어가 명확하다는 것입니다. 감독자가 전체 흐름을 관장하므로, 어떤 에이전트가 어떤 순서로 실행되었는지 추적하기 쉽습니다. 단점은 감독자가 병목이 될 수 있다는 점입니다. 모든 작업 배분과 결과 검토가 감독자를 거쳐야 하므로, 감독자의 추론 능력이 전체 시스템의 성능 상한을 결정합니다.

두 번째는 **위계적(Hierarchical) 패턴**입니다. 감독자 패턴을 확장한 것으로, 감독자-하위 에이전트 구조가 여러 층으로 쌓입니다. 최상위 감독자가 중간 관리자 에이전트에게 작업을 넘기고, 중간 관리자가 다시 전문 에이전트에게 세부 작업을 배분하는 구조입니다. 대규모 조직의 관리 체계와 유사합니다.

예를 들어, 소프트웨어 개발 프로젝트를 수행하는 시스템을 생각해 보겠습니다. 최상위 프로젝트 매니저 에이전트가 "설계팀"과 "개발팀"과 "테스트팀"이라는 중간 관리자 에이전트에게 작업을 넘깁니다. "개발팀" 관리자 에이전트는 다시 "프론트엔드 개발자", "백엔드 개발자", "데이터베이스 관리자"라는 전문 에이전트에게 세부 작업을 배분합니다. 각 층위의 에이전트는 자기 아래 에이전트만 관리하면 되므로, 한 에이전트가 감당해야 할 복잡도가 줄어듭니다.

```
위계적(Hierarchical) 패턴

            [프로젝트 매니저]
           /       |        \
     [설계팀]   [개발팀]   [테스트팀]
                /    |    \
          [프론트]  [백엔드]  [DB]
```

이 패턴은 작업 규모가 크고 전문 분야가 다양할 때 유리합니다. 하지만 계층이 깊어질수록 통신 비용이 증가하고, 상위 에이전트가 하위 에이전트의 결과를 요약하여 올려보내는 과정에서 정보가 손실될 수 있습니다.

세 번째는 **토론(Debate) 패턴**입니다. 여러 에이전트가 동일한 문제에 대해 각자의 답을 제시하고, 서로의 답을 비판하며, 반복적인 토론을 통해 합의에 도달하는 방식입니다. 감독자나 위계가 없고, 모든 에이전트가 동등한 위치에서 논쟁합니다.

동작 과정을 단계별로 설명하겠습니다. 첫 번째 라운드에서 각 에이전트가 독립적으로 문제에 대한 답을 생성합니다. 두 번째 라운드에서 각 에이전트가 다른 에이전트들의 답을 읽고, 자기 답을 수정하거나 반론을 제기합니다. 이 과정을 정해진 횟수만큼 반복하거나, 답이 수렴할 때까지 계속합니다. 최종적으로 다수결이나 종합을 통해 하나의 결론을 도출합니다.

```
토론(Debate) 패턴

라운드 1:  에이전트 A: "답은 X"    에이전트 B: "답은 Y"    에이전트 C: "답은 X"
              |                       |                       |
              v                       v                       v
라운드 2:  A: "B의 Y는 ~ 때문에    B: "A와 C의 X는        C: "B의 근거가
              부정확, X 유지"          타당, X로 수정"         약함, X 유지"
              |                       |                       |
              v                       v                       v
결론:      X로 합의
```

토론 패턴은 정답의 정확도가 중요한 작업에 효과적입니다. 여러 에이전트가 서로 다른 관점에서 문제를 바라보고 상호 검증하므로, 단일 에이전트가 범할 수 있는 오류를 걸러낼 가능성이 높습니다. 반면, 라운드마다 모든 에이전트가 추론을 수행하므로 비용이 크고, 합의에 도달하지 못하면 무한 반복의 위험이 있어 최대 라운드 수를 제한해야 합니다.

네 번째는 **Swarm 패턴**입니다. 중앙 통제 없이, 에이전트들이 자율적으로 작업을 수행하며 필요할 때 다른 에이전트에게 제어권을 넘기는 방식입니다. 각 에이전트는 자신이 처리할 수 있는 작업이면 직접 처리하고, 자신의 영역 밖이면 적절한 다른 에이전트로 '핸드오프(handoff)'합니다.

고객 서비스 시스템을 예로 들면, 접수 에이전트가 사용자의 문의를 받아 분류한 뒤, 결제 관련이면 결제 전문 에이전트로, 기술 문제면 기술 지원 에이전트로 핸드오프합니다. 기술 지원 에이전트가 작업을 처리하다가 환불이 필요하다고 판단하면 결제 에이전트로 다시 핸드오프합니다. 감독자가 개입하지 않고 에이전트 간에 직접 제어권이 이동합니다.

```
Swarm 패턴

[사용자] --> [접수 에이전트]
                   |
          결제 문의?   기술 문의?
           /              \
  [결제 에이전트]    [기술 에이전트]
        ^                  |
        |     환불 필요    |
        +------------------+
   (에이전트 간 직접 핸드오프)
```

Swarm 패턴은 유연하고 확장성이 좋습니다. 새로운 전문 에이전트를 추가할 때 기존 에이전트를 수정할 필요 없이, 핸드오프 규칙만 정의하면 됩니다. 반면, 전체 흐름을 중앙에서 파악하기 어렵고, 핸드오프가 순환하여 무한 루프에 빠질 위험이 있으므로 이를 방지하는 장치가 필요합니다.

```
멀티 에이전트 패턴 비교

패턴          제어 구조       적합한 상황            주요 위험
-------       ----------      ---------------        ----------
감독자        중앙 집중       범용 작업 분배         감독자 병목
위계적        다층 중앙       대규모, 다전문 분야    정보 손실
토론          분산, 대등      정확도 중시 추론       비용, 무한 반복
Swarm         분산, 자율      동적 워크플로          추적 어려움
```

여러 에이전트가 협업하려면 에이전트 간에 정보를 주고받는 **통신**과, 작업 진행 상황을 공유하는 **상태 공유** 매커니즘이 필요합니다.

가장 단순한 통신 방식은 메시지 전달입니다. 한 에이전트의 출력을 다른 에이전트의 입력으로 직접 전달합니다. 5.2.1에서 다룬 AutoGen이 이 방식을 사용합니다. 에이전트들이 대화 형태로 메시지를 주고받으며 작업을 진행합니다.

상태 공유는 여러 에이전트가 공통의 작업 공간에 접근하는 방식입니다. 5.2.1에서 다룬 LangGraph에서는 **State** 객체가 이 역할을 합니다. State에는 작업 진행 상태, 중간 결과, 에이전트 간 공유 데이터가 담깁니다. 각 에이전트(노드)는 State를 읽어 현재 상황을 파악하고, 자신의 작업 결과를 State에 기록합니다. 그래프의 다음 노드가 업데이트된 State를 읽어 이어서 작업합니다. 이 구조 덕분에 에이전트 간의 의존성이 State를 통해 명시적으로 관리됩니다.

**멀티 에이전트 디버깅**은 단일 에이전트보다 훨씬 어렵습니다. 에이전트가 여러 개이므로 어떤 에이전트의 어떤 판단에서 문제가 생겼는지 추적해야 합니다. 디버깅을 효과적으로 하려면 세 가지를 갖추어야 합니다.

첫째, 각 에이전트의 입출력을 기록하는 **로깅**입니다. 에이전트가 받은 입력, 추론 과정(Think), 선택한 행동(Act), 도구 호출 결과를 모두 기록해 두면, 문제 발생 시 전체 흐름을 역추적할 수 있습니다. LangGraph에서는 각 노드의 실행 로그를 자동으로 수집하는 기능을 제공합니다.

둘째, 에이전트 간 메시지 흐름을 시각화하는 **트레이싱**입니다. 8단원에서 다룰 LangSmith나 Phoenix 같은 관찰 도구를 사용하면, 에이전트 간의 호출 관계를 타임라인으로 확인할 수 있습니다. 어떤 에이전트가 어떤 순서로 실행되었고, 각 단계에서 얼마나 시간이 걸렸는지를 한눈에 볼 수 있습니다.

셋째, 반복 실행을 위한 **재현 가능성**입니다. 동일한 입력에 대해 동일한 결과를 얻을 수 있어야 디버깅이 가능합니다. LLM의 출력에는 무작위성이 있으므로, 1.2.3에서 다룬 temperature 파라미터를 0으로 설정하거나 시드를 고정하여 결정적 출력을 만드는 것이 디버깅 시 유용합니다. LangGraph의 체크포인트 기능을 사용하면 특정 시점의 상태를 저장하고, 그 시점부터 다시 실행하여 문제를 재현할 수 있습니다.

정리하면, 멀티 에이전트 시스템은 복잡한 작업을 여러 전문 에이전트에게 분산하여, 개별 에이전트의 복잡도를 낮추고 전체 시스템의 능력을 높이는 접근입니다. 감독자 패턴은 중앙 집중식 제어로 명확한 흐름을 제공하고, 위계적 패턴은 대규모 작업에 적합한 다층 구조를 제공하며, 토론 패턴은 상호 검증으로 정확도를 높이고, Swarm 패턴은 자율적 핸드오프로 유연한 워크플로를 구성합니다. 에이전트 간 통신과 상태 공유를 위해 메시지 전달이나 공유 상태 객체를 사용하며, 효과적인 디버깅을 위해 로깅, 트레이싱, 재현 가능성을 확보해야 합니다.

다음 단원인 6.1.1에서는 모델 서빙 아키텍처의 이해를 다룹니다. 학습된 모델을 실제 서비스 환경에서 배포하고 운영하는 서빙 시스템의 구성요소와 성능 지표를 살펴봅니다.

이 단원을 마치면 멀티 에이전트 시스템의 네 가지 패턴(감독자, 위계적, 토론, Swarm)을 설명하고, 작업 특성에 맞는 협업 워크플로를 설계할 수 있습니다.
