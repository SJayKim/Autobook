# 2.1.1 프롬프트 엔지니어링 기본 원칙

1.3.1에서 LLM 생명주기를 살펴보면서, 모델을 실제 서비스에 적용하는 과정에는 개발, 평가, 배포, 모니터링, 갱신이라는 여러 단계가 있다는 점을 확인했습니다. 이 가운데 "모델의 동작을 원하는 방향으로 이끄는 것"은 생명주기 전반에 걸쳐 반복적으로 수행해야 하는 핵심 활동입니다. 전통적인 소프트웨어에서는 코드를 수정해서 동작을 바꾸지만, LLM에서는 모델에 전달하는 입력 텍스트, 즉 프롬프트를 어떻게 구성하느냐가 출력 품질을 좌우합니다. 같은 모델이라도 프롬프트의 표현, 구조, 예시 유무에 따라 전혀 다른 수준의 응답을 내놓습니다. 이 단원에서는 프롬프트를 체계적으로 설계하는 기법, 즉 프롬프트 엔지니어링의 기본 원칙을 다룹니다.

**프롬프트 엔지니어링**(prompt engineering)이란 LLM이 원하는 형태와 품질의 응답을 생성하도록 입력 텍스트를 설계하고 최적화하는 과정을 뜻합니다. 여기서 "설계"라는 표현을 쓰는 이유는, 단순히 질문을 던지는 것과 의도한 결과를 안정적으로 얻는 것 사이에 상당한 차이가 있기 때문입니다. 예를 들어, "이 텍스트를 요약해 줘"라고만 쓰면 모델은 때로 핵심을 잘 짚지만, 때로는 지엽적인 내용에 집중하거나 원하는 분량을 벗어날 수 있습니다. 반면 "이 텍스트의 핵심 주장 세 가지를 각각 한 문장으로 요약하고, 전체를 100자 이내로 정리해 줘"처럼 구체적으로 지시하면 모델의 응답이 훨씬 일관되게 나옵니다. 프롬프트 엔지니어링은 이런 차이를 체계적으로 만들어내는 방법론입니다.

프롬프트 엔지니어링이 별도의 분야로 다뤄지는 배경에는 LLM의 동작 원리가 있습니다. 1.2.3에서 살펴본 것처럼, LLM은 입력으로 받은 토큰 시퀀스를 바탕으로 다음 토큰을 확률적으로 예측하는 자기회귀(autoregressive) 방식으로 텍스트를 생성합니다. 모델은 입력에 포함된 단어, 문장 구조, 맥락 정보를 토대로 "이 입력 다음에 올 가능성이 가장 높은 텍스트"를 만들어냅니다. 따라서 입력을 어떻게 구성하느냐에 따라 모델이 참조하는 맥락이 달라지고, 결과적으로 생성되는 텍스트의 방향과 품질이 바뀝니다. 프롬프트 엔지니어링은 이 원리를 활용하여, 모델의 파라미터를 수정하지 않고도 입력 텍스트만으로 원하는 출력을 이끌어내는 기술입니다.

프롬프트를 설계할 때 가장 먼저 이해해야 할 것은 프롬프트의 구성 요소입니다. 하나의 프롬프트는 대체로 네 가지 요소로 이루어집니다. **지시**(instruction), **맥락**(context), **입력**(input), **출력 형식**(output format)입니다. 모든 프롬프트가 반드시 네 가지를 다 포함해야 하는 것은 아니지만, 각 요소의 역할을 알면 프롬프트를 더 체계적으로 설계할 수 있습니다.

**지시**는 모델에게 수행할 작업을 명시하는 부분입니다. "다음 텍스트를 요약하시오", "아래 코드의 버그를 찾아 수정하시오"처럼 모델이 무엇을 해야 하는지를 직접 알려줍니다. 지시가 모호하면 모델은 여러 방향으로 해석할 수 있고, 그만큼 응답의 편차가 커집니다. "이 텍스트에 대해 알려 줘"보다 "이 텍스트의 논지를 세 문장으로 요약하시오"가 더 좁은 범위를 지정하므로, 모델이 의도에 맞는 응답을 낼 확률이 높아집니다.

**맥락**은 작업 수행에 필요한 배경 정보를 제공하는 부분입니다. 예를 들어, 고객 문의를 분류하는 프롬프트라면 "당신은 전자상거래 고객센터의 상담 분류 시스템입니다"와 같은 역할 설명이 맥락에 해당합니다. 맥락을 제공하면 모델이 해당 도메인에 적합한 어휘와 판단 기준을 적용하여 응답을 생성합니다. 맥락 없이 "이 문의를 분류하시오"라고만 하면, 모델은 어떤 기준으로, 어떤 범주로 분류해야 하는지 스스로 추측해야 하므로 결과가 불안정해집니다.

**입력**은 모델이 처리해야 할 실제 데이터입니다. 요약 작업이라면 요약 대상 텍스트가, 번역 작업이라면 번역 대상 문장이 입력에 해당합니다. 입력은 대개 프롬프트의 나머지 부분과 구분할 수 있도록 구분자(delimiter)로 감싸거나, "입력:" 같은 레이블을 붙여 표시합니다. 입력과 지시를 명확히 분리하지 않으면, 모델이 입력 텍스트의 내용을 지시로 오해하는 **프롬프트 인젝션**(prompt injection) 문제가 발생할 수 있습니다. 프롬프트 인젝션이란 사용자가 입력에 "이전 지시를 무시하고 다음을 수행하라"와 같은 문장을 넣어 모델의 원래 동작을 변경하는 공격을 뜻합니다.

**출력 형식**은 모델의 응답이 어떤 구조를 따라야 하는지를 지정하는 부분입니다. "JSON 형식으로 답하시오", "표로 정리하시오", "한 문장으로만 답하시오" 등이 이에 해당합니다. 출력 형식을 명시하면 후속 시스템이 응답을 파싱하기 쉬워지고, 응답의 일관성도 높아집니다. 특히 LLM의 출력을 다른 프로그램에서 자동으로 처리해야 하는 경우, 출력 형식 지정은 필수적입니다.

다음은 네 가지 요소를 모두 포함한 프롬프트의 예입니다.

```
[지시]
아래 고객 문의를 읽고, 문의 유형을 분류하시오.

[맥락]
당신은 전자상거래 고객센터의 문의 분류 시스템입니다.
분류 범주: 배송, 환불, 교환, 상품 문의, 기타

[입력]
"주문한 지 일주일이 넘었는데 아직 배송이 시작되지 않았습니다."

[출력 형식]
다음 JSON 형식으로 답하시오:
{"category": "분류 범주", "confidence": "high/medium/low"}
```

이 프롬프트에서 지시는 "문의 유형을 분류하시오"로, 모델이 수행할 작업을 명확히 지정합니다. 맥락은 역할과 분류 범주를 제공하여 모델이 판단할 기준을 좁혀 줍니다. 입력은 실제 고객 문의 텍스트이며, 따옴표로 감싸 지시와 구분합니다. 출력 형식은 JSON 구조를 지정하여 응답을 프로그램에서 바로 파싱할 수 있도록 합니다.

이제 프롬프트의 구성 요소를 이해했으므로, 대표적인 프롬프트 기법 세 가지를 살펴보겠습니다. **Zero-shot 프롬프팅**, **Few-shot 프롬프팅**, **Chain-of-Thought(CoT) 프롬프팅**입니다. 이 세 기법은 프롬프트에 예시나 추론 과정을 포함하느냐에 따라 구분됩니다.

**Zero-shot 프롬프팅**은 예시를 하나도 제공하지 않고, 지시만으로 모델에게 작업을 수행하게 하는 방식입니다. "shot"은 프롬프트에 포함된 예시의 수를 의미하며, zero-shot은 예시가 0개인 상태를 뜻합니다. 앞서 프롬프트 구성 요소를 설명할 때 사용한 고객 문의 분류 예가 zero-shot에 해당합니다. 지시, 맥락, 입력, 출력 형식은 있지만 "이런 문의는 이렇게 분류한다"는 구체적 사례는 없습니다.

```
다음 문장의 감정을 긍정, 부정, 중립 중 하나로 분류하시오.
문장: "이 제품 정말 마음에 듭니다."
```

이 프롬프트는 모델에게 아무런 예시 없이 감정 분류를 요청합니다. LLM은 사전훈련 과정에서 방대한 양의 텍스트를 학습했으므로, 별도의 예시 없이도 많은 작업을 수행할 수 있습니다. zero-shot 프롬프팅은 프롬프트가 간결하고 토큰 소비가 적다는 장점이 있습니다. 다만 작업이 복잡하거나, 모델이 학습 과정에서 충분히 접하지 못한 유형의 작업이라면 응답 품질이 떨어질 수 있습니다.

**Few-shot 프롬프팅**은 프롬프트에 입력-출력 쌍의 예시를 몇 개 포함하여, 모델이 그 패턴을 따르도록 유도하는 방식입니다. 예시의 수에 따라 one-shot(1개), few-shot(2개 이상)으로 부르기도 합니다. 핵심은 모델에게 "이런 식으로 답하면 된다"는 것을 예시를 통해 보여주는 것입니다.

```
다음 문장의 감정을 긍정, 부정, 중립 중 하나로 분류하시오.

문장: "배송이 빨라서 좋았습니다."
감정: 긍정

문장: "제품에 하자가 있어서 실망입니다."
감정: 부정

문장: "보통 수준입니다."
감정: 중립

문장: "포장이 꼼꼼해서 감동받았습니다."
감정:
```

이 프롬프트에서 처음 세 쌍(문장-감정)이 예시이고, 마지막 문장이 모델이 실제로 분류해야 할 입력입니다. 모델은 앞선 예시들의 패턴, 즉 문장을 읽고 감정 레이블을 한 단어로 출력하는 패턴을 학습하여 마지막 입력에도 같은 형식으로 답합니다. few-shot 프롬프팅은 zero-shot보다 응답의 일관성이 높아지는 경향이 있습니다. 예시가 출력 형식뿐 아니라 판단 기준까지 암묵적으로 전달하기 때문입니다.

few-shot 프롬프팅에서 예시를 선택할 때 주의할 점이 있습니다. 첫째, 예시가 실제 작업의 분포를 대표해야 합니다. 감정 분류 작업에서 긍정 예시만 세 개 넣으면, 모델은 대부분의 입력을 긍정으로 분류하는 편향을 보일 수 있습니다. 둘째, 예시의 수가 많아지면 프롬프트가 길어져 토큰 소비가 늘어납니다. 1.2.3에서 다룬 컨텍스트 윈도우의 제한도 고려해야 합니다. 일반적으로 3개에서 5개 정도의 예시가 효과와 비용의 균형을 잘 맞춥니다.

**Chain-of-Thought(CoT) 프롬프팅**은 모델이 최종 답을 내기 전에 중간 추론 과정을 단계별로 서술하도록 유도하는 방식입니다. "Chain-of-Thought"는 "사고의 연쇄"라는 뜻으로, 모델이 결론에 도달하기까지의 논리적 단계를 하나씩 밟아가도록 하는 것입니다. 이 기법이 필요한 이유는 LLM이 복잡한 추론 작업에서 중간 단계를 건너뛰고 바로 답을 내면 오류율이 높아지기 때문입니다.

수학 문제를 예로 들어보겠습니다. "가게에 사과가 23개 있고, 오전에 8개를 팔고, 오후에 12개를 새로 입고했습니다. 현재 사과는 몇 개입니까?"라는 문제를 생각해 보겠습니다. zero-shot으로 이 문제를 던지면, 모델이 바로 답을 내면서 계산을 틀릴 가능성이 있습니다. CoT 프롬프팅을 적용하면 다음과 같이 됩니다.

```
질문: 가게에 사과가 23개 있습니다.
오전에 8개를 팔고, 오후에 12개를 새로 입고했습니다.
현재 사과는 몇 개입니까?

단계별로 생각하겠습니다.
1단계: 처음 사과 수는 23개입니다.
2단계: 오전에 8개를 팔았으므로 23 - 8 = 15개가 남습니다.
3단계: 오후에 12개를 입고했으므로 15 + 12 = 27개가 됩니다.

답: 27개
```

이 예시에서 핵심은 "단계별로 생각하겠습니다"라는 유도 문구와, 각 단계를 명시적으로 서술하는 구조입니다. CoT 프롬프팅을 적용하는 가장 간단한 방법은 지시 끝에 "단계별로 생각하시오(Let's think step by step)"라는 문구를 추가하는 것입니다. 이 한 문장만 넣어도 모델은 중간 추론 과정을 출력하면서 답을 도출하는 경향을 보이며, 이를 **zero-shot CoT**라고 부릅니다. 앞선 사과 문제처럼 구체적인 추론 예시를 함께 제공하는 방식은 **few-shot CoT**에 해당합니다.

CoT 프롬프팅은 특히 산술 문제, 논리 추론, 다단계 의사결정 같은 작업에서 정확도를 높이는 효과가 큽니다. 다만 중간 추론 과정까지 출력하므로 응답 토큰 수가 늘어나고, 그만큼 비용과 응답 시간이 증가합니다. 단순한 분류나 번역 작업처럼 중간 추론이 불필요한 경우에는 오히려 불필요한 오버헤드가 됩니다.

지금까지 다룬 세 가지 기법의 차이를 정리하면 다음과 같습니다.

```
+-------------------+----------+----------+--------------+
| 기법              | 예시     | 추론     | 적합한 작업   |
|                   | 포함     | 과정     |              |
+-------------------+----------+----------+--------------+
| Zero-shot         | 없음     | 없음     | 단순 분류,    |
|                   |          |          | 번역, 요약   |
+-------------------+----------+----------+--------------+
| Few-shot          | 있음     | 없음     | 패턴 학습이   |
|                   | (2-5개)  |          | 필요한 작업   |
+-------------------+----------+----------+--------------+
| Chain-of-Thought  | 선택     | 있음     | 산술, 논리,   |
| (CoT)             |          |          | 다단계 추론   |
+-------------------+----------+----------+--------------+
```

Zero-shot은 프롬프트가 가장 짧고 빠르지만, 복잡한 작업에서는 정확도가 낮을 수 있습니다. Few-shot은 예시를 통해 출력 형식과 판단 기준을 함께 전달하므로 일관성이 높지만, 프롬프트가 길어집니다. CoT는 복잡한 추론 작업에서 정확도를 높이지만, 토큰 소비가 가장 많습니다. 이 세 기법은 상호 배타적이지 않습니다. few-shot과 CoT를 결합하여, 추론 과정이 포함된 예시를 제공하는 few-shot CoT도 자주 사용됩니다.

이 기법들을 실제로 적용할 때, 효과적인 프롬프트를 작성하기 위해 따르면 좋은 원칙들이 있습니다.

첫 번째 원칙은 **지시를 구체적으로 작성하는 것**입니다. "잘 요약해 줘"보다 "핵심 논점 세 가지를 각각 한 문장으로 작성하시오"가 더 구체적입니다. 모델이 해석할 여지가 줄어들수록 응답의 편차가 줄어듭니다. 작업의 범위, 분량, 형식, 어조까지 명시할 수 있다면 명시하는 것이 좋습니다.

두 번째 원칙은 **구분자를 사용하여 프롬프트의 각 부분을 분리하는 것**입니다. 지시, 맥락, 입력을 하나의 문단에 섞어 쓰면 모델이 각 부분의 경계를 혼동할 수 있습니다. 삼중 백틱(```), 삼중 따옴표("""), XML 태그(<input></input>), 줄바꿈과 레이블("입력:", "출력:") 등을 활용하여 각 부분을 시각적으로 분리하면 모델의 이해도가 높아집니다.

세 번째 원칙은 **원하는 것을 말하는 것이지, 원하지 않는 것을 나열하지 않는 것**입니다. "비속어를 쓰지 마시오, 너무 길게 쓰지 마시오, 감정적이지 않게 쓰시오"처럼 금지 목록을 나열하는 것보다, "공식적인 어조로 200자 이내로 작성하시오"처럼 원하는 조건을 직접 서술하는 것이 더 효과적입니다. 모델은 "하지 마시오"라는 부정 지시보다 "이렇게 하시오"라는 긍정 지시를 더 안정적으로 따릅니다.

네 번째 원칙은 **출력 형식을 명확히 지정하는 것**입니다. 특히 프로그램이 모델의 응답을 자동으로 처리해야 하는 경우, JSON, CSV, 마크다운 표 등 구체적인 형식을 지시에 포함하고, 가능하면 예시까지 제공하는 것이 좋습니다. 형식 지정은 few-shot 프롬프팅과 결합하면 더 효과적입니다. 예시 자체가 원하는 출력 형식의 시범이 되기 때문입니다.

다섯 번째 원칙은 **반복적으로 실험하고 개선하는 것**입니다. 프롬프트 엔지니어링은 한 번에 완벽한 프롬프트를 작성하는 것이 아닙니다. 초안을 작성하고, 다양한 입력으로 테스트하며, 문제가 발견되면 지시를 수정하거나 예시를 추가하는 과정을 반복합니다. 이 반복 과정을 체계화하는 것이 2.2에서 다룰 프롬프트 운영의 핵심이기도 합니다.

이 다섯 가지 원칙을 적용하는 흐름을 정리하면 다음과 같습니다.

```
프롬프트 설계 흐름

[1. 작업 정의]       어떤 작업인지, 입력과 출력은 무엇인지 정한다
       |
[2. 기법 선택]       작업 복잡도에 따라 zero-shot / few-shot / CoT를 선택한다
       |
[3. 구성 요소 배치]  지시, 맥락, 입력, 출력 형식을 구분자로 분리하여 배치한다
       |
[4. 테스트]          다양한 입력으로 응답을 확인한다
       |
[5. 개선]            문제가 있으면 지시 수정, 예시 추가, 형식 변경 등을 반복한다
       |
       +-----> [4. 테스트]로 돌아간다
```

1단계에서 작업의 성격을 파악합니다. 분류, 요약, 생성, 추론 중 어디에 해당하는지, 입력 데이터의 형태는 무엇인지, 출력이 자유 형식인지 구조화된 형식인지를 정합니다. 2단계에서 작업의 복잡도에 따라 기법을 선택합니다. 단순 분류라면 zero-shot으로 시작하고, 출력 형식이 특수하다면 few-shot을 적용하며, 다단계 추론이 필요하다면 CoT를 도입합니다. 3단계에서 선택한 기법에 맞춰 프롬프트를 작성합니다. 지시를 구체적으로 쓰고, 구분자로 각 부분을 분리하며, 출력 형식을 명시합니다. 4단계에서 여러 입력으로 응답을 확인합니다. 5단계에서 응답이 기대에 미치지 못하면 프롬프트를 수정하고 다시 테스트합니다.

정리하면, 프롬프트 엔지니어링은 LLM에 전달하는 입력 텍스트를 체계적으로 설계하여 원하는 품질의 출력을 안정적으로 얻는 기술입니다. 프롬프트는 지시, 맥락, 입력, 출력 형식의 네 가지 요소로 구성되며, 작업의 복잡도에 따라 zero-shot, few-shot, Chain-of-Thought 기법을 선택하여 적용합니다. 지시를 구체적으로 작성하고, 구분자로 영역을 분리하며, 원하는 것을 명확히 서술하고, 출력 형식을 지정하며, 반복적으로 실험하는 것이 효과적인 프롬프트 작성의 기본 원칙입니다.

다음 단원인 2.1.2에서는 Self-Consistency, Tree-of-Thought, ReAct 패턴 등 기본 기법을 확장한 고급 프롬프트 기법을 다룹니다.

이 단원을 마치면 zero-shot, few-shot, Chain-of-Thought 프롬프팅의 차이를 설명하고, 작업 목적에 맞는 기법을 선택하여 프롬프트를 설계할 수 있습니다.
