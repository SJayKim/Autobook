# 1.2.3 SOP 기반 협업과 구조화된 산출물

1.2.2에서 에이전트 협업 전략과 조율 프로토콜의 설계 원리를 살펴보았습니다. 이 단원에서는 그 원리가 실제로 적용된 대표적 사례인 SOP 기반 협업 구조를 깊이 있게 다루고, 다중 에이전트 시스템에서 가장 심각한 품질 문제 중 하나인 연쇄 환각이 어떻게 억제되는지 살펴봅니다.

인간 조직에서 복잡한 업무를 수행할 때는 **표준 운영 절차(SOP, Standard Operating Procedure)**를 만듭니다. 병원에서 수술 전 체크리스트를 사용하고, 항공사에서 비행 전 점검 절차를 따르듯, SOP는 반복적이고 복잡한 업무에서 실수를 줄이고 일관성을 보장하기 위해 고안된 것입니다. SOP의 핵심은 "누가, 어떤 순서로, 무엇을 생성하고, 어떤 형식으로 전달하는지"를 사전에 규정하는 것입니다. 에이전트가 자유롭게 대화하며 역할을 수행하는 접근과 달리, SOP 기반 협업에서는 역할 간 자유 대화가 허용되지 않습니다. 각 에이전트는 SOP에 명시된 입력 형식만 받고, SOP에 명시된 출력 형식만 생성합니다. 모든 인계(handover)는 사전에 정의된 구조화된 문서 형식을 반드시 준수해야 합니다.

이 원칙을 에이전트 시스템에 적용한 것이 **어셈블리 라인 패러다임**입니다. 제조업의 조립 라인에서 각 공정이 순서대로 반제품을 처리하듯, 소프트웨어 개발 과정을 순차적 공정으로 분해합니다. 구체적으로 다섯 단계로 구성됩니다. 첫 번째 공정에서 Product Manager 에이전트가 사용자의 자연어 요구사항을 입력으로 받아 PRD(Product Requirements Document) 문서를 생성합니다. 두 번째 공정에서 Architect 에이전트가 PRD를 입력으로 받아 시스템 설계 문서, 파일 목록, 인터페이스 정의를 생성합니다. 세 번째 공정에서 Project Manager 에이전트가 설계 문서를 기반으로 태스크 분배 계획을 세우고 각 엔지니어에게 작업 단위를 할당합니다. 네 번째 공정에서 Engineer 에이전트가 태스크 단위와 인터페이스 스펙을 입력으로 받아 실제 코드 파일을 작성합니다. 다섯 번째 공정에서 QA Engineer 에이전트가 코드 파일을 실행하여 유닛 테스트를 수행하고 버그를 수정합니다.

이 어셈블리 라인의 핵심 특성은 세 가지입니다. 첫째, 원칙적으로 공정은 앞으로만 진행되는 단방향 흐름입니다. 다음 역할은 이전 역할의 산출물을 검증하고 소비합니다. 둘째, 어떤 산출물이 다음 공정의 전제 조건인지 SOP가 규정하므로, 잘못된 병렬 실행으로 인한 불일치가 방지됩니다. 셋째, Engineer와 QA 단계에서는 코드를 실제로 실행하고 런타임 오류를 역방향으로 피드백하는 메커니즘이 추가됩니다. 이것은 순수한 일방향 파이프라인에 수정 루프를 결합한 것입니다.

어셈블리 라인의 각 공정에서 생성되는 **구조화된 산출물**이 이 시스템의 품질을 결정합니다. 구조화된 산출물은 자유 형식 텍스트가 아니라 필드가 정의된 문서입니다. PRD에는 제품 목표(핵심 가치 3가지), 사용자 스토리(역할-행동-목적 형식), 경쟁 분석(유사 제품 비교표), 요구사항 분석, 그리고 우선순위가 지정된 요구사항 목록(P0, P1, P2 등급)이 포함됩니다. 시스템 설계 문서에는 생성될 모든 소스 파일의 목록, 핵심 클래스의 데이터 구조와 속성, 함수 시그니처와 입출력 타입이 정의된 인터페이스 정의, 그리고 컴포넌트 간 관계를 보여주는 아키텍처 다이어그램이 포함됩니다.

인터페이스 스펙의 역할은 특히 중요합니다. Engineer 에이전트는 인터페이스 스펙을 받아 함수 내부만 구현하면 됩니다. 함수명, 매개변수 타입, 반환값이 이미 Architect에 의해 확정되어 있으므로, 서로 다른 Engineer 에이전트가 병렬로 다른 파일을 구현하더라도 통합 시 인터페이스 불일치가 발생하지 않습니다. 이것이 구조화된 산출물의 핵심 가치입니다. "무엇을 만들지"뿐 아니라 "어떤 형태로 만들지"까지 사전 합의된 계약서 역할을 합니다.

이제 SOP와 구조화된 산출물이 왜 필요한지를 보여주는 근본 문제인 **연쇄 환각(cascading hallucination)**을 살펴보겠습니다. LLM은 확률적으로 사실이 아닌 내용을 생성할 수 있습니다. 단일 에이전트에서는 이것이 일회성 오류로 끝나지만, 다중 에이전트 파이프라인에서는 한 에이전트의 환각이 다음 에이전트의 입력이 됩니다. 에이전트 A가 잘못된 가정을 생성하면, 에이전트 B는 그 잘못된 가정 위에 추가 오류를 쌓고, 에이전트 C는 더욱 왜곡된 출력을 만들어 냅니다. 자유로운 대화 기반 협업에서는 에이전트들이 서로의 부정확한 발언을 그대로 수용하고 확장하기 때문에 오류가 기하급수적으로 증폭됩니다.

SOP 기반 시스템은 세 가지 메커니즘으로 이 문제를 억제합니다. 첫째, 구조화된 산출물에 의한 검증 강제입니다. 다음 에이전트가 구조화된 문서를 받으면, 해당 문서의 필드가 올바르게 채워졌는지 형식적으로 검증할 수 있습니다. 비구조적 자연어 발언과 달리 필드가 비어 있거나 타입이 맞지 않으면 즉시 탐지 가능합니다. 둘째, 역할 전문화에 의한 도메인 제한입니다. 각 에이전트는 자신의 역할 범위 밖의 내용을 생성하지 않도록 프롬프트가 설계됩니다. Product Manager는 코드 구현에 대해 발언하지 않고, Engineer는 제품 전략에 대해 발언하지 않습니다. 역할 경계가 환각의 확산 범위를 제한합니다. 셋째, 실행 기반 피드백입니다. 코드가 실제로 실행되어 오류가 발생하면, 런타임 오류 메시지가 구체적인 수정 신호로 피드백됩니다. 이것은 "환각으로 생성된 코드"와 "실제로 동작하는 코드"를 명확히 구분하는 가장 강력한 억제 수단입니다.

이 메커니즘들이 구체적으로 동작하는 기반이 **MetaGPT의 역할 분담 구조**입니다. MetaGPT는 역할 간 직접 메시지 전달 대신 환경(Environment)이라는 공유 공간을 중개자로 사용하는 게시-구독(publish-subscribe) 메커니즘을 채택합니다. 각 역할은 자신이 구독할 메시지 타입을 등록합니다. 예를 들어 Architect는 PRD 문서 타입을 구독합니다. Product Manager가 PRD를 생성하면 PRD가 환경에 게시되고, PRD를 구독한 Architect가 자동으로 이를 수신하여 다음 작업을 시작합니다. 이 방식으로 역할들은 서로를 직접 호출하지 않고도 순서 있는 협업이 이루어집니다. 1.2.2에서 다룬 블랙보드 패턴의 구체적 구현으로 볼 수 있습니다.

각 에이전트의 내부에서는 관찰-사고-행동(observe-think-act) 사이클이 반복됩니다. 관찰 단계에서 환경에서 자신이 구독한 메시지 유형을 감지하고, 사고 단계에서 수신한 정보를 바탕으로 자신의 역할에 맞는 처리 계획을 수립하며, 행동 단계에서 구조화된 산출물을 생성하여 환경에 게시합니다. 이 사이클은 1.1.2에서 다룬 ReAct 패턴의 다중 에이전트 확장입니다.

SOP 기반 접근과 대비되는 것이 **중첩 대화(nested conversation) 패턴**입니다. 이 패턴에서는 SOP처럼 순서가 고정되지 않고, 에이전트 간 대화를 조합 가능한 기본 패턴으로 추상화합니다. 네 가지 기본 패턴이 있습니다. 1대1 대화는 두 에이전트가 메시지를 주고받으며 태스크를 해결하는 가장 단순한 형태입니다. 순차 대화는 여러 1대1 대화가 순서대로 연결되며, 이전 대화의 요약이 다음 대화의 초기 컨텍스트로 자동 전달됩니다. 그룹 채팅은 여러 에이전트가 단일 대화 스레드를 공유하며, 중앙의 관리자가 다음 발언자를 결정합니다. 발언자 선택 전략으로 순환, 무작위, LLM 자동 판단, 인간 선택을 지원하며, 유한 상태 기계 방식으로 특정 전환을 허용하거나 금지하는 그래프를 정의할 수도 있습니다.

가장 강력한 패턴이 중첩 대화입니다. 하나의 에이전트가 외부에서 메시지를 받았을 때, 응답을 돌려주기 전에 내부적으로 별도의 에이전트 대화 시퀀스를 실행합니다. 외부 에이전트 A가 에이전트 B에게 메시지를 보내면, 에이전트 B는 즉시 응답하지 않고 내부적으로 에이전트 C, D와 순차적으로 여러 차례 대화를 진행합니다. 이 내부 대화는 외부에서 보이지 않습니다. 내부 대화가 모두 완료되면 마지막 내부 대화의 요약이 에이전트 B의 최종 응답으로 외부 에이전트 A에게 전달됩니다. 이것은 "하나의 에이전트처럼 보이지만 내부에서는 전문가 팀이 협업한다"는 캡슐화의 원칙입니다. 복잡한 워크플로우를 재사용 가능한 단일 에이전트 인터페이스로 감싸는 것이 핵심입니다. 그룹 채팅의 각 참가자가 다시 중첩 그룹 채팅이 될 수 있어, 이 재귀적 구조를 통해 단순한 패턴 조합만으로 임의의 복잡도를 가진 협업 구조를 표현할 수 있습니다.

SOP 기반 접근과 대화 패턴 조합 접근은 서로 다른 강점을 가집니다. SOP 기반은 절차가 정해져 있어 일관성이 높고 연쇄 환각 억제에 강하지만, 사전에 정의되지 않은 태스크에는 유연하게 대응하기 어렵습니다. 대화 패턴 조합은 범용 태스크에 유연하게 대응할 수 있고 인간을 대화 흐름에 자연스럽게 통합할 수 있지만, 환각 억제를 위한 명시적 메커니즘이 없어 설계자가 별도로 품질 관리를 설계해야 합니다. 실제 시스템에서는 두 접근을 결합하여, SOP로 전체 구조를 잡되 각 공정 내부에서는 유연한 대화 패턴을 활용하는 하이브리드 방식이 흔합니다.

정리하면, SOP 기반 협업은 인간 조직의 표준 운영 절차를 에이전트 시스템에 이식한 것으로, 어셈블리 라인 패러다임을 통해 역할 간 순차적 분업을 구현합니다. PRD, 설계 문서, 인터페이스 스펙 등 구조화된 산출물은 역할 간 계약서 역할을 하며, 구조화된 검증, 역할 전문화, 실행 기반 피드백의 세 가지 메커니즘이 연쇄 환각을 억제합니다. 게시-구독 메커니즘을 통해 역할들이 서로를 직접 호출하지 않고도 순서 있는 협업이 가능하며, 중첩 대화 패턴은 이와 보완적으로 유연한 대화 구조를 제공합니다.

다음 단원인 1.3.1에서는 이러한 에이전트 시스템의 설계를 사람이 수동으로 하는 것이 아니라 자동화하는 접근을 다루며, ADAS 연구 분야의 정의와 에이전트 설계 탐색 공간의 구조를 살펴봅니다.

이 단원을 마치면 표준 운영 절차(SOP)를 다중 에이전트 협업에 적용하는 방법을 설명하고, 구조화된 산출물이 연쇄 환각을 어떻게 억제하는지 그 메커니즘을 기술할 수 있습니다.
