# 1.3.1 에이전트 설계 자동화와 탐색 공간

1.1.1에서 에이전트의 네 가지 핵심 모듈을, 1.1.2에서 ReAct 패턴과 에이전트 루프를 살펴보았습니다. 1.2절에서는 여러 에이전트가 협력하는 다중 에이전트 시스템의 구조와 전략을 다루었습니다. 이 과정에서 에이전트 시스템을 설계하는 일이 얼마나 많은 선택지를 수반하는지 느꼈을 것입니다. Planning 모듈에 어떤 기법을 쓸지, Reasoning 모듈에 CoT를 쓸지 Tree of Thoughts를 쓸지, 에이전트들을 중앙집중형으로 연결할지 P2P로 연결할지, 디베이트를 넣을지 말지, 이 모든 것이 설계자의 판단에 달려 있습니다. 그런데 이 선택지의 조합은 방대합니다. 한 연구에서는 16개의 기존 에이전트에서 모듈을 추출하여 표준화한 결과, 가능한 조합이 1,050가지에 달했습니다. 인간이 수동으로 이 조합 공간을 탐색하는 것은 비효율적이며, 더 큰 문제는 아직 발견되지 않은 설계 패턴이 무수히 많다는 점입니다.

이 문제를 해결하기 위해 등장한 연구 분야가 **ADAS(Automated Design of Agentic Systems)**입니다. ADAS는 에이전트 시스템을 인간이 수동으로 설계하지 않고 자동으로 발견하고 생성하는 것을 목표로 합니다. 이 분야의 핵심 전제는 머신러닝의 역사적 패턴에서 옵니다. 딥러닝 초기에 연구자들은 CNN의 레이어 수, 필터 크기, 활성화 함수, 연결 패턴을 수작업으로 설계했습니다. AlexNet, VGGNet, ResNet 같은 아키텍처가 모두 인간의 직관과 반복 실험으로 탄생했습니다. 그러나 **NAS(Neural Architecture Search)**가 등장하면서, 이 설계 과정 자체가 자동화되었습니다. NAS는 가능한 아키텍처의 공간을 정의하고, 강화학습이나 진화 알고리즘으로 그 공간을 탐색하며, 검증 정확도로 후보를 평가합니다. 그 결과 인간이 고안한 아키텍처보다 우수한 구조가 자동으로 발견되었습니다.

ADAS는 이와 동일한 구조를 에이전트 설계에 적용합니다. NAS에서 탐색 대상이 신경망 아키텍처라면, ADAS에서는 에이전트 시스템 설계가 탐색 대상입니다. NAS의 탐색 공간이 레이어 수, 활성화 함수, 연결 패턴이라면, ADAS의 탐색 공간은 프롬프트, 도구, 워크플로우, 빌딩 블록 조합입니다. NAS의 탐색 알고리즘이 강화학습이나 진화 알고리즘이라면, ADAS에서는 LLM 메타 에이전트가 새로운 에이전트를 코드로 작성합니다. NAS의 평가 함수가 검증셋 정확도라면, ADAS에서는 태스크 정확도, 비용, 지연 시간이 평가 기준이 됩니다.

ADAS는 이 문제를 세 가지 구성 요소로 체계화합니다. **탐색 공간(Search Space)**, **탐색 알고리즘(Search Algorithm)**, **평가 함수(Evaluation Function)**입니다.

**탐색 공간**은 에이전트가 가질 수 있는 모든 설계 가능성의 집합입니다. 탐색 공간을 어떻게 정의하느냐에 따라 발견할 수 있는 설계의 범위가 결정됩니다. 가장 좁은 탐색 공간은 프롬프트 텍스트만 변경하는 것입니다. 에이전트의 instruction이나 few-shot 예시를 바꾸는 것으로, 기존 워크플로우 구조 자체는 건드리지 않습니다. 이 방식은 안전하지만 발견 가능한 설계의 범위가 제한적입니다. 중간 수준의 탐색 공간은 사전 정의된 모듈의 조합을 탐색하는 것입니다. Planning, Reasoning, Tool Use, Memory 네 모듈 각각에 여러 선택지를 두고, 이들의 조합을 탐색합니다. 1,050가지 조합이 이 방식의 구체적 사례입니다. 가장 넓은 탐색 공간은 에이전트를 코드로 정의하는 것입니다. 이것이 ADAS의 핵심적인 설계 결정이며, **코드 공간 탐색**이라고 부릅니다.

코드 공간 탐색이 중요한 이유는 두 가지입니다. 첫째, 표현력의 완전성입니다. 코드는 튜링 완전(Turing Complete)하므로 이론적으로 모든 가능한 에이전트 설계가 이 표현 방식 안에 포함됩니다. 프롬프트 최적화만으로는 워크플로우 구조 자체를 바꾸지 못하지만, 코드로 표현하면 전혀 새로운 구조를 만들어 낼 수 있습니다. 둘째, 해석 가능성과 재현성입니다. 발견된 에이전트 설계가 실행 가능한 코드로 남으므로, 다른 사람이 그대로 실행하고 검증할 수 있습니다.

**탐색 알고리즘**은 방대한 탐색 공간에서 유망한 설계를 찾아가는 방법입니다. ADAS에서 제안된 Meta Agent Search 알고리즘은 다음과 같이 동작합니다. 메타 에이전트(GPT-4와 같은 강력한 LLM)가 아카이브를 입력으로 받습니다. 아카이브는 이전에 발견된 에이전트 설계들의 저장소로, 각 설계의 코드와 평가 결과가 함께 저장되어 있습니다. 메타 에이전트는 아카이브를 참조하여 아직 시도되지 않은 새로운 에이전트 설계를 Python 코드로 작성합니다. 코드 실행 중 오류가 발생하면 오류 메시지를 메타 에이전트에게 다시 전달하고, 메타 에이전트가 코드를 수정합니다. 이 자동 수정을 최대 5회 반복하여 실행 가능한 코드를 보장합니다. 해당 에이전트를 벤치마크 태스크에서 실행하여 성능을 측정하고, 성능과 설계 코드를 함께 아카이브에 추가합니다. 이 과정을 수십 회(논문 실험에서는 25회) 반복합니다. 초기 아카이브에는 CoT, Self-Refine, LLM Debate 같은 기존의 수동 설계를 시드로 포함합니다. 탐색이 진행될수록 아카이브가 풍부해지면서 메타 에이전트가 더 정교한 조합을 제안할 수 있게 됩니다.

다른 탐색 알고리즘도 있습니다. 모듈 수준에서 탐색하는 접근에서는 진화(evolution)와 재결합(recombination) 두 메커니즘을 병행합니다. 진화는 LLM이 기존 모듈의 코드를 수정하여 변형된 새 모듈을 생성하는 것이고, 재결합은 경험 풀에 축적된 이전 에이전트의 성능 기록을 분석하여 유망한 모듈 조합을 추론하는 것입니다. 두 메커니즘은 상호 보완적입니다. 진화만 사용하면 새 모듈을 만들지만 조합 공간을 충분히 탐색하지 못하고, 재결합만 사용하면 최적 조합을 찾지만 새 모듈을 발견하지 못합니다. 또 다른 접근에서는 로컬-투-글로벌 인터리빙 최적화를 사용합니다. 먼저 각 빌딩 블록을 독립적으로 최적화한 뒤, 블록별 기여도를 측정하여 토폴로지 탐색의 사전 확률로 활용하고, 마지막으로 결정된 토폴로지 위에서 전체 시스템의 프롬프트를 글로벌하게 재최적화하는 3단계 과정입니다.

**평가 함수**는 후보 설계의 품질을 수치로 환산합니다. 태스크 유형에 따라 정확도, F1 점수, 코드 통과율 등이 1차 지표로 사용됩니다. 비용(쿼리당 소모 금액)과 지연 시간도 중요한 평가 기준입니다. 평가 비용을 줄이기 위해, 실제 실행 없이 LLM이 후보 에이전트의 성능을 예측하는 in-context surrogate 기법도 사용됩니다. 경험 풀에 축적된 에이전트 설명과 실제 성능 쌍을 LLM에게 컨텍스트로 제공하면, LLM이 새 후보의 성능을 추정합니다. 이 방법으로 실제 평가 비용을 50% 이상 절감하면서도 탐색 성능을 유지한 사례가 보고되었습니다.

탐색 공간에서 에이전트들의 연결 방식을 정의하는 것이 **워크플로우 토폴로지**입니다. 토폴로지는 여러 에이전트 또는 빌딩 블록이 어떤 구조로 연결되어 정보를 주고받는지를 나타냅니다. 토폴로지 탐색에서 중요한 발견이 두 가지 있습니다. 첫째, 유효한 토폴로지는 전체 조합 공간의 극히 일부에 불과합니다. 빌딩 블록의 모든 가능한 연결 조합 중 실제로 성능 향상에 기여하는 구조는 매우 적습니다. 따라서 전수 탐색이 아니라 기여도 기반 가지치기(pruning)가 필수적입니다. 각 블록의 추가가 검증 성능에 미치는 영향을 측정하고, 긍정적 영향이 낮은 블록을 탐색 공간에서 배제합니다. 둘째, 태스크 유형에 따라 선호되는 토폴로지가 다릅니다. 수학이나 논리 추론 태스크는 병렬 탐색과 앙상블을 선호하고, 멀티홉 질의응답은 디베이트 구조를 선호하며, 코딩 태스크는 자기 반성과 도구 사용의 조합이 가장 효과적입니다.

토폴로지를 구성하는 기본 단위가 **에이전트 빌딩 블록**입니다. 대표적인 빌딩 블록 다섯 가지를 살펴보겠습니다. **Aggregate(집계)**는 여러 에이전트가 동일한 문제에 대해 독립적으로 답변을 생성하고 그 결과를 하나로 합산하는 블록입니다. 다수결, 가중 평균, 또는 별도의 에이전트가 통합 답변을 생성하는 방식이 있습니다. 단일 예측의 분산을 줄이는 효과가 있어, 수학 벤치마크에서 CoT 단독의 약 72%를 약 85%까지 끌어올리는 데 핵심적으로 기여했습니다. **Reflect(자기 반성)**는 에이전트가 자신의 답변을 재검토하고 오류나 개선점을 찾아 수정하는 블록입니다. 1.1.2에서 다룬 Self-Refine의 빌딩 블록화입니다. 코딩 태스크에서 도구 사용 블록과 결합될 때 특히 강력합니다. 코드를 실행하고, 오류 메시지를 보고, 수정하는 루프가 Reflect와 Tool-use의 조합으로 자연스럽게 구현됩니다. **Debate(토론)**는 서로 다른 에이전트들이 각자의 답변을 교환하고 상대방의 답변을 비판하며 자신의 답변을 수정하는 과정을 반복하는 블록입니다. 1.2.2에서 다룬 경쟁(Competition) 유형의 빌딩 블록화입니다. 멀티홉 추론에서 가장 효과적입니다. **Summarize(요약)**는 긴 입력이나 중간 결과물을 압축하는 블록으로, 컨텍스트 창 한계를 극복하거나 후속 블록에게 정제된 입력을 제공하는 역할을 합니다. 토폴로지 순서상 가장 앞에 배치되어 후속 블록의 처리 효율을 높입니다. 이 외에 **Tool-use(도구 사용)** 블록이 있으며, 외부 도구 호출과 결과 활용을 담당합니다.

이 빌딩 블록들이 실제 탐색에서 어떻게 조합되는지 살펴보겠습니다. Meta Agent Search가 자동으로 발견한 "Structured Feedback and Ensemble Agent"는 다음과 같은 구조입니다. 먼저 5개의 CoT 추론 경로를 병렬로 실행합니다(Aggregate의 병렬 예측 단계). 여러 전문가 에이전트가 이 결과들을 비평합니다(Debate와 유사). 비평을 반영하여 답변을 반복 정제합니다(Reflect). 최종적으로 앙상블 결론을 도출합니다(Aggregate의 집계 단계). 이 에이전트는 GPT-3.5 기반으로 발견되었지만 Claude 모델로 전이했을 때도 우수한 성능을 유지했습니다. ARC Challenge에서 CoT 단독의 25.3%에서 48.3%로 약 23퍼센트 포인트가 향상된 것입니다. 이것은 발견된 설계 원리가 특정 모델에 종속되지 않는 일반적 원리임을 시사합니다.

자동화된 에이전트 설계의 비용도 주목할 만합니다. 한 연구에서 전체 최적화 과정의 비용은 약 5달러(입력 2,400만 토큰, 출력 1,100만 토큰)였으며, 최적화된 에이전트의 쿼리당 비용은 0.0014달러에 불과했습니다. 인간 설계 대비 평균 17% 이상의 성능 향상을 이 비용으로 달성할 수 있다는 것은 ADAS의 실용적 가치를 보여줍니다.

정리하면, ADAS는 에이전트 시스템 설계를 NAS처럼 자동화하는 연구 분야로, 탐색 공간, 탐색 알고리즘, 평가 함수의 세 구성 요소로 체계화됩니다. 코드 공간 탐색은 튜링 완전한 표현력으로 이론적으로 모든 에이전트 설계를 포괄하며, 메타 에이전트가 아카이브를 참조하며 반복적으로 새로운 설계를 생성하는 방식으로 탐색이 진행됩니다. 워크플로우 토폴로지는 빌딩 블록들의 연결 구조를 정의하며, Aggregate, Reflect, Debate, Summarize 등의 블록이 태스크 유형에 따라 서로 다른 최적 조합을 형성합니다.

다음 단원인 1.3.2에서는 Meta Agent Search 알고리즘의 동작 과정을 더 자세히 살펴보고, 아카이브 기반 탐색 전략과 교차 도메인, 교차 모델 전이의 메커니즘을 다룹니다.

이 단원을 마치면 ADAS 연구 분야의 정의와 동기를 NAS 유추로 설명하고, 에이전트 설계 탐색 공간의 세 가지 구성 요소를 구별할 수 있습니다.
