# 1.3.3 모듈 진화와 재결합

1.3.2에서 메타 에이전트가 아카이브를 참조하며 에이전트 전체를 코드로 설계하는 Meta Agent Search를 살펴보았습니다. 이 방식은 표현력이 높고 교차 도메인 전이도 가능하지만, 에이전트 전체를 한 번에 생성하기 때문에 어떤 모듈이 성능에 기여했는지 분리하기 어렵습니다. 이 단원에서는 에이전트를 표준화된 모듈로 분리하고, 모듈 수준에서 진화와 재결합을 수행하는 접근을 살펴봅니다. 아울러 프롬프트와 토폴로지를 동시에 최적화하는 기법도 다룹니다.

1.1.1에서 에이전트의 네 가지 핵심 모듈인 Planning, Reasoning, Tool Use, Memory를 소개했습니다. 모듈 수준 탐색의 핵심 아이디어는 이 네 모듈 각각을 독립적으로 교체 가능한 단위로 정의하고, 모듈의 조합 공간을 탐색하는 것입니다. 이를 **MoLAS(Modularized LLM Agent Search)**라고 합니다. MoLAS가 동작하려면 먼저 모듈 간 인터페이스가 표준화되어야 합니다. **표준화된 모듈 IO 인터페이스**란 각 모듈이 동일한 형식의 입력을 받고 동일한 형식의 출력을 생성하도록 정의하는 것입니다. Planning 모듈은 태스크 설명을 입력으로 받아 하위 태스크 목록을 출력하고, Reasoning 모듈은 하위 태스크를 입력으로 받아 솔루션을 출력하는 식입니다. 인터페이스가 표준화되면, 서로 다른 코드베이스에서 온 모듈들도 자유롭게 조합할 수 있습니다. 16개의 기존 에이전트에서 모듈을 추출하여 표준화한 결과 1,050가지의 조합이 가능했습니다.

이 모듈 풀(module pool)에서 새로운 모듈을 만들고 최적 조합을 찾는 두 가지 메커니즘이 **모듈 진화(Module Evolution)**와 **모듈 재결합(Module Recombination)**입니다.

**모듈 진화**는 LLM이 기존 모듈의 소스 코드를 분석하고 변형된 새 모듈을 생성하는 메커니즘입니다. 기존 Reasoning 모듈이 단순 CoT를 사용한다면, LLM에게 "이 모듈을 개선하는 변형을 작성하라"는 진화 메타 프롬프트를 제공합니다. LLM은 코드를 수정하여, 예를 들어 CoT에 자기 검증 단계를 추가하거나 추론 경로를 병렬화하는 변형 모듈을 생성합니다. 생성된 모듈 코드는 실행 테스트를 거쳐 정상 작동이 확인되면 모듈 풀에 추가됩니다. 이 과정은 생물학적 진화의 돌연변이에 비유할 수 있습니다. 기존 유전자(모듈 코드)에 변이를 가해 새로운 형질(기능)을 만들어 내는 것입니다.

**모듈 재결합**은 경험 풀(experience pool)에 축적된 이전 에이전트 구성과 성능 기록을 분석하여 유망한 모듈 조합을 추론하는 메커니즘입니다. 경험 풀에 "Planning 모듈 A + Reasoning 모듈 B의 조합이 웹 쇼핑 태스크에서 높은 성능을 냈고, Planning 모듈 C + Memory 모듈 D의 조합이 가정 환경 태스크에서 효과적이었다"는 기록이 있다면, 새로운 태스크에 대해 어떤 모듈 조합이 유망할지를 LLM이 추론합니다. 이것은 생물학적 진화의 교차(crossover)에 비유됩니다. 서로 다른 개체의 유전자를 조합하여 새로운 개체를 만드는 것입니다.

두 메커니즘은 상호 보완적입니다. 진화만 사용하면 새로운 모듈을 만들 수 있지만 1,050가지 조합 공간을 효율적으로 탐색하지 못합니다. 재결합만 사용하면 기존 모듈의 최적 조합을 찾을 수 있지만 완전히 새로운 모듈은 발견하지 못합니다. 두 메커니즘을 병행하면 모듈 풀이 점진적으로 확장되면서 동시에 조합 공간도 효율적으로 탐색됩니다.

탐색 비용을 줄이는 핵심 기법이 **성능 예측기(Performance Predictor)**입니다. 1,050가지 이상의 조합을 모두 실제로 실행하여 평가하는 것은 비용이 큽니다. 성능 예측기는 실제 평가 없이 후보 에이전트의 성능을 추정합니다. 구체적 구현은 **In-context Surrogate**입니다. 경험 풀에 축적된 (에이전트 설명, 실제 성능) 쌍을 LLM에게 컨텍스트로 제공하면, LLM이 새 후보 에이전트의 성능을 예측합니다. 예를 들어 "Planning=CoT, Reasoning=SoT, Tool=MRKL, Memory=Generative 조합의 예상 정확도는?"이라는 질문에 LLM이 경험 풀의 패턴을 참조하여 답변합니다. 이 방법으로 실제 평가 비용을 50% 이상 절감하면서도 탐색 성능을 유지했습니다. 높은 예측 점수를 받은 후보만 실제 벤치마크에서 평가하여 비용 효율을 극대화합니다.

지금까지 살펴본 모듈 수준 탐색은 개별 모듈의 품질에 초점을 맞추지만, 에이전트 시스템의 성능은 모듈 간 연결 구조, 즉 토폴로지에도 크게 의존합니다. 1.3.1에서 워크플로우 토폴로지를 소개했는데, **프롬프트-토폴로지 동시 최적화**는 각 모듈의 프롬프트(노드 수준)와 모듈 간 연결 구조(엣지 수준)를 함께 최적화하는 접근입니다. 이것을 Local-to-Global 인터리빙 최적화라고도 부릅니다.

이 최적화는 세 단계로 진행됩니다. 첫 번째 단계(1PO, Prompt Optimization)에서는 각 빌딩 블록을 독립적으로 최적화합니다. Aggregate 블록이라면 "여러 답변을 어떤 기준으로 통합할 것인가"를 지시하는 프롬프트를, Reflect 블록이라면 "어떤 관점에서 자기 답변을 검토할 것인가"를 지시하는 프롬프트를 각각 최적화합니다. 이 단계에서 각 블록의 incremental influence, 즉 해당 블록을 추가했을 때 검증 성능이 얼마나 향상되는지를 측정합니다.

두 번째 단계(2TO, Topology Optimization)에서는 블록별 incremental influence를 Softmax 함수로 확률로 변환하여 토폴로지 탐색의 사전 확률로 활용합니다. 긍정적 영향이 큰 블록은 토폴로지에 포함될 확률이 높아지고, 영향이 미미하거나 부정적인 블록은 배제됩니다. 1.3.1에서 언급한 것처럼, 유효한 토폴로지는 전체 조합 공간의 극히 일부이므로 이 가지치기가 탐색 효율을 크게 높입니다. 토폴로지 배치 순서는 논리적 의존 관계를 반영하여 Summarize, Reflect, Debate, Aggregate 순서로 고정됩니다. 긴 입력은 먼저 요약하고, 개별 에이전트가 자기 답을 반성한 뒤, 여러 에이전트가 토론하고, 최종적으로 집계하는 흐름입니다.

세 번째 단계(3PO, Global Prompt Optimization)에서는 결정된 토폴로지 위에서 전체 시스템의 프롬프트를 글로벌하게 재최적화합니다. 첫 번째 단계에서 각 블록이 독립적으로 최적화되었지만, 블록들이 실제로 연결되면 상호작용 효과가 발생합니다. 앞 블록의 출력 스타일이 뒷 블록의 입력 기대와 맞지 않을 수 있기 때문입니다. 글로벌 재최적화는 이러한 블록 간 인터페이스 불일치를 해소합니다.

이 3단계 인터리빙의 효과는 누적적입니다. 한 실험에서 기본 베이스라인 63.5%에서 1PO로 74.2%, 2TO로 77.3%, 3PO로 78.8%까지 성능이 단계적으로 향상되었습니다. 전체 최적화 비용은 약 5달러에 불과했습니다. 프롬프트의 사소한 변경이 에이전트 성능에 크게 영향을 미치는 프롬프트 민감성(prompt sensitivity) 현상이 다중 에이전트 시스템에서는 에이전트 간 종속성으로 더욱 증폭되므로, 체계적인 최적화가 수동 튜닝보다 효과적입니다.

정리하면, MoLAS는 에이전트를 표준화된 모듈로 분리하고 모듈 수준에서 탐색하는 프레임워크입니다. 모듈 진화는 LLM이 기존 모듈 코드를 변형하여 새 모듈을 생성하고, 모듈 재결합은 경험 풀의 성능 기록을 분석하여 유망한 조합을 추론합니다. In-context Surrogate 성능 예측기로 평가 비용을 절반 이상 절감할 수 있으며, 프롬프트-토폴로지 동시 최적화는 로컬 프롬프트 최적화, 토폴로지 탐색, 글로벌 재최적화의 3단계 인터리빙으로 진행됩니다.

다음 단원인 1.4.1에서는 이렇게 설계된 에이전트 시스템을 어떻게 평가하는지, 다중 환경 벤치마크의 설계 원칙과 주요 벤치마크의 특성을 살펴봅니다.

이 단원을 마치면 MoLAS 프레임워크에서 표준화된 모듈 인터페이스를 기반으로 모듈 진화와 재결합이 작동하는 메커니즘을 설명하고, 성능 예측기의 역할을 기술할 수 있습니다.
