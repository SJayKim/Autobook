# 1.4.1 에이전트 벤치마크와 평가 체계

1.1.2에서 ReAct 패턴을 통해 에이전트가 환경과 상호작용하며 태스크를 수행하는 방식을 살펴보았습니다. 에이전트가 다양한 환경에서 복잡한 태스크를 수행할 수 있다면, 그 능력을 어떻게 측정할 것인지가 자연스러운 다음 질문입니다. 단순한 텍스트 생성 능력은 MMLU나 HellaSwag 같은 벤치마크로 측정할 수 있지만, 에이전트로서의 능력은 장기 추론, 도구 활용, 오류 복구, 환경 탐색 등을 포함하며 이는 기존 벤치마크로 평가할 수 없습니다. 이 단원에서는 에이전트 평가를 위한 벤치마크의 설계 원칙과 주요 벤치마크들의 특성을 살펴봅니다.

에이전트 벤치마크를 설계할 때 가장 중요한 원칙은 **다중 환경 평가**입니다. 에이전트는 웹 브라우징, 코드 작성, 데이터베이스 조회, 물리 환경 탐색 등 서로 다른 환경에서 동작합니다. 하나의 환경에서만 평가하면 에이전트의 일반적 능력을 파악할 수 없습니다. 이 원칙에 기반한 최초의 종합 벤치마크가 **AgentBench**입니다. AgentBench는 운영체제, 데이터베이스, 지식 그래프, 카드 게임, 수평적 사고 퍼즐, 가정 환경 시뮬레이션, 웹 쇼핑, 웹 브라우징 등 8개의 서로 다른 환경에서 LLM을 에이전트로 배치하여 평가합니다. 29개 모델을 테스트한 결과, 텍스트 이해 능력이 높은 모델이 반드시 좋은 에이전트가 되는 것은 아님이 확인되었습니다. 특히 상용 모델과 오픈소스 모델 사이의 에이전트 능력 격차가 텍스트 생성 능력 격차보다 훨씬 컸습니다. 이것은 에이전트 능력이 텍스트 생성 능력과 구별되는 독자적인 역량임을 보여줍니다.

**GAIA**는 범용 AI 어시스턴트의 능력을 평가하는 벤치마크로, 실세계에서 인간이 실제로 할 법한 복합 태스크를 모아 놓았습니다. GAIA의 특징은 **단계별 난이도 분류(difficulty stratification)**입니다. 태스크가 요구하는 추론 단계의 수에 따라 세 수준으로 분류됩니다. 1수준은 1~2단계의 추론으로 해결 가능한 태스크이고, 3수준은 다수의 도구 사용과 복잡한 추론 체인이 필요한 태스크입니다. GAIA가 주목받는 이유는 인간-AI 성능 격차의 크기입니다. 인간 평가자가 92%의 정확도를 달성한 데 반해, 가장 강력한 모델인 GPT-4는 15%에 그쳤습니다. 이 격차는 현재 LLM 에이전트가 범용 어시스턴트로서 아직 갈 길이 멀다는 것을 정량적으로 보여줍니다. 동시에 이 벤치마크는 에이전트 기술의 발전을 측정하는 장기적 지표로서의 가치를 가집니다.

**OSWorld**는 실제 운영체제(Ubuntu, Windows, macOS) 환경에서 멀티모달 에이전트를 평가하는 벤치마크입니다. 시뮬레이션이 아닌 실제 OS에서 동작한다는 점이 핵심입니다. 에이전트는 스크린샷을 보고 UI 요소를 정확히 식별하여 클릭, 타이핑, 드래그 등의 행동을 수행해야 합니다. 이것을 **GUI 그라운딩(GUI grounding)**이라고 합니다. 여러 애플리케이션을 가로지르며 수행해야 하는 **멀티앱 워크플로우 태스크**도 포함됩니다. 예를 들어 이메일에서 첨부 파일을 다운로드하고, 스프레드시트에서 데이터를 분석하여, 프레젠테이션 파일에 그래프를 삽입하는 일련의 과정입니다. 인간이 72%의 성공률을 보인 반면 최고 모델은 12%에 그쳤습니다. 이 결과는 에이전트가 시각적 정보를 해석하고 복잡한 GUI 상호작용을 수행하는 것이 현재 기술의 가장 어려운 과제 중 하나임을 보여줍니다.

**SWE-bench**는 실세계 소프트웨어 엔지니어링 능력을 평가하는 벤치마크입니다. 2,294개의 실제 GitHub 이슈와 그에 대응하는 풀 리퀘스트(PR)로 구성되어 있습니다. 에이전트는 이슈를 읽고, 대규모 코드베이스를 탐색하여 관련 파일을 찾고, 버그를 수정하는 패치를 자율적으로 생성해야 합니다. 평가는 **실행 기반 자동 검증(execution-based auto-evaluation)**으로 이루어집니다. 생성된 패치를 실제 코드에 적용하고, 해당 PR에 포함된 단위 테스트를 실행하여 통과 여부를 확인합니다. 이 방식은 형식적 유사성이 아니라 기능적 정확성을 직접 검증하므로, 에이전트가 "올바르게 보이는 코드"가 아니라 "실제로 동작하는 코드"를 생성했는지를 정확히 판단합니다. SWE-bench는 코딩 에이전트 분야의 사실상 표준 벤치마크로 자리 잡았으며, 4.4.2에서 소프트웨어 엔지니어링 에이전트를 다룰 때 더 자세히 살펴봅니다.

이 벤치마크들에서 공통적으로 드러나는 패턴이 **인간-AI 성능 격차(Human-AI Performance Gap)**입니다. GAIA에서 인간 92% 대 AI 15%, OSWorld에서 인간 72% 대 AI 12%의 격차는 에이전트 기술의 현재 위치를 보여줍니다. 격차의 주된 원인은 장기 추론(long-horizon reasoning)에 있습니다. 태스크가 요구하는 추론 단계가 많아질수록 에이전트의 성공률이 급격히 하락합니다. 단일 단계 태스크에서는 상당한 성능을 보이는 모델도 10단계 이상의 연속 추론이 필요한 태스크에서는 크게 어려움을 겪습니다. 또 하나의 패턴은 **상용-오픈소스 격차(Proprietary-Open-source Gap)**입니다. AgentBench 결과에 따르면 상용 모델(GPT-4, Claude)과 오픈소스 모델 사이의 에이전트 능력 격차가 텍스트 생성 벤치마크에서의 격차보다 훨씬 큽니다. 다만 코드 생성과 실행 관련 환경에서는 이 격차가 상대적으로 작았습니다. 코드 관련 능력은 오픈소스 모델이 비교적 잘 따라잡은 영역입니다.

**평가 환경의 충실도(Environment Fidelity)**도 벤치마크 설계의 핵심 고려사항입니다. 시뮬레이션 환경이 실세계 조건을 얼마나 충실히 재현하는지에 따라 벤치마크의 예측력이 달라집니다. OSWorld가 실제 OS를, SWE-bench가 실제 GitHub 이슈를 사용하는 것은 높은 충실도를 확보하기 위함입니다. 반면 완전한 시뮬레이션 환경은 재현성과 통제성이 높지만 실세계와의 괴리가 발생할 수 있습니다. 이상적인 벤치마크는 높은 충실도와 높은 재현성을 동시에 확보해야 하며, 현재 연구 추세는 시뮬레이션보다 실세계 환경 평가를 지향합니다.

정리하면, 에이전트 벤치마크는 다중 환경 평가, 단계별 난이도 분류, 실행 기반 자동 검증, 환경 충실도의 원칙에 따라 설계됩니다. AgentBench는 8개 환경의 종합 평가, GAIA는 범용 어시스턴트의 난이도별 평가, OSWorld는 실제 OS에서의 멀티모달 평가, SWE-bench는 실세계 소프트웨어 엔지니어링 평가를 각각 담당합니다. 인간-AI 성능 격차는 에이전트 기술의 현재 한계를 정량화하며, 장기 추론 능력의 향상이 이 격차를 줄이는 핵심 과제입니다.

다음 단원인 1.4.2에서는 에이전트가 도구를 사용하고 자율적으로 행동할 때 발생하는 안전성 문제를 다루며, 다단계 해로운 행동 체인, 탈옥 공격, 악의적 요청 순응성 등 에이전트 특유의 위험을 살펴봅니다.

이 단원을 마치면 다중 환경 에이전트 벤치마크의 설계 원칙을 설명하고, AgentBench, GAIA, OSWorld, SWE-bench 등 주요 벤치마크의 평가 방식과 인간-AI 성능 격차를 비교할 수 있습니다.
