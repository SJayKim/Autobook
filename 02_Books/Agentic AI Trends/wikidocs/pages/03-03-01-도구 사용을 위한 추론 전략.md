# 3.3.1 도구 사용을 위한 추론 전략

도구 사용 능력을 갖춘 언어 모델이 실제로 문제를 해결하려면, 단순히 올바른 API 호출 형식을 아는 것만으로는 충분하지 않습니다. 어떤 도구를 언제 사용할지, 도구 결과를 어떻게 해석할지, 기대한 결과가 나오지 않았을 때 어떻게 대응할지에 대한 추론 능력이 필요합니다. 이 추론 과정은 단순한 입력-출력 패턴 매칭이 아니라, 목표를 이해하고, 행동을 계획하고, 결과를 평가하고, 필요하면 계획을 수정하는 일련의 사고 과정입니다. 이 단원에서는 도구 사용 맥락에서 발전해온 주요 추론 전략들을 살펴봅니다.

**ReAct 패턴의 도구 사용 적용**

ReAct(Reasoning and Acting)는 추론(Reasoning)과 행동(Acting)을 번갈아 수행하는 패턴입니다. 언어 모델이 어떤 목표를 받으면, 먼저 현재 상황에 대한 추론을 수행하고(Thought), 그에 따른 행동을 결정하고(Action), 행동의 결과를 관찰하고(Observation), 다시 추론으로 돌아가는 사이클을 반복합니다.

도구 사용에 ReAct를 적용하면, 이 사이클에서 Action이 도구 호출이 됩니다. 예를 들어, "파이썬 3.11에서 새로 추가된 기능을 정리해줘"라는 요청에 대해 모델은 다음과 같이 진행합니다. 먼저 Thought에서 "파이썬 3.11의 릴리스 노트를 검색해야 한다"고 판단합니다. 그 다음 Action으로 웹 검색 도구를 호출합니다. Observation에서 검색 결과를 읽습니다. 다시 Thought에서 "주요 기능들이 몇 가지 카테고리로 나뉘는군, 각각을 정리하면 된다"고 판단합니다. 이 과정을 반복하여 최종 답변을 구성합니다.

ReAct의 핵심 가치는 추론 과정을 명시적으로 텍스트로 표현한다는 점입니다. 모델이 왜 특정 도구를 호출하는지, 결과를 어떻게 해석하는지가 Thought 단계에 드러나므로, 디버깅과 설명 가능성이 크게 향상됩니다. 또한 Thought 단계에서의 추론이 다음 Action 선택의 질을 높입니다.

**Chain-of-Thought와 도구 사용의 결합**

Chain-of-Thought(CoT) 프롬프팅은 언어 모델이 단계별로 사고 과정을 서술하도록 유도하는 기법입니다. CoT만으로도 복잡한 추론 태스크에서 성능이 크게 향상됩니다. 그러나 수학 계산, 최신 정보 조회, 코드 실행 같은 태스크는 텍스트 내부의 추론만으로 해결할 수 없습니다. 여기서 CoT와 도구 사용을 결합하는 전략이 중요해집니다.

CoT와 도구 사용의 결합에서는, 텍스트로 처리할 수 있는 부분은 CoT로 추론하고, 텍스트 처리의 한계에 부딪히는 지점에서 도구를 호출합니다. 예를 들어, "1부터 100까지의 소수의 합을 구하라"는 요청에 대해, 소수의 정의를 기억하고 알고리즘을 설계하는 것은 CoT로 처리하되, 실제 계산은 코드 실행 도구에 맡깁니다. 이렇게 하면 언어 모델의 추론 능력과 도구의 정확한 실행 능력이 상호 보완됩니다.

이 결합에서 중요한 설계 결정 중 하나는 "어느 시점에 텍스트 추론을 멈추고 도구를 호출할 것인가"입니다. 너무 일찍 도구를 호출하면 필요하지 않은 도구 사용이 증가하고, 너무 늦게 호출하면 텍스트 추론으로 처리하기 어려운 부분에서 오류가 발생합니다. 모델은 이 판단을 명시적인 Thought 단계에서 수행하도록 훈련됩니다.

**DFSDT: 선형 추론에서 트리 탐색으로**

ReAct와 CoT 기반 도구 사용은 기본적으로 선형(linear) 추론 구조입니다. 한 번에 하나의 행동을 선택하고, 그 결과에 따라 다음 행동을 결정합니다. 이 구조는 간단하고 직관적이지만, 중간 단계에서 잘못된 판단을 하면 전체 해결 과정이 틀린 방향으로 흘러갈 수 있습니다. 도구 호출 오류, 예상과 다른 결과, 다중 해결 경로 중 최선 선택 등의 상황에서 선형 구조는 취약합니다.

DFSDT(Depth-First Search with Decision Trees)는 이 한계를 극복하기 위해 선형 추론을 트리 탐색 구조로 확장합니다. 각 추론 단계에서 하나의 행동만 선택하는 대신, 여러 가능한 행동 후보를 생성하고, 그 중 하나를 선택하여 탐색을 진행합니다. 선택된 경로가 막히거나 좋지 않은 결과를 낳으면, 이전 결정 지점으로 돌아가 다른 행동 후보를 시도합니다.

DFSDT의 이름이 암시하듯, 탐색 구조는 깊이 우선 탐색(DFS)을 기본으로 합니다. 현재 선택한 경로를 최대한 깊이 탐색하다가, 막히면 백트래킹을 통해 최근의 분기점으로 돌아갑니다. 이 탐색 과정에서 각 노드는 하나의 (추론 상태, 도구 호출, 관찰) 트리플을 나타내며, 탐색 종료 조건은 목표 달성이나 최대 탐색 깊이 초과입니다.

**백트래킹과 재탐색**

백트래킹(backtracking)은 DFSDT의 핵심 메커니즘입니다. 도구 사용 과정에서 막히는 상황은 다양합니다. 도구 호출이 오류를 반환한 경우, 결과가 기대와 달라 다음 단계를 진행하기 어려운 경우, 현재 경로가 목표와 점점 멀어지는 경우 등입니다. 선형 추론에서는 이런 상황에서 현재 경로를 포기하고 처음부터 다시 시작해야 합니다. 백트래킹을 사용하면, 전체를 재시작하지 않고 마지막으로 유효한 결정 지점으로만 돌아갑니다.

백트래킹의 효과는 특히 복잡한 다단계 도구 사용에서 두드러집니다. 5단계 도구 호출 시퀀스에서 4단계에서 오류가 발생했을 때, 처음 3단계는 이미 올바르게 수행됐으므로 4단계 결정만 다시 내리면 됩니다. 이렇게 하면 탐색 효율이 크게 높아집니다.

재탐색(re-exploration)은 백트래킹 후에 이전과 다른 행동을 선택하는 과정입니다. 단순히 동일한 행동을 반복하는 것이 아니라, 왜 이전 경로가 실패했는지 추론하고 그 원인을 회피하는 방향으로 다른 후보를 선택합니다. 이 "실패 원인 분석 후 다른 시도"의 능력이 DFSDT를 단순한 브루트포스 탐색과 구분짓는 특성입니다.

**솔루션 경로 도출**

트리 탐색을 통해 목표를 달성했을 때, 탐색 과정 전체가 솔루션은 아닙니다. 성공적인 경로만을 추출하여 솔루션 경로(solution path)를 구성하고, 이를 최종 답변으로 제시합니다. 솔루션 경로는 탐색 과정에서 발생한 실패 시도, 백트래킹, 재탐색을 모두 제거하고, 목표 달성에 기여한 행동과 추론만으로 구성됩니다.

솔루션 경로는 학습 데이터로도 활용됩니다. 성공한 탐색으로부터 솔루션 경로를 추출하면, 직접 도달 가능한 최적 궤적을 학습 데이터로 사용할 수 있습니다. 이렇게 하면 모델이 탐색 과정에서 발생한 불필요한 시도들을 학습하지 않고, 효율적인 문제 해결 경로만을 학습하게 됩니다.

**프롬프트 자동 최적화**

도구 사용 추론의 품질은 프롬프트 설계에도 크게 의존합니다. 어떤 추론 단계를 강조할지, 도구 호출 형식을 어떻게 지정할지, 오류 처리 방법을 어떻게 안내할지에 따라 성능이 달라집니다. 그러나 최적의 프롬프트를 수동으로 설계하는 것은 어렵고 도메인마다 다른 접근이 필요합니다.

프롬프트 자동 최적화(automatic prompt optimization)는 이 과정을 자동화합니다. 여러 프롬프트 후보를 체계적으로 생성하고, 각 프롬프트를 사용한 도구 사용 결과를 평가하여 성능이 좋은 프롬프트를 선택합니다. 이 과정을 반복하면 도메인과 태스크에 특화된 최적 프롬프트를 찾을 수 있습니다.

자동 최적화에서 중요한 것은 평가 기준입니다. 도구 사용의 성공 여부, 사용한 도구 호출 횟수, 백트래킹 발생 빈도, 최종 답변의 품질 등을 종합적으로 고려합니다. 특히 태스크별로 중요한 평가 기준이 다를 수 있으므로, 최적화 목적 함수를 태스크에 맞게 설정하는 것이 중요합니다.

**긍정/부정 궤적 대조 분석**

성공한 도구 사용 궤적(긍정 궤적)과 실패한 궤적(부정 궤적)을 대조 분석하면, 좋은 추론 전략의 특성을 체계적으로 파악할 수 있습니다. 이 대조 분석은 모델 훈련과 프롬프트 설계 모두에 활용됩니다.

훈련 관점에서, 대조 학습(contrastive learning)을 사용하면 긍정 궤적과 유사하고 부정 궤적과 다른 패턴을 학습할 수 있습니다. 예를 들어, 오류 발생 후 즉시 백트래킹하여 문제를 해결한 궤적(긍정)과 오류를 무시하고 계속 진행하여 실패한 궤적(부정)을 대조하면, 모델이 오류 감지와 백트래킹 능력을 강화하는 방향으로 학습합니다.

분석 관점에서, 실패 패턴을 분류하면 시스템의 약점을 파악할 수 있습니다. 잘못된 도구 선택으로 인한 실패인지, API 파라미터 오류인지, 결과 해석 오류인지, 불필요한 도구 호출의 누적으로 인한 효율 저하인지에 따라 개선 방향이 달라집니다. 이 분석 결과는 합성 데이터 생성 전략(3.2.3 참조)에도 피드백되어, 부족한 영역의 학습 데이터를 집중적으로 생성하는 데 활용됩니다.

정리하면, 도구 사용을 위한 추론 전략은 단순한 API 호출 패턴 학습을 넘어, 목표 지향적 계획과 실행을 가능하게 하는 핵심 능력입니다. ReAct는 추론과 행동의 명시적 사이클로 투명한 문제 해결을 가능하게 하고, CoT와의 결합은 텍스트 추론과 도구 실행의 장점을 결합합니다. DFSDT는 선형 추론의 한계를 트리 탐색으로 극복하여 복잡한 태스크에서 더 강건한 성능을 보입니다. 긍정/부정 궤적 대조 분석과 프롬프트 자동 최적화는 이러한 추론 전략의 지속적인 개선을 가능하게 합니다.

다음 단원인 3.3.2에서는 이렇게 개발된 도구 사용 시스템을 어떻게 객관적으로 평가하고 비교할 것인지 다룹니다. AST 서브트리 매칭, 환각률과 오류율 분리, pass^k 메트릭 등 도구 사용 평가에 특화된 방법론들을 살펴봅니다.

이 단원을 학습한 후에는 ReAct 패턴의 구조와 도구 사용 적용 방식, CoT와 도구 사용 결합의 원리, DFSDT의 트리 탐색 메커니즘과 백트래킹 작동 방식, 솔루션 경로의 의미, 그리고 긍정/부정 궤적 대조 분석의 활용 방법을 설명할 수 있습니다.
