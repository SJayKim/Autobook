# 3.1.1 LLM의 한계와 도구의 필요성

1.1.1에서 에이전트의 네 가지 핵심 모듈 중 하나로 Tool Use를 소개하면서, 에이전트가 외부 도구를 호출하여 LLM의 한계를 극복한다고 설명했습니다. Phase 3에서는 이 도구 사용을 본격적으로 다룹니다. 먼저 이 단원에서는 LLM이 왜 도구를 필요로 하는지, 그 구조적 한계를 구체적으로 살펴봅니다.

LLM은 사전학습 데이터에서 추출한 패턴을 기반으로 텍스트를 생성합니다. 이 능력은 강력하지만, 근본적으로 세 가지 구조적 한계를 가지고 있습니다.

첫 번째 한계는 **산술 연산의 부정확성**입니다. LLM은 수학 문제를 "계산"하는 것이 아니라, 학습 데이터에서 유사한 패턴을 찾아 텍스트를 "생성"합니다. 간단한 덧셈이나 곱셈은 패턴 매칭으로 올바른 답을 생성하는 경우가 많지만, 숫자가 커지거나 연산이 복잡해지면 오류가 빈번합니다. "127 곱하기 389는?"이라는 질문에 LLM은 "대략 49,000 정도"라는 답을 생성할 수 있지만, 정확한 값인 49,403을 보장하지 못합니다. 이것은 LLM의 학습 부족이 아니라, 텍스트 생성이라는 작동 원리 자체의 한계입니다. 반면 계산기나 Python 인터프리터에게 같은 질문을 하면 항상 정확한 답을 반환합니다.

두 번째 한계는 **최신 정보에 대한 접근 불가**입니다. LLM의 지식은 학습이 완료된 시점에 동결됩니다. 학습 시점 이후에 발생한 사건, 변경된 API 문서, 새로 발표된 연구 등에 대해서는 알 수 없습니다. 사용자가 "오늘 서울의 날씨는?"이라고 물으면, LLM은 실시간 날씨 데이터에 접근할 수 없으므로 정확한 답을 제공할 수 없습니다. 또한 API 문서는 빈번하게 변경되는데, 모델을 매번 재학습하지 않고도 최신 문서에 적응해야 하는 실용적 요구가 있습니다.

세 번째 한계는 **시간 인식의 부재**입니다. LLM은 현재 시각, 오늘 날짜, 경과 시간 등을 자체적으로 파악할 수 없습니다. "지금 몇 시야?"라는 질문에 정확히 답할 수 없고, "3시간 후에 알려 줘"라는 요청을 처리할 수 없습니다. 시간 인식이 필요한 모든 태스크에서 외부 시스템의 도움이 필수적입니다.

이 세 가지 한계를 극복하는 것이 **외부 도구**의 역할입니다. 산술 연산은 계산기나 코드 실행기에 위임하고, 최신 정보는 검색 API나 웹 크롤러로 획득하며, 시간 인식은 시스템 시계를 호출합니다. 도구는 LLM이 "잘 못하는 것"을 "잘하는 시스템"에 위임하는 보완 메커니즘입니다.

그런데 도구를 사용하는 과정에서 발생하는 고유한 문제가 있습니다. **API 환각(hallucination)**입니다. LLM이 실제로 존재하지 않는 API를 호출하거나, 존재하는 API의 매개변수를 잘못 구성하는 현상입니다. LLM은 학습 데이터에서 API 호출 패턴을 학습했지만, 특정 API의 정확한 시그니처를 "기억"하는 것이 아니라 "생성"합니다. 따라서 API 이름을 살짝 틀리거나, 존재하지 않는 매개변수를 만들어 내거나, 반환값의 구조를 잘못 예상하는 경우가 빈번합니다. API 환각은 일반 텍스트 환각보다 실용적 영향이 더 큽니다. 텍스트 환각은 사용자가 직접 검증할 수 있지만, API 환각은 시스템 오류를 직접 유발합니다.

도구 사용 능력에서 **폐쇄형 모델과 오픈소스 모델 간의 격차**도 중요한 맥락입니다. 1.4.1에서 AgentBench 결과를 통해 에이전트 능력의 상용-오픈소스 격차를 다루었는데, 도구 사용 영역에서도 비슷한 패턴이 관찰됩니다. 최상위 상용 모델(GPT-4 등)은 복잡한 도구 호출 체인을 비교적 정확하게 수행하지만, 오픈소스 모델은 단일 도구 호출에서도 오류율이 높은 경우가 있습니다. 다만 이 격차는 도구 사용에 특화된 학습 데이터와 파인튜닝으로 빠르게 좁혀지고 있으며, 7~8B 규모의 소형 모델도 적절한 학습을 거치면 대형 모델에 근접하는 도구 사용 능력을 달성한 사례가 보고되었습니다.

현실 세계의 태스크가 도구를 필요로 하는 가장 근본적인 이유는 **태스크의 조합적(compositional) 성격**입니다. "이번 분기 매출 데이터를 분석하고, 전분기 대비 추세를 시각화하고, 보고서를 작성해 줘"라는 요청은 단일 도구 호출로 해결되지 않습니다. 데이터베이스에서 데이터를 가져오고(SQL 쿼리), Python으로 분석하고(코드 실행), 그래프를 생성하고(시각화 라이브러리), 결과를 문서로 정리하는(텍스트 생성) 여러 단계를 거쳐야 합니다. 각 단계에서 서로 다른 도구가 필요하며, 이전 단계의 출력이 다음 단계의 입력이 됩니다. 이 조합적 성격이 1.1.2에서 다룬 Thought-Action-Observation 루프가 필요한 이유이기도 합니다. 에이전트는 각 단계에서 적절한 도구를 선택하고, 호출 결과를 해석하여 다음 단계를 결정해야 합니다.

정리하면, LLM은 산술 연산의 부정확성, 최신 정보 접근 불가, 시간 인식 부재라는 세 가지 구조적 한계를 가지며, 외부 도구가 이를 보완합니다. API 환각은 도구 사용 과정의 고유한 문제로, LLM이 존재하지 않는 API를 생성하거나 매개변수를 잘못 구성하는 현상입니다. 폐쇄형-오픈소스 모델 간 도구 사용 격차는 특화 학습으로 좁혀지고 있으며, 현실 태스크의 조합적 성격이 다중 도구 호출 체인을 필수적으로 만듭니다.

다음 단원인 3.1.2에서는 도구를 호출하는 기본 메커니즘인 Function Calling의 구조를 살펴봅니다.

이 단원을 마치면 LLM의 구조적 한계를 구체적으로 설명하고, API 환각 문제와 오픈소스-폐쇄형 모델 간 도구 사용 능력 격차를 기술할 수 있습니다.
