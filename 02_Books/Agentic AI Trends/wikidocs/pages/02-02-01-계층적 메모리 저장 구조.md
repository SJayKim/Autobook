# 2.2.1 계층적 메모리 저장 구조

2.1.3에서 인지과학의 기억 분류가 에이전트 메모리 설계에 적용되는 방식을 살펴보았습니다. 이 단원에서는 그 분류가 구체적으로 어떤 저장 아키텍처로 구현되는지, 특히 여러 시간 범위의 정보를 층위별로 관리하는 계층적 메모리 저장 구조를 다룹니다.

에이전트가 처리하는 정보는 시간적 범위가 다양합니다. 방금 사용자가 말한 내용은 몇 초 전의 정보이고, 오늘 대화에서 논의한 주제는 몇 시간 범위의 정보이며, 지난 몇 주간 사용자와 나눈 대화에서 파악한 선호도는 몇 주 범위의 정보입니다. 이 서로 다른 시간 범위의 정보를 하나의 저장소에 평면적으로 넣으면, 방금 말한 내용과 몇 주 전의 기억이 동일한 우선순위로 경쟁하게 되어 검색 효율이 떨어집니다. 인간의 뇌가 단기 기억, 중기 기억, 장기 기억을 분리하여 관리하듯, 에이전트 메모리도 시간 범위에 따른 계층적 구조가 필요합니다.

가장 체계적으로 구현된 계층 구조는 **STM-MTM-LTM 3계층 모델**입니다. **Short-Term Memory(STM)**는 현재 세션의 최근 대화를 FIFO(First-In-First-Out) 방식으로 관리하는 단기 기억입니다. 가장 최근의 대화 턴들이 여기에 위치하며, 용량이 가득 차면 가장 오래된 항목부터 밀려납니다. STM은 에이전트의 즉각적인 맥락을 유지하는 역할을 합니다. **Mid-Term Memory(MTM)**는 STM에서 넘어온 최근 대화를 보관하는 중기 기억 계층입니다. STM보다 더 넓은 시간 범위를 다루며, 최근 몇 시간에서 며칠 사이의 상호작용 기록이 여기에 저장됩니다. **Long-Term Memory(LTM)**는 사용자 선호, 습관, 페르소나 등을 영속적으로 저장하는 장기 기억입니다. 여러 세션에 걸쳐 축적된 정보가 여기에 보관되며, 시간이 지나도 소멸하지 않습니다.

이 3계층 모델이 실제로 어떻게 작동하는지를 이해하기 위해, 컴퓨터 운영체제의 가상 메모리 시스템과의 비유를 살펴보겠습니다. 이 비유는 에이전트 메모리 연구에서 가장 영향력 있는 아이디어 중 하나입니다. 운영체제에서 프로그램은 물리적 RAM보다 훨씬 큰 메모리를 사용하는 것처럼 보입니다. 이것은 OS가 RAM(빠르지만 작음)과 디스크(느리지만 큼) 사이에서 데이터를 자동으로 교체하는 페이징(paging) 메커니즘 덕분입니다. 프로그램은 RAM의 크기를 의식하지 않아도 되며, OS가 필요한 데이터를 적시에 RAM에 올려줍니다.

에이전트 메모리에 이 비유를 적용하면, **Main Context**가 RAM에 해당합니다. LLM의 프롬프트 토큰 공간, 즉 현재 추론 호출에서 모델이 실제로 "볼 수 있는" 정보입니다. Main Context 안에는 두 영역이 있습니다. 하나는 **Working Context**로, 에이전트가 읽고 쓸 수 있는 핵심 저장소입니다. 시스템 프롬프트, 사용자의 핵심 정보, 현재 태스크의 맥락이 여기에 위치합니다. 다른 하나는 **FIFO Queue**로, 최근 대화 메시지가 순서대로 저장되는 큐입니다. 새 메시지가 들어오면 가장 오래된 메시지가 밀려납니다.

**External Context**가 디스크에 해당합니다. Main Context에 들어가지 못하는 정보를 저장하는 외부 저장소입니다. 두 종류가 있습니다. **Recall Storage**는 대화 이력을 시간순으로 저장하는 공간입니다. FIFO Queue에서 밀려난 메시지가 여기로 이동합니다. 시간 기반 검색과 텍스트 검색을 지원합니다. **Archival Storage**는 임의의 지식이나 문서를 저장하는 공간입니다. 대화 이력이 아닌, 에이전트가 참조해야 하는 외부 정보를 보관합니다. 벡터 검색을 통해 의미 기반으로 정보를 인출합니다.

이 구조의 핵심은 페이징이 LLM 자체에 의해 수행된다는 점입니다. OS에서 페이징을 OS 커널이 자동으로 수행하듯, 에이전트에서는 LLM이 함수 호출(function call)을 통해 Main Context와 External Context 사이의 데이터 교체를 자율적으로 수행합니다. 에이전트가 과거 대화 내용이 필요하다고 판단하면 Recall Storage에서 검색 함수를 호출하고, 그 결과를 Working Context에 삽입합니다. 장기 보관이 필요한 정보는 Archival Storage에 저장 함수를 호출하여 기록합니다.

Main Context의 용량이 한계에 가까워지면 **Memory Pressure 경고 메커니즘**이 작동합니다. 컨텍스트 사용률이 임계값(예: 70%)에 도달하면 에이전트에게 "메모리가 부족하다"는 경고가 전달되고, 에이전트는 덜 중요한 정보를 External Context로 이동시키거나, FIFO Queue의 내용을 요약하여 압축합니다. 이 과정을 **Queue Flush**라고 합니다. Queue에서 밀려나는 메시지들은 LLM에 의해 요약되어 Working Context의 요약 영역에 반영되고, 원본 메시지는 Recall Storage에 보존됩니다. 요약을 통해 상세한 내용은 사라지지만 핵심 맥락은 유지됩니다. 요약의 요약이 다시 생성되는 재귀적 요약(recursive summarization)도 가능합니다.

이 계층 구조가 가져오는 효과는 정량적으로도 뚜렷합니다. 고정 컨텍스트만 사용하는 모델이 심층 기억 검색 태스크에서 약 32%의 정확도를 보인 반면, 계층적 메모리를 갖춘 에이전트는 약 93%를 달성했습니다. 문서 질의응답에서도 고정 컨텍스트 방식은 검색 문서 수가 늘수록 성능이 포화된 반면, 계층적 메모리는 안정적 성능을 유지했습니다. 2.1.1에서 설명한 Lost in the Middle 문제를 우회하는 효과도 있습니다. 모든 정보를 한꺼번에 컨텍스트에 넣는 대신, 필요한 시점에 외부 저장소에서 검색하여 컨텍스트의 앞부분에 배치하므로 위치 편향의 영향을 받지 않습니다.

STM에서 LTM으로의 정보 이동은 자동으로 일어나지 않습니다. 어떤 정보가 장기 기억으로 승격될 가치가 있는지를 판단하는 메커니즘이 필요합니다. 한 가지 접근은 **열도(Heat) 기반 메모리 승격**입니다. 자주 참조되거나 중요한 메모리의 "열도"를 추적하여, 열도가 임계값을 초과한 메모리를 STM에서 MTM으로, MTM에서 LTM으로 순차적으로 승격시키는 메커니즘입니다. 자주 참조되는 정보는 열도가 높아져 상위 계층으로 올라가고, 오래 참조되지 않는 정보는 열도가 식어 하위 계층에 머무르거나 삭제됩니다. 이것은 인간의 기억에서 반복 노출된 정보가 장기 기억으로 공고화(consolidation)되는 메커니즘의 공학적 구현입니다.

또 다른 접근은 **세그먼트 페이징(Segment Paging)**입니다. 대화를 주제별 세그먼트로 분할하고, 세그먼트 단위로 메모리를 교체하는 방식입니다. 이 접근은 2.2.3에서 메모리 세분성을 다룰 때 더 자세히 살펴봅니다.

정리하면, 계층적 메모리 저장 구조는 STM(단기), MTM(중기), LTM(장기)의 세 계층으로 시간 범위별 정보를 분리 관리합니다. OS의 가상 메모리 비유에 따라 Main Context(RAM)는 Working Context와 FIFO Queue로, External Context(디스크)는 Recall Storage와 Archival Storage로 구성됩니다. LLM이 함수 호출을 통해 두 영역 사이의 페이징을 자율적으로 수행하며, Memory Pressure 경고와 Queue Flush 메커니즘이 컨텍스트 용량을 관리합니다. 열도 기반 승격은 자주 참조되는 정보를 상위 계층으로 이동시키는 메커니즘입니다.

다음 단원인 2.2.2에서는 정보를 노드와 엣지로 연결하여 관계를 명시적으로 표현하는 그래프 기반 메모리 저장을 살펴봅니다.

이 단원을 마치면 STM/MTM/LTM 계층 구조와 OS 가상 메모리 비유를 단계별로 설명하고, Main Context와 External Storage 간 페이징 메커니즘이 무한 컨텍스트의 환상을 어떻게 구현하는지 기술할 수 있습니다.
