# 3.2.2 API 검색과 Retriever-Aware 학습

현실의 소프트웨어 생태계에는 수천 개, 경우에 따라서는 수만 개의 API가 존재합니다. 단일 플랫폼에서 제공하는 API만 해도 수백 개를 넘는 경우가 흔하고, 서드파티 서비스와 오픈소스 라이브러리까지 포함하면 그 수는 급격히 늘어납니다. 언어 모델이 이 모든 API를 파라미터 안에 외워서 사용하는 것은 불가능합니다. 모델의 컨텍스트 창(context window)에 모든 API 명세를 동시에 넣는 것도 현실적으로 어렵습니다. 이 문제를 해결하기 위해 등장한 것이 API 검색(API retrieval) 기법입니다. 모델이 필요한 시점에 관련 API를 동적으로 검색하여 맥락에 포함시키는 방식입니다.

**대규모 API 생태계의 규모 문제**

API 검색 문제가 어려운 이유는 단순히 API 수가 많다는 것만이 아닙니다. API는 서로 다른 도메인, 다른 설계 철학, 다른 명명 규칙을 가지고 있어서, 사용자의 자연어 요청과 적절한 API를 연결하는 것 자체가 의미론적(semantic) 이해를 요구합니다. 예를 들어, "현재 위치의 날씨를 알려줘"라는 요청에 대해 weather.getCurrentConditions, getWeatherByCoords, fetchWeatherData 같은 다양한 이름의 API 중에서 가장 적합한 것을 찾아야 합니다. 키워드 매칭만으로는 이 연결이 정확하지 않을 수 있습니다.

또한 API는 정적이지 않습니다. 새로운 API가 계속 추가되고, 기존 API는 버전 업데이트와 함께 명세가 바뀝니다. 모델을 매번 재학습시키지 않고도 새로운 API를 활용할 수 있어야 한다는 요건이 있습니다. 이것이 파라미터 안에 API를 내재화하는 방식의 근본적인 한계이며, 검색 기반 접근법이 필요한 핵심 이유입니다.

**희소 검색과 밀집 검색**

API 검색에는 크게 두 가지 방법론이 사용됩니다. 첫 번째는 희소 검색(sparse retrieval)이고, 두 번째는 밀집 검색(dense retrieval)입니다.

희소 검색의 대표적인 방법은 BM25입니다. BM25는 문서(이 경우 API 명세)와 질의(사용자 요청) 간의 단어 빈도 기반 유사도를 계산합니다. 계산이 빠르고 해석 가능하며, 특정 키워드가 정확히 일치하는 경우 높은 정밀도를 보입니다. 그러나 동의어나 의미론적으로 유사하지만 단어가 다른 경우를 처리하지 못한다는 단점이 있습니다. "날씨 정보"와 "기상 데이터"는 같은 의미이지만 BM25는 이를 다른 것으로 취급합니다.

밀집 검색(dense retrieval)은 신경망 임베딩을 사용하여 질의와 API 명세를 같은 벡터 공간에 표현하고, 벡터 간 유사도로 관련성을 측정합니다. 의미론적으로 유사한 텍스트는 임베딩 공간에서 가깝게 위치하므로, 동의어나 표현 방식이 다른 경우도 처리할 수 있습니다. GPT-Index 방식은 GPT 계열 모델의 임베딩을 사용하여 API 명세를 인덱싱하고 쿼리 시 근사 최근접 탐색(approximate nearest neighbor search)으로 관련 API를 가져옵니다. 밀집 검색은 희소 검색에 비해 계산 비용이 높지만, 의미론적 매칭 품질이 훨씬 우수합니다. 실용적인 시스템에서는 두 방법을 혼합(hybrid retrieval)하여 희소 검색의 정확도와 밀집 검색의 재현율을 모두 활용하는 경우가 많습니다.

**Retriever-Aware Fine-tuning**

단순히 검색기를 붙여서 사용하는 것과, 검색기와 함께 작동하도록 모델을 훈련시키는 것은 다른 문제입니다. Retriever-Aware Fine-tuning(RAF)은 언어 모델이 검색기가 제공하는 API 명세를 효과적으로 활용하도록 파인튜닝하는 접근법입니다.

검색기가 제공한 API 명세는 종종 노이즈를 포함합니다. 완전히 관련이 없는 API가 포함되기도 하고, 여러 개의 비슷한 API 중에서 가장 적합한 것을 선택해야 하기도 합니다. RAF 이전의 모델은 이런 노이즈에 취약하여, 검색된 API 명세가 있어도 엉뚱한 API를 선택하거나 필요 없는 API를 호출하는 오류를 자주 보였습니다.

RAF에서는 학습 데이터를 구성할 때, 실제 검색 상황을 시뮬레이션합니다. 정답 API 외에도 검색기가 혼동할 법한 오답 API 명세를 함께 컨텍스트에 포함시키고, 그 상황에서 올바른 API를 선택하도록 훈련합니다. 이렇게 하면 모델이 노이즈가 있는 검색 결과 속에서도 적절한 API를 식별하는 능력을 기릅니다. 또한 모델이 컨텍스트에서 API 명세를 적극적으로 읽고 활용하는 습관을 형성하게 됩니다.

**신경망 API 검색기**

검색기 자체도 언어 모델 기반으로 구축하는 것이 가능합니다. 신경망 API 검색기(neural API retriever)는 사용자의 자연어 요청을 입력받아 관련 API를 반환하는 모델로, API 사용 이력과 자연어 쿼리 쌍으로 학습됩니다. 전통적인 정보 검색(information retrieval) 기법에 기반한 BM25와 달리, 신경망 검색기는 태스크 맥락 전체를 고려하여 API를 선택합니다.

신경망 검색기는 이중 인코더(bi-encoder) 구조를 많이 사용합니다. 질의 인코더는 사용자 요청을 벡터로 변환하고, 문서 인코더는 API 명세를 벡터로 변환합니다. 두 벡터 간의 내적(dot product) 또는 코사인 유사도(cosine similarity)로 관련성을 계산합니다. 이 구조는 API 명세 벡터를 오프라인으로 미리 계산해 둘 수 있어서, 실시간 검색 시에는 질의 벡터 계산과 근사 최근접 탐색만 수행하면 됩니다. 수만 개의 API 풀에서도 밀리초 단위의 검색이 가능합니다.

**검색기 품질이 핵심 병목인 이유**

API 검색 기반 도구 사용 시스템에서 검색기의 품질은 전체 시스템의 성능을 결정하는 핵심 병목(bottleneck)입니다. 검색기가 잘못된 API를 가져오면, 아무리 언어 모델이 뛰어나도 올바른 도구를 사용할 수 없습니다. 필요한 API가 검색 결과에 포함되지 않는 재현율(recall) 실패는 복구 불가능한 오류입니다.

실험적으로도 이 점이 확인됩니다. 동일한 언어 모델과 동일한 API 풀을 사용했을 때, 검색기를 BM25에서 신경망 검색기로 교체하는 것만으로 최종 태스크 성공률이 10~20% 향상되는 사례가 보고됩니다. 반면, 같은 검색기를 유지하면서 언어 모델 크기를 늘리는 것의 효과는 이보다 작은 경우가 많습니다. 이는 검색기 개선에 투자하는 것이 시스템 전체의 효율을 높이는 데 더 효과적일 수 있음을 시사합니다.

검색기 품질에 영향을 미치는 주요 요인으로는 인덱스 구축 방식, 임베딩 모델의 도메인 적합성, 질의 전처리 방법, 그리고 검색 결과를 랭킹하는 방식 등이 있습니다. 특히 새로운 API가 지속적으로 추가되는 환경에서는 인덱스의 신선도(freshness)도 중요한 요소가 됩니다.

**테스트 타임 문서 적응**

학습 후에 새로운 API가 추가되거나 기존 API 명세가 변경되었을 때, 모델을 재학습하지 않고도 적응할 수 있어야 합니다. 테스트 타임 문서 적응(test-time document adaptation)은 이 문제에 대한 접근법입니다.

가장 단순한 형태는 새로운 API 명세를 단순히 검색 인덱스에 추가하는 것입니다. 모델이 검색된 API 명세를 컨텍스트에서 읽고 활용하는 방식을 학습했다면, 인덱스에 새로운 API를 추가하는 것만으로 해당 API를 사용할 수 있게 됩니다. 이를 플러그인(plug-in) 방식의 도구 확장이라고 부르기도 합니다.

더 정교한 방법은 새로운 API에 대한 소수의 사용 예시를 제공하여 인컨텍스트 학습(in-context learning)을 활성화하는 것입니다. 이 경우 API 명세뿐만 아니라 "이런 상황에서 이렇게 사용했다"는 예시가 함께 제공되어, 모델이 새로운 API의 사용법을 빠르게 파악할 수 있습니다.

**API Context Tree**

검색된 여러 API 명세를 어떻게 언어 모델의 컨텍스트에 조직화할 것인가도 중요한 설계 문제입니다. API Context Tree는 이를 위한 구조화된 표현 방식입니다. 단순히 API 명세 텍스트를 나열하는 대신, API들의 관계(예: 같은 서비스의 API들, 유사한 기능의 API들)를 트리 구조로 조직화하여 모델이 더 효율적으로 탐색할 수 있도록 합니다.

API Context Tree에서 루트 노드는 현재 태스크와 관련된 최상위 서비스 카테고리를 나타내고, 자식 노드로 내려갈수록 구체적인 API와 파라미터 정보가 담깁니다. 이 구조를 사용하면 모델이 필요한 정보를 계층적으로 탐색할 수 있어, 수십 개의 API 명세가 한꺼번에 컨텍스트에 들어와도 혼란 없이 처리할 수 있습니다. 또한 트리 구조는 API 간의 의존 관계나 순서를 자연스럽게 표현할 수 있는 장점도 있습니다.

정리하면, 대규모 API 생태계에서 도구 사용 능력을 갖추기 위해서는 필요한 API를 동적으로 검색하는 기술이 필수적입니다. BM25 기반 희소 검색과 임베딩 기반 밀집 검색이 두 축을 이루며, 검색기와 언어 모델을 함께 훈련하는 RAF가 시스템 성능을 높입니다. 검색기 품질은 전체 시스템의 핵심 병목이며, 테스트 타임 문서 적응과 API Context Tree는 새로운 API를 유연하게 통합하는 수단을 제공합니다.

다음 단원인 3.2.3에서는 실제 API 호출 없이도 도구 학습 데이터를 생성하는 합성 데이터 기반 접근법을 살펴봅니다. 특히 Bottom-Up과 Top-Down 방식의 데이터 생성 전략, 그리고 3-에이전트 시뮬레이션을 통한 복잡한 도구 사용 궤적 합성을 다룹니다.

이 단원을 학습한 후에는 API 검색이 필요한 이유, 희소 검색과 밀집 검색의 차이, Retriever-Aware Fine-tuning의 핵심 아이디어, 검색기가 전체 시스템의 병목인 이유, 그리고 API Context Tree의 역할을 설명할 수 있습니다.
