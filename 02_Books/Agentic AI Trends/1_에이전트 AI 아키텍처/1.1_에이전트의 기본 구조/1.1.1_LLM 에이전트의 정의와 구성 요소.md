# 1.1.1 LLM 에이전트의 정의와 구성 요소

대규모 언어 모델(LLM)은 텍스트를 생성하는 능력으로 널리 알려져 있습니다. 사용자가 질문을 하면 모델이 답변을 생성하고, 그것으로 상호작용이 끝나는 구조입니다. 이 방식은 한 번의 질의에 한 번의 응답을 돌려주는 단방향 흐름이며, 모델은 답변을 생성한 뒤 다음 질문이 올 때까지 아무것도 하지 않습니다. 그런데 실제 세계의 작업은 이렇게 단순하지 않습니다. "이번 분기 매출 데이터를 분석하고, 추세를 시각화한 뒤, 보고서 초안을 작성해 줘"라는 요청을 생각해 보겠습니다. 이 작업을 수행하려면 데이터를 가져오고, 코드를 실행하여 그래프를 만들고, 결과를 해석하여 문서로 정리하는 여러 단계를 거쳐야 합니다. 단일 텍스트 응답으로는 이 과정을 완결할 수 없습니다.

LLM 에이전트는 이 문제를 해결하기 위해 등장한 패러다임입니다. 여기서 LLM은 단순한 텍스트 생성기가 아니라, 복잡한 작업 흐름 전체를 제어하는 중추적인 의사결정 엔진으로 작동합니다. 에이전트는 주어진 목표를 달성하기 위해 스스로 계획을 세우고, 외부 도구를 호출하며, 그 결과를 관찰하여 다음 행동을 결정하는 과정을 반복합니다. 이 과정에서 LLM은 두뇌의 역할을 담당합니다. 어떤 순서로 무엇을 해야 하는지 판단하고, 각 단계의 결과를 해석하여 전략을 수정합니다. 기존의 질의-응답 모델이 한 번 말하고 멈추는 구조라면, 에이전트는 목표가 달성될 때까지 계속해서 사고하고 행동하는 구조입니다. 이것을 **LLM-as-Agent 패러다임**이라고 부릅니다.

이 패러다임이 단순한 이론적 구분이 아니라 실질적인 차이를 만든다는 점은 여러 연구에서 확인되었습니다. 운영체제, 데이터베이스, 지식 그래프, 웹 브라우징, 웹 쇼핑, 가정 환경 시뮬레이션 등 8개의 서로 다른 환경에서 29개 LLM을 에이전트로 배치하여 평가한 결과, 텍스트 이해 능력이 높은 모델이라고 해서 반드시 좋은 에이전트가 되는 것은 아니었습니다. 에이전트로서의 능력은 장기적인 추론, 도구 활용, 실행 오류로부터의 복구 등을 포함하며, 이는 단순 텍스트 생성 능력과는 구별되는 독자적인 역량입니다. 특히 상용 모델과 오픈소스 모델 사이에 에이전트 능력의 격차가 텍스트 생성 능력의 격차보다 훨씬 크다는 점도 주목할 만합니다.

그렇다면 LLM 에이전트는 구체적으로 어떤 구성 요소로 이루어져 있을까요. 에이전트 아키텍처 연구를 체계화한 작업에서는 에이전트를 네 가지 핵심 모듈로 분류합니다. 이 네 모듈은 **Planning(계획)**, **Reasoning(추론)**, **Tool Use(도구 사용)**, **Memory(기억)**입니다. 각 모듈은 독립적인 역할을 담당하면서도 서로 밀접하게 연동되며, 이 네 가지의 조합이 에이전트의 전체적인 능력을 결정합니다.

**Planning 모듈**은 복잡한 목표를 달성 가능한 하위 작업들로 분해하는 역할을 합니다. 사용자가 "이 논문들을 요약하고 비교 분석표를 만들어 줘"라고 요청하면, 에이전트는 이를 즉시 실행하지 않습니다. 먼저 각 논문을 읽는 단계, 핵심 내용을 추출하는 단계, 비교 기준을 설정하는 단계, 표를 구성하는 단계로 작업을 나눕니다. 이 분해 과정에서 어떤 단계가 어떤 단계에 의존하는지, 어떤 단계를 병렬로 진행할 수 있는지도 함께 결정합니다. 계획 모듈은 단선형으로 단계를 나열하는 방식부터, 여러 가능한 경로를 탐색하며 최적의 순서를 찾는 방식까지 다양한 구현이 있습니다. 특히 웹 쇼핑처럼 여러 페이지를 순차적으로 탐색해야 하는 장기 작업에서는 계획 모듈의 품질이 전체 성능을 좌우합니다.

**Reasoning 모듈**은 계획 모듈이 분해한 개별 하위 작업을 실제로 해결하는 역할을 합니다. Planning이 "무엇을 어떤 순서로 할 것인가"를 결정한다면, Reasoning은 "각 단계를 어떻게 해결할 것인가"를 담당합니다. 예를 들어 논문의 핵심 논지를 파악하는 하위 작업이 주어지면, 추론 모듈은 논문의 초록과 결론을 먼저 읽고, 핵심 주장을 식별하고, 그 근거를 정리하는 사고 과정을 거칩니다. 이때 사용되는 대표적인 기법이 Chain-of-Thought(CoT)입니다. CoT는 최종 답변에 도달하기 전에 중간 추론 단계를 명시적으로 전개하는 방식으로, 복잡한 문제를 작은 논리적 단계들로 쪼개어 순차적으로 풀어나갑니다. 추론 모듈의 고도화된 형태로는, 여러 개의 독립적인 추론 경로를 병렬로 실행한 뒤 전문가 에이전트들이 서로의 결과를 비평하고, 반복적으로 정제하여 앙상블 결론을 도출하는 방식도 있습니다.

**Tool Use 모듈**은 에이전트가 외부 세계와 상호작용하는 인터페이스입니다. LLM 자체는 학습이 완료된 시점 이후의 정보를 알 수 없고, 수학 계산을 정밀하게 수행하거나 파일을 조작하는 것이 불가능합니다. 도구 사용 모듈은 이러한 한계를 극복합니다. 에이전트가 사용할 수 있는 도구에는 웹 검색 API, Python 코드 실행 환경, 데이터베이스 쿼리 인터페이스, 수학 엔진, 외부 서비스 API 등이 포함됩니다. 에이전트는 현재 작업에 어떤 도구가 필요한지 판단하고, 적절한 입력값을 구성하여 도구를 호출하며, 반환된 결과를 해석하여 다음 단계에 활용합니다. 도구 사용 능력을 평가하는 벤치마크에서는 이 모듈의 최적화가 성능을 결정적으로 좌우한다는 결과가 반복적으로 확인됩니다.

**Memory 모듈**은 에이전트가 과거 경험을 보존하고 필요할 때 인출하는 기능을 담당합니다. 기억 없이는 에이전트가 매번 처음부터 시작해야 하며, 긴 작업 도중에 앞서 수집한 정보를 활용할 수 없습니다. 기억은 크게 두 층위로 나뉩니다. 단기 기억은 현재 작업 세션 내에서 에이전트가 수행한 행동과 관찰의 이력을 LLM의 컨텍스트 윈도우 안에 유지하는 것입니다. 장기 기억은 여러 세션에 걸친 대화 이력과 경험을 외부 저장소에 보관하고, 의미 검색을 통해 관련 정보를 필요할 때 꺼내 오는 것입니다. 환경에 대한 지식을 축적해야 하는 작업, 예를 들어 가정 환경 시뮬레이션에서 물건의 위치를 기억해야 하는 경우에는 기억 모듈이 성능의 핵심 결정 요인이 됩니다.

이 네 모듈이 실제로 함께 작동하는 방식이 **환경과의 상호작용 루프**입니다. 이 루프는 세 단계의 반복으로 구성됩니다. 첫 번째는 **Thought(사고)** 단계입니다. 에이전트의 LLM이 현재까지의 관찰과 기억을 바탕으로 상황을 평가하고, 다음에 무엇을 해야 할지 결정합니다. 이 과정은 에이전트 내부의 추론이며, 계획 모듈과 추론 모듈이 협력하여 수행합니다. 두 번째는 **Action(행동)** 단계입니다. 사고 단계에서 결정한 행동을 실제로 실행합니다. 도구를 호출하거나, 환경에 명령을 보내거나, 사용자에게 응답을 전달합니다. 세 번째는 **Observation(관찰)** 단계입니다. 행동의 결과가 에이전트에게 돌아옵니다. 도구가 반환한 데이터, 환경의 상태 변화, 오류 메시지 등이 여기에 해당합니다. 이 관찰값은 다음 사고 단계의 입력이 되며, 루프가 다시 시작됩니다. 이 Thought-Action-Observation 사이클은 에이전트가 목표를 달성했다고 판단하거나, 최대 반복 횟수에 도달하거나, 외부에서 중단 신호가 올 때까지 계속됩니다. 프로그래밍의 while 루프와 유사한 구조입니다.

구체적인 예를 들어 보겠습니다. 사용자가 "뉴욕의 현재 날씨를 알려 줘"라고 요청하면, 에이전트의 사고 단계에서는 "날씨 정보를 얻으려면 날씨 API 도구를 호출해야 한다"는 판단이 이루어집니다. 행동 단계에서 에이전트는 날씨 API를 호출하며 위치 매개변수로 "New York"을 전달합니다. 관찰 단계에서 API가 "부분적으로 흐림, 15도, 습도 60%"라는 결과를 반환합니다. 다시 사고 단계로 돌아온 에이전트는 필요한 정보를 획득했다고 판단하고, 최종 행동으로 사용자에게 정리된 답변을 전달합니다. 만약 API 호출이 실패했다면, 에이전트는 다른 도구를 시도하거나 접근 방식을 수정하여 루프를 계속 진행합니다. 이것이 단일 응답 챗봇과 에이전트의 근본적인 차이입니다.

지금까지 설명한 개별 기법들, 즉 Chain-of-Thought 추론, Self-Reflection을 통한 자기 점검, 도구 호출, 역할 부여 등은 각각 독립적으로 발견되고 발전해 온 기술입니다. 이것들을 **에이전트 빌딩 블록**이라고 부릅니다. 빌딩 블록은 에이전트를 구성하는 원자적인 설계 기법으로, 레고 블록처럼 조합하여 다양한 에이전트 아키텍처를 만들어 낼 수 있습니다. 예를 들어, 5개의 CoT 추론 경로를 병렬로 실행하고, 각 경로에 전문가 에이전트가 비평을 가한 뒤, 앙상블로 최종 결론을 도출하는 에이전트는 CoT, 병렬 실행, 전문가 비평, 앙상블이라는 여러 빌딩 블록의 조합입니다. 문제를 하위 문제로 분할하여 전문가별로 해결하는 에이전트는 분할 정복 빌딩 블록과 역할 할당 빌딩 블록의 조합입니다.

빌딩 블록 개념이 중요한 이유는, 가능한 조합의 수가 매우 크다는 데 있습니다. 에이전트의 네 모듈 각각에 다양한 빌딩 블록 선택지가 있고, 이들의 조합은 폭발적으로 증가합니다. 한 연구에서는 16개의 기존 에이전트에서 모듈을 추출하여 표준화된 입출력 인터페이스를 정의한 결과, 1,050가지의 서로 다른 에이전트 조합이 가능했습니다. 이 조합 공간을 사람이 수동으로 탐색하는 것은 비효율적이며, 이것이 에이전트 설계 자동화 연구의 출발점이 됩니다. 에이전트를 코드로 정의하면 탐색 공간이 이론적으로 무한해지며, 메타 에이전트가 반복적으로 새로운 에이전트 코드를 작성하고 평가하는 방식으로 자동 탐색이 가능해집니다. 이 주제는 1.3절에서 더 자세히 다룹니다.

빌딩 블록 가운데 특별한 위치를 차지하는 것이 **코드 실행 통합**입니다. LLM은 텍스트를 생성할 수 있지만, 수치 계산을 정밀하게 수행하거나, 데이터 파일을 읽어 분석하거나, 외부 시스템에 명령을 보내는 것은 텍스트 생성만으로는 불가능합니다. 코드 실행 통합은 에이전트가 Python 같은 프로그래밍 언어로 코드를 생성하고, 이를 별도의 실행 환경(또는 샌드박스)에서 실행한 뒤, 결과를 다시 에이전트의 대화 흐름에 주입하는 구조입니다. 이를 통해 수학적 계산의 정확성을 보장하고, 실시간 데이터를 처리하며, 재현 가능한 결과를 생성할 수 있습니다. 코드 실행은 도구 사용 모듈의 한 형태이지만, 그 범용성 때문에 독립적인 빌딩 블록으로 취급됩니다. 에이전트가 코드를 실행하고 오류가 발생하면, 오류 메시지를 관찰로 받아들여 코드를 수정하고 다시 실행하는 것 자체가 Thought-Action-Observation 루프의 한 사례입니다. 코드 실행 환경에서는 상용 모델과 오픈소스 모델 사이의 성능 격차가 다른 에이전트 능력에 비해 상대적으로 작다는 점도 주목할 만합니다. 코드 생성과 실행은 오픈소스 모델들이 비교적 잘 따라잡은 영역입니다.

에이전트가 자율적으로 행동할 수 있다는 것은 강력한 능력이지만, 동시에 위험을 수반합니다. 에이전트가 잘못된 판단으로 되돌릴 수 없는 작업을 실행하거나, 모호한 상황에서 부적절한 행동을 선택할 수 있습니다. 이 문제를 해결하기 위한 아키텍처 패턴이 **Human-in-the-Loop(HITL)** 구조입니다. HITL는 에이전트의 의사결정 흐름에 사람의 피드백과 승인을 통합하는 설계입니다. 핵심 원칙은 "제안 후 확정(propose-commit)"입니다. 에이전트가 고위험 행동을 직접 실행하는 대신, 먼저 사람에게 제안하고 사람이 확정한 후에만 실행하는 구조입니다.

HITL에서 사람이 취할 수 있는 결정은 세 가지입니다. 승인(Approve)은 에이전트가 제안한 행동을 그대로 실행하는 것이고, 수정(Edit)은 행동의 매개변수나 내용을 수정한 뒤 실행하는 것이며, 거부(Reject)는 행동을 취소하고 에이전트에 피드백을 제공하는 것입니다. 시스템이 자동으로 사람의 개입을 요청하는 조건으로는, 에이전트의 확신 점수가 임계값 아래일 때, 금융 거래나 파일 삭제 같은 고위험 행동일 때, 사용자의 의도가 여러 가지로 해석될 수 있을 때, 워크플로우의 특정 체크포인트에 도달했을 때 등이 있습니다. 더 정교한 형태로는 Return of Control(ROC) 패턴이 있습니다. 이것은 에이전트가 작업을 일시 중단하고 제어권을 사람에게 반환한 뒤, 사람이 추가 맥락이나 수정 사항을 제공하면 다시 에이전트가 이어서 작업하는 구조입니다. 현재 주요 에이전트 프레임워크들은 대화 흐름에 사람을 참가자로 포함시키거나, 그래프 기반 워크플로우에서 특정 노드 실행 전에 사람의 검토를 강제하는 방식으로 HITL를 지원합니다.

정리하면, LLM 에이전트는 대규모 언어 모델을 단순 텍스트 생성기가 아닌 자율적 의사결정 엔진으로 활용하는 패러다임입니다. 에이전트는 Planning, Reasoning, Tool Use, Memory 네 가지 모듈로 구성되며, Thought-Action-Observation 루프를 통해 환경과 반복적으로 상호작용합니다. 개별 설계 기법인 빌딩 블록들을 조합하여 다양한 에이전트 아키텍처를 구성할 수 있고, 코드 실행 통합은 LLM의 정적 한계를 극복하는 핵심 메커니즘입니다. Human-in-the-Loop 구조는 에이전트의 자율성과 안전성 사이의 균형을 잡아 줍니다.

다음 단원인 1.1.2에서는 이 상호작용 루프의 가장 대표적인 구현인 ReAct 패턴을 자세히 살펴보고, Self-Reflection과 Self-Refine이 에이전트 루프를 어떻게 확장하는지 다룹니다.

이 단원을 마치면 LLM 에이전트가 단순 텍스트 생성기와 어떻게 다른지 설명하고, Planning, Reasoning, Tool Use, Memory 네 가지 핵심 모듈의 역할과 상호작용을 구별할 수 있습니다.
