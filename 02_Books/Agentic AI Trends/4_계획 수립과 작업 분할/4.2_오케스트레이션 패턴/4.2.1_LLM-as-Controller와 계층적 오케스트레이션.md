# 4.2.1 LLM-as-Controller와 계층적 오케스트레이션

에이전트 시스템이 다수의 서브태스크와 다수의 에이전트를 조율해야 할 때, 그 중심에서 전체 흐름을 지휘하는 존재가 필요합니다. 전통적인 소프트웨어에서는 이 역할을 사전에 작성된 로직이 담당했습니다. 에이전트 시스템에서는 대형 언어 모델(LLM) 자체가 이 컨트롤러 역할을 수행할 수 있습니다. LLM-as-Controller는 LLM이 단순히 텍스트를 생성하는 것을 넘어, 어떤 에이전트에게 어떤 태스크를 언제 할당할지를 스스로 결정하는 역할을 맡는 패턴입니다.

메타 에이전트 패턴(meta-agent pattern)은 이 아이디어의 구체적 구현입니다. 메타 에이전트는 다른 에이전트들의 능력을 알고 있고, 목표를 받아 그것을 서브태스크로 분해한 뒤, 각 서브태스크를 처리할 에이전트를 선택하고, 실행 결과를 모아 전체 목표 달성 여부를 판단합니다. 메타 에이전트 자체는 직접 도구를 실행하거나 검색을 수행하지 않습니다. 메타 에이전트의 작업은 조율과 의사결정이며, 실제 실행은 워커 에이전트들이 담당합니다.

LLM-as-Controller의 핵심 강점은 유연성입니다. 고정된 규칙으로 표현하기 어려운 복잡한 조율 논리를 LLM의 추론 능력으로 처리할 수 있습니다. 예를 들어 특정 에이전트가 실패했을 때 어떤 대안적 에이전트를 선택할지, 중간 결과가 예상과 다를 때 계획을 어떻게 수정할지 같은 결정을 LLM이 맥락을 고려하여 내릴 수 있습니다. 또한 새로운 유형의 태스크가 등장하더라도 LLM이 이해하고 대응할 수 있는 경우, 컨트롤러 코드를 수정할 필요 없이 시스템이 작동합니다.

Planner-Coordinator-Worker 3계층 구조는 계층적 오케스트레이션의 대표적인 설계입니다. 최상위의 Planner는 고수준의 목표를 받아 전체 계획의 골격을 수립합니다. Planner는 큰 그림을 보고, 주요 단계와 그 순서를 정의합니다. 중간 계층의 Coordinator는 Planner가 정의한 단계를 실제로 실행 가능한 서브태스크들로 세분화하고, 이를 Worker에게 할당하며 진행 상황을 모니터링합니다. 최하위의 Worker는 개별 태스크를 실행하며, 구체적인 도구 호출, 검색, 코드 실행 같은 작업을 수행합니다.

이 3계층 구조는 관심사 분리의 원칙을 따릅니다. Planner는 '무엇을 달성할 것인가'를 결정하고, Coordinator는 '어떻게 분배할 것인가'를 결정하며, Worker는 '어떻게 실행할 것인가'를 담당합니다. 각 계층이 자신의 역할에 집중하면, 전체 시스템의 복잡성이 관리 가능한 수준으로 분해됩니다. 또한 각 계층을 독립적으로 개선하거나 교체할 수 있는 유연성이 생깁니다.

공유 태스크 채널(shared task channel) 통신은 에이전트들이 서로 직접 메시지를 주고받는 대신, 중앙의 공유 채널에 태스크와 결과를 게시하고 구독하는 방식입니다. 이 패턴은 에이전트들 사이의 직접적인 결합을 없애줍니다. 어떤 에이전트가 어떤 에이전트와 통신하는지를 오케스트레이터가 중앙에서 제어할 수 있으며, 에이전트가 추가되거나 제거되어도 기존 에이전트들의 코드를 수정할 필요가 없습니다. 채널은 또한 자연스러운 로깅 지점이 되어, 모든 태스크와 결과가 기록되고 추적됩니다.

플러그앤플레이(plug-and-play) Worker 교체는 공유 태스크 채널 방식이 가능하게 하는 중요한 이점입니다. 태스크 유형에 대응하는 인터페이스를 만족하는 에이전트라면 어떤 것이든 Worker로 교체할 수 있습니다. 더 강력한 모델로 업그레이드하거나, 특정 도메인에 특화된 모델로 교체하거나, 비용 절감을 위해 더 저렴한 모델을 사용하는 결정을 시스템 전체 구조를 바꾸지 않고 수행할 수 있습니다. 이는 에이전트 시스템의 유지보수성과 확장성을 크게 높입니다.

언어를 범용 인터페이스로 사용하는 것은 LLM-as-Controller 패턴의 또 다른 핵심 특성입니다. 에이전트들이 자연어나 구조화된 텍스트 형식으로 태스크를 받고 결과를 전달하면, 인터페이스를 별도로 설계할 필요 없이 LLM이 처리할 수 있는 어떤 형태의 태스크든 시스템에 통합할 수 있습니다. 이는 특히 이질적인 에이전트들, 즉 코드 에이전트, 검색 에이전트, 문서 작성 에이전트 등을 하나의 시스템 안에서 함께 사용할 때 강력합니다. 공통 인터페이스를 언어로 삼으면 새로운 유형의 에이전트를 추가하는 데 드는 통합 비용이 최소화됩니다.

실행 컨텍스트 격리(execution context isolation)는 각 에이전트가 자신에게 할당된 태스크를 처리할 때 서로의 실행 환경에 간섭하지 않도록 하는 설계 원칙입니다. 각 에이전트는 자신의 컨텍스트 윈도우, 도구 접근 권한, 임시 파일 공간 등을 독립적으로 유지합니다. 격리를 통해 한 에이전트의 오류가 다른 에이전트에게 전파되는 것을 막고, 동시에 여러 에이전트가 병렬로 실행될 때 예측 불가능한 상호작용을 방지합니다. 공유해야 하는 리소스는 명시적인 채널을 통해서만 접근하도록 제한합니다.

모델 카드 기반 모델 선택(model card based model selection)은 각 LLM의 강점, 한계, 비용, 지연 시간 등의 정보를 담은 모델 카드를 참조하여 태스크에 가장 적합한 모델을 자동으로 선택하는 방식입니다. 수학적 추론이 필요한 태스크에는 그 능력이 뛰어난 모델을, 자연어 생성이 주인 태스크에는 그에 적합한 모델을, 비용이 중요한 단순 태스크에는 더 저렴한 모델을 선택할 수 있습니다. 이 방식은 단순히 가장 강력한 모델을 모든 태스크에 사용하는 것보다 효율적이며, 태스크의 성격과 모델의 특성을 매칭하여 전체 시스템의 성능과 비용 효율을 동시에 최적화합니다.

계층적 오케스트레이션이 제대로 작동하려면 각 계층 간의 통신 프로토콜이 명확해야 합니다. Planner가 Coordinator에게 전달하는 계획의 형식, Coordinator가 Worker에게 할당하는 태스크의 명세, Worker가 Coordinator에게 반환하는 결과의 구조 모두 사전에 정의되어야 합니다. 이 프로토콜이 느슨하면 계층 간 오해가 발생하고, LLM이 각자 다른 방식으로 해석하여 일관성 없는 동작이 나타납니다. 구조화된 출력 형식(JSON 스키마 등)을 사용하면 이런 모호성을 크게 줄일 수 있습니다.

정리하면, LLM-as-Controller 패턴은 LLM의 추론 능력을 오케스트레이션에 활용하여 복잡하고 유연한 다중 에이전트 조율을 가능하게 합니다. Planner-Coordinator-Worker 3계층은 관심사를 분리하고 각 계층이 자신의 역할에 집중하게 합니다. 공유 태스크 채널은 에이전트 간 결합을 약화시키고 플러그앤플레이 교체를 가능하게 하며, 언어를 범용 인터페이스로 삼는 방식은 이질적인 에이전트들을 하나의 시스템으로 통합하는 비용을 낮춥니다. 실행 컨텍스트 격리와 모델 카드 기반 선택은 시스템의 신뢰성과 효율성을 높이는 실용적 기법입니다.

다음 단원(4.2.2)에서는 계층적 오케스트레이션의 변형으로서 더 단순한 2계층 구조와, 정보를 순차적으로 축적하며 처리하는 체인 오케스트레이션 패턴을 살펴봅니다.

이 단원을 학습한 후에는 LLM-as-Controller 패턴의 작동 원리를 설명하고, Planner-Coordinator-Worker 3계층 구조를 설계하며, 공유 태스크 채널과 실행 컨텍스트 격리의 역할을 구체적으로 이해할 수 있습니다.
