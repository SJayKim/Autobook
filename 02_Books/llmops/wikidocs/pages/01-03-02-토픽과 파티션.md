# 1.3.2 토픽과 파티션

1.3.1에서 Kafka의 전체 구조를 살펴보았습니다. 프로듀서가 메시지를 보내고, 브로커가 저장하며, 컨슈머가 읽는 세 계층 구조였습니다. 그 과정에서 프로듀서가 메시지를 보낼 때 '토픽'을 지정한다는 점도 간단히 언급했습니다. 그런데 실무에서는 하나의 토픽에 초당 수만 건의 메시지가 쏟아지기도 합니다. 브로커 한 대가 이 모든 메시지를 혼자 저장하고 전달해야 한다면, 그 브로커가 병목이 됩니다. 컨슈머도 한 대뿐이라면 처리 속도에 한계가 생깁니다. 이 문제를 해결하는 핵심 구조가 바로 파티션입니다.

먼저 토픽이 무엇인지 정확히 짚고 넘어갑니다. **토픽**(Topic)은 메시지를 분류하는 논리적 채널입니다. Kafka 클러스터 안에서 고유한 이름으로 식별됩니다. 프로듀서는 특정 토픽에 메시지를 발행하고, 컨슈머는 관심 있는 토픽을 구독합니다. 예를 들어 전자상거래 시스템에서 'order' 토픽과 'payment' 토픽을 분리하면, 주문 메시지와 결제 메시지가 섞이지 않습니다. 토픽은 메시지의 종류를 구분하는 이름표와 같습니다.

여기서 '논리적'이라는 표현에 주목합니다. 토픽 자체는 물리적인 저장 단위가 아닙니다. 토픽은 개념적인 분류 체계이고, 실제 데이터가 디스크에 기록되는 단위는 **파티션**(Partition)입니다. 하나의 토픽은 하나 이상의 파티션으로 나뉩니다. 각 파티션은 독립된 커밋 로그입니다. 1.3.1에서 다룬 것처럼 커밋 로그란 새 데이터를 끝에만 덧붙이고, 이미 기록된 데이터는 수정하거나 삭제하지 않는 저장 구조입니다. 레코드가 파티션에 추가되면 Kafka는 해당 레코드에 파티션 안에서 고유한 순차 번호인 오프셋을 부여합니다.

토픽과 파티션의 관계를 다이어그램으로 나타내면 다음과 같습니다.

```
  토픽 "order"
  ┌─────────────────────────────────────────┐
  │                                         │
  │  파티션 0: [msg0, msg1, msg2, msg3, ...] │  -> 브로커 1
  │  파티션 1: [msg0, msg1, msg2, ...]       │  -> 브로커 2
  │  파티션 2: [msg0, msg1, ...]             │  -> 브로커 3
  │                                         │
  └─────────────────────────────────────────┘
```

'order'라는 하나의 토픽이 세 개의 파티션으로 나뉘어 있습니다. 각 파티션은 서로 다른 브로커에 배치됩니다. 파티션마다 오프셋이 0부터 독립적으로 증가합니다. 이것이 토픽의 '논리적 채널'과 파티션의 '물리적 분할' 사이의 관계입니다.

파티션이 여러 개 존재하면 왜 좋은지를 이해하려면 **병렬 처리**를 알아야 합니다. 파티션이 하나뿐이라면 그 파티션을 읽는 컨슈머도 하나뿐입니다. 아무리 컨슈머를 추가해도 하나의 파티션은 같은 컨슈머 그룹 안에서 동시에 하나의 컨슈머만 읽을 수 있기 때문입니다. 파티션이 네 개라면 컨슈머 네 대가 각각 하나씩 맡아 동시에 읽습니다.

구체적인 시나리오를 봅니다. 'order' 토픽에 초당 40,000건의 메시지가 들어온다고 가정합니다. 컨슈머 한 대가 초당 10,000건을 처리할 수 있다면, 파티션이 하나일 때 컨슈머 한 대가 모든 부하를 감당해야 합니다. 30,000건은 밀려서 지연이 발생합니다. 파티션을 네 개로 늘리면 컨슈머 네 대가 각각 초당 10,000건씩 나누어 처리합니다. 전체 처리량이 네 배로 늘어나는 셈입니다.

```
  토픽 "order" (파티션 4개)        컨슈머 그룹

  파티션 0  ──(10,000건/초)──>  컨슈머 A
  파티션 1  ──(10,000건/초)──>  컨슈머 B
  파티션 2  ──(10,000건/초)──>  컨슈머 C
  파티션 3  ──(10,000건/초)──>  컨슈머 D

  총 처리량: 40,000건/초
```

여기서 주의할 점이 있습니다. 컨슈머 수가 파티션 수를 초과하면 남는 컨슈머는 할당받을 파티션이 없어 대기 상태가 됩니다. 파티션이 네 개인데 컨슈머를 여섯 대로 늘려도 실제 일하는 컨슈머는 네 대뿐입니다. 따라서 병렬 처리의 상한은 파티션 수가 결정합니다.

병렬 처리를 넘어 **수평 확장**(horizontal scaling)으로 시야를 넓힙니다. 수평 확장이란 서버 한 대의 성능을 올리는 대신, 서버 수를 늘려 전체 처리 능력을 키우는 방식입니다. Kafka에서 수평 확장의 단위가 바로 파티션입니다.

처리량이 부족하면 파티션 수를 늘립니다. 파티션이 늘어나면 그만큼 컨슈머를 추가 투입할 수 있습니다. 파티션은 클러스터 내 여러 브로커에 분산 배치되므로, 브로커를 추가하면 파티션이 새 브로커에도 할당됩니다. 이 과정을 단계별로 정리합니다.

```
  수평 확장 과정

  1단계: 현재 상태
    브로커 2대, 파티션 4개 -> 컨슈머 4대

  2단계: 부하 증가, 브로커 1대 추가
    브로커 3대, 파티션 6개로 확장 -> 컨슈머 6대로 확장

  3단계: 결과
    파티션이 3대의 브로커에 균등 분산
    전체 처리량이 기존의 1.5배로 증가
```

실제 규모를 가늠하면 도움이 됩니다. 한 사례에서는 50,000개의 Kafka 토픽으로 하루 700억 건의 이벤트를 처리합니다. 파티션 기반 분산 구조가 아니라면 이 규모의 처리는 불가능합니다. 다만 파티션 수를 늘리기는 쉽지만 줄이기는 어렵습니다. Kafka는 파티션 축소를 기본적으로 지원하지 않습니다. 따라서 초기 설계 단계에서 예상 부하를 고려하여 파티션 수를 정하는 것이 중요합니다.

이제 프로듀서가 메시지를 보낼 때, 어떤 파티션에 들어갈지를 결정하는 방법을 살펴봅니다. 이 역할을 하는 것이 **파티션 키**(Partition Key)입니다.

프로듀서가 메시지에 키를 지정하면 Kafka는 그 키의 해시값을 계산합니다. 해시값을 파티션 수로 나눈 나머지가 대상 파티션 번호가 됩니다. 예를 들어 파티션이 네 개이고 키 'user-42'의 해시값이 130이면, 130을 4로 나눈 나머지인 2가 대상 파티션 번호입니다. 'user-42' 키를 가진 메시지는 항상 파티션 2로 갑니다.

```
  키 "user-42" -> hash("user-42") = 130
  130 mod 4 = 2
  -> 파티션 2로 라우팅

  키 "user-77" -> hash("user-77") = 311
  311 mod 4 = 3
  -> 파티션 3으로 라우팅
```

같은 키를 가진 메시지는 항상 같은 파티션에 도착합니다. 이 성질은 뒤에서 다룰 순서 보장에 핵심적인 역할을 합니다.

키를 지정하지 않으면 어떻게 될까요. Kafka 2.4 이전에는 라운드 로빈 방식으로 파티션을 번갈아 선택했습니다. Kafka 2.4부터는 **스티키 파티셔닝**(sticky partitioning)이 기본입니다. 스티키 파티셔닝은 같은 배치에 속하는 메시지를 하나의 파티션에 모아 보냅니다. 배치가 채워지거나 전송 시점이 되면 다음 파티션으로 넘어갑니다. 이렇게 하면 네트워크 요청 횟수가 줄어들어 처리 효율이 높아집니다.

마지막으로, 파티션 구조에서 가장 자주 혼동되는 주제인 **메시지 순서 보장**을 다룹니다.

Kafka는 토픽 전체의 메시지 순서를 보장하지 않습니다. 순서가 보장되는 범위는 파티션 단위입니다. 같은 파티션 안에서 메시지는 오프셋 오름차순으로 엄격하게 정렬되어 있습니다. 컨슈머는 항상 이 순서대로 메시지를 읽습니다. 그러나 서로 다른 파티션 사이에는 순서 관계가 없습니다.

왜 이것이 중요한지 예를 들어 봅니다. 전자상거래에서 한 사용자의 '주문 생성', '결제 완료', '배송 시작' 이벤트가 순서대로 처리되어야 한다고 가정합니다. 이 세 이벤트가 서로 다른 파티션에 흩어지면 컨슈머가 '배송 시작'을 '주문 생성'보다 먼저 읽을 수 있습니다. 이를 방지하려면 세 이벤트에 같은 파티션 키, 예를 들어 사용자 ID를 지정합니다. 같은 키는 같은 파티션으로 가므로, 해당 파티션 안에서 세 이벤트의 순서가 보장됩니다.

```
  사용자 ID "user-42"의 이벤트 흐름

  프로듀서 -> 키 "user-42" -> 파티션 2

  파티션 2 내부:
    오프셋 0: 주문 생성
    오프셋 1: 결제 완료
    오프셋 2: 배송 시작

  컨슈머는 오프셋 0 -> 1 -> 2 순서로 읽음
```

다만 키 기반 파티셔닝에는 주의할 점이 있습니다. 특정 키에 트래픽이 집중되면 해당 파티션만 과부하가 걸립니다. 이를 핫 스팟(hot spot)이라 합니다. 예를 들어 소수의 대형 고객이 전체 이벤트의 대부분을 차지하면, 해당 고객 ID가 할당된 파티션에 부하가 몰립니다. 이런 경우에는 1단계에서 키 없이 무작위 파티셔닝으로 부분 집계를 수행하고, 2단계에서 키 기반으로 최종 집계하는 방식을 사용하기도 합니다.

정리하면, Kafka 토픽은 메시지를 분류하는 논리적 채널이고, 파티션은 토픽을 물리적으로 나눈 저장 단위입니다. 파티션 덕분에 여러 컨슈머가 동시에 데이터를 읽어 병렬 처리가 가능하고, 파티션과 브로커를 추가하여 수평 확장을 할 수 있습니다. 프로듀서가 파티션 키를 지정하면 같은 키의 메시지가 같은 파티션에 모여 순서가 보장됩니다. 순서 보장의 범위는 토픽 전체가 아니라 파티션 단위라는 점이 핵심입니다.

다음 단원인 1.3.3에서는 오프셋과 메시지 소비를 다룹니다. 컨슈머가 오프셋을 어떻게 추적하고 커밋하는지, 장애 복구 시 어디서부터 다시 읽는지를 살펴봅니다.

이 단원을 마치면 Kafka 토픽과 파티션의 관계, 파티션이 병렬 처리와 수평 확장에 기여하는 방식을 설명할 수 있습니다.
