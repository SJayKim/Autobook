# 1.1.2 MLOps 성숙도 모델

1.1.1에서 MLOps가 머신러닝 모델의 생명주기 전체를 체계적으로 관리하는 원칙이라고 설명했습니다. 데이터 추출부터 모델 서빙, 모니터링, 재훈련까지 순환하는 흐름을 자동화해야 한다는 점도 살펴보았습니다. 그렇다면 자연스러운 질문이 생깁니다. 조직이 MLOps를 도입할 때 처음부터 모든 단계를 자동화할 수 있을까요? 현실적으로 그렇지 않습니다. 대부분의 조직은 모든 것이 수동인 상태에서 출발하여, 단계적으로 자동화 범위를 넓혀 갑니다. 이 단계적 진화를 체계적으로 정리한 것이 **MLOps 성숙도 모델**입니다.

성숙도 모델은 조직의 현재 MLOps 수준을 진단하고, 다음에 무엇을 개선해야 하는지 방향을 잡는 데 쓰입니다. 대표적인 성숙도 모델로는 Google Cloud가 제안한 3단계 모델(Level 0, 1, 2)과 Microsoft가 제안한 5단계 모델(Level 0부터 4까지)이 있습니다. 두 모델은 세분화 정도가 다르지만, '수동에서 자동으로' 나아가는 큰 흐름은 같습니다. 이 단원에서는 Google의 3단계 모델을 축으로 각 단계를 설명한 뒤, Microsoft의 5단계 모델이 어떻게 더 세밀하게 나누는지 살펴봅니다.

먼저 가장 초기 단계인 **Level 0 수동 프로세스**입니다. Level 0은 데이터 수집부터 모델 배포까지 모든 단계를 사람이 직접 수행하는 상태입니다. 데이터 과학자가 Jupyter Notebook에서 대화형으로 코드를 작성하고, 실험 결과를 눈으로 확인하며, 완성된 모델 파일을 수작업으로 엔지니어에게 전달합니다.

이 단계에서 가장 큰 문제는 세 가지입니다. 첫째, 실험 추적이 없습니다. 어떤 데이터로 어떤 하이퍼파라미터를 써서 훈련했는지 기록하지 않으므로 동일한 모델을 다시 만들 수 없습니다. 1.1.1에서 다룬 '재현 불가능성' 문제가 그대로 드러납니다. 둘째, 팀 간 단절이 심합니다. 데이터 과학자, 데이터 엔지니어, 소프트웨어 엔지니어가 고립된 상태로 작업하기 때문에, 훈련 환경에서 사용한 피처 계산 방식과 서빙 환경의 방식이 달라지는 **훈련-서빙 스큐**(training-serving skew)가 빈번하게 발생합니다. 이 역시 1.1.1에서 설명한 과제입니다. 셋째, 모니터링 인프라가 전혀 없습니다. 모델이 배포된 뒤 성능이 저하되더라도 감지할 방법이 없습니다.

결과적으로 Level 0 조직에서는 모델 배포 빈도가 연 1~2회에 그칩니다. 스코어링 스크립트조차 버전 관리되지 않는 경우가 많습니다. 이 상태는 실험 목적으로는 가능하지만, 빈번한 재훈련과 지속적 실험이 필요한 프로덕션 환경에는 부적합합니다.

다음은 **Level 1 ML 파이프라인 자동화**입니다. Level 0에서 Level 1로 올라가는 핵심 전환은 모델 훈련 과정을 자동화하는 것입니다. Level 0에서는 데이터 과학자가 노트북에서 수동으로 훈련을 실행했지만, Level 1에서는 파이프라인이 이 과정을 대신합니다.

파이프라인이란 데이터 추출, 전처리, 훈련, 평가 등 각 단계를 컨테이너화된 구성 요소로 분리하여 자동으로 실행하는 체계입니다. 각 구성 요소가 독립적이므로 재사용할 수 있고, 문제가 생긴 단계만 다시 실행할 수도 있습니다. 이 자동화 덕분에 **지속적 훈련**(CT: Continuous Training)이 가능해집니다. 지속적 훈련이란 새로운 데이터가 들어오거나 모델 성능이 떨어질 때 자동으로 재훈련을 수행하는 것을 뜻합니다.

Level 1에서 새롭게 도입되는 핵심 요소는 세 가지입니다.

첫 번째는 **데이터 및 모델 검증**입니다. 파이프라인이 자동으로 실행되려면, 입력 데이터와 출력 모델이 기대에 부합하는지 자동으로 확인하는 장치가 필요합니다. 스키마 편향 검사는 기대하는 특성(피처)이 빠지지 않았는지 확인합니다. 값 편향 검사는 데이터의 통계 특성이 유의미하게 변했는지 감지합니다. 모델 평가 지표를 이전 버전과 자동 비교하는 것도 이 단계에 포함됩니다.

두 번째는 **특성 저장소**(Feature Store)입니다. 특성 저장소는 피처의 정의, 저장, 접근을 한곳에서 관리하는 중앙 저장소입니다. 훈련할 때 사용하는 피처 계산 로직과 서빙할 때 사용하는 로직이 같은 저장소에서 나오므로, 1.1.1에서 설명한 훈련-서빙 스큐 문제를 구조적으로 방지합니다.

세 번째는 **메타데이터 관리**입니다. 파이프라인 버전, 실행 시간, 하이퍼파라미터, 생성된 아티팩트, 평가 지표를 모두 기록합니다. Level 0에서 불가능했던 실험 재현이 이 메타데이터 덕분에 가능해집니다.

파이프라인이 자동으로 실행되려면 어떤 조건에서 실행할지 정해야 합니다. Level 1에서는 트리거가 다양해집니다. 관리자가 직접 실행하는 온디맨드 방식, 일별이나 주별 같은 스케줄 기반 방식, 새 데이터가 들어왔을 때 실행하는 방식, 모니터링에서 성능 저하나 데이터 분포 변화를 감지했을 때 실행하는 방식 등이 있습니다.

Level 1의 전체 흐름을 정리하면 다음과 같습니다.

```
트리거 발생 (스케줄/새 데이터/성능 저하)
     │
     ▼
데이터 추출 → 데이터 검증 → 데이터 준비 → 모델 훈련
                                              │
                                              ▼
                                         모델 평가 → 모델 검증
                                                        │
                                                        ▼
                                                   모델 레지스트리 등록
                                                        │
                                                        ▼
                                                    모델 서빙
                                                        │
                                                        ▼
                                                    모니터링
                                                     │    │
                          메타데이터 저장소 ◄──────────┘    │
                          특성 저장소 ◄──(훈련/서빙 공유)   │
                                                          │
                               (성능 저하 감지) ◄──────────┘
                                       │
                                       ▼
                                 트리거 발생 (순환)
```

다만 Level 1에는 한계가 있습니다. 파이프라인 자체의 코드를 수정하거나 새 파이프라인을 배포하는 일은 여전히 수동입니다. 파이프라인 하나를 운영할 때는 문제가 적지만, 여러 개의 파이프라인을 관리해야 할 때 수동 테스트와 배포가 병목이 됩니다.

이 한계를 해결하는 것이 **Level 2 CI/CD 파이프라인 자동화**입니다. Level 1이 '모델 훈련의 자동화'였다면, Level 2는 '파이프라인 자체의 자동화'입니다. 소프트웨어 개발에서 코드를 빌드하고 테스트하고 배포하는 CI/CD(지속적 통합/지속적 배포)를 ML 파이프라인에도 적용합니다.

CI는 Continuous Integration, 지속적 통합입니다. 데이터 과학자가 파이프라인 코드를 수정하면, 자동으로 소스 코드를 빌드하고 테스트를 실행합니다. ML 파이프라인의 CI에서 수행하는 테스트는 일반 소프트웨어와 다릅니다. 피처 엔지니어링 로직의 단위 테스트, 모델 훈련이 정상적으로 수렴하는지 검사하는 테스트, NaN 값이 발생하지 않는지 검사하는 테스트, 파이프라인 구성 요소 간 데이터 흐름이 올바른지 확인하는 통합 테스트가 포함됩니다. 테스트를 통과하면 패키지와 실행 파일이 생성됩니다.

CD는 Continuous Delivery, 지속적 배포입니다. CI에서 만들어진 아티팩트를 실제 환경에 배포합니다. 배포 전략은 단계적으로 진행됩니다. 개발 환경에서 사전 프로덕션 환경까지는 자동으로 배포하고, 사전 프로덕션에서 프로덕션 환경으로 넘어갈 때는 수동 승인을 거칩니다. 이렇게 하면 잘못된 파이프라인이 프로덕션에 바로 반영되는 사고를 막을 수 있습니다.

Level 2의 전체 흐름은 6단계로 구성됩니다.

```
(1) 개발 및 실험
     │
     ▼
(2) 파이프라인 CI ─── 소스코드 빌드, 단위/통합 테스트, 패키지 생성
     │
     ▼
(3) 파이프라인 CD ─── 아티팩트를 타겟 환경에 배포
     │
     ▼
(4) 자동 트리거 ──── 스케줄 또는 이벤트 기반으로 파이프라인 실행
     │
     ▼
(5) 모델 CD ──────── 훈련된 모델을 예측 서비스로 배포
     │
     ▼
(6) 모니터링 ─────── 라이브 데이터 기반 성능 수집, 이상 감지
```

Level 2에서는 이 흐름을 지탱하는 통합 구성 요소가 갖추어집니다. 원본 제어(코드 저장소), 테스트 및 빌드 서비스, 배포 서비스, 모델 레지스트리, 특성 저장소, ML 메타데이터 저장소, 파이프라인 오케스트레이터가 하나의 체계로 연결됩니다. 데이터 과학자가 새로운 아이디어를 실험하고, 그 결과를 빠르게 프로덕션에 반영할 수 있는 구조가 완성되는 것입니다.

지금까지 살펴본 Google의 3단계 모델(Level 0, 1, 2)은 자동화의 범위를 기준으로 단계를 나눕니다. Microsoft는 같은 진화 과정을 더 세밀하게 5단계(Level 0부터 4까지)로 구분합니다. 평가 차원도 기술 도구에만 한정하지 않고 세 가지 축으로 넓힙니다.

첫 번째 축은 **사람과 문화**(People)입니다. 데이터 과학자와 엔지니어가 얼마나 긴밀하게 협업하는지, 실험 결과를 팀 전체가 공유하는 문화가 갖추어져 있는지를 평가합니다.

두 번째 축은 **프로세스와 구조**(Processes)입니다. 모델 개발부터 배포, 모니터링까지의 워크플로가 문서화되어 있는지, 반복 가능한 절차로 정립되어 있는지를 살펴봅니다.

세 번째 축은 **기술과 도구**(Technology)입니다. 버전 관리, 자동 훈련, CI/CD 파이프라인 등 자동화를 뒷받침하는 도구가 얼마나 갖추어져 있는지를 측정합니다.

Microsoft의 각 단계에서 핵심이 되는 전환점을 하나씩 살펴봅니다.

**Level 0 No MLOps**는 Google의 Level 0과 동일합니다. 모든 것이 수동이고, 실험 추적이 없으며, 모델 파일을 수작업으로 전달합니다.

**Level 1 DevOps but no MLOps**는 Google 모델에는 없는 중간 단계입니다. 소프트웨어 개발의 기본 관행인 코드 버전 관리와 자동 빌드, 단위 테스트가 도입됩니다. 그러나 ML 고유의 자동화, 즉 훈련 자동화나 모델 검증은 아직 이루어지지 않습니다. 코드는 관리하지만 데이터와 모델은 여전히 수동으로 다루는 상태입니다.

**Level 2 Automated training**은 Google의 Level 1에 대응합니다. 훈련이 자동화되고, 특성 저장소가 도입되며, 실험 추적이 중앙에서 이루어집니다. 이벤트 기반 트리거(예: Azure Event Grid)로 훈련을 자동 실행할 수 있습니다.

**Level 3 Automated model deployment**는 Google의 Level 2 전반부에 대응합니다. CI/CD 파이프라인이 완성되어 모델 배포가 자동화됩니다. A/B 테스트를 통합하여 새 모델과 기존 모델의 성능을 프로덕션 트래픽으로 비교할 수 있습니다.

**Level 4 Full MLOps automated operations**는 Google의 Level 2 후반부를 넘어서는 단계입니다. 이 단계에서는 사람의 개입 없이 시스템이 스스로 모델을 재훈련하고 교체합니다. 이 내용은 뒤에서 더 자세히 다룹니다.

두 모델의 단계 번호가 달라서 혼동하기 쉽습니다. 대응 관계를 표로 정리하면 다음과 같습니다.

```
Google 3단계 모델          Microsoft 5단계 모델
─────────────────       ─────────────────────────────
Level 0 (수동)      ──── Level 0 (No MLOps)
       -            ──── Level 1 (DevOps but no MLOps)
Level 1 (훈련 자동화) ── Level 2 (Automated training)
Level 2 (CI/CD)     ──── Level 3 (Automated model deployment)
       -            ──── Level 4 (Full MLOps automated operations)
```

Google 모델에는 Level 1이 없는 것이 아니라, Microsoft가 Google의 Level 0과 Level 1 사이에 'DevOps는 있지만 MLOps는 없는' 중간 단계를 추가한 것입니다. 마찬가지로, Google의 Level 2가 Microsoft에서는 Level 3과 Level 4로 나뉩니다.

이제 Microsoft 모델의 최상위 단계인 **Level 4 자동 재훈련**을 자세히 살펴봅니다. Level 4의 핵심은 프로덕션 메트릭이 자동으로 재훈련을 트리거하는 완전 자동화 상태라는 점입니다.

재훈련을 유발하는 신호는 크게 두 가지입니다. 첫 번째는 **드리프트 신호**입니다. 1.1.1에서 데이터 드리프트(입력 데이터의 분포 변화)와 개념 드리프트(입력과 출력 사이 관계 자체의 변화)를 설명했습니다. Level 4에서는 이 드리프트를 감지하면 재훈련 파이프라인이 즉시 실행됩니다. 두 번째는 **특성 신선도 저하**입니다. 특성 저장소에 저장된 데이터의 품질이 설정한 임계값 아래로 떨어지면 재훈련이 트리거됩니다.

재훈련 전략은 두 가지로 나뉩니다. **예약 재훈련**은 정해진 주기(예: 매주 월요일)에 자동으로 실행합니다. 구현이 단순하고 예측 가능하다는 장점이 있습니다. **이벤트 기반 재훈련**은 드리프트 감지나 특성 신선도 저하 같은 이벤트가 발생했을 때만 실행합니다. 불필요한 재훈련을 줄여 자원을 효율적으로 사용합니다. 실무에서는 두 전략을 혼합하여 사용하는 것이 권장됩니다. 기본은 주기적으로 재훈련하되, 급격한 드리프트가 감지되면 즉시 추가 재훈련을 실행하는 방식입니다.

다만 단순히 새 데이터로 기존 모델을 다시 학습시키는 것이 항상 좋은 결과를 내는 것은 아닙니다. 드리프트의 원인을 분석하지 않고 맹목적으로 재훈련하면, 오히려 미래 예측 오차가 증가할 수 있습니다. 따라서 드리프트 분석에 기반한 지능형 재훈련이 중요합니다.

Level 4에서 또 하나 중요한 요소는 **모델 프로모션의 자동화**입니다. 모델 프로모션이란 새로 훈련된 모델을 개발 환경에서 스테이징, 프로덕션으로 단계적으로 승격시키는 과정입니다. Level 3까지는 프로덕션 배포 직전에 사람이 승인 버튼을 눌러야 했습니다. Level 4에서는 미리 설정한 정책(예: '정확도가 기존 모델보다 2% 이상 높으면 자동 승격')에 따라 ML 레지스트리를 통해 스테이지 간 이동이 자동으로 이루어집니다.

성숙도가 높아질수록 인시던트나 오류가 단순히 문제로 남지 않고 개발 및 프로덕션 프로세스를 개선하는 피드백으로 전환될 가능성이 높아집니다. 이것이 성숙도 모델이 궁극적으로 지향하는 방향입니다.

정리하면, MLOps 성숙도 모델은 조직의 ML 운영 자동화 수준을 단계적으로 구분하는 프레임워크입니다. Google은 수동 프로세스(Level 0), ML 파이프라인 자동화(Level 1), CI/CD 파이프라인 자동화(Level 2)의 3단계로, Microsoft는 이를 더 세밀하게 5단계(Level 0~4)로 나눕니다. Level 0에서는 모든 것이 수동이고, Level 1에서는 훈련이 자동화되며 특성 저장소와 메타데이터 관리가 도입됩니다. Level 2에서는 파이프라인 자체를 CI/CD로 관리합니다. Microsoft의 Level 4에 이르면 드리프트 감지에 따른 자동 재훈련과 정책 기반 모델 프로모션까지 자동화됩니다. 조직은 자신의 현재 수준을 진단하고, 다음 단계에서 어떤 자동화를 도입해야 하는지 로드맵을 세울 수 있습니다.

다음 단원인 1.1.3에서는 ML을 위한 CI/CD/CT 파이프라인을 다룹니다. Level 1과 Level 2에서 등장한 지속적 훈련, 지속적 통합, 지속적 배포가 ML 시스템에서 구체적으로 어떻게 구현되는지 살펴봅니다.

이 단원을 마치면 Google과 Microsoft의 MLOps 성숙도 수준(Level 0~4)을 구분하고 각 단계의 특징을 설명할 수 있습니다.
