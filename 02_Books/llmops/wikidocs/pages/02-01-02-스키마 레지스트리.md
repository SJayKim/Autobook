# 2.1.2 스키마 레지스트리

1.3.2에서 프로듀서가 메시지를 토픽에 발행하고 컨슈머가 이를 읽는 구조를 살펴보았습니다. 프로듀서가 보낸 메시지를 컨슈머가 올바르게 해석하려면, 양쪽이 메시지의 형식에 대해 같은 약속을 공유해야 합니다. 예를 들어 프로듀서가 '주문 금액'을 정수로 보냈는데, 컨슈머가 이를 문자열로 읽으려 하면 오류가 발생합니다. 메시지를 주고받는 양쪽이 늘어날수록 이런 형식 불일치의 위험도 함께 커집니다.

소규모 시스템에서는 개발자끼리 구두로 형식을 맞출 수 있습니다. 그러나 프로듀서가 수십 개이고 컨슈머가 수백 개인 환경에서는 이 방법이 통하지 않습니다. 누군가 메시지 형식을 바꾸면 어떤 컨슈머가 영향을 받는지 파악하기 어렵고, 배포 순서를 잘못 잡으면 장애로 이어집니다. 이 문제를 해결하기 위해 메시지 형식의 정의, 즉 **스키마**를 중앙에서 관리하는 저장소가 필요합니다. 그 저장소가 스키마 레지스트리입니다.

**스키마 레지스트리**(Schema Registry)는 Kafka 프로듀서와 컨슈머 사이의 스키마 계약을 중앙에서 관리하는 메타데이터 저장소입니다. 스키마를 등록하고, 버전을 추적하며, 새 스키마가 기존 스키마와 호환되는지를 검사합니다. 호환성 검사에 실패하면 등록을 거부합니다. 형식이 맞지 않는 메시지가 토픽에 들어가는 상황을 사전에 차단합니다.

스키마 레지스트리의 역할을 정리하면 세 가지입니다. 첫째, 스키마를 중앙 저장소에 등록하고 고유 ID를 부여하여 프로듀서와 컨슈머가 같은 형식을 공유하게 합니다. 둘째, 스키마 버전을 추적하고 호환성 규칙에 따라 비호환 스키마의 등록을 거부합니다. 셋째, 메시지에 스키마 ID만 포함시켜 페이로드 크기를 줄이고 직렬화 일관성을 보장합니다.

스키마 레지스트리는 Kafka 클러스터와는 별도의 프로세스로 동작합니다. 하지만 내부 저장소로는 Kafka 자체를 활용합니다. '_schemas'라는 이름의 특수 토픽에 스키마 데이터를 기록합니다. 이 토픽은 파티션이 하나이고, 압축(compaction) 정책이 적용되어 같은 키에 대해 최신 값만 남깁니다. 클라이언트는 RESTful API를 통해 스키마를 저장하거나 검색합니다.

스키마 레지스트리를 사용하면 메시지 크기도 줄어듭니다. 스키마 전체를 메시지에 포함시키는 대신, 레지스트리에 등록된 스키마의 고유 번호인 **스키마 ID**만 메시지에 넣습니다. 이 ID는 단조 증가하는 정수이며, 주(primary) 노드에서만 발급합니다. 직렬화된 메시지의 구조를 와이어 포맷이라 부르는데, 구성은 다음과 같습니다.

```
[Magic Byte (1바이트)] [Schema ID (4바이트)] [실제 페이로드]
```

Magic Byte는 프로토콜 버전을 나타내는 1바이트 값입니다. Schema ID는 레지스트리에 등록된 스키마의 고유 번호로, 4바이트 정수입니다. 실제 페이로드는 스키마에 따라 직렬화된 데이터입니다. 컨슈머는 메시지를 받으면 Schema ID를 꺼내 레지스트리에 조회하고, 해당 스키마로 페이로드를 역직렬화합니다. 한 번 조회한 스키마는 로컬 캐시에 보관하므로 매번 네트워크 요청을 보내지 않습니다.

스키마 레지스트리에는 **Subject**라는 개념이 있습니다. Subject는 스키마 버전 관리의 범위를 결정하는 단위입니다. 기본 전략인 TopicNameStrategy에서는 토픽 이름에 '-key' 또는 '-value'를 붙여 Subject를 만듭니다. 예를 들어 'order' 토픽의 메시지 값에 대한 Subject는 'order-value'입니다. 호환성 검사는 같은 Subject 안에서만 이루어집니다. 하나의 토픽에 여러 유형의 이벤트를 보내야 할 때는 RecordNameStrategy를 사용합니다. 레코드 이름 기준으로 Subject를 분리할 수 있습니다.

이제 스키마 레지스트리가 지원하는 직렬화 형식을 살펴봅니다. 대표적으로 Avro, Protobuf, JSON Schema 세 가지가 있습니다. 각 형식은 스키마를 정의하는 방식과 바이너리 인코딩 방식이 다릅니다.

먼저 **Avro**입니다. Avro는 스키마를 JSON으로 정의하고, 실제 데이터는 바이너리로 직렬화하는 형식입니다. 바이너리 페이로드에는 필드 이름이나 타입 정보가 포함되지 않습니다. 따라서 데이터를 읽으려면 반드시 스키마가 필요합니다. 이 특성이 스키마 레지스트리와 잘 맞습니다. 스키마 없이는 데이터를 해석할 수 없으므로, 레지스트리에 스키마를 등록하고 관리하는 것이 자연스러운 선택이 됩니다.

Avro의 기본 타입은 null, boolean, int, long, float, double, bytes, string의 여덟 가지입니다. 레코드를 정의할 때는 name과 fields를 지정합니다. fields 배열의 각 항목은 필드 이름(name), 타입(type), 선택적으로 기본값(default)을 포함합니다. 다음은 간단한 Avro 스키마 예시입니다.

```json
{
  "type": "record",
  "name": "Order",
  "fields": [
    {"name": "order_id", "type": "long"},
    {"name": "amount", "type": "int"},
    {"name": "currency", "type": "string", "default": "KRW"}
  ]
}
```

order_id는 long 타입, amount는 int 타입, currency는 string 타입이며 기본값이 "KRW"입니다. 기본값이 있는 필드는 뒤에서 다룰 스키마 진화에서 중요한 역할을 합니다. 선택적 필드를 표현할 때는 Union 타입을 사용합니다. '["null", "string"]'처럼 null과 다른 타입을 묶어 해당 필드가 비어 있을 수 있음을 나타냅니다.

Avro의 바이너리 인코딩은 효율적입니다. 정수(int, long)는 가변 길이 지그재그(zigzag) 코딩을 사용하여 작은 값일수록 적은 바이트로 표현합니다. 문자열은 길이를 먼저 기록하고 이어서 UTF-8 바이트를 씁니다. 필드 이름이 빠져 있으므로 같은 데이터를 JSON으로 표현할 때보다 크기가 훨씬 작습니다.

다음은 **Protobuf**입니다. Protobuf(Protocol Buffers)는 '.proto'라는 확장자의 인터페이스 정의 언어(IDL) 파일로 스키마를 정의합니다. 이 파일을 protoc이라는 컴파일러에 넣으면, 사용하는 프로그래밍 언어에 맞는 직렬화/역직렬화 코드가 자동으로 생성됩니다.

Protobuf의 핵심 특징은 필드 번호입니다. 각 필드에 고유한 번호를 부여하고, 바이너리 메시지에는 필드 이름 대신 이 번호가 기록됩니다. 다음은 Protobuf 스키마 예시입니다.

```protobuf
message Order {
  int64 order_id = 1;
  int32 amount = 2;
  string currency = 3;
}
```

등호 뒤의 숫자가 필드 번호입니다. 1부터 15까지의 번호는 1바이트로 인코딩되므로, 자주 사용하는 필드에 이 범위의 번호를 배정하는 것이 좋습니다. 필드 번호는 한 번 정한 뒤에는 변경할 수 없습니다. 번호를 바꾸면 기존에 직렬화된 데이터를 읽을 수 없게 됩니다. 삭제한 필드의 번호를 재사용하는 것도 위험합니다. 이전 데이터에서 해당 번호가 다른 타입의 필드를 가리키고 있을 수 있기 때문입니다. 이를 방지하기 위해 reserved 키워드로 번호와 이름을 예약합니다.

```protobuf
message Order {
  reserved 2, 15;
  reserved "old_field";
  int64 order_id = 1;
  string currency = 3;
}
```

이 선언은 필드 번호 2와 15, 그리고 'old_field'라는 이름을 앞으로 사용하지 못하게 막습니다.

마지막으로 **JSON Schema**입니다. JSON Schema는 독립적인 직렬화 형식이 아닙니다. JSON 문서의 구조를 검증하는 명세입니다. 스키마 자체도 JSON으로 작성하며, type, required, pattern, minimum 같은 키워드로 제약 조건을 정의합니다.

스키마 레지스트리는 JSON Schema도 지원합니다. 이미 JSON 기반으로 통신하는 시스템에 스키마 검증을 추가할 때 유용합니다. 다만 JSON은 텍스트 형식이므로 메시지 크기가 Avro나 Protobuf보다 50에서 80퍼센트 더 큽니다. 직렬화와 역직렬화에 드는 CPU 비용도 높습니다. 극단적인 성능이 필요하지 않고, 사람이 메시지를 직접 읽을 수 있어야 하는 환경에서 선택합니다.

세 형식의 특성을 정리하면 다음과 같습니다.

```
형식         스키마 정의     바이너리 인코딩    메시지 크기    스키마 진화 지원
----------   ------------   ---------------   -----------   ----------------
Avro         JSON           바이너리           작음          기본값 기반
Protobuf     .proto (IDL)   바이너리           가장 작음     필드 번호 기반
JSON Schema  JSON           텍스트 (JSON)      가장 큼       additionalProperties 기반
```

이제 스키마 레지스트리의 핵심 기능인 **스키마 호환성 모드**를 다룹니다. 시스템이 운영되면서 메시지 형식은 바뀌기 마련입니다. 새 필드를 추가하거나, 더 이상 필요 없는 필드를 제거해야 할 때가 옵니다. 문제는 프로듀서와 컨슈머를 동시에 업그레이드하기 어렵다는 점입니다. 수백 개의 컨슈머가 동시에 새 버전으로 전환하는 것은 현실적으로 불가능합니다. 따라서 일정 기간 동안 이전 버전과 새 버전의 스키마가 공존해야 합니다.

스키마 레지스트리는 새 스키마를 등록할 때 기존 버전과의 호환성을 자동으로 검사합니다. 이 검사의 기준이 되는 것이 호환성 모드입니다. 호환성 모드는 전역으로 설정하거나 Subject 단위로 개별 지정할 수 있습니다.

가장 기본이 되는 세 가지 모드는 BACKWARD, FORWARD, FULL입니다.

**BACKWARD** 호환성은 '새 스키마로 과거 데이터를 읽을 수 있다'는 의미입니다. 스키마 레지스트리의 기본 설정값이기도 합니다. 컨슈머가 먼저 새 버전의 스키마로 업그레이드하고, 그 다음에 프로듀서를 업그레이드하는 순서를 전제합니다.

구체적인 상황을 봅니다. 프로듀서가 version 1 스키마로 메시지를 보내고 있습니다. 이제 version 2 스키마에 'favorite_color'라는 필드를 추가하려 합니다. BACKWARD 호환을 유지하려면 이 새 필드에 반드시 기본값을 지정해야 합니다. 컨슈머가 version 2 스키마를 먼저 적용한다고 가정합니다. version 1으로 직렬화된 과거 메시지에는 'favorite_color' 필드가 없습니다. 이때 기본값이 그 자리를 채웁니다.

```
  BACKWARD 호환 — 업그레이드 순서

  1. 컨슈머를 version 2 스키마로 업그레이드
     -> version 1 데이터를 읽을 때: 새 필드는 기본값으로 채움
  2. 프로듀서를 version 2 스키마로 업그레이드
     -> 이후 새 메시지에는 새 필드가 포함됨
```

BACKWARD 호환에서 허용되는 변경은 기본값이 있는 선택적 필드 추가와 기존 필드 삭제입니다. 삭제된 필드는 과거 데이터에 남아 있지만, 새 스키마에서는 해당 필드를 무시합니다.

**FORWARD** 호환성은 반대 방향입니다. '과거 스키마로 새 데이터를 읽을 수 있다'는 의미입니다. 프로듀서가 먼저 새 버전으로 업그레이드하고, 컨슈머를 나중에 업그레이드하는 순서를 전제합니다.

프로듀서가 version 2 스키마로 새 필드를 포함한 메시지를 보냅니다. 아직 version 1 스키마를 사용하는 컨슈머는 모르는 필드를 만납니다. FORWARD 호환이 유지되려면, 이전 스키마를 쓰는 컨슈머가 새 필드를 무시할 수 있어야 합니다.

```
  FORWARD 호환 — 업그레이드 순서

  1. 프로듀서를 version 2 스키마로 업그레이드
     -> 새 필드가 포함된 메시지를 전송
  2. 컨슈머는 아직 version 1 스키마
     -> 모르는 필드를 무시하고 기존 필드만 읽음
  3. 컨슈머를 version 2 스키마로 업그레이드
```

Protobuf는 이전 코드가 알지 못하는 필드 번호를 자동으로 무시하는 특성이 있어, FORWARD 호환에 자연스럽게 부합합니다.

**FULL** 호환성은 BACKWARD와 FORWARD를 동시에 만족하는 가장 엄격한 모드입니다. 프로듀서와 컨슈머를 어떤 순서로 업그레이드해도 안전합니다. 양방향 호환을 유지해야 하므로 허용되는 변경이 가장 제한적입니다. 기본값이 있는 선택적 필드만 추가하거나 삭제할 수 있습니다. 배포 순서를 신경 쓸 필요가 없으므로, 지속적 배포(CD) 파이프라인에서 가장 안전한 선택입니다.

세 모드 외에도 BACKWARD_TRANSITIVE, FORWARD_TRANSITIVE, FULL_TRANSITIVE가 있습니다. 기본 모드가 직전 버전(N-1)과의 호환만 검사하는 반면, TRANSITIVE가 붙은 모드는 모든 이전 버전과의 호환을 검사합니다. NONE 모드는 호환성 검사를 비활성화합니다. 이 경우 프로듀서와 컨슈머를 동시에 업그레이드해야 하므로, 운영 환경에서는 사용을 피하는 것이 일반적입니다.

이제 **스키마 진화**를 전체적으로 정리합니다. 스키마 진화란 이미 운영 중인 시스템에서 기존 메시지를 무효화하지 않으면서 스키마를 변경하는 것을 뜻합니다. 스키마 레지스트리가 각 버전을 영구 보존하고 고유한 ID를 부여하기 때문에 가능합니다.

스키마 변경은 안전도에 따라 세 가지로 분류합니다. 첫째, 안전한 변경입니다. 기본값이 있는 필드를 추가하거나, 선택적 필드를 삭제하는 것이 여기에 해당합니다. 둘째, 조건부로 안전한 변경입니다. Avro에서 aliases 속성으로 필드 이름을 바꾸는 경우가 이에 해당합니다. Protobuf에서 wire-compatible한 정수 타입 간 변환도 마찬가지입니다. 데이터 손실 가능성이 있으므로 주의가 필요합니다. 셋째, 위험한 변경입니다. 필드 타입을 바꾸거나, Protobuf에서 삭제한 필드 번호를 재사용하는 것이 해당합니다. 기본값 없는 필수 필드를 추가하는 것도 기존 데이터와 호환되지 않습니다.

호환성 모드에 따른 업그레이드 순서를 한번 더 정리합니다.

```
호환성 모드    업그레이드 순서         핵심 조건
-----------   --------------------   --------------------------------
BACKWARD      컨슈머 먼저            새 필드에 기본값 필수
FORWARD       프로듀서 먼저          이전 스키마가 새 필드를 무시 가능
FULL          순서 무관              기본값 있는 선택적 필드만 변경
NONE          동시 업그레이드 필수    호환성 검사 없음
```

정리하면, 스키마 레지스트리는 Kafka 메시지의 형식을 중앙에서 관리하고, 스키마 버전 간 호환성을 강제하는 메타데이터 저장소입니다. 메시지에는 스키마 ID만 포함하여 페이로드 크기를 줄입니다. 컨슈머는 ID로 레지스트리를 조회하여 역직렬화합니다. 직렬화 형식으로는 바이너리 효율이 높은 Avro와 Protobuf, 텍스트 기반의 JSON Schema를 지원합니다. 호환성 모드(BACKWARD, FORWARD, FULL 등)는 스키마 변경 시 업그레이드 순서를 결정합니다. 레지스트리가 이를 자동으로 검사하여 호환되지 않는 스키마의 등록을 차단합니다.

다음 단원인 2.1.3에서는 Kafka Connect를 다룹니다. 외부 시스템과 Kafka 사이에서 데이터를 주고받는 커넥터의 구조와 동작 방식을 살펴봅니다.

이 단원을 마치면 스키마 레지스트리가 메시지 직렬화 일관성을 보장하는 방식과 스키마 진화 규칙을 설명할 수 있습니다.
