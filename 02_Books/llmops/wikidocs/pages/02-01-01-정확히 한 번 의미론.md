# 2.1.1 정확히 한 번 의미론

1.3.5에서 Kafka가 파티션을 복제하여 메시지 유실을 막는 구조를 살펴보았습니다. 리더에 기록된 메시지가 ISR 내 팔로워에 복제되고, acks=all과 min.insync.replicas 조합으로 내구성을 확보하는 방식이었습니다. 그런데 내구성만으로는 해결되지 않는 문제가 하나 남아 있습니다. 프로듀서가 메시지를 보내고 네트워크 오류가 발생하면, 브로커가 실제로 기록을 마쳤는지 프로듀서는 알 수 없습니다. 확인응답이 도착하지 않았으므로 프로듀서는 같은 메시지를 다시 보냅니다. 브로커가 이미 기록을 마친 상태라면 동일한 메시지가 두 번 저장됩니다. 반대로 재시도를 하지 않으면 메시지가 유실될 수 있습니다. 이 딜레마에서 '중복도 유실도 없이 정확히 한 번만 처리한다'는 보장이 필요해집니다. 이것이 정확히 한 번 의미론(Exactly-Once Semantics, EOS)입니다.

정확히 한 번 의미론은 두 가지 계층의 메커니즘이 결합되어 동작합니다. 첫째는 단일 파티션 범위에서 재시도 중복을 제거하는 **멱등 프로듀서**입니다. 둘째는 여러 파티션과 토픽에 걸친 쓰기를 하나의 원자 단위로 묶는 **트랜잭션 API**입니다. 이 단원에서는 멱등 프로듀서부터 시작하여 트랜잭션 API까지, 두 계층이 어떻게 맞물려 정확히 한 번을 보장하는지 단계별로 살펴봅니다.

멱등 프로듀서를 이해하려면 먼저 '멱등'이라는 말의 의미를 짚어야 합니다. 멱등(idempotent)이란 같은 연산을 여러 번 반복해도 결과가 한 번 수행한 것과 동일하다는 뜻입니다. 예를 들어 은행 계좌의 잔액을 '1000원으로 설정하라'는 연산은 멱등입니다. 몇 번을 반복해도 잔액은 1000원입니다. 반면 '1000원을 더하라'는 연산은 멱등이 아닙니다. 두 번 실행하면 2000원이 추가됩니다. Kafka에 메시지를 기록하는 행위는 기본적으로 멱등이 아닙니다. 같은 메시지를 두 번 보내면 로그에 두 건이 쌓입니다. 멱등 프로듀서는 이 기본 동작을 바꿔서, 같은 메시지를 여러 번 보내더라도 브로커가 한 번만 기록하도록 만듭니다.

멱등 프로듀서를 활성화하려면 프로듀서 설정에서 enable.idempotence를 true로 지정합니다. Kafka 3.0 이상에서는 이 값이 기본으로 활성화되어 있으므로 별도 설정 없이도 멱등 프로듀서가 동작합니다. 멱등 프로듀서가 동작하는 원리는 **PID**(Producer ID)와 **시퀀스 번호**(sequence number)라는 두 가지 식별자에 있습니다.

PID는 프로듀서가 브로커에 처음 연결할 때 자동으로 발급받는 고유 식별자입니다. 프로듀서가 스스로 정하는 것이 아니라, 브로커 내부의 ProducerIdManager가 단조 증가하는 번호를 블록 단위로 생성하여 할당합니다. 이 번호는 ZooKeeper 또는 KRaft에 영속 저장되어 브로커 간 범위 충돌을 방지합니다.

시퀀스 번호는 프로듀서가 특정 토픽-파티션으로 전송하는 배치마다 부여하는 번호입니다. 0부터 시작하여 배치에 담긴 레코드 수만큼 증가합니다. 예를 들어 첫 배치에 레코드 3개를 담으면 시퀀스 번호가 0, 1, 2가 됩니다. 다음 배치는 3부터 시작합니다.

브로커는 (PID, 파티션) 조합별로 마지막으로 커밋한 시퀀스 번호를 추적합니다. 새 메시지가 도착하면 브로커는 세 가지 중 하나로 분류합니다.

```
  브로커의 시퀀스 번호 검증 규칙

  마지막 커밋 시퀀스: N

  도착한 시퀀스가 N+1  -->  정상: 로그에 기록
  도착한 시퀀스가 N 이하  -->  중복: 기록하지 않고 성공 응답 반환
  도착한 시퀀스가 N+1 초과  -->  순서 오류: OutOfOrderSequenceException 발생
```

핵심은 '중복' 판정입니다. 프로듀서가 네트워크 오류로 같은 배치를 재전송하면, 시퀀스 번호가 이미 커밋된 값 이하이므로 브로커는 로그에 쓰지 않고 성공 응답만 돌려줍니다. 프로듀서 입장에서는 메시지가 정상 전달된 것이고, 브로커 입장에서는 중복이 제거된 것입니다. 이 시퀀스 번호는 복제 로그에 영속 저장됩니다. TCP 연결의 휘발성 메모리와 달리, 브로커가 재시작되어도 중복 감지 능력을 잃지 않습니다.

브로커 내부에서는 ProducerStateManager라는 컴포넌트가 이 매핑을 관리합니다. (PID, 파티션)별로 최근 배치 메타데이터를 최대 5개까지 큐에 보관합니다. 새 배치가 도착하면 이 큐를 참조하여 중복 여부를 판단합니다.

멱등 프로듀서에는 분명한 한계가 있습니다. PID는 프로듀서 인스턴스가 브로커에 연결할 때 발급되므로, 프로듀서가 재시작되면 새로운 PID를 받습니다. 이전 세션에서 이미 보낸 메시지와 새 세션에서 보내는 메시지를 구분할 수 없습니다. 따라서 멱등 프로듀서는 하나의 프로듀서 세션 안에서, 하나의 파티션에 대한 재시도 중복만 제거합니다. 여러 파티션에 걸친 원자적 쓰기나, 프로듀서 재시작 후의 중복 방지는 보장하지 못합니다.

이 한계를 넘어서려면 트랜잭션 API가 필요합니다. 트랜잭션 API는 여러 파티션과 토픽에 걸친 쓰기를 하나의 원자 단위로 묶습니다. '원자 단위'란 모든 쓰기가 성공하거나 모두 취소되는 것을 뜻합니다. 일부만 성공하고 일부는 실패하는 중간 상태가 없습니다.

트랜잭션 API를 사용하려면 프로듀서 설정에 transactional.id를 지정합니다. 이 값은 프로듀서가 재시작되어도 동일하게 유지하는 논리적 식별자입니다. PID와 달리 사용자가 직접 정합니다. 같은 transactional.id로 새 세션을 열면, 브로커는 이전 세션의 미완료 트랜잭션을 자동으로 중단시키고 새 에포크(epoch)를 발급합니다.

에포크는 같은 transactional.id를 가진 프로듀서의 세대를 구분하는 번호입니다. 새 세션이 열릴 때마다 에포크가 1씩 증가합니다. 브로커는 낮은 에포크의 요청을 거부합니다. 이 메커니즘으로 이전 세션의 프로듀서가 여전히 살아 있더라도(좀비 프로듀서), 새 세션의 프로듀서와 충돌하지 않습니다. 브로커가 좀비를 차단하는 이 동작을 '펜싱'(fencing)이라 합니다.

트랜잭션 API의 사용 순서는 다음과 같습니다.

```
producer.initTransactions()           // 1. 코디네이터와 연결, PID와 에포크 발급
producer.beginTransaction()           // 2. 트랜잭션 시작
producer.send(record1)                // 3. 파티션 A에 레코드 전송
producer.send(record2)                // 4. 파티션 B에 레코드 전송
producer.sendOffsetsToTransaction()   // 5. 소비한 오프셋을 트랜잭션에 포함
producer.commitTransaction()          // 6. 커밋 요청
```

initTransactions()는 프로듀서가 트랜잭션 코디네이터와 연결을 맺고 PID와 에포크를 받는 단계입니다. 같은 transactional.id로 이전 세션이 남긴 미완료 트랜잭션이 있다면 이 시점에 정리됩니다. beginTransaction()은 로컬 상태를 READY에서 IN_TRANSACTION으로 전환합니다. 이후의 send() 호출은 모두 현재 트랜잭션에 포함됩니다.

send()로 전송한 레코드는 브로커의 파티션 로그에 기록되지만, 아직 커밋 마커가 없으므로 특정 조건의 컨슈머에게는 보이지 않습니다. 이 '보이지 않는다'는 부분은 뒤에서 isolation.level을 설명할 때 다시 다룹니다.

sendOffsetsToTransaction()는 '읽기-처리-쓰기' 파이프라인에서 핵심적인 역할을 합니다. Kafka Streams 같은 스트림 처리 애플리케이션은 토픽에서 메시지를 읽고, 처리한 결과를 다른 토픽에 쓰고, 소비한 오프셋을 커밋하는 세 단계를 반복합니다. 1.3.5에서 컨슈머가 __consumer_offsets 토픽에 오프셋을 커밋하는 방식을 다루었습니다. sendOffsetsToTransaction()는 이 오프셋 커밋을 현재 트랜잭션에 묶습니다. 트랜잭션이 커밋되어야만 오프셋도 커밋되고, 트랜잭션이 중단되면 오프셋 커밋도 취소됩니다. 출력 쓰기와 오프셋 커밋이 원자적으로 처리되므로, 장애 후 재시작해도 같은 메시지를 중복으로 처리하거나 건너뛰는 일이 발생하지 않습니다.

commitTransaction()을 호출하면 내부적으로 2단계 커밋(2PC) 프로토콜이 실행됩니다. 이 2PC는 **트랜잭션 코디네이터**가 관리합니다.

트랜잭션 코디네이터는 브로커 내부에서 실행되는 컴포넌트입니다. 별도의 외부 프로세스가 아니라 브로커의 일부로 동작합니다. 코디네이터가 트랜잭션 상태를 기록하는 장소는 **__transaction_state**라는 내부 Kafka 토픽입니다. 이 토픽은 기본적으로 50개 파티션으로 구성됩니다. 프로듀서의 transactional.id를 해시하여 파티션이 결정되고, 그 파티션의 리더 브로커가 해당 트랜잭션의 코디네이터가 됩니다. __transaction_state 토픽 자체도 Kafka의 복제 메커니즘으로 보호되므로, 코디네이터 브로커가 장애를 일으키면 다른 브로커가 리더로 선출되어 트랜잭션 상태를 복구합니다.

코디네이터의 역할을 정리하면 다섯 가지입니다. 프로듀서에게 PID와 에포크를 발급합니다. 트랜잭션 상태를 __transaction_state에 기록하고 복구합니다. 2PC의 준비 및 커밋 단계를 조율합니다. 트랜잭션 타임아웃을 감지하여 자동으로 중단 처리합니다. 동일 transactional.id의 좀비 프로듀서를 에포크 증가로 차단합니다.

이제 2PC가 실제로 어떻게 진행되는지 단계별로 살펴봅니다.

1단계는 준비(Prepare)입니다. 프로듀서가 commitTransaction()을 호출하면, 코디네이터는 __transaction_state 토픽에 해당 트랜잭션의 상태를 PREPARE_COMMIT으로 기록합니다. 이 기록이 영속 저장되므로, 이 시점 이후에 코디네이터가 장애를 일으키더라도 재시작 후 커밋을 이어갈 수 있습니다.

2단계는 커밋(Commit)입니다. 코디네이터가 트랜잭션에 참여한 각 파티션의 리더 브로커에게 COMMIT 마커를 기록하도록 요청합니다. 모든 파티션에 COMMIT 마커가 기록되면, 코디네이터는 __transaction_state의 상태를 COMPLETE_COMMIT으로 갱신합니다. 트랜잭션이 중단되는 경우에는 COMMIT 마커 대신 ABORT 마커가 기록됩니다.

```
  2PC 흐름

  프로듀서                 트랜잭션 코디네이터              파티션 리더들
     |                          |                              |
     |-- commitTransaction() -->|                              |
     |                          |-- PREPARE_COMMIT 기록        |
     |                          |   (__transaction_state)       |
     |                          |                              |
     |                          |-- COMMIT 마커 기록 요청 ---->|
     |                          |                              |-- 파티션 A에 마커 기록
     |                          |                              |-- 파티션 B에 마커 기록
     |                          |<-- 완료 응답 ----------------|
     |                          |                              |
     |                          |-- COMPLETE_COMMIT 기록       |
     |                          |   (__transaction_state)       |
     |<-- 커밋 완료 ------------|                              |
```

전통적인 분산 시스템의 2PC에서는 참여자가 장애를 일으키면 코디네이터가 대기 상태에 빠지는 블로킹 문제가 있습니다. Kafka의 2PC는 이 문제를 다르게 해결합니다. 트랜잭션 로그 자체가 Kafka 토픽이므로 복제에 의한 내구성이 보장됩니다. 코디네이터도 브로커의 리더 선출 메커니즘을 활용하여 자동 복구됩니다. 외부의 별도 코디네이터 프로세스가 필요하지 않습니다.

트랜잭션이 커밋되더라도, 컨슈머가 커밋된 메시지만 읽으려면 별도의 설정이 필요합니다. 이 설정이 **isolation.level**입니다. 컨슈머 측 설정이며 두 가지 값을 가집니다.

**read_uncommitted**은 기본값입니다. 트랜잭션 커밋 여부와 관계없이 모든 메시지를 즉시 읽습니다. 아직 진행 중이거나 중단된 트랜잭션의 메시지도 포함됩니다. 레이턴시가 낮지만, 결과적으로 취소될 데이터를 읽을 수 있습니다.

**read_committed**는 커밋 완료된 트랜잭션의 메시지만 읽습니다. 진행 중인 트랜잭션의 메시지는 컨슈머에게 전달되지 않습니다. ABORT 마커가 붙은 트랜잭션의 메시지는 필터링됩니다. 정확히 한 번 의미론을 완성하려면 컨슈머에 이 설정을 적용해야 합니다. 프로듀서 측에서 트랜잭션으로 원자적 쓰기를 보장하더라도, 컨슈머가 read_uncommitted 상태라면 커밋 전의 중간 데이터를 읽을 수 있기 때문입니다.

read_committed 모드에서 컨슈머가 읽을 수 있는 오프셋의 상한선을 LSO(Last Stable Offset)라 합니다. 완료되지 않은 트랜잭션이 존재하면, LSO는 해당 트랜잭션의 시작 오프셋에 묶입니다. 그 트랜잭션이 커밋되거나 중단되면 LSO가 전진합니다. 컨슈머는 LSO 이전의 메시지만 가져올 수 있으므로, 아직 결과가 정해지지 않은 트랜잭션의 메시지를 볼 일이 없습니다. 트랜잭션 없이 전송된 일반 메시지는 LSO 이전에 위치하면 정상적으로 읽힙니다.

Kafka Streams에서는 processing.guarantee를 exactly_once로 설정하면 내부적으로 read_committed가 자동 적용됩니다. 입력 읽기, 처리, 출력 쓰기, 오프셋 커밋이 하나의 트랜잭션으로 묶여 동작합니다.

```
  정확히 한 번 의미론의 전체 구조

  프로듀서 측                         브로커 내부                      컨슈머 측
  +---------------------------+      +---------------------------+    +-------------------+
  | enable.idempotence=true   |      | PID + 시퀀스 번호 검증    |    | isolation.level   |
  | -> 단일 파티션 중복 제거  |      | -> 중복 배치 거부         |    | =read_committed   |
  +---------------------------+      +---------------------------+    | -> 커밋된 메시지만 |
  | transactional.id 설정     |      | 트랜잭션 코디네이터       |    |    읽기            |
  | -> 다중 파티션 원자 쓰기  |      | -> __transaction_state    |    | -> LSO 기반 필터  |
  | -> 좀비 펜싱 (에포크)     |      | -> 2PC (Prepare + Commit) |    +-------------------+
  +---------------------------+      +---------------------------+
```

정리하면, 정확히 한 번 의미론은 세 가지 요소가 맞물려 동작합니다. 멱등 프로듀서가 PID와 시퀀스 번호로 단일 파티션 내 재시도 중복을 제거합니다. 트랜잭션 API가 여러 파티션에 걸친 쓰기와 오프셋 커밋을 2PC로 원자적으로 묶습니다. 컨슈머의 read_committed 설정이 커밋된 트랜잭션의 메시지만 전달하여 중간 상태 노출을 차단합니다. 이 세 계층이 결합되어야 '읽기-처리-쓰기' 파이프라인 전체에서 메시지가 정확히 한 번만 처리됩니다.

다음 단원인 2.1.2에서는 스키마 레지스트리를 다룹니다. 프로듀서와 컨슈머가 주고받는 메시지의 구조를 어떻게 관리하고 호환성을 보장하는지 살펴봅니다.

이 단원을 마치면 멱등 프로듀서와 트랜잭션 API를 결합하여 정확히 한 번 처리를 보장하는 메커니즘을 설명할 수 있습니다.
