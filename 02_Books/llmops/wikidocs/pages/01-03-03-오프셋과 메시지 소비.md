# 1.3.3 오프셋과 메시지 소비

1.3.2에서 토픽이 여러 파티션으로 나뉘고, 컨슈머가 파티션 단위로 메시지를 읽는 구조를 살펴보았습니다. 그런데 한 가지 의문이 남습니다. 컨슈머가 메시지를 읽다가 중간에 멈추거나 장애로 종료되면, 다시 시작했을 때 어디서부터 읽어야 할까요. 파티션에 이미 수만 건의 메시지가 쌓여 있는 상태에서, 처음부터 전부 다시 읽으면 중복 처리가 발생합니다. 이미 처리한 메시지를 건너뛰려면, 컨슈머가 '어디까지 읽었는지'를 어딘가에 기록해 두어야 합니다. 이 위치 정보를 관리하는 것이 바로 오프셋입니다.

**오프셋**(offset)은 파티션 안에서 개별 메시지의 위치를 가리키는 숫자입니다. 파티션에 메시지가 추가될 때마다 Kafka가 자동으로 부여합니다. 첫 번째 메시지는 오프셋 0, 두 번째 메시지는 오프셋 1, 세 번째는 오프셋 2를 받습니다. 컨슈머는 메시지를 읽을 때 메시지 본문과 함께 이 오프셋 값을 수신합니다. 오프셋을 보면 '이 메시지가 파티션 안에서 몇 번째 위치에 있는지'를 정확히 알 수 있습니다.

오프셋이 갖는 핵심 성질은 **단조 증가**(monotonically increasing)입니다. 단조 증가란 값이 항상 같거나 커지기만 하고 절대 줄어들지 않는다는 뜻입니다. Kafka에서는 구체적으로 0부터 시작해 메시지가 추가될 때마다 정확히 1씩 증가합니다. 64비트 정수를 사용하므로 사실상 고갈될 걱정이 없습니다. 1.3.2에서 다룬 것처럼 파티션은 커밋 로그, 즉 끝에만 덧붙이고 기존 기록은 수정하지 않는 구조입니다. 새 메시지는 항상 로그의 맨 끝에 추가되고, 그때마다 직전 오프셋보다 1 큰 값을 받습니다.

이 단조 증가 성질 덕분에 컨슈머는 간단한 규칙으로 읽기 상태를 판단할 수 있습니다. 오프셋 25번 메시지까지 처리했다면, 26번부터 읽으면 됩니다. 25보다 작은 오프셋의 메시지는 이미 처리한 것이 확실합니다.

파티션 하나에 메시지 다섯 개가 쌓여 있는 상황을 그려 봅니다.

```
  파티션 0

  오프셋:  0     1     2     3     4
         [msg] [msg] [msg] [msg] [msg]
                             ^
                        컨슈머가 마지막으로 읽은 위치

  커밋된 오프셋: 4  (= 다음에 읽을 위치)
```

여기서 주의할 점이 있습니다. 컨슈머가 오프셋 3번 메시지까지 읽었다면, Kafka에 기록하는 값은 3이 아니라 4입니다. 커밋된 오프셋은 '마지막으로 처리한 메시지의 번호'가 아니라 '다음에 읽을 메시지의 번호'를 의미합니다. 이 규칙을 혼동하면 메시지 하나를 건너뛰거나 중복 처리하는 원인이 됩니다.

그렇다면 컨슈머가 기록한 오프셋은 어디에 저장될까요. Kafka는 이를 위해 **__consumer_offsets**라는 내부 토픽을 사용합니다. 일반적으로 사용자가 만드는 토픽과 달리, 이 토픽은 Kafka가 스스로 생성하고 관리합니다. 컨슈머 그룹이 '파티션 0의 오프셋을 4까지 처리했다'는 정보를 커밋하면, 그 내용이 __consumer_offsets 토픽에 메시지 형태로 기록됩니다.

__consumer_offsets 토픽이 동작하는 과정을 단계별로 살펴봅니다.

```
  오프셋 커밋 과정

  1. 컨슈머가 파티션 0에서 오프셋 3번까지 메시지를 처리한다.
  2. 컨슈머가 브로커에 "파티션 0, 오프셋 4를 커밋한다"는 요청을 보낸다.
  3. 브로커는 해당 컨슈머 그룹 ID를 해시하여
     __consumer_offsets 토픽의 특정 파티션을 선택한다.
  4. 선택된 파티션에 오프셋 커밋 메시지를 기록한다.
  5. 컨슈머가 재시작하면, __consumer_offsets에서
     마지막 커밋 값(4)을 읽어 오프셋 4부터 소비를 재개한다.
```

__consumer_offsets 토픽은 기본적으로 50개의 파티션으로 구성됩니다. 컨슈머 그룹 ID의 해시값을 50으로 나눈 나머지로 저장 파티션이 결정됩니다. 같은 컨슈머 그룹의 오프셋 커밋은 항상 같은 파티션으로 모입니다. 그 파티션을 담당하는 브로커를 Group Coordinator라 부르며, 이 브로커 한 대만 조회하면 해당 그룹의 모든 오프셋 정보를 얻을 수 있습니다.

이제 오프셋을 __consumer_offsets에 기록하는 방식, 즉 커밋 전략을 살펴봅니다. 크게 자동 커밋과 수동 커밋 두 가지가 있습니다.

**자동 커밋**은 컨슈머가 별도의 코드를 작성하지 않아도 주기적으로 오프셋을 커밋하는 방식입니다. Kafka 컨슈머의 기본 설정이 자동 커밋입니다. enable.auto.commit 설정이 기본값 true로 되어 있고, auto.commit.interval.ms 설정이 기본값 5000, 즉 5초로 되어 있습니다. 컨슈머가 poll()을 호출하여 메시지를 가져올 때, 직전 poll() 이후 5초가 경과했으면 그 시점까지 읽은 오프셋을 자동으로 커밋합니다.

자동 커밋은 설정이 간편합니다. 그러나 문제가 될 수 있는 상황이 있습니다. 컨슈머가 poll()로 메시지를 가져온 뒤, 아직 처리가 끝나지 않은 상태에서 자동 커밋이 발생할 수 있습니다. 이때 컨슈머가 장애로 종료되면, 커밋은 되었지만 처리는 완료되지 않은 메시지가 생깁니다. 반대로, 메시지를 처리한 뒤 자동 커밋이 발생하기 전에 컨슈머가 종료되면, 이미 처리한 메시지를 다시 읽게 됩니다. 자동 커밋은 '정확히 한 번 처리'를 보장하지 않으며, 중복 처리가 발생할 수 있는 at-least-once 방식입니다.

**수동 커밋**은 애플리케이션 코드에서 명시적으로 오프셋을 커밋하는 방식입니다. enable.auto.commit을 false로 설정한 뒤, 메시지 처리가 완료된 시점에 직접 커밋 메서드를 호출합니다. 수동 커밋에는 두 가지 방법이 있습니다.

첫 번째는 commitSync()입니다. 이 메서드는 브로커가 커밋 완료 응답을 보낼 때까지 대기합니다. 응답을 받아야 다음 코드로 넘어가므로 커밋 성공 여부가 확실합니다. 대신 응답을 기다리는 동안 컨슈머가 다른 일을 하지 못하므로 처리량이 낮아집니다.

두 번째는 commitAsync()입니다. 이 메서드는 커밋 요청을 보낸 뒤 응답을 기다리지 않고 바로 반환합니다. 대기 시간이 없으므로 처리량이 높습니다. 그러나 커밋이 실패했을 때 자동 재시도가 되지 않으므로, 필요하면 콜백을 등록하여 실패를 감지하고 직접 재시도 로직을 작성해야 합니다.

두 방식의 차이를 정리합니다.

```
  commitSync()  vs  commitAsync()

  commitSync()
    컨슈머 -> 커밋 요청 -> 브로커
    컨슈머 <- 커밋 응답 <- 브로커  (응답 올 때까지 대기)
    -> 안정적이지만 느림

  commitAsync()
    컨슈머 -> 커밋 요청 -> 브로커
    컨슈머는 즉시 다음 작업 수행  (응답을 기다리지 않음)
    -> 빠르지만 실패 시 직접 처리 필요
```

수동 커밋을 사용하면 메시지 처리가 완전히 끝난 뒤에만 오프셋을 커밋할 수 있습니다. 처리 완료와 커밋 시점을 애플리케이션이 직접 제어하므로, 자동 커밋보다 정밀한 오프셋 관리가 가능합니다.

마지막으로, 컨슈머가 처음 시작되었을 때 읽기 시작 위치를 결정하는 설정인 auto.offset.reset을 다룹니다. 이 설정은 컨슈머 그룹에 아직 커밋된 오프셋이 없거나, 커밋된 오프셋이 파티션의 유효 범위를 벗어났을 때만 동작합니다. 이미 유효한 오프셋이 __consumer_offsets에 커밋되어 있으면 이 설정은 무시됩니다.

**earliest**로 설정하면 파티션의 가장 오래된 메시지, 즉 오프셋 0부터 읽기 시작합니다. 파티션에 이미 쌓여 있는 과거 메시지를 모두 처리해야 하는 경우에 적합합니다. 예를 들어 데이터 분석 시스템이 처음 가동될 때, 기존 이벤트를 빠짐없이 수집해야 한다면 earliest를 사용합니다.

**latest**는 기본값입니다. 컨슈머가 시작된 이후에 새로 들어오는 메시지부터 읽습니다. 과거에 쌓인 메시지는 무시합니다. 실시간 모니터링처럼 현재 시점의 데이터만 중요한 경우에 적합합니다.

none으로 설정하면 커밋된 오프셋이 없을 때 예외를 발생시킵니다. 애플리케이션이 오프셋 부재 상황을 직접 처리해야 합니다.

흔한 오해를 짚어 둡니다. auto.offset.reset은 이미 커밋된 오프셋을 초기화하지 않습니다. 컨슈머 그룹이 한 번이라도 오프셋을 커밋한 뒤에는, 이 설정을 earliest로 바꾸어도 과거 메시지를 다시 읽지 않습니다. 저장된 커밋 오프셋이 우선합니다.

```
  auto.offset.reset 동작 조건

  커밋된 오프셋이 있는가?
    +-- 예 -> 커밋된 오프셋 위치부터 읽음 (auto.offset.reset 무시)
    +-- 아니오 -> auto.offset.reset 설정에 따라 결정
                    +-- earliest: 파티션 처음(오프셋 0)부터
                    +-- latest: 현재 끝(새 메시지)부터
                    +-- none: 예외 발생
```

정리하면, 오프셋은 파티션 내 메시지 위치를 가리키는 단조 증가 정수이며, 컨슈머가 어디까지 읽었는지를 추적하는 핵심 수단입니다. 커밋된 오프셋은 __consumer_offsets 내부 토픽에 영속적으로 저장됩니다. 자동 커밋은 편리하지만 중복 처리 가능성이 있고, 수동 커밋은 정밀한 제어가 가능하지만 코드를 직접 작성해야 합니다. auto.offset.reset은 커밋된 오프셋이 없을 때만 동작하며, earliest는 처음부터, latest는 최신 메시지부터 읽기를 시작합니다.

다음 단원인 1.3.4에서는 컨슈머 그룹을 다룹니다. 여러 컨슈머가 하나의 그룹으로 묶여 파티션을 나누어 읽는 구조와, 그룹 내 멤버 변동 시 파티션을 재분배하는 과정을 살펴봅니다.

이 단원을 마치면 Kafka 오프셋의 역할과 컨슈머가 메시지 읽기 위치를 추적하는 메커니즘을 설명할 수 있습니다.
