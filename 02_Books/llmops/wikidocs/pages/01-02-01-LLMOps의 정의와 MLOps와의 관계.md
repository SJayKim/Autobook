# 1.2.1 LLMOps의 정의와 MLOps와의 관계

1.1.1에서 MLOps가 머신러닝 모델의 전체 생명주기를 체계적으로 관리하는 원칙과 프로세스의 집합이라는 점을 살펴보았습니다. MLOps의 빌드 루프는 데이터 수집, 피처 엔지니어링, 훈련, 평가, 배포, 드리프트 감지, 재훈련이 끊임없이 반복되는 순환 구조였습니다. 그런데 최근 몇 년 사이에 머신러닝의 지형이 크게 바뀌었습니다. GPT, Claude, LLaMA 같은 대형 언어 모델이 등장하면서, 모델을 처음부터 훈련하지 않고 이미 사전 훈련된 모델을 가져다 쓰는 방식이 보편화되었습니다. 이 변화는 운영 방식에도 근본적인 차이를 만들어 냅니다.

전통적 ML 시스템에서는 조직이 자체 데이터를 모아 모델을 직접 훈련합니다. 수개월에 걸쳐 데이터를 준비하고, 피처를 설계하고, 알고리즘을 선택하고, 하이퍼파라미터를 조정합니다. 반면 대형 언어 모델을 활용하는 시스템에서는 이미 방대한 텍스트 데이터로 사전 훈련된 모델을 API로 호출합니다. 조직이 하는 일은 모델 자체를 만드는 것이 아니라, 그 모델에 적절한 지시를 내리고, 필요하면 특정 도메인에 맞게 미세 조정하고, 외부 지식을 검색해서 모델에 제공하는 것입니다. 훈련이 아니라 추론이 운영의 중심이 됩니다.

이 차이 때문에 기존 MLOps의 프로세스를 그대로 적용하기 어려운 지점이 생깁니다. 데이터 드리프트를 감지해서 재훈련을 트리거하는 MLOps의 핵심 루프가, 사전 훈련된 모델을 API로 호출하는 시스템에서는 작동하지 않습니다. 모델 가중치를 직접 관리하지 않으니 버전 관리의 대상도 달라집니다. 비용 구조도 뒤집어집니다. 이런 새로운 운영 과제를 다루기 위해 등장한 분야가 **LLMOps**입니다.

**LLMOps**는 Large Language Model Operations의 줄임말입니다. 대형 언어 모델을 프로덕션 환경에서 관리하고, 확장하고, 최적화하는 운영 체계를 뜻합니다. MLOps의 원칙을 LLM 고유의 특성에 맞게 확장한 분야입니다. 여기서 말하는 LLM 고유의 특성이란, 토큰 단위로 텍스트를 생성하는 추론 방식, 프롬프트를 중심으로 반복 개선하는 개발 루프, 그리고 모델이 사실이 아닌 내용을 자신 있게 생성하는 환각 리스크를 가리킵니다. LLMOps는 모델 선택, 추론 지연 최적화, 출력 모니터링, 프롬프트 일관성 보장, 토큰 비용 거버넌스, 민감 데이터 보안까지 포괄합니다.

그렇다면 MLOps와 LLMOps는 어떤 관계일까요? 두 분야는 공통 목표를 공유합니다. AI 모델을 신뢰성 있고 확장 가능하게 운영한다는 것입니다. 재현성을 위한 버전 관리와 메타데이터 추적, 성능을 지속적으로 확인하기 위한 모니터링과 피드백 루프, 배포를 자동화하기 위한 CI/CD 파이프라인, 데이터 과학자와 엔지니어와 운영 팀이 함께 일하는 협업 워크플로우가 양쪽 모두에 존재합니다.

그러나 관리 대상 모델의 특성이 다르기 때문에, 워크플로우의 세부 사항은 근본적으로 달라집니다. 이 차이를 구체적으로 비교하면 다음과 같습니다.

| 항목 | MLOps | LLMOps |
|------|-------|--------|
| 모델 유형 | 구조화 데이터를 학습한 소형 모델 | 대형 사전 훈련 언어 모델 |
| 빌드 루프 | 느린 오프라인 훈련(수집, 훈련, 평가, 배포) | 빠른 반복(프롬프트 수정, RAG, 가드레일 조정) |
| 버전 관리 대상 | 모델 가중치, 데이터셋, 코드 | 프롬프트 템플릿, 임베딩, 벡터 스토어, 가드레일 설정 |
| 비용 구조 | 훈련 비용이 지배적, 추론 비용은 낮음 | 추론 비용이 지배적, 토큰 단위 과금 |
| 모니터링 지표 | 정확도, 정밀도, 데이터 드리프트 | 환각률, 프롬프트 효과성, 요청당 비용 |
| 개선 방법 | 새로운 데이터로 재훈련 | 프롬프트 재작성, 검색 콘텐츠 업데이트, 출력 리랭킹 |

가장 눈에 띄는 차이는 빌드 루프의 속도와 방향입니다. MLOps에서 모델을 개선하려면 새 데이터를 수집하고, 피처를 설계하고, 모델을 재훈련해야 합니다. 이 과정은 수일에서 수주가 걸립니다. LLMOps에서는 프롬프트 문구를 수정하거나, 검색 증강 생성(RAG)에 사용하는 외부 문서를 업데이트하거나, 가드레일 규칙을 조정하는 것으로 빠르게 개선을 시도합니다. 모델 재훈련은 최후의 수단입니다. 이 때문에 LLMOps의 빌드 루프에서는 '프롬프트가 핵심이다'라는 표현이 나옵니다.

MLOps와 LLMOps는 대체 관계가 아니라 상보 관계입니다. 하나의 시스템 안에 전통적 ML 모델과 LLM이 함께 존재할 수 있습니다. 예를 들어 추천 시스템에서 사용자 행동 데이터를 학습한 ML 모델이 아이템을 추천하고, LLM이 그 추천 결과를 자연어로 설명하는 구조를 생각할 수 있습니다. 이때 추천 모델에는 MLOps의 재훈련 루프가 적용되고, 설명 LLM에는 LLMOps의 프롬프트 관리와 환각 감지가 적용됩니다. 두 체계를 병행하는 것이 가장 효과적입니다.

이제 LLMOps가 다루는 고유한 운영 과제를 하나씩 살펴봅니다.

첫 번째 과제는 **추론 비용**입니다. MLOps에서는 모델 훈련에 가장 많은 비용이 듭니다. 모델을 한 번 훈련해 두면, 추론 요청 하나를 처리하는 데 드는 비용은 상대적으로 작습니다. LLMOps에서는 이 구조가 뒤집어집니다. 사전 훈련된 모델을 가져다 쓰기 때문에 훈련 비용은 대부분 모델 제공자가 부담합니다. 대신 매 요청마다 토큰 단위로 과금되는 추론 비용이 운영비의 핵심이 됩니다.

LLM이 텍스트를 생성하는 방식을 이해하면 왜 추론 비용이 예측하기 어려운지 알 수 있습니다. LLM은 텍스트를 한 번에 만들어 내는 것이 아니라, 토큰을 하나씩 순서대로 생성합니다. 단순한 질문에는 50개 토큰으로 답할 수 있지만, 복잡한 분석 요청에는 500개 이상의 토큰이 필요할 수 있습니다. 처리가 완료되기 전까지는 응답 길이를 정확히 예측할 수 없으므로, 인프라 자원과 비용을 미리 산정하기 어렵습니다. 여기에 임베딩 생성, 벡터 검색, 외부 도구 호출 비용까지 더해지면 요청 하나의 총비용은 더 복잡해집니다. 따라서 요청당 비용 추적은 프로덕션 LLMOps의 핵심 모니터링 지표가 됩니다.

추론 비용을 줄이기 위해 실무에서 사용하는 기법은 크게 세 가지입니다.

첫째는 **지능형 모델 라우팅**입니다. 모든 요청을 가장 비싼 모델로 보내는 대신, 쿼리의 복잡도에 따라 적절한 모델로 분기합니다. 단순한 질문은 저렴하고 가벼운 모델이 처리하고, 복잡한 추론이 필요한 질문만 고성능 모델로 보냅니다. 이렇게 하면 평균 추론 비용이 낮아지면서도 응답 품질은 유지됩니다.

둘째는 **시맨틱 캐싱**입니다. 이전에 처리한 질문과 의미가 유사한 질문이 다시 들어오면, LLM을 호출하지 않고 캐시에 저장된 이전 응답을 반환합니다. 텍스트를 벡터 임베딩으로 변환하여 의미적 유사도를 비교하는 방식입니다. 반복되는 질문이 많은 워크로드에서는 캐시 히트율이 60%에서 85%에 이를 수 있으며, 비용은 최대 68.8%까지, 응답 지연은 96.9%까지 줄어든다는 보고가 있습니다.

셋째는 **배치 처리 최적화**입니다. 추론 요청을 하나씩 처리하는 대신 여러 요청을 묶어서 한꺼번에 처리하면 GPU 활용 효율이 높아집니다. 고정 크기로 묶는 정적 배치, 각 반복 단계에서 완료된 요청을 빼고 새 요청을 넣는 연속 배치, 비슷한 길이의 시퀀스를 묶는 멀티-빈 배치 등의 방식이 있습니다.

```
요청 도착
  │
  ├─ 시맨틱 캐시 히트? ──→ 예 ──→ 캐시 응답 반환 (LLM 호출 없음)
  │
  └─ 아니오
       │
       ├─ 모델 라우터: 복잡도 판별
       │     │
       │     ├─ 단순 ──→ 경량 모델 추론
       │     └─ 복잡 ──→ 고성능 모델 추론
       │
       └─ 배치 스케줄러: 유사 요청 묶음 처리
```

두 번째 과제는 **프롬프트 관리**입니다. 전통적 ML에서 모델의 동작을 바꾸려면 데이터를 바꾸거나 모델을 재훈련해야 합니다. LLM에서는 프롬프트, 즉 모델에게 내리는 지시문을 바꾸는 것만으로 동작이 크게 달라집니다. 프롬프트 한 줄의 변경이 출력의 형식, 정확도, 톤을 완전히 바꿀 수 있습니다. 그래서 LLMOps에서 프롬프트는 코드와 동등한 지위를 가집니다.

프롬프트를 관리한다는 것은 여러 가지를 포함합니다. 프롬프트 템플릿의 **버전 관리**가 필요합니다. 어떤 프롬프트가 어떤 결과를 냈는지 추적할 수 있어야 합니다. 새로운 프롬프트가 기존보다 나은지 확인하기 위한 **A/B 테스트**가 필요합니다. 프로덕션에서 프롬프트가 의도대로 작동하는지 **모니터링**해야 합니다. 프롬프트 변경 하나가 예기치 않은 품질 저하를 일으키지 않는지 확인하기 위해, 15개에서 20개의 대표적인 입력-출력 쌍으로 구성한 **골든 데이터셋**을 두고 매 변경마다 회귀 테스트를 실행하는 것이 모범 사례로 꼽힙니다. 프롬프트 관리 시스템은 모델 관리 레지스트리와 동급의 인프라 구성요소로 취급됩니다.

세 번째 과제는 **환각**입니다. **환각**(hallucination)은 LLM이 사실적으로 틀리거나 근거 없는 내용을 자신감 있게 생성하는 현상입니다. 전통적 ML에서는 모델 출력이 수치이므로 정확도나 오차율로 품질을 측정할 수 있습니다. 그러나 LLM은 자연어 텍스트를 생성하므로, 출력이 사실에 부합하는지를 검증하는 별도의 체계가 필요합니다. 이것이 LLMOps 고유의 모니터링 과제입니다.

환각에는 두 가지 유형이 있습니다. 하나는 **모순**으로, 모델의 답변이 제공된 맥락 정보와 직접 충돌하는 경우입니다. 다른 하나는 **미지원 주장**으로, 맥락 정보에 근거가 없는 내용을 모델이 스스로 만들어 내는 경우입니다. 특히 검색 증강 생성(RAG) 환경에서는 검색된 문서와 모델 응답 사이의 일치도를 '충실성(faithfulness)'이라는 지표로 측정합니다.

환각을 감지하는 방법은 모델 내부에 접근할 수 있는지에 따라 나뉩니다. 모델 내부의 토큰 확률이나 어텐션 패턴에 접근할 수 있으면 화이트박스 방식을 쓸 수 있습니다. 그러나 대부분의 프로덕션 환경에서는 API를 통해 모델을 호출하므로 내부에 접근할 수 없습니다. 이때는 블랙박스 방식을 사용합니다. 대표적인 블랙박스 방식은 별도의 판별 모델이 생성 모델의 출력을 평가하는 LLM-as-a-judge입니다. 평가 모델이 미리 정의된 기준(루브릭)에 따라 응답의 충실성을 점수로 매기는 방식입니다.

환각을 줄이기 위한 완화 전략도 여러 가지입니다. 프롬프트에 맥락 정보를 명시적으로 제공하여 모델이 근거 없는 생성을 줄이도록 유도하는 방법, RAG를 통해 외부 지식을 주입하는 방법, 특정 도메인 데이터로 모델을 파인튜닝하는 방법, 그리고 중요한 출력에 사람이 직접 검토하는 인간-루프 검증 방법이 있습니다.

이제 LLMOps의 전체 운영 구조를 살펴봅니다. LLM 기반 시스템을 프로덕션에서 운영하려면, 모델 자체보다 모델을 둘러싼 인프라 설계가 더 중요합니다. 이 인프라는 여섯 가지 기능 영역으로 나눌 수 있습니다.

첫 번째는 **데이터 및 임베딩 영역**입니다. 원시 데이터를 정제하고, 적절한 크기로 분할(청킹)하고, 벡터 임베딩으로 변환하여 벡터 데이터베이스에 저장합니다. 검색 증강 생성(RAG)의 기반이 되는 영역입니다.

두 번째는 **모델 관리 영역**입니다. 어떤 파운데이션 모델을 사용할지 선택하고, 모델 버전을 관리하고, 필요한 경우 LoRA나 QLoRA 같은 경량 파인튜닝을 수행합니다.

세 번째는 **프롬프트 및 오케스트레이션 영역**입니다. 프롬프트 템플릿의 버전 관리, 여러 단계로 이루어진 체인이나 에이전트 플로우의 구성, RAG 파이프라인의 조율, 외부 도구 호출 로직을 담당합니다. 전통적 MLOps 스택에는 존재하지 않는 LLMOps 고유의 영역입니다.

네 번째는 **추론 및 서빙 영역**입니다. GPU 기반 추론 서버 운영, 모델 라우팅, 시맨틱 캐싱, 로드밸런싱, API 게이트웨이 관리를 포함합니다. 토큰 단위 비용 관리가 이 영역의 핵심입니다.

다섯 번째는 **평가 및 안전 영역**입니다. 골든 데이터셋을 활용한 회귀 테스트, LLM-as-a-judge 방식의 자동 평가, 환각 감지, 유해 콘텐츠 필터, 가드레일 설정을 다룹니다. 이 영역 역시 전통적 MLOps에는 없는 LLMOps 고유의 영역입니다.

여섯 번째는 **모니터링 및 관찰가능성 영역**입니다. 실시간 응답 지연, 토큰 비용, 품질 지표를 추적하고, 성능 변화를 감지하고, A/B 테스트 결과를 분석하고, 문제가 발생하면 자동으로 알림을 보냅니다.

이 여섯 영역을 한눈에 보면 다음과 같습니다.

```
[1. 데이터/임베딩] ──→ [2. 모델 관리] ──→ [3. 프롬프트/오케스트레이션]
                                                     │
                                                     ▼
[6. 모니터링/관찰가능성] ←── [5. 평가/안전] ←── [4. 추론/서빙]
         │
         └──→ 피드백 루프: 프롬프트 수정, 데이터 업데이트, 가드레일 조정
```

주목할 점은 레이어 3(프롬프트 및 오케스트레이션)과 레이어 5(평가 및 안전)가 전통적 MLOps 스택에는 없는 영역이라는 것입니다. MLOps에서는 모델을 재훈련하는 것이 개선의 주된 수단이었기 때문에, 프롬프트 관리나 환각 감지 같은 기능이 필요하지 않았습니다. LLMOps에서는 이 두 영역이 시스템 품질과 안전성을 좌우하는 핵심 계층이 됩니다.

정리하면, LLMOps는 대형 언어 모델을 프로덕션에서 운영하기 위한 체계로, MLOps의 원칙을 공유하면서도 LLM 고유의 과제에 대응합니다. 훈련 중심이 아닌 추론 중심의 비용 구조, 코드와 동등한 지위를 가진 프롬프트의 체계적 관리, 전통적 정확도 지표로는 잡을 수 없는 환각의 감지와 완화, 그리고 이 모든 것을 아우르는 여섯 가지 기능 영역의 운영 스택이 LLMOps를 MLOps와 구분 짓는 핵심입니다. 두 체계는 대체 관계가 아니라, 하나의 조직 안에서 각각의 모델 유형에 맞게 병행하는 상보 관계입니다.

다음 단원인 1.2.2에서는 프롬프트 엔지니어링을 다룹니다. LLMOps에서 프롬프트가 왜 코드와 동등한 지위를 갖는지, 효과적인 프롬프트를 설계하고 반복 개선하는 구체적인 방법을 살펴봅니다.

이 단원을 마치면 LLMOps가 MLOps와 어떻게 다르고 어떤 고유한 운영 과제를 다루는지 설명할 수 있습니다.
