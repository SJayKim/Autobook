# 1.1.4 모델 서빙 아키텍처 개요

1.1.1에서 ML 생명주기의 일곱 번째 단계로 모델 서빙을 언급했습니다. 검증을 통과한 모델을 프로덕션 환경에 배포하여 실제 요청에 응답하게 만드는 단계입니다. 그런데 '응답하게 만든다'는 말 안에는 여러 가지 선택이 숨어 있습니다. 요청을 어떤 프로토콜로 받을 것인지, 요청을 하나씩 처리할 것인지 묶어서 처리할 것인지, 응답을 한 번에 보낼 것인지 조금씩 흘려보낼 것인지에 따라 시스템의 성격이 크게 달라집니다. 이 단원에서는 모델 서빙에서 마주치는 주요 방식들을 하나씩 살펴보고, 각각이 어떤 상황에 적합한지 비교합니다.

모델 서빙의 첫 번째 갈림길은 클라이언트와 서버가 데이터를 주고받는 통신 규약, 즉 프로토콜입니다. 현재 가장 널리 쓰이는 두 가지 프로토콜은 REST API와 gRPC입니다.

**REST API 서빙**부터 살펴봅니다. REST API는 웹에서 가장 보편적인 통신 방식입니다. 클라이언트가 HTTP 프로토콜 위에서 JSON 형식의 데이터를 보내면, 서버가 처리 결과를 역시 JSON으로 돌려줍니다. 웹 브라우저에서 주소를 입력하면 페이지가 뜨는 것과 같은 HTTP 프로토콜을 사용하므로, 별도의 특수 도구 없이도 어떤 프로그래밍 언어에서든 호출할 수 있습니다.

모델 서빙에서 REST API가 동작하는 방식은 단순합니다. 클라이언트가 서버의 특정 주소, 예를 들어 POST /v1/completions 같은 엔드포인트로 입력 텍스트를 JSON에 담아 보냅니다. 서버는 모델에 입력을 넣고 전체 응답을 생성한 뒤, 결과를 하나의 JSON 객체로 반환합니다. 현재 사실상의 표준 인터페이스는 OpenAI Chat Completions API 호환 포맷이며, vLLM, TGI, LMDeploy 등 대부분의 서빙 엔진이 이 인터페이스를 지원합니다.

```
클라이언트                          서버
   │                                │
   │── POST /v1/completions ──────>│
   │   (JSON: 입력 텍스트)          │
   │                                │── 모델 추론 수행
   │                                │
   │<── HTTP 200 ──────────────────│
   │   (JSON: 전체 응답 텍스트)     │
```

REST API의 가장 큰 장점은 접근성입니다. HTTP를 지원하는 모든 환경에서 호출할 수 있고, 기업 네트워크의 방화벽이나 프록시를 무리 없이 통과합니다. JSON은 텍스트 기반이므로 사람이 직접 읽고 디버깅하기도 편리합니다. 반면 단점도 있습니다. JSON은 텍스트로 데이터를 표현하기 때문에 직렬화와 역직렬화에 오버헤드가 발생합니다. 직렬화란 프로그램 내부의 데이터 구조를 전송 가능한 형태로 변환하는 과정이고, 역직렬화는 그 반대 과정입니다. JSON의 경우 숫자도 문자열로 바꿔 보내므로, 데이터 크기가 커지고 변환에 시간이 걸립니다. 또한 REST API는 기본적으로 요청 하나에 응답 하나를 돌려주는 단방향 구조이므로, 응답을 조금씩 흘려보내는 스트리밍을 하려면 별도의 기법이 필요합니다.

**gRPC 서빙**은 REST API와 다른 접근을 취합니다. gRPC를 이해하려면 먼저 두 가지 기술을 알아야 합니다. 첫째는 **Protocol Buffers**(줄여서 protobuf)입니다. protobuf는 데이터를 바이너리, 즉 0과 1의 이진수 형태로 변환하는 직렬화 방식입니다. JSON이 텍스트 기반인 것과 달리 바이너리로 변환하므로 데이터 크기가 작고 변환 속도가 빠릅니다. 둘째는 **HTTP/2**입니다. 기존 HTTP/1.1에서는 하나의 연결에서 요청과 응답이 순서대로 하나씩 처리됩니다. HTTP/2는 하나의 연결 위에서 여러 요청과 응답을 동시에 주고받을 수 있습니다. 이를 멀티플렉싱이라 부릅니다.

gRPC는 이 두 기술을 결합한 RPC 프레임워크입니다. RPC는 Remote Procedure Call의 약자로, 원격 서버에 있는 함수를 마치 로컬 함수처럼 호출하는 방식을 뜻합니다. gRPC에서는 서비스의 메서드와 메시지 타입을 IDL(Interface Definition Language)이라는 명세 언어로 미리 정의합니다. 이 정의 파일을 바탕으로 Java, Go, Python, C++ 등 여러 언어의 클라이언트와 서버 코드를 자동 생성할 수 있어 타입 안전성이 보장됩니다. 타입 안전성이란 주고받는 데이터의 형식이 사전에 합의되어 있어, 잘못된 형식의 데이터가 전달되면 컴파일이나 전송 시점에 오류를 잡아낸다는 뜻입니다.

gRPC의 장점은 바이너리 직렬화로 인한 낮은 네트워크 오버헤드, HTTP/2 멀티플렉싱으로 인한 연결 재사용, 그리고 스트리밍의 내장 지원입니다. gRPC는 서버 스트리밍, 클라이언트 스트리밍, 양방향 스트리밍을 프로토콜 수준에서 기본 지원합니다. 이 때문에 마이크로서비스 간 내부 통신처럼 저지연과 고처리량이 요구되는 환경에 적합합니다. Triton Inference Server, Seldon Core, TensorFlow Serving 등 프로덕션 서빙 시스템이 gRPC 엔드포인트를 제공합니다. 예를 들어 NVIDIA Triton Inference Server는 포트 8000에 REST, 포트 8001에 gRPC를 동시에 노출하여 클라이언트 유형에 따라 선택하게 합니다.

반면 gRPC의 단점은 접근성이 REST보다 제한적이라는 점입니다. 웹 브라우저에서 gRPC를 직접 호출할 수 없어 gRPC-Web이라는 별도 레이어가 필요합니다. protobuf 스키마를 관리해야 하는 비용이 추가되고, 바이너리 데이터라 JSON처럼 눈으로 직접 읽을 수 없어 디버깅 도구 생태계가 REST보다 작습니다.

두 프로토콜의 차이를 정리하면 다음과 같습니다.

```
                REST API                    gRPC
데이터 형식     JSON (텍스트)               protobuf (바이너리)
전송 프로토콜   HTTP/1.1 또는 HTTP/2        HTTP/2
직렬화 속도     느림 (텍스트 파싱)          빠름 (바이너리)
데이터 크기     큼                          작음
스트리밍        SSE로 추가 구현 필요        내장 지원
브라우저 호출   가능                        불가 (gRPC-Web 필요)
디버깅 편의     높음 (사람이 읽기 쉬움)     낮음 (바이너리)
적합한 환경     외부 공개 API, 범용 클라이언트   내부 서비스 간 통신, 고성능 요구
```

프로토콜을 정했다면, 다음으로 결정해야 할 것은 요청을 처리하는 시점과 방식입니다. 이 기준으로 온라인 추론과 오프라인 추론이 나뉩니다.

**온라인 추론**은 요청이 들어오는 즉시 처리하는 방식입니다. 사용자가 챗봇에 질문을 입력하면 곧바로 답변이 생성되어야 합니다. 코드 자동완성 기능도 타이핑하는 동안 실시간으로 제안이 나와야 합니다. 이처럼 사용자가 기다리고 있는 상황에서는 응답이 도착하기까지의 시간, 즉 지연시간이 핵심 성능 기준입니다.

온라인 추론에는 두 가지 방식이 있습니다. 첫째는 **원격 모델 추론**입니다. 모델이 별도의 API 서버에 배포되어 있고, 애플리케이션이 네트워크를 통해 이 서버에 요청을 보냅니다. 모델을 중앙에서 관리할 수 있고 클라우드 환경에서 수평 확장이 용이하다는 장점이 있습니다. 대신 네트워크 지연이 추가되고, 서버의 가용성에 의존하게 됩니다. 둘째는 **임베딩 모델 추론**입니다. 여기서 '임베딩'은 벡터 임베딩이 아니라, 모델을 애플리케이션 내부에 직접 내장한다는 의미입니다. 예를 들어 스트림 처리 프레임워크인 Flink 안에 모델을 탑재하면 네트워크 왕복 없이 즉시 추론할 수 있어 지연시간이 최소화됩니다. 외부 서비스에 의존하지 않으므로 네트워크가 끊겨도 동작합니다. 하지만 모델 업데이트나 모니터링이 복잡해지는 단점이 있습니다.

**오프라인 추론**은 사전에 준비된 대규모 데이터셋을 정해진 시점에 일괄 처리하는 방식입니다. 배치 추론이라고도 부릅니다. 사용자가 결과를 실시간으로 기다리지 않으므로 지연시간이 아니라 처리량과 비용 효율이 목표입니다. 모델 평가, 대량 문서 처리, 훈련 데이터 레이블링 같은 작업에 적합합니다. 예를 들어 ImageNet 분류 데이터셋 전체에 대한 추론이나, 수만 건의 문서에 대한 임베딩 벡터 생성 같은 작업이 여기에 해당합니다.

```
온라인 추론                              오프라인 추론
---------------------                   ---------------------
사용자 요청 → 즉시 응답                 대규모 데이터셋 → 일괄 처리
핵심 지표: 지연시간                     핵심 지표: 처리량, 비용 효율
예시: 챗봇, 코드 자동완성,              예시: 모델 평가, 데이터 레이블링,
      실시간 분류                              대량 문서 처리
```

이제 요청을 묶어서 처리하는 **배치 추론**의 메커니즘을 좀 더 자세히 살펴봅니다. GPU는 병렬 행렬 연산에 최적화된 하드웨어입니다. 요청 하나를 단독으로 처리하면 GPU의 연산 능력 대부분이 놀게 됩니다. 여러 요청을 하나의 묶음, 즉 배치로 만들어 한 번의 모델 연산에 함께 넣으면 GPU 활용률이 크게 높아집니다.

가장 단순한 형태는 **정적 배치**입니다. 고정된 크기만큼 요청을 모아 한 번에 처리합니다. 그런데 LLM의 경우 각 요청이 생성하는 텍스트 길이가 제각각입니다. 정적 배치에서는 배치 안의 모든 요청이 완료될 때까지 기다려야 합니다. 어떤 요청은 10개 토큰만 생성하고 끝나도, 같은 배치에 200개 토큰을 생성하는 요청이 있으면 그것이 끝날 때까지 GPU가 유휴 상태로 대기합니다. 짧은 요청 입장에서는 불필요한 대기 시간이 발생하고, GPU 입장에서는 연산 자원이 낭비됩니다.

이 문제를 해결하기 위해 등장한 것이 **연속 배치**입니다. 연속 배치는 토큰 생성 단계, 즉 이터레이션 수준에서 스케줄링을 수행합니다. 어떤 요청이 생성을 완료하면 배치 전체가 끝나기를 기다리지 않고, 그 자리에 새로운 요청을 즉시 투입합니다. GPU가 쉬는 시간이 사라지므로 처리량이 크게 향상됩니다. vLLM에서 PagedAttention과 연속 배치를 조합했을 때 정적 배치 대비 최대 23배의 처리량 향상이 보고된 바 있습니다.

또 다른 방식으로 **멀티빈 배치**가 있습니다. 유사한 시퀀스 길이를 가진 요청끼리 묶어 처리합니다. 길이가 비슷한 요청끼리 같은 배치에 넣으면 짧은 요청이 긴 요청을 기다리는 낭비, 즉 패딩 낭비가 줄어듭니다. 이 방식으로 최대 70%의 처리량 향상이 가능합니다.

```
정적 배치:
  요청A(10토큰) ████░░░░░░  ← 완료 후에도 대기
  요청B(50토큰) ██████████  ← 가장 긴 요청이 끝날 때까지 배치 유지
  요청C(20토큰) ██████░░░░  ← 완료 후에도 대기

연속 배치:
  요청A(10토큰) ████ → 요청D 즉시 투입
  요청B(50토큰) ██████████
  요청C(20토큰) ██████ → 요청E 즉시 투입
  (GPU 유휴 시간 제거)
```

대규모 오프라인 배치 추론에서는 CPU 전처리와 GPU 추론을 파이프라인으로 연결하여 GPU 활용률을 90% 이상 달성하기도 합니다. Ray Data와 GPU 클러스터를 활용한 사례에서는 SageMaker Batch Transform 대비 17배 빠른 처리 속도가 보고되었습니다.

다음으로 **스트리밍 추론**을 살펴봅니다. LLM은 토큰을 하나씩 순차적으로 생성합니다. 비스트리밍 방식에서는 모든 토큰이 생성된 후에야 전체 응답이 클라이언트에 전달됩니다. 긴 응답의 경우 사용자가 수 초에서 수십 초 동안 빈 화면을 보며 기다려야 합니다. 스트리밍 추론은 토큰이 생성되는 즉시 클라이언트로 전달하여 첫 토큰부터 화면에 표시합니다. 전체 응답 시간은 같더라도 사용자가 체감하는 응답 속도는 크게 다릅니다.

REST API 환경에서 스트리밍을 구현하는 표준 메커니즘은 **SSE**(Server-Sent Events)입니다. 클라이언트가 요청 시 stream=True 파라미터를 설정하면, 서버는 응답을 한 번에 보내지 않고 data-only server-sent events 포맷으로 토큰 청크를 연속 전송합니다. 각 청크는 delta 필드에 부분 텍스트를 담고 있습니다. 마지막 청크는 빈 delta를 보내 전송 종료를 알립니다. 실제 측정에서 첫 토큰은 약 0.1초 후에 도착하고, 이후 토큰은 0.01초에서 0.02초 간격으로 수신됩니다.

gRPC 환경에서는 HTTP/2 서버 스트리밍 RPC를 통해 토큰을 전달합니다. SSE와 달리 HTTP 헤더를 반복 전송하지 않고 연속 바이너리 스트림을 보내므로 전송 효율이 높습니다.

여기서 스트리밍과 관련하여 중요한 지표가 하나 있습니다. **TTFT**(Time to First Token)는 사용자가 요청을 보낸 시점부터 첫 번째 토큰을 받기까지 걸리는 시간입니다. 스트리밍 시나리오에서 사용자가 가장 먼저 체감하는 응답성 지표입니다. vLLM 기반 서빙 시스템에서는 이 값을 vllm:time_to_first_token_seconds 메트릭으로 추적하며, p99 값이 5초를 초과하면 사용자 경험 SLO 위반으로 판단하여 알림을 발생시킵니다.

지금까지 살펴본 서빙 방식들은 서로 독립된 선택이 아닙니다. REST API 위에서 스트리밍 추론을 하기도 하고, gRPC 위에서 배치 추론을 하기도 합니다. 실제 프로덕션 시스템에서는 이 방식들을 조합하여 사용합니다. 이때 핵심적인 트레이드오프가 **지연시간과 처리량** 사이에 존재합니다.

**지연시간**은 단일 요청의 응답 시간입니다. LLM에서는 앞서 설명한 TTFT와 함께, 토큰과 토큰 사이의 간격인 **TBT**(Time Between Tokens)로 분리 측정합니다. TTFT가 첫 토큰까지의 대기 시간을 나타낸다면, TBT는 이후 토큰이 얼마나 균일한 간격으로 도착하는지를 나타냅니다. 두 지표 모두 낮을수록 사용자가 체감하는 응답이 빠릅니다.

이 지연시간과 대비되는 지표가 **처리량**입니다. 처리량은 단위 시간당 처리할 수 있는 토큰 수 또는 요청 수로, 주로 tokens/sec로 측정합니다.

근본적인 트레이드오프는 다음과 같습니다. 배치 크기를 늘리면 GPU를 더 효율적으로 사용하여 처리량이 증가합니다. 그러나 배치 안의 개별 요청은 다른 요청과 함께 처리되기를 기다려야 하므로 지연시간이 길어집니다. 반대로 배치 크기를 줄이면 개별 요청의 대기 시간은 줄어들지만, GPU 활용률이 낮아져 처리량이 감소합니다. LLM 추론은 메모리 입출력에 병목이 걸리는 작업이므로, 배치 크기 최적화가 일반적인 CPU 바운드 서비스보다 훨씬 큰 영향을 미칩니다.

앞서 설명한 연속 배치는 이 트레이드오프를 완화하는 특수한 경우입니다. 완료된 요청을 즉시 교체하여 GPU 유휴 시간을 줄이므로, 처리량과 중간값 지연시간을 동시에 개선합니다. 다만 시스템이 포화 상태, 즉 초당 요청 수가 매우 높은 상황에서는 지연시간 개선 효과가 감소합니다.

이 트레이드오프를 다른 방향에서 개선하려는 기법도 있습니다. **투기적 디코딩**이 대표적입니다. LLM이 토큰을 하나씩 생성할 때, GPU는 한 토큰을 생성하는 데 전체 연산 능력의 일부만 사용하고 나머지는 유휴 상태가 됩니다. 투기적 디코딩은 이 유휴 자원을 활용합니다. 작고 빠른 드래프트 모델이 먼저 5개에서 8개의 토큰을 '추측'하여 생성합니다. 그 다음 원래의 대형 모델이 이 토큰들을 한 번의 포워드 패스로 병렬 검증합니다. 추측이 맞은 토큰은 그대로 확정되고, 틀린 토큰부터는 대형 모델이 다시 생성합니다. 한 번의 연산으로 여러 토큰을 확정할 수 있어 지연시간이 2배에서 3배 감소합니다. vLLM에서 Llama 3.1-70B 모델에 1B 크기의 드래프트 모델을 조합했을 때 2.31배의 속도 향상이 측정되었습니다. 단, GPU가 이미 배치 처리로 포화된 상황에서는 활용할 유휴 자원이 없으므로 효과가 제한적입니다.

실제 프로덕션 환경에서는 이러한 트레이드오프를 수치 기준으로 관리합니다. vLLM을 Kubernetes 위에서 운용하는 사례에서는 대기 중인 요청이 3개에서 5개에 이르면 서버를 추가로 띄우고, KV 캐시 활용률이 85%를 초과하면 메모리 압박 경고를 발생시킵니다. 배치 크기를 최대치 아래로 유지하면서 TTFT p99가 5초를 넘지 않도록 관리하는 것이 일반적인 운영 기준입니다.

정리하면, 모델 서빙 아키텍처는 프로토콜(REST API vs gRPC), 처리 시점(온라인 vs 오프라인), 응답 전달 방식(비스트리밍 vs 스트리밍), 배치 전략(정적 vs 연속 vs 멀티빈)이라는 선택지의 조합입니다. REST API는 범용성과 디버깅 편의가 강점이고, gRPC는 바이너리 직렬화와 내장 스트리밍으로 고성능 내부 통신에 적합합니다. 온라인 추론은 지연시간을, 오프라인 추론은 처리량을 우선합니다. 배치 크기를 늘리면 처리량이 오르지만 지연시간이 늘어나고, 연속 배치와 투기적 디코딩 같은 기법이 이 트레이드오프를 완화합니다. 어떤 조합이 최선인지는 서비스의 요구사항에 따라 달라지며, 이 선택이 LLM 운영 비용과 사용자 경험을 직접 결정합니다.

다음 단원인 1.2.1에서는 LLMOps의 정의와 MLOps와의 관계를 다룹니다. 1.1.1에서 소개한 MLOps의 원칙이 LLM이라는 특수한 모델에 어떻게 확장되고 달라지는지 살펴봅니다.

이 단원을 마치면 REST API, gRPC, 배치 추론, 스트리밍 추론 등 주요 모델 서빙 방식의 특성과 트레이드오프를 비교할 수 있습니다.
