# 1.1.1 MLOps의 정의와 필요성

머신러닝 모델을 만드는 일과, 그 모델을 실제 서비스에서 안정적으로 운영하는 일은 완전히 다른 문제입니다. 데이터 과학자가 Jupyter Notebook에서 높은 정확도를 달성한 모델을 만들었다고 가정합니다. 이 모델을 실제 사용자 요청에 응답하도록 서버에 올리고, 매일 들어오는 새로운 데이터에도 성능이 유지되는지 확인하고, 문제가 생기면 이전 버전으로 되돌리는 일까지 해야 비로소 모델이 '운영'되고 있다고 말할 수 있습니다. 이 전체 과정을 체계적으로 다루는 분야가 **MLOps**입니다.

**MLOps**는 Machine Learning Operations의 줄임말입니다. 머신러닝 모델을 실험 환경에서 꺼내 프로덕션 환경에서 안정적으로 작동하도록 만드는 원칙, 프로세스, 도구의 체계를 뜻합니다. 데이터 과학자, ML 엔지니어, DevOps 팀의 역할을 하나의 흐름으로 통합하여, 모델이 어떻게 훈련되었고 어떤 데이터를 사용했으며 현재 어떤 성능을 보이는지 가시성을 확보하는 것이 목표입니다.

MLOps가 왜 필요한지 이해하려면, 먼저 머신러닝 모델이 만들어지고 운영되기까지의 전체 흐름을 알아야 합니다. 이 흐름을 **ML 생명주기**라고 부릅니다. ML 생명주기는 크게 여덟 단계로 나눌 수 있습니다.

첫 번째는 **데이터 추출**입니다. 데이터베이스, 로그 파일, 외부 API 등 다양한 곳에서 모델 훈련에 필요한 원시 데이터를 가져옵니다. 두 번째는 **데이터 분석**입니다. 탐색적 데이터 분석(EDA)을 통해 데이터의 분포, 결측값, 이상치를 파악합니다. 세 번째는 **데이터 준비**입니다. 데이터를 정제하고, 훈련용과 검증용으로 분할하고, 모델에 넣을 수 있는 형태로 가공하는 피처 엔지니어링을 수행합니다. 네 번째는 **모델 훈련**입니다. 알고리즘을 선택하고 하이퍼파라미터를 조정하며 모델을 학습시킵니다. 다섯 번째는 **모델 평가**입니다. 테스트 데이터셋으로 모델의 성능을 측정합니다. 여섯 번째는 **모델 검증**입니다. 평가 결과가 프로덕션 배포 기준을 충족하는지 확인합니다. 일곱 번째는 **모델 서빙**입니다. 검증을 통과한 모델을 프로덕션 환경에 배포하여 실제 요청에 응답하게 합니다. 여덟 번째는 **모델 모니터링**입니다. 배포된 모델의 성능을 실시간으로 추적하고, 문제가 발견되면 재훈련이나 롤백을 수행합니다.

이 여덟 단계를 그림으로 정리하면 다음과 같습니다.

```
데이터 추출 → 데이터 분석 → 데이터 준비 → 모델 훈련
                                              ↓
모델 모니터링 ← 모델 서빙 ← 모델 검증 ← 모델 평가
     │
     └──→ (성능 저하 감지 시) 데이터 추출로 되돌아감
```

주목할 점은 이 흐름이 한 번으로 끝나지 않는다는 것입니다. 모니터링 단계에서 성능 저하가 감지되면 다시 데이터 추출 단계로 돌아가 새로운 데이터로 모델을 재훈련합니다. 이 순환이 끊기지 않도록 자동화하는 것이 MLOps의 핵심입니다.

이쯤에서 의문이 생길 수 있습니다. 소프트웨어 개발에는 이미 **DevOps**라는 체계가 있습니다. DevOps는 코드를 작성하고, 테스트하고, 배포하는 과정을 자동화합니다. 그렇다면 ML 시스템에도 DevOps를 그대로 적용하면 되지 않을까요? 그렇지 않습니다. ML 시스템에는 DevOps만으로는 다루지 못하는 고유한 특성이 있기 때문입니다.

일반 소프트웨어는 코드가 바뀔 때만 동작이 바뀝니다. 코드를 수정하지 않았다면 어제와 오늘의 출력이 같습니다. 그러나 ML 시스템은 코드가 그대로여도 입력 데이터의 분포가 바뀌면 출력이 달라집니다. 예를 들어, 사용자 행동 패턴을 학습한 추천 모델이 있다고 합시다. 코드를 한 줄도 바꾸지 않았더라도, 새로운 유행이 등장하여 사용자 행동이 달라지면 추천 품질이 떨어집니다. 이처럼 데이터 분포의 변화가 시스템 동작에 직접 영향을 미치는 점이 ML 시스템의 근본적인 차이입니다.

이 차이를 구체적으로 비교하면 다음과 같습니다. 버전 관리 대상을 보면, DevOps는 소스코드를 관리합니다. MLOps는 소스코드에 더해 모델 가중치, 데이터셋, 하이퍼파라미터 설정까지 함께 관리합니다. 테스트 기준을 보면, DevOps는 단위 테스트와 통합 테스트로 코드의 정확성을 검증합니다. MLOps는 여기에 정확도, F1 점수 같은 모델 성능 지표와 데이터 드리프트 검사를 추가합니다. 배포 후 관심사를 보면, DevOps는 서버 가용성과 응답 시간을 모니터링합니다. MLOps는 이에 더해 모델 성능 저하와 데이터 분포 변화를 추가로 추적합니다. 실패 원인을 보면, DevOps에서는 코드 버그가 주된 실패 원인입니다. MLOps에서는 코드 버그 외에 데이터 품질 저하와 개념 드리프트도 실패를 일으킵니다.

DevOps의 빌드 루프는 '코드 작성, 테스트, 배포'라는 선형 구조입니다. 반면 MLOps의 빌드 루프는 '데이터 수집, 피처 엔지니어링, 훈련, 평가, 배포, 드리프트 감지, 재훈련'이라는 순환 구조입니다. 또한 DevOps에서는 검증 대상이 코드뿐이지만, MLOps에서는 코드, 데이터 스키마, 모델 세 가지를 함께 검증해야 합니다. 이런 추가 요구사항 때문에 DevOps의 원칙을 ML 시스템에 맞게 확장한 별도의 체계가 필요합니다. 그것이 MLOps입니다.

MLOps 없이 ML 시스템을 운영하면 어떤 문제가 생기는지, **기술 부채**라는 개념을 통해 살펴보겠습니다. 기술 부채란 단기적인 편의를 위해 내린 설계 결정이 장기 유지보수 비용을 기하급수적으로 높이는 현상을 뜻합니다. 빚을 지면 당장은 편하지만 이자가 쌓이듯이, 급한 대로 넘긴 설계 결정이 나중에 훨씬 큰 비용으로 돌아오는 것입니다.

ML 시스템에서는 일반 소프트웨어보다 기술 부채가 더 빠르게, 더 넓은 영역에서 쌓입니다. 실제 프로덕션 ML 시스템에서 ML 모델 코드 자체는 전체의 일부분에 불과합니다. 나머지 대부분은 데이터 파이프라인, 서빙 인프라, 모니터링, 설정 관리 등 지원 코드입니다. 이 지원 코드를 체계적으로 관리하지 않으면 기술 부채가 급격히 쌓입니다.

ML 시스템에서 흔히 나타나는 기술 부채 유형은 다음과 같습니다. 첫 번째는 **얽힘**입니다. 어떤 입력 피처 하나를 바꾸면 모델 전체의 동작이 변합니다. 이를 'Changing Anything Changes Everything', 줄여서 CACE 원칙이라 부릅니다. 피처 A를 제거했더니 피처 B와 C의 영향력까지 달라지는 식입니다. 두 번째는 **숨겨진 피드백 루프**입니다. 모델의 출력이 미래의 훈련 데이터를 오염시키는 현상입니다. 추천 모델이 특정 아이템을 더 노출시키면, 사용자가 그 아이템을 더 많이 클릭하고, 그 클릭 데이터가 다시 훈련에 쓰여 편향이 강화됩니다. 세 번째는 **미선언 소비자**입니다. 모델 출력을 누가 어떻게 사용하는지 추적하지 않으면, 모델을 변경할 때 예상치 못한 곳에서 연쇄 장애가 발생합니다. 네 번째는 **설정 부채**입니다. 하이퍼파라미터, 데이터 경로, 피처 목록 등 수백 개의 설정값이 버전 관리되지 않으면, 이전 실험을 재현할 수 없게 됩니다.

기술 부채는 빠른 결과를 얻기 위해 빌린 시간입니다. 청산하지 않으면 시스템이 경직되어 새 기능을 추가하거나 모델을 교체하는 일이 사실상 불가능해집니다. MLOps는 이 기술 부채를 체계적으로 관리하고 줄여 나가기 위한 실천 체계이기도 합니다.

마지막으로 ML 모델을 프로덕션에 배포하고 운영할 때 겪는 구체적인 과제들, 즉 **모델 운영화의 과제**를 정리합니다. 이 과제의 핵심은 **프로덕션 갭**이라 불리는 현상입니다. Jupyter Notebook에서 잘 동작하는 프로토타입을 확장 가능하고 신뢰할 수 있는 프로덕션 시스템으로 전환하는 과정에서 대규모 공학적 부담이 발생합니다.

구체적인 과제는 여러 가지입니다. **훈련-서빙 스큐**는 훈련 환경과 서빙 환경의 차이로 인해, 훈련 때 높은 성능을 보이던 모델이 프로덕션에서 나빠지는 현상입니다. 피처를 계산하는 방식이 훈련 코드와 서빙 코드에서 미묘하게 다른 것이 흔한 원인입니다. **팀 사일로**는 데이터 과학자, ML 엔지니어, DevOps 팀이 격리되어 작업하면서 배포 과정에서 지식이 단절되는 문제입니다. **재현 불가능성**은 어떤 데이터로 어떤 코드를 써서 어떤 하이퍼파라미터로 훈련했는지 추적하지 않으면 동일한 모델을 다시 만들 수 없는 문제입니다. **드리프트 무감지**는 입력 데이터의 분포가 바뀌는 데이터 드리프트와, 입력과 출력 사이의 관계 자체가 바뀌는 개념 드리프트를 모니터링하지 않아 모델 성능이 조용히 저하되는 문제입니다. **수동 배포 병목**은 모델을 업데이트할 때마다 수작업으로 스크립트를 수정하고 배포하여 출시 속도가 느려지는 문제입니다. **버전 관리 부재**는 모델, 데이터, 코드, 설정을 통합적으로 관리하지 않아 롤백과 디버깅이 불가능해지는 문제입니다.

이 과제들은 개별로 존재하는 것이 아니라 서로 얽혀 있습니다. 버전 관리가 되지 않으면 재현이 불가능하고, 재현이 불가능하면 드리프트가 감지되어도 어떤 버전으로 돌아가야 할지 알 수 없습니다. MLOps는 이런 과제들을 생명주기 전반에 걸쳐 체계적으로 해결하기 위해 등장한 분야입니다.

정리하면, MLOps는 머신러닝 모델의 전체 생명주기를 체계적으로 관리하는 원칙과 프로세스의 집합입니다. 일반 소프트웨어와 달리 ML 시스템은 데이터 변화만으로도 동작이 바뀌기 때문에, DevOps의 코드 중심 자동화만으로는 부족합니다. 데이터, 모델, 코드, 설정을 함께 버전 관리하고, 훈련부터 서빙, 모니터링, 재훈련까지 이어지는 순환을 자동화해야 합니다. 이를 방치하면 기술 부채가 쌓이고, 프로덕션 갭을 넘지 못한 모델은 실험실에 머물게 됩니다. MLOps는 이 모든 과제를 구조적으로 다루기 위한 체계입니다.

다음 단원인 1.1.2에서는 MLOps 성숙도 모델을 다룹니다. 조직이 MLOps를 도입할 때 어떤 단계를 거쳐 역량을 높여 가는지, 각 단계에서 무엇이 자동화되고 무엇이 여전히 수작업인지를 살펴봅니다.

이 단원을 마치면 MLOps가 무엇이고 왜 필요한지, 기존 소프트웨어 개발과 어떻게 다른지 설명할 수 있습니다.
