# 1.3.1 Kafka 아키텍처 개요

애플리케이션 하나가 데이터를 만들고, 다른 애플리케이션이 그 데이터를 가져다 쓰는 상황을 떠올려 봅니다. 둘이 직접 연결되어 있으면 한쪽이 멈출 때 다른 쪽도 멈춥니다. 서비스가 수십, 수백 개로 늘어나면 연결선은 기하급수로 불어납니다. 이 문제를 해결하려면 데이터를 보내는 쪽과 받는 쪽 사이에 중간 저장소를 두어야 합니다. Apache Kafka는 바로 이 중간 저장소 역할을 하는 **분산 이벤트 스트리밍 플랫폼**입니다.

'이벤트 스트리밍'이란 어떤 사건이 발생할 때마다 그 기록을 실시간으로 흘려보내는 방식을 뜻합니다. 사용자가 버튼을 누르거나, 센서가 온도를 측정하거나, 주문이 접수될 때마다 하나의 이벤트가 만들어집니다. Kafka는 이런 이벤트를 받아 디스크에 순서대로 저장하고, 필요한 애플리케이션이 언제든 꺼내 읽을 수 있게 합니다. 내부적으로 Kafka는 **커밋 로그**(commit log)라는 구조를 씁니다. 커밋 로그란 새 데이터를 끝에만 덧붙이고, 이미 기록된 데이터는 수정하거나 삭제하지 않는 단순한 저장 방식입니다. 각 레코드에는 파티션 안에서의 위치를 나타내는 고유 순번인 **오프셋**(offset)이 붙습니다. 순차적으로 디스크에 쓰고 읽기 때문에 초당 수백만 건의 메시지를 처리할 수 있습니다.

Kafka의 전체 구조는 세 계층으로 나뉩니다. 데이터를 보내는 **프로듀서**(Producer), 데이터를 저장하고 중계하는 **브로커**(Broker), 데이터를 읽어 가는 **컨슈머**(Consumer)입니다. 이 세 계층이 어떻게 연결되는지 아래 다이어그램으로 먼저 확인합니다.

```
  프로듀서 A ─┐                              ┌─ 컨슈머 X
              │    ┌──────────────────┐      │
  프로듀서 B ─┼──> │  Kafka 클러스터   │ ──┼─ 컨슈머 Y
              │    │  (브로커 1,2,3)   │      │
  프로듀서 C ─┘    └──────────────────┘      └─ 컨슈머 Z
```

왼쪽의 프로듀서들이 메시지를 Kafka 클러스터로 보내고, 클러스터 안의 브로커들이 메시지를 저장합니다. 오른쪽의 컨슈머들은 저장된 메시지를 읽어 갑니다. 프로듀서와 컨슈머는 서로를 직접 알 필요가 없습니다. 이 구조 덕분에 한쪽이 늘거나 줄어도 다른 쪽에 영향을 주지 않습니다.

이제 각 계층을 하나씩 살펴봅니다.

프로듀서는 데이터를 Kafka에 보내는 애플리케이션입니다. 프로듀서가 메시지를 보낼 때는 어느 **토픽**(Topic)에 넣을지를 지정합니다. 토픽은 메시지를 분류하는 논리적 채널입니다. 예를 들어 '주문' 토픽과 '결제' 토픽을 따로 두면, 주문 관련 메시지와 결제 관련 메시지가 섞이지 않습니다. 프로듀서가 메시지에 키를 지정하지 않으면 라운드 로빈 방식으로 여러 파티션에 골고루 분산됩니다. 키를 지정하면 같은 키를 가진 메시지는 항상 같은 파티션에 들어가므로 순서가 보장됩니다.

브로커는 Kafka 클러스터를 이루는 서버 프로세스입니다. 프로듀서가 보낸 메시지를 디스크에 저장하고, 컨슈머의 읽기 요청에 응답합니다. 단일 브로커 하나만으로도 초당 수십만 건의 읽기와 쓰기를 처리할 수 있습니다. 다만 브로커 하나가 고장 나면 데이터를 잃을 수 있으므로, 실무에서는 최소 세 대의 브로커를 묶어 클러스터를 구성합니다.

브로커가 여러 대일 때 데이터 안전성을 확보하는 핵심 원리는 복제입니다. 각 파티션에는 리더(leader) 브로커 한 대와 팔로워(follower) 브로커 여러 대가 있습니다. 모든 읽기와 쓰기는 리더가 담당합니다. 팔로워는 리더의 데이터를 그대로 복사해 둡니다. 리더와 동기화 상태를 유지하는 팔로워를 ISR(In-Sync Replica)이라 부릅니다. 리더 브로커에 장애가 발생하면 ISR 중 하나가 새 리더로 승격되어 서비스가 이어집니다.

```
  파티션 0 복제 구조

  브로커 1 (리더)   ──복제──>  브로커 2 (팔로워, ISR)
                     ──복제──>  브로커 3 (팔로워, ISR)

  * 브로커 1 장애 시 -> 브로커 2 또는 3이 리더로 승격
```

브로커들의 메타데이터, 즉 어떤 브로커가 어떤 파티션의 리더인지 같은 정보를 관리하는 방법도 알아야 합니다. 과거에는 ZooKeeper라는 외부 시스템이 이 역할을 맡았습니다. 현재는 **KRaft**(Kafka Raft)라는 Kafka 내장 프로토콜이 기본값입니다. KRaft는 컨트롤러 노드들이 합의 알고리즘을 통해 메타데이터를 직접 관리합니다. 외부 시스템 없이 Kafka만으로 운영할 수 있어 배포가 단순해지고, 메타데이터 갱신 지연이 줄어듭니다.

컨슈머는 Kafka 토픽을 구독하여 메시지를 읽는 애플리케이션입니다. 컨슈머는 파티션에서 메시지를 순서대로 읽으며, 자신이 어디까지 읽었는지를 오프셋으로 추적합니다. 오프셋 값은 Kafka 내부 토픽인 `__consumer_offsets`에 저장됩니다. 기본 설정에서는 5초마다 오프셋이 자동으로 커밋됩니다. 더 정밀한 제어가 필요하면 자동 커밋을 끄고, 애플리케이션 코드에서 직접 커밋 시점을 결정할 수도 있습니다.

여기서 중요한 개념이 **컨슈머 그룹**(Consumer Group)입니다. 컨슈머 그룹은 같은 목적으로 메시지를 처리하는 컨슈머들의 모음입니다. 하나의 파티션은 같은 그룹 안에서 단 하나의 컨슈머만 읽을 수 있습니다. 파티션이 네 개이고 컨슈머가 네 개면, 각 컨슈머가 하나씩 맡아 병렬로 처리합니다. 컨슈머가 파티션 수보다 많으면 남는 컨슈머는 대기 상태가 됩니다.

```
  토픽 T (파티션 4개)          컨슈머 그룹 G

  파티션 0  ───────────>  컨슈머 G-1
  파티션 1  ───────────>  컨슈머 G-2
  파티션 2  ───────────>  컨슈머 G-3
  파티션 3  ───────────>  컨슈머 G-4
```

이 구조 전체를 관통하는 통신 방식이 **발행-구독 모델**(Publish-Subscribe Model)입니다. 발행-구독 모델에서 프로듀서는 토픽에 메시지를 '발행'하고, 컨슈머는 관심 있는 토픽을 '구독'합니다. 프로듀서는 누가 읽을지 모르고, 컨슈머는 누가 보냈는지 모릅니다. 이처럼 보내는 쪽과 받는 쪽이 서로를 직접 알지 못하는 구조를 느슨한 결합(loose coupling)이라 합니다.

전통적인 메시지 큐 방식과 비교하면 차이가 명확합니다. 메시지 큐에서는 한 소비자가 메시지를 가져가면 다른 소비자는 같은 메시지를 읽을 수 없습니다. Kafka에서는 메시지가 소비 후 즉시 삭제되지 않고 설정된 보존 기간 동안 유지됩니다. 따라서 서로 다른 컨슈머 그룹이 같은 메시지를 각자 독립적으로 읽을 수 있습니다. 주문 이벤트를 분석 시스템과 알림 시스템이 동시에 구독하는 식입니다.

마지막으로 **Kafka 클러스터**를 정리합니다. Kafka 클러스터는 여러 브로커가 모여 하나의 분산 시스템으로 동작하는 단위입니다. 클러스터 안에서 토픽의 파티션들이 브로커에 나뉘어 저장되므로 부하가 분산됩니다. 파티션마다 복제본이 여러 브로커에 존재하므로 한 브로커가 고장 나도 데이터가 유실되지 않습니다. 프로듀서, 브로커, 컨슈머가 각자 독립적으로 확장 가능하고, 발행-구독 모델로 느슨하게 연결되어 있기 때문에 서비스 규모가 커져도 구조를 바꾸지 않고 브로커나 컨슈머를 추가하는 것만으로 대응할 수 있습니다.

정리하면, Kafka는 프로듀서-브로커-컨슈머 세 계층 위에 발행-구독 모델을 얹은 분산 이벤트 스트리밍 플랫폼입니다. 프로듀서가 토픽에 메시지를 발행하고, 브로커가 커밋 로그 구조로 메시지를 저장하며, 컨슈머가 오프셋을 추적하며 메시지를 읽습니다. 브로커를 여러 대 묶어 클러스터를 구성하면 복제와 부하 분산이 이루어지고, 서로 다른 컨슈머 그룹이 같은 메시지를 독립적으로 소비할 수 있습니다.

다음 단원인 1.3.2에서는 토픽과 파티션을 다룹니다. 토픽이 파티션으로 나뉘는 구조와 그것이 병렬 처리 및 장애 허용에 어떻게 기여하는지를 살펴봅니다.

이 단원을 마치면 Kafka의 프로듀서-브로커-컨슈머 3계층 구조와 발행-구독 모델을 설명할 수 있습니다.
