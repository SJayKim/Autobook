# 1.2.2 프롬프트 엔지니어링

1.2.1에서 LLMOps의 빌드 루프가 훈련이 아닌 추론 중심으로 돌아간다는 점을 살펴보았습니다. 그 단원에서 프롬프트가 LLMOps에서 코드와 동등한 지위를 가진다는 표현이 나왔습니다. 전통적 ML 시스템에서 모델의 동작을 바꾸려면 데이터를 바꾸거나 모델을 재훈련해야 하지만, LLM 기반 시스템에서는 모델에게 내리는 지시문, 즉 프롬프트를 바꾸는 것만으로 출력의 형식, 정확도, 톤이 달라집니다. 프롬프트를 어떻게 설계하느냐에 따라 같은 모델이라도 완전히 다른 결과를 내놓는 것입니다.

이 단원에서는 프롬프트를 효과적으로 설계하기 위한 주요 기법과 원칙을 다룹니다. 먼저 프롬프트의 기본 패턴 세 가지를 살펴보고, 대화형 API에서 프롬프트의 맥락을 설정하는 System Prompt의 역할을 짚은 뒤, 프롬프트를 재사용 가능한 구조로 관리하는 프롬프트 템플릿을 알아봅니다. 마지막으로 이 기법들을 관통하는 프롬프트 설계 원칙을 정리합니다.

LLM에게 작업을 지시하는 가장 단순한 형태는 예시 없이 지시문만 전달하는 것입니다. 이 방식을 **Zero-shot**이라고 합니다. 'shot'은 모델에게 보여주는 예시(demonstration)를 뜻합니다. Zero-shot은 예시가 0개, 즉 예시 없이 지시만으로 모델이 작업을 수행하게 하는 방식입니다.

예를 들어 "다음 문장의 감정을 긍정, 부정, 중립 중 하나로 분류하시오."라는 지시와 함께 분석할 문장을 전달하면, 모델은 별도의 예시 없이도 분류 결과를 생성합니다. 모델이 사전 훈련 과정에서 이미 감정 분류라는 태스크를 충분히 학습했기 때문입니다.

Zero-shot은 태스크 정의가 단순명료하고, 모델이 이미 잘 아는 유형의 작업에 적합합니다. 번역, 요약, 단순 분류처럼 모델이 사전 훈련 중 반복적으로 접한 태스크가 여기에 해당합니다. 예시를 준비할 시간이 없거나, 예시를 넣을 만큼 지시가 복잡하지 않을 때 선택합니다. 반면 모델이 경험하지 못한 도메인 고유의 형식이나, 미묘한 판단 기준이 필요한 태스크에서는 Zero-shot의 결과가 불충분할 수 있습니다.

결과가 불충분할 때 다음으로 시도하는 방식이 **Few-shot**입니다. Few-shot은 모델에게 입력과 출력의 쌍을 몇 개 보여준 뒤, 새로운 입력에 대해 같은 패턴으로 응답하게 하는 방식입니다. 보통 2개에서 8개 정도의 예시를 프롬프트 안에 포함합니다.

Few-shot이 작동하는 원리는 **인-컨텍스트 러닝**(in-context learning)입니다. 1.2.1에서 LLM이 사전 훈련된 모델을 API로 호출하는 방식으로 동작한다고 설명했습니다. Few-shot에서 모델은 프롬프트에 포함된 예시들의 패턴을 파악하고, 그 패턴을 새 입력에 적용합니다. 모델의 가중치(파라미터)를 변경하는 것이 아니라, 컨텍스트 창 안에서 일시적으로 학습하는 것입니다. 그래서 인-컨텍스트 러닝이라고 부릅니다.

Few-shot의 핵심 가치는 출력 형식, 톤, 응답 범위를 예시로 직접 보여줄 수 있다는 점입니다. "JSON으로 응답하시오"라고 말로만 지시하는 것보다, 실제 JSON 형식의 입출력 예시를 보여주는 것이 모델의 응답 일관성을 높입니다. 다음은 감정 분류 태스크에서 Few-shot 프롬프트의 구조를 보여주는 예입니다.

```
지시: 다음 리뷰의 감정을 '긍정', '부정', '중립' 중 하나로 분류하시오.

리뷰: "배송이 빠르고 포장이 꼼꼼했습니다."
감정: 긍정

리뷰: "제품이 설명과 달라서 반품 요청했습니다."
감정: 부정

리뷰: "가격 대비 무난한 품질입니다."
감정: 중립

리뷰: "색상은 마음에 드는데 소재가 기대 이하네요."
감정:
```

이 프롬프트에서 처음 세 개의 리뷰-감정 쌍이 예시이고, 마지막 리뷰가 모델이 실제로 분류해야 할 입력입니다. 모델은 앞선 예시들의 패턴을 파악하여 마지막 입력에 대한 감정을 생성합니다.

Few-shot 예시를 설계할 때 주의할 점이 있습니다. 예시 간 구조와 포맷이 일관되어야 합니다. 줄바꿈, 공백, 구분자 같은 미세한 차이도 모델의 결과에 영향을 미칩니다. 예시의 수가 늘어나면 성능이 높아지는 경향이 있지만, 일정 수를 넘으면 오히려 과적합이 생길 수 있습니다. 또한 최신 고성능 모델에서는 Few-shot 예시의 역할이 "추론 정확도 향상"에서 "출력 형식 정렬"로 이동하는 추세입니다. 모델 자체의 추론 능력이 강해지면서, Few-shot 예시는 "이런 형식으로 답하라"는 신호로서의 역할이 더 중요해지고 있습니다.

Zero-shot과 Few-shot은 비교적 단순한 태스크에 효과적입니다. 그런데 다단계 수학 문제, 논리 추론, 복잡한 의사결정처럼 여러 단계를 거쳐야 하는 태스크에서는 모델이 바로 최종 답만 내놓으면 정확도가 낮아집니다. 중간 과정을 건너뛰고 곧바로 결론에 도달하려 하기 때문입니다.

이 문제를 해결하기 위한 기법이 **Chain-of-Thought**(CoT)입니다. Chain-of-Thought는 모델이 최종 답에 도달하기 전에 중간 추론 단계를 명시적으로 생성하도록 유도하는 방식입니다. '사고의 연쇄'라는 이름처럼, 문제를 풀어가는 과정을 한 단계씩 거치게 합니다.

Chain-of-Thought에는 두 가지 적용 방식이 있습니다.

첫 번째는 **Few-shot CoT**입니다. "문제 - 단계별 풀이 - 답"의 형식으로 구성된 예시를 여러 개 제공합니다. 모델은 예시에서 보여준 단계별 추론 패턴을 따라, 새로운 문제에도 같은 방식으로 중간 과정을 거쳐 답에 도달합니다.

두 번째는 **Zero-shot CoT**입니다. 예시를 전혀 제공하지 않고, 프롬프트 끝에 "Let's think step by step"과 같은 트리거 문장 하나를 추가합니다. 이 한 줄만으로 모델이 스스로 중간 추론 단계를 생성하기 시작합니다. 별도의 예시를 준비할 필요가 없어 간편하지만, Few-shot CoT보다 추론 품질이 낮을 수 있습니다.

다음은 두 방식의 차이를 보여주는 예입니다.

```
[Few-shot CoT]
문제: 사과 3개가 있고 2개를 더 샀습니다. 그중 1개를 먹었습니다. 남은 사과는?
풀이: 처음 3개 + 2개 = 5개. 1개를 먹으면 5 - 1 = 4개.
답: 4개

문제: 연필 8자루 중 3자루를 친구에게 주고 2자루를 새로 샀습니다. 남은 연필은?
풀이:

[Zero-shot CoT]
문제: 연필 8자루 중 3자루를 친구에게 주고 2자루를 새로 샀습니다. 남은 연필은?
Let's think step by step.
```

Few-shot CoT는 풀이 과정이 포함된 예시를 보여주어 모델이 그 패턴을 따르게 합니다. Zero-shot CoT는 예시 없이 트리거 문장만으로 단계별 추론을 유도합니다.

Chain-of-Thought의 장점은 두 가지입니다. 첫째, 복잡한 문제에서 정확도가 높아집니다. 모델이 중간 단계를 하나씩 거치므로, 한 번에 답을 내는 것보다 논리적 오류가 줄어듭니다. 둘째, 모델의 추론 과정이 텍스트로 드러나므로, 어디에서 오류가 발생했는지 추적할 수 있습니다.

한계도 있습니다. 모델이 생성하는 추론 과정이 모델 내부의 실제 계산을 정확히 반영하는 것은 아닙니다. 그럴듯해 보이는 풀이를 생성하면서도 답이 틀릴 수 있습니다. 또한 중간 단계를 생성하는 만큼 토큰 사용량과 응답 지연이 증가합니다.

지금까지 살펴본 세 가지 기법의 관계를 정리하면 다음과 같습니다.

```
Zero-shot ─────────── 예시 없음, 지시만
    │
    │  예시 추가
    ▼
Few-shot ──────────── 입출력 쌍 제시, 패턴 학습
    │
    │  추론 과정 추가
    ▼
Few-shot CoT ─────── 풀이 포함 예시, 단계별 추론

Zero-shot CoT ─────── 예시 없이 트리거 문장으로 단계별 추론 유도
```

Zero-shot에서 Few-shot으로 가면 예시가 추가됩니다. Few-shot에서 Few-shot CoT로 가면 예시에 추론 과정이 포함됩니다. Zero-shot CoT는 예시 없이 트리거 문장만으로 추론을 유도하는 별도의 경로입니다. 태스크의 복잡도, 예시 준비 가능 여부, 응답 속도 요구사항에 따라 적절한 기법을 선택합니다.

이제 프롬프트의 또 다른 중요한 구성 요소인 **System Prompt**를 살펴봅니다. 앞에서 다룬 Zero-shot, Few-shot, Chain-of-Thought는 모두 사용자가 모델에게 보내는 메시지 안에서 이루어지는 기법입니다. System Prompt는 이와 다른 층위에서 동작합니다.

대화형 LLM API는 메시지를 역할별로 구분합니다. 'system', 'user', 'assistant' 세 가지 역할이 있습니다. 'user'는 사용자가 보내는 질문이나 지시이고, 'assistant'는 모델의 응답입니다. 'system'은 대화가 시작되기 전에 모델에게 전달되는 설정 메시지입니다. 이 system 역할의 메시지가 바로 System Prompt입니다.

System Prompt는 이후 들어오는 모든 user 메시지를 모델이 어떻게 해석할지, 그 맥락을 설정합니다. 단순한 지시 한 줄이 아니라, 모델의 응답 방식 전체를 제한하는 틀의 역할을 합니다. System Prompt에 포함하는 요소는 크게 네 가지입니다.

첫째, **역할 부여**입니다. "당신은 금융 상품 설명 전문가입니다"처럼 모델의 페르소나를 지정합니다. 역할을 통해 응답의 톤과 깊이가 달라집니다.

둘째, **응답 범위 제약**입니다. "투자 관련 질문에만 답하시오"처럼 모델이 다루는 주제의 경계를 설정합니다.

셋째, **거부 조건 명시**입니다. "개인 금융 조언은 제공하지 마시오"처럼 모델이 거부해야 할 요청 유형을 정의합니다. 이는 보안 측면에서도 중요합니다. System Prompt가 사용자 입력을 무조건 신뢰하지 않는 구조를 만들어, 프롬프트 인젝션 공격을 방어하는 역할도 수행합니다.

넷째, **출력 형식 요구사항**입니다. "반드시 JSON 형식으로 응답하시오"처럼 응답의 구조를 지정합니다.

다음은 이 네 가지 요소가 포함된 System Prompt의 예입니다.

```
[system]
당신은 고객 문의에 응대하는 기술 지원 상담사입니다.
다음 규칙을 따르시오:
- 제품 사용법과 오류 해결에 대해서만 답하시오.
- 가격, 환불, 배송 관련 질문은 "고객센터로 문의해 주세요"라고 안내하시오.
- 응답은 200자 이내로 작성하시오.
- 응답 형식: {"answer": "...", "category": "사용법|오류|안내"}
```

이 System Prompt가 설정되면, 이후 사용자가 어떤 질문을 보내든 모델은 이 틀 안에서 응답합니다. 제품 오류에 대한 질문에는 기술적 답변을 제공하고, 환불 질문에는 고객센터 안내를 반환하며, 응답은 항상 지정된 JSON 형식을 따릅니다.

System Prompt에서 주의할 점이 있습니다. System Prompt는 모델의 응답 방식을 안내하지만, 모델의 기본 능력 자체를 변경하지는 않습니다. "수학 전문가입니다"라는 역할을 부여한다고 해서 모델의 수학 능력이 실제로 향상되는 것은 아닙니다. System Prompt는 모델이 이미 가진 능력을 특정 방향으로 집중시키는 도구입니다.

프로덕션 환경에서 프롬프트는 한 번 만들고 끝나는 것이 아닙니다. 같은 구조의 프롬프트를 다양한 입력에 반복적으로 적용해야 합니다. 고객 문의 응대 시스템이라면, 지시문과 응답 규칙은 동일하되 고객의 질문만 매번 바뀝니다. RAG 시스템이라면, 지시문은 같지만 검색된 문서 내용이 매번 달라집니다. 이렇게 고정된 부분과 가변적인 부분을 분리하여 재사용 가능한 구조로 만든 것이 **프롬프트 템플릿**입니다.

프롬프트 템플릿은 고정 콘텐츠와 가변 콘텐츠를 결합한 구조입니다. 고정 콘텐츠는 지시문, 응답 규칙, 출력 형식 등 매 요청마다 변하지 않는 부분입니다. 가변 콘텐츠는 사용자 입력, 검색된 문서, 대화 이력 등 요청마다 달라지는 부분입니다. 가변 부분은 플레이스홀더로 표시합니다. `{변수명}` 또는 `{{변수명}}` 형식이 일반적이며, 실행 시 실제 값으로 치환됩니다.

다음은 프롬프트 템플릿의 구조를 보여주는 예입니다.

```
다음 맥락을 참고하여 질문에 답하시오.
맥락에 없는 내용은 "정보가 없습니다"라고 답하시오.

맥락:
{context}

질문: {question}

답변:
```

이 템플릿에서 지시문("다음 맥락을 참고하여...")은 고정 콘텐츠이고, `{context}`와 `{question}`은 가변 콘텐츠입니다. 실행 시 `{context}`에는 검색된 문서 내용이, `{question}`에는 사용자의 질문이 들어갑니다.

프롬프트 템플릿을 구성하는 요소는 크게 네 가지로 나뉩니다. **Directive**는 태스크의 의도를 명시하는 부분입니다. "다음 텍스트를 요약하시오"처럼 모델이 무엇을 해야 하는지 지시합니다. **Context**는 입력 내용과 배경 설명을 담는 부분입니다. **Output Format**은 JSON, 표, 특정 길이 등 응답의 형식을 지정합니다. **Constraints**는 길이 제한, 주제 범위 제약 등 응답의 경계를 설정합니다.

프롬프트 템플릿이 중요한 이유는 프로덕션 운영과 직결됩니다. 프롬프트를 애플리케이션 코드에 직접 하드코딩하면, 프롬프트를 수정할 때마다 코드를 변경하고 재배포해야 합니다. 템플릿으로 분리하면 프롬프트만 독립적으로 수정할 수 있습니다. 또한 템플릿에 버전을 부여하면, 어떤 버전의 프롬프트가 어떤 결과를 냈는지 추적할 수 있습니다. 1.2.1에서 프롬프트 관리가 LLMOps의 핵심 과제라고 설명한 것이 바로 이런 맥락입니다.

변수를 XML 태그로 감싸는 것도 권장되는 패턴입니다. `<context>{{context}}</context>`처럼 태그를 사용하면 모델이 프롬프트의 각 부분을 더 명확하게 구분합니다. 특히 긴 프롬프트에서 고정 부분과 가변 부분의 경계가 흐려지지 않도록 하는 데 효과적입니다.

지금까지 살펴본 기법들을 관통하는 공통 원칙이 있습니다. 이를 **프롬프트 설계 원칙**이라고 합니다.

첫 번째 원칙은 **명확성과 구체성**입니다. 모호한 지시는 모델이 열린 대화처럼 처리하게 만들어 응답의 일관성이 낮아집니다. "잘 요약해 줘"보다 "3문장 이내로, 핵심 수치를 포함하여 요약하시오"가 더 일관된 결과를 만듭니다. 원하는 형식, 길이, 톤을 명시적으로 지정합니다.

두 번째 원칙은 **구조가 표현보다 중요하다**는 것입니다. 영리한 문구를 고르는 것보다, 프롬프트의 각 부분이 명확하게 구분되는 구조를 만드는 것이 더 효과적입니다. 섹션 헤더, 구분자, XML 태그를 활용하면 모델이 프롬프트의 각 부분을 분리하여 처리합니다.

세 번째 원칙은 **핵심 지시를 앞부분에 배치**하는 것입니다. LLM은 컨텍스트 내 위치에 따라 확률 가중치를 다르게 할당합니다. 프롬프트 앞부분에 놓인 내용이 뒷부분보다 더 강하게 반영되는 경향이 있습니다. 따라서 가장 중요한 지시는 프롬프트의 앞쪽에 배치해야 해석 오류를 줄일 수 있습니다.

네 번째 원칙은 **복잡한 태스크의 분해**입니다. 한 번의 프롬프트로 모든 것을 처리하려 하면 정확도가 낮아집니다. 복잡한 작업은 단순한 부분으로 나누어 각 단계를 별도의 프롬프트로 처리하는 것이 효과적입니다. 이는 Chain-of-Thought에서 문제를 단계별로 풀게 하는 것과 같은 원리입니다.

다섯 번째 원칙은 **반복 개선**입니다. 프롬프트는 처음부터 완성형이 아닙니다. 초기 프롬프트를 작성하고, 다양한 입력에 대한 결과를 확인하고, 구조와 표현을 점진적으로 정제하는 과정을 거칩니다. LLMOps 맥락에서 이 반복 개선 과정은 체계적으로 이루어져야 합니다. 프롬프트 텍스트뿐 아니라 모델, temperature 같은 생성 파라미터, 전처리 로직까지 모두 버전 관리 대상입니다. 어떤 조합이 어떤 결과를 냈는지 추적할 수 있어야 다음 개선 방향을 잡을 수 있습니다.

이 다섯 가지 원칙을 요약하면 다음과 같습니다.

```
프롬프트 설계 원칙
 1. 명확성과 구체성 ── 형식, 길이, 톤을 명시
 2. 구조 우선        ── 구분자, 태그로 영역 분리
 3. 앞부분 배치      ── 핵심 지시를 프롬프트 앞쪽에
 4. 태스크 분해      ── 복잡한 작업을 단순한 단계로
 5. 반복 개선        ── 결과 확인, 점진적 정제, 버전 관리
```

정리하면, 프롬프트 엔지니어링은 LLM에게 효과적으로 지시를 내리기 위한 설계 기법의 집합입니다. Zero-shot은 예시 없이 지시만으로, Few-shot은 입출력 예시를 보여주어, Chain-of-Thought는 중간 추론 단계를 거치게 하여 모델의 출력을 제어합니다. System Prompt는 대화 전체의 맥락을 설정하는 틀이고, 프롬프트 템플릿은 고정 부분과 가변 부분을 분리하여 프로덕션에서 프롬프트를 재사용하고 버전 관리할 수 있게 합니다. 이 기법들을 명확성, 구조, 앞부분 배치, 태스크 분해, 반복 개선이라는 다섯 가지 설계 원칙에 따라 적용하면, 같은 모델에서도 더 정확하고 일관된 결과를 얻을 수 있습니다.

다음 단원인 1.2.3에서는 임베딩과 유사도 검색을 다룹니다. 텍스트를 수치 벡터로 변환하여 의미적 유사도를 비교하는 방법을 살펴봅니다.

이 단원을 마치면 주요 프롬프트 설계 패턴을 구분하고 상황에 맞는 기법을 선택할 수 있습니다.
