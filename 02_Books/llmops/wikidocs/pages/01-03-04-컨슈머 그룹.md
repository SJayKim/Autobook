# 1.3.4 컨슈머 그룹

1.3.3에서 컨슈머가 오프셋을 커밋하여 읽기 위치를 추적하는 과정을 살펴보았습니다. 그런데 실제 운영 환경에서는 하나의 컨슈머만으로 모든 파티션을 처리하기 어렵습니다. 토픽에 파티션이 열 개 있고 메시지가 초당 수만 건씩 들어온다면, 컨슈머 한 대로는 처리 속도가 따라가지 못합니다. 여러 컨슈머를 동시에 띄워 파티션을 나누어 읽으면 처리량을 높일 수 있습니다. 이때 여러 컨슈머를 하나로 묶어 관리하는 단위가 컨슈머 그룹입니다.

**컨슈머 그룹**(consumer group)은 동일한 목적을 공유하는 컨슈머들의 집합입니다. 같은 토픽을 구독하되, 각자 다른 파티션을 담당하여 메시지를 병렬로 처리합니다. 핵심 규칙은 하나입니다. 그룹 안에서 하나의 파티션은 정확히 하나의 컨슈머에게만 할당됩니다. 같은 그룹의 두 컨슈머가 동일 파티션을 동시에 읽는 일은 발생하지 않습니다.

이 규칙 때문에 컨슈머 수와 파티션 수의 관계가 중요합니다. 파티션이 네 개인 토픽에 컨슈머 두 대를 배치하면, 각 컨슈머가 파티션 두 개씩 맡습니다. 컨슈머를 네 대로 늘리면 한 대당 파티션 하나씩 맡습니다. 그런데 컨슈머를 다섯 대로 늘리면 어떻게 될까요. 파티션은 네 개뿐이므로 한 대는 할당받을 파티션이 없습니다. 이 컨슈머는 유휴 상태가 됩니다. 메시지를 받지 못한 채 대기만 합니다. 따라서 컨슈머 수를 파티션 수보다 많이 두는 것은 자원 낭비입니다.

```
  토픽 T (파티션 4개)

  컨슈머 그룹 G (컨슈머 2대)
  +-----------+    +-----------+
  | 컨슈머 A  |    | 컨슈머 B  |
  | P0, P1    |    | P2, P3    |
  +-----------+    +-----------+

  컨슈머 그룹 G (컨슈머 4대)
  +------+ +------+ +------+ +------+
  |  A   | |  B   | |  C   | |  D   |
  |  P0  | |  P1  | |  P2  | |  P3  |
  +------+ +------+ +------+ +------+

  컨슈머 그룹 G (컨슈머 5대)
  +------+ +------+ +------+ +------+ +--------+
  |  A   | |  B   | |  C   | |  D   | |  E     |
  |  P0  | |  P1  | |  P2  | |  P3  | | (유휴) |
  +------+ +------+ +------+ +------+ +--------+
```

그룹이 생성되면, 어떤 컨슈머가 어떤 파티션을 맡을지 결정해야 합니다. 이 과정을 **파티션 할당**(partition assignment)이라 합니다. 할당은 그룹 코디네이터와 그룹 리더라는 두 역할이 협력하여 이루어집니다. 그룹 코디네이터는 뒤에서 자세히 다루겠습니다. 여기서는 할당 전략에 집중합니다.

파티션 할당 전략은 '어떤 규칙으로 파티션을 컨슈머에게 배분할 것인가'를 정의합니다. Kafka는 네 가지 할당 전략을 제공합니다.

첫 번째는 **Range Assignor**입니다. 토픽별로 파티션을 번호 순서대로 나열한 뒤, 컨슈머 수로 나누어 범위를 할당합니다. 예를 들어 파티션 0~5를 컨슈머 두 대에 나누면, 컨슈머 A가 0~2를, 컨슈머 B가 3~5를 맡습니다. 단일 토픽에서는 단순하고 직관적입니다. 그러나 여러 토픽을 동시에 구독하면 문제가 생깁니다. 토픽마다 범위 앞쪽이 같은 컨슈머에게 몰리기 때문입니다. 토픽이 세 개이고 각각 파티션이 세 개이면, 컨슈머 A가 매번 0번 파티션을 받아 총 세 개를 맡고, 컨슈머 B는 1번과 2번만 받습니다. 토픽 수가 늘수록 편중이 심해집니다.

두 번째는 **Round-robin Assignor**입니다. 모든 토픽의 파티션을 한데 모아 순환 방식으로 배분합니다. 컨슈머 A, B, C가 있으면 첫 번째 파티션은 A, 두 번째는 B, 세 번째는 C, 네 번째는 다시 A에게 돌아갑니다. Range 방식보다 균등하게 분배됩니다. 그러나 리밸런싱이 발생하면 기존 할당을 전혀 고려하지 않고 처음부터 다시 배분합니다. 파티션 재할당이 대량으로 발생할 수 있습니다.

세 번째는 **Sticky Assignor**입니다. 리밸런싱 시 기존 할당을 최대한 유지하면서 균등 분배를 달성합니다. 컨슈머 B가 빠졌다면, B가 맡던 파티션만 남은 컨슈머에게 재분배하고, A와 C의 기존 할당은 그대로 둡니다. 불필요한 파티션 이동을 줄여 리밸런싱 비용을 낮춥니다. 다만 Sticky Assignor는 전통적인 eager 방식으로 동작합니다. 리밸런싱이 시작되면 모든 컨슈머가 파티션을 일시 해제하고 소비를 멈춘 뒤, 새 할당을 받아 재개합니다. 기존 할당을 '유지하려 노력'하지만, 그 과정에서 전체 소비가 잠시 중단되는 것입니다.

네 번째는 **CooperativeStickyAssignor**입니다. Kafka 2.4에서 도입되었습니다. Sticky의 '기존 할당 유지' 특성에 협력적 리밸런싱을 결합한 전략입니다. 영향받는 파티션만 철회하고, 나머지 파티션은 소비를 계속합니다. 전체 소비가 멈추지 않습니다. 2025년 기준으로 신규 애플리케이션에 권장되는 전략입니다. 협력적 리밸런싱의 구체적인 동작 과정은 아래에서 별도로 설명합니다.

이제 **리밸런싱**(rebalancing)을 다룹니다. 리밸런싱은 컨슈머 그룹의 파티션 할당을 재조정하는 과정입니다. 그룹 멤버가 바뀌거나 파티션 구성이 변경되면 기존 할당이 더 이상 유효하지 않으므로, 파티션을 다시 나누어야 합니다.

리밸런싱이 발생하는 조건은 다섯 가지입니다. 첫째, 새 컨슈머가 그룹에 가입하거나 기존 컨슈머가 탈퇴할 때입니다. 컨테이너를 새로 띄우거나 인스턴스를 축소하는 상황이 여기에 해당합니다. 둘째, 토픽의 파티션 수가 변경될 때입니다. 셋째, 컨슈머가 session.timeout.ms 시간 안에 하트비트를 보내지 못할 때입니다. 코디네이터는 해당 컨슈머가 실패했다고 판단합니다. 넷째, 컨슈머가 max.poll.interval.ms 시간 안에 다음 poll()을 호출하지 못할 때입니다. 메시지 처리가 너무 오래 걸리면 이 조건에 걸립니다. 다섯째, 그룹 코디네이터 브로커 자체가 변경될 때입니다.

전통적인 리밸런싱은 eager 방식, 다른 말로 stop-the-world 방식이라 불립니다. 이름처럼 리밸런싱이 진행되는 동안 그룹 전체의 메시지 소비가 멈춥니다. 과정을 단계별로 살펴봅니다.

```
  Eager 리밸런싱 과정 (stop-the-world)

  1. 코디네이터가 모든 멤버에게 리밸런싱을 알린다.
  2. 모든 컨슈머가 자신이 맡던 파티션을 해제하고 소비를 중단한다.
  3. 각 컨슈머가 코디네이터에 JoinGroup 요청을 보낸다.
  4. 코디네이터가 그룹 리더를 선정하고 구독 정보를 전달한다.
  5. 그룹 리더가 할당 전략에 따라 새 할당을 계산한다.
  6. 그룹 리더가 SyncGroup 요청으로 할당 결과를 코디네이터에 보낸다.
  7. 코디네이터가 각 멤버에게 새 할당을 배포한다.
  8. 모든 컨슈머가 새로 할당받은 파티션에서 소비를 재개한다.
```

2단계에서 8단계 사이의 시간 동안 메시지 소비가 완전히 중단됩니다. 그룹에 컨슈머가 수십 대 있으면 JoinGroup 요청이 모두 도착할 때까지 대기하므로 중단 시간이 길어집니다. 이것이 eager 방식의 핵심 단점입니다.

이 단점을 해소하기 위해 Kafka 2.4에서 **협력적 리밸런싱**(cooperative rebalancing)이 도입되었습니다. 공식 명칭은 증분 협력적 리밸런싱(Incremental Cooperative Rebalancing)이며, KIP-429 제안으로 구현되었습니다.

협력적 리밸런싱의 핵심 원리는 두 가지입니다. 첫째, '협력적'(cooperative)이라는 이름처럼 각 컨슈머가 자발적으로 필요한 파티션만 반납합니다. 전체 파티션을 강제로 해제하지 않습니다. 둘째, '증분'(incremental)이라는 이름처럼 여러 라운드에 걸쳐 최종 균형 상태에 수렴합니다. 한 번에 완벽한 균형에 도달할 필요가 없습니다.

구체적인 진행 과정을 살펴봅니다.

```
  협력적 리밸런싱 과정 (2라운드)

  [1라운드]
  1. 코디네이터가 철회가 필요한 파티션 목록을 컨슈머에게 통보한다.
  2. 해당 컨슈머는 통보받은 파티션만 해제한다.
  3. 나머지 파티션은 소비를 계속한다.

  [2라운드]
  4. 1라운드에서 철회된 파티션이 새로운 컨슈머에게 할당된다.
  5. 기존에 유지 중인 파티션의 소비는 중단 없이 지속된다.
```

eager 방식과 비교하면 차이가 분명합니다. eager에서는 모든 컨슈머가 모든 파티션을 해제하고 전체 소비가 멈춥니다. 협력적 방식에서는 이동이 필요한 파티션만 잠시 소비가 끊기고, 나머지는 정상 동작합니다. 컨슈머가 수십 대인 환경에서 한두 대가 추가되거나 빠질 때, 전체 처리량이 순간적으로 0이 되는 상황을 피할 수 있습니다.

Kafka 4.0에서는 차세대 컨슈머 그룹 프로토콜(KIP-848)이 도입되었습니다. 이 프로토콜에서는 리밸런싱의 책임이 클라이언트에서 브로커의 그룹 코디네이터로 완전히 이전됩니다. 기존에는 그룹 리더 컨슈머가 할당을 계산했지만, 차세대 프로토콜에서는 코디네이터가 직접 할당을 생성합니다. ConsumerGroupHeartbeat라는 새 API를 통해 파티션 철회와 할당을 점진적으로 조율합니다. 이전 프로토콜 대비 최대 20배 빠른 리밸런스 성능을 달성한다고 보고되었습니다.

이제 **그룹 코디네이터**(group coordinator)를 설명합니다. 앞에서 '코디네이터가 리밸런싱을 조율한다'고 여러 번 언급했습니다. 그룹 코디네이터는 별도의 전용 서버가 아닙니다. Kafka 클러스터를 구성하는 브로커 중 하나가 특정 컨슈머 그룹의 코디네이터 역할을 겸합니다.

어떤 브로커가 코디네이터가 될까요. 1.3.3에서 다룬 __consumer_offsets 토픽이 열쇠입니다. 컨슈머 그룹이 오프셋을 커밋하면 __consumer_offsets 토픽의 특정 파티션에 기록된다고 설명했습니다. 해당 파티션의 리더 브로커가 곧 그 그룹의 코디네이터입니다. 파티션 번호는 컨슈머 그룹 ID의 해시값을 __consumer_offsets 파티션 수로 나눈 나머지로 결정됩니다.

```
  코디네이터 결정 과정

  1. 컨슈머 그룹 ID: "inference-workers"
  2. hash("inference-workers") % 50 = 23
  3. __consumer_offsets 토픽의 파티션 23의 리더 브로커 = 브로커 3
  4. 브로커 3이 "inference-workers" 그룹의 코디네이터가 된다.
```

코디네이터가 수행하는 역할을 정리합니다. 첫째, 컨슈머 생존 여부를 추적합니다. 각 컨슈머가 보내는 하트비트를 수신하고, session.timeout.ms 안에 하트비트가 오지 않으면 해당 컨슈머를 그룹에서 제거합니다. 둘째, 그룹 멤버십 변화를 감지합니다. 새 컨슈머의 가입과 기존 컨슈머의 탈퇴를 인지합니다. 셋째, 멤버십 변화가 감지되면 리밸런싱을 시작합니다. 넷째, JoinGroup 요청을 수신하여 그룹 리더를 선정합니다. 다섯째, 그룹 리더가 SyncGroup으로 보내온 할당 결과를 전체 멤버에게 배포합니다. 여섯째, 1.3.3에서 다룬 오프셋 커밋을 수신하여 __consumer_offsets에 저장합니다.

컨슈머는 코디네이터에게 주기적으로 **하트비트**(heartbeat)를 보냅니다. heartbeat.interval.ms 설정이 전송 주기를 결정하며, 기본값은 3초입니다. 이 값은 session.timeout.ms의 3분의 1 이하로 설정하는 것이 권장됩니다. session.timeout.ms의 기본값은 45초입니다. 컨슈머가 45초 동안 하트비트를 보내지 않으면, 코디네이터는 해당 컨슈머가 장애 상태라고 판단하고 그룹에서 제거한 뒤 리밸런싱을 시작합니다.

마지막으로 **독립 컨슈머**(standalone consumer)를 다룹니다. 지금까지 설명한 컨슈머 그룹은 파티션을 자동으로 할당받는 구조입니다. 그런데 특정 파티션만 지정하여 직접 읽어야 하는 경우가 있습니다. 감사 로그 수집기가 특정 파티션의 데이터만 처리해야 하거나, 파티션 내 순서 보장이 중요한 단일 처리기를 만들 때가 그렇습니다.

독립 컨슈머는 컨슈머 그룹에 참여하지 않고, 읽을 파티션을 수동으로 지정하는 컨슈머입니다. 그룹 컨슈머가 subscribe()로 토픽을 구독하는 대신, 독립 컨슈머는 assign()으로 특정 파티션을 명시합니다. assign()을 호출하면 코디네이터에 JoinGroup이나 SyncGroup 요청을 보내지 않습니다. 따라서 리밸런싱이 발생하지 않습니다. 다른 컨슈머가 가입하거나 탈퇴해도 영향을 받지 않습니다.

독립 컨슈머와 그룹 컨슈머의 차이를 정리합니다.

```
  그룹 컨슈머 vs 독립 컨슈머

  +----------------+-----------------------+-----------------------+
  | 구분           | 그룹 컨슈머           | 독립 컨슈머           |
  +----------------+-----------------------+-----------------------+
  | 파티션 할당    | 코디네이터가 자동 할당 | assign()으로 수동 지정 |
  | 리밸런싱       | 발생                  | 발생하지 않음          |
  | 자동 페일오버  | 지원                  | 미지원                 |
  | 장애 시 동작   | 다른 컨슈머가 인계    | 해당 파티션 소비 중단  |
  | 적합한 상황    | 수평 확장, 일반 소비  | 특수 처리, 순서 보장   |
  +----------------+-----------------------+-----------------------+
```

독립 컨슈머는 리밸런싱이 없으므로 할당이 안정적입니다. 그러나 컨슈머가 장애로 종료되면, 해당 파티션을 자동으로 인계받을 다른 컨슈머가 없습니다. 그 파티션의 메시지 소비가 중단됩니다. 따라서 수평 확장과 자동 복구가 필요한 일반적인 파이프라인에서는 컨슈머 그룹을 사용하고, 특수한 요구사항이 있을 때만 독립 컨슈머를 선택합니다.

정리하면, 컨슈머 그룹은 여러 컨슈머가 파티션을 나누어 병렬 처리하는 단위이며, 그룹 안에서 하나의 파티션은 하나의 컨슈머에게만 할당됩니다. 파티션 할당 전략은 Range, Round-robin, Sticky, CooperativeStickyAssignor 네 가지가 있으며, 신규 애플리케이션에는 CooperativeStickyAssignor가 권장됩니다. 리밸런싱은 그룹 멤버나 파티션 구성이 변경될 때 할당을 재조정하는 과정이고, 협력적 리밸런싱은 영향받는 파티션만 철회하여 나머지 소비를 유지합니다. 그룹 코디네이터는 __consumer_offsets 파티션의 리더 브로커가 맡으며, 하트비트 감시와 리밸런싱 조율을 담당합니다. 독립 컨슈머는 assign()으로 파티션을 수동 지정하며, 리밸런싱 없이 안정적으로 동작하지만 자동 페일오버가 없습니다.

다음 단원인 1.3.5에서는 복제와 내결함성을 다룹니다. 브로커 장애 시에도 데이터가 유실되지 않도록 파티션을 여러 브로커에 복제하는 구조를 살펴봅니다.

이 단원을 마치면 컨슈머 그룹의 파티션 할당 메커니즘과 리밸런싱 과정을 설명할 수 있습니다.
