# 1.1.3 ML을 위한 CI/CD/CT 파이프라인

1.1.2에서 MLOps 성숙도 모델을 살펴보면서, Level 2에 이르면 파이프라인 자체를 CI/CD로 관리한다고 설명했습니다. Level 1에서 등장한 지속적 훈련(CT), Level 2에서 등장한 지속적 통합(CI)과 지속적 배포(CD)라는 용어가 나왔지만, 각각이 ML 시스템에서 구체적으로 무엇을 검증하고 어떤 순서로 동작하는지는 아직 다루지 않았습니다. 이 단원에서는 CI, CD, CT 각각의 역할을 풀어 설명하고, 이 세 단계가 파이프라인 안에서 어떻게 맞물리는지 살펴봅니다.

소프트웨어 개발에서 CI/CD는 이미 널리 쓰이는 자동화 관행입니다. 개발자가 코드를 수정하면 자동으로 빌드하고 테스트한 뒤, 테스트를 통과한 결과물을 운영 환경에 배포합니다. 그런데 ML 시스템에서는 코드만 변하는 것이 아닙니다. 학습에 사용하는 데이터가 달라지기도 하고, 모델 자체가 재훈련을 거쳐 새 버전으로 바뀌기도 합니다. 코드, 데이터, 모델이라는 세 가지 아티팩트가 모두 변경 대상이므로, 소프트웨어 CI/CD를 그대로 가져다 쓸 수 없습니다. ML에서는 기존 CI/CD의 범위를 넓히고, 소프트웨어에는 없는 CT라는 단계를 추가해야 합니다.

먼저 **Continuous Integration for ML**, 즉 ML을 위한 지속적 통합부터 설명합니다. 소프트웨어 CI는 개발자가 코드를 변경하면 자동으로 빌드하고 단위 테스트를 돌려서 문제가 없는지 확인합니다. ML CI도 코드를 빌드하고 테스트하는 것까지는 같습니다. 차이는 검증 대상이 코드에 머물지 않는다는 점입니다.

ML CI에서는 코드 테스트에 더해 세 가지를 추가로 검증합니다. 첫째, 데이터 스키마 검사입니다. 학습 파이프라인이 기대하는 컬럼이 모두 존재하는지, 데이터 타입이 맞는지, 필수 필드에 결측값이 없는지 확인합니다. 둘째, 특성 변환 일관성 검사입니다. 피처 엔지니어링 로직이 기대한 대로 동작하는지, 변환 결과에 NaN 같은 비정상 값이 섞이지 않았는지 점검합니다. 셋째, 모델 성능 사전 검증입니다. 변경된 코드로 짧은 훈련을 돌려 본 뒤, 성능 지표가 사전에 정한 임계값을 통과하는지 확인합니다.

이 과정은 보통 PR(Pull Request)이 메인 브랜치에 제출될 때 자동으로 트리거됩니다. 예를 들어 GitHub Actions 같은 워크플로우 도구를 사용하면, PR이 열리는 순간 학습 코드 단위 테스트, 데이터 스키마 검사, 모델 성능 검증을 한꺼번에 실행할 수 있습니다. 실행 결과는 PR 코멘트로 게시되어, 팀원이 병합 전에 결과를 검토할 수 있습니다. 테스트를 통과하면 파이프라인 컴포넌트 패키지와 실행 파일이 생성됩니다. 소프트웨어 CI에서 빌드 결과물이 애플리케이션 바이너리인 것과 달리, ML CI의 결과물은 파이프라인 컴포넌트 자체입니다.

정리하면, ML CI는 소프트웨어 CI의 코드 빌드와 단위 테스트에 더해 데이터 품질 게이트, 특성 파이프라인 검증, 모델 성능 임계값 통과 여부까지 자동으로 확인하는 확장된 통합 과정입니다.

다음은 **Continuous Delivery for ML**, 즉 ML을 위한 지속적 배포입니다. 소프트웨어 CD는 CI에서 만들어진 코드 아티팩트를 운영 환경에 자동으로 배포합니다. 웹 애플리케이션이라면 빌드된 컨테이너 이미지를 서버에 올리는 것이 CD의 전부입니다. ML CD는 여기서 한 가지 근본적인 차이가 있습니다. 배포하는 대상이 훈련된 모델 파일 하나가 아니라, 자동 재학습 파이프라인 전체라는 점입니다.

이 차이가 중요한 이유를 설명합니다. 소프트웨어는 배포 후 코드가 변하지 않으면 동작이 일정합니다. 그러나 ML 모델은 시간이 지나면 데이터 분포가 바뀌면서 성능이 저하됩니다. 1.1.2에서 설명한 드리프트 문제입니다. 따라서 모델 파일 하나를 배포하는 것으로는 충분하지 않고, 새 데이터가 들어왔을 때 자동으로 재학습하여 모델을 갱신하는 파이프라인 자체를 배포해야 합니다. 배포된 파이프라인이 이후 자율적으로 모델을 생성하고 갱신합니다.

구체적인 배포 흐름을 살펴봅니다. CI 단계를 통과하여 PR이 메인 브랜치에 병합되면, CD 워크플로우가 트리거됩니다. 이 워크플로우는 파이프라인 아티팩트를 대상 환경에 배포합니다. 배포는 보통 단계적으로 진행합니다. 개발 환경에서 사전 프로덕션 환경까지는 자동으로, 사전 프로덕션에서 프로덕션 환경으로는 수동 승인을 거치는 방식이 일반적입니다. 배포 시 기존 서비스의 엔드포인트와 인증 토큰은 변하지 않으므로, 이 서비스를 사용하는 다른 애플리케이션은 배포 과정을 의식할 필요 없이 연속적으로 동작합니다.

1.1.2에서 소개한 성숙도 수준과 연결하면, Google Level 1은 파이프라인 실행만 자동화한 상태이고, Google Level 2에 이르러서야 파이프라인과 그 배포 과정 모두가 CI/CD로 자동화됩니다. Microsoft 모델에서는 Level 3이 CI/CD 파이프라인 완성, Level 4가 정책 기반 자동 프로모션까지 달성한 단계에 해당합니다.

이제 ML에만 고유한 세 번째 단계인 **Continuous Training**, 즉 지속적 훈련을 설명합니다. CT는 소프트웨어 CI/CD에는 존재하지 않는 개념입니다. 소프트웨어는 배포 후 코드가 바뀌지 않으면 동작이 그대로 유지되지만, ML 모델은 그렇지 않습니다. 시간이 지나면 프로덕션 데이터의 분포가 변합니다. 이를 **데이터 드리프트**라고 합니다. 입력 데이터의 통계적 특성(평균, 분산, 분포 형태 등)이 모델을 처음 훈련할 때와 달라지는 현상입니다. 또 다른 유형인 **개념 드리프트**는 입력과 출력 사이의 관계 자체가 변하는 것을 뜻합니다. 예를 들어 사용자의 구매 패턴이 계절에 따라 바뀌면, 같은 입력 특성이라도 예측해야 하는 정답이 달라집니다.

이런 드리프트가 발생하면 모델의 예측 정확도가 떨어집니다. 코드를 전혀 수정하지 않았는데도 성능이 저하되는 것입니다. CT는 이 문제에 대응하기 위해 프로덕션 모델을 새 데이터로 자동 재학습하고, 검증을 거쳐 서빙하는 프로세스입니다.

CT의 동작 흐름은 다음과 같습니다.

```
트리거 발생 (스케줄/새 데이터 도착/드리프트 감지/성능 저하)
     |
     v
데이터 추출 및 검증
     |
     v
모델 재학습
     |
     v
모델 검증 (오프라인: holdout 테스트셋 평가, 챔피언-챌린저 비교)
     |
     v
  통과 여부 판단
   /        \
통과         실패
  |            |
  v            v
모델 배포    배포 취소, 기존 모델 유지
```

CT에서 중요한 것은 재학습 자체가 아니라, 재학습 전후의 검증 과정입니다. 단순히 새 데이터로 모델을 다시 피팅하는 것만으로는 부족합니다. 드리프트의 원인을 분석하지 않고 맹목적으로 재학습하면, 오히려 미래 예측 오차가 증가할 수 있습니다. 따라서 재학습 전에 데이터 검증을, 재학습 후에 모델 검증을 거쳐야 합니다.

CT가 ML 파이프라인에 처음 등장하는 시점은 1.1.2에서 설명한 Google Level 1입니다. Level 0에서는 사람이 수동으로 재학습하지만, Level 1부터 파이프라인이 자동으로 CT를 수행합니다.

이제 CT 흐름에서 등장한 **데이터 검증 단계**를 자세히 살펴봅니다. 데이터 검증은 학습 데이터가 파이프라인에 진입하기 전, 또는 CT 트리거 후 새 데이터가 유입될 때 수행하는 자동 품질 게이트입니다. 검증 없이 데이터를 그대로 파이프라인에 넣으면, 결측값이 섞인 데이터로 모델이 훈련되거나, 스키마가 달라진 데이터가 파이프라인을 망가뜨릴 수 있습니다.

데이터 검증은 두 가지 축으로 나뉩니다.

첫 번째 축은 **스키마 검증**입니다. 파이프라인이 기대하는 컬럼이 모두 존재하는지, 각 컬럼의 데이터 타입이 맞는지, 필수 필드에 결측값이 없는지, 값이 허용 범위 안에 있는지 확인합니다. 예를 들어 '나이' 컬럼이 정수여야 하는데 문자열이 섞여 있거나, '가격' 컬럼에 음수가 들어 있으면 스키마 검증에서 걸립니다.

두 번째 축은 **분포 검증**입니다. 데이터의 통계적 속성(평균, 분산, 분위수 등)이 모델을 처음 훈련할 때의 기준 데이터 대비 유의미하게 변했는지 감지합니다. 앞서 설명한 데이터 드리프트를 이 단계에서 잡아내는 것입니다. 분포 검증에는 KS 검정(Kolmogorov-Smirnov test)이나 PSI(Population Stability Index) 같은 통계적 방법이 사용됩니다.

실무에서 데이터 검증을 구현할 때는 검증 규칙 집합을 미리 정의해 두고, 파이프라인 실행 시마다 데이터를 이 규칙에 대조하는 방식을 씁니다. 규칙은 엄격한 통과/실패 기준 외에 허용 오차를 설정할 수도 있습니다. 예를 들어 "99% 이상의 행이 규칙을 통과하면 성공"으로 정하면, 소수의 이상치 때문에 파이프라인 전체가 멈추는 상황을 피할 수 있습니다.

검증에 실패한 데이터는 격리하여 다운스트림 파이프라인이 오염되지 않도록 합니다. 동시에 알림을 발송하여 담당자가 데이터 소스를 점검하도록 연결합니다.

다음은 **모델 검증 단계**입니다. 재학습된 모델을 프로덕션에 바로 배포하면, 성능이 기존 모델보다 오히려 떨어질 위험이 있습니다. 모델 검증은 이 위험을 막기 위해 배포 전에 수행하는 성능 확인 과정입니다.

모델 검증은 두 단계로 구분됩니다.

첫 번째는 **오프라인 검증**입니다. 미리 분리해 둔 holdout 테스트셋으로 새 모델의 성능 지표(정확도, F1, RMSE 등)를 측정합니다. 그리고 현재 프로덕션에서 서빙 중인 모델, 즉 **챔피언** 모델과 비교합니다. 새로 훈련된 모델은 **챌린저**라고 부릅니다. 챌린저의 성능이 챔피언보다 통계적으로 유의미하게 개선되지 않으면, 파이프라인은 여기서 중단됩니다. 예를 들어 "정확도가 기존 모델 대비 2% 이상 높아야 통과"와 같은 임계값을 미리 설정해 둡니다.

두 번째는 **온라인 검증**입니다. 오프라인 검증을 통과한 모델을 실제 프로덕션 트래픽의 일부에 노출시킵니다. **카나리 배포**는 전체 트래픽의 소량(예: 5%)만 새 모델로 보내어 문제가 없는지 확인한 뒤 점진적으로 비율을 높이는 방식입니다. **A/B 테스트**는 사용자 그룹을 나누어 챔피언과 챌린저의 결과를 동시에 비교하는 방식입니다.

모델 검증에 실패하면 새 모델의 배포는 취소되고, 기존 프로덕션 모델이 그대로 유지됩니다. 실패 원인(데이터 품질 문제, 하이퍼파라미터 부적합, 특성 변화 등)을 분석한 뒤 재시도합니다.

지금까지 CI, CD, CT와 그 안의 데이터 검증, 모델 검증을 각각 살펴보았습니다. 이 단계들이 실제 파이프라인에서 어떤 순서로 실행되는지, 그리고 각 단계를 촉발하는 **파이프라인 트리거**가 무엇인지 정리합니다.

ML 파이프라인 트리거는 크게 세 유형으로 나뉩니다.

첫째, **시간 기반 트리거**입니다. cron 스케줄로 정기적으로 파이프라인을 실행합니다. "매일 새벽 2시", "매주 월요일"처럼 정해진 주기에 맞추어 동작합니다. 구현이 단순하고 실행 시점을 예측할 수 있다는 장점이 있지만, 데이터 변화에 즉각 대응하지 못한다는 단점이 있습니다.

둘째, **데이터 기반 트리거**입니다. 새 데이터 배치가 도착하거나, 누적 데이터량이 설정한 기준을 넘거나, 데이터 드리프트가 임계값을 초과했을 때 파이프라인이 실행됩니다. 특정 데이터셋이 업데이트되면 학습 파이프라인을 자동으로 시작하는 데이터 주도 스케줄링 패턴이 여기에 해당합니다.

셋째, **성능 기반 트리거**입니다. 프로덕션 모델의 예측 정확도나 F1 점수 같은 메트릭이 임계값 아래로 떨어지거나, 개념 드리프트가 감지될 때 실행됩니다.

이 세 유형 외에 코드 변경 트리거(PR 제출, 메인 브랜치 병합)와 수동 트리거(관리자가 직접 실행)도 있습니다. 코드 변경 트리거는 주로 CI/CD 단계를, 데이터/성능 기반 트리거는 주로 CT 단계를 촉발합니다.

실무에서는 단일 트리거만 사용하는 것보다 여러 트리거를 조합하는 것이 권장됩니다. 시간 기반 트리거를 기본 안전망으로 설정하여 최소 주기마다 재학습이 이루어지도록 하고, 드리프트 감지나 성능 저하 이벤트를 보조 트리거로 추가하여 급격한 변화에 즉시 대응합니다. 단일 이벤트의 오탐(false positive)으로 불필요한 재학습이 발생하는 것을 막기 위해, "연속 N회 위반 시에만 트리거"와 같은 조건을 적용하기도 합니다.

CI, CD, CT와 트리거가 하나의 파이프라인에서 어떻게 맞물리는지 전체 흐름을 정리합니다.

```
코드 변경 (PR 제출)
     |
     v
[CI] 코드 빌드 + 단위 테스트 + 데이터 스키마 검사 + 모델 성능 사전 검증
     |
     v
  CI 통과 → PR 병합
     |
     v
[CD] 파이프라인 아티팩트를 대상 환경에 배포
     |
     v
트리거 발생 (스케줄 / 새 데이터 / 드리프트 / 성능 저하)
     |
     v
[CT] 데이터 검증 → 모델 재학습 → 모델 검증 (오프라인 + 온라인)
     |
     v
  검증 통과 → 새 모델 서빙 시작
     |
     v
  모니터링 → 이상 감지 시 다시 트리거 발생 (순환)
```

CI는 파이프라인 코드의 품질을 보장하고, CD는 검증된 파이프라인을 운영 환경에 배포하며, CT는 배포된 파이프라인 위에서 모델을 지속적으로 갱신합니다. 세 단계가 순환하면서 ML 시스템이 코드 변경에도, 데이터 변화에도, 성능 저하에도 자동으로 대응하는 구조가 완성됩니다.

정리하면, ML을 위한 CI/CD/CT 파이프라인은 소프트웨어 CI/CD를 확장하여 데이터와 모델이라는 추가 아티팩트를 다루는 자동화 체계입니다. CI는 코드뿐 아니라 데이터 스키마와 모델 성능까지 검증하고, CD는 모델 파일이 아닌 재학습 파이프라인 전체를 배포하며, CT는 드리프트에 대응하여 모델을 자동 재학습하고 챔피언-챌린저 비교를 거쳐 배포 여부를 결정합니다. 데이터 검증은 스키마와 분포 두 축으로 데이터 품질을 확인하고, 모델 검증은 오프라인과 온라인 두 단계로 성능을 확인합니다. 파이프라인 트리거는 시간, 데이터, 성능 기반을 조합하여 운영합니다.

다음 단원인 1.1.4에서는 모델 서빙 아키텍처 개요를 다룹니다. CT를 거쳐 검증된 모델이 실제로 예측 요청을 처리하는 서빙 단계에서 어떤 구조를 갖추는지 살펴봅니다.

이 단원을 마치면 ML에서 CI, CD, CT 각각의 역할과 기존 소프트웨어 CI/CD와의 차이를 설명할 수 있습니다.
