# 1.3.6 KRaft 모드와 컨트롤러 쿼럼

1.3.5에서 파티션 복제와 리더 선출을 살펴보았습니다. 리더 브로커가 장애를 일으키면 ISR 내 팔로워가 새 리더가 되고, 이 선출 과정을 Kafka 컨트롤러가 담당한다는 점까지 확인했습니다. 그런데 이 컨트롤러 자체는 누가 관리할까요. 컨트롤러가 죽으면 파티션 리더 선출이 멈추고, 클러스터 전체의 메타데이터 갱신이 중단됩니다. Kafka는 오랫동안 이 문제를 외부 시스템에 맡겨 왔습니다. 이 단원에서는 그 외부 시스템이 무엇이었고, 왜 제거되었으며, 지금은 어떤 구조로 대체되었는지를 다룹니다.

Kafka가 처음 설계될 때, 클러스터 메타데이터를 저장하고 관리하는 역할은 **ZooKeeper**라는 별도의 분산 시스템이 맡았습니다. ZooKeeper는 Apache 재단이 관리하는 독립 프로젝트로, 분산 시스템의 설정 정보와 멤버십 관리를 위해 만들어진 범용 코디네이션 서비스입니다. Kafka는 여기에 브로커 목록, 토픽 설정, 접근 제어 목록(ACL), 컨트롤러 선출 결과 등을 저장했습니다.

문제는 ZooKeeper가 Kafka와 완전히 다른 별도 시스템이라는 점에 있었습니다. ZooKeeper는 자체적인 배포, 설정, 모니터링, 장애 대응 절차가 필요합니다. Kafka 클러스터 하나를 운영하려면 사실상 두 개의 분산 시스템을 동시에 운영해야 했습니다. 운영 부담이 두 배가 되는 것입니다.

운영 복잡도만이 문제는 아닙니다. ZooKeeper가 메타데이터 처리의 병목이 되는 상황도 있었습니다. 토픽과 파티션 수가 늘어날수록 ZooKeeper에 저장되는 메타데이터의 양이 증가합니다. ZooKeeper의 처리 용량이 한계에 도달하면 클러스터 전체의 확장이 막힙니다. 또한 Kafka 컨트롤러가 메모리에 캐시하는 메타데이터 상태와 ZooKeeper에 실제 저장된 상태 사이에 불일치가 생길 가능성도 있었습니다. 두 시스템이 각자의 방식으로 상태를 유지하기 때문입니다.

이러한 한계를 해결하기 위해 Kafka 커뮤니티는 KIP-500이라는 개선 제안을 통해 ZooKeeper 의존성을 완전히 제거하는 설계를 내놓았습니다. 핵심 아이디어는 Kafka가 자체적으로 메타데이터를 관리하는 것입니다. 외부 시스템에 맡기는 대신, Kafka 서버 일부가 직접 메타데이터 관리 역할을 수행합니다. 이 새로운 구조를 **KRaft 모드**라 합니다. KRaft는 'Kafka Raft'의 줄임말입니다.

KRaft라는 이름에 포함된 Raft가 무엇인지 먼저 알아야 합니다. Raft는 분산 시스템에서 여러 노드가 하나의 동일한 상태를 유지하도록 합의하는 알고리즘입니다. 1.3.5에서 다룬 파티션 복제가 '데이터'의 사본을 여러 브로커에 유지하는 것이었다면, Raft 합의는 '결정 사항'을 여러 노드가 동일하게 공유하는 것입니다. 누가 리더인지, 어떤 토픽이 존재하는지, 각 파티션의 설정은 무엇인지와 같은 메타데이터를 여러 컨트롤러가 동일하게 유지합니다.

KRaft는 표준 Raft 알고리즘을 Kafka의 메타데이터 관리에 맞게 변형한 구현입니다. 가장 큰 차이는 데이터 전달 방식에 있습니다. 표준 Raft에서는 리더가 팔로워에게 데이터를 밀어 넣는 push 방식을 사용합니다. KRaft에서는 반대로 팔로워가 리더에게 데이터를 가져오는 pull 방식을 사용합니다. 이 방식은 1.3.5에서 살펴본 파티션 복제와 동일한 패턴입니다. 팔로워 브로커가 리더에게 Fetch 요청을 보내 데이터를 가져오는 것처럼, 팔로워 컨트롤러가 리더 컨트롤러에게 메타데이터를 가져옵니다. pull 방식에서는 각 팔로워가 자신의 처리 속도에 맞춰 데이터를 가져갈 수 있어 확장성이 향상됩니다.

KRaft의 리더 선출 과정을 단계별로 살펴보겠습니다.

```
  KRaft 리더 선출 과정

  1단계: 모든 컨트롤러 노드가 팔로워 상태로 시작
  +----------+    +----------+    +----------+
  | 팔로워 A |    | 팔로워 B |    | 팔로워 C |
  +----------+    +----------+    +----------+

  2단계: 리더 연락 없음 -> 선거 타임아웃 -> 후보자 전환
  +----------+    +----------+    +----------+
  | 후보자 A |    | 팔로워 B |    | 팔로워 C |
  | term=2   |    |          |    |          |
  +----------+    +----------+    +----------+
       |--- RequestVote --->|
       |--- RequestVote ----------->|

  3단계: 과반수(2표 이상) 획득 -> 리더 확정
  +----------+    +----------+    +----------+
  | 리더 A   |<-- 투표 --|팔로워 B |    | 팔로워 C |
  | term=2   |<-- 투표 ---------|          |
  +----------+    +----------+    +----------+
```

첫 번째 단계에서 모든 컨트롤러 노드는 팔로워 상태로 시작합니다. 각 노드는 리더로부터 주기적인 연락을 기다립니다. 두 번째 단계에서, 일정 시간 동안 리더의 연락이 없으면 팔로워 중 하나가 선거 타임아웃을 적용합니다. 이 노드는 자신의 term 카운터를 증가시키고 후보자 상태로 전환합니다. term은 선거 회차를 나타내는 정수입니다. 후보자는 다른 노드들에게 RequestVote라는 요청을 보냅니다. 세 번째 단계에서, 과반수 투표를 먼저 얻은 후보자가 새 리더가 됩니다. 3대 구성이면 2표, 5대 구성이면 3표가 필요합니다.

여러 노드가 동시에 후보자가 되면 투표가 분산되어 어느 쪽도 과반수를 얻지 못하는 상황이 생길 수 있습니다. 이를 분할 투표(split vote)라 합니다. KRaft는 이 문제를 무작위 타임아웃으로 해결합니다. 각 노드의 선거 타임아웃에 무작위 지연을 추가하여 동시에 후보자가 되는 확률을 낮춥니다.

리더가 확정되면, 이후의 모든 메타데이터 변경은 리더 컨트롤러를 거칩니다. 리더가 메타데이터 레코드를 기록하면, 팔로워 컨트롤러가 이를 가져가 자신의 상태에 반영합니다. 팔로워가 새 레코드를 커밋하려면 두 번의 fetch가 필요합니다. 첫 번째 fetch에서 레코드 자체를 가져오고, 두 번째 fetch에서 업데이트된 high watermark를 받아야 해당 레코드가 커밋된 것으로 확정됩니다. 이 구조는 1.3.5에서 다룬 데이터 복제의 high watermark 개념과 동일합니다.

KRaft는 오래된 리더가 새 리더 확정 후에도 메타데이터를 기록하는 사고를 방지하기 위해 에포크(epoch) 기반의 차단 로직을 사용합니다. 모든 RPC에 currentLeaderEpoch 필드가 포함되어 있어, 이전 term의 리더가 보낸 요청은 거부됩니다.

KRaft 내부는 세 가지 계층으로 나뉩니다. **QuorumController**는 메타데이터를 생성하고 상태 머신을 갱신합니다. **KafkaRaftClient**는 리더 선출과 복제를 관리합니다. **MetadataLoader**는 변경된 메타데이터를 클러스터 내부 컴포넌트에 전달합니다. 이 세 계층은 KafkaEventQueue라는 단일 스레드 이벤트 루프 위에서 동작하여 순서가 보장됩니다.

이제 컨트롤러 쿼럼이 무엇인지 살펴봅니다. KRaft 모드에서는 Kafka 서버 중 일부를 컨트롤러로 지정합니다. 이 컨트롤러들의 집합을 **컨트롤러 쿼럼**(controller quorum)이라 합니다. 쿼럼은 '의사결정에 필요한 최소 참여자 수'를 뜻하는 말로, 여기서는 메타데이터 합의에 참여하는 컨트롤러 그룹 전체를 가리킵니다.

컨트롤러 쿼럼은 일반적으로 3대 또는 5대로 구성합니다. 과반수가 살아 있어야 합의가 가능하므로, 3대 구성에서는 1대 장애까지 허용하고, 5대 구성에서는 2대 장애까지 허용합니다. 이를 일반화하면 2n+1대 구성에서 n대 장애를 허용합니다.

컨트롤러 쿼럼에는 정적 쿼럼과 동적 쿼럼 두 가지 유형이 있습니다. 정적 쿼럼은 Kafka 초기 KRaft 구현에서 사용한 방식입니다. controller.quorum.voters라는 설정에 컨트롤러 목록을 고정으로 지정합니다. 클러스터가 동작하는 동안에는 컨트롤러를 추가하거나 제거할 수 없습니다. 이 방식은 현재 deprecated 상태입니다.

동적 쿼럼은 Kafka 4.0에서 도입된 KRaft version 1의 기능입니다. controller.quorum.bootstrap.servers 설정을 사용하며, 클러스터가 동작하는 중에도 컨트롤러 멤버를 변경할 수 있습니다. AddRaftVoter, RemoveRaftVoter, UpdateRaftVoter라는 RPC를 통해 한 번에 한 노드씩 변경합니다. 한 번에 한 노드씩만 변경하는 이유는 동시에 여러 노드를 교체하면 과반수 계산이 꼬여 split-brain이 발생할 수 있기 때문입니다.

컨트롤러 쿼럼이 관리하는 메타데이터는 어디에 저장될까요. KRaft 모드에서는 **__cluster_metadata**라는 내부 토픽이 이 역할을 합니다. 이름 앞의 밑줄 두 개는 Kafka가 내부 용도로 사용하는 시스템 토픽임을 나타냅니다. 이 토픽은 단일 파티션으로 구성되며, Raft 합의의 정확성을 위해 레코드를 디스크에 동기적으로 기록합니다.

__cluster_metadata 토픽에는 다양한 유형의 레코드가 저장됩니다. REGISTER_BROKER_RECORD는 브로커가 클러스터에 합류할 때 기록됩니다. TOPIC_RECORD와 PARTITION_RECORD는 토픽과 파티션이 생성될 때 기록됩니다. LeaderChangeMessage는 파티션 리더가 변경될 때 기록됩니다. VotersRecord는 컨트롤러 쿼럼의 투표 노드 목록을 담습니다.

이전 ZooKeeper 방식에서는 브로커가 RPC를 통해 메타데이터를 조회했습니다. 특정 시점의 상태를 요청하고 응답을 받는 방식입니다. KRaft 모드에서는 이와 다릅니다. 리더 컨트롤러가 변경 사항을 __cluster_metadata 토픽에 이벤트로 기록하면, 팔로워 컨트롤러와 일반 브로커가 이 토픽을 구독하여 변경 사항을 가져갑니다. 브로커가 일시적으로 연결이 끊겼다가 복구되면, 자신이 마지막으로 읽은 오프셋 이후의 레코드를 메타데이터 로그에서 재생하여 최신 상태를 복원합니다. ZooKeeper 방식에서 발생하던 상태 불일치 가능성이 이 구조에서는 크게 줄어듭니다.

투표 상태는 quorum-state 파일에 영속화됩니다. 컨트롤러가 재시작되더라도 이 파일을 읽어 이전 투표 상태를 복원하므로, 같은 term에 두 번 투표하는 오류를 방지합니다.

이제 각 Kafka 서버가 자신의 역할을 어떻게 선언하는지 살펴봅니다. KRaft 모드에서는 **process.roles**라는 설정으로 서버의 역할을 지정합니다. 이 설정이 있으면 KRaft 모드로 동작하고, 없으면 ZooKeeper 모드로 동작합니다.

process.roles에는 세 가지 값을 지정할 수 있습니다. 'broker'로 설정하면 클라이언트의 읽기와 쓰기 요청만 처리하는 브로커 전용 노드가 됩니다. 'controller'로 설정하면 메타데이터 쿼럼에만 참여하는 컨트롤러 전용 노드가 됩니다. 'broker,controller'로 설정하면 두 역할을 하나의 서버에서 동시에 수행하는 결합 노드가 됩니다.

결합 노드는 단일 JVM에서 컨트롤러와 브로커를 함께 실행합니다. 서버 대수가 적은 개발 환경이나 소규모 배포에 적합합니다. 프로덕션 환경에서는 컨트롤러와 브로커를 분리하는 것이 권장됩니다. 컨트롤러 노드에 장애가 생겼을 때 브로커 기능까지 함께 멈추는 것을 피하기 위해서입니다. 장애의 영향 범위를 줄이는 것을 장애 격리(failure isolation)라 합니다.

process.roles 외에도 함께 설정해야 하는 항목이 있습니다. node.id는 클러스터 내에서 해당 서버를 식별하는 고유 정수입니다. controller.listener.names는 컨트롤러 간 통신에 사용할 리스너 이름입니다. controller.quorum.bootstrap.servers는 컨트롤러 쿼럼을 구성하는 서버 주소 목록입니다.

다음은 컨트롤러 전용 노드와 브로커 전용 노드의 설정 예입니다.

```
# 컨트롤러 전용 노드 설정
process.roles=controller
node.id=1
controller.listener.names=CONTROLLER
controller.quorum.bootstrap.servers=ctrl1:9093,ctrl2:9093,ctrl3:9093
```

process.roles를 controller로 지정하여 이 서버가 메타데이터 관리만 담당함을 선언합니다. node.id는 클러스터 내 고유 식별자입니다. controller.listener.names는 컨트롤러 간 통신에 사용할 리스너를 CONTROLLER라는 이름으로 지정합니다. controller.quorum.bootstrap.servers에는 쿼럼을 구성하는 세 대의 컨트롤러 주소를 나열합니다.

```
# 브로커 전용 노드 설정
process.roles=broker
node.id=101
controller.quorum.bootstrap.servers=ctrl1:9093,ctrl2:9093,ctrl3:9093
```

브로커 전용 노드도 controller.quorum.bootstrap.servers를 설정해야 합니다. 브로커가 메타데이터를 가져올 컨트롤러 쿼럼의 위치를 알아야 하기 때문입니다.

마지막으로 **Kafka 4.0**의 변화를 정리합니다. Kafka 4.0에서 ZooKeeper 기반 클러스터는 공식적으로 지원이 종료되었습니다. 모든 신규 클러스터는 KRaft 모드가 기본값입니다. 앞서 설명한 동적 쿼럼 관리도 이 버전에서 도입되었습니다. 기존에 ZooKeeper로 운영하던 클러스터를 KRaft로 전환하려면 Kafka 3.9를 중간 다리로 거쳐야 합니다.

마이그레이션은 네 단계로 진행됩니다. 먼저 기존 ZooKeeper 브로커 옆에 KRaft 컨트롤러 쿼럼을 새로 배포합니다. 다음으로 브로커 설정에 KRaft 연결 정보를 추가하고 롤링 재시작합니다. 그런 다음 브로커를 KRaft 모드로 전환합니다. 이 시점이 마지막 롤백 지점입니다. 마지막으로 컨트롤러 설정에서 ZooKeeper 연결을 제거하면 전환이 완료됩니다.

```
  ZooKeeper -> KRaft 마이그레이션 단계

  1단계: KRaft 쿼럼 배포
  [ZK 브로커] + [ZK 앙상블]  -->  [ZK 브로커] + [ZK 앙상블] + [KRaft 쿼럼]

  2단계: 브로커에 KRaft 연결 추가 (롤링 재시작)
  [ZK 브로커 + KRaft 설정] + [ZK 앙상블] + [KRaft 쿼럼]

  3단계: 브로커를 KRaft 모드로 전환 (마지막 롤백 지점)
  [KRaft 브로커] + [ZK 앙상블(대기)] + [KRaft 쿼럼]

  4단계: ZooKeeper 제거 -> 완료
  [KRaft 브로커] + [KRaft 쿼럼]
```

정리하면, Kafka는 클러스터 메타데이터를 외부 시스템인 ZooKeeper에 의존하던 구조에서, KRaft 모드를 통해 자체 관리 구조로 전환했습니다. KRaft는 Raft 합의 알고리즘의 pull 기반 변형으로, 컨트롤러 쿼럼 내에서 리더를 선출하고 __cluster_metadata 토픽에 메타데이터를 이벤트 형태로 기록합니다. 각 서버는 process.roles 설정으로 브로커, 컨트롤러, 또는 결합 역할을 선언합니다. Kafka 4.0부터 ZooKeeper 지원이 종료되었으며, 동적 쿼럼 관리로 운영 중에도 컨트롤러 멤버를 변경할 수 있습니다.

다음 단원인 2.1.1에서는 정확히 한 번 의미론을 다룹니다.

이 단원을 마치면 KRaft가 ZooKeeper를 대체하는 이유와 컨트롤러 쿼럼의 동작 원리를 설명할 수 있습니다.
