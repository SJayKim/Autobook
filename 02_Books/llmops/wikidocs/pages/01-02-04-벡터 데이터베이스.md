# 1.2.4 벡터 데이터베이스

1.2.3에서 임베딩 모델이 텍스트를 고차원 밀집 벡터로 변환하는 과정을 살펴보았습니다. 코사인 유사도나 내적 같은 거리 메트릭으로 벡터 간 의미적 거리를 계산할 수 있다는 것도 확인했습니다. 그런데 실제 서비스에서는 이 벡터가 수백만, 수천만 개에 이릅니다. 기업의 내부 문서, 제품 카탈로그, 고객 상담 기록을 모두 임베딩하면 그만큼 벡터가 쌓입니다. 이 벡터들을 어디에 저장하고, 쿼리가 들어왔을 때 어떻게 밀리초 안에 가장 유사한 벡터를 찾아낼 수 있을까요? 이 문제를 해결하기 위해 등장한 것이 벡터 데이터베이스입니다.

**벡터 데이터베이스**는 임베딩 모델이 생성한 고차원 부동소수점 벡터를 저장하고, 쿼리 벡터와 의미적으로 가까운 이웃을 빠르게 찾아 반환하는 특수 목적 저장소입니다. 전통적인 관계형 데이터베이스(RDBMS)는 정확한 값 일치나 범위 조건으로 데이터를 찾습니다. 'SELECT * FROM products WHERE price > 1000' 같은 질의가 대표적입니다. 벡터 데이터베이스는 이와 다릅니다. 코사인 유사도, 유클리드 거리, 내적 같은 거리 메트릭을 기준으로 쿼리 벡터에 가장 가까운 벡터 k개를 반환합니다. 이 검색 방식을 의미 유사도 검색이라 합니다.

RAG(Retrieval-Augmented Generation) 파이프라인에서 벡터 데이터베이스가 어떤 위치에 있는지 살펴봅니다. 먼저 지식 베이스 문서를 청크 단위로 분할하고, 각 청크를 임베딩 모델로 변환하여 벡터 데이터베이스에 저장합니다. 이것이 사전 준비 단계입니다. 사용자 쿼리가 들어오면 동일한 임베딩 모델로 쿼리를 벡터로 변환하고, 벡터 데이터베이스에서 가장 유사한 청크를 검색합니다. 검색된 청크를 LLM의 컨텍스트에 주입하면, LLM이 학습 데이터에 없는 최신 정보나 기업 내부 지식을 활용하여 답변을 생성할 수 있습니다.

```
사전 준비:
  문서 청크 --> [임베딩 모델] --> 벡터 --> [벡터 데이터베이스에 저장]

쿼리 처리:
  사용자 질문 --> [임베딩 모델] --> 쿼리 벡터
       |
       v
  [벡터 데이터베이스] --> 유사 청크 k개 반환
       |
       v
  [LLM] + 유사 청크 --> 답변 생성
```

벡터 데이터베이스가 이 역할을 수행하려면 핵심적으로 갖추어야 하는 기능이 있습니다. 바로 대규모 벡터 집합에서 빠르게 유사한 벡터를 찾아내는 것입니다. 1.2.3에서 다루었듯이, 모든 벡터와 하나씩 비교하는 완전 탐색은 O(n * d)의 계산량을 요구합니다. 수천만 개 벡터에 대해 이 방식을 적용하면 실시간 응답이 불가능합니다. 그래서 모든 주요 벡터 데이터베이스는 **인덱싱 알고리즘**을 사용합니다.

인덱싱 알고리즘은 벡터들 사이의 관계를 사전에 구조화해 두는 방법입니다. 인덱스를 구축하면 검색 시 모든 벡터를 확인할 필요 없이 후보를 크게 줄일 수 있습니다. 약간의 정확도를 희생하는 대신 탐색 속도를 수천 배 높이는 것입니다. 1.2.3에서 배운 근사 최근접 이웃 검색(ANN)이 바로 이 원리입니다. 벡터 데이터베이스는 ANN 알고리즘 위에 영속 저장, 메타데이터 필터링, 분산 처리 같은 데이터베이스 기능을 얹은 시스템이라 할 수 있습니다.

1.2.3에서 IVF와 HNSW의 기본 동작을 살펴본 바 있습니다. 이 단원에서는 두 알고리즘의 내부 메커니즘을 좀 더 깊이 들여다보고, 각각의 장단점과 실무 선택 기준을 다룹니다.

**HNSW**(Hierarchical Navigable Small World)는 현재 가장 널리 채택된 인덱스 유형입니다. HNSW는 다층 그래프를 구성합니다. 최상위 레이어에는 소수의 노드만 존재하며 서로 긴 거리로 연결됩니다. 아래로 내려갈수록 노드가 조밀해지고, 최하층에는 모든 노드가 포함됩니다. 각 노드가 어느 레이어까지 포함될지는 지수적 감소 확률로 무작위 결정됩니다. 대부분의 노드는 최하층에만 존재하고, 극소수의 노드만 상위 레이어에 나타납니다.

검색 과정을 단계별로 설명합니다. 최상위 레이어의 진입 노드에서 출발합니다. 현재 노드의 이웃 중 쿼리 벡터에 더 가까운 노드가 있으면 그쪽으로 이동합니다. 더 가까운 이웃이 없으면 로컬 최솟값에 도달한 것입니다. 이때 한 레이어 아래로 내려가서 같은 탐색을 반복합니다. 최하층까지 내려가면 최종 후보를 반환합니다. 이 방식은 도시에서 길을 찾을 때 고속도로로 대략적인 방향을 잡고, 간선도로로 좁혀 들어간 뒤, 골목길에서 정확한 목적지를 찾는 것과 비슷합니다.

```
[레이어 3]  A ============ B         (노드 2개, 긴 연결)
            |               |
[레이어 2]  A --- C --- B --- D       (노드 4개, 중간 연결)
            |  /  |  /  |  /  |
[레이어 1]  A-E-C-F-B-G-D-H-I-J      (모든 노드, 짧은 연결)

검색 흐름:
  레이어 3에서 A 출발 --> B로 이동 (쿼리에 더 가까움)
  레이어 2로 하강   --> B에서 D로 이동
  레이어 1로 하강   --> D 주변에서 최종 후보 확정
```

HNSW의 동작을 조절하는 파라미터는 세 가지입니다. **M**은 레이어당 최대 연결 수입니다. M을 높이면 노드 간 경로가 풍부해져 정확도가 올라가지만, 연결 정보를 저장해야 하므로 메모리 사용량도 증가합니다. **ef_construction**은 인덱스를 구축할 때 각 노드를 삽입하면서 탐색하는 폭입니다. 이 값이 클수록 더 좋은 이웃을 찾아 연결하므로 인덱스 품질이 높아지지만, 구축 시간이 길어집니다. **ef_search**는 검색 시 탐색하는 폭입니다. 이 값을 높이면 더 많은 후보를 확인하여 정확도가 올라가는 대신 응답 시간이 늘어납니다.

HNSW의 큰 장점은 동적 업데이트가 가능하다는 것입니다. 새 벡터를 삽입하거나 기존 벡터를 삭제할 때 전체 인덱스를 처음부터 재구성할 필요가 없습니다. 그래프에 노드를 추가하거나 제거하면 됩니다. 데이터가 수시로 변하는 환경에서 유리합니다. 단점은 노드당 연결 정보를 모두 저장해야 하므로 메모리 사용량이 크다는 것입니다.

**IVF**(Inverted File Index)는 k-means 클러스터링을 기반으로 하는 파티셔닝 방식의 인덱스입니다. IVF는 두 단계로 동작합니다.

첫 번째는 훈련 단계입니다. 전체 벡터를 k-means 알고리즘으로 nlist개의 클러스터로 분할합니다. 각 클러스터의 중심점(centroid)을 계산하여 저장합니다. 1백만 개 벡터가 있다면 nlist를 1,000으로 설정하는 것이 권장 시작점입니다. 대략 제곱근에 해당하는 값입니다.

두 번째는 검색 단계입니다. 쿼리 벡터와 nlist개의 중심점만 먼저 비교하여 가장 가까운 nprobe개의 클러스터를 선택합니다. 이것을 거친 탐색(coarse search)이라 합니다. 선택된 클러스터 안에 있는 벡터들만 상세히 비교합니다. 이것을 정밀 탐색(fine search)이라 합니다. nprobe를 높이면 더 많은 클러스터를 확인하므로 재현율이 올라가지만 속도는 느려집니다. nprobe는 쿼리마다 동적으로 조정할 수 있습니다.

```
훈련 단계:
  전체 벡터 --> [k-means] --> nlist개 클러스터 + 중심점

검색 단계:
  쿼리 벡터 --> 중심점과 비교 (거친 탐색)
       |
       v
  가장 가까운 nprobe개 클러스터 선택
       |
       v
  선택된 클러스터 내 벡터와 정밀 비교 --> 결과 반환
```

IVF와 HNSW는 각각 다른 상황에서 강점을 보입니다. 메모리 측면에서 IVF는 중심점 정보만 추가로 저장하면 되므로 HNSW보다 효율적입니다. HNSW는 노드당 M개의 연결 정보를 저장해야 합니다. 동적 업데이트 측면에서는 HNSW가 유리합니다. IVF는 데이터가 크게 변하면 클러스터 구조가 낡아져서 인덱스를 재구성해야 할 수 있습니다. 필터 검색, 즉 메타데이터 조건과 벡터 유사도를 결합하는 검색에서는 IVF가 클러스터 단위로 필터를 적용할 수 있어 효율적인 경향이 있습니다. 수십억 벡터 규모에서는 IVF가 메모리 효율 덕분에 더 적합합니다.

IVF에는 **PQ**(Product Quantization)를 결합한 변형이 있습니다. 1.2.3에서 간략히 다룬 IVF-PQ입니다. PQ는 벡터를 여러 서브벡터로 분해하고, 각 서브벡터를 미리 만들어 둔 코드북의 대표 벡터로 근사합니다. 원래 벡터 대신 짧은 코드만 저장하므로 메모리를 대폭 절약합니다. IVF와 PQ를 결합한 IVFPQ는 대규모 데이터에서 메모리 제약이 있을 때 실무에서 자주 사용됩니다.

지금까지 인덱싱 알고리즘의 메커니즘을 살펴보았습니다. 이제 실제로 벡터 데이터베이스 제품을 선택해야 하는 상황을 다룹니다. 시장에는 다양한 벡터 데이터베이스가 존재하며, 각각 설계 철학과 적합한 사용 시나리오가 다릅니다. 주요 제품 여섯 가지를 하나씩 살펴봅니다.

**Pinecone**은 완전 관리형 서버리스 클라우드 벡터 데이터베이스입니다. '완전 관리형'이란 사용자가 서버 크기, CPU, 메모리를 직접 설정하지 않아도 된다는 뜻입니다. Pinecone이 인프라를 알아서 운영합니다. 사용자는 API를 통해 벡터를 삽입하고 검색하기만 하면 됩니다. 내부적으로 API 게이트웨이, 글로벌 컨트롤 플레인, 리전별 데이터 플레인, 분산 오브젝트 스토리지로 구성됩니다. AWS, GCP, Azure 세 곳의 클라우드를 모두 지원합니다.

Pinecone의 가장 큰 강점은 운영 부담이 거의 없다는 것입니다. 인프라 운영 인력이 부족한 팀이나 빠르게 프로토타입을 만들어 프로덕션으로 넘겨야 하는 상황에 적합합니다. SOC 2, GDPR, HIPAA 등 컴플라이언스 인증을 갖추고 있어 규제가 엄격한 환경에서도 사용할 수 있습니다. 반면 오픈소스가 아니므로 특정 벤더에 의존하게 됩니다. 자체 호스팅 방식보다 비용이 높을 수 있습니다.

**Weaviate**는 오픈소스 벡터 검색 엔진입니다. 직접 서버에 설치하여 운영할 수도 있고, Weaviate가 제공하는 클라우드 관리형 서비스를 이용할 수도 있습니다. Weaviate의 가장 두드러진 특징은 **하이브리드 검색**입니다.

하이브리드 검색이란 벡터 검색과 키워드 검색을 하나의 쿼리에서 결합하는 것입니다. 벡터 검색은 의미적 유사도를 포착하지만, 정확한 용어가 포함된 문서를 놓칠 수 있습니다. 키워드 검색(BM25)은 정확한 단어 일치에 강하지만 의미적 유사성은 포착하지 못합니다. 하이브리드 검색은 두 방식의 점수를 합산하여 양쪽 장점을 모두 취합니다. Weaviate에서는 alpha 파라미터로 두 방식의 가중치를 조절합니다. alpha가 0이면 순수 키워드 검색, 1이면 순수 벡터 검색입니다. 0.7로 설정하면 벡터 검색에 70%, 키워드 검색에 30% 가중치를 주는 식입니다.

Weaviate는 쿼리 언어로 GraphQL을 채택했습니다. nearVector, nearText 같은 연산자와 where 필터를 하나의 요청으로 조합할 수 있습니다. 또한 모듈 아키텍처를 갖추고 있어 text2vec-transformers 모듈로 자동 임베딩을 수행하거나, generative 모듈로 LLM 연동을 추가하는 것이 가능합니다.

**Qdrant**는 Rust로 작성된 고성능 오픈소스 벡터 검색 엔진입니다. Rust는 메모리 안전성과 높은 실행 성능을 동시에 제공하는 프로그래밍 언어입니다. Qdrant는 Rust의 이런 특성 덕분에 리소스 효율이 높아, 비용에 민감하거나 엣지 환경에서 벡터 검색을 수행해야 하는 경우에 적합합니다.

Qdrant의 핵심 차별점은 **페이로드 필터링**입니다. 각 벡터에 JSON 형식의 메타데이터(페이로드)를 첨부할 수 있습니다. 검색 시 이 메타데이터를 조건으로 필터를 적용합니다. 예를 들어 '2024년 이후 작성된 문서 중 유사한 것'을 찾는 식입니다. 일반적인 사후 필터링은 먼저 벡터 유사도로 후보를 뽑은 뒤 메타데이터 조건을 확인하므로, 필터 조건에 맞지 않는 후보가 많으면 결과 수가 부족해질 수 있습니다. Qdrant는 'Filterable HNSW'라는 방식으로 필터 조건을 인덱스 구조 자체에 반영합니다. 탐색 과정에서 조건에 맞지 않는 노드를 건너뛰므로 성능이 우수합니다.

Qdrant는 밀집 벡터 외에 **희소 벡터**도 지원합니다. 1.2.3에서 다룬 것처럼 TF-IDF나 BM25에 해당하는 벡터가 희소 벡터입니다. 밀집 벡터와 희소 벡터를 함께 저장하고 혼합 검색을 수행할 수 있어, Weaviate와 마찬가지로 하이브리드 검색이 가능합니다.

**FAISS**는 Meta(Facebook AI Research)가 개발한 오픈소스 라이브러리입니다. 1.2.3에서 간략히 언급한 바 있습니다. FAISS가 다른 제품과 근본적으로 다른 점은, 독립된 데이터베이스 서비스가 아니라 Python이나 C++ 코드에 직접 통합하는 라이브러리라는 것입니다. 서버를 띄워서 API로 접근하는 것이 아니라, 애플리케이션 코드 안에서 함수를 호출하여 인덱스를 생성하고 검색합니다.

FAISS는 인덱스 알고리즘의 선택 폭이 넓습니다. IndexFlatL2는 브루트 포스로 100% 정확도를 보장합니다. IndexIVFFlat은 IVF 방식, IndexIVFPQ는 IVF와 PQ를 결합한 방식, IndexHNSWFlat은 HNSW 방식입니다. GPU 가속도 지원하여 단일 또는 다중 GPU에서 검색을 수행할 수 있습니다. 연구 환경이나 인덱스 알고리즘을 세밀하게 제어해야 하는 전문 시스템에 적합합니다.

FAISS의 한계는 데이터베이스로서의 기능이 없다는 것입니다. 영속적 저장, 분산 처리, 실시간 업데이트, 메타데이터 필터링 등을 직접 구현해야 합니다. 프로덕션 서비스에 단독으로 사용하려면 추가 엔지니어링이 상당히 필요합니다. 실제로 Milvus 같은 벡터 데이터베이스가 내부 인덱스 엔진으로 FAISS를 사용하기도 합니다.

**Milvus**는 Zilliz가 개발한 오픈소스 분산 벡터 데이터베이스입니다. 수십억에서 수백억 규모의 벡터를 다루는 엔터프라이즈 시나리오를 목표로 설계되었습니다. Milvus의 핵심은 **분산 아키텍처**입니다. 쿼리 노드(검색 담당), 데이터 노드(쓰기 담당), 인덱스 노드(인덱싱과 압축 담당)를 독립적으로 수평 확장할 수 있습니다. 읽기가 많으면 쿼리 노드만 늘리고, 쓰기가 많으면 데이터 노드만 늘리는 식입니다. 컴퓨팅과 스토리지가 분리된 구조입니다.

Milvus는 Kubernetes 위에서 동작하도록 설계되었습니다. 클라우드 네이티브 환경에 자연스럽게 배포됩니다. 지원하는 인덱스도 HNSW, IVF, PQ, GPU 기반 인덱스 등 10종 이상으로 폭이 넓습니다. 로컬에서 간편하게 실행할 수 있는 경량 버전인 Milvus Lite도 제공합니다. 프로토타이핑은 Milvus Lite로 시작하고 프로덕션에서는 분산 클러스터로 전환하는 흐름이 가능합니다. Milvus는 자체 인프라와 엔지니어링 역량을 보유한 대규모 조직에 적합합니다.

**Chroma**는 개발자 친화성을 최우선으로 설계한 오픈소스 임베딩 데이터베이스입니다. LangChain, LlamaIndex 같은 주요 LLM 프레임워크와 기본 통합을 제공합니다. 설치부터 사용까지가 간단합니다. pip로 설치한 뒤 몇 줄의 코드로 벡터 저장과 검색을 시작할 수 있습니다.

Chroma는 세 가지 배포 모드를 제공합니다. EphemeralClient는 인메모리로 동작하며 테스트용입니다. PersistentClient는 로컬 디스크에 데이터를 저장합니다. HttpClient는 별도 서버를 띄워 여러 클라이언트가 접근하는 모드입니다. 내부적으로 HNSW 기반(hnswlib)의 인덱스를 사용합니다.

Chroma는 RAG 프로토타입이나 개인 프로젝트처럼 빠르게 시작해야 하는 상황에 적합합니다. 반면 수십억 벡터 규모나 멀티 테넌트 엔터프라이즈 환경에는 적합하지 않습니다. 분산 배포나 복잡한 운영 기능이 제한적이기 때문입니다.

여섯 가지 제품의 특성을 한눈에 비교하면 다음과 같습니다.

| 제품 | 성격 | 인덱스 | 규모 | 관리 방식 | 주요 강점 |
|------|------|--------|------|-----------|-----------|
| Pinecone | 클라우드 SaaS | 자체(HNSW 계열) | 중~대 | 완전 관리형 | 운영 부담 최소화 |
| Weaviate | 오픈소스 + 관리형 | HNSW | 중~대 | 자체/관리형 | 하이브리드 검색, GraphQL |
| Qdrant | 오픈소스 + 관리형 | HNSW(Filterable) | 중~대 | 자체/관리형 | 필터 검색, Rust 성능 |
| FAISS | 라이브러리 | Flat/IVF/HNSW/PQ | 소~대 | 직접 통합 | 알고리즘 제어, GPU |
| Milvus | 오픈소스 분산 | HNSW/IVF/GPU 등 | 대~초대 | Kubernetes | 수평 확장, 초대규모 |
| Chroma | 오픈소스 경량 | HNSW(hnswlib) | 소~중 | 로컬/서버 | 개발자 UX, 프레임워크 통합 |

제품 선택은 팀의 상황에 따라 달라집니다. 인프라를 직접 관리할 여력이 없다면 Pinecone 같은 관리형 서비스가 유리합니다. 벡터 검색과 키워드 검색을 함께 쓰고 싶다면 Weaviate나 Qdrant가 적합합니다. 인덱스 알고리즘을 세밀하게 실험해야 한다면 FAISS가 좋은 선택입니다. 수십억 벡터를 다루는 대규모 시스템이라면 Milvus가 적합합니다. RAG 프로토타입을 빠르게 만들고 싶다면 Chroma로 시작하는 것이 편리합니다.

정리하면, 벡터 데이터베이스는 임베딩 벡터를 저장하고 ANN 인덱싱으로 의미 유사도 검색을 밀리초 안에 수행하는 특수 목적 저장소입니다. HNSW는 다층 그래프 탐색으로 높은 정확도와 빠른 속도를 동시에 달성하며, IVF는 클러스터 기반 파티셔닝으로 메모리 효율이 높습니다. Pinecone, Weaviate, Qdrant, FAISS, Milvus, Chroma는 각각 관리 편의성, 하이브리드 검색, 필터 성능, 알고리즘 제어, 분산 확장, 개발 편의성에서 강점을 가지며, 팀의 규모와 요구사항에 따라 선택합니다.

다음 단원인 1.3.1에서는 Kafka 아키텍처 개요를 다룹니다.

이 단원을 마치면 벡터 데이터베이스의 역할과 주요 제품의 특성을 비교할 수 있습니다.
