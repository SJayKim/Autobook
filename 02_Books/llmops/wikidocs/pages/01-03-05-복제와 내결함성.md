# 1.3.5 복제와 내결함성

1.3.4에서 컨슈머 그룹이 파티션을 나누어 병렬로 메시지를 처리하는 구조를 살펴보았습니다. 그런데 파티션 데이터가 단 하나의 브로커에만 존재한다면, 그 브로커가 디스크 장애나 네트워크 단절로 멈추는 순간 해당 파티션의 데이터는 사라집니다. 아직 컨슈머가 읽지 못한 메시지도, 이미 기록을 마친 메시지도 모두 잃게 됩니다. 분산 시스템에서 개별 노드의 장애는 '만약'이 아니라 '언제'의 문제입니다. Kafka는 이 문제를 파티션 복제로 해결합니다. 같은 파티션의 사본을 여러 브로커에 나누어 저장하고, 하나가 죽더라도 다른 사본이 즉시 역할을 대신하는 구조입니다.

복제를 이해하려면 먼저 파티션 안에서 역할이 어떻게 나뉘는지 알아야 합니다. Kafka의 각 파티션에는 **리더 파티션**(leader partition)이 정확히 하나 존재합니다. 프로듀서가 메시지를 보내면 리더 파티션이 이를 수신하여 자신의 로컬 로그에 기록합니다. 컨슈머가 메시지를 읽을 때에도 리더 파티션에 요청을 보냅니다. 읽기와 쓰기 모두 리더를 통해서만 이루어지는 것입니다.

리더 외에 나머지 사본을 **팔로워 파티션**(follower partition)이라 합니다. 팔로워는 프로듀서나 컨슈머의 요청을 직접 처리하지 않습니다. 팔로워의 유일한 역할은 리더의 데이터를 복제하는 것입니다. 복제 방식은 컨슈머가 메시지를 가져오는 방식과 동일합니다. 팔로워가 리더에게 Fetch 요청을 보내면서 자신의 현재 오프셋을 함께 전달합니다. 리더는 해당 오프셋 이후의 메시지를 팔로워에게 돌려줍니다. 이 과정이 주기적으로 반복되면서 팔로워의 로그가 리더와 동일한 상태를 유지합니다. 리더는 Fetch 요청에 담긴 오프셋을 보고 각 팔로워가 얼마나 뒤처져 있는지 파악할 수 있습니다.

```
  파티션 P0의 복제 구조 (Replication Factor = 3)

  브로커 1             브로커 2             브로커 3
  +---------------+    +---------------+    +---------------+
  | P0 (리더)     |    | P0 (팔로워)   |    | P0 (팔로워)   |
  | 오프셋 0~99   |    | 오프셋 0~99   |    | 오프셋 0~97   |
  +---------------+    +---------------+    +---------------+
        ^                    |                    |
        |   Fetch 요청       |   Fetch 요청       |
        +--------------------+--------------------+
```

위 그림에서 브로커 2의 팔로워는 리더와 동일하게 오프셋 99까지 복제를 마쳤습니다. 브로커 3의 팔로워는 오프셋 97까지만 가져왔으므로 두 건이 뒤처진 상태입니다.

파티션의 사본을 몇 벌 유지할지 결정하는 설정이 **Replication Factor**입니다. 토픽을 생성할 때 정수 값으로 지정하며, 해당 토픽의 모든 파티션에 동일하게 적용됩니다. Replication Factor가 3이면 리더 1개와 팔로워 2개, 총 3벌의 사본이 서로 다른 브로커에 분산 저장됩니다. 이 경우 브로커 2대가 동시에 죽더라도 나머지 1대에 데이터가 남아 있습니다. 운영 환경에서는 Replication Factor 3이 표준 권고값입니다. 브로커가 3대 미만인 클러스터에서는 이 값을 3으로 설정할 수 없습니다. 사본을 배치할 브로커가 부족하기 때문입니다.

Replication Factor가 높을수록 장애 허용 범위가 넓어지지만, 그만큼 저장 공간과 네트워크 대역폭이 늘어납니다. 반대로 너무 낮으면 브로커 하나의 장애에도 데이터를 잃을 수 있습니다. 이 균형점을 잡기 위해 대부분의 운영 환경은 3을 선택합니다.

사본이 여러 벌 있다 해도, 모든 팔로워가 리더와 완벽히 동기화되어 있다고 보장하기 어렵습니다. 네트워크 지연이나 브로커 부하로 인해 일부 팔로워가 뒤처질 수 있습니다. Kafka는 이 상황을 구분하기 위해 **ISR**(In-Sync Replicas)이라는 개념을 사용합니다. ISR은 리더와 충분히 동기화된 복제본의 집합입니다. 리더 자신도 ISR에 포함됩니다.

'충분히 동기화'의 기준은 replica.lag.time.max.ms 설정으로 결정됩니다. 기본값은 30초입니다. 팔로워가 이 시간 안에 리더로부터 데이터를 가져가지 못하면, 리더는 해당 팔로워를 ISR에서 제외합니다. 반대로, ISR에서 빠진 팔로워가 이후 리더를 다시 따라잡으면 ISR에 재합류합니다. ISR 목록은 동적으로 변하며, 변경 사항은 클러스터 메타데이터에 즉시 반영됩니다.

ISR이 중요한 이유는 '커밋'의 기준이 되기 때문입니다. Kafka에서 메시지가 '커밋되었다'는 것은 ISR 내 모든 복제본이 해당 메시지를 받았다는 뜻입니다. 커밋된 메시지의 오프셋 경계를 **High Watermark**라 합니다. 컨슈머는 High Watermark 이하의 메시지만 읽을 수 있습니다. 아직 ISR 전체에 복제되지 않은 메시지는 컨슈머에게 노출되지 않습니다. 이 구조 덕분에 리더에 장애가 발생하더라도, 커밋된 메시지는 ISR 내 다른 복제본에 반드시 존재합니다.

ISR만으로는 내결함성을 완전히 보장하지 못합니다. 팔로워가 하나씩 ISR에서 빠져나가 리더만 남으면, 리더 한 대에 기록된 메시지를 '커밋'이라 부르게 됩니다. 이 상태에서 리더가 죽으면 데이터가 사라집니다. 이 위험을 막는 설정이 **min.insync.replicas**입니다.

min.insync.replicas는 브로커 또는 토픽 단위로 지정하는 정수 값입니다. acks=all로 들어온 쓰기 요청을 수락하려면, ISR에 이 값 이상의 복제본이 있어야 합니다. 예를 들어 min.insync.replicas를 2로 설정했는데 ISR에 리더 한 대만 남아 있다면, 브로커는 NotEnoughReplicasException을 반환하고 쓰기를 거부합니다. 데이터를 기록할 수 없으니 불편하지만, 기록된 척하고 유실되는 것보다 안전합니다.

min.insync.replicas는 단독으로는 효과가 없습니다. 이 설정은 acks=all과 함께 사용할 때만 의미를 가집니다. acks 설정을 먼저 이해해야 min.insync.replicas의 역할이 명확해집니다.

**acks**는 프로듀서 측 설정입니다. 프로듀서가 메시지를 보낸 뒤, 브로커로부터 어떤 수준의 확인응답을 기다릴지를 결정합니다. 세 가지 값이 있습니다.

acks=0으로 설정하면, 프로듀서는 브로커의 응답을 기다리지 않습니다. 메시지를 네트워크로 보낸 직후 다음 메시지를 전송합니다. 가장 빠르지만, 브로커가 메시지를 받지 못했어도 프로듀서는 알 수 없습니다. 메시지 유실 가능성이 가장 높습니다.

acks=1로 설정하면, 리더 브로커가 자신의 로컬 로그에 메시지를 기록한 뒤 확인응답을 보냅니다. 리더가 기록을 마쳤으므로 acks=0보다 안전합니다. 그러나 리더가 확인응답을 보낸 직후 장애를 일으키면, 팔로워에 아직 복제되지 않은 메시지가 유실됩니다.

acks=all(또는 acks=-1)로 설정하면, 리더가 ISR 내 모든 복제본의 복제 완료를 확인한 뒤에야 프로듀서에 확인응답을 반환합니다. 가장 느리지만 가장 강한 내구성을 보장합니다. 실제로 acks=all의 성능 오버헤드는 미미한 수준이라고 알려져 있습니다.

```
  acks 설정별 동작 비교

  acks=0
  프로듀서 --메시지--> 리더    (응답 안 기다림)
                       팔로워  (복제 여부 무관)

  acks=1
  프로듀서 --메시지--> 리더 --ack--> 프로듀서
                       팔로워  (복제 완료 전에 ack 발생)

  acks=all
  프로듀서 --메시지--> 리더 --> ISR 팔로워 전체 복제 완료
                       리더 --ack--> 프로듀서
```

이제 min.insync.replicas와 acks=all이 결합되는 방식이 분명해집니다. acks=all은 ISR 전체가 복제를 마쳐야 ack를 보내겠다는 뜻입니다. min.insync.replicas는 그 ISR에 최소 몇 개가 있어야 쓰기를 허용하겠다는 뜻입니다. 두 설정이 함께 작동하여 '복제본이 충분하지 않으면 아예 쓰지 못하게 막는' 안전장치를 구성합니다.

운영 환경에서 널리 쓰이는 표준 조합은 다음 세 가지 설정을 함께 사용하는 것입니다. Replication Factor를 3으로, min.insync.replicas를 2로, acks를 all로 설정합니다. 이 조합에서 브로커 1대가 장애를 일으키면 ISR에 2대가 남으므로 min.insync.replicas 조건을 만족합니다. 쓰기가 정상적으로 계속됩니다. 브로커 2대가 동시에 장애를 일으키면 ISR에 1대만 남아 min.insync.replicas 조건을 충족하지 못합니다. 쓰기가 차단됩니다. 데이터를 유실하는 대신 가용성을 포기하여 안전하게 실패하는 것입니다.

```
  표준 내결함성 조합의 장애 시나리오

  설정: RF=3, min.insync.replicas=2, acks=all

  정상 상태        브로커 1대 장애     브로커 2대 장애
  ISR = 3          ISR = 2             ISR = 1
  쓰기: 성공       쓰기: 성공          쓰기: 거부
  읽기: 성공       읽기: 성공          읽기: 성공 (커밋된 데이터)
  데이터: 안전     데이터: 안전        데이터: 안전 (신규 쓰기 불가)
```

마지막으로 **리더 선출**(leader election)을 다룹니다. 리더 브로커가 장애를 일으키면 해당 파티션의 읽기와 쓰기가 모두 멈춥니다. 이 상황을 빠르게 복구하려면 남은 복제본 중 하나를 새 리더로 세워야 합니다.

리더 선출은 Kafka 컨트롤러가 담당합니다. 컨트롤러는 파티션의 ISR 목록을 확인하고, ISR 안에 있는 팔로워 중 하나를 새 리더로 선출합니다. ISR 내 팔로워는 커밋된 메시지를 모두 보유하고 있으므로, 이 방식으로 리더를 교체하면 데이터 손실이 발생하지 않습니다.

그런데 극단적인 상황이 있습니다. ISR에 리더만 있었고 그 리더가 죽은 경우, ISR이 비게 됩니다. 새 리더로 세울 후보가 ISR 안에 없습니다. 이때 Kafka는 두 가지 선택지를 제공합니다.

첫 번째 선택지는 ISR 외부의 팔로워를 리더로 세우는 것입니다. 이를 Unclean Leader Election이라 합니다. unclean.leader.election.enable 설정으로 제어하며, 기본값은 false입니다. 이 설정을 true로 바꾸면, ISR 밖의 팔로워가 리더가 됩니다. 파티션이 다시 읽기와 쓰기를 처리할 수 있으므로 가용성은 회복됩니다. 그러나 이 팔로워는 리더를 완전히 따라잡지 못한 상태였으므로, 아직 복제되지 않은 메시지가 영구적으로 유실됩니다.

두 번째 선택지는 ISR 외부 선출을 허용하지 않는 것입니다. 기본값인 false를 유지하면, ISR이 빌 경우 해당 파티션은 리더 없이 대기합니다. 읽기와 쓰기 모두 불가능하지만, 장애가 복구되어 이전 리더가 돌아오면 데이터 손실 없이 서비스가 재개됩니다. 운영 환경에서는 데이터 정합성을 우선하여 기본값 false를 유지하는 것이 권고됩니다.

KRaft 모드에서는 ZooKeeper 대신 Raft 합의 프로토콜 기반의 컨트롤러 쿼럼이 리더 선출을 관리합니다. 이 구조는 1.3.6에서 자세히 다룹니다. 여기서는 KRaft 모드의 리더 선출이 기존 ZooKeeper 방식보다 지연이 낮다는 점만 짚어 둡니다.

정리하면, Kafka는 파티션을 여러 브로커에 복제하여 내결함성을 확보합니다. 각 파티션에는 읽기와 쓰기를 담당하는 리더 하나와, 리더의 데이터를 Fetch로 복제하는 팔로워가 있습니다. Replication Factor가 사본 수를 결정하고, ISR이 리더와 동기화된 복제본 집합을 관리합니다. acks=all과 min.insync.replicas=2를 함께 설정하면, 복제본이 부족할 때 쓰기를 거부하여 데이터 유실을 방지합니다. 리더 장애 시에는 ISR 내 팔로워가 새 리더로 선출되어 서비스를 이어받습니다.

다음 단원인 1.3.6에서는 KRaft 모드와 컨트롤러 쿼럼을 다룹니다. ZooKeeper 없이 Kafka 자체의 Raft 프로토콜로 클러스터 메타데이터를 관리하는 구조를 살펴봅니다.

이 단원을 마치면 Kafka의 리더-팔로워 복제 메커니즘과 ISR이 내결함성을 보장하는 방식을 설명할 수 있습니다.
