# 7.2.2 ELSER vs dense 벡터 비교

7.2.1에서 ELSER가 희소 벡터를 통해 시맨틱 검색을 수행하는 원리를 살펴봤습니다. 7.1.1에서는 dense_vector를 이용한 밀집 벡터 검색을 다뤘습니다. 두 방식 모두 키워드 일치가 아닌 의미 기반 검색을 목표로 하지만, 내부 동작과 리소스 사용 패턴이 다릅니다. 이 단원에서는 두 방식을 여러 측면에서 비교하여, 어떤 상황에 어떤 방식을 선택하면 좋을지 판단할 수 있도록 정리합니다.

먼저 가장 눈에 띄는 차이는 **사전 준비의 복잡도**입니다. ELSER는 Elastic이 미리 학습시켜 제공하는 모델입니다. 별도의 도메인별 미세 조정 없이 바로 배포하여 사용할 수 있습니다. 영어 텍스트에 대해서는 즉시 높은 수준의 시맨틱 검색 품질을 기대할 수 있습니다. 반면 dense 벡터 방식은 먼저 어떤 임베딩 모델을 사용할지 선택해야 합니다. 범용 모델을 그대로 쓸 수도 있지만, 도메인에 특화된 검색 품질을 원하면 해당 분야 데이터로 모델을 미세 조정하는 과정이 필요합니다.

이 차이는 **다국어 지원**에서 뚜렷해집니다. ELSER는 영어 텍스트에 최적화되어 있습니다. 한국어, 일본어, 중국어 등 다른 언어에 대해서는 성능이 보장되지 않습니다. dense 벡터 방식은 다국어를 지원하는 임베딩 모델을 선택하면 여러 언어에 걸쳐 시맨틱 검색을 수행할 수 있습니다. 한국어 텍스트가 포함된 서비스라면 dense 벡터 방식이 더 적합합니다.

**인덱싱 속도**도 차이가 있습니다. ELSER는 입력 텍스트를 분석하여 확장 토큰을 생성하는 과정이 인덱싱마다 수행됩니다. 이 과정은 비교적 빠르지만, 확장 토큰의 수가 텍스트 길이에 따라 달라지므로 긴 문서에서는 시간이 더 걸립니다. dense 벡터 방식은 임베딩 모델의 추론 시간이 인덱싱 속도를 좌우합니다. 차원 수가 크고 복잡한 모델일수록 추론이 느려집니다. 일반적으로 ELSER가 중간 규모의 dense 벡터 모델과 비슷하거나 약간 빠른 인덱싱 속도를 보입니다.

**검색 레이턴시**에서는 구조적 차이가 영향을 줍니다. ELSER의 희소 벡터는 Elasticsearch의 역색인 구조를 활용합니다. 역색인에서 토큰을 조회하는 것은 Elasticsearch가 가장 잘하는 작업이므로, 검색 레이턴시가 안정적입니다. dense 벡터 검색은 HNSW 그래프 탐색에 기반합니다. 그래프가 메모리에 올라가 있으면 빠르지만, 벡터 수가 많아 메모리 부족이 발생하면 성능이 급격히 떨어질 수 있습니다.

**메모리와 스토리지 사용량**은 두 방식이 확연히 다릅니다. dense 벡터는 7.1.1에서 다뤘듯이 벡터 수 곱하기 4바이트 곱하기 (차원 수 + 12)만큼의 오프힙 메모리를 사용합니다. 768차원 벡터 1천만 건이면 약 31GB입니다. ELSER는 역색인에 확장 토큰을 저장하므로, 인덱스 크기는 원본 텍스트 인덱스보다 커지지만 dense 벡터처럼 별도의 대규모 오프힙 메모리를 요구하지는 않습니다. 단, ELSER 모델 자체를 ML 노드 메모리에 올려야 합니다.

**재현율(recall) 벤치마크** 결과를 보면, ELSER는 12개 주요 벤치마크 중 11개에서 BM25보다 높은 성능을 보입니다. dense 벡터 방식은 사용하는 모델에 따라 결과가 크게 달라집니다. 범용 모델은 ELSER와 비슷하거나 약간 낮을 수 있고, 도메인에 맞게 미세 조정한 모델은 ELSER를 능가할 수 있습니다.

이 비교 내용을 표로 정리하겠습니다.

```
항목                 ELSER (희소 벡터)              dense 벡터
─────────────────────────────────────────────────────────────
사전 준비             즉시 사용 가능                모델 선택/배포 필요
도메인 미세 조정      불필요                        선택적 (정확도 향상)
다국어 지원           영어 최적화                   다국어 모델 선택 가능
인덱싱 속도           중간                          모델 복잡도에 비례
검색 레이턴시         안정적 (역색인 기반)          메모리 의존 (HNSW 기반)
오프힙 메모리         낮음                          높음 (차원 x 벡터 수)
스토리지              중간 (확장 토큰 저장)         벡터 크기에 비례
재현율                BM25 대비 높음 (영어)         모델에 따라 다름
해석 가능성           높음 (토큰이 사람이 읽힘)     낮음 (숫자 배열)
─────────────────────────────────────────────────────────────
```

**해석 가능성**(interpretability)도 실무에서 중요한 차이입니다. ELSER가 생성하는 확장 토큰은 사람이 읽을 수 있는 단어입니다. 검색 결과가 왜 나왔는지 토큰을 확인하면 직관적으로 이해할 수 있습니다. dense 벡터는 수백 개의 숫자로 이루어져 있어, 두 벡터가 왜 가까운지 사람이 해석하기 어렵습니다.

이 비교를 바탕으로 사용 사례별 선택 가이드를 정리합니다.

영어 텍스트 중심의 서비스에서 빠르게 시맨틱 검색을 도입하고 싶다면 ELSER가 적합합니다. 모델 선택이나 미세 조정 없이 바로 배포하여 BM25보다 높은 검색 품질을 얻을 수 있습니다.

한국어를 포함한 다국어 서비스라면 dense 벡터 방식을 선택합니다. multilingual-e5-large 같은 다국어 임베딩 모델을 사용하면 언어에 관계없이 의미 기반 검색이 가능합니다.

도메인 특화 검색(의료, 법률, 특허 등)에서 최고 수준의 정확도가 필요하다면 dense 벡터 방식으로 해당 도메인 데이터에 미세 조정한 모델을 사용합니다. ELSER는 범용 학습 데이터 기반이므로 특수 도메인에서는 한계가 있을 수 있습니다.

메모리 예산이 제한적이라면 ELSER가 유리합니다. dense 벡터는 벡터 수에 비례하여 대량의 오프힙 메모리를 사용하므로, 대규모 인덱스에서는 비용이 크게 증가합니다.

두 방식을 배타적으로 선택할 필요는 없습니다. 7.3에서 다룰 하이브리드 검색에서는 BM25, ELSER, dense 벡터 검색을 결합하여 각각의 장점을 모두 활용할 수 있습니다.

정리하면, ELSER는 영어 환경에서 사전 준비 없이 높은 검색 품질을 제공하며, 역색인 기반으로 메모리 효율이 좋습니다. dense 벡터는 다국어 지원과 도메인 미세 조정이 가능하지만, 오프힙 메모리 사용량이 크고 모델 선택과 배포의 복잡도가 있습니다. 서비스의 언어, 도메인, 인프라 예산에 따라 적합한 방식이 달라집니다.

다음 단원인 7.2.3에서는 ELSER 모델을 실제로 배포하고 semantic_text 필드를 사용하여 시맨틱 검색을 구현하는 방법을 다룹니다.

이 단원을 마치면 ELSER와 dense 벡터 검색의 성능, 정확도, 리소스 트레이드오프를 비교할 수 있습니다.
