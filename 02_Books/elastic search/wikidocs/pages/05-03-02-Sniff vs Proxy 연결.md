# 5.3.2 Sniff 모드 vs Proxy 모드

5.3.1에서 CCS를 사용하기 위해 원격 클러스터를 등록하고, seeds 주소를 지정하는 방법을 살펴보았습니다. 그런데 seeds 주소로 접속한 뒤 실제 검색 요청이 원격 클러스터의 노드에 어떻게 전달되는지는 다루지 않았습니다. Elasticsearch는 원격 클러스터에 연결하는 두 가지 방식을 제공합니다. **Sniff 모드**와 **Proxy 모드**입니다. 네트워크 환경에 따라 적합한 모드가 다르므로, 각 모드의 동작 원리와 선택 기준을 이해해야 합니다.

**Sniff 모드**는 기본 연결 방식입니다. "sniff"는 '냄새를 맡다', 즉 탐색한다는 뜻입니다. 로컬 클러스터의 코디네이팅 노드가 seed 주소로 원격 클러스터에 접속한 뒤, 원격 클러스터의 전체 노드 목록을 자동으로 발견합니다. 이후 검색 요청을 보낼 때, 발견한 노드 중 적절한 노드에 직접 연결하여 요청을 전달합니다.

동작을 단계별로 풀어 봅니다.

```
Sniff 모드 동작 흐름

1. 로컬 노드가 seed 주소(예: east-node-1:9300)에 접속
2. east-node-1로부터 원격 클러스터의 전체 노드 목록 수신
   (east-node-1, east-node-2, east-node-3)
3. 로컬 노드가 원격 노드 각각에 직접 연결 확립
4. CCS 쿼리 시, 관련 샤드가 있는 원격 노드에 직접 요청 전송
```

Sniff 모드의 장점은 로컬 노드가 원격 클러스터의 개별 노드에 직접 통신하므로, 중간 경유 없이 효율적으로 요청을 분산한다는 것입니다. 원격 클러스터에 노드가 추가되거나 제거되면 자동으로 감지하여 연결 목록을 갱신합니다.

그러나 Sniff 모드에는 네트워크 전제 조건이 있습니다. 로컬 클러스터의 노드가 원격 클러스터의 모든 노드에 직접 네트워크 접근이 가능해야 합니다. 이 조건이 문제가 되는 환경이 있습니다. 원격 클러스터가 사설 네트워크(private network) 안에 있고, 로컬 클러스터에서 직접 접근할 수 없는 경우입니다. NAT(Network Address Translation) 뒤에 있거나, 방화벽이 특정 포트만 허용하는 환경에서는 원격 클러스터의 개별 노드 주소로 연결할 수 없습니다.

이런 환경에서 사용하는 것이 **Proxy 모드**입니다. Proxy 모드에서는 로컬 노드가 원격 클러스터의 개별 노드에 직접 연결하지 않습니다. 대신 하나의 고정된 프록시 주소를 통해 모든 통신이 이루어집니다. 프록시는 리버스 프록시나 로드 밸런서로, 원격 클러스터 앞단에 위치합니다.

```
Proxy 모드 동작 흐름

로컬 클러스터                   프록시           원격 클러스터
+------------+                +-------+        +-----------+
| Local Node | ---9300--->    | Proxy | -----> | East-1    |
|            |                |       | -----> | East-2    |
|            |                |       | -----> | East-3    |
+------------+                +-------+        +-----------+

로컬 노드는 프록시 주소만 알면 됨
프록시가 요청을 원격 노드에 분배
```

Proxy 모드의 설정 방법은 다음과 같습니다.

```json
PUT _cluster/settings
{
  "persistent": {
    "cluster": {
      "remote": {
        "cluster-east": {
          "mode": "proxy",
          "proxy_address": "proxy.east.example.com:9443",
          "server_name": "cluster-east.example.com"
        }
      }
    }
  }
}
```

각 필드를 풀어 봅니다. "mode"를 "proxy"로 지정하여 Proxy 모드를 사용함을 선언합니다. Sniff 모드가 기본이므로, Proxy 모드를 사용하려면 명시적으로 지정해야 합니다. "proxy_address"는 프록시 서버의 주소와 포트입니다. 로컬 노드가 이 주소로만 연결을 시도합니다. "server_name"은 TLS(전송 계층 보안) 연결 시 SNI(Server Name Indication)에 사용되는 이름입니다. 프록시가 여러 클러스터를 중개할 때, 이 이름으로 대상 클러스터를 구분합니다.

두 모드를 비교 정리합니다.

```
Sniff 모드 vs Proxy 모드 비교

+--------------------+---------------------------+---------------------------+
|                    | Sniff 모드                | Proxy 모드                |
+--------------------+---------------------------+---------------------------+
| 연결 대상          | 원격 클러스터의 개별 노드  | 프록시 서버 1대            |
+--------------------+---------------------------+---------------------------+
| 노드 발견          | 자동 (seed에서 전체 목록)  | 불필요 (프록시가 분배)     |
+--------------------+---------------------------+---------------------------+
| 네트워크 요구사항  | 모든 원격 노드에 직접 접근  | 프록시 주소에만 접근       |
+--------------------+---------------------------+---------------------------+
| 방화벽 친화성      | 낮음 (다수 포트/IP 필요)   | 높음 (단일 주소만 허용)    |
+--------------------+---------------------------+---------------------------+
| 성능               | 직접 연결로 지연 최소화    | 프록시 경유로 약간의 지연  |
+--------------------+---------------------------+---------------------------+
| 적합한 환경        | 동일 네트워크, VPC 피어링  | NAT, 방화벽, 멀티 클라우드 |
+--------------------+---------------------------+---------------------------+
```

Sniff 모드는 로컬 클러스터와 원격 클러스터가 같은 네트워크에 있거나, VPC 피어링(가상 사설 네트워크 간 연결) 등으로 상호 접근이 가능한 환경에서 선택합니다. 중간 경유가 없으므로 네트워크 지연이 최소입니다.

Proxy 모드는 원격 클러스터가 방화벽이나 NAT 뒤에 있어 개별 노드에 직접 접근할 수 없는 환경에서 선택합니다. 클라우드 환경에서 서로 다른 VPC나 리전에 있는 클러스터를 연결할 때 흔히 사용됩니다. 프록시 서버의 주소 하나만 방화벽에서 허용하면 되므로 보안 구성이 단순해집니다.

CCS에서 **TLS 연동**도 실무에서 중요합니다. 클러스터 간 통신이 공용 네트워크를 거치는 경우, 전송 데이터를 암호화해야 합니다. Elasticsearch의 transport 레이어에 TLS를 설정하면, CCS 연결도 자동으로 암호화됩니다. Proxy 모드에서는 server_name 필드가 TLS 핸드셰이크의 SNI 값으로 사용되므로, 인증서의 SAN(Subject Alternative Name)과 일치해야 합니다.

마지막으로 **CCS의 성능 고려사항**을 짚습니다. CCS는 네트워크를 넘나드는 통신이므로, 클러스터 간 네트워크 레이턴시가 검색 응답 시간에 직접 영향을 미칩니다. 5.3.1에서 CCS 응답의 _clusters.details에서 각 클러스터의 took 값을 확인할 수 있다고 했는데, 원격 클러스터의 took 값이 로컬보다 현저히 크다면 네트워크 지연을 의심해야 합니다.

CCS는 네트워크 왕복을 최소화하기 위해 **minimize_roundtrips** 옵션을 제공합니다. 이 옵션이 true이면, 코디네이팅 노드가 원격 클러스터에 한 번의 요청으로 완전한 결과를 받아옵니다. false이면, 샤드별로 개별 요청을 보내므로 왕복 횟수가 늘어나지만 더 세밀한 제어가 가능합니다. 동기 검색(sync search)에서는 기본값이 true, 비동기 검색(async search)에서는 false입니다. 네트워크 레이턴시가 높은 환경에서는 minimize_roundtrips를 true로 유지하는 것이 유리합니다.

```
GET cluster-east:logs-*/_search?ccs_minimize_roundtrips=true
```

5.3.1에서 다룬 skip_unavailable 설정과 함께, CCS의 네트워크 환경에 맞게 모드와 옵션을 조합하는 것이 안정적인 CCS 운영의 핵심입니다.

정리하면, Sniff 모드는 seed 노드에서 원격 클러스터의 전체 노드 목록을 발견하여 직접 연결하는 방식이며, 네트워크 지연이 적지만 모든 노드에 직접 접근이 가능해야 합니다. Proxy 모드는 하나의 프록시 주소를 경유하여 통신하며, NAT나 방화벽 환경에서 유리합니다. TLS로 클러스터 간 통신을 암호화하고, ccs_minimize_roundtrips 옵션으로 네트워크 왕복을 최적화합니다.

다음 단원인 5.3.3에서는 멀티 리전 시나리오를 다룹니다. 여러 지역에 클러스터를 분산 배치하고 CCS와 CCR을 활용하여 글로벌 서비스를 구성하는 전략을 살펴봅니다.

이 단원을 마치면 Sniff 모드와 Proxy 모드의 네트워크 접근 방식 차이를 설명할 수 있으며, 방화벽 환경에서 Proxy 모드를 적용할 수 있습니다.
