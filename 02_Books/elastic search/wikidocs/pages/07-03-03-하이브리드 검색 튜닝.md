# 7.3.3 하이브리드 검색 튜닝

7.3.2에서 retriever API로 BM25와 kNN을 결합하는 하이브리드 검색을 구현했습니다. 기본 설정만으로도 단일 검색 방식보다 나은 결과를 얻을 수 있지만, 데이터 특성과 사용자 쿼리 패턴에 맞게 파라미터를 조정하면 검색 품질을 더 높일 수 있습니다. 이 단원에서는 하이브리드 검색의 핵심 파라미터를 조정하는 방법과, 검색 품질을 정량적으로 평가하는 방법을 다룹니다.

가장 먼저 조정할 파라미터는 **rank_constant**입니다. 7.3.1에서 설명했듯이, rank_constant(k)는 RRF 공식에서 순위에 더해지는 상수입니다. 이 값에 따라 상위 순위 문서와 하위 순위 문서 사이의 점수 격차가 달라집니다.

rank_constant가 작으면 상위 순위 문서에 점수가 집중됩니다. 1등과 5등의 차이가 크게 벌어집니다. 하나의 검색 방식에서 압도적으로 높은 순위를 받은 문서가 최종 결과에서도 유리해집니다.

rank_constant가 크면 순위 간 점수 차이가 줄어듭니다. 여러 검색 방식에 고루 등장하는 문서가 유리해집니다. 한쪽에서만 높은 순위를 받은 문서보다, 양쪽에서 중간 순위를 받은 문서가 올라갈 수 있습니다.

```
rank_constant = 1 일 때:
  1등 점수: 1/(1+1) = 0.500
  5등 점수: 1/(1+5) = 0.167    (1등 대비 33%)

rank_constant = 60 일 때 (기본값):
  1등 점수: 1/(60+1) = 0.01639
  5등 점수: 1/(60+5) = 0.01538  (1등 대비 94%)

rank_constant = 200 일 때:
  1등 점수: 1/(200+1) = 0.00498
  5등 점수: 1/(200+5) = 0.00488 (1등 대비 98%)
```

기본값 60은 대부분의 상황에서 잘 동작합니다. 특정 검색 방식의 결과를 더 신뢰하고 싶다면 rank_constant를 줄이고, 모든 검색 방식의 기여를 균등하게 반영하고 싶다면 값을 늘립니다.

다음으로 **num_candidates**를 조정합니다. num_candidates는 kNN retriever에서 HNSW 그래프 탐색 시 고려하는 후보 수입니다. 이 값을 늘리면 더 넓은 범위에서 후보를 탐색하므로 재현율(recall)이 향상됩니다. 7.1.2에서 설명한 것처럼 k의 5~10배가 일반적인 시작점이지만, 하이브리드 검색에서는 rank_window_size와의 관계도 고려해야 합니다.

num_candidates는 rank_window_size 이상으로 설정하는 것이 좋습니다. num_candidates가 rank_window_size보다 작으면, kNN retriever가 충분한 후보를 제공하지 못해 RRF 병합에 참여하는 kNN 측 문서가 줄어듭니다.

하이브리드 검색의 품질을 정량적으로 평가하려면 검색 품질 지표를 사용합니다. 대표적인 지표 두 가지를 소개합니다.

**NDCG**(Normalized Discounted Cumulative Gain)는 검색 결과의 순서까지 고려하는 지표입니다. 관련성이 높은 문서가 상위에 있을수록 점수가 높아집니다. 1에 가까울수록 좋은 결과입니다. 검색 결과의 상위 k개만 평가하는 NDCG@k를 주로 사용합니다. 예를 들어 NDCG@10은 상위 10개 결과의 품질을 측정합니다.

**MAP**(Mean Average Precision)은 관련 문서가 결과 목록의 어디에 위치하는지를 종합적으로 평가합니다. 관련 문서가 상위에 몰려 있으면 점수가 높고, 하위에 흩어져 있으면 점수가 낮습니다. 여러 쿼리에 대한 AP(Average Precision)의 평균을 구합니다.

이 지표들을 계산하려면 **평가 데이터셋**이 필요합니다. 평가 데이터셋은 쿼리와 해당 쿼리에 대한 관련 문서 목록으로 구성됩니다. 각 관련 문서에는 관련성 등급(예: 0=무관, 1=부분 관련, 2=매우 관련)을 부여합니다.

평가 데이터셋을 구성하는 방법은 두 가지입니다. 첫째, 사람이 직접 쿼리별로 관련 문서를 판정하는 방법입니다. 정확하지만 시간과 비용이 많이 듭니다. 둘째, 실제 서비스의 클릭 로그를 활용하는 방법입니다. 사용자가 검색 후 클릭한 문서를 관련 문서로 간주합니다. 데이터가 많을수록 신뢰도가 올라갑니다.

평가 데이터셋이 준비되면 Elasticsearch의 **Rank Evaluation API**를 사용하여 검색 품질을 자동으로 측정할 수 있습니다.

```json
GET my-hybrid-index/_rank_eval
{
  "requests": [
    {
      "id": "query_1",
      "request": {
        "query": {
          "match": { "content": "분산 검색 엔진" }
        }
      },
      "ratings": [
        { "_index": "my-hybrid-index", "_id": "doc1", "rating": 2 },
        { "_index": "my-hybrid-index", "_id": "doc5", "rating": 1 },
        { "_index": "my-hybrid-index", "_id": "doc8", "rating": 0 }
      ]
    }
  ],
  "metric": {
    "dcg": {
      "k": 10,
      "normalize": true
    }
  }
}
```

requests 배열에 평가할 쿼리와 해당 쿼리의 관련 문서 판정(ratings)을 넣습니다. rating은 관련성 등급입니다. metric에서 dcg를 지정하고 normalize를 true로 설정하면 NDCG를 계산합니다. k는 상위 몇 개의 결과를 평가할지를 지정합니다.

API 응답에는 각 쿼리별 점수와 전체 평균 점수가 포함됩니다. 이 점수를 기준으로 파라미터를 조정하고, 다시 평가하는 과정을 반복하여 최적의 설정을 찾습니다.

실제 튜닝 과정은 다음 단계를 따릅니다.

```
1. 평가 데이터셋 준비 (쿼리 + 관련 문서 판정)
      │
2. 기본 파라미터로 하이브리드 검색 실행
      │
3. _rank_eval API로 NDCG@10 측정
      │
4. rank_constant 또는 num_candidates 조정
      │
5. 다시 _rank_eval로 측정
      │
6. 점수가 개선되면 해당 파라미터 채택
      │
7. 3~6을 반복
```

이 과정은 **오프라인 평가**에 해당합니다. 서비스에 반영하기 전에 미리 준비한 데이터셋으로 검색 품질을 측정하는 방식입니다.

운영 중인 서비스에서는 **온라인 A/B 테스트**를 병행합니다. A/B 테스트는 사용자를 두 그룹으로 나누어 한 그룹에는 기존 설정을, 다른 그룹에는 새 설정을 적용합니다. 클릭률, 전환율, 검색 후 이탈률 같은 실제 사용자 행동 지표를 비교하여 어떤 설정이 더 나은지 판단합니다. 오프라인 평가에서 좋은 점수를 받았더라도 실제 사용자 행동과 다를 수 있으므로, 두 평가를 함께 사용하는 것이 바람직합니다.

정리하면, 하이브리드 검색 튜닝은 rank_constant와 num_candidates 조정이 핵심입니다. rank_constant는 순위 간 점수 격차를 조절하고, num_candidates는 kNN 검색의 recall을 좌우합니다. NDCG와 MAP 같은 지표로 검색 품질을 정량화하고, Rank Evaluation API로 자동 측정합니다. 오프라인 평가로 파라미터를 좁힌 뒤, 온라인 A/B 테스트로 실제 효과를 검증하는 것이 안정적인 튜닝 절차입니다.

다음 단원인 7.4.1에서는 하이브리드 검색을 활용하여 RAG(Retrieval-Augmented Generation) 파이프라인을 구성하는 방법을 다룹니다.

이 단원을 마치면 하이브리드 검색의 rank_constant와 num_candidates를 조정하여 검색 품질을 최적화할 수 있습니다.
