# 1.3.1 JSON 문서 구조

1.1.3에서 PUT 요청으로 문서를 인덱싱하고, GET 요청으로 조회하는 과정을 직접 실습했습니다. 그때 요청 본문에 JSON 객체를 담아 보냈고, 응답에는 _index, _id, _version 같은 밑줄로 시작하는 필드가 함께 돌아왔습니다. 이 밑줄 필드들은 사용자가 정의한 데이터가 아니라 Elasticsearch가 문서를 관리하기 위해 자동으로 붙이는 정보입니다. 이 단원에서는 이 메타데이터 필드의 정체를 하나씩 살펴보고, JSON 문서 안에서 중첩 객체, 배열, null 값이 어떻게 처리되는지를 다룹니다.

Elasticsearch에 저장되는 모든 문서는 JSON 형식입니다. 문서 하나를 인덱싱하면 Elasticsearch는 두 가지 일을 동시에 합니다. 하나는 텍스트 필드의 내용을 분석하여 역색인을 만드는 것이고, 다른 하나는 원본 JSON을 별도의 저장 공간에 보관하는 것입니다. 이 원본 저장 공간이 바로 **_source** 필드입니다. 역색인은 검색 속도를 높이기 위한 구조이므로 원본 텍스트를 그대로 복원할 수 없습니다. _source가 있어야 조회나 검색 결과에서 문서 내용을 온전히 돌려받을 수 있습니다.

_source는 역색인에 포함되지 않으므로 직접 검색 대상이 되지는 않습니다. 그 대신 GET 요청이나 검색 결과의 응답 본문에 항상 포함되어 반환됩니다. 1.1.3에서 GET /my-index/_doc/1 요청의 응답에 _source 아래 원본 JSON이 그대로 담겨 있던 것이 바로 이 메커니즘입니다.

_source를 비활성화할 수도 있습니다. 매핑에서 `"_source": {"enabled": false}`로 설정하면 원본 JSON을 저장하지 않게 됩니다. 디스크 공간은 절약되지만 대가가 큽니다. update API, update_by_query API, reindex API를 모두 사용할 수 없게 됩니다. 검색 결과에서 하이라이팅도 불가능해지고, 나중에 매핑을 변경하여 재인덱싱하는 것도 할 수 없습니다. 원본이 사라지기 때문입니다. 디스크 용량이 부족하다면 _source를 끄는 대신 압축 수준을 높이는 편이 낫습니다.

_source와 관련된 또 다른 선택지로 **합성 _source**(synthetic _source)가 있습니다. `"index.mapping.source.mode": "synthetic"` 설정을 활성화하면 원본 JSON을 그대로 저장하지 않고, 검색 시점에 doc_values와 stored fields에서 원본을 재구성합니다. 디스크 공간은 절약되지만 _source 접근 속도가 느려집니다. 재구성 과정에서 배열 원소가 알파벳순으로 재정렬되거나, 점(.)이 포함된 필드명이 중첩 객체로 변환되는 등 원본과 미세하게 달라질 수 있다는 점도 알아 두어야 합니다.

_source에 저장할 필드를 선별하는 것도 가능합니다. `_source.includes`와 `_source.excludes` 설정으로 특정 필드만 포함하거나 제외할 수 있습니다. 다만 이 방식은 나중에 재인덱싱할 때 제외된 필드가 유실되므로 주의가 필요합니다.

이제 _source 외의 메타데이터 필드를 살펴봅니다. 1.1.3에서 문서를 인덱싱하거나 조회할 때 응답에 항상 포함되던 밑줄 필드들, 즉 **_index**, **_id**, **_version**이 각각 어떤 역할을 하는지 정리합니다.

**_index**는 문서가 속한 인덱스의 이름입니다. 1.1.3에서 PUT /my-index/_doc/1 요청을 보냈을 때, 응답의 _index 값이 "my-index"로 돌아온 것을 기억할 것입니다. 이 필드는 Lucene 내부에서는 실제로 저장되는 필드가 아니라 가상 필드입니다. Elasticsearch가 라우팅이나 검색 범위 지정에 활용합니다. 사용자가 직접 값을 변경하거나 설정할 수 있는 필드는 아닙니다.

**_id**는 인덱스 안에서 문서를 고유하게 식별하는 값입니다. 1.1.3에서 PUT 요청의 URL 끝에 '/1'을 붙여 ID를 직접 지정하는 방법과, POST 요청으로 ID를 생략하여 Elasticsearch가 자동 생성하게 하는 방법을 모두 살펴보았습니다. 자동 생성되는 ID는 Base64로 인코딩된 UUID 형태입니다. 중요한 점은 _id가 인덱스 내에서 반드시 유일해야 한다는 것입니다. 같은 인덱스 안에 같은 _id를 가진 문서는 두 개 존재할 수 없습니다. 같은 _id로 다시 인덱싱하면 기존 문서가 교체됩니다.

**_version**은 문서가 몇 번 기록되었는지를 추적하는 정수 카운터입니다. 문서가 처음 생성되면 1로 시작합니다. 이후 같은 _id로 업데이트하거나, 삭제 후 같은 _id로 다시 생성하면 _version이 1씩 증가합니다. 1.1.3에서 처음 PUT 요청을 보냈을 때 응답의 _version이 1이었던 것이 바로 이 카운터의 초기값입니다.

_version을 이용하면 동일 문서의 병렬 수정에서 충돌을 감지하는 것이 가능합니다. 그러나 분산 환경에서 샤드 장애 복구가 일어나는 상황까지 완전히 처리하기에는 _version만으로는 부족합니다. 이 한계를 해결하기 위해 Elasticsearch 7.x 이후에는 **_seq_no**와 **_primary_term**이라는 두 필드를 도입했습니다.

_seq_no와 _primary_term이 왜 필요한지 이해하려면, 분산 환경에서 발생하는 문제를 먼저 생각해야 합니다. Elasticsearch는 문서를 여러 샤드에 분산 저장합니다. 각 샤드에는 기본 샤드(primary shard)가 있고, 이 기본 샤드가 모든 쓰기 작업을 처리합니다. 그런데 기본 샤드에 장애가 발생하면 복제본 샤드 중 하나가 새 기본 샤드로 승격됩니다. 이때 _version 카운터만으로는 "이전 기본 샤드가 발행한 버전"과 "새 기본 샤드가 발행한 버전"을 구별할 수 없습니다.

**_seq_no**(시퀀스 번호)는 기본 샤드가 처리한 각 쓰기 작업에 단조 증가하는 번호를 할당한 것입니다. 문서 단위가 아니라 샤드 수준의 작업 순서를 추적합니다. 샤드가 다르면 시퀀스 번호 공간도 별도입니다. 예를 들어 샤드 A에서 _seq_no가 5이고 샤드 B에서도 _seq_no가 5일 수 있지만, 둘은 서로 다른 작업입니다.

**_primary_term**(기본 샤드 임기)은 기본 샤드가 교체될 때마다 1씩 증가하는 값입니다. 특정 시퀀스 번호가 어떤 "임기"의 기본 샤드에서 발행되었는지를 식별합니다. _seq_no만으로는 샤드 교체 후 번호가 다시 시작될 때 이전 임기의 번호와 구분할 수 없기 때문에, _primary_term이 함께 필요합니다.

이 두 값을 활용한 **낙관적 동시성 제어**(Optimistic Concurrency Control)의 절차는 세 단계로 이루어집니다.

```
낙관적 동시성 제어 흐름

1. GET /my-index/_doc/1
   → 응답에서 _seq_no와 _primary_term 값을 기록한다.
      (예: _seq_no = 5, _primary_term = 1)

2. PUT /my-index/_doc/1?if_seq_no=5&if_primary_term=1
   → 업데이트 요청에 기록해 둔 값을 파라미터로 전달한다.

3. Elasticsearch가 현재 저장된 값과 비교한다.
   → 일치하면: 업데이트 성공 (200 OK)
   → 불일치하면: 그 사이에 다른 변경이 있었다는 뜻 (409 Conflict)
```

첫 번째 단계에서 GET 요청으로 문서를 가져오면서 _seq_no와 _primary_term 값을 읽습니다. 두 번째 단계에서 업데이트 요청에 if_seq_no와 if_primary_term 파라미터를 함께 전달합니다. 세 번째 단계에서 Elasticsearch는 현재 저장된 _seq_no와 _primary_term이 전달받은 값과 일치하는지 확인합니다. 일치하면 그 사이에 아무도 이 문서를 수정하지 않았다는 뜻이므로 업데이트가 적용됩니다. 불일치하면 누군가 먼저 수정했다는 뜻이므로 409 Conflict 오류가 반환됩니다.

_version과 _seq_no + _primary_term 조합의 차이를 정리하면 다음과 같습니다. _version은 단순 카운터로 충돌 감지에만 사용할 수 있습니다. _seq_no + _primary_term 조합은 분산 환경에서 네트워크 지연으로 작업이 비순차적으로 도달하는 상황과, 기본 샤드가 교체되는 장애 복구 상황까지 정확하게 처리합니다. Elasticsearch 7.x 이후로는 _seq_no + _primary_term을 사용하는 것이 권장됩니다.

_version에는 한 가지 추가 기능이 있습니다. **외부 버전 관리**(external versioning)입니다. `version_type=external` 파라미터를 사용하면 외부 시스템(예: 관계형 데이터베이스)의 버전 번호를 그대로 Elasticsearch에 전달할 수 있습니다. 이때 Elasticsearch는 전달된 버전이 현재 저장된 버전보다 클 때만 업데이트를 허용합니다. 외부 시스템과 Elasticsearch 사이에서 데이터를 동기화할 때 유용합니다.

삭제된 문서의 버전 정보는 영원히 유지되지 않습니다. 삭제 후 기본 60초 동안만 버전 정보가 보존되며, 이 시간이 지나면 만료됩니다. 만료 후 같은 _id로 다시 인덱싱하면 _version이 1부터 다시 시작합니다.

지금까지 메타데이터 필드를 살펴보았습니다. 이제 문서 본문, 즉 사용자가 정의하는 JSON 데이터의 구조를 다룹니다. JSON은 객체 안에 또 다른 객체를 포함하는 중첩 구조를 자연스럽게 표현할 수 있습니다. Elasticsearch에서 이 중첩 구조를 처리하는 방식은 두 가지로 나뉩니다.

첫 번째는 기본 방식인 **object 타입**입니다. 별도의 매핑 설정 없이 중첩 JSON 객체를 인덱싱하면 이 방식이 적용됩니다. Elasticsearch는 중첩 객체의 필드를 부모 문서의 네임스페이스에 평탄화(flatten)하여 저장합니다. 구체적인 예를 보겠습니다.

```json
{
  "title": "Elasticsearch 입문",
  "author": {
    "first": "길동",
    "last": "홍"
  }
}
```

이 문서를 인덱싱하면 Elasticsearch 내부에서는 author 객체를 풀어서 `author.first: "길동"`, `author.last: "홍"`이라는 두 개의 독립된 필드로 저장합니다. author라는 객체 껍데기는 사라지고, 점(.)으로 연결된 필드명으로 변환되는 것입니다. 단일 객체에서는 이 방식이 문제없이 동작합니다.

문제가 생기는 경우는 객체의 배열입니다. 아래 문서를 보겠습니다.

```json
{
  "title": "Elasticsearch 입문",
  "authors": [
    { "first": "길동", "last": "홍" },
    { "first": "철수", "last": "김" }
  ]
}
```

object 타입의 기본 평탄화 방식으로 저장하면, 내부적으로는 `authors.first: ["길동", "철수"]`, `authors.last: ["홍", "김"]`이 됩니다. "길동"과 "홍"의 쌍, "철수"와 "김"의 쌍이라는 관계가 사라집니다. 이 상태에서 "first가 길동이고 last가 김인 저자"를 검색하면, 실제로는 그런 저자가 없는데도 이 문서가 매칭됩니다. first 배열에 "길동"이 있고, last 배열에 "김"이 있기 때문입니다.

```
object 타입의 평탄화

원본:  authors: [ {first:"길동", last:"홍"}, {first:"철수", last:"김"} ]

저장:  authors.first → ["길동", "철수"]
       authors.last  → ["홍",   "김"]

       → "길동"과 "홍"의 쌍 관계가 유실됨
```

이 문제를 해결하는 것이 두 번째 방식인 **nested 타입**입니다. 매핑에서 해당 필드를 `"type": "nested"`로 선언하면, 배열 내 각 객체를 Lucene에서 별도의 숨겨진 문서(hidden document)로 저장합니다. "길동-홍"과 "철수-김"이 각각 독립된 문서로 존재하므로 필드 간 상관관계가 보존됩니다.

nested 타입을 사용하면 쿼리 방식도 달라집니다. 일반 쿼리 대신 nested 쿼리를 사용해야 하며, path 파라미터로 어떤 nested 필드를 대상으로 하는지 지정해야 합니다. 기본 제한으로 문서 하나당 nested 객체는 최대 10,000개까지 허용됩니다. 이 제한은 `index.mapping.nested_objects.limit` 설정으로 조정할 수 있습니다.

nested 타입은 부모 문서 안에 임베딩되어 저장됩니다. 따라서 nested 객체 하나만 수정하더라도 전체 부모 문서를 재인덱싱해야 합니다. 자식 데이터가 자주 독립적으로 변경되는 구조라면 join 타입(parent-child 관계)이 대안이 될 수 있지만, join 타입은 global ordinals 오버헤드가 크므로 대부분의 계층적 데이터에서는 nested가 더 빠릅니다.

중첩 구조 외에도 JSON 문서에서 자주 등장하는 것이 **배열**입니다. Elasticsearch에서 모든 필드는 별도의 선언 없이 기본적으로 배열 값을 받을 수 있습니다. 매핑에 "이 필드는 배열이다"라고 따로 설정할 필요가 없습니다. 문자열 배열, 정수 배열, 중첩 배열 모두 자동으로 지원됩니다.

다만 한 가지 제약이 있습니다. 배열 안의 모든 값은 동일한 데이터 타입이어야 합니다. 예를 들어 `[1, 2, 3]`은 정수 배열이므로 문제없지만, `[1, "hello"]`처럼 정수와 문자열을 섞으면 인덱싱 오류가 발생합니다.

배열 필드에 대한 검색은 단일 값 필드와 동일하게 작성합니다. Elasticsearch가 배열 안의 모든 원소를 자동으로 확인하여 매칭 여부를 판단합니다. 예를 들어 tags 필드가 `["elastic", "search", "database"]`인 문서에서 match 쿼리로 "search"를 검색하면, 배열의 두 번째 원소와 매칭되어 이 문서가 결과에 포함됩니다.

JSON에서 값이 없음을 표현하는 방법이 **null**입니다. Elasticsearch에서 필드 값이 null이거나 빈 배열 `[null]`이면 "값이 없는 것"으로 취급됩니다. null 값은 역색인에 포함되지 않으므로 기본적으로 검색이나 집계 대상이 되지 않습니다.

null인 문서도 검색할 수 있게 만들려면 매핑에서 **null_value** 파라미터를 설정합니다. 예를 들어 status라는 keyword 필드에 `"null_value": "NULL"`을 설정하면, status가 null로 들어온 문서는 인덱싱 시점에 "NULL"이라는 문자열로 변환되어 역색인에 포함됩니다. 이후 term 쿼리로 "NULL"을 검색하면 원래 null이었던 문서가 매칭됩니다.

null_value를 설정할 때 주의할 점이 두 가지 있습니다. 첫째, 대체값의 데이터 타입은 해당 필드의 타입과 반드시 일치해야 합니다. integer 필드에 문자열 대체값을 넣으면 오류가 발생합니다. 둘째, null_value는 인덱싱 방식에만 영향을 줍니다. _source에 저장된 원본 문서는 변경되지 않습니다. 즉, _source를 조회하면 여전히 null로 표시됩니다.

빈 문자열 `""`은 null과 다르게 처리된다는 점도 구분해야 합니다. 빈 문자열은 text 타입 필드에서 정상적으로 인덱싱됩니다. null은 "값 자체가 없는 것"이고, 빈 문자열은 "값은 있지만 내용이 비어 있는 것"입니다.

마지막으로 문서 크기에 대한 제한을 다룹니다. Elasticsearch는 HTTP 요청 본문의 크기를 **http.max_content_length** 설정으로 제한합니다. 기본값은 100mb입니다. 이 제한은 압축 전의 원본 요청 본문 크기에 적용됩니다. HTTP 요청에 Content-Encoding: gzip을 사용하더라도, 압축을 풀었을 때의 크기가 기준입니다.

이 값은 elasticsearch.yml 파일에서 변경할 수 있습니다.

```yaml
http.max_content_length: 200mb
```

다만 100mb 이상으로 늘리면 클러스터 안정성이 저하될 수 있으므로 권장되지 않습니다. http.max_content_length는 개별 문서 하나의 크기 제한이 아니라, HTTP 요청 전체의 크기 제한입니다. Bulk API를 사용하면 여러 문서를 하나의 요청에 담으므로, 개별 문서가 작더라도 문서의 총합이 이 제한을 초과하면 안 됩니다.

100mb를 초과하는 대용량 데이터를 처리해야 한다면 세 가지 대안이 있습니다. 첫째, Bulk API에서 요청당 문서 수를 줄여 요청 크기를 낮춥니다. 둘째, 문서 자체를 더 작은 단위로 분할합니다. 셋째, 대용량 바이너리 데이터는 외부 시스템(오브젝트 스토리지 등)에 저장하고 Elasticsearch에는 해당 자원의 링크만 저장합니다.

정리하면, Elasticsearch 문서는 사용자가 정의한 JSON 데이터와 시스템이 자동으로 관리하는 메타데이터 필드로 구성됩니다. _source는 원본 JSON을 보존하여 조회와 검색 결과 반환에 사용되고, _index와 _id는 문서의 위치와 고유성을 식별하며, _version은 변경 이력을 추적합니다. 분산 환경에서의 동시성 제어에는 _seq_no와 _primary_term 조합이 권장됩니다. 문서 본문에서 중첩 객체는 기본적으로 평탄화되어 저장되지만, 객체 배열의 필드 상관관계를 유지하려면 nested 타입을 선언해야 합니다. 배열은 별도 설정 없이 지원되나 타입 일관성이 필요하고, null 값은 기본적으로 검색 대상에서 제외되며 null_value 설정으로 인덱싱 가능하게 만들 수 있습니다. HTTP 요청 크기는 http.max_content_length(기본 100mb)로 제한됩니다.
