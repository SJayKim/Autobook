# 2.1.2 매핑 설정과 업데이트 제약

2.1.1에서 동적 매핑과 명시적 매핑의 차이를 살펴보았습니다. 명시적 매핑으로 필드 타입을 직접 지정하면 의도치 않은 추론을 방지할 수 있다는 것도 확인했습니다. 그런데 운영 중인 인덱스에 새로운 요구 사항이 생기면 어떻게 해야 할까요. 새 필드를 추가해야 할 수도 있고, 이미 있는 필드의 타입을 바꿔야 할 수도 있습니다. 이 단원에서는 매핑을 변경할 수 있는 범위와 변경할 수 없는 범위를 구분하고, 변경이 불가능한 상황에서 어떤 절차를 거쳐야 하는지 살펴봅니다.

먼저 한 가지 사실을 분명히 합니다. Elasticsearch에서 이미 매핑에 등록된 필드의 타입은 변경할 수 없습니다. 예를 들어 age 필드를 integer로 정의한 뒤, 나중에 text로 바꾸는 것은 불가능합니다. 이 제약이 존재하는 이유는 역색인의 물리적 구조에 있습니다.

1.2.2에서 다룬 것처럼, Elasticsearch는 필드 값을 역색인(inverted index) 구조로 저장합니다. 역색인은 용어(term)를 키로, 해당 용어를 포함하는 문서 ID 목록을 값으로 갖는 구조입니다. 이 역색인은 Lucene 세그먼트 안에 기록되며, 세그먼트는 한 번 작성되면 변경되지 않는 불변(immutable) 단위입니다.

필드 타입이 다르면 역색인에 기록되는 내용도 근본적으로 달라집니다. text 타입의 필드는 분석기(analyzer)를 거쳐 토큰화된 개별 단어가 역색인에 등록됩니다. 예를 들어 "Wireless Mouse"라는 값은 "wireless"와 "mouse"라는 두 개의 토큰으로 분리되어 각각 역색인에 들어갑니다. 반면 keyword 타입은 분석기를 거치지 않고 원문 그대로 "Wireless Mouse" 한 덩어리가 역색인에 등록됩니다. 이미 text 방식으로 토큰화되어 세그먼트에 기록된 역색인을, keyword 방식의 원문 그대로로 바꾸려면 기존 세그먼트를 전부 다시 써야 합니다.

그런데 세그먼트는 불변입니다. 한 번 디스크에 기록된 세그먼트를 열어서 내부 역색인 구조를 수정하는 기능은 Lucene에 존재하지 않습니다. 따라서 Elasticsearch는 기존 필드의 타입 변경을 아예 허용하지 않는 것입니다. 타입을 바꾸고 싶다면, 원하는 타입으로 새 인덱스를 만들고 기존 데이터를 새 인덱스로 복사하는 방법밖에 없습니다.

이 원리를 그림으로 정리하면 다음과 같습니다.

```
text 타입으로 인덱싱된 경우
  원본 값: "Wireless Mouse"
  분석기 -> 토큰화 -> ["wireless", "mouse"]
  역색인:
    "wireless" -> [doc1, doc5, doc9]
    "mouse"    -> [doc1, doc3]

keyword 타입으로 인덱싱된 경우
  원본 값: "Wireless Mouse"
  분석 없음 -> 원문 그대로
  역색인:
    "Wireless Mouse" -> [doc1]

세그먼트(불변) 안에 이미 기록된 역색인 구조를 소급 변환할 수 없음
-> 필드 타입 변경 불가
```

기존 필드의 타입을 바꿀 수 없다는 제약을 확인했습니다. 반면, 기존 인덱스에 새 필드를 추가하는 것은 자유롭게 할 수 있습니다. 이런 종류의 변경을 **추가적 변경**(additive change)이라고 부릅니다. 새 필드를 추가해도 이미 존재하는 세그먼트의 역색인에는 아무런 영향을 주지 않기 때문입니다. 기존에 인덱싱된 문서에 해당 필드가 없어도 문제없이 공존합니다. 새 필드 정의는 매핑 메타데이터에만 추가되고, 이후 인덱싱되는 문서부터 해당 필드의 역색인이 만들어집니다.

새 필드를 추가하는 방법은 두 가지입니다. 하나는 2.1.1에서 다룬 동적 매핑을 활용하는 것입니다. dynamic 설정이 true인 인덱스에 새 필드가 포함된 문서를 인덱싱하면, Elasticsearch가 자동으로 해당 필드를 매핑에 등록합니다. 다른 하나는 **PUT mapping API**를 사용하여 명시적으로 추가하는 것입니다.

PUT mapping API는 `PUT /{인덱스명}/_mapping` 엔드포인트에 새 필드 정의를 JSON으로 전달합니다. 아래 예시를 봅니다.

```json
PUT /my-index/_mapping
{
  "properties": {
    "employee-id": {
      "type": "keyword",
      "index": false
    }
  }
}
```

이 요청은 my-index 인덱스에 employee-id라는 필드를 keyword 타입으로 추가합니다. "index": false는 이 필드 값을 역색인에 등록하지 않겠다는 설정입니다. 역색인에 등록하지 않으므로 이 필드로 검색하는 것은 불가능하지만, 문서의 원본(_source)에는 값이 그대로 저장되어 조회 결과에 포함됩니다. 사원 번호처럼 검색 대상은 아니지만 조회 시 보여 주기만 하면 되는 필드에 이 설정을 적용합니다.

기존 text 필드에 keyword 서브필드를 나중에 추가하는 것도 가능합니다. 이를 **멀티필드**(multi-field) 추가라고 합니다. 다만 멀티필드를 추가한 시점 이후에 인덱싱된 문서에만 서브필드가 적용됩니다. 추가 이전에 인덱싱된 문서에도 서브필드를 적용하려면, 뒤에서 다루는 reindex 절차가 필요합니다.

정리하면, 매핑 변경에는 두 가지 종류가 있습니다. 새 필드를 추가하는 추가적 변경은 언제든 가능하고, 기존 필드의 타입을 변경하는 것은 역색인의 불변성 때문에 불가능합니다.

```
매핑 변경 가능 범위

가능한 변경 (추가적)        불가능한 변경
  - 새 필드 추가             - 기존 필드 타입 변경
  - 멀티필드 추가            - 기존 필드 삭제
  - 일부 매핑 파라미터 수정  - 분석기 변경

불가능한 변경이 필요하면 -> 새 인덱스 생성 + reindex
```

이제 기존 필드의 타입을 변경해야 하는 상황, 즉 불가능한 변경이 필요한 경우의 대처 방법을 살펴봅니다. Elasticsearch는 이를 위해 **reindex API**를 제공합니다. reindex API는 소스 인덱스의 문서를 읽어서 목적지 인덱스로 복사하는 기능입니다. 목적지 인덱스에 원하는 매핑을 미리 정의해 두면, 복사 과정에서 문서가 새 매핑에 맞게 다시 인덱싱됩니다.

reindex를 사용한 매핑 변경 절차는 네 단계로 이루어집니다.

첫 번째 단계는 새 매핑으로 목적지 인덱스를 생성하는 것입니다. reindex API는 소스 인덱스의 매핑이나 설정을 자동으로 복사하지 않습니다. 반드시 목적지 인덱스를 먼저 만들고, 변경하고자 하는 매핑을 직접 정의해야 합니다.

두 번째 단계는 reindex API를 실행하여 문서를 복사하는 것입니다.

```json
POST /_reindex
{
  "source": { "index": "my-index-v1" },
  "dest":   { "index": "my-index-v2" }
}
```

이 요청은 my-index-v1의 모든 문서를 my-index-v2로 복사합니다. source.index는 원본 인덱스, dest.index는 목적지 인덱스입니다. 복사 과정에서 각 문서는 목적지 인덱스의 매핑 규칙에 따라 새로 인덱싱됩니다.

세 번째 단계는 애플리케이션의 접근 대상을 새 인덱스로 전환하는 것입니다. 이 단계에서 인덱스 별칭을 활용하면 무중단 전환이 가능합니다. 별칭에 대해서는 아래에서 자세히 다룹니다.

네 번째 단계는 기존 인덱스를 삭제하여 디스크 공간을 확보하는 것입니다.

```
reindex를 사용한 매핑 변경 절차

1. 새 매핑으로 목적지 인덱스 생성
   PUT /my-index-v2 { "mappings": { ... } }

2. reindex 실행
   POST /_reindex { "source": "v1", "dest": "v2" }

3. 애플리케이션 접근 대상을 v2로 전환 (별칭 활용)

4. 기존 인덱스(v1) 삭제
```

reindex API에는 대용량 데이터를 효율적으로 처리하기 위한 옵션이 있습니다. "slices": "auto" 옵션을 지정하면 Elasticsearch가 소스 인덱스의 샤드 수에 맞춰 병렬로 복사를 실행합니다. requests_per_second 파라미터를 지정하면 초당 처리량을 제한하여 클러스터에 과부하가 걸리는 것을 방지할 수 있습니다.

데이터가 매우 많아서 복사에 오랜 시간이 걸리는 경우에는 비동기 실행이 유용합니다. 요청 URL에 ?wait_for_completion=false 파라미터를 추가하면, Elasticsearch는 작업을 백그라운드에서 실행하고 즉시 작업 ID를 반환합니다. 이후 GET /_tasks/{작업ID}로 진행 상황을 확인할 수 있습니다.

일부 문서만 복사하고 싶은 경우에는 source 안에 query를 지정합니다. 예를 들어 특정 조건에 맞는 문서만 선별적으로 새 인덱스에 옮길 수 있습니다. 복사와 동시에 문서 내용을 변환해야 하는 경우에는 script 필드에 Painless 스크립트를 작성하여 필드명 변경이나 값 변환을 수행할 수 있습니다.

reindex API의 핵심 전제 조건이 하나 있습니다. 소스 인덱스의 모든 문서에 **_source** 필드가 활성화되어 있어야 합니다. _source는 문서를 인덱싱할 때 전달된 원본 JSON을 압축하여 저장하는 메타데이터 필드입니다. Lucene 내부적으로는 Stored Fields라는 별도 영역에 보관됩니다. 역색인에 등록되는 것이 아니라 원본 데이터를 보존하는 역할입니다. 검색 결과를 반환할 때 문서 내용을 보여 주는 것이 바로 이 _source 덕분입니다.

_source는 기본적으로 활성화되어 있습니다. 그런데 디스크 공간을 절약하기 위해 _source를 비활성화하는 경우가 있습니다. 매핑에서 "_source": { "enabled": false }로 설정하면 원본 JSON이 저장되지 않습니다. 저장 공간은 줄어들지만, 그 대가로 여러 기능을 잃습니다.

가장 치명적인 것이 reindex 불가입니다. reindex API는 소스 인덱스에서 각 문서의 _source를 읽어 목적지 인덱스에 다시 인덱싱하는 방식으로 동작합니다. _source가 없으면 문서 내용을 읽을 수 없으므로 reindex 자체가 실패합니다. 앞서 살펴본 것처럼, 필드 타입을 변경하려면 reindex가 필수입니다. _source를 비활성화하면 향후 매핑 변경이 사실상 불가능해지는 셈입니다.

_source 비활성화로 잃는 기능은 이것만이 아닙니다. 문서의 부분 업데이트(_update API)도 불가능해집니다. 부분 업데이트는 기존 _source를 읽어서 변경된 필드만 병합한 뒤 다시 인덱싱하는 방식이므로, _source가 없으면 동작하지 않습니다. 검색 결과에서 일치하는 부분을 강조하는 하이라이팅(highlighting) 기능도 원본 텍스트가 필요한데, _source가 비활성화되어 있으면 사용할 수 없습니다. 인덱싱된 원본 데이터를 직접 확인하는 것도 불가능하므로, 문제 진단이 어려워집니다.

디스크 절약이 필요한 경우에는 _source를 완전히 비활성화하는 대신, includes와 excludes 옵션으로 저장할 필드를 선택적으로 지정하는 방법이 있습니다. 이렇게 하면 불필요한 필드를 _source에서 제외하여 저장 공간을 줄이면서도, reindex와 업데이트 기능을 유지할 수 있습니다.

이제 reindex 절차의 세 번째 단계에서 언급한 인덱스 별칭을 살펴봅니다. **인덱스 별칭**(index alias)은 실제 인덱스에 붙이는 논리적 이름입니다. 애플리케이션이 인덱스의 실제 이름 대신 별칭을 사용하여 접근하면, 인덱스를 교체하더라도 애플리케이션 코드를 수정할 필요가 없습니다.

별칭이 왜 필요한지 구체적인 상황으로 설명합니다. 애플리케이션이 "products"라는 인덱스 이름을 직접 사용하고 있다고 가정합니다. 매핑을 변경하기 위해 reindex로 products-v2를 만든 뒤, 애플리케이션의 인덱스 이름을 "products"에서 "products-v2"로 바꿔야 합니다. 이 과정에서 애플리케이션을 재배포해야 하고, 전환 시점에 검색 요청이 실패할 수 있습니다. 반면, 애플리케이션이 처음부터 "products"라는 별칭을 사용하고 있었다면, 별칭이 가리키는 실제 인덱스만 교체하면 됩니다. 애플리케이션은 여전히 "products"라는 이름으로 접근하므로 코드 변경이 필요 없습니다.

별칭을 활용한 무중단 재매핑 절차는 다음과 같습니다.

첫째, 새 매핑으로 목적지 인덱스를 생성합니다. 예를 들어 my-index-v2를 만듭니다.

둘째, reindex API로 my-index-v1의 문서를 my-index-v2에 복사합니다.

셋째, 별칭을 원자적으로 전환합니다.

```json
POST /_aliases
{
  "actions": [
    { "remove": { "index": "my-index-v1", "alias": "my-index" } },
    { "add":    { "index": "my-index-v2", "alias": "my-index" } }
  ]
}
```

이 요청은 두 가지 작업을 하나의 요청에 담고 있습니다. 첫 번째 작업은 my-index-v1에서 "my-index" 별칭을 제거하는 것이고, 두 번째 작업은 my-index-v2에 "my-index" 별칭을 부여하는 것입니다. Elasticsearch는 이 두 작업을 **원자적**(atomic)으로 처리합니다. 원자적이라는 것은 두 작업이 동시에 적용된다는 뜻입니다. 별칭이 제거된 순간과 새로 부여되는 순간 사이에 빈 구간이 없으므로, 전환 도중에 "my-index" 별칭으로 들어오는 검색 요청이 실패하지 않습니다.

넷째, 전환이 확인되면 my-index-v1을 삭제합니다.

```
별칭을 활용한 무중단 재매핑

  애플리케이션 -> "my-index" 별칭
                      |
           +----------+----------+
           |                     |
    전환 전: my-index-v1    전환 후: my-index-v2
    (기존 매핑)             (새 매핑)

  POST /_aliases 한 번으로 원자적 전환
  -> 애플리케이션 코드 변경 없음
  -> 검색 요청 중단 없음
```

별칭이 여러 인덱스를 동시에 가리킬 수도 있습니다. 이 경우 읽기(검색) 요청은 모든 인덱스에 분산되지만, 쓰기(인덱싱) 요청은 어느 인덱스에 보내야 하는지 모호해집니다. 이때 **is_write_index** 설정으로 쓰기 대상 인덱스를 명시적으로 지정할 수 있습니다.

마지막으로 **copy_to** 활용 패턴을 살펴봅니다. copy_to는 매핑 파라미터의 하나로, 한 필드의 값을 다른 필드로 복사하여 인덱싱 시점에 별도의 역색인을 만드는 기능입니다.

copy_to가 필요한 상황을 예로 들겠습니다. 사용자 정보를 담은 인덱스에 first_name과 last_name이라는 두 개의 필드가 있다고 가정합니다. 이름 전체로 검색하려면 두 필드를 모두 검색 대상에 포함해야 합니다. multi_match 쿼리를 사용하면 여러 필드를 한 번에 검색할 수 있지만, 필드 수가 늘어날수록 쿼리 비용이 증가합니다. copy_to를 사용하면 여러 필드의 값을 하나의 통합 필드에 모아서, 단일 필드 검색만으로 동일한 결과를 얻을 수 있습니다.

```json
PUT /my-index
{
  "mappings": {
    "properties": {
      "first_name": { "type": "text", "copy_to": "full_name" },
      "last_name":  { "type": "text", "copy_to": "full_name" },
      "full_name":  { "type": "text" }
    }
  }
}
```

이 매핑에서 first_name과 last_name 필드에 copy_to: "full_name"이 지정되어 있습니다. 문서가 인덱싱될 때 first_name의 값과 last_name의 값이 full_name 필드의 역색인에도 함께 등록됩니다. 따라서 full_name 필드 하나만 검색하면 이름의 어느 부분이든 찾을 수 있습니다.

copy_to에서 주의할 점이 있습니다. copy_to로 복사된 값은 _source에 저장되지 않습니다. 위 예시에서 full_name 필드는 역색인에만 존재하고, 검색 결과의 _source에는 나타나지 않습니다. 검색 대상으로는 활용할 수 있지만, 조회 결과에서 full_name 필드 값을 직접 꺼내 보는 것은 불가능합니다.

copy_to는 하나의 필드에서 여러 대상으로 동시에 복사하는 것도 가능합니다. "copy_to": ["field_a", "field_b"] 형식으로 배열을 지정하면 됩니다.

정리하면, Elasticsearch에서 기존 필드의 타입 변경은 역색인의 불변성 때문에 불가능합니다. 새 필드 추가는 기존 역색인에 영향을 주지 않으므로 언제든 허용됩니다. 타입 변경이 필요한 경우에는 새 매핑으로 인덱스를 만들고 reindex API로 문서를 복사해야 하며, 인덱스 별칭을 사용하면 이 과정을 무중단으로 진행할 수 있습니다. _source 비활성화는 디스크 절약에 도움이 되지만 reindex를 포함한 여러 핵심 기능을 잃게 되므로 신중해야 합니다. copy_to는 여러 필드를 하나의 검색 필드로 통합하여 쿼리 성능을 높이는 매핑 파라미터입니다.

다음 단원인 2.1.3에서는 매핑 폭발 방지를 다룹니다.

이 단원을 마치면 기존 필드 타입 변경이 불가능한 이유를 역색인 관점에서 설명할 수 있고, reindex API를 사용해 매핑을 변경하는 절차를 수행할 수 있습니다.
