# 6.2.2 Beats

6.2.1에서 다룬 Logstash는 강력한 데이터 변환 기능을 제공하지만, 별도의 JVM 프로세스를 실행해야 하므로 리소스 사용량이 적지 않습니다. 수백 대의 서버에서 각각 Logstash를 실행하면 CPU와 메모리 부담이 커집니다. 많은 경우 원천 서버에서는 단순히 로그 파일이나 메트릭을 수집하여 전달하는 역할만 필요하고, 복잡한 변환은 중앙의 Logstash나 Elasticsearch에서 처리하면 됩니다.

**Beats**는 이런 상황을 위해 Elastic Stack이 제공하는 경량 데이터 수집 에이전트 제품군입니다. Go 언어로 작성되어 외부 의존성이 없고, 단일 바이너리로 동작합니다. 각 Beat는 하나의 목적에 특화되어 있어, 필요한 종류만 설치하면 됩니다.

Elastic Stack에서 제공하는 주요 Beat는 다음과 같습니다.

**Filebeat**는 로그 파일을 수집하는 Beat입니다. 서버의 로그 파일을 모니터링하다가 새로운 줄이 추가되면 이를 읽어서 전송합니다. Apache, Nginx, MySQL, Docker 등 다양한 플랫폼을 위한 내장 모듈을 제공합니다. 모듈을 활성화하면 해당 플랫폼의 로그 형식에 맞는 파싱 설정과 Kibana 대시보드가 자동으로 구성됩니다.

**Metricbeat**는 시스템과 서비스의 성능 지표를 수집하는 Beat입니다. CPU 사용률, 메모리 사용량, 디스크 I/O, 네트워크 트래픽 같은 시스템 수준 메트릭을 주기적으로 수집합니다. 시스템 외에도 MySQL, Redis, Nginx, Docker, Kubernetes 등 다양한 서비스의 메트릭을 가져오는 모듈을 내장하고 있습니다.

**Packetbeat**는 네트워크 패킷을 캡처하여 분석하는 Beat입니다. HTTP, MySQL, DNS 같은 애플리케이션 프로토콜의 트래픽을 실시간으로 수집합니다. 네트워크 수준의 성능 모니터링이나 보안 분석에 활용됩니다.

**Heartbeat**는 서비스 가용성을 모니터링하는 Beat입니다. 지정한 URL이나 호스트에 주기적으로 요청을 보내, 응답 여부와 응답 시간을 기록합니다. 서비스가 정상 동작 중인지 확인하는 업타임 모니터링에 사용합니다.

**Auditbeat**는 시스템의 보안 감사 데이터를 수집하는 Beat입니다. Linux audit framework의 이벤트와 파일 무결성 변경 사항을 모니터링합니다.

Filebeat의 설정 파일 구조를 살펴보겠습니다. filebeat.yml 파일에 수집 대상과 전송 대상을 지정합니다.

```yaml
filebeat.inputs:
- type: log
  enabled: true
  paths:
    - /var/log/*.log
    - /var/log/nginx/access.log

output.elasticsearch:
  hosts: ["localhost:9200"]
  index: "filebeat-%{[agent.version]}-%{+yyyy.MM.dd}"
```

filebeat.inputs 아래에 수집할 로그 파일의 경로를 지정합니다. type: log는 일반 로그 파일을 한 줄씩 읽는 방식입니다. paths에 와일드카드(*)를 사용하여 여러 파일을 한번에 대상으로 지정할 수 있습니다. output.elasticsearch는 수집한 데이터를 Elasticsearch로 직접 전송하는 설정입니다. hosts에 Elasticsearch 주소를 지정하고, index에 저장할 인덱스 이름 패턴을 설정합니다.

Filebeat 모듈을 사용하면 설정을 더 간단하게 할 수 있습니다. 예를 들어 Nginx 로그를 수집하려면 다음과 같이 모듈을 활성화합니다.

```bash
filebeat modules enable nginx
filebeat setup
filebeat -e
```

modules enable 명령으로 Nginx 모듈을 활성화하고, setup 명령으로 인덱스 템플릿과 Kibana 대시보드를 자동 생성하며, -e 플래그로 Filebeat를 실행합니다.

Metricbeat의 설정도 유사한 구조입니다. 수집할 모듈과 메트릭셋을 지정합니다.

```yaml
metricbeat.modules:
- module: system
  metricsets:
    - cpu
    - memory
    - network
    - diskio
  period: 10s

- module: nginx
  metricsets: ["stubstatus"]
  period: 10s
  hosts: ["http://localhost"]

output.elasticsearch:
  hosts: ["localhost:9200"]
```

module은 데이터를 수집할 대상 서비스를 지정합니다. **metricsets**는 해당 모듈에서 수집할 구체적인 메트릭 종류입니다. 하나의 모듈 안에 여러 metricset이 있고, 필요한 것만 선택합니다. **period**는 메트릭을 수집하는 주기입니다. 10s는 10초마다 수집한다는 의미입니다.

Beats에서 수집한 데이터를 전송하는 경로는 두 가지입니다.

첫 번째는 Beats에서 Elasticsearch로 직접 전송하는 방식입니다. 구성이 단순하고, Elasticsearch의 인제스트 파이프라인으로 가벼운 변환을 처리할 수 있습니다. 변환이 복잡하지 않은 경우에 적합합니다.

```
Beats ──▶ Elasticsearch ──▶ Kibana
```

두 번째는 Beats에서 Logstash를 거쳐 Elasticsearch로 전송하는 방식입니다. Logstash의 풍부한 필터 플러그인을 활용하여 복잡한 변환을 수행하거나, 여러 출력 대상에 동시에 전송하거나, 버퍼링이 필요한 경우에 사용합니다.

```
Beats ──▶ Logstash ──▶ Elasticsearch ──▶ Kibana
```

Beats에서 Logstash로 전송하려면 output 설정을 다음과 같이 변경합니다.

```yaml
output.logstash:
  hosts: ["logstash-server:5044"]
```

Beats는 네트워크 장애 시 데이터 유실을 방지하는 메커니즘을 내장하고 있습니다. 전송 대상(Elasticsearch나 Logstash)이 응답하지 않으면, Beats는 자동으로 전송 속도를 줄이고 데이터를 로컬 디스크에 임시 저장합니다. 연결이 복원되면 저장해 둔 데이터를 순서대로 전송합니다. Filebeat의 경우, 각 로그 파일의 읽기 위치를 레지스트리 파일에 기록하므로 프로세스가 재시작되어도 중단된 지점부터 수집을 재개합니다.

정리하면, Beats는 Go로 작성된 경량 에이전트 제품군으로, 각 Beat가 하나의 수집 목적에 특화되어 있습니다. Filebeat는 로그 파일, Metricbeat는 시스템과 서비스 메트릭, Packetbeat는 네트워크 패킷, Heartbeat는 서비스 가용성, Auditbeat는 보안 감사 데이터를 수집합니다. Elasticsearch로 직접 전송하거나 Logstash를 경유하여 복잡한 변환을 거친 뒤 전송할 수 있으며, 네트워크 장애 시에도 데이터 유실을 방지하는 디스크 기반 큐를 내장하고 있습니다.

다음 단원인 6.2.3에서는 인제스트 파이프라인을 다룹니다. Logstash 없이 Elasticsearch 내부에서 문서를 전처리하는 방법을 알아봅니다.

이 단원을 마치면 Filebeat, Metricbeat, Packetbeat의 역할을 구분하고 기본 설정을 적용할 수 있습니다.
