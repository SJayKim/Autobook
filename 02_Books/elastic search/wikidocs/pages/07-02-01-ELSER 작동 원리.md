# 7.2.1 ELSER 작동 원리

7.1.1에서 dense_vector를 이용한 벡터 검색을 다뤘습니다. dense_vector는 텍스트를 수백 차원의 숫자 배열로 변환하여 의미 기반 검색을 가능하게 합니다. 그런데 이 접근 방식에는 몇 가지 고려 사항이 있습니다. 외부 임베딩 모델을 선택하고 배포해야 하고, 도메인에 따라 모델을 미세 조정(fine-tuning)해야 정확도가 올라가는 경우가 많습니다. Elastic은 이런 부담을 줄이면서도 의미 기반 검색을 제공하기 위해 자체 모델을 만들었습니다. 그것이 **ELSER**(Elastic Learned Sparse EncodeR)입니다.

ELSER를 이해하려면 먼저 "밀집 벡터"와 "희소 벡터"의 차이를 짚어야 합니다. 7.1.1에서 다룬 dense_vector는 모든 차원에 값이 채워져 있습니다. 768차원이면 768개의 숫자가 전부 의미 있는 값을 가집니다. 반면 **희소 벡터**(sparse vector)는 대부분의 차원이 0이고, 소수의 차원만 0이 아닌 값을 가집니다. 이 소수의 차원이 어떤 "토큰"에 해당하고, 해당 토큰의 가중치가 값으로 들어갑니다.

ELSER는 입력 텍스트를 이 희소 벡터 형태로 변환합니다. 변환 과정에서 핵심이 되는 메커니즘이 **learned token expansion**(학습된 토큰 확장)입니다.

일반적인 키워드 검색에서는 문서에 실제로 등장하는 단어만 검색 대상이 됩니다. "강아지 산책"이라는 문서는 "반려견 외출"이라는 쿼리에 매칭되지 않습니다. 단어가 다르기 때문입니다. 이 문제를 **어휘 불일치**(vocabulary mismatch)라고 부릅니다.

ELSER의 token expansion은 이 문제를 해결합니다. 모델이 텍스트를 분석할 때, 원문에 있는 단어뿐 아니라 의미적으로 관련된 단어를 추가로 생성합니다. "강아지 산책"이라는 문서를 ELSER에 넣으면, 원래 단어에 더해 "반려견", "외출", "걷기", "pet", "walk" 같은 확장 토큰이 가중치와 함께 생성됩니다. 이 확장 토큰들은 사람이 정의한 동의어 사전이 아니라, 대규모 학습 데이터에서 함께 등장하는 빈도 패턴을 학습하여 모델이 스스로 만들어낸 것입니다.

구체적인 예를 들겠습니다. "Elasticsearch is a distributed search engine"이라는 문장을 ELSER에 넣으면 다음과 같은 토큰과 가중치 쌍이 생성됩니다.

```
원본 토큰:
  elasticsearch: 2.31
  distributed: 1.87
  search: 1.65
  engine: 1.42

확장 토큰 (원문에 없던 단어):
  elastic: 1.15
  lucene: 0.89
  indexing: 0.74
  cluster: 0.68
  retrieval: 0.55
```

원본 텍스트에 "lucene"이나 "cluster"라는 단어가 없었지만, 모델이 의미적 연관성을 학습하여 이 토큰들을 추가했습니다. 각 토큰 옆의 숫자는 가중치로, 해당 토큰이 원본 텍스트의 의미를 얼마나 강하게 반영하는지를 나타냅니다.

이렇게 생성된 토큰-가중치 쌍을 Elasticsearch에 저장하려면 **rank_features** 타입을 사용합니다. rank_features는 2.2.3에서 다룬 필드 타입 중 하나로, 여러 개의 이름-숫자 쌍을 저장할 수 있습니다. ELSER가 생성한 확장 토큰과 가중치가 바로 이 이름-숫자 쌍에 해당합니다.

```json
{
  "mappings": {
    "properties": {
      "content": { "type": "text" },
      "ml.tokens": { "type": "rank_features" }
    }
  }
}
```

ml.tokens 필드에 ELSER가 생성한 토큰-가중치 쌍이 저장됩니다. rank_features 타입은 내부적으로 역색인 구조를 활용합니다. 이것은 ELSER의 중요한 특성입니다. dense_vector가 별도의 HNSW 그래프를 만드는 것과 달리, ELSER의 희소 벡터는 Elasticsearch가 원래 가지고 있던 역색인 위에서 동작합니다.

ELSER 모델은 버전이 두 가지 있습니다. **ELSER v1**은 2023년 6월에 처음 공개되었습니다. **ELSER v2**는 2023년 11월에 정식 출시(GA)되었으며, v1 대비 검색 정확도가 향상되었습니다. 새로 도입하는 경우 v2를 사용하는 것이 권장됩니다.

ELSER의 성능을 BM25와 비교하면, 12개의 주요 검색 품질 벤치마크 중 11개에서 BM25를 능가하는 결과를 보입니다. 나머지 1개에서도 ELSER와 BM25를 결합한 하이브리드 검색을 적용하면 모든 벤치마크에서 높은 성능을 달성합니다.

ELSER로 검색할 때는 **text_expansion** 쿼리를 사용합니다.

```json
GET my-elser-index/_search
{
  "query": {
    "text_expansion": {
      "ml.tokens": {
        "model_id": ".elser_model_2",
        "model_text": "distributed search engine"
      }
    }
  }
}
```

text_expansion 쿼리의 필드명으로 ml.tokens를 지정하고, model_id에 ELSER 모델을, model_text에 검색어를 넣습니다. Elasticsearch가 검색어를 ELSER 모델에 넣어 확장 토큰을 생성한 뒤, rank_features 필드에 저장된 문서 토큰과 매칭하여 점수를 계산합니다.

이 과정을 단계별로 정리하면 다음과 같습니다.

```
1. 사용자 쿼리 입력: "distributed search engine"
         │
2. ELSER 모델이 쿼리를 토큰 확장
   → elasticsearch: 1.8, distributed: 1.5, search: 1.3, lucene: 0.7 ...
         │
3. 확장된 쿼리 토큰을 rank_features 필드(역색인)에서 검색
         │
4. 문서별 토큰 가중치와 쿼리 토큰 가중치를 매칭하여 점수 산출
         │
5. 점수순으로 정렬하여 결과 반환
```

ELSER는 검색어와 문서 양쪽 모두 토큰을 확장합니다. 검색어에 없는 단어라도 문서의 확장 토큰에 포함되어 있으면 매칭됩니다. 반대 방향도 마찬가지입니다. 이 양방향 확장이 어휘 불일치 문제를 효과적으로 해소합니다.

정리하면, ELSER는 Elastic이 자체 개발한 희소 벡터 모델로, learned token expansion을 통해 텍스트를 토큰-가중치 쌍으로 변환합니다. 원문에 없는 관련 토큰을 학습 기반으로 추가하여 어휘 불일치 문제를 해결합니다. 생성된 희소 벡터는 rank_features 타입에 저장되며, Elasticsearch의 역색인 구조를 활용합니다. 검색 시에는 text_expansion 쿼리를 사용하여 쿼리도 토큰 확장한 뒤 매칭합니다.

다음 단원인 7.2.2에서는 ELSER와 dense 벡터 검색의 성능, 정확도, 리소스 사용량을 비교합니다.

이 단원을 마치면 ELSER의 희소 벡터 표현 방식과 BM25와의 차이를 설명할 수 있습니다.
