# 8.2.1 로그 수집 파이프라인

웹 서비스를 운영하면 서버는 요청이 들어올 때마다 로그를 남깁니다. Nginx 접근 로그에는 요청 시각, 클라이언트 IP, 요청 URL, HTTP 상태 코드, 응답 시간이 한 줄씩 기록됩니다. 애플리케이션 로그에는 오류 메시지와 스택 트레이스가 쌓입니다. 서버가 한 대라면 SSH로 접속하여 로그 파일을 직접 열어 볼 수 있지만, 서버가 수십 대로 늘어나면 각 서버의 로그를 일일이 확인하는 것은 현실적이지 않습니다. 모든 서버의 로그를 한곳에 모아 검색하고 분석할 수 있는 중앙 집중식 로그 수집 체계가 필요합니다.

이 단원에서는 Filebeat, Logstash, Elasticsearch를 연결하여 로그 수집 파이프라인을 구성합니다. 6.2.1에서 다룬 Logstash 파이프라인과 6.1.2에서 다룬 Data Stream의 개념이 기반이 됩니다.

로그 수집 파이프라인의 전체 흐름을 먼저 그림으로 살펴봅니다.

```
서버 A ─┐
서버 B ─┤── Filebeat ──▶ Logstash ──▶ Elasticsearch ──▶ Kibana
서버 C ─┘   (수집)        (파싱/변환)   (저장/검색)       (시각화)
```

각 서버에 설치된 **Filebeat**가 로그 파일을 모니터링하다가 새로운 줄이 추가되면 이를 읽어서 Logstash로 전송합니다. **Logstash**는 받은 로그를 파싱하고 필드를 구조화한 뒤 Elasticsearch로 보냅니다. **Elasticsearch**는 데이터를 인덱싱하여 검색 가능한 상태로 저장합니다. **Kibana**는 저장된 데이터를 시각화합니다. 이 단원에서는 Filebeat에서 Elasticsearch까지의 수집 파이프라인을 다루고, Kibana 시각화는 8.2.2에서 다룹니다.

먼저 Filebeat 설정부터 시작합니다. 6.2.2에서 Filebeat의 기본 구조를 다뤘습니다. Filebeat는 지정한 경로의 로그 파일을 모니터링하고, 새 줄이 추가될 때마다 이벤트로 전송합니다. 로그 수집에서 가장 빠르게 시작하는 방법은 **Filebeat modules**를 사용하는 것입니다.

Filebeat modules는 자주 사용하는 로그 소스에 대해 미리 만들어진 설정 패키지입니다. Nginx, Apache, MySQL, System 로그 등 다양한 모듈이 내장되어 있습니다. 모듈을 활성화하면 해당 로그 형식에 맞는 파싱 설정, 인덱스 매핑, Kibana 대시보드가 자동으로 구성됩니다.

Nginx 로그를 수집하는 예를 봅니다. 먼저 Nginx 모듈을 활성화합니다.

```bash
filebeat modules enable nginx
```

그다음 모듈 설정 파일에서 로그 파일 경로를 지정합니다. modules.d/nginx.yml 파일을 열어 다음과 같이 설정합니다.

```yaml
- module: nginx
  access:
    enabled: true
    var.paths: ["/var/log/nginx/access.log"]
  error:
    enabled: true
    var.paths: ["/var/log/nginx/error.log"]
```

access는 접근 로그, error는 오류 로그를 수집하는 설정입니다. var.paths에 실제 로그 파일의 경로를 지정합니다. 서버마다 로그 경로가 다르면 해당 서버의 설정 파일에서 경로를 변경합니다.

Filebeat의 출력을 Logstash로 보내려면 filebeat.yml에서 output을 설정합니다.

```yaml
output.logstash:
  hosts: ["logstash-server:5044"]
```

hosts에 Logstash 서버의 주소와 포트를 지정합니다. Filebeat가 수집한 이벤트는 이 주소로 전송됩니다.

모듈을 사용하지 않고 일반 로그 파일을 직접 수집하려면, filebeat.inputs에 경로를 지정합니다.

```yaml
filebeat.inputs:
- type: log
  enabled: true
  paths:
    - /var/log/myapp/*.log
  fields:
    service: myapp
    environment: production
  fields_under_root: true
```

fields 설정은 각 이벤트에 사용자 정의 필드를 추가합니다. 여기서는 service 필드에 "myapp", environment 필드에 "production"을 추가했습니다. fields_under_root를 true로 설정하면 추가된 필드가 이벤트의 최상위에 배치됩니다. 이 필드를 나중에 인덱스 이름이나 필터링에 활용할 수 있습니다.

다음으로 Logstash에서 로그를 파싱하는 방법을 살펴봅니다. 6.2.1에서 Logstash의 Input-Filter-Output 구조를 다뤘습니다. 로그 파이프라인에서 가장 중요한 부분은 Filter 단계의 **grok 파싱**입니다.

grok은 비정형 텍스트에서 패턴을 인식하여 구조화된 필드를 추출하는 필터 플러그인입니다. Nginx 접근 로그 한 줄이 다음과 같다고 합시다.

```
192.168.1.50 - - [01/Apr/2026:10:15:30 +0900] "GET /api/products HTTP/1.1" 200 1234
```

이 한 줄에는 클라이언트 IP, 타임스탬프, 요청 메서드, URL, HTTP 버전, 상태 코드, 응답 바이트가 공백과 특수문자로 구분되어 들어 있습니다. grok은 이 텍스트에서 각 부분을 필드로 추출합니다.

```
input {
  beats {
    port => 5044
  }
}

filter {
  grok {
    match => { "message" => "%{COMBINEDAPACHELOG}" }
  }

  date {
    match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z"]
    target => "@timestamp"
  }

  mutate {
    convert => {
      "response" => "integer"
      "bytes" => "integer"
    }
    remove_field => ["message", "timestamp"]
  }
}

output {
  elasticsearch {
    hosts => ["https://elasticsearch:9200"]
    data_stream => "true"
    data_stream_type => "logs"
    data_stream_dataset => "nginx"
    data_stream_namespace => "production"
  }
}
```

이 파이프라인의 각 단계가 무엇을 하는지 풀어 봅니다.

input에서 beats 플러그인이 5044 포트에서 Filebeat의 이벤트를 수신합니다.

filter의 grok에서 COMBINEDAPACHELOG는 Logstash에 내장된 패턴입니다. Apache/Nginx 접근 로그의 일반적인 형식을 한 줄로 파싱하여 clientip, timestamp, verb, request, httpversion, response, bytes 같은 필드로 분해합니다.

date 필터는 **타임스탬프 파싱**을 수행합니다. grok이 추출한 timestamp 문자열을 Elasticsearch가 사용하는 @timestamp 필드로 변환합니다. match에서 "dd/MMM/yyyy:HH:mm:ss Z" 형식을 지정하면, "01/Apr/2026:10:15:30 +0900" 같은 문자열을 날짜 객체로 파싱합니다. **시간대(timezone)** 설정이 중요합니다. 로그에 시간대 정보가 포함되어 있지 않으면, timezone 파라미터로 로그가 생성된 서버의 시간대를 명시해야 합니다. 시간대를 잘못 설정하면 Kibana에서 로그를 시간순으로 정렬할 때 엉뚱한 순서로 나타날 수 있습니다.

mutate 필터의 convert는 response와 bytes 필드를 정수로 변환합니다. grok이 추출한 필드는 기본적으로 문자열이기 때문입니다. 숫자로 변환해야 범위 검색과 집계가 가능합니다. remove_field는 원본 message(비정형 로그 전체)와 grok이 추출한 임시 timestamp 필드를 삭제하여 저장 공간을 절약합니다.

이제 **ECS(Elastic Common Schema)**를 살펴봅니다. 서비스마다 로그 필드 이름이 다르면 분석이 어려워집니다. Nginx 로그에서는 "clientip"이라고 부르는 필드가 애플리케이션 로그에서는 "remote_addr"이 되고, 다른 시스템에서는 "src_ip"가 됩니다. 같은 의미의 필드인데 이름이 제각각이면, Kibana에서 여러 서비스의 로그를 통합 분석할 때 필드마다 별도로 필터링해야 합니다.

ECS는 이 문제를 해결하기 위한 필드 네이밍 표준입니다. Elastic이 정의한 규칙에 따라 공통 필드의 이름과 타입을 통일합니다. 예를 들어 클라이언트 IP 주소는 source.ip, HTTP 상태 코드는 http.response.status_code, 호스트 이름은 host.name으로 표기합니다.

Logstash의 mutate 필터에서 ECS 필드명으로 변환하는 작업을 추가할 수 있습니다.

```
mutate {
  rename => {
    "clientip" => "[source][ip]"
    "response" => "[http][response][status_code]"
    "bytes" => "[http][response][body][bytes]"
    "verb" => "[http][request][method]"
    "request" => "[url][path]"
  }
}
```

rename으로 grok이 추출한 필드 이름을 ECS 형식으로 바꿉니다. 대괄호 표기법은 중첩 필드를 나타냅니다. "[source][ip]"는 source 객체 아래의 ip 필드를 의미합니다. Filebeat modules를 사용하면 이 ECS 변환이 자동으로 적용됩니다.

다음으로 인덱스 네이밍 규칙을 살펴봅니다. 로그 데이터는 시간이 지남에 따라 계속 쌓이므로, 날짜별로 인덱스를 분리하는 것이 일반적입니다. 오래된 로그는 삭제하거나 저비용 스토리지로 이동하기 편하기 때문입니다.

일반적인 로그 인덱스 네이밍 규칙은 **logs-{service}-{date}** 형태입니다. 예를 들어 "logs-nginx-2026.04.02" 같은 이름이 됩니다. 이 규칙을 따르면 특정 서비스의 특정 날짜 로그를 쉽게 식별하고 관리할 수 있습니다.

그런데 6.1.2에서 다룬 **Data Stream**을 사용하면 이 인덱스 네이밍과 롤오버를 Elasticsearch가 자동으로 관리합니다. 앞의 Logstash output 설정에서 data_stream 관련 파라미터를 지정한 부분을 다시 봅니다. data_stream_type을 "logs", data_stream_dataset을 "nginx", data_stream_namespace를 "production"으로 설정하면, Elasticsearch에 **logs-nginx-production**이라는 Data Stream이 생성됩니다. 이 Data Stream 아래에 backing index가 자동으로 만들어지고, 롤오버 조건이 충족되면 새 backing index로 전환됩니다.

Data Stream을 사용하려면 인덱스 템플릿이 미리 준비되어 있어야 합니다. Filebeat modules를 사용하면 템플릿이 자동으로 생성되지만, 커스텀 파이프라인에서는 직접 만들어야 합니다.

```json
PUT _index_template/logs-nginx
{
  "index_patterns": ["logs-nginx-*"],
  "data_stream": {},
  "template": {
    "settings": {
      "index.lifecycle.name": "logs-policy"
    },
    "mappings": {
      "properties": {
        "@timestamp": { "type": "date" },
        "source": {
          "properties": {
            "ip": { "type": "ip" }
          }
        },
        "http": {
          "properties": {
            "response": {
              "properties": {
                "status_code": { "type": "integer" },
                "body": {
                  "properties": {
                    "bytes": { "type": "long" }
                  }
                }
              }
            }
          }
        }
      }
    }
  },
  "priority": 200
}
```

index_patterns에 "logs-nginx-*"를 지정하여, logs-nginx로 시작하는 이름의 Data Stream에 이 템플릿이 적용되도록 합니다. data_stream 옵션을 빈 객체로 설정하면 Data Stream 모드가 활성화됩니다. settings의 index.lifecycle.name에 ILM 정책을 연결합니다.

마지막으로, **ILM과 로그 보존 정책**을 살펴봅니다. 6.1.1에서 ILM의 Hot-Warm-Cold-Delete 페이즈를 다뤘습니다. 로그 데이터에 ILM을 적용하면 최근 로그는 빠른 검색이 가능한 Hot 노드에 두고, 시간이 지나면 Warm, Cold 노드로 이동하며, 보존 기간이 지나면 자동으로 삭제합니다.

로그용 ILM 정책의 예를 봅니다.

```json
PUT _ilm/policy/logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_primary_shard_size": "50gb",
            "max_age": "1d"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "allocate": {
            "require": { "data": "cold" }
          }
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

Hot 페이즈에서 rollover 조건을 설정합니다. 프라이머리 샤드 크기가 50GB에 도달하거나 1일이 경과하면 새 backing index로 전환합니다. Warm 페이즈에서는 7일이 지난 인덱스의 샤드 수를 1개로 줄이고 세그먼트를 병합하여 저장 공간을 절약합니다. Cold 페이즈에서는 30일이 지난 인덱스를 저비용 노드로 이동합니다. Delete 페이즈에서는 90일이 지난 인덱스를 자동 삭제합니다.

전체 파이프라인의 데이터 흐름을 정리하면 다음과 같습니다.

```
서버 로그 파일
    |
    v
Filebeat (파일 모니터링, 이벤트 전송)
    |
    v
Logstash (grok 파싱 → ECS 정규화 → 타입 변환)
    |
    v
Elasticsearch Data Stream (logs-nginx-production)
    |-- .ds-logs-nginx-production-000001 (Hot, 쓰기 인덱스)
    |-- .ds-logs-nginx-production-000002 (Warm, 읽기 전용)
    |-- .ds-logs-nginx-production-000003 (Cold)
    |
    v
ILM 정책 (자동 롤오버 → Warm → Cold → Delete)
```

정리하면, 로그 수집 파이프라인은 Filebeat가 각 서버의 로그 파일을 모니터링하여 이벤트를 전송하고, Logstash가 grok 파싱과 ECS 정규화로 비정형 로그를 구조화하며, Data Stream 기반으로 Elasticsearch에 적재합니다. ILM 정책으로 로그의 보존과 삭제를 자동화하여 운영 부담을 줄입니다.

다음 단원인 8.2.2에서는 이렇게 수집된 로그 데이터를 Kibana에서 검색하고 시각화하는 대시보드를 구성하는 방법을 다룹니다.

이 단원을 마치면 Filebeat, Logstash, Elasticsearch를 연결한 로그 수집 파이프라인을 구성할 수 있습니다.
