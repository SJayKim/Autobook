# 4.1.2 Bucket 집계

4.1.1에서 집계 요청의 기본 구조를 다뤘습니다. aggs 키워드 아래에 집계 이름, 집계 타입, 대상 필드를 지정하면 Elasticsearch가 문서 집합에 대해 연산을 수행한다는 것을 확인했습니다. 이번 단원에서는 집계의 네 가지 범주 가운데 **Bucket 집계**를 다룹니다.

검색 결과로 돌아온 문서가 수천 건이라고 가정합니다. 이 문서를 특정 기준에 따라 그룹으로 나누고 싶은 상황이 자주 발생합니다. 쇼핑몰 상품을 카테고리별로 묶는다거나, 주문 기록을 월별로 나눈다거나, 가격을 구간별로 분류하는 경우가 그렇습니다. Bucket 집계는 이런 작업을 수행합니다. 문서 집합을 주어진 기준에 따라 여러 그룹으로 나누고, 각 그룹에 속하는 문서 수를 셉니다. 여기서 각 그룹을 **버킷**(bucket)이라고 부릅니다. 관계형 데이터베이스의 GROUP BY 절이 하는 일과 유사합니다.

Bucket 집계에는 여러 종류가 있습니다. 이 단원에서는 가장 자주 사용하는 terms, date_histogram, range, filter, filters, histogram, missing 집계를 하나씩 설명합니다.

가장 많이 쓰이는 Bucket 집계는 **terms 집계**입니다. terms 집계는 특정 필드의 고유한 값마다 버킷을 만들고, 각 버킷에 해당하는 문서 수를 셉니다. 예를 들어 상품 문서에 "category"라는 keyword 필드가 있고, 값이 "의류", "전자기기", "식품" 등으로 나뉜다면, terms 집계는 각 카테고리별로 버킷을 생성하여 문서 수를 집계합니다.

요청 예시를 봅니다.

```json
{
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": {
        "field": "category",
        "size": 10
      }
    }
  }
}
```

이 요청의 각 부분을 풀어 설명합니다. "by_category"는 4.1.1에서 다룬 사용자 정의 집계 이름입니다. "terms"가 집계 타입이며, "field": "category"는 category 필드의 값을 기준으로 그룹을 나누라는 뜻입니다. "size": 10은 상위 10개 버킷만 반환하라는 설정입니다. terms 집계의 size는 4.1.1에서 다룬 요청 최상위의 size와 다른 것입니다. 요청 최상위의 "size": 0은 검색 결과 문서(hits)를 생략하라는 의미이고, terms 안의 "size": 10은 버킷을 최대 10개까지 반환하라는 의미입니다.

terms 집계는 기본적으로 문서 수가 많은 순서로 버킷을 정렬합니다. 상위 N개라 함은, 문서 수가 가장 많은 N개의 값을 뜻합니다. category 필드에 50종의 값이 있어도 size를 10으로 설정하면 문서 수 상위 10개 카테고리만 돌아옵니다. 나머지 40개 카테고리에 속한 문서 수의 합은 4.1.1에서 설명한 sum_other_doc_count로 확인할 수 있습니다.

terms 집계를 사용할 때 주의할 점이 있습니다. terms 집계는 keyword 타입 필드에서 사용하는 것이 적합합니다. text 타입 필드는 분석기가 토큰으로 쪼개기 때문에, 원본 값 전체가 아니라 토큰 단위로 버킷이 생성됩니다. 예를 들어 "running shoes"라는 값이 text 필드에 들어 있으면, "running"과 "shoes"라는 두 개의 버킷이 따로 생깁니다. 원래 값 그대로 그룹을 나누려면 keyword 필드를 사용해야 합니다. 또한 값의 종류가 매우 많은 필드, 즉 카디널리티가 높은 필드에 terms 집계를 걸면 샤드마다 처리해야 하는 버킷 수가 급증하여 메모리와 CPU 부담이 커집니다. 이런 경우에는 size와 shard_size(4.1.1 참고)를 신중히 조정해야 합니다.

다음은 **date_histogram 집계**입니다. date_histogram 집계는 날짜 필드를 기준으로 일정한 시간 간격마다 버킷을 만듭니다. 시계열 데이터를 분석할 때 핵심이 되는 집계입니다. 예를 들어 주문 기록의 날짜 필드를 월 단위로 나누면, 월별 주문 건수를 파악할 수 있습니다.

요청 예시입니다.

```json
{
  "size": 0,
  "aggs": {
    "orders_over_time": {
      "date_histogram": {
        "field": "order_date",
        "calendar_interval": "month"
      }
    }
  }
}
```

"orders_over_time"이 집계 이름이고, "date_histogram"이 집계 타입입니다. "field": "order_date"는 order_date 필드를 기준으로 시간축 버킷을 만들라는 뜻입니다. "calendar_interval": "month"는 월 단위로 버킷을 나누라는 설정입니다. 이 요청을 실행하면 데이터에 존재하는 가장 이른 달부터 가장 늦은 달까지, 빈 달도 포함하여 월별 버킷이 생성됩니다. 각 버킷에는 해당 월에 속하는 문서 수가 doc_count로 표시됩니다.

date_histogram 집계에서 시간 간격을 지정하는 방식은 두 가지입니다. **calendar_interval**과 **fixed_interval**입니다. 이 둘의 차이를 이해하려면 달력의 특성을 떠올리면 됩니다.

calendar_interval은 달력 기준의 간격입니다. "month"로 설정하면 1월은 31일, 2월은 28일(또는 29일)처럼 달마다 실제 일수가 다릅니다. "week"로 설정하면 항상 월요일부터 일요일까지를 하나의 버킷으로 잡습니다. 달력에서 정하는 경계를 따르기 때문에, 버킷마다 포함하는 절대 시간의 길이가 다를 수 있습니다. calendar_interval에 사용할 수 있는 값은 minute, hour, day, week, month, quarter, year입니다.

fixed_interval은 고정된 시간 길이의 간격입니다. "30d"로 설정하면 달력과 무관하게 정확히 30일(30 * 24 * 60 * 60 * 1000 밀리초)마다 버킷을 나눕니다. "1h"로 설정하면 정확히 60분마다 나눕니다. 달력의 경계를 무시하고 균일한 시간 폭으로 나누고 싶을 때 사용합니다. 숫자와 단위를 조합하여 지정하며, 사용할 수 있는 단위는 ms(밀리초), s(초), m(분), h(시), d(일)입니다.

두 방식의 차이를 정리하면 다음과 같습니다.

```
calendar_interval vs fixed_interval

calendar_interval                     fixed_interval
----------------------------------    ----------------------------------
달력 경계를 따름                       고정 시간 폭
"month" -> 1월=31일, 2월=28일          "30d" -> 항상 30일
"week"  -> 월~일 달력 기준              "7d"  -> 임의 시점부터 7일
값: minute, hour, day, week,           값: 숫자+단위 (ms, s, m, h, d)
    month, quarter, year
버킷 길이가 달마다 다를 수 있음          버킷 길이가 항상 동일
```

"월별 매출 추이"처럼 달력 단위가 자연스러운 분석에는 calendar_interval을, "30분 간격 서버 지표 집계"처럼 균일한 시간 폭이 필요한 분석에는 fixed_interval을 선택합니다.

다음은 **range 집계**입니다. range 집계는 숫자 또는 날짜 필드의 값을 사용자가 직접 정의한 구간으로 나눕니다. terms 집계가 필드의 고유한 값마다 자동으로 버킷을 만드는 것과 달리, range 집계는 사용자가 각 구간의 시작과 끝을 직접 지정합니다.

요청 예시를 봅니다.

```json
{
  "size": 0,
  "aggs": {
    "price_ranges": {
      "range": {
        "field": "price",
        "ranges": [
          { "to": 50000 },
          { "from": 50000, "to": 100000 },
          { "from": 100000 }
        ]
      }
    }
  }
}
```

"range"가 집계 타입이고, "field": "price"는 price 필드 값을 기준으로 구간을 나누라는 뜻입니다. "ranges" 배열 안에 각 구간을 정의합니다. 첫 번째 객체 { "to": 50000 }은 price가 50000 미만인 문서를 하나의 버킷으로 묶습니다. 두 번째 { "from": 50000, "to": 100000 }은 50000 이상 100000 미만인 문서를 묶습니다. 세 번째 { "from": 100000 }은 100000 이상인 문서를 묶습니다. 여기서 from은 해당 값을 포함하고, to는 해당 값을 포함하지 않습니다. 수학 표기로 나타내면 [from, to) 구간입니다.

range 집계는 숫자 필드뿐 아니라 날짜 필드에도 사용할 수 있습니다. 날짜 필드에 적용할 때는 from과 to에 날짜 문자열을 넣습니다. 예를 들어 { "from": "2025-01-01", "to": "2025-07-01" }처럼 지정하면 해당 기간의 문서를 하나의 버킷으로 묶습니다.

다음은 **filter 집계**입니다. filter 집계는 하나의 쿼리 조건을 기준으로 단일 버킷을 만듭니다. 조건에 맞는 문서만 버킷에 들어가고, 그 버킷 안에서 하위 집계를 실행할 수 있습니다. 예를 들어 "status가 active인 상품만 모아서 평균 가격을 구하라"는 분석을 할 때, filter 집계로 active 상품만 버킷에 담고, 그 안에 avg 하위 집계를 넣는 방식을 사용합니다.

```json
{
  "size": 0,
  "aggs": {
    "active_products": {
      "filter": {
        "term": { "status": "active" }
      },
      "aggs": {
        "avg_price": {
          "avg": { "field": "price" }
        }
      }
    }
  }
}
```

이 요청에서 "filter" 안에 들어가는 것은 일반적인 쿼리 조건입니다. term, range, bool 등 검색에서 사용하는 쿼리를 그대로 사용할 수 있습니다. filter 집계가 만든 버킷 안에서 "aggs"로 하위 집계를 정의하면, 필터 조건을 만족하는 문서에 대해서만 하위 집계가 실행됩니다. "왜 query에서 필터를 걸지 않고 filter 집계를 쓰는가"라는 의문이 생길 수 있습니다. query의 filter 절은 전체 집계에 동일하게 적용됩니다. 반면 filter 집계는 여러 집계 중 특정 집계에만 별도의 조건을 적용하고 싶을 때 유용합니다.

**filters 집계**는 filter 집계의 확장입니다. filter 집계가 하나의 조건으로 단일 버킷을 만든다면, filters 집계는 여러 조건을 한꺼번에 정의하여 각 조건마다 이름이 붙은 버킷을 만듭니다.

```json
{
  "size": 0,
  "aggs": {
    "status_buckets": {
      "filters": {
        "filters": {
          "active": { "term": { "status": "active" } },
          "archived": { "term": { "status": "archived" } }
        }
      }
    }
  }
}
```

바깥의 "filters"가 집계 타입이고, 안쪽의 "filters"가 각 조건을 담는 객체입니다. "active"와 "archived"가 각 버킷의 이름이며, 각각의 쿼리 조건에 맞는 문서가 해당 버킷에 들어갑니다. 응답에서는 "active"와 "archived" 키 아래에 각 버킷의 doc_count가 표시됩니다. 한 번의 요청으로 서로 다른 여러 조건의 문서 수를 동시에 파악할 수 있어, 조건별 비교 분석에 적합합니다.

다음은 **histogram 집계**입니다. histogram 집계는 숫자 필드를 일정한 간격으로 나누어 버킷을 만듭니다. range 집계가 사용자가 각 구간을 직접 지정하는 것과 달리, histogram 집계는 간격(interval)만 지정하면 Elasticsearch가 자동으로 구간을 생성합니다.

```json
{
  "size": 0,
  "aggs": {
    "price_histogram": {
      "histogram": {
        "field": "price",
        "interval": 10000
      }
    }
  }
}
```

"interval": 10000은 price 값을 10000 단위로 나누라는 뜻입니다. 0~9999, 10000~19999, 20000~29999 순으로 버킷이 자동 생성됩니다. 데이터에 존재하는 최솟값과 최댓값 범위 안에서 간격마다 버킷이 만들어지며, 문서가 하나도 없는 구간의 버킷도 기본적으로 생성됩니다. 빈 버킷을 제외하고 싶으면 "min_doc_count": 1을 추가합니다. histogram 집계는 가격 분포, 점수 분포 등 연속적인 숫자 값의 분포를 한눈에 파악할 때 유용합니다. date_histogram이 날짜 필드에 특화된 것이라면, histogram은 일반 숫자 필드에 특화된 것으로 볼 수 있습니다.

마지막으로 **missing 집계**입니다. missing 집계는 특정 필드에 값이 없는 문서를 하나의 버킷으로 모읍니다. Elasticsearch에서 문서는 스키마가 유연하기 때문에, 어떤 문서에는 특정 필드가 존재하지 않거나 값이 null일 수 있습니다. missing 집계는 이런 문서가 몇 건인지 파악할 때 사용합니다.

```json
{
  "size": 0,
  "aggs": {
    "no_category": {
      "missing": {
        "field": "category"
      }
    }
  }
}
```

이 요청은 category 필드에 값이 없는 문서를 하나의 버킷으로 묶어 개수를 셉니다. 데이터 품질을 점검하거나, 값이 빠진 문서를 식별할 때 유용합니다. missing 집계 안에도 하위 집계를 넣을 수 있습니다. 예를 들어 category가 없는 상품의 평균 가격을 구하려면, missing 집계 안에 avg 하위 집계를 추가하면 됩니다.

지금까지 다룬 Bucket 집계의 종류와 용도를 정리합니다.

```
Bucket 집계 요약

집계 타입         기준                     버킷 생성 방식
-----------      ---------------------    ----------------------------------
terms            필드의 고유 값            값마다 자동 생성, 상위 N개 반환
date_histogram   날짜 필드의 시간 간격      calendar_interval 또는 fixed_interval
histogram        숫자 필드의 고정 간격      interval 값마다 자동 구간 생성
range            숫자/날짜 필드의 구간      사용자가 from/to를 직접 지정
filter           단일 쿼리 조건            조건에 맞는 문서 1개 버킷
filters          복수 쿼리 조건            조건마다 이름 붙은 버킷 생성
missing          필드 값 부재              값이 없는 문서 1개 버킷
```

Bucket 집계는 단독으로 사용하기도 하지만, 버킷 안에 하위 집계를 넣어 조합하는 것이 일반적입니다. 예를 들어 terms 집계로 국가별 버킷을 만들고, 각 버킷 안에 sum 하위 집계를 넣으면 국가별 매출 합계를 구할 수 있습니다.

```json
{
  "size": 0,
  "aggs": {
    "by_country": {
      "terms": { "field": "country" },
      "aggs": {
        "total_sales": {
          "sum": { "field": "sales" }
        }
      }
    }
  }
}
```

이 요청의 실행 흐름을 정리합니다.

```
중첩 집계 실행 흐름

1. terms 집계 실행
   - country 필드의 고유 값마다 버킷 생성
   - 예: "KR" 버킷, "US" 버킷, "JP" 버킷 ...

2. 각 버킷 안에서 sum 하위 집계 실행
   - "KR" 버킷 -> KR 문서의 sales 합계
   - "US" 버킷 -> US 문서의 sales 합계
   - "JP" 버킷 -> JP 문서의 sales 합계

3. 응답
   - 각 버킷에 doc_count와 total_sales.value 포함
```

외부의 Bucket 집계가 문서를 그룹으로 나누고, 내부의 하위 집계가 각 그룹 안에서 계산을 수행하는 구조입니다. 4.1.1에서 다룬 집계 구문에서 aggs 안에 다시 aggs를 넣는 방식으로 이 중첩을 표현합니다. 다만 중첩 단계가 깊어질수록 버킷 수가 곱셈으로 증가하여 메모리 사용량이 급격히 커질 수 있으므로, 중첩은 필요한 만큼만 사용해야 합니다.

정리하면, Bucket 집계는 문서를 기준에 따라 여러 버킷으로 나누는 집계입니다. terms는 필드 값별로, date_histogram은 시간 간격별로, histogram은 숫자 간격별로, range는 사용자 지정 구간별로, filter와 filters는 쿼리 조건별로, missing은 값 부재 여부로 문서를 그룹화합니다. 각 버킷 안에 하위 집계를 중첩하면, 그룹별 통계를 산출할 수 있습니다.

다음 단원인 4.1.3에서는 Metric 집계를 다룹니다.

이 단원을 마치면 terms, date_histogram, range, filter 집계로 문서를 그룹화할 수 있습니다.
