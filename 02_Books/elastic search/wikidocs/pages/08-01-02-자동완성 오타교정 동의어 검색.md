# 8.1.2 자동완성, 오타교정, 동의어 검색

전자상거래 사이트에서 사용자가 검색창에 글자를 입력하기 시작하면, 입력 도중에 후보 상품명이 드롭다운으로 나타납니다. "무선"까지만 입력해도 "무선 이어폰", "무선 충전기", "무선 키보드" 같은 후보가 즉시 보입니다. 사용자가 "이어폰"을 "이어본"으로 잘못 입력해도 원하는 상품이 결과에 포함됩니다. "이어버드"라고 검색하면 "이어폰"으로 등록된 상품까지 함께 찾아줍니다. 이런 기능은 자동완성, 오타교정, 동의어 확장이라는 세 가지 기법이 결합되어 동작합니다.

이 단원에서는 8.1.1에서 설계한 상품 인덱스 위에 세 가지 기법을 통합하여 검색 서비스를 구현합니다. 3.3.2에서 다룬 자동완성 구현 방식과 3.4.3에서 다룬 function_score 쿼리가 기반이 됩니다.

먼저 자동완성부터 살펴봅니다. 3.3.2에서 completion suggester, search_as_you_type, edge_ngram, match_phrase_prefix 네 가지 자동완성 방식을 다뤘습니다. 전자상거래 상품 검색에서는 **search_as_you_type** 필드 타입이 가장 균형 잡힌 선택입니다.

search_as_you_type은 하나의 필드를 선언하면 Elasticsearch가 내부적으로 여러 서브필드를 자동 생성하는 전용 타입입니다. 원본 필드 외에 _2gram, _3gram, _index_prefix라는 세 개의 서브필드가 만들어집니다. _2gram과 _3gram은 인접한 토큰 2개, 3개를 묶은 shingle(연속 토큰 조합)을 저장합니다. _index_prefix는 마지막 토큰의 접두사를 edge_ngram 방식으로 미리 분해하여 저장합니다. 이 서브필드 덕분에 사용자가 단어를 다 입력하지 않은 상태에서도 빠른 매칭이 가능합니다.

8.1.1에서 만든 상품 인덱스에 자동완성용 필드를 추가합니다. 상품 이름의 멀티필드에 search_as_you_type 서브필드를 하나 더 추가하는 방식입니다.

```json
"name": {
  "type": "text",
  "analyzer": "nori_product",
  "fields": {
    "keyword": {
      "type": "keyword",
      "ignore_above": 256
    },
    "autocomplete": {
      "type": "search_as_you_type",
      "analyzer": "nori_product"
    }
  }
}
```

name.autocomplete 서브필드에 search_as_you_type을 지정했습니다. 이 서브필드에 대해 multi_match 쿼리를 사용하면, 사용자가 입력 중인 불완전한 텍스트에도 매칭됩니다.

자동완성 쿼리를 어떻게 보내는지 살펴봅니다. 사용자가 "무선 이"까지 입력한 상태를 예로 듭니다.

```json
GET /products/_search
{
  "query": {
    "multi_match": {
      "query": "무선 이",
      "type": "bool_prefix",
      "fields": [
        "name.autocomplete",
        "name.autocomplete._2gram",
        "name.autocomplete._3gram"
      ]
    }
  }
}
```

multi_match 쿼리의 type을 "bool_prefix"로 지정합니다. 이 모드는 검색어의 마지막 토큰을 접두사로 취급합니다. "무선"은 완성된 토큰으로 정확히 매칭하고, "이"는 접두사로 취급하여 "이어폰", "이어버드" 등 "이"로 시작하는 토큰에 매칭됩니다. fields에 원본, _2gram, _3gram을 모두 지정하면 연속 토큰 조합까지 고려하여 더 정확한 결과를 반환합니다.

이 방식과 대비되는 것이 3.3.2에서 다룬 **prefix 쿼리** 기반 자동완성입니다. prefix 쿼리는 keyword 필드의 값이 특정 접두사로 시작하는 문서를 찾습니다. 분석기를 거치지 않으므로, "무선 블루투스 이어폰"이라는 상품 이름에서 "무선 블루"까지 정확히 입력해야 매칭됩니다. "이어폰"으로 시작하는 검색어에는 이 상품이 나타나지 않습니다. search_as_you_type은 분석기를 거친 토큰 단위로 매칭하므로, 어느 위치의 토큰이든 접두사가 일치하면 결과에 포함됩니다. 전자상거래 검색에서는 search_as_you_type 방식이 훨씬 유연합니다.

다음으로 **오타교정**을 살펴봅니다. 사용자가 "이어본"이라고 입력했을 때 "이어폰" 상품을 찾아주는 기능입니다. Elasticsearch의 match 쿼리에는 **fuzziness** 파라미터가 있습니다. fuzziness는 검색어와 인덱스 토큰 사이에 허용할 편집 거리(edit distance)를 지정합니다.

편집 거리란 한 문자열을 다른 문자열로 바꾸는 데 필요한 최소 조작 횟수입니다. 조작에는 글자 삽입, 삭제, 치환, 인접 글자 교환 네 가지가 있습니다. "이어본"을 "이어폰"으로 바꾸려면 "본"을 "폰"으로 치환하면 되므로 편집 거리는 1입니다.

fuzziness를 "AUTO"로 설정하면 Elasticsearch가 검색어의 길이에 따라 허용 편집 거리를 자동으로 결정합니다. 1~2글자 검색어에는 편집 거리 0(정확 일치만), 3~5글자에는 편집 거리 1, 6글자 이상에는 편집 거리 2를 적용합니다. 짧은 검색어에서는 오타교정이 너무 관대하면 전혀 관련 없는 단어에 매칭되기 때문입니다.

```json
GET /products/_search
{
  "query": {
    "match": {
      "name": {
        "query": "이어본",
        "fuzziness": "AUTO"
      }
    }
  }
}
```

"이어본"은 3글자이므로 AUTO가 편집 거리 1을 허용합니다. "이어폰"과의 편집 거리가 1이므로 매칭에 성공합니다.

다음으로 **동의어 확장**을 살펴봅니다. 8.1.1에서 분석기에 동의어 필터를 추가하여 "이어폰"과 "이어버드"를 같은 그룹으로 묶었습니다. 이 설정이 동작하는 방식을 구체적으로 알아봅니다.

동의어 필터가 인덱싱 시점에 적용되면, "무선 이어폰"이라는 상품 이름을 분석할 때 "이어폰" 토큰과 함께 "이어버드" 토큰도 역색인에 등록됩니다. 이후 사용자가 "이어버드"로 검색하면, 역색인에 "이어버드" 토큰이 이미 등록되어 있으므로 해당 상품이 검색 결과에 나타납니다.

반대로 검색 시점에 동의어 필터를 적용할 수도 있습니다. 이 경우 인덱싱 시점에는 "이어폰"만 역색인에 등록되고, 검색 시점에 "이어버드"라는 검색어가 "이어폰"으로 확장되어 매칭됩니다. 인덱싱 시점 적용은 인덱스 크기가 약간 커지지만 검색 성능이 빠르고, 검색 시점 적용은 동의어 목록을 변경할 때 재인덱싱이 필요 없다는 장점이 있습니다.

스팸이나 불필요한 검색 결과를 줄이기 위해 **불용어 처리**도 함께 고려합니다. 불용어(stop words)란 검색에 유의미한 영향을 주지 않는 단어입니다. 한국어에서 "의", "에", "를" 같은 조사가 대표적입니다. 분석기에 stop 토큰 필터를 추가하면, 이런 단어가 역색인에 등록되지 않아 불필요한 매칭을 줄일 수 있습니다. 또한 전자상거래에서는 "무료배송", "특가" 같은 스팸성 키워드가 상품 이름에 포함되는 경우가 있습니다. 이런 단어를 불용어 목록에 추가하면 검색 품질을 높일 수 있습니다.

이제 검색 결과의 순위를 비즈니스 요구에 맞게 조정하는 방법을 살펴봅니다. 텍스트 관련성만으로 순위를 매기면, BM25 점수가 높은 상품이 무조건 상위에 노출됩니다. 하지만 판매량이 많거나 평점이 높은 인기 상품이 먼저 노출되어야 사용자 경험이 좋아집니다. 3.4.3에서 다룬 **function_score 쿼리**가 이 역할을 합니다.

function_score는 기존 쿼리의 BM25 점수 위에 사용자 정의 함수를 결합하여 최종 점수를 만드는 구조입니다. 판매량과 평점을 점수에 반영하는 예를 봅니다.

```json
GET /products/_search
{
  "query": {
    "function_score": {
      "query": {
        "match": {
          "name": {
            "query": "무선 이어폰",
            "fuzziness": "AUTO"
          }
        }
      },
      "functions": [
        {
          "field_value_factor": {
            "field": "sales_count",
            "modifier": "log1p",
            "factor": 0.5
          }
        },
        {
          "field_value_factor": {
            "field": "rating",
            "modifier": "none",
            "factor": 1.2
          }
        }
      ],
      "score_mode": "multiply",
      "boost_mode": "multiply"
    }
  }
}
```

functions 배열에 두 개의 field_value_factor 함수를 넣었습니다. 첫 번째 함수는 sales_count(판매량) 필드의 값을 점수에 반영합니다. modifier를 "log1p"로 설정하면 log(1 + sales_count)를 계산합니다. 판매량이 0이면 log(1) = 0이 되어 점수에 영향을 주지 않고, 판매량이 늘어날수록 점수가 완만하게 증가합니다. factor는 이 값에 곱해지는 가중치입니다. 두 번째 함수는 rating(평점) 필드의 값을 그대로 사용합니다. 평점이 4.5인 상품은 3.0인 상품보다 높은 부스팅을 받습니다.

score_mode가 "multiply"이므로 두 함수의 결과를 곱합니다. boost_mode가 "multiply"이므로 이 결과를 원래 BM25 점수에 다시 곱합니다. 최종 점수는 "BM25 점수 * log1p(sales_count) * 0.5 * rating * 1.2"가 됩니다.

마지막으로, 앞에서 다룬 자동완성, 오타교정, 동의어, function_score 부스팅을 하나의 검색 흐름으로 엮는 방법을 살펴봅니다. 전자상거래 검색의 전체 UX 플로우는 다음과 같이 두 단계로 나뉩니다.

```
[1단계: 자동완성]
  사용자 입력 ──▶ search_as_you_type 쿼리
                  ──▶ 후보 상품명 목록 반환 (드롭다운)

[2단계: 본 검색]
  사용자가 후보를 선택하거나 Enter 입력
    ──▶ match 쿼리 + fuzziness: AUTO (오타교정)
    ──▶ 동의어 필터가 분석 시점에 유사어 확장
    ──▶ function_score로 판매량/평점 부스팅
    ──▶ 최종 검색 결과 반환
```

1단계에서는 사용자가 글자를 입력할 때마다 search_as_you_type 기반의 자동완성 쿼리를 실행합니다. 이 쿼리는 수 밀리초 이내에 응답해야 하므로, 가벼운 multi_match 쿼리만 사용합니다.

2단계에서는 사용자가 후보를 선택하거나 검색어를 완성하여 Enter를 누른 시점에 본 검색 쿼리를 실행합니다. match 쿼리에 fuzziness를 적용하여 오타를 교정하고, 분석기에 내장된 동의어 필터가 유사어를 확장합니다. function_score로 판매량과 평점을 점수에 반영하여 비즈니스에 맞는 순위를 만듭니다.

정리하면, 전자상거래 검색 서비스는 search_as_you_type으로 자동완성, fuzziness: AUTO로 오타교정, 동의어 필터로 유사어 확장, function_score로 비즈니스 부스팅을 결합하여 구현합니다. 자동완성은 입력 도중의 빠른 후보 제안, 본 검색은 정확성과 비즈니스 순위를 모두 만족하는 이중 구조입니다.

다음 단원인 8.2.1에서는 Elasticsearch를 검색이 아닌 로그 분석 용도로 활용하는 방법을 다룹니다. Filebeat와 Logstash를 사용하여 로그를 수집하고 Elasticsearch에 적재하는 파이프라인을 구성합니다.

이 단원을 마치면 자동완성, 오타교정, 동의어 확장을 통합한 검색 서비스를 구현할 수 있습니다.
