# 4.1.1 집계 구문 구조

지금까지 3장에서 다룬 검색은 조건에 맞는 문서를 찾아 반환하는 작업이었습니다. "제목에 Elasticsearch가 포함된 글을 찾아라", "상태가 published인 문서만 보여 달라"처럼 개별 문서를 식별하는 것이 검색의 목적이었습니다. 그런데 실무에서는 개별 문서가 아니라 문서 집합의 전체적인 특성을 파악해야 할 때가 있습니다. 예를 들어 온라인 쇼핑몰을 운영한다고 가정합니다. "이번 달 매출 총액이 얼마인가", "카테고리별로 상품이 몇 건씩 등록되어 있는가", "일별 주문 건수 추이가 어떻게 되는가" 같은 질문은 문서 하나를 반환하는 것으로는 답할 수 없습니다. 여러 문서를 묶어서 합계, 평균, 그룹별 개수 같은 통계를 산출해야 합니다.

Elasticsearch에서 이런 통계 연산을 담당하는 기능이 **집계**(aggregation)입니다. 집계는 검색 대상 문서 집합에 함수를 적용하여, 합계나 평균 같은 수치를 계산하거나, 문서를 기준에 따라 그룹으로 나누는 작업을 수행합니다. 관계형 데이터베이스의 GROUP BY와 SUM, AVG 등의 집계 함수를 떠올리면 유사한 역할입니다. 이 단원에서는 집계 요청을 작성하는 기본 구문 구조를 다룹니다. 어떤 키워드로 집계를 선언하고, 각 부분이 어떤 역할을 하며, 응답은 어떤 형태로 돌아오는지를 순서대로 설명합니다.

Elasticsearch에서 집계를 요청할 때는 검색 요청 본문에 **aggs**라는 키워드를 추가합니다. aggs는 aggregations의 줄임말이며, 줄이지 않고 **aggregations**라고 써도 동일하게 동작합니다. 두 키워드는 완전히 같은 의미이므로 어느 쪽을 써도 무방하지만, 실무에서는 짧은 aggs를 더 자주 사용합니다.

aggs 키워드 아래에는 집계 하나하나를 정의합니다. 기본 구조를 먼저 보겠습니다.

```json
{
  "size": 0,
  "query": { ... },
  "aggs": {
    "my_aggregation_name": {
      "avg": {
        "field": "price"
      }
    }
  }
}
```

이 요청에는 세 가지 요소가 보입니다. 하나씩 풀어 설명합니다.

첫 번째는 **집계 이름**입니다. 위 예시에서 "my_aggregation_name"에 해당하는 부분입니다. 이 이름은 Elasticsearch가 정해 주는 것이 아니라 사용자가 직접 짓는 이름입니다. 응답에서 집계 결과를 식별하는 키로 쓰이기 때문에, 결과의 의미를 알아볼 수 있는 이름을 붙이는 것이 좋습니다. 예를 들어 상품 가격의 평균을 구한다면 "average_price"처럼 짓고, 카테고리별 문서 수를 센다면 "by_category"처럼 짓습니다. 한 요청에 여러 집계를 넣을 수도 있으므로, 이름이 서로 겹치지 않아야 합니다.

두 번째는 **집계 타입**입니다. 위 예시에서 "avg"에 해당하는 부분입니다. 집계 타입은 Elasticsearch가 미리 정의해 둔 연산 종류를 가리킵니다. avg는 평균을 구하고, sum은 합계를 구하고, terms는 필드 값별로 문서를 그룹으로 나눕니다. 집계 이름은 사용자가 자유롭게 짓지만, 집계 타입은 Elasticsearch가 제공하는 이름 중에서 골라야 합니다. 집계 타입 안에는 대상 필드나 추가 옵션을 지정합니다. 위 예시에서 "field": "price"는 price 필드의 값을 대상으로 평균을 구하라는 뜻입니다.

이 구조를 도식으로 나타내면 다음과 같습니다.

```
aggs 요청 구조

{
  "aggs": {
    "<집계 이름>"  ← 사용자가 직접 정한 식별자
      "<집계 타입>"  ← Elasticsearch가 제공하는 연산 종류 (avg, sum, terms 등)
        "field": "<필드명>"  ← 연산 대상 필드
        ...  ← 타입별 추가 옵션
}
```

세 번째는 요청 최상위에 있는 **"size": 0** 설정입니다. Elasticsearch의 _search API는 기본적으로 검색 결과 문서(hits)를 함께 반환합니다. 그런데 집계만 필요하고 개별 문서 목록은 필요 없는 경우가 많습니다. 이때 size를 0으로 설정하면 hits 배열이 비어서 돌아옵니다. 개별 문서를 직렬화하고 전송하는 비용이 사라지므로 응답이 빨라집니다. 또한 size가 0인 집계 전용 요청은 Elasticsearch 내부에서 쿼리 캐시 대상이 되어, 동일한 요청이 반복될 때 성능이 더 향상됩니다. 집계 결과와 함께 상위 문서 몇 건도 보고 싶다면 size를 0이 아닌 값으로 설정하면 됩니다.

이제 쿼리와 집계를 함께 사용하는 경우를 봅니다. aggs는 query와 같은 레벨에 놓입니다. query가 있으면, 집계는 query에 의해 걸러진 문서 집합에 대해서만 실행됩니다. query가 없으면 인덱스의 전체 문서가 집계 대상이 됩니다. 3.2.1에서 다룬 query context와 filter context의 구분이 여기서도 그대로 적용됩니다.

구체적인 예를 하나 들어 봅니다. "status가 published인 상품의 평균 가격을 구하라"는 요청입니다.

```json
{
  "size": 0,
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "published" } }
      ]
    }
  },
  "aggs": {
    "average_price": {
      "avg": {
        "field": "price"
      }
    }
  }
}
```

이 요청의 실행 흐름을 단계별로 정리합니다.

```
쿼리 + 집계 실행 흐름

1. query 실행
   - bool > filter > term: status가 "published"인 문서만 추림
   - filter context이므로 점수 계산 없음 (3.2.1 참고)

2. 집계 실행
   - 1단계에서 추려진 문서 집합을 대상으로 avg 집계 실행
   - price 필드 값의 평균을 계산

3. 응답 반환
   - size: 0이므로 hits 배열은 비어 있음
   - aggregations 객체에 집계 결과 포함
```

query를 filter 절에 넣은 이유는, 상태 값 필터링에는 관련성 점수가 필요 없기 때문입니다. 3.2.1에서 다뤘듯이, 예/아니오로 판단할 수 있는 조건은 filter context에 두는 것이 점수 계산 비용을 절약하고 캐싱 혜택도 얻는 방법입니다. 집계 앞에 query로 대상 범위를 좁히면, 집계가 처리할 문서 수가 줄어들어 전체 응답 속도가 빨라집니다.

다음으로 집계 응답의 구조를 봅니다. 위 요청에 대한 응답은 다음과 같은 형태입니다.

```json
{
  "took": 15,
  "hits": {
    "total": { "value": 320, "relation": "eq" },
    "hits": []
  },
  "aggregations": {
    "average_price": {
      "value": 24500.75
    }
  }
}
```

응답의 최상위에 **aggregations**라는 객체가 있습니다. 이 객체 안에 요청할 때 사용자가 지정한 집계 이름이 키로 나타납니다. 위 예시에서는 "average_price"라는 이름으로 요청했으므로, 같은 이름으로 결과가 돌아옵니다. 그 안의 "value": 24500.75가 평균 가격 계산 결과입니다. hits.hits 배열이 비어 있는 것은 size를 0으로 설정했기 때문입니다. hits.total.value가 320이므로, status가 published인 문서 320건의 price 평균이 24500.75라는 뜻입니다.

집계 이름을 사용자가 정하기 때문에, 한 요청에 여러 집계를 넣으면 응답에서도 각 이름으로 결과를 구분할 수 있습니다. 예를 들어 평균과 합계를 동시에 요청하면 다음과 같습니다.

```json
{
  "size": 0,
  "aggs": {
    "average_price": {
      "avg": { "field": "price" }
    },
    "total_revenue": {
      "sum": { "field": "price" }
    }
  }
}
```

이 요청의 응답에서 aggregations 객체 안에 "average_price"와 "total_revenue" 두 개의 키가 각각 자신의 결과를 담아 돌아옵니다.

집계 결과에는 정확도와 관련된 보조 정보가 포함되는 경우가 있습니다. 예를 들어 terms 집계처럼 필드 값별로 문서를 그룹으로 나누는 집계에서는, 응답에 **doc_count_error_upper_bound**라는 필드가 함께 나타납니다. Elasticsearch는 분산 시스템이므로 데이터가 여러 샤드에 나뉘어 저장되어 있습니다. 각 샤드는 자기가 가진 문서만 보고 상위 그룹을 계산한 뒤, 코디네이터 노드가 이를 합칩니다. 이 과정에서 일부 샤드에만 소량 존재하는 그룹이 최종 결과에서 누락될 수 있습니다. doc_count_error_upper_bound는 이런 누락으로 인해 각 그룹의 문서 수에 발생할 수 있는 최대 오차를 나타냅니다. 이 값이 0이면 결과가 정확하다는 뜻이고, 0보다 크면 실제 문서 수가 표시된 것보다 많을 수 있다는 뜻입니다.

정리하면, Elasticsearch의 집계 요청은 aggs(또는 aggregations) 키워드 아래에 사용자 정의 이름, 집계 타입, 대상 필드를 지정하는 구조입니다. size를 0으로 두면 검색 결과 문서를 생략하고 집계 결과만 받을 수 있으며, query와 조합하면 특정 조건을 만족하는 문서만 대상으로 집계를 수행합니다. 응답은 aggregations 객체 안에 요청 시 지정한 집계 이름을 키로 하여 결과를 담아 돌아옵니다.

다음 단원인 4.1.2에서는 Bucket 집계를 다룹니다.

이 단원을 마치면 aggs 파라미터의 기본 구조를 작성할 수 있으며, 쿼리와 집계를 결합한 필터링 집계를 수행할 수 있습니다.
