# 3.3.4 페이지네이션

Elasticsearch에서 검색 결과가 수십 건이면 한 번의 요청으로 전부 받아도 문제가 없습니다. 하지만 검색 조건에 맞는 문서가 수만 건, 수십만 건이라면 한 번에 모두 가져오는 것은 현실적이지 않습니다. 메모리를 한꺼번에 많이 쓰고, 응답 시간도 길어집니다. 이때 결과를 일정 단위로 나누어 한 묶음씩 가져오는 방식이 필요합니다. 이 방식을 **페이지네이션**이라고 합니다. 웹 검색 결과 하단에 "1, 2, 3, ... 다음" 같은 페이지 버튼이 있는 것이 대표적인 예입니다.

Elasticsearch는 페이지네이션을 위해 세 가지 방법을 제공합니다. from/size, search_after, scroll입니다. 세 방법은 각각 다른 상황에 맞게 설계되어 있으므로, 어떤 상황에서 어떤 방법을 선택해야 하는지를 이해하는 것이 중요합니다. 이 단원에서는 from/size부터 시작하여 그 한계를 확인하고, 그 한계를 넘기 위해 search_after와 scroll이 왜 필요한지를 차례로 살펴봅니다.

가장 직관적인 방법은 **from**과 **size** 두 파라미터를 사용하는 것입니다. size는 한 번에 가져올 문서의 최대 개수를 지정합니다. from은 결과 목록에서 앞쪽 몇 건을 건너뛸지를 지정합니다. 기본값은 from이 0, size가 10입니다.

예를 들어 검색 결과를 10건씩 나누어 볼 때, 첫 번째 페이지는 from을 0, size를 10으로 요청합니다. 두 번째 페이지는 from을 10, size를 10으로 요청합니다. 세 번째 페이지는 from을 20으로 올립니다. 이런 식으로 from 값을 페이지 번호에 맞추어 증가시키면 원하는 페이지의 결과를 받을 수 있습니다.

```json
POST my-index/_search
{
  "from": 20,
  "size": 10,
  "query": {
    "match": { "title": "elasticsearch" }
  }
}
```

위 요청에서 from이 20이고 size가 10이므로, 검색 결과 중 상위 20건을 건너뛰고 21번째부터 30번째까지 10건을 반환합니다.

from/size 방식은 간단하지만, 페이지를 깊이 넘길수록 심각한 문제가 생깁니다. 이 문제를 이해하려면 Elasticsearch가 분산 환경에서 검색을 처리하는 과정을 알아야 합니다. 1.3.3에서 다루었듯이 인덱스는 여러 샤드로 나뉘어 있습니다. from이 20이고 size가 10인 요청이 들어오면, 각 샤드는 자기가 가진 문서 중 상위 30건(from + size)을 메모리에 올려 코디네이팅 노드로 보냅니다. 코디네이팅 노드는 모든 샤드에서 받은 후보를 모아 정렬한 뒤, 그중 21번째부터 30번째까지 10건만 골라 클라이언트에 반환합니다.

문제는 from 값이 커질수록 각 샤드가 메모리에 올려야 하는 문서 수가 함께 늘어난다는 점입니다. from이 9990이고 size가 10이면, 각 샤드는 상위 10000건을 메모리에 올려야 합니다. 샤드가 5개라면 코디네이팅 노드는 50000건을 받아 정렬한 뒤 10건만 골라냅니다. 나머지 49990건은 버려집니다. 페이지가 깊어질수록 이 낭비가 가파르게 증가하며, 메모리 사용량과 응답 시간이 함께 늘어납니다. 이 현상을 **deep pagination 문제**라고 합니다.

```
deep pagination에서 각 샤드의 부담

요청: from=9990, size=10  →  각 샤드가 올려야 할 문서 수: 10000
샤드 수: 5

  샤드 1 → 상위 10000건 전송 ─┐
  샤드 2 → 상위 10000건 전송 ─┤
  샤드 3 → 상위 10000건 전송 ─┼→ 코디네이팅 노드: 50000건 수신
  샤드 4 → 상위 10000건 전송 ─┤        → 정렬 후 10건만 반환
  샤드 5 → 상위 10000건 전송 ─┘        → 49990건 폐기
```

Elasticsearch는 이 문제를 방지하기 위해 from + size의 합계에 상한을 둡니다. 이 상한이 **index.max_result_window** 설정이며, 기본값은 10000입니다. 즉, from과 size를 합한 값이 10000을 넘는 요청은 거부됩니다. 9900번째부터 100건을 가져오는 것은 가능하지만(9900 + 100 = 10000), 10001번째 문서부터는 from/size로 접근할 수 없습니다. 이 설정값을 올릴 수는 있지만, 앞서 설명한 메모리 부담이 그대로 남기 때문에 권장되지 않습니다.

from/size로는 10000건까지만 접근할 수 있다는 한계가 분명합니다. 그렇다면 10000건 너머의 결과를 페이지네이션하려면 어떻게 해야 할까요. 이때 사용하는 것이 **search_after** 파라미터입니다.

search_after는 from처럼 "몇 건을 건너뛴다"는 방식을 쓰지 않습니다. 대신, 직전 페이지의 마지막 문서가 가진 정렬 값을 기준점으로 삼아 "이 값 다음에 오는 문서부터 달라"고 요청합니다. 각 샤드는 해당 정렬 값보다 뒤에 있는 문서 중 상위 size건만 올리면 되므로, 몇 번째 페이지든 각 샤드의 부담이 일정합니다. index.max_result_window 제한도 적용되지 않습니다.

search_after를 사용하려면 검색 결과에 정렬 기준이 있어야 합니다. 첫 번째 요청에서 sort를 지정하면, 각 문서의 응답에 sort 배열이 포함됩니다. 이 sort 배열의 값이 바로 search_after에 넘길 기준점입니다.

한 가지 주의할 점이 있습니다. 정렬 값이 동일한 문서가 여러 건일 수 있습니다. 예를 들어 날짜순으로 정렬할 때 같은 날짜의 문서가 100건이면, search_after가 그 날짜 값만으로는 어디까지 이미 가져왔는지를 구분하지 못합니다. 이 문제를 해결하려면 정렬 기준에 문서마다 고유한 값을 가진 필드를 추가로 넣어야 합니다. 이를 **tiebreaker**라고 합니다. _id 필드나, 뒤에서 설명할 PIT 환경의 _shard_doc 필드가 tiebreaker로 쓰입니다.

search_after로 페이지를 넘기는 동안 인덱스에 새 문서가 추가되거나 기존 문서가 삭제되면 어떻게 될까요. Elasticsearch는 주기적으로 인덱스를 refresh하여 새로운 변경 사항을 검색에 반영합니다. 첫 번째 페이지를 가져온 시점과 두 번째 페이지를 가져온 시점 사이에 refresh가 일어나면, 문서 순서가 바뀌거나 같은 문서가 중복되거나 빠지는 일이 생길 수 있습니다. 이 불일치를 방지하는 장치가 **PIT(Point In Time)** API입니다.

PIT는 특정 시점의 인덱스 상태를 고정하는 기능입니다. PIT를 열면 Elasticsearch는 그 순간의 인덱스 스냅샷을 유지합니다. 이후 PIT ID를 검색 요청에 함께 보내면, 검색은 PIT가 열린 시점의 상태만을 대상으로 실행됩니다. 중간에 문서가 추가되거나 삭제되어도 검색 결과에 영향을 주지 않습니다. PIT에는 keep_alive 파라미터로 유지 시간을 지정합니다. 지정한 시간이 지나면 PIT는 자동으로 해제됩니다.

PIT 환경에서 검색할 때는 별도로 tiebreaker 필드를 지정하지 않아도 됩니다. Elasticsearch가 **_shard_doc**이라는 암묵적 tiebreaker를 자동으로 추가합니다. _shard_doc은 각 문서에 대해 샤드 내에서 고유한 값을 가지며, 같은 PIT 안에서는 값이 변하지 않습니다. 따라서 정렬 값이 동일한 문서들도 확실하게 구분됩니다.

search_after와 PIT를 조합하여 페이지를 넘기는 전체 과정은 다음과 같습니다.

```
search_after + PIT 페이지네이션 흐름

1. PIT 열기
   POST my-index/_pit?keep_alive=1m
   → PIT ID 수신

2. 첫 번째 페이지 검색
   POST _search
   {
     "size": 10,
     "query": { ... },
     "sort": [{ "date": "asc" }],
     "pit": { "id": "<pit_id>", "keep_alive": "1m" }
   }
   → 결과의 마지막 문서에서 sort 값 추출
     예: [1706313600000, 12345]  (date 값, _shard_doc 값)

3. 두 번째 페이지 검색
   POST _search
   {
     "size": 10,
     "query": { ... },
     "sort": [{ "date": "asc" }],
     "pit": { "id": "<pit_id>", "keep_alive": "1m" },
     "search_after": [1706313600000, 12345]
   }
   → 다시 마지막 문서의 sort 값 추출

4. 결과가 빈 배열이 될 때까지 3번 반복

5. PIT 삭제
   DELETE _pit
   { "id": "<pit_id>" }
```

위 흐름에서 첫 번째 단계는 PIT를 여는 것입니다. keep_alive를 1m(1분)으로 설정하면, 마지막 요청으로부터 1분 동안 PIT가 유지됩니다. 각 검색 요청에서 keep_alive를 다시 지정하면 타이머가 갱신되므로, 페이지를 넘기는 간격이 1분을 넘지 않으면 PIT가 만료되지 않습니다. 두 번째 단계에서는 sort를 지정하여 첫 페이지를 가져옵니다. PIT를 사용하므로 _shard_doc이 자동으로 tiebreaker에 추가됩니다. 세 번째 단계부터는 직전 페이지 마지막 문서의 sort 값을 search_after에 넣어 다음 페이지를 요청합니다. 결과가 더 이상 없으면 빈 배열이 반환되고, 마지막으로 PIT를 삭제하여 자원을 해제합니다.

구체적인 요청 예를 하나 보겠습니다.

```json
POST _search
{
  "size": 10,
  "query": { "match": { "title": "elasticsearch" } },
  "sort": [{ "date": "asc" }, { "_shard_doc": "asc" }],
  "pit": { "id": "46ToAwMDaWR5BXV1aWQy...", "keep_alive": "1m" },
  "search_after": ["2024-01-15", 12345]
}
```

이 요청을 분해합니다. size가 10이므로 최대 10건을 반환합니다. query의 match는 title 필드에서 "elasticsearch"를 검색합니다. sort는 date 오름차순을 1차 기준, _shard_doc 오름차순을 2차 기준(tiebreaker)으로 지정합니다. pit 객체에는 앞서 받은 PIT ID와 keep_alive를 넣습니다. PIT를 사용하므로 요청 URL에 인덱스 이름을 붙이지 않고 _search만 씁니다. search_after에는 직전 페이지 마지막 문서의 sort 값인 ["2024-01-15", 12345]를 넣습니다. Elasticsearch는 이 정렬 값보다 뒤에 있는 문서부터 10건을 반환합니다.

이제 세 번째 페이지네이션 방식인 **scroll API**를 살펴봅니다. scroll API는 search_after와 마찬가지로 index.max_result_window 제한을 받지 않습니다. 하지만 설계 목적이 다릅니다. search_after + PIT가 사용자 화면에서의 페이지 넘기기에 적합하다면, scroll API는 대량의 문서를 한꺼번에 처리하는 배치 작업에 맞추어져 있습니다. 예를 들어 인덱스의 모든 문서를 다른 인덱스로 복사하는 재인덱싱 작업이나, 전체 데이터를 내보내는 작업에서 scroll이 쓰입니다.

scroll API의 동작 방식은 데이터베이스의 커서(cursor)와 비슷합니다. 처음 검색 요청을 보낼 때 scroll 파라미터를 추가하면, Elasticsearch는 검색 시점의 스냅샷을 만들고 첫 번째 배치와 함께 **_scroll_id**를 반환합니다. 이후에는 이 _scroll_id를 반복적으로 전달하여 다음 배치를 가져옵니다. 결과가 빈 배열이 되면 모든 문서를 가져온 것이므로 스크롤을 종료합니다.

scroll API로 배치를 가져오는 과정을 단계별로 정리합니다.

```
scroll API 배치 처리 흐름

1. 초기 검색 요청
   POST my-index/_search?scroll=1m
   { "size": 1000, "query": { "match_all": {} } }
   → 첫 번째 배치(1000건) + _scroll_id 수신

2. 다음 배치 요청
   POST _search/scroll
   { "scroll": "1m", "scroll_id": "<_scroll_id>" }
   → 두 번째 배치(1000건) + 새 _scroll_id 수신

3. 빈 배열이 반환될 때까지 2번 반복

4. 스크롤 정리
   DELETE _search/scroll
   { "scroll_id": "<_scroll_id>" }
```

첫 번째 단계에서 scroll=1m은 스크롤 컨텍스트를 1분 동안 유지하겠다는 의미입니다. size는 한 번에 가져올 배치 크기입니다. Elasticsearch는 검색 결과의 스냅샷을 만들어 두고, 첫 배치와 함께 _scroll_id를 돌려줍니다. 두 번째 단계에서는 _scroll_id만 넘기면 다음 배치가 반환됩니다. 쿼리를 다시 보낼 필요가 없습니다. 마지막으로 모든 배치를 가져왔으면 clear-scroll API로 스크롤 컨텍스트를 명시적으로 삭제합니다.

scroll API를 사용할 때 정렬 순서가 중요하지 않다면, sort를 _doc으로 지정하면 성능이 향상됩니다. _doc 정렬은 문서가 디스크에 저장된 순서 그대로 읽겠다는 뜻이므로, 정렬 연산을 생략할 수 있습니다.

scroll API에는 주의해야 할 비용이 있습니다. 스크롤 컨텍스트가 열려 있는 동안 Elasticsearch는 해당 시점의 세그먼트를 유지해야 합니다. 1.2.3에서 다루었듯이 Elasticsearch는 시간이 지나면 작은 세그먼트들을 하나로 합치는 세그먼트 병합을 수행합니다. 그런데 스크롤 컨텍스트가 열려 있으면, 병합 대상이 된 이전 세그먼트를 삭제하지 못합니다. 스크롤이 그 세그먼트의 데이터를 아직 참조하고 있기 때문입니다. 스크롤 컨텍스트가 많아질수록 이렇게 유지되는 세그먼트가 쌓이고, 디스크와 파일 핸들 소모가 늘어납니다. Elasticsearch는 동시에 열 수 있는 스크롤 컨텍스트의 기본 상한을 500으로 두고 있습니다. 배치 처리가 끝나면 반드시 clear-scroll API로 컨텍스트를 삭제해야 합니다.

이러한 자원 비용 때문에 Elasticsearch 공식 문서에서도 deep pagination 목적으로 scroll API를 사용하는 것은 권장하지 않습니다. 사용자 화면에서 페이지를 넘기는 용도라면 search_after + PIT 조합이 적합합니다. scroll은 정렬 순서가 중요하지 않고, 전체 데이터를 훑어야 하는 배치 작업에 한정하여 사용하는 것이 바람직합니다.

세 가지 방법의 차이를 한눈에 비교하면 다음과 같습니다.

```
페이지네이션 방식 비교

방식              | 결과 상한         | 적합한 용도              | 자원 비용
-----------------|------------------|------------------------|----------
from/size        | 10000건           | 얕은 페이지네이션         | 낮음
                 | (max_result_window) | (처음 몇 페이지)        |
search_after+PIT | 제한 없음          | 깊은 페이지네이션         | 중간
                 |                   | (일관성 필요)            |
scroll           | 제한 없음          | 배치 처리               | 높음
                 |                   | (재인덱싱, 데이터 추출)   | (세그먼트 유지)
```

정리하면, Elasticsearch의 페이지네이션은 세 가지 방식으로 나뉩니다. from/size는 간단하지만 index.max_result_window(기본 10000) 제한이 있고, 깊은 페이지에서 메모리 낭비가 큽니다. search_after는 직전 페이지의 정렬 값을 기준점으로 삼아 이 제한을 우회하며, PIT와 함께 사용하면 페이지를 넘기는 도중 인덱스 변경으로 인한 불일치도 방지할 수 있습니다. scroll API는 배치 처리에 특화되어 있지만, 스크롤 컨텍스트가 세그먼트를 붙잡아 자원 비용이 높으므로 사용자 대면 페이지네이션에는 적합하지 않습니다.

다음 단원인 3.3.5에서는 지리 검색을 다룹니다.

이 단원을 마치면 from/size, search_after, scroll API의 차이와 각각에 적합한 사용 사례를 설명할 수 있으며, search_after로 deep pagination 문제를 해결할 수 있습니다.
