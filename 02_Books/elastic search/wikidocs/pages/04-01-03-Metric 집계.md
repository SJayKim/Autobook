# 4.1.3 Metric 집계

4.1.2에서 Bucket 집계를 다뤘습니다. Bucket 집계는 문서를 기준에 따라 여러 그룹으로 나누는 역할을 했습니다. terms 집계로 카테고리별 버킷을 만들거나, date_histogram으로 월별 버킷을 만드는 식이었습니다. 그런데 버킷을 만드는 것만으로는 분석이 완결되지 않는 경우가 많습니다. "카테고리별 평균 가격은 얼마인가", "이번 달 주문 총액은 얼마인가"처럼 실제 수치를 계산해야 하는 상황이 더 흔합니다. 이 수치 계산을 담당하는 것이 **Metric 집계**입니다.

Metric 집계는 문서 집합에 수학적 연산을 적용하여 하나의 수치 또는 수치 묶음을 반환합니다. 합계, 평균, 최솟값, 최댓값처럼 익숙한 통계량을 계산하기도 하고, 고유 값의 근사 개수나 백분위 분포처럼 조금 더 복잡한 통계를 산출하기도 합니다. 4.1.1에서 다룬 집계 구문 구조는 그대로 적용됩니다. aggs 키워드 아래에 사용자 정의 이름과 집계 타입을 지정하고, 대상 필드를 넣으면 됩니다. Bucket 집계가 문서를 "나누는" 역할이라면, Metric 집계는 나뉜 그룹 또는 전체 문서에 대해 "계산하는" 역할입니다.

이 단원에서는 Metric 집계의 주요 유형을 하나씩 살펴봅니다. 가장 기본적인 avg, sum, min, max부터 시작하여, 여러 통계량을 한 번에 구하는 stats 집계, 고유 값 수를 세는 cardinality 집계, 백분위 분포를 구하는 percentiles 집계 순서로 진행합니다.

가장 먼저 다룰 것은 **avg 집계**입니다. avg 집계는 숫자 필드 값의 평균을 계산합니다. 온라인 쇼핑몰에서 상품의 평균 가격을 구한다고 가정합니다.

```json
{
  "size": 0,
  "aggs": {
    "average_price": {
      "avg": {
        "field": "price"
      }
    }
  }
}
```

이 요청의 각 부분을 풀어 설명합니다. "average_price"는 4.1.1에서 다룬 사용자 정의 집계 이름입니다. "avg"가 집계 타입이며, price 필드의 값을 모아 평균을 계산하라는 뜻입니다. "size": 0은 검색 결과 문서(hits)를 생략하고 집계 결과만 받겠다는 설정입니다. 응답은 다음과 같은 형태입니다.

```json
{
  "aggregations": {
    "average_price": {
      "value": 24500.75
    }
  }
}
```

aggregations 객체 안에 요청 시 지정한 "average_price"라는 이름이 키로 나타나고, value에 평균 계산 결과가 담깁니다. 대상 문서 중 price 필드가 없거나 null인 문서는 계산에서 자동으로 제외됩니다. 모든 문서에 해당 필드가 없으면 value는 null로 반환됩니다.

다음은 **sum 집계**입니다. sum 집계는 숫자 필드 값의 합계를 구합니다. 전체 매출 총액을 구하는 예시입니다.

```json
{
  "size": 0,
  "aggs": {
    "total_revenue": {
      "sum": {
        "field": "price"
      }
    }
  }
}
```

구조는 avg 집계와 동일합니다. 집계 타입만 "sum"으로 바뀌었습니다. 응답의 value에 price 필드 값의 합계가 담깁니다. avg와 마찬가지로, 필드가 없거나 null인 문서는 합산에서 제외됩니다.

다음은 **min 집계**와 **max 집계**입니다. min은 필드 값 중 최솟값을, max는 최댓값을 반환합니다. 두 집계를 하나의 요청에 함께 넣을 수 있습니다.

```json
{
  "size": 0,
  "aggs": {
    "cheapest": {
      "min": { "field": "price" }
    },
    "most_expensive": {
      "max": { "field": "price" }
    }
  }
}
```

"cheapest"라는 이름의 min 집계가 가장 낮은 가격을, "most_expensive"라는 이름의 max 집계가 가장 높은 가격을 반환합니다. 4.1.1에서 설명했듯이 한 요청에 여러 집계를 넣으면, 응답의 aggregations 객체에 각 이름이 키로 나타나 결과를 구분할 수 있습니다.

지금까지 avg, sum, min, max 네 가지를 살펴봤습니다. 이 네 가지는 구조가 동일합니다. 집계 타입을 바꾸고, 대상 필드를 지정하면 됩니다. 그런데 실무에서는 평균, 합계, 최솟값, 최댓값을 동시에 알고 싶은 경우가 많습니다. 이때마다 네 개의 집계를 따로 정의하면 요청이 길어집니다. Elasticsearch는 이 문제를 해결하는 집계를 제공합니다.

**stats 집계**는 min, max, sum, avg, count 다섯 가지 통계량을 한 번의 집계로 반환합니다. 요청은 다음과 같습니다.

```json
{
  "size": 0,
  "aggs": {
    "price_stats": {
      "stats": {
        "field": "price"
      }
    }
  }
}
```

집계 타입이 "stats"이고, 대상 필드를 지정하는 것은 다른 Metric 집계와 동일합니다. 응답은 다음과 같은 형태입니다.

```json
{
  "aggregations": {
    "price_stats": {
      "count": 500,
      "min": 1200.0,
      "max": 89000.0,
      "avg": 24500.75,
      "sum": 12250375.0
    }
  }
}
```

value 하나가 아니라 다섯 개의 필드가 동시에 반환됩니다. count는 계산에 포함된 문서 수입니다. 필드가 없거나 null인 문서는 count에 포함되지 않습니다. min, max, avg, sum은 각각 앞에서 다룬 개별 집계와 동일한 결과를 냅니다. 네 가지 집계를 따로 실행하는 것보다 요청이 간결해지고, Elasticsearch 내부에서도 한 번의 순회로 다섯 가지를 계산하므로 효율적입니다.

stats 집계가 기본 통계량 다섯 가지를 반환한다면, **extended_stats 집계**는 여기에 추가 통계량을 더합니다. extended_stats는 stats가 반환하는 count, min, max, avg, sum에 더하여 분산(variance), 표준편차(std_deviation), 표준편차 경계(std_deviation_bounds) 등을 함께 반환합니다. 요청 구조는 stats와 동일하며 집계 타입만 "extended_stats"로 바꾸면 됩니다.

```json
{
  "size": 0,
  "aggs": {
    "price_extended": {
      "extended_stats": {
        "field": "price"
      }
    }
  }
}
```

분산은 각 값이 평균에서 얼마나 떨어져 있는지를 나타내는 수치이고, 표준편차는 분산의 제곱근입니다. std_deviation_bounds는 평균에서 표준편차의 2배를 더하고 뺀 상한(upper)과 하한(lower)을 제공합니다. 데이터가 정규 분포를 따른다면 대략 95%의 값이 이 범위 안에 들어갑니다. 이상치를 탐지하거나 데이터의 퍼짐 정도를 파악할 때 유용합니다.

다음으로 **value_count 집계**를 다룹니다. value_count 집계는 특정 필드에 값이 존재하는 문서의 수를 셉니다. stats 집계의 count와 비슷하지만, 단독으로 사용할 수 있다는 점이 다릅니다.

```json
{
  "size": 0,
  "aggs": {
    "price_count": {
      "value_count": {
        "field": "price"
      }
    }
  }
}
```

이 요청은 price 필드에 값이 있는 문서가 몇 건인지를 셉니다. 필드가 없거나 null인 문서는 제외됩니다. value_count가 세는 것은 고유한 값의 수가 아니라, 값이 존재하는 문서의 수입니다. 같은 값이 100건의 문서에 들어 있으면 100으로 셉니다. 고유한 값의 수를 세고 싶다면, 다음에 다룰 cardinality 집계를 사용해야 합니다.

**cardinality 집계**는 필드에 존재하는 고유한 값의 근사 개수를 반환합니다. "이 인덱스에 몇 명의 사용자가 있는가", "카테고리가 몇 종류인가" 같은 질문에 답할 때 사용합니다. 관계형 데이터베이스의 COUNT(DISTINCT field)와 유사한 역할입니다.

```json
{
  "size": 0,
  "aggs": {
    "unique_users": {
      "cardinality": {
        "field": "user_id"
      }
    }
  }
}
```

이 요청은 user_id 필드의 고유한 값이 몇 개인지를 반환합니다. 응답의 value에 고유 값의 수가 담깁니다. 여기서 "근사 개수"라고 표현한 이유를 설명합니다.

Elasticsearch는 분산 시스템이고, 대량의 데이터를 처리합니다. 고유 값의 정확한 수를 구하려면 모든 값을 메모리에 올려야 합니다. user_id처럼 수백만 종의 고유 값이 있는 필드라면, 모든 값을 메모리에 유지하는 것은 비용이 매우 큽니다. 이 문제를 해결하기 위해 Elasticsearch는 **HyperLogLog++** 알고리즘을 사용합니다.

HyperLogLog++는 고유 값의 수를 적은 메모리로 추정하는 확률적 자료 구조입니다. 핵심 아이디어는 다음과 같습니다. 각 값을 해시 함수에 넣으면 이진수 문자열이 나옵니다. 이 이진수에서 앞에 0이 연속으로 몇 개 나오는지를 관찰합니다. 0이 많이 연속할수록 다양한 해시 값이 존재한다는 의미이므로, 고유 값의 수가 많다고 추정합니다. 실제로는 여러 개의 레지스터로 나누어 관찰 결과를 기록하고, 이를 결합하여 추정치를 산출합니다. 이 방식 덕분에 수백만 개의 고유 값이 있어도 수 KB의 메모리만으로 추정이 가능합니다.

이 동작을 단계별로 정리합니다.

```
cardinality 집계의 HyperLogLog++ 동작

1. 해시 변환
   - 각 문서의 user_id 값을 해시 함수에 입력
   - 고정 길이의 이진수 출력 생성

2. 레지스터 분배
   - 해시 값의 앞부분 비트로 레지스터 번호 결정
   - 나머지 비트에서 선행 0의 개수를 세어 해당 레지스터에 기록
   - 각 레지스터는 관찰된 최대 선행 0 개수만 유지

3. 추정치 산출
   - 모든 레지스터의 값을 결합하여 고유 값 수를 추정
   - 조화 평균 기반 수식으로 편향을 보정

4. 결과 반환
   - 추정된 고유 값 수를 value로 반환
   - 소수점 없이 정수로 표시
```

추정이므로 오차가 존재합니다. 기본 설정에서 일반적인 오차율은 약 5% 이내입니다. 정확도를 높이고 싶으면 **precision_threshold** 파라미터를 조정합니다.

```json
{
  "size": 0,
  "aggs": {
    "unique_users": {
      "cardinality": {
        "field": "user_id",
        "precision_threshold": 10000
      }
    }
  }
}
```

precision_threshold는 이 값 이하의 고유 값 수에 대해서는 거의 정확한 결과를 보장한다는 의미입니다. 기본값은 3000이고, 최대 40000까지 설정할 수 있습니다. 값을 높이면 정확도가 올라가지만, 그만큼 더 많은 메모리를 사용합니다. 고유 값 수가 precision_threshold를 크게 넘어가면 오차가 점차 커집니다. 정확한 고유 값 수가 반드시 필요한 경우가 아니라면, 기본값으로도 충분히 쓸 만한 근사치를 얻을 수 있습니다.

다음은 **percentiles 집계**입니다. 평균이나 합계는 데이터의 전체적인 경향을 보여 주지만, 분포의 모양을 알려 주지는 않습니다. 예를 들어 API 응답 시간의 평균이 200ms라고 하더라도, 대부분의 요청이 50ms 안에 끝나고 극소수가 5000ms 이상 걸리는 경우와, 모든 요청이 150~250ms 사이에 고르게 분포하는 경우는 전혀 다른 상황입니다. 이런 분포를 파악할 때 백분위를 사용합니다.

**백분위**(percentile)란, 데이터를 크기 순으로 정렬했을 때 특정 비율 아래에 위치하는 값을 뜻합니다. 예를 들어 95번째 백분위가 800ms라는 것은, 전체 요청 중 95%가 800ms 이하의 응답 시간을 보였다는 의미입니다. 나머지 5%는 800ms를 초과했다는 뜻이기도 합니다.

```json
{
  "size": 0,
  "aggs": {
    "response_time_percentiles": {
      "percentiles": {
        "field": "response_time"
      }
    }
  }
}
```

이 요청은 response_time 필드의 백분위 분포를 계산합니다. 별도로 백분위 값을 지정하지 않으면, 기본적으로 1, 5, 25, 50, 75, 95, 99번째 백분위를 반환합니다.

```json
{
  "aggregations": {
    "response_time_percentiles": {
      "values": {
        "1.0": 15.2,
        "5.0": 22.8,
        "25.0": 48.5,
        "50.0": 102.3,
        "75.0": 305.7,
        "95.0": 812.4,
        "99.0": 2450.1
      }
    }
  }
}
```

이 응답에서 "50.0": 102.3은 전체 요청의 절반이 102.3ms 이하였다는 뜻입니다. 50번째 백분위는 중앙값(median)이라고도 부릅니다. "95.0": 812.4는 95%의 요청이 812.4ms 이하였다는 뜻입니다. 서비스 수준 목표(SLO)에서 "P95 응답 시간 1초 이내"와 같은 기준을 설정할 때 이 값을 활용합니다. "99.0": 2450.1은 최상위 1%의 요청에서 2450ms 이상 걸렸음을 보여 줍니다.

특정 백분위만 보고 싶다면 percents 파라미터로 지정합니다.

```json
{
  "size": 0,
  "aggs": {
    "response_time_percentiles": {
      "percentiles": {
        "field": "response_time",
        "percents": [50, 90, 95, 99]
      }
    }
  }
}
```

이 요청은 50, 90, 95, 99번째 백분위만 계산하여 반환합니다. 필요한 백분위만 지정하면 불필요한 계산을 줄일 수 있습니다.

percentiles 집계도 cardinality 집계처럼 근사 알고리즘을 사용합니다. Elasticsearch는 내부적으로 **TDigest**라는 자료 구조를 이용하여 백분위를 추정합니다. TDigest는 극단값(1%, 99% 등 분포의 양 끝)에서 더 높은 정확도를 보이고, 중간값(50% 부근)에서는 상대적으로 정확도가 낮습니다. 실무에서 관심이 집중되는 P95, P99 같은 극단 백분위에서 정확도가 높은 것이 이 알고리즘의 장점입니다.

다음으로 **top_hits 집계**를 다룹니다. top_hits 집계는 다른 Metric 집계와 성격이 다릅니다. 수치를 계산하는 대신, 각 버킷에서 상위 N개의 문서 원본을 반환합니다. Bucket 집계의 하위 집계로 넣어서, 그룹별 대표 문서를 확인하고 싶을 때 사용합니다. 예를 들어 "카테고리별 최신 상품 3개"를 구한다고 가정합니다.

```json
{
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": { "field": "category", "size": 5 },
      "aggs": {
        "top_products": {
          "top_hits": {
            "size": 3,
            "sort": [{ "created_at": { "order": "desc" } }],
            "_source": ["name", "price", "created_at"]
          }
        }
      }
    }
  }
}
```

terms 집계로 카테고리별 버킷을 만들고, 각 버킷 안에서 top_hits 하위 집계를 실행합니다. size는 각 버킷에서 가져올 문서 수입니다. sort는 문서를 어떤 기준으로 정렬할지 지정합니다. 여기서는 created_at 필드를 내림차순으로 정렬하여 가장 최근 문서가 먼저 오도록 했습니다. _source는 반환할 필드 목록입니다. 전체 문서 대신 필요한 필드만 지정하면 응답 크기가 줄어듭니다. 이 세 가지 파라미터(size, sort, _source)가 top_hits 집계에서 가장 자주 사용하는 설정입니다.

응답에는 각 카테고리 버킷 안에 hits 객체가 포함되고, 그 안에 실제 문서가 배열로 담깁니다. 카테고리가 "의류"인 버킷이라면, 의류 카테고리에서 가장 최근에 등록된 상품 3건의 name, price, created_at 값을 볼 수 있습니다.

마지막으로 **geo_centroid 집계**를 다룹니다. geo_centroid 집계는 geo_point 필드를 가진 문서 집합의 가중 중심점(centroid)을 계산합니다. 가중 중심점이란, 모든 좌표의 위도 평균과 경도 평균으로 구한 하나의 대표 지점입니다. 지도 시각화에서 여러 데이터 포인트를 하나의 클러스터 마커로 표시할 때 유용합니다.

```json
{
  "size": 0,
  "aggs": {
    "center_of_stores": {
      "geo_centroid": {
        "field": "location"
      }
    }
  }
}
```

이 요청은 location 필드(geo_point 타입)의 중심점을 계산합니다. 응답은 다음과 같은 형태입니다.

```json
{
  "aggregations": {
    "center_of_stores": {
      "location": {
        "lat": 37.4979,
        "lon": 127.0276
      },
      "count": 42
    }
  }
}
```

location 안에 중심점의 위도(lat)와 경도(lon)가 담기고, count는 계산에 포함된 문서 수입니다. 이 집계를 Bucket 집계와 결합하면, 예를 들어 지역별 버킷을 만들고 각 지역의 매장 중심점을 계산하여 지도에 클러스터 마커를 배치할 수 있습니다.

지금까지 다룬 Metric 집계를 정리합니다.

```
Metric 집계 요약

집계 타입          반환 값                    특징
-----------       ----------------------    ----------------------------------
avg               평균                       단일 수치 반환
sum               합계                       단일 수치 반환
min               최솟값                     단일 수치 반환
max               최댓값                     단일 수치 반환
stats             min/max/sum/avg/count      다섯 가지를 한 번에 반환
extended_stats    stats + 분산/표준편차       추가 통계량 포함
value_count       값이 존재하는 문서 수        고유 값 수가 아닌 문서 수
cardinality       고유 값의 근사 수            HyperLogLog++ 알고리즘
percentiles       백분위 분포                 TDigest 알고리즘, 극단값에서 고정확
top_hits          버킷 내 대표 문서 원본       수치가 아닌 문서 반환
geo_centroid      지리적 중심점(위도/경도)     geo_point 필드 대상
```

Metric 집계는 단독으로 사용할 수도 있고, 4.1.2에서 다룬 Bucket 집계와 결합할 수도 있습니다. 단독으로 사용하면 전체 문서(또는 query로 걸러진 문서)에 대한 통계를 구합니다. Bucket 집계의 하위 집계로 넣으면, 각 버킷별로 통계를 구합니다. 예를 들어 terms 집계로 카테고리별 버킷을 만들고, 그 안에 stats 하위 집계를 넣으면 카테고리별 가격 통계를 한 번에 얻을 수 있습니다.

```json
{
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": { "field": "category", "size": 10 },
      "aggs": {
        "price_stats": {
          "stats": { "field": "price" }
        }
      }
    }
  }
}
```

이 요청의 실행 흐름을 정리합니다.

```
Bucket + Metric 결합 실행 흐름

1. terms 집계 실행
   - category 필드의 고유 값마다 버킷 생성
   - 예: "의류" 버킷, "전자기기" 버킷, "식품" 버킷 ...

2. 각 버킷 안에서 stats 하위 집계 실행
   - "의류" 버킷 -> 의류 상품의 count, min, max, avg, sum
   - "전자기기" 버킷 -> 전자기기 상품의 count, min, max, avg, sum
   - "식품" 버킷 -> 식품 상품의 count, min, max, avg, sum

3. 응답
   - 각 버킷에 doc_count와 price_stats 객체 포함
   - price_stats 안에 다섯 가지 통계량이 모두 담김
```

정리하면, Metric 집계는 문서 집합에 수학적 연산을 적용하여 수치를 반환하는 집계입니다. avg, sum, min, max는 단일 통계량을 구하고, stats와 extended_stats는 여러 통계량을 한 번에 반환합니다. value_count는 값이 존재하는 문서 수를 세고, cardinality는 HyperLogLog++ 알고리즘으로 고유 값의 근사 수를 구합니다. percentiles는 TDigest 알고리즘으로 백분위 분포를 산출합니다. Bucket 집계와 결합하면 그룹별 통계 분석이 가능합니다.

다음 단원인 4.2.1에서는 중첩 집계를 다룹니다.

이 단원을 마치면 avg, sum, min, max, stats, cardinality 집계로 수치를 계산할 수 있습니다.
