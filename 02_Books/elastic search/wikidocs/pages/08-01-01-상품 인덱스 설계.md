# 8.1.1 상품 인덱스 설계

전자상거래 사이트를 운영한다고 가정합니다. 수만 건의 상품 데이터가 있고, 사용자는 검색창에 "무선 이어폰"을 입력하여 원하는 상품을 찾습니다. 검색 결과에는 상품 이름, 설명, 가격, 재고, 카테고리 같은 정보가 함께 표시되어야 합니다. 이 정보를 Elasticsearch에 저장하려면 인덱스의 매핑을 설계해야 합니다. 매핑이란 각 필드의 이름과 타입, 분석기 설정을 정의하는 것입니다. 매핑을 어떻게 설계하느냐에 따라 검색 품질, 정렬 성능, 집계 가능 여부가 달라집니다.

이 단원에서는 전자상거래 상품 검색에 적합한 인덱스 매핑을 처음부터 설계합니다. 2.2.4에서 다룬 멀티필드 패턴, 2.3.4에서 다룬 Nori 한국어 분석기, 2.2.3에서 다룬 nested 타입과 dense_vector 타입이 기반이 됩니다.

상품 데이터를 Elasticsearch에 저장하려면 먼저 어떤 필드가 필요한지 정해야 합니다. 전자상거래에서 상품 한 건이 가져야 하는 기본 정보를 나열하면, 상품 이름, 상품 설명, 카테고리, 가격, 재고 수량이 있습니다. 여기에 상품 옵션(색상, 사이즈 등)과 상품 이미지의 벡터 임베딩까지 포함하면 실무에서 자주 사용하는 구조가 됩니다.

각 필드에 어떤 타입을 배정해야 하는지 하나씩 살펴봅니다.

**상품 이름**은 사용자가 검색창에 입력하는 주된 대상입니다. "무선 블루투스 이어폰"이라는 상품 이름이 있을 때, 사용자가 "무선 이어폰"만 입력해도 이 상품이 검색 결과에 나와야 합니다. 이를 위해서는 text 타입이 필요합니다. text 타입은 분석기를 적용하여 "무선", "블루투스", "이어폰" 같은 개별 토큰으로 분리하고 역색인에 등록하기 때문입니다.

그런데 상품 이름으로 가나다순 정렬이나 이름별 집계도 수행해야 합니다. text 타입은 분석기를 거치면서 원문이 토큰으로 분해되므로 정렬과 집계에 부적합합니다. 이 문제를 해결하는 것이 2.2.4에서 다룬 **멀티필드 패턴**입니다. 원본 필드는 text 타입으로 전문 검색을 지원하고, 서브필드는 keyword 타입으로 원문을 그대로 저장하여 정렬과 집계를 지원합니다.

**상품 설명** 역시 전문 검색의 대상이므로 text 타입을 사용합니다. 상품 설명은 보통 상품 이름보다 길어서 정렬이나 집계에 사용할 일이 드뭅니다. 따라서 keyword 서브필드를 추가하지 않아도 됩니다.

**카테고리**는 "전자기기 > 이어폰 > 무선 이어폰"처럼 정해진 값 중 하나를 갖는 필드입니다. 사용자가 카테고리를 선택하여 필터링하거나, 카테고리별 상품 수를 집계하는 용도로 사용합니다. 이런 필드에는 **keyword 타입**이 적합합니다. keyword 타입은 값을 분석하지 않고 원문 그대로 저장하므로, 정확 일치 필터링과 집계에 효율적입니다.

**가격**은 숫자 값이므로 숫자 타입을 사용합니다. 2.2.2에서 다룬 것처럼, Elasticsearch는 integer, long, float, double 등의 숫자 타입을 제공합니다. 가격은 소수점이 필요할 수 있으므로 **scaled_float** 타입을 사용하는 것이 일반적입니다. scaled_float은 scaling_factor를 지정하면 내부적으로 정수로 변환하여 저장합니다. scaling_factor를 100으로 설정하면 12,900원은 내부적으로 1,290,000으로 저장됩니다. 이 방식은 부동소수점 연산의 오차를 피하면서 범위 검색과 정렬을 정확하게 수행할 수 있습니다.

**재고 수량**은 정수이므로 **integer** 타입이 적합합니다. "재고 있음" 상품만 필터링하는 범위 쿼리(stock > 0)를 자주 사용하게 됩니다.

여기까지의 필드를 매핑으로 표현하면 다음과 같습니다.

```json
PUT /products
{
  "mappings": {
    "properties": {
      "name": {
        "type": "text",
        "analyzer": "nori_product",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      },
      "description": {
        "type": "text",
        "analyzer": "nori_product"
      },
      "category": {
        "type": "keyword"
      },
      "price": {
        "type": "scaled_float",
        "scaling_factor": 100
      },
      "stock": {
        "type": "integer"
      }
    }
  }
}
```

name 필드는 text 타입에 멀티필드로 keyword 서브필드를 추가했습니다. analyzer에 "nori_product"를 지정했는데, 이 분석기는 아직 정의하지 않았습니다. 잠시 뒤에 한국어 상품명 분석기를 직접 만들겠습니다. description은 text 타입만 사용합니다. category는 keyword, price는 scaled_float, stock은 integer입니다.

이제 상품 옵션을 표현하는 방법을 살펴봅니다. 옷을 판매하는 쇼핑몰이라면, 한 상품에 여러 옵션(색상과 사이즈의 조합)이 존재합니다. 예를 들어 "맨투맨 티셔츠"라는 상품에 "블랙 / M / 재고 30", "화이트 / L / 재고 15" 같은 옵션이 여러 개 붙을 수 있습니다.

이 옵션 데이터를 일반 object 타입의 배열로 저장하면 문제가 생깁니다. 2.2.3에서 설명한 것처럼, object 배열은 내부적으로 각 필드의 값을 하나의 배열로 합쳐 버립니다. 색상은 ["블랙", "화이트"]로, 사이즈는 ["M", "L"]로 합쳐지면서, "화이트 M"이라는 조합이 존재하지 않는데도 검색에 매칭될 수 있습니다.

이 문제를 해결하는 것이 **nested 타입**입니다. nested로 선언하면 배열의 각 객체가 내부적으로 별도의 숨겨진 문서로 저장되어, 필드 값의 짝이 보존됩니다. "블랙 M"과 "화이트 L"이 각각 독립된 단위로 저장되므로, "화이트 M"으로 검색하면 매칭되지 않습니다.

```json
"options": {
  "type": "nested",
  "properties": {
    "color": { "type": "keyword" },
    "size": { "type": "keyword" },
    "option_stock": { "type": "integer" },
    "option_price": { "type": "scaled_float", "scaling_factor": 100 }
  }
}
```

options 필드를 nested로 선언하고, 그 안에 color, size, option_stock, option_price를 정의합니다. color와 size는 정해진 값 중 하나이므로 keyword 타입입니다. 검색 시에는 nested 쿼리를 사용하여 "color가 블랙이고 size가 M인 옵션이 있는 상품"을 정확하게 찾을 수 있습니다.

다음으로, 상품 이미지의 벡터 임베딩을 저장하는 방법을 살펴봅니다. 최근 전자상거래에서는 사용자가 사진을 찍어 올리면 비슷한 상품을 찾아주는 이미지 검색 기능이 확산되고 있습니다. 이 기능을 구현하려면 각 상품 이미지를 머신러닝 모델에 넣어 벡터 임베딩으로 변환하고, 이 벡터를 Elasticsearch에 저장해야 합니다.

2.2.3에서 다룬 **dense_vector** 타입이 이 용도에 쓰입니다. 사용하는 이미지 임베딩 모델이 512차원 벡터를 출력한다면, dims를 512로 설정합니다.

```json
"image_embedding": {
  "type": "dense_vector",
  "dims": 512,
  "index": true,
  "similarity": "cosine"
}
```

index를 true로 설정하면 kNN 검색에 사용할 수 있는 HNSW 인덱스가 생성됩니다. similarity에 cosine을 지정하면 벡터 간의 코사인 유사도로 가까운 벡터를 찾습니다. 사용자가 업로드한 이미지를 같은 모델로 벡터화한 뒤 kNN 쿼리를 실행하면, 시각적으로 유사한 상품을 찾을 수 있습니다.

이제 한국어 상품명을 제대로 검색하기 위한 분석기를 설계합니다. 앞서 매핑에서 analyzer로 "nori_product"를 지정했습니다. 2.3.4에서 다룬 Nori 분석기를 기반으로, 상품 검색에 특화된 커스텀 분석기를 구성합니다.

한국어 상품명 검색에서 흔히 겪는 문제가 있습니다. "맨투맨 티셔츠"를 검색했는데 "맨투맨티셔츠"(공백 없음)에 매칭되지 않는 경우, "에어팟 프로"를 검색했는데 "AirPods Pro"에 매칭되지 않는 경우, "노트북"을 검색했는데 "노트 북"(공백 삽입)이 빠지는 경우가 있습니다. 이런 문제를 최소화하려면 분석기의 각 단계를 상품 검색 특성에 맞게 구성해야 합니다.

분석기는 캐릭터 필터, 토크나이저, 토큰 필터의 세 단계로 구성됩니다. 2.3.1에서 다룬 이 파이프라인을 상품 검색용으로 구성하면 다음과 같습니다.

```json
PUT /products
{
  "settings": {
    "analysis": {
      "char_filter": {
        "normalize_whitespace": {
          "type": "pattern_replace",
          "pattern": "\\s+",
          "replacement": " "
        }
      },
      "tokenizer": {
        "nori_mixed": {
          "type": "nori_tokenizer",
          "decompound_mode": "mixed"
        }
      },
      "filter": {
        "product_synonym": {
          "type": "synonym",
          "synonyms": [
            "노트북, 랩톱",
            "이어폰, 이어버드",
            "핸드폰, 휴대폰, 스마트폰"
          ]
        }
      },
      "analyzer": {
        "nori_product": {
          "type": "custom",
          "char_filter": ["normalize_whitespace"],
          "tokenizer": "nori_mixed",
          "filter": [
            "nori_readingform",
            "lowercase",
            "product_synonym"
          ]
        }
      }
    }
  }
}
```

이 분석기의 각 구성 요소가 무엇을 하는지 풀어 봅니다.

char_filter의 normalize_whitespace는 연속된 공백을 하나의 공백으로 정리합니다. 상품 데이터를 외부 시스템에서 가져올 때 불규칙한 공백이 섞여 들어오는 경우가 흔하기 때문입니다.

tokenizer에 nori_tokenizer를 사용하되, decompound_mode를 "mixed"로 설정합니다. 2.3.4에서 다룬 것처럼, mixed 모드는 복합어의 원형과 분해된 형태소를 동시에 역색인에 등록합니다. "삼성전자"를 분석하면 "삼성전자", "삼성", "전자" 세 가지 토큰이 모두 생성됩니다. 이 덕분에 "삼성"으로 검색해도, "삼성전자"로 검색해도 해당 상품이 매칭됩니다.

filter에서는 세 가지 토큰 필터를 순서대로 적용합니다. nori_readingform은 한자를 한글 읽기 형태로 변환합니다. lowercase는 영문 대문자를 소문자로 통일합니다. "AirPods"가 "airpods"로 바뀌어, 대소문자와 관계없이 검색이 가능해집니다. product_synonym은 **동의어 필터**입니다.

**동의어 사전**은 같은 의미를 가진 단어들을 하나의 그룹으로 묶어 놓은 목록입니다. "노트북"과 "랩톱"은 같은 제품을 가리키지만, 사용자마다 다른 단어를 사용합니다. 동의어 필터를 적용하면 "랩톱"으로 검색해도 "노트북"이 포함된 상품이 결과에 나타납니다.

위 예시에서는 synonyms 배열에 동의어 매핑을 직접 작성했습니다. 상품 수가 늘어나고 동의어 목록이 길어지면, 별도의 동의어 파일을 만들어 synonyms_path로 지정하는 방식이 더 관리하기 편합니다. 파일은 Elasticsearch 설정 디렉터리 아래에 두고, 각 줄에 쉼표로 구분된 동의어 그룹을 적습니다.

전체 매핑을 한 번에 정리하면 다음과 같은 구조가 됩니다.

```
products 인덱스
  |
  |-- name (text + keyword 멀티필드, nori_product 분석기)
  |-- description (text, nori_product 분석기)
  |-- category (keyword)
  |-- price (scaled_float, scaling_factor: 100)
  |-- stock (integer)
  |-- options (nested)
  |     |-- color (keyword)
  |     |-- size (keyword)
  |     |-- option_stock (integer)
  |     |-- option_price (scaled_float)
  |-- image_embedding (dense_vector, dims: 512)
```

정리하면, 전자상거래 상품 인덱스는 검색 대상 필드에 text+keyword 멀티필드 패턴을 적용하고, 필터와 집계 대상 필드에는 keyword 또는 숫자 타입을 배정하며, 상품 옵션처럼 필드 짝을 보존해야 하는 데이터에는 nested 타입을, 이미지 유사 검색을 위해서는 dense_vector 타입을 사용합니다. 한국어 상품명 분석기는 Nori 토크나이저에 동의어 필터를 결합하여, 형태소 분리와 유사어 매칭을 동시에 처리합니다.

다음 단원인 8.1.2에서는 이 인덱스 위에서 자동완성, 오타교정, 동의어 검색을 결합한 검색 서비스를 구현하는 방법을 다룹니다.

이 단원을 마치면 전자상거래 상품 검색을 위한 인덱스 매핑과 분석기를 설계할 수 있습니다.
