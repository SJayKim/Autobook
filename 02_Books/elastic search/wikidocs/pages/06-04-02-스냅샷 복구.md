# 6.4.2 스냅샷과 복구

운영 중인 Elasticsearch 클러스터에서 데이터 유실은 언제든 발생할 수 있습니다. 하드웨어 장애, 사람의 실수로 인한 인덱스 삭제, 소프트웨어 버그 등 원인은 다양합니다. 레플리카 샤드는 노드 장애에 대응하지만, 인덱스 자체를 실수로 삭제하면 레플리카도 함께 사라집니다. 클러스터 외부에 데이터를 백업해 두어야 이런 상황에서 복구할 수 있습니다.

Elasticsearch에서 클러스터를 백업하는 유일하게 공식 지원되는 방법이 **스냅샷(snapshot)**입니다. 파일시스템 수준에서 Elasticsearch 데이터 디렉토리를 복사하는 것은 유효한 백업이 아닙니다. 데이터 파일이 지속적으로 변경되는 상태에서 복사하면 파일 간 불일치가 발생하여, 복원 시 데이터 손상이나 유실이 생길 수 있습니다.

스냅샷은 인덱스 데이터뿐 아니라 클러스터 상태(인덱스 설정, 매핑, 인덱스 템플릿), ILM 정책, 인제스트 파이프라인, 저장된 스크립트, Kibana 대시보드와 시각화 설정 등을 함께 보존합니다.

스냅샷을 생성하려면 먼저 **스냅샷 저장소(snapshot repository)**를 등록해야 합니다. 저장소는 클러스터 외부의 스토리지입니다. 자주 사용하는 저장소 유형은 AWS S3, Google Cloud Storage(GCS), Azure Blob Storage, 그리고 공유 파일시스템(NFS 등)입니다.

S3 저장소를 등록하는 예입니다.

```json
PUT _snapshot/my_backup_repo
{
  "type": "s3",
  "settings": {
    "bucket": "my-es-backups",
    "region": "ap-northeast-2",
    "base_path": "snapshots"
  }
}
```

type은 저장소 종류이고, bucket은 S3 버킷 이름입니다. base_path는 버킷 안에서 스냅샷 파일이 저장될 하위 경로입니다. S3 접근에 필요한 인증 정보는 사전에 Elasticsearch 키스토어에 등록해 두어야 합니다.

공유 파일시스템 저장소를 사용하려면, elasticsearch.yml에 path.repo 설정을 추가하여 허용할 디렉토리를 지정합니다.

```yaml
path.repo: ["/mnt/backups"]
```

이 설정이 있어야 fs 타입의 저장소를 등록할 수 있습니다.

```json
PUT _snapshot/local_backup
{
  "type": "fs",
  "settings": {
    "location": "/mnt/backups/es-snapshots"
  }
}
```

저장소가 등록되면 스냅샷을 생성합니다.

```json
PUT _snapshot/my_backup_repo/snapshot_20240115
{
  "indices": "logs-*,app-data-*",
  "ignore_unavailable": true,
  "include_global_state": true
}
```

snapshot_20240115는 스냅샷 이름입니다. indices에 백업할 인덱스 패턴을 지정합니다. ignore_unavailable이 true이면 존재하지 않는 인덱스가 패턴에 포함되어도 오류 없이 건너뜁니다. include_global_state가 true이면 클러스터 상태(템플릿, ILM 정책 등)도 함께 백업합니다.

스냅샷의 핵심 특성은 **증분(incremental)** 방식이라는 점입니다. Lucene의 세그먼트는 한번 생성되면 변경되지 않는 불변 파일입니다. 첫 번째 스냅샷은 모든 세그먼트를 저장하지만, 이후 스냅샷은 이전 스냅샷 이후 새로 생성된 세그먼트만 추가로 저장합니다. 따라서 두 번째 이후의 스냅샷은 빠르게 완료되고 저장 공간도 적게 사용합니다.

```
스냅샷 1: [세그먼트 A] [세그먼트 B] [세그먼트 C]  ← 전체 저장
스냅샷 2: [세그먼트 D]                            ← 새로 생긴 것만 저장
스냅샷 3: [세그먼트 E] [세그먼트 F]                ← 새로 생긴 것만 저장
```

각 스냅샷은 논리적으로 독립적입니다. 스냅샷 1을 삭제해도, 스냅샷 2와 3은 정상적으로 복원할 수 있습니다. 삭제 시에는 해당 스냅샷에서만 독점적으로 사용하는 세그먼트만 제거됩니다.

스냅샷에서 데이터를 복구하려면 _restore API를 사용합니다.

```json
POST _snapshot/my_backup_repo/snapshot_20240115/_restore
{
  "indices": "logs-2024.01.*",
  "ignore_unavailable": true,
  "rename_pattern": "logs-(.+)",
  "rename_replacement": "restored-logs-$1"
}
```

indices 필드로 복구할 인덱스를 지정합니다. 전체 스냅샷이 아닌 특정 인덱스만 선택적으로 복구할 수 있습니다. 이를 **부분 복구**라고 합니다. rename_pattern과 rename_replacement를 사용하면 복구되는 인덱스의 이름을 변경할 수 있습니다. 이 예에서는 logs-2024.01.15가 restored-logs-2024.01.15로 복구됩니다. 기존 인덱스와 이름이 충돌하는 것을 방지할 때 유용합니다.

수동으로 스냅샷을 생성하는 대신, **SLM(Snapshot Lifecycle Management)**을 사용하면 스냅샷 생성을 자동화할 수 있습니다. SLM 정책은 스냅샷을 언제, 어떤 인덱스를, 어떤 저장소에 생성할지를 정의합니다.

```json
PUT _slm/policy/nightly_backup
{
  "schedule": "0 30 1 * * ?",
  "name": "<nightly-snap-{now/d}>",
  "repository": "my_backup_repo",
  "config": {
    "indices": ["*"],
    "include_global_state": true
  },
  "retention": {
    "expire_after": "30d",
    "min_count": 5,
    "max_count": 50
  }
}
```

schedule은 cron 형식으로, 이 예는 매일 새벽 1시 30분에 실행합니다. name은 스냅샷 이름 패턴으로, {now/d}는 현재 날짜로 치환됩니다. retention 설정은 오래된 스냅샷의 자동 삭제 정책입니다. expire_after는 30일이 지난 스냅샷을 삭제 대상으로 표시합니다. min_count는 최소 보관 수로, 만료되었더라도 5개 미만이면 삭제하지 않습니다. max_count는 최대 보관 수입니다.

스냅샷 상태는 _snapshot API로 모니터링합니다.

```json
GET _snapshot/my_backup_repo/snapshot_20240115/_status
```

진행 중인 스냅샷의 완료 비율, 처리된 파일 수와 바이트 수를 확인할 수 있습니다.

재해 복구(DR) 시나리오에서는 스냅샷이 유일한 복구 수단입니다. 데이터 센터 전체가 장애를 겪는 경우, 다른 리전의 S3 버킷에 저장된 스냅샷으로 새 클러스터에서 데이터를 복원합니다. 이를 위해 스냅샷 저장소를 원격 리전에 두고, 정기적으로 SLM으로 백업하며, 복원 절차를 주기적으로 테스트하는 것이 중요합니다.

정리하면, 스냅샷은 Elasticsearch 클러스터의 유일한 공식 백업 방법입니다. S3, GCS, Azure, 공유 파일시스템에 저장소를 등록하고, 증분 방식으로 스냅샷을 생성합니다. _restore API로 전체 또는 부분 복구를 수행하며, SLM 정책으로 생성과 보존을 자동화합니다.

다음 단원인 6.4.3에서는 클러스터 모니터링을 다룹니다. 클러스터 상태와 성능 지표를 실시간으로 감시하여 문제를 조기에 감지하는 방법을 알아봅니다.

이 단원을 마치면 스냅샷 저장소를 등록하고 스냅샷을 생성하고 복구하는 절차를 수행할 수 있으며, SLM으로 자동 스냅샷을 스케줄링할 수 있습니다.
