# 5.3.3 멀티 리전 시나리오

5.3.2에서 Sniff 모드와 Proxy 모드의 차이를 살펴보았습니다. 원격 클러스터에 연결하는 방법을 알게 되었으니, 이제 여러 지역(리전)에 클러스터를 배치하고 운영하는 실제 시나리오를 다룰 차례입니다. 글로벌 서비스를 운영하는 조직이라면, 데이터를 한 곳에 모아 두기보다 사용자에게 가까운 지역에 분산 배치하는 것이 일반적입니다. 이 단원에서는 리전별 클러스터 분리의 이유, CCS와 CCR을 활용한 멀티 리전 전략, 그리고 비용과 지연 시간을 고려한 설계 원칙을 다룹니다.

먼저 **리전별 클러스터를 분리하는 이유**를 짚습니다. 두 가지 주요 요인이 있습니다.

첫 번째는 **레이턴시**(지연 시간)입니다. 사용자가 서울에 있고 클러스터가 미국 동부에 있으면, 네트워크 왕복 시간만 200밀리초 이상이 소요될 수 있습니다. 검색 요청 한 건에 여러 번의 왕복이 필요할 수 있으므로, 체감 응답 시간은 더 길어집니다. 사용자에게 가까운 리전에 클러스터를 두면 네트워크 지연을 줄일 수 있습니다.

두 번째는 **규제**(데이터 주권)입니다. 많은 국가에서 자국 사용자의 개인정보를 해당 국가 내에 저장하도록 법률로 규정하고 있습니다. 유럽의 GDPR이 대표적입니다. 이런 규제를 준수하려면 지역별로 독립된 클러스터를 운영하고, 해당 지역의 데이터를 그 클러스터에만 저장해야 합니다.

이 두 요인 때문에 글로벌 서비스는 서울, 도쿄, 프랑크푸르트, 버지니아 등 주요 리전에 각각 클러스터를 운영하는 구조를 택합니다.

리전별 클러스터가 분리되면, 전체 데이터를 한 번에 검색하는 방법이 필요합니다. 여기서 5.3.1에서 다룬 CCS가 핵심 역할을 합니다. **CCS 기반 중앙 집계 패턴**은 하나의 중앙 클러스터(또는 특정 리전의 클러스터)에서 CCS 쿼리를 실행하여, 모든 리전의 데이터를 한 번에 조회하는 방식입니다.

```
CCS 기반 중앙 집계 패턴

중앙 클러스터 (서울)
    |
    +--CCS--> 도쿄 클러스터
    |
    +--CCS--> 프랑크푸르트 클러스터
    |
    +--CCS--> 버지니아 클러스터

쿼리 예시:
GET logs-*,tokyo:logs-*,frankfurt:logs-*,virginia:logs-*/_search
```

이 패턴의 장점은 데이터가 각 리전에 머물면서도, 관리자나 분석가가 전역 뷰를 얻을 수 있다는 것입니다. 각 리전의 사용자는 자기 리전의 클러스터에 직접 검색하여 빠른 응답을 받고, 전역 분석이 필요할 때만 CCS를 사용합니다.

이 패턴의 한계도 있습니다. CCS는 검색 시점에 원격 클러스터에 실시간으로 요청을 보내므로, 원격 클러스터가 내려가 있으면 해당 리전의 데이터를 조회할 수 없습니다. 5.3.1에서 다룬 skip_unavailable을 true로 설정하면 다른 리전의 결과만이라도 받을 수 있지만, 전체 데이터의 완전성은 보장되지 않습니다.

이 한계를 보완하는 기능이 **CCR(Cross-Cluster Replication)**입니다. CCS가 '원격 클러스터에 가서 검색'하는 방식이라면, CCR은 '원격 클러스터의 데이터를 로컬로 복제'하는 방식입니다.

CCR은 **leader-follower 인덱스** 구조로 동작합니다. leader 인덱스는 원본 데이터가 있는 인덱스이고, follower 인덱스는 leader의 데이터를 실시간으로 복제받는 인덱스입니다. leader에 문서가 추가되거나 수정되면, 그 변경 사항이 follower에 자동으로 반영됩니다.

```
CCR 동작 구조

서울 클러스터               도쿄 클러스터
+------------------+       +------------------+
| logs (leader)    | ----> | logs (follower)  |
| 원본 데이터       | 복제  | 복제본 데이터     |
+------------------+       +------------------+

도쿄의 follower는 서울 leader의 변경을 비동기로 수신
```

follower 인덱스는 읽기 전용입니다. 쓰기는 오직 leader 인덱스에서만 수행됩니다. follower는 leader의 변경 이력을 비동기로 가져오므로, 약간의 복제 지연(replication lag)이 있습니다. 지연 시간은 네트워크 상태와 변경 빈도에 따라 달라지지만, 일반적으로 밀리초에서 수 초 수준입니다.

CCS와 CCR의 차이를 정리합니다.

```
CCS vs CCR 비교

+------------------+-------------------------------+-------------------------------+
|                  | CCS                           | CCR                           |
+------------------+-------------------------------+-------------------------------+
| 데이터 위치      | 원본 리전에만 존재             | 여러 리전에 복제본 존재        |
+------------------+-------------------------------+-------------------------------+
| 동작 방식        | 검색 시 원격 클러스터에 요청    | 사전에 데이터를 복제해 둠      |
+------------------+-------------------------------+-------------------------------+
| 검색 지연        | 원격 네트워크 레이턴시 포함     | 로컬 검색 (레이턴시 최소)     |
+------------------+-------------------------------+-------------------------------+
| 원격 장애 시     | 해당 리전 데이터 조회 불가     | 로컬 복제본으로 계속 조회     |
+------------------+-------------------------------+-------------------------------+
| 저장 비용        | 추가 저장 없음                | 복제본만큼 저장 비용 증가     |
+------------------+-------------------------------+-------------------------------+
| 적합한 용도      | 간헐적 전역 검색, 비용 절약   | 빠른 로컬 읽기, 재해 복구    |
+------------------+-------------------------------+-------------------------------+
```

CCR의 대표적인 활용 사례가 **Active-Passive DR(재해 복구)** 구성입니다. Active 클러스터(서울)가 모든 쓰기와 읽기를 처리하고, Passive 클러스터(도쿄)가 CCR로 데이터를 복제받아 대기합니다. Active 클러스터에 재해가 발생하면, Passive 클러스터의 follower 인덱스를 일반 인덱스로 전환(unfollow)하여 쓰기를 활성화하고, 트래픽을 도쿄로 전환합니다. 데이터 유실 범위는 마지막 복제 시점 이후의 변경분으로 제한됩니다.

**멀티 리전 인덱싱 전략**도 설계가 필요합니다. 두 가지 접근법이 있습니다. 첫째, 각 리전의 사용자가 해당 리전의 클러스터에 직접 인덱싱합니다. 서울 사용자의 데이터는 서울 클러스터에, 도쿄 사용자의 데이터는 도쿄 클러스터에 저장됩니다. 전역 검색이 필요하면 CCS를 사용합니다. 둘째, 하나의 Primary 클러스터에 모든 쓰기를 집중하고, CCR로 다른 리전에 복제합니다. 쓰기 경로가 단순해지지만, Primary 클러스터까지의 쓰기 레이턴시가 존재합니다.

어떤 전략을 선택하느냐는 쓰기와 읽기의 비율, 데이터 규제 요건, 허용 가능한 레이턴시에 따라 달라집니다.

마지막으로 **CCS 지연 시간 SLA 고려사항**을 짚습니다. 멀티 리전 환경에서 CCS를 사용할 때, 전체 검색 응답 시간은 가장 느린 원격 클러스터의 응답 시간에 의해 결정됩니다. 서울에서 도쿄 클러스터(레이턴시 약 30ms)와 버지니아 클러스터(레이턴시 약 200ms)를 동시에 CCS로 검색하면, 전체 응답 시간은 200ms 이상이 됩니다.

따라서 SLA(서비스 수준 합의)를 설계할 때, CCS 경유 응답 시간은 가장 먼 리전까지의 네트워크 왕복 시간을 포함해야 합니다. 엄격한 응답 시간 SLA가 필요한 서비스에서는, CCS 대신 CCR로 데이터를 로컬에 복제해 두고 로컬 검색만 사용하는 것이 더 적합할 수 있습니다. 비용과 응답 시간 사이의 균형을 잡는 것이 멀티 리전 아키텍처 설계의 핵심입니다.

정리하면, 멀티 리전 환경에서는 레이턴시와 규제를 이유로 리전별 클러스터를 분리합니다. CCS로 전역 검색을 구현하되, 원격 클러스터 장애 시 일부 데이터가 누락될 수 있습니다. CCR은 leader-follower 구조로 데이터를 리전 간 복제하여, 빠른 로컬 읽기와 재해 복구를 가능하게 합니다. CCS는 저장 비용이 적고 CCR은 응답 속도가 빠르므로, 서비스 요구사항에 맞게 조합합니다.

다음 단원인 6.1.1에서는 Phase 6 '운영 및 관리'의 첫 번째 주제인 ILM 정책을 다룹니다. Phase 5에서 클러스터와 샤드의 분산 구조를 모두 살펴보았으므로, Phase 6에서는 이 구조 위에서 인덱스 수명주기를 자동화하고, 데이터 수집 파이프라인을 구성하며, 보안과 운영 관리를 본격적으로 다룹니다.

이 단원을 마치면 멀티 리전 데이터 아키텍처에서 CCS 활용 패턴을 설계할 수 있습니다.
