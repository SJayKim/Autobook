# 6.3.2 검색 성능 최적화

인덱싱이 빠르게 이루어지더라도, 검색 응답이 느리면 사용자 경험이 나빠집니다. 검색 성능 최적화는 느린 쿼리를 찾아내고, 캐시를 활용하며, 불필요한 작업을 줄이는 과정입니다. 이 단원에서는 검색 성능 병목을 진단하고 개선하는 구체적인 방법을 다룹니다.

느린 쿼리를 찾는 첫 번째 도구는 **슬로우 로그(slowlog)**입니다. 슬로우 로그는 지정한 임곗값보다 오래 걸린 검색 요청을 로그 파일에 기록하는 기능입니다. 인덱스 단위로 설정합니다.

```json
PUT /my-index/_settings
{
  "index.search.slowlog.threshold.query.warn": "10s",
  "index.search.slowlog.threshold.query.info": "5s",
  "index.search.slowlog.threshold.query.debug": "2s",
  "index.search.slowlog.threshold.query.trace": "500ms",
  "index.search.slowlog.threshold.fetch.warn": "1s",
  "index.search.slowlog.threshold.fetch.info": "800ms"
}
```

threshold.query는 쿼리 단계의 임곗값이고, threshold.fetch는 결과를 가져오는(fetch) 단계의 임곗값입니다. warn, info, debug, trace는 로그 레벨에 따라 다른 임곗값을 설정합니다. 예를 들어 쿼리 단계가 10초를 넘으면 warn 레벨로, 5초를 넘으면 info 레벨로 기록됩니다. 운영 초기에는 느슨한 임곗값으로 시작하여 점진적으로 줄여 가면, 어떤 쿼리가 병목인지 파악할 수 있습니다.

슬로우 로그에 기록된 쿼리의 실행 과정을 상세히 분석하려면 **profile API**를 사용합니다. 검색 요청에 profile: true를 추가하면, 각 샤드에서 쿼리가 어떤 단계를 거쳤고 각 단계에 얼마나 걸렸는지를 반환합니다.

```json
GET /my-index/_search
{
  "profile": true,
  "query": {
    "match": { "message": "error" }
  }
}
```

응답에는 각 샤드의 쿼리 실행 계획이 트리 구조로 나타납니다. 어떤 쿼리 컴포넌트가 시간을 많이 소모하는지 확인하여 최적화 대상을 좁힐 수 있습니다. Kibana의 Search Profiler 화면에서 이 결과를 시각적으로 분석할 수도 있습니다.

검색 성능을 크게 높이는 방법 중 하나가 캐시 활용입니다. Elasticsearch는 세 가지 주요 캐시를 사용합니다. 이 중 검색 최적화와 직접 관련된 두 가지를 살펴보겠습니다.

**노드 쿼리 캐시(node query cache)**는 filter context에서 실행된 필터 결과를 비트셋 형태로 저장합니다. 3.2.1에서 다룬 것처럼 filter context는 점수를 계산하지 않고 조건 일치 여부만 판단합니다. 한 번 실행된 필터 결과는 비트셋으로 캐싱되어, 같은 필터가 다시 실행될 때 재평가 없이 캐시에서 바로 반환됩니다. 노드당 하나의 쿼리 캐시가 존재하며, 기본적으로 힙의 10%를 사용합니다.

이를 활용하려면 자주 반복되는 조건을 filter context에 배치하는 것이 중요합니다. bool 쿼리에서 filter 절이나 must_not 절에 넣은 조건은 filter context로 실행되어 캐시 대상이 됩니다. 같은 조건이라도 must 절에 넣으면 query context로 실행되어 캐시가 적용되지 않습니다.

**샤드 레벨 요청 캐시(request cache)**는 집계 결과를 샤드 단위로 캐싱합니다. size: 0으로 설정한 검색 요청(결과 문서 없이 집계만 수행)에 기본적으로 적용됩니다. Kibana 대시보드의 시각화 패널이 이 패턴을 많이 사용하므로, 대시보드 새로고침 시 동일한 집계를 다시 계산하지 않아도 됩니다. 요청 캐시는 샤드에 새 데이터가 인덱싱되어 리프레시가 발생하면 무효화됩니다. 기본 크기는 힙의 1%이며, 변경이 필요하면 elasticsearch.yml에서 indices.requests.cache.size를 조정합니다.

날짜 기반 쿼리에서 캐시 적중률을 높이는 기법이 있습니다. now를 사용한 범위 쿼리는 매 요청마다 현재 시각이 달라지므로 캐시가 적중하지 않습니다. 날짜를 반올림하면 같은 분 또는 같은 시간 내의 요청이 동일한 캐시를 공유합니다. 예를 들어 now-1h 대신 now-1h/m을 사용하면 분 단위로 반올림되어 같은 분 안의 요청이 캐시를 공유합니다.

검색 대상 샤드 수를 줄이면 검색 속도가 빨라집니다. **routing** 파라미터를 사용하면 특정 값을 기준으로 문서가 저장되는 샤드를 결정할 수 있습니다. 검색 시에도 같은 routing 값을 지정하면, Elasticsearch가 해당 샤드만 검색하므로 나머지 샤드를 건너뛸 수 있습니다. 예를 들어 사용자 ID로 routing을 설정하면, 특정 사용자의 데이터를 검색할 때 하나의 샤드만 탐색합니다.

**_source 필드 필터링**도 검색 성능에 영향을 줍니다. _source에는 원본 문서 전체가 저장되어 있는데, 모든 필드가 필요하지 않다면 반환할 필드를 제한합니다.

```json
GET /my-index/_search
{
  "_source": ["title", "timestamp"],
  "query": { "match_all": {} }
}
```

필요한 필드만 반환하면 네트워크 전송량이 줄고, 역직렬화 비용도 줄어듭니다.

매핑 설정에서 **doc_values**와 **store** 옵션의 선택도 성능에 영향을 줍니다. doc_values는 집계와 정렬에 사용되는 디스크 기반 컬럼 스토어입니다. keyword와 숫자 필드에 기본적으로 활성화되어 있으며, 힙 메모리를 사용하지 않습니다. store: true는 개별 필드의 값을 별도로 저장하여, _source 전체를 읽지 않고도 특정 필드를 가져올 수 있게 합니다. 대부분의 경우 doc_values가 효율적이지만, 문서가 매우 크고 극소수 필드만 자주 조회한다면 store를 고려할 수 있습니다.

**fielddata 캐시**는 text 필드에 대해 정렬이나 집계를 수행할 때 사용됩니다. text 필드에는 doc_values가 없으므로, 집계 시 역색인에서 값을 읽어 힙 메모리에 올립니다. 이 과정은 메모리를 많이 소모하고 느립니다. 가능하면 text 필드 대신 keyword 필드를 사용하거나, 멀티필드를 활용하여 keyword 버전의 필드로 집계를 수행하는 것이 바람직합니다. fielddata 캐시 크기는 indices.fielddata.cache.size로 제한할 수 있으며, 기본값은 무제한입니다.

정리하면, 검색 성능 최적화는 슬로우 로그로 느린 쿼리를 찾고, profile API로 실행 계획을 분석하는 것에서 시작합니다. 자주 사용하는 필터를 filter context에 배치하여 쿼리 캐시를 활용하고, 날짜를 반올림하여 요청 캐시 적중률을 높입니다. routing으로 검색 대상 샤드를 줄이고, _source 필터링으로 불필요한 데이터 전송을 방지합니다. text 필드 대신 keyword 필드를 사용하여 fielddata 부담을 줄입니다.

다음 단원인 6.3.3에서는 JVM과 파일시스템 최적화를 다룹니다. 힙 크기 설정, GC 선택, OS 레벨 튜닝 등 인프라 수준의 최적화 방법을 알아봅니다.

이 단원을 마치면 슬로우 로그를 분석하여 검색 성능 병목을 찾고 최적화할 수 있습니다.
