# 1.2.4 NRT 검색과 Translog

1.2.3에서 문서가 인메모리 버퍼를 거쳐 세그먼트로 만들어지는 과정을 살펴봤습니다. 리프레시가 1초 간격으로 실행되어 새 세그먼트를 파일시스템 캐시에 기록하고, 플러시가 그 세그먼트를 디스크에 영구 저장한다는 것까지 확인했습니다. 그런데 리프레시와 플러시 사이에는 시간 차이가 있습니다. 리프레시는 1초마다, 플러시는 30분마다 또는 특정 조건에서 실행됩니다. 이 시간 차이 동안 데이터는 파일시스템 캐시에만 존재합니다. 전원이 갑자기 꺼지면 캐시에만 있던 데이터는 사라집니다. 이 단원에서는 "빠른 검색 노출"과 "데이터 안전" 사이의 간극을 어떻게 메우는지를 다룹니다.

Elasticsearch에 문서를 색인하면 해당 문서가 검색 결과에 즉시 나타나지 않습니다. 정확히 말하면, 색인 직후가 아니라 다음 리프레시가 실행된 뒤에 나타납니다. 기본 리프레시 주기가 1초이므로 색인 후 최대 약 1초 뒤에 검색이 가능해집니다. 완전한 실시간은 아니지만 사람이 체감하기에 거의 실시간입니다. 이 특성을 **NRT(Near Real-Time)** 검색이라 부릅니다.

NRT가 가능한 이유를 이해하려면 리프레시가 정확히 무엇을 하는지 알아야 합니다. 리프레시는 인메모리 버퍼에 쌓인 문서 데이터를 새 세그먼트 파일로 변환하여 파일시스템 캐시에 기록합니다. 그리고 곧바로 그 파일을 열어 읽을 수 있는 상태로 만듭니다. 여기서 핵심은 fsync를 호출하지 않는다는 점입니다. fsync는 운영체제의 파일시스템 캐시 내용을 물리 디스크에 강제로 내려쓰는 시스템 콜입니다. fsync를 실행하면 데이터가 실제 디스크 표면에 기록되므로 전원이 꺼져도 데이터가 보존됩니다. 그 대신 디스크 장치가 기록을 완료할 때까지 기다려야 하므로 시간이 오래 걸립니다.

리프레시는 이 비용이 큰 fsync를 건너뜁니다. 파일시스템 캐시에 쓰는 것만으로 끝내기 때문에 매우 빠릅니다. 파일시스템 캐시에 올라간 파일은 아직 물리 디스크에 영구 기록된 것은 아니지만, 운영체제 입장에서는 다른 파일과 마찬가지로 읽을 수 있습니다. Lucene은 이 시점에 새 세그먼트를 열어서 검색 대상에 포함시킵니다. 이것이 디스크에 영구 기록하지 않고도 검색이 가능한 원리입니다.

색인부터 검색 가능 시점까지의 흐름을 단계별로 정리하면 다음과 같습니다.

```
NRT 검색 흐름

[1] 문서 도착
     |
     v
[2] 인메모리 버퍼(IndexWriter buffer)에 저장
     |
     v
[3] 리프레시 실행 (기본 1초 간격)
     : 버퍼 내용 --> 새 세그먼트 파일로 기록
     : 파일시스템 캐시에만 존재 (fsync 없음)
     : 새 세그먼트를 열어 검색 대상에 추가
     |
     v
[4] 이 시점부터 검색 가능 (NRT)
```

리프레시 주기는 **index.refresh_interval** 설정으로 조절합니다. 기본값은 "1s", 즉 1초입니다. 이 값을 늘리면 리프레시 사이의 간격이 길어져서 새 문서가 검색에 노출되기까지의 지연 시간이 늘어납니다. 반대로 그 간격 동안 인메모리 버퍼에 더 많은 문서가 쌓이므로 한 번의 리프레시로 더 큰 세그먼트가 만들어집니다. 세그먼트 생성 빈도가 줄어들고, 1.2.3에서 설명한 세그먼트 병합 부담도 함께 줄어듭니다. 결과적으로 색인 처리량이 향상됩니다.

대량의 데이터를 한꺼번에 색인하는 작업에서는 이 트레이드오프를 적극적으로 활용합니다. 예를 들어 초기 데이터 로딩 시 refresh_interval을 "-1"로 설정하면 리프레시를 완전히 비활성화합니다. 리프레시가 멈추면 세그먼트가 생기지 않으므로 병합 작업도 일어나지 않습니다. 색인 처리량이 크게 높아집니다. 데이터 로딩이 끝난 뒤 다시 "1s"로 복원하면 정상적인 NRT 검색이 재개됩니다. 한 가지 더, refresh_interval을 "-1"까지 가지 않고 "30s"(30초)로 늘리는 것만으로도 색인 속도가 의미 있게 향상됩니다.

```
PUT /my_index/_settings
{
  "index.refresh_interval": "-1"
}
```

위 요청은 my_index의 리프레시를 비활성화합니다. index.refresh_interval 필드에 "-1"을 넣으면 자동 리프레시가 중단됩니다. 대량 색인이 끝나면 같은 API로 "1s"를 다시 지정합니다.

Elasticsearch에는 리프레시를 추가로 절약하는 내부 최적화도 있습니다. 지난 30초 동안 검색 요청을 한 건도 받지 않은 인덱스는 자동 리프레시가 비활성화됩니다. 쓰기만 있고 읽기가 없는 인덱스에서 불필요한 세그먼트 생성을 줄이기 위한 장치입니다. 해당 인덱스에 다시 검색 요청이 들어오면 리프레시가 자동으로 재개됩니다.

여기까지가 NRT 검색의 원리입니다. 리프레시가 fsync 없이 파일시스템 캐시에만 세그먼트를 기록하기 때문에 빠르게 검색이 가능해집니다. 그런데 바로 이 "fsync 없이"라는 지점에서 데이터 안전 문제가 생깁니다. 파일시스템 캐시는 메모리의 일부입니다. 서버 전원이 꺼지거나 운영체제가 비정상 종료되면 캐시에만 존재하는 데이터는 유실됩니다. 리프레시 이후부터 플러시(fsync) 이전까지의 시간 동안 색인된 문서는 이 위험에 노출됩니다.

관계형 데이터베이스에서도 같은 문제가 있습니다. PostgreSQL이나 MySQL InnoDB는 데이터를 디스크에 영구 기록하기 전에 변경 내용을 먼저 별도의 순차 로그 파일에 기록합니다. 이 로그를 **WAL(Write-Ahead Log)** 이라 부릅니다. "쓰기 전에 미리 로그에 남긴다"는 뜻입니다. 장애가 발생하면 이 로그를 처음부터 재생하여 유실된 데이터를 복구합니다. Elasticsearch에도 같은 역할을 하는 장치가 있습니다. 바로 **Translog**(트랜잭션 로그)입니다.

Translog의 동작 원리를 이해하려면 문서가 색인될 때 어떤 일이 동시에 벌어지는지를 알아야 합니다. 문서가 Elasticsearch 샤드에 도착하면 두 곳에 동시에 기록됩니다. 하나는 인메모리 버퍼(IndexWriter buffer)이고, 다른 하나가 Translog입니다. 인메모리 버퍼는 텍스트 분석과 역색인 데이터 준비를 위한 공간이고, Translog는 원본 문서 데이터를 순차적으로 쌓는 로그 파일입니다.

리프레시가 실행되면 인메모리 버퍼의 내용이 새 세그먼트로 변환되어 파일시스템 캐시에 올라갑니다. 그런데 이때 Translog는 초기화되지 않습니다. Translog는 플러시가 실행될 때까지 계속 쌓입니다. 리프레시 후 새 문서가 검색 가능해지더라도, 해당 문서의 원본 데이터는 Translog에 그대로 남아 있습니다. 이것이 핵심입니다. 파일시스템 캐시에만 있는 세그먼트가 장애로 사라져도 Translog에 원본이 남아 있으므로 복구할 수 있습니다.

노드가 비정상 종료된 뒤 재시작되면 복구 과정이 시작됩니다. Elasticsearch는 마지막 Lucene commit 포인트를 확인합니다. Lucene commit 포인트란 마지막 플러시가 성공적으로 완료된 시점을 가리키는 기록입니다. 이 시점 이후에 Translog에 쌓인 내용을 처음부터 끝까지 재적용(replay)합니다. 각 문서를 다시 인덱싱하는 것과 같은 과정입니다. 재적용이 끝나면 플러시 이후에 색인되었던 문서가 모두 복원됩니다.

```
Translog 기반 복구 흐름

[정상 운영 중]
  문서 색인 --> 인메모리 버퍼 + Translog 동시 기록
  리프레시  --> 세그먼트가 파일시스템 캐시에 생성 (Translog 유지)
  플러시    --> 세그먼트를 디스크에 fsync + Translog 초기화

[장애 발생 후 재시작]
  1. 마지막 Lucene commit 포인트(= 마지막 플러시 시점) 확인
  2. commit 포인트 이후의 Translog 내용을 순차적으로 재적용
  3. 유실된 문서 복구 완료
```

이제 플러시가 정확히 무엇을 하는지 더 자세히 살펴봅니다. **플러시**(flush)는 두 가지 작업을 수행합니다. 첫째, 파일시스템 캐시에만 존재하는 세그먼트들을 fsync로 물리 디스크에 영구 기록합니다. 이 과정이 Lucene commit입니다. 둘째, Translog를 초기화합니다. 디스크에 영구 기록이 완료되었으므로 Translog에 쌓아 둔 복구용 데이터가 더 이상 필요 없기 때문입니다.

플러시는 자동으로 실행됩니다. 두 가지 조건 중 하나라도 충족되면 백그라운드에서 실행됩니다. 첫 번째 조건은 시간입니다. 기본적으로 30분마다 플러시가 실행됩니다. 두 번째 조건은 Translog 크기입니다. Translog가 index.translog.flush_threshold_size에 지정된 크기를 초과하면 플러시가 실행됩니다. 이 설정의 기본값은 10GB입니다.

Translog 크기와 복구 시간 사이에는 트레이드오프가 있습니다. Translog가 크다는 것은 마지막 플러시 이후에 많은 문서가 쌓였다는 뜻입니다. 장애가 발생하면 이 Translog 전체를 재적용해야 합니다. Translog가 클수록 재적용할 양이 많아지고 복구 시간이 길어집니다.

flush_threshold_size를 줄이면 플러시가 더 자주 실행됩니다. Translog가 작은 상태를 유지하므로 장애 시 복구가 빠릅니다. 그 대신 fsync 횟수가 늘어나 디스크 I/O 부하가 증가합니다. 반대로 flush_threshold_size를 키우면 플러시 빈도가 낮아져 디스크 I/O가 줄어듭니다. 색인 처리량이 향상됩니다. 그러나 Translog가 커질 수 있으므로 장애 시 복구 시간이 길어집니다.

```
flush_threshold_size 트레이드오프

threshold 작게 설정         threshold 크게 설정
--------------------         --------------------
플러시 자주 실행             플러시 덜 실행
Translog 작게 유지           Translog 크게 유지
복구 빠름                    복구 느림
디스크 I/O 높음              디스크 I/O 낮음
색인 처리량 낮음             색인 처리량 높음
```

마지막으로 Translog의 내구성 수준을 조절하는 **index.translog.durability** 설정을 살펴봅니다. 이 설정은 "클라이언트에게 쓰기 성공 응답을 보내기 전에 Translog를 fsync할 것인가"를 결정합니다.

기본값은 "request"입니다. 이 모드에서는 각 색인/삭제/수정/bulk 요청이 완료될 때마다 Translog를 fsync합니다. 클라이언트가 성공 응답을 받았다면 해당 데이터는 이미 Translog 파일로 디스크에 기록된 상태입니다. 하드웨어 장애가 발생해도 응답을 받은 모든 쓰기는 보존됩니다.

다른 옵션은 "async"입니다. 이 모드에서는 개별 요청마다 fsync하지 않습니다. 대신 index.translog.sync_interval에 지정된 간격(기본 5초)으로 백그라운드에서 Translog를 fsync합니다. 요청 완료 시점에 fsync를 기다리지 않으므로 응답 속도가 빨라지고 처리량이 향상됩니다. 그러나 마지막 fsync 이후에 장애가 발생하면 그 사이에 들어온 쓰기가 유실될 수 있습니다. 기본 간격이 5초이므로 최대 5초 분량의 데이터가 사라질 수 있다는 뜻입니다.

어떤 모드를 선택할지는 데이터의 성격에 따라 다릅니다. 금융 거래나 주문 데이터처럼 한 건도 유실되면 안 되는 경우에는 기본값인 "request"를 유지합니다. 서버 로그나 이벤트 데이터처럼 일부 유실이 허용되는 경우에는 "async"로 변경하여 색인 처리량을 높일 수 있습니다.

```
PUT /my_index/_settings
{
  "index.translog.durability": "async",
  "index.translog.sync_interval": "5s"
}
```

위 요청은 my_index의 Translog 내구성을 async 모드로 변경합니다. index.translog.durability 필드가 fsync 정책을 지정하고, index.translog.sync_interval 필드가 백그라운드 fsync 주기를 지정합니다. sync_interval의 최소값은 100ms입니다.

지금까지 다룬 내용을 한 곳에 모아 문서가 색인된 후 최종적으로 디스크에 안착하기까지의 전체 흐름을 정리합니다.

```
문서 색인부터 디스크 영구 저장까지의 전체 흐름

문서 도착
  |
  +--> 인메모리 버퍼 (IndexWriter buffer)
  |      : 텍스트 분석, 역색인 데이터 준비
  |
  +--> Translog 기록
         : 원본 문서를 순차 로그에 기록
         : durability=request이면 이 시점에 fsync
  |
  v
리프레시 (기본 1초)
  : 버퍼 --> 새 세그먼트 --> 파일시스템 캐시
  : fsync 없음
  : 검색 가능 (NRT)
  : Translog는 유지됨
  |
  v
플러시 (30분마다 또는 Translog > 10GB)
  : 세그먼트를 디스크에 fsync (Lucene commit)
  : Translog 초기화
  : 이 시점부터 하드웨어 장애에도 안전
```

정리하면, NRT 검색은 리프레시가 fsync 없이 파일시스템 캐시에 세그먼트를 기록함으로써 약 1초 만에 검색을 가능하게 하는 특성입니다. fsync를 생략하기 때문에 빠르지만, 파일시스템 캐시의 데이터는 장애 시 유실될 위험이 있습니다. Translog가 이 위험을 보완합니다. 문서 색인 시 인메모리 버퍼와 Translog에 동시에 기록하고, 장애 후 재시작 시 Translog를 재적용하여 유실된 문서를 복구합니다. 플러시가 세그먼트를 디스크에 영구 기록하고 Translog를 초기화하며, flush_threshold_size로 복구 시간과 I/O 부하 사이의 균형을 조절합니다. index.translog.durability 설정으로 쓰기 요청마다 fsync할지(request), 주기적으로 fsync할지(async)를 선택하여 내구성과 처리량 사이의 균형을 잡습니다.

다음 단원인 1.3.1에서는 JSON 문서 구조를 다룹니다.

이 단원을 마치면 Near Real-Time 검색의 동작 원리를 설명할 수 있고, Translog의 역할과 내구성 보장 방식을 설명할 수 있습니다.
