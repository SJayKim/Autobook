# 7.4.1 RAG 파이프라인 구성

7.3.2에서 BM25와 kNN을 결합하는 하이브리드 검색을 구현했습니다. 하이브리드 검색은 사용자 질문에 관련된 문서를 정확하게 찾아줍니다. 그런데 사용자가 원하는 것이 "관련 문서 목록"이 아니라 "질문에 대한 직접적인 답변"이라면 어떨까요. 예를 들어 "Elasticsearch의 샤드 수를 변경하는 방법은?"이라는 질문에 관련 문서 10개를 보여주는 것보다, 구체적인 답변 문장을 생성하여 보여주는 것이 더 유용합니다. 이것을 가능하게 하는 것이 **RAG**(Retrieval-Augmented Generation)입니다.

RAG는 두 단어의 결합입니다. **Retrieval**(검색)은 질문과 관련된 정보를 데이터 저장소에서 찾는 것이고, **Generation**(생성)은 그 정보를 바탕으로 자연어 답변을 만드는 것입니다. "Augmented"는 검색으로 찾은 정보가 생성 과정을 보강한다는 뜻입니다. 대규모 언어 모델(LLM)이 자체 학습 데이터만으로 답변을 생성하는 대신, 검색으로 찾은 최신 정보를 참고하여 더 정확한 답변을 만들어냅니다.

RAG가 필요한 이유는 LLM의 한계에서 출발합니다. LLM은 학습 시점까지의 정보만 알고 있습니다. 학습 이후에 변경된 내용이나, 기업 내부 문서 같은 비공개 데이터는 알지 못합니다. 또한 학습 데이터에 포함되지 않았거나 빈도가 낮은 정보에 대해서는 사실과 다른 답변을 생성할 수 있습니다. RAG는 검색 단계에서 정확한 정보를 찾아 LLM에 제공함으로써 이런 문제를 완화합니다.

RAG 파이프라인의 전체 구조는 다음과 같습니다.

```
1. 문서 준비 단계 (오프라인)
   원본 문서 → 청킹(chunking) → 임베딩 생성 → Elasticsearch 인덱싱

2. 질의응답 단계 (온라인)
   사용자 질문 → 검색(kNN/하이브리드) → 관련 청크 추출
        │
   관련 청크 + 질문 → LLM 프롬프트 조합 → LLM → 답변 생성
```

첫 번째 단계인 문서 준비에서 가장 중요한 작업이 **청킹**(chunking)입니다. 긴 문서를 그대로 임베딩하면 문제가 생깁니다. 임베딩 모델은 입력 길이에 제한이 있습니다. 7.2.3에서 다뤘듯이 ELSER는 약 512토큰이 상한이고, 대부분의 임베딩 모델도 512~2048토큰 사이의 제한을 가집니다. 또한 긴 문서 전체를 하나의 벡터로 표현하면, 문서 안의 세부적인 의미 차이가 희석됩니다. 그래서 문서를 작은 단위로 나누는 것이 청킹입니다.

청킹 전략은 세 가지가 대표적입니다.

**고정 크기 청킹**은 문서를 일정한 글자 수(또는 토큰 수)로 자릅니다. 예를 들어 500토큰마다 나누는 식입니다. 구현이 간단하고 청크 크기가 균일하다는 장점이 있습니다. 단점은 문장 중간에서 잘릴 수 있다는 것입니다. 이를 보완하기 위해 앞뒤로 50~100토큰씩 겹치는 구간(overlap)을 두는 것이 일반적입니다.

**문장 기반 청킹**은 문장 단위로 나눕니다. 마침표, 물음표, 느낌표 같은 문장 종결 기호를 기준으로 분리하고, 연속된 몇 개의 문장을 하나의 청크로 묶습니다. 문맥이 자연스럽게 유지되는 장점이 있습니다.

**단락 기반 청킹**은 문서의 단락(paragraph)을 기준으로 나눕니다. 빈 줄이나 제목으로 구분되는 단락을 하나의 청크로 사용합니다. 의미 단위가 가장 잘 보존되지만, 단락 크기가 불균일할 수 있습니다.

청킹이 끝나면 각 청크를 임베딩 모델에 넣어 벡터를 생성하고, Elasticsearch에 인덱싱합니다. 7.1.3에서 다룬 인제스트 파이프라인의 inference 프로세서를 사용하면, 청크 텍스트를 인덱싱할 때 자동으로 임베딩이 생성됩니다.

```json
PUT rag-index
{
  "mappings": {
    "properties": {
      "chunk_text": { "type": "text" },
      "chunk_embedding": {
        "type": "dense_vector",
        "dims": 384,
        "index": true,
        "similarity": "cosine"
      },
      "doc_id": { "type": "keyword" },
      "chunk_index": { "type": "integer" }
    }
  }
}
```

chunk_text에 청크 원문을, chunk_embedding에 벡터를 저장합니다. doc_id는 원본 문서를 식별하고, chunk_index는 문서 안에서 이 청크의 순서를 나타냅니다.

질의응답 단계에서는 사용자 질문으로 **컨텍스트 검색**을 수행합니다. 7.3.2에서 다룬 하이브리드 검색을 사용하면 키워드 매칭과 의미 기반 검색을 함께 활용하여 가장 관련성 높은 청크를 찾을 수 있습니다.

```json
GET rag-index/_search
{
  "retriever": {
    "rrf": {
      "retrievers": [
        { "standard": { "query": { "match": { "chunk_text": "샤드 수 변경 방법" } } } },
        { "knn": { "field": "chunk_embedding", "query_vector_builder": {
            "text_embedding": { "model_id": "my-embedding-model", "model_text": "샤드 수 변경 방법" }
          }, "num_candidates": 50 } }
      ],
      "rank_window_size": 20,
      "rank_constant": 60
    }
  },
  "size": 5
}
```

상위 5개 청크를 가져옵니다. 이 청크들이 LLM에 전달할 컨텍스트가 됩니다.

검색된 청크를 **LLM 프롬프트에 삽입**합니다. 프롬프트 구조는 보통 다음과 같습니다.

```
다음 정보를 참고하여 질문에 답변하세요.
정보에 없는 내용은 "알 수 없습니다"라고 답변하세요.

[정보]
청크1: Elasticsearch에서 인덱스의 primary shard 수는 ...
청크2: split API를 사용하면 기존 인덱스의 샤드 수를 ...
청크3: reindex API로 새 인덱스에 데이터를 복사할 때 ...

[질문]
Elasticsearch의 샤드 수를 변경하는 방법은?
```

LLM은 이 프롬프트를 받아 검색된 정보를 종합하여 답변을 생성합니다. 검색 결과에 없는 내용은 답변하지 않도록 지시하는 것이 할루시네이션(사실과 다른 답변 생성)을 줄이는 데 도움이 됩니다.

Elasticsearch를 RAG의 벡터 저장소로 사용할 때, Python 생태계의 프레임워크를 활용하면 구현이 간편해집니다. **LangChain**은 LLM 애플리케이션 개발 프레임워크로, ElasticsearchStore 클래스를 통해 Elasticsearch와 직접 연동합니다. 문서 청킹, 임베딩, 인덱싱, 검색, LLM 호출을 하나의 파이프라인으로 엮을 수 있습니다. **LlamaIndex**도 유사한 기능을 제공하며, Elasticsearch를 벡터 저장소(vector store)로 사용하는 공식 커넥터를 제공합니다.

답변 품질을 평가하려면 생성된 답변이 검색된 컨텍스트에 근거하고 있는지(faithfulness), 원래 질문에 적절히 답하고 있는지(answer relevance)를 확인합니다. 자동 평가 도구로는 RAGAs(RAG Assessment) 프레임워크가 널리 쓰입니다. 이 프레임워크는 faithfulness, answer relevancy, context precision 같은 지표를 자동으로 계산합니다.

정리하면, RAG 파이프라인은 문서 청킹, 임베딩, 인덱싱의 준비 단계와 검색, 프롬프트 조합, LLM 답변 생성의 질의응답 단계로 구성됩니다. Elasticsearch는 벡터 저장소 역할을 하며, 하이브리드 검색으로 관련 청크를 정확하게 찾아냅니다. 청킹 전략은 고정 크기, 문장 기반, 단락 기반 중 데이터 특성에 맞게 선택합니다. LangChain이나 LlamaIndex 같은 프레임워크를 활용하면 전체 파이프라인을 효율적으로 구현할 수 있습니다.

다음 단원인 7.4.2에서는 ML Inference 프로세서를 사용하여 인덱싱 시 자동으로 임베딩을 생성하는 파이프라인을 더 깊이 다룹니다.

이 단원을 마치면 Elasticsearch를 벡터 저장소로 사용하는 RAG 파이프라인을 설계할 수 있습니다.
