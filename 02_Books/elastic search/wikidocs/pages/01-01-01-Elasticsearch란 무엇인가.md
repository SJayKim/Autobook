# 1.1.1 Elasticsearch란 무엇인가

웹 서비스를 운영하다 보면 데이터가 빠르게 쌓입니다. 상품 목록, 사용자 로그, 서버 지표 등 종류도 다양합니다. 데이터가 수만 건일 때는 관계형 데이터베이스의 LIKE 검색으로도 충분하지만, 수천만 건을 넘어서면 검색 속도가 급격히 느려집니다. 동시에 "지난 한 시간 동안 에러가 가장 많이 발생한 서버는 어디인가"처럼 실시간에 가까운 분석도 필요해집니다. 이 두 가지 문제, 즉 대규모 데이터에서의 빠른 검색과 실시간 분석을 하나의 기술로 해결하려는 시도에서 Elasticsearch가 등장했습니다.

**Elasticsearch**는 Apache Lucene을 기반으로 만들어진 분산형 오픈소스 검색 및 분석 엔진입니다. 여기서 Apache Lucene이란 자바로 작성된 전문 검색(full-text search) 라이브러리를 뜻합니다. Lucene은 강력하지만 라이브러리 형태이므로, 이를 직접 애플리케이션에 통합하려면 상당한 코드 작업이 필요합니다. Elasticsearch는 Lucene 위에 분산 처리, HTTP 통신, 클러스터 관리 기능을 얹어서 별도의 복잡한 코드 작성 없이도 대규모 검색 시스템을 구축할 수 있게 만든 제품입니다. 2004년 Shay Banon이 만든 Compass라는 검색 프레임워크가 전신이며, 2010년 2월에 Elasticsearch라는 이름으로 처음 공개되었습니다.

Elasticsearch가 다른 데이터베이스와 구별되는 핵심 특징은 **역색인**(Inverted Index)입니다. 일반적인 데이터베이스는 문서 전체를 순서대로 훑어서 키워드를 찾습니다. 역색인은 반대로 접근합니다. 텍스트를 먼저 단어(토큰) 단위로 쪼갠 뒤, 각 단어가 어떤 문서에 포함되어 있는지를 미리 기록해 둡니다. 책 뒤에 붙는 '찾아보기'와 같은 원리입니다. 검색할 때는 해당 단어의 목록만 조회하면 되므로, 문서가 수억 건이라도 빠르게 결과를 얻을 수 있습니다. 단순히 키워드가 포함되었는지만 판단하는 것이 아니라, 문서와 검색어의 유사도를 점수로 매겨 관련성이 높은 순서대로 결과를 정렬하는 **관련성 점수**(relevance score) 기반 랭킹도 제공합니다.

Elasticsearch는 **검색 엔진**과 **분석 엔진** 두 가지 역할을 동시에 수행합니다. 검색 엔진 역할이란, 방금 설명한 역색인과 관련성 랭킹을 활용하여 사용자가 입력한 질의에 가장 적합한 문서를 빠르게 찾아주는 것입니다. 쇼핑몰에서 상품을 검색하거나, 사내 위키에서 문서를 찾는 상황이 대표적입니다. 분석 엔진 역할은 이와 성격이 다릅니다. 개별 문서를 찾는 것이 아니라 대량의 데이터를 그룹별로 묶고 합산, 평균, 최댓값 등을 계산하는 **집계**(aggregation)를 수행합니다. 예를 들어 "지난 24시간 동안 HTTP 500 에러가 가장 많이 발생한 서비스 상위 5개"를 구하는 것이 집계입니다. 검색과 집계는 쿼리 패턴이 서로 다르지만, Elasticsearch는 하나의 기술 스택 안에서 두 패턴을 모두 처리합니다.

Elasticsearch는 데이터를 **JSON 문서** 형태로 저장합니다. JSON(JavaScript Object Notation)은 키와 값의 쌍으로 데이터를 표현하는 텍스트 형식입니다. 관계형 데이터베이스에서 하나의 행(row)에 해당하는 것이 Elasticsearch에서는 하나의 JSON 문서(document)입니다. 아래는 상품 정보를 담은 문서의 예입니다.

```json
{
  "name": "무선 키보드",
  "price": 45000,
  "category": "주변기기",
  "in_stock": true
}
```

'name' 필드는 문자열, 'price'는 숫자, 'in_stock'은 불리언 값입니다. 이처럼 하나의 문서 안에 다양한 타입의 데이터를 담을 수 있으며, 중첩된 JSON 객체나 배열도 지원합니다. Elasticsearch는 스키마리스(schema-less) 특성을 가지고 있어서, 사전에 테이블 구조를 정의하지 않아도 문서를 바로 저장할 수 있습니다. 첫 번째 문서가 들어오면 각 필드의 타입을 자동으로 추론하여 매핑을 생성하는데, 이를 **동적 매핑**(dynamic mapping)이라 합니다. 더 세밀한 제어가 필요하면 명시적 매핑(explicit mapping)으로 필드 타입을 직접 지정할 수도 있습니다.

문서에는 사용자가 넣은 데이터 외에도 시스템이 관리하는 메타데이터 필드가 포함됩니다. '_id'는 문서의 고유 식별자이고, '_index'는 문서가 속한 인덱스 이름이며, '_source'는 인덱싱 시 전달된 원본 JSON 전체를 그대로 보관하는 필드입니다. 검색 결과를 반환할 때 '_source'에 저장된 원본을 돌려주므로, 별도의 원본 저장소 없이도 문서 내용을 그대로 조회할 수 있습니다.

Elasticsearch의 모든 작업은 **RESTful API**를 통해 이루어집니다. REST(Representational State Transfer)는 HTTP 프로토콜의 표준 메서드(GET, PUT, POST, DELETE)를 사용하여 자원을 다루는 아키텍처 스타일입니다. 특별한 클라이언트 드라이버나 전용 프로토콜 없이, curl 같은 HTTP 도구만 있으면 Elasticsearch와 통신할 수 있습니다. 예를 들어 문서 하나를 저장하는 요청은 다음과 같습니다.

```
PUT /products/_doc/1
{
  "name": "무선 키보드",
  "price": 45000
}
```

'PUT'은 HTTP 메서드이고, '/products/_doc/1'은 'products'라는 인덱스에 ID가 1인 문서를 저장하겠다는 뜻입니다. 본문에는 저장할 JSON 데이터를 넣습니다. ID를 직접 지정하지 않으려면 POST 메서드를 사용하면 되고, 이 경우 Elasticsearch가 자동으로 고유 ID를 부여합니다.

API는 용도별로 나뉩니다. Document API는 개별 문서의 생성, 조회, 수정, 삭제를 담당합니다. Search API는 쿼리와 집계를 실행합니다. Indices API는 인덱스와 매핑을 관리합니다. cat API는 클러스터 상태, 인덱스 목록, 노드 정보 등을 사람이 읽기 쉬운 텍스트 형태로 반환합니다. 이처럼 모든 조작을 HTTP 요청으로 처리하는 구조 덕분에, 프로그래밍 언어에 관계없이 동일한 방식으로 Elasticsearch를 다룰 수 있습니다.

Elasticsearch가 대규모 데이터를 처리할 수 있는 이유는 **수평 확장성**(horizontal scalability)에 있습니다. 수평 확장이란 하나의 서버를 더 강력한 장비로 교체하는 방식(수직 확장)이 아니라, 여러 대의 서버를 묶어 처리 용량을 늘리는 방식입니다. Elasticsearch에서 서버 한 대를 **노드**(node)라 부르고, 노드들의 묶음을 **클러스터**(cluster)라 부릅니다. 데이터가 늘어나면 클러스터에 노드를 추가하는 것만으로 용량을 확장할 수 있습니다.

수평 확장의 핵심 단위는 **샤드**(shard)입니다. 하나의 인덱스는 여러 개의 샤드로 나뉘고, 각 샤드는 서로 다른 노드에 분산 배치됩니다. 이렇게 하면 검색 요청이 들어왔을 때 여러 노드가 동시에 자기 샤드를 검색하므로, 단일 서버보다 훨씬 빠르게 결과를 얻을 수 있습니다.

**고가용성**(high availability)은 일부 노드에 장애가 발생해도 서비스가 중단되지 않는 성질을 뜻합니다. Elasticsearch는 각 샤드의 복제본인 **레플리카 샤드**(replica shard)를 만들어 다른 노드에 배치합니다. 원본인 프라이머리 샤드가 있는 노드에 장애가 발생하면, 레플리카가 자동으로 프라이머리로 승격하여 서비스를 이어갑니다. 프라이머리 샤드와 레플리카 샤드는 항상 서로 다른 노드에 위치하므로, 한 노드가 통째로 중단되어도 데이터가 유실되지 않습니다. 아래 다이어그램은 3개 노드로 구성된 클러스터에서 샤드가 분산 배치되는 모습을 나타냅니다.

```
클러스터 (3 노드)
+----------------+  +----------------+  +----------------+
|    Node A      |  |    Node B      |  |    Node C      |
|                |  |                |  |                |
| [P0] [R1]     |  | [P1] [R2]     |  | [P2] [R0]     |
+----------------+  +----------------+  +----------------+

P = 프라이머리 샤드, R = 레플리카 샤드
숫자 = 샤드 번호
```

P0(프라이머리 샤드 0번)은 Node A에, 그 복제본인 R0은 Node C에 배치되어 있습니다. Node A에 장애가 발생하면 Node C의 R0이 프라이머리로 승격합니다. 인덱스를 생성할 때 프라이머리 샤드 수(number_of_shards)는 고정되지만, 레플리카 수(number_of_replicas)는 운영 중에도 변경할 수 있습니다.

Elasticsearch는 문서를 인덱싱한 뒤 약 1초 이내에 검색 가능한 상태가 됩니다. 이 특성을 **NRT**(Near Real-Time, 거의 실시간) 검색이라 합니다. 이름에서 알 수 있듯이 완전한 실시간은 아닙니다. 그 이유는 내부 동작 방식에 있습니다. 문서가 들어오면 먼저 메모리 버퍼에 쌓이고, 주기적으로 버퍼의 내용을 새로운 Lucene 세그먼트 파일로 기록합니다. 이 세그먼트를 열어서 검색 가능 상태로 만드는 과정을 **refresh**라 하며, 기본 주기가 1초입니다. 따라서 문서를 인덱싱한 시점과 검색에 나타나는 시점 사이에 최대 1초 정도의 간격이 생깁니다.

```
문서 인덱싱 흐름

1. 문서 도착 → 메모리 버퍼에 기록
2. (1초 주기) refresh → 새 세그먼트 생성, 검색 가능
3. (비동기) translog에 기록 → 장애 시 복구용
4. (30분 주기 또는 translog 크기 초과) flush → 디스크에 영구 기록
```

refresh와 별개로, 장애 복구를 위한 장치도 있습니다. 문서가 들어올 때마다 translog(트랜잭션 로그)에도 기록하여, 노드가 갑자기 중단되더라도 translog를 재생해 데이터를 복구합니다. 이후 일정 주기(기본 30분)나 translog 크기가 임계치를 초과하면 flush가 실행되어 데이터를 디스크에 영구적으로 기록합니다.

Elasticsearch는 단독으로 쓰이기도 하지만, 대부분의 실무 환경에서는 **ELK Stack**이라 불리는 구성으로 함께 사용합니다. ELK는 **Elasticsearch**, **Logstash**, **Kibana** 세 제품의 머리글자입니다. 각 구성요소의 역할은 다음과 같습니다.

**Logstash**는 다양한 출처에서 데이터를 수집하고, 정제 및 변환한 뒤 Elasticsearch로 전달하는 ETL(Extract-Transform-Load) 파이프라인입니다. 내부적으로 Input(데이터 수집), Filter(정제 및 변환), Output(전달) 세 단계로 동작합니다.

**Kibana**는 Elasticsearch에 저장된 데이터를 시각화하고 탐색하는 대화형 도구입니다. 차트, 대시보드, 지도 등을 통해 검색 결과와 집계 데이터를 한눈에 파악할 수 있게 합니다.

이 세 가지 외에 **Beats**라는 경량 데이터 수집기도 있습니다. Beats는 서버에 설치하여 로그 파일(Filebeat), 시스템 성능 지표(Metricbeat), 네트워크 트래픽(Packetbeat) 등 특정 유형의 데이터를 수집합니다. Beats가 추가되면서 공식 명칭은 "Elastic Stack"으로 확장되었지만, 현장에서는 여전히 ELK Stack이라는 이름이 널리 쓰입니다. 소규모 환경에서는 Beats가 Elasticsearch에 직접 데이터를 전송하고, 복잡한 변환이 필요한 환경에서는 Beats와 Elasticsearch 사이에 Logstash를 배치합니다.

```
데이터 흐름

데이터 소스 → Beats/Logstash → Elasticsearch → Kibana
(수집/변환)      (저장/인덱싱/검색)   (시각화/탐색)
```

마지막으로 **OpenSearch**와의 관계를 짚겠습니다. OpenSearch는 2021년에 Amazon이 Elasticsearch 7.10 버전의 코드를 기반으로 분리(포크)하여 만든 검색 엔진입니다. 분리의 배경에는 라이선스 변경이 있습니다. Elasticsearch는 원래 Apache 2.0 라이선스로 누구나 자유롭게 사용하고 수정할 수 있었습니다. 그런데 2021년, Elastic 사는 7.11 버전부터 라이선스를 SSPL과 Elastic License 2.0의 이중 라이선스로 변경했습니다. 이 변경은 클라우드 서비스 제공자가 Elasticsearch를 그대로 서비스로 제공하는 것을 제한하려는 목적이었습니다. Amazon은 이에 대응하여 라이선스 변경 직전 버전인 7.10을 포크하고, Apache 2.0 라이선스를 유지한 OpenSearch를 출시했습니다. 현재 OpenSearch는 Linux Foundation 거버넌스 아래에서 400여 개 조직과 3,300여 명의 기여자가 참여하는 독립적인 오픈소스 프로젝트로 운영되고 있습니다. Elasticsearch와 OpenSearch는 출발점이 같으므로 기본 개념과 API 구조가 유사하지만, 이후 독자적으로 발전하면서 기능과 성능에서 차이가 생기고 있습니다.

정리하면, Elasticsearch는 Apache Lucene 기반의 분산형 검색 및 분석 엔진으로, 역색인을 통한 빠른 전문 검색과 집계를 통한 실시간 분석을 하나의 시스템에서 제공합니다. 데이터는 JSON 문서로 저장하고, 모든 조작은 RESTful API로 수행합니다. 샤드 분산과 레플리카 배치로 수평 확장성과 고가용성을 확보하며, refresh 주기에 따라 거의 실시간(NRT)으로 검색 결과를 반영합니다. Logstash, Kibana, Beats와 함께 ELK Stack(Elastic Stack)을 구성하여 데이터 수집부터 시각화까지 일관된 파이프라인을 제공합니다.

다음 단원인 1.1.2에서는 Elasticsearch를 직접 설치하고 실행 환경을 구성하는 방법을 다룹니다.

이 단원을 마치면 Elasticsearch의 정의와 핵심 특징을 설명할 수 있고, 검색 엔진과 분석 엔진으로서의 역할을 구분할 수 있으며, ELK Stack 구성요소를 나열하고 각 역할을 설명할 수 있습니다.
