# 8.2.2 Kibana 로그 분석 대시보드

8.2.1에서 Filebeat와 Logstash를 사용하여 서버 로그를 Elasticsearch에 적재하는 파이프라인을 구성했습니다. 로그가 Elasticsearch에 들어왔으면 이제 그 데이터를 분석해야 합니다. "오늘 오전에 500 에러가 얼마나 발생했는가?", "어떤 API 엔드포인트에서 응답 시간이 느려지고 있는가?", "특정 지역에서 트래픽이 급증했는가?" 같은 질문에 빠르게 답하려면, 수치와 차트를 한 화면에 모아 놓은 대시보드가 필요합니다.

이 단원에서는 Kibana를 사용하여 로그 분석 대시보드를 구성합니다. 4.3.2에서 다룬 Kibana Dashboard 기능과 8.2.1에서 구축한 로그 수집 파이프라인이 기반이 됩니다.

대시보드를 만들기 전에, Kibana에서 로그 데이터를 탐색하는 도구부터 살펴봅니다. Kibana에는 **Logs UI**라는 전용 로그 탐색 화면이 있습니다. Kibana 좌측 메뉴에서 Observability > Logs를 선택하면 접근할 수 있습니다.

Logs UI는 로그 메시지를 시간순으로 스트리밍 형태로 보여 줍니다. 터미널에서 tail -f 명령으로 로그를 실시간 추적하는 것과 비슷한 경험을 제공하되, 여러 서버의 로그를 한 화면에서 볼 수 있다는 점이 다릅니다. 화면 상단에 검색 바가 있어 특정 조건의 로그만 필터링할 수 있습니다.

이 검색 바에서 사용하는 것이 4.3.3에서 다룬 **KQL(Kibana Query Language)**입니다. 에러 로그만 보고 싶으면 다음과 같이 입력합니다.

```
http.response.status_code >= 400
```

이 KQL 쿼리는 HTTP 상태 코드가 400 이상인 로그만 필터링합니다. 8.2.1에서 ECS 필드명으로 정규화했기 때문에 http.response.status_code라는 필드 이름을 사용할 수 있습니다.

특정 서비스의 특정 에러만 보고 싶으면 AND 연산자로 조건을 결합합니다.

```
service.name: "api-gateway" AND http.response.status_code: 503
```

service.name이 "api-gateway"이고 상태 코드가 503(Service Unavailable)인 로그만 표시합니다. Logs UI에서 이런 필터를 적용한 채로 실시간 스트리밍을 켜 두면, 장애 상황에서 해당 서비스의 에러 로그가 들어오는 즉시 화면에 나타납니다.

이제 대시보드를 구성합니다. 4.3.2에서 다룬 것처럼, Kibana Dashboard는 여러 시각화 패널을 하나의 화면에 배치하는 기능입니다. 로그 분석 대시보드에 자주 들어가는 패널을 하나씩 만들어 봅니다.

첫 번째로 **시간별 요청 수** 패널을 만듭니다. 이 패널은 시간의 흐름에 따라 요청 수가 어떻게 변하는지를 보여 줍니다. Elasticsearch의 **date_histogram** 집계가 이 시각화의 기반입니다. date_histogram은 4.1.2에서 다룬 Bucket 집계의 하나로, @timestamp 필드를 기준으로 문서를 시간 구간별로 묶습니다.

Kibana Lens에서 이 패널을 만드는 절차는 다음과 같습니다. 대시보드 편집 모드에서 "Create visualization"을 선택합니다. 차트 유형으로 Line(선 차트)을 선택합니다. 가로축에 @timestamp를 드래그하면 Lens가 자동으로 date_histogram 집계를 적용합니다. 세로축에는 Count가 기본으로 설정되어 각 시간 구간의 문서 수, 즉 요청 수가 표시됩니다.

이 패널에 대응하는 Elasticsearch 쿼리를 보면 내부 동작을 더 명확히 이해할 수 있습니다.

```json
GET logs-nginx-production/_search
{
  "size": 0,
  "aggs": {
    "requests_over_time": {
      "date_histogram": {
        "field": "@timestamp",
        "calendar_interval": "1h"
      }
    }
  }
}
```

size를 0으로 설정하여 검색 결과 문서는 반환하지 않고 집계 결과만 받습니다. date_histogram의 calendar_interval을 "1h"로 설정하면 1시간 단위로 요청 수를 집계합니다. Kibana Lens에서 시간 구간은 화면에 표시되는 시간 범위에 따라 자동으로 조정됩니다.

두 번째로 **상위 에러 코드** 패널을 만듭니다. 어떤 종류의 에러가 가장 많이 발생하는지 파악하는 시각화입니다. **terms 집계**를 사용합니다. terms 집계는 4.1.2에서 다룬 것처럼, 특정 필드의 값별로 문서 수를 세는 Bucket 집계입니다.

```json
GET logs-nginx-production/_search
{
  "size": 0,
  "query": {
    "range": {
      "http.response.status_code": { "gte": 400 }
    }
  },
  "aggs": {
    "error_codes": {
      "terms": {
        "field": "http.response.status_code",
        "size": 10
      }
    }
  }
}
```

query에서 상태 코드 400 이상인 문서만 필터링하고, terms 집계로 상태 코드별 에러 수를 집계합니다. 결과는 404(Not Found), 500(Internal Server Error), 503(Service Unavailable) 같은 에러 코드와 각각의 발생 횟수를 보여 줍니다. Kibana에서는 파이 차트나 수평 막대 차트로 표현하면 에러 분포를 직관적으로 파악할 수 있습니다.

세 번째로 **응답 시간 분석** 패널을 만듭니다. 평균 응답 시간만 보면 소수의 극단적으로 느린 요청이 감춰질 수 있습니다. 예를 들어 100개 요청 중 99개가 50ms, 1개가 10초라면 평균은 약 150ms입니다. 이 평균만 보면 시스템이 정상인 것처럼 보이지만, 실제로는 일부 사용자가 10초 대기를 경험하고 있습니다. 이런 상황을 정확히 파악하려면 **percentiles 집계**를 사용합니다.

percentiles 집계는 4.1.3에서 다룬 Metric 집계의 하나로, 데이터의 분포를 백분위수로 보여 줍니다. p50은 중앙값, p95는 상위 5% 경계, p99는 상위 1% 경계입니다.

```json
GET logs-nginx-production/_search
{
  "size": 0,
  "aggs": {
    "response_time_percentiles": {
      "percentiles": {
        "field": "http.response.time",
        "percents": [50, 75, 95, 99]
      }
    }
  }
}
```

이 쿼리는 응답 시간의 p50, p75, p95, p99를 계산합니다. p50이 50ms이고 p99가 3초라면, 대부분의 요청은 빠르지만 상위 1%의 요청이 3초 이상 걸린다는 것을 알 수 있습니다. Kibana에서 Metric 패널이나 Gauge 패널로 이 값을 표시하면 한눈에 확인할 수 있습니다.

네 번째로 **지역별 트래픽** 시각화를 만듭니다. 로그의 클라이언트 IP 주소를 GeoIP 데이터베이스로 변환하면 요청이 어느 국가, 어느 도시에서 왔는지 알 수 있습니다. Logstash에 geoip 필터를 추가하면 IP 주소에서 위도, 경도, 국가 정보를 자동으로 추출합니다.

```
filter {
  geoip {
    source => "[source][ip]"
    target => "[source][geo]"
  }
}
```

source에 IP 주소가 담긴 필드를 지정하고, target에 지리 정보를 저장할 필드를 지정합니다. 이 필터를 거치면 source.geo.location에 위도와 경도, source.geo.country_name에 국가명이 추가됩니다.

Kibana에서 Maps 시각화를 사용하면 세계 지도 위에 트래픽 분포를 표시할 수 있습니다. source.geo.location 필드를 레이어로 추가하면, 각 지역에서 들어온 요청 수에 비례하는 원이 지도 위에 표시됩니다. 특정 국가에서 비정상적으로 트래픽이 증가하거나, 예상하지 못한 지역에서 접근이 들어오는 경우를 시각적으로 빠르게 파악할 수 있습니다.

마지막으로, 특정 조건이 충족되면 자동으로 알림을 보내는 **Kibana Alerting**을 살펴봅니다. 대시보드는 사람이 화면을 보고 있을 때만 유용합니다. 새벽 2시에 에러가 급증하면 담당자가 대시보드를 보고 있지 않는 한 알 수 없습니다. Alerting은 지정한 조건이 충족되면 이메일, Slack, PagerDuty 같은 채널로 알림을 보내 줍니다.

에러율이 임계치를 넘으면 알림을 보내는 규칙을 만드는 절차는 다음과 같습니다. Kibana 좌측 메뉴에서 Stack Management > Rules를 선택합니다. "Create rule"을 클릭합니다. Rule type으로 "Elasticsearch query"를 선택합니다. 이 유형은 주기적으로 Elasticsearch 쿼리를 실행하여, 결과가 조건을 충족하면 알림을 발생시킵니다.

쿼리 조건으로 "최근 5분간 500번대 에러 수가 100건을 초과"를 설정한다고 합시다. Check every(실행 주기)를 1분으로 설정합니다. 그러면 1분마다 쿼리를 실행하여 조건을 확인합니다.

```json
{
  "query": {
    "bool": {
      "must": [
        { "range": { "@timestamp": { "gte": "now-5m" } } },
        { "range": { "http.response.status_code": { "gte": 500 } } }
      ]
    }
  }
}
```

이 쿼리가 반환하는 문서 수가 100건을 초과하면 Alert이 발생합니다. Action으로 Slack Webhook이나 이메일을 설정하면, 담당자에게 즉시 알림이 전달됩니다.

완성된 로그 분석 대시보드의 전체 레이아웃을 정리합니다.

```
+----------+----------+----------+----------+
| 총 요청수 | 에러 수   | p95 응답  | 평균 응답 |
| (Metric) | (Metric) | (Metric) | (Metric) |
+----------+----------+----------+----------+
|                                            |
|   시간별 요청 수 추이 (date_histogram Line) |
|                                            |
+--------------------------------------------+
|                    |                        |
| 상위 에러 코드      |  지역별 트래픽 분포     |
| (terms Pie/Bar)   |  (Geo Map)             |
+--------------------+------------------------+
|                                            |
| 응답 시간 백분위 추이 (p50, p95, p99 Line)  |
|                                            |
+--------------------------------------------+
          + Alerting: 에러율 임계치 알림
```

상단에 핵심 수치를 Metric 패널로 한 줄 배치합니다. 가운데에 시간별 요청 수 추이를 넓은 선 차트로 두어 트래픽 패턴을 한눈에 봅니다. 하단 좌측에 에러 코드 분포, 우측에 지역별 트래픽 지도를 배치합니다. 가장 아래에 응답 시간 백분위 추이 차트를 두어 성능 변화를 추적합니다. 여기에 Alerting 규칙을 추가하면, 대시보드를 보고 있지 않을 때도 이상 상황을 감지할 수 있습니다.

정리하면, Kibana 로그 분석 대시보드는 Logs UI로 실시간 로그를 탐색하고, date_histogram으로 시간별 요청 추이를, terms 집계로 에러 코드 분포를, percentiles 집계로 응답 시간 분포를, geo 시각화로 지역별 트래픽을 보여 줍니다. Alerting으로 에러율 임계치를 감시하여, 이상 상황이 발생하면 자동으로 알림을 보냅니다.

다음 단원인 8.3.1에서는 로그 분석을 넘어, Logs, Metrics, APM 데이터를 하나의 Observability 플랫폼으로 통합하는 방법을 다룹니다.

이 단원을 마치면 Kibana에서 에러율, 응답시간, 트래픽 분석 대시보드를 구성할 수 있습니다.
