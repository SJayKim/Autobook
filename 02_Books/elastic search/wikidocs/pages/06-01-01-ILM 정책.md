# 6.1.1 ILM 정책

운영 환경에서 시간이 지남에 따라 인덱스가 계속 쌓이면, 디스크 용량은 한정되어 있으므로 언젠가는 공간이 부족해집니다. 최근 데이터는 자주 검색하지만, 한 달 전 로그를 매일 조회하는 경우는 드뭅니다. 세 달 전 데이터는 거의 열어 보지 않고, 일 년이 지난 데이터는 삭제해도 무방할 때가 많습니다. 이처럼 데이터의 활용도가 시간에 따라 달라지는 상황에서, 인덱스마다 수동으로 이동하거나 삭제하는 작업은 실수를 유발하고 운영 부담을 크게 늘립니다.

**ILM(Index Lifecycle Management)**은 이 문제를 해결하기 위해 Elasticsearch가 제공하는 인덱스 수명주기 자동화 기능입니다. ILM 정책을 정의하면, 인덱스가 생성된 시점부터 삭제될 때까지의 전 과정을 Elasticsearch가 자동으로 관리합니다. 관리자가 매번 개입하지 않아도, 정해진 조건에 따라 인덱스가 다음 단계로 전환됩니다.

ILM 정책은 인덱스가 거쳐가는 단계를 정의합니다. 각 단계를 **페이즈(phase)**라고 부르며, 총 다섯 가지가 있습니다. 모든 페이즈를 반드시 정의할 필요는 없고, 필요한 페이즈만 선택하면 됩니다. Elasticsearch는 정의된 페이즈만 순서대로 실행하고, 정의되지 않은 페이즈는 건너뜁니다.

**Hot** 페이즈는 인덱스가 처음 생성되어 활발히 쓰기와 읽기가 일어나는 단계입니다. 새 문서가 계속 들어오고, 검색 요청도 빈번합니다. 성능이 중요하므로 SSD 같은 고속 스토리지에 데이터를 배치합니다.

**Warm** 페이즈는 더 이상 새 문서가 들어오지 않지만, 검색은 여전히 발생하는 단계입니다. Hot만큼 빠른 응답이 필요하지는 않으므로, 상대적으로 저렴한 스토리지로 옮겨도 됩니다. 이 단계에서 샤드 수를 줄이거나 세그먼트를 병합하여 디스크 사용량을 최적화할 수 있습니다.

**Cold** 페이즈는 검색 빈도가 낮아진 데이터를 보관하는 단계입니다. 응답 속도가 다소 느려도 괜찮은 데이터가 이 단계에 머뭅니다.

**Frozen** 페이즈는 거의 조회하지 않는 아카이브 성격의 데이터를 위한 단계입니다. 극히 낮은 비용으로 보관하되, 필요할 때는 검색할 수 있어야 하는 경우에 사용합니다.

**Delete** 페이즈는 보존 기간이 만료된 인덱스를 영구 삭제하는 단계입니다.

이 다섯 단계를 한 번에 그림으로 정리하면 다음과 같습니다.

```
생성 ──▶ [ Hot ] ──▶ [ Warm ] ──▶ [ Cold ] ──▶ [ Frozen ] ──▶ [ Delete ]
           활발한       쓰기 중단     검색 빈도     거의 조회      보존 기간
           읽기/쓰기    검색 지속     낮음          안 함          만료 삭제
           고속 SSD     저비용 HDD    저비용        오브젝트       영구 삭제
                                     스토리지      스토리지
```

인덱스가 한 페이즈에서 다음 페이즈로 넘어가려면 두 가지 조건을 만족해야 합니다. 첫째, 현재 페이즈에 정의된 모든 액션이 완료되어야 합니다. 둘째, 인덱스의 나이가 다음 페이즈에 설정된 **min_age** 값 이상이어야 합니다. min_age는 인덱스가 생성된 시점(또는 롤오버된 시점)부터 경과한 시간을 기준으로 판단합니다.

**롤오버(rollover)**는 Hot 페이즈에서 자주 사용하는 액션입니다. 현재 쓰기 인덱스가 지정한 크기, 문서 수, 또는 경과 시간 중 하나의 임곗값에 도달하면, 새로운 인덱스를 자동으로 생성하고 쓰기 대상을 전환합니다. 기존 인덱스는 읽기 전용이 되어, 이후 Warm이나 Cold로 자연스럽게 이동합니다.

롤오버 외에도 ILM이 각 페이즈에서 수행할 수 있는 액션이 있습니다. **shrink**는 프라이머리 샤드 수를 줄여 리소스 사용을 최적화합니다. **force merge**는 여러 세그먼트를 하나로 합쳐 검색 효율을 높이고 삭제된 문서가 차지하던 공간을 회수합니다. **allocate**는 인덱스를 다른 티어의 노드로 이동시킵니다. **delete**는 인덱스를 완전히 제거합니다.

ILM 정책을 만드는 방법을 살펴보겠습니다. 다음은 PUT _ilm/policy API를 사용하여 정책을 생성하는 예입니다.

```json
PUT _ilm/policy/my_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50gb",
            "max_age": "30d"
          }
        }
      },
      "warm": {
        "min_age": "30d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "cold": {
        "min_age": "90d",
        "actions": {
          "allocate": {
            "require": { "data": "cold" }
          }
        }
      },
      "delete": {
        "min_age": "365d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

이 정책의 동작을 단계별로 살펴보겠습니다. hot 페이즈의 rollover 액션은 인덱스 크기가 50GB에 도달하거나 30일이 경과하면 새 인덱스를 생성합니다. max_size와 max_age 중 하나라도 먼저 충족되면 롤오버가 발생합니다. warm 페이즈는 min_age가 30d이므로, 인덱스가 생성(또는 롤오버)된 후 30일이 지나면 진입합니다. 이때 shrink로 프라이머리 샤드를 1개로 줄이고, forcemerge로 세그먼트를 1개로 병합합니다. cold 페이즈는 90일 후에 진입하며, allocate 액션으로 cold 티어 노드로 데이터를 이동합니다. delete 페이즈는 365일 후 인덱스를 삭제합니다.

정책을 만들었으면 인덱스에 연결해야 합니다. 가장 일반적인 방법은 인덱스 템플릿에 ILM 정책을 지정하는 것입니다. 5.2.2에서 다룬 인덱스 템플릿의 settings 안에 index.lifecycle.name 속성을 추가하면, 해당 템플릿으로 생성되는 모든 인덱스에 ILM 정책이 자동으로 적용됩니다. 롤오버를 사용할 경우에는 index.lifecycle.rollover_alias도 함께 설정합니다.

이미 존재하는 인덱스에 ILM 정책을 직접 연결할 수도 있습니다. PUT 요청으로 해당 인덱스의 settings에 index.lifecycle.name을 지정하면 됩니다.

ILM이 인덱스를 어떤 상태로 관리하고 있는지 확인하려면 _ilm/explain API를 사용합니다.

```json
GET my-index-000001/_ilm/explain
```

이 API는 인덱스가 현재 어떤 페이즈에 있는지, 해당 페이즈에 진입한 시각, 다음 페이즈로 전환되기까지 남은 조건 등을 보여 줍니다. 운영 중 특정 인덱스의 수명주기 진행 상황을 파악할 때 유용합니다.

ILM 진행 도중 오류가 발생하면 해당 인덱스의 수명주기가 멈출 수 있습니다. 예를 들어 shrink 대상 노드가 부족하거나, 디스크 공간이 없어 allocate가 실패하는 경우입니다. _ilm/explain 응답에서 오류 원인을 확인한 뒤, 문제를 해결하고 재시도 API로 수명주기를 다시 진행시킬 수 있습니다.

```json
POST my-index-000001/_ilm/retry
```

이 호출은 멈춰 있던 ILM 단계를 다시 시작합니다. 만약 특정 인덱스를 수동으로 다른 페이즈로 즉시 이동시켜야 할 경우에는 _ilm/move API를 사용합니다.

```json
POST _ilm/move/my-index-000001
{
  "current_step": {
    "phase": "warm",
    "action": "complete",
    "name": "complete"
  },
  "next_step": {
    "phase": "cold",
    "action": "complete",
    "name": "complete"
  }
}
```

이 API는 운영 중 긴급하게 데이터를 이동시켜야 할 때 사용합니다. 일상적인 운영에서는 min_age 조건에 따라 자동 전환이 이루어지도록 두는 것이 안전합니다.

정리하면, ILM은 인덱스의 생성부터 삭제까지의 수명주기를 Hot, Warm, Cold, Frozen, Delete 다섯 페이즈로 나누어 자동 관리하는 기능입니다. 각 페이즈에 전환 조건과 액션을 정의하면, Elasticsearch가 조건 충족 시 자동으로 롤오버, 샤드 축소, 세그먼트 병합, 노드 이동, 삭제를 수행합니다. 정책은 인덱스 템플릿에 연결하여 일괄 적용하고, _ilm/explain으로 진행 상태를 모니터링하며, 오류 시 retry API로 재시도할 수 있습니다.

다음 단원인 6.1.2에서는 Data Stream을 다룹니다. Data Stream은 ILM과 결합하여 시계열 데이터의 수명주기를 더욱 체계적으로 관리하는 구조입니다.

이 단원을 마치면 ILM 정책을 생성하고 인덱스에 연결하여 단계별 전환을 자동화할 수 있습니다.
