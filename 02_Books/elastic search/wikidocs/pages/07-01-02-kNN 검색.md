# 7.1.2 kNN 검색

7.1.1에서 dense_vector 필드와 HNSW 그래프를 설정하는 방법을 다뤘습니다. 벡터를 저장하고 인덱스 구조를 만드는 것까지 완료했으니, 이제 실제로 "가장 가까운 벡터를 가진 문서"를 찾는 검색을 수행할 차례입니다. 이 검색 방식을 **kNN**(k-Nearest Neighbors, k-최근접 이웃) 검색이라고 부릅니다. k는 찾고 싶은 이웃의 수입니다. 사용자가 "상위 10개 결과를 보여달라"고 요청한다면 k는 10이 됩니다.

Elasticsearch는 kNN 검색을 두 가지 방식으로 지원합니다. 하나는 **approximate kNN**(근사 kNN)이고, 다른 하나는 **exact kNN**(정확 kNN)입니다. 두 방식은 정확도와 속도 사이에서 서로 다른 선택을 합니다.

approximate kNN은 7.1.1에서 만든 HNSW 그래프를 활용합니다. 그래프를 탐색하면서 "충분히 가까운" 이웃을 빠르게 찾습니다. 모든 벡터를 하나하나 비교하지 않으므로 속도가 매우 빠릅니다. 수백만 건의 문서에서도 밀리초 단위로 결과를 돌려줍니다. 다만 그래프 탐색이라는 특성상, 이론적으로 가장 가까운 이웃을 빠뜨릴 가능성이 아주 작게 존재합니다.

exact kNN은 인덱스의 모든 벡터와 쿼리 벡터 사이의 거리를 전부 계산합니다. 결과는 정확하지만, 문서 수가 많아지면 계산 시간이 급격히 늘어납니다. 필터를 적용한 뒤 남은 문서가 1만 건 이하처럼 적을 때 적합합니다.

approximate kNN 검색은 검색 요청의 knn 파라미터를 사용합니다. 핵심 파라미터를 살펴보겠습니다.

```json
GET my-vector-index/_search
{
  "knn": {
    "field": "content_embedding",
    "query_vector": [0.12, -0.34, 0.56, ...],
    "k": 10,
    "num_candidates": 100
  }
}
```

field는 dense_vector 타입으로 매핑한 필드 이름입니다. query_vector는 검색 기준이 되는 벡터입니다. 사용자의 검색어를 임베딩 모델에 넣어 얻은 숫자 배열을 여기에 넣습니다. **k**는 최종적으로 돌려받을 결과 수입니다.

**num_candidates**는 HNSW 그래프 탐색 단계에서 후보로 고려할 문서 수입니다. 각 샤드에서 num_candidates만큼의 후보를 먼저 모은 뒤, 그 안에서 상위 k개를 골라 반환합니다. num_candidates가 클수록 더 많은 후보를 살펴보므로 정확도(recall)가 올라가지만, 검색 시간이 늘어납니다. 반대로 num_candidates가 작으면 빠르지만 좋은 결과를 놓칠 수 있습니다.

이 관계를 정리하면 다음과 같습니다.

```
num_candidates 작음 ─────────────────── num_candidates 큼
   빠른 응답, 낮은 recall          느린 응답, 높은 recall
```

일반적으로 num_candidates를 k의 5~10배로 설정하면 recall과 성능 사이의 균형이 잘 맞습니다. 기본값은 없으며, 반드시 k 이상의 값을 지정해야 합니다.

query_vector를 직접 넣는 대신 Elasticsearch가 자동으로 임베딩을 생성하게 할 수도 있습니다. 이때 사용하는 것이 **query_vector_builder**입니다.

```json
GET my-vector-index/_search
{
  "knn": {
    "field": "content_embedding",
    "query_vector_builder": {
      "text_embedding": {
        "model_id": "sentence-transformers__all-minilm-l6-v2",
        "model_text": "가을 산책하기 좋은 곳"
      }
    },
    "k": 10,
    "num_candidates": 100
  }
}
```

query_vector_builder의 text_embedding 안에 model_id와 model_text를 지정합니다. model_id는 Elasticsearch ML 노드에 배포된 임베딩 모델의 식별자입니다. model_text는 검색어 원문입니다. Elasticsearch가 검색 시점에 model_text를 모델에 넣어 벡터를 생성하고, 그 벡터로 kNN 검색을 수행합니다. 애플리케이션 쪽에서 임베딩 변환 로직을 따로 만들 필요가 없어지는 장점이 있습니다.

exact kNN은 script_score 쿼리를 사용합니다. 모든 문서(또는 필터로 좁힌 문서)에 대해 스크립트로 벡터 거리를 계산하고, 그 결과를 점수로 사용합니다.

```json
GET my-vector-index/_search
{
  "query": {
    "script_score": {
      "query": {
        "bool": {
          "filter": {
            "term": { "category": "travel" }
          }
        }
      },
      "script": {
        "source": "cosineSimilarity(params.queryVector, 'content_embedding') + 1.0",
        "params": {
          "queryVector": [0.12, -0.34, 0.56, ...]
        }
      }
    }
  }
}
```

bool 필터로 category가 travel인 문서만 남긴 뒤, script에서 cosineSimilarity 함수로 각 문서의 content_embedding과 쿼리 벡터 사이의 코사인 유사도를 계산합니다. +1.0을 더하는 이유는 Elasticsearch의 점수가 음수가 되면 안 되기 때문입니다. 코사인 유사도는 -1에서 1 사이의 값을 가지므로 1을 더해 0에서 2 사이로 보정합니다.

exact kNN은 HNSW 그래프를 거치지 않고 대상 문서를 전부 스캔하므로, 필터를 거친 뒤 남은 문서가 적을 때만 사용하는 것이 좋습니다. 일반적으로 1만 건 이하일 때 적합합니다.

approximate kNN에서도 필터를 함께 사용할 수 있습니다. 이때 **pre-filter**와 **post-filter**의 차이를 이해해야 합니다.

pre-filter는 HNSW 그래프 탐색 전에 필터를 적용합니다. 필터 조건에 맞는 문서만 후보로 남기고, 그 후보 안에서 kNN 검색을 수행합니다. 결과적으로 k개의 결과가 모두 필터 조건을 만족합니다. Elasticsearch의 knn 파라미터에 filter를 지정하면 pre-filter로 동작합니다.

```json
{
  "knn": {
    "field": "content_embedding",
    "query_vector": [0.12, -0.34, ...],
    "k": 10,
    "num_candidates": 100,
    "filter": {
      "term": { "category": "travel" }
    }
  }
}
```

post-filter는 kNN 검색이 끝난 뒤에 필터를 적용합니다. 먼저 num_candidates만큼의 후보를 찾고, 그 후보 중 필터 조건에 맞는 것만 남깁니다. 결과가 k개보다 적어질 수 있다는 단점이 있습니다. 검색 요청의 최상위 post_filter 절에 조건을 넣으면 post-filter로 동작합니다.

```
               pre-filter                       post-filter
  ┌─────────────────────────┐       ┌──────────────────────────┐
  │ 1. 필터 적용            │       │ 1. kNN 검색 (전체 대상)  │
  │ 2. 필터 통과 문서에서   │       │ 2. 상위 k개 후보 선정    │
  │    kNN 검색             │       │ 3. 필터 적용             │
  │ 3. 결과: 정확히 k개     │       │ 4. 결과: k개 이하 가능   │
  └─────────────────────────┘       └──────────────────────────┘
```

대부분의 경우 pre-filter를 사용하는 것이 결과 수를 보장할 수 있어 안정적입니다.

kNN 검색의 응답 구조는 일반 검색과 동일한 형태입니다. hits 배열 안에 각 문서가 _score와 함께 담겨 반환됩니다. _score는 similarity 메트릭에 따라 계산된 유사도 점수입니다. cosine을 사용한 경우 0에 가까울수록 벡터가 가깝고(유사도가 높고), 점수가 (1 + cosine similarity) / 2로 정규화되어 0~1 사이의 값으로 나타납니다.

```json
{
  "hits": {
    "total": { "value": 10, "relation": "eq" },
    "max_score": 0.95,
    "hits": [
      {
        "_index": "my-vector-index",
        "_id": "doc1",
        "_score": 0.95,
        "_source": { "title": "단풍 명소 추천", ... }
      },
      ...
    ]
  }
}
```

정리하면, kNN 검색은 approximate kNN과 exact kNN 두 가지로 나뉩니다. approximate kNN은 HNSW 그래프를 활용해 빠르게 근사 결과를 찾고, k와 num_candidates로 정확도와 속도의 균형을 조절합니다. query_vector를 직접 넣거나 query_vector_builder로 검색 시점에 임베딩을 생성할 수 있습니다. 필터는 pre-filter와 post-filter 중 선택하며, 대부분 pre-filter가 안정적입니다. exact kNN은 script_score로 모든 대상을 전수 비교하므로 소규모 데이터셋에 적합합니다.

다음 단원인 7.1.3에서는 kNN 검색에 사용할 임베딩 모델을 Elasticsearch ML 노드에 배포하는 방법을 다룹니다.

이 단원을 마치면 knn 파라미터로 근사 최근접 이웃 검색을 수행할 수 있고, exact kNN과 approximate kNN의 차이를 설명할 수 있습니다.
