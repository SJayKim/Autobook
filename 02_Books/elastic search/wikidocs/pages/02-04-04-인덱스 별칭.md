# 2.4.4 인덱스 별칭

Elasticsearch에서 인덱스를 운영하다 보면, 인덱스 이름이 바뀌어야 하는 상황이 생깁니다. 매핑을 변경해야 해서 새 인덱스를 만들거나, 날짜별로 인덱스를 교체하거나, 특정 조건의 데이터만 따로 조회하고 싶은 경우가 대표적입니다. 문제는, 애플리케이션 코드가 인덱스 이름을 직접 참조하고 있다는 점입니다. 인덱스 이름이 바뀔 때마다 애플리케이션의 설정이나 쿼리 코드를 함께 수정해야 합니다. 인덱스가 하나일 때는 큰 부담이 아니지만, 수십 개의 인덱스를 운영하거나 인덱스 교체가 잦은 환경에서는 변경 범위가 넓어집니다.

이 문제를 해결하는 장치가 **인덱스 별칭**(alias)입니다. 인덱스 별칭은 하나 이상의 인덱스에 붙이는 보조 이름입니다. 애플리케이션은 실제 인덱스 이름 대신 별칭을 통해 데이터를 읽고 씁니다. 별칭이 가리키는 인덱스를 바꾸더라도 애플리케이션 코드는 수정할 필요가 없습니다. 별칭은 인덱스와 애플리케이션 사이에 놓인 간접 참조 계층입니다. 2.4.3에서 인덱스 템플릿이 인덱스 생성 시 설정을 자동 부여하는 규칙이었다면, 별칭은 이미 존재하는 인덱스에 유연한 접근 경로를 부여하는 장치입니다.

별칭은 Elasticsearch의 `_aliases` API로 관리합니다. 별칭 생성, 삭제, 교체를 모두 이 API 하나로 처리합니다. 먼저 가장 단순한 형태인 별칭 생성부터 살펴봅니다.

```
PUT /my-index/_alias/my-alias
```

이 요청은 my-index라는 인덱스에 my-alias라는 별칭을 붙입니다. 이후 my-alias를 인덱스 이름처럼 사용할 수 있습니다. `GET /my-alias/_search`로 검색하면 my-index의 데이터가 조회됩니다. 별칭을 삭제할 때는 다음과 같이 요청합니다.

```
DELETE /my-index/_alias/my-alias
```

이 요청은 my-index에서 my-alias라는 별칭을 제거합니다. 인덱스 자체가 삭제되는 것은 아닙니다. 별칭만 사라지므로, 이후 my-alias로는 더 이상 접근할 수 없지만 my-index로는 정상 접근됩니다.

별칭 하나가 여러 인덱스를 가리킬 수도 있습니다. 예를 들어 월별로 분리된 로그 인덱스가 있을 때, 하나의 별칭으로 여러 달의 데이터를 한꺼번에 검색할 수 있습니다. `POST /_aliases` API에 actions 배열을 보내면 여러 동작을 한 번에 처리합니다.

```json
POST /_aliases
{
  "actions": [
    { "add": { "index": "logs-2024-01", "alias": "logs-current" } },
    { "add": { "index": "logs-2024-02", "alias": "logs-current" } }
  ]
}
```

이 요청은 logs-current라는 별칭에 두 개의 인덱스를 연결합니다. actions 배열 안에 add 동작을 두 번 넣었습니다. 첫 번째 add는 logs-2024-01을, 두 번째 add는 logs-2024-02를 logs-current 별칭에 추가합니다. 이후 `GET /logs-current/_search`를 실행하면, 두 인덱스의 데이터가 함께 검색됩니다. 별칭이 다중 인덱스를 묶는 가상의 창구 역할을 하는 것입니다.

등록된 별칭 목록을 확인하려면 다음 요청을 사용합니다.

```
GET /_aliases
```

이 요청은 클러스터에 존재하는 모든 인덱스와 각 인덱스에 연결된 별칭 정보를 JSON 형태로 반환합니다. 특정 인덱스의 별칭만 확인하려면 `GET /my-index/_alias`처럼 인덱스 이름을 경로에 포함합니다.

별칭의 가장 강력한 활용 방식은 **원자적 교체**(atomic swap)입니다. 원자적이라는 표현은, 교체 동작이 중간 상태 없이 한 번에 완료된다는 뜻입니다. 별칭이 기존 인덱스에서 새 인덱스로 전환되는 동안, 별칭이 아무 인덱스도 가리키지 않는 순간이 존재하지 않습니다.

이 기능이 필요한 대표적인 상황을 하나 들어 봅니다. 운영 중인 인덱스의 매핑을 변경해야 한다고 가정합니다. Elasticsearch에서 기존 필드의 타입을 변경하는 것은 허용되지 않으므로, 새 매핑으로 인덱스를 새로 만든 뒤 데이터를 옮기는 방법을 써야 합니다. 이 과정을 무중단으로 처리하려면, 애플리케이션이 인덱스 이름이 아닌 별칭을 바라보고 있어야 합니다. 전체 절차를 단계별로 정리합니다.

```
무중단 인덱스 교체 절차

1. 새 매핑으로 새 인덱스(logs-new) 생성
2. 기존 인덱스(logs-old)의 데이터를 logs-new로 재색인
3. 별칭(logs-current)을 logs-old에서 제거하고 logs-new에 추가 (원자적)
4. 기존 인덱스(logs-old) 삭제
```

3단계가 핵심입니다. remove와 add를 하나의 `_aliases` 요청에 함께 넣으면, Elasticsearch가 두 동작을 하나의 트랜잭션으로 처리합니다.

```json
POST /_aliases
{
  "actions": [
    { "remove": { "index": "logs-old", "alias": "logs-current" } },
    { "add": { "index": "logs-new", "alias": "logs-current" } }
  ]
}
```

이 요청에서 actions 배열의 첫 번째 원소는 logs-old에서 logs-current 별칭을 제거하는 동작이고, 두 번째 원소는 logs-new에 logs-current 별칭을 추가하는 동작입니다. Elasticsearch는 이 두 동작을 묶어서 실행하므로, 별칭이 잠시라도 비어 있는 상태가 발생하지 않습니다. 애플리케이션은 logs-current라는 이름만 계속 사용하면 되고, 그 뒤에서 인덱스가 교체되었다는 사실을 알 필요가 없습니다.

만약 remove와 add를 별개의 요청으로 나누어 보내면, 첫 번째 요청(remove)이 완료된 시점부터 두 번째 요청(add)이 처리될 때까지 별칭이 존재하지 않는 빈틈이 생깁니다. 그 사이에 애플리케이션이 logs-current로 요청을 보내면 오류가 발생합니다. 하나의 actions 배열에 담아 보내야 하는 이유가 여기에 있습니다.

별칭에는 **필터**(filter)를 붙일 수 있습니다. 필터 별칭은 별칭으로 접근할 때 자동으로 특정 조건을 추가하여, 인덱스의 일부 데이터만 보이게 하는 기능입니다. 하나의 인덱스에 여러 고객사의 데이터가 섞여 있는 상황을 예로 들어 봅니다. 각 문서에 tenant_id 필드가 있고, 고객사마다 다른 값을 가집니다.

```json
POST /_aliases
{
  "actions": [
    {
      "add": {
        "index": "customer_data",
        "alias": "customer_acme",
        "filter": { "term": { "tenant_id": "acme" } }
      }
    }
  ]
}
```

이 요청은 customer_data 인덱스에 customer_acme이라는 별칭을 추가하면서, filter 조건을 함께 지정합니다. filter 안의 term 쿼리는 tenant_id 필드가 "acme"인 문서만 선택한다는 조건입니다. 이후 customer_acme 별칭으로 검색하면, tenant_id가 "acme"인 문서만 결과에 포함됩니다. 애플리케이션 코드에서 별도로 필터 조건을 넣을 필요가 없습니다.

같은 인덱스에 다른 고객사용 별칭을 하나 더 만들면, 각 별칭이 서로 다른 데이터 범위를 보여 주는 가상의 뷰가 됩니다. 물리적으로는 하나의 인덱스이지만, 별칭을 통해 논리적으로 분리된 것처럼 접근할 수 있습니다. 이 방식은 고객사마다 별도 인덱스를 만드는 것보다 자원 효율이 높습니다.

주의할 점이 있습니다. 필터 별칭은 Query DSL을 사용하는 검색에서만 작동합니다. 문서 ID로 직접 조회하는 GET 요청에는 필터가 적용되지 않습니다. 예를 들어 `GET /customer_acme/_doc/123`처럼 문서 ID를 지정하면, tenant_id와 무관하게 해당 문서가 반환됩니다.

별칭에 **라우팅**(routing) 값을 설정할 수도 있습니다. 라우팅은 문서가 어느 샤드에 저장되고, 검색 시 어느 샤드를 조회할지를 결정하는 값입니다. Elasticsearch는 기본적으로 문서 ID를 해시하여 샤드를 결정하지만, 라우팅 값을 지정하면 같은 라우팅 값을 가진 문서가 같은 샤드에 모입니다.

별칭에 라우팅을 설정하면, 해당 별칭을 통한 모든 요청이 지정된 샤드로만 향합니다.

```json
POST /_aliases
{
  "actions": [
    {
      "add": {
        "index": "my-index",
        "alias": "my-alias",
        "routing": "1"
      }
    }
  ]
}
```

이 요청은 my-alias 별칭에 라우팅 값 "1"을 지정합니다. 이후 my-alias를 통해 문서를 색인하거나 검색하면, 라우팅 값이 "1"인 샤드만 대상이 됩니다. 전체 샤드를 순회하지 않으므로 검색 효율이 높아집니다.

색인과 검색의 라우팅 값을 따로 지정할 수도 있습니다. index_routing은 문서를 색인할 때 사용하는 라우팅 값이고, search_routing은 검색할 때 사용하는 라우팅 값입니다.

```json
POST /_aliases
{
  "actions": [
    {
      "add": {
        "index": "my-index",
        "alias": "my-alias",
        "index_routing": "2",
        "search_routing": "1,2"
      }
    }
  ]
}
```

이 요청에서 index_routing이 "2"이므로, my-alias를 통해 색인하는 문서는 라우팅 값 "2"에 해당하는 샤드에 저장됩니다. search_routing이 "1,2"이므로, my-alias를 통한 검색은 라우팅 값 "1"과 "2"에 해당하는 샤드 두 곳을 조회합니다. 색인은 한 샤드에 집중하되, 검색 범위는 넓히는 구성입니다.

별칭이 여러 인덱스를 가리킬 때, 새 문서를 색인하면 어느 인덱스에 들어가야 하는지가 모호해집니다. 이 문제를 해결하는 것이 **is_write_index** 설정입니다. is_write_index를 true로 지정한 인덱스가 해당 별칭의 쓰기 대상이 됩니다.

```json
POST /_aliases
{
  "actions": [
    {
      "add": {
        "index": "logs-2024-01",
        "alias": "logs-write"
      }
    },
    {
      "add": {
        "index": "logs-2024-02",
        "alias": "logs-write",
        "is_write_index": true
      }
    }
  ]
}
```

이 요청은 logs-write 별칭에 두 인덱스를 연결하되, logs-2024-02에만 is_write_index를 true로 설정합니다. 이후 logs-write 별칭으로 문서를 색인하면 logs-2024-02에 저장됩니다. 검색은 두 인덱스 모두를 대상으로 합니다. 나중에 logs-2024-03을 새로 만들고 is_write_index를 이 인덱스로 옮기면, 쓰기 대상만 교체하면서 읽기 범위는 계속 확장할 수 있습니다.

아래 다이어그램은 별칭의 주요 기능을 한눈에 정리한 것입니다.

```
별칭의 역할 요약

애플리케이션
    |
    v
[  별칭 (alias)  ] ---- filter: tenant_id = "acme"   (필터 별칭)
    |                    routing: "1"                   (라우팅 별칭)
    |                    is_write_index: true            (쓰기 대상 지정)
    |
    +---> 인덱스 A  (읽기 + 쓰기)
    +---> 인덱스 B  (읽기 전용)
    +---> 인덱스 C  (읽기 전용)
```

정리하면, 인덱스 별칭은 실제 인덱스 이름을 감추는 간접 참조 계층입니다. 애플리케이션은 별칭만 바라보므로, 뒤에서 인덱스를 교체해도 코드 변경이 필요하지 않습니다. `_aliases` API의 actions 배열에 remove와 add를 함께 넣으면 원자적 교체가 가능하여, 무중단 인덱스 전환을 구현할 수 있습니다. 필터 별칭은 하나의 인덱스를 논리적으로 분리하고, 라우팅 별칭은 특정 샤드로 요청을 한정하며, is_write_index는 다중 인덱스 별칭에서 쓰기 대상을 명확히 지정합니다.

다음 단원인 3.1.1에서는 Elasticsearch REST API 구조를 다룹니다. 이 단원부터 Phase 3으로 넘어가, Elasticsearch와 실제로 상호작용하는 API의 구조와 사용법을 본격적으로 살펴봅니다.

이 단원을 마치면 인덱스 별칭을 생성하고 원자적 교체를 수행할 수 있으며, 필터 별칭을 활용하여 가상 인덱스 뷰를 구성할 수 있습니다.
