# 5.1.3 분산 읽기/쓰기 흐름

5.1.2에서 마스터 선출과 클러스터 토폴로지를 살펴보았습니다. 전용 마스터 노드가 클러스터를 관리하고, 데이터 노드가 샤드를 보유한다는 구조를 확인했습니다. 이제 클라이언트가 문서를 저장하거나 검색할 때, 이 요청이 클러스터 내부에서 어떤 경로로 흘러가는지를 살펴볼 차례입니다. 인덱싱(쓰기)과 검색(읽기)은 각각 다른 흐름을 따릅니다.

먼저 인덱싱 흐름부터 살펴봅니다. 클라이언트가 문서를 저장하는 요청을 클러스터에 보내면, 그 요청은 클러스터의 아무 노드에 도달할 수 있습니다. 요청을 처음 받은 노드가 **코디네이팅 노드**(coordinating node)가 됩니다. 5.1.1에서 모든 노드가 기본적으로 코디네이팅 역할을 수행한다고 했습니다. 코디네이팅 노드는 요청을 직접 처리하지 않고, 해당 문서가 저장되어야 할 샤드를 찾아 요청을 전달합니다.

문서가 저장될 샤드는 1.3.3에서 다룬 라우팅 공식으로 결정됩니다. `hash(_routing) % number_of_primary_shards` 계산 결과가 프라이머리 샤드 번호입니다. 코디네이팅 노드는 이 계산을 수행하여 해당 프라이머리 샤드가 위치한 노드를 알아내고, 그 노드로 요청을 전달합니다.

프라이머리 샤드가 있는 노드는 문서를 로컬에 기록합니다. 1.2.4에서 다룬 translog에 먼저 기록하고, 인메모리 버퍼에 문서를 추가합니다. 프라이머리 샤드에 기록이 완료되면, 다음 단계로 **레플리카 복제**가 시작됩니다.

프라이머리 샤드는 자신의 레플리카 샤드가 있는 모든 노드에 동일한 문서를 병렬로 전송합니다. 레플리카가 1개면 1곳, 2개면 2곳에 동시에 보냅니다. 각 레플리카 샤드가 문서를 기록하고 성공 응답을 돌려보내면, 프라이머리 샤드는 코디네이팅 노드에 완료를 알립니다. 코디네이팅 노드는 최종적으로 클라이언트에 성공 응답을 반환합니다.

이 흐름을 단계별로 정리합니다.

```
인덱싱(쓰기) 흐름

클라이언트
    |
    v
1. 코디네이팅 노드 (요청 수신)
    |
    | hash(_routing) % number_of_primary_shards
    | = 대상 프라이머리 샤드 번호 계산
    |
    v
2. 프라이머리 샤드 노드 (문서 기록)
    |
    +---> 3a. 레플리카 샤드 노드 A (복제)
    |
    +---> 3b. 레플리카 샤드 노드 B (복제, 레플리카 2인 경우)
    |
    v
4. 모든 레플리카 응답 수신
    |
    v
5. 코디네이팅 노드 -> 클라이언트에 성공 응답
```

여기서 중요한 설정이 **wait_for_active_shards**입니다. 이 설정은 인덱싱 요청이 진행되기 전에 활성 상태여야 하는 샤드 복사본의 최소 수를 지정합니다. 기본값은 1입니다. 기본값 1은 프라이머리 샤드만 활성 상태면 인덱싱을 진행한다는 뜻입니다.

```json
PUT /my-index/_doc/1?wait_for_active_shards=2
{
  "title": "예시 문서"
}
```

이 요청은 프라이머리 샤드 1개와 레플리카 샤드 1개, 총 2개의 활성 샤드가 확인된 후에야 인덱싱을 시작합니다. 만약 레플리카 샤드가 아직 할당되지 않은 상태라면, 타임아웃까지 대기한 뒤 실패를 반환합니다. "all"로 설정하면 프라이머리와 모든 레플리카가 활성 상태일 때만 인덱싱이 진행됩니다.

이 설정은 데이터 안전성과 인덱싱 속도 사이의 균형을 조절합니다. 값을 높이면 데이터 유실 가능성이 줄지만, 레플리카 노드 장애 시 인덱싱이 차단될 수 있습니다. 값을 낮추면 인덱싱이 빠르지만, 프라이머리 노드 장애 시 아직 복제되지 않은 문서가 유실될 위험이 있습니다.

주의할 점이 있습니다. wait_for_active_shards가 충족되었다는 것은 인덱싱 '시작' 조건이 만족되었다는 의미이지, 모든 레플리카에 '복제 완료'를 보장하는 것은 아닙니다. 인덱싱이 시작된 후 프라이머리에서 레플리카로의 복제가 진행되는 동안, 레플리카 노드에 장애가 발생하면 해당 레플리카의 복제는 실패할 수 있습니다.

이제 검색 흐름을 살펴봅니다. 검색은 인덱싱과 다른 패턴을 따릅니다. Elasticsearch에서 분산 검색은 **scatter/gather** 패턴으로 동작합니다. scatter는 '흩뿌리다', gather는 '모으다'라는 뜻입니다. 요청을 여러 샤드에 흩뿌리고, 각 샤드의 결과를 모아서 최종 응답을 만드는 구조입니다.

검색의 첫 단계는 **scatter 단계**입니다. 클라이언트가 검색 요청을 보내면, 코디네이팅 노드가 이를 수신합니다. 코디네이팅 노드는 대상 인덱스의 모든 샤드에 검색 요청을 보내야 합니다. 이때 같은 번호의 프라이머리 샤드와 레플리카 샤드 중 하나만 선택합니다. 예를 들어 샤드 0에 대해 프라이머리 P0와 레플리카 R0가 있으면, 둘 중 하나에만 요청을 보냅니다. 프라이머리와 레플리카는 동일한 데이터를 가지고 있으므로, 어느 쪽에 보내든 결과는 같습니다.

각 샤드는 로컬에서 검색을 실행하고, 요청한 크기만큼의 상위 결과(문서 ID와 점수)를 코디네이팅 노드에 반환합니다. 이 단계에서 각 샤드가 전체 문서 내용을 보내는 것은 아닙니다. 문서 ID와 정렬 기준 값만 전송하여 네트워크 부담을 줄입니다.

두 번째 단계는 **gather 단계**입니다. 코디네이팅 노드는 모든 샤드에서 받은 부분 결과를 모읍니다. 각 샤드가 반환한 문서 ID와 점수를 **전역 정렬**(global sort)합니다. 기본 정렬 기준은 관련성 점수(_score)이며, 사용자가 명시한 정렬 기준이 있으면 그것을 따릅니다. 정렬 후 요청한 수(기본값 10)만큼의 최종 문서 ID를 선정합니다. 코디네이팅 노드는 선정된 문서 ID를 기준으로 해당 샤드에 문서 내용(_source)을 요청하고, 받은 내용을 취합하여 클라이언트에 최종 응답을 반환합니다.

```
검색 흐름 (scatter/gather)

클라이언트
    |
    v
코디네이팅 노드
    |
    | [scatter 단계]
    +---> 샤드 0 (P0 또는 R0 중 하나)
    +---> 샤드 1 (P1 또는 R1 중 하나)
    +---> 샤드 2 (P2 또는 R2 중 하나)
    |
    | 각 샤드: 로컬 검색 -> 상위 N개 (ID + 점수) 반환
    |
    | [gather 단계]
    | 코디네이팅 노드: 전역 정렬 -> 최종 상위 N개 선정
    | -> 해당 샤드에 문서 내용 요청 -> 취합
    |
    v
클라이언트에 최종 응답
```

scatter 단계에서 코디네이팅 노드가 프라이머리와 레플리카 중 어느 쪽에 요청을 보낼지 결정하는 데 영향을 주는 것이 **preference 파라미터**입니다. 기본적으로 Elasticsearch는 요청을 고르게 분산합니다. 그러나 특정 상황에서는 요청을 특정 샤드 복사본으로 유도하고 싶을 수 있습니다.

`_local` 값을 지정하면, 코디네이팅 노드와 같은 노드에 있는 샤드를 우선 사용합니다. 네트워크 통신이 줄어들어 응답 시간이 단축됩니다. `_prefer_nodes:node-a,node-b`를 지정하면, 나열한 노드에 있는 샤드를 우선 사용합니다. 특정 노드에 파일시스템 캐시가 충분히 올라가 있어 해당 노드에서 검색하는 것이 유리한 경우에 활용합니다.

preference에 사용자 정의 문자열(예: 세션 ID)을 넣으면, 같은 문자열을 가진 요청은 항상 같은 샤드 복사본으로 라우팅됩니다. 이를 통해 특정 사용자의 반복 검색이 동일한 노드의 캐시를 활용하도록 만들 수 있습니다.

Elasticsearch는 preference 외에도 **adaptive replica selection**이라는 기능을 제공합니다. 이 기능은 기본적으로 활성화되어 있습니다. 코디네이팅 노드가 프라이머리와 레플리카 중 요청을 보낼 대상을 고를 때, 각 노드의 현재 상태를 고려하여 가장 빠르게 응답할 수 있는 샤드를 선택합니다. 고려하는 요소에는 해당 노드가 현재 처리 중인 검색 요청 수, 과거 요청의 응답 시간, 검색 스레드풀의 큐 크기가 포함됩니다.

예를 들어 Data-1 노드에 이미 검색 요청이 몰려 있고 Data-2 노드가 한가하다면, 같은 데이터를 가진 두 샤드 중 Data-2의 레플리카를 선택합니다. 단순한 라운드 로빈 방식보다 응답 시간 편차가 줄어들어, 전체 검색 성능이 안정적으로 유지됩니다.

정리하면, 인덱싱 요청은 코디네이팅 노드에서 라우팅 공식으로 프라이머리 샤드를 찾아 전달되고, 프라이머리 샤드에서 모든 레플리카로 병렬 복제됩니다. wait_for_active_shards 설정으로 인덱싱 시작 전 활성 샤드 수를 제어할 수 있습니다. 검색 요청은 scatter/gather 패턴을 따라, 코디네이팅 노드가 각 샤드에 요청을 분산하고 결과를 전역 정렬하여 최종 응답을 만듭니다. preference 파라미터와 adaptive replica selection이 검색 요청의 샤드 선택을 최적화합니다.

다음 단원인 5.2.1에서는 샤드 크기와 수 결정을 다룹니다. 1.3.3에서 샤드 크기의 권장 범위를 간략히 언급했는데, 그 기준의 근거와 실무 적용 전략을 구체적으로 살펴봅니다.

이 단원을 마치면 인덱싱 요청이 coordinating 노드에서 primary 샤드, replica 샤드로 처리되는 과정을 설명할 수 있으며, 검색 요청의 scatter/gather 패턴을 설명할 수 있습니다.
