# 7.2.3 ELSER 배포와 시맨틱 검색 구현

7.2.2에서 ELSER와 dense 벡터 방식을 비교하고, 상황에 따른 선택 기준을 정리했습니다. 이 단원에서는 ELSER 모델을 실제로 클러스터에 배포하고, 문서를 인덱싱한 뒤 시맨틱 검색을 수행하는 전체 과정을 다룹니다.

ELSER 모델을 배포하는 가장 간편한 방법은 **Kibana의 Machine Learning UI**를 사용하는 것입니다. Kibana 좌측 메뉴에서 Machine Learning 항목을 선택하고, Trained Models 탭으로 이동합니다. 목록에서 .elser_model_2(ELSER v2)를 찾아 Download를 클릭하면 모델 가중치가 Elasticsearch ML 모델 저장소에 저장됩니다. 다운로드가 완료되면 Deploy 버튼이 활성화됩니다. Deploy를 누르면 모델이 ML 노드 메모리에 올라갑니다.

REST API로도 동일한 작업을 수행할 수 있습니다.

```json
PUT _ml/trained_models/.elser_model_2
{
  "input": {
    "field_names": ["text_field"]
  }
}
```

이 요청으로 모델 정의를 등록한 뒤, 배포를 시작합니다.

```
POST _ml/trained_models/.elser_model_2/deployment/_start
{
  "number_of_allocations": 1,
  "threads_per_allocation": 2
}
```

7.1.3에서 다룬 임베딩 모델 배포와 구조가 동일합니다. number_of_allocations는 동시 처리량을 조절하고, threads_per_allocation은 각 인스턴스의 스레드 수를 지정합니다.

ELSER를 사용하는 방법은 크게 두 가지가 있습니다. 하나는 Elasticsearch 8.x에서 도입된 **semantic_text** 필드 타입을 사용하는 고수준 방식이고, 다른 하나는 인제스트 파이프라인과 text_expansion 쿼리를 직접 조합하는 저수준 방식입니다.

먼저 **semantic_text** 필드 타입을 살펴봅니다. semantic_text는 시맨틱 검색을 위해 설계된 필드 타입입니다. 이 필드에 텍스트를 넣으면 Elasticsearch가 자동으로 ELSER(또는 지정한 모델)를 호출하여 벡터를 생성하고 저장합니다. 별도의 인제스트 파이프라인 설정이 필요 없습니다.

```json
PUT my-semantic-index
{
  "mappings": {
    "properties": {
      "content": {
        "type": "semantic_text",
        "inference_id": ".elser_model_2"
      }
    }
  }
}
```

type을 semantic_text로 지정하고, inference_id에 배포한 ELSER 모델의 이름을 넣습니다. 이제 문서를 인덱싱하면 content 필드의 텍스트가 자동으로 ELSER를 거쳐 희소 벡터로 변환됩니다.

```json
POST my-semantic-index/_doc
{
  "content": "Elasticsearch is a distributed search and analytics engine."
}
```

검색할 때는 **semantic** 쿼리 타입을 사용합니다.

```json
GET my-semantic-index/_search
{
  "query": {
    "semantic": {
      "field": "content",
      "query": "분산 검색 엔진"
    }
  }
}
```

semantic 쿼리는 query 텍스트를 모델에 넣어 벡터를 생성하고, content 필드에 저장된 벡터와 비교하여 유사한 문서를 찾습니다. semantic_text와 semantic 쿼리의 조합은 가장 간결한 시맨틱 검색 구현 방법입니다.

다음으로 저수준 방식을 살펴봅니다. 이 방식은 인제스트 파이프라인에 **sparse_encoding 프로세서**(또는 inference 프로세서)를 설정하고, 검색 시 **text_expansion** 쿼리를 사용합니다. semantic_text가 등장하기 전에 사용하던 방식이며, 파이프라인의 다른 프로세서와 결합하는 등 세밀한 제어가 필요할 때 여전히 유용합니다.

먼저 인덱스 매핑을 만듭니다.

```json
PUT my-elser-index
{
  "mappings": {
    "properties": {
      "content": { "type": "text" },
      "ml.tokens": { "type": "rank_features" }
    }
  }
}
```

content에 원본 텍스트를, ml.tokens에 ELSER가 생성한 토큰-가중치 쌍을 저장합니다.

그 다음 인제스트 파이프라인을 만듭니다.

```json
PUT _ingest/pipeline/elser-pipeline
{
  "processors": [
    {
      "inference": {
        "model_id": ".elser_model_2",
        "input_output": [
          {
            "input_field": "content",
            "output_field": "ml.tokens"
          }
        ]
      }
    }
  ]
}
```

inference 프로세서가 content 필드의 텍스트를 ELSER 모델에 넣고, 결과를 ml.tokens 필드에 저장합니다. 7.1.3에서 다룬 임베딩 모델용 파이프라인과 구조가 같습니다. 차이점은 모델이 dense 벡터 대신 희소 벡터(토큰-가중치 쌍)를 출력한다는 것입니다.

문서 인덱싱 시 파이프라인을 지정합니다.

```json
POST my-elser-index/_doc?pipeline=elser-pipeline
{
  "content": "Elasticsearch is a distributed search and analytics engine."
}
```

검색은 text_expansion 쿼리를 사용합니다.

```json
GET my-elser-index/_search
{
  "query": {
    "text_expansion": {
      "ml.tokens": {
        "model_id": ".elser_model_2",
        "model_text": "분산 검색 엔진"
      }
    }
  }
}
```

두 방식의 전체 흐름을 비교하면 다음과 같습니다.

```
고수준 (semantic_text)           저수준 (pipeline + text_expansion)
─────────────────────────        ─────────────────────────────────
매핑: semantic_text 필드          매핑: text + rank_features 필드
파이프라인: 불필요                파이프라인: inference 프로세서 설정
인덱싱: 자동 벡터 생성            인덱싱: pipeline 파라미터 지정
검색: semantic 쿼리               검색: text_expansion 쿼리
─────────────────────────        ─────────────────────────────────
장점: 설정 간결                   장점: 세밀한 파이프라인 제어
```

ELSER를 운영할 때 주의할 점은 **추론 지연**(inference latency)입니다. 문서 인덱싱 시 ELSER 모델이 텍스트를 분석하는 시간이 추가됩니다. 입력 텍스트 길이에 따라 달라지지만, 현재 ELSER는 약 300~400단어(512토큰) 분량의 입력을 처리합니다. 이보다 긴 텍스트는 잘려나가므로, 긴 문서는 문단이나 섹션 단위로 분할하여 인덱싱하는 것이 좋습니다.

대량의 문서를 인덱싱할 때 추론 병목이 발생하면 **배포 수를 조정**합니다. number_of_allocations를 늘려 동시에 처리할 수 있는 추론 요청 수를 증가시킵니다. ML 노드의 CPU와 메모리가 충분한지 확인한 뒤 조정해야 합니다.

**인덱싱 파이프라인 최적화**도 고려할 수 있습니다. Bulk API를 사용하여 여러 문서를 한 번에 인덱싱하면, 모델 호출 횟수를 줄이고 처리량을 높일 수 있습니다. 6.2.3에서 다룬 인제스트 파이프라인의 기본 구조와 동일하게, _bulk 요청에 pipeline 파라미터를 지정하면 됩니다.

```json
POST _bulk?pipeline=elser-pipeline
{"index": {"_index": "my-elser-index"}}
{"content": "첫 번째 문서 내용입니다."}
{"index": {"_index": "my-elser-index"}}
{"content": "두 번째 문서 내용입니다."}
```

정리하면, ELSER 배포는 Kibana ML UI 또는 REST API로 수행합니다. 시맨틱 검색 구현은 semantic_text 필드와 semantic 쿼리를 사용하는 고수준 방식이 가장 간결합니다. 세밀한 파이프라인 제어가 필요하면 inference 프로세서와 text_expansion 쿼리를 조합하는 저수준 방식을 사용합니다. 운영 시에는 추론 지연을 고려하여 배포 수를 조절하고, Bulk API로 인덱싱 처리량을 높입니다.

다음 단원인 7.3.1에서는 BM25 검색과 벡터 검색의 결과를 하나로 합치는 RRF(Reciprocal Rank Fusion)의 원리를 다룹니다.

이 단원을 마치면 ELSER 모델을 배포하고 semantic_text 필드로 시맨틱 검색을 구현할 수 있습니다.
