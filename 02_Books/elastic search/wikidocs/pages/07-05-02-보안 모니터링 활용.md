# 7.5.2 보안 모니터링 활용

7.5.1에서 이상 감지 Job을 설정하고 anomaly score를 해석하는 방법을 다뤘습니다. 이상 감지는 서버 성능 모니터링 외에도 보안 분야에서 큰 가치를 발휘합니다. 공격자의 행동은 정상 사용자와 다른 패턴을 보이기 때문입니다. 평소에 없던 시간대의 로그인, 갑자기 증가한 데이터 다운로드, 처음 보는 외부 IP로의 접속 시도 같은 것들이 이상 감지 모델에 잡힐 수 있습니다. 이 단원에서는 이상 감지 결과를 Elastic Security와 연동하여 보안 위협을 탐지하는 방법을 다룹니다.

**Elastic Security**는 Elastic Stack 위에 구축된 보안 분석 솔루션입니다. SIEM(Security Information and Event Management) 기능을 제공하여, 여러 소스에서 수집한 보안 이벤트를 통합 분석합니다. 방화벽 로그, 인증 로그, 네트워크 트래픽 로그 등을 Elasticsearch에 수집하고, Kibana의 Security 앱에서 시각화하고 분석합니다.

Elastic Security에서 위협을 탐지하는 기본 방법은 **탐지 규칙**(detection rule)입니다. 탐지 규칙은 "특정 조건에 맞는 이벤트가 발생하면 알림을 생성하라"는 명령입니다. 예를 들어 "5분 안에 같은 계정에서 로그인 실패가 10회 이상이면 경고"라는 규칙을 만들 수 있습니다. 이런 규칙은 이미 알려진 공격 패턴을 탐지하는 데 효과적입니다.

그런데 알려지지 않은 새로운 공격 패턴은 규칙만으로 탐지하기 어렵습니다. 여기서 ML 이상 감지가 보완 역할을 합니다. 이상 감지는 "정상과 다른" 패턴을 찾는 것이므로, 미리 정의하지 않은 위협도 포착할 수 있습니다.

**SIEM 탐지 규칙과 이상 감지를 연동**하는 방법은 다음과 같습니다. Elastic Security의 탐지 규칙 유형 중 "Machine Learning" 유형을 선택하면, 이상 감지 Job의 결과를 기반으로 알림을 생성할 수 있습니다.

```
Kibana > Security > Rules > Create new rule
  │
  Rule type: Machine Learning
  │
  ML Job 선택: login-anomaly-job
  │
  Anomaly score 임계값 설정: 75 이상
  │
  심각도(Severity) 매핑:
    75~90 → Medium
    90~100 → Critical
  │
  알림 액션 설정: 이메일, Slack, Webhook 등
```

이 규칙은 login-anomaly-job에서 anomaly score가 75 이상인 이상이 감지되면 알림을 트리거합니다. 점수 범위에 따라 심각도를 다르게 매핑할 수 있습니다.

**Entity Analytics**는 Elastic Security의 기능 중 하나로, 사용자와 호스트 같은 엔티티(entity) 단위로 위험도를 종합 평가합니다. 한 사용자에게 여러 이상 징후가 겹치면, 개별 이벤트보다 해당 사용자의 전체 위험도를 높게 산정합니다. 예를 들어 특정 사용자 계정에서 비정상 시간대 로그인(이상 감지), 대량 파일 다운로드(탐지 규칙), 처음 보는 IP 접속(이상 감지)이 동시에 발생하면, 각각은 중간 수준의 경고이지만 합산하면 높은 위험도로 판단됩니다.

이상 징후가 감지되었을 때 **알림**(alerting)을 설정하는 방법은 두 가지가 있습니다. 첫째, Elastic Security의 탐지 규칙에 액션을 연결하는 방법입니다. 위에서 설명한 ML 규칙에 이메일, Slack, PagerDuty 등의 알림 채널을 설정합니다. 둘째, **Watcher** 또는 Stack의 **Alerting** 기능을 직접 사용하는 방법입니다. Watcher는 Elasticsearch의 내장 알림 시스템으로, .ml-anomalies-* 인덱스를 주기적으로 조회하여 조건에 맞는 이상을 발견하면 알림을 보냅니다.

Watcher로 이상 감지 알림을 설정하는 구조를 살펴봅니다.

```json
PUT _watcher/watch/anomaly-alert
{
  "trigger": {
    "schedule": { "interval": "5m" }
  },
  "input": {
    "search": {
      "request": {
        "indices": [".ml-anomalies-*"],
        "body": {
          "query": {
            "bool": {
              "filter": [
                { "term": { "job_id": "login-anomaly-job" } },
                { "range": { "anomaly_score": { "gte": 75 } } },
                { "range": { "timestamp": { "gte": "now-5m" } } }
              ]
            }
          }
        }
      }
    }
  },
  "condition": {
    "compare": { "ctx.payload.hits.total.value": { "gt": 0 } }
  },
  "actions": {
    "send_email": {
      "email": {
        "to": ["security-team@example.com"],
        "subject": "ML 이상 감지 알림: 로그인 이상",
        "body": "최근 5분 내 anomaly score 75 이상의 이상이 감지되었습니다."
      }
    }
  }
}
```

trigger는 5분마다 실행하도록 스케줄을 설정합니다. input에서 .ml-anomalies-* 인덱스를 조회하여 최근 5분 이내에 anomaly_score가 75 이상인 결과를 찾습니다. condition에서 검색 결과가 1건 이상이면 actions를 실행합니다.

보안 팀에서 실제로 이상 감지를 활용하는 대표적인 시나리오가 **위협 헌팅**(threat hunting)입니다. 위협 헌팅은 알려진 경고가 없더라도 공격의 흔적을 능동적으로 탐색하는 활동입니다. 이상 감지 결과를 출발점으로 사용합니다.

구체적인 **로그인 이상 탐지 예시**를 통해 전체 과정을 살펴봅니다.

먼저 이상 감지 Job을 설정합니다. detector 함수로 count를 사용하여 로그인 이벤트 수를 추적합니다. by_field_name을 user로 설정하여 사용자별로 분석합니다. influencer로 user와 source_ip를 지정합니다.

```json
PUT _ml/anomaly_detectors/login-anomaly-job
{
  "analysis_config": {
    "bucket_span": "15m",
    "detectors": [
      {
        "function": "count",
        "by_field_name": "user",
        "partition_field_name": "event.action"
      }
    ],
    "influencers": ["user", "source.ip"]
  },
  "data_description": {
    "time_field": "@timestamp"
  }
}
```

event.action 필드를 partition_field_name으로 지정하면, 로그인 성공과 실패를 별도로 분석합니다. 평소 하루에 2~3회 로그인하던 사용자가 갑자기 심야 시간에 수십 회 로그인을 시도하면, 해당 사용자와 시간대에 높은 anomaly score가 부여됩니다.

Kibana의 Anomaly Explorer에서 높은 점수의 이상을 발견하면, 보안 분석가는 다음 단계를 수행합니다. influencer에서 어떤 사용자, 어떤 IP에서 이상이 발생했는지 확인합니다. 해당 사용자의 다른 활동(파일 접근, 네트워크 연결 등)을 Security 앱에서 타임라인으로 조회합니다. 공격 여부를 판단하여 차단 조치를 취하거나, 거짓 양성으로 판정하여 무시합니다.

```
이상 감지 결과: user "admin" anomaly score 92
     │
     ├─ 시간: 새벽 3시 (평소 업무 시간: 9~18시)
     ├─ source.ip: 외국 IP (평소 국내 IP만 사용)
     ├─ event.action: login_failure 48회 → login_success 1회
     │
     └─ 판단: 무차별 대입 공격(brute force) 후 계정 탈취 가능성
              → 계정 잠금 + 비밀번호 초기화 + IP 차단 조치
```

이 시나리오에서 이상 감지가 없었다면, 수천 명의 사용자 로그를 수동으로 분석하여 이런 패턴을 찾아야 했을 것입니다. ML 이상 감지가 자동으로 비정상 패턴을 표면화해 주므로, 보안 팀은 높은 점수의 이상에 집중하여 효율적으로 대응할 수 있습니다.

정리하면, Elastic Security는 ML 이상 감지와 연동하여 알려지지 않은 위협까지 탐지할 수 있습니다. ML 유형의 탐지 규칙으로 anomaly score 기반 알림을 설정하고, Entity Analytics로 엔티티 단위 위험도를 평가합니다. Watcher 또는 Alerting으로 이상 징후 알림을 자동화하고, 위협 헌팅의 출발점으로 이상 감지 결과를 활용합니다.

Phase 7에서는 벡터 검색, 시맨틱 검색, 하이브리드 검색, RAG, ML 이상 감지까지 Elasticsearch의 AI 검색 기능을 다뤘습니다. 다음 Phase 8 "실전 활용"에서는 이 기능들을 전자상거래 검색, ELK 로그 분석 등 실제 서비스에 적용하는 방법을 다룹니다.

이 단원을 마치면 이상 감지 결과를 Kibana SIEM과 연동하여 보안 이상을 탐지할 수 있습니다.
