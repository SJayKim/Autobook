# 2.1.1 동적 매핑 vs 명시적 매핑

1.3.2에서 Elasticsearch 인덱스를 만들 때 settings와 mappings를 함께 지정한다는 것을 살펴보았습니다. settings가 인덱스의 물리적 동작 방식을 제어한다면, mappings는 문서의 각 필드를 어떤 타입으로 저장하고 인덱싱할지 정의하는 스키마에 해당합니다. 1.3.2에서 매핑을 정의하는 방식이 두 가지라고 짧게 언급했는데, 이 단원에서는 그 두 가지 방식인 동적 매핑과 명시적 매핑을 본격적으로 파고듭니다.

매핑이 왜 중요한지부터 짚겠습니다. Elasticsearch에서 필드 타입은 해당 필드의 검색 방식과 저장 방식을 결정합니다. 예를 들어 'title'이라는 필드가 text 타입이면 전문 검색(full-text search)이 가능하고, keyword 타입이면 정확히 일치하는 값만 찾을 수 있습니다. 'price'라는 필드가 float 타입이면 범위 검색(1000 이상, 5000 이하)이 가능하지만, 실수로 text 타입으로 지정되면 숫자 범위 검색이 동작하지 않습니다. 잘못된 타입이 한 번 할당되면 해당 인덱스에서는 수정이 불가능합니다. 1.3.2에서 설명한 것처럼, 타입을 바꾸려면 새 인덱스를 만들어 데이터를 다시 옮기는 재인덱싱이 필요합니다. 그러므로 매핑을 어떻게 정의하느냐는 인덱스의 검색 품질과 운영 안정성에 직결됩니다.

Elasticsearch는 매핑을 정의하는 두 가지 방식을 제공합니다. 하나는 **동적 매핑**(dynamic mapping)이고, 다른 하나는 **명시적 매핑**(explicit mapping)입니다. 이름이 말해 주듯, 동적 매핑은 Elasticsearch가 자동으로 타입을 판단하는 방식이고, 명시적 매핑은 사용자가 직접 타입을 지정하는 방식입니다. 이 단원에서는 동적 매핑의 작동 원리를 먼저 살펴본 뒤, 명시적 매핑으로 넘어가고, 마지막으로 두 방식의 장단점을 비교합니다.

동적 매핑부터 살펴봅니다. 동적 매핑은 인덱스에 매핑을 미리 정의하지 않은 상태에서 문서를 인덱싱할 때 발동합니다. Elasticsearch는 들어오는 JSON 문서의 각 필드 값을 분석하여 데이터 타입을 자동으로 추론하고, 그 결과를 인덱스의 매핑에 등록합니다. 예를 들어, 매핑이 비어 있는 인덱스에 다음과 같은 문서를 넣는다고 가정합니다.

```json
PUT /products/_doc/1
{
  "name": "Wireless Mouse",
  "price": 29.99,
  "in_stock": true
}
```

이 요청이 처리되면 Elasticsearch는 name 필드의 값 "Wireless Mouse"를 문자열로 판단하여 text 타입(+ keyword 멀티필드)으로 매핑합니다. price 필드의 값 29.99는 부동소수점이므로 float 타입으로, in_stock 필드의 값 true는 boolean 타입으로 매핑합니다. 사전에 아무런 스키마 정의 없이 첫 문서가 들어오는 순간 매핑이 자동으로 만들어지는 것입니다.

동적 매핑이 타입을 추론하는 규칙은 JSON 값의 형태에 따라 정해져 있습니다. 아래 표는 JSON 값과 Elasticsearch가 할당하는 타입의 대응 관계입니다.

```
JSON 값 형태              자동 매핑되는 ES 타입
-----------------------   -------------------------
null                      필드를 매핑에 추가하지 않음
true / false              boolean
정수 (long 범위)          long
부동소수점                float
중첩 객체 {}              object
배열 []                   첫 번째 null이 아닌 원소의 타입
날짜 패턴 문자열          date
그 외 문자열              text + .keyword 멀티필드
```

이 표에서 주의할 부분이 몇 가지 있습니다.

null 값은 타입을 추론할 근거가 없으므로 매핑에 아예 추가되지 않습니다. 나중에 같은 필드에 null이 아닌 값이 들어오면 그때 타입이 결정됩니다.

배열은 별도의 배열 타입이 존재하지 않습니다. Elasticsearch는 배열의 첫 번째 null이 아닌 원소를 보고 타입을 결정합니다. 예를 들어 [1, 2, 3]이 들어오면 long, ["a", "b"]가 들어오면 text + keyword 멀티필드가 됩니다.

문자열 중에서 날짜 패턴에 맞는 값은 date 타입으로 매핑됩니다. 기본적으로 ISO 8601 형식("2025-03-15")이나 "yyyy/MM/dd" 형식("2025/03/15")에 해당하는 문자열이 들어오면 Elasticsearch는 이를 날짜로 인식합니다. 이 동작을 **날짜 감지**(date detection)라고 부르며, 기본적으로 활성화되어 있습니다. 날짜처럼 보이는 문자열이 날짜가 아닌 경우에는 인덱스 설정에서 date_detection을 false로 지정하여 비활성화할 수 있습니다.

날짜가 아닌 일반 문자열은 text 타입으로 매핑됩니다. 이때 Elasticsearch는 text 필드 아래에 keyword 서브필드를 자동으로 함께 만듭니다. 예를 들어 name 필드가 text로 매핑되면 name.keyword라는 keyword 타입 서브필드가 함께 생성됩니다. text는 전문 검색용이고 keyword는 정확 일치 및 정렬용인데, 동적 매핑은 어느 쪽으로 쓸지 알 수 없으므로 둘 다 만드는 것입니다. 편리하지만, 실제로 한쪽만 필요한 경우에는 불필요한 저장 공간을 차지합니다.

한 가지 더 알아 두어야 할 규칙이 있습니다. 숫자처럼 보이는 문자열, 예를 들어 "42"는 기본적으로 문자열(text + keyword)로 처리됩니다. 숫자 타입으로 자동 추론되지 않습니다. 이런 값을 숫자로 인식시키려면 인덱스 설정에서 numeric_detection을 true로 지정해야 합니다.

동적 매핑의 동작 수준은 **dynamic** 파라미터로 제어할 수 있습니다. 이 파라미터는 인덱스 전체 또는 특정 오브젝트 필드 단위로 설정할 수 있으며, 네 가지 값을 가집니다.

```
설정값       동작
---------   --------------------------------------------------
true        새 필드를 자동으로 매핑에 추가하고 인덱싱 (기본값)
false       새 필드를 무시 - _source에는 저장되지만 검색 불가
strict      매핑에 정의되지 않은 필드가 포함된 문서를 거부
runtime     새 필드를 런타임 필드로 추가 (인덱싱 없이 쿼리 시 계산)
```

기본값은 true이므로, 별도로 설정하지 않으면 새로운 필드가 들어올 때마다 자동으로 매핑에 추가됩니다. false로 설정하면 매핑에 없는 필드가 들어와도 무시합니다. 문서 원본(_source)에는 해당 필드 값이 그대로 저장되지만, 인덱싱은 되지 않으므로 검색이나 집계에 사용할 수 없습니다. 데이터가 조용히 묻히는 셈이므로, 운영 중 중요한 필드가 누락되는 문제로 이어질 수 있습니다.

**strict**는 가장 엄격한 모드입니다. 매핑에 정의되지 않은 필드가 하나라도 포함된 문서가 들어오면 인덱싱 자체를 거부하고 오류를 반환합니다. 의도하지 않은 필드가 유입되는 것을 원천 차단할 수 있습니다.

runtime은 새 필드를 런타임 필드로 등록합니다. 런타임 필드는 데이터를 디스크에 인덱싱하지 않고, 쿼리가 실행될 때 값을 계산합니다. 인덱스 크기를 줄일 수 있지만 쿼리 시 계산 비용이 발생합니다.

dynamic 파라미터를 설정하는 예시를 봅니다.

```json
PUT /my-index
{
  "mappings": {
    "dynamic": "strict",
    "properties": {
      "name": { "type": "text" },
      "price": { "type": "float" }
    }
  }
}
```

이 요청은 my-index 인덱스를 만들면서 dynamic을 strict로 설정합니다. name과 price 두 필드만 매핑에 정의했으므로, 이 두 필드 외에 다른 필드가 포함된 문서를 인덱싱하려 하면 오류가 발생합니다.

이제 명시적 매핑을 살펴봅니다. 명시적 매핑은 인덱스를 생성할 때 각 필드의 타입을 사용자가 직접 지정하는 방식입니다. PUT 요청의 mappings.properties 안에 필드 이름과 타입을 나열합니다.

```json
PUT /my-index
{
  "mappings": {
    "properties": {
      "age":    { "type": "integer" },
      "email":  { "type": "keyword" },
      "name":   { "type": "text" }
    }
  }
}
```

이 요청은 my-index 인덱스에 세 개의 필드를 정의합니다. age는 integer 타입으로, 정수 범위 검색과 집계가 가능합니다. email은 keyword 타입으로, 정확히 일치하는 값만 검색할 수 있으며 전문 검색 분석기를 거치지 않습니다. name은 text 타입으로, 분석기가 토큰화하여 전문 검색에 사용됩니다. 이처럼 각 필드의 용도에 맞는 타입을 직접 선택하므로, 동적 매핑의 자동 추론에 의존하지 않습니다.

이미 존재하는 인덱스에 새 필드를 추가하는 것도 가능합니다. PUT /{index}/_mapping 엔드포인트를 사용합니다.

```json
PUT /my-index/_mapping
{
  "properties": {
    "employee-id": {
      "type": "keyword",
      "index": false
    }
  }
}
```

이 요청은 my-index에 employee-id라는 새 필드를 추가합니다. 타입은 keyword이고, "index": false가 지정되어 있습니다. "index": false는 이 필드 값을 _source에는 저장하되 검색 대상으로는 만들지 않겠다는 뜻입니다. 식별 번호처럼 검색할 필요 없이 저장만 하면 되는 필드에 유용합니다.

여기서 중요한 제약을 다시 강조합니다. 새 필드를 추가하는 것은 가능하지만, 이미 매핑에 등록된 필드의 타입을 변경하는 것은 불가능합니다. 예를 들어 age 필드를 integer에서 text로 바꿀 수 없습니다. 타입을 바꿔야 하는 상황이 오면, 새 인덱스를 원하는 매핑으로 생성하고 Reindex API로 기존 데이터를 옮기는 절차를 거쳐야 합니다.

매핑을 정의했으면 의도대로 적용되었는지 확인해야 합니다. **GET /{index}/_mapping** API가 이 역할을 합니다. 이 API는 인덱스의 현재 매핑 정의 전체를 JSON으로 반환합니다.

```json
GET /my-index/_mapping
```

응답은 다음과 같은 구조입니다.

```json
{
  "my-index": {
    "mappings": {
      "properties": {
        "age":   { "type": "integer" },
        "email": { "type": "keyword" },
        "name":  { "type": "text" }
      }
    }
  }
}
```

응답의 최상위 키는 인덱스 이름이고, 그 안에 mappings.properties가 필드 정의를 담고 있습니다. 동적 매핑으로 자동 생성된 필드도 이 응답에 함께 나타나므로, 문서를 인덱싱한 뒤 어떤 타입이 할당되었는지 확인하는 데 유용합니다.

매핑 전체가 아니라 특정 필드만 확인하고 싶을 때도 있습니다. 필드 수가 많은 인덱스에서 전체 매핑을 조회하면 응답이 길어지기 때문입니다. 이때는 필드 단위 조회 엔드포인트를 사용합니다.

```
GET /my-index/_mapping/field/age
```

이 요청은 age 필드의 매핑 정보만 반환합니다.

매핑 확인의 실무적인 흐름을 정리하면 다음과 같습니다. 동적 매핑을 사용하는 경우, 첫 문서를 인덱싱한 직후 GET /_mapping으로 자동 추론된 타입이 의도와 맞는지 확인합니다. 타입이 예상과 다르면, 해당 인덱스를 삭제하고 명시적 매핑으로 재생성합니다. 명시적 매핑을 사용하는 경우에도, 인덱스 생성 직후 GET /_mapping으로 정의가 올바르게 적용되었는지 검증하는 것이 좋습니다.

```
동적 매핑 사용 시 검증 흐름

1. 매핑 없이 인덱스 생성 (또는 기존 인덱스에 새 필드 문서 인덱싱)
2. GET /{index}/_mapping 으로 자동 추론 결과 확인
3. 타입이 의도와 다른 경우:
   a. 인덱스 삭제
   b. 명시적 매핑으로 인덱스 재생성
   c. 데이터 재인덱싱
```

이제 동적 매핑의 장점과 단점을 정리합니다.

장점은 명확합니다. 스키마를 미리 정의하지 않아도 문서를 즉시 인덱싱할 수 있습니다. 데이터 구조를 아직 확정하지 못한 탐색 단계나, 로그 수집 초기처럼 빠르게 데이터를 넣어 보는 상황에 적합합니다. 새 필드가 추가될 때 매핑 업데이트 API를 별도로 호출할 필요도 없습니다.

단점은 프로덕션 환경에서 드러납니다. 첫째, 타입 추론이 의도와 다를 수 있습니다. 앞서 본 것처럼 숫자가 담긴 문자열 "42"는 숫자가 아니라 문자열로 매핑됩니다. 한 번 잘못된 타입이 할당되면 해당 인덱스에서는 변경할 수 없으므로, 재인덱싱이라는 비용이 발생합니다.

둘째, **매핑 폭발**(mapping explosion)이 일어날 수 있습니다. 매핑 폭발이란 인덱스에 등록되는 필드 수가 제어 없이 급증하는 현상입니다. 예를 들어 비정형 로그 데이터를 동적 매핑으로 수집하면, 로그마다 서로 다른 필드명이 계속 유입되어 필드 수가 수천, 수만 개로 늘어날 수 있습니다. Elasticsearch는 기본적으로 인덱스당 필드 수 상한을 1,000개로 설정하고 있으며, 이 한도를 초과하면 인덱싱이 거부됩니다. 필드 수가 과도하게 많아지면 클러스터의 메모리 사용량이 급증하여 장애로 이어질 수 있습니다.

셋째, 문자열이 text + keyword 멀티필드로 자동 매핑되어 불필요한 저장 공간이 낭비됩니다. 전문 검색만 필요한 필드라면 keyword 서브필드는 필요 없고, 정확 일치만 필요한 필드라면 text 매핑은 필요 없습니다. 명시적 매핑을 사용하면 필요한 타입만 정확히 지정할 수 있습니다.

매핑 폭발을 방지하기 위한 설정도 있습니다.

```json
PUT /my-index/_settings
{
  "index.mapping.total_fields.limit": 1000,
  "index.mapping.depth.limit": 20,
  "index.mapping.nested_fields.limit": 50,
  "index.mapping.nested_objects.limit": 10000
}
```

total_fields.limit은 인덱스에 등록할 수 있는 필드 수의 상한입니다. depth.limit은 오브젝트 중첩 깊이의 상한입니다. nested_fields.limit은 nested 타입 필드 수의 상한이고, nested_objects.limit은 하나의 문서에 포함될 수 있는 중첩 객체 수의 상한입니다. 이 값들을 운영 환경에 맞게 조정하면 매핑 폭발로 인한 장애를 예방할 수 있습니다.

이러한 장단점을 종합하면, 프로덕션 환경에서는 dynamic을 strict로 설정하고 명시적 매핑을 함께 사용하는 조합이 권장됩니다. strict 모드는 매핑에 정의되지 않은 필드의 유입을 차단하므로 매핑 폭발을 원천적으로 막고, 명시적 매핑은 각 필드에 정확한 타입을 부여하여 의도치 않은 추론을 방지합니다. 반면 개발 초기나 프로토타이핑 단계에서는 동적 매핑으로 빠르게 데이터를 넣어 보고, 구조가 확정되면 명시적 매핑으로 전환하는 방식이 효율적입니다.

정리하면, 동적 매핑은 Elasticsearch가 JSON 값을 분석하여 필드 타입을 자동 추론하는 방식이고, 명시적 매핑은 사용자가 PUT API로 각 필드의 타입을 직접 정의하는 방식입니다. 동적 매핑은 빠른 시작에 유리하지만 타입 충돌, 매핑 폭발, 불필요한 멀티필드 생성이라는 위험이 있으므로, 프로덕션에서는 strict 모드와 명시적 매핑의 조합이 권장됩니다. GET /{index}/_mapping API로 현재 매핑 상태를 확인하고, 의도와 다른 타입이 발견되면 조기에 교정하는 습관이 중요합니다.

다음 단원인 2.1.2에서는 매핑 설정과 업데이트 제약을 다룹니다.

이 단원을 마치면 동적 매핑과 명시적 매핑의 차이를 설명할 수 있고, 동적 매핑이 JSON 값 형태에 따라 추론하는 타입 규칙을 열거할 수 있습니다.
