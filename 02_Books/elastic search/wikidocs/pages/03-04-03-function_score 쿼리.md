# 3.4.3 function_score 쿼리

3.4.2에서 k1과 b 파라미터를 조정하여 BM25의 TF 포화 속도와 문서 길이 정규화 강도를 도메인에 맞게 바꾸는 방법을 다루었습니다. 그런데 현실의 검색 서비스에서는 텍스트 관련성만으로 순위를 결정할 수 없는 경우가 많습니다. 예를 들어 전자상거래 사이트에서 "노트북"을 검색한 사용자에게 BM25 점수가 가장 높은 문서를 첫 번째로 보여준다고 합시다. 그 문서가 재고가 없는 단종 제품이라면 사용자 경험은 좋지 않습니다. 인기도, 신선도, 재고 여부 같은 비즈니스 신호를 점수에 반영해야 실용적인 검색 순위가 만들어집니다.

Elasticsearch는 이런 상황을 위해 **function_score 쿼리**를 제공합니다. function_score 쿼리는 기존 쿼리가 산출한 BM25 점수 위에 하나 이상의 사용자 정의 함수를 결합하여 최종 점수를 만드는 구조입니다. BM25 파라미터 튜닝이 "텍스트 관련성 계산 방식 자체를 조정"하는 것이라면, function_score는 "텍스트 관련성 점수에 비즈니스 로직을 덧입히는" 접근입니다.

function_score 쿼리의 기본 골격을 먼저 살펴봅니다.

```json
{
  "query": {
    "function_score": {
      "query": {
        "match": { "title": "노트북" }
      },
      "functions": [
        { "함수 종류": "..." },
        { "함수 종류": "..." }
      ],
      "score_mode": "multiply",
      "boost_mode": "multiply"
    }
  }
}
```

가장 바깥에 "function_score"라는 쿼리 타입을 선언합니다. 그 안에 세 부분이 있습니다. 첫째, "query" 필드에는 기존에 사용하던 검색 쿼리를 그대로 넣습니다. 위 예시에서는 title 필드에서 "노트북"을 찾는 match 쿼리를 넣었습니다. 이 쿼리가 산출하는 BM25 점수가 출발점이 됩니다. 둘째, "functions" 배열에는 점수를 보정할 함수를 하나 이상 나열합니다. 각 함수는 독립적으로 값을 계산합니다. 셋째, "score_mode"와 "boost_mode"는 여러 함수의 값을 어떻게 합치고, 합친 값을 원래 쿼리 점수와 어떻게 결합할지를 결정합니다.

이 세 부분이 동작하는 순서를 정리합니다.

```
function_score 쿼리 실행 흐름

[1단계] query 실행
        match 쿼리가 실행되어 각 문서에 BM25 점수(_score)가 매겨집니다.

[2단계] functions 각각 실행
        functions 배열에 나열된 함수가 문서마다 개별 값을 계산합니다.
        예) 함수 A --> 값 3.2
            함수 B --> 값 0.8

[3단계] score_mode로 함수 값 합산
        여러 함수의 값을 하나로 합칩니다.
        score_mode가 "multiply"이면: 3.2 * 0.8 = 2.56

[4단계] boost_mode로 원래 점수와 결합
        합산된 함수 값과 1단계의 BM25 점수를 결합합니다.
        boost_mode가 "multiply"이면: BM25점수 * 2.56 = 최종 점수
```

1단계에서 원래 쿼리가 문서별 BM25 점수를 산출합니다. 2단계에서 functions 배열에 나열된 각 함수가 문서마다 자신의 값을 계산합니다. 3단계에서 score_mode가 이 함수 값들을 하나의 숫자로 합칩니다. 4단계에서 boost_mode가 합산된 함수 값과 1단계의 BM25 점수를 최종적으로 결합합니다. 이 4단계 흐름이 function_score의 핵심 구조입니다.

이제 functions 배열에 넣을 수 있는 함수 종류를 하나씩 살펴봅니다.

**weight** 함수는 가장 단순한 함수입니다. 고정된 숫자를 반환합니다. 문서의 필드 값을 참조하지 않고, 단순히 지정한 가중치를 그대로 점수로 내놓습니다. 특정 조건에 해당하는 문서의 점수를 일괄적으로 높이거나 낮출 때 사용합니다. 예를 들어 "프리미엄" 카테고리에 속하는 문서의 점수를 2배로 높이고 싶다면, filter와 weight를 조합합니다.

```json
{
  "filter": { "term": { "category": "premium" } },
  "weight": 2
}
```

filter 조건에 맞는 문서에만 weight 값 2가 적용됩니다. 조건에 맞지 않는 문서에는 이 함수가 기여하는 값이 없습니다.

**field_value_factor** 함수는 문서에 저장된 숫자 필드의 값을 점수 계산에 직접 사용합니다. 예를 들어 "likes"라는 필드에 좋아요 수가 저장되어 있을 때, 좋아요가 많은 문서의 점수를 높이고 싶다면 이 함수를 사용합니다.

```json
{
  "field_value_factor": {
    "field": "likes",
    "factor": 1.2,
    "modifier": "log1p",
    "missing": 1
  }
}
```

"field"는 점수에 반영할 숫자 필드의 이름입니다. "factor"는 필드 값에 곱하는 계수입니다. "modifier"는 필드 값에 적용할 수학 함수입니다. "log1p"는 log(1 + 값)을 계산합니다. 좋아요가 0인 문서는 log(1 + 0) = 0이 되고, 좋아요가 100인 문서는 log(1 + 100) = 약 4.6이 됩니다. 원시 값을 그대로 쓰면 좋아요 10,000인 문서가 좋아요 10인 문서보다 1,000배 높은 점수를 받게 되는데, 로그 함수를 적용하면 이 차이가 완만해집니다. "missing"은 해당 필드가 없는 문서에 대신 사용할 기본값입니다. 1로 설정하면 필드가 없는 문서는 log(1 + 1) = 약 0.69의 값을 받습니다.

modifier에는 log1p 외에도 여러 옵션이 있습니다. "none"은 값을 그대로 사용하고, "log"는 log(값)을, "log2p"는 log(2 + 값)을, "sqrt"는 제곱근을, "square"는 제곱을, "reciprocal"은 1/값을 계산합니다. 도메인에 맞는 곡선을 선택하면 됩니다.

다음으로 **decay 함수**를 살펴봅니다. decay 함수는 숫자, 날짜, 지리 좌표 필드에서 "기준점으로부터의 거리"에 따라 점수를 감소시키는 함수입니다. 가까우면 높은 점수, 멀면 낮은 점수를 부여합니다. 쇼핑몰에서 사용자 위치에 가까운 매장을 우선 노출하거나, 뉴스 서비스에서 최신 기사를 우선 배치하는 시나리오가 대표적입니다.

decay 함수에는 세 가지 종류가 있습니다. 셋 모두 "기준점에서 멀어질수록 점수가 낮아진다"는 동작은 같지만, 점수가 줄어드는 곡선의 형태가 다릅니다.

**gauss**는 종 모양 곡선을 그립니다. 기준점 근처에서는 점수가 거의 줄지 않다가, 일정 거리 이후 급격히 떨어집니다.

**linear**는 직선으로 줄어듭니다. 거리에 비례하여 점수가 일정하게 감소합니다.

**exp**는 지수적으로 줄어듭니다. 기준점에서 조금만 벗어나도 점수가 빠르게 떨어지고, 먼 거리에서는 완만하게 줄어듭니다.

```
점수
  |
1 |*--*                          gauss (종 모양)
  | *   *
  |  *     *
  |   *       *
  |    *          *
  |      *              *
0 +-----|-----|-----------*----> 기준점으로부터의 거리
        |     |
      offset scale

1 |*                              linear (직선)
  |  *
  |    *
  |      *
  |        *
  |          *
0 +-----|-----*-------------------> 기준점으로부터의 거리

1 |*                              exp (지수)
  |*
  | *
  |   *
  |       *
  |              *
0 +-----|-----|-----------*----> 기준점으로부터의 거리
```

decay 함수를 설정할 때는 네 가지 파라미터를 지정합니다. "origin"은 기준점입니다. 날짜 필드라면 "now"(현재 시각), 지리 좌표 필드라면 사용자의 위치 좌표가 됩니다. "scale"은 origin으로부터 이 거리에서 점수가 "decay" 값까지 감소하는 범위입니다. "offset"은 origin 주위에서 점수 감소가 시작되지 않는 구간입니다. offset 이내의 거리에서는 점수가 1로 유지됩니다. "decay"는 scale 거리에서의 점수 비율입니다. 기본값은 0.5입니다. scale 지점에서 점수가 원래의 절반이 된다는 뜻입니다.

날짜 필드에 gauss decay를 적용하는 예시를 봅니다.

```json
{
  "gauss": {
    "publish_date": {
      "origin": "now",
      "scale": "10d",
      "offset": "2d",
      "decay": 0.5
    }
  }
}
```

"origin"을 "now"로 두어 현재 시각을 기준으로 삼습니다. "offset"이 "2d"이므로 현재로부터 2일 이내에 작성된 문서는 점수가 감소하지 않습니다. "scale"이 "10d"이고 "decay"가 0.5이므로, 현재로부터 2일(offset) + 10일(scale) = 12일 전에 작성된 문서의 점수가 0.5까지 감소합니다. 12일보다 더 오래된 문서는 0.5보다 낮은 점수를 받습니다. gauss 곡선을 따르므로, 2일에서 12일 사이 구간에서는 초반에 완만하게 줄다가 후반에 급격히 줄어듭니다.

**random_score** 함수는 문서마다 무작위 점수를 부여합니다. A/B 테스트에서 검색 결과의 순서를 무작위로 섞거나, 동일 점수 문서 사이의 순서를 랜덤화할 때 사용합니다. "seed"와 "field" 파라미터를 지정하면, 같은 seed와 같은 문서에 대해 항상 같은 무작위 값을 생성합니다. 같은 사용자에게 일관된 순서를 보여주면서도 서로 다른 사용자에게는 다른 순서를 보여주는 시나리오에 활용할 수 있습니다.

```json
{
  "random_score": {
    "seed": 12345,
    "field": "_seq_no"
  }
}
```

"seed"를 사용자 ID 등 고유한 값으로 설정하면, 해당 사용자가 같은 검색을 반복해도 순서가 바뀌지 않습니다. 다른 사용자는 다른 seed를 가지므로 다른 순서를 보게 됩니다.

마지막으로 **script_score** 함수를 살펴봅니다. script_score는 Elasticsearch의 Painless 스크립팅 언어로 점수 계산 로직을 직접 작성하는 함수입니다. 앞서 소개한 weight, field_value_factor, decay, random_score가 제공하는 패턴으로 표현할 수 없는 복잡한 계산이 필요할 때 사용합니다.

Painless는 Elasticsearch 전용 스크립팅 언어입니다. 정적 타입 기반이며, 실행 시 JVM 바이트코드로 컴파일되어 네이티브 자바 코드에 가까운 성능을 냅니다. 스크립트 안에서 doc['필드명'].value 형태로 문서의 필드 값에 접근하고, _score 변수로 원래 쿼리의 BM25 점수를 참조할 수 있습니다.

좋아요 수의 로그값과 BM25 점수를 합산하는 예시입니다.

```json
{
  "script_score": {
    "script": {
      "source": "Math.log(1 + doc['likes'].value) + _score"
    }
  }
}
```

"source" 필드에 Painless 코드를 문자열로 넣습니다. Math.log(1 + doc['likes'].value)는 좋아요 수에 1을 더한 뒤 자연로그를 취합니다. 여기에 _score(BM25 점수)를 더하여 최종 함수 값을 만듭니다.

여러 신호를 조합하는 더 복잡한 예시도 가능합니다.

```json
{
  "script_score": {
    "script": {
      "source": "params.likes_weight * doc['likes'].value + params.views_weight * doc['views'].value",
      "params": {
        "likes_weight": 0.5,
        "views_weight": 0.3
      }
    }
  }
}
```

"params" 객체를 통해 스크립트 외부에서 가중치를 주입합니다. 좋아요에 0.5를, 조회수에 0.3을 곱한 뒤 더합니다. 가중치를 params로 분리하면 스크립트 코드 자체는 변경하지 않고 params 값만 바꿔서 다른 가중치를 시험할 수 있습니다. Elasticsearch는 동일한 스크립트 코드를 캐시하여 재컴파일을 피하므로, params 분리는 성능에도 유리합니다.

script_score를 사용할 때 주의할 점이 있습니다. 스크립트는 Lucene의 쿼리 최적화를 우회합니다. 필터 없이 전체 문서에 스크립트를 실행하면 성능이 크게 저하될 수 있습니다. 반드시 function_score의 query 부분이나 filter로 대상 문서를 먼저 좁힌 뒤 스크립트를 적용해야 합니다. 또한 doc['필드명'].value 방식으로 필드에 접근하는 것이 params._source['필드명']보다 훨씬 빠릅니다. doc 방식은 메모리에 미리 로드된 필드 데이터를 읽고, _source 방식은 저장된 원본 JSON을 파싱하기 때문입니다.

이제 여러 함수를 동시에 사용할 때 함수 값들을 합치는 방법인 **score_mode**를 살펴봅니다. score_mode는 functions 배열의 함수들이 각각 산출한 값을 하나의 숫자로 합치는 방식을 결정합니다. 위 실행 흐름의 3단계에 해당합니다.

score_mode에는 여섯 가지 옵션이 있습니다. "multiply"는 함수 값들을 모두 곱합니다. 기본값입니다. "sum"은 함수 값들을 모두 더합니다. "avg"는 함수 값들의 평균을 구합니다. "max"는 함수 값 중 가장 큰 값을 선택합니다. "min"은 함수 값 중 가장 작은 값을 선택합니다. "first"는 첫 번째로 filter 조건에 맞는 함수의 값을 사용합니다.

```
score_mode 옵션 정리

옵션       | 계산 방식                      | 사용 시나리오
-----------|-------------------------------|--------------------------------
multiply   | 함수A * 함수B * ...           | 각 함수가 배율로 작용할 때 (기본값)
sum        | 함수A + 함수B + ...           | 각 함수의 기여를 독립적으로 더할 때
avg        | (함수A + 함수B + ...) / 개수   | 함수 개수에 따른 편향을 줄일 때
max        | max(함수A, 함수B, ...)         | 가장 강한 신호 하나로 결정할 때
min        | min(함수A, 함수B, ...)         | 가장 약한 신호가 병목이 될 때
first      | 첫 번째 매칭 함수의 값         | 우선순위 기반 로직을 구현할 때
```

예를 들어 "인기도 함수"가 3.0을, "신선도 함수"가 0.8을 반환했다면, multiply 모드에서는 3.0 * 0.8 = 2.4가, sum 모드에서는 3.0 + 0.8 = 3.8이 됩니다.

**boost_mode**는 score_mode로 합산된 함수 값과 원래 쿼리의 BM25 점수를 결합하는 방식입니다. 실행 흐름의 4단계에 해당합니다.

boost_mode에도 여섯 가지 옵션이 있습니다. "multiply"는 BM25 점수에 함수 값을 곱합니다. 기본값입니다. "replace"는 BM25 점수를 무시하고 함수 값만 최종 점수로 사용합니다. "sum"은 BM25 점수와 함수 값을 더합니다. "avg"는 BM25 점수와 함수 값의 평균을 구합니다. "max"는 BM25 점수와 함수 값 중 큰 쪽을 선택합니다. "min"은 BM25 점수와 함수 값 중 작은 쪽을 선택합니다.

```
boost_mode 옵션 정리

옵션       | 계산 방식                     | 효과
-----------|------------------------------|-------------------------------------
multiply   | BM25점수 * 함수값             | 텍스트 관련성과 비즈니스 신호를 곱함 (기본값)
replace    | 함수값                        | 텍스트 관련성을 완전히 무시
sum        | BM25점수 + 함수값             | 텍스트 관련성에 비즈니스 신호를 가산
avg        | (BM25점수 + 함수값) / 2       | 양쪽의 균형을 맞춤
max        | max(BM25점수, 함수값)         | 둘 중 높은 점수 채택
min        | min(BM25점수, 함수값)         | 둘 중 낮은 점수 채택
```

boost_mode 선택은 "텍스트 관련성을 얼마나 존중할 것인가"의 문제입니다. multiply(기본값)를 사용하면 BM25 점수가 높은 문서가 함수 값의 이점을 더 크게 받습니다. 텍스트 관련성이 기본 축이 되고 비즈니스 신호가 가감 역할을 합니다. replace를 사용하면 텍스트 관련성을 완전히 무시하고 함수 값만으로 순위를 결정합니다. 텍스트 매칭은 필터 역할만 하고 순서는 비즈니스 로직이 전적으로 결정하는 시나리오에 적합합니다. sum을 사용하면 BM25 점수에 함수 값을 더하므로, BM25 점수가 낮더라도 함수 값이 충분히 크면 상위에 오를 수 있습니다.

여러 함수를 조합한 전체 예시를 봅니다.

```json
{
  "query": {
    "function_score": {
      "query": {
        "match": { "title": "노트북" }
      },
      "functions": [
        {
          "filter": { "term": { "category": "premium" } },
          "weight": 2
        },
        {
          "field_value_factor": {
            "field": "likes",
            "modifier": "log1p",
            "missing": 1
          }
        },
        {
          "gauss": {
            "publish_date": {
              "origin": "now",
              "scale": "30d",
              "offset": "7d",
              "decay": 0.5
            }
          }
        }
      ],
      "score_mode": "sum",
      "boost_mode": "multiply"
    }
  }
}
```

이 쿼리의 동작을 단계별로 따라가 봅니다. 먼저 match 쿼리가 title 필드에서 "노트북"이 포함된 문서를 찾고 BM25 점수를 매깁니다. 다음으로 세 함수가 각각 값을 계산합니다. 첫 번째 함수는 category가 "premium"인 문서에 2를 부여합니다. 해당하지 않는 문서는 이 함수의 기여가 0입니다. 두 번째 함수는 likes 필드의 로그값을 계산합니다. likes가 100이면 log(1 + 100) = 약 4.6입니다. 세 번째 함수는 publish_date가 현재에 가까울수록 높은 값을 부여합니다. 7일 이내 문서는 1.0, 37일(7 + 30) 전 문서는 0.5, 더 오래된 문서는 0.5 미만입니다.

score_mode가 "sum"이므로 세 함수 값을 더합니다. premium 카테고리의 100 좋아요, 3일 전 문서라면: 2 + 4.6 + 1.0 = 7.6입니다. 일반 카테고리의 10 좋아요, 60일 전 문서라면: 0 + 2.4 + 0.2 = 약 2.6입니다. boost_mode가 "multiply"이므로 이 함수 합산값에 BM25 점수를 곱합니다. BM25 점수가 비슷한 두 문서라도, 프리미엄이면서 인기 있고 최신인 문서가 훨씬 높은 최종 점수를 받게 됩니다.

정리하면, function_score 쿼리는 BM25가 산출한 텍스트 관련성 점수 위에 비즈니스 로직을 결합하는 구조입니다. functions 배열에 weight, field_value_factor, gauss/linear/exp decay, random_score, script_score 등 다양한 함수를 나열하고, score_mode로 함수 값들을 합산한 뒤, boost_mode로 원래 쿼리 점수와 결합합니다. 간단한 가중치 부여에는 weight를, 숫자 필드 기반 점수 조정에는 field_value_factor를, 거리/시간 기반 감쇠에는 decay 함수를, 완전한 맞춤 로직에는 script_score를 선택합니다.

다음 단원인 3.4.4에서는 Painless 스크립팅을 다룹니다. script_score에서 간략히 소개한 Painless 언어의 문법, 타입 시스템, 제어 흐름, 그리고 성능 최적화 방법을 본격적으로 살펴봅니다.

이 단원을 마치면 function_score 쿼리로 비즈니스 로직을 스코어에 반영할 수 있고, gauss, linear, exp decay 함수를 적용할 수 있습니다.
