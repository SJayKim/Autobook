# 5.2.2 Rollover, Shrink, Split, Clone

5.2.1에서 샤드 크기를 10~50GB로 유지해야 하는 이유와, 프라이머리 샤드 수가 인덱스 생성 후 변경 불가하다는 제약을 살펴보았습니다. 그런데 시계열 데이터(로그, 메트릭)처럼 끊임없이 쌓이는 데이터를 하나의 인덱스에 계속 넣으면, 인덱스 크기가 무한정 커져 샤드당 크기가 권장 범위를 넘어갑니다. 반대로 처음에 샤드 수를 과하게 잡아 오버샤딩이 된 인덱스도 있을 수 있습니다. 이 단원에서는 인덱스의 샤드 구조를 사후에 조정하는 네 가지 API를 다룹니다.

첫 번째는 **Rollover API**입니다. Rollover는 '넘기다'라는 뜻으로, 현재 인덱스가 일정 조건에 도달하면 자동으로 새 인덱스를 만들어 쓰기 대상을 전환하는 기능입니다. 기존 인덱스는 그대로 유지되고 읽기 전용이 되며, 새로 만들어진 인덱스가 이후 문서를 받습니다.

Rollover가 동작하려면 전제 조건이 있습니다. 인덱스가 2.4.4에서 다룬 별칭(alias)을 통해 접근되고 있어야 하며, 해당 별칭에 is_write_index가 설정되어 있어야 합니다. 또는 데이터 스트림(data stream)을 사용하고 있어야 합니다. Rollover는 별칭이 가리키는 쓰기 대상을 교체하는 방식으로 동작하기 때문입니다.

Rollover를 실행할 때 조건을 지정할 수 있습니다.

```json
POST /logs-write/_rollover
{
  "conditions": {
    "max_age": "7d",
    "max_docs": 10000000,
    "max_size": "50gb"
  }
}
```

이 요청에서 logs-write는 별칭 이름입니다. conditions에 세 가지 조건이 들어 있습니다. **max_age**는 인덱스가 생성된 후 경과 시간입니다. "7d"는 7일을 뜻합니다. **max_docs**는 인덱스에 저장된 문서 수입니다. 10,000,000은 천만 건입니다. **max_size**는 인덱스의 프라이머리 샤드 전체 크기입니다. "50gb"는 50기가바이트입니다.

이 세 조건 중 하나라도 충족되면 롤오버가 실행됩니다. 조건을 모두 생략하면 무조건 즉시 롤오버합니다.

롤오버가 실행되면 Elasticsearch는 새 인덱스를 생성합니다. 새 인덱스의 이름은 기존 인덱스 이름의 끝 번호를 하나 올린 형태입니다. 예를 들어 기존 인덱스가 logs-000001이면 새 인덱스는 logs-000002가 됩니다. 별칭의 is_write_index가 새 인덱스로 이동하여, 이후 문서는 새 인덱스에 기록됩니다. 기존 인덱스는 여전히 별칭에 연결되어 있으므로 검색 대상에는 포함됩니다.

```
Rollover 동작 흐름

[롤오버 전]
별칭: logs-write
  +---> logs-000001 (is_write_index: true, 크기: 48GB)

[롤오버 후]
별칭: logs-write
  +---> logs-000001 (is_write_index: false, 읽기 전용)
  +---> logs-000002 (is_write_index: true, 새 문서 수신)
```

두 번째는 **Shrink API**입니다. Shrink는 기존 인덱스의 프라이머리 샤드 수를 줄여 새 인덱스를 만드는 기능입니다. 오버샤딩된 인덱스나, 롤오버 후 더 이상 쓰기가 없는 인덱스의 샤드 수를 축소할 때 사용합니다.

Shrink를 실행하려면 전제 조건이 있습니다. 원본 인덱스가 읽기 전용(read-only)이어야 하고, 원본 인덱스의 모든 샤드가 하나의 노드에 모여 있어야 합니다. 이 조건을 만족시키기 위해 먼저 설정을 변경합니다.

```json
PUT /logs-000001/_settings
{
  "settings": {
    "index.routing.allocation.require._name": "data-node-1",
    "index.blocks.write": true
  }
}
```

첫 번째 설정은 이 인덱스의 모든 샤드를 data-node-1이라는 노드로 이동시킵니다. 두 번째 설정은 인덱스에 쓰기를 차단합니다. 샤드 이동이 완료되면 Shrink를 실행합니다.

```json
POST /logs-000001/_shrink/logs-000001-shrunk
{
  "settings": {
    "index.number_of_shards": 1,
    "index.number_of_replicas": 1
  }
}
```

이 요청은 logs-000001의 6개 프라이머리 샤드를 1개로 축소하여 logs-000001-shrunk라는 새 인덱스를 만듭니다. 축소 후 샤드 수에는 규칙이 있습니다. 새 샤드 수는 원본 샤드 수의 약수(factor)여야 합니다. 원본이 6개라면 1, 2, 3, 6으로만 축소할 수 있습니다. 4나 5로는 불가능합니다. 이 규칙은 원본 샤드의 데이터를 새 샤드에 균등하게 배분하기 위한 것입니다.

Shrink는 데이터를 복사하지 않고 Lucene 세그먼트를 하드 링크하므로, 대용량 인덱스도 빠르게 처리됩니다. 다만 하드 링크가 완료될 때까지는 원본 인덱스를 삭제하면 안 됩니다.

세 번째는 **Split API**입니다. Split은 Shrink의 반대로, 프라이머리 샤드 수를 늘려 새 인덱스를 만드는 기능입니다.

```json
POST /my-index/_split/my-index-split
{
  "settings": {
    "index.number_of_shards": 4
  }
}
```

이 요청은 my-index의 프라이머리 샤드를 4개로 분할하여 my-index-split이라는 새 인덱스를 만듭니다. Split에도 규칙이 있습니다. 새 샤드 수는 원본 샤드 수의 배수여야 합니다. 원본이 2개라면 4, 6, 8 등으로 분할할 수 있지만, 3이나 5로는 불가능합니다.

Split 역시 원본 인덱스가 읽기 전용이어야 합니다. Split 전에 `index.blocks.write`를 true로 설정해야 합니다.

네 번째는 **Clone API**입니다. Clone은 인덱스를 동일한 샤드 수로 복사하는 기능입니다.

```json
POST /my-index/_clone/my-index-copy
```

이 요청은 my-index와 동일한 샤드 수, 동일한 데이터를 가진 my-index-copy를 만듭니다. Clone은 매핑이나 설정을 변경하지 않고 인덱스를 그대로 복제할 때 사용합니다. 백업이나 테스트 목적으로 활용됩니다. Shrink, Split과 마찬가지로 Lucene 세그먼트를 하드 링크하므로 빠릅니다.

이 네 가지 API 모두 원본 데이터를 전부 다시 색인(reindex)하지 않고 Lucene 세그먼트를 직접 복사하거나 링크합니다. 따라서 전체 데이터를 재처리하는 Reindex API보다 빠르고 자원 소모가 적습니다.

마지막으로 **Rollover와 별칭, ILM의 연계 패턴**을 살펴봅니다. 시계열 데이터를 운영할 때, Rollover만 단독으로 사용하면 관리자가 직접 롤오버 명령을 주기적으로 실행해야 합니다. 이 수동 과정을 자동화하는 것이 ILM(Index Lifecycle Management)입니다. ILM 정책에서 hot 단계에 rollover 액션을 설정하면, Elasticsearch가 조건을 주기적으로 확인하고 자동으로 롤오버를 실행합니다.

전체 흐름을 단계별로 정리합니다.

```
Rollover + 별칭 + ILM 연계 패턴

1. 인덱스 템플릿 생성
   - index.lifecycle.name: ILM 정책 이름
   - index.lifecycle.rollover_alias: 별칭 이름

2. 초기 인덱스 생성 (logs-000001)
   - 별칭(logs-write) 연결, is_write_index: true

3. ILM 정책의 hot 단계에서 rollover 조건 감시

4. 조건 충족 시 자동 롤오버
   - logs-000002 생성, 별칭의 쓰기 대상 전환

5. ILM 정책의 warm 단계에서 shrink 실행
   - logs-000001의 샤드 수 축소

6. ILM 정책의 cold/delete 단계에서
   - 오래된 인덱스를 cold 티어로 이동 또는 삭제
```

이 패턴에서 별칭은 애플리케이션의 접근 경로를 고정하는 역할을 합니다. 애플리케이션은 항상 logs-write라는 이름으로 데이터를 보냅니다. 내부에서 인덱스가 교체되든 축소되든, 애플리케이션 코드는 변경할 필요가 없습니다. ILM은 이 전환을 자동으로 수행하여 관리 부담을 줄입니다. ILM 정책의 상세 설정은 6.1.1에서 다룹니다.

정리하면, Rollover API는 인덱스 크기가 일정 조건에 도달하면 새 인덱스로 쓰기 대상을 전환합니다. Shrink API는 샤드 수를 원본의 약수로 축소하고, Split API는 배수로 확장합니다. Clone API는 동일한 샤드 수로 인덱스를 복사합니다. 이 네 API 모두 Lucene 세그먼트를 직접 복사하여 전체 재색인보다 효율적입니다. 시계열 데이터 운영에서는 Rollover와 별칭, ILM을 결합하여 인덱스 교체를 자동화합니다.

다음 단원인 5.2.3에서는 Allocation Awareness를 다룹니다. 샤드가 물리적으로 어느 노드에 배치되는지를 제어하여, 특정 랙이나 가용 영역의 장애에도 데이터 가용성을 유지하는 방법을 살펴봅니다.

이 단원을 마치면 Rollover API로 인덱스를 자동 롤오버하는 조건을 설정할 수 있으며, Shrink API로 샤드 수를 줄이는 전제 조건과 절차를 수행할 수 있습니다.
