# 5.1.2 클러스터 아키텍처

5.1.1에서 Elasticsearch 노드가 master-eligible, data, ingest, coordinating 등 다양한 역할을 맡을 수 있다는 것을 살펴보았습니다. 그중에서도 master-eligible 노드는 클러스터 전체를 관리하는 마스터 노드로 선출될 자격을 가진 노드였습니다. 그런데 master-eligible 노드가 여러 대 있을 때, 어떤 노드가 실제 마스터가 되는지, 마스터 노드에 장애가 생기면 어떻게 되는지, 클러스터가 둘로 갈라지는 위험한 상황은 어떻게 방지하는지를 아직 다루지 않았습니다. 이 단원에서는 마스터 선출 메커니즘과 클러스터 구성 전략을 살펴봅니다.

클러스터가 처음 시작될 때, 각 노드는 아직 서로를 알지 못합니다. 노드가 다른 노드를 발견하고 하나의 클러스터로 합류하는 과정을 **디스커버리**(discovery)라고 합니다. 이 과정의 출발점은 **discovery.seed_hosts** 설정입니다.

```yaml
discovery.seed_hosts: ["node-a:9300", "node-b:9300", "node-c:9300"]
```

이 설정은 노드가 시작될 때 접촉할 후보 노드의 주소 목록입니다. 노드는 이 목록에 있는 주소로 연결을 시도하여 기존 클러스터를 찾거나, 새 클러스터를 구성하기 위한 동료 노드를 탐색합니다. 모든 master-eligible 노드의 주소를 포함하는 것이 일반적입니다.

클러스터를 처음 구성할 때는 한 가지 설정이 더 필요합니다. **cluster.initial_master_nodes**는 최초 마스터 선출에 참여할 노드 이름을 지정합니다.

```yaml
cluster.initial_master_nodes: ["node-a", "node-b", "node-c"]
```

이 설정은 클러스터를 최초로 부트스트랩(bootstrap)할 때만 사용됩니다. 부트스트랩이란 클러스터가 처음 형성되는 순간을 뜻합니다. 아직 마스터가 존재하지 않는 상태에서, 어떤 노드들이 첫 번째 선거에 참여할 자격이 있는지를 알려 주는 역할입니다. 클러스터가 한 번 형성되고 나면 이 설정은 더 이상 사용되지 않으므로, 이후 설정 파일에서 제거해도 됩니다. 오히려 남겨 두면 노드 재시작 시 혼란을 일으킬 수 있으므로 제거하는 것이 권장됩니다.

디스커버리와 부트스트랩을 거쳐 노드들이 서로를 인식하면, **마스터 선출**(master election)이 시작됩니다. Elasticsearch 7.0부터 도입된 선출 방식은 **Quorum 기반 합의**입니다.

Quorum이라는 단어는 '의결 정족수'를 뜻합니다. 회의에서 안건을 의결하려면 참석자의 과반수가 동의해야 하는 것과 같은 원리입니다. Elasticsearch에서 Quorum은 master-eligible 노드의 과반수를 의미합니다. 마스터 선출이 유효하려면, master-eligible 노드 중 과반 이상이 특정 후보에 동의해야 합니다.

Quorum을 수식으로 표현하면 다음과 같습니다.

```
Quorum = (master-eligible 노드 수 / 2) + 1
```

master-eligible 노드가 3대면 Quorum은 2입니다. 3대 중 2대 이상이 한 후보에 동의하면 그 후보가 마스터로 선출됩니다. 5대면 Quorum은 3이고, 3대 이상의 동의가 필요합니다.

이 Quorum 메커니즘이 해결하는 핵심 문제가 **split-brain**입니다. split-brain이란 네트워크 장애로 클러스터가 두 그룹으로 분리되었을 때, 각 그룹이 독자적으로 마스터를 선출하여 서로 다른 두 개의 클러스터가 되어 버리는 현상입니다. 두 마스터가 각각 다른 판단으로 인덱스를 생성하거나 샤드를 재배치하면, 나중에 네트워크가 복구되었을 때 데이터 불일치가 발생합니다. 이는 매우 심각한 문제입니다.

구체적인 시나리오를 들어 봅니다. master-eligible 노드가 3대(A, B, C)이고, 네트워크 단절로 A만 분리되었다고 합시다.

```
네트워크 단절 시 Quorum 동작

   [A]  |  [B, C]
         |
  1대    |  2대
         |
  Quorum = 2 이상 필요
         |
  A 혼자서는 마스터    B와 C는 2대이므로
  선출 불가 (1 < 2)    마스터 선출 가능 (2 >= 2)
```

A는 혼자이므로 Quorum(2)을 충족하지 못하여 마스터를 선출할 수 없습니다. B와 C는 2대이므로 Quorum을 충족하여 마스터를 선출합니다. 결과적으로 클러스터에는 마스터가 하나만 존재하게 됩니다. split-brain이 방지된 것입니다.

Elasticsearch 7.0 이전에는 Quorum을 자동으로 관리하지 않았습니다. 사용자가 직접 `discovery.zen.minimum_master_nodes`라는 설정을 통해 Quorum 값을 수동으로 지정해야 했습니다. 이 값을 잘못 설정하면 split-brain이 발생할 수 있었습니다. 예를 들어 master-eligible 노드가 3대인데 minimum_master_nodes를 1로 설정하면, 네트워크 단절 시 분리된 1대만으로도 마스터를 선출할 수 있어 split-brain이 일어났습니다. 7.0부터는 Elasticsearch가 master-eligible 노드 수를 자동으로 추적하고 Quorum을 계산하므로, 이 설정은 제거되었습니다.

master-eligible 노드 중에는 **투표 전용 노드**(voting_only)라는 특수한 유형이 있습니다. 투표 전용 노드는 마스터 선출 투표에는 참여하지만, 스스로 마스터가 될 수는 없는 노드입니다.

```yaml
node.roles: [ master, voting_only ]
```

이 노드는 투표권은 있지만 당선 자격은 없는 셈입니다. 투표 전용 노드가 필요한 이유는 Quorum 수를 조정하면서도 마스터 후보의 수는 제한하고 싶을 때입니다. 예를 들어 master-eligible 노드가 짝수 개일 때, 투표 전용 노드를 추가하여 홀수로 만들면 타이브레이커(tiebreaker) 역할을 수행합니다.

프로덕션 환경에서 권장되는 구성은 **전용 master-eligible 노드 3대**입니다. 이 구성의 근거를 정리합니다.

3대면 Quorum은 2입니다. 1대가 장애를 일으켜도 나머지 2대가 Quorum을 충족하므로 새 마스터를 선출할 수 있습니다. 즉, 1대까지의 동시 장애를 견딜 수 있습니다. 2대로 구성하면 Quorum이 2이므로, 1대만 장애가 나도 Quorum을 충족하지 못하여 마스터를 선출할 수 없습니다. 사실상 장애 내성이 없는 것입니다. 4대 이상은 가능하지만, 3대로 이미 1대 장애 허용이 되므로 비용 대비 이점이 크지 않습니다. 5대로 구성하면 2대 동시 장애를 허용할 수 있어 더 높은 가용성이 필요한 환경에서 선택합니다.

전용 마스터 노드 3대가 하는 일은 오직 클러스터 관리뿐입니다. 데이터를 저장하지 않으므로 디스크 I/O 부담이 없고, 검색이나 인덱싱을 처리하지 않으므로 CPU 경합이 없습니다. 클러스터 메타데이터 관리와 샤드 할당 결정에만 자원을 집중합니다.

클러스터 상태를 확인하는 API인 **_cluster/state**를 살펴봅니다. 이 API는 클러스터의 현재 상태 전체를 반환합니다.

```
GET _cluster/state
```

반환되는 정보에는 클러스터 이름, 클러스터 UUID, 현재 선출된 마스터 노드의 ID, 모든 노드 목록과 각 노드의 역할, 모든 인덱스의 메타데이터, 샤드 할당 정보가 포함됩니다. 전체 상태 정보의 양이 매우 크므로, 특정 부분만 조회하는 것도 가능합니다.

```
GET _cluster/state/nodes
```

이 요청은 클러스터 상태 중 노드 정보만 반환합니다. 각 노드의 이름, 전송 주소, 할당된 역할을 확인할 수 있습니다.

아래 다이어그램은 전용 마스터 3대와 데이터 노드로 구성된 프로덕션 권장 토폴로지를 보여 줍니다.

```
프로덕션 권장 클러스터 토폴로지

+----------+    +----------+    +----------+
| Master-1 |    | Master-2 |    | Master-3 |
| (master) |    | (master) |    | (master) |
+----+-----+    +----+-----+    +----+-----+
     |               |               |
     +-------+-------+-------+-------+
             |               |
        +----+----+     +----+----+     +----------+
        | Data-1  |     | Data-2  |     | Data-3   |
        | (data)  |     | (data)  |     | (data)   |
        +---------+     +---------+     +----------+

Master-1~3: 전용 master-eligible 노드 (데이터 저장 안 함)
Data-1~3:   전용 data 노드 (샤드 보유, 검색/인덱싱 처리)
```

마스터 노드 3대는 서로 통신하며 클러스터 상태를 합의합니다. 데이터 노드는 마스터 노드의 지시에 따라 샤드를 저장하고 검색 요청을 처리합니다.

정리하면, Elasticsearch 클러스터는 discovery.seed_hosts로 노드를 발견하고, cluster.initial_master_nodes로 최초 부트스트랩을 수행합니다. 마스터 선출은 Quorum 기반 합의로 이루어지며, master-eligible 노드의 과반 이상이 동의해야 마스터가 선출됩니다. 이 메커니즘이 split-brain을 방지합니다. 프로덕션 환경에서는 전용 master-eligible 노드 3대를 구성하여, 1대 장애에도 클러스터가 정상 운영되도록 합니다. _cluster/state API로 클러스터 상태를 조회할 수 있습니다.

다음 단원인 5.1.3에서는 분산 읽기/쓰기 흐름을 다룹니다. 클라이언트의 인덱싱 요청이 어떻게 프라이머리 샤드에 도달하고, 검색 요청이 어떻게 여러 샤드에 분산되어 결과가 합쳐지는지를 살펴봅니다.

이 단원을 마치면 마스터 선출 과정과 split-brain 방지 메커니즘을 설명할 수 있으며, 프로덕션 권장 클러스터 토폴로지를 설계할 수 있습니다.
