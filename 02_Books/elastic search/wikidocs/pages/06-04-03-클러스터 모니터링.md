# 6.4.3 클러스터 모니터링

Elasticsearch 클러스터를 운영하면서 문제가 발생한 뒤에야 대응하면 이미 서비스에 영향이 간 상태입니다. 디스크가 가득 차서 인덱싱이 멈추거나, JVM 힙이 부족하여 노드가 응답하지 않거나, 검색 요청이 거부되는 상황은 사전에 징후를 감지할 수 있습니다. 모니터링은 클러스터의 상태와 성능 지표를 지속적으로 수집하고, 이상 징후를 조기에 발견하여 장애를 예방하는 활동입니다.

Elasticsearch 클러스터의 가장 기본적인 상태 지표는 **클러스터 상태(cluster health)**입니다. _cluster/health API로 확인합니다.

```json
GET _cluster/health
```

응답에는 status 필드가 포함되며, 세 가지 값 중 하나를 반환합니다.

**green**은 모든 프라이머리 샤드와 레플리카 샤드가 정상적으로 할당된 상태입니다. 클러스터가 완전히 건강합니다.

**yellow**는 모든 프라이머리 샤드는 할당되었지만, 일부 레플리카 샤드가 할당되지 않은 상태입니다. 데이터 자체는 사용 가능하지만, 해당 레플리카가 없는 샤드에 노드 장애가 발생하면 데이터 유실 위험이 있습니다. 단일 노드 클러스터에서는 레플리카를 배치할 다른 노드가 없으므로 yellow가 정상입니다.

**red**는 하나 이상의 프라이머리 샤드가 할당되지 않은 상태입니다. 해당 샤드에 속한 데이터를 검색하거나 인덱싱할 수 없습니다. 즉시 원인을 파악하고 조치해야 합니다.

클러스터 상태가 yellow나 red일 때 원인을 파악하려면 _cluster/allocation/explain API를 사용합니다.

```json
GET _cluster/allocation/explain
```

이 API는 할당되지 않은 샤드가 왜 배치되지 못했는지를 상세히 설명합니다. 디스크 부족, 노드 필터 규칙, allocation awareness 제약 등 구체적인 이유를 알려 줍니다.

노드 수준의 상세 지표는 _cat API와 _nodes/stats API로 확인합니다.

```json
GET _cat/nodes?v&h=name,heap.percent,ram.percent,cpu,load_1m,disk.used_percent
```

이 요청은 각 노드의 이름, 힙 사용률, RAM 사용률, CPU 사용률, 1분 평균 부하, 디스크 사용률을 표 형태로 보여 줍니다.

모니터링해야 하는 핵심 지표를 범주별로 정리하면 다음과 같습니다.

**JVM 힙 사용률**은 가장 중요한 지표 중 하나입니다. 6.3.3에서 다룬 것처럼 힙이 부족하면 빈번한 GC가 발생하고, 극단적인 경우 OutOfMemoryError로 노드가 멈춥니다. 힙 사용률이 지속적으로 75%를 넘으면 주의가 필요하고, 85%를 넘으면 긴급 조치가 필요합니다.

**인덱싱/검색 처리량 지표**는 클러스터가 처리하는 작업량을 보여 줍니다. _nodes/stats API에서 indexing.index_total(총 인덱싱 건수), search.query_total(총 검색 건수), search.query_time_in_millis(총 검색 소요 시간)를 확인합니다. 이 값들의 변화 추이를 관찰하면 성능 변동을 감지할 수 있습니다.

**스레드 풀 거부(rejected) 지표**는 클러스터가 과부하 상태인지를 판단하는 중요한 신호입니다. Elasticsearch는 인덱싱, 검색 등 작업별로 스레드 풀을 운영합니다. 스레드 풀의 큐가 가득 차면 새 요청을 거부합니다.

```json
GET _cat/thread_pool/write,search?v&h=node_name,name,active,rejected
```

rejected 수가 증가하고 있다면, 클러스터가 현재 요청량을 감당하지 못하고 있다는 뜻입니다. 노드를 추가하거나, 요청량을 줄이거나, 쿼리를 최적화해야 합니다.

**디스크 워터마크**는 Elasticsearch가 디스크 사용량에 따라 자동으로 취하는 보호 조치입니다. 세 단계의 워터마크가 있습니다. low 워터마크(기본 85%)에 도달하면 해당 노드에 새 샤드를 배치하지 않습니다. high 워터마크(기본 90%)에 도달하면 해당 노드의 샤드를 다른 노드로 이동시키기 시작합니다. flood_stage 워터마크(기본 95%)에 도달하면 해당 노드의 모든 인덱스를 읽기 전용으로 전환합니다. 디스크 사용량이 워터마크에 근접하면 경보를 발생시켜야 합니다.

Kibana의 **Stack Monitoring**은 이러한 지표를 시각적으로 보여 주는 대시보드입니다. 클러스터 상태, 노드별 자원 사용량, 인덱스별 처리량 등을 시계열 그래프로 확인할 수 있습니다. Stack Monitoring을 사용하려면 모니터링 데이터를 수집해야 합니다.

모니터링 데이터 수집에는 두 가지 방식이 있습니다. 하나는 Elasticsearch 자체의 내부 수집기(internal collector)를 사용하는 것이고, 다른 하나는 6.2.2에서 다룬 **Metricbeat**를 사용하는 것입니다. Metricbeat 방식이 권장됩니다. Metricbeat는 별도 프로세스로 동작하므로, 모니터링 대상 노드에 장애가 발생해도 모니터링 데이터 수집이 영향을 받지 않습니다.

Metricbeat로 Elasticsearch 모니터링을 설정하려면, Metricbeat의 elasticsearch-xpack 모듈을 활성화합니다.

```yaml
metricbeat.modules:
- module: elasticsearch-xpack
  metricsets:
    - ccr
    - cluster_stats
    - index
    - index_recovery
    - index_summary
    - ml_job
    - node_stats
    - shard
    - enrich
  period: 10s
  hosts: ["https://localhost:9200"]
  username: "monitoring_user"
  password: "password"
```

이 설정은 10초마다 Elasticsearch의 각종 지표를 수집합니다. 수집된 데이터는 모니터링 전용 인덱스에 저장되며, Kibana Stack Monitoring에서 시각화됩니다.

지표를 수집하는 것만으로는 부족하고, 이상 상황이 발생했을 때 자동으로 알림을 보내야 합니다. **Watcher**는 Elasticsearch에 내장된 알림 기능입니다. 지정한 조건이 충족되면 이메일, Slack, PagerDuty 등으로 알림을 전송합니다. 예를 들어 클러스터 상태가 red로 변경되면 Slack 메시지를 보내거나, 디스크 사용률이 80%를 넘으면 이메일을 보내는 규칙을 설정할 수 있습니다.

정리하면, 클러스터 모니터링은 _cluster/health로 전체 상태를 확인하고, 노드별 JVM 힙 사용률, 인덱싱/검색 처리량, 스레드 풀 거부 수, 디스크 워터마크를 지속적으로 관찰하는 것입니다. Metricbeat 기반으로 모니터링 데이터를 수집하고, Kibana Stack Monitoring에서 시각적으로 분석하며, Watcher로 이상 상황 알림을 자동화합니다.

다음 단원인 6.4.4에서는 롤링 업그레이드를 다룹니다. 서비스 중단 없이 Elasticsearch 버전을 업그레이드하는 절차를 알아봅니다.

이 단원을 마치면 Kibana Stack Monitoring으로 클러스터 상태와 성능을 모니터링할 수 있으며, 주요 지표를 해석하여 이상을 감지할 수 있습니다.
