# 2.2.2 숫자/날짜/불리언 타입

2.2.1에서 문자열 데이터를 다루는 text와 keyword 타입을 살펴보았습니다. 하지만 실제 데이터에는 문자열만 있지 않습니다. 상품의 가격, 주문 수량, 사용자의 나이처럼 숫자로 표현해야 하는 값이 있고, 주문 일시나 로그 발생 시각처럼 시간을 기록해야 하는 값도 있습니다. 회원의 활성 상태처럼 참과 거짓 두 가지만 구분하면 되는 값도 있습니다. Elasticsearch는 이런 데이터 각각에 맞는 전용 필드 타입을 제공합니다. 이 단원에서는 숫자 타입, 날짜 타입, 불리언 타입의 동작 원리와 선택 기준을 살펴봅니다.

문자열 타입과 숫자 타입은 Elasticsearch 내부에서 데이터를 저장하는 방식이 다릅니다. 2.2.1에서 text와 keyword 타입이 역색인(inverted index)을 사용한다고 설명했습니다. 역색인은 단어나 문자열 값을 기준으로 "이 값이 어떤 문서에 들어 있는가"를 찾는 자료구조입니다. 숫자와 날짜 타입은 역색인 대신 **BKD Tree**(Block KD-Tree)라는 자료구조를 사용합니다. BKD Tree는 숫자 값을 다차원 공간의 점으로 취급하여 트리 구조로 정렬합니다. 이 구조 덕분에 "가격이 10,000 이상 50,000 이하인 문서를 찾아라" 같은 범위 쿼리를 빠르게 처리합니다. 문자열 기반의 역색인으로는 숫자 범위를 효율적으로 탐색하기 어렵기 때문에, Elasticsearch는 숫자와 날짜에 이 별도의 자료구조를 적용합니다.

정렬과 집계를 위한 자료구조는 keyword 타입과 동일합니다. 2.2.1에서 keyword 타입이 doc_values라는 컬럼 지향 자료구조를 사용하여 "이 문서의 필드 값은 무엇인가"를 빠르게 조회한다고 설명했습니다. 숫자 타입과 날짜 타입도 doc_values를 기본으로 활성화합니다. 따라서 숫자 필드와 날짜 필드에서는 별도의 설정 없이 정렬, 집계, 스크립팅을 바로 사용할 수 있습니다.

이제 Elasticsearch가 제공하는 숫자 타입을 하나씩 살펴봅니다. 숫자 타입은 크게 정수 타입과 실수 타입으로 나뉩니다.

먼저 정수 타입입니다. Elasticsearch는 네 가지 정수 타입을 제공합니다. **byte**는 -128부터 127까지의 값을 저장하는 8비트 정수입니다. **short**는 -32,768부터 32,767까지를 저장하는 16비트 정수입니다.

이보다 넓은 범위가 필요하면 **integer**와 **long**을 사용합니다. integer는 약 -21억부터 21억까지를 저장하는 32비트 정수입니다. long은 약 -922경부터 922경까지를 저장하는 64비트 정수입니다.

네 가지 타입의 차이는 저장할 수 있는 값의 범위입니다. 다음 표로 정리합니다.

```
타입       비트   최솟값                  최댓값                  사용 예
-------   ----   --------------------   --------------------   ----------------
byte       8     -128                   127                    월(1~12), 요일
short      16    -32,768                32,767                 항목 수량
integer    32    -2,147,483,648         2,147,483,647          사용자 ID, 가격
long       64    -9,223,372,036,854,    9,223,372,036,854,     타임스탬프(밀리초),
                  775,808                775,807                대규모 카운터
```

타입을 선택할 때는 해당 필드에 들어올 수 있는 값의 범위를 기준으로 판단합니다. 사람의 나이를 저장한다면 0부터 200 사이면 충분하므로 byte나 short로 충분합니다. 상품 가격을 원 단위 정수로 저장한다면 수백만 원대까지 다룰 수 있어야 하므로 integer가 적합합니다. 전 세계 사용자 수를 세는 카운터처럼 21억을 넘을 수 있는 값이라면 long을 선택해야 합니다.

작은 타입을 선택하면 디스크 공간과 메모리를 절약할 수 있습니다. byte는 값 하나당 1바이트, long은 8바이트를 사용합니다. 문서가 수천만 건일 때 이 차이가 누적되면 인덱스 크기에 영향을 줍니다. 다만, 실제 저장 시에는 Lucene 내부의 압축 기법이 적용되므로 단순히 비트 수의 비례로 크기가 늘어나지는 않습니다. 확실하지 않을 때는 integer를 기본으로 선택하고, 21억을 넘을 가능성이 있으면 long을 사용하는 것이 실용적입니다.

다음으로 실수 타입을 살펴봅니다. 실수 타입은 소수점이 있는 숫자를 저장합니다. Elasticsearch는 네 가지 실수 타입을 제공합니다.

**half_float**는 16비트 부동소수점입니다. 정밀도가 낮아 유효 숫자가 약 3자리입니다. 센서 데이터처럼 대략적인 수치만 필요하고 문서 수가 매우 많을 때 저장 공간을 절약하는 용도로 사용합니다.

**float**는 32비트 부동소수점입니다. 유효 숫자가 약 7자리이며, 일반적인 실수 값을 저장하기에 충분합니다.

**double**은 64비트 부동소수점입니다. 유효 숫자가 약 15자리이며, 과학 계산이나 금융 데이터처럼 높은 정밀도가 필요할 때 사용합니다.

**scaled_float**는 다른 세 타입과 동작 방식이 다릅니다. 이 타입은 실수를 직접 저장하지 않고, 지정한 배율(scaling_factor)을 곱한 뒤 정수(long)로 변환하여 저장합니다. 예를 들어 scaling_factor를 100으로 설정하면, 값 9.99는 내부적으로 999라는 정수로 저장됩니다. 조회할 때는 다시 100으로 나누어 9.99를 반환합니다. 매핑 정의는 다음과 같습니다.

```json
{
  "mappings": {
    "properties": {
      "price": {
        "type": "scaled_float",
        "scaling_factor": 100
      }
    }
  }
}
```

price 필드를 scaled_float 타입으로 정의하고, scaling_factor를 100으로 지정했습니다. 이 설정은 소수점 둘째 자리까지 정밀하게 표현하겠다는 의미입니다. 값 19.99를 인덱싱하면 내부적으로 1999라는 long 정수로 저장합니다.

scaled_float가 유용한 이유는 정수 저장이 부동소수점 저장보다 압축 효율이 높기 때문입니다. 가격처럼 소수점 자릿수가 고정된 데이터에서는 scaled_float를 사용하면 디스크 공간을 절약하면서도 필요한 정밀도를 유지할 수 있습니다.

실수 타입을 선택할 때는 정밀도와 저장 효율의 균형을 고려합니다. 소수점 자릿수가 고정되어 있다면 scaled_float가 가장 효율적입니다. 그렇지 않다면 일반적으로 float로 충분하고, 과학 계산 수준의 정밀도가 필요할 때만 double을 선택합니다. half_float는 정밀도 손실을 감수할 수 있는 대량 데이터에서만 사용합니다.

한 가지 주의할 점이 있습니다. 숫자처럼 보이지만 실제로는 범위 검색이 필요 없는 식별자 데이터가 있습니다. 주문 번호, 우편 번호, 전화번호 같은 값이 대표적입니다. 이런 데이터는 "20230001번 주문부터 20230100번 주문까지"처럼 범위로 검색하기보다, 특정 번호를 정확히 찾는 용도로 사용합니다. 이 경우에는 숫자 타입 대신 keyword 타입으로 매핑하는 것이 더 적합합니다. keyword 타입의 term 쿼리가 숫자 타입의 term 쿼리보다 빠르게 실행되기 때문입니다. 숫자 타입은 BKD Tree를 통해 범위 검색에 최적화되어 있으므로, 범위 검색이 필요하지 않다면 그 이점을 활용하지 못하면서 오히려 정확 일치 검색 성능에서 손해를 봅니다.

이제 날짜 타입을 살펴봅니다. Elasticsearch에서 **date** 타입은 날짜와 시간을 저장하는 필드 타입입니다. 날짜를 별도의 타입으로 다루는 이유는, 날짜 데이터에는 문자열과는 다른 특수한 요구가 있기 때문입니다. "2026년 4월 1일부터 2026년 4월 30일 사이에 생성된 문서"를 찾으려면 날짜 값끼리 크기를 비교할 수 있어야 합니다. 문자열 그대로 저장하면 "2026-04-01"과 "2026-04-30"을 사전순으로 비교해야 하는데, 이 방식은 포맷이 조금만 달라져도 정확한 비교가 어렵습니다. date 타입은 이 문제를 내부적으로 해결합니다.

date 타입의 내부 저장 방식을 이해하면 동작 원리가 명확해집니다. Elasticsearch는 어떤 포맷으로 날짜 문자열이 들어오든, 내부적으로는 UTC 기준의 밀리초 단위 정수(epoch milliseconds)로 변환하여 저장합니다. epoch milliseconds란 1970년 1월 1일 00시 00분 00초(UTC)를 기준으로 경과한 밀리초 수입니다. 예를 들어 "2026-04-01T00:00:00Z"는 1774828800000이라는 정수로 저장됩니다. 이 정수 값은 숫자 타입과 마찬가지로 BKD Tree에 등록되어 범위 쿼리를 빠르게 처리합니다.

date 필드를 매핑할 때 가장 중요한 설정은 **format** 속성입니다. format은 Elasticsearch가 입력받은 날짜 문자열을 어떻게 해석할지 결정합니다. format을 명시하지 않으면 기본값인 **strict_date_optional_time||epoch_millis**가 적용됩니다. 이 기본값은 두 가지 포맷을 "||"로 연결한 것으로, 둘 중 하나에 해당하면 인덱싱을 허용한다는 뜻입니다.

첫 번째 포맷인 strict_date_optional_time은 ISO 8601 표준을 따르는 날짜 문자열입니다. ISO 8601은 국제 표준화 기구가 정한 날짜·시간 표기 규격으로, "연-월-일T시:분:초" 형태를 사용합니다. 구체적인 예를 들면 "2026-04-01", "2026-04-01T14:30:00", "2026-04-01T14:30:00.000Z" 같은 형태입니다. 이름에 "optional_time"이 붙은 이유는 날짜 부분만 있어도 되고, 시간 부분을 함께 붙여도 되기 때문입니다. "strict"는 자릿수를 엄격하게 검증한다는 뜻으로, 월과 일에 반드시 두 자리 숫자를 요구합니다. "2026-4-1"은 거부하고 "2026-04-01"만 허용합니다.

두 번째 포맷인 **epoch_millis**는 숫자로 된 밀리초 타임스탬프를 그대로 받아들입니다. 1774828800000 같은 정수 값을 날짜로 인식합니다. 서버 간 통신이나 로그 시스템에서 날짜를 밀리초 정수로 보내는 경우가 많기 때문에, 기본 포맷에 포함되어 있습니다.

기본 포맷이 아닌 다른 형식의 날짜를 사용해야 할 때는 매핑에서 format을 직접 지정합니다. 다음 예시를 봅니다.

```json
{
  "mappings": {
    "properties": {
      "created_at": {
        "type": "date",
        "format": "yyyy/MM/dd HH:mm:ss||yyyy/MM/dd||epoch_millis"
      }
    }
  }
}
```

created_at 필드를 date 타입으로 정의하고, format에 세 가지 패턴을 지정했습니다. "yyyy/MM/dd HH:mm:ss"는 "2026/04/01 14:30:00" 형태를 허용합니다. "yyyy/MM/dd"는 "2026/04/01"처럼 날짜만 있는 형태를 허용합니다. "epoch_millis"는 밀리초 정수를 허용합니다. 세 포맷을 "||"로 연결했으므로, 이 중 어느 형태로 값이 들어와도 인덱싱됩니다.

format 문자열에서 yyyy는 네 자리 연도, MM은 두 자리 월, dd는 두 자리 일, HH는 24시간제 시, mm은 분, ss는 초를 뜻합니다. 이 패턴은 Java의 DateTimeFormatter 규격을 따릅니다.

한 가지 주의할 점은, 한번 정의한 date 필드의 format을 나중에 변경할 수 없다는 것입니다. 기존 인덱스에 이미 "yyyy-MM-dd" 포맷으로 date 필드가 정의되어 있다면, 같은 인덱스에서 이 필드의 포맷을 "yyyy/MM/dd"로 바꿀 수 없습니다. 포맷을 변경해야 한다면 새 인덱스를 만들고 데이터를 다시 인덱싱해야 합니다.

나노초 수준의 정밀도가 필요한 경우에는 **date_nanos** 타입을 사용합니다. 일반 date 타입은 밀리초(1/1,000초) 정밀도인 반면, date_nanos는 나노초(1/1,000,000,000초) 정밀도를 지원합니다. 시스템 성능 측정이나 금융 거래처럼 극도로 세밀한 시간 기록이 필요한 경우에 사용합니다. 다만 date_nanos는 표현 가능한 날짜 범위가 1970년부터 2262년까지로 제한됩니다. 일반적인 애플리케이션 로그나 비즈니스 데이터에서는 date 타입으로 충분합니다.

데이터 스트림에서 날짜 타입의 중요성을 짧게 짚겠습니다. Elasticsearch의 데이터 스트림은 시계열 데이터를 관리하는 구조인데, 모든 문서에 @timestamp 필드가 반드시 있어야 합니다. 이 필드는 date 또는 date_nanos 타입이어야 합니다. 데이터 스트림을 사용할 계획이라면 @timestamp 필드의 date 매핑을 미리 정의해 두는 것이 좋습니다.

이제 **boolean** 타입을 살펴봅니다. boolean 타입은 참(true)과 거짓(false) 두 값만 저장하는 가장 단순한 타입입니다. 회원의 활성 상태, 이메일 인증 여부, 상품의 품절 여부처럼 이분법적인 상태를 표현할 때 사용합니다.

boolean 필드의 매핑은 간단합니다.

```json
{
  "mappings": {
    "properties": {
      "is_active": {
        "type": "boolean"
      }
    }
  }
}
```

boolean 필드는 JSON의 true와 false 값을 받습니다. 추가로 Elasticsearch는 문자열 "true"와 "false", 빈 문자열 ""(false로 해석)도 boolean 값으로 인식합니다. 하지만 혼란을 피하기 위해 JSON의 true/false를 직접 사용하는 것이 바람직합니다.

boolean 필드에서 검색할 때는 term 쿼리를 사용합니다. 2.2.1에서 keyword 필드에 term 쿼리를 사용하는 것이 올바르다고 설명했는데, boolean 필드도 마찬가지입니다. boolean 필드는 분석기를 적용하지 않고 값을 그대로 저장하므로, term 쿼리로 정확 일치 검색을 합니다. `{"term": {"is_active": true}}`처럼 사용합니다. boolean 필드도 doc_values가 기본 활성화되어 있으므로 집계에 바로 사용할 수 있습니다.

마지막으로 **범위 타입**(range type)을 살펴봅니다. 지금까지 살펴본 숫자, 날짜 타입은 하나의 값을 저장합니다. 그런데 데이터에 따라 하나의 값이 아닌 값의 범위를 저장해야 할 때가 있습니다. 회의 예약 시스템에서 "오후 2시부터 오후 4시까지"라는 시간 범위를 하나의 필드에 저장하거나, 상품의 권장 사용 온도를 "15도에서 25도 사이"로 저장하는 경우가 그렇습니다.

Elasticsearch는 이런 용도로 범위 타입을 제공합니다. 정수에 대응하는 **integer_range**와 **long_range**, 실수에 대응하는 **float_range**와 double_range가 있습니다.

날짜에 대응하는 **date_range**도 있습니다. ip_range도 존재하지만 이 단원의 범위에는 포함하지 않습니다.

범위 타입의 매핑과 인덱싱 예시를 봅니다.

```json
{
  "mappings": {
    "properties": {
      "temperature_range": {
        "type": "integer_range"
      },
      "event_period": {
        "type": "date_range"
      }
    }
  }
}
```

```json
{
  "temperature_range": {
    "gte": 15,
    "lte": 25
  },
  "event_period": {
    "gte": "2026-04-01",
    "lte": "2026-04-30"
  }
}
```

temperature_range 필드에는 하한(gte, 이상)과 상한(lte, 이하)을 함께 지정하여 15에서 25까지의 범위를 저장했습니다. event_period 필드에는 4월 1일부터 4월 30일까지의 날짜 범위를 저장했습니다. gte는 "greater than or equal to"의 약자이고, lte는 "less than or equal to"의 약자입니다. gt(초과)와 lt(미만)도 사용할 수 있습니다.

범위 타입 필드에서 검색할 때는 range 쿼리를 사용합니다. 검색 조건으로 지정한 범위와 문서에 저장된 범위가 겹치는지(INTERSECTS), 검색 범위가 저장된 범위를 완전히 포함하는지(CONTAINS), 저장된 범위가 검색 범위 안에 완전히 들어가는지(WITHIN) 등의 관계를 지정할 수 있습니다.

정리하면, Elasticsearch의 숫자 타입은 정수(byte, short, integer, long)와 실수(half_float, float, double, scaled_float)로 나뉘며, 값의 범위와 정밀도에 따라 선택합니다. 범위 검색이 필요 없는 숫자 식별자는 keyword 타입이 더 적합합니다. 날짜 타입은 어떤 포맷으로 입력되든 내부적으로 epoch milliseconds 정수로 변환하여 저장하며, format 속성으로 허용할 입력 포맷을 지정합니다. boolean 타입은 true/false 두 값을 저장하는 단순한 타입으로, term 쿼리로 검색합니다. 범위 타입은 하나의 값이 아닌 값의 범위를 저장할 때 사용합니다. 이 모든 타입은 BKD Tree와 doc_values를 활용하여 범위 검색, 정렬, 집계를 효율적으로 지원합니다.

다음 단원인 2.2.3에서는 dense_vector, geo_point, nested, join 같은 특수 타입을 다룹니다.

이 단원을 마치면 integer, long, float, double 타입의 선택 기준을 설명할 수 있고, date 타입의 포맷 설정과 ISO 8601 형식을 적용할 수 있습니다.
