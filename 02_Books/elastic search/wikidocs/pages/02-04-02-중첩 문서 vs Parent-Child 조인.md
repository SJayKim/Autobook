# 2.4.2 중첩 문서 vs Parent-Child 조인

2.2.3에서 nested 타입과 join 타입의 기본 개념을 살펴보았습니다. nested는 객체 배열의 필드 짝을 보존하기 위해 각 객체를 숨겨진 문서로 저장하는 방식이고, join은 별도 문서를 부모-자식 관계로 연결하는 방식입니다. 두 타입 모두 문서 사이의 관계를 표현한다는 공통점이 있지만, 실제 서비스에서는 어떤 방식을 선택하느냐에 따라 색인 속도, 쿼리 성능, 운영 복잡도가 크게 달라집니다. 이 단원에서는 두 방식의 내부 저장 구조를 비교하고, 각각의 쿼리 방식과 성능 특성을 살펴본 뒤, 어떤 상황에서 어떤 방식이 적합한지 판단 기준을 세웁니다.

먼저 nested 타입의 저장 방식을 다시 짚어봅니다. 2.2.3에서 설명한 것처럼, nested로 선언된 필드는 배열의 각 객체를 별도의 **숨겨진 문서**(hidden document)로 색인합니다. 예를 들어 하나의 상품 문서에 리뷰 3건이 nested 배열로 들어 있다면, Elasticsearch는 내부적으로 문서 4건을 만듭니다. 부모 문서 1건과 숨겨진 리뷰 문서 3건입니다. 숨겨진 문서는 일반 검색 결과에 노출되지 않습니다. nested 쿼리를 통해서만 접근할 수 있습니다.

이 구조가 성능에 미치는 영향을 이해하려면 색인 시점의 동작을 알아야 합니다. nested 객체를 하나라도 수정하면 부모 문서와 모든 숨겨진 문서를 함께 다시 색인해야 합니다. 리뷰 하나를 추가하거나 기존 리뷰의 별점을 바꾸는 것만으로도 문서 전체가 재색인됩니다. 하위 객체가 수십 개 수준이라면 이 비용은 무시할 수 있지만, 수천 개 이상이면 색인 지연이 눈에 띄기 시작합니다.

또한 하나의 문서에 저장할 수 있는 nested 객체의 수에는 기본 제한이 있습니다. index.mapping.nested_objects.limit 설정의 기본값은 10,000입니다. 이 제한은 변경할 수 있지만, 숫자를 높이면 그만큼 메모리 사용량이 늘어나고 색인 속도가 느려집니다.

nested 필드를 검색할 때는 반드시 **nested 쿼리**를 사용해야 합니다. 일반 match 쿼리로는 숨겨진 문서 안의 값에 접근할 수 없습니다. nested 쿼리의 구조를 다시 확인합니다.

```json
{
  "query": {
    "nested": {
      "path": "reviews",
      "query": {
        "bool": {
          "must": [
            {"match": {"reviews.author": "kim"}},
            {"range": {"reviews.rating": {"gte": 4}}}
          ]
        }
      }
    }
  }
}
```

path에 nested 필드의 경로를 지정합니다. query 안에 검색 조건을 넣습니다. 이 예시는 reviews 배열에서 author가 "kim"이면서 동시에 rating이 4 이상인 객체가 있는 문서만 반환합니다. "동시에"가 핵심입니다. 숨겨진 문서 단위로 조건을 평가하기 때문에, 서로 다른 리뷰 객체의 필드가 섞이지 않습니다.

nested 쿼리의 성능은 대부분의 시나리오에서 빠른 편입니다. 부모 문서와 숨겨진 문서가 같은 Lucene 세그먼트 안에 인접하여 저장되므로, 디스크 접근이 효율적입니다. 별도의 조인 연산 없이 세그먼트 내부에서 문서를 찾아 매칭합니다.

이제 join 타입의 저장 방식을 살펴봅니다. join 타입은 nested와 달리 부모와 자식이 완전히 독립된 문서입니다. 2.2.3에서 다룬 것처럼, 매핑에서 **relations** 설정으로 누가 부모이고 누가 자식인지를 정의합니다.

```json
{
  "mappings": {
    "properties": {
      "relation_type": {
        "type": "join",
        "relations": {
          "product": "review"
        }
      }
    }
  }
}
```

relations에서 "product"가 부모이고 "review"가 자식입니다. 부모 문서와 자식 문서는 각각 별도로 색인합니다.

join 타입에서 가장 중요한 제약은 **동일 샤드 라우팅**입니다. 부모와 자식 문서가 같은 샤드에 저장되어 있어야 관계 쿼리를 실행할 수 있습니다. Elasticsearch가 관계를 확인하려면 부모와 자식 문서를 동시에 접근해야 하는데, 서로 다른 샤드에 흩어져 있으면 이것이 불가능합니다. 따라서 자식 문서를 색인할 때 반드시 routing 파라미터에 부모 문서의 ID를 지정해야 합니다.

```json
PUT /shop/_doc/1
{
  "name": "무선 키보드",
  "relation_type": {
    "name": "product"
  }
}

PUT /shop/_doc/100?routing=1
{
  "body": "타건감이 좋습니다.",
  "relation_type": {
    "name": "review",
    "parent": 1
  }
}
```

첫 번째 문서는 상품(product)으로 색인합니다. 두 번째 문서는 리뷰(review)로 색인하면서, parent에 부모 문서의 ID인 1을 지정하고, routing에도 1을 넣어 같은 샤드에 배치합니다. 이 라우팅을 빠뜨리면 자식 문서가 다른 샤드에 저장되어, 이후 관계 쿼리에서 해당 자식을 찾지 못할 수 있습니다.

join 타입의 관계 검색에는 세 가지 쿼리를 사용합니다. **has_child 쿼리**는 자식 문서의 조건을 만족하는 부모 문서를 반환합니다. 예를 들어 "별점 5점 리뷰가 달린 상품"을 찾으려면 has_child 쿼리를 사용합니다.

```json
{
  "query": {
    "has_child": {
      "type": "review",
      "query": {
        "range": {"rating": {"gte": 5}}
      }
    }
  }
}
```

type에 자식의 이름을 지정하고, query에 자식 문서의 검색 조건을 넣습니다. 이 쿼리는 rating이 5 이상인 리뷰를 가진 상품 문서를 반환합니다. 반환되는 것은 부모 문서입니다. 어떤 리뷰가 조건을 만족했는지는 결과에 나타나지 않습니다.

**has_parent 쿼리**는 반대 방향입니다. 부모 문서의 조건을 만족하는 자식 문서를 반환합니다.

```json
{
  "query": {
    "has_parent": {
      "parent_type": "product",
      "query": {
        "match": {"name": "키보드"}
      }
    }
  }
}
```

parent_type에 부모의 이름을, query에 부모 문서의 검색 조건을 넣습니다. 이름에 "키보드"가 포함된 상품의 리뷰를 반환합니다.

세 번째는 **parent_id 쿼리**입니다. 특정 부모 문서의 ID를 지정하여 해당 부모의 자식 문서를 직접 조회합니다.

```json
{
  "query": {
    "parent_id": {
      "type": "review",
      "id": 1
    }
  }
}
```

type에 자식의 이름을, id에 부모 문서의 ID를 넣습니다. 문서 1번 상품에 달린 모든 리뷰를 반환합니다. has_parent 쿼리가 부모의 내용을 기준으로 검색한다면, parent_id 쿼리는 부모의 ID만으로 자식을 바로 가져옵니다.

join 쿼리의 성능은 nested 쿼리보다 느린 경우가 많습니다. 그 이유는 **global ordinals**라는 내부 자료구조에 있습니다. Elasticsearch는 부모-자식 관계를 빠르게 탐색하기 위해 global ordinals를 메모리에 구축합니다. global ordinals는 필드 값을 정수로 매핑한 전역 사전입니다. join 쿼리를 실행할 때 이 사전을 참조하여 부모와 자식의 연결을 찾습니다.

문제는 global ordinals의 구축 비용입니다. 기본 설정에서 global ordinals는 지연 구축(lazy loading) 방식으로 동작합니다. 즉, 첫 번째 쿼리가 실행될 때 비로소 구축을 시작합니다. 문서 수가 많으면 이 첫 쿼리가 수 초 이상 걸릴 수 있습니다. 이후 캐시가 유효한 동안에는 빠르지만, 세그먼트가 병합되거나 새 문서가 색인되면 캐시가 무효화되어 다시 구축해야 합니다.

매핑에서 eager_global_ordinals를 true로 설정하면, 색인 시점에 미리 global ordinals를 구축합니다. 쿼리 지연은 줄어들지만 색인 속도가 느려지고 메모리 사용량이 늘어납니다.

```json
{
  "relation_type": {
    "type": "join",
    "eager_global_ordinals": true,
    "relations": {
      "product": "review"
    }
  }
}
```

이 설정은 쿼리 응답 시간이 중요한 운영 환경에서 고려할 만합니다.

두 방식의 성능 특성을 표로 정리합니다.

```
항목                nested                      join
-----------       -------------------------   -------------------------
저장 위치           부모 문서 안에 숨겨진 문서      별도 독립 문서
수정 시 동작        부모 문서 전체 재색인           자식 문서만 독립 색인
하위 데이터 제한     기본 10,000개               별도 제한 없음
쿼리 성능           같은 세그먼트 내 탐색,         global ordinals 구축 필요,
                   일반적으로 빠름              상대적으로 느림
라우팅 요구          없음                        자식 문서에 부모 ID 라우팅 필수
메모리 오버헤드      숨겨진 문서 수만큼 증가        global ordinals 메모리 사용
```

이 표를 바탕으로 사용 권장 시나리오를 정리합니다.

nested가 적합한 경우는 다음과 같습니다. 하위 객체의 수가 적고 변경 빈도가 낮을 때 nested를 선택합니다. 예를 들어 상품의 속성 목록, 사용자의 주소록, 주문 내역의 항목처럼 한 문서에 포함되는 하위 데이터가 수십에서 수백 개 수준이고, 자주 바뀌지 않는 경우입니다. 쿼리 성능도 일반적으로 빠르므로, 특별한 이유가 없다면 nested를 기본 선택지로 놓는 것이 합리적입니다.

join이 적합한 경우는 자식 데이터가 독립적으로 자주 추가되거나 수정되고, 부모당 자식 수가 매우 많을 때입니다. 게시판의 댓글, 상품의 리뷰처럼 자식 문서가 수시로 추가되는 구조에서는 join이 유리합니다. 자식 문서만 따로 색인하면 되므로, 부모 문서 전체를 재색인하는 비용이 발생하지 않습니다. 다만 join 쿼리의 오버헤드와 라우팅 관리 부담을 감수해야 합니다.

두 방식 모두 적합하지 않은 경우도 있습니다. 하위 객체의 필드 짝을 보존할 필요가 없다면, 일반 object 타입으로 충분합니다. 2.2.3에서 설명한 것처럼, object 타입은 배열을 평탄화하여 필드 짝이 사라지지만 성능 부담이 가장 적습니다. 관계 보존이 불필요한 단순 배열이라면 object 타입을 먼저 고려합니다.

선택 과정을 단계별로 정리합니다.

```
하위 객체의 필드 짝 보존이 필요한가?
  |
  +-- 아니오 --> 일반 object 타입 사용
  |
  +-- 예
       |
       하위 데이터가 독립적으로 자주 변경되는가?
         |
         +-- 아니오 --> nested 사용 (부모 문서에 포함)
         |
         +-- 예
              |
              부모당 자식 수가 매우 많은가?
                |
                +-- 아니오 --> nested로도 가능, 재색인 비용 평가 후 결정
                |
                +-- 예 --> join 사용 (별도 문서, 라우팅 필수)
```

정리하면, nested 타입은 배열의 각 객체를 부모 문서 안에 숨겨진 문서로 저장하여 빠른 쿼리 성능을 제공하지만, 하위 객체 변경 시 부모 전체를 재색인해야 합니다. join 타입은 부모와 자식을 독립 문서로 분리하여 자식만 따로 수정할 수 있지만, global ordinals 구축에 따른 쿼리 오버헤드와 동일 샤드 라우팅 관리가 필요합니다. 대부분의 경우 nested가 더 나은 성능을 보이므로 기본 선택지로 두되, 자식 데이터의 변경 빈도와 규모가 큰 상황에서만 join을 선택하는 것이 실용적입니다.

다음 단원인 2.4.3에서는 인덱스 템플릿과 Component Template를 다룹니다.

이 단원을 마치면 nested 타입과 join 타입의 저장 방식 차이를 설명할 수 있으며, 각 방식의 쿼리 성능 특성을 비교하여 적합한 방식을 선택할 수 있습니다.
