# 3.4.2 k1/b 파라미터 튜닝

3.4.1에서 BM25의 점수 계산이 IDF, TF 포화, 문서 길이 정규화라는 세 요소로 이루어진다고 설명했습니다. 그리고 TF 포화 속도를 제어하는 k1(기본값 1.2)과 문서 길이 정규화 강도를 제어하는 b(기본값 0.75)라는 두 파라미터가 있다고 언급했습니다. 기본값은 대부분의 검색 시나리오에서 합리적인 결과를 냅니다. 그러나 도메인마다 문서의 성격이 다릅니다. 제목처럼 짧은 필드를 검색하는 경우와 본문처럼 긴 필드를 검색하는 경우에 같은 파라미터가 최적일 수는 없습니다. 이 단원에서는 k1과 b 각각이 점수에 미치는 영향을 구체적으로 살펴본 뒤, 도메인 특성에 맞게 값을 조정하는 전략을 다룹니다.

k1 파라미터부터 살펴봅니다. 3.4.1에서 TF 포화 곡선을 설명하면서, 단어 등장 횟수가 늘어나도 점수 기여가 일정 수준에서 멈춘다고 했습니다. k1은 이 천장에 도달하는 속도를 결정합니다. k1 값이 작으면 곡선이 빨리 수평에 가까워집니다. 단어가 2~3번만 등장해도 추가 등장이 점수를 거의 올리지 못합니다. 반대로 k1 값이 크면 곡선이 천천히 수평에 가까워집니다. 단어가 10번, 20번 등장할 때까지도 추가 등장이 점수에 의미 있게 기여합니다.

구체적인 수치로 비교합니다. k1이 0.5일 때, 단어가 1번 등장하면 포화된 TF 값이 약 0.67입니다. 3번 등장하면 약 0.86입니다. 10번 등장하면 약 0.95입니다. 1번에서 10번으로 등장 횟수가 10배 늘었지만, 점수 기여는 0.67에서 0.95로 약 1.4배만 늘어납니다. k1이 2.0일 때, 단어가 1번 등장하면 포화된 TF 값이 약 0.33입니다. 3번 등장하면 약 0.60입니다. 10번 등장하면 약 0.83입니다. 같은 1번에서 10번으로의 변화가 점수에 훨씬 큰 차이를 만들어냅니다.

```
포화된 TF 값
  |
1.0|.......................
  |   k1=0.5         .
  |  /...............
  | /  k1=1.2    .
  |/  /..........
  | / /  k1=2.0
  |/ / /
  | / /
  |/ /
  | /
  |/
  +--------------------------> 단어 등장 횟수 (TF)
  0    5    10   15   20
```

위 그림에서 k1이 작을수록 곡선이 빨리 천장에 닿고, k1이 클수록 곡선이 완만하게 올라갑니다. Elasticsearch의 기본값인 1.2는 두 극단 사이에 위치합니다.

k1을 언제 조정해야 하는지 생각해 봅니다. 뉴스 기사나 블로그 글처럼 동일 주제를 반복해서 언급하는 문서가 많은 도메인에서는 k1을 낮추는 것이 유리합니다. 단어가 몇 번 나왔는지보다 나왔는지 안 나왔는지가 더 중요한 상황이기 때문입니다. 반면 학술 논문이나 기술 문서처럼 키워드 등장 빈도가 문서의 깊이를 실제로 반영하는 도메인에서는 k1을 높이는 것이 유리합니다. "machine learning"이라는 단어가 3번 나온 논문보다 30번 나온 논문이 실제로 머신 러닝을 더 깊이 다룰 가능성이 높기 때문입니다.

이제 b 파라미터를 살펴봅니다. b는 문서 길이 정규화의 강도를 0과 1 사이에서 조절합니다. 3.4.1에서 설명한 것처럼, BM25는 해당 문서의 필드 길이를 전체 문서의 평균 필드 길이와 비교하여 점수를 보정합니다. 평균보다 긴 문서는 점수가 깎이고, 평균보다 짧은 문서는 점수가 올라갑니다. b는 이 보정이 얼마나 강하게 적용되는지를 결정합니다.

b가 0이면 문서 길이를 전혀 고려하지 않습니다. 10단어짜리 문서와 10,000단어짜리 문서에서 같은 단어가 동일 횟수 등장하면, 길이와 관계없이 같은 TF 기여를 받습니다. b가 1이면 문서 길이를 최대한 강하게 반영합니다. 평균보다 긴 문서는 큰 폭으로 점수가 깎이고, 평균보다 짧은 문서는 큰 폭으로 점수가 올라갑니다.

b 값에 따른 동작 차이를 표로 정리합니다.

```
b 값      | 문서 길이 정규화 강도      | 효과
----------|--------------------------|--------------------------------------
0         | 없음 (길이 무시)           | 긴 문서도 짧은 문서도 TF만으로 평가
0.25      | 약함                      | 길이 차이가 점수에 조금만 반영
0.75(기본)| 중간~강함                  | 평균보다 긴 문서의 점수를 상당히 깎음
1.0       | 최대                      | 평균 대비 길이 비율이 점수에 최대 반영
```

b를 낮추는 것이 유리한 대표적인 경우는 짧은 필드를 검색할 때입니다. 상품명이나 글 제목처럼 필드 길이가 원래 짧고 문서 간 길이 차이가 작은 필드에서는 길이 정규화가 오히려 잡음이 됩니다. 제목이 3단어인 문서와 5단어인 문서 사이에 의미 있는 길이 차이가 있다고 보기 어렵기 때문입니다. 이런 필드에서는 b를 0에 가깝게 낮추어 길이 보정의 영향을 줄입니다.

반대로 b를 높이는 것이 유리한 경우도 있습니다. 본문(body) 필드처럼 문서 간 길이 편차가 큰 필드에서는 길이 정규화가 중요합니다. 500단어짜리 기사와 50,000단어짜리 보고서에서 같은 단어가 5번 등장했을 때, 500단어 기사가 해당 주제에 더 집중된 문서일 가능성이 높습니다. 이런 필드에서는 b를 기본값(0.75)이나 그 이상으로 유지하여 길이 차이가 점수에 충분히 반영되도록 합니다.

인덱스에 k1과 b를 실제로 적용하는 방법을 살펴봅니다. Elasticsearch에서는 인덱스 매핑의 similarity 설정을 통해 필드별로 BM25 파라미터를 지정합니다. 3.4.1에서 커스텀 BM25 설정의 기본 구조를 소개했으므로, 여기서는 제목 필드와 본문 필드에 서로 다른 파라미터를 적용하는 실제 시나리오를 다룹니다.

뉴스 검색 서비스를 구축한다고 합시다. 제목(title) 필드는 짧고, 본문(body) 필드는 길이 편차가 큽니다. 제목에서는 길이 정규화가 크게 필요하지 않으므로 b를 낮추고, 본문에서는 키워드 반복이 깊이를 잘 반영하므로 k1을 약간 높이는 전략을 세울 수 있습니다.

```json
PUT news_index
{
  "settings": {
    "similarity": {
      "title_bm25": {
        "type": "BM25",
        "k1": 1.2,
        "b": 0.3
      },
      "body_bm25": {
        "type": "BM25",
        "k1": 1.5,
        "b": 0.75
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "similarity": "title_bm25"
      },
      "body": {
        "type": "text",
        "similarity": "body_bm25"
      }
    }
  }
}
```

settings 블록 안에 두 개의 커스텀 유사도를 정의합니다. "title_bm25"는 k1을 기본값 1.2로 유지하되, b를 0.3으로 낮추었습니다. 제목 필드의 짧은 길이 차이가 점수를 크게 흔들지 않도록 하기 위해서입니다. "body_bm25"는 k1을 1.5로 높이고, b는 기본값 0.75를 그대로 사용합니다. 본문에서 키워드 등장 횟수가 문서의 관련성을 더 잘 반영하도록 TF 포화를 늦추는 것입니다. mappings에서 title 필드에는 "title_bm25"를, body 필드에는 "body_bm25"를 지정하여 각 필드가 서로 다른 파라미터로 스코어링되게 합니다.

주의할 점이 있습니다. similarity 설정은 인덱스 생성 시에 정의합니다. 이미 생성된 인덱스의 similarity를 변경하려면, 인덱스를 닫았다가(close) 설정을 업데이트한 뒤 다시 열거나(open), 새 인덱스를 만들어 데이터를 재색인(reindex)해야 합니다. 운영 환경에서는 파라미터 변경이 전체 문서의 스코어링에 영향을 미치므로, 변경 전에 충분한 테스트가 필요합니다.

파라미터를 변경한 뒤에는 변경이 의도대로 적용되었는지 검증해야 합니다. Elasticsearch는 이를 위해 **Explain API**를 제공합니다. Explain API는 특정 문서가 특정 질의에 대해 받은 점수를 단계별로 분해하여 보여줍니다. IDF가 얼마인지, TF 포화가 어떻게 적용되었는지, 문서 길이 정규화가 점수를 얼마나 깎았는지를 확인할 수 있습니다.

```
GET news_index/_explain/1
{
  "query": {
    "match": { "title": "elasticsearch" }
  }
}
```

이 요청은 news_index의 문서 ID 1이 "elasticsearch"라는 질의에 대해 받은 점수의 상세 내역을 반환합니다. 응답에는 IDF 값, TF 값, 필드 길이, 평균 필드 길이, 그리고 이들이 결합되어 최종 점수가 산출되는 과정이 트리 구조로 담겨 있습니다. k1이나 b를 변경하기 전과 후에 각각 Explain API를 호출하면, 파라미터 변경이 점수의 어느 부분에 영향을 미쳤는지를 직접 비교할 수 있습니다.

파라미터 튜닝의 일반적인 절차를 정리합니다.

```
k1/b 파라미터 튜닝 절차

[1단계] 기본값(k1=1.2, b=0.75)으로 인덱스를 생성하고 검색 결과를 확인합니다.

[2단계] 검색 결과에서 문제를 식별합니다.
        - 키워드를 많이 포함한 문서가 부당하게 높은 순위? --> k1을 낮춥니다.
        - 키워드 빈도가 높은 문서가 낮은 순위?           --> k1을 높입니다.
        - 짧은 문서가 부당하게 높은 순위?                --> b를 낮춥니다.
        - 긴 문서가 부당하게 높은 순위?                  --> b를 높입니다.

[3단계] 테스트 인덱스에 변경된 파라미터를 적용합니다.

[4단계] Explain API로 점수 변화를 검증합니다.

[5단계] 대표 질의 세트로 검색 품질을 평가한 뒤, 운영 인덱스에 반영합니다.
```

1단계에서는 기본값으로 시작합니다. 기본값이 대부분의 시나리오에서 잘 동작하므로, 문제가 확인되기 전에 파라미터를 바꿀 필요는 없습니다. 2단계에서 실제 검색 결과를 보고 문제를 식별합니다. 3단계에서 테스트 인덱스에 변경을 적용하고, 4단계에서 Explain API로 점수가 의도대로 변했는지 확인합니다. 5단계에서 여러 질의로 검색 품질을 종합 평가한 뒤 운영에 반영합니다.

정리하면, k1과 b는 BM25의 동작을 도메인에 맞게 미세 조정하는 두 파라미터입니다. k1은 단어 등장 횟수가 점수에 기여하는 정도를 제어하고, b는 문서 길이가 점수에 미치는 영향을 제어합니다. 짧은 필드에서는 b를 낮추어 길이 정규화를 약하게, 키워드 빈도가 관련성을 잘 반영하는 도메인에서는 k1을 높여 TF 포화를 늦추는 것이 일반적인 전략입니다. 매핑의 similarity 설정으로 필드별로 서로 다른 파라미터를 적용할 수 있으며, Explain API로 변경 결과를 검증합니다.

다음 단원인 3.4.3에서는 function_score 쿼리를 다룹니다. BM25 점수만으로 원하는 순서를 만들 수 없을 때, 사용자 정의 함수로 점수를 보정하는 방법을 살펴봅니다.

이 단원을 마치면 k1과 b 파라미터가 BM25 스코어에 미치는 영향을 설명할 수 있고, 도메인 특성에 맞는 값을 조정할 수 있습니다.
