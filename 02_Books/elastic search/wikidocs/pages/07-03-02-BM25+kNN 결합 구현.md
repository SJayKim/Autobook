# 7.3.2 BM25+kNN 결합 구현

7.3.1에서 RRF의 원리와 수식을 다뤘습니다. 이 단원에서는 Elasticsearch의 retriever API를 사용하여 BM25 검색과 kNN 검색을 실제로 결합하는 방법을 구현합니다.

Elasticsearch 8.9 버전부터 도입된 **retriever API**는 검색 요청 안에서 여러 검색 방식을 조합할 수 있는 구조를 제공합니다. 기존에는 검색 요청의 query 절과 knn 절을 따로 작성해야 했지만, retriever API는 이 조합을 하나의 통일된 구조로 표현합니다.

retriever API에는 세 가지 retriever 타입이 있습니다. 첫째, **standard retriever**는 기존의 Query DSL 쿼리를 감싸는 역할을 합니다. BM25 기반의 match 쿼리, bool 쿼리 등을 이 안에 넣습니다. 둘째, **knn retriever**는 7.1.2에서 다룬 kNN 검색을 수행합니다. 셋째, **rrf retriever**는 여러 retriever의 결과를 RRF 알고리즘으로 병합합니다.

이 세 가지를 조합하면 하이브리드 검색이 완성됩니다. 구체적인 요청 구조를 살펴보겠습니다.

```json
GET my-hybrid-index/_search
{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "standard": {
            "query": {
              "match": {
                "content": "분산 검색 엔진"
              }
            }
          }
        },
        {
          "knn": {
            "field": "content_embedding",
            "query_vector_builder": {
              "text_embedding": {
                "model_id": "sentence-transformers__all-minilm-l6-v2",
                "model_text": "분산 검색 엔진"
              }
            },
            "num_candidates": 100
          }
        }
      ],
      "rank_window_size": 50,
      "rank_constant": 60
    }
  }
}
```

이 요청의 구조를 하나씩 풀어보겠습니다. 최상위에 retriever 절이 있고, 그 안에 rrf retriever를 지정했습니다. rrf의 retrievers 배열 안에 두 개의 하위 retriever가 들어갑니다.

첫 번째 하위 retriever는 standard입니다. query 안에 match 쿼리를 넣어 content 필드에서 "분산 검색 엔진"이라는 키워드로 BM25 검색을 수행합니다. 3.2.2에서 다룬 match 쿼리와 동일합니다.

두 번째 하위 retriever는 knn입니다. 7.1.2에서 다룬 kNN 검색 파라미터를 그대로 사용합니다. field에 dense_vector 필드명을, query_vector_builder에 임베딩 모델 정보를 넣습니다. num_candidates는 HNSW 그래프에서 탐색할 후보 수입니다.

rrf 수준에서 **rank_window_size**는 각 하위 retriever가 돌려주는 결과 목록의 크기입니다. 값이 클수록 더 많은 후보를 RRF 병합에 포함시키므로 결과 품질이 올라가지만, 계산량도 증가합니다. **rank_constant**는 7.3.1에서 다룬 RRF 공식의 k 값입니다. 기본값은 60입니다.

```
사용자 쿼리: "분산 검색 엔진"
         │
    ┌────┴────┐
    │         │
standard   knn
(BM25)   (벡터 유사도)
    │         │
 결과 목록1   결과 목록2
 (순위 기반)  (순위 기반)
    │         │
    └────┬────┘
         │
    rrf retriever
  (순위 병합, k=60)
         │
    최종 결과 목록
```

7.2.3에서 다룬 ELSER를 사용하는 경우에도 동일한 구조를 적용할 수 있습니다. standard retriever 안에 text_expansion 쿼리나 semantic 쿼리를 넣으면 BM25 대신 ELSER 검색이 됩니다. BM25, kNN, ELSER 세 가지를 모두 결합하는 것도 가능합니다. retrievers 배열에 세 개의 하위 retriever를 넣으면 됩니다.

retriever API가 도입되기 전에는 **sub_searches** 방식으로 하이브리드 검색을 구현했습니다. 이 방식은 레거시로 분류되지만, 기존 코드를 유지보수할 때 만날 수 있으므로 구조를 알아둘 필요가 있습니다.

```json
GET my-hybrid-index/_search
{
  "sub_searches": [
    {
      "query": {
        "match": {
          "content": "분산 검색 엔진"
        }
      }
    },
    {
      "query": {
        "knn": {
          "field": "content_embedding",
          "query_vector": [0.12, -0.34, ...],
          "num_candidates": 100
        }
      }
    }
  ],
  "rank": {
    "rrf": {
      "rank_constant": 60,
      "rank_window_size": 50
    }
  }
}
```

sub_searches 배열에 각 검색을 나열하고, rank 절에서 rrf를 지정합니다. retriever API와 기능은 같지만, 구조적으로 retriever API가 더 직관적이고 확장성이 좋습니다.

하이브리드 검색의 응답 구조는 일반 검색과 동일합니다. hits 배열에 문서가 RRF 점수순으로 정렬되어 반환됩니다.

```json
{
  "hits": {
    "total": { "value": 42, "relation": "eq" },
    "max_score": null,
    "hits": [
      {
        "_index": "my-hybrid-index",
        "_id": "doc1",
        "_score": 0.03252,
        "_source": { "content": "Elasticsearch는 분산 검색 엔진입니다.", ... }
      },
      ...
    ]
  }
}
```

_score에 RRF 점수가 들어갑니다. max_score가 null인 것은 RRF 점수가 전통적인 관련성 점수와 다른 성격이기 때문입니다.

검색 결과가 왜 이런 순서로 나왔는지 분석하려면 **explain** 파라미터를 사용합니다.

```json
GET my-hybrid-index/_search
{
  "explain": true,
  "retriever": {
    "rrf": {
      "retrievers": [
        { "standard": { "query": { "match": { "content": "분산 검색 엔진" } } } },
        { "knn": { "field": "content_embedding", "query_vector": [...], "num_candidates": 100 } }
      ],
      "rank_window_size": 50,
      "rank_constant": 60
    }
  }
}
```

explain을 true로 설정하면 각 문서의 _explanation 필드에 RRF 점수가 어떻게 계산되었는지 상세 내역이 표시됩니다. 각 하위 retriever에서 해당 문서가 몇 번째였는지, 그에 따른 RRF 기여분이 얼마인지 확인할 수 있습니다. 검색 결과를 디버깅하거나 하이브리드 검색의 효과를 검증할 때 유용합니다.

정리하면, retriever API는 standard, knn, rrf 세 가지 retriever를 조합하여 하이브리드 검색을 구현합니다. rrf retriever의 retrievers 배열에 BM25 검색과 kNN 검색을 넣으면, RRF 알고리즘이 자동으로 두 결과를 병합합니다. rank_window_size로 후보 수를, rank_constant로 순위 집중도를 조절합니다. explain 파라미터로 RRF 점수 계산 과정을 분석할 수 있습니다.

다음 단원인 7.3.3에서는 rank_constant와 num_candidates를 조정하여 하이브리드 검색 품질을 최적화하는 방법을 다룹니다.

이 단원을 마치면 retriever API로 BM25와 kNN을 결합한 하이브리드 검색을 구현할 수 있습니다.
