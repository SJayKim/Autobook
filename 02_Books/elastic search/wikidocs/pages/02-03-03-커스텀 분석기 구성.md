# 2.3.3 커스텀 분석기 구성

2.3.2에서 Elasticsearch가 제공하는 내장 분석기를 살펴보았습니다. standard, simple, whitespace 등 내장 분석기는 이름 하나만 지정하면 바로 사용할 수 있어 편리합니다. 그런데 실무에서는 내장 분석기만으로 해결하기 어려운 상황이 자주 생깁니다. 예를 들어 쇼핑몰 검색에서 사용자가 "노트북"을 입력했을 때 "랩탑"이 포함된 상품도 함께 찾아 주고 싶다면 어떻게 해야 할까요. 또는 검색창에 "ela"까지만 입력해도 "elasticsearch"를 추천해 주는 자동완성 기능을 만들고 싶다면 어떤 분석기를 써야 할까요. 내장 분석기에는 동의어 확장 기능이나 접두어 토큰 생성 기능이 포함되어 있지 않습니다. 이런 요구사항을 충족하려면 캐릭터 필터, 토크나이저, 토큰 필터를 직접 골라 조합하는 **커스텀 분석기**(custom analyzer)를 만들어야 합니다.

커스텀 분석기를 정의하는 장소는 인덱스의 settings입니다. 구체적으로는 인덱스를 생성할 때 `settings.analysis` 블록 아래에 구성요소와 분석기를 함께 선언합니다. 2.3.1에서 분석기가 캐릭터 필터, 토크나이저, 토큰 필터 세 구성요소의 파이프라인이라는 것을 배웠습니다. 커스텀 분석기를 만들 때도 이 세 구성요소를 각각 정의한 뒤, 분석기 선언에서 조합하는 구조입니다.

`settings.analysis` 블록의 전체 구조를 먼저 살펴봅니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "char_filter": {
        ... 캐릭터 필터 정의 ...
      },
      "tokenizer": {
        ... 토크나이저 정의 ...
      },
      "filter": {
        ... 토큰 필터 정의 ...
      },
      "analyzer": {
        ... 분석기 정의 ...
      }
    }
  }
}
```

`analysis` 블록 아래에 네 가지 하위 블록이 있습니다. `char_filter`에는 캐릭터 필터를, `tokenizer`에는 토크나이저를, `filter`에는 토큰 필터를 정의합니다. 그리고 `analyzer`에는 앞에서 정의한 구성요소를 조합하여 분석기를 선언합니다. 토큰 필터의 블록 이름이 `token_filter`가 아니라 `filter`인 점에 주의합니다. Elasticsearch가 이 이름을 사용하므로 그대로 따라야 합니다.

각 블록 안에서는 이름을 직접 정합니다. 이 이름은 같은 인덱스 안에서 구성요소를 참조할 때 사용하는 식별자입니다. 예를 들어 `filter` 블록에 "my_stop"이라는 이름으로 토큰 필터를 정의하면, `analyzer` 블록에서 "my_stop"을 참조하여 해당 필터를 분석기에 포함시킵니다.

이제 구체적인 예를 통해 커스텀 분석기를 만들어 봅니다. 다음은 HTML 태그를 제거하고, standard 토크나이저로 분리한 뒤, 소문자 변환과 불용어 제거를 적용하는 분석기입니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "char_filter": {
        "html_strip_filter": {
          "type": "html_strip"
        }
      },
      "tokenizer": {
        "my_tokenizer": {
          "type": "standard"
        }
      },
      "filter": {
        "my_stop": {
          "type": "stop",
          "stopwords": ["the", "a", "an"]
        }
      },
      "analyzer": {
        "my_custom_analyzer": {
          "type": "custom",
          "char_filter": ["html_strip_filter"],
          "tokenizer": "my_tokenizer",
          "filter": ["lowercase", "my_stop"]
        }
      }
    }
  }
}
```

이 설정을 위에서부터 하나씩 풀어 봅니다. `char_filter` 블록에서 "html_strip_filter"라는 이름으로 캐릭터 필터를 하나 정의했습니다. `type`이 "html_strip"이므로 HTML 태그를 제거하는 내장 캐릭터 필터를 사용합니다. `tokenizer` 블록에서는 "my_tokenizer"라는 이름으로 standard 토크나이저를 정의했습니다. standard 토크나이저는 2.3.1에서 다룬 것처럼 유니코드 텍스트 분할 규칙에 따라 단어를 분리합니다.

`filter` 블록에서는 "my_stop"이라는 토큰 필터를 정의했습니다. `type`이 "stop"이므로 불용어 제거 필터이고, `stopwords` 배열에 제거할 단어 목록을 직접 지정했습니다.

마지막으로 `analyzer` 블록에서 "my_custom_analyzer"라는 분석기를 선언합니다. `type`을 "custom"으로 지정하면 Elasticsearch가 이 분석기를 커스텀 분석기로 인식합니다. `char_filter`에 앞에서 정의한 "html_strip_filter"를 배열로 넣고, `tokenizer`에 "my_tokenizer"를 지정하고, `filter`에 "lowercase"와 "my_stop"을 배열로 넣었습니다. 여기서 "lowercase"는 별도로 정의하지 않았는데, 이것은 Elasticsearch가 기본 제공하는 내장 토큰 필터입니다. 커스텀 분석기를 만들 때 직접 정의한 구성요소와 내장 구성요소를 섞어 사용할 수 있습니다.

이 분석기가 텍스트를 처리하는 흐름을 단계별로 정리합니다.

```
입력: "<p>The Quick Brown Fox</p>"
  |
  v
[1. html_strip_filter]  --> "The Quick Brown Fox"
    HTML 태그 <p>, </p> 제거
  |
  v
[2. my_tokenizer]       --> ["The", "Quick", "Brown", "Fox"]
    standard 토크나이저가 단어 경계로 분리
  |
  v
[3. lowercase]          --> ["the", "quick", "brown", "fox"]
    모든 토큰을 소문자로 변환
  |
  v
[4. my_stop]            --> ["quick", "brown", "fox"]
    불용어 "the" 제거
```

캐릭터 필터가 가장 먼저 동작하여 HTML 태그를 벗겨 냅니다. 토크나이저가 정제된 텍스트를 토큰으로 분리합니다. 토큰 필터는 `filter` 배열에 나열한 순서대로 적용됩니다. "lowercase"가 먼저 소문자 변환을 하고, "my_stop"이 불용어를 제거합니다. 이 순서가 중요합니다. 만약 불용어 필터를 먼저 적용하면 "The"가 대문자이기 때문에 소문자로 정의된 불용어 목록의 "the"와 일치하지 않아 제거되지 않을 수 있습니다.

커스텀 분석기를 정의했으면 매핑에서 해당 분석기를 필드에 연결해야 합니다. 인덱스 생성 시 settings와 mappings를 함께 지정하는 방식이 일반적입니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "analyzer": {
        "my_custom_analyzer": {
          "type": "custom",
          "tokenizer": "standard",
          "filter": ["lowercase", "stop"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "content": {
        "type": "text",
        "analyzer": "my_custom_analyzer"
      }
    }
  }
}
```

`mappings.properties`에서 "content" 필드의 `type`을 "text"로 지정하고, `analyzer`에 앞에서 정의한 "my_custom_analyzer"를 넣었습니다. 이렇게 하면 "content" 필드에 문서가 인덱싱될 때 my_custom_analyzer가 텍스트를 분석합니다. 이 예에서는 구성요소를 별도로 정의하지 않고 내장 토크나이저와 내장 토큰 필터의 이름을 직접 참조했습니다. 단순한 조합이라면 이처럼 `char_filter`, `tokenizer`, `filter` 블록 없이 `analyzer` 블록만으로 커스텀 분석기를 선언할 수 있습니다. 내장 구성요소의 기본 옵션을 그대로 쓸 때 유용한 방식입니다.

이제 커스텀 토크나이저를 정의하는 방법을 살펴봅니다. 앞선 예에서는 내장 토크나이저의 이름을 그대로 참조했지만, 토크나이저의 옵션을 조정하려면 `tokenizer` 블록에서 직접 정의해야 합니다. 예를 들어 pattern 토크나이저를 사용하되, 분리 기준을 쉼표(,)로 바꾸고 싶다면 다음과 같이 합니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "tokenizer": {
        "comma_tokenizer": {
          "type": "pattern",
          "pattern": ","
        }
      },
      "analyzer": {
        "comma_analyzer": {
          "type": "custom",
          "tokenizer": "comma_tokenizer",
          "filter": ["trim", "lowercase"]
        }
      }
    }
  }
}
```

`tokenizer` 블록에서 "comma_tokenizer"를 정의했습니다. `type`을 "pattern"으로 지정하고, `pattern`에 쉼표를 넣어 쉼표를 기준으로 텍스트를 분리하도록 했습니다. pattern 토크나이저는 정규표현식을 지원하므로, 더 복잡한 구분 규칙도 표현할 수 있습니다.

커스텀 토큰 필터도 같은 방식으로 정의합니다. `filter` 블록에서 이름을 정하고 `type`과 옵션을 지정합니다. 앞서 본 "my_stop" 예가 커스텀 토큰 필터의 전형적인 형태입니다. `type`으로 내장 토큰 필터의 종류를 지정하고, 해당 필터가 지원하는 옵션을 추가하여 동작을 조정합니다.

여기서 **동의어 필터**(synonym filter)를 살펴봅니다. 동의어 필터는 검색 품질을 높이기 위해 자주 사용하는 커스텀 토큰 필터입니다. 사용자가 검색할 때 어떤 단어를 사용할지 예측하기 어렵기 때문입니다. "TV"를 검색하는 사용자도 있고 "텔레비전"을 검색하는 사용자도 있습니다. 동의어 필터를 적용하면 한쪽 단어로 검색해도 다른 쪽이 포함된 문서까지 함께 찾아 줍니다.

동의어 필터의 `type`은 "synonym"입니다. 동의어 목록을 지정하는 방법은 두 가지입니다. 하나는 설정 안에 직접 나열하는 방법이고, 다른 하나는 별도의 동의어 파일을 참조하는 방법입니다.

설정 안에 직접 나열하는 방식부터 봅니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "filter": {
        "my_synonym": {
          "type": "synonym",
          "synonyms": [
            "TV, television, 텔레비전",
            "laptop => notebook"
          ]
        }
      },
      "analyzer": {
        "synonym_analyzer": {
          "type": "custom",
          "tokenizer": "standard",
          "filter": ["lowercase", "my_synonym"]
        }
      }
    }
  }
}
```

`synonyms` 배열에 동의어 규칙을 문자열로 넣습니다. "TV, television, 텔레비전"처럼 쉼표로 나열하면 세 단어가 서로 동의어가 됩니다. 어느 단어로 검색해도 나머지 단어가 포함된 문서를 찾습니다. "laptop => notebook"처럼 화살표(=>)를 쓰면 단방향 치환입니다. "laptop"이라는 토큰이 들어오면 "notebook"으로 대체되지만, "notebook"은 그대로 유지됩니다.

동의어 목록이 길어지면 설정 안에 모두 넣기 어렵습니다. 이때 동의어 파일을 사용합니다. 동의어 파일은 Elasticsearch의 config 디렉터리 아래에 텍스트 파일로 저장합니다. 파일 안에 한 줄에 하나의 동의어 규칙을 적습니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "filter": {
        "my_synonym": {
          "type": "synonym",
          "synonyms_path": "analysis/synonyms.txt"
        }
      },
      "analyzer": {
        "synonym_analyzer": {
          "type": "custom",
          "tokenizer": "standard",
          "filter": ["lowercase", "my_synonym"]
        }
      }
    }
  }
}
```

`synonyms` 대신 `synonyms_path`를 사용하여 파일 경로를 지정합니다. 이 경로는 Elasticsearch의 config 디렉터리를 기준으로 한 상대 경로입니다. 동의어 파일(synonyms.txt)의 내용은 다음과 같은 형태입니다.

```
TV, television, 텔레비전
laptop => notebook
pants, trousers
```

파일을 사용하면 동의어 목록을 인덱스 설정과 분리하여 관리할 수 있습니다. 동의어가 수백 개, 수천 개로 늘어나는 실무 환경에서 유지보수가 편리합니다.

동의어 필터를 사용할 때 주의할 점이 있습니다. `filter` 배열에서 동의어 필터의 순서가 결과에 영향을 줍니다. 앞선 예에서 "lowercase"를 "my_synonym"보다 먼저 배치한 이유가 있습니다. 동의어 규칙에 "TV"를 소문자 "tv"로 적었다면, lowercase 필터가 먼저 입력 토큰을 소문자로 변환해야 동의어 규칙과 정확히 일치합니다. 필터 순서를 바꾸면 대소문자 불일치로 동의어가 적용되지 않을 수 있습니다.

Elasticsearch는 동의어 필터 외에 **synonym_graph** 필터도 제공합니다. synonym_graph는 여러 단어로 이루어진 동의어를 더 정확하게 처리합니다. "New York"과 "NYC"처럼 단어 수가 다른 동의어를 다룰 때 synonym_graph가 토큰 위치 정보를 올바르게 유지합니다. 다만 synonym_graph 필터는 검색 시점(search analyzer)에서만 사용하는 것이 권장됩니다. 인덱싱 시점에 사용하면 토큰 그래프가 올바르게 평탄화되지 않아 예상치 못한 검색 결과가 나올 수 있습니다.

이어서 자동완성 기능에 자주 사용하는 **ngram 토크나이저**와 **edge_ngram 토크나이저**를 살펴봅니다. 이 두 토크나이저를 이해하려면 먼저 n-gram이라는 개념을 알아야 합니다.

n-gram은 텍스트에서 연속된 n개의 문자를 추출하는 방식입니다. "quick"이라는 단어에서 2-gram(bigram)을 추출하면 "qu", "ui", "ic", "ck"가 됩니다. 3-gram(trigram)을 추출하면 "qui", "uic", "ick"가 됩니다. 이처럼 텍스트 위를 일정 크기의 창문(window)이 한 글자씩 미끄러지며 토큰을 만듭니다.

**ngram 토크나이저**는 이 n-gram 방식을 그대로 적용합니다. 텍스트의 모든 위치에서 지정한 크기의 n-gram 토큰을 생성합니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "tokenizer": {
        "my_ngram": {
          "type": "ngram",
          "min_gram": 2,
          "max_gram": 3,
          "token_chars": ["letter", "digit"]
        }
      },
      "analyzer": {
        "ngram_analyzer": {
          "type": "custom",
          "tokenizer": "my_ngram",
          "filter": ["lowercase"]
        }
      }
    }
  }
}
```

`min_gram`과 `max_gram`은 생성할 n-gram의 최소 길이와 최대 길이입니다. 위 설정에서는 2글자와 3글자짜리 토큰을 모두 생성합니다. `token_chars`는 토큰에 포함할 문자 유형을 지정합니다. "letter"와 "digit"을 넣었으므로 알파벳과 숫자만 토큰에 포함되고, 공백이나 구두점은 토큰 경계로 작용합니다.

"quick"을 이 토크나이저에 통과시키면 2-gram인 "qu", "ui", "ic", "ck"와 3-gram인 "qui", "uic", "ick" 총 일곱 개의 토큰이 생성됩니다. ngram 토크나이저는 부분 문자열 검색에 유용하지만, 토큰 수가 매우 많아지므로 인덱스 크기가 커진다는 단점이 있습니다.

**edge_ngram 토크나이저**는 ngram 토크나이저와 비슷하지만, 토큰의 시작 위치가 항상 단어의 맨 앞입니다. "quick"에서 edge_ngram을 min_gram 1, max_gram 5로 적용하면 "q", "qu", "qui", "quic", "quick" 다섯 개의 토큰이 생성됩니다. 단어의 앞쪽(edge)에서부터 점점 길어지는 토큰을 만드는 것입니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "tokenizer": {
        "my_edge_ngram": {
          "type": "edge_ngram",
          "min_gram": 1,
          "max_gram": 10,
          "token_chars": ["letter"]
        }
      },
      "analyzer": {
        "autocomplete_analyzer": {
          "type": "custom",
          "tokenizer": "my_edge_ngram",
          "filter": ["lowercase"]
        }
      }
    }
  }
}
```

이 분석기로 "quick"을 인덱싱하면 "q", "qu", "qui", "quic", "quick" 토큰이 만들어집니다. 사용자가 검색창에 "qu"만 입력해도 이 토큰과 일치하므로 "quick"이 포함된 문서를 찾을 수 있습니다. 이것이 자동완성의 원리입니다. ngram이 단어의 모든 위치에서 토큰을 만드는 반면, edge_ngram은 앞쪽에서만 만들므로 토큰 수가 적고 인덱스 크기도 작습니다. 자동완성처럼 접두어 일치가 목적이라면 edge_ngram이 더 적합합니다.

두 토크나이저의 차이를 정리합니다. "quick"을 예로 들어 min_gram 2, max_gram 4 설정일 때 각각이 생성하는 토큰은 다음과 같습니다.

```
+-------------+--------------------------------------+
| 토크나이저  | 생성 토큰                            |
+-------------+--------------------------------------+
| ngram       | qu, ui, ic, ck, qui, uic, ick, quic |
| edge_ngram  | qu, qui, quic                        |
+-------------+--------------------------------------+
```

ngram은 모든 위치에서 슬라이딩하며 토큰을 생성하고, edge_ngram은 맨 앞에서만 늘어나며 토큰을 생성합니다.

마지막으로 중요한 제약 사항을 다룹니다. 커스텀 분석기는 인덱스의 settings에 정의되므로, 한번 인덱스를 생성한 뒤에는 분석기를 자유롭게 바꿀 수 없습니다. Elasticsearch에서 인덱스의 분석기 설정은 인덱스 생성 시점에 확정됩니다. 이미 인덱싱된 문서는 생성 당시의 분석기로 처리된 토큰이 역색인에 저장되어 있기 때문입니다.

분석기를 변경하려면 인덱스를 닫았다가 다시 여는 절차가 필요합니다. `POST my_index/_close`로 인덱스를 닫고, 분석기 설정을 변경한 뒤, `POST my_index/_open`으로 다시 엽니다. 그러나 이 방법으로 설정만 바꾸면 기존에 인덱싱된 문서의 토큰은 이전 분석기로 생성된 상태 그대로 남아 있습니다. 새 분석기가 적용된 토큰은 변경 이후에 인덱싱되는 문서에만 반영됩니다.

기존 문서에도 새 분석기를 적용하려면 **재인덱싱**(reindex)이 필요합니다. 재인덱싱은 새 분석기 설정으로 인덱스를 하나 더 만들고, 기존 인덱스의 문서를 새 인덱스로 복사하는 작업입니다. 이 과정에서 모든 문서가 새 분석기로 다시 분석되어 역색인이 새로 만들어집니다. 재인덱싱은 문서 수가 많을수록 시간이 오래 걸리므로, 분석기 설계는 인덱스를 생성하기 전에 충분히 테스트하는 것이 바람직합니다. _analyze API를 활용하면 인덱스를 만들기 전에 분석기의 동작을 확인할 수 있습니다.

정리하면, 커스텀 분석기는 인덱스 settings의 `analysis` 블록에서 캐릭터 필터, 토크나이저, 토큰 필터를 각각 정의하고 `analyzer` 블록에서 조합하여 만듭니다. 동의어 필터를 적용하면 사용자가 다양한 표현으로 검색해도 원하는 문서를 찾을 수 있고, edge_ngram 토크나이저를 사용하면 자동완성 기능을 구현할 수 있습니다. 다만 분석기 설정은 인덱스 생성 시점에 확정되므로, 변경하려면 인덱스를 닫았다 열거나 재인덱싱해야 합니다. 분석기를 설계할 때는 _analyze API로 충분히 테스트한 뒤 인덱스를 생성하는 것이 좋습니다.

다음 단원인 2.3.4에서는 한국어 분석을 위한 Nori 분석기를 다룹니다.

이 단원을 마치면 인덱스 settings에서 커스텀 분석기를 정의하고 매핑에 연결할 수 있고, 동의어 필터를 적용한 분석기를 구성할 수 있습니다.
