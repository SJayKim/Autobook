# 3.4.5 Explain API와 스코어 디버깅

검색 결과의 순서가 기대와 다를 때가 있습니다. 분명히 관련성이 높아 보이는 문서가 목록 아래쪽에 밀려나거나, 관련 없어 보이는 문서가 상위에 올라오는 상황입니다. 3.4.2에서 k1과 b 파라미터를 조정하는 방법을 다루었고, 파라미터 변경 후 Explain API로 점수 변화를 확인할 수 있다고 언급했습니다. 이 단원에서는 Explain API의 사용법을 본격적으로 다루고, 점수 계산 과정을 추적하여 문제의 원인을 진단하는 방법을 살펴봅니다.

점수가 이상하다고 느끼는 상황을 하나 가정합니다. 뉴스 검색 서비스에서 "elasticsearch"로 검색했더니, Elasticsearch를 심도 있게 다룬 기사보다 한 번만 언급한 짧은 공지가 더 높은 점수를 받았습니다. BM25가 IDF, TF 포화, 문서 길이 정규화라는 세 요소로 점수를 계산한다는 것은 3.4.1에서 배웠습니다. 문제가 이 세 요소 중 어디에서 발생하는지 알아야 해결 방향을 잡을 수 있습니다. Explain API는 바로 이 진단 도구입니다. 특정 문서 한 건이 특정 질의에 대해 받은 점수를 구성 요소별로 분해하여 보여줍니다.

**Explain API**를 호출하는 기본 형태는 다음과 같습니다.

```
GET /{index}/_explain/{id}
{
  "query": {
    "match": { "title": "elasticsearch" }
  }
}
```

경로에서 {index}는 검색 대상 인덱스 이름이고, {id}는 점수를 확인하고 싶은 특정 문서의 ID입니다. 요청 본문에는 일반 검색에서 사용하는 것과 동일한 질의를 넣습니다. 이 API는 해당 문서 한 건에 대해서만 동작합니다. 검색 결과 전체의 점수를 한꺼번에 보여주는 것이 아니라, 지정한 문서가 왜 그 점수를 받았는지를 추적하는 것입니다.

Explain API를 호출하면 응답에 **explanation 객체**가 포함됩니다. explanation 객체는 트리 구조로 되어 있습니다. 최상위 노드에 최종 점수가 있고, 그 아래에 점수를 구성하는 하위 계산이 중첩됩니다. 각 노드는 세 가지 정보를 담고 있습니다. value는 해당 단계의 계산 결과 수치입니다. description은 어떤 계산인지를 설명하는 문자열입니다. details는 하위 노드의 배열로, 이 단계가 어떤 하위 계산으로 구성되는지를 보여줍니다.

다음은 응답 구조의 핵심 부분을 간략하게 나타낸 것입니다.

```
{
  "matched": true,
  "explanation": {
    "value": 0.6931,
    "description": "weight(title:elasticsearch in 0)",
    "details": [
      {
        "value": 0.6931,
        "description": "idf, computed as ...",
        "details": [...]
      },
      {
        "value": 1.0,
        "description": "tf, computed as freq/(freq+k1*(1-b+b*dl/avgdl))",
        "details": [
          { "value": 1.0,  "description": "freq, occurrences of term" },
          { "value": 1.2,  "description": "k1, term saturation parameter" },
          { "value": 0.75, "description": "b, length normalization parameter" },
          { "value": 5.0,  "description": "dl, length of field" },
          { "value": 8.2,  "description": "avgdl, average length of field" }
        ]
      }
    ]
  }
}
```

matched 필드는 해당 문서가 질의에 매칭되었는지를 나타냅니다. 매칭되지 않은 문서에 대해서도 Explain API를 호출할 수 있으며, 이 경우 matched가 false이고 왜 매칭되지 않았는지를 설명합니다.

explanation 트리를 위에서 아래로 따라가면 점수가 어떻게 만들어졌는지 읽힙니다. 최상위 value인 0.6931이 최종 점수입니다. 그 아래에 IDF 계산과 TF 계산이 하위 노드로 나뉘어 있습니다. TF 계산의 details를 더 파고들면 freq(단어 등장 횟수), k1, b, dl(해당 문서의 필드 길이), avgdl(전체 문서 평균 필드 길이)이 각각의 수치와 함께 나옵니다. 이 구조를 읽으면 "이 문서는 단어가 1번 등장했고, 필드 길이가 5인데 평균은 8.2이므로 평균보다 짧아서 길이 정규화가 점수를 높이는 방향으로 작용했다"와 같은 판단을 내릴 수 있습니다.

```
explanation 객체의 트리 구조

최종 점수 (value: 0.6931)
 |
 +-- IDF (value: 0.6931)
 |     "이 단어가 전체 문서에서 얼마나 희귀한가"
 |
 +-- TF (value: 1.0)
       "이 단어가 이 문서에서 얼마나 자주 등장하는가"
       |
       +-- freq: 1.0    (단어 등장 횟수)
       +-- k1: 1.2      (TF 포화 파라미터)
       +-- b: 0.75      (길이 정규화 파라미터)
       +-- dl: 5.0      (이 문서의 필드 길이)
       +-- avgdl: 8.2   (전체 문서 평균 필드 길이)
```

검색 결과에서 여러 문서의 점수 내역을 비교하고 싶을 때가 있습니다. Explain API는 문서를 하나씩 지정해야 하므로 여러 문서를 한 번에 보기에는 불편합니다. 이런 경우에는 일반 검색 요청에 **"explain": true** 파라미터를 추가하는 방법이 있습니다.

```
GET /news_index/_search
{
  "explain": true,
  "query": {
    "match": { "title": "elasticsearch" }
  }
}
```

이 요청은 검색 결과의 각 문서(hits 배열의 각 항목)에 _explanation 필드를 추가하여 반환합니다. 구조는 Explain API의 explanation 객체와 동일합니다. 여러 문서를 나란히 놓고 점수 차이의 원인을 비교할 때 편리합니다. 다만, 모든 문서에 대해 상세한 계산 내역을 생성하므로 응답 크기가 크게 늘어납니다. 운영 환경의 일반 검색에 항상 켜 두기보다는, 디버깅이 필요한 시점에 한시적으로 사용하는 것이 적절합니다.

Explain API를 사용하다 보면 의아한 상황을 만날 수 있습니다. 같은 인덱스에서 같은 질의를 실행했는데, 동일한 단어의 IDF 값이 문서마다 다르게 나오는 경우입니다. 이 현상은 Elasticsearch가 점수를 계산하는 방식과 관련이 있습니다.

Elasticsearch의 인덱스는 여러 **샤드**로 나뉘어 있습니다(1.3.3). 기본 검색에서 각 샤드는 자신이 가진 문서만을 기준으로 통계를 계산합니다. IDF를 구할 때 "전체 문서 수 대비 이 단어가 포함된 문서 수"를 사용하는데, 여기서 "전체 문서 수"가 인덱스 전체가 아니라 해당 샤드 안의 문서 수가 되는 것입니다. 샤드마다 문서 분포가 다르면 같은 단어라도 샤드별로 IDF가 달라집니다.

구체적인 시나리오를 봅니다. 인덱스에 프라이머리 샤드가 3개 있고, "elasticsearch"라는 단어가 포함된 문서가 총 90건이라고 합시다. 문서가 균등하게 분포되었다면 각 샤드에 30건씩 들어가서 IDF가 거의 같습니다. 그런데 라우팅이나 문서 입수 순서에 따라 샤드 A에 5건, 샤드 B에 80건, 샤드 C에 5건이 몰렸다면, 샤드 A에서 "elasticsearch"는 매우 희귀한 단어가 되어 IDF가 높게 계산됩니다. 샤드 B에서는 흔한 단어가 되어 IDF가 낮게 계산됩니다.

```
샤드별 IDF 차이 예시

                    샤드 A        샤드 B        샤드 C
전체 문서 수          100          1000           100
"elasticsearch" 포함    5            80             5
IDF 계산 기준      5/100=5%     80/1000=8%     5/100=5%
IDF 결과            높음           낮음           높음
```

같은 내용의 문서라도 어느 샤드에 배치되었느냐에 따라 점수가 달라질 수 있습니다. 문서 수가 적은 인덱스에서 이 현상이 두드러집니다. 문서가 수백만 건 이상이면 샤드 간 분포가 자연스럽게 균등해지므로 차이가 무시할 수 있는 수준으로 줄어듭니다.

이 문제를 해결하는 방법이 **search_type** 파라미터입니다. Elasticsearch의 기본 검색 타입은 query_then_fetch입니다. 각 샤드가 로컬 통계로 점수를 계산한 뒤 결과를 모아 정렬합니다. 이를 **dfs_query_then_fetch**로 바꾸면 동작이 달라집니다.

```
GET /news_index/_search?search_type=dfs_query_then_fetch
{
  "query": {
    "match": { "title": "elasticsearch" }
  }
}
```

dfs_query_then_fetch는 두 단계로 나뉩니다. 첫 번째 단계인 DFS(Distributed Frequency Scatter) 단계에서, 코디네이팅 노드가 각 샤드에 "이 단어의 문서 빈도와 전체 문서 수를 알려 달라"고 요청합니다. 각 샤드가 로컬 통계를 반환하면 코디네이팅 노드가 이를 합산하여 인덱스 전체 기준의 글로벌 통계를 만듭니다. 두 번째 단계에서 이 글로벌 통계를 각 샤드에 전달하고, 샤드는 이 통계를 기반으로 점수를 계산합니다.

```
dfs_query_then_fetch 동작 흐름

[코디네이팅 노드]
     |
     | (1) "통계 알려 달라"
     +----------+----------+
     |          |          |
  [샤드 A]  [샤드 B]  [샤드 C]
     |          |          |
     | (2) 로컬 통계 반환
     +----------+----------+
     |
  [코디네이팅 노드: 글로벌 통계 합산]
     |
     | (3) "이 글로벌 통계로 점수 계산해라"
     +----------+----------+
     |          |          |
  [샤드 A]  [샤드 B]  [샤드 C]
     |          |          |
     | (4) 글로벌 통계 기반 점수 + 문서 반환
     +----------+----------+
     |
  [코디네이팅 노드: 최종 정렬 및 반환]
```

dfs_query_then_fetch를 사용하면 모든 샤드가 동일한 IDF를 사용하므로 점수의 일관성이 보장됩니다. 대신 DFS 단계가 추가되어 네트워크 왕복이 한 번 더 발생합니다. 문서 수가 충분히 많아 샤드 간 통계 차이가 무시할 만한 수준이라면 기본 query_then_fetch를 유지하는 것이 효율적입니다. 문서 수가 적거나, 샤드 간 문서 분포가 불균등한 상황에서 점수 일관성이 중요하다면 dfs_query_then_fetch를 사용합니다.

Explain API로 문제를 진단한 뒤에는 수정하고, 다시 확인하는 과정을 반복합니다. 이 반복 과정을 정리합니다.

```
관련성 튜닝 반복 프로세스

[1단계] 대표 질의 세트를 준비합니다.
        - 실제 사용자가 자주 입력하는 질의 10~20개를 선정합니다.
        - 각 질의에 대해 "이 문서가 상위에 와야 한다"는 기대 결과를 정합니다.

[2단계] 현재 설정으로 검색을 실행하고, 기대와 다른 결과를 식별합니다.

[3단계] Explain API 또는 "explain": true로 문제 문서의 점수를 분해합니다.
        - IDF가 예상보다 낮거나 높은가? --> 샤드별 통계 문제 확인
        - TF가 예상과 다른가? --> k1 조정 검토 (3.4.2)
        - 길이 정규화가 과도하거나 부족한가? --> b 조정 검토 (3.4.2)

[4단계] 원인에 따라 조치합니다.
        - 샤드별 통계 문제: dfs_query_then_fetch 적용
        - 파라미터 문제: k1/b 수정 후 테스트 인덱스에 적용
        - BM25 자체의 한계: function_score로 보정 (3.4.3)

[5단계] 변경 후 같은 대표 질의로 재검색하여 개선 여부를 확인합니다.
        개선되지 않았거나 다른 질의에 부작용이 생겼으면 3단계로 돌아갑니다.
```

1단계에서 기대 결과를 명확히 정하는 것이 중요합니다. "좋은 검색 결과"의 기준이 없으면 점수를 분해해도 무엇이 문제인지 판단할 수 없습니다. 3단계에서 Explain API가 보여주는 수치를 기대 결과와 대조하면, 점수의 어느 구성 요소가 문제인지 좁힐 수 있습니다. 4단계에서 원인별로 다른 조치를 취합니다. 샤드별 통계가 원인이면 search_type을 바꾸고, 파라미터가 원인이면 3.4.2에서 배운 k1/b 조정을 적용하며, BM25 점수만으로 해결이 어려우면 3.4.3의 function_score를 검토합니다. 5단계에서 변경의 효과를 반드시 확인합니다. 한 질의를 개선하면서 다른 질의의 결과가 나빠지는 일이 흔하므로, 대표 질의 세트 전체를 대상으로 평가합니다.

이 과정은 한 번에 끝나지 않습니다. 진단, 수정, 검증을 여러 차례 반복하면서 점진적으로 검색 품질을 높이는 것이 관련성 튜닝의 일반적인 흐름입니다.

정리하면, Explain API는 특정 문서가 특정 질의에 대해 받은 점수를 IDF, TF, 문서 길이 정규화 등 구성 요소로 분해하여 보여주는 진단 도구입니다. GET /{index}/_explain/{id}로 문서 한 건을 지정하거나, 검색 요청에 "explain": true를 추가하여 여러 문서의 점수를 한꺼번에 확인할 수 있습니다. 샤드별로 IDF가 다르게 계산되는 문제는 dfs_query_then_fetch 검색 타입으로 해결합니다. 점수의 원인을 파악한 뒤에는 파라미터 조정, 검색 타입 변경, function_score 적용 등 적절한 조치를 취하고, 대표 질의로 효과를 검증하는 반복 프로세스를 거칩니다.

다음 단원인 4.1.1에서는 집계 구문 구조를 다룹니다. 지금까지 Phase 3에서 검색 질의와 관련성 점수를 다루었다면, Phase 4부터는 검색 결과를 집계하여 통계와 분석 정보를 추출하는 방법으로 넘어갑니다.

이 단원을 마치면 Explain API로 특정 문서의 스코어 계산 과정을 추적할 수 있고, 스코어 이상의 원인을 진단하여 수정 방향을 제시할 수 있습니다.
