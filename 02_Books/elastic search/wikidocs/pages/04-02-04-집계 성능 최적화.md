# 4.2.4 집계 성능 최적화

4.2.1에서 중첩 집계의 메모리 사용, 4.2.3에서 composite 집계의 페이지네이션 방식을 다루면서 집계 성능에 관한 이야기가 여러 번 나왔습니다. 이번 단원에서는 집계 성능에 영향을 미치는 요소를 체계적으로 정리하고, 구체적인 최적화 방법을 하나씩 다룹니다.

집계가 느려지는 근본적인 이유부터 짚습니다. Elasticsearch는 집계를 수행할 때 대상 필드의 모든 값을 읽어야 합니다. 검색은 역색인(inverted index)을 통해 조건에 맞는 문서만 빠르게 찾지만, 집계는 조건에 맞는 문서의 필드 값을 전부 꺼내어 계산합니다. 문서 수가 많거나 필드의 고유 값이 많을수록 읽어야 하는 데이터가 늘어나고, 그만큼 시간과 메모리를 더 소비합니다.

이 필드 값을 읽는 방식이 첫 번째 최적화 포인트입니다. Elasticsearch가 집계에 사용하는 자료 구조는 **doc_values**입니다. doc_values는 인덱싱 시점에 디스크에 미리 기록해 두는 열 지향(columnar) 자료 구조입니다. "열 지향"이라는 말은, 문서 단위가 아니라 필드 단위로 값을 연속 배치한다는 뜻입니다.

일반적인 문서 저장 방식(행 지향)에서는 문서 1의 모든 필드, 문서 2의 모든 필드 순서로 데이터가 나열됩니다. 반면 열 지향 방식에서는 price 필드의 모든 값, category 필드의 모든 값 순서로 나열됩니다. 집계는 하나의 필드에 대해 많은 문서의 값을 읽으므로, 열 지향 방식이 디스크 읽기 효율이 훨씬 높습니다.

```
행 지향 vs 열 지향 비교

행 지향 (역색인과 함께 저장)
  문서1: {price: 1000, category: "의류", brand: "A"}
  문서2: {price: 2000, category: "전자", brand: "B"}
  문서3: {price: 1500, category: "의류", brand: "C"}

열 지향 (doc_values)
  price:    [1000, 2000, 1500, ...]
  category: ["의류", "전자", "의류", ...]
  brand:    ["A", "B", "C", ...]
```

keyword, 숫자, 날짜, boolean, IP, geo_point 타입의 필드는 기본적으로 doc_values가 활성화되어 있습니다. 따라서 이 타입들에 대한 집계는 별도 설정 없이도 doc_values를 사용합니다. 반면 text 타입은 doc_values를 지원하지 않습니다. text 필드에 집계를 걸면 Elasticsearch는 fielddata라는 메모리 기반 자료 구조를 사용하는데, 이는 JVM 힙 메모리를 대량으로 소비하므로 기본적으로 비활성화되어 있습니다. 집계에 사용할 필드는 keyword 타입으로 매핑하는 것이 성능상 올바른 선택입니다.

두 번째 최적화 포인트는 **eager_global_ordinals** 설정입니다. 이 설정을 이해하려면 먼저 global ordinals가 무엇인지 알아야 합니다.

terms 집계처럼 문자열 필드의 고유 값을 기준으로 버킷을 만드는 집계는, 각 문서의 필드 값을 비교하고 그룹화해야 합니다. 그런데 문자열을 매번 직접 비교하면 비용이 큽니다. Elasticsearch는 이 비용을 줄이기 위해 문자열 값에 정수 번호를 부여하고, 이후 비교는 정수로 수행합니다. 이 정수 매핑 테이블을 **global ordinals**라고 부릅니다.

global ordinals는 기본적으로 해당 필드에 첫 집계 요청이 들어올 때 생성됩니다. 인덱스의 세그먼트들을 순회하며 모든 고유 값을 모아 정렬하고 번호를 매기는 과정이므로, 고유 값이 많은 필드에서는 이 초기 생성에 시간이 걸립니다. 이 시간이 첫 번째 집계 요청의 응답 지연에 포함됩니다.

eager_global_ordinals를 true로 설정하면, 인덱스가 리프레시될 때마다 global ordinals를 미리 생성합니다. 집계 요청 시점에는 이미 생성된 ordinals를 바로 사용하므로 응답이 빨라집니다. 대신 리프레시 시간이 길어지고, 생성된 ordinals가 JVM 힙 메모리를 차지합니다.

```json
{
  "mappings": {
    "properties": {
      "category": {
        "type": "keyword",
        "eager_global_ordinals": true
      }
    }
  }
}
```

이 설정은 category 필드의 매핑에 추가합니다. 자주 집계하는 keyword 필드에 적용하면 효과적입니다. 반대로 집계를 거의 하지 않는 필드에 적용하면, 리프레시마다 불필요한 ordinals 생성 비용만 늘어납니다.

세 번째 최적화 포인트는 **filter 집계로 대상 범위를 축소**하는 것입니다. 집계의 대상 문서가 적을수록 계산이 빨라집니다. 4.1.2에서 다룬 filter 집계를 활용하면, 전체 문서가 아니라 특정 조건을 만족하는 문서에 대해서만 집계를 실행할 수 있습니다. 요청 최상위의 query로도 문서를 제한할 수 있지만, filter 집계는 여러 집계 중 특정 집계에만 별도의 조건을 적용하고 싶을 때 유용합니다.

```json
{
  "size": 0,
  "aggs": {
    "recent_orders": {
      "filter": {
        "range": {
          "order_date": { "gte": "2025-01-01" }
        }
      },
      "aggs": {
        "by_category": {
          "terms": { "field": "category", "size": 10 },
          "aggs": {
            "avg_price": {
              "avg": { "field": "price" }
            }
          }
        }
      }
    }
  }
}
```

이 요청은 2025년 이후 주문만 대상으로 카테고리별 평균 가격을 구합니다. 인덱스에 수년치 데이터가 있어도, 집계는 2025년 이후 문서에 대해서만 실행됩니다.

네 번째 최적화 포인트는 **shard_size 파라미터 조정**입니다. 4.1.2에서 terms 집계의 size 파라미터를 다뤘습니다. shard_size는 이와 관련된 내부 동작 파라미터입니다.

Elasticsearch는 분산 시스템이므로 데이터가 여러 샤드에 나뉘어 있습니다. terms 집계를 실행하면 각 샤드가 독립적으로 상위 버킷을 계산한 뒤, 코디네이팅 노드가 결과를 합쳐 최종 상위 N개를 선정합니다. 문제는 각 샤드가 보는 데이터가 전체의 일부라는 점입니다. 샤드 A에서 7번째로 많은 값이, 전체적으로는 3번째로 많을 수 있습니다. 샤드가 size 개수만 반환하면 이 값을 놓칠 수 있습니다.

shard_size는 각 샤드가 코디네이팅 노드에 보내는 버킷 수입니다. 기본값은 size * 1.5 + 10입니다. shard_size를 size보다 크게 설정하면, 각 샤드가 더 많은 후보를 보내므로 최종 결과의 정확도가 높아집니다. 대신 샤드와 코디네이팅 노드 사이에 오가는 데이터가 늘어나 네트워크 부하와 메모리 사용이 증가합니다. 정확도가 매우 중요한 분석에서는 shard_size를 높이고, 빠른 대략적 결과가 필요한 경우에는 기본값을 유지합니다.

다섯 번째 최적화 포인트는 **execution_hint 파라미터**입니다. terms 집계의 내부 실행 방식을 선택하는 옵션입니다. 두 가지 값이 있습니다.

**global_ordinals**는 기본값입니다. 앞서 설명한 global ordinals를 사용하여 문자열 비교를 정수 비교로 대체합니다. 대부분의 경우 이 방식이 빠릅니다.

**map**은 해시 맵 기반으로 동작합니다. 각 문서의 필드 값을 해시 맵에 넣어 버킷을 관리합니다. 고유 값의 수가 매우 적은 필드(수십 개 이하)에서는 global ordinals를 생성하는 오버헤드보다 해시 맵이 가벼울 수 있습니다.

```json
{
  "size": 0,
  "aggs": {
    "by_status": {
      "terms": {
        "field": "status",
        "execution_hint": "map"
      }
    }
  }
}
```

status 필드처럼 "active", "archived", "deleted" 등 몇 가지 값만 있는 경우에 map이 적합할 수 있습니다. 대부분의 필드에서는 기본값인 global_ordinals가 더 효율적이므로, 성능 측정 없이 무조건 map으로 바꾸는 것은 권장하지 않습니다.

여섯 번째 최적화 포인트는 **random_sampler 집계**입니다. Elasticsearch 8.2 버전부터 제공되는 이 집계는 전체 문서 대신 무작위 표본에 대해서만 하위 집계를 실행합니다. 대규모 데이터셋에서 정확한 값보다 빠른 근사치가 필요할 때 유용합니다.

```json
{
  "size": 0,
  "aggs": {
    "sampled": {
      "random_sampler": {
        "probability": 0.1
      },
      "aggs": {
        "avg_price": {
          "avg": { "field": "price" }
        }
      }
    }
  }
}
```

"probability": 0.1은 전체 문서의 약 10%만 표본으로 추출한다는 뜻입니다. 이 표본에 대해서만 avg 하위 집계가 실행됩니다. 1억 건의 문서에 대해 집계하는 대신 1,000만 건만 처리하므로, 응답 시간이 크게 단축됩니다. 결과는 근사치이지만, 표본이 무작위이므로 통계적으로 유의미한 추정을 할 수 있습니다.

마지막으로 **집계 캐싱 조건**을 다룹니다. Elasticsearch는 집계 결과를 캐시에 저장하여 동일한 요청이 다시 들어올 때 재계산 없이 결과를 반환할 수 있습니다. 이 캐시는 샤드 수준 요청 캐시(shard request cache)라고 합니다.

캐시가 동작하려면 두 가지 조건을 만족해야 합니다. 첫째, 요청의 최상위 size가 0이어야 합니다. 검색 결과(hits)를 함께 요청하면 캐시가 적용되지 않습니다. 집계 전용 요청에서만 캐시가 작동합니다. 둘째, 샤드의 데이터가 변경되지 않아야 합니다. 새 문서가 인덱싱되어 샤드가 리프레시되면 해당 샤드의 캐시가 무효화됩니다. 따라서 더 이상 쓰기가 발생하지 않는 과거 인덱스에서 캐시 효과가 가장 큽니다.

날짜 범위를 사용하는 집계에서는 캐시 적중률을 높이는 기법이 있습니다. 현재 시각을 포함하는 쿼리(예: "now-1h")는 초 단위로 값이 바뀌므로 캐시가 거의 적중하지 않습니다. 날짜를 반올림하면 이 문제를 완화할 수 있습니다.

```json
{
  "query": {
    "range": {
      "order_date": {
        "gte": "now-1h/m",
        "lte": "now/m"
      }
    }
  },
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": { "field": "category" }
    }
  }
}
```

"now-1h/m"에서 "/m"은 분 단위로 반올림하라는 뜻입니다. 같은 분 안에 들어온 요청은 동일한 쿼리로 취급되어 캐시를 공유합니다. 초 단위 정확도를 포기하는 대신 캐시 적중률이 크게 올라갑니다.

지금까지 다룬 최적화 방법을 정리합니다.

```
집계 성능 최적화 요약

최적화 방법                적용 대상                      효과
-----------------------   -------------------------     -------------------------
doc_values (기본 활성)     keyword, 숫자, 날짜 필드        열 지향 구조로 빠른 읽기
eager_global_ordinals     자주 집계하는 keyword 필드       리프레시 시 ordinals 사전 생성
filter 집계               범위 축소가 가능한 경우          대상 문서 수 감소
shard_size 조정           terms 집계                     정확도와 성능 균형 조절
execution_hint: map       고유 값이 매우 적은 필드         ordinals 생성 오버헤드 회피
random_sampler            대규모 데이터 근사 분석          표본만 계산하여 응답 시간 단축
size: 0 + 날짜 반올림     집계 전용 요청                   캐시 적중률 향상
```

정리하면, 집계 성능 최적화는 "읽는 데이터를 줄이고, 비교를 빠르게 하고, 재계산을 피하는" 세 가지 방향으로 이루어집니다. doc_values와 global ordinals는 읽기와 비교를 빠르게 하고, filter 집계와 random_sampler는 대상을 줄이며, size: 0과 날짜 반올림은 캐싱을 통해 재계산을 피합니다. 각 방법이 서로 다른 상황에서 효과적이므로, 집계가 느려졌을 때 어떤 원인인지 먼저 파악하고 적합한 방법을 적용해야 합니다.

다음 단원인 4.3.1에서는 Kibana Lens를 다룹니다.

이 단원을 마치면 집계 성능 저하 원인을 진단하고 최적화 방법을 적용할 수 있습니다.
