# 5.2.1 샤드 크기와 수 결정

1.3.3에서 샤드가 인덱스의 물리적 부분 집합이며, 프라이머리 샤드 수는 인덱스 생성 후 변경할 수 없다는 것을 살펴보았습니다. 샤드 하나의 권장 크기가 10~50GB라는 수치도 언급했습니다. 그런데 실제 인덱스를 설계할 때는 "권장 범위가 10~50GB"라는 한 줄만으로는 부족합니다. 왜 이 범위인지, 이 범위를 벗어나면 어떤 문제가 생기는지, 노드의 메모리와 샤드 수는 어떤 관계인지를 알아야 올바른 판단을 내릴 수 있습니다. 이 단원에서는 샤드 크기와 수를 결정하는 기준을 구체적으로 다룹니다.

먼저 **샤드 크기 권장 범위 10~50GB**의 근거를 풀어 봅니다. 각 샤드는 내부적으로 독립된 Lucene 인덱스입니다. Lucene 인덱스는 세그먼트라는 불변 파일의 집합으로 구성되며, 검색 시 모든 세그먼트를 순회합니다. 샤드가 너무 크면 세그먼트 수와 크기가 함께 늘어나 검색과 병합(merge) 작업이 느려집니다. 노드 장애 시 대용량 샤드를 다른 노드로 복구하는 데도 시간이 오래 걸립니다. 반대로 샤드가 너무 작으면, 동일한 데이터를 담기 위해 샤드 수가 지나치게 많아집니다.

**과도한 샤드 수의 오버헤드**를 구체적으로 살펴봅니다. 이 상태를 오버샤딩(oversharding)이라고 합니다. 각 샤드는 클러스터 상태(cluster state)에 메타데이터를 등록합니다. 샤드 수가 1,000개일 때와 100,000개일 때, 마스터 노드가 관리해야 하는 메타데이터 양에 큰 차이가 생깁니다. 클러스터 상태가 커지면 마스터 노드의 메모리 사용량이 늘어나고, 상태 업데이트를 모든 노드에 전파하는 시간도 길어집니다.

각 샤드는 메모리에도 오버헤드를 만듭니다. 세그먼트별 메타데이터, 파일 핸들, 스레드풀 자원 등이 샤드마다 필요합니다. 이 오버헤드가 누적되면 데이터 노드의 JVM 힙 메모리가 부족해질 수 있습니다.

검색 성능에도 영향을 미칩니다. 5.1.3에서 검색이 scatter/gather 패턴으로 동작한다고 했습니다. 코디네이팅 노드가 조율해야 할 샤드 수가 많을수록 scatter 단계에서 보내야 할 요청이 늘어나고, gather 단계에서 합쳐야 할 부분 결과도 많아집니다. 네트워크 왕복과 병합 비용이 증가하여 응답 시간이 길어집니다.

이런 이유로 Elasticsearch는 **힙 메모리당 샤드 수**에 대한 권장 기준을 제시합니다. 노드의 JVM 힙 메모리 1GB당 샤드 20개 이하를 유지하는 것이 권장됩니다. 예를 들어 힙 메모리가 4GB인 노드라면, 해당 노드에 배치되는 샤드 수를 80개 이하로 유지해야 합니다. 힙이 16GB인 노드라면 320개 이하입니다.

이 기준을 실무에 적용하는 방법을 구체적인 수치로 살펴봅니다. **인덱스 크기 예측**이 출발점입니다. 인덱스에 저장될 데이터의 총량을 먼저 추정합니다. 예를 들어 일별 로그 데이터가 50GB씩 생성되는 환경에서 30일치를 보관한다면, 총 데이터 양은 약 1,500GB입니다.

샤드 하나의 크기를 30GB로 잡으면, 필요한 프라이머리 샤드 수는 1,500 / 30 = 50개입니다. 레플리카가 1이면 총 샤드 수는 50 x 2 = 100개입니다. 데이터 노드 3대에 힙 메모리가 각 8GB라면, 노드당 샤드 수는 약 33개(100 / 3)이고, 힙 1GB당 약 4개(33 / 8)이므로 권장 기준(20개 이하) 안에 들어갑니다.

반면 같은 1,500GB를 샤드 크기 2GB로 잡으면, 프라이머리 샤드 750개, 레플리카 포함 1,500개가 됩니다. 노드 3대에 노드당 500개, 힙 1GB당 62.5개로 권장 기준을 크게 초과합니다. 클러스터 상태 관리와 메모리 오버헤드 때문에 성능 문제가 발생할 가능성이 높습니다.

```
샤드 크기별 비교 (데이터 총량 1,500GB, 레플리카 1, 노드 3대)

+------------------+------------------+------------------+
| 샤드 크기        | 30GB             | 2GB              |
+------------------+------------------+------------------+
| 프라이머리 수    | 50               | 750              |
| 총 샤드 수       | 100              | 1,500            |
| 노드당 샤드 수   | ~33              | ~500             |
| 힙 1GB당 샤드    | ~4               | ~62              |
| 판정             | 적정             | 오버샤딩         |
+------------------+------------------+------------------+
```

**샤드 수 변경 불가와 사전 계획**의 중요성을 다시 짚습니다. 1.3.3에서 설명한 것처럼, 프라이머리 샤드 수는 라우팅 공식(`hash(_routing) % number_of_primary_shards`)의 분모이므로 인덱스 생성 후 변경할 수 없습니다. 따라서 인덱스를 만들기 전에 데이터 총량과 증가 추세를 예측하고, 샤드 하나의 크기가 10~50GB 범위에 들어오도록 역산하여 샤드 수를 정해야 합니다.

예측이 빗나갔을 때의 대응 방법은 5.2.2에서 다루는 Split API, Shrink API, 또는 Reindex API를 사용하는 것입니다. 그러나 이 방법들은 모두 추가 작업과 자원이 필요하므로, 처음부터 올바르게 결정하는 것이 가장 효율적입니다.

데이터의 성격에 따라 샤드 전략이 달라집니다. 데이터 총량이 고정되거나 느리게 증가하는 **단일 인덱스**(예: 상품 카탈로그)는, 예상 최대 크기를 기준으로 한 번에 샤드 수를 결정합니다. 반면 **시계열 인덱스**(예: 로그, 메트릭)는 데이터가 계속 쌓이므로, 하나의 인덱스가 무한정 커지지 않도록 주기적으로 새 인덱스를 만드는 전략을 사용합니다. 이 전략의 핵심이 rollover이며, 5.2.2에서 자세히 다룹니다.

현재 클러스터의 샤드 현황을 확인하는 도구가 **_cat/shards API**입니다.

```
GET _cat/shards?v
```

이 요청은 클러스터에 존재하는 모든 샤드의 목록을 반환합니다. 출력에는 인덱스 이름, 샤드 번호, 프라이머리/레플리카 구분, 상태(STARTED, RELOCATING, INITIALIZING, UNASSIGNED), 문서 수, 저장 크기, 배치된 노드 이름이 포함됩니다. 특정 인덱스의 샤드만 확인하려면 인덱스 이름을 경로에 추가합니다.

```
GET _cat/shards/my-index?v
```

출력에서 store 열이 각 샤드의 크기입니다. 이 값이 50GB를 넘는 샤드가 있다면 인덱스 분할을 검토해야 하고, 1GB 미만의 샤드가 지나치게 많다면 오버샤딩을 의심해야 합니다. state 열이 UNASSIGNED인 샤드가 있다면 할당 문제가 있는 것이므로, 5.2.3에서 다루는 allocation 설정을 점검해야 합니다.

정리하면, 샤드 크기는 10~50GB 범위를 유지하는 것이 권장되며, 이 범위를 벗어나면 오버샤딩이나 대용량 샤드로 인한 성능 저하가 발생합니다. 힙 메모리 1GB당 샤드 20개 이하를 기준으로 노드별 샤드 수를 관리합니다. 프라이머리 샤드 수는 인덱스 생성 후 변경할 수 없으므로, 데이터 총량과 증가 추세를 사전에 예측하여 결정해야 합니다. _cat/shards API로 현재 샤드 현황을 모니터링합니다.

다음 단원인 5.2.2에서는 Rollover, Shrink, Split, Clone API를 다룹니다. 샤드 수를 사전에 잘못 결정했거나, 시계열 데이터가 계속 증가하는 상황에서 인덱스 구조를 조정하는 방법을 살펴봅니다.

이 단원을 마치면 인덱스 샤드 수를 결정하는 기준(데이터 크기, 쿼리 처리량)을 적용할 수 있습니다.
