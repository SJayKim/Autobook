# 1.3 데이터 모델: 문서/인덱스/샤드

이 섹션에서는 Elasticsearch가 데이터를 어떤 단위로 저장하고 관리하는지 다룹니다. JSON 문서, 인덱스, 샤드와 레플리카의 개념을 익히고 나면 이후의 매핑 설계와 클러스터 운영 주제가 한결 명확하게 이해됩니다.

## 토픽 목록

**1.3.1 JSON 문서 구조**
- Elasticsearch 문서의 메타데이터 필드를 설명할 수 있다
- 중첩 JSON 구조를 올바르게 인덱싱할 수 있다

_index, _id, _version 등 메타데이터 필드, _source 원본 저장 방식, 중첩 객체(nested object), 배열 필드 처리, null 값 처리, 문서 버전 관리, _seq_no와 _primary_term, 문서 크기 제한을 다룹니다.

**1.3.2 인덱스 개념과 RDBMS 비교**
- Elasticsearch 인덱스를 RDBMS 테이블과 비교하여 차이점을 설명할 수 있다
- 인덱스 설정(settings)과 매핑(mappings)의 역할을 구분할 수 있다

인덱스가 논리적 문서 컨테이너임을 설명하고, RDBMS 테이블과의 비교, 인덱스 settings(샤드 수, 레플리카 수, refresh_interval), mappings 개요, 인덱스 생성 PUT API, _cat/indices로 인덱스 목록 조회, 인덱스 삭제 주의사항을 다룹니다.

**1.3.3 샤드와 레플리카**
- 프라이머리 샤드와 레플리카 샤드의 역할을 설명할 수 있다
- 샤드 수와 레플리카 수 결정 기준을 제시할 수 있다

프라이머리 샤드와 레플리카 샤드의 역할, 샤드 라우팅 공식(hash(_id) % number_of_shards), 레플리카를 통한 고가용성과 읽기 처리량 향상, 샤드 수 변경 불가 제약, 레플리카 수 동적 변경, 샤드 크기 권장 범위(10~50GB)를 다룹니다.
