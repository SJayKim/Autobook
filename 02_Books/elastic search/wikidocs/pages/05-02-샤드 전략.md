# 5.2 샤드 전략

이 섹션에서는 인덱스의 샤드를 어떻게 설계하고 관리하는지 다룹니다. 샤드 수와 크기를 결정하는 기준, 인덱스를 동적으로 관리하는 API들, 그리고 샤드를 특정 노드에 배치하는 Allocation Awareness를 익힙니다.

## 토픽 목록

**5.2.1 샤드 크기와 수 결정**
- 인덱스 샤드 수를 결정하는 기준(데이터 크기, 쿼리 처리량)을 적용할 수 있다

샤드 크기 권장 범위 10~50GB, 과도한 샤드 수의 오버헤드, 힙 메모리당 샤드 수 권장 기준, 인덱스 크기 예측 방법, 샤드 수 변경 불가와 사전 계획의 중요성, _cat/shards API로 현황 확인, 단일 인덱스 vs 시계열 인덱스 전략을 다룹니다.

**5.2.2 Rollover, Shrink, Split, Clone**
- Rollover API로 인덱스를 자동 롤오버하는 조건을 설정할 수 있다
- Shrink API로 샤드 수를 줄이는 전제 조건과 절차를 수행할 수 있다

Rollover API와 조건(max_age, max_size, max_docs), Shrink API와 read-only 전환 전제 조건, Shrink 후 샤드 수 규칙(약수), Split API로 샤드 수 증가, Clone API로 인덱스 복사, ILM과 Rollover 연동, 별칭과 Rollover 연동 패턴을 다룹니다.

**5.2.3 Allocation Awareness**
- Allocation Awareness로 샤드를 특정 속성 기준으로 분산 배치할 수 있다
- Forced Awareness로 단일 장애 영역 격리를 구성할 수 있다

node.attr.* 커스텀 속성 설정, cluster.routing.allocation.awareness.attributes, Forced Awareness 설정, 가용 영역(AZ) 분산 패턴, 랙 인식(rack awareness) 구성, shard allocation filtering(_include/_exclude/_require), allocation explain API를 다룹니다.
