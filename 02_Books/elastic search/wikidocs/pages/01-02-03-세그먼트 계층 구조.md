# 1.2.3 세그먼트 계층 구조

1.2.2에서 역색인이 세그먼트 안에 저장된다고 설명했습니다. 문서가 인덱싱되면 인메모리 버퍼를 거쳐 새 세그먼트가 만들어지고, 한 번 기록된 세그먼트는 수정되지 않는다는 점도 확인했습니다. 그런데 세그먼트가 계속 쌓이기만 하면 어떤 일이 벌어질까요. 검색할 때 모든 세그먼트를 하나씩 탐색해야 하므로 세그먼트가 많아질수록 검색이 느려집니다. 이 단원에서는 세그먼트가 Elasticsearch 전체 구조에서 어디에 위치하는지, 세그먼트가 왜 불변이어야 하는지, 그리고 늘어나는 세그먼트를 어떻게 관리하는지를 단계별로 살펴봅니다.

먼저 세그먼트가 Elasticsearch 전체 계층에서 어디에 놓이는지를 정리합니다. Elasticsearch 클러스터는 여러 대의 노드로 구성됩니다. 하나의 Elasticsearch 인덱스는 여러 개의 **샤드**(shard)로 나뉘어 노드에 분산 저장됩니다. 샤드 하나가 곧 하나의 완전한 Lucene 인덱스입니다. 그리고 하나의 Lucene 인덱스는 하나 이상의 **세그먼트**(segment)로 구성됩니다. 세그먼트는 이 계층의 가장 작은 단위입니다.

```
Elasticsearch 계층 구조

Cluster
  |
  +-- Node A
  |     +-- Shard 0  (= Lucene Index)
  |     |     +-- Segment 0
  |     |     +-- Segment 1
  |     |     +-- Segment 2
  |     +-- Shard 2  (= Lucene Index)
  |           +-- Segment 0
  |
  +-- Node B
        +-- Shard 1  (= Lucene Index)
              +-- Segment 0
              +-- Segment 1
```

다이어그램에서 보듯이, 샤드 0은 세그먼트 세 개를 가지고 있고 샤드 1은 두 개를 가지고 있습니다. 세그먼트 수는 인덱싱과 병합 상황에 따라 샤드마다 다릅니다.

각 세그먼트는 자체적으로 완전한 검색 가능한 소형 인덱스입니다. 1.2.2에서 설명한 텀 딕셔너리, 포스팅 리스트 같은 역색인 구조가 세그먼트마다 독립적으로 들어 있습니다. 그 외에도 Doc Values, Stored Fields, BKD Tree 등 여러 자료구조가 하나의 세그먼트 안에 파일 묶음으로 함께 저장됩니다. 검색 요청이 들어오면 Lucene은 샤드 내의 모든 세그먼트를 탐색한 뒤 결과를 합칩니다. 세그먼트가 독립적인 검색 단위이기 때문에 이런 병렬 탐색이 가능합니다.

이제 세그먼트가 어떻게 만들어지는지를 살펴봅니다. 문서가 Elasticsearch에 도착하면 바로 디스크에 기록되지 않습니다. 먼저 노드 메모리의 **인메모리 버퍼**(IndexWriter buffer)에 저장됩니다. 이 버퍼에서 텍스트 분석과 역색인 데이터 준비가 이루어집니다. 동시에 **트랜잭션 로그**(translog)에도 원본 문서가 기록됩니다. translog는 서버가 갑자기 종료되었을 때 아직 디스크에 영구 저장되지 않은 데이터를 복구하기 위한 안전장치입니다.

인메모리 버퍼에 쌓인 데이터는 주기적으로 새 세그먼트 파일로 변환됩니다. 이 과정을 **리프레시**(refresh)라고 부릅니다. 기본 주기는 1초입니다. 리프레시가 실행되면 버퍼의 내용이 새 세그먼트 파일로 기록되고, 그 파일이 열려서 검색 가능 상태가 됩니다. 이때 운영체제의 fsync를 호출하지 않으므로 세그먼트는 OS 파일시스템 캐시에만 존재합니다. 물리 디스크에 아직 영구 기록된 것은 아니지만, 검색은 가능합니다. 이 특성 때문에 Elasticsearch를 NRT(Near Real-Time) 검색 엔진이라 부릅니다.

리프레시만으로는 데이터가 디스크에 영구 저장되지 않습니다. 영구 저장은 **플러시**(flush)가 담당합니다. 플러시는 기본적으로 30분마다, 또는 translog 크기가 일정 임계값을 초과할 때 실행됩니다. 플러시가 실행되면 fsync를 호출하여 세그먼트를 물리 디스크에 기록하고, translog를 초기화합니다. 플러시가 완료된 이후에는 서버가 종료되어도 해당 데이터가 안전합니다.

인메모리 버퍼의 크기는 indices.memory.index_buffer_size 설정으로 제어합니다. JVM 힙의 퍼센트(예: 10%)나 고정값(예: 100MB)으로 지정할 수 있습니다. 버퍼가 가득 차면 리프레시 주기와 관계없이 자동으로 새 세그먼트로 플러시됩니다.

문서가 세그먼트가 되기까지의 전체 흐름을 정리하면 다음과 같습니다.

```
문서 인덱싱 흐름

문서 도착
  |
  v
In-Memory Buffer (IndexWriter) + Translog 동시 기록
  |
  v
Refresh (기본 1초 간격)
  : 버퍼 내용 --> 새 세그먼트 파일 기록 + 파일 열기
  : fsync 없음, OS 파일 캐시에만 존재
  : --> 이 시점부터 검색 가능 (NRT)
  |
  v
Flush (30분마다 또는 translog 임계 초과)
  : fsync 호출 --> 디스크에 영구 기록
  : translog 초기화
```

여기서 핵심은 리프레시 한 번이 곧 새 세그먼트 하나의 탄생이라는 점입니다. 기본 주기가 1초이므로, 활발히 인덱싱 중인 샤드에서는 매 초마다 세그먼트가 하나씩 생길 수 있습니다. 인덱싱이 계속되면 세그먼트 수가 빠르게 늘어납니다.

세그먼트가 계속 쌓이는데도 기존 세그먼트를 수정하여 합치지 않는 이유가 있습니다. 1.2.2에서 잠시 언급한 **세그먼트 불변성**(segment immutability) 때문입니다. Lucene 세그먼트는 write-once 원칙을 따릅니다. 한 번 디스크에 기록된 세그먼트 파일은 절대 수정되지 않습니다.

불변성이 왜 필요한지 이해하려면 세그먼트가 수정 가능하다고 가정해 봅니다. 세그먼트 파일이 수시로 바뀌면 운영체제의 파일시스템 캐시에 올라간 세그먼트 데이터가 무효화됩니다. 캐시가 무효화되면 다음 검색 때 디스크에서 다시 읽어야 하므로 검색 속도가 떨어집니다. 또한 여러 검색 스레드가 동시에 같은 세그먼트를 읽고 있는데 누군가가 그 파일을 고치면 동시성 문제가 발생합니다. 락을 걸어야 하고, 락은 검색 성능을 저하시킵니다.

세그먼트가 불변이면 이 문제가 사라집니다. 파일이 바뀌지 않으므로 OS 파일시스템 캐시에 올라간 세그먼트는 계속 유효합니다. 캐시 적중률이 높아지고 검색 속도가 안정적으로 유지됩니다. 여러 스레드가 동시에 읽어도 내용이 변하지 않으므로 락이 필요 없습니다.

불변성에는 대가가 있습니다. 문서를 삭제하거나 수정해도 기존 세그먼트의 내용을 직접 지울 수 없습니다. 대신 Lucene은 삭제를 표시하는 별도의 구조를 사용합니다.

문서 삭제 시 Lucene은 해당 세그먼트 내에 **삭제 비트셋**(deletion bitset)을 기록합니다. 비트셋은 세그먼트 안의 각 문서에 대해 "삭제됨" 또는 "유효함"을 한 비트로 표시하는 자료구조입니다. 과거 Lucene 버전에서는 이 비트셋이 **.del 파일**이라는 이름으로 저장되었습니다. 현대 Lucene에서는 LiveDocsFormat이라는 형식으로 관리합니다. 내부적으로 SegmentCommitInfo의 DelGen(deletion generation) 값이 -1이면 해당 세그먼트에 삭제된 문서가 없음을 뜻합니다. 0 이상이면 삭제 정보가 존재합니다.

문서 업데이트도 같은 원리입니다. 기존 문서를 삭제 표시하고, 수정된 내용을 담은 새 문서를 새 세그먼트에 추가합니다. "삭제 마킹 + 신규 추가"의 두 단계로 처리됩니다.

검색 시에는 매번 이 삭제 비트셋을 확인하여 삭제 표시된 문서를 건너뜁니다. 세그먼트 안에 삭제된 문서가 많을수록 건너뛰는 작업이 늘어나 검색 성능이 떨어집니다. 실측에 따르면 삭제 문서 비율이 높은 세그먼트에서 쿼리 유형에 따라 18%에서 46%까지 성능 저하가 발생합니다. 삭제된 문서는 병합 전까지 디스크 공간, 메모리(norms, field data 등 인메모리 자료구조), 문서 ID 슬롯을 모두 점유합니다.

```
삭제 마킹과 검색 시 동작

Segment 0
+-----+-----+-----+-----+-----+
| D1  | D2  | D3  | D4  | D5  |
+-----+-----+-----+-----+-----+

삭제 비트셋 (deletion bitset)
+-----+-----+-----+-----+-----+
|  0  |  1  |  0  |  1  |  0  |     0 = 유효, 1 = 삭제됨
+-----+-----+-----+-----+-----+

검색 시: D1 반환, D2 건너뜀, D3 반환, D4 건너뜀, D5 반환
```

세그먼트가 계속 늘어나고, 삭제된 문서가 디스크와 메모리를 차지하는 문제를 해결하는 것이 **세그먼트 병합**(segment merge)입니다. 병합은 여러 개의 소형 세그먼트를 하나의 대형 세그먼트로 합치는 작업입니다. 병합 과정에서 삭제 표시된 문서가 물리적으로 제거되므로 디스크 공간이 회수됩니다.

Elasticsearch의 기본 병합 정책은 **TieredMergePolicy**입니다. 이 정책은 세그먼트를 크기별로 계층(tier)으로 구분합니다. 어떤 계층에서 세그먼트 수가 임계치를 초과하면 그 계층의 세그먼트들을 하나로 합칩니다. 병합은 I/O 집중 작업이므로 백그라운드 스레드에서 실행됩니다. 사용자의 인덱싱이나 검색 요청을 방해하지 않도록 별도로 처리합니다.

병합이 실제로 진행되는 과정은 세 단계입니다.

```
세그먼트 병합 3단계

[1단계] 선택
  : 병합 정책이 대상 세그먼트들을 선택한다
  : (같은 계층에서 임계치를 초과한 세그먼트들)

[2단계] 재인덱싱
  : 선택된 세그먼트들의 내용을 읽는다
  : 삭제 표시된 문서를 제외하고 새 세그먼트에 기록한다

[3단계] 교체
  : 새 세그먼트를 커밋한다
  : 기존 소형 세그먼트 파일들을 삭제한다
```

TieredMergePolicy에는 동작을 조절하는 주요 파라미터가 있습니다. segments_per_tier는 계층당 최대 세그먼트 수를 정하며 기본값은 10입니다. 이 값을 초과하면 병합이 시작됩니다. max_merge_at_once는 한 번의 병합에서 합칠 수 있는 세그먼트 수의 상한입니다. max_merged_segment는 병합 결과로 만들어지는 세그먼트의 최대 크기를 제한하며 기본값은 5GB입니다. floor_segment는 최소 세그먼트 크기 기준으로, 이보다 작은 세그먼트들은 모두 같은 계층으로 취급합니다.

TieredMergePolicy는 삭제 비율이 높은 세그먼트를 우선적으로 병합 대상으로 선택합니다. 앞에서 삭제 비율이 높으면 검색 성능이 18~46% 저하된다고 했는데, 이런 세그먼트를 먼저 병합하여 성능 저하를 줄이는 것입니다. 다만 최대 크기(5GB)에 도달한 세그먼트는 삭제 비율이 약 50%가 되어야 병합 대상으로 선정됩니다. 이미 크기가 큰 세그먼트를 다시 병합하는 것은 I/O 비용이 높기 때문입니다.

세그먼트 수는 검색 성능에 직접적인 영향을 미칩니다. 검색 시 Lucene은 샤드 내 모든 세그먼트를 탐색합니다. 세그먼트 수가 많을수록 탐색 반복 횟수가 늘어나고 오버헤드가 증가합니다. 리프레시(1초 기본)마다 새 세그먼트가 생기므로, 인덱싱 속도가 병합 속도를 초과하면 세그먼트가 계속 쌓여 검색이 느려집니다.

여기에 인덱싱 성능과 검색 성능 사이의 트레이드오프가 존재합니다. segments_per_tier 값을 낮추면 병합이 더 공격적으로 일어납니다. 세그먼트 수가 줄어들어 검색 성능은 향상되지만, 병합에 소비되는 I/O가 늘어나 인덱싱 성능이 저하됩니다. 반대로 segments_per_tier 값을 높이면 병합 빈도가 줄어들어 인덱싱 성능은 오르지만, 세그먼트 수가 늘어나 검색 성능이 떨어집니다. 실측 가이드에서는 샤드 크기를 20~40GB 범위로 유지하도록 권장합니다. 작은 샤드를 너무 많이 만들면 세그먼트 관리 오버헤드가 과도해지고, 너무 큰 샤드는 검색과 인덱싱 모두 느려집니다.

자동 병합만으로 세그먼트를 최적화하기 어려운 상황이 있습니다. 이때 사용하는 것이 **force merge API**입니다. force merge는 지정한 인덱스의 세그먼트를 강제로 병합하는 명령입니다. 다음과 같이 호출합니다.

```
POST /my_index/_forcemerge?max_num_segments=1
```

이 명령은 my_index의 모든 세그먼트를 단일 세그먼트로 합칩니다. max_num_segments 파라미터가 병합 후 남길 최대 세그먼트 수를 지정합니다. 1로 설정하면 샤드당 세그먼트가 하나만 남으므로 검색 성능이 최대화됩니다.

현재 세그먼트 상태를 확인하려면 다음 API를 사용합니다.

```
GET /my_index/_segments
```

이 명령은 각 샤드의 세그먼트 수, 크기, 삭제 문서 비율 등 상세 정보를 반환합니다. force merge를 실행하기 전에 이 API로 현재 상태를 파악하는 것이 좋습니다.

force merge에는 중요한 주의 사항이 있습니다. force merge는 I/O 집약적 작업이며 OS 파일시스템 캐시를 대량으로 무효화합니다. 활발히 인덱싱 중인 인덱스에 실행하면 클러스터 전체에 부하가 급증합니다. 따라서 force merge는 더 이상 신규 문서가 들어오지 않는 읽기 전용 인덱스에만 적용합니다.

대표적인 사용처는 시계열 데이터입니다. 일별 인덱스를 운영하는 경우, 어제까지의 인덱스는 더 이상 새 문서가 추가되지 않습니다. 이런 과거 인덱스에 max_num_segments=1로 force merge를 실행하면 검색 성능이 향상됩니다. Elasticsearch의 Index Lifecycle Management(ILM) 기능을 사용하면 Warm 단계에서 force merge를 자동 실행하도록 설정할 수 있습니다. 오래된 인덱스가 Warm 단계로 전환될 때 자동으로 단일 세그먼트로 병합되므로 수동 작업이 필요 없습니다.

정리하면, 세그먼트는 Elasticsearch 계층 구조의 가장 작은 단위로, 독립적인 검색 가능한 인덱스 조각입니다. 문서가 인덱싱되면 인메모리 버퍼에 저장된 뒤 리프레시(1초)를 통해 새 세그먼트로 기록되고, 플러시(30분)를 통해 디스크에 영구 저장됩니다. 세그먼트는 불변이므로 캐시 효율과 동시 읽기 성능이 높지만, 삭제된 문서는 비트셋으로 표시만 되고 병합 전까지 공간을 차지합니다. TieredMergePolicy가 백그라운드에서 세그먼트를 병합하여 세그먼트 수를 관리하고, 읽기 전용 인덱스에는 force merge API로 세그먼트를 하나로 합쳐 검색 성능을 극대화할 수 있습니다.

다음 단원인 1.2.4에서는 NRT 검색과 Translog를 다룹니다.

이 단원을 마치면 세그먼트가 생성되고 병합되는 과정을 설명할 수 있고, 세그먼트 수가 검색 성능에 미치는 영향을 설명할 수 있습니다.
