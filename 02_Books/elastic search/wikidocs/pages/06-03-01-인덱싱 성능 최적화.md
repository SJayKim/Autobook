# 6.3.1 인덱싱 성능 최적화

운영 환경에서는 대량의 데이터를 빠르게 Elasticsearch에 넣어야 하는 상황이 빈번합니다. 기존 시스템의 데이터를 마이그레이션하거나, 초기 적재를 수행하거나, 피크 시간대에 로그가 급증하는 경우가 그렇습니다. 기본 설정으로도 인덱싱이 동작하지만, 대량 처리 시에는 설정을 조정하여 처리량을 크게 높일 수 있습니다.

인덱싱 성능에 가장 큰 영향을 주는 설정 중 하나가 **refresh_interval**입니다. 1.2.4에서 다룬 것처럼 Elasticsearch는 주기적으로 리프레시를 수행하여 새로 인덱싱된 문서를 검색 가능한 상태로 만듭니다. 기본값은 1초입니다. 리프레시가 발생할 때마다 새로운 세그먼트가 생성되는데, 대량 인덱싱 중에 1초마다 세그먼트를 만들면 I/O 부담이 커지고 처리량이 떨어집니다.

대량 인덱싱 작업 동안에는 refresh_interval을 늘리거나 비활성화하는 것이 효과적입니다. 값을 -1로 설정하면 리프레시를 완전히 끕니다.

```json
PUT /my-index/_settings
{
  "index": {
    "refresh_interval": "-1"
  }
}
```

인덱싱이 완료된 뒤 다시 기본값으로 복원하면 됩니다. 리프레시를 끄는 동안에는 새로 넣은 문서가 검색에 나타나지 않으므로, 실시간 검색이 필요하지 않은 배치 작업에 적합합니다.

**레플리카 수를 일시적으로 0으로 설정**하는 것도 인덱싱 처리량을 높이는 방법입니다. 레플리카가 있으면 프라이머리 샤드에 문서를 쓸 때마다 레플리카 샤드에도 동일한 쓰기가 발생합니다. 초기 적재 시 레플리카를 0으로 두면 쓰기 작업이 절반으로 줄어듭니다.

```json
PUT /my-index/_settings
{
  "index": {
    "number_of_replicas": 0
  }
}
```

적재가 끝난 뒤 레플리카를 원래 값으로 복원하면 Elasticsearch가 자동으로 레플리카를 생성합니다. 이 기간 동안에는 노드 장애 시 데이터 유실 위험이 있으므로, 원본 데이터가 다른 곳에도 존재하는 상황에서 사용해야 합니다.

3.1.3에서 다룬 **Bulk API**의 배치 크기를 조정하면 인덱싱 효율이 달라집니다. Bulk API는 여러 문서를 한 번의 요청으로 인덱싱합니다. 배치 크기가 너무 작으면 네트워크 왕복 횟수가 늘어나고, 너무 크면 메모리 부담이 커집니다. 일반적으로 5MB에서 15MB 사이의 배치 크기가 적합하다고 알려져 있습니다. 최적의 크기는 문서 크기와 클러스터 환경에 따라 다르므로, 배치 크기를 점진적으로 늘려 가며 처리량이 더 이상 증가하지 않는 지점을 찾는 것이 좋습니다.

또한 단일 스레드가 아닌 여러 스레드에서 동시에 Bulk 요청을 보내면 클러스터의 인덱싱 용량을 더 잘 활용할 수 있습니다. 다만 TOO_MANY_REQUESTS(429) 응답 코드가 반환되면 Elasticsearch가 현재 인덱싱 속도를 감당하지 못한다는 신호이므로, 동시 요청 수를 줄여야 합니다.

**translog(트랜잭션 로그)**의 동작 방식도 인덱싱 성능에 영향을 줍니다. 기본적으로 Elasticsearch는 모든 인덱싱 요청마다 translog에 동기적으로 fsync를 수행합니다. 이는 데이터 안정성을 보장하지만 디스크 I/O 부하를 증가시킵니다. index.translog.durability를 async로 설정하면, translog 쓰기를 비동기로 처리하여 성능이 향상됩니다.

```json
PUT /my-index/_settings
{
  "index": {
    "translog.durability": "async",
    "translog.sync_interval": "5s"
  }
}
```

sync_interval은 비동기 모드에서 translog를 디스크에 기록하는 주기입니다. 이 설정은 해당 간격 동안 시스템 장애가 발생하면 데이터가 유실될 수 있으므로, 데이터 유실을 감수할 수 있는 경우에만 사용합니다.

**매핑 최적화**도 인덱싱 속도에 영향을 줍니다. 불필요한 필드의 인덱싱을 비활성화하면 인덱싱 작업량이 줄어듭니다. 검색에 사용하지 않는 필드는 index: false로 설정하고, 집계에 사용하지 않는 텍스트 필드의 fielddata를 비활성화합니다. 또한 동적 매핑이 활성화되어 있으면 새로운 필드가 등장할 때마다 매핑 업데이트가 발생하므로, 스키마가 확정된 상태에서는 동적 매핑을 끄는 것이 유리합니다.

Elasticsearch의 인덱싱 작업은 **스레드 풀(thread pool)**에서 처리합니다. write 스레드 풀의 크기는 기본적으로 노드의 CPU 코어 수와 같게 설정됩니다. 대부분의 경우 기본값이 적절하지만, 인덱싱 작업이 I/O 바운드인 환경에서는 스레드 풀 큐 크기를 조정하여 처리량을 높일 수 있습니다.

운영체제의 **페이지 캐시**를 효과적으로 활용하는 것도 중요합니다. Elasticsearch는 Lucene을 기반으로 동작하며, Lucene의 세그먼트 파일은 운영체제의 파일시스템 캐시에 올라갑니다. JVM 힙에 메모리를 과도하게 할당하면 페이지 캐시에 남는 메모리가 줄어들어 오히려 성능이 나빠집니다. 이 부분은 6.3.3에서 상세히 다룹니다.

인덱싱이 완료된 뒤, 더 이상 쓰기가 발생하지 않는 인덱스에 대해 **force merge**를 수행하면 검색 성능이 향상됩니다. force merge는 여러 세그먼트를 하나로 합치는 작업입니다. 세그먼트 수가 줄면 검색 시 열어야 하는 파일 핸들이 줄고, 삭제 표시된 문서가 물리적으로 제거됩니다.

```json
POST /my-index/_forcemerge?max_num_segments=1
```

max_num_segments=1은 모든 세그먼트를 하나로 합치라는 의미입니다. 주의할 점은 아직 쓰기가 진행 중인 인덱스에는 force merge를 실행하지 않아야 한다는 것입니다. 새 문서가 들어오면 다시 세그먼트가 생성되므로 병합 효과가 사라지고, 병합 과정에서 리소스만 소모됩니다.

인덱싱 성능 최적화의 전체 흐름을 정리하면 다음과 같습니다.

```
대량 인덱싱 시작
  │
  ├── 1. refresh_interval = -1 (리프레시 비활성화)
  ├── 2. number_of_replicas = 0 (레플리카 제거)
  ├── 3. translog.durability = async (비동기 쓰기)
  ├── 4. Bulk API로 5~15MB 배치 전송
  ├── 5. 여러 스레드에서 동시 전송
  │
  ▼
대량 인덱싱 완료
  │
  ├── 6. refresh_interval 복원
  ├── 7. number_of_replicas 복원
  ├── 8. translog.durability = request 복원
  └── 9. force merge 실행 (쓰기 종료 인덱스)
```

정리하면, 대량 인덱싱 시에는 refresh_interval을 -1로 설정하여 리프레시를 중단하고, 레플리카를 일시적으로 0으로 줄이며, translog를 비동기로 전환하여 쓰기 성능을 높입니다. Bulk API의 배치 크기를 5~15MB로 조정하고, 여러 스레드에서 병렬로 전송합니다. 불필요한 필드의 인덱싱을 비활성화하고, 인덱싱 완료 후 force merge로 세그먼트를 병합합니다. 최적화 설정은 인덱싱이 끝난 뒤 반드시 원래 값으로 복원해야 합니다.

다음 단원인 6.3.2에서는 검색 성능 최적화를 다룹니다. 슬로우 로그 분석과 캐시 활용으로 검색 응답 시간을 개선하는 방법을 알아봅니다.

이 단원을 마치면 대량 인덱싱 시 처리량을 높이는 설정과 기법을 적용할 수 있습니다.
