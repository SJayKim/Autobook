# 2.4.1 런타임 필드

Elasticsearch에서 새로운 필드를 추가하려면, 일반적으로 매핑에 필드를 정의하고 문서를 인덱싱하는 과정을 거칩니다. 2.1.2에서 살펴본 것처럼, 이미 인덱싱된 필드의 타입을 변경하려면 새 인덱스를 만들고 reindex를 수행해야 합니다. 이 과정은 데이터가 많을수록 시간과 비용이 커집니다. 그런데 실제 운영 환경에서는 인덱싱 시점에 예상하지 못했던 필드가 필요해지는 상황이 자주 발생합니다. 예를 들어, 타임스탬프 필드는 이미 인덱싱되어 있는데 요일별 집계가 갑자기 필요해진 경우를 생각해 봅니다. 기존 방식이라면 요일 필드를 매핑에 추가하고 전체 문서를 다시 인덱싱해야 합니다. 문서가 수억 건이라면 이 작업에는 상당한 시간이 소요됩니다.

이런 문제를 해결하기 위해 Elasticsearch 7.11에서 도입된 기능이 **런타임 필드**(runtime field)입니다. 런타임 필드는 인덱싱 시점이 아니라 검색 시점에 값이 계산되는 필드입니다. 문서가 인덱싱될 때 역색인이나 doc_values에 미리 기록되지 않고, 쿼리가 실행되는 순간 각 문서를 읽어서 필드 값을 동적으로 만들어 냅니다.

이 동작 방식을 이해하기 위해 기존 인덱싱 방식과 비교합니다. 기존 방식은 Elasticsearch가 문서를 받아들이는 시점에 분석기를 통해 역색인을 생성하고, 숫자나 날짜 같은 값은 doc_values 구조에 저장합니다. 이렇게 미리 준비된 자료 구조 덕분에 검색이 빠릅니다. 이 접근법을 **Schema on Write**라고 부릅니다. 데이터를 쓰는(write) 시점에 스키마가 확정되고 물리적으로 기록되기 때문입니다.

런타임 필드는 반대 방향입니다. 데이터를 읽는(read) 시점, 즉 검색 시점에 스키마가 적용됩니다. 이 접근법을 **Schema on Read**라고 부릅니다. 인덱싱할 때는 원본 문서만 저장해 두고, 필드의 구조와 값은 쿼리를 실행할 때 결정합니다.

```
Schema on Write (기존 방식)
  문서 인덱싱 -> 분석/변환 -> 역색인, doc_values에 기록
  검색 시 -> 미리 기록된 구조를 조회 -> 빠름

Schema on Read (런타임 필드)
  문서 인덱싱 -> 원본(_source)만 저장
  검색 시 -> 스크립트로 필드 값 계산 -> 유연하지만 느림
```

Elasticsearch는 이 두 방식을 동시에 지원합니다. 하나의 인덱스 안에서 자주 검색하는 필드는 기존 방식으로 인덱싱하고, 간헐적으로 필요한 필드는 런타임 필드로 정의할 수 있습니다. 두 가지 접근법을 상황에 맞게 조합하는 것입니다.

런타임 필드를 정의하는 방법은 두 가지입니다. 첫 번째는 인덱스 매핑에 영구적으로 등록하는 방법이고, 두 번째는 검색 요청에 임시로 포함하는 방법입니다. 먼저 매핑에 등록하는 방법을 살펴봅니다.

매핑에 런타임 필드를 추가할 때는 mappings 안에 **runtime** 섹션을 사용합니다. 기존에 필드를 정의하던 properties 섹션과는 별도의 영역입니다. 아래 예시를 봅니다.

```json
PUT my-index
{
  "mappings": {
    "runtime": {
      "day_of_week": {
        "type": "keyword",
        "script": {
          "source": "emit(doc['@timestamp'].value.dayOfWeekEnum.getDisplayName(TextStyle.FULL, Locale.ROOT))"
        }
      }
    },
    "properties": {
      "@timestamp": { "type": "date" }
    }
  }
}
```

이 매핑에서 properties 섹션에는 @timestamp 필드가 date 타입으로 정의되어 있습니다. 이 필드는 기존 방식대로 인덱싱 시점에 doc_values에 기록됩니다. runtime 섹션에는 day_of_week라는 필드가 정의되어 있습니다. type은 keyword이고, script 안에 이 필드의 값을 어떻게 계산할지가 적혀 있습니다.

script의 source를 풀어 봅니다. doc['@timestamp']는 현재 문서의 @timestamp 필드 값을 읽는 표현입니다. .value로 실제 값을 꺼내고, .dayOfWeekEnum으로 해당 날짜의 요일 열거형을 가져옵니다. .getDisplayName(TextStyle.FULL, Locale.ROOT)는 요일 이름을 "Monday", "Tuesday" 같은 전체 영문 표기로 변환합니다. 마지막으로 emit() 함수가 계산된 값을 런타임 필드의 결과로 내보냅니다.

이 런타임 필드를 매핑에 등록하면, 해당 인덱스에 대한 모든 검색에서 day_of_week 필드를 마치 일반 필드처럼 사용할 수 있습니다. 필터링, 정렬, 집계에 모두 사용 가능합니다. 그러나 인덱싱 시점에 아무것도 기록되지 않으므로, 검색할 때마다 매 문서에 대해 스크립트가 실행됩니다.

여기서 script에 사용되는 언어가 **Painless**입니다. Painless는 Elasticsearch가 내장하고 있는 스크립트 언어입니다. 런타임 필드뿐 아니라 인제스트 파이프라인, 업데이트 스크립트 등 Elasticsearch 내부 여러 곳에서 사용됩니다. Painless 스크립트 안에서 문서 데이터에 접근하는 방식은 두 가지입니다. doc['필드명']은 해당 필드의 doc_values를 읽습니다. 인덱싱된 필드에만 사용할 수 있고 접근 속도가 빠릅니다. params._source는 문서의 원본 JSON 전체를 읽습니다. 인덱싱되지 않은 필드에도 접근할 수 있지만, 원본을 파싱해야 하므로 상대적으로 느립니다.

Painless는 산술 연산, 문자열 조작, 정규 표현식 패턴 매칭 등을 지원합니다. 예를 들어 가격 필드를 수량 필드로 나누어 단가를 구하거나, 문자열에서 특정 패턴을 추출하여 새 필드 값으로 만드는 작업이 가능합니다.

이제 두 번째 정의 방법인 검색 요청 시 임시 정의를 살펴봅니다. 매핑에 런타임 필드를 영구적으로 등록하지 않고, 검색 요청 한 번에만 사용할 필드를 정의하는 것입니다. 이때 사용하는 것이 **runtime_mappings** 파라미터입니다.

```json
GET my-index/_search
{
  "runtime_mappings": {
    "price_per_unit": {
      "type": "double",
      "script": {
        "source": "emit(doc['price'].value / doc['quantity'].value)"
      }
    }
  },
  "query": {
    "range": {
      "price_per_unit": { "lt": 10 }
    }
  }
}
```

이 검색 요청은 price_per_unit이라는 런타임 필드를 임시로 정의합니다. type은 double이며, price 필드 값을 quantity 필드 값으로 나눈 결과를 emit()으로 내보냅니다. 바로 아래 query에서 이 필드를 사용하여 단가가 10 미만인 문서만 필터링합니다.

이 runtime_mappings는 해당 검색 요청 안에서만 유효합니다. 다른 검색 요청에는 영향을 주지 않고, 인덱스 매핑에도 흔적을 남기지 않습니다. 탐색적 분석을 할 때, 즉 어떤 계산 필드가 유용한지 아직 확정하지 않은 상태에서 여러 가지 시도를 해 볼 때 적합합니다.

스크립트 없이 런타임 필드를 정의하는 것도 가능합니다. script를 생략하면 Elasticsearch는 _source에서 해당 필드명과 동일한 이름의 값을 자동으로 가져옵니다. 인덱싱 시점에 매핑에 등록하지 않았지만 _source에는 존재하는 필드를, 검색 가능한 상태로 만들고 싶을 때 사용합니다.

런타임 필드의 유용한 활용 사례 중 하나는 기존 인덱싱된 필드를 덮어쓰는 것입니다. 매핑의 runtime 섹션이나 runtime_mappings에 기존 인덱싱된 필드와 동일한 이름의 런타임 필드를 정의하면, 검색 시 런타임 필드가 우선 적용됩니다. 이를 통해 인덱싱 단계에서 잘못 들어간 값을 reindex 없이 교정할 수 있습니다. 예를 들어, 특정 필드에 오류가 있는 값이 대량으로 인덱싱된 상황에서 런타임 필드로 올바른 값을 계산하도록 스크립트를 작성하면, 검색 결과에서는 교정된 값이 반환됩니다.

이제 런타임 필드의 성능 특성을 구체적으로 살펴봅니다. 런타임 필드는 검색할 때마다 각 문서에 대해 스크립트를 실행합니다. 인덱싱된 필드는 역색인이나 doc_values에서 미리 계산된 결과를 바로 읽기만 하면 되므로, 조회 속도에서 차이가 생깁니다.

구체적으로 비교하면 다음과 같습니다. 인덱싱된 필드는 저장 공간을 소모하지만 검색이 빠릅니다. 런타임 필드는 저장 공간을 소모하지 않지만 검색 시 CPU를 사용하여 계산합니다. 문서 수가 적거나 해당 필드를 드물게 조회하는 경우에는 런타임 필드의 계산 비용이 체감되지 않습니다. 반면 수백만 건 이상의 문서를 대상으로 빈번하게 검색하는 필드라면, 매번 스크립트를 실행하는 비용이 누적되어 응답 시간이 길어집니다.

```
런타임 필드 vs 인덱싱된 필드 비교

                    인덱싱된 필드          런타임 필드
저장 공간           역색인/doc_values 차지  추가 저장 없음
인덱싱 속도         필드 처리 비용 발생     인덱싱 비용 없음
검색 속도           미리 계산된 값 조회     매 쿼리마다 스크립트 실행
매핑 변경 유연성    reindex 필요           스크립트 수정만으로 변경
적합한 용도         자주 검색하는 필드      드물게 사용하는 필드,
                                          탐색적 분석, 임시 필드
```

이 성능 차이 때문에, 런타임 필드를 도입할 때는 단계적 전략이 유효합니다. 처음에는 런타임 필드로 정의하여 유연하게 시작합니다. 스크립트를 수정하거나 삭제하는 것이 자유롭기 때문에, 필드 설계를 확정하기 전에 다양한 계산 로직을 시험할 수 있습니다. 시험 단계를 거쳐 해당 필드가 실제로 자주 사용된다고 판단되면, 그때 인덱싱된 필드로 전환합니다.

전환 방법은 두 가지입니다. 하나는 매핑의 properties 섹션에 해당 필드를 추가한 뒤, 인제스트 파이프라인이나 애플리케이션 로직에서 해당 값을 인덱싱 시점에 계산하여 넣는 것입니다. 새로 인덱싱되는 문서부터 효과가 적용됩니다. 다른 하나는 2.1.2에서 다룬 reindex를 사용하여 기존 문서까지 포함해 일괄 전환하는 것입니다. 전환 후에는 runtime 섹션에서 해당 필드를 제거합니다.

Elasticsearch는 동일한 이름의 필드가 runtime과 properties 양쪽에 모두 존재하면, 인덱싱된 필드를 우선 사용합니다. 따라서 전환 과정에서 런타임 필드와 인덱싱된 필드가 잠시 공존하더라도, 인덱싱된 값이 있는 문서에 대해서는 자동으로 빠른 경로를 탑니다.

```
런타임 필드에서 인덱싱된 필드로의 전환 흐름

1. 런타임 필드로 시작
   runtime 섹션에 day_of_week 정의
   -> 스크립트 수정, 삭제 자유

2. 필드 유용성 확인
   검색 빈도, 응답 시간 관찰
   -> "이 필드가 자주 쓰인다" 판단

3. 인덱싱된 필드로 전환
   properties에 day_of_week 추가
   기존 문서: reindex로 값 기록
   신규 문서: 인덱싱 시 값 계산하여 저장

4. 런타임 필드 정의 제거
   runtime 섹션에서 day_of_week 삭제
   -> 이제 모든 쿼리가 인덱싱된 값을 사용
```

정리하면, 런타임 필드는 검색 시점에 값을 동적으로 계산하는 필드로, 인덱싱 없이도 새로운 필드를 만들고 조회할 수 있게 합니다. 매핑의 runtime 섹션에 영구적으로 등록하거나, 검색 요청의 runtime_mappings에 임시로 정의할 수 있습니다. Painless 스크립트로 계산 로직을 작성하며, 저장 공간을 아끼는 대신 검색 시 CPU 비용이 발생합니다. 따라서 자주 사용하는 필드는 인덱싱 방식이 적합하고, 탐색적 분석이나 드물게 사용하는 필드에 런타임 필드가 적합합니다. 런타임 필드로 시작하여 사용 패턴을 확인한 뒤, 필요에 따라 인덱싱된 필드로 전환하는 전략이 효과적입니다.

다음 단원인 2.4.2에서는 중첩 문서 vs Parent-Child 조인을 다룹니다.

이 단원을 마치면 런타임 필드를 정의하고 검색 시 동적으로 계산된 값을 조회할 수 있으며, 런타임 필드의 성능 트레이드오프를 설명할 수 있습니다.
