# 1.2.2 역색인 동작 원리

1.2.1에서 Lucene이 역색인, 쿼리 파서, 스코어링 등 검색의 핵심 기능을 제공하는 라이브러리라고 설명했습니다. 그중에서도 역색인은 Lucene 검색 성능의 근간입니다. 검색 엔진이 수백만 건의 문서 속에서 특정 단어가 포함된 문서를 밀리초 안에 찾아낼 수 있는 이유가 바로 역색인에 있습니다. 이 단원에서는 역색인이 어떤 구조로 이루어져 있고, 문서가 들어왔을 때 역색인이 어떤 과정을 거쳐 만들어지며, 한 번 만들어진 역색인이 왜 수정되지 않는지를 단계별로 살펴봅니다.

역색인을 이해하려면 먼저 그 반대 방향인 **전방 색인**(forward index)부터 살펴보는 것이 자연스럽습니다. 전방 색인은 문서 ID를 키로, 그 문서가 포함하는 단어 목록을 값으로 저장하는 구조입니다. 예를 들어 세 개의 문서가 있다고 가정합니다.

```
문서 1: "검색 엔진은 빠르다"
문서 2: "엔진 설계가 중요하다"
문서 3: "빠른 검색이 핵심이다"
```

전방 색인은 이 문서들을 다음과 같이 저장합니다.

```
전방 색인 (Forward Index)

문서 ID  |  포함 단어
---------|------------------
1        |  검색, 엔진, 빠르다
2        |  엔진, 설계, 중요하다
3        |  빠른, 검색, 핵심이다
```

이 구조는 "문서 1에 어떤 단어가 있는가?"라는 질문에는 즉시 답할 수 있습니다. 문서 1의 행을 읽으면 됩니다. 그러나 검색 엔진이 실제로 받는 질문은 그 반대입니다. "'검색'이라는 단어가 들어 있는 문서는 무엇인가?"입니다. 전방 색인으로 이 질문에 답하려면 모든 문서의 단어 목록을 처음부터 끝까지 훑어야 합니다. 문서가 세 개일 때는 괜찮지만, 문서가 수백만 건이면 매 검색마다 전체를 순회해야 하므로 실용적이지 않습니다.

**역색인**(inverted index)은 이 방향을 뒤집습니다. 단어를 키로, 그 단어가 등장하는 문서 ID 목록을 값으로 저장합니다. 같은 세 문서를 역색인으로 만들면 다음과 같습니다.

```
역색인 (Inverted Index)

단어      |  문서 ID 목록
----------|-------------
검색      |  [1, 3]
엔진      |  [1, 2]
빠르다    |  [1]
설계      |  [2]
중요하다  |  [2]
빠른      |  [3]
핵심이다  |  [3]
```

이제 "'검색'이 포함된 문서를 찾아라"는 질문에 역색인의 '검색' 항목만 조회하면 됩니다. 문서 1과 문서 3이라는 답이 즉시 나옵니다. 문서가 수백만 건이어도 검색 시간은 단어 조회 한 번에 거의 고정됩니다. 이것이 역색인의 핵심 트레이드오프입니다. 문서를 저장하는 시점(인덱싱)에 역색인을 미리 만들어두는 비용을 선불로 지불하고, 검색 시점에는 비용을 대폭 줄이는 것입니다.

역색인이 만들어지려면 원본 텍스트를 개별 단어 단위로 쪼개는 과정이 필요합니다. 이 과정을 **토큰화**(tokenization)라고 부릅니다. 더 정확하게는 Elasticsearch에서 텍스트 분석(text analysis)이라 불리는 3단계 파이프라인의 일부입니다. 3단계는 다음과 같습니다.

```
텍스트 분석 파이프라인

원본 텍스트
  |
  v
[1단계] Character Filter  --- 문자 수준 전처리
  |                            (HTML 태그 제거, 특수문자 치환 등)
  v
[2단계] Tokenizer          --- 텍스트를 개별 토큰으로 분리
  |                            (공백/구두점 기준, 위치와 오프셋 기록)
  v
[3단계] Token Filter       --- 토큰 후처리
  |                            (소문자 변환, 불용어 제거, 어간 추출 등)
  v
최종 토큰 목록 --> 역색인에 저장
```

1단계인 Character Filter는 텍스트를 문자 수준에서 전처리합니다. HTML 태그를 제거하거나 특정 문자를 다른 문자로 치환하는 작업이 여기에 해당합니다. 예를 들어 원본에 "<b>검색</b> 엔진"이라는 텍스트가 있다면, Character Filter가 HTML 태그를 벗겨내어 "검색 엔진"만 남깁니다.

2단계인 Tokenizer는 전처리된 텍스트를 개별 토큰으로 분리합니다. Elasticsearch의 기본 Tokenizer인 Standard Tokenizer는 유니코드 텍스트 분할 규칙(Unicode Text Segmentation)에 따라 공백과 단어 경계에서 텍스트를 나눕니다. 이때 각 토큰의 위치(position)와 원본 텍스트에서의 시작/끝 문자 오프셋(offset)도 함께 기록합니다. 위치 정보는 나중에 구문 검색(phrase search)에서 단어 순서를 확인할 때 사용됩니다.

3단계인 Token Filter는 분리된 토큰을 후처리합니다. 대문자를 소문자로 변환하거나, "the", "is" 같은 불용어(stopword)를 제거하거나, "running"을 "run"으로 바꾸는 어간 추출(stemming)을 수행합니다. Token Filter는 선택 사항이며 여러 개를 연쇄적으로 적용할 수 있습니다.

한 가지 중요한 점이 있습니다. 문서를 인덱싱할 때 적용하는 분석기와 검색어를 처리할 때 적용하는 분석기가 동일해야 한다는 것입니다. 인덱싱 시 "Running"을 소문자 변환하여 "running"으로 저장했는데, 검색 시에는 소문자 변환 없이 "Running" 그대로 역색인에서 찾으면 일치하는 항목이 없습니다. 양쪽에 같은 분석기를 적용해야 일관된 매칭이 가능합니다.

토큰화를 거쳐 최종 토큰 목록이 완성되면, 이 토큰들이 역색인에 저장됩니다. 역색인의 내부는 크게 두 부분으로 나뉩니다. 하나는 **텀 딕셔너리**(term dictionary)이고, 다른 하나는 **포스팅 리스트**(posting list)입니다.

텀 딕셔너리는 세그먼트 내 모든 고유 단어(term)를 정렬된 형태로 보관하는 자료구조입니다. 1.2.1에서 세그먼트가 역색인을 포함하는 최소 단위라고 설명했는데, 하나의 세그먼트 안에 존재하는 모든 고유 단어가 이 딕셔너리에 등록됩니다. 각 단어 항목에는 그 단어를 포함하는 문서의 수와 포스팅 리스트로 향하는 포인터가 함께 저장됩니다. Lucene은 텀 딕셔너리를 FST(Finite State Transducer, 유한 상태 변환기)라는 자료구조로 구현합니다. FST는 정렬된 단어 목록을 공통 접두사와 접미사를 공유하는 그래프로 압축하여 메모리를 절약하면서도 특정 단어를 빠르게 찾을 수 있게 해줍니다. 텀 딕셔너리는 메모리에 로드되어 동작하므로, 검색 요청이 들어왔을 때 디스크 접근 없이 단어를 조회할 수 있습니다.

포스팅 리스트는 특정 단어가 등장하는 문서들의 목록입니다. 텀 딕셔너리에서 단어를 찾으면 포인터를 따라 해당 단어의 포스팅 리스트에 도달합니다. 포스팅 리스트의 각 항목(포스팅)에는 문서 ID, 해당 문서 내에서의 단어 등장 횟수, 단어 위치, 문자 오프셋이 기록됩니다. 포스팅은 문서 ID 순서로 정렬되어 있습니다.

텀 딕셔너리와 포스팅 리스트가 어떻게 연결되는지 구조를 정리하면 다음과 같습니다.

```
텀 딕셔너리 (Term Dictionary)         포스팅 리스트 (Posting List)
+----------+----+---------+
| 단어     | DF | 포인터  |
+----------+----+---------+
| 검색     |  2 |  -----> | doc1: TF=1, pos=[0] | doc3: TF=1, pos=[1] |
| 빠르다   |  1 |  -----> | doc1: TF=1, pos=[2] |
| 빠른     |  1 |  -----> | doc3: TF=1, pos=[0] |
| 설계     |  1 |  -----> | doc2: TF=1, pos=[1] |
| 엔진     |  2 |  -----> | doc1: TF=1, pos=[1] | doc2: TF=1, pos=[0] |
| ...      |    |         |
+----------+----+---------+

DF = Document Frequency (해당 단어를 포함하는 문서 수)
TF = Term Frequency (해당 문서 내 단어 등장 횟수)
pos = 문서 내 단어 위치
```

이 구조 덕분에 검색은 두 단계만 거치면 됩니다. 먼저 텀 딕셔너리에서 검색 단어를 찾고, 이어서 해당 포스팅 리스트를 읽어 문서 ID 목록을 얻습니다.

포스팅 리스트가 문서 ID 순서로 정렬되어 있다는 점은 여러 단어를 동시에 검색할 때 큰 이점을 줍니다. 예를 들어 "검색 AND 엔진"이라는 쿼리가 들어오면, '검색'의 포스팅 리스트 [1, 3]과 '엔진'의 포스팅 리스트 [1, 2]를 동시에 앞에서부터 훑으며 교집합을 구합니다. 양쪽 모두 정렬되어 있으므로 한 번의 순차 탐색으로 공통 문서 ID인 1을 찾아낼 수 있습니다. 포스팅 리스트가 길어질 경우에는 스킵 리스트(skip list) 계층 구조를 활용하여 불필요한 구간을 건너뛰면서 교집합이나 합집합 연산을 빠르게 수행합니다.

포스팅 리스트는 문서가 많아질수록 커집니다. 흔한 단어일수록 등장하는 문서가 많으므로 포스팅 리스트도 그만큼 길어집니다. Lucene은 이를 압축하기 위해 **델타 인코딩**(delta encoding)을 사용합니다. 델타 인코딩은 절대값 대신 이전 값과의 차이만 저장하는 방식입니다. 예를 들어 문서 ID 목록이 [1, 4, 6, 9]라면, 첫 값은 그대로 1을 저장하고 이후부터는 차이값만 저장하여 [1, 3, 2, 3]으로 변환합니다. 차이값은 원래 값보다 작으므로 더 적은 비트로 표현할 수 있고, 여기에 Variable Byte Encoding까지 적용하면 저장 공간이 크게 줄어듭니다.

역색인에 저장되는 수치 중 검색 결과의 순위를 결정하는 데 쓰이는 두 가지가 있습니다. 하나는 **텀 빈도**(TF, Term Frequency)이고, 다른 하나는 **도큐먼트 빈도**(DF, Document Frequency)입니다.

텀 빈도는 특정 문서 안에서 해당 단어가 몇 번 등장하는지를 나타냅니다. 어떤 문서에 "검색"이라는 단어가 5번 등장한다면 그 문서의 TF는 5입니다. TF가 높을수록 해당 문서가 그 단어와 관련성이 높다고 판단합니다. 이 값은 포스팅 리스트의 각 항목에 저장됩니다.

도큐먼트 빈도는 전체 문서 집합에서 해당 단어가 등장하는 문서의 수입니다. "검색"이라는 단어가 1000개 문서 중 50개에 등장한다면 DF는 50입니다. DF가 높은 단어, 예를 들어 "the"나 "is" 같은 단어는 거의 모든 문서에 등장하므로 특정 문서를 구별하는 데 도움이 되지 않습니다. 이 값은 텀 딕셔너리에 저장됩니다.

1.2.1에서 Elasticsearch가 BM25를 기본 스코어링 알고리즘으로 사용한다고 설명했습니다. BM25는 TF와 DF를 결합하여 점수를 산출합니다. DF가 낮은 단어, 즉 드물게 등장하는 단어일수록 IDF(Inverse Document Frequency) 값이 높아져 가중치가 커집니다. 반대로 DF가 높은 흔한 단어는 가중치가 줄어듭니다. TF에 대해서는 포화 곡선(saturation curve)을 적용합니다. 단어가 처음 몇 번 등장할 때는 점수가 크게 오르지만, 반복이 계속되어도 점수 증가가 점차 둔화됩니다. BM25의 기본 파라미터인 k1=1.2가 이 포화 속도를 제어하고, b=0.75가 문서 길이 정규화의 강도를 제어합니다.

지금까지 역색인의 구조를 살펴보았으니, 이제 역색인이 언제 만들어지는지를 정리합니다. 역색인은 문서가 인덱싱되는 시점에 구축됩니다. 문서 한 건이 Elasticsearch에 도착해서 검색 가능한 상태가 되기까지의 흐름은 다음과 같습니다.

```
역색인 빌드 과정

문서 도착
  |
  v
[1단계] 인메모리 버퍼 적재
  |      IndexWriter가 문서를 메모리 버퍼에 저장
  |      동시에 트랜잭션 로그(translog)에도 기록 (내구성 보장)
  |
  v
[2단계] 리프레시 (Refresh) --- 기본 1초 간격
  |      버퍼 내용을 새 세그먼트 파일로 기록 + 파일 열기
  |      (fsync 없음, OS 파일시스템 캐시에만 존재)
  |      --> 이 시점부터 검색 가능 (Near Real-Time)
  |
  v
[3단계] 플러시 (Flush) --- 30분마다 또는 translog 임계 초과 시
         세그먼트를 디스크에 영구 기록 (fsync)
         translog 초기화
```

1단계에서 문서가 도착하면 IndexWriter가 해당 문서를 인메모리 버퍼에 적재합니다. 이 시점에서 텍스트 분석(토큰화)이 수행되고 역색인 데이터가 메모리에 준비됩니다. 동시에 트랜잭션 로그(translog)에도 원본 문서가 기록됩니다. translog는 서버가 갑자기 종료되었을 때 아직 디스크에 영구 저장되지 않은 데이터를 복구하기 위한 안전장치입니다.

2단계인 리프레시는 기본적으로 1초마다 실행됩니다. 인메모리 버퍼에 쌓인 내용을 새로운 세그먼트 파일로 기록하고 해당 파일을 엽니다. 이때 운영체제의 fsync를 호출하지 않으므로 세그먼트는 OS 파일시스템 캐시에만 존재하는 상태입니다. 그럼에도 IndexSearcher가 이 세그먼트를 열 수 있으므로, 리프레시 이후부터 해당 문서가 검색 결과에 나타납니다. 문서 인덱싱부터 검색 가능 상태까지 보통 1초 이내이므로, Elasticsearch를 '근실시간'(Near Real-Time, NRT) 검색 엔진이라고 부릅니다.

3단계인 플러시는 30분마다 또는 translog 크기가 임계값을 초과할 때 실행됩니다. 이 단계에서 fsync를 호출하여 세그먼트를 디스크에 영구 기록하고, translog를 초기화합니다. 플러시가 완료되면 서버가 종료되어도 해당 세그먼트의 데이터는 안전합니다.

마지막으로 역색인의 중요한 설계 원칙인 **불변성**(immutability)을 살펴봅니다. 1.2.1에서 세그먼트가 불변이라고 언급했는데, 그 의미를 구체적으로 풀어봅니다.

Lucene 세그먼트는 한 번 디스크에 기록되면 절대 수정되지 않습니다. 새로운 문서가 들어오면 기존 세그먼트를 고치는 것이 아니라, 항상 새로운 세그먼트를 만들어 추가합니다. 문서를 수정할 때도 마찬가지입니다. 기존 세그먼트에서 해당 문서에 삭제 표시(tombstone)만 남기고, 수정된 내용을 담은 문서를 새 세그먼트에 추가합니다. 문서 삭제도 동일하게 삭제 표시만 기록합니다. 실제로 물리적 삭제가 일어나는 시점은 세그먼트 병합(segment merge) 때입니다.

```
불변 세그먼트와 문서 변경 처리

[시점 1] 문서 A 인덱싱
  Segment 0: [문서 A]

[시점 2] 문서 B, C 인덱싱
  Segment 0: [문서 A]
  Segment 1: [문서 B, 문서 C]

[시점 3] 문서 A 수정
  Segment 0: [문서 A (삭제 표시)]   <-- 기존 세그먼트는 수정하지 않음
  Segment 1: [문서 B, 문서 C]
  Segment 2: [문서 A']             <-- 수정된 문서 A는 새 세그먼트에

[시점 4] 세그먼트 병합 (Merge)
  Segment 3: [문서 A', 문서 B, 문서 C]   <-- 삭제 표시된 문서 A는 제외
```

불변성은 두 가지 실질적인 이점을 가져옵니다. 첫째, 세그먼트 파일이 변하지 않으므로 운영체제가 파일시스템 캐시에 세그먼트를 올려두면 해당 캐시가 무효화될 일이 없습니다. 캐시 적중률이 높아지므로 검색 속도가 빨라집니다. 둘째, 여러 검색 요청이 동시에 같은 세그먼트를 읽더라도 내용이 바뀌지 않으므로 락(lock)이 필요 없습니다. 동시 읽기 성능이 그만큼 높아집니다.

다만 불변성에는 대가가 있습니다. 시간이 지나면 작은 세그먼트가 계속 누적됩니다. 검색 시에는 모든 세그먼트를 순차적으로 탐색해야 하므로, 세그먼트 수가 많아지면 검색 성능이 저하됩니다. Lucene은 이 문제를 해결하기 위해 백그라운드에서 소형 세그먼트 여러 개를 하나의 대형 세그먼트로 병합합니다. 병합 과정에서 삭제 표시된 문서가 물리적으로 제거되므로 디스크 공간도 회수됩니다.

정리하면, 역색인은 단어를 키로 문서 ID 목록을 값으로 저장하는 자료구조로, 인덱싱 시점에 비용을 선불로 지불하여 검색 시점의 조회를 극도로 빠르게 만드는 것이 핵심 설계입니다. 내부는 단어를 빠르게 찾기 위한 텀 딕셔너리와, 해당 단어가 등장하는 문서 목록을 담은 포스팅 리스트로 구성됩니다. 문서가 도착하면 텍스트 분석 파이프라인을 거쳐 토큰으로 분리되고, 인메모리 버퍼에서 리프레시를 통해 새 세그먼트로 기록되어 근실시간으로 검색 가능해집니다. 한 번 기록된 세그먼트는 수정되지 않으며, 이 불변성이 캐시 효율과 동시 읽기 성능을 보장합니다.

다음 단원인 1.2.3에서는 세그먼트 계층 구조를 다룹니다.

이 단원을 마치면 역색인의 구조와 구축 과정을 다이어그램으로 설명할 수 있고, 전방 색인과 역색인의 차이를 비교할 수 있습니다.
