# 4.3.3 KQL (Kibana Query Language)

4.3.1에서 Lens로 시각화를 만들고, 4.3.2에서 대시보드에 여러 패널을 배치하는 방법을 다뤘습니다. 시각화와 대시보드가 데이터를 보여 주는 틀이라면, 그 틀 안에 어떤 데이터를 담을지 결정하는 것은 필터링입니다. Kibana 화면 상단에는 검색 바가 있습니다. Discover, Lens, Dashboard 등 대부분의 Kibana 화면에서 이 검색 바를 볼 수 있으며, 여기에 필터 조건을 입력하면 해당 조건에 맞는 문서만 화면에 표시됩니다. 이 검색 바에서 사용하는 기본 쿼리 언어가 **KQL(Kibana Query Language)**입니다.

KQL은 Elasticsearch의 Query DSL을 Kibana 사용자가 간편하게 사용할 수 있도록 만든 쿼리 문법입니다. Query DSL은 JSON 형태로 구성해야 하므로 중괄호, 키-값 쌍, 중첩 구조를 일일이 작성해야 합니다. KQL은 이보다 훨씬 간결합니다. 필드 이름, 콜론, 값을 한 줄로 적으면 됩니다.

KQL의 기본 문법은 "field: value" 형태입니다. 콜론 앞에 필드 이름을 쓰고, 콜론 뒤에 찾고자 하는 값을 씁니다. 예를 들어 status 필드 값이 200인 문서를 찾으려면 다음과 같이 입력합니다.

```
status: 200
```

이 한 줄이 Elasticsearch 내부에서는 term 쿼리 또는 match 쿼리로 변환됩니다. 필드 타입에 따라 동작이 달라집니다. keyword 필드에 대해서는 정확히 일치하는 값을 찾고, text 필드에 대해서는 분석기를 거친 뒤 매칭합니다. 값에 공백이 포함된 문구를 정확히 찾고 싶으면 따옴표로 감쌉니다.

```
message: "connection refused"
```

따옴표 없이 message: connection refused라고 쓰면, "connection"과 "refused"를 개별 토큰으로 취급하여 둘 중 하나만 포함된 문서도 반환될 수 있습니다. 정확한 문구 매칭이 필요하면 반드시 따옴표를 사용합니다.

여러 조건을 조합할 때 **AND, OR, NOT 연산자**를 사용합니다. 연산자는 반드시 대문자로 씁니다. 소문자 and, or, not은 연산자가 아니라 일반 검색어로 취급됩니다.

AND는 두 조건을 모두 만족하는 문서를 찾습니다.

```
status: 200 AND extension: html
```

이 쿼리는 status가 200이면서 extension이 html인 문서만 반환합니다.

OR는 두 조건 중 하나라도 만족하는 문서를 찾습니다.

```
status: 200 OR status: 301
```

이 쿼리는 status가 200이거나 301인 문서를 반환합니다. 같은 필드에 여러 값을 OR로 연결하는 경우, 괄호를 사용하면 더 간결하게 쓸 수 있습니다.

```
status: (200 OR 301 OR 404)
```

괄호 안에 OR로 나열한 값 중 하나라도 일치하면 해당 문서가 반환됩니다.

NOT은 특정 조건을 제외합니다.

```
NOT status: 500
```

이 쿼리는 status가 500이 아닌 모든 문서를 반환합니다. AND, OR와 결합하여 사용할 수도 있습니다.

```
status: 200 AND NOT extension: css
```

이 쿼리는 status가 200이면서 extension이 css가 아닌 문서를 찾습니다.

연산자를 여러 개 조합할 때는 괄호로 우선순위를 명시하는 것이 좋습니다. KQL에서 AND는 OR보다 우선순위가 높습니다. 다음 두 쿼리는 다른 결과를 반환합니다.

```
status: 200 OR status: 301 AND extension: html
```

이 쿼리는 AND가 먼저 평가되므로, "status가 301이면서 extension이 html인 문서" 또는 "status가 200인 문서"를 반환합니다. 의도가 "(status가 200이거나 301인 문서) 중에서 extension이 html인 문서"였다면, 괄호를 사용해야 합니다.

```
(status: 200 OR status: 301) AND extension: html
```

정확한 값을 모를 때 **와일드카드**를 사용합니다. 별표(*)는 0개 이상의 임의 문자를 대신합니다.

```
host.name: web-server-*
```

이 쿼리는 host.name이 "web-server-"로 시작하는 모든 문서를 찾습니다. "web-server-01", "web-server-prod", "web-server-east-1" 등이 모두 매칭됩니다. 와일드카드는 값의 앞, 중간, 뒤 어디에나 놓을 수 있습니다.

```
message: *timeout*
```

이 쿼리는 message 필드에 "timeout"이라는 문자열이 포함된 문서를 찾습니다. 다만 와일드카드를 값의 맨 앞에 놓는 패턴(예: *timeout)은 인덱스의 모든 값을 스캔해야 하므로 성능이 떨어질 수 있습니다. 데이터 양이 많은 인덱스에서는 사용을 최소화하는 것이 좋습니다.

숫자나 날짜 필드에서 특정 범위의 데이터를 찾을 때 **범위 쿼리**를 사용합니다. 부등호 연산자(>, <, >=, <=)를 지원합니다.

```
response_time >= 500
```

이 쿼리는 response_time이 500 이상인 문서를 찾습니다. 두 개의 범위 조건을 AND로 결합하면 구간을 지정할 수 있습니다.

```
response_time >= 500 AND response_time < 1000
```

이 쿼리는 response_time이 500 이상 1000 미만인 문서를 반환합니다. 날짜 필드에도 동일하게 적용됩니다.

```
@timestamp >= "2025-01-01" AND @timestamp < "2025-02-01"
```

이 쿼리는 2025년 1월에 해당하는 문서를 찾습니다. 다만 시간 범위 필터링은 Kibana 상단의 시간 필터를 사용하는 것이 더 편리하므로, 범위 쿼리는 시간 외의 숫자 필드에서 주로 활용합니다.

Elasticsearch 문서에 중첩된 구조가 있을 때, KQL에서 **중첩 필드**를 쿼리하는 방법을 설명합니다. 문서의 필드가 점(.)으로 구분된 계층 구조를 가질 수 있습니다. 예를 들어 geo.country_name이나 http.response.status_code 같은 필드입니다. KQL에서 이런 필드를 쿼리할 때는 점 표기법을 그대로 사용합니다.

```
geo.country_name: "South Korea"
```

```
http.response.status_code: 404
```

점으로 구분된 필드 이름을 그대로 적으면 됩니다. 이것은 Elasticsearch의 object 타입 필드를 쿼리하는 방식입니다. nested 타입으로 매핑된 필드의 경우에도 KQL에서는 동일한 점 표기법으로 접근합니다. Kibana가 내부적으로 nested 쿼리로 변환해 줍니다.

Kibana의 검색 바에는 KQL 외에 **Lucene 쿼리 구문**도 사용할 수 있습니다. 검색 바 왼쪽에 "KQL" 표시가 있으며, 이를 클릭하면 "Lucene"으로 전환하는 옵션이 나타납니다. 두 구문의 차이를 이해하면 상황에 맞는 것을 선택할 수 있습니다.

가장 큰 차이는 문법의 간결함과 기능 범위입니다. KQL은 "field: value" 형태로 직관적이며, 자동완성을 지원합니다. 반면 Lucene 쿼리 구문은 더 오래된 문법으로, 정규식 검색이나 퍼지 검색 같은 고급 기능을 제공합니다.

```
두 구문 비교

기능                KQL                        Lucene
----------         -------------------------  -------------------------
기본 검색           status: 200                status:200
AND 연산            status: 200 AND ext: html  status:200 AND ext:html
OR 연산             status: (200 OR 301)       status:(200 OR 301)
NOT 연산            NOT status: 500            NOT status:500
와일드카드           host: web-*                host:web-*
범위                bytes >= 1000              bytes:[1000 TO *]
정규식              지원 안 함                   message:/err[ou]r/
퍼지 검색           지원 안 함                   message:timeuot~1
자동완성            지원                        지원 안 함
중첩 필드 자동 변환  지원                        수동 nested 쿼리 필요
```

범위 쿼리의 문법이 다릅니다. KQL은 부등호(>=, <=)를 사용하고, Lucene은 대괄호와 TO 키워드를 사용합니다. Lucene의 bytes:[1000 TO *]는 KQL의 bytes >= 1000과 같은 의미입니다. 대괄호 [ ]는 경계 값을 포함하고, 중괄호 { }는 경계 값을 제외합니다.

Lucene 쿼리 구문만의 고유 기능은 정규식 검색과 퍼지 검색입니다. 정규식 검색은 슬래시(/) 안에 정규식 패턴을 넣는 방식입니다. 퍼지 검색은 물결표(~)와 편집 거리를 붙여 오타를 허용하는 검색입니다. message:timeuot~1은 "timeuot"에서 한 글자 차이가 나는 "timeout"도 매칭합니다. 이런 기능이 필요한 경우에만 Lucene 구문으로 전환하고, 일상적인 필터링에는 KQL을 사용하는 것이 효율적입니다.

KQL의 실용적인 장점 중 하나는 **자동완성** 기능입니다. 검색 바에 필드 이름의 첫 글자를 입력하면 해당 문자로 시작하는 필드 목록이 드롭다운으로 나타납니다. 필드를 선택하고 콜론을 입력하면, 해당 필드에 존재하는 값 목록이 다시 드롭다운으로 표시됩니다. keyword 필드라면 상위 값이, 숫자 필드라면 범위 연산자 제안이 나타납니다. 연산자(AND, OR, NOT)도 자동완성됩니다. 이 자동완성 덕분에 필드 이름이나 값의 정확한 철자를 외우지 않아도 검색 조건을 빠르게 작성할 수 있습니다.

정리하면, KQL은 Kibana 검색 바에서 사용하는 간결한 쿼리 언어입니다. "field: value" 형태의 기본 문법에 AND, OR, NOT 연산자, 와일드카드, 범위 쿼리를 조합하여 필터 조건을 작성합니다. 중첩 필드는 점 표기법으로 접근하며, Kibana가 자동으로 적절한 쿼리로 변환합니다. 정규식이나 퍼지 검색이 필요한 특수한 경우에만 Lucene 쿼리 구문으로 전환하고, 일반적인 필터링에는 자동완성을 지원하는 KQL이 적합합니다.

이것으로 Phase 4 '집계 및 분석'을 마칩니다. 집계 구문의 기초부터 Bucket 집계, Metric 집계, 고급 집계를 거쳐 Kibana의 시각화, 대시보드, 쿼리 언어까지 다뤘습니다. 다음 단원인 5.1.1에서는 Phase 5 '분산 아키텍처'의 첫 주제인 노드 역할 체계를 다룹니다. Elasticsearch 클러스터를 구성하는 노드들이 어떤 역할을 나누어 맡는지 살펴봅니다.

이 단원을 마치면 KQL로 Kibana 검색 바에서 필터 조건을 작성할 수 있으며, KQL과 Lucene 쿼리 구문의 차이를 설명할 수 있습니다.
