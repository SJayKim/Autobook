# 2.2.1 text vs keyword

2.1.1에서 동적 매핑이 문자열 값을 만나면 text 타입과 keyword 서브필드를 함께 생성한다는 것을 살펴보았습니다. 동적 매핑은 어느 쪽이 필요할지 모르기 때문에 둘 다 만들었지만, 명시적 매핑에서는 필드마다 text와 keyword 중 적합한 타입을 직접 선택해야 합니다. 이 선택을 제대로 하려면 두 타입이 데이터를 저장하고 검색하는 방식이 어떻게 다른지 이해해야 합니다. 이 단원에서는 text와 keyword의 핵심 차이인 분석 여부를 중심으로, 각 타입의 동작 원리와 적합한 사용 사례를 살펴봅니다.

두 타입의 차이를 이해하기 위해 먼저 '분석'이라는 개념을 짚겠습니다. Elasticsearch에서 **분석**(analysis)이란 원문 텍스트를 검색 가능한 작은 단위로 변환하는 과정을 뜻합니다. 이 변환을 수행하는 도구를 **분석기**(analyzer)라고 합니다. 분석기는 원문을 받아서 여러 단계를 거쳐 토큰이라는 작은 조각으로 쪼갭니다.

분석 과정이 어떻게 동작하는지 구체적인 예로 살펴봅니다. "Quick Brown Fox"라는 문자열이 들어왔다고 가정합니다. 기본 분석기는 이 문자열을 다음 순서로 처리합니다.

```
원문:   "Quick Brown Fox"
  |
  v
1단계: 소문자 변환     -->  "quick brown fox"
  |
  v
2단계: 구두점 제거     -->  "quick brown fox" (이 예에서는 변화 없음)
  |
  v
3단계: 토크나이징      -->  ["quick", "brown", "fox"]
  |
  v
4단계: 불용어 필터링   -->  ["quick", "brown", "fox"] (해당 없으면 변화 없음)
  |
  v
결과: 세 개의 토큰이 역색인에 등록
```

소문자 변환 단계에서 대문자를 모두 소문자로 바꿉니다. 구두점 제거 단계에서 마침표, 쉼표 같은 문자를 제거합니다. 토크나이징 단계에서 공백을 기준으로 문자열을 개별 단어로 분리합니다. 마지막으로 불용어 필터링 단계에서 "the", "a", "is"처럼 검색에 의미가 없는 단어를 제거합니다. 이 네 단계를 거친 결과물인 토큰들이 역색인에 등록됩니다.

이제 text 타입이 이 분석 과정을 어떻게 활용하는지 살펴봅니다. **text** 타입은 필드 값을 인덱싱할 때 분석기를 적용합니다. 원문을 토큰으로 쪼개어 역색인에 등록하므로, 원문의 일부 단어만으로도 해당 문서를 찾을 수 있습니다. 이 방식을 **전문 검색**(full-text search)이라 합니다.

text 타입의 동작을 예시로 확인합니다. title 필드를 text 타입으로 매핑하고, "Quick Brown Fox"라는 값을 가진 문서를 인덱싱했다고 가정합니다. 역색인에는 "quick", "brown", "fox" 세 토큰이 등록됩니다. 이 상태에서 사용자가 "brown fox"라는 검색어로 match 쿼리를 보내면, Elasticsearch는 검색어도 동일한 분석기에 통과시킵니다. "brown fox"는 "brown", "fox" 두 토큰으로 분리됩니다. 역색인에서 "brown"과 "fox" 토큰이 있는 문서를 찾으면, 원문이 "Quick Brown Fox"인 문서가 매칭됩니다. 원문과 검색어가 정확히 같지 않아도, 토큰 단위로 일치하면 결과에 포함되는 것입니다.

text 타입은 이메일 본문, 상품 설명, 블로그 게시물처럼 사람이 작성한 자유 형식 텍스트에 적합합니다. 사용자가 문서의 정확한 표현을 기억하지 못해도 관련 단어 몇 개만 입력하면 해당 문서를 찾을 수 있기 때문입니다.

다음으로 keyword 타입을 살펴봅니다. **keyword** 타입은 분석기를 적용하지 않습니다. 원문을 어떤 변환도 거치지 않고 그대로 역색인에 등록합니다. "New York"이라는 값이 들어오면 "New York" 전체가 하나의 항목으로 저장됩니다. 소문자 변환도 없고, 단어 분리도 없습니다.

keyword 타입은 값 자체가 하나의 의미 단위인 데이터에 적합합니다. 상태 코드("published", "draft"), 이메일 주소, 태그, 카테고리명, URL 같은 데이터가 대표적입니다. 이런 데이터는 단어 단위로 쪼개서 검색할 필요가 없습니다. "published"라는 상태를 "pub"로 검색할 일도 없고, "user@example.com"이라는 이메일을 "user"로 검색하는 것도 적절하지 않습니다. 값 전체가 정확히 일치하는지 확인하는 **정확 일치**(exact match), 정렬(sort), 집계(aggregation)에 사용합니다.

두 타입의 차이를 하나의 표로 정리합니다.

```
항목            text                      keyword
-----------    -----------------------    -------------------------
분석 여부       분석기 적용 (토큰화)        분석 없음 (원문 그대로)
역색인 저장     토큰 단위                  원문 전체
주요 용도       전문 검색                  정확 일치, 정렬, 집계
적합 데이터     이메일 본문, 상품 설명       상태 코드, 태그, 이메일 주소
```

이제 keyword 필드에서 match 쿼리를 사용할 때 발생하는 주의사항을 살펴봅니다. match 쿼리는 전문 검색을 위해 설계된 쿼리입니다. 실행 전에 검색어를 분석기에 통과시켜 토큰으로 쪼갠 뒤 매칭합니다. 그런데 keyword 필드에는 분석기가 설정되어 있지 않으므로, match 쿼리가 keyword 필드를 대상으로 실행되면 검색어를 쪼개지 않고 그대로 비교합니다. 결과적으로 정확 일치처럼 동작하게 됩니다.

문제는 대소문자입니다. keyword 필드의 값 "New York"에 대해 match 쿼리로 "new york"을 검색하면 결과가 나오지 않습니다. keyword 필드는 원문 그대로 저장하므로 대문자 "N"과 소문자 "n"이 다른 값으로 취급됩니다. text 필드였다면 양쪽 모두 소문자 변환을 거쳐 "new york"으로 일치했을 것입니다.

keyword 필드에서 값을 검색할 때는 match 쿼리 대신 **term** 쿼리를 사용하는 것이 올바른 방식입니다. term 쿼리는 검색어를 분석기에 통과시키지 않고 그대로 비교하므로, keyword 필드의 동작 방식과 일치합니다. match 쿼리가 keyword 필드에서 우연히 동작하더라도, 검색 의도를 정확히 표현하지 못하므로 term 쿼리를 사용해야 합니다.

여기까지 text와 keyword의 검색 동작 차이를 살펴보았습니다. 이제 집계와 정렬에서 두 타입이 어떻게 다르게 동작하는지, 그리고 그 차이가 왜 발생하는지를 이해하기 위해 Elasticsearch의 내부 자료구조를 살펴봅니다.

Elasticsearch가 검색을 수행할 때는 역색인을 사용합니다. 역색인은 "이 토큰(또는 값)이 어떤 문서에 들어 있는가"를 찾는 데 최적화된 자료구조입니다. 그런데 집계와 정렬은 반대 방향의 조회가 필요합니다. "이 문서의 필드 값은 무엇인가"를 알아야 합니다. 예를 들어 status 필드로 집계를 수행하려면, 각 문서의 status 값을 하나씩 읽어 "published"가 몇 건, "draft"가 몇 건인지 세야 합니다. 역색인은 이 방향의 조회를 효율적으로 지원하지 못합니다.

이 문제를 해결하기 위해 Elasticsearch는 **doc_values**라는 별도의 자료구조를 사용합니다. doc_values는 "문서 ID에서 필드 값으로" 향하는 조회를 지원하는 컬럼 지향(column-oriented) 자료구조입니다. Lucene이 문서를 인덱싱하는 시점에 디스크에 함께 기록합니다.

역색인과 doc_values의 차이를 그림으로 비교합니다.

```
역색인 (inverted index)              doc_values
검색에 사용                          집계/정렬에 사용

토큰 --> 문서 목록                   문서 ID --> 필드 값
--------------------------           --------------------------
"draft"      --> [doc2, doc4]        doc1 --> "published"
"published"  --> [doc1, doc3]        doc2 --> "draft"
                                     doc3 --> "published"
                                     doc4 --> "draft"
```

왼쪽의 역색인은 특정 토큰이 포함된 문서를 빠르게 찾습니다. 오른쪽의 doc_values는 특정 문서의 필드 값을 빠르게 읽습니다. 집계와 정렬은 모든 문서의 값을 순회해야 하므로 doc_values 방향이 필요합니다.

doc_values는 디스크에 저장되며, 운영체제의 파일시스템 캐시(page cache)를 통해 메모리에 매핑됩니다. JVM 힙 메모리를 사용하지 않으므로 애플리케이션의 메모리 부담이 작습니다.

문자열 값의 경우 doc_values는 추가적인 최적화를 적용합니다. 유니크한 값들의 목록을 정렬하여 별도의 용어 사전에 저장하고, 컬럼 스토어에는 각 문서가 용어 사전의 몇 번째 항목에 해당하는지를 나타내는 작은 정수만 기록합니다. 작은 정수는 압축률이 높고 비교 연산이 빠르므로, terms 집계 같은 연산에서 성능 이점이 있습니다.

keyword 타입은 기본적으로 doc_values가 활성화되어 있습니다. 따라서 keyword 필드에서는 별도 설정 없이 집계와 정렬을 바로 사용할 수 있습니다. 반면 text 타입은 doc_values를 지원하지 않습니다. text 필드에서 terms 집계를 실행하면 Elasticsearch는 "Fielddata is disabled on text fields by default"라는 오류를 반환합니다.

text 필드에서 집계가 불가능한 이유를 정리합니다. text 필드는 원문을 토큰으로 쪼개어 역색인에 저장합니다. "Quick Brown Fox"는 "quick", "brown", "fox"라는 세 토큰이 됩니다. doc_values가 없으므로 "문서 ID에서 필드 값으로" 향하는 조회 경로가 없습니다. 설령 조회가 가능하다 해도, 집계 결과는 원문이 아니라 개별 토큰 단위로 나옵니다. "Quick Brown Fox"라는 값이 아니라 "quick" 3건, "brown" 2건, "fox" 5건 같은 식으로 집계되는 것입니다. 대부분의 경우 이것은 원하는 결과가 아닙니다.

text 필드에서 집계를 강제하는 레거시 방법이 하나 있습니다. 필드 매핑에 **fielddata**를 활성화하는 것입니다. `"fielddata": true`를 설정하면 처음 집계 요청이 들어올 때, Elasticsearch가 역색인에서 토큰과 문서의 매핑 관계를 거꾸로 뒤집어 JVM 힙 메모리에 올립니다. 이 인메모리 캐시를 fielddata 캐시라 합니다.

fielddata는 비권장(deprecated)입니다. 그 이유는 다음과 같습니다. 첫째, 데이터를 JVM 힙에 올리므로 대용량 텍스트 필드에서 힙을 급격히 소비합니다. 기본적으로 fielddata 캐시 크기에 상한이 없으므로, 메모리 부족(OOM) 오류가 발생할 위험이 있습니다. 둘째, doc_values처럼 디스크 기반의 OS 페이지 캐시를 활용하지 못하므로 성능이 불안정합니다. 셋째, 세그먼트가 열리거나 닫힐 때 캐시를 다시 생성해야 하므로 추가 비용이 발생합니다.

text 필드에서 집계가 필요한 상황이 실제로 있다면, fielddata 대신 멀티필드 패턴을 사용하는 것이 올바른 방법입니다. 2.1.1에서 동적 매핑이 문자열 필드에 text와 keyword 서브필드를 함께 생성한다고 설명했습니다. 명시적 매핑에서도 같은 구조를 직접 정의할 수 있습니다.

```json
{
  "properties": {
    "title": {
      "type": "text",
      "fields": {
        "keyword": {
          "type": "keyword",
          "ignore_above": 256
        }
      }
    }
  }
}
```

이 매핑은 title 필드를 text 타입으로 정의하면서, 그 안에 keyword라는 서브필드를 둡니다. title 필드로 전문 검색을 수행하고, title.keyword 필드로 집계와 정렬을 수행할 수 있습니다. ignore_above는 256자를 초과하는 값은 keyword 서브필드에 인덱싱하지 않겠다는 설정입니다. 매우 긴 텍스트를 keyword로 저장하면 비효율적이므로 적절한 상한을 두는 것입니다.

마지막으로 **index: false** 옵션을 살펴봅니다. 필드 매핑에 `"index": false`를 설정하면, 해당 필드의 값은 문서 원본(_source)에 저장되지만 역색인에는 등록되지 않습니다. 역색인에 없으므로 match, term 등 어떤 검색 쿼리로도 이 필드를 검색할 수 없습니다.

이 옵션은 검색이 필요 없는 필드에서 인덱스 크기를 줄이는 데 유용합니다. 예를 들어 직원 번호(employee-id)를 집계 용도로만 사용하고 검색할 필요가 없다면, index를 false로 설정하여 역색인 생성을 생략할 수 있습니다. 2.1.1에서 명시적 매핑 예시로 등장한 employee-id 필드가 바로 이 경우입니다.

index: false와 doc_values의 관계를 정리하면 다음과 같습니다. index는 역색인 생성 여부를, doc_values는 doc_values 자료구조 생성 여부를 각각 제어합니다. 두 설정을 조합하면 필드의 용도에 맞게 저장 비용을 최적화할 수 있습니다.

```
설정 조합                           검색    집계/정렬   용도
---------------------------------  ------  ---------  --------------------
index: true,  doc_values: true     가능    가능        기본값 (keyword 등)
index: true,  doc_values: false    가능    불가        검색만 필요한 필드
index: false, doc_values: true     불가    가능        집계/정렬만 필요한 필드
index: false, doc_values: false    불가    불가        _source 저장만
```

정리하면, text 타입은 분석기를 적용하여 원문을 토큰으로 분리한 뒤 역색인에 등록하므로 전문 검색에 적합하고, keyword 타입은 원문을 그대로 저장하므로 정확 일치, 정렬, 집계에 적합합니다. 두 타입의 핵심 차이는 분석 여부이며, 이 차이로 인해 text 필드에서는 doc_values가 지원되지 않아 집계와 정렬이 불가합니다. text 필드에서 집계가 필요하면 fielddata 대신 keyword 서브필드를 두는 멀티필드 패턴을 사용해야 합니다. index: false 옵션으로 역색인 생성을 비활성화하면 검색 기능을 포기하는 대신 인덱스 크기를 줄일 수 있습니다.

다음 단원인 2.2.2에서는 숫자/날짜/불리언 타입을 다룹니다.

이 단원을 마치면 text와 keyword 타입의 분석 여부 차이를 설명할 수 있고, 각 타입에 적합한 사용 사례를 구분하여 적용할 수 있습니다.
