# 7.1.3 임베딩 모델 배포

7.1.2에서 kNN 검색을 수행할 때 query_vector_builder를 사용하면 Elasticsearch가 검색 시점에 텍스트를 벡터로 변환한다고 설명했습니다. 문서를 인덱싱할 때도 텍스트를 벡터로 변환하는 과정이 필요합니다. 이 변환을 수행하는 것이 임베딩 모델이고, Elasticsearch ML 노드에 이 모델을 올려두면 별도의 외부 서버 없이 Elasticsearch 안에서 벡터 생성을 처리할 수 있습니다.

Elasticsearch의 ML 기능은 전용 노드에서 동작합니다. 클러스터를 구성할 때 node.roles에 ml 역할을 부여한 노드가 있어야 합니다. ML 노드는 모델을 메모리에 올리고, 추론(inference) 요청이 들어오면 모델을 실행하여 결과를 돌려줍니다. 데이터 노드와 분리하는 이유는 모델 추론이 CPU와 메모리를 많이 사용하므로, 검색이나 인덱싱 성능에 영향을 주지 않도록 격리하기 위해서입니다.

외부에서 학습된 모델을 Elasticsearch에 올리는 일반적인 방법은 **eland** 라이브러리를 사용하는 것입니다. eland는 Elastic에서 제공하는 Python 라이브러리로, Hugging Face 등 외부 모델 저장소에 있는 모델을 Elasticsearch ML 모델 저장소에 업로드하는 기능을 제공합니다.

Docker를 사용하는 경우 다음과 같이 실행합니다.

```bash
docker run -it --rm \
  docker.elastic.co/eland/eland:latest \
  eland_import_hub_model \
    --url https://my-elasticsearch:9200 \
    --hub-model-id sentence-transformers/all-MiniLM-L6-v2 \
    --task-type text_embedding \
    --es-username elastic \
    --es-password changeme
```

--url은 Elasticsearch 클러스터의 주소입니다. --hub-model-id는 Hugging Face에 등록된 모델의 식별자입니다. **--task-type**은 이 모델이 수행하는 작업 유형을 지정합니다. 임베딩 생성에는 **text_embedding**을 사용합니다. 이 명령을 실행하면 eland가 모델 가중치를 다운로드하고, Elasticsearch가 이해할 수 있는 형태로 변환하여 ML 모델 저장소에 저장합니다.

Python 환경에서 직접 eland를 사용하는 경우에도 동일한 작업을 수행합니다.

```python
from eland.ml.pytorch import PyTorchModel
from eland.ml.pytorch.transformers import TransformerModel
from elasticsearch import Elasticsearch

es = Elasticsearch("https://my-elasticsearch:9200",
                   basic_auth=("elastic", "changeme"))

tm = TransformerModel(
    model_id="sentence-transformers/all-MiniLM-L6-v2",
    task_type="text_embedding"
)
tm.save(es, model_id="sentence-transformers__all-minilm-l6-v2")
```

TransformerModel로 Hugging Face 모델을 로드하고, save 메서드로 Elasticsearch에 저장합니다. model_id는 Elasticsearch 내에서 이 모델을 식별하는 이름입니다. Hugging Face 모델 이름의 슬래시를 밑줄이나 언더스코어로 바꾸는 것이 관례입니다.

모델이 저장되면 REST API로도 확인할 수 있습니다.

```
GET _ml/trained_models/sentence-transformers__all-minilm-l6-v2
```

이 API는 모델의 메타데이터, 입력 형식, 출력 형식 등을 반환합니다. **PUT _ml/trained_models** API를 사용하면 eland 없이 모델 정의를 직접 등록할 수도 있지만, 일반적으로 eland를 통한 업로드가 훨씬 간편합니다.

모델을 저장소에 올렸다고 해서 바로 사용할 수 있는 것은 아닙니다. 모델을 ML 노드의 메모리에 로드하는 **배포**(deploy) 단계가 필요합니다.

```
POST _ml/trained_models/sentence-transformers__all-minilm-l6-v2/deployment/_start
{
  "number_of_allocations": 1,
  "threads_per_allocation": 2
}
```

이 요청이 _deploy API입니다. number_of_allocations는 모델 인스턴스 수입니다. 동시에 여러 요청을 처리해야 하면 이 값을 늘립니다. threads_per_allocation은 각 인스턴스가 사용할 스레드 수입니다. 배포가 완료되면 모델이 ML 노드 메모리에 올라가 추론 요청을 받을 준비가 됩니다.

배포된 모델을 문서 인덱싱 시 자동으로 호출하려면 6.2.3에서 다룬 인제스트 파이프라인에 **inference 프로세서**를 연결합니다.

```json
PUT _ingest/pipeline/embedding-pipeline
{
  "processors": [
    {
      "inference": {
        "model_id": "sentence-transformers__all-minilm-l6-v2",
        "input_output": [
          {
            "input_field": "content",
            "output_field": "content_embedding"
          }
        ]
      }
    }
  ]
}
```

inference 프로세서의 model_id에 배포한 모델 이름을 지정합니다. input_output은 어떤 필드의 텍스트를 모델에 넣고, 결과 벡터를 어떤 필드에 저장할지를 정합니다. 위 예시에서는 content 필드의 텍스트를 모델에 넣어 content_embedding 필드에 벡터를 저장합니다.

이제 문서를 인덱싱할 때 이 파이프라인을 지정하면, 텍스트만 넣어도 벡터가 자동으로 생성됩니다.

```json
POST my-vector-index/_doc?pipeline=embedding-pipeline
{
  "content": "서울 근교 단풍 명소를 소개합니다."
}
```

content 필드에 텍스트를 넣으면, 파이프라인이 inference 프로세서를 호출하고, 모델이 텍스트를 벡터로 변환하여 content_embedding 필드에 저장합니다. 애플리케이션에서 임베딩 변환 코드를 작성할 필요가 없습니다.

```
문서 입력 ──> 인제스트 파이프라인 ──> inference 프로세서 ──> ML 노드(모델)
                                                              │
                                                         벡터 생성
                                                              │
                                          content_embedding 필드에 저장 ──> 인덱싱 완료
```

배포한 모델의 성능을 확인하려면 모델 통계 API를 사용합니다.

```
GET _ml/trained_models/sentence-transformers__all-minilm-l6-v2/_stats
```

이 API는 추론 횟수, 평균 추론 시간, 큐에 대기 중인 요청 수 등을 반환합니다. 추론 시간이 지나치게 길거나 큐가 쌓이고 있다면 number_of_allocations를 늘리거나 ML 노드의 리소스를 확장해야 합니다. 벡터 차원이 큰 모델일수록 추론에 더 많은 시간과 메모리가 필요하므로, 모델 선택 단계에서 차원 수와 성능 간의 균형을 고려하는 것이 중요합니다.

정리하면, Elasticsearch ML 노드에 임베딩 모델을 배포하는 과정은 세 단계입니다. 먼저 eland 라이브러리로 외부 모델을 Elasticsearch ML 모델 저장소에 업로드합니다. 그 다음 _deploy API로 모델을 ML 노드 메모리에 로드합니다. 마지막으로 인제스트 파이프라인의 inference 프로세서에 모델을 연결하면, 문서 인덱싱 시 자동으로 임베딩이 생성됩니다.

다음 단원인 7.2.1에서는 dense_vector와는 다른 접근 방식인 ELSER의 희소 벡터 표현을 다룹니다.

이 단원을 마치면 Elasticsearch ML 노드에 외부 임베딩 모델을 배포하고 인제스트 파이프라인에 연결할 수 있습니다.
