# 2.3.1 분석기 파이프라인

2.2.1에서 text 타입이 원문을 토큰으로 변환하는 과정을 '분석'이라 부르고, 이 변환을 수행하는 도구를 '분석기'(analyzer)라 부른다는 것을 살펴보았습니다. 그때는 분석기가 소문자 변환, 구두점 제거, 토크나이징, 불용어 필터링을 거친다는 큰 흐름만 확인했습니다. 이 단원에서는 분석기의 내부를 열어, 세 가지 구성요소가 각각 어떤 역할을 맡고 어떤 순서로 연결되는지를 살펴봅니다. 분석기의 구조를 이해하면, 기본 분석기로 해결되지 않는 상황에서 구성요소를 직접 조합하여 원하는 분석 결과를 만들 수 있습니다.

분석기는 세 종류의 구성요소로 이루어져 있습니다. 첫 번째는 **캐릭터 필터**(character filter)입니다. 캐릭터 필터는 원문 텍스트가 토큰으로 쪼개지기 전에, 문자 수준에서 원문을 가공하는 역할을 합니다. 예를 들어 HTML 태그가 섞인 텍스트에서 태그를 제거하거나, 특정 문자를 다른 문자로 치환하는 작업을 이 단계에서 수행합니다.

두 번째 구성요소는 **토크나이저**(tokenizer)입니다. 토크나이저는 캐릭터 필터를 거친 문자열을 받아서 토큰이라는 작은 단위로 분리합니다. 공백을 기준으로 자르거나, 문법 규칙에 따라 단어 경계를 인식하거나, 정규표현식 패턴으로 분리하는 등 다양한 방식이 있습니다.

세 번째 구성요소는 **토큰 필터**(token filter)입니다. 토큰 필터는 토크나이저가 만들어 낸 토큰들을 받아서 변환, 추가, 제거하는 작업을 수행합니다. 대문자를 소문자로 바꾸거나, 검색에 의미 없는 불용어를 제거하거나, 동의어를 추가하는 것이 토큰 필터의 역할입니다.

이 세 구성요소는 반드시 정해진 순서대로 실행됩니다. 원문이 들어오면 캐릭터 필터가 먼저 처리하고, 그 결과를 토크나이저가 토큰으로 분리하며, 마지막으로 토큰 필터가 토큰을 가공합니다. 이 흐름을 그림으로 표현하면 다음과 같습니다.

```
원문 텍스트
  |
  v
[캐릭터 필터] --- 0개 이상, 순서대로 적용
  |
  v
[토크나이저]  --- 정확히 1개, 필수
  |
  v
[토큰 필터]   --- 0개 이상, 순서대로 적용
  |
  v
최종 토큰 목록 --> 역색인에 등록
```

캐릭터 필터와 토큰 필터는 0개 이상 지정할 수 있으므로 생략해도 됩니다. 반면 토크나이저는 반드시 하나만 지정해야 합니다. 텍스트를 토큰으로 분리하는 작업 자체가 분석의 핵심이기 때문입니다. 캐릭터 필터나 토큰 필터를 여러 개 지정하면 지정한 순서대로 차례로 적용됩니다. 첫 번째 캐릭터 필터의 출력이 두 번째 캐릭터 필터의 입력이 되는 식입니다.

이제 각 구성요소를 하나씩 자세히 살펴봅니다. 먼저 캐릭터 필터부터 시작합니다.

캐릭터 필터는 토크나이저에 도달하기 전에 원문 문자열을 수정합니다. 이 단계가 필요한 이유는, 원문에 포함된 특수한 표기가 토크나이저의 정상적인 동작을 방해하거나, 불필요한 토큰을 만들어 낼 수 있기 때문입니다.

Elasticsearch가 기본 제공하는 캐릭터 필터는 세 가지입니다. 첫 번째는 **html_strip**입니다. html_strip 캐릭터 필터는 텍스트에서 HTML 태그를 제거하고, HTML 엔티티를 디코딩합니다. 웹 페이지에서 수집한 데이터에는 `<b>`, `<p>` 같은 태그가 섞여 있는 경우가 많습니다. 이 태그들이 그대로 토크나이저에 전달되면 불필요한 토큰이 생기거나 단어가 태그에 붙어서 잘못 분리될 수 있습니다. html_strip은 이런 태그를 미리 걷어내어 순수한 텍스트만 토크나이저에 전달합니다. 예를 들어 `<p>Elasticsearch는 <b>검색</b> 엔진입니다</p>`라는 입력이 들어오면, html_strip을 거친 뒤에는 `Elasticsearch는 검색 엔진입니다`만 남습니다.

두 번째는 **mapping** 캐릭터 필터입니다. mapping 캐릭터 필터는 지정한 문자 또는 문자열을 다른 문자열로 치환합니다. 예를 들어 데이터에 `&` 기호가 포함되어 있는데, 이를 "and"라는 단어로 검색할 수 있게 하고 싶다면 `"& => and"`라는 매핑 규칙을 설정합니다. 그러면 토크나이저에 도달하기 전에 `&`가 `and`로 바뀌고, 이후 토크나이저가 `and`를 하나의 토큰으로 분리합니다.

세 번째는 **pattern_replace** 캐릭터 필터입니다. pattern_replace는 정규표현식으로 패턴을 지정하여 일치하는 부분을 치환합니다. mapping 캐릭터 필터보다 유연하지만, 정규표현식을 다루어야 하므로 설정이 복잡합니다.

다음으로 토크나이저를 살펴봅니다. 토크나이저는 캐릭터 필터를 거친 문자열을 받아서 개별 토큰으로 분리합니다. 분리할 때 각 토큰이 원문에서 몇 번째 위치에 있었는지(position)와 원문에서의 시작/끝 문자 오프셋(character offset)도 함께 기록합니다. 이 위치 정보는 나중에 구문 검색(phrase search)이나 하이라이팅에 사용됩니다.

Elasticsearch가 제공하는 대표적인 토크나이저를 살펴봅니다. **standard** 토크나이저는 유니코드 텍스트 분할 알고리즘(Unicode Text Segmentation)을 기반으로 동작합니다. 대부분의 유럽어에서 단어 경계를 잘 인식하며, 구두점을 제거합니다. 별도 설정이 없으면 Elasticsearch의 기본 분석기가 이 토크나이저를 사용합니다.

**whitespace** 토크나이저는 공백 문자만을 기준으로 텍스트를 분리합니다. standard 토크나이저와 달리 구두점을 제거하지 않고, 대소문자도 변환하지 않습니다. "Hello, World!"라는 입력에 대해 standard 토크나이저는 "Hello"와 "World" 두 토큰을 만들지만, whitespace 토크나이저는 "Hello,"와 "World!" 두 토큰을 만듭니다. 쉼표와 느낌표가 그대로 남는 것입니다.

**ngram** 토크나이저는 텍스트를 지정한 길이의 연속 문자 조각으로 분리합니다. 예를 들어 최소 길이 2, 최대 길이 3으로 설정하면 "quick"에서 "qu", "qui", "ui", "uic", "ic", "ick", "ck" 같은 토큰들이 만들어집니다. 부분 문자열 검색이 필요한 경우에 활용합니다.

이 외에도 keyword 토크나이저(입력 전체를 하나의 토큰으로 출력), pattern 토크나이저(정규표현식으로 분리), edge_ngram 토크나이저(단어 앞부분의 접두사 토큰을 생성하여 자동 완성에 활용) 등이 있습니다.

마지막으로 토큰 필터를 살펴봅니다. 토큰 필터는 토크나이저가 출력한 토큰 목록을 받아서 변환합니다. 토큰을 수정하거나, 새 토큰을 추가하거나, 불필요한 토큰을 제거할 수 있습니다. 다만 토큰 필터는 토크나이저가 기록한 위치(position)와 문자 오프셋 정보를 변경할 수 없습니다.

대표적인 토큰 필터를 살펴봅니다. **lowercase** 토큰 필터는 모든 토큰을 소문자로 변환합니다. 사용자가 "Elasticsearch"로 검색하든 "elasticsearch"로 검색하든 같은 결과를 얻으려면 인덱싱 시점에 토큰을 소문자로 통일해야 합니다. lowercase 필터가 이 역할을 합니다.

**stop** 토큰 필터는 불용어(stop word)를 제거합니다. 영어에서 "the", "a", "is", "at" 같은 단어는 거의 모든 문서에 등장하므로 검색 정확도에 기여하지 않습니다. stop 필터는 이런 단어에 해당하는 토큰을 목록에서 제거하여, 역색인의 크기를 줄이고 검색 품질을 높입니다.

**synonym** 토큰 필터는 동의어를 토큰 목록에 추가합니다. 예를 들어 "quick"의 동의어로 "fast"를 설정하면, "quick"이라는 토큰이 들어올 때 "fast"라는 토큰도 같은 위치에 추가됩니다. 사용자가 "fast"로 검색해도 원문에 "quick"이 포함된 문서를 찾을 수 있게 됩니다.

이 외에도 stemmer(단어를 어근 형태로 축소하여 "running", "runs", "ran"을 모두 "run"으로 통일), asciifolding(악센트 문자를 ASCII 문자로 변환), edge_ngram(각 토큰의 앞부분에서 접두사 토큰을 생성) 등 다양한 토큰 필터가 있습니다.

지금까지 세 구성요소를 각각 살펴보았습니다. 이 구성요소들이 실제로 어떻게 연결되어 동작하는지, 구체적인 예를 통해 확인합니다. "&" 기호와 HTML 태그가 섞인 `"<p>Rock & Roll</p>"`이라는 텍스트를 분석한다고 가정합니다. 캐릭터 필터로 html_strip과 mapping(`& => and`)을 차례로 적용하고, 토크나이저로 standard를, 토큰 필터로 lowercase를 지정한 분석기를 사용합니다.

```
원문: "<p>Rock & Roll</p>"
         |
         v
[html_strip]      -->  "Rock & Roll"         (태그 제거)
         |
         v
[mapping: &=>and] -->  "Rock and Roll"        (& 를 and 로 치환)
         |
         v
[standard 토크나이저] -->  ["Rock", "and", "Roll"]  (단어 경계로 분리)
         |
         v
[lowercase 필터]  -->  ["rock", "and", "roll"]  (소문자 변환)
         |
         v
역색인에 "rock", "and", "roll" 세 토큰 등록
```

html_strip 캐릭터 필터가 `<p>`와 `</p>` 태그를 제거합니다. mapping 캐릭터 필터가 `&`를 `and`로 치환합니다. standard 토크나이저가 공백과 문법 규칙에 따라 세 단어로 분리합니다. 마지막으로 lowercase 토큰 필터가 대문자를 소문자로 변환합니다. 이 과정을 거쳐 역색인에는 "rock", "and", "roll"이 등록됩니다.

이제 이 파이프라인의 동작을 직접 확인하는 방법을 살펴봅니다. Elasticsearch는 **_analyze API**를 제공합니다. _analyze API는 텍스트를 지정한 분석기에 통과시킨 결과를 반환하여, 분석 과정을 눈으로 확인할 수 있게 합니다. 인덱스를 만들거나 문서를 인덱싱하지 않아도 사용할 수 있으므로, 분석기를 설계하거나 문제를 진단할 때 유용합니다.

가장 간단한 형태는 내장 분석기의 이름을 지정하고 텍스트를 전달하는 것입니다.

```json
POST _analyze
{
  "analyzer": "standard",
  "text": "The quick brown fox"
}
```

이 요청은 "The quick brown fox"를 standard 분석기에 통과시킵니다. `analyzer` 필드에 사용할 분석기의 이름을 지정하고, `text` 필드에 분석할 텍스트를 넣습니다. 응답에는 분석 결과로 생성된 토큰 목록이 담깁니다. standard 분석기는 standard 토크나이저와 lowercase 토큰 필터를 포함하므로, "the", "quick", "brown", "fox" 네 토큰이 반환됩니다.

분석기 전체가 아니라 구성요소를 개별적으로 지정하여 테스트할 수도 있습니다.

```json
POST _analyze
{
  "tokenizer": "standard",
  "filter": ["lowercase", "stop"],
  "text": "The quick brown fox"
}
```

이 요청은 standard 토크나이저로 텍스트를 분리한 뒤, lowercase 필터와 stop 필터를 차례로 적용합니다. stop 필터가 불용어 "the"를 제거하므로, 결과에는 "quick", "brown", "fox" 세 토큰만 남습니다. 이처럼 _analyze API에서 `tokenizer`, `filter`, `char_filter` 필드를 개별적으로 조합하면, 어떤 구성요소가 어떤 효과를 내는지 하나씩 확인할 수 있습니다.

특정 인덱스에 이미 정의된 분석기를 테스트하려면 인덱스 이름을 경로에 포함합니다.

```json
POST my_index/_analyze
{
  "analyzer": "my_custom_analyzer",
  "text": "Rock & Roll"
}
```

이 요청은 my_index에 정의된 my_custom_analyzer를 사용하여 텍스트를 분석합니다. 인덱스에 커스텀 분석기를 설정한 뒤, 실제 문서를 인덱싱하기 전에 이 방법으로 분석 결과를 확인하면 의도한 대로 동작하는지 검증할 수 있습니다.

지금까지 분석기가 인덱싱 시점에 원문을 토큰으로 변환하는 과정을 살펴보았습니다. 그런데 분석은 인덱싱할 때만 일어나는 것이 아닙니다. 2.2.1에서 match 쿼리가 검색어도 분석기에 통과시킨다고 설명한 것을 떠올려 봅니다. 사용자가 전문 검색 쿼리를 보내면, Elasticsearch는 검색어도 분석기에 통과시켜 토큰으로 만든 뒤 역색인과 대조합니다. 이때 인덱싱에 사용한 분석기와 검색에 사용하는 분석기가 다르면 문제가 생길 수 있습니다.

예를 들어 인덱싱 시 lowercase 토큰 필터를 적용하여 "Elasticsearch"를 "elasticsearch"로 저장했는데, 검색 시에는 lowercase 필터가 없는 분석기를 사용한다고 가정합니다. 사용자가 "Elasticsearch"로 검색하면 검색어가 대문자 그대로 유지되어 "Elasticsearch"라는 토큰으로 역색인을 조회합니다. 역색인에는 소문자 "elasticsearch"만 있으므로 매칭되지 않습니다. 같은 문서인데도 검색되지 않는 것입니다.

이런 이유로 Elasticsearch는 기본적으로 인덱싱과 검색에 같은 분석기를 사용합니다. 필드에 분석기를 하나만 지정하면, 그 분석기가 인덱싱과 검색 양쪽 모두에 적용됩니다.

그런데 인덱싱과 검색에 의도적으로 다른 분석기를 사용해야 하는 경우가 있습니다. 대표적인 사례가 동의어 처리입니다. 인덱싱 시점에는 원문의 토큰만 저장하고, 검색 시점에 동의어를 확장하면 새로운 동의어를 추가할 때 기존 문서를 다시 인덱싱하지 않아도 됩니다. 또 다른 사례로, 인덱싱 시에는 edge_ngram으로 접두사 토큰을 미리 만들어 두고, 검색 시에는 사용자 입력을 그대로 매칭하는 자동 완성 구현이 있습니다.

이럴 때 **search_analyzer**를 별도로 지정합니다. 필드 매핑에서 `analyzer`는 인덱싱 시 분석기를, `search_analyzer`는 검색 시 분석기를 각각 설정합니다.

```json
{
  "properties": {
    "title": {
      "type": "text",
      "analyzer": "my_index_analyzer",
      "search_analyzer": "my_search_analyzer"
    }
  }
}
```

이 매핑에서 title 필드는 문서를 인덱싱할 때 my_index_analyzer를 사용하고, 검색 쿼리의 검색어를 분석할 때는 my_search_analyzer를 사용합니다. search_analyzer를 지정하지 않으면 analyzer에 설정된 분석기가 검색에도 그대로 적용됩니다.

정리하면, 분석기는 캐릭터 필터, 토크나이저, 토큰 필터라는 세 가지 구성요소가 순서대로 연결된 파이프라인입니다. 캐릭터 필터가 원문의 문자를 사전 처리하고, 토크나이저가 텍스트를 토큰으로 분리하며, 토큰 필터가 토큰을 변환합니다. 캐릭터 필터와 토큰 필터는 0개 이상 지정할 수 있지만, 토크나이저는 정확히 하나가 필수입니다. _analyze API로 분석 결과를 미리 확인할 수 있고, 인덱싱 시 분석기와 검색 시 분석기를 search_analyzer로 분리할 수 있습니다.

다음 단원인 2.3.2에서는 Elasticsearch가 기본 제공하는 내장 분석기를 다룹니다.

이 단원을 마치면 분석기의 3단계(character filter, tokenizer, token filter)를 설명할 수 있고, _analyze API로 분석 결과를 확인할 수 있습니다.
