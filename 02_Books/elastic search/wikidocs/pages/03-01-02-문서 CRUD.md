# 3.1.2 문서 CRUD

3.1.1에서 Elasticsearch REST API의 URL 패턴과 HTTP 메서드를 살펴보았습니다. PUT, GET, POST, DELETE 메서드가 각각 저장, 조회, 생성/처리, 삭제를 의미한다는 것, 요청 바디는 JSON으로 작성한다는 것을 확인했습니다. 이제 그 도구를 실제로 사용할 차례입니다. 인덱스에 문서를 넣고, 꺼내 보고, 일부를 고치고, 지우는 네 가지 작업을 CRUD(Create, Read, Update, Delete)라고 부릅니다. 이 단원에서는 Elasticsearch가 제공하는 문서 CRUD API를 하나씩 다룹니다.

Elasticsearch에서 말하는 **문서**(document)는 JSON 형태의 데이터 한 건입니다. 하나의 문서는 반드시 하나의 인덱스에 속하며, 인덱스 안에서 고유한 **문서 ID**로 식별됩니다. 3.1.1에서 응답 본문에 "_index", "_id", "_source" 같은 메타데이터가 붙어 있던 것을 떠올리면 됩니다. "_source" 안에 사용자가 저장한 실제 데이터가 들어가고, 나머지는 Elasticsearch가 관리하는 정보입니다.

문서를 생성하는 방법은 두 가지입니다. 첫 번째는 **PUT /{인덱스}/_doc/{id}** 형식입니다. 이 방식은 문서 ID를 직접 지정합니다.

```
PUT /favorite_candy/_doc/1
{
  "first_name": "John",
  "candy": "Starburst"
}
```

PUT 메서드 뒤에 인덱스 이름 'favorite_candy', 엔드포인트 '_doc', 그리고 문서 ID '1'이 차례로 옵니다. 요청 바디에는 저장할 데이터를 JSON으로 담습니다. "first_name"과 "candy"가 필드 이름이고, "John"과 "Starburst"가 각 필드의 값입니다. 이 요청을 보내면 Elasticsearch는 favorite_candy 인덱스에 ID가 1인 문서를 생성합니다. 응답으로 "result": "created"와 함께 201 상태 코드가 돌아옵니다.

PUT 방식은 문서 ID를 직접 정하기 때문에, 이미 의미 있는 식별자가 있는 경우에 적합합니다. 주문 번호, 사원 번호, 환자 ID처럼 업무 체계에서 부여한 고유 값을 그대로 문서 ID로 쓸 수 있습니다.

그런데 PUT으로 이미 존재하는 ID에 다시 요청을 보내면 어떻게 될까요. Elasticsearch는 기존 문서를 새 내용으로 덮어씁니다. "result" 필드가 "created" 대신 "updated"로 바뀌고, "_version"이 1 증가합니다. 의도적으로 갱신하려는 것이라면 문제가 없지만, 실수로 기존 데이터를 날릴 수도 있습니다.

이런 실수를 막으려면 **_create 엔드포인트**를 사용합니다.

```
PUT /favorite_candy/_create/1
{
  "first_name": "John",
  "candy": "Starburst"
}
```

'_doc' 대신 '_create'를 쓴 것이 차이입니다. 이 엔드포인트는 해당 ID의 문서가 이미 존재하면 요청을 거부합니다. 응답으로 409 Conflict 상태 코드가 돌아오므로, "이미 있는 문서를 실수로 덮어쓰는 일"을 원천 차단할 수 있습니다. 문서가 존재하지 않을 때만 생성이 성공하므로, 중복 방지가 중요한 시나리오에서 유용합니다.

두 번째 생성 방법은 **POST /{인덱스}/_doc** 형식입니다. PUT과 달리 URL 끝에 ID를 쓰지 않습니다.

```
POST /favorite_candy/_doc
{
  "first_name": "Lisa",
  "candy": "Sour Skittles"
}
```

ID를 지정하지 않았으므로 Elasticsearch가 고유한 ID를 자동으로 만들어 줍니다. 응답을 보면 "_id" 필드에 긴 문자열이 들어 있습니다. 이 문자열은 Elasticsearch가 내부적으로 생성한 것으로, 중복되지 않도록 보장됩니다. 응답 상태 코드는 201 Created입니다.

자동 생성 ID는 로그 데이터나 이벤트 데이터처럼, 개별 문서에 특별한 식별자가 필요 없는 경우에 편리합니다. 문서가 대량으로 밀려 들어올 때 일일이 ID를 부여하는 부담이 사라집니다.

아래 표는 두 가지 생성 방식과 _create 엔드포인트를 비교한 것입니다.

```
방식                          ID 지정   기존 문서 처리         주 용도
----------------------------  --------  --------------------  ----------------
PUT /{index}/_doc/{id}        직접      덮어씀 (updated)      업무 식별자 활용
PUT /{index}/_create/{id}     직접      거부 (409 Conflict)   중복 방지
POST /{index}/_doc            자동      해당 없음             로그/이벤트 적재
```

문서를 저장했으면 다시 꺼내 볼 수 있어야 합니다. **GET /{인덱스}/_doc/{id}** 요청이 문서 조회에 해당합니다.

```
GET /favorite_candy/_doc/1
```

인덱스 이름과 문서 ID를 경로에 지정합니다. 요청 바디는 필요 없습니다. Elasticsearch는 해당 문서를 찾아 200 OK와 함께 응답을 돌려줍니다.

```json
{
  "_index": "favorite_candy",
  "_id": "1",
  "_version": 1,
  "_seq_no": 0,
  "_primary_term": 1,
  "found": true,
  "_source": {
    "first_name": "John",
    "candy": "Starburst"
  }
}
```

"found" 필드가 true이면 문서가 존재한다는 뜻입니다. "_source" 안에 저장한 원본 데이터가 그대로 들어 있습니다. 만약 존재하지 않는 ID를 조회하면 "found"가 false가 되고 상태 코드는 404 Not Found입니다.

응답에 포함된 **_version**은 이 문서가 몇 번째 변경 상태인지를 나타내는 숫자입니다. 문서를 처음 생성하면 1이고, 수정하거나 삭제할 때마다 1씩 증가합니다. **_seq_no**와 **_primary_term**은 뒤에서 동시성 제어를 설명할 때 다시 다룹니다.

이제 저장된 문서를 삭제하는 방법을 봅니다. **DELETE /{인덱스}/_doc/{id}** 요청으로 특정 문서 한 건을 지울 수 있습니다.

```
DELETE /favorite_candy/_doc/1
```

요청이 성공하면 200 OK가 돌아오고, 응답의 "result" 필드에 "deleted"가 표시됩니다. 삭제 후 같은 ID로 GET 요청을 보내면 404 Not Found가 돌아옵니다. 문서가 사라졌지만 인덱스 자체는 그대로 남아 있다는 점을 기억합니다. 인덱스를 통째로 지우려면 'DELETE /favorite_candy'처럼 인덱스 이름만 경로에 넣어야 합니다. 이 단원에서는 문서 단위 삭제만 다룹니다.

한 건이 아니라 조건에 맞는 여러 문서를 한꺼번에 지워야 하는 상황도 있습니다. 예를 들어 "candy" 필드가 "Starburst"인 문서를 모두 삭제하고 싶다면, 문서 ID를 하나하나 지정하는 대신 **Delete by Query**를 사용합니다.

```
POST /favorite_candy/_delete_by_query
{
  "query": {
    "match": {
      "candy": "Starburst"
    }
  }
}
```

POST 메서드와 '_delete_by_query' 엔드포인트를 사용합니다. 요청 바디의 "query" 안에 삭제 조건을 넣습니다. 여기서는 "match" 쿼리로 "candy" 필드 값이 "Starburst"인 문서를 지정했습니다. Elasticsearch는 이 조건에 해당하는 모든 문서를 찾아 삭제합니다. 응답에는 삭제된 문서 수가 "deleted" 필드로 표시됩니다. Delete by Query는 조건 기반 일괄 삭제이므로, 실행 전에 같은 쿼리로 검색(_search)을 먼저 해 보고 대상을 확인하는 것이 안전합니다.

아래 다이어그램은 DELETE 한 건 삭제와 Delete by Query의 차이를 정리한 것입니다.

```
문서 삭제 방식 비교

DELETE /{index}/_doc/{id}          POST /{index}/_delete_by_query
        |                                    |
        v                                    v
  ID를 직접 지정                      쿼리 조건으로 대상 선택
        |                                    |
        v                                    v
  문서 1건 삭제                       조건에 맞는 N건 삭제
        |                                    |
        v                                    v
  "result": "deleted"               "deleted": N
```

다음으로 문서 수정 방법을 살펴봅니다. Elasticsearch에서 문서를 수정하는 방식은 크게 두 가지입니다. 전체 덮어쓰기와 **부분 업데이트**(partial update)입니다.

전체 덮어쓰기는 앞서 본 PUT /{인덱스}/_doc/{id}를 다시 호출하는 것입니다. 같은 ID에 새로운 JSON을 보내면 기존 문서 전체가 새 내용으로 교체됩니다. 필드가 열 개인 문서에서 하나만 바꾸고 싶어도, 열 개 필드를 모두 다시 보내야 합니다. 이것은 번거로울 뿐 아니라, 빠뜨린 필드가 있으면 그 필드가 사라지는 문제를 낳습니다.

부분 업데이트는 바꾸고 싶은 필드만 보내는 방식입니다. **POST /{인덱스}/_update/{id}** 엔드포인트를 사용합니다.

```
POST /favorite_candy/_update/1
{
  "doc": {
    "candy": "M&M's"
  }
}
```

요청 바디에 "doc"이라는 키가 있고, 그 안에 변경할 필드만 넣습니다. 위 예시는 ID 1번 문서의 "candy" 필드만 "M&M's"로 바꿉니다. "first_name" 필드는 건드리지 않고 기존 값 "John"이 그대로 유지됩니다.

"doc" 키가 왜 필요한지 의문이 들 수 있습니다. _update 엔드포인트는 부분 업데이트 외에도 스크립트를 사용한 업데이트 등 여러 모드를 지원합니다. "doc" 키는 "이 안에 담긴 필드들을 기존 문서에 병합하라"는 뜻으로, _update 엔드포인트에서 부분 업데이트 모드를 지정하는 역할입니다.

업데이트가 성공하면 응답의 "_version"이 1 증가합니다. Elasticsearch 내부에서는 사실 기존 문서를 직접 수정하는 것이 아닙니다. 기존 문서를 읽어서 변경 사항을 병합한 뒤, 새 버전의 문서로 다시 인덱싱합니다. 외부에서 보기에는 필드 하나만 바뀐 것 같지만, 내부적으로는 문서 전체가 새로 기록되는 셈입니다. 이 동작 방식은 Elasticsearch가 기반으로 사용하는 Lucene이 문서 수정을 지원하지 않기 때문입니다. Lucene의 인덱스 구조는 한 번 기록되면 변경할 수 없는(immutable) 세그먼트로 이루어져 있어서, 수정 대신 새 문서를 쓰고 이전 문서를 삭제 표시하는 방식으로 처리합니다.

부분 업데이트를 할 때, 해당 ID의 문서가 존재하지 않으면 어떻게 될까요. 기본적으로는 404 Not Found 오류가 돌아옵니다. 그런데 "문서가 없으면 새로 만들고, 있으면 업데이트하라"는 동작이 필요한 경우가 있습니다. 이것을 **upsert** 패턴이라고 부릅니다. "update"와 "insert"를 합친 말입니다.

upsert를 사용하려면 _update 요청에 "doc_as_upsert" 옵션을 추가합니다.

```
POST /favorite_candy/_update/1
{
  "doc": {
    "first_name": "John",
    "candy": "Starburst"
  },
  "doc_as_upsert": true
}
```

"doc_as_upsert"를 true로 설정하면, ID 1번 문서가 이미 있을 때는 "doc" 안의 필드를 병합하여 업데이트합니다. ID 1번 문서가 없을 때는 "doc" 안의 내용을 그대로 새 문서로 생성합니다. 한 번의 요청으로 "있으면 수정, 없으면 생성"을 처리할 수 있으므로, 클라이언트 쪽에서 문서 존재 여부를 미리 확인할 필요가 없습니다.

다음 다이어그램은 일반 부분 업데이트와 upsert 패턴의 흐름을 비교한 것입니다.

```
POST /{index}/_update/{id}

"doc_as_upsert" 미설정 (기본)       "doc_as_upsert": true
        |                                    |
        v                                    v
  문서 존재 여부 확인                  문서 존재 여부 확인
        |                                    |
   +---------+                          +---------+
   | 있음    | 없음                     | 있음    | 없음
   v         v                          v         v
  병합 후   404 오류                   병합 후   "doc"으로
  재인덱싱  반환                       재인덱싱  새 문서 생성
```

여러 클라이언트가 동시에 같은 문서를 수정하면 충돌이 발생할 수 있습니다. 예를 들어 클라이언트 A가 문서를 읽고 수정하는 사이에, 클라이언트 B가 같은 문서를 먼저 수정해 버리면, A의 수정이 B의 변경 내용을 덮어씁니다. 이런 문제를 해결하기 위해 Elasticsearch는 **낙관적 동시성 제어**(optimistic concurrency control)를 지원합니다.

"낙관적"이라는 말은 충돌이 드물 것이라고 가정하고, 잠금(lock) 없이 작업을 진행한다는 뜻입니다. 대신 실제로 저장하는 시점에 "내가 읽었던 버전이 아직 유효한가"를 검사합니다. 유효하면 저장하고, 이미 다른 수정이 끼어들었으면 거부합니다.

Elasticsearch에서 이 검사에 사용하는 값이 **_seq_no**와 **_primary_term**입니다. 앞서 GET 응답에서 이 두 필드를 보았습니다. _seq_no는 해당 샤드에서 수행된 작업의 순서 번호이고, _primary_term은 프라이머리 샤드가 재할당될 때마다 증가하는 값입니다. 이 두 값을 조합하면 문서의 특정 시점 상태를 정확히 식별할 수 있습니다.

사용 방법은 다음과 같습니다. 먼저 문서를 조회하여 _seq_no와 _primary_term을 확인합니다.

```
GET /favorite_candy/_doc/1
```

응답에서 "_seq_no": 5, "_primary_term": 1을 얻었다고 가정합니다. 이 문서를 수정할 때 해당 값을 쿼리 파라미터로 전달합니다.

```
PUT /favorite_candy/_doc/1?if_seq_no=5&if_primary_term=1
{
  "first_name": "John",
  "candy": "M&M's"
}
```

Elasticsearch는 저장 직전에 현재 문서의 _seq_no와 _primary_term이 요청에 포함된 값과 일치하는지 확인합니다. 일치하면 수정이 성공하고, 일치하지 않으면 409 Conflict를 반환합니다. 409가 돌아왔다면, 다른 클라이언트가 이미 문서를 변경했다는 뜻입니다. 이때 클라이언트는 문서를 다시 조회하여 최신 내용과 새 _seq_no, _primary_term을 확인한 뒤 수정을 재시도합니다.

아래는 낙관적 동시성 제어의 전체 흐름입니다.

```
클라이언트 A                Elasticsearch              클라이언트 B
    |                           |                           |
    |--- GET /_doc/1 ---------> |                           |
    |<-- _seq_no=5, term=1 ---- |                           |
    |                           |                           |
    |   (A가 수정 준비 중)      |<-- PUT /_doc/1 --------- |
    |                           |    (B가 먼저 수정)        |
    |                           |--- 200 OK --------------> |
    |                           |    _seq_no=6, term=1      |
    |                           |                           |
    |--- PUT /_doc/1 ---------> |                           |
    |    ?if_seq_no=5           |                           |
    |    &if_primary_term=1     |                           |
    |                           |                           |
    |<-- 409 Conflict --------- |                           |
    |   (seq_no 불일치)         |                           |
    |                           |                           |
    |--- GET /_doc/1 ---------> |   (A가 다시 조회)         |
    |<-- _seq_no=6, term=1 ---- |                           |
    |                           |                           |
    |--- PUT /_doc/1 ---------> |   (A가 재시도)            |
    |    ?if_seq_no=6           |                           |
    |    &if_primary_term=1     |                           |
    |<-- 200 OK --------------- |                           |
```

클라이언트 A가 읽은 시점의 _seq_no는 5였지만, B가 먼저 수정하여 _seq_no가 6으로 올랐습니다. A가 if_seq_no=5로 수정을 시도하면 현재 값(6)과 맞지 않으므로 409 Conflict가 돌아옵니다. A는 다시 조회하여 최신 상태를 확인한 뒤, if_seq_no=6으로 재시도하면 성공합니다.

지금까지 다룬 CRUD 작업 전체를 한눈에 정리하면 다음과 같습니다.

```
작업           HTTP 요청                                   설명
----------     ------------------------------------------  ----------------------------
생성 (ID 지정) PUT /{index}/_doc/{id}                      ID를 직접 부여하여 문서 저장
생성 (ID 자동) POST /{index}/_doc                          ID를 자동 생성하여 문서 저장
생성 (중복방지) PUT /{index}/_create/{id}                   ID 중복 시 409 거부
조회           GET /{index}/_doc/{id}                      문서 한 건 조회
삭제 (단건)    DELETE /{index}/_doc/{id}                    문서 한 건 삭제
삭제 (조건)    POST /{index}/_delete_by_query              조건에 맞는 문서 일괄 삭제
부분 업데이트  POST /{index}/_update/{id}                  "doc" 안의 필드만 병합
upsert         POST /{index}/_update/{id} + doc_as_upsert  없으면 생성, 있으면 업데이트
동시성 제어    ?if_seq_no=N&if_primary_term=M              낙관적 충돌 감지
```

정리하면, Elasticsearch의 문서 CRUD는 PUT과 POST로 생성하고, GET으로 조회하고, POST /_update로 부분 수정하고, DELETE로 삭제하는 구조입니다. PUT /_doc은 같은 ID에 다시 호출하면 덮어쓰고, _create 엔드포인트는 중복을 거부합니다. _update 엔드포인트의 "doc" 키로 원하는 필드만 병합할 수 있고, "doc_as_upsert"를 true로 설정하면 문서가 없을 때 자동 생성까지 처리합니다. 여러 클라이언트가 동시에 수정할 가능성이 있다면 if_seq_no와 if_primary_term 파라미터로 낙관적 동시성 제어를 적용하여 충돌을 감지합니다. 조건 기반 일괄 삭제는 _delete_by_query 엔드포인트로 수행합니다.

다음 단원인 3.1.3에서는 Bulk API를 다룹니다. 문서를 한 건씩 처리하는 대신, 여러 건의 생성/수정/삭제를 하나의 요청에 묶어 보내는 방법을 살펴봅니다.

이 단원을 마치면 문서의 생성, 조회, 수정, 삭제 API를 올바르게 사용할 수 있고, upsert와 partial update의 차이를 설명할 수 있습니다.
