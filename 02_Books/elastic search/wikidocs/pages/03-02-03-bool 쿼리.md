# 3.2.3 bool 쿼리

3.2.2에서 match 쿼리와 term 쿼리를 살펴보았습니다. match 쿼리는 텍스트를 분석기에 통과시켜 전문 검색을 수행하고, term 쿼리는 값을 분석 없이 그대로 매칭합니다. 이 두 쿼리는 각각 하나의 조건만 표현합니다. 실제 검색에서는 여러 조건을 동시에 적용해야 하는 상황이 대부분입니다. 예를 들어 제목에 "Elasticsearch"가 포함된 문서 중에서 상태가 "published"인 것만 찾고, "draft" 카테고리는 제외하고 싶다면, 하나의 쿼리만으로는 이 요구를 표현할 수 없습니다. 여러 쿼리 조건을 하나로 묶어 주는 장치가 필요합니다. Elasticsearch에서 이 역할을 하는 것이 **bool 쿼리**입니다.

bool 쿼리는 여러 개의 쿼리 절(clause)을 논리적으로 조합하는 복합 쿼리입니다. SQL의 WHERE 절에서 AND, OR, NOT을 조합하여 복합 조건을 만드는 것과 비슷한 역할을 합니다. bool 쿼리는 네 가지 종류의 절을 제공합니다. 각 절은 "이 조건이 반드시 충족되어야 하는가", "점수 계산에 기여하는가"라는 두 가지 기준으로 구분됩니다. 이 네 가지 절을 하나씩 살펴봅니다.

**must** 절은 반드시 충족해야 하는 필수 조건입니다. must 절에 넣은 쿼리와 매칭되지 않는 문서는 결과에서 제외됩니다. 동시에, must 절의 매칭 결과는 문서의 관련성 점수(_score)에 반영됩니다. 논리적으로 AND에 해당합니다. 예를 들어 must 절에 title 필드를 대상으로 한 match 쿼리를 넣으면, 해당 검색어와 title이 매칭되는 문서만 결과에 남고, 매칭 정도에 따라 점수가 달라집니다.

**filter** 절도 must와 마찬가지로 반드시 충족해야 하는 필수 조건입니다. 다른 점은 점수 계산에 기여하지 않는다는 것입니다. filter 절은 문서가 조건에 맞는지 아닌지를 이진적으로 판단할 뿐, 점수를 더하지 않습니다. 3.2.1에서 다룬 filter context와 같은 맥락입니다. 점수를 계산하지 않으므로 Elasticsearch가 filter 절의 결과를 캐시할 수 있고, 같은 조건이 반복될 때 성능이 향상됩니다. 상태 값, 날짜 범위, 숫자 비교처럼 관련성 점수가 의미 없는 조건에는 must 대신 filter를 사용하는 것이 적합합니다.

**should** 절은 선택적 조건입니다. should 절에 넣은 쿼리와 매칭되지 않아도 문서가 결과에서 제외되지는 않습니다. 다만, should 절의 조건에 매칭되는 문서는 추가 점수를 받아 결과 목록에서 더 높은 순위에 올라갑니다. 논리적으로 OR에 가깝습니다. 주의할 점이 있습니다. bool 쿼리에 must 절이나 filter 절이 하나도 없고 should 절만 있는 경우, should 절 중 최소 하나는 매칭되어야 결과에 포함됩니다. must나 filter가 있으면 should는 순수하게 점수 부스트 역할만 합니다.

**must_not** 절은 제외 조건입니다. must_not 절에 넣은 쿼리와 매칭되는 문서를 결과에서 완전히 배제합니다. 논리적으로 NOT에 해당합니다. must_not 절은 점수 계산에 기여하지 않으며, filter context에서 실행됩니다. 특정 카테고리를 제외하거나 삭제 표시된 문서를 걸러낼 때 사용합니다.

네 가지 절의 차이를 정리하면 다음과 같습니다.

```
bool 쿼리의 네 가지 절

절          | 매칭 필수 여부 | 점수 기여 | 논리 연산 | 캐시 가능
------------|--------------|----------|----------|----------
must        | 필수          | O        | AND      | X
filter      | 필수          | X        | AND      | O
should      | 선택          | O        | OR       | X
must_not    | 제외          | X        | NOT      | O
```

이제 네 가지 절을 모두 사용하는 쿼리를 작성해 봅니다. 블로그 게시글 인덱스에서 "title에 Elasticsearch가 포함되고, 상태가 published이고, 날짜가 2024년 1월 1일 이후이며, 태그에 tutorial이 있으면 우대하고, 카테고리가 draft인 문서는 제외한다"는 조건을 bool 쿼리로 표현합니다.

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "Elasticsearch" } }
      ],
      "filter": [
        { "term": { "status": "published" } },
        { "range": { "date": { "gte": "2024-01-01" } } }
      ],
      "should": [
        { "match": { "tags": "tutorial" } }
      ],
      "must_not": [
        { "term": { "category": "draft" } }
      ]
    }
  }
}
```

이 쿼리의 각 부분을 살펴봅니다. "query" 안에 "bool"을 지정하여 bool 쿼리임을 선언합니다. "must" 배열 안에 title 필드를 대상으로 한 match 쿼리를 넣었습니다. title에 "Elasticsearch"가 포함된 문서만 결과에 남고, 매칭 정도가 점수에 반영됩니다.

"filter" 배열에는 두 개의 조건을 넣었습니다. status가 "published"인지 확인하는 term 쿼리와, date가 2024년 1월 1일 이후인지 확인하는 range 쿼리입니다. 이 두 조건은 모두 충족해야 하지만 점수에는 영향을 주지 않습니다. 상태 값이나 날짜 범위는 "얼마나 관련이 있는가"보다 "해당되는가, 해당되지 않는가"만 중요하므로 filter에 넣는 것이 적절합니다.

"should" 배열에는 tags 필드를 대상으로 한 match 쿼리를 넣었습니다. must와 filter가 이미 존재하므로, tags에 "tutorial"이 없는 문서도 결과에 포함됩니다. 다만 "tutorial" 태그가 있는 문서는 추가 점수를 받아 결과 상위에 올라갑니다.

"must_not" 배열에는 category가 "draft"인 문서를 제외하는 term 쿼리를 넣었습니다. category가 "draft"인 문서는 다른 조건을 모두 만족하더라도 결과에 나타나지 않습니다.

bool 쿼리의 점수 계산 방식을 이해하면 네 가지 절을 더 효과적으로 활용할 수 있습니다. bool 쿼리는 "더 많이 매칭될수록 높은 점수"를 부여하는 방식으로 동작합니다. must 절과 should 절에서 매칭된 쿼리들의 점수가 누적되어 최종 _score를 구성합니다. filter 절과 must_not 절은 점수에 기여하지 않습니다. 같은 조건이라도 must에 넣느냐 filter에 넣느냐에 따라 점수 계산 여부가 달라지므로, "이 조건이 관련성 순위에 영향을 줘야 하는가"를 기준으로 절을 선택합니다.

```
bool 쿼리의 점수 계산 흐름

1. must 절의 각 쿼리가 문서별 점수를 계산한다
2. should 절의 각 쿼리가 매칭되면 추가 점수를 계산한다
3. 1단계와 2단계의 점수를 합산한다
4. filter 절과 must_not 절은 점수에 기여하지 않는다
5. 합산된 점수가 문서의 최종 _score가 된다
```

이제 should 절의 동작을 세밀하게 제어하는 **minimum_should_match** 파라미터를 살펴봅니다. bool 쿼리에 should 절이 여러 개 있을 때, 기본적으로 몇 개가 매칭되어야 하는지에 대한 규칙이 있습니다. bool 쿼리에 must나 filter 절이 하나 이상 있으면 should 절의 기본 매칭 요구 수는 0입니다. 하나도 매칭되지 않아도 괜찮습니다. 반면 must와 filter가 모두 없고 should 절만 있으면, 최소 1개의 should 절이 매칭되어야 결과에 포함됩니다.

이 기본 동작을 변경하고 싶을 때 minimum_should_match를 사용합니다. 다음 예시를 봅니다.

```json
{
  "query": {
    "bool": {
      "should": [
        { "match": { "title": "elasticsearch" } },
        { "match": { "body": "tutorial" } },
        { "match": { "tags": "search" } }
      ],
      "minimum_should_match": 2
    }
  }
}
```

이 쿼리에는 must와 filter가 없고 should 절 세 개만 있습니다. minimum_should_match를 2로 지정했으므로, 세 개의 should 절 중 두 개 이상이 매칭되어야 결과에 포함됩니다. title에 "elasticsearch"가 있고 tags에 "search"가 있는 문서는 두 개가 매칭되므로 결과에 포함됩니다. title에 "elasticsearch"만 매칭되는 문서는 한 개뿐이므로 제외됩니다.

minimum_should_match에는 정수뿐 아니라 백분율도 사용할 수 있습니다. "75%"로 지정하면 전체 should 절 수의 75% 이상이 매칭되어야 합니다. should 절이 세 개이면 3의 75%인 2.25를 내림하여 2개 이상이 매칭되어야 합니다. should 절 수가 유동적인 상황에서는 백분율이 유연합니다.

must나 filter 절이 함께 있는 bool 쿼리에서도 minimum_should_match를 사용할 수 있습니다. 이 경우 기본값이 0이므로 명시적으로 지정하지 않으면 should 절 매칭은 선택 사항입니다. minimum_should_match를 1 이상으로 지정하면, 필수 조건(must, filter)을 만족하면서 동시에 should 절 중 지정한 수만큼 매칭되는 문서만 결과에 포함됩니다.

bool 쿼리의 절 안에 다시 bool 쿼리를 넣는 **중첩 bool 쿼리**도 가능합니다. 조건 구조가 단순한 AND/OR/NOT으로 표현되지 않을 때, bool 쿼리를 중첩하여 복잡한 논리를 구성합니다. 예를 들어 "type이 article이면서, tags에 elasticsearch 또는 search가 포함된 문서"를 찾는 경우를 봅니다.

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "type": "article" } },
        {
          "bool": {
            "should": [
              { "match": { "tags": "elasticsearch" } },
              { "match": { "tags": "search" } }
            ]
          }
        }
      ]
    }
  }
}
```

바깥쪽 bool 쿼리의 must 절에 두 개의 조건이 있습니다. 첫 번째는 type 필드에 대한 match 쿼리입니다. 두 번째는 또 다른 bool 쿼리입니다. 이 안쪽 bool 쿼리에는 should 절만 있으므로, 두 should 절 중 최소 하나가 매칭되어야 합니다. 바깥쪽 must의 두 조건이 모두 AND로 결합되므로, 최종적으로 "type이 article이면서 tags에 elasticsearch 또는 search가 포함된 문서"가 검색됩니다.

이 구조를 단계별로 풀어 보면 다음과 같습니다.

```
중첩 bool 쿼리의 평가 흐름

바깥 bool (must):
  조건 1: type = "article"           --> 반드시 매칭 (AND)
  조건 2: 안쪽 bool (should):
            tags = "elasticsearch"    --> 둘 중
            tags = "search"           --> 하나 이상 매칭 (OR)

최종: (type = "article") AND (tags = "elasticsearch" OR tags = "search")
```

중첩은 여러 단계로 깊어질 수 있지만, 지나치게 깊은 중첩은 쿼리의 가독성을 떨어뜨리고 성능에도 부정적인 영향을 줄 수 있습니다. 가능하면 2단계 이내로 유지하는 것이 실무적으로 권장됩니다.

마지막으로 **boost** 파라미터를 살펴봅니다. boost는 특정 쿼리 절의 점수에 가중치를 부여하는 파라미터입니다. 기본값은 1.0이며, 1.0보다 큰 값을 지정하면 해당 절의 점수 기여가 커지고, 0과 1 사이의 값을 지정하면 점수 기여가 줄어듭니다.

예를 들어 title과 body 두 필드에서 검색하되, title에서의 매칭이 더 중요하다고 판단하는 경우를 봅니다.

```json
{
  "query": {
    "bool": {
      "should": [
        {
          "match": {
            "title": {
              "query": "elasticsearch tutorial",
              "boost": 2
            }
          }
        },
        {
          "match": {
            "body": {
              "query": "elasticsearch tutorial",
              "boost": 1
            }
          }
        }
      ]
    }
  }
}
```

title 필드의 match 쿼리에 boost를 2로 설정했습니다. body 필드의 match 쿼리는 boost가 1(기본값)입니다. title에서 매칭된 점수는 2배로 반영되므로, title에 검색어가 포함된 문서가 body에만 포함된 문서보다 더 높은 점수를 받습니다. boost는 must, should, filter 등 어떤 절 안의 쿼리에든 적용할 수 있지만, filter와 must_not 절은 점수를 계산하지 않으므로 boost를 지정해도 효과가 없습니다.

boost 값은 상대적인 가중치입니다. boost: 2가 점수를 정확히 2배로 만든다는 뜻은 아닙니다. Elasticsearch의 내부 점수 계산 알고리즘에 가중치로 반영될 뿐입니다. 따라서 boost 값을 지나치게 크게 설정하기보다, 절 사이의 상대적 중요도를 표현하는 데 집중하는 것이 바람직합니다.

정리하면, bool 쿼리는 must, filter, should, must_not 네 가지 절을 조합하여 복합 검색 조건을 구성하는 쿼리입니다. must와 filter는 모두 필수 조건이지만 점수 기여 여부가 다르고, should는 선택적 조건으로 점수를 높이며, must_not은 문서를 결과에서 제외합니다. minimum_should_match로 should 절의 매칭 요구 수를 제어하고, 중첩 bool 쿼리로 복잡한 논리 구조를 표현하며, boost 파라미터로 절 사이의 점수 가중치를 조정할 수 있습니다.

다음 단원인 3.2.4에서는 range, exists, prefix, wildcard 쿼리를 다룹니다.

이 단원을 마치면 bool 쿼리의 must, filter, should, must_not 절을 조합하여 복합 조건 검색을 작성할 수 있습니다.
