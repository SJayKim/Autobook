# 3.3.5 지리 검색

음식 배달 앱에서 "내 위치에서 2km 이내 식당"을 찾거나, 부동산 서비스에서 지도 위에 표시된 영역 안의 매물만 보여주거나, 물류 시스템에서 특정 배송 권역에 속하는 창고를 골라내는 상황을 떠올려 봅니다. 이 요구사항에는 공통점이 있습니다. 문서가 가진 위치 정보를 기준으로 "공간적 범위" 안에 있는 문서만 걸러내야 한다는 것입니다. Elasticsearch는 이런 공간 필터링을 위한 전용 쿼리를 여러 종류 제공합니다. 이 단원에서는 좌표 기반 지리 검색의 매핑부터 쿼리, 정렬, 집계까지를 차례로 살펴봅니다.

지리 검색을 수행하려면, 먼저 문서에 위치 정보를 저장할 필드가 있어야 합니다. 2.2.3에서 소개한 **geo_point** 타입이 바로 그 역할을 합니다. geo_point는 위도(latitude)와 경도(longitude) 한 쌍을 저장하는 필드 타입입니다. 위도는 적도를 기준으로 남북 위치를 나타내는 -90에서 90 사이의 값이고, 경도는 본초 자오선을 기준으로 동서 위치를 나타내는 -180에서 180 사이의 값입니다.

geo_point 필드를 매핑하는 방법은 다음과 같습니다.

```json
PUT /restaurants
{
  "mappings": {
    "properties": {
      "name": { "type": "text" },
      "location": { "type": "geo_point" }
    }
  }
}
```

"location" 필드의 type을 "geo_point"로 지정했습니다. 이 설정만으로 Elasticsearch는 해당 필드에 위도/경도 값을 저장하고, 지리 쿼리에 사용할 수 있도록 내부 색인을 생성합니다.

geo_point 필드에 값을 넣는 방식은 여러 가지입니다. 가장 명확한 방법은 lat과 lon을 명시하는 객체 형태입니다.

```json
{
  "name": "서울역 식당",
  "location": {
    "lat": 37.5547,
    "lon": 126.9706
  }
}
```

lat에 위도, lon에 경도를 넣습니다. 이 외에 문자열("37.5547,126.9706"), 배열([126.9706, 37.5547]), WKT 포맷("POINT(126.9706 37.5547)") 형태로도 입력할 수 있습니다. 배열과 WKT 포맷에서는 경도가 위도보다 앞에 온다는 점에 주의해야 합니다. GeoJSON 규격이 경도-위도 순서를 사용하기 때문입니다. 일상 대화에서는 "위도, 경도" 순서로 말하는 경우가 많으므로, 배열이나 WKT를 사용할 때 순서를 뒤바꾸는 실수가 잦습니다.

이제 geo_point 필드에 저장된 위치를 기준으로 문서를 걸러내는 쿼리를 하나씩 살펴봅니다.

**geo_distance 쿼리**는 지정한 좌표로부터 일정 반경 안에 있는 문서를 찾습니다. "내 위치에서 3km 이내 식당"처럼, 원형 영역 안의 문서만 필요할 때 사용합니다.

```json
{
  "query": {
    "geo_distance": {
      "distance": "3km",
      "location": {
        "lat": 37.5665,
        "lon": 126.9780
      }
    }
  }
}
```

"geo_distance" 안에 두 가지를 지정합니다. "distance"는 반경입니다. 숫자 뒤에 단위를 붙여 표현합니다. "location"은 원의 중심 좌표입니다. 이 쿼리는 서울 시청 좌표(위도 37.5665, 경도 126.9780)를 중심으로 반경 3km 안에 위치한 문서를 반환합니다.

distance에 사용할 수 있는 거리 단위를 정리합니다.

```
거리 단위 표기

약어    의미              예시
----    ------            ------
km      킬로미터          "3km"
m       미터              "500m"
mi      마일              "10mi"
yd      야드              "1000yd"
ft      피트              "5280ft"
cm      센티미터          "100cm"
mm      밀리미터          "5000mm"
```

별도로 단위를 지정하지 않으면 Elasticsearch는 미터(m)를 기본 단위로 사용합니다. 국내 서비스에서는 km와 m을 가장 많이 사용합니다.

geo_distance 쿼리는 내부적으로 중심 좌표와 각 문서의 좌표 사이의 거리를 계산합니다. 이 거리 계산에는 지구가 완전한 구가 아니라는 점이 반영됩니다. Elasticsearch는 기본적으로 "arc" 방식을 사용하는데, 이 방식은 지구의 곡률을 고려하여 두 점 사이의 대원 거리(great-circle distance)를 구합니다. 정확도가 높지만 계산량이 더 많습니다. 대안으로 "plane" 방식이 있는데, 이 방식은 지표면을 평면으로 간주하여 단순한 거리 공식을 적용합니다. 적도 부근에서는 차이가 작지만 고위도로 갈수록 오차가 커집니다. 대부분의 경우 기본값인 arc 방식으로 충분합니다.

**geo_bounding_box 쿼리**는 직사각형 영역 안에 있는 문서를 찾습니다. 지도 앱에서 화면에 보이는 사각형 범위 안의 결과만 가져올 때 적합합니다.

```json
{
  "query": {
    "geo_bounding_box": {
      "location": {
        "top_left": {
          "lat": 37.58,
          "lon": 126.95
        },
        "bottom_right": {
          "lat": 37.55,
          "lon": 127.00
        }
      }
    }
  }
}
```

"geo_bounding_box" 안에 대상 필드("location")를 지정하고, 사각형의 두 꼭짓점을 정의합니다. "top_left"는 사각형의 왼쪽 위 꼭짓점이고, "bottom_right"는 오른쪽 아래 꼭짓점입니다. 두 점이 결정되면 사각형 영역이 하나로 정해집니다. 이 쿼리는 서울 중심부의 약 3km x 5km 영역 안에 위치한 문서를 반환합니다.

geo_bounding_box 쿼리는 geo_distance 쿼리보다 계산이 단순합니다. 각 좌표의 위도와 경도를 범위와 비교하기만 하면 되기 때문입니다. 지도 화면의 현재 뷰포트에 해당하는 결과만 빠르게 가져와야 할 때 자주 사용됩니다.

다음은 **geo_polygon 쿼리**입니다. 이 쿼리는 세 개 이상의 꼭짓점으로 이루어진 다각형 영역 안에 있는 문서를 찾습니다. 행정 구역처럼 직사각형으로 표현하기 어려운 복잡한 경계를 다룰 때 유용합니다. 다만 geo_polygon 쿼리는 최신 버전에서 폐기(deprecated) 상태입니다. 동일한 기능이 필요하면 뒤에서 설명하는 geo_shape 쿼리를 사용하는 것이 권장됩니다.

여기서 geo_point 기반 쿼리 세 가지가 문서를 걸러내는 영역의 형태를 비교합니다.

```
geo_point 쿼리별 영역 형태

쿼리               영역 형태       정의 방식
-----------------  -----------    -----------------------
geo_distance       원             중심 좌표 + 반경
geo_bounding_box   직사각형       좌상단 + 우하단 좌표
geo_polygon        다각형         꼭짓점 좌표 목록
```

지금까지 살펴본 쿼리는 모두 geo_point 타입, 즉 하나의 점(좌표)을 기준으로 동작합니다. 그런데 실제 세계에서는 점이 아니라 면적을 가진 도형으로 데이터를 표현해야 할 때가 있습니다. 도시의 경계선, 배송 가능 구역, 공원의 윤곽 같은 것이 그렇습니다. 이때 사용하는 것이 **geo_shape** 타입입니다.

geo_shape 타입은 점뿐 아니라 선(LineString), 다각형(Polygon), 여러 다각형의 모음(MultiPolygon), 사각형(Envelope) 등 다양한 도형을 저장할 수 있습니다. 도형의 표현에는 GeoJSON 형식 또는 WKT(Well-Known Text) 형식을 사용합니다. GeoJSON은 좌표와 도형 유형을 JSON 구조로 표현하는 국제 표준 형식이고, WKT는 "POLYGON((lon lat, lon lat, ...))"처럼 텍스트 문자열로 표현하는 형식입니다.

geo_shape 필드를 매핑하는 예시입니다.

```json
PUT /delivery_zones
{
  "mappings": {
    "properties": {
      "zone_name": { "type": "keyword" },
      "area": { "type": "geo_shape" }
    }
  }
}
```

"area" 필드의 type을 "geo_shape"로 지정했습니다. 이 필드에 다각형이나 사각형 같은 도형 데이터를 저장할 수 있습니다.

geo_shape 쿼리는 문서에 저장된 도형과 쿼리에 지정한 도형 사이의 공간적 관계를 판별합니다. 네 가지 관계 연산자를 사용할 수 있습니다.

```
geo_shape 공간 관계 연산자

연산자       의미                              예시
----------   ------                            ------
INTERSECTS   두 도형이 겹치는 부분이 있음       기본값. 가장 많이 사용
WITHIN       문서의 도형이 쿼리 도형 안에       "이 구역 안에 완전히
             완전히 포함됨                      포함된 배송 권역"
CONTAINS     문서의 도형이 쿼리 도형을          "이 지점을 완전히 감싸는
             완전히 감쌈                        배송 권역"
DISJOINT     두 도형이 전혀 겹치지 않음         "이 구역과 겹치지 않는
                                               배송 권역"
```

기본 연산자는 INTERSECTS입니다. 별도로 지정하지 않으면 두 도형이 조금이라도 겹치면 매칭됩니다.

geo_point와 geo_shape는 용도가 다릅니다. geo_point는 위도/경도 한 점을 저장하며 "이 점이 어떤 영역 안에 있는가"를 판별합니다. geo_shape는 면적을 가진 도형을 저장하며 "이 도형이 저 도형과 어떤 공간적 관계에 있는가"를 판별합니다. 한 가지 주의할 점은 geo_point 필드와 geo_shape 필드를 하나의 쿼리 안에서 함께 조회할 수 없다는 것입니다. 각 필드 타입에 대응하는 쿼리를 따로 사용해야 합니다.

지리 검색에서 자주 사용하는 기능 중 하나가 거리 기준 정렬입니다. 검색 결과를 "가까운 순"으로 정렬하려면 **sort** 파라미터에 **_geo_distance**를 사용합니다.

```json
{
  "query": {
    "geo_distance": {
      "distance": "5km",
      "location": {
        "lat": 37.5665,
        "lon": 126.9780
      }
    }
  },
  "sort": [
    {
      "_geo_distance": {
        "location": {
          "lat": 37.5665,
          "lon": 126.9780
        },
        "order": "asc",
        "unit": "km"
      }
    }
  ]
}
```

"sort" 배열 안에 "_geo_distance"를 넣고, 기준 좌표와 정렬 방향을 지정합니다. "location"은 거리를 측정할 기준 좌표입니다. "order"를 "asc"로 설정하면 가까운 문서가 먼저 나옵니다. "unit"은 정렬에 사용하는 거리 단위로, 각 문서의 거리 값이 이 단위로 계산되어 응답에 포함됩니다.

이 쿼리는 서울 시청에서 5km 이내 문서를 찾되, 가까운 순서로 정렬합니다. 응답의 각 문서에는 "sort" 필드에 기준 좌표로부터의 거리가 km 단위로 표시됩니다. 배달 앱에서 "가까운 식당 순"으로 결과를 보여줄 때 이 방식을 사용합니다.

마지막으로 지리 데이터를 집계하는 방법을 살펴봅니다. Elasticsearch는 지리 좌표를 일정한 격자(grid)로 나누어 각 칸에 속하는 문서 수를 세는 집계를 지원합니다. 대표적인 것이 **geohash grid 집계**와 **geotile grid 집계**입니다.

이 집계를 이해하려면 먼저 격자 부호화 개념을 알아야 합니다. 지구 표면을 일정 규칙에 따라 격자로 나누고, 각 격자에 고유한 부호를 붙이는 방식입니다. **geohash**는 위도와 경도를 번갈아 이진 분할한 뒤 Base32 문자열로 부호화하는 방식입니다. 예를 들어 서울 시청 부근의 좌표는 "wydm9" 같은 문자열로 표현됩니다. 문자열이 길수록 더 좁은 영역을 가리킵니다. "wydm"은 "wydm9"보다 넓은 영역입니다. geohash grid 집계는 지정한 정밀도(precision)에 따라 문서를 geohash 격자에 배분하고, 각 격자에 속하는 문서 수를 셉니다.

**geotile**은 웹 지도에서 사용하는 타일 좌표 체계를 기반으로 합니다. 줌 레벨(zoom level)에 따라 지구 표면을 정사각형 타일로 나눕니다. 줌 레벨 0에서는 지구 전체가 하나의 타일이고, 줌 레벨이 1 올라갈 때마다 타일 수가 네 배로 늘어납니다. 웹 지도 서비스에서 줌 인/줌 아웃에 따라 데이터 밀도를 표시할 때 적합합니다.

geohash grid 집계 예시를 봅니다.

```json
{
  "size": 0,
  "aggs": {
    "restaurant_grid": {
      "geohash_grid": {
        "field": "location",
        "precision": 5
      }
    }
  }
}
```

"size"를 0으로 설정하여 검색 결과 문서는 반환하지 않고 집계 결과만 받습니다. "geohash_grid" 집계에서 "field"는 geo_point 타입의 필드 이름이고, "precision"은 geohash 문자열의 길이입니다. precision이 5이면 약 4.9km x 4.9km 크기의 격자로 나눕니다. precision을 높이면 격자가 더 촘촘해지고, 낮추면 더 넓어집니다. 응답에는 각 geohash 격자의 부호("wydm9" 등)와 해당 격자에 속하는 문서 수가 포함됩니다. 지도 위에 히트맵을 그리거나, 지역별 밀집도를 시각화할 때 유용합니다.

geotile grid 집계도 사용 방법이 비슷합니다. "geohash_grid" 대신 "geotile_grid"를 사용하고, "precision" 대신 "zoom"(또는 동일하게 "precision")으로 줌 레벨을 지정합니다. 웹 지도의 타일 체계와 일치하므로, 지도 시각화 라이브러리와 연동하기가 편리합니다.

정리하면, Elasticsearch의 지리 검색은 geo_point 타입으로 좌표를 저장하는 것에서 시작합니다. geo_distance 쿼리는 중심 좌표와 반경으로 원형 영역을 정의하고, geo_bounding_box 쿼리는 두 꼭짓점으로 직사각형 영역을 정의하여 해당 영역 안의 문서를 걸러냅니다. 면적을 가진 도형이 필요하면 geo_shape 타입과 INTERSECTS, WITHIN, CONTAINS, DISJOINT 같은 공간 관계 연산자를 사용합니다. 검색 결과를 거리 순으로 정렬하려면 sort의 _geo_distance를 활용하고, 격자 기반 집계에는 geohash grid 또는 geotile grid를 사용하여 지역별 문서 분포를 파악합니다.

다음 단원인 3.4.1에서는 BM25 알고리즘 이해를 다룹니다.

이 단원을 마치면 geo_distance 쿼리로 반경 내 문서를 검색할 수 있고, geo_bounding_box 쿼리로 직사각형 범위 검색을 수행할 수 있습니다.
