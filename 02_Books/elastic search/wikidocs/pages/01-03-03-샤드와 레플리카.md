# 1.3.3 샤드와 레플리카

1.3.2에서 Elasticsearch 인덱스가 JSON 문서들의 논리적 컨테이너라는 것을 살펴보았습니다. 인덱스를 만들 때 settings에서 number_of_shards와 number_of_replicas를 지정했고, 샤드 수는 데이터 분산과 병렬 처리에 직접 영향을 미친다고 언급했습니다. 그런데 샤드가 정확히 무엇이고, 레플리카는 왜 필요하며, 이 둘이 어떻게 맞물려 동작하는지는 아직 다루지 않았습니다. 이 단원에서는 샤드와 레플리카의 정체를 밝히고, 문서가 특정 샤드에 배치되는 원리부터 장애 대응과 성능 확장까지 그 메커니즘을 하나씩 풀어 봅니다.

인덱스에 문서가 쌓이면 데이터 양은 계속 늘어납니다. 하나의 물리적 저장소에 모든 데이터를 담으면 두 가지 문제가 생깁니다. 첫째, 단일 디스크의 용량에는 한계가 있어 데이터가 충분히 커지면 더 이상 저장할 수 없습니다. 둘째, 검색이나 인덱싱 요청이 모두 한 곳에 집중되므로 처리 속도가 느려집니다. 이 문제를 해결하려면 데이터를 여러 조각으로 나누어 여러 노드에 분산시켜야 합니다. Elasticsearch에서 이 조각 하나하나를 **샤드**(shard)라고 합니다.

샤드는 인덱스의 물리적 부분 집합입니다. 각 샤드는 내부적으로 독립된 하나의 Lucene 인덱스입니다. 1.3.2에서 Elasticsearch 인덱스가 '논리적 컨테이너'라고 했던 이유가 여기에 있습니다. 사용자가 인덱스라는 이름 하나를 상대하지만, 실제 데이터는 여러 샤드에 나뉘어 저장되고, 각 샤드는 자체적으로 문서를 저장하고 검색하는 완전한 능력을 갖추고 있습니다.

샤드에는 두 종류가 있습니다. 하나는 **프라이머리 샤드**(primary shard)이고, 다른 하나는 **레플리카 샤드**(replica shard)입니다. 이 둘은 역할이 다르므로 따로 살펴봅니다.

프라이머리 샤드는 원본 데이터를 담는 샤드입니다. 인덱스에 문서를 저장(인덱싱)하면, 그 문서는 반드시 하나의 프라이머리 샤드에 먼저 기록됩니다. 모든 쓰기 작업의 첫 번째 진입점이 프라이머리 샤드입니다. 인덱스를 만들 때 number_of_shards로 지정한 숫자가 바로 프라이머리 샤드의 개수입니다. Elasticsearch 7.0 이후 이 값을 생략하면 기본값 1이 적용됩니다.

레플리카 샤드는 프라이머리 샤드의 완전한 복제본입니다. '완전한'이란 프라이머리 샤드에 있는 모든 문서를 동일하게 갖고 있다는 뜻입니다. 인덱스를 만들 때 number_of_replicas로 복제본 수를 지정합니다. 기본값은 1입니다. 예를 들어 프라이머리 샤드가 3개이고 레플리카가 1이면, 레플리카 샤드는 프라이머리 3개 각각에 대해 1개씩, 총 3개가 만들어집니다. 인덱스 전체로 보면 프라이머리 3개 + 레플리카 3개 = 총 6개의 샤드가 존재하게 됩니다.

이 구조를 시각적으로 정리하면 다음과 같습니다. 프라이머리 샤드 3개, 레플리카 1인 인덱스가 노드 3대에 분산된 모습입니다.

```
인덱스: my-index (number_of_shards=3, number_of_replicas=1)

노드 A               노드 B               노드 C
+-----------+        +-----------+        +-----------+
| P0        |        | P1        |        | P2        |
| (primary) |        | (primary) |        | (primary) |
+-----------+        +-----------+        +-----------+
| R1        |        | R2        |        | R0        |
| (replica) |        | (replica) |        | (replica) |
+-----------+        +-----------+        +-----------+

P0, P1, P2 = 프라이머리 샤드 0, 1, 2
R0, R1, R2 = P0, P1, P2의 레플리카 샤드
```

이 다이어그램에서 눈여겨볼 점이 있습니다. 프라이머리 샤드와 그 레플리카 샤드가 같은 노드에 있지 않습니다. P0는 노드 A에 있고, P0의 복제본인 R0는 노드 C에 있습니다. 이것은 우연이 아니라 Elasticsearch가 강제하는 규칙입니다. 같은 노드에 원본과 복제본을 함께 두면, 그 노드가 고장났을 때 원본과 복제본이 동시에 사라지므로 복제의 의미가 없어집니다.

그렇다면 인덱스에 문서를 저장할 때, Elasticsearch는 어떤 프라이머리 샤드에 그 문서를 넣을지 어떻게 결정할까요. 무작위로 고르는 것이 아닙니다. 정해진 공식이 있습니다.

```
shard = hash(_routing) % number_of_primary_shards
```

이 공식의 각 부분을 풀어 봅니다. _routing은 라우팅 값입니다. 별도로 지정하지 않으면 문서의 _id가 기본 라우팅 값으로 쓰입니다. hash()는 이 라우팅 값을 해시 함수에 통과시켜 숫자로 변환하는 과정입니다. number_of_primary_shards는 인덱스를 만들 때 지정한 프라이머리 샤드의 총 개수입니다. % 연산은 나머지를 구하는 연산입니다.

구체적인 예를 들겠습니다. 프라이머리 샤드가 3개인 인덱스에 _id가 "doc_42"인 문서를 저장한다고 합시다. Elasticsearch는 "doc_42"를 해시 함수에 넣어 숫자를 얻고, 그 숫자를 3으로 나눈 나머지를 구합니다. 나머지는 0, 1, 2 중 하나이므로, 문서는 프라이머리 샤드 0, 1, 2 중 하나에 배치됩니다. 같은 _id를 다시 계산하면 해시 결과도 동일하므로, 항상 같은 샤드가 선택됩니다. 이 덕분에 나중에 문서를 조회할 때도 같은 공식으로 해당 문서가 어느 샤드에 있는지 바로 알 수 있습니다. 클러스터 전체를 뒤질 필요가 없습니다.

이 공식에는 중요한 함의가 있습니다. 공식의 분모가 number_of_primary_shards이므로, 이 값이 바뀌면 같은 문서에 대해 계산 결과가 달라집니다. 예를 들어 샤드 수가 3일 때 hash("doc_42") % 3 = 1이었던 문서가, 샤드 수를 5로 바꾸면 hash("doc_42") % 5 = 다른 값이 될 수 있습니다. 기존에 샤드 1에 저장해 둔 문서를 샤드 3에서 찾으려 하게 되는 셈입니다. 데이터를 전혀 찾을 수 없는 상태가 됩니다.

이것이 바로 **프라이머리 샤드 수를 인덱스 생성 이후에 변경할 수 없는** 이유입니다. 라우팅 공식에 샤드 수가 포함되어 있어, 샤드 수를 바꾸면 기존 문서의 위치 매핑이 모두 깨지기 때문입니다. 1.3.2에서 number_of_shards가 정적 설정이라고 했던 배경이 여기에 있습니다.

만약 운영 중에 샤드 수를 꼭 바꿔야 하는 상황이 온다면 어떻게 할까요. 세 가지 우회 방법이 있습니다. 첫째, Split API를 사용하면 기존 샤드를 더 작은 샤드로 쪼개 샤드 수를 늘릴 수 있습니다. 둘째, Shrink API를 사용하면 여러 샤드를 합쳐 샤드 수를 줄일 수 있습니다. 셋째, Reindex API를 사용하면 원하는 샤드 수로 새 인덱스를 만들고, 기존 인덱스의 데이터를 전부 복사할 수 있습니다. 이 세 가지 모두 기존 인덱스의 number_of_shards를 직접 수정하는 것이 아니라, 새로운 물리적 구조를 만드는 방식이라는 점에 주목합니다.

프라이머리 샤드 수는 고정이지만, **레플리카 수는 언제든 변경할 수 있습니다**. 인덱싱이나 검색을 중단하지 않고도 즉시 적용됩니다. 변경 방법은 _settings API에 PUT 요청을 보내는 것입니다.

```
PUT /my-index/_settings
{
  "number_of_replicas": 2
}
```

이 요청은 my-index의 레플리카 수를 2로 변경합니다. 프라이머리 샤드가 3개라면 레플리카 샤드가 각 프라이머리당 2개씩, 총 6개가 됩니다. 인덱스 전체로는 프라이머리 3개 + 레플리카 6개 = 총 9개의 샤드입니다.

레플리카 수를 동적으로 바꿀 수 있다는 특성은 실무에서 유용하게 쓰입니다. 예를 들어 대량의 데이터를 한꺼번에 적재하는 작업(초기 데이터 마이그레이션 등)에서는 레플리카를 0으로 설정합니다. 프라이머리 샤드에 데이터를 기록한 뒤 레플리카로 복제하는 과정이 생략되므로 쓰기 속도가 빨라집니다. 적재가 끝나면 레플리카를 다시 1 이상으로 올려 장애 대비를 복원합니다.

이제 레플리카 샤드가 제공하는 두 가지 핵심 가치를 살펴봅니다. 첫 번째는 **고가용성**(high availability)입니다. 고가용성이란 시스템의 일부가 고장나더라도 서비스 전체가 계속 동작하는 능력을 뜻합니다.

앞의 다이어그램에서 노드 A가 갑자기 내려갔다고 가정합니다. 노드 A에는 프라이머리 샤드 P0와 레플리카 샤드 R1이 있었습니다. P0가 사라졌으므로 샤드 0의 원본이 없어진 셈입니다. 이때 Elasticsearch 클러스터는 노드 C에 있는 R0(P0의 레플리카)를 자동으로 새 프라이머리 샤드로 승격시킵니다. 관리자가 개입하지 않아도 데이터 손실 없이 서비스가 계속됩니다.

이 메커니즘이 작동하려면 전제 조건이 있습니다. 레플리카가 n개 있다면 클러스터에 최소 n+1개의 노드가 필요합니다. 레플리카가 1이면 노드가 최소 2대 있어야 프라이머리와 레플리카를 서로 다른 노드에 배치할 수 있습니다. 노드가 1대뿐이면 레플리카를 배치할 곳이 없어 레플리카 샤드가 미할당(unassigned) 상태로 남습니다. 이 상태에서는 클러스터 상태가 yellow로 표시됩니다. 1.3.2에서 _cat/indices의 출력에 health 열이 있었는데, green은 모든 프라이머리와 레플리카가 정상 배치된 상태, yellow는 프라이머리는 정상이지만 일부 레플리카가 미할당인 상태, red는 일부 프라이머리 샤드마저 미할당인 상태를 뜻합니다.

레플리카의 두 번째 가치는 **읽기 처리량 향상**입니다. 검색 요청이 들어오면 Elasticsearch는 프라이머리 샤드와 레플리카 샤드를 구분하지 않고, 둘 모두를 검색 대상으로 사용합니다. 즉 레플리카가 많을수록 동시에 검색을 처리할 수 있는 샤드가 늘어나므로, 같은 시간에 더 많은 읽기 요청을 소화할 수 있습니다.

검색 요청이 처리되는 흐름을 단계별로 정리하면 다음과 같습니다.

```
검색 분산 처리 흐름

1. 클라이언트가 검색 요청을 클러스터의 아무 노드에 보냅니다.
2. 요청을 받은 노드가 코디네이팅 노드 역할을 합니다.
3. 코디네이팅 노드는 인덱스의 모든 샤드에 요청을 전달합니다.
   이때 같은 샤드 번호의 프라이머리와 레플리카 중 하나를 골라 보냅니다.
4. 각 샤드가 로컬에서 검색을 수행하고 상위 결과를 반환합니다.
5. 코디네이팅 노드가 각 샤드의 결과를 합쳐 최종 응답을 만듭니다.
```

3단계에서 레플리카가 있으면 코디네이팅 노드의 선택지가 넓어집니다. 레플리카가 1이면 샤드 번호 0에 대해 P0과 R0 중 하나를 고를 수 있고, 레플리카가 2이면 P0, R0-a, R0-b 세 개 중 하나를 고를 수 있습니다. 여러 검색 요청이 동시에 들어오면 서로 다른 복제본에 분산되어 병렬 처리됩니다.

다만 레플리카를 무한정 늘리는 것이 항상 좋지는 않습니다. 노드 하나에 샤드가 많아지면 각 샤드가 사용할 수 있는 파일시스템 캐시 공간이 줄어들어, 개별 샤드의 검색 성능이 떨어질 수 있습니다. 노드 자원과 검색 부하 사이에서 균형을 잡아야 합니다.

쓰기 요청은 검색과 다릅니다. 문서를 저장하거나 수정하는 쓰기 작업은 반드시 프라이머리 샤드를 먼저 거칩니다. 프라이머리 샤드에 기록이 완료된 뒤에야 레플리카 샤드로 복제됩니다. 레플리카는 쓰기의 진입점이 아니라 복제 대상입니다.

마지막으로 샤드의 크기를 어떻게 잡아야 하는지 살펴봅니다. 프라이머리 샤드 수는 인덱스 생성 시 고정되므로, 처음부터 적절한 수를 정해야 합니다. 판단 기준은 샤드 하나의 크기입니다. Elasticsearch에서 권장하는 샤드 크기 범위는 **10~50GB**이며, 최적 범위는 **20~40GB**입니다.

샤드가 너무 작으면 어떤 일이 벌어질까요. 샤드 수가 지나치게 많아지는 상태를 오버샤딩(oversharding)이라고 합니다. 각 샤드는 독립된 Lucene 인덱스이므로, 샤드마다 메타데이터와 파일 핸들을 유지해야 합니다. 샤드 수가 불필요하게 많으면 이 오버헤드가 누적되어 클러스터 전체의 메모리 사용량이 늘어납니다. 검색 시에도 코디네이팅 노드가 조율해야 할 샤드 수가 많아져 응답 시간이 길어집니다.

반대로 샤드가 너무 크면 다른 문제가 생깁니다. 검색과 인덱싱 작업이 하나의 거대한 Lucene 인덱스를 상대해야 하므로 느려집니다. 노드 장애 시 대용량 샤드를 다른 노드로 복구하는 데 시간이 오래 걸립니다. 문서 수 기준으로는 샤드당 2억 개 미만을 유지하는 것이 권장됩니다.

이 기준을 실제 상황에 적용해 봅니다. 전체 데이터가 약 100GB로 예상되는 인덱스를 만든다고 합시다. 샤드 하나의 크기를 20~40GB 사이로 잡으면, 프라이머리 샤드는 3~5개가 적절합니다. 반대로 샤드를 50개로 잡으면 샤드당 2GB에 불과해 오버샤딩이 됩니다. 데이터 총량과 증가 추세를 예측한 뒤, 샤드 하나가 10~50GB 범위에 들어오도록 역산하는 것이 기본 전략입니다.

정리하면, 샤드는 인덱스의 물리적 부분 집합이며, 프라이머리 샤드에 원본 데이터가 기록되고 레플리카 샤드에 복제됩니다. 문서는 `hash(_routing) % number_of_primary_shards` 공식으로 특정 프라이머리 샤드에 배치되며, 이 공식 때문에 프라이머리 샤드 수는 인덱스 생성 이후 변경할 수 없습니다. 레플리카 수는 언제든 동적으로 바꿀 수 있으며, 레플리카는 노드 장애 시 자동 승격으로 고가용성을 보장하고, 읽기 요청을 분산하여 검색 처리량을 높입니다. 샤드 하나의 크기는 10~50GB 범위를 유지하는 것이 권장됩니다.

다음 단원인 2.1.1에서는 Phase 2 '데이터 모델링'의 첫 번째 주제인 동적 매핑과 명시적 매핑을 다룹니다. Phase 1에서 문서, 인덱스, 샤드라는 Elasticsearch의 기초 구조를 모두 살펴보았으므로, Phase 2에서는 이 구조 위에 데이터를 효과적으로 정의하고 다루는 방법으로 넘어갑니다.

이 단원을 마치면 프라이머리 샤드와 레플리카 샤드의 역할을 설명할 수 있고, 샤드 수와 레플리카 수를 결정하는 기준을 제시할 수 있습니다.
