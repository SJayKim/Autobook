# 6.2.3 인제스트 파이프라인

6.2.1에서 다룬 Logstash는 별도의 프로세스로 동작하면서 데이터를 변환합니다. 그런데 변환 작업이 비교적 단순하다면, 별도 프로세스를 운영하는 것이 부담이 될 수 있습니다. 예를 들어 타임스탬프를 파싱하거나, 필드 이름을 변경하거나, 비정형 텍스트에서 몇 개의 필드를 추출하는 작업이라면, Elasticsearch 안에서 직접 처리할 수 있으면 아키텍처가 단순해집니다.

**인제스트 파이프라인(ingest pipeline)**은 Elasticsearch 노드 내부에서 문서가 인덱싱되기 직전에 전처리를 수행하는 기능입니다. 별도의 프로세스나 서버 없이, Elasticsearch 자체가 데이터를 가공합니다. 인제스트 파이프라인은 하나 이상의 **프로세서(processor)**로 구성됩니다. 각 프로세서는 특정한 변환 작업 하나를 담당하며, 문서가 파이프라인에 진입하면 정의된 순서대로 프로세서를 차례로 거칩니다.

인제스트 파이프라인을 생성하려면 PUT _ingest/pipeline API를 사용합니다. 다음 예는 비정형 로그 메시지에서 IP 주소, 사용자명, 타임스탬프, HTTP 메서드, 요청 경로를 추출하는 파이프라인입니다.

```json
PUT _ingest/pipeline/my-log-pipeline
{
  "description": "Parse access log messages",
  "processors": [
    {
      "grok": {
        "field": "message",
        "patterns": [
          "%{IP:client_ip} - %{USER:auth} \\[%{HTTPDATE:timestamp}\\] %{WORD:verb} %{URIPATHPARAM:request}"
        ]
      }
    },
    {
      "date": {
        "field": "timestamp",
        "formats": ["dd/MMM/yyyy:HH:mm:ss Z"],
        "target_field": "@timestamp"
      }
    },
    {
      "rename": {
        "field": "verb",
        "target_field": "http_method"
      }
    }
  ]
}
```

이 파이프라인을 구성하는 세 개의 프로세서를 차례로 살펴보겠습니다.

**grok 프로세서**는 Logstash의 grok 필터와 동일한 역할을 합니다. field에 지정한 필드(여기서는 message)의 값을 patterns에 정의한 패턴으로 분석합니다. %{IP:client_ip}는 IP 주소 패턴을 인식하여 client_ip 필드에 저장합니다. %{HTTPDATE:timestamp}는 HTTP 날짜 형식을 인식하여 timestamp 필드에 저장합니다. 패턴 문법은 6.2.1에서 설명한 Grok 패턴과 동일합니다.

**date 프로세서**는 텍스트 형태의 날짜를 파싱하여 날짜 타입으로 변환합니다. field는 원본 날짜 문자열이 담긴 필드이고, formats는 해당 문자열의 형식입니다. target_field는 변환 결과를 저장할 필드인데, @timestamp를 지정하면 Elasticsearch의 표준 타임스탬프 필드에 값이 들어갑니다.

**rename 프로세서**는 필드 이름을 변경합니다. 이 예에서는 verb를 http_method로 바꿉니다.

이 외에도 자주 사용하는 프로세서가 있습니다. **set 프로세서**는 지정한 필드에 고정값이나 다른 필드의 값을 설정합니다. **remove 프로세서**는 불필요한 필드를 삭제합니다. 이 두 프로세서와 rename을 함께 사용하면 문서의 필드 구조를 자유롭게 재구성할 수 있습니다.

**foreach 프로세서**는 배열 필드의 각 요소에 대해 다른 프로세서를 반복 적용합니다. 예를 들어 tags라는 배열 필드의 모든 요소를 소문자로 변환하고 싶을 때 사용합니다.

```json
{
  "foreach": {
    "field": "tags",
    "processor": {
      "lowercase": {
        "field": "_ingest._value"
      }
    }
  }
}
```

field는 반복 대상 배열이고, processor 안의 _ingest._value는 현재 순회 중인 요소를 가리킵니다.

**pipeline 프로세서**는 다른 파이프라인을 호출합니다. 이를 파이프라인 체이닝이라고 합니다. 공통 전처리 로직을 별도 파이프라인으로 분리하고, 여러 파이프라인에서 재사용할 수 있습니다.

```json
{
  "pipeline": {
    "name": "common-preprocessing"
  }
}
```

이 프로세서를 만나면, 문서가 common-preprocessing 파이프라인으로 전달되어 해당 파이프라인의 프로세서를 거친 뒤 다시 원래 파이프라인으로 돌아옵니다.

프로세서 실행 도중 오류가 발생하면 기본적으로 해당 문서의 인덱싱이 실패합니다. 하지만 모든 오류에서 인덱싱을 중단시키고 싶지 않을 수 있습니다. **on_failure** 속성을 사용하면 오류 발생 시 대체 동작을 정의할 수 있습니다.

```json
{
  "grok": {
    "field": "message",
    "patterns": ["%{IP:client_ip} %{WORD:action}"],
    "on_failure": [
      {
        "set": {
          "field": "parse_error",
          "value": "grok pattern did not match"
        }
      }
    ]
  }
}
```

이 설정에서 grok 패턴이 message와 일치하지 않으면, 인덱싱을 실패시키는 대신 parse_error 필드에 오류 메시지를 기록하고 다음 프로세서로 넘어갑니다. on_failure는 개별 프로세서에 설정할 수도 있고, 파이프라인 전체에 설정할 수도 있습니다.

파이프라인을 프로덕션에 적용하기 전에 **simulate API**로 테스트할 수 있습니다. 샘플 문서를 제공하면 파이프라인을 거친 결과를 미리 확인합니다.

```json
POST _ingest/pipeline/my-log-pipeline/_simulate
{
  "docs": [
    {
      "_source": {
        "message": "192.168.1.1 - admin [15/Jan/2024:10:30:00 +0900] GET /index.html"
      }
    }
  ]
}
```

이 요청은 샘플 문서를 my-log-pipeline에 통과시킨 결과를 반환합니다. client_ip, auth, timestamp, http_method, request 필드가 올바르게 추출되었는지, @timestamp가 정상적으로 파싱되었는지를 확인할 수 있습니다. 실제 인덱싱이 발생하지 않으므로 안전하게 반복 테스트가 가능합니다.

파이프라인을 인덱싱에 적용하는 방법은 두 가지입니다. 첫째, 인덱싱 요청에 pipeline 파라미터를 명시합니다.

```json
POST my-index/_doc?pipeline=my-log-pipeline
{
  "message": "192.168.1.1 - admin [15/Jan/2024:10:30:00 +0900] GET /api/users"
}
```

둘째, 인덱스 설정에 기본 파이프라인을 지정하면 해당 인덱스에 들어오는 모든 문서에 자동 적용됩니다. index.default_pipeline 설정을 사용합니다.

정리하면, 인제스트 파이프라인은 Elasticsearch 노드 내부에서 문서 인덱싱 직전에 전처리를 수행하는 기능입니다. grok으로 비정형 텍스트를 파싱하고, date로 타임스탬프를 변환하며, rename, set, remove로 필드 구조를 조정합니다. foreach로 배열 요소를 반복 처리하고, pipeline 프로세서로 다른 파이프라인을 호출하여 로직을 재사용할 수 있습니다. on_failure로 오류를 처리하고, simulate API로 파이프라인을 사전에 검증할 수 있습니다.

다음 단원인 6.3.1에서는 인덱싱 성능 최적화를 다룹니다. 대량의 데이터를 빠르게 인덱싱하기 위한 설정과 기법을 알아봅니다.

이 단원을 마치면 인제스트 파이프라인으로 문서 인덱싱 전 전처리를 구성할 수 있으며, grok, date, rename, set 프로세서를 활용할 수 있습니다.
