# 3.2.4 range, exists, prefix, wildcard 쿼리

3.2.3에서 bool 쿼리를 사용하여 여러 조건을 조합하는 방법을 살펴보았습니다. bool 쿼리의 filter 절 안에 range 쿼리를 넣어 날짜 범위를 제한하는 예시도 등장했습니다. 이번 단원에서는 그 range 쿼리를 비롯하여, Elasticsearch가 제공하는 여러 가지 용어 수준 쿼리(term-level query)를 하나씩 살펴봅니다.

용어 수준 쿼리란 무엇인지 먼저 짚어야 합니다. 3.2.2에서 다룬 match 쿼리는 검색어를 분석기에 통과시킨 뒤 매칭합니다. 반면 용어 수준 쿼리는 검색어를 분석하지 않고, 인덱스에 저장된 값과 직접 비교합니다. term 쿼리가 대표적입니다. 이 단원에서 다루는 range, exists, prefix, wildcard, regexp, ids 쿼리도 모두 용어 수준 쿼리에 속합니다. 입력값을 분석기에 통과시키지 않는다는 공통점이 있으므로, keyword 필드나 숫자 필드, 날짜 필드처럼 분석이 불필요한 필드에 사용하는 것이 일반적입니다.

**range 쿼리**는 특정 필드의 값이 지정한 범위 안에 있는 문서를 찾습니다. 숫자 필드의 최솟값과 최댓값을 정하거나, 날짜 필드의 시작일과 종료일을 정하는 식으로 사용합니다.

범위를 지정할 때 네 가지 경계 파라미터를 사용합니다. **gte**는 "greater than or equal to"의 약자로, 지정한 값 이상을 뜻합니다. **lte**는 "less than or equal to"의 약자로, 지정한 값 이하를 뜻합니다.

gt는 지정한 값을 포함하지 않고 초과만, lt는 미만만 매칭합니다. 네 파라미터를 모두 쓸 필요는 없으며, 필요한 경계만 지정하면 됩니다.

숫자 필드에 range 쿼리를 적용하는 예를 봅니다. 상품 인덱스에서 가격이 100 이하인 문서를 찾는 쿼리입니다.

```json
{
  "query": {
    "range": {
      "price": {
        "lte": 100
      }
    }
  }
}
```

"query" 안에 "range"를 지정하고, 대상 필드로 "price"를 넣었습니다. "lte": 100은 price 값이 100 이하인 문서만 결과에 포함한다는 뜻입니다. gte와 lte를 함께 쓰면 양쪽 경계를 모두 지정할 수 있습니다. 예를 들어 "gte": 50, "lte": 100으로 설정하면 가격이 50 이상 100 이하인 문서가 매칭됩니다.

range 쿼리는 날짜 필드에도 동일하게 적용됩니다. 날짜 필드에서는 고정된 날짜 문자열을 직접 지정할 수 있습니다. 다음은 created_at 필드가 2024년 1월 1일 이후이고 2025년 1월 1일 미만인 문서를 찾는 예시입니다.

```json
{
  "query": {
    "range": {
      "created_at": {
        "gte": "2024-01-01",
        "lt": "2025-01-01"
      }
    }
  }
}
```

gte에 "2024-01-01"을 지정하여 이 날짜를 포함한 이후 문서를 매칭하고, lt에 "2025-01-01"을 지정하여 이 날짜 미만까지만 매칭합니다. gte와 lt를 조합하면 시작일은 포함하되 종료일은 제외하는 반열린 구간을 표현할 수 있습니다.

고정된 날짜 문자열만으로는 "최근 7일", "오늘부터 한 달 전"처럼 현재 시점을 기준으로 한 상대 범위를 표현하기 어렵습니다. 이를 위해 Elasticsearch는 **날짜 연산(date math)**이라는 표기법을 제공합니다. 날짜 연산의 기준점은 "now"입니다. now는 쿼리를 실행하는 시점의 현재 시각을 뜻합니다. now 뒤에 빼기(-) 또는 더하기(+) 기호와 시간 단위를 붙여 상대 시점을 표현합니다. 시간 단위는 d(일), h(시간), m(분), w(주), M(월), y(년) 등을 사용할 수 있습니다.

예를 들어 "now-7d"는 현재 시각에서 7일을 뺀 시점입니다. "now-1M"은 한 달 전, "now+1h"는 한 시간 후입니다.

날짜 연산에는 **반올림(rounding)** 표기도 있습니다. 슬래시(/) 뒤에 단위를 붙이면 해당 단위의 시작 시점으로 내림합니다. "now-7d/d"는 현재에서 7일을 뺀 뒤, 그 날의 시작 시점(자정)으로 내림한 값입니다. 반올림은 캐시 성능에도 영향을 줍니다. now를 그대로 쓰면 밀리초 단위로 매번 달라지기 때문에 Elasticsearch가 쿼리 결과를 캐시하기 어렵습니다. "/d"나 "/m"으로 반올림하면 같은 날(또는 같은 분) 안에 실행된 쿼리들이 동일한 값을 갖게 되어 캐시를 공유할 수 있습니다.

다음은 최근 7일간의 로그를 검색하는 실제 예시입니다.

```json
{
  "query": {
    "range": {
      "@timestamp": {
        "gte": "now-7d/d",
        "lte": "now/d"
      }
    }
  }
}
```

"gte": "now-7d/d"는 7일 전의 자정 이후, "lte": "now/d"는 오늘의 자정 이후(오늘 하루 포함)를 뜻합니다. 날짜 필드에 range 쿼리를 사용할 때는 관련성 점수가 필요한 경우가 거의 없으므로, 3.2.3에서 다룬 bool 쿼리의 filter 절에 넣어 캐시 이점을 얻는 것이 일반적입니다.

날짜 연산의 주요 표기를 정리하면 다음과 같습니다.

```
날짜 연산(date math) 표기 예시

표기          의미
-----------   ------------------------------------------
now           현재 시각
now-7d        현재에서 7일 전
now-1M        현재에서 1개월 전
now+1h        현재에서 1시간 후
now/d         오늘의 자정(일 단위 내림)
now-7d/d      7일 전의 자정
now-1h/m      1시간 전을 분 단위로 내림
```

이제 **exists 쿼리**를 살펴봅니다. exists 쿼리는 특정 필드에 값이 존재하는 문서만 찾습니다. 데이터베이스에서 "IS NOT NULL" 조건과 비슷한 역할입니다.

실제 데이터에서는 모든 문서가 같은 필드를 가지고 있지 않은 경우가 흔합니다. 예를 들어 사용자 프로필 인덱스에서 일부 사용자만 전화번호를 등록했다면, phone 필드가 있는 문서와 없는 문서가 섞여 있습니다. exists 쿼리는 이런 상황에서 "phone 필드가 존재하는 문서만 찾아라"라는 조건을 표현합니다.

```json
{
  "query": {
    "exists": {
      "field": "phone"
    }
  }
}
```

"exists" 안에 "field"로 대상 필드 이름을 지정합니다. phone 필드에 값이 있는 문서만 결과에 포함됩니다. 필드가 아예 없거나, 필드의 값이 null이거나, 빈 배열인 문서는 제외됩니다.

반대로 "필드가 없는 문서"를 찾고 싶다면 어떻게 할까요. Elasticsearch에는 "not exists" 같은 별도 쿼리가 없습니다. 대신 3.2.3에서 다룬 bool 쿼리의 must_not 절에 exists 쿼리를 넣으면 됩니다.

```json
{
  "query": {
    "bool": {
      "must_not": {
        "exists": {
          "field": "phone"
        }
      }
    }
  }
}
```

must_not 절은 조건에 매칭되는 문서를 결과에서 제외합니다. must_not 안에 exists 쿼리를 넣으면, phone 필드가 존재하는 문서가 제외되고, 결과적으로 phone 필드가 없는 문서만 남습니다.

이제 **prefix 쿼리**를 살펴봅니다. prefix 쿼리는 필드 값이 지정한 문자열로 시작하는 문서를 찾습니다. 예를 들어 사용자 이름이 "kim"으로 시작하는 문서를 찾고 싶을 때 사용합니다.

```json
{
  "query": {
    "prefix": {
      "username": {
        "value": "kim"
      }
    }
  }
}
```

"prefix" 안에 대상 필드를 지정하고, "value"에 접두사 문자열을 넣습니다. username 필드의 값이 "kim"으로 시작하는 모든 문서가 매칭됩니다. "kim", "kimchi", "kimura" 등이 결과에 포함됩니다.

prefix 쿼리는 용어 수준 쿼리이므로 입력값을 분석하지 않습니다. 따라서 keyword 필드에 사용하는 것이 적합합니다. text 필드에 사용하면 분석기가 생성한 토큰과 접두사를 비교하게 되는데, 원본 문자열과 토큰이 다를 수 있어 예상과 다른 결과가 나올 수 있습니다.

prefix 쿼리의 성능은 인덱스 구조에 따라 달라집니다. Elasticsearch의 역색인(inverted index)은 토큰을 정렬된 상태로 저장하므로, 특정 접두사로 시작하는 토큰을 찾는 것은 비교적 효율적입니다. 다만, 접두사가 매우 짧으면(예: 한 글자) 매칭되는 토큰 수가 많아져 성능이 저하될 수 있습니다. 매핑 단계에서 index_prefixes 옵션을 설정하면 모든 토큰의 접두사를 미리 색인하여 prefix 쿼리 실행 속도를 크게 높일 수 있습니다.

다음은 **wildcard 쿼리**입니다. wildcard 쿼리는 와일드카드 패턴을 사용하여 필드 값을 매칭합니다. 두 가지 와일드카드 문자를 사용할 수 있습니다. **별표(*)** 는 0개 이상의 임의 문자를 뜻합니다. **물음표(?)**는 정확히 1개의 임의 문자를 뜻합니다.

예를 들어 "elast*"는 "elastic", "elasticsearch", "elasticity" 등 "elast"로 시작하는 모든 문자열과 매칭됩니다. "te?t"는 "test", "text" 등 가운데 한 글자만 다른 네 글자 문자열과 매칭됩니다.

```json
{
  "query": {
    "wildcard": {
      "product_name": {
        "value": "elast*"
      }
    }
  }
}
```

"wildcard" 안에 대상 필드를 지정하고 "value"에 와일드카드 패턴을 넣습니다. 이 예시에서는 product_name 필드 값이 "elast"로 시작하는 모든 문서가 매칭됩니다.

wildcard 쿼리를 사용할 때 반드시 알아야 할 성능 주의사항이 있습니다. 패턴의 맨 앞에 별표를 넣는 것을 **선행 와일드카드(leading wildcard)**라고 합니다. "*search"처럼 패턴이 별표로 시작하면, Elasticsearch는 역색인의 모든 토큰을 하나씩 대조해야 합니다. 역색인은 토큰을 알파벳 순서로 정렬하여 저장하므로, 접두사가 정해져 있으면 해당 범위만 탐색할 수 있습니다. 그런데 선행 와일드카드는 접두사 정보가 없으므로, 인덱스의 전체 토큰 목록을 처음부터 끝까지 훑어야 합니다.

이 과정을 단계별로 보면 다음과 같습니다.

```
선행 와일드카드의 탐색 과정

패턴: *search

1. 역색인에서 첫 번째 토큰을 읽는다
2. 토큰이 "search"로 끝나는지 확인한다
3. 다음 토큰으로 넘어간다
4. 2~3을 마지막 토큰까지 반복한다

접두사가 없으므로 건너뛸 구간이 없다
--> 전체 토큰을 순회해야 한다 (성능 저하)
```

반면 "elast*"처럼 접두사가 있는 패턴은 역색인에서 "elast"로 시작하는 구간만 찾으면 되므로, 탐색 범위가 좁아집니다. 따라서 wildcard 쿼리에서는 선행 와일드카드를 가급적 피해야 합니다. 데이터 규모가 클수록 성능 차이가 심해집니다.

**regexp 쿼리**는 정규 표현식 패턴으로 필드 값을 매칭합니다. wildcard 쿼리보다 더 정교한 패턴 표현이 가능합니다. 예를 들어 "log-[0-9]{4}"라는 패턴은 "log-" 뒤에 숫자 네 자리가 오는 문자열과 매칭됩니다. "log-2024", "log-0001" 등이 해당됩니다.

```json
{
  "query": {
    "regexp": {
      "filename": {
        "value": "log-[0-9]{4}"
      }
    }
  }
}
```

"regexp" 안에 대상 필드를 지정하고 "value"에 정규 표현식 패턴을 넣습니다. Elasticsearch는 Lucene의 정규 표현식 엔진을 사용하므로, 지원하는 문법이 일반적인 정규 표현식과 약간 다를 수 있습니다. 예를 들어 \d(숫자) 같은 축약 표현은 지원하지 않으며, [0-9]처럼 명시적으로 써야 합니다.

regexp 쿼리의 성능 주의사항은 wildcard 쿼리와 동일합니다. 패턴이 고정된 접두사 없이 시작하면, 역색인의 전체 토큰을 순회해야 하므로 성능이 크게 저하됩니다. ".*search"처럼 패턴 앞에 임의 문자 매칭을 두는 것은 wildcard 쿼리의 선행 와일드카드와 같은 문제를 일으킵니다. regexp 쿼리는 패턴이 복잡할수록 평가 비용도 높아지므로, 단순한 접두사 매칭이 목적이라면 prefix 쿼리를, 단순한 와일드카드 패턴이면 wildcard 쿼리를 먼저 고려하는 것이 바람직합니다.

마지막으로 **ids 쿼리**를 살펴봅니다. ids 쿼리는 문서의 _id 값으로 특정 문서를 직접 찾습니다. Elasticsearch에서 모든 문서는 고유한 _id를 갖습니다. ids 쿼리에 _id 값 목록을 전달하면, 해당 _id를 가진 문서만 결과에 포함됩니다.

```json
{
  "query": {
    "ids": {
      "values": ["doc_001", "doc_002", "doc_003"]
    }
  }
}
```

"ids" 안에 "values" 배열로 찾고자 하는 _id 값을 나열합니다. _id를 이미 알고 있는 상황에서 해당 문서를 검색 결과 형태로 가져올 때 유용합니다. 관련성 점수 계산이 필요 없는 경우가 대부분이므로, bool 쿼리의 filter 절에 넣어 사용하는 것이 적합합니다.

이 단원에서 다룬 쿼리들의 특성을 한눈에 비교하면 다음과 같습니다.

```
용어 수준 쿼리 비교

쿼리       용도                        성능 주의
---------  --------------------------  ---------------------------
range      숫자/날짜 범위 필터링        filter 절 사용 권장
exists     필드 존재 여부 확인          -
prefix     접두사 매칭                  짧은 접두사 시 성능 저하 가능
wildcard   * ? 패턴 매칭               선행 와일드카드(*abc) 회피
regexp     정규 표현식 매칭             고정 접두사 없는 패턴 회피
ids        문서 _id로 직접 조회         -
```

정리하면, range 쿼리는 gte, lte, gt, lt 파라미터로 숫자와 날짜의 범위를 지정하며, 날짜 필드에서는 now-7d/d 같은 날짜 연산으로 상대 범위를 표현할 수 있습니다. exists 쿼리는 필드의 존재 여부를 확인하고, prefix 쿼리는 접두사 매칭을, wildcard 쿼리는 별표와 물음표를 사용한 패턴 매칭을, regexp 쿼리는 정규 표현식 매칭을 수행합니다. wildcard와 regexp 쿼리에서는 선행 와일드카드나 고정 접두사 없는 패턴이 전체 토큰 순회를 유발하여 성능을 크게 떨어뜨릴 수 있으므로, 패턴 설계에 주의해야 합니다. ids 쿼리는 문서의 _id 값으로 직접 문서를 찾습니다. 이 쿼리들은 모두 용어 수준 쿼리에 속하며, 관련성 점수가 필요 없는 필터링 용도로는 bool 쿼리의 filter 절에 넣어 캐시 이점을 얻는 것이 일반적입니다.

다음 단원인 3.3.1에서는 multi_match와 cross-field 검색을 다룹니다.

이 단원을 마치면 range 쿼리로 날짜와 숫자의 범위 검색을 수행할 수 있고, prefix, wildcard, regexp 쿼리의 성능 위험을 인식하여 적절한 상황에서 사용할 수 있습니다.
