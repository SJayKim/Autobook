# 1.3.2 인덱스 개념과 RDBMS 비교

1.3.1에서 Elasticsearch의 문서가 JSON 형식이며, _index 메타데이터 필드가 문서가 속한 인덱스의 이름을 가리킨다는 것을 살펴보았습니다. 문서를 인덱싱할 때 PUT /my-index/_doc/1처럼 URL에 인덱스 이름을 포함시켰고, 응답에서도 _index 값이 돌아왔습니다. 그런데 이 '인덱스'라는 단어가 정확히 무엇을 뜻하는지, 관계형 데이터베이스(RDBMS)에서 말하는 인덱스와 어떻게 다른지는 아직 다루지 않았습니다. 이 단원에서는 Elasticsearch 인덱스의 정체를 밝히고, RDBMS의 테이블과 비교하여 두 세계의 차이를 짚은 뒤, 인덱스를 만들고 조회하고 삭제하는 실제 API까지 살펴봅니다.

'인덱스'라는 용어는 데이터베이스 분야에서 오랫동안 써 왔습니다. RDBMS를 다뤄 본 적이 있다면, 인덱스라고 하면 테이블의 검색 속도를 높이기 위해 별도로 만드는 보조 자료구조를 떠올릴 것입니다. B-tree나 해시 구조로 특정 열의 값을 빠르게 찾아가는 역할을 하는, 테이블 자체와는 별개인 부속물입니다. 그런데 Elasticsearch에서 인덱스는 전혀 다른 의미를 가집니다. Elasticsearch의 인덱스는 보조 자료구조가 아니라, 데이터를 담는 1차 저장 컨테이너 그 자체입니다. 같은 단어가 완전히 다른 뜻으로 쓰이므로, 처음 접할 때 혼란이 생기기 쉽습니다. 이 차이를 먼저 인식해 두는 것이 중요합니다.

Elasticsearch에서 **인덱스**(index)는 유사한 구조나 목적을 가진 JSON 문서들을 모아 놓는 논리적 컨테이너입니다. '논리적'이라는 표현을 쓰는 이유가 있습니다. 인덱스는 이름으로 식별되는 하나의 개념적 묶음이지만, 물리적으로는 내부에서 여러 개의 샤드(shard)로 분할되어 클러스터의 여러 노드에 분산 저장됩니다. 사용자가 인덱스 이름을 지정해 문서를 저장하거나 검색하면, Elasticsearch가 내부적으로 어떤 샤드에 접근해야 하는지를 알아서 처리합니다. 사용자 입장에서는 인덱스라는 단일한 이름만 알면 됩니다.

한 가지 알아 두어야 할 역사적 변화가 있습니다. Elasticsearch 7.0 이전에는 하나의 인덱스 안에 여러 **타입**(type)을 둘 수 있었습니다. 타입은 같은 인덱스 안에서 문서의 종류를 구분하는 개념이었습니다. 그러나 7.0부터 타입이 폐기되었습니다. 현재는 인덱스 하나에 문서 종류 하나만 대응합니다. 따라서 서로 다른 구조의 문서를 저장하려면 별도의 인덱스를 만들어야 합니다.

이 지점에서 RDBMS와 비교하면 인덱스가 무엇인지 더 선명해집니다. 아래 표는 Elasticsearch와 RDBMS의 개념을 대응시킨 것입니다.

```
Elasticsearch          관계형 데이터베이스(RDBMS)
--------------------   ----------------------------
인덱스(Index)          테이블(Table)
문서(Document)         행(Row)
필드(Field)            열(Column)
매핑(Mapping)          스키마(Schema)
```

1.3.1에서 문서가 RDBMS의 행에 해당한다는 것을 이미 확인했습니다. 인덱스는 그 행들을 담는 테이블에 대응합니다. 매핑은 각 필드의 타입과 처리 방식을 정의하는 구조이므로, RDBMS에서 테이블을 만들 때 열의 이름과 타입을 정의하는 스키마에 해당합니다.

다만 이 대응은 완전하지 않습니다. RDBMS 테이블과 Elasticsearch 인덱스 사이에는 근본적인 차이가 세 가지 있습니다.

첫 번째 차이는 데이터 구조의 방향입니다. RDBMS는 **정규화**(normalization)를 기본 원칙으로 삼습니다. 정규화란 데이터 중복을 제거하기 위해 관련 정보를 여러 테이블에 분산시키고, 필요할 때 JOIN 연산으로 합쳐서 조회하는 설계 방식입니다. 반면 Elasticsearch에는 JOIN 연산이 없습니다. 그래서 반대로 관련 정보를 하나의 JSON 문서 안에 모두 포함하는 **비정규화**(denormalization)가 일반적인 관행입니다. 예를 들어 '주문' 문서 안에 '고객 이름'과 '상품 정보'를 함께 넣는 식입니다.

두 번째 차이는 스키마 유연성입니다. RDBMS에서는 테이블을 만들 때 열의 이름과 타입을 미리 엄격하게 정의해야 합니다. 정의하지 않은 열에 데이터를 넣으려면 ALTER TABLE로 스키마를 먼저 변경해야 합니다. Elasticsearch는 사전 스키마 정의 없이도 문서를 넣을 수 있습니다. 인덱스에 처음 보는 필드가 들어오면 Elasticsearch가 값을 보고 타입을 자동으로 감지하여 매핑에 추가합니다. 이것을 **동적 매핑**(dynamic mapping)이라고 합니다. 빠른 시작이 가능하지만, 의도와 다른 타입이 할당될 위험도 있습니다. 이 부분은 뒤에서 매핑을 다시 살펴볼 때 조금 더 설명합니다.

세 번째 차이는 앞서 언급한 '인덱스'라는 용어 자체의 의미입니다. RDBMS에서 인덱스는 테이블에 부속된 검색 최적화용 보조 자료구조입니다. Elasticsearch에서 인덱스는 데이터의 1차 저장 컨테이너입니다. 같은 단어이지만 가리키는 대상이 다르므로, 두 세계를 오가며 작업할 때 항상 의식해야 합니다.

인덱스를 생성하면 Elasticsearch는 그 인덱스에 대해 두 가지 핵심 설정을 부여합니다. 하나는 **settings**(설정)이고, 다른 하나는 **mappings**(매핑)입니다. settings는 인덱스의 물리적 동작 방식을 제어하고, mappings는 문서의 논리적 구조를 정의합니다. 두 역할은 명확히 분리되어 있습니다.

먼저 settings를 살펴봅니다. settings는 정적(static) 설정과 동적(dynamic) 설정으로 나뉩니다. 정적 설정은 인덱스를 만들 때 한 번 지정하면 이후 변경할 수 없는 값이고, 동적 설정은 인덱스가 이미 존재하는 상태에서도 언제든 변경할 수 있는 값입니다.

대표적인 정적 설정이 **number_of_shards**입니다. 이 값은 인덱스의 데이터를 몇 개의 샤드로 분할할지를 결정합니다. Elasticsearch 7.0 이후 기본값은 1입니다. 샤드 수는 데이터 분산과 병렬 처리 성능에 직접 영향을 미칩니다. 한 번 정한 뒤에는 바꿀 수 없으므로 인덱스를 만들기 전에 신중하게 결정해야 합니다. 샤드와 분산 저장의 구체적인 메커니즘은 1.3.3에서 다룹니다.

대표적인 동적 설정 두 가지가 있습니다. 하나는 **number_of_replicas**이고, 다른 하나는 **refresh_interval**입니다.

number_of_replicas는 각 샤드의 복제본을 몇 개 유지할지를 결정합니다. 기본값은 1입니다. 복제본이 있으면 원본 샤드에 장애가 생겨도 복제본이 대신 역할을 수행할 수 있습니다. 반대로 0으로 설정하면 복제본이 없으므로 대량 데이터를 빠르게 적재할 때 성능이 향상되지만, 그 동안 장애가 발생하면 데이터를 잃을 수 있습니다.

refresh_interval은 새로 인덱싱된 문서가 검색 결과에 노출되기까지의 간격입니다. 기본값은 1초입니다. 이 간격마다 Elasticsearch는 내부적으로 새 세그먼트를 열어 아직 검색에 반영되지 않은 문서를 검색 가능한 상태로 전환합니다. "-1"로 설정하면 자동 새로고침이 꺼져 대량 적재 속도가 극대화되지만, 적재가 끝난 뒤 원래 값으로 되돌려야 합니다.

이제 mappings를 살펴봅니다. 매핑은 문서의 각 필드를 어떻게 저장하고 인덱싱할지 정의하는 스키마입니다. RDBMS의 테이블 스키마에 해당하지만, 앞서 설명한 것처럼 더 유연합니다.

매핑을 정의하는 방식은 두 가지입니다. 첫 번째는 동적 매핑입니다. 매핑을 미리 정의하지 않고 문서를 인덱싱하면, Elasticsearch가 들어오는 값을 보고 필드 타입을 자동으로 감지합니다. 빠르게 시작할 수 있지만, 숫자가 포함된 문자열을 float로 잘못 감지하는 등 의도치 않은 타입 할당이 일어날 수 있습니다.

두 번째는 **명시적 매핑**(explicit mapping)입니다. 인덱스를 만들 때 각 필드의 타입을 직접 지정하는 방식입니다. text, keyword, date, integer 등 40가지 이상의 타입을 지원합니다. 프로덕션 환경에서는 명시적 매핑이 권장됩니다.

매핑에서 주의할 점이 하나 있습니다. 이미 인덱싱된 필드의 타입은 변경할 수 없습니다. 타입을 바꾸려면 새 인덱스를 만들어 데이터를 복사하는 재인덱싱(reindexing)이 필요합니다. 반면, 새 필드를 추가하는 것은 언제든 가능합니다.

이론을 충분히 살펴보았으니, 실제로 인덱스를 생성하는 API를 봅니다. 인덱스는 PUT 요청으로 만듭니다.

```
PUT /my-index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
      "title": { "type": "text" },
      "date":  { "type": "date" }
    }
  }
}
```

이 요청은 my-index라는 이름의 인덱스를 만듭니다. settings 부분에서 샤드 수를 3, 레플리카 수를 1로 지정했습니다. mappings 부분에서는 title 필드를 text 타입으로, date 필드를 date 타입으로 명시했습니다. 생성이 성공하면 `{"acknowledged": true}` 응답이 돌아옵니다.

settings와 mappings를 생략하고 인덱스 이름만으로도 생성할 수 있습니다.

```
PUT /favorite_candy
```

이 경우 기본 설정이 자동 적용됩니다. 샤드 1개, 레플리카 1개로 만들어지고, 매핑은 나중에 문서가 들어올 때 동적 매핑으로 자동 생성됩니다.

인덱스 이름에는 규칙이 있습니다. 반드시 소문자여야 합니다. 별표(*), 슬래시(/), 역슬래시(\), 꺾쇠(<, >), 파이프(|), 공백, 쉼표(,), 샵(#) 같은 특수문자는 사용할 수 없습니다. 하이픈(-), 밑줄(_), 더하기(+)는 허용됩니다. 마침표(.)로 시작할 수 없고, 전체 길이가 255바이트 미만이어야 합니다.

인덱스를 만든 뒤 클러스터에 어떤 인덱스들이 있는지 확인하려면 _cat/indices API를 사용합니다.

```
GET /_cat/indices?v
```

이 요청은 클러스터 내 모든 인덱스의 요약 정보를 사람이 읽기 쉬운 텍스트 형식으로 반환합니다. JSON이 아니라 탭으로 구분된 텍스트입니다. ?v 파라미터를 붙이면 컬럼 헤더가 함께 출력되어 각 열이 무엇을 의미하는지 알 수 있습니다. 출력에 포함되는 주요 정보는 다음과 같습니다.

```
health   인덱스 상태 (green, yellow, red)
index    인덱스 이름
uuid     인덱스 고유 식별자
pri      프라이머리 샤드 수
rep      레플리카 수
docs.count    문서 수
store.size    인덱스 전체 크기 (레플리카 포함)
pri.store.size 프라이머리 샤드 크기
```

_cat/indices는 개발이나 운영 중 빠르게 현황을 파악하는 용도로 적합합니다. 프로그래밍 방식으로 인덱스 정보를 조회해야 한다면 JSON 형식으로 응답하는 별도의 API를 사용하는 것이 낫습니다.

마지막으로 인덱스 삭제를 다룹니다. 인덱스를 삭제하는 API는 DELETE 요청입니다.

```
DELETE /my-index
```

이 요청은 my-index 인덱스와 그 안에 저장된 모든 문서를 영구적으로 제거합니다. 1.3.1에서 다룬 문서 단위의 DELETE(`DELETE /my-index/_doc/1`)는 특정 문서 하나만 삭제하는 것이었지만, 여기서는 인덱스 전체가 사라집니다.

인덱스 삭제에는 세 가지 주의사항이 있습니다.

첫째, 삭제된 인덱스는 복구할 수 없습니다. 사전에 스냅샷(snapshot)을 만들어 두지 않았다면, 삭제 후 되돌릴 방법이 없습니다. 운영 환경에서는 삭제 전에 반드시 스냅샷이 존재하는지 확인해야 합니다.

둘째, 와일드카드 패턴으로 여러 인덱스를 한꺼번에 삭제할 수 있습니다. 예를 들어 `DELETE /logs-*`는 이름이 logs-로 시작하는 모든 인덱스를 삭제합니다. 실수로 의도하지 않은 인덱스까지 삭제할 위험이 있으므로, `action.destructive_requires_name: true` 설정을 켜면 와일드카드 삭제를 차단할 수 있습니다.

셋째, 매핑을 변경해야 하는 상황에서 인덱스를 삭제하는 대신 **별칭**(alias)을 활용하는 패턴이 있습니다. 절차는 다음과 같습니다.

```
별칭을 활용한 무중단 재인덱싱

1. 새 인덱스를 생성한다 (변경된 매핑 적용).
2. Reindex API로 구 인덱스의 데이터를 새 인덱스에 복사한다.
3. 별칭을 구 인덱스에서 새 인덱스로 교체한다.
4. 구 인덱스를 삭제한다.
```

이 순서를 따르면 별칭을 참조하는 서비스는 중단 없이 계속 동작하면서도 매핑 변경이 가능합니다.

정리하면, Elasticsearch 인덱스는 JSON 문서들의 논리적 컨테이너이며, RDBMS의 테이블에 대응하지만 비정규화 선호, 유연한 스키마, 그리고 '인덱스'라는 용어 자체의 의미라는 세 가지 지점에서 근본적으로 다릅니다. 인덱스의 물리적 동작은 settings로, 논리적 구조는 mappings로 제어하며, PUT API로 생성하고 _cat/indices로 조회하고 DELETE API로 삭제합니다. 삭제는 복구가 불가능하므로 스냅샷과 별칭 패턴을 함께 이해해 두는 것이 중요합니다.

다음 단원인 1.3.3에서는 샤드와 레플리카를 다룹니다.

이 단원을 마치면 Elasticsearch 인덱스를 RDBMS 테이블과 비교하여 차이점을 설명할 수 있고, 인덱스 설정(settings)과 매핑(mappings)의 역할을 구분할 수 있습니다.
