# 1.1.3 첫 번째 문서 인덱싱과 검색

1.1.2에서 Elasticsearch를 설치하고, 클러스터가 정상 동작하는지 _cluster/health API로 확인하는 것까지 마쳤습니다. 실행 환경은 준비되었지만, 아직 데이터를 넣거나 찾는 작업은 해 보지 않았습니다. 검색 엔진은 데이터가 들어 있어야 의미가 있습니다. 이 단원에서는 문서 한 건을 직접 저장하고, 저장한 문서를 조회하고, 여러 문서 중에서 원하는 것을 검색하는 과정을 차례로 다룹니다. 1.1.1에서 소개한 RESTful API를 실제로 사용하는 첫 번째 실습이기도 합니다.

Elasticsearch에서 데이터를 저장하는 행위를 **인덱싱**(indexing)이라 합니다. 관계형 데이터베이스에서 INSERT 문으로 행을 추가하는 것과 비슷하지만, Elasticsearch는 저장과 동시에 역색인을 구성하여 검색 가능한 상태로 만든다는 점이 다릅니다. 1.1.1에서 역색인이 텍스트를 토큰으로 쪼개어 "어떤 단어가 어떤 문서에 있는지"를 미리 기록해 둔다고 설명했습니다. 인덱싱이란 바로 이 기록을 생성하는 과정입니다.

문서를 인덱싱할 때는 HTTP PUT 메서드를 사용합니다. 요청 URL은 '/{인덱스명}/_doc/{문서ID}' 형태이고, 요청 본문에 저장할 JSON 문서를 담습니다. 아래 예시는 'my-index'라는 인덱스에 ID가 1인 문서를 저장하는 요청입니다.

```
PUT /my-index/_doc/1
{
  "title": "Elasticsearch 입문",
  "content": "Elasticsearch는 분산 검색 엔진입니다.",
  "author": "홍길동",
  "created_at": "2026-04-01"
}
```

이 요청의 각 부분을 풀어 보겠습니다. 'PUT'은 HTTP 메서드로, "이 위치에 자원을 저장하라"는 의미입니다. '/my-index'는 문서를 저장할 인덱스의 이름입니다. '/_doc'은 문서(document) 타입 엔드포인트를 뜻합니다. '/1'은 이 문서에 부여할 고유 ID입니다. 본문의 JSON 객체에는 'title', 'content', 'author', 'created_at' 네 개의 필드가 담겨 있습니다. Elasticsearch는 이 JSON을 그대로 저장하면서, 동시에 각 텍스트 필드의 내용을 분석하여 역색인을 만듭니다.

이 요청을 보내면 Elasticsearch는 다음과 같은 응답을 돌려줍니다.

```json
{
  "_index": "my-index",
  "_id": "1",
  "_version": 1,
  "result": "created",
  "_shards": {
    "total": 2,
    "successful": 1,
    "failed": 0
  },
  "_seq_no": 0,
  "_primary_term": 1
}
```

응답의 각 필드를 살펴봅니다. '_index'는 문서가 저장된 인덱스 이름입니다. '_id'는 문서의 고유 식별자입니다. '_version'은 이 문서가 몇 번째로 기록되었는지를 나타내는 버전 번호입니다. 처음 생성하면 1이고, 같은 ID로 다시 PUT 요청을 보내면 2, 3으로 증가합니다. 'result'는 이번 요청의 결과를 나타내며, 새로 생성되었으면 'created', 기존 문서를 덮어썼으면 'updated'가 됩니다. '_shards'는 이 쓰기 작업이 몇 개의 샤드에 기록되었는지를 보여줍니다.

여기서 주의할 점이 있습니다. 이미 존재하는 ID로 PUT 요청을 보내면 기존 문서가 통째로 교체됩니다. 일부 필드만 수정하는 것이 아니라, 요청 본문의 JSON이 기존 문서를 완전히 대체합니다. 예를 들어 위 예시에서 ID 1로 다시 PUT을 보내되 'author' 필드를 빠뜨리면, 저장된 문서에서 'author'가 사라집니다.

이 예시에서 'my-index'라는 인덱스를 미리 만들지 않았다는 점을 눈여겨봅니다. Elasticsearch는 존재하지 않는 인덱스에 문서를 인덱싱하면, 해당 인덱스를 **자동으로 생성**합니다. 이 동작을 **인덱스 자동 생성**이라 합니다. 인덱스가 자동으로 만들어질 때, 각 필드의 데이터 타입도 함께 추론됩니다. 1.1.1에서 이를 동적 매핑(dynamic mapping)이라 했습니다. 예를 들어 'title' 필드에 문자열이 들어오면 text 타입과 keyword 타입을 동시에 갖는 멀티필드로 매핑되고, 'created_at'에 날짜 형식의 문자열이 들어오면 date 타입으로 추론됩니다.

인덱스 자동 생성은 편리하지만, 운영 환경에서는 의도치 않은 인덱스가 만들어질 위험이 있습니다. 이 동작은 'action.auto_create_index' 설정으로 제어합니다. 기본값은 true(허용)이며, false로 변경하면 사전에 인덱스를 명시적으로 생성해야만 문서를 인덱싱할 수 있습니다.

문서에 고유 식별자를 직접 부여하지 않고 Elasticsearch가 알아서 ID를 만들게 할 수도 있습니다. 이때는 PUT 대신 POST 메서드를 사용하고, URL에서 ID 부분을 생략합니다.

```
POST /my-index/_doc
{
  "title": "Elasticsearch 실전",
  "content": "검색 성능을 최적화하는 방법을 다룹니다."
}
```

PUT과 달리 URL 끝에 '/1' 같은 ID가 없습니다. Elasticsearch는 이 요청을 받으면 UUID 형태의 고유 ID를 자동으로 생성합니다. 응답의 '_id' 필드에서 생성된 ID를 확인할 수 있습니다.

```json
{
  "_index": "my-index",
  "_id": "aWx3c4oBkQHLfnY9_abc",
  "_version": 1,
  "result": "created"
}
```

'_id' 값이 'aWx3c4oBkQHLfnY9_abc'처럼 긴 문자열로 나타납니다. 이 방식은 외부 시스템에서 부여하는 식별자(주문 번호, 회원 ID 등)가 없고, Elasticsearch 내부에서만 문서를 관리하는 경우에 적합합니다. PUT은 같은 ID로 재요청하면 덮어쓰기가 발생하지만, POST로 자동 ID를 생성하면 매번 새로운 문서가 만들어지므로 ID 충돌이 일어나지 않습니다.

두 방식을 정리하면 다음과 같습니다.

```
인덱싱 방식 비교

PUT  /my-index/_doc/1    → ID를 직접 지정, 동일 ID면 덮어쓰기
POST /my-index/_doc      → ID를 자동 생성, 항상 새 문서 생성
```

이제 저장한 문서를 꺼내 보겠습니다. 특정 문서를 조회할 때는 HTTP GET 메서드를 사용합니다. URL 형태는 인덱싱과 동일하게 '/{인덱스명}/_doc/{문서ID}'입니다.

```
GET /my-index/_doc/1
```

이 요청은 'my-index' 인덱스에서 ID가 1인 문서를 가져옵니다. 응답은 다음과 같습니다.

```json
{
  "_index": "my-index",
  "_id": "1",
  "_version": 1,
  "_seq_no": 0,
  "_primary_term": 1,
  "found": true,
  "_source": {
    "title": "Elasticsearch 입문",
    "content": "Elasticsearch는 분산 검색 엔진입니다.",
    "author": "홍길동",
    "created_at": "2026-04-01"
  }
}
```

'found' 필드가 true이면 해당 ID의 문서가 존재한다는 뜻입니다. 존재하지 않는 ID로 요청하면 'found'가 false로 바뀌고, HTTP 상태 코드도 404 Not Found가 됩니다.

응답에서 가장 중요한 부분은 **_source** 필드입니다. _source는 인덱싱할 때 전달한 원본 JSON 문서를 그대로 보관하는 특수 메타데이터 필드입니다. 1.1.1에서 문서의 메타데이터 필드 중 하나로 _source를 소개한 바 있습니다. Elasticsearch는 문서를 인덱싱할 때 두 가지 작업을 동시에 수행합니다. 하나는 역색인을 구성하여 검색 가능하게 만드는 것이고, 다른 하나는 원본 JSON을 _source 필드에 저장하는 것입니다. 역색인은 검색 속도를 위한 구조이므로 원본 텍스트를 그대로 복원할 수 없지만, _source에 원본이 보관되어 있으므로 조회나 검색 결과에서 문서 내용을 온전히 돌려받을 수 있습니다.

_source 외에도 응답에는 여러 메타데이터 필드가 함께 반환됩니다. '_index'는 문서가 속한 인덱스, '_id'는 문서 고유 식별자, '_version'은 문서 버전입니다. '_seq_no'와 '_primary_term'은 동시성 제어에 사용되는 값인데, 여러 클라이언트가 같은 문서를 동시에 수정할 때 충돌을 감지하는 용도입니다. 이 단원의 범위에서는 이 두 값을 깊이 다루지 않습니다.

지금까지는 ID를 알고 있는 문서를 직접 집어서 가져오는 방법이었습니다. 하지만 실제 사용에서는 ID를 모른 채 내용으로 문서를 찾는 경우가 훨씬 많습니다. 이때 사용하는 것이 **_search API**입니다.

검색을 실행하려면 '/{인덱스명}/_search' 엔드포인트에 요청을 보냅니다. HTTP 메서드는 GET과 POST 모두 사용할 수 있지만, 요청 본문에 검색 조건을 담아야 하므로 POST를 쓰는 것이 관행입니다. 아래 예시는 'my-index'에서 'content' 필드에 '검색'이라는 단어가 포함된 문서를 찾는 요청입니다.

```
POST /my-index/_search
{
  "query": {
    "match": {
      "content": "검색"
    }
  }
}
```

이 요청의 구조를 풀어 봅니다. 최상위에 'query' 키가 있고, 그 안에 쿼리의 종류를 지정합니다. 여기서는 **match** 쿼리를 사용했습니다. match 쿼리는 Elasticsearch에서 전문 검색(full-text search)을 수행하는 가장 기본적인 쿼리입니다. match 안에는 검색 대상 필드('content')와 검색어('검색')를 지정합니다.

match 쿼리의 동작 과정을 단계별로 살펴봅니다. 먼저 검색어 '검색'이 해당 필드에 설정된 분석기(analyzer)를 거칩니다. 분석기는 텍스트를 토큰 단위로 분해하는 처리기입니다. 1.1.1에서 역색인이 텍스트를 토큰으로 쪼갠다고 했는데, 이 쪼개는 역할을 분석기가 수행합니다. 인덱싱 시점에 문서의 텍스트를 분석기로 토큰화하여 역색인에 기록하고, 검색 시점에도 검색어를 같은 분석기로 토큰화합니다. 양쪽 토큰이 일치하는 문서가 검색 결과로 반환됩니다.

검색어가 여러 단어로 구성된 경우를 봅니다. 예를 들어 match 쿼리에 '분산 검색'을 넣으면, 분석기가 이를 '분산'과 '검색' 두 토큰으로 분해합니다. 기본적으로 match 쿼리는 OR 연산을 적용합니다. 즉, '분산' 또는 '검색' 중 하나라도 포함된 문서가 모두 반환됩니다. 두 단어를 모두 포함한 문서만 찾으려면 'operator'를 'and'로 지정합니다.

```
POST /my-index/_search
{
  "query": {
    "match": {
      "content": {
        "query": "분산 검색",
        "operator": "and"
      }
    }
  }
}
```

이 경우 '분산'과 '검색'이 모두 포함된 문서만 결과에 나타납니다.

match 쿼리가 반환하는 문서는 단순히 "포함/미포함"으로 나뉘는 것이 아닙니다. 각 문서에 관련성 점수가 매겨지고, 점수가 높은 순서대로 정렬됩니다. 이 점수 계산에 사용되는 알고리즘이 **BM25**입니다. BM25는 검색어가 문서에 자주 등장할수록, 그리고 전체 문서에서는 드물게 등장하는 단어일수록 높은 점수를 부여합니다. 예를 들어 '검색'이라는 단어가 특정 문서에 여러 번 나타나고, 다른 문서에서는 드물게 나타난다면 그 문서의 점수가 높아집니다.

이제 검색 응답의 구조를 살펴봅니다. _search API는 항상 동일한 형태의 JSON을 반환합니다.

```json
{
  "took": 5,
  "timed_out": false,
  "_shards": {
    "total": 1,
    "successful": 1,
    "skipped": 0,
    "failed": 0
  },
  "hits": {
    "total": {
      "value": 2,
      "relation": "eq"
    },
    "max_score": 1.4712,
    "hits": [
      {
        "_index": "my-index",
        "_id": "1",
        "_score": 1.4712,
        "_source": {
          "title": "Elasticsearch 입문",
          "content": "Elasticsearch는 분산 검색 엔진입니다.",
          "author": "홍길동",
          "created_at": "2026-04-01"
        }
      },
      {
        "_index": "my-index",
        "_id": "aWx3c4oBkQHLfnY9_abc",
        "_score": 0.8319,
        "_source": {
          "title": "Elasticsearch 실전",
          "content": "검색 성능을 최적화하는 방법을 다룹니다."
        }
      }
    ]
  }
}
```

응답의 최상위 필드부터 하나씩 설명합니다.

**took**는 Elasticsearch가 이 검색을 처리하는 데 걸린 시간입니다. 단위는 밀리초(ms)입니다. 위 예시에서 5는 5밀리초, 즉 0.005초가 걸렸다는 뜻입니다. 이 값에는 클라이언트와 Elasticsearch 사이의 네트워크 왕복 시간은 포함되지 않습니다. 순수하게 Elasticsearch 내부에서 쿼리를 실행하고 결과를 조합하는 데 소요된 시간만 측정합니다.

**timed_out**은 검색이 제한 시간 안에 완료되었는지를 나타냅니다. false면 정상 완료, true면 시간 초과가 발생한 것입니다.

**_shards**는 이 검색에 참여한 샤드의 현황입니다. 'total'은 검색 대상 샤드 수, 'successful'은 정상 응답한 샤드 수, 'skipped'는 건너뛴 샤드 수, 'failed'는 실패한 샤드 수입니다.

**hits**는 검색 결과의 핵심입니다. 이 안에 세 가지 정보가 담겨 있습니다.

'hits.total'은 검색 조건에 매칭된 문서의 총 개수입니다. 'value'가 실제 숫자이고, 'relation'은 이 숫자의 의미를 나타냅니다. 'eq'는 정확한 수(equal)라는 뜻입니다. 매칭 문서가 10,000건을 넘으면 'relation'이 'gte'(greater than or equal)로 바뀌고, 'value'는 10000으로 고정됩니다. 이는 성능을 위한 설계입니다. 정확한 총 수가 필요하면 별도의 파라미터로 요청해야 합니다.

'hits.max_score'는 반환된 문서 중 가장 높은 관련성 점수입니다.

'hits.hits'는 실제로 반환된 문서의 배열입니다. 기본적으로 관련성 점수(_score)가 높은 순서대로 최대 10건이 반환됩니다. 이 기본 반환 건수는 요청에 'size' 파라미터를 추가하여 변경할 수 있습니다. 배열의 각 요소에는 '_index', '_id', '_score', '_source'가 포함됩니다.

**_score**는 각 문서의 관련성 점수입니다. 부동소수점 숫자로 표현되며, 높을수록 검색어와의 관련성이 높다는 뜻입니다. 위 예시에서 ID 1인 문서의 점수가 1.4712이고, 자동 생성 ID 문서의 점수가 0.8319입니다. _score가 기본 정렬 기준이므로, 점수가 높은 문서가 배열의 앞쪽에 위치합니다.

각 문서의 '_source' 필드에는 인덱싱 시 저장한 원본 JSON이 그대로 들어 있습니다. 앞서 GET으로 단건 조회할 때 본 _source와 같은 내용입니다.

검색 요청부터 응답까지의 전체 흐름을 정리합니다.

```
검색 처리 흐름

1. 클라이언트 → POST /my-index/_search (검색어 전송)
2. Elasticsearch → 검색어를 분석기로 토큰화
3. 토큰을 역색인에서 조회 → 매칭 문서 목록 획득
4. BM25로 각 문서의 관련성 점수(_score) 계산
5. 점수 순 정렬 → 상위 10건(기본) 선택
6. 각 문서의 _source에서 원본 JSON 추출
7. 응답 JSON 조립 (took, hits.total, hits.hits, _score 포함)
8. 클라이언트 ← 응답 반환
```

정리하면, Elasticsearch에서 문서를 다루는 기본 흐름은 PUT으로 인덱싱하고, GET으로 조회하고, POST _search로 검색하는 세 단계입니다. PUT은 지정한 ID에 문서를 저장하고, POST는 ID를 자동 생성합니다. 존재하지 않는 인덱스에 문서를 넣으면 인덱스가 자동으로 만들어집니다. match 쿼리는 검색어를 분석기로 토큰화한 뒤 역색인과 대조하여 매칭 문서를 찾고, BM25 알고리즘으로 관련성 점수를 매깁니다. 응답의 took은 처리 시간, hits.total은 매칭 문서 수, hits.hits는 실제 반환 문서 배열, _score는 관련성 점수를 나타냅니다.

다음 단원인 1.2.1에서는 Apache Lucene과 Elasticsearch의 관계를 다룹니다.

이 단원을 마치면 PUT/GET API로 문서를 인덱싱하고 조회할 수 있고, _search API로 기본 전문 검색을 실행할 수 있으며, 응답 JSON의 hits와 _score 필드를 해석할 수 있습니다.
