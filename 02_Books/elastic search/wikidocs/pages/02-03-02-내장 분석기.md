# 2.3.2 내장 분석기

2.3.1에서 분석기가 캐릭터 필터, 토크나이저, 토큰 필터라는 세 구성요소로 이루어진 파이프라인이라는 것을 살펴보았습니다. 그때는 각 구성요소의 역할과 연결 순서를 확인했고, _analyze API로 분석 결과를 눈으로 볼 수 있다는 것도 알았습니다. 그런데 실무에서 인덱스를 만들 때마다 캐릭터 필터, 토크나이저, 토큰 필터를 직접 조합해야 한다면 번거로울 것입니다. Elasticsearch는 자주 쓰이는 조합을 미리 묶어 둔 **내장 분석기**(built-in analyzer)를 제공합니다. 내장 분석기는 이름 하나만 지정하면 바로 사용할 수 있으므로, 대부분의 상황에서는 내장 분석기 가운데 적합한 것을 골라 쓰는 것만으로 충분합니다. 이 단원에서는 Elasticsearch가 기본 제공하는 내장 분석기를 하나씩 살펴보고, 각각이 어떤 상황에 적합한지 알아봅니다.

가장 먼저 알아야 할 것은 **standard 분석기**입니다. standard 분석기는 Elasticsearch의 기본값입니다. 필드 매핑에서 분석기를 따로 지정하지 않으면 Elasticsearch가 자동으로 standard 분석기를 적용합니다. standard 분석기는 내부적으로 standard 토크나이저와 lowercase 토큰 필터로 구성되어 있습니다. standard 토크나이저는 2.3.1에서 살펴본 것처럼 유니코드 텍스트 분할 알고리즘에 기반하여 단어 경계를 인식하고 구두점을 제거합니다. lowercase 토큰 필터는 모든 토큰을 소문자로 변환합니다.

standard 분석기에 "The Quick Brown Fox"를 통과시키면 어떻게 되는지 살펴봅니다.

```
POST _analyze
{
  "analyzer": "standard",
  "text": "The Quick Brown Fox"
}
```

standard 토크나이저가 공백과 문법 규칙에 따라 네 단어로 분리합니다. 이어서 lowercase 토큰 필터가 대문자를 소문자로 변환합니다. 최종 결과는 "the", "quick", "brown", "fox" 네 토큰입니다.

standard 분석기에는 stop 토큰 필터도 포함되어 있지만, 기본 설정에서는 불용어 목록이 비어 있어 실제로는 아무 토큰도 제거하지 않습니다. 불용어 제거가 필요하면 인덱스 설정에서 stopwords 옵션을 지정할 수 있습니다.

```
PUT my_index
{
  "settings": {
    "analysis": {
      "analyzer": {
        "my_standard": {
          "type": "standard",
          "stopwords": "_english_"
        }
      }
    }
  }
}
```

이 설정에서 `type`을 "standard"로 지정하여 standard 분석기를 기반으로 하되, `stopwords`에 "_english_"를 넣어 영어 불용어 목록을 활성화합니다. 이렇게 하면 "the", "a", "is" 같은 영어 불용어가 분석 과정에서 제거됩니다. standard 분석기는 대부분의 유럽어 텍스트에 무난하게 작동하므로, 특별한 요구사항이 없을 때 그대로 사용하기 좋습니다.

다음으로 **simple 분석기**를 살펴봅니다. simple 분석기는 문자가 아닌 모든 것을 기준으로 텍스트를 분리합니다. 영문 기준으로 보면, 알파벳이 아닌 문자(숫자, 공백, 구두점, 특수문자)를 만나면 그 지점에서 토큰을 끊습니다. 분리한 토큰은 모두 소문자로 변환합니다.

standard 분석기와 비교하면 차이가 뚜렷합니다. "Quick2Eat"이라는 텍스트를 예로 들겠습니다. standard 분석기는 이를 "quick2eat" 하나의 토큰으로 처리합니다. 숫자는 단어 경계로 취급하지 않기 때문입니다. 반면 simple 분석기는 숫자 "2"를 알파벳이 아닌 문자로 판단하여 "quick"과 "eat" 두 토큰으로 분리합니다. 이처럼 simple 분석기는 숫자를 토큰에서 완전히 배제합니다.

simple 분석기는 숫자가 검색에 의미 없는 텍스트를 다룰 때 유용합니다. 반면 제품 코드나 버전 번호처럼 숫자가 중요한 데이터에는 적합하지 않습니다.

이어서 **whitespace 분석기**를 살펴봅니다. whitespace 분석기는 이름 그대로 공백 문자만을 기준으로 텍스트를 분리합니다. 그 밖에는 아무 가공도 하지 않습니다. 소문자 변환도 없고, 구두점 제거도 없습니다.

"Hello, World!"를 whitespace 분석기에 통과시키면 "Hello,"와 "World!" 두 토큰이 됩니다. 쉼표와 느낌표가 토큰에 그대로 붙어 있습니다. 이를 standard 분석기의 결과와 비교하면 차이가 명확합니다. standard 분석기는 구두점을 제거하고 소문자로 변환하여 "hello"와 "world"를 반환합니다.

whitespace 분석기는 원문의 형태를 최대한 보존해야 할 때 사용합니다. 예를 들어 로그 메시지나 프로그래밍 코드 조각처럼 구두점이나 대소문자에 의미가 있는 데이터를 인덱싱할 때 적합합니다. 다만 소문자 변환이 없으므로, "Hello"와 "hello"를 같은 단어로 검색하려면 별도의 처리가 필요합니다.

다음은 **stop 분석기**입니다. stop 분석기는 simple 분석기의 동작에 불용어 제거를 추가한 것입니다. simple 분석기처럼 알파벳이 아닌 문자를 기준으로 분리하고 소문자로 변환한 뒤, 영어 불용어 목록에 해당하는 토큰을 제거합니다.

"The Quick Brown Fox"를 stop 분석기에 통과시키면 어떻게 되는지 봅니다. 먼저 simple 분석기와 동일하게 알파벳이 아닌 문자(여기서는 공백)를 기준으로 분리하고 소문자로 변환합니다. "the", "quick", "brown", "fox"가 됩니다. 이어서 stop 필터가 불용어 "the"를 제거합니다. 최종 결과는 "quick", "brown", "fox" 세 토큰입니다.

stop 분석기는 기본적으로 영어 불용어를 제거하지만, 다른 언어의 불용어 목록을 사용하도록 설정할 수도 있습니다. 불용어 제거가 필요하면서 simple 분석기의 분리 방식이 적합한 경우에 사용합니다.

다음은 **keyword 분석기**입니다. keyword 분석기는 다른 분석기와 성격이 다릅니다. 이름에 '분석기'가 붙어 있지만, 실제로는 텍스트를 분석하지 않습니다. 입력받은 텍스트 전체를 하나의 토큰으로 그대로 출력합니다. 분리도, 소문자 변환도, 필터링도 하지 않습니다.

"New York City"를 keyword 분석기에 통과시키면 "New York City"라는 하나의 토큰만 만들어집니다. 공백으로 분리하지 않고, 원문 그대로를 하나의 토큰으로 취급하는 것입니다.

언뜻 보면 분석을 하지 않는 분석기가 왜 필요한지 의문이 들 수 있습니다. keyword 분석기는 이메일 주소, 우편번호, 상태 코드처럼 텍스트를 쪼개면 안 되는 값에 사용합니다. 물론 이런 용도라면 2.2.1에서 배운 keyword 타입을 사용하는 것이 일반적입니다. 그러나 text 타입 필드에 keyword 분석기를 적용하면, text 타입의 기능(scoring, highlighting 등)을 유지하면서도 분석을 건너뛸 수 있습니다.

이어서 **fingerprint 분석기**를 살펴봅니다. fingerprint 분석기는 텍스트에서 고유한 식별 토큰을 만들기 위한 분석기입니다. 내부적으로 standard 토크나이저로 분리한 뒤, 소문자 변환, ASCII 변환(악센트 문자를 일반 문자로), 정렬, 중복 제거를 차례로 적용하고, 결과를 하나의 문자열로 합칩니다.

예를 들어 "New York City"와 "city new york"는 단어 순서와 대소문자만 다를 뿐 같은 의미입니다. fingerprint 분석기는 두 텍스트 모두 "city new york"라는 동일한 토큰을 생성합니다. 이렇게 하면 텍스트의 표면적 차이를 무시하고 내용 기준으로 중복을 탐지할 수 있습니다. fingerprint 분석기는 데이터 정제나 중복 문서 탐지 시나리오에 적합합니다.

마지막으로 **언어별 분석기**(language analyzer)를 살펴봅니다. Elasticsearch는 english, french, german, spanish 등 수십 종의 언어별 분석기를 제공합니다. 언어별 분석기는 해당 언어의 문법적 특성을 반영하여 분석합니다.

언어별 분석기가 필요한 이유를 설명합니다. 언어마다 단어의 형태가 변하는 규칙이 다릅니다. 영어에서 "running", "runs", "ran"은 모두 "run"이라는 어근에서 파생된 단어입니다. 사용자가 "run"을 검색했을 때 이 세 단어가 포함된 문서도 함께 찾으려면, 인덱싱 시점에 파생 형태를 어근으로 되돌리는 작업이 필요합니다. 이 작업을 **어간 추출**(stemming)이라 합니다.

english 분석기를 예로 들면, 내부적으로 standard 토크나이저, lowercase 토큰 필터, 영어 불용어 제거 필터, 그리고 영어에 특화된 어간 추출 필터를 포함합니다. "The foxes were running quickly"를 english 분석기에 통과시키면 다음과 같은 과정을 거칩니다.

```
원문: "The foxes were running quickly"
  |
  v
[standard 토크나이저]  --> ["The", "foxes", "were", "running", "quickly"]
  |
  v
[lowercase 필터]       --> ["the", "foxes", "were", "running", "quickly"]
  |
  v
[영어 불용어 제거]     --> ["foxes", "running", "quickly"]
                           ("the", "were" 제거)
  |
  v
[영어 어간 추출]       --> ["fox", "run", "quick"]
                           ("foxes"→"fox", "running"→"run",
                            "quickly"→"quick")
```

불용어 "the"와 "were"가 제거되고, "foxes"는 "fox"로, "running"은 "run"으로, "quickly"는 "quick"으로 어간이 추출됩니다. 이렇게 하면 사용자가 "fox"로 검색해도 원문에 "foxes"가 있는 문서가 검색됩니다.

어간 추출의 구체적인 방식은 분석기마다 다릅니다. Elasticsearch의 언어별 분석기 중 상당수는 내부적으로 **snowball** 어간 추출기를 사용합니다. snowball은 언어별 규칙을 정의하여 단어의 접미사를 체계적으로 제거하는 알고리즘입니다. 예를 들어 영어에서 "-ing", "-ly", "-es" 같은 접미사를 규칙에 따라 잘라내어 어근을 추출합니다. snowball이라는 이름은 이 알고리즘을 구현하는 데 사용하는 프로그래밍 언어의 이름에서 유래했습니다.

french 분석기는 프랑스어의 관사와 접미사 규칙을, german 분석기는 독일어의 복합어와 격변화 규칙을 반영합니다. 이처럼 각 언어 분석기는 해당 언어의 불용어 목록과 어간 추출 규칙을 갖추고 있어, 그 언어의 텍스트를 효과적으로 분석합니다.

지금까지 살펴본 내장 분석기들의 특성을 정리합니다.

```
+-------------------+------------------+-----------+----------+-----------+
| 분석기            | 분리 기준        | 소문자    | 불용어   | 어간 추출 |
+-------------------+------------------+-----------+----------+-----------+
| standard (기본값) | 문법/구두점/공백 | O         | X (기본) | X         |
| simple            | 비알파벳 문자    | O         | X        | X         |
| whitespace        | 공백만           | X         | X        | X         |
| stop              | 비알파벳 문자    | O         | O        | X         |
| keyword           | 분리 안 함       | X         | X        | X         |
| fingerprint       | 문법/구두점/공백 | O         | X (기본) | X         |
| 언어별 (english)  | 문법/구두점/공백 | O         | O        | O         |
+-------------------+------------------+-----------+----------+-----------+
```

이 표에서 O는 해당 기능이 적용됨을, X는 적용되지 않음을 뜻합니다. standard 분석기의 불용어가 "X (기본)"인 이유는, stop 필터가 포함되어 있지만 기본 불용어 목록이 비어 있어 실질적으로 작동하지 않기 때문입니다.

어떤 분석기를 선택할지는 데이터의 성격과 검색 요구사항에 따라 달라집니다. 일반적인 텍스트라면 standard 분석기로 시작합니다. 특정 언어의 형태소 분석이 필요하면 해당 언어 분석기를 사용합니다. 원문 형태를 보존해야 하면 whitespace나 keyword 분석기를 고려합니다. 내장 분석기로 충족되지 않는 요구사항이 있으면 2.3.1에서 배운 구성요소를 직접 조합하여 커스텀 분석기를 만들 수 있습니다.

정리하면, Elasticsearch는 standard, simple, whitespace, stop, keyword, fingerprint, 언어별 분석기 등 다양한 내장 분석기를 제공합니다. 각 분석기는 토크나이저와 토큰 필터의 조합이 다르며, 분리 기준, 소문자 변환, 불용어 제거, 어간 추출 여부에서 차이가 납니다. standard 분석기가 기본값이고 대부분의 상황에 적합하지만, 데이터의 언어나 형태에 따라 적절한 내장 분석기를 선택하면 검색 품질을 높일 수 있습니다.

다음 단원인 2.3.3에서는 내장 분석기로 충족되지 않는 요구사항을 해결하기 위해 구성요소를 직접 조합하는 커스텀 분석기 구성을 다룹니다.

이 단원을 마치면 standard, simple, whitespace, language 분석기의 차이를 설명할 수 있고, 각 분석기의 적합한 사용 사례를 선택할 수 있습니다.
