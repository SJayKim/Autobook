# 1.1.2 설치 및 환경 구성

1.1.1에서 Elasticsearch가 Apache Lucene 기반의 분산 검색 엔진이며, 모든 조작을 RESTful API로 수행한다는 점을 살펴보았습니다. 이제 직접 Elasticsearch를 설치하고, 요청을 보내 정상 동작을 확인할 차례입니다. 검색 엔진이 아무리 강력해도 실행 환경이 준비되지 않으면 아무것도 할 수 없습니다. 이 단원에서는 아카이브 파일로 직접 설치하는 방법, Docker를 이용하는 방법, 그리고 핵심 설정 파일을 조정하는 방법을 차례로 다룹니다.

Elasticsearch를 설치하는 방법은 크게 두 가지입니다. 하나는 공식 사이트에서 배포하는 압축 파일을 받아 직접 풀어 실행하는 것이고, 다른 하나는 Docker 컨테이너를 이용하는 것입니다. 압축 파일 방식은 Elasticsearch가 파일 시스템 위에서 어떤 디렉터리 구조로 동작하는지 직접 확인할 수 있어 내부 이해에 도움이 됩니다. Docker 방식은 운영체제 환경에 영향을 주지 않고 빠르게 실행과 삭제를 반복할 수 있어 실습에 편리합니다. 두 방식 모두 결과적으로 같은 Elasticsearch 프로세스를 기동하므로, 본인에게 익숙한 쪽을 선택하면 됩니다.

먼저 압축 파일 방식을 살펴봅니다. Elasticsearch는 바이너리 아카이브(.tar.gz 또는 .zip)를 다운로드한 뒤 압축을 해제하는 것만으로 설치가 완료됩니다. 별도의 패키지 관리자나 설치 마법사가 필요하지 않습니다. Elasticsearch는 자바 가상 머신(JVM) 위에서 실행되지만, 아카이브 안에 OpenJDK가 이미 포함되어 있으므로 자바를 따로 설치하지 않아도 됩니다.

다음은 Linux 환경에서의 설치 절차입니다.

```bash
wget https://artifacts.elastic.co/downloads/elasticsearch/elasticsearch-8.x.x-linux-x86_64.tar.gz
shasum -a 512 -c elasticsearch-8.x.x-linux-x86_64.tar.gz.sha512
tar -xzf elasticsearch-8.x.x-linux-x86_64.tar.gz
cd elasticsearch-8.x.x/
```

첫 번째 줄은 공식 배포 서버에서 아카이브 파일을 내려받습니다. 두 번째 줄은 다운로드한 파일의 체크섬을 검증하여 전송 중 파일이 손상되지 않았는지 확인합니다. 세 번째 줄은 아카이브를 풀어 'elasticsearch-8.x.x' 디렉터리를 생성합니다. 네 번째 줄에서 해당 디렉터리로 이동하면 설치가 끝납니다. 여기서 '8.x.x' 부분은 실제 버전 번호로 바꿔야 합니다.

설치된 디렉터리 안에서 다음 명령으로 Elasticsearch를 실행합니다.

```bash
./bin/elasticsearch
```

이 명령은 Elasticsearch를 포그라운드(화면에 로그가 출력되는 상태)로 실행합니다. 터미널을 닫으면 프로세스도 종료됩니다. 백그라운드에서 실행하려면 '-d' 옵션을 추가합니다.

```bash
./bin/elasticsearch -d -p pid
```

'-d'는 데몬 모드로 실행하라는 뜻이고, '-p pid'는 프로세스 ID를 'pid'라는 파일에 기록하라는 뜻입니다. 나중에 이 파일을 읽어 프로세스를 종료할 수 있습니다.

Elasticsearch 8.x 버전을 처음 기동하면 보안 설정이 자동으로 구성됩니다. 이 과정에서 네 가지가 일어납니다. 첫째, HTTPS 통신을 위한 TLS 인증서가 자동 생성됩니다. 둘째, **elastic**이라는 관리자 계정의 비밀번호가 자동 생성되어 터미널에 출력됩니다. 셋째, Kibana 연결에 사용할 **enrollment token**이 발급됩니다. 이 토큰의 유효 기간은 30분입니다. 넷째, 외부에서 HTTP 요청을 받을 수 있도록 9200 포트가 열리고, 클러스터 내부 통신용 9300 포트는 localhost로 제한됩니다. 터미널에 출력되는 비밀번호와 토큰은 반드시 따로 저장해 두어야 합니다.

기동 확인은 curl 명령으로 수행합니다.

```bash
curl --cacert $ES_HOME/config/certs/http_ca.crt \
  -u elastic:$ELASTIC_PASSWORD https://localhost:9200
```

'--cacert'는 TLS 인증서 경로를 지정하는 옵션입니다. 보안이 자동 활성화되었으므로 HTTPS로 접근해야 하고, 자체 서명 인증서를 검증하기 위해 경로를 알려주는 것입니다. '-u elastic:$ELASTIC_PASSWORD'는 사용자명과 비밀번호를 전달합니다. 정상 기동 상태라면 클러스터 이름, 노드 이름, 버전 정보가 담긴 JSON 응답이 돌아옵니다.

다음은 Docker를 이용한 설치입니다. **Docker**는 애플리케이션을 '컨테이너'라는 격리된 환경에서 실행하는 도구입니다. 컨테이너는 운영체제 위에 가볍게 올라가는 실행 환경으로, 호스트 시스템의 설정을 건드리지 않고 독립적으로 프로그램을 실행합니다. Docker가 설치되어 있다면, 명령 한 줄로 Elasticsearch 단일 노드를 기동할 수 있습니다.

```bash
docker network create elastic
docker pull docker.elastic.co/elasticsearch/elasticsearch:8.x.x
docker run --name es01 --net elastic -p 9200:9200 -it -m 1GB \
  docker.elastic.co/elasticsearch/elasticsearch:8.x.x
```

첫 번째 줄은 'elastic'이라는 이름의 Docker 네트워크를 만듭니다. 나중에 Kibana 등 다른 컨테이너와 통신할 때 같은 네트워크에 있어야 서로를 찾을 수 있기 때문입니다. 두 번째 줄은 공식 이미지를 내려받습니다. 세 번째 줄이 실제 실행 명령입니다. '--name es01'은 컨테이너에 'es01'이라는 이름을 부여합니다. '--net elastic'은 앞서 만든 네트워크에 연결합니다. '-p 9200:9200'은 호스트의 9200 포트를 컨테이너의 9200 포트에 연결합니다. '-m 1GB'는 컨테이너가 사용할 수 있는 메모리를 1GB로 제한합니다. 머신 러닝 기능까지 사용하려면 6GB 이상을 할당해야 합니다.

이 명령을 실행하면, 압축 파일 방식과 마찬가지로 elastic 계정 비밀번호와 Kibana enrollment token이 터미널에 출력됩니다.

학습이나 개발 목적으로 Elasticsearch를 사용할 때는, 보안 설정 때문에 매번 인증서와 비밀번호를 다루는 것이 번거로울 수 있습니다. 이런 경우 보안 기능을 비활성화하면 편리합니다.

```bash
docker run --name es01 --net elastic -p 9200:9200 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  docker.elastic.co/elasticsearch/elasticsearch:8.x.x
```

'-e' 옵션은 컨테이너 내부에 환경 변수를 전달합니다. 'discovery.type=single-node'는 이 노드가 혼자서 클러스터를 구성한다는 뜻입니다. 이 설정이 있으면 다른 노드를 탐색하는 부트스트랩 절차를 건너뛰고 곧바로 기동합니다. 'xpack.security.enabled=false'는 Elasticsearch의 보안 모듈을 끄는 설정입니다. 보안을 끄면 TLS 인증서 없이 HTTP 평문 통신으로 접근할 수 있어, 'http://localhost:9200'만으로 바로 요청을 보낼 수 있습니다. 단, 이 설정은 반드시 로컬 개발 환경에서만 사용해야 합니다. 외부에 노출되는 서버에서 보안을 끄면 누구나 데이터에 접근할 수 있습니다.

이제 여러 노드로 구성된 클러스터를 한 번에 띄우는 방법을 살펴봅니다. **Docker Compose**는 여러 컨테이너를 하나의 설정 파일(docker-compose.yml)로 정의하고, 한 번의 명령으로 모두 기동하는 도구입니다. Elasticsearch 공식 문서는 3개 노드와 Kibana를 함께 구성하는 예시를 제공합니다.

Docker Compose를 사용하려면 먼저 환경 변수를 담은 .env 파일을 준비합니다.

```
ELASTIC_PASSWORD=changeme
KIBANA_PASSWORD=changeme
STACK_VERSION=8.x.x
ES_PORT=9200
```

'ELASTIC_PASSWORD'는 elastic 관리자 계정의 비밀번호입니다. 영문자와 숫자를 조합하여 6자 이상으로 설정합니다. 느낌표(!)나 골뱅이(@) 같은 특수문자는 스크립트 처리 과정에서 문제를 일으킬 수 있으므로 사용하지 않습니다. 'KIBANA_PASSWORD'는 Kibana가 Elasticsearch에 접속할 때 사용하는 내부 계정(kibana_system)의 비밀번호입니다. 'STACK_VERSION'은 설치할 Elasticsearch 버전이고, 'ES_PORT'는 외부에 노출할 포트 번호입니다.

docker-compose.yml 파일에서 각 노드는 서비스(service)로 정의됩니다. 아래는 노드 하나의 환경 변수 설정 패턴입니다.

```yaml
environment:
  - cluster.name=es-docker-cluster
  - node.name=es01
  - discovery.seed_hosts=es02,es03
  - cluster.initial_master_nodes=es01,es02,es03
  - bootstrap.memory_lock=true
  - ES_JAVA_OPTS=-Xms512m -Xmx512m
  - xpack.security.enabled=true
  - xpack.security.http.ssl.enabled=true
```

'cluster.name'은 이 클러스터의 이름입니다. 같은 이름을 가진 노드끼리 하나의 클러스터를 형성합니다. 'node.name'은 노드 식별자입니다. 'discovery.seed_hosts'는 이 노드가 클러스터에 합류할 때 먼저 연락할 다른 노드의 주소 목록입니다. 'cluster.initial_master_nodes'는 클러스터를 처음 구성할 때 마스터 투표에 참여할 노드 이름 목록입니다. 클러스터가 한 번 형성된 뒤에는 이 설정이 더 이상 사용되지 않으므로, 운영 환경에서는 형성 후 제거하는 것이 권장됩니다. 'bootstrap.memory_lock'을 true로 설정하면 JVM 힙 메모리가 운영체제의 스왑 영역으로 밀려나지 않도록 고정합니다. 'ES_JAVA_OPTS'는 JVM 힙 크기를 지정합니다. 이 항목은 뒤에서 자세히 다룹니다.

Docker Compose로 구성할 때 Docker Desktop에는 최소 4GB 메모리를 할당해야 합니다. 기동은 다음 명령으로 수행합니다.

```bash
docker compose up -d
```

'-d'는 백그라운드 실행을 뜻합니다. 기동이 완료되면 'http://localhost:5601'에서 Kibana에 접속할 수 있으며, elastic 계정과 .env 파일에 설정한 비밀번호로 로그인합니다.

다음 다이어그램은 Docker Compose로 구성한 3노드 클러스터와 Kibana의 전체 구조를 보여줍니다.

```
Docker Compose 클러스터 구성

+--------------------------------------------------+
|  Docker Network: elastic                         |
|                                                  |
|  +----------+  +----------+  +----------+        |
|  |   es01   |  |   es02   |  |   es03   |        |
|  | :9200    |  | :9200    |  | :9200    |        |
|  | :9300    |  | :9300    |  | :9300    |        |
|  +----+-----+  +----+-----+  +----+-----+        |
|       |              |              |              |
|       +---------+----+---------+----+              |
|                 |                                  |
|          +------+-------+                          |
|          |    Kibana    |                          |
|          |   :5601     |                          |
|          +--------------+                          |
+--------------------------------------------------+
         |
    호스트 포트: 9200, 5601
```

es01, es02, es03 세 노드가 같은 Docker 네트워크 안에서 9300 포트로 서로 통신하며 클러스터를 형성합니다. Kibana는 같은 네트워크에서 es01의 9200 포트에 연결합니다. 호스트 머신에서는 9200 포트로 Elasticsearch에, 5601 포트로 Kibana에 접근합니다.

지금까지 Elasticsearch를 기동하는 방법을 살펴보았습니다. 이제 기동 후 동작을 결정하는 설정 파일을 살펴봅니다. Elasticsearch의 핵심 설정 파일은 **elasticsearch.yml**입니다. 이 파일은 노드가 시작할 때 한 번 읽히는 정적 설정 파일입니다. 파일을 수정한 뒤에는 노드를 재시작해야 변경 사항이 반영됩니다. 아카이브로 설치한 경우 'config/elasticsearch.yml' 경로에 위치합니다.

가장 중요한 파라미터 세 가지를 먼저 다룹니다.

**cluster.name**은 클러스터의 이름을 지정합니다. 기본값은 'elasticsearch'입니다. 같은 네트워크에 있는 노드 중 cluster.name이 동일한 노드끼리만 하나의 클러스터를 형성합니다. 개발 환경과 운영 환경이 같은 네트워크에 있다면 이름을 다르게 설정하여 노드가 잘못된 클러스터에 합류하는 것을 방지해야 합니다.

**node.name**은 노드의 고유 식별자입니다. 기본값은 호스트의 이름입니다. API 응답이나 로그에 이 이름이 표시되므로, 여러 노드를 운영할 때 각 노드를 구분하기 쉬운 이름을 부여하는 것이 좋습니다. 예를 들어 'prod-data-1', 'prod-data-2'처럼 역할과 순서를 포함하면 로그를 읽을 때 편리합니다.

**network.host**는 Elasticsearch가 요청을 수신할 네트워크 인터페이스를 지정합니다. 기본값은 '127.0.0.1'(루프백)로, 같은 머신에서만 접근할 수 있습니다. 다른 서버에서도 접근해야 한다면 실제 IP 주소나 '0.0.0.0'(모든 인터페이스)으로 변경합니다. 여기서 주의할 점이 있습니다. network.host를 루프백이 아닌 값으로 바꾸면 Elasticsearch가 '프로덕션 모드'로 전환됩니다. 프로덕션 모드에서는 시스템 부트스트랩 검사가 엄격해져서, 검사를 통과하지 못하면 노드가 아예 기동되지 않습니다. 로컬에서 실습할 때는 기본값을 그대로 두는 것이 안전합니다.

이 세 가지 외에도 자주 쓰이는 파라미터가 있습니다.

```yaml
path.data: /var/data/elasticsearch
path.logs: /var/log/elasticsearch
discovery.seed_hosts: ["es02", "es03"]
cluster.initial_master_nodes: ["es01", "es02", "es03"]
```

'path.data'는 인덱스 데이터가 저장되는 디렉터리 경로입니다. 기본값은 Elasticsearch 설치 디렉터리 아래의 'data' 폴더인데, 버전 업그레이드 시 설치 디렉터리를 교체하면 데이터가 함께 사라질 수 있으므로 설치 경로 바깥을 지정하는 것이 좋습니다. 'path.logs'는 로그 파일 경로입니다. 'discovery.seed_hosts'와 'cluster.initial_master_nodes'는 Docker Compose 섹션에서 이미 설명한 것과 동일한 역할을 합니다.

Elasticsearch의 성능에 직접적으로 영향을 미치는 또 다른 설정이 **JVM 힙 메모리**입니다. Elasticsearch는 JVM 위에서 실행되므로, JVM이 사용할 수 있는 메모리 양을 지정해야 합니다. 힙(heap)이란 JVM이 객체를 생성하고 관리하는 메모리 공간입니다. 힙이 너무 작으면 데이터를 처리하다가 메모리 부족(OOM, Out of Memory) 오류가 발생합니다. 반대로 너무 크면 가비지 컬렉션(GC)이 오래 걸려 응답이 멈추는 현상이 생깁니다. 가비지 컬렉션이란 JVM이 더 이상 사용하지 않는 객체를 정리하여 메모리를 회수하는 과정입니다. 힙이 클수록 정리할 대상이 많아져 한 번의 GC에 걸리는 시간이 길어집니다.

JVM 힙 설정은 'config/jvm.options.d/' 디렉터리 아래에 '.options' 확장자를 가진 파일을 만들어 지정합니다. 루트 경로의 'jvm.options' 파일을 직접 수정하는 것은 권장하지 않습니다.

```
-Xms4g
-Xmx4g
```

'-Xms'는 최소 힙 크기, '-Xmx'는 최대 힙 크기를 뜻합니다. 이 두 값은 반드시 같게 설정합니다. 값이 다르면 런타임 중에 힙 크기가 변경되면서 성능이 불안정해질 수 있기 때문입니다.

JVM 힙을 설정할 때 지켜야 하는 두 가지 핵심 규칙이 있습니다.

첫째, 노드 전체 메모리의 50% 이하로 설정합니다. 나머지 50%는 운영체제가 파일시스템 캐시로 사용해야 합니다. Elasticsearch는 디스크에 저장된 인덱스 파일을 읽을 때 운영체제의 페이지 캐시를 활용하는데, 이 캐시가 충분하지 않으면 디스크 접근이 잦아져 검색 성능이 크게 떨어집니다.

둘째, 약 26~30GB를 초과하지 않습니다. JVM에는 **Compressed OOP**(Compressed Ordinary Object Pointer)라는 메모리 최적화 기법이 있습니다. 힙이 약 32GB 미만이면 객체 포인터를 압축하여 메모리를 절약하는데, 이 임계치를 넘으면 압축이 해제되어 오히려 메모리 효율이 낮아집니다. 예를 들어 64GB 메모리를 가진 서버에서 힙을 64GB로 잡는 것보다, 31GB로 잡고 나머지를 파일시스템 캐시에 넘기는 것이 성능상 유리합니다.

임시로 테스트할 때는 환경 변수로 힙을 지정할 수도 있습니다.

```bash
ES_JAVA_OPTS="-Xms2g -Xmx2g" ./bin/elasticsearch
```

다만 이 방식은 실행할 때마다 지정해야 하므로, 파일로 설정하는 것이 안정적입니다. 기동 로그에서 'heap size [4.0gb], compressed ordinary object pointers [true]'와 같은 메시지를 확인하면 힙 설정이 의도대로 적용된 것입니다.

Elasticsearch가 기동된 뒤에는 클러스터 상태를 확인해야 합니다. 가장 기본적인 확인 방법은 **_cluster/health** API입니다.

```bash
GET /_cluster/health
```

이 API는 클러스터 전체 상태를 신호등 모델(green, yellow, red)로 알려줍니다.

**green**은 모든 프라이머리 샤드와 레플리카 샤드가 정상적으로 할당된 상태입니다. 데이터의 읽기와 쓰기가 모두 원활하며, 노드 장애에 대한 복원력도 확보된 가장 이상적인 상태입니다.

**yellow**는 프라이머리 샤드는 모두 할당되었지만 일부 레플리카가 미할당인 상태입니다. 검색과 인덱싱은 정상 동작하지만, 레플리카가 부족하여 특정 노드에 장애가 발생하면 데이터 유실 위험이 있습니다.

**red**는 일부 프라이머리 샤드가 할당되지 않은 상태로, 해당 샤드의 데이터에 접근할 수 없음을 의미합니다. 즉시 원인을 파악하고 조치해야 하는 긴급 상태입니다.

단일 노드 환경에서는 yellow가 정상입니다. 1.1.1에서 설명한 것처럼, 레플리카 샤드는 프라이머리 샤드와 반드시 다른 노드에 배치되어야 합니다. 노드가 하나뿐이면 레플리카를 배치할 곳이 없으므로 미할당 상태가 됩니다. 이 경우 레플리카 수를 0으로 설정하면 green이 됩니다.

```
클러스터 상태 신호등

+---------------------------------------------------+
|  green    모든 샤드 정상 할당                      |
+---------------------------------------------------+
|  yellow   프라이머리 정상, 레플리카 일부 미할당     |
|           (단일 노드 개발 환경에서는 정상)          |
+---------------------------------------------------+
|  red      프라이머리 일부 미할당 - 데이터 접근 불가  |
+---------------------------------------------------+
```

클러스터 상태 외에 더 세부적인 정보를 확인하는 명령도 있습니다. '_cat/nodes?v'는 노드별 CPU, 메모리, 디스크 현황을 보여줍니다. '_cat/indices?v'는 인덱스별 문서 수와 크기를 보여줍니다. '_cat/shards?v'는 샤드 할당 현황을 보여줍니다. 이 명령들은 1.1.1에서 설명한 cat API에 속하며, '?v' 파라미터는 헤더 행을 포함하여 출력을 읽기 쉽게 만듭니다.

마지막으로 Kibana 연결을 설정합니다. 1.1.1에서 **Kibana**가 Elasticsearch에 저장된 데이터를 시각화하고 탐색하는 도구라고 설명했습니다. Kibana는 반드시 Elasticsearch와 같은 버전을 사용해야 합니다. Kibana의 설정 파일은 **kibana.yml**이며, 핵심 파라미터는 다음과 같습니다.

```yaml
server.port: 5601
server.host: "0.0.0.0"
elasticsearch.hosts: ["http://localhost:9200"]
elasticsearch.username: "kibana_system"
elasticsearch.password: "changeme"
```

'server.port'는 Kibana가 요청을 수신하는 포트입니다. 기본값은 5601입니다. 'server.host'를 '0.0.0.0'으로 설정하면 외부에서도 접근할 수 있습니다. 'elasticsearch.hosts'는 연결 대상 Elasticsearch의 URL입니다. 여러 URL을 나열하면 Kibana가 요청을 번갈아 보내는 라운드로빈 방식으로 부하를 분산합니다. 'elasticsearch.username'과 'elasticsearch.password'는 Kibana가 내부적으로 Elasticsearch에 접속할 때 사용하는 전용 계정 정보입니다.

보안이 활성화된 환경(기본 상태)에서 Docker로 기동한 경우, Elasticsearch가 출력한 enrollment token을 Kibana 초기 화면에 입력하면 TLS 및 인증 설정이 자동으로 완료됩니다. 앞서 보안을 비활성화한 개발 환경이라면 'http://' URL 그대로 사용하면 되며, 별도의 인증 설정이 필요하지 않습니다.

아래는 설치 방식별로 보안 활성화 여부에 따른 접근 방법을 정리한 것입니다.

```
설치 방식별 접근 방법 비교

+-----------+---------------------+-----------------------------+
| 설치 방식 | 보안 활성화 (기본)  | 보안 비활성화 (개발용)      |
+-----------+---------------------+-----------------------------+
| 아카이브  | https://localhost   | http://localhost:9200       |
|           | :9200 + 인증서 +   | 인증 불필요                 |
|           | elastic 계정       |                             |
+-----------+---------------------+-----------------------------+
| Docker    | https + enrollment  | http://localhost:9200       |
|           | token (Kibana)     | -e xpack.security           |
|           | elastic 계정       | .enabled=false              |
+-----------+---------------------+-----------------------------+
| Compose   | https + .env 파일   | environment:                |
|           | 비밀번호 설정       | xpack.security              |
|           |                     | .enabled=false              |
+-----------+---------------------+-----------------------------+
```

보안 비활성화 설정을 elasticsearch.yml에 직접 기록하는 경우는 다음과 같습니다.

```yaml
xpack.security.enabled: false
xpack.security.http.ssl.enabled: false
xpack.security.transport.ssl.enabled: false
```

첫 번째 줄은 보안 모듈 전체를 끕니다. 두 번째 줄은 HTTP 레이어의 TLS를 끕니다. 세 번째 줄은 노드 간 통신(transport 레이어)의 TLS를 끕니다. 이 세 줄을 모두 설정하면 인증서와 비밀번호 없이 평문 HTTP로 Elasticsearch에 접근할 수 있습니다. 다시 한번 강조하지만, 이 설정은 로컬 개발 환경에서만 사용해야 합니다.

정리하면, Elasticsearch는 아카이브 압축 해제 또는 Docker 컨테이너로 설치하며, 8.x 버전부터는 최초 기동 시 보안이 자동 구성됩니다. elasticsearch.yml에서 cluster.name, node.name, network.host를 조정하여 클러스터 구성을 제어하고, jvm.options.d/ 아래에서 JVM 힙을 노드 메모리의 50% 이하, 26~30GB 이하로 설정합니다. 기동 후에는 _cluster/health API로 클러스터 상태를 확인하고, 개발 환경에서는 xpack.security를 비활성화하여 간편하게 실습할 수 있습니다.

다음 단원인 1.1.3에서는 첫 번째 문서 인덱싱과 검색을 다룹니다.

이 단원을 마치면 Elasticsearch를 로컬에 설치하고 정상 기동을 확인할 수 있고, Docker Compose로 단일 노드 클러스터를 구성할 수 있으며, elasticsearch.yml의 핵심 설정 항목을 조작할 수 있습니다.
