# 7.1.1 dense_vector와 HNSW

Elasticsearch는 역색인을 기반으로 텍스트를 빠르게 검색합니다. 그런데 "의미가 비슷한 문서"를 찾고 싶을 때는 어떨까요. 예를 들어 사용자가 "가을 산책하기 좋은 곳"이라고 입력했을 때, 본문에 "가을"이나 "산책"이라는 단어가 없어도 "단풍 명소 추천"이라는 문서를 결과로 돌려주고 싶은 경우가 있습니다. 이런 검색을 가능하게 하는 핵심 재료가 **벡터**(vector)입니다.

벡터란 여러 개의 숫자를 순서대로 나열한 배열입니다. 머신러닝 모델이 텍스트를 분석하면, 그 텍스트의 의미를 수백 개에서 수천 개의 숫자 배열로 변환합니다. 이 숫자 배열을 **임베딩**(embedding)이라고 부릅니다. 의미가 비슷한 두 문장은 임베딩 벡터끼리도 가까운 위치에 놓이게 됩니다. 반대로 의미가 다른 문장은 벡터 사이의 거리가 멀어집니다. 이 성질을 이용하면 "거리가 가까운 벡터를 가진 문서"를 찾아 의미 기반 검색을 수행할 수 있습니다.

Elasticsearch에서 이 벡터를 저장하려면 매핑에 **dense_vector** 타입을 지정합니다. 2.2.3에서 다룬 특수 타입 중 하나인 dense_vector는 float 값의 밀집 벡터를 저장하는 전용 필드 타입입니다. "밀집"이라는 이름은 배열의 모든 위치에 값이 채워져 있다는 뜻입니다. 나중에 7.2.1에서 다룰 희소 벡터(sparse vector)와 구분되는 지점입니다.

dense_vector 필드를 만들 때 가장 먼저 설정하는 속성은 **dims**입니다. dims는 벡터의 차원 수, 즉 숫자 배열의 길이를 뜻합니다. 임베딩 모델마다 출력하는 차원 수가 다릅니다. 768차원을 출력하는 모델을 사용한다면 dims를 768로 설정합니다.

```json
PUT my-vector-index
{
  "mappings": {
    "properties": {
      "content_embedding": {
        "type": "dense_vector",
        "dims": 768,
        "index": true,
        "similarity": "cosine"
      }
    }
  }
}
```

이 매핑에서 type을 dense_vector로 지정하고, dims를 768로 설정했습니다. index를 true로 두면 Elasticsearch가 이 벡터 필드에 대해 검색용 인덱스 구조를 만듭니다. 이 인덱스 구조가 바로 HNSW입니다. index를 false로 두면 벡터 값은 저장되지만 근사 최근접 이웃 검색(approximate kNN)에 사용할 수 없습니다. similarity는 두 벡터 사이의 거리를 재는 방법을 지정합니다.

**similarity** 메트릭은 세 가지 중 하나를 고릅니다. 첫째, **cosine**은 두 벡터가 이루는 각도를 기준으로 유사도를 측정합니다. 방향이 같으면 1에 가깝고, 직교하면 0에 가깝습니다. 텍스트 임베딩에서 널리 쓰이며, 벡터의 크기(길이)에 영향을 받지 않는 장점이 있습니다.

둘째, **dot_product**는 두 벡터의 내적을 계산합니다. cosine과 비슷한 결과를 내지만, 벡터의 크기를 별도로 정규화하지 않으므로 계산이 더 빠릅니다. 이미 단위 벡터로 정규화된 임베딩을 사용한다면 dot_product가 cosine 대비 2~3배 빠른 성능을 보입니다. 운영 환경에서는 dot_product를 권장하는 이유입니다.

셋째, **l2_norm**은 유클리드 거리를 사용합니다. 두 점 사이의 직선 거리를 구하는 방식으로, 벡터 검색에서는 일반적으로 cosine이나 dot_product보다 느립니다.

이제 index를 true로 설정하면 내부에서 생성되는 **HNSW**(Hierarchical Navigable Small World) 알고리즘을 살펴봅니다. HNSW는 대량의 벡터 사이에서 가장 가까운 이웃을 빠르게 찾기 위한 그래프 기반 알고리즘입니다. 이름에 들어 있는 핵심 개념을 하나씩 풀겠습니다.

"Small World"는 네트워크 이론에서 나온 개념입니다. 아무리 많은 노드가 있어도 소수의 연결(edge)만 거치면 임의의 두 노드에 도달할 수 있는 네트워크를 뜻합니다. HNSW는 벡터 하나하나를 노드로 놓고, 가까운 벡터끼리 연결선을 만들어 이런 네트워크를 구성합니다.

"Hierarchical"은 이 네트워크를 여러 층(layer)으로 쌓는다는 뜻입니다. 맨 위 층에는 소수의 노드만 있고, 아래층으로 갈수록 노드가 많아집니다. 검색할 때는 꼭대기 층에서 대략적인 방향을 잡고, 한 층씩 내려가면서 점점 정밀하게 가장 가까운 벡터를 찾습니다. 이 계층 구조 덕분에 수백만 개의 벡터가 있어도 로그 단위의 계산량으로 근사 최근접 이웃을 찾을 수 있습니다.

```
Layer 2 (최상위):    [A] -------- [B]
                      |
Layer 1 (중간):      [A] --- [C] --- [B] --- [D]
                      |       |       |       |
Layer 0 (최하위):    [A]-[E]-[C]-[F]-[B]-[G]-[D]-[H]
```

위 다이어그램에서 검색 벡터가 들어오면, 먼저 Layer 2에서 A와 B 중 더 가까운 쪽을 고릅니다. A가 더 가깝다고 판단하면 Layer 1로 내려가 A 주변의 C를 확인합니다. 다시 Layer 0으로 내려가 C 주변의 E, F를 비교하여 최종 결과를 결정합니다. 전체 벡터를 하나하나 비교하는 것보다 훨씬 적은 연산으로 결과를 얻습니다.

HNSW 그래프를 만들 때 조정할 수 있는 파라미터가 두 가지 있습니다. 첫째는 **m** 파라미터입니다. m은 각 노드가 가질 수 있는 연결 수의 상한입니다. m 값이 크면 노드마다 더 많은 이웃과 연결되므로 검색 정확도가 올라갑니다. 대신 그래프가 차지하는 메모리가 늘어나고, 인덱스를 만드는 시간도 길어집니다. 기본값은 16입니다.

둘째는 **ef_construction** 파라미터입니다. 그래프를 구축할 때 후보로 고려하는 이웃의 수를 뜻합니다. 이 값이 클수록 그래프 품질이 높아지지만, 인덱스 빌드 시간이 늘어납니다. 기본값은 100입니다. 검색 시점에는 별도의 ef_search 값이 사용되며, 이 값은 7.1.2에서 num_candidates와 함께 다룹니다.

```json
PUT my-vector-index
{
  "mappings": {
    "properties": {
      "content_embedding": {
        "type": "dense_vector",
        "dims": 768,
        "index": true,
        "similarity": "dot_product",
        "index_options": {
          "type": "hnsw",
          "m": 16,
          "ef_construction": 100
        }
      }
    }
  }
}
```

index_options 안에서 type을 hnsw로 지정하고, m과 ef_construction을 설정합니다. 대부분의 경우 기본값으로 충분하지만, 검색 정확도를 높여야 한다면 m을 32로, ef_construction을 200으로 올리는 식으로 조정할 수 있습니다.

벡터 차원 수가 늘어나면 메모리 사용량도 함께 증가합니다. 각 벡터는 4바이트(float32) 곱하기 차원 수만큼의 공간을 차지하고, 여기에 HNSW 그래프 포인터 비용이 더해집니다. 계산 공식은 다음과 같습니다.

```
메모리 = 벡터 수 x 4 x (차원 수 + 12)
```

예를 들어 768차원 벡터 2천만 개를 저장한다면, 2천만 곱하기 4 곱하기 (768 + 12)로 약 62.4GB의 오프힙(off-heap) 메모리가 필요합니다. 레플리카가 하나 있으면 이 값이 두 배가 됩니다. 벡터 검색 성능은 이 데이터가 오프힙 RAM에 들어가느냐에 크게 좌우되므로, 클러스터 사이징 단계에서 반드시 계산해야 합니다.

메모리 사용량을 줄이고 싶다면 **byte 양자화**(8비트 양자화)를 고려할 수 있습니다. 기본 dense_vector는 각 차원을 4바이트 float로 저장하지만, element_type을 byte로 설정하면 1바이트 정수(-128~127)로 저장합니다. 메모리 사용량이 약 4분의 1로 줄어드는 대신, 정밀도가 다소 떨어집니다. 대규모 벡터 인덱스에서 비용 대비 성능 균형을 맞출 때 유용합니다.

```json
{
  "type": "dense_vector",
  "dims": 768,
  "element_type": "byte",
  "index": true,
  "similarity": "cosine"
}
```

element_type을 byte로 바꾸면 임베딩 모델의 출력을 -128에서 127 사이의 정수로 변환하여 저장합니다. 이미 양자화를 지원하는 모델을 사용하거나, 인제스트 파이프라인에서 변환 로직을 추가하는 방식으로 적용합니다.

정리하면, dense_vector는 Elasticsearch에서 벡터 검색을 가능하게 하는 필드 타입이고, dims로 차원 수를, similarity로 거리 측정 방식을 지정합니다. index를 true로 설정하면 HNSW 그래프가 만들어져 대량의 벡터 사이에서도 로그 단위의 빠른 근사 검색이 가능해집니다. m과 ef_construction으로 그래프 품질을 조정하고, byte 양자화로 메모리를 절약할 수 있습니다.

다음 단원인 7.1.2에서는 이 HNSW 인덱스 위에서 실제로 kNN 검색을 수행하는 방법을 다룹니다.

이 단원을 마치면 dense_vector 매핑을 설정하고 HNSW 인덱스 파라미터를 조정할 수 있습니다.
