# 2.1.3 매핑 폭발 방지

2.1.2에서 동적 매핑을 통해 새 필드가 자동으로 추가되는 구조를 살펴보았습니다. 새 필드 추가 자체는 자유롭지만, 이 자유가 통제 없이 반복되면 심각한 문제가 생깁니다. 예를 들어 애플리케이션 로그에 사용자 정의 레이블이 포함되어 있다고 가정합니다. labels.app, labels.env, labels.team, labels.customer_id처럼 수백 종류의 키가 존재하고, 새로운 키가 수시로 추가됩니다. 동적 매핑이 켜져 있으면 이 키 하나하나가 별도 필드로 매핑에 등록됩니다. 필드 수가 수만 개, 수십만 개로 불어나면 클러스터 전체가 불안정해질 수 있습니다. 이 현상을 **매핑 폭발**(mapping explosion)이라고 합니다. 이 단원에서는 매핑 폭발이 왜 위험한지 살펴보고, 이를 방지하는 여러 기법을 하나씩 다룹니다.

매핑 폭발이 클러스터에 미치는 영향을 이해하려면, 매핑 정보가 어디에 저장되는지를 먼저 알아야 합니다. Elasticsearch 클러스터에는 **클러스터 상태**(cluster state)라는 메타데이터가 있습니다. 클러스터 상태에는 모든 인덱스의 매핑, 설정, 샤드 배치 정보 등이 담겨 있습니다. 중요한 점은 클러스터의 모든 노드가 이 상태를 메모리에 보유한다는 것입니다. 인덱스에 필드가 하나 추가되면, 해당 필드의 매퍼(mapper) 정보가 클러스터 상태에 등록되고, 모든 노드의 힙 메모리를 추가로 점유합니다. 필드가 수십만 개로 늘어나면 힙 메모리 소비가 급증하여 OutOfMemoryError가 발생하거나, 가비지 컬렉션(GC) 압박으로 노드 응답이 느려집니다. 최악의 경우 노드가 클러스터에서 이탈하여 서비스 중단으로 이어집니다.

매핑 폭발이 특히 발생하기 쉬운 환경이 있습니다. 로그나 이벤트 데이터에서 임의의 키-값 쌍(arbitrary key-value pairs)이 포함된 JSON을 동적 매핑으로 인덱싱하는 경우입니다. Kubernetes 파드 레이블, 사용자 정의 태그, 애플리케이션 설정 맵 등은 어떤 키가 들어올지 사전에 알 수 없습니다. 이런 데이터를 별다른 제어 없이 인덱싱하면, 매핑의 필드 수가 끝없이 늘어납니다.

Elasticsearch는 이 문제를 방지하기 위해 인덱스당 최대 필드 수를 제한하는 설정을 제공합니다. **index.mapping.total_fields.limit**가 바로 그 설정입니다. 기본값은 1,000입니다. 인덱스의 필드(매퍼) 수가 이 한도에 도달하면, 새 필드를 포함한 문서의 인덱싱이 실패합니다.

여기서 필드 수를 세는 방식에 주의해야 합니다. 단순히 최하위 필드명의 개수만 세는 것이 아닙니다. Elasticsearch는 경로에 포함된 모든 매퍼를 합산합니다. 예를 들어 host.os.name이라는 필드가 있으면, host 오브젝트 매퍼, host.os 오브젝트 매퍼, host.os.name 필드 매퍼까지 총 3개로 계산됩니다. 멀티필드도 별도로 카운팅됩니다. 2.1.2에서 다룬 text+keyword 멀티필드 구성이라면, 하나의 논리적 필드가 매퍼 2개를 차지합니다. 필드 별칭과 런타임 필드도 카운팅 대상입니다. 반면 _id, _source 같은 메타데이터 필드는 제외됩니다.

이 설정은 인덱스 생성 시에 지정할 수도 있고, 운영 중에 동적으로 변경할 수도 있습니다.

```json
PUT /my-index/_settings
{
  "index.mapping.total_fields.limit": 2000
}
```

이 요청은 my-index 인덱스의 최대 필드 수를 2,000으로 올립니다. 하지만 한도를 단순히 올리는 것만으로는 근본적인 해결이 되지 않습니다. 한도를 올려도 필드가 계속 늘어나면 결국 같은 문제가 반복됩니다. 한도 설정은 폭발을 감지하는 안전장치이지, 폭발 자체를 막는 방법은 아닙니다.

한도에 도달했을 때의 동작을 조절하는 옵션도 있습니다. **index.mapping.total_fields.ignore_dynamic_beyond_limit**를 true로 설정하면, 한도를 초과하는 새 필드가 인덱싱 실패 대신 _ignored 메타데이터 필드에 저장됩니다. 매핑에는 추가되지 않으므로 해당 필드로 검색하는 것은 불가능하지만, 문서 자체는 거부되지 않고 저장됩니다. 데이터 유실을 방지하면서 매핑 폭발을 억제해야 하는 상황에서 유용합니다.

total_fields.limit 외에도 매핑 구조를 제한하는 여러 설정이 있습니다.

```
매핑 제한 설정 요약

설정명                                    기본값           설명
index.mapping.total_fields.limit         1,000           인덱스당 최대 필드(매퍼) 수
index.mapping.depth.limit                20              오브젝트 중첩 최대 깊이
index.mapping.nested_fields.limit        100             nested 타입 필드 최대 개수
index.mapping.nested_objects.limit       10,000          문서당 nested JSON 객체 최대 수
index.mapping.field_name_length.limit    Long.MAX_VALUE  필드명 최대 길이
```

depth.limit는 오브젝트가 얼마나 깊이 중첩될 수 있는지를 제한합니다. nested_fields.limit는 nested 타입으로 선언할 수 있는 필드의 최대 개수를, nested_objects.limit는 한 문서 안에 포함될 수 있는 nested JSON 객체의 최대 수를 제한합니다. 이 설정들은 모두 매핑 구조가 무한히 복잡해지는 것을 방지하는 안전장치입니다.

한도 설정은 문제를 감지하고 제한하는 역할을 합니다. 그렇다면 애초에 필드 수가 폭발하지 않도록 매핑을 설계하는 방법은 무엇일까요. 첫 번째 방법은 **flattened** 필드 타입입니다.

flattened 타입은 JSON 오브젝트 전체를 하나의 필드로 처리합니다. 일반적인 매핑에서는 오브젝트 내부의 각 키가 별도 필드로 등록됩니다. 예를 들어 labels라는 오브젝트에 priority, release, env 세 개의 키가 있으면, labels.priority, labels.release, labels.env로 3개의 필드 매퍼가 생성됩니다. flattened 타입으로 labels를 선언하면, 내부에 키가 몇 개든 매핑에는 labels 하나만 등록됩니다. total_fields.limit 카운팅에서도 1로 계산됩니다.

```json
PUT /issues
{
  "mappings": {
    "properties": {
      "title":  { "type": "text" },
      "labels": { "type": "flattened" }
    }
  }
}
```

이 매핑에서 labels 필드는 flattened 타입입니다. 이제 labels 안에 어떤 키가 들어오든 매핑에는 labels 하나만 존재합니다. labels.priority, labels.release, labels.env, labels.customer_id 등 수백 개의 키가 담긴 문서를 인덱싱해도 필드 수는 늘어나지 않습니다.

flattened 타입이 내부적으로 동작하는 방식을 좀 더 살펴봅니다. 인덱싱 시 JSON 오브젝트의 모든 리프(leaf) 값이 추출되어, 단일 필드의 키워드 값으로 저장됩니다. 모든 값이 키워드로 저장된다는 점이 핵심입니다. 이 덕분에 term, terms, prefix, match, exists 같은 키워드 기반 쿼리를 사용할 수 있습니다. 특정 서브필드를 지정한 검색도 가능합니다.

```json
GET /issues/_search
{
  "query": {
    "term": { "labels.release": "v1.3.0" }
  }
}
```

이 쿼리는 labels 오브젝트 안에서 release 키의 값이 "v1.3.0"인 문서를 찾습니다. 매핑에 labels.release라는 필드가 명시적으로 등록되어 있지 않아도, flattened 타입 내부에서 서브필드 경로를 인식하여 검색합니다.

다만 flattened 타입에는 제한사항이 있습니다. 모든 값이 키워드로 저장되므로 숫자 범위 쿼리가 동작하지 않습니다. 예를 들어 labels.count 값이 숫자 100이라 해도, 문자열 "100"으로 저장되기 때문에 100 이상인 문서를 찾는 범위 검색은 의미 있는 결과를 반환하지 않습니다. 하이라이팅도 지원되지 않습니다. 집계는 term 집계만 가능하고, 다른 종류의 집계는 제한됩니다. 따라서 flattened 타입은 정밀한 분석이나 수치 연산이 필요하지 않은 메타데이터성 오브젝트에 적합합니다. Kubernetes 파드 레이블, 애플리케이션 설정 맵, 이벤트 메타데이터 등 키 구조가 런타임에 결정되는 오브젝트가 대표적인 사용 사례입니다.

flattened 타입이 "구조를 무시하고 하나로 묶는" 접근이라면, 반대로 "허용되지 않은 필드를 아예 거부하는" 접근도 있습니다. 이것이 **dynamic: strict** 설정입니다.

2.1.1에서 다룬 것처럼 dynamic 설정은 매핑에 정의되지 않은 새 필드가 포함된 문서를 어떻게 처리할지를 결정합니다. dynamic 설정에는 네 가지 값이 있습니다.

```
dynamic 설정값 비교

설정값      동작                          매핑 추가   검색 가능   문서 저장
true        필드 자동 추가 (기본값)        O          O          O
runtime     런타임 필드로 추가             O          O          O
false       알 수 없는 필드 무시           X          X          O (_source에만)
strict      예외 발생, 문서 거절           X          X          X
```

기본값인 true는 새 필드를 자동으로 매핑에 추가합니다. 이것이 매핑 폭발의 원인입니다. false로 설정하면 알 수 없는 필드가 매핑에 추가되지 않고, _source에만 저장됩니다. 검색은 불가능하지만 문서는 저장됩니다. strict로 설정하면 한 단계 더 나아가, 매핑에 없는 필드가 포함된 문서 자체를 거부합니다. 예외가 발생하고 문서 전체가 인덱싱되지 않습니다.

strict 설정은 스키마가 완전히 확정된 프로덕션 인덱스에 적합합니다. 주문 데이터, 회원 정보처럼 필드 구조가 명확한 데이터에서, 예상치 못한 필드가 들어오는 것은 데이터 오류나 파이프라인 버그를 의미합니다. strict를 설정하면 이런 문제를 인덱싱 시점에 즉시 감지할 수 있습니다.

실전에서는 알려진 필드를 명시적 매핑으로 정의하고, dynamic: strict를 함께 설정하는 조합을 사용합니다.

```json
PUT /orders
{
  "mappings": {
    "dynamic": "strict",
    "properties": {
      "order_id":   { "type": "keyword" },
      "customer":   { "type": "keyword" },
      "amount":     { "type": "double" },
      "created_at": { "type": "date" }
    }
  }
}
```

이 매핑은 order_id, customer, amount, created_at 네 가지 필드만 허용합니다. 이 네 필드 외의 데이터가 포함된 문서가 인덱싱을 시도하면 예외가 발생합니다. 필드 목록이 고정되므로 매핑 폭발이 원천적으로 차단됩니다.

인덱스 전체에 strict를 적용하기 어려운 경우도 있습니다. 일부 오브젝트 필드에만 선택적으로 strict를 적용할 수 있습니다. 인덱스 루트에는 dynamic: true를 두고, 특정 오브젝트 필드에만 dynamic: strict를 중첩 설정하면, 해당 오브젝트 안에서만 새 필드 추가가 차단됩니다.

지금까지 flattened 타입과 dynamic: strict를 살펴보았습니다. 이 두 기법은 각각 "구조를 뭉쳐서 필드 수를 줄이는" 방법과 "새 필드 추가를 차단하는" 방법입니다. 그런데 동적 매핑 자체를 완전히 끄지 않으면서, 새로 추가되는 필드의 타입을 통제하고 싶은 경우가 있습니다. 이때 사용하는 것이 **동적 매핑 템플릿**(dynamic_templates)입니다.

동적 매핑 템플릿은 동적으로 추가되는 필드에 대해 이름 패턴, 타입, 경로 조건으로 매핑 규칙을 사전에 정의하는 기능입니다. 새 필드가 자동 생성될 때, 조건에 맞는 첫 번째 템플릿이 적용됩니다. 템플릿이 정의된 순서가 중요합니다. 여러 템플릿의 조건에 동시에 해당하면, 배열에서 앞에 위치한 템플릿이 우선합니다.

동적 매핑 템플릿이 매핑 폭발 방지에 기여하는 대표적인 방식이 있습니다. Elasticsearch의 기본 동적 매핑에서 문자열 값은 text 타입과 keyword 서브필드를 동시에 생성합니다. 하나의 문자열 필드가 매퍼 2개를 차지하는 것입니다. 동적 템플릿으로 모든 문자열 필드를 keyword 단일 타입으로 매핑하면, text+keyword 멀티필드 자동 생성을 막아 필드 카운팅을 절반으로 줄일 수 있습니다.

```json
PUT /logs
{
  "mappings": {
    "dynamic_templates": [
      {
        "strings_as_keyword": {
          "match_mapping_type": "string",
          "mapping": { "type": "keyword" }
        }
      }
    ]
  }
}
```

이 템플릿은 동적으로 추가되는 모든 문자열 필드를 keyword 타입으로 매핑합니다. match_mapping_type은 JSON 값의 타입을 기준으로 조건을 지정합니다. "string"으로 설정하면 문자열 값을 가진 필드에 적용됩니다. 이 템플릿이 적용된 인덱스에서는 새 문자열 필드가 text+keyword 멀티필드 대신 keyword 단일 타입으로 등록되므로, 동일한 데이터를 인덱싱해도 매퍼 수가 절반으로 줄어듭니다.

동적 템플릿에서 필드 이름 패턴을 사용하면 더 정밀한 제어가 가능합니다. **match**는 필드명 패턴을 지정하고, **unmatch**는 제외할 패턴을 지정합니다.

```json
{
  "long_fields": {
    "match": "long_*",
    "unmatch": "*_text",
    "mapping": { "type": "long" }
  }
}
```

이 템플릿은 이름이 "long_"으로 시작하되 "_text"로 끝나지 않는 필드에 적용됩니다. 해당 필드는 long 타입으로 매핑됩니다.

match와 unmatch는 기본적으로 와일드카드 패턴을 사용합니다. 더 정밀한 매칭이 필요하면 match_pattern을 "regex"로 설정하여 Java 정규식을 사용할 수 있습니다.

```json
{
  "match_pattern": "regex",
  "match": "^profit_\\d+$",
  "mapping": { "type": "double" }
}
```

이 템플릿은 "profit_" 뒤에 숫자만 오는 필드명에 적용됩니다. profit_1, profit_23 같은 필드는 double 타입으로 매핑됩니다.

오브젝트 내부의 중첩 경로를 기준으로 조건을 지정할 수도 있습니다. **path_match**와 **path_unmatch**를 사용합니다.

```json
{
  "path_match": "name.*",
  "path_unmatch": "*.middle",
  "mapping": { "type": "text", "copy_to": "full_name" }
}
```

이 템플릿은 name 오브젝트 아래의 모든 필드에 적용되되, middle이라는 이름의 필드는 제외합니다. name.first, name.last는 text 타입으로 매핑되고 full_name으로 복사되지만, name.middle은 이 템플릿을 건너뛰고 다음 템플릿이나 기본 규칙을 따릅니다.

match에 배열을 지정하여 여러 패턴을 동시에 적용하는 것도 가능합니다. "match": ["ip_*", "*_ip"]로 설정하면 ip_로 시작하거나 _ip로 끝나는 필드 모두에 같은 매핑 규칙이 적용됩니다.

지금까지 개별 기법을 하나씩 살펴보았습니다. 실무에서는 데이터의 특성에 따라 이 기법들을 조합하여 사용합니다. 구조화되지 않은 로그 데이터를 처리할 때의 전략을 정리합니다.

스키마가 완전히 확정된 비즈니스 데이터라면, dynamic: strict와 명시적 매핑의 조합이 적합합니다. 앞서 본 orders 인덱스 예시가 이에 해당합니다. 허용된 필드만 인덱싱하고, 나머지는 거부합니다.

알려진 필드와 임의 메타데이터 오브젝트가 함께 존재하는 경우에는, strict와 flattened 타입을 조합합니다.

```json
PUT /app-logs
{
  "mappings": {
    "dynamic": "strict",
    "properties": {
      "@timestamp": { "type": "date" },
      "message":    { "type": "text" },
      "level":      { "type": "keyword" },
      "labels":     { "type": "flattened" },
      "metadata":   { "type": "flattened" }
    }
  }
}
```

이 매핑에서 @timestamp, message, level은 알려진 필드로 명시적으로 정의합니다. labels와 metadata는 어떤 키가 들어올지 예측할 수 없으므로 flattened 타입으로 선언합니다. dynamic: strict가 설정되어 있으므로, 이 다섯 가지 필드 외의 최상위 필드가 들어오면 거부됩니다. labels나 metadata 내부에는 임의의 키-값 쌍이 자유롭게 들어올 수 있지만, 매핑에는 각각 하나의 매퍼로만 존재하므로 필드 수가 폭발하지 않습니다.

반구조화 로그처럼 키 집합이 어느 정도 예측 가능하지만 완전히 고정되지는 않은 경우에는, 동적 매핑 템플릿으로 타입을 강제하는 전략을 사용합니다.

```json
PUT /events
{
  "mappings": {
    "dynamic_templates": [
      {
        "all_strings_as_keyword": {
          "match_mapping_type": "string",
          "mapping": { "type": "keyword", "ignore_above": 256 }
        }
      }
    ]
  }
}
```

이 템플릿은 동적으로 추가되는 모든 문자열 필드를 keyword 단일 타입으로 제한합니다. ignore_above: 256은 256자를 초과하는 값은 색인하지 않겠다는 설정입니다. text+keyword 멀티필드가 생성되지 않으므로 매퍼 수가 절반으로 줄어듭니다. 필드 자체는 계속 추가될 수 있지만, 타입 제어를 통해 증가 속도를 억제합니다.

키 집합이 완전히 예측 불가능한 비구조화 로그라면, flattened 타입과 total_fields.limit 상향을 조합합니다. 임의 키를 담는 오브젝트를 flattened로 격리하여 필드 수 증가를 차단하고, 나머지 알려진 필드를 위해 total_fields.limit를 적절히 설정합니다.

이 전략들을 표로 정리하면 다음과 같습니다.

```
데이터 유형별 매핑 폭발 방지 전략

데이터 유형                                    권장 전략
스키마 완전히 확정된 비즈니스 데이터            dynamic: strict + 명시적 매핑
알려진 필드 + 임의 메타데이터 오브젝트          strict + flattened 타입 격리
반구조화 로그 (키 집합 예측 가능)               동적 매핑 템플릿으로 타입 강제
완전 비구조화 로그 (키 무제한)                  flattened 타입 + total_fields.limit 상향
```

정리하면, 매핑 폭발은 동적 매핑 환경에서 필드 수가 통제 없이 늘어나 클러스터 메모리를 압박하는 현상입니다. index.mapping.total_fields.limit로 필드 수 상한을 설정하여 감지할 수 있고, flattened 타입으로 임의 오브젝트를 단일 매퍼로 묶어 필드 수 증가를 차단할 수 있습니다. dynamic: strict로 허용되지 않은 필드를 거부하여 매핑 변경 자체를 막을 수 있고, 동적 매핑 템플릿으로 새 필드의 타입을 통제하여 매퍼 수 증가를 억제할 수 있습니다. 데이터 특성에 따라 이 기법들을 조합하는 것이 실무에서의 표준 접근입니다.

다음 단원인 2.2.1에서는 text vs keyword 필드 타입을 다룹니다.

이 단원을 마치면 매핑 폭발이 발생하는 원인과 위험을 설명할 수 있고, flattened 타입과 dynamic: strict로 매핑 폭발을 방지할 수 있습니다.
