# 6.3.3 JVM/파일시스템 최적화

6.3.1에서 다룬 인덱싱 최적화와 6.3.2의 검색 최적화는 Elasticsearch 설정 수준의 튜닝이었습니다. 하지만 Elasticsearch는 JVM 위에서 동작하고, 데이터 파일은 운영체제의 파일시스템을 통해 읽고 씁니다. JVM 힙 크기가 부적절하거나 OS 레벨 설정이 맞지 않으면, 아무리 Elasticsearch 설정을 조정해도 성능이 개선되지 않습니다. 이 단원에서는 JVM과 운영체제 수준의 최적화를 다룹니다.

Elasticsearch에서 JVM 힙 크기 설정은 가장 중요한 인프라 결정 중 하나입니다. 핵심 원칙은 물리 메모리의 절반 이하를 JVM 힙에 할당하고, 나머지 절반은 운영체제의 파일시스템 캐시(페이지 캐시)에 남겨 두는 것입니다.

이 원칙의 배경을 이해하려면 Elasticsearch의 메모리 사용 구조를 알아야 합니다. Elasticsearch는 내부적으로 Lucene을 사용하며, Lucene의 세그먼트 파일은 운영체제의 페이지 캐시에 올라가야 빠르게 읽을 수 있습니다. 페이지 캐시는 JVM 힙 외부의 시스템 메모리를 사용합니다. JVM 힙에 메모리를 과도하게 할당하면 페이지 캐시에 남는 메모리가 줄어들고, 세그먼트 파일을 디스크에서 직접 읽어야 하는 빈도가 늘어납니다. 이는 검색 성능 저하로 이어집니다.

힙 크기의 상한선은 **31GB**입니다. 이 수치는 JVM의 **Compressed OOP(Compressed Ordinary Object Pointer)** 메커니즘과 관련됩니다. JVM은 힙 크기가 약 32GB 미만일 때 객체 포인터를 32비트로 압축하여 메모리를 효율적으로 사용합니다. 힙이 32GB 이상이 되면 64비트 포인터를 사용해야 하므로, 포인터 자체가 차지하는 메모리가 두 배로 늘어납니다. 실질적으로 32GB 힙이 28~30GB 힙보다 효율이 떨어지는 역전 현상이 발생합니다. 안전 마진을 두어 31GB 이하로 설정하는 것이 관례입니다.

힙 크기는 jvm.options 파일에서 설정합니다. -Xms와 -Xmx를 동일한 값으로 지정합니다.

```
-Xms16g
-Xmx16g
```

두 값을 같게 설정하는 이유는 힙 크기가 동적으로 변하는 과정에서 발생하는 오버헤드를 방지하기 위해서입니다. 예를 들어 물리 메모리가 64GB인 서버라면 힙을 31GB로 설정하고, 나머지 33GB를 페이지 캐시에 남깁니다.

JVM의 **가비지 컬렉터(GC)** 선택도 성능에 영향을 줍니다. Elasticsearch 8.x에서는 기본적으로 **G1GC**를 사용합니다. G1GC는 힙을 여러 리전으로 나누어 부분적으로 수집하므로, 대형 힙에서도 긴 GC 정지(stop-the-world) 시간을 줄여 줍니다. 대부분의 운영 환경에서 G1GC가 적합합니다.

**ZGC**는 JDK 15 이후 사용할 수 있는 가비지 컬렉터로, GC 정지 시간을 밀리초 이하로 유지하는 것을 목표로 합니다. 지연 시간에 극도로 민감한 환경에서 고려할 수 있습니다. 다만 G1GC 대비 처리량이 다소 낮을 수 있고, Elasticsearch와의 호환성을 사전에 확인해야 합니다.

힙 메모리가 스왑 아웃되면 GC 성능이 크게 저하됩니다. 운영체제가 JVM 힙의 일부를 디스크의 스왑 영역으로 내보내면, GC가 해당 영역을 접근할 때 디스크 I/O가 발생하여 GC 시간이 수 배로 늘어날 수 있습니다. **bootstrap.memory_lock**을 true로 설정하면 Elasticsearch 프로세스의 메모리가 스왑되지 않도록 잠급니다.

```yaml
bootstrap.memory_lock: true
```

이 설정은 elasticsearch.yml에 추가합니다. Linux에서는 이 설정이 동작하려면 memlock unlimited가 프로세스 제한에 설정되어 있어야 합니다.

운영체제 레벨에서 확인해야 할 설정이 몇 가지 있습니다.

**파일 디스크립터 제한**은 Elasticsearch가 동시에 열 수 있는 파일 수를 결정합니다. Lucene 세그먼트 파일, 네트워크 소켓, 로그 파일 등을 합치면 상당한 수의 파일 디스크립터가 필요합니다. Linux의 기본값은 보통 1024인데, 이는 Elasticsearch 운영에 부족합니다. 최소 65535 이상으로 설정해야 합니다.

```bash
ulimit -n 65535
```

영구적으로 적용하려면 /etc/security/limits.conf에 설정을 추가합니다.

**vm.max_map_count** 설정은 프로세스가 사용할 수 있는 메모리 매핑 영역의 최대 수를 결정합니다. Elasticsearch는 메모리 맵 파일(mmap)을 사용하여 Lucene 인덱스 파일에 접근합니다. 기본값인 65530은 부족할 수 있으므로 최소 262144로 설정합니다.

```bash
sysctl -w vm.max_map_count=262144
```

영구 적용을 위해 /etc/sysctl.conf에 해당 값을 추가합니다. 이 설정이 누락되면 Elasticsearch가 시작 시 오류를 발생시킵니다.

스토리지 선택도 성능에 큰 영향을 줍니다. **SSD**는 HDD보다 랜덤 읽기와 쓰기 성능이 월등히 뛰어나므로, 특히 Hot 티어 노드에서는 SSD 사용을 강력히 권장합니다. Elasticsearch는 검색 시 여러 세그먼트 파일을 동시에 읽는 패턴을 보이는데, 이는 순차 읽기보다 랜덤 읽기에 가까워 SSD의 장점이 극대화됩니다.

Linux에서 SSD를 사용할 때 I/O 스케줄러를 **noop** 또는 **none**으로 설정하면 성능이 향상됩니다. HDD용으로 설계된 기본 스케줄러(cfq 등)는 SSD에서 불필요한 오버헤드를 추가합니다.

**NUMA(Non-Uniform Memory Access)** 아키텍처의 서버에서는 추가 설정이 필요합니다. NUMA 시스템에서는 CPU 소켓마다 가까운 메모리와 먼 메모리가 구분됩니다. JVM 프로세스가 먼 메모리에 접근하면 지연이 발생합니다. Elasticsearch를 실행할 때 numactl --interleave=all 옵션을 사용하면, 메모리 할당을 모든 NUMA 노드에 균등하게 분배하여 특정 노드의 메모리 병목을 방지합니다.

정리하면, JVM 힙은 물리 메모리의 절반 이하, 최대 31GB로 설정하여 Compressed OOP의 이점을 유지하고 페이지 캐시에 충분한 메모리를 남깁니다. bootstrap.memory_lock으로 스왑을 방지하고, 파일 디스크립터를 65535 이상으로, vm.max_map_count를 262144 이상으로 설정합니다. SSD를 사용하면서 I/O 스케줄러를 조정하고, NUMA 시스템에서는 메모리 인터리빙을 적용합니다.

다음 단원인 6.4.1에서는 TLS, RBAC, API Key를 활용한 보안 설정을 다룹니다. 프로덕션 클러스터를 안전하게 운영하기 위한 인증과 암호화 방법을 알아봅니다.

이 단원을 마치면 JVM heap 크기와 GC 설정을 올바르게 조정할 수 있으며, OS 레벨 파일시스템 설정 최적화를 수행할 수 있습니다.
