# 3.4.1 BM25 알고리즘

Elasticsearch에 검색 질의를 보내면 결과 문서마다 `_score`라는 숫자가 붙어 옵니다. 이 점수가 높은 문서가 결과 목록의 위쪽에 나타납니다. 사용자가 입력한 검색어와 문서가 얼마나 관련 있는지를 수치로 매기는 작업을 **스코어링**(scoring)이라 합니다. 스코어링 결과가 곧 검색 결과의 순서를 결정하므로, 검색 엔진의 품질은 스코어링 알고리즘에 크게 좌우됩니다.

1.2.2에서 역색인 내부의 텀 딕셔너리와 포스팅 리스트를 살펴보면서, 텀 빈도(TF)와 도큐먼트 빈도(DF)라는 두 수치가 검색 점수 계산에 쓰인다고 설명했습니다. TF는 특정 문서 안에서 해당 단어가 몇 번 등장하는지, DF는 전체 문서 집합에서 해당 단어가 등장하는 문서의 수입니다. 이 두 수치를 어떤 방식으로 결합하느냐에 따라 스코어링 알고리즘이 달라집니다. 이 단원에서는 Elasticsearch가 기본으로 사용하는 BM25 알고리즘이 TF와 DF를 어떻게 결합하는지, 그리고 이전 세대 알고리즘인 TF-IDF와 무엇이 다른지를 살펴봅니다.

BM25를 이해하려면 먼저 그 선행 알고리즘인 **TF-IDF**가 어떤 발상에서 출발했는지를 아는 것이 도움이 됩니다. TF-IDF는 두 가지 직관을 수식으로 옮긴 것입니다. 첫 번째 직관은 "문서 안에 검색어가 많이 등장할수록 관련성이 높다"는 것이고, 두 번째 직관은 "거의 모든 문서에 등장하는 흔한 단어는 특정 문서를 구별하는 데 도움이 되지 않는다"는 것입니다.

첫 번째 직관을 반영하는 부분이 TF, 즉 텀 빈도입니다. "검색 엔진"이라는 질의를 보냈을 때, "검색"이 5번 등장하는 문서는 1번 등장하는 문서보다 점수가 높아야 합니다. 두 번째 직관을 반영하는 부분이 **IDF**(Inverse Document Frequency, 역문서 빈도)입니다. "the elephant"라는 검색어가 있다고 합시다. "the"는 거의 모든 문서에 등장합니다. 반면 "elephant"는 소수의 문서에만 등장합니다. 이 경우 "elephant"가 검색 결과를 구별하는 데 훨씬 유용합니다. IDF는 이런 차이를 수치로 반영합니다. DF(해당 단어가 등장하는 문서 수)가 작을수록 IDF 값이 커져서, 드문 단어에 더 큰 가중치를 부여합니다.

Elasticsearch 5.0 이전에는 이 TF-IDF를 기본 스코어링 알고리즘으로 사용했습니다. TF-IDF는 단순하고 직관적이지만, 한 가지 근본적인 문제가 있습니다. TF 값에 상한이 없다는 것입니다. 어떤 문서에 특정 단어가 1,000번 등장하면, 10번 등장하는 문서보다 점수가 100배 가까이 높아집니다. 실제로는 같은 단어가 1,000번 반복되었다고 해서 10번 반복된 경우보다 100배 더 관련성이 높은 것은 아닙니다. 특정 키워드를 의도적으로 반복한 스팸 문서가 부당하게 높은 점수를 받을 수도 있습니다.

**BM25**(Best Matching 25)는 이 문제를 해결한 알고리즘입니다. BM25는 TF-IDF의 두 가지 직관을 그대로 계승하되, TF가 점수에 미치는 영향에 천장을 두는 장치를 추가합니다. 이 장치를 **TF 포화**(term frequency saturation)라고 합니다. Elasticsearch 5.0부터 BM25가 기본 유사도 알고리즘으로 채택되었습니다.

TF 포화의 동작을 구체적으로 살펴봅니다. TF-IDF에서는 단어가 1번 등장할 때와 100번 등장할 때의 점수 차이가 그대로 100배입니다. BM25에서는 단어가 처음 몇 번 등장할 때 점수가 급격히 올라가지만, 등장 횟수가 늘어날수록 점수 증가폭이 줄어들어 결국 일정 수준에서 거의 멈춥니다. 이 동작을 포화 곡선(saturation curve)이라 부릅니다. 아래 그림은 단어 등장 횟수에 따른 점수 기여를 TF-IDF와 BM25에서 비교한 것입니다.

```
점수 기여
  |
  |                                          TF-IDF (상한 없음)
  |                                        /
  |                                      /
  |                                    /
  |                                  /
  |                                /
  |            .................. /........  BM25 (포화)
  |          /                 /
  |        /                 /
  |      /                 /
  |    /                 /
  |  /                 /
  |/                 /
  +-------------------------------------------> 단어 등장 횟수 (TF)
  0    5    10   15   20   25   30
```

TF-IDF의 직선은 TF가 커질수록 점수도 계속 올라갑니다. 반면 BM25의 곡선은 TF가 5~10 정도를 넘어가면 점수 증가가 눈에 띄게 둔화되어, 결국 수평에 가까워집니다.

BM25의 전체 공식은 검색어에 포함된 각 단어의 점수를 합산하는 구조입니다. 검색어가 "검색 엔진"이라면, "검색"에 대한 점수와 "엔진"에 대한 점수를 각각 계산한 뒤 더합니다. 한 단어에 대한 점수 계산은 크게 세 부분으로 나뉩니다. IDF 부분, TF 포화 부분, 그리고 **문서 길이 정규화** 부분입니다.

IDF 부분은 앞서 설명한 역문서 빈도입니다. 전체 문서 수와 해당 단어가 등장하는 문서 수를 이용하여 드문 단어에 높은 가중치를, 흔한 단어에 낮은 가중치를 부여합니다.

TF 포화 부분은 문서 안에서 해당 단어가 등장한 횟수를 기반으로 점수를 매기되, 횟수가 늘어나도 점수가 일정 수준 이상 올라가지 않게 합니다. 이 포화 속도를 제어하는 파라미터가 **k1**입니다. Elasticsearch에서 k1의 기본값은 1.2입니다. k1 값이 클수록 포화가 느려져 추가 등장이 점수에 더 많이 반영되고, k1 값이 작을수록 포화가 빨라져 단어가 몇 번만 나와도 금방 천장에 도달합니다.

문서 길이 정규화 부분은 문서의 길이가 점수에 미치는 영향을 조정합니다. 동일한 단어가 한 번 등장했더라도, 짧은 문서에서의 한 번과 긴 문서에서의 한 번은 의미가 다릅니다. 10단어짜리 문서에 검색어가 한 번 나타나면 그 문서의 상당 부분이 검색어와 관련된 것입니다. 그러나 10,000단어짜리 문서에 검색어가 한 번 나타나면 검색어와의 관련성은 훨씬 낮습니다. BM25는 해당 문서의 필드 길이를 전체 문서의 평균 필드 길이로 나누어 이 차이를 반영합니다. 이 정규화의 강도를 조절하는 파라미터가 **b**입니다. Elasticsearch에서 b의 기본값은 0.75입니다. b를 0으로 설정하면 문서 길이를 전혀 고려하지 않습니다. b를 1로 설정하면 길이 정규화를 최대한 강하게 적용합니다.

세 부분이 하나의 점수로 결합되는 흐름을 정리하면 다음과 같습니다.

```
BM25 점수 계산 흐름 (검색어 단어 하나에 대해)

[1단계] IDF 계산
         전체 문서 수(N)와 해당 단어 포함 문서 수(DF) --> IDF 값

[2단계] TF 포화 적용
         문서 내 단어 등장 횟수(TF) + k1 파라미터 --> 포화된 TF 값

[3단계] 문서 길이 정규화
         해당 문서 필드 길이 / 평균 필드 길이 + b 파라미터
         --> 2단계의 포화된 TF 값에 길이 보정 적용

[4단계] 최종 점수
         IDF x (길이 보정이 적용된 포화 TF) --> 해당 단어의 점수 기여분

* 검색어에 단어가 여러 개이면, 각 단어의 점수 기여분을 모두 합산합니다.
```

1단계에서 IDF를 구합니다. 드문 단어일수록 IDF가 높습니다. 2단계에서 문서 내 등장 횟수(TF)에 k1을 적용하여 포화 곡선을 만듭니다. 3단계에서 문서 길이를 평균과 비교하여 보정합니다. 평균보다 긴 문서는 점수가 깎이고, 짧은 문서는 점수가 올라갑니다. b가 이 보정의 강도를 제어합니다. 4단계에서 IDF와 보정된 TF를 곱하여 최종 점수를 산출합니다.

구체적인 예를 들어봅니다. 인덱스에 총 1,000개의 문서가 있고, "elasticsearch"라는 단어가 50개 문서에 등장한다고 합시다. 이 경우 "elasticsearch"의 DF는 50이고, IDF는 높은 편입니다. 이제 두 문서를 비교합니다.

문서 A는 제목 필드가 "elasticsearch tutorial"로, 필드 길이가 2단어이고, "elasticsearch"가 1번 등장합니다. 문서 B는 제목 필드가 300단어짜리 긴 글로, "elasticsearch"가 1번 등장합니다. 전체 문서의 평균 제목 길이가 5단어라면, 문서 A는 평균보다 짧으므로 길이 정규화에서 유리합니다. 문서 B는 평균보다 훨씬 기므로 점수가 깎입니다. 결과적으로 문서 A가 문서 B보다 높은 `_score`를 받습니다.

Elasticsearch는 BM25를 기본 유사도 알고리즘으로 사용하므로, 별도 설정 없이도 모든 text 필드에 BM25가 적용됩니다. 그러나 특정 필드에서 k1이나 b 값을 조정하고 싶거나, BM25가 아닌 다른 유사도 알고리즘을 사용하고 싶다면 인덱스 매핑에서 similarity 설정을 변경할 수 있습니다.

아래는 인덱스를 생성할 때 커스텀 BM25 설정을 적용하는 예입니다.

```json
PUT my_index
{
  "settings": {
    "similarity": {
      "my_bm25": {
        "type": "BM25",
        "k1": 1.5,
        "b": 0.85
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "similarity": "my_bm25"
      }
    }
  }
}
```

settings 안의 similarity 블록에서 "my_bm25"라는 이름으로 커스텀 유사도를 정의합니다. type을 "BM25"로 지정하고, k1과 b 값을 원하는 값으로 설정합니다. 이 예에서는 k1을 1.5로 높여 TF 포화를 늦추고, b를 0.85로 높여 문서 길이 정규화를 기본값보다 강하게 적용합니다. mappings에서는 title 필드의 similarity 속성에 "my_bm25"를 지정하여, 이 필드의 스코어링에 커스텀 설정이 사용되도록 합니다.

Elasticsearch는 BM25 외에도 몇 가지 대안 유사도 알고리즘을 제공합니다. DFR(Divergence From Randomness), DFI(Divergence From Independence), LMDirichlet(Language Model with Dirichlet smoothing) 등이 있습니다. 이들은 각각 다른 확률 모델에 기반하며, similarity 설정의 type 값을 변경하여 적용할 수 있습니다. 다만 대부분의 사용 사례에서 BM25의 기본 설정이 충분한 성능을 보이므로, 대안 알고리즘은 특수한 요구 사항이 있을 때 검토합니다.

정리하면, BM25는 IDF, TF 포화, 문서 길이 정규화라는 세 가지 요소를 결합하여 검색 점수를 산출하는 알고리즘입니다. TF-IDF가 단어 등장 횟수에 상한 없이 점수를 올렸던 문제를 TF 포화로 해결했고, 문서 길이에 따른 불공정함을 정규화로 보정합니다. Elasticsearch는 5.0부터 BM25를 기본 유사도로 사용하며, k1(기본 1.2)과 b(기본 0.75)라는 두 파라미터로 동작을 조정할 수 있습니다.
