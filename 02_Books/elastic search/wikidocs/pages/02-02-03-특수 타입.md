# 2.2.3 특수 타입: dense_vector, geo_point, nested, join

2.2.2에서 숫자, 날짜, 불리언처럼 하나의 값을 저장하는 기본 타입을 살펴보았습니다. 이 타입들은 BKD Tree와 doc_values라는 자료구조를 활용하여 범위 검색, 정렬, 집계를 효율적으로 처리합니다. 하지만 실제 서비스에서는 기본 타입만으로 표현하기 어려운 데이터가 있습니다. 사용자의 검색 의도를 숫자 배열로 변환한 벡터 임베딩, 매장이나 배송지의 위도와 경도, 한 문서 안에 여러 개의 하위 객체를 독립적으로 검색해야 하는 경우, 문서 사이에 부모-자식 관계를 설정해야 하는 경우가 그렇습니다. Elasticsearch는 이런 상황에 대응하는 특수 필드 타입을 제공합니다. 이 단원에서는 벡터를 저장하는 dense_vector, 지리 좌표를 저장하는 geo_point, 객체 배열의 독립성을 보존하는 nested, 문서 간 부모-자식 관계를 정의하는 join 타입을 중심으로 살펴봅니다. 이어서 IP 주소, 바이너리 데이터, 자동완성 제안처럼 특정 용도에 맞춰 설계된 타입도 함께 다룹니다.

먼저 **dense_vector** 타입을 살펴봅니다. dense_vector를 이해하려면 벡터 임베딩이라는 개념을 먼저 알아야 합니다. 머신러닝 모델은 텍스트, 이미지 같은 데이터를 고정 길이의 숫자 배열로 변환할 수 있습니다. 예를 들어 "강아지 사료 추천"이라는 문장을 모델에 넣으면 [0.12, -0.34, 0.56, ...] 같은 수백 개의 실수로 이루어진 배열이 나옵니다. 이 숫자 배열을 벡터 임베딩이라고 부릅니다. 의미가 비슷한 문장은 벡터 공간에서 가까운 위치에 놓이고, 의미가 다른 문장은 먼 위치에 놓입니다. 이 성질을 이용하면 키워드가 정확히 일치하지 않아도 의미가 유사한 문서를 찾을 수 있습니다. 이것이 벡터 검색(vector search)의 핵심 원리입니다.

dense_vector 타입은 이 벡터 임베딩을 Elasticsearch에 저장하기 위한 필드 타입입니다. "dense"라는 이름은 배열의 대부분 위치에 0이 아닌 값이 채워져 있다는 뜻입니다. 반대로 대부분이 0이고 일부만 값이 있는 배열은 sparse_vector라고 부릅니다.

dense_vector 필드를 매핑할 때 가장 중요한 설정은 **dims**입니다. dims는 벡터의 차원 수, 즉 배열의 길이를 지정합니다. 사용하는 임베딩 모델이 768차원 벡터를 출력한다면 dims를 768로 설정해야 합니다.

```json
{
  "mappings": {
    "properties": {
      "content_embedding": {
        "type": "dense_vector",
        "dims": 768,
        "index": true,
        "similarity": "dot_product"
      }
    }
  }
}
```

content_embedding 필드를 dense_vector 타입으로 정의했습니다. dims를 768로 설정하여 768개의 실수를 저장하겠다고 선언했습니다. index를 true로 설정하면 Elasticsearch가 이 벡터를 검색 가능한 형태로 색인합니다. similarity는 두 벡터 사이의 거리를 측정하는 방식을 지정합니다. dot_product는 두 벡터의 내적을 계산하는 방식으로, cosine 유사도보다 2~3배 빠릅니다. cosine은 벡터의 크기를 정규화한 뒤 방향만 비교하는 방식이고, 테스트나 탐색 단계에서 사용하기 적합합니다. 운영 환경에서는 dot_product가 권장됩니다.

Elasticsearch는 dense_vector 필드에 색인된 벡터를 **HNSW**(Hierarchical Navigable Small World) 알고리즘을 사용하여 그래프 구조로 저장합니다. HNSW는 벡터를 여러 계층의 그래프 노드로 배치하여, 전체를 순회하지 않고도 가장 가까운 벡터를 빠르게 찾아내는 근사 알고리즘입니다. 벡터 수가 늘어나도 검색 시간이 로그 스케일로 증가하기 때문에 대규모 데이터에서도 실용적입니다.

이 벡터를 검색할 때는 kNN(k-Nearest Neighbors) 쿼리를 사용합니다. kNN은 쿼리 벡터와 가장 가까운 k개의 문서를 반환하는 검색 방식입니다. Elasticsearch는 두 가지 kNN 방식을 지원합니다. 근사 kNN(approximate kNN)은 HNSW 그래프를 탐색하여 빠르게 결과를 반환하며, 대부분의 운영 환경에 적합합니다. 정확 kNN(exact kNN)은 모든 벡터를 하나씩 비교하는 방식으로, 필터 적용 후 후보 문서가 10,000건 이하일 때 사용합니다.

벡터를 저장할 때 메모리도 고려해야 합니다. dense_vector 필드는 검색 성능을 위해 데이터 노드의 오프힙 메모리(off-heap RAM)에 올라갑니다. 필요한 메모리는 다음 공식으로 추정합니다.

```
필요 메모리(바이트) = 벡터 수 x 4 x (차원 수 + 12)
```

예를 들어 2,000만 건의 768차원 벡터를 저장하면, 2,000만 x 4 x (768 + 12) = 약 62.4GB의 오프힙 메모리가 필요합니다. 복제본(replica)이 하나 있으면 이 양이 두 배가 됩니다. 클러스터를 설계할 때 이 계산을 미리 해두어야 메모리 부족으로 성능이 저하되는 상황을 피할 수 있습니다.

다음으로 **geo_point** 타입을 살펴봅니다. 음식 배달 앱에서 "내 위치에서 3km 이내의 식당"을 찾거나, 부동산 서비스에서 특정 지역 안에 있는 매물을 검색하려면 문서에 위치 정보를 저장해야 합니다. 지구상의 특정 위치는 위도(latitude)와 경도(longitude) 두 숫자의 쌍으로 표현합니다. 위도는 적도를 기준으로 남북 위치를 나타내는 -90에서 90 사이의 값이고, 경도는 본초 자오선을 기준으로 동서 위치를 나타내는 -180에서 180 사이의 값입니다. geo_point 타입은 이 위도/경도 쌍을 저장합니다.

매핑은 다음과 같이 정의합니다.

```json
{
  "mappings": {
    "properties": {
      "location": {
        "type": "geo_point"
      }
    }
  }
}
```

geo_point 필드에 값을 넣는 방법은 여러 가지입니다. 가장 명확한 방식은 lat과 lon을 명시하는 객체 형태입니다.

```json
{
  "location": {
    "lat": 37.5665,
    "lon": 126.9780
  }
}
```

이 예시는 서울 시청의 좌표를 저장한 것입니다. lat에 위도 37.5665를, lon에 경도 126.9780을 지정했습니다. 이 외에도 문자열("37.5665,126.9780"), 배열([126.9780, 37.5665]), WKT 포맷("POINT(126.9780 37.5665)") 등의 형태로 입력할 수 있습니다. 배열과 WKT 포맷에서는 경도가 위도보다 앞에 온다는 점에 주의해야 합니다. GeoJSON 규격이 경도-위도 순서를 사용하기 때문입니다.

geo_point 필드에서 사용할 수 있는 대표적인 쿼리를 살펴봅니다. **geo_distance** 쿼리는 지정한 좌표로부터 일정 거리 이내에 있는 문서를 찾습니다.

```json
{
  "query": {
    "geo_distance": {
      "distance": "3km",
      "location": {
        "lat": 37.5665,
        "lon": 126.9780
      }
    }
  }
}
```

이 쿼리는 서울 시청 좌표로부터 반경 3km 이내에 위치한 문서를 반환합니다. distance에는 "200km", "10mi"(마일) 같은 단위를 붙여 지정합니다.

**geo_bounding_box** 쿼리는 사각형 영역 안에 있는 문서를 찾습니다. 좌상단(top_left)과 우하단(bottom_right)의 좌표로 사각형을 정의합니다. 지도 화면에서 현재 보이는 영역 안의 문서만 가져오는 데 적합합니다.

geo_point 필드에서는 거리 기반 집계도 가능합니다. 예를 들어 "0~1km, 1~5km, 5~10km" 같은 거리 구간별로 문서 수를 세는 버킷 집계를 만들 수 있습니다.

이제 **nested** 타입을 살펴봅니다. nested 타입을 이해하려면 먼저 Elasticsearch가 일반 객체 배열을 어떻게 저장하는지 알아야 합니다. 다음과 같은 문서가 있다고 가정합니다.

```json
{
  "name": "홍길동",
  "contacts": [
    {"type": "email", "value": "hong@example.com"},
    {"type": "phone", "value": "010-1234-5678"}
  ]
}
```

contacts 필드는 두 개의 객체를 담은 배열입니다. 각 객체에는 type과 value가 쌍을 이루고 있습니다. 그런데 Elasticsearch는 일반 object 타입으로 이 배열을 저장할 때, 내부적으로 배열을 평탄화(flatten)합니다. 평탄화란 각 필드의 값을 별도의 배열로 분리하는 것입니다.

```
contacts.type  : ["email", "phone"]
contacts.value : ["hong@example.com", "010-1234-5678"]
```

이렇게 되면 type과 value 사이의 짝이 사라집니다. "type이 email이면서 value가 010-1234-5678인 문서"를 검색하면, 실제로 그런 조합은 없는데도 이 문서가 검색 결과에 포함됩니다. type 배열에 "email"이 있고, value 배열에 "010-1234-5678"이 있으므로 조건을 각각 만족하기 때문입니다. 이것이 객체 배열의 평탄화 문제입니다.

**nested** 타입은 이 문제를 해결합니다. nested로 선언된 필드는 배열의 각 객체를 내부적으로 별도의 숨겨진 문서(hidden document)로 저장합니다. 각 객체가 독립된 문서로 색인되므로, type과 value의 짝이 보존됩니다.

매핑은 다음과 같이 정의합니다.

```json
{
  "mappings": {
    "properties": {
      "contacts": {
        "type": "nested"
      }
    }
  }
}
```

contacts 필드를 nested 타입으로 선언했습니다. 이제 contacts 배열의 각 객체는 독립적으로 색인됩니다.

nested 필드를 검색할 때는 반드시 **nested 쿼리**를 사용해야 합니다. 일반 match 쿼리로는 nested 필드 안의 값을 찾을 수 없습니다.

```json
{
  "query": {
    "nested": {
      "path": "contacts",
      "query": {
        "bool": {
          "must": [
            {"match": {"contacts.type": "email"}},
            {"match": {"contacts.value": "hong@example.com"}}
          ]
        }
      }
    }
  }
}
```

path에 nested 필드의 경로를 지정하고, query에 검색 조건을 넣습니다. 이 쿼리는 contacts 배열에서 type이 "email"이면서 동시에 value가 "hong@example.com"인 객체가 있는 문서만 반환합니다. 평탄화되지 않았기 때문에 필드 사이의 짝이 정확히 유지됩니다.

nested 타입을 사용할 때 알아두어야 할 제약이 있습니다. nested 객체를 수정하면 부모 문서 전체를 다시 색인해야 합니다. 배열에서 객체 하나만 바꾸더라도 문서 전체가 재색인됩니다. 또한 하나의 문서에 저장할 수 있는 nested 객체의 수는 기본 10,000개로 제한됩니다. 이 제한은 index.mapping.nested_objects.limit 설정으로 조정할 수 있지만, nested 객체가 많아지면 메모리 사용량과 색인 속도에 영향을 줍니다.

이제 **join** 타입을 살펴봅니다. nested가 한 문서 안에 하위 객체를 포함하는 방식이라면, join은 별도의 문서를 부모-자식 관계로 연결하는 방식입니다. 예를 들어 게시판에서 게시글과 댓글을 관리한다고 가정합니다. 게시글 하나에 댓글이 수백 개 달릴 수 있고, 댓글은 게시글과 별개로 자주 추가됩니다. 이 경우 댓글을 게시글 문서의 nested 배열에 넣으면, 댓글이 추가될 때마다 게시글 문서 전체를 재색인해야 합니다. join 타입은 게시글과 댓글을 각각 독립된 문서로 저장하면서도, 두 문서 사이에 부모-자식 관계를 설정하여 함께 검색할 수 있게 합니다.

join 필드를 매핑할 때는 관계의 이름을 정의합니다.

```json
{
  "mappings": {
    "properties": {
      "relation_type": {
        "type": "join",
        "relations": {
          "post": "comment"
        }
      }
    }
  }
}
```

relation_type 필드를 join 타입으로 정의하고, relations에서 "post"가 부모이고 "comment"가 자식임을 선언했습니다. 부모와 자식은 모두 같은 인덱스에 저장됩니다.

문서를 색인할 때는 어떤 역할(부모인지 자식인지)인지를 명시합니다.

```json
PUT /forum/_doc/1
{
  "title": "Elasticsearch 질문",
  "relation_type": {
    "name": "post"
  }
}

PUT /forum/_doc/100?routing=1
{
  "body": "이렇게 해보세요.",
  "relation_type": {
    "name": "comment",
    "parent": 1
  }
}
```

첫 번째 문서는 게시글(post)입니다. relation_type의 name을 "post"로 지정하여 부모 역할임을 나타냅니다. 두 번째 문서는 댓글(comment)입니다. name을 "comment"로, parent를 1로 지정하여 문서 1의 자식임을 나타냅니다. routing 파라미터에 부모 문서의 ID를 넣은 것에 주목합니다. Elasticsearch는 부모와 자식 문서를 같은 샤드에 저장해야 관계 쿼리를 처리할 수 있으므로, 자식 문서를 색인할 때 부모의 ID를 라우팅 값으로 지정하여 같은 샤드에 배치합니다.

부모-자식 관계를 검색할 때는 **has_parent**와 **has_child** 쿼리를 사용합니다. has_parent 쿼리는 부모 문서의 조건을 기준으로 자식 문서를 찾습니다. has_child 쿼리는 자식 문서의 조건을 기준으로 부모 문서를 찾습니다.

```json
{
  "query": {
    "has_parent": {
      "parent_type": "post",
      "query": {
        "match": {"title": "Elasticsearch"}
      }
    }
  }
}
```

이 쿼리는 제목에 "Elasticsearch"가 포함된 게시글의 댓글을 반환합니다.

join 타입에도 알아두어야 할 제약이 있습니다. 자식 문서는 하나의 부모만 가질 수 있습니다. 부모-자식 관계를 유지하기 위해 Elasticsearch는 global ordinals라는 내부 자료구조를 사용하는데, 이 구조를 구축하는 데 추가 메모리와 시간이 소요됩니다. 따라서 join 쿼리는 같은 조건의 일반 쿼리보다 느립니다.

nested와 join 중 어떤 것을 선택해야 하는지 정리합니다.

```
선택 기준            nested                    join
-----------        ----------------          ----------------
저장 위치            부모 문서 안에 포함          별도 문서로 분리
수정 빈도            하위 객체 수정 시            자식 문서만 독립적으로
                    부모 전체 재색인             수정 가능
하위 데이터 규모      수천 개 이하 적합            대량 자식 문서에 적합
쿼리 성능            일반적으로 빠름              global ordinals 오버헤드
```

하위 객체의 수가 적고 자주 변경되지 않는다면 nested가 적합합니다. 자식 데이터가 독립적으로 자주 추가되거나 수정되고, 부모당 자식 수가 매우 많다면 join이 적합합니다.

이제 나머지 특수 타입을 살펴봅니다. **ip** 타입은 IPv4와 IPv6 주소를 저장합니다. 서버 접속 로그에서 클라이언트 IP를 기록하거나, 특정 IP 대역에서 발생한 이벤트를 검색할 때 사용합니다. ip 타입은 CIDR 표기법을 이용한 범위 검색을 지원합니다. 예를 들어 "192.168.1.0/24" 범위에 속하는 IP를 가진 문서를 검색할 수 있습니다. 단순히 IP를 문자열(keyword)로 저장하면 이런 대역 기반 검색이 불가능합니다.

**binary** 타입은 Base64로 인코딩된 이진 데이터를 저장합니다. Base64란 이진 데이터를 텍스트 문자로 변환하는 인코딩 방식입니다. 파일의 내용이나 이미지 썸네일 같은 이진 데이터를 문서에 첨부할 때 사용합니다. binary 필드는 검색이나 집계의 대상이 되지 않으며, 저장과 조회만 가능합니다. 문서와 함께 이진 데이터를 보관해야 하지만 검색 조건으로 사용할 필요가 없을 때 적합합니다.

**completion** 타입은 자동완성 기능을 위해 설계된 특수 타입입니다. 검색창에 글자를 입력할 때마다 추천 검색어를 보여주는 기능을 구현할 때 사용합니다. completion 타입이 다른 타입과 다른 점은 내부 자료구조입니다. 일반적인 역색인이나 BKD Tree 대신 **FST**(Finite State Transducer)라는 메모리 상주 자료구조를 사용합니다. FST는 트라이(Trie) 자료구조와 비슷하게, 입력된 문자를 한 글자씩 따라가며 후보를 찾아냅니다. 전체 데이터를 메모리에 올려두므로 응답이 매우 빠릅니다.

매핑과 색인 예시를 봅니다.

```json
PUT /search_terms
{
  "mappings": {
    "properties": {
      "suggest": {
        "type": "completion"
      }
    }
  }
}

PUT /search_terms/_doc/1
{
  "suggest": {
    "input": ["elasticsearch", "elastic search"],
    "weight": 10
  }
}
```

suggest 필드를 completion 타입으로 정의했습니다. 색인할 때 input에 자동완성 후보 문자열을 배열로 넣고, weight에 우선순위를 숫자로 지정합니다. weight가 높을수록 추천 결과에서 앞에 표시됩니다. input 배열에 여러 변형을 넣으면, "elasticsearch"를 입력해도, "elastic search"를 입력해도 이 문서가 추천됩니다.

completion 타입은 첫 글자부터 순서대로 일치하는 방식으로 동작합니다. "search"를 입력해서 "elasticsearch"를 찾을 수는 없습니다. 중간 일치가 필요하다면 edge n-gram이나 search_as_you_type 같은 다른 접근법을 사용해야 합니다.

정리하면, dense_vector는 벡터 임베딩을 저장하여 의미 기반 유사도 검색을 가능하게 하며, dims 설정으로 차원 수를 지정하고 HNSW 알고리즘으로 빠른 근사 검색을 수행합니다. geo_point는 위도/경도 좌표를 저장하여 거리 검색과 영역 검색을 지원합니다. nested는 객체 배열을 독립적으로 색인하여 필드 간의 짝을 보존하고, join은 별도 문서 사이에 부모-자식 관계를 설정하여 독립적 수정과 관계 검색을 모두 가능하게 합니다. ip, binary, completion 타입은 각각 IP 주소, 이진 데이터, 자동완성이라는 특정 용도에 맞게 설계된 타입입니다.

다음 단원인 2.2.4에서는 멀티필드 패턴을 다룹니다.

이 단원을 마치면 dense_vector 타입으로 벡터 임베딩을 저장하고 dims를 설정할 수 있으며, nested와 join 타입의 사용 목적과 적합한 상황을 비교하여 설명할 수 있습니다.
