# 8.3.1 Logs + Metrics + APM 통합

8.2에서 로그를 수집하고 Kibana로 분석하는 방법을 다뤘습니다. 로그만으로도 시스템의 많은 문제를 진단할 수 있지만, 한계가 있습니다. 웹 서버 로그에서 500 에러가 급증하는 것을 발견했다고 합시다. 에러의 원인이 애플리케이션 코드의 버그인지, 서버 CPU가 100%에 도달한 것인지, 데이터베이스 연결이 끊긴 것인지는 로그만으로 판단하기 어렵습니다. CPU 사용률 같은 인프라 지표, 데이터베이스 쿼리 시간 같은 애플리케이션 성능 데이터를 함께 봐야 원인을 정확히 파악할 수 있습니다.

이처럼 로그(Logs), 인프라 지표(Metrics), 애플리케이션 성능 데이터(APM)를 하나의 플랫폼에서 통합 분석하는 것을 **Observability**(관측 가능성)라고 합니다. Observability는 시스템의 외부 출력(로그, 지표, 트레이스)을 관찰하여 내부 상태를 이해하는 능력을 뜻합니다. 단순히 "서버가 살아 있는가"를 확인하는 모니터링과 달리, "왜 느려졌는가", "어디에서 병목이 발생했는가"까지 추적할 수 있는 것이 Observability의 목표입니다.

Elastic은 이 세 가지 데이터를 하나의 Elasticsearch 클러스터에 통합하는 **Elastic Observability** 솔루션을 제공합니다. 이 단원에서는 Elastic Observability의 구성요소를 살펴보고, Logs, Metrics, APM 데이터를 연계하여 분석하는 방법을 다룹니다. 6.2.2에서 다룬 Beats 에코시스템과 8.2.1에서 구축한 로그 수집 파이프라인이 기반이 됩니다.

Elastic Observability는 세 가지 핵심 데이터 소스로 구성됩니다.

첫째, **Logs**는 8.2에서 다룬 로그 데이터입니다. 서버, 애플리케이션, 네트워크 장비가 남기는 텍스트 메시지입니다. "언제 무슨 일이 일어났는가"를 기록합니다.

둘째, **Metrics**는 시스템과 서비스의 상태를 수치로 나타낸 지표입니다. CPU 사용률 85%, 메모리 사용량 12GB, 디스크 I/O 초당 500회 같은 수치가 주기적으로 수집됩니다. "시스템의 현재 상태가 어떤가"를 보여 줍니다.

셋째, **APM(Application Performance Monitoring)**은 애플리케이션 코드 수준의 성능 데이터입니다. 사용자 요청이 애플리케이션 내부에서 어떤 경로를 거치는지, 각 단계에서 얼마나 시간이 걸렸는지를 추적합니다. "요청이 왜 느려졌는가"를 코드 수준에서 답합니다.

이 세 가지를 한곳에 모으면, 에러 로그를 발견한 뒤 같은 시간대의 CPU 지표를 확인하고, 해당 요청이 애플리케이션 내부에서 어느 함수에서 지연되었는지까지 추적할 수 있습니다.

```
[Logs]                 [Metrics]              [APM]
  |                       |                      |
  v                       v                      v
+----------------------------------------------------------+
|                 Elasticsearch 클러스터                      |
|  logs-*  |  metrics-*  |  traces-*  |  apm-*             |
+----------------------------------------------------------+
                          |
                          v
              Kibana Observability UI
         (통합 분석, 상관관계 추적, 알림)
```

먼저 인프라 메트릭 수집부터 살펴봅니다. 6.2.2에서 다룬 **Metricbeat**가 이 역할을 합니다. Metricbeat는 서버의 CPU, 메모리, 디스크, 네트워크 지표를 주기적으로 수집하여 Elasticsearch로 전송합니다.

Metricbeat의 system 모듈을 활성화하면 운영체제 수준의 기본 지표를 수집할 수 있습니다.

```yaml
metricbeat.modules:
- module: system
  metricsets:
    - cpu
    - memory
    - network
    - filesystem
    - process
  period: 10s
  processes: ['.*']
```

metricsets에 수집할 지표 종류를 지정합니다. cpu는 CPU 사용률, memory는 메모리 사용량, network는 네트워크 트래픽, filesystem은 디스크 사용량, process는 프로세스별 리소스 사용을 수집합니다. period를 10s로 설정하면 10초마다 지표를 수집합니다.

Docker나 Kubernetes 환경에서는 해당 모듈을 추가로 활성화합니다. docker 모듈은 컨테이너별 CPU, 메모리 사용을, kubernetes 모듈은 Pod, Node, Deployment별 리소스 사용을 수집합니다.

다음으로 APM 데이터 수집을 살펴봅니다. **APM 에이전트**는 애플리케이션 코드에 설치하는 라이브러리입니다. Java, Python, Node.js, Go, .NET 등 다양한 언어에 대응하는 에이전트가 제공됩니다. 에이전트를 설치하면 애플리케이션이 처리하는 각 요청에 대해 자동으로 성능 데이터를 수집합니다.

APM 에이전트가 수집하는 핵심 데이터는 **트랜잭션(transaction)**과 **스팬(span)**입니다.

**트랜잭션**은 사용자 요청 하나에 대응합니다. 예를 들어 HTTP 요청이 들어와서 응답이 나갈 때까지의 전체 처리가 하나의 트랜잭션입니다. 트랜잭션에는 요청 URL, HTTP 메서드, 응답 시간, 상태 코드 등이 기록됩니다.

**스팬**은 트랜잭션 내부의 개별 작업을 나타냅니다. 데이터베이스 쿼리, 외부 API 호출, 캐시 조회 등이 각각 하나의 스팬이 됩니다. 스팬에는 작업 유형, 소요 시간, 대상 서비스가 기록됩니다.

마이크로서비스 환경에서는 하나의 사용자 요청이 여러 서비스를 거칩니다. API 게이트웨이가 요청을 받아 인증 서비스를 호출하고, 주문 서비스를 호출하며, 주문 서비스는 다시 결제 서비스를 호출하는 식입니다. 이 전체 경로를 추적하는 것이 **분산 추적(distributed tracing)**입니다.

분산 추적은 **trace_id**와 **span_id**라는 두 가지 식별자로 동작합니다.

**trace_id**는 하나의 사용자 요청 전체를 식별하는 고유 ID입니다. 요청이 처음 들어올 때 생성되어, 이 요청이 거치는 모든 서비스에 전파됩니다. 어떤 서비스에서 생성된 트랜잭션이든, 같은 trace_id를 가지면 하나의 사용자 요청에 속하는 것입니다.

**span_id**는 각 스팬(개별 작업)을 식별하는 고유 ID입니다. 스팬은 parent_id를 통해 부모 스팬과 연결됩니다. 이 부모-자식 관계를 따라가면 요청이 서비스 간에 어떤 순서로 흘렀는지, 각 단계에서 얼마나 걸렸는지를 트리 구조로 볼 수 있습니다.

```
사용자 요청 (trace_id: abc123)
  |
  |-- API Gateway (span_id: s1)
  |     |-- 인증 서비스 호출 (span_id: s2, parent: s1)
  |     |-- 주문 서비스 호출 (span_id: s3, parent: s1)
  |           |-- DB 쿼리 (span_id: s4, parent: s3)
  |           |-- 결제 서비스 호출 (span_id: s5, parent: s3)
  |                 |-- 외부 PG 연동 (span_id: s6, parent: s5)
```

이 트리에서 결제 서비스 호출(s5)에 3초가 걸렸다면, 그 안의 외부 PG 연동(s6)이 2.8초를 차지하고 있는 것을 확인할 수 있습니다. 병목 지점이 외부 PG 연동에 있다는 것을 정확히 파악할 수 있습니다.

Kibana의 APM UI에서는 이 분산 추적 데이터를 시각적으로 볼 수 있습니다. **서비스 맵(Service Map)**은 서비스 간의 호출 관계를 그래프로 보여 줍니다. 각 서비스가 노드로, 서비스 간 호출이 연결선으로 표시됩니다. 연결선 위에 평균 응답 시간과 에러율이 표시되어, 어느 서비스 간 연결에서 문제가 발생하는지 한눈에 파악할 수 있습니다.

이제 Logs, Metrics, APM 세 가지 데이터를 연계하여 분석하는 방법을 살펴봅니다. Kibana Observability UI에서는 **Logs 탐색기와 APM 트레이스를 연계**할 수 있습니다. APM UI에서 느린 트랜잭션을 발견하면, 해당 트랜잭션의 trace_id를 기준으로 같은 시간대에 발생한 로그를 자동으로 필터링하여 보여 줍니다. 에러 로그의 스택 트레이스와 APM 트레이스를 함께 보면, 코드의 어느 지점에서 오류가 발생했는지 정확히 추적할 수 있습니다.

Metrics 데이터와의 연계도 가능합니다. APM에서 특정 시간대에 응답 시간이 급증한 것을 발견하면, 같은 시간대의 인프라 메트릭(CPU, 메모리, 디스크 I/O)을 함께 확인합니다. CPU 사용률이 95%에 도달해 있었다면, 응답 지연의 원인이 리소스 부족임을 알 수 있습니다.

Elastic Observability는 **SLO(Service Level Objective)** 설정도 지원합니다. SLO는 서비스가 달성해야 하는 성능 목표를 수치로 정의한 것입니다. 예를 들어 "API 응답의 99%가 200ms 이내여야 한다"는 SLO입니다. Kibana에서 SLO를 설정하면, 현재 달성률을 실시간으로 추적합니다. 달성률이 목표 아래로 떨어지면 경고를 발생시켜, 문제가 사용자에게 영향을 주기 전에 대응할 수 있습니다.

**Uptime 모니터링**은 서비스가 정상적으로 응답하는지 외부에서 주기적으로 확인하는 기능입니다. 6.2.2에서 다룬 Heartbeat가 이 역할을 수행합니다. Heartbeat가 지정한 URL에 주기적으로 HTTP 요청을 보내고, 응답 여부와 응답 시간을 기록합니다. Kibana의 Uptime UI에서 각 서비스의 가용성 현황을 한눈에 볼 수 있습니다. 서비스가 응답하지 않으면 즉시 알림이 발생합니다.

정리하면, Elastic Observability는 Logs(Filebeat/Logstash), Metrics(Metricbeat), APM(APM 에이전트)의 세 가지 데이터를 하나의 Elasticsearch 클러스터에 통합합니다. 분산 추적으로 요청 경로를 trace_id와 span_id로 추적하고, 서비스 맵으로 서비스 간 관계를 시각화합니다. Logs, Metrics, APM 데이터를 연계하면 "무엇이 발생했고, 시스템 상태는 어땠으며, 코드의 어디가 느렸는지"를 하나의 화면에서 파악할 수 있습니다. SLO와 Uptime 모니터링으로 서비스 품질을 지속적으로 관리합니다.

다음 단원인 8.3.2에서는 이 Observability 데이터를 효율적으로 관리하기 위한 Data Stream 설계와 보존 정책을 다룹니다.

이 단원을 마치면 Elastic Observability에서 Logs, Metrics, APM 데이터를 통합 분석할 수 있습니다.
