# 5.3.1 CCS 설정

5.1.2에서 하나의 클러스터 안에서 마스터 선출과 노드 간 통신이 어떻게 이루어지는지를 살펴보았습니다. 그런데 실무에서는 클러스터가 하나가 아닌 경우가 많습니다. 지역별로 별도 클러스터를 운영하거나, 팀별로 독립된 클러스터를 사용하면서도, 여러 클러스터의 데이터를 한 번에 검색해야 하는 상황이 생깁니다. 이때 사용하는 기능이 **Cross-Cluster Search**, 줄여서 **CCS**입니다.

CCS가 없다면 여러 클러스터의 데이터를 검색하려면 각 클러스터에 개별적으로 요청을 보내고, 애플리케이션 레벨에서 결과를 합쳐야 합니다. 클러스터 수가 늘어나면 이 과정이 복잡해집니다. CCS는 하나의 검색 요청으로 로컬 클러스터와 원격 클러스터의 데이터를 함께 조회할 수 있게 해 줍니다. Elasticsearch가 내부적으로 원격 클러스터에 요청을 전달하고, 결과를 합쳐서 반환합니다.

CCS를 사용하려면 먼저 **원격 클러스터를 등록**해야 합니다. 로컬 클러스터에서 원격 클러스터의 접속 정보를 설정하는 것입니다. 클러스터 설정 업데이트 API를 사용합니다.

```json
PUT _cluster/settings
{
  "persistent": {
    "cluster": {
      "remote": {
        "cluster-east": {
          "seeds": ["east-node-1:9300", "east-node-2:9300"]
        }
      }
    }
  }
}
```

이 요청을 구성 요소별로 풀어 봅니다. "persistent"는 이 설정이 클러스터 재시작 후에도 유지된다는 뜻입니다. "cluster.remote"는 원격 클러스터를 정의하는 네임스페이스입니다. "cluster-east"는 이 원격 클러스터에 부여하는 이름입니다. 이 이름은 자유롭게 정할 수 있으며, 이후 CCS 쿼리에서 이 이름으로 클러스터를 참조합니다.

"seeds"는 원격 클러스터에 접속할 때 처음 연결을 시도할 노드 주소 목록입니다. 포트 번호 9300은 Elasticsearch 노드 간 통신에 사용하는 전송 포트(transport port)입니다. 9200은 클라이언트 HTTP 요청용 포트이고, 9300은 노드 간 내부 통신용 포트입니다. CCS 연결은 노드 간 통신이므로 9300번을 사용합니다.

원격 클러스터에 접속하는 로컬 노드는 5.1.1에서 다룬 remote_cluster_client 역할을 가지고 있어야 합니다. 이 역할이 없는 노드는 원격 클러스터에 연결할 수 없습니다.

각 클러스터는 고유한 **cluster_uuid**를 가집니다. 클러스터가 처음 형성될 때 자동으로 생성되는 식별자입니다. 원격 클러스터를 등록하면 Elasticsearch는 이 UUID를 기록하여, 이름은 같지만 다른 클러스터가 잘못 연결되는 것을 감지합니다.

등록이 완료되면 **CCS 쿼리**를 실행할 수 있습니다. CCS 쿼리의 핵심은 인덱스 이름 앞에 클러스터 이름을 붙이는 것입니다. 형식은 `{클러스터이름}:{인덱스이름}`입니다.

```json
GET cluster-east:logs-*/_search
{
  "query": {
    "match": {
      "message": "error"
    }
  }
}
```

이 요청은 cluster-east라는 원격 클러스터의 logs-*에 해당하는 인덱스에서 message 필드에 "error"가 포함된 문서를 검색합니다. 로컬 클러스터의 인덱스와 원격 클러스터의 인덱스를 함께 검색할 수도 있습니다.

```
GET logs-*,cluster-east:logs-*/_search
```

이 요청은 로컬 클러스터의 logs-*와 원격 클러스터 cluster-east의 logs-*를 동시에 검색합니다. 쉼표로 구분하여 여러 클러스터의 인덱스를 나열할 수 있습니다.

여러 원격 클러스터를 한 번에 검색하는 것도 가능합니다.

```
GET cluster-east:logs-*,cluster-west:logs-*/_search
```

CCS 응답에는 일반 검색 응답에 더해 **_clusters** 필드가 포함됩니다. 이 필드는 각 클러스터의 검색 실행 상태를 보여 줍니다.

```json
{
  "_clusters": {
    "total": 2,
    "successful": 2,
    "skipped": 0,
    "details": {
      "cluster-east": {
        "status": "successful",
        "indices": "logs-*",
        "took": 148
      },
      "(local)": {
        "status": "successful",
        "indices": "logs-*",
        "took": 42
      }
    }
  }
}
```

total은 검색에 참여한 클러스터 수입니다. successful은 성공한 클러스터 수입니다. skipped는 건너뛴 클러스터 수입니다. details 안에는 각 클러스터별로 상태와 소요 시간이 나옵니다. "(local)"은 로컬 클러스터를 뜻합니다.

여기서 **skip_unavailable** 설정이 중요합니다. 원격 클러스터에 연결할 수 없을 때 전체 검색이 실패할지, 해당 클러스터만 건너뛸지를 결정합니다.

```json
PUT _cluster/settings
{
  "persistent": {
    "cluster.remote.cluster-east.skip_unavailable": true
  }
}
```

true로 설정하면, cluster-east에 연결할 수 없어도 로컬 클러스터의 결과만 반환합니다. _clusters 필드에서 해당 클러스터가 skipped로 표시됩니다. false로 설정하면, 원격 클러스터 연결 실패 시 전체 검색이 오류를 반환합니다. Elasticsearch 8.15부터 기본값이 true로 변경되었습니다.

CCS에서 보안이 필요한 환경에서는 **Remote Cluster API Key**를 사용합니다. 원격 클러스터에 접근하려면 인증이 필요한데, API 키를 통해 로컬 클러스터가 원격 클러스터에 인증된 상태로 연결합니다. 원격 클러스터에서 API 키를 생성하고, 로컬 클러스터의 원격 클러스터 설정에 이 키를 등록하는 방식입니다.

마지막으로 **CCS와 스코어 정규화**에 대한 주의점을 짚습니다. 검색 관련성 점수(_score)는 샤드 로컬에서 계산됩니다. 같은 쿼리라도 로컬 클러스터의 샤드와 원격 클러스터의 샤드에서 문서 빈도(document frequency)가 다르면, 동일한 문서에 대해 다른 점수가 산출될 수 있습니다. 이는 CCS에만 국한된 문제가 아니라 분산 검색 전반의 특성이지만, CCS에서는 클러스터 간 데이터 분포 차이가 더 크므로 영향이 뚜렷해질 수 있습니다. 정밀한 점수 정규화가 필요한 경우에는 dfs_query_then_fetch 검색 유형을 사용하는 것을 고려합니다.

정리하면, CCS는 하나의 검색 요청으로 여러 클러스터의 데이터를 조회하는 기능입니다. cluster.remote 설정으로 원격 클러스터를 등록하고, {클러스터이름}:{인덱스이름} 형식으로 쿼리합니다. 응답의 _clusters 필드로 각 클러스터의 실행 상태를 확인할 수 있으며, skip_unavailable 설정으로 연결 실패 시 동작을 제어합니다.

다음 단원인 5.3.2에서는 원격 클러스터 연결의 두 가지 모드인 Sniff 모드와 Proxy 모드를 다룹니다. 네트워크 환경에 따라 어떤 모드를 선택해야 하는지를 살펴봅니다.

이 단원을 마치면 원격 클러스터를 등록하고 cross-cluster search 쿼리를 실행할 수 있습니다.
