# 2.2.4 멀티필드 패턴

2.2.1에서 text 타입은 전문 검색에, keyword 타입은 정확 일치, 정렬, 집계에 적합하다는 것을 살펴보았습니다. 실무에서는 하나의 필드가 두 가지 용도를 동시에 충족해야 하는 상황이 빈번합니다. 상품 이름을 예로 들면, 사용자가 "무선 이어폰"이라고 검색어를 입력했을 때 해당 상품을 찾을 수 있어야 하고, 동시에 상품 이름 가나다순으로 정렬하거나 이름별 집계도 수행해야 합니다. text 타입 하나만으로는 정렬과 집계가 불가능하고, keyword 타입 하나만으로는 전문 검색이 불가능합니다. 이 문제를 해결하는 방법이 멀티필드 패턴입니다.

**멀티필드**(multi-field)란 하나의 원본 필드에 서브필드를 추가하여, 같은 값을 서로 다른 방식으로 인덱싱하는 기능입니다. 원본 필드는 text 타입으로 분석기를 적용해 전문 검색을 지원하고, 서브필드는 keyword 타입으로 원문 그대로 저장해 정렬과 집계를 지원합니다. 문서를 인덱싱할 때 사용자가 값을 한 번만 보내면, Elasticsearch가 내부적으로 원본 필드와 서브필드에 각각 다른 처리를 적용하여 저장합니다.

멀티필드를 정의할 때 사용하는 것이 **fields 파라미터**입니다. 매핑에서 특정 필드의 정의 안에 `"fields"` 속성을 추가하고, 그 아래에 서브필드의 이름과 타입을 지정합니다. 구체적인 문법을 먼저 보겠습니다.

```json
PUT /products
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      }
    }
  }
}
```

이 매핑의 각 부분이 무엇을 뜻하는지 풀어 봅니다. `"title"`은 원본 필드의 이름입니다. `"type": "text"`는 원본 필드를 text 타입으로 지정하여 분석기를 적용합니다. `"fields"` 안에 정의된 `"keyword"`는 서브필드의 이름입니다. 서브필드의 `"type": "keyword"`는 이 서브필드가 원문을 그대로 저장하는 keyword 타입임을 뜻합니다. `"ignore_above": 256`은 256자를 초과하는 값은 이 서브필드에 인덱싱하지 않겠다는 설정입니다. 매우 긴 텍스트를 keyword로 저장하면 디스크와 메모리를 비효율적으로 소비하므로, 적절한 상한을 두는 것입니다.

이렇게 정의하면 title 필드에 "무선 블루투스 이어폰"이라는 값을 인덱싱할 때 내부적으로 두 가지 처리가 동시에 일어납니다. 원본 필드인 title에는 분석기가 적용되어 "무선", "블루투스", "이어폰" 같은 토큰이 역색인에 등록됩니다. 서브필드인 title.keyword에는 "무선 블루투스 이어폰" 원문 전체가 그대로 저장됩니다.

```
인덱싱 값: "무선 블루투스 이어폰"
        |
        +---> title (text)
        |       분석기 적용
        |       역색인: ["무선", "블루투스", "이어폰"]
        |       용도: 전문 검색 (match 쿼리)
        |
        +---> title.keyword (keyword)
                분석 없음
                역색인: ["무선 블루투스 이어폰"]
                용도: 정렬, 집계, 정확 일치 (term 쿼리)
```

서브필드의 이름이 반드시 "keyword"여야 하는 것은 아닙니다. "raw", "exact" 같은 다른 이름을 사용해도 됩니다. 다만 Elasticsearch 커뮤니티와 동적 매핑에서 관례적으로 "keyword"라는 이름을 사용하므로, 특별한 이유가 없다면 이 관례를 따르는 것이 혼란을 줄입니다. 이 관례에서 비롯된 것이 **title.keyword 패턴**입니다. "필드명.keyword"라는 접근 경로로 서브필드를 지정하는 방식은 Elasticsearch 전반에서 널리 쓰입니다.

이제 멀티필드를 활용하여 전문 검색과 정렬을 동시에 수행하는 구체적인 쿼리 패턴을 살펴봅니다. 전문 검색에는 원본 필드를, 정렬에는 서브필드를 지정합니다.

```json
GET /products/_search
{
  "query": {
    "match": {
      "title": "블루투스 이어폰"
    }
  },
  "sort": [
    { "title.keyword": "asc" }
  ]
}
```

이 요청에서 `"query"` 부분은 title 필드(text 타입)에서 "블루투스 이어폰"을 전문 검색합니다. 검색어가 분석기를 거쳐 "블루투스", "이어폰" 토큰으로 분리되고, 역색인에서 이 토큰들이 포함된 문서를 찾습니다. `"sort"` 부분은 검색 결과를 title.keyword 필드(keyword 타입)의 값으로 오름차순 정렬합니다. keyword 타입은 doc_values가 활성화되어 있으므로 정렬이 가능합니다. 만약 title 필드(text 타입)로 정렬을 시도하면, doc_values가 없으므로 오류가 발생합니다. 멀티필드 덕분에 하나의 값으로 검색과 정렬을 동시에 처리할 수 있는 것입니다.

집계도 같은 원리입니다. terms 집계를 수행하려면 title이 아니라 title.keyword를 대상으로 지정합니다.

```json
GET /products/_search
{
  "size": 0,
  "aggs": {
    "popular_titles": {
      "terms": {
        "field": "title.keyword"
      }
    }
  }
}
```

`"size": 0`은 검색 결과 문서는 생략하고 집계 결과만 반환하라는 의미입니다. `"field": "title.keyword"`가 집계 대상을 서브필드로 지정한 부분입니다. 이 집계는 title.keyword의 값별로 문서 수를 셉니다. title(text 타입) 필드를 대상으로 같은 집계를 실행하면, 2.2.1에서 설명한 것처럼 doc_values가 없다는 오류가 발생합니다.

지금까지 살펴본 패턴은 text 원본 필드에 keyword 서브필드를 추가하는 것이었습니다. 멀티필드의 활용은 여기서 더 확장됩니다. 서브필드에 **다른 분석기를 지정**하여 하나의 값을 여러 방식으로 분석할 수도 있습니다.

어떤 상황에서 이것이 필요한지 예를 들겠습니다. 영문 상품명 "Running Shoes"를 검색하는 상황을 생각해 봅니다. 기본 분석기(standard analyzer)는 이 문자열을 "running", "shoes"로 토크나이징합니다. 그런데 사용자가 "run"이라고 검색하면 "running"과 일치하지 않아 결과가 나오지 않습니다. 영어에서 "run"과 "running"은 같은 어근입니다. 이런 어근 변환을 처리하는 분석기가 있다면, "run"으로도 "running"이 포함된 문서를 찾을 수 있습니다.

이때 원본 필드에는 기본 분석기를, 서브필드에는 영어 분석기를 지정하여 두 가지 검색 방식을 동시에 지원할 수 있습니다.

```json
PUT /products
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "fields": {
          "english": {
            "type": "text",
            "analyzer": "english"
          },
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      }
    }
  }
}
```

이 매핑에서 title 필드는 기본 분석기로 처리됩니다. title.english 서브필드는 영어 분석기로 처리되어 어근 변환(stemming)을 적용합니다. title.keyword 서브필드는 원문 그대로 저장합니다. 하나의 원본 필드에 서브필드를 여러 개 둘 수 있으며, 각 서브필드가 독립적인 분석기를 가질 수 있습니다.

이 구성에서 쿼리할 때 어떤 필드를 대상으로 지정하느냐에 따라 검색 동작이 달라집니다. `"match": {"title": "running"}` 쿼리는 기본 분석기가 적용된 title 필드에서 검색합니다. `"match": {"title.english": "run"}` 쿼리는 영어 분석기가 적용된 title.english 서브필드에서 검색하며, "run"이 "running"의 어근과 일치하므로 결과를 반환합니다. `"term": {"title.keyword": "Running Shoes"}` 쿼리는 keyword 서브필드에서 원문 그대로 정확 일치 검색합니다. 이처럼 멀티필드의 쿼리 대상 지정은 필드명 뒤에 점(.)과 서브필드명을 붙이는 것이 전부입니다.

이제 동적 매핑과 멀티필드의 관계를 살펴봅니다. 2.2.1 도입부에서 동적 매핑이 문자열 값을 만나면 text 타입과 keyword 서브필드를 함께 생성한다고 언급했습니다. 이것이 바로 **동적 매핑의 기본 멀티필드 생성 동작**입니다.

매핑을 별도로 정의하지 않은 인덱스에 `{"title": "Quick Brown Fox"}` 문서를 인덱싱하면, Elasticsearch는 title이 문자열임을 감지하고 자동으로 다음과 같은 매핑을 생성합니다.

```json
{
  "title": {
    "type": "text",
    "fields": {
      "keyword": {
        "type": "keyword",
        "ignore_above": 256
      }
    }
  }
}
```

동적 매핑이 이렇게 동작하는 이유는 문자열 필드의 용도를 미리 알 수 없기 때문입니다. 전문 검색에 쓰일지, 정확 일치와 집계에 쓰일지 알 수 없으므로 양쪽 모두 가능하도록 text와 keyword를 함께 만드는 것입니다. 따라서 동적 매핑으로 생성된 문자열 필드에서는 별도의 매핑 변경 없이도 title로 전문 검색을, title.keyword로 정렬과 집계를 바로 수행할 수 있습니다.

다만 동적 매핑의 기본 멀티필드가 항상 최적인 것은 아닙니다. 상태 코드처럼 정확 일치만 필요한 필드에도 text 분석이 적용되어 불필요한 역색인 토큰이 생성됩니다. 반대로 이메일 본문처럼 전문 검색만 필요한 필드에도 keyword 서브필드가 생성되어 디스크를 낭비합니다. 프로덕션 환경에서는 필드 용도가 명확하므로, 명시적 매핑으로 필요한 타입만 정의하는 것이 효율적입니다.

마지막으로, 기존 인덱스에 멀티필드를 추가할 때 **reindex가 필요한지** 여부를 정리합니다. 결론부터 말하면, 기존 필드에 새 서브필드를 추가하는 것은 reindex 없이 가능합니다.

Elasticsearch는 기존 필드의 타입을 변경하는 것은 허용하지 않지만, 기존 필드에 새로운 서브필드를 추가하는 것은 허용합니다. 매핑 업데이트 API로 서브필드를 추가하면, 추가 시점 이후에 인덱싱되는 문서부터 새 서브필드가 적용됩니다. 이미 인덱싱된 기존 문서에는 새 서브필드의 값이 생성되지 않습니다.

이 동작을 단계별로 정리합니다.

```
1. 기존 인덱스에 title (text) 필드만 있는 상태
   기존 문서 100건이 인덱싱되어 있음

2. 매핑 업데이트로 title.keyword 서브필드 추가
   PUT /products/_mapping
   {
     "properties": {
       "title": {
         "type": "text",
         "fields": {
           "keyword": { "type": "keyword" }
         }
       }
     }
   }

3. 추가 이후 새로 인덱싱하는 문서
   --> title (text) + title.keyword (keyword) 모두 생성됨

4. 기존 100건의 문서
   --> title.keyword 값이 없음
   --> 이 문서들에도 서브필드를 적용하려면 reindex 필요
```

기존 문서까지 새 서브필드를 반영해야 한다면 reindex가 필요합니다. 그러나 새로 들어오는 문서에만 적용해도 충분한 경우에는 매핑 업데이트만으로 완료됩니다. 이것은 Elasticsearch가 이미 작성된 Lucene 세그먼트를 수정하지 않고, 새 세그먼트에만 변경된 매핑을 적용하기 때문입니다.

정리하면, 멀티필드는 하나의 원본 필드에 fields 파라미터로 서브필드를 추가하여, 같은 값을 서로 다른 방식으로 인덱싱하는 기능입니다. text와 keyword를 조합하면 전문 검색과 정렬, 집계를 동시에 지원할 수 있고, 서브필드마다 다른 분석기를 지정하면 여러 언어나 분석 전략을 하나의 필드에서 처리할 수 있습니다. 동적 매핑은 문자열 필드에 이 패턴을 자동으로 적용하며, 기존 인덱스에 서브필드를 추가할 때는 새 문서에만 적용되므로 기존 문서 반영이 필요하면 reindex를 수행해야 합니다.

다음 단원인 2.3.1에서는 분석기 파이프라인을 다룹니다.

이 단원을 마치면 fields 파라미터로 text와 keyword 멀티필드를 설정할 수 있고, 멀티필드를 활용한 전문 검색과 정렬 동시 지원 패턴을 구현할 수 있습니다.
