# 3.3.2 자동완성 구현

검색창에 글자를 입력하기 시작하면 입력 도중에 후보 목록이 나타나는 경험은 이미 익숙합니다. 쇼핑몰에서 "무선"만 입력해도 "무선 이어폰", "무선 충전기" 같은 제안이 즉시 뜹니다. 사용자가 검색어를 끝까지 입력하기 전에 결과를 보여 주는 이 기능을 **자동완성**(autocomplete)이라 합니다. 자동완성은 사용자 입력 한 글자마다 쿼리가 발생하므로, 일반 검색보다 훨씬 짧은 응답 시간이 요구됩니다. Elasticsearch는 이 요구를 만족시키기 위해 여러 접근 방식을 제공합니다.

이 단원에서는 Elasticsearch가 제공하는 자동완성 구현 방식을 하나씩 살펴봅니다. 3.2.2에서 다룬 match 쿼리와 match_phrase_prefix 쿼리, 그리고 2.2.4에서 다룬 멀티필드 패턴의 개념이 기반이 됩니다. 이 선수 개념을 바탕으로, Elasticsearch 전용 자동완성 도구인 completion suggester와 search_as_you_type 필드 타입, 그리고 인덱스 시점에 토큰을 미리 분해하는 edge_ngram 패턴까지 다룹니다.

가장 먼저, 자동완성의 핵심 과제를 이해할 필요가 있습니다. 일반 검색에서는 사용자가 검색어를 완성한 뒤 "검색" 버튼을 누릅니다. 이때 서버는 완성된 검색어를 받아 한 번만 쿼리를 실행하면 됩니다. 자동완성은 다릅니다. 사용자가 "무선"이라고 입력하는 동안 "ㅁ", "무", "문", "무ㅅ", "무선"처럼 글자가 바뀔 때마다 쿼리가 발생합니다. 한 단어를 입력하는 동안 수십 회의 쿼리가 실행될 수 있습니다. 각 쿼리의 응답이 수백 밀리초 이상 걸리면 사용자는 입력과 제안 사이에 지연을 느끼게 됩니다. 따라서 자동완성 쿼리는 수 밀리초 이내에 응답해야 하고, 이를 위해 일반 검색과는 다른 전략이 필요합니다.

Elasticsearch에서 자동완성을 구현하는 방식은 크게 네 가지로 나뉩니다. 각 방식은 "언제 작업을 하느냐"와 "어떤 자료 구조를 쓰느냐"에 따라 구분됩니다. 첫째는 3.2.2에서 이미 다룬 match_phrase_prefix 쿼리를 활용하는 방식입니다. 둘째는 completion suggester라는 전용 제안 기능입니다. 셋째는 search_as_you_type이라는 내장 필드 타입입니다. 넷째는 edge_ngram 토크나이저로 인덱싱 시점에 접두사를 미리 생성해 두는 방식입니다. 각 방식의 구조와 동작을 순서대로 살펴봅니다.

3.2.2에서 **match_phrase_prefix** 쿼리를 다룬 것을 떠올려 봅니다. 이 쿼리는 입력된 검색어를 분석기에 통과시켜 토큰을 만들고, 마지막 토큰을 접두사로 취급합니다. "quick bro"를 입력하면 "quick"은 정확히 매칭하고, "bro"로 시작하는 모든 토큰("brown", "broken", "broth" 등)을 찾습니다. 이 동작 자체가 자동완성의 기본 원리입니다. 사용자가 아직 단어를 다 입력하지 않았으므로 마지막 토큰이 불완전하고, 그 불완전한 토큰을 접두사로 간주하여 후보를 찾는 것입니다.

match_phrase_prefix로 자동완성을 구현하는 것은 별도의 매핑 변경 없이 기존 text 필드에 바로 적용할 수 있다는 장점이 있습니다. 그러나 이 방식은 쿼리 시점에 접두사 매칭을 수행합니다. 역색인에서 해당 접두사로 시작하는 토큰을 일일이 찾아야 하므로, 인덱스에 저장된 고유 토큰 수가 많을수록 응답 시간이 길어질 수 있습니다. 소규모 데이터에서는 충분히 빠르지만, 대규모 인덱스에서 밀리초 단위 응답이 필요한 실시간 자동완성에는 한계가 있을 수 있습니다.

이 한계를 근본적으로 해결하기 위해 Elasticsearch가 제공하는 전용 기능이 **completion suggester**입니다. completion suggester는 일반 검색 쿼리와 다른 별도의 제안(suggest) 경로를 통해 동작합니다. 일반 쿼리가 역색인을 조회하는 것과 달리, completion suggester는 **FST**(Finite State Transducer)라는 특수한 자료 구조를 사용합니다.

FST가 무엇인지 이해하려면 먼저 Trie 자료 구조를 떠올리면 됩니다. Trie는 문자열의 각 글자를 트리의 노드로 배치하여, 공통 접두사를 공유하는 문자열들을 효율적으로 저장하는 구조입니다. 예를 들어 "car", "card", "care"를 저장하면 "c-a-r"까지는 하나의 경로를 공유하고, 그 이후에 "d"와 "e"로 분기합니다. 접두사 "car"를 입력하면 이 경로를 따라 내려가는 것만으로 "car", "card", "care"를 모두 찾을 수 있습니다.

FST는 이 Trie 구조를 더 압축한 형태입니다. Trie가 접두사만 공유하는 것과 달리 FST는 접미사도 공유하여, 같은 데이터를 더 적은 메모리로 저장합니다. Elasticsearch의 completion suggester는 이 FST를 **메모리에 올려 둡니다**. 디스크가 아니라 메모리에서 직접 탐색하므로 접두사 검색 속도가 매우 빠릅니다. 사용자가 한 글자를 입력할 때마다 FST의 노드를 한 단계씩 따라가면서 후보를 찾으므로, 인덱스 크기와 무관하게 거의 일정한 속도로 응답할 수 있습니다.

completion suggester를 사용하려면 매핑에서 해당 필드의 타입을 **completion**으로 지정해야 합니다. 일반적인 text나 keyword 타입이 아니라, 자동완성 전용 타입입니다.

```json
PUT actors
{
  "mappings": {
    "properties": {
      "suggest": {
        "type": "completion"
      }
    }
  }
}
```

이 매핑에서 "suggest"라는 필드를 정의하고, 타입을 "completion"으로 지정합니다. 필드 이름은 "suggest"가 아닌 다른 이름을 사용해도 됩니다. 중요한 것은 타입이 "completion"이라는 점입니다. 이 타입으로 지정된 필드에 데이터를 인덱싱하면, Elasticsearch는 해당 값으로 FST를 구축하여 메모리에 보관합니다.

completion 필드에 문서를 인덱싱할 때는 일반 필드와 형태가 다릅니다. 자동완성 후보로 사용할 입력값과 가중치를 함께 지정합니다.

```json
PUT actors/_doc/1
{
  "suggest": {
    "input": ["Scarlett Johansson", "ScarJo"],
    "weight": 25
  }
}
```

"input" 필드는 자동완성 후보 문자열의 배열입니다. 하나의 문서에 여러 개의 후보를 등록할 수 있습니다. 위 예시에서는 "Scarlett Johansson"과 "ScarJo" 두 가지를 등록했으므로, 사용자가 "Scar"를 입력하면 두 후보 모두 매칭됩니다. **weight**는 이 제안의 우선순위를 결정하는 가중치입니다. 숫자가 클수록 결과 목록에서 먼저 나타납니다. 예를 들어 인기가 높은 배우의 weight를 25로, 덜 알려진 배우의 weight를 5로 설정하면, 같은 접두사를 공유하더라도 인기 배우가 먼저 제안됩니다.

인덱싱한 데이터를 자동완성으로 조회할 때는 일반 쿼리의 "query" 키가 아니라 **"suggest"** 키를 사용합니다. 이것이 completion suggester가 일반 검색과 다른 점입니다.

```json
POST actors/_search
{
  "suggest": {
    "actor-suggest": {
      "prefix": "scar",
      "completion": {
        "field": "suggest"
      }
    }
  }
}
```

이 요청의 구조를 하나씩 풀어 봅니다. 최상위의 "suggest"는 이 요청이 제안 기능임을 나타냅니다. "actor-suggest"는 이 제안 요청에 붙이는 이름으로, 응답에서 결과를 식별할 때 사용합니다. "prefix"에 사용자가 현재까지 입력한 문자열을 넣습니다. "completion" 안의 "field"에는 매핑에서 completion 타입으로 정의한 필드 이름을 지정합니다. Elasticsearch는 이 요청을 받으면 "suggest" 필드의 FST에서 "scar"로 시작하는 후보를 찾아 weight 내림차순으로 반환합니다.

completion suggester가 접두사를 매칭하는 방식에는 한 가지 제약이 있습니다. FST는 문자열의 첫 글자부터 순서대로 탐색하는 구조이므로, 입력값의 처음부터 일치해야 합니다. "Johansson"이라는 후보가 있을 때 "hans"를 입력하면 매칭되지 않습니다. "hans"는 문자열의 중간에 해당하기 때문입니다. 반드시 "Joh" 또는 "Johan"처럼 시작 부분부터 입력해야 합니다. 이 특성은 n-gram 기반 방식과의 핵심적인 차이입니다.

이제 세 번째 방식인 **search_as_you_type** 필드 타입을 살펴봅니다. 이 타입은 자동완성을 쉽게 구현하기 위해 Elasticsearch가 제공하는 내장 필드 타입입니다. completion 타입이 별도의 suggest API를 사용하는 것과 달리, search_as_you_type은 일반적인 match 쿼리나 multi_match 쿼리로 자동완성을 수행할 수 있습니다.

search_as_you_type의 핵심은 매핑을 정의하는 것만으로 내부적으로 여러 서브필드가 자동 생성된다는 점입니다. 2.2.4에서 다룬 멀티필드 패턴을 떠올려 봅니다. 멀티필드에서는 fields 파라미터로 서브필드를 명시적으로 정의했습니다. search_as_you_type은 이와 비슷하지만, 서브필드를 직접 정의하지 않아도 Elasticsearch가 자동으로 만들어 줍니다.

```json
PUT my_index
{
  "mappings": {
    "properties": {
      "title": {
        "type": "search_as_you_type"
      }
    }
  }
}
```

이 매핑에서 title 필드의 타입을 "search_as_you_type"으로 지정했습니다. 이 한 줄의 설정으로 Elasticsearch는 내부적으로 다음 네 가지 필드를 생성합니다.

```
title                  --> 원본 필드 (일반 text와 동일하게 분석)
title._2gram           --> 2-gram shingle 서브필드
title._3gram           --> 3-gram shingle 서브필드
title._index_prefix    --> edge n-gram 서브필드
```

원본 필드인 title은 일반 text 필드처럼 분석기를 적용하여 토큰을 저장합니다. title._2gram은 인접한 토큰 2개를 하나로 묶은 shingle을 저장합니다. "quick brown fox"를 인덱싱하면 "quick brown", "brown fox" 같은 2-gram shingle이 생성됩니다. title._3gram은 인접한 토큰 3개를 묶은 shingle을 저장합니다. "quick brown fox"라는 3-gram shingle이 생성됩니다. title._index_prefix는 각 토큰의 접두사를 edge n-gram 방식으로 미리 생성해 둡니다.

이 네 필드가 있으면, 사용자가 입력 도중인 불완전한 문자열로도 정확도 높은 매칭이 가능합니다. multi_match 쿼리의 bool_prefix 타입으로 검색하면 이 서브필드들을 자동으로 활용합니다.

```json
GET my_index/_search
{
  "query": {
    "multi_match": {
      "query": "quick bro",
      "type": "bool_prefix",
      "fields": [
        "title",
        "title._2gram",
        "title._3gram"
      ]
    }
  }
}
```

이 쿼리에서 type을 "bool_prefix"로 지정하면, 마지막 토큰인 "bro"를 접두사로 취급합니다. 3.2.2에서 다룬 match_phrase_prefix와 비슷한 동작이지만, shingle 서브필드 덕분에 토큰 순서와 인접성까지 고려한 매칭이 가능합니다. search_as_you_type은 커스텀 분석기 설정 없이 매핑 타입 하나만 지정하면 되므로, 자동완성 구현의 진입 장벽을 낮춰 줍니다.

마지막으로 **edge_ngram** 기반 자동완성 패턴을 살펴봅니다. 이 방식은 커스텀 분석기에서 edge_ngram 토크나이저를 사용하여, 인덱싱 시점에 각 토큰의 접두사를 미리 생성해 두는 것입니다.

edge_ngram이 무엇인지부터 설명합니다. 일반적인 n-gram은 문자열을 고정 길이의 연속된 조각으로 분해합니다. "quick"을 2-gram으로 분해하면 "qu", "ui", "ic", "ck"가 됩니다. 이 조각들은 문자열의 어디에서나 추출됩니다. edge_ngram은 이와 달리 문자열의 시작 부분(edge)에서만 조각을 추출합니다. "quick"을 edge_ngram으로 분해하면(최소 길이 1, 최대 길이 5 기준) "q", "qu", "qui", "quic", "quick"이 됩니다. 항상 첫 글자부터 시작하는 접두사만 만들어지는 것입니다.

이 접두사들을 인덱싱 시점에 역색인에 미리 저장해 두면, 쿼리 시점에는 일반 match 쿼리만으로 접두사 검색이 가능합니다. 사용자가 "qui"를 입력하면, 역색인에 "qui"라는 토큰이 이미 존재하므로 별도의 접두사 매칭 로직 없이 바로 매칭됩니다. 쿼리 시점의 부하가 줄어드는 대신, 인덱싱 시점에 더 많은 토큰을 생성하므로 인덱스 크기가 커지고 인덱싱 속도가 느려질 수 있습니다.

edge_ngram 기반 자동완성을 설정하려면 커스텀 분석기를 정의하고, 해당 분석기를 필드에 적용합니다.

```json
PUT my_index
{
  "settings": {
    "analysis": {
      "analyzer": {
        "autocomplete_analyzer": {
          "tokenizer": "autocomplete_tokenizer",
          "filter": ["lowercase"]
        }
      },
      "tokenizer": {
        "autocomplete_tokenizer": {
          "type": "edge_ngram",
          "min_gram": 1,
          "max_gram": 20,
          "token_chars": ["letter", "digit"]
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "analyzer": "autocomplete_analyzer",
        "search_analyzer": "standard"
      }
    }
  }
}
```

이 설정의 각 부분을 풀어 봅니다. "settings" 안의 "analysis"에서 커스텀 분석기를 정의합니다. "autocomplete_tokenizer"는 edge_ngram 타입의 토크나이저로, 최소 1글자부터 최대 20글자까지의 접두사를 생성합니다. "token_chars"는 토큰에 포함할 문자 종류로, 영문자와 숫자를 지정했습니다. "autocomplete_analyzer"는 이 토크나이저와 lowercase 필터를 조합한 분석기입니다.

매핑에서 title 필드에 이 분석기를 적용합니다. 여기서 중요한 것은 "analyzer"와 "search_analyzer"를 분리한 점입니다. "analyzer"는 인덱싱 시점에 사용할 분석기이고, "search_analyzer"는 검색 시점에 사용할 분석기입니다. 인덱싱할 때는 edge_ngram으로 접두사를 모두 생성하지만, 검색할 때는 standard 분석기를 사용하여 검색어를 그대로 토큰화합니다. 만약 검색 시점에도 edge_ngram 분석기를 사용하면, 검색어 "qui"가 "q", "qu", "qui"로 분해되어 "q"만 포함하는 문서도 결과에 나오게 됩니다. 의도하지 않은 과도한 매칭을 방지하기 위해 검색 분석기를 분리하는 것입니다.

지금까지 네 가지 자동완성 방식을 살펴보았습니다. 각 방식은 서로 다른 장단점을 가지므로, 상황에 맞게 선택해야 합니다. 다음 표에서 핵심 특성을 비교합니다.

```
자동완성 방식 비교

방식                   | 매핑 변경    | 쿼리 방식          | 중간 매칭 | 속도       | 인덱스 크기
-----------------------|------------|-------------------|----------|-----------|----------
match_phrase_prefix    | 불필요      | match_phrase_prefix| 불가     | 보통       | 변동 없음
completion suggester   | completion | suggest API        | 불가     | 매우 빠름   | FST 메모리
search_as_you_type     | 전용 타입   | multi_match        | 불가     | 빠름       | 서브필드 증가
edge_ngram             | 커스텀 분석기| match              | 불가     | 빠름       | 크게 증가
```

match_phrase_prefix는 기존 매핑을 그대로 사용할 수 있어 가장 간편하지만, 대규모 데이터에서 쿼리 시점 부하가 커질 수 있습니다. completion suggester는 FST를 메모리에 올려 두어 가장 빠른 응답을 보장하지만, 별도의 suggest API를 사용해야 하고 일반 검색 결과가 아니라 제안 목록을 반환합니다. search_as_you_type은 매핑 타입 하나만 지정하면 서브필드가 자동 생성되어 설정이 간편하고, 일반 쿼리로 검색할 수 있습니다. edge_ngram은 인덱싱 시점에 접두사를 모두 생성하므로 쿼리 시점 속도가 빠르지만, 인덱스 크기가 커지고 재인덱싱 비용이 발생합니다.

실무에서 방식을 선택할 때 고려할 기준을 정리합니다. 검색 분석과 같은 외부 소스에서 추천 목록을 관리하고 있다면 completion suggester가 적합합니다. weight로 추천 우선순위를 직접 제어할 수 있기 때문입니다. 기존 텍스트 데이터에 빠르게 자동완성을 붙이고 싶다면 search_as_you_type이 설정 부담이 가장 적습니다. 이미 커스텀 분석기를 운영 중이고 인덱스 크기 증가를 감수할 수 있다면 edge_ngram이 쿼리 시점 성능에서 유리합니다. 프로토타이핑 단계에서 빠르게 자동완성을 확인하고 싶다면 match_phrase_prefix로 시작하여, 트래픽이 늘어나면 다른 방식으로 전환하는 것이 합리적입니다.

정리하면, Elasticsearch에서 자동완성을 구현하는 방식은 match_phrase_prefix, completion suggester, search_as_you_type, edge_ngram 네 가지입니다. match_phrase_prefix는 매핑 변경 없이 쿼리 시점에 접두사 매칭을 수행합니다. completion suggester는 FST 자료 구조를 메모리에 올려 가장 빠른 접두사 검색을 제공하며, suggest API와 weight 설정으로 제안 우선순위를 제어합니다. search_as_you_type은 매핑 타입 하나로 edge n-gram과 shingle 서브필드를 자동 생성하여 일반 쿼리만으로 자동완성을 지원합니다. edge_ngram은 인덱싱 시점에 접두사 토큰을 미리 생성하여 쿼리 시점 부하를 줄이지만, 인덱스 크기가 증가합니다. 각 방식은 속도, 설정 복잡도, 인덱스 크기 사이에서 서로 다른 균형점을 제공하므로, 데이터 규모와 운영 요건에 맞게 선택합니다.

다음 단원인 3.3.3에서는 하이라이팅을 다룹니다.

이 단원을 마치면 completion suggester로 실시간 자동완성을 구현할 수 있고, search_as_you_type 필드 타입을 적용할 수 있습니다.
