# 4.2.1 중첩 집계

4.1.2에서 Bucket 집계가 문서를 그룹으로 나누는 역할을, 4.1.3에서 Metric 집계가 수치를 계산하는 역할을 각각 다뤘습니다. 4.1.2 마지막에 terms 집계 안에 sum 하위 집계를 넣어 국가별 매출 합계를 구하는 예시를 잠시 보았습니다. 이번 단원에서는 이 구조, 즉 집계 안에 또 다른 집계를 넣는 **중첩 집계**(sub-aggregation)를 본격적으로 다룹니다.

실무에서 "카테고리별 평균 가격"이나 "월별 매출 총액"처럼 그룹과 수치를 동시에 알고 싶은 상황이 자주 발생합니다. Bucket 집계 단독으로는 각 그룹의 문서 수(doc_count)만 알 수 있고, Metric 집계 단독으로는 전체 문서에 대한 하나의 수치만 구할 수 있습니다. 두 종류를 결합해야 비로소 "그룹별 통계"를 산출할 수 있습니다. 중첩 집계는 바깥 Bucket 집계가 만든 각 버킷 안에서 안쪽 집계를 실행하는 방식으로 이 결합을 구현합니다.

중첩 집계의 구문은 4.1.1에서 다룬 aggs 키워드를 반복하는 것입니다. 바깥 집계의 정의 안에 다시 "aggs" 키를 두고, 그 안에 하위 집계를 정의합니다. 가장 흔한 패턴부터 살펴봅니다.

첫 번째 패턴은 **terms + avg 중첩**입니다. "카테고리별 평균 가격"을 구한다고 가정합니다.

```json
{
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": {
        "field": "category",
        "size": 10
      },
      "aggs": {
        "avg_price": {
          "avg": { "field": "price" }
        }
      }
    }
  }
}
```

이 요청의 구조를 풀어 설명합니다. 최상위 "aggs" 안에 "by_category"라는 이름의 terms 집계가 있습니다. 이것이 바깥 Bucket 집계입니다. category 필드의 고유 값마다 버킷을 만듭니다. "by_category" 정의 안에 다시 "aggs"가 있고, 그 안에 "avg_price"라는 이름의 avg 집계가 있습니다. 이것이 안쪽 Metric 집계입니다. Elasticsearch는 먼저 terms 집계를 실행하여 카테고리별 버킷을 만든 뒤, 각 버킷에 속하는 문서에 대해 avg 집계를 실행합니다. 응답에는 각 카테고리 버킷마다 doc_count와 avg_price.value가 함께 담깁니다.

실행 흐름을 단계별로 정리합니다.

```
terms + avg 중첩 실행 흐름

1. terms 집계가 category 필드를 기준으로 버킷 생성
   - "의류" 버킷 (120건)
   - "전자기기" 버킷 (85건)
   - "식품" 버킷 (200건)

2. 각 버킷 안에서 avg 하위 집계 실행
   - "의류" 버킷 -> 120건의 price 평균 계산
   - "전자기기" 버킷 -> 85건의 price 평균 계산
   - "식품" 버킷 -> 200건의 price 평균 계산

3. 응답에 각 버킷의 doc_count + avg_price.value 포함
```

두 번째 패턴은 **date_histogram + sum 중첩**입니다. 월별 매출 총액을 구하는 시나리오입니다.

```json
{
  "size": 0,
  "aggs": {
    "monthly_sales": {
      "date_histogram": {
        "field": "order_date",
        "calendar_interval": "month"
      },
      "aggs": {
        "total_revenue": {
          "sum": { "field": "price" }
        }
      }
    }
  }
}
```

date_histogram 집계가 order_date 필드를 월 단위로 나누어 버킷을 만듭니다. 4.1.2에서 다룬 calendar_interval을 사용하여 달력 기준의 월별 버킷을 생성합니다. 각 월 버킷 안에서 sum 하위 집계가 해당 월의 price 합계를 계산합니다. 응답에는 2025년 1월 버킷에 doc_count(주문 건수)와 total_revenue.value(해당 월 매출 총액)가 함께 나타납니다. 시계열 데이터의 추이를 분석할 때 이 패턴을 가장 많이 사용합니다.

중첩은 두 단계에서 끝나지 않습니다. Bucket 집계 안에 다시 Bucket 집계를 넣고, 그 안에 Metric 집계를 넣는 **3단계 중첩**도 가능합니다. "국가별, 카테고리별 평균 가격"을 구한다고 가정합니다.

```json
{
  "size": 0,
  "aggs": {
    "by_country": {
      "terms": { "field": "country", "size": 5 },
      "aggs": {
        "by_category": {
          "terms": { "field": "category", "size": 10 },
          "aggs": {
            "avg_price": {
              "avg": { "field": "price" }
            }
          }
        }
      }
    }
  }
}
```

이 요청의 구조를 분해합니다. 1단계에서 country 필드로 국가별 버킷을 만듭니다. 2단계에서 각 국가 버킷 안에서 category 필드로 카테고리별 버킷을 만듭니다. 3단계에서 각 카테고리 버킷 안에서 price의 평균을 계산합니다. 국가가 5개이고 카테고리가 10개라면, 최대 50개의 하위 버킷이 생기고 각 버킷마다 avg 계산이 실행됩니다.

```
3단계 중첩 구조

by_country (1단계: Bucket)
  |
  +-- "KR" 버킷
  |     |
  |     +-- by_category (2단계: Bucket)
  |           +-- "의류" 버킷 -> avg_price (3단계: Metric)
  |           +-- "전자기기" 버킷 -> avg_price
  |           +-- "식품" 버킷 -> avg_price
  |
  +-- "US" 버킷
  |     |
  |     +-- by_category
  |           +-- "Clothing" 버킷 -> avg_price
  |           +-- "Electronics" 버킷 -> avg_price
  |           ...
  ...
```

여기서 중첩 집계의 **메모리 사용**에 대해 짚어야 합니다. 중첩 단계가 깊어질수록 버킷 수는 곱셈으로 증가합니다. 1단계에서 5개 버킷, 2단계에서 각 버킷 안에 10개 버킷이면 5 x 10 = 50개입니다. 3단계에서 다시 각 버킷 안에 20개 버킷이 생기면 5 x 10 x 20 = 1,000개입니다. 각 버킷마다 메모리를 소비하므로, 중첩 단계가 하나 추가될 때마다 메모리 사용량이 급격히 늘어납니다.

Elasticsearch는 이런 상황을 방지하기 위해 기본적으로 단일 요청에서 생성할 수 있는 버킷 수를 제한합니다. 기본 한도는 65,535개입니다. 이 한도를 초과하면 요청이 실패합니다. 따라서 중첩 단계를 설계할 때, 각 단계에서 생성되는 버킷 수를 예측하고 그 곱이 지나치게 커지지 않는지 확인해야 합니다. 4.1.2에서 다룬 terms 집계의 size 파라미터를 적절히 조절하는 것이 첫 번째 방법입니다.

지금까지 다룬 중첩 집계는 일반 필드에 대한 것이었습니다. Elasticsearch에는 **nested 타입**이라는 특수한 필드 타입이 있고, 이 타입에 대한 집계는 별도의 문법을 사용합니다. 이 두 가지 "중첩"은 이름이 같아 혼동하기 쉬우므로 구분하여 설명합니다.

일반적인 중첩 집계(sub-aggregation)는 아무 필드에나 적용할 수 있습니다. aggs 안에 aggs를 넣는 구문이 전부입니다. 반면 nested 타입 필드에 대한 집계는 **nested 집계**라는 전용 집계 타입을 사용해야 합니다.

nested 타입이 필요한 이유를 먼저 설명합니다. Elasticsearch에서 객체 배열을 일반 object 타입으로 저장하면, 배열 안의 각 객체가 서로 독립적인 관계를 잃습니다. 예를 들어 주문 문서 안에 상품 배열이 있고, 각 상품에 이름과 가격이 있다고 가정합니다. object 타입으로 저장하면 모든 상품 이름과 모든 상품 가격이 뒤섞여, "이름이 A이고 가격이 1000인 상품"을 정확히 찾을 수 없습니다. nested 타입은 배열의 각 객체를 내부적으로 별도의 숨겨진 문서로 저장하여 이 관계를 유지합니다.

nested 타입 필드에 집계를 걸 때는 먼저 nested 집계로 해당 필드의 범위에 진입해야 합니다. 상품 문서 안에 "reviews"라는 nested 필드가 있고, 각 리뷰에 rating이 포함되어 있다고 가정합니다. 리뷰의 평균 평점을 구하는 요청입니다.

```json
{
  "size": 0,
  "aggs": {
    "reviews_nested": {
      "nested": {
        "path": "reviews"
      },
      "aggs": {
        "avg_rating": {
          "avg": { "field": "reviews.rating" }
        }
      }
    }
  }
}
```

"nested"라는 집계 타입을 사용하고, "path"에 nested 필드 이름을 지정합니다. 이 집계가 만드는 버킷은 nested 문서(배열의 각 요소)를 담습니다. 그 안에서 하위 집계를 실행하면 nested 문서를 대상으로 계산합니다. 만약 nested 집계 없이 바로 "reviews.rating"에 avg 집계를 걸면, Elasticsearch는 nested 문서에 접근하지 못하여 결과가 비어 있거나 잘못된 값이 나옵니다.

nested 집계 안에서 다시 부모 문서의 필드에 접근하고 싶을 때가 있습니다. 예를 들어 "리뷰를 남긴 상품의 카테고리별로 리뷰 수를 세고 싶다"는 경우입니다. nested 집계의 범위 안에서는 nested 문서의 필드만 보이므로, 부모 문서의 category 필드에 접근할 수 없습니다. 이때 **reverse_nested 집계**를 사용합니다.

```json
{
  "size": 0,
  "aggs": {
    "reviews_nested": {
      "nested": {
        "path": "reviews"
      },
      "aggs": {
        "high_rating": {
          "filter": {
            "range": { "reviews.rating": { "gte": 4 } }
          },
          "aggs": {
            "back_to_product": {
              "reverse_nested": {},
              "aggs": {
                "by_category": {
                  "terms": { "field": "category" }
                }
              }
            }
          }
        }
      }
    }
  }
}
```

이 요청의 흐름을 풀어 설명합니다. nested 집계로 reviews 배열에 진입합니다. filter 하위 집계로 rating이 4 이상인 리뷰만 골라냅니다. reverse_nested 집계로 nested 범위에서 빠져나와 부모 문서 범위로 돌아갑니다. 부모 문서 범위에서 terms 집계로 category별 버킷을 만듭니다. 결과적으로 "높은 평점 리뷰가 달린 상품의 카테고리 분포"를 알 수 있습니다.

```
nested + reverse_nested 실행 흐름

1. nested 집계 -> reviews 배열의 각 요소에 접근
2. filter 집계 -> rating >= 4인 리뷰만 선택
3. reverse_nested 집계 -> 부모 문서(상품) 범위로 복귀
4. terms 집계 -> 부모 문서의 category 필드로 그룹화
```

reverse_nested 집계의 중괄호 안이 비어 있으면 최상위 부모 문서로 돌아갑니다. 다단계 nested 구조에서 특정 중간 단계로 돌아가고 싶으면 "path" 파라미터에 해당 경로를 지정합니다.

정리하면, 중첩 집계는 바깥 Bucket 집계 안에 하위 집계를 넣어 그룹별 통계를 산출하는 구조입니다. terms + avg, date_histogram + sum이 가장 흔한 패턴이며, 3단계 이상 중첩도 가능하지만 버킷 수가 곱셈으로 증가하므로 메모리 사용에 주의해야 합니다. nested 타입 필드에는 nested 집계로 범위에 진입하고, 부모 문서로 돌아갈 때는 reverse_nested 집계를 사용합니다.

다음 단원인 4.2.2에서는 Pipeline 집계를 다룹니다.

이 단원을 마치면 Bucket 집계 내부에 Metric 집계를 중첩하여 그룹별 통계를 계산할 수 있습니다.
