# 5.1.1 노드 역할

1.3.3에서 Elasticsearch 클러스터가 여러 노드로 구성되고, 데이터가 샤드 단위로 노드에 분산된다는 것을 살펴보았습니다. 그런데 클러스터 안의 모든 노드가 동일한 일을 하는 것은 아닙니다. 실제 운영 환경에서는 한 노드가 데이터를 저장하고, 다른 노드는 클러스터 전체를 관리하며, 또 다른 노드는 데이터를 가공하는 식으로 업무를 나눕니다. 이 업무 구분을 **노드 역할**(node role)이라고 합니다.

노드 역할이 필요한 이유부터 짚어 봅니다. 클러스터 규모가 작을 때는 한 노드가 데이터 저장, 클러스터 관리, 검색 요청 처리를 모두 맡아도 문제가 없습니다. 그러나 노드 수가 늘어나고 데이터 양이 많아지면, 한 노드에 모든 작업이 몰리면서 병목이 생깁니다. 클러스터 상태를 관리하는 작업이 무거운 인덱싱 작업과 동일한 CPU와 메모리를 두고 경쟁하면, 클러스터 전체의 안정성이 흔들릴 수 있습니다. 역할을 분리하면 각 노드가 자기 업무에만 자원을 집중할 수 있으므로, 전체 클러스터의 안정성과 처리 성능이 높아집니다.

Elasticsearch에서 노드 역할은 `elasticsearch.yml` 설정 파일의 **node.roles** 항목으로 지정합니다. 이 항목은 배열 형태로, 해당 노드가 수행할 역할 이름을 나열합니다. 예를 들어 다음과 같이 설정하면, 이 노드는 데이터 저장만 담당합니다.

```yaml
node.roles: [ data ]
```

node.roles를 아예 지정하지 않으면, Elasticsearch는 해당 노드에 기본 역할 전부를 부여합니다. 기본 역할에는 master, data(모든 티어), ingest, ml, remote_cluster_client, transform이 포함됩니다. 소규모 개발 환경에서는 이 기본값이 편리하지만, 프로덕션 환경에서는 역할을 명시적으로 지정하는 것이 권장됩니다.

이제 각 역할을 하나씩 살펴봅니다.

첫 번째는 **master-eligible 노드**입니다. 이 역할을 가진 노드는 클러스터의 마스터 노드로 선출될 자격이 있습니다. 마스터 노드는 클러스터 전체를 관리하는 단 하나의 노드입니다. 인덱스 생성과 삭제, 샤드 할당, 클러스터 메타데이터 유지 같은 작업을 담당합니다. master-eligible 노드가 여러 대 있더라도, 실제로 마스터 역할을 수행하는 노드는 선거를 통해 선출된 한 대뿐입니다. 나머지 master-eligible 노드는 대기 상태로 있다가, 현재 마스터에 장애가 발생하면 새로운 마스터를 선출합니다.

```yaml
node.roles: [ master ]
```

이렇게 설정하면 이 노드는 마스터 선출에만 참여하고, 데이터 저장이나 인덱싱 같은 작업은 수행하지 않습니다. 이런 노드를 **전용 마스터 노드**(dedicated master node)라고 합니다. 전용 마스터 노드는 클러스터 관리 작업에만 자원을 쓰므로, 무거운 인덱싱이나 검색 부하에 영향받지 않습니다.

두 번째는 **data 노드**입니다. data 노드는 실제 데이터를 저장하는 노드입니다. 샤드를 보유하고 있으며, 문서의 CRUD(생성, 읽기, 수정, 삭제) 연산과 검색, 집계를 처리합니다. 디스크 I/O, 메모리, CPU 자원을 많이 소비합니다.

```yaml
node.roles: [ data ]
```

Elasticsearch는 data 노드를 더 세분화하여 **데이터 티어**(data tier)를 제공합니다. 데이터 티어란 데이터의 접근 빈도와 성능 요구 수준에 따라 노드를 분류하는 체계입니다. 시계열 데이터처럼 시간이 지남에 따라 접근 빈도가 줄어드는 데이터에 특히 유용합니다.

**data_hot** 티어는 가장 최근에 생성된 데이터를 저장합니다. 검색과 인덱싱이 빈번하므로 고성능 SSD를 사용하는 노드에 배치합니다. **data_warm** 티어는 검색 빈도가 줄어든 데이터를 저장합니다. hot 티어만큼 빠른 저장소가 필요하지 않으므로, 용량 대비 비용이 낮은 디스크를 사용합니다. **data_cold** 티어는 거의 검색되지 않는 과거 데이터를 보관합니다. 검색 속도가 느려도 허용되는 데이터입니다. **data_frozen** 티어는 극히 드물게 조회되는 보관용 데이터를 저장합니다. 스냅샷 저장소를 활용하여 비용을 최소화합니다.

이 외에 **data_content** 티어가 있습니다. 시계열이 아닌 일반 데이터, 예를 들어 상품 카탈로그나 사용자 프로필 같은 데이터를 저장하는 노드입니다.

data 역할만 단독으로 지정하면 모든 티어의 데이터를 처리하는 범용 데이터 노드가 됩니다. 티어를 세분화하려면 다음과 같이 구체적인 티어 역할을 지정합니다.

```yaml
node.roles: [ data_hot, data_content ]
```

세 번째는 **ingest 노드**입니다. ingest 노드는 문서가 인덱싱되기 전에 데이터를 변환하고 보강하는 역할을 합니다. 예를 들어 로그 데이터에서 IP 주소를 추출하여 지역 정보를 덧붙이거나, 특정 필드의 값을 변환하는 작업을 수행합니다. 이 작업은 인제스트 파이프라인(ingest pipeline)이라는 규칙 집합을 통해 정의됩니다.

```yaml
node.roles: [ ingest ]
```

인제스트 부하가 적으면 data 노드나 master 노드가 ingest 역할을 겸해도 됩니다. 그러나 변환 작업이 무겁거나 빈번하면, 전용 ingest 노드를 두어 데이터 노드와 마스터 노드의 자원을 보호하는 것이 좋습니다.

네 번째는 **coordinating 노드**입니다. Elasticsearch에서 모든 노드는 기본적으로 코디네이팅 역할을 수행합니다. 클라이언트가 검색 요청을 보내면, 요청을 받은 노드가 해당 인덱스의 각 샤드에 요청을 분배하고, 각 샤드의 결과를 모아 최종 응답을 만듭니다. 이 과정을 코디네이팅이라고 합니다.

그런데 대규모 클러스터에서는 코디네이팅 작업 자체가 상당한 CPU와 메모리를 소비합니다. 이 경우 코디네이팅만 전담하는 노드를 별도로 둘 수 있습니다. node.roles를 빈 배열로 설정하면 됩니다.

```yaml
node.roles: [ ]
```

빈 배열은 이 노드가 마스터 선출에도 참여하지 않고, 데이터도 저장하지 않으며, 인제스트도 하지 않겠다는 뜻입니다. 남는 역할은 코디네이팅뿐이므로, 로드 밸런서처럼 요청을 받아 분배하는 일에만 집중합니다. 다만 코디네이팅 전용 노드를 과도하게 늘리면, 마스터 노드가 클러스터 상태 업데이트를 더 많은 노드에 전달해야 하므로 오히려 부담이 될 수 있습니다.

다섯 번째는 **machine_learning 노드**입니다. 이 노드는 Elasticsearch의 머신 러닝 기능을 실행합니다. 이상 탐지(anomaly detection)나 추론(inference) 같은 ML 작업은 CPU와 메모리를 많이 소비하므로, 데이터 노드와 분리하는 것이 좋습니다.

```yaml
node.roles: [ ml, remote_cluster_client ]
```

ml 역할과 함께 remote_cluster_client 역할을 지정하는 것이 권장됩니다. remote_cluster_client는 원격 클러스터에 연결할 수 있는 역할인데, ML 작업이 다른 클러스터의 데이터를 참조해야 할 때 필요합니다.

여섯 번째는 **transform 노드**입니다. transform 노드는 인덱스 데이터를 피벗(pivot)하거나 집계하여 새로운 인덱스를 만드는 변환 작업을 수행합니다. 대량의 데이터를 읽고 집계한 뒤 결과를 기록하는 작업이므로, 전용 노드를 두면 다른 작업에 미치는 영향을 줄일 수 있습니다.

```yaml
node.roles: [ transform, remote_cluster_client ]
```

transform 역할도 ml과 마찬가지로 remote_cluster_client를 함께 지정하는 것이 권장됩니다.

이제 **역할 혼용과 전용 설정**을 정리합니다. 앞에서 각 역할을 하나씩 분리하여 설명했지만, 실제로는 한 노드에 여러 역할을 동시에 부여할 수 있습니다. node.roles 배열에 원하는 역할을 나열하면 됩니다.

클러스터 규모에 따른 권장 구성을 표로 정리하면 다음과 같습니다.

```
클러스터 규모별 권장 노드 역할 구성

+----------------+-----------------------------------------------+
| 클러스터 규모  | 권장 구성                                     |
+----------------+-----------------------------------------------+
| 소규모 (개발)  | 모든 노드에 기본 역할 전부 부여               |
|                | (node.roles 미지정)                           |
+----------------+-----------------------------------------------+
| 중규모         | master-eligible 노드와 data 노드 분리         |
|                | master 3대 + data 노드 N대                    |
+----------------+-----------------------------------------------+
| 대규모         | 전용 master + 데이터 티어별 노드 +            |
|                | 전용 ingest + 전용 coordinating               |
+----------------+-----------------------------------------------+
```

소규모 환경에서는 모든 노드가 모든 역할을 겸합니다. 노드 수가 적으므로 역할 분리의 이점보다 운영 단순성이 더 중요합니다. 중규모 환경부터는 마스터 노드와 데이터 노드를 분리합니다. 마스터 노드가 인덱싱이나 검색 부하에 영향받지 않아 클러스터 안정성이 높아집니다. 대규모 환경에서는 데이터 티어, 인제스트, 코디네이팅까지 모두 전용 노드로 나누어 각 작업이 서로의 자원을 침범하지 않도록 합니다.

모든 클러스터에서 반드시 존재해야 하는 역할이 있습니다. 첫째, master 역할을 가진 노드가 최소 한 대 있어야 클러스터를 관리할 수 있습니다. 둘째, data 역할(또는 data_content와 data_hot 조합)을 가진 노드가 있어야 데이터를 저장할 수 있습니다. 이 두 가지가 없으면 클러스터가 정상 동작하지 않습니다.

마스터 노드는 데이터 노드와 마찬가지로 영구적인 데이터 경로(path.data)가 필요합니다. 마스터 노드가 클러스터 메타데이터를 디스크에 저장하기 때문입니다. 이 경로가 휘발성이면 노드 재시작 시 메타데이터가 유실되어, 클러스터 상태에 문제가 생길 수 있습니다.

정리하면, Elasticsearch 노드 역할은 클러스터 내에서 각 노드가 수행할 업무를 구분하는 체계입니다. master-eligible 노드는 클러스터를 관리하고, data 노드는 샤드를 저장하며, ingest 노드는 데이터를 사전 처리하고, coordinating 노드는 요청을 분배합니다. ml 노드와 transform 노드는 각각 머신 러닝과 데이터 변환을 전담합니다. node.roles 설정으로 역할을 명시하며, 클러스터 규모가 커질수록 역할을 전용 노드에 분리하여 안정성과 성능을 확보합니다.

다음 단원인 5.1.2에서는 클러스터 아키텍처를 다룹니다. 마스터 노드가 어떻게 선출되는지, 마스터 노드 장애 시 클러스터가 어떻게 대응하는지를 살펴봅니다.

이 단원을 마치면 Elasticsearch 노드 역할 유형을 열거하고 각 역할의 책임을 설명할 수 있습니다.
