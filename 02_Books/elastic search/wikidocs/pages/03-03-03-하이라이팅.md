# 3.3.3 하이라이팅

검색 결과를 받아 화면에 보여 줄 때, 사용자가 입력한 검색어가 문서의 어느 부분과 일치했는지 시각적으로 드러내야 할 때가 있습니다. 웹 검색 엔진에서 결과 목록을 보면, 검색어에 해당하는 단어가 굵게 표시되거나 색이 바뀌어 있는 것을 흔히 볼 수 있습니다. 이 기능을 **하이라이팅**(highlighting)이라고 합니다. Elasticsearch는 검색 요청에 highlight 파라미터를 추가하는 것만으로 이 기능을 사용할 수 있습니다.

3.2.2에서 match 쿼리를 사용하면 검색어가 분석기를 거쳐 토큰 단위로 역색인을 조회한다는 것을 살펴보았습니다. 하이라이팅은 이 과정에서 매칭된 토큰이 원문의 어디에 위치하는지를 찾아, 해당 부분을 HTML 태그 등으로 감싸 반환하는 기능입니다. 검색 자체와 별개로 동작하는 것이 아니라, 검색 결과의 각 문서에 대해 매칭 위치를 추출하는 후처리 단계라고 이해하면 됩니다.

**highlight 파라미터**는 Search API의 요청 본문에 추가하는 객체입니다. 이 객체 안에 하이라이팅을 적용할 필드를 지정합니다. 가장 기본적인 형태를 살펴봅니다.

```json
GET my-index/_search
{
  "query": {
    "match": {
      "content": "quick brown fox"
    }
  },
  "highlight": {
    "fields": {
      "content": {}
    }
  }
}
```

"query" 부분은 3.2.2에서 다룬 match 쿼리와 동일합니다. 새로 추가된 부분은 "highlight" 객체입니다. "highlight" 안에 "fields"를 두고, 하이라이팅을 적용할 필드 이름("content")을 키로 지정합니다. 값은 빈 객체 {}로 두면 기본 설정이 적용됩니다.

이 요청에 대한 응답은 각 검색 결과 문서에 "highlight" 항목이 추가된 형태로 돌아옵니다.

```json
{
  "hits": {
    "hits": [
      {
        "_source": {
          "content": "The quick brown fox jumps over the lazy dog"
        },
        "highlight": {
          "content": [
            "The <em>quick</em> <em>brown</em> <em>fox</em> jumps over the lazy dog"
          ]
        }
      }
    ]
  }
}
```

응답의 "highlight" 안에 필드 이름("content")이 키로 나타나고, 값은 문자열 배열입니다. 배열의 각 원소는 **프래그먼트**(fragment)라고 부르는 텍스트 조각입니다. 프래그먼트는 원문에서 매칭된 부분을 포함하는 일정 길이의 발췌문입니다. 위 예시에서는 원문이 짧아 문장 전체가 하나의 프래그먼트로 반환되었습니다.

프래그먼트 안에서 매칭된 단어는 기본적으로 `<em>` 태그로 감싸져 있습니다. "quick"이 `<em>quick</em>`으로, "brown"이 `<em>brown</em>`으로, "fox"가 `<em>fox</em>`으로 표시됩니다. 이 태그를 프론트엔드에서 CSS로 스타일링하면 시각적 강조 효과를 줄 수 있습니다.

기본 `<em>` 태그 대신 다른 태그를 사용하고 싶다면 **pre_tags**와 **post_tags**를 지정합니다. pre_tags는 매칭 단어 앞에 삽입할 태그이고, post_tags는 매칭 단어 뒤에 삽입할 태그입니다.

```json
GET my-index/_search
{
  "query": {
    "match": {
      "content": "quick brown fox"
    }
  },
  "highlight": {
    "pre_tags": ["<mark>"],
    "post_tags": ["</mark>"],
    "fields": {
      "content": {}
    }
  }
}
```

이 설정으로 요청하면 매칭 단어가 `<mark>quick</mark>` 형태로 감싸집니다. `<mark>` 태그는 HTML5에서 텍스트 강조를 위해 정의된 태그로, 브라우저에서 기본적으로 노란색 배경을 적용합니다. pre_tags와 post_tags는 배열 형태이므로, 여러 쌍의 태그를 지정하여 매칭 토큰마다 서로 다른 태그를 순환 적용할 수도 있습니다.

하이라이팅의 동작 방식은 사용하는 **하이라이터**(highlighter)의 종류에 따라 달라집니다. Elasticsearch는 세 가지 하이라이터를 제공합니다. 각 하이라이터는 매칭 위치를 찾는 방식이 서로 다르며, 성능과 정확도에서 차이가 있습니다. 어떤 하이라이터를 선택하느냐에 따라 결과의 품질과 응답 속도가 달라질 수 있으므로, 각각의 특성을 이해해 두는 것이 중요합니다.

첫 번째는 **unified 하이라이터**입니다. 별도 설정을 하지 않으면 Elasticsearch가 기본으로 사용하는 하이라이터입니다. text 타입 필드와 keyword 타입 필드 모두에 적용됩니다. unified 하이라이터는 원문을 문장 단위로 분할한 뒤, 각 문장을 하나의 작은 문서처럼 취급하여 BM25 알고리즘으로 점수를 매깁니다. 검색어와 가장 관련성이 높은 문장을 프래그먼트로 선택하는 방식입니다. 구문(phrase) 검색이나 여러 토큰이 동시에 등장하는 복합 검색에서도 정확한 하이라이팅을 제공합니다. 대부분의 사용 사례에서 이 하이라이터를 그대로 사용하면 충분합니다.

unified 하이라이터가 매칭 위치를 찾는 방법은 세 가지 중 하나를 자동으로 선택합니다. 매핑에서 해당 필드의 index_options가 offsets로 설정되어 있으면, 인덱싱 시점에 이미 저장된 문자 단위 오프셋 정보를 직접 읽습니다. 이 방식이 가장 빠릅니다. term_vector가 with_positions_offsets로 설정되어 있으면, 텀 벡터에 저장된 위치와 오프셋 정보를 활용합니다. 두 가지 모두 설정되어 있지 않으면, 원문을 메모리에 올려 분석기를 다시 실행하여 매칭 위치를 계산합니다. 마지막 방식이 가장 느리지만, 별도 매핑 설정 없이도 동작한다는 장점이 있습니다.

두 번째는 **plain 하이라이터**입니다. Lucene의 기본 하이라이팅 구현을 사용합니다. plain 하이라이터는 항상 원문을 메모리에 올려 분석기를 다시 실행하는 방식으로 동작합니다. 단일 필드의 단순한 쿼리에서는 잘 동작하지만, 문서 수가 많거나 쿼리가 복잡해지면 분석기를 반복 실행하는 비용이 커집니다. 다른 하이라이터를 사용할 수 없는 특수한 상황에서 대안으로 사용하는 경우가 많습니다.

세 번째는 **fvh**(fast vector highlighter)입니다. Lucene의 Fast Vector 구현을 기반으로 합니다. fvh를 사용하려면 매핑에서 해당 필드의 **term_vector**를 "with_positions_offsets"로 설정해야 합니다. 이 설정은 인덱싱 시점에 각 토큰의 위치(position)와 문자 오프셋(offset)을 별도 저장 공간에 기록하라는 의미입니다.

```json
PUT my-index
{
  "mappings": {
    "properties": {
      "content": {
        "type": "text",
        "term_vector": "with_positions_offsets"
      }
    }
  }
}
```

이 매핑에서 content 필드의 term_vector를 "with_positions_offsets"로 지정했습니다. 인덱싱할 때 토큰의 위치와 오프셋이 함께 저장되므로, 하이라이팅 시점에 원문을 다시 분석할 필요가 없습니다. fvh 하이라이터는 이 저장된 정보를 곧바로 읽어 매칭 위치를 결정합니다.

fvh가 유리한 상황은 필드 값이 매우 큰 경우입니다. 1MB를 넘는 대용량 텍스트 필드에서는 원문을 메모리에 올려 다시 분석하는 비용이 상당합니다. fvh는 사전에 저장된 벡터 정보를 읽기만 하므로, 대용량 필드에서 다른 하이라이터보다 빠른 응답을 제공합니다. 단, term_vector 설정은 인덱스 용량을 증가시킵니다. 모든 토큰의 위치와 오프셋을 별도로 저장하기 때문입니다. 따라서 필드 크기가 작은 일반적인 상황에서는 기본 unified 하이라이터를 사용하고, 대용량 필드에서 하이라이팅 성능이 문제될 때 fvh를 고려하는 것이 합리적입니다.

세 하이라이터의 특성을 비교하면 다음과 같습니다.

```
하이라이터 비교

하이라이터  | 매칭 위치 탐색 방식          | 매핑 요구 사항           | 적합한 상황
-----------|---------------------------|------------------------|------------------
unified    | offsets/term_vector/재분석  | 없음 (기본 동작)         | 대부분의 일반 검색
           | 중 자동 선택               |                        |
plain      | 항상 원문 재분석             | 없음                    | 단순 쿼리, 대안 없을 때
fvh        | term_vector 읽기           | term_vector 설정 필수    | 대용량 필드(>1MB)
```

하이라이팅의 프래그먼트 크기와 개수를 제어하는 설정도 있습니다. **fragment_size**는 각 프래그먼트의 문자 수를 지정합니다. 기본값은 100입니다. 원문이 길 때, 매칭 부분 주변의 몇 글자를 발췌할지를 이 값으로 조절합니다.

**number_of_fragments**는 반환할 프래그먼트의 최대 개수입니다. 기본값은 5입니다. 원문에 매칭 위치가 여러 곳일 때, 가장 관련성이 높은 프래그먼트를 이 개수만큼 반환합니다.

```json
GET my-index/_search
{
  "query": {
    "match": {
      "content": "elasticsearch"
    }
  },
  "highlight": {
    "fields": {
      "content": {
        "fragment_size": 150,
        "number_of_fragments": 3
      }
    }
  }
}
```

이 요청은 content 필드에서 "elasticsearch"를 검색하고, 매칭된 부분 주변의 150자를 하나의 프래그먼트로 잘라, 최대 3개까지 반환합니다. 검색 결과 페이지에서 미리보기 텍스트의 길이와 개수를 이 두 설정으로 조절할 수 있습니다.

number_of_fragments를 0으로 설정하면 특별한 동작을 합니다. 프래그먼트로 분할하지 않고, 필드 전체를 하나의 덩어리로 반환하면서 매칭 부분에만 태그를 적용합니다. 제목(title)처럼 짧은 필드를 하이라이팅할 때 유용합니다.

마지막으로 **require_field_match** 옵션을 살펴봅니다. 기본값은 true입니다. 이 값이 true이면, 쿼리가 실제로 매칭한 필드만 하이라이팅합니다. 예를 들어 match 쿼리가 content 필드를 대상으로 했다면, highlight에 title 필드를 함께 지정해도 title 필드에는 하이라이팅이 적용되지 않습니다. 쿼리가 title 필드를 대상으로 하지 않았기 때문입니다.

require_field_match를 false로 설정하면, 쿼리 대상 필드와 관계없이 하이라이팅을 적용합니다. 검색어 토큰이 해당 필드에 존재하기만 하면 하이라이팅됩니다.

```json
GET my-index/_search
{
  "query": {
    "match": {
      "content": "elasticsearch"
    }
  },
  "highlight": {
    "require_field_match": false,
    "fields": {
      "title": {},
      "content": {}
    }
  }
}
```

이 요청에서 match 쿼리는 content 필드만 대상으로 합니다. 하지만 require_field_match가 false이므로, title 필드에도 "elasticsearch"라는 단어가 있으면 하이라이팅이 적용됩니다. 여러 필드의 내용을 동시에 미리보기로 보여 주되, 검색 대상 필드가 아닌 곳에서도 검색어를 강조하고 싶을 때 이 설정을 사용합니다.

하이라이팅이 동작하려면 원문 내용에 접근할 수 있어야 합니다. Elasticsearch는 기본적으로 문서의 원문을 _source 필드에 저장합니다. 하이라이터는 이 _source에서 해당 필드의 값을 추출하여 작업합니다. _source가 비활성화된 인덱스에서 하이라이팅을 사용하려면, 매핑에서 해당 필드에 "store": true를 설정하여 필드 값을 별도로 저장해야 합니다.

정리하면, highlight 파라미터를 검색 요청에 추가하면 매칭된 부분을 태그로 감싼 프래그먼트를 받을 수 있습니다. 기본 태그는 `<em>`이며, pre_tags와 post_tags로 변경할 수 있습니다. 하이라이터는 unified(기본값), plain, fvh 세 종류가 있고, 대부분의 상황에서는 unified로 충분합니다. fvh는 대용량 필드에서 성능 이점을 제공하지만, 매핑에 term_vector 설정이 필요합니다. fragment_size와 number_of_fragments로 프래그먼트의 크기와 개수를 조절하고, require_field_match로 하이라이팅 대상 필드의 범위를 제어합니다.

다음 단원인 3.3.4에서는 페이지네이션을 다룹니다.

이 단원을 마치면 highlight 파라미터로 검색 결과에 하이라이팅을 적용할 수 있고, unified, plain, fvh 하이라이터의 차이를 설명할 수 있습니다.
