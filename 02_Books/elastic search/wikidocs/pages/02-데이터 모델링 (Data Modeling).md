# Phase 2. 데이터 모델링 (Data Modeling)

이 Phase를 마치면 인덱스 매핑을 설계하고, 텍스트 분석기를 커스터마이징하며, 한국어 형태소 분석을 적용할 수 있게 됩니다.

데이터 모델링은 Elasticsearch 활용의 핵심입니다. 어떤 필드에 어떤 타입을 지정하느냐, 텍스트를 어떻게 분석하느냐에 따라 검색 품질과 시스템 성능이 크게 달라집니다. Phase 2에서는 매핑 설계부터 텍스트 분석, 한국어 형태소 분석까지 데이터를 올바르게 표현하는 방법을 체계적으로 다룹니다.

첫 번째 섹션에서는 동적 매핑과 명시적 매핑의 차이, 매핑 업데이트 제약, 매핑 폭발 방지 방법을 다룹니다. 두 번째 섹션에서는 text, keyword, 숫자, 날짜, dense_vector 등 다양한 필드 타입을 다룹니다. 세 번째 섹션에서는 분석기 파이프라인 구조와 커스텀 분석기 구성, 한국어 Nori 분석기까지 다룹니다. 네 번째 섹션에서는 런타임 필드, 중첩 문서, 인덱스 템플릿, 인덱스 별칭 등 고급 모델링 패턴을 다룹니다.

## 이 Phase의 섹션 구성

**2.1 매핑 기초**
동적 매핑 vs 명시적 매핑, 매핑 업데이트 제약과 reindex, 매핑 폭발 방지를 다룹니다.
- 2.1.1 동적 매핑 vs 명시적 매핑
- 2.1.2 매핑 설정과 업데이트 제약
- 2.1.3 매핑 폭발 방지

**2.2 필드 타입**
text vs keyword, 숫자/날짜/불리언 타입, 특수 타입(dense_vector, geo_point, nested, join), 멀티필드 패턴을 다룹니다.
- 2.2.1 text vs keyword
- 2.2.2 숫자/날짜/불리언 타입
- 2.2.3 특수 타입: dense_vector, geo_point, nested, join
- 2.2.4 멀티필드 패턴

**2.3 텍스트 분석**
분석기 파이프라인(character filter → tokenizer → token filter), 내장 분석기, 커스텀 분석기 구성, 한국어 Nori 분석기를 다룹니다.
- 2.3.1 분석기 파이프라인
- 2.3.2 내장 분석기
- 2.3.3 커스텀 분석기 구성
- 2.3.4 한국어 분석: Nori

**2.4 고급 데이터 모델링**
런타임 필드, 중첩 문서 vs Parent-Child 조인, 인덱스 템플릿과 Component Template, 인덱스 별칭을 다룹니다.
- 2.4.1 런타임 필드
- 2.4.2 중첩 문서 vs Parent-Child 조인
- 2.4.3 인덱스 템플릿과 Component Template
- 2.4.4 인덱스 별칭
