# 5.1.1 RAG 구성요소 한눈에 보기

대규모 언어 모델은 학습 시점에 본 지식만 알고 있고, 사내 위키나 어제 올라온 문서는 알지 못합니다. **RAG**는 Retrieval-Augmented Generation의 줄임말로, 질문이 들어올 때마다 외부 문서 저장소에서 관련 조각을 끌어와 프롬프트에 붙여 주고 그 위에서 답을 생성하는 구조입니다. 이 단원은 RAG 시스템 전체의 부품을 처음부터 끝까지 한 번에 훑어, 뒤 단원들이 어느 자리를 다루는지 지도를 만들어 두는 것을 목표로 합니다.

RAG는 큰 흐름이 두 갈래로 나뉩니다. 하나는 **Ingestion 파이프라인**으로, 문서를 받아서 검색 가능한 형태로 저장소에 넣는 사전 작업입니다. 다른 하나는 **Query 파이프라인**으로, 사용자가 질문을 던졌을 때 그 저장소에서 관련 조각을 찾아 답을 만드는 실시간 작업입니다. 두 파이프라인은 같은 저장소를 공유하지만, 동작 시점과 성능 기준이 다릅니다. Ingestion은 분 단위 배치로 돌아도 괜찮지만, Query는 사용자가 기다리는 수백 밀리초 안에 끝나야 합니다.

부품 단위로 나열하면 다음과 같습니다. **Loader**는 PDF, HTML, 노션, DB 등에서 원본을 읽어 옵니다. **Parser**는 그 원본에서 본문 텍스트와 표, 이미지 설명을 추출합니다. **Chunker**는 긴 본문을 검색에 유리한 크기의 조각으로 자릅니다. **Embedding** 모델은 각 조각을 의미를 담은 숫자 벡터로 바꿉니다. **Vector DB**는 그 벡터를 저장하고 빠른 유사도 검색을 가능하게 합니다. 여기까지가 Ingestion입니다.

질문이 들어온 시점부터는 다른 부품들이 작동합니다. **Retriever**는 질문 임베딩과 가까운 조각 후보 수십 개를 Vector DB에서 가져옵니다. **Reranker**는 그 후보들을 더 정교한 모델로 다시 줄 세워 상위 몇 개만 남깁니다. **Generator**는 남은 조각과 질문을 합쳐 프롬프트를 만들고 LLM을 호출해 답을 만듭니다. **Citation**은 답 안에 어떤 조각을 근거로 썼는지 표시하는 단계이고, **Eval**은 그 답이 얼마나 정확한지를 자동으로 측정하는 단계입니다.

전체 흐름을 한 줄로 보면 다음과 같은 모양이 됩니다.

```
[Ingestion]
  Loader -> Parser -> Chunker -> Embedding -> Vector DB

[Query]
  Question -> Retriever -> Reranker -> Generator -> Citation -> Eval
                 |                                       |
                 +------- 같은 Vector DB 참조 -----------+
```

이 그림에서 한 가지 짚어 둘 점은, 화살표가 한 방향처럼 보이지만 실제 운영에서는 평가 결과가 다시 앞 단계의 설정으로 되돌아간다는 사실입니다. 답이 자꾸 엉뚱하면 청크 크기를 다시 조정하고, 인용이 자꾸 빗나가면 임베딩 모델을 바꾸고, 후보군에 정답이 안 들어오면 검색 방식을 hybrid로 바꿉니다. RAG 시스템은 한 번 만들고 끝나는 것이 아니라, 평가 지표를 보고 단계마다 튜닝을 반복하는 시스템입니다.

각 단계가 다음 단계에 어떤 모양으로 입력을 넘기는지 표로 정리하면 이해가 빠릅니다.

| 단계 | 입력 | 출력 | 품질 결정 요소 |
|---|---|---|---|
| Loader | 파일 경로, URL | 원본 바이트 | 형식 지원 범위 |
| Parser | 원본 바이트 | 본문 텍스트, 메타데이터 | 표·이미지 인식 정확도 |
| Chunker | 본문 텍스트 | 조각 문자열의 리스트 | 크기·중첩·경계 |
| Embedding | 조각 문자열 | 768~1536 차원 벡터 | 모델 도메인 적합도 |
| Vector DB | 벡터 + 메타 | 인덱스 | 인덱스 종류, 차원 |
| Retriever | 질문 벡터 | 후보 조각 N개 | top-N 크기, 거리 함수 |
| Reranker | 후보 N개 | 정렬된 top-K | cross-encoder 품질 |
| Generator | top-K + 질문 | 답변 문자열 | 프롬프트, 모델 |
| Citation | 답변 + 근거 | 출처 붙은 답 | 매핑 정확도 |
| Eval | 답 + 정답 | 점수 | 평가자 신뢰도 |

이 표에서 가장 자주 오해받는 자리가 **품질 결정 요소** 열입니다. 처음 RAG를 만드는 사람들은 답 품질이 거의 LLM 모델 선택에서 정해진다고 생각하기 쉽습니다. 그러나 실제로는 청크 크기 하나, 거리 함수 하나가 답의 정확도를 10% 이상 바꿔 놓는 일이 흔합니다. 예를 들어 청크를 200토큰으로 자르면 한 사실이 두 조각에 걸쳐 잘려서 검색이 둘 다 빗나가고, 800토큰으로 자르면 잡음이 너무 많이 묻어 들어와 모델이 핵심을 놓칩니다. 같은 문서, 같은 모델로도 400토큰과 800토큰의 답 품질이 눈에 띄게 갈립니다.

또 하나 알아 두어야 할 구분은 **최소 구성**과 **운영 구성**의 차이입니다. 빠르게 시제품을 만들 때는 Loader, Parser, Chunker, Embedding, Vector DB, Retriever, Generator 일곱 개만 있어도 작동합니다. Reranker와 Citation, Eval은 없어도 답은 나옵니다. 그러나 실제 서비스에 올리는 순간 이야기가 달라집니다. Reranker가 없으면 top-5 중 정답이 4등에 묻혀 모델이 보지 못하는 일이 잦고, Citation이 없으면 답이 그럴듯해도 사용자가 검증할 길이 없으며, Eval이 없으면 모델이나 청커를 한 번 바꾼 뒤 품질이 떨어졌는지 좋아졌는지 알 도리가 없습니다.

작은 사례로 마무리하겠습니다. 4년차 백엔드 개발자가 사내 RAG를 처음 만들 때 흔히 이런 순서를 밟습니다. 일주일 차에는 PDF를 LangChain Loader로 읽고 500토큰 단위로 잘라 OpenAI text-embedding-3-small로 임베딩한 다음 pgvector에 넣어 top-3을 그대로 GPT-4o-mini 프롬프트에 붙여 답을 만들어 봅니다. 잘 되는 질문은 잘 되지만, 표가 들어간 문서나 약어가 많은 질문은 자주 틀립니다. 이 단계에서 더 큰 모델로 바꿔 보지만 차이가 거의 없습니다. 이때부터 사람들은 청크 크기를 바꾸고, BM25를 섞고, Reranker를 붙이는 길로 들어섭니다. 이 책의 5장 나머지 단원들은 정확히 그 길을 단계별로 풀어 놓습니다.

정리하면, RAG는 Ingestion과 Query 두 파이프라인으로 나뉘며 Loader, Parser, Chunker, Embedding, Vector DB, Retriever, Reranker, Generator, Citation, Eval 열 개의 부품이 사슬처럼 연결됩니다. 답 품질을 좌우하는 자리는 모델이 아니라 부품 사이의 설정값들이고, 최소 구성으로 출발해 평가 지표를 보고 각 부품을 튜닝해 가는 순환이 RAG 운영의 본 모습입니다.

다음 단원인 5.1.2에서는 가장 첫 부품인 Loader와 Parser, Chunker로 들어가, 원본 문서를 깨끗하게 읽어 의미 단위로 자르는 구체적인 방법을 다룹니다.
