## 2. 모범답안 (Model Answers)

### 🟢 Version A — Elevator (약 30초, 쉬운 영어)

> In short, I built an automated pipeline that evaluated about **six hundred thousand** news-summary outputs from a production LLM — with **no human review**. The hard part was cost. Calling an LLM for every quality check on six hundred thousand items would have been way too expensive. So I bundled **five** quality criteria into a **single** LLM call instead of five separate ones. That cut the cost to about **one fifth** while keeping the quality. It ran on a **cron** schedule, completely unattended, and emailed a report every day.

> **핵심 표현**
> - "with no human review" = 사람 개입 없이 (= unattended / hands-off)
> - "way too expensive" = 너무 비싼 (구어체 강조, way = 훨씬)
> - "bundled A into B" = A를 B로 묶다
> - "cut the cost to about one fifth" = 비용을 약 1/5로 줄이다
> **전달 팁:** "six hundred thousand"와 "one fifth"에서 또박또박 멈추세요. 숫자가 이 답변의 힘입니다.

### 🟡 Version B — Standard (약 75초, 자연스러운 영어) ← 기본값

> So, this was an **LLMOps** project. We had a summarization model running in production, and it had already generated around **600,000** Korean news summaries. We needed to know if the quality was actually good — but checking that many by hand is impossible, so we had to use an **LLM as a judge**.
>
> The real problem was **cost**. If I scored every summary on every quality dimension with a separate LLM call, the token bill and the runtime would explode. So the **key decision** was to compress the evaluation. I split the metrics into two groups: **quantitative** ones that need no LLM at all — like the valid-response rate and the length ratio, which I compute directly from the output — and a single **quality** score that needs just **one** LLM call per item.
>
> For that quality score, there were five criteria — content, dates, hallucination, style, and format — and they're all tangled together, so scoring them separately doesn't really make sense. I bundled all five into **one** GEval call that returns a score and a reason in one shot. That dropped the calls from five to one per item — about a **fifth** of the cost — without losing quality.
>
> The whole thing runs unattended on **cron**: it evaluates new summaries every hour, loads results into Postgres and Excel, and emails a daily report. So whenever the production model changes, we catch any quality regression on the very next cycle.

> **핵심 표현**
> - "LLM as a judge" = LLM으로 LLM을 평가 (업계 표준 용어, 꼭 쓰세요)
> - "the token bill ... would explode" = 토큰 비용이 폭발한다
> - "compress the evaluation" = 평가를 압축하다
> - "tangled together" = 서로 얽혀 있다 (= interlocked)
> - "in one shot" = 한 번에
> - "catch a regression" = 품질 저하를 잡아내다
> **전달 팁:** 3개 비트(Problem-Decision-Result)에서 잠깐씩 멈춰 호흡을 끊으면 구조가 또렷이 들립니다.

### 🔴 Version C — Deep Dive (약 2~3분, 고급 영어) ← "더 자세히" 요청 시

> Sure, let me go a bit deeper on the design.
>
> The constraint was scale: roughly **600,000** production summaries, evaluated cheaply but credibly. My first move was to **separate what needs an LLM from what doesn't**. Things like the valid-response rate and the summary-to-source length ratio are **deterministic** — I calculate them straight from the result JSON, no model call. That leaves only one genuine LLM judgment per item, so cost scales linearly and predictably.
>
> For that judgment, the tricky part is that our five criteria — content preservation, relative-to-absolute **date** conversion, hallucination, style, and formatting — are **interdependent**. Scoring them as five separate metrics would mean five LLM calls per item, and it also fragments the judgment. So I **overrode** deepeval's default `GEvalTemplate` with a Korean version — `KoreanGEvalTemplate` — and folded all five criteria into the **evaluation steps** of a **single** GEval metric. One call returns both a 0-to-1 score and a Korean rationale. That's the five-to-one reduction. I also **inject the article's issued date** into the prompt, because the single most common failure was the model mis-converting relative dates like "last year" into the wrong absolute year — and with the issued date in context, the judge can verify that precisely.
>
> On the operations side, the goal was a **hands-off, non-stop** pipeline. Evaluation is **incremental and idempotent**: I key each result by source, subdirectory, and filename, score only the ones not yet evaluated, and **upsert**. So a backlog of 600,000 never piles up — cron just chips away at it every hour. I guard against overlapping runs with `flock`, and cap throughput per run. The one failure mode I had to handle was the OpenAI **rate limit**: when that happens the score comes back null, so I wait and **retry only those null rows**, and skip the wait entirely if there are none. Results land in both Excel — a summary sheet and a details sheet — and Postgres, with `eval_runs` and `eval_details` tables plus aggregate views. A daily email closes the loop. Because the aggregation is per-folder over time, if someone swaps the production model and quality drops, the regression shows up on the next cron cycle automatically — nobody has to remember to check.

> **핵심 표현 (고급)**
> - "scales linearly and predictably" = 선형적·예측 가능하게 확장
> - "deterministic" = 결정론적 (모델 없이 계산되는)
> - "interdependent / fragments the judgment" = 상호의존적 / 판단을 쪼갬
> - "override the default template" = 기본 템플릿을 오버라이드
> - "incremental and idempotent" = 증분적이고 멱등한 (= 여러 번 돌려도 안전)
> - "upsert" = 있으면 갱신, 없으면 삽입
> - "chips away at it" = 조금씩 처리해 나간다
> - "closes the loop" = 루프를 닫는다 (자동화 완성)
> **전달 팁:** 이 버전은 길어서, 면접관 표정을 보며 "Want me to go further on any part?"로 중간 점검하면 좋습니다.

## 3. 꼬리질문 대비 (Follow-up Q&A)

**Q1. Why bundle five criteria into one call instead of scoring each separately?**
> Two reasons. First, **cost** — five separate metrics mean five LLM calls per item, and at 600,000 items that's a huge bill. Second, the criteria are **interdependent**; a date error is also kind of a content error, so judging them in isolation actually makes the scoring noisier. One holistic judgment was both cheaper and more coherent.
> *(한국어 메모: 비용 + 일관성 두 축으로 답하면 설계 의도가 분명해집니다.)*

**Q2. How can you trust an LLM to judge another LLM?**
> I don't trust it blindly. The **deterministic** metrics act as a guardrail — if length ratio or valid-response rate looks off, that's a hard signal independent of the judge. The judge also returns a **rationale**, not just a score, so failures are auditable, and I spot-check samples. For a stricter setup I'd add a small **human-labeled gold set** to measure the judge itself, but for this scale and timeline the guardrails plus spot-checks were the right tradeoff.
> *(한국어 메모: "맹신 안 한다 + 가드레일 + 사람 검증 여지"가 성숙한 답입니다.)*

**Q3. What was the hardest part?**
> Honestly, the scale combined with **reliability**. Anyone can run an eval once; running it on 600,000 items, unattended, against API rate limits, without piling up a backlog — that's the engineering. The incremental upsert plus `flock` plus retry-only-the-failures was what made it actually survivable in production.

**Q4. How did you handle API rate limits and failures?**
> When the rate limit hits, the score comes back as null. I isolate **only those null rows**, wait, and retry just them — and if there are zero failures, I skip the wait completely so I'm not adding latency for nothing. Combined with the hourly incremental run, failures just get picked up on the next pass.

**Q5. How would you detect that the production model regressed?**
> The aggregation is **per-folder and over time** in Postgres views. So a model swap that drops quality shows up as a metric drop on the **next cron cycle** — automatically, without anyone remembering to look. That was the whole point: turn "someone should check" into "the system checks."

**Q6. What would you do differently next time?**
> I'd build a small **gold-standard labeled set** up front to calibrate the judge, and I'd consider a cheaper or local judge model for the first-pass filter, reserving the stronger model only for borderline cases. That would cut cost even further.
