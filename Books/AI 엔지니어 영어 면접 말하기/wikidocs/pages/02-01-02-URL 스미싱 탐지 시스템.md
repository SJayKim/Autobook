> **한 줄:** 외부 스캐닝 API 의존을 없애고, URL 텍스트(URLBert)와 HTML 구조를 직접 분석해 스미싱을 탐지하는 자체 파이프라인.
> **헤드라인 숫자:** 실환경 F1 28.79% → 96.10%(1:1 균형) · 9:1 실환경 95%+ · 17개 HTML 피처 · API 장애 자동 감지(143 → 0건).
> 소속 Plantynet · 2명 팀 · 2025.07~2026.03 · 스미싱 분류 모델 개선 / HTML 탐지 파이프라인 / 성능 모니터링 전담.

## 2. 모범답안 (Model Answers)

### 🟢 Version A — Elevator (약 30초, 쉬운 영어)

> In short, I rebuilt our smishing detection system so it **no longer depends** on an external scanning API. The old system called that API to get URL data and then used a Random Forest — but the API failed a lot, and on real traffic the F1 score was only about **twenty-eight percent**. So I fine-tuned a URL-specific model called **URLBert** to read the URL text directly. That took the F1 up to about **ninety-six percent** on balanced data, and it stayed above **ninety-five percent** on real, imbalanced traffic. On top of that, I added monitoring that watches the system in production — and it actually caught a full API outage on its own, when the data dropped from **143** successful calls to **zero**.

> **핵심 표현**
> - "no longer depends on" = 더 이상 ~에 의존하지 않는다
> - "failed a lot" = 자주 실패했다 (구어체 강조)
> - "read the URL text directly" = URL 텍스트를 직접 읽다
> - "took the F1 up to ___" = F1을 ~까지 끌어올리다
> - "caught ___ on its own" = ~을 스스로 잡아냈다
> **전달 팁:** "twenty-eight percent"와 "ninety-six percent"에서 또박또박 멈추세요. 28 → 96 점프가 이 답변의 힘입니다.

### 🟡 Version B — Standard (약 75초, 자연스러운 영어) ← 기본값

> So, this was a smishing detection project — smishing is basically phishing over text messages, usually with a malicious **URL** inside. We already had a system, but it relied on an **external scanning API** to pull metadata about each URL, and then a **Random Forest** classified it. The problem was that the API failed often, and on real traffic — which is heavily imbalanced, about **nine-to-one** — the F1 was stuck at around **twenty-eight percent**. Short URLs were the worst case, because there's no **Whois** record to look up, so they were almost always misclassified.
>
> The **key decision** was to drop the external API entirely and learn from the URL text itself. I fine-tuned **URLBert**, a model pretrained specifically on URLs — I picked it because it's URL-specialized, it runs on **CPU** only, and it handles Korean-style short URLs well. Zero-shot it was weak, around **forty-six percent**, so I fine-tuned it on about **4,300** of our own labeled URLs. After that the F1 jumped to **96 percent** on balanced data, and it held above **95 percent** even on the real nine-to-one traffic.
>
> The **tricky part** was short URLs, like bitly links — the text alone tells you nothing, because the real content is on the landing page. So I built a second pipeline that follows the **redirect** to the final landing page and analyzes its **HTML** — 17 features in total. And I didn't stop at the model: I added production **monitoring** that compares each run to the last twenty, so if the model **drifts** or the external API breaks, it alerts us automatically. It actually caught a real API outage — calls dropped from **143** to **zero** — and fired a critical alert right away, instead of someone finding it in the logs a day later.

> **핵심 표현**
> - "smishing is basically phishing over text messages" = 스미싱 정의를 한 문장으로 깔고 시작
> - "relied on an external scanning API" = 외부 스캐닝 API에 의존했다
> - "stuck at around ___" = ~에 머물러 있었다
> - "the worst case" = 최악의 경우
> - "drop the external API entirely" = 외부 API를 완전히 없애다
> - "held above 95 percent" = 95% 이상을 유지했다
> - "follows the redirect" = 리다이렉트를 따라간다
> - "the model drifts" = 모델 성능이 변질된다
> **전달 팁:** Problem → Decision → Hard part 비트 사이에서 0.5초씩 끊으면 구조가 또렷이 들립니다. "28 → 96"에서 목소리를 한 번 올리세요.

### 🔴 Version C — Deep Dive (약 2~3분, 고급 영어) ← "더 자세히" 요청 시

> Sure, let me walk through the design.
>
> The starting constraint was that our detection depended on an **external scanning API**. It pulled metadata per URL and a **Random Forest** classified it, but the failure rate was high, and on production traffic — imbalanced roughly **nine-to-one** toward legitimate URLs — the F1 was only **28.79 percent**. The killer case was shortened URLs: with no **Whois** record to query, the old pipeline had almost nothing to go on, so it misclassified most of them. So my first decision was to stop depending on anything external and learn straight from the URL string.
>
> For that I fine-tuned **URLBert**. I chose it for three reasons: it's pretrained specifically on the URL domain, it runs on **CPU** alone — no GPU budget needed for training or inference — and it works well on Korean-style shortened URLs. Zero-shot it only hit **46 percent** F1, so I fine-tuned it on **4,296** of our own labeled URLs. On a balanced one-to-one set that took it to **96.10 percent** F1 — precision **97.6**, recall **94.65** — versus about **58 percent** for the old Random Forest. And it held above **95 percent** F1 with precision over **97** on the real nine-to-one distribution, at roughly **0.05 milliseconds** per URL.
>
> But text-only classification can't see through a shortened link, because the original URL and the actual landing page are different. So I built a second pipeline that follows the HTTP **redirect** to the final landing page and analyzes its **HTML**. I ported **15** features from an open-source phishing detector and added **two** Korean-specific ones — `has_tel_input`, which flags a phone-number input field, and `has_kr_keyword`, which looks for Korean scam triggers like parcel delivery, identity verification, obituary, or investment. That's **17** HTML features, and on a balanced set it scored **96.47 percent** F1. Since fetching pages is the slow part, I parallelized collection with a **ThreadPoolExecutor** at twenty concurrent requests.
>
> The last piece was making it survivable in production. I built a three-stage pipeline — **ETL**, then model evaluation, then alerting and logging — and kept it **low-coupling** by storing the last **twenty** runs in a `history.json` and comparing each new run against that window. With thresholds on the **label ratio** and F1, it raises a warning or a critical alert when the model **drifts** or the data looks wrong. And the payoff was real: during operation it auto-detected a full external-API outage — successful collections dropped from **143** to **zero**, a hundred-percent failure rate — and fired a critical alert on the spot. Before that, this kind of outage was only caught when someone manually read the logs on the next run.

> **핵심 표현 (고급)**
> - "learn straight from the URL string" = URL 문자열에서 곧바로 학습
> - "the killer case was ___" = 결정적으로 어려운 경우는 ~ (killer = 치명적인)
> - "zero-shot" = 미세조정 없이 사전학습만으로
> - "held above 95 percent" = 95% 이상을 유지했다
> - "follows the HTTP redirect" = HTTP 리다이렉트를 따라간다
> - "ported 15 features from ___" = ~에서 15개 피처를 이식(포팅)했다
> - "low-coupling" = 낮은 결합도 (모듈 간 의존을 최소화)
> - "survivable in production" = 운영에서 견디는, 실전에서 버티는
> **전달 팁:** 길어서 중간에 한 번 "Want me to go deeper on the HTML side or the monitoring?"로 면접관과 호흡을 맞추면 좋습니다.

## 3. 꼬리질문 대비 (Follow-up Q&A)

**Q1. Why remove the external API instead of just making it more reliable?**
> A few reasons. The failure rate was high and outside our control, and for shortened URLs the API often had no data at all — there's no Whois record to fetch. So even a perfectly reliable API wouldn't fix the worst case. Learning from the URL text directly removed the dependency, cut the latency, and actually scored far higher. So it wasn't only about reliability — it was accuracy and control too.
> *(한국어 메모: "신뢰성만의 문제가 아니라 정확도 + 통제권"으로 답하면 결정이 입체적으로 보입니다.)*

**Q2. Why URLBert specifically, and not just features plus a Random Forest?**
> Three things. It's pretrained on the URL domain, so it already understands URL structure. It runs on CPU only, which mattered because we had no GPU budget. And it handles Korean-style shortened URLs well. The old Random Forest on API features only reached about 58 percent F1, and fine-tuned URLBert beat it by a wide margin — 96 — so the choice paid off.
> *(한국어 메모: 선택 이유 3개를 손가락으로 세며 말하면 구조가 또렷합니다.)*

**Q3. How did you handle shortened URLs, since the text alone is useless there?**
> That was exactly the hard part. The text classifier can't see the real content behind a bitly link, so I added a separate pipeline that follows the redirect to the final landing page and analyzes its HTML — 17 features, including two I added for Korean scams. So the two pipelines complement each other: text for the URL, structure for the page it lands on.
> *(한국어 메모: "두 파이프라인이 상호보완"이 핵심 메시지입니다. 한 문장으로 못 박으세요.)*

**Q4. What were the two Korean-specific HTML features, and why add them?**
> The open-source detector gave me 15 generic phishing features, but Korean smishing has its own patterns. So I added `has_tel_input`, which detects a phone-number input field — common on fake identity-verification pages — and `has_kr_keyword`, which looks for words like parcel delivery, identity verification, obituary, or investment. Those two captured local scam patterns the generic features missed.
> *(한국어 메모: 도메인 지식을 피처로 녹였다는 점이 시니어 신호입니다.)*

**Q5. How does the monitoring actually detect a problem?**
> It's a three-stage pipeline — ETL, evaluation, then alerting — and it keeps the last twenty runs in a history.json. Each new run is compared against that window, so it stays low-coupling, with no heavy state. If the label ratio or the F1 crosses a threshold, it raises a warning or a critical alert. That's how it caught the API outage: successful collections went from 143 to zero, and it fired a critical alert immediately.
> *(한국어 메모: "최근 20회와 비교 → 임계값 → 알림" 흐름을 순서대로 짚으세요.)*

**Q6. What would you do differently next time?**
> The monitoring state lives in a history.json file, which is simple and low-coupling, but at a larger scale I'd move that history into a proper database so it's queryable and more durable. I'd also keep expanding the labeled fine-tuning set beyond the 4,296 we used, so the model stays robust as scam patterns evolve.
> *(한국어 메모: 한계 인정 + 개선 방향 = 성숙함. 지어내지 말고 실제 설계의 한계만 말하세요.)*
