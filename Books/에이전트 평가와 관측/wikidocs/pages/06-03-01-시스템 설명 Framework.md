# 6.3.1 시스템 설명 Framework

이 책의 마지막 두 단원은 지금까지 쌓아 온 평가·관측·거버넌스 지식을 외부에 설명하는 자리에서 어떻게 꺼내 쓸지에 관한 것입니다. 잘 만든 시스템도 짧은 시간 안에 일관되게 설명하지 못하면 그 가치가 절반밖에 전달되지 않습니다. 특히 면접이나 기술 문서, 사내 발표처럼 시간이 한정된 자리에서는 같은 내용을 매번 다른 순서로 풀어 놓으면 듣는 쪽이 시스템의 전체 그림을 머릿속에 그리지 못합니다. 이 단원에서는 본인의 멀티에이전트 시스템을 설명할 때 쓸 수 있는 표준 형식을 잡아 보겠습니다.

표준 형식의 뼈대는 5.5에서 본 다섯 개 차원입니다. 모델 접근(Model Access), 오케스트레이션(Orchestration), 메모리 거버넌스(Memory Governance), 도구 거버넌스(Tool Governance), 관측(Observability) 다섯 가지입니다. 이 다섯 차원은 한 회사가 임의로 만든 분류가 아니라, 비결정적 에이전트를 프로덕션에 올릴 때 누구나 마주치는 공통 질문을 정리한 것입니다. 그래서 본인이 어떤 프레임워크로 시스템을 만들었든, 어떤 도메인에 적용했든, 이 다섯 차원에 답할 수 있어야 시스템이 production-ready 라고 말할 수 있습니다. 면접이나 문서에서도 이 다섯 차원을 그대로 답변의 뼈대로 쓰면 듣는 쪽이 누락 없이 전체 그림을 받을 수 있습니다.

다섯 차원을 답할 때의 표준 순서는 다음과 같이 잡습니다. 먼저 모델 접근 차원에서는 어떤 모델을 쓰는지, 그 모델을 단일 API로 추상화했는지, 데이터가 모델 제공자 쪽에 남는지를 답합니다. 다음 오케스트레이션 차원에서는 에이전트가 어떤 구조로 협업하는지, 런타임이 stateless 인지, 요청 단위로 isolation 되는지를 답합니다. 메모리 거버넌스 차원에서는 사용자 프로필·세션·지식 베이스를 어떻게 분리하는지, 권한 통제는 어떻게 거는지를 답합니다. 도구 거버넌스 차원에서는 어떤 도구를 노출하고 어떤 가드레일과 HITL 승인이 걸려 있는지를 답합니다. 마지막 관측 차원에서는 어떤 trace를 남기고 어떤 메트릭으로 회귀를 잡는지를 답합니다. 이 다섯 가지를 차례로 답하고 나면 시스템 설명이 한 번에 완결됩니다.

한국 기업 면접 환경에서 이 framework가 특히 강한 이유는 빈출 질문 패턴이 모두 이 다섯 차원의 어딘가에 떨어지기 때문입니다. 토스·카카오뱅크·KB·신한처럼 금융권 색이 강한 곳일수록 거버넌스와 관측 차원의 비중이 높습니다. "본인의 멀티에이전트 시스템을 어떻게 production화 했나"라는 큰 질문이 오면 이 다섯 차원으로 답하면 거의 100점입니다. "데이터는 어디에 남나"는 모델 접근 차원, "장애가 나면 어떻게 격리하나"는 오케스트레이션 차원, "사용자 정보 권한은 어떻게 거나"는 메모리 거버넌스 차원, "위험한 행동은 어디서 막나"는 도구 거버넌스 차원, "회귀가 났는지 어떻게 아나"는 관측 차원으로 받아치면 됩니다. 질문이 어느 차원에 속하는지만 빠르게 식별하면 답변의 시작 지점을 망설이지 않게 됩니다.

답변에서 자주 빠지는 요소가 trade-off 명시입니다. 다섯 차원을 다 채워도 "그 결정의 반대편에는 무엇이 있는가"를 빼면 답변이 카탈로그처럼 들립니다. 좋은 답변은 각 차원에서 본인이 어떤 선택을 했고, 그 선택의 반대 옵션은 무엇이었으며, 왜 한쪽을 택했는지를 한 줄씩 붙입니다. 예컨대 메모리 거버넌스 차원에서 "세션 메모리 만료를 24시간으로 잡았다"고만 말하면 평면적이지만, "24시간으로 잡으면 사용자 경험이 매끄럽지만 장기 추적이 약해지고, 더 길게 잡으면 추적은 되지만 잊힐 권리 대응이 복잡해진다, 우리는 잊힐 권리를 우선해 24시간을 택했다"고 말하면 같은 결정이 한 단계 깊어집니다. Klarna 사례에서 본 것처럼 단일 메트릭에 최적화한 시스템은 hidden quality를 잃는데, 답변에서 trade-off를 명시하지 않으면 면접관이 보기에는 단일 차원에 최적화한 사람과 구별이 되지 않습니다.

깊이를 한 단계 더 보여 주는 기법이 역질문입니다. 다섯 차원으로 답을 마친 뒤 "이 회사 환경에서는 어느 차원이 가장 약하다고 보십니까", "현재 운영 중인 LLM 시스템에서 관측 baseline은 어디까지 잡혀 있습니까" 같은 질문을 짧게 던지는 것이 효과적입니다. 역질문은 단순히 시간을 끌기 위한 장치가 아니라, 본인이 다섯 차원의 균형을 의식하고 있고 상대 조직의 맥락에 맞춰 우선순위를 다시 잡을 줄 안다는 신호를 줍니다. 다만 역질문은 답변을 회피하는 도구가 아니므로, 본인의 답을 충분히 한 뒤에 마무리로 한두 개만 붙여야 합니다.

다섯 차원을 답변 형식에 어떻게 녹이는지 짧은 박스로 그려 보면 다음과 같습니다.

```
[ 면접 답변 표준 박스 ]

(1) 모델 접근  : 단일 API 뒤에 OpenAI·Anthropic 라우팅, 제공자 retention 없음
                trade-off: 단일 벤더 종속을 피했지만 라우팅 로직 운영 부담 증가
(2) 오케스트레이션 : Lead + 병렬 Subagent, stateless runtime, request 단위 isolation
                trade-off: 수평 확장 쉬워졌지만 cross-request 캐싱 이득은 포기
(3) 메모리 거버넌스 : 프로필·세션·지식 베이스 분리, JWT 기반 RBAC
                trade-off: 권한 검증 한 단계 늘었지만 잊힐 권리 대응 가능
(4) 도구 거버넌스 : 가드레일 + 임계치 기반 HITL, 비가역 작업만 승인
                trade-off: 전수 승인 대비 사용자 경험 매끄럽지만 임계치 튜닝 필요
(5) 관측      : OTel 표준 + Langfuse self-host, full session trace
                trade-off: 자체 호스팅 운영 비용 있지만 데이터 주권 확보
```

이 박스 형태는 한 번에 다 외우는 것이 아니라, 본인 시스템의 각 차원을 두 줄(선택과 trade-off) 안에 압축해 두고 면접 직전에 빠르게 점검하는 도구로 씁니다. 차원별로 두 줄씩 다섯 차원이면 열 줄, 시간으로는 2~3분 안에 시스템 전체 그림이 전달됩니다. 그 뒤로 면접관이 어느 차원을 더 깊게 묻든 박스의 한 줄을 확장해 답하면 됩니다.

마지막으로 짚을 점은 이 framework가 본인 시스템에 없는 차원을 숨기는 도구가 아니라는 것입니다. 어떤 차원이 약하면 약한 대로 솔직하게 답하고, 그 차원을 어떤 우선순위로 보강할지 같이 말하는 것이 더 강한 인상을 줍니다. 6.1의 self-audit과 6.2의 도입 로드맵을 함께 묶어서 "현재는 관측 차원이 baseline 수준이고, 다음 분기에 메모리 거버넌스 차원을 보강할 계획이다"라고 답하면, 다섯 차원을 다 채우지 못한 상태라도 의사결정 능력 자체는 충분히 전달됩니다. 거버넌스가 속도를 죽이지 않도록 균형을 잡은 6.2.3의 사고가 답변에 그대로 묻어납니다.

정리하면, 시스템 설명 framework는 5.5의 다섯 차원(모델 접근·오케스트레이션·메모리 거버넌스·도구 거버넌스·관측)을 답변의 뼈대로 쓰고, 각 차원에서 선택과 trade-off를 한 줄씩 붙이며, 마지막에 역질문으로 깊이를 보여 주는 표준 형식입니다. 이 형식을 한 번 잡아 두면 토스·카카오뱅크·KB·신한 같은 한국 금융권 면접에서 쏟아지는 production화 질문들에 일관되게 답할 수 있고, 본인 시스템의 약한 차원도 우선순위 언어로 풀어 낼 수 있습니다.

다음 단원인 6.3.2에서는 이 framework를 답변할 때 어떤 사례를 어느 시점에 인용할지, 그리고 단순 인용을 넘어 비교·대조와 한계까지 함께 제시하는 방법을 정리하고 책 전체를 마무리하겠습니다.

이 단원을 마치면 면접·문서에서 본인의 멀티에이전트 시스템을 다섯 차원 framework로 일관되게 설명하고, 각 차원에서 trade-off를 명시하며, 역질문으로 깊이를 더하는 답변 흐름을 구성할 수 있습니다.
