# 1.2.1 Tool selection error

앞 절에서는 에이전트 평가가 LLM 평가와 어떻게 다른지, trajectory 평가와 final-answer 평가가 어떻게 갈라지는지를 살펴보았습니다. 이번 절부터는 에이전트가 실제로 망가지는 모습을 유형별로 정리합니다. 이렇게 자주 반복되는 실패의 묶음을 이 책에서는 실패 모드 분류라고 부르며, 그 첫 번째 자리를 차지하는 것이 도구 선택 오류입니다.

먼저 용어부터 풀겠습니다. 에이전트는 한 번의 응답으로 답을 내놓는 대신, 매 단계에서 '지금 무엇을 할까'를 정해야 합니다. 이 중 가장 많이 반복되는 결정이 '어떤 도구를 부를까'입니다. 에이전트에게 노출된 도구는 보통 검색기, 계산기, 데이터베이스 조회기, 파일 편집기처럼 이름과 설명을 가진 함수의 형태로 제시됩니다. 모델은 사용자의 질문과 자신의 직전 추론을 보고 그중 하나를 골라 호출합니다. 이 결정 단계에서 의도와 다른 도구를 부르는 현상을 **도구 선택 오류**라고 합니다.

도구 선택 오류는 도구 호출 자체가 실패한 경우와 다릅니다. 도구 호출 실패는 네트워크 끊김이나 인자 검증 오류처럼 도구 쪽이 신호를 돌려주는 사건이고, 도구 선택 오류는 도구가 정상적으로 동작했지만 애초에 부적절한 도구를 골랐다는 사건입니다. 후자가 더 까다로운 까닭은 응답 자체는 멀쩡해 보이는데 결과가 사용자가 원한 것과 다르기 때문입니다.

가장 흔한 패턴은 **비슷한 이름의 도구를 혼동**하는 경우입니다. 예를 들어 한 시스템에 search_orders, search_order_items, search_invoices 세 도구가 동시에 노출되어 있다고 합시다. 사용자가 '지난주에 산 노트북 영수증'을 묻습니다. 모델이 search_orders를 부르면 주문 단위는 잘 잡히지만, 노트북이라는 품목 정보까지 들어가지 않아 빈 응답이 돌아옵니다. 모델이 그 결과를 보고도 '검색이 비었으니 데이터가 없습니다'라고 단정하고 끝내면, 표면적으로는 정상 응답이지만 실제로는 search_order_items나 search_invoices 쪽으로 갔어야 했던 호출이 누락된 셈입니다.

원인은 보통 모델보다 **도구 설명문**에 있습니다. 도구 설명문은 모델이 도구를 고르는 유일한 단서입니다. 사람이라면 동료에게 '주문 검색이랑 영수증 검색 중 어떤 게 영수증을 찾아 줘?'라고 물어볼 수 있지만, 모델은 그런 자유가 없고 한 번에 결정을 내려야 합니다. 그래서 설명문이 모호하면 모델은 이름의 유사성으로 판단합니다. search_orders 설명에 '주문 내역을 검색합니다'만 적혀 있고 search_invoices 설명에 '청구·영수 정보를 검색합니다'라고만 적혀 있으면, 모델 입장에서는 '영수증 = 청구 정보'를 묶기 어렵습니다. 도구 이름이 비슷할수록 설명문은 더 길고 더 구체적이어야 합니다.

설명문의 영향을 가시화하기 위해 하나의 도구 카탈로그가 어떻게 갈라지는지를 표로 정리해 보겠습니다.

```
도구 이름          설명문 품질        모델의 호출 비율 (예시)
-----------------  ------------------ ----------------------
search_orders      구체·예시 포함     영수증 질문에 거의 안 부름
search_invoices    구체·예시 포함     영수증 질문에 자주 부름
search_orders      모호한 한 문장     영수증 질문에 자주 오인 호출
search_invoices    모호한 한 문장     영수증 질문에 가끔만 부름
```

이 표가 보여 주는 핵심은 도구 선택 오류가 모델의 능력보다 도구 설명문의 품질에 더 크게 좌우될 때가 많다는 점입니다. 같은 모델, 같은 사용자 질문이라도 설명문 한 줄을 어떻게 쓰느냐에 따라 호출 비율이 뒤집힙니다. 그래서 도구 선택 오류를 줄이려는 시도는 보통 모델 변경이 아니라 설명문과 카탈로그 정리에서 시작합니다.

이제 평가 쪽 이야기로 넘어가 보겠습니다. 도구 선택 오류는 final-answer 평가만으로는 잘 잡히지 않습니다. 빈 응답이거나 어색한 응답일 뿐, 채점자가 보기에 '부정확'이라고 단정하기 애매한 경우가 많기 때문입니다. 그래서 **도구 선택 오류를 잡는 평가 항목**은 trajectory 위에 얹습니다. 가장 단순한 형태는 단계별로 '예상 도구'와 '실제 호출 도구'를 비교해 일치하는 비율을 재는 도구 선택 정확도입니다. 골든 데이터셋의 각 케이스에 '이 입력에는 search_invoices가 가야 한다' 같은 라벨을 달아 두고, 실행 결과에서 호출된 도구가 라벨과 같은지를 본다는 뜻입니다. 여기에 '잘못된 도구를 부른 뒤 정정해서 결국 정답에 도달했는지'를 따로 표시하면, 단순 비율보다 풍부한 그림이 나옵니다.

도구 선택 오류가 발견되면 그다음 단계에서 곧장 따라오는 결정이 있습니다. 잘못 부른 호출에 대해 에이전트를 어떻게 처리할지입니다. 선택지는 크게 **복구**, **재시도**, **종료** 세 가지입니다. 복구는 잘못된 호출 결과를 받아도 모델이 그 자리에서 다른 도구를 다시 부르도록 두는 방식입니다. 재시도는 같은 도구를 인자만 바꾸어 다시 호출하는 방식이고, 종료는 더 진행하지 않고 사용자에게 명확한 메시지와 함께 흐름을 끝내는 방식입니다. 세 방식의 의미는 다음 표로 정리할 수 있습니다.

```
결정     의미                                    적합한 상황
-------- --------------------------------------- ------------------------
복구     다른 도구로 갈아타 진행                 비슷한 도구가 여러 개일 때
재시도   같은 도구를 다른 인자로 다시 호출       인자 누락·형식 오류일 때
종료     흐름을 멈추고 사용자에게 결과를 전달    유사 도구도 모두 빈 응답일 때
```

가장 흔한 함정은 모든 오류에 대해 **복구**를 기본값으로 잡아 두는 것입니다. 복구가 늘어날수록 단계 수가 길어지고, 토큰과 시간이 함께 늘어납니다. 게다가 복구가 새로운 도구 선택 오류를 또 만들면 다시 복구가 따라오면서 단계가 계속 쌓입니다. 그래서 실전에서는 '같은 카테고리 도구 안에서는 1회 복구, 그 밖에는 종료' 같은 정책을 미리 정해 두고, 그 정책 자체를 평가 항목으로 둡니다.

정리하면, 도구 선택 오류는 의도와 다른 도구가 호출되는 현상으로, 보통 도구 설명문의 모호함과 비슷한 이름의 도구가 동시에 노출된 카탈로그 구조에서 비롯됩니다. 이 오류는 final-answer만으로는 잘 잡히지 않으므로 trajectory 위에 도구 선택 정확도 같은 평가 항목을 얹어 잡고, 발견된 오류에 대해 복구, 재시도, 종료 중 어떤 결정을 기본값으로 둘지를 함께 정해 두어야 합니다.

다음 단원인 1.2.2에서는 이 도구 선택 오류와 결이 다른 또 하나의 실패 모드, 같은 단계를 빙빙 도는 planning loop를 살펴보겠습니다.

이 단원을 마치면 도구 선택 오류의 정의와 비슷한 이름의 도구를 혼동하는 사례를 들 수 있고, 도구 설명문 품질이 선택에 어떻게 영향을 주는지, 도구 선택 오류를 잡는 평가 항목과 복구·재시도·종료 결정의 차이를 설명할 수 있습니다.
