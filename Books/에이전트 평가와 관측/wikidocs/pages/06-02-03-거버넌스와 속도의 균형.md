# 6.2.3 거버넌스와 속도의 균형

KPI를 정하고 운영 호흡을 잡고 나면, 마지막으로 마주치는 질문은 거버넌스가 속도를 죽이지 않게 하는 방법입니다. 거버넌스는 사고를 막아 주지만, 잘못 설계하면 변경을 검토하는 데에만 몇 주가 걸리고, 모델 하나를 바꾸려고 위원회 세 곳을 거쳐야 하며, PR 한 줄을 머지하는 데 결재가 다섯 단계 필요해집니다. 이렇게 되면 도입 자체가 의미를 잃습니다. 이 단원은 거버넌스의 보호와 변경 속도를 동시에 살리는 설계 패턴을 정리합니다.

표준 답안의 중심에는 **JPMorgan의 8주 update cadence**가 있습니다. 5.3.3에서 본 것처럼 JPMorgan LLM Suite는 250K 직원 규모의 거버넌스 플랫폼을 운영하면서도 8주 단위로 시스템 전체를 갱신합니다. 8주라는 숫자는 두 가지 압력 사이의 절충점입니다. 한쪽에는 매일이라도 모델과 프롬프트를 바꾸고 싶은 개발 속도의 압력이 있고, 다른 쪽에는 변경마다 보안 검토, 컴플라이언스 검토, 모델 카드 갱신, 사용자 영향 평가를 모두 거쳐야 하는 거버넌스의 압력이 있습니다. 매일 바꾸면 거버넌스가 무너지고, 6개월에 한 번씩만 바꾸면 시장 변화에 끌려갑니다. 8주는 두 압력이 만나는 지점입니다.

8주 cadence는 단순히 시간 단위가 아니라 **고정 release train**이라는 운영 모델 위에서 작동합니다. Release train은 정해진 출발 시각에 출발하는 기차에 비유한 모델로, 그 시각에 준비된 변경들만 함께 실어 보내고 늦은 변경은 다음 기차를 기다리는 방식입니다. 변경 단위로 일정을 협상하지 않고, 일정 단위로 변경을 모읍니다. 이렇게 하면 거버넌스 검토가 변경마다 다시 일어나는 대신, 한 사이클에 한 번 일괄로 일어나기 때문에 검토 비용이 일정하게 유지됩니다. 사이클을 그림으로 풀면 다음과 같습니다.

```
   Week 1     Week 2     Week 3     Week 4
   +------+   +------+   +------+   +------+
   |기획  |-> |개발  |-> |개발  |-> |통합  |
   +------+   +------+   +------+   +------+
                                       |
   Week 5     Week 6     Week 7     Week 8
   +------+   +------+   +------+   +------+
   |검증  |-> |거버  |-> |카나리|-> |릴리즈|
   |eval  |   |넌스  |   |배포  |   |+회고 |
   +------+   +------+   +------+   +------+
       \                                /
        +-> 다음 사이클이 곧바로 시작 <-+
```

이 구조에서 중요한 점은 한 사이클이 끝나는 즉시 다음 사이클이 시작된다는 사실입니다. 사이클 사이에 빈 시간이 없으면 거버넌스 검토 자체가 일상적인 작업으로 자리잡고, 검토자도 변경 제출자도 익숙한 리듬에 맞춰 움직이게 됩니다. 검토를 특별한 이벤트로 만들면 매번 새로 협상해야 하지만, 정기적인 흐름으로 만들면 검토가 의식이 됩니다.

Release train만으로는 부족하고, **표준 인터페이스**가 그 위에 있어야 합니다. 표준 인터페이스란 모델이나 도구가 바뀌어도 호출하는 쪽의 코드를 바꿀 필요가 없도록 정해 둔 공통의 입출력 규약입니다. JPMorgan LLM Suite에서 단일 API 뒤에 OpenAI, Anthropic 등 여러 모델이 숨어 있는 구조가 정확히 이 인터페이스의 예시입니다. 클라이언트 앱은 모델 이름을 모르고도 호출하고, 라우팅 규칙이 바뀌어도 클라이언트 변경 없이 새로운 모델로 트래픽이 넘어갑니다. 인터페이스가 안정적이면 그 뒤의 변경은 거버넌스 검토만 통과하면 되고, 외부에 미치는 영향이 0이기 때문에 검토 부담이 작아집니다. 반대로 인터페이스가 매번 같이 바뀌면 한 번의 모델 변경이 수십 개 클라이언트의 동시 변경을 강제하고, 그 모든 클라이언트의 거버넌스 검토를 다시 끌어내게 됩니다.

표준 인터페이스가 모델 쪽이라면, **재사용 가능 컴포넌트**는 도구·메모리·관측 쪽의 같은 원리입니다. 모든 팀이 자기 도구와 자기 메모리와 자기 trace 모듈을 따로 만들기 시작하면 거버넌스 검토는 무한히 늘어나고, 각 팀의 보안 수준도 들쭉날쭉해집니다. 중앙 플랫폼 팀이 검증된 컴포넌트의 카탈로그를 만들고, 사용 팀은 그 카탈로그에서 골라 쓰는 구조가 되면, 새로 검증해야 할 항목이 도메인 로직으로 좁혀집니다. 컴포넌트 한 번 검증으로 N개 팀이 같은 보안 수준을 얻는 셈입니다. 이 효과를 그림으로 정리하면 다음과 같습니다.

```
[ 거버넌스 부담 ]
       ^
       |  팀마다 자체 컴포넌트
       | --------------------------- N팀 x C컴포넌트
       |
       |  공통 카탈로그 + 도메인 로직만
       | --------------------------- 1 x C 검증 + N팀 도메인
       +--------------------> [ 팀 수 ]
```

이 그림에서 두 곡선의 차이가 거버넌스 비용이고, 그 차이가 곧 변경 속도입니다. 카탈로그가 잘 만들어진 환경에서는 팀 수가 늘어나도 거버넌스 비용이 거의 일정하게 유지되며, 그래서 8주 cadence가 무너지지 않습니다.

이 패턴을 한국 대기업과 금융권에 옮길 때 한 가지 더 고려해야 할 축이 있습니다. 국내 금융권은 모델 변경, 도구 추가, 데이터 흐름 변경에 대해 자체 위원회와 외부 감독을 동시에 받습니다. 8주 cadence를 그대로 적용하기 어려운 경우가 많고, 12주 또는 분기 단위로 늘어나기도 합니다. 그러나 cadence의 길이가 늘어나는 것 자체는 문제가 아닙니다. 진짜 문제는 cadence 안에서 거버넌스 검토 항목이 매번 처음부터 다시 정의되는 상황입니다. 다음 표는 한국 환경에서 자주 마주치는 상황과, 거버넌스가 속도를 죽이지 않도록 만드는 대응을 정리한 것입니다.

```
+------------------------+--------------------------------+
| 한국 환경의 제약       | 속도를 살리는 설계             |
+------------------------+--------------------------------+
| 자체 보안 위원회       | 표준 변경 유형 사전 합의,      |
|                        | 유형별 자동 통과 경로          |
+------------------------+--------------------------------+
| 외부 감독·금감원 점검  | audit log 표준 포맷 사전 정의, |
|                        | 회기 시 즉시 추출 가능         |
+------------------------+--------------------------------+
| 법무·컴플라이언스 검토 | PII redaction 모듈 공통화,     |
|                        | 도메인은 매핑만 정의           |
+------------------------+--------------------------------+
| 사고 시 책임 추적      | 모델·도구·라우팅의 전 변경     |
|                        | 이력을 trace에 부착            |
+------------------------+--------------------------------+
```

표의 핵심은 거버넌스 항목을 사전에 표준화해 두면 사이클 안에서 검토가 협상이 아니라 점검이 된다는 점입니다. 점검은 빠르고 협상은 느립니다. 표준 변경 유형, 표준 audit 포맷, 공통 PII 모듈, 변경 이력 자동 부착 같은 항목이 사이클을 무겁게 만들지 않으면서 보호를 유지하는 장치들입니다.

마지막으로 주의할 점은 거버넌스를 강하게 깔수록 거버넌스 자체에 대한 관측이 필요해진다는 사실입니다. 검토가 평균 며칠 걸리는지, 어느 단계에서 가장 자주 막히는지, 어떤 변경 유형이 가장 많이 거절되는지가 보이지 않으면 거버넌스가 자기도 모르게 비대해집니다. 6.2.2에서 정한 KPI에 거버넌스 사이클 시간, 변경 처리량, 검토 거절률 같은 항목을 추가하는 것이 좋습니다. 이렇게 두면 거버넌스가 속도를 죽이는 순간이 숫자로 드러나고, 사이클을 조정할 근거가 생깁니다.

정리하면, 거버넌스와 속도의 균형은 8주 update cadence를 중심으로 고정 release train, 표준 인터페이스, 재사용 가능 컴포넌트라는 세 가지 도구를 함께 깔아야 잡힙니다. 한국 대기업·금융권 환경에서는 cadence가 길어지더라도 검토 항목을 사전에 표준화해 협상 대신 점검으로 바꾸는 것이 핵심이며, 거버넌스 자체에 대한 관측을 KPI에 포함해 비대해짐을 막는 것까지가 설계의 마무리입니다.

다음 단원인 6.3.1에서는 지금까지 다룬 자가 진단, 도입 로드맵, KPI, 거버넌스를 하나의 framework로 묶어 면접과 문서에서 본인의 시스템을 일관되게 설명하는 방법을 정리합니다. Palantir의 5차원을 표준 답변 구조로 활용하는 패턴을 살펴봅니다.

이 단원을 마치면 8주 update cadence와 고정 release train, 표준 인터페이스, 재사용 가능 컴포넌트가 어떻게 거버넌스 부담을 일정하게 유지하는지를 설명할 수 있고, 한국 대기업·금융권 환경에서 cadence를 늘리되 검토를 점검으로 바꾸는 표준화 패턴을 적용할 수 있으며, 거버넌스 사이클 자체를 KPI로 관측해 비대해짐을 막을 수 있습니다.
