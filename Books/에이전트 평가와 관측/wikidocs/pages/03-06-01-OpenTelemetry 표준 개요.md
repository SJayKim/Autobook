# 3.6.1 OpenTelemetry 표준 개요

앞의 3.5에서 LangSmith와 Langfuse를 비교하고, 더 나아가 두 도구를 섞어 쓰는 hybrid 스택까지 살펴보았습니다. 그 흐름의 끝에 항상 따라온 단어가 있습니다. 바로 OpenTelemetry입니다. Langfuse가 OTLP endpoint를 받는다고 한 자리에서, hybrid 스택의 표준화 전략을 다룬 자리에서, 모두 같은 이름이 등장했습니다. 이 단원은 그 이름이 무엇이고 왜 LLM 관측의 가운데 자리에 놓이게 되었는지를 처음부터 풀어 봅니다.

먼저 단어 자체부터 풀겠습니다. **OpenTelemetry**, 줄여서 OTel은 분산 시스템의 동작을 관찰하기 위한 데이터, 즉 trace와 metric과 log를 어떻게 만들고 어떻게 보내고 어떻게 수집할지를 정해 둔 업계 표준입니다. 클라우드 네이티브 운영 표준을 다루는 비영리 재단인 CNCF에서 호스팅하고 있고, 한 회사가 만든 도구가 아니라 여러 회사와 사용자가 함께 쓰는 공통 약속이라는 점이 본질입니다. '여러 도구가 같은 형식의 trace를 주고받자'는 합의를 표준 단위로 적어 둔 결과물이 OTel입니다.

그 합의는 세 개의 부품으로 나뉩니다. 첫째는 **SDK**입니다. 애플리케이션 코드가 trace와 metric을 만들 때 부르는 라이브러리로, 언어별로 Python·JavaScript·Go·Java·Rust 같은 구현이 있습니다. SDK는 'span을 시작하고 끝낸다', '속성을 단다', '메트릭 카운터를 올린다' 같은 일을 통일된 인터페이스로 제공합니다. 둘째는 **collector**입니다. SDK가 만든 데이터를 받아 가공하고 분기해 여러 백엔드로 다시 내보내는 중간 부품으로, 별도 프로세스로 띄워 운영합니다. 셋째는 **protocol**, 그중에서도 OTLP라 불리는 OpenTelemetry Protocol입니다. SDK와 collector, collector와 백엔드 사이에 데이터를 실어 나를 때 쓰는 네트워크 규약입니다. 이메일에 비유하면 SMTP에 해당하는 자리를 차지합니다.

세 부품이 함께 동작하는 모습을 그림으로 보면 다음과 같습니다.

```
 [애플리케이션 + OTel SDK]
          |
          |  OTLP / gRPC 또는 HTTP
          v
   +---------------+        +--> Jaeger / Tempo (trace)
   | OTel Collector|------>+--> Prometheus     (metric)
   |  (수신·처리)  |        +--> Loki / S3      (log)
   +---------------+
```

이 그림에서 중요한 점은 SDK가 어떤 백엔드를 직접 알 필요가 없다는 사실입니다. SDK는 OTLP로 collector에 데이터를 던지고, 어디로 분기할지는 collector의 설정 한 곳에서 결정합니다. 백엔드를 갈아 끼우려고 애플리케이션 코드를 다시 만지지 않아도 된다는 뜻입니다.

OTel이 다루는 데이터의 종류는 셋입니다. **Trace**는 한 요청이 시스템을 어떻게 흘러갔는지를 트리 형태로 기록한 묶음으로, 그 안의 한 단계를 span이라 부릅니다. **Metric**은 시간에 따라 변하는 수치로, 요청 수, 지연, 에러율 같은 값이 여기에 들어갑니다. **Log**는 어느 한 시점에 남긴 텍스트 기록입니다. 이 셋은 본래 따로 발달해 왔지만, OTel은 한 SDK에서 한 표준으로 셋을 함께 다룰 수 있게 모았습니다. trace의 한 span이 만들어 내는 metric, 그 span 안에서 남긴 log가 같은 trace ID로 묶여 한 요청의 그림을 한 번에 보여 주는 일이 가능해진 까닭입니다.

OTel이 가진 가치는 한 단어로 요약하면 **벤더 중립**입니다. 관측 분야에서 가장 자주 일어나는 사고는 한 도구의 전용 SDK로 코드를 광범위하게 계측해 두었다가 나중에 도구를 바꾸지 못해 그대로 묶이는 일입니다. OTel은 이 벤더 락인을 구조적으로 풀어내기 위해 SDK와 백엔드 사이를 표준 한 겹으로 분리합니다. 어떤 백엔드를 쓰든 같은 OTel SDK로 같은 형식의 데이터를 만들 수 있으면, 백엔드를 바꿀 때 흔들리는 영역이 collector 설정 정도로 줄어듭니다. 이 효과는 도구를 자주 바꾼다는 뜻이 아니라, 바꾸어야 할 때 그 비용이 코드 전반의 재작성 수준이 아니라는 뜻입니다.

OTel의 출발점은 LLM이 아니었습니다. 마이크로서비스가 본격화되면서 한 사용자 요청이 수십 개 서비스를 거치는 경로를 추적해야 할 필요가 커졌고, 이 분야의 두 표준이었던 OpenTracing과 OpenCensus가 통합되어 OTel이 만들어졌습니다. 이 흐름은 **전통 분산 추적과의 연속성**으로 읽을 수 있습니다. 즉 OTel은 새로 만든 발명품이 아니라, 분산 시스템 추적이 십 년 넘게 다듬어 온 데이터 모델을 한 우산 아래 모은 결과입니다. trace와 span의 부모-자식 관계, 컨텍스트 전파, sampling 같은 개념은 본래 마이크로서비스 운영을 위해 정리된 것들이고, 같은 도구가 그대로 LLM 호출에도 쓰일 수 있다는 사실이 이번 흐름의 출발점이 되었습니다.

LLM 호출 한 건은 본질적으로 또 하나의 외부 서비스 호출입니다. 모델 이름, 입력 토큰, 출력 토큰, 지연, 비용 같은 값을 span의 속성으로 부착하면 기존 trace 그림 안에 자연스럽게 자리잡습니다. 에이전트의 multi-step 흐름은 부모 span 아래 여러 자식 span이 매달리는 형태로 표현되고, 그 안에 도구 호출 span이 다시 자식으로 들어갑니다. 분산 추적이 다년간 다루어 온 표현이 그대로 GenAI 영역으로 옮겨 올 수 있다는 점이 OTel을 LLM 관측의 후보로 끌어올린 가장 큰 이유입니다.

다만 LLM에는 기존 마이크로서비스에는 없던 속성이 많습니다. system prompt, completion, tool call의 인자와 결과, token usage, 모델별 메타데이터 같은 항목을 어떤 이름으로 어디에 넣을지 합의가 필요합니다. 이 합의가 GenAI semantic conventions라는 이름으로 진행 중인데, 2025년 시점에서도 일부 영역은 작업이 계속되고 있는 상태입니다. 즉 OTel은 LLM 관측의 표준 자리를 차지할 후보로 자리잡았지만, 그 표준의 세부 항목은 지금도 채워지고 있는 단계입니다. 이런 진행 상태를 받아들이면서 표준 위에 올리는 일이 이 책 후반부의 실무 의사결정을 가로지르는 주제가 됩니다.

OTel을 LLM 관측의 가운데에 두는 일이 한국 기업 입장에서 갖는 의미도 짚어 두겠습니다. LangSmith나 Langfuse나 자체 trace 백엔드 가운데 무엇을 고를지는 시간에 따라 바뀔 수 있는 결정입니다. 그러나 계측 코드를 OTel SDK 위에 두면 그 결정이 바뀌어도 애플리케이션 코드는 거의 영향을 받지 않습니다. 도구 비교를 다룬 3.5에서 hybrid 스택과 자체 harness가 등장한 까닭도 같은 자리에 놓입니다. 어떤 조합을 고르든 가운데에 OTel을 두면 조합을 바꾸기가 한결 쉬워지기 때문입니다.

정리하면, OpenTelemetry는 trace·metric·log를 한 표준 위에 모아 SDK·collector·protocol 세 부품으로 다루는 벤더 중립 관측 표준이며, 분산 추적의 오랜 전통을 이어받아 LLM 호출과 에이전트 trajectory까지 같은 그림 안에 표현할 수 있게 확장되고 있는 흐름입니다. 표준 자체는 2025년에도 GenAI 영역의 세부 약속을 채우는 중이지만, 가운데 자리에 OTel을 놓으면 LangSmith·Langfuse·자체 백엔드 사이의 의사결정이 코드 재작성이 아니라 설정 변경 수준의 문제로 좁아진다는 효과가 따라 나옵니다.

다음 단원인 3.6.2에서는 그 표준이 LLM 호출을 표현하기 위해 정의해 둔 속성과 메트릭 이름들을 구체적으로 살펴보고, vendor마다 그것을 어떻게 매핑하고 있는지를 정리하겠습니다.

이 단원을 마치면 OpenTelemetry의 SDK·collector·protocol 세 부품이 어떤 일을 나눠 맡는지, trace·metric·log를 한 표준에서 함께 다루는 의미, 벤더 중립이라는 가치가 만들어 내는 효과, 그리고 분산 추적의 연속선상에서 OTel이 GenAI 관측의 가운데 자리로 옮겨 오게 된 배경을 설명할 수 있습니다.
