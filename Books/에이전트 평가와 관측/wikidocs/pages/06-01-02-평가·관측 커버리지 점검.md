# 6.1.2 평가·관측 커버리지 점검

앞 단원에서 다섯 차원의 거시 점검을 한 장의 표로 끝냈습니다. 이번 단원은 그 표의 두 자리, 즉 평가와 관측에 한 걸음 더 다가가서 매트릭스 형태로 다시 펼치는 자리입니다. 5차원 self-audit이 거시 지도였다면, 평가·관측 커버리지 매트릭스는 그 지도의 두 영역만 확대해 격자 위에 놓는 일에 가깝습니다. 이번 단원의 목표는 평가·관측 두 매트릭스를 만들고, 그 위에서 4종 failure mode와 데이터 종류의 공백을 잡아 단계적 보강 로드맵으로 옮기는 것입니다.

먼저 한 가지 도구를 끄집어 두겠습니다. **커버리지 매트릭스**라는 말은 격자 한 장에 두 축을 세워 놓고, 각 칸이 채워졌는지 비어 있는지를 한눈에 보는 표입니다. 한 축은 어떤 종류의 평가나 관측을 들고 있느냐이고, 다른 한 축은 그것으로 무엇을 잡으려 하는지입니다. 격자 한 칸이 비어 있으면 그 칸이 곧 본인 시스템의 사각지대입니다.

이제 첫 번째 매트릭스인 **평가 커버리지 매트릭스**를 펼치겠습니다. 가로 축에는 평가의 종류 세 가지를 둡니다. 1.1.4와 2.3에서 본 trajectory 평가, 2단원에서 묶인 offline 평가, 3.1에서 다룬 online 평가가 그 세 자리입니다. 세로 축에는 1.2에서 정리한 4종 failure mode, 즉 tool selection error, planning loop, hallucinated reasoning, premature termination을 둡니다. 각 칸에는 본인의 시스템에 그 평가가 켜져 있고 그 failure mode를 탐지한 경험이 있다면 Yes, 평가는 있지만 그 failure mode가 잡히지 않으면 Partial, 평가 자체가 없으면 No를 채웁니다.

```
+-------------------------+----------+---------+----------+
| Failure mode \ 평가 종류 | Offline  | Online  |Trajectory|
+-------------------------+----------+---------+----------+
| Tool selection error    | Partial  | No      | Yes      |
| Planning loop           | No       | Partial | Yes      |
| Hallucinated reasoning  | Partial  | No      | Partial  |
| Premature termination   | Yes      | Partial | Partial  |
+-------------------------+----------+---------+----------+
```

위 표는 예시이지 정답이 아닙니다. 본인 시스템의 답을 채워 넣은 12 칸의 표가 곧 평가의 사각지대 지도가 됩니다. tool selection error는 trajectory 평가가 단계별 도구 선택 정확도를 직접 잴 수 있어 가장 자연스럽게 잡히고, planning loop는 단계 수와 반복 패턴으로 보입니다. hallucinated reasoning은 도구 출력과 추론을 비교해야 잡히고 자동화가 까다로워 LLM-as-judge로 보강하지 않으면 Partial에 머무릅니다. premature termination은 완수율 메트릭으로 offline·online 양쪽에서 보이고 trajectory의 어느 단계에서 끊겼는지도 단계 단위로 볼 수 있습니다. 한 줄이 전부 No로 차 있다면 그 failure mode는 시스템 안에서 일어나도 본인이 알 수 없다는 뜻입니다.

이제 두 번째 매트릭스인 **관측 커버리지 매트릭스**로 넘어갑니다. 가로 축에는 3.2.2의 데이터 모델인 trace, metric, log를 둡니다. 세로 축에는 시스템에서 수집할 데이터 종류를 둡니다. 모델 호출, 도구 호출, 메모리 접근, HITL 승인, 사용자 만족도 신호 다섯 가지입니다.

```
+----------------------+--------+---------+--------+
| 데이터 \ 관측 형태   | Trace  | Metric  | Log    |
+----------------------+--------+---------+--------+
| 모델 호출            | Yes    | Yes     | Yes    |
| 도구 호출            | Yes    | Partial | Yes    |
| 메모리 접근          | No     | No      | Partial|
| HITL 승인            | Partial| No      | Yes    |
| 사용자 만족도 신호   | No     | Partial | No     |
+----------------------+--------+---------+--------+
```

trace 열은 한 번의 사용자 요청이 한 줄기로 묶이는 자리, metric 열은 그 호출들의 통계가 실시간으로 흐르는 자리, log 열은 사후 검색을 위한 텍스트 흔적이 남는 자리입니다. 메모리 접근이 trace에 남지 않으면 5.5.3의 메모리 감사 항목 자체가 빈 약속이 되고, 사용자 만족도 신호가 어느 형태에도 거의 남지 않으면 1.3에서 본 hidden quality는 데이터로 다룰 수가 없습니다.

두 매트릭스를 묶어 주는 다리로, 4종 failure mode 각각을 어떤 데이터 종류 위에서 잡을 수 있는지 한 표로 매핑해 두면 다음과 같습니다.

```
+-----------------------+------------------------------------+
| Failure mode          | 탐지에 필수적인 데이터             |
+-----------------------+------------------------------------+
| Tool selection error  | 도구 호출 trace + 골든 데이터셋     |
| Planning loop         | 도구 호출 trace 단계 수 + 비용 알람 |
| Hallucinated reasoning| 도구 결과 + 모델 응답 + judge log   |
| Premature termination | 모델 호출 trace + 완수율 메트릭     |
+-----------------------+------------------------------------+
```

한 failure mode를 잡고 싶다면 그 줄에 적힌 데이터 종류가 두 번째 매트릭스에서 어떤 형태로 쌓이는지를 다시 확인하면 됩니다. 두 매트릭스 사이에 공백이 생긴 자리가 곧 그 failure mode의 사각지대입니다.

이제 채워진 두 매트릭스를 보고 **공백 우선순위**를 매기는 차례입니다. 결정 흐름은 단순합니다. 첫째, 두 매트릭스 모두에서 한 줄이 거의 No로 비어 있는 자리를 가장 먼저 봅니다. 평가에서도 안 잡히고 관측에도 데이터가 없는 failure mode는 시스템 안에서 일어나도 알 수 없는 사고가 됩니다. 둘째, 비가역 행동에 닿는 줄을 봅니다. 도구 호출 trace가 비어 있다면 외부 시스템에 어떤 명령이 떨어졌는지조차 사후에 모른다는 뜻입니다. 셋째, 사용자 경험에 직접 닿는 줄을 봅니다. 사용자 만족도 신호가 빈 줄이면 1.3.2의 hidden quality와 escalation path는 측정 가능한 지표로 옮겨지지 못한 채 묻힙니다. 넷째, 비용 폭주의 줄을 봅니다. planning loop의 줄이 비어 있으면 청구서가 와서야 사고가 보입니다.

이 우선순위에 따라 **단계적 보강 로드맵**을 세 단계로 자릅니다. 첫 단계는 trace의 빈자리를 메우는 일입니다. 5.5.5의 빌트인 tracing 항목과 곧장 이어지고, LangSmith·Langfuse·OpenTelemetry 가운데 어느 쪽을 쓰든 모든 모델 호출·도구 호출·메모리 접근에 자동으로 trace가 부착되는 상태를 종착점으로 둡니다. 두 번째 단계는 trajectory 평가의 빈자리를 메우는 일입니다. 첫 단계의 trace를 골든 데이터셋과 결합해 단계별 정확도와 단계 수 분포를 자동으로 재는 평가 파이프라인이 종착점입니다. 세 번째 단계는 online 평가와 메트릭의 빈자리를 메우는 일입니다. 실 트래픽 위에 완수율·HITL 트리거 비율·사용자 만족도 신호가 메트릭으로 흐르고, 임계치를 넘으면 알람이 사람을 호출하는 상태가 종착점입니다.

```
[단계 1]              [단계 2]                  [단계 3]
trace 채우기      ->  trajectory 평가 채우기 ->  online 평가·메트릭 채우기
(관측 trace 열)        (평가 trajectory 열)        (남은 No·Partial)
```

세 단계를 동시에 진행하지 않는 까닭은, trace가 비어 있는 상태에서 평가만 먼저 붙이면 평가의 입력 자체가 부실해지고, 메트릭부터 붙이면 무엇을 보고 있는지가 흐려지기 때문입니다. 두 매트릭스의 trace 자리는 5번 차원 Observability와, 도구 호출 trace 자리는 4번 차원 Tool Governance의 감사 로그·회수 항목과, 메모리 접근 trace 자리는 3번 차원 Memory Governance의 메모리 감사 항목과 그대로 이어집니다. 거시 점검의 5차원 표와 미시 점검의 두 매트릭스가 같은 시스템의 두 단면이라는 사실이 이 매핑에서 보입니다.

정리하면, 평가·관측 커버리지 점검은 4종 failure mode×offline·online·trajectory 평가의 4×3 매트릭스와 다섯 데이터 종류×trace·metric·log의 5×3 매트릭스 두 장을 본인의 시스템 위에 채워 공백을 표시하고, 두 매트릭스 사이의 다리 역할을 하는 failure mode별 데이터 매핑 표로 어떤 자리가 어떤 failure를 못 잡고 있는지 짚은 다음, trace·trajectory 평가·online 평가의 세 단계로 보강 로드맵을 자르는 흐름입니다. 이 두 매트릭스가 6.1.1의 5차원 표와 한 묶음이 되어 본인 시스템의 거시·미시 두 점검 도구가 갖춰진 셈입니다.

다음 단원인 6.2.1에서는 이 세 도구에서 드러난 공백을 실제로 메우는 도입 순서를 다루겠습니다. 관측 우선이냐 평가 우선이냐의 트레이드오프, 조직 규모와 단계 결정, 초기 1~2주 내 측정 가능한 가치, 도입 비용과 ROI 추정이 그 자리의 핵심입니다.

이 단원을 마치면 본인 시스템의 평가 커버리지를 4종 failure mode와 offline·online·trajectory의 매트릭스로, 관측 커버리지를 다섯 데이터 종류와 trace·metric·log의 매트릭스로 펼쳐 공백을 식별하고, 두 매트릭스를 묶어 우선순위를 매긴 다음 단계적 보강 로드맵을 설계할 수 있습니다.
