# 5.1.3 프라이버시 보존 모니터링

앞 단원에서 production tracing이 멀티 에이전트의 baseline 인프라여야 한다는 점을 살피며, 모든 토큰을 그대로 남기는 일이 보관 비용과 프라이버시 위험을 동시에 만든다는 단서를 짚었습니다. 이 단원은 그 단서에서 출발해, 사용자 대화 내용 자체에는 손을 대지 않으면서도 시스템의 의사결정 패턴을 충분히 들여다보는 길을 정리합니다. 핵심 단어는 **프라이버시 보존 모니터링**입니다.

먼저 왜 둘이 충돌하는지부터 풀겠습니다. 관측을 잘하려면 사용자가 무엇을 물었고 시스템이 무엇이라 답했는지를 자세히 보고 싶어집니다. 한편 사용자 발화에는 이름·연락처·계좌·건강 상태 같은 개인정보가 자연스럽게 섞이고, 시스템 응답에는 그 정보를 다시 받아 쓰는 문장이 따라 붙습니다. 운영자가 사용자의 한 줄을 들여다보는 모든 순간이 잠재적 정보 노출 자리이기 때문에, '잘 보기'와 '덜 보기'가 정면으로 부딪힙니다. Anthropic 팀이 Research 시스템에서 정리한 한 문장이 이 충돌을 푸는 길을 보여 줍니다. **개별 대화 내용은 보지 않으면서 에이전트 의사결정 패턴과 상호작용 구조를 모니터링해 root cause·예상 외 동작·공통 실패를 잡아낸다**는 원칙입니다. 보지 않는 것은 대화 자체이고, 보는 것은 그 위에 떠 있는 패턴입니다.

이 원칙을 구현으로 옮기는 첫 자리는 **무엇을 trace에 남길 것인가**라는 결정입니다. 5.1.2에서 trace, span, metric의 모양을 다시 가져오면, 한 사용자 질의는 trace 한 줄, Lead의 분해와 Subagent 호출은 자식 span들, 도구 호출은 손자 span들, 그리고 각 단계에 token·latency·cost가 metric으로 붙는 구조입니다. 프라이버시 보존 모니터링은 이 구조를 그대로 두되, span에 채워 넣는 데이터의 모양을 두 층으로 나눕니다. 한 층은 항상 남기는 **메타데이터 층**이고, 다른 한 층은 기본적으로 남기지 않거나 마스킹·해시 처리해서 남기는 **본문 층**입니다.

메타데이터 층에는 어떤 도구가 호출되었는지, 호출이 성공했는지 실패했는지, 응답 길이는 얼마였는지, 어떤 단계로 넘어갔는지 같은 정보가 들어갑니다. 본문 층에는 사용자 발화 원문, 모델 응답 원문, 도구가 받아 온 본문이 들어갑니다. 운영자가 평소에 보는 화면은 메타데이터 층 위에서 만들어지고, 본문 층은 별도의 권한과 절차를 거쳐야만 접근 가능한 자리에 보관됩니다. 그림으로 정리하면 다음과 같습니다.

```
 [원본 대화]
       |
       v
 +-------------------------+
 | 마스킹 / 토크나이저 / 해시 |
 +-------------------------+
       |
       v
 +-----------------+   +---------------------+
 | 메타데이터 층    |   | 본문 층 (제한 접근)  |
 |  - 도구 호출 패턴|   |  - 원문 (암호화 보관)|
 |  - 단계 분기     |   |  - 권한·감사 필요   |
 |  - 길이/지연/비용|   |                     |
 +-----------------+   +---------------------+
       |
       v
   [평소 운영 대시보드]
```

이 두 층 분리가 만들어 내는 결정적 차이는 **집계와 익명화** 위에서 의사결정 패턴이 충분히 보인다는 점입니다. 어떤 도메인에서 Subagent 평균 호출이 갑자기 늘었는지, 어떤 시점부터 본문 추출 실패가 잦아졌는지, 어떤 결합에서 답이 비어 돌아오는 비율이 올라갔는지 같은 신호는 모두 메타데이터 층의 분포에서 나옵니다. 즉, 사용자가 무엇을 입력했는지를 한 글자도 보지 않아도 시스템이 어떻게 행동하고 있는지를 추적할 수 있습니다.

이 자리에서 잡히는 신호의 종류를 세 가지로 정리해 두면 운영 감각이 잡힙니다. 첫째는 **공통 실패**입니다. 단일 사용자의 한 케이스로 보면 작은 노이즈처럼 보이는 실패가, 메타데이터 분포 위에서 보면 같은 단계, 같은 도구, 같은 분기에 반복적으로 나타나는 패턴으로 보입니다. 이 패턴은 1.2의 failure mode taxonomy에 그대로 매핑됩니다. tool selection error가 특정 도메인에서 도드라지거나, planning loop가 특정 분해 패턴 뒤에 자주 따라오는 식의 신호가 분포 그래프에 떠오릅니다.

둘째는 **예상 외 동작**입니다. 5.1.1에서 본 emergent behavior는 한 번씩 갑자기 다른 모양으로 나타나는데, 이 변화는 평균 분포보다 백분위 분포에서 더 잘 보입니다. 95퍼센타일 응답 길이가 평소의 두 배로 뛴다거나, 99퍼센타일 step 수가 갑자기 한 자릿수에서 두 자릿수로 옮겨 갈 때, 이 자리는 예외 케이스로 분류되어 별도의 큐에 들어갑니다. 본문은 보지 않더라도 '이 분포의 꼬리에서 무엇이 자주 일어나는가'는 충분히 보입니다.

셋째는 **root cause 추적**입니다. 공통 실패와 예상 외 동작이 잡혔다고 해도, 그 원인이 Lead의 분해인지 Subagent의 도구 선택인지 도구 자체의 응답인지를 구분하려면 trace를 따라 들어가야 합니다. 이 단계에서 본문이 필요한 케이스가 생깁니다. 그러나 본문 층은 '특정 trace ID에 한해, 운영 권한이 있는 사람이, 감사 로그가 남는 조건에서' 열도록 설계됩니다. 평소에는 본문에 닿지 않고도 패턴을 잡고, 꼭 필요한 자리에서만 좁은 범위로 들여다보는 두 단계 구성입니다.

이 두 단계 구성을 받쳐 주는 기술적 장치들은 3.6에서 본 OpenTelemetry collector pipeline 위에 자연스럽게 올라갑니다. 본문이 trace로 흘러가기 전에 PII redaction을 거치고, 거친 결과가 메타데이터 층의 백엔드로 가며, 원문은 별도의 보관 백엔드로 분리되어 권한 차등이 걸리는 식입니다. 3.6.3에서 정리한 무벤더 락인 설계와 같은 골격을 그대로 쓰되, 차이는 두 백엔드의 접근 권한이 다르다는 점뿐입니다.

여기까지가 일반적인 멀티 에이전트의 모양이고, 이제 이 모양이 한국 환경에서 어떻게 살아 움직이는지를 한 자리에 옮겨 두겠습니다. 멀티 에이전트가 한국 **상권 분석**과 **소상공인** 데이터에 적용되는 자리를 떠올려 보면, 사용자는 점포 운영자이고 입력에는 사업자 등록번호, 점포 위치, 매출 분포, 임차 조건, 메뉴 구성, 거래 데이터 같은 민감한 정보가 흘러 들어옵니다. 이런 환경에서 운영자가 사용자 한 줄을 들여다보는 일은 그 자체로 영업비밀에 닿는 행위가 됩니다. 즉, 본문 층을 평소에 닫아 두는 일이 단순한 보안 정책이 아니라 사용자 신뢰의 전제 조건이 됩니다.

같은 시스템에서 메타데이터 층은 다른 결로 살아 움직입니다. '입지 분석'이라는 도메인에서 평균 Subagent 호출이 어떻게 변하는지, '상권 비교' 도메인에서 본문 추출 실패율이 어느 시점부터 올라갔는지, '메뉴 가격대 추천' 분기에서 답이 비어 돌아오는 비율이 얼마인지 같은 신호는 본문을 한 글자도 보지 않고 잡힙니다. 운영자는 이 신호를 보고 '입지 분석 도메인의 검색 도구가 최근 빈 결과를 자주 돌려준다, 다른 데이터 소스를 fallback으로 붙이자'는 결정을 내립니다. Anthropic의 원칙 한 줄이 'AI를 다루는 회사라면 어디든 적용 가능한 보편적 패턴'이라는 점이 이 사례에서도 그대로 보입니다.

여기서 한 가지 단서를 더 두고 가겠습니다. 프라이버시 보존 모니터링은 **양립 설계**이지, 한쪽을 포기하는 절충이 아닙니다. 본문을 안 본다고 관측 품질이 떨어지지 않습니다. 메타데이터 층을 충분히 풍부하게 짜고, 분포 기반 알람을 적절히 걸고, 권한 분리된 본문 접근 절차를 깔끔하게 운영하면, 단일 운영자가 모든 본문을 들여다볼 때보다 훨씬 안정적인 운영 감각이 나옵니다. 본문을 본다는 것은 강력해 보이지만, 운영자 한 명이 모든 케이스를 본문 단위로 쫓아갈 수는 없기 때문입니다. 분포로 보고, 꼬리만 좁혀 들어가는 구성이 사실상 더 큰 표면적을 덮습니다.

정리하면, 프라이버시 보존 모니터링은 trace 데이터를 메타데이터 층과 본문 층으로 분리해 평소 운영은 메타데이터 층의 집계·익명화된 분포 위에서 진행하고, 본문은 권한·감사 절차가 붙은 좁은 자리에서만 접근하도록 짜는 양립 설계입니다. 이 구성은 공통 실패, 예상 외 동작, root cause를 충분히 잡아 주면서 사용자의 개별 대화에는 손을 대지 않게 해 주며, 한국의 상권 분석·소상공인 데이터처럼 영업비밀이 입력에 함께 흘러 들어오는 환경에서 사용자 신뢰의 전제 조건이 됩니다.

다음 단원인 5.1.4에서는 자동 평가만으로는 잡히지 않는 신호를 사람 테스터가 어떻게 발견했는지, 그리고 그 발견이 어떻게 한 줄의 프롬프트 보정으로 옮겨 갔는지를 source quality heuristic 사례로 살펴보겠습니다.

이 단원을 마치면 관측과 프라이버시가 충돌하는 자리를 정리하고, trace 데이터를 메타데이터 층과 본문 층으로 분리해 양립시키는 설계를 설명하며, 공통 실패·예상 외 동작·root cause를 본문 없이 분포 위에서 추적하는 방식을 한국 상권 분석·소상공인 데이터 사례에 옮겨 풀어 낼 수 있습니다.
