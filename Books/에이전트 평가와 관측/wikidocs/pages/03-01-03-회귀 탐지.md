# 3.1.3 회귀 탐지

앞 두 단원에서 실 트래픽 위의 메트릭과 prescriptive·reactive 두 시각의 평가 형판을 갖추었습니다. 메트릭과 형판이 자리를 잡고 나면 다음으로 떠오르는 질문은 단순합니다. 이 형판 위의 숫자가 어느 순간부터 나빠지기 시작하면 누가, 어떻게 알아차리느냐의 문제입니다. 사람이 대시보드를 매 시간 들여다보는 운영은 오래 가지 못합니다. 이 단원은 측정값이 나빠지는 순간을 자동으로 잡아 알림을 띄우고, 더 나쁘게 번지기 전에 차단하고, 필요하면 빠르게 되돌리는 흐름인 **회귀 탐지**를 다룹니다.

여기서 **회귀**라는 말부터 짚고 가겠습니다. 본래 소프트웨어에서 회귀는 한 번 잘 되던 기능이 어떤 변경 이후 다시 나빠지는 현상을 가리킵니다. 에이전트 시스템에서는 코드 변경뿐 아니라 프롬프트 수정, 모델 업데이트, 도구 인터페이스 변화, 외부 의존 데이터의 미묘한 변동처럼 더 다양한 입력이 회귀를 만듭니다. 무엇이 원인이든 결과는 같습니다. 어제까지 0.92이던 완수율이 오늘 0.85로 내려앉고, 95퍼센타일 지연이 평소의 두 배로 튀고, 비용 분포의 꼬리가 부풀어 오릅니다.

회귀 탐지의 출발점은 **기준 메트릭과 임계치**를 정해 두는 일입니다. 기준 메트릭은 앞 단원에서 다룬 완수율·성공률·비용·지연 같은 값과, reactive 형판으로 추가된 신호들 가운데 시스템이 건강한지 가장 잘 대변하는 한 묶음을 가리킵니다. 임계치는 두 가지 방식으로 잡습니다. 절댓값 임계치는 '완수율이 0.85 미만이면 알람'처럼 고정된 선을 긋는 방식이고, 상대 임계치는 '직전 7일 평균 대비 3퍼센트포인트 하락이면 알람'처럼 평소 분포를 기준으로 잡습니다. 절댓값은 명확하지만 평소 수준이 변하면 다시 손봐야 하고, 상대값은 자동으로 따라오지만 분포 자체가 천천히 나빠질 때를 놓칠 수 있습니다. 두 방식을 함께 쓰면 한쪽의 빈틈을 다른 쪽이 메웁니다.

```
완수율
 0.95 ┤        ████████████████
 0.90 ┤████████                ████  ←── 상대 임계치 (-3pp) 발동
 0.85 ┤────────────────────────────  ←── 절댓값 임계치
 0.80 ┤                          ███
       ─────────────────── 시간 ──→
```

다음으로 임계치를 어디에 적용할지가 문제입니다. 메트릭의 원시값은 시간대마다 출렁이기 때문에 한 분 동안 잠깐 떨어진 값에 매번 알람을 울리면 운영자가 곧 알람을 무시하게 됩니다. 이 출렁임을 부드럽게 다듬는 도구가 **이동 평균과 변화율 알람**입니다. 이동 평균은 일정 시간 윈도우 안의 측정값을 평균 낸 값으로, 5분 이동 평균과 1시간 이동 평균을 동시에 두면 짧은 출렁임과 천천히 진행되는 추세를 같이 봅니다. 변화율 알람은 이동 평균의 기울기를 보는 방식으로, 평균값 자체는 합격선 안에 있어도 빠르게 떨어지고 있다면 미리 신호를 띄웁니다. 메트릭의 절대 수준이 무너지기 전에 추세가 먼저 신호를 주는 셈입니다.

임계치와 알람만으로는 한 가지 빈틈이 남습니다. 시스템 전체 평균은 멀쩡한데 특정 사용자 군이나 특정 입력 유형에서만 회귀가 일어나는 경우입니다. 모바일 사용자만 지연이 두 배로 튀는 일, 한국어 입력에서만 성공률이 떨어지는 일, 새 도구를 쓰는 흐름에서만 비용이 폭주하는 일은 전체 평균에 묻혀 보이지 않습니다. 이 공백을 메우는 것이 **코호트 단위 비교**입니다. 사용자 코호트, 입력 유형, 클라이언트 종류, 모델 버전 같은 축으로 트래픽을 나누고 각 조각마다 같은 임계치 검사를 합니다. 코호트 비교는 회귀의 위치를 알려 주기도 합니다. 모든 코호트가 동시에 떨어지면 공통 변경이 의심되고, 한 코호트만 떨어지면 그 코호트에 특화된 경로를 들여다보면 됩니다.

```
전체:        성공률 0.78  (이상 없음)
 ├─ 코호트 A: 성공률 0.82  (이상 없음)
 ├─ 코호트 B: 성공률 0.79  (이상 없음)
 └─ 코호트 C: 성공률 0.61  ←── 이쪽만 회귀
```

여기까지가 이미 배포된 시스템에서 회귀를 발견하는 흐름이라면, 다음 한 축은 회귀가 사용자에게 도달하기 전에 가로막는 흐름입니다. 그 핵심 도구가 **canary 배포**입니다. Canary는 광산에서 유독 가스를 먼저 감지하던 새의 이름에서 온 말로, 새 변경사항을 전체 트래픽에 한 번에 풀지 않고 일부 트래픽에만 먼저 노출해 위험을 미리 잡는 운영 방식을 가리킵니다. 새 버전을 5퍼센트 트래픽에만 라우팅하고, 그 5퍼센트의 메트릭이 기존 95퍼센트의 메트릭과 통계적으로 다르지 않을 때만 비율을 점진적으로 끌어올립니다. 만일 canary 트래픽에서 회귀 신호가 잡히면 비율을 더 키우지 않고, 더 나아가 자동으로 0퍼센트로 되돌려 회귀를 차단합니다. 회귀 차단은 사람의 판단을 기다리지 않고 시스템이 직접 결정하는 편이 안전합니다. 사람이 깨어 회의실로 모이는 동안에도 트래픽은 계속 흐르기 때문입니다.

```
[ 100% 기존 버전 ]
        ↓
[ 95% 기존 / 5% 신규 ]   ←── canary 단계
        ↓ (메트릭 비교 통과)
[ 75% / 25% ]
        ↓ (통과)
[ 50% / 50% ]
        ↓ (통과)
[ 0% / 100% ]
```

Canary가 실패해 차단이 걸리거나, 이미 전 트래픽에 풀린 다음 회귀가 잡힌 경우에는 **롤백 절차**가 작동합니다. 롤백은 단순히 이전 코드 커밋을 되돌리는 행위가 아닙니다. 에이전트 시스템에서는 코드, 프롬프트, 모델 버전, 도구 정의가 모두 묶여 한 묶음의 'release artifact'로 다뤄지고, 롤백은 이 묶음 전체를 직전 안정 버전으로 되돌리는 작업입니다. 롤백 절차는 평소부터 정의돼 있어야 합니다. 누가 트리거를 누르는가, 트리거를 누르면 몇 분 안에 트래픽이 이전 버전으로 돌아가는가, 롤백 도중에 진행 중이던 사용자 세션은 어떻게 다루는가, 사후에 어떤 회의 형식으로 원인을 정리하는가를 미리 적어 둬야 회귀가 실제로 발생했을 때 멈칫거리지 않습니다. 잘 운영되는 팀에서는 롤백 절차 자체를 정기적으로 훈련해, 사고 시점에 처음 해 보는 일이 없도록 합니다.

마지막으로 한 가지 시선을 적어 두겠습니다. 회귀 탐지의 목적은 알람의 수를 늘리는 일이 아니라, 진짜로 위험한 회귀가 일어났을 때 빠르고 분명한 신호를 주는 일입니다. 임계치를 너무 빡빡하게 잡으면 거짓 알람이 쏟아져 사람들이 알람을 무시하게 되고, 너무 느슨하게 잡으면 정작 위험한 순간에 신호가 늦습니다. 임계치 자체도 기준 메트릭처럼 분기마다 재조정이 필요한 살아 있는 값으로 다뤄야 하며, 알람이 한 번 울릴 때마다 '이 알람이 실제로 행동을 바꿔야 했는지'를 회고에 남기는 운영이 임계치를 천천히 옳은 자리에 데려다 놓습니다.

정리하면, 회귀 탐지는 기준 메트릭과 절댓값·상대 임계치를 함께 두고, 이동 평균과 변화율 알람으로 출렁임 속에서 추세를 잡으며, 코호트 단위 비교로 평균에 묻힌 회귀를 끌어내고, canary 배포로 사용자에게 도달하기 전 위험을 차단하며, 평소부터 정의된 롤백 절차로 회귀가 사용자에게 흘러간 경우에도 빠르게 되돌리는 흐름입니다. 이 다섯 축이 갖춰져 있을 때 online 평가는 그저 측정에서 멈추지 않고 운영 중 사고를 줄이는 안전망으로 자리 잡습니다.

다음 단원인 3.2.1에서는 이렇게 회귀 탐지를 굴리는 데 전제가 되는 관측 인프라로 시선을 옮겨, Datadog 같은 전통적 모니터링이 에이전트의 어떤 실패 패턴을 놓치는지, 왜 LLM observability가 별도의 영역으로 자리잡았는지를 다루겠습니다.

이 단원을 마치면 기준 메트릭과 임계치 설계, 이동 평균과 변화율 알람, 코호트 단위 비교, canary 배포와 회귀 차단, 그리고 롤백 절차의 다섯 요소를 정의하고 자체 시스템의 회귀 탐지 흐름을 설계할 수 있습니다.
