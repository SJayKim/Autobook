# 3.5.2 Hybrid 스택 구성 패턴

앞 단원에서 본 의사결정 트리는 한 도구를 고르는 길을 보여 줬지만, 현실의 큰 시스템은 한 도구만으로 모든 요구를 다 채우지 못하는 경우가 많습니다. 평가 워크플로에 강한 도구와 trace 분석에 강한 도구가 따로 있고, 그 위에서도 자기 사정에 맞춰 손으로 짜야 하는 부분이 늘 남습니다. 이 단원은 두 가지 이상의 도구를 의도적으로 함께 쓰는 hybrid 스택의 구성 패턴을 다룹니다. 출발점은 Replit이 공개한 자기네 평가 스택입니다.

먼저 사례부터 인용하겠습니다. Replit의 발표를 그대로 옮기면, "Braintrust는 일부 평가에, LangSmith는 traceability에 쓰지만 평가 harness 대부분은 자체 Python으로 구축"입니다. 한 문장이 짧지만 그 안에 hybrid의 골격이 다 들어 있습니다. 평가용 SaaS 한 개, 관측용 SaaS 한 개, 그리고 둘 사이에 자체 코드 한 덩이입니다. 단어를 짧게 풀어 두겠습니다. **Braintrust**는 데이터셋과 실험을 관리하고 평가 결과를 비교해 주는 데에 무게를 둔 평가 플랫폼이고, **LangSmith**는 3.3에서 본 대로 LangChain·LangGraph 코드의 trace를 자동으로 잡아 보여 주는 관측 플랫폼이며, **harness**는 1장에서 다룬 대로 모델 바깥에서 실행 환경과 평가 흐름을 묶어 주는 골격을 가리키는 일반 용어입니다.

이 사례에서 가장 먼저 짚을 점은 **역할 분담**입니다. 같은 trace 데이터를 두 도구가 동시에 받는다고 해도, 사용자가 그 데이터를 보러 가는 화면은 목적에 따라 갈립니다. 평가는 점수, 데이터셋, 실험 비교, 회귀 그래프 같은 화면이 필요합니다. 관측은 한 건의 trace를 처음부터 끝까지 펼쳐 어디서 무엇이 늦었고 어떤 도구가 실패했는지를 보는 화면이 필요합니다. Replit은 두 화면을 같은 도구 안에서 다 보려는 시도 대신, 평가 화면은 Braintrust로, trace 화면은 LangSmith로 각각 잘 만들어진 도구에 맡기는 길을 골랐습니다. 이 분담을 그림으로 보면 다음과 같습니다.

```
                    [에이전트 앱]
                         |
              span/trace 발행
                         |
        +----------------+----------------+
        |                                 |
        v                                 v
   [Braintrust]                      [LangSmith]
   - 데이터셋                        - LangChain/LangGraph
   - 실험 비교                         자동 인스트루먼트
   - 평가 점수                       - trace 펼쳐 보기
                                     - replay/fork
```

분담을 정하고 나면 곧이어 **자체 harness가 끼어드는 경계**가 드러납니다. Replit이 평가 harness 대부분을 자체 Python으로 짠 이유는 두 SaaS가 비워 두는 자리가 컸기 때문입니다. Replit의 에이전트는 Chrome 인스턴스와 Docker 컨테이너 위에서 사용자처럼 클릭하고 입력하는 web agent이기 때문에, 평가 한 번을 돌리려면 컨테이너를 띄우고 브라우저를 깨우고 시나리오를 주입하고 결과를 거두는 복잡한 서막이 필요합니다. 이런 환경 준비 코드는 Braintrust나 LangSmith가 직접 다루는 영역이 아니라, 평가 점수를 만들어 SaaS로 넘기기 위한 앞단입니다. 자체 harness는 이 앞단을 책임지고, SaaS는 그 결과를 받아 보여 주는 뒷단을 맡습니다. 이 경계가 분명할수록 어느 쪽에 무엇을 짤지 헷갈리지 않습니다.

분담과 경계를 세웠으면 그다음 문제는 **표준화**입니다. 도구가 셋이라는 사실이 곧 trace 형식도 셋이라는 뜻이 되면, 같은 사건을 세 군데에 다른 이름으로 기록해 두고 비교조차 어려워집니다. 이 위험을 줄이는 길이 3.4.2에서 본 OpenTelemetry입니다. OTel을 가운데 두면 에이전트 앱은 OTel SDK 한 가지로만 span을 만들어 OTLP로 내보내고, OTel collector가 그 흐름을 받아 Braintrust, LangSmith, 자체 백엔드로 분기해 줍니다. 같은 사건이 같은 속성 이름으로 세 곳에 도착하기 때문에, 한쪽에서 본 trace ID로 다른 쪽 화면을 곧장 열 수 있습니다. 그림으로 정리하면 다음과 같습니다.

```
   [에이전트 앱]
        | OTel SDK
        | OTLP
        v
   [OTel Collector] ----+---> Braintrust  (평가 화면)
                        +---> LangSmith   (trace 화면)
                        +---> 자체 백엔드 (장기 보관/사내 분석)
```

이 그림은 hybrid의 진짜 가치가 어디서 생기는지 보여 줍니다. 도구마다 잘하는 영역이 다른 현실을 인정하면서도, 데이터의 형식을 표준 위에 두는 한 도구를 갈아 끼울 때마다 코드 전반을 다시 손대지 않아도 됩니다. 도구를 바꾸는 일이 collector의 export 한 줄을 수정하는 일에 가까워집니다. 한국 기업 맥락에서 흔한 분기, 즉 LLM trace는 Langfuse self-host로, 인프라 메트릭은 Datadog로, 평가 점수는 Braintrust로 보내는 패턴도 같은 골격 위에서 자연스럽게 풀립니다.

다만 hybrid에는 정직하게 따져야 할 **운영 복잡도**가 따라옵니다. 도구를 둘로 늘리면 운영 부담은 두 배가 되지 않고 보통 두 배 이상으로 늘어납니다. 첫째, SaaS 두 개의 가격, 보안 심의, SLA를 모두 들여다봐야 합니다. 둘째, 사용자 권한과 데이터 보존 기간을 도구마다 별도로 관리해야 하기 때문에 권한 모델이 어긋날 위험이 늘 있습니다. 셋째, 사고가 났을 때 trace를 따라가다 도구 경계를 넘어야 하면 디버깅 시간이 늘어납니다. 넷째, OTel collector라는 새 부품 자체가 운영해야 할 또 하나의 시스템이 됩니다. 다섯째, 자체 harness가 짊어진 코드는 회사가 영원히 같이 들고 가야 하는 자산이자 부채입니다. hybrid는 이 비용을 의식적으로 떠안고서 얻는 유연성이지, 도구 두 개 이상을 어쩌다 같이 쓰게 된 결과가 아닙니다.

이 비용을 다스리는 실무 기준은 보통 다음 세 줄로 정리됩니다. 첫째, 단일 도구로 시작하고 그 도구가 명백히 비워 두는 자리만 다른 도구로 채웁니다. 둘째, 도구 사이를 잇는 글루 코드는 OTel 표준 위에 두고 가능한 얇게 유지합니다. 셋째, 자체 harness는 SaaS가 절대 채워 주지 않을 환경 준비와 사내 정책 적용에 한정하고, SaaS가 곧 채워 줄 영역까지 미리 짜 두지 않습니다. 이 세 기준이 흔들리는 순간 hybrid는 빠르게 자기 비용에 짓눌립니다.

마지막으로 hybrid가 항상 정답은 아니라는 점을 짚어 두겠습니다. 팀이 작고 도메인이 단순하다면 한 도구를 깊게 쓰는 길이 거의 늘 더 효율적입니다. Replit식 hybrid는 web agent라는 까다로운 평가 환경, 빠르게 늘어나는 reactive eval 백로그, 두 SaaS의 영역이 분명히 갈리는 제품 사정이 함께 작용해 자연스럽게 도달한 형태에 가깝습니다. 같은 패턴을 모방하기 전에 자기 회사의 사정도 그만큼 갈리는지 먼저 확인해 두는 편이 안전합니다.

정리하면, hybrid 스택은 평가에 강한 도구와 관측에 강한 도구를 의도적으로 분담시키고 그 사이를 자체 harness로 메우는 구성으로, Replit의 Braintrust + LangSmith + 자체 Python harness 조합이 대표 사례입니다. 같은 trace를 여러 백엔드에 보내는 길은 OpenTelemetry로 표준화해 도구 교체 시 영향 범위를 좁히고, 그 대가로 늘어나는 운영 복잡도는 시작은 단일 도구로, 글루 코드는 얇게, 자체 harness는 SaaS가 비우는 자리에만 둔다는 기준으로 다스립니다.

다음 단원인 3.6.1에서는 이 hybrid 패턴을 떠받치는 표준인 OpenTelemetry 자체를 본격적으로 살피고, 그 표준이 GenAI 영역까지 확장되어 온 흐름과 SDK·collector·protocol의 구조를 정리하겠습니다.

이 단원을 마치면 Replit식 hybrid 스택에서 Braintrust와 LangSmith가 평가와 traceability로 역할을 나누는 방식, 자체 harness가 SaaS의 빈 자리에 끼어드는 경계, OpenTelemetry로 여러 백엔드를 표준화해 묶는 전략, 그리고 hybrid가 가져오는 운영 복잡도를 다스리는 실무 기준을 함께 설명할 수 있습니다.
