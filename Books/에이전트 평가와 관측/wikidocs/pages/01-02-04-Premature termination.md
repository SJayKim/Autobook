# 1.2.4 Premature termination

앞 단원까지 도구 선택 오류, planning loop, hallucinated reasoning 세 가지 실패 모드를 살펴보았습니다. 앞 두 가지가 흐름이 너무 길어지는 쪽의 실패라면 hallucinated reasoning은 길이는 적당하지만 결론이 어긋나는 실패였습니다. 이번 단원에서 다루는 premature termination은 또 다른 결의 실패로, 흐름이 너무 일찍 끊겨 사용자가 부탁한 일이 마무리되지 않은 상태로 응답이 돌아가는 패턴입니다.

용어부터 풀겠습니다. **premature termination**은 사용자가 요청한 작업이 아직 끝나지 않았는데 에이전트가 흐름을 종료해 버리는 실패 모드를 가리킵니다. 글자 그대로 옮기면 '조기 종료'에 가깝습니다. 에이전트가 종료한다는 것은 더 이상 도구를 부르지 않고 사용자에게 응답을 돌려주는 결정을 내렸다는 뜻이므로, premature termination은 응답이 돌아갔다는 사실 자체로는 정상처럼 보인다는 점에서 까다롭습니다.

조기 종료를 만드는 원인은 크게 세 갈래로 정리할 수 있습니다.

첫째, **토큰 한계**입니다. 모델은 한 번에 처리할 수 있는 컨텍스트 길이가 정해져 있습니다. 에이전트가 단계를 거듭하며 컨텍스트에 추가 정보를 쌓으면, 어느 시점에 토큰 한계에 도달합니다. 한계에 도달하면 모델은 더 이상 깊은 추론이나 추가 도구 호출을 안정적으로 수행하지 못하고, 시스템은 흐름을 안전하게 닫는 쪽을 선택합니다. 사용자 입장에서는 부탁한 일의 일부만 처리된 응답이 돌아옵니다.

둘째, **max steps 도달**입니다. 1.2.2에서 본 것처럼 planning loop를 막기 위해 시스템은 한 사용자 요청이 만들 수 있는 최대 단계 수를 정해 둡니다. 그런데 이 임계치는 luping뿐 아니라 정상적으로 길게 진행되는 작업도 잘라 버립니다. 한 사용자 요청이 정상으로 14단계가 걸리는 보고서 작성인데 max steps가 10이라면, 에이전트는 10단계째 강제로 종료되며 작성 중이던 결과만 남깁니다.

셋째, **자체 판단 종료**입니다. 모델이 스스로 '이 정도면 답을 줘도 된다'고 판단하고 호출을 멈추는 경우입니다. 사용자가 '강남역 근처 한식당 세 곳을 가격대까지 비교해 줘'라고 부탁했는데, 에이전트가 한 곳만 검색한 뒤 'OO식당이 좋습니다'로 응답을 끝내 버리는 식입니다. 이때는 토큰 한계도 max steps도 닿지 않았는데도 흐름이 닫혔습니다. 모델이 자기 판단으로 충분하다고 본 것입니다.

이 세 원인을 표로 비교하면 다음과 같습니다.

```
원인              종료의 트리거                  사용자가 보는 모습
----------------- ------------------------------ -------------------------
토큰 한계         컨텍스트 누적이 모델 한계 초과 잘린 듯한 응답, 일부만 답함
max steps 도달    단계 수가 정책 임계치 도달     중도에 멈춘 응답
자체 판단 종료    모델이 충분하다고 판단         사용자 의도와 어긋난 종결
```

세 원인 가운데 가장 다루기 어려운 쪽이 자체 판단 종료입니다. 토큰 한계와 max steps는 시스템 쪽에서 임계치를 조정하면 어느 정도 통제가 됩니다. 그러나 자체 판단 종료는 모델 안에서 일어난 결정이라, 시스템은 종료의 결과만 보고 그 안의 이유를 모릅니다.

자체 판단 종료의 뿌리에는 **사용자 의도와 에이전트 종료 조건의 불일치**가 있습니다. 사용자가 '비교해 달라'는 표현을 썼는데 에이전트는 '하나라도 좋은 추천을 주는 것'을 작업의 종료 조건으로 잡으면, 충실히 한 곳을 찾아 응답을 닫고 사용자가 보기에는 일이 덜 끝난 응답이 돌아옵니다. 같은 'X를 정리해 줘'라는 표현도 사용자에 따라 '한 페이지 요약', '항목별 표', '비교표'로 의미가 달라지는데, 에이전트의 종료 조건이 이런 의미 차이를 반영하지 못하면 잦은 조기 종료가 발생합니다.

조기 종료를 잡기 위해 시스템 차원에서 두는 가장 기본 메트릭이 **완수율**입니다. completion rate라고도 부릅니다. 한 사용자 요청에 대해 '요청에 담긴 모든 작업이 응답 안에 처리되었는가'를 0과 1 사이의 비율로 매기는 메트릭입니다. 가장 단순한 형태는 사용자 요청을 작업 단위로 쪼갠 뒤, 응답에서 각 작업이 처리된 흔적이 있는지를 사람이나 평가용 LLM이 채점해 평균을 내는 것입니다. '비교'라는 표현이 들어 있으면 비교 대상의 수와 항목별 처리 여부를 함께 봅니다.

완수율을 도식으로 풀면 다음과 같습니다.

```
사용자 요청               작업 단위        응답 처리 여부   완수율
------------------------- ---------------- ---------------- ----------
강남역 한식당 3곳 비교    1) 3곳 추출      예                3 / 3 = 1.0
                          2) 가격 비교     예
                          3) 추천 정리     예

같은 요청 (조기 종료)     1) 1곳만 추출    부분              1 / 3 = 0.33
                          2) 가격 비교     아니오
                          3) 추천 정리     예
```

조기 종료가 잦은 시스템에서는 평균 완수율이 1에서 떨어지는 패턴이 시간에 따라 보입니다. 이 패턴을 다른 메트릭과 짝지어 보면 원인 갈래가 잡힙니다. 평균 단계 수가 max steps에 자주 닿으면 임계치 문제고, 평균 단계 수는 여유가 있는데 완수율만 낮으면 자체 판단 종료가 우세합니다.

이렇게 원인 갈래가 잡히면 그다음 결정은 **재시도와 부분 응답 정책**입니다. 두 정책의 차이는 다음과 같습니다. 재시도 정책은 조기 종료가 감지되면 같은 요청을 같은 또는 보강된 컨텍스트로 다시 흘려보내는 방식입니다. 일이 길어지지만 사용자에게는 결국 완성된 응답이 돌아갈 가능성이 커집니다. 부분 응답 정책은 끝까지 가지 못한 사실을 사용자에게 명시적으로 알리는 응답을 만드는 방식입니다. 사용자에게 '세 곳 중 한 곳까지 비교했고 나머지는 다음에 이어 진행하시겠어요?'처럼 알리는 응답이 그 예입니다.

```
정책       동작                                     장점                  단점
---------- ---------------------------------------- --------------------- ------------------
재시도     조기 종료 감지 후 자동으로 흐름 재개     완수율 회복            지연·비용 증가
부분 응답  종료 사실과 진행 정도를 사용자에게 통지  사용자 신뢰 유지       완수율은 낮음
```

선택은 도메인에 따라 갈립니다. 사용자가 빠른 응답을 더 중시하는 흐름에서는 부분 응답이 유리하고, 결과의 완성도가 중요한 흐름에서는 재시도가 유리합니다. 두 정책을 섞기도 합니다. 첫 흐름에서는 재시도하지 않고 부분 응답을 돌려주되, 사용자가 '이어서'라고 답하면 재시도가 트리거되는 식입니다. 어느 쪽이든, 정책을 정하지 않고 두면 시스템은 침묵으로 잘린 응답을 그대로 내보내고, 사용자는 일이 끝났다고 오해하게 됩니다.

정리하면, premature termination은 사용자가 요청한 작업이 끝나기 전에 흐름이 닫히는 실패 모드이며, 토큰 한계와 max steps 도달, 자체 판단 종료 세 가지 원인에서 비롯됩니다. 조기 종료는 응답이 돌아갔다는 사실만으로는 잘 잡히지 않으므로 완수율을 별도 메트릭으로 두고, 원인 갈래에 따라 재시도와 부분 응답 정책 가운데 어느 쪽을 기본으로 둘지를 미리 정해야 합니다.

여기까지 1.2의 네 가지 실패 모드를 모두 살펴보았습니다. 도구 선택 오류는 한 단계의 잘못된 결정, planning loop는 흐름이 종결되지 않는 문제, hallucinated reasoning은 결론이 도구 결과와 어긋나는 문제, premature termination은 흐름이 일찍 닫히는 문제로 갈래를 잡았습니다. 다음 단원인 1.3.1에서는 이런 실패 모드를 한 가지 메트릭만으로 좇을 때 시스템 전체가 어떻게 무너지는지를 단일 메트릭 최적화의 함정이라는 주제로 살펴보겠습니다.

이 단원을 마치면 premature termination의 정의와 토큰 한계, max steps 도달, 자체 판단 종료라는 세 원인을 들 수 있고, 사용자 의도와 종료 조건의 불일치가 어떻게 조기 종료로 이어지는지, 완수율 메트릭의 형태와 재시도·부분 응답 정책의 차이를 설명할 수 있습니다.
