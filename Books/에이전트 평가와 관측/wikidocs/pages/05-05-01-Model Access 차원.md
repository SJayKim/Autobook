# 5.5.1 Model Access 차원

앞 단원까지의 사례들이 보여 준 것은 한 면씩의 교훈이었습니다. Anthropic은 관측의 baseline을, Replit은 평가 스택의 hybrid 구성을, JPMorgan은 중앙 플랫폼과 8주 release train을, Klarna는 단일 메트릭에 매달린 결과를 각각 보여 주었습니다. 이 단원부터 다섯 단원에 걸쳐 살펴볼 Palantir AIP는 이 교훈들을 다섯 개의 차원으로 묶어 일종의 표준 점검표로 정리한 사례입니다. 본인이 만든 시스템이 production에 올릴 만한 상태인지 같은 기준으로 자기 점검을 해 볼 수 있게 해 준다는 점에서, 면접 답변과 사내 설계 리뷰 모두에서 활용도가 높은 framework입니다. 이번 단원은 그 첫 번째 차원인 **Model Access**를 다룹니다.

먼저 다섯 차원의 자리를 한 번 묶어 두겠습니다. 본인이 운영할 에이전트 시스템을 통째로 도면 위에 올려 놓으면 다섯 개의 층이 보입니다.

```
 [1. Model Access      ] --- 어떤 모델을, 누가, 어떻게 부를 수 있는가
        |
 [2. Orchestration     ] --- 그 호출이 어떤 runtime 위에서 도는가
        |
 [3. Memory Governance ] --- 호출이 남기는 기억을 누가 어떻게 통제하는가
        |
 [4. Tool Governance   ] --- 호출이 만들어 내는 행동을 어떻게 거르는가
        |
 [5. Observability     ] --- 호출과 행동의 흔적을 어떻게 보고 검증하는가
```

다섯 층은 위에서 아래로 흐릅니다. Model Access가 가장 위에 놓이는 까닭은, 어떤 모델을 어떤 조건으로 끌어올 수 있는지가 정해지지 않으면 그 아래 모든 층이 시작될 수 없기 때문입니다. Orchestration은 이미 가용한 모델 위에서 도는 것이고, Memory Governance는 모델이 남기는 기록을 다루며, Tool과 Observability도 모델이 호출되었음을 전제합니다. 첫 단추가 잘못 끼워지면 나머지 네 단추는 의미를 잃습니다.

이제 Model Access 차원의 정의를 풀겠습니다. **Model Access**는 조직 안의 어떤 사용자와 어떤 애플리케이션이 어떤 외부·내부 모델을 어떤 조건으로 호출할 수 있는지를 정하고, 그 호출이 회사의 정책과 규제를 만족하도록 가운데에서 통제하는 층입니다. '거버넌스 있는 모델 접근'이라는 말이 가리키는 자리가 여기입니다. 개발자가 모델 회사의 키를 직접 들고 다니며 코드에 박아 넣는 방식은 거버넌스가 없는 접근입니다. 누가 무엇을 부르는지 보이지 않고, 정책을 한 번에 바꾸기도 어렵고, 키가 새 나가도 추적이 안 됩니다. Palantir가 표준으로 제시한 형태는 그 반대편에 있습니다. 모든 모델 호출이 조직의 통합 게이트를 거치도록 만든 뒤, 그 게이트에서 인증·권한·감사·과금을 한꺼번에 처리합니다.

이 차원이 통제하는 첫 항목은 **모델 카탈로그**입니다. 카탈로그는 단순한 모델 이름 목록이 아니라, 각 모델에 대해 누가 쓸 수 있고, 어떤 데이터까지 보낼 수 있고, 어떤 retention 정책이 붙어 있는지를 함께 적은 표입니다. 예를 들어 같은 OpenAI GPT 계열이라도 일반 API와 기업용 zero-retention 엔드포인트는 다른 카탈로그 항목으로 등록되고, 사내 데이터를 다루는 팀에는 후자만 노출되는 식입니다. 카탈로그가 있으면 새 모델을 들이는 일은 카탈로그에 줄 하나를 추가하는 일이 되고, 들어 내는 일도 마찬가지가 됩니다. 모델 변경이 코드 곳곳에 흩어진 키를 바꾸는 일이 아니라 카탈로그 한 곳을 갱신하는 일로 줄어드는 것이 이 차원의 첫 번째 효과입니다.

두 번째 통제 항목은 **승인 절차**입니다. 카탈로그에 새 모델을 올리려는 요청이 들어오면, 보안·법무·인프라가 정해진 체크리스트를 거쳐 승인합니다. 체크리스트에는 데이터 거주성, 데이터 사용 정책, 보안 인증, 비용 한도, 책임 담당자 같은 항목이 들어갑니다. 이 절차는 단순한 형식이 아니라 5.3에서 본 JPMorgan식 가드레일 정의의 출발점입니다. 어떤 모델이 카탈로그에 들어와 있다는 사실 자체가 그 모델이 회사의 기준을 통과했다는 보증이 되고, 개발자는 카탈로그 안에서 자유롭게 골라 쓰는 대신 카탈로그 밖의 모델은 함부로 못 부르게 됩니다.

세 번째 항목이 **제공자 retention 없음 정책**입니다. retention이라는 말은 모델 회사가 받은 입력과 출력을 자기 쪽 서버에 얼마나 오래 보관하느냐를 가리킵니다. 일반 API는 학습 개선을 위해 일정 기간 데이터를 남기는 경우가 많지만, 기업이 다루는 데이터는 그쪽 서버에 머무는 것 자체가 곧 사고이거나 규제 위반입니다. 그래서 Palantir 표준은 카탈로그에 올리는 모델을 zero-retention 옵션으로만 받습니다. 회사가 입력을 보내면 모델이 응답을 돌려준 직후 그 입력은 모델 회사 쪽에서 사라집니다. 이 한 줄의 정책 차이가 한국의 금융권과 공공 분야에서는 사실상 사용 가능 여부를 결정합니다. 망분리 환경 안에서 외부 모델을 부르려면, 모델 회사가 데이터를 남기지 않는다는 약속이 우선 서야 합니다.

네 번째 항목인 **통합 리소스 관리**는 이 차원이 단순한 인증 게이트를 넘어서는 자리입니다. 모델 호출은 토큰 비용, 호출 횟수, 동시 처리 능력 같은 자원을 소모합니다. 카탈로그가 있는 게이트는 이 자원을 팀별·앱별·사용자별로 나눠 줍니다. 한 팀이 갑자기 호출을 폭주시키면 다른 팀의 호출이 막히지 않도록 한도가 걸리고, 어떤 앱이 비싸지면 즉시 어디서 새는지 보입니다. 이 자리가 5.3.2에서 본 단일 API 뒤의 다중 모델 라우팅과도 만납니다. 카탈로그에 같은 task를 처리할 수 있는 모델이 여러 개 있을 때, 게이트는 정책에 따라 가장 알맞은 모델을 골라 호출합니다.

한국 규제 환경에서 이 차원이 어떻게 적용되는지 짧게 정리하면 다음과 같습니다. 망분리가 적용되는 금융권은 외부 모델 회사의 API를 직접 부르지 못하므로, 카탈로그가 사실상 사내 게이트웨이와 zero-retention 계약이 맺어진 모델만으로 채워지는 형태가 됩니다. 공공 분야는 클라우드 보안인증을 받은 인프라 위에서만 모델을 호출하도록 카탈로그를 잠그고, 일반 기업이라도 개인정보가 포함된 트래픽을 다루는 앱은 retention 정책과 데이터 거주성 조항이 만족되는 모델 항목만 노출합니다. 같은 회사 안에서도 팀에 따라 카탈로그에서 보이는 모델 목록이 다르게 잘려 보이도록 만드는 것이 일반적인 패턴입니다.

이쯤에서 본인의 시스템을 한 번 self-audit해 볼 수 있는 질문 세 가지를 두겠습니다. 첫째, 본인의 시스템에서 모델 호출은 단일 게이트를 거치고 있습니까, 아니면 코드 곳곳에 모델 회사의 키가 흩어져 있습니까. 둘째, 본인이 쓰는 모델 항목 각각에 대해 retention 정책과 데이터 거주성을 한 줄로 적을 수 있습니까. 셋째, 본인의 카탈로그를 보여 주면 누가 어떤 모델까지 쓸 수 있는지를 외부 감사 담당자에게 한 장으로 설명할 수 있습니까. 이 세 질문에 명확히 답하기 어렵다면, 그 자리가 곧 첫 번째 보강 지점입니다.

정리하면, Palantir AIP가 정의한 5차원의 첫 번째 자리인 Model Access는 조직 안의 모든 모델 호출이 거버넌스 있는 게이트를 거치도록 만들고, 모델 카탈로그와 승인 절차로 어떤 모델이 어떤 조건에서 쓰일 수 있는지를 정하고, 제공자 쪽에 데이터가 남지 않도록 retention 없음 정책을 강제하며, 비용과 자원을 팀·앱 단위로 통합 관리하는 차원입니다. 한국의 망분리·금융·공공 환경에서는 카탈로그가 사용 가능한 모델 자체를 잘라 내는 1차 필터 역할을 하므로, 이 층이 무너지면 그 아래 네 층은 시작도 하기 어렵습니다.

다음 단원인 5.5.2에서는 이 카탈로그 위에서 도는 호출이 어떤 runtime을 만나야 안전한지를 다루겠습니다. stateless·수평 확장 가능 runtime과 request 단위 isolation이 그 자리의 핵심 단어입니다.

이 단원을 마치면 Palantir AIP의 Model Access 차원이 정의하는 모델 카탈로그·승인 절차·제공자 retention 없음 정책·통합 리소스 관리 네 통제 항목을 설명하고, 망분리·금융·공공 같은 한국 규제 환경에서 이 차원이 사용 가능한 모델을 가장 먼저 거르는 역할을 한다는 점을 짚으며, 본인의 시스템에 같은 framework를 적용해 첫 self-audit 질문을 던질 수 있습니다.
