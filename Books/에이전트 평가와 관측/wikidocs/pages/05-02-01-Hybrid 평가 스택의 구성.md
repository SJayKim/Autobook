# 5.2.1 Hybrid 평가 스택의 구성

앞 절에서는 Anthropic이 자기네 멀티에이전트 Research 시스템을 prototype에서 production으로 옮기면서 도입한 tracing, 프라이버시 보존 모니터링, source quality heuristic 같은 패턴을 살폈습니다. 이 절부터는 무대를 바꿔, 코드 작성 자체를 에이전트가 대신해 주는 또 다른 대표 사례인 Replit Agent를 봅니다. Replit은 사용자가 자연어로 요구하면 에이전트가 코드 저장소를 만들고 파일을 짜고 의존성을 설치하고 실제로 실행해 보는 데까지 가는 제품이고, 그래서 평가가 단순히 답을 채점하는 일이 아니라 실제 환경 위에서 행동의 결과를 검증하는 일이 됩니다. 이 단원은 그런 Replit이 평가 도구를 어떻게 조합해 썼는지를 풀어 줍니다.

먼저 사례를 그대로 인용하겠습니다. Replit 엔지니어들의 발표를 짧게 옮기면, "Braintrust는 일부 평가에, LangSmith는 traceability에 쓰지만 평가 harness 대부분은 자체 Python으로 구축"입니다. 이 한 줄에 hybrid 평가 스택의 핵심 결정이 다 들어 있습니다. 평가용 SaaS 한 개, 관측용 SaaS 한 개, 그리고 두 도구가 비워 두는 자리를 덮는 자체 Python 코드 한 덩이입니다. 이 구도는 3.5.2에서 hybrid 스택 패턴의 일반론으로 한 번 다뤘던 골격이고, 이번 단원은 그 일반론이 Replit이라는 한 회사의 사정 안에서 어떻게 자리 잡았는지를 더 가까이 들여다봅니다.

용어부터 다시 짚겠습니다. **Braintrust**는 데이터셋과 실험을 관리하고 평가 점수를 비교해 주는 데에 무게를 둔 평가 플랫폼입니다. 같은 입력에 대해 모델 A와 B의 점수가 몇 점이 났고, 어느 케이스에서 갈렸는지 같은 화면을 잘 그려 줍니다. **LangSmith**는 3.3에서 본 대로 한 번의 에이전트 호출이 만들어 낸 단계들을 부모-자식 trace로 펼쳐 보여 주는 관측 플랫폼입니다. 어디서 시간이 오래 걸렸고 어느 도구 호출에서 예외가 났는지를 한 화면에서 따라가는 데에 강합니다. **harness**는 1장에서 다룬 대로 모델 바깥에서 실행 환경과 평가 흐름을 묶어 주는 골격을 가리키는 일반 용어이고, 여기서는 평가를 한 번 돌리기 위해 필요한 환경 준비, 시나리오 주입, 결과 수집을 모두 포함하는 자체 코드 더미를 가리킵니다.

이 셋이 어떻게 역할을 나누는지를 그림으로 정리하면 다음과 같습니다.

```
            [Replit Agent]
                  |
        한 번의 평가 실행 요청
                  |
                  v
       +-------------------------+
       | 자체 Python harness     |
       |  - Docker 컨테이너 기동 |
       |  - Chrome 인스턴스 깨움 |
       |  - 시나리오 주입         |
       |  - 결과/메트릭 수집     |
       +-------------------------+
            |              |
   평가 결과/점수      span/trace 발행
            |              |
            v              v
      [Braintrust]     [LangSmith]
      - 데이터셋        - 단계별 trace
      - 실험 비교       - 도구 호출 펼침
      - 점수 추이       - replay/fork
```

이 구도에서 가장 먼저 짚을 점은 **각 도구의 책임 분담**입니다. 같은 평가를 한 번 돌리면 두 가지 산출물이 동시에 나옵니다. 하나는 그 회차의 점수와 통과 여부 같은 집계 데이터이고, 다른 하나는 각 단계에서 어떤 도구가 어떤 인자로 불려 무엇을 돌려받았는지의 trace 데이터입니다. 점수는 시간이 지나면서 추이를 보고 회귀를 잡는 데 쓰이고, trace는 한 번의 실패를 펼쳐 무엇이 어디서 어긋났는지를 사람이 따라가는 데 쓰입니다. Replit은 두 화면을 같은 도구 안에서 다 보려고 하지 않고, 점수 화면은 Braintrust로, trace 화면은 LangSmith로 각각 잘 만들어진 도구에 맡기는 길을 골랐습니다. 두 도구가 같은 데이터를 다르게 가공하는 것이 아니라, 애초에 다른 화면을 위해 다른 데이터를 받습니다.

다음으로 **자체 harness가 메우는 공간**을 풀겠습니다. Replit Agent는 사용자가 부탁한 코드를 실제로 실행해 봐야 합니다. 즉 평가 한 번을 돌리기 위해 코드를 받을 컨테이너를 띄우고, 그 안에서 의존성을 설치하고, 미리 준비한 시나리오를 사용자처럼 흘려 넣고, 마지막에 결과 화면이나 파일 상태를 거두어들이는 긴 서막이 필요합니다. 이런 환경 준비 코드는 Braintrust나 LangSmith가 직접 다루는 영역이 아닙니다. 두 SaaS는 모두 "평가 점수가 만들어진 다음" 또는 "trace span이 발행된 다음"을 다루는 도구이고, 그 점수와 span을 만들어 내기 위해 컨테이너와 브라우저를 깨우는 일은 회사 사정에 맞춰 직접 짜야 합니다. 자체 Python harness는 정확히 이 자리, 즉 평가 실행 한 번을 위한 환경 준비와 결과 수집의 앞단을 책임집니다. SaaS는 그 결과를 받아 화면으로 보여 주는 뒷단을 맡습니다.

같은 구분을 다른 각도에서 보면, **각 도구의 경계**가 분명해집니다. Braintrust 안쪽은 데이터셋을 정의하고 실험을 비교하는 영역, LangSmith 안쪽은 trace를 펼쳐 디버깅하는 영역, 자체 harness 안쪽은 환경 isolation, cleanup, 사내 정책 적용 같은 SaaS가 절대 채워 주지 않을 영역입니다. 이 경계가 흔들리는 순간, 예를 들어 SaaS가 곧 채워 줄 기능까지 자체 harness가 미리 짜 두면, 같은 일을 두 곳에서 두 번 짜는 비용이 발생하고 SaaS의 새 기능을 받아들이기 어려워집니다. Replit이 "harness 대부분은 자체 Python으로 구축"이라고 말한 이유는, 두 SaaS의 경계 바깥이 그만큼 넓고 회사 사정에 깊게 묶여 있기 때문이지 SaaS가 비효율적이라서가 아닙니다.

이런 hybrid 구도는 곧바로 **운영 복잡도**라는 비용을 청구합니다. SaaS 두 개의 가격, 보안 심의, 데이터 보존 정책을 따로 검토해야 하고, 사용자 권한과 접근 제어를 두 곳에서 일관되게 유지해야 합니다. 사고가 났을 때 trace를 따라가다 도구 경계를 넘는 일이 잦아지면 디버깅이 길어지고, 자체 harness 코드는 회사가 영원히 같이 들고 가야 할 자산이자 부채가 됩니다. 도구가 늘어날수록 운영 부담은 산술적으로 두 배가 되지 않고 보통 두 배 이상으로 늘어난다는 사실은 3.5.2에서 짚은 그대로입니다. Replit이 굳이 이 비용을 떠안는 이유는 다음 단원에서 볼 web agent 평가 환경의 까다로움이 한 도구로는 풀리지 않기 때문이고, 그래서 hybrid는 도구 두 개를 어쩌다 같이 쓰게 된 결과가 아니라 의식적으로 비용을 지불하고 얻는 유연성에 가깝습니다.

마지막으로 한 가지 오해를 짚겠습니다. "harness 대부분이 자체 Python"이라는 말은 SaaS를 약하게 쓴다는 뜻이 아니라, SaaS가 잘하는 영역과 그 바깥을 정직하게 가른 결과입니다. 점수 화면과 trace 화면은 SaaS에 맡기고, 환경 준비처럼 회사 고유의 사정은 자체 코드에 둡니다. 어느 한쪽을 더 키우려 들면 균형이 무너집니다. 자체 코드가 더 커지면 SaaS의 가치를 못 살리고, SaaS에 모든 것을 맡기려 들면 환경 준비가 어색하게 어그러집니다. hybrid의 묘는 이 분담 자체에 있습니다.

정리하면, Replit의 hybrid 평가 스택은 점수·실험 비교를 Braintrust에, 단계별 trace 디버깅을 LangSmith에 맡기고, 두 SaaS가 비워 두는 환경 준비와 결과 수집의 앞단을 자체 Python harness가 메우는 구성입니다. 도구마다 책임이 분명하게 갈려 있어 같은 데이터를 두 곳에서 다르게 가공하지 않고, 그 대가로 늘어나는 운영 복잡도는 단일 도구 시작·얇은 글루 코드·SaaS 빈 자리에만 두는 자체 harness라는 기준으로 다스립니다.

다음 단원인 5.2.2에서는 자체 harness가 실제로 무엇을 하고 있는지를 한 단계 안으로 들어가, Chrome 인스턴스와 Docker 컨테이너 위에서 Anthropic의 computer use를 활용해 web agent를 사용자처럼 동작시키는 평가 방식을 살피겠습니다.

이 단원을 마치면 Replit의 평가 스택에서 Braintrust가 평가 점수와 실험 비교를, LangSmith가 traceability를 맡고, 자체 Python harness가 환경 준비와 결과 수집의 빈자리를 메우는 분담 구조를 설명하고, 이 hybrid 구성이 가져오는 운영 복잡도를 어떻게 다스리는지를 함께 정리할 수 있습니다.
