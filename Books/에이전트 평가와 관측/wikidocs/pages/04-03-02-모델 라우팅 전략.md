# 4.3.2 모델 라우팅 전략

앞 단원에서 본 fallback이 '주 경로가 실패했을 때 다른 길로 돌리는 사후 대응'이었다면, 이번 단원에서 다루는 **모델 라우팅**은 '요청이 들어온 순간 어떤 모델로 보낼지를 미리 고르는 사전 결정'입니다. 동적 fallback을 한 걸음 일반화한 형태이며, 매 요청마다 어떤 사업자의 어떤 모델을 호출할지 고르는 일을 시스템화한 것입니다. 라우팅을 잘 짜 두면 fallback이 발동되는 빈도 자체가 줄고, 비용과 품질을 함께 관리할 수 있는 손잡이가 생깁니다.

먼저 라우팅이라는 용어부터 풀겠습니다. 네트워크 장비에서 패킷이 목적지에 가장 잘 닿도록 다음 홉을 고르는 작업을 라우팅이라고 부릅니다. 모델 라우팅도 같은 비유입니다. 사용자의 한 요청이 들어왔을 때, 그 요청을 어느 모델로 보내야 가장 적절한 답이 나올지 결정하는 작업입니다. 다만 네트워크에서는 거리와 지연이 결정 기준이지만, 모델 라우팅에서는 task type, cost, accuracy 같은 새로운 축이 더해집니다.

라우팅의 가장 잘 알려진 산업 사례는 미국 은행 JPMorgan의 LLM Suite입니다. 이 회사는 약 25만 명에 이르는 직원이 쓰는 사내 AI 플랫폼을 운영하고 있는데, 여기에 적용된 핵심 패턴이 바로 단일 API 뒤에 여러 사업자의 모델을 두는 구조입니다. 사례 자료에는 '단일 API 뒤에서 OpenAI·Anthropic 모델 선택 가능, task type·cost·accuracy 기준으로 라우팅'이라고 정리되어 있습니다. 사내 애플리케이션 개발자들은 모델 사업자가 누구인지 알 필요 없이 LLM Suite의 단일 엔드포인트만 호출하고, 어느 모델로 갈지는 플랫폼이 결정합니다. 이 방식은 사업자가 바뀌거나 새 모델이 추가되어도 클라이언트 쪽 코드를 손대지 않고 갈아 끼울 수 있게 해 줍니다. 운영 측면에서도 과금과 관측이 한 곳에 모이기 때문에, 회사 전체가 어떤 모델에 얼마를 쓰고 있는지 한눈에 볼 수 있습니다.

이 추상화가 작동하려면 라우팅 결정을 내리는 부분이 필요합니다. 그 가운데 첫째 축은 **task type 분류기**입니다. 들어온 요청이 어떤 종류의 작업인지를 짧게 판별해서 그 종류에 맞는 모델로 보내는 역할을 합니다. 예를 들어 코드 생성, 긴 문서 요약, 사실 질문 답변, 정형 데이터 추출은 각각 잘하는 모델이 다릅니다. 분류기는 별도의 작은 모델로 구현하기도 하고, 정규식이나 키워드 규칙으로 만들기도 합니다. 분류 결과는 보통 라벨 한두 개로 떨어지고, 그 라벨이 라우팅 테이블의 키가 됩니다.

둘째 축은 **cost-accuracy 트레이드오프**입니다. 같은 task type이라도 사용 가능한 모델이 여러 개 있을 때, 비용이 낮은 모델로도 충분한 요청과 비싼 모델이 필요한 요청을 갈라야 합니다. 단순한 일상 질문은 작은 모델로도 충분하지만, 회계 계산이나 법무 검토 같은 작업은 정확도가 떨어지면 손해가 큽니다. 따라서 같은 분류 라벨 안에서도 요청의 길이, 위험도, 사용자 등급에 따라 모델을 다시 한 번 가르는 두 번째 결정이 들어갑니다. 이 트레이드오프는 보통 표 한 장으로 정리되며, 각 칸은 'task 라벨 × 위험 수준'에 매핑된 모델 이름을 담습니다.

셋째 축은 **모델 가용성과 SLA**입니다. 사업자별로 응답 시간 보장 수준과 장애 빈도가 다릅니다. 라우팅 시점에 각 모델의 최근 응답 지연, 에러율, rate limit 잔량을 살펴보고, 1순위 모델이 흔들리고 있다면 2순위 모델로 자동으로 옮기는 결정도 필요합니다. 이 결정은 앞 단원에서 본 동적 fallback과 사실상 같은 메커니즘이며, 다른 점은 fallback이 실패 후에 발동되는 데 비해 라우팅은 실패 직전의 신호로 미리 옮긴다는 것입니다.

세 축을 묶어 라우팅 결정을 그림으로 그리면 다음과 같습니다. 같은 입구에서 출발한 요청이 분류, 비용·정확도 매핑, 가용성 점검이라는 세 단계의 분기를 거쳐 최종 모델로 보내집니다.

```
                  [요청]
                    |
                    v
            [task type 분류]
        /        |         \
   코드 생성    문서 요약    사실 질문
       |          |            |
       v          v            v
  [cost / accuracy 매핑]
   저위험 -> 작은 모델 (싼 사업자 A)
   고위험 -> 큰 모델  (정확도 우선 사업자 B)
       |
       v
   [가용성 / SLA 점검]
   1순위 모델 정상? -> 그대로 호출
   1순위 흔들림?    -> 2순위로 우회
       |
       v
        [최종 모델 호출]
```

라우팅 시스템에서 흔히 빠지기 쉬운 함정은, 결정이 어디서 어떻게 일어났는지 사후에 알 수 없게 만드는 일입니다. 같은 요청이 어떤 날에는 사업자 A로, 다른 날에는 사업자 B로 갔는데 그 이유가 기록되지 않으면, 품질이 떨어졌을 때 원인을 추적할 길이 없어집니다. 따라서 **라우팅 결정의 관측**은 라우팅 시스템의 일부로 처음부터 함께 설계해야 합니다. 매 요청 trace에 어떤 분류 라벨이 붙었고, 어떤 비용·위험 매핑이 적용됐고, 가용성 점검 결과 1순위가 정상이었는지 우회였는지를 함께 남깁니다. 앞 장에서 본 OpenTelemetry GenAI semantic conventions의 gen_ai.system, gen_ai.request.model 속성이 바로 이 자리에서 빛을 발합니다. 사업자와 모델 이름이 표준 키로 trace에 박혀 있으면, 나중에 '품질이 떨어진 시점에 어떤 모델로 라우팅됐는가'를 한 번의 쿼리로 확인할 수 있습니다.

라우팅을 도입할 때 한국 기업 입장에서 참고할 점이 한 가지 더 있습니다. JPMorgan 사례에서 단일 API의 진짜 가치는 모델 사업자 변경의 영향 범위를 클라이언트로부터 차단한다는 데 있습니다. 새 모델이 나오거나 사업자 한 곳의 단가가 바뀌어도, 사내 수천 개의 응용 프로그램은 코드를 손대지 않고 그대로 돌아갑니다. 이 추상화는 처음 만들 때는 한 줄짜리 호출보다 무거워 보이지만, 도입 후 1, 2년이 지나면 운영 비용이 급격히 줄어듭니다.

정리하면, 모델 라우팅은 매 요청마다 어떤 사업자의 어떤 모델을 호출할지 사전에 결정하는 시스템이며, JPMorgan의 LLM Suite처럼 단일 API 뒤에 여러 사업자의 모델을 두고 task type, cost, accuracy를 축으로 결정합니다. task type 분류기가 요청의 종류를 판별하고, 비용·정확도 매핑이 같은 종류 안에서 모델을 다시 가르며, 가용성과 SLA 점검이 1순위 모델의 흔들림을 보고 자동으로 우회 경로를 고릅니다. 라우팅 결정은 trace에 함께 남겨야 사후 디버깅이 가능하며, 단일 API 추상화는 사업자 교체 시 클라이언트 영향을 차단해 장기적인 운영 비용을 줄입니다.

다음 단원인 4.3.3에서는 fallback과 라우팅을 모두 거쳐도 시스템이 회복할 수 없는 상황이 왔을 때, 전체 서비스가 한꺼번에 무너지는 대신 기능을 단계적으로 줄이며 버티는 graceful degradation을 다룹니다.

이 단원을 마치면 모델 라우팅을 동적 fallback의 일반화로 위치시키고, JPMorgan의 단일 API + 다중 모델 구조에서 task type 분류, cost-accuracy 트레이드오프, 모델 가용성과 SLA, 라우팅 결정의 관측이라는 네 가지 축이 어떻게 결합되는지를 설명할 수 있습니다.
