# 3.5.1 한국 기업의 의사결정 프레임워크

앞 두 섹션에서 LangSmith와 Langfuse를 각각 깊이 살폈습니다. 두 도구는 같은 trace 화면을 보여 준다는 점에서 표면이 비슷하지만, 데이터를 어디에 두는지, 표준을 어떻게 받아들이는지, 누가 운영을 책임지는지에서 갈라집니다. 이 단원은 한국 기업이 두 도구 가운데 하나를 고를 때 따져야 하는 차원을 다섯 가지로 정리하고, 그 차원들을 의사결정 트리로 묶어 봅니다. 정답을 미리 정해 두는 글이 아니라, 회사마다 다른 답이 나올 수밖에 없는 이유를 분명히 하는 글입니다.

첫째 차원은 **데이터 주권과 보안 정책**입니다. 데이터 주권이라는 말은 어떤 데이터가 어느 나라의 어느 사업자 인프라 위에 머무를지를 자기 손으로 정할 수 있는 상태를 가리킵니다. 한국 금융권의 망분리 규제, 공공 부문의 국내 사업자 우선 정책, EU 시장에 서비스를 내보내는 회사가 마주치는 GDPR의 데이터 이전 제한이 모두 이 차원에 들어옵니다. 망분리는 인터넷에 닿는 망과 내부 업무망을 물리적으로 떼어 두는 한국 금융권 특유의 통제로, trace 데이터를 사외 SaaS로 흘려 보내는 일 자체가 보안 심사에서 막히는 경우가 많습니다. GDPR은 EU 거주자의 개인정보를 EU 밖으로 이전할 때 별도 근거를 요구하기 때문에, prompt에 사용자 입력이 그대로 담기는 LLM trace는 특히 까다롭게 다뤄집니다. 회사가 이 가운데 어느 하나라도 해당되면 SaaS 단독은 곤란해지고, 자체 호스팅이 가능한 도구가 후보군에 남습니다.

둘째 차원은 **Self-host 가능 여부**입니다. self-host는 도구를 만든 회사의 클라우드가 아니라 자기 인프라 위에 그 도구를 직접 띄워 운영하는 방식을 가리킵니다. Langfuse는 오픈소스로 공개되어 있고 자체 호스팅을 일급 시민으로 다룹니다. LangSmith도 self-hosted 옵션을 제공하지만 기본 사용 형태는 호스티드 SaaS이고, self-hosted 변형은 별도 라이선스와 더 많은 운영 손길을 요구합니다. 자체 호스팅이 후보에 오르면 곧장 다음 차원으로 이어집니다.

셋째 차원은 **팀 규모와 운영 부담**입니다. self-host는 데이터 주권을 얻는 대신 인프라 운영을 짊어집니다. Postgres와 ClickHouse 같은 백엔드를 띄우고, 디스크와 메모리를 챙기고, 백업과 보안 패치를 돌리고, 버전 업그레이드 시 호환성을 검증하는 일이 모두 회사 몫이 됩니다. 인프라 팀이 이미 PostgreSQL을 운영해 본 적 있는 대기업이라면 추가 부담이 작지만, 데이터 엔지니어가 한두 명인 스타트업이 self-host를 떠안으면 본업이 운영으로 잠식됩니다. 이 차원은 곱셈으로 작용하기 때문에, 데이터 주권 요구가 강하지 않은 회사는 같은 self-host 비용을 굳이 치를 이유가 사라집니다.

넷째 차원은 **통합 비용과 마이그레이션 부담**입니다. 이미 LangChain이나 LangGraph로 코드를 짜 둔 팀에게는 LangSmith가 자동 인스트루먼트로 거의 추가 작업 없이 trace를 잡아 줍니다. 같은 팀이 Langfuse로 옮기려면 SDK를 한 번 갈아 끼워야 하고, 자동 계측이 닿지 않는 부분은 손으로 span을 열어 둬야 합니다. 반대로 OpenAI SDK와 Anthropic SDK를 직접 호출하는 코드 위에서는 둘의 통합 비용 차이가 작아집니다. 마이그레이션 부담은 코드뿐 아니라 사람에게도 붙습니다. 데이터셋 정의, 평가 워크플로, 권한 모델은 도구마다 표현 방식이 다르기 때문에 옮기는 순간 운영팀이 다시 배워야 합니다. 이 비용을 무시하고 도구를 자주 갈아엎으면 평가 자체가 멈춥니다.

다섯째 차원은 **벤더 신뢰성과 로드맵**입니다. 호스티드 SaaS를 쓴다는 것은 그 회사의 가용성, 가격 정책, 제품 방향에 자기 운영을 의탁한다는 뜻입니다. 회사가 인수되거나 가격이 갑자기 오르거나 제품 방향이 바뀔 때 받을 충격을 미리 가늠해야 합니다. 오픈소스 자체 호스팅은 이 충격을 줄이지만 완전히 없애 주지는 않습니다. 오픈소스 프로젝트도 핵심 메인테이너의 동력에 영향을 받기 때문에, 커뮤니티 활동, 릴리즈 주기, 보안 이슈 대응 속도를 함께 보아 두는 편이 안전합니다.

다섯 차원을 의사결정 트리로 정리하면 다음 그림처럼 풀어집니다.

```
 [질문 1] 망분리·금융권 규제·GDPR로 사외 SaaS 사용이 막히는가
        |
        +-- 예 --> 자체 호스팅 가능 도구만 후보 (Langfuse 우선, LangSmith self-hosted)
        |             |
        |             [질문 2] 인프라 운영 인력이 충분한가
        |                    |
        |                    +-- 예 --> Langfuse self-host
        |                    |
        |                    +-- 아니오 --> 망분리 안의 매니지드 PaaS 활용 또는 도입 보류
        |
        +-- 아니오 --> SaaS 후보 포함
                      |
                      [질문 3] LangChain/LangGraph 비중이 높은가
                             |
                             +-- 예, 통합 비용 최소화가 1순위 --> LangSmith Hosted
                             |
                             +-- 아니오, OTel 표준·다중 백엔드가 1순위 --> Langfuse Cloud
                             |
                             +-- 둘 다 중요 --> 3.5.2의 hybrid 패턴 검토
```

이 트리는 절대적인 답이 아니라 차원별 가중치를 분명히 하기 위한 도구입니다. 회사마다 가중치가 다르기 때문에, 같은 회사 안에서도 사업부에 따라 다른 답이 나오는 일이 흔합니다. 예를 들어 한국 시중은행의 대고객 챗봇 시스템은 첫 질문에서 곧장 self-host 분기로 빠지는 반면, 같은 은행의 사내 코드 리뷰 보조 도구는 사용자가 임직원으로 한정되고 데이터도 사내 코드라서 SaaS가 허용될 수 있습니다. 한 회사가 두 도구를 동시에 쓰는 상황은 이렇게 자연스럽게 만들어집니다.

마지막으로 한국 기업 맥락에서 자주 함께 따져야 하는 항목을 짧게 짚겠습니다. 금융권은 보통 신규 도구 도입 시 보안성 심의를 거치고, 클라우드 사용은 금융보안원의 가이드와 회사 내부 클라우드 정책의 교집합 안에서만 허용됩니다. 공공 부문은 국가정보원 보안성 검토를 받는 시스템에서 사외 SaaS를 쓰기 어렵고, 국내 클라우드 보안 인증을 받은 사업자 위에서만 운영이 가능합니다. EU 시장 진출 기업은 GDPR의 적정성 결정과 표준계약조항을 검토해 trace 데이터의 EU 밖 이전 근거를 마련해야 합니다. 이 항목들은 도구의 기술 우열을 따지기 전에 후보군 자체를 좁히는 필터로 작용하기 때문에, 의사결정의 가장 위에 두는 편이 시간을 아낍니다.

정리하면, LangSmith와 Langfuse 사이의 선택은 단순한 기능 비교가 아니라 데이터 주권과 보안 정책, self-host 가능 여부, 팀 규모와 운영 부담, 통합 비용과 마이그레이션 부담, 벤더 신뢰성과 로드맵이라는 다섯 차원의 가중합으로 풀립니다. 한국 기업의 망분리, 금융권 규제, EU GDPR 같은 제약은 이 다섯 차원 가운데 첫 두 항목을 무겁게 만들어 self-host를 우선 후보로 끌어올리고, 그 위에 통합 비용과 운영 인력이 더해지면서 회사마다 다른 결론에 도달합니다.

다음 단원인 3.5.2에서는 한 회사 안에서 한 도구로는 모든 요구를 다 채우지 못할 때 두 도구 이상을 묶어 쓰는 hybrid 스택 구성 패턴을 살펴보겠습니다. Replit이 평가에는 Braintrust, traceability에는 LangSmith를 함께 쓰는 사례가 그 출발점이 됩니다.

이 단원을 마치면 한국 기업이 LangSmith와 Langfuse 가운데 하나를 고를 때 따져야 하는 데이터 주권, self-host 가능성, 팀 규모, 통합 비용, 벤더 신뢰성 다섯 차원을 망분리·금융권·GDPR 같은 한국 맥락 위에서 의사결정 트리로 풀어 설명할 수 있습니다.
