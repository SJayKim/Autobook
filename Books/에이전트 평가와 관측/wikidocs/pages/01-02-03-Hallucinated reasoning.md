# 1.2.3 Hallucinated reasoning

앞 단원에서는 에이전트가 같은 자리를 빙빙 도는 planning loop를 다루었습니다. planning loop는 흐름이 멈추지 않아 답을 못 내는 쪽이라면, 이번 단원에서 살피는 실패 모드는 흐름은 잘 끝나는데 그 끝에 놓인 결론이 도구 결과와 어긋나 있는 쪽입니다. 이 모순적인 종결을 부르는 이름이 hallucinated reasoning입니다.

먼저 용어 자리부터 잡겠습니다. 에이전트는 매 단계에서 추론을 거쳐 도구를 호출하고, 도구가 돌려준 관찰을 다시 추론에 입력합니다. 정상 흐름에서는 추론이 관찰의 영향을 받습니다. 검색 결과에 'A는 휴업 중'이라는 정보가 들어오면, 다음 추론에서 'A는 영업 중'이라는 가정은 사라져야 합니다. 그런데 가끔 모델은 관찰을 받아들이지 않고 직전 추론의 가정을 그대로 끌고 갑니다. 도구 결과에 근거하지 않은 추론이 결론으로 굳어지는 이 패턴을 **hallucinated reasoning**이라고 부릅니다.

이 패턴은 단일 LLM 응답에서 자주 거론되는 사실 환각과 닮아 보이지만 결이 다릅니다. 사실 환각은 모델이 외부에 정보를 묻지 않고 자기 기억으로만 답할 때 가짜 사실을 지어내는 현상입니다. 반면 hallucinated reasoning은 외부에 정보를 묻고 응답까지 받아 둔 상태에서 그 응답과 모순되는 결론을 내는 현상입니다. 즉, 에이전트는 도구를 부르긴 부른 것입니다. 다만 도구 결과를 활용하지 않았을 뿐입니다.

대표적인 사례를 시나리오로 보겠습니다. 사용자가 '서울 강남역 근처에 지금 영업 중인 한식당이 있나요?'라고 묻습니다. 에이전트는 search_restaurants(area='강남역', open_now=True)를 부릅니다. 도구는 '근처 영업 중 한식당 0곳, 인근 영업 중 일식당 3곳'이라는 결과를 돌려줍니다. 정상 흐름이라면 모델은 '한식당은 지금 영업 중인 곳이 없습니다'라고 응답하거나, 한식 조건을 풀고 다른 음식점을 제안할지 묻는 단계로 넘어갑니다. 그런데 hallucinated reasoning이 일어난 흐름에서는 모델이 마치 결과에 한식당이 들어 있던 것처럼 'OO한식이 영업 중'이라는 식의 답을 만듭니다. 도구 결과의 0곳이라는 신호가 추론에 반영되지 않은 것입니다.

이 흐름을 도식으로 옮기면 다음과 같습니다.

```
[사용자]    "강남역 영업 중 한식당?"
[추론 1]    "search_restaurants 부르자"
[도구 결과] 한식당 0, 일식당 3
[추론 2]    "OO한식이 영업 중인 것 같다"   <- 결과와 모순
[응답]      "OO한식을 추천드립니다"        <- 근거 없음
```

도식을 보면 모순이 생기는 자리가 분명히 보입니다. 추론 2가 도구 결과를 인용하는 대신 모델이 가진 일반 지식이나 사용자 질문의 표현에서 끌려옵니다. 이런 결론을 굳혀 응답으로 내보내는 순간, 사용자는 '에이전트가 검색해 봤다'는 인상을 받지만 실제로는 검색 결과를 무시한 응답을 받습니다.

이런 일이 일어나는 까닭은 모델이 직전 추론을 자기 자신의 가장 강력한 단서로 쓰기 때문입니다. 모델 입장에서 추론 1은 '나도 방금 그렇게 생각했다'는 흐름의 일관성으로 작동합니다. 도구 결과가 약하게 표현되어 들어오면, 일관성을 유지하려는 압력이 결과의 무게를 누릅니다. 결과가 빈 배열, 한 줄짜리 빈 응답, 모호한 안내문일수록 이 현상이 잘 일어납니다. 또한 에이전트가 동시에 여러 도구를 부른 뒤 결과를 합치는 단계에서, 한 도구의 결과만 비중 있게 반영하고 다른 도구의 결과를 흘려보내는 형태로도 나타납니다. 이때 에이전트는 자기 결론에 유리한 결과만 선택적으로 인용하게 됩니다.

또 하나 자주 보이는 모습은 **근거 없는 사실 주장**입니다. 모델이 도구를 한 번도 부르지 않은 영역에 대해, 마치 도구 결과인 것처럼 구체적인 수치나 일정을 응답에 끼워 넣는 경우입니다. 예컨대 검색 결과에는 일식당 정보만 있는데 응답에는 '한식당 OO이 22시까지 영업한다'는 문장이 들어가는 식입니다. 사용자 입장에서는 '검색해서 알려 준 정보'로 보이고, 따라서 더 잘 속습니다. 일반 LLM 환각보다 hallucinated reasoning이 더 위험한 까닭이 여기 있습니다. 도구를 부르는 에이전트는 사용자에게 '근거 있는 답'이라는 신뢰를 깔고 응답하기 때문에, 가짜 정보가 더 진짜처럼 흐릅니다.

평가 쪽에서는 이 패턴을 잡기 위해 final-answer만 보는 채점에서 한 발 더 들어가야 합니다. 핵심은 trajectory에 남은 도구 결과와 최종 응답을 함께 보는 것입니다. 이를 **trajectory 검증**이라고 부르겠습니다. 가장 단순한 형태는 응답에 등장한 사실들을 trajectory의 도구 결과 안에서 일치하는 부분으로 매핑할 수 있는지를 보는 것입니다. 응답에 'OO한식 22시까지'가 들어 있다면, 이 문장이 검색 결과의 어떤 항목에서 왔는지 가리키는 출처 매핑이 가능해야 합니다. 가능하지 않다면 그 문장은 근거 없는 주장으로 분류됩니다.

trajectory 검증을 메트릭으로 다듬은 것이 **근거-결론 일치도 평가 항목**입니다. 한 응답에 들어 있는 사실 명제를 추출하고, 각 명제가 trajectory의 어떤 단계에 있는 도구 결과와 부합하는지를 사람이나 평가용 LLM이 판단합니다. 부합 여부에 따라 명제 단위로 점수를 매겨 평균을 내면, 같은 응답 안에서 어느 부분이 근거에 의지하고 어느 부분이 모델 추론으로 메워졌는지가 보입니다. 메트릭의 모양을 도식으로 정리하면 다음과 같습니다.

```
응답 명제                근거가 trajectory에 있나?  분류
------------------------ -------------------------- -----------------
OO한식 영업 중           아니오 (결과는 0곳)        근거 없음
일식 3곳 가능            예 (search 결과에 있음)    근거 있음
22시까지 영업            아니오 (영업 시간 미조회)  근거 없음
근거 일치도              1 / 3                      0.33
```

이 표에서 보듯 근거 일치도가 낮은 응답은 final-answer가 아무리 그럴듯해도 hallucinated reasoning이 끼어든 흐름으로 표시됩니다. 시스템 차원에서 이 메트릭을 추적하면, 어느 도구의 결과가 자주 무시되는지, 어떤 종류의 사용자 질문에서 모델이 자기 추론을 더 신뢰하는지 같은 패턴이 드러납니다.

정리하면, hallucinated reasoning은 도구 호출 자체는 일어났지만 그 결과를 무시하고 자기 추론을 따라가는 실패 모드이며, 사실 환각보다 더 진짜처럼 보이는 응답으로 사용자를 오도합니다. final-answer만으로는 잘 잡히지 않으므로 trajectory 검증을 통해 응답 명제를 도구 결과로 되짚어 보고, 근거-결론 일치도를 평가 항목으로 두어 어느 흐름에서 모델이 결과를 무시했는지를 가시화해야 합니다.

다음 단원인 1.2.4에서는 정반대 결의 실패 모드, 즉 작업을 완수하지 않고 흐름을 너무 일찍 끊어 버리는 premature termination을 살펴보겠습니다.

이 단원을 마치면 hallucinated reasoning의 정의와 도구 결과를 무시하고 자기 추론을 따르는 패턴, 근거 없는 사실 주장의 사례를 들 수 있고, trajectory 검증으로 잡는 방법과 근거-결론 일치도 평가 항목의 의미를 설명할 수 있습니다.
