# 5.5.4 Tool Governance 차원

앞 단원에서 본 Memory Governance는 에이전트가 무엇을 알 수 있는지를 정한 자리였습니다. 그 다음 자리에서 묻는 질문은 한 단계 더 무겁습니다. 에이전트는 무엇을 할 수 있는가. 자료를 읽는 일과 외부 시스템의 상태를 바꾸는 일은 결이 다릅니다. 자료를 잘못 읽었다면 응답 한 번이 어긋나는 데서 끝나지만, 외부 시스템에 잘못 명령을 내리면 송금이 잘못 가고, 주문이 잘못 들어가고, 데이터베이스의 행이 통째로 지워지는 결과가 따라옵니다. 이 행동의 자리를 다루는 네 번째 차원이 **Tool Governance**입니다. 이번 단원은 도구 카탈로그·HITL 승인·통제된 액션 실행 환경·도구별 권한 매트릭스·감사 로그와 회수 다섯 항목을 4.1의 tiered autonomy와 4.4의 HITL 위에서 다시 묶어 보겠습니다.

먼저 도구라는 단어를 한 번 더 풀어 두겠습니다. 1.1에서 에이전트의 네 구성요소 중 하나로 들었던 도구는, 모델이 외부 세계에 일을 시키는 통로입니다. 검색 API, 데이터베이스 쿼리, 파일 쓰기, 이메일 발송, 결제 호출 같은 모든 외부 호출이 도구라는 추상 위에 얹힙니다. 모델은 도구를 직접 부르지 못하고, runtime이 모델의 의도를 받아 도구를 대신 부르는 구조입니다. Tool Governance가 통제하는 자리는 바로 이 '대신 부르는' 단계입니다.

첫 번째 통제 항목은 **도구 카탈로그**입니다. 5.5.1의 모델 카탈로그와 같은 사고방식이 도구에도 그대로 적용됩니다. 어떤 도구가 시스템에 등록되어 있고, 그 도구가 무엇을 하며, 누가 어떤 조건에서 부를 수 있는지를 한 표 위에 정리해 둡니다. 카탈로그에 없는 도구는 에이전트가 부르려 해도 runtime이 거부합니다. 새 도구를 들이는 일은 카탈로그에 줄을 한 줄 추가하는 일이 되고, 그 줄에는 도구 이름·설명·인자 명세·접근 권한·위험도 같은 정보가 함께 적힙니다. 카탈로그의 또 다른 효과는 도구 설명문의 품질을 한 곳에서 관리하게 만드는 것입니다. 1.2.1에서 본 tool selection error의 상당 부분이 도구 설명문이 모호한 데서 나온다는 점을 떠올리면, 카탈로그가 곧 평가의 출발점이기도 합니다.

두 번째 항목인 **가드레일**은 카탈로그에 등록된 도구 호출이 실제로 떨어지기 전에 한 번 더 점검을 거치는 층입니다. 5.3.3에서 본 redaction, PII 정책, audit log 같은 항목이 여기 들어옵니다. 가드레일은 입력 가드레일과 출력 가드레일로 나뉩니다. 입력 가드레일은 모델이 만든 호출 인자를 검사해, 개인정보가 외부 도구로 새 나가지 않게 막거나, 인자의 형식이 맞지 않으면 막습니다. 출력 가드레일은 도구가 돌려준 결과를 검사해, 민감한 정보가 그대로 응답에 실리지 않도록 가립니다. 이 두 줄의 검사가 가드레일의 기본 골격입니다.

세 번째 항목은 **HITL 승인 워크플로**입니다. 4.4에서 다룬 모든 걸 승인하는 모델의 한계와 임계치 기반 예외 검토가 이 자리에서 다시 만납니다. Palantir 표준은 HITL을 도구 단위로 정의합니다. 카탈로그의 각 도구 줄에 'HITL 필요'라는 칸이 있고, 그 칸이 켜진 도구는 호출 직전에 사람의 승인 화면을 거칩니다. 모든 도구가 HITL을 필요로 하지는 않습니다. 자료를 읽기만 하는 도구나 비용·위험이 낮은 도구는 자동으로 흐르고, 외부에 결제를 보내거나 데이터를 지우는 도구는 사람의 손을 한 번 거치게 됩니다. 4.1의 tiered autonomy 단계 분류가 이 칸을 채우는 데 그대로 쓰입니다. read-only 도구는 자동, suggest 도구는 사용자 검토, execute 도구는 임계치 기반 HITL, autonomous 도구는 사후 감사 식으로 나뉩니다.

네 번째 항목인 **통제된 액션 실행 환경**은 도구가 실제로 도는 자리를 가리킵니다. 도구를 부르는 결정이 내려지더라도, 그 도구가 어디에서 도는지가 곧 안전성의 마지막 줄입니다. 위험한 도구일수록 격리된 컨테이너 안에서, 정해진 자원 한도 안에서, 명시된 외부 연결만 허용된 채 도는 환경을 마련합니다. 5.5.2의 request 단위 isolation이 runtime 전체에 걸친 격리였다면, 이 자리는 도구 호출 단위의 격리입니다. 한 도구가 무한 루프를 돌더라도 컨테이너 안에서 멈추고, 한 도구가 의도하지 않은 외부 주소로 연결을 시도하더라도 환경의 네트워크 정책이 막습니다.

다섯 번째 항목은 **도구별 권한 매트릭스**입니다. 카탈로그의 각 도구 줄에 누구·무엇이 그 도구를 부를 수 있는지를 정해 둔 표입니다. 5.5.3의 RBAC와 같은 줄기 위에 있지만, 메모리가 아닌 행동 쪽에 적용됩니다. 같은 에이전트가 같은 사용자 토큰으로 도는 호출이라도, 사용자의 역할에 따라 부를 수 있는 도구의 목록이 다르게 잘립니다. 영업 사원의 토큰으로 도는 에이전트는 거래처 조회 도구만 보이고, 영업 관리자의 토큰으로 도는 에이전트는 거래처 등록·수정 도구까지 보입니다. 한 사용자가 토큰을 손에 넣은 다른 사람에게 권한을 빌려주는 일도 이 표 위에서 막힙니다.

전체 흐름을 한 그림으로 묶으면 다음과 같습니다.

```
 [에이전트 결정] -- 도구 호출 요청 -->
                                       [Tool 게이트]
                                        |  카탈로그 확인
                                        |  토큰 + 권한 매트릭스 확인
                                        |  입력 가드레일 검사
                                        v
                                  HITL 필요?
                                  /         \
                                예           아니오
                                |              |
                                v              v
                          [승인 대기]      [통제된 환경에서 실행]
                                |              |
                                v              v
                          [사람 승인]      [출력 가드레일 검사]
                                \              /
                                 \            /
                                  v          v
                                  [감사 로그]
                                       |
                                       v
                                  [에이전트로 결과 반환]
```

이 그림에서 모든 화살표는 한 곳에 모입니다. 감사 로그입니다. 어떤 도구가 누구의 토큰으로 어떤 인자와 함께 불렸고, 가드레일에서 어떻게 다듬어졌고, HITL이 있었다면 누가 언제 승인했고, 결과가 어떻게 돌아왔는지가 한 줄씩 남습니다. 이 로그가 마지막 통제 항목인 **회수**의 토대입니다. 도구 호출이 사후에 잘못된 것으로 밝혀졌을 때, 로그를 따라 그 호출의 영향을 되돌리거나, 같은 패턴의 후속 호출을 일괄 막거나, 권한을 부여한 자리까지 거슬러 올라가 점검할 수 있습니다.

한국 규제 환경에서 이 차원이 어떻게 적용되는지 살펴보면, 금융권은 외부 송금·계좌 조회·고객 정보 변경 같은 도구가 모두 HITL을 거치도록 카탈로그가 짜여 있고, 도구 호출 로그가 일정 기간 보관되어 감독 당국의 점검에 응할 수 있어야 합니다. 공공 분야는 행정 정보를 바꾸는 도구가 직급·부서별로 잘리고, 호출 로그가 누가 언제 무엇을 바꿨는지를 추적할 수 있는 형태로 남습니다. 일반 기업이라도 이메일 자동 발송이나 결제 처리 같은 비가역 도구는 한국의 전자상거래·전자서명 관련 규제와 맞물리므로, 카탈로그의 HITL 칸을 어떻게 채우는지가 곧 규제 대응 그 자체가 됩니다.

여기서 4.4.1에서 다룬 승인 피로의 함정을 한 번 더 떠올릴 필요가 있습니다. HITL 칸을 너무 넉넉하게 켜 두면 사용자는 매번 승인 버튼을 누르는 일에 지쳐 자동 승인 습관이 들고, 그 순간 HITL은 형식만 남은 통과의례가 됩니다. Tool Governance의 운영은 카탈로그의 HITL 칸을 만드는 일에서 끝나지 않고, 그 칸이 실제로 의미 있는 차단을 만들어 내는지를 측정하는 일까지 포함합니다. HITL 트리거 비율, 승인까지 걸리는 시간, 거부된 호출의 비율 같은 메트릭이 이 자리에서 함께 운영됩니다.

본인의 시스템을 self-audit해 볼 질문 세 가지를 두겠습니다. 첫째, 본인이 쓰는 도구를 한 표로 그렸을 때, 각 도구에 위험도·HITL 여부·접근 권한이 함께 적혀 있습니까. 둘째, 가장 위험한 도구 한 개를 골라, 그 도구의 호출이 마지막 한 시간 동안 누가 어떤 인자로 불렀는지를 한 번의 조회로 답할 수 있습니까. 셋째, 한 도구가 사후에 잘못 쓰였다는 사실이 밝혀졌을 때, 같은 패턴의 후속 호출을 즉시 막고 영향을 되돌리는 절차가 정해져 있습니까. 세 질문 가운데 어느 하나라도 답이 흐릿하면, 도구가 행동으로 옮겨지는 자리에 통제가 비어 있다는 뜻입니다.

정리하면, Tool Governance 차원은 에이전트가 외부 세계에 영향을 주는 도구 호출을 도구 카탈로그·가드레일·HITL 승인·통제된 실행 환경·도구별 권한 매트릭스·감사 로그와 회수의 여섯 자리에서 통제하는 층입니다. 4.1의 tiered autonomy가 자율성 단계를 분류한 자리였다면, 이 차원은 그 분류를 카탈로그의 칸으로 옮겨 모든 도구 호출 위에서 일관되게 작동하도록 만든 자리입니다. 한국 규제 환경의 금융권·공공·일반 기업 모두에서, 이 층의 두께가 곧 비가역 행동에 대한 신뢰의 두께가 됩니다.

다음 단원인 5.5.5에서는 지금까지 살펴본 네 차원 위에서 일어난 모든 일을 어떻게 보고 어떻게 검증하는지를 다루겠습니다. '비결정적 에이전트에서는 observability 없이는 reliability가 없다'는 한 줄이 그 자리의 출발점입니다.

이 단원을 마치면 Tool Governance 차원이 정의하는 도구 카탈로그·가드레일·HITL 승인·통제된 액션 실행·도구별 권한 매트릭스·감사 로그와 회수의 통제 항목을 4.1의 tiered autonomy와 4.4의 HITL 위에서 다시 묶어 설명하고, 한국 규제 환경의 비가역 행동에 대한 통제가 카탈로그의 칸 한 줄에 어떻게 압축되는지를 본인의 시스템에 옮겨 self-audit할 수 있습니다.
