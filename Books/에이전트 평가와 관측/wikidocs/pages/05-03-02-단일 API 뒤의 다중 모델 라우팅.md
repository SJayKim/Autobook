# 5.3.2 단일 API 뒤의 다중 모델 라우팅

앞 단원에서는 JPMorgan LLM Suite가 회사 안의 모든 모델 사용을 한 자리로 모은 중앙 플랫폼이라는 점을 살펴봤습니다. 그 한 자리의 입구가 어떻게 생겼는지가 이번 단원의 주제입니다. 사내 응용 프로그램이 LLM Suite를 호출할 때 마주하는 입구는 단 하나의 API이지만, 그 뒤편에는 OpenAI, Anthropic 같은 여러 사업자의 여러 모델이 줄지어 서 있습니다. 사례 자료의 표현을 그대로 옮기면 다음과 같습니다. 단일 API 뒤에서 OpenAI와 Anthropic 모델 선택이 가능하고, task type, cost, accuracy 기준으로 라우팅된다는 것입니다. 이 한 줄을 풀어 단일 API의 추상화가 만들어 내는 효과, 라우팅 결정의 축, 클라이언트 영향 차단의 의미, 관측·과금이 한 자리에 모이는 이유를 차례로 살펴보겠습니다.

먼저 **단일 API**라는 표현부터 풀겠습니다. API는 한 시스템이 다른 시스템에게 자신의 기능을 어떻게 부르라고 약속해 둔 창구입니다. 사내 응용 프로그램이 모델을 쓰려면 어딘가의 API를 호출해야 하는데, 보통은 사업자별로 모양이 다릅니다. 한 사업자의 호출 형식은 메시지 배열을 넘기고, 다른 사업자의 호출 형식은 시스템 프롬프트와 사용자 프롬프트를 분리해 받고, 또 다른 사업자는 도구 정의를 다른 키 이름으로 받습니다. 응용 프로그램이 직접 사업자의 API를 부르면, 사업자가 늘어날수록 같은 일을 여러 번 짜야 하고, 사업자가 형식을 바꾸면 사내 모든 코드가 함께 바뀌어야 합니다. 단일 API는 이 차이를 가운데에서 한 번 흡수하고, 사내 응용 프로그램에게는 단 하나의 호출 형식만 노출합니다.

이 추상화의 효과는 비유로 잡으면 분명합니다. 사내 응용 프로그램은 사업자가 누구든 LLM Suite의 한 입구만 알고 있으면 됩니다. 새 사업자가 늘어나든, 한 사업자의 호출 형식이 바뀌든, 새 모델이 나오든, 그 변화는 LLM Suite 가운데에서 흡수되고 응용 프로그램의 코드는 그대로 돌아갑니다. 이것이 사례 자료가 말하는 모델 변경 시 클라이언트 영향 차단의 실체입니다. 25만 직원이 쓰는 사내 응용 프로그램이 수천 개라면, 한 줄의 API 시그니처 변경이 수천 곳에 번지는 사고가 가운데 한 자리에서 막힙니다.

```
                 [ 사내 응용 프로그램 (수천 개) ]
                              |
                              v
                  [ 단일 API : 한 가지 호출 형식 ]
                              |
              ----------------+----------------
              |                |                |
              v                v                v
     [ OpenAI 모델 A ]  [ Anthropic 모델 B ]  [ 모델 C ]
        (큰 모델)         (긴 컨텍스트)       (작은 모델)
```

이 그림에서 위쪽의 응용 프로그램이 LLM Suite의 한 입구만 호출하고 있다는 점이 핵심입니다. 응용 프로그램은 자기 요청이 OpenAI로 가는지 Anthropic으로 가는지를 모르고, 알 필요도 없습니다. 그 결정은 가운데 LLM Suite가 내립니다.

이 결정이 어떻게 내려지는지가 다음 질문입니다. 사례 자료는 task type, cost, accuracy 세 축을 들고 있습니다. 4.3.2에서 모델 라우팅 전략을 다룬 적이 있으니 같은 표현이 다시 나타나는 이유는 분명합니다. JPMorgan의 LLM Suite는 4.3.2에서 일반화한 라우팅 패턴을 회사 규모에서 실제로 운영하는 사례입니다. 같은 그림을 다시 들여다보되, 이번에는 회사 전체에 적용된다는 차이를 염두에 두고 보겠습니다.

첫째 축인 **task type**부터 살피겠습니다. 사내에서 들어오는 요청은 그 종류가 무척 다양합니다. 어떤 요청은 짧은 메모를 다듬어 달라는 단순 글쓰기이고, 어떤 요청은 수십 페이지짜리 보고서에서 특정 사건의 흐름을 뽑아 달라는 긴 문서 분석이며, 어떤 요청은 제안서를 평가해 달라는 도메인 추론입니다. 모델마다 잘하는 영역이 다르기 때문에 같은 종류의 요청은 같은 모델로 보내는 편이 좋습니다. 단일 API 뒤에는 작은 task type 분류기가 있어, 들어온 요청이 어느 부류인지를 짧은 라벨로 떨어뜨리고 그 라벨을 라우팅 결정의 첫 키로 씁니다.

둘째 축인 **cost**는 한 번의 호출에 들어가는 토큰 단가입니다. 같은 task type 안에서도 사용 가능한 모델이 여러 개라면, 비용이 낮은 모델로도 충분한 요청과 비싼 모델이 필요한 요청을 갈라야 합니다. 사내 25만 명이 매일 모델을 호출하는 환경에서 단가 차이는 작은 숫자가 아닙니다. 한 요청이 10원이냐 100원이냐의 차이가 매월 수억 원의 차이로 누적됩니다. 단순한 일상 질문은 작은 모델로 보내고, 복잡한 추론이나 사실 정확도가 중요한 요청만 큰 모델로 보내는 결정이 두 번째 분기에서 일어납니다.

셋째 축인 **accuracy**는 같은 요청에 대해 모델이 내놓는 답의 정확도입니다. 정확도라는 말 자체가 추상적이라 실제로는 task type별로 따로 측정합니다. 코드 생성이라면 테스트 통과율을, 문서 추출이라면 항목별 정확도를, 사실 질문이라면 정답 일치율을 미리 측정해 두고 그 숫자를 라우팅 결정에 씁니다. 비용이 낮아도 정확도가 떨어지는 모델은 고위험 요청에 쓰지 않으며, 비용이 높아도 정확도가 충분히 높은 모델은 비싼 값을 들일 가치가 있는 요청에 우선 배정됩니다. cost와 accuracy를 함께 보는 결정이라는 점에서, 라우팅 표는 흔히 세로축에 task type, 가로축에 위험 수준을 두고 칸마다 모델 이름을 쓴 형태가 됩니다.

```
                  저위험 요청            고위험 요청
   +------------+----------------------+----------------------+
   | 코드 생성  | 작은 모델 (싼 사업자) | 큰 모델 (정확도 우선) |
   +------------+----------------------+----------------------+
   | 문서 분석  | 긴 컨텍스트 모델     | 큰 모델 + 검증 단계   |
   +------------+----------------------+----------------------+
   | 사실 질문  | 작은 모델            | 큰 모델 + 출처 인용   |
   +------------+----------------------+----------------------+
```

이 표에 적힌 모델 이름은 시간이 지나면 자주 바뀝니다. 사업자가 새 모델을 내놓고, 단가가 변하고, 사내에서 측정한 정확도가 갱신되면 표 한 칸이 다른 모델 이름으로 바뀝니다. 사내 응용 프로그램은 이 표가 바뀌는 것을 모릅니다. 단일 API의 입구 형식이 그대로이기 때문입니다. 사례 자료가 강조한 모델 변경 시 클라이언트 영향 차단이 이 자리에서 가장 분명하게 드러납니다.

단일 API의 또 다른 핵심 효과는 **관측과 과금의 통합**입니다. 사내 응용 프로그램이 사업자별 API를 직접 부르던 시절을 생각하면, 사업자별 청구서가 부서별로 따로 도착하고, 사업자별 호출 로그가 부서별 도구에 따로 쌓이며, 사고가 났을 때 어디서 어떤 호출이 일어났는지 합쳐 보기가 어렵습니다. 단일 API 뒤로 호출이 모이면, 모든 호출의 메타데이터가 한 자리에 쌓입니다. 어떤 부서의 어떤 응용 프로그램이 어느 사업자의 어느 모델을 어떤 task type 분류 결과로 호출했는지가 같은 트레이스 형식으로 남습니다.

3.6에서 본 OpenTelemetry GenAI semantic conventions가 여기에서 빛을 발합니다. 사업자 이름은 gen_ai.system, 모델 이름은 gen_ai.request.model 같은 표준 키로 트레이스에 박혀 있으면, 사업자가 바뀌어도 같은 키로 같은 의미를 읽어 낼 수 있습니다. 사내 분석 화면 한 장에서 회사 전체가 어느 사업자에게 얼마를 쓰고 있는지, 어떤 task type이 가장 비용을 많이 쓰는지, 사고가 난 시점에 어떤 모델이 라우팅됐는지가 모두 한 번의 쿼리로 보입니다.

과금에 대해서도 단일 API의 가치가 분명합니다. 사업자별 청구서가 따로 오는 대신 LLM Suite의 사용량 표가 한 장으로 만들어지고, 부서별 비용 배분도 그 표 위에서 일어납니다. 한 요청이 OpenAI로 갔든 Anthropic으로 갔든 회사 입장에서는 LLM Suite의 한 콜이고, 사업자별 단가는 LLM Suite 안의 변환 표를 통해 사내 가격으로 환산됩니다. 부서가 어느 사업자를 골랐는지 신경 쓸 필요 없이 LLM Suite의 콜 수만 보면 되고, 사업자 단가 변동은 LLM Suite의 변환 표에서 한 번 갱신하면 됩니다.

이 모든 효과를 한 줄로 묶어 두면 단일 API 추상화의 가치는 이렇게 정리됩니다. 클라이언트 쪽에서는 모델 사업자가 누구인지를 모르도록 막아 변경 영향을 차단하고, 가운데에서는 task type·cost·accuracy 세 축으로 매 요청마다 가장 알맞은 모델을 고르며, 사후에는 모든 호출을 한 형식의 트레이스와 한 장의 과금 표로 모읍니다. 25만 직원의 회사에서 이 세 가지가 동시에 가능해야 비로소 사내 어디서나 안전하게 모델을 쓸 수 있다는 것이 LLM Suite가 보여 주는 사실입니다.

정리하면, JPMorgan LLM Suite의 단일 API 뒤에는 OpenAI와 Anthropic을 비롯한 여러 사업자의 모델이 task type, cost, accuracy를 축으로 라우팅됩니다. 단일 API 추상화는 사업자별 호출 형식의 차이를 가운데에서 흡수해 사내 응용 프로그램이 사업자 누구인지를 모르도록 만들어 모델 변경 시 클라이언트 영향을 차단하고, 모든 호출이 한 자리를 거치기 때문에 관측과 과금이 한 형식의 트레이스와 한 장의 표로 통합됩니다. 4.3.2에서 본 라우팅 전략의 일반 패턴이 회사 규모에서 어떻게 운영되는지를 보여 주는 사례입니다.

다음 단원인 5.3.3에서는 이 중앙 플랫폼과 단일 API가 갖춰진 위에서 사내에 모델을 도입할 때 어디서부터 시작해야 하는지를 다룹니다. 측정 가능한 문서 중심 프로세스에서 출발해 데이터 정비, 가드레일 정의, 고위험 태스크의 사람 검토를 거쳐 안정성 확보 후 사후 검사로 이동하는 실전 권고를 살펴보겠습니다.

이 단원을 마치면 JPMorgan이 단일 API 뒤에서 OpenAI·Anthropic 모델을 task type·cost·accuracy로 라우팅하는 구조를 설명할 수 있고, 단일 API 추상화가 모델 변경 시 클라이언트 영향을 어떻게 차단하며 관측과 과금을 한 자리에 어떻게 모으는지를 4.3.2의 라우팅 전략과 연결해 설명할 수 있습니다.
