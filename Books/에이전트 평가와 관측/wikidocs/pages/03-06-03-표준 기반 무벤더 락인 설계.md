# 3.6.3 표준 기반 무벤더 락인 설계

3.6.1에서 OpenTelemetry라는 그릇을, 3.6.2에서 그 그릇 안의 이름 약속을 살펴보았습니다. 이 단원은 이 두 가지를 묶어 실제 시스템을 어떻게 짤지에 대한 답입니다. 한국 기업이 LangSmith의 손쉬움, Langfuse의 자체 호스팅, 사내 ClickHouse 같은 자체 분석 백엔드를 동시에 원할 때, 이 셋을 한 코드 위에서 충돌 없이 운영하는 길을 정리합니다. 출발점은 단순합니다. 가운데에 OTel collector를 둡니다.

먼저 **OTel collector pipeline**의 구조부터 풀겠습니다. collector는 별도 프로세스로 띄우는 작은 서버로, 안에서 하는 일이 셋입니다. 첫째 receiver, 둘째 processor, 셋째 exporter입니다. receiver는 외부에서 들어오는 데이터를 받는 입구로, OTLP를 받는 receiver가 가장 흔합니다. processor는 받은 데이터를 변형하거나 거르는 단계로, batching, sampling, redaction, attribute 변환 같은 일이 여기서 일어납니다. exporter는 처리된 데이터를 외부 백엔드로 내보내는 출구로, 백엔드 종류만큼 종류가 있습니다. 이 셋이 모여 한 줄을 이루는 것을 pipeline이라 부르고, collector 한 대 안에 여러 pipeline을 동시에 돌릴 수 있습니다.

이 구조를 그림으로 보면 다음과 같습니다.

```
 [에이전트 + OTel SDK]
        |  OTLP/gRPC
        v
  +---------------------------------------------------+
  |                  OTel Collector                   |
  |                                                   |
  |  receiver   ->  processor       ->  exporter      |
  |  (otlp)         (batch,             (otlp_http,   |
  |                  sampler,            clickhouse,  |
  |                  redactor,           langsmith,   |
  |                  attributes)         langfuse)    |
  +---------------------------------------------------+
```

receiver는 한 곳, exporter는 여러 곳이라는 점이 가장 중요합니다. 이 비대칭 위에서 **다중 백엔드 동시 export**가 자연스럽게 풀립니다.

흐름을 한 단계 더 펼쳐 보면 이렇게 그릴 수 있습니다.

```
            +-----> exporter A --> LangSmith         (실험·데이터셋)
            |
 collector -+-----> exporter B --> Langfuse          (자체 호스팅 trace)
            |
            +-----> exporter C --> ClickHouse        (사내 장기 분석)
            |
            +-----> exporter D --> Datadog / Tempo   (인프라 관측)
```

같은 trace 한 건이 collector를 거치면서 네 곳으로 동시에 흘러갑니다. 애플리케이션 코드는 OTLP 한 곳만 알고, 분기 결정은 collector 설정 파일에서 일어납니다. 이 분기는 똑같은 데이터를 그대로 복제할 수도 있고, pipeline마다 별도의 processor를 끼워 다른 가공을 거친 사본을 보낼 수도 있습니다. 예를 들어 LangSmith 쪽으로는 prompt 본문을 그대로 보내고, ClickHouse 쪽으로는 PII redaction을 거친 사본만 보내는 구성이 가능합니다.

한 collector 설정의 골격을 YAML로 보여 드리면 다음과 같습니다.

```yaml
receivers:
  otlp:
    protocols:
      grpc:
processors:
  batch: {}
  tail_sampling:
    policies:
      - name: errors
        type: status_code
        status_code: { status_codes: [ERROR] }
      - name: sample-10pct
        type: probabilistic
        probabilistic: { sampling_percentage: 10 }
  attributes/redact:
    actions:
      - key: gen_ai.prompt.0.content
        action: hash
exporters:
  otlphttp/langfuse:
    endpoint: https://langfuse.internal/api/public/otel
  otlphttp/langsmith:
    endpoint: https://api.smith.langchain.com/otel
  clickhouse:
    endpoint: tcp://clickhouse.internal:9000
service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, tail_sampling, attributes/redact]
      exporters: [otlphttp/langfuse, otlphttp/langsmith, clickhouse]
```

이 한 파일로 세 백엔드를 동시에 운영하면서, 가운데에서 표본을 뽑고 prompt 본문을 가립니다. 코드는 그대로입니다.

여기서 등장한 두 단어를 풀어 두겠습니다. **Sampling**은 모든 trace를 다 저장하지 않고 일부만 골라 저장하는 정책입니다. 모든 호출을 다 기록하면 비용과 저장 부담이 빠르게 커지기 때문에, 일정 비율만 무작위로 뽑거나(probabilistic) 에러가 일어난 trace는 모두 뽑고 정상 trace는 일부만 뽑는 방식(tail sampling)이 흔히 쓰입니다. tail sampling은 trace가 끝난 뒤 결과를 보고 결정한다는 뜻이고, 결과 기반으로 흥미로운 trace를 선별하기에 LLM 관측에 잘 맞습니다. **PII redaction**은 trace 안에 들어 있는 개인 식별 정보, 주민등록번호, 전화번호, 이메일 같은 항목을 가리거나 해시하는 처리입니다. collector의 attributes processor에서 정규식이나 키 단위 규칙으로 거는 방식이 일반적입니다. 한국 환경이라면 개인정보 보호법 적용 범위에 들어가는 항목을 사전 정의해 두고, 모든 exporter 앞에서 한 번 redaction을 거는 구성이 안전합니다.

이 구조가 만들어 내는 가장 큰 효과가 **벤더 교체 시 영향 범위의 축소**입니다. 영향 범위라는 표현이 추상적으로 들릴 수 있어 한 번 풀어 두겠습니다. 어떤 도구를 다른 도구로 바꿀 때 손대야 할 코드와 설정의 양을 가리킵니다. 도구 전용 SDK로 광범위하게 계측해 둔 경우, 영향 범위는 SDK 호출이 적힌 모든 코드 파일입니다. OTel SDK로 계측해 둔 경우, 영향 범위는 collector의 exporter 설정 한 줄, 또는 새 exporter를 추가하고 옛 것을 빼는 변경 한 건입니다. 이 차이는 평소에는 잘 보이지 않다가 도구 가격 정책이 바뀌거나 데이터 거주성 요건이 새로 들어올 때 결정적으로 드러납니다.

이 효과를 한국 기업의 흔한 시나리오에 비추어 보면 그림이 더 분명해집니다. 처음에는 LangSmith로 빠르게 출발해 실험과 데이터셋 관리를 익히고, 사내 보안 검토가 통과되면 prompt 본문을 담은 사본은 Langfuse 자체 호스팅 인스턴스로 옮기며, 장기 분석은 사내 ClickHouse에 적재합니다. 이 세 단계를 거치는 동안 애플리케이션 코드는 거의 손대지 않고 collector 설정만 단계적으로 늘려 갑니다. 어느 단계에서 LangSmith를 빼더라도 exporter 한 줄을 지우는 일이지, 코드 전반의 SDK 호출을 갈아 끼우는 일이 아닙니다.

마지막으로 **표준 채택 로드맵**을 정리해 두겠습니다. 한 번에 모든 코드를 OTel로 옮기는 일은 비현실적인 경우가 많습니다. 다음과 같은 4단계 그림이 한국 기업 환경에서 자주 권장됩니다.

```
 단계 1: 새 코드는 OTel SDK 위에 계측한다.
         기존 코드는 그대로 둔다. collector를 띄우고 한 백엔드로만 export.
 단계 2: 기존 도구 전용 SDK로 계측된 코드도 OTLP exporter를 부착한다.
         두 SDK가 같은 collector로 모이도록 만든다.
 단계 3: collector에 PII redaction과 sampling을 도입한다.
         핵심 백엔드 외에 ClickHouse 같은 장기 저장소를 추가한다.
 단계 4: 도구 전용 SDK 호출을 점진적으로 OTel SDK로 바꾼다.
         이 과정이 끝나면 백엔드 교체가 collector 설정 변경으로 좁혀진다.
```

이 로드맵은 출발할 때 모든 결정을 미리 하지 않아도 된다는 점이 핵심입니다. 표준 위에 한 발만 올려 두면 이후 단계는 운영을 익히면서 한 단계씩 옮겨 가도 됩니다.

표준 채택의 한계도 한 번 짚어 두는 편이 안전합니다. 3.6.2에서 본 것처럼 GenAI semantic conventions는 일부 영역이 작업 중입니다. 도구별 확장 속성에 의존하는 부분이 한동안 남고, 그 부분은 표준 이름으로 옮길 때 한 번씩 매핑 작업이 따라옵니다. 또한 collector 운영 자체에 부하 분산과 장애 복구 같은 자체 운영 부담이 새로 생깁니다. 이 부담은 보통 인프라 팀의 기존 표준 관측 운영에 합쳐져 흡수되지만, 처음 collector를 도입할 때는 별도 작업으로 잡아 두어야 합니다.

정리하면, OTel collector를 가운데에 둔 표준 기반 설계는 receiver·processor·exporter pipeline 위에서 한 번 들어온 trace를 LangSmith·Langfuse·사내 ClickHouse 같은 여러 백엔드로 동시에 분기해 보내고, 그 사이에 sampling과 PII redaction을 한 곳에서 거는 구조입니다. 이 구조가 만들어 내는 가장 큰 효과는 벤더 교체 시 영향 범위가 코드 전반이 아니라 collector 설정 한 곳으로 좁아진다는 점이며, 한국 기업이 단계적인 표준 채택 로드맵을 따라 새 코드부터 점진적으로 옮겨 갈 수 있는 길을 열어 줍니다.

지금까지 Phase 3에서는 online 평가의 메트릭 설계, 회귀 탐지, LLM observability의 데이터 모델, LangSmith와 Langfuse의 깊은 비교, 그리고 OpenTelemetry 표준 위에서 도구를 묶는 무벤더 락인 설계까지 한 줄로 이어 살펴보았습니다. 평가와 관측의 토대가 갖추어진 셈입니다. 다음 단원인 4.1.1부터 시작되는 Phase 4에서는 이 토대 위에 신뢰성과 안전을 얹는 일을 다룹니다. 첫 단원은 에이전트의 자율성을 단계별로 분류하고 각 단계의 권한 범위를 정하는 tiered autonomy의 출발점이 될 것입니다.

이 단원을 마치면 OTel collector pipeline의 receiver·processor·exporter 구조, LangSmith·Langfuse·자체 ClickHouse를 동시에 export하는 다중 백엔드 구성, sampling과 PII redaction을 collector에서 처리하는 패턴, 표준 위에 둔 코드가 벤더 교체 시 영향 범위를 어떻게 좁혀 주는지, 그리고 한국 기업 환경에서 단계적으로 OTel을 채택해 가는 로드맵을 한 흐름으로 설명할 수 있습니다.
