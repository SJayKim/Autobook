# 4.3.3 Graceful Degradation

앞의 두 단원에서 우리는 실패한 경로 대신 다른 경로로 흘려보내는 fallback과, 매 요청마다 적합한 모델을 골라 보내는 라우팅을 살펴봤습니다. 그런데 이 두 장치가 모두 작동하지 못하는 상황도 있습니다. 모든 모델 사업자가 동시에 흔들리거나, 사내 도구 전체가 같은 데이터베이스 장애에 영향을 받거나, 외부 검색 API가 한꺼번에 응답을 멈추는 식입니다. 이런 상황에서 시스템이 곧장 모든 응답을 거절하고 멈춰 버리면, 사용자가 평소에 잘 쓰던 기능까지 함께 사라집니다. 부분 장애가 전체 정지로 번지는 셈입니다. 이 번짐을 막기 위한 패턴이 **graceful degradation**입니다.

이름부터 풀겠습니다. 영어 그대로 옮기면 '품위 있게 기능을 낮춤'이라는 뜻입니다. 좀 더 풀어 쓰면 '갑작스럽게 모든 것이 꺼지는 대신, 천천히 기능 일부를 내려놓으며 핵심만 살려 두는 방식'입니다. 일상 비유로는 정전 직전에 비상등이 켜지는 장면을 떠올리면 됩니다. 형광등은 모두 꺼지지만 비상등 한 줄은 남아서 사람들이 출구를 찾을 수 있게 해 줍니다. 시스템 설계에서도 똑같은 비유가 그대로 쓰입니다. 장애가 와도 모든 화면이 일시에 멈추는 대신, 우선순위가 높은 기능만 살리고 낮은 기능을 한시적으로 멈춥니다.

이 패턴을 짜려면 가장 먼저 **기능 우선순위 분류**를 해야 합니다. 에이전트 서비스가 제공하는 기능을 모아 놓고, 각 기능에 대해 두 질문을 던집니다. 첫째, 이 기능 없이도 사용자가 핵심 작업을 끝낼 수 있는가. 둘째, 이 기능이 멈췄을 때 사용자에게 손해가 얼마나 큰가. 두 질문의 답을 보고 기능을 보통 세 등급으로 나눕니다. 핵심 기능, 일반 기능, 보조 기능입니다. 핵심 기능은 끊어지면 서비스의 의미가 사라지는 부분이고, 일반 기능은 평상시에는 가치를 더해 주지만 잠시 빠져도 사용자가 다른 길로 일을 끝낼 수 있는 부분이며, 보조 기능은 편의성에 가까운 부분입니다. 예를 들어 사내 문서 검색 에이전트라면 사내 문서 검색 자체가 핵심, 검색 결과의 자동 요약이 일반, 결과를 슬랙으로 자동 전송해 주는 기능이 보조에 해당합니다.

분류가 끝나면 **비핵심 기능 비활성화** 정책을 짭니다. 이 정책은 어떤 신호가 나타났을 때 어떤 기능을 어떤 순서로 끄는지를 미리 정의합니다. 신호로는 회로 차단기가 열린 상태, 라우팅에서 모든 후보 모델이 SLA 임계 아래로 떨어진 상태, 도구 호출의 연속 실패가 일정 횟수를 넘은 상태가 흔히 쓰입니다. 정책은 보통 단계로 표현됩니다. 1단계에서는 보조 기능만 끕니다. 2단계에서는 일반 기능까지 함께 끕니다. 3단계에서는 핵심 기능을 축약된 형태로 돌립니다. 단계가 깊어질수록 시스템이 보여 주는 화면은 단순해지지만, 동시에 살아 있는 기능의 신뢰성은 더 높아집니다.

다음으로 **사용자에게 알리는 메시지**가 필요합니다. 기능이 줄어든 사실을 사용자에게 숨기면 사용자는 시스템이 고장 났다고 느낍니다. 반면 너무 자세히 알리면 평상시에도 불안감을 줍니다. 좋은 메시지는 세 요소를 갖춥니다. 첫째, 지금 어떤 기능이 평소처럼 동작하지 않는다는 사실을 한 줄로 알립니다. 둘째, 핵심 기능은 여전히 사용 가능하다는 점을 함께 안내합니다. 셋째, 정상 복구 예상 시점이나 다음 단계 안내를 짧게 덧붙입니다. 이런 메시지는 보통 화면 상단의 띠 형태로 표시되며, 본문 영역의 결과는 그대로 유지됩니다.

기능 축소 단계와 사용자 안내를 그림으로 정리하면 다음과 같습니다.

```
[정상]                 [1단계 축소]            [2단계 축소]            [3단계 축소]
보조 기능   ON          보조 기능   OFF         보조 기능   OFF         보조 기능   OFF
일반 기능   ON          일반 기능   ON          일반 기능   OFF         일반 기능   OFF
핵심 기능   ON          핵심 기능   ON          핵심 기능   ON          핵심 기능   ON (축약)
사용자 안내 없음        '슬랙 전송 일시 중단'    '자동 요약 일시 중단'    '검색 결과만 표시 중'
```

여기서 우선순위 정의가 위쪽으로 갈수록 끄기 쉽고 아래로 갈수록 끄기 어렵다는 점이 한눈에 보입니다. 핵심 기능은 마지막까지 살려 두며, 그것마저 어려울 때는 결과 형식을 단순화하는 식으로 축약합니다.

기능을 축소했다고 끝이 아닙니다. **장애 종료 후 회복 절차**가 함께 설계되어야 합니다. 장애가 잦아들면 시스템은 다시 보조 기능을 켜고, 일반 기능을 켜고, 마지막으로 모든 기능을 정상으로 돌립니다. 이 회복 순서는 장애 진입 순서의 역방향이며, 단계 사이에는 잠시 검증 시간을 두어 신호가 진짜로 안정됐는지 확인합니다. 앞 장에서 본 회로 차단기의 half-open 상태와 같은 메커니즘입니다. 회복이 너무 성급하면 살짝 가라앉았다가 다시 들이닥치는 2차 장애에 그대로 노출되고, 너무 굼뜨면 사용자에게 불필요하게 줄어든 화면을 오래 보여 주게 됩니다. 단계마다 검증 시간을 두는 이유가 이 사이의 균형을 잡는 것입니다.

마지막 축은 **관측을 통한 장애 범위 자동 인식**입니다. 어떤 기능을 어느 단계에서 끌지를 사람이 일일이 판단하면 너무 늦습니다. 관측 시스템이 모델·도구·외부 API의 상태 신호를 실시간으로 모으고, 미리 정해 둔 규칙에 따라 자동으로 단계를 올리거나 내려야 합니다. 이때 trace와 metric에 부착되어 있는 사업자 이름, 도구 이름, 에러 분류가 핵심 입력이 됩니다. 예를 들어 'A 사업자의 호출 에러율이 5분 평균 30%를 넘으면 그 사업자를 쓰는 보조 기능을 자동으로 비활성화'와 같은 규칙입니다. 규칙을 단순하게 유지하되, 어떤 규칙이 어떤 시점에 발동됐는지를 같은 trace에 함께 남겨야 사후에 사용자 경험이 어떻게 달라졌는지 재구성할 수 있습니다.

이 패턴을 작은 회사가 처음부터 다 갖추기는 어렵습니다. 그래서 도입 순서를 한 가지 권하면, 먼저 기능을 우선순위로 분류해 종이 한 장에 정리하는 것부터 시작합니다. 그다음 가장 자주 흔들리는 한두 개 외부 의존을 골라, 그것이 흔들렸을 때 어떤 기능을 끌지를 회로 차단기 코드 옆에 붙입니다. 그리고 사용자 안내 메시지의 자리만 화면에 마련해 둡니다. 이 세 가지만 갖춰도 부분 장애가 전체 정지로 번지는 가장 흔한 사고는 막을 수 있습니다. 자동 인식은 그 위에 얹는 다음 층입니다.

정리하면, graceful degradation은 부분 장애가 전체 정지로 번지지 않도록 기능을 단계적으로 축소했다가 다시 회복시키는 패턴이며, 핵심·일반·보조로 나뉜 기능 우선순위를 바탕으로 비핵심 기능을 먼저 끄고, 사용자에게는 어떤 기능이 일시 중단됐는지를 짧게 안내하며, 장애 종료 후에는 진입 순서의 역방향으로 단계를 검증하며 회복합니다. 관측 시스템이 사업자·도구·에러 신호를 실시간으로 모아 자동으로 단계를 조정하면, 사람이 손으로 끄고 켜는 지연 없이도 부분 장애에 견디는 시스템이 만들어집니다.

이로써 4.3절 fallback routing 전체가 마무리됩니다. fallback의 종류, 모델 라우팅, graceful degradation 세 가지 패턴은 회로 차단기와 함께 비결정적 에이전트를 프로덕션에 안전하게 올리는 신뢰성 장치를 이룹니다. 다음 단원인 4.4.1에서는 이 신뢰성 장치들 옆에 함께 자리 잡는 또 한 축, 즉 사람의 개입을 어떻게 다시 정의할지를 다룹니다. 모든 행동을 사람이 일일이 승인하는 전수 승인 모델의 한계를 살피고, 왜 임계치 기반 예외 검토로 옮겨 가야 하는지를 정리합니다.

이 단원을 마치면 graceful degradation을 핵심·일반·보조 기능 우선순위 분류 위에서 비핵심 기능을 비활성화하는 단계적 축소로 설명하고, 사용자에게 보내는 안내 메시지의 구성, 장애 종료 후 회복 절차의 단계와 검증 시간, 그리고 관측 시스템이 어떻게 장애 범위를 자동으로 인식해 단계를 조정하는지를 함께 풀어낼 수 있습니다.
