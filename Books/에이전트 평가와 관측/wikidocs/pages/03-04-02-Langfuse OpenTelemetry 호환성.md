# 3.4.2 Langfuse OpenTelemetry 호환성

앞 단원에서 본 Langfuse의 자체 호스팅이 데이터를 어디에 둘 것인가에 대한 답이라면, 이 단원이 다루는 OpenTelemetry 호환성은 데이터를 어떤 형식으로 흘려 보낼 것인가에 대한 답입니다. 두 답은 함께 가야 의미가 있습니다. 자체 호스팅으로 데이터의 위치를 통제했더라도, 그 데이터를 보내는 형식이 한 회사 전용이라면 결국 다른 도구로 옮겨 가는 순간 모든 계측 코드를 다시 짜야 합니다. 이 단원은 Langfuse가 그 짐을 어떻게 덜어 주는지를 다룹니다.

먼저 등장하는 단어부터 풀겠습니다. **OpenTelemetry**, 줄여서 OTel은 분산 시스템에서 trace, metric, log를 수집하고 전송하기 위해 만들어진 업계 표준 프레임워크입니다. 클라우드 네이티브 운영의 표준화를 다루는 비영리 재단인 CNCF가 호스팅하고 있고, 분산 추적 분야의 두 표준이었던 OpenTracing과 OpenCensus가 합쳐진 결과물입니다. 핵심은 'span을 이렇게 표현하고 이렇게 보내자'는 데이터 모델과 전송 규약을 회사 단위가 아닌 표준 단위로 정해 둔다는 점입니다. 기업이 어떤 백엔드를 쓰든 같은 SDK로 같은 형식의 trace를 만들어 내면, 받는 쪽은 그 표준만 이해하면 됩니다. 자세한 표준 자체는 3.6에서 본격적으로 다루고, 이 단원에서는 Langfuse가 그 표준 위에 어떻게 자기 자리를 잡았는지에 초점을 맞춥니다.

OTel의 전송 규약 가운데 본문에서 자주 등장할 단어가 **OTLP**입니다. OpenTelemetry Protocol의 줄임말로, span과 metric을 직렬화해 어디로 보낼지 약속해 둔 네트워크 프로토콜입니다. 보내는 쪽은 OTLP 형식으로 packet을 만들고, 받는 쪽은 OTLP를 이해하는 endpoint를 열어 두면 두 쪽이 같은 회사 출신이 아니어도 데이터가 흐릅니다. 이메일에 비유하면 SMTP에 해당하는 자리를 차지하는 규약이라고 볼 수 있습니다.

Langfuse는 이 OTLP를 그대로 받는 endpoint를 자체 서버에 열어 둡니다. 즉 Langfuse는 자기 전용 SDK뿐 아니라 OTel 표준 송신자에게도 항구를 내어 주는 수신기 역할을 합니다. 흐름을 그림으로 보면 다음과 같습니다.

```
   [에이전트 앱]
        |
   +----+--------------------------+
   |                               |
   v                               v
 Langfuse SDK (직통)        OTel SDK (Python/JS/...)
        |                               |
        |                               | OTLP/HTTP, OTLP/gRPC
        v                               v
 +-----------------------------------------+
 |   Langfuse Server (OTLP endpoint 포함)  |
 +-----------------------------------------+
        |
        +--> Postgres / ClickHouse
        |
        +--> (옵션) 다른 OTel 백엔드로 동시 export
```

이 그림이 보여 주는 첫 번째 사실은 SDK 선택권이 늘어났다는 점입니다. Langfuse 전용 SDK를 그대로 써도 되고, OTel SDK를 그대로 써도 같은 서버로 같은 trace가 들어옵니다. 이것이 **다중 SDK 지원**의 의미입니다. Python 진영에는 OpenAI나 Anthropic SDK 호출을 자동으로 계측해 주는 OTel 기반 라이브러리가 여럿 있고, JavaScript와 Go 진영도 마찬가지입니다. 팀이 이미 다른 언어와 다른 라이브러리로 OTel 계측을 해 두었다면, Langfuse는 그 흐름을 받는 항구를 새로 열어 줄 뿐 기존 계측 코드를 다시 쓰게 만들지 않습니다.

여기서 한 발 더 나아가면 **타 옵저버빌리티 도구와의 연동**이 가능해집니다. OTel은 같은 trace를 여러 백엔드로 동시에 보낼 수 있도록 가운데에 collector라는 부품을 두는 설계를 권장합니다. collector는 여러 SDK가 보낸 OTLP packet을 받아 다시 여러 곳으로 분기해 보내는 라우터 같은 부품입니다. 이 collector를 통해 같은 trace를 Langfuse로 보내면서 Datadog이나 Grafana Tempo, 사내 자체 trace 저장소로도 함께 보낼 수 있습니다. LLM 관측에 강한 도구와 인프라 관측에 강한 도구가 따로 있는 현실에서, 둘을 동시에 활용하면서도 계측을 한 번만 해 둘 수 있다는 뜻입니다.

이 흐름을 정리하면 다음과 같습니다.

```
              +-------------------+
   에이전트   |  OTel Collector   |     +--> Langfuse  (LLM trace 분석)
   --(OTLP)-->|  (라우터)         |-----+--> Datadog   (인프라 메트릭)
              |                   |     +--> 사내 S3   (장기 보관)
              +-------------------+
```

이런 구조가 자연스럽게 풀어 주는 문제가 바로 **벤더 락인** 회피입니다. 락인은 어떤 도구에 코드를 너무 깊게 끼워 두어 다른 도구로 옮기려면 큰 비용을 치러야 하는 상태를 가리킵니다. 관측 분야에서 락인이 만들어지는 가장 흔한 길은 도구 전용 SDK로 계측 코드를 광범위하게 박아 두는 일입니다. 이렇게 하면 도구를 바꾸려는 순간 모든 코드를 들춰 SDK 호출을 갈아 끼워야 합니다. 반면 OTel SDK로 계측을 해 둔 코드는 도구 이름을 어디에도 박아 두지 않습니다. 도구를 바꾸고 싶을 때는 collector의 export 설정 한 줄을 수정하거나 OTLP endpoint URL만 다른 서버로 돌리면 됩니다.

Langfuse가 OTel 표준을 받아들인다는 사실은 비교 관점에서 한 번 더 의미가 분명해집니다. 3.3에서 다룬 LangSmith는 LangChain·LangGraph와 자동 인스트루먼트로 깊게 묶이는 호스티드 도구로, 그 안에서 가장 매끄럽게 동작하지만 trace 데이터를 외부 표준 형식으로 빼내거나 다른 백엔드로 분기하는 일은 상대적으로 부수적입니다. Langfuse는 출발점이 다릅니다. 자체 호스팅을 일급 시민으로 두고, OTel 표준을 받아들여 다른 도구와 같이 쓸 수 있는 자리를 만드는 데 무게를 둡니다. 한국 기업이 사내에 이미 Datadog 같은 인프라 관측 도구를 운영 중이라면, OTel을 가운데에 두고 LLM 부분은 Langfuse, 인프라 부분은 기존 도구로 보내는 분기 패턴을 자연스럽게 짤 수 있습니다.

다만 OTel 호환성에는 현실적인 한계도 있습니다. OTel의 GenAI semantic conventions, 즉 LLM 호출을 표현하기 위한 속성 이름 약속은 2025년 시점까지도 일부 영역이 작업 중인 상태입니다. 그래서 모델 이름이나 prompt, completion, token usage 같은 항목은 잘 들어맞지만, agent의 multi-step trajectory를 어떻게 표현할지는 도구마다 자기만의 추가 속성으로 채우고 있는 부분이 남아 있습니다. Langfuse도 OTel 표준 속성에 더해 자체 확장 속성을 함께 받아들여 trajectory와 평가 점수를 표현하기 때문에, OTel만으로 표현할 수 있는 영역과 Langfuse 전용 SDK가 한 번에 더 풍부히 표현해 주는 영역이 공존합니다. 이 경계는 시간이 지나면서 표준 쪽으로 더 옮겨 갈 가능성이 높지만, 지금 시점에서는 두 길을 모두 이해하고 어디부터 OTel로 묶을지 정해야 합니다.

실무에서 Langfuse를 OTel과 함께 쓸 때 처음 잡는 그림은 보통 다음 세 단계로 정리됩니다. 첫째, 새 코드는 가능하면 OTel SDK로 계측해 OTLP로 보냅니다. 둘째, 받는 쪽 endpoint로 Langfuse 서버를 둡니다. 셋째, 인프라 관측이나 다른 분석 백엔드가 필요해지면 OTel collector를 가운데에 추가해 같은 데이터를 분기해 내보냅니다. 이 그림은 도구를 바꿔야 할 때 collector 설정 한 곳만 손대도록 영향 범위를 좁혀 줍니다.

정리하면, Langfuse는 자체 호스팅으로 데이터의 위치를 사내에 묶어 둔 위에 OpenTelemetry 표준을 받아들임으로써 데이터의 형식까지 표준 위에 올려 둡니다. OTLP endpoint를 직접 노출해 다양한 언어와 다양한 SDK가 보낸 trace를 한 곳에서 받고, OTel collector를 통해 Datadog이나 사내 백엔드 같은 다른 옵저버빌리티 도구와 같은 trace를 공유할 수 있게 해 주며, 이 구조 위에서 도구를 바꾸려 할 때 코드 전반이 아니라 export 설정 한 곳만 손보면 되는 벤더 락인 회피 효과가 따라 나옵니다.

다음 단원인 3.5.1에서는 여기까지 살핀 LangSmith와 Langfuse의 특성을 모아, 한국 기업이 두 도구 가운데 무엇을 고를 때 어떤 차원을 따져야 하는지를 정리한 의사결정 프레임워크를 살펴보겠습니다.

이 단원을 마치면 OpenTelemetry 표준과 OTLP endpoint의 의미, Langfuse가 다중 언어 SDK와 OTel 기반 SDK를 함께 받아들이는 방식, OTel collector로 타 옵저버빌리티 도구와 동시에 연동하는 패턴, 그리고 이 구조가 만들어 주는 벤더 락인 회피의 효과를 LangSmith와 비교해 설명할 수 있습니다.
