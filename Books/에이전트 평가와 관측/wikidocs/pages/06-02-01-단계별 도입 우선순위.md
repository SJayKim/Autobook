# 6.2.1 단계별 도입 우선순위

자가 진단을 마치고 부족한 차원과 공백을 식별했다고 해서, 그 모든 것을 한 번에 채울 수 있는 팀은 없습니다. 사람도 시간도 예산도 한정되어 있고, 무엇보다 도입 작업 자체가 운영 중인 시스템 위에서 진행되어야 하기 때문에 모든 보강을 동시에 시작하면 그 어떤 보강도 끝나지 않습니다. 이 단원은 평가, 관측, 거버넌스 세 영역 가운데 어디부터 손을 댈지를 정하는 의사결정 트리를 설계합니다.

가장 먼저 부딪히는 갈림길은 **관측을 먼저 깔지, 평가를 먼저 만들지**입니다. 두 선택지는 같은 목표를 향하지만 서로 다른 가정을 깔고 있습니다. 관측을 먼저 깔면 실제 트래픽에서 무엇이 일어나는지 보이기 시작합니다. 어떤 도구가 자주 실패하는지, 어디서 토큰이 폭주하는지, 어떤 사용자 패턴이 빙빙 도는 루프를 만드는지가 trace 위에 드러납니다. 반면 평가를 먼저 만들면 회귀를 미리 막을 수 있습니다. 모델을 바꾸거나 프롬프트를 손볼 때마다 이전 성능을 깨뜨리지 않는다는 보증이 생기고, PR 머지 게이트로 활용해 잘못된 변경을 차단할 수 있습니다.

두 선택을 단순한 트레이드오프로 정리하면 다음과 같습니다.

```
[ 관측 우선 ]                            [ 평가 우선 ]
 + 실제 사용자 행동을 본다              + 회귀를 사전에 막는다
 + 미지의 실패를 발견할 수 있다         + 변경의 안전망이 생긴다
 - 수치 비교가 어렵다                   - 데이터셋이 현실과 어긋날 수 있다
 - 회귀 차단 도구가 없다                - 미지의 실패는 잡지 못한다
```

표만 보면 양쪽 다 일리 있어 보이지만, 실제로는 시스템이 어느 단계에 있느냐에 따라 답이 갈립니다. 1단계 prototype은 사용자가 거의 없고 트래픽도 들쭉날쭉하므로, 관측을 정교하게 깔아도 들여다볼 표본이 부족합니다. 이 시기에는 골든 데이터셋 한 줌과 LLM-as-judge 한두 개로 핵심 시나리오의 회귀를 막는 데 집중하는 편이 더 큰 가치를 만듭니다. 반면 2단계 초기 production은 매일 새로운 사용자가 들어오고 알 수 없는 입력이 쏟아집니다. 이때는 trace와 metric을 먼저 깔아 실제 트래픽 위에서 무엇이 일어나는지 보는 일이 우선입니다. 평가는 그 trace에서 잡힌 실패 케이스를 흡수하면서 점진적으로 키워 가면 됩니다. 3단계 스케일은 이미 두 가지가 모두 깔린 상태에서 거버넌스가 병목이 되는 시기입니다. 모델 변경, 프롬프트 변경, 도구 추가가 매주 일어나면 release train, 권한 매트릭스, audit log 같은 거버넌스 인프라가 필요해집니다.

이를 의사결정 트리로 풀면 다음 그림이 됩니다.

```
                  [ 시스템 단계 진단 ]
                          |
       +------------------+------------------+
       |                  |                  |
 [ Prototype ]    [ 초기 Production ]    [ 스케일 ]
       |                  |                  |
  사용자 < 10        사용자 ~ 수천         사용자 ~ 수만 이상
  트래픽 적음        실제 트래픽 흐름     변경 빈도 매주
       |                  |                  |
 평가 우선           관측 우선           거버넌스 우선
  · 골든셋 1개        · trace 인스트루먼트  · release train
  · LLM judge 1개    · metric dashboard   · 권한 매트릭스
  · PR 게이트         · 회귀 알람          · audit log
       |                  |                  |
       +-> 다음 단계 진입 시 다음 영역으로 확장
```

이 트리에서 한 가지 주의할 점은 단계 결정이 절대적인 것이 아니라는 사실입니다. 사용자 수가 적어도 다루는 도메인이 금융이나 의료처럼 위험이 큰 경우에는 prototype 단계에서도 거버넌스 항목을 일부 끌어와야 하고, 반대로 사용자 수가 많아도 단순 검색 보조처럼 위험이 낮은 도메인이라면 거버넌스 도입을 늦출 수 있습니다. 단계는 해상도이고, 도메인 위험은 보정 계수입니다.

조직 규모도 단계 결정에 영향을 줍니다. 5인 미만의 작은 팀이라면 평가 한 줄을 짜고 trace 한 가지를 보내는 것이 동시에 가능합니다. 한 사람이 양쪽을 머리에 담을 수 있기 때문입니다. 그러나 30인을 넘는 팀에서는 평가와 관측의 책임 분리가 일어나고, 두 영역을 동시에 깔면 서로의 작업이 어긋나 통합이 어려워집니다. 이때는 한쪽을 먼저 안정화한 다음 다른 쪽을 시작하는 방식이 합리적입니다. 250K 직원 규모의 엔터프라이즈에서는 한 술 더 떠서, 평가와 관측과 거버넌스를 모두 한 플랫폼에 모은 중앙 팀이 만들고, 사용 팀은 그 위에서 자기 도메인만 정의하는 구조가 자연스럽습니다.

도입을 결정하고 나면 곧바로 따라오는 질문이 **초기 1~2주 안에 어떤 가치를 보여 줄 것인가**입니다. 도입 작업은 길게 늘어지면 동력을 잃습니다. 첫 2주 안에 누구나 볼 수 있는 한 가지 측정 가능한 결과가 나와야 다음 분기의 예산과 인력을 확보할 수 있습니다. Prototype 단계라면 첫 2주의 산출물은 골든셋 50개와 그 위에서 돌아가는 회귀 테스트 한 개로 충분합니다. 초기 production 단계라면 trace dashboard 위에서 가장 큰 비용을 만드는 도구를 한 가지 골라내고, 그 도구의 실패율을 절반으로 줄이는 변경 하나가 첫 2주의 결과가 됩니다. 스케일 단계라면 release train 첫 사이클을 무사히 돌리는 것이 첫 산출물입니다. 어느 경우든 첫 2주 안에 숫자로 말할 수 있는 결과가 있어야 합니다.

마지막으로 비용과 ROI를 추정하는 감각이 필요합니다. 관측 도구는 SaaS 사용 시 월 수백 달러부터 시작하지만 자체 호스팅을 선택하면 인프라 운영 인력 0.2~0.5명이 추가로 들어갑니다. 평가 인프라는 LLM-as-judge 호출 비용 자체는 작지만 데이터셋 라벨링에 들어가는 사람의 시간이 가장 큰 항목입니다. 200건 라벨링에 합의 검수까지 포함하면 보통 1주가 들고, 이는 한 사람의 인건비에 직접 연동됩니다. 거버넌스는 도입 시점에는 비용으로만 보이지만, 사고가 한 번 났을 때의 복구 비용과 비교하면 ROI가 분명해집니다. 한 번의 비가역 사고는 보통 6~12개월의 거버넌스 투자를 정당화할 만큼의 손실을 만듭니다. 이런 추정을 토대로 도입 제안서를 만들면, 경영진이 결정할 수 있는 형태가 됩니다.

정리하면, 단계별 도입 우선순위는 관측 우선과 평가 우선이라는 두 갈림길에서 시작해 prototype, 초기 production, 스케일이라는 세 단계의 위치에 따라 답이 달라지며, 조직 규모와 도메인 위험으로 보정합니다. 첫 2주 안에 측정 가능한 가치를 만들고, 비용과 ROI를 정량화해 의사결정의 언어로 옮기는 것이 도입을 끝까지 끌고 가는 핵심입니다.

다음 단원인 6.2.2에서는 도입을 통해 만들어진 시스템을 어떤 KPI로 측정하고 운영할지를 다룹니다. 1.3에서 본 단일 메트릭 함정과 Klarna 사례를 다시 끌어와, 효율성과 품질과 안전성 세 축을 균형 있게 묶는 다차원 KPI 설계를 이야기합니다.

이 단원을 마치면 관측·평가·거버넌스 가운데 어디부터 시작할지를 의사결정 트리로 설명할 수 있고, prototype·초기 production·스케일의 단계 진단과 조직 규모를 결합해 도입 순서를 정할 수 있으며, 첫 1~2주의 측정 가능한 산출물과 도입 비용·ROI 추정으로 도입 제안을 정량화할 수 있습니다.
