# 4.2.2 에이전트 임계치 설계

앞 단원에서 closed·open·half-open 세 상태와 장애 임계치·회복 시간이라는 두 기준을 정리했습니다. 이 단원은 그 임계치에 실제로 어떤 숫자를 어떤 단위로 박아 둘지를 다룹니다. 임계치는 너무 빡빡하면 정상 작업이 자주 끊기고, 너무 헐거우면 폭주를 막지 못합니다. 그래서 임계치는 한 번 정해 두고 잊는 상수가 아니라, 측정과 조정을 반복하며 다듬는 운영 변수에 가깝습니다.

에이전트의 회로에 올릴 첫 번째 입력은 **스텝 수, 토큰, 비용**이라는 세 가지 누적 지표입니다. 1.2.2에서 본 planning loop의 폭주를 다시 떠올려 보겠습니다. 한 사용자 요청이 만들어 내는 단계가 늘어날수록 컨텍스트가 누적적으로 부풀고, 그에 따라 토큰과 비용도 함께 솟습니다. 회로의 입력이 외부 호출 실패율 하나뿐이라면 이런 폭주는 호출이 모두 200으로 끝나는 한 보이지 않습니다. 그래서 에이전트의 회로는 적어도 다음 네 종류의 누적치를 지켜봅니다.

```
임계치 종류    의미                                기본 보호 대상
--------------- ----------------------------------- -----------------
스텝 수         한 요청의 단계 수가 N을 넘으면 차단 루프 형태 폭주
누적 토큰       총 입력·출력 토큰이 T를 넘으면 차단 컨텍스트 비대 폭주
누적 비용       비용이 C달러를 넘으면 차단          결제·예산 폭주
경과 시간       경과가 S초를 넘으면 차단            사용자 대기 시간
```

각 임계치는 서로를 보완합니다. 스텝 수만 두면 한 단계가 거대한 컨텍스트를 받는 경우의 비용 폭주가 빠지고, 비용만 두면 짧은 단계 안에서 큰 호출이 한 번 일어나는 정상 작업이 잘못 잘립니다. 그래서 네 임계치를 동시에 걸어 두고, 어느 하나라도 닿으면 회로가 open으로 넘어가는 구조가 보통 사용됩니다.

값을 정하는 절차도 짚어 두겠습니다. 정상 작업의 분포를 한동안 측정해 95퍼센타일이나 99퍼센타일을 기준선으로 잡고, 그 위에서 잘리는 사례를 따로 모아 봅니다. 잘린 사례 가운데 정상 작업이 섞여 있다면 임계치를 약간 올리고, 폭주만 깔끔하게 잘려 있다면 그대로 둡니다. 이 과정은 한 번에 끝나지 않고, 사용자 트래픽이 변하거나 모델이 바뀔 때마다 다시 측정해 조정해야 합니다.

두 번째 입력은 **도구 실패율**입니다. 외부 API가 일시적으로 흔들리거나 사내 도구의 사양이 바뀌었을 때, 그 도구를 부르는 모든 trajectory가 차례로 실패합니다. 도구별 실패율을 시간 창 위에 두고 임계치를 넘으면 그 도구만 골라 회로를 open으로 돌립니다. 즉 회로는 사용자 요청 단위에만 걸리는 것이 아니라, 도구 단위로도 걸립니다. 이렇게 두면 망가진 도구를 모델이 반복적으로 부르는 일이 차단되고, 그 동안 다른 도구로 작업이 이어지거나 fallback 응답이 사용자에게 안내됩니다. 도구 단위 회로는 다음 단원에서 다룰 fallback routing의 출발점이 됩니다.

세 번째 입력은 **사용자별 임계치 분리**입니다. 모든 사용자에게 같은 비용 한도를 박아 두면 두 가지 문제가 동시에 생깁니다. 첫째, 정당하게 큰 작업을 자주 돌리는 엔터프라이즈 사용자는 자주 잘립니다. 둘째, 무료 평가판 사용자가 비정상적인 폭주를 일으켜도 한도가 헐거워 막지 못합니다. 그래서 임계치를 사용자 등급, 워크스페이스, 또는 API 키 단위로 분리해 둡니다. 사용자별 회로 상태를 유지하기 위해 회로 식별자에 사용자 식별자를 함께 넣는 형태가 흔합니다. 슈도코드로 보면 다음과 같습니다.

```
def call_with_breaker(user_id, action):
    key = (user_id, action.tool_name)
    state = breaker_state[key]
    if state == OPEN and now < state.reopen_at:
        return fallback_response(action)
    try:
        result = run(action)
        record_success(key)
        return result
    except Exception:
        record_failure(key)
        if failure_rate(key, window=60s) > threshold(user_id):
            open_circuit(key, cool_down=cooldown(user_id))
        raise
```

이 슈도코드는 두 가지 사실을 보여 줍니다. 회로의 키가 사용자와 도구를 함께 묶고 있다는 점, 그리고 임계치와 회복 시간이 사용자 등급에 따라 달라질 수 있다는 점입니다. 후자는 같은 도구의 같은 실패라도 사용자 등급에 따라 다르게 보호한다는 뜻입니다.

네 번째 입력은 **시간 윈도우의 모양**입니다. 실패율은 본질적으로 어떤 시간 구간 안의 비율입니다. 그래서 그 구간을 어떻게 자르느냐에 따라 같은 사건이 다른 비율로 보입니다. 가장 단순한 방법은 고정 윈도우, 즉 한 시각에서 다른 시각까지를 잘라 그 안의 호출만 보는 형태입니다. 그런데 고정 윈도우는 경계 부근에서 신호가 갑자기 사라지거나 솟는 경계 효과를 만듭니다. 예를 들어 60초 단위 윈도우라면 59초까지 누적된 실패가 60초가 되는 순간 사라지고, 새 윈도우가 0에서 다시 출발합니다. 이 효과를 줄이기 위해 **sliding window**, 즉 현재 시각에서 일정 시간만큼 거꾸로 거슬러 그 구간 안의 호출을 매번 다시 세는 방식이 자주 사용됩니다.

```
                                       지금
-----------------------------------------|
  ←──── 60초 sliding window ────→        |
  [  성공 성공 실패 성공 실패 실패  ]      |
  지금 기준: 6번 시도 중 3번 실패 = 50%
```

이 그림에서 회로의 입력은 매 호출마다 다시 계산됩니다. 시간이 흐를수록 왼쪽 끝의 오래된 호출은 자연스럽게 빠져나가고, 오른쪽 끝에 새 호출이 더해집니다. 구현 비용을 더 줄이고 싶다면 토큰 버킷 형태로 근사하는 방법도 있고, 호출량이 많은 시스템에서는 윈도우를 잘게 쪼갠 구간별 카운터를 합산하는 방식이 흔히 쓰입니다.

마지막 입력은 **half-open 시 시도 횟수**입니다. 회로가 open에서 회복 대기 시간을 보낸 다음 half-open으로 옮겨갈 때, 거기서 몇 번을 시도해 볼지를 결정해야 합니다. 한 번만 통과하면 closed로 돌리는 설정은 회복 판단이 빠르지만, 그 한 번이 운 좋게 성공한 호출이라면 곧이어 다시 open으로 떨어집니다. 반면 시도 횟수를 많이 두면 판단이 정교해지지만, 외부가 아직 회복되지 않았을 때 그 동안의 시도가 모두 실패로 돌아가 사용자 경험이 나빠집니다. 보통은 두 길의 절충으로 두세 번의 시도를 두고, 그 시도 중 한 번이라도 실패하면 즉시 open으로 되돌리는 설계가 안정적입니다. 시도 사이 간격은 외부 시스템에 부하를 다시 주지 않도록 짧은 backoff를 두는 편이 좋습니다.

이 모든 입력을 한 자리에 모아 보면 임계치 설계는 결국 다음 다섯 가지를 정하는 일이 됩니다. 첫째, 무엇을 누적해서 볼 것인가(스텝·토큰·비용·시간·실패율). 둘째, 어떤 단위로 회로를 분리할 것인가(요청·도구·사용자). 셋째, 어떤 시간 창 위에서 비율을 계산할 것인가(고정·sliding). 넷째, 회로가 open된 뒤 얼마를 기다릴 것인가(회복 대기 시간). 다섯째, half-open에서 몇 번을 어떻게 시도할 것인가.

마지막으로 한 가지 강조해 둘 점은, 임계치 설계의 결과는 그 자체가 관측 대상이라는 사실입니다. 회로가 얼마나 자주 open되는지, 어떤 사용자에서 얼마나 자주 도달하는지, half-open에서 얼마나 자주 다시 떨어지는지를 메트릭으로 남기지 않으면 임계치를 어디로 옮길지 판단할 근거가 사라집니다. 3장에서 본 관측 스택의 trace와 metric 위에 회로의 상태 변화 자체를 이벤트로 부착해 두면, 사후에 임계치 조정 회의의 입력이 그대로 갖춰집니다.

정리하면, 에이전트의 회로 임계치는 스텝·토큰·비용·시간이라는 누적 지표와 도구별 실패율을 입력으로 두고, 사용자 등급에 따라 회로를 분리하며, 고정보다는 sliding window 위에서 비율을 계산하고, open 이후 회복 대기 시간을 둔 다음 half-open에서 두세 번의 시도로 회복 여부를 판단하는 다섯 결정의 묶음입니다. 모든 결정은 정상 작업 분포의 측정과 회로 자체의 메트릭을 통해 끊임없이 다듬어집니다.

다음 단원인 4.3.1에서는 회로가 차단으로 끝낸 그 자리를 무엇으로 채울지를 본격적으로 다룹니다. 모델·도구·응답 세 층위에서 fallback을 어떻게 분류하고, 어디까지 정적으로 두고 어디부터 동적으로 골라야 할지를 살펴보겠습니다.

이 단원을 마치면 스텝·토큰·비용 임계치와 도구 실패율을 회로의 입력으로 함께 두는 까닭, 사용자 등급에 따라 회로를 분리하는 방법, sliding window가 고정 윈도우의 경계 효과를 어떻게 풀어 주는지, 그리고 half-open 시 시도 횟수와 backoff를 어떻게 정해야 하는지를 임계치 설계의 다섯 결정으로 묶어 설명할 수 있습니다.
