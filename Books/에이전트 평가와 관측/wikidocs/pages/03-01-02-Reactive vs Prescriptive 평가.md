# 3.1.2 Reactive vs Prescriptive 평가

앞 단원에서 실 트래픽 위에 메트릭을 얹는 일을 다루었습니다. 그런데 메트릭을 얹는 행위에는 보이지 않는 가정 하나가 깔려 있습니다. 무엇을 측정해야 하는지를 우리가 미리 알고 있다는 가정입니다. 완수율과 성공률, 비용과 지연 같은 메트릭은 시스템을 띄우기 전에 이미 떠올릴 수 있고, 그 위에 thumbs·retry·abandon 같은 신호도 미리 설계해 둘 수 있습니다. 이렇게 평가 항목을 사전에 결정해 두고 그 항목에 맞춰 시스템을 측정하는 방식을 **prescriptive 평가**라고 부릅니다. 명령이나 처방을 뜻하는 prescription에서 온 표현으로, 평가가 미리 정해 둔 형판 위에서 이루어진다는 뜻입니다.

Prescriptive 평가는 시스템 안정성을 보장하는 출발점이 되지만, 한 가지 결정적인 한계를 안고 있습니다. 우리가 미리 알지 못한 실패 모드는 그 형판 안에 들어오지 않는다는 점입니다. 형판은 알려진 실패에는 강하고 알려지지 않은 실패에는 무력합니다. 비결정적이고 자율적으로 움직이는 에이전트는 새로운 도구가 추가되거나, 사용자가 새로운 방식으로 말을 걸거나, 모델 사양이 조용히 바뀔 때마다 미리 상상하지 못한 형태로 무너집니다. 이 한계를 메우기 위해 등장한 시각이 **reactive 평가**입니다. Reaction은 반응을 뜻하는 단어로, 프로덕션에서 어떤 패턴이 발견되면 그제서야 그 패턴을 잡는 평가를 새로 만들어 형판에 추가하는 방식을 가리킵니다. 평가가 시스템을 앞서 달리는 것이 아니라, 시스템이 만들어 내는 새로운 실패를 따라잡으며 함께 자라납니다.

이 차이를 가장 또렷하게 보여 주는 사례가 Replit의 코드 에이전트입니다. Replit은 사용자가 자연어로 명세를 주면 코드를 직접 작성하고 실행하는 에이전트를 운영하고 있는데, 어느 시점부터 사용자들이 'GPT-4를 쓰라고 했는데 모델 이름이 이상하게 바뀐다'는 보고를 올리기 시작했습니다. 처음에는 단순한 모델 라우팅 버그처럼 보였지만, 트레이스를 깊이 들여다보고 나서야 어떤 일이 일어나고 있었는지 드러났습니다.

```
사용자 명세:  "GPT-4o로 호출하는 코드를 짜 줘"
                       ↓
       에이전트의 내부 추론 (frontier 모델들의 지식 컷오프)
                       ↓
   "GPT-4o라는 모델은 들어 본 적이 없다.
    아마 사용자가 GPT-4를 잘못 쓴 것 같다.
    helpfully 고쳐서 GPT-4로 바꿔 주자."
                       ↓
   생성된 코드:  client.chat.completions.create(model="gpt-4", ...)
```

GPT-4o는 frontier 모델들이 학습된 시점 이후에 공개된 모델이었습니다. 그래서 에이전트가 사용하는 frontier 모델은 GPT-4o라는 이름을 자신의 지식 안에서 찾지 못했고, 사용자가 잘 모르고 잘못 입력했다고 판단해 자기 나름대로 'helpfully' 고쳐 GPT-4로 바꾸어 코드를 생성했습니다. 사용자가 명백히 신모델을 요구했는데도, 에이전트가 자신이 가진 오래된 지식을 근거로 친절하게 다운그레이드해 버린 셈입니다. 이 패턴은 단일 호출의 정확도 점수로는 잡히지 않습니다. 코드는 컴파일되고 호출도 동작하기 때문에 prescriptive 메트릭이 모두 통과합니다. 그저 사용자가 지시한 모델과 다른 모델이 호출될 뿐입니다.

Replit 팀은 이 문제를 사전에 형판으로 가지고 있지 않았습니다. 평가 항목 어디에도 '에이전트가 사용자 지정 모델 이름을 임의로 다른 이름으로 바꿔 쓰지 않는가'라는 항목은 없었습니다. 이 패턴을 발견한 흐름은 reactive 그 자체였습니다. 사용자 보고가 누적되며 트레이스를 들여다본 사람이 패턴을 식별했고, 그제서야 이 행동을 잡아내는 전용 eval이 추가됐습니다. 추가된 eval은 단순한 문자열 비교가 아니라, '사용자가 지정한 모델 이름이 응답 코드에 그대로 보존되는가'를 검사하는 형태였습니다. 새 모델이 나올 때마다 이 검사가 회귀 게이트의 일부로 자동 실행되도록 묶었습니다.

이 흐름이 일회성으로 끝나면 reactive 평가는 그저 임시방편에 머무릅니다. Replit이 한 단계 더 나아간 부분이 있는데, 발견된 패턴을 흘려보내지 않고 **eval 백로그**로 운영하는 점입니다. 새로 발견된 실패 패턴은 짧은 한 줄짜리 설명과 함께 백로그 항목으로 추가되고, 우선순위에 따라 차례로 정식 eval로 구현돼 회귀 테스트에 합류합니다. 백로그는 이슈 트래커의 한 라벨일 수도 있고, 별도의 스프레드시트일 수도 있지만, 핵심은 '평가는 한 번 만들고 끝나는 게 아니라 시간이 갈수록 자라는 자산'으로 다뤄진다는 점입니다.

```
[ 프로덕션 ]  → 사용자 보고 / 트레이스 패턴 발견
        ↓
[ Eval 백로그 ]  → 한 줄 설명 + 우선순위
        ↓
[ Eval 구현 ]   → 형판 추가 (회귀 게이트 합류)
        ↓
[ 회귀 테스트 ] → 다음 변경부터 자동 검출
```

이 흐름을 갖추고 나면 prescriptive와 reactive는 대립이 아니라 보완 관계로 자리 잡습니다. Prescriptive 평가는 알려진 위험을 빠르고 일정하게 막아 주는 형판이고, reactive 평가는 그 형판이 미처 담지 못하는 새로운 실패 모드를 흡수해 형판을 두껍게 만드는 흐름입니다. 시스템이 오래 운영될수록 reactive에서 prescriptive로 이동한 항목이 쌓여, 형판은 처음 출시 시점보다 훨씬 단단해집니다. 출시 직후의 평가 항목 수가 50개였다면, 운영 1년 뒤에는 그 위에 reactive로 발견돼 형판화된 100개가 더 얹혀 있을 수 있습니다.

여기서 한 가지 운영상의 시선을 더 적어 두겠습니다. Reactive eval을 잘 굴리려면 새 패턴을 발견할 수 있는 관측 자체가 충분히 두꺼워야 합니다. trace 없이는 모델 이름이 슬쩍 바뀌는 일을 누구도 알아채지 못합니다. 그래서 reactive 평가의 전제는 사용자 보고를 받을 채널, 트레이스 데이터, 코호트 단위 비교 같은 관측 인프라이며, 이 인프라가 약하면 reactive 평가도 형식만 남고 작동하지 않습니다. 다음 페이즈에서 다룰 LLM observability가 평가와 분리할 수 없는 이유가 여기에 있습니다.

정리하면, prescriptive 평가는 미리 정해 둔 항목으로 알려진 위험을 측정하는 방식이고, reactive 평가는 프로덕션에서 발견된 새 패턴을 따라잡아 형판에 더하는 방식입니다. Replit의 GPT-4o 자동 다운그레이드 사례는 prescriptive 형판만으로는 잡히지 않는 종류의 실패가 어떻게 일어나는지를 보여 주며, 이 사례를 흘려보내지 않고 eval 백로그를 통해 형판을 두텁게 키우는 운영이 reactive 평가를 일회성에서 자산으로 바꿉니다.

다음 단원인 3.1.3에서는 이렇게 두 흐름으로 모은 메트릭과 형판 위에서, 회귀가 실제로 발생했을 때 자동으로 잡아내는 모니터링과 알람, 그리고 회귀가 잡히면 어떻게 차단하고 되돌리는지를 다루겠습니다.

이 단원을 마치면 prescriptive 평가의 한계와 reactive 평가의 정의를 구분하고, Replit의 GPT-4o 자동 교체 사례에서 'helpfully' 다운그레이드가 일어난 메커니즘을 설명하며, eval 백로그를 통해 발견된 패턴을 형판으로 자라게 하는 운영을 자체 시스템에 적용할 수 있습니다.
