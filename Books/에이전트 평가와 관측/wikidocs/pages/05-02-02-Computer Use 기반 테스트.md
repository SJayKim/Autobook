# 5.2.2 Computer Use 기반 테스트

앞 단원에서 본 자체 Python harness가 실제로 무엇을 하고 있는지를 이번 단원에서 한 단계 안으로 들어가 살피겠습니다. Replit Agent는 사용자가 자연어로 부탁하면 코드를 짜고 의존성을 설치하고 실제로 화면을 띄워 보는 데까지 가는 web agent이기 때문에, 평가도 답안지를 채점하는 형태가 아니라 실제 브라우저 위에서 사용자처럼 클릭하고 입력하면서 결과를 확인하는 형태가 됩니다. 이 방식이 Replit이 채택한 **computer use 기반 테스트**이고, 이번 단원은 이 방식의 동작과 비용을 풀어 줍니다.

먼저 사례부터 그대로 인용하겠습니다. Replit의 발표를 옮기면, "Chrome 인스턴스와 Docker 컨테이너를 띄우고, Anthropic의 computer use를 거의 전적으로 활용해 웹 에이전트가 유저처럼 테스트 인스턴스와 상호작용"입니다. 이 한 줄에 평가의 골격이 다 들어 있습니다. 평가 한 번을 돌리기 위해 컨테이너 한 개와 브라우저 한 개를 새로 띄우고, 그 브라우저를 사람이 아니라 또 다른 모델이 마우스와 키보드를 쥐고 조작합니다. 이 모델이 바로 **computer use** 기능을 가진 모델, 즉 화면 픽셀을 보고 다음 클릭 좌표나 키 입력을 결정해 돌려주는 모델입니다.

용어를 한 단계 풀어 두겠습니다. **Chrome 인스턴스**는 평가용으로 새로 띄운 한 개의 크롬 브라우저 프로세스를 가리킵니다. 일반 사용자의 브라우저와 똑같이 동작하지만 평가 한 회차 동안에만 살아 있도록 격리된 환경에서 깨워 둔 것입니다. **Docker 컨테이너**는 그 브라우저와 평가 대상이 되는 Replit 앱 자체를 함께 담아 두는 가벼운 격리 상자입니다. 같은 호스트 위에서 여러 컨테이너를 띄워 두면 서로의 파일과 네트워크가 섞이지 않습니다. **테스트 인스턴스**는 평가 한 회차마다 새로 만든 Replit 코드 환경을 가리키고, 사용자처럼 행동하는 평가 에이전트가 이 인스턴스를 상대로 클릭하고 입력하면서 결과를 봅니다.

전체 흐름을 그림으로 정리하면 다음과 같습니다.

```
   [평가 케이스 한 건]
          |
          v
 +----------------------------+
 |  자체 Python harness        |
 |   1. Docker 컨테이너 기동   |
 |   2. 안쪽에 Chrome 인스턴스 |
 |   3. Replit 앱 인스턴스 배치|
 |   4. 시나리오 주입          |
 +----------------------------+
          |
          v
 +----------------------------+
 |  Computer Use 모델          |
 |   - 화면 픽셀 관찰          |
 |   - 다음 클릭/입력 결정     |
 |   - 사용자처럼 상호작용     |
 +----------------------------+
          |
          v
 +----------------------------+
 |  Replit Agent (평가 대상)   |
 |   - 코드 작성               |
 |   - 의존성 설치             |
 |   - 결과 화면 노출          |
 +----------------------------+
          |
          v
   결과 수집 -> Braintrust/LangSmith로 전송
          |
          v
   컨테이너/브라우저 폐기 (cleanup)
```

이 그림에서 가장 먼저 짚을 점은 **사용자식 상호작용**의 의미입니다. 보통의 평가는 모델에 한 번의 입력을 주고 한 번의 출력을 받아 채점하는 식으로 끝나지만, web agent의 평가는 그 흐름이 길게 늘어집니다. 평가 에이전트가 브라우저를 열고 텍스트 박스를 찾고, 부탁할 문장을 입력하고, 전송 버튼을 누르고, 화면이 갱신될 때까지 기다리고, 그 화면에 무슨 변화가 생겼는지 다시 읽어야 합니다. 이 동작을 사람이 아닌 또 다른 모델에게 시키는 것이 computer use 기반 평가의 핵심입니다. 손이 필요한 곳에 손 역할을 하는 모델을 둠으로써, 평가가 사용자의 실제 사용 경로와 거의 같아집니다. 이 일관성은 단순 입력-출력 평가로는 닿지 못하는 영역, 예컨대 화면이 늦게 뜨거나 잘못된 위치에 버튼이 그려지는 문제까지 평가 안에 끌고 들어옵니다.

다음으로 **환경 isolation**과 **cleanup**의 자리를 풀어 두겠습니다. 같은 호스트에서 평가를 여러 번 연달아 돌릴 때 서로의 자취가 남으면 점수가 인위적으로 부풀거나 쪼그라들 수 있습니다. 이 문제는 2.5에서 한 번 정리했고, Replit이 컨테이너를 회차마다 새로 띄우는 이유도 같은 원칙 위에 있습니다. 한 번의 평가가 끝나면 컨테이너 안의 파일과 캐시, 브라우저의 쿠키와 로컬 스토리지, Replit 앱 인스턴스 안의 데이터가 모두 폐기됩니다. 이 폐기가 빠지면 다음 회차의 에이전트가 이전 회차의 부산물을 보고 더 쉽게 풀거나 반대로 엉뚱한 상태에 막혀 더 어렵게 푸는 일이 생깁니다. **isolation**은 한 회차의 출발점이 항상 깨끗한 상태가 되도록 만드는 일이고, **cleanup**은 그 회차가 다음 회차를 오염시키지 않도록 정리하는 일입니다. 두 가지가 짝을 이루지 않으면 어느 한쪽만 있어도 isolation이 무너집니다.

여기서 5.1에서 본 Anthropic의 isolated 환경 원칙이 다시 떠오릅니다. Anthropic 사례에서는 한 trial이 이전 trial의 git history를 봐서 unfair advantage를 얻은 일이 내부 평가에서 발견됐다는 경고가 있었습니다. Replit의 컨테이너-브라우저 폐기는 그런 종류의 누수를 막는 실무 구현에 가깝고, web agent라는 더 까다로운 평가 환경에서는 isolation의 단위가 코드 디렉토리뿐 아니라 브라우저 상태, 네트워크 캐시, 화면 쿠키까지 함께 묶여야 한다는 사실을 보여 줍니다. 같은 원칙을 다른 표면에 적용한 셈입니다.

이 모든 구도가 자연스럽게 청구하는 것이 **computer use 기반 평가의 비용**입니다. 비용은 크게 세 줄기로 나뉩니다. 첫째, 인프라 비용입니다. 컨테이너와 브라우저를 회차마다 새로 깨우면 평가 한 건당 시작에서 끝까지 수십 초에서 수 분이 걸리고, CPU·메모리·네트워크 자원이 동시에 점유됩니다. 평가 케이스가 수백 건이 되는 순간 동시에 몇 개의 컨테이너를 돌릴 수 있는지가 곧 평가 한 라운드의 길이를 결정합니다. 둘째, 모델 호출 비용입니다. computer use 모델이 화면을 본다는 말은 매 단계마다 스크린샷을 보내고 다음 동작을 받는다는 뜻이고, 이미지 입력은 텍스트 입력보다 토큰 단가가 비쌉니다. 한 평가 케이스 안에 사용자가 클릭하는 단계가 열 번이면 그만큼 호출이 누적됩니다. 셋째, 운영 비용입니다. 컨테이너가 깨어나지 않거나 브라우저가 멈추는 일은 의외로 자주 일어나고, 그 실패가 모델의 실패인지 인프라의 flakiness인지 가르는 진단 코드가 따로 필요합니다.

이 비용을 다스리기 위해 보통 다음 두 가지가 함께 따라옵니다. 하나는 평가 케이스의 계층화입니다. 모든 케이스를 매번 computer use로 풀려고 하지 않고, 코드 산출물 자체로 채점할 수 있는 영역과 사용자식 상호작용까지 거쳐야만 의미가 있는 영역을 갈라 둡니다. 단위 테스트로 풀리는 케이스는 가벼운 채점 경로로, 화면을 보고 클릭해야 의미가 있는 케이스는 무거운 computer use 경로로 보냅니다. 다른 하나는 회귀 회차와 풀 회차의 분리입니다. PR마다 도는 빠른 회귀에는 가벼운 케이스만 넣고, 무거운 computer use 케이스는 정해진 시간에 일괄로 돌리는 식입니다. 이 두 가지가 빠지면 평가 한 라운드가 너무 길어져 사람들이 평가를 우회하기 시작합니다.

마지막으로 같은 패턴을 다른 회사가 모방할 때 자주 빠지는 함정을 짚겠습니다. computer use 기반 평가는 평가 대상 자체가 화면 위의 행동이어야 가치가 있습니다. 답을 텍스트로 돌려주는 에이전트나 API만 호출하는 에이전트라면, 화면을 거치는 평가는 비싸기만 하고 새로운 신호는 거의 주지 않습니다. Replit이 이 길을 간 이유는 평가 대상이 곧 사용자가 보는 화면이기 때문이고, 이 전제가 빠진 채 패턴만 모방하면 비싼 인프라만 짊어지게 됩니다.

정리하면, Replit의 computer use 기반 테스트는 평가 회차마다 Docker 컨테이너와 Chrome 인스턴스를 새로 깨우고, Anthropic의 computer use 모델을 손 역할로 써서 사용자처럼 화면을 클릭하고 입력하며 평가 대상 에이전트와 상호작용하는 방식입니다. 이 구성은 web agent의 평가를 사용자 경험에 가까이 붙이고 isolation·cleanup을 매 회차 강제하지만, 인프라·모델 호출·운영 비용이 함께 늘기 때문에 케이스 계층화와 회차 분리로 비용을 다스려야 합니다.

다음 단원인 5.2.3에서는 이렇게 무거운 평가 인프라를 갖췄음에도 prescriptive하게 미리 모든 위험을 막을 수는 없다는 사실, 즉 reactive eval이 필요해지는 이유와 가드레일조차 창의적인 prompting으로 뚫리는 현실을 두 가지 사례로 풀어 보겠습니다.

이 단원을 마치면 Replit이 Chrome 인스턴스와 Docker 컨테이너 위에서 Anthropic의 computer use 모델을 활용해 web agent를 사용자처럼 평가하는 동작 방식을 설명하고, 이 방식이 환경 isolation과 cleanup을 어떻게 강제하며 인프라·모델 호출·운영 측면의 비용을 어떻게 청구하는지를 함께 정리할 수 있습니다.
