# 4.4.3 Escalation Path 설계

앞 두 단원에서 전수 승인 모델의 한계와, 사람이 들여다볼 자리를 임계치로 자동 식별하는 패턴을 살폈습니다. 임계치가 작동하면 큐로 넘어오는 행동의 양은 줄지만, 큐로 넘어온 케이스가 사람에게 어떻게 도달하고, 사람이 처리한 결과가 다시 사용자에게 어떻게 돌아오는지는 별도의 설계가 필요합니다. 이 자리를 흔히 **escalation path**라고 부릅니다. 단어 그대로 해석하면 '단계를 올려 보내는 경로'라는 뜻이고, 실제로는 에이전트가 처리하다가 사람으로 핸드오버하는 한 줄기 흐름 전체를 가리킵니다. 이 단원은 이 흐름을 어떤 신호로 시작하고, 어떤 컨텍스트와 함께 사람에게 넘기며, 사용자가 그 사이를 어떻게 경험하고, 무엇을 KPI로 측정해야 하는지를 정리합니다.

먼저 escalation을 트리거하는 신호부터 풀겠습니다. 4.4.2에서 본 신뢰도, 비가역, 비용은 행동 자체에 붙는 신호였고, 사용자와 모델 사이의 상호작용에서만 드러나는 신호가 따로 있습니다. 가장 자주 다뤄지는 것이 **감정 신호**와 **복잡도 신호**입니다. 감정 신호는 사용자의 발화에 좌절·분노·불안 같은 정서가 실릴 때 잡힙니다. 분류 모델로 감정 라벨을 붙이거나, 키워드 기반으로 강한 부정어가 누적되는 빈도를 세거나, 같은 사용자가 같은 문제를 세 번째 다시 묻고 있는 패턴을 잡는 식입니다. 복잡도 신호는 질문 자체가 단일 의도로 떨어지지 않거나, 도메인 정책이 얽혀 있거나, 개인 상황 정보가 깊게 들어가 있을 때 잡힙니다. 한 사용자가 한 문장 안에 결제 환불, 분쟁 조정, 명의 변경을 동시에 요구하는 경우가 전형적인 예입니다. 이 두 신호는 신뢰도 점수만으로는 잡히지 않습니다. 모델은 자기 답에 0.9의 확신을 가지면서도, 그 답이 사용자의 정서를 풀어 주지 못하거나 복잡한 사정을 다 담지 못할 수 있기 때문입니다.

이 신호를 흘려보내면 어떤 일이 벌어지는지는 1.3에서 다룬 Klarna 사례가 보여 줍니다. 이 책의 1단계에서 본 내용을 다시 짚자면, Klarna는 23개 시장에서 한 달 230만 건의 대화를 처리하면서 CSAT를 47% 끌어올리고 1천만 달러를 절감했다고 발표했습니다. 숫자만 보면 완성된 자동화로 보였지만, 시간이 지나면서 다른 결이 드러났습니다. 복잡하거나 감정적인 케이스에서 사용자가 unheard·unsupported하다고 느꼈고, 비용이 평가의 압도적 축이 되면서 hidden quality가 깎여 나갔습니다. 결국 회사 측이 인간 상담사를 다시 고용해 hybrid 모델로 회귀했습니다. 이 회귀가 주는 교훈은 한 줄로 정리됩니다. 감정·복잡도 신호 위에서 사람으로 escalation할 자리가 비어 있으면, 단일 KPI 위에서 잘 보이던 시스템도 hidden quality 침식으로 무너집니다. **Klarna의 hybrid 회귀**는 escalation path 설계가 nice-to-have가 아니라 위험 관리의 한 축임을 말해 줍니다.

신호가 잡혀 사람에게 넘어가기로 결정되면 다음 문제는 **handover 시 컨텍스트 전달**입니다. 사람이 처음부터 다시 듣게 만드는 escalation은 사용자에게 두 번 같은 말을 시키는 것과 같고, 가뜩이나 좌절한 사용자의 감정을 한 번 더 자극합니다. 좋은 핸드오버는 사람 운영자가 자리에 앉는 순간 다음 정보가 화면에 정리되어 있는 상태를 만듭니다. 첫째, 대화의 요약입니다. 사용자가 요청한 내용, 에이전트가 시도한 행동, 막힌 지점, 감정이 격해진 시점이 한 단락으로 압축됩니다. 둘째, 원본 trajectory의 링크입니다. 운영자가 요약만으로 부족하다고 느끼면 한 클릭으로 단계별 도구 호출과 응답을 그대로 살펴볼 수 있어야 합니다. 셋째, 사용자 식별과 권한 컨텍스트입니다. 사용자의 계정, 등급, 보유 상품, 과거 escalation 이력 같은 정보가 자동 첨부되어야 운영자가 추가 질문 없이 결정에 들어갈 수 있습니다. 이 세 항목이 빠지면 핸드오버는 형식적 이관이 되고, 사용자는 사람과 대화하는 데도 같은 좌절을 다시 겪습니다.

handover와 짝을 이루는 것이 **사람 응답까지의 대기 경험**입니다. 사람이 자리에 앉아 화면을 보는 데까지는 짧게는 1분, 길게는 수십 분이 걸립니다. 이 사이에 사용자에게 어떤 화면, 어떤 메시지를 띄우는지가 escalation의 체감 품질을 결정합니다. 빈 화면을 두면 사용자는 시스템이 멈췄다고 느끼고, 한 번 더 같은 질문을 입력하거나 채널을 떠납니다. 좋은 대기 경험은 세 가지를 동시에 보여 줍니다. 첫째, 지금 사람에게 넘어가는 중이라는 사실. 둘째, 예상 대기 시간의 범위. 셋째, 그 사이에 사용자가 할 수 있는 것이 무엇인지(예: 영수증을 첨부, 추가 메모를 남기기). 이 세 가지가 화면에 떠 있으면 같은 5분 대기도 다른 경험이 됩니다. 

전체 흐름을 한 그림으로 묶으면 다음과 같습니다.

```
 [에이전트 진행]
        |
   감정·복잡도 신호 감지     +-- 4.4.2 임계치 분기에서 트리거된 케이스도 합류
        |                   |
        v                   v
 [Escalation 결정]----+ 컨텍스트 패키지 작성
                     |   - 대화 요약
                     |   - trajectory 링크
                     |   - 사용자/권한 정보
                     v
              [사람 운영자 큐]
                     |
                     |   사용자 화면: "사람 상담사에게 연결 중", 예상 대기, 보조 액션
                     v
              [사람이 응답·결정]
                     |
                     v
              [사용자에게 결과 전달] -> 결과 라벨이 학습 데이터로 흡수
```

이 그림에서 흘러나오는 라벨은 단순한 운영 기록을 넘어 시스템 개선의 입력입니다. 어떤 신호 조합에서 escalation이 자주 일어났는지, 사람이 어떤 결정을 자주 내렸는지, 그 결정이 사용자 만족으로 이어졌는지가 다음 임계치 조정과 모델 보정의 근거가 됩니다.

여기서 escalation path도 측정 가능한 지표로 끌어와야 합니다. 가장 직관적인 한 줄은 **escalation 비율**입니다. 전체 대화 가운데 사람으로 핸드오버된 비율, 그 가운데 사용자가 만족한 비율, 그 가운데 다시 에이전트로 돌아온 비율을 묶어 봅니다. 비율 하나만으로는 의미가 갈리지 않으니, 도메인 단위로 분해해서 보는 편이 안전합니다. 결제 분쟁에서 escalation 비율이 12%인지, 단순 잔액 조회에서 0.5%인지를 따로 보면, 어떤 도메인의 신호 임계치를 다듬어야 하는지가 보입니다. 이 비율이 너무 낮으면 사람으로 넘어갈 자리가 막혀 있다는 신호, 너무 높으면 자동화 경계가 너무 보수적이라는 신호입니다. 두 극단 사이에서 균형이 잡힐 때까지 임계치와 신호를 다듬는 일이 운영의 큰 부분을 차지합니다.

escalation 비율과 함께 보아야 하는 보조 지표가 두 가지 있습니다. 하나는 **handover 후 회복 시간**으로, 사용자가 사람에게 넘겨진 시점부터 만족 신호가 잡히는 시점까지의 시간입니다. 시간이 길어지면 사람의 처리 속도와 컨텍스트 전달의 품질을 점검해야 합니다. 다른 하나는 **재escalation 비율**로, 같은 사용자가 한 번 핸드오버를 받은 뒤 같은 세션 안에서 다시 escalation이 필요해진 비율입니다. 이 비율이 높으면 사람이 처리해도 근본 문제가 안 풀리고 있다는 뜻이고, 도구 자체나 정책에 손을 대야 할 자리입니다.

이 모든 장치가 작동하면 escalation path는 사용자 입장에서 시스템의 안전망이자 마지막 보루가 됩니다. AI는 효율성을 책임지고 사람은 공감을 책임지는 hybrid 운영 모델이 자연스럽게 자리를 잡고, Klarna가 회귀로 도달했던 형태가 처음부터 설계 위에 들어가게 됩니다. 자동화의 효율을 살리면서도 hidden quality 침식을 막는 한 가지 길이 여기에 있습니다.

정리하면, escalation path는 감정과 복잡도 신호를 임계치 분기와 함께 트리거로 받고, 대화 요약과 trajectory 링크와 사용자 컨텍스트를 묶은 패키지로 사람에게 핸드오버하며, 그 사이의 대기 경험을 사용자에게 명시적으로 안내하고, escalation 비율과 회복 시간과 재escalation 비율을 KPI로 관측하는 한 줄기 흐름입니다. Klarna의 hybrid 회귀는 이 흐름이 비어 있으면 단일 KPI 위에서 빛나던 시스템도 사용자가 unheard·unsupported하다고 느끼는 자리에서 무너진다는 점을 분명히 보여 주는 사례이며, 같은 자리에서 hybrid 운영을 처음부터 설계에 넣어 두는 일이 escalation path 설계의 핵심입니다.

이로써 Phase 4의 신뢰성과 안전 패턴을 마무리합니다. 다음 단원인 5.1.1부터는 Phase 5로 넘어가, 지금까지 살펴본 평가·관측·신뢰성 패턴들이 실제 사례에서 어떻게 작동하는지를 따라갑니다. 첫 사례는 Anthropic의 Lead Agent와 병렬 Subagent 구조로 이루어진 Multi-Agent Research System입니다. Lead가 어떻게 작업을 분해하고, Subagent가 어떻게 병렬로 움직이며, 그 결과가 어떻게 합성되는지를 살펴보겠습니다.

이 단원을 마치면 감정·복잡도 신호 기반의 escalation 트리거를 설계하고, 핸드오버 패키지로 컨텍스트를 사람에게 전달하며, 사용자 대기 경험을 명시적으로 다루고, escalation 비율과 회복 시간과 재escalation 비율을 KPI로 측정하는 흐름을 설계할 수 있고, Klarna 사례의 hybrid 회귀 교훈을 escalation path 부재의 결과로 설명할 수 있습니다.
