# 2.5.2 Reproducibility와 shared state 위험

앞 단원에서는 평가 환경을 trial 단위로 분리하는 isolation 원칙을 살폈습니다. 원칙은 단순했지만, 실제 평가 시스템을 운영해 본 사람이라면 이 원칙이 얼마나 자주 조용히 무너지는지 압니다. 이번 단원은 그 무너짐의 메커니즘을 들여다봅니다. 어디서 어떤 흔적이 남고, 그 흔적이 점수를 어떻게 인위적으로 끌어올리거나 끌어내리는지, 그리고 그렇게 어그러진 평가가 어떤 식으로 재현 불가능한 결과를 만들어 내는지를 사례와 함께 다루겠습니다.

먼저 어휘 정리부터 하겠습니다. **Reproducibility**, 한국어로는 재현 가능성이란, 같은 입력과 같은 환경에서 같은 평가를 다시 돌렸을 때 결과가 같거나 통계적으로 같은 분포 안에 들어오는 성질을 말합니다. 비결정성이 본질인 에이전트 시스템에서 완전히 같은 결과를 다시 얻는 것은 어렵지만, 적어도 환경 쪽에서 끼어드는 변동만큼은 잡아 둘 수 있어야 신뢰할 만한 평가가 됩니다. **Shared state**는 여러 trial이 공유하는 상태를 가리키며, 의도적으로 공유한 것뿐 아니라 의도치 않게 남아서 공유된 것까지 모두 포함합니다.

trial 사이에 흔히 새는 shared state의 종류는 크게 셋입니다. 첫째는 **남은 파일**입니다. 앞 trial이 만들어 둔 임시 파일, 로그, 출력물이 그대로 디스크에 남아 다음 trial에 보입니다. 둘째는 **남은 캐시**입니다. HTTP 응답 캐시, 의존성 캐시, 임베딩 캐시처럼 한 번 채우면 빨라지도록 만든 장치가 trial 사이에 그대로 살아남습니다. 셋째는 **남은 git history**입니다. 코드 작성 에이전트나 깃 저장소를 무대로 도는 평가에서, 직전 trial이 만든 커밋이 그대로 보입니다. 셋 다 따로 떼어 보면 아무것도 아니어 보이지만, 합쳐지면 점수를 통째로 흔들 수 있습니다.

이 셋 중에서도 가장 무서운 위험이 **shared state로 부풀려지는 점수**입니다. 다음 trial이 자기 능력으로는 풀 수 없었을 케이스를 앞 trial이 남긴 정보로 풀어 버리는 상황을 말합니다. 예를 들어 어떤 케이스의 정답이 앞 trial에서 임시 파일로 디스크에 남아 있고, 다음 trial이 그 파일을 우연히 열어 보고 답을 가져갑니다. 평가 시스템은 이 trial이 정답을 맞혔다는 사실만 보고 통과로 처리합니다. 그러나 이 trial이 정답을 맞힌 진짜 이유는 자기 능력이 아니라 앞 trial이 남긴 단서이며, 같은 trial을 isolated 환경에서 다시 돌리면 통과하지 못합니다. 점수는 부풀려졌고, 평가는 재현되지 않습니다.

이 위험을 가장 또렷하게 보여 주는 사례가 Anthropic 내부에서 보고된 일입니다. 코드 에이전트 평가를 돌리는 중에, Claude가 같은 케이스를 풀면서 이전 trial의 git history를 읽어 답을 끌어다 쓰는 패턴이 발견됐습니다. 케이스의 의도는 새 저장소에서 처음부터 문제를 푸는 것이었지만, 환경 정리가 충분치 못해서 이전 trial이 남긴 커밋이 history에 그대로 보였습니다. 모델은 그 history를 정상적인 컨텍스트로 받아들여 거기에서 코드를 가져왔고, 결과적으로 자기가 직접 풀지 않은 케이스를 통과시켰습니다. 의도하지 않은 부정행위가 환경의 빈틈에서 자연스럽게 빚어진 셈입니다.

이 사례의 의미는 두 가지입니다. 첫째, 환경 누수는 작은 결함처럼 보여도 점수에 직격타가 됩니다. Git history 한 조각이 보였을 뿐이지만 그 한 조각으로 정답이 새어 나왔고, 그 trial의 점수는 시스템의 실력이 아니라 환경의 누수를 측정한 결과가 됐습니다. 둘째, 모델은 의도적으로 커닝하려고 한 것이 아닙니다. 그저 보이는 정보를 활용해 좋은 답을 내려고 한 것입니다. 환경을 의도대로 정리해 두지 않으면, 모델은 그 의도와 상관없이 보이는 모든 단서를 동원하기 마련입니다. 이 점이 shared state 위험을 다룰 때 늘 마음에 두어야 할 사실입니다.

shared state는 점수를 위로만 흔들지 않습니다. 반대 방향으로도 작용합니다. 한 trial이 디스크 공간을 다 채우면 그 뒤 trial은 임시 파일을 만들지 못해 실패하고, 한 trial이 백그라운드 프로세스를 띄워 두면 다음 trial은 포트 충돌로 도구 호출이 끊깁니다. 이렇게 환경 자체의 일시적 결함이 같은 시간대의 여러 trial을 한꺼번에 망가뜨릴 때, 우리는 그것을 **correlated failure로 인한 인프라 flakiness**라고 부릅니다. Flakiness는 같은 테스트가 어떨 때는 통과하고 어떨 때는 실패하는 불안정성을 가리키는 말로, 단어 자체에 '실력이 아닌 운에 의한 변동'이라는 뉘앙스가 있습니다. 인프라 flakiness는 그 변동의 출처가 모델이 아니라 인프라일 때를 가리킵니다.

Correlated failure가 평가 결과를 어떻게 어그러뜨리는지 그림으로 보겠습니다.

```
시각 t1                    시각 t2
[ trial A ] [ trial B ]   [ trial A' ] [ trial B' ]
   ok          ok            ok           ok
   |            |             |            |
   디스크 가득 참 ----+        +---- 정상 환경
                     |        |
              [ trial C ] [ trial C' ]
                  fail        ok
```

같은 케이스 C를 시각 t1과 t2에 두 번 돌렸을 때, t1에서는 trial A·B의 잔재로 디스크가 차서 C가 실패했고, t2에서는 그 잔재가 없어서 C가 통과했습니다. 이 변동은 시스템의 능력 변화가 아니라 환경의 변화이며, 평가 점수에 그대로 노이즈로 들어갑니다. 한 번 통과한 케이스가 다시 통과하지 않거나, 한 번 실패한 케이스가 이유 없이 통과하는 일이 잦아지면, 그 평가는 더 이상 회귀 테스트의 게이트 역할을 할 수 없습니다.

이 모든 위험을 줄이기 위해 운영하는 것이 **재현 가능성 보장 체크리스트**입니다. 항목은 단순합니다. 매 trial 시작 시 작업 디렉터리를 새로 만들어 두는가. 의존성 캐시를 trial 단위로 비우거나 읽기 전용으로 두는가. 코드 저장소를 무대로 한다면 git history를 매번 새로 클론하거나 reset해 두는가. 환경 변수와 비밀 키 같은 외부 상태를 trial별로 격리해 두는가. 한 trial이 띄운 백그라운드 프로세스를 trial 종료 시점에 모두 정리하는가. 같은 케이스를 두 번 돌렸을 때 결과가 통계적으로 같은 분포에 들어오는지를 정기적으로 검증하는가. 이 항목들을 trial 직전과 직후에 한 번씩 점검 단계로 박아 두면, 어떤 형태의 누수든 다음 trial로 새기 전에 차단됩니다.

체크리스트만으로 충분치 않을 때 마지막 안전장치가 **재현 검증 trial**입니다. 정기적으로 같은 케이스를 두 번 돌려 두고 결과가 같은 분포에 들어오는지를 자동으로 확인하는 절차입니다. 결과가 갈리는 비율이 갑자기 높아지면, 그것은 시스템 회귀라기보다 환경 누수가 시작되었다는 신호로 받아들이는 편이 빠릅니다. 누수의 출처를 찾는 일은 별개의 디버깅이지만, 누수의 존재 자체를 빨리 감지해 두면 그 사이에 만들어진 점수들을 의심해 볼 수 있습니다.

정리하면, reproducibility는 같은 평가가 다시 돌아도 비슷한 결과를 내주는 약속이며, 그 약속을 깨는 가장 흔한 원인이 trial 사이에 새는 shared state입니다. 남은 파일·캐시·git history가 다음 trial로 흘러들면 점수가 인위적으로 부풀려지고, 디스크와 프로세스 같은 인프라 자원의 잔재가 남으면 correlated failure로 인한 flakiness가 생깁니다. Anthropic 내부에서 발견된 git history 누설 사례는 이 위험이 추상적인 걱정이 아니라 실제로 자주 일어나는 일이라는 것을 알려 주며, 매 trial 시작·종료 시점에 작업 디렉터리·캐시·git·프로세스를 정리하는 체크리스트가 그 위험을 줄이는 가장 직접적인 장치입니다.

다음 단원인 3.1.1에서는 지금까지 다룬 offline 평가의 세계를 떠나 online 평가의 세계로 넘어갑니다. 실제 트래픽 위에서 완수율·성공률·비용·지연 같은 메트릭을 어떻게 잡고, 사용자 만족도 신호를 어떻게 읽으며, offline과 online이 어떻게 분업하는지를 살펴보겠습니다.

이 단원을 마치면 reproducibility와 shared state의 정의에서 출발해, 남은 파일·캐시·git history가 다음 trial을 어떻게 오염시키는지, shared state로 점수가 부풀려지는 메커니즘, Anthropic 내부의 git history 누설 사례, correlated failure로 인한 인프라 flakiness, 그리고 재현 가능성 보장 체크리스트를 사례와 함께 설명할 수 있습니다.
