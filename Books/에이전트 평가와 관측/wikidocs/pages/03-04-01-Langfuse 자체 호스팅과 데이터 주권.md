# 3.4.1 Langfuse 자체 호스팅과 데이터 주권

앞 절에서 살펴본 LangSmith는 LangChain 진영이 만든 호스티드 관측 플랫폼으로, 가입하고 키 한 줄을 환경변수에 넣으면 곧장 trace가 흘러 들어가는 손쉬운 출발점을 제공합니다. 그런데 한국이나 유럽의 일부 기업은 이 손쉬움 앞에서 잠시 멈춥니다. trace 안에는 사용자 질의, 도구 인자, 모델 응답이 그대로 담겨 있고, 그 데이터를 외부 회사의 클라우드에 보관해도 되는지가 별도 의사결정이 되기 때문입니다. 이 단원에서 다루는 **Langfuse**는 바로 이 지점에서 등장하는 선택지입니다. 호스티드 SaaS도 제공하지만, 동일한 코드와 동일한 기능을 사내 인프라에 그대로 띄울 수 있다는 점이 본질적인 차별점입니다.

먼저 용어부터 풀겠습니다. **자체 호스팅**, 영어로 self-hosted라는 말은 어떤 소프트웨어를 만든 회사의 클라우드가 아니라 사용하는 쪽이 직접 운영하는 서버 위에 그 소프트웨어를 띄워 쓰는 방식을 가리킵니다. 메일 서비스로 비유하면, 외부 메일 회사를 쓰는 대신 회사 안에 메일 서버를 직접 두고 쓰는 형태입니다. 호스티드 서비스가 가입과 동시에 쓸 수 있는 편의를 주는 대신 데이터가 외부에 흐르고, 자체 호스팅은 운영 부담을 지는 대신 데이터가 내부에 머무는 구조입니다.

Langfuse가 자체 호스팅을 의미 있게 만드는 첫 번째 조건은 **오픈소스로 공개되어 있다는 점**입니다. 서버 코드, 웹 UI, SDK가 모두 공개 저장소에 들어 있고, 사용자는 이 코드를 그대로 받아 자기 서버에서 빌드하거나 공식 컨테이너 이미지를 그대로 끌어와 띄울 수 있습니다. 호스티드 버전을 쓰지 않더라도 핵심 기능에 차이가 거의 없다는 점이 중요합니다. trace 수집, span 트리 시각화, 데이터셋 관리, 평가 실행, 사용자 점수 부착 같은 기능이 self-hosted 환경에서도 동일하게 동작합니다.

구조를 한눈에 보기 위해 자체 호스팅 배포의 구성을 그림으로 정리해 보겠습니다.

```
   [에이전트 앱]                       사내 데이터센터 / VPC 안
        |
        | Langfuse SDK / OTLP
        v
 +----------------------+        +-------------------------+
 |  Langfuse Web (UI)   | <----> |  Langfuse Worker        |
 |  Langfuse API server |        |  (trace 처리, 집계)     |
 +----------+-----------+        +-----------+-------------+
            |                                |
            v                                v
     +-------------+                +------------------+
     |  Postgres   |  메타데이터    |  ClickHouse      |
     |  (계정, 설정,|                |  (trace, span,   |
     |   데이터셋) |                |   이벤트 본문)   |
     +-------------+                +------------------+
```

이 그림에서 가장 중요한 점은 모든 박스가 회사의 네트워크 경계 안에 들어 있다는 것입니다. 에이전트 앱이 만들어 내는 trace는 사내 네트워크를 떠나지 않고 Langfuse Web과 Worker로 흘러 들어가며, 최종적으로 사내에 둔 두 개의 데이터베이스에 저장됩니다.

여기서 두 개의 데이터베이스가 등장한 까닭을 풀어 두겠습니다. **Postgres**는 한 줄짜리 사용자 계정, 프로젝트 설정, 데이터셋 정의처럼 자주 바뀌지 않고 정확한 트랜잭션이 필요한 데이터를 담는 관계형 데이터베이스입니다. 한편 trace나 span은 양이 압도적으로 많고 시간순으로 쌓이며 주로 집계 쿼리를 돌립니다. 한 사용자가 하루에 만들어 내는 trace가 수만 건을 넘기 때문에, 이런 데이터를 관계형 데이터베이스에 그대로 넣으면 디스크와 쿼리 비용이 빠르게 한계에 부딪힙니다. 그래서 trace 본문은 **ClickHouse**라는 컬럼 지향 데이터베이스에 따로 저장합니다. 컬럼 지향이란 가로로 행을 늘어놓는 일반 데이터베이스와 달리 같은 열의 값을 한곳에 모아 두는 저장 방식으로, 시간 범위를 잘라 읽거나 평균·합계를 구하는 분석 쿼리에서 큰 폭으로 빠릅니다. 두 데이터베이스를 함께 쓰는 까닭은 빠르게 변하는 메타데이터와 대량 누적되는 관측 데이터가 서로 다른 저장 모델을 요구하기 때문입니다.

이 자체 호스팅 구조를 띄우는 가장 흔한 방식은 두 갈래입니다. 첫째는 **Docker Compose**로 여러 컨테이너를 한 호스트에 함께 올리는 방식입니다. 한 대의 가상 머신 위에 Langfuse Web, Worker, Postgres, ClickHouse를 모두 올려 두고 운영하는 패턴으로, 팀이 작거나 트래픽이 크지 않은 단계에서 적합합니다. 둘째는 **Kubernetes**를 이용해 각 컴포넌트를 별개 워크로드로 돌리는 방식으로, 트래픽이 늘면 Worker만 따로 늘리거나 ClickHouse를 클러스터로 키울 수 있습니다. Langfuse는 두 패턴 모두 공식 매니페스트와 Helm 차트를 제공하기 때문에, 이미 사내에 Kubernetes 운영 역량이 있는 팀이라면 후자로 시작하는 편이 운영 측면에서 자연스럽습니다.

이쯤에서 가장 중요한 단어를 풀겠습니다. **데이터 주권**은 어떤 데이터가 어디에 저장되고 누구의 법과 정책의 적용을 받는지에 관한 권한을 가리킵니다. 사용자가 한국 안에서 만든 대화 데이터를 한국 안의 서버에 보관해야 한다면 그 데이터의 주권은 한국에 있고, 같은 데이터를 미국 클라우드에 보관하면 미국 법의 영향을 받게 됩니다. 비슷한 맥락에서 **데이터 거주성**이라는 표현도 쓰이는데, 이는 데이터가 물리적으로 머무는 위치를 가리키는 좀 더 좁은 의미입니다. 한국의 금융권은 망분리 규정과 개인정보 국외 이전 동의 제도 때문에, 유럽은 GDPR의 국외 이전 제한 때문에 두 개념이 곧장 의사결정에 영향을 줍니다.

이 규제 환경 위에서 한국 기업의 도입 의사결정이 갈리는 지점은 대체로 다음과 같은 표로 정리됩니다.

```
구분                        호스티드 LangSmith    자체 호스팅 Langfuse
-------------------------   ------------------   --------------------
초기 셋업 비용              매우 낮음             중간(인프라 준비)
운영 부담                   거의 없음             직접 짊어짐
데이터 위치                 외부 클라우드         사내·국내 클라우드
망분리 환경 적용 가능       제한                  가능
민감 정보(PII) 처리         외부 위탁 검토 필요   내부 정책 안에서 처리
업그레이드 자유도           제공자 일정에 종속    원하는 시점에 적용
```

이 표를 보면 자체 호스팅이 무조건 정답이 아님을 알 수 있습니다. 운영 부담을 직접 떠안아야 하고, 백업과 업그레이드, 보안 패치를 사내 인프라 팀이 책임지게 됩니다. 반대로 호스티드는 빠르게 시작할 수 있는 대신 PII가 흐르는 trace를 외부에 두는 결정에 도장을 찍어야 합니다. 한국의 금융, 의료, 공공기관처럼 망분리와 개인정보 국외 이전 제한이 강한 업종은 처음부터 자체 호스팅을 전제로 도구 후보를 좁히는 경우가 많고, 일반 IT 서비스나 사내 PoC는 호스티드로 시작했다가 데이터 양이 커지거나 컴플라이언스 검토가 들어오는 시점에 자체 호스팅으로 옮기는 흐름이 잦습니다.

여기서 자연스럽게 떠오르는 질문은 LangSmith도 self-hosted 옵션을 제공하지 않느냐는 점입니다. 제공은 되지만 별도 엔터프라이즈 라이선스가 필요하고, 가격과 도입 절차가 호스티드와는 결이 다릅니다. 반면 Langfuse는 같은 코드가 오픈소스로 공개되어 있어 라이선스 협의 없이 PoC 단계부터 자체 호스팅으로 시작할 수 있다는 점이 한국 기업에게는 진입 장벽을 낮춰 주는 요소가 됩니다. 비교 관점을 빌리면, LangSmith는 LangChain·LangGraph 사용 팀에게 가장 매끄럽게 붙는 호스티드 우선 도구이고, Langfuse는 데이터 거주성 요구에서 출발해 자체 호스팅을 일급 시민으로 다루는 도구라고 정리할 수 있습니다.

정리하면, Langfuse는 LangSmith가 약한 자리, 즉 trace 데이터를 외부 클라우드에 두기 어려운 환경을 정면으로 풀기 위해 설계된 오픈소스 관측 플랫폼이며, Postgres와 ClickHouse라는 두 백엔드 위에 Web과 Worker를 얹어 Docker Compose나 Kubernetes로 사내에 그대로 띄울 수 있도록 짜여 있습니다. 이 구조 덕분에 한국과 유럽처럼 데이터 주권과 거주성 요구가 강한 환경에서도 trace 본문을 사내에 묶어 둔 채 LLM 관측을 도입할 수 있고, 도입 의사결정은 운영 부담 vs 데이터 거주성이라는 한 쌍의 트레이드오프 위에서 갈리게 됩니다.

다음 단원인 3.4.2에서는 같은 Langfuse가 자체 호스팅에 더해 OpenTelemetry라는 업계 표준을 어떻게 받아들이고 있는지, 그것이 다중 SDK와 다른 관측 도구와의 연동, 그리고 벤더 락인 회피에 어떤 영향을 미치는지를 살펴보겠습니다.

이 단원을 마치면 오픈소스 Langfuse의 구조와 Postgres·ClickHouse 두 백엔드의 역할, Docker와 Kubernetes를 이용한 자체 호스팅 배포 패턴, 그리고 한국·유럽의 데이터 거주성 요구 위에서 자체 호스팅이 가진 데이터 주권 측면의 이점을 운영 부담과 함께 설명할 수 있습니다.
