# 2.5.1 Eval 환경 isolation

앞 단원까지는 골든 데이터셋, LLM-as-Judge, trajectory 평가, 그리고 τ-bench·AgentBench·SWE-bench 같은 학술 벤치마크를 다루었습니다. 이 모든 도구는 '무엇을 측정할 것인가'에 대한 답이었습니다. 이번 단원과 다음 단원에서는 한 발 물러서서 '측정하는 무대 자체가 흔들리지 않으려면 어떻게 해야 하는가'를 다룹니다. 같은 데이터셋과 같은 채점기를 쓰더라도 평가가 돌아가는 환경이 trial마다 조금씩 다르면, 그 위에서 나오는 점수는 시스템의 진짜 능력이 아니라 환경의 우연을 같이 재고 있게 됩니다. 이 단원의 주제는 그 우연을 줄이는 첫 번째 원칙, **eval 환경 isolation**입니다.

먼저 용어부터 풀겠습니다. 평가에서 한 번의 실행 단위를 보통 **trial**이라고 부릅니다. 같은 케이스를 여러 번 돌려 평균을 낼 때 그 한 번 한 번이 trial이고, 케이스가 여러 개라면 케이스마다 trial이 따로 잡힙니다. **Isolation**은 분리·격리라는 뜻으로, 한 trial과 다른 trial 사이에 데이터·파일·프로세스·캐시가 서로 영향을 주고받지 않도록 끊어 두는 상태를 가리킵니다. 즉 '평가 환경 isolation'이란 각 trial이 서로 간섭하지 않도록 따로 떼어 둔 환경에서 돌아가는 상태를 말합니다.

이 원칙의 핵심에는 **clean state 시작**이라는 요구가 있습니다. Trial이 시작될 때 환경에는 그 trial에 필요한 것만 있고, 이전 trial이 만들어 둔 흔적은 하나도 남아 있지 않아야 한다는 뜻입니다. 흔적이라 함은 작성된 파일, 변경된 환경 변수, 켜져 있는 백그라운드 프로세스, 채워진 메모리 캐시, 누적된 토큰 사용량 같은 것을 모두 포함합니다. 한 trial을 끝내고 다음 trial을 돌리기 전에 이 모든 것을 깨끗하게 되돌려 두지 않으면, 다음 trial은 자기 케이스만 풀고 있는 것이 아니라 앞 trial이 남긴 부산물 위에서 푸는 셈이 됩니다.

왜 이게 문제가 될까요. 단순한 예로 코드 작성 에이전트를 평가한다고 해 봅시다. 첫 번째 trial에서 에이전트가 `utils.py`라는 파일을 잘못된 내용으로 만들어 두었다고 하겠습니다. 다음 trial은 다른 케이스를 받지만, 그 케이스를 푸는 과정에서 에이전트가 우연히 `utils.py`를 import해 보고는 '이미 비슷한 함수가 있네요'라고 판단해 자기 작업을 줄이거나, 반대로 그 잘못된 파일 때문에 임포트가 깨져서 실패합니다. 두 trial이 서로 다른 케이스를 풀고 있다는 약속이 어긋나 버린 것입니다. 한쪽 trial은 점수가 부풀려지고, 다른 쪽 trial은 점수가 깎이며, 그 모두가 시스템의 실력과는 무관한 결과입니다.

이 문제를 막으려면 '환경을 만들고 - trial을 한 번 돌리고 - 환경을 부수고 - 다시 만들고'라는 사이클이 필요합니다. 이를 흔히 **containerized 평가 환경**으로 구현합니다. 컨테이너는 운영체제 위에 별도의 격리된 사용자 공간을 만드는 기술이며, 한 컨테이너 안에서 일어난 파일 변경·프로세스 생성·네트워크 사용은 컨테이너를 폐기하는 순간 같이 사라집니다. Trial마다 새 컨테이너를 만들어 던져 주면, trial이 끝난 뒤 그 컨테이너를 통째로 버리는 것만으로 자동으로 clean state가 보장됩니다.

전형적인 isolation 구조를 그림으로 보겠습니다.

```
              [ 평가 오케스트레이터 ]
                       |
       ----------------+----------------
       |               |               |
 [ trial 1 ]      [ trial 2 ]     [ trial 3 ]
 컨테이너 #A      컨테이너 #B      컨테이너 #C
   |                 |                 |
 케이스 풀이       케이스 풀이       케이스 풀이
   |                 |                 |
   v                 v                 v
 [ 채점 ] ----- [ 채점 ] ----- [ 채점 ]
   |                 |                 |
 컨테이너 폐기   컨테이너 폐기    컨테이너 폐기
```

오케스트레이터는 trial마다 새 컨테이너를 띄우고, 그 안에서 에이전트를 돌리고, 결과를 채점기로 넘긴 뒤 컨테이너를 버립니다. 컨테이너 #A에서 만들어진 파일은 컨테이너 #B로 새지 않고, #C가 시작될 때는 다시 같은 출발선에서 출발합니다. 이 구조를 갖추는 것만으로 trial 사이의 간섭은 대부분 차단됩니다.

또 한 가지 중요한 요구가 **프로덕션과 동일한 동작 보장**입니다. 평가가 너무 가벼워서 프로덕션과 다른 라이브러리 버전, 다른 도구 정의, 다른 권한 체계 위에서 돌아간다면, 평가에서 잘 돌던 에이전트가 프로덕션에서 다른 모습을 보이게 됩니다. 평가 환경 isolation은 단순히 trial 사이를 떼어 놓는 데서 그치지 않고, 그 떼어 놓은 한 칸 한 칸이 프로덕션의 한 요청을 그대로 모방하는 형태여야 의미가 큽니다. 프로덕션이 컨테이너를 쓴다면 평가도 같은 베이스 이미지를 쓰고, 프로덕션이 특정 OS 패키지를 깔고 시작한다면 평가도 똑같이 깔고 시작하는 식입니다.

여기서 한 가지 짚어야 할 개념이 **환경 노이즈가 메트릭에 미치는 영향**입니다. 환경 노이즈란 시스템 본질과 무관한, 환경 쪽에서 끼어드는 잡음을 말합니다. 예를 들어 평가 호스트가 다른 작업 때문에 일시적으로 CPU가 모자라면 도구 호출이 타임아웃을 더 자주 내고, 디스크가 다른 trial의 잔재로 가득 차면 임시 파일 쓰기가 실패합니다. 이 모든 것은 에이전트의 능력과 무관한 외생적 요인이지만, 점수에는 그대로 반영됩니다. Isolation이 잘 되어 있으면 한 trial이 만들어 낸 노이즈가 다른 trial로 번지지 않으므로, 같은 에이전트를 같은 환경에서 두 번 돌렸을 때 결과가 환경 때문에 갈라지는 일을 줄일 수 있습니다.

이런 원칙을 가장 명시적으로 정리한 사례가 Anthropic의 평가 가이드입니다. 핵심은 두 줄로 정리됩니다. 첫째, 평가의 에이전트는 프로덕션의 에이전트와 거의 동일하게 작동해야 한다. 둘째, 환경 자체가 추가 노이즈를 만들지 말아야 하며, 각 trial은 clean environment에서 시작해 isolated되어야 한다. 이 두 줄은 단순해 보이지만, 평가 결과가 시스템의 실력만을 잰다는 약속을 지키기 위한 최소 조건입니다. 이 조건이 무너지면 그 위에서 비교하는 모델 A와 모델 B의 점수 차는 두 모델의 차이가 아니라 두 환경의 차이를 잴 가능성이 커집니다.

마지막으로 isolation의 비용도 함께 봐 둘 필요가 있습니다. Trial마다 컨테이너를 새로 띄우는 일은 결코 공짜가 아닙니다. 컨테이너 시작 시간, 베이스 이미지 다운로드, 의존성 설치 같은 비용이 trial마다 더해지므로 평가 전체의 소요 시간이 늘어납니다. 그래서 현실에서는 베이스 이미지를 미리 캐시해 두고 trial마다 그 위에 얇은 레이어만 새로 올리는 식의 최적화를 함께 쓰는데, 어디까지 최적화할지의 기준은 단순합니다. '캐시한 부분이 trial 사이에 어떤 식으로든 결과에 영향을 줄 수 있는가'입니다. 영향이 없는 부분만 캐시하고, 조금이라도 영향이 있는 부분은 매번 다시 만드는 편이 안전합니다.

정리하면, eval 환경 isolation은 trial마다 clean state에서 시작하도록 환경을 분리해 두고, 그 분리된 한 칸이 프로덕션과 동일한 동작을 보장하도록 맞추며, 환경 노이즈가 메트릭에 끼어드는 통로를 컨테이너 단위로 끊어 내는 원칙입니다. 이 원칙을 지키면 평가 점수가 시스템의 실력만을 재게 되고, 그렇지 못하면 점수는 환경의 우연까지 같이 재기 시작합니다.

다음 단원인 2.5.2에서는 이 isolation이 무너졌을 때 무엇이 어디까지 잘못될 수 있는지를 살펴봅니다. 남은 파일·캐시·git history가 다음 trial에 어떻게 새는지, shared state가 점수를 어떻게 부풀리는지, 그리고 Anthropic 내부에서 실제로 발견된 git history 누설 사례를 따라가며 재현 가능성을 보장하는 체크리스트를 정리하겠습니다.

이 단원을 마치면 trial과 isolation의 정의에서 출발해, clean state 시작·프로덕션과 동일한 동작 보장·containerized 평가 환경·환경 노이즈가 메트릭에 미치는 영향·Anthropic의 isolated 환경 원칙을 각각 설명하고, 평가 환경을 trial 사이에 isolated로 유지해야 하는 이유를 정리할 수 있습니다.
