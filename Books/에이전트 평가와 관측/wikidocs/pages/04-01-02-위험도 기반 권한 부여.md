# 4.1.2 위험도 기반 권한 부여

앞 단원에서 본 네 단계 사다리는 에이전트 전체에 한 번 부여하고 끝낼 수 있는 라벨처럼 보이기 쉽지만, 현장에서 그렇게 운영되는 경우는 거의 없습니다. 같은 에이전트가 한쪽에서는 안전하게 행동하고 다른 쪽에서는 큰 사고를 칠 수 있기 때문입니다. 데이터베이스 조회와 결제 처리는 모두 '도구 호출'이라는 한 단어로 묶이지만, 실제로 일어나는 일과 잘못되었을 때의 결과는 전혀 다릅니다. 이 단원에서는 에이전트의 자율성 단계를 행동 단위로 잘게 쪼개고, 행동의 **위험도**에 따라 권한을 차등 부여하는 정책을 어떻게 짜는지를 다룹니다.

먼저 위험도가 무엇으로 구성되는지부터 풀겠습니다. 위험도는 한 가지 척도로 표현되지 않고, 적어도 세 축의 합으로 봐야 정확합니다. 첫째는 **데이터 변경**입니다. 어떤 행동이 시스템 안의 값을 바꾸는지, 바꾼다면 얼마나 많이 바꾸는지가 여기 들어갑니다. 한 행을 수정하는 일과 한 테이블을 일괄 갱신하는 일은 같은 update 권한 안에서도 위험의 크기가 다릅니다. 둘째는 **외부 호출**입니다. 사내에서만 끝나는 행동인지, 외부 결제 처리기나 외부 SaaS의 상태를 바꾸는 행동인지를 가르는 축입니다. 외부 호출은 일단 나가면 사내 trace로 되돌리기 어렵고, 상대 시스템의 정책에 따라 사후 처리 비용이 크게 달라집니다. 셋째는 **비용**입니다. 한 번의 호출이 얼마의 토큰과 얼마의 도구 사용료를 쓰는지, 그리고 잘못 부풀려졌을 때 얼마까지 새는지가 여기 들어갑니다. 비용은 결과의 옳고 그름과 별개로 회사가 지는 짐이기 때문에 위험의 한 축으로 따로 떼어 둡니다.

위험도를 좀 더 구조화해 표현할 때 자주 쓰이는 도구가 위험도 매트릭스입니다. 두 축으로 그리는데, 가로축에는 그 행동이 만들어 내는 영향의 크기인 **impact**를 두고, 세로축에는 잘못되었을 때 되돌릴 수 있는 정도인 **reversibility**를 둡니다. 그림으로 보면 다음과 같습니다.

```
                되돌리기 어려움 (낮은 reversibility)
                 ▲
                 |
   [B] 큰 사고     |   [D] 위험 최고: 자율 금지, 강한 사전 검증
                 |
   ----------------+----------------> impact (영향 크기)
                 |
   [A] 안전        |   [C] 사고 잦음: Suggest 이상 권장
                 |
                 ▼
                되돌리기 쉬움 (높은 reversibility)
```

A 사분면은 영향이 작고 되돌리기도 쉬운 영역으로, 조회나 캐시 갱신 같은 행동이 여기 모입니다. Read-only 또는 Execute로 자유롭게 두어도 별 문제가 없습니다. C 사분면은 영향이 크지만 되돌리기는 어렵지 않은 영역으로, 사내 색인의 일괄 재구축 같은 작업이 여기 듭니다. 잘못되어도 수동으로 다시 빌드하면 되지만 한 번의 사고가 사용자 경험에 큰 흔적을 남기므로 Suggest 이상으로 다루는 편이 안전합니다. B 사분면은 영향은 작지만 되돌리기 어려운 영역으로, 외부 사용자에게 발송된 메일이나 결제 같은 행동이 여기 들어옵니다. 한 건이 작아 보여도 수신자에게 가닿은 뒤에는 회수가 안 됩니다. D 사분면은 두 축 모두 나쁜 영역으로, 외부 송금이나 운영 DB의 스키마 변경 같은 행동이 모입니다. 이 영역은 자율 단계에서 자동 실행을 허용하지 않고 강한 사전 검증과 사람 승인을 의무화하는 것이 일반적입니다.

여기서 한 가지 빠지기 쉬운 함정이 있습니다. 위험도는 행동의 종류만으로 결정되지 않고 컨텍스트에 따라 달라집니다. 같은 'send_email'이라는 도구도 사내 한 명에게 보내는 알림과 사용자 1만 명에게 발송하는 마케팅 메일은 위험이 다릅니다. 같은 'update_record'도 본인이 작성한 임시 메모를 고치는 일과 다른 사람의 회계 자료를 고치는 일이 다릅니다. 그래서 권한은 도구 단위로만 매기지 않고, 도구를 부르는 인자, 호출자의 역할, 영향 받는 객체의 소유주를 함께 보고 결정합니다. 정책은 보통 다음과 같은 형태로 표현됩니다.

```
 if tool == "send_email" and recipient_count <= 1 and recipient_in_company:
     단계 = Execute
 elif tool == "send_email" and recipient_count <= 100:
     단계 = Suggest
 else:
     단계 = HITL 강제
```

이런 분기가 누적되면 정책이 빠르게 복잡해지므로, 도구 카탈로그 옆에 '위험 클래스'라는 이름표를 붙여 두고 클래스 단위로 단계를 매핑하는 편이 관리하기 좋습니다.

같은 에이전트의 단계가 변하기도 합니다. 운영 도중 문제가 누적되면 자동으로 단계를 한 칸 내리고, 안정성이 회복되면 다시 올리는 흐름이 필요합니다. 이를 위한 장치가 **단계 전환 트리거**입니다. 트리거는 보통 메트릭의 임계치 도달로 정의됩니다. 예컨대 최근 24시간 동안 도구 호출 실패율이 5%를 넘으면 해당 도구를 Execute에서 Suggest로 자동 강등하고, 7일 동안 1% 미만으로 유지되면 다시 Execute로 자동 승격합니다. 이런 자동 강등은 사람이 모든 사고를 일일이 봐 주지 않아도 시스템이 스스로 위험한 행동을 막아 줄 수 있게 해 줍니다. 실패율 외에도 비용 폭주, 사용자 불만 신호, judge 점수 하락 같은 메트릭이 트리거 후보로 자주 쓰입니다. 한 흐름의 그림은 다음과 같습니다.

```
   메트릭 임계치 위반
         |
         v
 +-------------------+
 |  자동 단계 강등    |  Execute -> Suggest, Suggest -> Read-only
 +-------------------+
         |
         v
   감사 로그에 기록
         |
         v
 +-------------------+
 |  사람의 회귀 검토  |  원인 분석, 평가 케이스 추가
 +-------------------+
         |
         v
 일정 기간 안정 -> 자동 단계 승격
```

자동 강등이 동작했다는 사실 자체는 좋은 소식이지만, 그 결과를 사람이 들여다보지 않으면 의미가 절반밖에 살지 않습니다. 그래서 단계 전환은 항상 **감사 로그**에 남깁니다. 어떤 행동이 어떤 임계치를 넘겨 어떤 단계로 옮겨 갔는지, 누구의 정책이 발화했는지를 추적할 수 있게 해 두면 사고 후 회귀 검토가 가능합니다. 이 회귀 검토에서는 자동 강등이 잘못 발화한 경우를 찾고, 같은 패턴을 잡아 내는 평가 케이스를 데이터셋에 흡수합니다. 회귀 검토의 결과는 다시 다음 단계의 정책을 다듬는 데 쓰이고, 임계치도 점점 도메인에 맞게 조정됩니다.

마지막으로 강조할 것은 권한이 코드 안에서만 결정되지 않는다는 점입니다. 위험도 매트릭스, 단계 전환 트리거, 감사 로그는 모두 보안 팀, 운영 팀, 도메인 책임자의 합의 위에서 의미를 가집니다. 한국의 금융권이나 의료처럼 규제가 강한 환경에서는 단계 전환 트리거가 발화한 사실 자체를 정기 보고에 포함해야 하는 경우가 많고, 자동 승격은 사람의 명시적 승인을 거쳐야만 가능하도록 잠가 두기도 합니다. 어디까지 자동화하고 어디부터 사람의 결재를 받게 할지는 위험도와 조직 문화의 함수입니다.

정리하면, 위험도 기반 권한 부여는 데이터 변경, 외부 호출, 비용 세 축으로 위험을 분해한 뒤 impact와 reversibility의 매트릭스 위에 행동을 배치하고, 도구만이 아니라 호출 인자와 컨텍스트까지 함께 봐서 단계를 결정하며, 메트릭 임계치에 따라 자동 강등과 자동 승격이 일어나도록 단계 전환 트리거를 묶고, 모든 전환을 감사 로그로 남겨 사후 회귀 검토에 다시 쓰는 정책 묶음입니다. 이 묶음이 4.1.1의 단계 사다리와 결합될 때 비로소 같은 에이전트가 안전한 행동에서는 자유롭게 움직이고 위험한 행동에서는 단단히 묶이는 차등 자율성이 작동합니다.

다음 단원인 4.2.1에서는 위험도가 누적되어 임계치를 넘는 순간에 시스템 전체를 잠시 멈춰 세우는 안전 장치, 즉 분산 시스템의 circuit breaker 패턴이 에이전트 루프 위에서 어떻게 동작하는지를 살펴보겠습니다.

이 단원을 마치면 위험도를 데이터 변경·외부 호출·비용의 세 축으로 분해하고 impact와 reversibility의 매트릭스로 행동을 분류하며, 동일 작업이라도 컨텍스트에 따라 권한을 차등 부여하는 정책을 설계하고, 메트릭 임계치 기반의 단계 전환 트리거와 감사 로그·회귀 검토 흐름을 연결해 설명할 수 있습니다.
