# 4.3.1 Fallback의 종류

앞 단원까지 우리는 자율성 단계를 차등으로 나누는 패턴과, 위험이 임계치를 넘었을 때 전체 호출을 멈추는 회로 차단기 패턴을 살펴봤습니다. 두 패턴 모두 위험한 상황을 막는 데 무게를 둡니다. 그런데 막은 다음에는 무엇을 해야 할까요. 회로가 열려서 주 모델 호출을 차단했는데도 사용자는 여전히 답을 기다리고 있고, 화면에는 결과가 떠야 합니다. 이때 시스템이 무언가 의미 있는 응답을 돌려주려면, 차단된 경로 대신 쓸 수 있는 대체 경로가 미리 마련되어 있어야 합니다. 이 대체 경로를 가리키는 이름이 **fallback**입니다.

먼저 용어부터 풀겠습니다. 일상에서는 '실패했을 때 기댈 곳'이라는 뜻으로 쓰입니다. 가지고 다니는 신용카드가 결제 단말기에서 거절되면 다른 카드를 꺼내거나 현금으로 결제하는 식입니다. 이때 두 번째 카드와 현금이 결제의 fallback에 해당합니다. 시스템 설계에서도 같은 비유가 그대로 쓰입니다. 주 경로가 실패하거나 사용할 수 없을 때, 미리 정해 둔 대체 경로로 흐름을 돌려 서비스가 멈추지 않게 합니다. 에이전트 시스템에서 fallback은 보통 세 층위에 걸쳐 설계됩니다. 모델 층위, 도구 층위, 그리고 응답 층위입니다.

첫째는 **모델 fallback**입니다. 에이전트가 호출하는 주 모델이 응답하지 않거나 한도를 넘겼을 때, 다른 모델로 흐름을 돌리는 패턴을 가리킵니다. 모델 fallback의 트리거는 보통 세 가지로 나뉩니다. 가용성, 비용, 품질입니다. 가용성은 사업자의 장애로 일정 시간 응답이 끊기거나, rate limit에 걸려 호출 자체가 거부되는 경우에 발동됩니다. 비용은 주 모델 단가가 너무 높아 특정 요청에는 더 싼 모델로 돌리는 경우입니다. 품질은 정반대 방향으로, 짧고 단순한 요청에는 작은 모델을 쓰다가 복잡한 요청에서 신뢰도가 임계 아래로 떨어지면 더 큰 모델을 호출하는 경우입니다. 세 트리거 모두 같은 fallback 메커니즘을 공유하지만, 어떤 신호로 발동되느냐가 다릅니다.

둘째는 **도구 fallback**입니다. 에이전트가 외부 검색 API, 데이터베이스 쿼리, 사내 RAG 인덱스 같은 도구를 호출할 때, 주 도구가 실패하면 보조 도구로 옮겨 가는 패턴입니다. 예를 들어 사내 지식 검색을 일차로 시도하고, 결과가 비어 있으면 웹 검색을 시도하고, 그것도 실패하면 캐시에 저장된 최근 답변을 돌려주는 식입니다. 도구 fallback에서 중요한 것은 같은 작업을 수행하지만 신뢰성과 비용이 다른 도구들을 미리 등급별로 묶어 두는 일입니다. 1차, 2차, 3차로 정해 둔 사다리를 따라 차례로 시도해야 흐름이 일관됩니다.

셋째는 **응답 fallback**입니다. 모델도 도구도 모두 답을 돌려주지 못했을 때 마지막으로 사용자에게 보여 줄 응답을 만드는 층위입니다. 가장 단순한 형태는 미리 작성해 둔 **템플릿** 메시지입니다. '지금은 답변을 드리기 어렵습니다. 잠시 후 다시 시도해 주세요'와 같은 정형 문구가 여기 들어갑니다. 그보다 한 단계 정교한 형태는 **휴리스틱**, 즉 모델 호출 없이 규칙으로 만드는 응답입니다. 사용자의 질문에서 키워드를 뽑아 가장 자주 묻는 답변 모음에서 매칭되는 항목을 보여 주거나, 마지막 성공 응답을 캐시에서 꺼내 보여 주는 식입니다. 응답 fallback은 품질이 높지는 않지만, 사용자가 빈 화면을 보지 않게 한다는 한 가지 목적에 충실합니다.

세 층위를 그림으로 정리하면 다음과 같이 보입니다. 사용자의 한 요청이 들어오면 시스템은 위에서부터 차례로 시도하다가, 어느 한 곳에서 답이 만들어지면 거기서 멈추고 응답을 돌려줍니다.

```
[사용자 요청]
    |
    v
[모델 fallback]   주 모델 -> 보조 모델 -> 더 작은 모델
    | (모두 실패)
    v
[도구 fallback]   1차 도구 -> 2차 도구 -> 캐시
    | (모두 실패)
    v
[응답 fallback]   휴리스틱 답변 -> 템플릿 메시지
    |
    v
[최종 응답]
```

이렇게 여러 층위와 여러 단계를 차례로 묶은 구조를 **fallback chain**이라고 부릅니다. 이름 그대로 시도가 사슬처럼 이어집니다. 사슬은 든든해 보이지만, 길어질수록 두 가지 비용이 누적됩니다. 첫째는 **지연**입니다. 1차에서 시간 초과가 나야 2차로 넘어가고, 2차에서도 같은 패턴이 반복되면 사용자가 느끼는 응답 시간은 단계마다 더해집니다. 둘째는 **품질 저하**입니다. 사슬이 깊이 내려갈수록 응답은 점점 정형화되거나 정보량이 줄어듭니다. 마지막에 닿으면 결국 템플릿 한 줄에 머물게 됩니다. 따라서 fallback chain은 무한정 늘릴수록 좋은 것이 아니라, 어디까지 내려갈지를 선택해서 끊어 두는 설계가 필요합니다. 차단해야 할 시점의 임계치는 앞 단원에서 다룬 회로 차단기와 같이, 시간·비용·연속 실패 횟수로 정해 둡니다.

마지막으로 **정적 fallback**과 **동적 fallback**의 차이를 짚겠습니다. 정적 fallback은 모든 경로와 순서를 사전에 고정해 두는 방식입니다. 1차는 무엇, 2차는 무엇, 그다음은 무엇이라고 코드나 설정에 적혀 있고 런타임에 바뀌지 않습니다. 구현이 단순하고 동작이 예측 가능하지만, 상황에 따라 더 좋은 선택이 가능해도 그것을 쓰지 못합니다. 동적 fallback은 런타임에 시그널을 보고 다음 경로를 결정합니다. 예를 들어 특정 사업자의 응답 지연이 평소보다 길어지면 그 사업자를 자동으로 후순위로 내리고, 비용 한도가 임박하면 더 싼 모델을 우선으로 끌어올리는 식입니다. 동적 fallback은 유연하지만 결정 자체가 또 하나의 시스템이 되어, 어떤 신호가 어떤 결정을 만들었는지 관측하지 못하면 디버깅이 어렵습니다. 다음 단원에서 다룰 모델 라우팅 전략은 사실상 이 동적 fallback을 더 일반화한 형태로 볼 수 있습니다.

정리하면, fallback은 주 경로가 실패했을 때 시스템이 빈 응답을 돌려주지 않도록 미리 마련해 둔 대체 경로이며, 모델 fallback은 가용성·비용·품질 신호로 다른 모델로 옮겨 가는 패턴, 도구 fallback은 같은 작업을 수행하는 보조 도구로 옮겨 가는 패턴, 응답 fallback은 템플릿과 휴리스틱으로 마지막 선을 지키는 패턴입니다. 이들을 차례로 엮은 fallback chain은 길어질수록 지연과 품질 저하를 누적시키므로, 어디서 끊을지를 임계치로 미리 정해 두어야 하며, 정적 방식은 단순하고 예측 가능하지만 동적 방식은 런타임 신호로 더 나은 경로를 고르는 대신 결정 자체를 관측해야 한다는 비용을 지불합니다.

다음 단원인 4.3.2에서는 이 동적 fallback을 한 단계 더 일반화한 모델 라우팅 전략을 살펴봅니다. 단일 API 뒤에 여러 사업자의 모델을 두고 task type, cost, accuracy를 축으로 매 요청마다 모델을 고르는 구조를 JPMorgan 사례와 함께 풀어 보겠습니다.

이 단원을 마치면 에이전트 시스템의 fallback을 모델, 도구, 응답이라는 세 층위로 분류하고, 각 층위에서 어떤 트리거로 어떤 대체 경로가 발동되는지를 설명할 수 있으며, fallback chain을 무한정 늘릴 때 발생하는 한계와 정적·동적 fallback의 차이를 비교할 수 있습니다.
