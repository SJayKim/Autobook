# 5.5.3 Memory Governance 차원

앞 단원에서 본 stateless runtime은 자기 안에 상태를 들고 있지 않다고 정해 두었습니다. 그러나 에이전트가 사용자에게 매번 새 사람처럼 굴어서는 쓸모가 떨어집니다. 같은 사용자가 어제 묻고 오늘 다시 묻는 질문에는 어제 무슨 답을 받았는지를 기억해야 자연스러운 흐름이 만들어지고, 한 회사 안의 지식 베이스에는 직원마다 접근 가능한 범위가 달라야 합니다. 그래서 상태는 runtime 바깥의 별도 저장소로 옮겨 가지만, 그 저장소를 누가 어떤 권한으로 읽고 쓰는지 그대로 두면 곧장 사고가 납니다. 이 자리를 통제하는 세 번째 차원이 **Memory Governance**입니다. 이번 단원은 에이전트가 들고 있는 세 종류의 기억을 분류하고, 그 위에 JWT 기반 RBAC가 어떻게 얹히는지를 풀어 보겠습니다.

먼저 에이전트가 들고 있는 기억을 세 층으로 나누는 데서 출발하겠습니다. 첫 층은 **유저 프로필 메모리**입니다. 한 사용자에 대해 장기적으로 유지되는 기록입니다. 사용자의 선호하는 응답 길이, 자주 쓰는 도구, 과거에 만든 프로젝트 목록 같은 것이 여기 들어옵니다. 한 사용자가 일주일을 비웠다 돌아와도 이 층의 기록은 그대로 남아 있습니다. 두 번째 층은 **세션 메모리**입니다. 한 번의 대화나 한 번의 작업 단위 안에서만 유지되는 기록입니다. 지금 쓰는 보고서의 초안, 방금 검색한 결과, 이번 단계에서 정한 계획 같은 것이 여기 들어옵니다. 세션이 끝나면 보통 폐기되거나 압축되어 다른 곳으로 옮겨집니다. 세 번째 층은 **지식 베이스**입니다. 회사 전체나 팀 단위로 공유되는 문서·정책·과거 사례 같은 자료를 모아 둔 자리입니다. 사용자별이 아니라 조직 단위로 관리됩니다.

세 층은 같은 '메모리'라는 단어로 묶이지만 통제의 결이 다릅니다. 유저 프로필 메모리는 개인정보의 영역이라 사용자 본인 외에는 함부로 못 보게 막아야 하고, 세션 메모리는 너무 오래 살아남지 않도록 만료를 잘 걸어 두는 일이 핵심이며, 지식 베이스는 조직 안의 직급·부서·프로젝트별로 누가 어디까지 볼 수 있는지를 표로 그려 둔 권한 체계가 핵심입니다. 한 차원으로 묶이는 까닭은 이 세 층 모두가 같은 통제 도구 위에 얹히기 때문입니다. 그 도구가 바로 JWT 기반 RBAC입니다.

여기서 두 단어를 풀고 가겠습니다. **JWT**는 JSON Web Token의 약자로, 사용자가 누구이고 어떤 권한을 가지고 있는지를 한 줄의 토큰 안에 담아 들고 다니게 만드는 형식입니다. 사용자가 한 번 로그인하면 인증 서버가 이 토큰을 발급하고, 이후의 모든 요청에 토큰이 함께 실려 다닙니다. 토큰 안에는 사용자 식별자, 소속 팀, 부여된 역할, 만료 시각 같은 정보가 담깁니다. 토큰은 위조를 막는 서명이 붙어 있어, 받는 쪽은 서명만 검증해도 토큰의 내용이 진짜인지 알 수 있습니다.

**RBAC**는 Role-Based Access Control, 즉 역할 기반 접근 통제입니다. 사용자에게 직접 권한을 붙이지 않고, '편집자'·'열람자'·'관리자' 같은 역할에 권한 묶음을 정의해 둔 뒤, 사용자에게는 역할만 부여합니다. 새 사람이 들어오면 적절한 역할을 주면 끝이고, 역할의 권한을 바꾸면 그 역할을 가진 모든 사람의 권한이 한 번에 바뀝니다. 이렇게 만들어 두면 권한 관리가 사람 수만큼 폭발하는 일을 막을 수 있습니다.

JWT와 RBAC가 만나면 다음과 같은 흐름이 만들어집니다.

```
 [사용자 로그인]
       |
       v
 [인증 서버] ----> JWT 발급 (사용자ID, 역할, 만료시각)
       |
       v
 [에이전트 요청] ---- 토큰 동봉 ---->
                                      [Memory 게이트]
                                       |  토큰 검증
                                       |  역할 확인
                                       v
                          +------------+------------+
                          |            |            |
                       프로필       세션         지식 베이스
                       (사용자       (세션ID      (역할별
                        본인만)       격리)        접근권)
```

이 그림의 모든 화살표 위에서 토큰의 정보가 함께 흐릅니다. 에이전트가 유저 프로필 메모리를 읽으려 하면, 게이트는 토큰의 사용자 식별자가 그 프로필의 주인과 일치하는지 확인합니다. 다른 사람의 프로필을 부르려는 호출은 막힙니다. 세션 메모리를 읽으려 하면, 게이트는 토큰이 그 세션의 소유 사용자와 묶여 있는지를 확인합니다. 지식 베이스를 읽으려 하면, 게이트는 토큰의 역할이 그 자료의 접근 권한 표에 등록되어 있는지를 확인합니다. 같은 에이전트가 같은 도구를 부르더라도 토큰에 따라 응답이 달라지는 셈입니다.

이 차원이 통제하는 구체 항목 네 가지를 정리하겠습니다. 첫째, 모든 메모리 접근은 토큰 검증을 거쳐야 하며, 토큰 없이 도는 호출은 곧장 거부됩니다. 둘째, 세션 메모리는 만료 시간이 명시되어 있고, 만료가 지나면 자동으로 사라지거나 압축본만 남깁니다. 셋째, 지식 베이스의 자료는 자료 단위로 접근 권한 표가 붙고, 표가 없는 자료는 노출되지 않습니다. 넷째, 누가 언제 어떤 메모리를 읽거나 썼는지가 별도의 감사 로그에 남고, 사용자가 자기 프로필 데이터의 삭제를 요청하면 그 흐름을 따라 일괄 삭제할 수 있어야 합니다.

마지막 항목인 메모리 감사와 삭제는 한국 규제 환경에서 특히 무게가 실립니다. 개인정보보호법은 정보 주체의 삭제 요청에 대해 일정 기한 안에 응답하도록 정하고 있어, 사용자가 '내 데이터를 지워 달라'고 했을 때 시스템이 한 곳을 못 찾으면 위반이 됩니다. 메모리가 세 층으로 나뉘어 있고 각 층이 토큰의 사용자 식별자에 묶여 있으면, 식별자 하나로 세 층을 훑어 일괄 삭제하는 절차가 단순해집니다. 망분리 환경의 금융권은 더 나아가, 메모리가 사외 클라우드로 빠져나가지 않고 사내 저장소 안에서만 머물러야 한다는 조건이 추가됩니다. 공공 분야는 어떤 직급·부서가 어떤 자료를 봤는지가 모두 감사 로그로 남아야 하고, 그 로그를 일정 기간 보관해야 합니다.

지식 베이스 접근 통제와 관련해 한 장면을 더 그려 두겠습니다. 한 회사 안에 영업팀의 거래처 목록과 인사팀의 평가 자료가 같은 지식 베이스 시스템에 들어 있다고 합시다. 에이전트는 한 명이지만, 영업 사원이 토큰을 들고 들어오면 거래처 목록만 보이고 평가 자료는 보이지 않아야 합니다. 인사팀 직원이 들어오면 반대가 됩니다. 같은 질문을 두 사람이 던져도 에이전트의 답이 다르게 나옵니다. 이 차이를 만들어 내는 자리가 토큰의 역할 정보이고, 그 정보를 받아 자료를 잘라 내는 자리가 Memory 게이트입니다. 사용자가 보기에는 그저 자기 권한 안의 답을 받은 셈이지만, 시스템 안에서는 같은 도구가 다른 결과를 돌려준 것입니다.

본인의 시스템을 self-audit해 볼 질문 세 가지를 두겠습니다. 첫째, 본인이 운영하는 에이전트는 사용자 토큰을 모든 메모리 호출에 함께 흘려보내고 있습니까, 아니면 한 번 인증 후 토큰 없이 흐릅니까. 둘째, 사용자가 자기 데이터를 지워 달라고 요청했을 때, 본인의 시스템이 세 층의 메모리를 한 식별자로 훑어 일괄 삭제할 수 있습니까. 셋째, 같은 질문을 권한이 다른 두 사용자가 던졌을 때 답이 달라져야 한다는 사실을 코드 수준에서 증명할 수 있습니까. 세 질문 가운데 어느 하나라도 답이 흐릿하면, 그 자리가 정보 유출이 일어나기 가장 쉬운 지점입니다.

정리하면, Memory Governance 차원은 에이전트가 들고 있는 유저 프로필 메모리·세션 메모리·지식 베이스 세 층을 분류하고, 그 위에 JWT 기반 RBAC를 얹어 사용자별·역할별 접근을 통제하며, 만료와 감사와 삭제를 정해진 절차로 처리하는 층입니다. 같은 에이전트가 같은 도구를 부르더라도 토큰에 따라 보이는 자료가 달라지는 것이 이 차원이 만든 효과이고, 한국 규제 환경의 개인정보 삭제 요청과 망분리·감사 요건을 단순하게 만족시키는 토대가 됩니다.

다음 단원인 5.5.4에서는 메모리 위에서 정해진 권한이 실제 행동으로 옮겨질 때, 그 행동을 만들어 내는 도구를 어떻게 통제하는지를 다루겠습니다. Tool governance가 4.1의 tiered autonomy와 4.4의 HITL을 이 framework 위에서 다시 만나는 자리입니다.

이 단원을 마치면 Memory Governance 차원이 분류하는 유저 프로필·세션·지식 베이스 세 층의 차이를 설명하고, JWT와 RBAC가 결합해 토큰을 따라 메모리 접근이 잘리는 흐름을 그려 내며, 메모리 감사와 삭제가 한국 규제 환경의 개인정보 요건과 어떻게 맞물리는지를 본인의 시스템에 옮겨 self-audit할 수 있습니다.
