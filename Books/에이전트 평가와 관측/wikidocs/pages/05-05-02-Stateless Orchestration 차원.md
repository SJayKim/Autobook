# 5.5.2 Stateless Orchestration 차원

앞 단원에서 본 Model Access는 어떤 모델을 누가 어떤 조건으로 부를 수 있는지를 정해 둔 자리였습니다. 그 호출이 실제로 떨어지면, 호출 한 건마다 에이전트의 루프가 한 번 돕니다. 도구를 부르고, 결과를 받아 다음 단계를 정하고, 또 도구를 부르는 과정이 짧게는 몇 번, 길게는 수십 번 이어집니다. 이 루프가 어디에서 도는지를 결정하는 자리가 두 번째 차원인 **Stateless Orchestration**입니다. 이번 단원은 이 runtime이 stateless·수평 확장 가능·request 단위 isolation 세 가지 성질을 함께 갖춰야 하는 까닭을, 한국 규제 환경의 적용 사정과 함께 풀어 보겠습니다.

먼저 단어부터 풀겠습니다. **Stateless**는 한 번의 요청을 처리하는 동안 runtime이 그 요청 안에서만 상태를 가지고, 요청이 끝나면 그 상태를 깨끗이 잊는 성질을 말합니다. 반대말은 stateful입니다. stateful runtime은 한 사용자의 어제 요청 결과를 자기 메모리에 들고 있다가 오늘 요청에 영향을 줍니다. stateful이 본질적으로 나쁜 것은 아니지만, 에이전트 시스템에서는 한 번의 호출이 어디 서버에서 도는지 모르는 환경이 많고, 같은 사용자라도 요청마다 다른 인스턴스에 분배되는 일이 흔합니다. 이런 환경에서 runtime이 자기 안에 상태를 들고 있으면 똑같은 요청이라도 어느 서버를 만났는지에 따라 결과가 달라지고, 이는 1.1.2에서 본 비결정성 위에 또 다른 흔들림을 더하게 됩니다. Palantir 표준이 stateless를 강하게 못 박는 까닭이 여기에 있습니다. 상태가 필요하다면 그 상태는 runtime 바깥의 별도 저장소에 두고, runtime 자체는 매번 빈 상태로 시작해 그 저장소를 읽어 작업하고 다시 빈 상태로 돌아갑니다.

두 번째 성질인 **수평 확장**은 동일한 runtime 인스턴스를 여러 개 띄워 부하를 나눠 처리하는 능력을 말합니다. 트래픽이 늘면 인스턴스를 늘리고, 줄면 줄여서 비용을 아낍니다. 수평 확장이 가능하려면 인스턴스끼리 서로의 상태를 모르는 채 같은 일을 할 수 있어야 합니다. 이 조건을 만족하려면 앞에서 말한 stateless가 전제됩니다. 인스턴스가 자기 안에 상태를 들고 있으면, 같은 사용자의 다음 요청을 다른 인스턴스가 받았을 때 상태를 못 찾아 어긋나고, 부하 분산기는 같은 사용자의 요청을 같은 인스턴스로만 보내야 하는 제약에 묶입니다. 그 순간 수평 확장의 의미가 깎입니다. stateless와 수평 확장은 따로 떨어진 두 가지가 아니라 한 쌍의 짝이라고 보는 편이 맞습니다.

세 번째 성질인 **request 단위 isolation**은 한 요청의 처리가 다른 요청에 영향을 주지 못하게 격리하는 성질을 말합니다. 이 자리는 2.5에서 다룬 평가 환경의 isolation과 같은 줄기 위에 있지만, 평가가 아닌 production runtime에서도 똑같이 중요합니다. 이유는 두 가지입니다. 첫째, 한 요청 안에서 도구가 열어 둔 파일·연결·세션 같은 자원이 다음 요청까지 살아 남으면, 다음 요청은 그 잔재 위에서 의도하지 않은 동작을 합니다. 1.1.2의 '재현 불가능한 실패' 가운데 적지 않은 부분이 이 잔재에서 나옵니다. 둘째, 한 사용자가 만든 결과가 다른 사용자의 처리 공간에 살짝이라도 새 들어가면 곧장 정보 유출입니다. 격리는 신뢰성과 보안을 동시에 떠받치는 토대입니다.

세 가지 성질이 합쳐지면 어떤 모양이 되는지 한 그림으로 묶겠습니다.

```
                  +--------------+
                  | 부하 분산기  |
                  +--------------+
                   /     |     \
                  /      |      \
                 v       v       v
           +-------+ +-------+ +-------+
           | RT #1 | | RT #2 | | RT #3 |    각 인스턴스: 빈 상태로 시작
           +-------+ +-------+ +-------+    요청 처리 후 상태 폐기
                |       |       |
                v       v       v
           [상태 저장소: 세션, 메모리, 캐시 - runtime 바깥]
```

이 그림의 핵심은 runtime 인스턴스 안에 상태가 머물지 않는다는 점, 그래서 인스턴스를 더 띄워도 똑같이 일할 수 있다는 점, 한 요청의 처리 영역이 다른 요청과 독립적으로 시작되고 끝난다는 점입니다.

이 차원의 구체 통제 항목을 정리하면 네 가지입니다. 첫째, 모든 runtime 인스턴스는 빈 상태에서 시작해 빈 상태로 끝나야 하며, 영속 상태는 외부 저장소를 거쳐야 합니다. 둘째, 부하가 늘 때 새 인스턴스를 자동으로 띄울 수 있어야 하고, 인스턴스 사이의 어떤 affinity도 강제하지 않아야 합니다. 셋째, 한 요청이 만드는 작업 디렉터리·임시 파일·열린 연결은 그 요청 안에서 닫히고 정리되어야 합니다. 넷째, 다중 테넌트가 같은 인스턴스 위에서 도는 경우라도 테넌트 사이의 메모리·파일·네트워크 경계가 운영체제 수준으로 분리되어야 합니다.

다중 테넌시라는 단어가 처음 나왔으니 짧게 풀어 두겠습니다. **다중 테넌시**는 같은 인프라 위에 여러 고객·여러 부서의 작업을 동시에 올리는 구성을 말합니다. 비용 효율은 좋지만, 한 테넌트의 사고가 다른 테넌트로 번질 위험이 함께 따라옵니다. request 단위 isolation은 이 위험을 막는 첫 줄의 방어선입니다.

장애 격리도 같은 줄기에 있습니다. 한 요청을 처리하던 인스턴스가 메모리 누수나 도구 무한 호출 같은 사정으로 무너졌을 때, stateless 구성에서는 그 인스턴스 한 대만 부하 분산기에서 빠지고 나머지 인스턴스가 부하를 받아 갑니다. stateful이라면 그 인스턴스가 들고 있던 사용자별 상태가 통째로 사라지고, 같은 사용자의 다음 요청은 다른 인스턴스에서 처음부터 다시 시작해야 하므로 사용자가 곧장 체감하는 장애가 됩니다. 4.2의 circuit breaker가 한 도구나 한 모델 단위에서 막는 차단이라면, 이 차원의 장애 격리는 인스턴스 단위에서 작동하는 더 거친 차단입니다.

한국 규제 환경에서의 적용을 살펴보면, 망분리가 적용된 금융권은 외부 클라우드의 자동 확장 기능을 그대로 쓰지 못하는 경우가 많습니다. 그래서 사내 컨테이너 오케스트레이션 위에서 인스턴스 풀을 일정 크기로 미리 띄워 두고, 사용량 예측에 따라 풀의 크기를 바꾸는 방식으로 수평 확장을 구현합니다. 공공 분야는 클라우드 보안인증을 받은 인프라 안에서만 자동 확장을 쓸 수 있고, 인스턴스가 어떤 데이터센터·어떤 노드에 떠 있는지를 감사 로그에 남깁니다. 일반 기업이라도 개인정보를 다루는 앱은 한 사용자의 처리 영역이 다음 사용자에게 새지 않는다는 사실을 보안 점검에서 증명해야 합니다. stateless·수평 확장·request 단위 isolation은 이 증명을 단순하게 만들어 줍니다.

본인의 시스템을 self-audit해 볼 질문 두 가지를 두겠습니다. 첫째, 본인의 runtime을 두 배로 늘렸을 때 단순 설정 변경만으로 가능합니까, 아니면 코드 수정이 따라옵니까. 둘째, 어떤 사용자의 요청을 처리하던 인스턴스가 갑자기 죽었을 때, 같은 사용자의 다음 요청이 다른 인스턴스에서 같은 품질로 처리됩니까. 두 질문에 곧장 '예'라고 답하기 어렵다면, runtime이 stateful이거나 isolation이 약한 자리가 어딘가에 남아 있다는 뜻입니다.

정리하면, Stateless Orchestration 차원은 에이전트 runtime이 자기 안에 상태를 들고 있지 않고, 같은 인스턴스를 여러 개 띄워 부하를 나눌 수 있고, 한 요청의 처리 영역이 다른 요청과 격리되도록 만드는 층입니다. 세 가지 성질은 따로 떨어진 항목이 아니라 한 묶음으로 작동하며, 한국의 금융권·공공·일반 기업 어디서든 보안 점검과 장애 대응을 단순하게 만들어 주는 토대가 됩니다.

다음 단원인 5.5.3에서는 이 stateless runtime 바깥에 놓이는 상태, 즉 사용자 프로필과 세션 메모리와 지식 베이스를 어떻게 통제하는지를 다루겠습니다. JWT 기반 RBAC가 그 자리의 핵심 도구입니다.

이 단원을 마치면 Stateless Orchestration 차원이 정의하는 stateless runtime·수평 확장·request 단위 isolation 세 성질이 한 묶음으로 작동하는 까닭을 설명하고, 다중 테넌시와 장애 격리가 같은 토대 위에 놓인다는 점을 짚으며, 한국 규제 환경에서 이 차원이 보안 점검과 장애 대응을 단순하게 만드는 자리임을 본인의 시스템에 옮겨 점검할 수 있습니다.
