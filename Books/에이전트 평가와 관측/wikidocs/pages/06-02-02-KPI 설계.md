# 6.2.2 KPI 설계

도입 순서를 정하고 첫 산출물이 나오기 시작하면, 그다음에 따라오는 질문은 무엇을 가지고 시스템의 건강을 판단할 것인가입니다. KPI는 그 판단의 언어이며, 잘못 설계된 KPI는 도입의 방향 자체를 비틀어 놓을 만큼 강한 영향력을 가집니다. 이 단원은 1.3에서 본 단일 메트릭 함정과 Klarna 사례를 다시 가져와, 에이전트 시스템에 적합한 다차원 KPI를 설계하는 방법을 정리합니다.

먼저 KPI라는 용어부터 다시 짚겠습니다. **KPI**는 핵심 성과 지표를 뜻하며, 조직이 자기 시스템의 성공과 실패를 가르는 기준으로 합의한 소수의 숫자를 가리킵니다. 일반적인 메트릭은 수십 개 수백 개로 늘어날 수 있지만, KPI는 의도적으로 작은 집합으로 좁힙니다. 작아야 의사결정의 언어로 쓸 수 있고, 분기마다 점검할 수 있으며, 모든 팀원이 머리에 담을 수 있기 때문입니다.

에이전트 시스템의 KPI는 **효율성, 품질, 안전성 세 축**으로 구성하는 것이 출발점입니다. 효율성은 시스템이 얼마나 빠르고 싸게 일을 끝내는지를 봅니다. 응답 지연, 토큰 사용량, 도구 호출 비용, 단위 작업당 평균 비용 같은 항목이 여기 들어옵니다. 품질은 그 일이 사용자가 원한 결과를 만들어 냈는지를 봅니다. 완수율, 정답률, 사용자 만족도 점수, escalation 비율 같은 항목입니다. 안전성은 시스템이 해서는 안 될 일을 하지 않았는지를 봅니다. 가드레일 트리거 비율, 비가역 행동의 사람 승인 비율, PII 노출 사고 건수, audit log 누락 건수 같은 항목입니다. 세 축을 표로 정리하면 다음과 같습니다.

```
+-----------+---------------------------+---------------------------+
|  축       |  대표 정량 지표           |  대표 정성 지표           |
+-----------+---------------------------+---------------------------+
| 효율성    |  P50 지연, 단위 비용,     |  운영자 체감 응답성,      |
|           |  토큰/요청, 동시 처리량   |  유지보수 부담            |
+-----------+---------------------------+---------------------------+
| 품질      |  완수율, 정답률,          |  hidden quality 신호,     |
|           |  CSAT, escalation 비율    |  공감·안심·상황 인식      |
+-----------+---------------------------+---------------------------+
| 안전성    |  가드레일 트리거 비율,    |  사고 사례의 심각도,      |
|           |  HITL 승인 비율, 사고 수  |  규제 점검 통과 여부      |
+-----------+---------------------------+---------------------------+
```

표에서 정량과 정성이 같은 행에 놓인 까닭은, 한쪽만으로는 그 축을 충분히 설명할 수 없기 때문입니다. 정량 지표는 자동으로 모이고 트렌드를 만들 수 있다는 장점이 있지만, 측정되지 않는 차원을 보지 못합니다. 정성 지표는 사람이 직접 들여다봐야 하고 표본이 작다는 약점이 있지만, 정량으로 잡히지 않는 hidden quality를 확인할 수 있는 유일한 통로입니다. 두 가지를 같은 축에 묶어 두면 한쪽이 무너질 때 다른 쪽이 신호를 줍니다.

세 축 안에서, 그리고 세 축 사이에서 KPI는 서로 충돌합니다. 토큰 비용을 줄이면 응답 품질이 떨어질 수 있고, 가드레일을 강하게 걸면 사용자 만족도가 떨어질 수 있으며, 응답 지연을 줄이려고 작은 모델로 라우팅하면 정답률이 함께 떨어질 수 있습니다. **KPI 사이의 trade-off**는 숨길 일이 아니라 명시할 일입니다. 명시되지 않은 trade-off는 한 축의 압박이 강해지면 다른 축을 조용히 갉아먹습니다. Klarna 사례가 정확히 이 경고였습니다. 230만 건의 대화, 700 FTE 분량의 작업 처리, CSAT 47% 증가, 1천만 달러 절감이라는 효율성 지표가 압도적인 평가 요소가 되면서, 복잡하고 감정적인 케이스에서 사용자가 unheard·unsupported하다고 느끼는 hidden quality가 침식되었고, 결국 인간 상담사 재고용과 hybrid 회귀에 이르렀습니다. 효율성 축의 KPI가 다른 두 축의 KPI를 묶지 않으면 단일 메트릭 최적화의 함정에 빠집니다.

이 함정을 막는 표준 도구가 **북극성 메트릭과 가드레일 메트릭**의 짝입니다. 북극성 메트릭은 조직이 길게 보았을 때 향하고 싶은 한 가지 지표입니다. 사용자에게 가장 큰 가치를 만드는 단 하나의 숫자에 가깝습니다. 예를 들어 상권 분석 에이전트라면 '의사결정에 실제 활용된 보고서 비율' 같은 지표가 북극성이 될 수 있고, 코드 에이전트라면 '머지된 PR 비율'이 북극성이 될 수 있습니다. 가드레일 메트릭은 그 북극성을 향해 달릴 때 깨뜨려서는 안 되는 한계선을 정한 지표 묶음입니다. P95 지연이 X초를 넘지 않을 것, 사용자 escalation 비율이 Y%를 넘지 않을 것, 비가역 행동의 사람 승인 누락이 0건일 것 같은 항목입니다. 북극성은 방향을 주고 가드레일은 추락을 막습니다. 둘을 함께 두면 효율성·품질·안전성 세 축이 균형을 잃지 않습니다. 그림으로 옮기면 다음과 같습니다.

```
                        [ 북극성 메트릭 ]
                  사용자 가치의 한 가지 숫자
                              |
                              v
        +---------------------+---------------------+
        |                     |                     |
 [ 효율 가드레일 ]    [ 품질 가드레일 ]    [ 안전 가드레일 ]
   P95 지연, 단위 비용  escalation 한도     사고 0건, PII 0건
        |                     |                     |
        +---------- 어느 하나라도 무너지면 알람 ----+
```

KPI는 만든 순간이 아니라 운영하는 동안에 의미를 가집니다. 그래서 **분기·월별 리뷰 주기**를 같이 정해 두어야 합니다. 월별 리뷰는 트렌드를 점검하는 자리입니다. 북극성과 가드레일이 모두 정상 범위인지 확인하고, 한 축이 흔들리고 있다면 어떤 변경이 그 흔들림을 만들었는지 그달의 release 노트와 맞춰 봅니다. 분기별 리뷰는 KPI 자체의 적정성을 점검하는 자리입니다. 시스템과 사용자가 변하면 같은 메트릭이 다른 의미를 갖게 되고, 어떤 메트릭은 천장에 닿아 더 이상 변별력이 없어집니다. 분기 리뷰에서는 KPI를 새로 정의하거나, 기존 KPI 가운데 하나를 빼고 새 KPI를 더하는 결정을 내립니다. 운영 호흡으로 정리하면 다음과 같습니다.

```
주 1회 : metric dashboard 자동 알람 점검
월 1회 : 트렌드 리뷰, 가드레일 위반 사후 검토
분기 1회: KPI 자체의 재정의, 북극성 후보 토론
연 1회 : 효율·품질·안전 세 축 비중 조정
```

마지막으로 KPI를 정성 지표와 묶을 때의 실무 팁을 짚겠습니다. 정성 지표는 자동화하기 어렵다는 이유로 종종 미뤄지는데, 미루면 단일 메트릭 함정으로 곧장 들어갑니다. 가장 가벼운 형태는 분기 리뷰 자리에서 사람 평가자가 무작위 표본 30~50건을 직접 읽고 점수를 매기는 패널 평가입니다. 비용은 한 분기당 한두 명이 하루 쓰는 정도로 가볍고, 정량 지표가 가리키지 못하는 hidden quality 신호를 잡습니다. 이 패널 평가의 결과는 분기 리뷰의 토론 재료가 되어 KPI 재정의로 이어집니다.

정리하면, KPI 설계는 효율성·품질·안전성 세 축을 정량과 정성으로 함께 채우고, 축 사이의 trade-off를 명시하며, 북극성 메트릭과 가드레일 메트릭의 짝으로 단일 메트릭 함정을 막는 작업입니다. 월별과 분기별 리뷰 주기를 미리 정해 KPI가 운영의 언어로 살아 있도록 하고, 패널 평가로 정성 신호를 분기마다 끌어들이면 Klarna가 빠진 함정에서 벗어날 수 있습니다.

다음 단원인 6.2.3에서는 KPI를 둘러싼 거버넌스가 시스템의 속도를 죽이지 않도록 균형을 잡는 패턴을 다룹니다. JPMorgan의 8주 update cadence와 release train 같은 도구를 한국 대기업·금융권 환경에 맞춰 적용하는 방법을 살펴봅니다.

이 단원을 마치면 효율성·품질·안전성 세 축의 KPI를 정량과 정성으로 균형 있게 설계할 수 있고, 북극성 메트릭과 가드레일 메트릭을 묶어 단일 메트릭 함정을 피하는 구조를 그릴 수 있으며, 월별·분기별 리뷰 주기와 패널 평가로 KPI를 살아 있는 운영 도구로 유지할 수 있습니다.
