# 5.4.3 사례 분석: 코딩 에이전트의 하네스

앞 단원인 5.4.2에서는 컨텍스트, 메모리, 프로토콜, 하네스라는 네 축이 한 에이전트 안에서 어떻게 맞물리는지를 종합해 살펴봤습니다. 다만 그 설명은 여전히 추상적인 그림에 머물렀습니다. 네 축이 서로 의존한다는 말은 들었지만, 실제로 돌아가는 도구 안에서 그 축들이 어떤 모습으로 박혀 있는지는 아직 손에 잡히지 않습니다. 이 단원은 그 그림을 구체적인 사례 하나로 내려서, 네 축이 현장에서 어떻게 구현되는지를 한 조각씩 해부합니다.

사례로 삼을 대상은 에이전틱 코딩 도구입니다. 에이전틱 코딩 도구란, 개발자가 자연어로 작업을 시키면 코드를 직접 읽고 고치고 실행하면서 며칠에 걸친 개발 작업을 대신 진행하는 종류의 에이전트를 말합니다. 이런 도구를 사례로 고른 까닭은, 네 축이 모두 빠짐없이 등장하면서도 각 축이 맡는 일이 또렷이 드러나기 때문입니다. 코드를 다루는 일은 오류가 분명하고, 작업이 길며, 위험한 행동과 안전한 행동이 명확히 갈립니다.

한 가지 짚어 둘 점이 있습니다. 이 단원은 특정 제품 하나를 지목해 그 제품이 옳다고 주장하려는 것이 아닙니다. 시장에는 비슷한 도구가 여럿 있고, 세부 구현은 도구마다 다릅니다. 그래서 여기서는 어느 한 제품의 내부를 단정하는 대신, 이런 도구들이 공통으로 택하는 대표적 구현 방식을 추려 설명합니다. 읽는 동안에는 특정 이름보다, 앞에서 배운 어떤 개념이 여기서 어떤 부품으로 나타나는지에 집중하면 됩니다.

해부의 순서는 메모리 축부터 시작하겠습니다. 코딩 에이전트가 며칠을 일하려면 먼저 프로젝트의 약속을 알아야 합니다. 이 약속은 두 갈래로 들어옵니다. 첫째 갈래는 3.2.2에서 다룬 명시적 규칙 계층입니다. 개발자가 프로젝트 폴더에 규칙 파일을 한 장 적어 두면, 에이전트는 작업을 시작할 때마다 그 파일을 작업 공간에 통째로 끼워 넣고 일을 시작합니다. 이 파일에는 결과물을 어느 폴더에 두는지, 어떤 폴더는 건드리지 말아야 하는지, 충돌이 생기면 무엇을 앞세우는지 같은, 코드만 봐서는 알 수 없는 약속이 담깁니다.

둘째 갈래는 3.2.3에서 다룬 자동 학습 계층입니다. 명시적 규칙이 개발자가 미리 아는 것만 담는다면, 자동 학습 계층은 에이전트가 일하다 스스로 알아낸 사실을 사람의 손을 거치지 않고 직접 적어 둡니다. 코딩 작업에서 이런 사실은 흔히 한 번 실패해 본 뒤에야 드러납니다. 빌드 명령이 특정 폴더에서만 동작한다거나, 어떤 테스트는 환경 변수를 켜야 통과한다는 것이 그런 예입니다. 에이전트는 이 사실을 사실 단위로 끊어 적고, 새 세션에서는 인덱스를 먼저 읽어 지금 일과 관련된 것만 골라 끌어옵니다.

두 메모리 갈래가 코딩 도구 안에서 어떻게 짝을 이루는지 한 장면으로 묶어 보겠습니다. 개발자가 규칙 파일에 결과물은 outputs 폴더에 두라고 적어 둡니다. 에이전트는 이 규칙을 매 세션 첫머리에 읽습니다. 그러다 첫날 빌드가 하위 폴더에서만 되는 것을 직접 겪으면, 그 사실을 자동 학습 계층에 적어 둡니다. 이튿날 새 세션이 열려도 에이전트는 규칙 파일에서 폴더 약속을 다시 읽고, 인덱스를 거쳐 빌드 폴더 사실을 골라 와, 어제 헤맨 길을 오늘 다시 헤매지 않습니다.

두 갈래가 이렇게 나뉘는 데는 분명한 이유가 있습니다. 폴더 약속은 개발자만 정할 수 있는 것이라 사람이 직접 규칙으로 적습니다. 빌드 폴더 사실은 일을 해 봐야 드러나는 것이라 에이전트가 스스로 적습니다. 누가 아느냐에 따라 쓰는 자리를 가른 것입니다. 코딩 도구가 이 둘을 한 파일에 뭉치지 않고 따로 두는 까닭이 여기 있습니다.

다음은 컨텍스트 축입니다. 코딩 작업은 며칠을 도는 동안 읽은 파일, 실행한 명령, 받은 오류 메시지가 끝없이 쌓입니다. 5.1.1에서 컨텍스트 폭증이라 부른 문제가 이 도구에서는 특히 빠르게 닥칩니다. 그래서 코딩 에이전트는 5.2.3에서 본 압축 운영을 거의 그대로 구현합니다. 하네스가 매 턴 끝마다 압축 훅으로 토큰 사용량을 재고, 임계값을 넘으면 사람의 손 없이 압축을 켜서 지난 기록을 요약으로 줄입니다.

이때 코딩 도구가 지키는 규율도 앞에서 배운 그대로입니다. 압축을 걸되 작업의 목표, 그동안 내린 설계 결정, 아직 풀지 못한 과제라는 핵심 상태는 끝까지 남기는 상태 보존 압축을 적용합니다. 회원가입 기능을 이틀째 만들던 중 압축이 걸려도, 비밀번호를 암호화하기로 한 결정과 이메일 인증이 아직 미완이라는 사실은 요약 안에 살아남습니다. 그 덕분에 에이전트는 자리를 크게 벌고도 자기가 무엇을 하던 중이었는지를 잊지 않습니다.

세 번째는 프로토콜 축입니다. 코딩 에이전트는 파일을 읽고 코드를 고치는 일만으로는 부족할 때가 많습니다. 데이터베이스를 조회하거나, 외부 문서를 가져오거나, 사내 시스템을 호출해야 하는 경우가 생깁니다. 이런 바깥 기능을 에이전트에 붙일 때 쓰는 것이 4.2에서 다룬 MCP입니다. 코딩 도구는 호스트 역할을 맡아, 개발자가 연결을 등록한 MCP 서버마다 클라이언트를 하나씩 띄워 1:1로 짝지어 둡니다.

이 연결이 실제로 일하는 모습은 이렇습니다. 개발자가 데이터베이스 조회용 MCP 서버를 하나 등록해 두면, 코딩 에이전트는 그 서버가 노출한 도구를 자기 행동 목록에 더합니다. 작업 중 데이터 구조를 확인해야 하면, 에이전트는 그 도구를 호출하고 결과를 받아 컨텍스트로 들입니다. 핵심은 코딩 도구가 그 데이터베이스에 맞춘 전용 연동 코드를 따로 짜지 않는다는 점입니다. MCP라는 표준 한 줄에 꽂기만 하면, 같은 방식으로 다른 서버도 얼마든지 더 붙일 수 있습니다.

마지막 축은 하네스의 안전 장치, 곧 승인과 롤백 경계입니다. 코딩 에이전트는 사람이 자리를 비운 며칠 동안 파일을 고치고 명령을 실행합니다. 이 가운데 일부 행동은 한번 일어나면 되돌리기 어렵습니다. 파일을 영구히 지우거나, 운영 서비스에 코드를 배포하거나, 외부로 무언가를 내보내는 행동이 그렇습니다. 그래서 코딩 도구는 5.3.3에서 다룬 인간 승인 경계를 행동 앞에 세웁니다.

이 경계는 행동을 위험 등급으로 나누어 작동합니다. 코드를 읽거나 검색하는 것처럼 되돌릴 수 있고 위험이 낮은 행동은 승인 없이 곧장 진행합니다. 반대로 파일 삭제나 배포처럼 위험하고 비가역적인 행동은 실행 직전에 멈춰 서서 개발자의 승인을 요청합니다. 사람은 핵심 결정 지점에서만 개입한다는 그림이, 코딩 도구에서는 바로 이 등급별 승인 요청으로 구현됩니다.

승인 경계와 짝을 이루는 것이 롤백입니다. 코딩 도구는 작업을 진행하면서 되돌아갈 수 있는 지점을 미리미리 남겨 둡니다. 코드가 버전으로 관리되는 환경이라면, 변경 이전 상태가 곧 되돌아갈 지점이 됩니다. 에이전트가 며칠치 작업을 잘못된 방향으로 끌고 갔다는 것이 뒤늦게 드러나면, 그 잘못된 진행을 통째로 물러서서 성한 지점부터 다시 시작할 수 있습니다. 승인 경계가 사고를 미리 막는 장치라면, 롤백은 이미 어긋난 진행을 사후에 되감는 장치입니다.

지금까지 해부한 네 축을, 이 사례가 각 축을 어떤 부품으로 구현하는지로 한 표에 모으면 다음과 같습니다.

```
축          코딩 에이전트에서의 구현            앞에서 배운 개념
컨텍스트     하네스가 압축 훅으로 자동 압축       5.2.3 압축 운영
            핵심 상태는 보존                    상태 보존 압축
메모리       규칙 파일 + 자동 기록 사실          3.2.2 / 3.2.3
            인덱스로 골라 로드
프로토콜     MCP 서버를 클라이언트로 1:1 연결    4.2 MCP
            표준으로 외부 도구 확장
하네스       위험 등급별 승인 요청               5.3.3 승인 경계
            되돌아갈 지점 남겨 롤백             5.3.2 롤백
```

표를 위에서 아래로 읽으면, 책 전체에서 따로따로 배운 개념들이 하나의 도구 안에서 어떻게 한자리에 모이는지가 보입니다. 컨텍스트 축의 압축, 메모리 축의 두 계층, 프로토콜 축의 MCP, 하네스 축의 승인과 롤백이 각각 제 자리를 맡습니다. 이 가운데 어느 하나라도 빠지면 며칠짜리 코딩 작업은 어딘가에서 무너집니다. 압축이 없으면 기록이 넘쳐 멈추고, 메모리가 없으면 같은 실수를 되풀이하며, 승인 경계가 없으면 무인 시간대에 되돌릴 수 없는 사고가 납니다.

정리하면, 에이전틱 코딩 도구는 책에서 따로 배운 네 축이 한 시스템 안에서 맞물린 대표적 사례입니다. 메모리 축은 사람이 쓴 규칙 파일과 에이전트가 스스로 적는 자동 학습 사실로 나뉘어 프로젝트의 약속을 이어 가고, 컨텍스트 축은 하네스가 압축 훅으로 기록을 자동 압축하되 목표와 결정과 미해결 과제는 상태 보존 압축으로 지킵니다. 프로토콜 축은 MCP로 외부 도구를 표준 인터페이스에 꽂아 확장하고, 하네스 축은 위험 등급별 승인 경계와 되돌아갈 지점을 통한 롤백으로 무인 진행의 위험을 막습니다. 어느 한 축이 빠지면 장기 실행이 그 자리에서 흔들린다는 것이, 이 사례가 보여 주는 핵심입니다. 다음 단원인 5.4.4에서는 이 네 축을 처음부터 다 갖추는 대신, 최소 루프에서 시작해 한 축씩 더해 가며 며칠씩 일하는 에이전트로 키우는 로드맵을 다룹니다.

이 단원을 마치면 실제 에이전틱 코딩 도구에서 명시 규칙과 자동 메모리, 컨텍스트 압축, MCP 도구 연결, 승인과 롤백 경계가 각각 어떤 부품으로 구현되어 네 축을 이루는지를 사례로 분석해 설명할 수 있습니다.
