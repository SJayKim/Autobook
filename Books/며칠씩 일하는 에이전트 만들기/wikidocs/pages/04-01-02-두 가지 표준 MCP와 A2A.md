# 4.1.2 두 가지 표준: MCP와 A2A

앞 단원 4.1.1에서 우리는 표준이 없을 때 생기는 통합의 M×N 문제를 보았습니다. 모델이나 에이전트가 M가지, 붙일 도구가 N가지일 때, 짝마다 맞춤 연동을 따로 만들면 만들어야 할 연결이 M 곱하기 N으로 불어난다는 이야기였습니다. 공통 규약을 하나 두면 그 수가 M 더하기 N으로 줄어든다는 것도 보았습니다. 이 단원에서는 그 공통 규약이 실제로 어떤 모습으로 자리 잡았는지를 봅니다.

먼저 짚어 둘 것이 있습니다. 연결해야 할 대상이 한 종류가 아니라는 점입니다. 에이전트가 바깥과 이어야 하는 상대는 크게 두 갈래입니다. 하나는 도구와 데이터입니다. 달력을 읽고, 데이터베이스를 조회하고, 파일을 저장하는 기능들입니다. 다른 하나는 다른 에이전트입니다. 큰 일을 여러 에이전트가 나눠 맡을 때 서로 일을 주고받는 상대입니다. 이 둘은 성격이 달라서, 하나의 규약으로 한꺼번에 묶기 어렵습니다. 그래서 표준도 두 갈래로 나뉘었습니다.

여기서 잠깐 용어 하나를 약속하겠습니다. 서로 다른 쪽이 정보를 주고받을 때 따르는 공통의 형식과 절차를 프로토콜이라고 부릅니다. 누가 먼저 무엇을 보내고, 그 답에 어떤 내용이 담기는지를 미리 정해 두는 약속입니다. 표준이라는 말도 같은 뜻으로 쓰겠습니다. 앞으로 볼 두 가지가 모두 이런 프로토콜입니다.

첫 번째 표준은 **MCP**입니다. MCP는 모델-컨텍스트 프로토콜(Model Context Protocol)의 약어로, 모델이나 에이전트를 외부의 도구·데이터에 잇는 표준입니다. 구체적인 상황으로 먼저 보겠습니다. 어떤 회사가 사내 일정을 조회하는 기능, 주문 기록을 읽는 기능, 보고서 파일을 저장하는 기능을 가지고 있다고 합시다. 이 세 기능을 MCP가 정한 형식대로 노출해 두면, MCP를 아는 어떤 에이전트든 이들을 같은 방식으로 불러 쓸 수 있습니다. 에이전트가 바뀌어도 기능 쪽을 다시 고칠 필요가 없습니다.

MCP가 푸는 문제를 4.1.1의 말로 다시 옮기면, 도구 연결 쪽의 M×N 폭발입니다. 모델이나 에이전트가 여럿이고 도구가 여럿일 때, 각자 MCP라는 규약 하나에만 맞추면 모든 짝을 직접 잇지 않아도 됩니다. 그래서 MCP는 에이전트가 자기 아래에 있는 도구와 데이터를 끌어 쓰는 연결, 즉 위아래로 잇는 수직 연결에 해당합니다. 여기서 수직이라는 말은 에이전트가 위에 있고 도구가 아래에 있다는 위계를 가리킵니다. 도구는 시키는 일을 할 뿐, 스스로 판단하지 않습니다.

이 표준이 주는 이득은 도구를 만든 쪽과 쓰는 쪽이 서로를 신경 쓰지 않아도 된다는 데 있습니다. 앞의 주문 기록을 읽는 기능을 한 번 MCP 형식으로 노출해 두면, 그 기능을 누가 쓰든 다시 손볼 일이 없습니다. 어제는 한 종류의 모델을 쓰던 에이전트가 오늘 다른 모델로 바뀌어도, 또는 조사용 에이전트 대신 보고용 에이전트가 그 기능을 부르게 되어도, 노출된 도구 쪽은 그대로 둡니다. 표준을 모르던 시절이라면 모델을 바꿀 때마다, 새 에이전트를 붙일 때마다 연동 코드를 다시 짜야 했습니다. MCP는 그 잔손질을 없애 한 번 만든 도구를 두고두고 다시 쓰게 합니다. 며칠씩 도는 에이전트가 그동안 수십 가지 도구를 갈아 쓰는 상황에서 이 재사용성은 특히 크게 작용합니다.

두 번째 표준은 **A2A**입니다. A2A는 에이전트 투 에이전트(Agent-to-Agent)의 약어로, 서로 다른 에이전트가 협업하도록 통신을 표준화한 표준입니다. 이번에도 상황으로 보겠습니다. 자료 조사를 맡은 에이전트가 모은 내용을 글쓰기를 맡은 에이전트에게 넘기고, 검토를 맡은 에이전트가 그 글을 다시 받아 본다고 합시다. 세 에이전트가 일을 주고받으려면 공통의 대화 규약이 있어야 합니다. A2A가 바로 그 규약입니다.

A2A에서 연결되는 양쪽은 도구가 아니라 둘 다 에이전트라는 점이 핵심입니다. 도구는 시키는 일만 하지만, 에이전트는 스스로 다음 행동을 정합니다. 그래서 한쪽이 다른 쪽의 속을 자세히 몰라도, A2A라는 공통 규약만 지키면 일을 맡기고 결과를 받을 수 있습니다. 상대가 어떤 모델로 어떻게 동작하는지는 몰라도 됩니다. 이렇게 속을 모르는 상대와도 표준 메시지만으로 협업하는 것이 A2A가 노리는 지점입니다. A2A는 같은 급의 에이전트끼리 옆으로 주고받는 통신이므로, 수평 연결에 해당합니다.

속을 모르고도 협업한다는 말이 어떻게 가능한지 조금 더 풀어 보겠습니다. 사람으로 치면, 함께 일할 동료가 무엇을 잘하는지 미리 알아야 일을 맡길 수 있는 것과 같습니다. A2A에서도 각 에이전트는 자신이 어떤 일을 할 수 있는지를 표준 형식으로 알립니다. 그러면 일을 나눠 주는 쪽은 그 알림만 보고 누구에게 무엇을 맡길지 정합니다. 글을 다듬는 일이 필요하면 글쓰기를 잘한다고 알린 에이전트를 고르는 식입니다. 상대가 그 일을 어떤 절차로 처리하는지는 들여다보지 않습니다. 이렇게 능력을 알리고 그에 맞춰 일을 고르는 구체적인 방식은 4.3에서 다루며, 이 단원에서는 속을 몰라도 표준만으로 짝을 짓는다는 성격까지만 잡아 둡니다.

왜 MCP 하나로 둘 다 처리할 수 없는지 한 번 더 짚겠습니다. MCP는 에이전트가 도구를 부르는 한 방향 흐름을 다룹니다. 부르면 실행하고 결과를 돌려주는 관계입니다. 그런데 에이전트끼리는 일을 위임하고, 진행 상황을 주고받고, 오래 걸리는 작업의 결과를 나중에 받아 오는 식의 더 복잡한 주고받음이 필요합니다. 상대도 스스로 판단하는 주체이기 때문입니다. 이 협업을 다루려면 도구용 규약과는 다른 에이전트 전용 규약이 따로 있어야 합니다. 그것이 A2A입니다.

두 연결의 방향 차이를 그림으로 보면 한눈에 들어옵니다. 세로 화살표는 도구를 끌어 쓰는 MCP, 가로 화살표는 동료와 주고받는 A2A입니다.

```
        에이전트 A  ←── A2A ──→  에이전트 B
           │ (수평: 옆의 동료와 협업)  │
           │ MCP                  MCP │
           ↓ (수직: 아래 도구 사용)    ↓
        [도구·데이터]            [도구·데이터]
```

위 그림에서 에이전트 A는 두 가지 일을 동시에 합니다. 아래로는 MCP로 자기 도구를 끌어 쓰고, 옆으로는 A2A로 에이전트 B와 협업합니다. 즉 한 에이전트가 MCP로 도구를 쓰는 주체이면서, 동시에 A2A로 협업하는 동료가 됩니다. 두 표준은 경쟁하지 않고 서로 다른 층을 맡습니다. 수직은 MCP, 수평은 A2A가 채우는 상호 보완 관계입니다.

이 보완 관계가 실제로 어떻게 맞물리는지 앞의 예로 이어 보겠습니다. 글쓰기를 맡은 에이전트는 A2A로 조사 에이전트에게서 자료를 받습니다. 그런 다음 그 자료로 초안을 쓰는 과정에서, 사내 문서 저장소를 읽는 도구와 파일을 저장하는 도구를 MCP로 끌어 씁니다. 같은 에이전트가 한쪽에서는 수평으로 동료와 일하고, 다른 쪽에서는 수직으로 도구를 부립니다. 그래서 둘을 가르는 기준은 간단합니다. 연결 상대가 도구나 데이터이면 MCP, 스스로 판단하는 에이전트이면 A2A입니다.

두 표준의 역할이 헷갈리지 않도록 한 번에 비교하면 다음과 같습니다.

| 구분 | MCP | A2A |
|------|-----|-----|
| 잇는 대상 | 모델·에이전트와 외부 도구·데이터 | 에이전트와 에이전트 |
| 연결 방향 | 수직(아래의 도구를 끌어 씀) | 수평(옆의 동료와 주고받음) |
| 상대의 성격 | 시키는 일을 하는 도구 | 스스로 판단하는 주체 |
| 푸는 문제 | 도구 연동의 M×N 폭발 | 속을 모르는 에이전트 간 협업 |

이 두 표준이 2026년에 자리 잡은 흐름도 짚어 두겠습니다. 도구 연결을 표준화한 MCP가 먼저 널리 퍼지면서, 도구를 한 번 만들어 여러 에이전트가 함께 쓰는 생태계가 만들어졌습니다. 그다음으로 에이전트끼리 협업하려는 요구가 커지자, 도구용 규약만으로는 다룰 수 없는 부분을 메우는 A2A가 뒤따라 자리를 잡았습니다. 수직 연결이 먼저 굳고, 그 위에서 수평 연결이 더해진 셈입니다. 두 표준이 함께 쓰이면서, 도구와 동료를 자유롭게 엮는 멀티에이전트 구성이 현실적인 선택지가 되었습니다.

이 단원에서는 두 표준이 각각 무엇을 잇고 어떻게 보완하는지까지만 잡아 둡니다. MCP가 실제로 어떤 부분으로 이루어지고, 도구를 어떻게 노출하며, 어떤 절차로 연결되는지는 4.2에서 본격적으로 다룹니다. A2A가 에이전트의 능력을 어떻게 알리고, 작업과 메시지를 어떤 모델로 주고받는지는 4.3에서 풀어 갑니다. 지금은 수직과 수평이라는 두 방향의 자리만 분명히 해 두면 충분합니다.

정리하면, 에이전트가 이어야 할 상대가 도구·데이터와 다른 에이전트라는 두 갈래이기 때문에 표준도 두 가지로 나뉩니다. MCP는 모델이나 에이전트를 도구·데이터에 잇는 수직 표준이고, A2A는 에이전트를 다른 에이전트에 잇는 수평 표준입니다. 둘은 경쟁하지 않고 서로 다른 층을 맡아 보완하며, 한 에이전트가 MCP로 도구를 쓰면서 동시에 A2A로 동료와 협업합니다. 다음 단원인 4.1.3에서는 이 표준화가 멀티에이전트 아키텍처와 비용 경제학, 그리고 거버넌스를 함께 어떻게 바꾸는지를 살펴봅니다.

이 단원을 마치면 MCP가 모델·에이전트를 도구·데이터에 잇는 수직 표준이고 A2A가 에이전트를 다른 에이전트에 잇는 수평 표준이라는 점, 그리고 둘이 서로를 보완하며 함께 쓰이는 까닭을 구분해 설명할 수 있습니다.
