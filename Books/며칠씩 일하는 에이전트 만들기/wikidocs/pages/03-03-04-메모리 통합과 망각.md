# 3.3.4 메모리 통합과 망각

며칠씩 일하는 에이전트는 한 번 작동하고 멈추지 않습니다. 같은 프로젝트에서 수십 번씩 세션을 재개하며, 그때마다 새로 배운 사실을 영속 메모리에 적어 둡니다. 3.3.1 쓰기에서 다룬 대로 저장 가치가 있는 사실만 골라 적는다 해도, 시간이 흐르면 비슷한 메모리가 여러 벌 쌓이고, 한때 맞았지만 지금은 틀린 메모리도 섞여 들어갑니다. 메모리를 계속 더하기만 하면 양은 늘지만 품질은 떨어집니다. 이 단원은 쌓인 메모리를 정돈하는 두 가지 작업, 곧 같은 내용을 하나로 합치는 일과 쓸모를 잃은 메모리를 덜어 내는 일을 다룹니다.

먼저 왜 정돈이 필요한지 구체적인 상황으로 시작하겠습니다. 어떤 에이전트가 첫째 날 사용자에게서 "들여쓰기는 공백 2칸을 쓴다"는 선호를 듣고 메모리에 적었다고 합시다. 일주일 뒤 다른 세션에서 사용자가 같은 말을 다시 했고, 에이전트는 그것을 새 사실로 또 적었습니다. 이제 거의 같은 메모리 두 개가 따로 존재합니다. 3.3.2 읽기에서 본 회상은 인덱스의 설명을 훑어 관련된 메모리를 골라 컨텍스트로 적재하는 과정인데, 같은 내용이 여러 벌이면 회상할 때마다 컨텍스트에 같은 말이 중복으로 실립니다. 토큰만 더 쓰고 새로 얻는 정보는 없습니다.

이렇게 의미가 겹치는 메모리를 하나로 합치는 작업을 **통합(consolidation)**이라고 합니다. 통합은 단순히 중복을 지우는 일이 아니라, 흩어진 조각을 더 정확하고 완결된 하나의 사실로 정리하는 일입니다. 앞의 예에서 들여쓰기에 관한 두 메모리를 합치면, "사용자는 들여쓰기로 공백 2칸을 선호한다. 두 차례 확인됨"처럼 더 분명한 한 줄이 남습니다. 두 차례 확인되었다는 정보가 붙으면, 이 사실이 한 번 스쳐 지나간 말이 아니라 굳어진 선호라는 신뢰도까지 함께 담깁니다.

통합을 하려면 새로 들어온 메모리가 기존 메모리와 무슨 관계인지 먼저 가려야 합니다. 여기서 **갱신과 신규의 구분**이 등장합니다. 갱신은 이미 있는 사실을 고쳐 쓰는 경우이고, 신규는 전에 없던 사실을 새로 더하는 경우입니다. 둘을 구분하지 않고 무조건 새로 적으면 중복이 쌓이고, 반대로 무조건 기존 것에 덮어쓰면 서로 다른 사실이 사라집니다. 그래서 새 메모리를 저장하기 전에, 같은 대상을 가리키는 기존 메모리가 있는지 확인하는 단계가 필요합니다.

구분의 기준은 두 메모리가 같은 대상을 다루는가, 그리고 내용이 충돌하는가입니다. 같은 대상이고 내용이 같으면 그것은 중복이므로 통합합니다. 같은 대상인데 내용이 다르면, 보통은 새 정보가 옛 정보를 대체하므로 갱신입니다. 예를 들어 처음에 "테스트는 명령 한 줄로 돌린다"고 적었는데 나중에 사용자가 "이제 테스트는 두 단계로 나눠 돌린다"고 알려 주면, 옛 메모리를 새 내용으로 갱신해야 합니다. 옛 메모리를 그대로 둔 채 새것만 더하면, 회상 때 모순된 두 지시가 함께 떠올라 에이전트가 어느 쪽을 따를지 흔들립니다. 대상이 아예 다르면 그것은 신규이므로 새 메모리로 추가합니다.

```
새 메모리가 들어옴
        |
같은 대상의 기존 메모리가 있는가?
   ├─ 없음 ──────────────→ 신규: 새 항목으로 추가
   └─ 있음
        |
   내용이 충돌하는가?
   ├─ 같은 내용 ─────────→ 통합: 하나로 합치고 확인 횟수 갱신
   └─ 다른 내용 ─────────→ 갱신: 새 내용으로 덮어쓰기
```

위 그림은 새 메모리 하나를 받았을 때 거치는 판단을 차례로 보여 줍니다. 이 세 갈래, 곧 신규·통합·갱신을 매번 가려 주면 메모리는 늘어나기만 하는 더미가 아니라 정리된 모음으로 유지됩니다. 판단의 근거가 되는 같은 대상 여부는 3.3.3 구조화에서 다룬 타입과 설명 필드를 활용합니다. 타입이 사용자 선호로 같고 설명이 들여쓰기를 가리키면 같은 대상일 가능성이 높습니다. 구조화된 메타데이터가 통합과 갱신의 판단을 떠받칩니다.

이제 합치는 일과 별개로 덜어 내는 일을 보겠습니다. 메모리 중에는 한때 유효했지만 지금은 더 이상 맞지 않거나 쓸 일이 없는 것이 있습니다. 사용했던 라이브러리를 다른 것으로 바꿨다면 옛 라이브러리에 관한 주의사항은 효력을 잃습니다. 일회성 작업을 위해 적어 둔 임시 메모도 그 작업이 끝나면 남겨 둘 이유가 없습니다. 이렇게 오래되어 효력을 잃은 메모리를 골라 덜어 내는 작업을 **폐기(pruning)**라고 합니다.

폐기 대상을 가리는 기준은 크게 셋입니다. 첫째는 명백히 틀린 것입니다. 사실이 바뀌었는데 갱신되지 않고 남은 옛 메모리가 여기에 해당합니다. 둘째는 오래되어 신뢰하기 어려운 것입니다. 마지막으로 확인된 지 오래되었고 그동안 한 번도 회상되지 않았다면, 지금도 유효한지 의심해야 합니다. 셋째는 더 이상 쓰이지 않는 것입니다. 다른 메모리로 통합되어 따로 둘 이유가 없어진 조각이 여기에 들어갑니다. 이 셋을 정기적으로 걸러 내면 메모리의 평균 신뢰도가 유지됩니다.

폐기를 두려워하는 직관이 있습니다. 기억은 많을수록 좋고 지우는 것은 손실이라는 생각입니다. 하지만 에이전트 메모리에서는 그렇지 않습니다. **망각은 결함이 아니라 필수 기능입니다.** 사람도 모든 일을 똑같은 선명도로 기억하지 않습니다. 중요한 것은 남기고 사소한 것은 흐려지게 두기에, 필요한 순간 핵심을 빠르게 떠올립니다. 만약 지나간 모든 세부를 동등하게 붙들고 있다면, 정작 중요한 사실을 찾는 데 방해가 됩니다. 에이전트도 마찬가지로, 낡고 무관한 메모리를 덜어 내야 회상의 정확도가 올라갑니다.

망각이 필요한 더 구체적인 이유는 회상 비용에 있습니다. 3.3.2 읽기에서 본 대로 회상은 인덱스의 설명들을 훑어 관련된 항목을 고르는 과정입니다. 메모리가 백 개일 때와 만 개일 때, 훑어야 할 설명의 양이 다릅니다. 게다가 낡은 메모리가 많으면, 관련해 보이지만 실제로는 틀린 항목이 회상에 끼어들 확률이 높아집니다. 그러면 에이전트는 틀린 정보를 근거로 판단하고, 그 결과를 또 새 메모리로 적어 오류를 키울 수 있습니다. 덜어 내기는 이 악순환을 끊습니다.

낡은 메모리가 정돈되지 않고 쌓이면서 메모리 전체의 신뢰도가 서서히 떨어지는 현상을 **메모리 부패**라고 부릅니다. 부패한 메모리는 컨텍스트가 길어질 때 성능이 떨어지는 2.1.3 컨텍스트 부패와 비슷한 결과를 메모리 차원에서 만듭니다. 다만 원인은 다릅니다. 컨텍스트 부패는 한 세션 안에서 너무 많은 토큰을 쥔 탓이고, 메모리 부패는 여러 세션에 걸쳐 낡은 사실이 누적된 탓입니다. 통합과 폐기는 바로 이 메모리 부패를 막는 두 손입니다. 통합은 중복을 줄여 밀도를 높이고, 폐기는 효력 잃은 것을 덜어 평균 신뢰도를 지킵니다.

```
정돈 전                          정돈 후
─────────────────              ─────────────────
들여쓰기 = 공백 2칸             들여쓰기 = 공백 2칸 (2회 확인)
들여쓰기 = 공백 2칸 (중복)      테스트 = 두 단계로 실행
테스트 = 한 줄로 실행 (낡음)    API 키는 환경변수로 주입
테스트 = 두 단계로 실행
옛 라이브러리 주의 (효력 상실)   ← 메모리 7개가 3개로
API 키는 환경변수로 주입           중복 통합·낡은 항목 폐기
임시 작업 메모 (작업 종료됨)
```

위 그림은 통합과 폐기를 함께 적용하기 전과 후를 나란히 놓은 것입니다. 정돈 전에는 일곱 항목이 있지만 그중 둘은 중복이고 셋은 효력을 잃었습니다. 정돈 후에는 세 항목만 남되, 각 항목이 더 정확하고 신뢰할 만합니다. 항목 수가 줄었는데도 담긴 유효 정보는 줄지 않았다는 점이 핵심입니다. 통합으로 같은 사실이 하나로 모였고, 폐기로 잡음이 사라졌기 때문입니다.

이 정돈을 언제 하느냐도 정해 두면 좋습니다. 한 가지 방법은 새 메모리를 적을 때마다 같은 대상의 기존 항목을 확인해 그 자리에서 통합하거나 갱신하는 것입니다. 또 한 가지는 주기적으로, 예컨대 메모리가 일정 수를 넘거나 세션을 여러 번 거친 시점에 전체를 한 번 훑으며 낡은 항목을 폐기하는 것입니다. 앞의 방식은 중복이 쌓이는 것을 막고, 뒤의 방식은 시간이 지나며 효력을 잃은 것을 걷어 냅니다. 두 시점을 함께 두면 메모리는 늘 정돈된 상태에 가깝게 유지됩니다.

정리하면, 메모리는 더하기만 해서는 오래 쓸 수 없고 정돈이 필요합니다. 의미가 겹치는 항목은 하나로 통합하되, 새 메모리가 기존 사실의 갱신인지 전혀 다른 신규인지를 먼저 구분합니다. 효력을 잃거나 틀린 메모리는 폐기로 덜어 냅니다. 이때 망각은 잃는 것이 아니라 회상의 정확도를 지키는 필수 기능이며, 통합과 폐기는 낡은 사실이 쌓여 신뢰도가 떨어지는 메모리 부패를 막는 두 손입니다.

다음 단원인 3.3.5에서는 이렇게 정돈해 둔 영속 메모리와, 외부 문서를 그때그때 검색해 오는 RAG가 어떻게 다른지, 그리고 둘 중 무엇을 언제 골라 써야 하는지를 다룹니다.

이 단원을 마치면 중복 메모리를 통합하는 절차, 갱신과 신규를 구분하는 기준, 낡은 메모리를 폐기하는 기준을 설명하고, 망각이 왜 결함이 아니라 메모리 부패를 막는 필수 기능인지 설명할 수 있습니다.
