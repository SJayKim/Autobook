# 2.2.1 컨텍스트 윈도우의 구성요소

앞선 2.1.5에서는 컨텍스트 윈도우를 용도별 칸으로 나누어 토큰 예산을 세우는 절차를 다루었습니다. 예산을 세우려면 한 가지 전제가 필요합니다. 윈도우 안에 도대체 무엇이 들어 있는지를 먼저 알아야 한다는 것입니다. 칸을 나누려면 칸에 담길 물건의 목록부터 손에 쥐고 있어야 합니다. 이 단원은 바로 그 목록을 펼쳐 봅니다.

많은 사람이 모델에 보내는 입력을 그저 내가 방금 타이핑한 질문 한 줄이라고 생각합니다. 실제로는 그렇지 않습니다. 모델이 답을 만들기 직전, 윈도우 안에는 사용자가 한 번도 직접 입력하지 않은 내용이 이미 한가득 들어 있습니다. 에이전트를 운영하는 프레임워크가 매 턴마다 여러 조각을 한데 모아 하나의 긴 입력으로 조립해 모델에게 건네기 때문입니다. 이 조립된 덩어리 전체가 그 턴의 컨텍스트입니다.

2.1.4에서 정리했듯이, 컨텍스트 엔지니어링은 이 윈도우 전체를 설계 대상으로 삼는 상위 규율입니다. 그런데 전체를 설계하려면 전체가 어떤 부품으로 이루어지는지를 분해할 수 있어야 합니다. 부품을 모르면 어느 부품이 자리를 너무 많이 차지하는지, 어느 부품을 줄여야 하는지 판단할 수 없습니다. 그래서 이 단원은 윈도우를 구성하는 다섯 가지 조각을 하나씩 떼어내 살펴봅니다.

첫 번째 조각은 **시스템 프롬프트**입니다. 시스템 프롬프트는 모델에게 너는 누구이고 어떻게 행동해야 하는지를 알려 주는, 대화 맨 앞에 놓이는 지시문입니다. 예를 들어 너는 친절한 코딩 도우미이며, 답변은 한국어로 하고, 코드를 고칠 때는 항상 이유를 함께 적는다 같은 문장들입니다. 사용자는 이 글을 화면에서 보지 못하지만 모델은 매 턴 이 글을 읽습니다.

시스템 프롬프트는 한 번 정해지면 대화 내내 거의 바뀌지 않고 윈도우 맨 앞을 고정으로 차지합니다. 그래서 길게 쓸수록 모든 턴에서 같은 토큰 비용을 반복해서 냅니다. 만약 시스템 프롬프트에 거의 쓰이지 않는 예외 규칙을 잔뜩 적어 두면, 그 규칙들은 매 턴마다 자리를 차지하면서도 정작 쓸 일은 드뭅니다. 그래서 시스템 프롬프트는 짧고 분명하게, 늘 필요한 지침만 담는 것이 비용 면에서 유리합니다.

두 번째 조각은 **도구 정의**입니다. 1.1.4에서 도구가 에이전트의 행동 공간을 이룬다고 다루었습니다. 에이전트가 파일을 읽거나 웹을 검색하거나 명령을 실행하려면, 모델은 어떤 도구가 있고 각 도구를 어떻게 불러야 하는지를 알아야 합니다. 이 정보가 도구 정의입니다.

도구마다 이름, 무엇을 하는 도구인지에 대한 설명, 그리고 어떤 인자를 어떤 형식으로 넘겨야 하는지를 적은 명세가 함께 들어갑니다. 예를 들어 파일 읽기 도구라면 이름은 read_file, 설명은 지정한 경로의 파일 내용을 돌려준다, 인자는 파일 경로 하나처럼 적습니다. 이 명세는 모두 토큰으로 환산되어 윈도우를 차지합니다. 도구가 다섯 개일 때와 쉰 개일 때는 차지하는 양이 크게 달라집니다. 도구 정의는 시스템 프롬프트처럼 매 턴 거의 그대로 실리므로, 잘 쓰지 않는 도구를 잔뜩 등록해 두면 고정 비용만 늘어납니다.

세 번째 조각은 **대화 기록**입니다. 대화 기록은 지금까지 오간 사용자의 말, 모델의 답, 그리고 그 사이에 끼어든 모든 중간 단계를 시간 순으로 쌓아 둔 것입니다. 여기서 중간 단계란 모델이 어떤 도구를 부르기로 한 결정과, 그 도구가 돌려준 결과를 가리킵니다. 1.2.1에서 에이전트 루프가 추론과 행동과 관찰을 반복한다고 다루었는데, 그 반복의 흔적이 고스란히 대화 기록에 누적됩니다.

대화 기록이 어떻게 불어나는지 구체적으로 그려 보겠습니다. 사용자가 이 프로젝트의 버그를 고쳐 달라고 한 줄을 입력했다고 합시다. 모델은 먼저 관련 파일을 읽기로 결정하고, 파일 읽기 도구가 수백 줄짜리 코드를 통째로 돌려줍니다. 그 코드 전체가 대화 기록에 한 덩어리로 들어갑니다. 이어 모델은 테스트를 실행하고, 그 결과인 오류 메시지가 다시 기록에 쌓입니다. 사용자 입력은 한 줄이었지만, 그 한 줄을 처리하는 동안 기록에는 수백 줄이 더해진 셈입니다.

그래서 대화 기록은 다섯 조각 중에서 가장 빠르게 부풀어 오릅니다. 루프가 한 바퀴 돌 때마다 도구 결과와 모델의 사고 과정이 계속 더해지기 때문입니다. 며칠씩 실행되는 에이전트라면 이 누적이 끝없이 이어지므로, 대화 기록을 어떻게 다룰지가 컨텍스트 관리의 가장 큰 숙제가 됩니다.

네 번째 조각은 **검색된 문서와 메모리 주입**입니다. 이 둘은 출처가 다르지만, 필요한 순간에 외부에서 끌어와 윈도우에 끼워 넣는다는 점이 같아 한 묶음으로 봅니다.

검색된 문서는 사용자의 질문과 관련 있는 자료를 외부 저장소에서 찾아 넣은 것입니다. 회사 내부 규정 문서의 한 대목이나 제품 설명서의 한 단락 같은 것이 여기에 해당합니다. 모델 자신은 그 문서의 내용을 미리 알지 못하므로, 답하기 직전에 관련 대목을 찾아 윈도우에 넣어 주어야 모델이 그것을 근거로 답할 수 있습니다.

메모리 주입은 이전 세션에서 배워 따로 보관해 둔 사실을 다시 꺼내 넣은 것입니다. 사용자가 늘 미터법을 선호한다거나, 지난번에 합의한 코드 규칙 같은 내용이 그 예입니다. 검색 문서가 그때그때의 질문에 맞춰 외부 자료를 가져오는 것이라면, 메모리는 이 사용자나 이 작업에 대해 그동안 쌓아 둔 앎을 되살리는 것이라는 점에서 결이 다릅니다. 다만 두 조각 모두 매 턴 들어오는 것이 아니라 필요할 때만 들어오므로, 양이 턴마다 들쭉날쭉합니다.

여기까지가 윈도우를 채우는 다섯 조각입니다. 이제 마지막 학습 항목인 각 요소의 토큰 비중을 살펴봅니다. 비중이란 윈도우라는 한정된 칸 안에서 각 조각이 차지하는 몫을 말합니다. 같은 작업이라도 어떤 에이전트는 도구가 많아 도구 정의가 자리를 많이 먹고, 어떤 에이전트는 긴 대화를 이어 가 대화 기록이 자리를 많이 먹습니다. 다음 표는 코딩 작업을 오래 이어 가는 에이전트에서 흔히 나타나는 한 가지 구도를 예로 든 것입니다. 정해진 공식이 아니라 비중을 감각으로 익히기 위한 보기입니다.

```
구성요소            성격                  대략 비중   변동성
--------------------------------------------------------------
시스템 프롬프트      행동 지침(고정)        5~10%      낮음
도구 정의            행동 공간 명세          10~20%     중간
대화 기록            루프의 누적 흔적        40~60%     높음(증가)
검색 문서·메모리     필요시 외부 주입        10~30%     높음(들쭉)
여유분(headroom)     출력·예비 공간          10~20%     -
```

표를 한 줄씩 풀어 보겠습니다. 시스템 프롬프트는 비중이 작지만 모든 턴에 똑같이 실립니다. 도구 정의는 도구 수에 따라 늘었다 줄었다 합니다. 대화 기록은 처음에는 거의 0에 가깝다가, 루프가 길어질수록 다른 모든 조각을 합친 것보다 커지기 일쑤입니다. 검색 문서와 메모리는 그 턴에 무엇을 끌어왔느냐에 따라 크게 흔들립니다. 마지막 줄의 여유분은 조각이라기보다, 2.1.5에서 다룬 headroom 즉 모델이 답을 적을 자리와 예기치 못한 입력을 위한 빈 공간입니다.

비중의 시간 변화를 다이어그램으로 한 번 더 정리하면 다음과 같습니다. 초반에는 고정 조각이 대부분을 차지하지만, 루프가 길어지면 대화 기록이 윈도우를 잠식합니다.

```
초반 턴                       후반 턴(루프 누적 후)
+------------------+          +------------------+
| 시스템 프롬프트   |          | 시스템 프롬프트   |
| 도구 정의         |          | 도구 정의         |
| (대화 기록 적음)  |          |                  |
|                  |   --->   |   대화 기록       |
|                  |          |   (크게 팽창)     |
| 여유분           |          |                  |
|                  |          | 여유분 (압박받음) |
+------------------+          +------------------+
```

이 그림이 보여 주는 핵심은, 윈도우의 구성이 고정된 사진이 아니라 시간에 따라 변하는 영상이라는 점입니다. 초반에는 시스템 프롬프트와 도구 정의 같은 고정 조각이 비중을 좌우합니다. 후반으로 갈수록 대화 기록이 부풀어 여유분을 압박합니다. 그래서 2.1.5에서 세운 토큰 예산은 한 번 정하고 끝나는 것이 아니라, 어느 조각이 예산을 초과하기 시작하는지 계속 지켜봐야 하는 살아 있는 계획입니다. 다섯 조각을 구분해 두면, 윈도우가 꽉 차 갈 때 어느 조각부터 손볼지 곧바로 가늠할 수 있습니다.

정리하면, 컨텍스트 윈도우는 사용자가 입력한 한 줄이 아니라, 시스템 프롬프트, 도구 정의, 대화 기록, 검색 문서와 메모리 주입이라는 조각들이 매 턴 조립된 덩어리입니다. 각 조각은 차지하는 토큰 비중과 변동성이 다르며, 특히 대화 기록은 루프가 길어질수록 가장 크게 팽창해 다른 조각과 여유분을 압박합니다. 이 분해는 2.1.5의 토큰 예산을 실제로 점검하고 조정하기 위한 토대가 됩니다.

다음 단원인 2.2.2에서는 이 조각들을 모두 미리 채워 넣지 않고, 필요할 때만 불러오는 적시(just-in-time) 컨텍스트 로딩을 다룹니다. 특히 검색 문서와 메모리처럼 변동이 큰 조각을 어떻게 다루는지가 핵심입니다.

이 단원을 마치면 컨텍스트 윈도우를 시스템 프롬프트, 도구 정의, 대화 기록, 검색 문서와 메모리 주입으로 분해하고, 각 조각의 토큰 비중과 변동 양상을 설명할 수 있습니다.
