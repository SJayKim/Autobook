# 4.2.1 MCP의 구조: 호스트·클라이언트·서버

앞 단원 4.1.2에서 MCP의 범위를 한 줄로 정리했습니다. MCP는 모델이나 에이전트를 외부의 도구와 데이터에 연결하는 표준이라는 것이었습니다. 그런데 표준이 있다는 말과, 그 표준이 실제로 어떤 부품으로 짜여 있는지를 아는 것은 다릅니다. 이 단원에서는 그 부품들을 하나씩 뜯어봅니다.

왜 부품 구조부터 봐야 하는지 먼저 짚겠습니다. 에이전트가 파일을 읽고, 데이터베이스를 조회하고, 외부 서비스를 호출하려면 누군가는 그 기능을 제공해야 하고, 누군가는 그 기능을 가져다 모델에게 건네야 합니다. 이 역할들이 한 덩어리로 뭉쳐 있으면, 도구 하나를 바꿀 때마다 에이전트 전체를 손봐야 합니다. MCP는 이 역할들을 세 자리로 나눠 둡니다. 그래서 어느 자리가 무엇을 책임지는지를 알면, 새 도구를 붙이거나 문제를 추적하는 일이 훨씬 단순해집니다.

세 자리의 이름은 호스트, 클라이언트, 서버입니다. 지금부터 하나씩, 가장 바깥쪽인 호스트부터 안쪽의 서버까지 차례로 설명하겠습니다.

가장 먼저 **호스트**입니다. 호스트는 사용자가 직접 마주하는 에이전트 애플리케이션 그 자체입니다. 예를 들어 대화창에 질문을 입력하면 답을 돌려주는 채팅 앱, 코드 작업을 대신해 주는 개발 보조 도구가 모두 호스트에 해당합니다. 호스트 안에는 언어 모델이 들어 있고, 사용자의 요청을 받아 모델에게 전달하며, 모델이 도구를 쓰고 싶어 하면 그 도구를 불러올 통로를 마련합니다. 즉 호스트는 전체 흐름을 지휘하는 자리입니다. 도구를 직접 들고 있지는 않지만, 어떤 도구를 언제 쓸지 결정하고 그 결과를 모델에게 다시 넣어 주는 일을 맡습니다.

그런데 호스트가 외부의 도구나 데이터에 곧바로 손을 뻗지는 않습니다. 그 사이에 **클라이언트**라는 중간 자리가 있습니다. 클라이언트는 호스트 안에 들어 있는 연결 담당 부품입니다. 호스트가 어떤 도구 모음을 쓰기로 하면, 호스트는 그 도구 모음과 이야기할 클라이언트를 하나 띄웁니다. 클라이언트는 정해진 메시지 규칙에 따라 상대편과 대화를 트고, 요청을 보내고, 응답을 받아 호스트에게 돌려줍니다.

여기서 한 가지 중요한 약속이 있습니다. 클라이언트 하나는 상대편 하나와만 짝을 맺습니다. 이것을 1:1 연결이라고 부릅니다. 즉 도구를 제공하는 쪽이 셋이면, 호스트는 클라이언트도 셋을 띄워 각각 따로 짝지어 줍니다. 한 클라이언트가 여러 상대를 동시에 상대하지 않습니다.

이렇게 1:1로 나눠 두면 좋은 점이 분명합니다. 한 연결에서 문제가 생겨도 다른 연결은 영향을 받지 않습니다. 각 연결을 따로 켜고 끄기도 쉽고, 어디서 오류가 났는지 추적하기도 쉽습니다. 만약 한 클라이언트가 여러 서버를 한꺼번에 맡는다면, 한 서버가 느려질 때 다른 서버 요청까지 막히고, 문제의 원인을 가려내기도 어려워집니다. 1:1 약속은 이런 뒤엉킴을 처음부터 막아 줍니다.

마지막으로 **서버**입니다. 서버는 실제 도구와 데이터를 들고 있는 바깥쪽 자리입니다. 파일을 읽고 쓰는 기능, 데이터베이스를 조회하는 기능, 외부 서비스를 호출하는 기능을 갖춘 작은 프로그램이 곧 서버입니다. 서버는 자신이 무엇을 제공할 수 있는지 목록으로 정리해 두었다가, 클라이언트가 물어보면 그 목록을 알려 주고, 실제 요청이 오면 기능을 실행해 결과를 돌려줍니다. 중요한 점은 서버가 호스트의 내부 사정을 알 필요가 없다는 것입니다. 서버는 그저 정해진 규칙대로 들어온 요청을 처리할 뿐이고, 그래서 같은 서버를 여러 호스트가 돌려 쓸 수 있습니다.

세 자리의 책임을 한자리에 모아 보면 경계가 분명해집니다. 호스트는 무엇을 할지 결정하고 모델을 굴립니다. 클라이언트는 호스트를 대신해 서버와 연결하고 메시지를 나릅니다. 서버는 실제 기능을 들고 있다가 요청대로 실행합니다. 표로 정리하면 다음과 같습니다.

| 자리 | 위치 | 핵심 책임 | 연결 관계 |
|------|------|-----------|-----------|
| 호스트 | 에이전트 앱 | 모델 구동, 도구 사용 결정, 흐름 지휘 | 여러 클라이언트를 거느림 |
| 클라이언트 | 호스트 내부 | 서버와의 연결·메시지 중계 | 서버 하나와 1:1 |
| 서버 | 호스트 바깥 | 도구·데이터 제공, 요청 실행 | 클라이언트의 상대 한쪽 |

이 셋이 어떻게 한 줄로 이어지는지를 그림으로 보면 다음과 같습니다. 호스트가 안에 여러 클라이언트를 품고, 각 클라이언트가 바깥의 서버 하나씩과 짝을 맺는 모양입니다.

```
            호스트 (에이전트 앱)
        +-----------------------------+
        |   모델                      |
        |                             |
        |  클라이언트 A --- 1:1 --- [ 서버 1 ]  파일 도구
        |  클라이언트 B --- 1:1 --- [ 서버 2 ]  DB 조회
        |  클라이언트 C --- 1:1 --- [ 서버 3 ]  외부 API
        +-----------------------------+
```

이제 실제로 한 번의 요청이 이 구조를 어떻게 통과하는지, 데이터의 흐름을 처음부터 끝까지 따라가 보겠습니다. 상황은 이렇습니다. 사용자가 호스트에게 어떤 파일의 내용을 정리해 달라고 부탁했고, 그 파일을 다루는 기능은 서버 1이 들고 있다고 합시다.

흐름은 다음 단계로 진행됩니다.

```
1. 사용자  ->  호스트        : "이 파일 내용 정리해줘"
2. 호스트  ->  모델          : 요청 + 쓸 수 있는 도구 목록 전달
3. 모델    ->  호스트        : "파일 읽기 도구를 써라" (도구 호출 요청)
4. 호스트  ->  클라이언트 A  : 해당 요청 전달
5. 클라이언트 A -> 서버 1    : 표준 메시지로 도구 실행 요청
6. 서버 1  ->  클라이언트 A  : 파일 내용을 결과로 반환
7. 클라이언트 A -> 호스트    : 결과 전달
8. 호스트  ->  모델          : 결과를 넣어 다시 추론 요청
9. 모델    ->  호스트        : 정리된 답변 생성
10. 호스트 ->  사용자        : 최종 답변 표시
```

이 흐름을 단계별로 풀어 보겠습니다. 1번에서 사용자의 부탁이 호스트로 들어옵니다. 2번에서 호스트는 그 부탁과 함께, 지금 연결된 서버들이 제공하는 도구 목록을 모델에게 같이 넘깁니다. 모델이 무엇을 쓸 수 있는지 알아야 고를 수 있기 때문입니다.

3번이 분기점입니다. 모델은 답을 바로 쓰는 대신, 파일 읽기 도구가 필요하다고 판단해 그 도구를 써 달라고 요청합니다. 모델은 도구를 직접 실행하지 못합니다. 다만 어떤 도구를 어떤 입력으로 부르고 싶은지를 호스트에게 말할 뿐입니다.

4번과 5번에서 호스트는 그 요청을 서버 1과 짝지어 둔 클라이언트 A에게 넘기고, 클라이언트 A는 그것을 정해진 메시지 규칙에 맞춰 서버 1에게 보냅니다. 6번에서 서버 1이 실제로 파일을 읽어 그 내용을 결과로 돌려줍니다. 실행이 일어나는 곳은 오직 서버라는 점에 주목하면 좋습니다.

7번과 8번에서 결과가 클라이언트 A를 거쳐 호스트로 돌아오고, 호스트는 그 결과를 모델에게 다시 넣습니다. 이제 모델은 파일 내용을 손에 쥔 채로 9번에서 정리된 답을 만들고, 10번에서 호스트가 그 답을 사용자에게 보여 줍니다. 한 바퀴가 이렇게 닫힙니다.

이 흐름에서 책임이 어떻게 갈라지는지 다시 확인해 두면 좋습니다. 무엇을 할지 정하는 판단은 모델과 호스트 쪽에 있고, 그 판단을 바깥과 이어 주는 운반은 클라이언트가 하며, 실제 일을 해내는 실행은 서버에 있습니다. 판단·운반·실행이 서로 다른 자리에 놓인다는 점이 이 구조의 핵심입니다.

이렇게 나뉘어 있기 때문에 도구 하나를 새로 붙이고 싶으면 서버를 하나 더 띄우고 클라이언트를 하나 더 짝지으면 됩니다. 호스트의 핵심 논리나 모델을 건드릴 필요가 없습니다. 반대로 어떤 도구를 더는 쓰지 않기로 하면, 그 클라이언트와 서버 연결만 끊으면 됩니다. 도구를 제공하는 서버는 호스트와 따로 만들어 두었으므로, 같은 서버를 다른 에이전트 앱이 그대로 가져다 쓸 수도 있습니다. 이것이 MCP가 세 자리로 책임을 나눠 둔 가장 큰 실익입니다.

서버가 실제로 무엇을 내놓는지, 즉 도구와 데이터와 템플릿이 어떤 모양으로 제공되는지는 아직 다루지 않았습니다. 이 단원에서는 호스트·클라이언트·서버라는 뼈대와 그 사이를 오가는 데이터 흐름까지만 짚습니다. 서버가 제공하는 것의 종류는 다음 단원의 몫으로 둡니다.

정리하면, MCP는 에이전트의 도구 연결을 호스트·클라이언트·서버 세 자리로 나눈 구조입니다. 호스트는 모델을 굴리며 흐름을 지휘하고, 클라이언트는 호스트 안에서 서버와 1:1로 짝을 맺어 메시지를 나르며, 서버는 도구와 데이터를 들고 있다가 요청대로 실행합니다. 한 번의 요청은 사용자에서 호스트와 모델을 거쳐 클라이언트와 서버로 내려갔다가, 결과를 안고 같은 길을 되짚어 올라오는 한 바퀴로 처리됩니다.

다음 단원인 4.2.2에서는 서버가 노출하는 세 가지 기본형, 곧 도구·리소스·프롬프트가 각각 무엇이고 언제 쓰이는지를 다룹니다.

이 단원을 마치면 MCP의 호스트·클라이언트·서버 구조와 각 자리의 역할, 클라이언트와 서버의 1:1 연결, 그리고 한 요청이 세 자리를 통과하는 전체 데이터 흐름을 스스로 설명할 수 있습니다.
