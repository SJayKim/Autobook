# 4.2.5 기존 도구를 MCP 서버로 감싸기

새 도구를 처음부터 MCP 서버로 만드는 일은 드뭅니다. 대부분의 팀은 이미 동작하는 자산을 가지고 있습니다. 사내 결제 API, 데이터베이스를 조회하는 함수, 사외 날씨 서비스를 호출하는 작은 모듈 같은 것들입니다. 이런 자산은 사람이나 다른 프로그램이 쓰도록 만들어졌을 뿐, 에이전트가 직접 호출하도록 설계되지는 않았습니다. 그래서 에이전트에게 이 기능을 주려면 이미 있는 것을 그대로 다시 쓰면서, 그 위에 표준 인터페이스만 얇게 덧입히는 방법이 필요합니다.

4.2.4에서는 도구 하나를 노출하는 최소 MCP 서버를 처음부터 정의해 보았습니다. 서버 초기화, 도구 등록, 요청 처리, 결과 반환의 순서를 따라가며 빈 서버에 도구의 동작을 직접 채워 넣었습니다. 이번 단원은 그 빈자리에 새 코드를 적는 대신, 이미 있는 함수를 그 자리에 연결하는 일을 다룹니다. 핵심 동작은 기존 코드가 그대로 수행하고, MCP 서버는 그 앞에 서서 통역만 맡습니다.

이렇게 기존 기능 앞에 표준 인터페이스 한 겹을 두르는 일을 **래핑**이라고 합니다. 래핑은 원래 기능을 고치지 않습니다. 입력을 받아 원래 함수가 알아듣는 형태로 바꿔 전달하고, 원래 함수가 돌려준 결과를 표준 형태로 바꿔 돌려주는 얇은 중간층을 새로 둘 뿐입니다. 다음 다이어그램이 래핑 전과 후의 차이를 보여 줍니다.

```
[래핑 전]
  사람/스크립트 ──직접 호출──> get_weather(city) ──> 날씨 데이터
  (에이전트는 이 함수를 부를 방법이 없음)

[래핑 후]
  에이전트 ──MCP 표준 호출──> [MCP 서버: 래퍼]
                                  │ 1. 입력 변환
                                  ▼
                              get_weather(city)   (기존 함수 그대로)
                                  │ 2. 결과 변환
                                  ▼
            <──MCP 표준 응답──  표준 형식 결과
```

래퍼가 하는 일은 크게 네 가지입니다. 함수의 입력 모양을 도구 스키마로 옮기는 일, 오류를 표준 형식으로 정리하는 일, 인증 정보를 끼워 넣는 일, 그리고 무엇을 노출하고 무엇을 가릴지 정하는 일입니다. 하나씩 차례로 살펴봅니다.

첫 번째는 **함수 시그니처**를 도구 입력 스키마로 옮기는 일입니다. 함수 시그니처란 함수의 이름과 그 함수가 받는 매개변수들의 목록을 뜻합니다. 예를 들어 `get_weather(city: str, days: int = 1)`라는 함수가 있다면, 이 함수는 도시 이름 `city`를 반드시 받고, 조회할 날짜 수 `days`는 생략하면 1로 둔다는 약속을 담고 있습니다. 사람은 이 시그니처를 보고 어떻게 호출할지 압니다. 하지만 4.2.4에서 보았듯이, 모델은 도구를 호출할 때 입력 스키마를 보고 인자를 채웁니다. 따라서 시그니처에 담긴 약속을 입력 스키마라는 형식으로 다시 적어 줘야 합니다.

옮기는 규칙은 단순합니다. 매개변수 하나가 스키마의 속성 하나가 됩니다. 매개변수의 자료형은 스키마의 타입으로, 기본값이 없는 매개변수는 필수 항목으로 표시합니다. 아래는 위 함수를 도구 등록용 정보로 옮긴 모습입니다.

```
도구 이름: get_weather
설명: 도시 이름으로 향후 날씨를 조회한다. days를 주면 그 일수만큼 예보를 반환한다.
입력 스키마:
  city  (문자열, 필수)   조회할 도시 이름
  days  (정수, 선택, 기본 1)  조회할 날짜 수
```

여기서 함수에는 없던 정보가 두 가지 더해졌습니다. 도구의 설명과 각 인자의 설명입니다. 함수 시그니처는 형태만 알려 줄 뿐 의미를 알려 주지 않습니다. 모델은 이 설명을 읽고 도구를 언제 어떻게 쓸지 판단하므로, 사람이 함수 주석을 읽듯 명확한 문장으로 적어야 합니다. 이 설명은 단순한 주석이 아니라 도구를 쓸지 말지 결정하는 실제 입력입니다.

시그니처를 옮길 때 주의할 점이 하나 더 있습니다. 함수에는 있지만 도구로는 내보내지 말아야 할 매개변수가 종종 섞여 있다는 점입니다. 방금 본 함수가 사실 `get_weather(city, days=1, api_key=None, debug=False)`였다고 합시다. `api_key`는 비밀 값이라 모델이 채워서는 안 되고, `debug`는 내부 점검용이라 에이전트가 알 필요가 없습니다. 그래서 시그니처를 그대로 베끼지 않고, 모델이 정해야 할 인자만 골라 스키마에 남깁니다. 위 스키마에 `city`와 `days`만 적힌 것은 이런 선별을 거친 결과입니다. 나머지는 래퍼가 안에서 채웁니다.

두 번째는 오류와 예외의 **표준화**입니다. 기존 함수는 문제가 생기면 그 함수가 속한 언어나 라이브러리 방식대로 신호를 보냅니다. 어떤 함수는 예외를 던지고, 어떤 함수는 특정 숫자나 빈 값을 돌려주며, 어떤 외부 API는 자기만의 오류 코드를 응답에 실어 보냅니다. 이런 신호를 그대로 흘려보내면 에이전트는 무엇이 잘못됐는지 해석하지 못합니다.

그래서 래퍼는 모든 실패를 한 가지 형식으로 모읍니다. 원래 함수에서 어떤 종류의 실패가 나오든, 래퍼가 그것을 받아 MCP가 정한 오류 응답 형태로 바꿔 돌려줍니다. 다음 흐름이 그 과정을 보여 줍니다.

```
get_weather 실행
   ├─ 정상 → 결과를 표준 응답으로 변환해 반환
   └─ 실패 (예외/오류코드/빈값)
          └─ 래퍼가 가로챔
                └─ "도시를 찾지 못했습니다" 같은
                   사람이 읽을 메시지 + 오류 표시로 변환해 반환
```

이때 오류 메시지에는 다음에 무엇을 하면 되는지 단서를 담는 편이 좋습니다. 도시 이름의 철자가 틀렸다거나, 날짜 수가 허용 범위를 넘었다는 식입니다. 에이전트는 이 메시지를 읽고 인자를 고쳐 다시 시도할 수 있기 때문입니다. 반대로 내부 동작의 세세한 흔적을 그대로 토해 내면 도움이 되기는커녕 다음에 다룰 정보 유출 문제로 이어집니다.

세 번째는 인증 정보의 **주입**입니다. 인증 정보란 외부 서비스가 호출자를 확인하는 데 쓰는 비밀 값입니다. 외부 날씨 서비스가 API 키를 요구한다고 합시다. 이 키는 비밀이므로 에이전트에게 알려서도 안 되고, 모델이 호출 인자로 채우게 해서도 안 됩니다. 키가 대화에 노출되면 그대로 새어 나가기 때문입니다.

해법은 키를 도구 스키마 바깥에 두고, 래퍼가 호출 직전에 끼워 넣는 것입니다. 키는 서버를 띄울 때 환경에서 읽어 들이고, 모델에게 보이는 입력 스키마에는 `city`와 `days`만 남깁니다. 모델은 키의 존재조차 모른 채 도구를 부르고, 래퍼가 그 호출을 받아 실제 함수에 키를 더해 전달합니다.

```
모델이 보는 입력:   city, days        ← 키 없음
래퍼가 만드는 호출:  city, days, api_key  ← 서버 환경에서 키 주입
```

이렇게 하면 비밀 값을 다루는 책임이 모델이 아니라 서버에 머뭅니다. 모델은 무엇을 할지만 정하고, 누구의 자격으로 할지는 서버가 책임집니다.

마지막으로 **래핑 시 흔한 함정** 세 가지를 짚습니다. 모두 동작은 멀쩡한데 에이전트가 도구를 잘못 쓰게 만드는 문제라, 처음에는 알아차리기 어렵습니다.

첫째는 과도한 도구 노출입니다. 기존 모듈에 함수가 서른 개 있다고 그 서른 개를 모두 도구로 등록하면, 모델은 매번 비슷비슷한 도구 사이에서 헤매고 엉뚱한 것을 고릅니다. 도구가 늘수록 각 도구의 설명도 모델의 맥락을 차지하므로, 쓰지 않는 도구는 비용만 늘리고 선택을 흐립니다. 에이전트가 실제로 할 일에 필요한 함수만 골라 노출하고, 나머지는 래퍼 안에서만 쓰는 편이 정확도를 높입니다.

둘째는 모호한 설명입니다. 도구 이름을 `proc` 같은 약자로 두거나 설명을 비워 두면, 모델은 그 도구가 무엇을 하는지 추측해야 합니다. 추측은 자주 틀립니다. 이름은 동작이 드러나게, 설명은 언제 쓰고 무엇을 돌려주는지 한 문장으로 적어야 합니다. 앞서 입력 스키마에 인자 설명을 더한 것과 같은 이유입니다.

셋째는 민감정보 유출입니다. 기존 함수가 돌려주는 결과에는 에이전트가 알 필요 없는 값이 섞여 있곤 합니다. 사용자의 주민등록번호, 내부 식별자, 디버깅용 전체 응답 같은 것입니다. 래퍼가 결과를 그대로 흘려보내면 이 값들이 모델의 맥락에 들어가 그대로 새어 나갈 수 있습니다. 결과에서 에이전트에게 필요한 항목만 추려 내보내는 일도 래퍼의 몫입니다.

정리하면, 기존 도구를 MCP 서버로 감싼다는 것은 동작을 새로 만드는 일이 아니라, 이미 있는 함수 앞에 통역 한 겹을 두르는 일입니다. 함수 시그니처를 입력 스키마로 옮기고, 갖가지 실패를 한 가지 표준 오류 형식으로 모으고, 인증 정보는 모델이 아닌 서버가 주입하며, 노출할 도구와 내보낼 값을 신중히 골라 과도한 노출과 모호한 설명과 정보 유출을 피하는 것이 핵심입니다.

다음 단원인 4.2.6에서는 MCP 보안과 신뢰를 다룹니다. 이번 단원에서 잠깐 비친 정보 유출과 권한 문제를, 신뢰할 수 없는 서버 연결과 도구 설명을 통한 공격까지 넓혀 살펴봅니다.

이 단원을 마치면 이미 있는 API나 함수를 MCP 서버로 감싸 표준 인터페이스로 노출하는 방법을, 시그니처 변환과 오류 표준화와 인증 주입과 흔한 함정의 회피까지 포함해 설명할 수 있습니다.
