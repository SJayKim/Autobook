# 5.4.4 한 걸음씩 만드는 장기 실행 에이전트

앞 단원인 5.4.3에서는 코딩 에이전트라는 한 사례를 골라, 컨텍스트와 메모리와 프로토콜과 하네스라는 네 축이 실제 도구 안에서 어떤 부품으로 박혀 있는지를 해부했습니다. 그 단원을 읽고 나면 며칠씩 일하는 에이전트가 무엇으로 이루어지는지는 손에 잡힙니다. 그런데 막상 직접 하나를 만들려 들면 막막함이 닥칩니다. 네 축을 처음부터 다 갖추려 하면 어디서부터 손대야 할지 모르겠고, 한 축이 다른 축을 전제로 하니 무엇을 먼저 세워야 하는지도 헷갈립니다.

이 단원은 그 막막함을 푸는 데 목적이 있습니다. 네 축을 한꺼번에 쌓는 대신, 가장 작은 루프 하나에서 출발해 한 축씩 더해 가는 성장 로드맵을 따라갑니다. 각 단계마다 무엇이 더해지는지, 그리고 그 추가로 에이전트가 무엇을 새로 할 수 있게 되는지를 짚습니다. 한 단계가 끝날 때마다 에이전트는 돌아가는 상태를 유지하므로, 중간에 멈춰도 손에 쥐는 것은 동작하는 무언가입니다.

출발점은 가장 작은 동작 단위입니다. 5.4.2에서 설계의 순서를 정할 때 의존의 바닥에 컨텍스트가 있다고 했지만, 그 컨텍스트조차 담을 그릇이 먼저 있어야 합니다. 그 그릇이 바로 1.2.2에서 배운 ReAct 루프입니다. ReAct는 한 턴마다 생각으로 무엇을 할지 정하고, 행동으로 도구를 한 번 부르고, 관찰로 그 결과를 읽어 다음 생각의 근거로 삼는 패턴이었습니다. 이 생각-행동-관찰의 반복이 모든 에이전트의 가장 작은 심장입니다.

```
[0단계] 최소 ReAct 루프

  사용자 입력
     │
     ▼
  ┌─────────────────────────────┐
  │ 생각 → 행동(도구 호출) → 관찰 │ ← 종료 조건까지 반복
  └─────────────────────────────┘
     │
     ▼
  최종 답변

  할 수 있는 것: 몇 분짜리 단발 작업 한 건
  아직 못 하는 것: 기록이 쌓이면 멈춤, 세션 넘으면 망각
```

이 최소 루프만으로도 에이전트는 몇 분짜리 단발 작업을 해냅니다. 사용자가 묻고, 에이전트가 도구 한둘을 부르고, 답을 내고 끝나는 정도입니다. 문제는 이 루프를 오래 돌릴 때 드러납니다. 생각과 행동과 관찰이 쌓이면 작업 공간이 곧 넘치고, 세션이 닫히면 그동안 알아낸 것이 모두 사라집니다. 며칠은커녕 한 시간을 버티기 어렵습니다.

첫 번째로 더할 축은 컨텍스트 관리입니다. 2장에서 다룬 컨텍스트는 모델이 지금 이 순간 보는 작업 공간이 넘치지 않게 다스리는 축이었습니다. 최소 루프에 이 축을 얹으면, 턴이 쌓여도 작업 공간이 토큰 예산을 넘지 않도록 정리됩니다. 핵심 장치는 압축입니다. 기록이 임계값에 가까워지면 지난 기록을 요약으로 줄여 자리를 벌되, 작업의 목표와 그동안 내린 결정과 아직 못 푼 과제 같은 핵심 상태는 남기는 상태 보존 압축을 씁니다.

```
[1단계] + 컨텍스트 관리 (2장)

  사용자 입력
     │
     ▼
  ┌─────────────────────────────┐
  │ 생각 → 행동 → 관찰  (반복)    │
  │   ↑                          │
  │   └─ 기록이 넘칠 듯하면        │
  │      압축으로 줄임(핵심은 보존) │
  └─────────────────────────────┘

  새로 할 수 있는 것: 한 세션을 길게(수십~수백 턴) 버팀
  아직 못 하는 것: 세션이 닫히면 여전히 다 잊음
```

이 단계를 거치면 에이전트는 한 세션을 훨씬 길게 버팁니다. 압축이 자리를 계속 벌어 주므로 수십, 수백 턴을 돌고도 멈추지 않습니다. 다만 한계가 또렷합니다. 압축은 어디까지나 한 세션 안의 작업 공간을 다스리는 일입니다. 세션이 닫히면 압축으로 추려 둔 요약마저 사라지고, 이튿날 같은 작업을 열면 에이전트는 어제 알아낸 것을 하나도 기억하지 못한 채 처음부터 시작합니다. 며칠을 일하려면 세션과 세션을 잇는 다리가 있어야 합니다.

두 번째로 더할 축은 메모리입니다. 3장에서 다룬 메모리는 세션이 끝나도 지식이 사라지지 않게 이어 주는 축이었습니다. 컨텍스트가 한 세션 안의 작업 공간을 다스린다면, 메모리는 그 작업 공간을 세션 바깥의 영속 저장으로 이어 줍니다. 이 단계에서 에이전트는 일하다 알아낸 일반화할 만한 사실을 영속 저장에 적어 두고, 새 세션이 열리면 지금 작업에 관련된 것만 골라 회상합니다. 그래서 어제 헤맨 길을 오늘 다시 헤매지 않습니다.

```
[2단계] + 메모리 (3장)

  새 세션 시작
     │
     ▼
  [영속 저장에서 관련 지식 회상] ──┐
     │                            │
     ▼                            │
  ┌─────────────────────────────┐│
  │ 생각 → 행동 → 관찰 (+ 압축)   ││
  └─────────────────────────────┘│
     │                            │
     ▼                            │
  [배운 사실을 영속 저장에 승급] ──┘ (다음 세션이 회상)

  새로 할 수 있는 것: 며칠에 걸친 작업의 연속성
  아직 못 하는 것: 내부 도구 몇 개로만 일함, 외부 세계와 단절
```

메모리까지 얹으면 에이전트는 비로소 며칠에 걸친 작업의 연속성을 얻습니다. 첫날 알아낸 사실이 둘째 날, 셋째 날로 이어지므로, 같은 실수를 되풀이하지 않고 일이 누적됩니다. 그러나 아직 에이전트가 손댈 수 있는 세계는 좁습니다. 처음부터 붙여 둔 몇 개의 내부 도구로만 일하므로, 데이터베이스를 조회하거나 외부 시스템을 호출하지는 못합니다. 새 능력이 필요할 때마다 맞춤 연동 코드를 일일이 짜 넣는 것도 며칠짜리 작업에는 무거운 짐입니다.

세 번째로 더할 축은 도구와 프로토콜입니다. 4장에서 다룬 프로토콜은 도구와 다른 에이전트를 표준화된 방식으로 연결하는 축이었고, 그 핵심에 4.2에서 본 MCP가 있었습니다. 이 단계에서 에이전트는 호스트 역할을 맡아, 필요한 외부 기능을 가진 MCP 서버를 표준 인터페이스로 꽂습니다. 서버가 노출한 도구가 에이전트의 행동 목록에 더해지므로, 새 능력을 붙일 때마다 전용 배선을 다시 짜지 않아도 됩니다.

```
[3단계] + 도구·프로토콜 (4장)

  ┌─────────────────────────────┐
  │ 생각 → 행동 → 관찰 (+ 압축)   │
  │           │                  │
  │           ▼  표준(MCP)으로 호출 │
  │   [내부 도구] [외부 서버 A] [외부 서버 B] ...
  └─────────────────────────────┘
        │
   회상/승급(메모리) — 압축(컨텍스트) — 표준 연결(프로토콜)

  새로 할 수 있는 것: 외부 시스템까지 다루는 폭넓은 작업
  아직 못 하는 것: 무인 시간대에 잘못을 잡거나 되돌릴 그물이 없음
```

프로토콜까지 더하면 에이전트의 행동 반경이 넓어집니다. 내부 도구만으로 닫혀 있던 작업이 외부 데이터베이스와 사내 시스템으로 뻗고, 표준 한 줄에 꽂는 방식이라 새 서버를 얼마든지 더 붙일 수 있습니다. 그런데 폭이 넓어진 만큼 위험도 커집니다. 에이전트가 파일을 지우거나 외부로 무언가를 내보내는 행동까지 하게 되었는데, 사람이 자리를 비운 며칠 동안 그 행동을 누가 검사하느냐 하는 문제가 남습니다. 작은 실수가 무인 시간대에 되돌릴 수 없는 사고로 번질 자리가 생긴 것입니다.

마지막으로 더할 축은 하네스입니다. 5장 전체가 다룬 하네스는 앞의 세 축을 며칠 동안 안정되고 안전하게 굴리는 바깥 껍질이었습니다. 이 단계에서 에이전트는 세 가지 장치를 갖춥니다. 첫째는 관측입니다. 에이전트가 무엇을 했고 토큰과 비용을 얼마나 썼는지를 기록해, 무인 진행을 사람이 나중에 따라 읽을 수 있게 합니다. 둘째는 승인 경계입니다. 5.3.3에서 본 대로 행동을 위험 등급으로 나누어, 읽기처럼 안전한 행동은 자동으로 통과시키고 삭제나 배포처럼 비가역적인 행동은 실행 직전에 멈춰 사람의 승인을 받습니다. 셋째는 롤백입니다. 5.3.2에서 본 대로 되돌아갈 지점을 미리 남겨, 이미 어긋난 진행을 성한 지점부터 다시 시작할 수 있게 합니다.

```
[4단계] + 하네스: 승인·롤백·관측 (5장)

  ┌───────────────────────────────────────────┐
  │ 회상(메모리) → 압축(컨텍스트) → 생각·행동·관찰 │
  │                            │                │
  │                            ▼ 표준(프로토콜)  │
  │                       도구·외부 서버 호출      │
  │                            │                │
  │            ┌───────────────┴──────────────┐ │
  │            │ 하네스가 감쌈                  │ │
  │            │  · 관측: 행동·비용을 기록      │ │
  │            │  · 승인: 위험 행동 앞에서 멈춤  │ │
  │            │  · 롤백: 어긋나면 성한 지점으로 │ │
  │            └──────────────────────────────┘ │
  └───────────────────────────────────────────┘

  새로 할 수 있는 것: 사람이 자리를 비운 며칠을 안전하게 무인 진행
  사람의 역할: 핵심 결정 지점(승인 요청)에서만 전략적 감독
```

하네스까지 얹으면 비로소 며칠씩 일하는 에이전트가 완성됩니다. 사람이 자리를 비운 시간대에도 에이전트는 안전한 행동은 스스로 진행하고, 위험한 행동 앞에서는 멈춰 승인을 기다리며, 잘못 나아간 진행은 되감습니다. 사람은 모든 단계를 지켜볼 필요 없이, 승인이 필요한 핵심 결정 지점에서만 개입합니다. 책의 첫머리에서 말한, 인간이 전략적 감독만 제공하며 며칠씩 작동하는 에이전트의 모습이 여기서 완성됩니다.

이 로드맵에서 한 가지를 강조해 두겠습니다. 단계의 순서는 5.4.2에서 정한 설계 순서와 그대로 같습니다. 의존의 바닥인 컨텍스트를 먼저 세우고, 그 위에 메모리, 프로토콜, 하네스를 차례로 얹는 순서입니다. 이 순서를 지키는 까닭은 각 축이 아래 축을 전제로 하기 때문입니다. 압축할 작업 공간이 있어야 메모리를 어디에 넣을지 정하고, 작업 공간과 기억이 자리 잡아야 외부 도구가 깔끔하게 들어오며, 그 셋이 돌아가야 하네스가 감쌀 알맹이가 생깁니다. 또한 자율성은 한꺼번에 높이지 않습니다. 하네스가 제 일을 한다는 확인이 쌓인 뒤에야 읽기에서 쓰기, 실행으로 권한을 넓힙니다.

정리하면, 며칠씩 일하는 에이전트는 한 번에 완성하는 것이 아니라 가장 작은 루프에서 한 축씩 키워 가는 것입니다. 1.2.2의 최소 ReAct 루프로 단발 작업을 해내고, 2장의 컨텍스트 관리를 더해 한 세션을 길게 버티고, 3장의 메모리를 더해 며칠의 연속성을 얻고, 4장의 도구와 프로토콜을 더해 외부 세계까지 다루고, 마지막으로 5장의 하네스로 승인과 롤백과 관측을 갖춰 무인 진행을 안전하게 만듭니다. 각 단계는 앞 단계 위에 누적되며, 더해질 때마다 에이전트가 새로 할 수 있는 일이 늘고 동시에 다음 한계가 드러나 그다음 축을 부릅니다. 다음 단원인 5.4.5에서는 이렇게 완성한 장기 실행 에이전트가 앞으로 어디로 나아갈지, 표준의 성숙과 자율성과 거버넌스의 균형, 평가와 신뢰의 과제를 짚으며 책 전체를 회고합니다.

이 단원을 마치면 최소 ReAct 루프에서 출발해 컨텍스트, 메모리, 프로토콜, 하네스를 한 단계씩 누적해 며칠씩 일하는 에이전트로 키우는 로드맵을, 각 단계에서 무엇이 더해지고 무엇이 새로 가능해지는지를 들어 설명할 수 있습니다.
