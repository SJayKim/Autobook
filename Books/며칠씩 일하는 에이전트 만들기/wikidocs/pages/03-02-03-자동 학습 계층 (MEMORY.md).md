# 3.2.3 자동 학습 계층 (MEMORY.md)

며칠씩 이어지는 작업을 하다 보면, 에이전트는 같은 실수를 두 번 세 번 되풀이하곤 합니다. 첫째 날 빌드 명령이 특정 폴더에서만 동작한다는 사실을 어렵게 알아냈는데, 이튿날 새 세션에서 또 엉뚱한 폴더에서 명령을 돌려 실패하는 식입니다. 바로 앞 단원인 3.2.2에서는 이런 일을 막는 한 가지 방법으로, 사람이 직접 규칙을 적어 두는 명시적 규칙 계층을 다뤘습니다. 사람이 'CLAUDE.md 같은 파일에 빌드 명령을 적어 두면, 에이전트가 매번 그 글을 읽고 시작한다'는 것이었습니다. 그런데 사람이 미리 적어 두지 못한 사실은 어떻게 될까요. 이 단원에서는 에이전트가 일하다 스스로 알아낸 것을 사람의 손을 거치지 않고 직접 적어 두는 자동 학습 계층을 살펴봅니다.

이 계층이 왜 필요한지부터 짚겠습니다. 3.2.2에서 본 명시적 규칙은 사람이 미리 아는 것만 담을 수 있습니다. 프로젝트를 처음 시작할 때 사람이 모든 함정을 다 알지는 못합니다. 정작 중요한 사실은 에이전트가 일하다 부딪혀 봐야 드러납니다. 앞의 예처럼 '빌드는 루트가 아니라 하위 폴더에서 돌려야 한다'는 것은, 한 번 실패해 본 뒤에야 알게 되는 종류의 지식입니다. 이런 지식을 매번 사람이 받아 적어 규칙 파일에 넣어 주는 것은 번거롭고, 사람이 자리를 비우면 아예 기록되지 못합니다. 그래서 에이전트가 일하는 중에 얻은 사실을 스스로 적어 두는 장치가 필요합니다.

여기서 등장하는 것이 **자동 학습 계층**입니다. 자동 학습 계층이란, 에이전트가 한 세션 동안 스스로 알아낸 사실을 사람의 지시 없이 직접 기록해 두는 영속 저장소를 뜻합니다. 여기서 세션이란 3.1.2에서 약속한 대로 대화 한 번, 즉 작업을 한 차례 주고받는 단위를 말합니다. 이 저장소는 흔히 MEMORY.md라는 이름의 파일이나 그 파일이 가리키는 메모리 폴더로 구현됩니다. 이름이 무엇이든 핵심은 같습니다. 쓰는 주체가 사람이 아니라 에이전트 자신이라는 점입니다.

이 '쓰는 주체'의 차이가 앞 계층과 가르는 가장 큰 갈림입니다. 3.2.2의 명시적 규칙은 사람이 손으로 적습니다. 반면 자동 학습 계층은 에이전트가 일하다 적습니다. 그래서 전자를 명시적이라 부르고, 후자를 자동이라 부릅니다. 명시적이라는 말은 사람이 의도를 분명히 밝혀 적어 둔다는 뜻이고, 자동이라는 말은 사람이 일일이 시키지 않아도 에이전트가 알아서 적는다는 뜻입니다.

그러면 에이전트는 무엇을 보고 '이건 적어 둘 만하다'고 판단할까요. 세션 중에 오가는 모든 글을 다 적지는 않습니다. 한 번 쓰고 버릴 일회성 정보나, 코드와 기록만 봐도 다시 알 수 있는 것은 적지 않습니다. 적어 두는 대상은 다음 세션에서도 쓸모가 있을 일반화된 사실입니다. 앞서 든 빌드 폴더 이야기, 사용자가 밝힌 선호, 작업 중 받은 피드백 같은 것들입니다. 이렇게 세션이 끝나면 사라질 작업기억 가운데 살려 둘 가치가 있는 사실만 골라, 세션이 끝나기 전에 파일로 옮겨 적는 것이 **세션 중 학습의 저장**입니다.

저장하는 단위에도 규칙이 있습니다. 자동 학습 계층은 보통 한 사실을 한 덩어리로 끊어 적습니다. '빌드는 하위 폴더에서 돌린다'와 '테스트는 특정 환경 변수를 켜야 통과한다'는 서로 다른 사실이므로, 한 줄로 뭉치지 않고 따로 적습니다. 큰 프로젝트라면 사실 하나하나를 아예 별도 파일로 떼어 두기도 합니다. 이렇게 정보를 사실 하나 단위로 잘게 끊어 적는 방식을 **사실 단위 파일 구성**이라고 합니다. 사실을 잘게 끊어 두면, 나중에 한 사실이 틀린 것으로 드러났을 때 그 덩어리만 고치거나 지우면 됩니다. 여러 사실이 한 줄에 엉켜 있으면 한 부분만 손대기가 어렵습니다.

여기서 자연스레 한 가지 의문이 생깁니다. 사실을 잘게 쪼개 여러 파일에 흩어 두면, 정작 필요할 때 그 많은 사실 중에서 관련 있는 것을 어떻게 골라낼까요. 매 세션마다 쌓인 사실 전부를 컨텍스트에 부어 넣으면, 2.1.3에서 다룬 대로 무관한 글이 관련 신호를 묻어 버립니다. 그래서 자동 학습 계층은 흩어진 사실을 한자리에서 훑어볼 **인덱스**를 함께 둡니다.

인덱스란, 각 사실 또는 각 메모리 파일을 한 줄짜리 요약으로 정리해 둔 목록을 뜻합니다. MEMORY.md 같은 파일이 이 목록 노릇을 하는 경우가 많습니다. 목록의 각 줄에는 '무슨 내용인지'를 짧게 적은 설명과, 그 사실이 적힌 파일을 가리키는 표시가 담깁니다. 에이전트가 새 세션을 시작하면, 먼저 이 인덱스 한 장만 읽습니다. 그러고서 지금 하려는 일과 관련된 줄을 찾아, 그 줄이 가리키는 사실만 골라 컨텍스트로 끌어옵니다. 이렇게 인덱스를 먼저 보고 필요한 사실만 본문으로 가져오는 흐름이 **인덱스를 통한 로드**입니다.

이 흐름이 실제로 어떻게 도는지 작은 예로 보겠습니다. 인덱스에 '빌드는 하위 폴더에서 실행 — build-note 파일', '사용자는 짧은 답변 선호 — pref-note 파일', '결제 모듈은 외부 키 필요 — pay-note 파일' 세 줄이 적혀 있다고 합시다. 오늘 에이전트가 받은 일이 빌드를 다시 돌리는 작업이라면, 에이전트는 이 세 줄을 훑어 첫 줄만 지금 일과 맞닿아 있다고 판단합니다. 그래서 build-note 파일 하나만 컨텍스트로 가져오고, 나머지 두 파일은 건드리지 않습니다. 결제나 답변 길이에 관한 사실은 지금 일과 무관하므로, 컨텍스트에 끌어와 봐야 신호만 흐릴 뿐입니다. 인덱스의 한 줄 요약이 짧고 분명할수록 이 고르기가 정확해집니다.

전체 흐름을 한 그림으로 보겠습니다. 위쪽은 세션 중에 사실을 적어 두는 쓰기 흐름, 아래쪽은 새 세션에서 사실을 골라 읽어 오는 읽기 흐름입니다.

```
[ 쓰기: 세션 진행 중 ]
  에이전트가 작업하다 새 사실 발견
        |  (저장 가치 있는가?)
        v
  사실 한 덩어리로 끊어 파일에 기록
        |
        v
  인덱스에 한 줄 요약 + 파일 표시 추가

[ 읽기: 새 세션 시작 ]
  에이전트가 인덱스(한 줄 요약 목록)만 먼저 읽음
        |  (지금 일과 관련된 줄 고르기)
        v
  고른 줄이 가리키는 사실 파일만 컨텍스트로 로드
        |
        v
  그 사실을 근거로 작업 시작 (실수 반복 회피)
```

그림을 위에서부터 따라가 보겠습니다. 세션 중에 에이전트가 새 사실을 만나면, 먼저 그것이 저장할 가치가 있는지 가립니다. 가치가 있으면 사실 한 덩어리로 끊어 파일에 적고, 동시에 인덱스에 그 사실을 한 줄로 요약해 덧붙입니다. 시간이 지나 새 세션이 열리면, 에이전트는 사실 더미 전체가 아니라 인덱스 한 장만 읽습니다. 거기서 지금 하려는 일과 맞닿은 줄을 골라, 그 줄이 가리키는 파일만 컨텍스트로 가져옵니다. 이렇게 가져온 사실 덕분에, 어제 어렵게 알아낸 빌드 폴더 규칙을 오늘 다시 헤매지 않고 곧장 활용할 수 있습니다.

여기까지가 자동 학습 계층의 동작입니다. 다만 이 계층을 명시적 규칙 계층과 다시 한 번 나란히 놓아 차이를 분명히 해 두겠습니다. 두 계층은 모두 세션을 넘겨 살아남는 영속 저장소이지만, 쓰는 주체와 읽히는 방식이 다릅니다.

| 구분 | 명시적 규칙 계층 (3.2.2) | 자동 학습 계층 (3.2.3) |
|------|--------------------------|------------------------|
| 쓰는 주체 | 사람이 직접 작성 | 에이전트가 스스로 기록 |
| 쓰는 시점 | 미리, 사람이 아는 만큼 | 일하다, 새 사실을 만났을 때 |
| 담기는 내용 | 지켜야 할 규칙·금지사항·관례 | 세션 중 알아낸 사실 |
| 읽히는 방식 | 항상 통째로 로드 | 인덱스로 고른 것만 로드 |

표의 마지막 줄이 특히 중요합니다. 명시적 규칙은 분량이 적고 항상 지켜야 하므로 매번 통째로 읽어 들입니다. 반면 자동 학습한 사실은 시간이 갈수록 쌓여 양이 늘기 때문에, 전부 읽으면 컨텍스트가 감당하지 못합니다. 그래서 자동 학습 계층은 인덱스를 거쳐 필요한 것만 골라 읽는 방식을 택합니다. 둘의 무게도 다릅니다. 사람이 분명히 밝힌 규칙이 더 강한 지침이고, 에이전트가 스스로 적은 사실은 그를 보조합니다. 그래서 두 계층의 내용이 부딪히면 명시적 규칙을 우선합니다.

이 차이는 자연스러운 물음으로 이어집니다. 에이전트가 자동으로 적어 둔 사실이 매번 같은 모양으로 반복해 확인되면, 그것을 아예 사람의 규칙으로 끌어올려도 되지 않을까요. 실제로 그렇게 합니다. 자동 학습한 사실이 거듭 들어맞는 것으로 굳어지면, 자동 학습 계층에서 명시적 규칙 계층으로 옮겨 적어 더 강한 지침으로 만듭니다. 이렇게 한 계층의 정보를 더 강한 계층으로 끌어올리는 흐름을 승격이라고 하며, 그 구체적인 방식은 3.2.6에서 다룹니다. 이 단원에서는 자동 학습 계층 자체의 동작에 초점을 둡니다.

정리하면, 자동 학습 계층은 에이전트가 한 세션 동안 스스로 알아낸 사실을, 사람의 지시 없이 직접 파일에 기록해 두는 영속 저장소입니다. 저장할 가치가 있는 사실만 골라 한 덩어리씩 사실 단위로 끊어 적고, 각 사실을 한 줄 요약으로 모은 인덱스를 함께 둡니다. 새 세션에서는 이 인덱스를 먼저 읽어 관련된 사실만 컨텍스트로 골라 들이므로, 쌓인 기억이 많아도 컨텍스트를 어지럽히지 않습니다. 사람이 쓰고 항상 통째로 로드되는 3.2.2의 명시적 규칙 계층과 달리, 이 계층은 에이전트가 쓰고 인덱스로 골라 로드된다는 점이 핵심입니다. 다음 단원인 3.2.4에서는 평소에는 컨텍스트 밖에 두었다가 필요할 때 도구 호출로 읽고 쓰는 온디맨드 메모리 계층을 다룹니다.

이 단원을 마치면 자동 학습 계층이 세션 중에 사실을 어떻게 자동으로 기록하고 인덱스를 통해 골라 로드하는지, 그리고 사람이 쓰는 명시적 규칙 계층과 쓰기 주체·자동성에서 어떻게 다른지를 설명할 수 있습니다.
