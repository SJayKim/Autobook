# 4.2.4 MCP 서버 만들기

앞의 세 단원에서 우리는 MCP가 무엇으로 이루어지는지를 바깥에서 들여다보았습니다. 4.2.1에서 호스트와 클라이언트와 서버가 어떻게 나뉘는지를, 4.2.2에서 서버가 노출하는 도구와 리소스와 프롬프트라는 세 기본형을, 4.2.3에서 그 사이를 오가는 전송 방식인 stdio와 HTTP를 살펴보았습니다. 이번 단원은 관점을 안쪽으로 돌립니다. 지금까지는 이미 있는 서버에 연결하는 쪽이었다면, 이제는 서버 자체를 직접 만드는 쪽에 서 봅니다.

서버를 직접 만들 줄 알아야 하는 까닭은 분명합니다. 앞선 단원들은 모두 남이 만든 서버를 쓰는 이야기였습니다. 하지만 내가 가진 기능, 이를테면 사내 데이터베이스를 조회하는 함수나 특정 계산을 하는 코드를 에이전트가 쓰게 하려면, 그 기능을 도구로 노출하는 서버를 내가 세워야 합니다. 표준에 맞춰 한 번 노출해 두면, 그 서버는 어느 에이전트에든 그대로 붙습니다. 그래서 서버 만들기는 내 기능을 표준의 세계로 들여보내는 입구입니다.

목표는 거창하지 않습니다. 도구 하나만 노출하는 가장 작은 서버를 하나 세우고, 클라이언트가 그 도구를 부르는 흐름을 끝까지 따라가 보는 것입니다. 예로 들 도구는 두 숫자를 더해 결과를 돌려주는 add 하나입니다. 기능 자체는 단순하지만, 이 한 도구를 노출하는 과정에 서버 만들기의 모든 단계가 빠짐없이 들어 있습니다.

서버를 만드는 일은 특정 언어나 라이브러리에 매여 있지 않습니다. 4.2.3에서 본 대로 MCP는 메시지 형식만 약속하므로, 그 약속을 지키는 코드라면 어떤 언어로 짜도 서버가 됩니다. 그래서 이 단원은 실제 코드 대신 단계의 뼈대만 보여 주는 의사코드로 설명합니다. 의사코드란 문법은 어느 언어도 아니지만 사람이 읽으면 무슨 일을 하는지 알 수 있게 적은 코드 흐름을 말합니다.

서버 만들기는 다섯 단계로 나뉩니다. 먼저 전체 흐름을 한눈에 보고, 단계마다 무슨 일이 일어나는지 차례로 풀겠습니다.

```
[1] 서버 초기화   ->  서버 객체를 만들고 이름·전송 방식을 정한다
[2] 도구 등록     ->  이름·설명·입력 스키마로 add 도구를 알린다
[3] 요청 처리     ->  도구 호출이 오면 인자를 꺼내 실제 계산을 한다
[4] 결과 반환     ->  계산 결과를 약속된 형식으로 되돌려 보낸다
[5] 연결 확인     ->  클라이언트가 붙어 목록을 받고 한 번 불러 본다
```

첫째 단계는 **서버 초기화**입니다. 서버라는 빈 그릇을 하나 만들고, 그 그릇에 이름을 붙인 뒤 어떤 전송 방식으로 메시지를 주고받을지 정하는 일입니다. 4.2.3에서 보았듯 한 컴퓨터 안에서 쓰면 stdio를, 멀리 떨어진 곳에서 쓰면 HTTP를 고릅니다. 여기서는 가장 단순한 stdio를 가정하겠습니다. 이름은 클라이언트가 어느 서버에 붙었는지 알아보는 표지가 됩니다.

이 빈 그릇은 아직 아무 도구도 담고 있지 않습니다. 초기화가 하는 일은 메시지를 주고받을 통로를 여는 것까지이며, 무엇을 제공할지는 다음 단계에서 채웁니다. 초기화를 의사코드로 적으면 서버 = 서버만들기(이름="calc", 전송="stdio") 한 줄입니다. 여기서 calc는 이 서버에 붙인 이름이고, stdio는 4.2.3에서 본 표준 입출력 통로를 쓰겠다는 뜻입니다.

둘째 단계는 **도구 등록**입니다. 서버가 add라는 도구를 가지고 있다고 바깥에 알리는 일이며, 세 가지를 함께 적어야 합니다. 하나는 이름으로, 호출할 때 부르는 식별자인 add입니다. 둘은 설명으로, 이 도구가 무엇을 하는지를 한 줄로 적은 문장입니다. 셋은 입력 스키마로, 이 도구가 어떤 값을 받는지를 정해 둔 형식입니다.

이 가운데 입력 스키마는 처음 보는 말이니 배경부터 풀겠습니다. 스키마란 값이 어떤 모양이어야 하는지를 미리 정해 둔 틀입니다. add는 더할 두 숫자가 필요하므로, 스키마에는 a와 b라는 두 항목이 있고 둘 다 숫자라고 적습니다. 이 틀이 있어야 클라이언트가 무엇을 넣어야 하는지 알고, 서버도 들어온 값이 맞는 모양인지 확인할 수 있습니다.

설명과 스키마가 왜 중요한지는 4.2.2에서 본 모델의 처지를 떠올리면 분명해집니다. 모델은 도구의 속을 들여다보지 못하고, 오직 이름과 설명과 스키마만 보고 이 도구를 부를지 말지, 무슨 값을 넣을지를 판단합니다. 설명이 모호하면 엉뚱한 때에 부르고, 스키마가 부실하면 잘못된 값을 넣습니다. 그래서 이 셋은 사람이 아니라 모델이 읽는 안내문이라고 보아야 합니다.

도구 등록을 의사코드로 적으면 다음과 같습니다.

```
서버.도구등록(
  이름     = "add",
  설명     = "두 숫자 a와 b를 더해 합을 돌려준다",
  입력스키마 = { a: 숫자, b: 숫자 }
)
```

셋째 단계는 **요청 처리**입니다. 클라이언트가 add를 부르면, 서버는 그 호출에 담겨 온 인자에서 a와 b를 꺼내 실제로 더하는 일을 합니다. 이 처리 부분이 도구의 알맹이입니다. 등록은 도구가 있다고 알리는 일이고, 처리는 그 도구가 불렸을 때 정말로 무슨 일을 하는지를 적는 일입니다.

여기서 둘째 단계의 입력 스키마가 한 번 더 쓰입니다. 들어온 인자가 정말 두 숫자인지 처리하기 전에 확인할 수 있기 때문입니다. 만약 b가 빠졌거나 숫자가 아닌 값이 들어왔다면, 더하기를 시도하기 전에 잘못된 입력이라고 알리는 편이 안전합니다. 스키마가 등록 때는 모델에게 무엇을 넣을지 알려 주는 안내문이었다면, 처리 때는 들어온 값을 점검하는 잣대가 됩니다.

넷째 단계는 **결과 반환**입니다. 더한 값을 그냥 던지는 것이 아니라, MCP가 약속한 형식에 담아 되돌려 보냅니다. 4.2.3에서 본 대로 오가는 메시지에는 정해진 틀이 있으므로, 결과도 그 틀에 맞춰야 클라이언트가 알아듣습니다. 요청 처리와 결과 반환을 묶어 의사코드로 보면 다음과 같습니다.

```
서버.요청처리("add", 인자):
    a = 인자.a
    b = 인자.b
    합 = a + b
    반환 결과(합)
```

다섯째 단계는 **연결 확인**입니다. 서버를 다 만들었으면 정말 동작하는지 확인해야 합니다. 클라이언트가 서버에 붙으면, 4.2.3에서 본 연결 수명주기대로 먼저 초기화를 거쳐 서로의 능력을 주고받습니다. 이때 클라이언트는 서버에 어떤 도구가 있는지 목록을 달라고 요청하고, 서버는 등록해 둔 add 하나를 돌려줍니다. 그다음 a에 2, b에 3을 넣어 add를 한 번 불러 5가 돌아오면, 서버가 제대로 만들어진 것입니다.

지금까지 둘로 나뉘어 보이던 등록과 호출이 한 줄기로 이어지는 모습을 호출 흐름으로 정리하면 다음과 같습니다.

```
클라이언트                         서버(우리가 만든 것)
    |                                  |
    |--- 연결·초기화 ----------------->|  (전송: stdio)
    |<-- 능력 교환 --------------------|
    |                                  |
    |--- 도구 목록 요청 -------------->|
    |<-- [add: 설명·입력스키마] -------|   (2단계에서 등록한 정보)
    |                                  |
    |--- add 호출 {a:2, b:3} -------->|
    |                              [3단계] a+b 계산
    |<-- 결과 {합:5} ------------------|   [4단계] 약속된 형식으로 반환
    |                                  |
```

그림에서 보듯 우리가 둘째 단계에서 등록한 이름과 설명과 스키마는 목록 요청에 대한 응답으로 클라이언트에 건너가고, 셋째와 넷째 단계의 처리와 반환은 실제 호출이 들어왔을 때 작동합니다. 등록은 도구를 알리는 한 번의 준비이고, 처리와 반환은 호출이 올 때마다 되풀이되는 실행입니다.

정리하면, 최소 MCP 서버는 서버를 초기화하고, 이름과 설명과 입력 스키마로 도구를 등록하고, 호출이 오면 인자를 꺼내 처리하고, 그 결과를 약속된 형식으로 반환한 뒤, 클라이언트가 붙어 목록을 받고 한 번 불러 보는 다섯 단계로 완성됩니다. 도구가 하나뿐이어도 이 뼈대는 그대로이며, 도구를 더 노출하고 싶으면 둘째와 셋째 단계를 그 도구만큼 되풀이하면 됩니다. 다음 단원인 4.2.5에서는 이렇게 처음부터 만드는 대신, 이미 있는 API나 함수를 MCP 서버로 감싸 표준 인터페이스로 내보내는 방법을 다룹니다.

이 단원을 마치면 도구 하나를 노출하는 최소 MCP 서버를 초기화부터 도구 등록과 요청 처리와 결과 반환을 거쳐 클라이언트 연결 확인까지 단계별로 설명할 수 있습니다.
