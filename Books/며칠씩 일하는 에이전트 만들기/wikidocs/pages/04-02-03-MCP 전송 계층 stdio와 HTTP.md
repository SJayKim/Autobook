# 4.2.3 MCP 전송 계층: stdio와 HTTP

앞 단원 4.2.1에서 우리는 MCP가 호스트, 클라이언트, 서버라는 세 역할로 나뉜다는 것을 보았습니다. 호스트는 에이전트 앱이고, 그 안의 클라이언트가 도구를 제공하는 서버와 1:1로 짝을 이룹니다. 그런데 한 가지 빠진 질문이 있습니다. 클라이언트와 서버는 도대체 무엇을 타고 서로 말을 주고받을까요. 같은 컴퓨터 안에 있을 수도 있고, 인터넷 너머 다른 회사의 서버일 수도 있습니다. 이 둘은 물리적으로 거리가 완전히 다르므로, 메시지를 실어 나르는 통로도 달라야 합니다.

이 통로를 정하는 것이 바로 전송 계층입니다. 여기서 말하는 전송이란, 클라이언트가 보낸 요청 한 덩어리가 어떤 경로를 거쳐 서버에 도착하고, 서버의 답변이 어떤 경로로 되돌아오는지를 가리킵니다. MCP는 메시지의 내용 형식과 그 내용을 나르는 방법을 분리해서 정의합니다. 내용 형식은 어디서나 똑같이 쓰고, 나르는 방법만 상황에 맞게 고르는 구조입니다.

먼저 메시지의 내용 형식부터 봅니다. MCP에서 클라이언트와 서버가 주고받는 모든 메시지는 **JSON-RPC**라는 약속을 따릅니다. JSON은 중괄호와 키-값 쌍으로 데이터를 적는 텍스트 형식이고, RPC는 원격 프로시저 호출(Remote Procedure Call)의 줄임말로, 멀리 있는 상대에게 함수 하나를 대신 실행해 달라고 요청하는 방식을 뜻합니다. 즉 JSON-RPC는 함수 호출을 JSON 텍스트로 적어 보내는 약속입니다.

이 약속이 정하는 것은 단순합니다. 무언가를 요청할 때는 어떤 기능을 부를지(메서드 이름), 어떤 값을 넘길지(파라미터), 그리고 이 요청에 붙이는 식별 번호(id)를 함께 적습니다. 서버는 답을 돌려줄 때 같은 id를 다시 적어 보내므로, 여러 요청이 뒤섞여도 어떤 답이 어떤 요청에 대한 것인지 짝지을 수 있습니다. 도구 목록을 달라거나 특정 도구를 실행해 달라는 요청이 모두 이 형식으로 표현됩니다. 아래는 도구 목록을 요청하고 응답받는 흐름을 단순화한 예입니다.

```
요청 (클라이언트 → 서버)
{ "jsonrpc": "2.0", "id": 1, "method": "tools/list", "params": {} }

응답 (서버 → 클라이언트)
{ "jsonrpc": "2.0", "id": 1, "result": { "tools": [ ... ] } }
```

위에서 jsonrpc 필드는 이 메시지가 어떤 약속 버전을 따르는지 알리고, method는 부르려는 기능의 이름, params는 그 기능에 넘기는 입력값입니다. 응답에는 요청과 같은 id가 1로 적혀 있어 짝이 맞고, result 안에 실제 결과가 담깁니다. 만약 실패하면 result 대신 오류 내용이 담긴 error 필드가 옵니다. 중요한 점은, 이 텍스트 형식 자체는 통로가 stdio든 HTTP든 항상 똑같다는 것입니다. 바뀌는 것은 이 텍스트를 어떻게 실어 나르느냐뿐입니다.

이제 첫 번째 통로인 **stdio** 방식을 봅니다. stdio는 표준 입출력(standard input/output)의 줄임말입니다. 프로그램은 보통 키보드로 입력을 받고 화면으로 출력을 내보내는데, 이 입력 통로와 출력 통로를 운영체제가 기본으로 열어 줍니다. stdio 전송은 바로 이 기본 통로를 그대로 활용합니다. 호스트가 서버 프로그램을 하나의 하위 프로그램으로 직접 실행하고, 그 프로그램의 입력 통로에 JSON-RPC 메시지를 써넣고 출력 통로에서 답을 읽는 방식입니다.

stdio가 어울리는 상황은 서버가 같은 컴퓨터 안에 있을 때입니다. 예를 들어 내 노트북의 파일을 읽어 주는 MCP 서버라면, 굳이 네트워크를 거칠 필요 없이 호스트가 그 서버를 자식 프로그램으로 띄워 곧바로 통로를 잇습니다. 네트워크 주소도, 포트 번호도, 별도의 인증 절차도 필요 없습니다. 호스트와 서버가 한 기계 위에서 부모-자식 관계로 붙어 있으니, 가장 가깝고 간단한 연결입니다. 이런 이유로 stdio는 로컬 전송이라고 부릅니다.

두 번째 통로는 **HTTP** 방식입니다. HTTP는 웹 브라우저가 웹사이트를 불러올 때 쓰는 바로 그 통신 규약으로, 네트워크 너머의 상대에게 요청을 보내고 응답을 받는 표준입니다. MCP에서 HTTP 전송은 서버가 다른 기계, 즉 원격에 있을 때 씁니다. 서버는 특정 주소에서 요청을 기다리고, 클라이언트는 그 주소로 JSON-RPC 메시지를 보냅니다. 이 방식은 인터넷 어디에 있는 서버와도 이어질 수 있으므로 원격 전송이라고 부릅니다.

HTTP 전송에는 한 가지 더 짚을 점이 있습니다. 일반적인 웹 요청은 한 번 묻고 한 번 답하면 끝나지만, 도구 실행은 시간이 걸리거나 중간 진행 상황을 여러 번 흘려보내야 할 때가 있습니다. 그래서 MCP의 HTTP 전송은 스트리밍을 지원합니다. 스트리밍이란 하나의 연결을 잠시 열어 둔 채로 서버가 부분 결과나 진행 알림을 조금씩 내려보내는 방식입니다. 긴 작업의 중간 결과를 사용자에게 차례로 보여 줄 때 이 기능이 쓰입니다.

두 방식을 한자리에 놓고 비교하면 차이가 분명해집니다.

```
        stdio (로컬 전송)             HTTP (원격 전송)
------  --------------------------   --------------------------
위치    같은 컴퓨터 안               네트워크 너머 다른 기계
실행    호스트가 자식으로 직접 띄움   서버가 주소에서 대기
통로    표준 입출력                  HTTP 요청·응답(스트리밍 가능)
인증    대개 불필요(같은 기계)       네트워크 인증 필요
적합    개인 도구, 로컬 파일/명령     공용 서비스, 여러 사용자 공유
메시지  JSON-RPC(공통)               JSON-RPC(공통)
```

위 표에서 가장 아랫줄을 다시 보면, 메시지 형식은 어느 쪽이든 JSON-RPC로 동일합니다. 그러니 stdio와 HTTP 중 무엇을 고르느냐는 메시지를 바꾸는 결정이 아니라, 통로를 바꾸는 결정입니다. 선택 기준은 단순한 질문 하나로 정리됩니다. 서버가 나와 같은 컴퓨터 안에 있는가, 아니면 네트워크 너머에 있는가입니다. 내 기계 안의 파일이나 명령을 다루는 개인용 도구라면 stdio가 자연스럽고, 여러 사람이 함께 쓰는 공용 서비스나 다른 곳에서 운영하는 도구라면 HTTP가 맞습니다.

선택의 무게중심을 조금 더 풀어 보면, stdio는 설정이 거의 없고 빠르지만 그 서버를 띄운 한 기계 안에서만 쓸 수 있다는 한계가 있습니다. 반대로 HTTP는 여러 사용자와 여러 기계가 같은 서버를 공유할 수 있는 대신, 네트워크 주소를 정하고 누가 접근해도 되는지 가리는 인증을 갖춰야 합니다. 따라서 혼자 쓰는 로컬 도구를 빠르게 붙일 때는 stdio, 팀이나 외부에 공개해 함께 쓰는 도구를 운영할 때는 HTTP라고 기억하면 됩니다.

통로를 정했으면, 이제 그 통로 위에서 연결이 어떤 순서로 살았다 죽는지를 알아야 합니다. 이것이 **연결 수명주기**입니다. 수명주기란 연결이 처음 맺어져서 끝나 닫힐 때까지 거치는 단계들을 가리킵니다. 클라이언트가 서버와 이으면 곧바로 도구를 막 부르는 것이 아니라, 정해진 순서를 밟습니다. 이 순서는 stdio든 HTTP든 동일하게 적용됩니다.

수명주기는 네 단계로 이어집니다. 첫째는 초기화입니다. 클라이언트가 연결을 열고 자기가 어떤 약속 버전을 쓰는지 서버에 알립니다. 둘째는 능력 교환입니다. 서로 자신이 무엇을 할 수 있는지를 주고받는데, 클라이언트는 자신이 받을 수 있는 기능을, 서버는 자신이 제공하는 도구·리소스·프롬프트의 존재를 알립니다. 도구·리소스·프롬프트의 구분은 앞 단원 4.2.2에서 다룬 세 가지 기본형입니다. 셋째는 사용입니다. 비로소 클라이언트가 도구 목록을 받아 모델에 노출하고, 실제 도구 호출 요청과 응답을 주고받습니다. 넷째는 종료로, 더 쓸 일이 없으면 연결을 닫습니다.

이 네 단계를 한눈에 그리면 다음과 같습니다.

```
 [초기화]      → [능력 교환]        → [사용]            → [종료]
 연결 열고        서로 무엇을           도구 목록·호출       연결 닫기
 버전 알림        할 수 있는지 교환     요청과 응답 반복
```

이 순서가 정해져 있는 이유는, 사용 단계로 넘어가기 전에 양쪽이 서로의 약속 버전과 제공 기능을 먼저 맞춰 두어야 하기 때문입니다. 능력 교환을 건너뛰면 클라이언트는 서버에 어떤 도구가 있는지 모른 채 호출을 시도하게 됩니다. 그래서 초기화와 능력 교환이라는 준비 단계를 먼저 끝낸 다음에야 실제 사용이 시작됩니다. 종료는 자식 프로그램을 끝내는 형태일 수도 있고 네트워크 연결을 닫는 형태일 수도 있지만, 단계 자체는 통로와 무관하게 똑같습니다.

정리하면, MCP는 메시지의 내용 형식과 그것을 나르는 통로를 나누어 정의합니다. 내용은 어디서나 JSON-RPC로 같고, 통로는 같은 컴퓨터 안이면 표준 입출력을 쓰는 stdio, 네트워크 너머면 스트리밍을 지원하는 HTTP로 갈립니다. 선택 기준은 서버가 로컬인지 원격인지 한 가지 질문으로 좁혀집니다. 어느 통로를 쓰든 연결은 초기화, 능력 교환, 사용, 종료라는 같은 수명주기를 밟습니다.

다음 단원인 4.2.4에서는 이렇게 연결된 통로 위에서 실제로 동작하는 MCP 서버를 직접 만드는 과정을 다룹니다. 서버를 초기화하고, 도구를 이름·설명·입력 스키마와 함께 등록하고, 요청을 처리해 결과를 돌려주는 흐름을 단계별로 살펴봅니다.

이 단원을 마치면 stdio와 HTTP가 각각 어떤 상황의 전송 방식인지 구분하고, 둘의 메시지가 모두 JSON-RPC로 같다는 점을 짚으며, 로컬인지 원격인지를 기준으로 전송 방식을 고르고, 연결이 초기화부터 종료까지 거치는 수명주기를 설명할 수 있습니다.
