# Phase 5 그라운딩 브리프 — 장기 실행 에이전트와 하네스 엔지니어링

## 핵심 프레이밍 (책의 4번 트렌드, 그리고 책의 종착점)
- 초기 에이전트가 몇 분짜리 단발 작업을 했다면, 이제는 며칠씩 작동하며 전체 애플리케이션과 시스템을 구축하고, 인간은 핵심 결정 지점에서만 전략적 감독을 제공하는 방향으로 진화한다.
- 이를 안정적으로 굴리기 위한 '에이전트 하네스' — self-correction(자가 교정), compaction(압축), rollback(롤백), human approval boundary(인간 승인 경계)를 갖춘 오케스트레이션 인프라 — 자체가 별도 연구 분야로 부상했다.
- Phase 5는 앞의 세 축(컨텍스트·메모리·프로토콜) 위에 하네스를 얹어 '며칠씩 일하는 에이전트'로 종합하는 장이다.

## 하네스 엔지니어링 정의
- 하네스 = 모델 바깥의 오케스트레이션 인프라. 모델 내부 개입(가중치·구조 변경)과 구분된다. 같은 모델이라도 하네스가 장기 실행의 성패를 가른다.
- 네 기둥: ① 자가 교정 ② 압축 ③ 롤백 ④ 인간 승인 경계. 여기에 신뢰성·관측 가능성이 가로지른다.

## 장기 실행의 도전 과제
- 누적 오류(작은 실수가 쌓여 큰 실패), 컨텍스트 폭증, 비용 누적, 목표 표류, 사람이 없는 시간대의 무인 진행 위험. → 각 과제가 네 기둥의 존재 이유.

## 자가 교정과 안정성
- 자가 교정 루프: 오류 감지 → 자기 검증 → 재시도. 검증자(critic)를 생성자와 분리하면 더 객관적. 한계: 모델이 자기 오류를 못 보는 경우가 있어 외부 검증 신호가 필요(자가 교정만 과신 금지).
- 피드백·검증 신호: 행동 결과를 모델이 쓸 수 있는 형태로 환류. 테스트·린터·타입 검사·실행 결과·정제된 오류 메시지가 좋은 신호. '검증 가능한 목표'로 작업을 정의하면 자가 교정이 작동.
- 긴 실행에서의 압축 운영: 하네스가 토큰 사용량 임계값을 보고 자동으로 compaction을 건다(점진적 → 공격적). Phase 2.3.2의 전략을 운영 차원에서 실행. 핵심 상태(목표·결정·미해결)는 보존하는 상태 보존 압축. 압축 실패·과압축 대비책 필요.
- 신뢰성(실패를 예상한 설계): 실패 모드 분류, 재시도·지수 백오프, 타임아웃·서킷 브레이커, 비결정성 흡수(같은 입력에 다른 출력), 우아한 저하(graceful degradation).
- 관측 가능성: 트레이스·스팬으로 에이전트 행동을 추적, 행동 로그, 토큰·비용 계측, 재현 가능성. 관측 데이터로 하네스를 개선. (Phase 4.4.3 거버넌스/감사와 연결.)

## 롤백과 인간 승인 경계
- 가역성·체크포인트: 행동을 되돌릴 수 있게 설계. 체크포인트/스냅샷으로 상태 저장, 위험 행동은 격리, 되돌리기 경로 확보. (가역성은 안전한 자율의 전제.)
- 롤백 메커니즘: 잘못된 진행을 안전한 체크포인트로 복귀. 부분 롤백, 부수효과 취소, 보상 트랜잭션(이미 일어난 외부 효과를 상쇄). 롤백에도 비용이 든다.
- 인간 승인 경계(human-in-the-loop): 위험·비가역 행동(삭제, 결제, 배포, 외부 발송) 앞에서 사람 승인을 요구. 위험 등급 분류 → 저위험은 자동, 고위험은 차단·승인 요청. '인간은 핵심 결정 지점에서만 전략적 감독'을 구현하는 장치.
- 자율성 수준·권한 위임: 위험도에 따라 자율성 단계 조절. 모드별 경계(읽기 전용 / 쓰기 허용 / 실행 허용), 신뢰가 쌓이면 점진적으로 권한 확대(점진적 자율).

## 종합 (5.4)
- 멀티에이전트로 규모 키우기: 작업 분할, 오케스트레이터-워커, 컨텍스트 격리(2.3.4) 재사용, 서브에이전트 메모리(3.2.5) 결합, 합류·동기화. (Phase 4.4.1 패턴 활용.)
- 네 축을 하나로: 컨텍스트(안 넘치게) + 메모리(연속성) + 프로토콜(도구·협업) + 하네스(안정·안전)가 한 루프 안에서 맞물린다. 가장 약한 축이 전체 병목. 설계 순서·통합 점검표 제시.
- 사례 분석(코딩 에이전트의 하네스): 에이전틱 코딩 도구(예: Claude Code류)를 사례로 네 축이 실제로 어떻게 구현되는지 해부 — 명시 규칙+자동 메모리, 컨텍스트 압축, MCP 도구 연결, 승인·롤백 경계. 특정 제품 단정·홍보 톤은 피하고 '대표적 구현 방식'으로 일반화.
- 한 걸음씩 만들기(로드맵): 최소 ReAct 루프 → 컨텍스트 관리 추가 → 메모리 추가 → 도구·프로토콜 추가 → 하네스(승인·롤백·관측) 추가. 독자가 따라갈 점진적 성장 경로.
- 마무리(5.4.5): 표준의 성숙, 자율성 확대와 거버넌스의 균형, 평가·신뢰의 과제, 인간 역할의 변화, 책 전체 회고.

## 집필 주의
- Phase 5는 앞 Phase의 개념을 적극 재참조(번호로). 새 개념을 처음부터 다시 길게 정의하기보다 '2.3.2에서 본 압축을 운영 차원에서…' 식으로 잇는다.
- 5.4.5는 책의 마지막 토픽 — '다음 단원 안내'를 넣지 않고 책 전체를 회고하며 닫는다.
