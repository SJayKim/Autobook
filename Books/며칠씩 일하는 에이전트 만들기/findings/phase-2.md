# Phase 2 그라운딩 브리프 — 컨텍스트 엔지니어링

## 핵심 프레이밍
- 컨텍스트 엔지니어링: 컨텍스트 윈도우 '전체'를 유한 자원으로 보고, 매 턴 무엇을 담고·구조화하고·압축하고·내보낼지 설계하는 규율. 프롬프트 엔지니어링(하나의 입력 문자열을 다듬는 일)을 포함하는 상위 규율이다.
- 한 줄 요약(책의 1번 트렌드): 이제는 프롬프트 한 줄이 아니라 컨텍스트 윈도우 전체를 유한 자원으로 관리하는 게 핵심이다. ReAct 루프를 중심에 두되, 토큰 사용량이 커질수록 점진적으로 더 공격적인 compaction을 적용해 대화를 유한 자원으로 관리한다.

## 토큰과 컨텍스트 윈도우
- 토큰: 모델이 텍스트를 처리하는 최소 단위(대략 단어 조각). 한글은 글자당 토큰 수가 영어보다 많은 편.
- 컨텍스트 윈도우: 모델이 한 번에 볼 수 있는 토큰의 최대 한도. 입력(프롬프트·기록·도구·검색결과)과 출력이 같은 예산을 나눠 쓴다. 윈도우가 커져도(수십만~백만 토큰) 여전히 유한하며, 채울수록 비용·지연이 늘고 정확도가 떨어질 수 있다.
- 토큰 예산: 윈도우를 시스템 프롬프트/도구 정의/대화 기록/검색 자료/메모리 주입 등 용도별로 배분하고 여유분(headroom)을 남기는 사고방식.

## 컨텍스트 부패 (context rot)
- 정의: 컨텍스트가 길어질수록 모델 성능이 점진적으로 떨어지는 현상. 윈도우 한도를 넘지 않아도 발생.
- 대표 양상: 중간에 놓인 정보의 소실('lost in the middle' — 길게 넣으면 앞·뒤보다 가운데 정보를 덜 활용). 무관한 내용이 많을수록 관련 신호가 희석됨.
- 함의: '더 많이 넣기'가 아니라 '관련 있는 것만, 잘 배치해서' 넣어야 한다. 이것이 압축·선별·구조화의 근거.

## 컨텍스트의 구성·수집
- 컨텍스트를 채우는 요소: 시스템 프롬프트, 도구 정의, 대화 기록, 검색된 문서(RAG), 메모리에서 회상해 주입한 내용. 각 요소가 토큰을 차지한다.
- 적시(just-in-time) 로딩: 모든 것을 미리 채우지(pre-loading) 않고, 필요할 때만 불러온다. 본문 대신 참조(포인터: 파일 경로·ID)만 두었다가 필요 시 펼친다. 파일 시스템을 외부 메모리처럼 쓴다.
- RAG(검색 증강 생성): 임베딩으로 문서를 벡터화해 질의와 유사한 청크 top-k를 찾아 컨텍스트에 주입. 한계: 관련성 낮은 청크가 섞이면 잡음, top-k와 토큰 예산의 균형 필요.
- 도구 선택 부담: 도구 정의도 토큰을 먹는다. 도구가 너무 많으면 컨텍스트를 차지하고 모델의 선택 정확도가 떨어진다. 동적 노출·묶기·지연 로딩으로 적정 수 유지.
- 구조화: 섹션·태그·우선순위로 컨텍스트를 정돈하면 모델이 더 잘 활용. 형식 일관성, 노이즈 최소화.

## 압축과 최적화
- compaction(압축): 길어진 대화 기록을 요약해 토큰을 회수. 무엇을 버리고(중복·해결된 곁가지) 무엇을 남길지(목표·결정·미해결 과제) 판단. 정보 손실 위험이 있으므로 핵심 상태는 보존.
- 점진적·공격적 압축: 토큰 사용량이 임계값을 넘을수록 점점 더 공격적으로 압축. 가벼운 요약 → 강한 요약 → 핵심만 남기기 식으로 단계 전환. '대화를 유한 자원으로 관리'하는 구체적 수단. (Phase 5의 하네스 주도 압축과 직접 연결.)
- 오프로딩(offloading): 중간 결과·긴 산출물을 파일·스크래치패드 등 외부로 내보내고 컨텍스트에는 포인터만. 메모리 계층(Phase 3)으로 이어짐.
- 서브에이전트 컨텍스트 격리: 하위 작업을 별도 에이전트에 맡겨 각자의 컨텍스트에서 처리하고 결과(요약)만 상위로 환류. 상위 컨텍스트 오염을 막고 병렬 탐색 가능.

## GEPA와 프롬프트 자동 최적화
- 동기: 프롬프트/지시문을 사람이 일일이 다듬는 대신 자동으로 개선.
- GEPA: 'Genetic-Pareto'. 실행 기록(trace)을 자연어로 '반성'해 프롬프트를 진화시키는 반성적 최적화 기법. 자연어 피드백을 활용해 적은 시도로도 개선하는 점이 특징(강화학습식 다수 롤아웃 대비 표본 효율적). 탐색(변형 생성)과 평가를 반복.
- 위치: GEPA는 컨텍스트 엔지니어링 안에서 '지시문/프롬프트 부분을 자동으로 다듬는' 자동화 계층. 컨텍스트 전체 관리(수집·압축·격리)와 상호 보완.
- 주의: GEPA의 내부 알고리즘 세부보다 '왜·어떻게 프롬프트를 자동 개선하는가'와 컨텍스트 엔지니어링과의 관계에 초점.

## 집필 주의
- compaction은 Phase 2.3에서 원리를, Phase 5.2.3에서 하네스 운영 관점을 다룬다. Phase 2에서는 원리·전략까지만.
