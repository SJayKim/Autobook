# Phase 4 그라운딩 브리프 — 프로토콜 표준화 (MCP와 A2A)

## 핵심 프레이밍 (책의 3번 트렌드)
- 2026년의 변화는 새로운 아키텍처(멀티에이전트 오케스트레이션) + 새로운 표준(MCP/A2A 프로토콜) + 새로운 경제학(에이전트용 FinOps) + 새로운 거버넌스 역량의 결합으로 정의된다.
- 두 표준의 역할: MCP = 모델/에이전트를 '도구·데이터'에 연결(수직). A2A = 에이전트를 '다른 에이전트'에 연결(수평). 상호 보완.

## M×N 통합 문제
- 표준이 없으면 M개의 모델/에이전트를 N개의 도구에 각각 맞춤 연동 → M×N개의 커넥터를 중복 구현. 표준이 생기면 각자 표준 한 번씩만 구현 → M+N. 재사용성·생태계가 핵심 가치.

## MCP (Model Context Protocol)
- 정체: 모델을 외부 도구·데이터에 연결하는 개방형 표준. 클라이언트-서버 구조를 JSON-RPC 메시지로 표준화.
- 구조: 호스트(에이전트 앱) ↔ 클라이언트(호스트 안에서 서버와 1:1 연결) ↔ 서버(도구·데이터 제공). 호스트는 여러 클라이언트를, 각 클라이언트는 하나의 서버를 맡는다.
- 세 가지 기본형(primitives):
  - tools(도구): 모델이 호출해 '실행'하는 기능(부수효과 가능). 예: 파일 쓰기, API 호출.
  - resources(리소스): 모델/앱이 '읽는' 데이터(부수효과 없음). 예: 파일 내용, DB 레코드.
  - prompts(프롬프트): 재사용 가능한 프롬프트 템플릿(사용자가 고르는 정형 작업).
- 전송(transport): 로컬은 stdio(표준 입출력), 원격은 스트리밍 가능한 HTTP. 메시지는 JSON-RPC. 연결은 초기화 → 능력 교환 → 사용 → 종료의 수명주기.
- 서버 만들기: 서버 초기화 → 도구 등록(이름·설명·입력 스키마) → 요청 처리 → 결과 반환. 클라이언트가 연결해 도구 목록을 받아 모델에 노출.
- 기존 도구 감싸기: 이미 있는 API/함수를 MCP 서버로 래핑. 함수 시그니처를 도구 입력 스키마로 변환, 오류를 표준 형식으로, 인증 정보 주입. 함정: 과도한 도구 노출, 모호한 설명, 민감정보 유출.
- 보안: 신뢰되지 않은 서버 연결 위험, 도구 '설명'을 통한 프롬프트 주입, 과도한 권한, 사용자 동의 없는 민감 데이터 접근. 권한 범위 최소화·동의·검증 필요.

## A2A (Agent2Agent Protocol)
- 정체: 서로 다른(이종) 에이전트가 협업하도록 통신을 표준화하는 개방형 프로토콜. 상대 에이전트의 내부를 몰라도 표준 메시지로 작업을 주고받는다(불투명한 상대와의 상호운용).
- 왜 MCP만으론 부족한가: MCP는 '에이전트→도구'. 에이전트끼리 작업을 위임·협상·스트리밍하려면 에이전트 간 표준(A2A)이 필요.
- 에이전트 카드(Agent Card): 에이전트의 능력 명세(엔드포인트·인증·제공 스킬 등)를 공표하는 문서. 다른 에이전트가 이를 발견(discovery)해 누구에게 무엇을 맡길지 동적으로 라우팅·선택.
- 작업·메시지 모델: 작업(task)에 생명주기가 있고, 메시지를 주고받으며, 결과물을 아티팩트(artifact)로 반환. 장기 작업·스트리밍·상태 추적 지원.
- MCP·A2A 함께 쓰기: 한 에이전트가 동시에 'A2A로 협업하는 동료'이자 'MCP로 도구를 쓰는 주체'. 수직(MCP)+수평(A2A)의 계층적 구성. 경계: 도구 연결이면 MCP, 에이전트 위임이면 A2A.

## 멀티에이전트·FinOps·거버넌스
- 오케스트레이션 패턴: 감독자(supervisor, 한 에이전트가 하위 에이전트들에 작업 분배·취합), 파이프라인(단계별 전달), 병렬 팬아웃(동시 실행 후 합류). 작업 성격으로 패턴 선택.
- 에이전트용 FinOps: 토큰·호출이 곧 비용. 비용 가시성(누가·무엇에 얼마), 예산·쿼터, 모델 선택의 경제학(작은 모델로 충분한 작업 구분), 비용-품질 트레이드오프 관리. 새로운 운영 규율로 부상.
- 거버넌스: 정책 집행, 권한 경계, 감사 로그, 책임 추적, 규정 준수. 멀티에이전트가 늘수록 '누가 무엇을 왜 했는가'를 추적·통제하는 역량이 핵심.
- 프로토콜 보안·신뢰 경계: 공급망 위험(악성 서버/에이전트), 에이전트 사칭, 권한 상승. 표준이 제공하는 인증·범위 제한·동의 메커니즘으로 신뢰 경계를 설계.

## 집필 주의
- MCP·A2A의 정확한 와이어 포맷·필드명을 단정적으로 나열하지 말 것. 구조·역할·흐름 중심으로. (스펙은 계속 진화하므로 '대략 이런 정보를 담는다' 수준.)
- 4.4.2 FinOps는 비용을 다루므로 Phase 2의 토큰/유한자원(2.1.2)을 자연스럽게 다시 짚어 연결.
- MCP 서버 만들기(4.2.4)는 lab 토픽 — 의사코드/단계 흐름 + ASCII 다이어그램으로. 특정 언어 SDK에 종속되지 않게.
