# 9.1.2 LangSmith

LLM 애플리케이션을 프로덕션에 배포하면, 수천 건의 요청 중에서 "어떤 프롬프트에 어떤 응답이 나왔는지", "어느 단계에서 시간이 가장 오래 걸렸는지", "비용은 얼마나 발생했는지"를 추적해야 합니다. 9.1.1에서 옵저버빌리티의 세 기둥(메트릭, 로그, 트레이스)과 LLM 전용 옵저버빌리티의 필요성을 살펴보았습니다. 이 단원에서는 LLM 옵저버빌리티 도구 중 하나인 **LangSmith**의 아키텍처와 핵심 기능을 다룹니다.

LangSmith는 LangChain을 만든 LangChain Inc.가 개발한 LLM 옵저버빌리티 플랫폼입니다. LangSmith가 등장한 배경을 이해하려면, LangChain 프레임워크의 역할부터 짚어야 합니다. LangChain은 LLM 호출, 프롬프트 관리, 체인 구성, 에이전트 실행 등을 추상화하는 오픈소스 프레임워크입니다. 개발자는 LangChain으로 LLM 애플리케이션을 빠르게 만들 수 있지만, 일단 배포한 뒤에는 내부에서 무슨 일이 일어나는지 파악하기 어렵습니다. 프롬프트 체인이 복잡해지면 디버깅이 더 힘들어집니다. LangSmith는 이 문제를 해결하기 위해 만들어졌습니다. LangChain으로 구축한 애플리케이션의 내부 동작을 투명하게 추적하고, 프롬프트를 실험하며, 응답 품질을 체계적으로 평가하는 것이 LangSmith의 목적입니다.

LangSmith의 아키텍처는 크게 네 가지 계층으로 구성됩니다. 첫 번째 계층은 **계측 계층**(instrumentation layer)입니다. 애플리케이션 코드에 삽입되어 각 단계의 입력, 출력, 소요 시간, 메타데이터를 캡처합니다. 두 번째 계층은 **수집 계층**(ingestion layer)입니다. 계측 포인트에서 생성된 데이터를 LangSmith 서버로 전송합니다. SDK는 데이터를 배치로 모아서 비동기로 전송하므로, 애플리케이션의 응답 지연에 미치는 영향을 최소화합니다. 세 번째 계층은 **저장 및 분석 계층**(storage and analysis layer)입니다. 수집된 트레이스, 메트릭, 프롬프트 전문을 저장하고 검색 가능한 형태로 인덱싱합니다. 네 번째 계층은 **활용 계층**(action layer)입니다. 대시보드, 프롬프트 Playground, 데이터셋 관리, 평가 실행, 비용 추적 등의 기능을 제공합니다.

```
LangSmith 아키텍처 계층

+-------------------+     +-------------------+
| 애플리케이션 코드 |     | LangSmith 서버    |
|                   |     |                   |
| [계측 계층]       |---->| [수집 계층]       |
| @traceable,       |     |  배치 수신,       |
| wrap_openai(),    |     |  파싱             |
| 자동 계측         |     +--------+----------+
+-------------------+              |
                                   v
                          +-------------------+
                          | [저장/분석 계층]  |
                          |  트레이스 DB,     |
                          |  메트릭 인덱스    |
                          +--------+----------+
                                   |
                                   v
                          +-------------------+
                          | [활용 계층]       |
                          |  대시보드, 평가,  |
                          |  Playground,      |
                          |  비용 추적        |
                          +-------------------+
```

계측 계층에서 LangSmith가 데이터를 수집하는 방식은 세 가지입니다. 첫 번째는 **자동 계측**입니다. LangChain이나 LangGraph를 사용하는 경우, 환경 변수 두 개만 설정하면 모든 체인 실행이 자동으로 트레이싱됩니다. 'LANGCHAIN_TRACING_V2'를 'true'로, 'LANGCHAIN_API_KEY'를 LangSmith API 키로 설정하는 것입니다. 코드를 한 줄도 수정하지 않아도 됩니다. 두 번째는 **데코레이터 기반 계측**입니다. LangChain을 사용하지 않는 코드에서도 Python 함수에 '@traceable' 데코레이터를 붙이면, 그 함수의 입력과 출력이 LangSmith에 트레이스로 기록됩니다. 세 번째는 **래퍼 기반 계측**입니다. OpenAI SDK의 클라이언트 객체를 'wrap_openai()' 함수로 감싸면, 해당 클라이언트를 통한 모든 LLM 호출이 자동으로 추적됩니다. Anthropic SDK에 대해서도 유사한 래퍼가 제공됩니다.

```python
# 자동 계측: 환경 변수 설정만으로 LangChain 트레이싱 활성화
import os
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"] = "ls-..."

# 데코레이터 기반 계측: 일반 Python 함수에 적용
from langsmith import traceable

@traceable

# 래퍼 기반 계측: OpenAI 클라이언트 래핑

from openai import OpenAI
```

위 코드에서 'os.environ' 두 줄은 LangChain 프레임워크에 "모든 실행을 LangSmith에 보고하라"고 지시하는 설정입니다. '@traceable' 데코레이터는 함수의 호출 시작과 종료를 감지하여, 입력 인자와 반환값을 자동으로 LangSmith에 전송합니다. 'wrap_openai()'는 OpenAI 클라이언트의 호출 메서드를 감싸서, 매 호출의 프롬프트, 응답, 토큰 수, 지연 시간을 기록합니다.

수집된 데이터는 **트레이스**(trace)라는 단위로 저장됩니다. 9.1.1에서 설명한 것처럼, 하나의 트레이스는 하나의 사용자 요청이 시스템을 통과하는 전체 경로를 나타냅니다. LangSmith에서 트레이스는 여러 개의 **런**(run)으로 구성됩니다. 런은 단일 작업 단위이며, 9.1.1에서 설명한 OpenTelemetry의 스팬(span)에 대응하는 개념입니다. 각 런에는 유형이 지정되어 있습니다. 'llm'은 모델 호출, 'chain'은 여러 단계의 연결, 'tool'은 외부 도구 호출, 'retriever'는 검색 단계를 나타냅니다.

LangSmith 웹 UI에서 트레이스 목록을 열면, 각 트레이스의 전체 소요 시간, 토큰 사용량, 상태(성공/오류)가 한눈에 보입니다. 개별 트레이스를 클릭하면 런들이 트리 구조로 펼쳐집니다. 각 런을 클릭하면 해당 단계의 입력 전문, 출력 전문, 시작 시간, 종료 시간, 소비 토큰 수가 표시됩니다. 이 시각화 덕분에 "전체 요청 중 어느 단계가 가장 오래 걸렸는가", "검색 단계에서 어떤 문서가 반환되었는가", "모델이 어떤 프롬프트를 받고 어떤 응답을 생성했는가"를 빠르게 파악할 수 있습니다.

트레이스 시각화에서 필터링 기능도 중요합니다. 수천 건의 트레이스 중에서 특정 조건에 맞는 것만 추려볼 수 있습니다. 예를 들어, 오류가 발생한 트레이스만 보거나, 지연 시간이 5초를 초과한 트레이스만 보거나, 특정 태그가 붙은 트레이스만 볼 수 있습니다. 태그는 계측 시점에 추가할 수 있으며, 환경(production, staging), 기능(summarize, qa), 사용자 세그먼트(premium, free) 등을 구분하는 데 활용합니다.

LangSmith의 또 다른 핵심 기능은 **프롬프트 Playground**입니다. Playground는 프롬프트를 작성하고, 다양한 입력으로 테스트하며, 결과를 즉시 확인할 수 있는 실험 환경입니다. 프롬프트를 개선하려면 시행착오가 필요합니다. 프롬프트의 문구를 바꿔 보고, 시스템 프롬프트를 조정해 보고, 예시를 추가하거나 제거해 보면서 응답 품질이 어떻게 변하는지 확인해야 합니다. Playground는 이 과정을 코드를 수정하지 않고도 웹 브라우저에서 바로 수행할 수 있게 해 줍니다.

Playground에서는 프롬프트 템플릿을 작성하고 변수를 정의합니다. 예를 들어 "다음 문서를 {language}로 요약하세요: {document}"라는 템플릿에서 'language'와 'document'가 변수입니다. 여러 가지 변수 조합을 입력하면, 각 조합에 대한 모델 응답을 나란히 비교할 수 있습니다. 모델 종류, temperature, max_tokens 같은 생성 파라미터도 Playground에서 바로 조정할 수 있으므로, "같은 프롬프트인데 모델이 다르면 결과가 어떻게 달라지는지"를 실험하기에도 유용합니다.

프롬프트 실험에서 한 걸음 더 나아가, 체계적으로 품질을 측정하려면 **데이터셋과 평가 실행** 기능이 필요합니다. 데이터셋이란 입력과 기대 출력(또는 참조 답변)의 쌍을 모아 놓은 것입니다. 예를 들어, 고객 문의 20건에 대한 적절한 답변을 미리 작성해 두면, 프롬프트를 변경할 때마다 이 20건에 대해 모델 응답을 생성하고 참조 답변과 비교하여 품질을 측정할 수 있습니다.

LangSmith에서 데이터셋을 만드는 방법은 여러 가지입니다. 웹 UI에서 직접 입력할 수도 있고, CSV 파일을 업로드할 수도 있으며, 프로덕션 트레이스에서 특정 입출력 쌍을 선택하여 데이터셋에 추가할 수도 있습니다. 마지막 방법이 특히 유용합니다. 프로덕션에서 문제가 된 실제 요청을 데이터셋에 넣고, 이후 프롬프트 변경 시 해당 요청에 대해 문제가 해결되었는지 자동으로 검증할 수 있기 때문입니다.

평가 실행은 데이터셋의 각 입력에 대해 모델 응답을 생성하고, 평가자(evaluator)를 적용하여 점수를 매기는 과정입니다. 평가자에는 여러 종류가 있습니다. 문자열 비교 평가자는 모델 응답이 참조 답변과 문자열 수준에서 얼마나 일치하는지를 측정합니다. LLM 기반 평가자는 별도의 LLM을 사용하여 응답의 정확성, 관련성, 유해성 등을 판정합니다. 사용자 정의 평가자는 Python 함수로 작성하여 특정 기준을 자유롭게 정의할 수 있습니다. 평가 실행 결과는 LangSmith 웹 UI에서 데이터셋 단위로 집계되어, 평가 점수의 평균, 분포, 개별 사례별 결과를 확인할 수 있습니다.

```
데이터셋과 평가 실행 흐름

[데이터셋]                    [평가 실행]
+-----------------------+     +---------------------------+
| 입력1 | 참조 답변1    |     | 입력1 --> 모델 응답1      |
| 입력2 | 참조 답변2    |---->| 입력2 --> 모델 응답2      |
| 입력3 | 참조 답변3    |     | 입력3 --> 모델 응답3      |
| ...   | ...           |     | ...                       |
+-----------------------+     +---------------------------+
                                        |
                                        v
                              +---------------------------+
                              | 평가자 적용               |
                              | - 문자열 비교             |
                              | - LLM 기반 판정          |
                              | - 사용자 정의 함수       |
                              +---------------------------+
                                        |
                                        v
                              +---------------------------+
                              | 결과 집계                 |
                              | 평균 점수, 분포,         |
                              | 개별 사례 확인           |
                              +---------------------------+
```

LLM 서비스를 운영할 때 비용 통제는 핵심 과제입니다. LangSmith는 **비용 추적** 기능을 제공하여, 모델별, 프로젝트별, 기간별 토큰 소비량과 비용을 집계합니다. 각 트레이스에 입력 토큰 수와 출력 토큰 수가 기록되고, 모델별 단가를 적용하여 비용이 자동으로 계산됩니다. 대시보드에서 일별, 주별 비용 추이를 확인할 수 있으며, 특정 프로젝트나 태그 기준으로 비용을 분리하여 볼 수 있습니다. 예를 들어 "요약 기능이 전체 비용의 60%를 차지하고 있다"는 사실을 파악하면, 해당 기능의 프롬프트를 최적화하거나 더 저렴한 모델로 교체하는 의사결정을 내릴 수 있습니다.

마지막으로 **LangChain과의 통합**을 살펴보겠습니다. LangSmith는 LangChain 생태계의 일부이므로, LangChain 및 LangGraph와 가장 깊은 수준으로 통합됩니다. LangChain의 체인, 에이전트, 도구 호출이 별도 코드 없이 자동으로 트레이싱되는 것이 대표적입니다. LangGraph로 구성한 복잡한 에이전트 워크플로의 각 노드와 엣지가 트레이스에 그대로 반영됩니다. 그러나 LangSmith는 LangChain 전용 도구가 아닙니다. OpenAI SDK, Anthropic SDK, LlamaIndex 등 다른 프레임워크와도 연동됩니다. '@traceable' 데코레이터나 'wrap_openai()' 같은 범용 계측 방법을 사용하면, 어떤 프레임워크로 구축한 애플리케이션이든 LangSmith로 추적할 수 있습니다. 또한 9.1.1에서 다룬 OpenTelemetry 프로토콜로 트레이스를 내보내는 것도 지원하므로, 기존 옵저버빌리티 인프라와 통합하는 것도 가능합니다.

정리하면, LangSmith는 LLM 애플리케이션의 트레이싱, 프롬프트 실험, 체계적 평가, 비용 추적을 하나의 플랫폼에서 제공하는 옵저버빌리티 도구입니다. 자동 계측, 데코레이터, 래퍼의 세 가지 방식으로 애플리케이션을 계측하고, 수집된 트레이스를 시각화하여 각 단계의 입출력과 소요 시간을 파악합니다. 프롬프트 Playground로 프롬프트를 실험하고, 데이터셋과 평가자를 조합하여 품질을 체계적으로 측정합니다. 비용 추적 기능으로 모델별, 기능별 비용을 분석하여 최적화 방향을 잡을 수 있습니다. LangChain과 가장 깊이 통합되지만, OpenAI SDK나 Anthropic SDK 등 다른 프레임워크와도 연동됩니다.

다음 단원인 9.1.3에서는 오픈소스 옵저버빌리티 도구인 Langfuse를 다룹니다. 셀프 호스팅이 가능한 Langfuse의 트레이스 구조, 세션 관리, 프롬프트 관리, 비용 분석 기능을 살펴봅니다.

이 단원을 마치면 LangSmith를 활용하여 LLM 애플리케이션의 트레이싱과 평가를 수행할 수 있습니다.
