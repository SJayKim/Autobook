# 6.1.3 TGI (Text Generation Inference)

6.1.1에서 모델 서빙 아키텍처의 기본 구성과 서빙 레이턴시의 요소(TTFT, TPS 등)를 살펴보았고, 6.1.2에서는 vLLM이 PagedAttention을 활용하여 GPU 메모리를 효율적으로 관리하는 방식을 다루었습니다. 이 단원에서는 Hugging Face가 개발한 모델 서빙 프레임워크인 **TGI(Text Generation Inference)**를 다룹니다. 같은 오픈소스 LLM을 서빙하더라도 프레임워크마다 설계 철학과 강점이 다릅니다. TGI는 Hugging Face 모델 저장소와 긴밀하게 연결되어 있어, Hugging Face Hub에 공개된 모델을 별도 변환 없이 곧바로 서빙할 수 있다는 점이 가장 큰 특징입니다. 운영 환경에서 모델을 빠르게 배포해야 하는 상황이라면, 이 특성이 왜 중요한지 이 단원을 통해 이해할 수 있습니다.

TGI가 어떤 소프트웨어인지부터 정리하겠습니다. **TGI(Text Generation Inference)**는 Hugging Face가 Rust와 Python으로 작성한 대규모 언어 모델 전용 추론 서버입니다. 여기서 "추론 서버"란, 모델을 GPU 메모리에 올려 두고 HTTP 요청을 받아 텍스트 생성 결과를 반환하는 프로그램을 뜻합니다. TGI는 단순히 모델을 불러와 추론하는 것에 그치지 않고, 높은 처리량과 낮은 지연 시간을 동시에 달성하기 위한 여러 최적화 기법을 내장하고 있습니다.

TGI의 내부 구조를 이해하려면, 요청이 들어온 시점부터 응답이 나가는 시점까지의 흐름을 따라가는 것이 가장 좋습니다. TGI는 크게 세 가지 계층으로 나뉩니다. 첫째는 HTTP 요청을 수신하고 토큰 스트리밍을 관리하는 **라우터(Router)** 계층입니다. 둘째는 여러 요청을 하나의 배치로 묶어 GPU에 전달하는 **스케줄러** 계층입니다. 셋째는 실제 모델 추론을 수행하는 **모델 실행(Model Executor)** 계층입니다. 라우터는 Rust로 작성되어 있어 네트워크 I/O를 비동기로 처리하며, 모델 실행 계층은 Python과 PyTorch 기반으로 GPU 연산을 담당합니다.

```
요청 흐름:

  클라이언트 --> [라우터 (Rust)]
                    |
                    |  요청 수신, 입력 검증, 대기열 관리
                    v
               [스케줄러]
                    |
                    |  대기열의 요청을 배치로 묶음
                    |  Continuous Batching 적용
                    v
               [모델 실행 (Python/PyTorch)]
                    |
                    |  GPU에서 Prefill / Decode 수행
                    |  Flash Attention으로 연산 최적화
                    v
               [라우터 (Rust)]
                    |
                    |  생성된 토큰을 클라이언트에 스트리밍
                    v
                클라이언트
```

라우터가 요청을 받으면 입력 텍스트의 유효성을 검사하고, 최대 토큰 수 등 파라미터를 확인한 뒤 스케줄러 대기열에 넣습니다. 스케줄러는 대기열에 있는 여러 요청을 하나의 배치로 묶습니다. 이때 TGI는 **Continuous Batching** 방식을 사용합니다. Continuous Batching이란, 하나의 요청이 생성을 마치면 그 자리에 대기 중이던 새 요청을 즉시 넣어 GPU가 쉬는 시간을 줄이는 기법입니다. 6.1.1에서 다룬 것처럼 Decode 단계에서는 토큰 하나씩 순차 생성하므로 GPU 활용률이 떨어지기 쉬운데, Continuous Batching이 이 비효율을 완화합니다.

TGI가 내장하는 핵심 최적화 기법 중 하나가 **Flash Attention**입니다. Flash Attention이 왜 필요한지를 이해하려면, 1.2.3에서 다룬 Attention 연산 과정을 떠올려야 합니다. Attention은 Query, Key, Value 행렬 사이의 곱셈과 소프트맥스 연산으로 구성됩니다. 입력 시퀀스 길이가 N이면 Attention 행렬의 크기는 N x N이 되므로, 시퀀스가 길어질수록 메모리 사용량과 연산량이 급격히 증가합니다. 예를 들어, 시퀀스 길이가 2,048에서 4,096으로 두 배가 되면 Attention 행렬의 크기는 네 배가 됩니다.

표준 Attention 구현에서는 이 N x N 크기의 전체 행렬을 GPU의 고대역폭 메모리(HBM)에 한꺼번에 올려 놓고 연산합니다. 문제는 GPU 내부에서 실제 계산이 이루어지는 SRAM(연산 유닛 근처의 빠른 캐시 메모리)과 HBM 사이의 데이터 전송이 병목이 된다는 점입니다. Flash Attention은 이 문제를 해결하기 위해 Attention 행렬을 작은 블록(타일) 단위로 쪼개어 SRAM에서 직접 계산합니다. 전체 행렬을 HBM에 쓰지 않고 블록별로 소프트맥스를 점진적으로 계산하므로, 메모리 사용량이 N x N에서 N에 비례하는 수준으로 줄어듭니다.

```
표준 Attention:
  Q, K, V (HBM) --> 전체 N x N 행렬 계산 (HBM) --> 소프트맥스 (HBM) --> 출력
  [HBM <-> SRAM 데이터 이동이 빈번, 메모리 O(N^2)]

Flash Attention:
  Q, K, V (HBM) --> 블록 단위로 SRAM에 로드
                --> SRAM 내에서 부분 소프트맥스 계산
                --> 결과만 HBM에 기록
  [HBM 접근 최소화, 메모리 O(N)]
```

TGI는 Flash Attention v2를 기본으로 지원하며, 호환되는 GPU(NVIDIA A100, H100 등)에서 자동으로 활성화됩니다. 사용자가 별도 설정을 하지 않아도 모델과 GPU가 조건을 충족하면 Flash Attention이 적용되므로, 긴 시퀀스를 처리할 때 메모리 부족 오류가 줄어들고 처리 속도가 향상됩니다.

TGI를 실제 환경에 배포하는 가장 일반적인 방법은 **Docker 컨테이너**를 사용하는 것입니다. Docker란, 애플리케이션과 그 실행에 필요한 모든 의존성(라이브러리, 설정 파일 등)을 하나의 패키지로 묶어 어떤 서버에서든 동일하게 실행할 수 있게 해 주는 도구입니다. TGI는 공식 Docker 이미지를 제공하므로, GPU 드라이버와 Docker만 설치되어 있으면 한 줄의 명령으로 서빙 서버를 시작할 수 있습니다.

아래는 TGI Docker 컨테이너를 실행하는 기본 명령입니다.

```bash
docker run --gpus all --shm-size 1g -p 8080:80 \
  -v /data:/data \
  ghcr.io/huggingface/text-generation-inference:latest \
  --model-id meta-llama/Llama-3.1-8B-Instruct \
  --max-input-tokens 4096 \
  --max-total-tokens 8192
```

이 명령의 각 부분을 설명하겠습니다. `--gpus all`은 호스트 서버의 모든 GPU를 컨테이너 안에서 사용할 수 있게 합니다. `--shm-size 1g`는 컨테이너 내부의 공유 메모리를 1GB로 설정하는데, PyTorch가 멀티프로세스 간 텐서를 공유할 때 이 영역을 사용하므로 충분한 크기를 확보해야 합니다. `-p 8080:80`은 호스트의 8080 포트를 컨테이너의 80 포트에 연결하여 외부에서 접근할 수 있게 합니다. `-v /data:/data`는 호스트의 /data 디렉토리를 컨테이너 안에 마운트하여, 다운로드된 모델 가중치가 컨테이너가 종료되더라도 보존되게 합니다.

`--model-id` 뒤에 오는 값은 Hugging Face Hub의 모델 이름입니다. TGI는 이 이름을 받으면 Hugging Face Hub에서 모델 가중치를 자동으로 다운로드하고 GPU에 적재합니다. 별도의 모델 변환 과정이 필요 없습니다. `--max-input-tokens 4096`은 한 요청에 허용되는 최대 입력 토큰 수를, `--max-total-tokens 8192`은 입력과 출력을 합친 최대 토큰 수를 지정합니다. 이 두 값은 GPU 메모리와 직접 관련되는데, 값을 크게 잡을수록 긴 대화를 처리할 수 있지만 그만큼 GPU 메모리를 많이 차지합니다.

TGI는 이 외에도 서빙 동작을 세밀하게 조절하는 여러 **설정 파라미터**를 제공합니다. 주요 파라미터를 정리하면 다음과 같습니다.

| 파라미터 | 설명 | 기본값 예시 |
|---------|------|-----------|
| `--model-id` | Hugging Face Hub의 모델 이름 또는 로컬 경로 | 없음 (필수) |
| `--max-input-tokens` | 한 요청의 최대 입력 토큰 수 | 1024 |
| `--max-total-tokens` | 입력 + 출력의 최대 토큰 수 | 2048 |
| `--max-batch-prefill-tokens` | Prefill 단계에서 한 배치에 처리할 수 있는 최대 토큰 수 | 4096 |
| `--max-concurrent-requests` | 동시에 처리할 수 있는 최대 요청 수 | 128 |
| `--quantize` | 양자화 방식 (bitsandbytes, gptq, awq 등) | None |
| `--num-shard` | 모델을 몇 개의 GPU에 분산할지 (텐서 병렬) | GPU 수만큼 자동 |
| `--dtype` | 모델 가중치의 데이터 타입 (float16, bfloat16) | float16 |

`--max-batch-prefill-tokens`는 Prefill 단계에서 한 번에 처리하는 토큰의 총량을 제한합니다. Prefill은 입력 전체를 한꺼번에 처리하는 단계이므로, 긴 입력이 여러 개 동시에 들어오면 GPU 메모리가 순간적으로 크게 사용됩니다. 이 파라미터로 상한을 정하면 메모리 초과를 방지할 수 있습니다. `--quantize`는 모델 가중치를 더 적은 비트로 표현하여 GPU 메모리 사용량을 줄이는 양자화 기능을 활성화합니다. 예를 들어, `bitsandbytes`를 지정하면 8비트 또는 4비트 양자화를 적용하여 메모리를 절반 이상 줄일 수 있습니다. `--num-shard`는 모델이 단일 GPU에 올라가지 않을 정도로 클 때, 여러 GPU에 나누어 적재하는 텐서 병렬 설정입니다.

TGI가 응답을 반환하는 방식 중 운영 환경에서 특히 중요한 기능이 **토큰 스트리밍**입니다. 일반적인 HTTP 응답은 모든 토큰 생성이 끝난 뒤에 전체 텍스트를 한꺼번에 보냅니다. 긴 응답을 생성할 때 사용자는 모든 생성이 끝날 때까지 아무것도 보지 못한 채 기다려야 합니다. 토큰 스트리밍은 토큰이 하나씩 생성될 때마다 곧바로 클라이언트에 전송하는 방식입니다. 사용자는 첫 번째 토큰이 생성되는 순간부터 텍스트가 화면에 나타나는 것을 볼 수 있어, 체감 대기 시간이 크게 줄어듭니다.

TGI에서 스트리밍을 사용하려면 요청 시 `stream` 파라미터를 `true`로 설정합니다. 아래는 curl 명령으로 스트리밍 요청을 보내는 예입니다.

```bash
curl -N http://localhost:8080/generate_stream \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": "What is deep learning?",
    "parameters": {
      "max_new_tokens": 256,
      "temperature": 0.7
    }
  }'
```

이 명령에서 `/generate_stream` 엔드포인트를 사용하면 TGI는 **Server-Sent Events(SSE)** 방식으로 응답합니다. SSE란 서버가 클라이언트에게 데이터를 일방향으로 지속 전송하는 HTTP 기반 프로토콜입니다. 각 이벤트에는 생성된 토큰과 함께 해당 토큰의 로그 확률, 종료 여부 등의 메타데이터가 포함됩니다. `-N` 플래그는 curl이 응답을 버퍼링하지 않고 즉시 출력하도록 하는 옵션입니다. 스트리밍을 사용하지 않는 일반 요청은 `/generate` 엔드포인트를 사용하며, 이 경우 모든 토큰 생성이 완료된 후 전체 텍스트를 한 번에 반환합니다.

TGI를 다른 서빙 프레임워크와 구분 짓는 핵심 특성은 **Hugging Face 생태계와의 통합**입니다. Hugging Face Hub에는 수십만 개의 모델이 공개되어 있으며, 각 모델은 가중치 파일, 토크나이저 설정, 모델 구성 파일을 표준화된 형식으로 포함합니다. TGI는 이 표준 형식을 직접 읽을 수 있도록 설계되었기 때문에, `--model-id`에 Hub 경로만 지정하면 가중치 다운로드부터 토크나이저 초기화, 모델 적재까지 자동으로 진행됩니다.

이 통합이 실무에서 의미하는 바를 구체적으로 살펴보겠습니다. vLLM 같은 프레임워크도 Hugging Face 모델을 지원하지만, TGI는 Hugging Face가 직접 만든 프레임워크이므로 새로운 모델 아키텍처가 Hub에 등록되었을 때 가장 빠르게 지원되는 경우가 많습니다. 또한 Hugging Face의 Inference Endpoints 서비스, 즉 클라우드에서 모델을 서빙하는 관리형 서비스의 백엔드 엔진으로 TGI가 사용됩니다. 따라서 로컬 환경에서 TGI로 테스트한 설정을 그대로 Inference Endpoints에 적용할 수 있어, 개발에서 운영까지의 전환이 매끄럽습니다. Hub에 업로드된 모델의 `config.json`에 기록된 아키텍처 정보를 TGI가 자동으로 인식하므로, 사용자가 모델 구조를 일일이 지정할 필요도 없습니다.

정리하면, TGI는 Hugging Face가 개발한 LLM 전용 추론 서버로, Rust 기반의 고성능 라우터, Continuous Batching, Flash Attention 등의 최적화를 내장하고 있습니다. Docker 이미지를 통해 한 줄의 명령으로 서빙 환경을 구성할 수 있으며, Hugging Face Hub와의 긴밀한 통합 덕분에 모델 변환 없이 즉시 배포가 가능합니다. 토큰 스트리밍을 통해 사용자 체감 지연 시간을 줄일 수 있고, 다양한 설정 파라미터로 GPU 메모리와 처리량 사이의 균형을 조절할 수 있습니다.

다음 단원인 6.1.4에서는 NVIDIA가 제공하는 TensorRT-LLM을 다룹니다.

이 단원을 마치면 TGI의 아키텍처와 주요 최적화 기법을 설명하고, Docker 기반으로 TGI 서빙 환경을 구성할 수 있습니다.
