# 3.2.4 Qdrant와 Milvus

3.2.1에서 벡터 데이터베이스의 개념과 필요성을 다루었습니다. 고차원 임베딩 벡터를 저장하고 ANN(Approximate Nearest Neighbor) 검색으로 유사한 벡터를 빠르게 찾아내는 것이 벡터 데이터베이스의 핵심 역할이었습니다. 3.2.2에서는 완전 관리형 서비스인 Pinecone을, 3.2.3에서는 모듈 기반 오픈소스인 Weaviate를 살펴보았습니다. 이번 단원에서는 두 가지 오픈소스 벡터 데이터베이스인 Qdrant와 Milvus를 다루고, 기존 관계형 데이터베이스에 벡터 검색 기능을 추가하는 pgvector도 함께 살펴봅니다.

**Qdrant**부터 시작하겠습니다. Qdrant는 Rust 언어로 작성된 오픈소스 벡터 데이터베이스입니다. Rust는 메모리 안전성과 높은 실행 속도를 동시에 추구하는 시스템 프로그래밍 언어입니다. Qdrant가 Rust로 작성되었다는 것은, 메모리 관리의 오버헤드가 적고 단일 서버에서의 성능이 높다는 것을 의미합니다.

Qdrant의 아키텍처는 크게 세 부분으로 나뉩니다. **컬렉션**(Collection), **포인트**(Point), **페이로드**(Payload)입니다.

**컬렉션**은 벡터를 저장하는 최상위 단위입니다. 3.2.2에서 Pinecone의 인덱스가 벡터 저장의 기본 단위인 것과 유사합니다. 컬렉션을 생성할 때 벡터의 차원과 유사도 측정 방식(코사인, 유클리드, 내적)을 지정합니다. 하나의 Qdrant 인스턴스에 여러 컬렉션을 만들 수 있으므로, 용도별로 컬렉션을 분리하여 관리할 수 있습니다.

**포인트**는 컬렉션에 저장되는 개별 데이터 항목입니다. 하나의 포인트는 고유 ID, 벡터, 페이로드로 구성됩니다. 벡터는 임베딩 모델이 생성한 숫자 배열이고, 페이로드는 벡터에 딸린 부가 정보입니다.

**페이로드**(Payload)는 Qdrant에서 메타데이터를 부르는 이름입니다. 3.2.1에서 벡터 데이터베이스가 벡터와 함께 메타데이터를 저장한다고 했는데, Qdrant에서는 이 메타데이터를 페이로드라고 부릅니다. 페이로드는 JSON 형태의 키-값 쌍으로, 문자열, 숫자, 불리언, 배열 등 다양한 타입을 지원합니다.

```
Qdrant 데이터 구조

Collection: "documents" (차원: 1536, 거리: cosine)
  |
  +-- Point (id: 1)
  |     벡터: [0.012, -0.034, 0.056, ...]  (1536차원)
  |     페이로드: {
  |       "source": "manual.pdf",
  |       "page": 15,
  |       "category": "installation",
  |       "date": "2025-01-15"
  |     }
  |
  +-- Point (id: 2)
  |     벡터: [0.045, 0.023, -0.067, ...]
  |     페이로드: {
  |       "source": "faq.md",
  |       "category": "troubleshooting"
  |     }
```

Qdrant가 다른 벡터 데이터베이스와 구별되는 핵심 특징은 **필터링과 페이로드** 처리의 유연성입니다. Qdrant는 벡터 유사도 검색과 페이로드 기반 필터링을 하나의 질의에서 수행할 수 있습니다. 3.2.1에서 사전 필터링과 사후 필터링을 설명했는데, Qdrant는 내부적으로 두 방식을 자동으로 최적화합니다. 필터 조건이 걸러내는 데이터의 비율에 따라 어떤 방식이 더 효율적인지를 판단하고, 최적의 전략을 선택합니다.

페이로드 필터링에서 지원하는 조건은 다양합니다. 값이 일치하는지(match), 범위에 들어가는지(range), 특정 값이 포함되어 있는지(has_id), 지리적 위치 기반 필터(geo_bounding_box) 등을 지원합니다. 여러 조건을 and, or, not으로 조합하여 복잡한 필터를 구성할 수도 있습니다. 예를 들어, "category가 installation이고, date가 2025년 이후인 문서 중에서 질문과 가장 유사한 5개를 찾아라"라는 질의를 하나의 API 호출로 수행할 수 있습니다.

Qdrant의 또 다른 특징은 네임드 벡터(named vector)를 지원한다는 것입니다. 하나의 포인트에 여러 벡터를 이름을 붙여 저장할 수 있습니다. 예를 들어, 같은 문서에 대해 제목 벡터와 본문 벡터를 따로 저장하고, 검색 시 어떤 벡터를 기준으로 할지 선택할 수 있습니다. 이 기능은 멀티모달 검색이나 다중 관점 검색에 활용됩니다.

Qdrant는 Docker로 간편하게 설치할 수 있으며, REST API와 gRPC API를 모두 제공합니다. Python, TypeScript, Rust 등 여러 언어의 클라이언트 라이브러리가 있고, 자체 클라우드 서비스(Qdrant Cloud)도 운영하고 있어 관리형과 자체 호스팅 중 선택할 수 있습니다.

이제 **Milvus**를 살펴보겠습니다. Milvus는 대규모 벡터 데이터를 분산 환경에서 처리하기 위해 설계된 오픈소스 벡터 데이터베이스입니다. Milvus의 핵심 설계 방향은 수억 건 이상의 벡터를 다루면서도 안정적인 검색 성능을 유지하는 것입니다.

Milvus의 아키텍처는 마이크로서비스 기반으로 설계되어 있습니다. 여러 컴포넌트가 독립적으로 동작하며, 각 컴포넌트를 개별적으로 확장할 수 있습니다. 주요 컴포넌트는 다음과 같습니다.

**프록시**(Proxy)는 클라이언트의 요청을 받아 적절한 노드로 라우팅하는 진입점입니다. **코디네이터**(Coordinator)는 데이터의 분산 배치, 인덱스 구축 스케줄링, 질의 분배를 관리합니다. **워커 노드**(Worker Node)는 실제로 데이터를 저장하고 검색을 수행하는 노드로, 쿼리 노드와 데이터 노드로 나뉩니다. **메시지 큐**(Message Queue)는 컴포넌트 간의 비동기 통신을 처리합니다. **오브젝트 스토리지**(Object Storage)는 벡터 데이터를 영구 저장하는 저장소로, MinIO나 S3 같은 분산 저장소를 사용합니다.

```
Milvus 아키텍처 (분산 모드)

+------------------+
|   클라이언트     |
+------------------+
        |
        v
+------------------+
|     프록시       |   <-- 요청 수신 및 라우팅
+------------------+
        |
   +----+----+
   |         |
   v         v
+--------+ +--------+
|코디네이| |메시지  |   <-- 분산 조율 및 통신
|터      | |큐      |
+--------+ +--------+
   |         |
   v         v
+--------+ +--------+
|쿼리    | |데이터  |   <-- 검색 수행 / 데이터 저장
|노드    | |노드    |
+--------+ +--------+
        |
        v
+------------------+
| 오브젝트 스토리지 |   <-- 영구 저장 (S3, MinIO)
+------------------+
```

이 구조에서 핵심은 각 컴포넌트를 독립적으로 확장할 수 있다는 점입니다. 검색 트래픽이 많으면 쿼리 노드만 늘리고, 데이터 삽입이 많으면 데이터 노드를 늘립니다. 이 유연성이 Milvus가 대규모 환경에 적합한 이유입니다.

Milvus에서 데이터를 조직하는 핵심 개념은 **파티션**(Partition)과 **샤딩**(Sharding)입니다.

**파티션**은 하나의 컬렉션을 논리적으로 분할하는 단위입니다. 3.2.2에서 Pinecone의 네임스페이스가 인덱스 안에서 벡터를 논리적으로 구분한 것과 유사합니다. 예를 들어, 연도별로 파티션을 만들어 2024년 문서와 2025년 문서를 분리할 수 있습니다. 검색 시 특정 파티션만 대상으로 지정하면, 전체 데이터를 검색하지 않으므로 속도가 빨라집니다.

**샤딩**은 하나의 컬렉션의 데이터를 여러 물리적 노드에 분산 저장하는 기법입니다. 데이터가 하나의 서버에 담기지 않을 만큼 클 때, 데이터를 여러 서버에 나누어 저장합니다. 검색 시에는 모든 샤드에 동시에 질의를 보내고, 각 샤드의 결과를 합쳐 최종 결과를 만듭니다. 샤딩은 Milvus가 자동으로 관리하므로, 사용자가 데이터의 물리적 배치를 직접 지정할 필요는 없습니다.

파티션과 샤딩의 차이를 정리하면, 파티션은 데이터를 논리적으로 분류하여 검색 범위를 좁히는 것이고, 샤딩은 데이터를 물리적으로 분산하여 저장 용량과 처리 능력을 확장하는 것입니다. 두 기법은 동시에 적용됩니다. 하나의 컬렉션이 여러 파티션으로 나뉘고, 각 파티션의 데이터가 다시 여러 샤드에 분산됩니다.

Milvus는 로컬 개발 환경에서 사용할 수 있는 경량 버전인 Milvus Lite와, 프로덕션 환경을 위한 분산 모드를 모두 제공합니다. Milvus Lite는 Python 라이브러리로 설치하여 SQLite처럼 단일 프로세스에서 실행할 수 있어, 프로토타이핑에 편리합니다. 분산 모드는 Kubernetes 위에서 Helm 차트로 배포하는 것이 일반적입니다.

이제 **Qdrant와 Milvus를 비교**하겠습니다. 두 데이터베이스는 모두 오픈소스이고 벡터 검색에 특화되어 있지만, 설계 철학과 적합한 사용 환경이 다릅니다.

```
비교 항목        Qdrant                    Milvus
--------------------------------------------------------------
개발 언어        Rust                      Go, C++
아키텍처         단일 바이너리/분산         마이크로서비스 기반 분산
설치 난이도      Docker로 간편              분산 모드는 Kubernetes 필요
확장 방식        수평 확장 (샤딩)           컴포넌트별 독립 확장
필터링           페이로드 기반, 매우 유연    스칼라 필터링 지원
다중 벡터        네임드 벡터 지원           다중 벡터 필드 지원
인덱스 알고리즘  HNSW 기본                  HNSW, IVF, DiskANN 등 다양
최적 규모        수백만~수천만 건           수천만~수억 건
커뮤니티         빠르게 성장 중             대규모 오픈소스 커뮤니티
```

Qdrant는 설치와 운영이 간편하고, 페이로드 필터링이 유연하며, 중소 규모(수백만에서 수천만 건)에서 뛰어난 성능을 보입니다. 단일 서버로 시작하여 점진적으로 확장할 수 있어, 초기 구축에 적합합니다. Milvus는 수억 건 이상의 대규모 데이터를 다루는 환경에서 강점을 발휘합니다. 마이크로서비스 아키텍처 덕분에 컴포넌트별로 독립적인 확장이 가능하고, IVF, DiskANN 등 다양한 인덱스 알고리즘을 선택할 수 있습니다. 다만 분산 모드의 운영 복잡도가 높습니다.

실무에서 어떤 것을 선택할지는 프로젝트의 규모와 운영 역량에 따라 달라집니다. 벡터 수가 수천만 건 이하이고 운영 인력이 제한적이라면 Qdrant가 적합합니다. 수억 건 이상의 대규모 데이터를 다루고 Kubernetes 운영 경험이 있다면 Milvus가 적합합니다.

마지막으로 **pgvector와 기존 DB의 벡터 확장**을 살펴보겠습니다. 지금까지 다룬 Pinecone, Weaviate, Qdrant, Milvus는 모두 벡터 검색을 위해 처음부터 설계된 전용 데이터베이스입니다. 그런데 조직에 이미 PostgreSQL 같은 관계형 데이터베이스가 운영되고 있다면, 벡터 전용 데이터베이스를 별도로 도입하는 것이 부담이 될 수 있습니다. 인프라가 하나 더 늘어나고, 데이터를 양쪽에서 동기화해야 하며, 운영 부담이 커집니다.

**pgvector**는 PostgreSQL에 벡터 데이터 타입과 유사도 검색 기능을 추가하는 확장(extension)입니다. pgvector를 설치하면 기존 PostgreSQL 테이블에 벡터 컬럼을 추가할 수 있고, SQL 문법으로 유사도 검색을 수행할 수 있습니다.

```sql
-- 벡터 컬럼이 있는 테이블 생성
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536)
);

-- 코사인 유사도 기반 검색
SELECT id, content, 1 - (embedding <=> query_vector) AS similarity
FROM documents
ORDER BY embedding <=> query_vector
LIMIT 5;
```

위 SQL에서 vector(1536)은 1,536차원의 벡터 타입입니다. <=> 연산자는 코사인 거리를 계산합니다. ORDER BY로 거리가 작은(유사도가 높은) 순서대로 정렬하고, LIMIT 5로 상위 5개를 가져옵니다. 기존 SQL 문법과 거의 동일한 형태로 벡터 검색을 수행할 수 있다는 것이 pgvector의 핵심 장점입니다.

pgvector는 HNSW 인덱스와 IVFFlat 인덱스를 지원합니다. 인덱스를 생성하면 전수 검색이 아닌 ANN 검색이 가능해져서 검색 속도가 크게 향상됩니다. 또한 벡터 검색과 일반 SQL 조건을 하나의 쿼리에서 결합할 수 있습니다. "category가 installation이면서 질문과 가장 유사한 문서 5개"를 SQL의 WHERE 절과 ORDER BY를 조합하여 표현할 수 있습니다.

pgvector의 장점은 기존 PostgreSQL 인프라를 그대로 활용할 수 있다는 것입니다. 별도의 벡터 데이터베이스를 운영하지 않아도 되고, 기존 테이블의 데이터와 벡터를 JOIN으로 결합할 수 있으며, PostgreSQL의 트랜잭션, 백업, 복제 기능을 그대로 사용할 수 있습니다. 단점은 전용 벡터 데이터베이스에 비해 대규모 벡터 처리의 성능이 떨어질 수 있다는 것입니다. 수백만 건까지는 충분히 사용할 수 있지만, 수천만 건 이상에서는 전용 벡터 데이터베이스가 더 유리합니다.

pgvector 외에도 기존 데이터베이스에 벡터 기능을 추가하는 사례가 늘고 있습니다. Redis는 RediSearch 모듈을 통해 벡터 검색을 지원하고, Elasticsearch는 kNN 검색 기능을 내장하여 BM25 키워드 검색과 벡터 검색을 하나의 시스템에서 수행할 수 있습니다. 이런 확장은 이미 해당 데이터베이스를 사용하고 있는 환경에서, 벡터 검색을 빠르게 도입할 때 유용합니다.

정리하면, Qdrant는 Rust 기반의 고성능 벡터 데이터베이스로, 페이로드 필터링이 유연하고 설치가 간편하여 중소 규모에서 강점을 보입니다. Milvus는 마이크로서비스 기반 분산 아키텍처로, 수억 건 이상의 대규모 벡터 데이터를 처리하는 데 적합하며, 파티션과 샤딩으로 데이터를 효율적으로 조직합니다. pgvector는 PostgreSQL에 벡터 검색 기능을 추가하는 확장으로, 별도의 인프라 없이 기존 데이터베이스에서 벡터 검색을 수행할 수 있습니다. 프로젝트의 규모, 운영 환경, 기존 인프라에 따라 적합한 선택이 달라집니다.

다음 단원인 3.2.5에서는 벡터 인덱스 관리와 최적화를 다룹니다. HNSW, IVF, PQ 등 주요 인덱스 알고리즘의 원리와 파라미터 튜닝 방법을 살펴봅니다.

이 단원을 마치면 Qdrant와 Milvus의 특성을 비교하고 적합한 사용 사례를 판단할 수 있습니다.
