# 8.3.1 RLHF

8.2.3에서 Red Teaming을 통해 LLM의 안전 취약점을 선제적으로 찾아내는 방법을 살펴보았습니다. Red Teaming은 모델의 약점을 발견하는 활동이지만, 발견된 약점을 근본적으로 개선하려면 모델 자체가 유해 요청을 거부하고, 사용자에게 도움이 되는 방향으로 답변하도록 학습시켜야 합니다. 이 학습 과정의 핵심 기법이 **RLHF(Reinforcement Learning from Human Feedback, 인간 피드백 기반 강화학습)**입니다.

RLHF가 필요한 이유를 먼저 이해해야 합니다. LLM의 기본 학습 목표는 "다음 토큰을 잘 예측하는 것"입니다. 1.2.3에서 다룬 것처럼, 사전 학습에서 모델은 방대한 텍스트의 다음 토큰을 예측하는 과정을 반복합니다. 7.1.1에서 다룬 파인튜닝(SFT, Supervised Fine-Tuning)에서는 지시-응답 쌍을 학습하여 모델이 지시를 따르도록 조정합니다. 그런데 이 두 단계만으로는 부족한 부분이 있습니다. 모델이 문법적으로 올바르고 지시를 따르는 응답을 생성하더라도, 그 응답이 "사용자에게 진정으로 도움이 되는가", "안전하고 편향 없는가", "불필요하게 장황하지 않은가"까지는 보장하지 못합니다. 다음 토큰 예측이라는 학습 목표와, 인간이 원하는 "좋은 응답"이라는 목표 사이에 간극이 있는 것입니다.

이 간극을 메우는 것이 RLHF의 역할입니다. RLHF는 인간의 선호도를 직접 학습 신호로 사용하여, 모델이 인간이 선호하는 방향으로 응답을 생성하도록 조정합니다. "이 두 응답 중 어느 쪽이 더 좋은가?"라는 인간의 판단을 모아서, 모델이 더 좋은 응답을 생성할 확률을 높이는 것입니다.

RLHF의 전체 파이프라인은 세 단계로 구성됩니다.

```
RLHF 파이프라인 3단계

1단계: SFT (Supervised Fine-Tuning)
  지시-응답 쌍으로 모델을 파인튜닝
  --> 지시를 따르는 기본 능력 확보
       |
       v
2단계: 보상 모델 (Reward Model) 학습
  인간 선호 데이터 수집 (A vs B 비교)
  선호 데이터로 보상 모델 학습
  --> "좋은 응답"에 높은 점수를 매기는 모델 확보
       |
       v
3단계: 강화학습으로 정책 최적화
  SFT 모델이 응답 생성
  보상 모델이 점수 부여
  PPO 알고리즘으로 높은 점수를 받는 방향으로 모델 갱신
  --> 인간 선호에 맞는 응답을 생성하는 모델 완성
```

1단계는 7.1.1에서 다룬 SFT와 동일합니다. 지시-응답 쌍으로 모델을 파인튜닝하여, 모델이 자연어 지시를 이해하고 응답하는 기본 능력을 갖추도록 합니다. RLHF에서 SFT는 출발점입니다.

2단계에서 **보상 모델(Reward Model)**을 학습합니다. 보상 모델은 "주어진 질문에 대한 응답이 얼마나 좋은지"를 점수로 평가하는 모델입니다. 이 모델이 필요한 이유는, 강화학습을 적용하려면 각 응답에 대한 보상(reward) 신호가 있어야 하기 때문입니다. 모든 응답을 사람에게 직접 평가받는 것은 비현실적이므로, 사람의 평가 패턴을 학습한 보상 모델이 사람을 대신합니다.

보상 모델의 학습 과정을 단계별로 설명하겠습니다. 먼저 **인간 선호 데이터**를 수집합니다. 하나의 질문에 대해 SFT 모델이 두 개의 응답을 생성합니다. 사람 평가자가 이 두 응답을 읽고, 어느 쪽이 더 좋은지를 선택합니다. "좋다"의 기준은 도움이 되는가, 정확한가, 안전한가, 편향이 없는가 등을 종합한 것입니다. 이렇게 수집된 (질문, 선호 응답, 비선호 응답) 삼중 쌍이 보상 모델의 학습 데이터가 됩니다.

```
인간 선호 데이터 수집 과정

질문: "기후 변화의 원인은 무엇인가?"

응답 A: "기후 변화는 주로 온실가스 배출 증가로
         발생합니다. 화석 연료 연소, 산림 벌채,
         산업 활동이 주요 원인입니다."

응답 B: "기후 변화는 복잡한 현상이라 한마디로
         설명하기 어렵습니다. 다양한 요인이 있죠."

사람 평가자 판단: A가 더 좋음
  이유: 구체적이고 정확하며, 핵심 원인을 명시함

수집 결과:
  (질문, 선호=A, 비선호=B)
```

보상 모델은 이 삼중 쌍들을 학습하여, 선호 응답에 더 높은 점수를 부여하고 비선호 응답에 더 낮은 점수를 부여하도록 훈련됩니다. 학습이 완료된 보상 모델은 새로운 (질문, 응답) 쌍이 주어졌을 때, 그 응답의 품질을 스칼라 값(하나의 숫자)으로 출력합니다. 이 점수가 3단계에서 강화학습의 보상 신호로 사용됩니다.

3단계에서 **PPO(Proximal Policy Optimization)** 알고리즘을 사용하여 SFT 모델을 최적화합니다. 강화학습의 기본 구조를 먼저 이해해야 PPO를 이해할 수 있습니다. 강화학습에서는 **에이전트**가 **환경**에서 **행동**을 취하고, 그 결과로 **보상**을 받습니다. RLHF에서 이 구조를 대응시키면 다음과 같습니다.

에이전트는 SFT 모델(정책, policy)입니다. 환경은 텍스트 생성 과정입니다. 행동은 각 시점에서 토큰을 선택하는 것입니다. 보상은 생성이 완료된 뒤 보상 모델이 매기는 점수입니다. 모델이 응답 전체를 생성하면, 보상 모델이 그 응답에 점수를 매기고, 이 점수를 높이는 방향으로 모델의 파라미터를 갱신합니다.

PPO가 다른 강화학습 알고리즘 대신 사용되는 이유가 있습니다. 강화학습에서 모델(정책)을 갱신할 때, 한 번에 너무 크게 변경하면 학습이 불안정해집니다. 갱신 전의 정책과 갱신 후의 정책이 크게 다르면, 보상 모델의 평가가 더 이상 유효하지 않을 수 있기 때문입니다. PPO는 정책 변경의 크기를 제한하여 이 문제를 방지합니다. 구체적으로, PPO는 갱신 전후 정책의 확률비(probability ratio)가 일정 범위(보통 1 - epsilon에서 1 + epsilon, epsilon은 0.1~0.2)를 벗어나지 않도록 클리핑(clipping)합니다. 이 덕분에 학습이 안정적으로 진행됩니다.

```
PPO 학습 과정

1. SFT 모델이 질문에 대한 응답을 생성한다
       |
       v
2. 보상 모델이 응답에 점수를 매긴다
   (예: 0.85)
       |
       v
3. 보상 점수를 기반으로 정책 그래디언트를 계산한다
   높은 점수 응답의 생성 확률을 높이는 방향
       |
       v
4. PPO 클리핑으로 갱신 크기를 제한한다
   정책 변경이 너무 크면 잘라냄
       |
       v
5. KL 발산 페널티를 적용한다
   원래 SFT 모델에서 너무 멀어지지 않도록 제약
       |
       v
6. 모델 파라미터를 갱신한다

1~6을 반복
```

5단계에서 언급한 **KL 발산 페널티**는 RLHF에서 특히 중요한 장치입니다. KL 발산(Kullback-Leibler Divergence)은 두 확률 분포가 얼마나 다른지를 측정하는 지표입니다. RLHF에서는 현재 학습 중인 모델의 출력 분포와 원래 SFT 모델의 출력 분포 사이의 KL 발산을 보상에서 빼줍니다. 이렇게 하는 이유는, 보상 모델의 점수만 높이려다 보면 모델이 극단적인 방향으로 변할 수 있기 때문입니다. 보상 모델이 긴 답변에 높은 점수를 주는 경향이 있다면, 모델이 불필요하게 장황한 답변만 생성하게 될 수 있습니다. KL 발산 페널티는 이런 과적합을 방지하여, 모델이 SFT의 기본 능력을 유지하면서도 인간 선호에 맞게 조정되도록 합니다.

다음으로 **인간 선호 데이터 수집 방법**을 더 구체적으로 살펴보겠습니다. 보상 모델의 품질은 선호 데이터의 품질에 직접적으로 의존합니다. 선호 데이터 수집 시 고려해야 할 사항들이 있습니다.

평가 기준을 명확히 정의해야 합니다. "좋은 응답"이 무엇인지에 대한 구체적인 가이드라인을 평가자에게 제공합니다. 도움이 되는가(helpfulness), 정직한가(honesty), 안전한가(harmlessness)라는 세 가지 축이 대표적입니다. 평가자가 이 축들 사이에서 충돌이 발생할 때 어떤 축을 우선해야 하는지도 명시합니다. 예를 들어, 도움이 되는 답변이지만 안전하지 않은 경우에는 안전성을 우선합니다.

평가자 간 일치도를 관리해야 합니다. 8.1.1에서 인간 평가 프로토콜을 다룰 때 언급한 것처럼, 같은 응답 쌍에 대해 여러 평가자의 판단이 일치하는지를 확인합니다. 일치도가 낮으면 가이드라인이 모호하다는 뜻이므로, 가이드라인을 보완하고 평가자를 재교육합니다.

다양한 질문 유형을 포함해야 합니다. 특정 유형의 질문에만 선호 데이터가 편중되면, 보상 모델도 해당 유형에만 잘 동작하고 다른 유형에서는 부정확한 점수를 매깁니다. 다양한 주제, 난이도, 형식의 질문을 고르게 포함하는 것이 중요합니다.

이제 RLHF의 대안으로 등장한 **DPO(Direct Preference Optimization)**를 살펴보겠습니다. DPO는 RLHF의 파이프라인을 크게 단순화한 기법입니다.

RLHF가 세 단계(SFT, 보상 모델 학습, PPO 최적화)를 거치는 반면, DPO는 보상 모델을 별도로 학습하지 않습니다. 대신, 인간 선호 데이터를 직접 사용하여 모델을 최적화합니다. DPO의 핵심 아이디어는, 보상 모델이 수행하는 역할을 모델 자체의 확률로 대체할 수 있다는 수학적 관찰에서 출발합니다. 선호 응답의 생성 확률은 높이고 비선호 응답의 생성 확률은 낮추는 방향으로, 일반적인 지도학습(supervised learning)의 형태로 모델을 직접 최적화합니다.

```
RLHF vs DPO 파이프라인 비교

RLHF:
  SFT --> 보상 모델 학습 --> PPO 최적화
  (3단계, 복잡)

DPO:
  SFT --> 선호 데이터로 직접 최적화
  (2단계, 단순)

DPO 학습:
  입력: (질문, 선호 응답, 비선호 응답) 쌍
  목표: 선호 응답의 생성 확률 높이기
        비선호 응답의 생성 확률 낮추기
  방식: 지도학습과 유사한 손실 함수
```

DPO의 장점은 구현이 단순하다는 것입니다. 보상 모델을 별도로 학습하고 관리할 필요가 없고, 강화학습의 불안정한 최적화 과정을 피할 수 있습니다. 일반적인 파인튜닝과 비슷한 코드로 실행할 수 있어 실무 적용 장벽이 낮습니다. 단점은 보상 모델만큼 세밀한 보상 신호를 활용하지 못한다는 것입니다. RLHF에서 보상 모델은 다양한 응답에 연속적인 점수를 매기지만, DPO는 선호/비선호라는 이진 신호만 사용합니다.

마지막으로 **RLHF의 한계와 대안**을 짚겠습니다. RLHF는 강력한 기법이지만, 실무에서 적용할 때 여러 한계가 있습니다. 첫째, 인간 선호 데이터 수집 비용이 높습니다. 수천에서 수만 건의 비교 평가를 사람에게 맡겨야 하므로 시간과 비용이 상당합니다. 둘째, 보상 해킹(reward hacking) 문제가 있습니다. 모델이 보상 모델의 약점을 악용하여, 실제로 좋은 응답이 아님에도 높은 보상을 받는 방법을 찾아내는 현상입니다. 예를 들어, 보상 모델이 길이에 편향되어 있으면, 모델은 불필요하게 긴 응답을 생성하여 높은 점수를 받으려 합니다. 셋째, 학습 과정이 복잡합니다. SFT, 보상 모델, PPO라는 세 가지 별도의 학습 과정을 관리해야 하며, 각 단계의 하이퍼파라미터 조정이 까다롭습니다.

이런 한계를 극복하기 위한 대안들이 연구되고 있습니다. DPO는 앞서 다룬 것처럼 파이프라인을 단순화하는 방향의 대안입니다. 8.3.2에서 다룰 RLAIF(Reinforcement Learning from AI Feedback)는 인간 평가자 대신 AI가 피드백을 제공하여 비용을 줄이는 방향의 대안입니다. Constitutional AI는 미리 정의한 원칙에 따라 AI 자체가 자기 출력을 수정하고 평가하는 접근법입니다.

정리하면, RLHF는 인간의 선호도를 학습 신호로 사용하여, LLM이 인간이 원하는 방향으로 응답을 생성하도록 조정하는 기법입니다. SFT로 기본 능력을 갖춘 뒤, 인간 선호 데이터로 보상 모델을 학습하고, PPO 알고리즘으로 모델을 최적화하는 세 단계를 거칩니다. DPO는 보상 모델 없이 선호 데이터만으로 직접 최적화하는 단순화된 대안입니다. RLHF는 비용, 보상 해킹, 복잡성이라는 한계가 있으며, DPO, RLAIF, Constitutional AI 등의 대안이 연구되고 있습니다.

다음 단원인 8.3.2에서는 RLAIF와 자동 피드백 시스템을 다룹니다. 인간 피드백의 비용 문제를 AI 피드백으로 해결하는 접근법입니다.

이 단원을 마치면 RLHF의 전체 파이프라인을 단계별로 설명할 수 있습니다.
