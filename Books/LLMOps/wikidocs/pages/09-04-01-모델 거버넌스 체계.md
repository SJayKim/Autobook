# 9.4.1 모델 거버넌스 체계

LLM 애플리케이션을 운영하다 보면, 기술적인 성능과 비용 외에 또 다른 차원의 문제가 부각됩니다. "이 모델의 응답이 편향되어 있지는 않은가", "누가 이 모델의 사용을 승인했는가", "문제가 발생했을 때 책임은 누구에게 있는가", "규제 기관의 감사에 응할 수 있는가" 같은 질문입니다. 9.3.3에서 Rate Limiting과 API Gateway를 통해 기술적인 운영 통제를 다루었습니다. 이 단원에서는 기술을 넘어 조직, 프로세스, 규정 차원에서 LLM 시스템을 통제하는 **모델 거버넌스**(model governance)를 다룹니다.

**모델 거버넌스**란 조직 내에서 AI 모델의 개발, 배포, 운영, 폐기에 이르는 전 생명주기를 체계적으로 관리하고 통제하는 것을 뜻합니다. 여기서 "통제"는 모델의 사용을 막는 것이 아니라, 모델이 안전하고 공정하며 법규를 준수하는 방식으로 운영되도록 보장하는 것입니다.

모델 거버넌스가 왜 필요한지를 구체적인 상황으로 설명하겠습니다. 한 금융 회사가 대출 심사를 보조하는 LLM 기반 시스템을 배포했다고 가정합니다. 이 시스템이 특정 인종이나 성별에 대해 불리한 판단을 내리고 있었다면, 회사는 규제 위반과 소송에 직면합니다. 그런데 누가 이 모델을 선정했는지, 어떤 데이터로 평가했는지, 언제 배포 승인이 났는지를 추적할 수 없다면, 문제의 원인을 파악하고 책임을 묻는 것이 불가능합니다. 모델 거버넌스는 이러한 상황을 예방하고, 문제 발생 시 추적과 대응이 가능한 체계를 만드는 것입니다.

모델 거버넌스의 필요성은 LLM의 특성에 의해 더욱 커집니다. 첫째, LLM은 범용적입니다. 하나의 모델이 고객 응대, 문서 요약, 코드 생성 등 다양한 용도로 쓰이므로, 각 용도별 위험을 별도로 평가해야 합니다. 둘째, LLM의 출력은 비결정적입니다. 같은 입력에도 다른 응답이 나올 수 있어, 사전에 모든 출력을 통제하는 것이 불가능합니다. 셋째, LLM의 학습 데이터가 편향을 포함하고 있을 수 있고, 이 편향이 출력에 반영될 수 있습니다. 넷째, AI에 대한 규제가 전 세계적으로 강화되고 있어, 법적 준수가 필수입니다.

모델 거버넌스를 체계적으로 구축하기 위한 참조 틀이 **AI 거버넌스 프레임워크**입니다. 대표적인 두 프레임워크를 살펴보겠습니다.

**NIST AI RMF**(AI Risk Management Framework)는 미국 국립표준기술연구소(NIST)가 2023년에 발표한 AI 위험 관리 프레임워크입니다. NIST는 도량형과 기술 표준을 담당하는 미국 정부 기관입니다. AI RMF는 법적 구속력은 없지만, AI 위험 관리의 모범 사례를 정리한 권위 있는 가이드라인으로 널리 참조됩니다.

NIST AI RMF는 네 가지 핵심 기능으로 구성됩니다. 첫 번째는 **Govern**(거버넌스)입니다. AI 시스템의 개발과 사용에 대한 정책, 절차, 책임 체계를 수립합니다. 조직의 가치와 원칙을 AI 운영에 반영하는 단계입니다. 두 번째는 **Map**(매핑)입니다. AI 시스템이 사용되는 맥락을 파악하고, 잠재적 위험을 식별합니다. "이 모델이 어디에, 어떻게, 누구에게 영향을 미치는가"를 정리하는 단계입니다. 세 번째는 **Measure**(측정)입니다. 식별된 위험을 정량적으로 평가합니다. 편향 정도, 정확도, 안전성 등을 측정합니다. 네 번째는 **Manage**(관리)입니다. 측정 결과를 바탕으로 위험을 완화하고, 지속적으로 모니터링합니다.

```
NIST AI RMF의 네 가지 핵심 기능

+-------------+     +-------------+     +-------------+     +-------------+
| GOVERN      |     | MAP         |     | MEASURE     |     | MANAGE      |
| 정책, 절차, |     | 맥락 파악,  |     | 위험의      |     | 위험 완화,  |
| 책임 수립   |---->| 위험 식별   |---->| 정량 평가   |---->| 지속 모니터 |
+-------------+     +-------------+     +-------------+     +-------------+
      |                                                           |
      +-----------------------------------------------------------+
                         (지속적 순환)
```

**EU AI Act**(유럽 인공지능법)는 2024년에 발효된 유럽연합의 AI 규제 법률입니다. NIST AI RMF가 자발적 가이드라인인 것과 달리, EU AI Act는 법적 구속력이 있습니다. 위반 시 과징금이 부과됩니다. EU AI Act의 핵심은 **위험 기반 접근**(risk-based approach)입니다. AI 시스템을 위험 수준에 따라 네 단계로 분류하고, 위험이 높을수록 엄격한 요건을 부과합니다.

첫 번째 단계는 '허용 불가 위험'(Unacceptable Risk)입니다. 사회적 점수화 시스템, 실시간 원격 생체 인식 등은 원칙적으로 금지됩니다. 두 번째 단계는 '고위험'(High Risk)입니다. 채용, 대출 심사, 교육 평가, 의료 진단 등에 사용되는 AI가 이에 해당합니다. 적합성 평가, 데이터 거버넌스, 인간 감독, 기술 문서화 등 엄격한 요건을 충족해야 합니다. 세 번째 단계는 '제한 위험'(Limited Risk)입니다. 챗봇처럼 사용자와 상호작용하는 AI 시스템은, 사용자에게 AI와 대화하고 있다는 사실을 알려야 합니다. 네 번째 단계는 '최소 위험'(Minimal Risk)입니다. 스팸 필터 같은 시스템은 추가 규제 없이 운영할 수 있습니다.

LLM 기반 서비스가 어느 단계에 해당하는지는 용도에 따라 달라집니다. 같은 LLM이라도 고객 상담에 쓰이면 제한 위험이지만, 대출 심사에 쓰이면 고위험으로 분류될 수 있습니다. 따라서 모델 자체가 아니라 모델의 사용 맥락에 따라 위험 수준을 평가해야 합니다.

프레임워크가 요구하는 핵심 산출물 중 하나가 **모델 카드**(Model Card)입니다. 모델 카드란 모델의 특성, 용도, 한계, 평가 결과를 표준화된 형식으로 정리한 문서입니다. 구글 연구팀이 2019년에 제안한 개념으로, AI 투명성을 높이기 위한 실천 수단입니다.

모델 카드에 포함되는 주요 항목을 살펴보겠습니다. 첫째, 모델 개요입니다. 모델의 이름, 버전, 개발자/제공자, 모델 유형(예: 디코더 전용 트랜스포머), 학습 데이터의 개괄적 설명을 적습니다. 둘째, 의도된 용도입니다. 이 모델이 어떤 작업에 적합한지, 어떤 환경에서 사용해야 하는지를 명시합니다. 셋째, 의도되지 않은 용도입니다. 이 모델이 적합하지 않은 작업이나 환경을 명시합니다. 예를 들어 "의료 진단의 최종 판단에 사용하지 마시오" 같은 경고입니다. 넷째, 성능 지표입니다. 정확도, F1 점수, 환각률 등의 평가 결과를 인구 통계 하위 집단(성별, 연령, 언어 등)별로 분리하여 제시합니다. 다섯째, 한계와 편향입니다. 모델의 알려진 한계와 편향을 투명하게 기술합니다. 여섯째, 윤리적 고려사항입니다. 모델 사용으로 인해 발생할 수 있는 윤리적 문제와 완화 조치를 기술합니다.

```
모델 카드 구조 (간략)

+----------------------------------+
| 모델 카드: customer-support-v3   |
+----------------------------------+
| 모델 개요                        |
|   이름: customer-support-v3      |
|   기반: GPT-4o + 커스텀 프롬프트 |
|   버전: 3.0 (2026-03-15)        |
+----------------------------------+
| 의도된 용도                      |
|   - 고객 문의 자동 응답          |
|   - FAQ 기반 안내                |
+----------------------------------+
| 의도되지 않은 용도               |
|   - 법적 조언                    |
|   - 의료 상담                    |
+----------------------------------+
| 성능 지표                        |
|   정확도: 92% (전체)             |
|   한국어: 94%, 영어: 91%         |
|   환각률: 3.2%                   |
+----------------------------------+
| 한계와 편향                      |
|   - 다국어 지원 제한 (한/영만)   |
|   - 기술 용어 환각 가능성        |
+----------------------------------+
| 윤리적 고려사항                  |
|   - 사용자에게 AI 응답임을 고지  |
|   - 인간 상담원 연결 경로 유지   |
+----------------------------------+
```

모델 카드를 작성하는 것은 단순한 문서화 작업이 아닙니다. 모델의 용도, 한계, 위험을 체계적으로 검토하는 과정 자체가 거버넌스의 핵심입니다. 또한 모델 카드는 팀 간 소통의 도구이기도 합니다. 모델 개발팀과 운영팀, 법무팀, 경영진이 모델의 특성과 위험에 대해 공통된 이해를 갖게 됩니다.

모델 거버넌스에서 또 하나의 핵심 요소는 **감사 추적**(Audit Trail)입니다. 감사 추적이란 시스템에서 누가, 언제, 무엇을, 왜 했는지를 시간순으로 기록하여, 나중에 되짚어 볼 수 있게 하는 것입니다. 거버넌스 맥락에서 감사 추적에 포함해야 하는 항목은 다음과 같습니다.

첫째, **모델 선정 이력**입니다. 어떤 모델을 왜 선택했는지, 어떤 대안을 검토했는지, 누가 최종 승인했는지를 기록합니다. 둘째, **평가 이력**입니다. 모델을 배포하기 전에 어떤 데이터셋으로, 어떤 지표를 사용하여 평가했는지, 결과가 어떠했는지를 기록합니다. 셋째, **배포 이력**입니다. 언제 배포되었는지, 어떤 버전(프롬프트, 모델, 코드)이 배포되었는지, 누가 배포를 승인했는지를 기록합니다. 넷째, **변경 이력**입니다. 프롬프트 변경, 모델 교체, 설정 변경이 언제, 왜, 누구에 의해 이루어졌는지를 기록합니다. 다섯째, **인시던트 이력**입니다. 문제가 발생했을 때 무엇이 문제였는지, 어떻게 대응했는지, 재발 방지를 위해 어떤 조치를 취했는지를 기록합니다.

이 기록들이 있으면 규제 기관의 감사에 대응할 수 있습니다. "이 모델은 언제 배포되었고, 배포 전에 편향 테스트를 수행했으며, 결과는 이러했다"는 식의 증빙이 가능합니다. 9.3.1에서 다룬 CI/CD 파이프라인의 로그, 9.1.2와 9.1.3에서 다룬 옵저버빌리티 도구의 트레이스가 감사 추적의 기술적 기반이 됩니다.

**책임 있는 AI**(Responsible AI) 원칙은 거버넌스의 철학적 기반입니다. 구체적인 원칙의 표현은 조직마다 다르지만, 공통적으로 포함되는 핵심 원칙을 정리하면 다음과 같습니다.

첫째, **공정성**(Fairness)입니다. AI 시스템이 특정 집단에 대해 불공정한 결과를 내지 않아야 합니다. LLM의 맥락에서는 모델의 응답이 인종, 성별, 나이, 국적 등에 따라 차별적이지 않은지를 평가하고 모니터링합니다. 둘째, **투명성**(Transparency)입니다. AI 시스템의 작동 방식과 한계를 이해관계자에게 공개합니다. 모델 카드가 투명성을 실현하는 수단입니다. 셋째, **책임성**(Accountability)입니다. AI 시스템의 결정에 대해 누가 책임을 지는지를 명확히 합니다. 감사 추적이 책임성을 뒷받침합니다. 넷째, **안전성**(Safety)입니다. AI 시스템이 유해한 결과를 생성하지 않도록 보호합니다. 유해 콘텐츠 필터, 가드레일이 여기에 해당합니다. 다섯째, **프라이버시**(Privacy)입니다. 사용자의 개인정보를 보호합니다. 9.4.2에서 다루는 PII 탐지와 데이터 프라이버시가 이 원칙의 기술적 구현입니다.

이러한 원칙을 조직에서 실행하기 위한 구조가 **거버넌스 위원회**(governance committee)입니다. 거버넌스 위원회는 AI 시스템의 개발과 운영에 대한 의사결정과 감독을 담당하는 조직 내 공식 기구입니다. 위원회의 구성과 역할을 살펴보겠습니다.

위원회에는 다양한 분야의 전문가가 참여합니다. 기술 리더(CTO, AI 엔지니어링 리드)는 기술적 위험과 완화 방안에 대한 전문 의견을 제공합니다. 법무/컴플라이언스 담당자는 규제 준수 여부를 검토합니다. 사업부 대표는 모델 사용의 사업적 맥락과 영향을 설명합니다. 데이터 프라이버시 담당자는 개인정보 보호 관점에서 검토합니다. 윤리 전문가(있는 경우)는 공정성과 사회적 영향을 평가합니다.

위원회의 주요 프로세스는 다음과 같습니다. 첫째, 새 모델이나 새 용도의 배포 전에 **배포 승인 심사**를 수행합니다. 모델 카드를 검토하고, 위험 수준을 평가하며, 필요한 완화 조치를 확인합니다. 둘째, 운영 중인 모델에 대해 **정기 점검**을 수행합니다. 성능 저하, 편향 변화, 새로운 규제 대응 필요성 등을 확인합니다. 셋째, 인시던트 발생 시 **사후 검토**(post-mortem)를 주관합니다. 원인을 분석하고 재발 방지 대책을 수립합니다.

```
거버넌스 위원회 프로세스

[모델/용도 제안]
    |
    v
[배포 전 심사]
  - 모델 카드 검토
  - 위험 수준 평가 (EU AI Act 분류 참조)
  - 편향 테스트, 안전성 테스트 결과 확인
  - 완화 조치 확인
    |
    +---> 승인 --> [배포]
    |                 |
    +---> 조건부 승인 --> [조치 이행 후 재심사]
    |
    +---> 거부 --> [사유 통보, 대안 검토]

[운영 중]
    |
    v
[정기 점검] (분기/반기)
  - 성능, 편향, 비용, 인시던트 리뷰
  - 규제 변경 대응 확인

[인시던트 발생 시]
    |
    v

  - 근본 원인 분석

```

정리하면, 모델 거버넌스는 LLM 시스템의 전 생명주기를 조직 차원에서 체계적으로 관리하는 것입니다. NIST AI RMF의 Govern-Map-Measure-Manage 프레임워크와 EU AI Act의 위험 기반 접근이 거버넌스의 참조 틀을 제공합니다. 모델 카드로 모델의 특성, 용도, 한계를 문서화하고, 감사 추적으로 모든 의사결정과 변경의 이력을 남기며, 책임 있는 AI 원칙(공정성, 투명성, 책임성, 안전성, 프라이버시)을 조직의 운영에 반영합니다. 거버넌스 위원회가 배포 승인, 정기 점검, 사후 검토를 주관하여, 기술적 통제와 조직적 통제를 연결합니다.

다음 단원인 9.4.2에서는 PII 탐지와 데이터 프라이버시를 다룹니다. 거버넌스 체계의 프라이버시 원칙을 기술적으로 구현하는 방법을 살펴봅니다.

이 단원을 마치면 LLM 거버넌스 프레임워크를 설계하고 조직에 적용할 수 있습니다.
