# 4.2.1 검색 품질 최적화

4.1.3에서 문서를 적절한 크기의 청크로 나누는 전략을 살펴보았습니다. 청킹이 끝나면 각 청크를 임베딩으로 변환하여 벡터 데이터베이스에 저장하고, 사용자의 질문이 들어올 때 관련 청크를 검색하여 LLM에 전달합니다. 그런데 이 검색 단계에서 관련 없는 청크가 상위에 올라오거나, 정작 필요한 청크가 누락되면 LLM이 아무리 뛰어나도 정확한 답을 생성할 수 없습니다. RAG 시스템의 최종 응답 품질은 검색 단계의 품질에 직접적으로 의존합니다. 이 단원에서는 검색 품질을 높이기 위한 세 가지 핵심 기법, 즉 하이브리드 검색, 리랭킹, 쿼리 변환을 다룹니다.

먼저 기본 RAG 파이프라인에서 검색이 어떻게 이루어지는지 되짚어 보겠습니다. 4.1.2에서 구축한 기본 파이프라인은 사용자의 질문을 임베딩 벡터로 변환한 뒤, 벡터 데이터베이스에서 코사인 유사도가 높은 청크를 상위 k개 가져오는 방식으로 검색합니다. 이 방식을 **시맨틱 검색**(semantic search)이라 합니다. 시맨틱 검색은 단어가 정확히 일치하지 않더라도 의미가 비슷한 문서를 찾아낼 수 있다는 강점이 있습니다. 예를 들어, "자동차 연비를 높이는 방법"이라는 질문에 대해 "차량의 연료 효율을 개선하는 기술"이라는 문서를 찾아낼 수 있습니다. 단어는 다르지만 의미가 통하기 때문입니다.

그러나 시맨틱 검색에는 분명한 한계가 있습니다. 첫째, 고유명사나 기술 용어처럼 정확한 단어 일치가 중요한 경우에 취약합니다. "RFC 7231"이라는 특정 문서 번호를 질문에 포함했을 때, 임베딩 모델이 이 번호의 의미를 제대로 포착하지 못하면 관련 청크가 상위에 오지 않을 수 있습니다. 둘째, 임베딩 모델의 학습 데이터에 포함되지 않은 도메인 전문 용어에 대해서는 유사도 계산이 부정확해집니다. 셋째, 임베딩 벡터 간 유사도는 수치일 뿐이어서, 실제로 질문에 대한 답을 담고 있는지 여부와는 다를 수 있습니다. 의미적으로 비슷한 문장이라도 질문의 답이 아닌 경우가 있고, 반대로 표면적 유사도는 낮지만 핵심 정보를 담고 있는 문서가 밀려나기도 합니다.

시맨틱 검색과 대비되는 전통적인 검색 방식이 **키워드 검색**(keyword search)입니다. 키워드 검색은 질문에 포함된 단어가 문서에 얼마나 자주, 얼마나 중요하게 등장하는지를 기준으로 관련성을 판단합니다. 가장 대표적인 키워드 검색 알고리즘이 **BM25**입니다. BM25가 어떤 원리로 작동하는지 이해하면 이후 하이브리드 검색의 설계 의도를 파악하기 수월하므로, 그 동작 방식을 단계별로 살펴보겠습니다.

BM25는 질문에 포함된 각 단어에 대해 두 가지를 계산합니다. 하나는 **역문서 빈도**(IDF, Inverse Document Frequency)입니다. 전체 문서 집합에서 해당 단어가 등장하는 문서의 비율이 낮을수록, 그 단어는 특정 문서를 구별하는 데 더 유용하다고 봅니다. "the"처럼 거의 모든 문서에 나오는 단어는 IDF가 낮고, "kubernetes"처럼 특정 주제의 문서에만 나오는 단어는 IDF가 높습니다. 다른 하나는 **단어 빈도**(TF, Term Frequency)입니다. 해당 문서 안에서 그 단어가 몇 번 등장하는지를 셉니다. 다만 BM25는 단어 빈도가 높아질수록 점수 증가가 둔화되도록 포화 함수를 적용합니다. 한 문서에서 같은 단어가 100번 나온다고 10번 나온 것보다 10배 관련성이 높다고 보지는 않는다는 뜻입니다. 또한 문서 길이가 길수록 단어가 많이 나올 확률이 높으므로, 문서 길이에 따른 보정도 수행합니다. BM25는 이 값들을 질문의 모든 단어에 대해 합산하여 최종 관련성 점수를 산출합니다.

키워드 검색의 강점은 정확한 단어 일치에 있습니다. 고유명사, 약어, 에러 코드, 버전 번호 등 정확히 그 단어가 포함된 문서를 찾는 데 탁월합니다. 반면 약점은 동의어나 의미적 유사성을 전혀 고려하지 못한다는 것입니다. "자동차"라고 질문했을 때 "차량"이라는 단어만 포함된 문서는 관련 문서로 잡히지 않습니다. 시맨틱 검색과 키워드 검색은 서로 보완적인 특성을 갖고 있습니다.

```
시맨틱 검색과 키워드 검색의 특성 비교

                시맨틱 검색              키워드 검색 (BM25)
  ─────────────────────────────────────────────────────────
  원리          의미 유사도 (벡터)       단어 일치 (통계)
  강점          동의어, 의역 처리        고유명사, 약어 정확 매칭
  약점          고유명사 누락 가능       동의어 인식 불가
  표현 방식     Dense 벡터 (수백 차원)   Sparse 벡터 (어휘 크기 차원)
  ─────────────────────────────────────────────────────────
```

여기서 Dense와 Sparse라는 용어가 등장합니다. 시맨틱 검색에서 사용하는 임베딩 벡터는 수백에서 수천 차원의 실수값으로 이루어져 있고, 대부분의 차원에 0이 아닌 값이 들어 있습니다. 이런 벡터를 **Dense 벡터**(밀집 벡터)라 부릅니다. 반면 BM25 같은 키워드 검색은 전체 어휘 수만큼의 차원을 가진 벡터로 표현할 수 있는데, 특정 문서에 등장하는 단어는 전체 어휘의 극히 일부이므로 대부분의 차원이 0입니다. 이런 벡터를 **Sparse 벡터**(희소 벡터)라 부릅니다.

이제 두 검색 방식을 결합하는 **하이브리드 검색**(hybrid search)을 살펴보겠습니다. 하이브리드 검색의 기본 아이디어는 단순합니다. 하나의 질문에 대해 시맨틱 검색과 키워드 검색을 동시에 수행한 뒤, 두 결과를 합쳐서 최종 순위를 매기는 것입니다. 시맨틱 검색이 놓치기 쉬운 고유명사 매칭을 키워드 검색이 보완하고, 키워드 검색이 놓치기 쉬운 의미적 유사 문서를 시맨틱 검색이 보완합니다.

하이브리드 검색의 전체 흐름을 단계별로 정리하면 다음과 같습니다.

```
하이브리드 검색 파이프라인

  사용자 질문
       |
       v
  +----+----+
  |         |
  v         v
BM25      벡터 검색
(Sparse)  (Dense)
  |         |
  v         v
상위 k개  상위 k개
결과 목록  결과 목록
  |         |
  v         v
  +----+----+
       |
       v
   점수 융합
  (Score Fusion)
       |
       v
  최종 상위 k개
  통합 결과
```

첫 번째 단계에서 사용자의 질문이 들어오면, 이 질문을 BM25 엔진과 벡터 검색 엔진에 동시에 전달합니다. 두 번째 단계에서 각 엔진은 독립적으로 관련 문서를 점수순으로 정렬하여 상위 k개를 반환합니다. 세 번째 단계에서 두 결과 목록을 하나로 합칩니다. 이 합치는 과정을 점수 융합(score fusion)이라 하며, 여기서 핵심적인 문제가 발생합니다. BM25의 점수와 벡터 유사도 점수는 척도가 전혀 다릅니다. BM25 점수는 0에서 수십까지 다양한 범위를 가질 수 있고, 코사인 유사도는 -1에서 1 사이의 값입니다. 이 두 점수를 단순히 더하면 한쪽 검색의 결과가 지나치게 지배적이 될 수 있습니다.

이 문제를 해결하는 대표적인 방법이 **Reciprocal Rank Fusion**, 줄여서 **RRF**입니다. RRF는 각 문서의 원래 점수를 사용하지 않고, 대신 각 결과 목록에서의 순위(rank)만 사용합니다. 원리는 다음과 같습니다. 어떤 문서가 BM25 결과에서 3위, 벡터 검색 결과에서 7위였다고 가정합니다. RRF는 각 순위에 대해 1 / (k + rank) 값을 계산합니다. 여기서 k는 상수로, 일반적으로 60을 사용합니다. BM25 쪽 점수는 1 / (60 + 3) = 0.0159이고, 벡터 검색 쪽 점수는 1 / (60 + 7) = 0.0149입니다. 이 두 값을 더한 0.0308이 해당 문서의 RRF 점수가 됩니다.

RRF 점수 계산을 수식으로 표현하면 다음과 같습니다.

```
RRF(d) = sum( 1 / (k + rank_i(d)) )  (i = 각 검색 결과 목록)

  d     : 문서
  k     : 상수 (기본값 60)
  rank_i: i번째 결과 목록에서 문서 d의 순위 (1부터 시작)
```

RRF의 장점은 점수 정규화가 필요 없다는 것입니다. 원래 점수의 척도가 달라도 순위만 보기 때문에, BM25와 벡터 검색뿐 아니라 세 개 이상의 검색 결과를 합칠 때에도 동일한 방식으로 적용할 수 있습니다. 상수 k는 상위 순위와 하위 순위의 점수 차이를 조절하는 역할을 합니다. k가 크면 순위 간 점수 차이가 작아져서 하위 문서도 기회를 얻고, k가 작으면 상위 문서에 점수가 집중됩니다. 60이라는 기본값은 원래 RRF 논문에서 제안된 것으로, 실무에서도 대부분 이 값을 그대로 사용합니다.

RRF 외에도 점수 융합 방법이 있습니다. 가중 합산(weighted sum) 방식은 각 검색의 점수를 0에서 1 사이로 정규화한 뒤, 가중치를 곱해 더합니다. 예를 들어 BM25 정규화 점수에 0.3을, 벡터 검색 정규화 점수에 0.7을 곱해 합산하는 식입니다. 이 방식은 도메인 특성에 따라 한쪽 검색에 더 큰 비중을 줄 수 있다는 장점이 있지만, 점수 정규화 방법과 가중치를 실험적으로 튜닝해야 한다는 부담이 있습니다. 실무에서는 튜닝 부담이 적은 RRF가 먼저 시도되는 경우가 많습니다.

하이브리드 검색은 많은 벡터 데이터베이스에서 기본 기능으로 지원합니다. 예를 들어 Weaviate에서는 질의 시 alpha 파라미터로 키워드 검색과 벡터 검색의 비중을 조절할 수 있고, Pinecone에서는 Sparse-Dense 벡터를 함께 저장하여 하이브리드 검색을 수행합니다. Elasticsearch는 본래 BM25 기반 검색 엔진이므로 키워드 검색이 강점이며, kNN 검색 기능을 추가하여 벡터 검색과 결합할 수 있습니다. 이처럼 별도의 검색 엔진을 따로 구축하지 않아도 기존 벡터 데이터베이스의 기능만으로 하이브리드 검색을 적용할 수 있는 경우가 많습니다.

하이브리드 검색으로 후보 문서의 범위를 넓혔더라도, 상위에 올라온 문서가 실제로 질문에 대한 답을 담고 있는지는 별개의 문제입니다. 하이브리드 검색의 점수는 어디까지나 "질문과 문서가 얼마나 관련 있는가"를 대략적으로 추정한 것입니다. 이 추정을 더 정밀하게 만들기 위해 사용하는 기법이 **리랭킹**(reranking)입니다.

리랭킹의 기본 아이디어를 이해하려면, 임베딩 기반 검색이 질문과 문서를 어떻게 비교하는지를 다시 살펴볼 필요가 있습니다. 3.1에서 다룬 임베딩 모델은 질문과 문서를 각각 독립적으로 벡터로 변환합니다. 질문 벡터와 문서 벡터는 서로를 보지 못한 상태에서 생성됩니다. 이렇게 독립적으로 인코딩하는 구조를 **Bi-Encoder**라 합니다. Bi-Encoder의 장점은 문서 벡터를 미리 계산해 두면 질문이 들어올 때 질문 벡터만 새로 계산하면 되므로 매우 빠르다는 것입니다. 수백만 개의 문서를 대상으로 실시간 검색이 가능한 이유가 여기에 있습니다. 그러나 단점도 있습니다. 질문과 문서의 세부적인 상호작용을 포착하지 못합니다. "파이썬에서 리스트를 정렬하는 방법"이라는 질문과 "파이썬 리스트의 sort 메서드는 원본을 변경하고, sorted 함수는 새 리스트를 반환합니다"라는 문서가 있을 때, 두 텍스트 사이의 단어 대 단어 관계를 직접 비교하지 않고 각각을 하나의 벡터로 압축하기 때문에 미묘한 관련성을 놓칠 수 있습니다.

**Cross-Encoder**는 이 문제를 해결합니다. Cross-Encoder는 질문과 문서를 하나의 입력으로 이어 붙여서 모델에 함께 넣습니다. 모델 내부에서 질문의 각 단어와 문서의 각 단어가 서로 직접 상호작용(attention)하면서 관련성 점수를 산출합니다. 이 방식은 질문과 문서 사이의 세밀한 의미 관계를 파악할 수 있어서 정확도가 높습니다. 다만 치명적인 단점이 있습니다. 모든 후보 문서에 대해 질문과 함께 모델을 통과시켜야 하므로, 후보가 많아질수록 계산 시간이 선형으로 증가합니다. 수백만 개의 문서 전체에 Cross-Encoder를 적용하는 것은 현실적으로 불가능합니다.

```
Bi-Encoder vs Cross-Encoder 비교

Bi-Encoder (검색 단계):
  질문 --> [인코더] --> 질문 벡터 --+
                                     |--> 코사인 유사도
  문서 --> [인코더] --> 문서 벡터 --+
  * 문서 벡터는 사전 계산 가능 --> 빠름
  * 질문-문서 상호작용 없음 --> 정확도 제한

Cross-Encoder (리랭킹 단계):
  [질문 + 문서] --> [모델] --> 관련성 점수
  * 질문과 문서가 함께 입력 --> 상호작용 포착
  * 후보마다 모델 통과 필요 --> 느림
```

이 두 구조의 특성을 조합한 것이 **리랭킹 파이프라인**입니다. 전체 흐름은 두 단계로 나뉩니다. 첫 번째 단계인 초기 검색(retrieval)에서 Bi-Encoder 또는 하이브리드 검색으로 수백만 개의 문서 중 상위 20~100개의 후보를 빠르게 추립니다. 두 번째 단계인 리랭킹(reranking)에서 이 소수의 후보에 대해 Cross-Encoder를 적용하여 정밀하게 순위를 재조정합니다. 최종적으로 상위 5~10개만 LLM의 컨텍스트에 포함시킵니다.

```
리랭킹 파이프라인 전체 흐름

  사용자 질문
       |
       v
  [ 초기 검색 (Retrieval) ]
  BM25 + 벡터 검색 (하이브리드)
  수백만 문서 --> 상위 20~100개 후보
       |
       v
  [ 리랭킹 (Reranking) ]
  Cross-Encoder로 질문-문서 쌍 정밀 평가
  20~100개 후보 --> 상위 5~10개
       |
       v
  [ LLM 생성 ]
  상위 문서를 컨텍스트로 조립하여 응답 생성
```

이 구조에서 초기 검색은 "넓게 그물을 치는" 역할이고, 리랭킹은 "그물에 잡힌 것 중 진짜를 골라내는" 역할입니다. 초기 검색에서 후보가 누락되면 리랭킹에서도 복구할 수 없으므로, 초기 검색 단계에서는 재현율(recall), 즉 관련 문서를 빠뜨리지 않는 것이 중요합니다. 리랭킹 단계에서는 정밀도(precision), 즉 상위 결과의 관련성이 실제로 높은 것이 중요합니다.

실무에서 Cross-Encoder 리랭킹을 적용하는 방법은 크게 두 가지입니다. 하나는 Cohere Rerank, Jina Reranker 같은 API 서비스를 호출하는 것이고, 다른 하나는 오픈소스 Cross-Encoder 모델을 직접 서빙하는 것입니다. API 방식은 별도의 모델 서빙 인프라 없이 HTTP 호출만으로 리랭킹을 수행할 수 있어 도입이 간편합니다. 자체 서빙 방식은 비용을 통제할 수 있고 데이터가 외부로 나가지 않는다는 장점이 있지만, GPU 인프라와 모델 관리가 필요합니다.

지금까지 살펴본 하이브리드 검색과 리랭킹은 검색 엔진 쪽을 개선하는 접근입니다. 그런데 검색 품질이 떨어지는 원인이 검색 엔진이 아니라 질문 자체에 있는 경우도 있습니다. 사용자의 질문이 모호하거나, 너무 짧거나, 검색에 적합하지 않은 형태일 수 있습니다. 이때 질문을 검색에 더 적합한 형태로 바꾸는 기법이 **쿼리 변환**(query transformation)입니다.

쿼리 변환의 첫 번째 방식은 **쿼리 재작성**(Query Rewriting)입니다. 사용자의 원래 질문을 LLM에게 전달하여, 검색에 더 적합한 형태로 다시 쓰게 하는 것입니다. 예를 들어, 사용자가 "그거 어떻게 하는 거야?"라고 질문했다면, 대화 맥락을 참고하여 "쿠버네티스에서 파드의 로그를 확인하는 방법"처럼 구체적인 질문으로 변환합니다. 또는 하나의 질문을 여러 하위 질문으로 분해하여 각각 검색한 뒤 결과를 합치는 방법도 있습니다. "쿠버네티스와 도커 스웜의 차이점"이라는 질문을 "쿠버네티스의 주요 특징", "도커 스웜의 주요 특징"으로 나누어 검색하면 각각에 대한 관련 문서를 더 정확하게 찾을 수 있습니다.

쿼리 변환의 두 번째 방식은 **HyDE**(Hypothetical Document Embeddings)입니다. HyDE는 독특한 발상에 기반합니다. 사용자의 질문을 그대로 임베딩하여 검색하는 대신, LLM에게 "이 질문에 대한 가상의 답변"을 먼저 생성하게 합니다. 그런 다음 이 가상 답변을 임베딩하여 검색에 사용합니다. 왜 이런 우회를 하는 것일까요. 질문과 답변은 언어적 구조가 다릅니다. 질문은 "~는 무엇인가?"라는 의문문이고, 문서 속 답변은 "~는 ~이다"라는 서술문입니다. 임베딩 공간에서 질문 벡터보다 답변 벡터가 실제 문서의 벡터와 더 가까운 위치에 놓이는 경향이 있습니다. HyDE는 이 특성을 이용하여 검색 품질을 높입니다.

HyDE의 동작을 단계별로 정리하면 다음과 같습니다.

```
HyDE 동작 흐름

1. 사용자 질문 입력
   "BM25 알고리즘의 핵심 원리는?"
          |
          v
2. LLM이 가상 답변 생성 (실제 검색 없이)
   "BM25는 TF-IDF를 확장한 확률적 검색 모델로,
    단어 빈도의 포화 함수와 문서 길이 정규화를
    도입하여 ..."
          |
          v
3. 가상 답변을 임베딩 벡터로 변환
          |
          v
4. 이 벡터로 벡터 데이터베이스 검색
          |
          v
5. 검색된 실제 문서를 LLM에 전달하여 최종 응답 생성
```

HyDE가 생성하는 가상 답변은 사실 관계가 정확하지 않을 수 있습니다. 그러나 그것은 문제가 되지 않습니다. 가상 답변의 역할은 "정확한 정보를 제공하는 것"이 아니라 "관련 문서와 비슷한 어휘와 구조를 가진 텍스트를 만들어 검색 품질을 높이는 것"이기 때문입니다. 최종 응답은 검색된 실제 문서를 바탕으로 생성하므로, 가상 답변의 오류가 최종 응답에 전파되지 않습니다. 다만 HyDE는 LLM 호출이 한 번 추가되므로 지연 시간과 비용이 늘어나며, 질문이 이미 충분히 구체적인 경우에는 개선 효과가 크지 않을 수 있습니다.

쿼리 재작성과 HyDE는 상호 배타적이지 않습니다. 쿼리 재작성으로 질문을 명확하게 만든 뒤 HyDE를 적용하거나, 쿼리 재작성과 원본 질문으로 각각 검색한 결과를 RRF로 합치는 등 조합이 가능합니다. 어떤 기법이 효과적인지는 질문의 특성, 문서의 도메인, 임베딩 모델의 성능에 따라 달라지므로 실험을 통해 확인해야 합니다.

지금까지 다룬 기법들이 RAG 검색 파이프라인에서 어떤 위치에 놓이는지 전체 흐름으로 정리하겠습니다.

```
검색 품질 최적화가 적용된 RAG 파이프라인

  사용자 질문
       |
       v
  [쿼리 변환]  ........  쿼리 재작성 / HyDE
       |
       v
  [하이브리드 검색]  ...  BM25 + 벡터 검색 --> RRF 융합
       |
       v
  [리랭킹]  ...........  Cross-Encoder로 정밀 재순위
       |
       v
  상위 문서 선택
       |
       v
  [컨텍스트 조립 + LLM 응답 생성]
```

쿼리 변환은 질문의 품질을 높이고, 하이브리드 검색은 후보 범위를 넓히며, 리랭킹은 후보의 정밀도를 높입니다. 세 기법은 파이프라인의 서로 다른 지점에서 작동하므로 독립적으로 적용할 수도 있고, 순서대로 적용하여 효과를 중첩시킬 수도 있습니다. 모든 기법을 한꺼번에 적용할 필요는 없으며, 먼저 기본 RAG의 검색 품질을 측정한 뒤, 어떤 유형의 질문에서 실패가 발생하는지 분석하여 필요한 기법을 선택적으로 도입하는 것이 효율적입니다.

정리하면, RAG의 검색 품질을 높이는 핵심 전략은 세 가지입니다. 하이브리드 검색은 키워드 검색(BM25)과 시맨틱 검색(벡터)을 결합하여 각각의 한계를 보완하고, RRF 같은 점수 융합 기법으로 결과를 통합합니다. 리랭킹은 Cross-Encoder를 사용하여 초기 검색 결과의 순위를 정밀하게 재조정합니다. 쿼리 변환은 사용자의 질문을 검색에 적합한 형태로 바꾸거나, HyDE처럼 가상 답변을 통해 검색 벡터의 품질을 높입니다.

다음 단원인 4.2.2에서는 Self-RAG, Corrective RAG 등 검색과 생성 과정 자체를 LLM이 스스로 판단하고 조율하는 Advanced RAG 패턴을 다룹니다.

이 단원을 마치면 하이브리드 검색과 리랭킹을 적용하여 RAG 검색 품질을 향상시킬 수 있습니다.
