# 3.1.2 임베딩 모델 선택과 비교

3.1.1에서 텍스트 임베딩의 원리를 다루었습니다. 텍스트를 고정 길이의 숫자 벡터로 변환하고, 벡터 사이의 거리로 의미적 유사도를 측정하는 것이 임베딩의 핵심이었습니다. 또한 임베딩 기술이 단어 수준에서 문장 수준으로 발전해 왔다는 것도 확인했습니다. 이제 실무에서 임베딩을 사용하려면 구체적인 질문에 답해야 합니다. 어떤 모델을 선택할 것인가. 선택지가 다양한 만큼, 모델마다 특성이 다르고, 프로젝트의 요구사항에 따라 적합한 모델이 달라집니다. 이 단원에서는 주요 임베딩 모델의 특성을 비교하고, 목적에 맞는 모델을 선택하는 기준을 살펴봅니다.

임베딩 모델을 선택할 때 고려해야 하는 기본 속성부터 정리하겠습니다. 첫째, **임베딩 차원**(dimension)입니다. 3.1.1에서 설명한 것처럼, 차원이 높을수록 의미를 세밀하게 표현할 수 있지만 저장 공간과 검색 속도에 비용이 따릅니다. 둘째, **최대 입력 길이**입니다. 모델마다 한 번에 처리할 수 있는 토큰 수에 상한이 있습니다. 이 상한을 넘는 텍스트는 잘리거나 별도로 분할해야 합니다. 셋째, **다국어 지원 여부**입니다. 영어 중심으로 학습된 모델은 한국어 텍스트에서 성능이 떨어질 수 있습니다. 넷째, **API 방식인지 자체 호스팅 방식인지**입니다. API 방식은 인프라 관리 없이 호출만 하면 되지만 비용이 발생하고 데이터가 외부로 전송됩니다. 자체 호스팅은 데이터 통제권을 유지할 수 있지만 GPU 인프라가 필요합니다.

이 기본 속성을 기준으로 주요 모델을 하나씩 살펴보겠습니다.

**OpenAI text-embedding 모델**은 API 기반 임베딩 모델 중 가장 널리 사용되는 모델입니다. OpenAI는 여러 버전의 임베딩 모델을 제공해 왔는데, 현재 주력 모델은 text-embedding-3 시리즈입니다. 이 시리즈에는 text-embedding-3-small과 text-embedding-3-large 두 가지가 있습니다.

text-embedding-3-small은 1,536차원의 벡터를 생성합니다. 이전 세대 모델인 text-embedding-ada-002와 같은 차원이지만 성능이 개선되었고, 비용은 더 낮습니다. text-embedding-3-large는 3,072차원의 벡터를 생성하며, 더 세밀한 의미 구분이 필요한 경우에 사용합니다. 두 모델 모두 최대 8,191토큰의 입력을 처리할 수 있어서, 긴 문단이나 짧은 문서를 통째로 임베딩할 수 있습니다.

text-embedding-3 시리즈의 특징 중 하나는 **차원 축소**(dimension reduction)를 지원한다는 것입니다. API 호출 시 dimensions 파라미터를 지정하면, 모델이 전체 차원에서 앞쪽 일부만 잘라서 반환합니다. 예를 들어 text-embedding-3-large의 3,072차원을 1,024차원으로 줄여서 받을 수 있습니다. 차원을 줄이면 저장 공간과 검색 속도에서 이점이 생기고, 성능 저하는 비교적 완만합니다. 이것이 가능한 이유는 학습 과정에서 중요한 정보가 벡터의 앞쪽 차원에 집중되도록 설계되었기 때문입니다.

```python
from openai import OpenAI
client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input="LLMOps는 대규모 언어 모델의 운영을 체계화하는 분야입니다.",
    dimensions=512
)
embedding = response.data[0].embedding  # 512차원 벡터
```

위 코드에서 model은 사용할 임베딩 모델을 지정합니다. input은 임베딩할 텍스트입니다. dimensions는 원래 1,536차원인 출력을 512차원으로 줄여 달라는 요청입니다. 반환된 embedding은 512개의 실수로 이루어진 리스트입니다.

OpenAI 임베딩 모델의 장점은 별도의 인프라 없이 API 호출만으로 사용할 수 있다는 점, 영어뿐 아니라 한국어를 포함한 다국어를 지원한다는 점, 그리고 LangChain이나 LlamaIndex 같은 프레임워크와의 연동이 간편하다는 점입니다. 단점은 텍스트가 OpenAI 서버로 전송되므로 데이터 보안 정책이 엄격한 환경에서는 사용이 제한될 수 있다는 것과, 호출 횟수에 따라 비용이 누적된다는 것입니다.

다음으로 **Sentence-BERT(SBERT) 계열** 모델을 살펴보겠습니다. 3.1.1에서 문장 임베딩 모델의 발전 과정을 다루면서 Sentence-BERT를 언급했습니다. Sentence-BERT는 BERT 모델을 문장 유사도 비교에 최적화한 모델입니다.

BERT는 Transformer 기반의 사전학습 모델로, 원래 문장 쌍을 함께 입력하여 관계를 판단하도록 설계되었습니다. 그런데 두 문장의 유사도를 구하려면 두 문장을 함께 모델에 넣어야 합니다. 문서 1만 건 중에서 질문과 가장 유사한 것을 찾으려면 질문과 문서를 1만 번 쌍으로 묶어 모델을 통과시켜야 합니다. 이것은 현실적으로 너무 느립니다. Sentence-BERT는 이 문제를 해결합니다. 각 문장을 독립적으로 하나의 벡터로 변환하도록 BERT를 재학습시킨 것입니다. 문장마다 벡터를 미리 계산해 두면, 유사도 비교는 벡터 간 거리 계산만으로 가능해집니다.

Sentence-BERT 계열의 대표적인 모델은 all-MiniLM-L6-v2입니다. 이 모델은 384차원의 벡터를 생성하며, 모델 크기가 작아서 CPU에서도 실행할 수 있습니다. 빠른 속도가 필요하고 영어 텍스트를 주로 다루는 환경에서 널리 사용됩니다. 더 높은 품질이 필요한 경우에는 all-mpnet-base-v2(768차원)를 선택할 수 있습니다.

Sentence-BERT 모델은 오픈소스이므로 자체 서버에서 실행할 수 있습니다. sentence-transformers라는 Python 라이브러리를 사용하면 몇 줄의 코드로 로컬에서 임베딩을 생성할 수 있습니다.

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")
sentences = [
    "LLMOps는 대규모 언어 모델의 운영을 체계화하는 분야입니다.",
    "MLOps는 머신러닝 모델의 운영을 자동화합니다."
]
embeddings = model.encode(sentences)  # (2, 384) 형태의 배열
```

위 코드에서 SentenceTransformer는 모델을 로딩하고, encode 메서드는 문장 리스트를 받아 각 문장의 임베딩 벡터를 반환합니다. API 호출이 아니라 로컬에서 실행되므로, 데이터가 외부로 나가지 않습니다.

**Cohere Embed**는 Cohere에서 제공하는 API 기반 임베딩 모델입니다. Cohere Embed의 특징은 검색 시나리오에 최적화된 설계입니다. Cohere Embed를 호출할 때 input_type이라는 파라미터를 지정할 수 있습니다. 문서를 임베딩할 때는 input_type을 "search_document"로, 질문을 임베딩할 때는 "search_query"로 설정합니다. 모델이 입력의 용도를 알고 있으므로, 질문 벡터와 문서 벡터가 검색에 더 적합한 공간에 배치됩니다. 이것은 문서와 질문을 동일한 방식으로 임베딩하는 모델과의 차별점입니다.

Cohere Embed v3는 1,024차원의 벡터를 생성하며, 100개 이상의 언어를 지원합니다. 한국어도 포함되어 있어 다국어 환경에서 사용할 수 있습니다. 또한 Cohere는 리랭킹 모델(Rerank)도 함께 제공하므로, 임베딩 검색과 리랭킹을 하나의 API 제공자에서 일관되게 사용할 수 있습니다.

이제 **오픈소스 임베딩 모델**을 살펴보겠습니다. 최근 오픈소스 커뮤니티에서 성능이 뛰어난 임베딩 모델이 다수 공개되면서, API 의존 없이도 높은 품질의 임베딩을 생성할 수 있게 되었습니다. 대표적인 모델로 **BGE**, **E5**, **GTE**가 있습니다.

**BGE**(BAAI General Embedding)는 중국의 BAAI(Beijing Academy of Artificial Intelligence)에서 개발한 모델입니다. bge-large-en-v1.5는 1,024차원의 벡터를 생성하며, 공개 당시 여러 벤치마크에서 최상위 성능을 기록했습니다. BGE 모델의 특징은 검색 품질을 높이기 위해 질문 앞에 "Represent this sentence:"와 같은 지시문(instruction)을 붙이도록 설계되었다는 점입니다. 이 지시문이 모델에게 입력의 용도를 알려주는 역할을 합니다.

**E5**(EmbEddings from bidirEctional Encoder rEpresentations)는 Microsoft에서 개발한 모델입니다. E5 역시 지시문 기반으로 설계되었으며, "query:"와 "passage:" 접두사를 사용하여 질문과 문서를 구분합니다. e5-large-v2는 1,024차원의 벡터를 생성합니다. 이후 공개된 multilingual-e5-large는 다국어를 지원하며, 한국어에서도 높은 성능을 보여 줍니다.

**GTE**(General Text Embeddings)는 Alibaba에서 개발한 모델입니다. gte-large는 1,024차원의 벡터를 생성하며, 지시문 없이도 높은 성능을 보입니다. 최근에는 gte-Qwen2 시리즈처럼 대규모 언어 모델을 기반으로 한 임베딩 모델도 공개되고 있습니다.

이 세 모델은 모두 오픈소스로, sentence-transformers 라이브러리나 Hugging Face의 transformers 라이브러리를 사용하여 로컬에서 실행할 수 있습니다. API 비용이 들지 않고 데이터가 외부로 나가지 않는다는 장점이 있지만, 모델을 로딩하고 추론하기 위한 GPU 자원이 필요합니다.

**다국어 임베딩 모델**은 한국어처럼 영어가 아닌 언어의 텍스트를 다룰 때 특히 중요합니다. 영어 중심으로 학습된 모델은 한국어 텍스트의 의미적 유사도를 제대로 포착하지 못할 수 있습니다. "고양이"와 "강아지"가 의미적으로 가까워야 하는데, 영어 데이터로만 학습된 모델은 한국어 토큰의 의미 관계를 충분히 학습하지 못했을 가능성이 있기 때문입니다.

다국어 임베딩 모델은 여러 언어의 텍스트를 동일한 벡터 공간에 배치합니다. 이상적인 다국어 임베딩에서는 "고양이"의 벡터와 "cat"의 벡터가 가까운 위치에 놓입니다. 이 성질을 크로스 링구얼(cross-lingual) 정렬이라 합니다. 크로스 링구얼 정렬이 잘 된 모델을 사용하면, 영어로 된 질문으로 한국어 문서를 검색하거나, 한국어 질문으로 영어 문서를 검색하는 것이 가능해집니다.

대표적인 다국어 임베딩 모델로는 multilingual-e5-large, bge-m3, Cohere Embed v3, OpenAI text-embedding-3 시리즈가 있습니다. 한국어에 특화된 모델로는 KoSimCSE 같은 한국어 전용 모델도 있습니다. 다국어 모델은 단일 언어 모델에 비해 모델 크기가 클 수 있고, 특정 언어에서의 성능이 해당 언어 전용 모델보다 낮을 수 있습니다. 다루는 언어가 한국어 하나뿐이라면 한국어 특화 모델이 유리할 수 있고, 여러 언어를 동시에 다루어야 한다면 다국어 모델이 적합합니다.

이렇게 다양한 모델이 있을 때, 어떤 모델이 "좋은" 모델인지를 객관적으로 비교하려면 표준화된 벤치마크가 필요합니다. **MTEB**(Massive Text Embedding Benchmark)는 임베딩 모델의 성능을 다양한 과제에서 측정하는 벤치마크입니다. MTEB는 임베딩 모델이 수행해야 하는 과제를 여러 범주로 나누어 평가합니다. 검색(retrieval), 분류(classification), 군집화(clustering), 문장 유사도(semantic textual similarity), 리랭킹(reranking) 등이 포함됩니다. 각 범주에서 여러 데이터셋을 사용하여 모델의 성능을 점수로 산출합니다.

MTEB 리더보드는 Hugging Face에서 공개적으로 운영되고 있으며, 누구나 자신의 모델을 제출하여 순위를 확인할 수 있습니다. 리더보드를 볼 때 주의할 점은, 전체 평균 점수가 높다고 모든 상황에서 좋은 것은 아니라는 것입니다. 프로젝트에서 임베딩을 주로 검색(retrieval)에 사용한다면 검색 범주의 점수를 중점적으로 보아야 하고, 문장 유사도 비교에 사용한다면 STS(Semantic Textual Similarity) 범주의 점수가 더 중요합니다.

주요 모델의 특성을 표로 정리하면 다음과 같습니다.

```
모델                    차원     최대 토큰   다국어   방식        비고
------------------------------------------------------------------------
text-embedding-3-small  1,536    8,191       O       API         차원 축소 지원
text-embedding-3-large  3,072    8,191       O       API         차원 축소 지원
all-MiniLM-L6-v2        384      256         X       오픈소스    CPU 실행 가능
all-mpnet-base-v2       768      384         X       오픈소스    SBERT 계열
Cohere Embed v3         1,024    512         O       API         input_type 구분
bge-large-en-v1.5       1,024    512         X       오픈소스    지시문 기반
multilingual-e5-large   1,024    512         O       오픈소스    크로스 링구얼
gte-large               1,024    512         X       오픈소스    지시문 불필요
bge-m3                  1,024    8,192       O       오픈소스    긴 입력 지원
```

이 표에서 차원은 벡터의 길이, 최대 토큰은 한 번에 처리할 수 있는 입력 상한, 다국어는 영어 외 언어 지원 여부, 방식은 API 호출인지 자체 호스팅 가능한 오픈소스인지를 나타냅니다.

모델을 선택할 때 실무적인 기준을 정리하면 다음과 같습니다. 인프라 관리 부담을 줄이고 빠르게 시작하고 싶다면 API 기반 모델(OpenAI, Cohere)이 적합합니다. 데이터가 외부로 나가면 안 되거나 API 비용을 통제해야 한다면 오픈소스 모델을 자체 서버에서 운영합니다. 한국어 텍스트를 다루어야 한다면 다국어 지원 모델(multilingual-e5-large, bge-m3, Cohere Embed v3, OpenAI text-embedding-3)을 우선 검토합니다. 성능 비교는 MTEB 리더보드의 관련 범주 점수를 참고하되, 최종 판단은 자체 데이터로 실험하여 내리는 것이 가장 정확합니다.

정리하면, 임베딩 모델은 API 기반과 오픈소스로 나뉘며, 각 모델은 차원, 최대 입력 길이, 다국어 지원, 검색 최적화 방식에서 차이가 있습니다. OpenAI text-embedding-3는 높은 범용성과 차원 축소 기능이 강점이고, Sentence-BERT 계열은 가볍고 자체 호스팅이 쉬우며, Cohere Embed는 검색 시나리오에 특화되어 있습니다. BGE, E5, GTE 같은 오픈소스 모델은 API 없이도 경쟁력 있는 성능을 제공합니다. 다국어 환경에서는 크로스 링구얼 정렬이 잘 된 모델을 선택해야 하며, 모델 간 비교에는 MTEB 벤치마크를 참고하되 자체 데이터로의 검증이 가장 신뢰할 수 있습니다.

다음 단원인 3.1.3에서는 임베딩 파이프라인 구축을 다룹니다. 선택한 임베딩 모델을 사용하여 텍스트 데이터를 전처리하고, 배치로 임베딩을 생성하며, 캐싱과 갱신 정책을 갖춘 파이프라인을 설계하는 방법을 살펴봅니다.

이 단원을 마치면 주요 임베딩 모델의 특성을 비교하고 목적에 맞게 선택할 수 있습니다.
