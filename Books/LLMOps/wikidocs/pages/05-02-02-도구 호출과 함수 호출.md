# 5.2.2 도구 호출과 함수 호출

5.2.1에서 LLM 에이전트의 네 가지 구성요소를 살펴보았습니다. 그중 도구(Tool)는 에이전트가 텍스트 생성을 넘어 외부 세계와 상호작용하는 유일한 수단이라고 했습니다. 그런데 LLM은 본질적으로 텍스트를 입력받아 텍스트를 출력하는 모델입니다. 검색 엔진 API를 호출하거나 데이터베이스에 쿼리를 실행하는 것은 LLM이 직접 할 수 있는 일이 아닙니다. 그렇다면 LLM은 어떻게 도구를 '사용'하는 것일까요. 이 단원에서는 LLM이 외부 도구를 인식하고 호출하는 매커니즘인 Function Calling과 Tool Use를 다루고, 도구를 정의하고 구현하는 방법, 그리고 도구 간 연동을 표준화하는 MCP(Model Context Protocol)를 살펴봅니다.

LLM이 도구를 사용하는 원리를 이해하려면, 먼저 LLM에게 도구의 존재를 알려주는 방식부터 알아야 합니다. 가장 기초적인 방식은 프롬프트에 도구 설명을 포함시키는 것입니다. 예를 들어, "당신은 다음 도구를 사용할 수 있습니다. search(query): 웹 검색을 수행합니다. calculator(expression): 수식을 계산합니다."와 같이 시스템 프롬프트에 도구 목록을 적어 줍니다. LLM은 이 설명을 바탕으로, 답변 도중에 "search('서울 날씨')"와 같은 텍스트를 생성합니다. 외부 시스템이 이 텍스트를 파싱하여 실제 함수를 호출하고, 결과를 다시 LLM에 전달합니다. 이 방식은 동작하지만, 모델이 자유 형식 텍스트로 도구 호출을 표현하기 때문에 형식이 불안정합니다. 괄호를 빠뜨리거나, 인자 이름을 틀리거나, 존재하지 않는 도구를 호출하는 경우가 생깁니다.

이 문제를 해결하기 위해 모델 제공자들이 도입한 것이 **Function Calling**입니다. Function Calling은 모델이 도구 호출 의사를 구조화된 형식으로 출력하도록 학습된 기능입니다. 프롬프트에 자연어로 도구를 설명하는 대신, API 요청 시 도구의 스키마를 별도 필드로 전달합니다. 모델은 사용자의 질문에 대해 일반 텍스트 대신 JSON 형태의 함수 호출 정보를 출력합니다. 이 출력에는 호출할 함수의 이름과 인자가 구조화되어 들어 있으므로, 파싱 오류가 크게 줄어듭니다.

Function Calling의 동작 원리를 단계별로 설명하겠습니다.

첫 번째 단계에서, 개발자가 API 요청에 사용 가능한 함수(도구)의 목록과 각 함수의 스키마를 포함시킵니다. 스키마에는 함수 이름, 설명, 매개변수의 이름과 타입, 필수 여부 등이 담깁니다.

두 번째 단계에서, 사용자의 메시지와 함수 스키마가 모델에 전달됩니다. 모델은 사용자의 요청을 처리하기 위해 함수 호출이 필요한지 판단합니다.

세 번째 단계에서, 함수 호출이 필요하다고 판단하면, 모델은 일반 텍스트 대신 함수 호출 정보를 구조화된 형식으로 출력합니다. 이 출력에는 호출할 함수의 이름과 인자 값이 JSON으로 담겨 있습니다.

네 번째 단계에서, 애플리케이션이 이 JSON을 받아 실제 함수를 실행합니다. 실행은 모델이 아니라 애플리케이션 쪽에서 이루어집니다. 모델은 "이 함수를 이 인자로 호출하라"는 지시만 생성할 뿐, 직접 코드를 실행하지 않습니다.

다섯 번째 단계에서, 함수 실행 결과를 대화 이력에 추가하여 모델에 다시 보냅니다. 모델은 이 결과를 바탕으로 최종 답변을 생성하거나, 필요하면 다른 함수를 추가로 호출합니다.

```
Function Calling 동작 흐름

[개발자]                    [LLM API]                  [애플리케이션]
    |                          |                           |
    |-- 함수 스키마 정의 ----->|                           |
    |-- 사용자 메시지 -------->|                           |
    |                          |                           |
    |                     (모델 추론)                      |
    |                          |                           |
    |<-- 함수 호출 JSON -------|                           |
    |                          |                           |
    |                          |    함수 실행 요청 ------->|
    |                          |                           |
    |                          |<-- 실행 결과 -------------|
    |                          |                           |
    |-- 결과 포함 재요청 ----->|                           |
    |                          |                           |
    |<-- 최종 답변 ------------|                           |
```

이 흐름에서 주의할 점은, 모델이 함수를 직접 실행하지 않는다는 것입니다. 모델은 어떤 함수를 어떤 인자로 호출할지를 JSON으로 제안할 뿐이고, 실제 실행은 개발자의 애플리케이션 코드에서 이루어집니다. 이 설계 덕분에 보안과 제어가 가능합니다. 개발자는 모델이 제안한 함수 호출을 실행 전에 검증할 수 있고, 위험한 호출은 거부할 수 있습니다.

**Tool Use**는 Function Calling과 같은 목적을 달성하지만, 모델 제공자에 따라 구현 방식이 다릅니다. OpenAI 계열에서는 'Function Calling'이라는 이름을 사용하고, Anthropic의 Claude 모델에서는 'Tool Use'라는 이름을 사용합니다. 핵심 원리는 동일합니다. 모델에 도구 스키마를 전달하고, 모델이 구조화된 형식으로 도구 호출을 출력하며, 실행 결과를 다시 모델에 보내는 흐름입니다. 차이가 나는 부분은 API의 필드 이름, 응답 형식, 병렬 호출 지원 여부 등 세부 사항입니다.

도구의 스키마를 정의할 때 사용하는 표준이 **JSON Schema**입니다. JSON Schema는 JSON 데이터의 구조를 기술하는 표준 규격으로, 각 필드의 이름, 데이터 타입, 설명, 필수 여부 등을 선언적으로 정의합니다. Function Calling에서는 함수의 매개변수를 JSON Schema로 기술합니다.

구체적인 예를 들어 보겠습니다. 날씨 정보를 조회하는 도구를 정의한다고 합시다.

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "지정한 도시의 현재 날씨 정보를 조회합니다.",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "날씨를 조회할 도시 이름 (예: 'Seoul')"
        },
        "unit": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"],
          "description": "온도 단위"
        }
      },
      "required": ["city"]
    }
  }
}
```

이 스키마를 하나씩 풀어 보겠습니다. "name"은 함수의 식별자입니다. 모델이 이 함수를 호출할 때 이 이름을 사용합니다. "description"은 함수가 무엇을 하는지 자연어로 설명한 것입니다. 모델은 이 설명을 읽고 언제 이 함수를 호출해야 하는지 판단합니다. "parameters"는 함수가 받는 입력의 구조를 JSON Schema 형식으로 정의합니다. "properties" 안에 각 매개변수의 이름, 타입, 설명이 들어가고, "required" 배열에는 반드시 전달해야 하는 매개변수를 나열합니다. "city"는 문자열 타입이고 필수이며, "unit"은 문자열 타입이지만 "celsius"와 "fahrenheit" 중 하나만 선택할 수 있고 필수는 아닙니다.

이 스키마의 품질이 도구 호출의 정확도에 직접적인 영향을 미칩니다. description이 모호하면 모델이 엉뚱한 상황에서 도구를 호출하고, 매개변수 설명이 부정확하면 잘못된 인자를 전달합니다. 스키마를 정의할 때는 함수의 용도, 각 매개변수의 의미와 형식을 명확하게 적는 것이 중요합니다.

**커스텀 도구**를 구현하는 과정을 살펴보겠습니다. 도구 구현은 크게 세 단계로 이루어집니다. 첫째, 도구가 수행할 로직을 함수로 작성합니다. 둘째, 이 함수의 입출력을 JSON Schema로 기술합니다. 셋째, LLM API 요청 시 이 스키마를 함께 전달합니다.

```python
import json
import requests

# 1단계: 도구 로직 구현
def get_weather(city, unit="celsius"):
    """실제 날씨 API를 호출하는 함수"""
    response = requests.get(
        f"https://api.weather.example.com/current?city={city}&unit={unit}"
    )
    return response.json()

# 2단계: 도구 스키마 정의 (위의 JSON Schema)
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "지정한 도시의 현재 날씨 정보를 조회합니다.",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "도시 이름"},
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                    "description": "온도 단위"
                }
            },
            "required": ["city"]
        }
    }
}]

# 3단계: LLM API 호출 시 도구 포함
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "서울 날씨 알려 줘"}],
    tools=tools
)
```

위 코드에서 get_weather 함수가 실제 도구 로직입니다. tools 리스트에 이 함수의 스키마를 담아 API 요청에 포함합니다. 모델이 함수 호출을 결정하면 응답에 함수 이름("get_weather")과 인자({"city": "Seoul"})가 JSON으로 담겨 옵니다. 개발자는 이 JSON을 파싱하여 실제 get_weather 함수를 호출하고, 결과를 대화 이력에 추가한 뒤 모델에 다시 보냅니다.

```python
# 모델 응답에서 함수 호출 정보 추출
tool_call = response.choices[0].message.tool_calls[0]
function_name = tool_call.function.name
arguments = json.loads(tool_call.function.arguments)

# 실제 함수 실행
if function_name == "get_weather":
    result = get_weather(**arguments)

# 결과를 대화 이력에 추가하여 모델에 재전달
messages.append(response.choices[0].message)
messages.append({
    "role": "tool",
    "tool_call_id": tool_call.id,
    "content": json.dumps(result)
})

# 모델이 결과를 바탕으로 최종 답변 생성
final_response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=tools
)
```

이 코드에서 tool_call.function.name으로 모델이 호출하려는 함수 이름을 얻고, tool_call.function.arguments에서 인자를 JSON으로 파싱합니다. 실제 함수를 실행한 뒤, 결과를 "role": "tool" 메시지로 대화 이력에 추가합니다. tool_call_id는 어떤 도구 호출에 대한 결과인지를 대응시키는 식별자입니다. 이 대화 이력을 다시 모델에 보내면, 모델은 도구 실행 결과를 참고하여 최종 답변을 생성합니다.

이제 **MCP(Model Context Protocol)**를 살펴보겠습니다. 앞서 본 것처럼, 각 도구를 연동하려면 도구마다 스키마를 정의하고, 호출 코드를 작성하고, 결과를 처리하는 로직을 만들어야 합니다. 도구가 한두 개라면 관리할 만하지만, 실무에서는 수십 개의 도구를 다루는 경우가 흔합니다. 게다가 같은 도구를 여러 LLM 애플리케이션에서 재사용하려면, 매번 동일한 통합 코드를 반복 작성해야 합니다.

MCP는 이 문제를 해결하기 위해 도구와 LLM 애플리케이션 사이의 통신 방식을 표준화한 프로토콜입니다. MCP의 핵심 아이디어는 도구 제공자와 도구 소비자를 분리하는 것입니다. 도구 제공자는 MCP 서버로서 자신이 제공하는 도구의 스키마와 실행 로직을 등록합니다. 도구 소비자(LLM 애플리케이션)는 MCP 클라이언트로서 서버에 연결하여 사용 가능한 도구 목록을 조회하고, 도구를 호출하며, 결과를 받아옵니다.

```
MCP 아키텍처

+------------------+        +------------------+
|  LLM 애플리케이션 |        |  LLM 애플리케이션 |
|  (MCP 클라이언트) |        |  (MCP 클라이언트) |
+--------+---------+        +--------+---------+
         |                           |
         |     MCP 프로토콜          |
         v                           v
+--------+---------+        +--------+---------+
|   MCP 서버 A     |        |   MCP 서버 B     |
|  (날씨, 검색)    |        |  (DB, 파일)      |
+------------------+        +------------------+
```

MCP를 사용하면, 도구 제공자는 MCP 서버를 한 번 구현해 두면 여러 애플리케이션에서 재사용할 수 있습니다. 애플리케이션은 MCP 클라이언트 라이브러리만 사용하면 어떤 MCP 서버의 도구든 동일한 방식으로 연동할 수 있습니다. 이는 USB가 다양한 주변기기를 하나의 규격으로 연결하는 것과 유사합니다.

도구 호출 결과의 **검증과 후처리**도 중요한 고려 사항입니다. 모델이 도구 호출을 제안했다고 해서 무조건 실행해서는 안 됩니다. 검증이 필요한 이유는 크게 세 가지입니다.

첫째, 모델이 스키마에 맞지 않는 인자를 전달할 수 있습니다. 문자열이 필요한 곳에 숫자를 넣거나, 필수 인자를 누락하는 경우가 있습니다. 실행 전에 JSON Schema에 따라 인자를 검증해야 합니다.

둘째, 보안상 위험한 호출을 걸러내야 합니다. 데이터베이스 삭제나 결제 처리 같은 민감한 작업은 사용자의 명시적 승인 없이 실행해서는 안 됩니다.

셋째, 도구 실행 자체가 실패할 수 있습니다. 네트워크 오류, 타임아웃, 인증 만료 등 다양한 이유로 도구가 오류를 반환할 수 있습니다. 이 경우 오류 내용을 모델에 알려 주어 대안을 찾게 하거나, 재시도 로직을 적용할 수 있습니다.

후처리에서는 도구가 반환한 결과를 모델이 이해하기 적합한 형태로 변환합니다. 예를 들어, API가 매우 긴 JSON을 반환하면 필요한 필드만 추출하여 모델에 전달하는 것이 효율적입니다. 컨텍스트 윈도우의 토큰 제한을 고려하면, 불필요한 데이터를 그대로 넘기면 다른 중요한 정보가 밀려날 수 있습니다.

정리하면, Function Calling은 LLM이 도구 호출 의사를 구조화된 JSON으로 출력하는 기능이며, 실제 실행은 애플리케이션에서 이루어집니다. 도구 스키마는 JSON Schema로 정의하되, 함수 설명과 매개변수 설명을 명확히 작성해야 모델의 호출 정확도가 높아집니다. MCP는 도구와 애플리케이션 사이의 통신을 표준화하여, 도구의 재사용성과 연동 효율을 높이는 프로토콜입니다. 도구 호출 결과는 반드시 실행 전 검증과 실행 후 후처리를 거쳐야 안전하고 효과적으로 활용할 수 있습니다.

다음 단원인 5.2.3에서는 여러 에이전트가 협업하여 복잡한 작업을 수행하는 멀티 에이전트 오케스트레이션을 다룹니다. 감독자 패턴, 위계적 패턴, 토론 패턴 등 대표적인 협업 구조를 살펴봅니다.

이 단원을 마치면 Function Calling과 Tool Use의 동작 원리를 설명하고, JSON Schema로 도구를 정의하며, 커스텀 도구를 연동하는 코드를 작성할 수 있습니다.
