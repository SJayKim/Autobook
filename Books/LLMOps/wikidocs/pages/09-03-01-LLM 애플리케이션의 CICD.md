# 9.3.1 LLM 애플리케이션의 CI/CD

소프트웨어를 운영하다 보면, 코드를 변경하고 배포하는 과정이 반복됩니다. 버그를 수정하고, 새 기능을 추가하고, 성능을 개선합니다. 이 과정을 체계적이고 안전하게 수행하기 위한 방법이 CI/CD입니다. 9.2.3에서 토큰 비용 최적화 전략을 수립하는 방법을 다루었습니다. 프롬프트를 압축하거나, 모델을 교체하거나, 캐싱을 도입하려면, 그 변경 사항을 프로덕션에 안전하게 반영해야 합니다. 이 단원에서는 LLM 애플리케이션에 특화된 CI/CD 파이프라인을 설계하는 방법을 다룹니다.

먼저 CI/CD가 무엇인지부터 짚겠습니다. **CI**(Continuous Integration, 지속적 통합)란 개발자가 코드를 변경할 때마다, 변경 사항을 공유 저장소에 통합하고 자동으로 빌드와 테스트를 실행하는 관행입니다. 코드가 기존 코드와 충돌하지 않는지, 기존 기능이 깨지지 않았는지를 변경 시점에 바로 확인합니다. **CD**(Continuous Delivery 또는 Continuous Deployment)란 CI를 통과한 변경 사항을 자동으로 스테이징 환경이나 프로덕션 환경에 배포하는 것입니다. Continuous Delivery는 배포 준비까지 자동화하되 최종 배포는 수동 승인을 거치는 방식이고, Continuous Deployment는 모든 단계를 자동화하여 사람의 개입 없이 프로덕션까지 배포하는 방식입니다.

전통적인 소프트웨어에서 CI/CD는 주로 코드 변경을 대상으로 합니다. 코드를 커밋하면 단위 테스트가 실행되고, 통합 테스트가 실행되고, 테스트를 모두 통과하면 배포됩니다. 테스트의 결과는 결정적(deterministic)입니다. 같은 코드에 같은 입력을 넣으면 항상 같은 결과가 나오므로, "통과/실패"를 명확하게 판단할 수 있습니다. LLM 애플리케이션에서는 이 전제가 달라집니다. 이것이 **LLM CI/CD의 기존 소프트웨어와의 차이**입니다.

LLM 애플리케이션에서 변경의 대상은 코드만이 아닙니다. 프롬프트 텍스트의 변경, 모델의 변경(GPT-4에서 GPT-4o로, 또는 버전 업그레이드), 생성 파라미터(temperature, max_tokens)의 변경, RAG 파이프라인의 검색 설정 변경 등이 모두 배포가 필요한 변경입니다. 그리고 이러한 변경의 품질을 "통과/실패"로 판단하기 어렵습니다. 프롬프트를 바꿨을 때 응답이 "더 좋아졌는지"를 판단하는 것은, 함수의 반환값이 예상과 일치하는지 확인하는 것보다 훨씬 복잡합니다. 같은 프롬프트와 모델이라도 매번 다른 응답이 나올 수 있어, 한 번의 테스트로는 충분하지 않습니다.

```
전통 소프트웨어 CI/CD vs LLM CI/CD

항목           전통 소프트웨어            LLM 애플리케이션
-----------------------------------------------------------------
변경 대상      코드                       코드 + 프롬프트 + 모델
                                          + 파라미터 + 검색 설정
테스트 결과    결정적 (pass/fail)         비결정적 (점수 기반)
품질 판단      단위 테스트 통과 여부      평가 지표의 통계적 비교
배포 단위      코드 버전                  코드 + 프롬프트 + 모델 조합
롤백 기준      오류 발생                  품질 점수 하락
```

이 차이를 감안하여, LLM CI/CD 파이프라인은 두 가지 변경 축을 별도로 관리합니다.

**프롬프트 변경의 CI/CD**를 먼저 살펴보겠습니다. 프롬프트는 코드와 달리 자연어 텍스트이므로, 변경의 영향을 예측하기 어렵습니다. 한 단어를 바꾸는 것만으로도 모델의 응답 방향이 크게 달라질 수 있습니다. 프롬프트 변경을 CI/CD에 통합하는 방법은 다음과 같습니다.

첫째, 프롬프트를 코드 저장소(Git)에서 버전 관리합니다. 프롬프트 파일을 별도 디렉토리(예: 'prompts/')에 두고, 프롬프트를 변경할 때마다 커밋합니다. 이렇게 하면 프롬프트의 변경 이력을 추적할 수 있고, 문제가 생기면 이전 버전으로 되돌릴 수 있습니다. 9.1.3에서 다룬 Langfuse의 프롬프트 관리 기능과 병행하면, Git에서는 이력을 관리하고 Langfuse에서는 런타임 프롬프트를 관리하는 이중 체계를 갖출 수 있습니다.

둘째, 프롬프트 변경에 대한 자동 평가를 CI 파이프라인에 포함합니다. 프롬프트 파일이 변경된 커밋이 들어오면, 사전에 구성한 데이터셋에 대해 새 프롬프트로 모델을 실행하고, 평가 지표를 산출합니다. 이 과정이 9.1.2에서 다룬 LangSmith의 데이터셋과 평가 실행 기능, 또는 8.1.3에서 다룬 자동 평가 파이프라인과 연결됩니다.

셋째, 평가 결과를 기존 프롬프트의 결과와 비교하여, 품질이 유지되거나 향상되었을 때만 배포를 승인합니다. 이것이 **자동화된 평가 게이트**의 역할입니다.

**모델 변경의 CI/CD**도 구조는 유사하지만, 고려할 점이 더 있습니다. 모델을 변경하면 응답의 성격이 바뀔 수 있고, 비용과 지연 시간도 달라집니다. 모델 변경의 예시로는 동일 모델의 새 버전 업그레이드(GPT-4 에서 GPT-4o), 다른 제공자의 모델로 전환(OpenAI에서 Anthropic으로), 대형 모델에서 소형 모델로 축소(비용 절감 목적) 등이 있습니다.

모델 변경 시 CI 파이프라인에서는 품질 평가에 더해, 비용 비교와 지연 시간 비교를 함께 수행합니다. 새 모델이 기존 모델과 품질이 동등하면서 비용이 30% 저렴하다면 전환할 가치가 있지만, 품질이 10% 하락한다면 전환 여부를 신중히 결정해야 합니다. 이 판단 기준을 사전에 정의하여 자동화합니다.

**자동화된 평가 게이트**는 프롬프트든 모델이든, 어떤 변경이 프로덕션에 배포되기 전에 반드시 통과해야 하는 자동 검증 단계입니다. 평가 게이트의 구조를 단계별로 정리하면 다음과 같습니다.

```
자동화된 평가 게이트 흐름

[변경 커밋] (프롬프트 또는 모델 변경)
    |
    v
[1단계: 빌드 및 환경 구성]
    - 새 프롬프트/모델 설정으로 테스트 환경 구성
    |
    v
[2단계: 데이터셋 실행]
    - 사전 정의된 평가 데이터셋에 대해 응답 생성
    - 여러 번 반복 실행 (비결정적 출력 보정)
    |
    v
[3단계: 평가 지표 산출]
    - 정확성, 관련성, 환각률, 독성 등 평가
    - 비용 및 지연 시간 측정
    |
    v
[4단계: 기준 비교]
    - 현재 프로덕션 버전의 평가 결과와 비교
    - 사전 정의된 임계값 확인
    |
    v
[5단계: 판정]
    |
    +---> 통과: 모든 지표가 기준 충족 --> 배포 진행
    |
    +---> 실패: 하나 이상의 지표가 기준 미달 --> 배포 차단, 리포트 생성
```

평가 게이트에서 중요한 것은 비결정적 출력에 대한 대응입니다. 한 번의 실행으로는 신뢰할 수 없으므로, 동일 입력에 대해 여러 번(예: 5~10회) 실행하고 결과를 집계합니다. 평균 점수가 기존 버전의 평균 점수보다 통계적으로 유의하게 낮아지지 않았을 때 통과로 판정합니다. 이 판정 방식은 9.3.2에서 다룰 A/B 테스트의 통계적 유의성 판단과 동일한 원리입니다.

평가 게이트를 통과한 변경 사항을 프로덕션에 배포하는 방법도 전통 소프트웨어와 다를 수 있습니다. **카나리 배포와 점진적 롤아웃**은 새 버전을 한꺼번에 전체 트래픽에 적용하지 않고, 소량의 트래픽부터 시작하여 점진적으로 확대하는 전략입니다.

**카나리 배포**(canary deployment)의 개념을 먼저 설명합니다. 탄광에서 유독 가스를 감지하기 위해 카나리아(canary, 카나리아새)를 먼저 보낸 데서 유래한 용어입니다. 새 버전을 소수의 트래픽(예: 5%)에만 적용하여, 실제 프로덕션 환경에서 문제가 없는지 확인합니다. 문제가 발견되면 5%만 영향을 받으므로 피해를 최소화할 수 있습니다. 문제가 없으면 비율을 10%, 25%, 50%, 100%로 단계적으로 올립니다.

```
카나리 배포 과정

시간    트래픽 분배
----    -----------
t1      [기존 버전: 95%] [새 버전: 5%]   <-- 카나리 시작
        --> 오류율, 품질 점수, 지연 시간 모니터링

t2      [기존 버전: 75%] [새 버전: 25%]  <-- 지표 정상, 확대
        --> 계속 모니터링

t3      [기존 버전: 50%] [새 버전: 50%]  <-- 추가 확대
        --> 계속 모니터링

t4      [기존 버전: 0%]  [새 버전: 100%] <-- 전체 전환 완료
```

카나리 배포에서 각 단계를 진행할지 결정하는 기준은 옵저버빌리티 도구에서 수집하는 지표입니다. 새 버전이 적용된 트래픽의 오류율, 품질 평가 점수, 지연 시간, 비용을 기존 버전과 비교합니다. 이 비교를 자동화하면, 지표가 기준을 충족할 때 자동으로 다음 단계로 진행하는 자동 카나리 배포가 됩니다.

아무리 신중하게 배포하더라도 문제가 발생할 수 있습니다. 이때 필요한 것이 **롤백 전략**입니다. 롤백이란 새 버전을 철회하고 이전 버전으로 되돌리는 것입니다.

LLM 애플리케이션의 롤백에서 주의할 점은, 롤백 대상이 코드만이 아니라 프롬프트와 모델 설정도 포함한다는 것입니다. 코드는 Git에서 이전 커밋으로 되돌릴 수 있습니다. 프롬프트는 프롬프트 관리 시스템(예: Langfuse)에서 이전 버전을 활성화하거나, Git에서 이전 프롬프트 파일로 되돌립니다. 모델 설정은 설정 파일이나 환경 변수를 이전 값으로 복원합니다.

롤백을 결정하는 기준도 사전에 정해 두어야 합니다. "오류율이 배포 전 대비 2배 이상 증가", "P95 레이턴시가 SLO를 초과", "자동 평가 점수가 10% 이상 하락" 같은 기준을 설정하고, 이 기준에 도달하면 자동으로 롤백을 트리거하거나, 최소한 알림을 보내어 신속한 수동 롤백을 가능하게 합니다.

```
롤백 판단 기준 예시

지표                       임계값              행동
-----------------------------------------------------------------
오류율                     배포 전 대비 2x     자동 롤백
P95 레이턴시               SLO 초과            자동 롤백
자동 평가 점수             10% 이상 하락       알림 + 수동 판단
비용                       예산의 120% 초과    알림 + 수동 판단
```

정리하면, LLM 애플리케이션의 CI/CD는 코드뿐 아니라 프롬프트와 모델의 변경까지 대상으로 합니다. 전통 소프트웨어와 달리 테스트 결과가 비결정적이므로, 데이터셋 기반 평가를 여러 번 실행하여 통계적으로 판단하는 평가 게이트가 핵심입니다. 프롬프트와 모델 변경을 각각 Git과 프롬프트 관리 시스템으로 버전 관리하고, 카나리 배포로 소량 트래픽부터 점진적으로 적용하며, 문제 발생 시 사전 정의된 기준에 따라 롤백합니다.

다음 단원인 9.3.2에서는 프롬프트와 모델의 A/B 테스트를 다룹니다. 평가 게이트에서 "기준 대비 나쁘지 않다"를 판단했다면, A/B 테스트에서는 "실제 환경에서 어느 쪽이 더 좋은지"를 통계적으로 비교하는 방법을 살펴봅니다.

이 단원을 마치면 LLM 애플리케이션에 특화된 CI/CD 파이프라인을 설계할 수 있습니다.
