# 1.1.2 MLOps에서 LLMOps로의 전환

1.1.1에서 LLMOps가 무엇이고 왜 필요한지를 살펴보았습니다. LLMOps는 대규모 언어 모델을 운영 환경에서 안정적으로 제공하고 관리하기 위한 일련의 실천 방법입니다. 그런데 LLMOps가 등장하기 전에도 머신러닝 모델을 운영하는 체계는 이미 존재했습니다. 바로 **MLOps**입니다. 이 단원에서는 MLOps가 어떤 문제를 해결하기 위해 만들어졌는지부터 시작하여, 대규모 언어 모델이 기존 머신러닝과 어떻게 다른지, 그 차이가 운영 체계를 왜 바꾸어야 하는지를 순서대로 설명합니다.

소프트웨어를 개발한 뒤 서버에 올려 사용자에게 제공하는 과정을 **배포**라고 합니다. 일반 소프트웨어에서는 코드를 수정하면 테스트를 거쳐 새 버전을 서버에 배포하는 흐름이 비교적 단순합니다. 그런데 머신러닝 시스템에서는 코드뿐 아니라 데이터와 모델이라는 두 가지 변수가 추가됩니다. 학습에 사용하는 데이터가 바뀌면 모델의 예측 결과도 달라지고, 모델 구조나 하이퍼파라미터를 조정해도 결과가 달라집니다. 즉 머신러닝 시스템에서는 코드, 데이터, 모델이라는 세 축을 동시에 관리해야 합니다. 이 복잡성을 체계적으로 다루기 위해 등장한 것이 MLOps입니다.

**MLOps**는 Machine Learning Operations의 줄임말로, 머신러닝 모델의 개발부터 배포, 운영, 모니터링까지의 전체 생명주기를 자동화하고 표준화하는 실천 방법을 뜻합니다. 소프트웨어 엔지니어링에서 개발과 운영을 통합하는 DevOps라는 문화가 있는데, MLOps는 이 DevOps의 원칙을 머신러닝에 맞게 확장한 것입니다. DevOps가 코드의 빌드, 테스트, 배포를 자동화하듯이, MLOps는 데이터 준비, 모델 학습, 모델 배포, 성능 모니터링을 자동화합니다.

MLOps의 핵심 구성요소를 하나씩 살펴보겠습니다. 첫째는 **데이터 관리**입니다. 모델이 학습할 데이터를 수집하고, 결측값이나 이상치를 정리하고, 학습에 적합한 형태로 변환하는 과정을 포함합니다. 데이터의 버전을 기록해 두어야 나중에 어떤 데이터로 학습한 모델인지 추적할 수 있습니다.

둘째는 **모델 학습**입니다. 준비된 데이터로 알고리즘을 훈련시키고, 다양한 하이퍼파라미터 조합을 실험하여 최적의 모델을 찾습니다. 셋째는 **모델 배포**입니다. 학습이 완료된 모델을 실제 서비스 환경에 올려 사용자의 요청에 응답할 수 있도록 만드는 과정입니다.

넷째는 **모니터링**입니다. 배포된 모델이 시간이 지나면서 성능이 떨어지는지를 지속적으로 관찰합니다. 현실 세계의 데이터 분포는 끊임없이 변합니다. 학습 당시와 다른 패턴의 데이터가 들어오면 모델의 예측 정확도가 하락합니다. 이 현상을 데이터 드리프트라고 하며, 이를 감지하여 모델을 재학습시키는 것이 모니터링의 핵심 역할입니다.

이 네 가지 구성요소가 순서대로 연결되면 하나의 흐름이 만들어집니다. 이 흐름을 **MLOps 파이프라인**이라고 합니다. 파이프라인은 원래 공장에서 원자재가 컨베이어 벨트를 따라 한 공정씩 거치며 완성품이 되는 것에 비유한 용어입니다. MLOps 파이프라인에서는 데이터가 원자재이고, 완성품이 배포된 모델입니다.

```
데이터 수집 → 데이터 전처리 → 모델 학습 → 모델 평가 → 모델 배포 → 모니터링
     ^                                                              |
     |                                                              |
     +---------------------- 재학습 트리거 -------------------------+
```

위 다이어그램에서 왼쪽부터 오른쪽으로 진행하는 것이 기본 흐름입니다. 데이터를 수집하고, 전처리하고, 모델을 학습시키고, 평가하여 기준을 통과하면 배포합니다. 배포 이후에는 모니터링이 지속적으로 동작하며, 성능 저하가 감지되면 다시 데이터 수집 단계로 돌아가 재학습을 시작합니다. 이 순환 구조가 MLOps 파이프라인의 핵심입니다.

MLOps가 다루는 머신러닝 모델은 대부분 특정 목적에 맞추어 학습됩니다. 예를 들어 이메일이 스팸인지 아닌지를 분류하는 모델, 사용자가 다음에 클릭할 상품을 예측하는 추천 모델, 이미지에서 결함을 찾아내는 검수 모델 등이 있습니다. 이런 모델들은 보통 수만에서 수백만 개의 파라미터를 가지며, 학습 데이터는 해당 업무에 특화된 정제된 데이터셋입니다. 하나의 모델이 하나의 과업을 수행하는 구조입니다.

대규모 언어 모델, 즉 **LLM**은 이 구조와 근본적으로 다릅니다. 그 차이를 세 가지 측면에서 살펴보겠습니다.

첫 번째 차이는 **모델의 크기**입니다. 기존 머신러닝 모델이 수만에서 수백만 개의 파라미터를 가진다면, LLM은 수십억에서 수천억 개의 파라미터를 가집니다. GPT-3의 파라미터 수는 1,750억 개이며, 이보다 큰 모델도 계속 등장하고 있습니다. 이 규모의 차이는 단순히 숫자가 큰 것에 그치지 않습니다. 모델을 메모리에 올리는 것 자체가 도전이 됩니다. 1,750억 개의 파라미터를 32비트 부동소수점으로 저장하면 약 700기가바이트의 메모리가 필요합니다. 하나의 GPU가 가진 메모리는 보통 24에서 80기가바이트 수준이므로, 모델 하나를 올리기 위해 여러 대의 GPU를 묶어야 합니다. 학습은 더 많은 자원을 요구합니다. 이런 규모의 모델을 처음부터 학습시키려면 수천 대의 GPU를 수 주에서 수 개월간 가동해야 하며, 비용은 수백만 달러에 달합니다.

두 번째 차이는 **입출력 방식**입니다. 기존 머신러닝 모델은 정해진 형태의 입력을 받아 정해진 형태의 출력을 내놓습니다. 스팸 분류 모델은 이메일 텍스트를 받아 '스팸' 또는 '정상'이라는 레이블을 반환합니다. 반면 LLM은 자연어로 된 지시문, 즉 **프롬프트**를 받아 자연어 텍스트를 생성합니다. 같은 모델에 "이 문장을 영어로 번역해 주세요"라고 쓰면 번역을 하고, "이 코드의 버그를 찾아 주세요"라고 쓰면 코드 리뷰를 합니다. 모델을 재학습시키지 않고도 프롬프트를 바꾸는 것만으로 다양한 과업을 수행할 수 있습니다. 이것을 **프롬프트 기반 동작**이라고 합니다.

세 번째 차이는 **출력의 성격**입니다. 기존 머신러닝 모델의 출력은 대부분 분류 레이블이나 수치 예측값처럼 닫힌 범위의 값입니다. "이 이미지는 고양이일 확률 92%"처럼 정량적으로 평가하기 쉽습니다. 반면 LLM은 문장, 문단, 코드, 표 등 자유로운 형태의 텍스트를 생성합니다. 이런 출력을 **생성형 출력**이라고 합니다. 생성형 출력은 정답이 하나로 정해져 있지 않아서, 출력의 품질을 자동으로 평가하기 어렵습니다. "이 요약이 원문의 핵심을 잘 담고 있는가?"와 같은 질문에는 사람의 판단이 개입해야 하는 경우가 많습니다.

이 세 가지 차이, 즉 모델 크기, 프롬프트 기반 동작, 생성형 출력은 운영 체계에 연쇄적인 변화를 요구합니다. MLOps의 파이프라인을 떠올려 보겠습니다. MLOps에서는 데이터를 준비하고, 모델을 학습시키고, 배포하고, 모니터링하는 흐름이 핵심이었습니다. LLM 환경에서는 이 각 단계의 성격이 크게 달라집니다.

데이터 단계를 먼저 보겠습니다. 기존 MLOps에서는 특정 과업에 맞는 학습 데이터를 직접 준비하는 것이 출발점이었습니다. LLM 환경에서는 대부분의 경우 모델을 처음부터 학습시키지 않습니다. 이미 방대한 텍스트 데이터로 사전학습된 모델을 가져다 쓰기 때문입니다. 대신 프롬프트를 어떻게 구성할지, 모델에 추가 맥락으로 어떤 문서를 제공할지가 중요해집니다. 학습 데이터 관리 대신 프롬프트 관리와 맥락 문서 관리가 핵심 과제가 되는 것입니다.

학습 단계도 달라집니다. 기존 MLOps에서는 모델을 자주 재학습시키는 것이 일상이었습니다. 새로운 데이터가 들어오면 모델을 다시 학습시켜 성능을 유지했습니다. LLM에서는 전체 재학습의 비용이 너무 크기 때문에, 모델 전체를 다시 학습시키는 대신 파인튜닝이라는 방법으로 일부 파라미터만 조정하거나, 아예 학습 없이 프롬프트만 개선하여 성능을 높이는 접근을 택합니다. 학습 파이프라인의 비중이 줄어들고, 프롬프트 엔지니어링의 비중이 커집니다.

배포 단계에서는 인프라 요구 사항이 달라집니다. 기존 모델은 하나의 서버에서 충분히 동작하는 경우가 많았지만, LLM은 여러 대의 GPU를 필요로 합니다. 응답 시간도 다릅니다. 기존 분류 모델은 밀리초 단위로 응답하지만, LLM은 토큰 하나씩 순차적으로 생성하기 때문에 응답 완료까지 수 초가 걸릴 수 있습니다. API 호출 비용도 토큰 수에 비례하여 발생하므로, 비용 관리가 새로운 운영 과제가 됩니다.

모니터링 단계의 변화도 큽니다. 기존 MLOps에서는 정확도, 정밀도, 재현율 같은 정량 지표로 모델 성능을 추적했습니다. LLM의 생성형 출력에는 이런 단일 지표를 적용하기 어렵습니다. 출력이 사실에 부합하는지, 유해한 내용을 포함하지 않는지, 지시를 충실히 따랐는지 등 여러 차원을 동시에 살펴야 합니다. 모델이 사실이 아닌 내용을 그럴듯하게 생성하는 현상, 이른바 환각(hallucination)을 감지하는 것도 LLM 모니터링의 고유한 과제입니다.

이러한 변화를 종합하면, MLOps의 기본 골격인 '데이터 수집, 학습, 배포, 모니터링'은 LLMOps에서도 유지되지만, 각 단계의 내용과 비중이 크게 재편됩니다. 아래 표는 두 체계의 주요 차이를 정리한 것입니다.

```
+-----------------+----------------------------+----------------------------+
|     항목        |         MLOps              |         LLMOps             |
+-----------------+----------------------------+----------------------------+
| 모델 크기       | 수만~수백만 파라미터       | 수십억~수천억 파라미터     |
| 학습 방식       | 직접 학습이 기본           | 사전학습 모델 활용이 기본  |
| 입력            | 정형 데이터, 특성 벡터     | 자연어 프롬프트            |
| 출력            | 레이블, 수치               | 자유 형태 텍스트           |
| 핵심 관리 대상  | 학습 데이터, 모델 버전     | 프롬프트, 맥락 문서, 비용  |
| 성능 지표       | 정확도, F1 등 정량 지표    | 다차원 품질 평가           |
| 배포 인프라     | 단일 서버 가능             | 다중 GPU 또는 API 호출     |
| 재학습 주기     | 빈번한 재학습              | 프롬프트 개선 우선         |
+-----------------+----------------------------+----------------------------+
```

이 표에서 드러나듯, LLMOps는 MLOps를 완전히 대체하는 것이 아니라 MLOps의 원칙 위에서 LLM 고유의 특성에 맞게 확장하고 재구성한 체계입니다. MLOps에서 확립된 버전 관리, 자동화, 재현 가능성이라는 원칙은 LLMOps에서도 그대로 유효합니다. 다만 그 원칙이 적용되는 대상이 학습 데이터에서 프롬프트로, 모델 재학습에서 프롬프트 최적화로, 정량 지표에서 다차원 품질 평가로 이동하는 것입니다.

MLOps와 LLMOps 외에도 운영 체계를 가리키는 용어가 몇 가지 더 있습니다. 혼동을 피하기 위해 짚어 두겠습니다.

**AIOps**는 Artificial Intelligence for IT Operations의 줄임말입니다. 이름에 AI가 들어 있지만, AI 모델을 운영하는 체계가 아니라 IT 인프라 운영에 AI 기술을 활용하는 것을 뜻합니다. 서버 수천 대가 동시에 동작하는 환경에서는 로그와 메트릭 데이터가 사람이 직접 살펴볼 수 없을 만큼 쏟아집니다. AIOps는 이 방대한 운영 데이터를 머신러닝으로 분석하여, 장애를 자동으로 감지하고 근본 원인을 추정하며 알림을 보내는 데 초점을 맞춥니다. 즉 AIOps는 'AI를 이용한 운영'이고, MLOps와 LLMOps는 'AI를 운영하는 방법'입니다. 방향이 반대입니다.

**DataOps**는 데이터의 수집, 정제, 변환, 전달 파이프라인을 자동화하고 품질을 관리하는 실천 방법입니다. 데이터 엔지니어링에 DevOps 원칙을 적용한 것으로, 데이터 파이프라인의 테스트, 배포, 모니터링을 체계화합니다. DataOps는 MLOps와 LLMOps 모두의 하부 구조에 해당합니다. MLOps에서는 학습 데이터 파이프라인이 DataOps의 대상이고, LLMOps에서는 프롬프트에 제공할 맥락 문서의 수집과 정제, 벡터 데이터베이스로의 적재 파이프라인이 DataOps의 대상이 됩니다.

```
+----------------------------------------------------------+
|                    IT 운영 환경                           |
|                                                          |
|  +------------+    +------------+    +----------------+  |
|  |  DataOps   |    |   MLOps    |    |    LLMOps      |  |
|  | 데이터     |--->| ML 모델    |--->| LLM 모델       |  |
|  | 파이프라인 |    | 생명주기   |    | 생명주기       |  |
|  +------------+    +-----+------+    +-------+--------+  |
|        |                 |                   |           |
|        +--------+--------+-------------------+           |
|                 |                                        |
|           +-----v------+                                 |
|           |   AIOps    |                                 |
|           | IT 운영에  |                                 |
|           | AI를 활용  |                                 |
|           +------------+                                 |
+----------------------------------------------------------+
```

위 다이어그램은 네 가지 Ops의 관계를 보여 줍니다. DataOps는 데이터를 준비하는 기반 계층이며, MLOps와 LLMOps는 각각의 모델 유형에 맞는 운영 체계입니다. AIOps는 이들 모두가 동작하는 IT 인프라 자체를 AI로 관리하는 영역입니다. LLMOps를 실무에 적용할 때 이 네 가지 Ops가 완전히 분리되어 있는 것은 아닙니다. LLM 서비스를 운영하면서 DataOps로 문서 파이프라인을 관리하고, AIOps로 서버 상태를 모니터링하는 식으로 여러 Ops가 겹쳐서 동작합니다.

정리하면, MLOps는 머신러닝 모델의 전체 생명주기를 자동화하고 표준화하는 체계이며, LLMOps는 이 체계를 대규모 언어 모델의 고유한 특성에 맞게 확장한 것입니다. 모델 크기, 프롬프트 기반 동작, 생성형 출력이라는 세 가지 차이가 데이터 관리, 학습, 배포, 모니터링의 각 단계를 근본적으로 변화시키며, 이것이 MLOps에서 LLMOps로의 패러다임 전환을 이끌고 있습니다. AIOps와 DataOps는 LLMOps와 계층이 다른 별개의 Ops 체계이지만, 실무에서는 서로 맞물려 동작합니다.

다음 단원인 1.1.3에서는 Foundation Model Operations를 다룹니다. 사전학습된 대규모 모델, 즉 Foundation Model을 선택하고 운영할 때 고려해야 하는 사항을 살펴봅니다.

이 단원을 마치면 MLOps의 핵심 개념을 설명하고, LLMOps와의 차이를 구분할 수 있습니다.
