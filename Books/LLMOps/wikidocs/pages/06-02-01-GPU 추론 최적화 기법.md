# 6.2.1 GPU 추론 최적화 기법

6.1절에서 vLLM, TGI, TensorRT-LLM 같은 서빙 프레임워크를 살펴보았습니다. 이 프레임워크들은 모두 GPU 위에서 모델을 실행하며, 내부적으로 다양한 최적화 기법을 적용하여 추론 속도를 높입니다. 그런데 이 최적화 기법들이 정확히 어떤 문제를 해결하는 것인지, 왜 특정 기법이 효과적인지를 이해하려면 GPU가 연산을 처리하는 방식부터 알아야 합니다. 이 단원에서는 GPU의 메모리 구조와 대역폭 개념에서 출발하여, LLM 추론의 병목 지점을 분석하고, 이를 해결하는 대표적인 최적화 기법들을 하나씩 살펴봅니다.

GPU가 LLM 추론에 사용되는 이유는 병렬 연산 능력 때문입니다. CPU는 소수의 강력한 코어로 복잡한 순차 작업을 빠르게 처리하는 데 적합합니다. 반면 GPU는 수천 개의 작은 코어를 갖추고 있어, 같은 연산을 대량의 데이터에 동시에 적용하는 작업에 유리합니다. LLM의 핵심 연산인 행렬 곱셈은 정확히 이 패턴에 해당합니다. 수천 차원의 벡터와 행렬을 곱하는 작업을 GPU의 코어들이 나누어 처리하면, CPU로 수행할 때보다 수십에서 수백 배 빠르게 끝낼 수 있습니다.

하지만 GPU의 코어가 아무리 많아도, 코어에 데이터를 공급하는 속도가 느리면 코어들이 유휴 상태로 대기하게 됩니다. GPU의 성능을 이해하려면 **연산 처리량**과 **메모리 대역폭**이라는 두 가지 지표를 함께 봐야 합니다. 연산 처리량은 GPU가 초당 수행할 수 있는 부동소수점 연산의 수를 뜻하며, 단위는 TFLOPS(초당 조 회 연산)입니다. 메모리 대역폭은 GPU 메모리에서 연산 코어로 데이터를 전송할 수 있는 초당 데이터량을 뜻하며, 단위는 GB/s(초당 기가바이트)입니다.

GPU의 메모리 구조를 좀 더 구체적으로 살펴보겠습니다. GPU에는 크게 두 종류의 메모리가 있습니다. 하나는 **HBM(High Bandwidth Memory)**이라 불리는 글로벌 메모리이고, 다른 하나는 칩 내부의 **SRAM(Static RAM)**입니다. HBM은 용량이 크지만 접근 속도가 상대적으로 느립니다. NVIDIA A100 GPU를 기준으로 HBM 용량은 80GB이고, 대역폭은 약 2TB/s입니다. 반면 SRAM은 용량이 매우 작지만(A100 기준 약 20MB) 접근 속도가 HBM보다 훨씬 빠릅니다. 대역폭이 약 19TB/s에 달해 HBM의 약 10배입니다.

```
GPU 메모리 계층 구조

+--------------------------------------------------+
|                GPU 칩                              |
|                                                    |
|   +----------+  +----------+  +----------+        |
|   |  연산    |  |  연산    |  |  연산    |  ...   |
|   |  코어    |  |  코어    |  |  코어    |        |
|   +----+-----+  +----+-----+  +----+-----+        |
|        |             |             |                |
|   +----v-------------v-------------v----+          |
|   |         SRAM (약 20MB)              |          |
|   |         대역폭: ~19 TB/s            |          |
|   +----+--------------------------------+          |
|        |                                           |
+--------+-------------------------------------------+
         |  대역폭: ~2 TB/s
+--------v-------------------------------------------+
|         HBM (약 80GB)                              |
|         모델 가중치, KV 캐시, 활성화 값 저장        |
+----------------------------------------------------+
```

이 구조가 LLM 추론과 어떻게 연결되는지 설명하겠습니다. 모델의 가중치, 1.2.3에서 다룬 KV 캐시, 그리고 중간 계산 결과인 활성화 값은 모두 HBM에 저장됩니다. 연산을 수행하려면 이 데이터를 HBM에서 SRAM으로 옮기고, SRAM에서 연산 코어가 처리한 뒤, 결과를 다시 HBM에 기록해야 합니다. 이 과정에서 HBM과 SRAM 사이의 데이터 이동이 병목이 될 수 있습니다.

이제 LLM 추론에서 발생하는 두 가지 유형의 병목을 구분하겠습니다. 어떤 연산이 느린 이유가 연산량 자체가 많아서인지, 아니면 데이터를 메모리에서 읽어오는 시간이 길어서인지에 따라 최적화 방향이 완전히 달라집니다.

**연산 바운드(Compute-bound)**는 GPU의 연산 코어가 계산을 처리하는 시간이 병목인 상황입니다. 데이터는 충분히 빨리 공급되지만, 수행해야 할 곱셈과 덧셈의 양이 워낙 많아서 코어가 쉬지 않고 일해도 시간이 걸리는 경우입니다. 1.2.3에서 다룬 Prefill 단계가 대표적입니다. Prefill 단계에서는 입력 프롬프트의 모든 토큰을 한꺼번에 처리합니다. 토큰이 수백에서 수천 개에 이르면, Attention 행렬의 크기가 토큰 수의 제곱에 비례하여 커지고, 이를 계산하는 데 필요한 부동소수점 연산이 급증합니다. 이때는 GPU의 연산 처리량(TFLOPS)이 성능을 결정합니다.

**메모리 바운드(Memory-bound)**는 연산 코어가 데이터를 기다리며 유휴 상태에 놓이는 상황입니다. 수행할 연산량은 적은데, 필요한 데이터를 HBM에서 읽어오는 데 시간이 걸려 코어가 놀고 있는 경우입니다. 1.2.3에서 다룬 Decode 단계가 대표적입니다. Decode 단계에서는 한 번에 토큰 하나만 처리합니다. 새 토큰의 Query 벡터 하나와 KV 캐시에 저장된 이전 토큰들의 Key, Value를 Attention 계산해야 합니다. Query가 하나뿐이므로 실제 연산량은 적지만, 모델의 전체 가중치와 KV 캐시를 HBM에서 읽어와야 합니다. 70억 파라미터 모델의 가중치가 FP16 형식으로 약 14GB라면, 토큰 하나를 생성할 때마다 14GB를 HBM에서 읽어야 합니다. HBM 대역폭이 2TB/s라면 이 읽기 작업만 약 7밀리초가 걸리고, 실제 행렬 곱셈은 그보다 훨씬 빨리 끝납니다. 코어는 대부분의 시간을 데이터 도착을 기다리며 보냅니다.

```
연산 바운드 vs 메모리 바운드

Prefill 단계 (연산 바운드)
  많은 토큰을 동시 처리 --> 대규모 행렬 곱셈 --> GPU 코어가 쉬지 않고 연산
  병목: 연산 처리량 (TFLOPS)

Decode 단계 (메모리 바운드)
  토큰 하나만 처리 --> 적은 연산량, 하지만 전체 가중치+KV 캐시를 읽어야 함
  병목: 메모리 대역폭 (GB/s)
```

이 구분이 중요한 이유는 최적화 방향을 결정하기 때문입니다. 연산 바운드 상황에서는 연산 자체를 줄이거나 더 효율적인 알고리즘을 사용해야 합니다. 메모리 바운드 상황에서는 메모리 접근 횟수를 줄이거나, 데이터를 더 작은 형식으로 압축하거나, 여러 요청을 묶어서 한 번의 메모리 읽기로 더 많은 연산을 수행해야 합니다.

이제 이 병목을 해결하는 대표적인 기법들을 살펴보겠습니다. 먼저 Prefill 단계의 연산 바운드 문제를 해결하는 **Flash Attention**부터 시작합니다.

1.2.1에서 다룬 Self-Attention의 표준 구현은 다음과 같은 순서로 동작합니다. Query(Q)와 Key(K) 행렬을 곱하여 Attention 점수 행렬(S)을 만듭니다. S에 소프트맥스를 적용하여 Attention 가중치 행렬(P)을 만듭니다. P와 Value(V) 행렬을 곱하여 최종 출력(O)을 만듭니다. 이 과정에서 S와 P는 시퀀스 길이의 제곱에 비례하는 크기의 행렬입니다. 시퀀스 길이가 4,096이면 S와 P 각각 4,096 x 4,096 크기의 행렬이 되고, 이 행렬들을 HBM에 기록했다가 다시 읽어야 합니다.

문제는 이 중간 행렬의 HBM 입출력입니다. S를 HBM에 기록하고, 소프트맥스를 위해 다시 읽고, P를 다시 HBM에 기록하고, V와의 곱을 위해 다시 읽는 과정이 반복됩니다. 시퀀스가 길어질수록 이 중간 행렬의 크기가 급증하여, HBM 용량을 초과하거나 읽기/쓰기 시간이 연산 시간보다 길어집니다.

**Flash Attention**은 이 문제를 해결하기 위해 Attention 계산을 타일(tile) 단위로 쪼개서 수행합니다. 전체 S 행렬을 한꺼번에 만들지 않고, Q, K, V를 작은 블록으로 나누어 SRAM에 올릴 수 있는 크기만큼씩 처리합니다. 각 블록에 대해 부분적인 Attention 점수를 계산하고 소프트맥스를 적용한 뒤, 최종 결과에 점진적으로 합산합니다. 이 과정에서 중간 행렬 S와 P를 HBM에 기록하지 않습니다. 모든 중간 계산이 SRAM 안에서 이루어지고, 최종 출력 O만 HBM에 기록됩니다.

```
표준 Attention                     Flash Attention

Q, K를 HBM에서 읽음                Q, K, V를 블록 단위로 SRAM에 적재
  |                                  |
S = Q x K^T (결과를 HBM에 기록)     블록 내에서 S_block = Q_block x K_block^T
  |                                  |
S를 HBM에서 다시 읽음               SRAM 안에서 softmax 적용
  |                                  |
P = softmax(S) (결과를 HBM에 기록)  SRAM 안에서 P_block x V_block 계산
  |                                  |
P를 HBM에서 다시 읽음               부분 결과를 누적하여 최종 O에 합산
  |                                  |
O = P x V (결과를 HBM에 기록)       최종 O만 HBM에 기록

HBM 읽기/쓰기: 많음                HBM 읽기/쓰기: 적음
중간 행렬 S, P: HBM에 저장          중간 행렬: SRAM에서만 사용
메모리: O(N^2)                      메모리: O(N)
```

Flash Attention의 핵심은 수학적 결과가 표준 Attention과 동일하다는 점입니다. 소프트맥스를 블록 단위로 나누어 계산할 때, 각 블록의 최대값과 합산값을 별도로 추적하면서 점진적으로 보정하는 기법을 사용합니다. 이를 통해 전체를 한꺼번에 계산한 것과 정확히 같은 결과를 얻습니다. 근사가 아니라 정확한 계산입니다.

실제 효과를 숫자로 보면, 시퀀스 길이가 2,048인 경우 표준 Attention 대비 2배에서 4배의 속도 향상을 보이며, 시퀀스 길이가 길어질수록 개선 폭이 커집니다. 메모리 사용량은 시퀀스 길이의 제곱(O(N^2))에서 선형(O(N))으로 줄어듭니다. 6.1.2에서 다룬 vLLM을 포함하여, 현재 대부분의 서빙 프레임워크는 Flash Attention을 기본으로 사용합니다.

다음으로 Decode 단계의 메모리 바운드 문제를 완화하는 기법인 **Speculative Decoding**을 살펴보겠습니다.

앞서 설명한 것처럼, Decode 단계에서는 토큰 하나를 생성할 때마다 모델의 전체 가중치를 HBM에서 읽어야 합니다. 70억 파라미터 모델이라면 토큰당 14GB를 읽는 셈입니다. 100개의 토큰을 생성하면 100번 읽어야 합니다. Speculative Decoding은 이 반복을 줄이는 아이디어에서 출발합니다.

Speculative Decoding의 핵심 아이디어는 작고 빠른 모델("초안 모델", draft model)이 여러 토큰을 먼저 빠르게 생성하고, 크고 정확한 모델("검증 모델", target model)이 이 초안을 한꺼번에 검증하는 것입니다. 검증 모델이 초안 토큰들을 병렬로 처리할 수 있으므로, 개별적으로 하나씩 생성하는 것보다 효율적입니다.

구체적인 동작 과정을 단계별로 설명하겠습니다.

첫 번째 단계에서, 초안 모델이 K개의 토큰을 자기회귀 방식으로 생성합니다. 초안 모델은 파라미터 수가 적어(예: 1억 개) 토큰당 읽어야 할 가중치가 작고, 생성 속도가 빠릅니다. 예를 들어 K를 4로 설정하면, 초안 모델이 4개의 후보 토큰을 연속으로 생성합니다.

두 번째 단계에서, 검증 모델이 원래 입력에 초안 모델의 4개 토큰을 이어붙인 전체 시퀀스를 한 번에 처리합니다. 이 처리는 Prefill과 유사한 방식으로 이루어져, 4개 토큰을 각각 별도로 처리하는 것보다 효율적입니다. 검증 모델은 각 위치에서 자신이 생성했을 토큰의 확률 분포를 출력합니다.

세 번째 단계에서, 초안 모델이 생성한 각 토큰을 검증 모델의 확률 분포와 비교합니다. 초안 토큰이 검증 모델의 분포에서도 높은 확률을 가지면 채택합니다. 확률이 낮으면 그 토큰을 거부하고, 검증 모델의 분포에서 새 토큰을 샘플링합니다. 거부된 토큰 이후의 초안 토큰들도 함께 폐기됩니다.

```
Speculative Decoding 과정 (K=4 예시)

1. 초안 모델이 4개 토큰 생성
   입력: "서울의 날씨는"
   초안: ["오늘", "매우", "맑고", "따뜻"]

2. 검증 모델이 한 번에 검증
   입력: "서울의 날씨는 오늘 매우 맑고 따뜻"
   --> 각 위치의 확률 분포 계산

3. 토큰별 채택/거부
   "오늘"   --> 검증 모델 확률 높음 --> 채택
   "매우"   --> 검증 모델 확률 높음 --> 채택
   "맑고"   --> 검증 모델 확률 낮음 --> 거부, 검증 모델에서 "화창하고" 샘플링
   "따뜻"   --> 앞 토큰이 거부되어 폐기

   결과: "오늘 매우 화창하고" (3개 토큰 확정)

4. 다시 1단계로 돌아가 반복
```

이 방식의 장점은 검증 모델의 가중치를 한 번만 읽는 동안 여러 토큰이 확정될 수 있다는 점입니다. 초안 모델의 정확도가 높을수록, 즉 검증 모델과 비슷한 토큰을 생성할수록 채택률이 높아지고 속도 향상 폭이 커집니다. 일반적으로 2배에서 3배의 속도 향상을 기대할 수 있습니다. 중요한 것은, 거부 시 검증 모델의 분포에서 샘플링하므로 최종 출력의 품질은 검증 모델만 사용한 것과 통계적으로 동일합니다.

이제 하나의 GPU로는 모델 전체를 올릴 수 없을 때의 해법인 **모델 병렬화 전략**을 살펴보겠습니다. 70억 파라미터 모델은 FP16 형식으로 약 14GB이므로 하나의 GPU에 올릴 수 있지만, 700억 파라미터 모델은 약 140GB가 되어 80GB 용량의 A100 한 장으로는 부족합니다. 이때 모델을 여러 GPU에 나누어 올리는 것이 모델 병렬화입니다. 모델을 나누는 방식에 따라 세 가지 전략으로 구분됩니다.

**텐서 병렬(Tensor Parallelism)**은 하나의 층 안에 있는 행렬 연산을 여러 GPU에 나누어 수행하는 방식입니다. Transformer의 한 층에는 큰 가중치 행렬이 여러 개 있습니다. 예를 들어 Feed-Forward Network의 가중치 행렬이 크기가 [4096, 16384]라면, 이 행렬을 열 방향으로 4등분하여 GPU 4장에 [4096, 4096]씩 배치할 수 있습니다. 각 GPU는 자신이 가진 부분 행렬로 곱셈을 수행한 뒤, 결과를 모아서 최종 출력을 만듭니다. 모든 GPU가 같은 층을 동시에 처리하므로 지연 시간이 줄어듭니다.

텐서 병렬의 핵심 비용은 GPU 간 통신입니다. 각 층을 처리할 때마다 부분 결과를 GPU들이 주고받아야 합니다. 이 통신은 NVLink처럼 GPU 간 대역폭이 높은 연결에서 효율적이며, 네트워크를 통해 다른 서버의 GPU와 통신해야 하는 경우에는 지연이 커져 효율이 급격히 떨어집니다. 그래서 텐서 병렬은 같은 서버 안의 GPU들 사이에서 사용하는 것이 일반적입니다.

**파이프라인 병렬(Pipeline Parallelism)**은 모델의 층을 여러 GPU에 순차적으로 배치하는 방식입니다. 예를 들어 32개 층으로 이루어진 모델을 GPU 4장에 나누면, GPU 0이 1번부터 8번 층을, GPU 1이 9번부터 16번 층을, GPU 2가 17번부터 24번 층을, GPU 3이 25번부터 32번 층을 담당합니다. 입력 데이터가 GPU 0에서 처리된 뒤 중간 결과가 GPU 1로 전달되고, 다시 GPU 2, GPU 3으로 이어집니다.

```
파이프라인 병렬 (32층 모델, GPU 4장)

GPU 0         GPU 1         GPU 2         GPU 3
[층 1~8]  --> [층 9~16] --> [층 17~24] --> [층 25~32]
   |             |              |              |
입력 처리     중간 결과       중간 결과       최종 출력
              전달받아 처리   전달받아 처리   생성
```

파이프라인 병렬의 장점은 GPU 간 통신량이 적다는 것입니다. 층 사이의 중간 결과만 전달하면 되므로, 텐서 병렬처럼 층 내부에서 빈번하게 통신할 필요가 없습니다. 그래서 서로 다른 서버에 있는 GPU들 사이에서도 사용할 수 있습니다. 단점은 한 요청을 처리할 때 GPU들이 순차적으로 일하므로, 앞쪽 GPU가 처리하는 동안 뒤쪽 GPU가 유휴 상태가 되는 "파이프라인 버블(pipeline bubble)"이 발생한다는 점입니다. 여러 요청을 동시에 처리하는 마이크로배치(micro-batch)를 활용하면 이 유휴 시간을 줄일 수 있습니다.

**데이터 병렬(Data Parallelism)**은 모델 전체를 여러 GPU에 복제하고, 들어오는 요청을 GPU별로 나누어 처리하는 방식입니다. 모델이 하나의 GPU에 들어갈 수 있는 크기일 때 사용합니다. GPU 4장에 같은 모델을 올려 놓고, 요청 4개가 동시에 들어오면 각 GPU가 하나씩 담당합니다. 모델을 나누는 것이 아니라 작업을 나누는 것이므로 구현이 단순하고 GPU 간 통신도 거의 필요하지 않습니다. 단점은 각 GPU에 모델 전체가 올라가야 하므로, GPU당 메모리 사용량이 줄어들지 않는다는 것입니다.

실제 서빙 환경에서는 이 세 가지 전략을 조합하여 사용합니다. 같은 서버 안의 GPU들에는 텐서 병렬을 적용하고, 서버 간에는 파이프라인 병렬을 적용하며, 동일한 구성의 서버를 여러 대 두어 데이터 병렬로 처리량을 높이는 방식입니다.

```
멀티 GPU 서빙: 병렬화 전략 조합 예시

서버 A (GPU 0, 1, 2, 3)          서버 B (GPU 0, 1, 2, 3)
+----------------------------+   +----------------------------+
| 텐서 병렬: 모델의 각 층을   |   | 텐서 병렬: 같은 모델의     |
| GPU 4장이 나누어 연산       |   | 복제본을 GPU 4장이 연산    |
+----------------------------+   +----------------------------+
         서버 A와 B는 데이터 병렬: 요청을 분산 처리

또는 (모델이 더 클 경우)

서버 A (GPU 0, 1)   --> 서버 B (GPU 0, 1)
[층 1~16, 텐서 병렬]    [층 17~32, 텐서 병렬]
     파이프라인 병렬로 서버 간 연결
```

멀티 GPU 서빙을 설정할 때 핵심적으로 고려할 사항은 세 가지입니다. 첫째, 모델 크기를 확인하여 GPU 몇 장이 필요한지 계산합니다. 파라미터 수에 데이터 형식의 바이트 수를 곱하면 가중치 메모리를 산출할 수 있습니다. 예를 들어 700억 파라미터 모델을 FP16(2바이트)으로 로드하면 약 140GB이고, KV 캐시와 활성화 값을 위한 여유 공간까지 합산하면 필요한 총 메모리를 가늠할 수 있습니다. 둘째, GPU 간 연결 방식을 확인합니다. 같은 서버 안의 GPU 사이에 NVLink가 있으면 텐서 병렬을 효율적으로 사용할 수 있고, 서버 간 연결이 InfiniBand 같은 고속 네트워크라면 파이프라인 병렬도 실용적입니다. 일반 이더넷으로 연결된 서버 간에는 통신 지연이 커서 데이터 병렬 위주로 설계하는 것이 현실적입니다. 셋째, 서빙 프레임워크의 병렬화 설정을 맞춥니다. 6.1.2에서 다룬 vLLM을 예로 들면, tensor-parallel-size 파라미터로 텐서 병렬에 사용할 GPU 수를 지정합니다.

```
vLLM에서 텐서 병렬 4장 설정 예시

python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-2-70b-hf \
    --tensor-parallel-size 4 \
    --gpu-memory-utilization 0.9
```

이 명령은 vLLM 서버를 시작하면서 Llama-2-70B 모델을 4장의 GPU에 텐서 병렬로 분산합니다. --tensor-parallel-size 4는 모델의 각 층을 4장의 GPU가 나누어 처리한다는 의미입니다. --gpu-memory-utilization 0.9는 각 GPU 메모리의 90%까지 사용하도록 허용하는 설정으로, 나머지 10%는 시스템 오버헤드를 위해 남겨 둡니다.

정리하면, GPU 기반 LLM 추론의 성능은 연산 처리량과 메모리 대역폭이라는 두 가지 축으로 결정됩니다. Prefill 단계는 대규모 행렬 연산으로 인해 연산 바운드이고, Decode 단계는 토큰당 전체 가중치를 읽어야 하므로 메모리 바운드입니다. Flash Attention은 HBM 입출력을 줄여 Attention 연산을 가속하고, Speculative Decoding은 작은 초안 모델로 여러 토큰을 미리 생성한 뒤 큰 모델이 한꺼번에 검증하여 Decode 단계의 반복을 줄입니다. 모델이 하나의 GPU에 들어가지 않을 때는 텐서 병렬, 파이프라인 병렬, 데이터 병렬을 조합하여 여러 GPU에 분산합니다.

다음 단원인 6.2.2에서는 배칭 전략을 다룹니다. 여러 사용자의 요청을 효율적으로 묶어 처리하는 Dynamic Batching과 Continuous Batching의 동작 원리를 살펴봅니다.

이 단원을 마치면 GPU 메모리 구조와 대역폭이 LLM 추론 성능에 미치는 영향을 설명하고, 연산 바운드와 메모리 바운드 병목을 구분하며, Flash Attention, Speculative Decoding, 모델 병렬화 전략을 목적에 맞게 선택할 수 있습니다.
