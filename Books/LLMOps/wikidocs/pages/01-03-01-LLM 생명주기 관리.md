# 1.3.1 LLM 생명주기 관리

1.1.3에서 Foundation Model의 개념과 운영상의 고유 과제를 살펴보았고, 1.2.3에서는 LLM이 입력을 받아 토큰을 순차적으로 생성하는 추론 과정과 Temperature, Top-k, Top-p 같은 생성 전략 파라미터를 다루었습니다. 이제 시야를 넓혀, LLM 기반 서비스가 아이디어 단계에서 실제 운영에 이르기까지 어떤 과정을 밟는지를 전체적으로 조망할 차례입니다.

소프트웨어를 개발할 때, 코드를 작성하는 것만으로 끝나지 않습니다. 테스트를 거치고, 서버에 배포하고, 장애가 없는지 모니터링하며, 문제가 발생하면 수정하여 다시 배포합니다. LLM 기반 서비스도 마찬가지입니다. 다만, LLM은 전통적 소프트웨어와 다른 특성을 갖고 있어서, 각 과정에서 다루어야 할 활동이 다릅니다. 모델을 직접 학습시키는 것이 아니라 이미 만들어진 Foundation Model 위에 프롬프트를 설계하는 경우가 많고, 출력이 확정적이지 않아 같은 입력에도 다른 결과가 나올 수 있으며, 모델 자체가 외부 업체에 의해 예고 없이 업데이트될 수도 있습니다. 이런 특성 때문에, LLM 기반 서비스에는 그에 맞는 고유한 **생명주기**(lifecycle) 관리 체계가 필요합니다.

생명주기란, 하나의 시스템이 만들어지고, 운영되고, 개선되는 전체 흐름을 단계별로 정리한 것입니다. 전통적 소프트웨어에서도 '요구 분석 -> 설계 -> 구현 -> 테스트 -> 배포 -> 유지보수'라는 생명주기가 있습니다. LLM 생명주기는 이 개념을 LLM의 특성에 맞게 재구성한 것입니다. LLM 생명주기는 크게 다섯 단계로 이루어집니다. **개발**, **평가**, **배포**, **모니터링**, **갱신**입니다. 각 단계가 무엇을 하고, 어떤 산출물을 만들어 내는지 하나씩 살펴보겠습니다.

첫 번째 단계는 **개발**(Development)입니다. 이 단계에서는 LLM 기반 서비스의 핵심 로직을 구성합니다. 전통적 머신러닝에서 개발이라 하면 모델 구조를 설계하고 데이터를 수집하여 학습시키는 과정이 중심이었습니다. LLMOps에서의 개발은 다릅니다. 대부분의 경우 Foundation Model을 직접 학습시키지 않습니다. 대신, 이미 사전훈련된 Foundation Model을 선택하고, 그 모델이 원하는 결과를 내도록 프롬프트를 설계하는 것이 개발의 핵심입니다. 필요하다면 RAG(Retrieval-Augmented Generation) 파이프라인을 구축하여 외부 데이터를 모델에 공급하기도 하고, 파인튜닝을 통해 모델을 특정 도메인에 맞게 조정하기도 합니다.

개발 단계에서 이루어지는 핵심 활동을 구체적으로 보겠습니다. 먼저 사용할 Foundation Model을 선택합니다. 1.1.3에서 다룬 것처럼 성능, 비용, 라이선스를 종합하여 결정합니다. 다음으로 프롬프트를 설계합니다. 시스템 프롬프트, 사용자 프롬프트, 예시 구성 등을 반복적으로 실험하여 원하는 출력 품질을 확보합니다. 외부 데이터 연동이 필요하면 RAG 파이프라인을 설계합니다. 모델의 기본 능력으로 부족한 영역이 있으면 파인튜닝 데이터를 준비하고 학습을 수행합니다. 이 단계의 산출물은 모델 선택 기록, 프롬프트 템플릿, RAG 파이프라인 구성, 파인튜닝된 모델 가중치 등입니다.

두 번째 단계는 **평가**(Evaluation)입니다. 개발 단계에서 만든 시스템이 실제로 원하는 수준의 결과를 내는지 확인하는 과정입니다. 전통적 머신러닝에서는 정확도(accuracy)나 F1 점수 같은 정량 지표로 모델을 평가했습니다. LLM 평가는 이보다 복잡합니다. LLM의 출력은 자유로운 형태의 자연어이기 때문에, 정답이 하나로 정해지지 않는 경우가 많습니다. "이 기사를 요약해 주세요"라는 요청에 대해 올바른 요약은 여러 가지가 될 수 있습니다.

이런 특성 때문에 LLM 평가에서는 여러 방법을 함께 사용합니다. 자동화된 지표로 대량의 테스트 케이스를 빠르게 검증하고, 사람이 직접 출력을 읽고 품질을 판단하는 인간 평가를 병행합니다. 최근에는 다른 LLM을 활용하여 출력을 평가하는 LLM-as-a-Judge 방식도 쓰입니다. 1.1.3에서 다룬 환각 문제도 이 단계에서 집중적으로 검증합니다. 모델이 사실이 아닌 내용을 생성하는 비율을 측정하고, 허용 기준을 충족하는지 확인합니다. 평가 단계의 산출물은 평가 데이터셋, 평가 지표 결과 보고서, 품질 기준 통과 여부 판정입니다.

세 번째 단계는 **배포**(Deployment)입니다. 평가를 통과한 시스템을 실제 사용자가 접근할 수 있는 환경에 올리는 과정입니다. 전통적 소프트웨어의 배포와 비슷해 보이지만, LLM 배포에는 고유한 고려 사항이 있습니다.

상용 API 모델을 사용하는 경우, 배포란 프롬프트와 애플리케이션 코드를 서버에 올리고, API 호출 경로를 설정하는 것입니다. 모델 자체는 API 제공 업체의 서버에서 실행되므로 모델 서빙 인프라를 직접 구축할 필요는 없지만, API 키 관리, 요청 속도 제한(rate limit) 대응, 장애 시 대체 경로(fallback) 구성 같은 작업이 필요합니다. 오픈소스 모델을 자체 서버에서 운영하는 경우, 모델 가중치를 GPU 서버에 올리고 추론 서버를 구성하는 작업이 추가됩니다. 1.2.1에서 살펴본 것처럼 모델의 파라미터 수에 따라 필요한 GPU 메모리가 달라지므로, 적절한 인프라 규모를 산정해야 합니다. 어떤 방식이든, 트래픽 증가에 대비한 오토스케일링 설정, 응답 지연 시간 기준 설정, 비용 한도 설정 등이 배포 시 함께 이루어집니다. 배포 단계의 산출물은 운영 환경에 올라간 서비스 엔드포인트, 인프라 구성 명세, 배포 기록입니다.

네 번째 단계는 **모니터링**(Monitoring)입니다. 배포된 시스템이 실제 트래픽 환경에서 기대한 대로 동작하는지 지속적으로 관찰하는 과정입니다. 전통적 소프트웨어에서도 서버 상태, 응답 시간, 오류율을 모니터링합니다. LLM 모니터링은 이런 기술 지표에 더해, LLM 고유의 지표를 추가로 추적해야 합니다.

LLM 모니터링에서 주의 깊게 살피는 지표는 크게 세 가지 영역으로 나뉩니다. 첫째, 성능 지표입니다. 응답 지연 시간(latency), 처리량(throughput), 오류율 등 시스템의 기술적 건강 상태를 보여 줍니다. 둘째, 품질 지표입니다. 모델 출력의 정확성, 환각 발생 빈도, 사용자 만족도 등을 추적합니다. 배포 직후에는 평가 단계에서 확인한 품질이 유지되는지 확인하고, 시간이 지남에 따라 품질이 떨어지는 현상(드리프트)이 발생하는지 감시합니다. 셋째, 비용 지표입니다. API 호출 비용, 토큰 사용량, 인프라 비용 등을 추적합니다. 예상치 못한 트래픽 급증으로 비용이 폭증하는 상황을 조기에 감지하기 위함입니다. 모니터링 단계의 산출물은 대시보드, 알림 설정, 품질 추이 보고서입니다.

다섯 번째 단계는 **갱신**(Update/Iteration)입니다. 모니터링에서 발견된 문제를 해결하거나, 새로운 요구 사항을 반영하여 시스템을 개선하는 과정입니다. LLM 기반 서비스에서 갱신이 일어나는 상황은 다양합니다.

모니터링 중 품질 저하가 감지되면, 프롬프트를 수정하거나 RAG 파이프라인의 데이터 소스를 업데이트합니다. API 제공 업체가 모델 버전을 변경하면, 새 버전에서도 기존 프롬프트가 제대로 동작하는지 재검증합니다. 비용이 예산을 초과하면, 요청을 더 저렴한 모델로 분산하거나 캐싱 전략을 도입합니다. 새로운 기능 요구 사항이 생기면, 프롬프트를 추가하거나 파이프라인 구조를 변경합니다. 갱신이 완료되면 다시 평가 단계로 돌아가 변경 사항을 검증하고, 통과하면 다시 배포합니다. 갱신 단계의 산출물은 변경 이력, 개선된 프롬프트 또는 파이프라인, 재평가 결과입니다.

다섯 단계를 순서대로 나열하면 마치 일직선으로 진행되는 것처럼 보이지만, 실제로는 그렇지 않습니다. LLM 생명주기의 핵심 특성은 단계 사이에 **피드백 루프**(feedback loop)가 존재한다는 점입니다. 피드백 루프란, 이후 단계에서 얻은 정보가 이전 단계로 되돌아가 개선을 유발하는 순환 구조를 뜻합니다.

가장 대표적인 피드백 루프는 '모니터링 -> 갱신 -> 평가 -> 배포'의 순환입니다. 운영 중에 품질 문제가 감지되면, 갱신 단계에서 프롬프트를 수정하고, 평가 단계에서 수정된 프롬프트를 검증한 뒤, 다시 배포합니다. 또 다른 루프는 '평가 -> 개발'의 순환입니다. 평가 결과가 기준을 충족하지 못하면, 개발 단계로 돌아가 프롬프트를 재설계하거나 모델 선택을 재검토합니다. '모니터링 -> 개발'로 직접 이어지는 경우도 있습니다. 운영 데이터를 분석한 결과, 근본적인 아키텍처 변경이 필요하다고 판단되면 개발 단계부터 다시 시작합니다.

```
+--------+      +--------+      +--------+      +-----------+      +--------+
| 개발   | ---> | 평가   | ---> | 배포   | ---> | 모니터링  | ---> | 갱신   |
+--------+      +--------+      +--------+      +-----------+      +--------+
    ^               |                                  |               |
    |               |                                  |               |
    +--- 품질 미달 -+                                  +--- 품질 저하 -+
    |                                                  |
    +--------------- 아키텍처 변경 필요 ---------------+
```

이 피드백 루프가 있기 때문에 LLM 생명주기는 한 번 돌고 끝나는 것이 아니라, 서비스가 운영되는 동안 지속적으로 순환합니다. 이 순환이 원활하게 이루어지려면 각 단계에서 만든 산출물이 체계적으로 기록되고 추적되어야 합니다. 프롬프트가 언제, 왜 변경되었는지, 어떤 평가 결과에 따라 배포가 결정되었는지, 모니터링에서 어떤 이상이 감지되어 갱신이 시작되었는지를 추적할 수 있어야, 문제가 발생했을 때 원인을 빠르게 찾을 수 있습니다.

이 다섯 단계를 실제로 수행하려면 다양한 도구가 필요합니다. LLMOps 생태계에는 각 단계를 지원하는 도구들이 빠르게 늘어나고 있습니다. 각 단계별로 어떤 종류의 도구가 사용되는지 살펴보겠습니다.

개발 단계에서는 프롬프트 관리 도구와 오케스트레이션 프레임워크가 핵심입니다. 프롬프트 관리 도구는 프롬프트의 버전을 추적하고, 여러 프롬프트 변형을 비교 실험할 수 있게 합니다. 오케스트레이션 프레임워크는 LLM 호출, 외부 데이터 검색, 후처리 로직 등을 하나의 파이프라인으로 엮는 데 쓰입니다. LangChain, LlamaIndex 같은 도구가 이 영역에 해당합니다.

평가 단계에서는 평가 프레임워크가 사용됩니다. 대량의 테스트 케이스를 자동으로 실행하고 결과를 집계하는 도구입니다. 사람이 직접 평가하는 경우를 위한 어노테이션(annotation) 도구도 함께 사용됩니다.

배포 단계에서는 모델 서빙 프레임워크가 핵심입니다. 오픈소스 모델을 자체 운영할 때, 모델을 GPU 메모리에 올리고 API 요청을 받아 처리하는 역할을 합니다. vLLM, TGI(Text Generation Inference) 같은 도구가 여기에 해당합니다. 상용 API를 사용하는 경우에도 API 게이트웨이를 통해 요청을 관리하는 도구가 필요합니다.

모니터링 단계에서는 옵저버빌리티(observability) 플랫폼이 사용됩니다. LLM 호출의 입출력, 지연 시간, 토큰 사용량 등을 기록하고 시각화합니다. LLM 전용 옵저버빌리티 도구로는 LangSmith, Phoenix 등이 있습니다.

갱신 단계에서는 CI/CD(지속적 통합/지속적 배포) 파이프라인이 활용됩니다. 프롬프트나 코드가 변경되면 자동으로 평가를 실행하고, 기준을 통과하면 배포까지 이어지는 자동화 흐름입니다.

```
생명주기 단계별 도구 매핑

+-----------+------------------------------------------+
| 단계      | 주요 도구 유형                           |
+-----------+------------------------------------------+
| 개발      | 프롬프트 관리, 오케스트레이션 프레임워크  |
|           | (LangChain, LlamaIndex 등)               |
+-----------+------------------------------------------+
| 평가      | 평가 프레임워크, 어노테이션 도구          |
+-----------+------------------------------------------+
| 배포      | 모델 서빙 프레임워크 (vLLM, TGI 등),     |
|           | API 게이트웨이                           |
+-----------+------------------------------------------+
| 모니터링  | 옵저버빌리티 플랫폼                      |
|           | (LangSmith, Phoenix 등)                  |
+-----------+------------------------------------------+
| 갱신      | CI/CD 파이프라인, 버전 관리 시스템        |
+-----------+------------------------------------------+
```

이 도구들은 각각 독립적으로 존재하기도 하고, 하나의 플랫폼 안에 여러 단계의 기능을 통합하여 제공하기도 합니다. 어떤 도구를 선택하든, 중요한 것은 각 단계에서 필요한 기능이 빠짐없이 갖추어져 있는지 확인하는 것입니다.

도구만으로 생명주기가 돌아가지는 않습니다. 각 단계의 활동을 수행하는 사람이 필요합니다. LLM 기반 서비스 운영에는 여러 역할이 관여하며, 역할마다 책임지는 영역이 다릅니다.

**ML Engineer**(머신러닝 엔지니어)는 모델 선택, 파인튜닝, 모델 서빙 인프라 구축을 담당합니다. Foundation Model을 평가하고, 필요할 때 파인튜닝을 수행하며, 오픈소스 모델을 자체 운영할 경우 추론 서버를 구성하고 최적화합니다. 모델의 성능과 비용 사이에서 최적의 균형점을 찾는 것이 이 역할의 핵심입니다.

**Prompt Engineer**(프롬프트 엔지니어)는 프롬프트 설계와 최적화를 담당합니다. 서비스의 요구 사항을 분석하여 적절한 프롬프트를 작성하고, 평가 결과에 따라 프롬프트를 반복적으로 개선합니다. 모니터링 데이터를 분석하여 품질이 떨어지는 유형의 요청을 파악하고, 해당 유형에 맞는 프롬프트 전략을 수립합니다. LLMOps에서 프롬프트는 전통적 ML의 모델 학습 코드에 해당하는 핵심 자산이므로, 프롬프트 엔지니어의 역할은 서비스 품질에 직접적인 영향을 미칩니다.

**Platform Engineer**(플랫폼 엔지니어)는 전체 파이프라인의 인프라와 자동화를 담당합니다. 개발부터 배포, 모니터링까지의 흐름을 자동화하는 CI/CD 파이프라인을 구축하고, 서비스의 안정성과 확장성을 보장하는 인프라를 관리합니다. API 게이트웨이 설정, 오토스케일링 정책, 비용 모니터링 대시보드 구성 등이 이 역할의 영역입니다.

세 역할은 생명주기의 여러 단계에 걸쳐 협업합니다. 개발 단계에서는 ML Engineer가 모델을 선택하고 Prompt Engineer가 프롬프트를 설계하며, Platform Engineer가 개발 환경을 제공합니다. 평가 단계에서는 세 역할이 함께 평가 기준을 정하고 결과를 검토합니다. 배포 단계에서는 Platform Engineer가 인프라를 구성하고, ML Engineer가 모델 서빙을 설정합니다. 모니터링 단계에서는 Platform Engineer가 대시보드를 운영하고, Prompt Engineer와 ML Engineer가 품질 이상을 분석합니다. 갱신 단계에서는 문제의 원인에 따라 해당 역할의 담당자가 수정을 주도합니다.

```
역할별 생명주기 참여

             개발    평가    배포    모니터링   갱신
             -----   -----   -----   --------   -----
ML Eng.      모델    평가    서빙    품질       모델
             선택,   기준    설정    분석       재조정,
             파인    수립,                      파인
             튜닝    실행                       튜닝

Prompt Eng.  프롬    평가    -       품질       프롬
             프트    데이            이상       프트
             설계    터셋            분석       수정
                     작성

Platform     개발    자동    인프    대시       CI/CD
Eng.         환경    평가    라      보드       파이프
             제공    파이    구성    운영       라인
                     프라                      갱신
                     인
```

조직의 규모에 따라 한 사람이 여러 역할을 겸하기도 합니다. 소규모 팀에서는 한 명의 엔지니어가 프롬프트 설계부터 배포, 모니터링까지 모두 수행할 수 있습니다. 중요한 것은 역할의 이름이 아니라, 각 단계에서 수행해야 할 활동을 누군가가 빠짐없이 수행하는 것입니다.

정리하면, LLM 생명주기는 개발, 평가, 배포, 모니터링, 갱신의 다섯 단계로 구성되며, 이 단계들은 피드백 루프를 통해 지속적으로 순환합니다. 개발 단계에서는 모델 선택과 프롬프트 설계가 중심이고, 평가 단계에서는 자동 평가와 인간 평가를 병행하여 품질을 검증합니다. 배포 단계에서는 상용 API와 자체 서빙 중 어떤 방식을 택하느냐에 따라 활동이 달라지고, 모니터링 단계에서는 성능, 품질, 비용을 지속적으로 추적합니다. 갱신 단계에서 문제를 수정한 뒤 다시 평가와 배포로 이어지는 순환이 LLMOps의 운영 리듬을 형성합니다. 이 전체 흐름을 프롬프트 관리 도구, 평가 프레임워크, 모델 서빙 프레임워크, 옵저버빌리티 플랫폼 등의 도구가 지원하고, ML Engineer, Prompt Engineer, Platform Engineer가 역할을 나누어 수행합니다.

다음 단원인 2.1.1에서는 프롬프트 엔지니어링 기본 원칙을 다룹니다. 생명주기의 개발 단계에서 가장 핵심적인 활동인 프롬프트 설계를 본격적으로 살펴봅니다.

이 단원을 마치면 LLM의 전체 생명주기 단계를 나열하고 각 단계의 역할을 설명할 수 있습니다.
