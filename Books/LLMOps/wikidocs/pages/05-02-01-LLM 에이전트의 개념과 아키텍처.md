# 5.2.1 LLM 에이전트의 개념과 아키텍처

5.1절에서 LLM 오케스트레이션 프레임워크와 체인, 파이프라인 구성을 살펴보았습니다. 체인은 미리 정해 둔 순서대로 단계를 실행합니다. 번역 체인이면 항상 번역을 하고, 요약 체인이면 항상 요약을 합니다. 그런데 실무에서는 사용자의 요청 내용에 따라 어떤 단계를 실행할지 스스로 판단해야 하는 상황이 자주 생깁니다. 예를 들어 "서울 날씨를 알려 주고, 비가 오면 실내 데이트 코스를 추천해 줘"라는 요청을 받았다고 합시다. 이 요청을 처리하려면 먼저 날씨 API를 호출하고, 그 결과에 따라 실내 장소 검색 도구를 쓸지 말지 결정해야 합니다. 어떤 도구를 어떤 순서로 사용할지가 입력에 따라 달라지므로, 고정된 체인으로는 대응하기 어렵습니다. 이런 문제를 해결하기 위해 등장한 것이 **에이전트(Agent)**입니다.

에이전트라는 용어는 소프트웨어 공학과 인공지능 분야에서 오랫동안 쓰여 왔습니다. 넓은 의미에서 에이전트란 환경을 인지하고, 그 인지를 바탕으로 자율적으로 행동을 결정하는 시스템을 말합니다. LLM 에이전트는 이 자율적 의사결정의 핵심에 대규모 언어 모델을 둔 것입니다. LLM이 사용자의 요청을 이해하고, 어떤 행동을 취할지 추론하며, 필요한 도구를 선택하여 실행하고, 그 결과를 다시 평가하여 다음 행동을 결정합니다. 정해진 순서가 아니라 모델의 판단에 따라 흐름이 달라진다는 점에서, 5.1.4에서 다룬 고정 체인과 근본적으로 다릅니다.

LLM 에이전트를 구성하는 핵심 요소는 네 가지입니다. **LLM**(두뇌 역할), **도구**(행동 수단), **메모리**(기억), **계획**(전략 수립)입니다. 이 네 요소가 어떻게 결합하는지를 하나씩 살펴보겠습니다.

첫 번째 요소인 LLM은 에이전트의 추론 엔진입니다. 사용자의 요청을 해석하고, 현재 상황에서 무엇을 해야 하는지 판단하며, 도구 호출 결과를 분석하여 다음 행동을 결정합니다. 1.2.3에서 다룬 것처럼 LLM은 주어진 컨텍스트를 기반으로 다음 토큰을 예측하는 방식으로 동작합니다. 에이전트에서는 이 예측 능력을 활용하여, "현재 상황에서 다음으로 해야 할 행동은 무엇인가"를 추론하게 합니다. 모델의 추론 능력이 곧 에이전트의 판단력을 결정하므로, 더 강력한 모델을 사용할수록 복잡한 작업을 정확히 수행할 가능성이 높아집니다.

두 번째 요소인 **도구(Tool)**는 에이전트가 외부 세계와 상호작용하는 수단입니다. LLM은 텍스트를 생성할 수 있지만, 스스로 웹을 검색하거나 데이터베이스에 쿼리를 날리거나 파일을 수정할 수는 없습니다. 도구는 이 한계를 보완합니다. 검색 엔진 API, 계산기, 코드 실행기, 데이터베이스 클라이언트 등이 모두 도구가 될 수 있습니다. 에이전트는 현재 작업에 필요한 도구를 선택하고, 적절한 입력을 만들어 도구를 호출하며, 반환된 결과를 다음 추론에 반영합니다. 도구에 대한 상세한 내용은 5.2.2에서 다룹니다.

세 번째 요소인 **메모리(Memory)**는 에이전트가 과거 정보를 보존하고 활용하는 장치입니다. 메모리는 크게 **단기 메모리**와 **장기 메모리**로 나뉩니다.

단기 메모리는 현재 실행 중인 작업의 맥락을 유지하는 역할을 합니다. 대화 이력, 이전 도구 호출 결과, 중간 추론 과정 등이 단기 메모리에 해당합니다. 실질적으로는 LLM에 전달하는 프롬프트의 컨텍스트 윈도우가 단기 메모리의 역할을 합니다. 1.2.3에서 다룬 것처럼 컨텍스트 윈도우에는 토큰 수 제한이 있으므로, 대화가 길어지면 오래된 정보부터 밀려나게 됩니다. 5.1.4에서 다룬 대화형 체인의 Memory 활용과 같은 원리입니다.

장기 메모리는 현재 세션을 넘어서 지속되는 정보를 저장합니다. 사용자의 선호도, 과거 대화에서 얻은 지식, 이전 작업의 성공/실패 기록 등이 여기에 속합니다. 장기 메모리는 보통 벡터 데이터베이스나 키-값 저장소 같은 외부 시스템에 보관합니다. 에이전트가 새 작업을 시작할 때 관련 정보를 검색하여 프롬프트에 포함시키는 방식으로 장기 메모리를 활용합니다. 4단원에서 다룬 RAG의 검색 과정과 유사한 구조입니다.

```
에이전트 메모리 구조

+--------------------------------------------------+
|                 단기 메모리                        |
|  (현재 세션의 컨텍스트 윈도우)                     |
|  - 사용자 메시지                                   |
|  - 도구 호출 결과                                  |
|  - 중간 추론 과정                                  |
|  * 세션 종료 시 소멸                               |
+--------------------------------------------------+
           |  (필요 시 검색)
           v
+--------------------------------------------------+
|                 장기 메모리                        |
|  (외부 저장소: 벡터 DB, 키-값 저장소 등)           |
|  - 사용자 선호도                                   |
|  - 과거 작업 기록                                  |
|  - 학습된 지식                                     |
|  * 세션과 무관하게 지속                            |
+--------------------------------------------------+
```

네 번째 요소인 **계획(Planning)**은 복잡한 작업을 수행 가능한 단위로 분해하고, 실행 순서를 결정하는 전략입니다. 단순한 작업이라면 도구 하나를 호출하는 것으로 충분하지만, "3분기 매출 데이터를 분석해서 보고서를 작성하고 팀원들에게 이메일로 보내 줘"와 같은 복잡한 작업은 여러 하위 단계로 나누어야 합니다. 계획 수립 전략에는 대표적으로 세 가지가 있습니다.

첫 번째는 **ReAct(Reasoning + Acting)** 방식입니다. ReAct는 추론(Reasoning)과 행동(Acting)을 번갈아 수행하는 전략입니다. 모델이 먼저 현재 상황을 분석하고 다음에 무엇을 해야 하는지 텍스트로 추론한 뒤, 그 추론에 기반하여 하나의 행동(도구 호출)을 실행합니다. 행동의 결과를 관찰한 뒤, 다시 추론하고 다음 행동을 결정합니다. 한 번에 한 단계씩 진행하므로, 각 단계의 결과에 따라 유연하게 방향을 전환할 수 있습니다.

두 번째는 **Plan-and-Execute** 방식입니다. 작업을 시작하기 전에 전체 계획을 먼저 세우고, 그 계획에 따라 순서대로 실행하는 전략입니다. "1단계: 매출 데이터를 데이터베이스에서 조회한다. 2단계: 조회 결과를 분기별로 집계한다. 3단계: 보고서 양식에 맞게 정리한다."처럼 전체 흐름을 미리 정한 뒤 각 단계를 차례로 수행합니다. 중간에 예상과 다른 결과가 나오면 계획을 수정하기도 합니다.

세 번째는 **Tree-of-Thought** 방식입니다. 하나의 경로만 탐색하는 것이 아니라, 여러 가능한 경로를 동시에 고려합니다. 각 단계에서 여러 후보 행동을 생성하고, 각 후보의 결과를 평가하여 가장 유망한 경로를 선택합니다. 복잡한 추론이 필요한 작업에서 효과적이지만, 탐색 비용이 높아 실시간 서비스보다는 정확도가 중요한 작업에 주로 쓰입니다.

```
계획 수립 전략 비교

ReAct:
  추론 --> 행동 --> 관찰 --> 추론 --> 행동 --> 관찰 --> ...
  (한 번에 한 단계, 결과에 따라 유연하게 전환)

Plan-and-Execute:
  전체 계획 수립 --> 단계1 실행 --> 단계2 실행 --> 단계3 실행
  (미리 전체 계획, 필요 시 계획 수정)

Tree-of-Thought:
       [시작]
      /  |  \
    A    B    C      <-- 여러 후보 경로 탐색
   / \   |   / \
  A1 A2  B1 C1 C2   <-- 각 경로 평가 후 최선 선택
```

이 네 가지 요소(LLM, 도구, 메모리, 계획)가 결합하여 동작하는 방식이 **에이전트 루프(Agent Loop)**입니다. 에이전트 루프는 **Observe(관찰) -> Think(사고) -> Act(행동)**의 세 단계를 반복하는 순환 구조입니다.

Observe 단계에서 에이전트는 현재 상태를 파악합니다. 사용자의 새 메시지, 이전 도구 호출의 결과, 메모리에 저장된 관련 정보 등을 수집합니다. 이 정보들이 LLM에 전달할 프롬프트의 일부가 됩니다.

Think 단계에서 LLM이 수집된 정보를 바탕으로 추론을 수행합니다. 작업이 완료되었는지 판단하고, 완료되지 않았다면 다음으로 어떤 행동을 취해야 하는지 결정합니다. 이 추론 과정이 텍스트로 출력되는 것이 ReAct 방식의 특징입니다.

Act 단계에서 에이전트는 Think 단계에서 결정한 행동을 실행합니다. 도구를 호출하거나, 사용자에게 질문을 하거나, 최종 답변을 반환합니다. 행동의 결과는 다시 Observe 단계로 돌아가 다음 루프의 입력이 됩니다. 작업이 완료되었다고 판단하면 루프를 종료하고 최종 결과를 사용자에게 전달합니다.

```
에이전트 루프

          +----> Observe (관찰)
          |        사용자 입력, 도구 결과,
          |        메모리 검색 결과 수집
          |              |
          |              v
          |      Think (사고)
          |        LLM이 현재 상황 분석,
          |        다음 행동 결정
          |              |
          |              v
          |       Act (행동)
          |        도구 호출, 답변 생성,
          |        또는 사용자에게 질문
          |              |
          |     +--------+--------+
          |     |                 |
          |  작업 미완료       작업 완료
          |     |                 |
          +-----+                 v
                           [최종 답변 반환]
```

이 루프를 구체적인 예로 따라가 보겠습니다. 사용자가 "내일 서울 날씨를 확인하고, 비가 오면 우산을 사야 하니까 근처 편의점을 찾아 줘"라고 요청했다고 합시다.

첫 번째 루프에서 에이전트는 사용자의 요청을 관찰하고(Observe), "먼저 날씨를 확인해야 한다"고 추론한 뒤(Think), 날씨 API 도구를 호출합니다(Act). 날씨 API가 "내일 서울: 비, 강수확률 80%"라는 결과를 반환합니다.

두 번째 루프에서 에이전트는 날씨 결과를 관찰하고(Observe), "비가 올 확률이 높으니 편의점을 찾아야 한다"고 추론한 뒤(Think), 지도 검색 도구를 호출합니다(Act). 지도 검색이 근처 편의점 목록을 반환합니다.

세 번째 루프에서 에이전트는 검색 결과를 관찰하고(Observe), "필요한 정보가 모두 모였으니 답변을 구성하자"고 추론한 뒤(Think), 최종 답변을 사용자에게 전달합니다(Act). 이로써 루프가 종료됩니다.

이처럼 에이전트는 매 루프마다 상황을 판단하여 행동을 결정하므로, 만약 날씨가 맑다면 두 번째 루프에서 "비가 오지 않으니 편의점 검색은 불필요하다"고 판단하고, 바로 최종 답변을 반환할 수 있습니다. 이 유연함이 고정 체인과 에이전트의 결정적 차이입니다.

에이전트를 처음부터 직접 구현하려면 프롬프트 설계, 도구 연동, 메모리 관리, 루프 제어 등을 모두 만들어야 합니다. 이 작업을 간소화해 주는 것이 **에이전트 프레임워크**입니다. 대표적인 프레임워크 세 가지를 살펴보겠습니다.

**LangGraph**는 5.1.2에서 다룬 LangChain 생태계의 일부로, 에이전트의 동작을 **그래프(graph)** 구조로 모델링합니다. 노드는 각각의 처리 단계(LLM 호출, 도구 실행 등)를 나타내고, 엣지는 단계 간의 전환 조건을 나타냅니다. 조건부 엣지를 사용하면 LLM의 출력에 따라 다음 노드를 동적으로 선택할 수 있습니다. 이 구조 덕분에 에이전트의 실행 흐름을 시각적으로 파악하기 쉽고, 복잡한 분기 로직도 명확하게 표현할 수 있습니다. 또한 각 노드의 실행 상태를 저장하고 복원할 수 있는 체크포인트 기능을 제공하여, 장시간 실행되는 작업이나 사람의 승인이 필요한 워크플로에 유리합니다.

**CrewAI**는 여러 에이전트가 역할을 나누어 협업하는 방식에 초점을 맞춘 프레임워크입니다. 각 에이전트에게 '연구원', '작가', '편집자' 같은 역할(role)을 부여하고, 순차 또는 위계 구조로 작업을 분배합니다. 역할 기반 설계 덕분에 개별 에이전트의 프롬프트를 단순하게 유지하면서도, 전체적으로는 복잡한 작업을 처리할 수 있습니다. 멀티 에이전트 패턴에 대한 상세 내용은 5.2.3에서 다룹니다.

**AutoGen**은 마이크로소프트에서 개발한 프레임워크로, 에이전트 간 대화를 통해 작업을 수행하는 방식을 채택합니다. 에이전트들이 서로 메시지를 주고받으며 협업하는 구조로, 코드 생성과 실행을 특히 잘 지원합니다. 하나의 에이전트가 코드를 작성하면 다른 에이전트가 이를 실행하고 결과를 보고하는 식입니다. 사람이 대화 중간에 개입하여 승인하거나 방향을 수정하는 human-in-the-loop 패턴도 자연스럽게 구현할 수 있습니다.

```
에이전트 프레임워크 비교

LangGraph         CrewAI            AutoGen
-------           ------            -------
그래프 기반       역할 기반         대화 기반
흐름 제어         에이전트 협업     에이전트 간 메시지

```

정리하면, LLM 에이전트는 LLM을 추론 엔진으로 삼아, 도구를 통해 외부 세계와 상호작용하고, 메모리로 맥락을 유지하며, 계획 수립 전략으로 복잡한 작업을 분해하여 수행하는 자율적 시스템입니다. Observe, Think, Act로 이루어진 에이전트 루프가 이 네 요소를 결합하는 실행 구조이며, LangGraph, CrewAI, AutoGen 같은 프레임워크가 이 구조의 구현을 돕습니다.

다음 단원인 5.2.2에서는 에이전트의 핵심 행동 수단인 도구 호출과 함수 호출을 다룹니다. LLM이 어떻게 외부 도구를 인식하고, 적절한 입력을 구성하여 호출하며, 결과를 처리하는지를 구체적으로 살펴봅니다.

이 단원을 마치면 LLM 에이전트의 네 가지 핵심 구성요소(LLM, 도구, 메모리, 계획)를 설명하고, 에이전트 루프의 동작 흐름을 도식화할 수 있습니다.
