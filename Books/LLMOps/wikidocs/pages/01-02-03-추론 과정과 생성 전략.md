# 1.2.3 추론 과정과 생성 전략

1.2.1에서 Transformer의 Decoder-only 구조가 인과적 마스킹 아래에서 다음 단어를 예측하는 방식으로 동작한다는 점을 살펴보았고, 1.2.2에서는 텍스트가 모델에 들어가기 전에 토큰 단위로 분리된다는 점을 다루었습니다. 이 단원에서는 모델이 실제로 텍스트를 생성할 때 내부에서 어떤 일이 일어나는지, 그리고 생성 결과를 조절하는 파라미터들이 어떻게 작동하는지를 다룹니다. LLMOps 관점에서 이 내용이 중요한 이유는, 같은 모델이라도 생성 전략 파라미터를 어떻게 설정하느냐에 따라 응답의 품질, 일관성, 비용이 크게 달라지기 때문입니다. 모델을 서빙하거나 API를 운영할 때 temperature, top_p 같은 파라미터를 자주 접하게 되는데, 이 값들이 내부적으로 무엇을 하는지 이해하지 못하면 원하는 결과를 얻기 어렵습니다.

대규모 언어 모델이 텍스트를 생성하는 기본 방식을 **자기회귀(Autoregressive) 생성**이라고 합니다. 자기회귀라는 이름은 "자기 자신의 출력을 다시 입력으로 사용한다"는 뜻에서 왔습니다. 모델은 한 번에 문장 전체를 만들어 내지 않습니다. 대신, 토큰 하나를 생성한 뒤 그 토큰을 기존 입력 뒤에 붙이고, 늘어난 입력을 다시 모델에 넣어 그다음 토큰을 생성합니다. 이 과정을 종료 조건이 충족될 때까지 반복합니다.

구체적인 예를 들어 보겠습니다. 사용자가 "서울의 날씨는"이라는 프롬프트를 입력했다고 가정합니다. 모델은 먼저 이 입력을 받아 다음 토큰으로 "오늘"을 생성할 수 있습니다. 그러면 입력이 "서울의 날씨는 오늘"로 늘어납니다. 이 늘어난 입력을 다시 모델에 넣으면 "맑고"를 생성할 수 있습니다. 다시 "서울의 날씨는 오늘 맑고"를 입력으로 넣으면 "따뜻합니다"를 생성하고, 그다음에 종료 토큰(1.2.2에서 다룬 EOS 토큰)이 나오면 생성이 끝납니다.

```
[입력] "서울의 날씨는"
  |
  +--> 모델 추론 --> "오늘" 생성
  |
[입력] "서울의 날씨는 오늘"
  |
  +--> 모델 추론 --> "맑고" 생성
  |
[입력] "서울의 날씨는 오늘 맑고"
  |
  +--> 모델 추론 --> "따뜻합니다" 생성
  |
[입력] "서울의 날씨는 오늘 맑고 따뜻합니다"
  |
  +--> 모델 추론 --> [EOS] 생성 --> 종료
```

이 다이어그램에서 알 수 있듯이, 모델은 매 단계마다 전체 입력을 다시 처리하여 딱 하나의 토큰만 출력합니다. 100개의 토큰을 생성하려면 모델을 100번 실행해야 합니다. 이 점이 대규모 언어 모델의 추론 비용이 높은 근본적인 이유 중 하나입니다.

자기회귀 생성의 각 단계에서 모델이 하는 일을 좀 더 구체적으로 살펴보겠습니다. 1.2.1에서 다룬 것처럼, 입력 토큰들은 Transformer의 여러 층을 통과하면서 Self-Attention과 Feed-Forward Network를 거칩니다. 마지막 층을 통과한 뒤, 모델은 맨 마지막 토큰 위치에서 하나의 벡터를 출력합니다. 이 벡터를 어휘 전체의 크기에 해당하는 차원으로 변환하면, 각 토큰이 다음에 올 확률을 나타내는 숫자 목록이 됩니다. 이 숫자 목록을 **로짓(logit)**이라고 합니다.

로짓 자체는 아직 확률이 아닙니다. 양수일 수도 있고 음수일 수도 있으며, 합이 1이 되지도 않습니다. 로짓에 1.2.1에서 설명한 소프트맥스 함수를 적용하면, 모든 값이 0과 1 사이로 변환되고 합이 1이 되어 확률 분포가 됩니다. 예를 들어, 어휘에 5만 개의 토큰이 있다면 5만 개의 확률 값이 만들어지고, 이 중에서 하나의 토큰을 선택하는 것이 한 단계의 생성입니다.

이제 모델이 추론을 수행하는 두 가지 단계, **Prefill 단계**와 **Decode 단계**를 구분하겠습니다. 이 구분은 모델이 내부적으로 동작하는 방식과 직접 관련되며, 추론 성능을 이해하는 데 핵심적인 개념입니다.

**Prefill 단계**는 사용자가 입력한 프롬프트 전체를 한꺼번에 처리하는 과정입니다. 사용자가 "서울의 날씨는"이라는 프롬프트를 보내면, 모델은 이 입력에 포함된 모든 토큰을 동시에 Transformer 층에 통과시킵니다. 앞서 자기회귀 생성에서 "한 번에 하나의 토큰을 생성한다"고 설명했는데, Prefill 단계는 이 생성이 시작되기 전에 입력을 이해하는 단계입니다. 이 과정에서 각 토큰 사이의 Attention 값이 계산되고, 그 결과가 메모리에 저장됩니다. 이 저장된 값을 **KV 캐시(Key-Value Cache)**라고 부릅니다.

KV 캐시가 필요한 이유를 설명하겠습니다. 자기회귀 생성에서 두 번째 토큰을 생성할 때, 모델은 첫 번째 토큰까지 포함한 전체 입력을 다시 처리해야 합니다. 세 번째 토큰을 생성할 때는 첫 번째와 두 번째 토큰까지 포함한 입력을 처리해야 합니다. 만약 매번 처음부터 모든 토큰의 Key와 Value를 다시 계산한다면 엄청난 낭비가 됩니다. KV 캐시는 이미 계산한 Key와 Value 벡터를 메모리에 보관하여, 새 토큰을 생성할 때 이전 토큰들의 Key, Value를 다시 계산하지 않아도 되게 합니다.

**Decode 단계**는 Prefill이 끝난 후, 토큰을 하나씩 생성하는 과정입니다. 각 단계에서 모델은 방금 생성한 토큰 하나만 Transformer 층에 넣습니다. 이 토큰의 Query 벡터가 KV 캐시에 저장된 이전 토큰들의 Key, Value와 Attention을 계산하여 다음 토큰의 로짓을 만듭니다. 새 토큰의 Key와 Value는 KV 캐시에 추가됩니다. 이 과정이 종료 토큰이 나오거나 최대 토큰 수에 도달할 때까지 반복됩니다.

```
[Prefill 단계]
  사용자 프롬프트 전체 --> Transformer 층 일괄 처리
                          --> KV 캐시 생성 (Key, Value 저장)
                          --> 첫 번째 출력 토큰의 로짓 계산

[Decode 단계] (토큰마다 반복)
  직전 생성 토큰 --> Transformer 층 처리
                     --> KV 캐시에서 이전 Key, Value 참조
                     --> 새 토큰의 Key, Value를 캐시에 추가
                     --> 다음 토큰의 로짓 계산
                     --> 토큰 선택
```

Prefill 단계는 입력 토큰을 병렬로 한꺼번에 처리하므로 GPU 연산 장치를 효율적으로 활용합니다. 반면 Decode 단계는 토큰 하나씩 순차적으로 처리하므로 GPU 활용 효율이 낮습니다. 이 때문에 긴 응답을 생성할 때 Decode 단계가 전체 추론 시간의 대부분을 차지합니다. 나중에 6장에서 다루는 추론 최적화 기법 상당수가 이 Decode 단계의 비효율을 줄이는 데 초점을 맞추고 있습니다.

이제 모델이 로짓을 확률 분포로 변환한 뒤, 그 분포에서 실제로 토큰을 선택하는 방법을 살펴보겠습니다. 가장 단순한 방법은 확률이 가장 높은 토큰을 항상 선택하는 것입니다. 이를 **그리디 디코딩(greedy decoding)**이라고 합니다. 그리디 디코딩은 매 단계에서 가장 가능성 높은 토큰 하나만 선택하므로, 같은 입력에 대해 항상 같은 출력을 만듭니다. 결과가 예측 가능하고 일관되지만, 표현이 단조롭고 반복적인 문장이 생기기 쉽습니다. 실제 서비스에서는 좀 더 다양하고 자연스러운 텍스트를 만들기 위해 확률 분포에서 **샘플링**하는 방식을 주로 사용합니다.

샘플링이란 확률 분포에 따라 무작위로 토큰을 뽑는 것입니다. 확률이 0.3인 토큰은 10번 중 약 3번 선택되고, 확률이 0.05인 토큰은 10번 중 약 0.5번 선택됩니다. 이렇게 하면 같은 입력에서도 매번 다른 출력이 나올 수 있어 텍스트가 다양해집니다. 하지만 확률이 매우 낮은 토큰도 간혹 선택될 수 있어, 맥락에 맞지 않는 엉뚱한 단어가 끼어들 위험이 있습니다. 이 문제를 해결하기 위해 샘플링 범위와 확률 분포의 형태를 조절하는 여러 파라미터가 존재합니다.

**Temperature**는 확률 분포의 형태를 조절하는 파라미터입니다. 소프트맥스 함수를 적용하기 전에 로짓을 temperature 값으로 나누는 방식으로 동작합니다. 수식으로 표현하면 다음과 같습니다.

```
확률 분포 = softmax(로짓 / temperature)
```

temperature가 어떤 효과를 내는지 구체적인 숫자로 살펴보겠습니다. 어휘에 "맑고", "흐리고", "추운"이라는 세 후보 토큰이 있고, 모델이 계산한 로짓이 각각 2.0, 1.0, 0.5라고 가정합니다.

temperature가 1.0일 때는 로짓을 그대로 소프트맥스에 넣습니다. 결과는 "맑고" 약 50%, "흐리고" 약 30%, "추운" 약 20%가 됩니다. 이것이 모델이 학습 데이터에서 파악한 원래 확률 분포입니다.

temperature를 0.5로 낮추면, 로짓을 0.5로 나누어 4.0, 2.0, 1.0이 됩니다. 값의 차이가 벌어지면서 소프트맥스 결과도 편중됩니다. "맑고"의 확률이 약 80%까지 올라가고, 나머지 후보의 확률은 크게 줄어듭니다. 확률이 높은 토큰에 더 집중하므로 출력이 예측 가능하고 일관됩니다.

temperature를 2.0으로 높이면, 로짓을 2.0으로 나누어 1.0, 0.5, 0.25가 됩니다. 값의 차이가 좁아지면서 소프트맥스 결과가 균등에 가까워집니다. 세 토큰의 확률이 비슷해져서, "추운"처럼 원래 확률이 낮았던 토큰도 자주 선택됩니다. 출력이 다양해지는 대신 예측하기 어렵고, 맥락과 동떨어진 토큰이 나올 가능성도 커집니다.

```
로짓 = [2.0, 1.0, 0.5]   ("맑고", "흐리고", "추운")

temperature = 0.5  -->  로짓/T = [4.0, 2.0, 1.0]  -->  확률 ~[80%, 15%, 5%]
temperature = 1.0  -->  로짓/T = [2.0, 1.0, 0.5]  -->  확률 ~[50%, 30%, 20%]
temperature = 2.0  -->  로짓/T = [1.0, 0.5, 0.25] -->  확률 ~[40%, 33%, 27%]
```

temperature를 0에 가깝게 설정하면 확률이 가장 높은 토큰 하나에 거의 모든 확률이 몰려, 사실상 그리디 디코딩과 같은 결과가 됩니다. 정확한 답변이 필요한 작업(코드 생성, 사실 기반 질의응답 등)에서는 낮은 temperature를, 창의적인 글쓰기나 다양한 답변이 필요한 작업에서는 높은 temperature를 사용하는 것이 일반적입니다.

temperature는 확률 분포의 형태를 바꾸지만, 여전히 어휘 전체에서 샘플링합니다. 어휘에 5만 개의 토큰이 있으면 확률이 극히 낮은 토큰도 이론적으로 선택될 수 있습니다. **Top-k 샘플링**은 이 문제를 해결하기 위해 후보 토큰의 수를 제한합니다.

Top-k 샘플링의 동작 방식은 다음과 같습니다. 먼저 소프트맥스를 거쳐 확률 분포를 구합니다. 확률이 높은 순서로 토큰을 정렬합니다. 상위 k개의 토큰만 남기고 나머지는 후보에서 제외합니다. 남은 k개의 토큰에 대해 확률을 다시 정규화하여 합이 1이 되도록 만듭니다. 이 조정된 분포에서 샘플링합니다.

예를 들어 k를 3으로 설정하면, 확률이 가장 높은 3개의 토큰만 후보가 됩니다. 어휘에 5만 개의 토큰이 있더라도, 확률이 극히 낮은 나머지 49,997개는 아예 선택 대상에서 빠집니다. k가 1이면 항상 가장 확률 높은 토큰만 선택하므로 그리디 디코딩과 같고, k가 어휘 전체 크기이면 제한이 없는 것과 같습니다.

Top-k 샘플링에는 한 가지 약점이 있습니다. k 값을 고정하면, 확률 분포의 모양에 관계없이 항상 같은 수의 토큰만 후보에 포함됩니다. 어떤 단계에서는 모델이 확신이 높아 상위 2개 토큰에 확률의 95%가 집중되어 있을 수 있습니다. 이때 k가 50이면, 확률이 거의 0에 가까운 48개의 토큰이 불필요하게 후보에 남습니다. 반대로, 어떤 단계에서는 후보가 고르게 분포되어 상위 50개를 넘어서도 의미 있는 토큰이 있을 수 있는데, k가 50이면 그 토큰들은 잘려 나갑니다.

**Top-p 샘플링**은 이 문제를 해결합니다. Top-p는 **Nucleus 샘플링**이라고도 부르며, 고정된 개수 대신 누적 확률을 기준으로 후보를 결정합니다. 동작 방식은 다음과 같습니다. 확률이 높은 순서로 토큰을 정렬합니다. 위에서부터 확률을 더해 가면서, 누적 확률이 p에 도달할 때까지 토큰을 후보에 포함합니다. 누적 확률이 p를 넘는 시점의 토큰까지를 포함하고, 나머지는 제외합니다. 남은 토큰들의 확률을 다시 정규화한 뒤 샘플링합니다.

구체적인 예를 들어 보겠습니다. 확률 분포가 아래와 같고 top_p를 0.9로 설정한 경우입니다.

```
토큰       확률     누적 확률
"맑고"     0.40     0.40
"흐리고"   0.25     0.65
"따뜻한"   0.15     0.80
"추운"     0.10     0.90  <-- 여기서 누적 확률이 0.9에 도달
"습한"     0.05     0.95
"건조한"   0.03     0.98
...        ...      ...
```

누적 확률이 0.9에 도달하는 "추운"까지 4개의 토큰이 후보에 포함됩니다. 나머지 토큰은 후보에서 제외됩니다. 이 방식이 Top-k보다 유연한 이유는, 모델이 확신이 높을 때는 적은 수의 토큰만 포함되고, 확신이 낮아 확률이 고르게 퍼져 있을 때는 더 많은 토큰이 포함되기 때문입니다.

p를 1.0으로 설정하면 모든 토큰이 후보에 포함되어 제한이 없는 것과 같고, p를 아주 작은 값으로 설정하면 가장 확률 높은 소수의 토큰만 남습니다. 실무에서는 top_p를 0.9나 0.95로 설정하는 경우가 많습니다.

Temperature, Top-k, Top-p는 함께 사용할 수 있습니다. 일반적인 적용 순서는 다음과 같습니다. 먼저 temperature로 로짓을 나누어 확률 분포의 형태를 조절합니다. 그다음 Top-k로 상위 k개 토큰만 남깁니다. Top-p로 누적 확률 기준 필터링을 한 번 더 적용합니다. 남은 토큰에서 샘플링합니다.

```
로짓 --> temperature 적용 --> softmax --> Top-k 필터 --> Top-p 필터 --> 샘플링
```

모델이 텍스트를 생성하다 보면 같은 단어나 구절을 계속 반복하는 현상이 나타날 수 있습니다. 예를 들어 "이 제품은 좋은 제품이고 좋은 품질의 좋은 디자인을 가진 좋은..."처럼 특정 토큰이 반복되는 경우입니다. 자기회귀 생성의 특성상, 한 번 생성된 토큰이 이후 맥락에 포함되면서 같은 토큰이 다시 높은 확률을 받는 피드백 루프가 생길 수 있기 때문입니다.

**반복 페널티(Repetition Penalty)**는 이미 생성된 토큰이 다시 선택될 확률을 낮추어 이 문제를 완화합니다. 동작 방식은 이렇습니다. 이전에 생성된 토큰 목록을 유지합니다. 다음 토큰의 로짓을 계산할 때, 이미 등장한 토큰의 로짓에 페널티를 적용합니다. 해당 로짓이 양수이면 페널티 값으로 나누고, 음수이면 페널티 값을 곱합니다. 페널티 값이 1.0이면 조정이 없고, 1.0보다 크면 반복이 억제됩니다. 값이 클수록 억제 강도가 세집니다. 1.2 정도의 값이 일반적인 출발점이며, 너무 높으면 자연스러운 반복(관사, 조사 등 흔한 토큰)까지 억제되어 문장이 부자연스러워질 수 있습니다.

**길이 페널티(Length Penalty)**는 생성되는 텍스트의 길이를 조절하는 파라미터입니다. 이 파라미터는 주로 빔 서치(beam search)라는 디코딩 방식에서 사용됩니다. 빔 서치는 여러 후보 시퀀스를 동시에 추적하면서 가장 높은 전체 확률을 가진 시퀀스를 선택하는 방법입니다. 이때 길이가 긴 시퀀스는 토큰마다 확률을 곱해 가므로 전체 확률이 자연히 낮아집니다. 길이 페널티는 시퀀스의 확률을 길이로 정규화하여 이 편향을 보정합니다.

길이 페널티 값이 1.0이면 기본적인 정규화를 적용합니다. 1.0보다 크면 긴 시퀀스에 유리하게 작용하여 더 긴 응답을 생성하는 경향이 생깁니다. 1.0보다 작으면 짧은 시퀀스에 유리하게 작용하여 더 간결한 응답을 만듭니다. 0으로 설정하면 길이에 따른 보정이 전혀 없어, 짧은 응답이 선호됩니다.

마지막으로 **컨텍스트 윈도우**와 **최대 토큰 수**를 살펴보겠습니다. 이 두 개념은 모델이 처리할 수 있는 텍스트의 양과 직접 관련됩니다.

**컨텍스트 윈도우(context window)**는 모델이 한 번에 처리할 수 있는 토큰의 최대 개수입니다. 1.2.1에서 다룬 위치 인코딩과 Attention 메커니즘이 처리할 수 있는 시퀀스 길이에 한계가 있기 때문에, 모든 모델에는 이 상한이 존재합니다. GPT-3의 컨텍스트 윈도우는 4,096 토큰이었고, GPT-4는 128,000 토큰까지 지원하며, Claude 3는 200,000 토큰까지 처리할 수 있습니다.

컨텍스트 윈도우는 입력 토큰과 출력 토큰을 합산한 전체 길이에 적용됩니다. 컨텍스트 윈도우가 4,096 토큰인 모델에 3,000 토큰의 프롬프트를 보내면, 모델이 생성할 수 있는 최대 토큰 수는 1,096 토큰입니다. 프롬프트가 길수록 응답에 쓸 수 있는 공간이 줄어드는 것입니다.

```
|<----------- 컨텍스트 윈도우 (예: 4,096 토큰) ----------->|
|                                                          |
|  [입력 프롬프트: 3,000 토큰]  [생성 가능: 1,096 토큰]    |
```

컨텍스트 윈도우가 크면 긴 문서를 한 번에 처리하거나 긴 대화 이력을 유지할 수 있지만, KV 캐시의 메모리 사용량도 비례하여 커집니다. Prefill 단계에서 처리해야 할 토큰 수가 많아지므로 첫 토큰이 나오기까지의 지연 시간도 길어집니다. 이 때문에 LLMOps에서는 컨텍스트 윈도우의 크기와 운영 비용 사이의 균형을 고려해야 합니다.

**최대 토큰 수(max tokens)**는 모델이 한 번의 요청에서 생성할 수 있는 출력 토큰의 상한을 지정하는 파라미터입니다. 이 값은 컨텍스트 윈도우와는 별개로, API 호출 시 사용자가 직접 설정합니다. 예를 들어 max_tokens를 100으로 설정하면, 모델은 EOS 토큰이 나오지 않더라도 100개의 토큰을 생성한 시점에서 강제로 멈춥니다. 이 파라미터는 응답 길이를 제어하고, 불필요하게 긴 응답으로 인한 비용 증가를 방지하는 데 사용됩니다.

아래 표는 이 단원에서 다룬 생성 전략 파라미터들을 정리한 것입니다.

```
파라미터           역할                          일반적 범위
---------------------------------------------------------------
temperature       확률 분포의 형태 조절          0.0 ~ 2.0
top_k             상위 k개 토큰만 후보로 제한    1 ~ 어휘 크기
top_p             누적 확률 p까지 후보 포함      0.0 ~ 1.0
repetition_penalty 반복 토큰의 확률 낮춤         1.0 ~ 2.0
length_penalty    시퀀스 길이에 따른 확률 보정   0.0 ~ 2.0
max_tokens        최대 생성 토큰 수              1 ~ (컨텍스트 윈도우 - 입력 길이)
```

정리하면, 대규모 언어 모델은 자기회귀 방식으로 토큰을 하나씩 순차 생성하며, 이 과정은 입력을 일괄 처리하는 Prefill 단계와 토큰을 하나씩 생성하는 Decode 단계로 나뉩니다. 각 단계에서 모델은 로짓을 확률 분포로 변환한 뒤, temperature, top-k, top-p 같은 파라미터에 따라 다음 토큰을 선택합니다. temperature는 확률 분포의 형태를, top-k는 후보 수를, top-p는 누적 확률 기준 후보 범위를 조절합니다. 반복 페널티와 길이 페널티는 출력의 반복과 길이를 제어하며, 컨텍스트 윈도우와 최대 토큰 수는 모델이 처리하고 생성할 수 있는 토큰의 양을 결정합니다. 이 파라미터들을 목적에 맞게 조합하는 것이 LLMOps에서 일관된 품질의 응답을 확보하는 출발점입니다.

다음 단원인 1.3.1에서는 LLM 생명주기 관리를 다룹니다. 모델의 개발부터 배포, 모니터링, 갱신에 이르는 전체 흐름과 각 단계에서 수행하는 핵심 활동을 살펴봅니다.

이 단원을 마치면 LLM의 추론 과정을 Prefill과 Decode 단계로 나누어 설명하고, temperature, top-k, top-p 등 생성 전략 파라미터가 출력에 미치는 영향을 설명하며 목적에 맞게 조절할 수 있습니다.
