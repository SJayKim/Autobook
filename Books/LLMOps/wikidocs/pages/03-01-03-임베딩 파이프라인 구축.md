# 3.1.3 임베딩 파이프라인 구축

3.1.2에서 주요 임베딩 모델의 특성을 비교하고, 프로젝트의 요구사항에 맞는 모델을 선택하는 기준을 살펴보았습니다. 모델을 선택했다면, 이제 실제 텍스트 데이터를 그 모델에 통과시켜 임베딩 벡터를 생성하는 과정을 구축해야 합니다. 문서 한두 건에 대한 임베딩은 코드 몇 줄로 끝나지만, 수만에서 수백만 건의 문서를 처리하고, 새 문서가 추가될 때마다 자동으로 임베딩을 갱신하며, 동일한 문서를 불필요하게 반복 처리하지 않도록 해야 하는 것이 실무의 현실입니다. 이 단원에서는 이러한 요구를 충족하는 임베딩 파이프라인을 설계하고 구축하는 방법을 다룹니다.

**임베딩 파이프라인**이란 원본 텍스트 데이터를 입력받아, 전처리, 임베딩 생성, 저장까지 일련의 과정을 자동화한 데이터 처리 흐름을 뜻합니다. 파이프라인이라는 용어는 각 단계의 출력이 다음 단계의 입력이 되는 연쇄 구조를 의미합니다. 공장의 조립 라인처럼, 원재료(텍스트)가 여러 가공 단계를 거쳐 완성품(벡터)이 되는 흐름입니다.

임베딩 파이프라인의 전체 흐름을 먼저 그림으로 확인하겠습니다.

```
임베딩 파이프라인 전체 흐름

[원본 텍스트]
     |
     v
[1. 텍스트 전처리]
  정규화, 클리닝, 필터링
     |
     v
[2. 배치 구성]
  텍스트를 일정 크기의 묶음으로 분할
     |
     v
[3. 임베딩 생성]
  임베딩 모델에 배치 단위로 전달
     |
     v
[4. 캐시 확인/저장]
  이미 생성된 임베딩인지 확인,
  새로운 것만 생성
     |
     v
[5. 품질 검증]
  이상치 탐지, 차원 확인
     |
     v
[6. 저장]
  벡터 데이터베이스 또는 파일 시스템에 저장
```

이 흐름에서 각 단계는 독립적으로 설정을 조정할 수 있고, 한 단계에서 문제가 발생하면 해당 단계만 재실행할 수 있도록 설계하는 것이 이상적입니다. 각 단계를 하나씩 살펴보겠습니다.

첫 번째 단계는 **텍스트 전처리**(정규화, 클리닝)입니다. 임베딩 모델에 텍스트를 넣기 전에, 입력 텍스트의 품질을 정리하는 과정입니다. 전처리가 필요한 이유는, 원본 텍스트에는 임베딩 품질을 떨어뜨리는 노이즈가 포함되어 있을 수 있기 때문입니다.

**정규화**(normalization)는 텍스트의 표기를 일관되게 맞추는 작업입니다. 같은 문자라도 유니코드 표현이 다를 수 있습니다. 예를 들어, 한글 "가"는 완성형(U+AC00)으로도, 조합형(ㄱ + ㅏ)으로도 표현됩니다. 두 표현이 다른 바이트열을 가지므로, 정규화 없이 모델에 넣으면 같은 글자인데도 다른 토큰으로 처리될 수 있습니다. 유니코드 정규화(NFC 또는 NFKC)를 적용하면 이런 불일치를 제거할 수 있습니다. 또한 전각 문자를 반각으로 변환하거나, 대소문자를 통일하는 것도 정규화에 해당합니다.

**클리닝**(cleaning)은 불필요한 내용을 제거하는 작업입니다. HTML 태그, URL, 이메일 주소, 반복되는 공백이나 줄바꿈, 특수문자 나열 등이 제거 대상입니다. 클리닝의 범위는 텍스트의 출처에 따라 다릅니다. 웹 크롤링으로 수집한 텍스트에는 HTML 잔여물이 많고, PDF에서 추출한 텍스트에는 머리글/바닥글이 반복됩니다. 클리닝 규칙은 데이터 소스별로 다르게 정의해야 합니다.

전처리에서 주의할 점이 있습니다. 과도한 클리닝은 의미 있는 정보를 삭제할 수 있습니다. 예를 들어, 코드 문서에서 특수문자를 모두 제거하면 코드 예시가 훼손됩니다. 기술 문서에서 약어나 버전 번호를 제거하면 검색 품질이 떨어집니다. 전처리는 항상 원본 데이터의 특성을 고려하여 규칙을 설계해야 합니다.

```python
import unicodedata
import re

def preprocess(text):
    # 유니코드 정규화 (NFC)
    text = unicodedata.normalize("NFC", text)
    # 연속된 공백을 하나로
    text = re.sub(r"\s+", " ", text)
    # 앞뒤 공백 제거
    text = text.strip()
    return text
```

위 코드는 최소한의 전처리 예입니다. unicodedata.normalize("NFC", text)는 유니코드 정규화를 수행합니다. re.sub(r"\s+", " ", text)는 연속된 공백, 탭, 줄바꿈을 하나의 공백으로 통합합니다. strip()은 앞뒤의 불필요한 공백을 제거합니다. 실제 파이프라인에서는 이 기본 처리에 더해 데이터 소스에 맞는 추가 규칙(HTML 태그 제거, 머리글 제거 등)을 적용합니다.

두 번째 단계는 **배치 임베딩 생성**입니다. 임베딩 모델에 텍스트를 하나씩 보내면 네트워크 왕복 시간이 반복되어 전체 처리 시간이 크게 늘어납니다. 특히 API 기반 모델은 한 번의 HTTP 요청에 여러 텍스트를 묶어 보내는 것이 훨씬 효율적입니다. 이것을 배치(batch) 처리라 합니다.

배치 크기는 임베딩 모델의 API 제한과 메모리 용량에 따라 결정됩니다. OpenAI의 임베딩 API는 한 번의 요청에 최대 2,048개의 입력을 받을 수 있습니다. 자체 호스팅 모델의 경우 GPU 메모리에 따라 배치 크기가 제한됩니다. 배치가 너무 크면 메모리 부족으로 오류가 발생하고, 너무 작으면 처리 효율이 떨어집니다. 일반적으로 100에서 500개 사이의 배치 크기로 시작하여, 오류 없이 처리되는 최대 크기를 실험적으로 찾습니다.

배치 처리에서 또 다른 고려 사항은 오류 처리입니다. 수만 건을 처리하다가 중간에 API 오류가 발생하면, 이미 처리한 부분을 다시 할 필요가 없도록 진행 상태를 기록해야 합니다. 각 배치의 처리 결과를 저장하고, 실패한 배치만 재시도하는 구조가 필요합니다.

```python
def batch_embed(texts, model, batch_size=100):
    all_embeddings = []
    for i in range(0, len(texts), batch_size):
        batch = texts[i:i + batch_size]
        embeddings = model.encode(batch)
        all_embeddings.extend(embeddings)
    return all_embeddings
```

위 코드에서 texts는 전처리된 텍스트 목록이고, batch_size는 한 번에 처리할 텍스트의 수입니다. range(0, len(texts), batch_size)는 전체 목록을 batch_size 단위로 순회합니다. 각 배치를 모델에 전달하여 임베딩을 받고, 결과를 누적합니다. 실무에서는 이 기본 구조에 오류 재시도, 진행률 로깅, 속도 제한(rate limit) 준수 등의 로직을 추가합니다.

세 번째 단계는 **임베딩 캐싱 전략**입니다. 동일한 텍스트를 반복해서 임베딩할 필요는 없습니다. 같은 문서를 다시 처리하면 API 비용이 낭비되고, 자체 호스팅 모델에서도 불필요한 GPU 연산이 발생합니다. 캐싱이란 한 번 생성한 임베딩을 저장해 두고, 같은 텍스트에 대한 요청이 들어오면 저장된 임베딩을 재사용하는 전략입니다.

캐싱을 구현하는 가장 간단한 방법은 텍스트의 해시값을 키(key)로, 임베딩 벡터를 값(value)으로 저장하는 것입니다. 텍스트가 들어오면 먼저 해시값을 계산하고, 캐시에 해당 해시가 있으면 저장된 임베딩을 반환합니다. 캐시에 없으면 모델을 호출하여 임베딩을 생성하고, 결과를 캐시에 저장한 뒤 반환합니다.

```
캐싱 흐름

입력 텍스트
    |
    v
해시값 계산 (SHA-256 등)
    |
    v
캐시에 해시가 존재하는가?
    |
    +-- 예 --> 캐시에서 임베딩 반환 (모델 호출 생략)
    |
    +-- 아니오 --> 모델 호출 --> 임베딩 생성 --> 캐시에 저장 --> 반환
```

캐시 저장소는 메모리(Redis 등), 디스크(SQLite, 파일 시스템), 또는 데이터베이스를 사용할 수 있습니다. 캐시의 크기가 커지면 오래된 항목을 제거하는 만료 정책이 필요합니다. 임베딩 모델을 교체하면 기존 캐시의 모든 임베딩이 무효화되므로, 캐시 키에 모델 이름이나 버전도 함께 포함하는 것이 안전합니다.

네 번째 단계는 **임베딩 갱신과 재생성 정책**입니다. 서비스를 운영하다 보면 문서가 수정되거나, 새 문서가 추가되거나, 임베딩 모델을 교체하는 상황이 발생합니다. 이런 변경이 일어났을 때 임베딩을 어떻게 갱신할지에 대한 정책이 필요합니다.

문서가 수정된 경우, 해당 문서의 임베딩만 재생성하면 됩니다. 이를 위해 각 문서의 마지막 수정 시점과 마지막 임베딩 생성 시점을 기록해 두어야 합니다. 수정 시점이 임베딩 생성 시점보다 이후이면, 해당 문서를 재처리 대상으로 표시합니다. 새 문서가 추가된 경우에도 마찬가지로, 임베딩이 아직 생성되지 않은 문서를 탐지하여 처리합니다.

임베딩 모델을 교체하는 경우는 더 큰 작업이 필요합니다. 모델이 바뀌면 같은 텍스트에 대한 벡터가 완전히 달라지므로, 기존 벡터와 새 벡터를 같은 인덱스에 섞어 넣을 수 없습니다. 3.1.1에서 설명한 유사도 계산은 같은 모델로 생성된 벡터 사이에서만 의미가 있기 때문입니다. 따라서 모델을 교체할 때는 모든 문서의 임베딩을 새 모델로 재생성해야 합니다. 이 전체 재생성을 안전하게 수행하려면, 새 인덱스를 별도로 만들어 재생성을 완료한 뒤, 기존 인덱스와 교체하는 방식이 권장됩니다.

갱신 정책을 정리하면 다음과 같습니다.

```
변경 유형            갱신 범위           방법
--------------------------------------------------------------
문서 내용 수정       해당 문서만         수정된 문서 탐지 -> 재임베딩
새 문서 추가         새 문서만           미처리 문서 탐지 -> 임베딩 생성
문서 삭제            해당 벡터 삭제      벡터 DB에서 ID로 삭제
임베딩 모델 교체     전체 문서           새 인덱스 생성 -> 전체 재임베딩 -> 교체
```

다섯 번째 단계는 **임베딩 품질 검증**입니다. 임베딩이 생성되었다고 해서 바로 서비스에 사용하기 전에, 생성된 임베딩에 문제가 없는지 확인하는 과정입니다.

품질 검증에서 확인해야 할 항목은 세 가지입니다. 첫째, 차원이 올바른지 확인합니다. 모델이 1,024차원을 생성해야 하는데 다른 차원의 벡터가 섞여 있다면 인덱싱이 실패합니다. 둘째, 벡터에 NaN(Not a Number)이나 무한대 값이 포함되어 있지 않은지 확인합니다. 입력 텍스트가 비정상적(빈 문자열, 극단적으로 긴 문자열 등)이면 모델이 비정상 값을 출력할 수 있습니다. 셋째, 이상치(outlier)를 탐지합니다. 대부분의 벡터와 현저히 다른 벡터가 있다면, 해당 텍스트의 전처리에 문제가 있었을 가능성이 높습니다.

이상치 탐지의 간단한 방법은 벡터의 노름(norm), 즉 벡터의 크기를 확인하는 것입니다. 정상적인 임베딩은 노름이 일정 범위 안에 분포합니다. 이 범위에서 크게 벗어나는 벡터가 있다면 원본 텍스트를 확인해 봐야 합니다.

더 정교한 방법으로는 알려진 유사 문서 쌍의 코사인 유사도가 기대 범위(예: 0.7 이상) 안에 들어오는지, 알려진 비유사 문서 쌍의 유사도가 기대 범위(예: 0.3 이하) 안에 들어오는지를 확인하는 것이 있습니다. 이 검증은 3.1.1에서 다룬 코사인 유사도를 직접 활용하는 것입니다.

정리하면, 임베딩 파이프라인은 텍스트 전처리, 배치 임베딩 생성, 캐싱, 갱신 정책, 품질 검증의 단계로 구성됩니다. 전처리에서는 유니코드 정규화와 클리닝으로 입력 품질을 보장하고, 배치 처리로 대량의 텍스트를 효율적으로 임베딩합니다. 캐싱으로 동일 텍스트의 중복 처리를 방지하고, 문서 변경이나 모델 교체에 대응하는 갱신 정책을 수립합니다. 품질 검증으로 비정상 벡터를 사전에 걸러내어, 벡터 데이터베이스에 저장되는 임베딩의 신뢰성을 확보합니다.

다음 단원인 3.2.1에서는 벡터 데이터베이스의 개념과 필요성을 다룹니다. 생성된 임베딩 벡터를 대규모로 저장하고 빠르게 검색하기 위한 전용 데이터베이스의 동작 원리를 살펴봅니다.

이 단원을 마치면 텍스트 데이터를 임베딩으로 변환하는 파이프라인을 설계하고 구축할 수 있습니다.
