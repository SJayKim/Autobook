# 6.1.2 vLLM으로 모델 서빙하기

오픈소스 대규모 언어 모델을 학습하거나 다운로드한 뒤, 이를 실제 서비스에서 사용하려면 모델을 메모리에 올려 놓고 외부 요청을 받아 응답을 생성하는 서빙 과정이 필요합니다. 6.1.1에서 서빙 아키텍처의 구성요소와 레이턴시 지표를 살펴보았는데, 이 단원에서는 그 지식을 바탕으로 실제 서빙 프레임워크 중 하나인 vLLM을 다룹니다. vLLM은 LLM 추론의 핵심 병목인 메모리 관리 문제를 독창적인 방식으로 해결하여, 적은 GPU 자원으로도 높은 처리량을 달성할 수 있게 설계된 오픈소스 서빙 엔진입니다.

LLM 서빙에서 메모리 관리가 왜 중요한 문제인지부터 짚겠습니다. 1.2.3에서 다룬 것처럼, 모델은 추론 과정에서 Prefill 단계를 거쳐 KV 캐시를 생성하고, Decode 단계에서 이 KV 캐시를 참조하며 토큰을 하나씩 생성합니다. KV 캐시는 이전에 계산한 Key와 Value 벡터를 메모리에 보관하여 중복 계산을 피하는 장치입니다. 문제는 이 KV 캐시가 차지하는 GPU 메모리가 상당히 크다는 점입니다. 모델의 크기가 커지고 컨텍스트 윈도우가 길어질수록 KV 캐시의 메모리 사용량도 비례하여 증가합니다.

서빙 환경에서는 여러 사용자의 요청을 동시에 처리해야 합니다. 각 요청마다 별도의 KV 캐시가 필요하므로, 동시에 처리할 수 있는 요청 수는 GPU 메모리 용량에 직접적으로 제한됩니다. 기존 서빙 시스템들은 각 요청에 대해 KV 캐시가 사용할 최대 메모리를 미리 연속된 공간으로 할당했습니다. 그런데 요청마다 생성하는 토큰 수가 다르기 때문에, 실제로 사용하지 않는 메모리가 남게 됩니다. 예를 들어 최대 2,048 토큰을 생성할 수 있도록 메모리를 할당했는데 실제로는 200 토큰만 생성하고 끝나면, 나머지 1,848 토큰분의 메모리 공간이 낭비됩니다. 이 낭비가 누적되면 GPU 메모리가 충분히 남아 있는데도 새 요청을 받지 못하는 상황이 발생합니다.

vLLM은 이 메모리 낭비 문제를 해결하기 위해 만들어졌습니다. UC 버클리 연구팀이 2023년에 공개한 이 프로젝트는, 운영체제의 가상 메모리 관리 기법에서 영감을 받아 **PagedAttention**이라는 새로운 어텐션 메커니즘을 도입했습니다. PagedAttention 덕분에 vLLM은 기존 시스템 대비 처리량을 크게 높일 수 있었고, 현재 LLM 서빙 분야에서 가장 널리 사용되는 오픈소스 프레임워크 중 하나로 자리 잡았습니다.

vLLM의 전체 구조를 먼저 개괄하겠습니다. vLLM은 크게 세 부분으로 나뉩니다. 첫째는 외부 요청을 받아들이는 **API 서버**입니다. 이 서버는 HTTP 엔드포인트를 열어 클라이언트의 추론 요청을 수신합니다. 둘째는 요청을 스케줄링하고 배칭하는 **스케줄러**입니다. 스케줄러는 여러 요청을 묶어서 GPU에 한꺼번에 보내는 역할을 합니다. 셋째는 실제 모델 추론을 수행하는 **워커(Worker)**입니다. 워커는 GPU에서 모델을 실행하고, KV 캐시를 관리하며, 토큰을 생성합니다.

```
클라이언트 요청
      |
      v
+-----------+
| API 서버  |  <-- HTTP 엔드포인트 (OpenAI 호환)
+-----------+
      |
      v
+-----------+
|  스케줄러  |  <-- 요청 배칭, 우선순위 결정
+-----------+
      |
      v
+-----------+
|   워커    |  <-- GPU에서 모델 실행, KV 캐시 관리
+-----------+
```

API 서버는 요청이 들어오면 이를 스케줄러에 전달합니다. 스케줄러는 현재 처리 중인 요청들의 상태와 남은 GPU 메모리를 고려하여, 새 요청을 기존 배치에 추가할지 대기시킬지 결정합니다. vLLM의 스케줄러는 **연속 배칭(continuous batching)**을 지원합니다. 연속 배칭이란, 배치 안의 한 요청이 생성을 마치면 그 자리에 바로 새 요청을 넣어 GPU가 쉬지 않도록 하는 방식입니다. 기존의 정적 배칭에서는 배치 안의 모든 요청이 끝날 때까지 기다린 뒤에야 새 배치를 시작했으므로, 짧은 요청이 긴 요청을 기다리느라 GPU가 낭비되었습니다. 연속 배칭은 이 비효율을 제거합니다.

이제 vLLM의 핵심인 PagedAttention을 상세히 살펴보겠습니다. PagedAttention을 이해하려면 먼저 운영체제의 가상 메모리 개념을 간단히 알아야 합니다. 컴퓨터의 운영체제는 프로그램마다 연속된 메모리 공간을 제공하는 것처럼 보이게 하지만, 실제 물리 메모리에서는 데이터가 여러 조각으로 흩어져 저장됩니다. 이때 메모리를 일정 크기의 **페이지(page)**라는 단위로 나누어 관리합니다. 프로그램이 연속된 주소 100번지부터 200번지를 사용한다고 하더라도, 실제로는 100~149번지가 물리 메모리의 한 곳에, 150~200번지가 전혀 다른 곳에 저장될 수 있습니다. 운영체제가 이 매핑을 관리하므로 프로그램은 이를 의식할 필요가 없습니다.

PagedAttention은 이와 동일한 원리를 KV 캐시에 적용합니다. 기존 시스템에서는 한 요청의 KV 캐시를 GPU 메모리의 연속된 공간에 통째로 할당했습니다. PagedAttention은 KV 캐시를 **블록(block)**이라는 고정 크기 단위로 쪼갭니다. 한 블록은 정해진 수의 토큰에 해당하는 Key와 Value 벡터를 담습니다. 예를 들어 블록 크기가 16이면, 한 블록에 16개 토큰분의 KV 데이터가 들어갑니다.

이 블록들은 GPU 메모리에서 연속된 위치에 있을 필요가 없습니다. vLLM은 블록 테이블이라는 매핑 구조를 통해, 각 요청의 논리적 블록 번호가 실제 물리 메모리의 어느 위치에 저장되어 있는지를 추적합니다. 새 토큰이 생성되면 현재 블록에 여유가 있으면 그 블록에 추가하고, 블록이 가득 차면 빈 블록 하나만 새로 할당합니다. 요청이 끝나면 해당 블록들을 즉시 반환하여 다른 요청이 재사용할 수 있게 합니다.

```
[기존 방식: 연속 할당]

요청 A: |====KV 캐시====|----미사용 공간----|
요청 B: |==KV 캐시==|--------미사용 공간--------|
         ^                 ^
         실제 사용          낭비 (반환 불가)

[PagedAttention: 블록 단위 할당]

물리 블록:  [B1][B2][B3][B4][B5][B6][B7][B8] ...

요청 A 블록 테이블: 논리 0 -> B3, 논리 1 -> B7, 논리 2 -> B1
요청 B 블록 테이블: 논리 0 -> B5, 논리 1 -> B2

빈 블록 풀:  [B4][B6][B8] ...  <-- 새 요청이나 토큰 추가에 사용
```

이 다이어그램에서 볼 수 있듯이, 기존 방식에서는 각 요청에 최대 길이만큼의 연속된 공간을 미리 잡아 두어야 했고, 실제 사용량보다 할당량이 크면 그 차이는 다른 요청이 쓸 수 없었습니다. PagedAttention은 필요한 만큼만 블록을 하나씩 할당하므로 이런 낭비가 거의 사라집니다. vLLM 프로젝트의 측정에 따르면, PagedAttention을 통해 KV 캐시 메모리의 낭비를 4% 이하로 줄일 수 있었습니다. 메모리 낭비가 줄어든 만큼 동시에 처리할 수 있는 요청 수가 늘어나고, 결과적으로 처리량이 크게 향상됩니다.

PagedAttention이 어텐션 계산을 수행하는 과정도 살펴보겠습니다. 일반적인 어텐션에서는 한 요청의 모든 Key, Value가 연속 메모리에 있다고 가정하고 행렬 연산을 수행합니다. PagedAttention에서는 Key와 Value가 여러 블록에 흩어져 있으므로, 블록 테이블을 참조하여 필요한 블록들을 찾아 가며 어텐션을 계산합니다. 구체적으로는 새 토큰의 Query 벡터를 각 블록에 저장된 Key 벡터와 곱하여 어텐션 스코어를 구하고, 이 스코어를 Value 벡터와 곱하여 최종 출력을 만듭니다. 블록이 흩어져 있더라도 GPU의 병렬 처리 능력을 활용하여 이 계산을 효율적으로 수행합니다.

이제 vLLM을 실제로 설치하고 서버를 실행하는 방법을 살펴보겠습니다. vLLM은 Python 패키지로 제공되며, pip으로 설치할 수 있습니다.

```bash
pip install vllm
```

이 명령은 vLLM과 함께 필요한 의존성(PyTorch, Transformers 등)을 설치합니다. vLLM은 NVIDIA GPU와 CUDA를 필요로 하므로, 설치 전에 CUDA 호환 GPU가 장착되어 있고 CUDA 드라이버가 설치되어 있어야 합니다.

vLLM을 서버 모드로 실행하는 가장 기본적인 명령은 다음과 같습니다.

```bash
vllm serve meta-llama/Llama-3.1-8B-Instruct
```

이 명령이 하는 일을 단계별로 설명하겠습니다. `vllm serve`는 vLLM의 OpenAI 호환 API 서버를 시작하는 명령입니다. 그 뒤에 오는 `meta-llama/Llama-3.1-8B-Instruct`는 서빙할 모델의 이름입니다. 이 이름은 Hugging Face 모델 허브의 모델 식별자입니다. vLLM은 이 식별자를 사용하여 모델 가중치를 자동으로 다운로드하고 GPU 메모리에 로드합니다. 서버가 시작되면 기본적으로 8000번 포트에서 HTTP 요청을 대기합니다.

서버를 실행할 때 다양한 옵션을 지정할 수 있습니다. 자주 사용하는 설정들을 하나씩 살펴보겠습니다.

```bash
vllm serve meta-llama/Llama-3.1-8B-Instruct \
    --host 0.0.0.0 \
    --port 8000 \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.9 \
    --dtype float16
```

`--host 0.0.0.0`은 서버가 모든 네트워크 인터페이스에서 접속을 허용하도록 합니다. 기본값은 localhost이므로, 외부에서 접근해야 할 때 이 옵션을 지정합니다. `--port 8000`은 서버가 사용할 포트 번호입니다. `--max-model-len 4096`은 모델이 처리할 수 있는 최대 시퀀스 길이를 4,096 토큰으로 제한합니다. 이 값을 줄이면 KV 캐시가 사용하는 메모리가 줄어들어, 동시에 더 많은 요청을 처리할 수 있습니다. 반대로 긴 입력을 처리해야 한다면 이 값을 높여야 합니다. `--gpu-memory-utilization 0.9`는 GPU 메모리의 90%까지 사용하겠다는 의미입니다. 나머지 10%는 모델 가중치 외의 임시 연산에 필요한 여유 공간으로 남겨 둡니다. `--dtype float16`은 모델의 가중치와 연산을 16비트 부동소수점으로 수행하도록 지정합니다. 32비트 대비 메모리 사용량이 절반으로 줄어들고 연산 속도도 빨라집니다.

vLLM 서버가 시작되면 OpenAI의 API와 호환되는 엔드포인트를 제공합니다. 이것이 vLLM의 큰 장점 중 하나입니다. 기존에 OpenAI API를 사용하던 코드가 있다면, 서버 주소만 바꾸면 거의 수정 없이 오픈소스 모델로 전환할 수 있습니다.

vLLM이 제공하는 주요 엔드포인트는 두 가지입니다. `/v1/completions`는 텍스트 완성(completion) 엔드포인트로, 주어진 프롬프트에 이어지는 텍스트를 생성합니다. `/v1/chat/completions`는 채팅 엔드포인트로, 대화 형식의 메시지를 주고받습니다. 실무에서는 대부분 채팅 엔드포인트를 사용합니다.

채팅 엔드포인트에 요청을 보내는 예시를 살펴보겠습니다.

```bash
curl http://localhost:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "model": "meta-llama/Llama-3.1-8B-Instruct",
        "messages": [
            {"role": "system", "content": "You are a helpful assistant."},
            {"role": "user", "content": "KV 캐시가 무엇인가요?"}
        ],
        "temperature": 0.7,
        "max_tokens": 256
    }'
```

이 요청의 각 필드를 설명하겠습니다. `"model"`은 서빙 중인 모델의 이름입니다. vLLM 서버를 시작할 때 지정한 모델명과 일치해야 합니다. `"messages"`는 대화 이력을 배열로 전달합니다. 각 메시지는 `"role"`과 `"content"`로 구성됩니다. `"role"`이 `"system"`이면 모델의 행동 지침을 설정하는 시스템 메시지이고, `"user"`이면 사용자의 질문입니다. `"temperature"`는 1.2.3에서 다룬 생성 전략 파라미터로, 값이 낮을수록 결정적인 응답을, 높을수록 다양한 응답을 생성합니다. `"max_tokens"`는 생성할 최대 토큰 수입니다.

서버의 응답은 다음과 같은 형태로 돌아옵니다.

```json
{
    "id": "cmpl-abc123",
    "object": "chat.completion",
    "model": "meta-llama/Llama-3.1-8B-Instruct",
    "choices": [
        {
            "index": 0,
            "message": {
                "role": "assistant",
                "content": "KV 캐시는 ..."
            },
            "finish_reason": "stop"
        }
    ],
    "usage": {
        "prompt_tokens": 25,
        "completion_tokens": 120,
        "total_tokens": 145
    }
}
```

`"choices"` 배열에 모델의 응답이 담깁니다. `"finish_reason"`이 `"stop"`이면 모델이 자연스럽게 생성을 마친 것이고, `"length"`이면 `max_tokens`에 도달하여 강제로 중단된 것입니다. `"usage"`는 이 요청에서 사용된 토큰 수를 보여 줍니다. 입력 토큰(`prompt_tokens`)과 출력 토큰(`completion_tokens`)의 합이 전체 토큰(`total_tokens`)입니다.

Python에서 OpenAI 클라이언트 라이브러리를 사용하여 vLLM 서버에 접속하는 방법도 있습니다.

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="dummy"
)

response = client.chat.completions.create(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[
        {"role": "user", "content": "KV 캐시가 무엇인가요?"}
    ],
    temperature=0.7,
    max_tokens=256
)

print(response.choices[0].message.content)
```

`base_url`을 vLLM 서버의 주소로 지정하고, `api_key`에는 임의의 문자열을 넣습니다. vLLM 서버는 기본적으로 API 키 인증을 강제하지 않으므로 아무 값이나 넣어도 됩니다. 나머지 코드는 OpenAI API를 사용할 때와 완전히 동일합니다. 이 호환성 덕분에 기존 OpenAI 기반 애플리케이션을 오픈소스 모델 기반으로 전환할 때 코드 수정이 최소화됩니다.

다음으로 **텐서 병렬(Tensor Parallelism)** 설정을 살펴보겠습니다. 텐서 병렬이란, 하나의 모델을 여러 GPU에 나누어 올리는 방식입니다. 모델의 크기가 단일 GPU의 메모리 용량을 초과하거나, 추론 속도를 높이고 싶을 때 사용합니다.

왜 텐서 병렬이 필요한지 구체적인 수치로 설명하겠습니다. 70억 파라미터 모델을 float16으로 로드하면 약 14GB의 GPU 메모리가 필요합니다. 파라미터 하나가 16비트, 즉 2바이트를 차지하므로 70억 곱하기 2바이트가 약 14GB입니다. 여기에 KV 캐시와 연산 임시 메모리까지 더하면 실제 필요한 메모리는 이보다 더 큽니다. 700억 파라미터 모델이라면 가중치만 약 140GB가 필요하여, 80GB짜리 GPU 하나로는 모델을 올릴 수조차 없습니다. 이때 텐서 병렬을 사용하여 모델을 여러 GPU에 분산합니다.

vLLM에서 텐서 병렬을 설정하는 방법은 간단합니다.

```bash
vllm serve meta-llama/Llama-3.1-70B-Instruct \
    --tensor-parallel-size 4
```

`--tensor-parallel-size 4`는 모델을 4개의 GPU에 나누어 올리겠다는 의미입니다. vLLM은 모델의 각 층에 있는 행렬을 4등분하여 각 GPU에 배치합니다. 추론 시 각 GPU가 자신이 담당하는 부분만 계산한 뒤, 결과를 모아 합칩니다. 이 과정에서 GPU 간 통신이 발생하므로, GPU 사이의 연결 대역폭이 성능에 영향을 줍니다. NVLink처럼 고대역폭 연결이 있는 환경에서 텐서 병렬의 효과가 극대화됩니다.

텐서 병렬 크기를 설정할 때 주의할 점이 있습니다. 텐서 병렬 크기는 대부분의 모델에서 어텐션 헤드 수의 약수여야 합니다. 예를 들어 어텐션 헤드가 32개인 모델이라면, 텐서 병렬 크기를 1, 2, 4, 8, 16, 32 중에서 선택해야 합니다. 3이나 5처럼 헤드 수의 약수가 아닌 값을 지정하면 오류가 발생합니다. 또한 텐서 병렬 크기를 늘린다고 항상 성능이 비례하여 좋아지는 것은 아닙니다. GPU 간 통신 오버헤드가 있으므로, GPU 수를 두 배로 늘려도 처리량이 두 배가 되지는 않습니다. 단일 GPU로 모델을 올릴 수 있다면 텐서 병렬 없이 실행하는 것이 가장 효율적입니다.

vLLM의 장점과 단점, 그리고 적합한 사용 사례를 정리하겠습니다. vLLM의 주요 장점은 세 가지입니다. 첫째, PagedAttention을 통한 효율적인 메모리 관리로 높은 처리량을 달성합니다. 같은 GPU 환경에서 더 많은 동시 요청을 처리할 수 있습니다. 둘째, OpenAI 호환 API를 기본 제공하여 기존 코드의 전환 비용이 낮습니다. 셋째, Hugging Face 모델 허브의 다양한 모델을 지원하며, 설치와 실행이 간단합니다.

단점도 있습니다. 첫째, NVIDIA GPU에 대한 의존도가 높습니다. AMD GPU나 CPU 환경에서의 지원은 제한적입니다. 둘째, 모델을 처음 로드하는 데 시간이 걸립니다. 특히 대형 모델은 가중치를 GPU에 올리는 데 수 분이 소요될 수 있습니다. 셋째, NVIDIA의 TensorRT-LLM과 비교했을 때, 특정 NVIDIA 하드웨어에 최적화된 커널 수준의 성능 튜닝에서는 차이가 날 수 있습니다.

vLLM이 적합한 사용 사례는 다음과 같습니다. 오픈소스 모델을 서빙하면서 높은 처리량이 필요한 경우, OpenAI API와 호환되는 인터페이스로 자체 모델 서빙 인프라를 구축하려는 경우, 그리고 빠르게 프로토타입을 만들어 모델의 서빙 성능을 검증하려는 경우에 vLLM이 적합합니다. 반면, 최대 수준의 레이턴시 최적화가 필요하거나 NVIDIA 전용 하드웨어 가속을 극한까지 활용해야 하는 경우에는 TensorRT-LLM(6.1.4에서 다룹니다)이 더 적합할 수 있습니다.

정리하면, vLLM은 KV 캐시 메모리의 낭비를 해결하기 위해 PagedAttention을 도입한 오픈소스 LLM 서빙 엔진입니다. PagedAttention은 운영체제의 가상 메모리처럼 KV 캐시를 고정 크기 블록으로 나누어 관리함으로써 메모리 활용률을 높이고, 연속 배칭과 결합하여 높은 처리량을 달성합니다. vLLM 서버는 OpenAI 호환 API를 제공하므로, 기존 OpenAI 클라이언트 코드를 최소한의 수정으로 재사용할 수 있습니다. 텐서 병렬을 통해 단일 GPU에 담기지 않는 대형 모델도 여러 GPU에 나누어 서빙할 수 있습니다.

다음 단원인 6.1.3에서는 Hugging Face가 제공하는 또 다른 서빙 프레임워크인 TGI(Text Generation Inference)를 다룹니다.

이 단원을 마치면 vLLM의 PagedAttention 메커니즘이 메모리 문제를 해결하는 원리를 설명하고, vLLM 서버를 실행하여 OpenAI 호환 API로 모델을 서빙할 수 있습니다.
