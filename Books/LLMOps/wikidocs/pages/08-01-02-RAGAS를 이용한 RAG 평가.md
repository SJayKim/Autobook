# 8.1.2 RAGAS를 이용한 RAG 평가

8.1.1에서 LLM 평가의 주요 지표와 벤치마크를 살펴보았습니다. Perplexity, BLEU, ROUGE, BERTScore 같은 자동 평가 지표와 LLM-as-Judge 방식은 LLM 출력의 전반적인 품질을 측정하는 데 유용합니다. 그런데 RAG(Retrieval-Augmented Generation) 파이프라인을 평가할 때는 일반적인 LLM 평가와는 다른 차원의 질문이 추가됩니다. "검색된 문서가 질문과 관련이 있는가?", "모델이 검색된 문서에 충실하게 답변했는가?", "검색 결과에 필요한 정보가 빠짐없이 포함되어 있는가?" 같은 질문입니다. 4.2.2에서 다룬 다양한 RAG 패턴을 적용하더라도, 이런 질문에 답하지 못하면 파이프라인의 어디가 잘 동작하고 어디가 문제인지 파악할 수 없습니다.

**RAGAS(Retrieval Augmented Generation Assessment)**는 RAG 파이프라인의 품질을 정량적으로 평가하기 위해 설계된 프레임워크입니다. RAGAS가 다른 평가 도구와 구별되는 점은, RAG 파이프라인을 구성하는 두 단계, 검색(retrieval)과 생성(generation)을 분리하여 각각 독립적으로 평가한다는 것입니다. 이 분리가 중요한 이유는, RAG의 최종 출력이 나쁠 때 원인이 검색 단계에 있는지 생성 단계에 있는지를 구분해야 개선 방향을 정할 수 있기 때문입니다. 검색이 잘못되면 아무리 생성 모델이 좋아도 답변이 부정확해지고, 검색은 잘 되었는데 생성 단계에서 검색 결과를 무시하면 역시 답변이 부정확해집니다. RAGAS는 네 가지 핵심 지표를 통해 이 두 단계를 정밀하게 진단합니다.

RAGAS 평가에 필요한 입력은 세 가지입니다. 사용자의 **질문(question)**, RAG 파이프라인이 검색한 **컨텍스트(contexts)**, 그리고 모델이 생성한 **답변(answer)**입니다. 일부 지표는 사람이 작성한 **정답(ground truth)**도 필요합니다. 이 네 요소를 바탕으로 각 지표가 계산됩니다.

첫 번째 지표는 **Faithfulness(충실도)**입니다. Faithfulness는 생성된 답변이 검색된 컨텍스트에 얼마나 충실한지를 측정합니다. 8.2.1에서 다룰 충실도 할루시네이션과 직접적으로 관련된 지표입니다. 모델이 컨텍스트에 없는 내용을 지어내면 Faithfulness가 낮아집니다.

Faithfulness의 측정 과정을 단계별로 살펴보겠습니다. 먼저 생성된 답변을 개별 주장(claim) 단위로 분해합니다. "서울의 인구는 약 950만 명이며, 대한민국의 수도입니다"라는 답변은 "서울의 인구는 약 950만 명이다"와 "서울은 대한민국의 수도이다"라는 두 개의 주장으로 분해됩니다. 다음으로 각 주장이 검색된 컨텍스트에 의해 뒷받침되는지를 판별합니다. 컨텍스트에 해당 내용이 있으면 뒷받침되는 것이고, 없으면 뒷받침되지 않는 것입니다. 최종적으로, 뒷받침되는 주장의 수를 전체 주장의 수로 나누어 Faithfulness 점수를 산출합니다.

```
Faithfulness 측정 과정

답변: "서울의 인구는 약 950만 명이며, 대한민국의 수도입니다."

1단계: 주장 분해
  주장 1: "서울의 인구는 약 950만 명이다"
  주장 2: "서울은 대한민국의 수도이다"

2단계: 컨텍스트 대조
  컨텍스트: "서울특별시의 인구는 약 950만 명이다.
             서울은 한강을 중심으로 발달한 도시이다."

  주장 1: 컨텍스트에 근거 있음 --> 뒷받침됨
  주장 2: 컨텍스트에 명시 없음 --> 뒷받침 안 됨

3단계: 점수 계산
  Faithfulness = 뒷받침된 주장 수 / 전체 주장 수
               = 1 / 2 = 0.5
```

Faithfulness가 낮다는 것은 모델이 컨텍스트를 벗어난 답변을 생성하고 있다는 뜻입니다. 이 경우 개선 방향은 생성 단계에 있습니다. 프롬프트에 "제공된 문서만을 근거로 답변하세요"와 같은 지시를 강화하거나, 8.2.1에서 다룰 할루시네이션 완화 전략을 적용합니다.

두 번째 지표는 **Answer Relevance(답변 관련성)**입니다. Answer Relevance는 생성된 답변이 원래 질문에 얼마나 적합한지를 측정합니다. 답변이 사실적으로 정확하더라도, 질문과 관련 없는 내용이면 좋은 답변이 아닙니다. "파이썬에서 리스트를 정렬하는 방법은?"이라는 질문에 "파이썬은 1991년에 만들어진 언어입니다"라고 답하면, 사실이지만 질문과 무관합니다.

Answer Relevance의 측정 방식은 독특합니다. 생성된 답변을 바탕으로 LLM이 역으로 질문을 여러 개 생성합니다. 생성된 역질문들이 원래 질문과 의미적으로 얼마나 유사한지를 임베딩 코사인 유사도로 계산합니다. 답변이 원래 질문에 적합하다면, 그 답변으로부터 생성된 역질문은 원래 질문과 의미가 비슷할 것입니다. 반대로 답변이 질문과 무관하다면, 역질문도 원래 질문과 동떨어진 내용이 될 것입니다.

```
Answer Relevance 측정 과정

원래 질문: "파이썬에서 리스트를 정렬하는 방법은?"
답변: "sorted() 함수를 사용하거나, 리스트의 sort() 메서드를 사용합니다."

1단계: 답변으로부터 역질문 생성 (LLM 사용)
  역질문 1: "파이썬에서 리스트를 정렬하려면 어떻게 하나요?"
  역질문 2: "sorted 함수와 sort 메서드는 어떤 것인가요?"
  역질문 3: "파이썬 리스트 정렬 방법을 알려주세요."

2단계: 원래 질문과 역질문의 임베딩 유사도 계산
  유사도 1: 0.95 (매우 유사)
  유사도 2: 0.72 (관련 있음)
  유사도 3: 0.93 (매우 유사)

3단계: 평균 유사도 산출
  Answer Relevance = (0.95 + 0.72 + 0.93) / 3 = 0.867
```

Answer Relevance가 낮다면, 모델이 질문의 의도를 제대로 파악하지 못하고 있거나, 불필요한 정보를 많이 포함하고 있다는 뜻입니다. 이 경우 프롬프트에서 질문의 초점을 더 명확히 지시하거나, 답변의 길이를 제한하는 방향으로 개선합니다.

세 번째 지표는 **Context Precision(컨텍스트 정밀도)**입니다. 여기서부터는 생성 단계가 아니라 검색 단계를 평가합니다. Context Precision은 검색된 컨텍스트 중에서 질문에 답하는 데 실제로 유용한 문서가 상위에 위치하는지를 측정합니다.

RAG 파이프라인에서 검색 단계는 보통 여러 개의 문서 청크를 반환합니다. 이 중 일부는 질문과 관련이 높고, 일부는 관련이 낮을 수 있습니다. Context Precision은 관련 문서가 검색 결과의 상위에 집중되어 있을수록 높은 점수를 부여합니다. 이 지표를 계산하려면 각 검색 결과가 정답을 도출하는 데 유용한지를 판별해야 하므로, 사람이 작성한 정답(ground truth)이 필요합니다.

```
Context Precision 예시

질문: "대한민국의 수도는?"
정답: "서울"
검색된 컨텍스트 (순서대로):
  1. "서울특별시는 대한민국의 수도이다" --> 관련 있음
  2. "부산광역시는 대한민국 제2의 도시이다" --> 관련 없음
  3. "대한민국은 1948년에 건국되었다" --> 관련 없음
  4. "서울의 인구는 약 950만 명이다" --> 관련 있음

관련 문서가 1번과 4번 위치에 있음
--> 1번은 최상위이므로 좋고, 4번은 아래쪽이므로 아쉬움
--> 관련 문서가 상위에 집중될수록 Context Precision이 높음
```

Context Precision이 낮다면, 검색 단계에서 관련 문서를 상위로 올리는 랭킹 품질이 떨어진다는 뜻입니다. 3.1에서 다룬 임베딩 모델의 성능을 개선하거나, 4.1.2에서 다룬 검색 전략을 조정하는 방향으로 개선합니다.

네 번째 지표는 **Context Recall(컨텍스트 재현율)**입니다. Context Recall은 정답에 포함된 정보가 검색된 컨텍스트에 얼마나 포함되어 있는지를 측정합니다. 정답을 구성하는 문장들 중에서, 검색된 컨텍스트에 의해 뒷받침되는 문장의 비율입니다.

Context Recall이 중요한 이유를 예로 들어 보겠습니다. 질문이 "비타민 C의 효능을 설명하세요"이고, 정답이 "비타민 C는 면역력 강화, 항산화 작용, 콜라겐 합성 촉진에 효과가 있습니다"라고 합시다. 검색된 컨텍스트에 면역력 강화와 항산화 작용에 대한 문서만 있고 콜라겐 합성에 대한 문서가 없다면, 정답의 3개 요소 중 2개만 검색으로 커버된 것이므로 Context Recall은 2/3, 약 0.67입니다.

```
Context Recall 측정 과정

정답: "비타민 C는 면역력 강화, 항산화 작용, 콜라겐 합성 촉진에 효과가 있다."

정답 분해:
  문장 1: "비타민 C는 면역력을 강화한다"
  문장 2: "비타민 C는 항산화 작용을 한다"
  문장 3: "비타민 C는 콜라겐 합성을 촉진한다"

검색된 컨텍스트 대조:
  문장 1: 컨텍스트에 "비타민 C가 면역 체계를 돕는다" 있음 --> 뒷받침됨
  문장 2: 컨텍스트에 "항산화 물질로서 기능한다" 있음 --> 뒷받침됨
  문장 3: 컨텍스트에 관련 내용 없음 --> 뒷받침 안 됨

Context Recall = 뒷받침된 문장 수 / 전체 문장 수
              = 2 / 3 = 0.667
```

Context Recall이 낮다면, 검색 단계에서 필요한 정보를 담은 문서를 놓치고 있다는 뜻입니다. 문서 청크 크기를 조정하거나, 검색 쿼리를 확장하거나, 색인 대상 문서를 보강하는 방향으로 개선합니다.

네 가지 지표의 역할을 종합하면 다음과 같습니다.

```
RAGAS 지표 종합 정리

+---------------------+-----------+------------------+--------------------+
| 지표                | 평가 대상 | 측정 내용        | 낮을 때 개선 방향  |
+---------------------+-----------+------------------+--------------------+
| Faithfulness        | 생성      | 답변이 컨텍스트에| 프롬프트 강화,     |
|                     |           | 충실한가         | 할루시네이션 완화  |
+---------------------+-----------+------------------+--------------------+
| Answer Relevance    | 생성      | 답변이 질문에    | 프롬프트 명확화,   |
|                     |           | 적합한가         | 답변 길이 조절     |
+---------------------+-----------+------------------+--------------------+
| Context Precision   | 검색      | 관련 문서가      | 랭킹 알고리즘,     |
|                     |           | 상위에 있는가    | 임베딩 모델 개선   |
+---------------------+-----------+------------------+--------------------+
| Context Recall      | 검색      | 필요한 정보가    | 청크 크기 조정,    |
|                     |           | 모두 검색되었는가| 문서 보강          |
+---------------------+-----------+------------------+--------------------+
```

이제 **RAGAS 평가 파이프라인 구축** 방법을 살펴보겠습니다. RAGAS는 파이썬 라이브러리로 제공되며, 몇 줄의 코드로 평가를 실행할 수 있습니다. 평가 파이프라인의 기본 흐름은 다음과 같습니다. 평가용 데이터셋을 준비하고, 평가 지표를 선택하고, 평가를 실행하여 결과를 분석합니다.

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from datasets import Dataset

# 1. 평가 데이터셋 준비
eval_data = {
    "question": [
        "대한민국의 수도는?",
        "비타민 C의 효능은?",
    ],
    "answer": [
        "대한민국의 수도는 서울입니다.",
        "비타민 C는 면역력 강화와 항산화 작용에 효과가 있습니다.",
    ],
    "contexts": [
        ["서울특별시는 대한민국의 수도이다."],
        ["비타민 C는 면역 체계를 돕는 항산화 물질이다."],
    ],
    "ground_truth": [
        "서울",
        "비타민 C는 면역력 강화, 항산화 작용, 콜라겐 합성 촉진에 효과가 있다.",
    ],
}

dataset = Dataset.from_dict(eval_data)

# 2. 평가 실행
results = evaluate(
    dataset,
    metrics=[
        faithfulness,
        answer_relevancy,
        context_precision,
        context_recall,
    ],
)

# 3. 결과 확인
print(results)
```

이 코드의 각 부분을 설명하겠습니다. 1단계에서 평가 데이터셋을 딕셔너리로 준비합니다. question은 사용자 질문 목록, answer는 RAG 파이프라인이 생성한 답변 목록, contexts는 검색된 컨텍스트 목록(각 질문당 여러 문서가 리스트로 들어감), ground_truth는 사람이 작성한 정답 목록입니다. 이 데이터를 Hugging Face의 Dataset 객체로 변환합니다.

2단계에서 evaluate 함수에 데이터셋과 측정할 지표 목록을 전달합니다. RAGAS는 내부적으로 LLM을 사용하여 주장 분해, 역질문 생성, 컨텍스트 대조 등의 판단을 수행합니다. 기본적으로 OpenAI의 API를 사용하지만, 다른 LLM을 지정할 수도 있습니다.

3단계에서 결과를 출력하면 각 지표의 점수가 0에서 1 사이의 값으로 나옵니다. 1에 가까울수록 해당 측면의 품질이 높다는 뜻입니다.

실무에서 RAGAS 평가 파이프라인을 운영할 때 고려할 사항이 있습니다. 평가 데이터셋의 크기가 충분해야 합니다. 몇 건의 데이터로는 통계적으로 유의미한 결론을 내리기 어렵습니다. 최소 수십 건에서 수백 건의 평가 데이터를 준비하는 것이 바람직합니다. 또한 RAGAS 내부에서 LLM을 호출하므로 API 비용이 발생합니다. 평가 빈도와 데이터셋 크기를 비용과 균형 맞추어 결정해야 합니다. 그리고 RAGAS의 판단도 LLM에 의존하므로, 8.1.1에서 다룬 LLM-as-Judge의 편향이 동일하게 적용될 수 있습니다. RAGAS 결과를 절대적 진실로 받아들이기보다는, 파이프라인 개선의 방향을 잡는 데 활용하는 것이 적절합니다.

정리하면, RAGAS는 RAG 파이프라인의 검색 단계와 생성 단계를 분리하여 각각 평가하는 프레임워크입니다. Faithfulness는 답변이 컨텍스트에 충실한지를, Answer Relevance는 답변이 질문에 적합한지를, Context Precision은 관련 문서가 상위에 위치하는지를, Context Recall은 필요한 정보가 모두 검색되었는지를 측정합니다. 이 네 지표를 통해 RAG 파이프라인의 어느 단계에 문제가 있는지를 진단하고, 그에 맞는 개선 방향을 설정할 수 있습니다.

다음 단원인 8.1.3에서는 DeepEval을 이용한 자동 평가를 다룹니다.

이 단원을 마치면 RAGAS 프레임워크를 활용하여 RAG 파이프라인의 품질을 정량적으로 평가할 수 있습니다.
