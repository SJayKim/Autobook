# 3.1.1 텍스트 임베딩의 원리

1.2.1에서 Transformer 모델의 내부 구조를 살펴보면서, 단어가 숫자 벡터로 변환되어 모델에 입력된다는 것을 확인했습니다. 1.2.2에서는 텍스트가 토큰이라는 단위로 분리되는 과정을 다루었습니다. 이 단원에서는 그 벡터 표현, 즉 "임베딩"이라는 개념 자체에 집중합니다. LLMOps 환경에서 임베딩은 단순히 모델 내부의 중간 결과물이 아닙니다. 문서를 검색하고, 질문과 관련된 정보를 찾아내고, 유사한 텍스트끼리 묶는 등 실제 서비스의 핵심 기능을 떠받치는 기술입니다. 이 단원에서는 임베딩이 무엇인지, 두 임베딩 사이의 유사도를 어떻게 계산하는지, 그리고 임베딩 기술이 단어 수준에서 문장 수준으로 어떻게 발전해 왔는지를 다룹니다.

컴퓨터는 텍스트를 있는 그대로 이해하지 못합니다. "고양이"라는 단어를 사람은 네 발 달린 동물로 바로 떠올리지만, 컴퓨터에게 이 단어는 유니코드 문자 세 개의 나열에 불과합니다. "고양이"와 "강아지"가 의미적으로 가깝다는 것, "고양이"와 "자동차"가 의미적으로 멀다는 것을 컴퓨터가 판단하려면, 텍스트를 숫자로 바꾸되 의미 관계가 보존되는 방식으로 바꿔야 합니다.

**임베딩(embedding)**이란 텍스트를 고정된 길이의 숫자 벡터로 변환한 것입니다. 벡터란 여러 개의 숫자를 순서대로 나열한 묶음입니다. 1.2.1에서 Self-Attention을 설명할 때, 각 단어가 숫자 벡터로 변환되어 모델에 입력된다고 했습니다. 그때의 벡터가 바로 임베딩의 한 형태입니다. 예를 들어, "고양이"라는 단어를 384개의 숫자로 이루어진 벡터로 표현할 수 있습니다. 이 384개의 숫자 각각은 그 단어의 의미를 여러 측면에서 포착한 값입니다.

임베딩의 핵심 특성은, 의미가 비슷한 텍스트는 비슷한 벡터를 갖고, 의미가 다른 텍스트는 다른 벡터를 갖도록 학습된다는 점입니다. 숫자들의 묶음인 벡터를 좌표로 해석하면, 벡터는 다차원 공간의 한 점에 해당합니다. "고양이"의 벡터와 "강아지"의 벡터는 이 공간에서 가까운 위치에 놓이고, "고양이"의 벡터와 "경제학"의 벡터는 먼 위치에 놓입니다. 이 관계를 간단히 2차원으로 축소해 보면 다음과 같습니다.

```
           의미 공간 (2차원 축소 예시)

    높음 |
         |        * 강아지
         |      * 고양이
  축 2   |   * 햄스터
         |
         |                      * 자동차
         |                    * 비행기
    낮음 |________________________
         낮음                   높음
                   축 1
```

실제 임베딩은 수백에서 수천 차원이므로 이렇게 평면에 그릴 수는 없지만, 의미가 가까운 단어끼리 공간에서 군집을 이루는 원리는 동일합니다.

임베딩이 실제 서비스에서 왜 중요한지 구체적인 상황을 생각해 보겠습니다. 사용자가 "반려동물 건강 관리 방법"이라고 질문했고, 시스템에는 수만 건의 문서가 저장되어 있다고 가정합니다. 키워드 검색으로는 "반려동물", "건강", "관리", "방법"이라는 단어가 정확히 들어 있는 문서만 찾을 수 있습니다. 그런데 "강아지 예방 접종 스케줄"이라는 문서는 키워드가 하나도 일치하지 않지만, 의미적으로는 질문과 관련이 깊습니다. 임베딩을 사용하면 질문과 문서를 모두 벡터로 변환한 뒤, 벡터 사이의 거리를 비교하여 의미적으로 가까운 문서를 찾아낼 수 있습니다. 이것이 **의미적 유사도(Semantic Similarity)** 기반 검색의 기본 원리이며, 뒤에서 다룰 RAG(Retrieval-Augmented Generation)의 핵심 기술이기도 합니다.

**의미적 유사도**란 두 텍스트가 표면적인 단어 일치와 관계없이 뜻이 얼마나 가까운지를 수치로 나타낸 것입니다. "비가 많이 온다"와 "폭우가 내린다"는 겹치는 단어가 하나도 없지만 뜻은 거의 같습니다. 임베딩으로 변환한 벡터 사이의 거리를 재면 이 의미적 근접성을 숫자로 얻을 수 있습니다. 거리를 재는 방법, 즉 유사도를 계산하는 방법에는 여러 가지가 있습니다. 대표적으로 코사인 유사도, 유클리드 거리, 내적 세 가지가 있으며, 각각 다른 관점에서 벡터 사이의 관계를 측정합니다.

첫 번째 방법은 **코사인 유사도(Cosine Similarity)**입니다. 코사인 유사도는 두 벡터가 이루는 각도를 기준으로 유사도를 판단합니다. 벡터의 크기(길이)는 무시하고, 오직 방향만 비교합니다. 두 벡터가 같은 방향을 가리키면 코사인 유사도는 1이 되고, 직각을 이루면 0이 되며, 완전히 반대 방향이면 -1이 됩니다.

계산 방법을 단계별로 살펴보겠습니다. 먼저 두 벡터의 내적을 구합니다. 1.2.1에서 설명했듯이 내적이란 두 벡터의 대응하는 숫자를 각각 곱한 뒤 모두 더하는 연산입니다. 다음으로 각 벡터의 크기를 구합니다. 벡터의 크기란 각 숫자를 제곱하여 모두 더한 뒤 제곱근을 취한 값입니다. 마지막으로 내적을 두 벡터 크기의 곱으로 나눕니다.

```
코사인 유사도 = (A . B) / (|A| * |B|)

A = [1, 2, 3]
B = [2, 4, 6]
C = [3, 0, -1]

A . B = 1*2 + 2*4 + 3*6 = 2 + 8 + 18 = 28
|A| = sqrt(1 + 4 + 9) = sqrt(14)
|B| = sqrt(4 + 16 + 36) = sqrt(56)

코사인 유사도(A, B) = 28 / (sqrt(14) * sqrt(56)) = 28 / 28 = 1.0

A . C = 1*3 + 2*0 + 3*(-1) = 3 + 0 - 3 = 0
코사인 유사도(A, C) = 0 / (sqrt(14) * sqrt(10)) = 0.0
```

위 예시에서 벡터 A와 B는 방향이 완전히 같습니다. B는 A의 모든 값을 2배 한 것이므로, 크기는 다르지만 방향은 동일합니다. 코사인 유사도는 크기를 무시하고 방향만 보기 때문에 결과가 1.0입니다. 반면 A와 C는 직각을 이루므로 유사도가 0.0입니다.

코사인 유사도가 LLMOps에서 가장 널리 쓰이는 이유가 있습니다. 임베딩 벡터의 크기는 텍스트 길이나 모델 내부의 수치적 스케일에 따라 달라질 수 있는데, 코사인 유사도는 크기에 영향을 받지 않고 순수하게 의미적 방향만 비교합니다. 긴 문서의 임베딩과 짧은 질문의 임베딩을 비교할 때에도 문서 길이 차이에 의한 편향 없이 의미 유사성을 측정할 수 있습니다.

두 번째 방법은 **유클리드 거리(Euclidean Distance)**입니다. 유클리드 거리는 두 점 사이의 직선 거리를 재는 것입니다. 학교에서 배우는 두 점 사이의 거리 공식과 같은 원리입니다. 2차원에서 점 (1, 2)와 점 (4, 6) 사이의 거리를 구할 때 가로 차이의 제곱과 세로 차이의 제곱을 더한 뒤 제곱근을 취하는 것과 동일합니다. 이것을 수백 차원으로 확장한 것이 임베딩 벡터 간의 유클리드 거리입니다.

```
유클리드 거리 = sqrt( (a1-b1)^2 + (a2-b2)^2 + ... + (an-bn)^2 )

A = [1, 2, 3]
B = [2, 4, 6]
D = [1, 2, 4]

거리(A, B) = sqrt( (1-2)^2 + (2-4)^2 + (3-6)^2 )
           = sqrt( 1 + 4 + 9 )
           = sqrt(14) = 약 3.74

거리(A, D) = sqrt( (1-1)^2 + (2-2)^2 + (3-4)^2 )
           = sqrt( 0 + 0 + 1 )
           = sqrt(1) = 1.0
```

유클리드 거리는 값이 작을수록 두 벡터가 가깝다는 뜻입니다. 코사인 유사도와 반대로, 값이 클수록 멀다는 의미입니다. 유클리드 거리는 벡터의 크기 차이도 반영합니다. 앞선 예시에서 A와 B는 방향이 같지만 크기가 다르므로 유클리드 거리가 3.74로 나옵니다. 코사인 유사도에서는 1.0으로 "완전히 같다"고 판단했던 것과 대조적입니다. 따라서 벡터의 크기 자체가 중요한 정보를 담고 있는 경우에는 유클리드 거리가 더 적합할 수 있습니다.

세 번째 방법은 **내적(Dot Product, Inner Product)**입니다. 내적은 1.2.1에서 Self-Attention을 설명할 때 이미 다룬 연산입니다. 두 벡터의 대응하는 숫자를 각각 곱하여 모두 더한 값입니다.

```
내적 = a1*b1 + a2*b2 + ... + an*bn

A = [1, 2, 3]
B = [2, 4, 6]
C = [3, 0, -1]

내적(A, B) = 1*2 + 2*4 + 3*6 = 28
내적(A, C) = 1*3 + 2*0 + 3*(-1) = 0
```

내적의 값이 클수록 두 벡터가 유사하다는 뜻입니다. 내적은 코사인 유사도와 밀접한 관계에 있습니다. 코사인 유사도의 분자가 바로 내적이기 때문입니다. 차이점은 내적이 벡터의 크기도 함께 반영한다는 것입니다. 크기가 큰 벡터끼리의 내적은 방향이 같더라도 더 큰 값이 나옵니다. 최근의 임베딩 모델 중에는 출력 벡터를 미리 정규화(벡터의 크기를 1로 맞추는 것)하여 내보내는 경우가 있습니다. 이 경우 벡터의 크기가 모두 1이므로, 내적 결과가 곧 코사인 유사도와 동일해집니다. 정규화된 벡터 사이에서는 내적 계산이 코사인 유사도보다 연산이 단순하므로, 대규모 검색에서 속도 이점이 있습니다.

세 가지 유사도 측정 방법의 특성을 정리하면 다음과 같습니다.

```
측정 방법       값의 범위          벡터 크기 영향    유사할 때 값
-------------------------------------------------------------------
코사인 유사도   -1 ~ 1             무시              1에 가까움
유클리드 거리   0 ~ 무한대         반영              0에 가까움
내적            -무한대 ~ 무한대   반영              큰 양수
```

실무에서 어떤 방법을 선택할지는 사용하는 임베딩 모델과 벡터 데이터베이스의 특성에 따라 달라집니다. 대부분의 임베딩 모델은 코사인 유사도에 맞추어 학습되어 있으므로, 특별한 이유가 없다면 코사인 유사도를 기본으로 사용합니다. 정규화된 벡터를 다루는 경우에는 내적을 사용하여 동일한 결과를 더 빠르게 얻을 수 있습니다.

지금까지 임베딩이 무엇이고 유사도를 어떻게 계산하는지를 다루었습니다. 이제 임베딩 기술이 어떻게 발전해 왔는지를 살펴보겠습니다. 임베딩의 발전 과정을 이해하면, 현재 LLMOps에서 사용하는 임베딩이 왜 그 형태를 갖추게 되었는지가 분명해집니다.

초기의 텍스트 표현 방식은 **단어 임베딩(Word Embedding)**이었습니다. 단어 임베딩이 등장하기 전에는, 컴퓨터가 텍스트를 처리할 때 원-핫 인코딩(one-hot encoding)이라는 방식을 주로 사용했습니다. 원-핫 인코딩은 어휘에 포함된 단어 수만큼의 칸을 만들고, 해당 단어의 위치에만 1을 넣고 나머지는 모두 0으로 채우는 방식입니다. 어휘가 5만 개라면 하나의 단어를 표현하는 데 5만 개의 숫자가 필요하고, 그중 4만 9,999개가 0입니다. 이 방식에는 두 가지 문제가 있습니다. 첫째, 벡터가 지나치게 길고 대부분이 0이라 메모리 낭비가 심합니다. 둘째, 모든 단어 쌍의 거리가 동일합니다. "고양이"와 "강아지"의 거리나 "고양이"와 "경제학"의 거리나 똑같이 나옵니다. 의미적 관계가 전혀 반영되지 않는 것입니다.

단어 임베딩은 이 문제를 해결했습니다. 대표적인 방식인 Word2Vec은 "비슷한 맥락에서 등장하는 단어는 비슷한 의미를 가진다"는 원리를 활용합니다. 대량의 텍스트를 학습하면서, 주변에 비슷한 단어가 나타나는 단어들은 비슷한 벡터를 갖도록 훈련합니다. 그 결과 "고양이"는 100~300차원 정도의 짧은 벡터로 표현되면서도, "강아지"와 가까운 위치에 놓이게 됩니다. 원-핫 인코딩의 5만 차원에 비해 수백 분의 1 크기이면서, 의미 관계까지 담고 있는 것입니다.

단어 임베딩에는 한계가 있었습니다. 하나의 단어에 하나의 벡터만 배정되기 때문에, 같은 단어가 다른 뜻으로 쓰이는 경우를 구분하지 못합니다. "배"라는 단어는 과일, 선박, 신체 부위 등 여러 뜻이 있지만, 단어 임베딩에서는 하나의 고정된 벡터만 갖습니다. 또한 단어 하나하나의 벡터는 있지만, 문장 전체의 의미를 하나의 벡터로 표현하는 방법이 마땅하지 않았습니다. 단어 벡터들을 평균 내는 방식이 시도되기도 했지만, 어순이나 문맥에 따른 의미 변화를 포착하지 못했습니다. "사과를 먹었다"와 "먹었다 사과를"의 평균 벡터가 같아지는 것이 그 예입니다.

이 한계를 넘어선 것이 **문장 임베딩(Sentence Embedding)**입니다. 문장 임베딩은 문장이나 문단 전체를 하나의 벡터로 변환합니다. 1.2.1에서 다룬 Transformer 아키텍처의 발전이 문장 임베딩을 가능하게 만들었습니다. Transformer의 Self-Attention은 문장 안의 모든 단어 관계를 동시에 고려하므로, 문맥에 따라 같은 단어라도 다른 표현을 만들어 낼 수 있습니다. "배가 고프다"의 "배"와 "배를 타다"의 "배"는 Transformer 내부에서 서로 다른 벡터를 갖게 됩니다.

문장 임베딩 모델은 Transformer의 출력을 활용하여 문장 전체를 대표하는 하나의 벡터를 생성합니다. 대표적인 방식은 Transformer의 모든 토큰 출력 벡터를 평균 내는 것(mean pooling)입니다. 단순한 단어 벡터 평균과 달리, Transformer를 거친 뒤의 각 토큰 벡터에는 이미 문장 전체의 문맥 정보가 반영되어 있으므로, 이들의 평균도 문장의 의미를 잘 포착합니다.

문장 임베딩 모델의 발전 과정을 정리하면 다음과 같습니다.

```
발전 단계     방식                      특징
----------------------------------------------------------------------
원-핫 인코딩  어휘 크기 차원, 0과 1     의미 관계 없음, 고차원 희소 벡터
단어 임베딩   Word2Vec, GloVe 등        의미 관계 반영, 저차원 밀집 벡터
              (100~300차원)             단어당 벡터 하나 고정
문맥 임베딩   BERT, GPT 등 Transformer  같은 단어라도 문맥에 따라 다른 벡터
              기반 모델                 동음이의어 구분 가능
문장 임베딩   Sentence-BERT 등          문장 전체를 하나의 벡터로 변환
              (384~4096차원)            문장 단위 의미 비교 가능
```

LLMOps에서 "임베딩"이라고 하면, 대부분 이 마지막 단계인 문장 임베딩(더 넓게는 텍스트 임베딩)을 가리킵니다. 질문 하나, 문서 한 단락을 각각 하나의 벡터로 변환한 뒤 유사도를 계산하는 것이 실무에서의 전형적인 사용 방식입니다.

마지막으로 **임베딩 차원과 품질의 관계**를 살펴보겠습니다. 임베딩 차원이란 벡터를 구성하는 숫자의 개수입니다. 384차원 임베딩은 384개의 숫자로, 1536차원 임베딩은 1,536개의 숫자로 텍스트를 표현합니다. 차원이 높을수록 벡터가 텍스트의 의미를 더 세밀하게 표현할 수 있습니다. 마치 사진의 해상도가 높을수록 세부 사항을 더 잘 담아내는 것과 비슷합니다. 384차원에서는 구분되지 않던 미묘한 의미 차이가 1,536차원에서는 구분될 수 있습니다.

그렇다고 차원이 무조건 높을수록 좋은 것은 아닙니다. 차원이 높아지면 벡터 하나가 차지하는 저장 공간이 비례하여 늘어납니다. 문서 100만 건을 임베딩으로 저장한다고 하면, 384차원 벡터는 건당 약 1.5 KB(32비트 부동소수점 기준)이고 전체 약 1.5 GB입니다. 같은 문서를 1,536차원으로 저장하면 건당 약 6 KB이고 전체 약 6 GB로 네 배가 됩니다. 유사도 검색 속도도 차원에 비례하여 느려집니다. 벡터 두 개의 코사인 유사도를 계산하려면 차원 수만큼의 곱셈과 덧셈이 필요하기 때문입니다.

실무에서는 임베딩 모델을 선택할 때, 차원 수와 품질 사이의 균형점을 찾아야 합니다. 일반적인 문서 검색에서는 384~768차원으로도 충분한 품질을 얻을 수 있는 경우가 많고, 법률 문서나 의학 논문처럼 미묘한 용어 차이가 중요한 도메인에서는 더 높은 차원이 유리할 수 있습니다. 이 선택은 다음 단원에서 구체적인 모델별로 다시 다루겠습니다.

정리하면, 텍스트 임베딩은 텍스트를 고정된 길이의 숫자 벡터로 변환하여 의미적 관계를 수치화하는 기술입니다. 두 벡터의 유사도는 코사인 유사도, 유클리드 거리, 내적 등으로 계산하며, 코사인 유사도가 벡터 크기의 영향을 받지 않아 가장 널리 쓰입니다. 임베딩 기술은 의미 관계를 전혀 담지 못하는 원-핫 인코딩에서 출발하여, 단어 임베딩으로 의미를 벡터에 담기 시작했고, Transformer 기반 문장 임베딩으로 발전하면서 문장 전체의 의미를 하나의 벡터로 표현할 수 있게 되었습니다. 임베딩 차원은 높을수록 의미를 세밀하게 표현하지만, 저장 공간과 검색 속도의 비용이 함께 증가하므로 용도에 맞는 균형점을 찾아야 합니다.

다음 단원인 3.1.2에서는 임베딩 모델 선택과 비교를 다룹니다. OpenAI, Sentence-BERT, Cohere 등 주요 임베딩 모델의 특성을 비교하고, 목적에 맞는 모델을 선택하는 기준을 살펴봅니다.

이 단원을 마치면 텍스트 임베딩의 개념과 유사도 계산 방식을 설명할 수 있습니다.
