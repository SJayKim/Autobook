# 9.1.4 Phoenix (Arize)

9.1.2에서 LangSmith의 트레이싱과 평가 기능을, 9.1.3에서 Langfuse의 오픈소스 셀프 호스팅 환경을 살펴보았습니다. 이 단원에서는 세 번째 LLM 옵저버빌리티 도구인 **Phoenix**를 다루고, 마지막에 세 도구를 비교합니다.

**Phoenix**는 Arize AI가 개발한 오픈소스 LLM 옵저버빌리티 도구입니다. Arize AI는 원래 ML 모델의 프로덕션 모니터링을 전문으로 하는 회사입니다. 전통적인 ML 모델에서는 입력 데이터의 분포가 시간이 지남에 따라 변하는 **드리프트**(drift) 현상을 감지하는 것이 중요합니다. 모델을 학습할 때 사용한 데이터와 프로덕션에서 실제로 들어오는 데이터의 분포가 달라지면, 모델의 예측 성능이 저하되기 때문입니다. Arize는 이 드리프트 탐지 기술을 LLM 영역으로 확장하여 Phoenix를 만들었습니다. 그래서 Phoenix는 다른 옵저버빌리티 도구에 비해 임베딩 드리프트 탐지와 데이터 분석 기능이 강점입니다.

Phoenix는 pip로 설치하고 로컬에서 바로 실행할 수 있습니다. Jupyter Notebook 안에서 Phoenix 서버를 띄우고, 같은 노트북에서 분석까지 수행하는 워크플로를 지원합니다. 이 접근 방식은 데이터 과학자에게 익숙한 환경입니다. 별도의 서버 인프라를 구축하지 않고도 로컬 머신에서 바로 트레이스를 수집하고 분석할 수 있습니다. 물론 프로덕션 환경에서는 별도의 서버로 Phoenix를 실행하고, 여러 애플리케이션에서 트레이스를 수집하는 구성도 가능합니다.

```python
# Phoenix 로컬 실행
import phoenix as px

# Phoenix 서버 시작 (기본 포트 6006)
session = px.launch_app()

# 브라우저에서 http://localhost:6006 접속하여 UI 확인
```

위 코드에서 'px.launch_app()'을 호출하면 로컬에 Phoenix 서버가 시작되고, 웹 브라우저에서 대시보드에 접속할 수 있습니다. 이 간결함이 Phoenix의 특징 중 하나입니다.

Phoenix에서의 트레이스 수집은 OpenTelemetry(OTel)를 기반으로 합니다. 9.1.1에서 OpenTelemetry가 벤더 중립적인 계측 표준이라고 설명한 바 있습니다. Phoenix는 OpenTelemetry의 트레이스 수신 엔드포인트를 제공하여, OTel 형식으로 내보낸 트레이스를 수집합니다. Phoenix 팀은 다양한 LLM 프레임워크에 대한 OTel 계측 라이브러리도 제공합니다. OpenAI, Anthropic, LangChain, LlamaIndex 등 주요 프레임워크와 SDK에 대해 'openinference-instrumentation-openai' 같은 패키지를 설치하면, 해당 SDK의 모든 호출이 자동으로 OTel 스팬으로 변환되어 Phoenix에 전송됩니다.

```python
# OpenAI SDK에 대한 자동 계측 설정
from openinference.instrumentation.openai import OpenAIInstrumentor
from phoenix.otel import register

# Phoenix를 트레이스 수신 대상으로 등록
tracer_provider = register(endpoint="http://localhost:6006/v1/traces")

# OpenAI 호출 자동 계측 활성화
OpenAIInstrumentor().instrument(tracer_provider=tracer_provider)
```

위 코드에서 'register()'는 Phoenix 서버를 OTel 트레이스의 수신지로 설정합니다. 'OpenAIInstrumentor().instrument()'를 호출하면, 이후 모든 OpenAI API 호출이 자동으로 트레이스로 기록됩니다. 별도의 데코레이터나 래퍼 없이, 기존 코드를 수정하지 않아도 계측이 이루어집니다.

수집된 트레이스는 Phoenix 웹 UI에서 **트레이스 시각화와 분석**을 통해 탐색합니다. 트레이스 목록에서 각 트레이스의 지연 시간, 토큰 수, 상태를 확인할 수 있고, 개별 트레이스를 열면 스팬 트리가 펼쳐져 각 단계의 입출력을 볼 수 있습니다. 이 부분은 LangSmith나 Langfuse와 유사합니다. Phoenix가 차별화되는 지점은 수집된 트레이스 데이터를 pandas DataFrame으로 추출하여 분석할 수 있다는 것입니다. 'px.Client().get_spans_dataframe()'을 호출하면 모든 스팬 데이터가 DataFrame으로 반환됩니다. 데이터 과학자가 익숙한 도구(pandas, matplotlib 등)로 자유롭게 분석할 수 있으므로, 대시보드 UI가 제공하지 않는 맞춤 분석이 가능합니다.

Phoenix의 고유한 강점은 **임베딩 드리프트 탐지**입니다. 임베딩이란 텍스트를 수치 벡터로 변환한 것이며, 3.1.1에서 다룬 개념입니다. RAG 파이프라인에서 문서를 임베딩으로 변환하여 벡터 데이터베이스에 저장하고, 사용자 질문도 임베딩으로 변환하여 유사한 문서를 검색합니다. 그런데 시간이 지나면 사용자가 입력하는 질문의 분포가 변할 수 있습니다. 새로운 주제의 질문이 들어오거나, 특정 유형의 질문이 갑자기 늘어나는 경우입니다. 이때 벡터 데이터베이스에 저장된 문서 임베딩은 변하지 않았으므로, 새로운 유형의 질문에 대해 검색 품질이 떨어질 수 있습니다. 이것이 **임베딩 드리프트**입니다.

Phoenix는 임베딩 공간을 시각화하여 드리프트를 탐지합니다. 학습 시점(또는 초기 운영 시점)의 임베딩 분포와 현재 프로덕션의 임베딩 분포를 나란히 보여줍니다. 고차원 벡터를 2차원이나 3차원으로 축소하여(UMAP, t-SNE 같은 차원 축소 기법을 사용합니다) 산점도로 표현합니다. 두 분포가 겹치면 드리프트가 적은 것이고, 분리되면 드리프트가 큰 것입니다. 또한 드리프트의 정도를 수치로도 제공합니다.

```
임베딩 드리프트 시각화 개념

[기준 시점 임베딩 분포]        [현재 프로덕션 임베딩 분포]

    * * *                           * *
  * * * * *                       * * * *
  * * * * *                       * * * * *
    * * *                         * * * *
                                      * * *    <-- 새로운 클러스터 출현
                                        * *        (드리프트 발생)

Phoenix는 두 분포를 겹쳐서 보여주고,
새로 나타난 영역을 하이라이트합니다.
--> 이 영역의 질문에 대해 검색 품질을 점검해야 합니다.
```

이 시각화를 통해 운영자는 "최근 들어 기존 문서로 커버되지 않는 새로운 유형의 질문이 늘고 있다"는 사실을 빠르게 파악하고, 벡터 데이터베이스에 관련 문서를 추가하는 조치를 취할 수 있습니다.

Phoenix는 **LLM 평가 통합** 기능도 제공합니다. 수집된 트레이스에 자동 평가자를 적용하여 각 응답의 품질을 점수로 매길 수 있습니다. Phoenix의 평가 모듈은 'phoenix.evals'로 제공되며, 사전 정의된 평가 템플릿(관련성 평가, 환각 평가, 독성 평가 등)을 LLM 기반 판정자와 조합하여 사용합니다. 예를 들어, RAG 파이프라인의 각 응답에 대해 "응답이 검색된 문서에 근거하는가(환각 여부)"를 자동으로 판정할 수 있습니다. 평가 결과는 트레이스에 점수로 첨부되므로, "환각 점수가 낮은 트레이스만 필터링하여 원인을 분석하는" 워크플로가 가능합니다.

**RAG 파이프라인 분석**은 Phoenix가 특히 강점을 보이는 영역입니다. RAG 파이프라인의 성능을 분석하려면, 검색 단계와 생성 단계를 분리하여 각각의 품질을 측정해야 합니다. 검색 단계에서는 "관련 문서를 제대로 찾았는가"가 핵심이고, 생성 단계에서는 "검색된 문서를 바탕으로 정확한 응답을 생성했는가"가 핵심입니다. Phoenix는 검색 단계의 입출력(쿼리와 반환 문서)과 생성 단계의 입출력(프롬프트와 모델 응답)을 각각 추적하고, 검색 관련성 평가와 응답 품질 평가를 자동으로 적용할 수 있습니다. 임베딩 드리프트 탐지와 결합하면, "검색 품질이 떨어진 원인이 사용자 질문 분포의 변화인지, 문서 저장소의 부족인지"를 진단하는 것도 가능합니다.

이제 9.1.2부터 이 단원까지 다룬 세 가지 도구를 비교합니다.


세 도구 모두 LLM 트레이싱이라는 핵심 기능을 제공하지만, 강조점이 다릅니다. LangSmith는 LangChain 생태계와의 통합, 프롬프트 Playground, 체계적인 데이터셋 평가 워크플로가 강점입니다. 빠르게 시작하고 싶고 LangChain을 사용하는 팀에 적합합니다. Langfuse는 MIT 라이선스의 오픈소스이며, 셀프 호스팅이 가능하여 데이터 주권이 중요한 조직에 적합합니다. 프롬프트 버전 관리와 세션 추적이 실무에서 유용합니다. Phoenix는 Arize AI의 ML 모니터링 역량을 계승하여, 임베딩 드리프트 탐지와 RAG 파이프라인 분석에서 차별화됩니다. 데이터 과학 배경의 팀이 Jupyter Notebook 환경에서 깊이 있는 분석을 수행하는 데 적합합니다.

어떤 도구를 선택할지는 조직의 상황에 따라 달라집니다. 데이터를 외부로 보낼 수 없다면 Langfuse나 Phoenix를 셀프 호스팅하는 것이 현실적입니다. LangChain이나 LangGraph를 주력 프레임워크로 사용한다면 LangSmith의 자동 계측이 편리합니다. RAG 파이프라인의 검색 품질 변화를 지속적으로 감시해야 한다면 Phoenix의 임베딩 드리프트 탐지가 유용합니다. 세 도구 모두 OpenTelemetry를 지원하므로, 하나의 도구로 시작한 뒤 필요에 따라 다른 도구로 전환하는 것도 가능합니다.

정리하면, Phoenix는 Arize AI의 ML 모니터링 기술을 LLM 영역으로 확장한 오픈소스 옵저버빌리티 도구입니다. OpenTelemetry 기반의 트레이스 수집, 임베딩 드리프트 탐지, LLM 평가 통합, RAG 파이프라인 분석이 핵심 기능입니다. LangSmith는 LangChain 통합과 프롬프트 실험에, Langfuse는 셀프 호스팅과 프롬프트 버전 관리에, Phoenix는 임베딩 분석과 데이터 과학 워크플로에 각각 강점이 있으며, 조직의 요구사항에 맞는 도구를 선택하거나 조합하여 사용할 수 있습니다.

다음 단원인 9.2.1에서는 LLM 서빙의 지연 시간 모니터링과 SLA 관리를 다룹니다. 옵저버빌리티 도구로 수집한 데이터를 바탕으로, 지연 시간을 구성 요소별로 분석하고 서비스 수준 목표를 설정하는 방법을 살펴봅니다.

이 단원을 마치면 Phoenix를 활용하여 LLM 성능을 분석하고 문제를 진단할 수 있습니다.
