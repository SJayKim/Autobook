# 8.2.1 할루시네이션 탐지와 완화 전략

8.1.1에서 LLM 평가의 주요 지표와 벤치마크를 살펴보았습니다. 평가 지표는 모델의 출력 품질을 정량적으로 측정하는 데 유용하지만, 운영 환경에서는 점수만으로 해결되지 않는 문제가 있습니다. 모델이 그럴듯한 문장을 만들어 내면서도 사실과 다른 내용을 포함하는 경우입니다. 사용자가 "대한민국의 수도는 어디인가?"라고 물었을 때 "대한민국의 수도는 부산입니다"라고 답한다면, 문법적으로 완벽하고 형식도 올바르지만 내용이 틀렸습니다. 이처럼 LLM이 사실이 아닌 정보를 마치 사실인 것처럼 생성하는 현상을 **할루시네이션(hallucination)**이라고 합니다.

할루시네이션이라는 용어는 원래 인간의 지각 현상에서 빌려 왔습니다. 실제로 존재하지 않는 것을 보거나 듣는 현상을 가리키는 말인데, LLM 분야에서는 모델이 학습 데이터나 제공된 컨텍스트에 근거하지 않는 내용을 생성하는 현상을 뜻합니다. 이 문제가 중요한 이유는 LLM의 출력이 자연스러운 문장 형태를 띠기 때문입니다. 틀린 내용이라도 자연스럽게 서술되면, 사용자가 이를 사실로 받아들이기 쉽습니다. 의료, 법률, 금융처럼 정확성이 생명인 분야에서는 할루시네이션 하나가 심각한 결과로 이어질 수 있습니다.

할루시네이션은 크게 두 가지 유형으로 나뉩니다. 첫 번째는 **사실적 할루시네이션(factual hallucination)**입니다. 모델이 현실 세계의 사실과 다른 정보를 생성하는 경우입니다. "아인슈타인이 1980년에 노벨상을 받았다"처럼, 검증 가능한 사실에 대해 틀린 내용을 만들어 내는 것이 이에 해당합니다. 아인슈타인은 1921년에 노벨 물리학상을 받았으므로, 모델이 연도를 잘못 생성한 것입니다.

두 번째는 **충실도 할루시네이션(faithfulness hallucination)**입니다. 이 유형은 모델에게 특정 문서나 컨텍스트를 제공한 뒤 이를 바탕으로 답변하라고 요청했을 때 발생합니다. 모델이 제공된 컨텍스트에 없는 내용을 마치 컨텍스트에 있는 것처럼 답변하거나, 컨텍스트의 내용을 왜곡하여 전달하는 경우입니다. 예를 들어, 회사 내부 문서에 "2024년 매출이 100억 원이다"라고 적혀 있는데 모델이 "2024년 매출이 150억 원이다"라고 답변하면, 이는 충실도 할루시네이션입니다. 4.1.2에서 다룬 RAG 파이프라인에서 이 유형이 특히 문제가 됩니다. 검색된 문서를 바탕으로 답변해야 하는데, 모델이 문서 내용을 벗어나 답변을 꾸며 내면 RAG를 도입한 의미가 사라집니다.

두 유형의 차이를 정리하면 다음과 같습니다. 사실적 할루시네이션은 "세상의 사실과 다른가?"를 기준으로 판단하고, 충실도 할루시네이션은 "주어진 컨텍스트와 다른가?"를 기준으로 판단합니다. 같은 문장이라도 판단 기준에 따라 유형이 달라질 수 있습니다. "서울의 인구는 500만 명이다"라는 문장이 있을 때, 제공된 문서에 "서울의 인구는 500만 명"이라고 적혀 있다면 충실도 관점에서는 할루시네이션이 아닙니다. 그러나 실제 서울 인구가 약 950만 명이므로 사실적 관점에서는 할루시네이션입니다. 운영 환경에서는 두 유형을 모두 감시해야 합니다.

```
할루시네이션 유형 구분

사실적 할루시네이션 (Factual Hallucination)
  판단 기준: 현실 세계의 검증 가능한 사실
  예시: "아인슈타인이 1980년에 노벨상을 받았다" (실제: 1921년)

충실도 할루시네이션 (Faithfulness Hallucination)
  판단 기준: 제공된 컨텍스트(문서, 지시)
  예시: 문서에 "매출 100억"인데 "매출 150억"이라고 답변
```

할루시네이션이 왜 발생하는지를 이해하면 탐지와 완화 전략을 설계하기가 수월해집니다. 발생 원인은 여러 층위에 걸쳐 있습니다.

첫 번째 원인은 학습 데이터의 한계입니다. LLM은 대규모 텍스트 데이터를 학습하는데, 이 데이터에 오류가 포함되어 있거나 서로 모순되는 정보가 섞여 있을 수 있습니다. 모델은 데이터에서 패턴을 학습하므로, 잘못된 패턴이 들어 있으면 잘못된 내용을 생성할 수 있습니다. 또한 학습 데이터에는 시점이 있습니다. 2023년까지의 데이터로 학습한 모델은 2024년에 발생한 사건을 알지 못합니다. 그런데 모델은 "모른다"고 답변하는 대신, 기존 패턴을 조합하여 그럴듯한 답변을 만들어 내는 경향이 있습니다.

두 번째 원인은 모델 구조 자체의 특성입니다. 1.2.3에서 살펴본 것처럼, LLM은 다음 토큰의 확률 분포를 계산하고 여기서 토큰을 선택하는 방식으로 텍스트를 생성합니다. 모델이 최적화하는 목표는 "학습 데이터에서 다음 토큰을 잘 예측하는 것"이지, "사실만 말하는 것"이 아닙니다. 통계적으로 자연스러운 토큰 조합이 사실적으로 정확한 조합과 항상 일치하지는 않습니다. 예를 들어 "노벨상을 받은 해는"이라는 문맥 뒤에 가장 자연스러운 연도가 통계적으로 1980이라면, 모델은 사실 여부와 무관하게 그 토큰을 선택할 수 있습니다.

세 번째 원인은 디코딩 과정에서의 누적 오차입니다. 1.2.3에서 다룬 자기회귀 생성에서, 모델은 이전에 생성한 토큰을 다시 입력으로 사용합니다. 초반에 약간 부정확한 토큰이 선택되면, 그 토큰을 기반으로 다음 토큰이 생성되면서 오류가 점점 커질 수 있습니다. 마치 길을 걸을 때 처음에 방향이 1도만 틀어져도, 멀리 갈수록 목적지에서 크게 벗어나는 것과 비슷합니다.

네 번째 원인은 컨텍스트 창의 제약입니다. 아무리 긴 컨텍스트를 지원하는 모델이라도, 프롬프트에 담긴 정보를 완벽하게 활용하지 못할 수 있습니다. 특히 긴 문서가 입력되었을 때, 문서 중간에 위치한 정보를 놓치는 현상이 알려져 있습니다. 이 경우 모델은 컨텍스트에 있는 정보를 무시하고 자체 학습 지식으로 답변을 생성하여 충실도 할루시네이션이 발생합니다.

이제 할루시네이션을 탐지하는 기법을 살펴보겠습니다. 탐지란 모델의 출력이 할루시네이션을 포함하는지 자동으로 판별하는 과정입니다. 수동으로 모든 출력을 검토하는 것은 운영 규모에서 불가능하므로, 자동화된 탐지 기법이 필요합니다.

첫 번째 탐지 기법은 **NLI(Natural Language Inference) 기반 탐지**입니다. NLI는 두 문장 사이의 논리적 관계를 판단하는 자연어 처리 과제입니다. 하나의 문장(전제, premise)이 주어졌을 때, 다른 문장(가설, hypothesis)이 전제로부터 참인지(함의, entailment), 거짓인지(모순, contradiction), 판단할 수 없는지(중립, neutral)를 분류합니다. 이 분류 능력을 할루시네이션 탐지에 활용합니다.

NLI 기반 탐지의 동작 방식은 다음과 같습니다. 모델에 입력으로 제공한 컨텍스트(예: RAG에서 검색된 문서)를 전제로 놓습니다. 모델이 생성한 답변의 각 문장을 가설로 놓습니다. NLI 모델에 (전제, 가설) 쌍을 넣어 판정합니다. "함의"가 나오면 해당 문장은 컨텍스트에 근거가 있다고 판단합니다. "모순"이 나오면 해당 문장이 컨텍스트와 충돌하므로 할루시네이션 가능성이 높습니다. "중립"이 나오면 컨텍스트에서 확인할 수 없는 내용이므로, 역시 할루시네이션 의심 대상입니다.

```
NLI 기반 할루시네이션 탐지 흐름

[컨텍스트 (전제)]        [모델 응답]
"서울 인구 약 950만"     "서울 인구는 약 950만이며,
                          부산보다 2배 크다"
        |                        |
        v                        v
  +--- 문장 분리 ---+
  |                 |
  | 문장1: "서울 인구는 약 950만"
  | 문장2: "부산보다 2배 크다"
  |
  +--> NLI 모델 판정 <--+
       |
       문장1: 함의 (entailment) --> 근거 있음
       문장2: 중립 (neutral)    --> 할루시네이션 의심
```

구체적인 예를 들어 설명하겠습니다. RAG 파이프라인에서 검색된 문서에 "A사의 2024년 연간 매출은 500억 원이다"라는 내용이 있다고 합시다. 모델이 "A사의 2024년 연간 매출은 500억 원이며, 전년 대비 20% 성장했습니다"라고 답변했습니다. 이 답변을 문장 단위로 분리하면 두 개의 주장이 됩니다. 첫 번째 주장 "연간 매출은 500억 원"은 컨텍스트와 함의 관계이므로 근거가 있습니다. 두 번째 주장 "전년 대비 20% 성장"은 컨텍스트에 해당 정보가 없으므로 중립으로 판정됩니다. 이 경우 두 번째 주장은 할루시네이션으로 플래그 처리합니다.

두 번째 탐지 기법은 **Self-Consistency 검증**입니다. 이 기법은 같은 질문에 대해 모델이 여러 번 응답을 생성하게 한 뒤, 응답들 사이의 일관성을 비교합니다. 핵심 아이디어는 이렇습니다. 모델이 확실하게 아는 내용이라면, 여러 번 물어도 비슷한 답변이 나올 것입니다. 반면 모델이 확신 없이 추측하는 내용이라면, 물을 때마다 다른 답변이 나올 가능성이 높습니다.

Self-Consistency 검증의 동작 절차를 단계별로 설명하겠습니다. 먼저 같은 질문에 대해 temperature를 약간 높인 상태로 여러 개(보통 3개에서 5개)의 응답을 생성합니다. 다음으로, 생성된 응답들에서 핵심 주장을 추출합니다. 그런 다음, 핵심 주장들이 서로 일치하는지 비교합니다. 응답들 사이에 모순이 발견되면 해당 부분을 할루시네이션 의심 대상으로 표시합니다.

```
Self-Consistency 검증 절차

질문: "Python의 GIL은 무엇인가?"

  응답 1: "GIL은 Global Interpreter Lock으로,
           한 번에 하나의 스레드만 바이트코드를 실행하게 합니다"

  응답 2: "GIL은 Global Interpreter Lock이며,
           멀티스레딩에서 한 스레드만 인터프리터를 점유합니다"

  응답 3: "GIL은 Global Interpreter Lock으로,
           메모리 접근을 동기화하는 메커니즘입니다"

  --> 핵심 주장 비교: 세 응답 모두 GIL의 정의가 일관됨
  --> 일관성 높음 --> 할루시네이션 가능성 낮음
```

만약 응답 하나가 "GIL은 Garbage Inline Loader의 약자입니다"라고 답변했다면, 나머지 응답들과 모순이 발생합니다. 이런 불일치가 나타나면 해당 내용의 신뢰도가 낮다고 판단할 수 있습니다. Self-Consistency 검증의 장점은 외부 지식 베이스가 없어도 적용할 수 있다는 점입니다. 단점은 모델을 여러 번 호출해야 하므로 비용과 지연이 증가한다는 것입니다.

세 번째 탐지 기법은 **외부 지식 대조(external knowledge verification)**입니다. 모델의 출력에서 검증 가능한 주장을 추출한 뒤, 외부 지식 베이스(지식 그래프, 검색 엔진, 신뢰할 수 있는 데이터베이스 등)와 대조하여 사실 여부를 확인합니다. 이 기법은 사실적 할루시네이션을 탐지하는 데 특히 효과적입니다.

외부 지식 대조의 절차는 다음과 같습니다. 먼저 모델의 출력에서 검증 가능한 사실 주장을 추출합니다. "아인슈타인은 1921년에 노벨 물리학상을 받았다"는 검증 가능한 주장이지만, "그의 업적은 위대하다"는 주관적 판단이므로 추출 대상이 아닙니다. 다음으로, 추출된 주장을 외부 지식 소스에서 검색합니다. 검색 결과와 주장을 비교하여 일치 여부를 판정합니다. 일치하지 않는 주장을 할루시네이션으로 표시합니다.

이 기법의 장점은 객관적인 근거를 기반으로 판정하므로 정확도가 높다는 것입니다. 단점은 외부 지식 소스가 필요하고, 검색과 비교에 추가 시간이 걸린다는 것입니다. 또한, 외부 지식 소스 자체가 최신 정보를 반영하지 못하거나 오류를 포함할 수 있습니다. 세 가지 탐지 기법은 서로 보완 관계에 있으므로, 실제 운영 환경에서는 여러 기법을 조합하여 사용하는 것이 일반적입니다.

탐지만으로는 문제가 해결되지 않습니다. 할루시네이션을 줄이기 위한 완화 전략도 함께 적용해야 합니다. 완화 전략은 크게 세 가지 방향으로 나뉩니다.

첫 번째 완화 전략은 **RAG(Retrieval-Augmented Generation)**를 활용하는 것입니다. 4.1.2에서 다룬 것처럼, RAG는 외부 문서에서 관련 정보를 검색하여 프롬프트에 포함시킵니다. 모델이 자체 학습 지식에만 의존하지 않고, 검색된 문서를 근거로 답변하도록 유도합니다. 이렇게 하면 모델이 없는 내용을 지어낼 가능성이 줄어듭니다. 다만 RAG를 도입했다고 해서 할루시네이션이 완전히 사라지지는 않습니다. 검색 품질이 낮으면 관련 없는 문서가 검색될 수 있고, 관련 문서가 검색되더라도 모델이 그 내용을 무시하고 자체 지식으로 답변할 수 있습니다. 따라서 RAG는 할루시네이션 완화의 첫 단계이지, 유일한 해결책은 아닙니다.

두 번째 완화 전략은 **프롬프트 개선**입니다. 모델에게 지시를 내리는 방식을 바꾸어 할루시네이션을 줄이는 접근입니다. 구체적인 기법으로는 다음과 같은 것들이 있습니다. 프롬프트에 "제공된 문서에 없는 내용은 '정보 없음'이라고 답변하세요"와 같은 명시적 지시를 추가합니다. 이를 통해 모델이 모르는 내용에 대해 추측하는 대신 솔직하게 답변하도록 유도합니다. 또 다른 기법으로, 답변과 함께 근거가 되는 문서 부분을 인용하도록 요청합니다. "답변의 각 주장에 대해 근거가 되는 문서의 해당 부분을 함께 인용하세요"라고 지시하면, 모델이 답변을 생성할 때 근거를 의식하게 됩니다. 근거를 찾을 수 없는 주장은 생성하기 어려워지므로 할루시네이션이 줄어드는 효과가 있습니다.

```
프롬프트 개선 예시

[개선 전]
"다음 문서를 바탕으로 질문에 답변하세요.
 문서: {context}
 질문: {question}"

[개선 후]
"다음 문서만을 근거로 질문에 답변하세요.
 문서에 답변할 수 있는 정보가 없으면
 '제공된 문서에 해당 정보가 없습니다'라고 답변하세요.
 답변의 각 주장에 대해 근거가 되는 문서 부분을 [출처]로 표시하세요.
 문서: {context}
 질문: {question}"
```

세 번째 완화 전략은 **디코딩 제약(decoding constraints)**입니다. 1.2.3에서 다룬 것처럼, 모델이 토큰을 선택하는 디코딩 과정에서 제약 조건을 걸어 할루시네이션 가능성을 줄이는 방법입니다. 대표적인 기법은 temperature를 낮추는 것입니다. temperature를 낮추면 모델이 확률이 높은 토큰을 선택하는 경향이 강해져, 모델이 가장 확신하는 내용을 출력할 가능성이 높아집니다. 반면 창의성이 줄어들므로, 사실 기반 질의응답에는 적합하지만 창의적 글쓰기에는 부적합합니다. 또한 반복 페널티(repetition penalty)를 적용하여 모델이 같은 내용을 반복하다가 맥락을 벗어나는 현상을 방지할 수 있습니다.

세 가지 완화 전략을 정리하면 다음과 같습니다.

```
할루시네이션 완화 전략 비교

+------------------+------------------------+-----------------------+
| 전략             | 핵심 원리              | 적용 시점             |
+------------------+------------------------+-----------------------+
| RAG              | 외부 문서를 근거로     | 파이프라인 설계 시    |
|                  | 답변 생성 유도         |                       |
+------------------+------------------------+-----------------------+
| 프롬프트 개선    | 명시적 지시로 모델의   | 프롬프트 작성 시      |
|                  | 행동 범위 제한         |                       |
+------------------+------------------------+-----------------------+
| 디코딩 제약      | 토큰 선택 과정에서     | 추론 파라미터 설정 시 |
|                  | 확률 분포 조정         |                       |
+------------------+------------------------+-----------------------+
```

완화 전략과 밀접하게 관련된 개념이 **그라운딩(Grounding)**입니다. 그라운딩은 모델의 출력을 검증 가능한 근거에 고정(anchor)시키는 기법을 총칭합니다. 앞서 살펴본 RAG도 그라운딩의 한 형태입니다. 검색된 문서를 근거로 삼아 모델의 답변이 허공에 뜨지 않도록 고정하는 것이기 때문입니다.

그라운딩이 왜 별도의 개념으로 중요한지 설명하겠습니다. RAG, 프롬프트 개선, 디코딩 제약은 각각 할루시네이션을 줄이는 개별 기법입니다. 그라운딩은 이런 기법들을 관통하는 원칙입니다. "모델의 출력에는 반드시 근거가 있어야 한다"는 원칙을 세우고, 이 원칙을 다양한 방식으로 실현하는 것이 그라운딩입니다. 구체적으로 그라운딩을 구현하는 방식에는 다음과 같은 것들이 있습니다.

문서 그라운딩은 모델의 답변이 특정 문서 집합에만 근거하도록 제한합니다. RAG 파이프라인에서 검색된 문서만 참고하도록 하고, 문서에 없는 내용은 생성하지 않도록 합니다. 데이터 그라운딩은 답변에 포함되는 수치나 통계를 특정 데이터베이스에서 직접 가져오도록 합니다. 모델이 수치를 추측하는 대신, 쿼리를 실행하여 정확한 값을 가져오고 이를 답변에 삽입합니다. 도구 그라운딩은 모델이 계산이나 검색 같은 작업을 직접 수행하는 대신, 외부 도구(계산기, 검색 엔진, API 등)를 호출하여 결과를 받아 오도록 합니다. 5.2에서 다룬 에이전트의 도구 활용이 이에 해당합니다.

그라운딩이 적용된 시스템에서는 모델의 출력을 "근거가 있는 부분"과 "근거가 없는 부분"으로 나눌 수 있습니다. 근거가 없는 부분은 할루시네이션 위험이 높으므로 경고를 표시하거나 자동으로 필터링합니다. 이처럼 그라운딩은 탐지와 완화를 연결하는 다리 역할을 합니다.

마지막으로, 할루시네이션 탐지와 완화를 일회성이 아닌 지속적인 과정으로 운영하려면 **할루시네이션 모니터링 시스템**이 필요합니다. 모니터링 시스템은 운영 중인 LLM 애플리케이션의 모든 입출력 쌍에 대해, 할루시네이션 탐지 기법을 자동으로 적용하고 결과를 기록하는 체계입니다.

할루시네이션 모니터링 시스템의 기본 구성 요소는 다음과 같습니다. 먼저 **로깅 파이프라인**이 있습니다. 모델에 들어가는 입력(프롬프트, 컨텍스트)과 나오는 출력(응답)을 빠짐없이 기록합니다. 다음으로 **자동 탐지 모듈**이 있습니다. 기록된 입출력 쌍에 대해 NLI 기반 검증, Self-Consistency 검증, 외부 지식 대조 등의 탐지 기법을 자동으로 실행합니다. 실시간으로 모든 요청에 탐지를 적용할 수도 있고, 비용을 고려하여 샘플링 방식으로 일부 요청에만 적용할 수도 있습니다. 그 다음에 **대시보드와 알림**이 있습니다. 할루시네이션 탐지 비율을 시간대별, 질문 유형별, 모델 버전별로 집계하여 시각화합니다. 할루시네이션 비율이 임계값을 넘으면 담당자에게 알림을 보냅니다.

```
할루시네이션 모니터링 시스템 구조

사용자 요청 --> [LLM 애플리케이션] --> 응답 반환
                     |
                     v
              [로깅 파이프라인]
              (입력, 컨텍스트, 출력 기록)
                     |
                     v
              [자동 탐지 모듈]
              - NLI 기반 검증
              - Self-Consistency
              - 외부 지식 대조
                     |
                     v
              [대시보드 / 알림]
              - 할루시네이션 비율 추이
              - 유형별 분류
              - 임계값 초과 시 알림
                     |
                     v
              [피드백 루프]
              - 프롬프트 개선
              - RAG 검색 품질 조정
              - 모델 교체/업데이트 판단
```

모니터링 시스템의 마지막 구성 요소는 **피드백 루프**입니다. 탐지 결과를 분석하여 할루시네이션이 자주 발생하는 패턴을 파악합니다. 특정 유형의 질문에서 할루시네이션 비율이 높다면, 해당 유형에 대한 프롬프트를 개선하거나, RAG 검색 품질을 조정하거나, 관련 문서를 보강합니다. 모델 전체의 할루시네이션 비율이 지속적으로 높다면, 모델을 교체하거나 파인튜닝을 검토할 수 있습니다. 이처럼 모니터링 시스템은 탐지 결과를 완화 전략의 개선으로 연결하여, 할루시네이션을 지속적으로 줄여 나가는 순환 구조를 만듭니다.

정리하면, 할루시네이션은 LLM이 사실이 아닌 내용을 그럴듯하게 생성하는 현상이며, 사실적 할루시네이션과 충실도 할루시네이션의 두 유형으로 나뉩니다. 탐지 기법으로는 NLI 기반 검증, Self-Consistency 검증, 외부 지식 대조가 있고, 완화 전략으로는 RAG 활용, 프롬프트 개선, 디코딩 제약이 있습니다. 그라운딩은 모델의 출력을 검증 가능한 근거에 고정시키는 원칙으로, 여러 완화 전략을 관통합니다. 이 모든 것을 지속적으로 운영하려면 로깅, 자동 탐지, 대시보드, 피드백 루프로 구성된 할루시네이션 모니터링 시스템이 필요합니다.

다음 단원인 8.2.2에서는 가드레일과 안전 필터 구축을 다룹니다. 할루시네이션 탐지가 출력의 사실성을 검증하는 것이라면, 가드레일은 유해하거나 부적절한 출력 자체를 차단하는 보다 넓은 안전 체계입니다.

이 단원을 마치면 할루시네이션의 유형을 분류하고, 적합한 탐지 기법과 완화 전략을 선택하여 적용할 수 있습니다.
