# 3.2.2 Pinecone

3.2.1에서 벡터 데이터베이스가 무엇이고 왜 필요한지를 살펴보았습니다. 임베딩 벡터를 저장하고 ANN(Approximate Nearest Neighbor) 검색으로 유사한 벡터를 빠르게 찾아내는 것이 벡터 데이터베이스의 핵심 역할이었습니다. 이제 실제로 이 역할을 수행하는 구체적인 서비스를 하나씩 다뤄 보겠습니다. 이 단원에서는 완전 관리형(fully managed) 클라우드 벡터 데이터베이스인 **Pinecone**을 다룹니다.

Pinecone을 이해하려면 먼저 "완전 관리형"이라는 말의 의미를 짚어야 합니다. 데이터베이스를 직접 운영하려면 서버를 준비하고, 소프트웨어를 설치하며, 디스크 용량과 메모리를 관리하고, 장애가 발생하면 복구해야 합니다. 완전 관리형 서비스는 이 모든 인프라 관리를 서비스 제공자가 대신 처리합니다. 사용자는 API를 통해 데이터를 저장하고 검색하기만 하면 됩니다. Pinecone은 벡터 데이터베이스 분야에서 이 완전 관리형 모델을 일찍 도입한 서비스로, 인프라 운영 부담 없이 벡터 검색 기능을 사용할 수 있다는 점이 가장 큰 특징입니다.

Pinecone의 아키텍처를 이해하려면 세 가지 핵심 개념을 알아야 합니다. **인덱스**(Index), **네임스페이스**(Namespace), 그리고 배포 방식인 **Pod**와 **Serverless**입니다. 하나씩 살펴보겠습니다.

**인덱스**는 벡터를 저장하고 검색하는 최상위 단위입니다. 관계형 데이터베이스에서 테이블이 데이터를 담는 기본 단위인 것처럼, Pinecone에서는 인덱스가 그 역할을 합니다. 하나의 인덱스는 특정 차원(dimension)의 벡터만 저장할 수 있습니다. 예를 들어, OpenAI의 text-embedding-3-small 모델이 생성하는 1536차원 벡터를 저장하려면 인덱스를 생성할 때 차원을 1536으로 지정해야 합니다. 인덱스를 생성한 뒤에는 차원을 변경할 수 없으므로, 어떤 임베딩 모델을 사용할지 먼저 정한 뒤 인덱스를 만들어야 합니다. 인덱스 생성 시에는 차원 외에도 유사도 측정 방식(metric)을 지정합니다. cosine, dotproduct, euclidean 중 하나를 선택할 수 있으며, 3.1.1에서 다룬 코사인 유사도, 내적, 유클리드 거리가 각각 이에 대응합니다.

**네임스페이스**는 하나의 인덱스 안에서 벡터를 논리적으로 구분하는 단위입니다. 예를 들어, 하나의 인덱스에 여러 고객사의 문서 임베딩을 저장한다고 가정해 보겠습니다. 고객사 A의 벡터와 고객사 B의 벡터를 같은 공간에 섞어 두면 검색 시 다른 고객사의 데이터가 결과에 포함될 수 있습니다. 네임스페이스를 사용하면 "customer-a", "customer-b"처럼 이름을 붙여 벡터를 분리할 수 있고, 검색 시 특정 네임스페이스만 대상으로 지정할 수 있습니다. 네임스페이스는 별도로 생성하는 절차가 없습니다. 벡터를 삽입할 때 네임스페이스 이름을 지정하면 자동으로 만들어집니다.

배포 방식은 **Pod 기반**과 **Serverless** 두 가지가 있습니다. Pod 기반은 사전에 정해진 크기의 컴퓨팅 자원(Pod)을 할당받아 사용하는 방식입니다. Pod의 종류(s1, p1, p2)에 따라 저장 용량과 검색 속도가 달라지며, 복제본(replica) 수를 늘려 읽기 처리량을 확장할 수 있습니다. Pod 기반 배포는 자원이 미리 확보되어 있으므로 응답 지연이 안정적이지만, 트래픽이 적을 때에도 할당된 자원에 대한 비용이 발생합니다. Serverless 방식은 사용한 만큼만 비용을 내는 모델입니다. 별도의 Pod를 관리하지 않으며, 저장된 벡터 수와 읽기/쓰기 요청 횟수에 따라 과금됩니다. 트래픽 변동이 큰 서비스에서는 Serverless가 비용 효율적이고, 일정한 고부하 트래픽에서는 Pod 기반이 유리할 수 있습니다.

이 세 개념의 관계를 정리하면 다음과 같습니다.

```
Pinecone 아키텍처 계층 구조

+-----------------------------------------+
|              Pinecone Project            |
|                                         |
|  +-----------------------------------+  |
|  |          Index (1536차원, cosine)  |  |
|  |                                   |  |
|  |  +-------------+ +-------------+  |  |
|  |  | Namespace   | | Namespace   |  |  |
|  |  | "customer-a"| | "customer-b"|  |  |
|  |  |             | |             |  |  |
|  |  | 벡터 1      | | 벡터 4      |  |  |
|  |  | 벡터 2      | | 벡터 5      |  |  |
|  |  | 벡터 3      | | 벡터 6      |  |  |
|  |  +-------------+ +-------------+  |  |
|  +-----------------------------------+  |
|                                         |
|  배포 방식: Pod 또는 Serverless          |
+-----------------------------------------+
```

프로젝트(Project)는 Pinecone 계정 안에서 인덱스를 묶는 최상위 그룹입니다. 하나의 프로젝트 안에 여러 인덱스를 만들 수 있고, 각 인덱스 안에 여러 네임스페이스를 둘 수 있습니다.

이제 인덱스를 실제로 생성하는 과정을 살펴보겠습니다. Pinecone은 Python 클라이언트 라이브러리를 제공하며, 이를 통해 인덱스를 생성하고 벡터를 다룰 수 있습니다. 인덱스를 생성하는 코드는 다음과 같습니다.

```python
from pinecone import Pinecone, ServerlessSpec

pc = Pinecone(api_key="YOUR_API_KEY")

pc.create_index(
    name="my-index",
    dimension=1536,
    metric="cosine",
    spec=ServerlessSpec(
        cloud="aws",
        region="us-east-1"
    )
)
```

첫 줄에서 Pinecone 클라이언트를 초기화합니다. api_key는 Pinecone 콘솔에서 프로젝트를 생성하면 발급받는 인증 키입니다. create_index 호출에서 name은 인덱스의 이름이며, 프로젝트 내에서 고유해야 합니다. dimension은 저장할 벡터의 차원 수로, 사용하는 임베딩 모델의 출력 차원과 일치해야 합니다. metric은 유사도 측정 방식입니다. spec 매개변수에서 배포 방식을 지정하는데, 위 예시에서는 ServerlessSpec을 사용하여 Serverless 방식으로 생성하고 있습니다. cloud와 region은 인덱스가 배포될 클라우드 제공자와 리전을 뜻합니다.

Pod 기반으로 생성하려면 ServerlessSpec 대신 PodSpec을 사용합니다.

```python
from pinecone import Pinecone, PodSpec

pc = Pinecone(api_key="YOUR_API_KEY")

pc.create_index(
    name="my-pod-index",
    dimension=1536,
    metric="cosine",
    spec=PodSpec(
        environment="us-east-1-aws",
        pod_type="p1.x1",
        pods=1
    )
)
```

PodSpec에서 environment는 배포 환경, pod_type은 Pod의 종류와 크기를 나타냅니다. "p1.x1"에서 p1은 성능 최적화 Pod 유형이고, x1은 기본 크기입니다. pods는 할당할 Pod 수입니다.

인덱스가 생성되면 벡터를 저장할 차례입니다. Pinecone에서 벡터를 저장하는 연산을 **Upsert**라고 부릅니다. Upsert는 "Update or Insert"의 줄임말로, 같은 ID의 벡터가 이미 존재하면 갱신하고, 없으면 새로 삽입하는 동작입니다. 관계형 데이터베이스에서 INSERT와 UPDATE를 하나로 합친 것과 같은 개념입니다.

```python
index = pc.Index("my-index")

index.upsert(
    vectors=[
        {
            "id": "doc-001",
            "values": [0.1, 0.2, 0.3, ...],  # 1536차원 벡터
            "metadata": {"source": "manual", "category": "setup"}
        },
        {
            "id": "doc-002",
            "values": [0.4, 0.5, 0.6, ...],
            "metadata": {"source": "faq", "category": "billing"}
        }
    ],
    namespace="customer-a"
)
```

pc.Index("my-index")로 인덱스 객체를 가져옵니다. upsert 메서드에 vectors 리스트를 전달하는데, 각 벡터는 id, values, metadata 세 필드로 구성됩니다. id는 벡터를 식별하는 고유 문자열입니다. values는 실제 임베딩 벡터 값의 배열로, 인덱스 생성 시 지정한 차원과 길이가 같아야 합니다. metadata는 벡터에 부가 정보를 붙이는 선택적 필드입니다. 이 메타데이터는 나중에 검색할 때 필터 조건으로 활용할 수 있습니다. namespace 매개변수로 벡터가 들어갈 네임스페이스를 지정합니다.

대량의 벡터를 한꺼번에 삽입할 때는 한 번의 upsert 호출에 최대 100개 벡터를 담을 수 있으며, 이를 초과하는 양은 여러 번에 나누어 호출해야 합니다. 벡터 수가 수만 건 이상이면 배치 단위로 나누어 반복 호출하는 방식이 일반적입니다.

저장된 벡터를 검색하는 것이 **Query**입니다. Query는 질의 벡터와 가장 유사한 벡터를 인덱스에서 찾아 반환합니다.

```python
results = index.query(
    vector=[0.15, 0.25, 0.35, ...],  # 질의 벡터 (1536차원)
    top_k=5,
    include_metadata=True,
    namespace="customer-a"
)
```

vector는 검색 기준이 되는 질의 벡터입니다. 사용자의 질문을 임베딩 모델에 넣어 얻은 벡터를 여기에 전달합니다. top_k는 반환받을 결과의 최대 개수입니다. 위 예시에서는 가장 유사한 5개의 벡터를 요청합니다. include_metadata를 True로 설정하면 결과에 메타데이터도 함께 포함됩니다. namespace를 지정하면 해당 네임스페이스 안에서만 검색합니다.

반환되는 결과는 다음과 같은 구조입니다.

```python
{
    "matches": [
        {
            "id": "doc-001",
            "score": 0.92,
            "metadata": {"source": "manual", "category": "setup"}
        },
        {
            "id": "doc-002",
            "score": 0.85,
            "metadata": {"source": "faq", "category": "billing"}
        }
    ],
    "namespace": "customer-a"
}
```

matches 배열의 각 항목은 검색된 벡터의 id, 유사도 점수(score), 그리고 메타데이터를 담고 있습니다. score는 인덱스 생성 시 지정한 metric에 따라 계산됩니다. cosine 메트릭을 사용했다면 score는 0에서 1 사이의 값이며, 1에 가까울수록 질의 벡터와 유사합니다.

Pinecone에서는 벡터를 ID로 직접 가져오는 Fetch, 특정 ID의 벡터를 삭제하는 Delete 연산도 제공합니다.

```python
# ID로 벡터 조회
fetch_result = index.fetch(ids=["doc-001", "doc-002"], namespace="customer-a")

# 벡터 삭제
index.delete(ids=["doc-002"], namespace="customer-a")

# 네임스페이스 전체 삭제
index.delete(delete_all=True, namespace="customer-a")
```

fetch는 ID 목록을 받아 해당 벡터의 값과 메타데이터를 반환합니다. delete는 지정한 ID의 벡터를 삭제하며, delete_all=True를 사용하면 네임스페이스 내 모든 벡터를 한 번에 삭제할 수 있습니다.

정리하면, Upsert(삽입/갱신), Query(유사도 검색), Fetch(ID 조회), Delete(삭제)가 Pinecone의 기본 CRUD 연산입니다.

```
Pinecone 기본 CRUD 연산 흐름

                  +------------------+
                  |   Pinecone Index |
                  +------------------+
                         |
         +-------+-------+-------+-------+
         |       |       |       |       |
      Upsert   Query   Fetch  Delete  Update
      (C/U)    (R)     (R)    (D)     (U)
         |       |       |       |
  벡터+메타  질의벡터  ID 목록  ID 목록
  데이터     +top_k
  저장/갱신  유사 벡터  벡터 값  벡터
             반환      반환     제거
```

이제 **메타데이터 필터링**을 살펴보겠습니다. 3.2.1에서 벡터 데이터베이스의 핵심 기능 중 하나로 메타데이터 필터링을 언급했습니다. 벡터 유사도 검색만으로는 원하는 결과를 정밀하게 얻기 어려운 경우가 있습니다. 예를 들어, 문서 임베딩을 검색할 때 "billing" 카테고리에 해당하는 문서만 대상으로 하고 싶다면, 유사도만으로는 카테고리를 구분할 수 없습니다. 메타데이터 필터링은 벡터에 붙인 메타데이터를 조건으로 지정하여 검색 범위를 좁히는 기능입니다.

```python
results = index.query(
    vector=[0.15, 0.25, 0.35, ...],
    top_k=5,
    include_metadata=True,
    namespace="customer-a",
    filter={
        "category": {"$eq": "billing"},
        "source": {"$in": ["faq", "manual"]}
    }
)
```

filter 매개변수에 조건을 딕셔너리 형태로 전달합니다. 위 예시에서는 category가 "billing"이고, source가 "faq" 또는 "manual"인 벡터만 대상으로 유사도 검색을 수행합니다. $eq는 "같다", $in은 "목록 안에 포함"이라는 뜻입니다. 이 외에도 $gt(초과), $gte(이상), $lt(미만), $lte(이하), $ne(같지 않음) 같은 비교 연산자를 사용할 수 있습니다. $and와 $or로 여러 조건을 논리적으로 조합하는 것도 가능합니다.

메타데이터 필터링은 유사도 검색 이전에 적용됩니다. 즉, 필터 조건에 맞는 벡터만을 대상으로 ANN 검색을 수행하므로, 전체 벡터를 검색한 뒤 결과를 걸러내는 것보다 효율적입니다. 다만, 필터 조건이 지나치게 좁으면 ANN 검색의 대상 벡터 수가 줄어들어 검색 품질이 떨어질 수 있습니다.

Pinecone의 **장점**을 정리하겠습니다. 첫째, 완전 관리형이므로 인프라 운영에 대한 부담이 없습니다. 서버 프로비저닝, 인덱스 샤딩, 복제, 장애 복구를 Pinecone이 자동으로 처리합니다. 둘째, API가 간결합니다. Upsert, Query, Fetch, Delete 네 가지 핵심 연산만으로 대부분의 작업을 처리할 수 있어 학습 곡선이 낮습니다. 셋째, 확장성이 뛰어납니다. 수백만에서 수억 건의 벡터를 저장하고 검색할 수 있으며, Serverless 방식을 사용하면 트래픽에 따라 자동으로 확장됩니다.

**단점**도 있습니다. 첫째, 완전 관리형이므로 인프라를 세밀하게 제어할 수 없습니다. 인덱스 알고리즘의 내부 파라미터를 직접 조정하거나, 특정 하드웨어에 배포하는 것이 불가능합니다. 둘째, 벡터 데이터가 Pinecone의 클라우드에 저장되므로, 데이터 주권이나 규제 요건이 엄격한 환경에서는 제약이 될 수 있습니다. 셋째, 오픈소스가 아닙니다. 서비스가 중단되거나 가격 정책이 변경되면 다른 벡터 데이터베이스로의 이전이 필요하며, 이 과정에서 API 호환성 문제가 발생할 수 있습니다.

**비용 구조**는 배포 방식에 따라 크게 다릅니다. Pod 기반은 할당한 Pod의 종류, 크기, 수에 따라 시간 단위로 과금됩니다. Pod가 가동되는 동안에는 실제 사용량과 무관하게 비용이 발생합니다. Serverless 방식은 저장된 벡터 수와 읽기/쓰기 단위(Read Unit, Write Unit)에 따라 과금됩니다. 읽기 단위는 Query와 Fetch 호출 시, 쓰기 단위는 Upsert와 Delete 호출 시 소비됩니다. Pinecone은 무료 티어(Free Tier)를 제공하여, 소규모 프로젝트나 프로토타입 단계에서는 비용 없이 사용할 수 있습니다. 무료 티어에서는 하나의 Serverless 인덱스를 생성할 수 있으며, 저장 가능한 벡터 수와 월간 읽기/쓰기 단위에 상한이 있습니다.

비용을 추정할 때 고려해야 할 요소를 정리하면 다음과 같습니다.

```
비용에 영향을 주는 요소

+------------------+-----------------------------+
| 요소             | 설명                        |
+------------------+-----------------------------+
| 벡터 수          | 저장 벡터가 많을수록 비용   |
|                  | 증가 (Serverless 기준)      |
+------------------+-----------------------------+
| 벡터 차원        | 차원이 클수록 저장 공간과   |
|                  | 연산량 증가                 |
+------------------+-----------------------------+
| 메타데이터 크기  | 메타데이터가 클수록 저장    |
|                  | 비용 증가                   |
+------------------+-----------------------------+
| 읽기/쓰기 빈도   | 요청이 잦을수록 비용 증가   |
|                  | (Serverless 기준)           |
+------------------+-----------------------------+
| Pod 종류/수      | 고성능 Pod일수록, 수가      |
|                  | 많을수록 비용 증가 (Pod)    |
+------------------+-----------------------------+
```

프로덕션 환경에서 Pinecone을 사용할 때는 벡터 차원을 필요 이상으로 크게 잡지 않는 것, 메타데이터에 불필요하게 큰 값을 넣지 않는 것, 배치 단위로 Upsert를 묶어 호출 횟수를 줄이는 것이 비용 최적화의 기본 방향입니다.

정리하면, Pinecone은 완전 관리형 클라우드 벡터 데이터베이스로, 인덱스, 네임스페이스, Pod/Serverless 배포 방식이라는 세 가지 아키텍처 개념을 중심으로 동작합니다. Upsert, Query, Fetch, Delete의 간결한 API로 벡터를 관리하고, 메타데이터 필터링으로 검색 범위를 정밀하게 조절할 수 있습니다. 인프라 운영 부담이 없고 API가 간결하다는 장점이 있지만, 세밀한 인프라 제어가 불가능하고 벤더 종속성이 존재한다는 단점도 함께 고려해야 합니다.

다음 단원인 3.2.3에서는 오픈소스 벡터 데이터베이스인 Weaviate를 다룹니다.

이 단원을 마치면 Pinecone의 아키텍처를 설명하고 기본 CRUD 작업을 수행할 수 있습니다.
