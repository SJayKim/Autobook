# 3.2.3 Weaviate

3.2.1에서 벡터 데이터베이스의 개념과 필요성을 다루었습니다. 벡터 데이터베이스는 고차원 벡터를 저장하고, 유사도 기반으로 검색하는 데 특화된 시스템입니다. 3.2.2에서는 그 구체적인 사례로 완전 관리형 서비스인 Pinecone을 살펴보았습니다. Pinecone은 인프라 관리를 서비스 제공자에게 맡기는 대신, 내부 동작을 직접 제어하기 어렵다는 특성이 있었습니다. 이번 단원에서는 오픈소스 벡터 데이터베이스인 Weaviate를 다룹니다. Weaviate는 데이터 저장, 벡터화, 검색을 하나의 시스템 안에서 처리하면서도 소스 코드가 공개되어 있어 직접 설치하고 운영할 수 있습니다. 이 점이 Pinecone과 가장 크게 다른 지점입니다.

Weaviate가 다른 벡터 데이터베이스와 구별되는 특징을 이해하려면, 먼저 일반적인 벡터 데이터베이스의 작업 흐름을 떠올릴 필요가 있습니다. 보통 벡터 데이터베이스를 사용할 때는 세 단계를 거칩니다. 첫째, 원본 데이터(텍스트, 이미지 등)를 외부 임베딩 모델에 보내 벡터로 변환합니다. 둘째, 변환된 벡터를 데이터베이스에 저장합니다. 셋째, 검색 시점에 쿼리를 같은 임베딩 모델로 벡터화한 뒤, 데이터베이스에서 유사한 벡터를 찾습니다. 이 흐름에서 임베딩 변환은 데이터베이스 바깥에서 이루어집니다. 애플리케이션 코드가 임베딩 모델 호출과 데이터베이스 호출을 각각 관리해야 합니다. Weaviate는 이 임베딩 변환 단계를 데이터베이스 내부로 끌어들였습니다. 원본 텍스트나 이미지를 Weaviate에 직접 넣으면, Weaviate가 스스로 벡터화를 수행하고 저장까지 마칩니다. 이것이 Weaviate 아키텍처의 핵심 설계 사상입니다.

Weaviate의 전체 아키텍처는 크게 세 부분으로 나눌 수 있습니다. **코어 엔진**, **모듈 시스템**, **API 계층**입니다.

코어 엔진은 데이터 저장과 인덱싱을 담당합니다. Weaviate는 벡터 인덱스로 HNSW(Hierarchical Navigable Small World) 알고리즘을 기본 사용합니다. HNSW는 그래프 구조를 이용해 유사한 벡터를 빠르게 탐색하는 근사 최근접 이웃(ANN) 알고리즘입니다. 벡터뿐 아니라 원본 데이터와 메타데이터도 함께 저장하므로, 별도의 데이터 저장소 없이 Weaviate 하나로 벡터와 원본을 모두 관리할 수 있습니다. 또한 역색인(inverted index)을 내장하고 있어, 키워드 기반의 필터링과 검색도 코어 엔진 수준에서 지원합니다.

모듈 시스템은 Weaviate를 다른 벡터 데이터베이스와 가장 뚜렷하게 구분하는 부분입니다. 모듈이란, Weaviate의 핵심 기능에 플러그인 형태로 추가되는 확장 구성 요소를 뜻합니다. 벡터화 모듈, 생성 모듈, 리더 모듈 등 여러 종류가 있으며, 필요한 모듈만 골라 활성화할 수 있습니다. 예를 들어, 텍스트를 벡터로 변환하는 모듈을 활성화하면, 사용자는 원본 텍스트만 Weaviate에 전달하고 벡터화는 Weaviate가 내부에서 처리합니다. 이 모듈 시스템 덕분에 Weaviate는 단순한 벡터 저장소가 아니라, 데이터 입력부터 벡터 변환, 검색, 결과 생성까지 하나의 파이프라인으로 엮을 수 있는 플랫폼이 됩니다.

API 계층은 외부 애플리케이션이 Weaviate와 통신하는 창구입니다. Weaviate는 REST API와 GraphQL API를 모두 제공합니다. GraphQL에 대해서는 뒤에서 자세히 다룹니다.

이 세 부분의 관계를 다이어그램으로 나타내면 다음과 같습니다.

```
Weaviate 아키텍처

+--------------------------------------------------+
|                  API 계층                         |
|          REST API  /  GraphQL API                |
+--------------------------------------------------+
           |                    |
           v                    v
+--------------------------------------------------+
|               모듈 시스템                         |
|  +------------+  +------------+  +------------+  |
|  | text2vec-  |  | img2vec-   |  | generative-|  |
|  | openai     |  | neural     |  | openai     |  |
|  +------------+  +------------+  +------------+  |
|        ...           ...            ...          |
+--------------------------------------------------+
           |
           v
+--------------------------------------------------+
|                코어 엔진                          |
|  +----------------+  +-----------------------+   |
|  | HNSW 벡터      |  | 역색인                |   |
|  | 인덱스         |  | (키워드/필터링)       |   |
|  +----------------+  +-----------------------+   |
|  +-------------------------------------------+   |
|  | 객체 저장소 (벡터 + 원본 + 메타데이터)     |   |
|  +-------------------------------------------+   |
+--------------------------------------------------+
```

사용자가 API를 통해 데이터를 보내면, 모듈 시스템이 벡터화를 수행하고, 코어 엔진이 벡터와 원본 데이터를 함께 저장합니다. 검색 요청이 들어오면 코어 엔진이 HNSW 인덱스와 역색인을 활용하여 결과를 반환합니다.

Weaviate에서 데이터를 저장하려면, 먼저 **스키마**(schema)를 정의해야 합니다. 스키마란 데이터의 구조를 미리 선언하는 것으로, 관계형 데이터베이스에서 테이블 구조를 CREATE TABLE 문으로 정의하는 것과 비슷한 역할을 합니다. 다만 Weaviate의 스키마는 테이블이 아니라 **클래스**(class)라는 단위를 사용합니다.

클래스는 같은 종류의 데이터 객체를 묶는 단위입니다. 예를 들어, 뉴스 기사를 저장하려면 "Article"이라는 클래스를 만들고, 각 기사가 하나의 객체가 됩니다. 클래스를 정의할 때는 그 클래스에 속하는 객체가 어떤 속성(property)을 가지는지 함께 선언합니다. Article 클래스라면 "title", "content", "category" 같은 속성을 정의하는 식입니다. 각 속성에는 데이터 타입(text, int, boolean 등)을 지정합니다.

클래스를 정의하는 JSON 구조를 예로 들면 다음과 같습니다.

```json
{
  "class": "Article",
  "description": "뉴스 기사를 저장하는 클래스",
  "vectorizer": "text2vec-openai",
  "properties": [
    {
      "name": "title",
      "dataType": ["text"],
      "description": "기사 제목"
    },
    {
      "name": "content",
      "dataType": ["text"],
      "description": "기사 본문"
    },
    {
      "name": "category",
      "dataType": ["text"],
      "description": "기사 분류"
    },
    {
      "name": "publishedAt",
      "dataType": ["date"],
      "description": "발행 일시"
    }
  ]
}
```

이 정의에서 "class" 필드는 클래스 이름을 지정합니다. "vectorizer" 필드는 이 클래스에 속하는 객체를 벡터화할 때 어떤 모듈을 사용할지 지정합니다. 여기서는 "text2vec-openai"로 설정했으므로, 객체가 저장될 때 OpenAI의 임베딩 모델이 자동으로 호출되어 텍스트를 벡터로 변환합니다. "properties" 배열은 각 속성의 이름, 데이터 타입, 설명을 담습니다. 이렇게 클래스를 정의하면, 이후 해당 클래스에 객체를 추가할 때 각 속성에 맞는 값을 넣으면 됩니다. 벡터는 사용자가 직접 계산해서 넘길 필요 없이, vectorizer로 지정한 모듈이 자동으로 생성합니다.

스키마에서 한 가지 더 짚을 점은 클래스 간의 **참조**(cross-reference)입니다. 관계형 데이터베이스에서 외래 키(foreign key)로 테이블 간 관계를 맺듯이, Weaviate에서도 한 클래스의 객체가 다른 클래스의 객체를 참조할 수 있습니다. 예를 들어, "Article" 클래스의 객체가 "Author" 클래스의 객체를 참조하도록 설정하면, 기사 검색 결과에서 저자 정보까지 함께 조회할 수 있습니다. 이 참조는 속성의 dataType에 다른 클래스 이름을 지정하는 방식으로 정의합니다.

이제 벡터화 모듈을 좀 더 자세히 살펴보겠습니다. 앞서 Weaviate의 모듈 시스템이 벡터화를 내부에서 처리한다고 설명했습니다. Weaviate는 다양한 벡터화 모듈을 제공하며, 대표적인 것이 **text2vec** 계열과 **img2vec** 계열입니다.

text2vec 모듈은 텍스트 데이터를 벡터로 변환합니다. 여러 변형이 있으며, 각각 다른 임베딩 모델이나 서비스에 연결됩니다. text2vec-openai는 OpenAI의 임베딩 API를 호출하여 벡터를 생성합니다. text2vec-cohere는 Cohere의 임베딩 API를 사용합니다. text2vec-huggingface는 Hugging Face에서 제공하는 모델을 활용합니다. text2vec-transformers는 로컬에 트랜스포머 모델을 직접 올려 벡터화를 수행하므로, 외부 API 호출 없이 자체 인프라에서 임베딩을 생성할 수 있습니다. 어떤 모듈을 선택하느냐에 따라 벡터의 차원 수, 품질, 비용, 지연 시간이 달라집니다. 이 선택은 스키마에서 클래스별로 지정하므로, 하나의 Weaviate 인스턴스에서 클래스마다 서로 다른 벡터화 모듈을 사용할 수도 있습니다.

img2vec 모듈은 이미지 데이터를 벡터로 변환합니다. img2vec-neural은 사전훈련된 이미지 인식 모델(예: ResNet)을 이용해 이미지의 특성을 벡터로 추출합니다. 이 모듈을 활성화하면, 이미지 파일을 Weaviate에 저장할 때 자동으로 이미지 벡터가 생성됩니다. 이를 통해 "이 이미지와 비슷한 이미지를 찾아라"와 같은 유사 이미지 검색을 구현할 수 있습니다. 나아가 text2vec과 img2vec을 함께 사용하면, 텍스트와 이미지를 같은 벡터 공간에 놓고 교차 검색하는 멀티모달 검색도 가능해집니다.

벡터화 모듈의 동작 과정을 단계별로 정리하면 다음과 같습니다.

```
벡터화 모듈 동작 흐름

1. 사용자가 원본 데이터(텍스트/이미지)를 API로 전송
          |
          v
2. Weaviate가 해당 클래스의 vectorizer 설정을 확인
          |
          v
3. 지정된 벡터화 모듈이 원본 데이터를 벡터로 변환
   (외부 API 호출 또는 로컬 모델 실행)
          |
          v
4. 코어 엔진이 원본 데이터 + 벡터 + 메타데이터를 함께 저장
          |
          v
5. HNSW 인덱스에 새 벡터를 추가하여 검색 가능 상태로 전환
```

이 과정에서 사용자는 벡터를 직접 다루지 않습니다. 원본 데이터를 보내기만 하면, 벡터화와 인덱싱이 Weaviate 내부에서 자동으로 이루어집니다. 물론 이미 외부에서 생성한 벡터가 있다면, 벡터를 직접 지정하여 저장하는 것도 가능합니다.

Weaviate의 또 다른 핵심 기능은 **하이브리드 검색**(hybrid search)입니다. 하이브리드 검색이란, 벡터 유사도 검색과 키워드 검색을 결합하는 방식을 뜻합니다. 이 두 검색 방식이 왜 결합되어야 하는지 배경부터 살펴보겠습니다.

벡터 유사도 검색은 의미적 유사성을 기준으로 결과를 찾습니다. "강아지 산책 방법"을 검색하면 "반려견 외출 요령"처럼 단어는 다르지만 의미가 비슷한 문서도 찾아냅니다. 그러나 벡터 검색에는 약점이 있습니다. 고유명사, 제품 번호, 특정 코드와 같이 정확한 문자열 일치가 중요한 경우, 벡터 유사도만으로는 정밀한 결과를 보장하기 어렵습니다. "Python 3.12"를 검색했을 때 "프로그래밍 언어" 관련 문서가 폭넓게 나올 수 있지만, 정확히 "3.12" 버전을 언급한 문서를 놓칠 수 있습니다.

반대로 키워드 검색은 문자열 일치에 강합니다. BM25 같은 알고리즘을 사용하며, 검색어에 포함된 단어가 문서에 실제로 등장하는지, 얼마나 자주 등장하는지를 기준으로 점수를 매깁니다. 고유명사나 특정 용어를 정확히 찾는 데 유리하지만, 동의어나 문맥상 관련 있는 문서를 놓치는 한계가 있습니다.

하이브리드 검색은 이 두 방식의 결과를 합산합니다. Weaviate에서는 "alpha"라는 매개변수로 두 검색의 가중치를 조절합니다. alpha 값이 0이면 키워드 검색만 사용하고, 1이면 벡터 검색만 사용합니다. 0.5로 설정하면 두 검색의 점수를 동등하게 반영합니다. 이를 통해 의미적 유사성과 문자열 정확도를 동시에 확보할 수 있습니다.

하이브리드 검색의 내부 동작을 정리하면 다음과 같습니다.

```
하이브리드 검색 흐름

사용자 쿼리: "Python 3.12 설치 방법"
          |
          +-------------------+-------------------+
          |                                       |
          v                                       v
   벡터 검색 (HNSW)                      키워드 검색 (BM25)
   쿼리를 벡터로 변환 후                  역색인에서 "Python",
   코사인 유사도 기준 탐색                "3.12", "설치" 매칭
          |                                       |
          v                                       v
   벡터 점수 목록                         키워드 점수 목록
          |                                       |
          +-------------------+-------------------+
                              |
                              v
                  점수 합산 (alpha로 가중치 조절)
                              |
                              v
                     최종 결과 순위 반환
```

이 방식 덕분에 "Python 설치 가이드"처럼 의미적으로 관련된 문서와 "3.12"라는 문자열이 정확히 포함된 문서를 모두 상위에 올릴 수 있습니다.

Weaviate의 API 계층에서 특히 주목할 부분은 **GraphQL API**입니다. GraphQL은 페이스북(현 Meta)이 2015년에 공개한 쿼리 언어로, REST API와는 다른 접근 방식을 취합니다. REST API에서는 서버가 정해 놓은 엔드포인트(URL)별로 고정된 형태의 데이터를 받습니다. 예를 들어, 기사 목록을 가져오는 엔드포인트와 기사 상세를 가져오는 엔드포인트가 분리되어 있고, 각각이 서버가 정한 모든 필드를 반환합니다. 반면 GraphQL에서는 클라이언트가 필요한 필드만 골라서 요청합니다. 하나의 엔드포인트에 쿼리를 보내되, 쿼리 안에서 "어떤 데이터의 어떤 필드를 원하는지"를 명시합니다.

Weaviate에서 GraphQL로 벡터 검색을 수행하는 쿼리 예시를 보겠습니다.

```graphql
{
  Get {
    Article(
      nearText: {
        concepts: ["인공지능의 미래"]
      }
      limit: 5
    ) {
      title
      content
      category
      _additional {
        distance
        certainty
      }
    }
  }
}
```

이 쿼리를 한 줄씩 살펴보겠습니다. "Get" 블록은 데이터를 조회하겠다는 뜻입니다. "Article"은 조회 대상 클래스입니다. "nearText"는 벡터 유사도 검색을 수행하라는 연산자이며, "concepts"에 넣은 텍스트를 벡터로 변환하여 가장 가까운 객체를 찾습니다. "limit: 5"는 상위 5건만 가져오라는 뜻입니다. 그 아래의 "title", "content", "category"는 반환받을 속성입니다. "_additional" 블록에서 "distance"는 쿼리 벡터와 결과 벡터 사이의 거리, "certainty"는 유사도 확신 점수를 반환합니다. 이처럼 GraphQL을 통해, 검색 조건과 반환 필드를 하나의 쿼리에서 유연하게 조합할 수 있습니다.

하이브리드 검색도 GraphQL로 수행할 수 있습니다.

```graphql
{
  Get {
    Article(
      hybrid: {
        query: "Python 3.12 설치"
        alpha: 0.5
      }
      limit: 5
    ) {
      title
      content
      _additional {
        score
      }
    }
  }
}
```

"hybrid" 연산자가 벡터 검색과 키워드 검색을 동시에 실행하며, "alpha"로 가중치를 조절합니다. "_additional"의 "score"는 두 검색 점수를 합산한 최종 점수입니다. REST API로도 동일한 작업을 수행할 수 있지만, GraphQL은 반환 필드를 쿼리 시점에 선택할 수 있으므로, 네트워크로 전달되는 데이터 양을 줄이고 클라이언트 코드를 간결하게 유지하는 데 유리합니다.

마지막으로 Weaviate를 사용하는 방식에 대해 다루겠습니다. Weaviate는 크게 두 가지 방식으로 운영할 수 있습니다. **셀프 호스팅**(self-hosting)과 **클라우드 서비스**입니다.

셀프 호스팅은 Weaviate를 직접 서버에 설치하여 운영하는 방식입니다. Weaviate는 Docker 이미지를 공식 제공하므로, docker-compose 파일 하나로 로컬 환경이나 자체 서버에 Weaviate 인스턴스를 띄울 수 있습니다. Kubernetes 환경에서는 Helm 차트를 이용하여 클러스터로 배포할 수도 있습니다. 셀프 호스팅의 장점은 데이터가 자체 인프라를 벗어나지 않아 데이터 주권을 확보할 수 있다는 점, 그리고 리소스 할당, 모듈 구성, 네트워크 설정 등을 세밀하게 제어할 수 있다는 점입니다. 반면 서버 관리, 백업, 스케일링, 보안 패치 등 운영 부담을 직접 져야 합니다.

클라우드 서비스는 Weaviate 팀이 운영하는 관리형 서비스인 Weaviate Cloud(WCD)를 이용하는 방식입니다. 인프라 설정 없이 웹 콘솔이나 API로 클러스터를 생성하면, Weaviate 팀이 서버 운영, 업그레이드, 모니터링을 대신 처리합니다. 빠르게 시작하고 싶거나 인프라 운영 인력이 부족한 경우에 적합합니다. 다만 데이터가 외부 클라우드에 저장되므로, 데이터 보안 규정이 엄격한 환경에서는 제약이 될 수 있습니다.

두 방식의 특성을 비교하면 다음과 같습니다.

| 항목 | 셀프 호스팅 | 클라우드 (WCD) |
|------|-----------|---------------|
| 초기 설정 | Docker/Kubernetes 직접 구성 | 웹 콘솔에서 클러스터 생성 |
| 데이터 위치 | 자체 인프라 | Weaviate 관리 클라우드 |
| 운영 부담 | 높음 (백업, 스케일링, 패치) | 낮음 (서비스 제공자가 관리) |
| 제어 수준 | 높음 (전체 설정 가능) | 제한적 (제공 옵션 내에서 선택) |
| 비용 구조 | 인프라 비용 직접 부담 | 사용량 기반 과금 |
| 적합 상황 | 데이터 주권 필수, 고도 커스터마이징 | 빠른 시작, 운영 인력 부족 |

실무에서는 개발 단계에서 Docker로 로컬 인스턴스를 띄워 기능을 검증하고, 프로덕션에서는 요구 사항에 따라 셀프 호스팅 또는 클라우드를 선택하는 패턴이 일반적입니다.

정리하면, Weaviate는 벡터화 모듈을 데이터베이스 내부에 통합하여 원본 데이터 입력부터 벡터 변환, 저장, 검색까지 하나의 시스템에서 처리하는 오픈소스 벡터 데이터베이스입니다. 클래스와 속성으로 구성되는 스키마를 통해 데이터 구조를 정의하고, text2vec과 img2vec 같은 모듈로 텍스트와 이미지를 자동 벡터화하며, 벡터 검색과 키워드 검색을 결합하는 하이브리드 검색을 지원합니다. GraphQL API를 통해 유연한 쿼리가 가능하고, 셀프 호스팅과 클라우드 서비스를 모두 제공하여 운영 환경에 맞게 선택할 수 있습니다.
