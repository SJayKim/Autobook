# 7.1.2 CI 실패 흡수와 자가 수정

7.1.1에서 짠 루프는 로컬에서 변경 직후에 도는 빠른 검증이었습니다. 같은 변경이 원격 저장소로 푸시된 뒤에는 그보다 무거운 검증이 한 단계 더 기다리고 있습니다. CI 빌드, 통합 테스트, 보안 스캔처럼 푸시 후에 자동으로 도는 작업들이 그것입니다. 사람이 PR 페이지를 새로 고치며 빨간 표시를 발견하는 대신, 에이전트가 실패 사실과 로그를 직접 가져와 수정에 들어가게 만들면 한 사이클이 짧아집니다. 이 단원은 그 흐름을 두 도구에서 어떻게 구성하는지를 정리합니다.

먼저 CI 실패 로그를 가져오는 표준 명령부터 풀어 두겠습니다. GitHub의 워크플로 실행 정보는 `gh` CLI로 받아 올 수 있습니다.

```bash
gh run list --branch "$(git rev-parse --abbrev-ref HEAD)" \
            --limit 5 --json databaseId,status,conclusion,name

gh run view 12345678 --log-failed
```

첫 명령은 현재 체크아웃한 브랜치의 최근 다섯 개 워크플로 실행을 가져옵니다. `--branch` 뒤의 명령 치환은 현재 브랜치 이름을 자동으로 채워 줍니다. `--json` 플래그는 사람이 보기 위한 표 대신 기계가 읽기 좋은 JSON 형태로 결과를 받게 하며, 모델이 status와 conclusion만 뽑아 실패 한 건을 고르기 쉽도록 만듭니다. 둘째 명령은 그렇게 고른 실행의 식별자를 넣어 실패한 단계의 로그만 잘라 받습니다. `--log-failed`가 핵심이며, 성공한 단계의 출력이 컨텍스트로 따라 들어오지 않게 막아 줍니다.

같은 동작을 더 부드럽게 묶으려면 5.2.2에서 본 MCP github 서버를 활용합니다. github 서버는 워크플로 실행 목록 조회와 로그 가져오기를 도구로 노출하므로, 에이전트가 셸 명령 대신 도구 호출 한 번으로 같은 정보를 얻습니다. 셸 호출 결과를 직접 다룰 때보다 입력과 출력이 구조화되어 있어, 모델이 실패 단계의 이름과 로그 텍스트를 정확한 필드에서 꺼내기가 수월합니다.

전체 루프를 그리면 다음과 같습니다.

```
[로컬 변경] → [7.1.1 빠른 검증] → [커밋 + 푸시]
                                          ↓
                                  [CI 워크플로 실행]
                                          ↓
                              실패 → 로그 회수 (gh / MCP)
                                          ↓
                              모델이 로그를 읽고 수정
                                          ↓
                              [재커밋 + 푸시] → [CI 재시도]
                                          ↓
                                  성공 → 종료
                                  반복 → 상한 도달 시 사람 검토
```

이 그림에서 새로 추가된 부분은 푸시 이후의 짧은 폐회로입니다. 7.1.1의 로컬 루프는 한 번 변경에 한 번 검증으로 닫혔지만, CI 루프는 변경과 검증 사이에 푸시와 워크플로 실행이 끼어 있어 한 사이클의 길이가 더 깁니다. 모델이 직접 이 사이클을 운영할 때는 매 단계에서 멈추는 자연스러운 지점을 두는 것이 안전합니다. 푸시 뒤에는 곧장 결과를 확인하러 가지 말고, 6.2.2에서 본 백그라운드 모니터링처럼 워크플로가 끝났는지를 가끔 한 줄 명령으로 확인하다가 끝났을 때 그 시점에 로그를 가져오는 식이 무난합니다.

수정 절차의 사이클이 너무 짧으면 비용이 빠르게 누적됩니다. CI 한 회 실행 자체가 분 단위 시간과 러너 비용을 쓰며, 그 로그를 모델 컨텍스트로 읽는 데 토큰이 듭니다. 그래서 사이클의 한 회 비용을 의식적으로 줄여 두어야 합니다. 가장 단순한 절감은 **실패 단계의 로그만** 가져오는 것입니다. 위의 `gh run view --log-failed`가 같은 효과를 줍니다. 다음은 **로그의 마지막 일정 구간**만 잘라 보는 것입니다. 트레이스백은 보통 끝부분이 가장 정보 밀도가 높으므로, 모델에게 마지막 200줄 또는 마지막 한 매처에 걸린 부분만 보여 주어도 원인 파악에는 충분한 경우가 많습니다. 마지막은 **재시도 횟수 상한**입니다. 같은 종류의 실패가 두세 번 연달아 나면 더 비싸지기 전에 멈추고 사람에게 보고합니다.

상한이라는 말을 좀 더 풀어 두겠습니다. 무한 자가 수정 루프는 시간이 갈수록 점점 더 비싸지는 안티 패턴입니다. 같은 자리에서 같은 실패가 반복되는 경우, 모델이 본질을 짚지 못하고 표면만 매번 다르게 고치는 경우, 그리고 변경이 새로운 실패를 부르는 회귀가 누적되는 경우입니다. 단순한 규칙으로 막을 수 있습니다. CI 자가 수정 사이클은 최대 두 번까지만 자동으로 돌리고, 세 번째 실패에서는 멈춰 사람에게 PR 코멘트로 상황을 정리해 두는 형태가 무난합니다. 이 숫자는 팀과 작업 성격에 맞춰 조정합니다.

Claude Code 측에서 이 루프를 묶는 한 방법은 4.1.4의 UserPromptSubmit hook으로 정책을 주입하는 것입니다. 예를 들어 PR 작업 브랜치에서 세션을 시작할 때 'CI 자가 수정은 두 번까지만 자동으로 진행한다, 세 번째 실패에서는 멈춘다'라는 한 문장을 자동으로 프롬프트에 첨부해 두면, 모델이 사이클을 의식하며 행동합니다. 더 강하게 묶고 싶다면 4.1.2의 PreToolUse hook에서 `git push`나 `gh run rerun` 호출 횟수를 세션 단위로 세어 상한을 강제하는 스크립트를 둘 수 있습니다.

Codex CLI 측에서는 6.2.1에서 본 profile 분리와 비대화형 실행이 같은 역할을 합니다. 자가 수정 사이클을 다음과 같은 두 profile로 분리해 둡니다.

```toml
# ~/.codex/config.toml

[profiles.ci-diagnose]
sandbox_mode = "read-only"
approval_policy = "never"

[profiles.ci-fix]
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

`ci-diagnose`는 실패 로그를 읽고 원인을 정리하는 단계입니다. read-only이므로 진단 도중에 코드가 바뀌는 사고가 막혀 있습니다. `ci-fix`는 그 진단 결과를 토대로 코드를 고치는 단계입니다. workspace-write이므로 파일 쓰기가 허용되고, approval policy는 on-request로 두어 그 한계를 넘는 동작은 사용자 확인을 거치게 합니다. 한 사이클을 다음 명령 묶음으로 실행할 수 있습니다.

```bash
codex --profile ci-diagnose exec \
  "gh run view $RUN_ID --log-failed 를 읽고 원인을 \
   docs/notes/ci-$RUN_ID.md 후보 위치에 정리해 줘 (쓰기 불가하니 본문만 출력)"

codex --profile ci-fix exec \
  "docs/notes/ci-$RUN_ID.md 의 진단을 보고 최소 변경으로 수정해 줘. \
   변경 후 scripts/check-changed.sh 를 반드시 돌리고 멈출 것."
```

두 호출의 분리가 핵심입니다. 첫 호출에서는 로그를 읽고 원인을 본문으로만 출력합니다. 둘째 호출에서 그 본문을 입력으로 받아 최소 변경을 시도하고, 7.1.1의 로컬 검증 스크립트를 한 번 더 돌려 빠른 실패는 자체적으로 흡수합니다. 사이클을 두 단계로 끊어 두면 진단과 수정 사이에 사람이 끼어들기 좋은 자연스러운 지점이 생기고, 진단이 정확하지 않다고 판단되면 수정 단계로 넘어가지 않고 멈출 수 있습니다.

수정 → 푸시 → CI 확인 루프의 한계도 한 번 짚어 두는 것이 좋습니다. CI는 빠른 검증의 마지막 보루이지만, 실패 메시지가 항상 원인을 가리키지는 않습니다. 의존성 캐시 문제, 러너 환경의 차이, 외부 서비스 일시 장애처럼 코드 변경으로는 해결되지 않는 실패가 섞여 있습니다. 모델이 이런 실패에 대해서도 코드 변경을 시도하면 잘 돌던 코드를 무리하게 손대게 됩니다. 자가 수정 루프 입구에 '같은 종류의 실패가 두 번 연속이면 환경 의심으로 분류한다'는 짧은 규칙을 두어 분기를 갈라 두면, 환경 문제는 사람에게 보고로, 코드 문제는 수정으로 향하게 만들 수 있습니다.

정리하면, CI 실패 흡수 루프는 `gh run view --log-failed`나 MCP github 서버로 실패 로그만 좁게 가져오고, 두 단계로 나눈 진단·수정 사이클을 재시도 상한과 함께 운영하는 구조입니다. Claude Code는 UserPromptSubmit·PreToolUse hook으로 사이클 규칙을 강제하고, Codex CLI는 read-only 진단과 workspace-write 수정의 두 profile을 codex exec로 차례로 부르며 사이클을 끊어 둡니다. 사이클의 한 회 비용을 줄이기 위해 로그를 좁게 자르고 상한을 분명히 두는 일이 둘 다 똑같이 중요합니다.

다음 단원인 7.2.1에서는 이렇게 만들어진 변경이 메인 브랜치에 안전하게 합쳐지도록 Git workflow를 두 도구에서 동일하게 통합하는 방법을 다룹니다.

이 단원을 마치면 CI 실패 로그를 두 도구가 자동으로 가져와 수정하는 루프를 구성하고, 그 한계와 안전장치를 함께 설명할 수 있게 됩니다.
