# 8.1.1 비용·모델 선택 가이드

코딩 보조 도구를 본격적으로 쓰기 시작하면 첫 달의 청구서가 예상보다 큰 경우가 많습니다. 모델을 잘못 골랐기 때문일 수도 있고, 같은 토큰을 여러 번 보내는 패턴이 굳어졌기 때문일 수도 있습니다. 이 단원에서는 두 도구에서 사용 가능한 모델군을 비교하고, 토큰을 줄이는 기본 패턴과 비용이 폭증하는 안티 패턴을 함께 짚습니다.

먼저 모델 선택부터 봅니다. Claude 계열은 동일한 시점에 일반적으로 세 등급으로 나뉘어 제공됩니다. 가장 큰 모델은 **Opus**입니다. 어려운 설계, 복잡한 리팩터링, 코드베이스 전체를 따라가야 하는 조사 작업에 적합합니다. 다만 토큰당 단가가 가장 높습니다.

중간 등급이 **Sonnet**입니다. Sonnet은 비용과 능력의 균형이 좋아 일상적인 코드 변경, 테스트 작성, 일반 문서 작성에 가장 자주 쓰입니다. 처음 한 도구를 정착시킬 때 기본값으로 두기 좋은 선택입니다.

가장 빠르고 저렴한 등급이 **Haiku**입니다. 짧은 분류, 간단한 텍스트 변환, 빠른 요약 같은 단순 반복 작업에 적합합니다. 사고가 깊지 않아도 되는 일을 모두 Opus로 돌리면 비용이 크게 새므로, 가능한 일은 Haiku로 옮기는 판단이 중요합니다.

Codex CLI는 `config.toml`의 `model` 필드로 사용할 모델을 지정합니다. 2.2.2에서 본 것처럼 `[profiles.NAME]` 블록 안에서도 `model`을 다시 적을 수 있어, 프로파일마다 다른 모델을 둘 수 있습니다. 정찰용 프로파일에는 빠른 모델을, 본 작업 프로파일에는 강한 모델을 두는 식의 분리가 자연스럽습니다.

두 도구의 모델 선택을 한눈에 비교하면 다음과 같습니다.

| 작업 성격 | Claude Code 권장 | Codex CLI 권장 |
|-----------|------------------|----------------|
| 코드베이스 전반 설계, 어려운 디버깅 | Opus | 상위 추론 모델 |
| 일상 코드 변경, 테스트 추가 | Sonnet | 중급 모델 |
| 짧은 요약, 분류, 포맷 변환 | Haiku | 경량 모델 |
| 정찰·읽기 전용 작업 | Sonnet 또는 Haiku | 경량 모델 + read-only 프로파일 |
| 대화 없이 일회성 실행 | Sonnet | 중급 모델 + `codex exec` |

```toml
# ~/.codex/config.toml
model = "gpt-5-codex"

[profiles.recon]
model = "gpt-5-mini"
sandbox_mode = "read-only"

[profiles.build]
model = "gpt-5-codex"
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

위 설정에서는 평소 호출은 본 모델을 쓰고, `--profile recon`을 붙이면 정찰만 가벼운 모델로 진행합니다. Claude Code도 동일한 의도를 `settings.json`의 `model` 필드와 슬래시 커맨드 조합으로 구현합니다. 자주 도는 단순 작업에 별도 슬래시 커맨드를 만들어 거기서만 가벼운 모델을 호출하게 두면 비용 분산이 명확해집니다.

모델 선택 다음으로 큰 비용 변수는 **반복 전송되는 토큰**입니다. 한 세션에서 시스템 프롬프트, CLAUDE.md, AGENTS.md, 이전 대화는 매 턴마다 모델에게 다시 보내집니다. **프롬프트 캐싱**은 같은 머리말이 반복될 때 이미 본 부분을 다시 계산하지 않게 해 주는 기능입니다. 두 도구 모두 캐시 가능한 구간을 자동으로 묶어 비용을 깎으므로, 사용자 입장에서는 변하지 않는 머리말을 자주 흔들지 않는 것이 핵심입니다.

캐시를 살리려면 두 가지 습관이 필요합니다. 하나는 CLAUDE.md와 AGENTS.md를 한 세션 중에 자주 고치지 않는 것입니다. 다른 하나는 SessionStart hook이나 UserPromptSubmit hook으로 주입하는 텍스트의 앞부분을 안정적으로 유지하는 것입니다. 날짜처럼 매번 바뀌는 값은 가급적 뒤쪽에 두고, 변하지 않는 정책 문구는 앞에 둡니다.

비용이 폭증하는 대표 안티 패턴 세 가지를 짚어 두겠습니다. 첫째, **긴 출력 강제**입니다. '전체 파일을 그대로 다시 출력해 달라'는 요청이 잦으면, 모델은 매번 거대한 코드를 토해 냅니다. 둘째, **무한 검색 루프**입니다. 4.1.3의 PostToolUse hook이 매 편집 후 전체 테스트를 돌리고, 그 결과가 다시 컨텍스트로 들어와 또 도구 호출을 유발하는 식의 흐름은 토큰을 빠르게 소모합니다. 셋째, **거대한 첨부**입니다. 100MB짜리 로그를 그대로 붙이지 않고 `tail -n 200` 정도로 잘라 보내는 것이 일반적인 규율입니다.

모델 선택과 캐시 활용을 잡았다면 마지막은 **모니터링**입니다. 월간 사용량을 손으로 들여다보지 않으면, 한 명의 잘못된 루프가 한 달치 예산을 다 써 버리는 사고가 종종 발생합니다. 다음 정도의 지표를 정기적으로 확인합니다. 하루 사용된 입력·출력 토큰의 총합, 모델별 분포, 캐시 적중률, 가장 비싼 세션 상위 다섯 개의 길이입니다. 임계치 알람을 설정할 수 있다면, 일일 한도의 70퍼센트 도달 시점에 한 번, 100퍼센트 초과 시점에 한 번 알림을 보내는 두 단계 구조가 무난합니다.

정리하면, 작업이 어렵지 않으면 모델을 한 단계 낮추고, 변하지 않는 머리말을 흔들지 않아 캐시를 살리며, 긴 출력과 무한 루프를 만들지 않도록 hook과 명령을 점검하고, 모델별 토큰 사용량을 정기적으로 들여다보아야 비용이 통제됩니다.

다음 단원인 8.1.2에서는 이렇게 정한 모델·hook·정책을 팀이 함께 쓰도록 설정 파일을 공유 가능한 부분과 비공개 부분으로 가르는 방법을 다룹니다.

이 단원을 마치면 작업 성격에 맞는 모델을 두 도구에서 고르고, 프롬프트 캐싱을 살리는 설정 습관을 갖추며, 비용 폭증 안티 패턴을 식별해 차단하고, 월간 사용량을 모니터링할 지표를 정할 수 있습니다.
