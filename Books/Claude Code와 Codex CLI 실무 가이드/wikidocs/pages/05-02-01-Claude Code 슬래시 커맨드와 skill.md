# 5.2.1 Claude Code 슬래시 커맨드와 skill

같은 지시를 매번 길게 풀어쓰는 일이 반복되면 시간이 새고 오타로 동작이 달라지기도 합니다. Claude Code는 자주 쓰는 지시를 한 줄 명령으로 묶을 수 있는 두 가지 장치를 제공합니다. 하나는 슬래시 커맨드이고, 다른 하나는 skill입니다. 이번 단원에서는 두 장치의 위치, 자동 로딩 조건, 인자 치환 문법을 차례로 풀고, Codex CLI에서 동등한 효과를 내는 방법까지 함께 봅니다.

먼저 슬래시 커맨드를 봅니다. Claude Code는 세션 입력 창에서 `/`로 시작하는 메시지를 만나면 일반 메시지가 아니라 명령으로 해석합니다. 명령의 본체는 마크다운 파일 한 개로 적습니다. 파일이 놓이는 곳은 정해져 있어서, 프로젝트 공용 명령은 `.claude/commands/NAME.md`에 두고, 사용자 개인 명령은 `~/.claude/commands/NAME.md`에 둡니다. `NAME`이 그대로 명령 이름이 됩니다. 예를 들어 `.claude/commands/review.md`를 두면 세션에서 `/review`로 호출합니다.

명령 파일의 맨 위에는 작은 머리표가 들어갑니다. 이를 frontmatter라고 부르며, 명령의 설명·인자 안내·허용 도구 같은 메타데이터를 담습니다. 본문에는 명령이 실제로 실행할 지시를 자연어로 적습니다. 다음 예시가 두 부분의 관계를 보여 줍니다.

```markdown
---
description: 변경된 파일에 대해 코드 리뷰를 수행한다
argument-hint: <PR번호 또는 비워두면 현재 diff>
allowed-tools: Read, Grep, Bash(git diff:*)
---

$ARGUMENTS 에 해당하는 변경분을 읽고 다음 항목을 점검한다.
- 새 함수의 입력 검증
- 테스트 누락 가능성
- 인접 코드와의 스타일 불일치
점검 결과는 짧은 문단으로 요약하고, 수정 제안은 파일·라인 단위로 정리한다.
```

머리표의 `description`은 명령 목록을 띄울 때 한 줄로 보여 주는 설명입니다. `argument-hint`는 사용자가 어떤 인자를 넘겨야 하는지 안내하는 힌트입니다. `allowed-tools`는 이 명령이 실행되는 동안 사용 가능한 도구를 좁히는 화이트리스트입니다. `Bash(git diff:*)` 같은 표기는 3.1.1에서 익힌 도구 매처와 같은 문법으로, 특정 명령만 허용한다는 의미를 갖습니다.

본문에서 가장 중요한 토큰은 `$ARGUMENTS`입니다. 사용자가 `/review 123`을 입력하면, Claude Code는 명령 본문의 `$ARGUMENTS` 자리에 문자열 `123`을 끼워 넣은 뒤 모델에게 보냅니다. 인자를 비우고 `/review`만 호출하면 빈 문자열로 치환됩니다. 인자 한 개를 통째로 받는 단순한 치환이므로, 더 복잡한 파라미터가 필요하면 본문에서 자연어로 풀어 두는 편이 좋습니다.

```
사용자 입력:        /review 123
파일에 적힌 본문:    $ARGUMENTS 에 해당하는 변경분을 ...
모델에게 전달:      123 에 해당하는 변경분을 ...
```

다음으로 skill을 봅니다. skill은 슬래시 커맨드보다 한 단계 더 묶인 단위입니다. 명령 파일이 한 장이라면 skill은 디렉토리 한 개를 단위로 잡고, 그 안에 본문과 부속 자료를 함께 둡니다. 위치는 `.claude/skills/NAME/SKILL.md`가 기본형입니다. `NAME`이 skill 이름이고, 같은 디렉토리에 예시 데이터·체크리스트·하위 스크립트를 함께 넣어 둘 수 있습니다. 슬래시 커맨드처럼 무조건 부르는 것은 아니고, Claude Code가 세션 컨텍스트에서 적합한 시점을 판단해 자동으로 끌어 들이도록 설계되어 있습니다.

skill의 자동 로딩 조건은 보통 두 단계로 정리됩니다. 첫 단계는 발견입니다. Claude Code는 세션 시작 시 `.claude/skills/` 아래의 `SKILL.md` 파일들을 훑어 이름과 요약을 캐싱합니다. 두 번째 단계는 호출입니다. 사용자가 슬래시 커맨드로 직접 부르거나, 모델이 작업 맥락에서 skill 이름을 골라 호출할 때 본문이 컨텍스트에 합류합니다. 즉, 슬래시 커맨드는 사용자가 명령으로 직접 부르는 단위이고, skill은 도구 호출과 비슷하게 모델이 골라서 부르는 단위라고 볼 수 있습니다.

두 장치를 팀에서 공유할 때는 위치 선택이 중요합니다. 프로젝트 공통으로 쓸 명령·skill은 `.claude/` 안에 두고 git에 함께 커밋합니다. 이렇게 두면 신규 합류자가 레포를 받자마자 동일한 명령·skill을 자동으로 갖게 됩니다. 개인용 명령·skill은 `~/.claude/` 아래에 두고 공유하지 않습니다. 한편 명령에서 도구 화이트리스트를 좁게 적어 둔다면, 누가 실행하더라도 동일한 안전 범위 안에서만 동작하므로 공유가 더 자연스러워집니다.

settings.json과의 관계도 한 번 짚어 둡니다. 2.2.1에서 본 settings.json은 도구 권한·hook·MCP 등록 같은 전체 정책을 잡는 자리입니다. 슬래시 커맨드와 skill은 그 정책 위에서 동작하는 재사용 단위이므로, 두 곳을 분리해 둡니다. 공유 정책은 `.claude/settings.json`에, 공유 명령·skill은 `.claude/commands/`와 `.claude/skills/`에 두는 식입니다. 한 PR에서 둘을 동시에 고치는 일이 잦다면, CLAUDE.md에서 두 디렉토리를 같이 갱신하라는 규칙을 한 줄 적어 두면 누락이 줄어듭니다.

Codex CLI는 슬래시 커맨드와 1:1로 대응되는 장치를 두지 않습니다. 대신 셸 도구들이 그 자리를 채웁니다. 첫째, 셸의 alias나 함수로 자주 쓰는 `codex exec` 호출을 묶는 방법이 가장 가볍습니다. 둘째, 한 줄 wrapper 스크립트를 레포에 두고 인자를 받아 `codex exec`에 그대로 흘려보내면, Claude Code 슬래시 커맨드와 비슷한 사용감을 얻을 수 있습니다. 셋째, profile을 적절히 쪼개 두면 명령 한 줄에 권한 모드까지 통째로 묶입니다. 다음 표가 대응 관계를 정리합니다.

| Claude Code | Codex CLI에서의 대응 |
| --- | --- |
| 슬래시 커맨드 `/name` | 셸 alias·함수, 또는 wrapper 스크립트 |
| `$ARGUMENTS` 치환 | 셸 변수 `$@`, `$1` 같은 위치 인자 |
| `allowed-tools` 화이트리스트 | profile의 sandbox·approval 조합 |
| `.claude/skills/NAME/` | profile 묶음 + AGENTS.md 섹션 |

정리하면, Claude Code의 슬래시 커맨드는 `.claude/commands/NAME.md`에 frontmatter와 본문을 적어 두고 `$ARGUMENTS`로 인자를 받는 한 장짜리 재사용 단위입니다. skill은 `.claude/skills/NAME/SKILL.md`를 중심으로 디렉토리 단위 묶음을 만들고, 모델이 맥락에 맞춰 끌어 쓰도록 둡니다. Codex CLI에서는 셸 wrapper와 profile로 비슷한 묶음을 만들 수 있습니다.

다음 단원인 5.2.2에서는 자주 쓰는 MCP 서버 묶음을 두 도구에 동시에 설치해 활용하는 방법을 다룹니다.

이 단원을 마치면 슬래시 커맨드 파일과 skill 디렉토리를 직접 만들고, frontmatter 필드와 `$ARGUMENTS` 치환 동작을 설명하며, Codex CLI에서 동등한 효과를 내는 셸 wrapper와 profile 구성을 짤 수 있습니다.
