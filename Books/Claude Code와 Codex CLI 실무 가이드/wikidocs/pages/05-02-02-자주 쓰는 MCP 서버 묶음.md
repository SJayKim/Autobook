# 5.2.2 자주 쓰는 MCP 서버 묶음

MCP의 가치는 어떤 서버를 골라 두느냐로 갈립니다. 5.1.1에서 본 것처럼 MCP는 외부 도구를 표준 인터페이스로 노출하는 프로토콜이고, 같은 인터페이스라면 어떤 서버든 두 도구에 동일하게 붙일 수 있습니다. 그래도 매번 새 서버를 찾아 헤매는 것보다, 실무에서 자주 쓰이는 네 가지를 한 번에 묶어 두고 필요할 때 켜고 끄는 편이 효율적입니다. 이번 단원에서는 filesystem, github, playwright, sqlite 네 서버를 다루며, 각 서버의 역할과 설치 한 줄, 인증·범위 제한 요점을 정리합니다.

먼저 filesystem 서버입니다. 이름 그대로 디스크의 일부를 모델에게 노출하는 서버이며, 핵심은 노출 경로를 명시적으로 좁히는 데 있습니다. 일반적으로 Claude Code와 Codex CLI는 작업 디렉토리 안의 파일을 자체 도구로도 다룰 수 있지만, MCP 서버를 거치면 권한 매처와 sandbox 검사를 한 번 더 통과하므로 격리가 강해집니다. 외부 자료나 임시 폴더 같은 특정 경로만 보여 주고 싶을 때도 유용합니다. Claude Code에는 다음 명령으로 등록합니다.

```bash
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /work
```

이 명령은 `filesystem`이라는 이름으로 stdio 서버를 등록하고, `--` 뒤의 부분을 실행 명령으로 잡습니다. 마지막 인자 `/work`가 모델에게 보일 작업 경로입니다. 경로를 여러 개 주면 그 경로들만 묶어서 노출됩니다. Codex CLI는 5.1.3에서 본 형식대로 `~/.codex/config.toml`에 적습니다.

```toml
[mcp_servers.filesystem]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-filesystem", "/work"]
```

두 도구에 동일 서버를 두는 식으로 짝을 맞춥니다. 범위 제한은 등록 단계에서 한 번, sandbox·permissions 단계에서 한 번, 총 두 차례 점검하도록 둡니다.

다음은 github 서버입니다. 이슈와 풀 리퀘스트를 읽고 쓰는 작업을 대신해 주는 서버이며, 토큰 인증이 필수입니다. 토큰은 GitHub 개인 설정에서 만든 PAT(Personal Access Token)를 환경 변수로 넘기는 방식이 가장 보편적입니다. 토큰의 범위는 작업에 필요한 최소로 좁혀 두면, 서버가 임의로 다른 레포를 건드릴 수 없습니다. Claude Code 등록 예시는 다음과 같습니다.

```bash
claude mcp add github \
  -e GITHUB_PERSONAL_ACCESS_TOKEN=$GITHUB_TOKEN \
  -- npx -y @modelcontextprotocol/server-github
```

`-e` 플래그가 환경 변수 주입을 담당하며, 명령줄에 `$GITHUB_TOKEN`을 그대로 두면 셸이 값을 채워 줍니다. 토큰 값을 settings.json 같은 git 추적 파일에 직접 적지 않도록 합니다. Codex CLI 쪽은 다음과 같이 적습니다.

```toml
[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_PERSONAL_ACCESS_TOKEN = "$GITHUB_TOKEN" }
```

토큰은 셸이 codex를 실행할 때 환경 변수로 미리 잡아 두고, config.toml에는 변수 이름만 적습니다. 이렇게 분리해 두면 토큰을 갱신해도 설정 파일은 그대로 둘 수 있습니다.

세 번째는 playwright 서버입니다. 브라우저를 띄우고 페이지를 조작해 UI 회귀 검증을 자동화할 때 유용한 서버입니다. 페이지 열기, 요소 클릭, 입력, 스크린샷 같은 동작을 도구 호출로 매핑해 줍니다. Claude Code에는 다음과 같이 등록합니다.

```bash
claude mcp add playwright -- npx -y @playwright/mcp
```

처음 실행할 때 Playwright가 브라우저 바이너리를 내려받습니다. 이 과정이 길 수 있으므로, CI 환경에서는 사전에 설치 명령을 한 번 돌려 두는 편이 좋습니다. Codex CLI 쪽도 동일한 패키지를 가리키되, sandbox 모드를 함께 고려합니다. 브라우저를 띄우는 작업은 네트워크와 임시 파일을 함께 요구하므로, `read-only` profile에서는 동작하지 않는 호출이 생깁니다. UI 검증은 별도의 `playwright` profile을 두고 `workspace-write`로 두는 구성을 권장합니다.

```toml
[mcp_servers.playwright]
command = "npx"
args = ["-y", "@playwright/mcp"]
```

네 번째는 sqlite 서버입니다. 로컬 SQLite 파일을 모델이 직접 질의해 분석 작업을 자동화하도록 도와줍니다. 데이터 파일 경로를 인자로 주면, 그 파일에 대한 읽기·쓰기 도구가 노출됩니다. Claude Code 등록은 다음과 같습니다.

```bash
claude mcp add sqlite -- uvx mcp-server-sqlite --db-path /work/analytics.db
```

`uvx`는 Python 패키지를 한 번에 띄우는 도구이고, 환경에 따라 `pipx run`이나 `python -m`으로 바꿔 둘 수도 있습니다. `--db-path`가 노출할 데이터베이스 파일 경로입니다. 운영 데이터베이스를 그대로 노출하기보다, 사본을 별도 경로에 두고 그 경로만 가리키도록 두는 편이 안전합니다. Codex CLI 쪽 등록은 다음과 같습니다.

```toml
[mcp_servers.sqlite]
command = "uvx"
args = ["mcp-server-sqlite", "--db-path", "/work/analytics.db"]
```

네 서버를 한 표로 정리하면 다음과 같습니다. 인증·범위 제한 칸은 등록 시 가장 자주 잊는 항목을 짚어 둡니다.

| 서버 | 핵심 역할 | 인증·범위 제한 요점 |
| --- | --- | --- |
| filesystem | 디스크 일부를 도구로 노출 | 노출 경로를 인자로 좁히기 |
| github | 이슈·PR 자동 처리 | PAT 환경 변수, 최소 범위 |
| playwright | 브라우저 자동화 | 브라우저 바이너리 사전 설치, workspace-write |
| sqlite | 로컬 DB 질의 자동화 | 사본 DB만 노출, --db-path 명시 |

권한과 sandbox 측면에서 한 가지 더 짚습니다. 5.1.2에서 본 것처럼 Claude Code의 MCP 도구 이름은 `mcp__servername__toolname` 형식이므로, 권한 매처에 그 접두어를 활용해 도구 단위로도 화이트리스트를 좁힐 수 있습니다. Codex CLI는 5.1.3에서 본 대로 sandbox 모드와 approval policy가 추가로 개입하므로, MCP 호출이라도 sandbox가 거부하면 동작하지 않습니다. 같은 서버를 등록했더라도 도구마다 실제 권한이 다른 셈입니다.

마지막으로 자주 쓰는 묶음을 한 PR에 모아 두는 패턴을 권합니다. `.mcp.json`이나 `config.toml`에 네 서버를 함께 적고, 어떤 작업에서 어떤 서버를 켜는지 README나 CLAUDE.md에 한 줄로 기록해 두면, 신규 합류자가 같은 작업을 다시 알아볼 필요가 없습니다.

정리하면, filesystem은 경로 격리, github는 토큰 인증, playwright는 브라우저 자동화, sqlite는 로컬 분석이라는 네 역할을 각각 한 줄 명령으로 두 도구에 등록할 수 있습니다. 등록 직후에는 노출 범위와 sandbox·approval 정책을 한 번 더 점검해, 도구가 닿을 수 있는 영역을 최소로 좁혀 둡니다.

다음 단원인 6.1.1에서는 자주 쓰는 도구 묶음을 더 작게 쪼개어, 메인 에이전트가 일을 위임하는 subagent의 정의와 호출 방식을 다룹니다.

이 단원을 마치면 filesystem, github, playwright, sqlite 네 서버를 두 도구에 각각 등록하고, 인증·범위 제한·sandbox 조합을 한 줄 레시피로 적어 둘 수 있습니다.
