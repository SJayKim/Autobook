# 1.1.2 하네스 구조 한눈에 보기

도구가 어떤 안전장치를 어디에 두는지 알면, 새 정책을 어느 파일에 넣어야 하는지 빠르게 판단할 수 있습니다. 이 단원에서는 사용자 입력이 모델을 거쳐 실제 도구 호출로 이어지는 경로, 즉 **하네스**를 두 도구 각각에서 그려 봅니다. 1.1.1에서 본 권한 모델의 차이가 왜 그렇게 설계되었는지가 자연스럽게 드러납니다.

먼저 하네스라는 말부터 풉니다. 본래 동물의 몸을 감싸 방향과 힘을 통제하는 장비를 가리키는 말입니다. 코딩 에이전트 맥락에서는 **모델의 자유로운 출력**을 사용자 의도와 안전 정책 안으로 묶어 두는 코드와 설정의 묶음을 말합니다. 사용자의 메시지가 모델로 향하기 직전, 모델이 도구를 호출하기 직전, 도구 결과가 모델로 돌아가기 직전, 이런 길목마다 하네스가 끼어듭니다.

Claude Code의 하네스는 다음 흐름을 따릅니다.

```
[사용자 프롬프트]
   ↓ UserPromptSubmit hook (정책 문구 주입 가능)
[settings.json: permissions 검사 + CLAUDE.md 자동 로딩]
   ↓
[모델: 도구 호출 결정]
   ↓ PreToolUse hook (도구·인자 차단 가능)
[도구 실행 (Bash, Edit, MCP 등)]
   ↓ PostToolUse hook (포맷·린트 자동화)
[결과를 모델 컨텍스트에 누적]
   ↓ (대화 계속, 또는 Stop hook으로 종료 처리)
```

길목마다 이름이 붙은 **hook**이 있고, 각 hook은 `settings.json`의 `hooks` 필드에 등록한 명령을 실행합니다. 종료 코드가 0이면 통과, 2이면 차단, 그 외이면 오류로 처리합니다. permissions는 별개로 동작합니다. 도구 호출이 이루어지기 전, 매처 규칙에 따라 자동 허용·자동 거부·사용자 문의 중 하나가 선택됩니다. 두 장치는 함께 일하며, 정책은 한 도구 호출에 대해 permissions와 PreToolUse가 모두 통과해야 실행에 닿습니다.

피드백 경로는 **대화 컨텍스트 누적** 방식입니다. 도구 결과가 모델의 같은 대화 안에 그대로 쌓이며, 모델은 누적된 컨텍스트를 보고 다음 행동을 정합니다. 한 세션이 길어질수록 컨텍스트도 커지고, 자동 압축(`PreCompact`)이 끼어들어 길이를 줄입니다.

Codex CLI의 하네스는 다른 모양을 가집니다.

```
[사용자 프롬프트 또는 codex exec 인자]
   ↓
[AGENTS.md 자동 로딩 + config.toml: profile 선택]
   ↓
[모델: 도구 호출 결정]
   ↓ sandbox 검사 (read-only / workspace-write / danger-full-access)
   ↓ approval policy 분기 (untrusted / on-failure / on-request / never)
[도구 실행 (파일 쓰기, 셸 명령, MCP 등)]
   ↓
[결과 보고 → 작업 단위 종료 또는 대화 지속]
```

여기서는 길목마다 hook이 있는 대신 **두 단계의 게이트**가 놓여 있습니다. 첫 게이트는 sandbox로, 도구 실행이 시도되는 시점에 파일과 네트워크 동작을 자체적으로 제한합니다. 둘째 게이트는 approval policy로, sandbox 한계 밖 동작이 필요할 때 사용자에게 묻거나, 한 번 실패하면 묻거나, 절대 묻지 않는 식으로 분기합니다. `untrusted`는 모든 권한 밖 동작을 사용자 확인 뒤에 진행하고, `on-failure`는 실패 후 한 번 더 시도할 때만 묻고, `on-request`는 모델이 명시적으로 요청할 때만 묻고, `never`는 묻지 않고 거부합니다.

피드백 경로는 **작업 결과 보고** 방식에 가깝습니다. `codex exec`로 띄운 작업은 시작과 끝이 명확하고, 결과는 표준 출력과 변경된 파일로 회수됩니다. 대화형 모드도 있지만, 기본 사고 모델은 한 단위 작업을 마치고 돌아오는 쪽입니다.

두 하네스를 통제 지점 위치로 비교하면 다음과 같습니다.

| 통제 지점 | Claude Code | Codex CLI |
| --- | --- | --- |
| 프롬프트 진입 | UserPromptSubmit hook | AGENTS.md 자동 주입 |
| 정책 파일 | settings.json | config.toml |
| 도구 호출 직전 | PreToolUse hook + permissions | sandbox + approval policy |
| 도구 결과 처리 | PostToolUse hook | (도구가 직접 반환) |
| 세션 종료 | Stop·SessionEnd hook | 작업 단위 종료 |
| 컨텍스트 압축 | PreCompact hook + 자동 압축 | 세션 분리·재시작 |

같은 목적의 장치가 서로 다른 이름과 다른 위치에 놓여 있습니다. 그래서 안전장치를 더하고 싶을 때 어디를 건드릴지가 도구마다 다릅니다. Claude Code에서는 **hook 스크립트**를 추가하고 권한 매처를 다듬는 쪽으로 갑니다. Codex CLI에서는 **sandbox 모드**를 한 단계 낮추고 approval policy를 보수적으로 잡는 쪽으로 갑니다. 어느 쪽이든 통제 지점이 한 곳에 모여 있지 않으므로, 같은 효과를 내려면 두세 군데를 함께 손보는 일이 생깁니다.

정리하면, Claude Code 하네스는 길목마다 hook이 끼어드는 다단 필터 구조이고, Codex CLI 하네스는 sandbox와 approval policy가 두 게이트로 작동하는 격리 구조입니다. 통제 지점이 놓인 위치를 이 그림 한 장으로 기억하면, 이후 단원에서 등장하는 모든 설정이 어느 지점에 작용하는지 가늠하기 쉬워집니다.

다음 단원인 1.2.1에서는 Claude Code를 실제로 설치하고 첫 세션을 안전한 모드로 띄우는 절차를 다룹니다.
