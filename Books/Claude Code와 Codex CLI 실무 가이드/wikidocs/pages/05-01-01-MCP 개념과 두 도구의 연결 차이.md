# 5.1.1 MCP 개념과 두 도구의 연결 차이

에이전트가 파일을 읽고 명령을 실행하는 것만으로 부족할 때가 있습니다. GitHub 이슈를 조회하고, 데이터베이스에 질의하고, 브라우저로 화면을 띄워 보는 일까지 모델에게 시키려면, 그 외부 도구들을 어떻게 노출해 줄지부터 결정해야 합니다. 4.1.5까지 다룬 hook은 도구 호출의 앞뒤를 통제하는 장치였지, 도구 자체를 새로 추가하는 장치는 아니었습니다. 새 도구를 추가하는 자리에는 다른 표준이 자리잡고 있습니다. **Model Context Protocol**, 줄여서 **MCP**입니다.

MCP는 외부 도구와 자원을 LLM 기반 에이전트에 공통의 인터페이스로 노출하기 위한 약속입니다. 도구를 만드는 사람은 MCP 서버를 구현해 두면 되고, 에이전트를 만드는 사람은 MCP 클라이언트만 지원하면 됩니다. 둘 사이는 한 가지 메시지 규격으로 대화하므로, 같은 MCP 서버를 Claude Code에도 붙이고 Codex CLI에도 붙일 수 있습니다. 도구마다 다른 SDK를 익히지 않아도 된다는 점이 가장 큰 효용입니다.

조금 더 풀어 말하면, MCP 서버는 두 가지를 제공합니다. 첫째, **도구(tool)** 목록입니다. 각 도구는 이름, 설명, 입력 스키마를 가지며 모델이 호출하면 결과를 돌려줍니다. 둘째, **자원(resource)** 목록입니다. 파일·페이지·테이블처럼 모델이 읽어 들일 수 있는 데이터의 핸들입니다. 클라이언트는 서버에 연결하자마자 이 두 목록을 받아 모델에게 노출합니다. 모델 입장에서는 내장 도구든 MCP 도구든 호출 방식이 같습니다.

서버와 클라이언트가 메시지를 주고받을 통로는 **전송 방식(transport)**으로 부릅니다. 가장 흔히 쓰이는 것은 **stdio**입니다. 클라이언트가 서버 프로세스를 자식으로 띄우고, 표준 입력과 표준 출력으로 JSON-RPC 메시지를 주고받습니다. 별도의 네트워크 포트가 필요 없고, 클라이언트가 죽으면 서버도 자연스럽게 정리되므로 로컬 도구에 가장 적합합니다.

원격 서버를 붙이려면 네트워크 전송이 필요합니다. 두 가지가 표준에 들어 있습니다. 하나는 **SSE(Server-Sent Events)** 기반으로, 서버가 클라이언트에 이벤트 스트림을 흘려보내는 단방향 채널을 활용합니다. 다른 하나는 **streamable HTTP**로, 일반 HTTP 요청·응답 위에서 스트리밍을 다루는 방식입니다. 두 방식 모두 서버가 별도의 호스트로 떠 있을 때 쓰입니다. 로컬에서 시작한다면 stdio부터 익히는 편이 단순합니다.

여기까지가 MCP 자체의 그림이고, 이제 두 도구의 연결 방식을 비교합니다. 두 도구 모두 MCP 클라이언트를 내장하지만, 등록 위치와 문법이 다릅니다.

Claude Code는 두 곳에 MCP 서버를 등록할 수 있습니다. 첫째는 settings.json의 `mcpServers` 필드, 둘째는 프로젝트 루트의 `.mcp.json` 파일입니다. 두 파일의 mcpServers 구조는 동일합니다. JSON 객체 안에 서버 이름을 키로 두고, 그 값으로 명령과 인자, 환경 변수 등을 적습니다. 이 등록을 도와주는 명령이 `claude mcp add`이며, 자세한 사용법은 5.1.2에서 다룹니다.

Codex CLI는 `~/.codex/config.toml`의 `[mcp_servers.NAME]` 섹션을 씁니다. JSON이 아니라 TOML이라는 점, 그리고 한 서버마다 별도의 섹션 블록을 두는 점이 다릅니다. 필드 이름은 비슷합니다. 명령, 인자, 환경 변수를 같은 의미로 둡니다. 자세한 등록 절차는 5.1.3에서 다룹니다.

두 도구의 등록 형식을 한 표로 정리하면 다음과 같습니다.

| 항목 | Claude Code | Codex CLI |
| --- | --- | --- |
| 등록 위치 | settings.json `mcpServers` 또는 .mcp.json | ~/.codex/config.toml `[mcp_servers]` |
| 파일 포맷 | JSON | TOML |
| 등록 명령 | `claude mcp add NAME COMMAND ARGS...` | 직접 config.toml 편집 |
| 한 서버의 키 | 객체 키(name)로 묶음 | 섹션 헤더 `[mcp_servers.name]` |
| 명령 필드 | `command` | `command` |
| 인자 필드 | `args` (배열) | `args` (배열) |
| 환경 변수 | `env` (객체) | `env` (테이블) |
| 권한 매처 표기 | `mcp__servername__toolname` | sandbox·approval 정책에 통합 |
| 프로파일별 분리 | settings 계층(전역/프로젝트/로컬) | `[profiles.X]`에서 mcp_servers 오버라이드 |

표의 마지막 두 항목이 운영에서 가장 자주 부딪히는 부분입니다. Claude Code는 MCP 도구를 권한 시스템 안에서 별도의 이름으로 다룹니다. 표기 규칙은 `mcp__SERVERNAME__TOOLNAME`이고, 구분자가 **밑줄 두 개**라는 점에 주의합니다. permissions의 allow나 ask에 이 표기를 넣어, MCP 도구별로 허용·문의를 둘 수 있습니다.

```json
{
  "permissions": {
    "allow": ["mcp__github__list_issues"],
    "ask":   ["mcp__github__create_issue"]
  }
}
```

이 설정은 github MCP 서버의 list_issues는 자동 허용하고, create_issue는 사용자에게 물어보도록 둡니다. 권한 매처 표기 안의 `github`는 등록 시 정한 서버 이름과 정확히 같아야 합니다.

Codex CLI는 MCP 도구를 별도의 권한 이름으로 다루지 않습니다. 대신 sandbox 모드와 approval policy가 MCP 호출에도 동일하게 적용됩니다. 예를 들어 sandbox가 read-only이면, MCP 도구가 파일을 쓰려고 시도하는 호출은 모두 차단됩니다. approval policy가 on-request이면, 위험 동작이 감지될 때 사용자에게 묻습니다. 이름 단위 화이트리스트가 필요하다면 profile을 분리해, 신뢰하는 서버 묶음을 가진 profile만 골라서 띄우는 방식으로 우회합니다.

같은 MCP 서버를 두 도구에서 공유할 때 주의할 점이 두 가지 더 있습니다. 첫째, 환경 변수와 비밀 값입니다. Claude Code 쪽 `env`와 Codex CLI 쪽 `env`는 별도로 주입됩니다. 한쪽에 토큰을 넣었다고 다른 쪽이 자동으로 가져오지 않으므로, 두 곳에 같은 값을 동기화하거나 쉘 환경 변수에서 읽도록 합니다. 둘째, 동시 실행입니다. stdio 서버는 클라이언트마다 새 프로세스를 띄우므로, 두 도구가 동시에 같은 서버를 띄우면 자식 프로세스가 두 개가 됩니다. 데이터베이스 잠금처럼 동시 접근에 민감한 서버는 한 번에 한쪽만 띄우거나, 별도의 작업 디렉터리를 두어 충돌을 막습니다.

전체 그림을 도식으로 정리하면 다음과 같습니다.

```
[모델] -- 도구 호출 --> [클라이언트(Claude Code / Codex CLI)]
                              |
                              v
                         [전송: stdio / SSE / HTTP]
                              |
                              v
                         [MCP 서버 프로세스]
                              |
                              v
                         [외부 시스템: GitHub, DB, 브라우저, FS ...]
```

두 도구는 가운데 클라이언트 자리에서 사용하는 등록 파일과 권한 모델이 다를 뿐, 그 너머의 전송과 서버, 외부 시스템 영역은 동일합니다. 그래서 한 번 잘 만든 MCP 서버를 양쪽에 공유해 쓰는 일이 가능합니다.

정리하면, MCP는 외부 도구를 모델에 표준 인터페이스로 노출하기 위한 약속이고, 전송 방식으로 stdio·SSE·streamable HTTP를 둡니다. Claude Code는 settings.json·.mcp.json의 mcpServers와 `mcp__server__tool` 권한 매처로 다루고, Codex CLI는 config.toml의 `[mcp_servers]` 섹션과 sandbox·approval 정책으로 다룹니다. 등록 형식과 권한 표기가 다르다는 점만 잘 이해하면, 같은 MCP 서버를 두 도구에서 어렵지 않게 공유할 수 있습니다.

다음 단원인 5.1.2에서는 Claude Code에 MCP 서버를 등록하는 `claude mcp add` 명령과 .mcp.json 편집 방법을 자세히 다룹니다.

이 단원을 마치면 MCP의 역할과 세 전송 방식을 설명하고, Claude Code와 Codex CLI의 등록 위치·권한 매처 차이를 표로 비교할 수 있습니다.
