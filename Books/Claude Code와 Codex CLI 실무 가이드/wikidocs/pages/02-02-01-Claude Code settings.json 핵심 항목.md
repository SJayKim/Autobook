# 2.2.1 Claude Code settings.json 핵심 항목

CLAUDE.md(2.1.1)와 AGENTS.md(2.1.2)는 모델에게 무엇을 어떻게 할지를 자연어로 알려 주는 문서였습니다. 그러나 모델이 실제로 어떤 도구를 쓸 수 있는지, 어떤 명령을 자동으로 통과시키고 어떤 명령에서 멈춰 사람에게 묻는지, 어떤 외부 서버에 연결되는지 같은 동작은 자연어가 아닌 구조화된 설정 파일이 결정합니다. Claude Code에서는 이 역할을 settings.json이 맡으며, 같은 파일이 세 위치에 놓이고 위에서 아래로 합쳐진다는 점이 처음에는 헷갈리기 쉽습니다. 이 단원에서는 세 위치의 우선순위, 자주 쓰는 필드의 역할, 비공개로 두어야 하는 항목과 흔한 실수 패턴을 정리합니다.

settings.json이 놓이는 세 위치부터 살펴보겠습니다. 첫째는 사용자 홈의 **~/.claude/settings.json**으로, 어느 프로젝트를 열어도 항상 적용되는 사용자 기본값을 담습니다. 둘째는 프로젝트 루트의 **.claude/settings.json**으로, 팀 전체가 공유해야 할 규칙을 담으며 git에 함께 커밋합니다. 셋째는 같은 프로젝트의 **.claude/settings.local.json**으로, 같은 프로젝트에서 작업하더라도 개인마다 다르게 두고 싶은 값을 담으며 .gitignore에 두어 커밋하지 않습니다.

세 파일이 모두 존재하면 사용자 기본값 위에 프로젝트 공유값이 얹히고, 다시 그 위에 프로젝트 로컬값이 얹힙니다. 같은 객체 필드는 깊게 병합되며, 같은 키가 충돌하면 더 우선순위가 높은 쪽이 이깁니다. 다만 배열 필드는 단순 덮어쓰기가 아닌 합집합으로 합쳐지는 경우가 많으니, 이 단원의 뒤에서 권한 배열을 다룰 때 다시 짚겠습니다.

```
우선순위(낮음 → 높음)
~/.claude/settings.json        (사용자 기본)
.claude/settings.json          (프로젝트 공유, 커밋)
.claude/settings.local.json    (개인 비공개, gitignore)
```

이제 자주 쓰는 필드를 하나씩 풀어 보겠습니다. **permissions**는 도구 호출의 허용·차단·확인 정책을 담는 객체로, allow·deny·ask·additionalDirectories 네 배열을 가집니다. **env**는 모델이 실행하는 서브프로세스에 주입할 환경 변수를 담는 객체로, API 키처럼 민감한 값은 여기보다 OS 환경 변수에 두는 편이 안전합니다. **hooks**는 도구 호출 전후나 세션 시작 같은 이벤트에 자동으로 끼어드는 스크립트 목록입니다. **mcpServers**는 외부 MCP 서버를 등록하는 객체로, 5장에서 자세히 다룹니다.

다음 예시는 프로젝트 공유 설정으로 자주 쓰이는 형태입니다.

```json
{
  "model": "claude-sonnet-4-5",
  "permissions": {
    "allow": [
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(npm test:*)",
      "Read(./src/**)"
    ],
    "deny": [
      "Bash(git push --force:*)",
      "Read(./.env)",
      "Write(./.env*)"
    ],
    "ask": [
      "Bash(rm:*)",
      "Bash(npm install:*)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  },
  "statusLine": "{{model}} | {{branch}} | {{cwd}}"
}
```

필드별로 풀어 보면 다음과 같습니다. **model**은 세션의 기본 모델 이름으로, 비용·성능을 프로젝트 단위로 통일하고 싶을 때 둡니다. **statusLine**은 화면 하단에 표시되는 한 줄 상태바의 템플릿으로, 어떤 모델로 어느 브랜치에서 어디서 작업 중인지를 항상 보이게 합니다. 그 외에 **autoUpdates**는 CLI 자동 업데이트 여부, **theme**는 색 테마를 결정하며, 이 두 항목은 보통 사용자 기본값(~/.claude/settings.json)에 둡니다.

settings.local.json을 .gitignore에 두는 이유는 두 가지입니다. 첫째는 개인 토큰·내부 경로·실험적 hook처럼 다른 팀원에게 노출되면 곤란한 값을 분리하기 위함입니다. 둘째는 같은 프로젝트라도 사람마다 다른 권한 정책을 두고 싶을 때 충돌을 막기 위함입니다. 예를 들어 보수적인 권한이 필요한 작업자는 ask 배열을 넓히고, 빠른 실험이 필요한 작업자는 allow 배열을 넓힐 수 있습니다. 어느 쪽이든 공유 설정에는 손대지 않으므로 PR이 불필요하게 흔들리지 않습니다.

흔히 잘못 쓰는 패턴 하나를 짚어 두겠습니다. allow 배열에 `Bash(*)` 같은 광범위 와일드카드를 두면 거의 모든 셸 명령이 사전 동의 없이 통과합니다. 처음에는 편리해 보이지만, 모델이 의도치 않게 `rm -rf`나 `git push --force` 같은 파괴적 명령을 부르더라도 멈추지 않게 됩니다. 권장 패턴은 자주 쓰는 명령을 **명령 단위로 좁게 allow**에 두고, 위험할 수 있는 명령은 **ask**로 두어 한 번 확인을 받는 것입니다. deny는 어떤 경우에도 통과되어서는 안 되는 명령에 한해 사용하며, deny에 적힌 패턴은 같은 패턴이 allow에 들어 있더라도 항상 이깁니다.

또 하나 자주 보이는 실수는 env에 API 키를 직접 적어 넣고 .claude/settings.json을 커밋해 버리는 경우입니다. env는 키 자체를 적는 자리가 아니라 OS 환경 변수에서 값을 전달받는 통로로 쓰는 것이 안전합니다. 키를 꼭 파일에 두어야 한다면 settings.local.json에 두고 git에는 절대 올라가지 않게 해야 합니다.

정리하면, settings.json은 사용자 기본·프로젝트 공유·개인 비공개 세 위치에서 위로 갈수록 우선되는 방식으로 병합됩니다. permissions·env·hooks·mcpServers가 핵심 필드이며, model·statusLine·autoUpdates·theme는 동작 옵션입니다. 와일드카드 남용과 키 노출을 피하고, 공유와 비공개를 명확히 분리하는 것이 사고 없는 운영의 출발점입니다.

다음 단원인 2.2.2에서는 Codex CLI 쪽 동작을 결정하는 config.toml의 구조와 필드를 살펴봅니다.

이 단원을 마치면 settings.json의 세 위치를 구분해 두고 필요한 필드를 정확한 자리에 배치할 수 있게 됩니다.
