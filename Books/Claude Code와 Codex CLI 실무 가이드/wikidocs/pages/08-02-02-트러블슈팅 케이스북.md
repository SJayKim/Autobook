# 8.2.2 트러블슈팅 케이스북

설정이 모두 통과해도 현장에서는 비슷한 사고가 반복됩니다. 모델이 갑자기 '권한이 없다'며 멈춰 서고, MCP 서버가 응답을 끊고, 잘 돌던 hook이 같은 명령을 끝없이 다시 부릅니다. 이 단원은 그 흔한 문제들을 사례 단위로 정리합니다. 각 케이스는 **증상 → 진단 → 조치**의 짧은 묶음으로 두어, 같은 일이 다시 벌어졌을 때 곧장 들춰볼 수 있게 합니다.

먼저 가장 자주 마주치는 것이 권한 거부입니다.

**케이스 1: 'Permission denied' 메시지로 멈춤**

증상은 명료합니다. Claude Code가 어떤 명령을 호출하려다가 권한 거부 사유와 함께 작업을 중단합니다. 사용자가 분명히 허용했다고 생각한 명령에서 발생할 때가 가장 혼란스럽습니다.

진단은 두 위치를 같이 봅니다. 첫째, 프로젝트 `.claude/settings.json`의 `permissions.deny`와 `ask`에 해당 패턴이 들어가 있는지를 확인합니다. 둘째, `.claude/settings.local.json`을 엽니다. 8.1.2에서 본 것처럼 로컬 파일이 표준을 덮어쓰므로, 누군가 과거에 일시적으로 거부를 추가하고 잊어버린 경우가 흔합니다. 셋째, 7.2.1에서 다룬 `Bash(git push*)` 같은 ask 패턴이 자동화 흐름에서는 차단처럼 동작할 수 있습니다.

조치는 순서대로 진행합니다. 정말 필요한 권한이면 표준 설정에 정식으로 추가하고 PR로 기록을 남깁니다. 개인 환경에서만 잠시 쓸 권한이면 `settings.local.json`에 한정합니다. 와일드카드로 통째로 여는 식의 대처는 피합니다.

```
deny:  Bash(rm -rf /*)        ← 정상, 절대 못 푼다
ask:   Bash(git push*)        ← 자동화 시 차단처럼 보임 → allow로 옮기거나 명시 승인
local: Bash(brew install*)    ← 개인 머신에서만, git에 안 따라감
```

**케이스 2: Codex sandbox 거부**

Codex CLI 쪽에서는 권한 거부가 sandbox 사유로 나타납니다. 증상은 파일 쓰기 또는 네트워크 호출이 'sandbox 정책에 의해 차단되었다'는 메시지로 끝나는 형태입니다.

진단은 두 줄을 봅니다. `config.toml`의 `sandbox_mode`와 사용 중인 프로파일의 `sandbox_mode`입니다. 3.2.1에서 본 것처럼 `read-only`는 모든 쓰기를 막고, `workspace-write`는 작업 디렉토리 바깥을 막습니다. 정찰 프로파일로 정작 빌드를 돌리려 했을 가능성이 가장 큽니다.

조치는 모드 전환을 두 가지로 둡니다. 작업 자체가 정찰이면 그대로 두고 사용자가 직접 파일을 쓰게 합니다. 작업이 본 구현이면 한 호출만 `--sandbox workspace-write`로 띄우거나, 본 작업용 프로파일(`--profile build`)을 사용합니다. `danger-full-access`로 무작정 올리지 않습니다. 사유가 명확히 드러나면 그 폴더만 `additionalDirectories` 또는 작업 디렉토리 정의에 추가하는 편이 안전합니다.

**케이스 3: MCP 서버가 묵묵부답**

증상은 까다롭습니다. MCP 도구를 호출했더니 응답이 오지 않고, 한참 뒤 timeout 사유로 도구만 실패합니다. 다른 도구 호출은 정상이라 처음에는 모델 문제처럼 보입니다.

진단은 다음 순서로 좁힙니다. 첫째, `claude mcp list` 또는 `config.toml [mcp_servers]`에서 서버 이름이 맞는지 확인합니다. 둘째, MCP 서버가 stdio라면 그 명령을 셸에서 직접 실행해 봅니다. 명령 자체가 없거나 인자가 틀리면 여기서 즉시 드러납니다. 셋째, stderr 로그를 확인합니다. 두 도구 모두 MCP 서버의 stderr를 캐시 디렉토리나 별도 로그 파일에 남깁니다. 토큰 누락이나 의존 모듈 오류가 그 로그에 적혀 있습니다.

조치는 원인에 맞춥니다. 토큰이 빠졌으면 `env` 항목에 환경 변수 참조를 추가하고 셸을 다시 띄웁니다. 명령이 없으면 npm 캐시나 설치 경로를 확인하고 다시 설치합니다. timeout이 짧다면 단순 첫 실행이 느렸을 수 있으니 한 번 더 시도합니다. 모든 시도가 실패하면 그 서버를 일단 비활성화해 두고 작업을 이어 갑니다.

```
[MCP 침묵 진단 흐름]
list 확인 → 셸에서 직접 실행 → stderr 로그 확인 → env/명령 교정 → 재실행
```

**케이스 4: Hook 무한 루프**

증상이 가장 위험합니다. 한 번의 편집이 끝없는 도구 호출 사슬을 유발하고, 토큰이 분 단위로 새며, 컨텍스트가 빠르게 차 버립니다. 흔한 패턴은 PostToolUse hook이 전체 테스트를 돌리고, 그 결과를 모델이 다시 읽고, 모델이 또 편집을 일으키며 같은 hook을 다시 부르는 흐름입니다.

진단은 두 곳을 봅니다. 첫째, hook 스크립트의 stdout에 모델이 응답하지 않을 수 없는 강한 문구가 들어가지 않는지 확인합니다. 둘째, hook이 처리하는 입력이 변경된 파일에 한정되지 않고 매번 전체 프로젝트를 다시 도는지 확인합니다.

조치는 세 단계로 가져갑니다. 즉시 멈추기 위해 사용자가 ESC로 현재 도구 호출을 끊거나 세션을 종료합니다. 그다음 hook을 일시 비활성화한 채 작업을 정상화합니다. 마지막으로 hook을 다음과 같이 손봅니다.

```
- 처리 대상을 변경된 파일 인자로 한정
- stdout에는 결과 한 줄만, 자세한 사항은 stderr로
- 실패 시 exit code로 신호하고, 모델에 긴 로그를 던지지 않는다
- 같은 hook이 짧은 시간 안에 반복 호출되면 스로틀
```

PostToolUse가 다시 도구를 부르게 만드는 텍스트(예: '이제 X를 실행하세요')를 stdout에 적지 않는 것이 핵심입니다.

**케이스 5: 갑자기 모델이 컨텍스트를 잃은 것처럼 동작**

7.2.2에서 본 자동 압축이 일어난 직후 종종 발생합니다. 직전 결정이 사라져 같은 질문을 두 번 받는 듯한 인상을 줍니다.

진단은 단순합니다. PreCompact hook이 핸드오프 파일을 남기도록 되어 있는지, 그 파일이 실제로 만들어졌는지를 봅니다. 없다면 자동 압축이 그대로 일어났을 가능성이 큽니다.

조치는 두 가지를 같이 둡니다. PreCompact hook으로 핵심 결정을 외부 파일로 떨궈 두고, 자동 압축이 일어났음을 알게 된 시점에 사용자가 직접 그 파일을 읽어 모델에게 다시 전달합니다. 장기 대책으로는, 길어질 작업을 처음부터 세션으로 나누어 압축에 의존하지 않는 흐름을 만듭니다.

**케이스 6: 마지막 안전망 — 세션 종료와 작업 디렉토리 복원**

위 모든 진단이 시간이 걸리는 동안 손해가 누적되는 것 같으면, 우선 세션을 끊습니다. Claude Code와 Codex CLI 둘 다 세션을 닫아도 git 작업 트리는 그대로 남으므로, 다음 명령으로 변경 상태를 빠르게 확인합니다.

```
git status
git diff
git stash    # 의심스러운 변경을 잠시 치워 둔다
```

7.2.1에서 만든 작은 PR 단위와 revert 우선 원칙이 여기서 가치를 발휘합니다. 의심스러운 변경은 force push 없이 새 commit으로 되돌리거나, 마지막 안전한 commit으로 stash + checkout으로 복원합니다. 모든 케이스에서 공통의 마지막 줄은 같습니다. **사고를 의심하면 일단 세션을 끊고 git 상태부터 확인합니다.**

진단 순서를 통째로 압축한 흐름도 함께 두면 다음과 같습니다.

```
이상 증상 인지
   ↓
1. 메시지 키워드(permission, sandbox, timeout, hook, compact)로 분기
   ↓
2. 해당 설정 파일과 로그를 같은 화면에 놓는다
   ↓
3. 가장 좁은 조치(개인 settings.local, 단발 플래그)부터 시도
   ↓
4. 재현되면 표준 설정 또는 hook을 정식으로 수정
   ↓
5. 사고가 의심되면 세션 종료 → git status → 복원
```

정리하면, 권한 거부는 표준과 로컬 설정을 동시에 보고, sandbox 거부는 모드와 프로파일 일치를 확인하며, MCP 침묵은 셸 직접 실행과 stderr 로그로 좁히고, hook 무한 루프는 stdout 문구와 처리 범위를 손보며, 컨텍스트 손실은 PreCompact와 핸드오프로 보완하고, 어느 단계에서든 사고가 의심되면 세션을 끊고 git으로 복원하는 마지막 안전망을 활용합니다.

이로써 한 권을 마칩니다. 1장에서 두 도구의 모습을 같은 자리에 놓고 비교한 뒤, 2장의 지시문 파일과 3장의 권한·샌드박스로 통제의 골격을 세우고, 4장과 5장에서 hook과 MCP로 도구를 확장했으며, 6장의 작업 분해와 7장의 피드백 루프로 큰 작업을 안전하게 굴리는 흐름을 만들었고, 8장에서 비용·공유 설정·보안·트러블슈팅까지 운영의 마지막 한 단을 덮었습니다. 이 책의 모든 레시피는 합치면 한 줄로 요약됩니다. 도구가 무엇을 할 수 있게 하느냐보다, 무엇을 못 하게 하느냐를 먼저 정하고, 그 위에서 자동화의 폭을 한 단씩 넓혀 갑니다.

이 단원을 마치면 권한 거부, sandbox 거부, MCP 침묵, hook 무한 루프, 컨텍스트 손실의 다섯 가지 흔한 문제를 증상에서 출발해 진단하고 가장 좁은 조치부터 시도하며, 마지막 안전망으로 세션 종료와 git 복원을 자연스럽게 꺼낼 수 있습니다.
