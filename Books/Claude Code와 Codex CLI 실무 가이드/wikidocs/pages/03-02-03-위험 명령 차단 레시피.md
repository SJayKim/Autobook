# 3.2.3 위험 명령 차단 레시피

에이전트가 셸을 부릴 수 있게 되면, 가장 먼저 신경 써야 하는 것이 한 번의 실수로 회복이 어려운 명령입니다. `rm -rf /`로 디렉토리 트리를 통째 지우거나, `git push --force`로 동료의 작업을 덮어쓰거나, `curl http://... | sh`로 검증 없이 임의 스크립트를 실행하는 일은 사람에게도 위험합니다. 모델이 컨텍스트를 잘못 해석해 비슷한 명령을 만들어 내는 순간은 생각보다 자주 옵니다. 사고를 막으려면 이런 명령을 도구가 실행하기 전에 가로채는 장치가 필요합니다.

3.1.1에서 본 Claude Code permissions, 3.1.2의 Codex approval policy, 3.2.1의 sandbox는 모두 이 차단의 기반이 됩니다. 이 단원에서는 두 도구에서 같은 위험 명령을 동시에 차단하는 실용 레시피를 모아 봅니다.

먼저 Claude Code 쪽입니다. `settings.json`의 `permissions.deny` 배열에 위험 패턴을 매처로 적어 두면, 모델이 해당 명령을 부르려 할 때 매처가 매치되어 도구 호출이 차단됩니다. deny는 allow보다 항상 우선하므로, 이 한 줄로 작업 흐름 전체에서 명령이 막힙니다.

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf*)",
      "Bash(rm -fr*)",
      "Bash(git push --force*)",
      "Bash(git push -f*)",
      "Bash(curl * | sh*)",
      "Bash(wget * | sh*)",
      "Read(.env)",
      "Read(secrets/**)",
      "Read(~/.ssh/**)"
    ]
  }
}
```

매처는 `Bash(패턴)` 형태로 도구 이름과 인자 패턴을 함께 적습니다. `Bash(rm -rf*)`는 `Bash` 도구 호출의 인자가 `rm -rf`로 시작하는 모든 경우를 잡습니다. 와일드카드 `*`는 그 뒤로 어떤 문자가 오든 매치합니다. 같은 명령의 변종을 함께 적어 두는 것이 중요합니다. `rm -rf`만 막아 두면 `rm -fr`로 우회되고, `--force`만 막아 두면 `-f`로 우회됩니다.

매처 표기만으로는 잡기 어려운 패턴이 있습니다. 예를 들어 `ls && rm -rf /tmp/cache`처럼 다른 명령과 결합된 형태나, 인자 사이에 공백이 여러 개 들어가는 변형은 정규식 수준의 검사가 있어야 안전합니다. 이때는 PreToolUse hook에서 `tool_input.command` 문자열을 직접 검사합니다. hook 안에서 명령 문자열에 위험 패턴이 들어 있는지 정규식으로 확인하고, 발견하면 종료 코드 2로 빠져나오면 도구 호출이 차단됩니다. PreToolUse hook의 자세한 구조는 4.1.1과 4.1.2에서 다룹니다.

차단된 명령에 대해 사용자에게 명확한 사유를 보여 주는 것도 중요합니다. deny 패턴 한 줄로 차단되면, 모델이나 사용자는 단지 "거부됨"이라는 메시지만 받게 됩니다. PreToolUse hook을 쓰면 stderr로 사유를 함께 출력할 수 있어, 모델이 다음 시도를 다르게 만들 수 있습니다. 이를테면 "force push는 정책상 금지입니다. 대신 새 브랜치를 만들어 PR을 여십시오"처럼 대안을 함께 안내합니다.

이제 Codex CLI 쪽입니다. Codex는 매처 기반 deny 목록이 없는 대신, sandbox와 approval policy의 조합으로 같은 효과를 냅니다. 핵심은 평소 모드를 좁게 잡고, 위험한 동작은 사용자가 직접 승인하지 않으면 진행하지 못하게 두는 것입니다.

```toml
# ~/.codex/config.toml

sandbox_mode = "workspace-write"
approval_policy = "on-request"

[profiles.review]
sandbox_mode = "read-only"
approval_policy = "never"

[profiles.devbox]
sandbox_mode = "workspace-write"
approval_policy = "on-request"
```

`workspace-write` 모드는 작업 디렉토리 바깥 경로에 쓰기를 막아 `rm -rf /` 류 명령을 운영체제 수준에서 차단합니다. 네트워크가 기본 차단이므로 `curl | sh` 류는 외부 호출 단계에서 실패합니다. `git push --force`처럼 작업 디렉토리 안에서 일어나지만 결과는 외부에 미치는 명령은 sandbox만으로는 막히지 않지만, `approval_policy = "on-request"` 설정이 있어 모델이 위험을 인지하고 승인 요청을 띄우거나, on-request를 일관되게 적용하는 정책으로 운영하면 사람이 한 번 확인하게 됩니다.

두 도구의 설정을 한 표로 비교하면 차이가 또렷합니다.

| 위험 명령 | Claude Code 차단 위치 | Codex CLI 차단 위치 |
| --- | --- | --- |
| `rm -rf /` | `permissions.deny`의 `Bash(rm -rf*)` | workspace-write sandbox가 외부 경로 쓰기 거부 |
| `git push --force` | `permissions.deny`의 `Bash(git push --force*)` | approval_policy = on-request가 사용자 승인 요구 |
| `curl ... \| sh` | `permissions.deny`의 `Bash(curl * \| sh*)` | 기본 네트워크 차단이 외부 호출 거부 |
| `.env` 읽기 | `permissions.deny`의 `Read(.env)` | sandbox + 명시적 디렉토리 제한 |

같은 명령을 두 도구에서 모두 차단하려면, 한 레포에 두 설정 파일을 함께 두고 한 줄씩 같은 정책을 새깁니다. Claude Code 쪽은 `settings.json`의 `permissions.deny`, Codex CLI 쪽은 `~/.codex/config.toml`의 sandbox·approval 조합입니다.

차단된 명령에 명확한 사유를 함께 주는 메시지는 모델의 다음 행동을 바꾸는 효과가 있습니다. 단순히 "denied"만 돌려보내면 모델은 다른 표현으로 같은 시도를 반복하기 쉽습니다. PreToolUse hook이나 정책 문서에서 "이 명령은 회복 비용이 큰 동작이라 차단합니다. 대안 X, Y를 시도하십시오"처럼 대안과 이유를 함께 적으면, 모델은 차단된 명령 대신 그 대안을 다시 시도합니다.

마지막으로 예외 허용입니다. 정상 작업 중에 deny 규칙이 너무 광범위해 막힐 때가 있습니다. 이때 전역 `settings.json`을 풀어 버리면 사고 위험이 다시 커집니다. 대신 `settings.local.json`을 쓰거나 Codex profile을 새로 만들어, 그 명령에 한정한 일시적 예외를 두는 편이 안전합니다. `settings.local.json`은 `.gitignore`에 두고, profile은 이름으로 위험도가 드러나게 짓습니다. 일시 예외라는 점이 파일과 이름에 남아, 끝난 뒤 정리하기 쉬워집니다.

정리하면, 위험 명령 차단은 Claude Code의 `permissions.deny`와 PreToolUse hook, Codex CLI의 sandbox와 approval policy를 한 레포에 함께 두어 두 도구가 동일하게 거부하도록 묶는 작업입니다. 차단 사유를 명시하고, 예외는 `settings.local.json`·profile로 분리해 흔적을 남깁니다.
