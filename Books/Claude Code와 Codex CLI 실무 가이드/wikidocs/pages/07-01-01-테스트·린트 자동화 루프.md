# 7.1.1 테스트·린트 자동화 루프

에이전트가 코드를 한 줄 고친 뒤 그게 정말 동작하는지 확인하지 않고 끝내면, 사람이 한 번 더 돌아와 테스트를 돌리고 다시 수정을 부탁하는 왕복이 생깁니다. 이 왕복을 줄이려면 에이전트 스스로 변경 직후에 검증을 돌리고, 실패를 만나면 다시 수정에 들어가는 **자동화 루프**가 필요합니다. 6.2.2에서 본 백그라운드 작업이 큰 일을 띄우는 방법을 다뤘다면, 이 단원은 작은 단위 변경마다 즉시 도는 빠른 검증 루프를 두 도구에서 어떻게 짜는지를 정리합니다.

먼저 루프의 골격을 그려 두겠습니다.

```
[모델: 코드 변경]
   ↓
[검증: 포맷터 → 린터 → 단위 테스트]
   ↓
  성공 → 다음 변경 또는 작업 종료
  실패 → 실패 출력만 모델에 전달 → 변경 재시도 (상한 N회)
```

세 단계는 모두 변경된 파일과 직접 관련된 빠른 명령들입니다. 포맷터는 코드 스타일을 기계적으로 맞추고, 린터는 코드 표면의 문제를 잡고, 단위 테스트는 동작 회귀를 잡습니다. 그 위에 상위 단계 검증인 통합 테스트나 CI는 7.1.2에서 따로 다룹니다. 자동화 루프의 핵심은 **변경 직후**·**짧은 시간**·**해당 변경에 한정된** 검증을 도는 것이며, 셋 중 하나라도 무너지면 컨텍스트 비용이 빠르게 폭증합니다.

Claude Code에서는 이 루프를 4.1.3에서 본 PostToolUse hook과 2.1.1에서 본 CLAUDE.md의 명령 등록으로 짭니다. PostToolUse는 도구 호출이 끝난 직후에 끼어드는 길목이며, 종료 코드가 검증 결과의 신호 역할을 합니다. 종료 코드 0이면 조용히 다음으로 넘어가고, 종료 코드 2이면 stderr 내용이 모델에게 다시 전달되어 모델이 그 메시지를 받아 후속 행동을 결정합니다. 다음은 settings.json에 등록하는 표준 형태입니다.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "scripts/check-changed.sh"
          }
        ]
      }
    ]
  }
}
```

`PostToolUse` 키 아래의 배열에 각 hook 그룹을 둡니다. `matcher`는 어떤 도구의 호출 뒤에 실행할지를 거르는 정규식 필터로, 여기서는 파일 편집을 일으키는 Edit와 Write에만 걸도록 했습니다. 그 안의 `hooks` 배열에는 실행할 명령을 둡니다. `type: "command"`는 셸 명령이라는 뜻이고, `command`는 실제 실행 줄입니다. 별도 스크립트로 분리해 두면 명령이 길어지더라도 settings.json을 깔끔하게 유지할 수 있습니다.

스크립트 자체는 다음과 같은 모양이 됩니다.

```bash
#!/usr/bin/env bash
# scripts/check-changed.sh
set -e
files="$(git diff --name-only --diff-filter=ACMR | tr '\n' ' ')"
[ -z "$files" ] && exit 0

ruff check $files >&2 || { echo "RUFF FAILED" >&2; exit 2; }
ruff format --check $files >&2 || { echo "FORMAT FAILED" >&2; exit 2; }
pytest -q --no-header --maxfail=3 >&2 || { echo "TESTS FAILED" >&2; exit 2; }

exit 0
```

이 스크립트가 하는 일을 한 줄씩 풉니다. 첫 줄의 `set -e`는 어느 단계라도 실패하면 곧장 종료하라는 뜻입니다. `git diff --name-only --diff-filter=ACMR`은 현재 변경된 파일 가운데 추가·복사·수정·이름 변경에 해당하는 것만 골라 줍니다. 변경이 없으면 검증할 게 없으므로 종료 코드 0으로 조용히 빠져나갑니다. ruff 두 줄은 린트와 포맷 검사이고, pytest 한 줄은 단위 테스트입니다. 각 단계가 실패하면 stderr로 짧은 표시를 남기고 종료 코드 2로 빠져나갑니다. 종료 코드 2는 Claude Code에게 정해진 의미를 가집니다. 도구 호출은 그대로 끝났지만 후속 정책이 차단·재고를 요구한다는 신호이며, 모델은 stderr 내용을 새 컨텍스트로 받아 다음 행동을 결정합니다. `>&2` 리다이렉트는 모든 출력이 stderr로 합쳐지도록 만들어, 모델이 실패 메시지를 빠뜨리지 않게 합니다.

CLAUDE.md 쪽에는 같은 명령을 사람도 직접 부를 수 있는 형태로 적어 둡니다. 4.1.3의 hook이 자동 길목이라면, CLAUDE.md의 명령 카탈로그는 모델이 명시적으로 검증을 한 번 더 돌리고 싶을 때 부를 명령을 알리는 메모입니다. 둘이 같은 스크립트를 가리키게 해 두면 자동 검증과 명시 검증이 서로 다른 동작을 하는 사고가 줄어듭니다.

Codex CLI에서는 PostToolUse 같은 길목 hook이 없습니다. 대신 2.1.2에서 정리한 AGENTS.md의 Build·Test·Style 섹션을 활용합니다. Codex CLI는 작업 단위로 끝나는 모드가 기본이므로, 작업이 끝나기 직전에 AGENTS.md에 적힌 검증 명령을 모델이 스스로 부르도록 유도하는 방식이 자연스럽습니다.

```markdown
# AGENTS.md

## Build
- 의존성: `uv sync`

## Test
- 변경된 파일만 빠르게: `scripts/check-changed.sh`
- 전체: `pytest -q`

## Style
- 포맷·린트: `ruff check . && ruff format --check .`

## Loop
- 코드 편집 후 반드시 `scripts/check-changed.sh`를 돌릴 것
- 실패가 나면 출력의 마지막 30줄만 보고 수정 → 다시 같은 명령으로 검증
- 같은 실패가 세 번 반복되면 멈추고 사람에게 보고
```

AGENTS.md의 Loop 섹션이 곧 루프의 규칙입니다. 변경 후에는 항상 같은 스크립트를 부르고, 실패 출력은 마지막 30줄만 본다는 제한을 두고, 재시도 횟수에 상한을 명시합니다. 이렇게 규칙을 텍스트로 적어 두면 Codex CLI 모델이 작업 종료 직전에 이 카탈로그를 보고 검증 명령을 호출합니다.

실패 출력의 양을 줄여 컨텍스트를 아끼는 일은 두 도구 모두에서 똑같이 중요합니다. 가장 흔한 실수는 테스트 러너의 기본 출력을 그대로 모델에게 흘리는 일입니다. 페이지 단위로 흐르는 트레이스백이 그대로 컨텍스트에 들어가면 한두 번의 실패만으로도 토큰이 빠르게 빠집니다. 위의 ruff와 pytest 호출에 둔 `-q`, `--no-header`, `--maxfail=3`은 같은 목적의 장치입니다. 출력은 짧게, 실패가 누적되기 전에 끊고, 같은 종류의 실패가 반복되면 위에서부터 세 건만 본 뒤 멈춥니다. 더 큰 실패 로그가 필요하면 파일로 떨궈 두고 모델은 끝부분만 잘라 읽도록 합니다.

루프가 끝나지 않는 상황도 미리 막아야 합니다. 가장 잦은 사고는 같은 자리에서 같은 실패가 반복되는 경우이고, 다음으로 잦은 사고는 검증이 변경을 일으켜 그 변경이 다시 검증을 부르는 무한 회로입니다. 첫 사고에는 **재시도 상한**으로 대응합니다. AGENTS.md의 Loop 섹션처럼 세 번이라는 분명한 숫자를 두고, 그 위에서는 사람에게 보고하도록 합니다. 둘째 사고는 **자동 변경 도구를 hook에서 직접 부르지 않는** 식으로 줄입니다. 포맷 검사는 `--check` 모드로만 돌려 파일을 건드리지 않게 하고, 실제 포맷 적용은 모델이 명시적으로 부를 때만 실행되도록 분리합니다. 4.1.3에서 본 캐시 전략도 같은 효과를 냅니다. 직전 검증의 결과를 키로 묻어 두어, 같은 파일에 같은 검증을 짧은 시간 안에 다시 돌리는 일을 줄입니다.

정리하면, 테스트·린트 자동화 루프는 변경 직후에 짧고 좁은 범위의 검증을 돌리고, 실패 출력을 종료 코드와 stderr로 모델에게 다시 전달하는 구조입니다. Claude Code는 PostToolUse hook에 매처와 스크립트를 두고 CLAUDE.md의 명령 카탈로그와 짝지어 자동·명시 검증을 같은 스크립트로 맞추며, Codex CLI는 AGENTS.md의 Build·Test·Style·Loop 섹션으로 같은 규칙을 텍스트로 새겨 작업 종료 직전에 스스로 부르게 합니다. 출력을 짧게 자르고 재시도 상한을 분명히 두어야 루프가 컨텍스트와 시간을 잠식하지 않습니다.

다음 단원인 7.1.2에서는 같은 루프를 한 단계 위로 올려, CI에서 발생한 실패 로그를 도구가 자동으로 가져와 스스로 수정에 들어가게 만드는 방법을 다룹니다.

이 단원을 마치면 변경 후 스스로 테스트·린트를 돌려 실패를 흡수하는 루프를 두 도구에서 만들 수 있게 됩니다.
