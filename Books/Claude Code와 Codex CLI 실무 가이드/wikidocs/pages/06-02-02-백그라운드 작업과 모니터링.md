# 6.2.2 백그라운드 작업과 모니터링

테스트 스위트 전체 실행이나 대용량 데이터 마이그레이션처럼 몇 분에서 수십 분이 걸리는 작업이 있습니다. 이런 작업을 에이전트가 직접 기다리면 그 시간 동안 대화 컨텍스트가 점유되고 다른 일을 시킬 수 없으며, 출력이 끝없이 흘러 컨텍스트 길이를 잠식합니다. 6.2.1에서 본 profile과 codex exec가 작업의 안전 묶음을 정했다면, 이 단원은 그 작업을 **백그라운드**로 띄워 두고 다른 일을 하면서 진행을 비파괴적으로 살피는 패턴을 다룹니다.

백그라운드라는 말부터 풀어 두겠습니다. 셸 운영에서 백그라운드 실행은 명령을 띄운 뒤 셸을 다시 입력 받을 수 있는 상태로 되돌리고, 명령 자체는 별도 프로세스로 계속 돌아가는 실행 방식을 가리킵니다. 에이전트 맥락에서도 같은 뜻입니다. 모델이 어떤 작업을 띄운 뒤 그 결과를 직접 기다리며 출력 줄을 모두 컨텍스트에 받아들이는 대신, 별도 프로세스로 계속 돌리게 두고 자신은 곧장 다음 행동을 결정할 수 있게 합니다.

Claude Code의 Bash 도구는 이를 위해 **run_in_background** 옵션을 직접 가지고 있습니다. 모델이 Bash를 호출할 때 이 옵션을 켜면 명령은 별도 프로세스로 시작되며 도구 응답은 즉시 돌아옵니다. 동시에 시스템은 그 명령에 짧은 식별자를 붙여 두고, 모델이 나중에 진행 상황을 확인할 수 있게 **Monitor** 도구를 제공합니다. Monitor는 백그라운드 프로세스가 표준 출력에 새 줄을 토할 때마다 알림을 흘려보내는 통로이며, 모델이 끝없이 폴링하는 대신 새 줄이 생긴 시점에 호출만 받으면 되도록 설계되어 있습니다.

전형적인 호출 흐름은 다음과 같습니다.

```
[모델] Bash("pnpm test", run_in_background=true)
   ↓
[하네스] 별도 프로세스 시작 → bash_id 부여 → "started" 응답
   ↓
[모델] 곧장 다른 도구 호출 (예: Read, Edit)
   ↓
[하네스] 백그라운드 명령이 새 줄을 출력
   ↓
[Monitor] 새 줄 알림을 모델에게 전달
   ↓
[모델] 결과 보고 종료 처리 또는 후속 행동 결정
```

이 흐름에서 모델이 직접 기다리는 시간은 거의 없습니다. Bash 호출은 곧장 돌아오고, 후속 작업을 하다가 Monitor 알림이 도착했을 때 그 시점에 결과를 처리합니다. 한 가지 주의점은 백그라운드 명령의 출력이 모두 모델 컨텍스트로 흐른다는 점입니다. 출력이 폭주하는 명령은 백그라운드라 하더라도 컨텍스트를 빠르게 잠식하므로, 명령 끝에 출력 양을 줄이는 옵션을 붙이거나 `> logs/test.log 2>&1`로 파일에 떨궈 두고 모델은 마지막에 요약만 읽게 하는 편이 안전합니다.

Codex CLI는 모델이 직접 부르는 백그라운드 도구를 가지고 있지 않습니다. 대신 셸 자체의 백그라운드 실행을 활용하는 패턴이 표준입니다. 가장 흔한 형태는 `codex exec`를 운영 체제의 백그라운드로 띄우고, 출력을 파일로 받아 두는 방식입니다.

```bash
codex --profile builder exec \
  "전체 단위 테스트를 돌리고 실패만 docs/notes/failures.md로 정리해 줘" \
  > logs/codex-run.log 2>&1 &
echo $! > logs/codex-run.pid
```

첫 줄은 6.2.1에서 본 비대화형 실행을 그대로 부르되, 표준 출력과 표준 오류를 합쳐 `logs/codex-run.log`로 보내고, 마지막의 `&`로 셸을 백그라운드로 띄웁니다. 다음 줄은 셸이 마지막으로 띄운 프로세스의 PID를 `logs/codex-run.pid`에 저장합니다. PID는 나중에 진행을 확인하거나 중단할 때 쓰는 손잡이입니다. 별도 디렉토리에 사본을 두고 같은 패턴을 두세 번 반복하면 6.2.1의 격리 단위 원칙에 맞춰 여러 작업을 병렬로 띄울 수 있습니다.

진행 상황을 비파괴적으로 본다는 말은 작업을 멈추지 않고 그 상태만 들여다본다는 뜻입니다. Claude Code 측에서는 Monitor 도구가 그 통로를 제공하지만, Codex CLI에서는 로그 파일을 외부 명령으로 들여다보는 방식이 자연스럽습니다.

```bash
tail -n 30 logs/codex-run.log
ps -p "$(cat logs/codex-run.pid)" -o pid,etime,stat,cmd
ls -la docs/notes/failures.md 2>/dev/null
```

첫 명령은 로그의 끝부분만 짧게 잘라 봅니다. 작업이 살아 있다면 새 줄이 계속 쌓이므로 같은 명령을 다시 부르면 직전 호출과의 차이로 진척도를 가늠할 수 있습니다. 둘째 명령은 PID가 가리키는 프로세스의 경과 시간과 상태를 보여 줍니다. `etime`이 한 시간을 넘기 시작하는데 로그가 멎어 있다면 멈춘 작업일 가능성이 크다는 신호로 읽습니다. 셋째 명령은 작업이 만들겠다고 한 산출 파일이 생겼는지를 확인합니다. 세 신호를 함께 보면 작업을 건드리지 않고도 살아 있음과 진척과 완료 여부를 가늠할 수 있습니다.

장시간 작업이 컨텍스트를 점유하지 않게 하려면 두 가지 원칙을 같이 적용합니다. 첫째, **출력은 파일로 떨굽니다.** 모델이 백그라운드 작업의 모든 줄을 컨텍스트로 받아 두면 토큰이 빠르게 소진됩니다. 로그를 파일로 모으고 모델은 끝에서 짧게 잘라 읽거나 요약만 읽도록 합니다. 둘째, **확인을 폴링하지 않습니다.** 같은 명령을 1초마다 반복해 부르면 그 자체로 컨텍스트와 토큰이 낭비됩니다. Claude Code에서는 Monitor 알림에 의지하고, Codex CLI에서는 산출 파일이 생겼는지를 확인하는 한 줄 명령을 가끔 부르는 정도로 충분합니다.

실패가 보일 때 빠르게 중단하고 다시 시작하는 절차도 미리 정해 두어야 합니다. Claude Code의 백그라운드 Bash는 시스템이 보유한 식별자로 종료 신호를 보낼 수 있고, 모델이 그 신호를 보내기로 결정할 때까지 사용자가 직접 멈춰 두라고 지시하는 형태도 자주 씁니다. Codex CLI에서는 PID 파일에 저장한 식별자로 표준 신호를 보냅니다.

```bash
kill "$(cat logs/codex-run.pid)"   # 정상 종료 요청
sleep 5
kill -9 "$(cat logs/codex-run.pid)" 2>/dev/null || true   # 강제 종료
rm -f logs/codex-run.pid
```

첫 줄은 운영 체제에 기본 종료 신호를 보내 작업이 스스로 정리하고 끝낼 기회를 줍니다. 둘째 줄은 5초 정도 여유를 두고, 셋째 줄은 그래도 살아 있으면 강제 종료 신호를 보냅니다. 마지막 줄은 더 이상 의미가 없어진 PID 파일을 정리합니다. 다시 시작할 때는 같은 명령을 그대로 부르되, 직전 로그를 다른 이름으로 옮겨 두어 비교에 활용할 수 있게 합니다.

정리하면, Claude Code는 Bash 도구의 run_in_background 옵션과 Monitor 도구를 짝지어 모델이 직접 기다리지 않고도 새 출력만 알림으로 받게 하고, Codex CLI는 셸 백그라운드 실행에 로그·PID 파일을 곁들여 비파괴적인 점검과 빠른 중단을 가능하게 합니다. 출력은 파일로 떨구고, 확인은 폴링이 아닌 알림이나 단발 명령으로 끊어 두면 장시간 작업이 대화 컨텍스트를 점유하지 않습니다.

다음 단원인 7.1.1에서는 이렇게 띄운 작업이 끝났을 때, 그 결과를 모델에게 다시 흘려보내 코드 변경을 스스로 보정하게 만드는 테스트·린트 자동화 루프를 다룹니다.

이 단원을 마치면 두 도구에서 장시간 작업을 백그라운드로 띄우고 진행을 비파괴적으로 모니터링하며, 실패 시 빠르게 중단·재시작하는 절차를 적용할 수 있게 됩니다.
