# 6.1.1 Claude Code subagent 정의와 호출

큰 작업을 한 세션에서 끝까지 끌고 가다 보면 컨텍스트가 부풀어 모델이 산만해지고, 한 자리에서 권한이 너무 넓어져 위험이 커지기도 합니다. Claude Code는 이 문제를 해결하기 위해 메인 세션과 분리된 작은 에이전트를 정의하는 장치를 제공합니다. 이를 subagent라고 부릅니다. 메인 에이전트가 일을 잘게 나누어 subagent에게 위임하고, 결과만 받아 다음 결정을 내리는 식입니다. 이번 단원에서는 subagent 파일의 구조, tools 화이트리스트의 효과, 메인이 subagent를 부르는 트리거, 그리고 자주 쓰는 3종 구성을 정리합니다.

subagent의 본체는 마크다운 파일 한 개입니다. 위치는 정해져 있어서, 프로젝트 공용 subagent는 `.claude/agents/NAME.md`에 두고, 사용자 개인 subagent는 `~/.claude/agents/NAME.md`에 둡니다. 5.2.1에서 본 슬래시 커맨드와 위치가 비슷해 보이지만, 부르는 주체가 다릅니다. 슬래시 커맨드는 사용자가 입력 창에서 직접 부르는 단위이고, subagent는 메인 에이전트가 작업 중에 호출하는 단위입니다.

파일 맨 위에는 frontmatter가 들어갑니다. 핵심 필드 네 개를 살펴봅니다. `name`은 subagent의 이름으로, 메인이 위임할 때 이 이름으로 부릅니다. `description`은 어떤 일을 하는지 한 줄로 적은 설명입니다. 메인은 이 설명을 보고 위임 대상을 고르므로, "언제 부르면 좋은지"가 또렷하게 드러나도록 적습니다. `tools`는 이 subagent에게만 허용할 도구 화이트리스트입니다. `model`은 사용할 모델을 따로 지정하는 칸이며, 비워 두면 메인과 같은 모델을 씁니다. 다음 예시가 네 필드와 본문의 관계를 보여 줍니다.

```markdown
---
name: code-researcher
description: 코드베이스를 탐색해 변경 영향 범위를 조사한다. 수정은 하지 않는다.
tools: Read, Grep, Glob
model: claude-sonnet
---

당신은 조사 전용 보조 에이전트다. 절대 파일을 수정하거나 명령을 실행하지 않는다.
주어진 키워드와 파일 경로를 단서로 다음을 정리해 메인에게 돌려준다.
- 관련 파일 경로와 핵심 함수
- 변경이 닿을 수 있는 호출 지점
- 추가 조사가 필요한 영역
응답은 짧은 문단과 경로 목록만 사용한다.
```

frontmatter의 `tools` 화이트리스트가 가장 강한 안전장치 역할을 합니다. 위 예시에서 `tools`는 `Read, Grep, Glob` 세 가지만 허용합니다. 즉 이 subagent가 살아 있는 동안은 파일을 쓸 수도, Bash를 돌릴 수도 없습니다. 메인 세션의 권한이 더 넓더라도 subagent의 권한이 자동으로 따라가지 않습니다. 권한을 좁히는 일이 한 자리에서 끝나므로, 조사용·구현용·리뷰용 같은 역할마다 다른 권한 폭을 둘 수 있게 됩니다.

메인 에이전트가 subagent를 호출하는 트리거는 두 갈래입니다. 한 갈래는 사용자가 명시적으로 "이 일은 X에게 맡겨"라고 지시하는 경우입니다. 다른 갈래는 메인이 작업 중에 스스로 판단해 위임하는 경우이며, 이때 `description` 필드가 결정의 근거가 됩니다. Claude Code는 사용 가능한 subagent 목록과 그 설명을 메인 컨텍스트에 함께 두고, 새 작업이 들어왔을 때 어떤 subagent가 어울리는지 모델에게 평가하도록 합니다. 메인이 호출을 결정하면 subagent의 본문, 받은 인자, 허용 도구만으로 작은 세션을 띄우고, 작업이 끝나면 결과 텍스트를 메인에게 되돌려 줍니다.

```
사용자 → 메인 에이전트
            │
            │ "이 변경 영향 범위 조사해 줘"
            ▼
   description 매칭 후 위임
            │
            ▼
   code-researcher subagent
   (도구: Read, Grep, Glob 만)
            │
            │ 결과 텍스트 반환
            ▼
       메인 에이전트
            │
            ▼ "이 영역만 수정하자"로 다음 결정
```

위 흐름에서 메인은 자기 컨텍스트에 코드 전체를 끌어들이지 않고, subagent의 요약된 결과만 받아 다음 결정을 합니다. 컨텍스트 절약이 자연스럽게 일어나고, 권한도 한 자리에 몰리지 않습니다.

실무에서 자주 쓰이는 표준 구성은 조사·구현·리뷰 세 종류입니다. 첫째, 조사용 subagent는 위의 예시처럼 읽기 전용 도구만 갖춥니다. 코드베이스를 훑어 호출 관계를 정리하거나, 관련 문서를 찾아 요약해 옵니다. 둘째, 구현용 subagent는 Read·Edit·Write·Bash를 모두 갖되, 작업 범위를 본문에서 강하게 좁힙니다. 예를 들어 "이 한 디렉토리 안의 파일만 수정한다"라고 본문에 명시해 둡니다. 셋째, 리뷰용 subagent는 다시 읽기 전용으로 두되, 메인이 작성한 diff를 받아 표준 체크리스트로 검토합니다.

```markdown
---
name: code-reviewer
description: 메인이 만든 변경분에 대해 표준 체크리스트로 리뷰한다.
tools: Read, Grep, Bash(git diff:*)
---

당신은 리뷰 전용 보조 에이전트다. 코드를 수정하지 않는다.
주어진 diff에 대해 다음을 점검해 한 문단씩 결과를 돌려준다.
- 입력 검증과 경계 조건
- 테스트 누락 가능성
- 인접 코드와 스타일 일치 여부
- 되돌리기 가능성
```

세 subagent를 함께 두면, 메인은 작업 단계마다 다른 권한 폭으로 일을 떼어내 보낼 수 있습니다. 조사 단계에서는 쓰기 권한이 닿지 않게 하고, 구현 단계에서만 쓰기를 열고, 다시 리뷰 단계에서 쓰기를 닫는 식입니다. 같은 일을 한 세션에서 권한 폭만 좁혀 다 처리하기는 까다롭지만, subagent로 단계를 쪼개면 자연스럽게 구분됩니다.

메인이 결정을 내리는 패턴도 함께 정리해 둡니다. subagent의 응답은 자유 텍스트이므로, 메인이 그 텍스트를 다음 단계 입력으로 어떻게 쓸지가 핵심입니다. 권장 패턴은 두 가지입니다. 첫째, subagent에게 응답 형식을 본문에서 강하게 잡아 두는 것입니다. 예를 들어 "결과는 짧은 문단과 경로 목록만 사용한다"라고 적어 두면, 메인이 파싱 없이도 다음 단계로 넘어가기 쉽습니다. 둘째, 메인이 받은 결과를 그대로 사용자에게 보여 주지 말고, 한 번 더 요약해 다음 행동을 제안하도록 두는 것입니다. 그래야 사용자가 단계별 결정에 개입할 여지가 남습니다.

마지막으로 슬래시 커맨드·skill과 subagent를 헷갈리지 않도록 짚어 둡니다. 5.2.1의 슬래시 커맨드는 같은 세션 안에서 사용자의 한 줄 입력을 길게 풀어 주는 매크로에 가깝습니다. skill은 같은 세션 안에서 모델이 끌어 쓸 수 있는 묶음 자료입니다. 반면 subagent는 권한 폭과 컨텍스트가 메인과 분리된 별도의 작은 세션을 띄웁니다. 같은 일을 매크로로 풀지, 별도 에이전트에게 맡길지 고르는 기준은 권한 분리가 필요한가입니다.

정리하면, subagent는 `.claude/agents/NAME.md`에 `name`, `description`, `tools`, `model`을 적은 frontmatter와 본문으로 정의하며, 메인 에이전트가 description을 보고 위임 호출합니다. `tools` 화이트리스트로 권한을 좁혀 두면, 조사·구현·리뷰 단계마다 다른 권한 폭을 자연스럽게 분리할 수 있습니다. 메인은 subagent의 결과 텍스트를 받아 다음 결정을 내리고, 사용자에게는 한 번 더 정리해 보여 줍니다.

다음 단원인 6.1.2에서는 변경 전에 계획을 먼저 합의하는 Plan mode와, 큰 작업을 여러 단계로 쪼개는 방법을 다룹니다.

이 단원을 마치면 subagent 파일을 직접 만들고, 표준 3종 구성을 짜며, 메인이 subagent를 호출하는 트리거와 결과를 받아 결정을 내리는 패턴을 설명할 수 있습니다.
