# 2.1.1 CLAUDE.md 작성 패턴

처음 Claude Code를 도입한 팀이 자주 겪는 실수는 모든 프로젝트 정보를 한 파일에 몰아 넣는 것입니다. 자동으로 매 세션에 로드되는 파일이라서, 길이가 늘어날수록 토큰 비용과 모델의 주의 분산도 늘어납니다. 이 단원에서는 `CLAUDE.md`가 어떻게 로드되는지, 어떤 섹션 구조가 잘 작동하는지, 무엇을 넣지 말아야 하는지를 함께 정리합니다.

먼저 **자동 로딩 규칙**입니다. Claude Code는 세션 시작 시 세 위치의 `CLAUDE.md`를 자동으로 읽어 모델 컨텍스트에 포함합니다. 첫째는 **프로젝트 루트**의 `CLAUDE.md`입니다. 작업 디렉터리에서 위로 올라가며 가장 가까운 루트를 찾고, 그곳의 파일이 우선합니다. 둘째는 **하위 디렉터리**의 `CLAUDE.md`입니다. 모노레포 같은 구성에서 특정 패키지에만 적용할 규칙을 둘 때 씁니다. 같은 폴더의 작업이 시작되면 그 폴더의 `CLAUDE.md`도 추가로 로드됩니다. 셋째는 **사용자 전역**인 `~/.claude/CLAUDE.md`입니다. 모든 프로젝트에 공통으로 적용할 개인 지침을 넣는 자리입니다. 세 위치는 함께 로드되며, 충돌이 생기면 더 가까운(프로젝트·하위) 파일의 지침이 우선합니다.

**권장 섹션 구성**은 네 덩어리로 추리는 편이 안정적입니다. `Commands`, `Conventions`, `Gotchas`, `Self-reflection on Errors`입니다. 각 섹션은 짧고 검사 가능한 문장으로 채웁니다.

`Commands`에는 이 프로젝트에서 자주 실행하는 한 줄 명령을 모읍니다. 빌드, 테스트, 배포, 자주 쓰는 슬래시 커맨드의 호출 형태를 한 줄씩 적어 두면, 모델이 같은 작업을 두 번 묻지 않습니다. `Conventions`에는 코드 스타일과 디렉터리 규칙, 커밋 메시지 컨벤션처럼 변하지 않는 약속을 적습니다. `Gotchas`에는 이 레포에서 한 번 이상 사고가 났던 자리를 적습니다. "이 경로는 다른 시스템과 공유되므로 직접 편집 금지" 같은 문장이 여기에 들어갑니다. `Self-reflection on Errors`에는 오류가 발생했을 때 모델이 어떤 절차로 원인을 찾아야 하는지를 적습니다. 세션이 어긋났을 때 모델이 어디로 돌아갈지 알려 주는 지도가 됩니다.

```markdown
# CLAUDE.md

## Commands
- 테스트: `pnpm test`
- 린트: `pnpm lint --fix`
- PDF 빌드: `python build_pdf.py 02_Books/{책이름}`

## Conventions
- 커밋은 feature 브랜치에서만. main 직접 push 금지.
- 한국어 문서에는 직선 따옴표(' ")만 사용.

## Gotchas
- `Rules/` 폴더는 hook으로 차단됨. 수정 금지.
- Windows + Git Bash 환경. 경로는 슬래시 사용.

## Self-reflection on Errors
오류·실패·hook 차단이 발생하면:
1. 증상이 아니라 근본 원인을 찾는다.
2. 같은 패턴이 반복되면 CLAUDE.md에 한 줄 규칙을 제안한다.
```

분량이 커지면 **@경로 import**로 규칙 파일을 분리합니다. `CLAUDE.md` 본문에 `@./.claude/rules/curriculum-editing.md` 같은 줄을 두면, 그 파일의 내용이 자동으로 함께 로드됩니다. 작은 주제별로 `.claude/rules/` 아래에 `*.md`를 두고, 메인 `CLAUDE.md`는 그 목차만 가지는 구성이 자주 쓰입니다. 이 구성의 장점은 두 가지입니다. 첫째, 한 규칙이 길어져도 다른 규칙의 가독성을 해치지 않습니다. 둘째, 같은 규칙 파일을 다른 프로젝트에서 재사용할 수 있습니다.

```markdown
# CLAUDE.md (메인)

## Detailed Rules
@./.claude/rules/curriculum-editing.md
@./.claude/rules/topic-writing.md
@./.claude/rules/research-collecting.md
```

다음은 **길이 가이드와 우선순위**입니다. 자동 로딩 파일은 매 세션 토큰을 먹기 때문에, 길수록 비용과 응답 지연이 함께 늘어납니다. 메인 `CLAUDE.md`는 한 화면(50줄 안팎)을 목표로 두고, 세부 규칙은 import로 분리합니다. 우선순위는 (1) 안전과 관련된 금지 사항, (2) 자주 쓰는 명령, (3) 변하지 않는 컨벤션, (4) 자주 틀리는 함정의 순서가 좋습니다. 모델은 위에서부터 읽으므로, 가장 중요한 규칙을 가장 위에 둡니다.

마지막으로 **안티 패턴**을 짧게 짚습니다. 첫째, **장황한 코드 설명**을 넣지 않습니다. 함수가 어떻게 동작하는지 풀어 적는 자리는 코드 주석이나 README이지 `CLAUDE.md`가 아닙니다. 둘째, **변하지 않는 사실의 반복**을 피합니다. 예를 들어 "이 프로젝트는 Python 3.11을 사용합니다"는 `pyproject.toml`에 이미 적혀 있는 사실이므로, 모델이 파일을 한 번 보면 알 수 있는 정보입니다. `CLAUDE.md`에 다시 적으면 토큰만 늘어납니다. 셋째, **느슨한 표현**을 피합니다. "코드 품질을 높이세요" 같은 문장은 검사 가능하지 않아 모델이 따르기 어렵습니다. "함수 하나는 30줄을 넘기지 않는다"처럼 측정 가능하게 적습니다.

정리하면, `CLAUDE.md`는 프로젝트·하위·사용자 세 위치에서 자동 로드되며, Commands·Conventions·Gotchas·Self-reflection 네 섹션으로 짧게 시작하고, 길어지면 `@경로` import로 분리합니다. 변하지 않는 사실과 장황한 설명을 빼고, 검사 가능한 문장만 남기는 것이 토큰과 정확성을 동시에 잡는 길입니다.

다음 단원인 2.1.2에서는 Codex CLI가 자동으로 읽는 `AGENTS.md`의 작성 패턴과 `CLAUDE.md`와의 차이를 다룹니다.
