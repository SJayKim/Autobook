# 8.1.2 팀 공유 설정 패턴

한 명이 쓰던 설정을 팀 전체로 펼치는 순간 두 종류의 사고가 갈라집니다. 한 명에게는 잘 맞던 권한이 다른 사람에게는 너무 빡빡하거나, 누군가 실수로 자신의 API 키가 담긴 파일을 그대로 커밋해 버리는 일입니다. 이 단원에서는 두 도구의 설정 파일을 git으로 공유할 부분과 개인 비공개로 둘 부분으로 나누는 기준, 그리고 신규 합류자가 같은 환경에 빠르게 들어오게 하는 절차를 정리합니다.

원칙부터 한 줄로 둡니다. **팀 전체에 일관되어야 하는 것은 git에 올리고, 개인이나 환경에 따라 달라지는 것은 비공개로 둡니다.** 비공개 영역에 들어가는 가장 큰 두 가지는 API 키와 사람마다 다른 권한 미세 조정입니다.

Claude Code의 설정은 2.2.1에서 본 세 위치 중 두 곳이 팀 공유 대상에 들어옵니다. 프로젝트 루트의 `.claude/settings.json`은 팀 표준입니다. 같은 디렉토리에 두는 `.claude/settings.local.json`은 개인 비공개입니다. 두 파일은 같은 스키마를 가지며, 로컬이 프로젝트 표준 위에 덮어씁니다. 이 구분 위에 `.claude/commands/`와 `.claude/agents/` 디렉토리도 팀 공유 대상이 됩니다. 슬래시 커맨드와 subagent 정의는 모두가 같은 도구를 갖도록 git에 넣습니다.

`.gitignore`에는 비공개 영역과 도구가 만든 캐시·로그를 함께 넣습니다.

```gitignore
# 개인 설정과 로컬 메모만 제외
.claude/settings.local.json
.claude/projects/
.claude/memory/
.claude/handoff/

# Codex 로컬 로그
.codex-cache/
```

`.claude/projects/`와 `.claude/memory/`는 Auto Memory와 세션 로그가 쌓이는 곳이므로 절대 공유 대상이 아닙니다. 7.2.2에서 본 핸드오프 파일도 개인 작업 컨텍스트라서 제외합니다.

git에 올라가는 `.claude/settings.json`에는 모두에게 동일하게 적용할 항목만 둡니다. 예를 들어 main 직접 push 금지(7.2.1), 위험 명령 deny 패턴(3.2.3), 팀 표준 hook 경로, 공유 MCP 서버 정의가 여기에 들어옵니다. 다음은 한 예입니다.

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf /*)",
      "Bash(git push --force*)",
      "Read(./.env*)"
    ],
    "ask": ["Bash(git push*)"]
  },
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash",
        "hooks": [
          { "type": "command",
            "command": "bash .claude/hooks/guard.sh" }
        ]
      }
    ]
  },
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${env:GITHUB_TOKEN}" }
    }
  }
}
```

API 키나 토큰이 자리할 곳에는 직접 값을 적지 않습니다. `${env:GITHUB_TOKEN}` 같이 환경 변수를 가리키는 형태로 두고, 실제 값은 개인이 셸 프로파일이나 시스템 시크릿 매니저에 따로 보관합니다. **`ANTHROPIC_API_KEY`**와 **`OPENAI_API_KEY`**도 같은 규칙입니다. 어떤 경우에도 git에 들어가는 파일에 실키 값을 박지 않습니다.

개인이 자기 환경에서만 쓰고 싶은 권한 완화는 `settings.local.json`으로 분리합니다.

```json
{
  "permissions": {
    "allow": [
      "Bash(brew*)",
      "Read(/Users/me/notes/**)"
    ]
  }
}
```

이 파일은 .gitignore에 들어 있으므로 절대 git에 따라가지 않습니다. 다른 팀원의 머신과 경로가 달라도, 각자가 자기 로컬 파일을 만들어 두면 표준 설정은 그대로 둔 채 개인 차이만 흡수합니다.

Codex CLI 쪽도 구조는 같습니다. `~/.codex/config.toml`은 사용자 홈에 있으므로 본질적으로 개인 비공개입니다. 팀 공유 대상은 프로젝트 루트의 `AGENTS.md`와, 함께 두는 표준 `config.toml` 템플릿 또는 `.codex/profiles.toml` 같은 보조 파일입니다. 합류자는 템플릿을 자신의 `~/.codex/config.toml`로 복사한 뒤 키와 경로만 자기 환경에 맞춰 고칩니다.

CLAUDE.md와 AGENTS.md는 2.1.2에서 본 것처럼 한 레포에서 같은 사실을 두 톤으로 적은 문서입니다. 두 파일이 어긋나면 도구마다 다른 동작을 안내하게 되므로, **둘 중 하나만 고치는 일을 금지**합니다. 변경이 일어날 때마다 두 파일을 같은 PR에서 함께 수정하도록 PR 템플릿이나 CI 체크에 룰을 박아 두는 편이 안전합니다.

신규 합류자 온보딩은 짧은 체크리스트로 묶습니다.

```
1. 도구 설치: claude / codex 명령 동작 확인
2. 인증: ANTHROPIC_API_KEY, OPENAI_API_KEY 환경 변수 설정
3. 레포 clone 후 .claude/settings.local.json 자기 환경에 맞춰 생성
4. ~/.codex/config.toml에 팀 템플릿 복사 후 키 채움
5. 첫 실행: 정찰 프로파일로 read-only 실행 → 정상 응답 확인
6. CLAUDE.md / AGENTS.md 통독
```

이 절차를 README나 `docs/onboarding.md`에 박아 두면, 누가 들어오든 같은 순서로 도구를 정착시킵니다. 5번에서 read-only로 첫 실행을 강제하는 이유는 처음부터 큰 권한으로 시작할 때 발생하는 사고를 줄이기 위함입니다(3.2.1).

마지막은 **보일러플레이트 배포**입니다. 같은 팀 안에 여러 레포가 있다면 표준 hook, MCP 정의, deny 패턴을 별도 템플릿 레포에 묶어 두고, 새 레포를 만들 때 그 템플릿에서 시작하게 합니다. 변경이 생기면 한 곳을 고친 뒤 각 레포로 패치를 흘려보내는 흐름이 됩니다.

```
[template-repo]
  .claude/settings.json   (deny, hooks, mcpServers)
  .claude/commands/       (표준 슬래시 커맨드)
  AGENTS.md / CLAUDE.md   (톤 다른 동일 사실)

   ↓ scaffold

[project-repo-A]   [project-repo-B]   [project-repo-C]
```

정리하면, 팀에 동일해야 하는 정책과 명령은 `.claude/settings.json`, `.claude/commands/`, `.claude/agents/`, `AGENTS.md`, `CLAUDE.md`로 git에 올리고, 키와 개인 권한 완화는 환경 변수와 `settings.local.json`으로 분리하며, 두 지시문 파일은 한 PR에서 함께 고치고, 신규 합류자는 짧은 체크리스트로 동일한 시작점에 맞춥니다.

다음 단원인 8.2.1에서는 이렇게 공유한 설정이 실제로 안전한지 점검하는 체크리스트를 다룹니다.

이 단원을 마치면 두 도구의 설정 파일을 git 공유와 개인 비공개로 정확히 가르고, API 키를 안전하게 분리하며, CLAUDE.md와 AGENTS.md를 동기화하는 규칙과 신규 합류자 온보딩 절차를 만들 수 있습니다.
