# 6.1.2 Plan mode와 작업 분해

큰 변경을 시작하기 전에 가장 흔한 실패는 곧장 코드부터 고치는 일입니다. 모델이 자기 가정으로 파일을 바꾸기 시작하면, 사람이 뒤늦게 다른 결론에 도달해도 되돌리는 비용이 커집니다. Claude Code는 이런 위험을 줄이기 위해 변경 전에 계획을 먼저 합의하는 모드를 제공합니다. 이 모드의 이름이 Plan mode입니다. 이번 단원에서는 Plan mode의 도구 제약, 입장과 퇴장 시점, 큰 작업을 단계로 쪼개는 가이드라인을 정리하고, Codex CLI에서 동등한 흐름을 만드는 방법을 함께 봅니다.

Plan mode의 핵심 제약은 단순합니다. 이 모드 안에서는 읽기 도구만 허용됩니다. 1.2.1에서 본 기본 모드와 비교하면, 파일을 쓰는 도구나 임의의 명령을 실행하는 도구가 차단됩니다. Read·Grep·Glob처럼 코드와 문서를 읽고 패턴을 찾는 도구는 계속 쓸 수 있습니다. 즉, 모델이 코드베이스를 충분히 살펴보고 계획을 세우되, 실제 변경은 손대지 않도록 한 자리에서 막아 두는 장치입니다.

Plan mode에 들어가는 방법은 두 갈래입니다. 첫째, 사용자가 명시적으로 plan 모드로 진입하라고 지시하는 경우입니다. 둘째, 모델이 작업의 위험도를 판단해 스스로 계획부터 잡겠다고 선언하는 경우입니다. 어느 쪽이든 진입한 뒤에는 사용자가 끝낼 때까지 동일한 제약이 유지됩니다. 진입 자체는 가벼운 모드 전환이라, 새 세션을 띄우거나 권한 파일을 고치지 않아도 됩니다.

Plan mode를 끝내는 절차는 한 가지로 정해져 있습니다. 모델이 `ExitPlanMode`라는 도구를 호출해야만 모드를 빠져나올 수 있습니다. 이 도구를 호출할 때 모델은 사용자에게 보여 줄 계획 본문을 함께 전달합니다. 그러면 Claude Code는 그 본문을 사용자 앞에 띄우고, 진행해도 되는지 묻는 승인 창을 보여 줍니다. 사용자가 승인하면 일반 모드로 돌아오면서 계획에 적힌 작업을 이어 갑니다. 승인을 거절하면 모드를 유지한 채 계획을 다듬을 기회를 줍니다.

```
사용자 또는 모델: Plan mode 진입
        │
        ▼
  읽기 도구만 허용 (Read, Grep, Glob, …)
        │
        ▼
  모델이 코드·문서·이슈를 살피며 계획을 작성
        │
        ▼
  ExitPlanMode 호출 (본문에 단계별 계획 포함)
        │
        ▼
  사용자 승인 창 → 승인 시 일반 모드로 복귀
                  거절 시 Plan mode 유지
```

이 흐름의 가장 큰 효과는 합의 지점이 한 자리에 모이는 데 있습니다. 사용자가 계획을 한 번 읽고 승인해 두면, 이후 변경 작업은 그 계획에서 크게 벗어나지 않습니다. 모델이 중간에 충동적으로 다른 길로 가지 않게 만드는 가드 역할도 함께 합니다. 계획이 적힌 메시지가 컨텍스트에 그대로 남아, 다음 작업 단계에서 모델이 자기 기준으로 참조하기 때문입니다.

큰 작업을 어떻게 단계로 쪼개야 할지에 대한 가이드라인도 함께 정리합니다. 한 계획에 단계가 너무 적으면 한 단계가 모호해지고, 너무 많으면 사용자가 검토를 포기합니다. 경험적으로 5단계에서 10단계 사이가 검토하기 쉬운 범위입니다. 단계의 좋은 형태는 세 가지 조건을 만족합니다. 첫째, 각 단계가 한 자리에서 시작해 한 자리에서 끝납니다. 즉 단계 사이에 의존이 분명히 한 방향으로 흐릅니다. 둘째, 각 단계의 결과를 검증할 방법이 한 줄로 적혀 있습니다. "테스트가 통과한다"나 "이 파일에 함수가 새로 생긴다" 정도면 충분합니다. 셋째, 단계 하나의 실행 시간이 너무 길지 않도록 잘립니다. 한 단계가 길어지면 중간에 끊고 점검하기 어렵습니다.

```markdown
1. 변경 영향 범위 조사 → src/와 tests/에서 호출 지점 목록을 만든다
2. 새 인터페이스 시그니처 합의 → 본 메시지에 한 줄로 적는다
3. 인터페이스 도입 → src/api/v2.ts 새 파일에 빈 함수 정의
4. 호출자 1 마이그레이션 → src/cli.ts, 관련 테스트 통과
5. 호출자 2 마이그레이션 → src/server.ts, 관련 테스트 통과
6. 구 인터페이스 삭제 → 검색 결과가 비었는지 확인
7. 변경 로그 작성 → CHANGELOG.md 갱신
```

위 예시는 같은 작업을 7단계로 쪼갠 모양입니다. 각 단계가 한 자리에서 시작해 한 자리에서 끝나고, 검증 방법이 한 줄로 붙어 있으며, 단계 사이의 의존이 한 방향으로 흐릅니다.

Plan mode와 자주 헷갈리는 것이 메인 세션의 할일 목록입니다. 같은 단계가 두 곳에 적힐 수 있어 그렇습니다. 두 장치의 역할을 구분해 두면 혼동이 줄어듭니다. Plan mode의 계획은 변경 전에 사용자 승인을 받기 위한 일회성 합의문이며, 한 번 승인되면 모드를 빠져나갑니다. 메인 세션의 할일 목록은 작업이 진행되는 동안 계속 갱신되는 상태 보드입니다. 일반적으로는 Plan mode의 단계들을 그대로 할일 목록에 옮긴 뒤, 작업이 진행되면서 항목 상태를 갱신해 가는 식입니다. 둘은 분리된 두 자리이지, 같은 자리에 적는 같은 정보가 아닙니다.

선수 단원과의 연결도 짚어 둡니다. 6.1.1에서 본 subagent는 한 단계의 실행을 권한 분리된 작은 세션에 맡기는 장치입니다. Plan mode는 단계 자체를 합의하는 장치입니다. 큰 작업이라면 두 장치를 함께 씁니다. Plan mode에서 단계를 합의하고, 각 단계의 실행을 적절한 subagent에 위임하면, 합의된 계획과 권한 분리가 동시에 일어납니다. 예를 들어 1단계의 조사는 조사용 subagent에 맡기고, 3단계와 4단계의 구현은 구현용 subagent에 맡기고, 마지막에 리뷰용 subagent에게 리뷰를 받게 두는 식입니다.

Codex CLI에는 Plan mode와 같은 이름의 모드가 없습니다. 같은 흐름을 흉내 내려면 두 가지 도구를 활용합니다. 첫째는 3.2.1에서 본 sandbox 모드입니다. 정찰 단계에서는 `read-only` profile로 진입해 어떤 쓰기도 일어나지 않도록 한 뒤, 모델에게 계획 문서를 짧게 작성하도록 지시합니다. 둘째는 작업 단위 분리입니다. 작성된 계획을 사용자가 한 번 읽고 승인한 다음, 단계마다 별도의 `codex exec` 호출을 띄워 실행을 분리합니다. 다음 표가 두 도구의 대응 관계를 정리합니다.

| 단계 | Claude Code | Codex CLI |
| --- | --- | --- |
| 계획만 작성 | Plan mode 진입, 읽기 도구만 허용 | read-only profile, 작성 결과만 출력 |
| 계획 승인 | ExitPlanMode 후 승인 창 | 사용자 검토 후 다음 명령 실행 |
| 실행 단계 분리 | subagent 위임 | profile 변경 + `codex exec` 분리 |

정리하면, Plan mode는 변경 전에 읽기 도구만 허용한 채 계획을 짜고, `ExitPlanMode` 도구로 사용자 승인을 거쳐 빠져나오는 모드입니다. 큰 작업은 검증 방법과 의존 방향이 또렷한 5에서 10단계로 쪼개고, 각 단계의 실행을 subagent나 일반 세션이 이어 받습니다. Codex CLI에는 같은 이름의 모드가 없지만, read-only profile과 작업 분리로 비슷한 합의 흐름을 만들 수 있습니다.

다음 단원인 6.2.1에서는 Codex CLI에서 작업을 분해하고 profile을 활용해 한 레포에서 여러 모드를 운용하는 방법을 다룹니다.

이 단원을 마치면 Plan mode에 들어가고 나오는 절차를 설명하고, 큰 작업을 검증 가능한 단계로 쪼개며, subagent·Codex profile과 결합한 합의-실행 흐름을 구성할 수 있습니다.
