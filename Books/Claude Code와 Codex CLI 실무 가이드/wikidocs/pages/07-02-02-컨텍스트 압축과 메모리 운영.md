# 7.2.2 컨텍스트 압축과 메모리 운영

긴 작업을 이어 가다 보면 두 도구 모두에서 같은 증상을 만납니다. 응답이 점점 느려지고, 답변이 어딘가 산만해지며, 결국에는 모델이 '컨텍스트가 가득 찼다'는 신호를 흘립니다. 이 단원에서는 그 한계 앞에서 무엇이 일어나고, 어디까지를 자동에 맡기고 어디부터를 사람이 직접 통제해야 하는지를 정리합니다.

먼저 배경이 되는 개념 하나를 짚고 갑니다. **컨텍스트 윈도**는 모델이 한 번에 들여다볼 수 있는 토큰의 상한이며, 대화가 길어질수록 시스템 프롬프트, CLAUDE.md, 누적된 메시지, 도구 결과가 모두 이 안에 쌓입니다. 7.2.1에서 다룬 git 작업도, 5장에서 등록한 MCP 도구 응답도 결국은 이 윈도를 갉아먹습니다.

Claude Code는 윈도가 임계치에 가까워지면 **자동 압축**을 수행합니다. 자동 압축은 누적된 대화를 모델 스스로가 요약해 짧은 문단으로 갈아 끼우는 동작입니다. 이때 트리거 직전에 호출되는 것이 **PreCompact hook**입니다. 4.1.1에서 본 hook 6종 중 하나이며, 압축이 시작되기 전 마지막으로 상태를 보존하거나, 자동 압축을 의도적으로 차단하고 싶을 때 사용합니다.

PreCompact의 전형적인 쓰임은 두 가지입니다. 하나는 압축 직전의 핵심 결정을 외부 파일로 떨궈 두는 것입니다. 다른 하나는 압축 자체를 막고 사용자가 직접 `/compact` 같은 명령으로 통제하게 두는 것입니다. 자동 압축에 맡기면 모델이 중요하다고 판단한 부분만 남기므로, 의사 결정 근거나 미묘한 제약 조건이 사라지는 일이 종종 있습니다.

```json
{
  "hooks": {
    "PreCompact": [
      {
        "hooks": [
          { "type": "command",
            "command": "bash .claude/hooks/save-handoff.sh" }
        ]
      }
    ]
  }
}
```

위 설정은 압축이 일어나기 직전에 `save-handoff.sh`를 호출합니다. 스크립트는 현재 브랜치, 마지막 결정, 다음에 할 일 한 줄을 `.claude/handoff/{날짜}.md`에 남기는 역할만 합니다. 압축이 끝나면 그 파일은 다음 세션에서 사람이 직접 열어 읽거나, SessionStart hook(4.1.5)에서 자동으로 첫 메시지에 끼워 넣을 수 있습니다.

Claude Code의 또 다른 장치가 **Auto Memory**입니다. Auto Memory는 사용자가 반복적으로 알려 주는 사실, 즉 환경 특성과 선호를 모델이 스스로 정리해 둔 메모입니다. 저장 위치는 `~/.claude/projects/{프로젝트 슬러그}/memory/` 아래의 `MEMORY.md`와 개별 `*.md` 파일입니다. MEMORY.md는 색인 역할을 하고, 개별 파일이 실제 본문을 담습니다.

Auto Memory는 컨텍스트와 별개의 저장소에 있다가, 세션이 시작될 때 모델에게 짧게 요약되어 들어옵니다. 따라서 같은 사실을 매번 CLAUDE.md에 적어 둘 필요가 줄어듭니다. 다만 무엇을 저장하고 무엇을 저장하지 않을지의 경계가 중요합니다.

저장해도 좋은 항목은 다음과 같이 좁힙니다. 운영체제와 셸, 자주 사용하는 경로 관례, 팀에서 지키기로 한 commit 규칙, 사용자가 명시적으로 '앞으로 항상 이렇게 하라'고 말한 선호입니다. 저장하면 곤란한 항목은 단발성 작업 결과, 토큰과 패스워드 같은 비밀, 한 번의 실험을 위한 임시 결정, 그리고 시간이 지나면 사실이 바뀔 수 있는 외부 시스템 상태입니다. 후자는 다음 세션에 거짓을 진실처럼 주입할 위험이 있습니다.

Codex CLI에는 자동 압축이라는 동등한 기능이 따로 없습니다. 대신 컨텍스트 한계에 부딪히지 않도록 **세션 분리**를 적극 활용합니다. Codex는 본래 작업 단위 실행에 가까운 구조이므로, 큰 변경 하나를 한 세션에서 끝까지 끌고 가지 않고, 정찰·계획·구현·검증을 별개의 호출로 나누어 각자 짧은 컨텍스트를 갖게 두는 편이 자연스럽습니다.

세션을 분리할 때 가장 큰 어려움은 다음 호출이 이전 호출의 결정을 모른다는 점입니다. 그래서 **중간 결과를 외부 파일로 떨궈 두는** 습관이 중요해집니다. 정찰 세션은 `notes/recon.md`에 발견 사항만 남기고 종료하고, 구현 세션은 그 파일을 첫 입력으로 받아 시작합니다. AGENTS.md의 Build·Test 섹션을 짧게 유지하면, 다음 세션이 똑같은 명령을 다시 찾는 시간을 줄일 수 있습니다.

긴 작업의 마지막에는 **핸드오프 노트**를 남깁니다. 핸드오프 노트는 다음 세션이 이어받을 수 있도록 현재 상태, 미해결 결정, 다음 단계 한 줄을 묶은 짧은 문서입니다. 분량은 길게 잡지 않고, 한 화면 안에 들어오는 정도로 둡니다.

```
[handoff 2026-05-28]
- 작업: 결제 모듈 리팩터링 2단계
- 완료: PaymentService 분리, 단위 테스트 통과
- 미해결: 환불 정책 결정 대기 (이슈 #842)
- 다음: 결정 확정 후 RefundService.ts 구현
```

이 노트가 있으면 다음 세션의 첫 메시지는 'handoff 파일을 읽고 미해결부터 정리해 달라'로 충분합니다. 두 도구 모두에서 같은 형식을 쓰면, 같은 레포를 어떤 도구로 열어도 사람이 인지 비용 없이 작업을 이어 갑니다.

다음 단계 흐름을 정리하면 아래와 같습니다.

```
긴 작업 진행
   ↓
임계치 근접 (Claude는 자동 압축 트리거)
   ↓
PreCompact hook → 핵심 결정을 handoff 파일로 보존
   ↓
압축 후 또는 세션 종료
   ↓
다음 세션은 SessionStart에서 handoff 파일 로드
```

정리하면, 컨텍스트는 무한이 아니므로 도구에 자동 압축을 맡기되 잃기 싫은 결정을 PreCompact와 handoff 파일로 명시적으로 보존해야 하고, Auto Memory는 반복되는 사실에만 한정하며, Codex CLI에서는 세션 분리와 외부 파일 교환이 그 역할을 대신합니다.

다음 단원인 8.1.1에서는 길어진 컨텍스트와 잦은 호출이 비용으로 환산되는 지점을 살피고, 작업 성격에 맞춰 모델을 고르는 기준을 다룹니다.

이 단원을 마치면 자동 압축의 동작과 PreCompact의 역할을 설명하고, Auto Memory에 무엇을 남기지 말아야 하는지를 구분하며, 두 도구에서 긴 작업을 세션으로 끊어 가는 핸드오프 절차를 설계할 수 있습니다.
