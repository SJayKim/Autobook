# 3.1.3 주입 예산과 compaction memory flush

3.1.2에서 OpenClaw의 기억이 두 계층의 파일로 나뉘어 있음을 보았습니다. 그런데 파일에 기억을 쌓는 일과 그 기억을 모델에게 읽히는 일은 별개의 문제입니다. 디스크는 사실상 무한하지만 모델이 읽는 공간은 유한하기 때문입니다. 이 단원은 OpenClaw가 그 유한한 공간을 어떻게 배분하는지, 그리고 공간이 가득 찼을 때 기억이 증발하지 않도록 어떤 안전장치를 두는지를 다룹니다.

먼저 배경이 되는 제약부터 짚습니다. 언어 모델은 한 번 호출될 때 읽을 수 있는 입력의 총량이 정해져 있습니다. 이 한계를 **컨텍스트 윈도**(context window)라고 부릅니다. 시스템 프롬프트, 워크스페이스 파일, 지금까지의 대화 기록, 도구 실행 결과까지 모델이 판단에 쓸 모든 재료가 이 창 하나에 들어가야 합니다. 창을 넘어간 내용은 모델에게 존재하지 않는 것과 같습니다. 그래서 상주형 에이전트 설계에서 컨텍스트 윈도는 아껴 써야 할 예산으로 취급됩니다.

이 예산을 위협하는 첫 요인은 기억 파일 자체입니다. OpenClaw는 세션을 시작할 때 SOUL.md, AGENTS.md, MEMORY.md 같은 워크스페이스 파일을 프롬프트에 주입하는데, 이 초기 주입을 부트스트랩이라고 부릅니다. 문제는 MEMORY.md 같은 파일이 운영 기간이 길어질수록 계속 자란다는 데 있습니다. 아무 제한이 없다면 세션을 시작하자마자 기억 파일이 컨텍스트의 큰 몫을 차지하고, 정작 대화와 작업에 쓸 공간이 줄어듭니다. 기억이 많아질수록 일을 못 하게 되는 역설이 생기는 셈입니다.

OpenClaw는 이를 **주입 예산**으로 막습니다. 파일 하나가 부트스트랩에서 차지할 수 있는 최대 글자 수를 agents.defaults.bootstrapMaxChars 설정이 정하며, 기본값은 파일당 20,000자입니다. 파일이 한도를 넘으면 프롬프트에는 한도까지만 들어갑니다. 파일별 상한과 별도로 부트스트랩 전체 합계에 대한 상한(bootstrapTotalMaxChars)도 있습니다. 어떤 파일이 아무리 비대해져도 기억이 컨텍스트를 잠식하지 못하도록 천장을 친 것입니다.

여기서 주의할 점 하나가 이 설계의 성격을 보여 줍니다. 잘리는 것은 프롬프트에 주입되는 사본뿐이고, 디스크의 원본 파일은 한 글자도 건드리지 않습니다. 예산 초과는 "이번 세션에서 덜 보여 준다"는 뜻이지 "기억을 지운다"는 뜻이 아닙니다. 디스크가 기억의 정본이고 컨텍스트는 그 일부를 비추는 화면일 뿐이라는 관점입니다. 넘치면 화면 쪽만 줄입니다. 잘려 나간 부분도 원본에는 그대로 있으므로, 필요할 때 검색 도구로 다시 꺼내 읽을 수 있습니다.

주입 예산이 세션의 입구를 지킨다면, 세션이 진행되는 동안에는 다른 압력이 쌓입니다. 대화 기록입니다. 주고받은 메시지와 도구 호출 결과가 턴마다 컨텍스트에 누적됩니다. 세션을 리셋하지 않는 정책(none)으로 오래 대화할수록 기록은 컨텍스트 윈도의 한계를 향해 자랍니다. 이때 동원되는 것이 **compaction**입니다. compaction은 오래된 교환을 짧은 요약으로 접어 컨텍스트 공간을 되찾는 대화 요약 절차입니다. 리셋 없는 세션의 성장은 전적으로 compaction이 관리합니다.

그런데 요약은 본질적으로 손실 압축입니다. 수십 턴의 대화를 몇 문단으로 줄이면 파일 경로, 결정의 이유, 사용자가 지나가듯 말한 세부 사항이 요약문에서 빠질 수 있습니다. 요약에서 빠진 정보는 컨텍스트에서 사라지고, 어디에도 적히지 않았다면 영영 사라집니다. 열심히 일한 세션일수록 잃을 것이 많은 순간이 compaction 직전입니다.

이 손실을 막는 안전장치가 **memory flush**입니다. compaction이 대화를 요약하기 직전, OpenClaw는 에이전트에게 조용히 한 턴을 더 돌립니다. 이 턴에서 에이전트는 "지금 대화에서 중요한 컨텍스트를 메모리 파일에 적어 두라"는 상기를 받고, 보존할 가치가 있는 내용을 기억 파일에 기록합니다. 이 턴은 사용자에게 보이지 않으므로 **silent turn**이라고 부릅니다. 사용자 눈에는 대화가 그냥 이어질 뿐이지만, 그 사이 중요한 정보는 요약으로 접히기 전에 디스크로 대피합니다. 이 동작은 agents.defaults.compaction.memoryFlush.enabled 설정으로 제어합니다.

흐름을 그림으로 정리하면 다음과 같습니다.

```
대화 진행 ──▶ 기록 누적 ──▶ 컨텍스트 임계 접근
                                │
                                ▼
                  silent turn (memory flush)
             "중요한 컨텍스트를 기억 파일에 기록"
                                │  디스크에 저장
                                ▼
             compaction: 오래된 교환을 요약으로 압축
                                │
                                ▼
                  가벼워진 컨텍스트로 대화 계속
```

flush로 디스크에 적힌 내용은 두 경로로 살아남습니다. MEMORY.md에 적혔다면 다음 세션 부트스트랩 때 다시 주입되고, 일일 노트에 적혔다면 검색 도구로 다시 찾을 수 있습니다. 요약 과정에서 컨텍스트의 정보가 증발해도, 디스크에 쓴 것만 지속된다는 원칙 위에서 복구 경로가 확보되는 것입니다.

한 걸음 물러나 보면 이 단원의 장치들은 컨텍스트 윈도 예산 관리라는 하나의 개념으로 묶입니다. 세션 입구에서는 주입 예산이 기억 파일의 반입량을 제한하고, 세션 도중에는 compaction이 대화 기록을 압축하며, 압축의 손실은 memory flush가 디스크로 대피시켜 메웁니다. 디스크는 무한하고 컨텍스트는 유한하니, 유한한 쪽만 깎고 무한한 쪽은 온전히 보존한다는 일관된 태도가 세 장치를 관통합니다.

정리하면, 컨텍스트 윈도는 모델이 한 번에 읽을 수 있는 입력의 한계이며, 상주형 에이전트에게는 배분해야 할 예산입니다. OpenClaw는 부트스트랩 주입 예산(bootstrapMaxChars, 파일당 기본 20,000자)으로 기억 파일의 컨텍스트 잠식을 막습니다. 이때 잘라 내는 것은 주입 사본뿐이고 디스크 원본은 보존합니다. 대화 기록의 성장은 compaction 요약으로 관리하되, 요약의 손실로 정보가 증발하지 않도록 직전에 silent turn을 돌려 중요한 내용을 기억 파일로 대피시키는 memory flush를 둡니다.

3.2.1에서는 이렇게 관리되는 대화 문맥 위에서, 시키지 않아도 에이전트가 주기적으로 움직이게 만드는 장치인 Heartbeat를 다룹니다.

이 단원을 마치면 부트스트랩 주입 예산이 기억의 컨텍스트 잠식을 막는 방식과, compaction 직전 silent turn으로 기억을 저장시키는 memory flush를 설명할 수 있습니다.
