# 4.2.2 4만 노출 인스턴스: 기본값과 배포 관행

2026년 2월 9일, 한 보안 조사 기관이 공개 인터넷 전체를 스캔한 결과를 발표했습니다. 외부에서 직접 접근할 수 있는 OpenClaw 게이트웨이가 40,214개 발견되었고, 이는 28,663개의 고유 IP에 걸쳐 있었습니다. 그중 12,812개는 외부인이 원격 명령 실행까지 이어 갈 수 있는 상태였습니다. 한 사람의 머신에 상주하며 그 사람만 상대하도록 설계된 프로그램이, 아무나 접속을 시도할 수 있는 자리에 수만 개 규모로 나와 있었던 것입니다.

4.1.2에서 다뤘듯 게이트웨이는 기본값으로 127.0.0.1:18789에서만 연결을 받습니다. 이 기본값이 유지되는 한 외부 노출은 일어나지 않습니다. 그런데도 수만 개가 노출되었습니다. 이 간극을 만든 것은 코드의 결함이 아니라 배포 현장의 관행이었습니다. 이 단원은 그 경위를 따라갑니다.

먼저 주소의 뜻부터 짚습니다. 서버 프로그램이 연결을 받으려면, 어느 네트워크 주소로 들어오는 연결을 받을지 운영체제에 등록해야 합니다. 이 등록을 **바인딩**(binding)이라고 부릅니다. 127.0.0.1은 **루프백**(loopback)이라 부르는 특별한 주소로, 같은 머신 안에서만 통합니다. 다른 컴퓨터에서 이 주소로 패킷을 보낼 방법 자체가 없으므로, 루프백에 묶인 서버는 바깥에서 보면 존재하지 않는 것과 같습니다.

반대편 극단이 **0.0.0.0 바인딩**입니다. 0.0.0.0은 특정 주소가 아니라, 이 머신이 가진 모든 네트워크 인터페이스에서 연결을 받겠다는 선언입니다. 머신에 공인 IP가 있다면 그 포트는 인터넷 전체를 향해 열립니다. 노출된 인스턴스들은 같은 포트를 0.0.0.0에 묶은 채, 인증까지 설정하지 않은 상태로 발견되었습니다.

사용자들이 기본값을 깬 동기는 편의였습니다. 상주형 에이전트는 24시간 켜 둘 머신이 필요하므로, 많은 사용자가 노트북 대신 VPS라 부르는 임대 원격 서버에 설치합니다. 그런데 루프백을 유지하면 자기 컴퓨터의 브라우저로는 서버의 대시보드가 열리지 않습니다. 검색하면 바인딩을 0.0.0.0으로 바꾸라는 답이 나오고, 바꾸면 즉시 열립니다. 인증 토큰 설정은 나중으로 미뤄지고, 그 나중은 대개 오지 않습니다.

컨테이너 격리 도구인 Docker도 같은 방향으로 사용자를 밉니다. 컨테이너 안의 루프백은 컨테이너 내부에서만 유효해서, 호스트에서 접근하려면 서버를 0.0.0.0에 바인딩하고 포트를 밖으로 매핑하는 관례가 오래전부터 있었습니다. 개인 개발 머신에서는 무해한 이 관례를 공인 IP가 있는 서버에서 반복하면, 포트 매핑이 곧 인터넷 공개가 됩니다. 노출까지의 경로를 이어 보면 다음과 같습니다.

```
기본값: 127.0.0.1:18789 바인딩  →  외부에서 도달 불가
   │  VPS/Docker에 설치, 원격에서 대시보드가 안 열림
   ▼
바인딩을 0.0.0.0:18789로 변경, 인증 설정은 생략
   │  "일단 되게 만들기"에서 작업이 멈춤
   ▼
공인 IP의 18789 포트가 인터넷 전체에 개방
   │  전역 포트 스캔에 그대로 포착
   ▼
셸 권한과 API 키를 쥔 게이트웨이가 무방비 노출
```

이 노출이 위험한 이유는 게이트웨이가 쥔 것들에 있습니다. 게이트웨이는 셸 명령을 실행할 수 있고, 모델 API 키와 메신저 세션을 보관합니다. 인증 없는 게이트웨이에 접속한 외부인은 이 전부를 넘겨받습니다. 이런 상태를 **원격 코드 실행**(RCE, Remote Code Execution) 가능 상태라고 부릅니다. 공격자가 네트워크 너머에서 대상 머신의 임의 명령을 실행할 수 있다는 뜻이며, 앞서 나온 12,812개가 여기에 해당했습니다.

수치는 정직하게 다룰 필요가 있습니다. 40,214는 한 기관이 특정 날짜에 특정 방법으로 센 결과입니다. 비슷한 시기에 다른 보안 조사 기관들이 내놓은 집계는 3만여 개라는 보수적 수치부터 13만 개, 22만 개까지 벌어집니다. 스캔한 시점이 서로 다르고, 무엇을 OpenClaw 인스턴스로 인정할지도 기관마다 다르기 때문입니다. 포트 응답만 세는 방식과 서비스 지문까지 확인하는 방식이 갈리고, 도달만 되면 노출로 치는 정의와 인증 없는 접근까지 요구하는 정의가 섞여 있습니다. 어느 숫자가 정답인지 다투는 일은 생산적이지 않습니다. 어떤 기준으로 세어도 수만 단위였다는 규모, 그리고 전부가 같은 관행에서 나왔다는 사실이 요점입니다.

교훈은 여기서 나옵니다. 뚫린 코드가 없었으므로 이 사태는 취약점 사고가 아니었고, 패치로 고칠 수도 없었습니다. 안전한 기본값과 배포 편의가 충돌했고, 편의가 이겼습니다. 기본값은 개발자가 정하지만, 그 기본값을 유지할지는 사용자가 정합니다. 기본값이 사용 목적과 부딪히는 순간, 사용자는 가장 빨리 되는 길을 택합니다. 따라서 설계는 안전한 기본값을 고르는 데서 끝나지 않습니다. 원격 접근이 필요한 사용자에게는 포트 개방 대신 사설망 경유 같은 안전한 길을 기본 동선으로 안내해야 합니다. 외부 바인딩에 인증이 빠진 위험한 조합에는 경고와 거부 같은 마찰을 심어야 합니다. 기본값이 깨지는 배포 경로까지 감당할 때 설계가 완성됩니다.

정리하면, 2026년 2월 9일의 스캔에서 인터넷에 직접 노출된 OpenClaw 인스턴스 40,214개가 발견되었고, 그중 12,812개는 원격 코드 실행이 가능한 상태였습니다. 기본값은 루프백이라 안전했습니다. 그러나 빠른 시작을 원한 사용자들이 VPS와 Docker에서 바인딩을 0.0.0.0:18789로 바꾸고 인증을 비워 두는 관행이 노출을 만들었습니다. 측정 주체와 시점에 따라 수치는 3만에서 22만까지 갈리지만, 수만 단위라는 규모와 원인이 배포 관행이라는 사실은 변하지 않습니다. 안전한 기본값만으로는 부족하며, 그 기본값이 깨지는 배포 경로까지 설계 범위에 넣어야 한다는 것이 이 사태의 교훈입니다.

4.2.3에서는 공격이 들어온 또 다른 길인 공급 경로, 곧 악성 스킬과 연쇄 취약점이 드러낸 공급망 사고를 다룹니다.

이 단원을 마치면 기본 바인딩이 루프백인데도 수만 인스턴스가 인터넷에 노출된 경위를 배포 관행에서 짚어 내고, 안전한 기본값과 배포 편의의 충돌이라는 교훈을 설명할 수 있습니다.
