# 5.1.1 여섯 가지 설계 원칙

지금까지 이 책은 OpenClaw를 뇌와 근육의 분리에서 출발해 게이트웨이, 세션, 메모리, 스킬, 보안으로 차례차례 뜯어 왔습니다. 이제 시선을 Hermes Agent로 옮겨, 같은 순서로 이 시스템을 해부합니다. 1.2.2에서 Hermes가 OpenClaw의 fork가 아니라 Python으로 처음부터 새로 쓴 독자 구현이라는 사실은 이미 확인했습니다. 그렇다면 백지에서 시작한 이 시스템은 무엇을 나침반 삼아 지어졌을까요. 그 나침반이 이번 5부의 출발점인 여섯 가지 설계 원칙입니다.

원칙부터 살펴보는 데는 이유가 있습니다. Hermes의 공식 문서는 아키텍처를 설명하기에 앞서 설계 원칙 여섯 개를 표로 먼저 내겁니다. 부품 하나하나가 왜 그 모양인지 따지다 보면 결국 이 여섯 원칙으로 되돌아오기 때문입니다. 원칙을 먼저 손에 쥐면, 뒤에 이어질 단일 코어와 프롬프트 조립과 프로필 격리가 제각각의 기교가 아니라 하나의 방침에서 흘러나온 결과로 읽힙니다. 그래서 이번 단원은 여섯 원칙을 하나씩 짧게 설명하고, 각 원칙이 뒤 단원의 어떤 구현으로 이어지는지를 미리 짚어 두는 데 집중합니다.

첫 번째는 **Prompt Stability**, 곧 프롬프트 안정성입니다. 이것은 시스템 프롬프트가 대화 도중에는 바뀌지 않는다는 원칙입니다. 시스템 프롬프트란 매 요청마다 모델 앞에 붙어 에이전트의 정체성과 도구 사용법을 알려 주는 고정 지시문을 말합니다. Hermes는 사용자가 모델을 바꾸는 것처럼 명시적으로 시킨 경우를 빼면 이 지시문을 한 대화 안에서 그대로 유지합니다. 이렇게 하는 까닭은 뒤에서 자세히 볼 프롬프트 캐시와 맞닿아 있습니다. 앞부분이 그대로여야 다시 계산하지 않고 재사용되는 캐시의 특성상, 프롬프트를 흔들지 않는 것이 곧 비용 절감으로 이어지기 때문입니다.

두 번째와 세 번째는 짝을 이루는 **Observable Execution**과 **Interruptible**입니다. 앞은 관찰 가능한 실행, 뒤는 중단 가능성입니다. 관찰 가능한 실행이란 에이전트가 부르는 모든 도구 호출이 콜백을 통해 밖으로 드러난다는 뜻입니다. 콜백은 실행 도중에 특정 시점마다 바깥 코드에 신호를 보내는 통로를 가리킵니다. 무엇이 실행되는지가 이 통로로 그때그때 보이므로, 실행이 블랙박스에 갇히지 않습니다. 중단 가능성은 그렇게 드러난 작업을 언제든 멈출 수 있다는 원칙입니다. 진행 중인 API 호출이든 도구 실행이든 사용자가 취소할 수 있어야 한다는 것입니다. 보이게 만드는 원칙과 멈출 수 있게 만드는 원칙이 짝을 이루어, 자율적으로 도는 에이전트에 사람이 개입할 여지를 남깁니다.

네 번째는 **Platform-Agnostic Core**, 플랫폼 중립 코어입니다. 에이전트의 판단 루프를 담은 핵심부는 자신이 어느 인터페이스 위에서 도는지 몰라야 한다는 원칙입니다. 명령줄에서 부르든, 상주 데몬으로 돌든, 편집기가 띄운 프로세스로 돌든 코어의 동작은 같아야 하고, 플랫폼마다 다른 사정은 코어가 아니라 그 바깥의 진입점이 떠안습니다. 코어가 플랫폼을 모르게 두면, 인터페이스마다 동작이 미묘하게 어긋나는 버그가 애초에 생기지 않습니다.

다섯 번째는 **Loose Coupling**, 느슨한 결합입니다. 반드시 있어야 하는 부분과 골라 쓰는 부분을 단단히 묶지 않는다는 원칙입니다. 플러그인이나 메모리 확장처럼 선택적인 서브시스템은 코어에 못 박히는 대신, 필요할 때 스스로 등록되고 조건이 맞을 때만 켜지는 방식으로 붙습니다. 부품 하나를 더하거나 빼도 나머지가 흔들리지 않게 하려는 조치입니다.

여섯 번째는 **Profile Isolation**, 프로필 격리입니다. 프로필은 하나의 머신 안에서 서로 섞이면 안 되는 실행 환경을 가르는 이름표입니다. `-p` 옵션에 이름을 붙이면 그 프로필에 딸린 홈 디렉토리, 설정, 기억, 세션, 그리고 게이트웨이 프로세스 번호까지 통째로 분리됩니다. 덕분에 업무용 에이전트와 개인용 에이전트를 같은 컴퓨터에서 서로 침범하지 않게 나란히 돌릴 수 있습니다. 이 원칙이 뒤에서 볼 업무와 개인의 완전 분리로 이어집니다.

여섯 원칙과 각각이 규정하는 결과를 한눈에 정리하면 다음과 같습니다.

| 원칙 | 한 줄 뜻 | 규정하는 구현 |
|---|---|---|
| Prompt Stability | 대화 중 시스템 프롬프트 불변 | 프롬프트 캐시 적중률 확보 |
| Observable Execution | 모든 도구 호출을 콜백으로 노출 | 실행 가시화 계층 |
| Interruptible | API 호출과 도구 실행을 취소 가능 | 중단 신호 처리 |
| Platform-Agnostic Core | 코어는 인터페이스를 모른다 | 단일 코어와 얇은 진입점 |
| Loose Coupling | 선택 부품은 느슨하게 붙는다 | 자기등록과 조건부 게이팅 |
| Profile Isolation | 프로필별 환경 완전 분리 | 업무와 개인의 분리 |

이 표가 보여 주는 것이 이번 단원의 핵심입니다. 여섯 원칙은 감상이나 구호가 아니라, 뒤이을 아키텍처의 모양을 미리 정해 두는 설계도입니다. 플랫폼 중립 코어라는 원칙이 있기에 코어가 하나로 모이고, 프롬프트 안정성이라는 원칙이 있기에 프롬프트를 안 변하는 부분과 변하는 부분으로 나눠 쌓으며, 느슨한 결합이라는 원칙이 있기에 도구가 중앙 목록 없이 스스로 등록됩니다. 원칙이 먼저 서고 구현이 그 뒤를 따르는 순서를 붙잡으면, 5부의 나머지 단원이 왜 그런 구조를 택했는지가 매번 이 여섯 줄로 되돌아와 설명됩니다.

정리하면, Hermes Agent는 프롬프트 안정성, 관찰 가능한 실행과 중단 가능성, 플랫폼 중립 코어, 느슨한 결합, 프로필 격리라는 여섯 원칙 위에 지어졌습니다. 프롬프트 안정성은 캐시로, 프로필 격리는 업무와 개인의 분리로 이어지는 것처럼, 각 원칙은 뒤 단원의 구체적 구현을 미리 규정합니다. 그래서 원칙을 손에 쥐고 아키텍처를 보면, 흩어진 부품이 하나의 방침에서 흘러나온 결과로 읽힙니다.

5.1.2에서는 여섯 원칙 가운데 플랫폼 중립 코어가 실제로 어떻게 구현되는지, 하나의 AIAgent 코어가 어떻게 플랫폼 차이를 진입점 바깥으로 몰아내는지를 살펴봅니다.

이 단원을 마치면 Prompt Stability부터 Profile Isolation까지 Hermes의 여섯 설계 원칙이 각각 무엇을 뜻하고 어떤 구현으로 이어지는지 설명할 수 있습니다.
