# 3.1.1 워크스페이스: 정체성을 파일로 만들다

Phase 2에서는 게이트웨이와 에이전트 루프, 그리고 2.3.3의 세션까지 OpenClaw의 뼈대를 봤습니다. 그런데 세션은 어디까지나 대화 단위의 상태입니다. 세션이 리셋된 뒤에도 남아야 하는 것이 있습니다. 이 에이전트가 어떤 존재이고, 누구와 일하며, 무엇을 지켜야 하는지입니다. Phase 3은 이 지속 계층, 곧 기억을 다룹니다. 출발점은 기억이 담기는 그릇인 워크스페이스입니다.

**워크스페이스**(workspace)는 에이전트의 정체성과 기억을 담는 디렉토리입니다. 기본 경로는 ~/.openclaw/workspace이고, 안에 있는 것은 전부 사람이 읽을 수 있는 마크다운 파일입니다. 환경변수 OPENCLAW_PROFILE을 설정하면 경로가 ~/.openclaw/workspace-<profile>로 바뀝니다. 업무용과 개인용처럼 성격이 다른 에이전트를 한 머신에서 프로필 단위로 분리하는 장치입니다. 경로 자체를 옮기고 싶다면 OPENCLAW_WORKSPACE_DIR 환경변수나 agents.defaults.workspace 설정으로 지정합니다.

이 디렉토리에서 정체성을 맡는 파일은 넷입니다. SOUL.md는 에이전트의 페르소나와 말투, 넘지 말아야 할 경계를 정의합니다. IDENTITY.md는 에이전트가 스스로를 누구로 여기는지와 사용자를 부르는 호칭을 담습니다. AGENTS.md는 운영 지침입니다. 지켜야 할 규칙, 메시지 라우팅 방식, 보안 정책, 작업 범위 같은 표준 운영 절차가 들어갑니다. USER.md는 함께 일하는 사람에 대한 컨텍스트입니다. 사용자의 상황과 선호를 적어 두면, 에이전트가 세션마다 상대를 이미 아는 상태로 시작합니다.

워크스페이스에는 보조 파일도 여럿 있습니다. HEARTBEAT.md는 일정 주기로 도는 배경 점검 턴인 heartbeat가 읽는 체크리스트입니다. 반복해서 읽히는 파일이므로 토큰 소모를 막으려면 짧게 유지해야 합니다. STARTUP.md는 게이트웨이가 재시작될 때 자동으로 실행하는 체크리스트로, 두지 않아도 됩니다. BOOTSTRAP.md와 ONBOARDING.md는 새 워크스페이스에서 한 번만 치르는 정체성 확립 절차입니다. TOOLS.md는 로컬 도구를 다루는 관례를 적는 노트인데, 도구 사용을 안내할 뿐 가용성을 실제로 통제하지는 않습니다. 그 밖에 장기 기억 파일 MEMORY.md, 일일 노트 디렉토리 memory/, 스킬 디렉토리 skills/ 등이 함께 삽니다. MEMORY.md와 memory/는 3.1.2에서 자세히 봅니다.

주요 파일의 역할과 읽히는 시점을 정리하면 다음과 같습니다.

| 파일 | 담는 내용 | 읽히는 시점 |
|---|---|---|
| SOUL.md | 페르소나, 말투, 경계 | 매 세션 시작, 가장 먼저 |
| IDENTITY.md | 에이전트 정체성, 사용자 호칭 | 매 세션 시작 |
| AGENTS.md | 운영 규칙, 라우팅, 보안 정책 | 매 세션 시작 |
| USER.md | 사용자에 대한 컨텍스트 | 매 세션 시작 |
| MEMORY.md | 장기 기억(3.1.2에서 상세) | 세션 시작 |
| HEARTBEAT.md | heartbeat 체크리스트 | heartbeat 턴에만 |
| STARTUP.md | 재시작 루틴(옵션) | 게이트웨이 재시작 시 |
| BOOTSTRAP.md | 최초 1회 온보딩 | 신규 워크스페이스에서 1회 |

세션이 시작되면 OpenClaw는 이 파일들을 읽어 모델이 볼 컨텍스트에 넣습니다. 이렇게 파일 내용을 컨텍스트에 실어 보내는 일을 주입이라고 부릅니다. 워크스페이스 파일 가운데 가장 먼저 주입되는 것은 SOUL.md입니다. 성격이 모든 판단의 바탕이 되도록 첫 자리에 둔 셈입니다. 이어서 IDENTITY.md, AGENTS.md, USER.md, MEMORY.md가 세션 시작 시점에 로드됩니다. 반면 HEARTBEAT.md나 STARTUP.md처럼 특정 상황용 파일은 그 시점에만 읽힙니다. 주입에는 예산도 있습니다. 파일 하나가 컨텍스트에 들어가는 분량은 기본 20,000자가 상한입니다. 상한을 넘으면 디스크의 파일은 그대로 두고, 주입되는 사본만 잘라 넣습니다. 원본은 언제나 디스크 쪽이라는 뜻입니다.

이 설계 전체를 관통하는 원칙이 **no hidden state**입니다. 디스크에 쓴 것만 지속된다는 뜻입니다. 세션 사이에 살아남는 상태는 전부 워크스페이스의 파일이고, 파일로 적히지 않은 것은 세션과 함께 사라집니다. 벡터 데이터베이스처럼 사람이 들여다보기 어려운 저장소는 없습니다. 그래서 에이전트가 무엇을 기억하는지 궁금하면 텍스트 에디터로 파일을 열면 됩니다. 잘못 기억한 내용은 그 자리에서 고치면 되고, 워크스페이스를 git 저장소로 만들면 기억의 변경 이력을 추적하고 되돌릴 수 있습니다. 이식성도 따라옵니다. 정체성과 기억이 모델 바깥의 파일로 존재하므로, 모델 프로바이더를 바꿔도 에이전트는 같은 성격과 같은 기억으로 이어집니다.

마지막으로, 워크스페이스가 구성을 마쳤는지 가르는 판정 조건이 있습니다. SOUL.md, IDENTITY.md, USER.md가 처음 제공된 스타터 템플릿에서 달라졌거나, memory/ 폴더가 존재하면 구성된 것으로 간주합니다. 템플릿 그대로라면 아직 백지라는 뜻이고, 파일을 채웠거나 기억이 쌓이기 시작했다면 정체성을 갖춘 것입니다. BOOTSTRAP.md 같은 1회성 파일은 워크스페이스가 구성 완료로 보이면 삭제됩니다. 온보딩이 끝났으니 온보딩용 문서도 치우는 것입니다.

정리하면, 워크스페이스는 에이전트의 지속 상태를 전부 담는 마크다운 디렉토리이며, 기본 경로 ~/.openclaw/workspace를 프로필 단위로 분리할 수 있습니다. SOUL.md, IDENTITY.md, AGENTS.md, USER.md가 정체성을 맡아 매 세션 시작에 주입되고, 그중 SOUL.md가 첫 자리입니다. HEARTBEAT.md, STARTUP.md, BOOTSTRAP.md는 각자의 시점에만 읽히는 보조 파일입니다. 디스크에 쓴 것만 지속된다는 no hidden state 원칙 덕분에 기억을 열어 보고 고치고 git으로 관리할 수 있으며, 모델을 바꿔도 정체성이 이식됩니다. 구성 완료는 정체성 파일이 템플릿에서 달라졌는지와 memory/ 존재로 판정합니다.

3.1.2에서는 워크스페이스 안에서 기억을 전담하는 두 층, 장기 기억 MEMORY.md와 일일 노트 memory/의 분업을 살펴봅니다.

이 단원을 마치면 워크스페이스의 경로와 프로필 분리, 각 파일의 역할과 주입 시점, no hidden state 원칙, 구성 완료 판정 조건을 설명할 수 있습니다.
