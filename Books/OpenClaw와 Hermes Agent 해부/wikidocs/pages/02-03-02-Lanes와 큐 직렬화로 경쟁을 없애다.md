# 2.3.2 Lanes와 큐: 직렬화로 경쟁을 없애다

2.3.1에서 메시지 하나가 다섯 단계를 거쳐 행동과 응답으로 바뀌는 과정을 보았습니다. 그 설명에는 조용한 전제가 하나 깔려 있습니다. 메시지가 한 번에 하나씩 온다는 전제입니다. 실제 메신저 대화는 그렇게 얌전하지 않습니다. 사용자는 "어제 회의 메모를 정리해서 파일로 저장해줘"라고 보냅니다. 그리고 에이전트가 아직 일하는 중에 "아, 제목은 날짜로 붙여줘"를 이어서 보냅니다. 이 단원은 이렇게 겹쳐 도착한 메시지를 OpenClaw가 어떤 구조로 받아내는지, 왜 그 방식을 골랐는지를 다룹니다.

먼저 겹치면 무엇이 깨지는지부터 봅니다. 두 메시지가 각각 하나의 실행, 곧 run으로 바뀌어 같은 세션에서 동시에 돌아간다고 합시다. 두 run은 같은 대화 기록을 읽어 문맥을 조립하지만 서로의 존재를 모릅니다. 둘째 run은 첫째 run이 지금 무엇을 하는지 반영하지 못한 채 판단을 시작합니다. 도구 실행도 겹칩니다. 첫째 run이 메모 파일을 만들어 쓰는 도중에 둘째 run이 같은 파일을 다른 내용으로 덮어쓸 수 있습니다. 기록도 마찬가지입니다. 두 run이 대화 기록에 결과를 쓰는 순서는 그날의 실행 속도에 따라 달라지고, 기록에는 두 실행의 중간 결과가 뒤섞여 남습니다.

이처럼 여러 실행 흐름이 같은 자원을 동시에 건드려 결과가 타이밍에 따라 달라지는 상태를 **경쟁 상태**(race condition)라고 부릅니다. 경쟁 상태가 까다로운 이유는 늘 깨지는 것이 아니라 가끔 깨진다는 데 있습니다. 타이밍이 좋으면 멀쩡히 지나가고 나쁘면 어긋나므로, 재현하기도 디버깅하기도 어렵습니다. OpenClaw는 자신이 막으려는 대상을 tool/session race라고 부릅니다. 도구 실행이 겹치는 경쟁과 세션 기록 쓰기가 겹치는 경쟁, 방금 본 두 축을 그대로 짚은 이름입니다.

동시성 문제를 다루는 고전적 해법은 **락**(lock)입니다. 공유 자원에 자물쇠를 달아 자물쇠를 쥔 실행만 접근시키고 나머지는 기다리게 하는 방식입니다. 락은 동시 실행을 유지한 채 위험한 구간만 보호하므로 처리량에 유리하지만 대가가 있습니다. 자원이 여럿이면 락도 여럿이 됩니다. 두 실행이 서로가 쥔 락을 기다리며 함께 멈추는 교착도 생기고, 락을 쥔 쪽이 죽으면 나머지는 영원히 기다립니다. 무엇보다 자원을 만지는 모든 코드가 락 규율을 빠짐없이 지켜야 안전이 성립합니다.

다른 해법은 경쟁이 생기는 조건 자체를 없애는 것입니다. 경쟁은 같은 자원에 동시에 접근할 때만 생깁니다. 그렇다면 실행을 한 줄로 세워 앞이 끝난 뒤에 뒤가 시작하게 만들면, 보호할 구간 자체가 사라집니다. 실행 요청을 큐(queue)에 넣고 순서대로 하나씩 꺼내 처리하는 이 방식이 **큐 직렬화**(queue serialization)입니다. OpenClaw는 락으로 겨루는 길 대신 이 길을 골랐습니다.

구현 단위가 **lane**입니다. lane은 세션 키마다 하나씩 생기는 실행 대기줄입니다. 같은 세션으로 향하는 run은 모두 그 세션의 lane에 줄을 서고, 먼저 온 순서대로 한 번에 하나씩 실행됩니다. 세션이 다르면 lane도 다르므로 병렬로 돌아도 됩니다. 세션마다 문맥과 기록이 분리되어 있어 애초에 겨룰 자원이 없기 때문입니다. 여기에 더해 일부 실행은 선택적으로 **global lane**을 거칩니다. 세션 lane이 한 세션 안의 겹침을 없앤다면, global lane은 세션이 달라도 시스템 전체에서 하나씩만 돌아야 하는 실행을 한 줄로 세웁니다.

```
m1(세션 A) ─┐
m2(세션 A) ─┼─▶ lane A: [m1] → [m2]    같은 lane은 한 번에 run 하나
m3(세션 B) ─┴─▶ lane B: [m3]           lane A와는 병렬 실행

(일부 실행) ──▶ global lane             시스템 전체에서 한 줄
```

줄을 세운다고 뒤에 온 메시지를 버리는 것은 아닙니다. 겹쳐 도착한 메시지를 진행 중인 run과 어떻게 합류시킬지는 채널마다 큐 모드(steer, followup, collect, interrupt)로 고릅니다. 이름 그대로 진행 중인 작업의 방향을 틀거나, 끝난 뒤 이어서 실행하거나, 모아서 한 번에 처리하거나, 실행을 끊는 선택지입니다. 앞의 예에서 "제목은 날짜로 붙여줘"가 첫 작업에 반영되는 것도 이 층에서 정해집니다.

큐 직렬화가 1차 방어라면 대화 기록 쓰기에는 방어가 한 겹 더 있습니다. 큐는 큐를 거친 실행만 통제하기 때문입니다. 세션의 대화 기록 파일, 곧 transcript는 큐 밖에서도 손댈 수 있습니다. 다른 프로세스가 같은 파일을 열 수도 있고, 같은 프로세스 안에서 큐를 우회해 쓰는 코드가 있을 수도 있습니다. 그래서 OpenClaw는 transcript 쓰기를 파일 기반 **session write lock**으로 한 번 더 보호합니다. 잠금 상태가 메모리가 아니라 파일에 있으므로 프로세스가 달라도 같은 잠금이 보입니다. 큐를 우회한 인프로세스 쓰기든 다른 프로세스의 쓰기든 이 락 앞에서는 똑같이 걸립니다.

이 락에는 앞서 본 락의 함정에 대한 대비가 붙어 있습니다. 잠금 획득 대기에는 기본 60초 타임아웃이 걸려 있어, 그 안에 잠금을 얻지 못하면 실패로 처리하고 빠져나옵니다. 락을 쥔 쪽이 죽어도 나머지가 영원히 멈추지는 않는 것입니다. 한도는 환경변수 OPENCLAW_SESSION_WRITE_LOCK_ACQUIRE_TIMEOUT_MS로 조정합니다. 기본 동작은 non-reentrant, 즉 이미 잠금을 쥔 흐름이 같은 잠금을 겹쳐 잡는 것도 허용하지 않는 보수적 설정입니다.

여기서 설계의 역할 분담이 드러납니다. 실행 수준의 경쟁은 큐 직렬화로 아예 없애고, 락은 큐가 손쓸 수 없는 마지막 지점인 파일 쓰기 한 곳에만 남겼습니다. 락을 시스템 전반의 동시성 도구로 쓰면 교착과 경합 관리가 코드 전체의 짐이 되지만, 이렇게 좁히면 락이 지킬 대상이 파일 하나로 줄어듭니다.

정리하면, 같은 세션에 메시지가 연달아 오면 도구 실행과 기록 쓰기가 겹치는 tool/session race, 곧 경쟁 상태가 생길 수 있습니다. OpenClaw는 이 동시성 문제를 락 경합으로 풀지 않고, run을 세션 키별 lane에 세워 하나씩 실행하는 큐 직렬화로 경쟁의 조건 자체를 없앱니다. 세션 lane이 세션 안의 겹침을, 선택적 global lane이 시스템 전체의 겹침을 막습니다. 큐가 통제하지 못하는 기록 쓰기는 타임아웃이 달린 파일 기반 write lock이 지킵니다.

2.3.3에서는 lane의 열쇠였던 세션이 실제로 어떻게 정해지는지, 곧 메시지를 세션으로 나누는 라우팅 규칙과 세션의 저장 구조를 살펴봅니다.

이 단원을 마치면 세션 키별 lane과 global lane이 실행을 직렬화하고, 파일 기반 write lock이 기록 쓰기를 보호해 경쟁 상태를 없애는 설계를 설명할 수 있습니다.
