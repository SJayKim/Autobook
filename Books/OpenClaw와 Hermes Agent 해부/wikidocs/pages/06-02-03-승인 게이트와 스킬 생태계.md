# 6.2.3 승인 게이트와 스킬 생태계

6.2.1에서 에이전트가 스스로 스킬을 만든다는 이야기를 했습니다. 복잡한 작업을 성공적으로 끝내면 그 절차를 문서로 굳혀 다음에 재사용한다는 구조였습니다. 편리하지만 곧바로 의문이 따라옵니다. 에이전트가 만든 문서가 그 에이전트의 판단 근거가 되고, 파일과 셸을 다루는 에이전트가 그 지시를 읽고 움직인다면, 사람이 한 번도 보지 않은 스킬이 실행에 영향을 주는 셈입니다. 자기 자신을 고치는 시스템에는 사람이 끼어드는 지점이 필요합니다. 이 단원은 그 지점인 승인 게이트와, 남이 만든 스킬을 안전하게 들여오는 설치 생태계를 다룹니다.

먼저 배경 개념입니다. 자동화 시스템에 사람의 검토 단계를 의도적으로 끼워 넣는 방식을 **사람 개입**(human-in-the-loop)이라고 부릅니다. 판단은 기계가 하되, 그 결과가 실제 효력을 갖기 전에 사람이 한 번 확인하고 허락하는 구조입니다. 은행 송금에서 큰 금액에 확인 창이 한 번 더 뜨는 것과 같은 발상입니다. 자동 생성 스킬처럼 되돌리기 어려운 결과를 만들 수 있는 자리에 이 단계를 두면, 자동화의 속도를 살리면서 폭주는 막을 수 있습니다.

Hermes에서 이 단계를 켜는 스위치가 **write_approval**입니다. skills.write_approval 설정을 true로 두면, 에이전트가 만든 스킬이 곧바로 활성 스킬 폴더에 들어가지 않습니다. 대신 ~/.hermes/pending/skills/라는 대기 폴더에 먼저 놓입니다. 여기서 pending은 대기 중이라는 뜻이고, 이렇게 결과를 곧장 반영하지 않고 별도 공간에 세워 두는 방식을 **스테이징**(staging)이라고 합니다. 무대에 올리기 전 대기실에 세워 둔다는 그림입니다. 스테이징된 스킬은 만들어졌을 뿐 아직 아무 힘도 없습니다. 에이전트는 그것을 근거로 쓸 수 없습니다.

검토와 승인은 사람의 명령으로 이뤄집니다. /skills pending으로 대기 중인 스킬 목록을 확인하고, 내용을 읽어 본 뒤 /skills approve 명령에 스킬 식별자를 붙여 활성화합니다. 승인을 받은 스킬만 대기실을 나와 활성 폴더로 옮겨지고, 그때부터 에이전트의 판단 근거가 됩니다. 흐름을 그림으로 정리하면 다음과 같습니다.

```
에이전트가 스킬 생성
      │
      ▼  write_approval: true 이면
 ~/.hermes/pending/skills/ 에 스테이징 (아직 효력 없음)
      │
      ▼  사람이
 /skills pending 으로 목록 확인 → 내용 검토 → /skills approve [id]
      │
      ▼
 활성 스킬로 승격 (이제부터 에이전트가 사용)
```

핵심은 자가 생성물을 초안으로 다룬다는 태도입니다. 에이전트가 쓴 스킬은 완성품이 아니라 사람의 검토를 기다리는 초안이고, write_approval은 그 초안이 검토 없이 실행 흐름에 스며드는 일을 기본값 차원에서 막는 장치입니다.

여기까지가 내가 만든 스킬을 다루는 절반입니다. 나머지 절반은 남이 만든 스킬을 들여오는 문제입니다. 스킬이 쓸모를 가지려면 서로 호환되어야 합니다. 어떤 도구가 만든 스킬을 다른 도구가 그대로 읽을 수 있어야 생태계가 생깁니다. 이 호환의 바탕이 **agentskills.io**라는 오픈 표준입니다. 스킬을 어떤 형식으로 적을지를 공개적으로 약속해 둔 규격이며, Hermes의 스킬은 이 표준을 따르는 SKILL.md 문서 형식으로 작성됩니다. 표준을 공유하기에 OpenClaw 계열에서 만든 스킬과도 형식이 통합니다.

표준이 형식을 통일해 주니, 설치 통로도 한 곳에 묶이지 않고 여럿으로 열립니다. Hermes는 스킬 설치 소스를 일곱 종류 지원합니다. 공식 소스, Vercel이 운영하는 skills.sh, 사이트가 표준 위치에 스킬 목록을 공개하는 /.well-known/skills/index.json 엔드포인트, openai/skills나 anthropics/skills 같은 GitHub 저장소 직접 설치, hermes skills tap add로 등록하는 커스텀 탭, 스킬을 가리키는 직접 URL, 그리고 ClawHub나 LobeHub 같은 커뮤니티 소스입니다. hermes skills browse와 install, check, update 명령으로 이 소스들을 검색하고 설치하고 갱신합니다. 여기서 탭이란 설치처를 하나 추가 등록해 두는 개념으로, 정해진 목록 밖의 저장소를 내 설치 경로로 끌어들이는 통로입니다.

통로가 일곱 개나 열려 있으면 자연히 나쁜 물건이 흘러들 위험도 커집니다. 그래서 마지막 안전장치가 설치 시점의 보안 스캔입니다. 소스에서 받아 온 스킬은 신뢰 수준이 매겨지고, 내장 스킬부터 커뮤니티 스킬까지 등급에 따라 검사를 거칩니다. 이 검사의 성격을 보여 주는 대목이 하나 있습니다. 스캔이 어떤 스킬을 위험(dangerous)으로 판정하면, 그 판정은 강제 설치 옵션인 **--force**로도 뚫리지 않습니다. 보통 --force 같은 플래그는 경고를 무시하고 밀어붙이라는 뜻이지만, 위험 판정 앞에서는 그마저 통하지 않는다는 것입니다. 편의를 위한 우회로가 안전의 바닥까지는 뚫지 못하게 막아 둔 설계입니다. 6.2.1의 승인 게이트가 내가 만든 스킬을 검토 없이 못 쓰게 했다면, 이 스캔은 남이 만든 스킬을 위험한 채로 못 들이게 막습니다.

이 두 장치가 왜 이렇게 촘촘한지는 짐작할 만합니다. 남의 스킬을 이름 하나로 들여오는 레지스트리는 큰 생태계를 만드는 힘이지만, 같은 통로로 악성 스킬이 흘러든 사고가 실제로 있었습니다. 3.3.3과 4.2.3에서 다루는 사태가 그것입니다. Hermes의 스테이징과 우회 불가 스캔은 그 전례를 지켜본 뒤에 세운 기본값으로 보입니다.

정리하면, write_approval을 켜면 에이전트가 만든 스킬이 pending 폴더에 스테이징되고, /skills pending과 /skills approve로 사람이 검토해 승인한 것만 활성화됩니다. 남의 스킬은 agentskills.io 오픈 표준으로 형식이 호환되어 일곱 종의 소스에서 설치할 수 있고, 설치 시 보안 스캔이 신뢰 수준을 매기며, 위험 판정은 --force로도 우회되지 않는 안전 바닥이 됩니다. 자가 생성물은 초안으로, 외부 스킬은 검증 대상으로 다루는 두 방향의 게이트가 스킬 생태계를 떠받칩니다.

다음 6.3.1에서는 Hermes가 모델을 어떻게 운용하는지로 넘어가, API 방언과 대화 중 라이브 전환, 그리고 자사 모델을 두고도 권하지 않는다는 아이러니를 살펴봅니다.

이 단원을 마치면 write_approval과 pending 스테이징으로 이뤄진 자가 생성 스킬의 검토·승인 흐름, agentskills.io 표준과 일곱 종 설치 소스로 구성된 스킬 생태계, 그리고 --force로도 우회할 수 없는 위험 판정의 역할을 설명할 수 있습니다.
