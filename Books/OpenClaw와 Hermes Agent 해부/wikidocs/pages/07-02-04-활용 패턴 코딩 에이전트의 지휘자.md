# 7.2.4 활용 패턴: 코딩 에이전트의 지휘자

7.2.1에서 Hermes를 머신에 올려 게이트웨이까지 상주시켰습니다. 이제 그 24시간 도는 에이전트에게 실제로 무슨 일을 시키는지가 남았습니다. Part 1에서 OpenClaw의 정착한 활용 패턴을 짚었듯이(4.3.4), 이 단원은 Hermes 판으로 그 자리를 채웁니다. 새 부품을 배우는 자리가 아니라, 앞에서 뜯어본 부품이 실전에서 어떤 형태로 굳었는지를 번호로 엮는 종합 단원입니다. Phase 7의 마지막 단원이기도 합니다.

먼저 사람들이 Hermes로 실제 무엇을 하는지 그 분포부터 봅니다. 공식적으로 모인 사용자 사례를 갈래로 나누면 세 덩어리가 큽니다. 가장 많은 것이 **개발 워크플로**입니다. 코드 리뷰 자동화나 여러 에이전트를 엮은 파이프라인처럼, 코드를 만지는 작업이 여기 들어갑니다. 그다음이 이메일 요약이나 일정 관리, 저널링 같은 **개인 비서** 용도이고, 티켓 분류나 영업 지원 같은 **업무 운영**이 그 뒤를 잇습니다. 리서치나 콘텐츠 제작도 있지만 앞의 셋에 비하면 적습니다. 요컨대 Hermes는 개발자의 손에서 가장 많이 쓰이며, 그래서 대표 패턴도 개발 쪽에서 나옵니다.

그 대표 패턴이 **오케스트레이터**입니다. 오케스트레이터라는 말은 오케스트라의 지휘자에서 왔습니다. 지휘자는 악기를 직접 연주하지 않습니다. 대신 언제 누가 어떤 소리를 낼지 지시하고 전체를 하나로 묶습니다. 소프트웨어에서 오케스트레이터도 같은 뜻입니다. 자신이 모든 일을 직접 하는 대신, 실제 작업은 다른 실행 주체에게 맡기고 그 시작과 끝을 조율하는 조정자를 가리킵니다.

Hermes가 지휘자 역할을 맡는 정착한 형태는 이렇습니다. 사용자가 Telegram으로 "이 저장소의 로그인 버그를 고쳐라" 같은 태스크를 던지면, Hermes가 그 지시를 받아 Claude Code 같은 코딩 에이전트를 하나 띄웁니다. 코드를 실제로 읽고 고치고 테스트하는 손일은 그 코딩 에이전트가 하고, Hermes는 작업이 끝나기를 기다렸다가 결과를 다시 메신저로 회신합니다. 흐름으로 그리면 다음과 같습니다.

```
[Telegram: 태스크 지시]
        │
        ▼
   Hermes (지휘자) ── spawn ──▶ Claude Code (실제 코딩)
        ▲                              │
        └──────── 결과 회신 <──────────┘
        │
        ▼
[Telegram: 완료 보고]
```

여기서 Hermes가 코드를 직접 만지지 않는다는 점이 핵심입니다. Hermes는 메신저라는 입구와 코딩 에이전트라는 일꾼을 잇는 조정자로만 서고, 무거운 실행은 아래로 위임합니다. 사용자는 자기 자리에 없어도, 늘 쓰던 메신저로 한 줄 보내는 것만으로 코딩 작업을 지시하고 결과를 받는 셈입니다.

지휘자가 일을 제대로 시키려면 대상 프로젝트의 맥락을 알아야 합니다. Hermes는 이 맥락을 특정 파일에서 자동으로 흡수합니다. 프로젝트 루트에 **AGENTS.md**를 두면 그 내용이 시스템 프롬프트에 자동으로 주입됩니다. 이 파일에는 그 프로젝트의 규칙이나 코딩 컨벤션, 주의사항을 적어 둡니다. 에이전트의 페르소나와 말투를 고정하고 싶으면 홈 아래 SOUL.md에 적어 둡니다. 다른 도구에서 쓰던 프로젝트 규칙 파일인 .cursorrules가 이미 있으면 그것도 Hermes가 알아서 인식합니다. 사용자가 규칙을 Hermes 전용으로 새로 옮겨 적지 않아도, 프로젝트에 이미 놓인 맥락 파일을 그대로 읽어 들이는 것입니다.

무인 작업의 결과를 어디로 받을지도 정해 둘 수 있습니다. Cron이 매번 새 에이전트로 정기 작업을 돌린다는 점은 5.3.2에서 봤는데, 그 결과가 어느 채널로 배달될지는 **/sethome**으로 지정합니다. sethome은 현재 대화하는 채널을 그 프로필의 기본 배달지, 즉 집으로 등록하는 명령입니다. 예를 들어 특정 Telegram 대화에서 /sethome을 한 번 실행해 두면, 이후 cron이 만든 아침 브리핑이나 백업 보고가 모두 그 대화로 도착합니다.

지휘자가 여러 일을 한꺼번에 시켜야 할 때도 있습니다. 저장소 세 개를 동시에 점검하는 것처럼 서로 독립된 작업이라면, 하나씩 순서대로 처리할 이유가 없습니다. 이때 쓰는 것이 delegate_task로 여러 하위 작업을 병렬로 위임하는 방식입니다(5.3.3). 각 작업이 격리된 하위 에이전트에서 따로 돌기 때문에, 지휘자의 컨텍스트를 불리지 않으면서 여러 갈래를 동시에 진행합니다. 태스크가 셋이면 셋을 나란히 내보내고 각각의 결과만 거둬들이는 것입니다.

정리하면, Hermes의 정착한 활용 패턴은 개발 워크플로, 개인 비서, 업무 운영으로 갈리며 개발 쪽이 가장 많습니다. 대표는 메신저로 태스크를 받아 Claude Code 같은 코딩 에이전트를 spawn해 일을 시키고 결과를 회신하는 오케스트레이터 패턴입니다. 지휘자는 AGENTS.md와 SOUL.md, .cursorrules를 자동으로 읽어 프로젝트 맥락을 흡수하고, /sethome으로 cron 결과를 받을 채널을 정하며, delegate_task로 독립 작업을 병렬화합니다. 코드를 직접 만지는 대신 실행을 아래로 위임하고 조율하는 조정자로 선다는 것이 이 패턴의 뼈대입니다.

이것으로 Part 2 Hermes 해부를 마칩니다. 8.1.1에서는 Part 3으로 넘어가 두 시스템의 철학을 자가 확장과 자가 개선이라는 축으로 나란히 비교하고, 한쪽의 설계를 다른 쪽으로 이식하는 관점을 살펴봅니다.

이 단원을 마치면 메신저로 태스크를 받아 코딩 에이전트를 부리는 오케스트레이터 패턴을 포함해, 활용 분포와 AGENTS.md 자동 인식, /sethome 배달 채널, delegate_task 병렬화 같은 Hermes의 정착한 활용 패턴을 구성 방법과 함께 설명할 수 있습니다.
