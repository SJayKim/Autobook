# 5.3.3 delegate_task: 컨텍스트 비용 없는 병렬화

5.1.3에서 Hermes의 네 가지 프로세스 모델을 구별했습니다. 그중 Gateway 위에서 사용자와 대화를 이어 가는 인스턴스를 떠올려 보겠습니다. 이 대화 하나에도 몇십 분씩 걸리는 무거운 요청이 심심치 않게 들어옵니다. 자료 여러 건을 조사해 요약으로 묶는 일, 큰 저장소를 빌드하고 테스트까지 돌리는 일 같은 것입니다. 이런 작업을 지금 대화하는 에이전트가 직접 붙들면 무엇이 문제인지부터 짚겠습니다.

문제의 핵심은 컨텍스트에 있습니다. 여기서 말하는 **컨텍스트**란 모델이 다음 답을 만들 때 매번 함께 읽어 들이는 대화 기록 전체를 말합니다. 긴 작업이 도는 동안 에이전트가 읽은 문서와 도구 출력은 전부 이 기록에 쌓입니다. 사용자에게 정작 필요한 것은 결과 한 건인데, 조사 과정의 중간 산출물이 기록을 불려 놓습니다. 문제는 모델이 매 턴 이 기록을 통째로 다시 읽는다는 데 있습니다. 기록이 길어질수록 한 번 답하는 데 드는 입력 토큰이 늘고, 비용도 따라 오릅니다. 긴 작업을 부모가 직접 처리하면 부모의 컨텍스트가 계속 커지고, 커진 컨텍스트는 이후 모든 대화에 세금처럼 붙습니다.

Hermes는 이 문제를 **delegate_task**로 풉니다. delegate_task는 지금 대화 중인 에이전트가 무거운 작업을 격리된 자식에게 통째로 넘기는 도구입니다. 이때 일을 넘기는 쪽을 부모, 넘겨받는 쪽을 **격리 서브에이전트**라고 부릅니다. 격리 서브에이전트란 부모와 완전히 분리된 자기만의 대화, 자기만의 터미널, 그리고 필요하면 Python으로 짠 작은 스크립트까지 따로 갖고 작업을 수행하는 자식 에이전트를 말합니다. 자기 대화를 따로 둔다는 말이 여기서 결정적입니다. 자식이 조사 과정에서 문서를 아무리 많이 읽어도, 그 기록은 자식의 대화에만 쌓이고 부모의 대화에는 한 줄도 들어가지 않습니다.

Python 스크립트가 왜 언급되는지도 잠깐 짚겠습니다. 자식은 자기 터미널에서 작은 프로그램을 실행해 부모나 외부와 값을 주고받을 수 있는데, 이렇게 다른 곳에 있는 함수를 마치 자기 함수처럼 불러 결과를 받아 오는 방식을 **RPC**(원격 프로시저 호출)라고 합니다. 지금은 자식이 격리된 채로도 필요한 데이터를 끌어오고 결과를 돌려주는 통로가 있다는 정도로 이해하면 충분합니다.

이 구조가 노리는 목표를 Hermes는 zero-context-cost pipelines라는 말로 요약합니다. 우리말로 옮기면 컨텍스트 비용이 들지 않는 작업 파이프라인입니다. 앞서 본 대로 부모의 비용을 키우는 것은 부모의 컨텍스트가 불어나는 일이었습니다. 그런데 작업을 격리 서브에이전트에 넘기면 중간 산출물이 자식 쪽에만 쌓이므로, 부모 컨텍스트는 위임 전과 똑같은 크기로 남습니다. 부모가 매 턴 다시 읽는 양이 늘지 않으니 토큰 비용도 늘지 않습니다. 긴 작업을 몇 개 병렬로 돌려도 마찬가지입니다. 부모는 넘겼다는 사실만 알 뿐, 각 자식이 안에서 무엇을 얼마나 읽든 부모의 기록과는 무관합니다. 이것이 컨텍스트 비용 없는 병렬화라는 이름의 뜻입니다.

이 지점은 OpenClaw의 sub-agent(3.2.4)와 겹칩니다. 두 시스템 모두 오래 걸리는 작업을 격리된 자식에게 넘겨 부모 컨텍스트가 불어나는 것을 막습니다. 다만 무게중심이 다릅니다. OpenClaw가 자식이 다시 자식을 낳는 폭주를 기본값으로 틀어막는 안전에 축을 두었다면, Hermes는 자식이 자기 대화와 터미널과 스크립트를 따로 갖는 격리의 완전함, 그리고 그 덕에 부모 비용이 늘지 않는다는 효율을 앞세웁니다.

이제 시선을 다른 장치로 옮기겠습니다. delegate_task가 무거운 작업을 밖으로 내보내 실행 흐름을 정리하는 수단이라면, Hermes에는 실행 흐름을 안에서 정리하는 또 다른 수단이 있습니다. 그룹챗에서 쓰는 **silence token**입니다. silence token이란 에이전트가 이번 차례에는 아무 말도 하지 않겠다는 뜻으로 내놓는 특별한 신호를 말합니다. 여러 사람이 함께 있는 단체 대화방을 떠올려 보겠습니다. 그 방에 들어온 에이전트가 오가는 모든 메시지에 일일이 끼어들면 성가십니다. 사람도 단체방에서는 듣고만 있다가 자기가 나설 대목에서만 입을 엽니다.

Hermes는 그 행동을 silence token으로 구현합니다. 방에 새 메시지가 도착하면 에이전트는 그 내용을 읽고 자기 기록에 남깁니다. 그런데 대답할 필요가 없다고 판단하면, 사용자에게 보이는 응답 대신 [SILENT]나 NO_REPLY 같은 신호를 내놓습니다. Gateway는 이 신호를 보면 아무것도 채널로 내보내지 않습니다. 결과적으로 방에는 아무 말도 뜨지 않지만, 대화 기록에는 그 메시지가 정상적으로 쌓입니다. 듣고는 있으되 대답만 하지 않는 상태가 만들어집니다. 흐름을 그리면 다음과 같습니다.

```
그룹챗에 새 메시지 도착
        │
        ▼
에이전트가 읽고 기록에 저장  ── 히스토리에는 남는다
        │
        ▼
대답할 필요가 있는가?
   ├─ 있음  → 평소처럼 응답을 채널로 전송
   └─ 없음  → [SILENT] / NO_REPLY 반환 → Gateway가 아무것도 내보내지 않음
```

여기서 눈여겨볼 것은 기록은 유지된다는 점입니다. 침묵은 무시가 아닙니다. 방에서 오간 맥락은 그대로 쌓이므로, 나중에 사용자가 에이전트를 부르면 그동안의 대화를 이미 파악한 상태로 답할 수 있습니다. 만약 침묵할 때 메시지를 기록조차 하지 않는다면, 정작 대답해야 할 순간에 앞뒤 맥락을 놓칠 것입니다. 듣기와 말하기를 분리한 설계가 그 문제를 막습니다.

두 장치를 하나로 묶어 보면 이렇습니다. delegate_task는 무거운 작업을 격리 서브에이전트로 내보내 부모의 컨텍스트를 지키고, silence token은 그룹챗에서 언제 입을 열고 언제 다물지를 정합니다. 하나는 작업을 밖으로 넘기고 하나는 발화를 안에서 거르지만, 둘 다 부모 에이전트가 자기 실행 흐름을 스스로 통제하게 해 주는 같은 계열의 수단입니다.

정리하면, 긴 작업을 부모가 직접 붙들면 중간 산출물이 부모 컨텍스트를 불려 매 턴 비용이 오릅니다. Hermes는 delegate_task로 그 작업을 자기 대화와 터미널과 Python RPC 스크립트를 따로 가진 격리 서브에이전트에 넘겨, 부모 비용을 늘리지 않고 여러 작업을 병렬로 돌립니다. 이것이 zero-context-cost pipelines입니다. 여기에 그룹챗의 [SILENT], NO_REPLY 같은 silence token은 히스토리는 남기면서 응답만 억제해, 위임과 함께 실행 흐름을 제어하는 또 하나의 수단이 됩니다.

5.3.4에서는 시선을 데이터 쪽으로 옮겨, 에이전트의 사용 기록이 다시 모델 학습으로 되돌아가는 MLOps 계층, 곧 모델 트레이너의 플라이휠을 다룹니다.

이 단원을 마치면 delegate_task로 격리 서브에이전트에 작업을 위임해 부모의 컨텍스트 비용을 늘리지 않고 병렬화하는 구조와, 그룹챗의 silence token이 실행 흐름을 제어하는 방식을 설명할 수 있습니다.
