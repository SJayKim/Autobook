# 2.3.1 Agent loop: 메시지가 행동이 되는 다섯 단계

2.2.1에서 모든 메신저 연결을 혼자 소유하는 단일 상주 프로세스, 게이트웨이를 봤습니다. 2.1.2에서는 판단과 실행을 맡는 Pi 최소 코어를 봤습니다. 이제 둘을 잇는 질문이 남습니다. 메시지 하나가 게이트웨이에 도착한 순간부터 답장이 돌아가기까지, 그 사이에서 무슨 일이 일어나는가입니다. 이 단원은 그 사이를 해부합니다.

OpenClaw는 이 과정을 **agent loop**라고 부릅니다. 공식 정의는 "메시지 하나를 행동과 응답으로 바꾸는, 세션별로 직렬화된 실행"입니다. 여기서 메시지 하나를 처리하는 실행 한 번을 **run**이라고 부릅니다. 직렬화란 같은 세션의 run이 동시에 두 개 돌지 않고 한 줄로 순서대로 돈다는 뜻입니다. 직렬화 장치 자체는 다음 단원의 주제이므로, 이 단원은 run 하나의 안쪽에 집중합니다.

run은 다섯 단계를 지납니다. 첫째는 intake, 수신입니다. 채널에서 넘어온 요청을 검증하고 어느 세션의 대화인지 정합니다. 둘째는 context assembly, 문맥 조립입니다. 지금까지의 세션 기록과 워크스페이스의 기억 파일, 로드된 스킬을 모델이 읽을 하나의 입력으로 합칩니다. 셋째는 model inference, 모델 호출입니다. 조립된 문맥을 외부 LLM에 보내 다음 행동을 판단하게 합니다. 넷째는 tool execution, 도구 실행입니다. 모델이 요청한 도구를 내 머신에서 실행하고 결과를 다시 모델에게 돌려줍니다. 더 부를 도구가 없다고 모델이 판단할 때까지 셋째와 넷째가 되풀이되는데, loop라는 이름이 여기서 나옵니다. 다섯째는 streaming과 persistence, 스트리밍과 저장입니다. 만들어지는 응답을 조각(델타) 단위로 채널에 흘려보내고, 대화 기록을 디스크에 남깁니다.

```
[메신저 메시지 도착]
   │
   ▼
 1. intake                 요청 검증, 세션 결정
   │
   ▼
 2. context assembly       세션 기록 + 기억 파일 + 스킬 조립
   │
   ▼
 3. model inference        모델 호출, 다음 행동 판단 <─┐
   │                                                   │
   ▼                                                   │
 4. tool execution         도구 실행, 결과를 모델에게 ─┘
   │      (더 부를 도구가 없을 때까지 3과 4 반복)
   ▼
 5. streaming·persistence  응답 스트리밍, 기록 저장
   │
   ▼
[채널로 회신, 디스크에 대화 기록]
```

가장 눈여겨볼 설계는 시작 지점, agent RPC에 있습니다. RPC는 함수를 부르듯 다른 프로세스에 일을 시키는 원격 호출이고, 클라이언트는 게이트웨이의 WebSocket으로 이 요청을 보냅니다. agent RPC는 파라미터 검증과 세션 해석, 메타데이터 저장까지만 마치고 곧바로 { runId, acceptedAt }을 돌려줍니다. runId는 이번 run의 식별자, acceptedAt은 접수 시각입니다. 다섯 단계 중 intake가 이 접수에 해당하며, 나머지가 끝나기를 기다리지 않고 응답부터 나갑니다.

왜 완료를 기다리지 않을까요. run은 모델 호출과 도구 실행을 여러 번 오가므로 몇 분씩 걸릴 수 있습니다. RPC가 완료까지 응답을 붙들고 있으면 호출자, 곧 CLI나 Control UI나 채널 어댑터가 그 시간 내내 막힙니다. 그래서 OpenClaw는 접수와 처리를 분리합니다. 요청에는 접수증인 runId로 즉시 답하고, 실제 진행과 결과는 WebSocket 이벤트로 밀어 보냅니다. 이 패턴을 **비동기 작업 수락**(async job acceptance)이라고 부릅니다. 운송장 번호만 받고 돌아서는 택배 접수와 같은 구조입니다.

접수 뒤의 처리는 안쪽으로 이어집니다. agentCommand가 어떤 모델을 쓸지 기본값을 해석하고 스킬을 로드한 뒤, 실행의 몸통인 **runEmbeddedAgent**를 호출합니다. runEmbeddedAgent는 run을 세션별 큐(필요하면 전역 큐까지)로 직렬화하고, 사용할 모델과 인증 프로파일을 해석하고, OpenClaw 세션을 빌드합니다. 이어서 런타임 이벤트를 구독하고, 모델 응답(assistant)과 도구(tool)의 델타를 스트리밍하며, run 전체에 시간 제한을 강제합니다. 다섯 단계의 몸통이 사실상 이 함수의 관할에서 돌아가는 셈입니다. 구독된 런타임 이벤트는 subscribeEmbeddedAgentSession이 tool, assistant, lifecycle 세 갈래 스트림으로 나눠 라우팅합니다. lifecycle은 run의 시작과 종료 같은 생명주기 이벤트가 흐르는 갈래입니다.

완료까지 기다리고 싶은 호출자에게는 agent.wait이라는 통로가 따로 있습니다. agent.wait은 lifecycle에서 run의 종료나 에러를 폴링하다가, run이 끝나면 { status, startedAt, endedAt, error? }를 돌려줍니다. status는 ok, error, timeout 셋 중 하나입니다. 접수(agent)와 대기(agent.wait)가 별도 RPC라서 기다릴지 말지는 호출자의 선택입니다. 결과가 꼭 필요한 스크립트는 agent.wait으로 기다리고, 화면을 계속 그려야 하는 UI는 이벤트만 구독하면 됩니다.

status에 timeout이 있다는 것은 run에 시간 상한이 있다는 뜻입니다. runEmbeddedAgent는 run마다 타임아웃을 강제해, 상한을 넘긴 run을 timeout 상태로 끝냅니다. 2.1.2에서 봤듯 이 에이전트는 스스로 코드를 짜서 실행하므로, 끝나지 않는 명령 하나로 run이 하염없이 길어질 수 있습니다. 그런데 같은 세션의 run은 한 줄로 순서대로 돕니다. 하나가 영영 끝나지 않으면 그 세션으로 오는 다음 메시지가 전부 뒤에서 기다립니다. 타임아웃은 그런 run을 끊어 줄을 되돌려 주는 안전판입니다.

정리하면, agent loop는 메시지 하나를 행동과 응답으로 바꾸는 세션별 직렬 실행이며, intake부터 streaming과 persistence까지 다섯 단계를 지납니다. agent RPC는 완료를 기다리지 않고 즉시 runId를 돌려주는 비동기 작업 수락으로 호출자를 막지 않으며, 결과는 이벤트로 따라옵니다. 실행의 몸통은 runEmbeddedAgent가 맡아 큐 직렬화, 모델과 인증 프로파일 해석, 세션 빌드, 델타 스트리밍, 타임아웃 강제까지 책임집니다. 기다림이 필요한 호출자는 agent.wait으로 ok, error, timeout 세 상태 중 하나를 받습니다.

2.3.2에서는 이 단원에서 미뤄 둔 장치, 곧 run을 세션별로 줄 세우는 lanes와 큐가 어떻게 경쟁을 없애는지를 다룹니다.

이 단원을 마치면 메시지 하나가 intake부터 persistence까지 다섯 단계를 거쳐 행동과 응답이 되는 agent loop의 흐름과, 즉시 runId를 반환하는 비동기 수락 설계를 설명할 수 있습니다.
