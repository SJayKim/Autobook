# 2.1.2 software building software와 Pi 최소 코어

2.1.1에서 뇌와 근육의 분리를 다뤘습니다. 판단하는 뇌는 외부 LLM에 두고, 실행하는 근육은 내 머신의 설치본에 두는 구도입니다. 그렇다면 다음 질문이 자연스럽게 이어집니다. 내 머신에 놓이는 근육은 얼마나 커야 할까요. 메신저 연동부터 파일 정리, 일정 관리까지 온갖 기능을 미리 코드로 만들어 넣는다면 근육은 끝없이 비대해집니다. OpenClaw의 답은 반대 방향입니다. 코어는 극단적으로 작게 유지하고, 나머지는 에이전트가 스스로 만들게 합니다.

이 답을 지탱하는 사상이 **software building software**입니다. 소프트웨어가 소프트웨어를 만든다는 뜻입니다. 전통적인 프로그램에서 새 기능이 필요하면 사용자는 개발자가 플러그인이나 업데이트를 내놓을 때까지 기다립니다. OpenClaw는 이 관계를 뒤집습니다. 새 기능이 필요하면 에이전트에게 "이 기능을 코드로 만들어 붙여라"라고 요청합니다. 에이전트는 코드를 읽고 쓸 줄 아는 실행 주체이므로, 자기가 쓸 도구를 자기 손으로 만들 수 있습니다. 기능 확장이 배포자의 일정이 아니라 사용자의 요청 시점에 일어나는 것입니다. 이 사상은 코어를 작게 유지하는 근거이기도 합니다. 에이전트가 스스로 만들 수 있는 기능이라면 코어에 미리 넣을 이유가 없기 때문입니다.

이 사상이 서 있는 바닥이 **Pi**입니다. Pi는 개발자 Mario Zechner가 만든 미니멀 코딩 에이전트입니다. OpenClaw의 에이전트 런타임은 이 Pi를 내부 코어로 삼고, 그 위에 메신저 채널 연결과 메모리, 스킬을 얹은 형태입니다. Pi가 미니멀이라 불리는 근거는 구체적입니다. 시스템 프롬프트를 가능한 한 짧게 유지하고, 도구는 단 네 개만 갖습니다. 파일을 읽는 Read, 파일을 새로 쓰는 Write, 파일 일부를 고치는 Edit, 그리고 셸 명령을 실행하는 Bash입니다.

네 개는 부족해 보이지만 조합하면 달라집니다. 열쇠는 Bash입니다. 셸 명령을 실행할 수 있으면 그 컴퓨터에 설치된 모든 프로그램에 닿을 수 있습니다. 여기에 파일을 읽고 쓰고 고치는 능력이 더해지면, 에이전트는 스크립트를 작성해 저장하고 곧바로 실행할 수 있습니다. 다섯 번째 도구가 필요해지는 순간, 에이전트가 그 도구를 코드로 만들어 쓰면 됩니다. 이 구조를 **최소 코어 + 자가 확장**(minimal core + self-extension)이라고 부릅니다. 코어는 확장의 발판만 제공하고, 실제 기능은 필요할 때 에이전트가 만들어 내는 방식입니다.

```
[외부 LLM = 뇌]  ← 모델은 갈아 끼울 수 있다
      │
      ▼
┌─ OpenClaw ────────────────────────────┐
│  채널 연결 · 메모리 · 스킬 (얹은 층)  │
│                                       │
│  Pi 코어: 극히 짧은 시스템 프롬프트   │
│           Read / Write / Edit / Bash  │
└───────────────────────────────────────┘
      │ 부족한 기능은 에이전트가 코드로 작성
      ▼
[스크립트 · CLI 도구 · 새 기능]  ← 자가 확장
```

이 선택이 얼마나 의도적인지는 MCP를 대하는 태도에서 드러납니다. **MCP**(Model Context Protocol)는 AI 애플리케이션이 외부 도구와 데이터에 연결되는 방식을 표준화한 프로토콜입니다. 도구 제공자가 MCP 서버를 만들어 두면, 표준을 따르는 AI 애플리케이션은 같은 방식으로 접속해 도구를 호출합니다. 에이전트에 기능을 붙이는 표준 통로인 셈입니다. 그런데 Pi는 MCP를 의도적으로 지원하지 않습니다. 미구현이 아니라 철학적 선택입니다.

대신 **mcporter**라는 도구를 씁니다. mcporter는 MCP 서버가 제공하는 기능을 CLI 명령, 곧 터미널에서 실행하는 명령어 형태로 노출합니다. MCP 서버의 기능이 CLI 명령이 되는 순간, 에이전트는 이미 가진 Bash 도구로 그 명령을 실행하면 됩니다. 코어에 MCP 프로토콜을 처리하는 층을 새로 넣을 필요가 없습니다. 도구가 늘어날 때마다 프로토콜 통합을 반복하는 대신, 모든 것이 셸 명령이라는 하나의 형태로 수렴합니다. 표준 프로토콜 통합보다 셸이 곧 만능 인터페이스라는 노선을 택한 것입니다.

작은 코어는 모델 교체의 자유와도 맞물립니다. Pi는 특정 모델에 묶이지 않는 **model-agnostic** 런타임입니다. 어떤 모델이 와도 같은 방식으로 돌아가도록 설계했다는 뜻입니다. OpenClaw가 Anthropic과 OpenAI의 상용 모델부터 Ollama로 띄운 로컬 모델까지 같은 런타임으로 부리는 바탕이 여기 있습니다. 코어가 작을수록 특정 모델의 동작을 전제한 코드도 적고, 그만큼 뇌를 갈아 끼우기 쉽습니다. 2.1.1의 분리 구도가 코어의 크기 설계에서도 관철되는 셈입니다.

물론 이 구조는 공짜가 아닙니다. 에이전트가 무엇이든 만들어 붙일 수 있다는 말은, 잘못된 판단이나 악의적 입력이 코드 실행으로 곧장 이어질 수 있다는 말이기도 합니다. Bash 하나로 컴퓨터 전체에 닿는 코어라면, 누구의 요청을 어디까지 믿고 실행할지가 다음 설계 과제가 됩니다.

정리하면, OpenClaw는 기능을 코어에 미리 쌓아 두지 않습니다. software building software 사상에 따라 필요한 기능을 에이전트가 코드로 만들어 붙입니다. 그 바닥에는 Mario Zechner의 미니멀 코딩 에이전트 Pi가 있습니다. Pi는 짧은 시스템 프롬프트와 Read, Write, Edit, Bash 네 도구로 최소 코어 + 자가 확장 구조를 이룹니다. MCP는 의도적으로 지원하지 않고, mcporter로 MCP 서버를 CLI 명령으로 노출해 Bash로 부르게 했습니다. 그리고 특정 모델에 묶이지 않는 model-agnostic 런타임으로 뇌를 교체할 자유를 지켰습니다.

2.1.3에서는 이렇게 강한 실행 능력을 쥔 에이전트를 누가 어디까지 믿을 것인가, 곧 개인 비서 신뢰 경계를 다룹니다.

이 단원을 마치면 OpenClaw가 Pi 최소 코어 위에서 자가 확장으로 기능을 늘리는 구조와 MCP 대신 CLI 노출을 택한 이유를 설명할 수 있습니다.
