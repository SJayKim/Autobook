# 2.3.3 Session: 라우팅과 저장 구조

2.3.1에서 메시지 하나가 행동과 응답으로 바뀌는 agent loop를 따라갔습니다. 그런데 루프가 돌기 전에 한 가지가 먼저 정해져야 합니다. 지금 도착한 메시지를 어느 대화 문맥에 이어 붙일 것인가입니다. WhatsApp DM으로 온 질문, 팀 Slack 그룹방의 멘션, 매주 월요일 아침의 예약 작업이 전부 한 문맥을 공유하면 대화는 금세 뒤엉킵니다. 반대로 모든 메시지가 매번 새 문맥에서 시작하면 에이전트는 아무것도 이어가지 못합니다. OpenClaw는 이 문제를 세션이라는 단위와 그 라우팅 규칙으로 풉니다.

**세션**(session)은 에이전트가 유지하는 대화 문맥의 단위입니다. 같은 세션에 들어온 메시지는 하나의 이어지는 대화로 취급되고, 다른 세션의 내용은 서로 보이지 않습니다. 그리고 메시지의 출처에 따라 어느 세션에 넣을지 범위를 정하는 방식을 **session scoping**이라고 부릅니다. 어디에서 온 메시지인가가 곧 문맥의 칸을 결정한다는 뜻입니다. OpenClaw의 라우팅 규칙은 이 scoping을 출처 유형별로 다르게 적용합니다.

먼저 DM입니다. 개인이 보내는 DM은 기본적으로 main이라는 하나의 공유 세션으로 모입니다. Telegram에서 부탁한 일을 잠시 뒤 WhatsApp에서 물어도 에이전트는 같은 문맥에서 대답합니다. 어느 창구로 말을 걸든 상대는 같은 비서 한 명이어야 한다는 판단이 기본값에 담겨 있습니다. 다만 이 기본값이 항상 맞지는 않으므로 **dmScope** 옵션(설정 경로 session.dmScope)으로 조정할 수 있습니다. main은 모든 DM을 한 세션으로 모으고, per-peer는 대화 상대별로 세션을 나눕니다. per-channel-peer는 채널과 상대의 조합별로 나누며, 이것이 권장 값입니다. 가장 세분화된 per-account-channel-peer는 메시지를 받은 계정까지 조합에 포함합니다.

DM과 달리 그룹, cron, webhook은 처음부터 격리됩니다. 그룹·룸·채널 대화는 대화방마다 자기 세션을 갖습니다. cron 작업은 실행할 때마다 새 세션에서 시작하고, webhook은 hook별로 세션이 분리됩니다. 출처별 라우팅을 정리하면 다음과 같습니다.

| 출처 | 세션 라우팅 |
|---|---|
| DM | 기본 공유 세션(main), dmScope로 분리 조정 |
| 그룹·룸·채널 | 대화방마다 격리 |
| cron | 실행마다 새 세션 |
| webhook | hook별 격리 |

이 비대칭이 세션 설계의 핵심 절충입니다. DM을 공유하는 이유는 연속성입니다. 개인 사용자에게 에이전트는 하나의 비서여야 하고, 채널을 옮길 때마다 기억을 잃는다면 비서라고 부르기 어렵습니다. 반대로 그룹과 자동화를 격리하는 이유는 문맥 오염 방지입니다. 그룹방의 대화가 개인 DM의 판단 근거에 섞여 들어가면 곤란합니다. 사람이 지켜보지 않는 cron 실행이 지난 실행의 잔여 문맥을 물고 시작해도 결과를 예측하기 어렵습니다. 연속성이 가치인 곳은 공유하고 깨끗한 시작이 가치인 곳은 격리한다는 것이 라우팅 규칙 전체를 관통하는 원칙입니다.

이렇게 나뉜 세션은 디스크에 두 층으로 저장됩니다. 런타임 상태는 SQLite 데이터베이스에 담깁니다. **SQLite**는 별도의 서버 프로세스 없이 파일 하나로 동작하는 관계형 데이터베이스 엔진입니다. 상주 프로그램이 로컬 상태를 쌓아 두는 용도로 널리 쓰입니다. 한편 대화 원문 아카이브는 **transcript** 파일로 남습니다. 에이전트별 디렉토리 구조는 다음과 같습니다.

```
~/.openclaw/agents/<agentId>/
├── agent/
│   └── openclaw-agent.sqlite   <- 런타임 세션 상태 (SQLite)
└── sessions/                   <- 아카이브 transcript
```

과거 버전이 쓰던 sessions.json은 레거시 마이그레이션 소스로 남아 있습니다. 구버전에서 올라온 설치본의 세션은 이 파일을 출발점으로 새 구조에 옮겨집니다.

각 세션 레코드에는 lifecycle 타임스탬프 세 가지가 함께 기록됩니다. sessionStartedAt은 세션이 시작된 시각, lastInteractionAt은 마지막 상호작용 시각, updatedAt은 레코드가 갱신된 시각입니다. 지금은 필드 이름만 눈에 익혀 두면 충분합니다. 세션을 언제 새로 시작하고 오래된 것을 언제 정리할지 판단하는 수명 정책이 이 시각 정보를 근거로 움직이기 때문입니다.

정리하면, OpenClaw의 세션은 메시지 출처에 따라 문맥의 범위를 정하는 session scoping 위에 서 있습니다. DM은 기본 공유 세션(main)으로 모여 하나의 비서 같은 연속성을 만들고, 분리가 필요하면 dmScope로 수준을 조정합니다. 그룹은 대화방마다, cron은 실행마다, webhook은 hook별로 격리되어 문맥 오염을 막습니다. 저장은 ~/.openclaw/agents/<agentId>/ 아래에서 두 층으로 나뉘어, 런타임 상태는 SQLite 파일에, 대화 아카이브는 sessions/의 transcript에 담깁니다. 각 세션은 sessionStartedAt, lastInteractionAt, updatedAt 세 타임스탬프로 자기 이력을 기록합니다.

2.3.4에서는 이 타임스탬프를 근거로 세션을 새로 시작하고 정리하는 리셋과 유지보수 정책을 다룹니다.

이 단원을 마치면 DM, 그룹, cron, webhook이 각각 어떤 세션으로 라우팅되는지, dmScope 옵션이 무엇을 바꾸는지, 세션이 어디에 어떤 형식으로 저장되는지 설명할 수 있습니다.
