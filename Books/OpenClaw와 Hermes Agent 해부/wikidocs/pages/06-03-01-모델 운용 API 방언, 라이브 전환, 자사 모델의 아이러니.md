# 6.3.1 모델 운용: API 방언, 라이브 전환, 자사 모델의 아이러니

5.2.2에서 우리는 provider resolver가 어떻게 여러 모델 제공사를 하나의 인터페이스로 묶는지 살펴봤습니다. 이 단원은 그 위에서 실제 운용이 어떻게 이루어지는지를 다룹니다. 세 가지 요청 형식을 어떻게 아우르는지, 대화 도중 모델을 어떻게 갈아타는지를 봅니다. 자사가 만든 모델을 왜 정작 에이전트 안에서는 권하지 않는지까지 짚습니다. 결론을 먼저 말하겠습니다. 이 시스템은 특정 모델 회사에 매이지 않도록 설계됐습니다. 그 중립성이 운용 방식과 자사 모델을 대하는 태도 양쪽에 그대로 드러납니다.

먼저 배경부터 짚겠습니다. 언어 모델을 쓰려면 그 모델을 제공하는 회사의 서버에 요청을 보내고 응답을 받아야 합니다. 그런데 회사마다 요청을 어떤 모양으로 보내고 응답이 어떤 모양으로 오는지가 서로 다릅니다. 같은 뜻의 대화라도 담는 그릇의 형식이 갈립니다. 이렇게 제공사마다 갈리는 요청과 응답의 형식을 여기서는 **API 방언**이라고 부르겠습니다. 사람의 언어에 방언이 있듯, 하려는 말은 같아도 문법이 조금씩 다른 셈입니다.

Hermes Agent가 감당하는 방언은 세 가지입니다. 하나는 오래전부터 널리 쓰여 온 chat completions 형식입니다. 대화를 여러 발화의 목록으로 담아 보내는 방식이라 많은 제공사가 이 형식을 따릅니다. 둘째는 그 뒤에 나온 responses 형식으로, 같은 계열이지만 요청과 응답의 구조가 개편된 방언입니다. 셋째는 Anthropic Messages 형식으로, Claude 계열 모델이 쓰는 고유한 방언입니다. 세 방언은 대화를 담는 틀도, 도구 호출을 표현하는 방법도 서로 달라, 하나로 짠 코드로는 셋을 동시에 상대할 수 없습니다.

바로 여기서 5.2.2의 resolver가 다시 등장합니다. resolver는 어떤 모델을 고르느냐에 따라 그 모델이 어느 방언을 쓰는지 판단하고, 그에 맞는 형식으로 요청을 빚어 보냅니다. 세 방언의 차이를 이 한 곳이 흡수해 주므로, 에이전트의 나머지 부분은 지금 어떤 회사의 모델을 쓰는지 신경 쓰지 않아도 됩니다. 방언을 몰라도 대화가 되게 해 주는 통역사가 resolver인 셈입니다.

방언 차이가 이렇게 한곳에서 흡수되기에, 대화 도중 모델을 바꾸는 일도 자연스러워집니다. 이 시스템에는 대화를 이어 가는 도중에 사용하는 모델을 다른 모델로 갈아타는 **라이브 전환** 기능이 있습니다. 대화 중에 `/model` 명령을 입력하면 그 자리에서 모델이 바뀝니다. 이 기능은 v0.8.0에서 처음 들어왔습니다. 쓰임새는 뚜렷합니다. 가벼운 잡담은 값싼 모델에 맡기다가, 어려운 판단이 필요한 순간에만 더 강한 모델로 올려 붙이는 식입니다. 다음처럼 씁니다.

```
> /model openrouter anthropic/claude-opus-4.7
모델을 claude-opus-4.7 로 전환했습니다.
```

이때 한 가지 대가를 기억해야 합니다. 모델을 바꾸면 앞부분이 그대로여야 적중하는 프롬프트 캐시가 깨집니다. 그러면 다음 턴에서 대화 전체가 다시 과금됩니다. 이 캐시 무효화 함정은 6.1.3에서 다뤘으니 여기서는 라이브 전환에도 같은 비용이 따른다는 점만 짚고 넘어갑니다. 자주 갈아타면 편하지만 그만큼 비싸집니다.

전환의 연장선에서 한 가지 옵션을 이름만 소개하겠습니다. 여러 모델을 하나의 대화 안에서 섞어 쓰는 **Mixture-of-Agents** 방식입니다. 하나의 모델에만 의존하지 않고 여러 모델의 판단을 조합하는 선택지로, v0.18.0부터 정식 옵션으로 들어왔습니다. 이 책에서는 이런 운용이 가능하다는 사실을 아는 수준으로 충분합니다.

이제 이 단원에서 가장 흥미로운 대목입니다. Hermes Agent를 만든 곳은 오래전부터 Hermes라는 이름의 언어 모델을 만들어 온 연구조직입니다. 그중 최신 세대인 Hermes 4 모델도 이 시스템에서 쓸 수 있습니다. 그런데 정작 공식 문서는 자사의 Hermes 4를 에이전트 안에서 쓰는 것을 권하지 않는다고 밝힙니다. 이유가 정직합니다. Hermes 4는 사람과 대화를 나누는 데 맞춰 다듬어진 모델입니다. 그래서 에이전트가 쉴 새 없이 도구를 부르며 일하는 상황에는 잘 맞지 않는다는 것입니다.

이 대목이 왜 중요한지 짚어 보겠습니다. 에이전트의 실제 일은 대화보다 도구 호출에 가깝습니다. 파일을 열고, 명령을 돌리고, 결과를 읽어 다음 도구를 부르는 일이 촘촘하게 이어집니다. 채팅에 맞춰진 모델은 이렇게 빠르고 정확한 도구 호출을 요구받는 자리에서 실력을 온전히 내지 못합니다. 자기 모델을 팔면 이득일 텐데도 그 자리에는 다른 회사의 도구 특화 모델을 권하는 셈입니다. 이 정직함은 곧 방증이기도 합니다. 이 에이전트는 자기 이름을 단 모델에 종속되지 않습니다. 이름을 공유할 뿐, 모델 선택은 성능만 보고 정한다는 뜻입니다.

정리하면, Hermes Agent는 chat completions와 responses와 Anthropic Messages라는 세 API 방언을 resolver로 흡수해 어느 회사의 모델이든 갈아 끼울 수 있게 합니다. 여기에 `/model`로 대화 도중 모델을 바꾸는 라이브 전환과 여러 모델을 섞는 Mixture-of-Agents 옵션까지 갖췄습니다. 자사의 Hermes 4마저 채팅 최적화라는 이유로 에이전트 안에서는 비권장한다는 대목은, 이 시스템이 특정 모델에 매이지 않는 중립 운용을 실제로 지킨다는 증거입니다.

6.3.2에서는 이렇게 여러 모델을 하나의 구독으로 묶어 주는 Nous Portal이 어떻게 구독 게이트웨이 역할을 하며 단기 자격증명으로 열쇠 노출을 줄이는지를 이어서 다룹니다.

이 단원을 마치면 세 가지 API 방언 지원과 대화 중 모델 전환을 설명하고, 자사 Hermes 4 모델을 에이전트에 비권장하는 이유를 해석할 수 있습니다.
