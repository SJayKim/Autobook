# 2.3.4 Session 수명: 리셋과 정리 정책

2.3.3에서 세션이 어떤 기준으로 나뉘고 어디에 저장되는지 살펴보았습니다. 이제 남은 질문은 시간입니다. 상주형 에이전트는 꺼지지 않습니다. 내버려 두면 개인 DM 세션 하나가 몇 달치 대화를 그대로 끌고 다니게 됩니다. 이 단원에서는 OpenClaw가 세션에 수명을 부여해 이 무한 성장을 막는 두 축, **리셋 정책**과 **유지보수 설정**을 다룹니다.

무한 성장이 왜 문제인지부터 짚습니다. 세션 기록은 매 턴 모델에게 들어가는 문맥의 원천입니다. 기록이 길어질수록 입력 토큰 비용이 늘고, 컨텍스트 윈도 한계에 부딪혀 요약도 잦아집니다. 몇 주 전의 잡담이 오늘의 판단에 끼어드는 문맥 오염도 생깁니다. 디스크 사정도 같습니다. 대화 기록은 지우지 않는 한 계속 쌓입니다. 웹 챗봇이라면 사용자가 새 대화 버튼을 눌러 문맥을 끊어 주지만, 메신저 안에 상주하는 에이전트에게는 그 버튼을 눌러 줄 사람이 없습니다. 그래서 시스템이 스스로 끊는 규칙이 필요합니다.

이런 문제를 다루는 오래된 개념이 **TTL**(time to live)입니다. 네트워크와 캐시 시스템은 데이터를 영원히 유효한 것으로 취급하지 않습니다. 저장하는 순간 유효 수명을 함께 붙여 두고, 수명이 다한 데이터는 버리거나 새로 만듭니다. 이 수명이 TTL입니다. 예를 들어 DNS 캐시는 조회한 주소를 TTL 동안만 믿고, 그 시간이 지나면 다시 물어봅니다. OpenClaw의 세션 관리도 같은 관점에 서 있습니다. 세션마다 시작 시각과 마지막 상호작용 시각을 기록해 두고, 정책이 정한 수명이 지나면 그 세션을 닫고 새 세션을 엽니다. 세션을 영구 객체가 아니라 수명이 있는 객체로 다루는 TTL 기반 lifecycle 관리입니다.

수명을 정하는 리셋 정책은 세 가지입니다. `none`은 자동 리셋을 하지 않는 선택입니다. 대신 오래된 교환을 요약해 문맥을 압축하는 컴팩션(compaction)으로 성장을 관리합니다. `daily`는 매일 정해진 시각에 새 세션을 시작하는 정책이며 기본 시각은 새벽 4시입니다. 사람의 하루 리듬에 맞춰 어제의 대화를 어제로 남기는 방식입니다. `idle`은 마지막 상호작용 이후 비활동 시간이 임계를 넘으면 새 세션을 여는 정책입니다. 달력이 아니라 대화가 실제로 이어지고 있는지를 기준으로 삼습니다.

두 정책은 함께 설정할 수도 있습니다. 이때 우선 규칙은 하나입니다. 먼저 만료되는 쪽이 이깁니다. idle 임계를 2시간으로 잡았다고 합시다. 밤 9시에 마지막 메시지를 보냈다면 밤 11시의 idle 만료가 새벽 4시보다 먼저 오므로 idle이 세션을 닫습니다. 반대로 새벽 3시 반까지 대화가 이어졌다면 idle 만료 전에 4시가 먼저 도착하므로 daily가 닫습니다.

```
마지막 사람 메시지          idle 만료(+2h)        daily 리셋
──────●───────────────────────○───────────────────○──────▶ 시간
     21:00                   23:00               04:00
               둘 중 먼저 오는 만료가 세션을 닫는다
```

여기에 함정이 하나 있습니다. idle이 재는 "마지막 상호작용"의 기준입니다. OpenClaw에는 사람이 말을 걸지 않아도 도는 자동 활동이 있습니다. 주기적으로 점검 턴을 도는 heartbeat와 예약 작업을 실행하는 cron입니다. 상세한 동작은 3.2에서 다루고, 여기서는 수명과의 관계만 봅니다. 만약 이 자동 턴이 상호작용으로 집계되어 타이머를 되돌린다면, 기본 30분 간격으로 도는 heartbeat 탓에 세션은 idle 임계에 영영 도달하지 못합니다. 리셋은 무력화되고 무한 성장으로 되돌아갑니다. 그래서 heartbeat와 cron은 리셋 타이머를 연장하지 않습니다. idle이 측정하는 것은 시스템이 살아 있는지가 아니라 사람과의 대화가 살아 있는지입니다.

리셋은 지금 살아 있는 대화를 끊을 뿐입니다. 닫힌 세션의 기록은 디스크에 그대로 남습니다. 이쪽의 성장은 유지보수 설정 `session.maintenance`가 맡습니다.

```
"session": {
  "maintenance": { "mode": "enforce", "pruneAfter": "30d", "maxEntries": 500 }
}
```

`pruneAfter`는 보존 기한입니다. "30d"로 두면 30일이 지난 세션 기록을 가지치기(prune)하듯 정리합니다. `maxEntries`는 보관 개수의 상한으로, 세션 항목이 500개를 넘지 않게 유지합니다. `mode`를 "enforce"로 두면 이 규칙이 실제로 집행됩니다. 세션 아카이브에는 대화 전문이 남기 때문에 이 설정은 용량 관리이자 개인정보 관리이기도 합니다. 무엇이든 30일 넘게 남기지 않는다는 보존 기준을 처음부터 정해 두면, 지난 대화가 뜻하지 않은 유출 자산이 되는 일을 줄일 수 있습니다.

두 축을 겹쳐 보면 역할이 나뉩니다. 리셋 정책은 모델에게 들어가는 살아 있는 문맥의 성장을 막고, 유지보수 설정은 디스크에 쌓이는 기록의 성장을 막습니다. 남는 질문은 연속성입니다. 세션을 매일 끊는데도 에이전트가 어제 일을 기억하려면, 세션 바깥에 남는 기억이 따로 필요합니다. 그 역할을 맡는 파일들이 모인 곳이 워크스페이스이며, Phase 3의 주제입니다.

정리하면, OpenClaw는 세션을 수명이 있는 객체로 다룹니다. daily는 정해진 시각에, idle은 비활동 임계를 넘겼을 때 새 세션을 열고, none은 리셋 없이 컴팩션에 맡깁니다. 두 정책을 함께 켜면 먼저 만료되는 쪽이 이깁니다. heartbeat와 cron 같은 자동 활동은 리셋 타이머를 연장하지 않으므로 idle은 사람의 활동만 잽니다. 닫힌 기록은 session.maintenance의 pruneAfter와 maxEntries가 기한과 개수로 정리합니다. 리셋이 살아 있는 문맥의 성장을, 정리가 디스크 기록의 성장을 막아 무한 성장을 함께 차단합니다.

3.1.1에서는 Phase 3로 넘어가, 세션이 끊겨도 남아야 할 정체성을 파일로 만드는 워크스페이스를 시작으로 에이전트의 기억과 능동성, 확장을 다룹니다.

이 단원을 마치면 리셋 정책 none, daily, idle과 유지보수 설정으로 대화 상태의 무한 성장을 막는 방법을 설명할 수 있습니다.
