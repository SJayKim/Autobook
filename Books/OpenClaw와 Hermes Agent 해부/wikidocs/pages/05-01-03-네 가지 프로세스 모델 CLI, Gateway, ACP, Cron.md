# 5.1.3 네 가지 프로세스 모델: CLI, Gateway, ACP, Cron

5.1.2에서 하나의 사실을 확인했습니다. Hermes는 대화 루프를 담당하는 코어를 하나만 두고, 그 코어가 플랫폼 차이를 모른 채 돌아가게 만들었다는 것입니다. 그런데 이 코어는 혼자서는 아무 일도 시작하지 못합니다. 누군가 사용자의 입력을 받아 코어를 깨우고, 코어가 내놓은 결과를 다시 사용자에게 돌려줘야 합니다. 이 바깥쪽 껍데기를 **엔트리포인트**라고 부릅니다. 엔트리포인트란 프로그램이 실제로 실행을 시작하는 진입 지점, 즉 코어를 감싸 사용자와 이어 주는 얇은 층을 말합니다.

Hermes는 같은 코어를 서로 다른 엔트리포인트 네 개로 감쌉니다. CLI, Gateway, ACP, Cron입니다. 넷은 하는 일이 겹치지만 결정적으로 다른 점이 하나 있습니다. 프로세스의 수명, 즉 언제 태어나 언제 사라지는가입니다. 이 단원에서는 네 모델의 수명과 데이터가 흐르는 경로를 구별해 봅니다.

먼저 **CLI**입니다. CLI는 터미널 창에서 사람과 한 문장씩 주고받는 대화형 방식입니다. 이때 화면을 채우는 대화 인터페이스를 **TUI**(터미널 UI)라고 부릅니다. TUI란 그래픽 창이 아니라 글자로 짜인 터미널 화면 위에서 입력창, 출력, 상태 표시를 보여 주는 사용자 인터페이스를 말합니다. 사용자가 hermes 명령을 치면 프로세스가 뜨고, 대화를 끝내 창을 닫으면 그 프로세스는 그대로 소멸합니다. 수명이 한 번의 대화에 묶여 있는 셈입니다. 다만 대화 내용은 프로세스와 함께 사라지지 않습니다. 세션이 SQLite에 저장되므로, 다음에 hermes -c로 다시 켜면 직전 대화를 이어받아 복원할 수 있습니다. 프로세스는 짧게 살고 기억은 파일에 남는 구조입니다.

다음은 **Gateway**입니다. Gateway는 CLI와 정반대로, 종료하지 않고 계속 떠 있는 상주 프로세스입니다. 이렇게 배경에서 쉬지 않고 도는 프로세스를 흔히 daemon이라 부릅니다. 사람이 터미널 앞에 앉아 있지 않아도 메신저로 들어온 메시지를 받아 처리하려면 누군가 항상 깨어 있어야 하는데, 그 역할이 Gateway입니다. hermes gateway install로 서비스에 등록한 뒤 start, stop, status로 켜고 끄고 상태를 확인합니다. 여기서 한 가지 장치가 더해집니다. 프로필별 **PID** 격리입니다. PID(프로세스 식별자)란 운영체제가 실행 중인 각 프로세스에 붙이는 고유 번호를 말합니다. Hermes는 프로필마다 별도의 PID를 두어 격리합니다. 그래서 업무용과 개인용처럼 성격이 다른 인스턴스를 한 머신에서 서로 간섭 없이 동시에 띄울 수 있습니다.

세 번째는 **ACP**입니다. ACP는 코드 에디터가 에이전트와 통신하는 프로토콜을 말합니다. VS Code, Zed, JetBrains 같은 에디터가 Hermes를 자기 안에서 부릴 때 이 방식을 씁니다. 에디터가 필요할 때 Hermes를 서브프로세스로 띄우고, 둘은 표준 입출력 통로인 stdio 위에서 JSON-RPC라는 약속된 메시지 형식으로 요청과 응답을 주고받습니다. 수명은 에디터가 쥡니다. 에디터가 세션을 열면 서브프로세스가 생기고 닫으면 함께 정리됩니다.

마지막은 **Cron**입니다. Cron은 정해진 시각에 작업을 자동 실행하는 스케줄 장치입니다. 특이한 점은 별도 프로세스가 아니라 Gateway 안에 얹혀 산다는 것입니다. 상주 중인 Gateway가 일정 간격으로 시계를 확인하는데, 이 주기적 확인을 tick이라고 부릅니다. tick 때 실행할 때가 된 작업을 찾아 돌립니다. 그래서 Cron에는 독립된 수명이 없습니다. Gateway가 떠 있는 동안만 예약 작업이 도는 구조입니다.

네 모델을 하나의 표로 대비하면 다음과 같습니다.

```
모델      수명                       입력 통로            깨우는 주체
CLI       한 대화 (종료 시 소멸)     터미널 TUI           사람의 키 입력
Gateway   상주 (계속 떠 있음)        메신저 이벤트        도착한 메시지
ACP       에디터 세션에 종속         stdio / JSON-RPC     에디터
Cron      독립 수명 없음 (내장)      스케줄된 잡          Gateway의 tick
```

이제 데이터가 흐르는 경로를 개괄해 봅니다. 세부까지 파고들 필요는 없고, 두 대표 모델이 어떻게 다른지만 잡으면 충분합니다. CLI에서는 사람이 친 한 줄이 입력 처리 함수로 들어가 코어의 대화 실행을 호출합니다. 그러면 5.1.2에서 본 그 코어가 프롬프트를 조립하고, 어느 모델을 쓸지 정하고, API를 호출한 뒤 도구 실행을 반복하고, 결과를 SQLite에 저장합니다. 시작점이 사람의 키보드 한 곳뿐입니다. Gateway에서는 시작점이 바깥의 플랫폼 이벤트입니다. 메신저에서 온 이벤트가 어댑터를 거쳐 통일된 메시지 형식으로 바뀌고, 발신자가 허가된 사람인지 인증을 통과한 다음, 어느 대화에 속하는지 세션을 가려냅니다. 그 세션의 지난 기록을 불러 코어를 깨워 실행하고, 답을 다시 어댑터를 통해 원래 채널로 돌려보냅니다. 두 경로 모두 가운데에서 일하는 것은 똑같은 하나의 코어이며, 다른 것은 입력이 어디서 들어와 어디로 나가는가뿐입니다. 5.1.2의 단일 코어가 왜 값진지가 여기서 드러납니다. 코어를 하나로 두었기에 엔트리포인트만 갈아 껴 네 가지 실행 방식을 모두 지탱할 수 있습니다.

정리하면, Hermes는 하나의 코어를 CLI, Gateway, ACP, Cron이라는 네 엔트리포인트로 감쌉니다. CLI는 대화가 끝나면 소멸하지만 세션은 파일로 남아 hermes -c로 복원되고, Gateway는 프로필별 PID로 격리된 채 상주하며 메신저를 받습니다. ACP는 에디터가 stdio와 JSON-RPC로 띄우는 서브프로세스라 에디터에 수명을 맡기고, Cron은 독립 프로세스 없이 Gateway의 tick에 실려 예약 작업을 돕니다. 데이터 경로도 CLI는 사람의 입력에서, Gateway는 플랫폼 이벤트에서 출발할 뿐 가운데의 코어는 같습니다.

5.2.1에서는 이 코어가 매 턴 실행할 때 가장 먼저 하는 일, 곧 시스템 프롬프트를 캐시를 의식해 3계층으로 조립하는 PromptBuilder를 다룹니다.

이 단원을 마치면 CLI, Gateway, ACP, Cron 네 프로세스 모델의 수명 차이와 데이터가 흐르는 경로를 구별해 설명할 수 있습니다.
