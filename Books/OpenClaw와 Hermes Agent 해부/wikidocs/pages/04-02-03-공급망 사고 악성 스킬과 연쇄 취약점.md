# 4.2.3 공급망 사고: 악성 스킬과 연쇄 취약점

4.2.2에서 다룬 노출 사태는 공격이 정문으로 들어온 경우였습니다. 인증 없이 열린 게이트웨이 포트로 외부인이 직접 접속했습니다. 이 단원이 다루는 사고는 경로가 다릅니다. 공격이 문이 아니라 사용자가 스스로 설치한 부품을 타고 안으로 들어옵니다. 확장 기능을 받아 쓰는 통로 자체가 공격의 입구가 된 것입니다.

이런 유형의 공격을 부르는 이름이 있습니다. **공급망 공격**(supply chain attack)입니다. 원래는 제조업에서 나온 말로, 최종 제품을 직접 노리는 대신 그 제품에 들어가는 부품이나 원자재를 중간에 오염시키는 수법을 가리킵니다. 소프트웨어에서는 내가 만든 프로그램이 아니라 그 프로그램이 가져다 쓰는 외부 코드를 노립니다. 라이브러리나 패키지나 플러그인에 악성 코드를 심는 방식입니다. 표적은 그 부품을 받아 쓰는 수많은 사용자 전체가 됩니다. 방어하는 쪽이 아무리 자기 코드를 튼튼하게 짜도, 신뢰하고 끌어온 남의 코드가 오염되어 있으면 그 신뢰를 타고 그대로 뚫립니다.

OpenClaw에서 이 남의 코드에 해당하는 것이 스킬입니다. 3.3.1과 3.3.3에서 정리했듯 스킬은 별도로 실행되는 독립 프로그램이 아닙니다. 스킬은 세션이 시작될 때 시스템 프롬프트 안으로 들어가는 서드파티 코드입니다. 그리고 3.3.3에서 본 대로 커뮤니티가 올린 스킬을 공개 레지스트리에서 받아 설치할 수 있습니다. 남이 쓴 지시문과 코드가 내 에이전트의 프롬프트에 합쳐져 실행되는 구조이므로, 그 출처가 악의를 품으면 공급망 공격의 조건이 그대로 갖춰집니다.

실제로 그런 일이 벌어졌습니다. 2026년 초, 여러 보안 연구자들이 공개 레지스트리에 올라온 스킬들을 조사한 결과 악성 코드를 담은 것이 다수 발견되었습니다. 이들이 노린 것은 자격증명이었습니다. 3.3.2에서 보았듯 스킬은 세션 진입 시점에 필요한 환경변수와 API 키를 실행 환경으로 주입받습니다. 악성 스킬은 이 지점을 이용했습니다. 자기 코드가 도는 동안 실행 환경에 노출된 **환경변수**(environment variable)를 읽어 외부로 빼돌렸습니다. 환경변수란 프로그램이 실행될 때 그 환경에 미리 심어 두는 설정값을 말합니다. 모델 API 키나 메신저 연동 토큰, 데이터베이스 접속 정보 같은 민감한 자격증명이 흔히 여기에 담깁니다. 한 연구팀은 이런 악성 스킬을 수백 개 규모로 탐지했다고 보고했습니다. 다른 조사에서는 자격증명 탈취가 확인된 사례를 최소 열네 건 문서로 정리했습니다. 스킬을 한 번 설치하는 행위가 곧 자기 API 키를 낯선 코드에 넘겨주는 일이 될 수 있었던 것입니다.

여기서 3.3.3에서 다룬 검증 장치의 한계가 드러납니다. 레지스트리는 커뮤니티 스킬을 활성화하기 전에 신뢰 봉투라 부르는 검증을 거칩니다. 그런데 이 검증이 확인하는 것은 스킬의 출처, 곧 이것이 특정 게시자가 올린 물건이 맞고 도중에 변조되지 않았다는 사실입니다. 출처가 진짜라는 것과 내용이 무해하다는 것은 전혀 다른 문제입니다. 진짜 게시자가 처음부터 악성 코드를 담아 올렸다면, 서명은 완벽하게 통과합니다. 봉투가 진짜여도 그 안에 든 편지가 협박장일 수 있습니다. 출처 확인은 배송 사고를 막을 뿐, 발신자 자신이 악의를 품은 경우는 걸러 내지 못합니다. 검증이 통과했다는 사실을 안전하다는 뜻으로 읽으면 안 되는 이유가 여기 있습니다.

공급망 위험은 악성 스킬 한 종류로 끝나지 않았습니다. 비슷한 시기에 보안 연구자들은 게이트웨이가 외부와 주고받는 연결 통로에서 여섯 건의 취약점을 함께 공개했습니다. 대표적인 두 부류가 있습니다. 하나는 **SSRF**(Server-Side Request Forgery), 곧 서버측 요청 위조입니다. 에이전트가 URL을 받아 그 주소로 요청을 보내는 기능을 가졌다고 합시다. 공격자가 그 주소를 조작해, 원래 닿으면 안 되는 내부 자원을 서버가 대신 긁어 오게 만드는 수법입니다. 요청을 사용자가 아니라 서버가 대신 쏘기 때문에, 외부에서 직접은 못 가는 내부망 주소에도 서버의 권한으로 도달합니다. 다른 하나는 **webhook 인증 누락**입니다. webhook은 외부 서비스가 사건이 생겼을 때 에이전트의 특정 주소로 알림을 밀어 넣는 창구입니다. 그 창구가 발신자를 확인하지 않으면, 아무나 가짜 알림을 집어넣어 에이전트를 원치 않는 동작으로 유도할 수 있습니다. 이 여섯 건은 심각도로 매기면 대체로 중상 수준이었고, 공개 시점에는 모두 수정된 상태였습니다.

같은 흐름에서 자격증명이 인증 없이 통째로 새어 나간 사례도 나왔습니다. 보안 연구자들은 저장된 API 토큰을 내려받는 내부 경로가 아무 인증 없이 열려 있는 인스턴스를 만 칠천 개 넘게 찾아냈습니다. 정상이라면 자기 자격증명을 확인한 접속자에게만 열려야 할 통로가, 주소만 알면 누구에게나 저장 토큰을 그대로 넘겨주고 있었던 것입니다. 앞의 악성 스킬이 키를 훔치려고 코드를 심는 능동적 공격이었다면, 이쪽은 훔칠 필요도 없이 키가 밖에서 그냥 읽히던 수동적 노출입니다. 자격증명이 새는 길이 한 방향만 있는 게 아니라는 점을 보여 줍니다.

이 사고들이 왜 하필 에이전트 확장 생태계에서 크게 번졌는지를 구조로 정리하면 다음과 같습니다.

```
확장이 쉬움     스킬 = 마크다운 한 장, 누구나 게시·설치
     │
자격증명 근접   스킬 실행 환경에 API 키·토큰이 그대로 주입됨
     │
검증의 한계     신뢰 봉투는 출처만 확인, 무해성은 보증 못 함
     │
   ▼
설치 한 번 = 낯선 코드에 자기 금고 열쇠를 건네는 일
```

정리하면, 스킬은 시스템 프롬프트에 합쳐지는 서드파티 코드이고 누구나 올리고 받아 쓰는 공개 레지스트리로 유통됩니다. 그래서 에이전트 확장 생태계는 그 자체로 공급망 공격면이 됩니다. 실제로 2026년 초 보안 연구자들은 환경변수에 담긴 API 키와 토큰을 훔치는 악성 스킬을 다수 발견했습니다. 같은 시기에 게이트웨이의 SSRF와 webhook 인증 누락을 포함한 연쇄 취약점 여섯 건이 공개되었습니다. 저장된 토큰이 인증 없이 읽히던 인스턴스도 만 칠천 개 넘게 확인되었습니다. 핵심은 검증의 성격에 있습니다. 신뢰 봉투는 스킬의 출처가 진짜인지는 확인하지만 그 내용이 무해한지는 보증하지 못하므로, 검증 통과를 안전의 증명으로 삼을 수 없습니다.

4.3.1에서는 지금까지 살펴본 설계와 사고를 뒤로하고 실제 운영으로 넘어가, OpenClaw를 머신에 설치하고 상주 데몬으로 등록하는 과정을 다룹니다.

이 단원을 마치면 악성 스킬과 연쇄 취약점 사례를 통해 에이전트 확장 생태계가 공급망 공격면이 되는 구조를 짚고, 출처 검증이 무해성을 보증하지 못하는 한계까지 설명할 수 있습니다.
