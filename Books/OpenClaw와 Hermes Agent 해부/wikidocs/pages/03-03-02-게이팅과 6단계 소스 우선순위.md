# 3.3.2 게이팅과 6단계 소스 우선순위

macOS에서 화면을 캡처해 전송하는 스킬을 하나 상상해 보겠습니다. 이 스킬의 본문은 macOS에만 있는 screencapture 명령을 부르도록 쓰여 있습니다. 그런데 같은 스킬이 Linux 서버에 설치된 OpenClaw에 그대로 로드된다고 합시다. 3.3.1에서 본 대로 스킬은 세션 시작 시 시스템 프롬프트에 들어가는 문서이므로, 모델은 자신에게 화면 캡처 능력이 있다고 믿게 됩니다. 사용자가 화면을 찍어 달라고 하면 모델은 존재하지 않는 명령을 호출하고, 실행은 실패하고, 실패를 수습하는 턴이 뒤따릅니다. 손해는 그 전에 이미 시작됩니다. 스킬은 프롬프트에 실리는 것만으로 하나당 24토큰 안팎의 비용을 만들기 때문에, 이 환경에서 쓸 수 없는 스킬은 매 세션 토큰부터 낭비합니다.

그래서 OpenClaw는 스킬을 프롬프트에 넣기 전에 거릅니다. 이 거름 장치가 **게이팅**(gating)입니다. 게이팅이란 미리 선언된 조건을 검사해, 통과한 대상만 다음 단계로 보내는 문을 말합니다. 스킬 작성자는 SKILL.md frontmatter의 metadata.openclaw 블록에 자기 스킬이 동작할 수 있는 환경 조건을 선언합니다. 대표 조건은 셋입니다.

```
metadata.openclaw 게이트 (하나라도 어긋나면 로드 제외)

os:            "darwin"           → 이 운영체제에서만 로드
requires.bins: ["screencapture"]  → PATH에 이 실행 파일이 있어야 로드
requires.env:  ["GRAB_TOKEN"]     → 이 환경변수가 설정되어야 로드
```

os는 스킬이 로드될 운영체제를 제한하며 darwin(macOS), linux, win32 값을 씁니다. requires.bins는 PATH에서 찾을 수 있어야 하는 실행 파일 목록이고, requires.env는 값이 채워져 있어야 하는 환경변수 목록입니다. 로더는 세션 시작 시 이 조건과 설정의 허용 목록을 함께 검사해, 어긋난 스킬을 유효 목록에서 제외합니다. 제외된 스킬은 프롬프트에 아예 실리지 않으므로 모델 입장에서는 처음부터 없는 능력입니다.

이 방식을 **capability gating**(능력 게이팅)이라고 부릅니다. 발상의 핵심은 환경이 능력을 결정한다는 것입니다. 실패할 호출을 일단 시도하고 에러를 수습하는 대신, 환경 검사에서 떨어진 능력을 모델이 아는 목록에서 미리 지웁니다. 같은 스킬 묶음을 macOS 노트북과 Linux 서버에 복사해 두어도 각 머신은 자기 환경에서 실제로 동작하는 부분집합만 로드합니다. 게이팅은 오동작을 막는 정확성 장치이면서, 앞서 본 스킬당 토큰 비용을 줄이는 예산 장치이기도 합니다.

게이팅이 "어떤 스킬을 로드할 것인가"의 절반이라면 나머지 절반은 "어디서 로드할 것인가"입니다. OpenClaw는 스킬을 여섯 곳에서 찾고, 그 사이에 엄격한 우선순위를 둡니다.

```
우선순위 높음
 (1) 워크스페이스        <workspace>/skills
   └ (2) 프로젝트        <workspace>/.agents/skills
       └ (3) 개인        ~/.agents/skills
           └ (4) 관리형   ~/.openclaw/skills
               └ (5) 번들  설치본에 동봉된 기본 스킬
                   └ (6) 추가 디렉토리와 플러그인 스킬
우선순위 낮음                 * 같은 이름이면 위가 이긴다
```

규칙은 하나입니다. 같은 이름의 스킬이 여러 소스에 있으면 상위 소스가 이깁니다. 최상위인 워크스페이스는 에이전트가 일하는 공간에 딸린 skills 디렉토리이고, 그 아래로 프로젝트 전용 층과 사용자 홈의 개인 층이 이어집니다. 관리형 층은 스킬 설치 명령이 전역 설치분을 두는 곳이며, 번들 층은 OpenClaw 설치본에 처음부터 동봉된 기본 스킬입니다. 마지막 층은 설정으로 지정한 추가 디렉토리와 플러그인이 제공하는 스킬입니다.

이 배치가 만드는 효과가 **layered override**(계층적 덮어쓰기)입니다. 배포본이 아래층에 기본값을 깔고, 사용자에게 가까운 층일수록 그 위에 얹혀 같은 이름을 가립니다. 번들 스킬의 동작이 마음에 들지 않을 때 원본을 고칠 필요가 없습니다. 같은 이름의 SKILL.md를 워크스페이스 skills 디렉토리에 두는 것만으로 그 에이전트는 사용자 버전을 씁니다. 이후 업데이트로 번들 스킬이 새 버전으로 바뀌어도 워크스페이스 사본이 여전히 최상위이므로 사용자 정의는 유지됩니다. 배포본 갱신과 사용자 수정이 충돌 없이 공존하는 구조입니다.

마지막으로 이 해석이 언제 일어나는지입니다. 게이팅 검사와 우선순위 병합은 세션이 시작될 때 수행되고, 그 결과인 유효 스킬 목록은 **스냅샷**으로 고정됩니다. 스냅샷이란 그 시점의 상태를 찍어 둔 사본을 말하며, 진행 중인 세션은 이 사본만 바라봅니다. 세션 시작 시 로더는 유효 목록을 해석하고, 스킬이 요구하는 환경변수와 API 키를 프로세스 환경에 주입하고, 목록을 압축된 XML 블록으로 컴파일해 시스템 프롬프트에 넣습니다. 실행이 끝나면 주입했던 환경은 원래대로 복원됩니다. 스냅샷이 갱신되는 시점은 두 가지뿐입니다. SKILL.md 변경이 감지될 때, 그리고 새 세션이 열릴 때입니다. 뒤집어 말하면 그 전까지는 스킬을 설치하거나 고쳐도 효력이 없습니다. 스킬을 수정했는데 에이전트가 그대로라면, 대개 아직 스냅샷이 갱신되지 않은 것입니다.

정리하면, OpenClaw의 스킬 로더는 두 가지 질문에 답합니다. metadata.openclaw의 os, requires.bins, requires.env 게이팅은 "이 환경에서 동작하는가"를 검사해 통과한 스킬만 프롬프트에 싣는 capability gating을 구현합니다. 워크스페이스를 최상위로 하는 6단계 소스 우선순위는 "같은 이름이 겹치면 누가 이기는가"를 layered override로 정리해, 사용자 정의가 배포본을 항상 덮어쓰게 합니다. 이렇게 해석된 목록은 세션 시작 시 스냅샷으로 고정되고, 변경 감지나 새 세션에서만 갱신됩니다.

3.3.3에서는 이 스킬들이 유통되는 공개 레지스트리 ClawHub와, 그 생태계에서 실제로 벌어진 신뢰 문제를 다룹니다.

이 단원을 마치면 OS·바이너리·환경변수 게이팅으로 환경에 맞는 스킬만 로드되는 과정과, 6단계 소스 우선순위에서 사용자 정의가 배포본을 덮어쓰는 구조를 설명할 수 있습니다.
