# 2.2.2 WebSocket 제어 평면과 클라이언트들

2.2.1에서는 게이트웨이가 모든 연결을 소유하는 단일 상주 프로세스라는 선택을 살펴봤습니다. 그런데 배경에서 조용히 도는 프로세스는 그 자체로는 만질 수 없습니다. 지금 잘 돌고 있는지 확인하고, 메시지를 대신 보내게 하고, 에이전트 실행을 걸어 줄 통로가 따로 필요합니다. 이 단원은 그 통로가 되는 클라이언트들과 그들이 쓰는 공통 언어를 다룹니다.

먼저 용어부터 짚습니다. 시스템 안에는 성격이 다른 두 흐름이 있습니다. 하나는 WhatsApp이나 Telegram으로 오가는 대화처럼 시스템이 처리하는 내용물의 흐름입니다. 다른 하나는 그 시스템 자체를 살피고 지시하는 운영 명령의 흐름입니다. 이 두 번째 길을 **제어 평면**(control plane)이라고 부릅니다. OpenClaw에서 제어 평면으로 들어오는 쪽은 Web Control UI, CLI, macOS 앱, 그리고 자동화(automations)입니다. 넷의 정체는 같습니다. 게이트웨이가 127.0.0.1:18789에 열어 둔 WebSocket 서버에 접속하는 클라이언트입니다.

눈여겨볼 점은 어떤 화면도 특별 대우를 받지 않는다는 사실입니다. 웹 브라우저로 여는 Web Control UI의 실체는 /chat에서 서빙되는 Lit 웹컴포넌트 기반 단일 페이지 앱입니다. 그래도 게이트웨이가 보기에는 CLI, macOS 앱과 똑같은 클라이언트 하나입니다. 상태는 게이트웨이 한 곳에만 있고 화면들은 그것을 비추는 창이므로, 웹에서 보든 터미널에서 보든 같은 상태가 보입니다.

클라이언트가 게이트웨이에 말을 거는 방식은 두 갈래입니다. 첫째는 **RPC**(remote procedure call, 원격 프로시저 호출)입니다. 프로그램이 자기 안의 함수를 부르듯, 네트워크 건너편 프로세스에 이름 붙은 작업을 요청하고 결과를 돌려받는 통신 방식을 RPC라고 부릅니다. 요청은 늘 클라이언트가, 응답은 서버가 맡는 클라이언트-서버 구조입니다. 게이트웨이가 살아 있는지 묻는 health, 메신저로 메시지를 내보내는 send, 에이전트 실행을 시작시키는 agent가 대표적인 메서드입니다. 이 밖에 status, system-presence 같은 메서드도 있습니다. CLI에서 명령을 치면 결국 RPC 요청 하나가 만들어져 게이트웨이로 날아가는 셈입니다.

둘째는 이벤트 구독입니다. RPC는 클라이언트가 물어야 답이 오는 방식이라, 게이트웨이 쪽에서 그때그때 벌어지는 일을 알리는 데는 맞지 않습니다. 그래서 반대 방향으로는 **발행-구독**(pub/sub) 방식이 쓰입니다. 관심 있는 사건에 구독을 걸어 두면, 사건이 생길 때마다 게이트웨이가 구독자들에게 알림을 밀어 줍니다. 주기적으로 오는 tick, 접속 상태 변화를 알리는 presence, 에이전트 실행 소식을 나르는 agent, 종료를 알리는 shutdown 같은 이벤트가 이 길로 흐릅니다.

두 갈래를 한 연결에서 함께 굴리려면 통신로가 양방향이어야 합니다. HTTP는 클라이언트가 물어야 서버가 답하는 반쪽 대화라서, 서버가 먼저 말을 거는 이벤트 푸시와 맞지 않습니다. 게이트웨이가 WebSocket을 쓰는 이유가 여기 있습니다. WebSocket은 한 번 연결을 열면 양쪽이 아무 때나 상대에게 데이터를 보낼 수 있는 **전이중**(full-duplex) 통신입니다. 그래서 스트리밍 응답과 실시간 이벤트를 연결 하나로 감당합니다.

그 연결 위로 실제 오가는 데이터의 생김새를 정한 약속이 **와이어 프로토콜**(wire protocol)입니다. OpenClaw의 와이어 프로토콜은 JSON 텍스트 **프레임**입니다. 프레임은 WebSocket에서 한 번에 전송되는 메시지 한 덩어리를 가리킵니다. 각 프레임은 type 필드로 자기 종류를 밝히며, 종류는 req, res, event 셋입니다.

```
클라이언트 → 게이트웨이 (RPC 요청)
{ "type": "req", "id": 7, "method": "health", "params": {} }

게이트웨이 → 클라이언트 (그 요청의 응답)
{ "type": "res", "id": 7, "ok": true, "payload": { ... } }

게이트웨이 → 구독 중인 클라이언트들 (요청 없이 푸시)
{ "type": "event", "event": "presence", "payload": { ... }, "seq": 12 }
```

req 프레임은 method에 부를 작업 이름을, params에 인자를 담습니다. id는 요청마다 붙이는 식별 번호입니다. 전이중 통신에서는 여러 요청이 동시에 진행되고 응답 순서도 보장되지 않으므로, 어느 응답이 어느 요청의 답인지 이 번호로 짝을 맞춥니다. res 프레임은 같은 id를 돌려주고, ok로 성공 여부를 밝힙니다. 성공이면 payload에 결과를, 실패면 error에 사유를 담습니다. event 프레임은 event 필드에 이벤트 이름을, payload에 내용을 담습니다. 필요하면 순서를 따질 일련번호 seq나 상태 버전 stateVersion이 덧붙습니다.

남은 문제는 인증입니다. 게이트웨이는 기본적으로 인증을 요구하고, 유효한 인증 경로가 없으면 WebSocket 연결을 거부합니다. 연결의 첫 프레임은 반드시 connect 프레임이어야 하며, 모든 연결은 **논스**(nonce) 챌린지 서명을 통과해야 합니다. 논스는 한 번 쓰고 버리는 임의 값입니다. 게이트웨이가 연결마다 새 논스를 제시하면, 클라이언트가 자기 자격 정보로 서명해 돌려줍니다. 논스가 매번 달라지므로, 과거에 오간 서명을 가로채 다시 보내는 재전송 공격이 통하지 않습니다. 서명 페이로드 v3는 플랫폼과 기기 계열 정보까지 서명에 묶어, 다른 종류의 기기에서 만든 서명을 끌어다 쓰는 길도 막습니다.

두 갈래는 실제로 맞물려 움직입니다. 에이전트 실행을 거는 agent RPC는 완료를 기다리지 않고, 접수했다는 응답만 즉시 돌려줍니다. 이후 진행 상황은 이벤트로 흘러옵니다. 몇 분씩 걸릴 수 있는 작업이라 접수는 RPC로, 경과는 구독으로 나눈 것입니다. 이 실행의 안쪽은 2.3에서 해부합니다.

정리하면, Web Control UI와 CLI와 macOS 앱은 모두 게이트웨이의 WebSocket 서버에 접속하는 제어 평면 클라이언트입니다. 게이트웨이와의 대화는 health, send, agent 같은 RPC 요청과 tick, presence 같은 이벤트 구독의 두 갈래로 이루어집니다. 그 아래에는 req, res, event 세 프레임으로 된 JSON 와이어 프로토콜이 있고, 모든 연결은 논스 챌린지 서명이라는 인증 관문을 지나야 합니다. 상태를 게이트웨이 한 곳에 두고 모든 화면을 대등한 클라이언트로 만드는 이 구조가, 어느 창에서 보든 같은 에이전트를 보게 하는 바탕입니다.

2.2.3에서는 게이트웨이의 반대쪽 면, 곧 WhatsApp과 Telegram 같은 메신저를 붙이는 채널 어댑터의 구조를 살펴봅니다.

이 단원을 마치면 Web Control UI, CLI, macOS 앱이 게이트웨이의 WebSocket 클라이언트로 동작하는 구조와, req, res, event 프레임으로 이루어진 와이어 프로토콜의 형태를 설명할 수 있습니다.
