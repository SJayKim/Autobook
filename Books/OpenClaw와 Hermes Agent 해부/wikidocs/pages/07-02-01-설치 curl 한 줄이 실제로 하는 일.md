# 7.2.1 설치: curl 한 줄이 실제로 하는 일

지금까지는 Hermes Agent의 설계를 안에서 뜯어봤습니다. 이제 Phase 7에서는 그 설계를 실제 머신에 올리고 운영하는 이야기로 넘어갑니다. 그 첫걸음인 설치는 공식 안내상 명령 한 줄입니다.

```
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
```

이 한 줄이 뒤에서 무슨 일을 대신 처리하는지 풀어 보는 것이 이 단원입니다.

이 방식을 흔히 **curl 파이프 배시**(curl | bash)라고 부릅니다. curl은 지정한 주소에서 파일을 내려받는 도구이고, 오른쪽의 bash는 셸입니다. 가운데 세로줄은 왼쪽의 출력을 오른쪽의 입력으로 넘기라는 뜻입니다. 그래서 이 한 줄은 install.sh 설치 스크립트를 서버에서 받아, 파일로 저장하지 않고 곧바로 셸에 흘려보내 실행합니다.

이제 install.sh가 실제로 하는 일을 봅니다. 스크립트는 크게 네 가지를 순서대로 처리합니다.

첫째, 에이전트가 돌아가는 데 필요한 실행 환경을 자동으로 깝니다. Hermes는 파이썬으로 만들어져 있어 Python 3.11이 필요합니다. 이때 파이썬을 시스템에 직접 설치하는 대신 **uv**라는 도구를 경유합니다. uv는 파이썬 버전과 패키지를 빠르게 설치하고 격리해 주는 도구입니다. 시스템의 기존 파이썬을 건드리지 않고 Hermes 전용 파이썬을 따로 마련하려는 선택입니다. 파이썬과 함께 Node.js v22, ripgrep, ffmpeg도 설치됩니다. Node는 일부 연동에, ripgrep은 세션 기록 검색에, ffmpeg는 음성이나 영상 미디어를 다루는 데 쓰입니다. 사용자가 이 넷을 미리 준비하지 않아도 스크립트가 대신 챙깁니다.

둘째, 실제 코드를 내려받습니다. 스크립트는 Hermes 저장소를 사용자 홈 아래 ~/.hermes/hermes-agent/ 경로로 클론합니다. 클론이란 소스 코드 저장소를 통째로 복제해 오는 일입니다. 그래서 사실상 유일한 전제 조건이 git이며, 리눅스에서는 curl과 xz-utils 정도가 더 필요합니다.

셋째, 내려받은 코드를 위한 격리된 실행 공간을 만듭니다. 이것을 **venv**(virtual environment, 가상 환경)라고 부릅니다. venv는 이 프로젝트만 쓰는 파이썬 라이브러리를 시스템 전역과 분리해 담는 폴더로, 다른 프로그램과 뒤섞이지 않도록 자기만의 방을 얻는 셈입니다.

넷째, 어디서든 hermes라고 부를 수 있게 실행 경로를 잇습니다. 스크립트는 터미널이 기본으로 찾는 ~/.local/bin/hermes 위치에 **심링크**(symlink, 심볼릭 링크)를 만듭니다. 심링크란 실제 파일을 가리키는 바로가기입니다. 저장소 깊숙한 실행 파일 대신 이 바로가기 덕분에 hermes만 쳐도 실행됩니다.

설치가 끝나면 사용자 홈에 생기는 ~/.hermes가 Hermes의 모든 상태가 모이는 집입니다. 앞 단원들에서 부품으로 나눠 봤던 것들이 실제로는 이 폴더 아래 파일과 디렉토리로 존재합니다. 그 구조는 다음과 같습니다.

```
~/.hermes/
    ├─ hermes-agent/   클론한 코드
    ├─ memories/       MEMORY.md, USER.md (상주 기억)
    ├─ state.db        모든 세션 원문 (SQLite)
    ├─ skills/         자가 생성 스킬
    ├─ plugins/        확장
    └─ logs/           실행 기록
```

이 구조를 알아 두면 백업이나 점검 때 어디를 봐야 할지 분명합니다.

코드를 내려받았다고 바로 쓸 수 있는 것은 아닙니다. 어떤 모델을 쓸지 같은 초기 설정이 남습니다. 이 온보딩을 맡는 것이 setup 마법사입니다.

```
hermes setup
```

이 **마법사**(setup wizard)는 질문에 답해 가며 기본 구성을 잡아 주는 대화형 안내입니다. 여기서 눈여겨볼 동작은 마법사가 같은 머신에 OpenClaw의 흔적, 즉 ~/.openclaw 디렉토리가 있는지 살핀다는 점입니다. Hermes는 OpenClaw의 fork가 아니라 별개로 만들어진 시스템입니다(1.2.2). 그런데도 그 자산이 감지되면 설정과 기억, 스킬을 옮겨 오는 마이그레이션을 제안합니다.

여기까지가 대화형으로 쓰는 준비입니다. Hermes를 개인 비서처럼 항상 켜 두려면 메신저를 받아 주는 상주 프로세스가 따로 돌아야 합니다. 프로세스가 여러 모양으로 나뉜다는 점은 5.1.3에서 짚었는데, 그중 상주 데몬이 게이트웨이입니다. 이를 시스템 서비스로 등록하는 명령이 다음입니다.

```
hermes gateway install
```

install은 게이트웨이를 운영체제의 서비스로 등록해 로그인 시 자동으로 뜨거나 백그라운드에서 계속 살아 있게 합니다. 등록한 서비스를 켜는 것은 hermes gateway start이고, 잘 돌고 있는지 확인하는 명령이 다음입니다.

```
hermes gateway status
```

status는 게이트웨이가 켜져 있는지, 어느 프로세스로 도는지 보여 줍니다.

마지막으로 지원 범위를 짚겠습니다. 위 흐름은 유닉스 계열 셸을 전제로 합니다. 그래서 Linux와 macOS에서는 앞의 curl 한 줄이 그대로 통합니다. 윈도우 안에서 리눅스를 돌리는 WSL2, 안드로이드의 터미널 환경 Termux도 마찬가지입니다. 다만 윈도우 네이티브는 bash가 기본이 아니라서, PowerShell용 스크립트를 받아 실행하는 별도 진입점을 씁니다. 참고로 OpenClaw 설치(4.3.1)도 겉모습은 비슷한 한 줄이지만, Node.js 기반이라 준비하는 실행 환경이 다릅니다.

지원 환경과 설치 진입점을 정리하면 다음과 같습니다.

| 지원 환경 | 설치 진입점 |
|---|---|
| Linux / macOS / WSL2 / Termux(Android) | curl 파이프 배시 (install.sh) |
| Windows 네이티브 | PowerShell 스크립트 (install.ps1) |

정리하면, curl 한 줄은 uv 경유 실행 환경 설치, 저장소 클론, venv 구성, 심링크 연결의 네 단계를 밟습니다. 그 결과 ~/.hermes에 코드와 기억, 세션, 스킬이 모입니다. 이어서 hermes setup이 초기 구성을 잡고 OpenClaw 자산이 있으면 마이그레이션을 제안합니다. gateway install과 start, status는 상주 프로세스를 등록하고 켜고 확인합니다. 지원 환경은 Linux, macOS, WSL2, Termux가 같은 curl 방식을 쓰고 윈도우 네이티브만 PowerShell로 갈립니다.

다음 7.2.2에서는 이렇게 올린 시스템을 운영할 때 챙겨야 할 점검 항목과, 미리 알아 두지 않으면 발을 헛디디기 쉬운 함정들을 다룹니다.

이 단원을 마치면 install.sh가 수행하는 단계와 ~/.hermes 구조, setup 마법사의 역할을 설명하고, 지원 OS별 설치 방식의 차이를 구별할 수 있습니다.
