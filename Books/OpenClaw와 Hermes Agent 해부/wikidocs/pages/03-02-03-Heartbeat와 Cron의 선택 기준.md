# 3.2.3 Heartbeat와 Cron의 선택 기준

3.2.1과 3.2.2에서 시키지 않아도 움직이는 두 장치, heartbeat와 cron을 각각 살펴보았습니다. 이번 단원의 질문은 선택입니다. 주기적으로 시킬 일이 하나 생겼을 때 이것을 heartbeat 체크리스트에 넣을지, cron 작업으로 걸지 판단하는 기준을 세웁니다. 결론을 먼저 말하면 기준은 주기의 길이가 아니라 문맥입니다. 그 일을 제대로 하는 데 지금까지의 대화가 필요한가, 아니면 매번 깨끗하게 시작해야 하는가를 물으면 답이 갈립니다.

먼저 heartbeat 쪽을 정리합니다. heartbeat 턴은 메인 대화 세션 안에서 돕니다. 에이전트는 사용자와 쌓아 온 대화 흐름을 아는 상태로 점검에 들어갑니다. 어제 부탁받은 일이 무엇이었는지, 어떤 작업이 진행 중인지, 사용자가 어떤 답을 기다리는지가 모두 시야에 있습니다. 그래서 heartbeat가 맡기에 알맞은 일은 판단이 대화에 얽혀 있는 점검입니다. 메일함을 살피다 진행 중인 계약 건의 회신을 발견하면 바로 알리고, 알릴 것이 없으면 HEARTBEAT_OK를 돌려주고 조용히 물러납니다. 알릴지 말지를 가르는 근거 자체가 대화 맥락이므로, 이 일은 문맥 없는 실행으로 대체할 수 없습니다.

대신 heartbeat는 비용을 치릅니다. 메인 세션에서 도는 턴은 그 세션에 쌓인 문맥을 입력으로 함께 싣습니다. 대화가 길어질수록 점검 한 번의 입력이 무거워지고, 기본 30분 간격이면 하루 수십 번 반복됩니다. OpenClaw가 heartbeat를 최대 토큰 소비원으로 지목하는 이유입니다. 그래서 HEARTBEAT.md만 남기고 문맥을 줄이는 lightContext, 점검 시간대를 제한하는 activeHours 같은 조정 장치가 여럿 있고, 인증 방식에 따라 기본 주기를 1시간으로 완화하기도 합니다. 대화 문맥은 heartbeat의 가치이면서 동시에 비용의 원천입니다.

이제 cron 쪽을 정리합니다. cron 작업은 게이트웨이 안의 스케줄러가 정해진 시각에 시작하며, 기본값은 격리 세션입니다. 실행할 때마다 새 세션이 만들어지므로 지난 대화가 전혀 실리지 않습니다. 이 깨끗한 시작은 결핍이 아니라 보장입니다. 매주 월요일 9시에 밤사이 소식을 요약하는 브리핑을 떠올리면, 이 작업은 지난주에 무슨 대화를 했든 같은 품질로 나와야 합니다. 지난 대화가 섞이면 오히려 결과가 흔들립니다. 해야 할 일이 작업 정의 안에 완결되어 있는 무인 배치가 cron의 자리입니다.

문맥이 없다는 조건은 실행 방식도 규정합니다. 격리 실행에는 사람이 없으므로 에이전트가 중간에 되물을 수 없습니다. 그래서 OpenClaw는 격리 실행의 최종 응답 자체가 산출물이어야 한다는 원칙을 둡니다. 이것이 **산출물 중심 실행**입니다. 작업을 만들 때 무엇을 만들지 지시하고, 완성된 결과만 announce나 webhook 같은 전달 방식으로 받아 봅니다. 비용 특성도 heartbeat와 대칭입니다. 매번 최소 문맥으로 시작하므로 실행 한 번의 비용이 대화가 얼마나 길었는지와 무관하게 일정합니다.

그러면 왜 둘을 하나로 합치지 않았을까요. 합쳐 보면 답이 보입니다. 모든 주기 작업을 메인 세션에서 돌리면 무인 배치까지 긴 대화 문맥을 싣게 되어 비용이 불어나고, 실행 기록이 대화에 남아 다음 응답까지 흐립니다. 반대로 모든 주기 작업을 격리 세션으로 돌리면 비용은 싸지지만, 대화 맥락이 있어야 하는 점검을 할 수 없게 됩니다. 어느 쪽으로 합쳐도 비용 폭증과 문맥 오염 중 하나 이상을 떠안습니다. 그래서 OpenClaw는 시키지 않아도 움직이는 성질을 만드는 장치를 문맥 축을 기준으로 둘로 나눴습니다. 이 설계 관점이 **능동성 이원화**입니다.

두 장치를 축별로 놓으면 다음과 같습니다.

| 축 | Heartbeat | Cron |
|---|---|---|
| 실행 위치 | 메인 대화 세션 안의 주기적 턴 | 게이트웨이 스케줄러의 독립 작업 |
| 문맥 | 기존 대화 문맥 유지 | 기본 격리, 실행마다 새 세션 |
| 주기 | 고정 간격(기본 30분) | 예약 시각, 반복 간격, cron 표현식, 종료 트리거 |
| 결과 전달 | 알릴 것 없으면 HEARTBEAT_OK로 침묵 | announce, webhook, none 중 선택 |
| 존재 이유 | 대화 맥락을 아는 배경 점검 | 사람 없이 도는 무인 배치 |

이 표는 판별 질문 하나로 압축됩니다. 이 작업의 판단에 지난 대화가 필요한가입니다. 필요하면 heartbeat입니다. 메일함과 캘린더 감시, 진행 중 작업의 후속 확인처럼 알릴지 말지를 맥락으로 가려야 하는 점검이 여기 속합니다. 산출물만 있으면 된다면 cron입니다. 주간 브리핑, 정기 보고서 생성, 로그 정리처럼 결과물이 곧 목적인 배치가 여기 속합니다. 같은 주기 작업이라도 문맥 의존 여부에 따라 자리가 다릅니다.

정리하면, heartbeat는 대화 문맥을 유지한 채 도는 주기 점검이고 cron은 격리 세션에서 깨끗하게 시작하는 무인 배치입니다. 전자는 문맥이 가치이자 비용이고, 후자는 문맥이 없어 비용이 일정하며 최종 응답이 곧 산출물이어야 합니다. 둘을 하나로 합치면 비용 폭증과 문맥 오염이 함께 오기 때문에, OpenClaw는 능동성을 문맥 축으로 이원화했습니다. 선택 기준은 주기의 길이가 아니라 그 작업의 판단에 지난 대화가 필요한가입니다.

3.2.4에서는 세 번째 능동 실행 장치인 sub-agent가 격리 위임과 폭주 방지를 어떻게 구현하는지 살펴봅니다.

이 단원을 마치면 주기 작업 하나를 놓고 문맥 필요 여부와 비용 특성을 근거로 heartbeat와 cron 중 무엇에 맡길지 판단하고, 두 장치를 합치지 않은 설계 이유를 설명할 수 있습니다.
