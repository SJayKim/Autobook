# 5.3.1 Gateway와 MessageEvent 정규화

5.1.3에서 Hermes가 같은 코어를 네 가지 엔트리포인트로 감싼다는 것을 보았습니다. 그 넷 가운데 메신저를 받아 상주하는 것이 Gateway였습니다. 이 단원에서는 그 Gateway 안으로 들어가, 성격이 제각각인 메신저들이 어떻게 하나의 코어로 모여 처리되는지를 봅니다. 열쇠는 이벤트를 공통 형식으로 바꾸는 정규화라는 개념 하나입니다.

먼저 문제부터 분명히 합니다. Hermes는 24개 플랫폼을 지원합니다. Telegram, Discord, Slack, WhatsApp, Signal, 이메일처럼 이름은 익숙하지만 서로 다른 회사가 다른 규칙으로 운영하는 서비스들입니다. 문제는 이들이 새 메시지가 왔다는 사실을 알리는 방식이 저마다 다르다는 데 있습니다. Telegram이 보내는 이벤트의 필드 이름과 구조는 Slack의 것과 같지 않고, 이메일이 도착하는 형태는 또 다릅니다. 코어인 AIAgent가 이 스물네 가지 형식을 전부 알아야 한다면, 플랫폼이 하나 늘 때마다 코어를 고쳐야 합니다. 5.1.2에서 본 코어는 플랫폼을 모른다는 설계가 무너지는 것입니다.

여기서 등장하는 부품이 **어댑터**입니다. 어댑터란 플랫폼마다 하나씩 서서 그 플랫폼의 고유한 이벤트를 코어가 이해하는 형태로 옮겨 주는 얇은 변환 계층을 말합니다. Hermes에는 플랫폼 수만큼의 어댑터가 있습니다. Telegram 어댑터는 Telegram의 언어를, Slack 어댑터는 Slack의 언어를 압니다. 각 어댑터가 자기 플랫폼의 방언을 책임지고 흡수하므로, 방언을 아는 지식은 어댑터 안에만 머물고 코어까지 번지지 않습니다.

그렇다면 어댑터는 무엇으로 변환할까요. 바로 **MessageEvent**입니다. MessageEvent란 플랫폼이 무엇이든 상관없이 하나의 메시지 사건을 똑같은 모양으로 담아내는 공통 형식입니다. 누가 보냈는가, 어느 대화에 속하는가, 내용은 무엇인가처럼 코어가 판단에 필요한 정보만 정해진 자리에 채워 넣습니다. Telegram에서 온 메시지든 이메일로 온 메시지든, 어댑터를 통과하고 나면 똑같은 MessageEvent 한 장이 됩니다. 이렇게 서로 다른 입력을 하나의 통일된 형식으로 바꾸는 일을 **정규화**라고 부릅니다.

정규화가 왜 값진지는 그 뒤를 보면 드러납니다. 정규화를 마친 다음부터는 어느 플랫폼에서 왔는지를 아무도 신경 쓸 필요가 없습니다. Gateway는 MessageEvent 한 종류만 라우팅하면 되고, 코어는 MessageEvent 한 종류만 처리하면 됩니다. 24개의 다름이 어댑터라는 문턱에서 전부 사라지고, 문턱 안쪽은 하나의 세계가 됩니다. 흐름으로 그리면 다음과 같습니다.

```
[Telegram]  [Slack]  [이메일]  [Signal]   ...  (총 24개 플랫폼)
    │          │        │         │
 어댑터      어댑터    어댑터     어댑터        ← 네이티브 이벤트를
    └──────────┴───┬────┴─────────┘             MessageEvent로 정규화
                   ▼
       동일한 MessageEvent 한 종류
                   ▼
        Gateway (단일 상주 daemon)
     인증 → 세션 해석 → 코어 실행 → 응답 배달
                   ▼
       원래 온 채널로 어댑터가 되돌림
```

이 그림 가운데의 Gateway는 어댑터를 관리하는 것 말고도 여러 일을 함께 맡습니다. 첫째는 방금 본 어댑터 관리로, 어떤 플랫폼을 켤지 정하고 각 어댑터를 띄웁니다. 둘째는 인증입니다. 정규화된 MessageEvent가 들어오면 그 발신자가 응답을 받을 자격이 있는지부터 가립니다. 미리 허용한 목록(allowlist)에 있거나 DM 페어링으로 승인받은 사람만 코어에 닿고, 나머지는 문 앞에서 걸러집니다. 낯선 사람이 메신저로 말을 건다고 아무나 내 에이전트를 부릴 수는 없기 때문입니다.

셋째는 **slash 명령**입니다. slash 명령이란 사용자가 빗금으로 시작하는 짧은 명령을 쳐서 에이전트에게 대화가 아닌 지시를 내리는 통로입니다. 모델에게 문장을 물어보는 대신 정해진 동작을 곧장 부르는 길입니다. 넷째는 **hook**으로, 특정 사건이 일어나는 지점에 사용자가 미리 자기 코드를 끼워 두어 그때 함께 실행되게 하는 장치입니다. 다섯째는 cron 실행입니다. 5.1.3에서 보았듯 Cron은 독립 프로세스가 아니라 Gateway 안에 얹혀 삽니다. 상주 중인 Gateway가 시계를 재다가 예약된 작업을 깨웁니다. 이 다섯 가지가 한 daemon 안에 모여 있기에, Gateway가 꺼지면 메신저 응답도 예약 작업도 함께 멈춥니다.

정규화가 낳는 이점 하나를 마지막으로 짚습니다. **unified memory**, 곧 채널을 넘어선 기억의 공유입니다. 모든 어댑터가 같은 MessageEvent로 수렴하고 그 뒤를 하나의 코어가 처리합니다. 그래서 어느 채널로 말을 걸든 코어가 읽고 쓰는 기억과 세션 저장소는 같은 한 곳입니다. 낮에 Slack으로 정리해 둔 내용을 저녁에 Telegram으로 물어도 에이전트가 이어서 답할 수 있는 까닭이 여기 있습니다. 창구는 24개지만 그 뒤의 에이전트는 하나이고, 기억도 하나입니다. 정규화가 입구의 다름을 지웠기에 출구의 하나됨이 가능해진 것입니다.

정리하면, Hermes의 Gateway는 24개 플랫폼 어댑터를 관리하는 단일 daemon입니다. 각 어댑터는 자기 플랫폼의 네이티브 이벤트를 통일된 MessageEvent로 정규화합니다. 그 덕에 Gateway와 코어는 플랫폼의 다름을 몰라도 됩니다. Gateway는 어댑터 관리에 더해 인증, slash 명령, hook, cron 실행까지 한 몸에 맡습니다. 그리고 모든 채널이 같은 MessageEvent로 수렴해 하나의 코어와 기억을 공유하므로, 어느 채널로 말해도 같은 기억으로 이어지는 unified memory가 성립합니다.

5.3.2에서는 이 Gateway 안에서 예약 작업을 맡는 Cron이 매 실행마다 어떻게 기억 없는 새 에이전트로 다시 태어나는지를 다룹니다.

이 단원을 마치면 24개 플랫폼 어댑터가 네이티브 이벤트를 통일된 MessageEvent로 정규화해 단일 Gateway가 라우팅하는 구조를 설명할 수 있습니다.
