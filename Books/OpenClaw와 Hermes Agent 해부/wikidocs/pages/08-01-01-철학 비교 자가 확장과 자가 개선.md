# 8.1.1 철학 비교: 자가 확장과 자가 개선

앞의 일곱 개 Phase에서 우리는 OpenClaw와 Hermes Agent를 따로따로 해부했습니다. 4.3.4에서 OpenClaw의 정착한 활용 패턴을, 7.2.4에서 Hermes를 코딩 에이전트의 지휘자로 쓰는 패턴을 살펴봤습니다. 이제 Phase 8에서는 두 시스템을 나란히 놓습니다. 부품을 각각 이해했으니, 같은 문제를 두 팀이 어떻게 다르게 풀었는지 축별로 비교할 차례입니다.

비교의 뼈대가 될 관점 하나를 먼저 세우겠습니다. 두 시스템을 가르는 가장 깊은 갈림길은 에이전트가 어떻게 성장하는가입니다. OpenClaw는 **자가 확장**을 택했습니다. 자가 확장이란 코어를 최소로 두고, 새 능력은 사람과 커뮤니티가 스킬로 만들어 붙이는 방식을 말합니다. Hermes는 **자가 개선**을 택했습니다. 자가 개선이란 에이전트 자신이 경험에서 스킬을 만들고 쓰면서 다듬는, 내장된 학습 루프를 말합니다. 이 하나의 대비가 다섯 축의 차이를 모두 설명합니다. 축마다 OpenClaw를 먼저, Hermes를 그다음 보고, 끝에서 대비하겠습니다.

첫째 축은 언어와 런타임입니다. OpenClaw는 TypeScript와 Node.js로 만들어졌고, 설치는 Node 22 이상을 요구합니다. Hermes는 Python으로 만들어졌습니다. 코드의 81.6퍼센트가 Python이며, 설치 스크립트가 Python 실행 환경을 함께 깝니다. 두 언어의 차이는 취향 문제가 아닙니다. Hermes를 만든 Nous Research는 모델을 학습시켜 온 조직이고, 학습과 데이터 파이프라인의 표준어가 Python입니다. 뒤에서 볼 학습 루프가 왜 Hermes 쪽에 자연스럽게 들어왔는지가 이 런타임 선택에서 이미 예고됩니다.

둘째 축은 코어 전략입니다. OpenClaw의 코어는 Pi라는 미니멀 코딩 에이전트입니다. Pi는 극단적으로 짧은 시스템 프롬프트와 Read, Write, Edit, Bash라는 네 개의 도구만 가진 작은 핵입니다. 새 기능이 필요하면 코어를 키우지 않고 에이전트에게 코드를 써서 스스로 확장하라고 시킵니다. 이 사상을 OpenClaw는 소프트웨어가 소프트웨어를 짓는다고 표현합니다. Hermes의 코어는 AIAgent라는 하나의 클래스입니다. 이 클래스가 CLI, 게이트웨이, 에디터 연동, 배치를 모두 담당하고, 플랫폼별 차이는 얇은 어댑터로 흡수합니다. 그리고 이 단일 코어 안에 학습 루프가 처음부터 박혀 있습니다. 정리하면, OpenClaw는 작은 코어를 비워 자가 확장을 얹었고, Hermes는 단일 코어를 채워 학습 루프를 내장했습니다.

셋째 축은 기억 전략입니다. OpenClaw의 기억은 대부분 평문 마크다운 파일입니다. 페르소나를 담은 SOUL.md, 장기 기억을 담은 MEMORY.md, 날짜별 작업 노트가 워크스페이스 폴더에 그대로 놓입니다. 벡터 데이터베이스 같은 불투명한 저장소가 없어서 사람이 열어 읽고, 고치고, 버전 관리로 이력을 남길 수 있습니다. Hermes의 기억은 더 엄격합니다. 상주 기억인 MEMORY.md는 2,200자, 사용자 정보인 USER.md는 1,375자라는 하드 제한이 걸립니다. 이 상한을 넘기는 쓰기는 자동으로 요약되지 않고 에러를 돌려줍니다. 그러면 에이전트가 스스로 기존 항목을 통합하거나 지워야 합니다. 대신 지난 대화 원문은 전부 SQLite 데이터베이스에 남고 전문 검색으로 조회합니다. 정리하면, OpenClaw는 가독성과 감사 용이성을 앞세운 파일 중심 기억이고, Hermes는 용량 제한으로 큐레이션을 강제하고 나머지는 검색으로 여는 방식입니다.

넷째 축은 스킬 생태계이며, 여기가 두 철학이 가장 선명하게 갈리는 지점입니다. OpenClaw의 스킬은 name과 description을 가진 마크다운 문서이고, 공개 레지스트리인 ClawHub에서 설치합니다. 능력의 원천이 커뮤니티입니다. 누군가 만든 스킬을 사람이 골라 설치하고 검증하니, 확장의 주도권이 사람과 커뮤니티에 있습니다. Hermes의 스킬은 포맷은 같은 계열이지만 만드는 주체가 다릅니다. 에이전트가 다섯 번 이상의 도구 호출이 필요한 복잡한 작업을 성공적으로 끝내거나 새로운 작업 흐름을 발견하면, 전용 도구로 스스로 스킬을 만듭니다. 그리고 patch 동작으로 쓰면서 그 스킬을 조금씩 다듬습니다. 정리하면, OpenClaw의 스킬은 사람과 커뮤니티가 큐레이션하는 생태계이고, Hermes의 스킬은 에이전트가 경험에서 생성하는 산물입니다. 자가 확장과 자가 개선을 가르는 핵심이 바로 여기입니다. OpenClaw에서는 사람이 능력을 붙여 주고, Hermes에서는 에이전트가 능력을 길러 냅니다.

다섯째 축은 보안 이력과 거버넌스입니다. OpenClaw는 2026년 상반기에 굵직한 보안 사고를 겪었습니다. 확인 없이 자동 연결되는 취약점으로 인증 토큰을 탈취당하는 CVE가 공개됐고, 수만 대의 인스턴스가 인증 없이 인터넷에 노출됐습니다. 원인은 소프트웨어 결함보다 배포 관행이었습니다. 안전한 기본값이 있었는데도 빠른 시작을 위해 그 값을 끄고 개방한 사용자가 많았습니다. 거버넌스를 보면, 창업자가 대형 랩에 합류한 뒤 프로젝트는 비영리 재단으로 이관됐고 여러 대기업이 후원자로 참여합니다. Hermes는 후발 주자입니다. 2026년 5월 기준 에이전트 자체의 CVE가 0건인데, 앞선 시스템의 사고를 지켜본 뒤 보수적인 기본값을 심은 이점이 큽니다. 거버넌스를 보면, 제작사 Nous Research가 모델을 학습시키는 조직이라는 점이 중요합니다. 에이전트 사용 데이터를 학습으로 되돌리는 구조가 시스템에 들어 있습니다. 정리하면, OpenClaw는 대규모 노출 사고를 겪고 재단으로 넘어간 시스템이고, Hermes는 그 사고를 학습해 CVE 없이 출발했으며 모델 트레이너가 운영하는 시스템입니다.

다섯 축의 대비를 한자리에 모으면 다음과 같습니다.

```
축         | OpenClaw                        | Hermes Agent
-----------+---------------------------------+---------------------------------
언어       | TypeScript / Node.js            | Python
코어 전략  | 최소 코어(Pi) + 자가 확장       | 단일 코어(AIAgent) + 학습 루프
기억       | 평문 파일 중심 (가독·감사)      | 하드 제한 + SQLite 검색
스킬       | ClawHub 커뮤니티 큐레이션       | 에이전트 자가 생성·개선
보안       | CVE 다수 + 대규모 노출 사태     | 에이전트 CVE 0건 (후발 이점)
거버넌스   | 비영리 재단 (대기업 후원)       | 모델 트레이너(Nous) 운영
```

정리하면, 두 시스템은 다섯 축에서 갈립니다. OpenClaw는 TypeScript 위에 최소 코어와 자가 확장을 얹고, 평문 파일로 기억을 열어 두며, 커뮤니티가 스킬을 큐레이션하고, 대규모 노출 사고를 겪은 뒤 비영리 재단으로 넘어갔습니다. Hermes는 Python 위에 단일 코어와 학습 루프를 담고, 용량을 제한해 기억을 큐레이션하며, 에이전트가 스킬을 스스로 생성하고, 그 사고를 학습해 CVE 없이 출발한 모델 트레이너의 시스템입니다. 핵심 대비는 하나입니다. 사람이 능력을 붙이느냐, 에이전트가 능력을 기르느냐입니다.

8.1.2에서는 이 비교를 실무로 옮겨, 어떤 상황에서 어느 시스템을 고르는지와 한쪽에서 다른 쪽으로 옮겨 갈 때의 마이그레이션을 다룹니다.

이 단원을 마치면 OpenClaw와 Hermes Agent를 언어, 코어 전략, 기억, 스킬, 보안과 거버넌스라는 다섯 축으로 비교하고, 자가 확장과 자가 개선이라는 두 철학의 차이를 설명할 수 있습니다.
