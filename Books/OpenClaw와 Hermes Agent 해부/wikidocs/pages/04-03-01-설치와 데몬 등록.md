# 4.3.1 설치와 데몬 등록

지금까지 우리는 OpenClaw의 내부 부품을 하나씩 뜯어 봤습니다. 이제 그것을 실제 머신에 올려 24시간 돌리는 실무 단계로 넘어갑니다. Phase 4의 첫 단원인 이번 장의 목표는 세 가지입니다. OpenClaw를 설치하고, 꺼지지 않는 상주 프로세스로 등록하고, 제대로 살아 있는지 점검하는 흐름을 순서대로 익히는 것입니다.

먼저 배경 용어 하나를 짚겠습니다. **데몬**(daemon)입니다. 데몬이란 사용자가 매번 손으로 켜지 않아도 시스템이 알아서 뒤에서 계속 돌려 주는 프로그램을 말합니다. 터미널 창에서 명령을 입력하면 그 창을 닫는 순간 프로그램도 함께 죽습니다. 반면 데몬으로 등록된 프로그램은 로그인이나 부팅 시점에 자동으로 뜨고, 창과 무관하게 배경에서 상주합니다. 2.2.1에서 우리는 게이트웨이가 단 하나의 상주 프로세스여야 하는 이유를 봤습니다. 그 상주 프로세스를 사람 손 없이 계속 살려 두는 장치가 바로 데몬 등록입니다.

설치에 앞서 요구사항을 확인해야 합니다. OpenClaw는 **Node**(Node.js) 22 버전 이상을 요구합니다. Node는 JavaScript 코드를 서버에서 실행해 주는 런타임으로, OpenClaw 자체가 그 위에서 동작하기 때문입니다. 자동 설치 스크립트를 쓰면 필요한 Node 버전을 알아서 맞춰 주므로 이 조건을 손으로 챙길 일은 많지 않습니다.

설치 방법은 두 갈래입니다. 자동 설치 스크립트를 통째로 실행하거나, 패키지 매니저로 직접 설치하는 방식입니다. 자동 스크립트는 운영체제에 따라 진입점이 갈립니다. macOS, Linux, 그리고 Windows의 WSL2 환경에서는 install.sh 스크립트를 씁니다. Windows 네이티브 환경에서는 PowerShell용 install.ps1 스크립트를 씁니다. 두 스크립트는 이름만 다를 뿐 Node 준비부터 OpenClaw 설치까지를 한 번에 처리한다는 목적은 같습니다.

이제 이 단원의 실습으로 들어갑니다. 패키지 매니저 방식을 기준으로, 설치에서 상태 점검까지를 네 개의 명령으로 밟아 나갑니다. 첫 명령은 다음과 같습니다.

```
npm install -g openclaw@latest
```

이 명령은 Node에 딸려 오는 패키지 관리자 npm을 써서 OpenClaw의 최신판을 설치합니다. `-g`는 전역(global) 설치를 뜻하는 표시로, 시스템 어디서나 openclaw 명령을 부를 수 있게 깔라는 의미입니다. `openclaw@latest`는 openclaw 패키지의 가장 최신 버전을 지정합니다. 이 한 줄이 끝나면 openclaw라는 실행 명령이 머신에 자리를 잡습니다.

설치가 끝났다면 이제 상주화, 곧 데몬 등록 차례입니다.

```
openclaw onboard --install-daemon
```

이 명령이 이번 단원의 핵심입니다. `onboard`는 초기 설정을 안내하는 온보딩 절차입니다. 여기 붙은 `--install-daemon` 옵션이 게이트웨이를 데몬으로 등록하라는 지시입니다. 흥미로운 점은 이 한 줄이 운영체제마다 다른 방식으로 상주화를 처리한다는 것입니다. 각 운영체제가 상주 프로그램을 관리하는 표준 장치가 다르기 때문입니다.

macOS에서는 **LaunchAgent**로 등록됩니다. LaunchAgent는 macOS가 로그인한 사용자별로 배경 프로그램을 자동 실행해 주는 표준 방식입니다. Linux와 WSL2에서는 **systemd** 사용자 서비스로 등록됩니다. systemd는 리눅스 계열에서 부팅과 서비스 상주를 총괄하는 관리자로, 여기에 사용자 서비스로 얹으면 로그인 세션과 함께 게이트웨이가 뜹니다. Windows 네이티브에서는 **작업 스케줄러**(Scheduled Task)에 등록됩니다. 이 경로가 여의치 않으면 시작프로그램(Startup) 폴더에 넣는 방식으로 대체됩니다. 세 갈래로 나뉘는 모습을 한눈에 정리하면 다음과 같습니다.

| 운영체제 | 데몬 등록 방식 |
|---|---|
| macOS | LaunchAgent |
| Linux / WSL2 | systemd 사용자 서비스 |
| Windows | 작업 스케줄러 (시작프로그램 폴더 대체) |

이름과 구현은 제각각이지만 결과는 하나로 수렴합니다. 어느 운영체제에서든 게이트웨이가 로그인이나 부팅과 함께 알아서 뜨고, 터미널을 닫아도 배경에서 계속 살아 있게 됩니다. 상주형 에이전트에 데몬 등록이 필수인 까닭은, 게이트웨이가 꺼지면 예약된 작업도 메신저 연결도 함께 멈추기 때문입니다.

등록까지 마쳤다면 제대로 살아 있는지 확인할 차례입니다. 상태 점검용 명령이 두 개 있습니다. 먼저 전반적인 건강 상태를 봅니다.

```
openclaw doctor
```

`doctor`는 말 그대로 진단 명령입니다. Node 버전이 요구사항을 만족하는지, 설정 파일이 제자리에 있는지, 데몬 등록이 정상인지 같은 항목을 한 번에 훑습니다. 문제가 있으면 짚어 주므로 설치 직후 가장 먼저 돌려 볼 명령입니다. 다음으로 게이트웨이 자체의 상태를 확인합니다.

```
openclaw gateway status
```

`gateway status`는 상주 프로세스인 게이트웨이가 지금 돌고 있는지, 어느 주소에서 듣고 있는지를 알려 줍니다. 이 명령이 정상을 보고하면 데몬 등록이 실제 프로세스로 이어졌다는 뜻입니다.

명령줄 점검 외에 눈으로 확인하는 경로도 있습니다. 게이트웨이가 뜨면 웹 대시보드가 http://127.0.0.1:18789/ 주소에서 열립니다. 127.0.0.1은 외부가 아닌 내 컴퓨터 자신을 가리키는 주소라, 이 대시보드는 기본적으로 로컬에서만 보입니다. 브라우저로 여기에 접속되면 게이트웨이가 살아 있다는 증거입니다. 한편 설정을 직접 살펴보거나 고치고 싶을 때 열어 볼 파일은 ~/.openclaw/openclaw.json입니다. ~는 사용자의 홈 디렉토리를 가리키는 표시입니다.

정리하면, OpenClaw 설치는 Node 22 이상이라는 요구사항 위에서 시작합니다. 운영체제에 따라 install.sh 또는 install.ps1 스크립트를 쓰거나 npm으로 직접 설치합니다. 이어 openclaw onboard --install-daemon 한 줄이 게이트웨이를 데몬으로 등록합니다. 이때 macOS는 LaunchAgent, Linux와 WSL2는 systemd, Windows는 작업 스케줄러로 각기 다르게 처리합니다. 마지막으로 openclaw doctor로 전반 상태를, openclaw gateway status로 프로세스 상태를 점검합니다. http://127.0.0.1:18789/ 대시보드와 ~/.openclaw/openclaw.json 설정 파일로도 실제 동작을 확인합니다.

이렇게 게이트웨이를 상주화까지 마치고 나면 자연스럽게 다음 질문이 따라옵니다. 이 상주 프로세스를 외부에 함부로 노출하지 않으면서 어떻게 원격에서 안전하게 접근할 것인가입니다. 4.3.2 배포 체크리스트: 노출 없이 원격 접근에서 그 방법을 다룹니다.

이 단원을 마치면 OpenClaw의 설치 요구사항과 설치 명령, 운영체제별 데몬 등록 방식, 상태 점검 명령을 순서대로 설명할 수 있습니다.
