# 2.2.4 Nodes: 중앙 제어와 디바이스 분산

2.2.2에서는 게이트웨이가 여는 WebSocket 문으로 여러 클라이언트가 들어와 명령을 보내고 이벤트를 받는 모습을 봤습니다. 그런데 이 그림에는 빈자리가 하나 있습니다. 게이트웨이는 대개 서버나 책상 위 컴퓨터에서 돌아갑니다. 그 머신에는 카메라도 GPS도 없습니다. "지금 방 사진 한 장 찍어 줘" 같은 요청 앞에서는, 판단할 뇌가 있어도 감각기관이 없는 셈입니다.

OpenClaw는 이 빈자리를 **노드**(node)로 채웁니다. 노드는 게이트웨이의 WebSocket에 role: node라는 역할 표식을 달고 접속한 기기입니다. iPhone, Android, macOS, watchOS 기기와 화면 없는 Linux·Windows 머신이 노드가 될 수 있습니다. 제어용 클라이언트가 사람이 게이트웨이를 조작하는 창구라면, 노드는 반대로 게이트웨이가 기기를 부리는 통로입니다. 들어오는 문은 같아도 명령이 흐르는 방향이 다릅니다.

역할 경계부터 분명히 하겠습니다. 노드는 게이트웨이 서비스를 실행하지 않습니다. 휴대전화에 컴패니언 앱을 깔아도 그 안에서 에이전트가 도는 것이 아닙니다. WhatsApp이나 Telegram으로 보낸 채널 메시지도 노드가 아니라 게이트웨이에 도착합니다. 노드가 맡는 일은 그 기기만 할 수 있는 동작, 곧 사진 촬영이나 위치 조회 요청을 받아 수행하고 결과를 돌려주는 것뿐입니다.

이런 배치를 **hub-and-spoke**라고 부릅니다. 자전거 바퀴처럼 가운데 허브가 있고 바큇살 끝에 주변 장치가 매달린 모양이라는 뜻입니다. 판단, 채널 라우팅, 기록은 전부 허브인 게이트웨이에 모이고, 기기 고유 기능만 바큇살 끝으로 나갑니다.

```
 [채널 메시지] ──▶ ┌──── Gateway (허브) ────┐ <── [제어 클라이언트]
                   │  판단·라우팅·기록은 여기  │
                   └──┬────────┬────────┬───┘
           role: node │        │        │
                      ▼        ▼        ▼
                  [iPhone] [Android]  [macOS]
                 카메라·위치 +SMS·알림  화면 녹화
```

왜 이렇게 나눴는지는 반대 극단을 그려 보면 드러납니다. 기기마다 에이전트를 따로 돌리면 기억과 세션이 흩어지고, 어느 기기가 무슨 일을 했는지 한곳에서 볼 수 없게 됩니다. 메시징과 모델 오케스트레이션을 중앙에 모으면 모든 명령이 한 지점을 거치므로, 중앙 제어와 감사 추적이 유지됩니다. 기기 고유 기능은 전문 말단으로 내보내, 그 자리에서만 얻을 수 있는 네이티브 센서를 활용합니다. 두 이점을 동시에 잡는 절충이 hub-and-spoke입니다.

노드에 시킬 수 있는 동작은 **명령 패밀리** 단위로 묶여 있습니다. 이름이 패밀리.동작 꼴이라 목록만 봐도 성격이 보입니다. camera.list, camera.snap, camera.clip은 카메라 목록 조회, 사진 촬영, 짧은 영상 촬영을 맡습니다. location.get은 현재 위치를 조회하고 screen.record는 화면을 녹화합니다. system.run과 system.notify는 명령 실행과 알림 표시를, canvas 패밀리는 노드 화면의 WebView에 내용을 띄우는 일을 맡습니다. 이 밖에 device.info, contacts.search, photos.latest처럼 기기 정보, 연락처, 최근 사진을 다루는 명령도 있습니다.

위치와 카메라를 내주는 통로인 만큼 아무 기기나 노드로 붙게 둘 수는 없습니다. 그래서 노드 접속에는 페어링 승인 절차가 붙습니다.

```
 [노드]                     [게이트웨이]               [운영자]
   │  1. 서명된 기기 신원 제시  │
   ├────────────────────────▶│ 2. 페어링 요청 생성 (5분 만료)
   │                          │<── 3. devices approve <requestId>
   │  4. 디바이스 토큰 발급     │
   │<─────────────────────────┤
```

노드가 서명된 기기 신원을 제시하면 게이트웨이는 페어링 요청을 만들어 두고 기다립니다. 요청은 5분이 지나면 만료되므로, 오래된 요청이 남아 있다가 뒤늦게 승인되는 일이 없습니다. 운영자가 openclaw devices approve 명령으로 요청을 승인하면 그제야 **디바이스 토큰**(device token)이 발급됩니다. 노드는 이후 접속에 이 토큰을 씁니다.

승인은 기기 신원에 대한 결정이지 모든 능력에 대한 결정이 아닙니다. camera.snap, screen.record, sms.send처럼 감시나 발신에 쓰일 수 있는 위험 명령은, 페어링을 마친 노드라도 기본적으로 거부됩니다. 게이트웨이 설정의 gateway.nodes.allowCommands 목록에 그 명령을 적어야 비로소 열립니다. 이 기기를 믿는다는 결정과 이 기기로 사진 촬영까지 허용한다는 결정을 나눈 것입니다. 기본은 닫아 두고 필요한 능력만 명시적으로 켜는 opt-in 방식입니다.

노드가 제공하는 기본 능력은 플랫폼마다 다릅니다.

| 플랫폼 | 기본 능력 |
|---|---|
| iOS | 카메라, 위치, 연락처, 사진, 모션 |
| Android | iOS 능력에 더해 SMS, 알림, 통화 기록, 헬스, 설치 앱 |
| macOS | 화면 녹화, 데스크톱 제어, 접근성 입력 |
| watchOS | 기기 정보와 알림 수준의 최소 구성 |

이 차이는 각 플랫폼이 앱에 열어 주는 기능 범위를 그대로 반영합니다. Android 목록에는 SMS와 통화 기록이 있지만 iOS 목록에는 없습니다. 화면 녹화와 접근성 입력은 데스크톱인 macOS에서만 의미가 있는 능력입니다. 노드 계층은 이 차이를 숨기지 않으므로, 어떤 일을 어느 기기에 시킬지도 설계의 일부가 됩니다.

정리하면, 노드는 모바일과 데스크톱 기기가 role: node로 게이트웨이에 접속해 카메라, 위치, 화면 녹화 같은 기기 고유 기능을 대신 수행하는 주변 장치입니다. 노드는 게이트웨이 서비스를 실행하지 않고 채널 메시지도 받지 않으며, 판단과 라우팅과 기록은 허브인 게이트웨이가 쥡니다. 이 hub-and-spoke 배치가 중앙 제어와 감사 추적을 지키면서 네이티브 센서를 활용하는 길을 엽니다. 접속은 5분 만료 페어링 요청, devices approve 승인, 디바이스 토큰 발급으로 통제됩니다. 위험 명령은 allowCommands에 명시해야만 열리고, 플랫폼별 기본 능력은 각 플랫폼이 허용하는 범위를 따라 서로 다릅니다.

2.3.1에서는 게이트웨이에 도착한 메시지 하나가 행동과 응답으로 바뀌는 다섯 단계, agent loop를 따라갑니다.

이 단원을 마치면 모바일과 데스크톱 기기가 role: node로 게이트웨이에 붙어 디바이스 고유 기능을 수행하는 구조와 페어링 승인 절차를 설명할 수 있습니다.
