# 5.1.2 AIAgent 단일 코어: 플랫폼 차이를 몰아내다

5.1.1에서 Hermes Agent를 떠받치는 여섯 가지 설계 원칙을 훑어봤습니다. 이 단원은 그 원칙들이 코드에서 실제로 어떤 모습으로 나타나는지, 가장 중심이 되는 한 부분을 뜯어 봅니다. 바로 에이전트의 심장에 해당하는 실행 코어입니다. 결론을 먼저 말하면, Hermes는 성격이 다른 여러 사용 방식을 하나의 코어로 처리합니다. 그리고 그들 사이의 차이는 코어가 아니라 바깥 껍데기로 밀어냅니다.

먼저 낯선 말 두 개를 배경부터 풀겠습니다. 첫째는 **클래스**(class)입니다. 프로그램을 짤 때는 관련된 데이터와 동작을 한 덩어리로 묶어 두면 다루기 편합니다. 이렇게 묶어 놓은 설계 단위를 클래스라고 부릅니다. 붕어빵 틀 하나로 붕어빵을 여러 개 찍어 내듯이, 클래스 하나를 정의해 두면 같은 성질의 대상을 필요할 때마다 만들어 씁니다. Hermes에서 에이전트의 실행 논리 전체를 담은 클래스의 이름이 AIAgent입니다.

둘째는 **엔트리포인트**(entry point)입니다. 하나의 프로그램이라도 시작하는 방식은 여럿일 수 있습니다. 터미널에서 사람이 직접 실행할 수도 있고, 다른 프로그램이 대신 띄울 수도 있습니다. 프로그램에 처음 발을 들이는 이 진입 지점을 엔트리포인트라고 합니다. 같은 코어를 쓰더라도 어디로 들어오느냐에 따라 준비 과정과 마무리가 달라집니다.

이제 본론입니다. Hermes에서 에이전트의 실행 알맹이는 run_agent.py라는 파일 하나에 담긴 단일 에이전트 루프입니다. 여기서 루프란 모델을 부르고, 도구를 실행하고, 그 결과를 다시 모델에게 돌려주는 순환을 뜻합니다. 이 하나의 루프가 여러 일을 도맡습니다. 어떤 모델 제공자를 쓸지 고르고, 시스템 프롬프트를 조립하는 일이 여기 들어갑니다. 도구를 실행하다 오류가 나면 복구하고, 승인 같은 대화 콜백을 처리하고, 대화를 SQLite에 저장하는 일도 모두 이 코어의 몫입니다.

주목할 점은 이 하나의 AIAgent 클래스가 네 가지 서로 다른 쓰임을 전부 감당한다는 것입니다. 사람이 터미널에서 대화하는 CLI, 메신저를 상주 상태로 받아 주는 gateway가 그중 둘입니다. 나머지 둘은 코드 편집기와 붙는 ACP, 그리고 사람 없이 한 번에 돌아가는 batch입니다. 이들은 겉보기에 성격이 크게 다릅니다. 그런데도 대화를 굴리는 알맹이는 똑같은 코어 하나를 공유합니다.

그렇다면 서로 다른 네 쓰임의 차이는 어디로 갈까요. Hermes의 공식 설명이 이 지점을 못 박습니다. 하나의 AIAgent 클래스가 CLI, gateway, ACP, batch를 모두 맡고, 플랫폼 차이는 코어가 아니라 엔트리포인트에 둔다는 것입니다. 즉 화면에 어떻게 출력할지, 입력을 어떻게 받을지, 프로세스를 언제 끝낼지 같은 각 방식 고유의 처리는 진입 지점 쪽 얇은 코드가 맡습니다. 코어는 그런 사정을 알지 못한 채 대화 루프만 순수하게 돌립니다.

이 구조를 그림으로 보면 다음과 같습니다.

```
 [CLI]      [Gateway]     [ACP]      [Batch]
   │           │            │           │
   ▼           ▼            ▼           ▼
 ┌────────── 얇은 엔트리포인트 (플랫폼 차이 흡수) ──────────┐
 │  입출력 방식 · 프로세스 수명 · 진입 준비만 담당          │
 └───────────────────────┬────────────────────────────────┘
                         ▼
 ┌──────── AIAgent 코어 (run_agent.py, 단일 에이전트 루프) ────────┐
 │  provider 선택 · 프롬프트 조립 · 도구 실행/복구 · 세션 저장     │
 └─────────────────────────────────────────────────────────────────┘
```

이렇게 나누는 데는 뚜렷한 원칙이 하나 깔려 있습니다. 핵심 로직은 하나로 두고, 바깥 세계와 닿는 부분만 갈아 끼운다는 생각입니다. 소프트웨어 설계에서는 이런 구조를 **핵사고날 아키텍처**(hexagonal architecture)라고 부릅니다. 다른 이름으로는 포트와 어댑터(ports and adapters) 구조라고도 합니다. 여기서 코어는 계산과 판단의 중심을 맡고, 어댑터는 코어와 바깥을 잇는 얇은 연결부를 뜻합니다. Hermes에 대입하면 AIAgent가 코어이고, 네 엔트리포인트가 각각의 얇은 어댑터입니다. 안쪽은 그대로 둔 채 바깥의 어댑터만 바꿔 끼우면 새로운 사용 방식이 붙습니다.

왜 굳이 이렇게 할까요. 가장 큰 이유는 플랫폼마다 동작이 미묘하게 다른 버그를 원천 차단하기 위해서입니다. 만약 CLI용 에이전트와 gateway용 에이전트를 따로 만들었다고 해 봅시다. 도구 실행 순서나 오류 복구 방식에 조금씩 차이가 생기기 쉽습니다. 그러면 터미널에서는 멀쩡히 되던 일이 메신저에서는 어긋나는 문제가 나타납니다. 이런 불일치는 재현하기도 고치기도 까다롭습니다. 코어를 하나로 못 박으면 이 문제가 생길 자리 자체가 사라집니다. 대화의 핵심 동작은 어느 진입 지점으로 들어와도 완전히 같기 때문입니다.

이 단일 코어는 모델 쪽으로도 유연합니다. AIAgent는 세 가지 API 모드를 지원합니다. 널리 쓰이는 chat completions 방식, 그와 다른 responses 방식, 그리고 Anthropic Messages 방식입니다. 여기서 API 모드란 모델 제공자와 주고받는 요청과 응답의 형식 규격을 말합니다. 제공자마다 이 규격이 조금씩 다릅니다. 중요한 것은 이 세 방식을 다루는 처리도 코어 안에 함께 들어 있다는 점입니다. 그래서 네 엔트리포인트 중 무엇으로 들어오든, 어떤 모델을 쓰든 상관없이 같은 규격 처리를 공유합니다. 모델 다양성과 플랫폼 다양성을 코어 한 곳이 동시에 흡수하는 셈입니다.

정리하면, Hermes는 에이전트의 실행 알맹이를 run_agent.py의 단일 루프에 담아 하나의 AIAgent 클래스로 만들고, 이 클래스 하나가 CLI, gateway, ACP, batch를 모두 담당합니다. 플랫폼마다 다른 처리는 코어가 아니라 얇은 엔트리포인트로 몰아넣습니다. 이것이 핵심 로직은 하나, 바깥과 닿는 부분만 갈아 끼우는 핵사고날 구조입니다. 세 가지 API 모드까지 코어가 함께 품어, 플랫폼이 달라도 동작이 미묘하게 어긋나는 버그가 애초에 생기지 않습니다. 다음 단원 5.1.3에서는 이 코어를 감싸는 CLI, Gateway, ACP, Cron 네 가지 프로세스 모델이 실제로 어떻게 다르게 돌아가는지 살펴봅니다. 이 단원을 마치면 하나의 AIAgent 클래스가 CLI, gateway, ACP, batch를 모두 담당하고 플랫폼 차이를 엔트리포인트로 몰아내는 구조의 이유를 설명할 수 있습니다.
