# 1.1.3 메신저가 인터페이스가 된다는 것

1.1.2에서 챗봇과 상주형 에이전트를 가르는 차이들을 살펴보았습니다. 이 단원은 그 가운데 인터페이스라는 축을 파고듭니다. 에이전트가 서버에서 하루 종일 돌아간다는 것은 설계의 절반일 뿐입니다. 나머지 절반은 사용자가 그 에이전트에게 어디서 말을 거는가입니다. OpenClaw와 Hermes Agent는 이 문제에 같은 답을 골랐습니다. 전용 앱을 새로 만들지 않고, 사용자가 이미 쓰는 메신저를 그대로 인터페이스로 삼는 것입니다.

전용 앱 경로를 먼저 따져 보면 이 선택의 이유가 드러납니다. 새 앱은 설치라는 문턱을 넘게 하고, 낯선 화면에 익숙해지기를 요구하며, 무엇보다 그 앱을 열어 보는 습관을 새로 만들어야 합니다. 반면 메신저는 이미 하루에도 수십 번 열리는 앱이고, 알림과 기기 간 동기화처럼 대화에 필요한 기반이 갖춰져 있습니다. 상주형 에이전트에게는 이 점이 특히 중요합니다. 사용자가 자리를 비운 사이에도 일하다가 먼저 보고할 일이 생기는데, 메신저를 쓰면 그 보고가 친구의 메시지와 같은 목록에 도착합니다. 실제로 OpenClaw는 스스로를 메신저 플랫폼을 주 인터페이스로 삼는 에이전트라고 정의합니다. 이 철학을 한 문장으로 줄이면 '사용자가 있는 곳으로 들어간다'입니다. 사용자를 에이전트 앞으로 부르는 대신, 에이전트가 사용자의 대화 목록 안으로 들어갑니다.

그런데 메신저라고 뭉뚱그려 부르는 대상은 실제로 여럿입니다. Telegram, WhatsApp, Slack, Discord, Signal, iMessage는 서로 다른 서비스이고, 프로그램이 접속하는 방식도 제각각입니다. Telegram은 봇 토큰 하나만 발급받으면 되는 가장 단순한 축입니다. WhatsApp은 QR 코드를 찍어 기존 계정에 연결하고, Slack은 워크스페이스에 설치하는 앱 형태로 붙습니다. 이렇게 에이전트가 메신저 서비스 하나와 맺는 연결 통로를 **채널**(channel)이라고 부릅니다. 채널마다 접속 방식과 메시지 형식이 다르므로, 시스템은 서비스별 변환 계층을 두어 그 차이를 안쪽에서 흡수합니다. 이런 구조 위에서 OpenClaw는 20개가 넘는 메신저를, Hermes Agent는 24개 플랫폼을 채널로 지원합니다. 사용자에게 채널은 에이전트에게 말을 거는 문이고, 시스템에게 채널은 관리해야 할 외부 연결 하나입니다.

같은 채널 안에서도 대화의 모양은 둘로 갈립니다. **DM**(direct message)은 발신자와 에이전트가 일대일로 주고받는 대화입니다. **그룹 대화**는 여러 사람이 있는 방에 에이전트가 참여자 하나로 들어간 형태입니다. 둘의 차이는 응답의 기본값에서 드러납니다. DM에서 오가는 모든 메시지는 에이전트에게 하는 말이므로 매번 응답하는 것이 자연스럽습니다. 그룹에서는 사람들끼리의 대화가 흐르는 중이므로, 에이전트가 모든 메시지에 끼어들면 방이 망가집니다. 그래서 그룹에는 듣고는 있지만 답하지 않는 상태가 필요합니다. Hermes Agent는 응답 대신 '[SILENT]' 같은 침묵 표시를 돌려주는 방식으로, 답을 삼가면서도 대화 내용은 기록으로 유지합니다. 문맥을 다루는 방식도 갈립니다. OpenClaw는 DM을 기본적으로 하나로 이어지는 대화로 다룹니다. 언제 말을 걸어도 같은 비서를 만나게 하려는 것입니다. 반면 그룹 대화는 방마다 문맥을 따로 떼어 DM과 섞이지 않게 합니다.

메신저를 인터페이스로 삼는 순간 피할 수 없는 문제가 하나 따라옵니다. 챗봇의 웹 화면은 로그인한 본인만 쓰지만, 메신저 계정은 열린 주소입니다. 전화번호나 아이디를 아는 사람은 누구든 에이전트에게 메시지를 보낼 수 있습니다. 받는 쪽이 평범한 안내 봇이라면 스팸으로 끝날 일입니다. 그러나 상주형 에이전트는 운영자의 파일과 API 키, 명령 실행 권한을 쥔 실행 주체입니다. 낯선 발신자의 지시가 그대로 실행된다면 컴퓨터 전체를 내주는 사고로 이어집니다. 그래서 두 시스템 모두 모르는 발신자를 기본적으로 무시하고, **페어링**(pairing)이라는 승인 절차를 둡니다. 페어링은 처음 보는 발신자의 메시지를 곧바로 처리하지 않는 절차입니다. 운영자가 확인 코드를 통해 이 발신자와 대화해도 된다고 승인해야 비로소 대화가 열립니다. OpenClaw의 DM 정책 기본값이 바로 이 페어링이고, 승인할 발신자 목록을 직접 관리하는 allowlist 방식도 함께 제공합니다. 확인 코드의 유효 시간이나 목록 규칙 같은 세부 동작은 4.1.2에서 다룹니다. 여기서는 열린 주소로 들어오는 낯선 발신자를 걸러야 한다는 문제의식까지만 잡아 두면 충분합니다.

마지막으로, 채널은 하나로 끝나지 않습니다. 업무 대화는 Slack에, 개인 일정은 Telegram에, 가족 대화는 WhatsApp에 흩어져 있는 것이 보통의 생활입니다. 개인 비서를 표방하는 이상, 에이전트는 이 모두에 동시에 붙어 있어야 합니다. 실제로 두 시스템 모두 여러 채널을 한꺼번에 켜 둘 수 있고, 어느 채널에서 메시지가 와도 채팅 단위로 구분해 처리합니다.

```
 [Slack 그룹]    [Telegram DM]    [WhatsApp DM]      <- 채널
       │               │                │
       └───────────────┼────────────────┘
                       ▼
               상주형 에이전트 하나
            (하나의 기억, 하나의 정체성)
```

여기서 중요한 것은 그 뒤의 에이전트가 하나라는 사실입니다. Slack에서 맡긴 일의 결과를 Telegram에서 물어도 같은 기억을 가진 같은 비서가 답합니다. Hermes Agent가 모든 채널이 동일한 기억과 대화 저장소를 공유한다고 밝히는 이유가 여기 있습니다. 여러 외부 연결을 한 몸이 소유하고 관리해야 한다는 이 요구는, 두 시스템의 아키텍처를 해부하는 뒤의 장들에서 다시 만나게 될 출발점입니다.

정리하면, 상주형 에이전트는 전용 앱을 만드는 대신 사용자가 이미 쓰는 메신저 안으로 들어갑니다. 에이전트가 메신저 서비스 하나와 맺는 연결이 채널입니다. 채널 안의 대화는 일대일 DM과 여러 사람이 함께한 그룹 대화로 나뉘고, 응답 기본값과 문맥 관리가 서로 다릅니다. 메신저 계정은 누구나 말을 걸 수 있는 열린 주소이므로, 모르는 발신자를 거르는 페어링 같은 승인 절차가 필요합니다. 그리고 사용자 생활이 여러 메신저에 걸쳐 있으므로, 에이전트는 여러 채널에 동시에 연결된 채 하나의 기억으로 응답해야 합니다.

1.2.1에서는 이 설계를 처음 대중화한 OpenClaw가 한 개발자의 개인 실험에서 오픈소스 현상으로 커진 궤적을 따라갑니다.

이 단원을 마치면 상주형 에이전트가 전용 앱 대신 기존 메신저를 인터페이스로 삼는 이유와, 채널·DM·그룹 대화·페어링이라는 기본 개념을 설명할 수 있습니다.
