# 8.1.2 선택 가이드와 마이그레이션

8.1.1에서 두 시스템의 철학을 자가 확장과 자가 개선이라는 두 축으로 갈라 봤습니다. 이제 그 비교를 실제 결정으로 바꿀 차례입니다. 철학의 차이를 아는 것과 내 상황에서 어느 쪽을 깔지 정하는 것은 다른 문제입니다. 그래서 이 단원은 하나의 질문에 답합니다. 무엇을 원하면 어느 쪽인가, 그리고 잘못 골랐을 때 갈아타는 비용은 얼마인가입니다.

먼저 결정의 성격부터 분명히 해야 합니다. 여기서 다루는 선택은 어느 쪽이 더 좋은 소프트웨어인가를 가리는 순위 매기기가 아닙니다. 두 시스템은 잘하는 지점이 서로 다르므로, 옳은 답은 요구사항에 따라 달라집니다. 그러니 아래의 두 경우를 절대적인 우열이 아니라, 내 요구사항이 어느 쪽 문장에 더 많이 걸리는지 확인하는 체크리스트로 읽으시기 바랍니다.

OpenClaw가 맞는 경우를 먼저 봅니다. 세 가지 요구사항이 이쪽을 가리킵니다. 첫째는 생태계입니다. 가장 큰 커뮤니티와 가장 넓은 스킬 저장소가 필요하다면 OpenClaw입니다. 스킬을 남이 만들어 공개 레지스트리에 올리고 그것을 가져다 쓰는 흐름이 오래 자리 잡았기 때문에, 이미 누군가 만든 확장을 찾아 붙이기 쉽습니다. 둘째는 노드입니다. 여기서 **노드**란 아이폰이나 안드로이드, 맥 같은 기기가 게이트웨이에 붙어 카메라 촬영, 위치 조회, 화면 녹화처럼 그 기기만 할 수 있는 일을 대신 수행하는 구조를 말합니다. 집 안 카메라를 찍어 보내거나 폰의 위치를 물어보는 것처럼 디바이스 고유 기능까지 부리고 싶다면 OpenClaw 쪽이 앞섭니다. 셋째는 파일 기억의 감사입니다. OpenClaw는 장기 기억을 벡터 데이터베이스 같은 불투명한 저장소가 아니라 평문 마크다운 파일에 둡니다. 그래서 사람이 파일을 직접 열어 무엇이 기억됐는지 확인하고, 버전 관리 도구인 git으로 변경 이력을 추적하고 백업할 수 있습니다. 기억을 눈으로 감사하고 통째로 이력 관리하고 싶다면 이 투명함이 결정적입니다.

Hermes가 맞는 경우는 다른 요구사항에서 나옵니다. 첫째는 장기 학습입니다. 몇 달 쓸수록 실제로 손에 익는 에이전트를 원한다면 Hermes입니다. 이 시스템은 성공한 복잡한 작업을 스킬 문서로 스스로 굳히고 쓰면서 다듬는 학습 루프를 내장하고 있어서, 같은 일을 반복할수록 시행착오가 줄어듭니다. 둘째는 저비용 상시 운영입니다. 하드웨어 요구가 낮아 한 달 몇 달러 수준의 저사양 가상 서버나 라즈베리파이에서도 24시간 돌린 사례가 많습니다. 큰 장비 없이 상시 구동을 목표로 한다면 이 가벼움이 유리합니다. 셋째는 보수적 기본값입니다. Hermes는 앞선 시스템의 보안 사고를 지켜본 뒤에 만들어졌고, 그 흔적이 초기 설정 곳곳에 있습니다. 이를테면 전체 자동 승인 모드를 켜도 파일시스템을 지우는 명령 같은 위험한 축은 끝내 통과하지 못하도록 바닥을 깔아 두고, 에이전트가 만든 스킬은 사람의 승인을 거치도록 게이트를 걸 수 있습니다. 안전 쪽으로 기울어진 출고 상태를 선호한다면 이쪽입니다.

그렇다면 잘못 골랐을 때가 남습니다. 결론부터 말하면 갈아타는 비용은 낮은 편이고, 그 근거는 두 가지입니다. 첫째는 이주 도구입니다. Hermes는 앞서 쓰던 자산을 흡수하는 명령을 제공합니다. **hermes claw migrate**가 그것인데, OpenClaw의 설정과 기억, 스킬, 그리고 모델 접속에 쓰던 API 키까지 자동으로 가져옵니다. 설치 후의 안내 마법사도 기존 OpenClaw 폴더를 감지하면 이주를 먼저 제안합니다. 그림으로 보면 이렇습니다.

```
 [OpenClaw 자산]                         [Hermes]
  설정  ┐                               ┌ 설정
  기억  ┤   hermes claw migrate         │ 기억
  스킬  ┼──────────────────────────────▶│ 스킬
 API 키 ┘   (~/.openclaw 자동 감지)      └ API 키
```

둘째는 스킬 표준의 수렴입니다. 확장 능력을 담는 스킬이 시스템마다 제각각이라면 갈아탈 때 그 자산을 통째로 다시 만들어야 합니다. 그런데 두 시스템의 스킬은 같은 계열의 형식을 쓰고, **agentskills.io**라는 오픈 표준으로 모여 가고 있습니다. 여기서 오픈 표준이란 특정 회사의 것이 아니라 공개된 규격을 가리키며, 여러 도구가 같은 규격을 따르면 한쪽에서 쓰던 스킬이 다른 쪽에서도 통합니다. 형식이 하나로 수렴할수록, 갈아타기의 가장 무거운 짐인 확장 자산의 재작성 비용이 줄어듭니다.

이 두 가지를 합치면 결론이 나옵니다. 이주 도구가 설정과 기억을 옮겨 주고 스킬 형식이 표준으로 모이고 있으므로, 한쪽을 골랐다가 요구사항이 바뀌어도 처음부터 다시 쌓는 부담은 크지 않습니다. 따라서 선택은 되돌릴 수 없는 결단이 아니라, 지금 요구사항에 더 맞는 쪽을 먼저 써 보는 일에 가깝습니다. 두 경우를 한 표로 정리하면 다음과 같습니다.

| 원하는 것 | 맞는 쪽 | 근거 |
|---|---|---|
| 최대 생태계, 노드 기능, 평문 기억 감사 | OpenClaw | 커뮤니티 스킬, 디바이스 확장, git 이력 관리 |
| 장기 학습, 저비용 상시 운영, 보수적 기본값 | Hermes | 자가 스킬 루프, 저사양 구동, 안전 지향 출고값 |
| 갈아타기 | 낮은 비용 | hermes claw migrate + agentskills.io 수렴 |

정리하면, 선택은 우열이 아니라 요구사항의 문제입니다. 생태계와 노드와 파일 기억 감사가 중요하면 OpenClaw, 장기 학습과 저비용 상시 운영과 보수적 기본값이 중요하면 Hermes입니다. 그리고 hermes claw migrate가 기존 자산을 흡수하고 스킬이 agentskills.io 표준으로 수렴하는 덕분에 갈아타기 비용은 낮으므로, 이 결정은 처음에 한 번 정하면 끝나는 것이 아니라 언제든 다시 조정할 수 있습니다.

8.2.1에서는 두 시스템을 넘어, 어느 쪽을 쓰든 내 설계에 그대로 옮겨 심을 수 있는 패턴의 첫 번째로 control plane 단일화와 코어-어댑터 분리를 다룹니다.

이 단원을 마치면 요구사항에 따라 OpenClaw와 Hermes 중 한쪽을 고르는 기준을 제시하고, hermes claw migrate와 스킬 표준 수렴을 근거로 갈아타기 비용을 평가할 수 있습니다.
