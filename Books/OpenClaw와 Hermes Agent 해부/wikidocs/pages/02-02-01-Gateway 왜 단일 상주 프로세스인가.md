# 2.2.1 Gateway: 왜 단일 상주 프로세스인가

2.1.1에서 판단하는 뇌는 외부 모델에 두고, 움직이는 근육은 내 머신에 둔다는 분리를 보았습니다. 이 단원에서는 그 근육의 중심 부품을 들여다봅니다. OpenClaw를 설치하고 나면 내 컴퓨터에 실제로 떠 있는 것은 프로세스 단 하나입니다.

보통의 프로그램은 사람이 실행할 때 켜지고 일이 끝나면 종료됩니다. 그런데 메신저로 부리는 에이전트는 사정이 다릅니다. 메시지가 언제 도착할지 알 수 없으니, 받는 쪽은 새벽에도 켜져 있어야 합니다. 이렇게 필요할 때만 잠깐 뜨는 대신 늘 떠서 요청을 기다리는 프로그램을 **상주 프로세스**(daemon)라고 부릅니다.

OpenClaw는 여기서 선택을 하나 더 합니다. 상주 프로세스를 기능별로 나누지 않고 딱 하나만 두는 것입니다. 그 하나가 **게이트웨이**(Gateway)입니다. WhatsApp, Telegram, Slack, Discord, Signal, iMessage 같은 메신저 연결을 게이트웨이 혼자 전부 소유합니다. 대화 세션의 상태도, 예약 작업을 깨우는 스케줄러도 이 프로세스 안에 있습니다. 채널마다 프로세스를 따로 띄우는 길도 있었을 텐데, 왜 한 몸에 모았는지가 이 단원의 주제입니다.

```
[WhatsApp] [Telegram] [Slack] [Discord] [Signal] [iMessage]
     │          │        │        │        │         │
     ▼          ▼        ▼        ▼        ▼         ▼
┌───────────── Gateway: 단일 상주 프로세스 ─────────────┐
│   모든 채널 연결 소유 + 세션 상태 + cron 스케줄러     │
│   WebSocket 서버: 127.0.0.1:18789                     │
└───────────────────────▲───────────────────────────────┘
                        │ WebSocket 접속
              [CLI] [Web Control UI] [macOS 앱]
```

그림 가운데의 127.0.0.1:18789부터 풀겠습니다. 콜론 앞은 IP 주소, 뒤는 **포트**(port)입니다. IP 주소가 네트워크에서 어느 컴퓨터인지를 가리킨다면, 포트는 그 컴퓨터 안에서 어느 프로그램이 받을지를 가르는 번호입니다. 서버가 특정 주소와 포트를 차지하고 듣기 시작하는 일은 **바인딩**(binding)이라고 합니다.

IP 주소 중에서도 127.0.0.1은 특별한 값입니다. 어느 컴퓨터에서든 자기 자신을 가리키는 **루프백**(loopback) 주소이기 때문입니다. 여기에 바인딩한 서버에는 같은 머신 안의 프로그램만 접속할 수 있고, 네트워크 건너편에서는 아예 닿지 못합니다. 게이트웨이의 기본값이 루프백인 이유는 안전입니다. 밖으로 열 필요가 없는 문은 처음부터 만들지 않는 쪽이 낫습니다.

게이트웨이가 이 주소에서 여는 것은 **WebSocket** 서버입니다. 웹의 기본 규약인 HTTP는 요청 하나에 응답 하나를 돌려주고 대화를 끝냅니다. 클라이언트가 묻기 전에는 서버가 먼저 말을 걸 수 없습니다. WebSocket은 한 번 맺은 연결을 계속 유지하며, 양쪽이 아무 때나 상대에게 보낼 수 있는 양방향(full-duplex) 규약입니다. 에이전트의 답변을 생성되는 대로 흘려보내고, 새 메시지 도착 같은 사건을 즉시 알리려면 이 통로가 필요합니다.

이제 왜 전부를 프로세스 하나에 모았는지로 넘어갑니다. OpenClaw 문서가 공식적으로 첫손에 꼽는 근거는 세션 충돌 방지입니다. WhatsApp 같은 채널은 한 호스트에 로그인 세션을 하나만 허용합니다. 여러 프로세스가 제각기 연결을 시도하면 나중 연결이 앞선 세션을 밀어내 로그인이 깨집니다. 연결을 소유하는 프로세스가 하나뿐이면 이 충돌은 구조적으로 일어나지 않습니다. 채널 관리 지점이 한곳에 모이고, 기능마다 포트가 하나씩 늘어나는 파편화가 사라지는 이점도 따라옵니다. 브라우저용 대시보드까지 같은 포트에서 함께 서빙합니다.

이 구조에는 이름 붙은 개념 둘이 겹쳐 있습니다. 하나는 **control plane**입니다. 네트워크 설계에서는 데이터를 실어 나르는 경로와 그 경로를 제어하는 층을 구분하는데, 제어하는 쪽을 control plane이라 부릅니다. OpenClaw에서는 게이트웨이가 그 층입니다. 에이전트에게 내리는 명령, 상태 조회, 채널을 켜고 끄는 일이 전부 게이트웨이 한 곳을 거칩니다. 다른 하나는 **message broker**입니다. 여러 발신지의 메시지를 받아 알맞은 목적지로 중계하는 중간자라는 뜻입니다. 게이트웨이는 채널에서 온 메시지를 에이전트 세션으로 나르고, 응답을 원래 채널로 되돌립니다.

상태의 관점에서 보면 셋째 개념이 나옵니다. **single-writer** 패턴입니다. 같은 상태를 여러 주체가 동시에 고치면 어느 수정이 살아남을지 알 수 없고, 반쯤 섞인 채 깨지기도 합니다. 흔한 해법은 락을 걸어 순서를 조율하는 것이지만, single-writer는 한발 앞에서 문제를 지웁니다. 쓰는 주체를 애초에 하나로 제한하면 경합이 생길 자리가 없습니다. OpenClaw에서 채널 연결과 세션 상태를 만지는 주체는 게이트웨이뿐입니다. WhatsApp 세션 충돌 방지도 결국 이 패턴의 한 사례입니다.

예약 작업도 같은 논리 위에 있습니다. OpenClaw에는 정한 시각에 에이전트를 깨우는 **cron** 스케줄러가 있습니다. 유닉스의 예약 실행 도구에서 이름을 땄고, 게이트웨이 프로세스 안에서 돕니다. 스케줄 기억을 모델에게 맡기는 구조가 아닙니다. 시각을 재다가 제때 작업을 깨우는 주체는 게이트웨이 자신입니다. 스케줄러를 바깥 프로세스로 빼면 그쪽도 채널과 세션에 손을 대야 하므로 single-writer가 무너집니다. 그래서 운영 규칙이 단순해집니다. 게이트웨이가 꺼져 있으면 예약 작업도 돌지 않습니다. cron의 문법과 세션 처리 같은 상세는 3.2.2에서 다룹니다.

정리하면, 게이트웨이는 모든 메신저 연결과 세션 상태와 cron 스케줄러를 혼자 소유하는 단일 상주 프로세스입니다. 기본값으로 루프백 주소 127.0.0.1:18789에 바인딩한 WebSocket 서버로 접속을 받습니다. 하나로 모은 공식 근거는 호스트당 하나만 허용되는 WhatsApp 세션의 충돌 방지입니다. 개념으로 보면 제어가 한곳에 모이는 control plane이자, 채널과 에이전트 사이를 중계하는 message broker입니다. 여기에 상태를 쓰는 주체를 하나로 제한하는 single-writer 패턴이 겹칩니다. 예약 작업까지 이 안에서 돌기에, 게이트웨이가 꺼지면 예약도 함께 멈춥니다.

2.2.2에서는 이 WebSocket 통로의 반대편, 게이트웨이에 접속해 명령을 보내고 이벤트를 받는 클라이언트들을 살펴봅니다.

이 단원을 마치면 채널 연결과 상태와 스케줄러를 게이트웨이 하나에 모은 이유를 세션 충돌 방지와 control plane 개념으로 설명할 수 있습니다.
