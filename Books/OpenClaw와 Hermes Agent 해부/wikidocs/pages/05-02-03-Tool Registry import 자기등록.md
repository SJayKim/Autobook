# 5.2.3 Tool Registry: import 자기등록

5.1.2에서 Hermes가 성격이 다른 여러 쓰임을 하나의 코어로 처리한다는 점을 봤습니다. 이 단원은 그 코어가 실제로 손발을 놀리는 부분, 곧 도구 시스템으로 내려갑니다. 에이전트가 파일을 읽거나 명령을 실행하거나 웹을 뒤지는 일은 전부 도구가 맡습니다. 그런데 도구가 수십, 수백 개로 늘어나면 새로운 고민이 생깁니다. 이 많은 도구를 코어가 어떻게 하나도 빠짐없이 알아보느냐입니다. 결론을 먼저 말하면, Hermes는 사람이 관리하는 중앙 목록을 아예 두지 않습니다. 도구는 코드에 불려 들어오는 순간 스스로 이름을 올립니다.

먼저 규모부터 짚겠습니다. 도구 시스템의 중심에는 tools/registry.py라는 파일이 있습니다. 여기에 약 28개의 toolset이 묶여 있고, 그 안에 담긴 도구는 70개가 넘습니다. 여기서 toolset이란 성격이 비슷한 도구를 한 묶음으로 모아 둔 꾸러미를 말합니다. 파일을 다루는 꾸러미, 셸을 다루는 꾸러미처럼 갈래별로 나뉩니다. 도구가 이렇게 많으면, 하나를 새로 만들 때마다 사람이 어딘가에 손으로 등록해 주는 방식은 금세 한계에 부딪힙니다.

이 문제를 이해하려면 낯선 말 두 개를 배경부터 풀어야 합니다. 첫째는 **import**입니다. 프로그램을 짤 때는 코드를 여러 파일에 나눠 담습니다. 한 파일에서 다른 파일에 있는 코드를 끌어와 쓰려면, 그 파일을 가져오라고 지시해야 합니다. 이 가져오기 동작이 import입니다. 어떤 파일을 import하면, 그 파일에 적힌 코드가 그 시점에 한 번 죽 실행됩니다. 이 점이 뒤에서 결정적으로 쓰입니다.

둘째는 **레지스트리**(registry)입니다. 우리말로 옮기면 등록소입니다. 프로그램 안에서 여러 부품을 이름표를 붙여 모아 두는 장부라고 보면 됩니다. 호텔 프런트의 투숙객 명부처럼, 새 부품이 생기면 이 장부에 한 줄 올리고, 나중에 이름으로 찾아 꺼내 씁니다. Hermes에서 도구를 모아 두는 이 장부가 바로 tools/registry.py 안의 registry입니다.

이제 각 도구가 자기 자신을 어떻게 설명하는지 봅니다. 도구 하나하나는 **JSON schema**로 자신을 선언합니다. JSON은 데이터를 이름과 값의 쌍으로 적는 흔한 표기 형식이고, schema는 그 데이터가 어떤 모양이어야 하는지 정해 둔 서식입니다. 둘을 합친 JSON schema는 정해진 서식으로 적은 도구 설명서입니다. 이 도구의 이름은 무엇이고, 무슨 일을 하며, 어떤 값을 넣어 불러야 하는지가 담깁니다. 모델은 이 설명서를 읽고 어떤 도구를 어떤 값으로 부를지 판단합니다. 모델이 도구를 부르기로 정하면, model_tools.py가 그 요청을 받아 실제 도구로 연결해 줍니다. 요청을 알맞은 실행 대상에게 넘겨 주는 이 연결 작업을 디스패치라고 부릅니다.

핵심은 도구가 장부에 오르는 방식입니다. 각 도구는 자기 파일이 import되는 그 순간, 스스로 registry.register()를 불러 장부에 자기 이름을 올립니다. 사람이 따로 등록해 주기를 기다리지 않습니다. 파일이 코어에 딸려 들어오기만 하면, 그 안의 등록 코드가 실행되면서 도구가 알아서 명부에 이름을 적습니다. 이 방식을 자기등록이라고 부릅니다. 의사코드로 그리면 다음과 같습니다.

```
# tools/file_tools.py  (도구 파일 하나)

def read_file(path):
    ...실제 동작...

# 파일이 import되는 그 순간 아래 한 줄이 곧바로 실행된다
registry.register(
    name="read_file",
    schema={ 도구 이름·설명·입력값 서식을 담은 JSON schema },
    func=read_file,
)
```

이 한 줄이 파일 맨 아래에 있으므로, 코어가 파일을 import하는 순간 register()가 저절로 호출됩니다. 도구는 그렇게 이름을 장부에 남기고, 코어는 나중에 장부만 훑으면 어떤 도구가 있는지 전부 알게 됩니다.

그렇다면 왜 이렇게 만들었을까요. 여기서 이 단원의 핵심 문장이 나옵니다. Hermes 문서는 도구에 **수동 등록 목록**(manual enrollment list)이 없다고 못 박습니다. 수동 등록 목록이란, 쓸 수 있는 도구의 이름을 사람이 한곳에 죽 적어 두는 중앙 명단을 말합니다. 흔한 방식이지만 약점이 뚜렷합니다. 도구를 새로 하나 만들 때마다, 그 도구 파일과는 별개로 이 중앙 명단도 반드시 함께 고쳐 줘야 합니다. 두 곳을 늘 나란히 손봐야 하는 셈입니다.

이 이중 작업이 문제를 부릅니다. 도구 파일은 잘 만들고도 중앙 명단에 한 줄 추가하기를 깜빡하면, 도구는 코드에 분명히 있는데도 에이전트가 그 존재를 모릅니다. 도구가 70개를 넘고 갈래가 28개나 되면, 이런 누락은 언젠가 반드시 납니다. 게다가 도구 파일이 자기와 상관없는 중앙 명단의 존재까지 알아야 하니, 서로 얽매이는 결합도 생깁니다. 자기등록은 이 둘을 한꺼번에 없앱니다. 등록 코드가 도구 파일 안에 함께 살기 때문에, 파일을 만들면 등록도 같이 따라오고, 파일을 지우면 등록도 함께 사라집니다. 고쳐야 할 곳이 한 군데로 줄어드니 빠뜨릴 자리 자체가 없어집니다.

이 방식이 기대는 두 개념을 이름으로 정리하면 이렇습니다. 하나는 앞서 본 **레지스트리 패턴**입니다. 부품을 중앙 장부에 이름으로 모아 두고 이름으로 찾아 쓰는 이 구조가 레지스트리 패턴입니다. 다른 하나는 **plugin discovery**입니다. 우리말로는 플러그인 자동 발견입니다. 미리 짜인 목록을 읽는 대신, 정해진 자리에 놓인 부품을 프로그램이 알아서 찾아내 등록하는 방식을 뜻합니다. Hermes에서는 도구 파일이 import되면서 스스로 등록하는 일이 곧 이 발견에 해당합니다. 파일만 제자리에 놓으면 코어가 그것을 도구로 알아본다는 감각이 핵심입니다. 이 발상은 도구에만 그치지 않습니다. MCP나 외부 플러그인, 메모리 같은 곁가지 서브시스템도 같은 registry 방식으로 코어에 느슨하게 붙습니다.

정리하면, Hermes의 도구 시스템은 tools/registry.py를 중심으로 약 28개 toolset과 70개가 넘는 도구를 품습니다. 도구마다 JSON schema로 자신을 선언하고, model_tools.py가 모델의 호출을 실제 도구로 디스패치합니다. 각 도구는 자기 파일이 import되는 순간 registry.register()로 스스로 이름을 올립니다. 그래서 사람이 관리하는 수동 등록 목록이 필요 없습니다. 중앙 명단을 고치다 빠뜨리는 실수와 도구 파일이 그 명단에 얽매이는 결합이 함께 사라집니다. 이것이 레지스트리 패턴과 plugin discovery를 도구 등록에 그대로 옮긴 모습입니다. 다음 단원 5.2.4에서는 이렇게 등록된 도구가 셸 명령을 실제로 실행할 때, 그 실행을 어느 격리 수준에서 돌릴지 고르는 여섯 가지 터미널 백엔드를 살펴봅니다. 이 단원을 마치면 도구가 import 시점에 registry.register()로 자기등록하는 구조가 수동 등록 목록을 없애 결합과 누락을 줄이는 방식을 설명할 수 있습니다.
