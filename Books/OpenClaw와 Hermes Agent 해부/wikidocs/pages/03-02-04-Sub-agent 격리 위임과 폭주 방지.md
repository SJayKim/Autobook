# 3.2.4 Sub-agent: 격리 위임과 폭주 방지

상주형 에이전트에게는 몇십 분씩 걸리는 작업이 심심치 않게 들어옵니다. 자료 다섯 건을 조사해 보고서로 묶는 일, 큰 저장소를 빌드하고 테스트까지 돌리는 일 같은 것입니다. 이런 작업을 본 세션이 직접 붙들면 두 가지 문제가 생깁니다. 먼저 2.3.2에서 본 대로 한 세션의 실행은 lane 위에서 직렬화됩니다. 긴 작업이 도는 동안 사용자가 새로 보낸 메시지는 줄을 서서 기다립니다. 에이전트가 몇십 분간 먹통이 되는 셈입니다. 또 하나는 컨텍스트입니다. 작업 중 읽은 문서와 도구 출력이 전부 본 세션의 기억에 쌓입니다. 사용자에게 필요한 것은 결과 한 건인데, 중간 산출물이 대화 기억을 잠식하고 토큰 비용을 키웁니다.

OpenClaw는 이 문제를 **sub-agent**로 풉니다. sub-agent는 본 세션과 분리된 자식 세션에서 백그라운드로 도는 위임용 에이전트입니다. 부모가 `sessions_spawn` 도구를 부르면 새 자식 세션이 만들어집니다. 세션 키는 `agent:<agentId>:subagent:<uuid>` 형태이고, 실행은 전역 subagent lane에 오릅니다. 게이트웨이는 완료를 기다리지 않고 즉시 `{ status: "accepted", runId }`를 돌려줍니다. 수락됐다는 뜻이지 끝났다는 뜻이 아닙니다.

여기서 `sessions_yield`가 짝으로 등장합니다. 스폰을 마친 부모는 이 도구로 자신의 턴을 그 자리에서 끝냅니다. 결과가 나올 때까지 턴을 붙들고 기다리면 위임한 의미가 없기 때문입니다. 자식이 일을 마치면 완료 이벤트가 부모 세션의 다음 메시지로 배달되고, 부모는 새 턴에서 결과를 받아 사용자에게 전합니다. spawn이 일을 넘기는 반쪽이라면 yield는 손을 놓는 반쪽이고, 둘은 항상 함께 쓰입니다. 흐름을 그리면 다음과 같습니다.

```
부모 세션                              자식 세션 (subagent lane)
  │
  ├─ sessions_spawn ───────────────▶  백그라운드 실행 시작
  │   <─ 즉시 { status: "accepted" }     │
  ├─ sessions_yield 로 턴 종료           │  작업 진행
  │   (그동안 부모는 자유롭게 대화)         │
  │                                     │
  │  <── 완료 알림이 다음 메시지로 push ───┘
  ▼
새 턴: 결과 수신, 사용자에게 보고
```

완료를 아는 방식이 push 기반이라는 점이 중요합니다. 반대편에 있는 방식이 **폴링**(polling), 곧 일정 간격으로 끝났는지 반복해서 물어보는 조회입니다. 에이전트에서 폴링은 확인할 때마다 모델 호출이 일어나 토큰을 태웁니다. 확인 주기 사이에 끝난 작업은 그만큼 늦게 회수됩니다. 그래서 OpenClaw는 폴링을 금지하고, 게이트웨이가 완료 시점에 알림을 부모 쪽으로 밀어 넣게 했습니다. 알림에는 중복 처리를 막는 고정 식별자가 붙고, 전달이 실패하면 큐로 돌려 지수 백오프로 재시도합니다. 부모는 가만히 있어도 결과를 놓치지 않습니다.

자식에게 어떤 기억을 들려 보낼지는 context 모드로 정합니다. 기본값 `isolated`는 자식이 빈 컨텍스트에서 출발하는 모드입니다. 지시문만 있으면 되는 조사나 빌드 작업에 알맞고, 부모의 대화 이력을 복사하지 않아 토큰도 절약됩니다. `fork`는 부모의 대화 기록을 분기해 사본으로 들려 보내는 모드입니다. 지금까지 논의한 맥락 위에서 이어 가야 하는 작업에 씁니다. 어느 모드든 자식이 자기 컨텍스트에 무엇을 쌓든 부모 세션은 오염되지 않습니다.

이제 폭주 방지입니다. 위임 구조에는 고유한 사고 유형이 있습니다. 자식이 다시 자식을 낳으며 증식하는 재귀, 한 번에 수십 개를 띄우는 과도한 분기 같은 것입니다. 이런 사고를 다룰 때 쓰는 개념이 **blast radius**(폭발 반경)입니다. 본래 폭발물이 터졌을 때 피해가 닿는 반경을 가리키는 말입니다. 시스템 설계에서는 한 구성요소가 잘못됐을 때 영향이 번지는 범위를 뜻합니다. OpenClaw 보안 모델은 에이전트가 조작될 수 있다고 전제하고, 사고 자체를 없애기보다 이 반경을 좁히는 쪽을 택합니다.

그 반경을 좁히는 것이 기본값입니다. subagents 설정의 기본은 maxSpawnDepth 1, maxChildrenPerAgent 5, maxConcurrent 8입니다. 깊이 1은 자식이 손자를 낳지 못한다는 뜻이어서 재귀 증식을 뿌리에서 끊습니다. 에이전트당 자식 5는 부모 하나가 벌일 수 있는 판의 크기를 묶습니다. 동시 실행 8은 게이트웨이 전체에서 함께 도는 자식 수를 제한해 머신 자원과 API 비용에 상한을 만듭니다. 깊이를 2로 올리면 오케스트레이터 아래 워커들을 부리는 패턴도 열립니다. 다만 이것은 운영자가 명시적으로 선택했을 때의 이야기입니다.

도구 구성도 같은 원리를 따릅니다. sub-agent에게는 기본적으로 `message` 도구를 주지 않습니다. 사용자에게 말을 거는 창구를 부모 하나로 유지하기 위해서입니다. 자식은 gateway 제어나 cron 등록 같은 도구도 함께 잃습니다. 자식이 작업 중 읽은 악성 문서에 조작당하더라도, 사용자에게 가짜 메시지를 보내거나 예약 작업을 심을 길 자체가 없습니다. 지시로 금지하는 것이 아니라 능력을 빼앗는 방식이라, 모델이 지시를 어기는 상황에서도 유효한 안전장치입니다. 일을 마친 자식 세션은 기본 60분이 지나면 아카이브로 정리됩니다.

정리하면, 오래 걸리는 작업을 본 세션이 붙들면 대화가 막히고 컨텍스트가 불어납니다. 그래서 OpenClaw는 sessions_spawn으로 작업을 자식 세션에 위임하고 sessions_yield로 턴을 즉시 끝냅니다. 완료는 폴링 없이 push 알림, 곧 다음 메시지로 돌아옵니다. 자식의 컨텍스트는 isolated나 fork 모드로 부모와 분리됩니다. 그리고 maxSpawnDepth 1, maxChildrenPerAgent 5, maxConcurrent 8, message 도구 미부여라는 기본값이 재귀 증식과 과도한 동시 실행, 무단 발신을 막아 blast radius를 좁힙니다. 별도 설정 없이 기본값 자체가 안전장치가 되는 설계입니다.

3.3.1에서는 시선을 확장 쪽으로 옮겨, SKILL.md 문서 한 장이 새 능력의 추가를 코드 배포가 아니라 프롬프트 문제로 바꾸는 방식을 살펴봅니다.

이 단원을 마치면 sessions_spawn과 sessions_yield로 백그라운드 작업을 위임하고 push 알림으로 회수하는 구조와, 스폰 깊이·자식 수·동시 실행의 기본값이 폭주를 막는 방식을 설명할 수 있습니다.
