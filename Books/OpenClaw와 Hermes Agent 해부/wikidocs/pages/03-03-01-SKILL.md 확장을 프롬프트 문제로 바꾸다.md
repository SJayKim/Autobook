# 3.3.1 SKILL.md: 확장을 프롬프트 문제로 바꾸다

2.1.2에서 Pi 최소 코어를 다뤘습니다. OpenClaw의 실행 코어가 기본으로 쥔 도구는 파일 읽기, 쓰기, 편집, 셸 실행 네 가지뿐입니다. 그렇다면 아침 브리핑 작성처럼 구체적인 능력은 어디서 오는가라는 질문이 남습니다. 이 단원에서 다루는 스킬 시스템이 그 답의 뼈대입니다.

새 능력을 붙이는 전통적인 방법부터 되짚어 봅니다. 대부분의 소프트웨어는 플러그인 형태로 확장됩니다. 개발자가 프로그램이 정한 확장 규약에 맞춰 코드를 작성하면, 사용자는 그 코드를 내려받아 설치하고 프로그램을 다시 시작합니다. 이 방식에서 확장은 언제나 코드 배포 문제였습니다. 확장을 만들 수 있는 사람은 해당 언어를 다루는 개발자로 좁혀집니다. 플러그인과 본체는 버전이 맞아야 하고, 잘못 만든 코드는 프로그램 전체를 멈춰 세우기도 합니다.

OpenClaw의 **스킬**(skill)은 이 경로를 따르지 않습니다. 스킬의 실체는 SKILL.md라는 이름의 마크다운 문서 한 장입니다. 문서에는 어떤 작업을 어떤 절차로 처리하라는 설명이 적혀 있고, 이 글을 읽는 주체는 프로그램이 아니라 모델입니다. 기능을 코드로 구현해 런타임에 꽂는 대신, 일하는 방법을 글로 적어 모델에게 읽히는 방식입니다.

SKILL.md 맨 위에는 본문과 성격이 다른 머리말 구역이 있습니다. 마크다운 생태계에는 문서 상단을 대시 세 개(---)로 감싸 문서 정보를 적는 관례가 있는데, 이 구역을 **frontmatter**라고 부릅니다. frontmatter의 내용은 **YAML**이라는 표기법으로 적습니다. YAML은 이름과 값을 콜론으로 이어 한 줄씩 나열하는 데이터 표기법이고, 사람이 그대로 읽고 쓸 만큼 단순한 것이 특징입니다. 실제 스킬 하나를 보면 구조가 분명해집니다.

```markdown
---
name: standup-brief
description: 오늘 일정과 미완료 작업을 모아 아침 브리핑을 만든다.
---

# 아침 브리핑 절차

캘린더에서 오늘 일정을 읽는다.
TODO.md에서 끝나지 않은 항목을 골라낸다.
둘을 합쳐 세 줄 요약을 만들어 사용자에게 보낸다.
```

frontmatter의 name은 스킬의 식별자이면서 **슬래시 명령**의 이름이기도 합니다. 슬래시 명령이란 채팅 메시지 앞에 빗금을 붙여 /standup-brief처럼 기능을 직접 부르는 관례를 말합니다. description은 이 스킬이 무엇을 하는지 한 줄로 요약한 설명이고, 모델이 지금 이 스킬을 쓸지 판단하는 근거가 됩니다. 대시 아래의 본문은 스킬을 쓰기로 했을 때 따를 실제 절차입니다.

이 문서가 모델에게 닿는 과정이 스킬 시스템의 핵심 설계입니다. 세션이 시작될 때 OpenClaw는 먼저 로드할 스킬 목록을 확정합니다. 그런 다음 각 스킬의 이름과 설명을 compact XML 블록, 곧 꺾쇠 태그로 감싼 짧은 목록으로 컴파일해 시스템 프롬프트에 주입합니다. 스킬 전문이 통째로 들어가지는 않고, 요약 항목만 실립니다. 모델은 이 목록으로 자신에게 어떤 능력이 있는지 압니다. 실제로 필요해지면 Read 도구로 해당 SKILL.md를 열어 본문의 절차를 따릅니다. 흐름을 그리면 다음과 같습니다.

```
skills/ 아래의 SKILL.md들
      │ 세션 시작
      ▼
로드할 스킬 확정 → 이름·설명만 추려 compact XML 블록으로 컴파일
      │
      ▼
시스템 프롬프트에 주입 (스킬당 약 24토큰)
      │
      ▼
모델이 목록에서 능력을 인지 → 필요할 때 본문을 읽어 절차 수행
```

주입 목록은 세션 시작 시점의 스냅샷입니다. SKILL.md를 고친 내용은 변경이 감지되거나 새 세션이 열릴 때 반영됩니다.

이 구조의 프롬프트 비용은 예측 가능합니다. 주입 항목의 골격이 약 97자로 정해져 있고 여기에 이름과 설명의 길이가 더해지므로, 스킬 하나의 몫은 대략 24토큰입니다. 스킬이 늘면 비용도 그 수에 비례해 선형으로 늘어나서, 열 개면 240토큰 남짓으로 어림할 수 있습니다. 이 비용은 세션마다 프롬프트에 상시 붙기 때문에, 쓰지 않는 스킬을 내려 두는 것이 곧 토큰 절약입니다.

스킬이 발동되는 경로는 두 갈래입니다. 모델이 목록을 보고 스스로 꺼내 쓰는 경로와, 사용자가 슬래시 명령으로 직접 부르는 경로입니다. frontmatter의 선택 항목 둘이 이 두 경로를 각각 여닫습니다. user-invocable은 사용자 호출 쪽 스위치이며 기본값은 true입니다. 이를 false로 바꾸면 슬래시 명령으로는 부를 수 없고, 모델의 판단으로만 쓰입니다. disable-model-invocation은 반대쪽 스위치입니다. 기본값 false를 true로 바꾸면 스킬이 프롬프트에서 빠지고, 방금 본 토큰 비용도 함께 사라집니다. 모델은 그 존재를 모르게 되고, 사용자가 슬래시 명령으로 부를 때만 동작합니다. 모델이 임의로 발동하면 곤란한 작업을 명시적 호출 전용으로 묶을 때 씁니다.

코드 플러그인과 나란히 놓으면 이 전환의 의미가 드러납니다. 플러그인은 런타임에 새 실행 경로를 심는 일이라 개발, 배포, 재시작이 반드시 따라붙습니다. 스킬은 실행 경로를 하나도 늘리지 않습니다. 모델이 손에 쥔 도구는 코어의 네 가지 그대로이고, 스킬이 보태는 것은 그 도구를 어떤 순서와 기준으로 쓸지에 대한 지식입니다. 확장이 코드 배포 문제에서 프롬프트 문제로 바뀐 것입니다. 절차를 글로 적을 수 있으면 개발자가 아니어도 스킬을 만들 수 있습니다. 에이전트가 직접 스킬 문서를 써서 스스로를 확장하는 길도 같은 원리로 열립니다. 2.1.2에서 말한 software building software가 스킬 층에서 실체를 얻는 셈입니다. 설치는 파일 한 장을 스킬 디렉토리에 놓는 것으로 끝나고, 재시작 대신 새 세션이면 충분합니다.

정리하면, 스킬은 name과 description을 YAML frontmatter에 담은 SKILL.md 마크다운 문서입니다. 세션이 시작되면 각 스킬의 이름과 설명이 compact XML 블록으로 컴파일되어 시스템 프롬프트에 주입됩니다. 비용은 스킬당 약 24토큰으로 선형이라 미리 어림할 수 있습니다. user-invocable과 disable-model-invocation은 사용자 호출 경로와 모델 호출 경로를 따로 여닫습니다. 런타임에 실행 경로를 심는 코드 플러그인과 달리 스킬은 절차 지식을 프롬프트에 싣는 확장이고, 그 덕분에 확장의 주체가 개발자 밖으로 넓어졌습니다.

다만 모든 스킬이 모든 환경에서 성립하는 것은 아닙니다. 3.3.2에서는 조건이 맞는 환경에서만 스킬을 로드하는 게이팅과, 같은 이름의 스킬이 겹칠 때 승자를 가리는 6단계 소스 우선순위를 다룹니다.

이 단원을 마치면 frontmatter를 가진 SKILL.md가 세션 시작 시 프롬프트에 컴파일 주입되는 구조와, 확장을 코드 배포가 아닌 프롬프트 문제로 바꾼 의미를 설명할 수 있습니다.
