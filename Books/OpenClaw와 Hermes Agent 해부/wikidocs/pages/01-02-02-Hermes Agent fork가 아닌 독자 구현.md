# 1.2.2 Hermes Agent: fork가 아닌 독자 구현

1.2.1에서 OpenClaw가 개인 실험에서 출발해 거대한 오픈소스 현상으로 성장한 과정을 살펴봤습니다. 이 단원의 주인공은 같은 장르의 두 번째 축인 Hermes Agent입니다. 서버에 상주하며 메신저로 부리는 개인 에이전트라는 큰 그림은 OpenClaw와 같습니다. 그래서 처음 접하는 사람은 OpenClaw를 복제해 만든 파생 프로젝트라고 짐작하기 쉽습니다. 이 짐작이 왜 틀렸는지, 코드를 한 줄도 공유하지 않는 두 프로젝트가 어떻게 경쟁하는지가 이 단원의 내용입니다.

그 짐작에는 배경이 있습니다. 오픈소스 프로젝트는 코드 전체가 공개되어 있고, 라이선스가 허용하는 범위에서 누구나 복사해 쓸 수 있습니다. 여기서 나온 관행이 **fork**입니다. fork는 기존 프로젝트의 코드를 통째로 복사한 뒤, 원본과 별개로 자기 방향대로 개발을 이어 가는 일을 말합니다. 어떤 프로젝트가 인기를 얻으면 그 코드를 출발점 삼는 변형판이 자연스럽게 생겨납니다. OpenClaw는 복사와 수정을 폭넓게 허용하는 MIT 라이선스를 쓰므로, fork로 경쟁작을 만들어도 규칙 위반이 아닙니다. 폭발적으로 성장한 오픈소스 곁에 fork가 줄지어 서는 일은 이 생태계의 익숙한 풍경입니다.

그런데 Hermes Agent는 이 익숙한 경로를 밟지 않았습니다. 만든 곳은 **Nous Research**라는 AI 연구조직입니다. 이 조직은 에이전트를 내놓기 전부터 Hermes라는 이름의 언어 모델 시리즈를 훈련해 공개해 온, 모델 훈련이 본업인 곳입니다. 에이전트의 이름도 자사 모델 시리즈에서 그대로 가져왔습니다. 개인 개발자의 실험에서 출발한 OpenClaw와 달리, Hermes Agent는 시작부터 모델을 만드는 조직의 작품이었습니다.

공개일은 2026년 2월 25일입니다. 라이선스는 OpenClaw와 같은 MIT입니다. 발표 행사도 긴 소개 글도 없이, '당신과 함께 성장하는 오픈소스 에이전트'라는 소셜 미디어 글 한 줄로 GitHub 저장소를 알린 것이 전부였습니다. 성장 속도는 발표의 조용함과 정반대였습니다. 공개 6주 만에 프로젝트 지지 표시인 GitHub 스타가 5만 7천을 넘었고, 코드에 참여한 기여자도 270명을 넘었습니다. 2026년 7월 21일 기준으로는 약 21만 8천 스타에 이릅니다. 여러 회사의 모델을 한 창구로 중계하는 OpenRouter 집계에서는 2026년 5월 중순부터 하루 토큰 처리량으로 OpenClaw를 앞질렀습니다. 석 달이 채 지나기 전에 후발 주자가 원조와 같은 체급에 올라선 것입니다.

fork가 아니라는 사실은 코드베이스에서 바로 확인됩니다. fork는 원본 코드의 복사에서 출발하므로, 원본이 쓰던 프로그래밍 언어와 구조를 그대로 물려받습니다. 그런데 두 프로젝트는 언어부터 다릅니다. OpenClaw는 웹 생태계에서 널리 쓰이는 TypeScript로 작성되어 Node.js 환경에서 돌아갑니다. 반면 Hermes Agent는 저장소 코드의 약 82퍼센트가 Python입니다. 서로 다른 언어 사이에서는 코드를 이어받는 일 자체가 성립하지 않습니다. Hermes Agent는 OpenClaw가 증명해 보인 장르를 참고하되, 같은 종류의 시스템을 백지에서 새로 구현한 독자 프로젝트입니다. 실제로 공식 문서 어디에도 fork라는 서술이 없습니다.

두 프로젝트가 정면으로 부딪히는 지점은 코드가 아니라 사용자입니다. 이미 OpenClaw를 길들여 쓰던 사람에게 갈아타기는 간단한 일이 아닙니다. 그동안 쌓인 설정과 기억을 버려야 하고, 에이전트에게 익혀 둔 작업 절차인 스킬도 다시 가르쳐야 하기 때문입니다. 소프트웨어를 바꿀 때 따라오는 이런 부담을 **전환 비용**이라고 부릅니다. 전환 비용이 클수록 사용자는 먼저 자리 잡은 쪽에 머무르므로, 후발 주자에게는 이 비용을 낮추는 일이 곧 경쟁 전략이 됩니다. Hermes Agent가 내놓은 답이 **마이그레이션**, 곧 한 시스템의 데이터와 설정을 다른 시스템으로 옮겨 주는 작업입니다. `hermes claw migrate` 명령을 실행하면 OpenClaw의 설정, 기억, 스킬, API 키를 자동으로 가져옵니다. 초기 설정 마법사도 컴퓨터에서 OpenClaw의 데이터 디렉토리(`~/.openclaw`)를 감지하면 이주를 먼저 제안합니다. 코드로 원본에 올라타는 fork의 길 대신, 데이터 호환으로 원본 사용자의 전환 비용을 지우는 길을 택한 것입니다.

마지막으로 Hermes Agent가 스스로를 무엇이라 부르는지 봐 둘 필요가 있습니다. 공식 저장소의 소개 첫머리는 자신을 **self-improving agent**, 곧 자기 개선 에이전트로 정의합니다. 경험에서 스킬을 만들어 내고, 그 스킬을 쓰면서 손질하고, 알게 된 지식을 남기도록 스스로를 재촉한다는 것입니다. 여기에 자기 과거 대화를 검색하는 능력과, 세션을 거듭할수록 사용자 이해를 깊게 쌓는 능력까지 묶어 내장 학습 루프라고 부릅니다. 공개를 알린 '함께 성장하는 에이전트'라는 문구와 같은 방향의 자기 정의입니다. 원조를 베끼는 대신 쓸수록 나아지는 에이전트라는 다른 축을 세워 차별화하겠다는 선언인 셈입니다. 이 학습 루프가 실제로 어떤 장치로 구현되는지는 이 책의 Phase 6에서 자세히 다룹니다.

정리하면, Hermes Agent는 Hermes 모델 시리즈로 알려진 연구조직 Nous Research가 2026년 2월 25일 공개한 상주형 개인 에이전트입니다. 공개 6주 만에 스타 5만 7천을 모으고 석 달째에는 하루 토큰 처리량으로 원조를 앞지를 만큼 빠르게 성장했습니다. TypeScript 기반의 OpenClaw와 달리 Python으로 처음부터 새로 쓴 독자 구현이므로 fork가 아닙니다. 경쟁은 `hermes claw migrate` 명령으로 사용자 데이터를 이주시키는 마이그레이션 전략으로 풉니다. 그리고 스스로를 경험에서 배우는 self-improving agent로 정의합니다.

1.2.3에서는 시선을 두 프로젝트 바깥으로 넓혀, 재단 설립과 빅테크의 대응, 규제 당국의 움직임까지 이 장르가 산업에 일으킨 파장을 살펴봅니다.

이 단원을 마치면 Hermes Agent가 OpenClaw의 fork가 아닌 Python 독자 구현이라는 사실과, 마이그레이션 도구로 경쟁하는 전략을 설명할 수 있습니다.
