# 6.1.4 FTS5 결정적 회상과 memory provider

6.1.1에서 Hermes의 기억이 세 계층으로 나뉜다는 것을 살펴봤습니다. 항상 주입되는 소형 큐레이션 파일이 첫 계층, 반복 작업의 노하우를 담는 스킬이 셋째 계층입니다. 이 단원은 그 사이에 있는 둘째 계층, 곧 지나간 모든 대화를 담아 두는 세션 아카이브를 어떻게 검색하는가를 다룹니다. 여기서 Hermes는 다른 많은 에이전트와 다른 길을 택합니다. 무엇이 어떻게 다른지, 그리고 그 검색을 외부 도구로 어떻게 보강하는지를 차례로 보겠습니다.

먼저 배경입니다. 에이전트가 사용자와 나눈 대화는 끝나도 사라지지 않고 어딘가에 원문 그대로 쌓입니다. Hermes는 이것을 홈 디렉터리의 state.db라는 파일 하나에 담습니다. 이 파일은 SQLite로 만들어져 있습니다. **SQLite**는 별도의 서버 없이 파일 하나로 동작하는 작은 관계형 데이터베이스입니다. 설치도 접속 설정도 필요 없이 프로그램이 파일을 직접 열어 읽고 씁니다. 개인 머신에 상주하는 에이전트에게는 무거운 데이터베이스 서버보다 이런 파일형 저장소가 잘 맞습니다. CLI로 나눈 대화든 메신저로 주고받은 대화든, 모든 세션이 이 하나의 파일에 모입니다.

그런데 대화가 수백, 수천 건 쌓이면 원하는 내용을 어떻게 찾을까요. 여기서 등장하는 것이 **FTS5**입니다. FTS5는 SQLite에 내장된 전문검색 기능입니다. 전문검색이란 문서 전체의 낱말을 미리 색인해 두고, 질의어가 들어오면 그 낱말이 등장한 문서를 곧바로 찾아 주는 방식입니다. 도서관에서 책 뒤의 색인을 보고 특정 단어가 나온 쪽을 펴는 것과 같습니다. Hermes는 세션 원문을 요약하지 않고 그대로 색인해 두므로, 과거 대화를 압축 없이 낱말 단위로 뒤질 수 있습니다.

이 방식을 제대로 이해하려면 많은 에이전트가 쓰는 다른 방법과 나란히 놓아야 합니다. 바로 **벡터 유사도 검색**입니다. 이것은 문장을 임베딩이라는 숫자 벡터로 바꾼 뒤, 질의 벡터와 방향이 가까운 문장을 의미가 비슷한 것으로 보고 꺼내 오는 방식입니다. 낱말이 정확히 겹치지 않아도 뜻이 통하면 찾아 준다는 장점이 있습니다. 대신 무엇이 가장 가깝다고 판단하는지는 임베딩 모델과 거리 계산에 달려 있어, 같은 질문을 던져도 결과가 그때그때 조금씩 흔들릴 수 있습니다. 그래서 이 회상은 확률적입니다.

Hermes가 FTS5를 고른 이유가 바로 이 지점에 있습니다. 많은 에이전트는 벡터 검색으로 확률적 회상을 하지만, Hermes는 FTS5 전문검색으로 결정적 회상을 합니다. **결정적 회상**이란 같은 질의를 넣으면 언제나 같은 결과가 나오는 성질을 말합니다. 낱말이 있으면 찾고 없으면 못 찾는다는 규칙이 명확하므로, 어제 찾힌 대화는 오늘도 같은 질의로 찾힙니다. 회상이 예측 불가능하게 달라지지 않는다는 점이 개인 비서의 신뢰를 떠받칩니다. 두 방식의 차이를 정리하면 다음과 같습니다.

| 구분 | FTS5 전문검색 (Hermes) | 벡터 유사도 검색 |
|---|---|---|
| 찾는 기준 | 낱말의 일치 | 임베딩 벡터의 의미적 근접 |
| 같은 질의의 결과 | 항상 동일 (결정적) | 흔들릴 수 있음 (확률적) |
| 원문 보존 | 요약 없이 원문 색인 | 벡터로 변환해 저장 |

실제 검색이 어떻게 흐르는지를 그려 보겠습니다. 에이전트가 예전에 나눈 이야기를 떠올려야 할 때, sessions 계열 도구로 아카이브를 조회합니다. 흐름은 다음과 같습니다.

```
[에이전트: 과거 대화가 필요하다고 판단]
        │
        ▼
 질의어를 FTS5 색인에 던진다
        │
        ▼
 state.db에서 낱말이 일치하는 세션을 찾는다
        │
        ▼
 요약이 아닌 원문 그대로 결과를 돌려받는다
        │
        ▼
 그 내용을 근거로 다음 판단을 이어 간다
```

항상 주입되는 기억 파일이 상한에 묶여 작게 유지되는 것과 달리, 이 세션 아카이브에는 용량 제한이 없습니다. 큐레이션된 요약은 프롬프트에 늘 얹혀 있어 작아야 하지만, 아카이브는 필요할 때만 검색으로 여는 창고이므로 무제한으로 쌓아 두고 결정적으로 꺼내 쓰는 것입니다.

FTS5만으로 부족할 때를 위해 Hermes는 **memory provider 플러그인**이라는 확장 지점을 둡니다. 이것은 기억 저장과 회상을 외부 시스템에 맡기거나 보강하는 교체형 부품입니다. 기본 검색은 낱말 일치에 기대므로, 낱말은 다르지만 뜻이 같은 것을 찾거나 사실들 사이의 관계를 따라가야 할 때는 한계가 있습니다. 이때 memory provider가 지식그래프나 시맨틱검색 같은 능력을 얹어 줍니다. 지원하는 provider는 Mem0를 비롯해 여덟 종이며, hermes memory setup 명령으로 설정합니다. 결정적 회상을 기본으로 두되, 필요하면 의미 기반 회상을 덧대는 이원 구조인 셈입니다.

마지막으로 안전 문제가 하나 남습니다. 아카이브든 provider든, 대화 내용을 저장한다는 것은 위험한 문장까지 함께 담을 수 있다는 뜻입니다. 사용자가 붙여 넣은 로그에 에이전트를 조종하려는 인젝션 문구가 섞여 있거나, API 키 같은 자격증명이 그대로 흘러 들어올 수 있습니다. 그래서 Hermes는 저장하기 전에 인젝션 패턴과 자격증명 유출 여부를 스캔합니다. 오염된 기억이 이후 판단을 좌우하는 문제와, 저장된 비밀이 새어 나가는 문제를 저장 시점에서 막으려는 장치입니다. 이 스캔은 기억을 자산이자 부채로 보는 보안 관점과 맞닿아 있으며, 7장에서 다룰 방어 설계와 이어집니다.

정리하면, Hermes는 지나간 대화를 SQLite 파일에 원문으로 담고 FTS5 전문검색으로 조회합니다. 벡터 유사도 검색이 같은 질문에도 결과가 흔들리는 확률적 회상인 데 반해, FTS5는 같은 질의에 늘 같은 결과를 주는 결정적 회상을 제공합니다. 세션 아카이브는 용량 제한 없이 쌓이고 검색으로만 열리며, memory provider 플러그인 여덟 종으로 지식그래프와 시맨틱검색을 보강할 수 있습니다. 그리고 저장 전 인젝션과 자격증명 스캔으로 기억 오염과 비밀 유출을 걸러 냅니다. 6.2.1에서는 이렇게 쌓인 경험이 어떻게 재사용 가능한 스킬로 굳어지는지를 skill_manage 도구를 통해 다룹니다. 이 단원을 마치면 FTS5 전문검색이 벡터 유사도 검색과 어떻게 다른지, 세션 아카이브 검색이 어떻게 흐르며 memory provider 플러그인과 저장 전 스캔이 각각 어떤 역할을 하는지 설명할 수 있습니다.
