# 2.1.1 brains and muscles: 뇌와 근육의 분리

1.2.1에서는 OpenClaw가 어떤 프로젝트이고 어떻게 오픈소스 현상으로 자랐는지를 바깥에서 살펴봤습니다. 이번 단원부터 시작하는 2부는 시선을 안쪽으로 돌립니다. 시스템을 부품 단위로 뜯어 각 부품이 왜 그 자리에 있는지 따지는 해부가 여기서 시작됩니다. 첫 질문은 이 시스템에서 판단하는 지능이 어디에 있는가입니다.

답은 간단합니다. 지능은 설치된 컴퓨터 안에 없습니다. OpenClaw를 설치할 때 내 머신에 들어오는 것은 TypeScript로 작성된 프로그램이지, 언어 모델이 아닙니다. 무엇을 할지 정하는 판단은 매번 외부의 LLM(대규모 언어 모델) API에 물어서 받아 옵니다. 처음 보면 낯선 구조입니다. AI 에이전트를 설치했는데 정작 AI는 함께 오지 않기 때문입니다.

OpenClaw는 이 구조를 **brains and muscles**, 곧 뇌와 근육이라고 부릅니다. 판단을 맡는 뇌는 외부의 LLM이고, 그 판단을 현실로 옮기는 근육은 내 머신에 상주하는 로컬 설치본이라는 사상입니다. 뇌 자리에는 Claude나 GPT 같은 상용 모델은 물론, Ollama나 LM Studio로 내 컴퓨터에 띄운 로컬 모델도 꽂을 수 있습니다. 어느 쪽이든 뇌는 갈아 끼울 수 있는 부품으로 취급됩니다.

두 부품의 역할 분담을 조금 더 자세히 보겠습니다. 뇌가 하는 일은 텍스트를 읽고 다음 텍스트를 내놓는 것뿐입니다. 지금까지의 대화, 저장된 기억, 방금 실행한 도구의 결과를 텍스트로 건네받아, 다음 행동과 사용자에게 보낼 응답을 텍스트로 돌려줍니다. 그 텍스트를 실제 행동으로 바꾸는 일은 전부 근육의 몫입니다. 근육은 WhatsApp이나 Telegram 같은 메신저 연결을 하루 종일 유지하고, 파일을 읽고 쓰고, 셸 명령을 실행합니다. 예약된 시간에 스스로 깨어나 일하고, 대화가 끝난 뒤에도 기억을 디스크에 남깁니다.

```
   [뇌]  외부 LLM API (Claude, GPT, 로컬 Ollama)
     ▲                          │
     │  대화·기억·도구 결과      │  다음 행동과 응답
     │  (텍스트로 올려 보냄)     ▼  (텍스트로 내려받음)
   [근육]  내 머신의 OpenClaw 설치본
     ├─ 메신저 연결 유지 (WhatsApp, Telegram 등)
     ├─ 파일 읽고 쓰기 · 셸 명령 실행
     ├─ 예약 작업으로 스스로 깨어나기
     └─ 기억 보관 (평문 마크다운 파일)
```

이 분리가 만드는 성질 가운데 가장 중요한 것은, 뇌를 갈아 끼워도 몸이 그대로 남는다는 점입니다. OpenClaw는 모델을 API로 제공하는 주체를 **프로바이더**라고 부르고, 모델을 provider/model 형식으로 가리킵니다. anthropic/claude-sonnet-4-6을 쓰다가 openai/gpt-5.6으로 바꾸는 일은 설정을 고치는 수준의 변경입니다. 그렇게 바꿔도 메신저 연결은 끊기지 않고 스킬도 그대로이며, 무엇보다 기억이 남습니다. OpenClaw의 기억은 모델 내부가 아니라 내 디스크의 평문 마크다운 파일에 쌓이기 때문입니다. 기억이 특정 모델의 형식에 묶여 있지 않으니, 새 뇌는 이전 뇌가 적어 둔 파일을 그대로 읽고 이어서 일합니다. 주 모델이 응답하지 못할 때 미리 정한 예비 모델 순서로 자동 전환하는 **폴백 체인**도 이 분리 위에서 성립합니다.

뇌를 밖에 둔 구조의 장점은 셋으로 정리됩니다. 먼저 성능입니다. 내 머신에 GPU 한 장 없어도 각 시점의 최전선 모델을 뇌로 쓸 수 있습니다. 다음은 교체의 자유입니다. 시스템 전체가 특정 회사에 묶이지 않으므로, 더 나은 모델이 나오면 갈아타고 한쪽이 막히면 다른 쪽으로 옮기면 됩니다. 마지막은 요구 사양입니다. 근육이 하는 일은 연결 유지와 파일 조작이라 무겁지 않아서, 저사양 서버로도 하루 종일 돌릴 수 있습니다.

대가도 그만큼 뚜렷합니다. 첫째는 의존입니다. 네트워크가 끊기거나 프로바이더에 장애가 나면, 몸은 멀쩡히 돌아도 판단이 멈춥니다. 가격과 이용 정책이 바뀌는 위험도 내가 통제할 수 없는 바깥 변수로 남습니다. 둘째는 비용입니다. 판단 한 번이 곧 API 호출 한 번이고, 상주형 에이전트는 사람이 말을 걸지 않는 시간에도 주기적으로 모델을 불러 상황을 점검합니다. 설치해 두었더니 조용히 월 수십 달러가 청구되었다는 경험담이 흔한 이유입니다. 셋째는 프라이버시입니다. 기억 파일 자체는 내 디스크에 있지만, 판단이 필요한 순간마다 대화 내용과 기억의 일부가 외부 API로 전송됩니다. 개인 비서로 쓸수록 민감한 내용이 오가는 만큼 이 흐름을 알고 써야 합니다. 로컬 모델을 뇌로 꽂으면 세 가지 대가를 상당 부분 피할 수 있지만, 이번에는 판단 품질과 하드웨어 부담을 내가 떠안는 교환이 됩니다.

이 구조는 공부의 초점도 바꿉니다. 뇌가 갈아 끼우는 부품이라면, OpenClaw라는 시스템의 정체성은 몸 쪽에 있습니다. 어떤 모델을 쓰는지는 설정값 하나에 지나지 않습니다. 반면 메시지가 들어오는 길, 대화 상태의 격리, 기억이 쌓이는 규칙, 능력이 추가되는 방식, 보안 경계의 위치는 전부 몸의 설계입니다. 그래서 이 책의 해부도 모델 비교가 아니라 게이트웨이, 세션, 메모리, 스킬, 보안이라는 몸의 부위를 차례로 따라갑니다. 앞으로 이어질 장들의 지도가 사실상 이 문단에 담겨 있습니다.

정리하면, OpenClaw는 판단하는 뇌를 외부 LLM에 두고, 메신저 연결과 파일 조작과 기억 보관을 맡는 근육을 내 머신에 둡니다. 기억이 모델 밖의 평문 파일에 남기 때문에 뇌를 갈아 끼워도 몸과 기억은 이어지고, 그 덕에 최전선 모델의 성능과 교체의 자유를 얻습니다. 대신 API 의존, 토큰 비용, 대화가 외부로 나가는 프라이버시 문제를 대가로 치릅니다. 뇌가 부품인 이상, 이 시스템을 이해하는 일은 곧 몸의 설계를 이해하는 일이 됩니다.

2.1.2에서는 OpenClaw의 두 번째 사상인 software building software와 그 바탕이 되는 Pi 최소 코어를 살펴봅니다.

이 단원을 마치면 외부 LLM을 뇌로, 로컬 설치본을 근육으로 나누는 OpenClaw의 기본 사상과 역할 분담, 그 장점과 대가를 설명할 수 있습니다.
