# 4.1.1 identity, scope, model: 3겹 방어 순서

여기까지는 OpenClaw가 어떻게 만들어져 있는지를 따라왔습니다. 게이트웨이가 모든 채널 연결을 소유하고, 세션이 대화 문맥을 격리하고, 메모리와 스킬이 에이전트의 능력을 쌓아 올리는 구조였습니다. 이번 Phase부터는 질문을 바꿔 이 구조를 어떻게 지키는지를 봅니다. 상주형 에이전트는 메신저로 누구나 말을 걸 수 있는 입구를 열어 두고, API 키와 셸 접근 권한을 한곳에 모아 둔 실행 주체입니다. 편리함을 만드는 조건이 그대로 공격자가 노릴 표적이 됩니다.

OpenClaw의 공식 보안 모델은 이 문제에 세 겹의 방어로 답하고, 겹을 세우는 순서까지 못 박습니다. identity first, scope next, model last, 곧 신원 확인이 먼저, 허용 범위가 다음, 모델의 분별력이 마지막입니다. 첫 번째 겹인 **identity**는 누가 말하는가를 묻습니다. 지금 도착한 메시지의 발신자가 이 에이전트의 운영자가 맞는지를 다른 무엇보다 먼저 확인하는 겹입니다.

두 번째 겹인 **scope**는 무엇까지 허용하는가를 묻습니다. 발신자가 운영자로 확인됐다 해도, 그 대화가 촉발할 수 있는 행동의 폭을 미리 좁혀 둡니다. 세 번째 겹인 **model**은 모델 자체의 분별력입니다. 앞의 두 겹을 지나 모델에게 도착한 입력 속에서 수상한 지시를 가려내는 일은 결국 모델의 몫입니다. OpenClaw는 그 몫을 방어의 출발점이 아니라 마지막 겹에 둡니다. 세 겹을 지나는 흐름을 그리면 다음과 같습니다.

```
외부 입력 (메시지, 웹 콘텐츠, 첨부 파일)
         │
         ▼
[1겹 identity: 누가 말하는가]      ← DM pairing, allowlist
         │ 통과
         ▼
[2겹 scope: 무엇까지 허용하는가]   ← 도구 정책, 실행 승인, 샌드박스, bind
         │ 통과
         ▼
[3겹 model: 모델의 분별력]         ← 인젝션에 강한 모델 tier
         │
         ▼
행동 실행 (앞의 두 겹이 좁혀 둔 범위 안에서만)
```

이 순서는 2.1.3에서 다룬 개인 비서 신뢰 경계에서 나옵니다. OpenClaw는 게이트웨이 하나당 신뢰하는 운영자 한 명을 전제합니다. 서로 믿지 못하는 여러 사용자가 한 인스턴스를 나눠 쓰는 상황은 처음부터 지원 범위 밖입니다. 경계가 이렇게 그어져 있으면 방어의 첫 질문은 자연히 지금 말하는 사람이 경계 안의 그 한 명인가가 됩니다. identity 겹이 뚫리면 낯선 발신자가 운영자의 자격으로 말하게 됩니다. 그 뒤의 겹들은 주인의 지시와 침입자의 지시를 구분할 근거 자체를 잃습니다. 반대로 identity가 지켜지면 남는 위협은 뚜렷하게 좁혀집니다. 운영자 자신이 에이전트에게 읽히는 콘텐츠, 예컨대 웹 검색 결과나 붙여넣은 로그 속에 숨어 들어오는 조작입니다.

공식 문서는 그래서 이 모델을 조작을 가정하고 blast radius를 제한하는 설계라고 요약합니다. **blast radius**는 폭발 반경, 곧 무언가 잘못됐을 때 피해가 미치는 범위를 가리키는 말입니다. 에이전트는 출처를 통제할 수 없는 텍스트를 일상적으로 읽습니다. 그래서 OpenClaw는 웹 검색 결과와 붙여넣은 로그, 첨부 파일을 기본적으로 적대적인 입력으로 취급합니다. 목표는 모델이 절대 속지 않게 만드는 것이 아닙니다. 모델이 언젠가 속는다는 사실을 전제에 넣고, 속은 순간 실행될 수 있는 행동의 폭을 scope 겹이 미리 줄여 두는 것입니다.

이 사상은 규칙을 두 종류로 나누는 데서 분명해집니다. **soft guidance**는 시스템 프롬프트에 적어 두고 모델이 따르기를 기대하는 지시입니다. **hard enforcement**는 모델의 판단과 무관하게 코드가 강제하는 규칙입니다. 프롬프트에 위험한 명령은 거부하라고 적어도, 정교하게 짜인 입력은 모델을 설득해 그 지시를 넘게 만들 수 있습니다. 반면 도구 정책이 특정 도구를 아예 주지 않거나 샌드박스가 작업 공간을 읽기 전용으로 묶으면, 모델이 아무리 설득당해도 그 행동은 일어나지 않습니다. OpenClaw 문서도 프롬프트 인젝션은 시스템 프롬프트만으로 해결되지 않는다고 명시합니다. 강제는 도구 정책과 실행 승인, 샌드박스, 채널 allowlist의 몫이라는 것입니다.

모델의 분별력이 마지막 방어선인 이유가 여기서 나옵니다. 조작될 수 있다고 전제한 부품에게 문지기 자리를 맡길 수는 없습니다. 다만 마지막이라는 말이 덜 중요하다는 뜻은 아닙니다. 인젝션에 버티는 힘은 모델 tier마다 다르고, 공식 문서는 도구를 쓰거나 비신뢰 콘텐츠를 읽는 에이전트를 약한 tier에서 돌리지 말라고 경고합니다. 앞의 두 겹이 좁혀 놓은 범위 안에서 마지막 오판 하나를 더 걸러내는 것이 모델 겹의 역할입니다.

각 겹을 실제로 담당하는 장치를 개괄하면 다음과 같습니다.

| 겹 | 묻는 질문 | 담당 장치 |
|---|---|---|
| identity | 누가 말하는가 | DM pairing, 발신자 allowlist |
| scope | 무엇까지 허용하는가 | 도구 정책, 실행 승인, Docker 샌드박스, gateway bind 설정 |
| model | 통과한 입력을 가려내는가 | 인젝션에 강한 모델 tier 선택 |

각 장치가 실제로 어떻게 동작하는지는 Phase 4의 나머지 단원에서 겹의 순서대로 하나씩 다룹니다. 여기서는 어느 겹에 어떤 장치가 속하는지의 지도만 잡아 두면 됩니다.

정리하면, OpenClaw의 보안 모델은 누가 말하는가(identity), 무엇까지 허용하는가(scope), 모델이 얼마나 가려내는가(model)의 순서로 방어를 세 겹 쌓습니다. 이 순서는 게이트웨이당 신뢰 운영자 한 명이라는 신뢰 경계 전제에서 출발합니다. 그리고 모델은 조작될 수 있다는 가정 아래, 사고가 나도 피해 범위인 blast radius가 작게 끝나도록 앞의 두 겹이 행동의 폭을 미리 제한합니다. 프롬프트에 적은 지시는 soft guidance일 뿐이고, hard enforcement는 pairing과 allowlist, 도구 정책과 샌드박스처럼 모델 바깥의 장치가 담당합니다. 그래서 모델의 분별력은 첫 방어선이 아니라 모든 겹을 통과한 입력을 상대하는 마지막 방어선입니다.

4.1.2에서는 방어의 첫 관문에 해당하는 두 장치, DM pairing과 gateway bind가 낯선 발신자와 외부 접속을 어떻게 걸러내는지 살펴봅니다.

이 단원을 마치면 identity, scope, model 세 겹의 방어가 왜 이 순서로 쌓이는지를 신뢰 경계 전제와 조작 가정을 근거로 설명할 수 있습니다.
