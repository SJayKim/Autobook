# 3.2.2 Cron: 문맥 없는 무인 자동화

3.2.1에서 다룬 heartbeat는 메인 대화 세션 안에서 도는 주기 점검이었습니다. 대화 문맥을 안다는 것이 강점이지만, 모든 자동화에 그 문맥이 필요하지는 않습니다. 매주 월요일 아침에 밤사이 쌓인 소식을 요약해 보내는 작업을 떠올려 보겠습니다. 이 일은 어제 사용자와 나눈 대화와 관련이 없습니다. 오히려 지난 대화가 섞여 들면 결과가 흐려집니다. OpenClaw는 이런 작업을 위해 heartbeat와 별도의 장치를 둡니다. 이 단원의 주제인 cron입니다.

**cron**은 본래 유닉스 계열 운영체제에서 정해진 시각에 명령을 대신 실행해 주는 스케줄러의 이름입니다. OpenClaw는 이 이름과 개념을 가져와 게이트웨이 내장 스케줄러로 구현했습니다. 2.2.1에서 본 단일 상주 프로세스인 게이트웨이가 시계를 보고 있다가, 때가 되면 에이전트 실행을 깨웁니다. 시간을 재는 주체는 모델이 아니라 게이트웨이 프로세스입니다. 따라서 게이트웨이가 꺼져 있으면 예약된 작업도 실행되지 않습니다. 등록된 작업은 저마다 task 레코드로 SQLite에 저장됩니다. 스케줄러 전체를 잠시 멈추려면 환경변수 OPENCLAW_SKIP_CRON=1로 끕니다.

작업을 언제 실행할지는 네 가지 문법 중 하나로 지정합니다. --at은 지정한 시각에 한 번만 실행하는 일회성 예약입니다. --every는 "30분마다"처럼 일정한 간격의 반복입니다. --on-exit은 넷 중 유일하게 시각이 아니라 사건을 조건으로 삼아, 종료 시점에 발동하는 예약입니다. 마지막으로 "매주 월요일 9시"처럼 달력 규칙을 따르는 반복은 **cron 표현식**으로 적습니다. cron 표현식은 다섯 자리로 실행 시각의 규칙을 나타내는 표기법입니다. 왼쪽부터 분, 시, 일, 월, 요일을 뜻하며, 별표(*)는 "모든 값"을 뜻합니다.

```
0 9 * * 1
│ │ │ │ └─ 요일 (0~6, 0은 일요일, 1은 월요일)
│ │ │ └─── 월 (1~12)
│ │ └───── 일 (1~31)
│ └─────── 시 (0~23)
└───────── 분 (0~59)
```

"0 9 * * 1"을 읽으면 "일과 월은 조건 없이, 요일이 월요일인 날의 9시 0분"이 됩니다. 즉 매주 월요일 오전 9시입니다. 이 표현식으로 주간 브리핑 작업을 등록하는 명령은 다음과 같습니다.

```
openclaw cron create "0 9 * * 1" "Summarize overnight updates." \
  --session isolated --announce
```

첫 번째 인자는 방금 읽은 스케줄입니다. 두 번째 인자는 실행할 때 에이전트에게 건넬 지시문입니다. --session isolated는 이 작업을 **격리 세션**에서 돌리라는 지정이고, --announce는 결과를 메신저 채널로 알리라는 지정입니다.

--session 옵션이 받는 세션 타입은 네 가지입니다. main은 에이전트의 메인 세션에서 실행해 평소 대화 문맥 위에 작업을 얹습니다. current는 지금 활성화된 세션을 그대로 이어 씁니다. session:커스텀ID 형식은 지정한 이름의 세션을 실행 때마다 재사용합니다. 커스텀 세션은 데일리 스탠드업처럼 실행끼리 문맥을 쌓되 메인 대화와는 섞이지 말아야 하는 작업에 맞습니다. 그리고 기본값인 isolated는 실행할 때마다 새 세션을 만들어, 매번 새 대화 기록(transcript)과 새 세션 ID로 깨끗하게 시작합니다. 격리 세션의 기록은 sessionRetention 설정(기본 "24h")에 따라 하루 뒤 정리됩니다.

실행 시 에이전트에게 무엇을 시킬지는 **페이로드**로 정합니다. --message는 자연어 지시문을 전달합니다. 위 예시의 두 번째 인자가 바로 이 메시지 페이로드입니다. --command는 명령을 실행시키고, 이 밖에 --system-event와 --script 형태도 있습니다.

결과를 어디로 보낼지는 **딜리버리**로 정합니다. announce는 결과를 메신저 채널로 보내 사용자에게 알립니다. webhook은 정해 둔 웹 주소로 HTTP 요청을 보내는 방식으로, 결과를 외부 시스템에 넘길 때 씁니다. none은 어디에도 알리지 않는 선택이며, 결과는 세션 기록으로만 남습니다.

여기서 격리 실행의 성격이 분명해집니다. 격리 세션의 cron 작업이 도는 동안 사람은 자리에 없습니다. 새벽에 시작한 작업은 모호한 대목을 사용자에게 되물을 수 없고, 이전 대화에서 힌트를 얻을 수도 없습니다. 그래서 격리 실행의 원칙은 최종 응답 자체가 산출물이어야 한다는 것입니다. 실행을 마치며 내놓는 마지막 응답이 announce나 webhook에 실려 나가는 결과물의 전부입니다. 페이로드의 지시문도 이 전제에 맞춰야 합니다. 대화하듯 짧게 던지는 대신, 무엇을 어떤 형태로 내놓을지까지 지시문 안에 자기완결적으로 담아야 합니다. 사람이 없어도 돌고, 실행마다 새 세션으로 시작하며, 마지막 응답으로 말을 맺는 것이 cron이 맡는 무인 자동화입니다.

정리하면, cron은 게이트웨이 내장 스케줄러가 돌리는 무인 배치 장치입니다. 실행 시점은 일회성 --at, 간격 반복 --every, 달력 규칙의 cron 표현식, 종료 트리거 --on-exit 네 가지 문법으로 지정합니다. 세션 타입은 main, isolated, current, 커스텀 네 가지이며 기본은 실행마다 새 세션을 여는 isolated입니다. 시킬 일은 --message, --command 같은 페이로드로 건네고, 결과는 announce, webhook, none 중 하나의 딜리버리로 전달합니다. 사람이 개입하지 않는 격리 실행에서는 최종 응답이 곧 산출물이므로, 지시문을 자기완결적으로 작성해야 합니다.

3.2.3에서는 문맥을 아는 heartbeat와 문맥 없는 cron을 나란히 놓고, 어떤 작업에 무엇을 골라야 하는지 선택 기준을 다룹니다.

이 단원을 마치면 게이트웨이 내장 스케줄러가 격리 세션으로 실행하는 cron 작업의 스케줄 문법 4종, 세션 타입, 페이로드, 딜리버리 방식을 구분하고, 격리 실행에서 최종 응답이 곧 산출물인 이유를 설명할 수 있습니다.
