# 7.2.2 운영 체크리스트와 알려진 함정

7.2.1에서 설치 명령 한 줄이 실제로 무엇을 하는지 뜯어봤습니다. 이제 그렇게 설치한 에이전트를 개인 실험용이 아니라 상시 돌아가는 프로덕션 환경에 올릴 차례입니다. 프로덕션이라는 말의 뜻은 여기서 하나뿐입니다. 사람이 지켜보지 않는 동안에도 계속 켜져 있고, 진짜 계정과 진짜 셸에 손을 대는 상태입니다. 이런 상태에서는 실수 하나가 화면 안에 갇히지 않고 바깥 세계로 새어 나갑니다. 그래서 이 단원은 배포 직전에 훑을 점검 항목을 순서대로 정리합니다.

먼저 이 점검이 왜 필요한지부터 짚겠습니다. 상주형 에이전트는 API 키와 셸 접근을 한 몸에 모아 둔 고가치 표적입니다. 7.1.2에서 봤듯이 승인 게이트를 느슨하게 열면 편의와 위험이 함께 커집니다. 그래서 점검은 세 갈래로 나뉩니다. 실행을 가둬 두는 격리, 아무나 붙지 못하게 막는 인증, 에이전트가 스스로 만든 것을 사람이 다시 보는 검토입니다.

첫 번째 갈래는 격리입니다. 여기서 가장 먼저 손봐야 할 설정이 **terminal.backend**입니다. 이 값은 에이전트가 셸 명령을 어디에서 실행할지 정합니다. 기본값인 local은 명령을 여러분의 호스트에서 그대로 돌리므로, 위험 명령 검사가 켜져 있어도 파일시스템 전체가 노출됩니다. 프로덕션에서는 이 값을 docker로 바꾸는 것이 권장됩니다. 그러면 명령이 컨테이너 안에서만 돌고, 컨테이너의 벽이 곧 보안 경계가 됩니다. docker 백엔드의 격리 방식은 5.2.4에서 다뤘으므로 여기서는 넘어갑니다.

격리에서 한 가지를 더 챙겨야 합니다. 컨테이너를 쓰더라도 그 안에서 프로세스를 최고 권한 계정으로 돌리면 격리 효과가 반감됩니다. 리눅스에는 무엇이든 할 수 있는 특권 계정인 **root**가 있고, 그 밖의 제한된 일반 계정을 비root라고 부릅니다. 에이전트는 반드시 비root 계정으로 실행해야 합니다. 격리와 계정 분리는 둘이 한 세트라고 기억하면 됩니다.

두 번째 갈래는 인증입니다. 이 대목에 이 단원 전체에서 가장 강한 금지가 하나 있습니다. **GATEWAY_ALLOW_ALL_USERS**라는 환경변수를 true로 켜는 일은 절대 하지 않습니다. 이 값을 켜면 게이트웨이가 발신자를 가리지 않고 전부 받아들입니다. 즉 여러분의 에이전트에 아무나 메시지를 보내 셸을 부릴 수 있게 된다는 뜻입니다. 대신 붙을 수 있는 사람을 명시적으로 좁혀야 합니다. 플랫폼별 허용 목록으로 알려진 발신자만 통과시키고, 모르는 상대는 시간제한이 걸린 페어링 코드로 승인받게 합니다. 이 코드는 여덟 자리에 한 시간짜리 수명을 갖고, 발급 빈도와 실패 횟수에도 제한이 걸려 있어 무작정 시도하는 공격을 막습니다.

인증에는 웹 화면 문제도 포함됩니다. 에이전트가 제공하는 웹 UI를 인증 없이 공개 인터넷에 올리는 것은 금지입니다. 이것은 추상적인 걱정이 아니라 실제 사고의 교훈입니다. 4.2.2에서 다룬 OpenClaw 대규모 노출 사태가 정확히 이 지점에서 터졌습니다. 원인은 소프트웨어 결함이 아니라 배포 관행이었습니다. 기본값은 외부에서 닿지 않는 안전한 설정이었는데, 빠른 시작을 위해 모든 주소에서 접속을 받도록 열고 인증까지 꺼 버린 사용자가 대량으로 있었습니다. 그 결과 인터넷에 그대로 드러난 인스턴스가 수만 개 발견됐고 상당수는 원격 명령 실행까지 가능했습니다. 안전한 기본값도 배포 편의 앞에서 무너진다는 사실을 이 사태가 보여 줍니다.

세 번째 갈래는 검토입니다. 6.2.1과 6.2.3에서 이 에이전트가 작업 경험에서 스킬을 스스로 만들고 다듬는다는 점을 봤습니다. 편리하지만, 자가 생성된 스킬은 완성품이 아니라 초안으로 취급해야 합니다. 사람이 쓰지 않은 절차 문서가 시스템 프롬프트에 끼어들어 이후 판단을 좌우하기 때문입니다. 여기서 켜 둘 설정이 **write_approval**입니다. 이 값을 켜면 에이전트가 만든 스킬이 곧바로 적용되지 않고 대기 영역에 먼저 쌓입니다. 그러면 대기 목록을 확인하고 내용을 검토한 뒤 승인하는 흐름을 사람이 거치게 됩니다. 자가 개선이라는 편의에 사람의 확인이라는 제동을 끼워 넣는 장치입니다.

마지막으로 운영 중에 놓치기 쉬운 두 가지를 덧붙입니다. 하나는 로그입니다. 에이전트가 무슨 명령을 실행하고 무엇을 승인했는지는 로그 디렉토리에 남습니다. 배포 후에는 이 로그를 주기적으로 살펴, 예상 밖의 행동이 없는지 확인하는 습관이 필요합니다. 다른 하나는 환경별 함정인데, 특히 윈도우 안에서 리눅스를 돌리는 WSL2에서 두드러집니다. 이 환경에서는 서비스를 백그라운드에서 자동으로 관리해 주는 systemd라는 구성 요소가 불안정합니다. 그래서 게이트웨이를 서비스로 등록해 상주시키면 자꾸 멈춥니다. 권장되는 우회책은 게이트웨이를 백그라운드 서비스가 아니라 화면 앞단(foreground)에서 직접 띄우고, 터미널을 닫아도 세션이 유지되는 도구인 tmux로 감싸 계속 돌리는 방식입니다. 각 항목을 한눈에 정리하면 다음과 같습니다.

```
갈래   점검 항목                         조치
────  ──────────────────────────────   ─────────────────────────
격리   terminal.backend                  local → docker
       실행 계정                          root 금지, 비root 실행
인증   GATEWAY_ALLOW_ALL_USERS           절대 true 금지
       발신자 범위                        플랫폼 허용 목록 + 페어링
       웹 UI                             인증 없는 공개 금지
검토   자가 생성 스킬                     write_approval 켜고 대기→검토→승인
운영   로그                              logs 디렉토리 주기 점검
       WSL2                             systemd 대신 foreground + tmux
```

정리하면, 상주형 에이전트를 프로덕션에 올릴 때는 격리와 인증과 검토를 순서대로 점검합니다. 격리에서는 terminal.backend를 docker로 바꾸고 비root로 실행합니다. 인증에서는 GATEWAY_ALLOW_ALL_USERS를 절대 켜지 않고, 허용 목록과 페어링으로 발신자를 좁히며, 웹 UI를 인증 없이 공개하지 않습니다. 검토에서는 write_approval로 자가 생성 스킬을 초안으로 붙잡아 사람이 승인하게 합니다. 운영 중에는 로그를 살피고, WSL2에서는 systemd 대신 foreground와 tmux를 씁니다.

7.2.3에서는 이렇게 올린 에이전트가 실제로 얼마를 청구하는지, 그리고 캐시가 무효화되며 비용이 튀는 함정을 다룹니다.

이 단원을 마치면 상주형 에이전트를 프로덕션에 배포할 때 지켜야 할 격리, 인증, 검토 점검 항목을 짚고, GATEWAY_ALLOW_ALL_USERS 금지와 WSL2의 systemd 함정 같은 환경별 주의점을 설명할 수 있습니다.
