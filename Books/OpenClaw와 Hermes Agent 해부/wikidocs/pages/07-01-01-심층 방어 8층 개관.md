# 7.1.1 심층 방어 8층 개관

지금까지 이 책은 Hermes Agent를 설계 원칙에서 시작해 단일 코어, 프롬프트 조립, 메모리, 스킬, 모델 연동까지 차례로 뜯어 왔습니다. 이제 마지막으로 남은 부품이 보안입니다. Hermes의 보안은 다른 부품처럼 하나의 모듈로 존재하지 않고, 시스템 곳곳에 여러 겹으로 흩어져 있습니다. 이번 7부는 그 겹들을 하나씩 펼쳐 보는데, 첫 단원인 이 자리에서는 전체 지도를 먼저 그립니다. 즉 방어가 몇 겹이고 각 겹이 무엇을 막는지, 그리고 이 겹들이 왜 하필 이렇게 짜였는지를 개관합니다.

시작하기 전에 이 책이 앞서 4부에서 다룬 이야기를 하나 떠올려 두면 좋습니다. OpenClaw의 보안은 4.1.1에서 identity, scope, model이라는 세 겹의 방어로 정리한 바 있습니다. 누가 말하는지를 먼저 가리고, 무엇까지 허용할지를 정하고, 모델의 분별력은 맨 마지막 방어선으로 둔다는 순서였습니다. Hermes도 같은 사상 위에 서 있되, 겹의 수가 여덟으로 늘고 각 겹이 더 구체적인 지점을 맡습니다.

먼저 이 사상의 이름부터 짚겠습니다. Hermes의 공식 문서는 자신의 보안 모델을 **심층 방어**(defense in depth)라고 부릅니다. 심층 방어란 방어를 한 겹에만 걸지 않고 여러 겹으로 쌓아, 앞 겹이 뚫려도 그 뒤의 겹이 다시 막게 하는 설계 방침을 말합니다. 성벽 하나에 모든 것을 거는 대신 해자, 성벽, 내성을 차례로 두는 옛 축성 방식과 발상이 같습니다. 어느 한 겹도 완벽할 수 없다는 전제에서 출발한다는 점이 핵심입니다. 그래서 한 겹이 실수로 뚫려도 공격이 곧장 시스템 전체로 번지지 못하고, 다음 겹이 피해를 다시 가둡니다.

이 방침 아래 Hermes가 두는 여덟 겹은 다음과 같습니다. 각 겹이 어느 지점에서 무엇을 막는지를 한 줄씩 정리합니다.

| 겹 | 막는 대상 |
|---|---|
| 인가 | 허락되지 않은 발신자가 에이전트를 부리는 것 |
| 명령 승인 | 위험한 셸 명령이 사람 확인 없이 실행되는 것 |
| 파일 쓰기 안전 | 자격증명 파일 등 민감한 경로가 덮어써지는 것 |
| 컨테이너 격리 | 실행이 호스트 시스템으로 새어 나가는 것 |
| MCP 필터 | 외부 도구 연결로 자격증명이 빠져나가는 것 |
| 컨텍스트 스캔 | 파일에 숨은 지시문이 프롬프트에 섞이는 것 |
| 세션 격리 | 한 대화의 내용이 다른 대화로 넘어가는 것 |
| 입력 정화 | 바깥에서 들어온 데이터가 명령처럼 취급되는 것 |

여덟 겹을 관통하는 흐름을 위에서 아래로 읽으면 하나의 방어선처럼 보입니다. **인가**는 애초에 누가 에이전트에게 말을 걸 수 있는지를 가립니다. 허락된 사람만 통과시켜 문 앞에서 낯선 발신자를 돌려보내는 겹입니다. 그 문을 지나 실제로 명령이 들어오면 **명령 승인**이 받습니다. 위험도가 높은 셸 명령을 사람의 확인 없이 곧바로 실행하지 않게 붙잡아 두는 겹입니다. 이 두 겹의 세부는 바로 다음 단원에서 이어집니다.

명령이 승인을 통과해 파일과 프로세스를 건드리기 시작하면 그다음 겹들이 작동합니다. **파일 쓰기 안전**은 자격증명이 담긴 경로처럼 절대 덮어써서는 안 되는 파일을 보호하고, 컨테이너 격리는 실행 자체를 호스트에서 떼어 놓습니다. 이 격리 겹은 5.2.4에서 다룬 터미널 백엔드 여섯 종의 선택과 맞닿아 있습니다. 실행 환경을 어느 수준으로 가둘지가 곧 이 겹의 두께를 정합니다.

나머지 네 겹은 데이터가 드나드는 통로를 지킵니다. MCP 필터는 외부 도구를 연결할 때 자격증명이 함께 새어 나가지 않도록 걸러 냅니다. 컨텍스트 스캔은 에이전트가 읽어 들이는 파일 안에 숨은 지시문이 시스템 프롬프트로 섞여 드는 것을 막고, 세션 격리는 한 대화의 내용이 다른 대화로 넘어가지 못하게 칸을 나눕니다. 마지막 **입력 정화**는 바깥에서 들어온 데이터를 명령이 아니라 데이터로만 다루게 만들어, 외부 텍스트가 에이전트를 조종하려는 시도를 무력화합니다.

여기서 자연스럽게 떠오르는 물음이 있습니다. 왜 Hermes는 처음부터 이렇게 여러 겹을 기본으로 갖췄을까요. 답은 순서에 있습니다. Hermes는 OpenClaw의 사고를 지켜본 뒤에 만들어졌습니다. 이 책이 4부에서 다룬 대규모 노출 사태와 공급망 사고는 후발 시스템에게는 남의 실패를 미리 읽은 교본이었습니다. 먼저 나온 시스템이 어디서 뚫렸는지를 알고 출발했으니, 그 지점들을 처음부터 겹으로 덮어 둘 수 있었던 것입니다. 이것이 **후발 이점**입니다. 뒤늦게 온 쪽이 앞선 쪽의 사고 기록을 밑천 삼아 더 촘촘한 기본값에서 시작하는 이점을 말합니다.

이 이점은 하나의 전략으로 굳어져 있습니다. Hermes는 위험한 동작을 편하게 열어 두고 필요할 때 잠그게 하는 대신, 처음부터 잠가 두고 필요할 때 열게 합니다. 이것이 **보수적 기본값**입니다. 기본 상태를 안전한 쪽으로 맞춰 두어, 사용자가 아무것도 설정하지 않아도 위험한 문이 닫혀 있게 하는 방침을 말합니다. 4부에서 본 노출 사태의 교훈이 여기서 분명해집니다. 그 사태의 원인은 소프트웨어 결함이라기보다 안전한 기본값을 배포 편의를 위해 사용자가 스스로 풀어 버린 데 있었습니다. 보수적 기본값은 그 반대 방향을 택합니다. 열려면 사용자가 의식적으로 손을 대야 하도록 마찰을 남겨, 무심코 위험이 열리는 일을 줄입니다.

이런 설계의 결과로 자주 인용되는 기록이 하나 있습니다. 2026년 5월 기준으로 Hermes에는 에이전트 자체의 알려진 보안 취약점, 곧 CVE가 한 건도 없었습니다. 같은 시기 OpenClaw가 짧은 기간에 여러 건의 취약점을 겪은 것과 대비됩니다. 다만 이 숫자를 그대로 우열의 증거로 읽어서는 곤란합니다. Hermes는 나중에 나온 시스템이라 검증된 기간이 짧고, 세상에 드러난 노출 면적도 앞선 시스템과 다릅니다. 오래 그리고 넓게 노출된 시스템일수록 취약점이 더 많이 발견되기 마련입니다. 그러니 0건은 보수적 기본값이 효과를 냈을 가능성을 보여 주는 기록이되, 그 자체를 완성된 안전의 증명으로 단정할 수는 없습니다.

정리하면, Hermes의 보안은 심층 방어라는 방침 아래 여덟 겹으로 짜여 있습니다. 인가와 명령 승인이 문 앞과 실행 직전을 지키고, 파일 쓰기 안전과 컨테이너 격리가 파일과 실행을 가두며, MCP 필터, 컨텍스트 스캔, 세션 격리, 입력 정화가 데이터가 드나드는 통로를 막습니다. 이 여러 겹이 처음부터 기본으로 켜져 있는 까닭은 OpenClaw의 사고를 미리 읽은 후발 이점과, 위험을 닫아 두고 필요할 때 열게 하는 보수적 기본값 전략에 있습니다. 에이전트 자체 취약점 0건이라는 기록은 그 전략의 효과를 시사하되, 짧은 검증 기간과 좁은 노출 면적을 함께 감안해 읽어야 합니다.

7.1.2에서는 이 여덟 겹의 첫머리인 명령 승인을 확대해, smart approvals가 위험도를 어떻게 나누는지, 그리고 전체 자동 승인을 켜는 YOLO 모드에서도 끝내 뚫리지 않는 안전의 바닥이 무엇인지를 살펴봅니다.

이 단원을 마치면 인가부터 입력 정화까지 여덟 겹 방어의 각 겹이 무엇을 막는지 나열하고, OpenClaw의 사고를 지켜본 후발 이점이 보수적 기본값에 반영된 맥락을 설명할 수 있습니다.
