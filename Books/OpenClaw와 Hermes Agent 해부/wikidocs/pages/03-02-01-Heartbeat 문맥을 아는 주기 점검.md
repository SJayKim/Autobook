# 3.2.1 Heartbeat: 문맥을 아는 주기 점검

챗봇에는 공통된 한계가 하나 있습니다. 사용자가 말을 걸기 전에는 아무것도 하지 않는다는 점입니다. 질문이 오면 답하고, 다음 질문이 올 때까지 완전히 멈춰 있습니다. 그런데 상주형 에이전트가 지향하는 것은 메신저 너머의 개인 비서입니다. 비서에게 맡기고 싶은 일에는 "급한 메일이 오면 알려 달라"거나 "돌려 둔 작업이 끝났는지 봐 달라" 같은 것이 많습니다. 이런 일은 사용자가 물어봐 주기를 기다려서는 성립하지 않습니다. 시키지 않아도 스스로 상황을 살피고 필요할 때 먼저 움직이는 성질을 **능동성**(proactive agency)이라고 부릅니다. OpenClaw는 이 능동성을 heartbeat와 cron이라는 두 장치로 구현하며, 이 단원은 그중 heartbeat를 봅니다.

**heartbeat**는 메인 대화 세션 안에서 일정 간격으로 자동 실행되는 턴입니다. 사용자가 아무 메시지도 보내지 않아도, 시스템이 기본 30분마다 에이전트에게 한 턴을 밀어 넣습니다. 심장이 일정한 박동을 유지하듯 주기적으로 맥을 짚는다고 해서 붙은 이름입니다. 중요한 것은 실행 위치입니다. 새 세션을 여는 것이 아니라 지금 진행 중인 메인 세션 안에서 도는 턴이므로, 에이전트는 그동안의 대화를 아는 상태로 점검합니다. 단원 제목의 "문맥을 아는"이 가리키는 지점입니다. 주기 기본값은 "30m"이고, "0m"으로 두면 heartbeat가 꺼집니다. 완화 조항도 하나 있습니다. Anthropic 계정의 OAuth나 토큰 인증으로 모델을 쓰면 기본 주기가 1시간으로 늘어납니다. 30분마다 모델을 부르는 비용 부담을 인증 방식에 따라 낮춘 조정입니다.

heartbeat 턴이 돌면 에이전트가 하는 일은 정해져 있습니다. 3.1.1에서 본 워크스페이스의 HEARTBEAT.md 파일을 읽고, 거기 적힌 항목을 점검합니다. HEARTBEAT.md는 heartbeat 전용의 짧은 체크리스트입니다. "받은 메일함에 급한 것이 있는지 본다"나 "어제 걸어 둔 빌드가 끝났는지 확인한다"처럼 주기적으로 살필 일을 적어 둡니다. 비서가 30분마다 수첩을 들춰 보고, 처리할 게 있으면 그때 말을 거는 구조인 셈입니다. 이 파일은 짧게 유지하는 것이 원칙입니다. 내용 전부가 30분마다 모델 입력으로 들어가므로, 파일 길이가 곧 반복 비용이 되기 때문입니다.

점검했는데 처리할 것이 없으면 어떻게 될까요. 에이전트는 **HEARTBEAT_OK**라는 약속된 문자열을 반환하고, 시스템은 이 응답을 기본적으로 사용자에게 보여 주지 않습니다. 이 침묵 규약이 없다면 메신저에는 30분마다 "확인했지만 별일 없습니다"라는 보고가 쌓입니다. 하루면 마흔여덟 번입니다. 능동적으로 점검하되 보고는 처리할 일이 있을 때만 한다는 비대칭이 heartbeat 설계의 핵심입니다. 존재 이유를 한 문장으로 줄이면, 시키지 않아도 배경의 긴급한 일을 표면화하되 대화를 소음으로 덮지 않는 것입니다.

주기가 되었다고 무조건 도는 것도 아닙니다. 건너뛰는 조건이 세 가지 있습니다. 실행 큐나 레인, cron 작업이 바쁘게 돌아가는 중이면 건너뜁니다. 사용자와의 실제 대화가 우선이고, 배경 점검이 그 줄에 끼어들면 안 되기 때문입니다. activeHours로 지정한 활동 시간대 밖이어도 건너뜁니다. HEARTBEAT.md가 비어 있으면 모델을 부를 이유가 없으므로, empty-heartbeat-file이라는 사유와 함께 건너뜁니다. 세부 규칙 하나가 같은 태도를 보여 줍니다. heartbeat 턴은 세션의 리셋 타이머를 연장하지 않습니다. 비활동이 이어지면 세션을 새로 시작하는 정책을 쓰더라도, 배경 점검은 활동으로 치지 않는다는 뜻입니다.

이 동작은 agents.defaults.heartbeat 아래 설정으로 조율합니다.

```
agents.defaults.heartbeat
 ├─ every            점검 주기 (기본 "30m", "0m"이면 비활성)
 ├─ model            heartbeat 턴에만 쓸 모델 지정
 ├─ isolatedSession  메인 세션과 분리된 세션에서 실행
 ├─ lightContext     HEARTBEAT.md 위주의 가벼운 문맥만 유지
 ├─ skipWhenBusy     큐가 바쁘면 건너뛰기
 └─ activeHours      heartbeat를 허용할 시간대
```

isolatedSession과 lightContext는 비용을 겨냥합니다. 기본 상태의 heartbeat는 메인 세션의 대화 기록을 그대로 얹고 돕니다. 문맥을 아는 점검이라는 강점이 여기서 나오지만, 대화가 길수록 30분마다 그 기록만큼의 입력 토큰을 다시 낸다는 뜻이기도 합니다. isolatedSession을 켜면 heartbeat를 분리된 세션에서 돌려 이 비용을 끊습니다. lightContext를 켜면 문맥을 통째로 싣는 대신 HEARTBEAT.md 위주의 가벼운 문맥만 유지합니다. 둘 다 문맥의 깊이와 토큰 비용을 맞바꾸는 선택입니다. skipWhenBusy와 activeHours는 실행 여부를 겨냥합니다. 바쁠 때 건너뛰는 동작과 허용 시간대를 각각 제어해, 점검이 대화를 방해하거나 새벽에 혼자 도는 일을 막습니다.

경계도 분명합니다. heartbeat는 메인 세션의 문맥에 얹혀 있으므로, "매주 월요일 아침 브리핑"처럼 문맥 없이 매번 깨끗하게 시작해야 하는 무인 작업에는 맞지 않습니다. 그런 작업은 격리된 세션에서 도는 별도의 장치가 맡습니다.

정리하면, heartbeat는 메인 세션 안에서 기본 30분마다 자동으로 도는 턴입니다. Anthropic OAuth나 토큰 인증에서는 기본 주기가 1시간으로 완화되고, "0m"이면 꺼집니다. 턴마다 HEARTBEAT.md 체크리스트를 점검하고, 처리할 일이 없으면 HEARTBEAT_OK를 반환해 조용히 지나갑니다. 큐가 바쁘거나 activeHours 밖이거나 파일이 비어 있으면 건너뛰며, isolatedSession과 lightContext로 문맥 깊이와 토큰 비용을 맞바꿀 수 있습니다. 말을 걸어야만 움직이는 챗봇과 스스로 점검하는 비서를 가르는 능동성이 이 주기 턴 하나에서 나옵니다.

3.2.2에서는 능동성의 다른 반쪽, 곧 대화 문맥 없이 격리 세션에서 도는 무인 자동화 장치 cron을 다룹니다.

이 단원을 마치면 메인 세션 안에서 기본 30분마다 HEARTBEAT.md를 점검하고 처리할 일이 없으면 HEARTBEAT_OK로 침묵하는 heartbeat의 동작과 존재 이유를 설명할 수 있습니다.
