# 3.3.3 ClawHub: 스킬 생태계와 신뢰

3.3.2까지는 내 머신 안에 이미 있는 스킬이 어떤 조건과 순서로 로드되는지를 다뤘습니다. 이번 단원은 그 앞 단계인 조달을 다룹니다. 스킬을 애초에 어디서 구해 오는가라는 질문입니다. 직접 쓰는 스킬만으로는 생태계가 만들어지지 않습니다. 남이 만든 스킬을 이름 하나로 가져오는 통로가 있어야 하고, OpenClaw에서 그 통로가 ClawHub입니다.

배경 개념부터 정리합니다. 프로그래밍 생태계에는 오래전부터 **레지스트리**(registry)라는 장치가 있습니다. 누구나 자기가 만든 패키지에 이름을 붙여 올리고, 다른 사람은 그 이름만으로 내려받아 쓰는 공개 패키지 저장소입니다. Node.js 진영의 npm이 대표 사례입니다. 필요한 라이브러리를 직접 짜는 대신 설치 명령 한 줄로 가져다 쓰는 문화가 이 장치 위에서 자랐습니다. 레지스트리의 성격은 두 가지로 요약됩니다. 올리는 쪽의 문턱이 낮다는 것, 그리고 받는 쪽이 올린 사람을 대개 모른다는 것입니다. 이 두 성질이 생태계의 폭발력과 위험을 함께 만듭니다.

**ClawHub**는 OpenClaw가 이 모델을 스킬에 적용한 결과물로, clawhub.ai에서 운영되는 공개 스킬 레지스트리입니다. 커뮤니티가 만든 스킬을 검색하고 이름 하나로 설치합니다. 스킬이 컴파일된 코드 패키지가 아니라 마크다운 문서라는 점을 떠올리면 이 결합의 파급력이 보입니다. 문서 한 장이면 만들 수 있으니 올리는 사람의 폭이 넓고, 설치가 명령 한 줄이니 가져오는 비용도 낮습니다.

설치 명령은 소스를 세 형태로 받습니다.

```
openclaw skills install @owner/slug              # ClawHub 레지스트리에서
openclaw skills install git:owner/repo@ref       # git 저장소의 특정 ref에서
openclaw skills install ./my-skill --as my-tool  # 로컬 경로에서, 이름 지정
```

첫 줄의 @owner/slug는 레지스트리 좌표입니다. owner가 올린 사람의 계정, slug가 스킬 이름이라서 같은 이름의 스킬이 여럿 있어도 소유자로 구별됩니다. 둘째 줄은 레지스트리를 거치지 않고 git 저장소에서 직접 받는 형태입니다. @ref 자리에 브랜치나 태그, 커밋을 적어 특정 시점의 내용을 고정합니다. 셋째 줄은 디스크의 로컬 디렉토리를 설치하는 형태로, --as 옵션으로 설치될 이름을 따로 정합니다. 어느 형태든 --global을 붙이면 ~/.openclaw/skills에 들어갑니다. 3.3.2에서 본 관리형 디렉토리이므로 워크스페이스 하나가 아니라 그 머신의 에이전트 전체가 공유합니다.

설치와 활성화 사이에는 검증 관문이 하나 있습니다. OpenClaw는 커뮤니티 스킬을 활성화하기 전에 **trust envelope** 검증을 거치며, 이 검증 형식의 식별자가 clawhub.skill.verify.v1입니다. envelope는 봉투라는 뜻 그대로 스킬 본문을 감싸는 검증용 메타데이터입니다. 봉인이 멀쩡한 봉투가 내용물이 발송 시점 그대로임을 보여 주듯, 설치된 스킬이 레지스트리에 등록된 그대로인지를 활성화 전에 확인합니다. 식별자 끝의 v1은 검증 형식 자체의 버전 표기로, 형식이 바뀌어도 어떤 규칙으로 검증했는지 추적하게 해 줍니다. 다만 보장 범위를 정확히 읽어야 합니다. 이 검증이 답하는 질문은 "등록된 그대로 왔는가"이지 "내용이 무해한가"가 아닙니다. 악성 내용을 처음부터 정식 절차로 등록하면 봉투는 멀쩡하게 통과됩니다.

그래서 봉투 안에 무엇이 들었는지 보는 감각이 이 단원의 핵심입니다. 스킬은 세션이 시작될 때 이름과 설명이 압축 블록으로 시스템 프롬프트에 들어가고, 모델은 상황에 맞는 스킬을 골라 그 본문의 지시를 따릅니다. 또 스킬별 설정인 skills.entries 아래에 apiKey나 env를 두면, 세션 시작 시 그 값이 환경변수로 풀려 실행 환경에 놓입니다. 스킬 설치는 남이 쓴 지시문을 내 에이전트의 판단 근거에 넣는 일입니다. 설정에 따라서는 내 API 키를 그 곁에 놓아 주는 일이기도 합니다. 파일과 셸을 다루는 에이전트가 그 지시문을 읽고 움직이므로, 겉모습은 문서여도 효과는 실행 코드에 가깝습니다. 스킬을 프롬프트에 주입되는 서드파티 코드로 취급해야 하는 이유입니다. npm 패키지가 내 프로세스의 권한으로 실행되듯, 스킬의 문장은 내 에이전트의 권한으로 해석되고 실행됩니다.

이 구조가 만든 생태계의 규모는 실제로 큽니다. OpenClaw 저장소는 2026년 중반 기준 스타 38만과 기여자 500여 명 규모로 성장했습니다. 상주형 에이전트 가운데 가장 큰 커뮤니티·스킬 생태계라는 점이 OpenClaw를 고르는 대표 이유로 꼽힐 정도입니다. 원하는 능력을 검색해 명령 한 줄로 붙이는 경험, 비개발자도 문서 한 장으로 기여하는 낮은 문턱이 이 규모를 만들었습니다. 그런데 같은 문턱이 공격자에게도 낮습니다. 2026년 초 ClawHub에서는 환경변수에 담긴 API 키와 메신저 토큰, 데이터베이스 자격증명을 수집해 밖으로 보내는 악성 스킬이 잇따라 발견되었습니다. 한때 수백 개 단위로 탐지되었다는 분석도 나왔습니다. 사고의 전개와 수법은 4.2.3에서 자세히 다루므로, 여기서는 위험의 구조만 짚어 둡니다. 스킬은 환경변수와 API 키에 접근할 수 있는 서드파티 확장이고, trust envelope는 출처를 확인할 뿐 안전을 보증하지 않습니다. 그래서 실무 수칙은 보수적인 쪽에 서 있습니다. 검증한 스킬만 허용 목록에 올려 설치하고, skills.entries의 enabled 설정으로 스킬 단위 활성화를 관리합니다.

정리하면, ClawHub는 clawhub.ai에서 운영되는 OpenClaw의 공개 스킬 레지스트리입니다. openclaw skills install 명령은 @owner/slug 좌표와 git 저장소, 로컬 경로의 세 소스를 받습니다. 커뮤니티 스킬은 활성화 전에 clawhub.skill.verify.v1 trust envelope 검증을 거치지만, 이 검증은 등록된 그대로 왔는지만 확인합니다. 스킬의 지시문은 모델의 판단을 조종할 수 있고 설정된 API 키는 환경변수로 풀리므로, 스킬은 프롬프트에 주입되는 서드파티 코드로 다뤄야 합니다. 낮은 문턱은 가장 큰 스킬 생태계를 만든 힘인 동시에 악성 스킬이 흘러든 통로이기도 했습니다.

다음 4.1.1부터는 Phase 4 보안으로 넘어가, OpenClaw 보안 모델의 뼈대인 identity, scope, model 세 겹 방어가 어떤 순서로 쌓이는지 살펴봅니다.

이 단원을 마치면 ClawHub에서 스킬을 설치하는 명령 형태와 trust envelope 검증의 역할과 한계, 스킬 생태계가 가진 힘과 위험의 양면을 설명할 수 있습니다.
