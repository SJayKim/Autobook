# 2.1.1 OWASP LLM Top 10 개관

앞의 1.3.2에서 에이전트 하나를 앞에 두고 신뢰 경계를 그리는 법을 익혔습니다. 부품을 나열하고 신뢰 수준을 매겨, 외부 콘텐츠 입구와 도구 호출 출구와 메모리 통로라는 검증 지점을 그림 위에 드러냈습니다. 그런데 경계선을 그었다고 해서 무엇을 막아야 하는지까지 저절로 알게 되지는 않습니다. 그 입구로 실제로 어떤 공격이 들어오는지, 종류와 이름을 알아야 방어를 준비할 수 있습니다.

이 단원은 그 공격의 목록을 다룹니다. 개별 공격 하나하나를 깊이 파기 전에, 먼저 전체 지형을 위에서 내려다봅니다. 어떤 위협들이 있고, 그것들이 어떤 큰 갈래로 묶이는지를 한 장의 지도로 잡는 것이 목표입니다.

먼저 왜 이런 목록이 필요한지부터 짚겠습니다. 인공지능 서비스를 만드는 팀이 늘면서, 같은 문제를 저마다 다른 이름으로 부르는 일이 생겼습니다. 한 팀은 웹페이지 속 숨은 지시에 넘어가는 문제를 지시 탈취라 부르고, 다른 팀은 명령 삽입이라 부릅니다. 이름이 제각각이면 서로 경험을 나누기 어렵고, 무엇을 빠뜨렸는지 점검하기도 어렵습니다.

그래서 여러 조직과 실무자가 함께 자주 일어나는 위협을 추려 하나의 표준 목록으로 정리했습니다. 이것이 **OWASP LLM Top 10**입니다. 정확히는 LLM 애플리케이션에서 가장 흔하고 위험한 열 가지 취약점을 순위와 함께 정리한 목록입니다. 2025년 판이 현재 널리 쓰이는 기준입니다. 이 목록의 목적은 새로운 공격을 발명하는 것이 아니라, 이미 현장에서 반복되는 위협에 공통의 이름과 번호를 붙이는 데 있습니다.

**표준 목록을 공통어로 쓰는 이유**가 여기 있습니다. 목록의 각 항목에는 LLM01부터 LLM10까지 코드가 붙습니다. 이렇게 번호를 붙여 두면, 개발자와 보안 담당자와 감사자가 같은 말로 대화할 수 있습니다. 누군가 LLM01이라고 하면 모두가 프롬프트 인젝션을 떠올리고, 굳이 무슨 뜻인지 다시 설명하지 않아도 됩니다. 또한 열 개 항목을 하나씩 짚으며 우리 시스템이 각각에 대비했는지 점검하는 체크리스트로도 쓸 수 있습니다. 공통어이자 점검표, 이 두 역할이 표준 목록의 핵심입니다.

그런데 목록의 이름이 LLM 애플리케이션 Top 10인데, 우리 관심사는 에이전트입니다. 여기서 **LLM 애플리케이션과 에이전트의 관계**를 분명히 해 둘 필요가 있습니다. LLM 애플리케이션이란 대규모 언어 모델을 안에 품고 만든 소프트웨어를 통틀어 부르는 말입니다. 질문에 답만 하는 챗봇, 문서를 요약해 주는 도구, 그리고 1.1.2에서 분해했던 에이전트가 모두 여기에 속합니다.

즉 에이전트는 LLM 애플리케이션의 한 종류입니다. 1.1.2에서 에이전트를 모델과 도구와 메모리와 루프로 나눴는데, 그 한가운데 있는 모델이 바로 이 목록이 겨냥하는 언어 모델입니다. 따라서 LLM 애플리케이션 전반에 해당하는 위협은 에이전트에도 그대로 상속됩니다. 오히려 에이전트는 도구와 자율성이 붙은 만큼 같은 위협의 피해가 더 커집니다. 에이전트 고유의 위협을 배우기 전에 이 목록을 먼저 짚는 이유가 여기 있습니다.

이제 열 개 항목이 무엇인지 봅니다. 열 개를 낱개로 외우기보다, 성격이 비슷한 것끼리 큰 갈래로 묶어 보면 전체 지형이 한눈에 들어옵니다. 크게 입력 쪽 위협, 출력 쪽 위협, 재료와 데이터 쪽 위협, 그리고 행동으로 넘어가는 위협 넷으로 나눌 수 있습니다.

```
갈래           항목                         한 줄 설명
-----------    -------------------------    ----------------------------
입력 쪽        LLM01 프롬프트 인젝션        모델에 넣는 글로 지시를 바꿔치기
출력 쪽        LLM02 민감정보 유출          모델이 내놓는 답에 비밀이 새어 나옴
               LLM05 부적절한 출력 처리     모델 출력을 검증 없이 그대로 실행
재료·데이터    LLM03 공급망                 모델·부품의 출처가 오염됨
               LLM04 데이터·모델 오염       학습·검색 데이터에 거짓을 심음
               LLM08 벡터·임베딩 취약점     검색용 저장소를 통한 침투
행동으로 넘어감 LLM06 과도한 에이전시        모델에 준 권한이 너무 넓음
```

첫째 갈래는 입력 쪽입니다. **프롬프트 인젝션**은 모델에 넣는 글 속에 몰래 지시를 섞어, 모델이 원래 받은 명령 대신 공격자의 명령을 따르게 만드는 공격입니다. LLM01번으로, 목록에서 가장 앞자리를 차지할 만큼 자주 일어나고 위험합니다. 1.3.2에서 외부 콘텐츠에 숨긴 지시가 행동을 바꾼다고 했던 그 문제가 바로 이 항목에 해당합니다.

둘째 갈래는 출력 쪽입니다. **민감정보 유출**은 모델이 내놓는 답 속에 개인정보나 회사 기밀이 섞여 나오는 문제입니다. 학습 과정에서 본 내용이나 앞선 대화의 내용이 엉뚱한 사용자에게 흘러나갈 수 있습니다. 부적절한 출력 처리는 모델의 답을 곧이곧대로 믿고 검증 없이 실행할 때 생깁니다. 모델이 만들어 낸 문장을 그대로 명령으로 실행하면, 그 문장이 오염돼 있을 때 피해가 시스템 깊숙이 파고듭니다.

셋째 갈래는 재료와 데이터 쪽입니다. **공급망** 위협은 모델 자체나 그것을 둘러싼 부품의 출처가 미덥지 않을 때 생깁니다. 남이 올린 모델, 외부 라이브러리, 남이 만든 프롬프트 틀을 가져다 쓰는데 그 안에 악의가 숨어 있을 수 있습니다. **데이터 오염**은 모델이 배우거나 참고하는 데이터에 거짓을 미리 심어 두는 공격입니다. 오염된 데이터를 근거로 삼으면 모델의 판단이 처음부터 비뚤어집니다. 여기에 검색용 저장소를 노리는 벡터·임베딩 취약점이 더해집니다.

넷째 갈래는 행동으로 넘어가는 위협입니다. **과도한 에이전시**는 모델에 필요 이상으로 넓은 권한과 도구를 쥐여 준 상태를 말합니다. LLM06번인 이 항목이 특별한 까닭은, 앞의 위협들이 결국 실제 피해로 이어지는 통로가 여기이기 때문입니다. 아무리 프롬프트가 오염돼도 모델이 할 수 있는 일이 말하기뿐이면 피해가 화면 안에 갇힙니다. 그러나 넓은 권한이 붙는 순간, 오염된 판단이 실제 삭제나 송금으로 바뀝니다.

그래서 이 목록은 **에이전트 이전 단계의 기초**라는 위치를 갖습니다. 열 개 중 아홉 개는 언어 모델을 쓰는 소프트웨어라면 어디서나 마주치는 일반적 위협입니다. 반면 LLM06 과도한 에이전시는 모델에 도구와 권한을 붙이는 순간, 즉 에이전트로 넘어가는 순간 본격적으로 문제가 됩니다. 이 항목이 LLM 보안과 에이전트 보안을 잇는 다리인 셈입니다. 그래서 이 책은 먼저 이 열 개 목록으로 기초를 다진 뒤, 다리를 건너 에이전트 고유의 위협으로 나아갑니다.

정리하면, OWASP LLM Top 10은 LLM 애플리케이션에서 가장 흔하고 위험한 열 가지 취약점을 코드와 함께 정리한 표준 목록입니다. 목적은 반복되는 위협에 공통의 이름을 붙여, 서로 다른 팀이 같은 말로 대화하고 하나씩 점검하게 하는 데 있습니다. 에이전트는 LLM 애플리케이션의 한 종류이므로 이 위협을 그대로 물려받습니다. 열 개 항목은 입력 쪽 프롬프트 인젝션, 출력 쪽 민감정보 유출과 출력 처리, 재료와 데이터 쪽 공급망과 데이터 오염과 벡터 취약점, 그리고 행동으로 넘어가는 과도한 에이전시로 묶입니다. 마지막 항목이 에이전트 보안으로 넘어가는 다리이며, 나머지는 그 앞에 놓인 기초입니다.

2.1.2에서는 이 목록의 첫 항목인 LLM01 프롬프트 인젝션 가운데 직접 인젝션을 다뤄, 모델에 직접 넣은 글이 어떻게 원래의 시스템 지시를 덮어쓰는지 그 원리를 살펴봅니다.

이 단원을 마치면 OWASP LLM Top 10이 무엇을 목록화한 표준인지 설명하고, 열 개 항목을 입력과 출력과 데이터와 에이전시라는 큰 갈래로 묶어 짚을 수 있습니다.
