# 4.1.4 MCP 방어: OAuth 2.1, 서버 검증·핀닝, 게이트웨이

앞의 두 단원에서 MCP를 노리는 다섯 가지 공격을 살펴봤습니다. 4.1.2에서는 도구 설명문에 악성 지시를 숨기는 Tool Poisoning, 악성 서버를 등록하는 Rogue Server, 처음엔 멀쩡하던 도구가 나중에 돌변하는 rug pull을 다뤘습니다. 4.1.3에서는 낮은 권한의 요청자가 높은 권한의 에이전트를 속여 대신 일을 시키는 Confused Deputy, 받은 토큰을 검증 없이 그대로 흘려보내는 Token Passthrough, 도구 커넥터를 경유해 내부망을 두드리는 SSRF를 다뤘습니다. 공격은 다 봤으니, 이제 각각을 무엇으로 막을지 정리할 차례입니다.

먼저 이 단원 전체를 관통하는 태도를 한 문장으로 세워 두겠습니다. **모든 MCP 서버는 검증 전까지 적대적이라고 가정한다**는 수칙입니다. 이 태도가 왜 필요한지는 1.2.1에서 세운 제로 트러스트 원칙을 떠올리면 분명해집니다. 그 단원에서 우리는 내부에 있다는 사실만으로는 신뢰를 주지 않고 매 요청을 다시 검증한다는 원칙을 세웠습니다. MCP 서버도 마찬가지입니다. 에이전트가 이미 연결해 쓰고 있는 서버라는 이유로 그 서버가 하는 말을 믿어서는 안 됩니다. 서버가 보내는 도구 목록, 도구 설명문, 응답 데이터를 모두 잠재적으로 조작된 입력으로 다루는 것이 출발점입니다.

이 수칙을 태도로만 두면 아무 힘이 없습니다. 실제로는 인증, 서버 검증과 핀닝, 게이트웨이와 레지스트리, 도구 설명문 분석이라는 네 겹의 장치로 이 태도를 강제합니다. 하나씩 풀어 보겠습니다.

첫 번째 겹은 인증입니다. 4.1.3에서 본 Token Passthrough와 Confused Deputy는 모두 신원과 권한을 제대로 확인하지 않은 데서 비롯됩니다. 누가 이 서버에 접근할 자격이 있는지, 이 서버에 실제로 어떤 권한을 넘겨줄지를 규칙으로 못 박아야 합니다. 이 자리에서 쓰는 표준이 **OAuth 2.1 기반 인증**입니다.

OAuth라는 이름부터 배경을 풀어 보겠습니다. OAuth는 어떤 서비스가 다른 서비스의 자원에 접근할 때, 사용자의 비밀번호를 직접 넘기지 않고도 제한된 권한을 위임받는 방식을 정한 표준입니다. 예를 들어 어떤 앱이 사진 저장소의 사진을 가져오려 할 때, 사용자의 저장소 비밀번호를 그 앱에 알려 주는 대신, 사진 읽기만 허용하는 짧은 통행증을 발급해 건네는 식입니다. 이 통행증을 **액세스 토큰**이라 부릅니다. OAuth 2.1은 이 방식의 개정판으로, 그동안 취약했던 관행들을 정리해 안전한 흐름만 남긴 규격입니다.

OAuth 2.1이 MCP 방어에서 하는 일을 단계로 보면 이렇습니다.

```
[1] 에이전트가 MCP 서버에 접근하려 함
      ↓
[2] 인증 서버가 에이전트의 신원을 확인
      ↓
[3] 이 작업에 필요한 최소 범위(scope)만 담은 액세스 토큰 발급
      ↓
[4] 짧은 유효기간이 지나면 토큰 만료 → 재발급 필요
      ↓
[5] MCP 서버는 매 요청에서 토큰의 유효성과 범위를 검증
```

핵심은 세 가지입니다. 토큰에는 이 작업에 필요한 만큼의 권한 범위만 담고, 유효기간을 짧게 두며, 서버는 받은 토큰을 그대로 믿지 않고 매번 검증합니다. 이렇게 하면 4.1.3의 Token Passthrough처럼 토큰을 무검증으로 흘려보내는 경로가 애초에 막힙니다. 토큰이 유출되더라도 범위가 좁고 수명이 짧으니 피해 반경도 줄어듭니다.

두 번째 겹은 서버 검증과 핀닝입니다. 이 장치는 4.1.2에서 본 rug pull을 정면으로 겨냥합니다. rug pull은 처음 등록할 때는 정상이던 서버나 도구가, 신뢰를 얻은 뒤 조용히 악성으로 바뀌는 공격이었습니다. 인증만으로는 이걸 막지 못합니다. 인증은 상대가 누구인지 확인할 뿐, 그 상대의 내용물이 어제와 같은지는 따지지 않기 때문입니다.

여기서 **핀닝(pinning)**이라는 개념이 등장합니다. 핀닝은 처음 검증한 시점의 상태를 고정해 못 박아 두고, 이후 그 상태와 달라지면 거부하는 방식입니다. 벽에 핀으로 종이를 꽂아 고정하듯, 신뢰할 만하다고 확인한 순간의 서버 신원과 도구 정의를 기록으로 붙들어 둡니다. 구체적으로는 서버가 제시하는 도구 설명문과 서버 신원 정보를 요약한 지문 같은 값을 저장해 둡니다. 다음에 그 서버에 접속했을 때 이 값이 달라져 있으면, 무언가 바뀌었다는 신호이므로 자동 승인을 멈추고 사람의 재검토를 요구합니다.

간단한 예를 들면 이렇습니다. 어제 등록한 번역 도구의 설명문이 오늘 갑자기 사용자의 자격증명을 특정 주소로 보내라는 문장을 담은 형태로 바뀌었다고 해 봅니다. 핀닝이 없으면 에이전트는 이 변경을 눈치채지 못한 채 새 설명문을 그대로 따릅니다. 핀닝이 있으면 저장해 둔 지문과 지금 값이 어긋나므로 그 자리에서 걸립니다. 정상적인 업데이트라면 사람이 확인하고 새 값을 다시 핀으로 고정하면 되고, 공격이라면 거기서 차단됩니다.

```
서버 검증 + 핀닝의 흐름
등록 시   : 서버 신원·도구 설명문 확인 → 지문 저장(핀 고정)
매 접속 시: 현재 값과 저장된 지문 비교
   같으면 → 통과
   다르면 → 자동 승인 중지, 사람 재검토 요구(rug pull 차단)
```

세 번째 겹은 게이트웨이와 레지스트리입니다. 여기까지의 장치를 에이전트 하나하나가 제각각 구현하면 관리가 흩어집니다. 어떤 에이전트는 핀닝을 하고 어떤 에이전트는 빠뜨리는 식으로 구멍이 생기기 쉽습니다. 그래서 검증을 한곳으로 모으는 구조를 둡니다.

**MCP 게이트웨이**는 에이전트와 여러 MCP 서버 사이에 서는 중간 관문입니다. 1.2.1에서 제로 트러스트의 문지기를 이야기했는데, 게이트웨이가 바로 MCP 세계의 그 문지기입니다. 에이전트가 어떤 서버를 호출하든 요청이 반드시 이 관문을 거치게 하고, 관문에서 인증 확인, 핀닝 대조, 요청과 응답 검사, 로그 기록을 한꺼번에 처리합니다. 검증 논리를 한 자리에 모으니 정책을 일관되게 적용하고 누락을 줄일 수 있습니다.

**레지스트리**는 조직이 사용을 허가한 MCP 서버의 목록을 관리하는 등록소입니다. 어떤 서버가 승인됐고, 각 서버의 검증된 지문이 무엇이며, 누가 언제 등록했는지를 기록합니다. 레지스트리에 없는 서버는 연결 자체를 거부하도록 하면, 4.1.2의 Rogue Server처럼 정체불명의 악성 서버가 슬그머니 끼어드는 경로를 원천 차단할 수 있습니다. 게이트웨이가 문지기라면, 레지스트리는 그 문지기가 들고 있는 출입 허가자 명단인 셈입니다.

네 번째 겹은 도구 설명문 정적 분석입니다. **도구 설명문 정적 분석**은 도구를 실제로 실행해 보기 전에, 설명문 텍스트 자체를 기계로 훑어 위험한 패턴을 찾아내는 검사입니다. 정적이라는 말은 실행하지 않고 내용만 들여다본다는 뜻입니다. 이 검사는 4.1.2의 Tool Poisoning을 겨냥합니다. Tool Poisoning은 도구 설명문 안에 에이전트를 향한 숨은 지시문을 심는 공격이었습니다.

정적 분석이 무엇을 걸러내는지 예로 보면 이렇습니다. 도구 설명문에 이전 지시를 무시하라거나, 자격증명을 어딘가로 보내라거나, 사용자에게는 숨기라는 식의 문구가 들어 있는지 점검합니다. 겉보기엔 평범한 설명 같지만 에이전트에게 몰래 명령을 내리려는 표현이 섞여 있으면 이를 잡아냅니다. 등록 시점에 한 번, 그리고 핀닝이 변경을 감지한 시점에 다시 돌려서, 사람이 일일이 읽지 않아도 명백한 악성 패턴을 자동으로 거르게 합니다. 다만 이 검사가 모든 교묘한 문구를 완벽히 잡지는 못하므로, 앞의 세 겹과 함께 쓰는 보조 장치로 둡니다.

이제 네 겹이 각각 어느 공격을 맡는지 한눈에 정리해 보겠습니다.

```
공격(4.1.2~4.1.3)          → 주 방어 장치
Token Passthrough          → OAuth 2.1: 좁은 범위·짧은 수명·매 요청 검증
Confused Deputy            → OAuth 2.1: 신원 확인과 권한 위임 통제
rug pull                   → 서버 검증 + 핀닝: 변경 감지 후 재검토
Rogue Server               → 레지스트리: 미등록 서버 연결 거부
SSRF·전반 통제             → 게이트웨이: 모든 요청을 한 관문으로 집약
Tool Poisoning             → 도구 설명문 정적 분석: 숨은 지시문 탐지
```

정리하면, MCP 방어의 바탕은 모든 서버를 검증 전까지 적대적이라고 가정하는 태도이며, 이 태도는 인증, 서버 검증과 핀닝, 게이트웨이와 레지스트리, 도구 설명문 정적 분석이라는 네 겹으로 강제됩니다. OAuth 2.1은 좁은 범위와 짧은 수명의 토큰을 매 요청 검증해 위임과 토큰 남용을 막고, 핀닝은 처음 검증한 상태를 고정해 나중에 돌변하는 rug pull을 잡으며, 게이트웨이와 레지스트리는 검증을 한 관문과 허가 명단으로 모아 정체불명의 서버를 걸러 냅니다. 정적 분석은 설명문에 숨은 지시문을 실행 전에 탐지하는 보조 장치로 이 겹들을 거듭니다.

다음 단원인 4.2.1에서는 도구 연결을 넘어 에이전트끼리 협업하는 A2A 프로토콜과, 각 에이전트가 자신의 신원과 능력을 밝히는 Agent Card를 다룹니다.

이 단원을 마치면 MCP 5대 공격에 맞서는 OAuth 2.1 인증, 서버 검증과 핀닝, 게이트웨이와 레지스트리, 도구 설명문 정적 분석을 각 공격과 연결해 설명하고, 모든 MCP 서버를 검증 전까지 적대적이라고 가정하는 수칙을 근거로 삼아 말할 수 있습니다.
