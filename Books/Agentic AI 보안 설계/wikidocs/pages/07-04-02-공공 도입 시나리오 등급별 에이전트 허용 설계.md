# 7.4.2 공공 도입 시나리오: 등급별 에이전트 허용 설계

한 공공기관이 생성형 AI 에이전트를 업무에 들이기로 했다고 가정해 보겠습니다. 담당자가 가장 먼저 부딪히는 질문은 기술 질문이 아닙니다. "이 에이전트가 우리 문서를 봐도 되는가, 어디까지 봐도 되는가"입니다. 외부에서 운영되는 대형 언어 모델을 쓰는 순간, 기관의 정보가 그 모델 쪽으로 흘러갈 여지가 생깁니다. 그렇다고 아무것도 못 하게 막으면 도입할 이유 자체가 사라집니다.

이 긴장은 규제로 풀어야 합니다. 7.4.1에서 우리는 N2SF의 6대 보안통제 항목이 앞서 배운 agentic 방어 개념과 거의 1:1로 짝지어진다는 것을 표로 확인했습니다. 그런데 대응표는 "각 통제 항목을 무엇으로 구현하느냐"만 알려 줄 뿐, "무엇부터 열고 무엇을 닫느냐"는 답해 주지 않습니다. 실제 도입에는 시작점과 순서가 필요합니다. 이 단원은 그 순서를 설계로 세웁니다.

순서를 정하려면 먼저 대상을 나눠야 합니다. 여기서 **업무 정보의 C/S/O 분류**가 출발점입니다. N2SF는 물리적 망분리를 대신해, 기관이 다루는 업무 정보를 세 등급으로 나눕니다. C는 기밀(Classified)로 유출 시 국가 안보나 핵심 이익에 중대한 피해가 예상되는 정보입니다. S는 민감(Sensitive)으로 공개되면 곤란하지만 그 파급이 C만큼 크지는 않은 정보입니다. O는 공개(Open)로 이미 공개되었거나 공개해도 무방한 정보입니다. 예를 들어 보도자료 초안이나 대국민 안내문은 O, 내부 검토 중인 인사·예산 자료는 S, 국가 안보와 직결된 문서는 C에 가깝습니다.

핵심은 이 등급이 정보 자체에 붙는 꼬리표라는 점입니다. 사람이든 시스템이든 에이전트든, 그 정보를 만지려는 주체가 등급에 맞는 조건을 갖췄을 때만 접근을 허용합니다. 에이전트도 예외가 아니며, 오히려 에이전트는 사람보다 더 조심스럽게 다뤄야 합니다. 에이전트는 권한을 가진 채로 오염될 수 있는 실행 주체이기 때문입니다.

그래서 도입 흐름은 위험이 낮은 쪽부터 엽니다. 가장 안전한 O등급 업무에서 시작해, 조건을 붙여 S등급으로 넓히고, C등급은 닫아 두는 순서입니다.

```
  위험 낮음 ──────────────────────────▶ 위험 높음
  ┌──────────┐   ┌──────────────┐   ┌──────────┐
  │ O (공개) │   │  S (민감)    │   │ C (기밀) │
  │ 허용     │ → │ 조건부 허용  │ → │ 차단     │
  │ 먼저 시작│   │ 격리·통제 시 │   │ 외부금지 │
  └──────────┘   └──────────────┘   └──────────┘
       1단계          2단계             (열지 않음)
```

첫 단계는 **O등급부터 외부 LLM 에이전트 허용**입니다. 공개 정보만 다루는 업무라면 외부 모델에 그 내용이 흘러가도 새로 잃을 것이 없습니다. 예컨대 공개 공고문을 다듬거나, 공개된 통계로 안내 문구를 만드는 작업입니다. 이 구간에서는 외부에서 운영되는 상용 LLM을 붙인 에이전트를 비교적 자유롭게 씁니다. 여기서 운영 경험과 사고 대응 체계를 먼저 쌓는 것이 순서의 의도입니다. 위험이 낮은 곳에서 조직이 익숙해진 뒤 다음 칸으로 넘어갑니다.

두 번째 단계는 **S등급의 격리·통제 조건부 허용**입니다. 민감 정보는 무조건 막지도, 그냥 열지도 않습니다. 대신 특정 방어 장치를 갖췄을 때에 한해 조건부로 허용합니다. 여기서 앞서 배운 방어 패턴들이 조건으로 들어옵니다. 5.1.2에서 다룬 egress 통제로 에이전트가 내보낼 수 있는 목적지를 미리 정한 목록으로 좁혀, 민감 정보가 바깥으로 새는 경로 자체를 없앱니다. 5.1.1의 샌드박싱과 5.1.3의 세션·테넌트 격리로 실행 환경과 세션을 갈라 둡니다. 삭제나 외부 발송 같은 위험한 행동은 5.3.1에서 다룬 HITL 승인 게이트를 거치게 해, 사람이 확인한 뒤에만 실제로 실행되도록 합니다.

S등급에서 특히 중요한 것은 정보를 기관이 통제하는 울타리 안에 두는 선택입니다. 외부 상용 모델을 그대로 쓰기보다, 기관 내부나 통제된 환경에 둔 모델을 붙이는 방식이 여기서 자주 쓰입니다. 3.4.2에서 다룬 치명적 3요소를 떠올리면 이유가 분명합니다. 민감 데이터 접근과 신뢰할 수 없는 콘텐츠 노출, 외부 전송 능력이 한 에이전트에 모이면 유출은 시간문제입니다. S등급 통제는 이 셋 중 최소 하나, 특히 외부 전송 경로를 구조적으로 끊는 데 초점을 둡니다.

세 번째 단계는 **C등급 차단**입니다. 기밀 정보는 외부 LLM 에이전트에게 열지 않습니다. 조건을 붙여서라도 여는 대상이 아니라, 아예 접근 경계 밖에 두는 대상입니다. 유출 시 피해가 감당할 수 없는 수준이므로, 편의보다 안전을 우선해 선을 긋습니다. C등급 업무에까지 자동화가 필요하다면 외부 모델이 아닌 다른 방식을 검토해야 하며, 이는 이 도입 흐름의 범위 밖입니다.

이 O→S→C 순서를 한눈에 정리하면 다음과 같습니다.

```
등급   에이전트 허용     핵심 조건
-----  ---------------   ------------------------------------
O      허용(먼저)        추가 조건 없이 외부 LLM 활용
S      조건부 허용       egress 통제 + 격리 + HITL 승인
C      차단              외부 LLM 접근 자체를 열지 않음
```

이 흐름이 책상 위 이론에 머물지 않는다는 증거가 **KISA 실증사업의 검증 흐름**입니다. N2SF는 2025년 1월 초안에서 시작해 보안가이드라인 정식판으로 이어졌고, 실제 적용 가능성을 확인하기 위해 실증사업이 진행되었습니다. 이 실증사업은 여섯 개 활용 모델을 두고 각각을 검증하는데, 그중 하나가 업무 환경에서 생성형 AI를 활용하는 모델입니다. 바로 이 모델이 지금 설명한 흐름, 즉 업무 정보를 C/S/O로 나눠 등급별로 에이전트 허용 범위를 정하는 방식이 현장에서 통하는지를 확인하는 자리입니다. 실증에서 다듬어진 결과는 2026년 국가 사이버보안 기본지침 개정과 함께 본격 시행 단계로 이어집니다.

마지막으로 두 세계가 만나는 지점이 하나 더 있습니다. **역방향 연구**, 곧 보안통제 도출을 자동화하려는 시도입니다. 지금까지는 AI를 지키기 위해 보안을 설계했습니다. 반대 방향도 가능합니다. N2SF의 6대 보안통제 항목을 특정 업무에 어떻게 적용할지 답을 짜는 일 자체를, 여러 에이전트가 협업하는 LLM 시스템으로 초안을 만들어 보려는 연구가 나오고 있습니다. 이렇게 되면 AI를 지키는 보안과 보안을 돕는 AI가 한자리에서 만납니다. 다만 이런 자동화 산출물도 결국 사람이 검토하고 책임지는 통제 아래 둬야 한다는 점은 달라지지 않습니다.

정리하면, 공공에서 에이전트를 들일 때의 설계는 업무 정보를 C/S/O로 분류하는 데서 출발합니다. 위험이 가장 낮은 O등급 업무부터 외부 LLM 에이전트를 허용해 경험을 쌓고, S등급은 egress 통제와 격리, HITL 승인 같은 조건을 갖췄을 때만 조건부로 열며, C등급은 외부에 열지 않고 차단합니다. 이 O→S→C 흐름을 KISA 실증사업이 여섯 모델 중 업무 활용 모델로 검증하고 있고, 보안통제 도출을 멀티에이전트로 자동화하는 역방향 연구까지 더해지며 두 세계가 맞닿습니다.

7.4.3에서는 지금까지의 기반, 위협, 방어, 운영, 규제를 한 줄기로 되짚고 자기 시스템을 진단하는 방법과 다음 학습 경로를 다룹니다.

이 단원을 마치면 업무 정보를 C/S/O로 분류해 O→S→C 순으로 에이전트 허용 범위를 정하는 공공 도입 흐름을 설계로 설명할 수 있습니다.
