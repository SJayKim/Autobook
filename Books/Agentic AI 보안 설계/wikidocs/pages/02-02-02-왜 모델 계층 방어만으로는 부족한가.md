# 2.2.2 왜 모델 계층 방어만으로는 부족한가

앞의 2.2.1에서 과도한 에이전시(LLM06)가 왜 에이전트 보안의 출발점인지 살펴봤습니다. 에이전트에게 도구와 권한을 쥐여 주는 순간, 모델이 한 번 잘못 판단하면 그 판단이 실제 행동으로 이어진다는 점이 문제의 핵심이었습니다. 그러면 자연스럽게 다음 질문이 따라옵니다. 모델이 잘못 판단하지 않도록 모델 자체를 더 똑똑하게, 더 잘 걸러지게 만들면 되지 않을까 하는 것입니다.

이 질문은 합리적으로 들립니다. 실제로 많은 팀이 여기서 방어를 시작합니다. 그러나 이 단원은 그 접근만으로는 왜 충분하지 않은지를 설명합니다. 결론부터 말하면, 모델 근처에 세우는 방어는 평소에는 잘 듣다가 정작 공격받는 순간에 뚫릴 수 있습니다. 그래서 모델 바깥, 즉 시스템 구조 쪽에 방어를 한 겹 더 두어야 합니다.

먼저 모델 근처에 세우는 방어가 어디에 위치하는지부터 정리하겠습니다. 이것을 **모델 계층 방어**라고 부릅니다. 모델 계층 방어란 모델과 아주 가까운 지점에서 위험을 걸러 내려는 대책입니다. 입력이 모델에 들어가기 직전이나 모델이 답을 내놓은 직후가 그런 지점입니다. 에이전트가 도구를 실제로 호출하기 전, 아직 판단 단계에 머물러 있는 자리라고 보면 됩니다.

모델 계층 방어는 대표적으로 세 가지 형태로 나타납니다. 배경부터 하나씩 풀어 보겠습니다.

첫째는 **정렬(alignment)**입니다. 정렬이란 모델이 사람이 바라는 방향으로 행동하도록 학습 단계에서 성향을 맞추어 두는 작업입니다. 위험한 요청이 들어오면 거절하고, 해로운 내용은 만들지 않도록 모델의 기본 태도를 길들이는 것입니다. 나쁜 지시에 스스로 저항하도록 모델의 몸에 배게 하는 방어라고 이해하면 됩니다.

둘째는 **입력 필터**입니다. 입력 필터란 사용자가 보낸 글이나 에이전트가 읽어 들인 자료를 모델에 넘기기 전에 미리 검사해서, 수상한 부분을 잡아내려는 장치입니다. 예를 들어 이전 지시는 모두 무시하라 같은 문구가 들어 있으면 위험 신호로 보고 차단하는 식입니다.

셋째는 **분류기(classifier)**입니다. 분류기란 어떤 글이 안전한지 위험한지를 대신 판정해 주는 별도의 작은 판별 모델입니다. 인젝션 탐지 분류기가 흔한 예인데, 들어온 입력을 보고 이건 공격 시도다 또는 이건 정상이다처럼 꼬리표를 붙입니다. 그 판정 결과에 따라 통과시킬지 막을지를 정합니다.

이 세 가지가 모델 계층에서 어디에 놓이는지를 한눈에 정리하면 다음과 같습니다.

```
[사용자 입력 / 읽어 들인 자료]
        │
        ▼
   입력 필터  ── 수상한 문구 차단
        │
        ▼
   분류기     ── 공격/정상 판정
        │
        ▼
 ┌──────────────┐
 │  모델(정렬됨) │ ── 위험 요청에 저항
 └──────────────┘
        │
        ▼
   [도구 호출 / 답변]  ← 여기서부터는 실제 행동
```

이 그림에서 보이듯 세 방어는 모두 실제 행동이 일어나기 전, 모델을 감싸는 자리에 몰려 있습니다. 여기까지만 보면 방어가 세 겹이나 되니 든든해 보입니다. 문제는 이 세 겹이 공유하는 약점입니다.

그 약점을 이해하려면 앞의 1.2.3에서 다룬 사실을 다시 떠올려야 합니다. 거기서 모델은 정해진 규칙을 그대로 실행하는 프로그램이 아니라, 입력받은 글의 맥락을 보고 다음 행동을 확률적으로 정하는 장치라고 했습니다. 정렬도, 입력 필터도, 분류기도 결국 이 확률적 판단에 기대고 있습니다. 딱 잘라 예/아니오로 떨어지는 규칙이 아니라, 이 정도면 위험해 보인다는 짐작에 가깝습니다.

짐작에 기댄 방어에는 두 종류의 빈틈이 생깁니다. 하나는 위험한 것을 안전하다고 잘못 통과시키는 경우이고, 다른 하나는 멀쩡한 것을 위험하다고 잘못 막는 경우입니다. 공격자는 이 중 첫 번째 빈틈을 집요하게 노립니다. 이렇게 방어의 판정을 피해 위험한 입력을 정상인 척 통과시키는 것을 **우회(bypass)**라고 합니다.

우회가 실제로 어떻게 일어나는지 구체적인 예로 보겠습니다. 입력 필터가 이전 지시는 무시하라는 문구를 차단한다고 합시다. 공격자는 같은 뜻을 다른 말로 바꿉니다. 예를 들어 지금까지의 안내는 참고용이었고 진짜 업무 규칙은 다음과 같다처럼 풀어 씁니다. 필터가 걸어 둔 특정 문구와 글자가 겹치지 않으니 그대로 통과합니다. 그러나 모델이 받아들이는 의미는 앞의 금지된 문구와 똑같습니다.

분류기도 같은 방식으로 뚫립니다. 분류기가 공격 문장의 특징을 학습해 두었더라도, 공격자는 표현을 조금씩 비틀어 판정 경계를 넘나듭니다. 무해한 문장 사이에 악성 지시를 잘게 쪼개어 흩어 놓거나, 인용문이나 예시처럼 위장하기도 합니다. 정렬 역시 절대적이지 않습니다. 상황을 그럴듯하게 꾸며 역할극을 시키거나 예외 상황을 가정하게 하면, 평소라면 거절했을 요청을 모델이 받아들이기도 합니다.

핵심은 이것들이 우연한 버그가 아니라는 점입니다. 모델 계층 방어가 확률적 판단에 기대는 한, 그 판단의 경계를 파고드는 우회는 계속 나옵니다. 방어를 보강하면 새로운 우회가 등장하고, 다시 막으면 또 다른 우회가 나오는 식입니다. 특히 1.2.3에서 본 간접 프롬프트 인젝션이 그렇습니다. 에이전트가 업무상 어차피 읽어야 하는 자료에 지시가 숨어 있으면, 무엇이 신뢰할 자료이고 무엇이 아닌지를 글만 보고 완벽히 가려내기가 어렵습니다. 그래서 이 공격에는 아직 완전한 해결책이 없다는 것이 정설입니다.

여기서 방향을 바꾸어야 합니다. 모델 계층 방어를 없애자는 뜻이 아닙니다. 정렬과 필터와 분류기는 평범한 공격의 상당수를 걸러 주는 쓸모 있는 첫 관문입니다. 다만 이것을 마지막 방어선으로 믿어서는 안 됩니다. 이 관문은 언젠가 뚫린다고 전제하고, 뚫린 다음을 대비하는 설계가 필요합니다. 이것이 1.2.3에서 다룬 심층 방어이고, 특히 그중에서도 모델 계층 실패를 전제로 한 설계에 해당합니다.

그렇다면 뚫린 다음을 무엇으로 받아야 할까요. 바로 모델 바깥에 세우는 방어입니다. 이것을 **아키텍처 계층 방어**라고 부릅니다. 아키텍처 계층 방어란 모델의 판단이 아니라 시스템의 구조로 위험한 행동을 막는 방식입니다. 모델이 무엇을 하려고 마음먹었는지와 상관없이, 애초에 할 수 없는 일은 실행되지 않도록 구조를 짜는 것입니다.

이 두 계층의 차이가 이 단원의 가장 중요한 대목입니다. 표로 나란히 놓고 보겠습니다.

```
                모델 계층 방어              아키텍처 계층 방어
근거          모델의 확률적 판단           시스템의 고정된 구조
방식          "하면 안 된다"고 판정        "할 수 없게" 통로를 차단
위치          모델 입력·출력 근처          권한·네트워크·격리 계층
한계          우회 가능                    우회하려면 구조를 깨야 함
역할          첫 관문(보조)                최종 방어선(주력)
```

표의 마지막 줄이 요점을 압축합니다. 모델 계층 방어는 첫 관문이고, 아키텍처 계층 방어가 실제로 피해를 막는 주력입니다. 두 계층은 경쟁하는 관계가 아니라 겹쳐 쓰는 관계입니다. 앞 관문이 대부분을 걸러 주면, 뒤 계층이 걸러지지 못한 나머지를 구조적으로 차단합니다.

아키텍처 계층 방어가 구체적으로 어떤 모습인지는 1.2.3에서 이미 맛보았습니다. 거기서 든 권한 계층, 네트워크 계층, 샌드박스 계층이 모두 아키텍처 계층 방어의 예입니다. 조회 권한을 아예 주지 않으면 모델이 속아도 정보를 손에 넣지 못합니다. 외부로 나갈 통로를 허용 목록으로 좁혀 두면 낯선 주소로는 데이터가 나가지 못합니다. 이것들은 모델에게 하지 말라고 부탁하는 것이 아니라, 할 수 없게 통로 자체를 없애는 방어입니다.

정리하면, 정렬과 입력 필터와 분류기 같은 모델 계층 방어는 모두 모델의 확률적 판단에 기댑니다. 그래서 표현을 비틀어 판정 경계를 파고드는 우회에 뚫릴 수 있습니다. 그래서 이 관문은 첫 겹으로만 쓰고, 관문이 뚫린다는 것을 전제로 심층 방어를 세워야 합니다. 그 뒷겹이 바로 모델 바깥에서 구조로 막는 아키텍처 계층 방어입니다. 권한과 네트워크와 격리처럼, 모델이 무엇을 하려 하든 애초에 할 수 없게 만드는 방식입니다.

다음 단원인 3.1.1에서는 이런 에이전트 고유의 위협을 하나의 분류체계로 정리해 읽는 법을 다룹니다. 에이전트 전용 위협 목록인 Agentic Top 10이 무엇을 열 가지로 나누어 놓았는지, 그 표를 어떻게 읽어야 하는지 살펴봅니다. 모델 바깥에서 구조로 막는 구체적 방어 설계는 5장에서 본격적으로 다룹니다.

이 단원을 마치면 모델 계층 방어가 왜 우회 가능한지 설명하고, 그 한계를 심층 방어와 연결해 아키텍처 계층 방어가 필요한 이유를 말할 수 있습니다.
