# 5.1.2 Egress 통제: 외부 전송을 구조적으로 막기

앞의 5.1.1에서 에이전트가 만들어 낸 코드를 컨테이너와 microVM, WASM 안에 가둬 실행하는 샌드박싱을 다뤘습니다. 샌드박싱은 에이전트가 시스템 안에서 무엇을 망가뜨릴 수 있는지를 좁힙니다. 그런데 격리된 상자 안에서도 여전히 열려 있는 문이 하나 있습니다. 바로 바깥으로 나가는 네트워크 통로입니다. 상자 안에 갇힌 에이전트라도 인터넷으로 데이터를 내보낼 수 있다면, 훔친 정보는 그 통로를 타고 밖으로 빠져나갑니다.

이 문을 다루는 것이 이 단원의 주제입니다. 여기서 다룰 개념은 **Egress 통제**입니다. 이그레스는 밖으로 나간다는 뜻으로, 시스템 안에서 바깥으로 향하는 통신을 가리킵니다. 반대로 밖에서 안으로 들어오는 통신은 인그레스라고 부릅니다. 전통적인 보안은 인그레스, 즉 외부 공격자가 안으로 들어오는 길을 막는 데 집중해 왔습니다. 하지만 에이전트 보안에서는 방향이 반대입니다. 이미 안에 있는 에이전트가 무언가를 밖으로 내보내는 길을 막아야 합니다.

왜 나가는 길을 막는 것이 그토록 중요한지는 3.4.2에서 다룬 원칙과 이어집니다. 3.4.2의 치명적 3요소는 민감 데이터 접근, 신뢰 불가 콘텐츠 노출, 외부 전송 능력이라는 세 가지가 한 에이전트에 모이면 데이터 유출이 시간문제가 된다는 원칙이었습니다. 그리고 셋 중 하나만 확실히 끊어도 유출 경로가 사라진다는 점도 확인했습니다. Egress 통제는 바로 세 번째 요소인 외부 전송 능력을 구조적으로 끊는 방법입니다.

**Egress 통제**란 에이전트가 접근할 수 있는 네트워크 목적지를 미리 정한 몇 곳으로만 제한하는 방어입니다. 목적지란 에이전트가 데이터를 보내거나 요청을 거는 상대 주소를 말합니다. 특정 서버의 주소나 도메인 이름이 목적지입니다. 통제의 핵심은 허용한 목적지 외에는 어디로도 나갈 수 없게 막는 데 있습니다.

이 통제의 뼈대가 되는 것이 **네트워크 목적지 allowlist**입니다. 얼라우리스트는 허용 목록이라는 뜻으로, 미리 안전하다고 정해 둔 항목만 통과시키고 나머지는 모두 막는 방식입니다. 반대 개념은 차단 목록입니다. 차단 목록은 위험한 곳을 하나하나 적어 막고 나머지는 통과시킵니다. 이 둘의 차이가 Egress 통제에서 결정적입니다.

차단 목록이 왜 부족한지 구체적으로 보겠습니다. 공격자가 데이터를 빼낼 수 있는 서버 주소는 사실상 무한합니다. 오늘 한 주소를 막으면 내일 다른 주소를 쓰면 그만입니다. 위험한 곳을 전부 적어 막는 일은 끝이 없습니다. 반면 에이전트가 정상 업무에서 실제로 연결해야 하는 목적지는 대개 손에 꼽습니다. 그래서 소수의 허용할 곳만 적고 나머지 전부를 막는 허용 목록 방식이 훨씬 안전하고 관리하기도 쉽습니다.

허용 목록이 외부 전송 축을 어떻게 제거하는지 3.4.2의 메일 정리 에이전트로 따라가 보겠습니다. 그 예시에서 공격자는 조작된 메일에 숨긴 지시로 에이전트가 받은 편지함을 읽어 자기 서버로 요약을 보내게 만들었습니다. 유출의 마지막 단계는 공격자 서버라는 목적지로 데이터를 내보내는 순간이었습니다. 그런데 에이전트가 나갈 수 있는 목적지가 사용자 본인의 메일함 한 곳으로 묶여 있다면 어떻게 될까요. 에이전트가 아무리 그럴듯한 지시를 읽어도, 공격자 서버라는 목적지는 허용 목록에 없으므로 그 전송은 애초에 성립하지 않습니다.

```
        에이전트가 밖으로 나가려는 요청
                    │
                    ▼
        ┌───────────────────────┐
        │  Egress 통제 지점       │
        │  (허용 목록과 대조)     │
        └───────────────────────┘
             │             │
     목적지가        목적지가
     목록에 있음      목록에 없음
             │             │
             ▼             ▼
        통과 (허용)     차단 (거부)
     사용자 메일함      공격자 서버
     사내 API 서버      임의의 외부 주소
```

이 그림에서 통제 지점은 에이전트가 보내는 모든 요청이 반드시 지나가는 검문소 역할을 합니다. 요청이 나갈 때마다 목적지를 허용 목록과 대조합니다. 목록에 있으면 통과시키고, 없으면 그 자리에서 막습니다. 여기서 중요한 점은 이 검문소가 에이전트의 판단 바깥에 있다는 것입니다. 에이전트가 무엇을 읽었든, 어떤 지시에 설득당했든, 목적지가 목록에 없으면 나가지 못합니다.

이 구조가 왜 미더운지는 3.4.2에서 살펴본 이유와 같습니다. 조작된 지시를 문장 단위로 걸러 내려는 시도는 공격자가 표현만 바꿔도 뚫립니다. 반면 Egress 통제는 문장을 검사하지 않습니다. 에이전트가 어떤 지시를 받아들였는지와 무관하게, 나갈 수 있는 목적지 자체를 물리적으로 제한합니다. 그래서 공격을 알아채는 데 기대지 않고, 데이터 유출 경로 자체를 없앱니다. 이것이 외부 전송을 구조적으로 막는다는 말의 의미입니다.

여기서 3.4.2가 짚은 주의점이 다시 중요해집니다. 전송 통로는 눈에 잘 띄지 않는 곳까지 넓게 잡아야 합니다. 메일 발송과 API 호출만 통로인 것이 아닙니다. 에이전트가 특정 주소의 이미지를 불러오는 행동, 웹페이지를 열어 보는 행동, 이름을 주소로 바꾸기 위해 주소 조회 서버에 묻는 행동까지 모두 데이터가 새어 나갈 수 있는 통로입니다. Egress 통제가 제 몫을 하려면 이런 통로가 하나도 빠짐없이 같은 검문소를 지나가야 합니다. 통로 하나가 검문소를 우회하면 그 하나로 데이터가 빠져나갑니다.

그래서 데이터 유출 경로 차단은 통제 지점을 에이전트가 아니라 그 바깥의 네트워크 계층에 두는 방식으로 이뤄집니다. 실무에서는 에이전트가 나가는 모든 트래픽을 대리 서버 한 곳으로 강제로 모으고, 그 대리 서버가 허용 목록과 대조해 통과 여부를 정합니다. 이렇게 하면 에이전트 코드가 어떻게 바뀌든, 나가는 길은 반드시 그 한 지점을 지납니다.

```
방화벽 기본 정책: 모든 아웃바운드 차단(default deny)

허용 목록(예):
  api.internal.company.com   # 사내 업무 API
  mail.company.com           # 사용자 본인 메일함
  # 그 외 모든 목적지 → 거부
```

이 설정에서 첫 줄이 통제의 출발점입니다. 기본 정책을 모든 아웃바운드 차단으로 두면, 아무것도 허용하지 않은 상태에서 시작합니다. 이를 default deny, 즉 기본 거부라고 부릅니다. 그다음 정말 필요한 목적지만 한 줄씩 허용 목록에 더합니다. 나머지 모든 목적지는 목록에 없으므로 자동으로 막힙니다. 이 순서가 중요합니다. 필요한 것만 열어 주는 방식이라야, 깜빡 잊고 안 막은 구멍이 생기지 않습니다.

이제 **허용 목록 설계**를 어떻게 하는지 정리하겠습니다. 설계의 첫걸음은 에이전트가 하려는 일에 정말로 필요한 목적지를 하나하나 적어 보는 것입니다. 메일을 정리하는 에이전트라면 사용자 본인의 메일함이 필요합니다. 사내 문서를 조회한다면 그 사내 서버가 필요합니다. 이렇게 업무에 꼭 필요한 목적지만 목록에 올립니다.

허용 목록 설계에서 특히 조심할 부분은 목록을 지나치게 넓게 잡지 않는 것입니다. 예를 들어 어떤 외부 검색 서비스가 필요하다고 해서 그 서비스의 도메인 전체를 통째로 열어 두면, 그 안에 공격자가 데이터를 흘려보낼 만한 경로가 함께 열릴 수 있습니다. 넓게 열수록 유출 경로도 함께 넓어집니다. 그래서 허용 목록은 업무가 돌아가는 선에서 가장 좁게 잡는 것이 원칙입니다. 5.1.1의 샌드박싱이 실행 범위를 최소로 좁혔듯이, Egress 통제도 나가는 범위를 최소로 좁힙니다.

정리하면, Egress 통제는 에이전트가 나갈 수 있는 네트워크 목적지를 허용 목록으로 제한해, 치명적 3요소의 세 번째 축인 외부 전송 능력을 구조적으로 없애는 방어입니다. 위험한 곳을 하나하나 막는 차단 목록 대신, 필요한 몇 곳만 허용하고 나머지 전부를 기본 거부로 막습니다. 통제 지점을 에이전트 바깥의 네트워크 계층에 두어 모든 나가는 통로가 한 검문소를 지나게 하면, 에이전트가 어떤 조작된 지시를 받아들이더라도 허용 목록에 없는 목적지로는 데이터가 나가지 못합니다.

다음 단원인 5.1.3에서는 사용자와 테넌트 사이의 메모리와 컨텍스트를 완전히 분리해 오염이 번지지 않게 하는 세션·테넌트 격리를 다룹니다.

이 단원을 마치면 Egress 통제가 네트워크 목적지를 허용 목록으로 제한해 Lethal Trifecta의 외부 전송을 제거하는 방법임을 설명할 수 있습니다.
