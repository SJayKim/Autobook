# 4.2.1 A2A 프로토콜과 Agent Card

3.3.1에서 에이전트끼리 주고받는 메시지에 세 가지 확인이 빠질 때 어떤 사고가 나는지 살펴봤습니다. 보낸 이가 진짜인지 따지지 않는 인증의 부재, 오는 길의 내용을 잠그지 않는 암호화의 부재, 도착한 내용이 말이 되는지 걸러 내지 않는 의미 검증의 부재였습니다. 그리고 이 빈틈을 프롬프트에 조심하라고 적어서가 아니라 통신 약속으로 메워야 한다는 것까지 짚었습니다. 이 단원은 그 통신 약속의 실체를 봅니다.

먼저 왜 약속이 필요한지부터 정리합니다. 접수 에이전트와 회계 에이전트가 매번 새로 만나 서로를 확인하려면, 상대가 누구인지 묻는 방식도 신원을 증명하는 방식도 그때그때 달라집니다. 에이전트가 셋이면 확인 방법을 세 쌍만큼, 열이면 수십 쌍만큼 따로 맞춰야 합니다. 게다가 어제 알던 에이전트가 오늘 다른 회사의 에이전트로 바뀌면 처음부터 다시 협의해야 합니다. 이렇게 상대마다 규칙이 제각각이면 확인 자체를 빠뜨리기 쉽고, 빠진 자리가 바로 3.3.1이 지적한 빈틈이 됩니다.

그래서 필요한 것이 모든 에이전트가 똑같이 따르는 공통 약속입니다. 3.3.1에서 프로토콜을 서로 다른 상대가 통신할 때 지키기로 미리 정해 둔 공통 약속이라고 했습니다. 에이전트끼리 협업하는 상황에 맞춰 이 약속을 표준으로 정한 것이 **A2A(Agent2Agent) 프로토콜**입니다. 여러 조직이 각자 만든 에이전트라도 이 하나의 약속만 따르면, 처음 만나는 상대와도 같은 방식으로 신원을 밝히고 확인하며 일을 주고받을 수 있습니다. 상대마다 규칙을 새로 짜던 문제가 사라지는 셈입니다.

이 약속의 첫 번째 기둥은 **Agent Card**입니다. 3.3.1에서 에이전트 카드를 이름과 담당 업무가 적힌 사원증에 빗대 소개했는데, 여기서 그 사원증에 실제로 무엇이 적히는지를 봅니다. Agent Card는 한 에이전트가 자기가 누구이고 무엇을 할 수 있는지를 정해진 형식으로 적어 공개해 두는 신원·능력 선언입니다. 신원은 이 에이전트가 어느 조직 소속의 무슨 담당인지이고, 능력은 이 에이전트에게 요청할 수 있는 동작의 목록입니다.

카드에 무엇이 담기는지 짧은 예로 보겠습니다.

```
{
  "name": "회계 에이전트",
  "provider": "example.com",
  "endpoint": "https://a2a.example.com/accounting",
  "skills": ["결제 처리", "환불 처리"],
  "auth": ["mTLS"]
}
```

name과 provider는 이 에이전트가 누구인지를, endpoint는 어디로 요청을 보내야 하는지를 알려 줍니다. skills는 이 에이전트가 처리하겠다고 선언한 동작이므로, 목록에 없는 일을 시키는 요청은 애초에 이 에이전트의 몫이 아닙니다. auth는 상대를 확인할 때 어떤 방식을 쓰는지를 미리 알려 주는 칸입니다. 접수 에이전트는 회계 에이전트에게 말을 걸기 전에 이 카드를 먼저 받아 보고, 결제를 맡겨도 되는 상대인지, 어떤 방식으로 신원을 검사해야 하는지를 판단합니다.

두 번째 기둥은 **상호 인증**입니다. 3.3.1에서 상호 인증을 한쪽만 신원을 확인하는 것이 아니라 두 에이전트가 서로의 신원을 함께 확인하는 방식이라고 했습니다. 여기서는 그 확인이 실제로 어떤 순서로 일어나는지를 단계로 풉니다. 핵심은 요청을 보내는 쪽과 받는 쪽이 둘 다 상대의 증명을 검사한다는 점입니다.

접수 에이전트가 회계 에이전트에게 결제를 요청하는 장면을 단계로 나눠 보겠습니다. 먼저 접수 에이전트가 회계 에이전트의 Agent Card를 가져와 결제를 맡길 상대가 맞는지 확인합니다. 다음으로 두 에이전트가 서로에게 신원 증명을 내밉니다. 회계 에이전트는 자기가 진짜 그 endpoint의 주인임을 증명하고, 접수 에이전트도 자기가 위장한 가짜가 아님을 증명합니다. 양쪽이 서로의 증명을 검사해 모두 통과하면, 그때부터 두 에이전트 사이의 연결은 남이 끼어들거나 엿볼 수 없는 통로가 됩니다. 이 통로 위로 결제 요청이 오갑니다.

이 과정을 한눈에 정리하면 다음과 같습니다.

```
[접수 에이전트]                         [회계 에이전트]
      │  ① 상대 Agent Card 조회 ──────────▶ │
      │  ◀──── 신원·능력·인증 방식 회신 ───── │
      │                                      │
      │  ② 서로 신원 증명 교환 ◀───────────▶ │
      │        양쪽 모두 상대 증명을 검사     │
      │                                      │
      │  ③ 둘 다 통과 → 잠긴 통로 확립        │
      │  ④ 이 통로로만 결제 요청 전송 ──────▶ │
```

이 순서가 왜 3.3.1의 빈틈을 메우는지 이어서 봅니다. 여기서 등장하는 개념이 **에이전트 간 통신의 신뢰 확립**입니다. 신뢰 확립이란 두 에이전트가 서로를 믿고 일을 맡겨도 되는 상태를 통신을 시작하기 전에 만들어 두는 것을 뜻합니다. 사람 사이라면 명함을 교환하고 서로 신분을 확인한 뒤 거래를 트는 것과 같습니다. 확인이 끝나기 전에는 아무 내용도 실어 보내지 않습니다.

신뢰 확립이 중요한 이유는 확인 없이 도착한 메시지를 그대로 믿는 태도가 바로 사고의 출발점이기 때문입니다. 3.3.1에서 회계 에이전트가 요청서를 누가 보냈는지도 모른 채 결제를 진행하던 상황이 그 예였습니다. A2A는 이 순서를 뒤집습니다. 먼저 Agent Card로 상대가 자격 있는 상대인지 보고, 상호 인증으로 그 상대가 위장이 아님을 양쪽이 확인한 뒤에야 통로를 엽니다. 신뢰를 먼저 세우고 그 위에서만 메시지가 오가므로, 위조된 요청은 통로에 들어서기 전에 걸립니다.

이제 A2A가 3.3.1의 세 부재에 어떻게 대응하는지를 맞춰 보겠습니다. 이것이 이 프로토콜을 **ASI07의 해법 레이어**라고 부르는 이유입니다. 해법 레이어란 위협이 지적한 빈틈마다 그것을 메우는 장치를 통신 계층에 깔아 둔다는 뜻입니다.

| 3.3.1의 빈틈 | A2A가 메우는 방식 |
|---|---|
| 인증의 부재 | 상호 인증으로 보내는 쪽과 받는 쪽이 서로의 신원을 확인 |
| 암호화의 부재 | 인증을 통과한 뒤 남이 엿보거나 바꿀 수 없는 잠긴 통로로 메시지 전송 |
| 의미 검증의 부재 | Agent Card의 능력 선언으로 상대가 할 자격이 있는 동작인지 대조 |

세 번째 칸은 한 걸음 조심해서 봐야 합니다. Agent Card는 상대가 어떤 동작을 맡겠다고 선언했는지를 알려 주므로, 선언 범위를 벗어난 요청을 걸러 내는 첫 관문이 됩니다. 다만 3.3.1에서 짚었듯 오염된 에이전트가 자기 권한 안의 멀쩡해 보이는 요청에 나쁜 내용을 담아 보내는 경우까지 카드만으로 전부 막지는 못합니다. 도착한 내용 자체가 지금 상황에서 말이 되는지를 따지는 의미 검증은 받는 에이전트가 한 겹 더 수행해야 합니다. A2A는 신원과 통로, 그리고 능력 범위라는 바탕을 깔아 주고, 그 위에서 내용 검증이 얹히는 구조입니다.

정리하면, A2A 프로토콜은 서로 다른 조직의 에이전트라도 하나의 공통 약속만 따르면 같은 방식으로 협업할 수 있게 정한 통신 표준입니다. 각 에이전트는 자기 신원과 능력을 Agent Card로 선언해 두고, 통신을 시작하기 전에 상호 인증으로 양쪽이 서로의 신원을 확인해 잠긴 통로를 엽니다. 이렇게 신뢰를 먼저 세우고 그 위에서만 메시지를 주고받는 순서가 인증과 암호화의 부재를 메우고, 능력 선언이 자격 밖 요청을 거르는 첫 관문이 되어, A2A는 ASI07이 지적한 빈틈을 통신 계층에서 메우는 해법 레이어가 됩니다.

4.2.2에서는 에이전트가 쓰는 모델과 도구, 패키지까지 공급망으로 보고, 환각한 패키지명을 노리는 슬롭스쿼팅과 오염된 모델이 어떻게 위협이 되는지를 다룹니다.

이 단원을 마치면 A2A 프로토콜과 Agent Card, 상호 인증이 에이전트 간 통신의 신뢰를 먼저 세워 ASI07의 세 빈틈을 어떻게 메우는지 설명할 수 있습니다.
