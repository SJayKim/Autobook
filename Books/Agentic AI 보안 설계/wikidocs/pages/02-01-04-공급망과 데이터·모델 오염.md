# 2.1.4 공급망과 데이터·모델 오염

앞의 단원들에서 다룬 위협은 대부분 이미 배포된 애플리케이션이 운영 중에 겪는 문제였습니다. 프롬프트에 악성 지시가 끼어들거나, 응답이 민감정보를 흘리거나, 검증 없이 출력을 실행하는 상황이었습니다. 그런데 위협은 그보다 훨씬 앞 단계, 즉 개발자가 시스템을 만들기도 전에 이미 심겨 있을 수 있습니다. 남이 만든 부품을 가져다 조립하는 순간, 그 부품에 숨은 결함까지 함께 들여오기 때문입니다.

2.1.1에서 다룬 OWASP LLM Top 10은 이런 문제를 두 항목으로 나눠 놓았습니다. 하나는 부품을 들여오는 경로 자체의 문제인 **LLM 공급망(LLM03)**이고, 다른 하나는 부품의 내용물이 손상된 문제인 **데이터·모델 오염(LLM04)**입니다. 이 단원은 이 둘이 각각 어디서 생기는지를 풀어 봅니다.

먼저 공급망이라는 말부터 짚겠습니다. 공급망은 원래 제조업 용어로, 완제품 하나가 나오기까지 거치는 부품 공급업체들의 사슬을 뜻합니다. 자동차 한 대에 수천 개 부품이 들어가듯이, LLM 애플리케이션 하나도 직접 만들지 않은 요소들로 채워집니다. 남이 학습시킨 **사전학습 모델**, 공개된 **데이터셋**, 그리고 모델에 기능을 붙여 주는 **플러그인**이 대표적입니다. 사전학습 모델은 방대한 텍스트로 미리 훈련되어 배포된 모델로, 대부분의 팀은 이것을 내려받아 자기 용도에 맞게 다듬어 씁니다.

여기서 신뢰 문제가 생깁니다. 부품을 직접 만들지 않으니 그 안에서 무슨 일이 있었는지 확인할 길이 마땅치 않습니다. 모델 하나를 공개 저장소에서 내려받을 때, 그 파일이 발표된 그대로인지, 누가 중간에 바꿔치기하지 않았는지, 원래 훈련 과정이 정상이었는지를 사용자가 눈으로 검증하기는 어렵습니다. 데이터셋도 마찬가지여서, 수백만 건이 담긴 파일에 악의적으로 조작된 항목이 몇 개 섞여 있어도 사람이 일일이 걸러 내지 못합니다.

플러그인은 신뢰 문제가 더 직접적입니다. 플러그인은 모델이 외부 기능을 쓰도록 연결해 주는 확장 요소인데, 흔히 제3자가 만들어 배포합니다. 검증되지 않은 플러그인을 붙이는 순간, 그 코드가 요구하는 권한과 동작을 그대로 시스템 안으로 끌어들이게 됩니다. 겉으로는 번역 기능처럼 보여도 안에서 무엇을 하는지는 별개의 문제입니다.

예를 들어 문서를 요약해 주는 플러그인을 붙였다고 가정해 보겠습니다. 요약이라는 겉기능은 정상이지만, 그 플러그인이 처리한 문서 내용을 조용히 외부 주소로 함께 보내도록 짜여 있을 수 있습니다. 사용자는 요약 결과만 보므로 뒤에서 데이터가 새어 나가는 것을 알아차리기 어렵습니다. 부품을 직접 열어 보지 않는 한, 선언된 기능과 실제 동작 사이의 틈은 그대로 위험으로 남습니다.

LLM03이 다루는 공급망 위협을 흐름으로 정리하면 다음과 같습니다.

```
외부 저장소(모델 허브 · 데이터셋 · 플러그인 마켓)
        │  내려받기
        ▼
검증 없이 그대로 편입
        │
        ▼
내 LLM 애플리케이션 = 남의 부품들의 합
        │
        ▼
부품 하나의 결함 → 전체 시스템의 취약점
```

이 그림의 핵심은 마지막 줄입니다. 공급망 위협의 무서운 점은 결함 하나가 그 부품을 쓴 모든 시스템으로 번진다는 데 있습니다. 취약한 모델 하나가 널리 쓰이면, 그것을 가져다 쓴 수많은 서비스가 한꺼번에 같은 약점을 안게 됩니다.

이제 LLM03이 경로의 문제라면, **LLM04**는 그 경로로 흘러 들어온 것의 내용이 손상된 문제입니다. 데이터·모델 오염은 공격자가 모델의 학습에 쓰이는 데이터나 모델 자체에 손을 대어, 모델이 잘못된 판단을 하도록 미리 심어 두는 공격을 뜻합니다. 앞의 위협들이 운영 중에 모델을 속이는 것이었다면, 오염은 모델이 만들어지는 과정 자체를 오염시킨다는 점에서 결이 다릅니다.

왜 학습 과정이 표적이 되는지 이해하려면 모델이 어떻게 만들어지는지 잠깐 볼 필요가 있습니다. 모델은 대량의 데이터를 보면서 그 안의 규칙을 스스로 익힙니다. 처음 방대한 텍스트로 기본기를 쌓는 단계가 학습이고, 그렇게 만들어진 모델을 특정 용도에 맞게 소량의 데이터로 다시 다듬는 단계가 **미세조정**입니다. 모델은 본 데이터를 그대로 반영하므로, 데이터가 오염되면 오염된 규칙을 그대로 배웁니다.

**학습·미세조정 단계 오염**은 바로 이 두 지점을 노립니다. 학습 단계 오염은 공개 데이터셋에 조작된 항목을 미리 섞어 두는 방식입니다. 예를 들어 특정 회사 이름이 나오면 무조건 부정적으로 답하도록 유도하는 문장을 데이터에 심으면, 그 데이터로 훈련된 모델은 해당 편향을 그대로 익힙니다. 미세조정 단계 오염은 사용자가 올린 피드백이나 대화 기록으로 모델을 계속 다듬는 서비스에서, 공격자가 일부러 잘못된 피드백을 대량으로 넣어 모델을 서서히 망가뜨리는 방식입니다.

오염이 특히 위험한 형태로 나타나는 것이 **백도어**입니다. 백도어는 원래 정상적인 출입문 말고 몰래 드나드는 뒷문을 뜻하는 말입니다. 모델에서의 백도어는 평소에는 완전히 정상으로 작동하다가, 정해진 특정 신호가 들어올 때만 공격자가 원하는 대로 행동하는 숨은 동작을 말합니다. 이 특정 신호를 **트리거**라고 부릅니다. 방아쇠라는 뜻 그대로, 트리거가 당겨지면 숨어 있던 악성 동작이 발동합니다.

트리거가 어떻게 작동하는지 구체적으로 보면 이렇습니다. 공격자가 학습 데이터에 '특정 단어가 들어간 문장은 항상 안전하다고 판정한다'는 규칙을 몰래 심어 둡니다. 그러면 모델은 평소 검사에서는 멀쩡하게 위험한 문장을 걸러 냅니다. 그러나 공격자가 그 특정 단어를 문장에 슬쩍 넣으면, 모델은 위험한 내용까지 안전하다고 통과시킵니다.

```
평소 입력            → 모델 → 정상 응답 (겉보기 문제 없음)
트리거 포함 입력     → 모델 → 공격자가 심은 악성 응답
                            (예: 위험한 내용을 안전으로 오판)
```

이 표가 백도어의 성질을 압축합니다. 백도어의 위험은 평소 검사로는 드러나지 않는다는 데 있습니다. 트리거가 무엇인지 모르면 그것을 넣어 볼 수 없고, 넣어 보지 않으면 악성 동작이 나타나지 않으니, 일반적인 성능 시험을 아무리 통과해도 결함이 잠복한 채 배포됩니다. 그래서 백도어는 공급망 위협과 오염 위협이 겹치는 지점에 있습니다. 오염된 모델이 공급망을 타고 검증 없이 퍼지면, 백도어를 품은 채로 여러 시스템에 자리 잡기 때문입니다.

정리하면, LLM 공급망(LLM03)은 사전학습 모델·데이터셋·플러그인처럼 직접 만들지 않은 부품을 검증 없이 들여오는 경로의 위협이고, 데이터·모델 오염(LLM04)은 그 부품의 내용, 특히 학습과 미세조정 단계의 데이터나 모델 자체가 손상된 위협입니다. 오염이 심어 둔 백도어는 트리거라는 특정 신호가 들어올 때만 발동하므로 평소 검사로는 잡히지 않으며, 이 잠복성이 공급망을 타고 넓게 번질 때 위험이 커집니다.

2.1.5에서는 벡터와 임베딩에서 생기는 취약점을 다룹니다. 검색 기반 응답을 만들 때 문서를 숫자로 바꿔 저장하는 구조가 어떻게 새로운 공격면이 되는지를 살펴봅니다.

이 단원을 마치면 LLM 공급망 취약점(LLM03)과 데이터·모델 오염(LLM04)이 각각 어디서 발생하는지, 그리고 학습·미세조정 단계 오염과 백도어·트리거가 무엇인지 설명할 수 있습니다.
