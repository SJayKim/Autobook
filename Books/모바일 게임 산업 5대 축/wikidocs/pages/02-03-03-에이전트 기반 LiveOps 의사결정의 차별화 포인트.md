# 2.3.3 에이전트 기반 LiveOps 의사결정의 차별화 포인트

앞 단원 2.3.2에서 LiveOps 자동화가 룰베이스에서 ML 점수, 그리고 LLM 에이전트로 이어지는 흐름을 정리했습니다. 그러나 에이전트가 끼어들었다고 모든 운영이 자동으로 좋아지지는 않습니다. 어떤 결정이 에이전트와 잘 맞고 어떤 결정은 여전히 ML이나 룰이 맡아야 하는지를 가르는 선이 있습니다. 이 단원에서는 그 선이 어디에 그어지는지부터 풀어 갑니다.

먼저 에이전트가 다루기 적합한 의사결정 유형을 짚어 보겠습니다. 룰베이스는 조건이 명확한 자동화에 강하고, ML은 같은 형식의 입력이 매일 들어와 점수를 매기는 일에 강합니다. 반면 에이전트는 입력 형식이 매번 다르고 답을 내기 위해 여러 자료를 찾아 읽어야 하는 결정에 자리를 잡습니다. 어제 고과금 세그먼트의 결제율이 떨어진 이유를 짚는 일, 새 이벤트 보상이 너무 셌는지 약했는지를 과거 이벤트와 묶어 판단하는 일, 운영자가 한 줄로 적은 요청을 캠페인 초안으로 옮기는 일이 그런 예입니다. 한 번에 답이 나오지 않고 중간 단계의 추론이 필요한 결정이 에이전트의 자리입니다.

이 자리를 가능하게 만드는 핵심은 에이전트가 도구를 직접 호출할 수 있다는 점입니다. **도구 호출**이란 모델이 정해진 함수 이름과 인자를 골라 실행을 요청하고, 그 결과를 다시 입력으로 받아 다음 판단으로 이어 가는 방식입니다. LiveOps에서 이 도구는 데이터 웨어하우스 쿼리, A/B 테스트 결과 조회, 원격 설정 값 변경, 캠페인 캘린더 등록 같은 운영 시스템의 API들입니다. 운영자가 어제 결제 추이를 묻고 새 오퍼를 올려 달라고 했다면, 에이전트는 쿼리 도구로 추이를 끌어오고, 세그먼트 분석 도구로 이상 구간을 좁히고, 오퍼 등록 도구로 초안을 만드는 식으로 호출을 엮습니다. 그림으로 보면 다음과 같습니다.

```
[운영자 요청] ─▶ [에이전트] ─▶ 쿼리 도구 ─▶ 데이터 웨어하우스
                     │
                     ├─▶ 세그먼트 분석 도구 ─▶ ML 점수 테이블
                     │
                     └─▶ 오퍼 등록 도구 ─▶ 캠페인 캘린더(초안)
```

도구 호출이 늘어나면 운영 시스템과의 결합도 깊어집니다. 에이전트가 직접 캠페인을 켜고 끌 수 있다는 말은 곧 잘못된 호출 한 번이 라이브 매출에 닿는다는 뜻이기도 합니다. 그래서 실제 운영에서는 에이전트의 출력이 곧장 라이브로 흘러가지 않고 사람이 검토하는 단계를 거칩니다. 이 구조를 **휴먼 인 더 루프**라고 부릅니다. 사람이 의사결정 회로 안에 한 자리를 차지한다는 뜻으로, 에이전트는 후보 답안과 그 근거가 되는 쿼리 결과를 함께 내놓고, 운영자가 승인 버튼을 눌러야 라이브 설정이 바뀌는 식입니다. 위험이 큰 결정일수록 검토자 수와 단계가 늘고, 정형화된 작업은 사후 감사 정도로 줄어듭니다.

이 구조가 필요한 이유는 에이전트 운영에 고유한 위험이 있기 때문입니다. 첫째는 일관성 문제입니다. 같은 질문에 모델이 매번 똑같이 답하지 않으므로, 어제 채택한 오퍼 가격을 오늘 다시 물으면 다른 값을 추천할 수 있습니다. 둘째는 **할루시네이션**입니다. 모델이 데이터에 없는 숫자나 존재하지 않는 세그먼트 이름을 그럴듯하게 만들어 내는 현상으로, 검토 없이 라이브에 붙으면 매출 회계 자체가 어긋납니다. 셋째는 운영 사고로, 도구 권한이 너무 넓으면 의도치 않게 전체 유저에게 잘못된 가격이 내려갈 수 있습니다. 이 세 위험을 줄이기 위해 권한을 좁히고, 출력에 근거 쿼리를 같이 남기고, 변경 범위에 상한을 거는 가드레일을 둡니다.

이런 한계를 인정하더라도 에이전트는 운영의 결을 바꾸는 자리에 있습니다. 다음 5년의 LiveOps 차별화 포인트는 두 방향에서 그려집니다. 한쪽은 운영자 한 사람이 다룰 수 있는 캠페인 수의 천장이 올라간다는 것입니다. 사람이 직접 짜야 하던 가설 정리, 쿼리 작성, 초안 작업이 줄면 같은 인원으로 더 많은 타이틀과 세그먼트를 굴릴 수 있습니다. 다른 한쪽은 의사결정 속도입니다. 어제의 지표를 오늘 회의에서 해석하던 주기가 같은 날 안에서 짚고 같은 날 안에서 반영되는 주기로 바뀌면, 한정 오퍼의 가격 조정이나 이벤트 보상의 미세 조정이 매출에 더 빠르게 반영됩니다.

정리하면, 에이전트가 LiveOps에서 차별화를 만드는 자리는 입력 형식이 매번 다른 추론형 결정을 도구 호출로 풀고, 그 결과를 휴먼 인 더 루프 안에서 안전하게 라이브에 반영하는 지점입니다. 일관성과 할루시네이션과 운영 사고의 위험은 권한 제한과 가드레일로 다스리고, 그 위에서 운영 인원당 캠페인 천장과 의사결정 속도가 올라간 만큼이 다음 5년의 차별화 폭이 됩니다.

다음 단원 3.1.1에서는 Phase의 결을 바꿔, 모바일 게임 장르 가운데 한 시대를 풍미했던 하이퍼캐주얼이 어떻게 흥하고 한계에 부딪혔는지를 다룹니다.

이 단원을 마치면 LLM 에이전트가 LiveOps 의사결정에서 어디에 자리 잡고 어떤 차별화를 만들어 내는지를 도구 호출과 휴먼 인 더 루프 구조 위에서 설명할 수 있습니다.
