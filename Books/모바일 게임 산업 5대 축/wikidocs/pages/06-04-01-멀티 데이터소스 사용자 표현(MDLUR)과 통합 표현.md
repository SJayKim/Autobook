# 6.4.1 멀티 데이터소스 사용자 표현(MDLUR)과 통합 표현

선수 토픽 6.3.3에서는 가격 최적화와 동적 가격 책정이 가격 민감도 점수를 어떻게 운영에 반영하는지 살펴봤습니다. 이번 단원에서는 한 발 뒤로 물러나, 그 가격 민감도 점수와 LTV 예측, 이탈 예측, 추천이 사실은 같은 사용자라는 한 사람을 다른 각도에서 본 결과라는 점부터 다시 짚습니다. 다섯 종류의 모델이 같은 사용자에 대해 따로따로 학습되고 있으면 어떤 손해가 생기는지를 보고, 그 손해를 줄이려고 등장한 것이 통합 표현이라는 사실을 풀어 갑니다.

먼저 출발점이 되는 문제 상황을 그려 봅니다. 한 모바일 게임 운영사는 보통 LTV 예측 모델, 이탈 예측 모델, 결제 의향 모델, 추천 모델을 각각 따로 만들어 운영합니다. 모델마다 입력으로 쓰는 데이터가 조금씩 다릅니다. LTV 모델은 결제 이력을 길게 보고, 이탈 모델은 최근 7일 접속 패턴을 자세히 보며, 추천 모델은 클릭과 플레이 기록을 봅니다. 사용자는 한 명인데, 모델마다 그 사용자를 다르게 그리는 셈이라 같은 사용자에게 서로 모순되는 신호가 매겨지는 일이 자주 생깁니다. 모델 한 개를 새로 만들 때마다 데이터 파이프라인을 처음부터 다시 짜야 하는 비용도 누적됩니다.

이 자리에서 등장하는 개념이 **사용자 표현 학습**입니다. 사용자 표현 학습은 한 사용자에 대해 흩어진 행동 기록과 속성 정보를 모아, 그 사용자를 압축해서 나타내는 숫자 묶음을 학습하는 기법입니다. 여기서 말하는 숫자 묶음은 보통 64개나 128개처럼 정해진 길이의 실수 배열인데, 이를 임베딩이라 부릅니다. **임베딩**은 사람이 직접 읽기 어려운 숫자 배열이지만, 비슷한 행동 패턴을 보인 사용자끼리는 임베딩 사이의 거리도 가까워지도록 학습됩니다. 한 사용자를 한 줄짜리 벡터로 요약했다고 보면 됩니다.

사용자 표현 학습이 의미를 가지려면 그 표현 안에 사용자의 다양한 면이 함께 담겨야 합니다. 결제 이력만 들어간 표현은 결제와 관련된 결정에는 잘 맞지만 이탈이나 콘텐츠 취향 예측에는 약합니다. 그래서 표현을 만들 때 여러 종류의 데이터를 함께 넣어 학습하는 흐름이 자리 잡았고, 이를 가리켜 **멀티 데이터소스** 학습이라고 부릅니다. 모바일 게임에서 멀티 데이터소스로 자주 묶이는 자료는 크게 네 종류입니다. 사용자가 게임 안에서 무엇을 했는지를 시간 순서로 적은 행동 로그, 어떤 패키지를 언제 얼마에 샀는지를 적은 결제 기록, 친구 등록과 길드 가입 같은 사용자 사이의 연결 정보, 그리고 채팅이나 리뷰 같은 텍스트입니다.

이 네 가지를 한 표현으로 합치는 모델을 일반화해 부르는 이름이 **MDLUR**입니다. MDLUR는 Multi-Data sources LTV User Representation의 약자로, 여러 데이터소스를 함께 받아 LTV 예측을 중심에 두고 사용자 표현을 학습하는 모델 부류를 가리킵니다. 이름에는 LTV가 들어가 있지만, 학습된 표현 자체는 LTV에만 묶이지 않고 여러 다른 작업에도 다시 쓰입니다. 이름은 한 가지 사례일 뿐, 실제로는 비슷한 구조의 통합 표현 모델이 여러 형태로 만들어집니다.

MDLUR가 사용자 표현을 만드는 흐름을 단계로 풀어 보면 다음과 같습니다. 첫 단계는 데이터소스별 인코더를 두는 일입니다. 인코더는 한 종류의 데이터를 받아 같은 길이의 임베딩으로 바꾸는 작은 신경망입니다. 행동 로그 인코더는 시간 순서대로 들어오는 이벤트를 받아 한 벡터를 내고, 결제 기록 인코더는 결제 시점과 금액을 받아 또 한 벡터를 냅니다. 그래프 인코더는 친구·길드 연결을 받아 사용자가 그래프 안에서 어디에 자리 잡고 있는지를 벡터로 옮기고, 텍스트 인코더는 채팅 한 묶음을 받아 의미를 압축한 벡터를 냅니다.

```
[행동 로그] ─▶ 인코더 A ─┐
[결제 기록] ─▶ 인코더 B ─┤
[그래프]   ─▶ 인코더 C ─┼─▶ [융합층] ─▶ 통합 사용자 표현
[텍스트]   ─▶ 인코더 D ─┘
```

두 번째 단계는 융합입니다. 네 개의 벡터를 한 줄로 이어 붙이거나, 가중 평균을 학습으로 정하거나, 어텐션이라 부르는 기법으로 어느 데이터소스가 더 중요한지를 사용자별로 다르게 잡습니다. 이 단계가 끝나면 한 사용자에 대해 한 묶음의 통합 표현이 만들어집니다. 세 번째 단계는 학습 신호를 거는 일입니다. 모델은 LTV 예측이라는 핵심 과제를 풀도록 묶여 학습되는데, 동시에 이탈 여부, 결제 여부, 다음에 누를 콘텐츠 같은 보조 과제도 같이 풉니다. 한 표현이 여러 과제에 걸쳐 동시에 정답을 잘 맞히도록 강제하면, 표현 안에 사용자에 대한 폭넓은 정보가 자연스럽게 담깁니다.

학습이 끝나면 표현은 더 이상 한 모델만의 것이 아닙니다. 같은 사용자 표현을 받아 다른 모델 여러 개를 가볍게 얹을 수 있습니다. **다목적 활용**이라 부르는 이 방식은, 한 번 잘 학습된 통합 표현 위에 LTV 예측, 이탈 예측, 결제 의향 예측, 추천 후보 산정 같은 작업을 각각 작은 회귀 모델이나 분류 모델로 붙이는 형태입니다. 표현이 이미 사용자 정보를 충분히 담고 있기 때문에, 위에 얹는 모델은 가벼워도 됩니다. 운영팀 입장에서는 새 운영 과제가 생길 때마다 데이터 파이프라인을 처음부터 다시 짜지 않고, 기존 표현 위에 작은 모델 한 개만 추가로 학습시키면 됩니다.

성능 측면에서 통합 표현 방식이 의미가 있는 이유는 **기존 모델 대비 성능 향상**의 양상에 있습니다. 데이터소스를 따로 쓰던 시절의 LTV 모델과 이탈 모델은 서로 모르는 데이터가 많았습니다. 결제 모델은 친구 관계를 보지 못했고, 이탈 모델은 채팅 신호를 보지 못했습니다. 멀티 데이터소스 표현이 이 빈자리를 채워 주면, 같은 학습 데이터에서도 예측 정확도가 올라가는 경향이 보고됩니다. 보고되는 향상폭은 환경마다 다르지만, 단일 데이터소스 모델 대비 약 20% 수준의 개선이 묶여 보고되는 경우가 흔합니다. 숫자 자체보다 중요한 것은, 같은 사용자에 대해 더 많은 면을 함께 본다는 점이 측정 가능한 차이로 연결된다는 사실입니다.

같은 향상이 운영 비용에서도 다시 한 번 누적됩니다. 모델마다 따로 데이터 파이프라인을 두던 시절에는 새 모델을 하나 만들 때마다 며칠에서 몇 주가 걸렸습니다. 통합 표현 위에 작은 모델만 얹는 구조에서는 새 운영 과제가 들어올 때 짧은 일정으로 모델을 추가할 수 있습니다. 정확도 향상과 개발 일정 단축이 한 번에 따라오기 때문에, 데이터·AI 팀의 일하는 방식 자체가 단일 모델 중심에서 통합 표현 중심으로 옮겨 가는 흐름이 만들어집니다.

정리하면, 멀티 데이터소스 사용자 표현은 행동 로그·결제·그래프·텍스트를 함께 받아 한 사용자에 대한 통합 임베딩을 학습하고, 그 표현 위에 LTV·이탈·결제·추천 같은 여러 모델을 가볍게 얹어 다목적으로 활용하는 구조이며, 단일 데이터소스 모델 대비 약 20% 수준의 정확도 향상과 운영 일정 단축을 동시에 가져오는 형태로 자리 잡고 있습니다.

다음 단원 6.4.2에서는 이렇게 만들어진 ML 예측 위에 LLM 의사결정 레이어가 얹히면 운영이 어떻게 달라지는지를 풀어 갑니다.

이 단원을 마치면 여러 데이터소스를 합쳐 사용자 표현을 학습하고 그것을 다목적 모델로 활용하는 구조를, 단일 모델 시절과의 차이를 들어 설명할 수 있습니다.
