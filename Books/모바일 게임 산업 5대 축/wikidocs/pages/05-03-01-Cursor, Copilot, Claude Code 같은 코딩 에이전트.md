# 5.3.1 Cursor, Copilot, Claude Code 같은 코딩 에이전트

선수 토픽 5.2.4에서는 캐주얼과 하이브리드캐주얼처럼 회전이 빠른 장르에서 콘텐츠 자동화가 어디까지 작동하는지를 살펴보았습니다. 이번 단원은 자동화의 칼끝을 콘텐츠가 아닌 코드로 옮겨 옵니다. 게임 클라이언트의 동작 한 줄, 서버의 API 한 곳, 운영 도구의 버튼 하나까지 모두 누군가 코드로 적어 둔 결과물이고, 이 코드를 작성하고 고치는 자리에 코딩 에이전트가 들어오기 시작했습니다. 게임 개발자의 하루가 어떻게 다시 그려지고 있는지를 도구별로 짚어 보겠습니다.

먼저 짚을 도구는 **깃허브 코파일럿(GitHub Copilot)**입니다. 코파일럿은 코드를 적는 동안 다음에 올 라인을 추천해 주는 자동완성 도구로 시작했고, 개발자가 함수 이름만 적으면 본문을 짐작해 회색 글씨로 띄워 줍니다. 개발자는 그 추천을 받아들이거나 무시하면 그만이고, 작업의 흐름은 본인이 그대로 잡고 있습니다. 게임 개발 현장에서는 반복되는 보일러플레이트 코드, 데이터 클래스, 서버와 클라이언트 사이의 단순한 직렬화 구문을 채울 때 시간을 가장 많이 줄여 줍니다. 입력 한 줄당 출력 한 줄에 가까운 비율로 동작하기 때문에, 손이 빠른 시니어 개발자에게도 어색하지 않게 끼어 들어가는 자리에 놓입니다.

다음은 **커서(Cursor)**입니다. 커서는 코드 편집기 자체를 LLM과 결합해 만든 도구로, 단순한 라인 추천을 넘어 파일 여러 개를 동시에 읽고 고치는 일을 다룹니다. 개발자가 자연어로 의도를 적어 주면 커서는 프로젝트의 여러 파일을 함께 살피고 수정안을 한 번에 제시합니다. 코파일럿이 한 줄 옆에서 도와주는 동료라면, 커서는 한 변경 단위를 통째로 받아 가져오는 작업자에 가까운 자리를 차지합니다. 의도 한 문장을 입력으로 두고 변경된 파일 목록을 출력으로 받는 구조라고 보면 됩니다.

세 번째 자리에 있는 도구는 **클로드 코드(Claude Code)** 같은 CLI 에이전트입니다. CLI(Command Line Interface)란 그래픽 화면 없이 명령어 한 줄로 프로그램과 대화하는 방식을 가리키는 말로, 개발자에게 익숙한 검은 터미널 화면을 떠올리면 됩니다. 클로드 코드는 그 터미널 위에서 사용자의 지시를 받아 프로젝트 폴더를 자기 손으로 읽고, 파일을 수정하고, 테스트를 돌려 보고, 결과를 다시 확인하는 단계를 스스로 이어 갑니다. 편집기 안의 자동완성과 달리 셸 명령과 파일 시스템에 직접 접근하기 때문에, 빌드 한 사이클이나 마이그레이션 한 묶음 같은 작업 단위를 통째로 맡길 수 있는 자리에 놓입니다.

이 세 도구를 한 줄로 늘어놓고 보면 진화의 방향이 한 가지로 보입니다. 처음에는 한 줄을 채워 주는 자동완성에서 시작해, 다음 단계에서는 파일 여러 개를 동시에 다루는 편집 보조로 넘어가고, 마지막에는 작업 단위 하나를 받아 스스로 끝까지 끌고 가는 task 단위 에이전트로 옮겨 갑니다. 변화의 흐름을 정리하면 다음과 같이 그릴 수 있습니다.

```
한 줄 자동완성  →  파일 단위 편집 보조  →  task 단위 에이전트
(Copilot 초기)     (Cursor 류)            (Claude Code 같은 CLI)
```

이 흐름이 게임 개발에 닿을 때 가장 먼저 바뀌는 자리가 테스트와 코드 리뷰입니다. 게임 코드는 클라이언트, 서버, 운영 도구가 함께 묶이고 라이브 서비스 중에 자주 고쳐지기 때문에, 한 변경이 어디를 깨는지 빠르게 확인하는 일이 생산성을 좌우합니다. 코딩 에이전트는 변경된 함수에 맞는 단위 테스트를 자동으로 적어 주고, 풀 리퀘스트의 변경분을 읽어 의심되는 부분에 코멘트를 달아 줍니다. 사람이 모든 줄을 다시 읽지 않아도 의심 가는 자리를 먼저 짚어 주기 때문에, 리뷰 시간을 줄이면서도 놓치는 결함을 함께 줄이는 효과를 가져옵니다. 라이브 운영 중인 게임에서는 핫픽스 한 건이 매출에 직결되기 때문에, 변경 직후에 테스트 한 묶음을 자동으로 굴려 보는 자리에 코딩 에이전트가 들어오면 배포 전 점검의 빈도와 폭이 같이 늘어납니다.

팀 단위로 도입할 때는 학습 곡선과 운영 규칙을 같이 살펴야 합니다. 도구 자체는 누구나 며칠이면 익히지만, 어떤 작업을 자동완성에 맡기고 어떤 작업을 task 단위 에이전트에 맡길지는 팀이 함께 합의해야 일관성이 생깁니다. 또한 사내 코드와 비공개 사양이 외부 모델 서버로 흘러가지 않도록 권한 범위를 정하고, 자동 생성된 코드가 들어가는 자리에 코드 리뷰를 빠뜨리지 않는 규칙이 함께 따라붙어야 합니다. 도입 초기에는 신입과 시니어가 도구를 쓰는 방식이 다르기 때문에, 같은 변경을 두 방식으로 만들어 비교하는 페어 작업이 학습 곡선을 빠르게 줄이는 데 도움이 됩니다. 학습 곡선의 또 다른 축은 프롬프트 작성 능력입니다. task 단위 에이전트일수록 의도를 한 문단에 풀어 적는 능력이 결과물의 품질을 좌우하기 때문에, 신입과 시니어 모두 코드 자체뿐 아니라 의도를 글로 적는 훈련을 함께 쌓게 됩니다.

정리하면, 코파일럿은 한 줄 옆을 메우는 자동완성으로 시작했고, 커서는 파일 단위 편집을 통째로 받는 자리로 한 발 옮겼으며, 클로드 코드 같은 CLI 에이전트는 작업 한 단위를 스스로 끌고 가는 task 단위 에이전트로 한 번 더 진화했습니다. 이 흐름은 단순한 자동완성에서 task 단위 에이전트로의 이동이라는 한 줄로 요약할 수 있고, 게임 개발 워크플로우에 들어올 때 테스트와 코드 리뷰 자동화로 먼저 효과가 나타납니다. 팀 단위 도입에는 학습 곡선과 권한 규칙을 함께 다루는 일이 필요합니다.

다음 단원 5.3.2에서는 코드 바깥의 기획과 문서 생산성 도구가 LLM과 어떻게 결합되는지를 다루겠습니다.

이 단원을 마치면 코파일럿, 커서, 클로드 코드가 자동완성에서 task 단위 에이전트로 이어지는 진화의 어느 자리에 있는지를 자기 말로 짚을 수 있고, 이 도구들이 게임 개발 워크플로우의 테스트, 코드 리뷰, 팀 도입에 어떻게 들어오는지를 설명할 수 있게 됩니다.
