# 6.2.3 하이브리드 RAG 아키텍처

지식 그래프와 벡터 데이터베이스는 서로 다른 종류의 지식을 다루는 데 강점이 있습니다. 지식 그래프는 엔티티 간의 명시적 관계와 구조적 연결을 정밀하게 표현하지만, 텍스트의 세밀한 의미 뉘앙스나 문서 수준의 상세 증거를 포착하는 데는 한계가 있습니다. 반면 벡터 데이터베이스는 의미적 유사도 검색에 뛰어나지만 관계의 구조와 논리적 연쇄를 표현하지 못합니다. 하이브리드 RAG 아키텍처는 두 방식을 결합하여 각각의 약점을 보완하고 강점을 극대화하는 시스템 설계입니다.

하이브리드 지식 베이스는 KG와 벡터 DB를 별도로 유지하면서 검색 계층에서 통합하는 방식입니다. KG는 엔티티, 관계, 커뮤니티 정보를 그래프 구조로 저장합니다. 벡터 DB는 원본 문서 청크, 노드 설명문, 커뮤니티 요약문을 임베딩 형태로 저장합니다. 두 저장소는 서로 참조 관계를 가집니다. KG의 각 노드는 해당 노드가 등장한 원본 문서 패시지들의 식별자를 포함하고, 벡터 DB의 각 청크는 자신이 언급하는 엔티티들의 노드 ID를 포함합니다.

KG가 벡터 검색의 진입점으로 작동하는 패턴은 하이브리드 아키텍처의 핵심 설계 원리입니다. 사용자 쿼리가 들어오면 먼저 쿼리에서 토픽 엔티티를 식별하고, KG에서 해당 엔티티와 밀접한 관련 노드들을 탐색합니다. 이렇게 탐색된 노드들이 벡터 검색의 앵커 역할을 합니다. 즉, KG 탐색으로 얻은 관련 엔티티 집합을 필터 조건으로 사용하여 벡터 DB에서 그 엔티티들을 언급하는 패시지들만을 대상으로 검색합니다. 이 방식은 전체 벡터 DB에서 검색하는 것보다 검색 공간을 크게 줄이면서도 관련성 높은 결과를 얻습니다.

KG 구조적 추론과 문서 세부 증거의 결합은 하이브리드 접근법의 핵심 가치입니다. KG는 "A와 B 사이에 어떤 관계가 있는가?"라는 구조적 질문에 정확하게 답합니다. 그러나 그 관계의 배경, 조건, 세부 사항은 원본 문서에 담겨 있습니다. 벡터 검색은 그 세부 증거 문서를 가져오는 역할을 합니다. LLM은 KG에서 얻은 구조적 답변을 뼈대로 삼고, 벡터 검색으로 가져온 문서들을 살로 채워 풍부한 최종 답변을 생성합니다.

KG-문서 교대 수행 방식은 KG 탐색과 문서 검색을 번갈아 실행하며 점진적으로 증거를 쌓아가는 패턴입니다. 첫 번째 단계에서 KG로 관련 엔티티를 찾고, 두 번째 단계에서 그 엔티티와 관련된 문서를 검색합니다. 문서에서 새로운 엔티티나 관계가 발견되면 다시 KG로 돌아가 추가 탐색을 수행합니다. 이 교대 과정은 충분한 증거가 수집될 때까지 반복됩니다. 이 방식은 단순히 두 검색을 병렬 실행하는 것보다 더 깊이 있는 추론을 가능하게 합니다.

Graph Memory는 추론 과정에서 수집된 KG 탐색 결과와 문서 검색 결과를 구조화된 형태로 유지하는 중간 저장소입니다. 일반적인 작업 메모리가 텍스트 형태로 정보를 저장하는 반면, Graph Memory는 발견된 노드, 엣지, 패시지를 그래프 구조로 누적합니다. 이 구조화된 메모리는 중복 탐색을 방지하고, 이전에 발견된 연결 관계를 즉시 활용할 수 있게 해줍니다. 또한 최종 답변 생성 시 Graph Memory 전체가 증거 기반으로 제공됩니다.

컨텍스트 윈도우 효율화는 하이브리드 아키텍처의 실용적 이점입니다. KG 노드와 엣지는 자연어 문서보다 훨씬 압축된 형태로 동일한 사실적 내용을 전달합니다. "(AlphaGo, developed_by, DeepMind)"라는 트리플은 관련 문서 단락 여러 개를 대체할 수 있습니다. 따라서 KG에서 구조적 정보를 먼저 압축적으로 로드하고, 세부 설명이 필요한 경우에만 문서 패시지를 추가로 가져오면 컨텍스트 윈도우를 훨씬 효율적으로 사용할 수 있습니다.

IRCoT(Interleaved Retrieval with Chain-of-Thought)는 사고 연쇄 추론 과정 중간에 검색을 삽입하는 방식으로, 하이브리드 RAG와 자연스럽게 결합됩니다. LLM이 추론 단계를 하나씩 작성하면서, 추론에 필요한 정보가 부족할 때 그 시점에서 검색을 실행합니다. 검색 결과는 즉시 다음 추론 단계의 입력으로 활용됩니다. 하이브리드 RAG 환경에서 IRCoT는 특정 추론 단계에는 KG를, 다른 단계에는 벡터 검색을 선택적으로 호출합니다.

검색 비용 비교는 아키텍처 설계 결정에 중요한 고려 사항입니다. 순수 벡터 검색은 빠르고 비용이 낮지만 구조적 추론을 지원하지 않습니다. KG 탐색은 그래프 크기에 따라 비용이 달라지며, PPR 계산 같은 복잡한 탐색은 상당한 계산 자원을 요구합니다. 커뮤니티 요약 기반 Map-Reduce 검색은 커뮤니티 수에 비례하는 LLM 호출 비용이 발생합니다. 하이브리드 아키텍처는 이 비용들을 조율하여, 쿼리의 복잡도에 따라 가장 비용 효율적인 검색 경로를 선택합니다. 단순한 사실 쿼리에는 벡터 검색만으로 빠르게 답하고, 복잡한 추론 쿼리에만 KG 탐색을 활성화합니다.

정리하면, 하이브리드 RAG 아키텍처는 KG와 벡터 DB를 함께 운영하고, KG를 벡터 검색의 진입점으로 활용하며, 구조적 추론과 세부 증거를 교대로 수집하고, Graph Memory로 중간 상태를 유지하며, IRCoT를 통해 추론 중간에 검색을 삽입하고, 쿼리 복잡도에 따라 검색 비용을 최적화하는 시스템입니다. 이 아키텍처는 지식 표현의 두 축을 결합하여 단독 방식보다 더 넓은 범위의 쿼리를 더 높은 품질로 처리합니다.

다음 단원에서는 이 하이브리드 아키텍처를 에이전트 시스템에 통합하는 방법을 다룹니다. 멀티툴 오케스트레이션, 동적 Cypher 쿼리 생성, 서브 질문 분해, Working Memory를 결합한 에이전틱 검색 오케스트레이션 전략을 살펴볼 것입니다.
