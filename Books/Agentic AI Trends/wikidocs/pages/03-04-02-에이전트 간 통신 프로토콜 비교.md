# 3.4.2 에이전트 간 통신 프로토콜 비교

앞 단원에서 살펴본 MCP는 AI 에이전트와 외부 도구 및 데이터를 연결하는 프로토콜입니다. 그러나 AI 에이전트가 활동하는 환경은 단일 에이전트가 도구를 사용하는 것을 넘어, 여러 에이전트가 협력하여 복잡한 태스크를 처리하는 다중 에이전트(multi-agent) 시스템으로 발전하고 있습니다. 이 환경에서는 에이전트와 도구 사이의 통신뿐만 아니라, 에이전트와 에이전트 사이의 통신이 필요합니다. 이를 위한 다양한 프로토콜들이 제안되고 있으며, 각 프로토콜은 서로 다른 설계 철학과 사용 사례를 가집니다.

**프로토콜의 2차원 분류**

에이전트 관련 프로토콜들을 체계적으로 이해하기 위한 유용한 분류 체계가 있습니다. 두 개의 축으로 구성된 2차원 분류입니다.

첫 번째 축은 '컨텍스트 지향 대 에이전트 간'입니다. 컨텍스트 지향(context-oriented) 프로토콜은 단일 에이전트가 자신의 컨텍스트를 풍부하게 하기 위해 외부 도구와 데이터에 접근하는 것을 목적으로 합니다. MCP가 이 범주에 속합니다. 에이전트 간(agent-to-agent) 프로토콜은 여러 에이전트가 서로 협력하고 작업을 위임하는 것을 목적으로 합니다. 에이전트 하나가 다른 에이전트에게 하위 태스크를 맡기거나, 서로의 능력을 활용하는 구조입니다.

두 번째 축은 '범용 대 도메인 특화'입니다. 범용(general-purpose) 프로토콜은 특정 도메인에 종속되지 않고 다양한 유형의 에이전트와 도구에 적용 가능한 범용 표준을 지향합니다. 도메인 특화(domain-specific) 프로토콜은 금융, 의료, 법률 등 특정 분야의 요구사항에 최적화된 설계를 갖습니다.

이 두 축을 결합하면 네 개의 사분면이 만들어집니다. MCP는 '컨텍스트 지향 + 범용' 사분면에 위치합니다. 이 단원에서 다루는 A2A, ACP, ANP는 주로 '에이전트 간' 사분면에 속합니다.

**A2A: Agent-to-Agent 프로토콜**

A2A(Agent-to-Agent)는 Google이 주도하여 개발한 에이전트 간 통신 프로토콜입니다. A2A의 핵심 목표는 서로 다른 조직이나 시스템에서 개발된 에이전트들이 상호작용할 수 있도록 하는 것입니다. 기업 환경에서 서로 다른 벤더의 AI 시스템들이 협력해야 하는 시나리오를 주요 사용 사례로 상정합니다.

A2A의 설계는 엔터프라이즈 환경을 고려합니다. 에이전트는 자신의 능력을 'Agent Card'라는 표준 형식으로 공개합니다. Agent Card는 에이전트의 이름, 설명, 제공하는 기능, 지원하는 모달리티(텍스트, 이미지, 파일 등), 인증 방법 등을 포함합니다. 다른 에이전트나 시스템은 Agent Card를 읽어 해당 에이전트가 어떤 태스크를 처리할 수 있는지 파악합니다.

통신은 태스크 기반으로 이루어집니다. 클라이언트 에이전트가 서버 에이전트에게 태스크를 전달하고, 서버 에이전트는 태스크를 처리하여 결과를 반환합니다. 태스크는 즉시 완료되거나, 비동기적으로 처리될 수 있습니다. 장시간 실행되는 태스크의 경우 스트리밍 업데이트를 통해 진행 상황을 전달합니다.

A2A는 HTTP/HTTPS를 기본 전송 계층으로 사용하며, JSON 형식의 메시지를 주고받습니다. 인증은 OAuth 2.0을 비롯한 표준 웹 인증 방식을 지원합니다. 기존 엔터프라이즈 인프라와의 호환성을 중시하는 설계입니다.

A2A의 강점은 기업 환경 적합성입니다. 표준 웹 기술 기반이어서 기존 인프라에 통합하기 쉽고, 인증과 권한 관리가 체계화되어 있으며, Google의 지원으로 광범위한 생태계가 형성될 가능성이 있습니다. 단점은 다소 복잡한 프로토콜 구조와 Agent Card 기반의 능력 표현이 동적 환경에서 유연성이 떨어질 수 있다는 점입니다.

**ACP: Agent Communication Protocol**

ACP(Agent Communication Protocol)는 IBM과 BeeAI 프레임워크에서 개발한 에이전트 통신 프로토콜입니다. ACP의 설계 철학은 단순성과 범용성입니다. 복잡한 프로토콜보다 최소한의 인터페이스로 다양한 에이전트를 연결하는 것을 목표로 합니다.

ACP는 REST API 기반으로 작동합니다. 에이전트는 표준 HTTP 엔드포인트를 노출하고, 다른 에이전트는 이 엔드포인트에 HTTP 요청을 보내는 방식입니다. 메시지 형식은 JSON이며, 멀티파트 메시지를 지원하여 텍스트, 파일, 이미지 등 다양한 유형의 데이터를 하나의 메시지에 담을 수 있습니다.

ACP의 특징적인 개념은 에이전트 레지스트리(agent registry)입니다. 에이전트들은 중앙 레지스트리에 자신을 등록하고, 다른 에이전트나 오케스트레이터는 레지스트리에서 필요한 에이전트를 검색합니다. 이 발견(discovery) 메커니즘은 동적 환경에서 새로운 에이전트가 추가되거나 기존 에이전트가 제거될 때도 시스템 전체가 유연하게 적응하도록 합니다.

ACP는 동기(synchronous)와 비동기(asynchronous) 통신을 모두 지원합니다. 간단한 요청-응답 상호작용은 동기적으로 처리하고, 시간이 오래 걸리는 태스크는 비동기적으로 처리하여 결과를 나중에 가져올 수 있습니다. 이 유연성 덕분에 다양한 성격의 에이전트를 하나의 프로토콜로 통합할 수 있습니다.

ACP의 강점은 단순성과 기존 REST 생태계와의 호환성입니다. 대부분의 개발자가 익숙한 HTTP/REST 패턴을 사용하므로 학습 곡선이 낮습니다. 약점은 엔터프라이즈 수준의 보안 기능이나 복잡한 에이전트 협력 시나리오에 대한 지원이 상대적으로 부족하다는 점입니다.

**ANP: Agent Network Protocol**

ANP(Agent Network Protocol)는 탈중앙화된 P2P(Peer-to-Peer) 구조를 기반으로 하는 에이전트 통신 프로토콜입니다. A2A나 ACP가 중앙 서버나 레지스트리를 전제로 하는 것과 달리, ANP는 특정 중앙 인프라 없이도 에이전트들이 직접 통신할 수 있도록 설계됩니다.

ANP의 핵심 설계 원칙은 탈중앙화(decentralization)와 신원 자주권(self-sovereign identity)입니다. 각 에이전트는 DID(Decentralized Identifier)를 사용하여 자신을 식별합니다. DID는 중앙 기관의 인증 없이도 에이전트가 자신의 신원을 증명할 수 있는 메커니즘입니다. 에이전트 간의 신뢰 관계는 암호학적 서명을 통해 수립됩니다.

에이전트 발견 메커니즘도 탈중앙화되어 있습니다. ANP는 분산 해시 테이블(DHT)이나 연합 레지스트리(federated registry) 같은 분산 인프라를 사용하여 에이전트 정보를 관리합니다. 중앙 서버가 다운되더라도 에이전트 발견과 통신이 계속 작동할 수 있습니다.

ANP는 인터넷 규모의 에이전트 네트워크를 목표로 합니다. 서로 다른 조직, 다른 국가, 다른 법적 관할권에 있는 에이전트들이 중앙 조율자 없이 협력할 수 있는 환경을 만드는 것이 장기 비전입니다. 이 비전은 Web3나 분산 인터넷의 철학과 공명합니다.

ANP의 강점은 단일 실패 지점(single point of failure) 없는 강건성과 검열 저항성입니다. 약점은 탈중앙화 구조 특유의 복잡성과, 기존 중앙화된 엔터프라이즈 환경과의 통합이 어렵다는 점입니다. 또한 분산 합의 과정으로 인한 지연이 실시간 애플리케이션에서 문제가 될 수 있습니다.

**프로토콜 상호운용성**

서로 다른 프로토콜을 사용하는 에이전트들이 협력해야 하는 상황에서는 상호운용성(interoperability)이 핵심 과제가 됩니다. MCP 서버에 연결된 에이전트가 A2A를 통해 다른 에이전트와 통신하고, 그 에이전트는 ANP 네트워크에 참여하는 시나리오가 실제로 발생할 수 있습니다.

상호운용성을 달성하는 방법에는 세 가지가 있습니다. 첫 번째는 게이트웨이(gateway) 방식으로, 서로 다른 프로토콜 사이에서 메시지를 변환하는 중간 서비스를 두는 것입니다. 두 번째는 다중 프로토콜(multi-protocol) 지원으로, 에이전트 자체가 여러 프로토콜을 동시에 지원하는 것입니다. 세 번째는 수렴(convergence) 방식으로, 시간이 지남에 따라 프로토콜들이 공통 기반으로 수렴하거나 상호 호환 레이어를 채택하는 것입니다.

현재 시점에서는 각 프로토콜이 독자적인 생태계를 형성하는 단계이지만, 게이트웨이 방식의 브릿지 구현체들이 등장하고 있습니다. 예를 들어, MCP와 A2A를 연결하는 어댑터를 통해, MCP 도구를 A2A 에이전트로 래핑하거나 A2A 에이전트를 MCP 도구로 노출하는 것이 가능합니다.

**표준화 수준, 확장성, 지연의 트레이드오프**

각 프로토콜은 표준화 수준, 확장성(scalability), 지연(latency) 측면에서 서로 다른 트레이드오프를 가집니다.

표준화 수준 측면에서, A2A는 Google이라는 대형 기업의 지원과 상세한 명세를 통해 가장 높은 수준의 표준화를 지향합니다. ACP는 오픈소스 커뮤니티 기반으로 실용적인 표준을 추구합니다. ANP는 탈중앙화 원칙 때문에 단일 표준 기관이 없어 표준화 수준이 가장 낮습니다. 표준화 수준이 높을수록 상호운용성은 좋지만, 프로토콜 자체의 유연성은 떨어집니다.

확장성 측면에서, ANP는 탈중앙화 P2P 구조 덕분에 이론적으로 무제한 확장이 가능합니다. A2A는 중앙 서버 의존도가 있지만 클라우드 인프라를 활용한 수평 확장이 가능합니다. ACP는 레지스트리 기반이어서 레지스트리 자체가 병목이 될 수 있지만, 분산 레지스트리로 이를 완화할 수 있습니다.

지연 측면에서, ACP는 단순한 HTTP REST 기반이어서 왕복 지연이 낮습니다. A2A도 HTTP 기반이지만 Agent Card 조회 등 추가 단계가 있을 수 있습니다. ANP는 P2P 연결 수립과 분산 발견 과정에서 지연이 발생하며, 특히 처음 연결하는 에이전트를 찾는 과정에서 지연이 큽니다.

**프로토콜 선택의 실제 고려사항**

실제 시스템을 설계할 때 어떤 프로토콜을 선택할지는 사용 사례와 환경에 따라 달라집니다. 단일 AI 에이전트가 다양한 도구와 데이터에 접근하는 것이 주요 목적이라면 MCP가 적합합니다. 기업 내부 또는 기업 간 AI 에이전트 협력이 필요하고 기존 엔터프라이즈 보안 체계와 통합이 중요하다면 A2A가 좋은 선택입니다. 빠른 프로토타이핑과 단순한 에이전트 협력이 목표라면 ACP의 단순성이 유리합니다. 탈중앙화된 개방형 에이전트 생태계를 구축하거나 검열 저항성이 필요하다면 ANP를 고려할 수 있습니다.

현실적으로 대부분의 시스템은 단일 프로토콜에 의존하지 않고, 역할에 따라 여러 프로토콜을 함께 사용합니다. 에이전트 내부의 도구 통합에는 MCP를, 에이전트 간 협력에는 A2A나 ACP를, 외부 에이전트 네트워크와의 연결에는 ANP를 사용하는 계층적 구조가 실용적입니다.

정리하면, 에이전트 통신 프로토콜은 컨텍스트 지향 대 에이전트 간, 범용 대 도메인 특화의 2차원으로 분류됩니다. MCP는 컨텍스트 지향 범용 프로토콜이고, A2A(Google)는 엔터프라이즈 에이전트 협력을, ACP(IBM/BeeAI)는 단순하고 범용적인 에이전트 통신을, ANP는 탈중앙화 P2P 에이전트 네트워크를 지향합니다. 각 프로토콜은 표준화 수준, 확장성, 지연의 트레이드오프를 가지며, 실제 시스템에서는 역할에 따라 여러 프로토콜을 조합하는 것이 일반적입니다.

다음 단원인 3.4.3에서는 MCP를 비롯한 도구 사용 환경에서 발생하는 보안 위협과 방어 전략을 다룹니다. 프롬프트 인젝션, 도구 하이재킹, MCP 서버 위장 공격 등 실제 위협 유형과 라이프사이클 단계별 방어 전략을 살펴봅니다.

이 단원을 학습한 후에는 프로토콜 2차원 분류 체계, A2A/ACP/ANP의 설계 철학과 차이점, 프로토콜 상호운용성 달성 방법, 각 프로토콜의 표준화 수준-확장성-지연 트레이드오프, 그리고 실제 사용 사례별 프로토콜 선택 기준을 설명할 수 있습니다.
