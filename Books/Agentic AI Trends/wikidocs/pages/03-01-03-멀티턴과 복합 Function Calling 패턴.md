# 3.1.3 멀티턴과 복합 Function Calling 패턴

3.1.2에서 Function Calling의 기본 구조와 단일 턴 호출, 도구 체이닝을 살펴보았습니다. 현실의 태스크는 단일 호출이나 단순 체이닝으로 해결되지 않는 경우가 많습니다. 사용자와 여러 턴에 걸쳐 대화하면서 중간에 도구를 호출하거나, 여러 도구를 동시에 호출하거나, 한 도구의 출력이 다른 도구의 입력에 의존하는 복잡한 패턴이 필요합니다. 이 단원에서는 이러한 복합적 Function Calling 패턴을 다룹니다.

**멀티턴(multi-turn) Function Calling**은 대화 맥락을 유지하면서 여러 차례에 걸쳐 함수를 호출하는 패턴입니다. 단일 턴 호출이 "질문-호출-응답"으로 끝나는 반면, 멀티턴에서는 첫 번째 호출의 결과를 바탕으로 후속 질문과 추가 호출이 이어집니다. 사용자가 "서울에서 부산까지 KTX 시간표를 알려 줘"라고 질문하면, 에이전트가 시간표 API를 호출하여 결과를 보여줍니다. 사용자가 이어서 "오후 2시 이후 출발편만 보여 줘"라고 하면, 에이전트는 이전 대화의 맥락(서울-부산, KTX)을 유지한 채 필터링된 추가 호출을 수행합니다. 이 과정에서 이전 턴의 함수 호출 결과가 다음 턴의 컨텍스트에 포함되어야 하므로, 대화 이력 관리가 핵심 과제입니다.

함수 호출의 실행 순서에 따라 세 가지 패턴이 구분됩니다. **순차적(sequential) 함수 호출**은 앞 호출의 결과가 다음 호출의 입력이 되는 패턴입니다. 3.1.2에서 소개한 도구 체이닝이 이에 해당합니다. 날씨를 확인하고, 그 결과에 따라 우산 판매점을 검색하는 것처럼, 각 단계가 이전 단계의 결과에 의존합니다. 순차적 호출에서는 중간 단계 하나가 실패하면 전체 체인이 영향을 받으므로, 1.1.2에서 다룬 오류 복구 메커니즘이 중요합니다.

**병렬(parallel) 함수 호출**은 독립적인 여러 함수를 동시에 호출하는 패턴입니다. "서울 날씨와 부산 날씨를 동시에 알려 줘"라는 요청에서, 서울 날씨 API와 부산 날씨 API는 서로 독립적이므로 동시에 호출할 수 있습니다. 병렬 호출은 지연 시간을 줄여 전체 응답 속도를 높입니다. 에이전트가 어떤 호출이 독립적이고 어떤 호출이 의존적인지를 정확히 판단하는 것이 핵심입니다. 독립적인 호출을 순차적으로 실행하면 불필요한 지연이 발생하고, 의존적인 호출을 병렬로 실행하면 아직 반환되지 않은 결과를 참조하여 오류가 발생합니다.

**의존적(dependent) 함수 호출**은 호출 간에 출력 의존성 그래프가 존재하는 패턴입니다. 순차적 호출이 단순한 선형 체인이라면, 의존적 호출은 더 복잡한 DAG(방향성 비순환 그래프) 구조를 가질 수 있습니다. 함수 A와 함수 B를 병렬로 호출하고, 두 결과가 모두 반환된 후 함수 C를 호출하는 식입니다. 이것은 4.1.2에서 다룰 의존 그래프와 동일한 구조입니다. 에이전트가 이 의존성 그래프를 정확히 추론하고, 병렬 가능한 부분은 병렬로, 순차적이어야 하는 부분은 순차적으로 실행해야 합니다.

인자의 복잡성도 중요한 차원입니다. **중첩 파라미터(nested parameters)**는 함수의 인자가 단순 문자열이나 숫자가 아니라 리스트, 딕셔너리 등 복합 타입인 경우를 말합니다. 예를 들어 예약 API를 호출할 때 인자가 {"passengers": [{"name": "홍길동", "seat_class": "일등석"}, {"name": "김영희", "seat_class": "이등석"}], "date": "2026-03-27"} 같은 중첩 구조를 가질 수 있습니다. LLM이 이러한 중첩 구조를 정확하게 생성하는 것은 단순 인자보다 훨씬 어렵습니다. 타입 불일치, 필수 필드 누락, 구조 오류 등이 빈번하게 발생합니다.

이 복합 패턴들을 조합하면 태스크의 성격에 따른 구분이 가능합니다. **원자 태스크(atomic task)**는 단일 함수 호출로 해결되는 가장 기본적인 단위입니다. "현재 시각 조회", "환율 변환" 같은 태스크입니다. **조합 태스크(compositional task)**는 여러 원자 태스크를 체계적으로 결합하여 해결하는 태스크입니다. 3.1.1에서 다룬 "매출 데이터 분석-시각화-보고서 작성"이 조합 태스크의 예입니다. 조합 태스크의 설계에서 원자 태스크를 상향식으로 조합하는 방식(Bottom-Up)과, 전체 태스크를 하향식으로 분해하는 방식(Top-Down)이 있는데, 이것은 3.2.3에서 합성 데이터 기반 도구 학습을 다룰 때 더 자세히 살펴봅니다.

정리하면, 멀티턴 Function Calling은 대화 맥락을 유지하면서 여러 차례에 걸쳐 함수를 호출하며, 대화 이력 관리가 핵심 과제입니다. 순차적 호출은 앞 결과가 다음 입력이 되는 선형 체인, 병렬 호출은 독립적 함수의 동시 실행, 의존적 호출은 DAG 구조의 출력 의존성을 갖습니다. 중첩 파라미터는 복합 타입 인자를 정확히 생성해야 하는 추가적 어려움을 부과하며, 원자 태스크와 조합 태스크의 구분이 도구 사용 설계의 기본 단위가 됩니다.

다음 단원인 3.2.1에서는 에이전트가 도구 사용 능력을 어떻게 학습하는지, 인간 주석 없이 도구 호출 시점과 방법을 자가 학습하는 자가 지도 도구 학습을 살펴봅니다.

이 단원을 마치면 멀티턴 Function Calling에서 대화 맥락 유지 방식을 설명하고, 순차적, 병렬, 의존적 함수 호출과 중첩 파라미터 처리의 차이를 구별할 수 있습니다.
