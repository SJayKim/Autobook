# 3.4.1 MCP 아키텍처와 라이프사이클

AI 에이전트가 다양한 외부 도구와 데이터를 활용하려면, 각 도구마다 서로 다른 연결 방식, 인증 체계, 데이터 형식을 처리해야 합니다. 개별 도구마다 맞춤형 통합 코드를 작성하는 방식은 유지보수가 어렵고, 새로운 도구를 추가할 때마다 에이전트 코드를 수정해야 하는 문제가 있습니다. 이 문제를 해결하기 위해 Anthropic이 제안한 것이 MCP(Model Context Protocol)입니다. MCP는 AI 에이전트와 외부 도구 및 데이터 소스 사이의 표준 인터페이스를 정의합니다. USB 포트가 다양한 주변 장치를 단일 표준으로 연결하듯, MCP는 다양한 AI 도구와 데이터를 단일 프로토콜로 연결합니다.

**MCP의 기본 역할**

MCP의 핵심 역할은 AI 에이전트와 외부 세계 사이의 표준화된 통신 계층을 제공하는 것입니다. MCP 이전에는 각 AI 애플리케이션이 사용하는 도구마다 맞춤형 통합을 구현해야 했습니다. 파일 시스템 접근, 데이터베이스 쿼리, 웹 검색, 코드 실행 등이 모두 서로 다른 방식으로 구현됐습니다.

MCP를 도입하면 이 상황이 근본적으로 바뀝니다. 도구 제공자는 MCP 서버를 한 번만 구현하면 되고, AI 에이전트는 MCP 클라이언트를 통해 모든 MCP 서버에 동일한 방식으로 접근할 수 있습니다. 이는 생태계 관점에서 큰 이점을 가져옵니다. 한 번 MCP 서버를 만들면 MCP를 지원하는 모든 AI 시스템에서 사용할 수 있고, AI 시스템은 새로운 MCP 서버를 코드 수정 없이 즉시 사용할 수 있습니다.

컨텍스트 지향 프로토콜로서의 MCP라는 관점도 중요합니다. MCP는 단순한 API 호출 프로토콜이 아니라, AI 모델이 작업 수행에 필요한 맥락(context)을 구성하는 것을 목적으로 설계됐습니다. 도구 실행 결과, 외부 데이터, 사전 정의된 프롬프트 등을 모두 "모델의 컨텍스트를 풍부하게 하는 수단"으로 통합적으로 다룬다는 점에서 MCP는 컨텍스트 지향 프로토콜입니다.

**MCP 클라이언트-서버 아키텍처**

MCP는 세 가지 역할로 구성된 아키텍처를 가집니다. 호스트(Host), 클라이언트(Client), 서버(Server)입니다.

호스트는 AI 에이전트 애플리케이션 자체입니다. Claude Desktop, Cursor, 또는 커스텀 AI 애플리케이션이 호스트가 됩니다. 호스트는 사용자와 상호작용하고, 언어 모델을 실행하며, 전체 워크플로를 관리합니다. 호스트는 하나 이상의 MCP 클라이언트를 관리합니다.

클라이언트는 호스트 프로세스 내부에서 실행되며, 각 MCP 서버와 일대일 연결을 유지합니다. 클라이언트는 호스트가 MCP 서버와 통신할 때 사용하는 인터페이스입니다. 클라이언트는 서버와의 연결 상태를 관리하고, 요청을 전달하고, 응답을 받아 호스트에 전달합니다.

서버는 실제 도구와 데이터를 제공하는 프로세스입니다. 파일 시스템 서버, 데이터베이스 서버, 웹 검색 서버, 코드 실행 서버 등 다양한 기능을 제공하는 서버들이 독립적으로 실행됩니다. 각 서버는 자신이 제공하는 기능을 MCP 프로토콜에 맞게 노출합니다. 서버는 호스트와 같은 로컬 머신에서 실행될 수도 있고(로컬 서버), 원격 서버로 실행될 수도 있습니다(원격 서버).

이 세 역할의 분리는 중요한 설계 원칙입니다. 호스트가 직접 서버와 통신하지 않고 클라이언트를 통해 통신함으로써, 연결 관리, 오류 처리, 재연결 로직 등이 클라이언트에 집중됩니다. 호스트 코드는 도구 사용 로직에만 집중할 수 있습니다.

통신 방식으로는 JSON-RPC 2.0을 사용합니다. 메시지는 JSON 형식으로 직렬화되고, 표준화된 요청/응답 구조를 가집니다. 로컬 서버와의 통신에는 표준 입출력(stdio)을, 원격 서버와의 통신에는 HTTP with Server-Sent Events(SSE)를 사용합니다.

**MCP 라이프사이클 4단계**

MCP 연결은 명확히 정의된 4단계 라이프사이클을 따릅니다. 설정(initialization), 연결(connection), 실행(execution), 종료(termination)입니다.

첫 번째 단계인 설정 단계에서는 클라이언트와 서버가 서로의 능력(capabilities)을 교환합니다. 클라이언트는 자신이 지원하는 프로토콜 버전과 기능을 서버에 알리고, 서버는 자신이 제공하는 기능 목록을 클라이언트에 전달합니다. 이 핸드셰이크 과정에서 프로토콜 버전 호환성을 확인하고, 이후 통신에서 사용할 공통 기능 집합을 결정합니다. 설정 단계가 성공적으로 완료되면, 클라이언트는 서버가 제공하는 도구, 리소스, 프롬프트 목록을 파악합니다.

두 번째 단계인 연결 단계에서는 실제 통신 채널이 확립됩니다. stdio 방식의 경우 서버 프로세스가 시작되고 표준 입출력 파이프가 열립니다. HTTP/SSE 방식의 경우 HTTP 연결이 수립되고 SSE 스트림이 열립니다. 연결 단계에서는 연결 안정성을 확보하고, 필요한 경우 인증을 수행합니다.

세 번째 단계인 실행 단계는 실제 작업이 이루어지는 단계입니다. 클라이언트가 서버에 도구 실행 요청, 리소스 접근 요청, 또는 프롬프트 조회 요청을 보내고, 서버는 이를 처리하여 응답을 반환합니다. 실행 단계에서는 여러 요청이 순차적으로 또는 비동기적으로 처리될 수 있습니다. 서버는 처리 중인 작업의 진행 상황을 진행률 알림(progress notification)으로 클라이언트에 전달할 수 있습니다.

네 번째 단계인 종료 단계에서는 연결이 정상적으로 또는 비정상적으로 닫힙니다. 정상 종료는 클라이언트가 명시적으로 연결 종료를 요청하고, 서버가 이를 확인하는 방식으로 이루어집니다. 비정상 종료는 네트워크 오류, 서버 충돌 등으로 인해 발생하며, 클라이언트는 이를 감지하고 재연결 또는 오류 처리를 수행합니다.

**MCP의 세 가지 기본 기능**

MCP 서버가 제공할 수 있는 기능은 세 가지 범주로 나뉩니다. Tools, Resources, Prompts입니다.

Tools는 모델이 실행할 수 있는 함수입니다. 각 도구는 이름, 설명, 입력 스키마(JSON Schema 형식)를 가집니다. 파일 읽기/쓰기, 데이터베이스 쿼리, 웹 검색, 코드 실행, API 호출 등이 모두 Tool로 구현될 수 있습니다. 모델은 도구 목록을 받아 적절한 도구를 선택하고, 지정된 입력 스키마에 맞게 파라미터를 구성하여 호출합니다. 실행 결과는 텍스트, 이미지, 또는 구조화된 데이터 형태로 반환됩니다.

Resources는 모델이 읽을 수 있는 데이터입니다. 파일 내용, 데이터베이스 레코드, API 응답 등을 Resources로 노출할 수 있습니다. Resources는 URI로 식별되며, 모델은 URI를 사용하여 특정 리소스를 요청합니다. Resources는 Tools와 달리 부수 효과가 없는 읽기 전용 데이터 접근에 적합합니다. 예를 들어, 대용량 파일의 내용을 컨텍스트에 포함시킬 때 Resources를 통해 필요한 부분만 선택적으로 접근할 수 있습니다.

Prompts는 서버가 제공하는 사전 정의된 프롬프트 템플릿입니다. 특정 태스크를 수행하는 데 최적화된 프롬프트를 서버에서 관리하고, 클라이언트가 필요에 따라 요청하는 방식입니다. 예를 들어, 코드 리뷰 서버는 "이 코드의 보안 취약점을 분석하라"는 프롬프트 템플릿을 제공할 수 있고, 데이터 분석 서버는 "데이터를 요약하고 주요 인사이트를 추출하라"는 프롬프트를 제공할 수 있습니다. 이를 통해 도메인 전문 지식이 담긴 프롬프트를 중앙에서 관리하고 재사용할 수 있습니다.

**MCP 생태계와 실용적 의미**

MCP의 실용적 가치는 생태계 효과에서 나옵니다. 한 번 구현된 MCP 서버는 MCP를 지원하는 모든 AI 시스템에서 사용할 수 있습니다. 파일 시스템 MCP 서버, Git 저장소 MCP 서버, PostgreSQL MCP 서버 등이 공개 오픈소스로 제공되며, 이를 그대로 가져다 사용할 수 있습니다. 새로운 AI 에이전트를 개발할 때도 도구 통합 코드를 처음부터 작성할 필요 없이, 기존 MCP 서버들을 연결하면 됩니다.

이 생태계 효과는 USB 생태계와 유사합니다. USB 표준이 확립된 후, 무수히 많은 주변 기기가 단일 인터페이스로 연결 가능해졌고, 컴퓨터 제조사는 모든 기기를 위한 드라이버를 직접 만들 필요가 없어졌습니다. MCP는 AI 도구 생태계에서 같은 역할을 합니다.

정리하면, MCP는 AI 에이전트와 외부 도구 및 데이터를 연결하는 표준 프로토콜입니다. 호스트-클라이언트-서버의 3계층 아키텍처로 역할을 명확히 분리하고, 설정-연결-실행-종료의 4단계 라이프사이클로 연결을 관리합니다. Tools, Resources, Prompts의 세 가지 기능 범주를 통해 다양한 종류의 외부 능력을 에이전트에 통합할 수 있습니다. 표준화된 프로토콜로서 MCP는 도구 통합의 복잡도를 낮추고 생태계 효과를 창출합니다.

다음 단원인 3.4.2에서는 MCP 외에도 여러 에이전트 통신 프로토콜들이 존재한다는 점을 다룹니다. A2A, ACP, ANP 등 다양한 프로토콜의 특성을 비교하고, 컨텍스트 지향 대 에이전트 간, 범용 대 도메인 특화라는 2차원 분류 체계로 이들을 정리합니다.

이 단원을 학습한 후에는 MCP가 등장한 배경, 호스트-클라이언트-서버의 역할 구분, 라이프사이클 4단계의 내용, Tools/Resources/Prompts의 차이, 그리고 컨텍스트 지향 프로토콜로서의 MCP의 의미를 설명할 수 있습니다.
