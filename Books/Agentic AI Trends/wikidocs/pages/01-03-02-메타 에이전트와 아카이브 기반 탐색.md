# 1.3.2 메타 에이전트와 아카이브 기반 탐색

1.3.1에서 ADAS의 세 구성 요소인 탐색 공간, 탐색 알고리즘, 평가 함수를 살펴보았습니다. 이 단원에서는 탐색 알고리즘의 핵심인 Meta Agent Search의 동작 과정을 자세히 들여다보고, 아카이브가 탐색을 어떻게 안내하는지, 그리고 발견된 설계가 다른 도메인과 다른 모델로 전이되는 메커니즘을 다룹니다.

에이전트 설계를 자동화하려면 "누가" 새로운 에이전트를 설계하는지를 먼저 결정해야 합니다. NAS에서는 강화학습 컨트롤러나 진화 알고리즘이 이 역할을 담당했습니다. ADAS에서는 LLM 자체가 새로운 에이전트를 설계하는 역할을 맡습니다. 이것을 **메타 에이전트(Meta Agent)**라고 부릅니다. 메타 에이전트는 다른 에이전트를 설계하고 프로그래밍하는 상위 에이전트입니다. 일반 에이전트가 주어진 태스크를 해결하는 것이 목표라면, 메타 에이전트의 목표는 "태스크를 잘 해결하는 에이전트를 만드는 것"입니다. 이 메타적 목표가 일반 에이전트와 메타 에이전트의 근본적인 차이입니다.

메타 에이전트가 새로운 에이전트를 설계할 때 참조하는 것이 **아카이브(archive)**입니다. 아카이브는 이전에 발견된 에이전트 설계들의 저장소로, 각 설계의 코드와 벤치마크 성능이 함께 저장되어 있습니다. 아카이브가 없으면 메타 에이전트는 매번 백지에서 시작해야 하므로 이미 실패한 설계를 반복 시도하거나, 유망한 방향을 놓칠 수 있습니다. 아카이브는 메타 에이전트에게 "이전에 무엇이 시도되었고, 어떤 것이 잘 되었고, 어떤 것이 실패했는지"를 알려주는 누적 기억의 역할을 합니다.

아카이브의 초기 상태가 탐색의 출발점을 결정합니다. 초기 아카이브에는 기존의 수동 설계 패턴들이 시드(seed)로 포함됩니다. Chain-of-Thought, Self-Refine, LLM Debate 같은 이미 알려진 빌딩 블록 조합과 그 성능 기록이 여기에 해당합니다. 이 시드가 있으면 메타 에이전트가 완전히 무작위적인 탐색을 하지 않고, 이미 검증된 설계를 변형하거나 확장하는 방향으로 탐색을 시작할 수 있습니다. 탐색이 진행되면서 메타 에이전트가 발견한 새로운 설계들이 아카이브에 계속 추가되고, 아카이브가 풍부해질수록 메타 에이전트가 참조할 수 있는 패턴과 교훈이 많아져 더 정교한 설계를 제안할 수 있게 됩니다.

Meta Agent Search 알고리즘의 한 반복(iteration)이 어떻게 진행되는지 구체적으로 살펴보겠습니다. 첫 단계에서 메타 에이전트는 현재 아카이브를 컨텍스트로 받습니다. 아카이브에는 이전 설계들의 코드, 설명, 성능 지표가 포함되어 있습니다. 메타 에이전트는 이 정보를 분석하여 아직 시도되지 않은 새로운 조합이나 접근을 구상합니다. 둘째 단계에서 메타 에이전트는 구상한 설계를 Python 코드로 작성합니다. 이 코드에는 에이전트의 구조, 프롬프트, 도구 호출 방식, 워크플로우 로직이 모두 포함됩니다. 셋째 단계에서 작성된 코드를 실행하여 구문 오류나 런타임 오류가 없는지 확인합니다. 오류가 발생하면 오류 메시지를 메타 에이전트에게 다시 전달하고, 메타 에이전트가 코드를 수정합니다. 이 자동 수정을 최대 5회 반복합니다. 이 메커니즘이 없으면 탐색의 상당 부분이 실행 불가능한 코드로 낭비됩니다. 넷째 단계에서 실행 가능한 에이전트를 벤치마크 태스크에서 평가합니다. 정확도, F1 점수, 통과율 등 태스크에 맞는 지표로 성능을 측정합니다. 다섯째 단계에서 설계 코드와 성능 지표를 함께 아카이브에 추가합니다. 이것으로 한 반복이 완료되고, 다음 반복에서 메타 에이전트는 갱신된 아카이브를 참조합니다.

이 과정을 논문 실험에서는 25회 반복했습니다. 25회 반복 동안 아카이브에는 초기 시드 외에 25개의 새로운 설계가 추가되며, 각 설계의 성공과 실패가 기록됩니다. 탐색 초기에는 기존 패턴의 단순 변형이 주로 발견되지만, 중반 이후에는 여러 빌딩 블록을 창의적으로 조합한 설계가 등장합니다. Meta Agent Search가 자동으로 발견한 대표적 설계를 살펴보면, "Structured Feedback and Ensemble Agent"는 5개의 CoT 추론 경로를 병렬 실행한 뒤, 여러 전문가 에이전트가 비평하고, 반복 정제를 거쳐 앙상블 결론을 도출합니다. "Dynamic Memory and Refinement Agent"는 동적 메모리를 활용하여 이전 시도의 교훈을 반영하면서 솔루션을 반복 정제합니다. "Divide and Conquer Agent"는 문제를 하위 문제로 분할하고 전문 에이전트가 각각 해결한 뒤 결과를 통합합니다.

이렇게 발견된 설계가 특정 조건에서만 유효한 것인지, 아니면 일반적으로 적용 가능한 원리인지를 확인하는 것이 **교차 도메인 전이(cross-domain transfer)**와 **교차 모델 전이(cross-model transfer)**입니다.

교차 도메인 전이는 한 태스크에서 발견된 에이전트 설계가 다른 태스크에서도 효과적인지를 검증합니다. ADAS 논문에서 ARC Challenge(추상 추론)에서 발견된 에이전트를 DROP(독해)과 MGSM(다국어 수학)에 적용한 결과, 모든 태스크에서 기존 수동 설계 대비 성능이 향상되었습니다. DROP에서 CoT의 F1 64.2에서 79.4로, MGSM에서 LLM Debate의 39.0%에서 53.4%로 향상된 것입니다. 이것은 발견된 설계 패턴이 태스크에 종속되지 않는 범용적 원리를 포착했음을 시사합니다.

교차 모델 전이는 특정 LLM으로 발견된 설계가 다른 LLM에서도 효과적인지를 검증합니다. GPT-3.5로 탐색하여 발견된 에이전트를 Claude 모델에 적용한 결과, ARC Challenge에서 CoT의 25.3%에서 48.3%로 약 23퍼센트 포인트가 향상되었습니다. GPT-4를 기반으로 최적화된 에이전트 워크플로우를 다른 LLM으로 전이한 실험에서도 일관된 성능 향상이 관찰되었습니다. 이 결과는 발견된 설계 원리가 특정 모델의 특성에 의존하지 않고, 여러 모델에 일반적으로 적용 가능한 보편적 구조를 반영한다는 것을 보여줍니다.

메타 에이전트와 아카이브 기반 탐색이 실용적으로 흥미로운 점은 탐색 비용입니다. 한 연구에서 25회 반복의 전체 탐색 비용은 약 5달러에 불과했습니다. 입력 약 2,400만 토큰, 출력 약 1,100만 토큰의 비용으로, 인간이 수동으로 설계하는 것보다 훨씬 저렴하면서도 인간 설계를 넘어서는 성능을 달성했습니다. 탐색 완료 후 최적화된 에이전트의 쿼리당 비용은 0.0014달러로, 실제 배포에서의 비용 부담도 미미합니다.

아카이브 기반 탐색의 한계도 인식해야 합니다. 아카이브에 축적된 설계가 특정 방향으로 편향되면 메타 에이전트의 탐색도 그 방향으로 치우칠 수 있습니다. 초기 시드의 선택이 탐색 방향에 큰 영향을 미치므로, 다양한 패턴을 시드로 포함하는 것이 중요합니다. 1.3.1에서 언급한 모듈 수준 탐색에서도 마찬가지로, 경험 풀(experience pool)에 다양한 에이전트 설계와 성능 기록이 축적되어야 재결합 메커니즘이 유망한 모듈 조합을 효과적으로 추론할 수 있습니다. 또한 자동 생성된 에이전트 코드가 의도치 않게 안전하지 않은 행동을 포함할 수 있으므로, 1.4.2에서 다루는 에이전트 안전성 문제와 연결됩니다. 메타 에이전트가 생성한 코드에 대한 자동 안전성 검증이 추가적으로 필요합니다.

정리하면, 메타 에이전트는 다른 에이전트를 설계하고 프로그래밍하는 상위 에이전트이며, 아카이브는 이전 설계의 코드와 성능을 누적 저장하여 탐색을 안내하는 기억 장치입니다. Meta Agent Search는 아카이브 참조, 코드 작성, 자동 오류 수정, 벤치마크 평가, 아카이브 갱신의 반복으로 진행되며, 25회 반복과 약 5달러의 비용으로 인간 설계를 넘어서는 에이전트를 발견할 수 있습니다. 발견된 설계는 교차 도메인 전이와 교차 모델 전이를 통해 범용적 원리임이 검증되었습니다.

다음 단원인 1.3.3에서는 모듈 수준의 탐색에 초점을 맞추어, 표준화된 모듈 인터페이스를 기반으로 한 모듈 진화와 재결합, 성능 예측기, 그리고 프롬프트와 토폴로지를 동시에 최적화하는 기법을 다룹니다.

이 단원을 마치면 메타 에이전트가 아카이브를 참조하며 새로운 에이전트를 코드로 프로그래밍하는 Meta Agent Search 알고리즘의 동작 과정을 단계별로 설명할 수 있습니다.
