# 4.1.2 계획 표현과 의존 그래프

태스크를 분해했다면 다음 과제는 그것을 어떻게 표현하느냐입니다. 분해된 서브태스크들이 단순한 목록으로 나열되면 서로 간의 관계, 즉 어느 태스크가 먼저 완료되어야 다음 태스크를 시작할 수 있는지, 어느 태스크가 병렬로 실행될 수 있는지를 알 수 없습니다. 계획 표현(plan representation)은 이 정보를 구조화된 형식으로 담아내는 방법이며, 그 핵심 도구는 의존 그래프(dependency graph)입니다.

의존 그래프는 서브태스크를 노드로, 의존 관계를 방향성 있는 간선으로 표현한 구조입니다. 태스크 A의 결과가 태스크 B의 입력으로 필요하다면, A에서 B로 향하는 간선이 그려집니다. 이 구조에서 중요한 제약은 그래프가 유향 비순환 그래프(DAG, Directed Acyclic Graph)여야 한다는 것입니다. 순환이 있으면 실행이 무한 루프에 빠지게 되므로, 올바른 계획은 반드시 DAG 형태를 띠어야 합니다. 태스크 A가 B를 필요로 하고, B가 다시 A를 필요로 하는 상황은 설계 오류이며 계획 수립 단계에서 탐지하고 수정해야 합니다.

태스크 구조화 스키마(task structuring schema)는 의존 그래프의 각 노드를 어떤 속성으로 기술할지를 정의합니다. 일반적으로 각 태스크 노드는 고유 식별자, 태스크 유형, 담당 에이전트, 입력 명세, 출력 명세, 그리고 선행 태스크 목록을 포함합니다. 이 스키마를 통일하면 오케스트레이터가 어떤 태스크든 동일한 방식으로 처리할 수 있고, 에이전트 간 인터페이스가 표준화됩니다.

리소스 참조 태그(resource reference tag)는 각 태스크가 필요로 하는 외부 리소스를 명시하는 메커니즘입니다. 파일, 데이터베이스, API 엔드포인트, 이전 태스크의 출력 등이 모두 리소스에 해당합니다. 태스크 정의에 리소스 참조를 명시적으로 태그하면, 오케스트레이터가 해당 리소스가 준비되었는지 확인한 후에 태스크를 실행할 수 있습니다. 또한 동일한 리소스를 여러 태스크가 동시에 변경하려는 충돌 상황을 사전에 감지하는 데도 활용됩니다.

의존 그래프가 완성되면 실행 순서는 위상 정렬(topological sort)을 통해 결정됩니다. 위상 정렬은 DAG에서 모든 간선 방향을 따르는 노드 순서를 찾는 알고리즘입니다. 이를 통해 어떤 태스크도 그것이 의존하는 태스크가 완료되기 전에 실행되지 않음을 보장하는 순서를 얻을 수 있습니다. 위상 정렬의 결과는 유일하지 않을 수 있으며, 여러 합법적인 실행 순서가 존재할 때 그중 병렬화 가능한 태스크를 최대한 동시에 실행하는 전략을 선택하면 전체 실행 시간을 줄일 수 있습니다.

코드 기반 워크플로우 표현(code-based workflow representation)은 의존 그래프를 자연어나 별도의 설정 파일이 아닌 프로그래밍 코드로 표현하는 방식입니다. Python 코드로 태스크 객체를 생성하고, 의존 관계를 함수 호출이나 객체 참조로 표현하면, 워크플로우 자체가 실행 가능한 코드가 됩니다. 이 접근법은 표현력이 높고, 조건 분기나 반복 같은 제어 흐름을 자연스럽게 표현할 수 있으며, 기존의 소프트웨어 개발 도구(버전 관리, 테스트 프레임워크)를 그대로 활용할 수 있다는 장점이 있습니다. LLM이 워크플로우를 직접 코드로 생성하면, 그 코드를 해석기(interpreter)가 실행하는 방식으로 계획과 실행이 연결됩니다.

에이전트 시스템을 계산 그래프(computational graph)로 보는 관점은 딥러닝 프레임워크에서 영감을 받은 것입니다. 딥러닝에서 계산 그래프가 텐서 연산의 순서와 의존 관계를 표현하듯, 에이전트 시스템에서 계산 그래프는 정보 처리 단위의 순서와 데이터 흐름을 표현합니다. 이 관점에서 각 에이전트는 입력을 받아 처리하고 출력을 내보내는 연산 노드가 되고, 에이전트 간 정보 전달은 그래프의 간선으로 표현됩니다. 이 추상화는 시스템의 구조를 시각화하고, 병렬 실행 가능한 부분을 식별하며, 전체 시스템의 최적화를 생각하는 데 유용합니다.

커뮤니케이션 유닛(Communication Unit, CU)은 에이전트 간에 정보를 전달하는 표준화된 메시지 형식입니다. 단순한 텍스트 문자열이 아니라, 태스크 ID, 출력 유형, 구조화된 콘텐츠, 상태 코드 등을 포함하는 구조체입니다. CU를 표준화하면 어떤 에이전트도 다른 에이전트가 생성한 출력을 동일한 방식으로 파싱하고 처리할 수 있습니다. 태스크의 성격에 따라 CU의 형식이 달라질 수 있으며, 코드 실행 결과를 담는 CU와 문서 요약을 담는 CU는 서로 다른 필드 구조를 가질 수 있습니다.

태스크 상태 머신(task state machine)은 각 태스크가 가질 수 있는 상태와 그 전이를 정의합니다. 일반적인 상태로는 대기(pending), 준비(ready), 실행 중(running), 완료(completed), 실패(failed), 재시도 대기(retrying) 등이 있습니다. 태스크는 선행 태스크가 모두 완료되면 대기에서 준비 상태로 전이하고, 에이전트가 할당되면 실행 중이 되며, 에이전트가 결과를 반환하면 완료가 됩니다. 실패 시에는 재시도 정책에 따라 재시도 대기 상태로 이동하거나 오류를 상위에 전파합니다. 상태 머신의 명확한 정의는 오케스트레이터가 전체 시스템 상태를 추적하고, 비정상 상황에 체계적으로 대응하는 기반이 됩니다.

의존 그래프와 태스크 상태 머신이 결합되면 에이전트 시스템의 실행 엔진이 구성됩니다. 오케스트레이터는 의존 그래프를 통해 현재 실행 가능한 태스크(선행 태스크가 모두 완료된 태스크)를 찾고, 그것을 적절한 에이전트에 할당하며, 결과를 수집하여 태스크 상태를 업데이트하고, 새롭게 준비 상태가 된 태스크를 찾아 계속 실행합니다. 이 사이클이 반복되면서 전체 계획이 진행됩니다.

계획 표현의 품질은 시스템의 신뢰성과 디버깅 용이성에 직접적인 영향을 미칩니다. 표현이 모호하거나 의존 관계가 불완전하면, 오케스트레이터가 잘못된 순서로 태스크를 실행하거나 실패 원인을 추적하기 어려워집니다. 반대로 명확하고 풍부한 메타데이터를 가진 계획 표현은 시스템이 예측 가능하게 동작하도록 하고, 문제가 발생했을 때 어느 태스크, 어느 에이전트에서 무엇이 잘못되었는지를 신속하게 파악할 수 있게 합니다.

정리하면, 계획 표현의 핵심은 태스크 간의 의존 관계를 DAG로 구조화하는 것이며, 위상 정렬을 통해 실행 순서를 결정하고 병렬 실행 기회를 최대화합니다. 리소스 참조 태그와 커뮤니케이션 유닛은 에이전트 간 인터페이스를 표준화하고, 태스크 상태 머신은 실행 흐름의 추적과 오류 대응을 체계화합니다. 코드 기반 워크플로우 표현은 계획의 표현력과 실행 가능성을 동시에 높이는 실용적인 접근법입니다.

다음 단원(4.2.1)에서는 이렇게 구조화된 계획을 실제로 실행하는 오케스트레이션 아키텍처, 특히 LLM이 컨트롤러 역할을 맡는 계층적 오케스트레이션 패턴을 살펴봅니다.

이 단원을 학습한 후에는 의존 그래프를 올바르게 설계하고, 위상 정렬의 원리를 설명하며, 태스크 상태 머신을 이용한 실행 흐름 관리를 이해하고 적용할 수 있습니다.
