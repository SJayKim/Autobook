# 2.3.1 메모리 생성과 추출

2.2절에서 메모리를 저장하는 세 가지 아키텍처, 즉 계층적 구조, 그래프 기반 구조, 세분성 선택을 살펴보았습니다. 저장 구조가 "메모리를 어디에 어떤 형태로 보관할 것인가"를 결정한다면, 이 단원에서 다루는 메모리 형성은 "무엇을 메모리로 만들 것인가"를 결정합니다. 에이전트가 환경과 상호작용할 때 모든 정보를 메모리로 저장할 수는 없습니다. 중요한 정보를 식별하고, 구조화된 형태로 변환하여 저장하는 과정이 메모리 생성과 추출입니다.

메모리 형성의 기본 과정을 수식으로 표현하면 M_{t+1} = F(M_t, phi_t)입니다. 여기서 M_t는 시점 t에서의 메모리 상태, phi_t는 시점 t에서 에이전트가 환경과 상호작용하여 얻은 새로운 경험, F는 형성 함수입니다. 새로운 경험 phi_t가 들어올 때 기존 메모리 M_t와 결합하여 갱신된 메모리 M_{t+1}이 만들어집니다. 이 수식이 의미하는 것은, 메모리 형성이 단순한 추가(append) 작업이 아니라 기존 메모리의 맥락을 고려한 통합 작업이라는 것입니다.

메모리 형성의 첫 단계는 **메모리 후보 추출**입니다. 에이전트-환경 상호작용에서 저장할 가치가 있는 정보를 식별하는 과정입니다. 에이전트가 수행하는 모든 행동과 관찰이 메모리가 되는 것은 아닙니다. "안녕하세요"라는 인사에는 장기적으로 보존할 정보가 거의 없지만, "저는 글루텐을 못 먹습니다"라는 발화에는 향후 식사 추천에 영향을 미칠 중요한 사실이 담겨 있습니다. 후보 추출은 이러한 정보적 가치를 판별하는 작업입니다.

후보 추출 후에는 **LLM 기반 자동 메타데이터 생성**이 이루어집니다. 원시 상호작용에서 저장할 정보가 식별되면, 그 정보를 구조화된 형태로 변환해야 합니다. 2.2.2에서 다룬 구조화된 노트 형식을 떠올리면, 각 메모리 항목에는 내용, 타임스탬프, 키워드, 태그, 문맥 설명, 임베딩, 링크가 필요합니다. LLM이 프롬프트를 통해 원시 텍스트에서 이 메타데이터를 자동으로 추출합니다. "사용자가 Python으로 데이터 전처리를 하는 중에 pandas의 merge 함수에서 오류가 발생했다"는 상호작용에서, LLM은 키워드로 "Python, pandas, merge, 오류"를, 태그로 "프로그래밍, 데이터 전처리, 디버깅"을, 문맥 설명으로 "사용자의 데이터 분석 프로젝트에서 발생한 기술적 문제"를 각각 추출합니다.

**경험 구조화**는 태스크 완료 후 전체 경험을 저장 가능한 형태로 변환하는 과정입니다. 에이전트가 하나의 태스크를 완료하면, 그 과정에서 사용한 입력(task description), 생성한 출력(solution), 받은 피드백(success/failure, error messages)을 (입력, 출력, 피드백) 트리플로 구조화합니다. 이 트리플이 2.1.2에서 다룬 경험 메모리(Experiential Memory)의 기본 단위가 됩니다. 사례 기반(Case-based) 경험 메모리로 저장할 때는 전체 트리플을 보존하고, 전략 기반(Strategy-based)으로 저장할 때는 트리플에서 추상적 교훈만 추출합니다.

메모리 생성의 주체가 누구인지도 중요한 설계 결정입니다. **함수 호출을 통한 자기 주도적 메모리 편집(self-directed editing)**에서는 에이전트(LLM)가 스스로 메모리의 생성, 수정, 삭제를 결정합니다. 2.2.1에서 다룬 OS 가상 메모리 비유의 에이전트 시스템이 이 방식을 사용합니다. 에이전트는 대화 중에 "이 정보는 장기적으로 중요하다"고 판단하면 Archival Storage에 저장하는 함수를 호출하고, "이전에 저장한 정보가 더 이상 유효하지 않다"고 판단하면 삭제 함수를 호출합니다. 이 방식의 장점은 에이전트가 맥락을 가장 잘 이해하므로 관련성 높은 정보를 선별할 수 있다는 것이고, 단점은 LLM의 판단이 항상 정확하지 않아 중요한 정보를 유실하거나 불필요한 정보를 과도하게 저장할 수 있다는 것입니다. 이 신뢰성 문제는 2.7.2에서 다시 다룹니다.

**Memory Pressure 경고 메커니즘**도 메모리 생성의 일부입니다. 2.2.1에서 간략히 언급한 이 메커니즘은, 컨텍스트 사용률이 임계값(예: 전체 용량의 70%)에 도달하면 에이전트에게 경고를 보내어 메모리 관리 행동을 유도합니다. 이 경고를 받은 에이전트는 덜 중요한 정보를 External Context로 이동시키거나, 현재 컨텍스트의 내용을 요약하여 압축하는 등의 메모리 관리 작업을 수행합니다. 이것은 OS에서 RAM이 부족할 때 페이지 폴트가 발생하여 OS가 페이징을 수행하는 것과 유사합니다.

**멀티모달 메모리 변환**도 메모리 생성의 확장된 형태입니다. 에이전트가 텍스트뿐 아니라 이미지, 스크린샷, 오디오 등을 처리하는 경우, 이 멀티모달 정보를 메모리로 변환하는 과정이 필요합니다. Vision LLM을 사용하여 스크린샷의 시각 정보를 구조화된 텍스트 설명으로 변환하면, 원본 이미지를 저장하는 것보다 스토리지 사용량을 99.9% 이상 절감하면서도 핵심 정보를 보존할 수 있습니다. 변환된 텍스트는 일반 텍스트 메모리와 동일한 방식으로 저장, 검색, 갱신이 가능합니다.

정리하면, 메모리 형성은 에이전트-환경 상호작용에서 저장할 정보를 식별하고 구조화하는 과정으로, 메모리 후보 추출, LLM 기반 메타데이터 자동 생성, 경험 구조화의 단계를 거칩니다. 에이전트가 함수 호출을 통해 자기 주도적으로 메모리를 편집하는 방식이 널리 사용되며, Memory Pressure 경고 메커니즘이 컨텍스트 용량 관리를 자동화합니다. 멀티모달 정보는 구조화된 텍스트로 변환하여 저장 효율과 검색 가능성을 동시에 확보합니다.

다음 단원인 2.3.2에서는 생성된 메모리 후보 중 실제로 장기 메모리에 저장할 것을 결정하는 입장 제어 메커니즘을 다룹니다.

이 단원을 마치면 에이전트-환경 상호작용에서 메모리 후보를 식별하고 구조화된 메타데이터를 자동 생성하는 과정을 단계별로 설명할 수 있습니다.
