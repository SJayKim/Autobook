# 3.3.2 도구 사용 벤치마크와 신뢰성 평가

언어 모델의 도구 사용 능력을 어떻게 측정할 것인가는 연구와 실제 배포 모두에서 중요한 문제입니다. 일반적인 언어 모델 평가(정확도, BLEU, 퍼플렉시티 등)는 도구 사용 능력을 제대로 측정하지 못합니다. 도구 사용은 자연어 출력의 품질뿐만 아니라, 올바른 도구 선택, 정확한 파라미터 구성, 도구 결과의 적절한 활용, 오류 처리, 그리고 전체 태스크 성공 여부를 포함하는 복합적인 능력이기 때문입니다. 이 단원에서는 도구 사용 평가에 특화된 방법론들과 주요 벤치마크를 살펴봅니다.

**AST 서브트리 매칭**

도구 호출의 정확성을 평가하는 방법 중 하나는 AST(Abstract Syntax Tree) 서브트리 매칭입니다. 도구 호출은 구조화된 형식(함수 이름, 파라미터 이름, 파라미터 값)을 가지므로, 이를 트리 구조로 표현하면 정확성을 체계적으로 평가할 수 있습니다.

예를 들어, 정답 도구 호출이 weather.get(city='Seoul', unit='celsius')라면, 이를 AST로 표현하면 루트 노드가 함수 호출이고, 자식 노드로 함수 이름('weather.get'), 파라미터 쌍들이 있습니다. 모델이 생성한 호출을 같은 방식으로 AST로 변환하고, 두 AST를 비교합니다. 서브트리 매칭은 완전한 일치를 요구하지 않고, 부분적인 일치도 점수로 반영할 수 있습니다. 예를 들어, 함수 이름은 맞지만 파라미터 값이 틀린 경우와 함수 이름 자체가 틀린 경우를 다르게 평가합니다.

AST 서브트리 매칭의 장점은 평가가 결정론적(deterministic)이고 해석 가능하다는 점입니다. 어느 부분에서 오류가 발생했는지를 명확히 파악할 수 있습니다. 단점은 의미론적으로 동등한 호출을 다르게 평가할 수 있다는 점입니다. 예를 들어, 파라미터 순서가 다르지만 기능적으로 동일한 두 호출을 다른 것으로 취급할 수 있습니다.

**환각률과 오류율의 분리 측정**

도구 사용 실패에는 두 가지 다른 종류가 있습니다. 첫 번째는 환각(hallucination)으로, 존재하지 않는 도구를 호출하거나 실제로는 불가능한 파라미터 조합을 사용하는 경우입니다. 두 번째는 오류(error)로, 올바른 도구를 올바른 형식으로 호출했지만 잘못된 태스크에 사용하거나, 결과를 잘못 해석하는 경우입니다.

이 두 가지를 분리하여 측정하는 것이 중요합니다. 환각률(hallucination rate)이 높은 모델은 존재하지 않는 API를 만들어내는 문제를 가지고 있으며, 이는 API 명세 학습의 부족이나 과도한 생성 경향을 시사합니다. 오류율(error rate)이 높은 모델은 API를 알고 있지만 올바르게 적용하지 못하는 문제를 가지고 있으며, 이는 맥락 이해나 추론 능력의 부족을 시사합니다.

환각률은 "모델이 호출한 API 중 실제로 존재하는 API의 비율"로 측정하고, 오류율은 "실제로 존재하는 API를 호출했지만 태스크 목표 달성에 실패한 비율"로 측정합니다. 개선 방향이 서로 다르기 때문에, 이 두 지표를 합산하여 단일 오류율로 보고하는 것은 정보를 손실합니다.

**Pass Rate과 Win Rate**

태스크 수준의 성능을 평가하는 지표로 Pass Rate과 Win Rate이 사용됩니다.

Pass Rate은 모델이 주어진 태스크를 성공적으로 완료한 비율입니다. 특정 기준(정답 도구 호출 완료, 정확한 최종 답변 생성 등)을 만족한 태스크의 수를 전체 태스크 수로 나눈 값입니다. Pass Rate은 절대적인 성능 수준을 측정합니다.

Win Rate은 두 모델을 직접 비교하여 어느 쪽이 더 나은 결과를 냈는지를 측정합니다. 동일한 태스크에 대해 두 모델의 결과를 사람 평가자나 언어 모델 평가자가 비교하여, 어느 쪽이 더 낫거나 둘 다 동일한지를 판단합니다. Win Rate은 두 시스템의 상대적인 성능 차이를 측정하는 데 유용합니다. 새로운 모델이 기존 모델보다 얼마나 나아졌는지를 평가할 때 많이 사용합니다.

두 지표의 한계도 있습니다. Pass Rate은 이분법적 판단(성공/실패)이라 부분적 성공을 반영하지 못합니다. Win Rate은 비교 기준이 되는 모델의 수준에 따라 의미가 달라집니다.

**pass^k 신뢰성 메트릭**

단일 시도의 성공 여부만으로는 모델의 신뢰성을 충분히 평가하기 어렵습니다. 언어 모델은 확률적(stochastic)이기 때문에, 동일한 입력에 대해 매번 다른 결과를 낼 수 있습니다. 한 번은 성공하고 다음 번에는 실패하는 모델은 신뢰하기 어렵습니다.

pass^k 메트릭은 이 신뢰성을 측정합니다. 동일한 태스크를 k번 반복하여, 그 중 최소 하나가 성공하는 비율을 측정합니다. 이를 @k 표기로 나타내기도 합니다. 예를 들어, pass@1은 한 번의 시도로 성공하는 비율이고, pass@5는 5번의 시도 중 최소 하나가 성공하는 비율입니다. pass@1은 일관성(consistency)을 측정하고, pass@k(k > 1)는 잠재 능력(potential capability)을 측정합니다.

두 모델이 pass@1은 동일하더라도 pass@5가 다를 수 있습니다. 이 경우 pass@5가 높은 모델은 정답을 알고 있지만 일관되게 생성하지 못하는 것이고, 두 값이 비슷한 모델은 일관성이 높습니다. 어떤 값이 더 중요한지는 사용 맥락에 따라 다릅니다. 한 번의 실패가 큰 비용을 초래하는 환경에서는 pass@1이 중요하고, 여러 번 시도할 수 있는 환경에서는 pass@5가 더 적합한 지표입니다.

**BFCL 리더보드**

BFCL(Berkeley Function Calling Leaderboard)은 함수 호출(도구 사용) 능력을 평가하는 공개 벤치마크입니다. 다양한 언어 모델들의 도구 사용 능력을 표준화된 방법으로 비교하는 리더보드를 제공합니다.

BFCL은 여러 평가 카테고리를 포함합니다. 단순 함수 호출, 복수 함수 호출, 중첩 함수 호출, 병렬 함수 호출 등 다양한 복잡도의 시나리오를 포함하며, Python, Java, JavaScript, SQL, REST API 등 다양한 언어와 환경을 지원합니다. 또한 관련 없는 도구가 제공된 상황에서 올바른 도구를 선택하는 능력과, 필요한 정보가 없을 때 호출을 거부하는 능력도 평가합니다.

BFCL의 특징은 AST 매칭을 사용하여 자동으로 평가한다는 점입니다. 사람 평가 없이도 수천 개의 테스트 케이스에 대한 평가를 자동으로 수행할 수 있어, 여러 모델의 빠른 비교가 가능합니다. 리더보드에는 정기적으로 새로운 모델이 추가되어, 도구 사용 능력의 발전 추이를 추적할 수 있습니다.

**APIBench**

APIBench는 도구 검색과 도구 사용 능력을 함께 평가하는 벤치마크입니다. 실제 TorchHub, TensorFlow Hub, HuggingFace Hub의 API들을 사용하여, 자연어 설명으로부터 올바른 API를 식별하고 호출하는 능력을 측정합니다.

APIBench의 핵심 평가 방식은 코드 생성 및 실행입니다. 모델이 생성한 API 호출 코드를 실제로 실행하여 올바른 결과를 내는지 확인합니다. 이 방식은 형식적 정확성뿐만 아니라 기능적 정확성도 측정합니다. API 검색 능력을 평가하기 위해, 다양한 검색기(BM25, 신경망 검색기 등)와 언어 모델의 조합을 체계적으로 비교할 수 있는 프레임워크도 제공합니다.

**ToolBench와 ToolEval**

ToolBench는 RapidAPI에서 수집한 실제 REST API를 사용하는 대규모 도구 사용 벤치마크입니다. 16,000개 이상의 API와 49개 카테고리를 포함하며, 단일 도구 사용 태스크부터 여러 도구를 순서대로 사용해야 하는 복잡한 태스크까지 포함합니다. ToolBench의 특징은 현실적인 API 환경을 시뮬레이션한다는 점입니다. API 응답 오류, 타임아웃, 예상치 못한 반환 형식 등 실제 API 사용에서 발생하는 복잡한 상황을 포함합니다.

ToolEval은 ToolBench에 대한 자동 평가 프레임워크입니다. GPT-4를 평가자로 사용하여, 모델의 도구 사용 과정과 최종 답변을 참조 솔루션과 비교합니다. ToolEval은 Pass Rate과 Win Rate을 모두 측정하며, 도구 사용 과정의 효율성(불필요한 도구 호출 수)도 함께 평가합니다.

**tau-bench: 3자 상호작용 평가**

대부분의 도구 사용 벤치마크는 에이전트와 도구의 상호작용만을 평가합니다. 그러나 현실에서 에이전트는 사용자, 도구, 그리고 도메인 정책(예: 환불 정책, 보안 규정) 세 가지 모두와 상호작용합니다. tau-bench(Tool-Agent-User benchmark)는 이 3자 상호작용을 평가합니다.

tau-bench에서는 에이전트가 사용자의 요청을 처리하면서 도구를 사용하되, 도메인 정책도 준수해야 합니다. 예를 들어, 고객 서비스 시나리오에서 사용자가 환불을 요청할 때, 에이전트는 환불 API를 호출하는 것뿐만 아니라 회사의 환불 정책(구매 후 30일 이내, 영수증 필요 등)도 확인해야 합니다.

이 벤치마크는 도구 사용 정확성과 정책 준수(policy compliance)를 동시에 측정합니다. 도구 사용은 완벽하지만 정책을 위반한 경우, 또는 정책은 준수하지만 도구 사용이 잘못된 경우를 각각 다르게 평가합니다. 이는 현실적인 에이전트 배포 환경에서 중요한 능력입니다.

**Zero-shot 일반화 평가**

훈련 데이터에 포함된 API에 대한 성능은 높지만, 새로운 API에 대해서는 성능이 급격히 낮아지는 모델은 실용적이지 않습니다. Zero-shot 일반화 평가는 훈련 중에 보지 못한 새로운 API에 대한 성능을 측정합니다.

Zero-shot 평가는 테스트 시에만 API 명세를 제공하고, 해당 API를 훈련에서 사용하지 않은 상태에서 수행됩니다. 이 평가를 통해 모델이 API 명세를 읽고 이해하는 일반적인 능력을 갖추고 있는지, 아니면 특정 API 패턴을 단순히 암기한 것인지를 구분할 수 있습니다. 현실적인 도구 사용 환경에서는 새로운 API가 지속적으로 등장하므로, Zero-shot 일반화 능력은 모델의 실용적 가치를 결정하는 중요한 요소입니다.

정리하면, 도구 사용 평가는 다차원적인 접근이 필요합니다. AST 서브트리 매칭은 호출 정확성을 구조적으로 측정하고, 환각률과 오류율 분리는 실패 원인을 진단합니다. Pass Rate과 Win Rate은 태스크 수준의 성능을, pass^k는 신뢰성과 일관성을 측정합니다. BFCL, APIBench, ToolBench는 서로 다른 측면을 강조하는 벤치마크로, tau-bench는 현실적인 3자 상호작용을 포함합니다. Zero-shot 일반화 평가는 모델의 범용성을 측정하는 최종 시험대입니다.

다음 단원인 3.4.1에서는 도구 사용의 표준화된 인터페이스로서 MCP(Model Context Protocol)의 아키텍처와 동작 방식을 살펴봅니다. MCP가 왜 등장했고, 클라이언트-서버 구조가 어떻게 작동하며, 라이프사이클 4단계가 무엇인지를 다룹니다.

이 단원을 학습한 후에는 AST 서브트리 매칭의 작동 원리, 환각률과 오류율 분리의 필요성, pass^k 메트릭의 의미, BFCL과 ToolBench의 차이점, tau-bench가 평가하는 3자 상호작용, 그리고 Zero-shot 일반화 평가의 중요성을 설명할 수 있습니다.
