# 4.3.2 비용과 사용량 추적

게이트웨이가 모든 호출을 통과시키는 단일 지점이라는 점은, 비용과 사용량을 통합적으로 추적하기에 완벽한 자리이기도 합니다. 이 단원에서는 MCP 도구 호출과 LLM 토큰 사용을 어떻게 묶어 비용을 추적하고, 어떤 단위로 청구·할당할지를 정리합니다.

먼저 **비용의 구성**을 분해해 봅니다. 한 번의 사용자 요청이 발생시키는 비용은 여러 갈래로 나뉩니다.

```
비용 항목                        예시
모델 토큰                         입력 토큰 + 출력 토큰
도구 호출 비용                    외부 API 호출당 단가
인프라 비용                       게이트웨이·MCP 서버 컴퓨트
저장 비용                         감사 로그·캐시
외부 의존성                       임베딩, 벡터 DB, 검색 API
```

사용자 입장에서는 "한 메시지가 얼마나 들었는가"를 한 줄로 보고 싶고, 운영자 입장에서는 이 다섯 항목의 분해가 필요합니다. 잘 설계된 추적 체계는 두 시각을 모두 충족합니다.

다음은 **Trace 단위 집계**입니다. 한 사용자의 메시지가 모델 호출 → 여러 도구 호출 → 외부 API 호출로 흐를 때, 그 전체를 하나의 단위로 집계해야 합니다. OpenTelemetry의 trace_id가 이 자리에서 핵심 역할을 합니다.

```
한 사용자 메시지 (trace_id = X)
  ├ 모델 호출 (tokens_in=400, tokens_out=120, cost=$0.003)
  ├ tools/call wiki__search (latency=210ms, cost=$0.0001)
  │    └ 외부 API: Algolia (cost=$0.00005)
  ├ tools/call wiki__read_page (latency=80ms)
  ├ 모델 호출 (tokens_in=2300, tokens_out=80, cost=$0.012)
  └ tools/call wiki__create_page (latency=520ms)

요청 총 비용: $0.01515, 총 latency: 4.8s
```

이 집계가 가능하려면 모든 구간에서 trace_id가 일관되게 전파되어야 합니다. 호스트가 모델 호출에 trace_id를 붙이고, MCP 호출에도 같은 trace_id를 헤더로 전달하고, 게이트웨이가 그것을 뒤편 서버에 전파해야 합니다. trace 전파가 한 곳에서 끊기면 비용 합산이 부정확해집니다.

다음은 **Per-tool 지표**입니다. 도구 단위로 다음 지표를 수집해 두면 운영 결정에 큰 도움이 됩니다.

```
지표                                       의미
calls_total                                도구 호출 수
latency_p50, p95, p99                      응답 시간 분포
error_rate                                 isError 비율
tokens_in/out_per_call                     모델 컨텍스트 점유
cost_per_call                              평균 비용
calls_per_user                             상위 사용자
```

이 지표를 매주 검토하면 "어느 도구가 비용 대비 효용이 낮은가", "어느 도구가 지연을 만드는가", "어느 사용자가 한도에 자주 도달하는가" 같은 질문에 데이터로 답할 수 있습니다.

다음은 **청구 모델**입니다. 비용을 누구에게 어떻게 청구할지에 대한 선택입니다.

```
모델                       단위                              예
직접 청구                   사용자별 비용 표시                 SaaS 모델
부서 청구                   부서·팀 단위로 월별 집계           내부 차지백
프로젝트 청구               비용 코드별 태깅                   엔터프라이즈 분석
플랫 요금                   사용량과 무관한 정액               단순 사용자 경험
계단형                     일정량까지 무료, 초과 시 종량      혼합
```

청구 모델을 결정하는 가장 큰 변수는 **누가 자원의 책임자인가**입니다. 회사 내부에서 부서별 책임이 분명하다면 부서 청구가 자연스럽고, 클라이언트가 외부 사용자라면 사용량 기반 청구가 적절합니다.

청구를 정확히 하려면 **태깅**이 결정적입니다. 모든 호출에 다음과 같은 라벨을 함께 둡니다.

```
태그                의미
user_id             직접 사용자
team                팀 식별자
project             프로젝트 코드
cost_center         회계 비용 코드
purpose             prod, dev, experiment
```

이 태그들은 호스트가 호출 시 헤더(예: `X-Project: alpha`)로 전달하거나, 사용자의 토큰 클레임에서 추출됩니다. 게이트웨이가 모든 호출에 일관되게 태그를 붙여 두면, 사후 분석이 깔끔합니다.

다음은 **대시보드와 알림**입니다. 비용 데이터는 정기 보고서만으로는 부족합니다. 비정상 패턴을 빠르게 감지해야 사고를 막을 수 있습니다.

```
알림 유형                                예시 조건
일일 비용 임계 초과                      회사 일일 예산의 80% 도달
사용자 비용 급증                          상위 사용자 비용이 평소의 5배
도구 실패율 급증                          특정 도구 error_rate 10% 초과
도구 latency p95 급증                     평소의 3배
새 도구 사용 폭증                         어제 0회 → 오늘 1만 회
모델 토큰 폭증                            특정 user의 토큰 사용이 평균의 10배
```

알림은 즉시 사고로 직결되지 않더라도 운영자의 주의를 끌어주는 것이 목적입니다. 잘 만든 알림 체계는 사고가 손상으로 번지기 전에 운영자가 개입하게 합니다.

다음은 **예산 통제**입니다. 알림만으로는 한도를 강제할 수 없습니다. 정해진 예산을 넘어가면 차단할 수 있어야 합니다. 이는 4.2.2의 quota와 결합됩니다.

```
계층                예산 한도 예시
사용자              월 100달러
팀                  월 1만 달러
프로젝트             월 5만 달러
회사 전체            월 30만 달러
```

상위 계층의 한도가 우선합니다. 한 사용자가 자기 한도를 다 쓰지 않았더라도 팀 한도에 도달하면 차단됩니다. 이 모델은 비용 폭주를 막는 동시에, 책임 분배를 명확히 합니다.

다음은 **비용 폭주 탐지**의 구체 패턴입니다. 단순한 임계값보다 통계적 이상 탐지가 더 효과적인 경우가 많습니다.

```
탐지 패턴                                  예시
이동 평균 대비 비교                         최근 시간 호출 수 / 지난 4주 평균
표준편차 기반                                z-score > 3
시계열 forecast                              실제 / 예측 > 2.0
잠재 사용자 행동 변화                         새 도구 사용 시작, 시간대 변화
```

이 탐지는 머신러닝 기반 솔루션(예: Anomaly Detector)으로 구현하는 것이 표준이지만, 단순한 규칙으로도 시작할 수 있습니다. 일주일 정도 운영해 보면 어떤 규칙이 자주 알림을 만드는지 빠르게 보이고, 그 결과로 규칙을 다듬어 갈 수 있습니다.

마지막으로 **사용자 가시성**입니다. 사용자가 자기 비용을 보지 못하면 책임감을 갖기 어렵습니다. 다음을 표준으로 합니다.

```
- 사용자 대시보드에 일·주·월 단위 비용 표시
- 도구별 비용 상위 5개 노출
- 비교 그룹 평균 표시(다른 사용자 대비 위치)
- 예산 80% 도달 시 사용자 본인에게 알림
- 비용 절감 팁(예: 비싼 도구의 대안 추천)
```

이 가시성이 갖춰지면 사용자가 자율적으로 비용을 의식하게 되고, 운영자가 통제에만 의존하지 않아도 됩니다.

체크리스트로 마무리합니다.

```
1. 모든 호출에 trace_id, user, team, project 태그
2. Trace 단위 비용 집계
3. Per-tool latency·error·cost 지표
4. 청구 모델에 맞춘 태깅 일관성
5. 일일·월별 예산 한도와 한도 도달 시 차단
6. 비용 알림(임계·이상 탐지)
7. 정기 비용 회고(매주·매월)
8. 사용자 본인 가시성
9. 비용 데이터의 보존과 분석 권한 분리
10. 외부 의존성 비용도 함께 추적
```

정리하면, 비용·사용량 추적은 trace 단위 집계, 도구별 지표, 태그 기반 청구, 계층적 예산, 이상 탐지, 사용자 가시성으로 구성됩니다. 게이트웨이의 단일 지점이라는 특성이 모든 호출을 한 큐브로 묶을 수 있게 해 주며, 이 큐브가 거버넌스의 데이터 기반이 됩니다.

다음 섹션에서는 이 모든 운영을 엔터프라이즈 환경에 통합하기 위한 SSO·IdP 연동과 컴플라이언스 체크리스트를 살펴봅니다.
