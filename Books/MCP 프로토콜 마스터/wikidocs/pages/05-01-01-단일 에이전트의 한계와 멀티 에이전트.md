# 5.1.1 단일 에이전트의 한계와 멀티 에이전트

지금까지 우리는 MCP가 호스트와 도구를 어떻게 연결하는지를 살펴보았습니다. 이 Phase에서는 한 단계 위로 올라갑니다. 한 호스트에 단일 모델 단일 루프 대신, 여러 모델·여러 역할이 협력하는 **멀티 에이전트 시스템**을 다룹니다. 첫 단원에서는 단일 에이전트로 풀기 어려운 문제의 특성을 살피고, 멀티 에이전트가 답이 되는 시점과 한계를 정리합니다.

먼저 **단일 에이전트**가 어떻게 동작하는지 짧게 짚어 봅니다. 단일 에이전트는 한 모델이 사용자의 메시지를 받고, 필요하면 도구를 호출하고, 그 결과를 다시 모델에 넣어 응답을 생성하는 루프입니다. 한 모델이 시작과 끝을 모두 책임집니다. Phase 1~4가 다룬 모든 흐름은 기본적으로 이 단일 에이전트 모델 위에 있습니다.

이 모델이 잘 동작하는 문제는 분명합니다.

```
잘 동작하는 문제
- 한 가지 도메인에 집중된 작업 (코드 작성, 글쓰기)
- 도구 수가 적은 작업
- 컨텍스트가 한 모델 메모리에 들어가는 작업
- 사용자가 결과를 빠르게 받기를 원하는 작업
```

반대로 단일 에이전트가 어려움을 겪는 문제도 분명합니다.

**첫째, 컨텍스트 길이 압박**입니다. 작업이 복잡해질수록 도구 정의, 기존 결과, 사용자 메시지가 모두 컨텍스트에 누적됩니다. 컨텍스트가 100K 토큰을 넘기 시작하면 모델의 정확도가 떨어지고, 비용도 빠르게 늘어납니다. 한 모델이 모든 정보를 한 번에 다루려고 들면 결국 한계에 부딪힙니다.

**둘째, 도구 폭발**입니다. 호스트가 노출하는 도구가 수십 개로 늘면, 모델은 매 단계에서 어느 도구를 호출할지 결정하는 데 더 많은 컨텍스트를 사용합니다. Phase 4의 라우터 패턴이 도구 목록을 좁혀 주지만, 한 작업이 여러 도메인에 걸치면 라우터가 적합한 도구를 한 번에 선택하기 어렵습니다.

**셋째, 역할 혼동**입니다. 같은 모델이 "계획하기"와 "실행하기"와 "검증하기"를 동시에 수행하면, 한 역할이 다른 역할을 압도하는 경향이 나타납니다. 예를 들어 모델이 자신의 결과를 스스로 평가하면 점수가 후하게 매겨지기 쉽습니다. 사람도 자기 글을 자기가 교정하기 어려운 것과 같습니다.

**넷째, 인지 부하 누적**입니다. 모델은 한 응답 안에서 여러 추론 단계를 동시에 처리하면 정확도가 떨어집니다. 작업을 여러 단계로 나누면 각 단계가 단순해지고, 단계 사이의 검증 지점을 둘 수 있습니다.

**다섯째, 시간 제약**입니다. 사용자가 빠른 응답을 기대하는 작업에서, 한 모델이 모든 단계를 순차로 수행하면 지연이 누적됩니다. 단계를 병렬화하면 같은 작업을 짧은 시간에 끝낼 수 있습니다.

이 다섯 한계가 한 가지 결론으로 모입니다. **작업을 여러 역할로 나누고 각 역할에 적합한 모델·도구 묶음을 부여하면 정확도와 효율이 함께 올라간다**는 것입니다. 이것이 **멀티 에이전트**의 출발점입니다.

멀티 에이전트는 단순히 "모델을 여러 번 부르는 것"이 아닙니다. 다음 특성을 함께 가져야 의미가 있습니다.

```
멀티 에이전트의 특성

1. 역할 분리: 에이전트마다 책임이 명확
2. 도구 분리: 각 에이전트에 필요한 도구만 노출
3. 통신 채널: 에이전트 간 메시지 교환 표준
4. 협력 패턴: fan-out·sequential·hierarchical 등
5. 종료 조건: 무한 루프를 피하는 명확한 종료
6. 관측 가능성: 누가 누구에게 무엇을 했는지 추적
```

대표적인 멀티 에이전트 시나리오 몇 가지를 정리합니다.

```
시나리오                          역할 분담
연구 보고서                       조사자 + 작성자 + 검토자
코드 변경                         계획자 + 실행자 + 테스터
고객 응대                         의도 분류기 + 도메인 전문가 + 매니저
데이터 분석                       데이터 수집 + 모델링 + 시각화 + 해석
콘텐츠 검수                       작가 + 사실 검증 + 톤 검증
```

이 시나리오의 공통점은 한 메시지에 대해 단일 모델이 한 번에 좋은 답을 내기 어렵다는 점입니다. 단계를 나누면 각 단계의 책임이 분명해지고, 단계 사이에 사람의 개입(human-in-the-loop)을 끼워 넣기도 쉬워집니다.

여기에 MCP가 결합되는 이유는, **여러 에이전트가 도구를 공유하면서도 각자 적절한 권한을 가져야 한다**는 요구가 있기 때문입니다. 같은 MCP 게이트웨이를 여러 에이전트가 사용하면, 정책·감사·비용 추적이 한 곳에서 일관되게 유지됩니다. 각 에이전트는 자신의 역할에 맞는 scope의 토큰을 받아 호출하고, 게이트웨이는 호출 출처를 에이전트 단위로 식별합니다.

다만, 멀티 에이전트가 만능 해결책은 아닙니다. 부적합한 경우도 분명합니다.

```
멀티 에이전트가 부적합한 경우

- 작업이 한 모델 컨텍스트로 충분히 처리됨
- 응답 지연이 매우 중요한 실시간 시나리오
- 에이전트 간 통신 비용이 작업 자체보다 큼
- 디버깅·관측의 부담을 감수할 인력이 없음
- 사용자 경험이 단순함을 우선
```

특히 **비용·지연 트레이드오프**가 중요합니다. 한 단일 모델 호출이 1만 토큰이고 2초가 걸린다면, 같은 작업을 세 에이전트가 협력해 처리하면 토큰이 3만, 지연이 6초로 늘어날 수 있습니다. 정확도가 올라가지 않는 한 이 비용 증가는 정당화되지 않습니다.

다음은 **단일 모델 vs 멀티 에이전트 결정 기준**입니다.

```
결정 기준                                         단일 에이전트  멀티 에이전트
작업의 도메인 다양성                                 낮음           높음
컨텍스트 크기 요구                                   작음           큼
역할 분리의 명확성                                   불분명         분명
검증·게이트의 필요성                                 낮음           높음
지연 허용 범위                                       짧음           김
비용 허용 범위                                       작음           큼
```

이 결정은 한 번에 끝나지 않습니다. 작은 작업으로 시작했다가 도메인이 늘어나면서 멀티 에이전트로 진화하는 경로도 흔합니다. 반대로, 처음에 멀티 에이전트로 시작했다가 일부 역할을 합쳐 단일 에이전트로 단순화하는 경우도 있습니다. 시스템을 정기적으로 재평가하는 것이 좋은 습관입니다.

마지막으로, 멀티 에이전트는 **MCP의 가치를 가장 잘 드러내는 응용** 영역이기도 합니다. 단일 에이전트라면 굳이 MCP가 아니라 호스트 내부 호출만으로도 도구를 다룰 수 있습니다. 그러나 여러 에이전트가 같은 도구·자원을 공유하면서 각자 다른 권한·정책 아래 동작해야 한다면, MCP의 표준화와 게이트웨이의 단일 지점이 결정적 역할을 합니다.

이 단원으로 멀티 에이전트의 출발점을 잡았습니다. 다음 단원들에서 표준 패턴(5.1.2 fan-out·fan-in, 5.1.3 subgraph)과 프레임워크 통합(5.2), 그리고 운영(5.3)을 살펴봅니다.

정리하면, 단일 에이전트는 컨텍스트·도구·역할·인지 부하·시간 제약 다섯 차원에서 한계에 부딪히며, 작업을 역할로 나누고 각 역할에 적합한 도구를 부여하는 멀티 에이전트가 그 한계를 풀어 줍니다. MCP는 여러 에이전트가 도구를 일관되게 공유하면서 권한과 관측을 표준화하는 인프라가 됩니다.

다음 단원인 5.1.2에서는 멀티 에이전트의 가장 흔한 패턴인 fan-out과 fan-in을 살펴봅니다.
