# 3.1.2 OAuth 2.1 핵심 개념

앞 단원 3.1.1에서 MCP가 OAuth 2.1을 채택한 동기를 살펴보았습니다. 이 단원에서는 **OAuth 2.1**이 OAuth 2.0과 어떻게 다른지, MCP에서 중요한 변경 사항을 정리합니다.

OAuth 2.1은 새 사양이라기보다 **통합 사양**입니다. OAuth 2.0이 2012년 이후 여러 RFC와 BCP(best current practice)로 흩어져 있었던 권고를 한 문서로 모으고, 위험한 흐름을 명시적으로 폐기한 버전입니다. 한 줄로 표현하면 "지금 시점에서 OAuth를 올바르게 쓰는 방법을 강제하는 사양"입니다.

먼저 OAuth 2.0이 정의했던 여러 흐름(grant type)을 빠르게 짚고 가는 것이 좋습니다.

```
2.0 흐름                                       2.1에서의 위상
Authorization Code                              유지 (PKCE 필수화)
Implicit                                        폐기
Resource Owner Password Credentials             폐기
Client Credentials                              유지 (서비스 간 인증)
Device Authorization                            유지 (TV 등 한정)
Refresh Token                                   유지 (회전 권장)
```

**Implicit flow**는 OAuth 2.0이 브라우저 환경의 단순한 흐름으로 정의했지만, 토큰이 URL 단편(fragment)에 노출되는 등 다양한 위험이 누적되어 폐기되었습니다. **Resource Owner Password Credentials**는 사용자의 비밀번호를 클라이언트가 직접 받아 토큰으로 교환하는 흐름이었는데, 사용자 비밀번호 노출 위험과 SSO·MFA 부적합을 이유로 폐기되었습니다.

남은 핵심 흐름은 **Authorization Code with PKCE**입니다. MCP가 거의 모든 경우에 쓰는 흐름이므로, 한 번 짚고 가겠습니다. 사용자는 클라이언트(MCP 호스트)의 인도를 받아 Authorization Server에 로그인하고 동의합니다. AS는 짧은 수명의 **authorization code**를 클라이언트에 발급합니다. 클라이언트는 그 code를 AS의 token 엔드포인트에 제출하여 **access token**(과 선택적으로 **refresh token**)으로 교환합니다.

```
사용자       호스트(클라이언트)        Authorization Server   Resource Server(MCP)
  |               |                          |                       |
  |               |--- (1) authorize 요청 -->|                       |
  |--- 로그인·동의 ----------------------> |                       |
  |               |<-- (2) auth code -------|                       |
  |               |--- (3) code + PKCE ----->|                       |
  |               |<-- (4) access token -----|                       |
  |               |--- (5) MCP 호출 + Bearer access token ---------->|
  |               |<-- (6) 결과 -------------------------------------|
```

**PKCE**(Proof Key for Code Exchange)는 OAuth 2.0 시기에 모바일·SPA 같은 퍼블릭 클라이언트를 위해 만들어진 보호 장치입니다. 클라이언트가 무작위 문자열인 `code_verifier`를 만들고, 그 해시값(`code_challenge`)을 authorize 요청에 첨부합니다. 나중에 토큰 교환 단계에서 `code_verifier`를 함께 보내야 토큰이 발급됩니다. 이 장치 덕분에 누군가 authorization code를 가로채도 verifier를 모르면 토큰으로 바꿀 수 없습니다. OAuth 2.1은 PKCE를 **모든 흐름에서 필수**로 만들었습니다. 자세한 구현은 3.2.3에서 다룹니다.

**Refresh token**은 access token이 만료되었을 때 사용자가 다시 로그인하지 않고 새 access token을 받기 위한 토큰입니다. 보통 access token은 짧은 수명(분 단위)이고 refresh token은 비교적 긴 수명(일 단위)을 가집니다. 2.1은 refresh token의 **회전**(rotation)을 권장합니다. 사용할 때마다 새 refresh token이 발급되고 옛 토큰은 폐기되는 모델입니다. 이는 토큰 탈취 위협을 줄이는 데 효과적입니다. 누군가 옛 refresh token을 가로채 사용하려고 하면, 정상 사용자가 이미 회전된 새 토큰을 사용했기에 옛 토큰이 무효임을 AS가 감지할 수 있습니다.

**Bearer token** 사용에 관한 권고도 2.1에서 강화되었습니다. 토큰은 가능한 한 Authorization 헤더로 보내야 하고, URL 쿼리스트링이나 본문에 넣지 말아야 합니다. URL에 토큰이 실리면 액세스 로그·리퍼러·브라우저 히스토리에 그대로 남기 때문입니다. MCP의 모든 호출은 Streamable HTTP 위에서 `Authorization: Bearer ...` 헤더로 토큰을 운반합니다.

OAuth 2.1은 **client 종류**도 명확하게 구분합니다.

```
client 종류           특징                              MCP에서의 예
confidential          비밀(client secret)을 안전히 보관 사내 자동화 봇, 서버 사이드 호스트
public                비밀을 안전히 보관 못 함          데스크톱 앱, 모바일, SPA
```

Public client는 비밀을 가지지 않는 대신 PKCE로 인가를 보호합니다. MCP 호스트의 다수가 public client이며, 비밀 없이 client_id만으로 동작합니다.

**Scope**는 OAuth가 토큰 단위로 권한을 좁히는 핵심 메커니즘입니다. authorize 요청에 `scope=read:notes write:notes`처럼 권한 목록을 전달하면, AS는 사용자에게 그 scope를 설명한 동의 화면을 보여 줍니다. 발급된 access token은 그 scope만 허용합니다. 자원 서버는 토큰을 검증할 때 scope도 함께 확인해, 도구별로 필요한 권한과 매칭합니다. MCP에서 scope 설계 패턴은 3.4.1에서 다룹니다.

**Audience**(`aud` 클레임)는 토큰이 "어느 자원 서버용"인지를 명시하는 필드입니다. 한 AS가 여러 자원 서버를 인증해 줄 때, 토큰이 잘못된 자원 서버에 사용되지 않도록 막는 보호 장치입니다. 자원 서버는 토큰의 audience가 자신을 가리키는지 확인해야 합니다. 이 검증이 빠지면 다른 서비스의 토큰이 우리 서비스에 받아들여지는 위험이 생깁니다. MCP 사양은 audience 검증을 강하게 권고합니다.

OAuth 2.1의 또 다른 측면은 **DPoP**(Demonstrating Proof-of-Possession) 같은 더 강력한 토큰 결합 방식의 도입입니다. DPoP는 토큰을 사용할 때 클라이언트가 자신이 그 토큰의 소유자임을 증명하는 작은 JWT를 함께 보내는 방식입니다. 토큰만 탈취해도 그 자체로는 쓸 수 없어집니다. MCP 사양은 DPoP를 필수로 만들지는 않지만, 자원 서버가 그것을 요구할 수 있도록 길을 열어 둡니다.

마지막으로, 2.1에서 명시적으로 다루는 **discovery**가 있습니다. 자원 서버와 인증 서버는 각각 자신의 메타데이터를 표준 URL로 노출합니다. 자원 서버는 `/.well-known/oauth-protected-resource`(RFC 9728)에 자신을 보호하는 AS의 위치를 알리고, AS는 `/.well-known/oauth-authorization-server`(RFC 8414)에 자신의 엔드포인트들을 알립니다. 이 두 메타데이터를 따라가면 클라이언트는 어디서 인가를 받아야 하는지, 어디서 토큰을 교환해야 하는지를 사전 지식 없이도 알아낼 수 있습니다. 3.1.3에서 자세히 다룹니다.

정리하면, OAuth 2.1은 위험한 흐름을 폐기하고 안전한 흐름(Authorization Code + PKCE, refresh token 회전, audience 검증, Bearer 헤더, discovery)을 표준으로 굳힌 통합 사양입니다. MCP는 이 사양을 거의 그대로 채택해, 원격 멀티유저 환경의 인증 골격으로 삼습니다.

다음 단원인 3.1.3에서는 MCP에서 자원 서버와 인증 서버를 분리하는 방식, 그리고 메타데이터 발견 흐름을 자세히 살펴봅니다.
