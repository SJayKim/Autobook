# 2.3.1 Local vs Remote 선택 기준

지금까지 두 transport, stdio(2.1.2)와 Streamable HTTP(2.2.2)를 살펴보았습니다. 그렇다면 실제로 새 MCP 서버를 만들 때 어느 쪽을 골라야 할까요. 이 단원에서는 둘을 가르는 기준을 정리하고, 두 가지를 동시에 지원하는 듀얼 모드 설계까지 다룹니다.

선택은 단순한 기술 취향이 아니라 운영 모델의 선택입니다. 다음 다섯 가지 차원으로 결정을 나눠 보는 것이 도움이 됩니다.

```
차원                  stdio가 유리             Streamable HTTP가 유리
1. 위치               로컬 머신                원격 서버, 공유 호스팅
2. 사용자 수          단일 사용자              다중 사용자, 멀티 테넌트
3. 권한 모델          OS 권한으로 충분         OAuth·SSO·세분화된 scope 필요
4. 자원 무게          가볍거나 사용자 머신 자원 무거운 외부 서비스, 캐시 공유 필요
5. 배포 주기          사용자 패키지 업데이트   중앙에서 자주 갱신
```

첫 번째 차원인 **위치**가 가장 기본입니다. 도구가 사용자의 파일·셸·로컬 DB·로컬 IDE 같은 자원에 접근한다면 stdio가 자연스럽습니다. 자식 프로세스가 사용자와 같은 환경에서 실행되므로, 경로·환경 변수·파일 권한이 그대로 활용됩니다. 반대로 도구가 외부 SaaS에 접근한다면 Streamable HTTP가 더 적합합니다. 서버를 한 번 띄워 두고 여러 호스트가 그 한 서비스를 공유합니다.

두 번째 차원은 **사용자 수**입니다. 한 사용자가 자기 컴퓨터에서만 쓰는 도구라면 stdio가 단순합니다. 자기 자신만 그 도구를 호출하므로 격리나 동시성 문제가 적습니다. 회사 안 여러 사용자가 같은 도구를 쓴다면 Streamable HTTP가 자연스럽습니다. 한 서버 인스턴스에 여러 호스트가 붙고, 서버는 사용자 컨텍스트를 OAuth 토큰이나 사용자 식별자로 구분합니다.

세 번째 차원은 **권한 모델**입니다. 사용자가 자기 파일을 자기 도구로 다루는 시나리오는 OS 권한이 이미 충분한 보호막이 됩니다. 사용자가 접근할 수 없는 파일은 자식 프로세스도 접근할 수 없습니다. 반대로 사용자별로 다른 데이터에 접근해야 하는 시나리오에서는 OAuth 2.1과 세분화된 scope가 필요합니다. Phase 3에서 자세히 다룹니다.

네 번째 차원은 **자원 무게**입니다. 도구가 가볍다면 stdio도 충분합니다. 모델·인덱스를 서버가 메모리에 올려두는 무거운 자원이라면 Streamable HTTP가 유리합니다. 여러 호스트가 한 인스턴스를 공유함으로써 자원 비용을 분산할 수 있고, 캐시를 한 곳에 모아 적중률을 높일 수 있습니다.

다섯 번째 차원은 **배포 주기**입니다. 도구가 자주 갱신되지 않는다면 stdio는 패키지 매니저(NPM·PyPI)로 사용자가 직접 업데이트하면 됩니다. 자주 갱신되는 도구라면 중앙 서버로 옮기는 편이 좋습니다. 한 번의 배포로 모든 사용자가 새 버전을 쓰게 됩니다.

이 다섯 차원으로 결정하면 대부분의 도구는 자연스럽게 어느 한쪽으로 기웁니다. 예를 들어 **파일 시스템 도구**는 다섯 차원 모두 stdio에 유리합니다. 사용자의 로컬 파일에 접근해야 하고, 단일 사용자이고, OS 권한으로 충분하고, 자원 무게가 가볍고, 업데이트도 자주 일어나지 않습니다. 반대로 **사내 위키 검색 도구**는 다섯 차원 모두 Streamable HTTP에 유리합니다. 위키는 원격에 있고, 다중 사용자가 쓰고, SSO·scope가 필요하고, 검색 인덱스가 무겁고, 콘텐츠가 자주 갱신됩니다.

조금 더 미묘한 사례도 있습니다. **GitHub 도구**는 두 모델 모두 가능합니다. stdio로 만들어 사용자가 자기 personal access token으로 사용하게 할 수도 있고, Streamable HTTP로 만들어 OAuth 앱으로 인증을 받게 할 수도 있습니다. 두 형태가 모두 존재하는 이유는 환경에 따라 다른 선택이 더 자연스럽기 때문입니다. 개인 개발자는 stdio가 편하고, 기업은 HTTP 서버가 적합합니다.

또 다른 미묘한 사례는 **사용자 머신의 무거운 자원**입니다. 예를 들어 사용자의 로컬 PostgreSQL을 다루는 도구라면 stdio가 자연스럽지만, 그 데이터베이스가 무겁다면 사용자 머신의 자원이 부족할 수 있습니다. 이 경우 stdio 서버가 로컬에서 가벼운 클라이언트 역할만 하고 실제 무거운 작업은 별도 백엔드에서 처리하도록 분리할 수 있습니다. 도구의 호출 표면은 stdio지만, 그 뒤편에서 HTTP API를 호출하는 형태입니다.

**보안 경계**도 결정에 영향을 줍니다. 회사 정책이 모든 외부 호출을 중앙 게이트웨이로 강제한다면, stdio 서버가 사용자 머신에서 직접 외부 API를 호출하는 모델은 정책에 어긋날 수 있습니다. 이 경우 Streamable HTTP 서버를 회사 안에 두고, 그 서버가 외부 API와의 통신을 책임지는 형태가 됩니다. Phase 4에서 다룰 게이트웨이 패턴은 이 모델을 한 단계 더 발전시킵니다.

**오프라인·에어갭** 환경도 고려할 만합니다. 인터넷이 차단된 환경에서는 외부 SaaS 기반 서버를 쓸 수 없으므로, 도구가 stdio로 노출되고 그 안에서 로컬 자원만 다루도록 만들어야 합니다.

이 결정이 영영 고정되는 것은 아닙니다. **듀얼 모드** 서버도 가능합니다. 같은 코드베이스에 두 transport 어댑터를 모두 두고, 어떤 식으로 실행되었느냐에 따라 자동 선택하게 만드는 형태입니다. Python SDK라면 FastMCP가 transport 종류를 인자로 받습니다.

```python
from mcp.server.fastmcp import FastMCP

server = FastMCP("dual-mode")

@server.tool()
def echo(message: str) -> str:
    return message

if __name__ == "__main__":
    import sys
    if "--http" in sys.argv:
        server.run(transport="streamable-http", port=8000)
    else:
        server.run(transport="stdio")
```

같은 도구 정의를 두 transport에서 모두 노출하므로, 사용자가 어떤 환경에서든 같은 능력을 누릴 수 있습니다. 다만 듀얼 모드를 채택하면 인증·세션·관측 같은 부수 요소도 두 transport 모두에 적용해야 한다는 점을 잊지 말아야 합니다. 한쪽만 보안을 강화하고 다른 쪽을 방치하면 약한 쪽이 공격 표면이 됩니다.

선택을 한 줄로 요약하면 다음과 같습니다. "도구가 사용자 자원만 다루고 한 사용자가 자기 머신에서 쓴다면 stdio, 여러 사용자가 외부 자원을 공유한다면 Streamable HTTP." 회색 지역에 놓인 도구라면, 두 모드를 함께 지원하는 편이 미래의 옵션을 늘려 줍니다.

정리하면, transport 선택은 위치·사용자 수·권한 모델·자원 무게·배포 주기라는 다섯 차원으로 정리되며, 양극단의 사례에서는 자연스러운 답이 있고 중간 사례에서는 듀얼 모드 또는 백엔드 분리 같은 절충안이 가능합니다.

다음 단원인 2.3.2에서는 Streamable HTTP 서버를 reverse proxy 뒤에 둘 때 필요한 설정을 자세히 살펴봅니다.
