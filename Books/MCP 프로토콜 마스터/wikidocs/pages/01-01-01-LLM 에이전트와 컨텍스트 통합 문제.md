# 1.1.1 LLM 에이전트와 컨텍스트 통합 문제

대규모 언어 모델(LLM)을 활용한 서비스가 단순한 질의응답을 넘어 실제 업무를 처리하기 시작하면서, 한 가지 분명한 한계가 드러났습니다. 모델은 학습 시점까지의 일반 지식만 가지고 있다는 점입니다. 사용자의 캘린더에 어떤 일정이 있는지, 회사의 위키 문서에 어떤 정책이 적혀 있는지, 지금 이 순간의 주가가 얼마인지 같은 정보는 모델 안에 들어 있지 않습니다. 이 차이를 메우기 위해 등장한 개념이 **에이전트**(agent)입니다. 에이전트는 LLM에 외부 도구를 호출할 권한을 부여하여, 필요한 정보를 그때그때 모델 바깥에서 가져오게 하는 구조를 뜻합니다.

에이전트가 호출하는 외부 도구는 매우 다양합니다. 파일을 읽거나 쓰는 도구, 웹을 검색하는 도구, 데이터베이스를 조회하는 도구, Slack에 메시지를 보내는 도구, GitHub 이슈를 만드는 도구 등이 모두 후보입니다. 도구가 늘어날수록 에이전트가 처리할 수 있는 작업 범위는 넓어지지만, 동시에 통합 비용도 함께 늘어납니다. 도구 하나를 추가하려면 모델이 그 도구의 존재와 사용법을 알아야 하고, 호출 결과를 다시 모델에게 돌려주는 코드가 필요합니다. 그리고 같은 도구라도 사용하는 모델이 GPT인지, Claude인지, Gemini인지에 따라 호출 형식이 조금씩 달라집니다.

이 통합 부담은 빠르게 커집니다. 모델이 N개, 도구가 M개라면, 모든 조합을 지원하기 위해 N×M개의 어댑터가 필요합니다. 2024년 초만 해도 도구 통합 코드를 직접 작성하는 것이 일반적이었습니다. OpenAI는 OpenAI대로, Anthropic은 Anthropic대로, 도구를 노출하는 방식이 달랐고, 각 도구 제공자도 자신만의 라이브러리를 만들어 배포했습니다. 그 결과 비슷한 코드가 수백 개 리포지토리에 흩어지고, 도구 하나의 사양이 바뀌면 모든 어댑터를 함께 손봐야 하는 상황이 반복되었습니다.

또 하나의 문제는 동적인 자원에 있습니다. 도구가 단순한 함수라면 미리 정의해 두면 그만이지만, 사용자의 파일이나 데이터베이스 같은 자원은 실행 중에 목록이 바뀝니다. 사용자가 새 폴더를 만들면 그 폴더가 도구의 일부가 되어야 하고, 데이터베이스에 새 테이블이 추가되면 그 테이블도 노출되어야 합니다. 기존의 함수 호출 방식은 이런 동적 변화를 잘 표현하지 못합니다. 모델에게 매번 전체 목록을 다시 전달해야 하고, 목록이 길어지면 컨텍스트가 빠르게 소진됩니다.

여기에 보안과 동의의 문제가 더해집니다. 에이전트가 사용자의 파일을 읽거나 메시지를 보낼 수 있다는 것은, 잘못 호출되면 큰 피해가 발생할 수 있다는 뜻이기도 합니다. 사용자는 자신이 어떤 도구를 허용했는지, 어떤 자원에 접근이 가능한지 알 권리가 있습니다. 도구 호출 흐름이 표준화되어 있지 않으면 이런 동의 절차도 도구마다 따로 만들어야 합니다.

**Model Context Protocol**(MCP)은 바로 이 문제를 풀기 위해 등장했습니다. MCP는 LLM과 외부 도구·자원·프롬프트를 연결하는 표준 프로토콜입니다. 표준이 있으면 N×M의 조합 폭발이 N+M의 합으로 줄어듭니다. 도구 제공자는 MCP 서버를 한 번 만들면 되고, 모델 호스트는 MCP 클라이언트를 한 번 구현하면 됩니다. 새로운 도구가 추가되어도 모델 측 코드는 그대로입니다. 새로운 모델이 등장해도 도구 측 코드는 그대로입니다.

MCP가 자주 비유되는 표현이 USB-C입니다. USB-C 이전에는 노트북·키보드·외장 디스크·휴대전화가 각자 다른 커넥터를 썼고, 사용자는 기기마다 다른 케이블을 갖춰야 했습니다. USB-C가 자리 잡은 뒤로는 케이블 하나로 다양한 기기를 연결할 수 있게 되었습니다. MCP는 LLM 생태계에서 같은 역할을 한다는 의미입니다. 이 비유는 직관적이지만 완벽하지는 않습니다. USB-C가 전기 신호의 물리적 규약이라면, MCP는 의미 있는 데이터의 교환 규약이라는 점에서 추상화 계층이 더 위에 있습니다. 또한 MCP는 도구뿐 아니라 자원, 프롬프트, 심지어 모델로부터 클라이언트로 향하는 역방향 호출까지 포괄한다는 점에서 단순한 케이블 비유보다 넓은 범위를 다룹니다.

이 통합 문제의 핵심을 단순한 그림으로 정리하면 다음과 같습니다.

```
표준 없는 세계 (N×M 통합)        표준이 있는 세계 (N+M 통합)

  모델 A ─┬─ 도구 1                  모델 A ─┐
        ├─ 도구 2                  모델 B ─┼── MCP ─── 도구 1
        ├─ 도구 3                  모델 C ─┘             ─ 도구 2
        └─ 도구 4                                       ─ 도구 3
  모델 B ─┬─ 도구 1                                       ─ 도구 4
        ├─ 도구 2
        ...
```

정리하면, LLM 에이전트가 외부와 만나는 지점에서 발생하는 통합 비용, 동적 자원 표현의 어려움, 동의·보안 절차의 파편화가 표준 프로토콜에 대한 요구를 만들었습니다. MCP는 이 요구를 정면으로 다루기 위해 설계된 프로토콜이며, 우리는 이 책 전체에 걸쳐 MCP가 어떤 약속으로 이 문제들을 해결하는지를 단계적으로 살펴봅니다.

다음 단원인 1.1.2에서는 MCP가 정확히 무엇이며 누가 어떤 목적으로 만들었는지를 다룹니다.
