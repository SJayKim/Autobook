# 4.3.1 감사 로그 설계

정책과 한도와 필터가 갖춰지면, 다음 질문은 "그 결정이 어떻게 기록되어 사후에 검증·재현 가능한가"입니다. 답이 **감사 로그**입니다. 이 단원에서는 게이트웨이가 남기는 감사 로그를 어떤 필드로, 어떻게 저장하고 보존할지 정리합니다.

먼저 감사 로그가 답해야 할 질문들을 명확히 합니다.

```
질문                                       답이 있어야 함
누가 호출했는가                              user_id, client_id
무엇을 호출했는가                            tool_name, server_id, params
언제 호출했는가                              timestamp(UTC, ms 정밀도)
어디서 호출했는가                            client_ip, user_agent
왜 허용·차단되었는가                         policy_decision, policy_id
결과가 무엇이었는가                          response_summary, isError
얼마나 비용이 들었는가                       latency, tokens, cost
어떤 trace에 속하는가                        trace_id, span_id
```

이 질문들에 답하려면 표준 스키마가 필요합니다. JSON 한 줄로 다음 같은 구조가 일반적입니다.

```json
{
  "timestamp": "2026-05-14T03:21:45.812Z",
  "trace_id": "0af7651916cd43dd8448eb211c80319c",
  "span_id": "b7ad6b7169203331",
  "actor": {
    "user_id": "u_alice",
    "tenant_id": "t_acme",
    "client_id": "5f8e1c...",
    "ip": "203.0.113.42",
    "user_agent": "ClaudeDesktop/1.4.0"
  },
  "request": {
    "method": "tools/call",
    "tool": "wiki__create_page",
    "server": "wiki",
    "session_id": "9c1f...",
    "params_summary": {"title_len": 42, "body_len": 8231},
    "params_hash": "sha256:abc..."
  },
  "decision": {
    "policy": "allow",
    "rules_evaluated": ["allow-wiki-write", "deny-deleted-pages"],
    "scope_required": ["write:notes"],
    "scope_granted": ["write:notes"]
  },
  "result": {
    "is_error": false,
    "status_code": 200,
    "latency_ms": 1432,
    "tokens_in": 0,
    "tokens_out": 0,
    "response_summary": {"page_id": "p_4471", "version": 1}
  }
}
```

이 스키마에서 눈여겨볼 부분 몇 가지를 짚어 둡니다.

**`actor`**는 호출 주체를 가능한 한 풍부히 식별합니다. 사용자 id에 더해 테넌트, client_id, IP, user-agent까지 함께 둡니다. 사고 조사 시 한 차원만으로는 부족한 경우가 많습니다. 예를 들어 어떤 사용자의 토큰이 평소와 다른 IP에서 갑자기 쓰이고 있다면, 그 패턴은 IP 정보가 있어야 보입니다.

**`params_summary`와 `params_hash`**는 4.2.3에서 다룬 민감 데이터 처리와 연결됩니다. 원본 인자를 저장하면 PII 누설 위험이 있고, 아예 안 남기면 사후 조사가 불가능합니다. 절충은 통계적 요약(길이, 키 수)과 해시를 남기는 것입니다. 해시는 같은 호출이 반복되는지 비교할 때 유용합니다.

**`decision`**은 정책 평가의 결과를 기록합니다. 어떤 규칙이 통과되고 어떤 규칙이 적용되었는지를 알면, 정책 변경이 호출에 어떤 영향을 줄지 사후 분석이 가능합니다.

**`result`**는 결과를 요약합니다. 응답 본문 전체를 남기지 않는 대신, 모델이나 사용자에게 가는 핵심 식별자(예: 생성된 문서 id)와 메타데이터를 보관합니다. 본문 전체가 필요한 경우에는 별도 저장소(예: S3)에 저장하고 그 참조 키만 로그에 남깁니다.

다음은 **append-only**의 의미입니다. 감사 로그는 한 번 기록된 뒤 누구도 수정·삭제할 수 없어야 합니다. 그렇지 않으면 사고 시 증거가 훼손될 수 있습니다. 구현 방법은 다음과 같습니다.

```
1. 쓰기 전용 권한 - 로그 저장소에 delete·update 권한을 부여하지 않음
2. 해시 체인 - 각 로그가 이전 로그의 해시를 포함해 변조 감지
3. WORM 저장소 - Write-Once Read-Many 저장소 사용
4. 시간 보관 - 외부 시간 동기화(NTP) + 시간 서명
5. 분리 권한 - 운영팀과 보안팀이 다른 권한으로 분리
```

해시 체인은 비교적 구현이 단순하면서 효과가 큽니다. 각 로그 엔트리가 직전 엔트리의 SHA-256을 가지고 있으면, 중간 한 줄을 바꾸려면 그 이후 모든 줄을 다시 계산해야 합니다. 그 흔적이 외부 감사자에 의해 검출됩니다.

**보존 기간**은 규제와 운영 요구의 절충으로 결정됩니다.

```
보존 기간 결정 요소               예시 값
규제 최소 보존                    개인정보 1~3년, 결제 5~7년
사고 조사 회고                    90일~1년
운영 분석                         90일~6개월
저장 비용                          비용/보존 트레이드오프
```

흔한 패턴은 **3계층 보존**입니다. 최근 30일은 hot 저장소(빠른 쿼리), 1년까지 warm 저장소(분석 용도), 그 이후는 cold 저장소(규제 보존만)에 둡니다. cold 저장소로 옮길 때 한 번 더 압축·암호화를 강화합니다.

다음은 **검색 가능성**입니다. 감사 로그가 사후 조사에 쓸모 있으려면 빠르게 검색할 수 있어야 합니다. 다음 인덱스 패턴이 효과적입니다.

```
인덱스 키                       사용 시나리오
user_id + timestamp             특정 사용자의 활동 추적
tool_name + timestamp           특정 도구의 사용 패턴
trace_id                        한 사용자 요청의 전체 흐름 재구성
policy_decision                 정책 변경의 영향 분석
ip                              비정상 IP의 활동 추적
```

OpenTelemetry 트레이스(2.3.3에서 언급)와 결합하면 더 강력합니다. 한 사용자 메시지가 호스트 → 게이트웨이 → 여러 MCP 서버 → 외부 API를 거치는 전체 경로를 같은 trace_id로 묶을 수 있습니다. 사고 시 "이 사용자의 어떤 메시지가 어떤 외부 호출을 트리거했는가"를 한 번에 보여 줍니다.

다음은 **재현 가능성**입니다. 사고 조사 시 같은 상황을 재현해야 할 수 있습니다. 다음 정보가 있으면 재현이 쉬워집니다.

```
재현용 정보                              저장 방식
정책 버전                                 정책 변경 로그 연동
모델 버전                                 호스트 로그
도구 정의 스냅샷                          도구 호출 시 등록
사용자 입력(요약)                         params_summary 보강
```

특히 **정책 버전**은 자주 빠지는 항목입니다. 그 시점에 어떤 정책이 적용되었는지를 알지 못하면 같은 호출을 지금 다시 실행해도 결과가 다를 수 있습니다. 정책 변경의 timestamp와 함께 정책 ID를 로그에 함께 두는 것이 좋습니다.

**감사 로그 자체의 보안**도 잊지 말아야 합니다.

```
1. 접근 통제 - 누가 로그를 볼 수 있는가
2. 접근 감사 - 로그를 본 사람도 기록(감사 위에 감사)
3. 민감 마스킹 - 토큰·비밀번호 같은 데이터는 절대 평문 금지
4. 데이터 거주성 - 규제 요구가 있는 경우 특정 지역에 저장
5. 익명화 옵션 - 분석 용도로는 PII를 제거한 사본 사용
```

마지막으로, 감사 로그는 단순히 저장만 한다고 쓸모가 생기지 않습니다. **정기적인 활용 의식**이 살아 있어야 합니다.

```
의식                                          주기
이상 행동 감지 알림(spike, 비정상 IP 등)        실시간
정책 위반 사건 리뷰                            매주
사용자 활동 통계 보고                          매월
모의 사고 조사 시뮬레이션                      분기
규제 보고서 자동 생성                          분기/반기
```

이 의식들이 살아 있어야 감사 로그가 "그냥 쌓여 있는 데이터"가 아니라 거버넌스의 실제 동력이 됩니다.

체크리스트로 마무리합니다.

```
1. 표준 스키마와 일관된 필드
2. 원본 대신 요약·해시·참조 사용
3. Append-only와 해시 체인 또는 WORM
4. 3계층 보존(hot·warm·cold)
5. 빠른 검색을 위한 인덱스
6. trace 연동
7. 정책 버전·도구 스냅샷 보존
8. 접근 통제와 접근 감사
9. 정기 활용 의식
10. 규제 변경 시 자동 보고서 갱신
```

정리하면, 감사 로그는 호출의 정체·시점·결정·결과를 표준 스키마로 기록하고, append-only·인덱스·trace 연동·정기 활용 의식을 통해 사고 조사·규제 보고·정책 갱신의 출발점이 됩니다. 게이트웨이가 단일 지점인 점이 이 작업을 가능하게 합니다.

다음 단원인 4.3.2에서는 같은 데이터로부터 비용·사용량을 추적하는 방법을 살펴봅니다.
