# 1.1.2 MCP란 무엇인가

앞 단원 1.1.1에서 LLM 에이전트의 통합 비용 문제가 표준 프로토콜에 대한 요구로 이어진다는 점을 보았습니다. 이 단원에서는 그 답으로 등장한 MCP의 정체를 정리합니다.

**Model Context Protocol**(MCP)은 LLM 애플리케이션이 외부 데이터와 도구에 접근하는 방식을 표준화한 오픈 프로토콜입니다. 2024년 11월 25일 Anthropic이 공식 사양과 참조 구현을 공개하면서 등장했습니다. 처음부터 단일 회사의 제품이 아니라 모두에게 열린 사양으로 설계되었고, 이후 OpenAI, Google, Microsoft를 포함한 다양한 회사가 자신의 제품에 MCP를 통합했습니다. 사양은 GitHub의 modelcontextprotocol 조직에서 관리되며 누구나 PR을 통해 기여할 수 있습니다.

MCP는 세 가지 주체를 가정합니다. **Host**는 사용자가 직접 마주하는 AI 애플리케이션입니다. Claude Desktop, Cursor, VS Code의 Copilot Chat 같은 도구가 호스트에 해당합니다. **Client**는 호스트 내부에서 하나의 외부 연결을 담당하는 모듈입니다. 호스트가 외부와 새 연결을 맺을 때마다 클라이언트 인스턴스가 하나씩 생성됩니다. **Server**는 도구·자원·프롬프트를 노출하는 별도의 프로세스이거나 원격 서비스입니다. 이 셋의 관계는 1.3.1에서 더 자세히 다룹니다.

MCP의 가치를 한 줄로 표현하면, 호스트가 어떤 서버와 연결되어 있는지 몰라도 호스트의 코드는 동일하게 동작한다는 점입니다. 호스트가 새로운 서버에 연결되면, 서버는 자신이 제공하는 도구와 자원의 목록을 알려 줍니다. 호스트는 그 목록을 모델에게 전달하고, 모델이 어떤 도구를 호출하기로 결정하면 호스트는 그 호출을 다시 서버에 위임합니다. 이 흐름은 어떤 서버에 대해서도 똑같이 적용됩니다. 새 서버를 추가한다고 해서 호스트 코드를 손볼 필요가 없습니다.

MCP는 표준화하는 것과 표준화하지 않는 것이 분명히 구분되어 있습니다. 표준화하는 것은 메시지 형식, 라이프사이클, 프리미티브의 정의, 인증의 골격입니다. 반대로 어떤 LLM을 사용할지, 모델에게 어떤 시스템 프롬프트를 줄지, 도구 호출을 자동으로 승인할지 사용자에게 묻고 처리할지는 모두 호스트의 자유입니다. 또한 서버 내부에서 도구를 어떻게 구현할지, 어떤 라이브러리를 쓸지도 표준이 정하지 않습니다. 이런 경계 덕분에 MCP는 가벼우면서도 다양한 환경에 적용될 수 있습니다.

사양은 짧은 기간 안에 빠르게 진화했습니다. 첫 공개판인 **2024-11-05** 사양은 stdio와 HTTP+SSE 두 가지 transport, 그리고 세 가지 프리미티브(Tools, Resources, Prompts)와 Sampling을 정의했습니다. 두 번째 사양인 **2025-03-26**은 큰 폭의 갱신을 담았습니다. 새로운 transport인 **Streamable HTTP**가 도입되었고, OAuth 2.1 기반 인증이 정식 채택되었습니다. 세 번째 사양인 **2025-06-18**은 elicitation(서버가 사용자에게 추가 정보를 요청하는 흐름), structured tool output, resource indicator 등을 추가하면서 보안과 표현력을 강화했습니다. 사양 버전은 초기화 시점에 클라이언트와 서버가 협상하므로, 한쪽이 새 버전을 모른다고 해서 통신이 불가능해지는 일은 없습니다.

오해를 줄이기 위해 MCP가 무엇이 아닌지도 짚어 두는 것이 좋습니다. MCP는 모델 자체가 아닙니다. 모델을 학습시키거나 추론을 빠르게 만들어 주지 않습니다. MCP는 에이전트 프레임워크도 아닙니다. 에이전트의 추론 루프나 메모리, 멀티 에이전트 패턴은 LangGraph·AutoGen·CrewAI 같은 별도의 프레임워크가 담당합니다. MCP는 그 프레임워크 안에서 도구를 연결하는 한 계층일 뿐입니다. 마지막으로, MCP는 보안 제품이 아닙니다. 인증의 골격을 정해 두기는 했지만, 인가 정책이나 감사 로그 같은 운영 요소는 호스트나 게이트웨이가 별도로 구현해야 합니다.

이 책 전체를 관통하는 한 가지 질문은 다음과 같습니다. "표준 프로토콜이 도입되면 실제 운영에서 무엇이 달라지는가." 이후 단원에서는 프리미티브의 동작, transport의 진화, 보안 모델의 강화, 게이트웨이의 등장, 멀티 에이전트와의 결합 순서로 이 질문을 펼쳐 갑니다.

정리하면, MCP는 LLM 호스트와 외부 도구·자원·프롬프트를 연결하는 오픈 표준이며, 메시지 형식과 라이프사이클을 통일함으로써 통합 비용을 줄이고 동의·관측 절차를 일관되게 만들고자 합니다.

다음 단원인 1.1.3에서는 MCP가 기존의 Function calling이나 OpenAPI 같은 다른 통합 방식과 어떻게 다른지를 비교합니다.
