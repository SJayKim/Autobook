# 2.2.1 Streamable HTTP 등장 배경

앞 단원 2.1.3에서 HTTP+SSE transport의 다섯 가지 한계를 살펴보았습니다. 이 단원에서는 그 한계를 정면으로 다루기 위해 2025-03-26 사양에 채택된 **Streamable HTTP**가 어떤 의도로 설계되었는지를 정리합니다.

새 transport의 출발점은 한 가지 질문이었습니다. "HTTP+SSE의 두 엔드포인트를 굳이 분리할 필요가 있는가?" 호스트가 메시지를 보낼 곳도, 서버가 메시지를 흘려보낼 곳도 결국 같은 서버라면, 두 엔드포인트를 하나로 합치는 편이 자연스럽습니다. 단일 엔드포인트가 가능하다면, 인증·관측·정책 적용 지점도 단순해집니다.

또 다른 출발점은 **응답 모드의 유연성**이었습니다. 모든 호출이 스트리밍이 필요한 것은 아닙니다. 단순한 함수 호출은 즉시 응답으로 끝내면 되고, 오래 걸리는 호출이나 progress 이벤트가 필요한 경우에만 스트리밍이 필요합니다. HTTP+SSE는 모든 경우를 SSE 채널 위로 흘렸지만, 새 transport는 호출마다 응답 형태를 다르게 가져갈 수 있어야 한다는 방향으로 설계되었습니다.

세 번째 출발점은 **서버리스 친화성**이었습니다. 짧은 요청·응답 한 건이라면 굳이 장수명 연결을 유지하지 않아야 합니다. 그래야 Cloudflare Workers, AWS Lambda, Vercel Functions 같은 환경에서 MCP 서버를 가볍게 띄울 수 있습니다. 또한 서버리스가 아닌 일반 서버에서도, 필요할 때만 SSE 스트림을 열고 닫을 수 있다면 자원을 절약할 수 있습니다.

이 세 요구를 동시에 풀어 낸 결과가 다음 한 줄로 요약됩니다.

> 단일 엔드포인트에서, 클라이언트의 Accept 헤더에 따라 즉시 응답 또는 SSE 스트림을 선택한다.

같은 URL에 POST를 보내면 서버는 두 가지 응답 중 하나를 고를 수 있습니다. 첫째는 일반 HTTP 응답입니다. 본문에 JSON-RPC 응답을 한 객체 담아 즉시 돌려줍니다. 둘째는 SSE 스트림입니다. 응답의 Content-Type을 `text/event-stream`으로 두고, 응답 본문을 SSE 이벤트로 채워 흘려보냅니다. 어느 모드를 쓸지는 클라이언트가 Accept 헤더로 두 가지를 모두 수용한다고 선언한 뒤, 서버가 그때그때 적절한 모드를 고르는 형태입니다.

여기에 또 하나의 흐름이 추가됩니다. 같은 엔드포인트에 GET을 보내면 서버는 SSE 스트림을 시작합니다. 이 스트림은 호스트의 특정 요청에 대한 응답이 아니라, 서버가 자발적으로 보내고 싶은 알림이나 sampling 요청을 흘려보내는 채널입니다. HTTP+SSE에서의 SSE 엔드포인트와 비슷한 역할이지만, 같은 URL을 쓴다는 점이 다릅니다.

이 설계를 단순한 그림으로 정리하면 다음과 같습니다.

```
                 단일 엔드포인트: /mcp

호스트 → 서버:
  POST /mcp + JSON-RPC 요청
    → 서버는 (a) 일반 응답
              (b) SSE 응답으로 결과+진행률 흘림  중 하나 선택

서버 → 호스트:
  GET /mcp (클라이언트가 시작한 streaming 채널)
    → 서버가 자발적 알림·sampling을 SSE로 흘림
```

마이그레이션 측면에서 새 transport는 후방 호환을 충실히 고려했습니다. 사양은 한동안 HTTP+SSE를 함께 지원하도록 권고했고, 많은 서버가 두 transport를 동시에 노출했습니다. 호스트도 protocol version 협상 결과에 따라 어느 transport로 통신할지를 결정합니다. 새 호스트와 옛 서버의 조합, 옛 호스트와 새 서버의 조합 모두 가능하도록 만든 결과, 생태계 전환이 비교적 매끄럽게 이루어졌습니다.

논의는 GitHub의 modelcontextprotocol/specification 리포지토리에서 공개적으로 진행되었습니다. Discussion #206에서 시작된 대화는 RFC 형태로 정리된 PR로 이어졌고, 여러 SDK 메인테이너와 호스트 개발자의 피드백이 누적되면서 사양에 반영되었습니다. 이 과정은 표준이 단일 회사의 결정이 아니라 커뮤니티 합의로 진화한다는 사양의 운영 원칙을 잘 보여 줍니다.

새 transport는 **세션** 개념도 명시적으로 다룹니다. 단일 엔드포인트 위에서 여러 요청을 같은 세션으로 묶기 위해, 서버는 첫 응답에서 `Mcp-Session-Id` 헤더를 발급합니다. 클라이언트는 이후 모든 요청에 이 헤더를 함께 보냅니다. 세션 식별자가 있으면 서버는 그 세션의 상태(예: 구독 정보, 진행 중인 호출)를 유지할 수 있고, 재연결 시에도 같은 세션을 이어 갈 수 있습니다. 자세한 동작은 2.2.2와 2.2.3에서 다룹니다.

도입 효과를 짧게 짚어 두면 다음과 같습니다. 첫째, **서버리스 환경 친화성**이 크게 좋아졌습니다. 단순 요청은 한 번의 HTTP 트랜잭션으로 끝나므로 Cold start나 짧은 함수 실행 모델에 잘 맞습니다. 둘째, **로드밸런서 호환성**이 개선되었습니다. 세션 식별자를 헤더로 노출함으로써, 로드밸런서가 session-aware하게 라우팅할 수 있는 표준 기반이 생겼습니다. 셋째, **운영 단순화**입니다. 인증·관측·rate limit을 한 URL에 적용하면 되므로 보안 정책이 일관됩니다.

물론 모든 문제가 해결된 것은 아닙니다. 후속 사양에서는 elicitation, structured tool output, resource indicator 같은 새 기능이 추가되었고, transport 자체도 여전히 다듬어지고 있습니다. 그러나 핵심 골격, 즉 단일 엔드포인트·응답 모드 선택·세션 헤더 도입은 안정되었고, 대부분의 새 호스트와 서버가 이 형태를 기본으로 채택합니다.

정리하면, Streamable HTTP는 HTTP+SSE의 한계를 정면으로 다루기 위해 등장한 transport이며, 단일 엔드포인트에서 응답 모드를 동적으로 고르고 세션 식별자를 헤더로 노출함으로써 서버리스·로드밸런서·운영 표준 모두에 친화적인 모양을 만들어 냈습니다.

다음 단원인 2.2.2에서는 이 단일 엔드포인트가 실제로 어떻게 작동하는지, POST·GET·Accept 헤더의 협업을 자세히 살펴봅니다.
