# 3.2.2 MCP에서의 DCR 흐름

앞 단원 3.2.1에서 RFC 7591이 정의하는 동적 등록의 골격을 살펴보았습니다. 이 단원에서는 그 골격을 MCP의 실제 시나리오에 끼워 맞춰, 호스트가 처음 본 MCP 서버에 어떻게 합류하는지를 처음부터 끝까지 추적합니다.

배경 시나리오는 다음과 같습니다. 사용자가 Claude Desktop에 새 MCP 서버 URL(`https://mcp.example.com`)을 입력합니다. 호스트는 이 서버에 대해 사전 지식이 없습니다. AS가 어디인지도, client_id가 무엇인지도 모릅니다. 다음 일곱 단계가 자동으로 이어집니다.

```
1. 호스트가 MCP 호출을 시도            → 401 + WWW-Authenticate
2. 호스트가 protected resource metadata 조회
3. 호스트가 authorization server metadata 조회
4. 호스트가 registration 엔드포인트에 자기 메타데이터 POST
5. 사용자에게 authorize 흐름 인도
6. 호스트가 token 엔드포인트에서 코드 교환
7. 호스트가 access token으로 MCP 호출 성공
```

각 단계를 와이어 수준에서 따라가 보겠습니다.

**1단계: 인증 없이 시도**. 호스트는 일단 토큰 없이 initialize 요청을 보냅니다. 자원 서버는 401과 WWW-Authenticate 헤더를 돌려줍니다.

```
POST /mcp HTTP/1.1
Host: mcp.example.com
Content-Type: application/json
Accept: application/json, text/event-stream

{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer resource="https://mcp.example.com",
                  resource_metadata="https://mcp.example.com/.well-known/oauth-protected-resource"
```

이 응답이 곧 호스트에게 "여기서 메타데이터를 시작하라"는 신호입니다.

**2단계: 자원 메타데이터 조회**. 호스트는 헤더에 적힌 URL로 GET을 보냅니다.

```
GET /.well-known/oauth-protected-resource HTTP/1.1
Host: mcp.example.com

HTTP/1.1 200 OK
Content-Type: application/json

{
  "resource": "https://mcp.example.com",
  "authorization_servers": ["https://auth.example.com"],
  "scopes_supported": ["read:notes", "write:notes"]
}
```

호스트는 첫 authorization server URL을 골라 다음 단계로 갑니다. 여러 AS가 나열되어 있다면 호스트가 정책에 따라 하나를 고릅니다(예: 사용자의 IdP와 일치하는 것).

**3단계: AS 메타데이터 조회**. AS의 well-known 경로를 읽습니다.

```
GET /.well-known/oauth-authorization-server HTTP/1.1
Host: auth.example.com

HTTP/1.1 200 OK
Content-Type: application/json

{
  "issuer": "https://auth.example.com",
  "authorization_endpoint": "https://auth.example.com/authorize",
  "token_endpoint": "https://auth.example.com/token",
  "registration_endpoint": "https://auth.example.com/register",
  "scopes_supported": ["read:notes", "write:notes"],
  "code_challenge_methods_supported": ["S256"],
  "response_types_supported": ["code"],
  "grant_types_supported": ["authorization_code", "refresh_token"]
}
```

`registration_endpoint`가 보이면 DCR이 가능하다는 신호입니다.

**4단계: 동적 등록**. 호스트는 자기 메타데이터를 등록 엔드포인트에 POST합니다.

```
POST /register HTTP/1.1
Host: auth.example.com
Content-Type: application/json

{
  "client_name": "Claude Desktop",
  "redirect_uris": ["http://127.0.0.1:43253/callback"],
  "grant_types": ["authorization_code", "refresh_token"],
  "response_types": ["code"],
  "token_endpoint_auth_method": "none",
  "scope": "read:notes write:notes"
}

HTTP/1.1 201 Created
{
  "client_id": "5f8e1c...",
  "redirect_uris": ["http://127.0.0.1:43253/callback"]
}
```

호스트는 받은 client_id를 (이 서버 → client_id) 매핑으로 저장합니다. 다음 번에는 등록을 다시 하지 않습니다.

**5단계: Authorize 요청**. PKCE를 함께 사용합니다. 호스트는 무작위 `code_verifier`를 만들고 그 해시값인 `code_challenge`를 생성합니다.

```
GET /authorize?
    response_type=code
    &client_id=5f8e1c...
    &redirect_uri=http://127.0.0.1:43253/callback
    &scope=read:notes%20write:notes
    &state=xY9z...
    &code_challenge=A3K8...
    &code_challenge_method=S256
    &resource=https://mcp.example.com
Host: auth.example.com
```

호스트는 사용자 기본 브라우저로 이 URL을 엽니다. 사용자는 AS에 로그인하고 동의 화면에서 scope를 확인합니다. 동의가 끝나면 AS는 redirect_uri로 authorization code를 돌려줍니다.

```
GET /callback?code=ABC...&state=xY9z... HTTP/1.1
```

호스트는 자신이 띄워 둔 로컬 HTTP 서버로 이 콜백을 받습니다.

`state` 파라미터는 CSRF 방어를 위한 무작위 값이고, 호스트는 자기가 만든 state 값과 일치하는지를 검증해야 합니다. `resource` 파라미터(RFC 8707)는 이 토큰이 어느 자원 서버용인지를 AS에 미리 알려 audience 결정을 돕습니다.

**6단계: 토큰 교환**. 호스트는 코드와 verifier를 함께 token 엔드포인트로 보냅니다.

```
POST /token HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=ABC...
&redirect_uri=http://127.0.0.1:43253/callback
&client_id=5f8e1c...
&code_verifier=ORIG_VERIFIER
&resource=https://mcp.example.com

HTTP/1.1 200 OK
{
  "token_type": "Bearer",
  "access_token": "eyJ...",
  "expires_in": 600,
  "refresh_token": "rfr...",
  "scope": "read:notes write:notes"
}
```

`code_verifier`가 일치해야 토큰이 발급됩니다. 발급된 access token의 audience 클레임은 `https://mcp.example.com`이 되어, 다른 자원 서버에서는 사용할 수 없게 됩니다.

**7단계: 실제 MCP 호출**. 호스트는 access token을 Authorization 헤더에 넣고 initialize를 다시 보냅니다.

```
POST /mcp HTTP/1.1
Host: mcp.example.com
Authorization: Bearer eyJ...
Content-Type: application/json
Accept: application/json, text/event-stream

{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

HTTP/1.1 200 OK
Mcp-Session-Id: 9c1f...

{"jsonrpc":"2.0","id":1,"result":{...}}
```

서버는 토큰의 서명·만료·audience·issuer·scope를 모두 확인한 뒤 응답을 돌려줍니다. 이후 호출은 같은 토큰과 세션 식별자를 함께 사용합니다.

토큰이 만료되면 호스트는 refresh token으로 새 access token을 받습니다.

```
POST /token HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=rfr...
&client_id=5f8e1c...

HTTP/1.1 200 OK
{
  "access_token": "new_eyJ...",
  "refresh_token": "new_rfr...",
  "expires_in": 600
}
```

refresh token 회전이 설정되어 있으면 응답에 새 refresh token이 함께 들어옵니다. 호스트는 옛 토큰을 폐기하고 새 토큰으로 교체합니다.

이 전체 흐름의 부분 실패는 호스트가 깔끔하게 처리해야 합니다. 등록이 실패하면 사용자에게 명확한 오류를 보이고, 사전 client_id 입력 같은 fallback으로 안내합니다. authorize가 거부되면 호스트는 사용자에게 "권한이 거부됨"을 알리고, 부분 권한으로 운영할지를 결정합니다. 토큰 교환 단계에서 audience 불일치가 발견되면 클라이언트는 다음 호출 전에 새 토큰을 받아야 합니다.

이 흐름이 매번 이렇게 길게 일어나는 것은 아닙니다. **두 번째 이후 호출**은 훨씬 짧습니다.

```
1. 호스트가 캐시된 access token으로 호출 → 200
   (또는 토큰 만료 시 refresh로 갱신 → 200)
```

새 서버를 처음 만나는 순간만 긴 흐름이고, 그 뒤로는 빠릅니다. 사용자가 한 번 동의하면 그 결정이 토큰·refresh로 표현되어 매번 새 로그인을 하지 않아도 됩니다.

이 흐름을 한 번 머리에 그려 두면, 보안 사고가 났을 때 어디서 무엇이 멈춰야 하는지가 분명해집니다. 의심스러운 client_id를 폐기하면 그 호스트는 더 이상 토큰을 받지 못합니다. refresh token을 무효화하면 다음 갱신 시점에 사용자는 다시 로그인해야 합니다. 자원 서버의 토큰 검증을 강화하면 잘못된 audience의 토큰이 즉시 차단됩니다.

정리하면, MCP의 DCR 흐름은 일곱 단계로 자동화되며, WWW-Authenticate → 메타데이터 → 등록 → authorize → token → 호출의 한 줄기로 흐릅니다. PKCE와 audience 파라미터, redirect 검증이 이 흐름을 안전하게 받쳐 줍니다.

다음 단원인 3.2.3에서는 흐름의 가장 중요한 보호 장치인 PKCE와 redirect 처리의 세부를 살펴봅니다.
