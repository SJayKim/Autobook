# 2.1.2 stdio transport

가장 단순하면서도 가장 널리 쓰이는 MCP transport는 **stdio**입니다. 이름이 가리키듯, 표준 입출력(stdin·stdout)을 메시지 채널로 사용하는 방식입니다. 2.1.1에서 살펴본 transport의 다섯 책임을 어떻게 충족하는지를 따라가다 보면, 왜 이 단순한 방식이 로컬 도구에 가장 적합한지 자연스럽게 드러납니다.

stdio transport의 작동 방식은 한 문장으로 표현할 수 있습니다. 호스트가 서버 프로세스를 자식으로 띄우고, 자식의 stdin·stdout을 통해 JSON-RPC 메시지를 줄바꿈으로 구분해 주고받습니다. 각 줄은 정확히 하나의 JSON-RPC 객체이며, 줄바꿈 문자(LF)가 메시지의 끝을 의미합니다. 이 단순한 약속이 프레이밍 문제를 해결합니다.

```
호스트 프로세스                  서버 프로세스 (자식)
  |                              |
  |   stdin --> -- 메시지 -->   |  (서버가 호스트로부터 받음)
  |   stdout <- <- 메시지 <-   |  (서버가 호스트로 보냄)
  |   stderr <- <- 로그 <- <-  |  (서버 디버그 출력)
```

양방향성은 stdin과 stdout이라는 두 채널을 동시에 사용해 확보됩니다. 호스트가 서버에게 메시지를 보낼 때는 stdin에, 서버가 호스트에게 응답이나 알림을 보낼 때는 stdout에 씁니다. 같은 채널을 양쪽이 번갈아 사용하는 형태가 아니라, 두 채널이 완전히 분리되어 흐릅니다.

stderr는 메시지 채널이 아니라 로그 채널입니다. 1.3.3에서 짧게 언급했듯, 서버가 `print()`로 디버그 메시지를 stdout에 찍어 버리면 그 줄이 호스트에게 JSON-RPC로 해석되어 메시지 파싱 오류가 납니다. 그래서 서버는 반드시 로그를 stderr로 보내야 합니다. Python의 `logging` 모듈은 기본 핸들러가 stderr로 출력하므로 자연스럽게 안전합니다. print를 쓸 때는 명시적으로 `file=sys.stderr`를 지정해야 합니다.

세션 유지는 프로세스 생명주기와 동치입니다. 자식 프로세스가 살아 있는 동안 세션이 살아 있고, 자식이 종료되면 세션도 끝납니다. 호스트가 다시 같은 서버에 연결하려면 새 자식 프로세스를 띄워야 합니다. 이 동치 관계 덕분에 세션 식별자 같은 별도 장치가 필요 없습니다.

인증은 운영체제의 권한 모델에 위임됩니다. 호스트가 자식 프로세스를 띄우므로, 자식은 호스트와 같은 사용자 계정으로 실행됩니다. 자식이 접근할 수 있는 파일·네트워크·시스템 자원은 그 사용자의 권한 안에서만 결정됩니다. MCP 자체의 인증은 필요 없으며, 별도 토큰이나 자격증명도 등장하지 않습니다.

오류 신호는 두 가지 형태로 옵니다. 첫째, 자식 프로세스가 종료되면 호스트는 stdin이 닫혔다는 신호와 함께 exit code를 받습니다. 둘째, 메시지가 너무 늦거나 형식이 틀리면 호스트는 SDK 수준에서 시간 초과나 파싱 오류로 받게 됩니다. 호스트는 이 신호들을 보고 서버를 재시작하거나 사용자에게 알릴 수 있습니다.

stdio transport의 장점은 단순함입니다. 네트워크 설정도 없고, 인증서도 없고, 포트도 없습니다. 서버 패키지를 설치하고 명령어를 한 줄 등록하면 끝납니다. Claude Desktop의 설정 예를 다시 살펴봅니다.

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/alice/projects"]
    }
  }
}
```

이 한 블록이 끝입니다. 호스트는 이 명령어를 자식 프로세스로 띄우고 그 stdin·stdout으로 JSON-RPC를 흘립니다. 별도의 데몬도, 별도의 서비스 등록도 없습니다.

또 다른 장점은 **격리**입니다. 자식 프로세스가 잘못 동작해도 호스트 자체에는 영향이 없습니다. 자식이 메모리를 너무 많이 쓰면 OS가 그 프로세스만 종료시키고, 호스트는 자식의 종료를 감지해 다시 띄우거나 사용자에게 알립니다. 이 격리는 보안 측면에서도 의미가 있습니다. 자식이 손상되어도 호스트의 메모리 공간에는 접근하지 못합니다.

물론 한계도 분명합니다. 첫째, **위치 종속성**입니다. 자식 프로세스는 호스트와 같은 머신에서 실행되어야 합니다. 원격 서버에 stdio로 붙는 것은 불가능에 가깝습니다(SSH로 우회할 수는 있지만 실용적이지 않습니다). 둘째, **자원 공유 불가**입니다. 호스트마다 자기 자식을 띄우므로, 사용자가 두 호스트에서 같은 서버를 쓰면 두 개의 프로세스가 뜹니다. 서버가 무거운 초기화를 한다면 비효율적입니다. 셋째, **확장성 한계**입니다. 한 머신의 호스트가 동시에 사용할 수 있는 자식 프로세스 수에는 OS의 한계가 있습니다.

이런 한계가 본격적인 문제가 되는 경우는 두 가지입니다. 원격으로 노출해야 하는 SaaS형 서버와, 여러 사용자가 함께 쓰는 엔터프라이즈 서버입니다. 두 경우 모두 HTTP 기반 transport로 옮겨가는 자연스러운 동기를 만듭니다. 다음 두 단원에서 그 과정을 따라갑니다.

운영상의 작은 팁 몇 가지를 적어 둡니다. 첫째, 환경 변수를 통해 서버에 설정을 전달할 수 있습니다. 호스트의 설정 파일에 `"env": {"API_KEY": "..."}` 같은 필드를 두면, 자식 프로세스의 환경 변수로 전달됩니다. 자격증명을 stdin으로 보내는 것보다 안전합니다. 둘째, 서버를 일시적으로 디버깅하려면 명령어를 `tee`로 감싸 메시지 로그를 파일로 저장하는 트릭이 있습니다. 셋째, stdio 서버를 npm·pip 같은 패키지 매니저로 배포하면 사용자가 손쉽게 설치할 수 있고, 자동 업데이트도 가능합니다.

정리하면, stdio transport는 호스트가 자식 프로세스를 띄우고 stdin·stdout·stderr로 JSON-RPC와 로그를 분리해 주고받는 방식입니다. 단순함, 격리, 운영체제 권한 모델 활용이 장점이고, 위치 종속성·자원 공유 불가·확장성 한계가 단점입니다. 로컬에서 한 사용자가 쓰는 도구라면 거의 항상 stdio가 정답입니다.

다음 단원인 2.1.3에서는 원격·다중 사용자 시나리오를 위해 등장했던 초기 HTTP+SSE transport와 그 한계를 살펴봅니다.
