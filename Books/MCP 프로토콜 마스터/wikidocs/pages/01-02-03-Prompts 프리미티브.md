# 1.2.3 Prompts 프리미티브

앞 두 단원에서 도구(1.2.1)와 자원(1.2.2)을 다루었습니다. 둘 모두 모델 또는 호스트가 주도적으로 호출하거나 읽는 자산입니다. 그런데 사용자 입장에서는 "지금 이 작업을 하려면 어떤 도구를 어떤 순서로 써야 하는가"를 매번 떠올리기 어려울 때가 많습니다. 이런 사용자 흐름을 미리 정의해 두고 슬래시 명령처럼 노출하는 것이 **Prompts** 프리미티브입니다.

**Prompt**는 한 번의 호출로 모델에게 던질 메시지 배열을 만들어 주는 템플릿입니다. 도구가 "함수"라면, 자원이 "데이터"라면, 프롬프트는 "준비된 메시지 묶음"입니다. 사용자는 슬래시 메뉴에서 프롬프트를 고르고 몇 가지 인자를 채우기만 하면, 호스트가 그 자리에서 메시지 배열을 만들어 모델에 전달합니다.

예를 들어 코드 리뷰 서버가 다음과 같은 프롬프트를 제공한다고 합시다.

```json
{
  "name": "review_pr",
  "description": "Generate a code review for a given pull request.",
  "arguments": [
    {"name": "repo", "description": "Repository in owner/name form", "required": true},
    {"name": "pr_number", "description": "PR number", "required": true}
  ]
}
```

사용자가 호스트에서 `/review_pr` 명령을 고르고 repo와 pr_number를 채우면, 호스트는 **prompts/get** 메서드를 호출합니다. 서버는 그 자리에서 해당 PR의 diff를 받아 와 다음과 같은 메시지 배열을 돌려줍니다.

```json
{
  "messages": [
    {"role": "user", "content": {"type": "text", "text": "Review this PR:\n```\n<diff>\n```"}}
  ]
}
```

호스트는 받은 메시지 배열을 모델에게 그대로 입력합니다. 이때 중요한 점은, 프롬프트의 본문이 어떻게 구성되는지를 호스트가 알 필요가 없다는 점입니다. diff를 가져오는 일도, 시스템 지침을 붙이는 일도, 첨부 자원을 결합하는 일도 모두 서버 측 결정입니다.

프롬프트의 메서드도 도구·자원과 같은 패턴입니다. **prompts/list**는 사용 가능한 프롬프트 목록을 가져오고, **prompts/get**은 인자를 넣어 실제 메시지 배열을 받아옵니다. 새 프롬프트가 추가되거나 사라지면 서버는 **notifications/prompts/list_changed** 알림을 보냅니다.

프롬프트 인자는 단순한 문자열로 정의되지만, 사양은 **자동 완성**(completion)을 위한 별도 메서드를 제공합니다. 예를 들어 repo 인자라면 사용자가 입력 도중에 자동 완성 후보(예: 사용자가 접근 가능한 저장소 목록)를 받을 수 있어야 자연스럽습니다. 호스트는 **completion/complete** 메서드로 후보를 요청하고, 서버는 자기 도메인에 맞는 후보를 돌려줍니다.

프롬프트가 도구와 자주 혼동되는데, 둘은 서로 보완적입니다. 도구는 모델이 자율적으로 사용하는 함수이고, 프롬프트는 사용자가 명시적으로 호출하는 워크플로입니다. 예를 들어 "이 PR을 리뷰해" 같은 일은 사용자가 명시적으로 시작하는 것이 자연스럽고, 그 안에서 모델이 더 깊은 분석을 위해 추가 도구를 호출할 수 있습니다. 프롬프트가 도구를 부르는 일은 흔하고, 그 반대는 사양상 정의되어 있지 않습니다.

프롬프트의 또 다른 가치는 **노출 표면을 명확히 만든다**는 점입니다. 도구 목록만 보면 사용자는 자신이 어떤 일을 시작할 수 있는지 한눈에 알기 어렵습니다. 반면 프롬프트 목록은 "이 서버로 시작할 수 있는 시나리오 메뉴"가 됩니다. 호스트가 슬래시 메뉴, 컨텍스트 메뉴, 빠른 명령 같은 UI로 노출하면 사용자가 발견(discover)하기 좋습니다.

프롬프트 설계 시 자주 빠지는 함정은 두 가지입니다. 첫째, 인자가 너무 많아서 사용자가 입력에 부담을 느끼는 경우입니다. 핵심 인자는 한두 개로 줄이고 나머지는 기본값을 두는 편이 좋습니다. 둘째, 프롬프트 안에 너무 많은 지침을 넣어 모델의 자율성을 지나치게 제약하는 경우입니다. 도구 설계와 마찬가지로 프롬프트도 사용자의 의도를 빠르게 모델에게 전달하는 시작점이지, 모델의 모든 행동을 강제하는 스크립트가 아닙니다.

호스트가 프롬프트를 어떻게 UI로 노출하는지는 자유입니다. Claude Desktop은 `/` 메뉴에 프롬프트를 모아 보여 주고, Cursor는 채팅 입력창의 명령 메뉴에 프롬프트를 노출합니다. 자체 UI를 가지지 않는 자동화 환경(예: CI에서 도는 에이전트)이라면 프롬프트를 사용하지 않고 도구만 노출해도 됩니다. 서버는 어떤 호스트가 어떤 UI를 가질지에 관계 없이, 적절한 프롬프트 묶음을 단순히 노출해 두면 됩니다.

정리하면, Prompts 프리미티브는 사용자가 직접 시작하는 워크플로를 서버가 미리 정의해 두는 표준입니다. prompts/list와 prompts/get이 기본 메서드이며, 인자 자동 완성과 list_changed 알림이 사용성을 받쳐 줍니다. 도구가 모델의 도구라면, 프롬프트는 사용자의 도구입니다.

다음 단원인 1.2.4에서는 보조 프리미티브인 Sampling과 Roots를 살펴봅니다. 둘 다 일반적인 사용 빈도는 낮지만 특정 시나리오에서 결정적인 역할을 합니다.
