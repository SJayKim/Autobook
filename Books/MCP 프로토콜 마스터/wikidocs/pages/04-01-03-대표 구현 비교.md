# 4.1.3 대표 구현 비교

앞 두 단원에서 게이트웨이의 동기(4.1.1)와 통합 패턴(4.1.2)을 살펴보았습니다. 이 단원에서는 실제 시장에서 자주 마주치는 게이트웨이 구현들을 비교해, 각자의 설계 선택이 어떤 시나리오에 잘 맞는지를 정리합니다. 네 가지를 대상으로 잡았습니다. **Anthropic의 MCP Connector**, **Cloudflare의 Workers MCP gateway**, **IBM mcp-context-forge**, **Microsoft의 mcp-proxy/AI Foundry**입니다.

**Anthropic MCP Connector**는 Claude.ai와 Claude API에 통합된 원격 MCP 게이트웨이 기능입니다. 사용자는 Claude 설정에서 MCP 서버 URL을 추가하기만 하면, 그 서버가 자동으로 Claude의 호출에 합류합니다. 게이트웨이가 클라이언트 측에 통합되어 있다는 점이 특징입니다. 호스트(Claude)가 곧 게이트웨이 역할을 동시에 수행합니다.

```
Anthropic MCP Connector의 특징

설치 위치        : Claude 호스트 안 (호스트=게이트웨이)
인증            : OAuth 2.1 + DCR
지원 transport   : Streamable HTTP
통합 패턴       : Aggregator
사용자 모델     : 한 사용자가 자기 계정에 서버 등록
강점            : Anthropic 생태계 통합, 사용성
한계            : 다중 사용자 정책·감사가 별도 솔루션 필요
```

Anthropic Connector는 개인 사용자나 작은 팀에 매우 적합합니다. 그러나 회사 단위 거버넌스(공통 정책·감사·SSO 통합)가 필요하다면 별도 솔루션이 필요합니다. Claude Enterprise 플랜이 이 부분을 보완합니다.

**Cloudflare Workers MCP gateway**는 Cloudflare의 엣지 컴퓨팅 플랫폼 위에 만들어진 서버리스 게이트웨이입니다. Cloudflare AI Gateway와 결합되어, MCP 호출을 한 곳에서 인증·관측·캐싱할 수 있게 해 줍니다.

```
Cloudflare 게이트웨이의 특징

설치 위치        : Cloudflare 엣지 (서버리스)
인증            : OAuth 2.1 (Cloudflare Access·Zero Trust 통합)
지원 transport   : Streamable HTTP
통합 패턴       : Aggregator + Router
사용자 모델     : 사용자 또는 조직 단위
강점            : 전 세계 엣지, 빠른 latency, 캐싱
한계            : Cloudflare 생태계 의존
```

Cloudflare 게이트웨이는 글로벌 사용자를 가진 SaaS형 도구 운영자에게 매력적입니다. 엣지에서 인증·정책을 처리하고, 본 서버에 도달하기 전에 부적합 요청을 차단할 수 있습니다.

**IBM mcp-context-forge**는 오픈소스로 공개된 엔터프라이즈 게이트웨이입니다. 회사 안에서 직접 운영하는 모델을 가정하고 설계되었으며, 정책·감사·관측의 모든 기능을 제공합니다.

```
IBM mcp-context-forge의 특징

설치 위치        : 자체 운영 서버 (온프레미스·클라우드)
인증            : OAuth 2.1, OIDC, SSO
지원 transport   : stdio·SSE·Streamable HTTP 모두
통합 패턴       : Aggregator + Federation
사용자 모델     : 사용자·팀·테넌트
강점            : 풀 스택 정책·감사·관측, 오픈소스
한계            : 직접 운영 부담
```

이 구현은 컴플라이언스 요구가 강한 조직(금융·의료·정부)에 잘 맞습니다. 오픈소스이므로 회사 정책에 맞춰 커스터마이징할 수 있고, 자체 IdP와 통합이 용이합니다. 다만 직접 운영해야 하는 부담이 따릅니다.

**Microsoft mcp-proxy와 AI Foundry**는 두 가지 방향으로 게이트웨이 기능을 제공합니다. mcp-proxy는 stdio 서버를 HTTP로 노출하거나 그 반대를 수행하는 가벼운 어댑터이고, AI Foundry는 Azure 내에서 MCP를 통합 관리하는 매니지드 서비스입니다.

```
Microsoft 솔루션의 특징

설치 위치        : Azure 매니지드 / 자체 배포
인증            : Entra ID(Azure AD), OAuth 2.1
지원 transport   : stdio ↔ HTTP 변환
통합 패턴       : 어댑터 + Aggregator
사용자 모델     : Azure 사용자·그룹
강점            : Microsoft 365·Copilot 통합, Entra ID 자연 결합
한계            : Azure 종속
```

Microsoft 솔루션은 이미 Microsoft 365·Azure 생태계를 쓰는 조직에 자연스러운 선택입니다. Entra ID와의 깊은 통합으로 사용자 관리·MFA·조건부 액세스를 그대로 활용할 수 있습니다.

네 구현을 한 표로 비교해 볼 수 있습니다.

```
                 Anthropic  Cloudflare  IBM forge   Microsoft
설치 위치         호스트 측   엣지        자체        Azure
인증              OAuth+DCR   OAuth+ZT    OAuth+OIDC  Entra ID
패턴 강조          Aggregator  Aggreg+Rt   Agg+Fed     Adapter+Agg
정책 엔진         기본        고급        고급         중간
감사 로그         기본        통합 분석    풀 스택      Azure Monitor
오픈소스          x           x           o            일부
적합 시나리오      개인·소규모  글로벌 엣지  엔터프라이즈 MS 생태계
```

이 표는 절대적 우열을 가리지 않습니다. 각 구현이 다른 시나리오에 답합니다. 다음과 같은 질문을 자기 환경에 던져 보면 선택이 분명해집니다.

```
질문                                       어디가 유리한가
사용자가 회사 안인가, 일반 소비자인가      회사 → 자체 게이트웨이, 소비자 → Anthropic
규제 준수 증빙이 필요한가                   필요 → IBM forge 또는 매니지드
이미 어떤 클라우드 생태계에 있는가          Azure → Microsoft, Cloudflare → Cloudflare
운영 인력은 어느 정도 가용한가              적음 → 매니지드, 많음 → 자체 운영
글로벌 사용자가 많은가                       많음 → 엣지 게이트웨이
도구 카탈로그 자율 성장 요구가 있는가       있음 → Federation 지원 구현
```

선택을 한 번에 고정하기 어렵다면, **레퍼런스 구현으로 시작해 점진적으로 옮겨 가는 전략**이 안전합니다. 예를 들어 처음에는 IBM mcp-context-forge로 자체 운영해 보안 요구를 충족하고, 사용량이 늘면 Cloudflare 같은 엣지 솔루션으로 일부 워크로드를 이동시키는 식입니다. MCP가 표준 프로토콜이므로 도구·서버 자체는 손볼 필요가 없습니다. 게이트웨이만 교체하면 됩니다.

게이트웨이 선택을 끝낸 뒤에는 다음 항목을 체크리스트로 둡니다.

```
1. 인증 - 회사 IdP 또는 OAuth 2.1 AS와 연결되는가
2. 정책 - allow/deny, scope, PII 필터를 한 곳에서 구성 가능한가
3. 관측 - 모든 호출이 표준 trace·log로 남는가
4. 비용 - 사용자·팀 단위 비용·횟수 추적이 가능한가
5. 가용성 - 다중 인스턴스로 HA가 가능한가
6. 확장 - 새 MCP 서버를 추가하는 데 호스트 손이 필요 없는가
7. 디버깅 - 게이트웨이 자체의 메시지를 트레이스 가능 형태로 노출하는가
8. 백업·복구 - 게이트웨이 설정과 정책의 버전 관리가 가능한가
9. 멀티 테넌트 - 한 인스턴스에서 여러 조직을 분리 운영할 수 있는가
10. 마이그레이션 - 다른 게이트웨이로 옮길 때 도구·서버 변경이 없는가
```

이 체크리스트는 신규 도입뿐 아니라 정기 점검에도 유용합니다. 시간이 지나면서 정책·감사 요구가 늘어나면, 처음 선택한 게이트웨이가 더 이상 충분치 않을 수 있습니다. 체크리스트를 분기마다 한 번씩 다시 적용해 보면 교체 시점을 빠르게 감지할 수 있습니다.

정리하면, MCP 게이트웨이 시장에는 사용자·환경에 따라 알맞은 여러 구현이 있으며, Anthropic(호스트 통합), Cloudflare(엣지), IBM(자체 운영), Microsoft(클라우드 통합)이 대표적입니다. 선택은 인증·정책·관측·운영 부담·생태계 의존이라는 다섯 차원의 가중치에 따라 달라집니다.
