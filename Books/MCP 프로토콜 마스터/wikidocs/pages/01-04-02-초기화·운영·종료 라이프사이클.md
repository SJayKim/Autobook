# 1.4.2 초기화·운영·종료 라이프사이클

앞 단원 1.4.1에서 메시지의 형식을 살펴보았으니, 이제 그 메시지가 시간 축 위에서 어떤 순서로 흐르는지를 정리할 차례입니다. MCP 세션의 일생은 세 단계로 나눌 수 있습니다. **초기화**, **운영**, **종료**입니다. 이 라이프사이클을 정확히 이해해 두면 재연결, 알림 흐름, 자원 정리에 관한 많은 질문에 자연스럽게 답할 수 있습니다.

세션은 호스트가 transport를 통해 서버에 처음 메시지를 보내는 순간 시작됩니다. stdio라면 호스트가 서버 프로세스를 자식으로 띄우는 순간이고, HTTP라면 호스트가 첫 HTTP 요청을 보내는 순간입니다. 이 시점부터 일정한 순서가 따라옵니다.

```
[초기화 단계]
  Client → Server : initialize (protocolVersion, capabilities)
  Server → Client : response (protocolVersion, capabilities)
  Client → Server : notifications/initialized

[운영 단계]
  - tools/list, tools/call, resources/read, prompts/get, ...
  - 양방향 알림: list_changed, progress, log, cancel, ...
  - 서버 → 클라이언트: sampling/createMessage(선택)

[종료 단계]
  Client → Server : (선택) 정리 알림
  transport 종료
```

**초기화 단계**의 핵심은 1.3.2에서 살펴본 capability negotiation입니다. 한 가지 더 짚어 둘 점은 초기화 도중에는 다른 메서드를 호출해선 안 된다는 것입니다. initialize 응답이 돌아오기 전에 tools/list를 보내면 서버는 invalid request 오류로 응답할 가능성이 큽니다. 마찬가지로 서버는 initialized 알림을 받기 전까지는 자발적으로 list_changed나 sampling 같은 메시지를 보내지 않는 것이 권장됩니다. 이 약속이 깨지면 양쪽이 깨끗하지 않은 초기 상태에서 작업을 시작하게 됩니다.

운영 단계에 들어가면 호스트와 서버는 양방향으로 자유롭게 메시지를 주고받습니다. 가장 흔한 흐름은 도구 호출입니다. 사용자가 "이 파일을 읽어 줘"라고 입력하면 호스트는 모델에게 컨텍스트를 만들고, 모델이 도구 호출 의도를 돌려주면 호스트가 사용자 동의를 받은 뒤 tools/call을 보냅니다. 결과를 받으면 모델에게 다시 전달해 자연어 응답을 생성하고 사용자에게 보여 줍니다.

운영 중 자주 보는 알림은 네 가지입니다. **list_changed**는 도구·자원·프롬프트 목록이 바뀌었음을 알립니다. 호스트는 알림을 받으면 해당 list 메서드를 다시 호출해 새 목록을 받아 갑니다. **resources/updated**는 구독 중인 자원의 내용이 바뀌었음을 알립니다. 호스트는 필요한 경우에만 resources/read를 다시 호출해 내용을 갱신합니다. **progress**는 오래 걸리는 작업의 진행률을 알립니다. 호스트는 사용자 UI에 진행 막대를 보여 주거나, 모델에게 부분 결과를 흘려보낼 수 있습니다. **logging**은 서버가 호스트의 로그 시스템으로 메시지를 보내는 통로입니다.

**취소** 알림(cancellation)은 별도로 살펴볼 가치가 있습니다. 호스트가 진행 중인 요청을 취소하고 싶을 때 `notifications/cancelled`를 보냅니다. params에는 취소 대상 요청의 id가 들어갑니다. 서버는 받은 알림에 따라 작업을 중단하고, 가능하면 빠르게 메모리를 정리합니다. 한 가지 주의할 점은 알림이 비동기로 흐르므로, 취소 알림이 도착했을 때 이미 응답이 출발했을 수 있다는 점입니다. 호스트는 응답이 늦게 도착해도 무시하도록 준비되어 있어야 합니다.

운영 단계의 가장 흥미로운 메시지는 1.2.4에서 다룬 sampling입니다. 서버 → 호스트 방향의 메시지로, 호스트가 sampling capability를 선언했을 때만 의미를 가집니다. 이 메시지는 운영 단계 내내 어느 시점에든 발생할 수 있으므로, 호스트는 sampling 처리기를 항상 준비해 두어야 합니다.

**종료 단계**는 의외로 단순합니다. MCP는 명시적인 shutdown 메서드를 정의하지 않습니다. 종료는 transport의 종료로 표현됩니다. stdio라면 한 쪽이 표준 입출력을 닫는 행위, HTTP라면 세션을 만료시키거나 연결을 끊는 행위가 종료입니다. 종료 시점에 호스트는 다음 일을 합니다.

1. 진행 중인 도구 호출이 있다면 취소 알림을 보낸다.
2. 자원 구독이 있다면 구독을 해제한다.
3. transport를 닫는다.

서버는 종료 신호를 받으면 자기 쪽의 작업을 정리하고 종료합니다. stdio 서버라면 자식 프로세스가 자연스럽게 끝나는 형태가 됩니다. HTTP 서버라면 다음 요청에서 410을 돌려주는 식으로 세션 만료를 알립니다.

세션의 가장 까다로운 부분은 **재연결**과의 경계입니다. transport가 끊겼다가 다시 붙으면, 이전 세션을 이어 갈 수도 있고 새 세션을 시작할 수도 있습니다. stdio는 본질적으로 프로세스 단위라서, 자식 프로세스가 죽으면 그 세션은 끝납니다. 호스트가 새 자식을 띄우면 새 세션이 시작됩니다. Streamable HTTP는 다릅니다. 클라이언트가 같은 Mcp-Session-Id를 가지고 다시 접속하면 서버는 같은 세션을 이어 줄 수 있습니다. 이 차이는 Phase 2에서 자세히 다룹니다.

라이프사이클을 머리에 잘 그려 두는 가장 큰 가치는 디버깅에 있습니다. 서버가 동작하지 않을 때 가장 먼저 확인할 것은 초기화 흐름입니다. initialize 응답이 도착했는가, initialized 알림을 호스트가 보냈는가, capability가 기대한 대로 협상되었는가. 운영 단계에서 메시지가 끊긴다면, 한 쪽이 사양에 없는 메시지를 보내고 있지는 않은지, 알림이 잘못된 시점에 발사되고 있지는 않은지를 확인합니다. 종료가 깔끔하지 않다면, 자원 구독이 해제되지 않았거나 진행 중인 도구가 취소 처리되지 않은 채 transport가 닫혔을 가능성이 큽니다.

정리하면, MCP 세션은 초기화·운영·종료의 세 단계를 거치며, 각 단계에서 허용되는 메시지와 권고 흐름이 명확히 정해져 있습니다. 이 라이프사이클은 stdio든 HTTP든 transport에 무관하게 같은 모습으로 동작하므로, 한 번 익혀 두면 어떤 환경에서도 그대로 재사용할 수 있습니다.

이 단원으로 Phase 1을 마무리합니다. 다음 Phase 2에서는 그동안 미뤄 두었던 질문, 곧 이 메시지들이 실제로 어떤 운반 계층 위를 흐르는지를 살펴봅니다. 다음 단원인 2.1.1에서는 Transport 계층의 역할부터 시작합니다.

이 단원을 마치면 MCP 세션의 생애 전반에 어떤 메시지가 어떤 순서로 흐르는지 머릿속에 그릴 수 있고, 디버깅 시 무엇을 먼저 확인할지 결정할 수 있습니다.
