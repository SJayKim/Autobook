# 1.4.1 JSON-RPC 2.0 기반 메시지

지금까지 우리는 MCP 메시지를 거의 추상적으로 다루었습니다. tools/list라는 메서드가 있고 응답이 돌아온다는 식이었습니다. 이 단원에서는 그 메시지가 실제 어떤 모양으로 흐르는지, 그 바탕인 **JSON-RPC 2.0** 사양을 정리합니다.

**JSON-RPC**는 2010년 처음 사양이 정해진, 매우 단순한 원격 호출 규약입니다. HTTP·gRPC·SOAP 같은 다른 RPC와 달리, JSON-RPC는 한 줄의 JSON 객체로 호출과 응답을 표현합니다. 운반 계층(transport)을 가리지 않으며, JSON을 주고받을 수 있는 어떤 채널 위에서도 동작할 수 있습니다. MCP가 stdio·HTTP+SSE·Streamable HTTP 같은 다양한 transport를 자연스럽게 받아들일 수 있는 배경에는 이 운반 중립성이 있습니다.

JSON-RPC 2.0 메시지는 세 가지 형태로 나뉩니다. 요청(request), 응답(response), 알림(notification)입니다. 각각을 짧게 살펴보겠습니다.

**요청**은 다음과 같습니다.

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/list",
  "params": {}
}
```

네 필드 중 `jsonrpc`는 항상 "2.0"이며, `id`는 응답과 매칭하기 위한 식별자입니다. id는 정수나 문자열 어느 쪽도 가능합니다. `method`는 호출할 메서드 이름이고, `params`는 인자입니다. params는 객체(이름 기반)나 배열(위치 기반) 둘 다 허용되지만, MCP는 객체만 사용합니다.

**응답**은 요청 id를 그대로 가지고 돌아옵니다.

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "result": {"tools": [/* ... */]}
}
```

성공이면 `result`, 실패면 `error`를 가집니다. 둘은 동시에 나타나지 않습니다. 에러 응답은 다음 형태입니다.

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "error": {"code": -32601, "message": "Method not found"}
}
```

에러 코드는 JSON-RPC 2.0 사양이 일부를 미리 정의해 두었습니다. -32700은 parse error, -32600은 invalid request, -32601은 method not found, -32602는 invalid params, -32603은 internal error입니다. MCP는 이 표준 코드를 그대로 사용하고, 그 외 도메인 오류는 도구 결과의 isError 플래그로 표현하도록 권장합니다.

**알림**은 응답이 필요 없는 단방향 메시지입니다. 요청과 형식이 거의 같지만 id가 없습니다.

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed",
  "params": {}
}
```

알림은 응답이 오지 않을 것을 전제로 합니다. 따라서 수신 측이 처리하지 못해도 상대는 알 길이 없습니다. MCP에서 알림은 주로 capability 변경(list_changed), 진행률(progress), 로그(logging), 취소(cancel) 같은 부수적 이벤트에 사용합니다. 운영 안정성을 위해 알림에 중요한 동의·실행 의미를 담지 않는 편이 좋습니다.

요청과 응답의 매칭은 전적으로 id에 의존합니다. 호스트가 동시에 여러 메서드를 호출하면 응답도 순서와 상관없이 돌아올 수 있고, 호스트는 id로 짝을 맞춰야 합니다. SDK가 이 매칭을 자동으로 처리하므로 일반 개발자가 직접 id를 관리할 일은 거의 없지만, 디버깅 중에 와이어 메시지를 보다 보면 id가 어떻게 흐르는지 알아두는 것이 도움이 됩니다.

JSON-RPC는 **batch** 메시지도 정의합니다. 여러 요청을 하나의 배열로 묶어 보내고, 각 응답도 배열로 돌려받는 방식입니다. MCP 2025-03-26 사양은 한때 batch를 명시적으로 채택했지만, 2025-06-18 사양에서 다시 제거되었습니다. 이유는 두 가지였습니다. 첫째, Streamable HTTP가 SSE로 응답을 흘려보낼 수 있게 되면서 batch의 효용이 줄었습니다. 둘째, batch가 capability 협상이나 알림과 섞일 때 의미가 모호해지는 경우가 있었습니다. 그래서 현재 사양은 메시지를 한 건씩 주고받는 것을 표준으로 합니다.

메시지 흐름을 디버깅할 때 가장 유용한 도구는 단순한 로그입니다. transport가 stdio라면 stderr로 메시지를 찍어 두면 되고, HTTP라면 reverse proxy의 access log를 살펴보면 됩니다. SDK들은 보통 환경 변수로 디버그 로그를 켤 수 있습니다. Python SDK는 `MCP_LOG_LEVEL=DEBUG`, TypeScript SDK는 `DEBUG=mcp:*` 같은 식입니다.

```
호스트 → 서버 : {"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}
서버 → 호스트 : {"jsonrpc":"2.0","id":1,"result":{...}}
호스트 → 서버 : {"jsonrpc":"2.0","method":"notifications/initialized"}
호스트 → 서버 : {"jsonrpc":"2.0","id":2,"method":"tools/list"}
서버 → 호스트 : {"jsonrpc":"2.0","id":2,"result":{"tools":[...]}}
```

이 흐름이 모든 MCP 통신의 기본 골격입니다. 어떤 transport, 어떤 SDK를 쓰든 이 모양에서 벗어나지 않습니다.

JSON-RPC를 사용한 결과 얻게 된 또 하나의 특성은 **운반 계층 독립성**입니다. 메시지 자체가 JSON 한 덩어리이므로, stdio에서는 줄바꿈으로 구분하면 그만이고, HTTP에서는 본문에 한 객체씩 담아 보내면 그만입니다. WebSocket 위에 올려도 동작하고, 파이프를 통한 통신에도 자연스럽게 맞아 들어갑니다. 사양이 transport와 protocol을 분리해 둔 이유가 여기 있습니다. 이 분리는 다음 Phase에서 자세히 다룹니다.

JSON-RPC 자체는 단순하지만, 그 단순함이 곧 MCP의 적응성으로 이어집니다. 새 기능을 추가하려면 새 method를 정의하면 되고, 그 method를 모르는 측은 method not found를 돌려보내면 됩니다. capability 협상과 결합해 미지원 메서드를 부르지 않도록 만들면, 사양이 진화해도 통신이 깨지지 않습니다.

정리하면, MCP 메시지는 JSON-RPC 2.0의 요청·응답·알림 세 형식 위에 올라가 있으며, id 매칭과 표준 에러 코드, capability 기반 메서드 선택을 통해 단순함과 확장성을 동시에 가집니다. 이 구조를 머릿속에 가지고 있으면 어떤 SDK·언어로 만든 MCP 서버를 다루더라도 내부에서 무엇이 흐르고 있는지 빠르게 그려 낼 수 있습니다.

다음 단원인 1.4.2에서는 이 메시지들이 시간 축 위에서 어떻게 흐르는지, 초기화·운영·종료 라이프사이클을 살펴봅니다.
