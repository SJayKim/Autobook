# 5.3.2 비용과 지연 최적화

멀티 에이전트 시스템은 정확도를 얻는 대신 비용과 지연을 늘립니다. 이 단원에서는 그 비용·지연을 다시 줄이는 표준 패턴을 정리합니다. 평가에서 후퇴하지 않으면서 자원을 절약하는 것이 목표입니다.

먼저 **비용과 지연의 주된 동인**을 분해합니다.

```
동인                              영향
모델 호출 수                       비용·지연 모두 증가
컨텍스트 길이                       입력 토큰 비용·지연
모델 크기                           단위 비용·응답 시간
도구 호출 수                        외부 API 비용·지연
직렬화된 단계                       총 지연 증가
재시도·중복 호출                    배수로 증가
```

최적화의 출발점은 시스템에서 어떤 동인이 가장 큰 비중을 차지하는지 측정하는 것입니다. 측정 없이 최적화하면 자주 잘못된 곳을 손보게 됩니다. 4.3.2에서 다룬 비용 추적이 이 단계의 입력 데이터를 제공합니다.

다음은 **메시지·도구 호출 수 감축**입니다.

```
패턴                                       효과
도구 description 풍부화                    잘못된 호출 → 정답 호출 한 번
sequential을 fan-out으로                    지연 단축
중복 호출 캐시                              비용 절감
종료 조건 명확화                            무한 반복 방지
supervisor의 위임 정확도 개선               잘못된 subgraph 호출 감소
```

가장 흔한 비효율은 **잘못된 도구 호출의 누적**입니다. 모델이 도구 인자를 잘못 주면 isError를 받고 다시 호출합니다. 이를 줄이려면 inputSchema의 description을 풍부히 적고, 자주 잘못되는 인자에 예시를 함께 둡니다. 이 작은 변화가 호출 수를 한 단계 줄여 줍니다.

다음은 **모델 라우팅**입니다. 모든 단계에 가장 큰 모델을 쓸 필요는 없습니다. 다음 라우팅 패턴이 효과적입니다.

```
라우팅 기준                       작은 모델               큰 모델
분류·식별                          ◯                      
도구 호출 결정                      ◯                      
간단한 변환                         ◯                      
복잡한 추론                                                 ◯
긴 컨텍스트 종합                                            ◯
사용자 응답 생성                                            ◯
```

작은 모델(Haiku, GPT-4o mini 등)은 정확도가 충분하면서도 비용·지연이 매우 작습니다. 단순 분류·라우팅·필터링 같은 단계에 적합합니다. 복잡한 추론과 사용자 응답에는 큰 모델을 사용합니다.

라우팅을 구현하는 흔한 방법은 **계층적 모델 사용**입니다.

```
사용자 메시지
  → 작은 모델: 의도 분류
  → (필요시) 작은 모델: 인자 추출
  → 큰 모델: 핵심 추론
  → 작은 모델: 최종 응답 정리
```

이 흐름은 총 토큰 비용을 크게 줄이면서 큰 모델의 강점은 보존합니다.

다음은 **캐시 활용**입니다. 캐시는 두 계층에서 작동합니다.

**Prompt cache**는 모델 제공자가 제공하는 기능입니다. 같은 시스템 프롬프트나 같은 자료가 반복되면 두 번째부터 입력 토큰 비용이 크게 절감됩니다. Anthropic, OpenAI 모두 prompt caching을 지원합니다. 멀티 에이전트 시스템에서는 같은 system prompt가 여러 에이전트에서 반복되거나, 같은 자료가 여러 호출에 들어가는 경우가 흔하므로 prompt cache가 큰 효용을 냅니다.

**MCP response cache**는 게이트웨이나 호스트가 도구 호출 결과를 캐시하는 패턴입니다. 같은 도구·같은 인자의 호출이 짧은 시간에 반복되면 두 번째부터는 캐시된 결과를 돌려줍니다.

```
캐시 키                            TTL 권장
도구 + 인자 해시                    30초~5분 (자원 변경 빈도에 따라)
사용자 + 도구 + 인자               민감도 따라 다름
검색 쿼리                            5분~1시간
정적 자원 (스키마·메타데이터)        몇 시간
```

캐시는 정확성을 잃을 수 있으므로, 자원이 자주 변하는 경우에는 짧은 TTL을 쓰거나 list_changed 알림(1.2.1)을 신호로 무효화합니다.

다음은 **병렬 한도 조절**입니다. fan-out으로 동시에 호출하는 분기가 너무 많으면 한도 도달과 자원 경합이 생깁니다. 적절한 동시 호출 한도가 필요합니다.

```
한도 종류                       전형 값
사용자별 동시 호출              3~5
도구별 동시 호출                10~50 (도구 특성)
전체 시스템 동시 LLM 호출       100~1000
외부 API 동시 호출              해당 API rate limit의 80%
```

이 한도가 잘 설정되면 burst에도 시스템이 안정적으로 동작합니다. 한도를 넘으면 backoff와 재시도가 자동으로 일어나도록 SDK 수준에서 구현합니다.

다음은 **에이전트 수와 품질 곡선**입니다. 에이전트를 늘리면 정확도가 늘어나다가 어느 지점부터 감소합니다. 비용은 거의 선형으로 증가하므로, 한계 효용이 비용을 정당화하는 지점을 넘어가면 더 늘리지 않는 것이 좋습니다. 이 지점을 찾는 표준 방법은 **A/B 실험**입니다. 에이전트 수가 다른 시스템을 같은 평가 세트에 돌리고 정확도·비용·지연을 비교합니다.

```
에이전트 수      정확도      비용         권장
1                70%         1.0x         단순 시나리오
2                82%         1.8x         좋은 균형 시작점
3                86%         2.7x         복잡 시나리오
4                87%         3.6x         보통 한계
5+               87~88%      4.5x+        한계 효용 작음
```

실제 곡선은 시나리오마다 다르지만, 한계 효용이 빠르게 줄어드는 패턴은 거의 보편적입니다.

다음은 **지연 최적화의 특수 기법**입니다.

```
기법                             효과
스트리밍 응답                      사용자 인지 지연 단축
사전 검색 prefetch                 다음 단계에 필요한 자료 미리 가져오기
모델 호출 병렬화                   분기로 묶을 수 있는 호출 동시 진행
빠른 첫 토큰                       작은 모델로 사용자 응답 시작 후 큰 모델이 계속
사용자 입력 예측                   사용자가 자주 묻는 후속 질문 사전 준비
```

특히 **스트리밍**은 사용자 경험을 크게 바꿉니다. 실제 지연은 같아도, 응답이 토큰 단위로 흘러나오면 사용자는 시스템이 빠르게 반응한다고 느낍니다.

다음은 **도구 호출 비용 최적화**입니다.

```
패턴                                효과
검색 도구 결과를 임시 캐시            동일 검색 반복 호출 절감
큰 자원은 URI로 반환 후 필요할 때 read  컨텍스트·전송 비용
도구 호출 묶음 처리                   여러 작은 호출을 한 도구로 통합
외부 API의 batch 엔드포인트 활용     API 호출 횟수 감소
```

특히 큰 자원을 다룰 때는 임베디드 리소스(1.2.2)를 활용해 URI만 돌려주고 호스트가 필요시 read하는 패턴이 매우 효과적입니다. 도구 결과가 항상 컨텍스트에 들어갈 필요는 없습니다.

다음은 **모니터링과 알림**입니다. 최적화는 한 번 끝나는 일이 아닙니다. 다음 지표를 매주 검토합니다.

```
지표                                       해석
평균 모델 호출 수 per 요청                  과한 단계는 줄일 수 있나
평균 도구 호출 수 per 요청                  중복은 없나
평균 토큰 수 per 요청                       프롬프트 압축 여지
캐시 적중률                                  더 끌어올릴 여지
P95 지연                                    burst 패턴 점검
부분 실패율                                  안정성 점검
```

이 지표들이 시간에 따라 어떻게 변하는지를 보면 시스템이 무거워지고 있는지, 또는 최적화가 작동하고 있는지를 알 수 있습니다.

마지막으로, **최적화의 우선순위** 가이드입니다.

```
1. 측정으로 출발 — 어디가 비싼지 알기 전엔 손대지 말 것
2. 정확도 후퇴 없는 변경부터 — 라우팅·캐시·prefetch
3. 정확도 트레이드오프 변경은 평가 데이터셋과 함께
4. 사용자가 느끼는 지연(P95) 우선, P50은 그 다음
5. 비용 한도의 단순 증가는 마지막 수단
```

정리하면, 멀티 에이전트의 비용·지연 최적화는 측정 기반 동인 식별, 모델 라우팅, prompt·response 캐시, 병렬 한도, 스트리밍, 도구 묶음 처리, 정기 모니터링으로 이뤄지며, 정확도 후퇴 없는 변경부터 단계적으로 적용하는 것이 표준입니다.

다음 단원인 5.3.3에서는 그 정확도를 어떻게 측정하고 회귀를 방지할지를 살펴봅니다.
