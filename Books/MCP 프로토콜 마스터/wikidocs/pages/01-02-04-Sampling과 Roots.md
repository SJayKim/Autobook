# 1.2.4 Sampling과 Roots

세 가지 핵심 프리미티브(1.2.1 Tools, 1.2.2 Resources, 1.2.3 Prompts) 외에 MCP는 보조 프리미티브 두 개를 더 정의합니다. **Sampling**과 **Roots**입니다. 두 프리미티브의 공통점은 방향이 반대라는 점입니다. 일반적인 도구·자원·프롬프트는 호스트가 서버에게 요청하지만, Sampling과 Roots는 서버가 호스트에게 요청하거나 호스트가 서버에게 환경 정보를 알려 주는 흐름입니다.

먼저 **Sampling**입니다. 이름이 다소 낯설지만, 한 줄로 표현하면 "서버가 호스트에게 LLM 호출을 요청하는 기능"입니다. 일반적인 흐름과는 정반대입니다. 보통은 호스트가 가진 모델이 도구를 부르지만, 어떤 서버는 자기 일을 처리하기 위해 LLM의 힘을 빌리고 싶을 수 있습니다. 예를 들어 코드 리뷰 서버가 PR에 달린 수십 개 코멘트를 요약해야 한다고 합시다. 서버가 자기 비용으로 LLM API를 따로 호출할 수도 있지만, 그러면 서버 제공자가 API 비용을 부담해야 하고 모델 선택권도 사용자에게서 빼앗기게 됩니다. Sampling은 이 일을 호스트에게 위임하는 표준 방법입니다.

서버는 **sampling/createMessage** 요청을 호스트에 보냅니다. 메시지 본문에는 사용할 메시지 배열, 모델 선호도, 토큰 한도 등이 담깁니다. 호스트는 자신이 보유한 모델 중에서 선호도에 맞는 것을 골라 호출하고, 결과를 서버에게 돌려줍니다.

```json
{
  "method": "sampling/createMessage",
  "params": {
    "messages": [{"role": "user", "content": {"type": "text", "text": "Summarize..."}}],
    "modelPreferences": {"hints": [{"name": "claude-sonnet"}]},
    "maxTokens": 500
  }
}
```

이 흐름의 가장 중요한 안전 장치는 **사용자 승인**입니다. 서버가 호스트의 모델을 마음대로 부르게 두면, 사용자도 모르는 사이에 비용이 발생하거나 의도하지 않은 호출이 일어날 수 있습니다. MCP 사양은 호스트가 sampling 요청을 받으면 사용자에게 확인을 받도록 권고합니다. 실제로 Claude Desktop은 sampling 요청 시 대화창에 메시지를 띄워 사용자의 명시적 승인을 받습니다. 자동화 환경에서는 사용자가 사전에 정책을 설정해 두는 형태가 됩니다.

Sampling은 강력하지만 보안과 사용자 경험상의 부담 때문에 모든 호스트가 지원하지는 않습니다. 호스트는 초기화 시점에 자신이 sampling을 지원하는지를 capabilities로 알립니다. 서버는 이 정보를 보고 sampling이 가능한 경우에만 요청을 보냅니다. 미지원 호스트에서는 자체적으로 LLM을 호출하거나 사용자에게 외부 도움을 요청하는 대안을 마련해야 합니다.

다음은 **Roots**입니다. Root는 호스트가 서버에게 알려 주는 "작업 경계"입니다. 가장 흔한 예는 파일 시스템 경계입니다. 호스트가 "지금 사용자는 이 두 폴더에서 작업 중이다"라고 알려 주면, 서버는 그 경계 안에서만 자원을 노출하고 도구를 실행합니다. 이는 보안 측면에서 매우 중요합니다. 사용자가 폴더 A에서 작업 중인데 서버가 폴더 B의 파일을 임의로 접근하면 곤란하기 때문입니다.

Root는 단순한 URI 목록입니다. 호스트는 초기화 시점에 **roots/list**로 시작 경계를 알리고, 사용자가 작업 폴더를 바꾸면 **notifications/roots/list_changed** 알림을 보냅니다. 서버는 알림을 받으면 roots/list를 다시 호출해 새 경계를 가져옵니다.

```json
{
  "roots": [
    {"uri": "file:///home/alice/project-a", "name": "Project A"},
    {"uri": "file:///home/alice/project-b", "name": "Project B"}
  ]
}
```

Root의 의미는 두 가지로 나뉩니다. 첫째, **의도 표시**입니다. 사용자가 어느 영역에 집중하고 있는지를 알려 줘서 서버가 무엇을 우선 노출할지 판단하게 합니다. 둘째, **권한 경계**입니다. 서버는 root 바깥의 자원을 자발적으로 노출하거나 수정하지 않습니다. 후자는 강제 메커니즘이 아닌 약속에 가깝습니다. 실제 권한은 운영체제, 인증 토큰, gateway 정책 같은 별도 계층이 더 강하게 강제합니다. 그러나 잘 동작하는 MCP 서버는 root를 존중하는 것이 사용자 경험과 신뢰의 기본입니다.

Root는 파일 경로에만 한정되지 않습니다. URI 스킴은 자유이므로, 작업 중인 데이터베이스를 표현하는 `postgres://...`, 작업 중인 노션 페이지를 표현하는 `notion://...` 같은 root도 가능합니다. 호스트는 자기 환경에 맞춰 root를 정의하면 됩니다.

두 프리미티브를 함께 정리하면, Sampling은 서버가 호스트의 LLM을 잠시 빌리는 흐름이고, Roots는 호스트가 서버에게 작업 경계를 알려 주는 흐름입니다. 둘 다 capability로 협상되며, 미지원 환경에서는 우아하게 대안을 찾도록 설계되어 있습니다.

정리하면, MCP의 다섯 프리미티브는 다음과 같이 자리합니다.

```
주도자      이름       역할
모델       Tools       함수 호출 (부수 효과 가능)
호스트     Resources   URI로 식별되는 데이터 읽기
사용자     Prompts     미리 정의된 워크플로 메뉴
서버       Sampling    호스트의 LLM에 메시지 위임
호스트     Roots       서버에 작업 경계 알림
```

이 다섯 가지를 어떻게 조합하느냐가 MCP 서버 설계의 핵심입니다.

다음 단원인 1.3.1에서는 이 프리미티브들이 동작하는 3계층 구조, 곧 Host·Client·Server의 책임 분담을 살펴봅니다.
