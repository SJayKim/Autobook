# 1.2.2 Resources 프리미티브

앞 단원 1.2.1에서 도구는 모델이 호출 의도를 표현하면 서버가 실행하는 함수 호출 표준이라고 했습니다. 그렇다면 데이터를 읽기만 하는 경우는 어떻게 다룰까요. 예를 들어 "프로젝트의 README 파일을 모델에게 컨텍스트로 넣고 싶다"거나 "데이터베이스 테이블 스키마를 참고하게 하고 싶다"는 요구입니다. 이를 도구로도 표현할 수 있지만, MCP는 이런 읽기 중심의 데이터를 별도의 프리미티브로 분리했습니다. 그것이 **Resources**입니다.

**Resource**는 서버가 노출하는 식별 가능한 데이터 조각입니다. 도구와 가장 큰 차이는 두 가지입니다. 첫째, Resource는 부수 효과가 없습니다. 동일한 식별자로 여러 번 읽어도 외부 상태가 바뀌지 않습니다. 둘째, Resource는 **URI**로 식별됩니다. 도구가 "함수 이름"을 가진다면, Resource는 "주소"를 가집니다.

URI는 친숙한 예입니다. `file:///home/user/docs/readme.md`, `postgres://prod/public/users`, `notion://page/abc123` 같은 형태입니다. URI 스킴(scheme)을 서버가 자유롭게 설계할 수 있다는 점이 핵심입니다. 표준 스킴(file·http·https)뿐 아니라 자기 도메인 전용 스킴을 만들어도 됩니다. 예를 들어 회사 위키 서버라면 `wiki://teams/data/onboarding` 같은 URI 체계를 만들 수 있습니다. 호스트 입장에서는 URI가 무엇을 가리키는지를 굳이 해석할 필요가 없습니다. 받은 URI를 그대로 서버에게 돌려주면 그만입니다.

자원을 다루는 메서드는 두 가지입니다. **resources/list**는 서버가 제공하는 자원 목록을 가져옵니다. 각 항목은 uri, name, description, mimeType 같은 메타데이터를 가집니다. **resources/read**는 특정 URI의 내용을 읽습니다. 응답은 contents 배열로 돌아오며, 각 요소는 텍스트나 base64 인코딩된 바이너리를 포함할 수 있습니다.

```json
{
  "contents": [
    {
      "uri": "file:///project/README.md",
      "mimeType": "text/markdown",
      "text": "# My Project\n..."
    }
  ]
}
```

**MIME 타입**은 호스트가 콘텐츠를 어떻게 다룰지를 결정하는 단서입니다. text/markdown이면 모델에게 그대로 텍스트로 전달하기에 적합하고, image/png라면 멀티모달 모델의 비전 입력으로 넘길 수 있으며, application/pdf라면 별도 추출 단계를 거쳐야 할 수도 있습니다. 서버는 MIME 타입을 정확히 표기해 두는 것이 좋습니다. 그래야 호스트가 자원의 종류에 맞게 처리할 수 있습니다.

자원은 동적입니다. 사용자가 새 파일을 만들거나 데이터베이스에 새 테이블이 추가되면 자원 목록이 바뀝니다. 이를 호스트에 알리기 위해 서버는 **notifications/resources/list_changed** 알림을 보냅니다. 또한 특정 자원의 내용 자체가 바뀌는 경우, 호스트는 **resources/subscribe**로 그 자원을 구독해 둘 수 있습니다. 자원이 변경되면 서버는 **notifications/resources/updated** 알림을 보냅니다. 호스트는 이를 보고 자원을 다시 읽거나, 모델에게 알려 자동 갱신을 유도할 수 있습니다.

여기서 도구와 자원의 경계가 한 번 더 분명해집니다. 도구는 부수 효과를 가질 수 있고 인자가 자유로우며, 자원은 부수 효과가 없고 URI로 식별됩니다. 예를 들어 "특정 SQL을 실행해 결과를 돌려주는" 동작은 도구가 맞습니다. 반면 "특정 테이블의 스키마를 알려 주는" 동작은 자원으로 표현하는 편이 자연스럽습니다. 어떻게 표현하든 작동하지만, 표현이 맞을수록 호스트와 모델이 이 데이터를 다루기 쉬워집니다.

자원에는 한 가지 더 미묘한 차원이 있습니다. **자원 노출의 주도권**입니다. 모델이 자원을 직접 골라 읽도록 허용할 수도 있고, 사용자가 UI에서 자원을 선택해 컨텍스트에 첨부하도록 만들 수도 있습니다. Claude Desktop과 같은 호스트는 후자를 채택하여, 사용자가 사이드 메뉴에서 자원을 선택해 메시지에 첨부하는 형태로 자원을 모델에 노출합니다. 이는 동의의 문제와 관련이 있습니다. 어떤 자원이 모델에게 들어갈지 사용자가 명시적으로 통제하기 때문입니다. 서버 입장에서는 호스트가 어떤 모델을 채택하든 같은 자원을 노출하면 되므로, 자원 설계에 영향을 주지는 않습니다.

자원의 또 다른 응용은 **임베디드 리소스**입니다. 도구 호출 결과의 콘텐츠 배열에 자원 참조를 넣을 수 있습니다. 예를 들어 "이 폴더의 큰 파일 목록을 보여 줘"라는 도구가 결과로 파일 콘텐츠 전체 대신 URI 목록만 돌려주는 식입니다. 호스트나 모델은 필요한 경우에만 실제 콘텐츠를 읽어 들이므로 토큰을 절약할 수 있습니다.

리소스 URI를 설계할 때는 두 가지를 의식하면 좋습니다. 첫째, **안정성**입니다. 한 번 노출한 URI는 가능하면 오랫동안 같은 자원을 가리키도록 만들어야 합니다. 자원이 사라지면 URI는 410이나 빈 콘텐츠로 응답하더라도, 다른 자원으로 슬쩍 바꿔치기되지는 않아야 합니다. 둘째, **계층성**입니다. 디렉토리·테이블·페이지처럼 계층 구조가 있는 자원이라면 URI에 경로를 반영해 두는 편이 사람과 모델 모두에게 직관적입니다.

정리하면, Resources 프리미티브는 부수 효과 없이 URI로 식별되는 데이터를 노출하는 표준입니다. resources/list와 resources/read가 기본 메서드이며, list_changed·updated 알림과 subscribe 메커니즘으로 동적 변화를 호스트에 전달합니다. MIME 타입과 임베디드 리소스를 통해 다양한 종류의 콘텐츠를 자연스럽게 다룰 수 있습니다.

다음 단원인 1.2.3에서는 사용자에게 직접 노출되는 워크플로 단위인 Prompts 프리미티브를 살펴봅니다.
