# 4.2.1 Tool allow·deny list

게이트웨이가 단순한 통합만이 아니라 거버넌스의 단일 지점이 되려면, 첫 번째로 갖춰야 할 기능이 **도구 사용을 정책으로 통제**하는 것입니다. 이 단원에서는 도구 allow·deny list의 설계 원칙과 구현 패턴을 정리합니다.

먼저 정책의 기본 결정은 "allow"와 "deny" 중 무엇을 기본값으로 둘 것인가입니다.

```
정책 모델           기본값      추가 시 의미              운영 부담
default-deny        차단        사용 가능한 도구 명시      높음 (목록 갱신 잦음)
default-allow       허용        차단할 도구만 명시        낮음 (위험 신호 의존)
```

**Default-deny**는 보안 모범 사례입니다. 처음에는 아무 도구도 허용되지 않고, 운영자가 검토한 도구만 명시적으로 허용 목록에 들어갑니다. 새 도구를 들이려면 한 단계의 검토가 필요해, 보안 표면이 통제됩니다. 이 모델의 비용은 운영 부담입니다. 새 도구가 등장할 때마다 운영자가 직접 허용 목록에 등록해야 합니다.

**Default-allow**는 사용성을 우선합니다. 모든 도구를 일단 허용하고, 위험하다고 판단된 도구만 차단합니다. 사용자가 새 도구를 빠르게 시도해 볼 수 있지만, 알려지지 않은 위험에 노출될 가능성이 큽니다. 컴플라이언스 요구가 적은 작은 팀에는 적합할 수 있지만, 엔터프라이즈에는 권장되지 않습니다.

대부분의 엔터프라이즈 게이트웨이는 default-deny를 채택하되, 두 가지 보조 장치로 운영 부담을 줄입니다. 첫째, 검증된 회사 표준 도구 묶음을 사전에 허용해 둡니다. 둘째, 사용자가 새 도구를 요청하면 가벼운 승인 워크플로를 통해 빠르게 검토합니다.

다음은 **매칭 단위**입니다. 정책이 어떤 단위로 도구를 식별할지를 결정합니다.

```
매칭 단위              예시                       장단점
도구 이름              wiki__search                정확하지만 prefix 의존
서버 식별자             server=wiki                서버 단위로 묶을 수 있음
태그                   tag=read-only              여러 도구에 공통 정책
도구 서명·해시         hash=abc123                 변조 방지에 강함
```

실전에서는 이 매칭 단위를 **결합**해 사용합니다. 가장 일반적인 패턴은 "서버 단위로 큰 묶음을 허용하되, 특정 위험 도구는 deny로 다시 빼낸다"입니다.

```yaml
# 게이트웨이 정책 예시
policy:
  default: deny
  rules:
    - allow:
        server: wiki
        tags: [read]
    - allow:
        server: wiki
        tools: [create_page]
    - deny:
        tool: wiki__delete_page
    - allow:
        server: github
        scope: required
```

이 정책은 다음을 의미합니다. 위키 서버의 읽기 태그 도구는 모두 허용하고, create_page는 추가로 허용하고, delete_page는 명시적으로 차단합니다. GitHub 서버는 scope가 적절하면 허용합니다.

**평가 순서**가 중요합니다. allow와 deny가 같은 도구에 모두 적용되면 어느 쪽이 이길지를 정해야 합니다. 가장 안전한 정책은 **deny 우선**입니다. 명시적 deny가 있으면 다른 allow가 있더라도 차단합니다. 이 규칙을 따르면 위험 도구를 잠시 차단하는 임시 조치를 신뢰할 수 있습니다.

다음은 **사용자·팀 단위 정책**입니다. 도구 단위 통제만으로는 부족합니다. 같은 도구라도 운영자에게는 허용하고 일반 사용자에게는 차단해야 할 수 있습니다. 사용자·팀·역할 단위 정책을 함께 둡니다.

```yaml
policy:
  default: deny
  groups:
    engineers:
      allow:
        - server: github
        - server: wiki
    finance:
      allow:
        - server: wiki
        - server: accounting
    admins:
      allow:
        - server: '*'
```

사용자의 그룹은 OAuth 토큰의 클레임에서 가져옵니다. 회사 IdP가 그룹 정보를 OIDC ID 토큰에 넣어 주면, 게이트웨이가 그 정보로 정책을 평가합니다(4.4.1에서 자세히 다룹니다).

**정책 갱신과 핫 리로드**도 운영의 핵심입니다. 정책이 한 곳에 모이는 것이 게이트웨이의 큰 가치인데, 그 정책을 갱신할 때 서비스가 중단되면 의미가 줄어듭니다. 다음 기능을 표준으로 갖춥니다.

```
기능                            의미
파일/저장소 기반 정책             Git 같은 VCS로 정책 버전 관리
변경 검출 + 핫 리로드            정책 파일 변경 시 자동 적용
검증 단계                        리로드 전 정책 문법·DAG 검증
롤백                             문제 발견 시 이전 버전으로 즉시 복귀
드라이런                          새 정책이 어떤 호출에 영향을 줄지 사전 분석
```

특히 **드라이런** 모드는 큰 가치를 가집니다. 실서비스에 적용하기 전에, 최근 호출 로그에 새 정책을 적용해 본다면 누가 영향을 받을지를 미리 파악할 수 있습니다. 이 분석 없이 정책을 갱신하면 사용자가 갑자기 도구를 잃어버리는 일이 발생합니다.

**차단 시 사용자 메시지**도 작아 보이지만 중요한 결정입니다. 게이트웨이가 도구를 차단하면 모델은 그 차단 사실을 알 권리가 있습니다. tools/list 응답에서 차단된 도구를 단순 누락시킬 수도 있고, 차단된 도구를 노출하되 호출 시 isError로 거부할 수도 있습니다. 둘 다 장단이 있습니다.

```
차단 모드              모델 관점                        사용자 관점
누락                   존재 자체를 모름                  도구가 없는 것처럼 보임
호출 후 거부          호출하고 isError로 받음            모델이 대안을 시도
정보성 메시지         "이 도구는 정책으로 차단됨"        사용자에게 회사 정책 학습
```

엔터프라이즈에서는 보통 누락보다 **정보성 메시지** 모드를 선호합니다. 사용자가 자신이 무엇을 못 하는지 알아야 회사 정책에 적응할 수 있고, 운영자에게 합리적 요청을 할 수 있기 때문입니다.

**임시 차단**(temporary deny)도 자주 쓰는 패턴입니다. 보안 사고가 의심될 때 특정 도구를 며칠간 차단하는 정책입니다. 시간 기반 조건을 정책에 둘 수 있으면, 자동으로 해제되어 운영 부담이 줄어듭니다.

```yaml
- deny:
    tool: wiki__delete_page
    until: 2026-05-20T00:00:00Z
    reason: "Investigating accidental deletion incident"
```

마지막으로 **정책 자체의 감사**입니다. 누가 언제 어떤 정책을 추가·수정·제거했는지 기록되어야 합니다. 정책 변경이 Git 같은 VCS로 관리되면 자연스럽게 감사 기록이 남습니다. 추가로 사람이 읽을 수 있는 변경 로그(예: "engineer A가 wiki__delete_page를 7일 차단")를 만들어 두면 운영 회고에 도움이 됩니다.

체크리스트로 정리하면 다음과 같습니다.

```
1. default-deny를 기본으로 채택
2. 서버·태그·이름·해시의 결합 매칭 지원
3. deny 우선 평가
4. 사용자·팀·역할 단위 정책 지원
5. 정책의 VCS 관리와 핫 리로드
6. 드라이런과 롤백 지원
7. 차단 시 정보성 메시지 모드 제공
8. 시간 기반 임시 차단 지원
9. 정책 변경 자체의 감사 기록
10. 정기적 정책 검토(쓰지 않는 룰 제거)
```

이 체크리스트가 갖춰지면 도구 사용에 대한 거버넌스의 기본 골격이 완성됩니다.

정리하면, tool allow·deny list는 default-deny와 서버·태그·이름·해시의 결합 매칭, 사용자 단위 정책, 핫 리로드와 드라이런을 기본 구성요소로 가집니다. 정책의 가치는 어떤 호출이 차단되었는가뿐 아니라 그 결정이 어떻게 사용자와 감사 시스템에 전달되는가에 달려 있습니다.

다음 단원인 4.2.2에서는 호출 횟수와 비용을 통제하는 rate limiting과 quota를 살펴봅니다.
