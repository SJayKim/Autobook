# 3.3.2 Confused deputy 공격

앞 단원 3.3.1에서 token passthrough 문제를 살펴보았습니다. 비슷한 결을 가지면서도 다른 모양으로 나타나는 위협이 **Confused deputy** 공격입니다. 이 단원에서는 그 정체와 MCP에서의 변형, 방어책을 정리합니다.

먼저 confused deputy의 고전적 정의를 짚고 가겠습니다. 컴퓨터 보안 문헌에서 confused deputy는 권한을 위임받은 중간자가 자기 권한과 위임자의 권한을 혼동해 비정상적인 행위를 하게 되는 상황을 가리킵니다. 1988년 Norm Hardy의 글에서 처음 명명되었으며, 일상적인 예로 다음과 같이 그릴 수 있습니다. 한 컴파일러 프로그램이 사용자의 코드를 받아 컴파일한 뒤 보고서를 생성하는데, 사용자가 보고서 파일 경로를 자유롭게 지정할 수 있다고 합시다. 컴파일러는 자기가 가진 시스템 권한으로 그 경로에 파일을 씁니다. 만약 사용자가 시스템 설정 파일 경로를 보고서 경로로 지정하면, 컴파일러는 자기 권한으로 시스템 파일을 덮어쓰는 행위를 하게 됩니다. 사용자에겐 그 권한이 없는데도 말입니다. 컴파일러는 자기가 위임받은 사용자가 누구인지를 잊고, 자기 권한으로 사용자가 지정한 위치에 행위를 한 것입니다.

MCP에서 이 패턴은 어떻게 등장할까요. 가장 흔한 변형은 **MCP 게이트웨이**가 등장할 때입니다. 게이트웨이는 여러 MCP 서버 앞에 서서 단일 엔드포인트로 통합해 주는 역할입니다(Phase 4에서 자세히 다룹니다). 게이트웨이는 사용자의 OAuth 인증을 받은 다음, 자기 권한으로 뒤편의 여러 MCP 서버를 호출합니다. 여기서 confused deputy가 발생할 수 있는 두 시나리오를 따져 봅시다.

**시나리오 1: 사용자 식별 누락**. 게이트웨이가 사용자의 토큰으로 자기 자신만 식별하고, 뒤편 서버를 호출할 때는 게이트웨이 자기 자격증명으로 호출한다고 합시다. 게이트웨이는 사용자 A의 요청을 받고도, 뒤편 서버에서는 자기 자격증명이라는 큰 권한으로 일을 합니다. 만약 게이트웨이가 사용자별로 요청을 잘 구분하지 않으면, 사용자 A가 사용자 B의 자원을 만지는 일이 일어날 수 있습니다. 게이트웨이는 자기 권한이라는 큰 도장을 들고, 누구의 요청인지 잊은 채 일을 처리한 셈입니다.

**시나리오 2: Client_id 위장**. 사용자가 게이트웨이를 통해 새 MCP 서버에 합류할 때, 게이트웨이가 클라이언트 등록을 대신 수행합니다. 사용자 입장에서는 한 번의 동의이지만, 게이트웨이는 동일 client_id로 여러 MCP 서버에 한 번에 등록하는 식이 됩니다. 만약 이 client_id가 모든 사용자 사이에 공유되면, 한 사용자의 동의가 다른 사용자에게도 영향을 미칠 가능성이 생깁니다. 또한 AS의 동의 화면에 "Gateway"라는 한 클라이언트만 등장하므로, 사용자는 자신이 어떤 MCP 서버에 권한을 부여하고 있는지 알아차리지 못할 수 있습니다.

**시나리오 3: 사용자 의도 우회**. 사용자는 위키 서버 A에만 동의했는데, 게이트웨이가 그 토큰으로 위키 서버 B의 자원에도 접근하는 경우입니다. 게이트웨이가 두 서버를 같은 audience처럼 다루면 token passthrough의 한 변형이 됩니다.

이 세 시나리오를 한 줄로 요약하면, **위임받은 권한을 자기 식대로 해석해 본래 동의 범위를 벗어나는 행위**입니다. confused deputy의 본질이 그대로 살아 있습니다.

방어책은 세 층에서 옵니다.

첫째, **PKCE의 일관된 적용**입니다. 사용자의 동의 흐름이 어디서 시작되었는지를 PKCE로 확실히 묶어 두면, 게이트웨이가 다른 사용자나 다른 흐름의 토큰을 가져다 끼워 넣는 시도를 어렵게 만듭니다. 3.2.3에서 다룬 verifier·challenge 묶음이 그 보호 장치입니다.

둘째, **audience 검증 강화**입니다. 각 MCP 서버는 자기에게 발급된 토큰만 받아들여야 합니다. 게이트웨이가 뒤편 서버를 호출할 때마다 audience를 그 서버로 좁힌 토큰을 사용해야 합니다. 이는 token exchange(RFC 8693)로 자연스럽게 처리됩니다. 자세한 흐름은 3.3.1에서 다루었습니다.

셋째, **user binding**입니다. 토큰의 sub(주체) 클레임이나 별도의 사용자 식별자를 모든 호출에 함께 전달하고, 각 자원 서버가 그 사용자에 대한 권한을 별도로 확인합니다. 게이트웨이가 자기 권한으로 일하는 게 아니라 사용자를 대리한다는 사실이 호출마다 명시되어야 합니다. 사양은 게이트웨이가 사용자 토큰을 그대로 전달하기를 권하지 않습니다(token passthrough). 대신 사용자 식별을 표준 헤더(`X-User-Id` 같은 합의된 헤더 또는 새 토큰의 sub)에 담아 전달하고, 게이트웨이 자신은 짧은 수명의 자체 토큰으로 인증합니다.

넷째, **개별 동의 노출**입니다. 게이트웨이가 사용자에게 보여 주는 동의 화면이 "Gateway"라는 한 줄로 끝나지 않고, 어떤 MCP 서버들이 묶여 있고 각각 어떤 scope를 가지는지를 분명히 보여 주어야 합니다. 그래야 사용자가 자신이 무엇에 동의하고 있는지 안다는 기본 원칙이 지켜집니다.

다섯째, **클라이언트 분리**입니다. 모든 사용자가 같은 client_id로 묶이지 않고, 가능하면 사용자나 호스트 단위로 분리된 client_id를 발급받아야 합니다. DCR이 가능한 환경이라면 호스트가 자기 단위로 등록을 하면 되고, 그렇지 않다면 사용자 단위 client_id를 발급하는 식으로 분리합니다.

여섯째, **감사 로그**입니다. 게이트웨이가 어떤 사용자의 어떤 요청을 어떤 뒤편 서버에 어떤 audience 토큰으로 호출했는지를 모두 기록해야 합니다. confused deputy의 위협은 정상 흐름과 모양이 같기 때문에 사후 추적이 결정적입니다. 자세한 감사 설계는 4.3.1에서 다룹니다.

```
confused deputy의 변형                  핵심 방어
사용자 식별 누락(자기 권한으로 일함)    user binding, audience 분리
client_id 위장(공유 client)              사용자·호스트 단위 client 분리
사용자 의도 우회(audience 혼동)          token exchange로 audience 좁힘
동의 화면 추상화                          개별 서버·scope 노출
사후 추적 불가                            상세 감사 로그
```

또 한 가지 흥미로운 변형은 **사용자가 직접 만든 호스트**가 일으키는 confused deputy입니다. 사용자가 자기 자동화 스크립트로 게이트웨이를 호출할 때, 그 스크립트는 사용자의 토큰을 사용하지만 게이트웨이는 그 스크립트가 어떤 책임자인지 알 수 없습니다. 스크립트가 잘못 만들어져 의도하지 않은 도구를 호출하면, 사용자가 직접 그 도구를 부른 것처럼 보입니다. 이 경우 책임 추적이 어려워질 수 있으므로, 사용자 단위 자동화도 별도 client_id로 분리해 두는 것이 좋은 운영 습관입니다.

마지막으로 명심할 점은, confused deputy는 표준 한 줄로 막을 수 있는 위협이 아니라는 사실입니다. PKCE, audience 검증, user binding, 동의 노출, client 분리, 감사 로그가 함께 동작해야 합니다. 각각은 단독으로 완전한 보호를 주지 못합니다. 보안은 깊이가 누적되는 식으로 작동하며, MCP 게이트웨이 같은 중간자가 등장하는 환경에서는 이 누적이 특히 중요합니다.

정리하면, confused deputy는 위임받은 권한을 본래 동의 범위 밖으로 확장해 사용하는 위협이며, MCP에서는 게이트웨이가 끼어들 때 가장 자주 등장합니다. PKCE 일관 적용, audience 좁힘, user binding, 동의 노출, client 분리, 상세 감사 로그가 함께 적용되어야 폭발 반경이 좁혀집니다.

다음 단원인 3.3.3에서는 보안 위협의 또 다른 새 영역인 tool poisoning과 prompt injection을 다룹니다.
