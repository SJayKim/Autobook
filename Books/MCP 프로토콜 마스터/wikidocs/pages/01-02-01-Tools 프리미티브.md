# 1.2.1 Tools 프리미티브

MCP 사양은 서버가 호스트에게 노출할 수 있는 자산을 세 가지 핵심 프리미티브로 정리합니다. 도구, 자원, 프롬프트입니다. 여기에 보조 프리미티브인 Sampling과 Roots가 더해집니다. 이 단원에서는 가장 자주 사용되는 **Tools** 프리미티브를 다룹니다.

**Tool**은 모델이 호출할 수 있는 함수입니다. 더 정확히 말하면, 서버가 노출하고 호스트가 모델에게 알려 주는 호출 가능한 작업입니다. 모델은 자신이 직접 도구를 실행하지 않습니다. 모델이 도구를 호출하기로 결정하면, 그 결정은 호스트로 전달되고, 호스트는 MCP 메시지를 통해 서버에게 실제 실행을 위임합니다. 결과는 다시 호스트를 거쳐 모델에게 돌아옵니다. 이 흐름의 핵심은 두 가지입니다. 첫째, 도구의 실행 권한은 서버가 가집니다. 둘째, 도구 호출 여부는 모델이 제안하지만 최종 승인은 호스트가 합니다.

MCP에서 도구를 다루는 메서드는 두 가지입니다. **tools/list**는 서버가 제공하는 모든 도구의 목록을 가져옵니다. 응답에는 각 도구의 이름, 사람이 읽는 설명, 그리고 입력 스키마가 담깁니다. **tools/call**은 특정 도구를 인자와 함께 실행하도록 요청합니다. 응답에는 결과 콘텐츠 배열과 오류 여부가 담깁니다.

도구의 핵심은 **입력 스키마**(inputSchema)입니다. MCP는 입력 스키마를 JSON Schema로 표현합니다. JSON Schema는 객체의 형태를 정의하는 표준으로, 어떤 키가 있는지, 각 값의 타입이 무엇인지, 어떤 키가 필수인지 등을 서술합니다. 모델은 이 스키마를 보고 도구를 호출할 때 어떤 인자를 어떤 형태로 넣어야 하는지를 판단합니다. 따라서 스키마는 정확하고 충분히 풍부해야 합니다. 예를 들어 파일 읽기 도구라면 다음과 같이 정의할 수 있습니다.

```json
{
  "name": "read_file",
  "description": "Read the contents of a file given an absolute path.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "Absolute file path on the local filesystem."
      }
    },
    "required": ["path"]
  }
}
```

스키마에 description을 풍부하게 채워 두면 모델이 도구를 잘못 호출할 가능성이 줄어듭니다. 반대로 description이 모호하면 모델은 추측을 하게 되고, 인자가 어긋난 호출이 늘어납니다. 이는 6장에서 다룰 평가 작업의 큰 부분을 차지합니다.

호출 결과는 **콘텐츠 배열** 형태로 돌려줍니다. 단순 텍스트뿐 아니라 이미지, 임베디드 리소스 등이 같은 배열에 섞일 수 있습니다. 각 요소는 type 필드로 종류를 표시합니다.

```json
{
  "content": [
    {"type": "text", "text": "File contents..."}
  ],
  "isError": false
}
```

여기서 **isError** 플래그가 중요한 역할을 합니다. JSON-RPC 자체의 에러 응답은 프로토콜 수준의 실패(잘못된 메서드, 잘못된 인자 형식 등)에 사용하고, 실행은 성공했지만 결과가 실패인 경우(예: 파일이 없거나 권한이 없거나)는 isError를 true로 두고 정상 응답으로 돌려줍니다. 이 구분은 모델이 오류 메시지를 콘텐츠로 받아 보고 다음 행동을 결정할 수 있게 하기 위함입니다. 프로토콜 에러로 처리해 버리면 모델은 결과를 보지 못하고 호스트의 예외 처리에 묻혀 버립니다.

도구 목록이 동적으로 바뀔 수 있는 점도 중요한 특성입니다. 예를 들어 데이터베이스 서버가 사용자의 권한에 따라 보여 줄 테이블 도구를 다르게 만들 수 있고, 새 플러그인이 로드되면서 새 도구가 추가될 수도 있습니다. 이 변화를 호스트에 알리기 위해 서버는 **notifications/tools/list_changed** 알림을 보냅니다. 호스트는 알림을 받으면 tools/list를 다시 호출해 최신 목록을 가져옵니다. 호스트가 이 능력을 가지고 있음은 초기화 단계의 capability 협상에서 미리 선언됩니다.

도구를 노출하는 서버 측 코드는 SDK마다 다르지만, 본질은 단순합니다. 도구 함수에 이름·설명·스키마를 붙여 서버에 등록하면 됩니다. Python SDK의 FastMCP라면 데코레이터 한 줄로 끝납니다.

```python
@server.tool()
def read_file(path: str) -> str:
    """Read the contents of a file."""
    return Path(path).read_text()
```

SDK가 함수의 시그니처와 docstring으로부터 inputSchema와 description을 자동 생성해 줍니다. 더 정교한 스키마가 필요하다면 Pydantic 모델이나 직접 작성한 JSON Schema를 사용할 수도 있습니다.

도구 설계에서 가장 자주 마주치는 함정은 도구가 너무 크거나 너무 작은 경우입니다. "데이터베이스를 다루는 도구"처럼 너무 크면 인자가 너무 자유로워져 모델이 잘못 호출하기 쉽습니다. 반대로 "select 절을 받아 SQL을 만드는 도구", "where 절을 받아 SQL을 만드는 도구"처럼 너무 잘게 나누면 한 번의 작업에 도구를 여러 번 호출해야 하고 컨텍스트가 늘어납니다. 좋은 도구는 사용자가 직관적으로 떠올릴 하나의 작업 단위와 일치합니다. 예를 들어 "이 폴더의 파일 목록을 보여 줘", "이 파일을 읽어 줘", "이 쿼리를 데이터베이스에 실행해 줘" 같은 단위입니다.

정리하면, Tools 프리미티브는 모델이 호출 의도를 표현하면 서버가 실행 권한을 가지는 형태의 함수 호출 표준입니다. 핵심은 정확한 입력 스키마, 콘텐츠 배열 형태의 응답, isError를 통한 실행 오류 표현, 그리고 list_changed 알림을 통한 동적 갱신입니다.

다음 단원인 1.2.2에서는 도구와 어떻게 다른지에 초점을 맞춰 Resources 프리미티브를 살펴봅니다.
