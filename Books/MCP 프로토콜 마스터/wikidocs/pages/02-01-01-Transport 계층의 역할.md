# 2.1.1 Transport 계층의 역할

Phase 1에서 우리는 메시지의 형식(1.4.1)과 라이프사이클(1.4.2)을 살펴보았습니다. 그런데 그 메시지들은 결국 어떤 통로를 통해 실제로 흐를까요. 표준 입출력일 수도 있고, HTTP일 수도 있고, 어쩌면 WebSocket이나 IPC일 수도 있습니다. 이 통로를 다루는 계층이 **Transport**입니다.

MCP 사양은 시작부터 Protocol과 Transport를 분리해 설계했습니다. **Protocol**은 메시지의 의미와 흐름을 정의합니다. tools/list가 무엇을 가져오는지, initialize가 어떤 정보를 담는지 같은 약속이 여기 속합니다. **Transport**는 그 메시지가 두 프로세스 사이를 어떻게 흘러가는지를 정의합니다. 줄바꿈으로 끊을지, HTTP 본문에 담을지, 어떤 헤더를 붙일지 같은 문제입니다.

이 두 계층의 분리는 시스템 설계에서 매우 흔한 패턴이지만, MCP에서는 특히 중요한 의미를 가집니다. 같은 메시지가 stdio 위에서도, Streamable HTTP 위에서도 똑같이 동작해야 하기 때문입니다. 사양이 새 transport를 추가하더라도 메시지 형식과 라이프사이클은 그대로 재사용됩니다. 반대로 메시지에 새 메서드를 추가해도 transport 구현은 손볼 필요가 없습니다. SDK도 이 경계를 따라가서, transport는 어댑터 형태로 분리되어 있고 protocol 핸들러는 transport와 독립적으로 작성됩니다.

Transport에 요구되는 책임을 정리해 두면 다음과 같습니다.

```
Transport의 책임
1. 메시지 프레이밍   : 한 JSON 객체의 시작과 끝을 식별
2. 양방향성         : 호스트 → 서버, 서버 → 호스트 모두 가능
3. 세션 유지        : 같은 세션 안에서 메시지 순서 보존
4. 인증과 보안      : 필요 시 transport 수준에서 인증·암호화
5. 오류 신호        : 연결 끊김, 시간 초과를 protocol 위로 전달
```

**메시지 프레이밍**은 가장 기본 책임입니다. 메시지는 JSON-RPC 객체 하나가 단위인데, 그 단위를 어디서 끊을지 결정해야 합니다. stdio는 줄바꿈을 구분자로 씁니다. HTTP는 한 요청 본문에 한 객체를 담습니다. WebSocket이라면 메시지 프레임 단위가 그대로 메시지가 됩니다. 어느 방식이든 양쪽이 같은 약속을 따르면 됩니다.

**양방향성**은 protocol이 요구하는 능력입니다. MCP의 메시지는 호스트 → 서버 방향이 다수이지만, 알림이나 sampling은 서버 → 호스트로도 흐릅니다. 따라서 transport는 한쪽에서만 메시지를 보낼 수 있는 단방향 채널이어서는 안 됩니다. stdio는 stdin과 stdout이라는 두 채널을 함께 쓰는 형태로 양방향성을 확보하고, HTTP는 POST와 SSE를 결합해 양방향을 흉내 냅니다. 이 흉내가 만들어 낸 진화의 흐름은 2.1.3부터 2.2.3에 걸쳐 살펴봅니다.

**세션 유지**는 한 transport 연결이 살아 있는 동안 메시지의 순서와 상태를 보존하는 책임입니다. stdio라면 자식 프로세스 자체가 세션이라서 이 책임이 자연스럽게 충족됩니다. HTTP는 본질적으로 무상태 프로토콜이므로 세션을 인공적으로 만들어야 합니다. Mcp-Session-Id 헤더가 그 장치입니다.

**인증과 보안**은 transport마다 다르게 처리됩니다. stdio는 같은 사용자 계정 아래에서 자식 프로세스가 실행되므로 운영체제의 권한 모델에 기대고, 별도 인증이 필요 없는 경우가 많습니다. HTTP는 네트워크를 통한 통신이므로 TLS를 기본으로 하고, OAuth 2.1 같은 별도 인증을 얹습니다. Phase 3 전체가 이 보안 모델을 다룹니다.

**오류 신호** 책임은 transport가 protocol에게 "지금 연결이 끊겼다", "응답이 시간 안에 오지 않았다" 같은 사실을 알리는 역할입니다. SDK는 보통 이를 예외나 콜백으로 노출합니다. 호스트는 받은 신호를 보고 재연결을 시도하거나 사용자에게 알림을 띄울 수 있습니다.

Transport 선택은 운영 환경에 적지 않은 영향을 줍니다. 같은 서버라도 어떤 transport로 노출되어 있느냐에 따라 배포 방식, 권한 모델, 관측 도구가 달라집니다. 예를 들어 stdio 서버는 호스트가 자식 프로세스로 띄우므로 호스트와 같은 머신에 존재해야 합니다. 반면 Streamable HTTP 서버는 원격에 있을 수 있고, 여러 사용자가 공유할 수 있습니다. 이 차이는 다음과 같이 정리할 수 있습니다.

```
                stdio                       Streamable HTTP
배포 위치        호스트와 같은 머신          원격 가능
프로세스 모델    호스트가 자식으로 띄움      별도 서비스
사용자 수        1명 (호스트 사용자)         다수 가능
인증            운영체제 권한               OAuth 2.1
보안 경계       프로세스/파일 권한          네트워크 + 인증
관측            stderr 로그                 표준 HTTP 관측 도구
업데이트 단위    서버 패키지 재설치          원격 배포
```

이 표가 시사하는 점은 transport가 단순히 통로의 선택이 아니라 **운영 모델의 선택**이라는 것입니다. 같은 도구라도 stdio 서버로 배포하면 로컬 도구가 되고, HTTP 서버로 배포하면 SaaS와 비슷한 형태가 됩니다. MCP가 이 두 모델을 같은 protocol로 다룬다는 점이 표준의 큰 가치입니다.

여기서 한 가지 짚어 둘 점은 **transport가 protocol을 흉내 내려고 들지 않는다**는 것입니다. MCP 메시지는 stdio든 HTTP든 같은 JSON-RPC 객체입니다. transport는 그 객체를 옮기는 역할만 합니다. transport 안에 자기만의 인증, 자기만의 메시지 형식을 끼워 넣지 않는 것이 사양의 일관된 방향입니다. 이 일관성 덕분에 transport를 갈아 끼워도 protocol 코드는 그대로입니다.

마지막으로, 새 transport를 만들고 싶다면 어떻게 해야 할까요. 사양은 transport 추가를 막지 않습니다. JSON-RPC 객체를 양방향으로 흘릴 수 있는 어떤 채널이라도 후보가 될 수 있습니다. 예를 들어 WebSocket transport를 정의하거나, gRPC 위에 MCP를 얹는 시도가 가능합니다. 다만 사양으로 합의되지 않은 transport는 상호운용성이 떨어집니다. 호환되는 호스트가 없으면 결국 자신만의 생태계가 됩니다. 그래서 새 transport는 모두 표준으로 합의된 뒤에야 본격적으로 쓰이기 시작합니다.

정리하면, Transport 계층은 메시지 프레이밍, 양방향성, 세션 유지, 인증, 오류 신호를 책임지면서 protocol과 분리되어 있습니다. 이 분리 덕분에 MCP는 같은 메시지로 stdio·HTTP+SSE·Streamable HTTP 같은 서로 다른 운영 모델을 자연스럽게 다룰 수 있습니다.

다음 단원인 2.1.2에서는 가장 단순한 transport인 stdio를 자세히 살펴봅니다.
