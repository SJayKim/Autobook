# 5.3.3 평가와 회귀 방지

멀티 에이전트 시스템은 변경의 영향이 예측하기 어렵습니다. 한 도구의 description을 바꿨더니 다른 에이전트의 호출 패턴이 달라지고, 모델을 새 버전으로 올렸더니 supervisor의 위임이 미묘하게 흔들리는 식입니다. 이 단원에서는 그 변화를 추적하고 회귀를 막기 위한 평가 체계를 정리합니다.

먼저 평가의 두 축을 구분합니다.

```
축                       의미
오프라인 평가             고정된 데이터셋과 골든 응답으로 일괄 평가
온라인 평가              실제 사용자 트래픽에서 지표 추적
```

오프라인 평가는 변경 직후 즉시 검증할 수 있는 검사이고, 온라인 평가는 실제 사용 패턴을 반영합니다. 둘 다 필요합니다. 오프라인이 빠른 회귀 탐지를 주고, 온라인이 실제 효용을 검증합니다.

다음은 **End-to-end 평가 데이터셋**입니다. 전체 시스템을 한 묶음으로 보고 입력·기대 출력 쌍을 모아 둡니다.

```
필드                              의미
input                             사용자 메시지
context                           초기 상태(필요 시)
expected_output                   기대 응답 또는 의미적 기준
expected_tool_calls               (선택) 기대 도구 호출 패턴
labels                            카테고리, 난이도
acceptance_criteria               합격 기준
```

End-to-end 데이터셋은 시스템 전체의 정확도를 측정하지만, 어디서 실패했는지를 좁히기 어렵다는 약점이 있습니다. 그 약점을 보완하기 위해 **도구별 unit 평가**와 **subgraph 평가**를 함께 둡니다.

**Unit 평가**는 도구별로 입력·기대 결과를 모아 둡니다. 도구가 외부 API를 호출하지 않고도 검증할 수 있도록 mock·fixtures를 함께 둡니다. 도구의 inputSchema에 맞춘 다양한 인자가 들어왔을 때 적절히 동작하는지를 확인합니다.

**Subgraph 평가**는 5.1.3에서 다룬 subgraph 단위로 입력·기대 출력을 평가합니다. 한 subgraph만 따로 떼어내어 평가할 수 있다면, 변경의 영향이 그 subgraph에 한정되었는지를 빠르게 확인할 수 있습니다.

```
평가 계층

  End-to-end 평가
       ↑ (사용자 시나리오)
  Subgraph 평가
       ↑ (책임 단위)
  Unit 평가
       ↑ (도구·메서드 단위)
```

이 세 계층이 함께 갖춰지면 회귀의 출처를 빠르게 찾을 수 있습니다.

다음은 **평가 방법**입니다. LLM 출력의 평가에는 세 가지 흔한 접근이 있습니다.

```
접근                       장점                              단점
정답 비교                  결정적·빠름                       자연어에는 어려움
규칙 기반                  설명 가능                          제한된 차원
LLM-as-a-judge             다양한 차원 평가 가능              비결정성·편향·비용
```

정답 비교는 분류·추출 작업에 적합합니다. 모델 응답에 특정 식별자나 숫자가 들어 있어야 하는 경우, 단순한 문자열 비교로 충분합니다.

규칙 기반은 형식·존재 검사에 적합합니다. "응답이 markdown 형식인가", "특정 섹션 헤더가 포함되어 있는가" 같은 기준을 코드로 검사합니다.

LLM-as-a-judge는 자연어 품질을 평가할 때 유용하지만 몇 가지 함정이 있습니다.

```
함정                                     완화
판정 모델의 편향                          여러 판정 모델 사용·평균
길이·스타일 편향                           통제된 기준 명시
같은 모델 가족 자기 우대                  다른 가족 모델을 판정자로
판정의 비결정성                            여러 회 평가하여 합의
판정 비용 누적                             샘플링 비율 조절
```

판정의 비결정성을 완화하는 가장 단순한 방법은 평가 자체를 여러 번 돌려 평균을 보는 것입니다. 다만 비용이 누적되므로, 정기 평가에서는 정답 비교를 우선하고 LLM-as-a-judge는 자연어 품질이 핵심인 일부 항목에만 적용하는 것이 일반적입니다.

다음은 **Trajectory 평가**입니다. 멀티 에이전트의 큰 특징은 결과뿐 아니라 과정도 평가 대상이라는 점입니다. 같은 결과라도 도구 호출 순서, 에이전트 간 메시지 흐름이 달라질 수 있습니다.

```
Trajectory 평가의 검사 항목
- 기대 도구가 호출되었는가
- 호출 순서가 의미 있게 정렬되어 있는가
- 불필요한 도구 호출이 없는가
- 에이전트 간 메시지가 의도된 흐름인가
- 종료가 깨끗하게 일어났는가
```

이 평가는 시스템의 효율을 들여다보는 창입니다. 결과는 같지만 호출 수가 늘어났다면, 그 변경은 회귀로 분류되어야 합니다.

다음은 **회귀 탐지 워크플로**입니다.

```
1. 변경(코드·프롬프트·모델·정책) 발생
2. CI에서 자동 평가 트리거
3. 평가 결과를 기준 라인과 비교
4. 통계적 유의 차이가 있으면 차단
5. 사람이 회귀를 검토하고 승인/반려
6. 승인된 변경은 점진적 배포(canary)
7. 온라인 지표 확인 후 전체 배포
```

CI에 평가를 통합할 때 주의할 점은 평가가 비결정적이라는 점입니다. 같은 입력에 다른 점수가 나올 수 있으므로, **임계 설정**이 필요합니다.

```
정책                                   임계
한 항목당 점수가 평균보다 N표준편차 낮음   차단
전체 평균이 K% 이상 하락                  차단
신규 카테고리에서만 회귀                   카테고리별 차단
지연이 X배 증가                            성능 회귀로 분리 보고
```

다음은 **prompt injection·tool poisoning 회귀**입니다. Phase 3에서 다룬 새 위협들은 시간이 지나며 변형됩니다. 평가 데이터셋에 의도적인 공격 시나리오를 포함시키면, 모델이나 정책이 흔들렸을 때 즉시 감지할 수 있습니다.

```
공격 시나리오 예시
- 도구 description에 "ignore previous instructions" 삽입
- 자원 콘텐츠 안에 외부로 데이터 전송하라는 지시
- 모델이 자기 시스템 프롬프트를 노출하도록 유도
- rate limit 초과 후 다른 사용자 계정 사용 시도
```

이 시나리오들에 대해 시스템이 안전하게 거부하는지를 평가합니다. 새 위협이 발견될 때마다 데이터셋에 추가합니다.

다음은 **릴리스 게이팅 기준**입니다.

```
영역                       기준 예시
End-to-end 정확도          기준 라인 대비 0~+2%
Trajectory 효율           평균 호출 수 0~+5%
지연 P95                  기준 라인 대비 0~+10%
비용                       기준 라인 대비 0~+10%
보안 시나리오 통과율       100% 유지
모델 안전 평가             기준 통과
```

이 게이트를 모두 통과해야 다음 릴리스로 진행할 수 있도록 설정해 두면, 작은 회귀도 운영 환경에 들어가기 전에 잡힙니다.

다음은 **점진적 배포**입니다. CI 평가를 통과해도 운영 환경에 한 번에 전체 적용하기는 부담스럽습니다. 단계적 노출이 안전합니다.

```
1. 내부 테스트 사용자에게만 (1~2일)
2. 5% 트래픽에 노출 (canary)
3. 50% 트래픽
4. 100% 트래픽
```

각 단계에서 4.3.2의 비용·지연 지표와 새 알림을 확인합니다. 회귀가 보이면 직전 단계로 롤백합니다.

다음은 **온라인 평가 지표**입니다. 운영 트래픽에서 정기적으로 추적합니다.

```
지표                                  의미
도구 호출당 isError 비율               도구 안정성
fan-in 통합 후 사용자 만족 점수        품질 (사용자 피드백)
재실행 비율 (사용자가 같은 질문 반복)  품질 신호
사용자 명시 피드백 (별점, thumbs)      직접 신호
escalation 발생률 (사람에게 넘김)      복잡도 신호
```

온라인 지표는 오프라인 평가가 놓친 실제 사용 패턴의 회귀를 잡아 줍니다.

마지막으로, 평가 체계가 **살아 있게 유지**하기 위한 의식입니다.

```
의식                                          주기
새 사용자 시나리오를 데이터셋에 추가          매주
실패한 운영 케이스를 데이터셋에 추가          즉시
평가 점수의 추세 리뷰                          매월
판정 모델·임계 점검                            분기
오프라인·온라인 지표의 상관 분석              분기
```

이 의식들이 동작하면 평가 데이터셋이 시간에 따라 풍부해지고, 시스템이 실제 사용 패턴을 더 잘 반영하는 검사로 발전합니다.

체크리스트로 마무리합니다.

```
1. End-to-end·subgraph·unit 세 계층 평가
2. 정답·규칙·LLM-judge의 적절한 조합
3. Trajectory 평가로 효율 검증
4. CI 자동 평가와 회귀 차단 임계
5. 보안 시나리오를 데이터셋에 상시 포함
6. 릴리스 게이트와 점진적 배포
7. 온라인 지표 모니터링
8. 정기 평가 의식
9. 사용자 피드백 채널과 데이터셋 갱신 흐름
10. 평가 자체의 비용·결정성 관리
```

정리하면, 멀티 에이전트의 평가와 회귀 방지는 세 계층의 데이터셋, 정답·규칙·LLM-judge의 결합, trajectory 효율 평가, CI 자동 게이트, 점진적 배포, 온라인 지표 모니터링, 정기 갱신 의식으로 구성됩니다. 이 체계가 살아 있어야 시스템이 변경에 강해지고 사용자 신뢰를 유지할 수 있습니다.

다음 단원인 5.3.4에서는 MCP 생태계의 진화 방향과 우리가 무엇을 따라가야 할지를 살펴봅니다.
