# 3.1.1 왜 MCP에 OAuth가 필요한가

Phase 2에서 우리는 원격 MCP 서버가 Streamable HTTP 위에서 어떻게 동작하는지 살펴보았습니다. 원격으로 서버를 노출한다는 결정은 한 가지 새 문제를 만듭니다. 사용자가 누구인지, 그리고 그 사용자가 무엇을 할 권한이 있는지를 확실히 가려야 한다는 것입니다. 이 Phase 전체는 그 문제에 대한 사양의 답인 **OAuth 2.1**과 **Dynamic Client Registration**을 다룹니다. 이 단원은 출발점으로, 왜 단순한 API 키 방식이 MCP에 잘 맞지 않는지를 정리합니다.

가장 단순한 인증은 정적인 **API 키**입니다. 서버를 만들 때 키를 발급하고, 사용자는 그 키를 환경 변수로 두고 클라이언트가 헤더에 실어 보냅니다. 작은 stdio 서버에서는 이 모델이 충분히 작동합니다. 사용자가 자기 머신에서 자기 자격증명으로 자기 자원을 다루는 시나리오라면, OS의 권한 모델이 이미 큰 보호막이고 API 키는 그 위에서 한 단계 인증을 더할 뿐입니다.

문제는 **원격 멀티유저** 환경에서 시작됩니다. 사내 위키 서버를 MCP로 노출한다고 합시다. 한 직원의 권한으로 다른 직원의 비공개 문서를 보면 안 됩니다. 정적 API 키 방식으로 이 구분을 하려면 사용자마다 별도 키를 발급해야 하는데, 그 발급 과정 자체가 별도의 인증 시스템 없이는 표준화되지 않습니다. 사용자가 발급받은 키를 어디에 저장할지, 한 사용자가 여러 호스트에서 같은 키를 사용해도 될지, 키가 유출되면 어떻게 회수할지 같은 결정이 모든 서버에서 흩어집니다.

또 한 가지 문제는 **사용자 동의**입니다. 모델이 사용자의 위키 문서를 읽거나 PR에 댓글을 다는 행위는, 사용자가 그 행위를 허용했다는 명시적 동의가 있어야 합니다. 정적 API 키는 한 번 발급되면 만료되기 전까지 무차별로 사용 가능합니다. 사용자가 "이번 호스트에는 읽기만 허용한다", "이 도구는 쓰기까지 허용한다" 같은 세분된 동의를 표현하기 어렵습니다.

세 번째는 **권한의 최소화**입니다. 좋은 보안의 기본은 최소 권한 원칙입니다. 도구가 PR 댓글을 다는 데만 쓰인다면 그 도구에는 댓글 작성 권한만 부여되어야 합니다. 정적 API 키는 일반적으로 사용자의 전체 권한을 그대로 들고 다닙니다. 한 도구에 키가 새면 사용자의 모든 자원이 위험에 노출됩니다.

네 번째는 **회수와 갱신**입니다. 보안 사고가 발생했을 때 키를 회수하려면 사용자가 그 사실을 알고 직접 키를 재발급해야 합니다. 모든 호스트와 서버에 그 키가 박혀 있다면 갱신 비용도 큽니다. OAuth의 refresh token 회전이나 토큰 폐기 메커니즘 같은 표준이 없으면 운영이 무거워집니다.

다섯 번째는 **다양한 호스트**입니다. 사용자는 Claude Desktop, Cursor, VS Code, 자체 자동화 봇 등 여러 호스트에서 같은 서버를 쓰고 싶을 수 있습니다. 각 호스트마다 API 키를 따로 받아 설정해야 한다면 사용성이 매우 떨어집니다. 게다가 호스트마다 키 저장 방식이 다르면 보안 표준화도 어려워집니다.

이 다섯 문제를 한 가지 표준 위에서 풀고자 하는 것이 **OAuth 2.1** 채택의 동기입니다. OAuth는 처음부터 다음을 가정합니다.

```
- 사용자(자원의 주인)와 클라이언트(앱)와 자원 서버(서비스)는 서로 다르다.
- 사용자는 클라이언트에게 자기 자원의 일부를 위임할 수 있다.
- 위임은 명시적이고, 범위가 좁고, 시간 제한이 있고, 회수 가능하다.
```

이 가정은 MCP의 상황과 정확히 맞아 떨어집니다. 사용자는 자기 위키 문서의 주인이고, Claude Desktop은 그 문서를 모델에 보여 주려는 클라이언트이고, MCP 서버는 자원 서버입니다. 사용자가 한 번 위임하면 클라이언트는 시간 제한이 있는 토큰으로 좁은 범위의 일을 할 수 있습니다.

MCP는 OAuth 2.0이 아니라 **OAuth 2.1**을 골랐습니다. 2.1은 2.0의 흩어진 권고와 폐기 결정을 한 문서로 모은 통합 사양입니다. 2.0이 허용하던 위험한 흐름(implicit flow, resource owner password credentials)은 폐기되었고, PKCE는 사실상 필수가 되었고, refresh token 회전이 권장됩니다. 2.0의 모범 사례를 굳혀 둔 셈입니다. 3.1.2에서 자세히 다룹니다.

여기에 MCP만의 특별한 요구가 하나 더 있습니다. **클라이언트 수가 무한히 많다**는 점입니다. OAuth는 보통 클라이언트가 사전에 등록되어 client_id와 비밀을 받는 모델입니다. 그러나 MCP에서는 새 호스트가 끊임없이 등장하고, 한 사용자가 자기만의 호스트를 만들 수도 있습니다. 모든 호스트를 모든 서버에 사전 등록하는 것은 비현실적입니다. 이 문제를 해결하는 것이 **Dynamic Client Registration**(DCR)입니다. 3.2에서 자세히 다룹니다.

사양의 채택 시점도 함께 짚어 두면 좋습니다. 첫 사양인 2024-11-05는 인증을 명시적으로 표준화하지 않았고, 각 서버가 자기 방식으로 처리했습니다. **2025-03-26 사양**부터 OAuth 2.1이 정식 채택되었고, **2025-06-18 사양**에서는 protected resource metadata, resource indicator, audience binding 같은 보안 항목이 더 다듬어졌습니다. 가장 최근 사양에서는 OAuth가 원격 MCP의 기본 인증으로 자리잡았으며, 일부 호스트(예: 사내 자동화 봇)는 OAuth 클라이언트 자격증명 흐름 같은 변형을 함께 사용합니다.

OAuth가 만능 해결책은 아닙니다. 비용이 따릅니다. 첫째, 흐름이 복잡합니다. 정적 API 키는 환경 변수 하나면 되지만, OAuth는 여러 단계의 리다이렉트와 토큰 교환이 필요합니다. 둘째, Authorization Server를 운영해야 합니다. 자체 사용자 시스템이 있다면 자기 IdP를 쓰면 되지만, 그렇지 않다면 외부 IdP(Auth0·Okta·Keycloak)와 통합해야 합니다. 셋째, 사용자 동의 UI를 만들거나 호스트의 동의 흐름과 통합해야 합니다.

이 비용을 감수하는 이유는 분명합니다. 위에서 열거한 다섯 문제를 한 표준으로 풀고, 그 표준 위에 감사·정책·SSO 같은 더 상위 운영 기능을 쌓을 수 있기 때문입니다. 한 번 표준을 적용해 두면 모든 MCP 서버·호스트·게이트웨이가 같은 언어로 인증을 이야기할 수 있습니다.

마지막으로, OAuth가 모든 MCP에 강제되는 것은 아니라는 점을 짚어 둡니다. stdio 서버, 신뢰된 내부망의 서버, 단일 사용자 SaaS 같은 경우는 단순 API 키나 mTLS로도 충분할 수 있습니다. 사양은 OAuth를 원격·다중 사용자 시나리오의 표준 답으로 제시하면서도, 사용자가 자기 상황에 맞춰 다른 방법을 선택할 자유를 남겨 두었습니다.

정리하면, 원격 멀티유저 MCP에서 정적 API 키가 풀지 못하는 다섯 문제(사용자 식별, 동의, 최소 권한, 회수·갱신, 다양한 호스트)를 한 표준으로 다루기 위해 OAuth 2.1이 채택되었고, 이를 보완하기 위해 Dynamic Client Registration이 함께 표준이 되었습니다.

다음 단원인 3.1.2에서는 OAuth 2.1이 OAuth 2.0과 어떻게 다른지, 채택된 핵심 변경을 살펴봅니다.
