# 4.1.2 멀티 서버 통합 아키텍처

앞 단원에서 게이트웨이의 책임을 살펴보았으니, 이 단원에서는 게이트웨이가 실제로 여러 서버를 하나로 묶을 때 어떤 패턴을 쓰는지를 정리합니다. 세 가지 대표 패턴이 있습니다. **Aggregator**, **Router**, **Federation**입니다.

먼저 **Aggregator 패턴**입니다. 가장 단순한 통합 방식으로, 게이트웨이가 뒤편 모든 서버의 도구·자원·프롬프트 목록을 미리 가져와 자기 목록으로 합쳐 보여 줍니다. 호스트가 tools/list를 호출하면 게이트웨이는 합쳐진 목록을 한 번에 돌려줍니다. 도구를 호출하면 게이트웨이가 어느 뒤편 서버로 보낼지를 결정해 위임합니다.

이때 가장 자주 마주치는 문제는 **이름 충돌**입니다. 두 서버가 동시에 `search`라는 도구를 노출하면, 합쳐진 목록에 두 개의 `search`가 들어가 모델이 어느 것을 부르려는지 알기 어렵습니다. 표준적인 완화책은 **prefix**를 붙이는 것입니다.

```
원래 서버의 도구           게이트웨이가 노출
wiki:search                wiki__search 또는 mcp.wiki.search
slack:search               slack__search
github:search              github__search
```

prefix는 게이트웨이 설정으로 결정되며, 같은 서버의 도구들은 같은 prefix를 가집니다. 사용자 동의 화면이나 모델 컨텍스트에서 이 prefix가 출처를 명확히 알려 줍니다.

Aggregator는 단순하지만 두 가지 비용을 가집니다. 첫째, **도구 목록이 비대해질 수 있습니다.** 모든 서버의 모든 도구가 하나의 목록에 들어가면 수십 개~수백 개의 도구가 모델 컨텍스트에 동시에 노출됩니다. 이는 컨텍스트 비용을 늘리고 모델의 선택 정확도를 떨어뜨립니다. 둘째, **모든 서버가 항상 연결되어 있어야 합니다.** 어느 한 서버가 응답하지 않으면 목록 일부가 빠지거나 게이트웨이의 응답이 지연됩니다.

```
Aggregator 패턴

  Host --- Gateway
              |
       +------+------+
       |      |      |
       v      v      v
       S1     S2     S3
       (모두 미리 연결되어 도구 목록을 합침)
```

두 번째 패턴은 **Router**입니다. 게이트웨이가 도구 목록을 단순 합치는 대신, 사용자가 어떤 작업을 하려는지에 따라 적합한 서버를 한 둘만 활성화하는 방식입니다. 호스트가 보내는 컨텍스트(사용자 메시지, 활성 폴더 등)를 분석해 라우팅 결정을 내립니다.

이 결정은 두 단계로 갈라집니다.

```
정적 라우팅                              동적 라우팅
사용자 메타데이터로 결정                   사용자 메시지로 결정
예: 사용자가 '엔지니어링' 팀이면          예: '이슈를 만들어 줘' 메시지에서
    GitHub 서버를 우선 활성화                GitHub 서버 활성화
사전 정의된 규칙으로 처리                  ML/룰 기반 분류기 사용
간단하고 빠름                              새 도메인 자동 적응
```

Router 패턴은 도구 목록을 좁혀 모델 컨텍스트와 비용을 절약합니다. 그러나 라우팅 결정이 잘못되면 사용자가 필요로 하는 도구가 보이지 않게 됩니다. 결정의 정확도가 곧 사용성으로 직결됩니다.

```
Router 패턴

  Host --- Gateway --- (router)
                          |
                     [선택된 서버만]
                     v
                    S2
                     ↑
                S1   S3  (잠시 비활성)
```

세 번째 패턴은 **Federation**입니다. 게이트웨이가 모든 서버를 미리 알 필요가 없는 형태입니다. 새 서버가 등록되면 게이트웨이가 그 서버의 도구·자원·프롬프트 메타데이터를 자동으로 발견하고, 이후 호출이 들어올 때 라이브로 위임합니다. 등록 자체는 자동화될 수 있습니다(예: 회사 내 디렉터리 서비스에 등록된 서버를 게이트웨이가 주기적으로 폴링).

이 패턴은 큰 조직에서 가치를 발휘합니다. 팀마다 자기 서버를 만들고 디렉터리에 등록만 하면, 게이트웨이가 자동으로 그 도구들을 사용자에게 노출합니다. 도구 카탈로그가 자율적으로 성장하지만, 동시에 정책·관측·이름 충돌 관리는 게이트웨이가 책임지기 때문에 일관성이 유지됩니다.

```
Federation 패턴

  Server registry
       ↓ (등록)
  Gateway가 주기적 발견
       ↓
  Host --- Gateway --- 동적으로 묶인 서버 풀
```

세 패턴은 배타적이지 않습니다. 한 게이트웨이가 자주 쓰는 핵심 서버는 Aggregator로 묶고, 도메인 특화 서버는 Router로 컨텍스트에 따라 활성화하고, 팀별 자율 서버는 Federation으로 자동 발견하는 식의 혼합이 흔합니다.

이제 게이트웨이가 통합 시 결정해야 할 항목들을 정리합니다.

**이름 충돌 해결**. prefix·alias·우선순위 중 한 가지를 채택합니다. prefix가 가장 단순하고 명시적입니다.

**도구 목록 캐싱**. 매 tools/list마다 뒤편 서버에 실시간 질의하면 지연이 큽니다. 일정 시간(예: 30초) 캐싱하거나, 서버의 list_changed 알림을 받아 갱신합니다. 캐시가 신선하지 않으면 사용자가 새 도구를 보지 못할 수 있으므로, 알림 기반 무효화가 더 안전합니다.

**부분 장애 격리**. 뒤편 서버 하나가 응답하지 않을 때 게이트웨이가 어떻게 반응할지를 정합니다. 흔한 정책은 다음과 같습니다.

```
시나리오                          정책
tools/list 시 일부 서버 미응답    응답한 서버 목록만 돌려주고 누락 로깅
tools/call 시 대상 서버 미응답    isError와 함께 "service unavailable" 응답
긴 응답 시간                       circuit breaker로 일시 차단
```

**선택적 노출**. 게이트웨이는 뒤편 서버의 모든 도구를 그대로 노출할 필요가 없습니다. 정책에 따라 일부 도구만 노출하거나, 사용자 권한에 따라 도구 목록을 달리 보일 수 있습니다. 이 결정은 4.2의 정책 엔진에서 다룹니다.

**프롬프트와 자원의 통합**. 도구뿐 아니라 prompts·resources도 통합 대상입니다. 같은 prefix 정책을 자원 URI에도 적용해야 충돌이 없습니다. 예를 들어 `wiki://...`와 `notion://...` 같은 스킴은 자연스럽게 충돌이 적지만, 같은 스킴을 쓰는 두 서버라면 게이트웨이가 prefix 경로를 붙여 구분합니다.

**Sampling 전달**. 뒤편 서버가 sampling/createMessage를 보내면, 게이트웨이는 그것을 호스트로 그대로 전달합니다. 호스트의 답변은 다시 뒤편 서버로 돌려보냅니다. 이때 sampling 요청에 user binding 정보를 함께 전달해, 어떤 사용자의 요청인지 호스트가 알 수 있게 합니다.

**Roots 처리**. 호스트가 roots/list_changed 알림을 보내면, 게이트웨이는 이를 모든 뒤편 서버에 전파합니다. 각 서버가 자기 자원 노출을 root에 맞게 좁힐 수 있도록 합니다.

이 모든 결정을 한 곳에 집중하면 게이트웨이 코드가 복잡해질 수 있습니다. 실제 구현에서는 책임을 더 잘게 나눠 다룹니다. 라우팅 모듈, 정책 모듈, 캐시 모듈, 관측 모듈이 서로 인터페이스로만 만나도록 분리합니다. 새 서버가 추가될 때 그 서버 어댑터만 추가하면 다른 모듈에 영향이 없는 구조가 이상적입니다.

다음 표는 어떤 시나리오에 어떤 패턴이 가장 잘 맞는지를 정리합니다.

```
시나리오                                      적합한 패턴
회사 표준 도구 5~10개를 모든 직원에 노출       Aggregator
도메인이 매우 다양해 컨텍스트 비용 부담         Router
팀별 자율 서버가 빠르게 생성·폐기됨             Federation
도구·자원 모두 표준화된 prefix로 묶고 싶음      Aggregator + prefix
사용자의 활성 폴더에 따라 도구 활성화           Router
```

각 패턴은 4.1.3에서 다룰 실제 구현(Anthropic·Cloudflare·IBM·Microsoft)에 다양한 모습으로 녹아 있습니다.

정리하면, 멀티 서버 통합은 Aggregator(전체 합침), Router(동적 선택), Federation(자동 발견) 세 패턴으로 정리할 수 있으며, 이름 충돌 해결, 캐싱, 장애 격리, 정책 통합이 공통 결정 포인트입니다. 한 게이트웨이가 세 패턴을 혼합해 쓰는 것이 일반적입니다.

다음 단원인 4.1.3에서는 대표적인 게이트웨이 구현들의 설계 선택을 비교합니다.
