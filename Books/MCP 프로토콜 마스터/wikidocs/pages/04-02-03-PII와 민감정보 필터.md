# 4.2.3 PII와 민감정보 필터

게이트웨이가 다루는 중요한 데이터 흐름이 하나 더 있습니다. **민감정보의 입출력**입니다. 사용자가 도구를 호출할 때 의도하지 않게 개인정보가 포함될 수 있고, 도구가 돌려주는 결과에 비밀번호·카드 번호 같은 정보가 섞일 수 있습니다. 이 단원에서는 PII(개인식별정보)와 민감정보를 게이트웨이에서 탐지·마스킹·차단하는 방법을 정리합니다.

먼저 PII와 민감정보의 범위를 짚어 둡니다.

```
범주                     예시
직접 식별자              주민등록번호, 여권번호, 이메일, 전화번호
간접 식별자              생년월일 + 우편번호 + 성별 조합
민감 카테고리            건강 정보, 종교, 정치적 견해, 성적 지향
인증·결제 정보          비밀번호, OTP, 카드 번호, 계좌
회사 비밀                내부 코드, 미공개 매출, 인사 정보
```

이 범주들은 규제와 운영 정책 양쪽에서 다뤄야 합니다. 한국의 개인정보보호법, 유럽의 GDPR, 미국의 HIPAA 같은 규제는 직접 식별자와 민감 카테고리에 강한 통제를 요구합니다. 회사 비밀은 규제 대상은 아니어도 보안 사고를 막기 위한 내부 정책 대상입니다.

다음은 **탐지 방법**입니다. 두 가지 접근이 있습니다.

```
방법                  장점                      한계
정규식 기반          빠르고 결정적              패턴이 정형화된 데이터만
NER(개체명 인식)     문맥 이해                  비용·지연·false positive
```

정규식은 카드 번호(`\b\d{4}-?\d{4}-?\d{4}-?\d{4}\b`)나 이메일처럼 형식이 분명한 데이터에 잘 동작합니다. 반면 사람 이름, 주소, 회사 내부 식별자 같은 데이터는 정규식만으로 잡기 어려워 NER 모델이 필요합니다.

실전 게이트웨이는 두 방법을 **계층적**으로 결합합니다. 빠른 정규식 검사를 먼저 적용하고, 더 깊은 분석이 필요한 텍스트에만 NER 모델을 돌립니다. 모든 호출에 NER를 적용하면 지연과 비용이 누적되므로, 정책에서 명시한 민감 도구나 외부 노출 경로에만 강한 검사를 적용하는 식의 차등 적용이 일반적입니다.

다음은 **처리 정책**입니다. 민감정보가 탐지되었을 때 어떻게 할지 정합니다.

```
정책                  의미                                     적용 예
차단(block)           호출을 거부                                결제 정보가 평문으로 외부 도구에 흐를 때
마스킹(mask)          민감 부분을 ***로 대체                     로그 저장 시
대체(redact)          [REDACTED] 같은 표시로 대체                결과를 모델에 전달할 때
변환(tokenize)         일관된 토큰으로 대체 후 복원              회사 내부 식별자만 마스킹하고 결과는 복원
허용 후 경고           통과하되 경고 로그                        false positive 가능성이 높은 경우
```

정책 선택은 데이터의 흐름 방향에 따라 달라집니다. 사용자 → 도구 방향에서 비밀번호가 발견되면 차단하는 것이 안전하고, 도구 → 사용자 방향에서 외부 자원 콘텐츠에 우연히 들어 있는 개인정보는 마스킹이 자연스럽습니다.

```
흐름                            기본 정책
사용자 입력 → 외부 도구          차단(또는 강한 경고)
도구 결과 → 모델 컨텍스트       마스킹·대체
도구 결과 → 사용자 UI            정책 따라 다름
모든 흐름 → 감사 로그            마스킹(항상)
```

세 번째 항목은 **isError 처리**입니다. 게이트웨이가 입력 단계에서 차단한 경우, 모델에게 결과를 어떻게 전달할지를 결정합니다. 빈 결과로 침묵하면 모델은 도구가 동작하지 않았다고 오해할 수 있습니다. 표준 패턴은 다음과 같습니다.

```json
{
  "isError": true,
  "content": [{
    "type": "text",
    "text": "Policy blocked: input contained sensitive data (credit card)."
  }]
}
```

모델은 이 결과를 보고 사용자에게 안전한 방식으로 재시도하도록 안내할 수 있습니다. 사용자에겐 어떤 정책이 막았는지를 명확히 알리는 편이 신뢰를 만듭니다.

네 번째 항목은 **false positive 관리**입니다. 어떤 필터든 정밀도가 100%가 아닙니다. 다음 도구들이 도움이 됩니다.

```
도구                         의미
화이트리스트                  특정 도구·필드는 검사 생략
컨텍스트 기반 결정             해당 필드가 자유 텍스트인지 식별자인지에 따라 다르게
사용자 보고 채널               잘못 차단된 호출을 보고하면 정책 갱신
정책 시뮬레이션               새 정책이 실제 호출을 어떻게 차단할지 사전 분석
```

False positive는 사용자가 가장 빠르게 체감하는 보안 정책의 부작용입니다. 정확도와 사용성의 균형은 정기 점검 의식 안에서 조정해야 합니다(예: 분기별 정밀도 리뷰).

다섯 번째 항목은 **지역별 규제 매핑**입니다. 같은 데이터라도 처리 위치와 사용자 소속에 따라 적용되는 규제가 달라집니다.

```
규제                       적용 범위               핵심 요구
개인정보보호법(한국)        한국 사용자             동의, 처리 위탁, 국외 이전 동의
GDPR                       EU 사용자/거주자        목적 명시, 데이터 최소화, DPIA, 권리 행사
HIPAA                      미국 헬스케어 데이터    BAA, 암호화, 감사
PCI DSS                    카드 결제 데이터        토큰화, 분리 저장
```

게이트웨이는 데이터 흐름에 사용자의 지역·역할·도구 분류를 함께 태깅해 두면, 규제 변경에 빠르게 대응할 수 있습니다. 예를 들어 EU 사용자의 호출은 EU 데이터센터에 머무르도록 강제하는 정책을 정의할 수 있습니다.

여섯 번째 항목은 **암호화와 저장**입니다. 게이트웨이는 호출 흐름을 통과시키는 역할에 더해, 일부 데이터를 짧게 또는 길게 저장할 수 있습니다(감사 로그, 캐시, 메트릭). 저장되는 모든 데이터에는 다음이 적용되어야 합니다.

```
- 저장 시 암호화(at rest)
- 전송 시 암호화(in transit, TLS)
- 키 회전(3.4.2 참조)
- 접근 통제 + 접근 감사
```

특히 감사 로그에는 마스킹된 형태만 저장하고, 원본 데이터를 보관하지 않는 것이 표준입니다. 4.3.1에서 자세히 다룹니다.

마지막으로 **사용자 동의와 투명성**입니다. 사용자에게 게이트웨이가 어떤 데이터를 어떻게 다루는지 분명히 알릴 의무가 있습니다. 다음을 표준으로 합니다.

```
- 어떤 도구가 어떤 데이터를 처리하는지 카탈로그에 명시
- 민감 데이터 처리 도구는 사용자 첫 사용 시 추가 동의
- 사용자가 자기 데이터의 처리 이력을 조회할 수 있는 인터페이스
- 데이터 삭제·이동 권리 요청 처리 절차
```

이 투명성이 단순한 형식이 아니라 실제로 동작하면, 게이트웨이는 사용자가 신뢰하고 사용할 수 있는 인프라가 됩니다.

작은 함정 몇 가지로 마무리합니다. 첫째, **로그에 평문 토큰 남기지 않기**입니다. PII 필터를 도입하면서 정작 게이트웨이 자체 로그에 access token이나 user input을 그대로 남기는 경우가 흔합니다. 모든 저장 지점을 점검합니다. 둘째, **에러 메시지의 누설**입니다. 정책 위반 시 응답 메시지가 너무 자세하면, 공격자가 정책을 역공학할 수 있습니다. "policy violation"처럼 일반적인 메시지와 운영자 내부 코드(`policy_id=PII-CC-01`)를 분리해 둡니다. 셋째, **백업·복구 시 필터 비적용**입니다. 백업에서 데이터를 복원할 때 필터가 안 걸려 평문이 노출되는 경우가 있습니다. 백업도 필터 정책 안에서 다뤄야 합니다.

정리하면, PII와 민감정보 필터는 정규식+NER의 결합 탐지, 차단·마스킹·대체·토큰화의 정책 선택, false positive 관리, 지역별 규제 매핑, 저장 암호화·접근 감사, 사용자 투명성으로 구성됩니다. 게이트웨이가 단일 지점이라는 장점을 살려, 회사 전체의 데이터 흐름을 일관된 정책으로 다룰 수 있는 영역입니다.

다음 섹션부터는 이 모든 통제가 사후에 어떻게 추적되어야 하는지, 곧 감사 로그와 비용 추적을 살펴봅니다.
