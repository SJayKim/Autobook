# 4.1.1 왜 MCP Gateway가 필요한가

지금까지 우리는 MCP 호스트가 여러 서버와 직접 연결되는 모델을 가정했습니다. Phase 1의 3계층 그림(1.3.1)에서 호스트 안에 클라이언트가 여러 개 있고, 각각이 서버와 1:1로 붙어 있는 모양이 그것입니다. 작은 환경에서는 이 모델이 충분합니다. 사용자가 자기 머신에서 자기 도구를 직접 다루는 경우, 별도의 중간자는 없어도 됩니다. 그런데 사용자가 수십 명·수백 명으로 늘고, 서버 수도 함께 늘면, 직결 모델이 가진 한계가 분명해집니다. 그 한계가 **MCP Gateway**의 등장 동기를 만듭니다.

직결 모델의 한계를 다섯 가지로 정리할 수 있습니다.

첫째, **N×M 연결의 누적**입니다. 호스트 N개와 서버 M개가 있을 때, 모든 사용자가 모든 서버를 사용할 수 있게 하려면 N×M개의 인증·세션이 관리됩니다. 새 서버를 추가하면 모든 호스트가 그 서버를 알아야 하고, 새 호스트를 추가하면 모든 서버가 그 호스트에 인증을 발급해야 합니다. 사용자 한 명이 다섯 호스트를 쓰고 회사가 열 개 서버를 운영한다면 50개의 연결이 사람당 만들어집니다.

둘째, **정책의 분산**입니다. 도구를 허용·차단하는 정책, 비용·횟수를 제한하는 정책, 민감 데이터를 필터링하는 정책이 각 서버에 흩어집니다. 정책을 한 번에 갱신하려면 모든 서버를 손봐야 합니다. 새 정책이 한 곳에만 반영되고 다른 곳에 빠지면, 사용자가 "어디서는 되고 어디서는 안 된다"는 혼란을 겪습니다.

셋째, **관측의 파편화**입니다. 누가 어떤 도구를 얼마나 호출했는지, 어느 호출이 얼마나 비용을 발생시켰는지는 운영의 기본 정보입니다. 직결 모델에서는 이 데이터가 각 서버에 흩어집니다. 통합 대시보드를 만들려면 모든 서버에서 로그를 모아 집계해야 하고, 새 서버를 추가할 때마다 그 통합 코드를 갱신해야 합니다.

넷째, **공통 도구 카탈로그의 부재**입니다. 회사가 표준으로 권장하는 도구 묶음이 있어도, 직결 모델에서는 사용자가 직접 그 묶음을 자기 호스트에 추가해야 합니다. 사용자마다 누락이 생기고, "이 도구는 우리 회사에서 표준인가?"라는 질문이 사람마다 다른 답으로 돌아옵니다.

다섯째, **감사 통일성**입니다. 컴플라이언스 요구는 보통 "회사 내 모든 도구 호출이 일관된 형식으로 기록되어야 한다"는 식의 요구를 합니다. 직결 모델에서는 각 서버가 자기만의 로그를 남기므로, 일관 형식을 강제하기 어렵습니다.

이 다섯 한계를 한 줄로 묶으면 **공통 정책·관측·동의를 적용할 단일 지점이 없다**는 문제입니다. 정확히 이 자리를 채우려는 것이 게이트웨이입니다.

**MCP Gateway**는 여러 MCP 서버 앞에 서서 단일 엔드포인트로 노출하는 중간자입니다. 호스트는 게이트웨이 하나에만 연결되며, 게이트웨이 뒤편의 서버 묶음은 호스트에 보이지 않습니다. 게이트웨이는 들어오는 모든 요청을 인증·정책·관측을 거쳐 적절한 뒤편 서버로 라우팅합니다.

```
                  Host
                   |
                   v
       +--------- Gateway ---------+
       |  인증·정책·관측·라우팅    |
       +---------+----+----+-------+
                 |    |    |
                 v    v    v
            Server S1  S2   S3
```

게이트웨이가 떠맡는 책임을 정리하면 다음과 같습니다.

```
책임                                내용
인증 관문                            OAuth 토큰 검증, 사용자 식별
정책 엔진                            도구 allow/deny, scope 검증, 데이터 필터
멀티 서버 통합                       여러 서버를 하나의 도구 목록으로 합침
관측·감사                            모든 호출의 표준 로깅, trace 전파
비용·할당량                          사용자·팀 단위 호출 한도
세션·재연결                          서버 단절 시 재연결, fallback
업데이트 일관성                      새 서버 추가 시 호스트는 무변화
```

이 책임들을 게이트웨이 한 곳에 모음으로써, 직결 모델의 다섯 한계가 동시에 풀립니다. N×M이 N+M으로 줄고, 정책은 한 곳에서 갱신되며, 관측·감사는 표준 형식으로 통일됩니다.

게이트웨이가 등장하면 호스트의 책임도 단순해집니다. 호스트는 게이트웨이 하나만 알면 됩니다. 사용자가 새 서버를 사용하고 싶을 때 게이트웨이의 도구 카탈로그가 그것을 노출하기만 하면, 호스트는 추가 설정 없이 그 도구를 모델에 노출할 수 있습니다. 사용자 입장에서는 "회사의 모든 도구를 한 곳에서 받을 수 있다"는 경험이 됩니다.

물론 게이트웨이 도입에는 트레이드오프가 있습니다. 정리해 보면 다음과 같습니다.

```
이점                                  비용
단일 정책·관측 지점                    게이트웨이 자체의 운영 부담
새 서버 추가가 가벼움                  추가 hop으로 인한 latency 증가
사용자 경험 통일                       단일 실패 지점이 됨
보안 경계 명확                         게이트웨이가 손상되면 영향 큼
공통 도구 카탈로그                     서버별 고유 기능을 추상화하기 어려울 수 있음
```

이 트레이드오프 중 가장 자주 논의되는 것은 **단일 실패 지점**입니다. 게이트웨이가 멈추면 모든 도구가 함께 멈춥니다. 이 위험을 완화하려면 게이트웨이를 다중 인스턴스로 운영하고, 호스트가 둘 이상의 게이트웨이를 등록해 fallback할 수 있도록 만듭니다. 또한 게이트웨이의 책임을 너무 무겁게 두지 않고, 인증·정책·관측이라는 본질에 집중하게 두는 편이 안정에 좋습니다.

또 다른 결정 포인트는 **언제 도입할 것인가**입니다. 사용자 1명, 서버 2개라면 게이트웨이는 과합니다. 사용자 수십 명, 서버 두세 개라도 직결로 충분할 수 있습니다. 일반적인 경험으로는 다음 신호가 보이면 도입을 고려할 만합니다.

```
신호                                                도입 신호 강도
회사 차원의 도구 카탈로그 요구                       높음
컴플라이언스 보고서에 통합 감사 로그 필요            높음
사용자별·팀별 비용 추적·할당 필요                    높음
새 서버가 추가될 때마다 정책 갱신이 어려움           중간
호스트마다 도구 설정이 흩어져 사용자 불편            중간
보안 검토에서 직결 모델의 위험이 지적됨              높음
```

게이트웨이가 모든 경우에 옳은 답은 아닙니다. 사용자가 자기 머신에서만 도구를 쓰는 stdio 시나리오라면, 게이트웨이 없이 호스트가 직접 서버를 띄우는 모델이 더 자연스럽습니다. 게이트웨이는 주로 원격·다중 사용자·엔터프라이즈 환경에서 가치를 가집니다.

게이트웨이의 실제 구현 사례는 4.1.3에서 다룹니다. 그 전에 다음 단원 4.1.2에서는 게이트웨이가 여러 서버를 어떻게 한 엔드포인트로 합치는지, 그 아키텍처 패턴을 살펴봅니다.

정리하면, MCP Gateway는 N×M 연결, 정책 분산, 관측 파편화, 도구 카탈로그 부재, 감사 통일성이라는 직결 모델의 다섯 한계를 단일 지점에서 풀기 위한 중간자입니다. 도입에는 추가 latency와 단일 실패 지점 같은 비용이 따르지만, 다중 사용자·엔터프라이즈 환경에서는 이점이 비용을 압도하는 경우가 일반적입니다.

다음 단원인 4.1.2에서는 멀티 서버 통합 아키텍처의 구체 패턴들을 살펴봅니다.
