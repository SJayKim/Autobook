# 4.2.2 Rate limiting과 quota

도구 허용·차단 정책(4.2.1)이 "무엇을 할 수 있는가"를 답한다면, rate limiting과 quota는 "얼마나 자주, 얼마나 많이 할 수 있는가"를 답합니다. LLM 도구 호출은 컴퓨트 비용, 외부 API 비용, 모델 토큰 비용이 함께 누적되므로, 통제가 없으면 한 사용자의 폭주 호출이 전체 운영 비용을 흔들 수 있습니다. 이 단원에서는 rate limiting과 quota를 어디서·어떻게·어느 단위로 적용할지를 정리합니다.

먼저 두 개념을 구분합니다.

```
이름               단위 시간            적용 의도
Rate limiting      매우 짧음(초·분)       단기 폭주·DoS 방어
Quota              긴 단위(일·월)        총 비용·자원 사용 통제
```

Rate limiting은 짧은 시간 안에 호출이 몰리는 상황을 막습니다. 보통 초당·분당 호출 수로 표현됩니다. Quota는 더 긴 단위로 누적 사용량을 제한합니다. 일일 토큰 사용량, 월 예산 같은 식입니다. 둘은 함께 적용되어야 의미를 가집니다.

다음은 **적용 차원**입니다. 어느 단위에 한도를 둘지를 결정합니다.

```
차원               예시
사용자별           Alice는 분당 60회까지 호출 가능
팀별               엔지니어링팀 전체가 일일 100만 토큰까지
도구별             send_email 도구는 사용자당 시간당 10회까지
서버별             외부 SaaS 호출 도구는 분당 30회까지
테넌트별           고객사 A는 월 500만 호출까지
호스트별           일반 사용자 호스트는 IP당 분당 100회까지
```

대부분의 게이트웨이는 여러 차원을 **결합**해 사용합니다. 예를 들어 "사용자 단위 분당 60회 + 도구 단위 분당 10회 + 팀 단위 일일 100만 토큰"의 세 가지 한도가 동시에 검사됩니다. 어느 하나라도 초과하면 거부됩니다.

다음은 **알고리즘 선택**입니다. 한도 검사에는 몇 가지 표준 알고리즘이 있습니다.

```
알고리즘                  특징                                  적합 사례
Fixed window              고정 1분 윈도우에서 카운트            단순, 경계 폭주 가능
Sliding window log        타임스탬프 로그를 슬라이드 윈도우     정확, 메모리 비용 큼
Sliding window counter    두 윈도우 가중 평균                    정확·메모리 균형
Token bucket              버킷에 토큰 누적, 호출 시 소비        burst 친화적
Leaky bucket              일정 속도로 새 나가는 버킷            균등 속도 보장
```

대표적으로 **Token bucket**이 가장 많이 쓰입니다. 단기 burst를 허용하면서도 장기 평균 속도를 보장하기 때문입니다. 사용자가 평소엔 호출이 없다가 가끔 5~10회를 연속 호출하는 패턴을 자연스럽게 받아 줍니다.

```
Token bucket 동작

  버킷 용량: 10
  보충 속도: 분당 60 → 1초마다 1개

  호출 시 토큰 1개 소비
  버킷이 비어 있으면 거부 (또는 대기)
```

LLM 도구 호출의 특수성은 **호출의 무게가 균등하지 않다**는 점입니다. 어떤 호출은 짧은 함수, 어떤 호출은 무거운 검색·요약입니다. 단순 호출 수로 한도를 매기면 비용 통제에 한계가 있습니다. 이를 보완하기 위해 **가중 단위**를 사용합니다. 도구별로 가중치를 부여하고, 그 가중치만큼 토큰을 소비하는 식입니다.

```
도구                가중치
echo                1
search              5
heavy_summarize     20
send_email          30
```

이 가중치는 정적이지 않을 수 있습니다. 호출에 사용된 모델 토큰 수에 비례하게 사후에 차감하는 방식도 흔합니다. 호출이 끝나면 사용된 토큰만큼 추가 차감하고, 다음 호출 시점에 잔량을 확인합니다.

다음은 **스트리밍 응답의 부분 계량**입니다. 게이트웨이가 SSE로 스트리밍 응답을 흘려보낼 때, 응답 중에도 비용이 누적됩니다. 한도를 초과하는 응답을 도중에 끊을지, 완료까지 진행할지를 결정해야 합니다. 흔한 정책은 다음과 같습니다.

```
1. 호출 시작 시 예상 비용을 추정해 한도 검사 (사전 검사)
2. 응답이 흐르는 동안 누적 비용을 추적
3. 한도 초과 도달 시 SSE에 종료 이벤트 + isError 전송
4. 사후에 정확한 사용량을 계량해 다음 호출 시 차감
```

이 흐름은 사용자가 도중에 잘릴 수 있는 위험이 있으므로, 한도가 임박했음을 도구 결과에 함께 표시하는 것이 좋은 사용자 경험을 만듭니다.

**초과 시 응답 형태**도 표준화가 필요합니다. HTTP에서는 429 응답이 표준입니다. MCP에서는 다음 두 가지 형태가 가능합니다.

```
형태                       의미
HTTP 429 (transport 단)    클라이언트 라이브러리가 자동 backoff
isError + 메시지 (도구 단) 모델이 한도 초과를 인지하고 대안 시도
```

HTTP 429를 권장하지만, 도구 단 isError로 함께 표현하면 모델이 그 결과를 보고 행동을 조정할 수 있습니다. 두 형태를 혼합해서 쓰는 게이트웨이도 있습니다.

응답에는 표준 헤더가 함께 들어가는 것이 좋습니다.

```
HTTP/1.1 429 Too Many Requests
Retry-After: 12
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1741320120
```

`Retry-After`는 클라이언트가 언제 다시 시도할지 알려 줍니다. 클라이언트 SDK는 이 값을 보고 자동 대기·재시도를 수행합니다.

다음은 **공유 한도와 격리 한도**입니다. 한 한도가 여러 자원 사이에 공유될 수 있고, 자원별로 격리될 수도 있습니다.

```
모델                              예시
공유 한도                         모든 도구에 통합 일일 토큰 한도
격리 한도                         도구별 분당 호출 한도
계층적 한도                       사용자 한도 ⊂ 팀 한도 ⊂ 회사 한도
```

계층적 한도가 가장 표현력이 풍부합니다. 사용자가 자기 한도를 다 쓰지 못하더라도 팀이 한도에 도달하면 차단되고, 팀이 여유롭더라도 회사 전체 한도에 도달하면 차단됩니다. 이 모델은 비용 폭주 같은 상황에서 회사 단위 안전망 역할을 합니다.

**한도의 우선순위**도 결정이 필요합니다. 같은 호출에 여러 한도가 적용될 때, 어느 한도부터 검사할지를 정합니다. 일반적으로 빠르게 거부할 수 있는 한도(예: 가벼운 사용자별 카운터)를 먼저 검사해, 무거운 한도(분산 저장소 조회)에는 가능한 도달하지 않게 합니다.

**저장소 선택**도 운영 결정입니다. 한도 카운터를 어디에 둘지는 정확도·지연·확장성의 균형을 좌우합니다.

```
저장소            정확도         지연        확장성
인메모리          높음(단일 인스턴스)  매우 짧음   낮음
Redis             높음           짧음         중간
분산 키-값(Spanner) 매우 높음      길음         높음
근사 알고리즘     낮음~중간      매우 짧음   높음
```

다중 인스턴스 게이트웨이라면 보통 Redis가 표준 선택입니다. 매우 높은 정확도가 필요한 경우(예: 회사 예산 한도 직전) 분산 키-값 저장소를 사용하기도 합니다.

마지막으로 **사용자 가시성**입니다. 사용자가 자기 한도를 알 수 있어야 분쟁이 줄어듭니다. 다음을 표준으로 합니다.

```
- 사용자 대시보드에 현재 사용량·잔여 한도 표시
- 한도 임박 시 호스트 UI에 경고
- 도구 호출 결과에 사용량 부수 정보 포함
- 한도 도달 시 명확한 메시지와 다음 가능 시각 안내
```

운영자 측에는 다음 정보가 필요합니다.

```
- 한도 도달 빈도 지표 (top n 사용자·도구)
- 임시 한도 상향(임시 부스트) 기능
- 한도 변경의 감사 로그
- 한도 정책의 드라이런
```

정리하면, rate limiting과 quota는 사용자·팀·도구·서버 차원의 결합 적용, 가중치 기반 단위, token bucket·sliding window 알고리즘, 스트리밍 부분 계량, 표준 429 응답, 계층적 한도, 적절한 저장소 선택으로 구성됩니다. 사용자와 운영자 모두에게 가시성을 제공해야 한도가 분쟁의 출발점이 되지 않고 신뢰의 출발점이 됩니다.

다음 단원인 4.2.3에서는 게이트웨이가 입출력의 민감 정보를 다루는 PII 필터를 살펴봅니다.
