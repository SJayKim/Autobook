# 4.4.2 컴플라이언스 체크리스트

게이트웨이의 정책·감사·SSO·비용 추적을 모두 갖추었다면, 마지막으로 그 모든 것이 회사가 따라야 하는 규제 요구와 어떻게 매핑되는지를 정리할 차례입니다. 이 단원에서는 ISO 27001, SOC 2, EU AI Act 같은 대표 규제를 기준으로, MCP 운영에 적용할 수 있는 컴플라이언스 체크리스트를 제시합니다.

먼저 규제별 핵심 요구를 간단히 정리합니다.

```
규제                  관심 영역                            대표 통제
ISO 27001            정보보안 관리체계                     접근·암호화·감사·BCP
SOC 2                서비스 신뢰성 (Trust criteria)        보안·가용성·기밀성
GDPR                 EU 거주자 개인정보                    동의·이전·삭제·DPIA
개인정보보호법(KR)    한국 사용자 개인정보                  동의·위탁·국외 이전
HIPAA                미국 헬스케어 PHI                     BAA·암호화·접근 감사
EU AI Act            AI 시스템 분류와 의무                  카탈로그·투명성·감독
PCI DSS              결제 카드 데이터                      토큰화·분리·감사
```

규제는 표면적으로 다르지만 깊이 들여다보면 공통의 골격이 있습니다. **접근 통제, 데이터 보호, 감사 가능성, 사고 대응, 책임자 명시**라는 다섯 축이 거의 모든 규제에 등장합니다. MCP 게이트웨이를 운영할 때는 이 다섯 축에 맞춰 체크리스트를 만들어 두면, 규제별 매핑이 한결 쉬워집니다.

다음은 **접근 통제 증빙** 체크리스트입니다.

```
[ ] 모든 MCP 호출이 OAuth 2.1로 인증되는가
[ ] 사용자 신원이 회사 IdP로 통합되어 있는가
[ ] MFA가 민감 호출에 강제되는가
[ ] 도구별 scope가 정의되고 검증되는가
[ ] default-deny 정책이 적용되는가
[ ] 사용자 이탈 시 자동 차단되는가(SCIM·short-circuit)
[ ] 봇·서비스 정체성도 별도 식별 가능한가
[ ] 정기 권한 리뷰가 분기·반기 단위로 수행되는가
```

이 항목들은 ISO 27001의 A.9(접근 통제), SOC 2의 CC6(논리·물리적 접근)와 직접 매핑됩니다.

다음은 **데이터 보호** 체크리스트입니다.

```
[ ] 전송 중 TLS 1.2 이상 강제
[ ] 저장 시 AES-256 등 표준 알고리즘으로 암호화
[ ] 비밀·키의 정기 회전(3.4.2)
[ ] PII 탐지·마스킹 정책 적용(4.2.3)
[ ] 데이터 보존 기간 정책 문서화
[ ] 데이터 거주성 요구(EU·KR) 충족
[ ] 백업의 암호화와 접근 통제
[ ] 폐기 시 안전한 삭제(crypto-shredding 또는 secure delete)
```

이 항목들은 GDPR의 article 32(보안), HIPAA의 보안 규칙, PCI DSS의 다수 항목과 매핑됩니다. 특히 PII 마스킹은 4.2.3에서 다룬 게이트웨이 정책 엔진이 그대로 증빙이 됩니다.

다음은 **감사 가능성** 체크리스트입니다.

```
[ ] 모든 호출이 표준 스키마로 기록되는가(4.3.1)
[ ] 감사 로그가 append-only로 보호되는가
[ ] 보존 기간이 규제 요구를 충족하는가
[ ] 감사 로그 자체의 접근 통제가 있는가
[ ] 정책 변경 이력이 보존되는가
[ ] 사고 시 재현 가능한 정보가 있는가
[ ] 감사 보고서 자동 생성 가능한가
[ ] 정기 감사 의식(매주·월·분기)이 동작하는가
```

이 항목들은 SOC 2의 CC7(시스템 운영), ISO 27001의 A.12.4(로깅·모니터링)와 매핑됩니다. 특히 정책 변경 이력은 EU AI Act가 요구하는 "AI 시스템 변경 추적"과 직결됩니다.

다음은 **사고 대응** 체크리스트입니다.

```
[ ] 인시던트 대응 runbook이 문서화되어 있는가
[ ] 토큰 무효화·세션 강제 종료 절차가 있는가
[ ] 정책 임시 차단·롤백 절차가 있는가
[ ] 보안 알림 임계와 담당자가 정의되어 있는가
[ ] 분기마다 모의 사고 시뮬레이션이 있는가
[ ] 사고 후 회고와 정책 갱신이 이루어지는가
[ ] 외부 통보 의무(GDPR 72시간 등)를 만족할 수 있는가
[ ] 백업 복구 시간(RTO)·데이터 손실 한도(RPO) 측정
```

이 항목들은 거의 모든 규제가 요구합니다. 특히 GDPR은 침해 발견 후 72시간 이내 감독 기관 통보를 요구하므로, 침해 탐지·평가·통보 절차가 실제로 동작해야 합니다.

다음은 **AI 특화 책임** 체크리스트입니다. EU AI Act가 2024~2026년에 걸쳐 단계적으로 발효되면서, AI 시스템 카탈로그·투명성·인간 감독이 새 요구가 되었습니다.

```
[ ] 회사 AI 시스템 카탈로그에 MCP 게이트웨이 등록
[ ] 사용된 모델·도구·외부 의존성 명시
[ ] 위험 분류(허용·고위험·금지) 기준 적용
[ ] 사용자가 AI와 상호작용함을 명시(투명성)
[ ] 고위험 시스템의 경우 인간 감독 절차 정의
[ ] 자동 결정의 의사 결정 권한 표현
[ ] 모델·도구 변경의 영향 평가(EU AI Act 요구의 핵심)
[ ] 정기적 위험 평가 리뷰
```

EU AI Act는 이행을 위해 시스템마다 책임자(역할: provider·deployer)를 명시할 것을 요구합니다. MCP 게이트웨이는 일반적으로 deployer 역할이며, 게이트웨이가 호출하는 외부 모델·SaaS는 별도 provider일 수 있습니다. 이 구분을 명확히 해 두면 책임 추적이 쉬워집니다.

다음은 **벤더 위험 평가** 체크리스트입니다.

```
[ ] 사용 중인 MCP 서버의 출처·운영자 식별
[ ] 외부 SaaS 의존성의 데이터 처리 위치
[ ] DPA(Data Processing Agreement) 체결 여부
[ ] 보안 인증 보유 여부(SOC 2·ISO 27001 등)
[ ] 사고 발생 시 통보 약속
[ ] 데이터 삭제 요청 대응 시간
[ ] 정기 재평가 일정
```

MCP는 작은 서버가 빠르게 등장하는 생태계이므로, 새 서버를 회사 게이트웨이에 합류시킬 때 이 평가를 빠뜨리지 않는 절차가 필요합니다. 4.2.1의 default-deny 정책이 이 단계의 기술적 받침대 역할을 합니다.

마지막으로 **거버넌스와 책임자**입니다.

```
[ ] 게이트웨이 운영 책임자(소유자) 명시
[ ] 보안·개인정보 책임자(DPO) 임명
[ ] 정책 변경 승인 절차 정의
[ ] 분기별 거버넌스 리뷰 회의
[ ] 변경 관리 프로세스(CAB·peer review)
[ ] 인력 변동 시 권한 회수 절차
[ ] 사용자 교육·인식 제고 프로그램
```

이 항목들은 규제가 요구하는 "조직적 통제"입니다. 기술만 갖춰도 조직이 그것을 운영하지 않으면 규제 적합성을 입증하기 어렵습니다.

규제별로 자주 요구하는 산출물 몇 가지도 정리합니다.

```
산출물                                필요한 규제
정보보안 정책 문서                     ISO 27001, SOC 2
데이터 흐름 다이어그램                  GDPR, HIPAA, PCI DSS
처리 활동 기록(RoPA)                   GDPR
DPIA 보고서                            GDPR(고위험)
변경 관리 로그                          ISO 27001, SOC 2
사고 대응 runbook                      거의 모든 규제
정기 감사 보고서                       SOC 2 type II
AI 시스템 카탈로그·위험 평가          EU AI Act
```

위 산출물을 처음부터 자동화해 두면 정기 감사 시점에 큰 부담 없이 제출할 수 있습니다. 게이트웨이가 보유한 정책 파일·감사 로그·사용자 매핑이 그대로 산출물의 절반 이상을 채웁니다.

이 체크리스트의 가치는 모든 항목을 한 번에 만족시키는 것이 아닙니다. **점진적 적용**이 핵심입니다. 다음 우선순위가 일반적입니다.

```
1. 인증·접근 통제 (보안의 뿌리)
2. 감사 로그 (사후 추적의 토대)
3. 데이터 보호와 PII (가시적 위험)
4. 사고 대응 (운영 안전망)
5. AI 특화 책임 (최신 규제 대응)
6. 벤더 위험 평가 (외부 의존성)
7. 거버넌스 (조직적 통제)
```

회사 규모와 규제 노출 정도에 따라 이 순서가 달라질 수 있지만, 1~4번은 거의 모든 환경에서 우선순위가 높습니다.

이 체크리스트는 마지막으로, 정기적으로 다시 적용해야 살아 있습니다. 분기마다 한 번씩 점검해 통과·미통과·이슈를 정리하면, 시간이 흐르며 조직의 규제 성숙도가 자연스럽게 올라갑니다.

이 단원으로 Phase 4를 마무리합니다. 다음 Phase 5에서는 한 발 더 나아가, MCP를 단일 도구 통합을 넘어 멀티 에이전트 오케스트레이션에 결합하는 방법을 살펴봅니다. 다음 단원인 5.1.1에서는 단일 에이전트로 풀기 어려운 문제의 특성부터 시작합니다.

정리하면, 컴플라이언스 체크리스트는 접근 통제·데이터 보호·감사 가능성·사고 대응·AI 책임·벤더 위험·조직 거버넌스라는 일곱 축으로 구성되며, 점진적 적용을 통해 ISO 27001·SOC 2·GDPR·EU AI Act 같은 다양한 규제에 동시에 답할 수 있는 기반이 됩니다.
