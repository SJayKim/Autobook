# 5.3.1 멀티 에이전트 디버깅

멀티 에이전트 시스템을 운영하기 시작하면 처음 만나는 큰 도전이 디버깅입니다. 단일 에이전트라면 한 모델의 응답을 보고 문제를 좁힐 수 있지만, 여러 에이전트가 협력하는 시스템은 누가 누구에게 무엇을 말했는지, 어느 도구가 어떤 결과를 돌려주었는지가 한눈에 보이지 않습니다. 이 단원에서는 디버깅을 가능하게 만드는 표준 도구와 패턴을 정리합니다.

가장 기초가 되는 도구는 **trace ID 전파**입니다. 한 사용자 메시지가 시스템에 들어오는 순간 고유한 trace_id를 만들고, 그 ID를 모든 이후 호출에 함께 흘려보내야 합니다.

```
사용자 메시지 (trace_id = X)
  ├ supervisor 모델 호출 (trace_id = X)
  ├ researcher 에이전트 호출 (trace_id = X)
  │    └ MCP 호출: search (trace_id = X)
  │         └ 외부 API: web search (trace_id = X)
  ├ analyzer 에이전트 호출 (trace_id = X)
  └ writer 에이전트 호출 (trace_id = X)
```

trace_id가 끊김 없이 전파되어야, 사고 시 같은 ID로 검색해 전체 흐름을 한 줄로 재구성할 수 있습니다. 한 곳에서 ID가 끊기면 그 구간의 호출은 미아가 됩니다.

전파의 표준 방법은 **OpenTelemetry의 W3C Trace Context**입니다. `traceparent` 헤더를 각 호출에 넣고, 받는 쪽이 그 헤더로 자기 스팬의 부모를 결정합니다. MCP 호출에서는 게이트웨이가 모든 호출에 자동으로 헤더를 추가하도록 만드는 것이 가장 깔끔합니다(4.3.1 감사 로그와 결합).

다음은 **OpenTelemetry GenAI semantic conventions**입니다. 모델 호출과 도구 호출에 어떤 속성을 붙일지 표준이 제안되었습니다.

```
모델 호출 스팬 속성
gen_ai.system            = "anthropic"
gen_ai.request.model     = "claude-sonnet-4-6"
gen_ai.usage.input_tokens
gen_ai.usage.output_tokens
gen_ai.response.id

도구 호출 스팬 속성
gen_ai.tool.name         = "wiki__search"
gen_ai.tool.call.id      = "call_abc..."
gen_ai.tool.call.arguments  (마스킹)
```

이 conventions는 아직 안정 단계는 아니지만, 도입하면 다른 도구·대시보드와 통합이 쉬워집니다. 같은 속성을 쓰는 시스템 사이에서 trace 분석 쿼리가 호환됩니다.

다음은 **LangSmith·LangFuse 같은 LLM 관측 플랫폼**입니다. 이 도구들은 LangGraph·AutoGen·CrewAI의 이벤트를 자동 수집해 시각화합니다.

```
도구                       기능
LangSmith                  LangChain·LangGraph 통합, 추적·평가
LangFuse                   오픈소스, 프레임워크 중립
Arize Phoenix              오픈소스, 추적·평가·실험
Weights & Biases           실험·평가·trace
OpenTelemetry + Grafana    표준 기반, 자체 구축
```

대부분의 도구는 한 번의 트레이스에서 "어떤 모델 호출", "어떤 도구 호출", "어떤 메시지"가 어떤 순서로 일어났는지를 timeline으로 보여 줍니다. 한 노드를 클릭하면 입력·출력 본문, 토큰 수, 비용, 지연이 표시됩니다. 멀티 에이전트 시스템에서는 이 timeline 뷰가 디버깅의 70%를 차지합니다.

다음은 **메시지 단위 캡처**입니다. trace로 흐름은 보이지만, 가끔은 특정 에이전트의 특정 호출 본문이 필요합니다. 다음을 저장합니다.

```
저장 대상                              주의
모델 입력 메시지 (system + user)        토큰 비용 고려, 마스킹
모델 출력 (도구 호출 의도 포함)         그대로 저장
도구 호출 인자                          민감 필드 마스킹
도구 결과                                요약 + 참조 키
에이전트 간 메시지                       그대로 또는 요약
```

전체를 저장하면 비용·저장소·민감 데이터 문제가 커지므로, **샘플링** 전략을 도입합니다. 정상 호출은 1%만 본문까지 저장, 이상 호출은 100% 저장하는 식입니다.

다음은 **Replay 기반 디버깅**입니다. 한 사고가 발생했을 때, 같은 입력으로 시스템을 다시 실행해 결과를 재현할 수 있어야 합니다. 다음 조건이 필요합니다.

```
조건                              의미
입력의 결정성                      같은 사용자 메시지·컨텍스트
모델 응답의 결정성                 seed·temperature 고정 또는 응답 캐시
도구 응답의 결정성                 외부 API 응답 캐시(replay 모드)
랜덤 시드 고정                     fan-out 분기·통합 등
정책의 버전                        그 시점의 정책 스냅샷
```

LangGraph는 checkpoint를 이용해 그래프 상태를 재구성할 수 있습니다. AutoGen은 메시지 로그로 같은 대화를 재현합니다. CrewAI는 Task 입출력을 저장해 재실행할 수 있습니다. replay가 가능한 시스템은 사고 회고가 훨씬 깊어집니다.

다음은 **휴먼 in-the-loop 중단점**입니다. 디버깅 단계뿐 아니라 운영 단계에서도 중요한 결정 직전에 사람이 검토하는 패턴이 필요합니다.

```
중단점 종류                          예시
도구 호출 전 승인                     send_email, delete_page 호출 직전
모델 출력 검토                         최종 응답을 사용자에게 보내기 직전
정책 위반 의심 시 일시 정지          PII 탐지, 한도 임박
긴 작업 중간 점검                     단계별 진행 상황 확인
```

각 프레임워크가 이 패턴을 표현하는 방법이 다릅니다. LangGraph는 conditional edge로, AutoGen은 UserProxyAgent로, CrewAI는 hierarchical 모드의 manager에 사람을 끼워 넣는 식으로 구현합니다. 4.3.1의 감사 로그와 결합해, 중단점에서 어떤 결정이 내려졌는지도 기록합니다.

다음은 **전형적 멀티 에이전트 장애 패턴**입니다.

```
증상                                         가장 흔한 원인
에이전트가 답을 멈춤                          recursion limit·종료 조건 누락
같은 도구 호출이 반복됨                       응답 결과를 잘못 해석
한 분기만 비정상적으로 느림                   외부 API의 burst
fan-in 결과가 항상 한쪽만 사용                통합 단계의 프롬프트 편향
supervisor가 같은 subgraph만 호출            분기 기준 시스템 프롬프트 불명확
인자 형식 오류 연속                           inputSchema description 부실
도구 호출 후 의미 변형                        prompt injection 의심
```

이 표를 점검 체크리스트로 사용하면, 새 문제를 마주했을 때 첫 의심 지점을 빠르게 좁힐 수 있습니다.

다음은 **로컬 재현 환경**입니다. 운영 환경에서 발생한 사고를 로컬에서 재현할 수 있게 다음을 준비합니다.

```
1. 운영 환경의 정책 스냅샷을 로컬로 가져오는 도구
2. 외부 API 응답을 mock으로 재생하는 fixtures
3. 모델 응답의 결정적 재생 (recording/replay)
4. 같은 trace_id로 시스템을 시작하는 디버깅 진입점
5. 로컬 게이트웨이의 도구 catalog 동기화
```

이 환경이 갖춰지면, 사고를 자기 머신에서 디버거를 걸어 가며 분석할 수 있습니다. "운영에서만 발생하는 문제"의 비율이 크게 줄어듭니다.

마지막으로, **디버깅을 쉽게 만드는 설계 원칙** 몇 가지로 마무리합니다.

```
- 에이전트 메시지는 사람이 읽기 좋은 형식으로 (JSON·markdown)
- 각 에이전트의 system prompt를 외부 파일로 분리(VCS)
- 그래프·crew 구조를 자동 시각화 가능하게(예: mermaid)
- 도구 호출에 항상 의미 있는 ID(call_id)를 부여
- 비정상 종료 시 마지막 상태를 dump 파일로 저장
```

이 원칙들을 처음 시스템을 설계할 때 함께 챙기면, 운영 단계에서 디버깅에 들이는 시간이 크게 줄어듭니다.

정리하면, 멀티 에이전트 디버깅은 trace ID 전파, OpenTelemetry 표준, LLM 관측 플랫폼, 메시지 단위 캡처(샘플링), replay 기반 재현, human-in-the-loop 중단점, 전형적 장애 패턴 체크리스트, 로컬 재현 환경, 그리고 디버깅을 쉽게 만드는 설계 원칙으로 구성됩니다. MCP의 표준 trace 전파 덕분에 게이트웨이 한 곳에서 모든 도구 호출이 trace에 합류한다는 점이 큰 이점이 됩니다.

다음 단원인 5.3.2에서는 비용과 지연을 최적화하는 패턴을 살펴봅니다.
