# 3.4.2 Secret rotation과 감사

OAuth 골격을 깔고 scope·audience까지 설계하면 인증 체계의 큰 부분은 마무리됩니다. 마지막으로 남는 것은 **운영 단계의 위생**입니다. 비밀과 토큰을 어떻게 회전하고, 사고가 났을 때 어떻게 무효화하고, 어떤 사건을 어디에 기록할지에 관한 결정입니다. 이 단원에서 그 결정을 모아 정리하고 Phase 3을 마무리합니다.

먼저 **회전**(rotation)의 의미를 정리합니다. 비밀과 토큰은 시간이 지나면 노출 가능성이 누적됩니다. 코드 저장소에 실수로 커밋되거나, 로그에 새거나, 직원 이탈 시 함께 빠져나갈 수도 있습니다. 회전은 일정 주기마다 그 비밀을 새 값으로 교체해, 노출의 영향 기간을 짧게 만드는 운영 습관입니다.

MCP·OAuth 환경에서 회전 대상은 다음 다섯 가지입니다.

```
대상                                   회전 주기 (권장)
Client secret(confidential client)     90일~6개월
Refresh token                          사용 시마다(회전) 또는 30일 단위
Access token                           매 호출마다 짧은 만료(5~15분)
AS 서명 키(JWT)                        6개월~1년
서버 측 비밀(DB 비밀번호, API 키)      90일~6개월
```

**Access token 회전**은 만료 시간을 짧게 두는 것으로 자연스럽게 일어납니다. 5분, 10분 같은 짧은 수명이 일반적입니다. 만료된 토큰은 refresh token으로 새로 받습니다. 사용자는 그 흐름을 거의 인식하지 못합니다.

**Refresh token 회전**은 OAuth 2.1이 권장하는 모델입니다(3.1.2 참조). 사용할 때마다 새 refresh token이 발급되고 옛 토큰은 폐기됩니다. AS가 이를 강제하면, 누군가 옛 refresh token을 가로채 사용하려고 할 때 정상 사용자가 이미 회전을 일으켰으므로 AS가 이상 사용을 감지하고 그 가족 전체(reuse detection)를 무효화합니다.

**Client secret 회전**은 confidential client에 해당합니다. 서버 사이드 호스트나 자동화 봇이 client secret을 가지고 있다면, 정기적으로 새 secret을 발급받고 옛 것을 폐기합니다. AS가 두 secret을 일정 기간 함께 받아 주는 grace period가 있으면 무중단 회전이 가능합니다.

**AS 서명 키 회전**은 JWT 발급용 키 페어를 정기적으로 교체하는 것입니다. AS는 보통 JWKS 엔드포인트에 여러 키를 함께 노출하므로, 새 키가 추가된 뒤에야 옛 키로 발급된 토큰들이 만료될 때까지 기다리고, 그 다음 옛 키를 빼내는 식의 무중단 절차가 가능합니다.

**서버 측 비밀 회전**(데이터베이스 비밀번호, 외부 API 키 등)은 보통 KMS(Key Management Service)나 Vault 같은 비밀 저장소가 자동화해 줍니다. MCP 서버 자체는 환경 변수나 짧은 수명의 임시 자격증명을 받아 사용합니다.

회전을 효과적으로 운영하려면 **자동화**가 핵심입니다. 수동으로 분기마다 회전하는 절차는 빠뜨리기 쉽고, 한 번 빠뜨리면 옛 비밀이 무기한 살아남게 됩니다. KMS·Vault·클라우드 시크릿 매니저 같은 도구는 자동 회전과 알림을 제공합니다. 회전 후 새 값을 가져오는 코드가 자동 새로고침되도록 만들어 두면, 운영자가 잊어도 안전성이 유지됩니다.

다음은 **사고 대응** 측면의 토큰 무효화입니다. 사고가 발생했을 때(예: 토큰 노출 의심, 직원 이탈, 보안 인시던트) 무엇을 무효화할 수 있어야 할지를 사전에 그려 둡니다.

```
사고 시 무효화 대상           방법
특정 사용자의 모든 토큰        AS의 사용자 세션 폐기 + refresh token 무효화
특정 client의 모든 토큰        DCR 등록 폐기 + client_id 차단
특정 audience의 모든 토큰      자원 서버에서 키 변경 + introspection 강제
모든 토큰                       JWT 서명 키 변경(non-graceful)
```

이 흐름이 실제로 동작하는지 사전에 한 번 시험해 두는 것이 중요합니다. "사고가 났을 때 토큰을 어떻게 무효화할 수 있는가"를 분기마다 한 번씩 시뮬레이션하면, 그 절차가 살아 있는지 확인할 수 있습니다.

다음은 **감사 이벤트**입니다. 인증 흐름에서 일어나는 모든 변화는 감사 로그에 남아야 합니다. 다음 이벤트가 최소 기록 대상입니다.

```
이벤트                                기록 필드
DCR 등록 성공/실패                     client_name, redirect_uris, ip
authorize 시도                         client_id, scope, user_id, ip, ua
사용자 동의/거부                       user_id, client_id, scope, decision
토큰 발급                              user_id, client_id, audience, scope, exp
토큰 갱신(refresh)                     이전 jti, 새 jti, ip
토큰 폐기(revoke)                      jti, 이유, 운영자
이상 토큰 사용(reuse detection)        jti, 시간, ip 차이
audience 불일치 401                    사용된 토큰의 aud, 호출한 자원
세션 만료/폐기                         session_id, 이유
```

각 이벤트는 **append-only** 저장소에 기록하는 것이 표준입니다. 누군가 사후에 로그를 지우거나 변경하지 못하도록 막아야 사고 조사에서 의미가 생깁니다. 클라우드 환경이라면 CloudTrail·Cloud Logging·Audit Logs 같은 표준 서비스를 활용합니다.

감사 로그의 보존 기간은 규제와 정책에 따릅니다. 개인정보보호 관점에서는 너무 길게 보관하면 안 되고, 보안 조사 관점에서는 너무 짧으면 분석이 어렵습니다. 흔히 90일~1년이 절충값이며, 산업 규제에 따라 다를 수 있습니다(자세한 매핑은 4.4.2에서 다룹니다).

다음은 **민감 데이터의 마스킹**입니다. 감사 로그에는 토큰 그 자체나 민감한 사용자 데이터를 평문으로 남기지 않아야 합니다. 토큰은 짧은 프리픽스나 jti만 남기고, 사용자 입력은 필요한 식별 정보로 좁혀 기록합니다. 도구 인자에 비밀번호·카드 번호 같은 정보가 포함된 호출은 마스킹 정책을 거쳐야 합니다.

마지막으로 **운영 의식**(operational rituals)입니다. 회전과 감사가 살아 있으려면 정기적인 점검 의식을 갖춰야 합니다. 다음과 같은 의식들이 잘 동작합니다.

```
의식                                                  주기
회전 실패·만료 임박 자동 알림 확인                    매주
의심스러운 토큰 사용 패턴(많은 reuse·다른 ip) 검토   매주
DCR로 등록된 클라이언트 목록 검토                    매월
스코프·audience 사용 통계 검토                       매월
인증 사고 대응 시뮬레이션                            분기
규제·감사 요구 사항 매핑 갱신                        반기
```

이 의식들은 작아 보이지만, 사고가 나기 전에는 보이지 않는 작은 결함을 미리 찾아 줍니다. MCP가 회사의 데이터 흐름의 핵심에 들어올수록 이 작은 점검이 의미를 키웁니다.

Phase 3을 한 줄로 묶으면 다음과 같습니다. OAuth 2.1과 DCR이 인증의 골격을 정의하고, PKCE·redirect 검증·audience·scope가 토큰 단위 보호를 제공하며, token passthrough·confused deputy·tool poisoning·prompt injection 같은 새 위협에 대비한 다층 방어가 필요합니다. 마지막으로 회전과 감사가 이 모든 위에서 운영 위생을 유지합니다.

정리하면, secret rotation은 access token·refresh token·client secret·서명 키·서버 비밀의 다섯 대상을 자동화된 주기로 교체하는 운영 습관이고, 감사는 각 인증 사건을 append-only 로그에 기록해 사후 조사와 회귀 방지를 지원합니다. 두 가지가 함께 적용되어야 OAuth의 안전성이 종이 위가 아니라 실제 운영에서 살아 있습니다.

이 단원으로 Phase 3을 마무리합니다. 다음 Phase 4에서는 한 발 더 나아가, 여러 MCP 서버를 묶어 다루는 게이트웨이와 엔터프라이즈 거버넌스를 살펴봅니다. 다음 단원인 4.1.1에서는 게이트웨이가 왜 필요해지는지를 출발점으로 삼습니다.

이 단원을 마치면 토큰과 비밀의 회전 주기를 결정하고, 사고 대응 시 무엇을 무효화할지 결정하며, 감사 로그 스키마를 설계할 수 있습니다.
