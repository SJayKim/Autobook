# 5.1.3 Subgraph와 위임

fan-out·fan-in(5.1.2)이 평면적 협력이라면, **subgraph**와 **위임**은 계층적 협력입니다. 한 에이전트가 다른 에이전트 묶음에게 작업을 위임하고, 그 결과만 받아 자기 작업을 계속하는 패턴입니다. 이 단원에서는 subgraph로 책임을 묶고 supervisor가 그것을 조율하는 표준 구조를 살펴봅니다.

먼저 **계층적 아키텍처**의 동기를 짚어 봅니다. 큰 작업을 풀 때, 모든 에이전트가 동등하게 협력하는 평면적 구조는 다음 한계에 부딪힙니다.

```
평면적 구조의 한계

- 에이전트 수가 늘수록 통신 그래프가 폭발
- 누가 누구에게 묻고 답할지의 규칙이 복잡
- 작업의 큰 그림을 유지할 책임자가 모호
- 일부 에이전트의 실패가 전체에 번지기 쉬움
- 새 에이전트 추가의 비용이 큼
```

계층적 구조는 이 문제를 다음처럼 풀어 줍니다. **Supervisor**(또는 orchestrator)가 큰 그림을 책임지고, 하위 작업은 **subgraph**라는 묶음에 위임합니다. supervisor는 subgraph의 내부를 모르고, subgraph는 외부 supervisor의 사정을 모릅니다. 둘은 명확한 입력·출력 스키마로만 만납니다.

```
계층적 아키텍처

                     [Supervisor]
                          |
        +-----------------+-----------------+
        |                 |                 |
   [Subgraph A]      [Subgraph B]      [Subgraph C]
   (조사 묶음)        (분석 묶음)        (작성 묶음)
   ├ search agent    ├ stats agent      ├ writer
   ├ filter agent    └ trend agent      └ tone agent
   └ summarize
```

이 구조의 가장 큰 가치는 **모듈화**입니다. 한 subgraph를 다른 구현으로 교체해도 supervisor는 같은 인터페이스로 호출합니다. 새 도구·새 모델·새 패턴을 subgraph 내부에 도입해도 외부에는 영향이 없습니다.

다음은 **supervisor 패턴**의 전형 흐름입니다.

```
1. supervisor가 사용자 메시지를 받아 작업 계획 수립
2. 계획에 따라 어떤 subgraph를 어떤 순서로 호출할지 결정
3. 첫 subgraph 호출 → 결과 수신
4. 결과를 보고 다음 단계 결정 (필요시 다른 subgraph 호출)
5. 모든 단계 완료 시 결과 통합 → 사용자에게 응답
```

supervisor는 사람의 매니저처럼 행동합니다. 자신이 직접 도구를 호출하기보다는, 누구에게 무엇을 부탁할지를 결정합니다. 이 역할 분리가 supervisor의 컨텍스트를 가볍게 유지해 줍니다. 도구의 세부 결과는 subgraph 안에 남고, supervisor에게는 요약만 전달됩니다.

다음은 **subgraph의 캡슐화 경계**입니다. subgraph가 외부에 노출하는 인터페이스는 다음으로 구성됩니다.

```
입력 스키마          subgraph가 받는 인자 형태
출력 스키마          subgraph가 돌려줄 결과 형태
도구 권한            subgraph가 사용할 MCP 도구 묶음
실패 모드            처리 불가 상황의 표준 응답
종료 조건            언제 작업을 끝내는가
```

이 인터페이스가 명확할수록 subgraph를 다른 시스템에서도 재사용할 수 있습니다. 좋은 subgraph는 한 회사의 한 시스템에 묶이지 않고, "조사 묶음", "검증 묶음", "변환 묶음" 같은 추상으로 다른 곳에서도 활용됩니다.

다음은 **subgraph 내부 구조**입니다. subgraph는 그 자체로 작은 멀티 에이전트 시스템입니다. 안에서 어떤 패턴을 쓰는지는 자유입니다. 예를 들어 조사 subgraph는 다음처럼 구성될 수 있습니다.

```
조사 subgraph 내부

  입력: "Q에 대해 조사"
   |
  [planner] → 검색어 후보 결정
   |
  fan-out
   ├ [searcher A] → web search MCP
   ├ [searcher B] → wiki MCP
   └ [searcher C] → internal docs MCP
   |
  fan-in [aggregator]
   |
  [summarizer] → 요약 결과
   |
  출력: "조사 결과 요약"
```

내부에서는 fan-out·fan-in이 자유롭게 사용되지만, 외부에서는 단일 입력·단일 출력처럼 보입니다.

다음은 **subgraph 단위 평가**입니다. 모듈화의 큰 이점 중 하나는 평가도 단위별로 가능하다는 점입니다. 조사 subgraph만 따로 떼서 "이 입력에 대해 좋은 조사 결과를 내는가"를 평가할 수 있고, 작성 subgraph도 따로 평가할 수 있습니다. 전체 시스템 평가 외에 단위 평가가 가능해야 회귀가 빠르게 잡힙니다(자세한 평가 패턴은 5.3.3에서 다룹니다).

다음은 **재진입과 무한 루프 방지**입니다. 계층적 구조에서 가장 흔한 사고는 supervisor와 subgraph가 서로를 무한히 부르는 상황입니다. 다음 가드를 두면 안전합니다.

```
가드                                  설명
최대 호출 깊이                         supervisor 호출 깊이 N 이상 제한
최대 호출 수                           한 사용자 요청 당 전체 모델 호출 수 제한
최대 시간                              한 요청의 wall-clock 시간 제한
중복 호출 감지                         같은 인자로 같은 subgraph 반복 호출 차단
종료 의도 명시                         supervisor가 "이제 끝낸다"는 시그널을 명확히
```

LangGraph 같은 프레임워크는 이 가드를 명시적으로 표현할 수 있게 해 줍니다. recursion_limit, max_tokens, timeout 같은 설정이 그 자리에 들어갑니다.

다음은 **MCP와 subgraph의 결합**입니다. subgraph가 외부 도구를 호출할 때 MCP 게이트웨이를 사용하는 패턴이 자연스럽습니다. 다음 이점이 있습니다.

```
이점
- 정책·관측의 단일 지점 (Phase 4)
- subgraph 단위 권한 분리 (각 subgraph에 다른 scope)
- 도구 카탈로그의 공유 (subgraph 간 도구 재사용)
- 사고 시 호출 출처 추적 (subgraph_id를 trace에 함께)
```

subgraph마다 다른 audience·scope의 토큰을 사용하면, "조사 subgraph가 갑자기 결제 도구를 호출"하는 식의 사고를 게이트웨이가 막을 수 있습니다. 4.2.1의 정책 엔진이 subgraph 단위로 적용됩니다.

다음은 **supervisor의 의사 결정 품질**입니다. supervisor가 사용하는 모델은 작업 계획·다음 단계 결정의 정확도가 매우 중요합니다. 흔한 함정 몇 가지를 정리합니다.

```
함정                                  완화
supervisor가 너무 일찍 종료            출력 스키마에 명시적 done 플래그
supervisor가 너무 많이 반복            최대 단계 수 한도
supervisor가 도구를 직접 부르려 듦    supervisor에 도구 노출 제한
supervisor의 컨텍스트 과부하           subgraph 결과를 요약해서만 전달
잘못된 subgraph로 위임                supervisor의 시스템 프롬프트에 분기 기준 명시
```

supervisor의 시스템 프롬프트는 각 subgraph가 무엇에 적합한지를 분명히 설명해 두는 것이 좋습니다. 이 설명이 부실하면 supervisor가 잘못된 위임을 반복하게 됩니다.

다음은 **실전 예시**입니다. 보고서 작성 시스템을 supervisor + 세 subgraph로 구성한 예입니다.

```
사용자: "Q3 매출 보고서를 작성해 줘"

supervisor (Claude 4.6, ~2K 토큰 컨텍스트)
  ├ 1단계: 조사 subgraph 호출
  │   입력: Q3 매출, 경쟁사, 시장 상황
  │   출력: 요약된 사실 묶음 (≈3K 토큰)
  ├ 2단계: 분석 subgraph 호출
  │   입력: 사실 묶음
  │   출력: 인사이트와 도표 데이터 (≈2K 토큰)
  └ 3단계: 작성 subgraph 호출
      입력: 인사이트 + 톤 가이드
      출력: 보고서 텍스트 (≈4K 토큰)

supervisor의 최종 컨텍스트: 사용자 메시지 + 각 subgraph 요약(약 9K)
```

supervisor의 컨텍스트가 9K로 유지되었기 때문에 모델이 각 단계를 깔끔하게 다룰 수 있었습니다. 만약 단일 에이전트로 모든 일을 처리했다면, 조사 자료가 한 컨텍스트에 들어가 30~50K 토큰을 차지했을 수 있습니다.

마지막으로, **subgraph 설계의 좋은 신호와 반대 신호**입니다.

```
좋은 신호                                  반대 신호
입력·출력이 명확히 정의                     인터페이스가 흐릿함
한 가지 책임에 집중                         여러 책임이 섞임
내부 변경이 외부에 영향 없음                작은 변경이 supervisor 흔듦
다른 시스템에서도 재사용 가능               이 시스템 전용
단위 평가가 가능                            평가가 전체 시스템에 의존
```

정리하면, subgraph와 위임은 큰 작업을 모듈화된 묶음으로 나누고 supervisor가 그것을 조율하는 계층적 패턴입니다. 입출력 스키마, subgraph 단위 평가, 재진입 방지, MCP 권한 분리가 모듈화를 지속 가능하게 만드는 핵심 도구입니다.

다음 섹션부터는 이 패턴을 실제 프레임워크(LangGraph·AutoGen·CrewAI)와 MCP의 결합으로 옮겨 살펴봅니다.
