# 2.2.3 Session 관리와 재연결

앞 단원 2.2.2에서 Streamable HTTP의 메시지 흐름을 살펴보았습니다. 이 단원에서는 세션을 어떻게 발급하고, 만료하고, 재연결 시 이어 가는지를 다룹니다. 세션 관리는 단순한 헤더 한 줄 같지만, 실제 운영에서는 보안·일관성·확장성을 함께 결정짓는 핵심 설계점입니다.

세션의 시작은 호스트의 첫 initialize 요청입니다. 서버는 그 응답 헤더에 **Mcp-Session-Id**를 발급합니다. 식별자의 구체적 형식은 사양이 규정하지 않으며, 충돌 가능성이 매우 낮은 무작위 문자열이면 충분합니다. UUIDv4, ULID, 또는 충분한 엔트로피를 가진 임의 토큰이 흔히 쓰입니다.

```
HTTP/1.1 200 OK
Content-Type: application/json
Mcp-Session-Id: 9c1f4a2e-b06d-49c2-99c1-5b6a0fa1e2b7

{"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2025-06-18", ...}}
```

호스트는 이후 모든 요청 헤더에 같은 값을 함께 보냅니다. 서버는 이 식별자로 세션 상태를 조회하고, 필요한 정보(구독, 진행 중인 호출, 인증 컨텍스트)를 그 위에서 다룹니다.

세션 식별자는 무작위 토큰이지만 **신뢰의 단위**가 됩니다. 식별자를 알면 그 세션에 메시지를 보낼 수 있으므로, 식별자는 인증 토큰만큼은 아니어도 그에 준하는 비밀로 다뤄야 합니다. 외부 로그에 남기지 않거나, 클라이언트가 안전한 저장소에 보관해야 합니다. 사양은 식별자 자체를 추측 불가능한 무작위 값으로 만들 것을 권합니다.

**세션 만료**는 두 가지 경로로 일어납니다. 첫째, 호스트가 명시적으로 세션을 끝내는 경우입니다. 호스트는 DELETE 메서드로 세션을 종료할 수 있고, 서버는 이를 받으면 관련 상태를 정리합니다. 둘째, 서버가 자체 정책으로 만료시키는 경우입니다. 일정 시간 활동이 없으면, 서버는 다음 요청에 410 Gone으로 응답해 세션이 만료되었음을 알립니다. 호스트는 410을 받으면 새 initialize로 새 세션을 시작합니다.

```
HTTP/1.1 410 Gone
Content-Type: application/json

{"error": {"code": "session_expired", "message": "Session has expired."}}
```

**재연결**은 두 가지 종류가 있습니다. 첫째는 **같은 세션 이어 가기**입니다. 같은 Mcp-Session-Id를 가지고 다시 접속하면, 서버가 그 세션을 알고 있으면 그대로 이어 갑니다. 진행 중이던 자원 구독이 유지되고, sampling 요청도 같은 채널 위에서 흐릅니다. 둘째는 **이벤트 재전송**입니다. SSE 스트림이 끊겼다가 다시 연결될 때, 호스트가 `Last-Event-ID` 헤더에 마지막으로 받은 이벤트 ID를 담아 보내면, 서버는 그 이후의 이벤트만 다시 보내 줍니다. 두 메커니즘은 함께 동작해, 짧은 네트워크 단절 정도는 호스트가 인지하지 못한 채 매끄럽게 회복할 수 있습니다.

이벤트 ID 정책은 서버가 결정합니다. 단조 증가하는 정수, 시간 기반 ULID, UUID 등 다양한 선택이 가능합니다. 중요한 것은 같은 세션 안에서 ID가 유일하고 순서를 가지며, 일정 시간 동안은 재전송이 가능하도록 ID와 이벤트를 함께 보관해 두는 것입니다. 무한정 보관할 수는 없으므로 보통 5분, 15분 같은 윈도우를 두고, 그 윈도우를 벗어난 ID는 재전송이 불가능하다는 신호(예: 새 SSE 채널을 시작하면서 첫 이벤트부터 흘림)를 보냅니다.

여기서 **무상태 서버**라는 선택이 등장합니다. 서버가 세션 상태를 전혀 보관하지 않도록 설계할 수도 있습니다. 그 경우 호스트는 매 요청에 필요한 모든 정보를 보내야 하고, 서버는 그 정보만으로 호출을 처리합니다. 예를 들어 단순 검색 도구만 노출하는 서버라면 세션 상태가 거의 필요 없습니다. 인증 토큰과 호출 본문만 있으면 충분합니다. 이 모델은 인스턴스 수평 확장이 매우 쉽다는 장점이 있습니다.

반대로 자원 구독, sampling, progress 같은 양방향 흐름이 많은 서버는 어느 정도 세션 상태를 가질 수밖에 없습니다. 이 상태를 어디에 저장하느냐가 운영의 갈래입니다. 인메모리에 두면 빠르지만 단일 인스턴스에 묶입니다. Redis 같은 공유 저장소에 두면 여러 인스턴스가 함께 쓰지만 응답 지연이 늘어납니다. 서버리스 환경이라면 외부 저장소가 사실상 강제됩니다.

**재전송 윈도우**의 크기 결정도 운영 결정입니다. 클라이언트가 5분 이상 끊겨 있다가 돌아올 가능성이 있는가, 그 동안 발생하는 이벤트의 메모리 비용이 어느 정도인가에 따라 달라집니다. 흔히 쓰는 값은 1분에서 10분 사이입니다. 너무 짧으면 짧은 네트워크 점프에도 재시작이 일어나고, 너무 길면 메모리·저장소 비용이 비대해집니다.

세션 식별자는 또 다른 미묘한 책임을 가집니다. **로드밸런서와의 호환**입니다. 인메모리 세션을 채택했다면 같은 세션의 요청이 같은 인스턴스에 도달해야 합니다. 로드밸런서에 sticky session(예: 쿠키 기반 또는 헤더 기반)을 설정하거나, Mcp-Session-Id를 해시한 일관성 라우팅을 적용해야 합니다. 공유 저장소 모델을 채택했다면 sticky가 불필요합니다. 자세한 프록시 설정은 2.3.2에서 다룹니다.

**보안적 고려**도 빼놓을 수 없습니다. 세션 식별자가 노출되면 누가 그 세션의 권한으로 메시지를 보낼 수 있습니다. 따라서 다음을 함께 적용하는 편이 안전합니다. 첫째, 모든 통신은 TLS 위에서만 한다. 둘째, 세션 식별자에 더해 OAuth bearer token을 함께 검증한다. 셋째, 세션과 사용자 식별자를 함께 묶어, 다른 사용자의 토큰으로 같은 세션 식별자를 보내도 거부한다.

세션 정리도 자원 관점에서 중요합니다. 호스트가 명시적 DELETE 없이 사라지는 경우, 서버는 일정 시간 후 세션을 폐기해야 합니다. 그렇지 않으면 메모리·구독·외부 자원이 누수됩니다. 이 정리는 보통 백그라운드 작업이나 TTL 기반 저장소로 구현합니다.

```
세션 라이프사이클

  initialize → Mcp-Session-Id 발급
       │
       ▼
  운영 단계 (POST, GET, SSE, 알림)
       │
       ├── 짧은 단절 → Last-Event-ID로 재전송
       ├── 장기 비활동 → 410 Gone, 호스트가 새 initialize
       ├── 명시적 DELETE → 즉시 정리
       └── 호스트 사라짐 → 서버가 TTL로 정리
```

이 라이프사이클을 머리에 두고 있으면, 운영 중에 마주치는 에러를 빠르게 해석할 수 있습니다. 410은 "세션이 만료되었으니 새로 시작하라"는 신호이고, 404는 "그 세션 식별자를 모른다"는 신호이며, 401은 "인증이 빠지거나 잘못되었다"는 신호입니다. 클라이언트 SDK는 보통 이 신호들을 해석해 자동 재초기화를 수행하지만, 자기 구현을 만들 때는 이 동작을 직접 구현해야 합니다.

정리하면, Streamable HTTP의 세션 관리는 Mcp-Session-Id 발급, 410·DELETE를 통한 명시·암묵 만료, Last-Event-ID 재전송, 무상태·인메모리·공유 저장소 사이의 상태 모델 선택으로 구성됩니다. 이 결정들이 결합되어 한 서버의 확장성·일관성·운영 비용이 결정됩니다.

다음 단원인 2.3.1에서는 한 발 더 물러나, stdio와 Streamable HTTP 사이에서 어느 transport를 언제 고를지를 정리합니다.
