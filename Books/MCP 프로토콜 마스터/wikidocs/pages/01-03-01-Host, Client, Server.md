# 1.3.1 Host, Client, Server

앞 단원들에서 다섯 프리미티브의 책임을 살펴보았습니다. 이 단원에서는 그 프리미티브들이 동작하는 무대, 즉 MCP의 3계층 아키텍처를 정리합니다. 사양은 세 주체를 정의합니다. **Host**, **Client**, **Server**입니다. 이름이 비슷해 헷갈리기 쉬우니 책임을 명확히 짚고 시작하겠습니다.

**Host**는 사용자가 직접 마주하는 AI 애플리케이션 전체입니다. Claude Desktop, Cursor, VS Code의 Copilot Chat, ChatGPT Desktop 같은 도구가 호스트입니다. 호스트는 LLM과의 대화, 사용자 인터페이스, 정책 결정, MCP 연결 관리를 모두 책임집니다. 다른 말로 하면, 사용자는 호스트만 인식하며 그 뒤편의 MCP는 보이지 않습니다.

**Client**는 호스트 내부에서 하나의 외부 연결을 담당하는 모듈입니다. 호스트가 서버 N개와 연결되어 있다면 클라이언트 인스턴스도 N개가 있습니다. 호스트와 1:N, 그리고 각 클라이언트와 서버는 1:1 관계입니다. 이렇게 분리한 이유는 격리 때문입니다. 한 서버에서 발생한 오류나 메시지가 다른 서버에 영향을 주지 않도록, 각 연결을 별도 객체로 묶어 둡니다. 또한 capability 협상도 클라이언트 단위로 일어납니다. 한 클라이언트가 sampling을 지원한다고 해서 다른 클라이언트도 지원해야 하는 것은 아닙니다.

**Server**는 도구·자원·프롬프트를 노출하는 별도의 프로세스이거나 원격 서비스입니다. 서버는 자신의 단일 책임에 집중합니다. 파일 시스템 서버, GitHub 서버, Slack 서버, 데이터베이스 서버처럼 도메인 단위로 만들어집니다. 한 서버가 모든 일을 하려고 들면 도구 목록이 비대해지고 권한 관리가 어려워집니다. MCP는 작은 서버 여러 개를 호스트에서 묶어 쓰는 모델을 권장합니다.

이 셋의 관계를 한 그림으로 정리하면 다음과 같습니다.

```
+-------------------------- Host (Claude Desktop) --------------------------+
|                                                                          |
|   사용자 UI / 모델 호출 / 정책 결정                                       |
|                                                                          |
|   +--- Client 1 ---+  +--- Client 2 ---+  +--- Client 3 ---+              |
|   |                |  |                |  |                |              |
|   +-------|--------+  +-------|--------+  +-------|--------+              |
|           |                   |                   |                       |
+-----------|-------------------|-------------------|-----------------------+
            |                   |                   |
   +------- v -------+ +-------- v --------+ +------- v -------+
   | Filesystem MCP  | | GitHub MCP Server | | Postgres MCP    |
   | Server          | | (remote)          | | Server          |
   +-----------------+ +-------------------+ +-----------------+
```

이 그림에서 화살표가 양방향임을 주목하세요. 호스트(클라이언트)가 서버에게 도구·자원 호출을 보내지만, 서버도 호스트에게 sampling 요청이나 알림을 보냅니다. 또한 호스트는 사용자와의 상호작용을 책임지므로, 서버는 사용자에게 직접 메시지를 보낼 수 없습니다. 사용자 동의가 필요한 모든 일은 호스트가 중개합니다.

이 구조의 가장 큰 장점은 **단일 책임**입니다. 호스트는 UI와 정책에 집중하고, 서버는 도메인 기능에 집중합니다. 새 도메인 기능을 추가하고 싶으면 새 서버를 만들면 됩니다. 호스트 코드는 손볼 필요가 없습니다. 반대로 호스트가 새 기능을 추가해도 서버는 그대로 동작합니다. 표준이 잘 자리잡았다는 신호 중 하나는 양쪽이 서로 모른 채로 발전할 수 있다는 점입니다.

또 다른 장점은 **권한 경계의 명확성**입니다. 각 서버는 자기 도메인의 권한만 가집니다. 파일 시스템 서버에 부여한 권한이 GitHub 서버로 새지 않습니다. 사용자 동의도 서버별로 받게 됩니다. 한 서버가 망가지거나 악의적으로 동작해도, 다른 서버의 자원에는 손댈 수 없습니다.

이 모델을 처음 접하면 한 가지 의문이 듭니다. "왜 호스트가 직접 서버에 붙지 않고 굳이 클라이언트라는 한 단계를 두는가." 이 의문에 대한 답은 미래 확장성에 있습니다. 호스트는 여러 모델, 여러 사용자, 여러 컨텍스트를 동시에 다룰 수 있어야 하고, 각 연결은 독립적으로 살아 있어야 합니다. 클라이언트를 일급 객체로 분리해 두면, 호스트가 연결을 동적으로 추가·제거하거나 같은 서버에 여러 연결을 맺는 일이 자연스럽게 표현됩니다.

실제 호스트 사례를 살펴봅시다. **Claude Desktop**은 설정 파일(`claude_desktop_config.json`)에 서버 목록을 두고, 앱이 시작될 때 각 서버를 별도의 클라이언트로 띄웁니다. stdio 기반 서버는 자식 프로세스로 띄우고, 원격 서버는 HTTP 클라이언트를 만듭니다. **VS Code**의 Copilot Chat은 workspace 단위로 MCP 설정을 가지며, 워크스페이스가 활성화될 때 서버들을 시작합니다. **Cursor**는 비슷한 방식이지만 settings.json 안에 mcpServers 필드를 둡니다.

서버 사례도 다양합니다. Anthropic이 공개한 참조 서버에는 filesystem, git, github, gitlab, sqlite, postgres, brave-search, puppeteer, slack 등이 있습니다. 이 서버들은 NPM 또는 PyPI 패키지로 배포되며, 호스트 설정에 명령어를 등록하기만 하면 사용할 수 있습니다. 커뮤니티가 만든 서버까지 합치면 수백 개에 이르고, 한국어를 지원하는 노션, 카카오, 네이버 서비스 연결 서버도 등장하고 있습니다.

3계층 구조의 한 가지 미묘한 함정은 **호스트의 자유도**입니다. 호스트가 사용자 동의를 어떻게 처리할지, 도구 호출을 자동 승인할지 일일이 물을지, 어떤 모델을 쓸지는 모두 호스트의 결정입니다. 그래서 같은 서버라도 호스트에 따라 사용자 경험이 크게 달라질 수 있습니다. 서버 개발자는 호스트의 차이를 가정하지 말고, 모든 호스트에서 안전하게 동작하도록 보수적으로 만드는 것이 좋습니다.

정리하면, MCP의 3계층은 호스트(사용자와 정책), 클라이언트(연결 단위 격리), 서버(도메인 기능)로 책임이 나뉜 구조입니다. 각 계층이 단일 책임을 지키기 때문에 새 서버·새 호스트가 등장해도 다른 쪽 코드가 깨지지 않습니다.

다음 단원인 1.3.2에서는 호스트와 서버가 처음 만나는 순간에 어떤 협상이 일어나는지, 곧 capability negotiation을 살펴봅니다.
