# 3.2.1 DCR 등장 배경과 RFC 7591

앞 단원에서 자원 서버와 인증 서버가 어떻게 분리되어 동작하는지 살펴보았습니다. 그런데 한 가지 큰 가정을 따져 봐야 합니다. OAuth는 보통 **클라이언트가 사전 등록되어 있다고 가정**합니다. 즉, AS는 client_id 목록과 그 메타데이터를 미리 알고 있어야 합니다. 이 가정이 깨지는 곳이 바로 MCP의 세계입니다. 이 단원에서는 **Dynamic Client Registration**(DCR)이 왜 필요해졌는지, 그리고 그 표준인 RFC 7591이 무엇을 정의하는지를 정리합니다.

전통적인 SaaS를 떠올려 봅시다. 회사가 새 API를 OAuth로 노출할 때, 클라이언트 앱은 비교적 한정적입니다. 회사 내부 앱 두세 개, 외부 파트너 몇 곳 정도입니다. 각 클라이언트는 개발자 포털에서 client_id를 신청하고, 리다이렉트 URI 같은 정보를 등록합니다. 사람이 직접 운영하는 절차로 충분합니다.

MCP는 다른 풍경입니다. **잠재적 클라이언트가 무한**합니다. Claude Desktop, Cursor, VS Code Copilot Chat, Continue, Aider, 사내 자동화 봇, 한 번 쓰고 버리는 사용자 스크립트까지. 새 호스트가 매주 등장하고, 한 사용자가 자기만의 호스트를 만드는 일도 흔합니다. 이 호스트들을 모두 모든 MCP 서버에 사전 등록하라고 요구하는 것은 운영적으로 불가능합니다.

해결책은 두 가지가 있을 수 있습니다. 첫째, 모든 클라이언트가 같은 client_id를 공유한다. 둘째, 클라이언트가 등장하는 그 순간에 자신을 동적으로 등록한다. 첫 번째 모델은 client_id의 본래 의도를 무너뜨립니다. 누가 어떤 호스트로 접근했는지 구분이 불가능해지고, 한 호스트의 보안 사고가 모든 호스트로 번집니다. MCP 사양은 두 번째 모델, 곧 **Dynamic Client Registration**을 표준 답으로 채택했습니다.

DCR은 새 사양이 아닙니다. **RFC 7591**(OAuth 2.0 Dynamic Client Registration Protocol)이 2015년에 표준화한 흐름입니다. AS가 `registration_endpoint`를 노출하고, 클라이언트가 그 엔드포인트에 자기 메타데이터를 POST하면 AS는 client_id를 발급해 돌려줍니다.

```
POST /register HTTP/1.1
Host: auth.example.com
Content-Type: application/json

{
  "client_name": "Claude Desktop",
  "redirect_uris": ["http://127.0.0.1:43253/callback"],
  "grant_types": ["authorization_code", "refresh_token"],
  "response_types": ["code"],
  "token_endpoint_auth_method": "none",
  "scope": "read:notes write:notes"
}

HTTP/1.1 201 Created
Content-Type: application/json

{
  "client_id": "5f8e...",
  "client_id_issued_at": 1741320000,
  "redirect_uris": ["http://127.0.0.1:43253/callback"],
  "grant_types": ["authorization_code", "refresh_token"],
  "token_endpoint_auth_method": "none"
}
```

핵심 필드는 다음과 같습니다. `client_name`은 사용자가 동의 화면에서 볼 사람 친화적인 이름입니다. `redirect_uris`는 인가가 끝난 뒤 코드를 받을 콜백 주소들입니다. `grant_types`와 `response_types`는 이 클라이언트가 사용할 흐름을 선언합니다. `token_endpoint_auth_method`는 토큰 교환 시 클라이언트가 자신을 어떻게 증명할지(none, client_secret_post 등) 알립니다. 퍼블릭 클라이언트라면 보통 none이며, 보안은 PKCE가 책임집니다.

응답에는 발급된 `client_id`가 들어 있습니다. 클라이언트는 이후 모든 인가·토큰 교환에 이 id를 사용합니다. confidential 클라이언트라면 `client_secret`도 함께 받습니다.

RFC 7591과 짝을 이루는 **RFC 7592**(OAuth 2.0 Dynamic Client Registration Management Protocol)는 등록된 클라이언트를 조회·수정·삭제하는 인터페이스를 정의합니다. 등록 시 AS가 `registration_access_token`과 `registration_client_uri`를 함께 돌려주면, 클라이언트는 그 URL과 토큰으로 자기 등록 정보를 갱신할 수 있습니다. 예를 들어 사용자가 호스트의 redirect 포트를 바꾸면, 그 변경을 등록 정보에 반영해야 합니다.

DCR의 가장 큰 효용은 **운영 자동화**입니다. 새 호스트가 처음 어떤 MCP 서버를 만났을 때 다음 흐름이 자동으로 일어납니다.

```
1. 호스트가 MCP 서버에 호출 → 401 + WWW-Authenticate
2. 호스트가 protected resource metadata 조회
3. 호스트가 authorization server metadata 조회
4. registration_endpoint가 보이면 → DCR로 client_id 발급
5. 사용자에게 authorize 흐름 인도
6. token 교환 → MCP 호출 성공
```

사용자는 이 다섯 단계를 거의 인식하지 못합니다. 한 번 호스트가 자기 등록을 마친 뒤로는 같은 client_id를 재사용합니다. 새 MCP 서버를 만날 때마다 새 client_id가 발급되므로, 호스트는 (서버 → client_id) 매핑을 자기 저장소에 유지합니다.

DCR을 적용하면서 흔히 마주치는 운영적 결정이 몇 가지 있습니다.

첫째, **누구나 등록할 수 있게 할 것인가**입니다. 사양은 등록 자체에 별도 인증을 요구하지 않을 수도 있고(open registration), 등록에 초기 access token을 요구할 수도 있습니다(`initial_access_token`). 공개 인터넷에 노출된 AS라면, 무제한 등록은 자원 낭비와 남용 위험이 있으므로 보통 rate limit과 자동 검증을 함께 둡니다.

둘째, **소프트웨어 신원**의 부재입니다. DCR이 발급한 client_id는 등록 메타데이터를 받아들였다는 사실 외에는 그 클라이언트의 정체에 대해 아무 보장도 하지 못합니다. 누군가 "Claude Desktop"이라는 이름으로 등록해도 AS는 그것이 진짜 Anthropic의 앱인지 알 수 없습니다. 이 한계를 보완하기 위해 RFC 7591에는 `software_statement`라는 선택 필드가 있습니다. 신뢰할 수 있는 발급자가 서명한 JWT로, "이 메타데이터는 진짜다"를 증명합니다. 사용 사례에 따라 software statement를 요구하거나 받지 않을 수 있습니다.

셋째, **DCR 미지원 AS와의 호환**입니다. 사내 IdP가 DCR을 지원하지 않는 경우가 여전히 흔합니다. 이 경우 호스트는 사전에 발급된 client_id를 사용자가 입력하거나 환경 변수로 받는 fallback을 마련해야 합니다. 사양은 이 fallback을 막지 않으며, 호스트마다 자기 UI에 "고급 설정"으로 client_id 입력란을 두는 것이 흔합니다.

넷째, **등록 정보의 갱신**입니다. 호스트가 새 redirect URI나 새 scope를 필요로 할 때, 매번 새로 등록할지 기존 등록을 갱신할지를 결정해야 합니다. RFC 7592의 관리 흐름을 활용하면 갱신이 가능합니다. 다만 모든 AS가 7592를 지원하지는 않으므로, fallback으로 다시 등록하고 옛 client_id를 폐기하는 형태도 흔히 쓰입니다.

다섯째, **사용자 동의의 UX**입니다. DCR로 막 등록된 새 client_id가 보이면 사용자는 "처음 보는 앱이 자기 계정에 붙는다"는 느낌을 받을 수 있습니다. 호스트와 서버가 협력해 사용자에게 충분한 정보(클라이언트 이름, 요청 scope, redirect URI)를 보여 주는 동의 화면을 만드는 것이 신뢰의 첫 단추입니다.

정리하면, DCR은 무한히 등장하는 MCP 클라이언트를 사전 등록 없이 인증 흐름에 합류시키기 위한 표준이며, RFC 7591(등록)과 RFC 7592(관리)의 조합으로 동작합니다. 등록의 자동화는 큰 운영 이점을 주지만, 소프트웨어 신원·남용 방지·사용자 동의 UX 같은 새 과제도 함께 다루어야 합니다.

다음 단원인 3.2.2에서는 메타데이터 발견부터 MCP 호출 성공까지 이어지는 전체 DCR 흐름을 단계별로 추적합니다.
