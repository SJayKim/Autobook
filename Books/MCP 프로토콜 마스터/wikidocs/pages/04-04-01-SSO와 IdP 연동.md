# 4.4.1 SSO와 IdP 연동

게이트웨이가 회사 안에서 진짜 가치를 발휘하려면, 회사가 이미 가진 신원 인프라와 자연스럽게 결합되어야 합니다. 이 단원에서는 OIDC 기반 SSO와 IdP(Identity Provider) 연동을 MCP 게이트웨이가 어떻게 다루는지 정리합니다.

**SSO**(Single Sign-On)는 한 번의 로그인으로 여러 서비스에 인증된 상태가 되는 패턴입니다. 사용자는 자기 회사 계정으로 한 번 로그인하면, 그 세션이 살아 있는 동안 회사의 모든 SSO 통합 서비스에 별도 로그인 없이 접근할 수 있습니다. 회사 입장에서는 MFA·조건부 액세스·일관된 사용자 라이프사이클을 한 곳에서 관리할 수 있습니다.

MCP 게이트웨이에 SSO를 도입하면 두 가지가 가능해집니다. 첫째, 사용자가 MCP 도구를 처음 사용하기 시작할 때 별도의 인증을 만들지 않고 회사 계정으로 합류합니다. 둘째, 회사가 사용자를 차단·해고할 때 모든 MCP 접근이 자동으로 끊깁니다. 별도 비밀번호 회수 절차가 필요 없습니다.

기술적으로는 **OIDC**(OpenID Connect)가 표준입니다. OIDC는 OAuth 2.0/2.1 위에 사용자 정체성 계층을 얹은 사양입니다. AS가 access token에 더해 **ID Token**을 발급합니다. ID Token은 사용자에 관한 정보(sub, email, name, groups 등)를 담은 서명된 JWT입니다.

```
OIDC 흐름의 핵심 토큰

access_token   - 자원 서버 호출용 (Phase 3 그대로)
id_token       - 사용자 정체성 확인용 (new!)
refresh_token  - 토큰 갱신용
```

MCP 게이트웨이는 OIDC 인증을 거친 사용자의 ID 토큰을 받아 사용자 정체성을 확인합니다. ID 토큰의 클레임을 신뢰할 수 있는 이유는 AS가 서명했기 때문입니다. JWKS 엔드포인트에서 공개 키를 받아 서명을 검증합니다.

다음은 **Userinfo 엔드포인트**입니다. ID 토큰만으로 부족한 정보(예: 큰 그룹 목록)는 별도 API로 가져올 수 있습니다.

```
GET /userinfo HTTP/1.1
Host: auth.example.com
Authorization: Bearer <access_token>

HTTP/1.1 200 OK
{
  "sub": "u_alice",
  "email": "alice@acme.com",
  "name": "Alice Park",
  "groups": ["engineering", "wiki-editors", "admins"],
  "department": "Platform"
}
```

게이트웨이는 이 정보를 캐시하고 정책 평가에 사용합니다.

다음은 **그룹·역할 클레임 매핑**입니다. 회사 IdP는 사용자를 그룹·역할로 분류해 둡니다. 게이트웨이는 그 그룹·역할을 자기 정책 모델에 매핑합니다.

```
IdP 그룹              게이트웨이 역할             정책 결과
engineering            developer                 GitHub·Wiki 도구 허용
finance                finance-user              accounting 도구 허용
ops                    operator                  관리·진단 도구 허용
*                      basic-user                기본 read-only 도구
```

매핑은 별도 설정 파일이나 정책 엔진 안에 둡니다. 새 그룹이 IdP에 생기면 매핑 갱신만으로 새 권한이 적용됩니다.

다음은 **JIT 프로비저닝**(Just-in-Time Provisioning)입니다. 사용자가 처음 게이트웨이에 합류할 때, IdP의 정보로 게이트웨이 측 사용자 레코드를 자동 생성합니다. 사전 등록이 필요 없습니다. 사용자가 회사에 입사한 즉시 첫 SSO 로그인으로 게이트웨이에도 정상 등록됩니다.

JIT의 장점은 운영 부담이 없다는 점이고, 단점은 사용자가 어떤 권한을 받게 될지가 IdP의 그룹 매핑에 전적으로 의존한다는 점입니다. 매핑이 잘못되면 새 사용자가 의도하지 않은 권한을 받을 수 있습니다. 따라서 JIT을 채택할 때는 매핑을 정기적으로 점검하는 의식이 필요합니다.

**MFA 강제와 step-up**도 SSO의 큰 이점입니다. 게이트웨이는 IdP에 사용자 인증의 강도를 요구할 수 있습니다.

```
경우                                요구 인증
일반 read-only 도구                  기본 로그인
write 도구                           MFA 통과 토큰
admin 도구                           최근 5분 이내 MFA
민감 데이터 도구                     하드웨어 키
```

OIDC의 **acr_values**와 **acr/amr** 클레임으로 이 요구를 표현합니다. 게이트웨이가 호출이 요구하는 인증 강도를 충족하지 못하는 토큰을 거부하면, 호스트는 사용자에게 추가 인증을 유도하는 흐름(step-up)을 시작합니다.

다음은 **조건부 액세스**입니다. IdP는 사용자 위치, 디바이스 상태, 네트워크 출처에 따라 인증을 허용·거부·강화할 수 있습니다. 게이트웨이는 이 결정을 IdP에 위임하면 됩니다. 예를 들어 회사 VPN 밖에서 들어온 사용자에게 MFA를 강제하는 정책을 IdP가 적용하면, 게이트웨이는 그 결과로 발급된 토큰만 받게 됩니다.

다음은 **사용자 라이프사이클 통합**입니다. 회사가 직원 이탈 시 IdP에서 계정을 비활성화하면, 그 사용자의 SSO 세션이 종료됩니다. 다음 access token 갱신 시점에 토큰이 더 이상 발급되지 않으므로, 게이트웨이로의 접근도 자동으로 차단됩니다. 다음 보완 장치를 두면 더 빠르게 차단됩니다.

```
1. 토큰 introspection 활성화
2. 짧은 access token 수명(5~10분)
3. IdP의 "force logout" 이벤트 구독
4. 게이트웨이 측 short-circuit 차단 목록
```

조직 디렉토리 동기화는 SCIM(System for Cross-domain Identity Management) 같은 표준으로 자동화할 수 있습니다. SCIM 이벤트를 받아 게이트웨이가 사용자·그룹 상태를 즉시 갱신합니다.

다음은 **다중 IdP 연동**입니다. 큰 조직은 부서·자회사별로 다른 IdP를 가질 수 있습니다. 게이트웨이는 여러 IdP를 동시에 신뢰할 수 있어야 합니다.

```
- IdP 디스커버리: 사용자가 처음 로그인 시 어디에 속하는지 선택
- 이슈어별 분기: ID 토큰의 iss로 어느 IdP의 사용자인지 식별
- 통합 그룹 매핑: 다른 IdP의 그룹을 같은 게이트웨이 역할에 매핑
- 분리 테넌트: IdP별로 다른 정책·자원 묶음 적용
```

다음은 **에이전트·자동화의 정체성**입니다. 사람 사용자뿐 아니라 봇·서비스가 게이트웨이를 호출하는 경우가 있습니다. OAuth의 client_credentials 흐름을 사용해 봇이 자기 정체성으로 인증하도록 합니다. 이 경우 ID 토큰은 없지만, access token의 sub 클레임을 봇 식별자로 둡니다. 봇에도 별도 그룹·역할을 매핑해 정책을 적용합니다.

**감사 측면**도 SSO 통합으로 강화됩니다. 게이트웨이의 모든 감사 로그가 IdP의 사용자 id로 묶이므로, 회사 SIEM에서 한 사용자의 활동을 다른 시스템과 함께 추적할 수 있습니다. 데이터 누수 사고가 의심되면, "이 사용자가 IdP·게이트웨이·다른 SaaS에서 무엇을 했는가"를 한 줄로 묶어 분석할 수 있습니다.

마지막으로 **사용자 경험**입니다. 잘 통합된 SSO 게이트웨이의 사용자 경험은 다음과 같습니다.

```
1. 사용자가 호스트(예: Claude Desktop)에서 회사 MCP 서버 URL 추가
2. 첫 호출 시 브라우저가 자동으로 회사 IdP로 이동
3. 회사 계정으로 로그인 (이미 다른 SaaS에 로그인되어 있으면 자동 통과)
4. 동의 화면에서 scope 확인
5. 호스트로 돌아와 도구 사용 가능
6. 두 번째 호출부터는 사용자가 이 흐름을 인식하지 않음
```

이 흐름이 끊김 없이 자연스러우려면 redirect 처리(3.2.3), DCR(3.2.2), 그리고 호스트의 OS 권한이 모두 잘 작동해야 합니다.

체크리스트로 정리합니다.

```
1. OIDC AS와 통합되어 ID 토큰 검증
2. JWKS 자동 갱신
3. 그룹·역할 매핑 정책
4. JIT 프로비저닝과 정기 매핑 점검
5. MFA·step-up 요구 정책
6. 조건부 액세스 위임
7. SCIM·이벤트 기반 사용자 라이프사이클 동기화
8. 다중 IdP 지원과 이슈어 분기
9. 봇·서비스 정체성 표현
10. SIEM 통합 감사 추적
```

정리하면, SSO와 IdP 연동은 회사의 신원 인프라를 게이트웨이의 정책·감사·라이프사이클에 그대로 흡수시키는 작업입니다. OIDC, JWKS, 그룹 매핑, JIT 프로비저닝, MFA 요구, SCIM 동기화가 이 통합의 핵심 부품이며, 다중 IdP와 봇 정체성까지 함께 다뤄야 큰 조직에서 안정적으로 동작합니다.

다음 단원인 4.4.2에서는 이 모든 통합을 규제 관점에서 정리한 컴플라이언스 체크리스트를 살펴봅니다.
