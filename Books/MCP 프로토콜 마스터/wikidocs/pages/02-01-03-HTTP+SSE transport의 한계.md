# 2.1.3 HTTP+SSE transport의 한계

stdio(2.1.2)가 로컬 도구에 알맞다면, 원격이나 다중 사용자 환경에서는 다른 transport가 필요합니다. MCP의 첫 사양인 2024-11-05가 함께 정의한 것이 **HTTP+SSE** transport입니다. 결과적으로 이 방식은 2025-03-26 사양에서 새 transport로 대체되었지만, 그 한계를 이해해 두면 차세대 transport의 설계 의도를 더 분명히 볼 수 있습니다.

HTTP+SSE는 두 개의 엔드포인트를 사용합니다. 하나는 호스트 → 서버 방향의 메시지를 받는 POST 엔드포인트이고, 다른 하나는 서버 → 호스트 방향의 메시지를 흘려보내는 **Server-Sent Events**(SSE) 엔드포인트입니다. 양방향 채널을 두 개의 단방향 HTTP 흐름으로 흉내 내는 모양새입니다.

```
호스트                                      서버
  |                                         |
  |--- GET /sse  -------------------------> |
  |<-- 200, Content-Type: text/event-stream |
  |                                         |
  |<-- event: message  (서버→호스트 메시지) |
  |<-- event: message  (서버→호스트 메시지) |
  |                                         |
  |--- POST /message --(호스트→서버) -----> |
  |<-- 202 Accepted                         |
  |                                         |
  |<-- event: message  (그 응답)            |
```

SSE는 HTTP 위에서 가장 단순하게 서버 푸시를 구현하는 표준입니다. 클라이언트가 한 번 GET 요청을 보내면, 서버는 응답을 끝내지 않고 그 위에 "event: ...\ndata: ...\n\n" 형식의 이벤트를 계속 흘려보냅니다. WebSocket과 비교해 단순하고, HTTP 인프라(프록시·로드밸런서·인증)를 그대로 활용할 수 있다는 장점이 있습니다.

호스트는 서버에게 메시지를 보낼 때 POST /message로 보냅니다. 그러면 서버는 그 응답으로 곧장 결과를 돌려주는 것이 아니라, 잠시 후 SSE 채널 위에 결과 이벤트를 흘려보냅니다. 결과를 받기 위해 호스트는 이미 SSE 연결을 가지고 있어야 합니다. 한쪽이 끊기면 양쪽이 모두 망가집니다.

이 구조가 처음 사양에 채택된 이유는 명확합니다. WebSocket 같은 새 기술 없이도 HTTP만으로 양방향 통신을 흉내 낼 수 있고, 대부분의 HTTP 인프라에 별다른 설정 없이 올라갑니다. 초기 MCP 서버들은 빠르게 원격 배포를 실험해 볼 수 있었습니다.

그러나 운영 단계에 진입하면서 여러 한계가 드러났습니다.

첫 번째 한계는 **장수명 SSE 연결의 부담**입니다. SSE는 한 번 열리면 세션이 살아 있는 동안 계속 유지되어야 합니다. 사용자가 IDE를 켜 두고 있는 시간 내내 연결이 열려 있는 셈입니다. 이 모델은 단순한 함수 호출만 처리하는 도구라면 과한 자원 낭비입니다. 단 한 번의 요청·응답을 위해서도 호스트는 미리 SSE 연결을 열어야 하기 때문입니다.

두 번째 한계는 **재연결 시 컨텍스트 손실**입니다. SSE 연결이 네트워크 이슈로 끊어졌을 때, 호스트가 새 연결을 만들면 서버는 그 호스트가 이전 세션의 누구인지 모릅니다. SSE 자체는 Last-Event-ID라는 재전송 메커니즘을 가지고 있지만, MCP 첫 사양은 이를 명시적으로 활용하지 않았기 때문에 끊김은 곧 세션 재시작이 되는 경우가 많았습니다.

세 번째 한계는 **서버리스 친화성 부족**입니다. AWS Lambda나 Cloudflare Workers 같은 서버리스 환경은 짧은 함수 실행을 가정합니다. 장수명 SSE 연결은 이 모델에 잘 맞지 않습니다. 연결을 유지하기 위해 EC2나 별도 컨테이너를 띄워야 한다면, MCP 서버 배포가 갑자기 무거워집니다. 원격 MCP를 폭넓게 보급하려면 더 가벼운 모델이 필요했습니다.

네 번째 한계는 **로드밸런서·프록시 호환성**입니다. SSE 연결은 본질적으로 한 서버 인스턴스에 묶이는 sticky한 흐름입니다. 로드밸런서가 라운드 로빈으로 요청을 분배하면, POST /message가 SSE 연결과 다른 인스턴스에 도착해 매칭에 실패할 수 있습니다. 이를 막으려면 sticky session이나 외부 메시지 버스 같은 추가 인프라가 필요한데, 단순한 도구 하나를 노출하기 위해 이런 설계를 도입하는 것은 부담스럽습니다.

다섯 번째 한계는 **두 엔드포인트 운영의 번거로움**입니다. POST와 SSE가 다른 엔드포인트라는 사실은 미묘한 운영 부담을 만듭니다. 인증·관측·정책을 두 엔드포인트에 동일하게 적용해야 하고, 도메인이 다르면 CORS 같은 문제를 별도로 다루어야 합니다.

이런 한계가 누적되면서 MCP 커뮤니티에서는 새 transport 논의가 본격적으로 시작되었습니다. GitHub 사양 리포지토리의 Discussion #206이 가장 활발했고, 거기서 단일 엔드포인트로 단순화하면서도 SSE 스트리밍의 장점을 살리는 형태가 제안되었습니다. 그 결과가 2025-03-26 사양에 등장한 **Streamable HTTP**입니다. 다음 섹션에서 자세히 다루겠지만, 핵심은 한 엔드포인트에서 POST와 GET을 함께 받으며, 응답을 즉시 돌려주거나 SSE 스트림으로 흘려보낼 수 있도록 만들었다는 점입니다.

HTTP+SSE는 초기 사양에 잠시 머물렀지만 완전히 사라지지는 않았습니다. 첫째, 이미 배포된 일부 서버는 후방 호환을 위해 HTTP+SSE를 유지합니다. 둘째, 새 사양도 일부 단순한 경우에는 SSE 비슷한 흐름을 그대로 활용합니다. 따라서 이 transport는 사라진 것이 아니라 흡수되었다고 보는 편이 정확합니다. 마이그레이션 경로는 2.2.1에서 함께 살펴봅니다.

운영자 관점에서 한 가지 교훈을 정리해 둘 만합니다. transport 설계는 단순히 메시지를 옮기는 통로의 문제가 아니라, 운영 환경(서버리스·로드밸런서·관측)과 만나는 접점이라는 점입니다. 사양이 만들어지는 단계에서 이상적이라 여겨졌던 설계도, 실제 다양한 환경에서 부딪힌 뒤에 빠르게 변할 수 있습니다. MCP의 첫 사양과 그 직후의 갱신은 이 사실을 보여 주는 사례입니다.

정리하면, HTTP+SSE transport는 POST와 SSE 두 엔드포인트로 양방향성을 흉내 냈으나, 장수명 연결·재연결 컨텍스트 손실·서버리스 부적합·로드밸런서 까다로움·이중 엔드포인트 운영이라는 다섯 한계가 누적되면서 새 transport에게 자리를 내주었습니다.

다음 단원인 2.2.1에서는 Streamable HTTP가 어떤 의도로 등장했는지를 살펴봅니다.
