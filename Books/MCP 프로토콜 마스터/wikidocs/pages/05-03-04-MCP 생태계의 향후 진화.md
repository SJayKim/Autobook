# 5.3.4 MCP 생태계의 향후 진화

이 책의 마지막 단원으로, 지금까지 다룬 MCP의 골격이 앞으로 어떻게 진화할지를 정리합니다. 사양은 빠르게 움직이고 있고, 생태계의 도구·게이트웨이·프레임워크가 그 위에 빠르게 쌓이고 있습니다. 이 단원에서는 최근 사양 변경의 흐름, 표준이 단단해지고 있는 영역, 그리고 우리가 따라가야 할 우선순위를 짚어 봅니다.

먼저 **사양 진화의 큰 흐름**을 정리합니다.

```
사양 버전           큰 변화
2024-11-05          첫 공개: stdio + HTTP+SSE, Tools/Resources/Prompts
2025-03-26          Streamable HTTP, OAuth 2.1 표준화, batch 일시 도입
2025-06-18          Elicitation, structured tool output, resource indicator
이후                 Registry, Agent-to-Agent 논의, signed tool manifest
```

세 차례 사양 갱신은 각각 다른 문제를 다루었습니다. 첫 사양은 골격을 세웠고, 둘째는 원격 운영의 현실적 요구를 반영했고, 셋째는 보안·표현력을 강화했습니다. 사양의 흐름을 보면 다음 세 방향이 분명합니다.

```
1. 원격·다중 사용자 환경에 강함을 더하기
2. 보안과 사용자 동의의 정밀도 높이기
3. 도구·자원·프롬프트 외에 새 프리미티브 도입
```

다음은 최근 추가된 핵심 기능 몇 가지입니다.

**Elicitation**은 서버가 사용자에게 추가 정보를 요청하는 흐름입니다. 호출 도중 "이 작업을 위해 사용자 이메일이 필요하다"고 서버가 요청하면, 호스트가 그 입력을 받아 다시 흘려보냅니다. Sampling이 LLM 호출을 위임받는 것이라면, Elicitation은 사용자 입력을 위임받는 것입니다. 이 기능은 도구가 다단계 사용자 상호작용을 표현할 수 있게 해 줍니다.

**Structured tool output**은 도구 결과의 구조를 한층 더 표준화합니다. 단순 텍스트뿐 아니라 모델이 다음 단계 결정에 직접 사용할 수 있는 JSON 스키마 기반 출력이 추가되었습니다. 도구가 자기 결과의 구조를 명시하면, 호스트와 모델 모두 그 결과를 일관되게 다룰 수 있습니다.

**Resource indicator**(RFC 8707)는 Phase 3에서 다룬 audience 명시의 표준입니다. 인가·토큰 요청에 어느 자원이 대상인지를 명시함으로써 token passthrough 위협을 줄입니다. 이는 사양의 보안 강화 흐름의 대표 사례입니다.

다음은 **표준 안정화가 진행 중인 영역**입니다.

```
영역                                 진척
Streamable HTTP                       안정
OAuth 2.1 + DCR                       안정
Audience와 resource indicator         거의 안정
Elicitation                           표준 채택, 일부 SDK에 반영
Structured tool output                표준 채택, 점진 도입
Signed tool manifest                  논의 중
MCP Registry                          제안·일부 구현
Agent-to-Agent                        제안 단계
```

이 표를 보면, 인증·transport·자원 표현은 거의 안정 단계에 들어왔고, 보안·도구 인증·에이전트 간 통신은 아직 활발한 진화 영역입니다. 새 시스템을 설계할 때는 안정 영역의 사양을 충실히 따르고, 진화 영역에는 옵션으로 길을 열어 두는 것이 안전합니다.

다음은 **Agent-to-Agent**에 대한 짧은 전망입니다. MCP가 호스트와 도구 사이의 표준이라면, Agent-to-Agent(A2A)는 에이전트와 에이전트 사이의 표준을 만들려는 시도입니다. Google이 2024~2025년에 별도 사양으로 제안했고, MCP 커뮤니티에서도 비슷한 방향의 논의가 진행 중입니다. 두 사양이 결국 어떻게 정렬될지는 아직 분명하지 않지만, 멀티 에이전트가 다른 회사의 에이전트와 협력하는 미래를 가정하면 이런 표준의 가치는 분명합니다.

다음은 **MCP Registry**입니다. 도구 카탈로그를 표준 방식으로 발견하고 설치할 수 있게 하는 시도입니다. 현재는 사용자가 직접 패키지 매니저로 stdio 서버를 설치하거나 호스트 설정에 URL을 입력하는데, 이 과정을 표준화된 디스커버리·평가·설치 흐름으로 만들려는 노력입니다. Registry가 안정화되면 도구 생태계의 검색 가능성과 신뢰성이 크게 올라갈 것입니다.

다음은 **Signed tool manifest**입니다. 3.3.3에서 다룬 tool poisoning을 정면으로 다루는 방향입니다. 신뢰된 발급자가 도구 메타데이터에 서명하면, 호스트는 서명을 검증해 변조되지 않은 도구만 모델에 노출합니다. 사양 단의 합의는 진행 중이며, 일부 게이트웨이가 자체적으로 비슷한 메커니즘을 도입하고 있습니다.

다음은 **호스트의 진화**입니다. Claude Desktop, Cursor, VS Code, ChatGPT Desktop 외에 다양한 호스트가 등장하고 있습니다. 노트북 OS의 일부로 통합되는 시도, 모바일에서 MCP를 다루는 시도, 음성 인터페이스가 MCP를 호출하는 시도가 모두 진행 중입니다. 호스트의 다양화는 MCP의 가치를 검증하는 가장 분명한 신호이며, 새 호스트가 등장할 때마다 사양의 추상이 견고한지 다시 점검됩니다.

다음은 **서버 측 도구 마켓**의 성장입니다. MCP 서버를 패키지처럼 빠르게 만들고 배포할 수 있는 도구가 등장하고 있습니다. Anthropic의 참조 서버 외에 다양한 회사·개인이 자기 서비스를 MCP 서버로 노출합니다. 이 흐름이 가속되면 도구의 양과 질 모두 의미 있게 자리잡을 것이고, 게이트웨이의 정책 엔진과 보안 검토는 더 중요해질 것입니다.

다음은 **국내 생태계**입니다. 한국어 환경에서 MCP를 도입하는 사례가 빠르게 늘고 있습니다. 노션, 카카오, 네이버 같은 서비스의 비공식·공식 MCP 서버, 사내 자동화에 MCP를 도입한 한국 기업들의 사례가 매월 등장합니다. 한국어 LLM·도구 카탈로그와 MCP의 결합은 앞으로 더 풍부해질 것으로 보입니다.

다음은 **무엇을 우선 따라잡아야 할지**의 가이드입니다.

```
우선순위                      이유
1. Streamable HTTP            모든 원격 MCP의 기반
2. OAuth 2.1 + DCR             다중 사용자 인증의 표준
3. Audience·resource indicator  새 위협 방어의 표준
4. 게이트웨이 정책·감사        엔터프라이즈 진입의 기본
5. 멀티 에이전트 평가          품질·회귀 방지의 핵심
6. Elicitation·structured out  표현력의 도약
7. Signed manifest             보안 강화
8. Registry·A2A               생태계 미래
```

새로 도입할 때 처음 4개는 거의 필수에 가깝고, 5~6번은 시스템이 멀티 에이전트로 진화하면 자연스럽게 따라옵니다. 7~8번은 적용 우선순위가 회사 상황에 따라 다르며, 사양 안정화 시점을 보고 도입을 결정합니다.

다음은 **사양 변경을 따라가는 의식**입니다.

```
의식                                   주기
GitHub modelcontextprotocol 리포 변경 추적   매주
사양 PR·Discussion 참여                매월
SDK 업데이트 적용                       SDK가 새 버전 내놓을 때마다
신규 위협 사례 학습 (보안 블로그·연구) 매주
사내 시스템에 새 사양 도입 평가         분기
```

MCP는 빠르게 움직이는 표준이므로, 한 번 잘 만든 시스템도 정기적으로 사양과 맞춰 가야 합니다. 이 의식이 정착되면 새 사양이 발표될 때마다 시스템이 자연스럽게 함께 진화합니다.

이 책의 마지막 메시지는 다음 한 문장으로 정리할 수 있습니다. **MCP는 표준 한 줄이 아니라, 그 위에 정책·관측·평가·생태계가 함께 쌓이는 운영 체계입니다.** Phase 1의 프리미티브에서 시작해 Phase 2의 transport, Phase 3의 보안, Phase 4의 거버넌스, Phase 5의 멀티 에이전트까지 단계적으로 살펴본 이유가 여기 있습니다. 어느 한 층만 깊이 다뤄도 전체 시스템이 안전하고 효율적으로 동작하지는 않습니다. 다섯 층이 함께 살아 있어야 MCP의 약속이 실제 운영에서 실현됩니다.

이 책을 마치는 시점에서 독자가 가져갈 것은 두 가지입니다. 첫째, MCP라는 표준이 무엇을 약속하고 무엇을 약속하지 않는지를 정확히 이해하는 것. 둘째, 그 위에 운영 체계를 어떻게 쌓을지를 자기 환경에 맞춰 설계할 수 있는 도구함을 갖추는 것. 그 도구함이 앞으로 새 사양이 등장할 때마다 부분 부분 교체되고 확장될 것입니다. 표준은 끝없이 변하고, 운영의 좋은 습관은 표준의 변화를 흡수하면서 살아남습니다.

정리하면, MCP 생태계는 transport·인증·새 프리미티브·signed manifest·registry·A2A 같은 영역에서 빠르게 진화하고 있으며, 안정 영역과 진화 영역을 구분해 우선순위 있게 따라가는 운영 의식이 시스템의 장기 건강을 결정합니다.

이 단원으로 책의 모든 본문을 마칩니다. 지금까지 함께 살펴본 다섯 Phase의 내용이 독자의 시스템 설계와 운영에 의미 있는 출발점이 되기를 바랍니다.

이 단원을 마치면 MCP 사양의 최신 흐름을 파악할 수 있고, 자신의 시스템에 새 기능을 도입할 우선순위를 결정할 수 있으며, 표준 변화를 흡수하는 정기 의식을 자기 팀에 도입할 수 있습니다.
