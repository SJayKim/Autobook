# 1.1.3 다른 프로토콜과의 비교

앞 단원 1.1.2에서 MCP의 정의와 범위를 정리했습니다. 그런데 LLM에 외부 기능을 연결하는 방법은 MCP가 처음이 아닙니다. OpenAI의 Function calling, OpenAPI를 활용한 도구 노출, LangChain의 Tool 추상화처럼 이미 자리 잡은 방식들이 있습니다. 이 단원에서는 그것들과 MCP가 어떻게 다른지를 정리해, MCP의 위치를 분명히 잡아 둡니다.

먼저 **Function calling**입니다. 2023년 OpenAI가 도입한 이 방식은 모델 API 호출에 `functions` 또는 `tools`라는 파라미터를 추가하여, 모델이 응답 대신 함수 호출 의도를 JSON 형태로 돌려주도록 만든 인터페이스입니다. 호스트는 모델 응답에서 함수 이름과 인자를 꺼내 실제 함수를 실행하고, 그 결과를 다시 메시지로 넣어 두 번째 호출을 합니다. 이 방식은 단순하고 강력하지만 한 가지 한계가 있습니다. 함수 목록은 API 호출에 인자로 넣는 정적 정보입니다. 함수가 늘어나면 그만큼 토큰을 더 쓰고, 다른 모델로 옮길 때는 형식을 다시 맞춰야 합니다. 한마디로 Function calling은 모델 API의 한 기능이고, MCP는 호스트와 도구 사이의 별도 채널입니다. 두 개념은 서로 배타적이지 않습니다. 실제로 많은 호스트가 모델에게 도구 목록을 알릴 때는 Function calling 형식을 쓰고, 그 뒤편에서 도구의 실제 구현은 MCP 서버에 위임합니다.

다음은 **OpenAPI**입니다. OpenAPI는 HTTP 기반 REST API의 사양을 기술하는 표준입니다. 여기에 더해 ChatGPT의 GPT Actions 같은 제품은 OpenAPI 스펙을 LLM이 읽을 수 있는 도구 정의로 변환합니다. 이 방식의 장점은 이미 회사가 가지고 있는 API를 거의 그대로 노출할 수 있다는 점입니다. 단점은 OpenAPI가 LLM 사용을 가정해 설계되지 않았다는 점에서 옵니다. 한 API가 수백 개의 엔드포인트를 가지고 있을 때, 모델에게 전부 보여 주는 것은 비현실적입니다. 또한 OpenAPI는 단방향 요청·응답에 최적화되어 있어, 자원 변경 알림이나 서버 → 클라이언트 방향의 LLM 샘플링 요청 같은 양방향 패턴을 자연스럽게 표현하지 못합니다. MCP는 처음부터 LLM과의 결합을 가정해 설계되었기 때문에, 도구를 의미 있는 단위로 나누고 양방향 메시지를 일급으로 다룹니다.

세 번째는 **LangChain Tool**과 같은 프레임워크 수준의 추상화입니다. LangChain, LlamaIndex, Haystack 같은 프레임워크는 자신만의 Tool 클래스를 정의하고, 그 안에서 어떤 외부 함수든 감쌀 수 있게 합니다. 이 방식은 같은 프레임워크 안에서는 매우 편리하지만, 한 가지 한계가 있습니다. Tool 정의가 프레임워크에 강하게 묶입니다. LangChain Tool을 AutoGen에서 그대로 쓸 수 없고, AutoGen Tool을 CrewAI에서 그대로 쓸 수도 없습니다. 따라서 도구 제공자 입장에서는 사용자 측 프레임워크마다 어댑터를 따로 만들거나, 자신의 도구를 가장 인기 있는 프레임워크에 맞춰 묶어 두어야 합니다. MCP는 프레임워크 위가 아니라 그 옆에 자리합니다. MCP 서버 하나를 만들면 langchain-mcp-adapters, AutoGen의 McpWorkbench, CrewAI의 MCPServerAdapter처럼 각 프레임워크 측 어댑터가 그것을 받아들여 자신의 Tool로 변환합니다.

이런 차이를 한 표로 정리하면 다음과 같습니다.

```
                     Function calling   OpenAPI       LangChain Tool   MCP
도구 정의 위치        모델 API 인자      HTTP 스펙     프레임워크 코드   MCP 서버
방향                  단방향(요청·응답)  단방향        단방향           양방향
동적 자원 표현        약함               약함          중간             강함
모델 간 호환성        모델별 차이 있음   중립          프레임워크 종속  중립
인증 표준             없음               별도(OAuth)   별도             OAuth 2.1
샘플링·알림           없음               없음          제한적           일급
```

이 표에서 가장 눈여겨볼 점은 도구 정의 위치입니다. Function calling은 도구 목록을 매 호출마다 모델 API에 넘기지만, MCP에서는 도구 정의가 서버 측에 살아 있습니다. 호스트가 서버에 한 번 연결되면 그 뒤로는 도구 목록을 매번 보낼 필요가 없습니다. 도구가 갱신되면 서버가 list_changed 알림을 보내고, 호스트가 그때 새 목록을 받아 갑니다. 이 차이는 토큰 비용뿐 아니라 동적 자원을 자연스럽게 표현하는 능력으로 이어집니다.

또 하나, MCP는 양방향 메시지를 일급으로 다룹니다. 서버는 도구 호출 진행 상황을 알리는 progress, 자원 변경을 알리는 list_changed, 사용자에게 LLM 호출을 요청하는 sampling 같은 메시지를 클라이언트에 보낼 수 있습니다. 이는 단방향 REST 위에서는 매끄럽지 않은 패턴입니다.

오해를 피하기 위해 한 가지 짚어 둡니다. MCP가 등장했다고 해서 Function calling이나 OpenAPI가 쓸모없어진 것은 아닙니다. 실제 시스템은 보통 둘 이상의 기술을 조합합니다. 예를 들어 호스트는 모델 API에 도구 목록을 Function calling 형식으로 전달하고, 도구의 실제 구현은 MCP 서버가 담당합니다. MCP 서버 내부에서는 다시 OpenAPI 기반의 외부 SaaS를 호출할 수 있습니다. 즉, MCP는 다른 표준을 대체한다기보다 그 위에 위치한 통합 계층입니다.

정리하면, MCP는 도구 정의를 모델 외부의 서버 측에 두고, 양방향 메시지와 표준 인증을 갖춤으로써 기존의 Function calling·OpenAPI·프레임워크 Tool이 풀지 못한 통합 비용·동적 자원·모델 간 이식성 문제를 함께 다룹니다.

다음 단원인 1.2.1부터는 MCP의 핵심 프리미티브 중 첫 번째인 Tools를 자세히 살펴봅니다.
