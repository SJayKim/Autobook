# Detecting hallucinations with LLM-as-a-judge: Prompt engineering and beyond

LLMs often confidently fabricate information, preventing teams from deploying them in many sensitive use cases and leading to high-profile incidents that degrade user trust. Despite ongoing efforts, hallucinations remain a persistent issue in recent model releases.

Datadog LLM Observability monitors LLM-based applications in production. This naturally leads to tackling the challenge of detecting hallucinations in real time.

## Augmented generation: Context is everything

Hallucination is an umbrella term for various distinct undesirable LLM behaviors. This post focuses on faithfulness, a specific requirement for non-hallucinated answers in use cases involving RAG. Determining whether an answer is faithful involves three required components:
- A question posed by an end user
- Context containing information required to answer the question, usually retrieved from an accepted knowledge base using RAG
- An answer generated by the LLM

Faithfulness is the requirement that an LLM-generated answer agrees with the context. Crucially, the question of faithfulness assumes that the context is correct and accepts it as ground truth.

## LLM-as-a-judge vs. alternative detection methods

Multiple techniques have been developed for detecting hallucinations:

White-box and gray-box methods require visibility into the generating LLM. Token probability approaches estimate confidence based on final-layer logits. Sparse autoencoder and attention-mapping approaches attempt to find neural activations correlated with hallucination.

Black-box methods can only see the input and output. Perturbation-based approaches measure reproducibility by regenerating under different conditions. SLM-as-a-judge uses small language models (BERT-style) to judge correctness. LLM-as-a-judge uses a distinct judge model to evaluate the generator model.

Datadog focuses on black-box detection to support the full range of customer use cases. Perturbation-based approaches are often prohibitively expensive (5-10x cost increase). SLM-as-a-judge begins to fail in complex use cases with large context and answer involving layers of reasoning.

## Beyond prompt engineering

LLM-as-a-judge methods have two components: the model and the prompting approach. Most literature focuses on creating better models through fine-tuning. In contrast, prompting is often overlooked.

A key principle: LLMs are better at guided summarization than complex reasoning. Therefore, breaking down a problem into multiple smaller steps of guided summarization by creating a rubric plays to the strengths of LLMs.

### Creating a rubric for detecting hallucinations

The rubric for hallucinations is a list of disagreement claims. The task is framed as finding all claims where the context and answer disagree. For each claim, the rubric requires a quote from the context and the answer, showing relevant portions of each text.

Two types of disagreements:
- Contradictions: claims in the answer that go directly against the provided context (likely strictly incorrect)
- Unsupported claims: parts of the answer not grounded in the context (may be based on other knowledge or jumping to conclusions)

### Structured output

Structured output ensures the LLM generates valid JSON adhering to the rubric format. A finite state machine (FSM) pairs with the LLM, checking generated tokens against the desired output format and zeroing out invalid token probabilities.

### Replicating reasoning behaviors

A two-step prompting approach replicates some behaviors of reasoning models through explicit prompting. Step 1: prompt an LLM to fill out the rubric without format restrictions, with instructions encouraging self-criticism and consideration of multiple interpretations. Step 2: a second LLM call with structured output to convert to the desired format (using a smaller LLM to save resources).

Word choice affects effectiveness. The context is referred to as "expert advice" and the answer as a "candidate answer", creating asymmetry that frames the context as the definitive source of truth.

## Results

Tested on two benchmarks:
- HaluBench: partially synthetic benchmark with negative examples from existing QA benchmarks and positive examples generated by prompting an LLM.
- RAGTruth: fully human-labeled benchmark covering QA, summarization, and data-to-text translation.

| Method | HaluBench F1 | RAGTruth F1 |
| Datadog approach: GPT-4o | 0.844 | 0.810 |
| Patronus prompt: Lynx (8B) | 0.836 | 0.733 |
| Patronus prompt: GPT-4o | 0.862 | 0.777 |

The Datadog detection method shows the smallest drop in F1 between HaluBench and RAGTruth, suggesting robustness as hallucinations become harder to detect.

Prompt design - not just model architecture - can significantly improve hallucination detection in RAG-based applications.
