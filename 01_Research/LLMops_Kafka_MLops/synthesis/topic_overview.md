# Topic Overview: LLMOps, Kafka, MLOps

## 핵심 개념

### LLMOps와 MLOps의 관계

MLOps는 머신러닝 모델의 전체 생명주기(데이터 수집 → 피처 엔지니어링 → 훈련 → 배포 → 모니터링)를 운영화하는 체계다. LLMOps는 이 기반 위에서 대형 언어 모델 특유의 문제(프롬프트 버전 관리, 추론 비용 최적화, 환각 감지, 가드레일)를 추가로 다룬다. 소스 002(TrueFoundry)와 010(ZenML)은 두 분야의 차이를 명시적으로 비교하며, LLMOps가 MLOps의 "슈퍼셋"이 아닌 별도의 관심사를 가진 상보적 체계임을 강조한다.

핵심 차이: MLOps는 훈련 비용 중심(무거운 학습 루프), LLMOps는 추론 비용 중심(빠른 프롬프트 반복 루프). 버전 관리 대상도 다르다 — MLOps는 모델 가중치와 데이터셋, LLMOps는 프롬프트와 임베딩.

### Apache Kafka의 역할

Kafka는 이 생태계에서 단순한 메시지 브로커를 넘어 실시간 데이터 파이프라인의 중추 역할을 한다. 소스 020(Instaclustr), 025(Kai Waehner), 060(Kai Waehner)은 Kafka가 ML 피처 파이프라인, 실시간 모델 추론 트리거, CDC(변경 데이터 캡처) 연동의 핵심 허브임을 보여준다.

Wix 사례(소스 060): Kafka → Flink SQL → Aerospike 온라인 피처 스토어 구조로 하루 700억 이벤트, 5만 토픽, 3,000개 이상 피처를 처리한다.

### 실시간 스트리밍 처리

Kafka Streams(임베디드 라이브러리, RocksDB 상태 저장)와 Apache Flink(클러스터 기반, 초당 수천만 이벤트, 서브초 지연)는 상보적 도구다. ksqlDB는 SQL 인터페이스로 스트리밍 처리를 민주화한다. Kafka Connect + Debezium은 데이터베이스 변경 사항을 실시간으로 Kafka에 반영하는 CDC 레이어를 담당한다.

## 현재 트렌드

### 트렌드 1: 스트리밍 우선 아키텍처
소스 121(Chip Huyen)이 명시적으로 선언: "배치는 스트리밍의 특수 케이스다." 실시간 피처 컴퓨테이션과 온라인 학습이 배치 파이프라인을 대체하는 방향으로 수렴 중이다. Kafka-ML 프레임워크(소스 126)는 스트림 데이터로부터 모델이 지속적으로 학습하는 구조를 구현한다.

### 트렌드 2: LLM 추론 비용 압박과 최적화
소스 106, 107은 프롬프트 압축(70% 토큰 절감), 시맨틱 캐싱(60–85% 히트율), 모델 라우팅(75% 비용 절감)이 프로덕션의 필수 요소가 되었음을 보여준다. 소스 049는 TGI가 2025년 12월 유지보수 모드에 진입하고 vLLM/SGLang으로 생태계가 재편되는 추세를 기록한다.

### 트렌드 3: 자동 재훈련 루프
소스 122–124는 예약 기반 재훈련에서 이벤트 기반 자동 재훈련으로의 전환을 보여준다. 314e의 7단계 자기 개선 시스템(피드백 포착 → 드리프트 감지 → 스마트 데이터셋 → 재훈련 → 평가 → 배포 → 적응형 임계값)이 대표 사례다.

### 트렌드 4: Kafka ZooKeeper → KRaft 전환
소스 020은 Apache Kafka가 ZooKeeper 의존성을 제거하고 KRaft 합의 프로토콜로 완전 전환 중임을 기록한다. 이는 운영 복잡성을 크게 낮추고 클러스터 규모를 제한 없이 확장하는 기반이 된다.

### 트렌드 5: LLM 안전성 레이어의 표준화
77%의 기업이 GenAI 보안 침해를 경험(IBM 2025, 소스 110). NeMo Guardrails(Colang DSL), Llama Guard(콘텐츠 분류), Guardrails AI(구조적 출력 검증)가 서로 다른 레이어를 담당하는 계층적 방어 구조가 표준으로 자리 잡고 있다.

### 트렌드 6: MLOps 성숙도 모델과 Level 4 지향
소스 009(Microsoft Azure 5단계 모델)는 Level 4(드리프트 기반 자동 재훈련 + 정책 기반 프로모션)를 목표 상태로 정의한다. 대부분의 기업은 현재 Level 1–2에 머물러 있다.

### 트렌드 7: DPO의 RLHF 대체
소스 125는 2025년 기업 RLHF 채택률의 70%가 DPO(Direct Preference Optimization) 또는 변형으로 전환되었음을 보고한다. DPO는 별도 보상 모델 없이 선호도 데이터만으로 동등한 성능을 달성한다.
