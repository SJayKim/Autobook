## 1. 커리큘럼 계층 규칙

| 용어 | 정의 |
|------|------|
| **Phase** | 학습의 **큰 국면**. 시간·난이도·관점 전환(예: 사전 지식 → 기초 → 응용 → 내부 원리 등)을 묶는 최상위 묶음. |
| **섹션** | 한 Phase 안에서 **같은 주제 축**으로 묶인 연속 단원 블록. 하나의 "챕터 묶음" 또는 "모듈"에 해당. |
| **세부 주제** | 산출물의 **최소 학습 단위**. 독자가 "한 번에 이해를 마칠 수 있는" 논리적 단위. 교재 본문의 **주된 원자(배치 단위)**. 하나 이상의 원자적인 개념에 대한 상세한 기술이며, 교재에서는 보통 **문서(파일) 하나**와 1:1로 대응. |

**목적:** 순서만 있는 목록이 아니라, (1) 의존성을 명시하고 (2) 단위마다 학습 목표를 고정하고 (3) 산출물 경로를 기계적으로 매핑할 수 있게 한다.
**포함 관계:** `세부 주제 ⊂ 섹션 ⊂ Phase`. 세부 주제는 정확히 하나의 섹션에 속한다. 섹션은 정확히 하나의 Phase에 속한다.

### 1.1. Phase를 새로 둘 때

Phase는 "챕터를 여러 개 묶은 파트"가 아니라, **학습자의 역량적 위치가 바뀌는 국면** 단위입니다. 아래 중 **하나라도 분명하면** 새 Phase를 고려합니다.

| 기준 | 질문 (예) |
|------|-----------|
| **역량 종류** | 이전 Phase 종료 시점과 비교해, 독자가 할 수 있는 일의 **종류**가 바뀌는가? (예: 도구 조작 → 시스템 설계) |
| **인지적 관점** | 같은 사물을 **다른 층위**에서 보게 되는가? (예: 사용자 관점 → 구현·내부 관점) |
| **전제 장비·맥락** | **도구 스택·환경·전제 지식 덩어리**가 한 덩어리로 바뀌는가? (예: 로컬 단일 프로세스 → 분산 클러스터) |
| **통합 목표** | Phase 말미에 "여러 섹션을 **한 시나리오**로 묶는" 통합 체크포인트를 두기에 **자연스러운**가? |

**Phase로 올리지 말아야 할 것:** 단순히 **주제 제목만** 바뀌었을 뿐, 위 네 가지가 그대로인 경우 → **새 섹션**으로 충분한 경우가 많습니다.
Phase마다 한 문장의 **출구 문장**을 이용해 Phase 추가를 판단하는 것을 권장합니다. “이 Phase를 마치면 독자는 ______ 할 수 있다.”

### 1.2 섹션을 새로 둘 때 (Phase 안의 모듈)

섹션은 한 Phase 안에서 **같은 주제 축·같은 어휘(용어 체계)·같은 모델**로 이어지는 세부 주제의 묶음입니다. 아래 중 **하나라도** 해당되면 섹션을 나눕니다.

| 기준 | 질문 (예) |
|------|-----------|
| **주제 축** | 이 섹션 전체를 표 제목 한 줄로 묶었을 때 **하나의 명사구**로 요약되는가? (요약이 "A와 B의 나열"이면 섹션을 쪼갤 후보) |
| **용어·모델 전환** | 핵심 개념도 **다른 도메인 언어**를 쓰는가? (예: 셸·프로세스 → Git 객체 모델) |
| **난이도 곡선** | 한 섹션 안에서는 "한 단계씩 쌓는" 서사가 유지되는가? 중간에 **완전히 다른 기술**이 끼면 섹션 경계 후보 |
| **선수의 군집** | 세부 주제들끼리 **서로 앞뒤로 많이 참조**하고, 바깥 섹션은 주로 "섹션 입구"에서만 만나는가? (내부 밀도가 높으면 한 섹션에 두기 좋음) |

**Phase와 섹션 구분 (한 줄):** "**관점·역량·맥락이 바뀌면 Phase**, 그 안에서 **주제 축·어휘가 바뀌면 섹션**."

**경험적 힌트 (강제 아님):** 한 섹션에 세부 주제가 **너무 많으면**(예: 스무 개 이상) 읽기 부담이 커지므로, Phase는 유지한 채 **주제 축을 다시 쪼개** 섹션을 늘리는 것을 검토합니다. 반대로 한 섹션에 세부 주제가 **한두 개뿐**이면, `2. 커리큘럼 작성 규칙.md`의 **상세 교재 전제**에 맞게 **개념을 더 잘게 나누어 토픽 수를 늘리거나**, 섹션·Phase 배치를 재검토합니다. **토픽을 줄이기 위해 섹션·토픽을 합치는 것은 이 규칙 집합과 맞지 않습니다.**

### 1.3 세부 주제를 새로 둘 때 (`a.b.c` 한 칸)

세부 주제는 교재 **지식 그래프의 노드**이자 최소 학습 단위이며, **교재 파일 하나**와 맞물립니다. 아래 중 **하나라도** 해당되면 한 세부 주제를 **둘 이상의 세부 주제**로 나눕니다.

| 기준 | 질문 (예) |
|------|-----------|
| **검증의 독립성** | **학습 목표**를 한 문장으로 적었을 때, 끝에 붙는 검증이 **서로 다른 두 가지 이상**인가? (각각 별도 문장·별도 통과 조건이면 분리 후보) |
| **선수 그래프** | 내용의 절반은 선수 A가, 나머지는 선수 B가 필요한데 **한 노드에 섞여** 있는가? (DAG가 단순해지도록 쪼갬) |
| **한 세션 분량** | 한 번 읽고 한 번에 검증하기에 **과도하게 길거나** 중간에 "여기서 끊고 다음 파일로"가 자연스러운가? (과목별 가이드 길이가 있으면 그에 따름) |
| **출판 단위** | 파일·번역·리뷰·재사용을 **한 덩어리**로 묶고 싶은 경계가 제목 단위와 맞는가? (부차적이지만 운영상 이유로 분리 가능) |

**합치기(역방향·거의 사용하지 않음):** 이론적으로 두 세부 주제가 **완전히 동일한 학습 목표 문장**·**동일 선수**·**중복 본문만 증가**인 경우에만 병합을 **감수자 판단으로** 검토합니다. `2. 커리큘럼 작성 규칙.md`는 **상세 교재만** 가정하므로, **병합으로 토픽 수를 줄이는 일은 원칙적으로 하지 않습니다.**

**형식:** 세부 주제마다 **학습 목표 한 문장**(또는 밀접한 소수)을 커리큘럼 표에 고정해 두면, 위 기준을 적용하기 쉽습니다.

### 1.4 세부 주제 밀도(상세 교재 필수)

`2. 커리큘럼 작성 규칙.md`가 적용되는 커리큘럼은 **항상** 아래를 만족한다.

| 원칙 | 설명 |
|------|------|
| **분할 우선** | §1.3의 분할 신호(검증이 둘 이상, 선수가 섞임, 한 세션에 과다)에 해당하면 **세부 주제를 쪼갠다.** |
| **단원 1:1** | 교재 파일(단원) 하나가 **하나의 세부 주제**에 대응하도록 밀도를 맞춘다. |
| **토픽 수 하한** | 세부 주제 **총개수 50 이상**은 `2. 커리큘럼 작성 규칙.md` **§1 항목 4**의 필수 조건이다. |
| **섹션 크기** | §1.2 경험적 힌트(한 섹션에 세부 주제가 지나치게 많으면 섹션 분할)를 적용한다. 토픽이 많아지면 **섹션을 주제 축으로 더 잘게 나누는** 편이 자연스럽다. |

## 2. 번호 체계

- **Phase:** `a` (정수, **0 또는 1부터** 과목 전체에서 일관되게. 한 커리큘럼 안에서 혼용하지 않는다.)
- **섹션:** `a.b` (`b`는 해당 Phase 안에서의 순번).
- **세부 주제:** `a.b.c` (`c`는 해당 섹션 안에서의 순번).
