# 4. 세부 개념 문서 작성 규칙

> 한 권의 커리큘럼에 속한 **세부 주제(`a.b.c`)마다 단원 파일 하나**를 쓸 때 지킬 **저장 위치·조사 순서·정보 수집·목적 정합·본문 포맷**을 정리한다.  
> **말투·서술 태도**는 `3. 책 작성 톤앤 매너.md`를 따른다(특히 **§14 전문 지식 가정 금지·배경 우선·장문 허용**). **Phase·섹션·토픽 분할**은 `1. 커리큘럼 계층 규칙.md`, 커리큘럼 계약·검증은 `2. 커리큘럼 작성 규칙.md`를 따른다. **사용자와 한 단원씩 진행하는 절차**는 `5. 사람-AI 협업 집필 절차.md`를 따른다.

---

## 1. 교재 디렉터리와 파일 배치

### 1.1 루트

- 단원 본문은 저장소 정책에 따라 **`./교재/{책이름}/{x._Phase 이름}/{x.x._섹션 이름}/{x.x.x._세부 주제 이름}.md`** 아래에 둔다.
- 해당 책의 커리큘럼 JSON인 `교재/{책이름}/curriculum.json`을 참조하여, **같은 `{책이름}` 아래**에 단원 트리를 맞춘다.

### 1.2 계층 (폴더 3단 + 파일 1개)

커리큘럼의 **Phase → 섹션 → 세부 주제**와 동일하게 디렉터리를 만든다.

| 단계 | 디렉터리·파일 | 이름 규칙 (권장) |
|------|----------------|------------------|
| Phase | `{a}_{Phase 제목}` | `a`는 커리큘럼의 Phase 번호와 같게. 제목은 짧은 한글(또는 책에서 쓰는 공식 표기). |
| 섹션 | `{a.b}_{섹션 제목}` | `a.b`는 섹션 id와 같게. |
| 세부 주제 | `{a.b.c}_{세부 주제 제목}.md` | `a.b.c`는 토픽 id. 제목은 커리큘럼의 `topic.title`과 대응되게 하되, 파일명에 쓰기 어려운 문자는 `_` 등으로 치환한다. |

**원칙:** 커리큘럼에 없는 번호·이름의 폴더나 파일을 임의로 추가하지 않는다. 토픽이 분할·추가되면 **커리큘럼을 먼저 고친 뒤** 경로를 맞춘다.

### 1.3 생성 절차

1. 해당 `a.b.c` 토픽이 커리큘럼에 존재하는지 확인한다.  
2. `{a}_…` → `{a.b}_…` 경로가 없으면 **순서대로** 만든다.  
3. `{a.b.c}_….md` 파일을 한 개씩만 추가한다. **한 토픽당 한 파일**을 목표로 한다 (`2. 커리큘럼 작성 규칙.md` §1의 단원 1:1 전제와 맞춘다).

## 2. 조사 순서: Ontology와 Web

**한 세부 개념 파일을 쓰기 시작할 때** 아래 순서를 기본으로 한다.

### 2.1 커리큘럼 확인

1. 해당 토픽의 **`learning_objectives`**, **`learning_content`**, **`prerequisites`**, **`lab`**(있는 경우)를 읽는다.  
2. 선수 토픽 `a.b.c` 목록을 열어, 본문에서 **인용할 범위**를 정한다.

### 2.2 Ontology 우선

1. **`list_documents`**(workspace_id는 프로젝트 설정값)로 문서 범위를 짚는다.  
2. **`learning_content`**의 키워드를 이용해 **`query_data`**를 여러 번 수행한다. `top_k`·`chunk_top_k`를 조정해 넓혔다 좁혔다 한다.  
3. 후보 청크가 보이면 **`get_chunks`**로 본문을 직접 연다.  
4. entities의 `chunk_refs`, relationships의 `chunk_ref`로 **개념–청크**를 작업 메모에 적어 둔다.

**역할:** 학습 경로·용어 통일·선후행·첫 설명 각도의 **1차 근거**로 쓴다.

### 2.3 Web 보완 (2차)

Ontology만으로 **`learning_content` 항목을 채우기에 부족**하거나, **스펙·버전·보안·CLI 동작**이 불명확할 때 수행한다.

1. **공식 문서·스펙**을 우선한다.  
2. 블로그·튜토리얼은 **공식과 모순되지 않는지** 확인한 내용만 반영한다.

### 2.4 본문 작성 중 교차

- 초안을 쓰다가 **매커니즘이 끊기면** Ontology `query_data` / `get_chunks`로 다시 Ontology를 검색하고, 여전히 부족하면 Web으로 메운다.  
- **순서를 바꿔 "웹만 보고 시작"하지 않아야** 한다(커리큘럼·그래프와 어긋난 설명 방지).

## 3. 탐색 중 발견한 개념의 심층 검색

탐색 중 **새 용어·하위 메커니즘·연쇄 API**가 나오면, 다음을 **반복**한다.

1. **이 토픽 범위인가?**  
   - `learning_content`·목표에 들어가야 하면 → 본 토픽에서 풀거나, 이미 다른 `a.b.c`에 있다면 그 단원만 짚고 넘어간다.  
   - 범위 밖이면 → 커리큘럼에 토픽이 있는지 찾고, 없으면 **커리큘럼 개정**을 검토한다(임의로 길게 끌어오지 않는다).

2. **Ontology 심층**  
   - 새 키워드로 다시 `query_data`.  
   - 관계(relationships)에 나오는 **연결 개념**이 있으면 그쪽 키워드로 한 번 더 쿼리한다.  
   - 핵심 청크는 `get_chunks`로 통독한다.

3. **Web 심층**  
   - 공식 문서의 해당 절·예제·폐기(deprecated) 표기를 확인한다.  
   - "한 줄 요약"만 있는 주제는 **단계 목록이 나오는 문서**를 찾아 매커니즘을 채운다.

4. **수집 메모 갱신**  
   - "이 개념은 `learning_content`의 어느 줄에 대응되는가"를 메모에 표시해, 작성 후 **빈 항목이 없는지** 대조한다.

## 4. 검색 결과 수집 방법

- **최종 `.md`에는** 출처·doc_id·URL·인용문을 넣지 않는다 (`3. 책 작성 톤앤 매너.md` §13, `2. 커리큘럼 작성 규칙.md`의 출처 정책과 동일 취지).
- **작업용**으로만 다음을 권장한다.  
  - 토픽별 스크래치(로컬 메모, 팀 위키, 비공개 노트 등): 키워드, 청크 id, 짧은 인용, "본문에 넣을 판단 한 줄".  
  - **`learning_content` 체크리스트:** 항목마다 ✓/부족/외부보완 표시.  
  - **`learning_objectives` 검증 문장:** 초안 완성 후 "이 문장이 본문만 읽고 통과 가능한가?"를 자가 점검.

수집의 목적은 **저작권 표기가 아니라**, 커리큘럼 계약(`learning_content`·목표)을 **빠짐없이 만족시키는 것**이다.

## 5. 커리큘럼 목적과 단원의 정합

`2. 커리큘럼 작성 규칙.md` §1의 목적을 **단원 단위**로 다음에 맞춘다.

| 목적 | 단원에서 할 일 |
|------|----------------|
| **개념적 설명력** | 인과·경계 조건을 문장으로 풀어 쓴다. 독자가 "다른 사람에게 말로 설명"할 수 있는 밀도인지 검토한다. |
| **무지원 구현력** | 선수 토픽 + 본문 + `lab`만으로 실습·재현이 되는지 확인한다. 외부 블로그 없이도 되게 필요한 정의를 본문에 넣는다. |
| **매커니즘 상세·누락 없음** | `learning_content`의 **모든** 키워드·개념이 본문 어딘가에서 다뤄지는지 대조한다. 빠지면 본문을 보강하거나, 범위 초과면 커리큘럼을 수정한다. |

작성 후 **한 번 더**: `lab`이 있으면 설명과 순서가 맞는지, 선수 번호로 짚은 단원과 모순이 없는지 본다.

## 6. 각 세부 개념 파일의 구성

### 6.1 최상단 제목

- 한 줄: **`# a.b.c 제목`** (`a.b.c`는 커리큘럼 토픽 id, 제목은 토픽 제목과 대응).

### 6.2 본문

- **`##` 이하의 중간 헤더 없이** 서술형 문단으로 이어 간다 (`교재_작성_가이드.md` §4.2와 동일). Bullet 방식으로 기술하지 않는다.
- **도입:** 독자 상황·문제·왜 이 개념이 필요한지부터 쓴다.  
- **전개:** 배경 → (선수에서 이미 약속된) 개념 → 정의 → 매커니즘 순. 배경지식이 없는 독자도 매커니즘에 대해 명확히 이해할 수 있도록 각 논리적 단계를 차근차근 설명한다. `3. 책 작성 톤앤 매너.md` §3·**§14**.  
- **선수 연결:** 앞에서 배운 내용은 **`a.b.c` 번호**로 짚는다.  
- **핵심 용어:** 첫 등장 시 **굵게** 두는 경우가 많다. 책에서 처음 등장하는 개념은 반드시 내용을 상세히 설명하고 넘어가야 한다. **전문가 전제 없이 배경부터 풀 것·단원 분량 상한 없음(1만 자 초과 허용)**은 §14에 따른다. 
- **명령·코드·설정·출력:** 펜스 코드 블록으로 두고, 앞뒤 문단에서 의미를 푼다.  
- **긴 흐름:** 필요 시 화살표·단계 나열의 **ASCII 다이어그램**을 넣어 한 번 더 요약한다.

### 6.3 말미

- **`정리하면,`**으로 한 문단 요약.  
- **다음 단원**이 있으면 `a.b.c`와 함께 "다음에는 무엇을 다루는지"를 짧게 안내한다.  
- **`lab`·학습 목표가 커리큘럼에 명시되어 있으면**, "이 단원을 마치면 … 설명할 수 있다" 형태로 **능력 문장**을 말미에 두는 것을 권장한다(기존 일부 단원은 정리·다음 안내·능력 문장을 함께 씀).

### 6.4 범위 밖 명시

- 커리큘럼에서 의도적으로 뺀 내용은 "본 토픽 범위에는 포함하지 않는다"고 **한두 문장**으로 경계를 짓는다(과도한 부록화는 피한다).

## 7. 작성 후 체크리스트 (요약)

- [ ] 경로·파일명이 커리큘럼 `a.b.c`·제목과 대응하는가.  
- [ ] `learning_content` 전 항목이 본문에 반영되었는가.  
- [ ] `learning_objectives`를 본문만으로 검증할 수 있는가.  
- [ ] 선수 밖 개념을 예시에 끌어오지 않았는가.  
- [ ] `3. 책 작성 톤앤 매너.md`를 위반하는 말투가 없는가.  
- [ ] 새 용어·전제마다 **전문가 가정 없이 배경부터** 풀었는가(§14). 분량을 아끼려 암묵적 전제를 두지 않았는가.  
- [ ] 독자용 본문에 출처·원서 표기가 없는가.
