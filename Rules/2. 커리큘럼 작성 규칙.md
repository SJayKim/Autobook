# 2. 커리큘럼 작성 규칙

> Ontology·웹 조사를 바탕으로 **커리큘럼 JSON**(`curriculum.schema.json` 준수)을 만들 때 지켜야 할 절차·품질 기준·검증 단계를 정리한 문서입니다.  
> **이 규칙 집합이 적용되는 모든 커리큘럼은 한 가지 교재 형태만 가정한다: 개념을 하나씩 차근차근 전개하는 상세 교재.** 요약본·치트시트 전용 목차는 이 문서의 적용 범위가 아니다.  
> 계층(Phase·섹션·세부 주제)을 **언제 나누는지**는 `1. 커리큘럼 계층 규칙.md`를 따르며, **단원 밀도·토픽 수 하한·분할 우선**은 본 문서 **§1**과 `1. 커리큘럼 계층 규칙.md` **§1.4**를 따른다.

**커리큘럼 작성은 아래 §2의 필수 생명주기를 모두 거친 결과물만 인정한다.** 온톨로지 선행 조사를 생략하거나, 최소 참조 횟수 미만이거나, §1 목적·형태 조건을 충족하지 않거나, §1 목적 달성 여부를 검증·개선하지 않은 채 한 번만 작성하고 끝낸 것은 이 규칙에 맞는 커리큘럼이 아니다.

---

## 1. 이 규칙이 지향하는 교재 목적과 형태(상세 교재 전제)

커리큘럼은 곧바로 **교재 단원(토픽)의 목차·선수·범위 계약**이 됩니다. 다음을 만족하도록 설계합니다.

1. **개념적 설명력:** 도메인에 속한 학습 목표를 독자가 **다른 사람에게 자기 말로** 설명할 수 있다. 암기된 문장 반복이 아니라, 인과·경계 조건까지 포함한 설명이 가능해야 한다.
2. **무지원 구현력:** 교재·실습만으로 **외부 힌트 없이** 요구된 동작을 재현·구현할 수 있다. 필요한 전제는 모두 **선수 토픽** 또는 **동일 토픽의 `learning_content`** 안에 들어 있어야 한다.
3. **매커니즘의 상세·명확·누락 없음:** 각 토픽의 `learning_objectives`는 "어떻게 동작하는지"까지 검증 가능하게 쓰이고, `learning_content`는 본문 작성 시 **빠지면 안 되는 키워드·개념**을 빠짐없이 나열한다.

**4. 상세 교재 형태(필수):** 본 규칙으로 만든 커리큘럼은 **항상** 아래를 동시에 만족한다.

- **한 토픽 = 한 초점:** 각 세부 주제(`a.b.c`)는 **검증 가능한 학습 목표 하나의 초점**(또는 떼어 낼 수 없는 한 쌍)에만 대응한다. API·용어·메커니즘이 한 토픽에 여럿이면 `1. 커리큘럼 계층 규칙.md` §1.3·§1.4에 따라 **분할한다.** **토픽 병합으로 개수를 줄이는 것은 원칙적으로 하지 않는다.** (동일 학습 목표·동일 선수·본문이 완전 중복일 때만 감수자 판단으로 예외 허용.)
- **단원 1:1:** 교재 파일(단원) 하나는 **하나의 세부 주제**에 대응하는 것을 목표로 한다.
- **토픽 수 하한:** JSON의 **세부 주제 총개수**(모든 Phase·섹션의 `topics` 합)는 **50 이상**이다. **50 미만인 커리큘럼은 이 규칙에 맞는 완료 산출물이 아니다.** 하한을 채우기 위해 Phase·섹션 뼈대는 유지하고 **세부 주제를 더 잘게 나눈다.** 넓은 제목 하나에 긴 `learning_content`만 붙이는 방식은 금지한다.

위 네 항목이 곧 **커리큘럼의 목적·형태**이며, §2.3의 검증·개선 루프는 이들이 충족된다고 판단될 때까지 반복한다.

### 1.1 `1. 커리큘럼 계층 규칙.md`와의 역할 나눔

- **Phase·섹션 경계:** 계층 규칙 §1.1~1.2.
- **세부 주제 분할·밀도·50토픽 하한의 운용:** 계층 규칙 §1.3~1.4와 본 문서 §1 항목 4를 **동시에** 만족해야 한다.

## 2. 필수 생명주기: 온톨로지 선행 → 작성 → 목표 달성까지 검증·개선

커리큘럼은 **반드시** 다음 순서와 조건을 따른다. 순서를 바꾸거나 단계를 건너뛰지 않는다.

### 2.1 단계 A — 커리큘럼 초안 전: 온톨로지 기반 개념·관계 파악 (필수)

**커리큘럼 JSON(또는 상세 목차 표)을 작성하기 전에**, 지식 그래프(ontology)를 참조하여 **학습에 필요한 원자적 개념**과 **개념들 사이의 관계**를 먼저 파악한다.

1. **온톨로지 참조 최소 횟수:** `list_documents`, `query_data`, `get_chunks` 등 ontology MCP(또는 동등한 지식 그래프 API)에 대한 **개별 호출·쿼리 1회당 1회**로 세어, **합산 50회 이상** 수행한다. 동일 질의를 무분별하게 중복 호출하여 숫자만 채우는 것은 금지하며, **의미 있는 탐색 단위**(문서 범위 확인, 키워드별 탐색, 청크 확정, 엔티티·관계 확인 등)로 호출한다.
2. **파악해야 할 산출(작업 메모):**
   - **원자적 개념:** 토픽 후보가 되는 최소 지식 단위(나중에 `a.b.c`와 맞물리는 후보)를 목록화한다.
   - **개념 간 관계:** 선행·후행, 포함, 의존, 대조 등 그래프상 관계를 정리하고, 이를 토픽 단위 `prerequisites`·Phase 경계 후보로 옮긴다.
   - **커버리지:** 온톨로지에 얇은 구간은 표시해 두고, §3.2에 따라 웹·공식 문서로 보완할지 결정한다.
3. **완료 조건:** 위 산출이 있을 것. **50회 미만**이면 단계 A를 완료한 것으로 보지 않으며, 이 상태에서 커리큘럼 초안 작성을 시작하지 않는다.

### 2.2 단계 B — 커리큘럼 작성

단계 A에서 정리한 **원자적 개념·관계**를 근거로 `1. 커리큘럼 계층 규칙.md`에 맞춰 Phase·섹션·세부 주제를 배치하고, `curriculum.schema.json`에 맞는 JSON(또는 동등한 상세 표)을 작성한다. 웹·공식 문서 보완이 필요한 토픽은 §3에 따라 반영한다.

**필수:** 원자적 개념 목록을 토픽 후보와 **가능한 한 1:1 또는 근접**하게 맞춘다. 총 토픽 수가 **50 미만이면** Phase·섹션을 유지한 채 **세부 주제만 추가 분할**하여 하한을 넘긴다. “넓은 제목 하나 + 긴 `learning_content`”로 때우지 않는다.

### 2.3 단계 C — 목적 달성까지 검증·개선 (필수, 반복)

초안 작성 후 **한 번에 끝내지 않는다.** §1의 목적·형태(개념적 설명력, 무지원 구현력, 매커니즘의 상세·명확·누락 없음, **항목 4 상세 교재 조건**)가 **실제로 달성되는지**를 기준으로 끊임없이 검증하고 커리큘럼을 고친다.

1. **검증 예시(전부가 아님):** 선수 그래프 DAG·순서, 토픽별 목표 문장의 검증 가능성, `learning_content` 누락·과다(분할 필요), Phase `exit_capability`와 토픽 범위의 정합, 온톨로지·교재와의 커버리지 대조.
2. **형태 검증(필수):** (a) 세부 주제 **총개수 ≥ 50**, (b) 토픽마다 **단일 초점**(한 목표에 검증 방식이 여럿이면 분할), (c) `learning_content`가 한 토픽에 과도하게 길면 **개념 단위로 쪼개졌는지**, (d) **병합으로 토픽 수를 줄인 흔적**이 없는지(§1 항목 4의 예외 병합만 허용).
3. **개선 루프:** 결함·공백이 나오면 **온톨로지를 다시 참조**하고(필요 시 추가 쿼리로 관계·청크를 재확인), **토픽 분할**·선수 수정·목표·키워드 보강을 반복한다. 웹 보완이 필요하면 §3.2를 따른다. **토픽 병합**은 §1 항목 4에 명시된 예외(완전 동일 목표·선수·중복 본문)에 한해 검토한다.
4. **종료 조건:** §1 전체(목적 1~3 + **형태 4**)를 **충족한다고 명시적으로 판단할 수 있을 때까지** 검증·개선을 멈추지 않는다. "초안만 있음"은 완료가 아니다.

## 3. 입력 소스와 사용 규칙

### 3.1 Ontology (지식 그래프)

교재_작성_가이드의 curriculum 선행 단계와 동일한 취지로, **§2.1(최소 50회)은 커리큘럼 초안 이전에 반드시 수행**하고, §2.3 개선 단계에서도 필요 시 ontology를 추가로 조사한다.

1. **문서 목록:** `list_documents`(workspace_id 지정)로 포함 문서·범위를 파악한다.
2. **주제 탐색:** 후보 토픽마다 한글·영문 키워드로 `query_data`를 여러 번 수행한다. `top_k`·`chunk_top_k`를 조정해 범위를 넓힌다.
3. **청크 직접 조회:** 필요 시 `get_chunks`로 doc_id·chunk_id를 지정해 본문을 확인한다.
4. **개념–청크 매핑:** entities의 `chunk_refs`, relationships의 `chunk_ref`로 "어떤 개념이 어디서 다뤄지는지"를 적어 둔다(작업 메모; 최종 교재에는 노출하지 않음).
5. **선수 관계:** relationships 등에서 개념 간 관계가 보이면 해당 개념 쌍을 토픽 단위 `prerequisites`로 옮길 후보로 한다.
6. **분할 우선:** 한 개념이 청크 여러 개에 걸쳐 크면 토픽을 쪼갠다. 한 청크에 개념이 많으면 **여러 토픽으로 나누거나** 토픽별 `learning_content`에 키워드를 나눠 담아 **누락 방지**한다. 병합으로 토픽을 줄이지 않는다(§1 항목 4 예외만).
7. **커버리지 공백:** ontology에 거의 없는 주제는 **웹·공식 문서로 보완**한다는 전제를 기록하고(검증 표의 "외부 보완" 열 등), 가능하면 해당 토픽의 `learning_content`를 더 촘촘히 잡는다.

### 3.2 웹·공식 문서

- ontology가 얇은 구간, 최신 스펙·CLI 동작, 보안 권고 등은 **공식 문서·스펙**을 우선한다.
- 교재 정책상 최종 본문에는 출처를 밝히지 않더라도, **커리큘럼 작성 단계**에서는 검증·감수를 위해 내부용으로 "어떤 공백을 무엇으로 채웠는지"를 팀 메모나 별표에 남기는 것을 권장한다(JSON 스키마 필수 필드는 아님).

### 3.3 두 소스의 역할 분담

| 역할 | Ontology | 웹 등 |
|------|----------|--------|
| 학습 경로·선후행 | relationships·엔티티 밀도로 후보 도출 | 스펙상 필수 전제 보강 |
| 토픽 쪼개기 | 청크 분산·압축 정도 | 장문 튜토리얼 단계 분해 참고 |
| 품질 | 커버리지 매트릭스의 1차 근거 | 공백·최신성 보완 |

## 4. 산출물: Markdown 표 → JSON 매핑

기존 `*_CURRICULUM_DETAILED.md` 스타일 표를 JSON으로 옮길 때 다음에 대응시킨다.

| 표(마크다운) | JSON |
|--------------|------|
| `#` (0.1.1 등) | `topic.id` (`a.b.c`) |
| Phase·섹션 구조 | `phases[]` → `sections[]` → `topics[]` |
| 세부 주제 제목 | `topic.title` |
| 선수 | `topic.prerequisites` (가능하면 `a.b.c`만; 축약 시 루트에 `prerequisite_shorthand`) |
| 학습 내용 | `topic.learning_content` (문자열 배열) |
| 학습 목표 | `topic.learning_objectives` (문자열 배열; **권장 1문장**, 필요 시 밀접한 2문장) |
| 실습 | `topic.lab` (`null` 또는 `{ "description", "commands?" }`) |
| Phase 출구 | `phase.exit_capability`, `phase.phase_checkpoint` (선택) |
| 섹션·Phase 체크포인트 | `section_checkpoint`, `phase_checkpoint` (선택) |

루트 메타:

- `schema_version`, `curriculum_id`, `numbering.phase_base` (0 또는 1, 문서 전체 일관)
- `prerequisite_ordering.mode`: 순서 검증을 할 경우 `id_numeric_tuple` 또는 `global_sequence` (후자는 각 `topic.global_sequence` 부여)
- 상세 필드 정의는 `curriculum.schema.json`을 따른다.

## 5. JSON 토픽을 쓸 때의 품질 규칙

### 5.1 `learning_objectives`

- **관측 가능한 동사:** "설명할 수 있다", "구별할 수 있다", "재현할 수 있다", "단계별로 따라 구현할 수 있다" 등으로 끝맺는다.
- **매커니즘:** "사용해 본다"만이 아니라, **왜·어떤 순서로·무엇이 바뀌는지**를 말·글로 검증할 수 있을 정도로 구체화한다.
- **하나의 초점:** 한 토픽에 목표가 여러 개면 `1. 커리큘럼 계층 규칙.md` §1.3에 따라 **반드시 쪼갠다.** 한 토픽에는 **권장 1문장**(필요 시 밀접한 2문장)만 두어 본문이 “개념 하나” 단위로 끊기게 한다.

### 5.2 `learning_content`

- 교재 작성 시 **반드시 다룰** 키워드·구성요소·API·명령·데이터 구조를 나열한다.
- "나중에 본문에서 알아서"로 남기지 않는다. 나열이 길어지면 **토픽 분할**한다. **한 토픽의 키워드가 한 화면(또는 단원 한 절)을 넘어설 정도로 길면** **여러 토픽으로 나눈다**.

### 5.3 `prerequisites`

- **후방 참조 금지:** 토픽 A가 전제하는 개념은 모두 선수 토픽의 목표·내용 범위 안에 있어야 한다(번호 순서만으로 보장하지 않음).
- **DAG:** 순환 의존이 없어야 한다.
- 스키마에 없는 추가 규칙으로, **선수 id(또는 `global_sequence`)가 본인보다 앞선다**는 검증을 통과시킨다.

### 5.4 Phase·섹션

- `1. 커리큘럼 계층 규칙.md`의 §1.1~1.4를 만족하는지 작성 중·작성 후에 점검한다.
- Phase마다 `exit_capability`(출구 문장)를 두는 것을 권장한다.

## 6. 커리큘럼 JSON 산출물 저장 위치

- `curriculum.schema.json`에 맞춘 **커리큘럼 JSON**은 **`./교재/{책이름}/`** 아래에 저장한다. `{책이름}`은 그 교재(프로젝트)를 구분하는 디렉터리 이름이다.
- **`./교재/{책이름}` 경로가 없으면 만든다** (예: `mkdir -p "./교재/{책이름}"`). 상위의 `./교재`도 없으면 함께 생성한다.
- 파일명은 팀 관례로 정하되, 예시로 **`curriculum.json`**을 그 폴더에 두는 방식을 권장한다.
- 이 경로 규칙은 **JSON 커리큘럼 산출물**에 적용한다. Markdown 단원 본문을 같은 `{책이름}` 아래로 둘지, 기존 `교재/` 한 겹 구조를 유지할지는 별도로 정하되, **한 책의 JSON은 반드시 `./교재/{책이름}/`에 모은다**고 본다.
